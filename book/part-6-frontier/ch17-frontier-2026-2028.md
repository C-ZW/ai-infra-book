# ch17 — 前沿（2026–2028）：流量在變、模型在變、硬體在變

> **本章解決什麼問題**：前 16 章教你的是一套已經收斂的工程體系——KV cache 管理（ch05）、continuous batching（ch06）、P/D 分離與 KV 路由（ch10）、容量規劃（ch13）。但這個領域的地基還在移動：流量形態正在從 chat 換成 agentic、模型架構正在動搖 KV cache 的數學、硬體路線圖兩年一翻、電力取代晶片成為新瓶頸。本章用 2026-06 查證過的證據分析未來 2–3 年資源會砸向哪裡，並在最後給你一張「能力耐久性矩陣」：哪些知識會在你讀完本書前就過期、哪些十年不變。這是全書觀點最重的一章——凡是推測，我會標明是推測。

## 從你已知的出發

你經歷過「使用者行為改變壓垮原設計」。遊戲後端為人類玩家設計：人類有思考時間、有睡眠週期、操作頻率有上限。然後腳本和外掛來了——機器節奏、無間隔、無日夜——原本健康的容量模型一夜作廢。本章的核心論點是同一件事正發生在 LLM serving 上：**為人類打字節奏設計的推論平台，正在被機器發起的 agentic 流量重新定義**。你用 k6 做過 open-loop 負載測試，知道「無協調遺漏、不等回應就發下一發」的流量和真人流量是兩種生物——agentic workload 就是把你的負載測試變成生產流量。

你也從 framework churn 裡存活過：Express 到 NestJS、REST 到 GraphQL、跑過好幾代 ORM。回頭看，真正保值的不是任何框架的 API，而是 SQL、索引、佇列理論、量測方法論。本章最後那張耐久性矩陣，就是把你已經會做的「押注不變量」判斷，套用到 AI infra 的 2026–2028。

最後，你做過容量規劃的 headroom 思維——不確定的未來不是用猜的，是用對沖的。本章每個預測後面都附「如果我錯了，會錯在哪、你怎麼觀測到」。

## 流量形態革命：agentic workload 正在取代 chat

這是本章的核心論點，先把證據攤開，再講它顛覆了什麼。

### 證據：流量結構已經換血（2026-06）

a16z 與 OpenRouter 合作的「State of AI: 100 Trillion Token Study」（arXiv:2601.10088，2025-12 發布）是目前最大的推論需求面實證研究，分析了 OpenRouter 平台上超過 100 兆 token 的真實流量。幾個關鍵數字：

- **Programming 類請求佔比從 2025 年初的約 11% 漲到超過 50%**——而 coding 正是 agentic 化最徹底的場景。
- **Reasoning 模型的 token 佔比超過 50%**，2025 年 Q1 時接近零。
- **平均 prompt 長度從約 1.5k token 漲到 6k+（約四倍）**，輸出從約 150 漲到 400；programming 流量的輸入經常性超過 20k token。
- Agentic inference（模型規劃→呼叫工具→讀結果→迭代的多步驟序列）是平台上**成長最快的使用形態**。

旁證：Anthropic 的 Economic Index 報告顯示其 API 流量以自動化（automated）使用為主，Claude Code 已佔抽樣 API 流量的大宗，44% 的 API 流量對應到 computer/mathematical 類任務。llm-d 專案引述的生產數據則指出 agentic 流量的 input:output 比可以超過 **100:1**——絕大多數 token 是被反覆重送的 context，不是新內容。

把這些數字疊起來，結論很難迴避：**主流負載已經不是「一個人類打一段字、模型回一段字」，而是「一台機器帶著巨大的固定 context，以工具迴圈節奏連續轟炸 API」**。MCP（Model Context Protocol，2024-11 由 Anthropic 發布、現已交由 Linux Foundation 旗下基金會中立託管）把工具接入標準化之後，這個形態還在加速——標準化的工具定義直接膨脹在每個請求的 prefix 裡。

### 兩種流量，五個維度的差異

| 維度 | Chat（人類節奏） | Agentic（機器節奏） |
|---|---|---|
| 請求間隔 | 30–60s（人在思考、打字） | 0–10s（工具執行完就發） |
| Context 成長 | 每輪幾百 token，session 末端幾 k | 每輪 1–3k（工具結果），末端可達 100k+ |
| Prefix 重複率 | 高（多輪對話），但 session 短 | 極高（system prompt + 工具定義 + 全部歷史逐輪重送） |
| Input:output 比 | 個位數比 1 | 數十到 100+ 比 1 |
| 尖峰成因 | 日夜週期、行銷活動 | CI 觸發、cron、平行分支探索——機器同步性，比人類尖峰陡得多 |

第五點值得多說一句：agentic 流量會出現**平行分支探索**——一個任務 fork 出多個子 agent 同時跑（Kimi K2.6 甚至把 Agent Swarm 做成 API 原語，2026-04）。這在後端視角就是 fan-out 放大器：一個使用者動作，後面是幾十路併發推論。你在遊戲後端見過整點活動的同步尖峰；agentic 尖峰是它的加強版，因為觸發者本身就是程式。

### 它對前面每一章意味著什麼

| 前章設計 | Chat 時代的定位 | Agentic 時代的定位 |
|---|---|---|
| Prefix caching（ch05） | 錦上添花的優化（省 system prompt） | **生存必需**：97%+ 的輸入是重複 prefix，沒有它成本直接 ×30（下面 worked example 算給你看） |
| KV offloading 分層（ch05） | 少數長 context 場景才需要 | 標配：session KV 以數十 GB 計、跨呼叫存活分鐘級，HBM 裝不下也捨不得丟 |
| Chunked prefill / 排程（ch06） | 平衡長短請求 | 長 prefill 請求成為常態而非尾巴，token budget 的拉扯加劇 |
| P/D 分離（ch10） | 大規模才值得的複雜度 | 流量本身就是極端 prefill-heavy（input:output 100:1），xPyD 配比整個改寫 |
| KV-aware routing（ch10） | TTFT 優化手段 | **產品核心**：routing miss = 重算數十秒的 prefill，llm-d 量測 cache-aware vs cache-blind 的 P90 TTFT 差距達 57 倍（專案自報數據） |
| Autoscaling 訊號（ch13） | RPS 已經不準 | 更不準：一個 session 60 次呼叫，併發 session 數與呼叫頻率才是訊號；尖峰同步性逼你加大 headroom |
| Retry storm 防禦（ch15） | 人會放棄重試 | 機器不會：SDK 內建指數退避也擋不住一個寫錯的 agent 迴圈，admission control 從建議變成底線 |
| 成本歸因（ch16） | per-request 計量 | per-session、per-task 計量；cache 命中率直接決定毛利 |

一句話總結我的立場：**chat 時代的推論平台是 stateless web 服務加一點快取；agentic 時代的推論平台是一個分散式狀態機，KV cache 是它的 working set**。這不是漸進演化，是負載性質的相變——就像你的遊戲後端從「玩家請求」變成「整個世界的 bot 都在打你」。

## Worked example：agentic coding assistant vs chat 的流量畫像

用公開數據錨點建兩個流量畫像，然後推導 KV 容量與路由需求的差異。錨點：OpenRouter 研究的 programming 輸入經常 >20k token、平均輸出約 400；llm-d 引述的 input:output >100:1；Anthropic 對 agent 把任務拆成大量 API 呼叫的描述。以下具體參數是**我基於這些錨點的估算**（標示為估），模型用 ch03 的基準 Llama-3.3-70B（GQA、BF16 KV ≈ 320 KB/token）。

**畫像 A：客服型 chat session（估）**

- 8 輪對話，輪間隔約 45s（人類節奏），session 約 8 分鐘
- System prompt 500 token；每輪使用者輸入 80、模型輸出 350
- Session 末端 context：500 + 8 × 430 ≈ **3.9k token**
- 邏輯輸入總量（每輪重送歷史）：約 16.7k token；輸出 2.8k → input:output ≈ 6:1
- 末端 KV：3.9k × 320 KB ≈ **1.2 GB**；呼叫頻率約 1 次/分鐘

**畫像 B：agentic coding session（估）**

- 一個任務 = 60 次模型呼叫（工具迴圈：讀檔、改檔、跑測試、看結果）
- 靜態 prefix：system prompt + 工具/MCP 定義 ≈ 18k token（與「programming 輸入經常 >20k」一致）
- 每次呼叫新增約 2k token（工具結果 + 上一輪動作），輸出平均 1k（含 reasoning 摘要與工具呼叫）
- Context：第 n 次呼叫 ≈ 18k + 2k×n → 末端 ≈ **138k token**，全程平均 ≈ 78k
- 邏輯輸入總量：Σ(18k + 2k×(n−1)) ≈ 60×18k + 2k×1,770 ≈ **4.62M token**；輸出 60k → input:output ≈ **77:1**（與公開的 >100:1 同量級，驗證畫像合理）
- 呼叫頻率約 6 次/分鐘（工具執行 2–10s），是 chat 的 6–10 倍
- 平均 KV：78k × 320 KB ≈ **25 GB**；末端 138k ≈ **44 GB**

**推導一：prefix caching 從優化變成成本結構本身。** 無 cache 時這個 session 要 prefill 4.62M token；有 prefix cache（每輪只算新增的 ~2k suffix）時實際 prefill ≈ 18k + 59×2k ≈ 136k token——**34 倍差距，cache 命中率 97%**。換句話說，agentic 服務商的成本結構裡，prefill 成本幾乎完全由 cache 命中率決定。這就是為什麼各家 API 把 cached input 折扣壓到 90%+（Anthropic read 為基準價 10%、DeepSeek 更深，2026-06，見 ch16）——不是慈善，是 cache 命中後的邊際成本真的只剩零頭，而 agentic 客戶對這個價差極度敏感。

**推導二：KV 容量需求差 20–35 倍，分層儲存從可選變必選。** 1,000 條並發 chat session 的 KV ≈ 1.2 TB；1,000 條 agentic session ≈ 25 TB。ch03 算過 2×H100 留給 KV 的池子約 65 GB——agentic 場景下**兩三條 session 就吃滿**。但工具執行的 2–10s 間隔又意味著 KV 不能丟（丟了就是重算 78k prefill）。唯一出路是 ch05 的分層：HBM 只放正在 decode 的 working set，間隔期掉到 CPU RAM / NVMe，工具回來前搬回去。這正是 vLLM v0.22 把 multi-tier KV offloading 做成框架、Mooncake 把 KV 做成獨立儲存系統的需求面原因（2026-06）。

**推導三：路由的賭注高了兩個數量級。** Chat 的 routing miss 代價是重算 3.9k prefill——亞秒級，round-robin 勉強能活。Agentic 的 miss 是重算 78k prefill——以 ch03 的 prefill 速率估算是數十秒級，TTFT 直接爆掉 SLO。所以 prefix-aware routing（ch10）對 chat 是加分項、對 agentic 是 50–100 倍的 TTFT 槓桿。路由器必須知道「這條 session 的 KV 在哪個 replica」，這就是 session 黏性從 nice-to-have 變成架構約束的數學原因。

## Reasoning 經濟學：當模型開始「想」

第二個結構性變化：test-time compute。thinking/non-thinking 雙模式已是開源旗艦標配（DeepSeek-V4、Kimi K2 Thinking、Qwen3.x，2026-06），OpenRouter 上 reasoning 模型 token 佔比已過半。這改變了三本帳：

**成本帳：輸出 token 爆炸。** Thinking token 按輸出價計費，而輸出價中位數約是輸入價的 4 倍（2026 年第三方統計）。一個複雜請求在思考模式下可能花 20k–40k thinking token 才產出 500 個可見 token（第三方估算，量級可信）——使用者看到一段話，你的 GPU decode 了一本小冊子。對容量規劃（ch13）的衝擊直接：**decode 階段的負載權重大幅上修**，而且 thinking 長度的變異比輸出長度更大（同一個模型對不同難度的題目，thinking 可以差兩個數量級），p99 容量更難壓。

**控制面：thinking budget 成為產品參數。** 各家都在補控制手段——Anthropic 的 effort 參數與 thinking budget、Gemini 的 thinking budget、DeepSeek/Qwen 的雙模式開關（2026-06）。對 infra 工程師的含意：這些參數是你的 brownout 工具箱（ch15）的新成員——容量吃緊時降 effort 換容量，跟你當年降畫質保流暢是同一招。

**計價演化：從 $/token 走向 $/任務。** 這一段是推測，標明立場：我認為 2026–2028 會看到計價單位從 token 向任務/結果遷移。理由：(1) thinking token 讓 $/token 對客戶失去可預測性——同一個問題，模型「想多久」客戶無法控制也無法審計；(2) agentic 場景客戶買的本來就是任務完成，不是 token；(3) 經濟壓力方向一致——Epoch AI 的數據顯示固定能力水準的 $/token 每年下降 9–900 倍，token 本身正在失去作為計價單位的稀缺性。反方論點也誠實列出：token 是物理誠實的成本原子（ch03），任務計價把需求波動風險從客戶轉給供應商，定價工程難度高得多——所以遷移會先發生在垂直封閉產品（coding agent 訂閱制的「用量包」其實已經是變相任務計價），API 層的 $/token 還會活很久。

**矛盾的解法：價跌量爆。** $/token 崩跌與總帳單暴漲同時發生（Jevons 效應的教科書案例）：單價跌 10 倍、單任務 token 用量漲 30 倍，淨效果是推論容量需求持續上升。對你的職涯這是好消息：**推論 infra 的工作量跟 token 總量成正比，不跟單價成正比**。

## 「KV 是新的 Redis」：證據與反證

2026 年業界有一個流行論述：KV cache 正在成為獨立的儲存層級，地位類比當年的 Redis——每個嚴肅的架構圖裡都會有它一格。這個論點值得認真對待，也值得認真質疑。

### 證據

1. **系統已經長出來了。** Mooncake 把 KV cache 做成跨叢集儲存系統並撐起 Kimi 的生產流量（自報多承接 75% 請求）；LMCache 的 backend 清單長得像個儲存產品（CPU RAM、本地 SSD、Redis/Valkey、S3 相容物件儲存、GDS）；vLLM v0.22 內建 multi-tier offloading 框架；NVIDIA 做了 framework-agnostic 的 KVBM；SGLang 有 HiCache。（2026-06，見 ch05/ch10）四條獨立路線收斂到同一個抽象——「KV 是值得獨立管理的儲存層」已是工程共識。
2. **它被商品化定價了。** Gemini 的 explicit caching 按小時收儲存費——這是字面意義的儲存計價；Anthropic 的 cache write 按 TTL 分級收溢價（5 分鐘 1.25x、1 小時 2x）。當一個東西有了「容量 × 時間」的價格表，它就是儲存商品。
3. **需求是結構性的。** 上面 worked example 算過：agentic 流量 97% 的輸入是可重用 prefix。只要 agentic 流量持續成長，KV 重用的經濟價值就持續存在——這不是某個引擎的實作細節，是流量形態的數學性質。

### 反證

1. **KV 是衍生資料，不是 source of truth。** Redis 裡的 session 丟了會出事故；KV 丟了只是變慢——重算就有。它更像 page cache 或 materialized view：值得管理，但永遠有「重算」這條退路。ch10 算過傳輸 vs 重算的 crossover；硬體每進一代，重算就更便宜一分，KV 持久化的價值邊界就退一步。
2. **它跟模型版本、引擎版本、量化配置、甚至 attention backend 綁死**（ch05 講過失效矩陣）。Redis 的 value 是通用 bytes，KV 的 value 是「特定模型某一版的內部狀態」——模型一週一更的世界裡，這份「儲存」的經濟壽命以小時計。沒有任何嚴肅場景會把 KV 當資產存超過一天。
3. **架構移動目標直接威脅它的體積。** 下一節會講：如果 linear/hybrid attention 把 KV 砍掉 75%，或 sparse attention 改寫讀取模式，今天為「TB 級 KV 池」設計的儲存層投資，攤提期可能撐不到回本。

### 我的立場

「KV 是新的 Redis」做為**口號**是誇大，做為**方向**是對的。準確的說法是：KV cache 從引擎的私有實作細節，升格為平台層的一等資源——有自己的容量規劃、命中率 SLO、分層策略與計價。但它是快取階層，不是資料庫：押注「管理 KV 的能力」（分層、路由、失效、計量——全是你會的快取工程）是安全的；押注「某個特定 KV 儲存產品會成為標準件」是危險的，因為上游模型架構一變，整層的形狀就得重來。2026–2028 我預期：KV 管理能力內化進每個 serving 框架（已發生），但獨立的「KV 儲存廠商」市場不會像當年 Redis 那樣長出來——衍生資料養不起 source of truth 的生意。

## 模型架構是移動目標：ch03 的數學還能活多久

全書的記憶體帳都建立在 ch03 的 KV 公式上：每 token 狀態 × token 數，線性成長。2025–2026 有兩條路線在動搖它。

### 路線一：sparse attention（DeepSeek 路線）

DeepSeek 從 MLA（壓係數）走到 NSA（2025-02 論文）再到 DSA——V3.2-Exp（2025-09）首次量產，V4（2026-04 preview）以「token-wise compression + DSA」支撐 1M context。Sparse attention 的本質：**KV 還是全部存，但每步只挑一小撮來讀**（用輕量 indexer 選 top-k）。注意這對 ch03 兩本帳的影響不對稱——**容量帳幾乎不變**（除非搭配壓縮，V4 的 token-wise compression 細節尚未完全公開），**頻寬帳大幅縮小**：ch03 證明過長 context decode 的瓶頸是「每步重讀整份 KV」，DSA 把這項從 O(n) 降到 O(k)。直接後果是 1M context 在經濟上變得可行，以及 DeepSeek 敢把 cache 命中價壓到地板（2026-06，折扣倍率僅見第三方報導，引用前需查官方價目）。

### 路線二：linear / hybrid attention

純 linear attention（每層只維護固定大小的 RNN 式狀態）多年來品質不過關，2025–2026 收斂到的務實解是**混合**：大部分層用 linear、少數層保留 full attention。兩個量產級代表：**Qwen3.5**（2026-02）旗艦 397B-A17B 用 Gated Delta Networks + sparse MoE；**Kimi Linear**（arXiv:2510.26692）以 KDA + MLA 按 3:1 混合，自報 KV cache 減少最多 75%、1M context 下 decode 吞吐 6 倍，kernel 與 vLLM 支援都已就位。混合比例就是新的 KV 係數：3:1 混合 ≈ KV 帳直接打 25 折。

### 反方證詞：MiniMax 的回頭路

誠實呈現反證。MiniMax M1 用了 efficient attention，**M2（2025-10）卻退回 full attention**——官方工程部落格寫得很直白：在工業級規模下，linear/windowed 變體在長鏈推理任務上出現嚴重缺陷，且整條推論基礎設施（kernel、量化、cache 生態）對 efficient attention 的支援不夠成熟，full attention 是「最不脆弱的選擇」。然後劇情再轉折：**M3（2026）又帶著新的 sparse attention 回來了**，自報長 context 速度提升一個數量級。一年內同一家公司走了「linear → full → sparse」三步——這就是「移動目標」的最佳註腳：方向（降低 attention 成本）是確定的，路徑（linear vs sparse vs hybrid）還在劇烈震盪。

### 顛覆程度評估

| ch03 的結論 | Sparse attention 下 | Linear/hybrid 下 |
|---|---|---|
| KV 容量隨 context 線性成長 | 大致仍成立（全存） | **被打破**：linear 層是 O(1) 狀態，只剩 full 層線性成長（×混合比） |
| Decode 頻寬 = 權重 + 整份 KV 重讀 | **被打破**：KV 讀取量 sublinear | 部分成立：full 層仍重讀，linear 層讀固定狀態 |
| Prefill O(n²) | 被緩解（attend 子集） | 被緩解（linear 層 O(n)） |
| KV 是 append-only、可按 block 共享前綴 | 成立 | **語意改變**：RNN 狀態是壓縮摘要，不能像 KV block 那樣任意切前綴共享——prefix caching 必須改成「在 block 邊界存狀態快照」，vLLM 的 hybrid allocator（v0.21 起）就在處理這件事 |

我的評估：**ch03 的公式不會死，但會從「定律」降格為「參數化模型」**。「每 token 狀態 bytes × 成長函數」這個會計框架永遠成立，變的是係數（GQA/MLA 已經教過你係數會變）與成長函數（線性 → 分段線性 → 次線性）。2026–2028 你該押注的不是任何一條公式的數值，而是「拿到任何新架構的 config，十分鐘內重算出它的記憶體帳」這個能力。最深的系統含意在表格最後一行：**prefix caching、PagedAttention、KV routing 全都隱含「狀態可按 token 邊界分解」的假設，linear 層打破它**——這會是 2026–2028 推論引擎最髒的工程戰場之一。

### 多模態：另一條工程線

簡短帶過（它是工程差異不是理論顛覆）：多模態推論在引擎前面多了一段 encoder pipeline（vision encoder → projector → 進主幹），image token 把 prefill 帳單再推高一級、video 是 token 洪水。Serving 端的含意是異質 pipeline——encoder 是另一種資源畫像（compute-bound、可獨立擴縮、甚至放不同硬體），P/D 分離的思路天然延伸成 E/P/D 三段。原理上沒有新東西，全是 ch10 的決策模板再用一次。

三個比規格表更重要的結構性觀察：

**一、Rubin CPX 的插曲：P/D 分離正在硬體化——即使單一產品反覆。** NVIDIA 在 2025-09 發表 Rubin CPX：一張專做 prefill 的卡——128 GB GDDR7（不是 HBM！）、30 PFLOPS NVFP4、無 NVLink。設計邏輯完全是 ch10 的數學：prefill compute-bound，HBM 的頻寬是浪費，用便宜記憶體堆算力。結果 2026 年 3 月前後 CPX 被**擱置**——NVIDIA 以約 $200 億取得 Groq 技術授權與核心團隊（2025-12，非股權收購，Groq 仍獨立營運）後優先做 LPU 基因的 LPX 機架（主攻 decode），NVIDIA 高層公開表示 CPX 概念未死、可能在後代回歸（The Register，2026-06）。然後 Intel 在 Computex 2026 發表 Crescent Island——480 GB LPDDR5x、350W、PCIe 卡，明擺著接 CPX 留下的 prefill 真空，且直接對接 Dynamo 的分離式架構。讀法：**workload 的分化（prefill/decode/長 context）正在變成晶片 SKU 的分化**。P/D 分離從你在 ch10 學的軟體架構，升格為採購目錄上的選項。這也預答了本章 Lab 的辯題：硬體進步不是在淘汰 P/D 分離，是在把它鑄進矽裡。

**二、SRAM 路線被主流收編。** Groq 的 LPU 用 on-chip SRAM 換極端 decode 頻寬——物理上對 memory-bound 的 decode（ch03）是對症下藥，代價是容量小、模型要切碎。NVIDIA 用 ~$200 億把它的技術與核心團隊收編（授權＋人才移轉，2025-12），Cerebras 則完成 IPO（2026-05 上市，估值從 S-1 階段預估的約 $230 億跳到上市時約 $560 億）並拿下 OpenAI $100 億級算力協議（750MW 起、可擴至 2GW）。我的解讀（推測）：推論專用架構不再是「挑戰者敘事」，而是被整合進主流產品線的一個象限——2026–2028 你會看到同一家雲上「HBM 通用卡 + SRAM decode 專用 + 大容量便宜記憶體 prefill 專用」並存，ch13 的硬體選型矩陣會多出一個維度。

**三、rack 是新的伺服器。** NVL72 之後，NVIDIA（Vera Rubin NVL72）與 AMD（Helios）的產品單位都是整機架：72 顆 GPU、一個 NVLink/IF domain、液冷、以百 kW 計的功耗。對 infra 工程師的含意：ch09 的「TP 不出節點」邊界擴大成「TP 不出機架」、ch11 的拓撲感知排程以 rack 為單位、故障域與採購單位也是 rack。「一台 8 卡機」這個心智單位正在過時。

另外兩個影響供給的常數：HBM 產能仍是整個行業的配給瓶頸（HBM4 的爬坡決定 Rubin/MI400 的真實供貨量），以及單卡功耗一路從 700W（H100）爬到 1,400W（B300/MI355X）——這條線直接把我們推進下一節。

## 電力成為一級約束

2026 年的容量瓶頸排序已經改變：**不是買不到卡，是插不上電**。證據（2026-06）：

- Uptime Institute 預估全球 AI 專屬資料中心負載到 2026 年底達 **10 GW** 量級；單一 hyperscale AI 資料中心需要 100–300 MW 連續供電——一座中型城市的用電。
- 美國電網互聯排隊普遍超過 5 年，大型變壓器交期約 5 年；產業統計顯示 2026 年美國公告的約 12 GW 資料中心容量中僅 5 GW 實際動工。
- CoreWeave（SEC 文件）已運轉超過 **1 GW**、目標 2030 年 8 GW+，營收 backlog 近千億美元——neocloud 的競爭壁壘正從「有沒有卡」變成「有沒有電」。

對 infra 工程師的具體含意（前兩點是現在進行式，後兩點是我的推測）：

1. **功耗進入容量規劃公式。** ch13 的卡數計算之外要多一條：機架功率預算 ÷ 每卡 TDP。1,400W 的卡讓「有空位但沒電」成為真實的排程約束。
2. **電價結構影響工作負載排程。** Batch tier 吃離峰容量（ch16）的邏輯延伸：離峰「容量」未來部分是字面意義的離峰電力。
3. **Power-aware scheduling 會成為排程器的一等維度**（推測）：power capping、DVFS、依電價/碳強度搬移 batch 工作——K8s 生態已有雛形，我預期 2027–2028 它從 ESG 報表工具變成省真錢的容量工具。你在 ch11 學的排程器擴展點，到時候就是實作它的地方。
4. **訓練/推論的電力彈性差異會被制度化**（推測）：訓練可以暫停讓電、推論不行（有 SLO），GB300 世代開始出現平滑同步功率波動的機架級設計（廠商宣稱）。電網友善度可能成為新的採購規格。

這個約束對你是利多：電力工程拉長了供給端的時間常數，意味著「用軟體把既有 GPU 榨得更乾」——本書教的全部內容——的投資報酬率在 2026–2028 只會更高。

## 標準化與市場結構

**API 形狀已定，雙標準並存。** OpenAI ChatCompletions 是 de facto 標準，但注意一個 2026 的細節：DeepSeek-V4 同時相容 OpenAI 與 Anthropic API——Anthropic 的 Messages API 借 agentic 工具生態（MCP、Claude Code）成為第二事實標準。對你的含意：gateway 層（ch12）要會雙方言，但這層知識是薄的、可速學的。

**路由層全面換代。** Ingress NGINX 退役（2026-03）、Gateway API Inference Extension GA 並被 GKE/Envoy/NGINX/KServe/llm-d 共同採用（2026-06）——InferencePool + EPP 這套「身體感知路由」的 API 形狀大概率就是未來五年的標準件，值得深學（ch12 已講）。

**聚合層成為市場結構的一部分。** OpenRouter 2026-05 募資 $113M 定位企業推論路由；它的存在本身就是論據：當模型是可替換商品，路由與聚合就是價值節點。配合開放權重市場的換血——Meta 棄 Llama 轉閉源（Muse Spark，2026-04），中國系模型在 OpenRouter 佔比超過 45%（2026，前一年不到 2%）——模型層的競爭讓 serving 層的議價力上升。

**Neocloud 整併年。** S&P Global 統計 2025 年資料中心 M&A 超過 $690 億；2024–2025 簽的 GPU 合約 2026 下半陸續到期續約、資本市場收緊、第一代部署到壽——多方分析指向 2026 是 neocloud 淘汰賽之年。對讀者的實務含意（ch16 採購策略的延伸）：簽多年期合約前查對方的電力儲備與資產負債表，你的 GPU 供應商在 2026–2028 是有信用風險的。

**價格趨勢的正確讀法。** Epoch AI：固定能力水準的推論價格每年跌 9–900 倍。錯誤讀法是「推論會便宜到不重要」；正確讀法是 Jevons——單價崩跌餵養用量爆炸，**推論基礎設施的總盤子在漲，而利潤越來越取決於工程效率**（利用率、命中率、goodput——ch16 的排序）。效率工程師的稀缺度跟著總盤子走。

5. Rubin CPX 從發表到擱置的故事，給 infra 工程師的兩個教訓是什麼（一個關於架構方向、一個關於採購紀律）？
