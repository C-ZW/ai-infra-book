# Plan: text-transform plugin system

## 1. Core design constraint that drives everything

A decorator registry has one trap: `@register('upper')` only executes when its module is **imported**. If the core has to `import plugins.upper` to make that happen, adding a plugin means editing the core тАФ which the README forbids (line 10). So the central design decision is **how registration gets triggered without the core naming any plugin**.

Chosen approach: a `plugins/` package + **generic directory auto-discovery** (`pkgutil.iter_modules` + `importlib.import_module`). The core walks the package and imports every submodule it finds; it never references a plugin by name. Adding a plugin = dropping a new `.py` file into `plugins/`. No core edit. (Alternative for true third-party/out-of-tree plugins тАФ Python entry points тАФ covered in ┬з6.)

## 2. Proposed layout

```
text_transforms/
  __init__.py        # public API re-exports: register, run_pipeline, errors
  registry.py        # _REGISTRY dict, register(), run_pipeline(), error types
  discovery.py       # load_plugins(): scans the plugins package, idempotent
  plugins/
    __init__.py      # marks a package; NO plugin names listed here
    upper.py         # @register('upper')
    reverse.py       # @register('reverse')
tests/
  test_pipeline.py
pyproject.toml       # minimal: project name + pytest config (optional but recommended)
```

The core = `registry.py` + `discovery.py`. The `plugins/__init__.py` stays empty of plugin references тАФ that is what keeps it from becoming "the core you must edit."

## 3. Registry + decorator design

State: a module-level `_REGISTRY: dict[str, Callable[[str], str]]` in `registry.py`.

`register(name)` тАФ parametrized decorator:
- Validate `name` at the boundary: must be a non-empty `str` (reject `None`, empty, non-str) тЖТ raise `ValueError`/named error, not a silent skip.
- On duplicate name тЖТ raise `DuplicatePluginError` (fail loud; silent last-wins shadowing is a debugging nightmare). Exception: re-registering the *same* function object is a no-op, so re-running discovery is idempotent.
- Store `func`, then **return `func` unchanged** so each plugin remains a plain, directly-callable/testable function.

`run_pipeline(text, names)`:
- Ensure discovery has run (lazy trigger on first call, guarded тАФ see ┬з4).
- Validate `text` is `str`; validate `names` is a list/sequence of `str` (reject `None` / non-str entries with a clear message).
- Empty `names` тЖТ return `text` unchanged (identity is the defined, intentional behavior, not an accident).
- For each `name` in order: look up; if missing тЖТ `PluginNotFoundError` that lists available names and the failing index. Apply `func(text)`; **validate the result is a `str`** (a plugin returning `None`/bytes corrupts everything downstream); thread output into the next step. Wrap any exception the plugin throws in `PluginExecutionError(name, index)` with `raise ... from e` to preserve the traceback.
- Return the final string.

Error taxonomy (named types, never bare `Exception`):
```
PluginError(Exception)           # base, so callers can catch the whole family
тФЬтФА DuplicatePluginError
тФЬтФА PluginNotFoundError
тФЬтФА PluginLoadError               # a plugin module failed to import
тФФтФА PluginExecutionError          # a plugin raised at run time (carries name + position)
```

## 4. How new plugins stay decoupled (the crux)

`discovery.load_plugins()` in `discovery.py`:
- `import` the `plugins` package, then `pkgutil.iter_modules(plugins.__path__, plugins.__name__ + ".")`, **sorted** for deterministic order, and `importlib.import_module` each one. Importing runs the `@register(...)` side effects.
- **Idempotent + thread-safe**: guard with a module-level `_loaded` flag and a `threading.Lock` so concurrent first calls don't double-scan (the `register` same-func no-op is a second safety net).
- **Fail-fast on import error by default**: a plugin that can't import raises `PluginLoadError` (module name + chained original traceback) rather than vanishing silently. Offer an optional `strict=False` that logs-and-skips for operators who prefer degraded-but-running.
- The scan contains **zero plugin names** тАФ that is the decoupling guarantee. The function works identically whether `plugins/` has 2 files or 200.

Decoupling acceptance test (must be automated, ┬з5): add a brand-new plugin file *without touching `registry.py`/`discovery.py`* and prove `run_pipeline` picks it up.

## 5. Ordered implementation steps

1. **Scaffold** `text_transforms/` package + empty `plugins/__init__.py`; add minimal `pyproject.toml` with pytest config. (Optional: `git init`.)
2. **Errors** тАФ define the exception hierarchy in `registry.py` first; everything else references it.
3. **Registry** тАФ `_REGISTRY`, `register(name)` with name validation + duplicate guard.
4. **Discovery** тАФ `load_plugins()` with sorted scan, idempotency flag, lock, fail-fast import.
5. **Runner** тАФ `run_pipeline()` with input validation, lazy discovery trigger, ordered application, per-step output-type check, error wrapping, empty-list identity.
6. **Public API** тАФ `text_transforms/__init__.py` re-exports `register`, `run_pipeline`, and the error types.
7. **Example plugins** тАФ `plugins/upper.py` (`@register('upper')` тЖТ `s.upper()`), `plugins/reverse.py` (`@register('reverse')` тЖТ `s[::-1]`).
8. **Tests** тАФ see ┬з6 verification matrix.
9. **Run + lint** тАФ `pytest -q`; optionally `ruff`/`mypy` if the user wants them added.
10. **Report** тАФ summarize files + how to add a plugin; do not commit unless asked.

## 6. Verification

Run: `python -m pytest -q` from the repo root.

Test matrix (`tests/test_pipeline.py`):
- `run_pipeline("abc", ["upper"]) == "ABC"` тАФ single plugin.
- Ordered composition applies left-to-right (thread output through).
- **Order-sensitivity proof**: `upper`+`reverse` on `"abc"` give `"CBA"` in *both* orders, so they don't prove ordering. Use a dedicated asymmetric test plugin (e.g., one that appends `"!"`) to assert `["x","reverse"] != ["reverse","x"]`. Flagging this so the order test isn't a false positive.
- `run_pipeline("abc", []) == "abc"` тАФ empty pipeline identity.
- Unknown name тЖТ `PluginNotFoundError` (and message lists available names).
- Duplicate registration тЖТ `DuplicatePluginError`.
- Bad input (`text` not str; `names` containing non-str) тЖТ raises the boundary error.
- **Decoupling test**: write a temp module into the `plugins/` dir (or via a `tmp_path` package added to discovery), re-run `load_plugins()`, assert the new name resolves тАФ with assertion that `registry.py`/`discovery.py` were not modified. This directly tests README line 10.

Manual smoke: `python -c "from text_transforms import run_pipeline; print(run_pipeline('Hello', ['upper','reverse']))"` тЖТ `OLLEH`.

Tooling note: `ruff`/`mypy` are **not currently installed** тАФ I won't assume them; tests rely only on the confirmed pytest 9.0.3.

Scale-up alternative (note, not first implementation): for plugins shipped as separate installed packages, switch discovery to `importlib.metadata.entry_points(group="text_transforms.plugins")`. Same decoupling property, supports out-of-tree plugins, at the cost of requiring packaging/installation тАФ overkill for this in-repo task.

## 7. Risks / failure modes

| Risk | Impact | Mitigation |
|---|---|---|
| Registration never triggered (module not imported) | Plugin silently "missing" | Lazy `load_plugins()` on first `run_pipeline`; explicit, tested |
| Two plugins claim same name | Silent shadowing | `DuplicatePluginError` fail-loud; same-func re-register is no-op |
| Plugin import raises | Whole discovery dies, or plugin vanishes | Fail-fast `PluginLoadError` w/ chained traceback; opt-in skip mode |
| Plugin returns non-str / `None` | Downstream corruption | Per-step output-type validation |
| Re-running discovery | Double registration | Idempotency flag + lock + no-op re-register |
| Concurrent first call | Race in discovery | `threading.Lock` around load |
| **Unicode corruption** | `"silent wrong output"` | `str.upper()` changes length (`'├Я'тЖТ'SS'`); `s[::-1]` breaks grapheme clusters / combining marks / emoji ZWJ. Document as a known limitation of the *example* plugins; the core stays byte-faithful |
| Order assumption | Wrong result | Ordering is contractual + tested with an asymmetric plugin |
| Mutable global registry in tests | Cross-test bleed | Fixture to snapshot/restore `_REGISTRY` (and `_loaded`) |
