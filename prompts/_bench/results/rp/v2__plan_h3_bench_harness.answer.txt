# Design + Implementation Plan: Benchmark Harness

## 1. Goal & contract

Build a harness that takes a list of candidate functions and a list of shared inputs, calls each `candidate(input)` under a hard per-call timeout, records the outcome as one of `{ok, timeout, error, crashed}`, times only the successful calls, and prints a comparison table â€” and never lets a single misbehaving candidate hang or crash the harness itself.

Demonstrated against `candidates.py` (`fast_sum_to`, `slow_sum_to`, `TEST_INPUTS = [0, 1, 10, 100, -1]`).

## 2. What `candidates.py` actually forces (grounding the design)

The crux is in `slow_sum_to`:

```python
while i != n:   # i starts at 0 and only increments
```

For `n = -1` this is a **pure-CPU infinite loop** â€” no `sleep`, no I/O, no syscall, no checkpoint where the function voluntarily yields. This single fact eliminates the two "easy" timeout mechanisms (see Â§3) and drives the whole design. Expected per-input behavior the harness must produce:

| call | expected |
|---|---|
| `fast_sum_to(-1)` | `-1*0//2 = 0`, OK (fast path has no loop, doesn't hang) |
| `slow_sum_to(0/1/10/100)` | OK â†’ `0/1/55/5050` |
| `slow_sum_to(-1)` | **must be killed by timeout**, recorded as `timeout`, harness keeps going |

So the demo's correctness hinges entirely on the timeout mechanism genuinely *terminating* a runaway loop, not just abandoning it.

## 3. Isolation mechanism â€” the key decision

Three candidate mechanisms; the infinite pure-CPU loop is the discriminator:

| Mechanism | Kills `slow_sum_to(-1)`? | Failure mode | Verdict |
|---|---|---|---|
| `threading.Thread` + `join(timeout)` | **No** â€” Python threads can't be force-killed; the loop runs forever as a leaked (daemon) thread, contending for the GIL and skewing every later measurement | Silent resource leak; timings degrade over the run | Reject |
| `signal.SIGALRM` + handler raising | Yes *here* (signals are checked between bytecodes in pure Python) | Main-thread-only, Unix-only, one timer at a time; cannot interrupt C-extension/blocking calls; fragile if a candidate installs its own handler | Reject as primary |
| **`multiprocessing.Process` per call + `join(timeout)` â†’ `terminate()` â†’ `kill()`** | **Yes, reliably** â€” SIGTERM/SIGKILL reclaim the core regardless of what the child is doing | Pickling constraints + spawn overhead (mitigated, see Â§6/Â§10) | **Choose** |

**Decision: one fresh child process per call.** It's the only option that truly kills a runaway loop *and* isolates hard crashes (segfault / `os._exit` / OOM) so they can't take down the harness â€” exactly the "capture failures without crashing" requirement. Cost is process-spawn + IPC overhead, which we keep out of the reported timings by measuring inside the child.

Note: a `ProcessPoolExecutor` + `future.result(timeout)` is explicitly **not** used â€” on timeout the future raises but the worker keeps running and the pool wedges. A disposable process we own lets us `terminate()` it.

## 4. Harness architecture

Single new file, e.g. `benchmark.py`, importing nothing from `candidates.py` except via the public lists (`CANDIDATES`, `TEST_INPUTS`). Components:

- **`run_one(func, arg, timeout) -> Result`** â€” spawns a child, runs the call, enforces timeout, returns a structured `Result`. Pure, no printing.
- **child worker `_worker(func, arg, conn)`** â€” runs in the child: times the call with `perf_counter`, wraps in try/except, sends back a result payload over a `Pipe`. Never raises into the parent.
- **`benchmark(candidates, inputs, timeout, repeats=1) -> list[Result]`** â€” nested loop over `(candidate, input)`, collects results, guarantees cleanup.
- **`format_table(results) -> str`** â€” renders the comparison table from results only (no recomputation).
- **`main()`** under `if __name__ == "__main__":` â€” wires `candidates.py` lists in, picks a timeout, prints the table. The guard is mandatory (see Â§6, spawn).

## 5. Data model

One explicit result record per call â€” no tuples-by-position, no `None`-means-many-things:

```
Result:
  candidate: str        # func.__name__
  arg: repr(arg)
  status: "ok" | "timeout" | "error" | "crashed"
  elapsed_s: float | None   # set only when status == ok
  value_repr: str | None    # repr of return value, ok only
  error_type: str | None    # exception class name, error only
  error_msg: str | None     # short message, error only
  exitcode: int | None      # for crashed (segfault â†’ negative signal)
```

`status` is a closed set decided on purpose; the table renderer switches on it exhaustively.

## 6. Timeout + exception isolation mechanics (precise)

**Per call:**
1. Create a `multiprocessing.Pipe()` (or `Queue`). Start `Process(target=_worker, args=(func, arg, child_conn))`.
2. Parent `proc.join(timeout)`.
3. **If `proc.is_alive()` after join â†’ timeout path:** `proc.terminate()` (SIGTERM); `proc.join(grace=~0.5s)`; if still alive, `proc.kill()` (SIGKILL); final `join()` to reap (no zombies). Record `status=timeout`. This is what catches `slow_sum_to(-1)`.
4. **Else child finished â†’ drain the pipe:**
   - payload present and `status=ok` â†’ record value + elapsed (measured in child).
   - payload present and `status=error` â†’ record `error_type/error_msg` (candidate raised; captured, not propagated).
   - **no payload but exitcode â‰  0 â†’ `status=crashed`** (segfault, `os._exit`, OOM-kill, pickling-result failure). Record `exitcode`. This is the path that keeps a C-level crash from taking down the harness.
5. Always close pipe ends; the whole step sits in `try/finally` that force-kills any still-alive child so a harness exception can't orphan a process.

**Inside the child (`_worker`):** time `t0 = perf_counter()`, call `func(arg)`, `t1 = perf_counter()`; on success send `{ok, value, t1-t0}`; on `except BaseException as e` send `{error, type(e).__name__, str(e)}`. Wrap the *send* in its own try/except so an unpicklable return value degrades to a `crashed`/`error` record instead of a silent hang. Timing covers only the call â€” spawn, import, and IPC are excluded, keeping `fast_sum_to`'s microsecond timings honest.

**Spawn safety (macOS/Windows default = `spawn`):** the child re-imports the runner module, so `main()` **must** be under `if __name__ == "__main__":`; otherwise spawn recursively forks the harness. Candidates and inputs must be top-level importable/picklable â€” `fast_sum_to`, `slow_sum_to`, and the int inputs all satisfy this; document the constraint for future candidates (no lambdas/closures). Set the start method explicitly (`get_context("spawn")`) so behavior is identical across macOS/Linux rather than depending on the platform default.

## 10. Risks & mitigations

- **Spawn recursion** if `__main__` guard is missing â†’ harness fork-bomb. *Mitigation:* guard + explicit `spawn` context; called out in Â§6 as non-negotiable.
- **Non-picklable candidate/arg/return** (lambdas, closures, open handles) â†’ child can't start or can't reply. *Mitigation:* document top-level-only constraint; catch send failures in child â†’ degrade to `error`/`crashed`, never hang.
- **`terminate()` insufficient** (child ignores SIGTERM / uninterruptible) â†’ escalate to `kill()` after grace, then `join()` to reap; final `finally` sweep kills survivors.
- **Spawn overhead dwarfs fast functions** (ms-scale spawn vs Âµs-scale `fast_sum_to`) â†’ measure *inside* the child so reported time is call-only; note in output that cross-process overhead is excluded. Optionally `repeats`/best-of-N for stability.
- **Orphaned processes if the harness itself throws** mid-loop â†’ all spawning wrapped in `try/finally` cleanup.
- **Timer skew under load** (busy machine inflates timings, or delays the join) â†’ timeout is a generous bound (â‰¥1 s) well above real call times; report is comparative, not absolute SLA.
- **Platform variance** (SIGALRM-style assumptions, fork-vs-spawn differences) â†’ process approach + pinned `spawn` context removes main-thread/Unix-only constraints.
- **Silent misclassification** (a slow-but-correct call near the timeout boundary flips to `timeout`) â†’ keep timeout comfortably above observed `slow_sum_to(100)`; surface the chosen timeout in the table header so a near-boundary result is interpretable.
