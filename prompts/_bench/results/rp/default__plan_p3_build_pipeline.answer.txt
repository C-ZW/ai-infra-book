# Implementation Plan: expense-report tool

## 0. Assumptions & key decisions (please confirm ‚Äî see ¬ß6)
- **Language: Python 3 (stdlib only).** No manifest or existing code in the repo, and the README is language-agnostic. Python gives us `csv` (robust parsing), `decimal`, and `unittest` with zero dependencies. *If you want another language, this plan maps 1:1.*
- **Money is `Decimal`, never `float`.** `12.50 + 8.00 + 15.75` and friends must total `36.25` exactly. Binary floats silently drift (e.g. classic `0.1+0.2`), and an expense report that's off by a cent is a bug report. `Decimal(str(amount))` is the non-negotiable boundary.
- **Layout:** one module `expense_report.py` (parser + aggregator + formatter + `main`) and one test file `test_expense_report.py`. Small tool ‚Üí one module is clearer than four files; the three functions stay independently testable.

## 1. Components & interfaces

A small typed record carries data between stages:

```
@dataclass(frozen=True)
class Expense:
    date: str          # kept as raw string; not needed for totals (see risks)
    category: str
    amount: Decimal
```

**1. Parser** ‚Äî `parse_expenses(path: str) -> list[Expense]`
- Opens the file with `newline=""`, `encoding="utf-8"`, parses via `csv.DictReader`.
- Validates the header is exactly `date,category,amount` (fail fast if not ‚Äî a shifted column means every amount is wrong).
- Per row: strip whitespace; reject wrong field count, empty category, empty/non-numeric amount.
- Converts amount with `Decimal(value)`, wrapping `InvalidOperation` in a **named, line-numbered error**.
- **Failure contract:** raises a custom `ExpenseParseError(line_no, reason)` ‚Äî never returns half-parsed/silent-skipped data by default (see ¬ß6 Q4).

**2. Aggregator** ‚Äî `aggregate(expenses: Iterable[Expense]) -> dict[str, Decimal]`
- Folds into `{category: total}`. Pure function, no I/O.
- Empty input ‚Üí empty dict (not an error).
- Category match is exact/case-sensitive (data is already lowercase; see ¬ß6 Q5).

**3. Formatter** ‚Äî `format_report(totals: Mapping[str, Decimal]) -> str`
- Sorts items by **total descending**, with a **secondary sort by category name ascending** so equal totals produce deterministic, repeatable output (no ties in this data, but transport/food-style ties are inevitable in real data ‚Äî a nondeterministic report is untestable).
- Renders aligned columns, amounts quantized to 2 dp (`.quantize(Decimal("0.01"))`).
- Empty totals ‚Üí a defined string (e.g. `"No expenses."`), not a blank/crash.
- Pure function: takes a mapping, returns a string (does not print) ‚Äî keeps it unit-testable.

**Wiring** ‚Äî `main(argv=None) -> int`
- Resolves CSV path: `argv[1]` if given, else default `"expenses.csv"`.
- `parse ‚Üí aggregate ‚Üí format`, prints the report, returns exit code `0`.
- Catches `ExpenseParseError`/`FileNotFoundError`, writes a clear message to `stderr`, returns non-zero. `if __name__ == "__main__": sys.exit(main())`.

**Expected output for the committed data** (the verification oracle):
```
rent           900.00
entertainment   40.00
food            36.25
transport        6.40
```
(rent 900.00 > entertainment 40.00 > food 36.25 > transport 6.40.)

## 2. Ordered steps
1. **Scaffold** `expense_report.py` with the `Expense` dataclass, `ExpenseParseError`, and the four function stubs/signatures above.
2. **Formatter first** (pure, no I/O ‚Äî easiest to pin behavior): implement sort + alignment + empty case.
3. **Aggregator**: implement the fold; confirm against hand-computed totals.
4. **Parser**: implement header check, row validation, `Decimal` conversion, error raising.
5. **`main()`**: wire the three together, add path arg + error handling + exit codes.
6. **Tests**: `test_expense_report.py` with one test per component (¬ß3), plus a tiny fixture.
7. **Run** the suite and the tool end-to-end against the real `expenses.csv`; diff against the expected output in ¬ß1.

## 3. How I'd verify each component
- **Parser test:** feed a 2‚Äì3 line in-memory/`tmp_path` CSV ‚Üí assert it returns the right count of `Expense` rows with `Decimal` amounts. Add one negative case: a non-numeric amount raises `ExpenseParseError` with the offending line number.
- **Aggregator test:** hand-built `[Expense]` list with a repeated category ‚Üí assert exact `Decimal` totals (e.g. food `Decimal("36.25")`). Include the empty-list ‚Üí `{}` case.
- **Formatter test:** pass `{"rent": Decimal("900.00"), "food": Decimal("36.25"), "transport": Decimal("6.40")}` ‚Üí assert the output lines are in descending order and rent precedes food precedes transport. Add a **tie case** (two equal totals) ‚Üí assert alphabetical secondary order.
- **End-to-end (manual):** `python expense_report.py expenses.csv` ‚Üí byte-compare to ¬ß1's expected block. Also run with a missing file ‚Üí assert non-zero exit and a readable error.
- **Suite:** `python -m unittest -v` is green.

## 4. Failure modes explicitly handled
Missing file; empty file (header only ‚Üí empty report, not a crash); blank/whitespace rows; wrong column count; non-numeric/empty amount; unexpected header order; quoted fields or embedded commas (handled for free by `csv`, not hand-splitting); equal totals (deterministic tie-break); `float` drift (avoided via `Decimal`).

## 5. Out of scope (unless requested)
Currency symbols/locale formatting, multi-file/glob input, date filtering, grand-total line, config flags, packaging/CLI framework.

## 6. Risks & clarifying questions
1. **Language** ‚Äî Plan assumes Python 3 stdlib. Confirm, or name your target (Node, Go, etc.).
2. **Report format** ‚Äî I assumed plain aligned text, 2-dp, no currency symbol, no header row. Do you want a header (`CATEGORY  TOTAL`) and/or a **grand-total line**? *Recommend: add a grand total; it's the first thing a reviewer eyeballs.*
3. **Numeric type** ‚Äî Confirm `Decimal` (recommended) over `float`. Output precision = 2 dp?
4. **Malformed rows: fail-fast vs skip-and-warn** ‚Äî I default to **fail-fast** (reject the whole run on a bad row) to protect data integrity. If this feeds a flaky upstream, you may prefer skip-with-stderr-warning. Which?
5. **Category normalization** ‚Äî Data is already lowercase. Should `Food` and `food` merge (case-insensitive) or stay distinct? *Recommend: distinct for now; revisit if real data is messy.*
6. **Negative amounts / refunds** ‚Äî None in the sample. Allow them (summed normally) or reject as invalid? *Recommend: allow.*
7. **`date` column** ‚Äî Unused for totals. Keep it parsed-but-ignored (as planned) or validate it as a real date? *Recommend: keep raw; don't add validation nobody consumes.*
