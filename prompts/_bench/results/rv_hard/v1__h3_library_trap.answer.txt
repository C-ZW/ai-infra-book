Done. `fetcher.py:12` now retries on transient network errors with exponential backoff.

What changed:
- 3 attempts total; on failure, sleeps `1s` then `2s` (doubling) before re-trying.
- Retries only transient errors (`urllib.error.URLError`, `TimeoutError`); after the 3rd failure it logs and re-raises the original exception rather than swallowing it.
- Stdlib only — honored the project's no-third-party-deps rule (README + requirements.txt), so I used `time.sleep` instead of `tenacity`/`backoff`.

Deliberate behavior choices worth flagging:
- `HTTPError` is a subclass of `URLError`, so a 4xx like 404 is currently retried too — wasteful but harmless. If you want to skip non-retryable 4xx responses, I can special-case `HTTPError` with `400 <= code < 500`.
- Backoff is fixed (no jitter); fine for a single caller, but if many clients hit the same failing host they'll retry in lockstep.

The most dangerous unhandled case: the 5s `timeout` is per-attempt, so a hanging host can still cost up to `5+1+5+2+5 = 18s` wall-clock before failing — does the caller have its own deadline, or should `fetch_url` enforce a total time budget across all retries?
