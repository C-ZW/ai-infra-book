# ch10 — 旋轉的位置：RoPE 與相對位置之美

> **本章解決什麼問題**：ch09 揭穿了一個洞——self-attention 看不見順序（置換不變），所以位置資訊得**外加**進去；ch09 給的補法是正弦位置編碼，把一個固定的位置向量**加**到 embedding 上。本章換一個更漂亮、也是 2026 主流開源 LLM 事實標準的補法：RoPE（Rotary Position Embedding，旋轉位置嵌入）。它不加任何東西，而是把 query 與 key **依位置旋轉一個角度**。關鍵之美在於：兩個旋轉後向量做內積時，**絕對位置自動消去、只剩下相對位置**——這正是你在《圓的影子》裡反覆看過的 R(a)·R(b)=R(a+b)。本章動的還是三步骨架的第一步「打分」：不換打分公式（還是內積），而是在算內積**之前**先把 q、k 各自轉一個由位置決定的角度，讓相對位置自然滲進分數裡。它和 ch09 是同一個洞的兩種補法，你會看清為什麼「旋轉」這條路後來贏了。

## 從你已知的出發

你做過分散式系統，一定處理過「時間」這個麻煩東西。最天真的做法是給每個事件蓋一個**絕對時間戳**（wall-clock timestamp）：`2026-06-13T19:32:01Z`。但絕對時間戳有個老問題——兩台機器的時鐘不可能完全同步，你真正在乎的往往不是「這個事件發生在幾點幾分」，而是「這個事件**比另一個事件早還是晚、差多少**」。於是你學會用相對的東西來思考：Lamport 邏輯時鐘記的是「誰在誰之前」，向量時鐘記的是「相對的因果偏序」，TCP 序號在乎的是「這個封包跟上一個差幾號」。**絕對的位置常常是雜訊，相對的位置才是訊號。**

ch09 的正弦位置編碼，本質是給每個 token 蓋一個「絕對位置戳記」：第 0 個位置一個向量、第 1 個位置一個向量，加到 embedding 上。它確實能用（原始 Transformer 就靠它），而且設計者刻意挑了正弦函數，讓「相對位置可以用線性關係表達」——但相對位置在那裡是**間接**浮現的，要靠模型自己從絕對戳記裡學出來。

RoPE 問了一個更乾脆的問題：既然我們真正在乎的是**相對**位置，能不能讓相對位置**直接、無條件地**出現在注意力分數裡，連學都不用學？答案是能，而且用的工具你已經很熟——旋轉。

還有一個你會立刻有共鳴的工程角度：**相位（phase）**。如果你碰過訊號處理，或哪怕只是調過音訊的左右聲道延遲，你就知道「相位差」能編碼「時間差」——同一個正弦波，相位差多少就代表延遲了多少。RoPE 做的事，用一句話講就是「**把位置編碼成相位、把相對位置編碼成相位差**」。內積這個運算，剛好只看得到相位差。

在三步骨架裡，這一步坐在哪？

```text
            ┌──────────┐   ┌────────────┐   ┌──────────────┐
 query ──►  │ 1. 打分  │─► │ 2. 正規化  │─► │ 3. 加權混合  │ ──► 輸出
 keys  ──►  │  score   │   │  softmax   │   │ weighted sum │
 values ─────────────────────────────────────────┘
              │
              └─►【RoPE 在這裡】算內積之前，先把 q、k 各自依位置旋轉一個角度
```

RoPE 動的是第一步「打分」的**輸入**：不換內積這個打分公式，而是在 q·k 之前，把 q 轉 nθ、把 k 轉 mθ（n、m 是它們各自的位置）。注意它**完全不碰** value、不碰 softmax、不碰加權混合——位置資訊只透過「分數」這個閘門進場。記住這點，本章的所有故障模式都從這裡長出來。

## 先把問題收窄成 2 維：旋轉怎麼編碼位置

RoPE 的全貌是高維的，但它的靈魂在 2 維就講完了——因為 RoPE 把高維向量**切成一堆 2 維平面**，每個平面獨立做同一件事。所以我們先把整件事壓進一個平面，看清楚，再放大。

回憶《圓的影子》ch05：把一個 2 維向量「逆時針轉 α」這個動作，可以寫成一個 2×2 矩陣：

```text
R(α) = ┌ cos α   −sin α ┐
       └ sin α    cos α ┘

      R(α)·v = 把向量 v 逆時針轉 α 度
```

RoPE 的規則只有一句話：**位置在 p 的 token，它的 query（或 key）向量，先被旋轉 p·θ 這個角度，再拿去算內積。** 這裡 θ 是一個固定的「每走一步轉多少」的角速度。位置 0 不轉（轉 0 度），位置 1 轉 θ，位置 2 轉 2θ……位置 p 轉 pθ。位置愈靠後，轉得愈多。

```text
位置 0:  q 不轉           （角度 0）
位置 1:  q 轉 θ            （角度 1·θ）
位置 2:  q 轉 2θ           （角度 2·θ）
   ⋮
位置 p:  q 轉 pθ           （角度 p·θ）
```

就這樣。沒有要加的位置向量、沒有查表、沒有額外參數——只有「依位置轉一個角」。位置資訊**不是被加進向量**，而是被**烤進向量的方向**裡。

你可能會皺眉：把每個 token 各自轉個角度，怎麼就編碼了「順序」？關鍵不在單一向量轉了多少，而在**兩個向量做內積時會發生什麼**。這就是下一節的全部重點，也是 RoPE 之所以漂亮的地方。

## 之美：內積只剩相對位置

現在來算給你看。設 query 在位置 n、key 在位置 m。RoPE 把 q 轉 nθ、把 k 轉 mθ，然後算它們的內積。寫成矩陣：

```text
旋轉後的 q  =  R(nθ)·q
旋轉後的 k  =  R(mθ)·k

它們的內積 =  (R(nθ)·q)ᵀ · (R(mθ)·k)
           =  qᵀ · R(nθ)ᵀ · R(mθ) · k       ← 轉置把第一個矩陣翻過去
```

到這裡，《圓的影子》的兩個結論直接收掉它。第一，旋轉矩陣的**轉置等於反向旋轉**：`R(nθ)ᵀ = R(−nθ)`（轉 −nθ 就是把轉 nθ 退回去）。第二，旋轉會**結合相加**：`R(a)·R(b) = R(a+b)`（《圓的影子》ch05 的招牌結論，先轉 b 再轉 a 等於一次轉 a+b）。把兩條接起來：

```text
R(nθ)ᵀ · R(mθ) = R(−nθ) · R(mθ) = R(mθ − nθ) = R((m−n)θ)
```

於是整個內積塌縮成：

```text
(R(nθ)·q)ᵀ · (R(mθ)·k)  =  qᵀ · R((m−n)θ) · k
```

**停在這裡看十分鐘。** 左邊的式子裡，n 和 m 是兩個 token 的**絕對位置**——分別是第幾個。右邊呢？n 和 m 不見了，只剩下 **(m−n)**——它們的**相對距離**。換句話說：

> 兩個經 RoPE 旋轉後的向量做內積，結果**只依賴它們相隔多遠（m−n）**，跟它們各自站在序列的哪個絕對位置**完全無關**。

把 animal 和 it 一起往後挪十個位置，它們之間的注意力分數**一個字都不會變**，因為相對距離沒變。這正是你在分散式系統裡夢寐以求的性質——絕對時鐘漂移無所謂，只要相對順序對。RoPE 把「相對位置才是訊號」這件事，從「希望模型學得到」升級成「數學上保證如此」。

這就是本章的驚嘆點，請你能用自己的話講出來：**正弦編碼是把位置加進向量、希望模型從絕對戳記裡推出相對關係；RoPE 是把位置烤成旋轉角度，讓相對關係在內積那一步自動、無條件地浮現——連學都不必學。** 三步骨架的「打分」步沒換公式（還是內積），但因為打分的輸入被旋轉過，相對位置就免費掉進了分數裡。

### 接回複數：同一件事的另一個鏡頭

《圓的影子》ch07–08 教過：2 維平面上的旋轉，就是複數乘 e^{iα}。把 q、k 的那一對座標看成複數，RoPE 就是「把 q 乘 e^{inθ}、把 k 乘 e^{imθ}」。內積對應到「一個取共軛再相乘取實部」，算出來：

```text
(q·e^{inθ}) 與 (k·e^{imθ}) 的內積（取實部）
   ∝  實部[ q·k̄ · e^{i(m−n)θ} ]      ← 指數律 e^{a}/e^{b}=e^{a−b}，nθ 被 mθ 一減就剩 (m−n)θ
```

同一句話換複數講：**位置編成相位、相對位置就是相位差**，而內積只讀得到相位差。這是全書最漂亮的跨書接點之一——你在三角函數那本花了三章建立的「旋轉會相加」的肌肉記憶，在這裡一次回收，撐起了 2026 年幾乎每個開源大模型的位置編碼。需要重溫旋轉、複數、e^{iθ} 的，回《圓的影子》ch05、ch07–08；本書只用結論，不重推。

## 為什麼「相對」是對的歸納偏置（inductive bias）

數學漂亮是一回事，但你應該追問：把「只看相對位置」硬寫進架構，憑什麼是好的設計，而不是隨手挑的偏好？這牽涉到一個機器學習的關鍵概念，補一句鷹架：**歸納偏置（inductive bias）**——你在模型結構裡預先植入的假設，等於先替模型省下一批它本來得自己從資料學的東西。植入對的假設，模型學得又快又穩；植入錯的假設，模型得花力氣去抵消你的偏見。

「語言的意義主要由相對位置決定」這個假設，對不對？拿脊椎句子檢查：*"the animal didn't cross the street because it was too tired."* 把這整句話塞進一篇長文章的開頭、中段、還是結尾——it 指向 animal 的關係**一點都不會變**，因為那只取決於 it 和 animal 在句內**相隔多遠**，跟整句話落在文章第幾個字無關。再想你熟的場景：一段程式碼裡 `})` 該配哪個 `({`，看的是巢狀的相對深度，不是這段碼在檔案第幾行；一句 SQL 裡 `WHERE` 綁哪個 `FROM`，看的是子句的相對結構。**自然語言（和程式語言）的句法、指代關係，絕大多數是相對的、平移不變（shift-invariant）的。**

所以「相對位置不變」不是一個任意偏好，而是一個**和資料的真實結構對齊**的歸納偏置。正弦編碼把這件事留給模型「希望它學得到」；RoPE 直接把它焊進內積——模型開機就知道「平移整句不該改變注意力」，不必浪費容量去學這條規律。這跟你設計 schema 時的取捨是同一個道理：與其讓每個 query 自己去算「事件 A 是不是在事件 B 之前」，不如直接存一個相對偏序，把這條規律變成結構保證。**對的歸納偏置 = 把資料的對稱性寫進架構。** RoPE 把語言的「平移對稱」寫進了打分這一步。

這也解釋了一個你可能納悶的事：RoPE 零額外參數（不像可學習絕對位置嵌入，每個位置一個訓練向量），照理說「能力」更弱，怎麼反而贏？因為它不是靠參數量取勝，是靠**把對的假設免費送給模型**——省下來的容量拿去學真正需要學的東西。少即是多，前提是你少對了地方。

## Worked example：手算 RoPE 的相對位置

抽象式子看一遍不夠，動手算一次才會信。我們用 2 維、一個固定角速度 θ = 60°（cos 60° = 0.5、sin 60° = √3/2 ≈ 0.866，這兩個值你不用查表）。這就是脊椎——序章那個句子 *"the animal didn't cross the street because **it** was too tired."* 裡的 it 與 animal，現在各自帶上位置。

為了把「只剩相對位置」這件事看得最乾淨，我們先用最簡單的向量：旋轉**之前**，it 的 query 是 q=(1, 0)、animal 的 key 是 k=(1, 0)。它們指向同一個方向，原始內積 q·k = 1×1 + 0×0 = **1**（完全對齊，「最相似」）。現在給它們蓋上位置。

**情境 A**：animal 在位置 m=1、it 在位置 n=3（it 在句子裡比 animal 晚出現，相對距離 m−n = −2）。

```text
旋轉 it 的 q（轉 n·θ = 3×60° = 180°）：
   R(180°)·(1,0) = (cos180°, sin180°) = (−1, 0)

旋轉 animal 的 k（轉 m·θ = 1×60° = 60°）：
   R(60°)·(1,0) = (cos60°, sin60°) = (0.5, 0.866)

旋轉後內積 = (−1)(0.5) + (0)(0.866) = −0.5
```

**情境 B**：把兩個 token 整體往後挪 3 格——animal 在位置 m=4、it 在位置 n=6（相對距離還是 m−n = −2）。

```text
旋轉 it 的 q（轉 6×60° = 360°）：
   R(360°)·(1,0) = (cos360°, sin360°) = (1, 0)

旋轉 animal 的 k（轉 4×60° = 240°）：
   R(240°)·(1,0) = (cos240°, sin240°) = (−0.5, −0.866)

旋轉後內積 = (1)(−0.5) + (0)(−0.866) = −0.5
```

兩個情境的絕對位置完全不同（1 與 3，對上 4 與 6），但**內積都是 −0.5**。為什麼？因為相對距離都是 −2。用上一節的閉式核對：內積應該等於 `qᵀ·R((m−n)θ)·k = (1,0)·R(−120°)·(1,0) = cos(−120°) = −0.5`。對得上。

把這件事攤成一張表，看相對距離怎麼一手決定分數（q=k=(1,0)，所以內積 = cos(相對距離 × 60°)）：

```text
相對距離 |m−n|:   0      1      2      3      4
旋轉後內積:      +1.00  +0.50  −0.50  −1.00  −0.50
                  ▲                            
            完全對齊                            
            （同位置）  ← 隨距離拉開，內積開始振盪、衰減 →
```

注意一件事：**單一頻率**（單一個 θ）下，內積是 cos((m−n)θ)，它會**週期性振盪**——距離拉到讓角度繞滿一圈，分數又回到高點，這不是我們要的（我們要的是「離得愈遠、相關性傾向愈低」的單調感）。真正的 RoPE 不只用一個 θ，而是用**很多個不同的 θ**（下一節），把不同頻率的 cos 疊起來，淨效果才是「隨相對距離平滑衰減」（RoFormer 原文叫它 long-term decay）。你在《圓的影子》ch12 看過的「不同頻率疊加」，在這裡又一次決定了形狀。

> **可選的 5 行 numpy（讓骨架一目了然，數字與上面手算一致）**

```python
import numpy as np

def rope_2d(v, pos, theta):           # rotate a 2D vector by pos*theta
    a = pos * theta
    R = np.array([[np.cos(a), -np.sin(a)],
                  [np.sin(a),  np.cos(a)]])
    return R @ v

th = np.pi / 3                          # 60 degrees
q, k = np.array([1.0, 0.0]), np.array([1.0, 0.0])
for (m, n) in [(1, 3), (4, 6)]:         # (key pos, query pos), both rel = -2
    print(m, n, round(rope_2d(q, n, th) @ rope_2d(k, m, th), 4))
# 1 3 -0.5   /   4 6 -0.5   ← same score, same relative distance
```

## 從 2 維放大到 d 維：一堆獨立的旋轉平面

真實模型的 q、k 是 d 維（脊椎 d=4、原始 Transformer d_k=64）。RoPE 怎麼處理高維？很簡單粗暴：**把 d 維座標兩兩配對成 d/2 個 2 維平面，每個平面用上面那套旋轉，但每個平面配一個不同的角速度 θᵢ。**

```text
d 維向量  (x₀, x₁, x₂, x₃, …, x_{d−2}, x_{d−1})
          └─平面0─┘ └─平面1─┘        └─平面 d/2−1─┘
平面 0 用角速度 θ₀，平面 1 用 θ₁，…，每個平面在位置 p 各轉 p·θᵢ
```

角速度怎麼選？RoFormer 沿用正弦編碼那套幾何級數的頻率（這也是它和 ch09 一脈相承的地方）：

```text
θᵢ = base^(−2i/d)，  i = 0, 1, …, d/2−1，  base 通常取 10000
```

第 0 個平面 θ₀ = 1（最快、高頻，轉得最猛，捕捉**近距離**的位置差異）；最後一個平面 θ 趨近 base^{−1}（最慢、低頻，幾乎不轉，捕捉**遠距離**的粗略位置）。高頻平面管「差一兩個字」的精細，低頻平面管「差幾百個字」的大局——多頻率疊加，淨效果就是上一節說的「隨距離平滑衰減」。

把脊椎的 d=4 代進去算給你看（base=10000，d/2=2 個平面）：

```text
平面 i=0:  θ₀ = 10000^(0)    = 1       → 每走一步轉 1 弧度，約 6.3 步轉一圈（高頻）
平面 i=1:  θ₁ = 10000^(−1)   = 0.01    → 每走一步轉 0.01 弧度，約 628 步才轉一圈（低頻）
```

兩個平面，一快一慢：快的那個分辨「差 1 步還是差 2 步」這種貼身的位置差；慢的那個幾乎不動，要走幾百步它才轉得明顯，負責「差很遠」的粗粒度。原始 Transformer 的 d_k=64 就有 32 個平面，θ 從 1 一路幾何級數降到約 0.000133（最慢的平面要四萬多步才轉一圈）——等於布了一把從「貼身」到「全景」連續鋪滿的尺，每個相對距離都有某幾個平面對它最敏感。這正是「多頻率疊加避免別名」在做的事：不同距離由不同頻率的平面負責，疊起來既不週期性回繞、又隨距離平滑衰減。你在《圓的影子》ch13 看過的「不同頻率的正弦疊出任意形狀」（Fourier 的門口），在這裡又一次當了主角。

最關鍵的是：**因為每個平面都是旋轉，上一節「內積只剩相對位置」的結論對每個平面各自成立、加總後依然成立。** d 維 RoPE 後的內積 = 各平面內積之和 = `Σᵢ (那一對座標的 q·k) × cos((m−n)θᵢ)`，每一項都只看 (m−n)。所以放大到任意維度，「只依賴相對位置」這個性質**原封不動**。這是 RoPE 設計上的乾淨：高維只是把同一個 2 維魔術並排做了 d/2 次。

## RoPE vs 正弦 vs ALiBi：加在哪、動了什麼

把位置補進注意力，2017 到 2026 主要有三種思路。它們補的是同一個洞（ch09 的置換不變），但「動三步骨架的哪裡」不同——這張對照表是本章你最該帶走的東西：

```text
方法          動在哪                位置怎麼進場           相對位置是…
─────────────────────────────────────────────────────────────────────
正弦編碼       embedding（打分之前）  把位置向量【加】到      間接：靠正弦的
（ch09）                            token embedding 上     線性關係，模型自己學
─────────────────────────────────────────────────────────────────────
RoPE          q、k（打分之前）       把 q、k 各自【旋轉】    直接：內積後絕對
（本章）                            一個由位置決定的角      位置消去，只剩 m−n
─────────────────────────────────────────────────────────────────────
ALiBi         分數（打分之後、        在 q·k 分數上【加】一   直接：偏置正比於
（Press 2021） softmax 之前）         個正比於距離的負偏置     距離 |m−n|
─────────────────────────────────────────────────────────────────────
```

三者最大的差別在「位置進場的時機與形式」：

- **正弦編碼加在 value 也會帶到的 embedding 上**——位置混進了「內容」，順著殘差流一路傳下去。
- **RoPE 只旋轉 q 和 k，不碰 value**——位置只透過「分數」這個閘門影響注意力，不污染被搬運的內容（value）。這是 RoPE 比正弦乾淨的一點：位置該影響「我看哪裡」（分數），不該直接改「我搬什麼」（value）。
- **ALiBi 連向量都不動**，直接在算好的分數 q·k 上加一個 `−slope × |m−n|` 的線性懲罰（每顆頭一個固定的 slope，head-specific，訓練前定好、不學），距離愈遠扣愈多。它最便宜、外推性好，但表達力較死（只能線性遞減，無法表達「特別關注某個距離」）。

ALiBi 在三步骨架裡其實也是動「打分」——它在打分**之後**、softmax 之前加偏置，可以看成 RoPE 之外的另一條相對位置路線。`（2026-06）`的事實是：**RoPE 成了主流開源 LLM 的事實標準**——Llama 2/3、Qwen、Mistral、Gemma、DeepSeek 等普遍採用 RoPE 或其變體，ALiBi 的採用率遠低（照 landscape）。RoPE 贏在幾個工程上的甜蜜點：相對、零額外參數、不查表、和 FlashAttention 相容（旋轉是逐位置的廉價運算，不破壞分塊），而且配上後面要講的外推手法可以撐到很長的 context。這不是說 RoPE 數學上「最優」，而是它在「漂亮 × 便宜 × 能外推」三軸上同時夠好。

## 故障模式與直覺陷阱

RoPE 的優雅容易讓人以為它是位置問題的終結。它不是。下面是真實會壞的地方。

### 招牌失效：外推（extrapolation）失效——RoPE 看不懂沒見過的長度

這是本章最重要的故障，也是 2023 之後一整條研究線（PI、YaRN）要救的痛。

RoPE 雖然只看相對位置，但模型訓練時只見過**訓練長度以內**的相對距離。一旦推論時的序列超過訓練長度，就會出現它從沒見過的、極大的相對距離——那些 cos((m−n)θᵢ) 落在訓練時從未取樣到的角度區間。後果是注意力分數**失真、不穩定**：分數分布變得異常（熵爆高、變得平而模糊，或反過來在錯誤位置異常尖銳），模型「看不懂」這些超長距離，效果斷崖式下滑（Liu 等人觀察到的「災難性崩壞」，2026-06 查證）。直覺版：**RoPE 保證了相對位置的形式，但沒保證模型在沒見過的角度上還能正常運作**——形式對了，不代表外插能力憑空出現。

> **怎麼自我察覺**：如果你把一個 4k 訓練的模型直接餵 32k，發現 perplexity 突然爆掉、或長文後半段開始胡言亂語，先懷疑位置外推失效，而不是模型「不夠聰明」。

救法（一瞥，數學細節指向延伸閱讀）：

| 手法 | 想法（一句話） | 出處（2026-06 查證） |
|---|---|---|
| Position Interpolation（PI） | 把超長的位置索引**線性壓縮**回訓練範圍內（位置 8000 當成 1000 用），讓角度落回見過的區間 | Chen 2023 |
| YaRN | NTK-by-parts：高頻平面（管近距離）幾乎不壓、低頻平面（管遠距離）多壓，再加一個**注意力溫度**把分數整體縮小防止過尖 | Peng 2023（ICLR 2024） |

注意 YaRN 那個「注意力溫度」——它就是 ch05 的溫度旋鈕，回收一次：外推時分數會變得過尖／過散，調溫度把它拉回正常的銳利度。位置外推失效，本質上是 ch05–06「softmax 的尺度／銳利度」問題在長 context 下的重演。

### 陷阱二：以為「只看相對位置」等於「不在乎絕對位置」

RoPE 的內積只依賴相對位置——但這**不代表模型完全不知道絕對位置**。在因果遮罩（ch11）下，第 0 個 token 永遠只能看到自己，第 5 個 token 看得到前 6 個——能看到幾個 token 這件事本身就洩漏了絕對位置資訊。RoPE 拿掉的是「分數對絕對位置的直接依賴」，不是「模型對絕對位置一無所知」。把這兩件事混為一談，你會對「為什麼模型還是知道句首句尾」感到莫名其妙。

### 陷阱三：以為單一頻率就夠、忽略 base 這個旋鈕

如果你只用一個 θ，內積是 cos((m−n)θ)，它**週期性振盪**——距離拉到繞滿一圈，遠的 token 分數又跟近的一樣高，相對位置就**別名（aliasing）**了，模型分不清「差 2 格」和「差 14 格」。RoPE 靠多頻率疊加避開這個坑，而那個 base（通常 10000）控制頻率的鋪展範圍。`（2026-06）`長 context 模型常調大 base（如 100 萬）來拉長「不別名」的有效範圍——這也是長度外推的一條路。忘了 base 是個可調旋鈕，你會以為 RoPE 是個沒有超參的黑盒。

### 陷阱四：把 RoPE 的「衰減」當成硬性的距離截斷

RoPE 多頻率疊加帶來的 long-term decay 是**傾向**（遠的 token 平均而言分數較低），不是 ch15 那種**結構性的稀疏截斷**（直接不算遠的對）。RoPE 算的還是 full O(n²) 的所有對，遠的也照算，只是分數傾向小。把「衰減傾向」誤當「省了計算」，你就會以為 RoPE 解決了 O(n²)——它沒有，那是 Part IV 的事。

## 紙上推演

### 推演 1：用旋轉矩陣性質證明「RoPE 內積只依賴相對位置」 [15 分鐘] ★★

不查上文，自己從頭推一遍：query 在位置 n、key 在位置 m，RoPE 把它們各轉 nθ、mθ。寫出旋轉後內積 `(R(nθ)q)ᵀ(R(mθ)k)`，用「轉置＝反向旋轉」與「R(a)R(b)=R(a+b)」兩條，化簡到只剩 (m−n)。然後用一句話講清楚：這個結論為什麼讓「把整句往後挪」不改變任何注意力分數？

### 推演 2：手算兩個情境，驗證相對位置不變 [15 分鐘] ★★

用 θ=60°、旋轉前 q=k=(1,0)。算兩個情境的旋轉後內積：(a) key@2、query@5；(b) key@3、query@6。兩者相對距離都是 m−n = −3。各自把 q 轉 nθ、k 轉 mθ 算出座標再內積，確認兩者相等。再用閉式 `cos((m−n)θ) = cos(−180°)` 核對。

### 推演 3：RoPE 加在哪、和正弦差在哪 [10 分鐘] ★

口頭（或寫 200 字）回答三件事：(1) 正弦編碼加在哪個東西上、RoPE 旋轉哪兩個東西、ALiBi 加在哪一步？(2) 為什麼「RoPE 不碰 value」是個好性質？(3) 用《圓的影子》的一句話（提到 R(a)R(b)=R(a+b) 或 e^{iα}）解釋「為什麼旋轉是編碼位置的漂亮辦法」。

### 推演解答

**推演 1。** 旋轉後內積：

```text
(R(nθ)q)ᵀ (R(mθ)k)
  = qᵀ R(nθ)ᵀ R(mθ) k          ← (Av)ᵀ(Bw) = vᵀAᵀBw
  = qᵀ R(−nθ) R(mθ) k          ← 旋轉矩陣轉置 = 反向旋轉，R(α)ᵀ = R(−α)
  = qᵀ R(mθ − nθ) k            ← R(a)R(b) = R(a+b)，這裡 a=−nθ、b=mθ
  = qᵀ R((m−n)θ) k             ← 只剩相對位置 (m−n)
```

n 與 m 只透過差 (m−n) 出現，絕對位置消失。所以把 query 和 key 同時往後挪 k 格——n→n+k、m→m+k——差 (m−n) 不變，內積一個字都不變。直覺：旋轉同時轉 q 和 k，等於只看「兩者**之間**的夾角差」，跟它們各自轉到哪個絕對方向無關。這就是 RoPE 把「絕對位置是雜訊、相對位置是訊號」寫死進數學的方式。

常見錯路：忘了轉置會把旋轉**反向**（寫成 R(nθ)R(mθ)=R((m+n)θ)，得到 m+n 而非 m−n），那就不是相對位置了。R(nθ)ᵀ=R(−nθ) 那個負號是整個證明的關鍵。

**推演 2。** θ=60°，q=k=(1,0)。

情境 (a) key@m=2、query@n=5：

```text
query 轉 5×60°=300°：R(300°)(1,0) = (cos300°, sin300°) = (0.5, −0.866)
key   轉 2×60°=120°：R(120°)(1,0) = (cos120°, sin120°) = (−0.5, 0.866)
內積 = (0.5)(−0.5) + (−0.866)(0.866) = −0.25 − 0.75 = −1.0
```

情境 (b) key@m=3、query@n=6：

```text
query 轉 6×60°=360°：R(360°)(1,0) = (1, 0)
key   轉 3×60°=180°：R(180°)(1,0) = (−1, 0)
內積 = (1)(−1) + (0)(0) = −1.0
```

兩者都是 **−1.0**，相對距離都是 −3。閉式核對：`cos((m−n)·60°) = cos(−3×60°) = cos(−180°) = −1`。對得上。直覺收尾：相對距離 3（角度差 180°）讓兩個原本對齊的向量轉到完全相反方向，內積到最負——這就是「差這麼遠、相關性傾向最低」在單頻率下的長相。

**推演 3。** (1) 正弦編碼把位置向量**加**到 token embedding（內容）上；RoPE **旋轉** query 與 key（不碰 value）；ALiBi 在算好的 q·k **分數**上加一個正比於距離的負偏置。三者都動「打分」這一步的不同位置（前者改打分的輸入內容、RoPE 改打分輸入的方向、ALiBi 改打分的輸出）。(2) RoPE 不碰 value 是好事，因為位置該影響「該看哪裡（分數）」，不該直接竄改「被搬運的內容（value）」——把位置摻進 value 會讓位置雜訊順著殘差一路污染表示。(3) 因為旋轉會相加（R(a)R(b)=R(a+b)，或 e^{ia}·e^{ib}=e^{i(a+b)}），所以「q 轉 nθ、k 轉 mθ 再內積」會自動把角度一減、只剩相對的 (m−n)θ——相對位置不是硬塞的，是旋轉這個運算的代數天性免費給的。
