這本書要證明的事，可以濃縮成七句話。我把它們放在最前面，當作全書的驗收清單——讀完最後一章，你應該能對著另一個工程師，把每一句都用自己的話講清楚：

1. 注意力到底在算什麼（一句話：對所有位置做一次「依相關性加權的平均」——三步骨架）。
2. 為什麼要 query／key／value 三個角色、為什麼除以 √d、為什麼用 softmax。
3. multi-head 在分什麼工、位置編碼補了「集合沒有順序」這個洞。
4. 為什麼是 O(n²)、業界怎麼想「替這張沒有索引的表建索引」。
5. 注意力的理論身世（核迴歸／Hopfield／集合上的置換等變運算）——它不是憑空冒出來的。
6. 注意力權重能不能當「解釋」、機制可解釋性看到了什麼。
7. 2026 年「attention is all you need」還成立嗎。

這一章只負責一件事：讓你**想知道**這七題的答案。

## 從你已知的出發

你每天都在解「該看哪裡」這個問題，只是你不這麼叫它。

一個請求進到你的 load balancer，它得決定：這筆流量該送哪一台後端？一個查詢打到你的 router，它得決定：這條路徑該套哪一條規則？你的 cache 收到一個 key，它得決定：該回哪一筆 value，還是 miss 掉去打 DB？你寫的每一個 `if`、每一張 routing table、每一個 `WHERE` 子句，本質上都是在回答「**面對這筆輸入，我該看哪裡、取哪一筆**」。

這些決策有一個共同點：**規則是你手寫死的**。你決定了「user_id 落在這個區間就送 shard A」、「path 開頭是 `/api/v2` 就走新服務」、「這個 key 不在就 fallback 到 origin」。系統很聽話，因為每一條對應關係都是你親手鋪好的。資料一進來，它沿著你挖好的水道流。

現在把問題反過來。假設我不告訴你規則。我只給你一大堆「輸入→該看哪裡」的例子，要你的系統**自己長出**那張 routing table——而且這張表不是離散的 if-else，是一個對「每一筆輸入、每一個可能去處」都給出一個**連續權重**的東西：不是「送 A 或送 B」，而是「**0.7 看 A、0.2 看 B、0.1 看 C**，然後把三邊的結果按這個比例混起來」。

這就是注意力。它是一個**學出來的、軟性的、按相似度加權的查找**。你熟悉的 hash map 是「key 精確命中就取那一筆 value」——硬的、二元的、命中或不命中。注意力是它的連續版：query 對每一個 key 算一個相似度分數，把所有分數正規化成一組和為 1 的權重，再用這組權重去**混合所有的 value**。沒有「命中一筆」這回事，每一筆都按它的相關度貢獻一點。

我之所以從這裡開頭，是因為這座橋是本書最強的一座，後面會反覆走回來：

- **key-value 查找**（hash map／Redis／字典）——注意力就是它的軟性版本（ch04 把這座橋走到底）。
- **`WHERE`／`JOIN`／index**——注意力是「一張建不了索引的表，每筆新資料對全表算一次相似度」（ch07、ch14 接這個）。
- **append-only log／event sourcing**——生成文字時「只能讀已發生的、讀不到未來」，就是 causal mask（ch11）。
- **P99 與全表掃描翻車**——「對全表兩兩比對」資料一大就爆，這是 O(n²) 的代價（ch14）。

差別只有一個，但這個差別撐起了整個大型語言模型的時代：**那些查找規則，你以前手寫死，注意力讓模型自己學出來。**

## 一個代名詞，一道人類一眼就懂、機器卻要算的題

來看全書要反覆解的那個句子。它沿用了 Transformer 早期最有名的視覺化例子：

> *"the animal didn't cross the street because **it** was too tired."*
> （那隻動物沒有過街，因為**牠**太累了。）

讀到 **it** 的時候，你的腦子在零點幾秒內就完成了一件事：**it 指的是 animal，不是 street。**

你怎麼知道的？句子裡有兩個名詞可以當 it 的對象——animal 和 street，文法上兩個都通。但你立刻判定是 animal，理由是**世界知識**：會「累（tired）」的是動物，不是街道。注意這個判斷有多薄——只要把句尾一個字換掉：

> *"the animal didn't cross the street because **it** was too **wide**."*
> （……因為**它**太**寬**了。）

同一個句構、同一個 it，現在它指的是 **street** 了。因為「寬（wide）」是街的屬性，不是動物的。決定 it 看哪裡的，不是文法，而是「tired 還是 wide」這個離 it 好幾個字、語意上卻緊緊綁住的線索。（這種「換一個字、指代就翻面」的最小對照，是語言學裡 Winograd schema 那類常識消解題的經典設計；本書不深入語言學，只借它說明：要解對 it，模型非得「跨好幾個位置、去看對的那個字」不可。）

對人類，這叫常識，不費力。對一台只會做矩陣乘法的機器，這是一道貨真價實的難題：**它要能在處理 it 這個位置時，把目光投向句子裡正確的那幾個字**——多看 animal、少看 street，並且讓 tired 這個線索把天平壓向 animal。

關鍵的字是「**投向**」。本書要證明的就是：這個「該看哪裡」**不是被規定的，是被算出來的**。沒有人在模型裡寫一條「遇到 it 就回頭找最近的名詞」的規則。模型看到的只有一堆數字，它對「it 該看 animal」這件事的判斷，是 query 與 key 的內積、一次 softmax、一次加權混合——三步——的結果。

這就是脊椎句子。**全書每深入一層，就回來重算它一次**：ch04 算出 it 對三個位置的原始分數、ch05 用 softmax 把分數變成權重、ch06 看為什麼要先除以 √d、ch07 把它擺進完整的注意力矩陣、ch08 看多頭怎麼分工、ch09 發現不加位置就分不出順序、ch11 看遮罩怎麼讓它只能往左看、ch21 把這條 it→animal 的連結畫成一個具體的電路。到 ch23，我們會把這同一個 it，攤出八層讀法的總對帳表。

> 一個誠實的歸屬：這個 "the animal... because it was too tired" 的例子**不是出自 2017 年那篇原始論文**（Vaswani et al.「Attention Is All You Need」, arXiv 1706.03762, 2017-06）。它首見於 Google 2017 年 8 月的官方部落格〈Transformer: A Novel Neural Network Architecture for Language Understanding〉，之後經由 Jay Alammar 的〈The Illustrated Transformer〉這篇圖解廣為流傳（2026-06 查證）。原論文裡確實有代名詞消解的注意力視覺化，但這個被引用到爛的英文句子是部落格與圖解普及開來的。我特地點明這件事，是因為本書談「誰先做了什麼」時會很在意歸屬——這是後面 ch03 講「注意力到底是誰發明的」會反覆遇到的態度。

## 模型不讀字，模型讀向量（最小 ML 鷹架）

在往下走之前，得先補一塊地基，否則「it 看 animal」會懸在半空。這塊地基我在第一次需要時講清楚到夠用就好，之後各章直接拿來用，不再展開。

**第一件事：模型根本不認得 "animal" 這個字。** 它認得的是數字。所以送進模型之前，文字要先被切成一塊一塊的 token（大致是「字／詞／子詞」這個量級的單位，怎麼切是 tokenizer 的事，本書一句帶過，細節見《從後端到 AI Infra》ch03），然後每一個 token 被換成一個**向量**——一串實數，比如 512 個數字排成一排。這個「token → 向量」的對應表叫 **embedding（嵌入）**。

你可以把 embedding 想成一張巨大的查找表：每一個 token 在表裡都有一行，那一行就是它的向量。所以脊椎句子進到模型裡，其實長這樣：

```text
  token        embedding 向量（示意，真實是幾百維的實數）
  ─────        ──────────────────────────────────────
  animal   →   [ 0.21, -0.05,  0.88, ...,  0.13 ]
  street   →   [-0.40,  0.62,  0.07, ...,  0.55 ]
  it       →   [ 0.09,  0.31, -0.22, ...,  0.04 ]
```

向量不是隨便給的。embedding 的妙處在於：**語意相近的 token，向量也相近**（在向量空間裡靠得近——這裡的「近」就是內積／夾角的事，是相似度，深入見《矩陣是動詞》ch15）。所以 animal 跟 cat、dog 的向量會擠在一塊，跟 street 隔得遠些。這件事不是手工排的，是**學出來的**——接下來這就是第二塊地基。

**第二件事：「權重是學來的」是什麼意思。** 本書會反覆出現「W_Q／W_K／W_V 是學來的投影矩陣」這種話。先給一個夠用的直覺，ch04 真正用到時再接上：

模型裡有一大堆數字（embedding 表、各種矩陣），統稱**權重（weight）**。一開始它們是隨機的——這時模型對「it 看哪裡」毫無概念，胡亂分配權重。然後拿海量文字來「訓練」：每次給模型一段話，遮住下一個字要它猜，猜錯了就**微微調整那一大堆權重**，讓它下次猜得準一點點。重複幾兆次之後，那些權重就自己「沉澱」成能把語言處理得很好的數值——包括讓 embedding 表把語意近的 token 排在一起，也包括讓注意力的投影矩陣學會「看到 it 該往 animal 那邊投目光」。

這就是「學出來的」全部的意思：**不是人寫死的規則，是被資料反覆推著調出來的數字。** 至於怎麼「微微調整」（梯度下降、反向傳播）——那是訓練工程，**本書刻意不教**（梯度只在 ch06、ch17 點一下，數學指向《馴服無限》）。本書要看的是「注意力**已經學好**之後，在前向計算時到底在算什麼」。把訓練當成一個黑盒：它負責把權重調對，我們負責看懂這些調好的權重在跑什麼運算。

有了這兩塊地基，「it 看 animal」就站得住了：it、animal、street 都是向量；注意力用這些向量算出 it 對三個位置的權重；而讓這套運算「算得對」的那些投影矩陣，是訓練學出來的。本章到此都還只給直覺——數字版，ch04 見。

## 三步骨架：注意力的全部，就這三步

現在給出本書最重要的一張圖。它是全書的浮木：之後每介紹一個變體（√d 縮放、multi-head、RoPE、稀疏／線性注意力、FlashAttention、cross-attention、SSM 對照……），我都會明確點出「它動了這三步的**哪一步**、為什麼」。記住這三步，你就不會被一堆論文名詞淹沒。

```text
            ┌──────────┐   ┌────────────┐   ┌──────────────┐
 query ──►  │ 1. 打分  │─► │ 2. 正規化  │─► │ 3. 加權混合  │ ──► 輸出
 keys  ──►  │  score   │   │  softmax   │   │ weighted sum │
 values ─────────────────────────────────────────┘（值在此進場）
```

用脊椎的話講一遍，先**不給任何公式、不算任何數字**，只給直覺：

1. **打分（score）**：拿 it 這個位置當 **query**（「我在找什麼」），對句子裡每一個位置的 **key**（「我是什麼、可以被誰找到」）算一個「相關性分數」。直覺上，it 對 animal 的分數要高、對 street 的分數要低。分數怎麼算？標準作法是兩個向量做**內積**——內積大代表方向相近、相關性高（內積即相似度，快速複習見《矩陣是動詞》ch15）。

2. **正規化（normalize）**：上一步得到的是一排原始分數，數值大小沒有上限、也不能直接當「比例」用。這一步用 **softmax** 把它們壓成一組**和為 1 的權重**——比如 it 得到「看 animal 0.7、看 street 0.2、看 it 自己 0.1」這樣一個分布。它是一個機率分布（非負、加起來是 1，呼應《馴服隨機》）。softmax 還會**放大差距**：原本分數高一點的，正規化後權重拉得更高。

3. **加權混合（mix）**：拿這組權重，去對每個位置的 **value**（「我這裡實際存著什麼內容」）做**加權平均**。0.7 份 animal 的內容 ＋ 0.2 份 street 的內容 ＋ 0.1 份 it 自己的內容，混出一個新向量——這就是 it 這個位置在這一層的輸出。因為 animal 的權重最大，輸出主要承載了 animal 的資訊。it 這個位置，就這樣「**吸收了它該看的地方**」。

口訣，記住它：**相關性決定權重，權重決定混合。**

我認為這是整本書最該先被震撼到的一頁：**注意力的全部，就是「對所有位置做一次依相關性加權的平均」。** 沒有更多了。後面二十幾章看起來千變萬化的東西——除以 √d、多頭、RoPE、FlashAttention、Mamba——全都是在問同一件事的不同角度：這三步，**哪一步可以算得更準、更省、或換個算法**。Bahdanau 在 2014 年第一次讓 decoder 這麼做（ch03）、Vaswani 在 2017 年把它變成整個模型的唯一主軸（ch13），中間的跨度就是這三步從一個配件長成主角的過程。

注意這裡有一個**現在先埋、ch05 引爆**的反直覺點：第 3 步是「**混合全部**」，不是「挑一個最高的」。很多人第一次聽到注意力，腦中浮現的是「它找出最相關的那個詞」——這是錯的。它不挑一個，它把**所有**位置都按權重摻進來，只是相關的摻得多、不相關的摻得少（甚至少到趨近 0，但很少真的是 0）。「注意力＝找最相關的一個」是本書要打掉的第一個直覺陷阱，下一節就先點名。

## 一次純文字的試跑：給 it 三個直覺分數

來把三步骨架在脊椎上空跑一次。**全程純直覺、不出現任何真正的數字**（真正的脊椎數字從 ch04 起算，本章絕不偷跑），目的是讓你**感覺**到這三步怎麼把 it 推向 animal。

序列簡化成三個 token：**[animal, street, it]**。我們站在 it 這個位置，問它「該看哪裡」。

**第 1 步，打分（憑直覺）。** it 是個代名詞，它在找「我指的是誰」。it 對三個 key 的相關性，憑語意直覺排個高低：

```text
  it 對誰打分     直覺相關性     憑什麼（純直覺，非計算）
  ───────────     ─────────     ──────────────────────
  → animal        高            it 要指代一個名詞，animal 是合理對象
  → street        中            street 也是名詞、文法上可指代，但語意上輸給 animal
  → it 自己        低            指代自己沒帶來新資訊
```

我把這三個分數**故意只標成「高／中／低」**，不寫成 0.7／0.2／0.1，更不寫成 4／2／0——因為那是 ch04、ch05 要動手算的，現在寫出來就破壞了「該看哪裡是算出來的」這個全書懸念。這一步你只要接受：it 對 animal 的原始分數**應該**最高。

**第 2 步，正規化（憑直覺）。** 把「高／中／低」想像成被 softmax 壓成一組和為 1 的比例：animal 拿走最大的一份注意力預算、street 拿一小份、it 自己拿最少。三份加起來剛好是「一整份注意力」。

**第 3 步，加權混合（憑直覺）。** it 這個位置的輸出，就是「**大部分的 animal ＋ 一點點的 street ＋ 一丁點的 it 自己**」混出來的向量。混完之後，it 這個位置就「沾上了」animal 的資訊——模型在後續的計算裡，看 it 就等於看了一個「主要是 animal」的東西。**代名詞被解析了，而且是算出來的。**

這就是全書的縮影。你現在應該有一個明確的「我想知道」：那個「高／中／低」到底會被算成哪三個數字？softmax 把它們壓成怎樣的比例？為什麼要先除以一個 √d？——ch04、ch05、ch06 一題一題回答。本章只負責讓你**站到問題面前**。

## 故障模式與直覺陷阱

序章就該先豎起警示牌——這些是讀者（包括很多寫過 transformer 的人）最常踩的坑。本書的招牌段落就是這個「故障視角」，每一章都會有，因為「哪裡會出錯」往往比「正確答案是什麼」更能逼出真正的理解。序章先點名三個**直覺陷阱**，後面各章再各自引爆並修好。

| 直覺陷阱 | 錯在哪 | 在三步骨架的哪一步把你帶溝裡 | 哪一章修好 |
|---|---|---|---|
| 「注意力＝找出最相關的那一個詞」 | 它**混合全部**，相關的權重大、不相關的權重小但通常不是 0；softmax 幾乎不會真的給出 one-hot | 第 3 步（混合）：你以為是「選一個」，其實是「加權平均所有」 | ch05 |
| 「注意力天生就知道詞的順序」 | self-attention 本質是**集合**運算，置換不變——不外加位置資訊，"animal street it" 跟 "it street animal" 對它一模一樣 | 第 1 步（打分）：分數只看向量內容，看不見「誰在前誰在後」 | ch09、ch19 |
| 「注意力熱圖＝模型為什麼這樣判斷的解釋」 | 權重高不代表那就是模型決策的因果根因；這在學界至今（2026-06）是場沒有共識的嘴仗 | 不在運算、在**詮釋**：把相關性 dashboard 當因果根因 | ch20 |

第三個陷阱值得你現在就記住，因為它最誘人也最危險。看到 it 那一列在 animal 欄最濃，很容易脫口而出「你看，模型就是**因為**看了 animal 才解析出 it」。但「權重高」只說明「混合時 animal 那份摻得多」，它**不直接等於**「模型的判斷是由這個權重因果地造成的」。這正是你天天踩的工程陷阱的翻版：一個 metric 在 dashboard 上飆紅、跟事故同時發生，不代表它就是**根因**（相關≠因果，工程版見《馴服隨機》）。本書會把這個爭論留到 ch20 當高潮，誠實地告訴你：兩派吵到 2026 還沒分勝負，而且本書**不選邊**。

序章先把牌豎在這，是要你在後面每次看到漂亮的注意力熱圖時，心裡有一根弦：**這張圖在算什麼，跟它能解釋什麼，是兩件事。**

## 紙上推演

序章沒有數字要算（數字從 ch04 起）。這裡的推演是**口頭與紙筆的直覺練習**——目的不是算對，是逼你把「該看哪裡」這件事，翻譯成你自己的語言、接到你自己的經驗上。講得出來，才算真的站到了問題面前。

**推演 1：把「該看哪裡」翻成三個你熟的工程場景 [10 分鐘] ★**
注意力是一次「軟性的、學出來的查找」。請對下面三個你熟的後端機制，各寫一兩句：它原本是「硬」查找（規則手寫死、命中取一筆），如果把它變成注意力那種「軟」查找會長怎樣（query 對所有候選算相似度、按權重混合所有結果）？
(a) hash map / Redis 的 key-value 查找
(b) router / load balancer 的 routing 決策
(c) SQL 的 `JOIN`（一張表的列去另一張表找對應）

**推演 2：對著脊椎句子，把 it 該看 animal 的理由講清楚 [10 分鐘] ★★**
不准用「因為 it 指 animal」這種同義反覆。請口頭回答：
(a) 句子裡哪兩個名詞文法上都能當 it 的對象？
(b) 是哪一個字（線索）把天平壓向 animal、而不是 street？
(c) 如果把句尾的 tired 換成 wide，答案會怎麼變？這說明「該看哪裡」依賴的是文法還是語意？
(d) 用三步骨架的詞（打分／正規化／混合），描述「it 吸收了 animal 的資訊」這件事——但**不要寫任何數字**。

**推演 3：畫出全書地圖，標出你最想先懂哪一格 [10 分鐘] ★**
不看本章開頭的路線圖，憑剛讀完的印象，自己畫一張六個 Part 的全書地圖（一格一章名也行、只記得 Part 也行）。畫完對照開頭的圖，看漏了哪幾格。然後在地圖上圈出**你現在最想先懂的那一格**，並寫一句為什麼——這句話一年後回來看會很有意思。

### 推演解答

**推演 1 解答。**
(a) **hash map / Redis**：硬查找是「`get(key)`，key 精確相等才回那一筆 value，否則 miss」。軟查找版：拿 query 對**每一個**存著的 key 算相似度（內積），用 softmax 把相似度變成權重，回的不是某一筆 value，而是**所有 value 按權重混出來的一個新向量**。沒有 miss——最不像的 key 也貢獻一點點（權重很小）。這正是 ch04 要把 Q／K／V 對到 Redis 三個角色的那座橋。
(b) **router / load balancer**：硬版是「path 符合這條規則就送這台、其餘 fallback」。軟版：對每一台後端算一個「這筆請求跟你有多搭」的分數，然後把請求的「結果」按權重從各台混合回來——更像是把多個候選的回應加權平均，而不是二選一送一台。（真實 routing 當然不會這樣做；這只是用來體會「軟」的味道。）
(c) **`JOIN`**：硬版是「左表的列，找右表裡 `key` 相等的列接上」。軟版：左表的列當 query，對右表**每一列**算相似度，輸出是右表所有列的 value 按相似度加權的混合。這恰好是 ch07／ch12 的伏筆——self-attention 是同一張表的「軟 self-JOIN」，cross-attention 是兩張表的「軟 JOIN」；而它「對全表每一列都算一次」正是 O(n²) 的來源（ch14）。

**常見錯路**：把「軟」理解成「模糊匹配（取最像的那一筆）」。不是。模糊匹配仍然是「**取一筆**」，注意力是「**混合所有筆**」。差別就是上面那個直覺陷阱，ch05 會用脊椎數字把它釘死。

**推演 2 解答。**
(a) 文法上能當 it 對象的名詞是 **animal** 與 **street**（兩個都是單數名詞，都在 it 前面，文法上都通）。
(b) 把天平壓向 animal 的線索是 **tired（累）**——「累」是動物的屬性，不是街的。
(c) 換成 **wide（寬）**，答案翻成 **street**——「寬」是街的屬性。這說明「該看哪裡」依賴的是**語意／世界知識**，不是文法：句構一個字都沒變，只換了形容詞，指代就翻面了。這也正是為什麼這題對只會比對表面字串的舊方法很難——它需要「tired 配 animal、wide 配 street」這種跨位置的語意關聯。
(d) 三步版（無數字）：**打分**——站在 it 的位置，對 animal、street、it 各算一個相關性分數，因為 tired 這個語境，it 對 animal 的分數最高；**正規化**——把三個分數用 softmax 壓成和為 1 的權重，animal 拿最大份；**混合**——用這組權重對三個位置的 value 做加權平均，輸出主要是 animal 的內容，於是 it 這個位置「吸收了 animal」。

**常見錯路**：跳過 (c) 直接說「it 當然指 animal」。如果你的解釋對 tired/wide 的翻面無感，那你其實沒解釋「該看哪裡」，只是背了一個固定答案——而模型最不能做的就是背固定答案。

**推演 3 解答。**
沒有標準答案，這題是給你自己的。但有一個我想預先指出的事：很多人第一眼最想先懂的是 ch10 RoPE（旋轉位置編碼，因為它最漂亮，接《圓的影子》）或 ch16 FlashAttention（因為它最「實用」）。我的建議是——**忍住，照順序走到 ch06**。因為 RoPE 動的是第 1 步（打分時怎麼把位置塞進去）、FlashAttention 動的是「怎麼把這三步算得更省」，兩者都建立在「你已經徹底懂了沒加料的三步骨架」之上。ch04–ch06 那三章（Q/K/V、softmax、√d）是全書地基，地基沒夯實，後面那些漂亮的東西都會變成你只能背、不能講的名詞。你圈的那一格，就當作你讀到那裡時要驗收的「我終於懂了嗎」的考點。

## 自我檢核

口頭自答，講得出來才算過關。序章的檢核重在「你有沒有站到對的問題面前」，而不是「你會不會算」。

1. 用一句話說出本書要解決的核心問題——不要用「注意力」三個字，要描述那個**問題**本身。（提示：每個位置該看序列的哪裡，而且這件事是算出來、不是規定的。）
2. 脊椎句子裡的 it 指 animal 還是 street？是哪個字決定的？如果把那個字換掉會怎樣？——你能不靠背、靠推理講清楚嗎？
3. 三步骨架是哪三步？用脊椎的 it 把這三步各講一句（**不准出現數字**）。
4. 「注意力＝找出最相關的那一個詞」這句話錯在哪？正確的說法是什麼？
5. 為什麼「模型不讀字、讀向量」這件事是注意力的前提？embedding 在做什麼、它的向量為什麼語意近的會靠在一起？
6. 「權重是學來的」是什麼意思？本書為什麼**不教**那個「學」的過程（訓練）？
7. 看到一張漂亮的 it→animal 注意力熱圖，為什麼不能直接說「模型就是因為看了 animal 才解析出 it」？這跟你工程上「相關≠因果」的經驗怎麼對得上？
8. 不看書，畫出全書六個 Part 的大致地圖，並說出「三步骨架」這條主線怎麼貫穿其中。
