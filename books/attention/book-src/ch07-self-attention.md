# ch07 — 自注意力：每個 token 都是一次查找（O(n²) 的源頭）

> **本章解決什麼問題**：ch03 的 Bahdanau 注意力是「decoder 回頭看 encoder」——查的人和被查的人是兩群不同的 token。ch04–06 為了把三步骨架講清楚，只盯著一個查詢者：序章那個 it。但真正的 self-attention 有個更狠的設定：**句子裡每一個 token 都同時是查詢者、也是被查者，每個 token 都對全句做一次完整的查找**。這一章把脊椎從「只算 it 那一列」升級成「算完整的 3×3 注意力矩陣」，你會親眼看到那張 n×n 的方陣長出來——而 O(n²) 這個糾纏 transformer 一輩子的複雜度，就是從這張方陣的形狀直接讀出來的。本章不動三步骨架的任何一步，它動的是**規模**：把「一次查找」變成「n 次查找同時跑」，於是工作量從 n 變成 n²。

## 從你已知的出發

你做過這種事：手上有 N 筆資料，要找出「彼此最像的」配對。沒有索引、沒有預先排序，你能怎麼辦？老實的做法是雙重迴圈——拿第 1 筆比對其餘所有筆、第 2 筆比對其餘所有筆……一路到第 N 筆。這是 N×N 次比對，O(n²)。你在 code review 裡看到這種巢狀迴圈會皺眉，因為你知道資料一大它就爆。

self-attention 做的就是這件事，而且做得更徹底：它不是「找最像的配對」，而是**讓每一筆資料都對其他所有資料算一次相似度，再按相似度把大家混合進自己**。N 筆資料、每筆對 N 筆算分，總共 N×N 個相似度分數——一張 n×n 的方陣。這張方陣就是注意力矩陣（attention matrix），也是本書後面整個 Part IV「為什麼是 O(n²)」要對付的東西。

換個你更熟的角度：這像一個**沒有索引的 self-JOIN**。SQL 裡 `SELECT ... FROM t a JOIN t b ON ...`，如果 `ON` 條件能走索引（等值比對、可雜湊、可排序），查詢規劃器幫你把它降到接近 O(n log n) 甚至 O(n)。但 attention 的「ON 條件」是**相似度**——q 和 k 的內積要大。相似度不是等值、不可雜湊成桶、不能排序後二分查找（兩個向量「有多像」沒有全序關係可以建 B-tree）。所以這個 self-JOIN **建不了索引**，只能老老實實全表對全表比一遍。每對都要算，這就是 O(n²) 的根。

> 這條「建不了索引的表」的線，《從後端到 AI Infra》ch03 從**推論的記憶體與物理成本**那一面講得很細（n×n 矩陣在 HBM 裡多大、怎麼被反覆讀）。本書接棒講**原理那一面**：為什麼這個機制在數學上必然是 n×n，以及這件事 ch14 之後會逼出哪些妥協。兩本講同一張表的兩個側面。

在三步骨架裡，本章站在哪？答案有點特別——它哪一步都沒改：

```text
            ┌──────────┐   ┌────────────┐   ┌──────────────┐
 query ──►  │ 1. 打分  │─► │ 2. 正規化  │─► │ 3. 加權混合  │ ──► 輸出
 keys  ──►  │  score   │   │  softmax   │   │ weighted sum │
 values ─────────────────────────────────────────┘
                  │
                  └─►【self-attention：把這三步同時對 n 個 query 各跑一遍】
```

ch04–06 算的是「一個 query（it）跑完三步」。self-attention 是把**完全一樣的三步**，對序列裡的每一個 token 當 query 各跑一次。打分公式沒變、softmax 沒變、加權混合沒變——變的只是「跑幾次」。一個 query 跑出一個輸出向量；n 個 query 跑出 n 個輸出向量；而打分這一步，n 個 query 各對 n 個 key 算分，就攤開成一張 n×n 的分數表。三步骨架的形狀沒變，是它被**平行複製了 n 份**。本章的全部內容，就是看清這個「複製 n 份」帶來的後果。

## self vs cross：誰在查、誰被查

先把 ch03 和本章的差別釘死，因為這是 self-attention 名字裡那個「self」的全部含義。

ch03 的 Bahdanau 注意力，是翻譯場景：decoder 要生出一個中文詞，它的 query 去看 encoder 編好的英文 key/value。**查的人（query）來自目標序列，被查的人（key/value）來自源序列**——兩群是不同的 token。這叫 **cross-attention**（跨序列注意力，ch12 正式展開）。

self-attention 把這件事收進**同一個序列內部**：

```text
cross-attention（ch03 Bahdanau、ch12）：
   query  ← 來自序列 A（如中文目標端）
   key/value ← 來自序列 B（如英文源端）
   「A 的每個位置回頭看 B」

self-attention（本章）：
   query、key、value ← 全部來自同一個序列 X
   「X 的每個位置回頭看 X 自己（包含自己）」
```

具體說，self-attention 拿同一批 token 的向量 X，用三個學來的投影矩陣 W_Q、W_K、W_V，**從同一個 X 算出 Q、K、V 三套**（landscape §2 §3.2.3 原文：encoder self-attention 的 Q/K/V 同源自前一層）。於是序列裡每個 token 都有三個身分同時在身上：

- 當 **query** 時，它在問：「我（這個位置）該看序列裡的誰？」
- 當 **key** 時，它在被別人衡量：「別的位置在找的東西，像不像我？」
- 當 **value** 時，它準備好被取走：「如果有人決定看我，這是我要交出去的內容。」

回到脊椎那句 *"the animal didn't cross the street because **it** was too tired."* 簡化成 [animal, street, it]。在 ch04–06 我們只讓 it 當 query。但在真正的 self-attention 裡，animal 也是 query（它在問「我該看誰」）、street 也是 query、it 還是 query——**三個 token 同時各做一次查找**。同時，三個 token 也都是 key（被衡量）、都是 value（待取）。每個 token 一人分飾三角，這就是 self。

> **這裡了不起／反直覺在哪**：ch04 把 attention 比喻成「軟性字典查找」，那時 query 是外來的查詢條件、字典是別人的。self-attention 把這個比喻擰了一圈——**字典查它自己**。序列裡的每個詞，既是查詢條件、又是被索引的鍵、又是存著的值。it 在查 animal 的同時，animal 也在查它自己該看誰、street 也在查它自己該看誰。一句話裡所有詞**同時、互相**地查找彼此，一次把全句的兩兩關係都算出來。你第一次意識到「查的人和被查的人是同一批」時，會覺得這個設定有點繞——但正是這個「自己查自己」，讓一個位置能根據**全句語境**重新理解自己，這是 RNN 那種「一個字一個字往後傳」做不到的。

## 整張注意力矩陣：QKᵀ 是 n×n

現在把「每個 token 都當一次 query」這件事寫成矩陣，那張 n×n 的方陣就會自己冒出來。

ch04 算單一 query 時，打分是 q · k 一個內積。現在有 n 個 query，把它們疊成一個矩陣 **Q**（n 列，每列是一個 token 的 query 向量，維度 d）；同理把 n 個 key 疊成 **K**（n 列、維度 d）。打分這一步，要算「每個 query 對每個 key 的內積」——這正好是矩陣乘法 **Q Kᵀ**（內積即相似度、矩陣乘法即批量內積，運算深度見《矩陣是動詞》ch06、ch15）：

```text
        Kᵀ 的欄 = 每個 token 當 key
        ┌─ k_animal  k_street  k_it ─┐
 Q 的列 │                            │
 q_animal│  q_a·k_a   q_a·k_s   q_a·k_i│   ← animal 當 query 對三個 key 的分數
 q_street│  q_s·k_a   q_s·k_s   q_s·k_i│   ← street 當 query 的那一列
 q_it    │  q_i·k_a   q_i·k_s   q_i·k_i│   ← it 當 query（ch04–06 一直在算的就是這一列）
        └────────────────────────────┘
                  這是 QKᵀ：一張 n×n 的分數表
```

維度對一下就懂為什麼結果是 n×n：Q 是 n×d、Kᵀ 是 d×n，相乘消掉中間的 d，得到 **n×n**（web 查證：Q ∈ ℝ^{n×d} 乘 Kᵀ ∈ ℝ^{d×n}，結果 n×n，計算量 O(n²·d)，2026-06）。矩陣的第 (i, j) 格，就是「第 i 個 token 當 query、對第 j 個 token 當 key」的相似度分數。**n 個 query × n 個 key = n² 個分數**，一個都跑不掉。

接著三步骨架照常走：整張 QKᵀ 除以 √d（ch06），對**每一列**各做一次 softmax（ch05 的正規化，逐列做，因為每個 query 各自要一個「和為 1 的權重分布」），最後每一列的權重去加權混合 V，得到每個 token 各自的輸出。完整公式還是那一行，只是現在 Q、K、V 都是矩陣：

```text
Attention(Q, K, V) = softmax(QKᵀ / √d) · V
                     └──┬──┘ └──┬──┘
                   n×n 分數表   逐列正規化     最後乘 V (n×d) 得 n×d 的輸出
```

注意這裡的關鍵轉折：在 ch04–06，softmax 是對「一排三個分數」做一次。在 self-attention，softmax 是**對 n×n 矩陣的每一列各做一次**——共 n 次 softmax，每次處理一列的 n 個分數。三步骨架沒變，只是「正規化」這一步從做一次變成做 n 次（每個 query 一次）。

## 親手算脊椎的完整 3×3 矩陣

ch04–06 我們只填了這張表的**第三列**（it 當 query）。現在把三列都填滿，看完整的方陣。

脊椎維度 d=4、√d=2，三個 key 是 k_animal=(2,1,1,0)、k_street=(0,1,1,0)、k_it=(1,−1,0,0)（基準向量）。

it 那一列我們已經有了，是全書正典（ch06 釘下的）。但 animal 和 street 當 query 時，它們的 query 向量是多少？**基準表只定義了 q_it=(1,1,1,0,...)＝(1,1,1,1) 這一個 query 向量**（它是 it 經 W_Q 投影出來的）。為了不偽造全書脊椎，我在本章用兩個**僅供本章說明的輔助 query**（明確標示：這不是新的全書脊椎數字，只是為了把方陣填滿、看清形狀）：

```text
本章說明用輔助 query（非全書脊椎，僅本章）：
  q_animal = (2, 0, 0, 0)   ← 讓 animal 主要關注「自己這類實體」
  q_street = (0, 1, 1, 0)   ← 讓 street 在 animal/street 之間猶豫
  q_it     = (1, 1, 1, 1)   ← 全書正典 query（基準表），這一列數字不可動
```

逐格算內積（打分），三列各算對三個 key 的分數：

```text
animal 列（q_animal=(2,0,0,0)）：
  q_a·k_animal = 2·2+0·1+0·1+0·0 = 4
  q_a·k_street = 2·0+0·1+0·1+0·0 = 0
  q_a·k_it     = 2·1+0·(−1)+0+0  = 2          → raw [4, 0, 2]

street 列（q_street=(0,1,1,0)）：
  q_s·k_animal = 0·2+1·1+1·1+0·0 = 2
  q_s·k_street = 0·0+1·1+1·1+0·0 = 2
  q_s·k_it     = 0·1+1·(−1)+1·0+0= −1         → raw [2, 2, −1]

it 列（q_it=(1,1,1,1)，正典）：
  q_i·k_animal = 1·2+1·1+1·1+1·0 = 4
  q_i·k_street = 1·0+1·1+1·1+1·0 = 2
  q_i·k_it     = 1·1+1·(−1)+1·0+1·0 = 0       → raw [4, 2, 0]   ← 和 ch04 一模一樣
```

整張原始分數矩陣 QKᵀ：

```text
              key:animal  key:street  key:it
query:animal     4           0           2
query:street     2           2          −1
query:it         4           2           0      ← 正典那一列
```

除以 √d=2，再逐列 softmax。把每一列當成 ch05/ch06 做過的事重算一遍：

```text
animal 列：[4,0,2]/2 = [2, 0, 1]
   e²=7.389, e⁰=1.000, e¹=2.718, 和=11.107
   → [7.389/11.107, 1/11.107, 2.718/11.107] = [0.665, 0.090, 0.245]

street 列：[2,2,−1]/2 = [1, 1, −0.5]
   e¹=2.718, e¹=2.718, e^(−0.5)=0.607, 和=6.043
   → [2.718/6.043, 2.718/6.043, 0.607/6.043] = [0.450, 0.450, 0.100]

it 列：[4,2,0]/2 = [2, 1, 0]
   e²=7.389, e¹=2.718, e⁰=1.000, 和=11.107
   → [0.665, 0.245, 0.090]    ← 全書正典，和 ch06 完全吻合
```

完整的注意力權重矩陣（每一列和為 1）：

```text
              animal   street    it      這一列在說什麼
query:animal  0.665    0.090    0.245    animal 主要看自己（語境裡先確認「我是哪個實體」）
query:street  0.450    0.450    0.100    street 在 animal/street 間猶豫（軟、未定）
query:it      0.665    0.245    0.090    it 看 animal 最多 ← 序章的懸念在這裡被「算」出來
```

畫成 ASCII 熱圖（█ 最濃對應最大權重，往下到 ░ 最淡；看一列的濃淡分布，不看跨列比例）：

```text
              animal      street      it
query:animal  ███████     █░          ███
query:street  ████        ████        █░
query:it      ███████     ███         █░
              ↑↑↑                      ↑
        it 那一列在 animal 欄最濃——it→animal，序章的答案
```

讀這張熱圖：看 **it 那一列**（最下面），它在 **animal 欄**最濃（0.665），這就是序章一直懸著的問題——「it 指的是 animal 還是 street？」——被注意力機制**算**出來的答案：it 把最大的權重放在 animal 上。這不是誰規定的，是 q_it·k_animal=4 這個內積最大、經過縮放與 softmax 後自然贏出來的。ch04 算分、ch05 正規化、ch06 縮放，三章鋪墊就為了讓你現在看懂這一列為什麼長這樣。

而旁邊兩列在說別的故事：animal 那列主要看自己（一個實體 token 先確認自身語境）；street 那列在 animal 和 street 之間五五開（0.450/0.450，一個「軟、還沒拿定主意」的分布——回想 ch06，分數越接近，softmax 越平）。**重點不是這兩列的具體數字**（它們用的是本章輔助 query），重點是：self-attention 一次把**全句每個位置該看哪裡**都算出來了，不只是 it 那一列。三列同時、平行算完，這就是 self-attention 的威力。

如果你想看演算法骨架，ch06 那五行 numpy 一個字都不用改，差別只在餵進去的 Q 從「一個 query」變成「整個序列的 query 矩陣」：

```python
import numpy as np

def self_attention(X, W_Q, W_K, W_V):
    Q, K, V = X @ W_Q, X @ W_K, X @ W_V   # all from the SAME X — that's "self"
    d = K.shape[-1]
    scores = Q @ K.T / np.sqrt(d)          # (n, n) matrix — here O(n^2) is born
    weights = softmax(scores, axis=-1)     # softmax over each ROW (per query)
    return weights @ V                      # (n, d) — one output vector per token
```

`Q @ K.T` 那一行的輸出形狀是 `(n, n)`——O(n²) 就誕生在這一行。Q、K、V 全部來自同一個 `X`，這就是程式碼裡的「self」。把脊椎餵進去（X 是三個 token、W 是恰好投影出上面那些 q/k/v 的矩陣），`scores` 就是上面手算的那張除以 2 之後的表，`weights` 就是那張 3×3 權重矩陣，it 那一列印出來正是 `[0.665, 0.245, 0.090]`——和手算對得上。

## 為什麼必然是 O(n²)：把方陣的格子數出來

現在把複雜度從這張方陣直接讀出來，這是本章的硬核心。

注意力矩陣是 n×n，**格子數就是 n²**。每一格要算一個 query 對一個 key 的內積（d 維，d 次乘加），所以打分這一步的總計算量是 n²·d 次基本運算——複雜度 **O(n²·d)**（web 查證：原始論文 Table 1 明列 self-attention per-layer 複雜度 O(n²·d)，2026-06）。在我們關心序列長度 n 怎麼吃成本時，d 是固定的模型超參數，主導項就是 **n²**。

更要命的是**記憶體**：你得把這張 n×n 的分數表存下來才能做 softmax（softmax 要先知道整列才能正規化），所以空間也是 O(n²)（web 查證：儲存 n×n 注意力矩陣需要 O(n²) 記憶體，2026-06）。時間 O(n²)、空間 O(n²)，兩個都是平方。

把脊椎那張 3×3 數一遍找感覺：n=3，格子數 = 3² = 9。九個內積。小到不痛不癢。但 n² 的脾氣是「一開始溫和、後面失控」。把序列拉長：

```text
   n（序列長度）   n²（注意力矩陣格數）      相對 n=512 的倍數
   3                          9               —（脊椎玩具）
   512                  262,144               1×（短句／一段話）
   4,096             16,777,216              64×（一篇長文）
   100,000       10,000,000,000            ~38,000×（一本書／長對話）
   1,000,000  1,000,000,000,000           ~3,800,000×（超長 context）
```

讀這張表的方式：**context 長 10 倍，注意力成本變 100 倍**（10²）。從 512 拉到 4096（長 8 倍），成本變 64 倍（8²）。從一段話（n≈512）到一本書（n≈10萬），格子數從二十幾萬暴衝到一百億。這就是為什麼「長 context」在工程上這麼貴、為什麼 Part IV 整整三章都在想辦法不付這筆平方稅。

對照 RNN（ch02 那個被注意力取代的舊世界）：RNN 一個字一個字處理，第 i 步只更新一個固定大小的隱藏狀態，總共 n 步——複雜度 **O(n)**，線性。看起來 RNN 在複雜度上完勝？不。代價藏在另一個維度：RNN 的 n 步必須**循序執行**（第 i 步要等第 i−1 步算完），無法平行；而 self-attention 的 n×n 矩陣可以**一次性平行算完**——每一格互不依賴，GPU 一口氣全算。

原始論文 Table 1 把這個取捨講得很精準（web 查證，2026-06）：

```text
                 每層計算複雜度    循序操作數    最長路徑
   self-attention   O(n²·d)         O(1)         O(1)
   recurrent (RNN)  O(n·d²)         O(n)         O(n)
```

- **計算複雜度**：RNN 是 O(n·d²)（線性於 n），self-attention 是 O(n²·d)（平方於 n）。當 n < d 時 self-attention 反而更省（短序列、高維度，論文原話：句子表示時 n 通常小於 d）；但 n 一大，平方項就壓過一切。
- **循序操作數**：self-attention O(1)（全部平行），RNN O(n)（必須一步步來）。這是 self-attention 之所以能餵飽 GPU、訓練飛快的根本原因——它用「平方的總工作量」換「常數的循序深度」。
- **最長路徑**：任意兩個位置之間的資訊要走幾步才能互相影響。self-attention 是 O(1)（任兩個 token 直接在矩陣裡有一格相連，一步到位），RNN 是 O(n)（句首的資訊要經過 n 步才傳到句尾，路上一路衰減——這正是 ch02 講的長程依賴衰減）。

> **這裡了不起／反直覺在哪**：O(n²) 常被當成 self-attention 的「原罪」「缺陷」，後面三章都在補救它。但請先看清它**買到了什麼**：那張 n×n 的方陣，意思是**任意兩個 token 之間都有一條直達的連線**，句首和句尾一步互通。RNN 為了省成 O(n)，代價是句首到句尾要走 n 步、訊號一路衰減；self-attention 願意付 O(n²)，換來的是「全句任兩點直接對話」。O(n²) 不是 bug，是這個機制**敢讓所有位置直接互看**的明碼標價。Part IV 之後所有的高效注意力，本質都在問同一個問題：**能不能少付一點這筆錢，又別丟掉太多「直達連線」？**——那是 ch14 起的故事。

## 雙向：每個 token 看得到全句（包含未來）

self-attention 的 3×3 矩陣裡，**每一格都有值**——animal 看得到 it、street 看得到 it、it 也看得到 animal 和 street。沒有任何一格被擋住。這叫**雙向（bidirectional）**：每個位置都能看序列的全部，左邊右邊都看（web 查證：encoder self-attention 中每個位置可關注前一層所有位置，雙向，2026-06）。

用你的工程直覺：這像你能**同時讀到上下文兩邊**，不像 streaming 處理只能看已經流過的資料。it 在被解析時，左邊有 animal、street，右邊有 was、too、tired——雙向 self-attention 讓 it 把這些全看進來，於是「它累了」這個線索（在 it 右邊）也能幫忙確認「累的是會累的 animal、不是 street」。一次抓全句關係，這是雙向的威力。

但這裡埋著一個**重大轉折的伏筆**，我先點明、ch11 才拆：雙向意味著 it 看得到它**右邊**還沒出現的詞。在「理解一整句」的場景（像 BERT 做填空、做分類）這完全沒問題——句子是現成的，當然兩邊都能看。但在**生成**場景就出大事了：模型一個字一個字往外吐，生第 i 個字時，第 i+1 個字**還不存在**，它怎麼可能「看到未來」？如果允許看未來，訓練時模型會作弊（直接抄答案），推論時又無未來可看，對不上。

所以生成模型（GPT 那類）必須把這張矩陣的**上三角（看未來的那些格）擋掉**——這就是 ch11 的因果遮罩（causal mask）。本章先把話放這裡：**self-attention 的本性是雙向、全看；單向、只看左邊是後天用遮罩強加的限制。** 脊椎在這裡是雙向的（it 合法地看 animal、street，也看得到右邊），到 ch11 我們會把它改成「it 只能看左邊的 animal、street」，那個對稱會被打破——記住這個轉折。

## 故障模式與直覺陷阱

| 直覺陷阱 / 故障 | 為什麼錯／在哪一步把你帶溝裡 | 怎麼自我察覺、正確版是什麼 |
|---|---|---|
| 「self-attention 是某個位置去看別的位置，自己不看自己」 | 漏了對角線。矩陣的 (i,i) 格也有值——每個 query **包含自己**這個 key。it 那列裡 it→it 是 0.090，不是 0。一個 token 完全可以（也常常）把不少權重放在自己身上。 | 看矩陣對角線：它不是空的。self 的意思就是 query 集合＝key 集合，自己也在被查之列。 |
| 「O(n²) 是因為要做 n 次 softmax / 因為有多頭」 | 抓錯源頭。O(n²) 純粹來自 **QKᵀ 產生 n×n 個分數**（n 個 query × n 個 key）。多頭（ch08）是把 d 切開平行做，不改 n²；softmax 是逐列做、總共 n 列 ×每列 n 項＝也是 n² 量級，但它是 QKᵀ 的後果不是獨立原因。 | 問：哪一步輸出了 n×n？是 QKᵀ。把那一步的輸出形狀寫出來就抓到 O(n²) 的根。 |
| 「attention 矩陣是對稱的（i 看 j ＝ j 看 i）」 | 不對稱。第 (i,j) 格是 q_i·k_j，第 (j,i) 格是 q_j·k_i——因為 query 和 key 用**不同**投影（ch04「我找的≠我是的」），q_i·k_j ≠ q_j·k_i。脊椎裡 it→animal=0.665 但 animal→it=0.245，明顯不等。 | 比兩個對稱位置的格子：它們不相等。非對稱正是 Q≠K 的直接後果，這是 attention 比「相似度矩陣」更靈活的地方。 |
| 「self-attention 是 O(n²) 所以一定比 O(n) 的 RNN 慢」 | 只看計算量、漏看平行性。self-attention 循序深度 O(1)（全平行），RNN O(n)（必須一步步來）。在 GPU 上、序列不太長時，self-attention 反而**快得多**——它用「平方的總量」換「常數的循序深度」，剛好對上平行硬體。 | 分開看兩個指標：總工作量 vs 循序深度。O(n²) 貴在工作量，但 O(1) 循序讓它能餵飽 GPU。慢不慢要看 n 多大、硬體多平行。 |
| 「雙向（看全句）永遠比較好，為什麼還要遮罩」 | 把「理解」和「生成」混為一談。理解現成句子（BERT）當然雙向好；但生成是一個字一個字吐，生第 i 字時第 i+1 字不存在，看未來＝作弊＋推論對不上。 | 問這個任務「未來的 token 此刻存在嗎」。存在（整句現成）→ 可雙向；不存在（邊生邊看）→ 必須遮罩成單向（ch11）。 |
| 「QKᵀ 那張 n×n 矩陣一定要先整個算出來、存進記憶體」 | 這是 O(n²) **空間**的來源，但**不是數學上的必然**——它是「先全算、再 softmax」這種樸素實作的後果。FlashAttention（ch16）證明可以**分塊算、邊算邊合併 softmax**，永遠不實體化整張 n×n，省的是空間不是精度（精確、非近似）。 | 記住：O(n²) 的**時間**很難繞過（分數對數就是 n²），但 O(n²) 的**空間**可以靠不實體化整張矩陣來省（ch16）。兩者是不同的帳。 |

**和後面章節的鉤子**：這張 n×n 矩陣是全書下半本的主角。它太貴（O(n²)）→ ch14 把代價講透、ch15 用稀疏/線性「少算一些格或換掉 softmax」、ch16 用 FlashAttention「不實體化整張矩陣」；它看不見順序（重排 token 這張矩陣只是跟著重排，權重不變）→ ch09 必須外加位置編碼；它預設雙向 → ch11 用遮罩擋掉上三角換成單向。本章把這張表立起來，後面九章都在對它動手。

## 紙上推演

### 推演題

**第 1 題 [18 分鐘]（★★）** 親手填滿脊椎的完整 3×3 注意力矩陣。用三個 key k_animal=(2,1,1,0)、k_street=(0,1,1,0)、k_it=(1,−1,0,0)、d=4、√d=2，以及本章輔助 query q_animal=(2,0,0,0)、q_street=(0,1,1,0)、q_it=(1,1,1,1)：(a) 算出 3×3 的原始分數矩陣 QKᵀ；(b) 整張除以 √d；(c) 對每一列各做一次 softmax（一步不跳，寫出每列的 e 次方、求和、相除）；(d) 確認 it 那一列等於全書正典 [0.665, 0.245, 0.090]；(e) 一句話說「為什麼要逐列 softmax 而不是對整張矩陣做一次 softmax」。

**第 2 題 [12 分鐘]（★★）** O(n²) 從哪來、為什麼。(a) 不用算具體數字，說明 QKᵀ 的輸出形狀為什麼是 n×n（用 Q、Kᵀ 的維度推）；(b) 由此說明計算量為什麼是 O(n²·d)、記憶體為什麼是 O(n²)；(c) 對照 RNN 的 O(n)，解釋「為什麼 self-attention 計算量更大、卻常常更快」（提示：循序操作數 O(1) vs O(n)）。

**第 3 題 [10 分鐘]（★★★）** 注意力矩陣的非對稱性。用脊椎，算 it→animal 的權重（你已經有：0.665）和 animal→it 的權重（用本章的 q_animal 算 animal 那列，取 it 欄）。(a) 兩者相等嗎？(b) 從「query 和 key 用不同投影」解釋為什麼第 (i,j) 格 ≠ 第 (j,i) 格；(c) 如果硬要讓矩陣對稱，要犧牲什麼（提示：回想 ch04「我找的≠我是的」）。

**第 4 題 [10 分鐘]（★）** 把 O(n²) 對到你的後端直覺並估算。你有一個服務要對 n 筆進來的事件做「兩兩相關性比對」（沒有索引可用）。(a) n=1000 和 n=10000 時各要算幾次比對？(b) n 變 10 倍，成本變幾倍？(c) 把這個場景對到 self-attention 的哪一步，並說明「為什麼相似度比對建不了傳統索引」（一句話，預告 ch14）。

### 推演解答

**第 1 題。**
(a) 逐格內積（已在正文逐步算過）：

```text
              animal  street  it
query:animal     4       0      2
query:street     2       2     −1
query:it         4       2      0
```

(b) 除以 √d=2：

```text
              animal  street   it
query:animal    2.0     0.0    1.0
query:street    1.0     1.0   −0.5
query:it        2.0     1.0    0.0
```

(c) 逐列 softmax：
- animal 列 [2,0,1]：e²=7.389、e⁰=1.000、e¹=2.718，和=11.107 → [0.665, 0.090, 0.245]。
- street 列 [1,1,−0.5]：e¹=2.718、e¹=2.718、e⁻⁰·⁵=0.607，和=6.043 → [0.450, 0.450, 0.100]。
- it 列 [2,1,0]：e²=7.389、e¹=2.718、e⁰=1.000，和=11.107 → [0.665, 0.245, 0.090]。

(d) it 列 = **[0.665, 0.245, 0.090]**，和 ch06 的全書正典完全吻合 ✓（每列也驗：和皆=1.000）。
(e) 因為**每個 query 需要自己的一個「和為 1 的權重分布」**——animal 該怎麼分配注意力、it 該怎麼分配，是各自獨立的決定。對整張矩陣做一次 softmax 會把所有 query 的分數混在一起正規化，那就不是「每個 token 各自決定看哪裡」了。逐列＝每個查詢者各算各的預算。常見錯路：對欄做 softmax（那會變成「每個 key 被多少 query 看」的正規化，語義完全不同）。

**第 2 題。**
(a) Q 是 n×d（n 個 query、每個 d 維），Kᵀ 是 d×n。矩陣相乘 (n×d)·(d×n) 消掉中間的 d，得到 **n×n**——第 (i,j) 格是第 i 個 query 與第 j 個 key 的內積。
(b) n×n 個分數、每個分數是 d 維內積（d 次乘加），總計算量 ∝ n²·d ⇒ **O(n²·d)**；做 softmax 前必須把整張 n×n 表存著（softmax 要先看完整列才能正規化），所以空間 ∝ n² ⇒ **O(n²)**。
(c) self-attention 計算量 O(n²·d) 確實大於 RNN 的 O(n·d²)（當 n 大時），但 self-attention 的 n×n 每一格**互不依賴、可一次平行算完**（循序操作數 O(1)），而 RNN 的 n 步**必須循序**（第 i 步等第 i−1 步，循序操作數 O(n)）。在平行硬體（GPU）上，循序深度才是牆鐘時間的瓶頸——self-attention 用「平方的總工作量」換「常數的循序深度」，剛好對上平行硬體，所以常常更快。

**第 3 題。**
(a) it→animal = 0.665（it 列 animal 欄）；animal→it = 0.245（animal 列 it 欄）。**不相等**。
(b) 第 (i,j) 格是 q_i·k_j，第 (j,i) 格是 q_j·k_i。因為 query 和 key 經過**不同**的投影矩陣（W_Q≠W_K），同一個 token 當 query 的向量和當 key 的向量不同，所以 q_it·k_animal（it 在找的 vs animal 是的）≠ q_animal·k_it（animal 在找的 vs it 是的）。兩個內積算的是兩件不同的事。
(c) 要讓矩陣對稱，得逼 W_Q=W_K（query 投影＝key 投影），這樣 q_i·k_j 才會等於 q_j·k_i。但這就回到 ch04 講過的退化——「我找的」被迫等於「我是的」，每個 token 最像的永遠是自己（對角線最大），注意力失去「我在找一種我自己未必具備的東西」的表達力。非對稱是 Q≠K 換來的靈活性，不是缺陷。

**第 4 題。**
(a) n=1000：1000² = **1,000,000** 次比對；n=10000：10000² = **100,000,000** 次。
(b) n 變 10 倍，成本變 **100 倍**（10²）——這就是平方成長。
(c) 對應 self-attention 的**打分**步（算 QKᵀ）：每筆事件對每筆事件算一次相似度＝query 對 key 算內積，n×n。相似度比對建不了傳統索引，是因為 hash/B-tree 索引靠「鍵可雜湊或可排序」把查找降到 O(1)/O(log n)，但「兩個向量有多像」既不能雜湊成桶、也沒有全序可以二分——所以只能每對都算，逼出 O(n²)（ch14 詳述）。

## 自我檢核

口頭自答，講得出來才算過：

1. 用一句話說 self-attention 和 ch03 cross-attention 的差別（提示：query 和 key/value 是不是同源；「自己查自己」vs「查別群」）。
2. 為什麼 QKᵀ 的輸出是 n×n？把 Q（n×d）和 Kᵀ（d×n）的維度相乘推一次給自己聽。
3. O(n²) 到底是從哪一步、為什麼長出來的？（不是 softmax、不是多頭，是 QKᵀ 產生 n×n 個分數：n 個 query × n 個 key。）
4. self-attention 計算量 O(n²) 比 RNN 的 O(n) 大，為什麼它在 GPU 上常常更快？（循序操作數 O(1) vs O(n)，用平方總量換常數循序深度。）
5. 注意力矩陣為什麼**不**對稱？it→animal 和 animal→it 為什麼可以不相等？（Q≠K，q_i·k_j ≠ q_j·k_i。）
6. 「雙向」是什麼意思、它買到了什麼、又埋了什麼伏筆？（每個位置看全句、任兩點直達；但生成時看未來＝作弊，ch11 要用遮罩擋掉。）
7. 為什麼 self-attention 對每一列各做一次 softmax，而不是對整張矩陣做一次？（每個 query 各要一個和為 1 的權重分布、各自決定看哪裡。）
8. O(n²) 的「時間」和「空間」是同一筆帳嗎？哪一個可以靠不實體化整張矩陣省下來？（時間 O(n²) 難繞過，空間 O(n²) 可靠 FlashAttention 分塊省，ch16。）
