# ch02 — 固定向量的瓶頸：RNN 與 seq2seq 為何卡住

> **本章解決什麼問題**：上一章你看到一個代名詞「it」該看哪裡的懸念，也看到了三步骨架（打分→正規化→加權混合）的預告。但在三步骨架誕生之前，機器翻譯是怎麼做的？這一章帶你回到 2014 年——注意力還沒出現的世界。我們要看清楚當時主流的 seq2seq 為什麼會卡住：它把整個輸入句硬壓成「一個固定大小的向量」，然後要 decoder 只靠這個向量還原一切。這是一個**具體的痛**，注意力（ch03 開始）就是來治它的。看懂這個痛，你才會明白為什麼三步骨架那麼值錢。

## 從你已知的出發

你一定寫過這種程式：上游給你一個結構豐富、長度可變的物件，但下游介面只收一個**固定大小的欄位**。於是你得想辦法把它「塞進去」。

- 你要把一個任意長的使用者操作歷史，塞進一個固定 256 bytes 的 session token。
- 你要把一筆富 JSON 物件，序列化進一個固定 schema 的 protobuf，而那個 schema 早就定死了欄位數。
- 你要把一段不知道多長的 log，hash 成一個固定長度的 fingerprint。

這些場景都有同一個結構：**可變大小的資訊 → 固定大小的容器**。而你心裡很清楚，這件事一旦資料變大就會出事：要嘛截斷、要嘛覆蓋、要嘛把不同的東西雜湊到同一格（碰撞）。固定大小的容器有一個**資訊上限**，超過就丟東西，而且通常是**默默地**丟——沒有報錯，只是下游拿到的東西「不太對」。

2014 年的神經機器翻譯（neural machine translation, NMT）做的就是這件危險的事，而且還是核心架構。它把「整個來源句子」當成那個可變大小的富物件，把「一個固定維度的向量」當成那個塞不下的容器。整本書的主角——注意力——之所以會出現，第一個動機就是：**別再硬塞了，讓 decoder 需要時回頭去看原句**。

這一章我們先不碰任何 Q/K/V、不算任何 softmax。我們只做一件事：把這個「固定向量瓶頸」徹底講清楚，清楚到你能對另一個工程師用一句話講完——「它就是把一本書摘要成一句話，再要人從那句話把整本書翻譯出來」。

## seq2seq：把翻譯看成「編碼再解碼」

先把舞台搭好。2014 年有兩篇奠基論文，幾乎同時把「序列轉序列（sequence to sequence, seq2seq）」這個框架立了起來：

- **Sutskever、Vinyals、Le 2014**「Sequence to Sequence Learning with Neural Networks」（arXiv 1409.3215，NIPS 2014）。用多層 LSTM 把輸入序列編碼成一個固定維度向量，再用另一個 LSTM 從這個向量解碼出目標序列。WMT'14 英法達 34.8 BLEU，首次讓純神經網路在大規模翻譯上勝過當時的 phrase-based 統計系統。
- **Cho 等人 2014**「Learning Phrase Representations using RNN Encoder–Decoder」（EMNLP 2014）。提出 encoder-decoder 架構，並首次引入 **GRU**（gated recurrent unit）這種閘控循環單元。作者群裡就有 Cho、Bahdanau、Bengio——記住這幾個名字，下一章的注意力正是從這群人裡長出來的。

這個框架的骨架是兩個角色（你可以直接把它讀成「兩個 RNN 串起來」，RNN 內部怎麼運作不是本書主題，一句帶過：它就是一個逐步讀入、每步更新一個隱藏狀態的網路）：

```text
                       encoder（讀）                    decoder（寫）
  the   animal  ...  tired  <eos>          c          La  bête  ...  fatiguée
   │      │           │      │             │           │    │          │
   ▼      ▼           ▼      ▼             ▼           ▼    ▼          ▼
  h1 ─►  h2 ─► ... ─► hn ─► [收尾] ─────► context ─► s1 ─► s2 ─► ... ─► sm
                                          vector
  逐步讀入、每步更新隱藏狀態           只有這一個向量          只靠 c 起步，逐步吐出譯文
```

- **encoder（編碼器）**：逐字讀進來源句 *the animal didn't cross the street ...*，每讀一個字就更新一次自己的隱藏狀態 h。讀到句尾（`<eos>`）時，它把**最後一個隱藏狀態**當成整句的摘要，這就是那個固定維度向量，論文裡常叫它 **context vector（context 向量）**，我們記成 c。
- **decoder（解碼器）**：拿到 c 當起點，逐字生成目標句 *La bête ne traversait pas la rue ...*，每生成一個字就更新一次自己的隱藏狀態 s，直到吐出 `<eos>` 為止。

這是個很乾淨、很對稱的想法。把翻譯拆成「理解（encode）」和「重述（decode）」兩段，中間用一個向量交棒。問題全部都藏在那個交棒的向量裡。

### 那個向量到底裝了多少東西

c 是一個固定維度的向量——在 Sutskever 的論文裡是 1000 維左右。**不管來源句是 5 個字還是 50 個字，c 永遠是同一個大小**。

這就是整章的核心矛盾，我用一句話標出來，它值得你停下來想十秒：

> 來源句的資訊量隨長度線性成長，但裝它的容器大小是**常數**。

5 個字的句子，1000 維綽綽有餘。50 個字的句子呢？句首那個 *the animal* 的資訊，要在 encoder 走完 50 步的過程中，一路被後面 48 個字的更新「擠」著、改寫著，最後還要在 c 裡留下足夠的痕跡，好讓 decoder 在生成譯文第 40 個字時還能用到它。這幾乎不可能不損失。

這不是我事後諸葛。Cho 等人 2014 年另一篇分析論文「On the Properties of Neural Machine Translation」（arXiv 1409.1259）就直接量到了這個現象：**NMT 在短句上表現不錯，但句子越長、生詞越多，表現就急遽下滑**（performance degrades rapidly as the length of the sentence ... increase）。而 Bahdanau、Cho、Bengio 2014 那篇即將開啟注意力時代的論文（arXiv 1409.0473），開宗明義就把矛頭指向這裡，原文（2026-06 查證）：

> *"we conjecture that the use of a fixed-length vector is a bottleneck in improving the performance of this basic encoder-decoder architecture."*

「我們推測，使用固定長度的向量，是改進這個基本 encoder-decoder 架構的瓶頸。」——注意這句話的措辭：他們把瓶頸**精確地指認在那個固定長度向量上**，而不是 RNN、不是訓練、不是資料。這是一個外科手術式的診斷，而下一章的注意力就是對症的刀。

## 瓶頸的形狀：為什麼是「擠爆」而不是「不夠聰明」

工程師看到「長句翻不好」，第一反應通常是「網路不夠大／訓練不夠久」。這裡要點破一個直覺陷阱：**這個瓶頸是架構性的，不是規模性的**。

把 c 想成你系統裡那個固定大小的緩衝。你把網路加大（隱藏狀態從 1000 維變 2000 維）確實能延後爆掉的時間點，就像你把緩衝從 256 bytes 加到 512 bytes——但只要輸入長度可以無限長，而容器是常數，那條「資訊量 > 容器容量」的線**遲早會被跨過**。加大只是把那條線往右推，沒有消滅它。

更糟的是，這個瓶頸有兩個你很熟悉的失效特徵：

**一、損失是默默的、漸進的。** 它不會在 c 裝滿時拋一個 `BufferOverflowException`。它只是讓譯文「越往後越漂」——句子前半翻得好好的，後半開始張冠李戴、漏譯、自我重複。這正是你最怕的那種 bug：沒有 error log，只有「輸出不太對」。

**二、最舊的資訊最容易丟。** encoder 是逐步更新的，越早讀進來的字（句首的 *the animal*），被後續更新覆寫的次數越多。Sutskever 論文裡有一個著名的工程小手段可以佐證這件事：他們發現**把來源句的字序整個反轉**（但目標句不反轉）能顯著提升 BLEU。為什麼？因為反轉後，來源句句首的字離 decoder 開始生成的時間點更近，c 裡保留它的痕跡更新鮮——等於是承認「離 c 越遠的資訊損失越大」，然後用一個 hack 把最重要的部分挪近一點。這個 hack 有效，恰恰證明瓶頸是真的。

我認為這是本章最該被你記住的一頁：**一個能靠「反轉輸入順序」這種小聰明就明顯改善的系統，等於在告訴你它的資訊瓶頸卡在哪裡。** 注意力之後不需要反轉輸入，因為它根本不靠那個唯一的 c——它讓 decoder 每生成一個字，都能回頭直接看 encoder 的每一個隱藏狀態。這是 ch03 的事，但動機在這裡就成形了。

### 回扣三步骨架：seq2seq 缺了哪一步

把全書的三步骨架（打分 → 正規化 → 加權混合）拿來對照固定向量 seq2seq，你會發現一件很乾淨的事：**它連第一步都還沒有**。

```text
  步驟        三步骨架（注意力，ch03 起）            固定向量 seq2seq（本章，注意力之前）
  ──────────  ────────────────────────────────────  ────────────────────────────────────
  1. 打分      decoder 對 encoder 每個位置算          沒有打分：decoder 看不到各位置，
              相關性分數                             只看到一個壓好的 c
  2. 正規化    softmax 成一排和為 1 的權重            沒有正規化：沒有權重可言
  3. 加權混合  用權重對各位置 value 加權平均          沒有混合：context = 最後一個
                                                    隱藏狀態，固定不變
```

固定向量 seq2seq 裡，decoder 在生成第 1 個字和第 40 個字時，能依靠的來源資訊**是同一個 c**，完全沒有「這個目標字該特別看來源句哪裡」的機制。它沒有「打分」這個動作，所以也談不上正規化和混合。三步骨架的第一步——「query 對每個 key 算相關性分數」——在這個世界裡根本不存在；decoder 沒有 query，encoder 也沒有把各位置攤開來當 key/value，只交出了一個塌縮成一團的摘要。

所以你可以把整本書的故事，從這一章開始這樣講：**注意力做的第一件事，就是把那個塌成一團的 c 重新攤開成「一排可以被打分、被加權的東西」，然後補上三步骨架。** 固定向量瓶頸是「題目」，三步骨架是「答案」。

## RNN 的第二個痛：循序依賴（一句帶過，但記住它）

固定向量是 seq2seq 的第一個痛，也是本章主角。但 RNN 還有第二個痛，它在本書後面（ch07 講自注意力「全部位置一次算」時）會變成關鍵對照，所以這裡點到、不展開。

RNN 的隱藏狀態是**循序更新**的：h_t 要等 h_{t-1} 算完才能算，h_{t-1} 又要等 h_{t-2}……這是一條**必須一步接一步走的依賴鏈**。用你的話講：

- 這就是一條**沒法平行化的 pipeline**。前一個 stage 沒跑完，下一個 stage 動不了。你有 64 核也沒用，因為第 50 步在等第 49 步。
- 序列越長，這條鏈越長，訓練時想用 GPU 一次吃下整批的平行算力就越使不上。

（這條依賴鏈同時也是「長程依賴衰減」的來源——梯度沿著長鏈反覆相乘容易爆炸或消失，這是 LSTM/GRU 的閘控想緩解的問題；但反向傳播的數學不是本書主題，需要時見《馴服無限》談梯度。）

記住這個對照：**注意力後來最大的工程紅利之一，就是把「循序依賴」換成「所有位置一次平行算」**。固定向量瓶頸是「翻得準不準」的問題，循序依賴是「算得快不快、能不能平行」的問題——注意力一箭雙鵰，兩個都治。但這要到 ch07 才看得清楚，現在你只要把這個痛**記在帳上**。

## 一條更早的暗線：Graves 的可微分記憶（伏筆 ch18）

注意力不是 2014 年憑空蹦出來的。在 Bahdanau 把它正式用進翻譯之前，有一條更早、更抽象的線，值得在這裡埋個伏筆。

Alex Graves 在 2013–2014 年做的工作，已經有了「可微分注意力」的雛形：

- **Graves 2013**「Generating Sequences With RNNs」（arXiv 1308.0850）：在手寫合成裡用一個高斯核混合的可微分「位址」機制，讓網路自己決定下一筆畫該對齊到輸入文字的哪個位置——這已經是「軟性地指向某處」的雛形。
- **Graves、Wayne、Danihelka 2014**「Neural Turing Machines」（arXiv 1410.5401）：把一個神經網路控制器接上一塊外部記憶矩陣，用**可微分**的讀寫去存取它。它的「content-based addressing（基於內容的定址）」（2026-06 查證）是這樣運作的：控制器發出一個 key 向量，跟記憶體裡每一格做**相似度比對**，產生一排**正規化後的權重**，然後用這排權重去做一次「模糊的（blurry）」讀取——不是只讀一格，而是按權重讀**所有格的加權混合**。

把上一段慢讀一遍，你會發現它的形狀**就是三步骨架**：key 跟每格比相似度（打分）→ 正規化成權重（正規化）→ 按權重混合所有格（加權混合）。NTM 是用這套機制去模擬「可微分版的 Turing 機」，能學會 copy、sort、associative recall（聯想式回憶）這類小演算法。

所以這條暗線告訴你兩件事：

1. **「依相似度加權混合一塊記憶」這個想法，在被用來治翻譯瓶頸之前，已經先被用來造可微分的記憶體了。** 同一個機制，兩個動機。
2. 這個「軟性內容定址的聯想記憶」視角，會在 ch18 整個翻回來——到時你會看到：注意力跟現代 Hopfield 網路（一種聯想記憶）是等價的。NTM 就是那條線的源頭，先記著。

關於「誰最早發明 attention」，誠實的版本是這樣（照 landscape 的保守寫法，不選邊）：**Bahdanau 等人 2014 是「attention mechanism」這個詞在神經機器翻譯裡的正式確立者**；Graves 2013/2014 更早就用了可微分的注意力雛形但沒這樣命名；Jürgen Schmidhuber 則主張他 1990 年代初的早期工作（Fast Weight Programmers）更早。這是個沒有乾淨答案的歸屬爭議，本書的立場是：把 NMT 注意力的確立歸給 Bahdanau 2014，承認 Graves 的更早雛形與 Schmidhuber 的爭議，到此為止。

## 故障模式與直覺陷阱

| 陷阱 / 失效模式 | 錯誤直覺 | 正確理解 | 怎麼自我察覺 |
|---|---|---|---|
| 把瓶頸當成「網路不夠大」 | 「長句翻不好？把隱藏維度加大就好」 | 瓶頸是**架構性**的：容器固定、資訊隨長度成長，加大只是把爆掉點往右推、沒消滅它 | 你發現「再加大也只多撐幾個字」就該懷疑是架構而非規模 |
| 以為損失會報錯 | 「資訊塞不下，總會有什麼訊號吧」 | 損失是**默默漸進**的：句子越往後越漂、漏譯、重複，沒有 error | 看到「前半好、後半爛」的不對稱品質下滑，就是固定向量在漏 |
| 以為「反轉輸入」是玄學調參 | 「反轉來源句能提升 BLEU，純屬偶然」 | 它有效**正好證明瓶頸真實存在**——把重要資訊挪近 c 能減少損失，等於承認離 c 遠的資訊會丟 | 任何「靠重排輸入順序就明顯改善」的系統，都在暴露它的資訊瓶頸位置 |
| 把固定向量瓶頸跟 O(n²) 混為一談 | 「seq2seq 也是慢在二次方吧」 | 兩回事：固定向量瓶頸是**準確度**問題（資訊被擠爆）；RNN 的痛是**循序依賴**（O(n) 但不能平行）；O(n²) 是注意力**自己**後來的代價（ch07/ch14），跟這裡無關 | 分清「翻不準」（本章）、「算不快/不能平行」（RNN 循序）、「長 context 爆記憶體」（O(n²)，後面的事） |
| 以為注意力是為了「更快」才發明 | 「注意力一開始是效能優化」 | Bahdanau 注意力一開始是為了**翻得更準**（打破固定向量瓶頸），平行化紅利是 Transformer（ch07 起）才凸顯的副產物 | 記住動機的順序：先治準確度（ch03），平行化是後來的禮物 |

最該防的那一個，是第一個：**把架構性瓶頸誤判成規模問題**。這在你的工程生涯裡會反覆出現——當一個系統「越調越邊際遞減」時，往往不是參數沒調對，而是架構本身有個固定容器卡在那裡。2014 年的人花了不少力氣加大網路、反轉輸入，最後真正解決問題的是換架構（加注意力），不是加規模。

## 紙上推演

這一章沒有矩陣可算（脊椎的數字版從 ch04 才開始）。但「固定容器裝可變資訊」這件事，用粗略的數量級估計就能讓你**用算的**感受到瓶頸，而不只是用講的。

### 練習 1：固定向量能裝多少句子？ **[15 分鐘] ★★**

假設 context 向量 c 是 1000 維，每一維用 32-bit 浮點數存（這是個極度樂觀的上限估計——實務上一維遠遠存不了 32 bits 的「有效資訊」，但我們先用這個上限論證）。

(a) c 的資訊容量上限大約是多少 bits？
(b) 假設一個英文 token 平均要用約 15 bits 才能無損表示「它是哪個詞」（詞表約 3 萬詞，log₂(30000)≈14.9）。在這個樂觀上限下，c 最多能「無損裝下」幾個 token？
(c) 真實系統遠達不到這個上限（向量是連續的、訓練出來的有損表示，且要同時編碼詞序、語法、語意關係，不只是「哪些詞」）。用這個估計，解釋為什麼「5 個字翻得好、50 個字開始崩」在數量級上是合理的。

### 練習 2：把瓶頸對到你自己的翻車現場 **[10 分鐘] ★**

回想一個你真實遇過的「固定大小容器塞可變大小資料」的 bug（固定欄位被截斷、固定緩衝溢位、雜湊碰撞、protobuf 欄位塞爆……）。用三句話寫下：(a) 容器是什麼、上限多少；(b) 資料超過時**怎麼壞的**（報錯？默默截斷？覆蓋？）；(c) 這個失效模式跟固定向量瓶頸「最舊資訊先丟、默默漸進」的哪一點最像。

### 練習 3：為什麼循序依賴讓 RNN 難平行？ **[10 分鐘] ★★**

口頭（或寫下）對另一個工程師解釋：為什麼 RNN 算一個長度 50 的序列時，就算你有一張有幾千個核心的 GPU，也沒辦法把這 50 步同時算完？再多想一步：如果有一種架構能讓「50 個位置同時算」，它必須放棄 RNN 的什麼性質？（這是 ch07 自注意力的伏筆。）

### 推演解答

**練習 1。**

(a) 1000 維 × 32 bits/維 = 32,000 bits = 4,000 bytes（約 4 KB）。這是把每一維都當成能無損存 32 bits 來算的**上限**。

(b) 32,000 bits ÷ 15 bits/token ≈ **2,133 個 token**。

乍看之下「兩千個 token 還不少嘛」，但這正是這題的陷阱所在——也是要你算一遍才會服氣的地方：(b) 算出來的 2000 多，是**只記「有哪些詞」、而且假設每一維都能無損利用 32 bits** 的天花板。真實情況有三層折扣，每一層都很狠：

- **連續、有損**：c 的每一維是訓練出來的實數，不是一個乾淨的 32-bit 存格。它能承載的「有效資訊」遠少於 32 bits——可能只有幾個 bits 的可區分度。光這一層，上限就從 2000 掉到幾十的數量級。
- **不只記詞，還要記關係**：c 不能只記「出現了哪些詞」（那是一個無序的袋子），它還得編碼**詞序、語法結構、誰修飾誰、代名詞指誰**（正是脊椎句子裡 *it* 指代的那種資訊）。這些關係資訊吃掉大量容量。
- **要支撐整段解碼**：decoder 要靠這一個 c 連續生成幾十個字，c 必須同時保留「開頭講了什麼」和「結尾講了什麼」。

(c) 把這三層折扣疊起來，c 能可靠承載的「有效句子長度」就掉到**十幾到幾十個 token** 的數量級——這跟「短句翻得好、句子一長就崩」的實測現象在數量級上完全吻合。重點不是某個精確數字，而是：**容器是常數，需求隨長度成長，兩條線一定會交叉**。交叉點之後，多出來的資訊只能被默默丟掉。

**練習 2。** 這題沒有標準答案，但好的回答會點出固定向量瓶頸最陰險的特徵：**它跟你的截斷 bug 不一樣的地方在於——你的 protobuf 截斷至少還算「乾淨地」丟掉尾巴，固定向量是把所有位置的資訊「揉成一團」後一起變模糊，連哪裡被丟了都看不出來。** 最像的那一點通常是「最舊的資訊最容易丟」（你的環狀緩衝、LRU 快取、append 到滿的 log 都有這個味道）和「默默地壞、沒有 error」。

**練習 3。** RNN 的 h_t = f(h_{t-1}, x_t)：第 t 步的隱藏狀態是第 t−1 步的**函數**。所以第 50 步在數學上必須等第 49 步算完，第 49 步等第 48 步……這是一條長度 50 的**真依賴鏈**，不是排程問題，是資料依賴問題。GPU 的幾千個核心只能拿來**平行算每一步內部的矩陣乘法**，沒辦法把「步與步之間」的順序拆開——就像你不能在前一個 migration 還沒跑完時跑下一個依賴它的 migration。

再多想一步：要讓 50 個位置**同時**算，這個架構就必須放棄「第 t 步依賴第 t−1 步的隱藏狀態」這個遞迴結構——也就是不能再讓位置之間有循序依賴。代價是：它得用**別的方式**讓每個位置知道其他位置的存在。注意力給的答案是「每個位置直接對所有位置算一次相關性」——所有位置的計算彼此獨立、可以一次平行算完。但這個「對所有位置都算一次」也正是 O(n²) 的源頭（ch07/ch14）。天下沒有白吃的午餐：用二次方的計算量，換掉了循序依賴。這個交易划不划算，是 Part IV 整個 Part 在吵的事。
