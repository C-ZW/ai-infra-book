# ch09 — 位置編碼：集合沒有順序這個洞

> **本章解決什麼問題**：Part II 我們把三步骨架（打分→正規化→加權混合）整個拆乾淨了——QKV、softmax、√d、自注意力矩陣、多頭分工。但這套機制有一個你可能還沒察覺的、致命的洞：它**看不見順序**。把 [animal, street, it] 重排成 [it, street, animal]，每個 token 拿到的注意力權重一模一樣。本章證明這件事不是 bug 而是 attention 的本質——它對輸入的處理方式是「集合」不是「序列」（嚴格的理論版留 ch19）。看清這個洞，才會懂為什麼非得「外加」一塊位置資訊不可，以及原始 Transformer 那組漂亮的正弦位置編碼到底在補什麼。RoPE 怎麼用旋轉把這件事做得更漂亮，留 ch10。

到這裡，你應該已經能把三步骨架背起來：query 對每個 key 打分（內積）、softmax 正規化成和為 1 的權重、用權重混合所有 value。ch07 還畫出了完整的 n×n 注意力矩陣，ch08 讓多顆頭各看一個子空間。一切看起來很完整。

但有一件事，前面五章從頭到尾**沒有出現過**：序列裡每個 token 的「位置」。

回頭翻 ch04 到 ch08 任何一條公式——QKᵀ、softmax、加權混合 V——裡面有任何一個項目代表「animal 在第幾個位置、it 在第幾個位置」嗎？沒有。我們算 it 對 animal 的分數時，用的是 q_it 和 k_animal 兩個向量做內積，從頭到尾**沒有一個地方告訴模型「animal 在 it 前面好幾個字」**。

這不是我偷懶漏掉了。這是 attention 機制**結構性**的缺口。本章就是要把這個缺口攤開來看清楚，然後看人們怎麼補它。

## 從你已知的出發

你天天在處理「有沒有順序」這個區別，只是沒特別命名它。

想想你手上幾種資料結構。一個 `Array`（或 `List`）：`[a, b, c]` 和 `[c, b, a]` 是**不同**的東西，順序是它身分的一部分——你能 `arr[0]`、能 `slice`、能說「第三個元素」。但一個 `Set`（集合）：`{a, b, c}` 和 `{c, b, a}` 是**同一個** set，問它「第三個元素是誰」根本沒有意義，因為 set 不記順序。同理 hash map 的 key 集合、Redis 的一堆 key、一個 `SELECT` 沒帶 `ORDER BY` 撈回來的 result set——這些本質上都是**無序的袋子**，順序要嘛不存在，要嘛不保證。

現在問你一個工程問題：**如果你有一個只吃 Set 的函數，但你需要它知道元素的先後順序，你會怎麼辦？**

你不會去改 Set 的定義（Set 就是無序的，這是它的本性）。你會做的是——**給每個元素貼一個顯式的順序欄位**。一個 `index`、一個 `seq_no`、一個 `timestamp`、一個 `created_at`。你把「順序」這個原本不存在的資訊，當成一塊額外的資料，**手動塞進每個元素裡**。之後就算這些元素被丟進無序的容器、被任意重排，你仍然能靠那個欄位把順序還原出來。

```text
  無序的袋子（Set）          貼上順序戳記後
  ──────────────          ──────────────────
  { animal,                { (animal, pos=0),
    street,        ──►       (street, pos=1),
    it }                     (it,     pos=2) }
  問「第幾個」沒意義          現在每個元素自帶位置
```

**這正是位置編碼（positional encoding）在做的事，一字不差。** 等一下你會看到，self-attention 處理輸入的方式，本質上就是「把序列當成一個無序的袋子」。它需要順序資訊，但它的運算裡天生沒有順序的概念。所以工程上的解法跟你會做的一模一樣：**在每個 token 進入注意力之前，給它的向量蓋上一個「位置戳記」**，把「它排在第幾個」這件事，編碼成數字、塞進向量裡。

差別只在於戳記長什麼樣。你貼的是一個整數 `pos=2`；模型不能直接吃整數（後面會說為什麼），所以它貼的是一個跟 embedding 同維度的**向量**——而怎麼設計這個向量，就是本章後半的主題，也是接到 ch10 RoPE 那條漂亮路線的起點。

## 那個洞：self-attention 看不見順序

先把洞挖出來看。我用脊椎那個 it，做一次最直接的實驗：**把句子裡 token 的順序打亂，看 it 拿到的注意力權重會不會變。**

回收 ch06 宣告的全書正典脊椎數字（縮放後）。三個 token 的 key、value，以及 it 的 query：

```text
  q_it = (1, 1, 1, 1)

  k_animal = (2,  1, 1, 0)      v_animal = (1, 0)
  k_street = (0,  1, 1, 0)      v_street = (0, 1)
  k_it     = (1, -1, 0, 0)      v_it     = (1, 1)

  d = 4,  √d = 2
```

**原本的順序 [animal, street, it]。** it 對三個 key 打分（內積），除以 √d=2，softmax：

```text
  打分（內積 q_it·k）        ÷√d=2      softmax
  ─────────────────         ──────     ──────────────────
  q_it·k_animal = 4    →     2.0    →   0.665   ← 看 animal 最重
  q_it·k_street = 2    →     1.0    →   0.245
  q_it·k_it     = 0    →     0.0    →   0.090
```

（softmax 算法：e²=7.389、e¹=2.718、e⁰=1，和 11.107；7.389/11.107=0.665、2.718/11.107=0.245、1/11.107=0.090。這是 ch06 的正典那一組，全書通用。）

加權混合 value，輸出 ≈ (0.755, 0.335)，偏向 v_animal=(1,0) 的方向。一切正常，這就是前幾章那個結果。

**現在把順序倒過來，排成 [it, street, animal]。** 注意：我只是改了三個 token 排列的先後，**每個 token 自己的向量完全沒動**——animal 還是那個 k_animal=(2,1,1,0)、v_animal=(1,0)，只是現在它排在第三個。it 的 query 也沒變，還是 (1,1,1,1)。重算一次：

```text
  新順序   打分              ÷√d=2     softmax
  ──────   ───────────       ─────     ───────
  it       q_it·k_it=0    →   0.0   →   0.090
  street   q_it·k_street=2 →  1.0   →   0.245
  animal   q_it·k_animal=4 →  2.0   →   0.665   ← 還是看 animal 最重
```

看出來了嗎？權重整排只是**跟著 token 一起換了位子**：animal 仍然拿 0.665、street 仍然 0.245、it 自己仍然 0.090。it 給 **animal** 的權重，在兩種順序下**都是 0.665，一模一樣**。加權混合出來的輸出，**也是同一個 (0.755, 0.335)**——位元級相同（我用 Python 對過，兩個輸出完全相等）。

換句話說：**把 animal 放在 it 前面，還是放在 it 後面，attention 給的答案完全沒差。** 模型對「animal 到底排在 it 的左邊還是右邊」這件事，**毫無感覺**。

這就是那個洞。讓我把它講到底為什麼會這樣。

### 為什麼必然如此：分數只看「向量配對」，不看「位置」

it 對 animal 的分數是 q_it · k_animal。這個內積只跟**兩個向量本身**有關——q_it 的四個分量、k_animal 的四個分量，乘一乘加起來。它**根本沒有一個輸入端口**接收「animal 排第幾」這個資訊。你把 animal 移到哪裡，k_animal 這個向量都不變，所以 q_it · k_animal 永遠是同一個數。

softmax 也救不了。softmax 吃一排分數、吐一排和為 1 的權重，它對「分數的順序」是無所謂的——`softmax([4,2,0])` 和 `softmax([0,2,4])` 出來的數值集合是一樣的，只是跟著重排。最後的加權混合，是「權重 × 對應的 value」相加，加法本身可交換（`a+b = b+a`），所以混合的結果跟你用什麼順序把它們加起來無關。

三步骨架**每一步**都對順序無感：

```text
  1. 打分     score(q, kⱼ) = q·kⱼ    只看向量配對，與 j 排第幾無關
  2. 正規化   softmax(...)          重排分數→重排權重，集合不變
  3. 加權混合 Σⱼ αⱼ vⱼ              加法可交換，順序無所謂
```

三步全都過不了「順序」這一關，所以整個 attention 過不了。這不是哪一步沒寫好，是這個機制從骨子裡就是這樣設計的。

### 給這個性質一個正確的名字：置換等變

這個性質有一個精確的名字，本書全程用它：**置換等變（permutation equivariant）**。

「等變」的意思是：**你怎麼重排輸入，輸出就怎麼跟著重排，內容不變。** 用上面的實驗講：我把輸入從 [animal, street, it] 重排成 [it, street, animal]，輸出的那一排向量也就**對應地**重排了——animal 那個位置算出來的輸出向量，跟著 animal 跑到第三個位置去；但它的**數值**沒變。沒有任何資訊因為重排而改變或消失，順序的調換被「原封不動地搬過去」了。

寫成式子（直覺版，嚴格證明在 ch19 用置換矩陣 P 寫成 `SelfAttention(PX) = P · SelfAttention(X)`）：

```text
  把輸入序列重排 → 輸出序列就照同一種方式重排，每個位置算出的向量不變
  「等變」：排列作用在輸入上 = 排列作用在輸出上
```

有一個更狠的特例：如果你不在乎輸出的順序、只把它們聚合起來（比如全部相加、或取某一個固定 query 對它們的混合），那連重排都看不出來了——這叫**置換不變（permutation invariant）**。上面 it 那個 query 的輸出 (0.755, 0.335) 在兩種順序下完全相同，就是不變的展現：因為 it 這個 query 不管 animal 排哪，混合出來都一樣。

> **這裡反直覺在哪**：你大概一直以為 Transformer「理解語言」，而語言顯然是有順序的——"dog bites man" 和 "man bites dog" 意思天差地遠。但你剛剛親手算出來：**裸的 self-attention 分不出這兩句**。把 token 重排，它眼皮都不眨。所以 Transformer 能讀懂語序，**不是因為 attention 本身懂順序**——attention 一點都不懂——而是因為**有人在 token 進 attention 之前，先偷偷把位置資訊塞了進去**。這塊「塞進去」的東西，就是這一章的主角。少了它，Transformer 就是一台讀「詞袋（bag of words）」的機器。

為什麼 attention 會被設計成這樣？因為 ch07 講過，self-attention 是「每個 token 對所有 token 算一遍」的全連接運算——它把序列當成一個**所有元素彼此平等、沒有先後**的集合來處理。這正是它能完全平行、一次抓全句關係（對照 RNN 必須一步一步循序）的代價：它丟掉了 RNN 天生自帶的「我是一步一步讀過來的，所以我知道誰先誰後」。RNN 的順序資訊藏在「處理的時序」裡，attention 沒有時序，於是順序資訊也就沒地方藏了。

attention 是集合運算這件事，理論上的嚴格版（Set Transformer、置換等變的形式證明）留到 ch19——那時你會看到本章這個「洞」其實是 attention 深層身分的一個面向，而不只是個工程小麻煩。本章只需要你抓住結論：**裸 attention 看不見順序，順序必須外加。**

## 怎麼補：把「位置」變成一個向量加進去

洞看清楚了，補法的思路就很直接，跟你給 Set 元素貼 `seq_no` 一模一樣：**在 token 的 embedding 進入 attention 之前，給它加上一個代表位置的向量。**

注意動詞是「**加**」——逐元素相加。設 token 在第 pos 個位置，它的詞嵌入是 x（一個 d 維向量），位置編碼是 PE(pos)（也是一個 d 維向量），那麼真正送進第一層 attention 的，是：

```text
  輸入 = embedding(token) + PE(pos)
         ─────────────────   ───────
         「我是什麼字」        「我排第幾」
         （內容，學出來的）    （位置戳記）
```

這樣一來，原本只攜帶「我是 animal」的向量，現在同時攜帶了「我是 animal，而且我排在第 0 個位置」。等它去算內積、被別人打分時，位置資訊就一起參與了運算——animal 排在 it 前面還是後面，現在會體現在它的向量上，分數就會不同。洞補上了。

馬上有兩個合理的疑問，先擋掉：

**疑問一：為什麼是「加」，不是「接（concat）」？** 直覺上，把位置資訊接在 embedding 後面（拼成更長的向量）似乎更乾淨——內容歸內容、位置歸位置，井水不犯河水。確實有人這樣做。但原始 Transformer 選擇「加」，務實的理由是：加法不增加維度（concat 會讓向量變長、後面所有矩陣都得跟著變大、參數變多），而且模型有足夠的容量在訓練時自己學會「從這個混在一起的向量裡，把內容和位置該怎麼用拆開來用」。這是「夠用就好」的工程取捨，不是什麼深刻定理。

**疑問二：直接塞一個整數 pos=0、1、2 不就好了？** 不行，而且這個「為什麼不行」很值得想清楚——它直接決定了位置編碼該長什麼樣：

- **數值範圍會爆**。pos 可以從 0 跑到幾千、幾萬。把一個 0~50000 的大整數加進每個分量都是 ~O(1) 的 embedding 向量裡，這個位置項會直接**淹掉**內容項——模型還沒看清你是什麼字，就先被「你排第 38274 個」這個巨大的數字壓垮了。
- **不好泛化到沒見過的長度**。如果訓練時只見過 pos 到 512，模型對 pos=5000 這個從沒出現過的巨大整數會手足無措。
- **「絕對位置」不如「相對關係」有用**。對語言來說，真正重要的常常不是「it 在第 11 個字」這種絕對座標，而是「it 距離 animal 有幾步」這種**相對**關係。一個光禿禿的整數很難讓模型輕鬆算出相對距離。

所以一個好的位置編碼，至少要：**數值有界**（不會淹掉內容）、**每個位置唯一**（不能兩個位置撞成一樣，否則模型分不出）、**能讓模型方便地推出相對位置**、**最好能外推到比訓練更長的序列**。原始 Transformer 的正弦位置編碼，就是衝著這幾條設計的。

## 正弦位置編碼：用一堆不同頻率的波，給每個位置一個指紋

原始 Transformer（Vaswani et al. 2017，arXiv 1706.03762，§3.5；2026-06 對照 landscape）的做法，是用 sin 和 cos 函數，**直接算出** PE(pos)——不是學出來的，是一條固定公式（這點和下一節要對照的「可學習位置」很不同）。公式長這樣：

```text
  PE(pos, 2i)   = sin( pos / 10000^(2i / d_model) )    ← 偶數維用 sin
  PE(pos, 2i+1) = cos( pos / 10000^(2i / d_model) )    ← 奇數維用 cos
```

符號拆開：`pos` 是 token 在序列裡的位置（0, 1, 2, …）；`d_model` 是向量維度（原始論文 512）；`i` 跑 `0` 到 `d_model/2 - 1`，把維度兩兩配對（第 0、1 維一對，第 2、3 維一對……），每一對由同一個頻率的一組 sin/cos 組成。

公式裡那個 `10000^(2i/d_model)` 是**頻率控制鈕**。當 i=0，分母是 `10000^0 = 1`，頻率最高（波長最短，sin/cos 變得最快）；當 i 變大，分母指數成長到 10000，頻率最低（波長最長，sin/cos 變得最慢）。也就是說，**不同的維度對（i）用不同的波長去「看」位置**，波長從 2π 一路幾何級數地拉長到 10000·2π（2026-06 查證，與原論文一致）。

這招的直覺，借用你熟的東西：**這就是二進位**。一個整數的二進位表示，最低位（bit 0）每加 1 就翻一次（高頻），次低位每 2 次翻一次，更高位翻得更慢（低頻）——用一排「不同翻動速度的位元」就能唯一表示任何整數。正弦編碼是它的連續、平滑版：用一排「不同擺動速度的正弦波」，給每個位置一個獨一無二的**指紋**。低頻維度像高位元，粗略地標出「大概在序列哪一段」；高頻維度像低位元，精細地標出「在這一小段裡的確切位置」。

來算給你看。為了能手算，把 d_model 縮到 4（真實是 512，但機制一樣），這樣只有兩對維度：i=0（分母 10000⁰=1，高頻）和 i=1（分母 10000^(2/4)=10000^0.5=100，低頻）。算前幾個位置：

```text
  維度索引：     0          1          2           3
  公式：      sin(pos/1)  cos(pos/1)  sin(pos/100) cos(pos/100)
            （i=0 高頻）          （i=1 低頻）

  pos=0:    sin(0)=0.0000  cos(0)=1.0000  sin(0)=0.0000   cos(0)=1.0000
  pos=1:    sin(1)=0.8415  cos(1)=0.5403  sin(0.01)=0.0100 cos(0.01)=1.0000
  pos=2:    sin(2)=0.9093  cos(2)=−0.4161 sin(0.02)=0.0200 cos(0.02)=0.9998
  pos=3:    sin(3)=0.1411  cos(3)=−0.9900 sin(0.03)=0.0300 cos(0.03)=0.9996
```

（這幾個值我用 Python 算過：sin(1)≈0.8415、cos(1)≈0.5403、sin(2)≈0.9093、cos(2)≈−0.4161、sin(3)≈0.1411、cos(3)≈−0.9900；低頻那對因為 pos/100 還很小，sin 約等於 pos/100、cos 約等於 1，幾乎沒動。）

看這張表，幾件事一目了然：

- **每個位置的四個數合起來是獨一無二的**——沒有兩列長得一樣，所以模型能靠這個向量分辨位置。這滿足「每個位置唯一」。
- **數值全部落在 [−1, 1]**，因為 sin/cos 本來就有界。它不會像光禿禿的整數那樣淹掉 embedding。這滿足「數值有界」。
- **高頻那對（維度 0、1）在相鄰位置就明顯變化**（pos 0→1→2，sin 從 0 跳到 0.84 再到 0.91），負責精細定位；**低頻那對（維度 2、3）幾乎不動**（都還在 0.0x 和 0.99~1.00），它要到 pos 很大時才會明顯轉起來，負責粗略定位。高低頻分工，正是「二進位」那個直覺。

把這四個數加到 token 的 embedding 上，這個 token 就帶著它的位置指紋進 attention 了。

### 為什麼用 sin/cos：相對位置變成一個線性關係

正弦編碼最被稱道的性質——也是原論文選它而非可學習版本的明說理由之一——是：**對任何固定的偏移量 k，PE(pos+k) 可以寫成 PE(pos) 的一個線性函數。**

意思是：「位置 pos」和「位置 pos+k」這兩個編碼之間的關係，**只跟偏移量 k 有關，跟 pos 本身是多少無關**。位置 5 到位置 8（差 3）的編碼變換，和位置 100 到位置 103（差 3）的編碼變換，是**同一個**線性變換。這正是「相對位置」的數學體現——模型只要學會「差 k 步對應哪個變換」，就能在序列任何地方判斷兩個 token 相差幾步。

為什麼？因為 sin/cos 有那組你高中背過的和角公式（這也是《圓的影子》ch05 旋轉的核心）：

```text
  sin(θ + φ) = sin θ · cos φ + cos θ · sin φ
  cos(θ + φ) = cos θ · cos φ − sin θ · sin φ
```

把 θ = pos·ω（ω 是某個頻率）、φ = k·ω 代進去，你會發現 PE(pos+k) 的那一對 (sin, cos) 分量，恰好是 PE(pos) 那一對分量乘上一個**只由 k·ω 決定的 2×2 矩陣**——而那個矩陣，正是一個**旋轉矩陣**。換句話說：在每一對維度構成的平面上，「位置往前 k 步」=「把那個 (sin, cos) 向量旋轉一個固定角度 k·ω」。旋轉的角度只看 k，不看你從哪個 pos 出發。

我用上面 d=4 的高頻那對（ω=1）數值驗一次，取 pos=1、k=2，看 PE(3) 是不是 PE(1) 的線性（旋轉）變換：

```text
  和角公式：
  sin(3) = sin(1+2) = sin1·cos2 + cos1·sin2
                    = 0.8415·(−0.4161) + 0.5403·0.9093
                    = −0.3502 + 0.4913 = 0.1411  ✓（= 上表 sin(3)）

  cos(3) = cos(1+2) = cos1·cos2 − sin1·sin2
                    = 0.5403·(−0.4161) − 0.8415·0.9093
                    = −0.2248 − 0.7652 = −0.9900  ✓（= 上表 cos(3)）
```

對得上。PE(3) 確實是 PE(1) 經過一個「轉 2·ω 角度」的線性變換得來的，而這個變換只認 k=2，不認起點 pos=1。

> 把這句話記牢，因為它是 ch10 的種子：**「位置往前 k 步」在正弦編碼裡 = 「旋轉一個只由 k 決定的角度」。** 正弦編碼是把這件事**加在 embedding 上**間接達成的；RoPE（ch10）的洞見是——既然相對位置的本質是旋轉，那何不**直接旋轉 query 和 key**，讓內積天生只剩相對角度差？那是全書最漂亮的跨書接點（接《圓的影子》的旋轉與複數），這裡先埋下伏筆。

順帶補一條原論文宣稱的好處：因為 sin/cos 是對所有 pos（包括訓練時沒見過的大 pos）都算得出來的確定函數，正弦編碼**理論上可以外推到比訓練更長的序列**——你給 pos=5000，公式照樣吐一個合法的向量。（實務上裸正弦的外推效果有限，這條路通往 ALiBi、RoPE、Position Interpolation/YaRN 等後續工作；ch10 接手。本章只說原論文當初的動機。）

## 另一條路：可學習的絕對位置（BERT / GPT-2）

正弦編碼是「算出來的」（一條固定公式，無參數）。但你應該已經在想：既然其他東西（embedding、投影矩陣）全是學出來的，**位置為什麼不也讓模型自己學**？

確實有，而且是另一條主流路線——**可學習的絕對位置嵌入（learned absolute positional embedding）**。做法簡單到近乎偷懶：再開一張查找表，叫**位置嵌入表**，第 0 個位置一個可訓練向量、第 1 個位置一個、第 2 個……每個位置一行，跟詞嵌入表長得一模一樣，只是查的索引從「token id」換成「位置 pos」。訓練時這些向量跟著梯度一起被調出來。用的時候照樣是「embedding(token) + 位置嵌入(pos)」加進去。

```text
  詞嵌入表（查 token id）        位置嵌入表（查 pos，可學習）
  ─────────────────             ──────────────────────────
  "animal" → [ ... ]            pos 0 → [ ... ]  ← 訓練學出
  "street" → [ ... ]            pos 1 → [ ... ]  ← 訓練學出
  "it"     → [ ... ]            pos 2 → [ ... ]  ← 訓練學出
                                  ...
```

**BERT**（Devlin et al. 2018，arXiv 1810.04805）和 **GPT-2**（2019）走的都是這條（2026-06 對照 landscape §3 與獨立查證確認）。它的優缺點是清楚的工程取捨：

| | 正弦（固定公式） | 可學習絕對位置 |
|---|---|---|
| 參數 | 0（無參數） | 每個位置一個向量（佔參數） |
| 哪一步動了 | 都在「打分」前，改 query/key 的向量 | 同左 |
| 相對位置 | 天生線性可推（和角公式） | 不天生，要模型自己從資料學 |
| 外推到更長序列 | 公式照算（理論上可，實務有限） | **不行**——pos=5000 那一行根本沒被訓練過，是隨機初始值 |
| 採用 | 原始 Transformer | BERT、GPT-2、GPT-3 |

最值得記的是**最後一條**：可學習絕對位置有一個硬限制——**它只認得訓練時見過的位置**。位置嵌入表只開到訓練的最大長度（比如 512），第 513 行根本不存在；就算硬開出來，那一行從沒被任何資料訓練過，是個隨機向量，模型對它一無所知。所以用可學習絕對位置的模型，context 長度是**寫死的上限**，想變長只能重訓。正弦編碼至少在公式上沒這個牆（雖然實務外推也不理想）。這個「外推」痛點，正是 ch10 那一連串工作（RoPE、ALiBi、YaRN）要解的核心。

有意思的補充（2026-06 查證）：後來有研究發現，**可學習絕對位置在訓練後，常常自己長成類似正弦波的結構**——模型沒人教，居然自己摸索出了「不同頻率的週期波」這個好點子。這從側面說明正弦那組設計不是隨便挑的，而是踩中了某種「位置該怎麼編碼」的自然解。

不論哪條路，回到三步骨架看它們**動了哪一步**：位置編碼**不改 softmax、不改加權混合**，它在三步**之前**動手——改的是進入「打分」的那些 query/key 向量本身（給它們蓋上位置戳記），讓內積出來的分數開始攜帶位置資訊。換句話說，位置編碼是「打分」這一步的**輸入預處理**：它不換打分的算法，它換的是被打分的料。記住這個定位，ch10 的 RoPE 會把它推到更乾淨的地方——RoPE 同樣不碰 softmax 和混合，但它動 query/key 的方式從「相加」升級成「旋轉」。

## 故障模式與直覺陷阱

| 陷阱 | 錯誤直覺 | 會在哪一步把你帶溝裡 | 怎麼自我察覺 |
|---|---|---|---|
| 「Transformer 當然懂順序」 | 以為 attention 本身理解語序 | 你會以為拿掉位置編碼 Transformer 還能讀懂句子——其實它退化成詞袋，分不出 "dog bites man" 和 "man bites dog" | 自問：公式裡哪一項代表位置？（裸 attention 裡：沒有。） |
| 「重排輸入只是換個寫法」 | 以為重排 token 不影響本來就無序的東西 | 對「集合」確實無所謂，但對**語言**致命——你會低估位置編碼的必要性 | 自問：把 [animal, it] 對調，模型的輸出該不該變？該變（語意變了），但裸 attention 不變 |
| 「位置編碼是某種特殊層」 | 以為它是個複雜的可訓練模組 | 過度神化它 | 正弦版根本沒有參數，就是一條公式算出來加上去；可學習版也只是一張查找表 |
| 「直接加個位置整數就好」 | 以為 pos=0,1,2… 塞進去即可 | 大整數會淹掉 embedding、不能外推、難表達相對位置 | 自問：pos=38274 加進一個 ~O(1) 的向量會發生什麼？（內容被壓垮。） |
| 「正弦編碼能無限外推」 | 把原論文「理論上可外推」當成「實務上沒問題」 | 你會以為長 context 不需要額外處理——實務上裸正弦外推效果有限，這正是 RoPE/YaRN 存在的原因 | 區分「公式對任意 pos 都算得出」（真）vs「外推後注意力還準」（不一定，見 ch10） |

**最值得單獨拎出來的失效模式：拿掉位置編碼會發生什麼。** 這不是假設題——它是理解這一章必要性的試金石。如果一個 self-attention 模型完全不加任何位置資訊，它就是一台**詞袋（bag-of-words）機器**：給它 "the cat sat on the mat" 和把這六個字隨機打亂的任何一種排列，它的內部表示和輸出**完全相同**（置換不變）。它能告訴你「這句話裡有 cat、sat、mat」，但永遠告訴不了你「誰坐在誰上面」。對情感分類這種「有哪些詞」就八九不離十的任務，詞袋也許還能矇對；對任何依賴語序的任務（翻譯、推理、指代消解——正是脊椎那個 it→animal），它會徹底失敗。位置編碼不是錦上添花，它是讓 attention 從「讀詞袋」升級成「讀句子」的那塊**必要拼圖**。

（有一個 2026 的細節值得標時點、但別過度推論：研究發現純 decoder、配 causal mask 的自迴歸模型，即使**不**顯式加位置編碼，也能從「只能看左邊」這個遮罩結構裡**間接學到**一些位置資訊——因為遮罩本身打破了置換對稱性。這是個微妙的特例，不推翻本章主結論：encoder（雙向、無遮罩）一定要外加位置，主流 decoder 也都還是配 RoPE。本章講原理，這個特例知道即可，2026-06。）

## 紙上推演

### 推演題

**第 1 題（置換不變，脊椎，★，15 分鐘）**
用脊椎正典數字（q_it=(1,1,1,1)；k_animal=(2,1,1,0)、k_street=(0,1,1,0)、k_it=(1,−1,0,0)；√d=2；v_animal=(1,0)、v_street=(0,1)、v_it=(1,1)），把序列從 [animal, street, it] 重排成 [street, it, animal]，**重算** it 對三個 token 的注意力權重與最終輸出。它們會變嗎？用三步骨架解釋為什麼。

**第 2 題（手算正弦位置編碼，★★，20 分鐘）**
用 d_model=4 的正弦公式（i=0 分母=1、i=1 分母=100），算出 pos=4 和 pos=5 的四維位置編碼向量。它們落在什麼數值範圍？哪兩個維度變化快、哪兩個慢，為什麼？

**第 3 題（相對位置＝線性關係，★★★，20 分鐘）**
用和角公式，驗證 d_model=4 高頻那對維度（ω=1）裡，PE(pos=4) 可以由 PE(pos=2) 經過一個「只由偏移 k=2 決定」的線性變換得到。寫出那個 2×2 變換矩陣，並說明為什麼它「只認 k、不認起點」這件事就是「相對位置」。

**第 4 題（口頭，★，10 分鐘）**
向另一個工程師解釋：「為什麼直接把位置的整數 0,1,2,… 加進 embedding 行不通？」列三個理由；再解釋這跟你給一個 Set 元素貼 `seq_no` 是同一回事。

---

### 推演解答

**第 1 題。** 先算 [street, it, animal] 這個順序下的分數。注意每個 token 的向量沒變，只是排序變了：

```text
  新順序   打分（內積）         ÷√d=2    softmax
  ──────   ────────────        ─────    ──────
  street   q_it·k_street = 2 →  1.0  →   0.245
  it       q_it·k_it     = 0 →  0.0  →   0.090
  animal   q_it·k_animal = 4 →  2.0  →   0.665
```

（softmax 同一組數：e²=7.389、e¹=2.718、e⁰=1，和 11.107；0.245、0.090、0.665。）

加權混合：`0.245·(0,1) + 0.090·(1,1) + 0.665·(1,0) = (0.090+0.665, 0.245+0.090) = (0.755, 0.335)`。

**權重沒變、輸出沒變。** it 給 animal 的權重還是 0.665、給 street 還是 0.245、給自己還是 0.090；輸出還是 (0.755, 0.335)，和原本 [animal, street, it] 的結果**位元級相同**。

三步骨架解釋：打分只看 q_it·k 的向量配對，與 token 排第幾無關（k_animal 不管放哪都是同一個向量，分數恆為 4）；softmax 重排分數→重排權重，集合不變；加權混合是加法、可交換。三步全對順序無感，所以整個 attention 置換等變——這就是非加位置不可的洞。

**第 2 題。** d_model=4，i=0 分母=1（高頻）、i=1 分母=100（低頻）：

```text
  pos=4:  sin(4)=−0.7568   cos(4)=−0.6536   sin(0.04)=0.0400   cos(0.04)=0.9992
  pos=5:  sin(5)=−0.9589   cos(5)= 0.2837   sin(0.05)=0.0500   cos(0.05)=0.9988
```

（sin(4)≈−0.7568、cos(4)≈−0.6536、sin(5)≈−0.9589、cos(5)≈0.2837，皆 Python 複核。）所有值落在 [−1, 1]——這是 sin/cos 的天性，保證位置項不會淹掉內容項。**前兩維（i=0, 頻率高）變化快**：pos 4→5 時 sin 從 −0.76 跳到 −0.96、cos 從 −0.65 跳到 +0.28，劇烈擺動，負責精細定位。**後兩維（i=1, 頻率低）幾乎不動**：sin 從 0.040 到 0.050、cos 都還在 0.999，因為 pos/100 還很小，要 pos 大上百倍它才明顯轉起來，負責粗略定位。高低頻分工，正是「二進位的高位低位」那個直覺。

**第 3 題。** 取高頻對 ω=1，PE 那對分量是 (sin(pos), cos(pos))。要驗 PE(4) 由 PE(2) 經偏移 k=2 的線性變換得到。和角公式（θ=2、φ=2）：

```text
  sin(4) = sin(2+2) = sin2·cos2 + cos2·sin2 = 2·sin2·cos2 ... 
  用一般式：
  [ sin(pos+k) ]   [ cos k    sin k ] [ sin(pos) ]
  [ cos(pos+k) ] = [ −sin k   cos k ] [ cos(pos) ]

  代 k=2：cos2=−0.4161、sin2=0.9093
  [ sin4 ]   [ −0.4161   0.9093 ] [ sin2 ]   [ −0.4161   0.9093 ] [  0.9093 ]
  [ cos4 ] = [ −0.9093  −0.4161 ] [ cos2 ] = [ −0.9093  −0.4161 ] [ −0.4161 ]

  驗 sin4：−0.4161·0.9093 + 0.9093·(−0.4161) = −0.3784 − 0.3784 = −0.7568 ✓
  驗 cos4：−0.9093·0.9093 + (−0.4161)·(−0.4161) = −0.8268 + 0.1731 = −0.6537 ✓（≈cos4=−0.6536，四捨五入）
```

那個 2×2 矩陣 `[[cos2, sin2],[−sin2, cos2]]` 就是「旋轉 2 弧度」的旋轉矩陣（《圓的影子》ch05 的 R(θ)）。它**只含 k=2，完全沒有 pos**。所以無論你從 pos=2 還是 pos=200 出發，「往前 2 步」都是乘同一個矩陣——位置差 2 對應的變換永遠相同。這正是「相對位置」：模型不必記住每個絕對座標，只要學會「差 k 步＝乘某個固定矩陣」，就能在序列任何地方判斷相對距離。這個「相對位置＝旋轉」的觀察，就是 ch10 RoPE 的起跑線。

**第 4 題。** 三個理由：(1) **數值會爆**——pos 可達數萬，把大整數加進每個分量都 ~O(1) 的 embedding，位置項直接淹掉內容項，模型還沒看清你是什麼字就被「你排第幾」壓垮。(2) **不能外推**——訓練只見過 pos 到某上限，pos=5000 這個沒見過的巨大整數模型無從處理。(3) **難表達相對位置**——語言裡「差幾步」常比「絕對第幾個」重要，一個光禿禿的整數很難讓模型輕鬆推出相對距離（正弦編碼用和角公式天生解決這條，見第 3 題）。和 Set 貼 `seq_no` 同一回事：Set 本性無序、attention 本性置換等變，兩者都看不見順序；要表達順序，唯一辦法是給每個元素**外加一個顯式的位置欄位**——你貼整數 `seq_no`，模型貼一個有界、唯一、能推相對位置的位置向量。

## 延伸閱讀

- **Vaswani et al. 2017「Attention Is All You Need」§3.5**（arXiv 1706.03762，HTML 版 https://arxiv.org/html/1706.03762v7 ）——正弦位置編碼的原始出處。直接讀 §3.5 那幾行：公式、「波長從 2π 到 10000·2π 的幾何級數」、選正弦是因為「可能可以外推到更長序列」與「相對位置可表示為線性函數」這兩句原話。一頁不到，讀原文比讀二手解說值得。
- **Jay Alammar「The Illustrated Transformer」**（https://jalammar.github.io/illustrated-transformer/ ）——位置編碼那一段有把不同維度的正弦波畫成熱圖的視覺化，看一眼「高頻低頻分工」會比文字更直觀。脊椎那個 it 的例子也出自這條普及線（ch01 已標：例子源於 Google 2017 部落格，非原論文）。
- **Devlin et al. 2018「BERT」**（arXiv 1810.04805）——可學習絕對位置嵌入的代表作。不必細讀全文，知道它「用一張可訓練的位置查找表取代正弦公式、因此 context 長度寫死」即可，對照本章那張取捨表。
- **《圓的影子》ch05（旋轉矩陣 R(a)R(b)=R(a+b)）、ch07–08（複數與 e^{iθ}）**——本章「相對位置＝旋轉一個固定角度」的數學底子。讀完它們，ch10 的 RoPE 會像水到渠成：既然相對位置本質是旋轉，何不直接旋轉 query 和 key？這是全書最漂亮的跨書接點，先把旋轉的直覺擦亮。
- **ch10（RoPE 與相對位置）、ch19（注意力即集合運算）**——本章的兩個接續：ch10 把「相對位置＝旋轉」推到極致（直接旋轉 q/k，內積只剩角度差）；ch19 給「self-attention 置換等變」一個嚴格的理論版（Set Transformer），讓你看清這個「洞」其實是 attention 深層身分的一面。
