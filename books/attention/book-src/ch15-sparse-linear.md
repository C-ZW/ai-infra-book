# ch15 — 給注意力建索引：稀疏與線性注意力

> **本章解決什麼問題**：ch14 把刀架在脖子上——self-attention 是「一張建不了索引的表」，每個 query 對每個 key 都得算一次相似度，n×n 的矩陣讓計算與記憶體隨序列長度平方成長，長 context 一定爆。ch14 在結尾預告了兩個直覺方向：要嘛**少算一些對**（別每對都比），要嘛**換個算法**讓總量不要平方成長。本章就把這兩條路走到底：**稀疏注意力（sparse attention）**動三步骨架的第一步「打分」——只在部分 token 對之間算分數，硬把 n×n 砍出一堆洞；**線性注意力（linear attention）**動第二步「正規化」——把 softmax 換成一個能拆開的核函數，再用矩陣乘法的結合律把 (QKᵀ)V 重排成 Q(KᵀV)，O(n²) 一口氣降到 O(n)。兩條路都能省，但都付了同一筆學費：**它們是近似，不是精確**——省下的成本常常從「精確回憶」那裡扣回去。記住這筆學費，下一章（ch16）的 FlashAttention 才會顯出它的反常：不近似也能省。

## 從你已知的出發

你做後端做久了，對「O(n²) 怎麼救」其實有一整套肌肉記憶，只是沒意識到那就是本章在做的事。

第一招你天天用：**加索引，把全表掃描變成局部查找**。一張沒有索引的表，查一筆要 full scan O(n)；建了 B-tree 索引，查一筆降到 O(log n)。但 ch14 已經告訴你殘酷的事實——attention 的「相似度」**建不了傳統索引**，因為它不是等值比對（`WHERE id = 42`），而是「跟每個 key 算一個連續的相似度分數」，沒有可排序、可雜湊的鍵能讓你預先把資料切好。所以 attention 不能直接套 B-tree。但你還有第二招、第三招。

第二招：**局部性假設——只比對「附近的」資料，不比對全部**。你寫過滑動視窗（sliding window）的限流、寫過只看最近 N 筆的時間序列聚合、寫過「快取只存熱資料、冷資料不碰」。背後是同一個賭注：大部分有用的關聯都是**局部的**，遠處的關聯稀少到可以忽略或另外處理。這就是**稀疏注意力**的靈魂——別讓每個 token 看全部，只看鄰近的窗口，再挑幾個「全域重要」的 token 讓大家都能看到。

第三招你在查詢優化裡親手用過：**改變運算順序，避開那個爆炸的中間結果**。`(A · B) · C` 和 `A · (B · C)` 算出來一模一樣（矩陣乘法有結合律），但如果中間的 `A · B` 是一個巨大的矩陣，你就會選另一種括號法，先算那個小的中間結果。你在 SQL 裡見過：先 filter 再 join，跟先 join 再 filter，結果相同但成本天差地遠。**線性注意力的核心招數，就是把 attention 的乘法重新加括號，避開那個 n×n 的中間矩陣。**

在三步骨架裡，這兩條路各動哪一步？這是本章你最該帶走的一張定位圖：

```text
            ┌──────────┐   ┌────────────┐   ┌──────────────┐
 query ──►  │ 1. 打分  │─► │ 2. 正規化  │─► │ 3. 加權混合  │ ──► 輸出
 keys  ──►  │  score   │   │  softmax   │   │ weighted sum │
 values ─────────────────────────────────────────┘

   ▸ 稀疏注意力：改「打分」的連結結構——只算一部分 token 對（local 窗口＋少數全域）
   ▸ 線性注意力：改「正規化」——把 softmax 換成可拆開的核 φ(q)φ(k)ᵀ，再用結合律重排乘法順序
```

稀疏動「打分」的**連結結構**（哪些對要算）；線性動「正規化」的**形式**（用什麼函數把分數變權重）。第三步「加權混合」兩條路都沒換——還是「按權重對 value 做加權平均」。這是它們的共同點，也是為什麼它們仍然算「注意力」：三步骨架的骨還在，只是改了其中一根。

## 第一條路：稀疏注意力——別讓每個 token 看全部

先把問題量化。ch14 已經算過：序列長 n，標準 self-attention 要填滿一個 n×n 的注意力矩陣，每一格是一對 (query, key) 的相似度。n=1000 是一百萬格，n=100k 是一百億格。這一百億格裡，**真的有用的有多少？**

直覺告訴你：不多。在一篇長文章裡，第 5000 個字跟第 3 個字之間有強關聯的機率很低；大部分的句法、指代關係都發生在**附近**（脊椎句子裡 it 指 animal，兩者只隔幾個字）。如果絕大多數遠距離的格子權重都接近 0，那把它們**全部算出來再發現它接近 0**，就是純浪費。稀疏注意力的賭注就是：**事先決定哪些格子根本不算，只填一部分。**

問題變成：哪些格子留、哪些砍？這是稀疏注意力各個變體的全部分歧點。主流的切法有三種積木，混搭出不同方案：

```text
注意力矩陣（n×n），■=要算的對，·=砍掉不算
query↓ \ key→

(a) 局部窗口（local window）       (b) 全域 token（global）          (c) 隨機（random）
    每個 query 只看左右 w 個鄰居        少數特殊 token 大家都能看        每個 query 額外看 r 個隨機位置
  ■ ■ · · · · · ·                  ■ · · · · · · ■                ■ · · ■ · · · ·
  ■ ■ ■ · · · · ·                  ■ ■ · · · · · ■                · ■ · · · ■ · ·
  · ■ ■ ■ · · · ·                  ■ · ■ · · · · ■                · · ■ · · · ■ ·
  · · ■ ■ ■ · · ·                  ■ · · ■ · · · ■                ■ · · ■ · · · ·
  · · · ■ ■ ■ · ·                  ■ · · · ■ · · ■                · · ■ · ■ · · ·
  · · · · ■ ■ ■ ·                  ■ · · · · ■ · ■                · ■ · · · ■ · ·
  · · · · · ■ ■ ■                  ■ · · · · · ■ ■                · · · ■ · · ■ ·
  · · · · · · ■ ■                  ■ ■ ■ ■ ■ ■ ■ ■                · · · · ■ · · ■
  約 n·w 格（線性）                 全域那欄/列是滿的                 約 n·r 格（線性）
```

- **局部窗口（local window）**：每個 query 只跟它左右 w 個鄰居算分數。這直接編碼了「關聯多是局部的」這個歸納偏置（inductive bias，見 ch10）。格子數從 n² 降到約 n·w——**對固定的 w，這是線性的 O(n·w)**。
- **全域 token（global token）**：純局部會有個明顯的洞——遠處的兩個 token 永遠互相看不到，全局資訊傳不過去。補法是指定少數「全域 token」（例如分類用的 `[CLS]`、或每隔一段放一個），讓**所有 token 都能看它、它也能看所有 token**。少數幾根「全域柱子」就能讓資訊在遠處之間中轉，代價只多 O(n) 格。
- **隨機（random）**：再撒一些隨機連結，從圖論角度保證「任兩點之間有短路徑」（小世界性質），讓資訊傳遞的跳數不會太多。

把這三種積木組合起來，就是 landscape 裡那幾個有名的方案（2026-06 查證）：

| 方案 | 用了哪些積木 | 複雜度 | 出處 |
|---|---|---|---|
| **Sparse Transformer** | 固定模式：局部 + strided（每隔 l 個看一次），切給不同頭分工 | **O(n√n)** | Child et al. 2019 |
| **Longformer** | 局部滑動窗口 + 任務指定的全域 token | **O(n)**（線性） | Beltagy et al. 2020 |
| **BigBird** | 局部窗口 + 全域 + 隨機，三合一 | **O(n)**（線性） | Zaheer et al. 2020 |

Sparse Transformer（Child et al. 2019，作者群裡有 Radford 和 Sutskever）是這條路的開山之作，它的兩種稀疏模式（局部、跨步 strided）分給不同的注意力頭去負責，總複雜度 O(n√n)——還不是線性，但已經把平方咬掉一大塊。Longformer 和 BigBird（都是 2020）再進一步做到**線性**：Longformer 用「滑動窗口＋少數全域」，BigBird 加上隨機連結並從理論上證明這個稀疏模式仍保留足夠的表達力（它能逼近完整注意力，甚至圖靈完備——但這是「存在性」的保證，不代表實際訓練一定學得到）。

**稀疏注意力動了三步骨架的哪一步？** 第一步「打分」的**連結結構**。它沒改打分的公式（還是內積 q·k）、沒改 softmax、沒改加權混合——它改的是「哪些對要進入打分」。被砍掉的格子，等同於把那些位置的分數設成 −∞（回收 ch11 遮罩的招數：softmax 前設 −∞，softmax 後就是 0），讓那些 value 在加權混合時權重歸零。**稀疏注意力就是一個事先畫好、固定的遮罩**——只是這次遮的不是「未來」，而是「太遠、賭它不重要」的對。

## 第二條路：線性注意力——把乘法重新加括號

稀疏這條路有個哲學上的不滿足：它**靠假設**砍格子（賭遠處不重要）。能不能不砍任何一對、所有對都「算到」，但總成本還是不要平方成長？這聽起來像吃免費午餐，但線性注意力真的端出了一盤——代價藏在別處（後面會抓出來）。

它的關鍵洞察是回頭看 softmax 到底卡在哪。標準注意力（先忽略 √d 縮放，看結構）長這樣：

```text
標準注意力（一個 query 的視角）：

         Σⱼ exp(q·kⱼ) · vⱼ
output = ─────────────────       ← 分子：以 exp(q·kⱼ) 為權重，對所有 value 加權和
            Σⱼ exp(q·kⱼ)          ← 分母：權重總和（softmax 的正規化）
```

注意那個 **exp**。它把 q 和 k 死死綁在一起——`exp(q·kⱼ)` 沒辦法拆成「只跟 q 有關的部分」乘「只跟 kⱼ 有關的部分」。因為拆不開，你**被迫**先算出每一對的 `q·kⱼ`（這就是那個 n×n 矩陣），才能繼續。**exp 是 O(n²) 的元兇**，不是內積本身。

線性注意力的招數：**把 exp(q·k) 換成一個能拆開的核函數** φ(q)·φ(k)。這裡 φ 是一個「特徵映射」（feature map），把向量變換一下，使得相似度能寫成「q 變換後」和「k 變換後」的內積。換掉之後：

```text
線性注意力（換掉 exp 為可拆的核 φ）：

         Σⱼ [φ(q)·φ(kⱼ)] · vⱼ        φ(q)ᵀ · ( Σⱼ φ(kⱼ) vⱼᵀ )
output = ──────────────────────  =  ──────────────────────────
            Σⱼ φ(q)·φ(kⱼ)              φ(q)ᵀ · ( Σⱼ φ(kⱼ) )
                                              ▲
                                    φ(q) 可以提到 Σ 外面！
```

關鍵就在那一步：因為 φ(q) 跟 j 無關，它可以**提到求和符號外面**。原本「對每個 query 都要重新跑一遍對所有 key 的求和」，變成「先把所有 key 的部分 `Σⱼ φ(kⱼ)vⱼᵀ` 算好一次，每個 query 來查的時候直接乘」。這個「先把 key 那邊聚合好」的中間量，是一個 **d×d 的小矩陣**（跟序列長 n 無關），不再是 n×n 的大矩陣。

寫成整批矩陣形式，這正是「結合律重新加括號」：

```text
標準（O(n²)）：  output = softmax(Q Kᵀ) V       ← 先算 Q Kᵀ＝n×n 大矩陣（爆炸點）
線性（O(n)）：   output ∝ φ(Q) · ( φ(K)ᵀ V )   ← 先算 φ(K)ᵀ V＝d×d 小矩陣（不含 n²）
```

`(QKᵀ)V` 和 `Q(KᵀV)` 算出來相等（矩陣乘法結合律，見《矩陣是動詞》ch06），但**括號位置決定了你要不要實體化那個 n×n 中間矩陣**。標準注意力的 softmax 把括號**鎖死**在 `(QKᵀ)` 那邊（exp 拆不開，沒得選）；線性注意力把 softmax 換成可拆的核，**解除了這個鎖**，於是你可以選 `(KᵀV)` 那邊先算——而 KᵀV 是 d×d，跟 n 無關。

**線性注意力動了三步骨架的哪一步？** 第二步「正規化」。它把 softmax（exp 核）換成另一個核 φ(q)φ(k)ᵀ。打分還是某種相似度、加權混合還是加權平均，但「用什麼把分數變權重」這一步換了人——換成一個**能拆開、能讓你重排乘法順序**的函數。這是全章最該停下來想的一句：**O(n²) 不是內積的錯，是 softmax 那個 exp「拆不開」逼你先算大矩陣的錯；換一個拆得開的核，平方就消失了。**

landscape 裡這條路的三個代表（2026-06 查證）：

| 方案 | 怎麼換 softmax | 出處 |
|---|---|---|
| **Transformers are RNNs** | 用 φ(x)=elu(x)+1 當核，直接套結合律，O(n²)→O(n) | Katharopoulos et al. 2020 |
| **Performer** | 用隨機特徵（FAVOR+）**無偏估計** softmax 核本身——逼近真 softmax | Choromanski et al. 2020 |
| **Linformer** | 假設注意力矩陣**低秩**，把 K、V 在序列維壓到 k×d（k≪n） | Wang et al. 2020 |

Katharopoulos 的「Transformers are RNNs」是最直接的——它連 softmax 都不假裝逼近，直接挑一個簡單的 φ（elu+1）當核，然後吃結合律。論文標題的「are RNNs」是個漂亮的副產品：一旦寫成這個形式，自迴歸生成時那個 `Σ φ(kⱼ)vⱼᵀ` 可以**邊生成邊累加**（像維護一個 running state），於是線性注意力的生成過程**就是一個 RNN 的遞推**——固定大小的狀態、O(1) 每步。記住這個「壓進固定狀態」的影子，ch22 講 SSM/Mamba 時會發現它們是同一條線的延伸。

Performer 走另一條路：它不換掉 softmax 的語義，而是用隨機特徵去**逼近** exp 核本身（FAVOR+），所以它號稱「能估計真正的 softmax 注意力」。Linformer 則賭另一個假設——**注意力矩陣是低秩的**（rank 遠小於 n），所以把 K、V 沿序列方向投影到一個小維度 k 就夠了；這直接接上《矩陣是動詞》ch19–20 的低秩近似與 SVD：一個 n×n 但實際只有低秩結構的矩陣，可以用遠少於 n² 的數字逼近。

## Worked example：用結合律數出「省在哪」

抽象講了兩種乘法順序，現在動手數乘法次數，把「省」變成具體數字。我們不需要 softmax（線性注意力本來就拿掉它了），只看「先算哪個括號」對乘法量的影響。

設序列長 n、維度 d。三個矩陣都是 n×d：Q、K、V。我們要算 `Q Kᵀ V`（線性注意力去掉正規化後的骨架）。數的是純量乘法的次數（一個 a×b 矩陣乘 b×c 矩陣要做 a·b·c 次乘法）。

**路徑 A：先算 (QKᵀ)，再乘 V**（標準注意力的順序）

```text
步驟 1：Q Kᵀ      → (n×d)·(d×n) = n×n      乘法數 = n · d · n = n²·d
步驟 2：(QKᵀ) V   → (n×n)·(n×d) = n×d      乘法數 = n · n · d = n²·d
                                    ─────────────────────────────
                                    合計 A = 2·n²·d      ← 含 n²，爆炸
```

**路徑 B：先算 (KᵀV)，再被 Q 乘**（線性注意力的順序）

```text
步驟 1：Kᵀ V      → (d×n)·(n×d) = d×d      乘法數 = d · n · d = n·d²
步驟 2：Q (KᵀV)   → (n×d)·(d×d) = n×d      乘法數 = n · d · d = n·d²
                                    ─────────────────────────────
                                    合計 B = 2·n·d²      ← 只含 n 的一次方，線性
```

兩個結果**完全相等**（結合律保證），但成本差在 `n²·d` vs `n·d²`——也就是差在 **n vs d**。代進幾組數字：

```text
   n        d     A=2n²d（標準）        B=2nd²（線性）       A/B（>1 表示 B 較省）
─────────────────────────────────────────────────────────────────────────
   2        4              32                  64             0.50×（B 反而貴！n<d）
   4        2              64                  32             2.00×（B 省一半，n>d）
 1000      64     128,000,000           8,192,000            15.62×（n≫d，B 大省）
100000    128   2,560,000,000,000   3,276,800,000           781×（長 context，B 狂省）
```

停在這張表第一列看一件**反直覺、而且常被漏掉**的事：**當序列比維度還短（n<d）時，線性注意力路徑 B 反而比較貴**——n=2、d=4 時 A=32、B=64，標準路徑 A 才便宜。為什麼？看式子：A=2n²d、B=2nd²，兩者相等的 **crossover 恰好在 n=d**；A<B 當且僅當 **n<d**。直覺上：B 先算的 KᵀV 是 d×d 的矩陣，當 d 比 n 還大時，這個「小」矩陣其實比 n×n 還大，先算它就虧了。所以線性注意力**只在 n≫d（序列遠長於維度）時才真正划算**——而那正是長 context 的場景（n=10 萬、d=128 時省 781 倍）。短序列上硬上線性注意力，是賠錢生意。

> **可選的 numpy 雛形（數字與上面手算一致，驗證兩條路徑算出同一個結果）**

```python
import numpy as np
np.random.seed(0)
n, d = 8, 2
Q, K, V = np.random.randn(n, d), np.random.randn(n, d), np.random.randn(n, d)

path_A = (Q @ K.T) @ V          # (n x n) intermediate -> 2*n*n*d mults
path_B = Q @ (K.T @ V)          # (d x d) intermediate -> 2*n*d*d mults
print(np.allclose(path_A, path_B))           # True: associativity, identical result
print("A flops:", 2*n*n*d, " B flops:", 2*n*d*d)   # 256  vs  64
```

這就是線性注意力的**全部魔術，也是它的全部限制**：結合律讓你免費換到同一個答案，但「省」是有條件的（n≫d 才省）。沒有 softmax 那個拆不開的 exp 擋路，你才有資格選 B 這個括號法——這也再次說明，O(n²) 的鎖是 softmax 上的，不是內積上的。

## 故障模式與直覺陷阱

這兩條路是過去幾年最熱鬧的研究方向，也累積了最多「論文說很好、實際換上去退步」的故事。下面是真正會把你帶溝裡的地方。

### 招牌學費：近似會在「精確回憶」上吃虧

這是整章最重要的一句，也是 ch22 為什麼混合架構要保留少量 softmax 的伏筆。**稀疏和線性注意力都是近似（approximation），不是精確（exact）**——這跟下一章的 FlashAttention 形成最關鍵的對比。

近似在哪裡漏？在**精確回憶（exact recall）**——「從很遠的地方，精準地把某一個特定 token 撈回來」這種能力。

- **稀疏**漏在它砍掉的格子：如果 it 要看的那個 animal 剛好落在它的局部窗口外、又不是全域 token，那條連結**根本不存在**，分數連算都沒算，權重必然是 0。稀疏注意力賭「遠處不重要」，但語言裡偏偏有「呼應很遠的那個詞」的時候（長文章開頭定義的一個變數名、幾千字後才用到），這時稀疏就漏接。
- **線性**漏在核的表達力：softmax 那個 exp 有一個寶貴性質——它能把分數差**指數級放大**，做出非常「尖」的權重分布（ch05–06 算過，[2,1,0] 縮放後仍能給最相關的 token 0.665 的權重）。換成 φ(q)φ(k)ᵀ 這種低次的核，分布往往「軟趴趴」、難以做到那麼尖的聚焦。**當任務需要「幾乎只看那一個 token」（精確檢索）時，軟趴趴的線性核就糊掉了。** 加上線性注意力把所有 key 壓進一個固定大小的 d×d 狀態，n 個 token 的資訊擠進一個跟 n 無關的小盒子——序列愈長，這個盒子愈像個資訊漏斗，遠處的細節被後來的覆蓋稀釋（這正是 ch22 SSM「固定狀態的代價」的同一個病）。

`（2026-06）`的實證共識：在需要 in-context 精確回憶的任務（如大海撈針 needle-in-a-haystack、複製一長串、in-context learning）上，純線性／純稀疏注意力普遍**輸給** softmax 注意力。這就是為什麼混合架構（ch22）通常還是**保留少數幾層完整 softmax**——讓那幾層扛精確回憶，其餘交給便宜的線性層。**省下的成本，是從「能不能精準撈回遠處那一個」那裡扣的。** 天下沒有白吃的午餐，這一章的午餐錢記在這裡。

|---|---|---|---|
| 把 O(n) 當「永遠更快」 | 「線性注意力複雜度低，所以一定更快」 | 忽略 crossover 在 n=d：n<d 時線性反而**更貴**（worked example）；且常數因子大、kernel 不好寫 | 問自己「我的 n 真的遠大於 d 嗎？」短序列上線性是賠本 |

| 把低秩假設當普世真理 | 「注意力矩陣總是低秩，壓了不虧」 | Linformer 賭低秩；但需要尖銳、近 one-hot 的注意力（精確檢索）時，矩陣**不是**低秩，壓了就糊 | 任務愈需要「只盯一個 token」，低秩假設愈危險 |

### 自我察覺的徵兆

什麼時候你該停下來懷疑「是不是近似注意力害的」？三個訊號：(1) 模型在短 context 上正常、一拉長就掉鏈子，尤其是「需要引用很前面某個特定詞」的時候；(2) 換上高效注意力後，throughput 上去了但下游品質悄悄退步，且退步集中在「檢索／複製」類能力；(3) 你在 needle-in-a-haystack 這種「埋一根針在長文裡要模型找出來」的測試上分數崩掉。這三個都指向同一個根因：**你用精確回憶能力換了速度，而這個任務剛好需要精確回憶。**

## 紙上推演

### 推演 1：把稀疏 vs 線性各自「動了三步骨架哪一步」說清楚 [10 分鐘] ★

不看上文，口頭（或寫 200 字）回答：(1) 稀疏注意力動了三步骨架的哪一步？它改了那一步的什麼（公式還是別的）？被砍掉的格子，等同於在 softmax 前對那些位置做了什麼？(2) 線性注意力動了哪一步？它把什麼換成了什麼？為什麼這個替換是「解鎖結合律」的前提？(3) 兩條路的第三步「加權混合」有沒有變？

### 推演 2：用結合律證明線性注意力的 O(n)，並找出 crossover [20 分鐘] ★★

設 Q、K、V 都是 n×d。手算兩條路徑算 `Q Kᵀ V` 的乘法次數：(a) 先 `(QKᵀ)` 再乘 V；(b) 先 `(KᵀV)` 再被 Q 乘。各寫成 n 和 d 的式子。然後：(1) 哪條是 O(n²)、哪條是 O(n)？(2) 兩條路徑相等發生在 n 和 d 什麼關係時（crossover）？(3) 代入 n=10000、d=64，B 比 A 省幾倍？(4) 代入 n=32、d=64（短序列），哪條反而便宜？用這個結果反駁「線性注意力永遠更快」。

### 推演 3：解釋「為什麼近似注意力會在精確回憶上吃虧」 [15 分鐘] ★★

口頭講清楚兩件事：(1) **稀疏**為什麼會漏掉「呼應很遠的那個詞」？（提示：被砍的格子權重是多少、有沒有算過？）(2) **線性**為什麼難做出「幾乎只看那一個 token」的尖銳分布？（提示：softmax 的 exp 在放大分數差上有什麼線性核做不到的本事？）最後用一句話連到 ch22：為什麼 2026 的混合架構通常還是保留少數幾層完整 softmax？

### 推演解答

**推演 1。** (1) 稀疏動**第一步「打分」**——更精確說是打分的**連結結構**（哪些 token 對要進入打分）。它**不改打分的公式**（還是內積 q·k），改的是「算哪些對」：只算局部窗口＋少數全域（＋隨機）的對，其餘不算。被砍掉的格子，等同於在 softmax 前把那些位置的分數設成 **−∞**（回收 ch11），softmax 後權重變 0——所以它就是一個事先畫好的固定遮罩。(2) 線性動**第二步「正規化」**——把 softmax（exp 核 `exp(q·k)`）換成一個**可拆開**的核 `φ(q)·φ(k)`。這個替換是解鎖結合律的前提：因為 exp(q·k) 拆不開，括號被鎖在 `(QKᵀ)` 那邊，逼你先算 n×n 矩陣；換成可拆的 φ(q)φ(k)ᵀ 後，φ(q) 能提到求和外面，你才有資格改成 `Q(KᵀV)`、先算 d×d 的小矩陣。(3) 第三步「加權混合」**兩條都沒變**——還是「按權重對 value 做加權平均」。這是它們仍算「注意力」的原因：三步骨架的骨架完整，各自只換了一根。

**推演 2。**

```text
路徑 A：(Q Kᵀ) V
  Q Kᵀ:  (n×d)(d×n)=n×n   →  n·d·n = n²d 次乘法
  (·) V: (n×n)(n×d)=n×d   →  n·n·d = n²d 次乘法
  合計 A = 2 n²d                              ← O(n²)，含 n 的平方

路徑 B：Q (Kᵀ V)
  Kᵀ V:  (d×n)(n×d)=d×d   →  d·n·d = n d² 次乘法
  Q (·): (n×d)(d×d)=n×d   →  n·d·d = n d² 次乘法
  合計 B = 2 n d²                              ← O(n)，n 只有一次方
```

(1) A 是 O(n²)、B 是 O(n)（固定 d 時）。(2) crossover：`2n²d = 2nd²` ⟺ `n = d`。當 **n < d** 時 A < B（標準反而便宜）、**n > d** 時 B < A（線性才省）。(3) n=10000、d=64：A/B = n²d/(nd²) = n/d = 10000/64 ≈ **156 倍**。(4) n=32、d=64：n<d，A=2·32²·64=131072、B=2·32·64²=262144，**A 反而便宜**（B 是 A 的兩倍）。所以「線性注意力永遠更快」是錯的——它只在 n≫d 時划算，短序列上是賠本生意。常見錯路：背了「O(n) 比 O(n²) 快」就以為任何情況都成立，忘了大 O 藏掉的常數和 crossover；以及忘了線性注意力的 kernel 在硬體上常數因子更大、實測不一定贏（這裡只數乘法，沒算記憶體存取——而那正是 ch16 的主題）。

**推演 3。** (1) 稀疏漏掉遠處呼應，是因為那條連結的格子**被事先砍掉了，分數連算都沒算**——它的權重必然是 0（不是「算出來很小」，是「壓根沒這個對」）。當「呼應很遠的那個詞」剛好落在局部窗口外、又不是全域 token，模型就無從看它。(2) 線性難做尖銳分布，是因為 softmax 的 **exp 能把分數差指數級放大**——分數差一點點，exp 後權重差很多，於是能逼近「幾乎只看最大那個」（接近 one-hot 的 soft-argmax，ch05–06）。換成 φ(q)φ(k)ᵀ 這種低次核，沒有 exp 的指數放大，分布天生「軟」、難以集中到單一 token；加上所有 key 被壓進固定大小的 d×d 狀態，遠處細節被稀釋。(3) 連到 ch22：因為精確回憶（從遠處精準撈一個特定 token）是純線性／純稀疏的共同弱點，2026 的混合架構通常**保留少數幾層完整 softmax** 專門扛精確回憶，其餘層用便宜的線性／SSM——這是「省成本」和「保檢索」之間的工程折衷，跟你「熱路徑走快取、少數需要精確的走全查詢」是同一個取捨。
