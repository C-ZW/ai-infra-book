## 從你已知的出發

你職涯裡最常被半夜叫起來的故障，多半長同一個樣子：某張表沒加索引，資料量小的時候沒人發現，等到資料漲到一個量級，一個查詢從幾毫秒變成幾十秒，連鎖拖垮整個服務。你打開慢查詢日誌，看到那行刺眼的 `Seq Scan`（全表掃描），心裡咯噔一下——「這就是 N+1 / 沒走索引 / 兩兩比對的經典翻車」。

self-attention 在做的，正是這種你最怕的事，而且是**結構性地、無法靠加索引繞過地**在做。ch07 已經點過這個直覺：self-attention 是一個**沒有索引的 self-JOIN**——序列裡每個 token 當 query，要對全序列每個 token 當 key 算一次相似度，n 個 query × n 個 key = n² 個分數。本章要把這件事的兩個後果講透：

1. **代價會爆**：n² 的成長在 n 小的時候溫和到你不會注意，n 一大就以平方速度失控。這正是「資料量小時沒事、漲到一個量級就翻車」的數學版本。
2. **加不了索引**：你的 DB 翻車還有救——加個 B-tree、加個 hash 索引，把 O(n) 的全表掃描降到 O(log n) 或 O(1)。但 attention 的查找**建不了這種索引**。為什麼建不了，是本章的核心，也是後面三章為什麼要那麼費勁繞路的根本原因。

把這兩點釘住，你就懂了：Part IV 不是在介紹幾個炫技的新模型，而是在處理一個你太熟悉的工程問題——**一張慢表，但這次你連索引都加不上**。

在三步骨架裡，本章站在哪？和 ch07 一樣，它哪一步都沒改：

```text
            ┌──────────┐   ┌────────────┐   ┌──────────────┐
 query ──►  │ 1. 打分  │─► │ 2. 正規化  │─► │ 3. 加權混合  │ ──► 輸出
 keys  ──►  │  score   │   │  softmax   │   │ weighted sum │
 values ─────────────────────────────────────────┘
                  │
                  └─►【打分步要算 n×n 個分數——本章盤點這一步的成本帳單】
```

O(n²) 整個誕生在**打分（score）**這一步：n 個 query 各對 n 個 key 算內積，攤成 n×n。正規化（逐列 softmax）和加權混合（乘 V）跟著也是 n² 量級，但它們是打分產出 n×n 的**後果**，不是獨立的源頭。本章不引入任何新機制，它做的是一件後端工程師每天做的事：**把一個演算法的時間與空間複雜度算清楚，看它在規模上會不會炸。**

## 把 O(n²) 算到讓你心疼：長 context 下的爆炸

ch07 給過一張 n 與 n² 的成長表。本章把它重新算一遍、加上 2026 年真實模型的 context 尺度，讓「平方」這兩個字從抽象符號變成你能感覺到的痛。

先把帳對齊。注意力矩陣是 n×n，**格子數就是 n²**，每格一個 d 維內積（打分），所以時間是 O(n²·d)；要做逐列 softmax 就得把整張 n×n 表存著，所以空間是 O(n²)（web 查證：標準 self-attention 時間 O(n²·d)、空間 O(n²)，皆直接來自 QKᵀ 產生的 n×n 矩陣，2026-06）。d 是固定的模型超參數，我們關心 n 怎麼吃成本，主導項就是 **n²**。

把序列拉長，看格子數怎麼長（自算複核）：

```text
   n（序列長度）   n²（注意力矩陣格數）      相對 n=512 的倍數    現實對照
   3                          9               —                脊椎玩具
   512                  262,144               1×               一段話
   4,096             16,777,216              64×               一篇長文
   100,000       10,000,000,000          ~38,147×             一本中篇小說／長對話
   1,000,000  1,000,000,000,000       ~3,814,697×             整本書／超長 context
```

讀這張表，先抓住一條讀法規則，它比任何單一數字都好用：

> **context 長 k 倍，注意力的打分成本變 k² 倍。**

- context **加倍**（k=2）→ 成本變 **4 倍**（2²）。這就是 2026 年那句工程口頭禪「把 context window 翻倍，attention 那層的計算大致翻四倍」的數學出處（web 查證：doubling the context window roughly quadruples the attention computation，2026-06）。
- context 長 **10 倍**（k=10）→ 成本變 **100 倍**（10²）。
- 從一段話（n≈512）到一本書（n≈100k，長約 195 倍）→ 格子數從二十幾萬暴衝到 **一百億**（10¹⁰），成本約 38,000 倍。

這個「一開始溫和、後面失控」就是平方函數的全部脾氣。n=3 時九個格子，連零頭都算不上；但 n 每往上跳一個數量級，n² 就往上跳兩個。線性成長是爬樓梯，平方成長是坐電梯往下墜——你在 n 小的時候完全感覺不到地板在哪，等感覺到時已經摔下去了。這正是「資料量小時沒事、漲到一個量級就翻車」在數學上長的樣子。

2026 年這件事為什麼變得這麼要命？因為 context window 的軍備競賽。把幾個真實尺度擺進來感受一下（web 查證，皆標時點 2026-06）：

```text
   模型（2026-06，數字會變，看趨勢不看絕對值）       廣告 context window
   Claude Sonnet 4                                     1,000,000 tokens（1M）
   Gemini 2.5 Pro                                      2,000,000 tokens（2M）
   Gemini 3 Pro / Llama 4 Scout                       10,000,000 tokens（10M）
```

把 n=100 萬代進去：n² = **一兆**（10¹²）格。把 n=1000 萬代進去：n² = **一百兆**（10¹⁴）格。每一格還要乘上模型維度 d、再乘上層數、再乘上頭數。你開始明白為什麼「長 context」在 2026 年是貼著錢燒的奢侈品——每一倍的 context,attention 那層都在收平方的稅。

> **這裡了不起／反直覺在哪**：很多人以為「context 變長就是多塞點字進去，線性多花一點錢」。錯得離譜。塞進去的字是線性的（embedding、FFN 那些逐位置的運算確實是 O(n)）,但**讓每個字看見彼此**——attention 那一步——是平方的。你付的不是「多 n 個字」的錢，是「多 n 個字之間 n² 條兩兩連線」的錢。這就是為什麼一個只長了 10 倍的輸入，帳單卻長了 100 倍。**長 context 真正貴的不是資訊量，是資訊之間的關係數。**

至於記憶體那筆帳，ch07 已經點明「時間 O(n²) 和空間 O(n²) 是兩筆要分開算的帳」。空間那筆在真實硬體上會具體變成「這張 n×n 矩陣在 GPU 記憶體裡多大、怎麼被反覆讀寫」——那是**部署與推論物理**的問題，深度屬於《從後端到 AI Infra》ch03/ch05,本書不展開。本章只給你一個會心一驚的動機數字：web 查證提到，光是存下**單一層**、n=32k 序列的注意力矩陣，就可能吃掉一張 80GB HBM 顯卡裡的數十 GB（2026-06,具體數字隨實作與精度而異）。記住這個量級就好；為什麼會這樣、怎麼省，ch16 講想法、工作書講工程。

對照那個被注意力取代的舊世界——RNN（ch02）。RNN 一個字一個字處理，第 i 步只更新一個固定大小的隱藏狀態，總共 n 步，複雜度 **O(n)**,線性。從「成本對 n 的成長」單看，RNN 完勝：

```text
   架構             計算量對 n 的成長     直覺
   self-attention   O(n²)  ← 平方        每個 token 看全部 token
   RNN              O(n)   ← 線性        每個 token 只更新一個狀態,往後傳
```

那為什麼大家還是拋棄了 RNN、擁抱 O(n²) 的 attention?因為 ch07 講透的那個交易：RNN 的 n 步**必須循序**（第 i 步等第 i−1 步）,沒法平行，而且句首的資訊要走 n 步才傳到句尾、一路衰減；self-attention 的 n×n 每一格**互不依賴、一次平行算完**（循序深度 O(1)）,任兩個 token 一步直達。**O(n²) 不是 bug,是「敢讓所有位置直接互看」的明碼標價。** 本章要面對的問題正是：這個價，在長 context 下貴到付不起了，能不能殺價？——但別把孩子跟洗澡水一起倒掉（別為了省錢丟了「全句直達」）。

## 為什麼不加個索引：相似度查找的詛咒

到這裡，你身為後端工程師的本能該跳出來了：慢查詢的標準解法不就是**加索引**嗎？DB 全表掃描 O(n) 慢，加個 B-tree 索引降到 O(log n)、加個 hash 索引降到 O(1)。attention 是個 n×n 的全表對全表掃描，那……**給它建個索引不就好了？**

這個問題問得對極了——它正是 ch15 之後所有「高效注意力」研究的起點。而本章要給你的答案是：**attention 這張表，建不了傳統索引。** 不是沒人想到，是它的查找本質上不吃索引這一套。理解「為什麼建不了」,你才會懂後面那些繞路有多無奈、又多巧妙。

先回顧索引憑什麼快。你的 DB 索引能把查找從 O(n) 降到 O(log n)/O(1),靠的是**鍵的兩個性質**:

```text
   索引類型    靠鍵的什麼性質              把查找降到
   hash 索引    鍵可雜湊成桶(等值比對)      O(1)   —— WHERE id = 42
   B-tree      鍵可排序(有全序)            O(log n) —— WHERE age > 30、ORDER BY
```

關鍵字是 **等值** 和 **有序**。你查 `WHERE id = 42`,hash 函數一算就直接跳到那個桶，不用掃別人；你查 `WHERE age BETWEEN 20 AND 30`,B-tree 因為鍵排好序，二分找到範圍的兩端，中間整段直接拿。**索引之所以能跳過大部分資料，是因為「鍵的結構」事先告訴了你「不可能命中的那些，連看都不用看」。**

現在看 attention 的查找條件是什麼。它不是 `WHERE key = query`（等值）,也不是 `WHERE key > query`（有序）。它是：

```text
   attention 的「查詢條件」:找出與 query 內積最大(最相似)的 key——並且不是取一個,是按相似度混合全部
```

這個條件踩中了三個讓傳統索引失效的點，每一個都致命：

**第一，相似度不是等值，也沒有全序。** 「q 和 k 有多像」是一個連續的、多維度權衡出來的數值，不是「相等/不相等」那種離散桶，也不存在一個一維的大小排序能讓你「排好序後二分」。兩個四維向量 (2,1,1,0) 和 (1,−1,0,0),誰「比較大」?這問題本身沒意義——向量在高維空間裡的「像」是它們的夾角與長度共同決定的，投影到任何一個軸上排序都會丟掉資訊。沒有全序，B-tree 無從建起；不是等值，hash 桶無從分起。

**第二，查的不是「最像的那一個」,是「按相似度混合全部」。** 就算你退一步，只想找「最相似的前幾個 key」（這在向量檢索裡叫 nearest neighbor,確實有專門的索引，下面就說）,attention 也不滿足——它的三步骨架第三步是**加權混合所有 value**(ch05 的招牌反直覺：attention 不是取一筆，是混合全部、相關的權重大)。softmax 給每個 key 一個非零權重，連 it→it 那個 0.090 都在貢獻。要算這個混合，你**原則上需要每個 key 的分數**才能正規化（softmax 的分母是所有 e 次方的和）。「我只想看前幾名」這個能跳過大多數資料的捷徑，被 softmax 要「看完全部才能正規化」這件事堵死了。

**第三，query 不是固定的，是每筆資料現算的、而且還在變。** DB 索引能省事，還有個隱含前提：被索引的資料**相對靜止**,你建一次索引，之後無數次查詢都能用。但 self-attention 裡，每一層、每個 token 的 Q/K/V 都是**當場**由學來的投影矩陣算出來的（ch04）,而且**逐層在變**——這一層的輸出餵進下一層，下一層的 K 就全變了。就算你費勁為某一層建了索引，下一層作廢，等於沒建。這像一張**每讀一次就整張重寫**的表，你永遠來不及為它建索引。

把這三點合起來，結論很硬：

> **attention 的查找，是在一個高維連續空間裡、對一張每層都在變的表、做「按相似度加權混合全部」的操作。等值索引（hash）、有序索引（B-tree）的前提它一個都不滿足。所以沒有捷徑，每一對 (query, key) 都得老老實實算一次內積——這就是 O(n²) 在原理上無法靠「加索引」消除的根本原因。**

這裡值得補一個你可能會想到的反例，因為它恰好從反面印證了上面的論證（web 查證，2026-06）。向量資料庫（如做 RAG 的那些）不是有索引嗎？ANN（approximate nearest neighbor，近似最近鄰）用 LSH、HNSW 圖這些技巧，確實能在高維向量裡快速找「最像的前 k 個」。為什麼它行、attention 不行？三個差別：① ANN 找的是「**最像的少數幾個**」（top-k），不是「**混合全部**」——它能丟掉長尾，attention 的 softmax 不能；② ANN 是**近似**——它接受偶爾漏掉真正最像的那個，換取速度，而標準 attention 要精確；③ ANN 的索引建在**相對靜止**的向量庫上（你的文件 embedding 建好就不太動），不是每層現算現變的。

更深一層，連 ANN 也在跟一個叫**維度詛咒（curse of dimensionality）** 的東西搏鬥：web 查證指出，在高維空間裡，傳統樹狀索引（B-tree、KD-tree 那類靠空間切分的結構）的效能會退化到**還不如直接線性掃描**——因為維度一高，「鄰近」這個概念本身就變模糊了，空間切分救不了你（2026-06）。這從另一個角度確認了本章的論點：**高維相似度查找，先天就不是傳統索引的主場。** attention 不是不想加索引，是它的查找站在傳統索引的能力邊界之外。

而這，恰恰預告了 ch15 的兩條路為什麼長那個樣子——既然「精確地混合全部」建不了索引，那就只能在「精確」和「混合全部」這兩個前提上動刀。下一節說。

## 兩個方向：少算一些 vs 換個算法

既然 O(n²) 在原理上躲不掉（精確地讓每對 token 互看，就是 n² 個分數）,想殺價就只剩兩條路。它們對應「動三步骨架的哪一步」,你現在就能憑直覺推出來，後面 ch15/ch16 只是把細節填上。

把全表掃描慢的問題，放回你的後端腦袋，你本來就有兩個直覺解法：

```text
   想加速一個 O(n²) 的兩兩比對:

   方向 A:少比一些對         → 不是每對都算,只算「重要的/鄰近的」那些對
   方向 B:換個算法            → 別用那個逼你「兩兩都算」的公式,換一個能重排運算順序的
```

**方向 A——少算一些對（改「打分」的連結結構）。** 直覺是局部性假設：一個 token 真的需要看遍全句的每一個字嗎？多半不用——它最在意的是鄰近的幾個詞，加上少數幾個全域的「樞紐」。那就**只算這些對的分數**,其餘的對直接當作零（不連）。這就是**稀疏注意力（sparse attention）**：它把 n×n 那張稠密的表，挖成一張只有少數格子有值的稀疏表，於是要算的分數從 n² 降到 n√n（Sparse Transformer）甚至線性（Longformer/BigBird）。它動的是三步骨架的**第一步「打分」**——改的是「誰跟誰之間需要打分」的連結結構。對應你做的「只比對鄰近/重點資料，不全比」。代價：這是**近似**——你賭那些被跳過的對真的不重要，賭錯了就漏掉長程關係。

**方向 B——換個算法（改「正規化」）。** 這條路更刁鑽，直接挑戰那個逼你「兩兩都算」的元兇：softmax。標準 attention 是 softmax(QKᵀ)·V,那個 QKᵀ 必須先算出 n×n 才能餵給 softmax——n² 卡死在這。但如果把 softmax 換成一個能拆開的核函數 φ,讓分數寫成 φ(Q)φ(K)ᵀ 的形式，那就能玩一個你在查詢優化裡用爛的招：**矩陣乘法結合律**。本來是 (φ(Q)·φ(K)ᵀ)·V——先算 n×n 的 φ(Q)φ(K)ᵀ;改成 φ(Q)·(φ(K)ᵀ·V)——先算 d×d 的 φ(K)ᵀV,再被 Q 乘。中間那個大矩陣的尺寸從 n×n 縮成 d×d,複雜度從 O(n²·d) 降到 O(n·d²)——**對 n 線性**(web 查證：linear attention 用核函數替換 softmax,靠矩陣乘法結合律先算 KᵀV,把 O(n²d) 降到 O(nd²),2026-06)。這就是**線性注意力（linear attention）**,動的是三步骨架的**第二步「正規化」**——把 softmax 換掉，換來重排運算順序的自由。對應你把 (A·B)·C 改成 A·(B·C) 省掉大中間結果。代價：也是**近似**——φ 核近似不了 softmax 的全部威力，尤其在「精確回憶某個遠處 token」這種事上會吃虧。

```text
   方向    動三步骨架哪一步       做法                      複雜度      代價
   A 稀疏   第1步 打分(連結結構)   只算部分 token 對          n√n / O(n)  漏掉被跳過的關係
   B 線性   第2步 正規化(換 softmax) 核函數+結合律重排運算順序   O(n)        近似不了 softmax 的全域競爭
```

注意兩條路的共同點：它們都是**近似**——用一點精度，換 O(n²)→次二次方甚至線性的成本。這就埋下 ch15 結尾、ch22 會回收的伏筆：**近似注意力在「精確 token 級回憶」上會吃虧**,所以 2026 年的混合架構，往往還是要保留一小撮精確的 softmax 注意力層當「精確檢索」的後盾（這條線一路通到 ch22 的 SSM 與混合共識）。

那有沒有**不近似、也能省**的辦法？有——但它省的不是「算多少」，是「怎麼擺記憶體」。FlashAttention（ch16）證明：O(n²) 的**時間**（分數的個數就是 n²）很難繞過，但 O(n²) 的**空間**可以——靠**不把整張 n×n 矩陣實體化在慢記憶體裡**，而是分塊算、用 online softmax 邊算邊合併。它在數學上和標準 attention **一模一樣**（精確、非近似），省的是記憶體的搬運，不是計算的精度。這是和方向 A/B 本質不同的第三招，ch16 專章講。

於是 Part IV 的三章地圖就清楚了，全是對本章這張「建不了索引的表」動手：

```text
   ch14(本章) 把代價算透、說清為什麼建不了索引 —— 共同動機
   ch15        方向 A 稀疏 + 方向 B 線性 —— 近似省法(動打分/正規化)
   ch16        FlashAttention(精確省空間)+ KV 共享 —— 不近似的省法
   (ch22)      乾脆換掉 attention 的主幹:SSM/Mamba —— 用固定狀態換掉 n×n
```

## 故障模式與直覺陷阱

| 直覺陷阱 / 故障 | 為什麼錯／在哪一步把你帶溝裡 | 怎麼自我察覺、正確版是什麼 |
|---|---|---|
| 「context 變長，成本就線性多一點」 | 把「資訊量」和「資訊間的關係數」搞混。塞進去的 token 確實是線性多（embedding、FFN 是 O(n)）,但 attention 算的是**兩兩關係**,是 n²。context 長 10 倍，attention 那層貴 100 倍，不是 10 倍。 | 問自己：這一步算的是「每個 token 各自做的事」（線性）還是「token 之間的兩兩關係」（平方）?attention 是後者。 |
| 「O(n²) 慢，加個索引不就好了」 | 用對了 DB 直覺、用錯了地方。索引靠鍵「可雜湊（等值）/可排序（有全序）」跳過大部分資料；attention 的「相似度」既非等值也無全序，且要**混合全部**(softmax 要看完才能正規化),query 還逐層現算現變。三個前提一個不滿足。 | 問：這個查找的「鍵」能雜湊或排序嗎？要的是「最像的一個」還是「按相似度混合全部」?是混合全部，索引就救不了。 |
| 「向量資料庫的 ANN 索引能加速，那 attention 也能套用」 | 三個關鍵差別：ANN 找 **top-k 少數**（可丟長尾）、是**近似**（可接受漏掉）、建在**相對靜止**的向量庫上；attention 要**混合全部**、要**精確**、Q/K/V **每層現算現變**。ANN 解的是不同的問題。 | 對照三點：你能丟長尾嗎？能接受近似嗎？資料靜止嗎？attention 三個都是「不行」。 |
| 「O(n²) 是因為有多頭/要做 n 次 softmax」 | 抓錯源頭（ch07 同款陷阱）。O(n²) 純粹來自 **QKᵀ 產生 n×n 個分數**。多頭是把 d 切開平行做，不改 n²;softmax 逐列做是 n² 量級，但它是 QKᵀ 的後果，不是獨立原因。 | 問：哪一步輸出了 n×n?是打分（QKᵀ）。把那一步輸出形狀寫出來，就抓到唯一的根。 |
| 「既然 O(n²) 這麼糟，RNN 的 O(n) 一定整體更好」 | 只看計算量、漏看平行性與長程衰減（ch07）。RNN 的 O(n) 步必須**循序**（不能平行）、句首到句尾**衰減**;self-attention 付 O(n²) 換來循序深度 O(1)（餵飽 GPU）和任兩點直達。便宜不等於整體更好。 | 分開看三個指標：總工作量、循序深度、最長路徑。RNN 贏第一個，輸後兩個。 |
| 「稀疏/線性注意力既然能降到 O(n),那標準 attention 該被淘汰了」 | 漏看「近似的代價」。稀疏賭被跳過的對不重要、線性的核近似不了 softmax 的全域競爭——兩者在**精確 token 級回憶**上常吃虧。所以 2026 年主流是**混合**（保留少量精確 softmax 層）,不是純近似（ch22）。 | 問：這個近似在「精確記住遠處某個 token」上會不會出錯？會，就不能全靠它。 |
| 「FlashAttention 把 O(n²) 解決了」 | 混淆時間與空間兩筆帳。FlashAttention 省的是 O(n²) 的**空間**（不實體化 n×n）和記憶體搬運，**時間仍是 O(n²)**（分數個數就是 n²）,而且它是**精確**不是近似（ch16）。它讓 O(n²) 跑得起更長，沒消滅 O(n²)。 | 記住：時間 O(n²) 難繞、空間 O(n²) 可省。FlashAttention 動的是後者。 |

**和後面章節的鉤子**:本章把「O(n²) 的代價」和「為什麼建不了索引」這兩件事講透，它們是 Part IV 與 ch22 的共同動機。ch15 走「近似省」（稀疏改打分、線性改正規化）;ch16 走「精確省空間」(FlashAttention 不實體化 + KV 共享);ch22 乾脆換掉主幹（SSM 用固定狀態換掉 n×n,但脊椎在那裡會「斷」——固定狀態裝不下精確回憶，正好印證本章「混合全部要付的代價」）。記住本章那句讀法規則：**長 k 倍貴 k² 倍**——它是你往後讀每一個高效注意力時，心裡那把尺。

## 紙上推演

### 推演題

**第 1 題 [12 分鐘]（★）** 把平方稅算到底。用本章的 n² 成長關係，不查表自己算：(a) 一個模型 context 從 8k 提升到 32k（長 4 倍）,attention 打分那一步的成本變幾倍？(b) 從 4k 提升到 1M（長 250 倍）呢？(c) 用一句話向一個沒學過複雜度的同事解釋「為什麼 context 翻倍、attention 的帳單翻了四倍」（提示：你付的是關係數不是字數）。

**第 2 題 [15 分鐘]（★★）** 為什麼建不了索引——逐條反駁「加個索引就好」。你的後端同事說：「attention 慢就是沒加索引嘛，給 key 建個 B-tree 或 hash 不就 O(log n)/O(1) 了？」請用本章的三個理由逐條回他：(a) 為什麼「相似度」這個鍵既不能 hash 也不能 B-tree?(b) 就算只想找「最相似的前幾個」,為什麼 softmax 還是逼你看完全部？(c) 為什麼「每層現算現變」讓索引就算建了也白建？

**第 3 題 [12 分鐘]（★★）** 兩個方向各動哪一步。不展開公式，只憑三步骨架推理：(a) 稀疏注意力「只算部分 token 對」,動的是三步骨架的哪一步？為什麼？(b) 線性注意力「把 softmax 換成核函數、用結合律重排」,動的是哪一步？(c) 用結合律的直覺說明：把 (QKᵀ)V 改成 Q(KᵀV) 為什麼能把成本從 O(n²·d) 降到 O(n·d²)?(d) 這兩條路有什麼共同代價、ch22 為什麼還要保留少量精確 softmax 層？

**第 4 題 [10 分鐘]（★★）** ANN 為什麼行、attention 為什麼不行。你做過 RAG,知道向量資料庫用 ANN 索引能在幾百萬個 embedding 裡快速找「最像的前 k 個」。請說清楚 ANN 和 attention 的三個關鍵差別（top-k vs 混合全部、近似 vs 精確、靜止 vs 每層現變）,並由此解釋「為什麼 ANN 的成功不能直接搬來救 attention」。

### 推演解答

**第 1 題。**
(a) 長 k=4 倍 → 成本 k²=**16 倍**。
(b) 長 k=250 倍 → 成本 k²=**62,500 倍**。（平方的脾氣：輸入只長了兩個半數量級，成本長了快五個數量級。）
(c) 範例答：「我們不是多塞了 n 個字進去而已——attention 要算的是這 n 個字**之間的兩兩關係**,有 n² 條。字數翻倍，關係數翻四倍（2²）,所以帳單翻四倍。你付的是關係數，不是字數。」

**第 2 題。**
(a) 「相似度」是 q·k 內積，一個多維度權衡出來的連續值。hash 索引要鍵能**雜湊成桶**（靠等值比對）,但相似度不是「相等/不相等」;B-tree 要鍵**有全序**能排序後二分，但高維向量「誰比較大」沒有一維全序可言（投影到任一軸排序都丟資訊）。既非等值、又無全序，兩種索引的前提都不滿足。
(b) 因為 attention 的第三步是**加權混合所有 value**,而 softmax 的分母是**所有** key 分數的 e 次方之和——要算任何一個權重的正確值，你都得先知道全部分數才能正規化。「只看前幾名」這個能跳過大多數資料的捷徑，被「softmax 要看完全部才能正規化」堵死。（這也正是 ch05 那個招牌反直覺：attention 不是取一筆，是混合全部。）
(c) self-attention 裡每個 token 的 Q/K/V 都是當場由學來的投影矩陣算出、而且**逐層在變**（這一層的輸出餵下一層，下一層的 K 全變）。索引建在相對靜止的資料上才划算；這張表每讀一次（每過一層）就整張重寫，你建好索引它就作廢，等於白建。

**第 3 題。**
(a) 動**第一步「打分」**。稀疏改的是「誰跟誰之間需要算分數」的**連結結構**——本來 n×n 全連，稀疏只連局部窗口+少數全域樞紐，被跳過的對直接當零，於是要算的分數從 n² 降到 n√n 或線性。
(b) 動**第二步「正規化」**。把 softmax 這個正規化函數換成可拆開的核函數 φ,讓注意力寫成 φ(Q)φ(K)ᵀ 的形式——換掉 softmax 是為了解開「必須先算出整張 n×n 才能正規化」的死結。
(c) 結合律：(φ(Q)·φ(K)ᵀ)·V 先算 φ(Q)φ(K)ᵀ 是 n×n（大）,再乘 V;改成 φ(Q)·(φ(K)ᵀ·V) 先算 φ(K)ᵀV 是 **d×d**（小，與 n 無關）,再被 φ(Q) 乘。中間矩陣從 n×n 縮成 d×d,當 n≫d 時省得驚人——成本從 O(n²·d) 降到 O(n·d²),對 n 線性。這正是你在查詢優化裡把 (A·B)·C 重排成 A·(B·C) 省大中間結果的同一招。
(d) 共同代價：兩者都是**近似**——稀疏賭被跳過的對不重要（漏長程關係）、線性的核近似不了 softmax 的全域競爭（尤其精確回憶遠處某個 token 時）。所以 ch22 的混合架構往往保留一小撮（如 1-in-8）**精確 softmax 層**當「精確檢索」的後盾，其餘層才用便宜的近似/SSM——用少量精確層補住近似省下來的洞。

**第 4 題。**
三個關鍵差別：
- **top-k vs 混合全部**:ANN 找「最像的前 k 個」,可以大方丟掉長尾；attention 要按 softmax 權重**混合全部** value,連權重 0.09 的都在貢獻，丟不得。
- **近似 vs 精確**:ANN 接受偶爾漏掉真正最像的那個（換速度）;標準 attention 要精確的 softmax。
- **靜止 vs 每層現變**:ANN 的索引建在相對靜止的向量庫上（文件 embedding 建好不太動）;attention 的 Q/K/V 每層現算現變，索引建了就作廢。
由此：ANN 的成功建立在「可丟長尾、可近似、資料靜止」三個 attention 都不滿足的前提上，所以不能直接搬來救 attention。能搬的部分（近似、可丟一些對）正是 ch15 稀疏/線性在做的事——而它們也就因此繼承了 ANN 的代價：近似會在精確回憶上吃虧。

## 自我檢核

口頭自答，講得出來才算過：

1. 用一句話說「為什麼 context 長 10 倍、attention 那層貴 100 倍」（提示：平方，且付的是兩兩關係數不是字數）。
2. O(n²) 到底從三步骨架的哪一步、為什麼長出來？（打分：n 個 query × n 個 key = n×n 個分數；softmax 和混合是它的後果。）
3. attention 這張表為什麼建不了傳統索引？把三個理由講給一個只懂 DB 索引的同事聽（相似度非等值無全序、softmax 要混合全部看完才正規化、Q/K/V 每層現算現變）。
4. 向量資料庫的 ANN 索引明明能在高維向量裡快速查，為什麼救不了 attention?(top-k vs 混合全部、近似 vs 精確、靜止 vs 每層現變。)
5. 想殺掉 O(n²) 的兩個直覺方向是什麼？各動三步骨架的哪一步？（少算一些對=改打分的連結結構=稀疏；換掉 softmax 用結合律重排=改正規化=線性。）
6. 線性注意力靠矩陣乘法結合律省成本——把 (QKᵀ)V 重排成什麼？為什麼這樣就從 O(n²·d) 變 O(n·d²)?（重排成 Q(KᵀV),中間矩陣從 n×n 縮成 d×d。）
7. 稀疏和線性都是「近似」,代價是什麼？為什麼 2026 年主流是「混合」而非純近似？（近似在精確 token 級回憶上吃虧；混合保留少量精確 softmax 層補洞。）
8. O(n²) 的「時間」和「空間」是同一筆帳嗎？哪一個能不靠近似就省下來、靠什麼？（不是同一筆；空間可靠 FlashAttention 不實體化 n×n 來精確省下，時間難繞，ch16。）
