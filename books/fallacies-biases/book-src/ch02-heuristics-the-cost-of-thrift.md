## 從你已知的出發

你寫過 cache。你不會因為有 cache 就覺得自己偷懶——你知道 cache 是**正當的工程選擇**：每次都回源算（origin）正確，但慢、貴、扛不住流量；於是你存一份「夠新、夠準」的近似，絕大多數請求直接命中，省下的那筆算力與延遲是實打實的收益。cache 不是「壞掉的回源」，它是「在對的脈絡下，用一個可接受的近似換速度與成本」。

捷思就是大腦的 cache。

而你也知道 cache 會出哪種事：**stale**。資料變了、cache 沒失效，你拿著一份「曾經對、現在錯」的值繼續服務，而且——這是最要命的一點——**命中的那一刻你毫無感覺**，回應一樣是毫秒級、一樣理直氣壯。你不會在命中時收到「警告：這份可能過期」。你要等到下游對帳對不上、用戶投訴、或某次回源才發現它早就不對了。

把這兩件事疊起來，你就已經懂了本章的全部骨架，只是還沒給它名字：

- **cache 命中 = 系統一跑捷思**：自動、毫秒、不思考，給你一個「夠好的近似」。
- **cache 是正當的 = 捷思是理性的**：在它適用的脈絡裡，它平均夠好，省下的算力是真收益，不是偷懶。
- **stale = 偏誤**：環境變了、捷思沒換，你拿著「曾經對的規則」用在「現在不對的脈絡」——而且**命中那一刻你察覺不到**。

這就是全書的中央張力，用你的母語講一遍：**謬誤不是愚蠢，是在錯的脈絡用對的捷思——而且你察覺不到自己正在犯。** 你不會因為 cache 偶爾 stale 就把所有 cache 拆掉（那會讓系統慢到不能用）；同樣地，本書的目標不是叫你「別用捷思」（做不到，也不該），而是讓你認得**哪些捷思在哪些脈絡會 stale**，並在那些地方裝上一道失效機制。

你天天信任的近似還不只 cache。負載預估時你用「拇指法則」（rule of thumb）：「這台機器扛得住，差不多 8 成 CPU 以下就沒事」；容量規劃時你抓「去年雙十一的 1.5 倍」；rate limiter 取樣估流量而不是逐筆精算；估點（story point）時你說「這跟上次那個差不多大，給 5 點」。這些全是捷思——**不保證最優、但夠快夠省、平均夠好**。你不會為了估一個 story point 去建排隊模型。你信任近似，因為在絕大多數脈絡裡近似就夠了。

本章接下來要做的，就是把「捷思為什麼是理性的」講到你心服口服，然後立刻給你看它在哪一類脈絡會系統性翻車——這樣後面三章（可得性、代表性、錨定）你就有一把統一的尺去量它們。

## 捷思是什麼：演算法的人都認得的東西

先把名詞釘死，省得後面打架。本書統一用「**捷思**」（heuristic，啟發法只在此標註一次，之後一律捷思）。

對寫過程式的你來說，heuristic 一點都不陌生——它在演算法裡有精確的意思：**一個不保證得到最優解、但通常夠快、答案夠好的策略**。A\* 搜尋的 heuristic function 估「離終點還有多遠」，估得好就剪掉大片搜尋空間；若啟發式函數滿足可容許性（admissible，估算值不大於實際值），A\* 保證找到最短路；若不可容許，則不保證最短路，但讓你在可接受時間內拿到一條夠好的路。貪婪演算法（greedy）每步挑當下最好的，不回頭——常常不是全域最優，但又快又簡單。load balancer 的「round-robin」「最少連線」也是捷思：不去解一個全域最優分配問題，用一條簡單規則換掉天文數字的計算量。

人腦的捷思是同一回事，只是跑在生物硬體上、且你看不到原始碼。它的形狀是：

```text
捷思的本質（演算法人版本）

  正規解法（系統二）            捷思（系統一）
  ─────────────────────        ─────────────────────
  窮舉/精算/查全部資料          抓一個好取得的線索當代理
  保證最優或正確                平均夠好、偶爾翻車
  慢、費力、會累                快、自動、省力、永遠開著
  例：解貝氏、查實際基率         例：「像不像」「好不好想起來」

  替換的核心動作：
  「真正想知道的那個難問題」 → 偷偷換成「一個好回答的近似問題」
        （目標問題：機率多少？）   （代理問題：它有多像原型？）
```

最後那兩行是康納曼後來描述捷思的關鍵詞，他叫它 **attribute substitution（屬性替換）**：當你被問一個難算的問題（「這件事的機率多少？」），系統一悄悄把它換成一個好回答的問題（「這件事有多像我心裡的典型？」），然後拿後者的答案回報前者——**而你以為自己回答的是原來那題**。代理問題答得快、答得順，於是系統二沒理由懷疑，放行。

這個「換題」就是後面所有捷思偏誤的共同生產線。ch03 的可得性，是把「頻率多少」換成「例子多好想起來」；ch04 的代表性，是把「機率多少」換成「有多像原型」。本章不展開它們的細節（那是後三章的事），只要你先記住這個替換動作的形狀。

## 捷思是理性的：生態理性與「省力不是偷懶」

這節是本章的價值核心，也是最容易被講歪的地方。很多談偏誤的科普走的是「人類好蠢，看，又錯了」的調子。本書明確拒絕這條路線——不只因為它廉價，更因為它**事實上不對**。

捷思的正當性有兩條獨立的論證線，都不靠「人很笨」。

**第一條：在資訊不足、時間不夠時，捷思是理性的省力策略。** 經濟學家賽門（Herbert Simon）很早就指出，「完全理性」（把所有選項與資訊都算過、選出真正最優）是個**現實中達不到的假想**：你沒有無限算力、沒有完整資訊、沒有無限時間。他提出**有限理性（bounded rationality）**與**滿意化（satisficing，satisfy + suffice 的合成詞）**——人不是去找「最優解」，而是設一個「夠好就收」的門檻，碰到第一個越過門檻的選項就停。賽門 1978 年拿了諾貝爾經濟學獎，核心貢獻正是把「理性」從「最優化」重新定義成「在認知與資源限制下夠好的決策」（2026-06 查證）。

用你的話：你 debug 時不會把整個 codebase 讀完才動手，你抓最可疑的幾個地方先看——這不是不嚴謹，這是在「時間有限、不可能讀完」的現實約束下的**正確策略**。窮舉是奢侈品，多數情況你付不起。

**第二條（更強、也更晚才被講清楚）：在對的環境結構下，捷思甚至可能比「精算」更準。** 這是吉仁澤（Gerd Gigerenzer）與其團隊的**生態理性（ecological rationality）**主張：一個捷思的好壞**不能脫離它運作的環境來評判**——它「理性」的程度，取決於它**多貼合所處環境的結構**。他們研究一批「快而省（fast-and-frugal）」的捷思，發現一件反直覺的事：**在不確定、資料雜訊大的真實環境裡，看更少的資訊、用更簡單的規則，有時反而做出更準的判斷**（這叫「less-is-more」效應，2026-06 查證）。直覺的理由是：複雜模型在雜訊大、樣本有限時容易**過配適（overfit）**——把雜訊當訊號學進去；簡單規則參數少、不容易被雜訊帶偏，反而更穩。

這個你一定有感。一個塞滿特例分支、handle 了一堆只出現過一次的 edge case 的函式，往往比一個乾淨的簡單版本更脆、更難維護、在新情境下死得更難看——因為它**過配適**了過去那幾次的具體狀況。簡單規則的 robustness，是工程直覺裡早就有的東西。

把兩條線合起來，本章第一個要你記牢的結論是：

> **捷思不是偷懶，是理性的省力。** 它是大腦在「資訊不足、時間不夠、算力有限」這個人類永遠擺脫不了的約束下，演化／學習出來的近似策略。在它適用的環境裡，它平均夠好，甚至可能比精算更穩。**把每個偏誤都罵成「人很笨」，等於把 cache 罵成「壞掉的回源」——既不公道，也讓你看不見問題真正在哪。**

問題從來不在捷思本身，而在**錯置**：環境結構變了、或這個脈絡根本不在捷思的適用範圍內，捷思卻照跑。下一節就把這件事的形狀畫出來。

## 偏誤 = 捷思被錯置：同一機制的兩面

捷思和偏誤不是兩個東西，是**同一個機制的兩面**。同一條規則，在它的「生態棲位」裡運作就是捷思（省力、夠好），被搬到棲位外運作就是偏誤（系統性翻車）。沒有「壞的捷思」，只有「被放錯地方的捷思」。

回到 cache 的比喻，這件事就一目了然：

```text
同一機制的兩面：捷思 ←→ 偏誤

  脈絡對          脊椎機制              脈絡錯
  ──────          ──────              ──────
  cache 命中  ←─  存一份夠好的近似  ─→  cache stale
  （省力、對）      （省力，可能過期）      （省力、但錯，且無感）

  捷思        ←─  抓代理線索估難題  ─→  偏誤
  （平均夠好）      （換題作答）          （錯置到不該用的脈絡）

  關鍵：兩邊跑的是「同一個動作」。差別只在環境結構合不合。
        而且——命中和 stale，當下感覺完全一樣。
```

「同一個動作、環境合不合決定它是捷思還是偏誤」這句話，是本書接下來每一章的讀法。寫到任何一個認知偏誤時，本書都會逼自己回答三件事，請你也用這三問去檢核我有沒有偷懶：

1. **這個捷思原本對在哪？**（它把哪個難問題換成哪個好回答的代理問題，省了什麼，在什麼環境裡這個替換是對的）
2. **它在哪個脈絡翻車？**（環境結構怎麼變了，使得「代理線索」不再是「目標答案」的好近似）
3. **為什麼當事人察覺不到？**（為什麼系統一給的答案感覺一樣對，系統二沒有理由停下來覆核）

第三問是全書的核心張力，也是最該慢下來的地方。cache 命中和 stale 之所以難辦，正因為它們**在命中那一刻長得一模一樣**——同樣的延遲、同樣的格式、同樣的理直氣壯。捷思的偏誤也是：系統一給「女性主義銀行員更可能」（ch11）和給「銀行員更可能」，**主觀的確定感是一樣的**；你不會在犯合取謬誤的當下感到一絲遲疑。**偏誤之所以危險，不是因為你答錯，而是因為你答錯時的感覺，和你答對時一模一樣。** 這是本書反覆要把你按在椅子上的那句話。

這裡要先守住一條重要的**分野**（這是全書的概念骨架，不能混）。上面「對的捷思／錯的脈絡」這套框架，描述的是**認知偏誤**——捷思在錯場合誤觸發。本書還收另一類完全不同機制的陷阱：**形式謬誤**（肯定後件、否定前件，Part III）。形式謬誤錯在**演繹推論的結構本身無效**（P→Q 為真，不保證 Q→P 為真），**與內容無關、也不是某個捷思被錯置**。**不要把肯定後件講成「一個用錯場合的捷思」**——它不是捷思，是無效的推論形式。到 ch05 我們會看到，雙系統對形式謬誤的回扣只有一個方向：「為何系統一會覺得一個無效推論看起來合理」（系統一對邏輯形式不敏感，憑表面合理度放行），而**不是**「它跑了哪個捷思」。本章與接下來兩章談的是捷思／偏誤這一面；遇到形式謬誤時，這套 cache 比喻就不適用了，請先把它放一邊。

## 怎麼認得自己正在用捷思：徵兆

既然偏誤的核心難處是「察覺不到」，那實務上唯一的著力點，就是學會辨認**「我現在正在跑捷思」的徵兆**——不是等錯了才知道，而是在系統一交卷、系統二準備蓋章放行的那一瞬間，攔下來問一句「這題該走快取嗎」。

徵兆都是「主觀流暢度」的訊號。系統一的答案有一種特定的手感：

```text
「我正在跑捷思」的徵兆清單（攔截點）

  徵兆                          對應的快取狀態        該做的攔截動作
  ──────                        ──────              ──────
  答案來得太快太順、不費力        cache 命中（毫秒）    問：這題的代價配得上快取嗎？
  「感覺顯然」「一看就知道」      命中且高信心          高信心 ≠ 高正確；查一下回源
  完全沒感覺自己在「算」          純自動、系統二沒上線  停三秒，把問題講出聲（外化）
  第一個冒出來的數字後面          錨定已發生（ch04）    主動找一個獨立來源的數字
  「這跟上次那個一模一樣」        代表性已啟動（ch04）  問：上次那個的基率是多少？
  賭注大、不可逆、罕見           最不該走快取的脈絡    強制走系統二（冷路徑）
```

注意最後一列的判讀法：**不是所有捷思都要攔**。攔截是有成本的（系統二慢、費力、會累），全部都精算你會癱瘓——那等於把所有 cache 拆掉。真正的工程判斷是**選擇性失效**：在「賭注大、不可逆、罕見、或你已知這類脈絡會 stale」的地方，才付系統二的成本去回源；其餘照走快取。這跟你設計 cache 失效策略一模一樣——你不會每次請求都回源，你只在「資料剛變動」「這筆特別重要」時主動失效。

這也是為什麼「光知道有偏誤」幾乎不能讓你免疫（ch18 會把這條誠實講透）：知道 cache 會 stale，並不會讓你在命中的那一刻自動感覺到它 stale——命中就是命中，手感一樣順。真正有用的，是事先在「已知會 stale 的脈絡」上裝好失效規則（改環境），而不是要求自己「每次命中都更努力懷疑一下」（改人，幾乎無效）。這條線索本章先埋著，Part VI 收。

## worked example：同一個判斷，捷思版 vs 正規版

把抽象的「對的脈絡 vs 錯的脈絡」變成一個你能跟著走的具體例子。場景：你是 on-call，凌晨收到告警，要快速判斷「**這次 504 逾時，是不是 DB 又慢查詢了**」。我們把同一個判斷用兩種方式做一遍，標出捷思在哪個脈絡會給對答案、哪個脈絡會翻車。

```text
判斷題：這次 504，是不是 DB 慢查詢？

── 捷思版（系統一，3 秒）────────────────────────────────
  動作：抓「最好取得的線索」當代理
        線索 = 「上次 504 就是 DB 慢查詢」（好想起來、剛發生過）
  替換：把難問題「這次的原因是什麼」
        悄悄換成「這次有多像上次那次」     ← 代表性 + 可得性（ch03/04）
  輸出：「八成又是 DB。」                  ← 來得快、感覺顯然、高信心
  代價：幾乎為零（不用查任何 dashboard）

── 正規版（系統二，5 分鐘）──────────────────────────────
  動作：查實際證據，估「在 504 裡，DB 慢查詢佔幾成」
  需要的量（先別算，只列出來）：
    - 504 的各種成因基率（DB 慢查、上游逾時、GC、網路、限流…）
    - 這次的具體證據（DB 連線數、慢查詢 log、上游延遲…）
  方法：看 trace、比對時間軸、排除法
  代價：5 分鐘 + 清醒的腦

── 兩個脈絡，捷思的命運不同 ────────────────────────────
  脈絡 A：你的系統九成的 504 真的來自 DB 慢查詢（環境結構穩定）
    → 捷思版「八成又是 DB」≈ 正確，省下 5 分鐘  ✔ 這是 cache 命中
       「上次是 DB」是「這次原因」的好近似——因為環境真的這樣分布

  脈絡 B：上週剛上了新的上游服務、流量結構變了，504 的主因已悄悄
          換成「上游逾時」，但你的捷思還停在「504 = DB」的舊版
    → 捷思版「八成又是 DB」= 系統性錯，你會去翻 DB 翻半天、繞遠路  ✘
       這是 cache stale：規則曾經對，環境變了，規則沒換——
       而且你在咬定「又是 DB」的當下，毫無遲疑，手感和脈絡 A 一樣順

── 自我複核：捷思錯在哪一步？ ───────────────────────────
  不是錯在「用了上次的經驗」（那正是省力且常常對的捷思）。
  錯在「環境結構已經變了（主因換人了），捷思沒跟著失效」。
  攔截點：徵兆是「答案來得太快、感覺顯然、賭注其實不小（凌晨故障）」
         → 這正是該付系統二成本回源（查 trace）的脈絡。
```

走完這一遍，本章的主張就落地了：捷思版**在脈絡 A 是對的、該用的、省力又準**——罵它「偷懶」是不公道的；它**在脈絡 B 翻車，不是因為它變笨了，是因為環境變了而它沒失效**。你要對付的從來不是「用不用捷思」，而是「認得這次是脈絡 A 還是脈絡 B」——而最難的是，**站在當下，A 和 B 的手感一模一樣**。

## 直覺的陷阱

捷思這個概念本身，就埋著好幾個會把你帶下溝的直覺。挑三個具名的講。

```text
陷阱清單（捷思總論版）

  你以為                          其實                          自察徵兆
  ──────                          ──────                        ──────
  ① 偏誤 = 大腦壞了 / 人很笨       偏誤 = 對的捷思用錯脈絡。       你開始用「人類就是
                                  捷思在棲位裡是理性的省力。      笨」當解釋——這通常
                                  罵它笨＝罵 cache 是壞掉的回源。  代表你還沒搞懂機制。

  ② 知道有這偏誤就能免疫          知道 cache 會 stale，不會讓你   你心想「我懂偏誤，
     （「我懂偏誤，我不會中」）     在命中那刻感覺到它 stale。      所以我比較不會中」
                                  光知道幾乎不改善（ch18/ch19）。  ——這念頭本身就是
                                                                偏誤盲點（ch19）。

  ③ 捷思都該被消滅 / 換成精算      攔截有成本，全精算會癱瘓。      你想「以後每個判斷
     （「理性 = 凡事都慢慢算」）    對的策略是選擇性失效，          都慢慢算」——你撐
                                  不是拆掉所有快取。              不過三天，且多數
                                                                判斷精算並不更準。
```

**陷阱①「偏誤＝人很笨」** 是最普遍、也最該先拆的。它不只廉價，更**讓你看不見問題真正在哪**。如果偏誤是「腦子壞了」，那無解、也無從防；但如果偏誤是「對的捷思被錯置」，那問題就變得可操作——你要找的是「環境結構在哪變了」「這個捷思的棲位邊界在哪」，並在邊界外裝失效機制。把人罵笨會讓你停在情緒上；把它看成錯置會讓你動手解決。本書全程拒絕陷阱①——這不是替人類護短，是因為「錯置」才是事實、也才有用。

**陷阱②「知道就能免疫」** 是這本書最想讓你提防的自我感覺。你讀完二十章，會「知道」一大堆偏誤的名字——然後很自然地以為自己因此比較不會中。錯。知道 cache 會 stale，跟你在命中那一刻能不能感覺到它 stale，是兩件事；前者是宣告式知識，後者是即時偵測，而即時偵測幾乎不會因為前者而改善。更糟的是，「我懂偏誤所以我比較理性」這個念頭，本身就是 ch19 的**偏誤盲點**——你對別人的偏誤一眼看穿，對自己的全開。真正讓你少犯的，不是「知道」，是「改環境裝失效機制」（ch18）。請現在就對「我讀完這本書會更理性」這句話打個折，並把它記下來，到 ch19 我們算總帳。

**陷阱③「捷思都該換成精算」** 是工程師特別容易掉進去的——你習慣「正確性優先」，於是直覺認為理性就是凡事都走系統二慢慢算。但攔截系統一是有成本的（慢、費力、會累、且系統二自己也會偷懶），把所有判斷都精算的人不是更理性，是會**癱瘓**——而且如前所述，在雜訊大的環境裡精算還不一定更準（過配適）。理性的標的不是「消滅捷思」，是**選擇性失效**：認得哪些脈絡值得付系統二的成本，其餘信任快取。

## 紙上推演

以下三題，純紙上，不用查任何工具，也不用跑任何程式。每題先自己想到一個完整答案再看解答。

**推演題一（★，[10 分鐘]）：你工作中的三個拇指法則。**
列出你日常工作中真實在用的三個「拇指法則／捷思」（例如容量估算、值班分流、estimate、code review 的注意力分配……）。對每一個，回答三件事：(a) 它原本對在哪——把哪個難問題換成哪個好回答的代理問題、省了什麼；(b) 在什麼**脈絡**下它會失準（環境結構怎麼變會讓它 stale）；(c) 你會放一個什麼「徵兆」當攔截點。

**推演題二（★★，[15 分鐘]）：把一句話翻成「捷思換題」的形狀。**
拿這句你大概說過的話：「**這個服務看起來像之前那個出事的，八成也會出事。**」請：(a) 指出這裡「真正想知道的難問題」是什麼、「被偷換成的好回答代理問題」是什麼；(b) 說明在什麼脈絡下這個替換是對的（捷思命中）、什麼脈絡下會翻車（stale）；(c) 點出這個替換忽略了一個什麼量（提示：和「這類服務實際出事的比例」有關——但**機率怎麼算**指向《馴服隨機》，這裡只要指出「忽略了那個量」即可，不必計算）。

**推演題三（★★，[15 分鐘]）：把「捷思不是偷懶，是理性的省力」講給一個工程師聽。**
不准用「人類很笨」「要更理性」這類話。要求：(a) 用一個工程比喻（cache、A\* heuristic、greedy、satisficing 任選）說明捷思為何正當；(b) 講清楚「同一個機制，脈絡對是捷思、脈絡錯是偏誤」；(c) 用一句話收尾，講清楚「偏誤之所以危險，不在你答錯，在你答錯時的手感和答對一樣」。目標是讓對方能用自己的話複述。

### 推演解答

**題一解答。** 這題沒有標準答案，重點在你有沒有把三件事都答出「機制」而不是「感覺」。一個達標的範例（容量估算）：(a) 難問題是「下個季度真實尖峰負載是多少」，需要建模、看歷史分布、算成長曲線；捷思把它換成「去年同期 × 1.5」——省掉建模，用一個好取得的歷史錨。它在「業務成長平穩、流量結構不變」的環境裡命中。(b) 失準脈絡：突然有行銷活動、產品改版改變了用戶行為、或上了個會放大讀寫的新功能——環境結構變了，「× 1.5」這條規則 stale。(c) 攔截徵兆：「這個數字我幾乎沒想就喊出來了」「這季有沒有任何非常規的變動」——只要答案是有，就強制回源（重新建模）。**常見錯路**：只答「它會失準因為估不準」——這是同義反覆，沒指出**環境結構的哪個變化**讓它失準。捷思失準一定對應一個「環境變了、規則沒換」，講不出那個變化就是還沒懂。

**題二解答。** (a) 難問題：「這個服務**實際上**會不會出事」（牽涉它的真實架構、依賴、負載、與此刻的健康狀態）。被偷換成的代理問題：「這個服務**有多像**之前那個出事的」。這正是 ch04 的代表性捷思——把「機率」換成「像不像原型」。(b) 命中脈絡：如果「長得像」在你的環境裡真的和「會出事」高度相關（例如它們共用同一套有問題的框架、同一個會雪崩的依賴），那「像 → 會出事」是個夠好的近似，省下逐一排查。翻車脈絡：兩個服務只是**表面**像（同樣是 Node 寫的、同樣有 cache 層），底層成因毫無關係——這時「像」對「會出事」毫無預測力，捷思 stale。(c) 它忽略的量是「**像這樣的服務，實際出事的基率**」——「之前那個」可能只是一個生動的單一案例，而你沒問「這類服務一百個裡實際出幾個事」。這就是基率忽視的引信（ch10/ch04），但**演算法指向《馴服隨機》**，這裡只需指認「忽略了基率」。**常見錯路**：直接斷言「這句話是錯的」——不對，它在命中脈絡是對的、該用的捷思；它不是「錯的判斷」，是「在錯脈絡會錯置的捷思」。把它一律打成謬誤，等於犯了陷阱①。

**題三解答。** 一個達標的口頭版本：「你想想 cache。每次回源算最準，但慢、貴、扛不住流量，所以我們存一份夠好的近似，絕大多數請求直接命中——這不是偷懶，是正當的工程選擇，省下的算力是真收益。大腦也一樣：碰到難算的問題，它不會每次都精算，它抓一個好取得的線索當代理，給你一個夠快夠好的答案。在這個近似適用的環境裡，它就是對的、該用的，甚至比硬算更穩。**偏誤不是另一個東西——偏誤就是這份近似 stale 了**：環境變了、規則沒換，你拿著曾經對的捷思用在現在不對的脈絡。同一個機制，脈絡對是捷思、脈絡錯是偏誤。而最麻煩的是——**cache 命中和 stale，在命中那一刻手感完全一樣**；偏誤危險的地方不是你答錯，是你答錯時的確定感，和你答對時一模一樣，所以你當下完全不會懷疑。」**常見錯路**：講成「所以我們要少用捷思、多動腦」——這違背本章主張（攔截有成本、全精算會癱瘓且不一定更準），也沒抓到重點。重點不是少用，是認得棲位邊界、在邊界外裝失效。

## 自我檢核

口頭自答，講得出來才算過關。優先答「為什麼」。

1. 為什麼說「捷思不是偷懶，是理性的省力」？請給**兩條獨立**的論證（提示：一條來自賽門的有限理性，一條來自吉仁澤的生態理性／less-is-more），不要只說「因為比較快」。
2. 「捷思」和「偏誤」是兩個不同的東西嗎？用 cache 命中與 stale 解釋你的答案。
3. 康納曼說的「屬性替換（attribute substitution）」是什麼意思？它怎麼當「後面所有捷思偏誤的共同生產線」？
4. 為什麼「光知道有某個偏誤」幾乎不能讓你免疫？用 cache 的「命中那刻感覺不到 stale」來講。
5. 攔截系統一是有成本的，所以理性的策略不是「凡事精算」而是「選擇性失效」。你會在什麼樣的脈絡才付這個成本？（提示：賭注、可逆性、稀有度、已知棲位邊界）
6. 為什麼本書堅持不用「人類很笨」當解釋？這個拒絕除了「不夠厚道」，在**實務上**讓你多看見了什麼？
7. 形式謬誤（如肯定後件）為什麼**不能**用本章「對的捷思／錯的脈絡」框架去解釋？這條分野到 ch05 會怎麼接？
