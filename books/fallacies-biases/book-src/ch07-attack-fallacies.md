## 從你已知的出發

你每天在 code review 裡打仗。不是吵架那種打仗——是「我覺得這個設計有問題，你覺得沒有」這種，要靠論證解決的打仗。而 code review 是一個極好的場域，能讓你親眼看清本章三個謬誤的形狀，因為它們**全部**都在那裡反覆上演，只是你大概沒給它們起過名字。

先看一段你一定遇過的對話。你在 PR 底下留言：「這個 endpoint 沒有對 `limit` 參數設上限，別人傳 `limit=10000000` 進來會把 DB 拖垮。」對方回：「**所以你是說我們要對每一個參數都做防禦性驗證？那這專案永遠寫不完。**」

停。你有沒有覺得哪裡不對，但一時說不出哪裡？你說的是「這**一個**沒有上限的分頁參數會出事」，對方回的是「你主張**每一個參數**都要驗證」——他打的根本不是你說的那句話。他先把你的立場換成一個明顯過頭、明顯站不住的版本（「凡參數皆驗證」），然後從容地把那個版本打倒，最後讓在場的人覺得「啊你的要求太誇張了」。你原本那句精準的、只針對一個 endpoint 的意見，就這樣在沒被正面回應的情況下蒸發了。這叫**稻草人（straw man）**：他立了一個稻草做的假人替你站著，打倒假人，宣稱打倒了你。

再看第二種。同一個 PR，另一個 reviewer 路過，留言：「這 PR 是上週剛 onboard 的新人寫的吧？那大概問題不少，我先 request changes。」——他連 diff 都沒打開。這句話攻擊的是**寫程式的人**（新人、經驗少），而不是**這段程式本身**（它到底有沒有 bug）。新不新人跟這段 code 對不對，是兩件事；老鳥也會寫出 race condition，新人也可能交出乾淨的 PR。這叫**人身攻擊（ad hominem）**：把箭頭從論證轉到提論證的人身上。

第三種更隱蔽，因為它常常**看起來很合理**。設計討論裡有人說：「我們應該用這個架構，因為 Netflix 就是這樣做的。」——Netflix 這樣做，跟「我們」這樣做對不對，中間隔著一整個「我們的規模、流量、團隊、約束跟 Netflix 一不一樣」。如果不一樣，「Netflix 做了」對「我們該做」幾乎沒有證據力。這叫**訴諸權威（appeal to authority）**——但這裡要先埋一根刺，後面整章都圍著它轉：**訴諸權威不一定是謬誤**。「資安團隊評估過這個加密庫有已知漏洞」也是訴諸權威，而它是完全正當的證據。本章最值得你帶走的一件事，就是學會分辨這兩者的界線。

這三個謬誤有一個共同形狀，也是本章的脊椎：**它們打的都不是該打的那個對象**。稻草人打的是「被你捏歪的立場」，人身攻擊打的是「提論證的人」，訴諸權威是拿「誰說的」替代「說得對不對」。要抓住它們，你得先做一件 ch06 教過的事——**把論證真正的樣子重建出來**：對方真正主張什麼、隱藏前提是什麼、這個攻擊到底落在哪。沒重建，你只會「覺得怪怪的」卻說不出口，然後在會議上輸給一個其實沒回應你的人。

## 非形式謬誤：錯不在形式，在「你重建之後」才看得到

先把這一整個 Part 的定位釘死，免得跟 Part III 混了。

Part III 的形式謬誤（肯定後件、否定前件）有一個很乾淨的性質：**錯在結構、與內容無關，你不用知道句子在講什麼就能判它無效**。「P→Q，Q，∴P」這個形狀，不管 P、Q 代入什麼都無效——代入「下雨→地濕」是無效，代入「部署→告警」也是無效。你可以純機械地查它的結構。

非形式謬誤不是這樣。它的「錯」**藏在內容與脈絡裡**，你必須先搞懂這段話在講什麼、對方真正主張什麼、有哪些沒講出來的前提，才判得出它有沒有犯規。同一句話，換個脈絡可能就不是謬誤了——「Netflix 這樣做」在「我們規模跟 Netflix 相近、且這是 Netflix 的核心專長領域」的脈絡下，會變成一個有點分量的證據；在「我們是三人小團隊做內部工具」的脈絡下，就是個沒力的訴諸權威。**脈絡決定它是不是謬誤**，這正是本 Part 名字「脈絡壞了」的意思。

```text
形式謬誤 vs 非形式謬誤：錯在哪一層

  形式謬誤（Part III）              非形式謬誤（Part IV，本章起）
  ──────────────────              ──────────────────────────
  錯在「推論結構」                  錯在「脈絡與隱藏前提」
  與內容無關，可機械檢查            必須先重建論證、補出前提才抓得到
  代入什麼都無效                    同一句換個脈絡可能就不是謬誤
  例：肯定後件 P→Q,Q,∴P            例：稻草人、人身、訴諸權威
  抓法：對照四推論表                抓法：重建對方真正主張 → 看攻擊落在哪
```

這就是為什麼 ch06 的「重建隱藏前提（enthymeme）」是整個 Part IV 的鑰匙。攻擊型謬誤之所以難當場抓到，正因為它們利用的是「你**沒有**把對方的論證攤開來擺在桌上」——一旦你把對方真正主張的命題、跟對方實際攻擊的命題並排寫出來，謬誤就無所遁形：你會看到那兩個命題**根本不是同一個**。

雙系統在這裡的角色（回扣 ch01 的脊椎）跟形式謬誤有點像、又不完全相同。系統一對「論證的表面說服力」反應極快：對方語氣篤定、句子聽起來通順、結論朝著你本來就同意的方向——系統一就給一個「嗯有道理」的放行訊號，而**它不會自動去做「對方攻擊的，是不是我真正主張的那個東西」這項檢查**。那項檢查是系統二的活：慢、費力，要把兩個命題並排比對。系統二一偷懶，稻草人就過關了。所以攻擊型謬誤的「察覺不到」，落在一個很具體的地方：**你忙著反駁對方丟給你的那句話，完全沒注意到那句話不是你說的**。

> 嚴謹度標示：本章是「直覺版＋工程師的嚴謹」。謬誤的定義（哪種攻擊算稻草人、訴諸權威何時正當）在邏輯學界本身有細緻爭論，沒有像「四推論有效性」那樣的恆真標準答案；本章給的是主流、可操作、能讓你當場辨認與重建的版本，每一步都能口頭講出理由，但不宣稱它是唯一正確的劃界。凡引用具體研究或史實處另標時點。

## 稻草人：你打的不是他說的

稻草人的機制只有一句話：**先扭曲對方的立場（換成一個更弱、更極端、更好打的版本），再打倒那個版本，宣稱打倒了對方。**

它原本對在哪？這裡要誠實——稻草人跟訴諸權威不同，它**沒有一個「正當版本」**。它本質上就是一個論證上的犯規。但它之所以**有效**（為什麼人會上當、為什麼連好意的人也常不自覺地犯），是有機制的，而那個機制值得拆開：

**第一，簡化是必要的，扭曲是順手的。** 在真實對話裡，你本來就得「複述對方的意思」才能回應——「你的意思是說……」這個動作是必要的、也是省力的（系統一順手做的摘要）。問題是，複述的時候**把對方的立場往「好打」的方向偏一點點，幾乎不費力、而且你自己不會察覺**。你不是故意立稻草人，你是在「摘要對方」這個必要動作裡，順著自己的傾向把它摘歪了。這跟 ch03 的可得性、ch04 的代表性一樣，是一個對的動作（摘要）在錯的方向上滑了一下。

**第二，弱化版更好打，打贏的爽感是真的。** 把對方說成「所以你主張所有參數都要驗證／所以你是說我們永遠不該上線／所以你想要零風險」——這些極端版本都有一個共同點：**它們真的很爛、真的一打就倒**。你打倒它的時候會有真實的「我贏了」的回饋，而這個回饋會讓你（和旁觀者）誤以為原本那個議題已經解決了。

稻草人有幾個值得單獨命名的變種，因為它們在工程討論裡特別常見（2026-06 查證主流分類）：

```text
稻草人家族：都是「攻擊一個不是對方真正主張的東西」

  變種            做法                            工程現場例子
  ──────          ──────                          ──────
  稻草人          把對方立場換成更極端的版本再打   「所以你要每個參數都驗證？」
  （straw man）                                    （對方只說「這一個」）

  弱稻草人        從對方陣營挑「最弱的那個論證」    只反駁「想用 Rust 因為它潮」，
  （weak man）    打，假裝那代表整個立場           無視「想用 Rust 因為記憶體安全」

  空心稻草人      乾脆捏造一個沒人說過的立場再打    「有人說我們應該完全不寫測試」
  （hollow man）                                   （根本沒人這樣說）

  挑極端例        從一群人裡挑最極端的個案，        「看，有個用 microservice 的團隊
  （nutpicking）  假裝那代表整群人的典型主張        垮了」→ 假裝那代表所有人
```

稻草人的解藥，工程師其實天天在用，只是用在 code 上沒用在對話上——它叫 **steelman（鋼人，亦譯「鋼鐵人論證」）**，也就是邏輯學說的**善意原則（principle of charity）**。做法跟稻草人完全相反：在反駁對方之前，先把對方的立場重建成**最強、最合理、最難打倒的版本**——強到連對方都會說「對，我就是這個意思，而且你講得比我還清楚」。然後你再去打那個最強版。

steelman 為什麼是工程師的母語？因為這就是**設計 review 的正確姿勢**。你 review 一個你不喜歡的設計時，真正有價值的不是「找它最爛的地方酸兩句」，而是「假設這個設計的作者是對的，他到底在 optimize 什麼我沒看到的約束？」——先把它的最強動機重建出來，再判斷那個動機在你的脈絡下站不站得住。打贏一個稻草人，你什麼都沒學到；打贏一個 steelman，你才真的解決了問題（或者發現對方其實是對的）。

我認為這是本章最該帶進工作的一條：**在 code review 裡，預設先 steelman 對方的設計，再決定要不要反對。** 不是為了當好人，是因為打倒稻草人對你的系統沒有任何幫助——你拆掉的是一個本來就不存在的弱論證，真正的那個問題還站在那裡。

## 人身攻擊：你打的不是論證，是人

人身攻擊（ad hominem，拉丁文「對著人」）的機制：**不去反駁論證，改去攻擊提出論證的那個人——他的品格、身分、處境、或他自己也沒做到——藉此讓人覺得他的論證不成立。**

核心的邏輯錯誤在於：**論證的對錯，跟提論證的人是誰，是兩件獨立的事。** 一個討厭的人、一個有私心的人、甚至一個自己都做不到的人，**仍然可能提出一個對的論證**；一個你敬重的好人也可能講錯。把「我不喜歡這個人／這個人有問題」拿來當「所以他的論證錯」的理由，就是把箭頭射偏了。

人身攻擊有三種主要型態，分清楚它們有實際用處——因為其中有一條界線（最後會講）是合理的、不是謬誤（2026-06 查證主流分類）：

```text
人身攻擊三型：箭頭都射向人，不射向論證

  型態            做法                              工程現場例子
  ──────          ──────                            ──────
  辱罵型          直接攻擊品格/能力                  「這新人懂什麼，他的 PR 不用看」
  （abusive）     （與論證對錯無關）                 「你不就是個前端，懂什麼資料庫」

  處境型          指「他的處境/利益使他這樣說」       「你會推 K8s 還不是因為你想學
  （circumstantial）→ 所以他錯                       了好跳槽」→ 所以方案不該採

  你也一樣        「你自己也沒做到」                 「你叫我寫測試？你上週那個
  （tu quoque）   → 轉移焦點、假裝對方主張因此無效     PR 覆蓋率還不是 0」
```

三型裡，**辱罵型**最容易認（它最赤裸）；**處境型**和 **tu quoque** 才是真正陰險的，因為它們**聽起來像在講道理**。

處境型的陰險在於：「指出對方有利益動機」這件事**本身往往是真的、也值得知道**——一個推銷某雲服務的人，確實有動機誇大它的好處，你確實該提高警覺。但「他有動機這樣說」**不等於**「他說的就錯」。利益動機是一個提醒你「去查證」的訊號，不是一個「可以直接判他錯」的結論。把前者當後者用，就從「合理的警覺」滑進了「處境型人身攻擊」。差別在於：合理的版本是「他有利益，所以我要自己核對數據」；謬誤的版本是「他有利益，所以他的方案不用看」。

tu quoque（拉丁文「你也一樣」）的陰險在於它**有一種道德快感**：「你自己都做不到，憑什麼說我？」——這句話在情緒上很爽，但它在邏輯上完全沒回應對方的主張。「你上週覆蓋率也是 0」這件事，就算百分之百屬實，也**絲毫不影響**「我們該不該寫測試」這個問題的答案。對方是不是偽君子，跟對方說的對不對，是兩件事。一個天天抽菸的醫生叫你戒菸，他是偽君子沒錯——但「抽菸有害」這個論證不會因為他自己抽菸就變假。

**那條合理的界線在哪？** 這是本節最該慢下來的地方，因為它正是「攻擊人」唯一**不是**謬誤的場合。當議題本身**就是「該不該相信這個人說的話」**——也就是論點靠的是這個人的**證詞（testimony）或可信度（credibility）**，而不是靠一條獨立可查的論證——這時候攻擊他的可信度就**不是謬誤，而是相關證據**（2026-06 查證）。

把這條界線翻成工程場景：

- 「**這台機器真的壞了，相信我，我看過了。**」——這個主張的全部分量都壓在「相信我」上，是純證詞。這時候「可是你上次也說你看過、結果根本沒看」**就不是 tu quoque 謬誤**，而是對證詞可信度的正當質疑。因為這裡要判的就是「該不該採信他的話」，他過去的可靠度是直接相關的證據。
- 對比：「**這個演算法是 O(n log n)，因為它每層做 n 的工作、遞迴深度 log n。**」——這個主張**不靠**「相信我」，它擺出了一條任何人都能自己驗的論證。這時候「可是你資料結構考試當過」就是**辱罵型人身攻擊**：論證攤在那裡，你該去查那條論證對不對，而不是查提論證的人考試考幾分。

```text
判準：攻擊可信度是「相關證據」還是「人身攻擊謬誤」？

  問一句：這個主張靠的是「獨立可查的論證」還是「相信我（證詞）」？
  ──────────────────────────────────────────────────────────
  靠獨立論證  →  論證攤在桌上，去查論證。攻擊人 = 謬誤（射偏了）
  靠證詞/可信  →  要判的就是「該不該信他」，他的可靠度 = 相關證據（不是謬誤）
```

這條判準很好用：**先問「他的結論能不能不靠他這個人、由我自己獨立驗證？」**——能，就閉嘴去驗，攻擊他是謬誤；不能、整件事就壓在「信不信他」上，那評估他的可信度就是正事。

## 訴諸權威：你信的是「誰說的」，不是「說得對不對」

訴諸權威是本章最需要小心的一個，因為它**最常被誤判**——很多人學了「訴諸權威是謬誤」之後，就走向另一個極端，把「專家共識」也當成謬誤一筆勾銷，這比原本的謬誤更糟。所以這一節的重點不是「訴諸權威是謬誤」，而是**畫出它合理與謬誤的界線**。

先講機制。訴諸權威是：**拿「某個權威說了 X」當作「X 為真」的理由。** 問題不在「引用權威」這件事本身——人不可能事事親自驗證，相信領域專家的判斷是**理性且必要**的省力策略（這正是 ch02 的捷思框架：在資訊不足、時間不夠時，把判斷外包給可信的權威，是對的）。問題在於**這個外包在什麼條件下可靠、什麼條件下失效**。

主流的判準是：一個訴諸權威要當得起證據，大致要同時滿足幾個條件（2026-06 查證，綜合多個邏輯與批判思考來源）——

```text
訴諸權威：合理 vs 謬誤的判準（同時滿足才算合理證據）

  條件                          合理（是證據）            謬誤（射偏了）
  ──────                        ──────                    ──────
  ① 真的是該領域專家？           密碼學家談加密協定         名演員代言保健品
  ② 講的是他的專業範圍內？       醫生談疾病                 物理學家談總體經濟
  ③ 該領域有專家共識？           「地球暖化是人為」有共識   還在激烈爭論的前沿問題
  ④ 沒有明顯的利益/偏誤汙染？     獨立學會的評估             賣產品的人誇自己產品
  ⑤ 引用沒被斷章取義/過時？       現行的、完整脈絡的結論     十年前已被推翻的舊說
```

任何一條不滿足，「權威說了」對「它為真」的證據力就大幅下降，甚至歸零。最常見的謬誤型態是 ① 和 ②——**借一個在別處有威望的人，來替一個他根本不懂的領域背書**。物理學拿過諾貝爾獎，不會讓他對營養學的意見更可信；一個演員很會演醫生，不代表他懂醫學。威望會在人心裡**外溢**到他不懂的領域，這個外溢就是訴諸權威最常踩的雷（這也跟 ch03 的可得性、ch04 的代表性有暗線：「他這麼有名／這麼像專家」被當成「所以他說的對」）。

現在回到開頭那個「Netflix 就是這樣做的」。用上面五條過一遍，你就知道它為什麼**多半**是謬誤：

- ① Netflix 在大規模串流架構上**確實是專家**——這條過。
- ② 但你引用它，是想支持「**我們**也該這樣」，而「我們的脈絡」（規模、流量、團隊大小、約束）**不在** Netflix 那個結論的適用範圍內——這條多半不過。
- ③④⑤ 通常也沒被檢查。

所以「Netflix 這樣做」的真正問題，不是「引用了一個權威」（Netflix 確實是權威），而是**把一個在「Netflix 的脈絡」下成立的結論，搬到「我們的脈絡」下當證據，而兩個脈絡的關鍵差異沒被檢查**。換句話說，它常常同時是訴諸權威**和**不當類比（ch08 會把不當類比講透）——「我們的處境像 Netflix」這個被偷渡的隱藏前提，根本沒被證成。

反過來，「**資安團隊評估過這個加密庫有已知 CVE，建議別用**」是不是訴諸權威？是。它是不是謬誤？**不是**，而且它是好證據——① 資安團隊在這領域是相關專家、② 講的正是專業範圍內、③ 「有已知 CVE 就別用」在資安界有強共識、④ 他們沒有要賣你什麼、⑤ CVE 是現行的公開紀錄。五條全過。這時候你**不該**因為「這只是訴諸權威」就無視它——那才是真正的不理性。

> 守住的關鍵分野：**「訴諸權威」不等於「謬誤」。** 把所有「專家說」都當謬誤打掉，會讓你掉進一個更危險的反智陷阱——好像「我不信任何專家、凡事自己判斷」才叫理性。不是。理性的省力策略本來就包含「在對的條件下相信相關專家的共識」。本章要你學的不是「別信權威」，是「**用那五條判準，分清哪次的權威是證據、哪次是借威望背書**」。

## 直覺的陷阱

攻擊型謬誤這一區，有幾個直覺會反過來咬你——而且咬你的時候你感覺自己很對。挑三個具名的講。

**陷阱① 是本章最想拆的，因為它是「學了一半的批判思考」造成的傷害。** 很多人讀了幾條謬誤清單，最大的收穫竟然是學會了一句萬能反駁：「那只是訴諸權威。」於是資安團隊說某庫有漏洞、醫生說某做法有風險、編譯器團隊說這個 UB 不能依賴——全被一句「訴諸權威」打發掉。這不是理性，這是**把一個防禦性工具拿來當無視證據的藉口**。記住那五條判準的存在本身就是解藥：訴諸**相關領域、有共識、無利益汙染**的專家，是證據；你要打的只是「借不相關威望背書」那種。能不能講出「這次的權威是哪一條判準不過」，是你有沒有真懂的試金石——講不出哪條不過，你就只是在反智。

**陷阱② 訴諸謬誤（fallacy fallacy）** 是 ch06「有效性 vs 健全性」的直接延伸（landscape T10）。你在本章學會抓稻草人、人身、訴諸權威之後，會有一股強烈的衝動：抓到對方犯規，就宣布對方輸了、結論也錯了。**錯。** 指出對方犯了謬誤，只證明了「**這一條論證**沒有成功支持結論」，完全沒證明「結論本身為假」。對方可能用一個爛到爆的稻草人論證，去支持一個其實正確的結論——那個結論不會因為論證爛就跟著錯。把「你論證錯」當成「你結論錯」，你自己就犯了訴諸謬誤。實務上這很重要：你贏了辯論的姿態，不等於你站在對的一邊。

**陷阱③「我又沒罵人」** 是工程師特別容易中的，因為我們以為自己很客觀、很對事不對人。但人身攻擊**最危險的兩型（處境型、tu quoque）根本不需要粗魯**。「你會推這個方案，還不是因為這是你引進的、你想證明自己沒選錯」——這句話可以講得非常冷靜、非常「我只是陳述事實」，但它射的是對方的**動機**，不是對方論證的對錯。一個有私心的人提的方案，仍然可能是對的方案；動機是叫你去**獨立查證數據**的訊號，不是判他錯的理由。自察的徵兆很簡單：**當你發現自己在講「他之所以這樣說，是因為他……」的時候，停一下——你正準備用動機代替論證。**

## 紙上推演

以下三題，純紙上，不用查任何工具、不用跑任何程式。每題先自己寫出完整答案再看解答。重點全在「重建論證」這個動作——把對方真正主張的命題、跟攻擊實際落點，並排寫出來。

**推演題一（★，[10 分鐘]）：把稻草人還原成對方真正的主張。**
下面是一段（虛構但逼真的）設計討論。請：(a) 寫出 A 真正主張的命題（一句話）；(b) 寫出 B 實際攻擊的命題（一句話）；(c) 指出這兩個命題為什麼不是同一個、B 犯的是哪種稻草人變種；(d) 把 B 的回應改寫成「先 steelman A、再反對」的版本。

> A：「我覺得這個新 feature 應該先對 5% 流量放量，觀察一週再全面上線。」
> B：「所以你是說我們要對每個 feature 都搞一個月的漸進式發布流程？那我們什麼都別想上線了。」

**推演題二（★★，[15 分鐘]）：判斷三句「訴諸權威」哪句合理、哪句謬誤，差別在哪一條判準。**
用本章那五條判準（① 相關專家？② 專業範圍內？③ 有共識？④ 無利益汙染？⑤ 沒斷章取義/過時？），逐句判斷下面三句是「合理證據」還是「謬誤」，並指出是哪一條（或哪幾條）判準決定的：

> (甲)「我們的資安團隊掃出這個依賴有一個未修補的高危 CVE，建議換掉。」
> (乙)「某知名創投合夥人在 podcast 說微服務是過時的架構，所以我們該回頭用單體。」
> (丙)「這套加密方案沒問題，因為發明它的人是圖靈獎得主。」

**推演題三（★★★，[20 分鐘]）：辨認人身攻擊，並判斷哪一個其實不是謬誤。**
下面三句都把矛頭指向「人」。請對每一句：(a) 判定它是辱罵型、處境型、還是 tu quoque；(b) 判斷它是「人身攻擊謬誤」還是「對可信度的正當質疑」，並用本章的判準（這個主張靠獨立論證，還是靠證詞/相信我？）說明你的判斷。

> (甲)「他說這段 code 沒有 race condition——但他連 mutex 跟 semaphore 都分不清，別信。」
> (乙)「他說那台 DB 主機他親自檢查過、硬碟沒問題——可是上次他也說檢查過，結果根本沒登入。」
> (丙)「你叫大家寫文件？你自己的服務到現在連一行 README 都沒有。」

### 推演解答

**題一解答。**
(a) A 真正主張的命題：「**這一個** new feature 應該先對 5% 流量放量、觀察一週再全面上線。」——範圍是「這一個 feature」、強度是「一週」。
(b) B 實際攻擊的命題：「**每個** feature 都要搞**一個月**的漸進式發布流程。」——範圍被放大成「每個」、強度被放大成「一個月」。
(c) 這兩個不是同一個命題：B 在兩個維度上都把 A 的立場往「更極端、更好打」的方向偏移了（「這一個」→「每個」、「一週」→「一個月」），然後去打那個被放大的版本。這是典型的**稻草人**；具體說，它是把對方立場「過度一般化＋誇大強度」的標準稻草人（不是空心稻草人，因為 A 確實主張了漸進式放量這件事，B 是扭曲它而非無中生有）。
(d) steelman 版改寫：「我同意對高風險 feature 做漸進式放量觀察是對的——你的擔心應該是『出事的時候 blast radius 要小』，這點我完全認同。我想討論的是：**這一個** feature 的風險等級，值不值得用到一週的觀察窗？如果它是個低風險的文案改動，也許 5% 跑一天就夠了；如果它動到計費，那一週甚至更久都合理。我們能不能按風險分級訂放量時長，而不是一刀切？」——注意這個版本先把 A 的最強動機（控制 blast radius）重建出來、認同它，再針對「時長該不該一律一週」這個**真正的分歧點**討論。**常見錯路**：只說「B 在扭曲 A」，但講不出**扭曲在哪兩個維度**（範圍、強度）——抓稻草人的關鍵就是並排兩個命題、指出差在哪，含糊地說「他講得太誇張」沒有抓到謬誤的形狀。

**題二解答。**
(甲)**合理證據**。① 資安團隊是相關專家、② CVE 評估正是其專業範圍、③「有未修補高危 CVE 就該換」在資安界有強共識、④ 他們沒有要賣你東西（無利益汙染）、⑤ CVE 是現行公開紀錄。五條全過——這時候把它當「只是訴諸權威」而無視，才是不理性（陷阱①）。
(乙)**謬誤**。主要敗在 ① 和 ②：創投合夥人的專業是投資、不是系統架構；「微服務是否過時」也**不是有共識**的問題（③ 也不過，這是仍在爭論的設計取捨）。這是「借一個在別處有威望的人，替一個他不專精、且本無共識的技術問題背書」——標準的訴諸（不相關）權威。
(丙)**謬誤**。敗在 ②（範圍外）與一個更微妙的點：圖靈獎得主**可能**真的是密碼學家（那 ① 過），但「**發明它的人很厲害**」對「**這個方案安全**」幾乎沒有證據力——加密方案的安全性靠的是**公開的密碼分析與同行檢驗**，不是發明者的名望。事實上密碼學界有一條鐵律：別用沒被公開審查過的自製加密（"don't roll your own crypto"），這恰恰是說「發明者多厲害都不算數，要看分析」。所以(丙)是拿「名望」替代「該領域真正的證據形式（公開分析）」——是訴諸權威。**常見錯路**：把(甲)也判成謬誤（「不都是訴諸權威嗎」）——這就是陷阱①，沒把五條判準逐條過，只憑「出現了權威」就一律打掉。

**題三解答。**
(甲)**辱罵型人身攻擊，是謬誤**。「他連 mutex 跟 semaphore 都分不清」攻擊的是人的能力。關鍵在判準：「這段 code 有沒有 race condition」是一個**獨立可查的論證**——你可以自己讀 code、自己跑 race detector、自己審 lock 順序，**完全不需要靠他這個人的可信度**。論證攤在桌上，你該去查論證，而不是查他能力。所以攻擊他＝射偏了＝謬誤。（就算他真的分不清 mutex/semaphore，這段 code 也可能剛好是對的——這同時也是陷阱②的提醒。）
(乙)**這是對證詞可信度的正當質疑，不是謬誤**。注意它**長得很像 tu quoque**（「上次你也說……」），但形狀不同：他的主張「硬碟沒問題」**全部分量都壓在「我親自檢查過、相信我」這個證詞上**，沒有獨立論證攤在桌上給你查（你沒有他那次檢查的 log）。當一個主張就是靠「相信我」時，「你過去的可靠度」就是**直接相關的證據**——這正是本章那條界線：要判的就是「該不該採信他的話」，他過去說檢查了卻沒登入，是評估這次該不該信他的正當依據。
(丙)**tu quoque，是謬誤**。「你自己的服務連 README 都沒有」就算百分之百屬實，也**完全不影響**「大家該不該寫文件」這個問題的答案。對方是不是偽君子，跟「寫文件這件事該不該做」是兩回事——抽菸的醫生叫你戒菸，他偽君子歸偽君子，「抽菸有害」不會因此變假。**常見錯路**：把(乙)也判成 tu quoque 謬誤（「不都是翻舊帳嗎」）——差別在(乙)的主張靠純證詞、沒有獨立論證，而(丙)的主張（該不該寫文件）有獨立於「誰說的」的理由可談。判準永遠是那一句：**這個主張靠獨立論證，還是靠「相信我」？**

## 自我檢核

口頭自答，講得出來才算過關。優先答「為什麼」。

1. 非形式謬誤跟形式謬誤（肯定後件那種）在「錯在哪一層」上有什麼根本差別？為什麼非形式謬誤要先「重建論證」才抓得到？
2. 稻草人的機制是什麼？為什麼說它「沒有正當版本」，但又有一個讓人會犯它的機制？steelman（善意原則）為什麼是它的解藥、又為什麼正好是 code review 的正確姿勢？
3. 人身攻擊的三型（辱罵、處境、tu quoque）各是什麼？為什麼後兩型比辱罵型更陰險、更難當場抓到？
4. 攻擊一個人的可信度，**什麼時候不是謬誤**？請用「這個主張靠獨立論證，還是靠證詞／相信我」這條判準，自己造一個「不是謬誤」的例子和一個「是謬誤」的例子。
5. 「訴諸權威一律是謬誤」這句話為什麼是錯的、甚至危險？請講出讓「訴諸權威」變成合理證據的那幾條判準裡，最常被違反的是哪一兩條。
6. 什麼是訴諸謬誤（fallacy fallacy）？為什麼「我抓到你犯了稻草人，所以你的結論錯了」這句話本身也是謬誤？（提示：回扣 ch06 有效性 vs 健全性。）
7. 為什麼說攻擊型謬誤的「察覺不到」，落在「你忙著反駁對方丟給你的那句話，沒注意到那句話不是你說的」？這跟系統一／系統二的分工有什麼關係？
