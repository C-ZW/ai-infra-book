# ch06 — 有效不等於為真：有效性 vs 健全性

> **本章解決什麼問題**：ch05 立起「形式謬誤＝結構壞了」，但留下一個危險的副作用——很多人讀完會反過來相信「結構壞了，結論就一定是錯的」。這一章把這個誤會徹底拆掉。我們要分開兩個被日常語言攪在一起的概念：論證**有效（valid）**＝「前提若真、結論必真」的結構保證，與論證**健全（sound）**＝「有效，而且前提真的全為真」。形式對了結論仍可能假（前提錯），形式錯了結論仍可能真（瞎貓碰上死耗子）。這是評估任何論證的核心工具，也是 Part IV 整個非形式謬誤目錄的鑰匙：因為大多數非形式謬誤都藏在沒講出來的前提（隱藏前提）裡，你得先學會「把論證的骨架攤開、看它哪裡是結構、哪裡是前提真假」，才抓得到它們。

## 從你已知的出發

你每天都在跟一個你早就懂、卻可能沒命名過的區別打交道：**「測試過了」不等於「需求對了」。**

想一次 CI 全綠、但功能還是錯的經驗。發生了什麼？你的程式碼**正確地實現了你寫下的規格**——輸入經過邏輯，輸出符合斷言，這一段是對的、是「有效」的：給定這些前提（規格、輸入假設），這個輸出**必然**跟著來。問題出在更上游：那份規格本身錯了。需求理解錯了、邊界條件漏了、PM 給的假設根本不成立。**推論無懈可擊，前提是爛的。**結果就是一個「形式正確、結論錯誤」的系統。

這正是邏輯學裡「**有效但不健全（valid but unsound）**」的工程版：

```text
你的系統                          邏輯學的講法
────────────────────────────  ──────────────────────────────
程式碼正確實現了規格           推論有效（valid）：前提→結論的步沒問題
規格本身是錯的                 前提為假
所以功能是錯的                 所以結論不可信（論證不健全 unsound）
                              ── 但「不健全」不代表「結論一定假」
```

反過來的情形你也見過：有人用一個**爛到不行的理由**，得出一個**剛好正確**的結論。「這個服務一定有 memory leak，因為它是 Java 寫的」——理由是胡扯（Java 不蘊含 leak），但這個服務**確實**正好有 leak（被別的原因造成）。理由錯了，結論對了。如果你因為「他的推理很爛」就反過來咬定「所以根本沒 leak」，你會在生產環境踩雷。

這兩種錯位——**對的推理＋錯的前提**、**錯的推理＋對的結論**——是本章的全部。日常語言把「他講得有道理」「他說的是對的」「他的邏輯對」全攪成一坨，工程師其實有更精細的直覺，只是平常沒把它從程式碼搬到論證上。這一章就是搬過來。

## 兩個被攪在一起的問題：結構，與內容

評估一段推理，其實是在問兩個**完全獨立**的問題。把它們攪在一起，是日常爭論裡最常見的混亂來源。

```text
問題一（結構）：如果前提都是真的，結論是不是「非真不可」？
   → 這是「有效性 valid」。只看推論的形狀，不看前提是不是真的。

問題二（內容）：前提，實際上，是不是真的？
   → 這是前提的真假。要查事實、要查證據，不是邏輯的事。

論證健全（sound） = 問題一答「是」（有效） AND 問題二答「是」（前提全真）
```

關鍵在於：**這兩個問題互不干涉。** 你可以有效卻前提全假，可以無效卻前提全真，可以無效卻結論碰巧為真。把四種組合攤開看（這是本章最該記住的一張表，後面 worked example 會逐格填具體例子）：

```text
四象限（有效性 × 前提真假 → 結論可信度）

  有效 ＋ 前提全真  →  健全 sound      →  結論「必」為真  ★唯一能擔保的一格
  有效 ＋ 前提有假  →  有效但不健全     →  結論真假未定    （形式對、地基爛）
  無效 ＋ 前提全真  →  無效            →  結論真假未定    （前提真也救不了爛結構）
  無效 ＋ 前提有假  →  無效            →  結論真假未定    （兩頭都爛）

唯一一格擔保結論為真：有效 ＋ 前提全真 ＝ 健全。其餘三格，結論都可真可假。
```

讀懂這張表，你就讀懂了本章一半。注意右下到左上那條對角線：**只有左上角那一格**，能讓你從「論證」安全地走到「結論為真」。其他三格，結論的真假**這張表管不著**——你得另外去查事實。

這跟有效性的官方定義是同一回事，只是換句話講。邏輯學對「有效」的定義是（這是恆真的邏輯定義，不靠 landscape，但我在這裡自我複核一遍）：

> 一個演繹論證是**有效**的，若且唯若**不可能**「前提全真、而結論為假」同時成立。

換句話說，有效＝「前提真」這件事，**鎖死了**「結論真」。它不保證前提真——它只保證**這個鎖是好的**。鎖好不好，跟門後有沒有東西，是兩回事。健全＝鎖是好的（有效）**而且**門後真的有東西（前提真）。

（這裡的「有效」是**演繹**論證的標準。歸納論證——從觀察推一般規律——沒有這種「前提真則結論必真」的鐵保證，它只有「強／弱」的程度差別：前提真讓結論**更可能**真，但不擔保。本書談的形式有效性是演繹的事；歸納的可靠性是另一套，本書點到，機率機制指向《馴服隨機》。）

## 為什麼這對「形式謬誤」這麼重要：別把無效當成假

回到 ch05。你學了肯定後件、否定前件是無效的推論形式。一個誠實的危險就在這裡：學完「這種推論無效」，大腦的 S1 會偷偷塞給你一個結論——「**所以他講的是錯的**」。

不對。**無效的論證，結論可能是真的。**

```text
前提：若這個服務記憶體爆了（P），它會被 OOM killer 殺掉（Q）   P → Q
觀察：它被殺掉了（Q）
結論：所以它記憶體爆了（P）                              ← 肯定後件，無效

這個推論是無效的（Q 也可能是別人手動 kill、是 OOM 以外的訊號）。
但——這個服務「實際上」可能真的就是記憶體爆了。
推論無效 ≠ 結論為假。它只是「這條推論沒有成功證明結論」。
```

「無效」說的是**這條路沒走通**，不是**目的地不存在**。OOM killer 殺它的那一刻，它的記憶體可能真的爆了——只是「被殺掉」這個觀察，本身**推不出**那個結論，因為還有別的路通往「被殺掉」。你要證明它記憶體爆了，得去看別的證據（記憶體曲線、OOM 的 dmesg log），不能靠「它被殺了」這一步。

這裡也順手回扣一次本書脊椎的分野（見 ch01 的雙系統透鏡，也守住 ch05 立的界線）：形式謬誤之所以無效，是**邏輯結構的事**，跟用了哪個捷思無關。雙系統在這裡只解釋一件事——**為什麼 S1 會覺得「它被殺了，所以記憶體爆了」聽起來很合理**：因為 S1 對邏輯形式不敏感，它只感覺「P 和 Q 常常一起出現、這聽起來順」，就把答案放行了，S2 該做的結構檢查（「Q 是不是還有別的來源」）被省力預設蓋過。**這不是某個捷思被錯置，是 S1 根本沒在做結構檢查。** 別把肯定後件本身講成一個捷思。

## worked example：四種組合，逐格填具體例子

把上面那張四象限表，用工程情境逐格填滿。每一格我都標清楚「哪裡是結構、哪裡是前提真假」，並走一遍判定。慢一點，這張表是本章的骨。

```text
═══════════════════════════════════════════════════════════════
格 ①：有效 ＋ 前提全真 ＝ 健全（結論「必」為真）
═══════════════════════════════════════════════════════════════
前提1：所有通過型別檢查的程式，都沒有「把字串當數字加」的型別錯誤  P → Q
前提2：這支程式通過了型別檢查                                    P
結論：所以這支程式沒有「把字串當數字加」的型別錯誤              ∴ Q

結構：肯定前件（modus ponens），有效。
前提真假：前提1 在「TypeScript strict 模式、無 any、無 as 斷言」
         的限定下成立；前提2 可查 CI。兩前提都真。
判定：有效 ＋ 前提全真 ＝ 健全。結論「必」為真。
      （注意前提1 的限定條件——一旦混進 any/as，前提1 就變假，
        論證掉到格②。地基的脆弱性藏在前提，不在推論。）

═══════════════════════════════════════════════════════════════
格 ②：有效 ＋ 前提有假 ＝ 不健全（結論真假未定）
═══════════════════════════════════════════════════════════════
前提1：所有 5xx 錯誤都是後端的責任                              P → Q
前提2：這次事故是 5xx                                          P
結論：所以這次事故是後端的責任                                  ∴ Q

結構：肯定前件，有效。推論本身無懈可擊。
前提真假：前提1 為假——5xx 也可能是上游 LB 設定、DNS、
         打到後端前就壞掉。前提1 是個被偷渡的過度概括。
判定：有效但不健全。結論「真假未定」：
      這次事故「可能」真是後端責任，也可能不是——
      這個論證本身擔保不了。要查事故根因，不能靠這段推論。
      ★ 這格就是「CI 全綠但需求錯」的邏輯本體：推論對、前提爛。

═══════════════════════════════════════════════════════════════
格 ③：無效 ＋ 前提全真 ＋ 結論碰巧為真
═══════════════════════════════════════════════════════════════
前提1：若服務有 memory leak（P），長跑後記憶體會持續上升（Q）   P → Q
前提2：這服務長跑後記憶體持續上升（Q）                          Q
結論：所以這服務有 memory leak（P）                            ∴ P

結構：肯定後件（affirming the consequent），無效。
      反例：記憶體上升也可能是正常快取暖機、連線池長大，不是 leak。
前提真假：兩個前提都真（leak 確實會讓記憶體升；它確實在升）。
結論真假：這服務「實際上」可能真的有 leak——結論碰巧為真。
判定：無效。即使前提全真、即使結論碰巧真，
      這條推論「沒有成功證明」結論。你不能靠它下結論。
      ★ 前提全真也救不了爛結構——這是格③要你記住的。

═══════════════════════════════════════════════════════════════
格 ④：無效 ＋ 前提有假（兩頭都爛，但結論仍可能碰巧真）
═══════════════════════════════════════════════════════════════
前提1：若一個語言會自動垃圾回收（P），就不可能有記憶體問題（Q） P → Q
前提2：這服務沒有記憶體問題（Q）                                Q  （而且這前提還是假的）
結論：所以它用的是會自動垃圾回收的語言（P）                     ∴ P

結構：肯定後件，無效。
前提真假：前提1 為假（GC 語言照樣會 OOM、會 leak）；
         前提2 也可疑。兩頭都爛。
判定：無效且不健全。結論（它用 GC 語言）就算碰巧為真，
      這段推理也一文不值。
═══════════════════════════════════════════════════════════════
```

把四格放在一起看，一個事實會跳出來，而它正是本章最該讓你坐立難安的地方：**「結論是不是真的」這個你最在乎的問題，這四格裡有三格給不出答案。** 只有格①（健全）能擔保結論。其餘三格，你都得放下這段論證、另外去查事實。

我認為這是整本書裡最實用的一頁，因為它直接對應你 debug 的處境：很多時候你聽完一段聽起來很順的事故歸因（格②），或一段邏輯有洞但結論你早就信的猜測（格③），你的 S1 已經點頭了。這張表逼你的 S2 多問一句：**這是結構的問題，還是前提的問題？** 兩者要用完全不同的方法去驗。

## 隱藏前提：非形式謬誤都藏在這裡（Part IV 的鑰匙）

現實裡幾乎沒有人會把論證寫成上面那種「前提1、前提2、結論」的乾淨三段。真實的論證長這樣：

> 「這個方案不能用，它需要改 schema。」

這不是一個完整論證——它**省略了**一個沒講出來的前提。邏輯學給這種「缺了一塊的論證」一個名字：**省略三段論（enthymeme，亦稱省略論證）**，亞里斯多德在《修辭學》裡就討論過。它的定義是：**一個有前提被省略、靠聽者自己補上的論證**（這個定義我查證過，見延伸閱讀）。前提被省略，通常是因為「太顯然、不用說」——但**陷阱也正在這裡**：那塊沒講出來的前提，往往就是整段論證最可疑、最該被質疑的地方，省略它正好讓它躲過檢查。

把上面那句話的隱藏前提補出來：

```text
明說的前提：這個方案需要改 schema
隱藏前提  ：（任何需要改 schema 的方案都不能用）   ← 沒講出來的那塊
結論      ：所以這個方案不能用

補出來之後，論證變成有效的肯定前件（modus ponens）：
  改 schema → 不能用 ；  要改 schema ；  ∴ 不能用
結構沒問題。但那個隱藏前提「任何需要改 schema 的都不能用」——
真的嗎？這是個極端的過度概括。論證的真正弱點不在明說的話，
在那塊被當成「顯然」而省略掉的前提。
```

**這就是 Part IV（非形式謬誤）的鑰匙。** ch07 開始的稻草人、假兩難、滑坡、循環論證……它們大多不是「結構無效」（那是 ch05 形式謬誤的事），而是**藏了一個爛前提在沒講出來的地方**。要抓到它們，你得先做這一步：**把論證重建完整，把隱藏前提攤到陽光下，再單獨檢查那塊前提真不真。** 重建論證，是讀懂 Part IV 之前必須先練的功。

重建隱藏前提的工程直覺：這跟你讀別人程式碼時「找出被默默假設、但沒寫進去的不變量（invariant）」是同一個動作。一段程式碼「能跑」常常靠一堆沒寫出來的假設（「這個 list 永遠非空」「這個欄位永遠不是 null」「上游一定先呼叫過 init」）。論證也一樣，靠一堆沒講出來的前提撐著。把它們挖出來、寫下來、逐一質疑——這對你不是新技能，只是換了個場域。

## 直覺的陷阱

本章的招牌陷阱，是上面那條「無效≠假」延伸出的、有自己名字的具名謬誤。

**具名陷阱：訴諸謬誤 / 謬誤謬誤（the fallacy fallacy，又稱 argument from fallacy）。**

它長這樣：「**你犯了謬誤，所以你的結論是錯的。**」這句話本身，就是一個謬誤。

為什麼？因為「對方的論證沒成功支持結論」只證明了一件事——**這個論證失敗了**。它**沒有**證明「結論為假」。結論可能仍然是真的，只是對方剛好用了爛理由去支持它（回去看 worked example 的格③）。指出對方犯了謬誤，你贏得的是「他這段論證不算數」，**不是**「他錯了」。我查證過這個定義（見延伸閱讀，Internet Encyclopedia of Philosophy 與 landscape T10 一致）：argument from fallacy ＝ 因為一個支持某結論的論證有缺陷，就斷定該結論為假。

```text
最常被誤解成什麼 / 錯誤直覺在哪一步把你帶下溝 / 怎麼自我察覺
──────────────────────────────────────────────────────────────
誤解①：「形式謬誤 → 結論一定是錯的」
  下溝點：學完 ch05「肯定後件無效」，S1 自動把「無效」翻譯成
          「假」。但無效只是「這條路沒走通」，不是「目的地不存在」。
  自我察覺：當你聽到「他犯了 XX 謬誤」就準備宣布對方錯了——停。
          先問：我證明的是「這條論證失敗」，還是「結論為假」？
          十之八九只是前者。結論還沒判，得另外查。

誤解②：「他講得頭頭是道（有效），所以他是對的（結論真）」
  下溝點：把「有效」當「健全」。推論順暢只擔保結構，不擔保前提。
          一個有效論證可以從荒謬前提推出荒謬結論，一路順暢。
  自我察覺：聽完一段順到無法反駁的論證、卻隱隱覺得結論不對——
          別懷疑自己。去查它的前提，尤其是那塊沒講出來的隱藏前提。
          順暢是結構的事；結論真假在前提那邊。

誤解③：「相關方論證有漏洞 → 整個立場都垮」
  下溝點：訴諸謬誤的社交版。辯論中很爽，但邏輯上你只擊倒了
          一個論證，沒擊倒結論。對方換個好論證，結論可能照樣站。
  自我察覺：你很想用「你這是 OO 謬誤！」結束討論時，提醒自己：
          這頂多讓對方「這一招」失效，不讓對方「這個結論」變錯。
```

這個陷阱的危險，在於它讓「學過邏輯謬誤的人」變得**更容易**犯錯——因為你手上多了一把錘子（謬誤的名字），看什麼都像釘子。學會辨認謬誤的真正用途是「**降低你對這個論證的信任**」，不是「**反向斷定結論為假**」。論證失敗，結論待議——這是受過訓練的懷疑者該停在的地方。

## 紙上推演

純紙筆，不用跑任何程式。建議準備紙筆，把論證的骨架真的畫出來——「攤開結構」這件事，用看的會自我感覺良好、用寫的才會發現你漏了哪塊前提。

---

**第 1 題（★，[10 分鐘]）：判定四象限。** 下面三段推理，各判定它落在本章四象限表的哪一格（健全／有效但不健全／無效＋前提真／無效＋前提假），並說明你的判定理由（哪裡是結構、哪裡是前提真假）。

```text
(a) 前提1：所有部署到 prod 的版本都跑過 staging
    前提2：v2.3 部署到了 prod
    結論：v2.3 跑過 staging
(b) 前提1：若快取掛了，延遲會飆高
    前提2：延遲飆高了
    結論：快取掛了
(c) 前提1：所有用 Rust 寫的服務都沒有資料競爭（data race）
    前提2：這個服務沒有資料競爭
    結論：這個服務是用 Rust 寫的
```

---

**第 2 題（★★，[15 分鐘]）：重建隱藏前提並判定健全性。** 把下面這句真實會出現在設計討論裡的話，重建成完整論證（補出隱藏前提），判定它**有效**嗎、**健全**嗎，並指出它真正的弱點在哪。

> 「我們不該引入這個 library，GitHub 上它兩年沒更新了。」

---

**第 3 題（★★,[15 分鐘]）：拆訴諸謬誤。** 一位同事在 PR review 說：

> 「你這個論點犯了滑坡謬誤——『不加這個 lock 就會資料損毀、然後客訴、然後公司倒閉』。所以根本不需要加這個 lock。」

請解釋：這位同事的反駁，本身犯了什麼謬誤？把它拆開：原本那段話的問題是什麼、同事的反駁多走了哪一步變成謬誤、那個 lock 到底該不該加這件事，現在處於什麼狀態？

---

**第 4 題（★★★，[20 分鐘]）：構造反例。** 請各自構造一段**你工作領域**的論證，滿足：
- (a) 有效但不健全（推論無懈可擊，但因為某個前提是假的，結論不可信）；
- (b) 無效但結論碰巧為真（推論是肯定後件或否定前件，前提也許都真，但結論之所以真不是這段推論的功勞）。
構造完，對 (b) 明確指出：如果有人**靠這段論證**去下決策，會在什麼情況下踩雷？

---

### 推演解答

**第 1 題。**

- **(a) 健全。** 結構是肯定前件（部署 prod → 跑過 staging；v2.3 部署 prod；∴ 跑過 staging），有效。前提1（在「有強制 staging gate 的 pipeline」這個脈絡下）為真、前提2 可查，前提全真。有效＋前提全真＝健全，結論必為真。**常見錯路**：有人會說「但萬一有人手動跳過 staging 直接上 prod 呢？」——對，那正是在質疑**前提1 的真假**（前提1 變假），這時論證掉到「有效但不健全」。能看出這是在攻前提、不是攻結構，就對了。

- **(b) 無效（前提可真，結論真假未定）。** 結構是肯定後件（快取掛→延遲飆；延遲飆了；∴ 快取掛了），無效。反例：延遲飆高也可能是 DB 慢查詢、GC 停頓、上游流量暴增。前提可能都真，但這條推論推不出結論。快取**可能**真的掛了，但你得另外查（看快取的 metrics），不能靠「延遲飆高」這一步。**常見錯路**：把它判成「健全」或「有效」，因為「快取掛了延遲確實會飆」聽起來很對——但那是在確認前提1 的真，不是在確認推論的有效。結構是肯定後件，無效，沒得商量。

- **(c) 無效（且前提1 為假）。** 結構是肯定後件（Rust → 無 data race；無 data race；∴ 是 Rust），無效。而且前提1 本身也假——Rust 的安全保證是針對 safe 程式碼的 data race，`unsafe` 區塊、邏輯層的競態條件（race condition，跟編譯器層的 data race 不是同一件事）照樣可能出問題；何況「沒有 data race」的語言不只 Rust（單執行緒的服務天然沒有）。結構無效、前提又假，這結論（它是 Rust 寫的）就算碰巧真，也跟這段論證無關。

**第 2 題。** 重建：

```text
明說前提：這個 library 兩年沒更新了
隱藏前提：（任何兩年沒更新的 library 都不該引入）   ← 補出來的那塊
結論    ：不該引入這個 library

補上隱藏前提後，結構是肯定前件（modus ponens），有效。
```

**有效嗎？** 補出隱藏前提後，是。**健全嗎？** 取決於那塊隱藏前提真不真——而它正是弱點所在。「兩年沒更新」可以有兩種完全相反的解讀：(1) 專案死了、沒人維護、有漏洞沒人修；(2) 它**做完了、穩定了**——很多小而專一的 library（一個 left-pad 等級的工具、一個成熟的解析器）就是因為沒有要加的功能才不更新。隱藏前提「凡兩年沒更新都不該引入」把這兩種情況一刀切，**過度概括**，所以**不健全**。真正的弱點不在明說的「兩年沒更新」（這是可查的事實，多半為真），而在那塊被當成顯然、省略掉的前提。**討論該往哪走**：不是吵「到底幾年算舊」，而是把隱藏前提攤開、針對「這個 library 是死了還是做完了」去查（看 issue 回應速度、有無未修的安全公告、依賴它的專案數）。

**第 3 題。** 這位同事犯的是**訴諸謬誤 / 謬誤謬誤（the fallacy fallacy）**。拆開：

1. **原本那段話的問題**：「不加 lock → 資料損毀 → 客訴 → 公司倒閉」這個連鎖**可能**確實有滑坡的味道（每一步的必然性沒被證成，後面幾步尤其跳）。同事指出這一點，**本身不算錯**。
2. **多走的那一步**：同事從「你這段論證是滑坡（爛論證）」直接跳到「**所以根本不需要加這個 lock**」。這一步就是謬誤——他用「論證失敗」推出了「結論為假」。但「該不該加 lock」這個結論，**滑坡論證的失敗證明不了它是假的**。
3. **那個 lock 現在的狀態**：**待議，未判。** 那段滑坡論證沒有成功證明「必須加 lock」，但同樣地，指出它是滑坡也沒有證明「不必加 lock」。是否需要這個 lock，要回到真正的技術問題去判：這裡到底有沒有並發寫入、有沒有實際的競態窗口、不加會不會真的損毀資料。**用一個更好的論證重新評估**，而不是靠「你犯了滑坡謬誤」結案。**常見錯路**：覺得同事很厲害、一句話駁倒了對方。在辯論場上也許有效，在工程決策裡這是把「論證沒成功」誤當「結論為假」，可能讓一個**真的需要**的 lock 被砍掉——這就是訴諸謬誤會害你踩的雷。

**第 4 題（示範答案，你的具體例子會不同）。**

- **(a) 有效但不健全**：
  ```text
  前提1：所有經過 code review 的 PR 都沒有 bug
  前提2：這個 PR 經過了 code review
  結論：這個 PR 沒有 bug
  ```
  結構是肯定前件，有效。但前提1 為假（code review 抓不到所有 bug），所以不健全，結論不可信。這正是「綠燈 CI / 過了 review ≠ 正確」的邏輯本體：推論沒問題，前提（review 萬無一失）是幻想。

- **(b) 無效但結論碰巧真**：
  ```text
  前提1：若資料庫連線池滿了（P），健康檢查會逾時（Q）
  前提2：健康檢查逾時了（Q）
  結論：資料庫連線池滿了（P）          ← 肯定後件，無效
  ```
  前提可能都真，連線池**或許**真的就是滿的。但健康檢查逾時還有別的原因（網路、CPU 飽和、下游慢）。**靠這段論證去決策的踩雷情境**：你看到健康檢查逾時，就**直接**斷定是連線池滿、跑去調大 pool size——但這次的真凶其實是下游 API 變慢，pool 根本沒滿。你「治」錯了地方，問題沒解，還可能因為盲目放大 pool 而掩蓋或加劇真正的瓶頸。肯定後件的危險不是抽象的：它就是「症狀 → 直接咬定某個原因」的歸因捷徑，在 on-call 的凌晨三點最容易害你。

## 自我檢核

口頭回答，講不清楚就是還沒懂。優先回答「為什麼」。

1. 用你自己的話，講清楚「有效（valid）」和「健全（sound）」的差別。為什麼一個論證可以「有效卻不健全」？再舉一個你工作裡的「有效但不健全」實例（提示：CI 全綠但需求錯）。
2. 為什麼「無效的論證」不代表「結論為假」？用 worked example 格③（memory leak 那個）對另一個工程師講一遍。
3. 「訴諸謬誤（fallacy fallacy）」是什麼？為什麼「你犯了謬誤，所以你錯了」這句話本身是謬誤？指出對方論證有漏洞，到底證明了什麼、沒證明什麼？
4. 四象限表裡，哪一格——也只有哪一格——能擔保「結論為真」？其他三格，結論的真假該去哪裡判（提示：不在這張表裡）？
5. 什麼是「隱藏前提 / 省略三段論（enthymeme）」？為什麼說「重建隱藏前提」是讀懂 Part IV（非形式謬誤）的鑰匙？為什麼那塊被省略的前提，往往正是論證最可疑的地方？
6. 回扣脊椎：當你聽到一段無效的推論（如肯定後件）卻覺得它「聽起來很合理」，按 ch01 的雙系統透鏡，是 S1 在做什麼、S2 沒做什麼？（注意守住分野：這裡別把肯定後件講成一個捷思——它為何無效是結構的事，雙系統只解釋「為何它看起來有效」。）
7. 為什麼「學過邏輯謬誤的人」反而可能更容易犯訴諸謬誤？這對你「以後在 review 裡指出謬誤」這個習慣，該加上什麼自我提醒？
