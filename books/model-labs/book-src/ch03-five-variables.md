# ch03 — 五變數記分卡：差異的成因框架

> **本章解決什麼問題**：前兩章我們同意了兩件事——架構是公開的同一張藍圖（ch01），而模型是沿著 scaling 曲線「養」出來、不是「寫」出來的（ch02）。那就逼出一個問題：如果藍圖一樣、養法都叫「訓練」，為什麼八家養出來的模型強項天差地遠？本章交出全書的核心分析工具——**五變數記分卡**：差異記在哪幾個帳上、各記一筆。這是 ch04 開始「別急著排名」之前的最後一塊地基，也是 Part II 逐家拆解（ch05–12）與 Part III 跨家主題（ch13–19）每一章 `## 成因拆解` 都會回收的那把尺。讀完這章，你手上就有一張能套到任何一款新模型上的排查清單。

## 從你已知的出發

你做過無數次效能瓶頸診斷。系統慢了，你不會瞎猜，你跑一張固定的排查清單：**CPU 打滿了嗎？IO 卡住了嗎？哪裡在搶鎖？網路在抖嗎？快取沒命中嗎？** 你逐項排查，不是因為你知道答案，是因為你知道**慢有固定幾種成因**，把它們一個一個過一遍，主因自然浮出來。你那次把尖峰 RDS CPU 壓下 40%，不是運氣——是你把這張清單跑到底，定位到一條沒走索引的查詢。

「為什麼這家模型強」也有這樣一張清單。而且我想先用一個你更熟的對照，把它的形狀講清楚。

想像兩個團隊，**租用同一型號的 RDS 執行個體**——同樣的 vCPU、同樣的記憶體、同樣的 MySQL 版本。硬體藍圖一模一樣。但一個團隊的查詢 p99 是 8 毫秒，另一個是 800 毫秒。差一百倍。差在哪？你閉著眼睛都數得出來，效能差異就來自那幾個源頭：

```text
同一台 RDS，效能差異的五個源頭
─────────────────────────────────
  schema 設計      表怎麼拆、欄位型別、正規化程度
  index            建了哪些索引、覆蓋查詢沒
  查詢寫法         有沒有 N+1、有沒有全表掃描
  硬體配置         IOPS 上限、連線池、參數調校
  流量形狀         讀寫比、熱點、尖峰分布
```

藍圖（RDS 型號）一樣，跑出來的系統行為天差地遠——因為**差異不在藍圖，在藍圖之外的那五件事**。這正是 ch01 那句「架構不是護城河」落到地上的樣子：Transformer 是那台 RDS，大家租的是同一型；強弱差在你怎麼配、餵什麼、誰來調。

本章要做的，就是把「模型強弱」也壓成這樣一張**五格清單**。它不像 RDS 那張那麼乾淨（模型的五格會互相纏繞，等下會看到），但它的用法一模一樣：看到一款模型在某件事上很強，你不再說「他們比較厲害」這種廢話，你拿這張清單逐格問——**這個強項，主要記在哪一格的帳上？**

## 五變數：固定順序、固定用詞

先把五格擺出來。這五個名稱、這個順序，全書不變——Part II 每一家、Part III 每一個主題，`## 成因拆解` 都照這張表逐格歸因，不准各章自創分類。請把它們當成你那張瓶頸清單的「CPU/IO/鎖/網路/快取」：固定的座標軸。

| # | 變數 | 一句話定義 | 它解釋哪一類差異 |
|---|---|---|---|
| 1 | **資料**（data） | 餵什麼、配比、清洗、來源（網路／程式碼／合成／專有） | 模型「見過什麼世界」：中文強弱、程式碼強弱、知識廣度 |
| 2 | **後訓練·對齊**（post-training / alignment） | SFT、RLHF、RLAIF、RLVR、reward model、指令與性格塑形 | 好不好用、聽不聽話、會不會推理，很多來自這裡 |
| 3 | **算力·基礎設施**（compute / infra） | GPU/TPU 數量、自研晶片、叢集、互連、訓練工程成熟度 | 能訓多大、多久、多便宜；長上下文、超大模型可不可行 |
| 4 | **研究文化·品味**（research culture / taste） | 押哪個方向、實驗紀律、迭代速度、人才密度、價值取向 | 為什麼有人先做出 reasoning、有人先做對齊；最難被抄走 |
| 5 | **產品取向**（product orientation） | 服務誰（消費者／開發者／企業／自家產品）、開放或封閉 | 模型被「養成」什麼形狀，往往是被它要服務的產品壓出來的 |

下面逐格講一遍。每一格我給你一句定義、它管哪一類差異、再丟**一個迷你預告例子**——某一格如何主導某一家的某個強項。**注意：這些例子全是「預告」，這裡只點到「主因落在這格」，完整因果鏈留到 Part II 那家自己的章節展開。** 你現在只需要建立「哪種差異該往哪一格看」的直覺。

### 第 1 格：資料——模型見過什麼世界

模型的能力長在它見過的資料上。**沒見過的世界，它就學不會。** 這格管的是「餵什麼進去」：是英文網路為主還是中文語料厚實，是塞了大量 GitHub 程式碼還是偏向自然語言，是純爬來的網路文字還是摻了大量合成資料、專有資料。資料的**配比**（各種來源各佔多少）是各家最不公開的秘密——因為它幾乎直接決定模型「天生會什麼」。

對工程師我這樣翻譯：這就像你那台 RDS 的**流量形狀**。你餵系統什麼樣的讀寫流量，它就被磨成適應那種流量的樣子。模型餵什麼資料，就被磨成擅長那個領域。

> **迷你預告**：Qwen（阿里巴巴）的中文與多語言（號稱 119 種語言）強，主因落在這一格——背後是中文／亞洲語料的厚度。為什麼是阿里、這個優勢的影子是什麼，留到 ch12。

### 第 2 格：後訓練·對齊——把原料磨成好用的工具

ch02 講的 pretraining，養出來的是一塊**有能力但難用**的原料——它會接話、知道很多，但不一定聽你的指令、不一定按你要的格式答、可能講出有害內容。**後訓練（post-training）**就是把這塊原料磨成順手工具的全部工序：用人類示範教它照指令做（SFT）、用人類偏好教它什麼答案更好（RLHF）、用 AI 自評代替人工標註（RLAIF）、用「答案對不對／程式跑不跑得起來」這種客觀訊號做強化學習（RLVR）。

這些縮寫你現在不必全懂（ch15 會把這條演化線講透）。你只要抓住一件事：**「好不好用、聽不聽話、會不會推理」這些你天天有感的特質，大半不是 pretraining 給的，是後訓練給的。** 同一塊 pretraining 原料，後訓練手法不同，磨出來的性格就不同——有的乖、有的會頂嘴、有的特別會一步步推理。

> **迷你預告**：Anthropic 的 Constitutional AI（憲法式 AI，2022）——給模型一份寫好的「憲法」，讓它照原則**自我批評、自我修正**，用 AI 回饋取代大量人工標註——主因就落在這一格。它如何外溢成「聽話、適合 agent/coding」的口碑，留到 ch07。

### 第 3 格：算力·基礎設施——能不能養出某類模型的硬約束

這格管的是「你有多少肌肉、肌肉長什麼樣」：用幾萬張 GPU 還是幾十萬張、用 NVIDIA 還是自研晶片、叢集多大、晶片之間的互連多快、整套訓練工程多成熟。它決定的是**物理上的可能性邊界**——能訓多大的模型、訓多久、每個 token 多便宜、長上下文划不划算。

這格最像你的硬體配置與基礎設施，而且我要強調一個你會立刻有共鳴的點：**掌握整條 stack，才能做別人做不到的最佳化。** 你自建那套全團隊採用的多層 cache library、自己管基礎設施，換來的就是別人用代管服務拿不到的最佳化空間。算力這格的頂級玩法——自研晶片——就是這個邏輯放到國家級資本的版本。

> **迷你預告**：Google 的超長上下文（Gemini 1.5 Pro 早在 2024 就 ship 了 200 萬 token 的視窗，截至 2026-06）主因落在這一格——**自研 TPU 的垂直整合**讓長序列的訓練與服務成本可控。這條因果鏈是本章後面 `## 紙上推演` 要你親手拆的，這裡先記「長上下文→往算力這格看」。

### 第 4 格：研究文化·品味——最難被抄走的那一格

前三格（資料、後訓練、算力）你多少能用錢和人砸出來。**這一格不行。** 研究文化·品味管的是更軟、更貴的東西：這家押注哪個方向、實驗紀律嚴不嚴、迭代多快、人才密度多高、價值取向是什麼（衝能力？衝安全？衝效率？）。它解釋的是一個最關鍵的差異——**為什麼是這一家、在這個時間點、先做出了那件事。**

為什麼 OpenAI 先押對了「對思維鏈做 RL、讓模型在推論時多想一會兒」（reasoning 範式，o1，2024）？為什麼 Anthropic 一開始就把對齊與可解釋性當第一性原則？為什麼 DeepSeek 在出口管制的壓力下，磨出一整套效率文化？這些都不是「資料好」或「GPU 多」能解釋的——是一群人**選擇相信某個方向值得押**的結果。架構可以抄、權重可以蒸餾、技巧論文一發全世界都會，但「在對的時間押對方向的品味」抄不走。

對工程師我這樣翻譯：這像你們團隊的**工程文化**。為什麼有的團隊就是先想到該強制 OpenAPI 同步、該自建那層 cache、該把可觀測性做在前面？不是因為他們設備比較好，是因為他們的品味與紀律讓他們**先看見**那個該做的決定。文化是組織的索引——它決定你多快查到對的答案。

> **迷你預告**：reasoning 範式為什麼先在 OpenAI 落地（而不是別家），主因有一大塊落在這一格（先押 RLHF、先押 test-time compute 的研究取向）。展開留到 ch05 與 ch14。

### 第 5 格：產品取向——模型被它要服務的產品壓成什麼形狀

最後一格常被外行忽略，卻往往是最深的成因：**這個模型最終要服務誰？** 是消費者（要好聊、安全、別出包）、是開發者（要 API 穩、會寫程式、能當 agent）、是企業（要可控、可私有部署）、還是自家的某個產品（要餵 Search、餵 Workspace、餵社群平台）？要開放權重給全世界下載，還是閉源只走 API？

這格的威力在於它會**反過來塑形前四格**。你決定要服務 Search 這種行星級流量的產品，就被逼著把效率與長上下文做到極致（算力那格被產品壓著走）；你決定要做開放權重生態，就會出全尺寸的模型、選更寬鬆的授權（資料與產品策略一起被帶動）。模型最後長成什麼形狀，常常是「它要服務的產品」在背後施的力。

對工程師我這樣翻譯：這像你的系統**要服務的 SLA 與使用場景**。同一套微服務骨架，你拿去做面向消費者的高併發 API，跟拿去做內部後台批次，會被調校成完全不同的樣子——不是技術不同，是**要服務的東西不同**。

> **迷你預告**：Meta 曾用開放權重（Llama）養出龐大生態，又在 2026-04 轉向閉源前沿（Muse Spark）——這個戲劇性的轉向，主因正落在這一格（產品取向變了：從「送權重換生態」改成「靠自家 30 億用戶平台與專有資料變現」）。這是全書最「新鮮又反直覺」的個案，留到 ch08，標為 **2026 轉折、非定局**。

## 怎麼讀一個 `## 成因拆解`

五格擺好了，現在講**怎麼用**。從 ch05 開始，每一章結尾都有一個叫 `## 成因拆解` 的段落，它是這本書的招牌。它永遠是同一個三段式結構，你學會讀它，就學會了全書最重要的閱讀動作。

**第一段：五格歸因。** 一張表，把本章現象逐格過一遍，每格標一個記號、寫一句「為什麼」：

```text
記號     意思                     你該怎麼讀
──────────────────────────────────────────────
✓ 主因   這格是這個強項的主要成因   先看這幾格，因果鏈的主幹在這
△ 次因   有貢獻，但不是決定性的     第二層，會放大或修飾主因
（空白） 這格基本不相關             這個強項跟這格沒什麼關係
影子     這個強項的代價／盲點       同一組選擇讓他強在 X，也弱在 Y
```

讀法：**先掃 ✓，那是主幹；再看 △，那是修飾；空白的格子也是資訊**——它告訴你這個強項「不靠」什麼。比方一款模型在 coding 上強，若歸因表的「資料」與「後訓練」打 ✓、「算力」留白，那它在告訴你：這個強項是養法磨出來的，不是靠堆 GPU 堆出來的——這個判斷很重要，因為它預測了「別家就算 GPU 更多，也未必抄得到這個強項」。

**第二段：影子。** 這是這本書跟公關稿、跟排行榜最不一樣的地方，也最對你這種看慣 postmortem 的人的胃口。**每一個強項都有代價。** 讓一家強在 X 的那一組選擇，幾乎一定讓它弱在、或暴露在某個 Y 上。押安全與對齊到極致，可能讓你在某些開放度、速度上讓步；把算力暴力堆到極致，要付能源、成本、資料品質的帳；送出開放權重換生態，等於送出一部分護城河。**沒有只有優點的選擇**——這句話會在全書出現八次以上。一個 `## 成因拆解` 如果只講優點，它就是壞的。

**第三段：何時不成立。** 提醒你這個「強項」是**截至某個日期的快照，不是定律**。它會被哪類任務打臉（某家在數學強不代表在長文寫作強）、被哪家追上、被什麼時間因素推翻（下個版本、下個範式）。這一段是 ch04「誰最強測不準」那把懷疑論的尺，提前裝進每一章——**任何「X 在 Y 上領先」都帶著保存期限**。

讀完這三段，你應該能用一句話複述本章的帳：**「強在 X，因為〔哪幾格〕；影子是 Y。」** 這個句式全書統一，每一家、每一個主題收尾都結出這一句，到 ch21 累積成一張總帳。

## 為什麼幾乎沒有「單一主因」

最後一件，也是這張清單跟你那張 RDS 清單**最不一樣**的地方，必須講清楚，否則你會誤用整個框架。

RDS 那五格——schema、index、查詢、硬體、流量——大致是**可以拆開的**。你可以說「這次慢純粹是少了一個索引」，其他四格沒參與。乾淨、可隔離。

**五變數不是這樣。它們深度纏繞，幾乎找不到「只有一格是主因」的例子。** 原因是這五格在現實裡互為因果、彼此放大：

```text
五格互相牽動的一個典型迴路（以「Google 長上下文」為例骨架）

  產品取向（要服務 Search 這種行星級流量）
        │  逼出對效率的執念
        ▼
  算力·基礎設施（自研 TPU、垂直整合）
        │  讓長序列訓練／服務成本可控
        ▼
  研究文化·品味（DeepMind 的研究縱深押長上下文）
        │  把可行性變成真的做出來
        ▼
  「超長上下文」這個強項長出來
```

看出來了嗎？「Google 長上下文強」這一個強項，你沒辦法說它「只是因為 TPU」。是**產品壓力**逼出效率執念、**算力**（TPU 垂直整合）提供物理可行性、**研究文化**（DeepMind 底）選擇把它做出來——三格扣在一起才成立。抽掉任何一格，這個強項都不會以現在的樣子出現。

所以歸因表上才會有「✓ 主因」和「△ 次因」的分別：**我們標的是『主幹落在哪』，不是『只有這格』。** 一個誠實的 `## 成因拆解`，主因通常是兩到三格，而不是一格。任何「某家強純粹因為他資料好」這種單因解釋，**幾乎一定是把因果鏈砍斷了**——這正是 `## 紙上推演` 要訓練你抓的破綻。

這也是為什麼研究文化·品味（第 4 格）最難被抄走：別家可以複製 TPU 的算力（買晶片）、可以複製長上下文的技術（讀論文），但複製不了「在對的產品壓力下、用對的算力、押對方向」這整個**纏繞的迴路**。能抄走的是單格；抄不走的是格與格之間的扣法。

## 成因拆解

本章沒有一個「某家的強項」可拆——本章拆的是**這個框架自己**。所以這個 `## 成因拆解` 是一個示範：拿一句你常聽到的話「**Google 的長上下文很強**」（截至 2026-06；Gemini 1.5 Pro 早在 2024 就 ship 了 200 萬 token 視窗，是真實且耐久的招牌，⚠️ 更新的 Gemini 3.5 Pro 號稱 200 萬 token＋Deep Think，截至 2026-06-09 尚未 GA，視為目標非定局），用五格把它拆給你看。**這正是你讀完每一章後該能自己做的動作。**

**(a) 五格歸因**

| 變數 | 記號 | 為什麼 |
|---|---|---|
| 資料 | （空白） | 長上下文不是「見過更多資料」給的，是架構與算力給的——這格基本不相關 |
| 後訓練·對齊 | △ 次因 | 長視窗要「用得好」（在百萬 token 裡找對的東西）需後訓練調教，但不是長視窗本身的成因 |
| 算力·基礎設施 | ✓ 主因 | **自研 TPU 垂直整合**讓長序列的訓練與推論成本可控——長上下文每多一倍，注意力與 KV cache 成本就漲，能扛得起這個成本是物理前提 |
| 研究文化·品味 | ✓ 主因 | DeepMind 的研究縱深選擇押注長上下文這個方向，把「算力扛得起」變成「真的做出來」 |
| 產品取向 | △ 次因 | 要服務 Search／Workspace 這種行星級產品的壓力，逼出對效率與長上下文的執念 |

一句話總結：**強在長上下文，因為〔算力·基礎設施 ＋ 研究文化·品味〕，被〔產品取向〕推著走。** 主因落在兩格、被第三格放大——印證了上一節「幾乎沒有單一主因」。

**(b) 影子**

垂直整合自研 TPU 的這套選擇，代價是**綁死在自家生態與供應鏈上**。你不像買 NVIDIA 那樣有現成的全球工具鏈、社群、現貨市場——TPU 的整套軟硬體都得自己扛，外部開發者想用你的硬體做最佳化也更難。更廣的影子：長上下文能塞進去，不等於模型在百萬 token 裡**真的找得到、用得對**那根針——「視窗大」和「長距離推理可靠」是兩件事，後者常被前者的數字蓋過去（這個陷阱 ch04 會用 benchmark 的角度再戳）。

**(c) 何時不成立**

這個「長上下文強」的招牌，在幾種情況下會被打臉：(1) **被任務打臉**——你要的是短而難的數學或 coding，長上下文這個強項根本用不上，這時該看的是別格、別家。(2) **被別家追上**——上下文長度是會擴散的技術，2026-06 各家旗艦多半都往百萬 token 走，「長」正在變成 table stakes（基本門檻）而非差異化。(3) **被時間打臉**——這是**截至 2026-06 的快照**；下一個版本、下一個範式（比方「無限上下文」或「外部記憶」成熟）都可能讓「視窗多長」這件事不再是賣點。**記住：這是快照，不是定律。**

## 紙上推演

### 推演題1：把「Google 長上下文很強」歸到五格 **[15 分鐘] ★★**

**先不准看上面的 `## 成因拆解`。** 拿「截至 2026-06，Google 的長上下文很強」這句話，自己畫一張五格表，每格標 ✓主因／△次因／空白，並各寫一句「為什麼」。寫完再回去對照，看你把主因放對了沒。這是全書你要重複做幾十次的動作，第一次就練到位。

#### 推演解答

關鍵是分清「**這個強項的物理前提是什麼**」跟「**誰選擇把它做出來**」。

- **資料（空白）**：陷阱在這格。很多人直覺「強＝資料好」，但長上下文不是資料給的——你餵再多資料，視窗開不大、成本扛不住，照樣做不出長上下文。這格基本不相關，**留白本身就是答案**。
- **後訓練·對齊（△）**：把百萬 token 的視窗「用好」需要調教，但這是修飾，不是長視窗的成因。
- **算力·基礎設施（✓ 主因）**：這是物理前提。上下文每長一倍，注意力與 KV cache 的成本就漲；自研 TPU 的垂直整合讓 Google 扛得起這個成本。沒有這格，後面都不用談。
- **研究文化·品味（✓ 主因）**：算力扛得起只是「可以做」，DeepMind 選擇押注長上下文這個方向、真的把它做出來，是「做了」。可以做 ≠ 做了，中間隔著品味。
- **產品取向（△）**：要服務 Search/Workspace 的壓力，逼出對長上下文與效率的執念，放大了前兩格。

**常見誤解**：把主因放到「資料」格（被「強就是料多」的直覺騙了），或只放一格（「就是 TPU 嘛」）而漏掉研究文化。記住兩條：**長上下文往算力看，不往資料看**；**主因通常兩三格，不是一格**。你若能講出「算力是物理前提、文化是選擇、產品是推力」這條三格扣在一起的鏈，這題就過了。

### 推演題2：抓出「他們強純粹因為資料好」的破綻 **[10 分鐘] ★★**

有人跟你說：「某家模型 coding 特別強，純粹就是因為他們餵了比別人多的 GitHub 程式碼。」用本章的框架，指出這句話**為什麼幾乎一定是錯的**——不是說資料不重要，是說「純粹因為」這三個字錯在哪。

#### 推演解答

破綻在「**純粹**」——它把一條纏繞的因果鏈砍成單格。

餵 GitHub 程式碼（資料，第 1 格）確實是 coding 能力的必要原料，但它遠遠不夠。同一批程式碼資料，pretraining 完只是一塊「會接程式碼」的原料——它**會不會照你的指令寫、會不會自己跑測試、會不會 debug、能不能當 agent 多步完成任務**，這些是**後訓練·對齊**（第 2 格）磨出來的；要訓得動、要支援長的程式碼上下文，靠**算力**（第 3 格）；先看見「該把模型練成會用工具的 coding agent」這個方向，是**研究文化·品味**（第 4 格）；把產品定位在開發者、逼著模型往 agent/coding 長，是**產品取向**（第 5 格）。

所以正確的講法是：「coding 強，主因落在**資料 ＋ 後訓練**兩格，被**產品取向**推著走」——而不是「純粹因為資料」。「純粹因為 X」這種單因解釋是本書要你終生警惕的破綻；現實裡的強項，主幹幾乎都是兩三格扣在一起。**任何把模型強弱講成單一原因的說法，先假設它砍斷了因果鏈，再去找它漏了哪幾格。**

### 推演題3：用五格對帳你自己的系統 **[10 分鐘] ★**

把你那次「尖峰 RDS CPU 降 40%」的最佳化，對應回本章開頭那張「同一台 RDS 五個源頭」的清單：你那次的主因落在哪一格？是 schema、index、查詢、硬體還是流量？然後想一想——你當時是怎麼確定主因的？這個「逐格排查、定位主幹」的動作，跟本章要你對模型做的，是不是同一個動作？

#### 推演解答

這題沒有標準答案（看你當時的真實案例），重點是**讓你親身體會這個框架的用法不是新東西**——它就是你早已內化的瓶頸診斷，換了個領域。

多數人那次 RDS 最佳化的主因落在 **index 或查詢寫法**這格（一條沒走索引的查詢、一個 N+1）。而你當時怎麼確定的？你不是猜的——你跑 profiling、看慢查詢日誌、逐格排除，直到主幹浮現。**這正是 `## 成因拆解` 在做的事**：對「模型強弱」這個現象跑同一套逐格排查，標出主幹（✓）、修飾（△）、不相關（空白）。

差別只有一個，而且這個差別很重要：RDS 那五格大致可隔離（「純粹是少了個索引」常成立），但模型那五格深度纏繞（「純粹因為資料」幾乎不成立）。所以對模型做歸因時，你要比做 RDS 診斷時**更小心『單因』的誘惑**——主因通常是兩三格，不是一格。帶走這句：**你已經會這個動作了；本章只是把它指向一個新對象，並提醒你新對象的格子是纏在一起的。**

## 自我檢核

口頭回答，講得出來才算過關。優先答「為什麼」與「如果不這樣會怎樣」。

1. 五變數是哪五個？照固定順序、固定用詞講出來，並各用一句話說它管哪一類差異。（講不出順序與用詞，後面每一章的 `## 成因拆解` 你都會讀得卡。）
2. 為什麼「同一台 RDS、效能差百倍」這個對照，能說明「同一張藍圖、八家強項天差地遠」？兩邊的「藍圖之外那幾件事」各是什麼？
3. 一個 `## 成因拆解` 的三段是哪三段？為什麼說「只講優點的成因拆解就是壞的」——影子那段不寫，你會錯過什麼？
4. 為什麼五變數「幾乎沒有單一主因」，而你那張 RDS 排查清單卻常常能隔離出單一主因？這個差別會讓你在歸因時多警惕什麼？
5. 為什麼「研究文化·品味」這一格最難被抄走？別家能複製前三格的哪些東西、複製不了什麼？
6. 「Google 長上下文強」的主因為什麼落在算力與研究文化、而不是資料？如果有人堅持是「資料好」，你怎麼反駁？
7. 為什麼產品取向（第 5 格）會「反過來塑形前四格」？舉一個你能想到的方向（比方「要服務行星級流量的產品」會逼出什麼）。
8. 每一章 `## 成因拆解` 結尾那句「強在 X，因為〔哪幾格〕；影子是 Y」，為什麼要強制每章都結這一句？它最後累積成什麼？
