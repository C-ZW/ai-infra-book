# 寫作風格指南（內部文件，非書籍內容）

本文件是全書的寫作契約。每一章都必須遵守，否則全書會失去一致性。
書名（工作名）：《藍圖之外：同一套 Transformer，八家大模型為何各擅勝場》。

## 這本書是什麼、不是什麼

- **是**：一本「欣賞與直覺」的科普書，從**資深後端／分散式系統工程師的視角**讀當代大型語言模型產業。核心問題只有一個——**底層明明是同一張架構藍圖（decoder-only Transformer ＋ self-attention），為什麼各家養出來的模型強項天差地遠？** 讀完要能用自己的話，對另一個工程師講清楚「OpenAI、Google、Anthropic、Meta、Mistral、xAI、DeepSeek、Qwen 各擅長什麼、**為什麼**會這樣」，而不是背一張排行榜。
- **不是**：
  - **不是排行榜**——本書刻意不選「誰最強」。當代模型每幾個月洗牌一次；書要教的是**差異的成因結構**（為什麼會有差異、差異記在哪個帳上），不是某個時間點的名次。任何排名、版本號、價格、GPU 數量都**必須標日期戳**、用「截至 2026-06／以…著稱」而非「就是最好」。
  - **不是技術教科書**——不教你訓練模型、不推導 attention 公式、不教 PyTorch；機制只講到「能支撐『為什麼這家擅長這個』這個判斷」的深度。
  - **不是公關稿或產業八卦**——不吹捧也不唱衰任何一家；每個強項都要同時講出它的**影子（代價／盲點／何時不成立）**。觀點要冷靜、有依據、可被另一個工程師檢驗。
  - **不是轉職用書**——純興趣（2026-06-13 訪談定案，與工作無關）；但工程橋接要讓它讀起來像母語。
- **全書貫穿主張（核心商品，ch21 總收官驗收）**：
  1. **架構不是護城河**：Transformer 是 2017 年的公開論文，attention 怎麼算全世界都看得到。各家底層大同小異，差異不在藍圖。
  2. **模型是「養」出來的，不是「寫」出來的**：沒有人逐行規定模型的行為；你設定一個成長環境（資料、優化目標、算力），能力是**長出來的產物**。同一張藍圖、不同養法 → 不同風格。
  3. **五個變數解釋一切差異**：**資料 × 後訓練／對齊 × 算力與基礎設施 × 研究文化與品味 × 產品取向**。這五格是全書的記分卡（ch03 定調，每章回收）。
  4. **「最強」是測不準的，「風格」才是穩定的**：沒有單一冠軍；穩定可辨識的是各家的**取向與性格**。差異會持續（結構性成因），也會局部收斂（互相抄、趨同）。

## 讀者是誰

唯一讀者：Chewei，6 年以上資深後端工程師（TypeScript/Node.js、NestJS、AWS、K8s）。**技術底子厚，但不是 ML 專家**。可直接當橋接錨點的經歷與直覺（**第一次用某個錨點時點名具體場景，不要泛泛說「你的後端經驗」**）：

- 高併發系統（500K+ 使用者、10K+ CCU）、佇列（SQS）、冪等消費者、最終一致性、backpressure、rate limiting、queue 理論直覺。
- 微服務 greenfield（AWS ECS、TypeScript/NestJS）、自建輕量框架與多層 cache library（全團隊採用）、CI 強制 OpenAPI 同步。
- 效能工程：profiling（Clinic.js）、負載測試（k6/Vegeta）、瓶頸診斷（招牌敘事：尖峰 RDS CPU 降 40%）。
- 維運：K8s 基礎物件（Deployment/Service/HPA/CronJob）、Docker、CI/CD、CloudWatch、自建 FluentBit pipeline、event-driven 結算 pipeline（CronJob→SQS→idempotent consumer）。
- 資料層：MySQL/PostgreSQL 調校、Redis Pub/Sub。

**讀者已具備、可直接當錨點，不需重講**：分散式系統的調度／路由／autoscaling／快取／佇列／冪等／最終一致性；雲與 K8s 的工程意義；**而且他讀過自己書架上的《從後端到 AI Infra》**，所以對 **attention 作為推論期的機制、KV cache、prefill/decode、batching、quantization 的「概念層」已經熟**——談這些可以直接用、當錨點，不要從頭教（一句話喚醒即可）。

**要從較低基礎教起（本書主體，讀者接觸極少）**：模型**訓練側**的一切——pretraining 的資料與優化、scaling laws、後訓練（SFT/RLHF/RLAIF/RLVR）、對齊方法、reward model、MoE 與訓練期的工程、合成資料、評測方法學。用「你懂分散式系統與推論，現在把『模型怎麼被養出來』這一側補上」的節奏帶。**不堆數學**：scaling law 不推導，用「投入 vs 能力的一條可外推曲線」這種直覺講；RL 不寫損失函數，用「給原則讓它照鏡子自評」這種比喻講。

**跨書資源**：讀者書架上有《從後端到 AI Infra》（推論基礎設施、KV cache、PagedAttention、GPU 體系）、《馴服隨機：機率與統計》（分布、期望、抽樣、為什麼 benchmark 會騙人）。談推論成本（ch16）可一句話接上 AI Infra 書（「你那本推論書把這條算到底」）；談評測抽樣偏誤、信賴區間、為什麼小樣本排名不可靠（ch04）可接上機率書——**可以接，但不得依賴**，本書自身要把概念講到能懂。

- 人名：有把握的通行中譯才用＋原文第一次出現附原文（如 Ilya Sutskever、Demis Hassabis、Yann LeCun、梁文鋒）；沒把握就保留原文。
- 台灣用語基準：程式／程式碼（不用「代碼」）、最佳化（全書用「最佳化」，不混用「優化」）、記憶體（不用「內存」）、預設（不用「默認」）、品質（不用「質量」）、資訊（不用「信息」）、相依（dependency）、效能（performance）、運算／算力（compute）、權重（weights）、開放權重（open-weight，不寫「開源模型」除非真的連訓練碼都開）、推論（inference）、訓練（training）。**禁止簡體中文詞彙**。

## 每章骨架（必須遵守，標題名稱照抄——一致性掃描會 grep 這些標題）

1. **章首**：`# chNN — 章名：副標`，接著 blockquote：
   `> **本章解決什麼問題**：…（這章在全書「同一張藍圖、不同養法」地圖的哪裡、回答的是哪一格／哪一家的「為什麼」、跟前後章的關係，3–5 行）`
   每個 Part 的第一章（**ch01、ch05、ch13、ch20**）在 blockquote 後放**全書 ASCII 路線圖**（基準圖在 outline.md，照抄並把「◄你在這裡」標到本章），讀者偏好架構圖先行。
2. `## 從你已知的出發` — 橋接段：把本章核心問題錨定在讀者**具體的**工程實戰上（點名場景：你的多層 cache library、ECS 微服務、SQS 冪等消費者、RDS CPU 降 40%、HPA autoscaling、FluentBit pipeline…），或錨定在他讀過的 AI Infra／機率書概念上。要具體，不要泛泛。
3. **核心內容**（多個 `##` section）。敘事與機制交織，但一切**服務於「為什麼這家／這個現象會這樣」的因果**，不是公司流水帳。company 章節：先講「以什麼著稱」（標日期戳、查 landscape），再層層回答「為什麼是這家」。theme 章節：講清楚一個橫切面（如推理模型、對齊演化、算力競賽）如何在各家身上展開、為什麼。
4. `## 成因拆解` — **本書招牌段落（每章必有，名稱照抄）**。用 ch03 的「五變數記分卡」把本章現象**歸因**：哪幾格是主因、各記一筆。**必含三部分**（可用表格）：
   - **(a) 五格歸因**：資料／後訓練對齊／算力基礎設施／研究文化品味／產品取向，逐格說明這格在本章現象裡扮演什麼角色（主因打✓、次因打△、不相關留白），每格一句「為什麼」。
   - **(b) 影子**：這個強項（或這個選擇）的**代價／盲點**是什麼——同一組選擇讓他強在 X，也讓他弱在／暴露在 Y（對應讀者偏好的「故障視角」，必答）。
   - **(c) 何時不成立**：這個「強項」在什麼情況下會被打臉（被哪類任務、被哪家、被什麼時間因素推翻）；提醒讀者這是**截至某日期的快照、不是定律**。
5. `## 紙上推演` — 動手做的形式（純敘事科普書，無可執行 lab；2026-06-13 訪談定案）。每章 2~4 題，標注預估時間（如 **[15 分鐘]**）與難度（★~★★★）。題型：**口頭重講**一段因果、**翻譯**（把一個模型現象對應回五變數裡的某一格）、**反事實推理**（如果某家沒有 X 條件會怎樣）、**找破綻**（指出一個流行說法的問題，如「DeepSeek 只花 560 萬美金」）、**對帳**（把你維運過的某系統取捨對應到某家的取捨邏輯）。題目用 `### 推演題N`，緊接 `#### 推演解答` 給完整解答與討論（為什麼這樣想、常見誤解）。
6. `## 自我檢核` — 5~8 個問題，**口頭自答**：講得出來才算過關，講不清楚＝還沒懂。對齊「能不能把這段因果講給另一個工程師聽」的深度，**優先出「為什麼」與「如果不這樣會怎樣」**，少出「是什麼／哪一版」。
7. `## 延伸閱讀` — 真實連結（以 `_meta/landscape-2026-06.md` 驗證過的為優先），每條一句話說明為什麼值得讀、讀哪一段。優先：原始論文與技術報告（Attention Is All You Need、Chinchilla、InstructGPT、Constitutional AI、DeepSeek 技術報告…）、當事人的演講／訪談、可信的產業整理。

## 深度標準（每章驗收）

- **每章至少一個「被講透的成因個案」**：不是泛談「他們資料好」，而是**一條具體的因果鏈**講到底。例：Google 長上下文 ← TPU 垂直整合讓長序列訓練／服務成本可控 ＋ DeepMind 研究底 ＋ 要服務 Search 這個產品壓力；DeepSeek 低成本 ← MLA 砍 KV cache ＋ MoE 稀疏 ＋ GRPO 省掉 reward model ＋ 出口管制逼出的效率文化。
- **每章至少一個帶真實數字的具體例**（從 landscape 取，標日期戳）：版本、上下文長度、參數量級、GPU/TPU 數量級、訓練成本主張等——**有爭議的數字要同時給「主張值」與「爭議／但書」**（如 DeepSeek 訓練成本、各家 GPU 數量）。
- **五變數框架一致**：每章 `## 成因拆解` 用的五格名稱與 ch03 完全一致，不得各章自創分類。
- **事實照 landscape，不憑記憶**：任何版本號、排名、價格、硬體、訓練成本、技術歸屬（誰發明 MLA／GRPO／Constitutional AI）一律以 `landscape-2026-06.md` 為錨；章節作者可做 3–8 次自己章節的查證，但與 landscape 衝突要回報、不得擅自改基準。
- **沒有絕對排名**：禁止「X 是最強的模型」這類斷言。一律「截至 2026-06，X 以…著稱／在…benchmark 上領先」並點出 benchmark 的侷限。
- **影子必寫**：違反「只講優點」——每個強項都要有 `## 成因拆解` 的 (b)(c) 兩段把代價與失效條件講出來。
- **橋接點名具體場景**：第一次用某個讀者經歷當錨點時，點名真實系統（cache library、RDS、SQS…），不泛稱。

## 全書一致的隱喻與用語（ch01／ch03 定調，全書回收）

- 核心隱喻：**藍圖（架構）vs 養成（一切架構之外的事）**。模型是「養」出來的；各家是「同一張藍圖、不同養法」。
- 記分卡：**五變數**＝資料 / 後訓練·對齊 / 算力·基礎設施 / 研究文化·品味 / 產品取向。固定順序、固定用詞。
- 收束句式：每家／每主題用一句「**強在 X，因為〔哪幾格〕；影子是 Y**」收尾，全書累積成 ch21 總帳。
