# ch17 — 多模態：文字之外的戰場

> **本章解決什麼問題**：到 ch16 為止，我們談的「強項」幾乎都還在文字裡——coding、長文、推理、多語言。但文字 benchmark 正在飽和（MMLU-Pro、GPQA 在頂端早已擠到分不出高下，見 ch04），純文字的競爭快要榨乾。這一章把鏡頭轉向**文字之外**：影像、音訊、影片。要回答的是——為什麼「原生多模態（native multimodal）」會成為下一個差異化戰場、誰為何領先，以及這條戰線的成因主要記在五變數的哪幾格。本章是**橫切面**章節：不重講各家完整版本史（那是 Part II 與附錄 A 的事），只把「多模態」這一個切面如何在各家身上展開、為什麼，攤給你看。會頻繁接回 ch06（Google 的 TPU／長上下文鏈）與 ch08（Llama 4 是 Meta 第一個原生多模態），因為這一章的招牌因果，正好是那兩章的延伸。

## 從你已知的出發

你做過一件每個後端工程師遲早會做的事：**把很多種輸入格式，統一收斂成同一套 schema、進同一個服務**。

想想你那條 event-driven 結算 pipeline。上游來的東西從來不乾淨：有的是 webhook 推來的 JSON，有的是第三方丟的 CSV，有的是另一個內部服務發的 protobuf，欄位名稱、時區、金額單位、空值約定全都不一樣。你有兩種做法可選：

- **後接法（bolt-on）**：每來一種新格式，就在外面臨時加一個 adapter，把它「翻譯」成主流程看得懂的樣子，再塞進去。快、省事、不用動核心。但每個 adapter 都是一道接縫，接縫越多，邊界情況越多，最後你的核心服務其實從沒真正「理解」這些異質輸入——它只是被餵了一堆被勉強對齊過的代理資料。
- **原生統一法（native）**：你乾脆從一開始就設計一套**共用的 canonical schema**，讓所有來源在進系統的那一刻就活在同一個語意空間裡，下游所有邏輯都直接在這個統一表示上運作，不用每次都先猜「這筆是從哪個 adapter 來的」。前期設計貴得多，但一旦建好，跨來源的對帳、查詢、推理全都順了——因為它們本來就同源。

這章要講的「原生多模態 vs 後接多模態」，**幾乎就是這同一道取捨**，只是把「異質資料格式」換成了「文字／影像／音訊／影片這些異質模態」。一個模型要不要真的「原生看懂」影片，還是只是在文字模型外面黏一個影像編碼器當 adapter——這個選擇，跟你決定要不要為結算 pipeline 設計一套 canonical schema，是同一種工程判斷。差別只在尺度，與——你馬上會看到——它最後又一次回到了 ch06 那條算力的因果鏈上。

## 為什麼是「下一個」戰場：文字快榨乾了

先講清楚這一章為什麼存在於「現在」，而不是兩年前或兩年後。

截至 2026-06，純文字的競爭出現了一個結構性訊號：**頂級文字 benchmark 飽和了**。ch04 已經攤過這件事——MMLU-Pro（硬知識多選）頂端的模型擠在 88～94% 這個窄帶裡，GPQA Diamond（研究所級科學推理）前沿模型逼近 94%，AIME（競賽數學）對推理模型幾乎是滿分。當一張尺的刻度已經被大家踩到頂、再也分不出誰高誰低時，它就**不再是差異化的戰場**了。reasoning 模式同樣：到 2026 年，「會不會推理」已經是入場券（table stakes），不是區別點——八家全都有 reasoning 模式（見 ch14）。

當一個維度上大家都做到滿分，競爭自然會**外溢到還沒飽和的維度**。截至 2026-06，那個維度主要是兩塊：**agentic**（讓模型自己規劃、用工具、檢查自己的工作）和**多模態**。這一章談後者。多模態之所以是「下一個」戰場，不是因為它比文字重要，而是因為——

- 文字那條賽道的刻度用完了，**這裡刻度還很長**：影像、音訊、影片的理解，遠沒到「大家都滿分」的程度，還有大量空間可以拉開差距；
- 它直接連到**產品**：使用者真正想要的下一代體驗，幾乎都是多模態的——對著手機螢幕問問題、用語音即時對話、讓裝置端的模型看著鏡頭幫你做事。產品取向（五變數第五格）會把實驗室往這個方向硬推。

換句話說，多模態之所以「現在」變成戰場，一半是因為文字戰場打到沒仗可打了（飽和），另一半是因為產品的下一站非它不可。

## 原生 vs 後接：同一張藍圖下的兩種長法

這是本章要講透的那條因果。先把兩條路的**機制差異**講清楚，再講它為什麼會分出強弱。

記得全書的脊椎：底層是同一張藍圖（decoder-only Transformer ＋ self-attention），各家差在**養法**。多模態恰好是這個命題一個漂亮的縮影——同一張藍圖，兩種把「非文字」餵進去的養法。

### 後接法：把一個影像編碼器釘在文字模型外面

最常見、最省事的做法是這樣的：你已經有一個訓練好的純文字 LLM，現在想讓它「看得到圖」。於是你拿一個現成的視覺編碼器（vision encoder，把圖片轉成一串向量），在它後面接一個小小的**投影層（projection layer）**，把影像向量「翻譯」成 LLM 看得懂的 token 空間，再把這些 token 塞進文字模型的輸入裡。文字模型本體常常是**凍結的**（不動它的權重），你主要訓練那個投影層讓兩邊對齊。

這就是「後接（bolt-on）」——學界這條路線的代表是 LLaVA、MiniGPT-4 這類「frozen 視覺編碼器 ＋ 一層投影」的設計。它的好處跟你的結算 pipeline 加 adapter 一模一樣：**快、便宜、不用重訓核心**。但它的代價也一樣：影像理解是**事後黏上去的一層代理**，文字模型本體從沒在預訓練時真正「跟影像一起長大」。圖文之間的深層關係，只能擠過那一道窄窄的投影接縫來傳遞。

### 原生法：從預訓練第一天就把多種模態餵進同一個流

原生多模態（native multimodal）走的是另一條路：**從預訓練階段就把文字、影像、音訊、影片一起餵進同一個模型**，讓它在同一個 token 流裡學會跨模態的關係。

截至 2026-06，這條路最常被引為標竿的是 **Google 的 Gemini**：所有模態都被表示成同一個 token 流裡的離散 token，模型在一個統一的向量空間裡同時學文字、影像、音訊、影片——語意上相近的東西會聚在一起，不管它原本是哪種格式。**Meta 的 Llama 4**（2025-04 發表，見 ch08）走的是技術上叫「**early fusion（早期融合）**」的同一個家族：把文字 token 與視覺 token 在進入模型主幹的那一刻就**融進同一條骨幹**，於是可以拿大量「未標註的文字＋影像＋影片」一起做預訓練。Llama 4 是 Meta 第一個**原生多模態、也是第一個 MoE** 的 Llama——這兩件事擺在一起不是巧合，後面成因拆解會說為什麼。

> 對照你的 canonical schema：原生法就是「所有來源在進系統那一刻就活在同一個語意空間」。影像不是被翻譯成文字的代理，而是**本來就跟文字同源**地被模型認識。

### 為什麼原生通常贏：接縫的代價

兩條路放在同一張藍圖下，原生為什麼通常更強？關鍵在那道**接縫**。

後接法把跨模態的理解全部壓進「投影層」這一道窄門。文字模型本體是先學會純文字、之後才被勉強對齊到影像的，它對「圖裡這個東西和句子裡那個詞是同一件事」這種**深層對應**，只能透過那層薄薄的投影去猜。原生法不一樣——模型在預訓練時就**同時**見過「這張圖」與「描述它的文字」「相關的聲音」，跨模態的關聯是**長進權重深處**的，不是事後糊上去的一層。

這正是你結算 pipeline 的教訓放大版：每加一個 adapter，就多一道接縫、多一批邊界情況、多一個「核心其實沒真懂這個來源」的盲點；而一套共用的 canonical schema，雖然前期貴，但讓所有跨來源的推理都順，因為它們本來同源。**原生多模態 = 為模態設計的 canonical schema；後接多模態 = 一堆 adapter。** 接縫越少，跨模態推理越深、越不容易在邊界上出錯。

⚠️ 但別把這寫成定律。後接法不是「錯」，它在很多場景是**正確的工程取捨**——尤其當你只想給既有文字模型加個看圖能力、不想付重訓的天價時。學界也指出：融合的時機與方式沒有單一最優解，不同任務各有適合的點。所以正確的說法是：截至 2026-06，**原生路線在「深層跨模態理解」這個軸上通常更有上限**，但它的入場費（資料＋算力）也高得多——而那個入場費，就是下一節要講的門檻。

## 影片＝偽裝成多模態的長上下文問題

現在把這一章接回 ch06。這是本章最重要的一條因果鏈，也是「為什麼領先的名單跟長上下文的名單高度重疊」的真正原因。

問一個看似天真的問題：**為什麼影片是多模態裡最難的那一塊？**

答案不在「影片很複雜」這種模糊話，而在一個你最有直覺的地方——**序列長度**。一段影片在模型眼裡是什麼？是一連串影格，每一影格又被切成一堆視覺 token，再加上音軌的 token……加起來，**一段影片可以輕易是上百萬 token 的序列**。也就是說：

> **「能原生吃影片」與「能吃很長的上下文」，本質上是同一種能力。** 影片是一個偽裝成多模態問題的**長上下文（long-context）問題**。

這一下就把整章接回了 ch06 那條被講透的因果鏈。你在 ch06 學到：長上下文主要是一道**成本題**，不是演算法題——序列拉長，KV cache 的記憶體與頻寬會脹得很快（這是你那本 AI Infra 書算到底的帳），attention 的計算量也隨序列長度不便宜。誰能把超長序列的訓練／服務成本壓到可規模化，誰就敢把窗口開大。

把這條鏈套到影片上：

```
影片 = 上百萬 token 的超長序列
      |
      v  與「長上下文」是同一種能力（接 ch06）
能不能原生吃影片 = 能不能負擔超長序列的訓練/服務成本
      |
      v  成本由算力結構決定（接 ch06 的 TPU 鏈）
自研晶片/垂直整合 → 超長序列成本可控 → 敢做原生影片
      |
      v
原生多模態的領先名單，與長上下文的領先名單高度重疊
```

這就解釋了一個本來看起來像巧合的事實：截至 2026-06，**以原生多模態著稱的 Google**，正好也是**以超長上下文著稱的 Google**（見 ch06）——因為這兩個招牌長在同一條算力鏈上。Gemini 1.5 Pro 早在 2024 年就真的出貨過 200 萬 token 的窗口，Gemini 3.5 Pro 在 2026-05 的 I/O 上把 2M 窗口當頭條企圖（⚠️ 截至 2026-06-09 還沒 GA、沒有獨立 benchmark，當企圖讀不當事實）。能撐起 2M 文字窗口的那套基礎設施，正好就是能原生吃影片所需要的那套。**長上下文與原生影片，是 TPU 垂直整合這同一筆投資的兩個出口。**

反過來看 Meta：Llama 4 Scout 號稱業界最長、達 1,000 萬 token 的上下文窗口，而它同時是 Meta 第一個原生多模態模型——這同樣不是巧合，是同一條「能扛超長序列就能扛影片」的鏈。至於 Mistral，**Mistral Small 4**（2026-03-16，Apache 2.0）把原本分開的 Pixtral（視覺）／Magistral（推理）／Devstral（coding）**折進同一個 119B 的 MoE 模型**——這是「歐洲效率派」（見 ch09）的多模態打法：不拼影片那條最貴的賽道，而是把視覺能力以高效率的方式收進一個小而通用的開放權重模型裡，走「夠用、便宜、開放」的路線。

## 成因拆解

### (a) 五格歸因

把「為什麼原生多模態（尤其原生影片）會領先在少數幾家」這個現象，攤回五變數記分卡：

| 變數 | 主/次 | 為什麼 |
|---|---|---|
| **資料** | **✓主因** | 原生多模態要在預訓練就餵入海量「文字＋影像＋音訊＋影片」對齊資料。誰手上有大規模、配對好的多模態語料（如 Google 的 YouTube／Search／影像生態），誰就有東西可餵。沒有這批資料，原生路線根本起不了步。 |
| 後訓練·對齊 | △次因 | 多模態指令微調、讓模型聽話地描述圖／看影片回答，有貢獻；但它不是「原生 vs 後接」這個架構分野的主因，主因在預訓練怎麼餵與算力扛不扛得住。 |
| **算力·基礎設施** | **✓主因** | **影片＝超長序列，原生影片＝能負擔超長序列的訓練／服務成本（接 ch06 的 TPU 鏈）。這是本章被講透的因果：自研晶片／垂直整合把超長序列成本壓到可控，才敢做原生影片。長上下文的領先名單與原生多模態的領先名單因此高度重疊。** |
| 研究文化·品味 | △次因 | 願不願意付「原生重訓」的貴前期成本、而不是走便宜的後接捷徑，是一種研究品味的選擇（Google 的原生取向、Meta 的 early fusion 押注）；但它本身要靠資料與算力兩格才兌現得了。 |
| **產品取向** | **✓主因** | 使用者要的下一代體驗是影片／語音／裝置端。要餵 Search、YouTube、Workspace、3B 用戶平台這些**本來就是多模態**的產品，會把實驗室硬推向原生多模態——產品形狀把模型的養法壓成多模態。 |

一句話收束：**原生多模態（尤其原生影片）強在少數幾家，因為〔資料＋算力·基礎設施＋產品取向〕三格咬合；影子是更難評測、幻覺更隱蔽、成本更高。**

### (b) 影子

多模態這條戰線的強項，每一個都拖著一道對應的影子——而且這些影子比文字時代更難察覺：

- **評測更難（接 ch04）。** 文字至少還有 MMLU-Pro、GPQA、SWE-bench 這些（雖然會飽和、會污染）相對成熟的尺。多模態的評測**更不成熟、更難驗證**：一個模型「看懂」一張圖、「理解」一段影片到什麼程度，很難用一個乾淨的分數定下來。ch04 對文字 benchmark 的所有警告（污染、過擬合、飽和、刷榜），在多模態這裡只會更嚴重——因為連「正確答案」本身都更主觀、更難建乾淨的測試集。所以任何「某家多模態最強」的宣稱，可信度要打更大的折。
- **幻覺更隱蔽。** 文字幻覺你還可能讀出破綻；但模型對一張圖、一段影片「自信地說錯」時，**錯誤藏在感知層**，更難抓。它可能無中生有地描述圖裡根本沒有的物件、或把影片裡的因果順序講反，而表面上講得頭頭是道。模態越多，模型能「一本正經胡說」的表面積就越大，而你能查核的工具越少。
- **成本更高。** 這是 ch06 那條鏈的直接代價：影片是超長序列，原生處理它的訓練與推論成本，比純文字高一個量級。這道帳並非人人算得過來——所以原生影片才會集中在有自研晶片／垂直整合（接 ch16）的少數幾家手上。對沒有那種成本結構的實驗室，硬上原生影片就是燒錢黑洞。這也是為什麼 Mistral 這種效率派，選擇把多模態收進高效率的小 MoE，而不是去拼影片那條最貴的賽道。
- **接縫風險換成了重訓風險。** 原生路線雖然少了後接的接縫，但它把賭注押在「重訓一個多模態基座」上——而一旦這個基座在某個模態上沒長好，你沒有後接法那種「換掉一個 adapter 就好」的彈性，整個基座的問題會更難局部修。Llama 4 Behemoth（~2T 規模）在 2T 尺度上遇到 MoE 路由／chunked-attention 的麻煩、最終被擱置（見 ch08，⚠️ announced 未出貨），正是原生＋超大規模這條路前期賭注很重的提醒。

### (c) 何時不成立

這個「原生多模態領先」的判斷，在什麼情況會被打臉？

- **被任務打臉**：原生不是對所有多模態任務都更划算。如果你的需求只是「給既有文字模型加個看簡單圖表的能力」，後接法又快又便宜又夠用——這時候付原生重訓的天價反而是過度工程。學界也明說：融合時機沒有單一最優解，視任務而定。「原生通常上限更高」是**就深層跨模態理解這個軸**而言，不是放諸所有場景皆準。
- **被別家打臉**：後接法在快速進步。視覺編碼器、投影對齊的技巧一直在變強，後接模型在許多實用任務上的差距在縮小。而且「原生 vs 後接」的界線本身在模糊——有些設計是混合的（部分原生、部分後接）。把它看成一個光譜、而不是非黑即白的二分，比較誠實。
- **被時間打臉**：本章所有版本與狀態都是 **2026-06 的快照**。Gemini 3.5 Pro 的 2M 窗口截至 2026-06-09 還沒 GA、沒有獨立 benchmark（⚠️）；Llama 4 Behemoth 已擱置、Meta 在 2026-04 整個轉向閉源的 Muse Spark（見 ch08，⚠️ 2026 轉折非定局）；各家旗艦版本每幾個月就洗一次牌。三個月後（見 maintenance 重掃）這些十之八九要更新。**穩定的是「影片＝偽裝成長上下文的成本題、原生需要扛得起超長序列的算力結構」這個成因；會過期的是任何一家此刻的具體版本與名次。** 別把任何一個時間點的領先，當成多模態這場仗的終局。

## 紙上推演

### 推演題1 — 翻譯：把「Gemini 多模態強」對應回五變數 **[10 分鐘]** ★★

有人對你說：「Gemini 原生多模態強，就是因為 Google 有 YouTube，影片資料多。」這句話對嗎？把它對應回五變數，指出它**對在哪、漏了什麼最關鍵的一格**。（提示：這題 ch06 出過親戚版，但這次焦點在「原生影片」。）

#### 推演解答

對了一格，漏了並列的一格。YouTube／影像資料確實是真實優勢，對應到**資料**那一格——所以本章把資料記為 ✓主因，沒有留白。但這句話的問題是把資料當成**唯一**主因，漏掉了與它並列、而且常常更卡脖子的一格：**算力·基礎設施**。

關鍵反問：**就算別家也弄得到大量影片資料，他訓練得起、服務得起嗎？** 影片是上百萬 token 的超長序列，吃的是「把超長序列攤開算」的能力，而那回到 ch06 的 TPU 成本結構。所以正確歸因是：資料（✓）讓 Google **有東西可餵**，算力（✓）才讓他**餵得起、用得起**，再加上產品取向（✓，要餵 Search／YouTube 這些本來就多模態的產品）把他**硬推**向原生路線——三格咬合。只講資料，等於把一條三格的因果鏈砍成一格。這正是全書反覆訓練你避免的「單因謬誤」：當一個現象看起來有個很明顯的原因時，先問「還有沒有並列的另一格，少了它這個明顯原因也兌現不了？」

### 推演題2 — 反事實：如果 Google 沒有自研 TPU **[15 分鐘]** ★★

反事實推理：假設 Google 從沒做 TPU，跟所有人一樣跟 NVIDIA 排隊買 GPU、付市場價。它還能不能是「原生多模態（尤其原生影片）最著稱的那家」？把因果鏈倒過來推一遍。

#### 推演解答

很可能不能——至少不會是現在這個樣子。把鏈倒推：原生影片 ← 要負擔上百萬 token 超長序列的訓練／服務成本 ← 而那個成本由算力結構決定。如果 Google 的每一片算力都要跟 NVIDIA 用市場價買、還要排隊，這道「原生吃影片」的成本帳很可能算不過來——它會被逼著做**保守一點的選擇**：要嘛把影片能力做得很受限（短片段、低解析度、只給少數人用），要嘛乾脆走後接法（黏個影像編碼器，省下重訓的天價）。

注意這個反事實漂亮地證明了一件事：**Google 的多模態領先，主因不在「想到了什麼別人想不到的演算法」，而在它的成本結構不一樣**。把 TPU 抽掉，YouTube 的資料優勢還在、研究品味還在，但少了「扛得起超長序列」這一格，原生影片這條最貴的路就走不通——資料與品味會空轉。這也正是 ch06 那條鏈在多模態上的回收：**算力·基礎設施是「能不能做某類模型」的硬約束**，不是錦上添花。

常見誤解：以為抽掉 TPU，Google 還是能靠「資料多」或「人聰明」補回來。補不回來——資料餵不進去、品味實現不了，因為卡住的是**成本**這道硬約束，不是想法。

### 推演題3 — 對帳：你的 canonical schema vs 原生多模態 **[15 分鐘]** ★★

把你當年為結算 pipeline 設計 **canonical schema、讓所有異質來源在進系統那一刻就同源**這個決定，跟「原生多模態（vs 後接 adapter）」放在同一張取捨表上對帳。回報是什麼？前期代價是什麼？兩者結構一不一樣？

#### 推演解答

結構幾乎同構，只是把「異質資料格式」換成了「異質模態」。

- **回報**：兩者都是「同源 → 跨來源／跨模態的推理更深、更不容易在邊界出錯」。你的 canonical schema → 所有來源活在同一語意空間，跨來源對帳／查詢全都順，不用每次先猜「這筆從哪個 adapter 來」；原生多模態 → 文字／影像／影片在預訓練就同源，跨模態的深層對應長進權重深處，不必擠過一道窄投影。回報的**形狀**一樣：同源換來深層整合。
- **前期代價**：兩者都得付一筆貴的前期設計／訓練成本。你設計 canonical schema → 比臨時加 adapter 貴得多、慢得多；原生多模態 → 要重訓一個多模態基座，比後接黏一層貴一個量級，還賭上「基座沒長好就難局部修」的風險。
- **影子也同構**：後接法（adapter）的影子是接縫越多越脆、核心沒真懂來源；原生法（schema）的影子是前期重投資、且一旦基座／schema 設計失誤，牽一髮動全身（對應 Llama 4 Behemoth 在 2T 尺度擱置的教訓）。

差異只在尺度與可逆性：你的 schema 改起來還算可控；一個多模態預訓練基座一旦長歪，重來的代價是幾千萬美金級的。但「後接 adapter 求快 vs 原生統一求深」的取捨邏輯，從你的結算 pipeline 一路同構到一個前沿多模態模型。能看出這個同構，你就把「原生 vs 後接」這個架構分野真正讀懂了——它不是 ML 的玄學，是你每天在做的那道工程判斷，放大到了模態的尺度。
