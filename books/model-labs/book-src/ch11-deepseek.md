# ch11 — DeepSeek：效率創新如何撼動成本敘事

> **本章解決什麼問題**：前面幾章拆過用算力暴力換速度的 xAI（ch10）、用自研晶片把成本壓低的 Google（ch06）。這一章換 DeepSeek，回答一個逆向的問題——當你**沒有**最頂的算力（被出口管制卡住，只能用降規的 H800），你怎麼跟巨頭同台？答案是把五變數記分卡裡「**算力·基礎設施**」這一格的劣勢，逼成「**架構**」與「**後訓練·對齊**」兩格的優勢：用 MLA 砍 KV cache、用 MoE 稀疏化、用 GRPO 省掉一整個 reward model。本章的招牌個案不是某個技術，而是一個**數字的正確讀法**：那個傳遍全世界的「560 萬美金」，到底算了什麼、沒算什麼——這是全書「找破綻」訓練最經典的一題。對齊方法的完整譜系（→ch15）、算力競賽全貌（→ch16）、開放權重的哲學（→ch13）留給後面，本章只負責 DeepSeek 這一家。

## 從你已知的出發

你做過兩件事，剛好是讀懂這一章最好的兩把鑰匙。

第一件：你寫了那套**多層 cache library**，全團隊採用。它的核心價值是什麼？是**壓記憶體、省頻寬**——把昂貴的東西（一次次打到 RDS 的查詢）換成便宜的東西（一層層快取裡的命中）。你很清楚一條鐵律：在高併發系統裡，能省下來的記憶體與頻寬，就是能多服務的使用者、能少付的帳單。

第二件：你做過一次「**砍掉一個中間服務、簡化整條 pipeline**」的決定。某個環節本來要維護一個額外的元件——它要部署、要監控、要餵資料、自己也會壞——後來你發現可以用更直接的方式達到同樣的目的，把它整個拿掉，pipeline 反而更穩、更省。少一個會壞的東西，就少一條故障路徑。

DeepSeek 這一家，幾乎就是把你這兩個直覺，放大到「訓練一個前沿大模型」的尺度去執行。**MLA 就是他們的 cache library**——把 attention 推論期最貴的那塊記憶體（KV cache，你那本 AI Infra 書算到底的東西）壓進一個低維潛在空間。**GRPO 就是他們那次「砍掉中間服務」**——把強化學習裡那個又貴又麻煩的 reward model（你可以想成一個額外的評分服務）整個拿掉，改用一群樣本互相比較來打分。一個壓記憶體，一個簡化 pipeline，兩個你都做過。差別只在：你是為了讓尖峰 RDS CPU 降 40%，他們是為了在算力被卡死的情況下，還能把一個 671B 的模型訓出來。

而逼出這一切的，是一個你也很熟的處境：**資源受限**。你那次 RDS CPU 降 40% 的招牌敘事，不是因為你有無限的伺服器可以加，恰恰是因為**不能無腦加機器**，才被逼著去把瓶頸看透、把效率擰乾。DeepSeek 的整個工程文化，就是這個處境的國家級版本——被出口管制掐住算力，逼出了一整套省算力的發明。限制，是創新的母親。

## 以什麼著稱：用一個數字撼動了整個產業（截至 2026-06）

先講「是什麼」，標好日期戳，再回頭層層問「為什麼」。

截至 2026-06，DeepSeek 以一件事著稱：**用遠低於業界共識的成本，做出了接近前沿水準的開放權重模型，並因此在 2025 年初親手引爆了一場市場地震。** 把它釘在時間軸上：

- ✅ **DeepSeek-V3**（2024-12，arXiv:2412.19437）：一個 MoE 模型，**總參數 671B、每個 token 只激活 37B**，256 個路由專家；招牌技術包含 **MLA**、auxiliary-loss-free（無輔助損失）的負載平衡、**Multi-Token Prediction（MTP）**、**FP8** 訓練，吃了 14.8T 個訓練 token。
- ✅ **DeepSeek-R1**（**2025-01-20**）：一個 671B 的**開放權重推理模型**——就是它，震動了市場。它證明了一件事：OpenAI 的 o1（2024 末）開創的那套「對思維鏈做 RL」的推理範式，可以被便宜地、開放地重現出來。
- 🟡 **DeepSeek-V4**（Pro ＋ Flash 兩條線）據報於 **2026-04-24** 推出：MoE、約 **100 萬（1M）token 上下文**，agentic 分數被拿來跟 GPT-5.5、Claude Opus 並列討論，價格卻砍到極低（V4-Pro 的 75% 折扣於 2026-05-22 轉為永久）。⚠️ 這一段是**次級來源、變動極快**的：點版本號、確切定價三個月內十之八九要重掃，當「DeepSeek 持續用低價施壓」這個**方向**讀，別當定版事實。

> ⚠️ 注意措辭分寸：與其說「DeepSeek 是最便宜／最強的模型」，不如說——截至 2026-06，DeepSeek **以「用效率創新撼動了『訓練前沿模型一定很貴』的成本敘事」著稱**。「以效率著稱」是穩定的結構性事實；「現在誰最便宜、誰分數最高」是會過期的（benchmark 名次的洗牌見 ch04）。

但這一章真正的主角，不是上面任何一個版本號，而是一個**金額**。下一節我們就來拆它。

## 那個「560 萬美金」：一個被全世界讀錯的數字

2025 年 1 月，一個說法傳遍了財經與科技媒體：DeepSeek 只花了大約**560 萬美金**（**這個數字是由 GPU 時數〔約 278.8 萬 H800 小時 × 一個假設單價〕回推出來的社群估算，不是 DeepSeek 官方公布的金額**），就訓出了一個能跟 OpenAI、Google 同台的模型。這個數字之所以炸裂，是因為它跟外界對「前沿模型訓練成本」的想像差了**兩三個數量級**——大家以為這是「上億、甚至數十億美金」等級的遊戲。

於是市場用腳投票了。✅ **2025-01-27，NVIDIA 單日股價跌了約 17%，市值蒸發接近 6,000 億美金**（約 589B–600B，視收盤口徑），是當時史上單一公司單日最大的市值損失。連動整個 AI 硬體與電力類股一起重挫。市場當下的恐慌邏輯很簡單：**「如果訓練模型可以這麼便宜，那大家還需要買這麼多 GPU 嗎？」**

這就是全書「找破綻」訓練最經典的一題。這個數字**不是假的**，但它幾乎被所有人**讀錯了**。要把它讀對，得問三個問題。

**第一問：這 560 萬，算的是什麼？** ✅ 它的出處很明確——是 **DeepSeek-V3 技術報告**裡的一個數字：約 **278.8 萬個 H800 GPU 小時**，乘上一個假設的每小時租金，得到約 560 萬美金。但報告自己就講清楚了：這是**「最後一次成功的預訓練 run」的 GPU 算力成本，僅此而已**。

**第二問：那它沒算什麼？** 幾乎所有真正貴的東西都沒算進去：

- **研發成本**：在跑出那次成功的 run 之前，有多少次失敗的實驗、廢掉的 run？不算。
- **人事成本**：一整個頂尖研究團隊的薪水？不算。
- **硬體攤提**：那幾千張 H800 GPU 本身的採購／持有成本（這才是大頭，一張卡的價格遠超這個 run 的「租金」分攤）？不算。
- **前代模型的投資**：MLA、GRPO 這些是在 V2、在 DeepSeekMath 上一路試出來的；V3 是站在這些前作的肩膀上。前面那些投資？不算。

所以正確的讀法是：✅ **560 萬美金是「最後一次預訓練 run 的 GPU 帳單」，不是「打造這個模型的總成本」。** ⚠️ 把研發、硬體、人事全算進去，批評者估的「全包成本」高達 **13 億～16 億美金**等級——跟 560 萬差了大約**兩三百倍**。兩個數字其實都「對」，只是回答的是**完全不同的問題**。

**第三問：媒體還錯在哪？** ✅ 一個極常見的混淆是：**很多報導把 V3 的 560 萬，安到了 R1 頭上**。引爆股災的是 R1（2025-01-20 的推理模型），而那個 560 萬出自 V3 的技術報告、講的是 V3 的預訓練。R1 自己的訓練成本是另外報的（⚠️ 約 29.4 萬美金、512 張 H800 的說法流傳過，但這同樣是一個**窄口徑**的數字，跟 560 萬一樣不能當「總成本」讀）。一個講 V3 預訓練的窄數字，被張冠李戴成「R1 的造價」，再被簡化成「560 萬就能做一個前沿模型」——破綻就是這樣一層層長出來的。

> 用你的話講：這就像有人說「我這個服務一個月雲端帳單只要 50 美金」，然後別人聽成「他用 50 美金做出了整個系統」——漏掉了你三年的開發工時、那些報廢的架構、團隊薪水、還有你早就攤提掉的那批保留實例。**單一個窄口徑的成本數字，乘上一個錯誤的敘事框架，就會傳成神話。**

**但——破除迷思，不等於否定成就。** 這是這一題最容易走偏的地方。560 萬被讀錯了，不代表 DeepSeek 沒做出真東西。✅ 真正站得住、也是市場恐慌**沒有完全錯**的那部分是：**DeepSeek 確實用一整套效率發明，把「達到某個能力水準所需的算力」實實在在地往下壓了**——而且把權重開放出來，讓全世界都能驗證、複製。所以這場地震留下的、durable 的教訓不是「scaling 已死、不用買 GPU 了」，而是更精準的一句：✅ **「演算法效率，跟堆規模一樣重要。」** 證據是什麼？✅ NVIDIA 的股價**在幾個月內就漲回來了**——市場最終認清「更便宜的模型會帶來更多的 AI 應用、反而吃掉更多算力」（這叫 Jevons 弔詭：效率提升不一定減少總需求，常常反而放大它）。恐慌過頭了，但它指向的那個真實內核——效率是一種被低估的護城河——留了下來。

接下來，我們就拆這個「真實內核」是怎麼造出來的。

## 為什麼是 DeepSeek：被卡住算力，逼出一套省算力的發明

現在用五變數這把尺，層層回答「為什麼是這家」。這一章的主因有點特別：**它從一個劣勢開始**。

### 起點：出口管制把算力掐住

✅ 美國的出口管制，限制了最頂規的 NVIDIA 晶片賣進中國。DeepSeek 能拿到的，主要是**降規版的 H800**（互連頻寬被刻意閹割過的 H100）。⚠️ 它到底有幾千張卡是個祕密、數字未經證實，但方向很清楚：**它的算力，在「絕對火力」這一格上，是輸給 OpenAI、Google、Meta 那種等級的。**

關鍵的因果轉折在這裡：**當你不能用蠻力解決問題，你就被逼著用聰明解決問題。** xAI 的故事是「拼命堆算力換速度」（ch10）；DeepSeek 的故事剛好是它的鏡像——「算力堆不上去，只好把每一張卡的效率擰到極致」。這就是為什麼這一章的「算力·基礎設施」這一格，雖然在**火力**上是劣勢，卻在**效率文化**上是主因：是這個劣勢，**催生**了下面那一整套發明。

### MLA：他們的 cache library，砍掉 attention 最貴的記憶體

✅ **MLA（Multi-head Latent Attention，多頭潛在注意力）** 是 DeepSeek 的招牌，從 **DeepSeek-V2 開始**用。它解決的，正是你那本 AI Infra 書算到底的痛點：**KV cache。**

快速喚醒一下（你早就懂概念）：推論時，attention 要把過去每個 token 的 Key 與 Value 都存下來重複使用，這就是 KV cache。序列越長、batch 越大，這塊記憶體脹得越兇——它常常是長上下文服務的第一個瓶頸。業界最常見的省法是 **GQA（Grouped-Query Attention）**：讓多個 query head 共用同一組 KV，少存幾份。

MLA 走得更狠，而且**關鍵字不是「快取」，是「壓縮」**——而且是**有損的低秩壓縮**。它不是「少存幾份 KV」，而是在寫進快取**之前**，用一個訓練出來的低秩投影矩陣（low-rank projection），把每個 token 的 K、V 聯合**降維**壓成一個很小的「潛在向量（latent vector）」；推論真的要算 attention 時，再用另一個投影即時把它**展開**回近似的 K、V。概念上近似 **gzip／壓縮傳輸**：存的時候壓小、用的時候解開，本質是**用一點額外算力（投影的乘法），換回大量的記憶體空間與頻寬**。注意「低秩投影」＝**有損的降維**，不是無損地把原值原封不動存起來——這正是它和「單純快取」最大的分野：快取是把原物件照原樣放著，MLA 是把原物件**壓成一個更小、會丟掉一點資訊的近似表示**。效果就是：**KV cache 的記憶體佔用大幅縮小**，同樣的卡能塞下更長的序列、更大的 batch。而且 DeepSeek 的選擇很值得玩味——他們捨棄了業界預設的 GQA，押了自研的 MLA，因為在他們的實驗裡 MLA 不只更省記憶體，建模品質還比 GQA 好。

橋接到你（但要橋對地方）：**它和你那套多層 cache library 同構的，是「省記憶體與頻寬」這個目標**——你的 cache 把「昂貴的 RDS 查詢」換成「便宜的快取命中」，MLA 把「龐大的 KV cache」換成「小小的潛在向量」，兩者都是「用一點算力，換回記憶體空間與頻寬」。但**機制不完全是同一回事**：你的快取是把查詢結果**原樣**留著（命中就直接回，無損）；MLA 更像在你快取那層再加一道**有損壓縮**（gzip 那種），存進去前先壓、取出來時再解。把它記成「壓 KV cache 的記憶體足跡」（你那本 AI Infra 書算到底的東西）是對的，但別把它讀成「就是多放一層 cache」——它是**用低秩降維把那塊記憶體本身做小**。**而省下來的記憶體，就是省下來的算力與成本**；對一個算力被卡死的團隊來說，這一塊省下來的，是能不能訓得起、服務得起的差別。

### MoE：671B 的容量，37B 的帳單

✅ V3 是一個 **MoE（Mixture-of-Experts，混合專家）** 模型：總參數 671B，但每個 token 只激活 37B（256 個專家裡只路由到少數幾個）。

直覺很簡單：與其讓每一個 token 都跑過整個 671B 的龐大網路（貴），不如讓一個路由器決定「這個 token 該交給哪幾個專家處理」，只激活那一小撮。結果是**「擁有 671B 的知識容量，卻只付 37B 的計算帳單」**——又是一招「把昂貴換便宜」。MoE 不是 DeepSeek 發明的，Mistral、Qwen、Llama 4 也都愛用（見 ch09、ch12），但 DeepSeek 把它跟 MLA、MTP、FP8 疊在一起，效率是**一整疊招式的複利**，不是單一招。

⚠️ 這裡要插一個誠實的但書：MoE 的省，是「計算量」的省，不是「記憶體」的省——那 671B 的權重你還是得整個放進記憶體待命（隨時可能被路由到）。所以 MoE 省的是 FLOPs，不省 VRAM；真正配著它去省記憶體的，是 MLA 那一招。兩招分工，缺一不可。

### GRPO：砍掉那個又貴又麻煩的 reward model

✅ **GRPO（Group Relative Policy Optimization，群組相對策略最佳化）** 出自 DeepSeek 的 **DeepSeekMath**，後來成為 V3／R1 的引擎，也是 R1 那個推理模型背後的關鍵。它解決的是強化學習裡一個很實際的工程痛點。

先給最低限度的背景（這是訓練側，你接觸少，慢慢帶）：要讓模型「學會推理」，一種主流做法是用 RL——讓模型自己產生一堆答案，給「好答案」獎勵、「壞答案」懲罰，慢慢往好的方向調。傳統做法（如 PPO）需要額外訓練並一直掛著一個 **critic／reward model**：一個專門「打分」的模型，估計每個狀態「值多少分」。問題是這個打分模型**本身就跟主模型一樣大、一樣貴**——你等於要同時養兩個大模型，記憶體、算力都翻倍，工程也更複雜。

GRPO 的做法是：✅ **把那個 critic 整個拿掉。** 它改成這樣——對同一個問題，讓模型一次生成**一群**（a group of）答案，然後**讓這群答案互相比較**：在這群裡相對好的，往上推；相對差的，往下壓。基準不是一個「打分模型」算出來的，而是**這群樣本自己的平均**。一句話：**不靠一個外部裁判，改靠「同儕之間的相對排名」來給訊號。**

橋接到你：**這就是你那次「砍掉一個中間服務、簡化 pipeline」的決定。** 那個 reward model 就像 pipeline 裡一個額外的、自己也會壞、又吃資源的元件；GRPO 把它拿掉，用更直接的方式（組內相對比較）達到同樣的目的。

**為什麼這能省下將近一半的 VRAM？** 關鍵在那個 critic 有多大。傳統 RL（如 PPO）的打分模型（critic）**跟主模型（actor）一樣大**——訓練時，這兩個大模型的**權重、梯度、最佳化器狀態（optimizer state）**都要同時佔著 GPU 記憶體。GRPO 砍掉 critic，**≈ 直接釋放掉訓練期將近一半的 VRAM**：同一張卡，原本要分給兩個大模型，現在只養一個，省下來的空間能塞更大的 batch、跑更穩。**這在算力被卡住（只能用降規 H800）時特別關鍵**——當你每一格 VRAM 都得錙銖必較，砍掉一個跟主模型一樣大的元件，不是「擠一點邊角」級的省，是「能不能在這批卡上把 RL 跑起來」級的省。

結果是：**記憶體省一半量級、工程更簡單、少一條故障路徑。** 對一個算力受限的團隊，這不只是「優雅」，是「**訓得起**」的關鍵。而且這一招特別適合**可驗證**的任務（數學對不對、程式碼能不能編譯跑過）——因為那種任務的「好壞」有客觀標準，不太需要一個學出來的裁判去猜。這就是為什麼 R1 這類推理模型，能用這套又便宜又開放地練出來（完整的對齊方法譜系見 ch15）。

### 把四招串成一條因果鏈

這就是本章要講透的那一條鏈——注意它的**起點是劣勢**：

```
出口管制 → 只能用降規 H800（算力火力是劣勢）
      |
      v  不能用蠻力，只能用聰明
被逼出一整套「省算力」的效率發明
      |
      +-- MLA   : 壓 KV cache（省記憶體／頻寬，接你 cache library）
      +-- MoE   : 稀疏激活（省計算 FLOPs，671B 容量 37B 帳單）
      +-- GRPO  : 砍掉 reward model（省一半 RL 記憶體，接你砍中間服務）
      +-- MTP/FP8 : 更密的訓練訊號／更省的數值精度
      |
      v  一整疊招式的複利
用遠低於共識的算力，訓出接近前沿、且開放權重的模型
      |
      v  數字傳開（但被讀錯口徑）+ 市場恐慌
撼動「訓練前沿模型一定很貴」的成本敘事
```

注意這條鏈最反直覺的地方：**主因不是「他們算力多」，恰恰是「他們算力被卡住」。** 限制不是只帶來損害，它**重新分配**了一家公司必須擅長什麼——當火力這條路被堵死，效率這條路就被逼成了核心競爭力。

### (b) 影子

同一組讓 DeepSeek 強的選擇，也讓它暴露在幾個地方：

- **成本數字被過度簡化的病毒式傳播，是把雙面刃。** 「560 萬」既替 DeepSeek 賺到了全球聲量，也讓它被綁在一個**它自己都沒這樣宣稱過的神話**上。當外界用「560 萬就能做前沿模型」這個錯框架去檢驗它，任何「其實沒那麼便宜」的揭露，都會被當成「DeepSeek 騙人」——但真相是窄口徑數字被外界讀錯了。**靠一個容易被誤讀的數字成名，代價是你的真實成就也跟著被那個誤讀綁架。** 這就是你做系統最怕的「對外承諾被誇大解讀」——一旦傳開，要收回去比講清楚難十倍。
- **出口管制是一道長期的天花板。** 效率發明能把劣勢補到「同台」，但補不到「無限」。如果前沿持續往「更大規模、更多算力」推進，而 DeepSeek 能拿到的頂規算力長期被限制，那麼**效率買來的，是縮小差距，不是抹平差距**。今天靠 MLA／GRPO 追上來，不保證下一個需要絕對火力的範式（例如某種超大規模訓練）裡還能追上。劣勢逼出的優勢，本身也是被劣勢框住的。
- **開放權重的商業模式問題還沒解。** 把權重開放出去，賺到了驗證、聲量、生態採用——但**錢從哪裡來？** 你把最值錢的東西（模型權重）免費送出，別人可以直接拿去自架、不付你一毛 API 費。這跟你「把內部 cache library 開源給全公司」的影子一模一樣：採用率變護城河，但**採用率本身不付帳單**。低價 API ＋ 開放權重能換到影響力，能不能換到可持續的營收，截至 2026-06 仍是個開放問題（開源 vs 閉源的商業模式對立，見 ch13）。

### (c) 何時不成立

這個「效率撼動成本」的強項，會在什麼情況被打臉？

- **被任務打臉**：GRPO ＋ 可驗證獎勵這套，特別吃**有客觀對錯**的任務（數學、程式碼）。一旦進到**難驗證**的領域——創意寫作、開放式對話的細膩度、沒有標準答案的判斷——「靠組內相對排名」這套省下來的優勢就會縮小，因為那裡本來就需要更多人類偏好的訊號（reward model 沒那麼容易被砍掉）。效率招式不是萬靈藥，它在「能不能客觀打分」這條線上有邊界。
- **被別家打臉**：效率招式**最容易被抄**。MLA、GRPO 一發論文、一開權重，全世界隔週就能學。這正是全書 ch01 的命題——**架構與訓練法不是護城河**。今天 DeepSeek 靠這幾招領先效率敘事，但這幾招正在快速擴散成業界常識（ch20 會講「為什麼差異會收斂」）。真正難抄的是那層「願意在受限下把效率擰乾、還開放驗證」的研究文化，但連這個，別家也在補課。
- **被時間打臉**：本章所有版本號都是 **2026-06 的快照**。V4 的 1M 上下文、75% 永久降價那些（🟡）是次級來源、變動極快的；出口管制的鬆緊、下一代降規卡能拿到什麼，全都會變。三個月後（見 maintenance 重掃）這些數字十之八九要更新。記住：穩定的是「**算力被卡住 → 逼出效率發明 → 撼動成本共識**」這個**成因結構**，以及「560 萬是窄口徑、別當總成本」這個**讀法**；會過期的是任何一個具體的版本、價格與名次。

## 紙上推演

### 推演題1 — 找破綻：「DeepSeek 只花 560 萬美金就做出前沿模型」 **[15 分鐘]** ★★★

這是全書的招牌題。有人在群組裡丟出這句話。請你用 90 秒，不否定 DeepSeek 的真實成就、但精準指出這句話的破綻在哪。要點名「這個數字回答的是哪個問題、它沒回答哪個問題」。

#### 推演解答

完整的拆法分三刀，缺一刀都不算過關：

1. **口徑刀（這數字算了什麼）**：560 萬出自 V3 技術報告，是**「最後一次成功預訓練 run 的 GPU 算力帳單」**（約 278.8 萬 H800 小時 × 假設租金），不是「打造這個模型的總成本」。
2. **漏算刀（它沒算什麼）**：研發、失敗的 run、團隊薪水、那幾千張 H800 的硬體攤提（這才是大頭）、前代（V2／DeepSeekMath）的投資——全都不在裡面。批評者估的全包成本到 ⚠️ 13 億～16 億美金等級，跟 560 萬差兩三百倍。**兩個數字都對，回答的是不同問題。**
3. **張冠李戴刀（媒體還錯在哪）**：引爆股災的是 R1（2025-01-20），但 560 萬是 V3 的預訓練數字，被很多報導安到了 R1 頭上。一個窄口徑數字 ＋ 一個錯敘事 ＝ 神話。

**但收尾必須轉回來**：破除迷思 ≠ 否定成就。DeepSeek 確實用 MLA／MoE／GRPO 把「達到某水準所需的算力」實打實壓低了，還開放權重讓人驗證——市場恐慌**沒有完全錯**，它只是把「效率很重要」誇大成了「scaling 已死」。證據：NVIDIA 股價幾個月內就漲回來了。

常見誤解：很多人「找破綻」找過頭，變成「所以 DeepSeek 是吹牛、其實一點都不便宜」——這也錯了。正確的姿態是**校準**，不是**翻案**：數字被讀錯口徑，但底層的效率成就是真的。能同時講出「破綻」與「真內核」，才算真的讀懂這一題。

### 推演題2 — 翻譯：把 MLA 與 GRPO 對應回你的系統 **[10 分鐘]** ★★

不准用 ML 術語，把 MLA 和 GRPO 各自對應回你維運過的一個**具體**決定，並說出「省的是什麼資源」。

#### 推演解答

- **MLA ≈ 你的多層 cache library**：把昂貴龐大的東西（KV cache／一次次 RDS 查詢）換成小而便宜的東西（低維潛在向量／快取命中）。**省的是記憶體與頻寬**——而記憶體與頻寬，對算力受限的團隊就是「服務得起多少、付得起多少帳單」。
- **GRPO ≈ 你那次砍掉一個中間服務、簡化 pipeline**：把一個額外的、自己也會壞、又吃資源的元件（reward model／那個中間服務）整個拿掉，用更直接的方式（組內相對比較）達到同樣目的。**省的是算力與工程複雜度，還少一條故障路徑**。

關鍵反問（檢驗你真懂）：這兩招省的是**同一種**資源嗎？不是。MLA 省的主要是**推論期的記憶體**；GRPO 省的主要是**訓練期的算力與工程**。MoE 又不一樣，省的是**計算 FLOPs、但不省 VRAM**。能分清「哪一招省哪一種資源」，你就不會把「DeepSeek 便宜」當成一句模糊的口號，而是看得到它是**一整疊各司其職的省法疊起來的複利**。

### 推演題3 — 反事實：如果 DeepSeek 拿得到無限的頂規 GPU **[12 分鐘]** ★★★

假設出口管制從沒發生，DeepSeek 跟 OpenAI 一樣想買多少頂規 H100／B200 都行。它還會發明 MLA、GRPO 這套效率招式嗎？這個 thought experiment 想逼你看清什麼？

#### 推演解答

很可能**不會、至少不會這麼拚**。這正是本章最反直覺的成因：**那套效率發明，是被劣勢逼出來的，不是在資源充足下自然長出來的。** 當你可以無腦堆卡，「把每張卡擰乾」的邊際動機就低得多——你會像很多資源充足的大廠一樣，先用蠻力把東西做出來再說。

這個反事實要逼你看清的是全書一個核心觀念：**約束不只是損害，它會重新分配一家公司「必須擅長什麼」。** 你那次 RDS CPU 降 40%，也是因為**不能無腦加機器**才被逼著把瓶頸看透——如果預算無限，你大概會先加台機器，那個 40% 的洞察就不會被逼出來。限制是創新的母親，這句話在工程現場與在國家級實驗室是同一個機制。

但要小心別把這個浪漫化過頭（常見誤解）：約束**催生**了 DeepSeek 的效率優勢，但約束**同時也是它的天花板**（見影子 (b)）。同一個出口管制，逼出了今天的招式，也限制了明天的上限。約束的兩面性——既是創新的母親，也是規模的牢籠——要一起講，才完整。

## 自我檢核

口頭自答，講不清楚就是還沒懂：

1. 「560 萬美金」這個數字，**回答的是哪個問題、沒回答哪個問題**？為什麼說它「不是假的，但被讀錯了」？
2. 破除「560 萬」的迷思時，怎麼做到**不否定** DeepSeek 的真實效率成就？「校準」和「翻案」差在哪？
3. 為什麼 MLA 對應你的 cache library、GRPO 對應你「砍掉中間服務」？這兩招各自省的是**哪一種**資源（記憶體？算力？工程？）？
4. MoE 省的是計算還是記憶體？為什麼說「MoE 不省 VRAM，要配 MLA 才省記憶體」？兩招怎麼分工？
5. 為什麼說 DeepSeek 的「算力·基礎設施」這一格，是用**劣勢**當主因？如果它拿得到無限頂規 GPU，還會這麼拚效率嗎？
6. GRPO 砍掉的「reward model」是什麼、為什麼貴？「靠一群樣本互相比較」怎麼取代它？這招為什麼特別適合數學／程式碼這種任務？
7. 2025-01-27 NVIDIA 跌 17%、市值蒸發近 6,000 億，幾個月後又漲回來——這一來一回留下的**正確教訓**是哪一句？為什麼不是「scaling 已死」？
8. DeepSeek 的三個影子（成本數字被誤讀／出口管制天花板／開放權重商業模式）裡，哪一個跟你「把內部 library 開源給全公司」的影子最像？為什麼「採用率本身不付帳單」？
