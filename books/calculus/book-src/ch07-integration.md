## 從你已知的出發

你每個月都在為一個積分付錢。

AWS Fargate 的帳單單位是 vCPU-秒與 GB-秒（us-east-1 的 Linux 牌價約每 vCPU-秒 $0.000011，2026-06 查價），EC2 的世界則慣講 vCPU-hours、GB-hours。如果你的服務一整個月固定跑 4 vCPU，帳很好算：單價 × 4 × 秒數，一個乘法。但你開了 autoscaling：尖峰 12 vCPU、深夜 2 vCPU、部署時抖來抖去——「用量 × 時間」這個乘法當場失效，因為用量不是一個數，是一條隨時間起伏的曲線 u(t)。AWS 的計費器實際做的事：把整個月切成一秒一秒，每一秒記下當下的 vCPU 數，全部加總。把這句話畫出來，就是「u(t) 曲線下的面積」。

你也每天在用同一件事的另一面：Grafana 上盯著 QPS 曲線，想知道這一小時總共進來多少請求——你瞄的就是曲線下的面積；CloudWatch 把這個統計量直接叫 Sum。寫成本章的語言：

```text
總請求數 = ∫ QPS dt           ← 速率曲線下的面積 = 總量
本月帳單 = ∫ 單價 × u(t) dt
```

「速率會變的時候，總量 = 速率曲線下的面積」——這個直覺你早就有了。本章做三件事：給它一個正式的名字（積分，integral）、一台通用的機器（黎曼和）、和一個經得起盤問的地基（極限）。

## 曲線下的面積為什麼難

直邊的世界裡，面積是個已經解決的問題。長方形 = 底 × 高；三角形 = 對半切的長方形；任何多邊形都能切成三角形逐塊加總。兩千年的工具箱就這麼多——而且只要邊一彎，全套報廢：曲線下方的區域沒有「底 × 高」可言，用三角形去拼，永遠留著貼不平的縫。

ch02 看過人類的第一次突圍：阿基米德對拋物線弓形的 4/3。但那是手工藝——1/8 性質、平帳尾巴、雙重歸謬，每個零件都是為拋物線特製的，換一條曲線就全部重來。卡住一千八百年的不是答案，是通法。

現在重新拿起同一道題：y = x² 從 x = 0 到 x = 1，曲線下的面積是多少。我們甚至已經知道答案——ch02 末尾用弓形換算出 1/3。這次要的不是答案，是一條**不挑曲線的路**：一套對任何 f(x) 都按同樣步驟運轉的機器。差別在於，這次我們有極限。

## 黎曼和：切、近似、加總、取極限

機器叫黎曼和（Riemann sum），四個動作：

1. **切**（slice）：把 [0, 1] 切成 n 條等寬的直條，每條寬 1/n。
2. **近似**（approximate）：每條的彎頂用平頂取代——變成矩形，高度取這條範圍內某一點的 f 值。最老實的兩個選擇：左端點，或右端點。
3. **加總**（sum）：n 個矩形面積相加。這是有限和，國中數學。
4. **取極限**（limit）：n → ∞，看這串數逼近哪裡。

前三步全是初等操作，第四步是 ch03 的極限——抽掉它，機器只剩一堆永遠在近似的有限和。

n = 4 完整算一遍。分割點 0、1/4、2/4、3/4、1，每條寬 1/4：

```text
左和 L₄（每條的高取左端點的 f）：
  L₄ = (1/4)·[f(0) + f(1/4) + f(2/4) + f(3/4)]
     = (1/4)·(0 + 1/16 + 4/16 + 9/16)
     = (1/4)·(14/16)
     = 14/64 = 0.21875

右和 R₄（每條的高取右端點的 f）：
  R₄ = (1/4)·[f(1/4) + f(2/4) + f(3/4) + f(1)]
     = (1/4)·(1/16 + 4/16 + 9/16 + 16/16)
     = (1/4)·(30/16)
     = 30/64 = 0.46875
```

右和的長相（示意圖，比例不嚴格；`*` 是曲線 y = x²，每條矩形的高取右端點）：

```
  y
  1 ┤                      ┌──────*
    │                      │    **│
    │                      │   *  │
    │                      │ *    │
    │               ┌──────*      │
    │               │    * │      │
    │               │   *  │      │
    │        ┌──────* *    │      │
    │        │    * │      │      │
    │ ┌──────*  *   │      │      │
    └─┴──────┴──────┴──────┴──────┴── x
      0     1/4    2/4    3/4     1
```

兩個數都不是答案，但它們的**位置**極有價值。x² 在 [0, 1] 上遞增，所以每一條裡，左端點是全條最矮處、右端點是全條最高處：左矩形整個藏在曲線下面，右矩形把曲線整個蓋住。逐條取界、逐條相加：

```text
L₄ ≤ 真實面積 ≤ R₄
0.21875 ≤ 真實面積 ≤ 0.46875
```

認出來了嗎？內接給下界、外切給上界——ch02 的夾擠，換上現代制服。更妙的是這對界的寬度免費可知：右和與左和逐條相減，中間項全部對銷，只剩頭尾——

```text
Rₙ − Lₙ = (1/n)·[f(1) − f(0)] = 1/n    ← 不用算出任何一個和，就知道夾多緊
```

切 10 條，區間寬 0.1；切 100 條，寬 0.01。「誤差要多小有多小，而且事先講得出要切幾條」——這是 ch02 那條引理的現代版，也是你壓測報告裡上下界承諾的數學版。

實際收斂給你看。n = 10：分割點 0, 0.1, …, 1，平方數加總（0+1+4+…+81 = 285、1+4+…+100 = 385）：

```text
L₁₀ = (1/10)·(285/100) = 0.285
R₁₀ = (1/10)·(385/100) = 0.385
```

| n | 左和 Lₙ | 右和 Rₙ | 區間寬 1/n |
|---|---|---|---|
| 4 | 0.21875 | 0.46875 | 0.25 |
| 10 | 0.285 | 0.385 | 0.1 |
| 100 | 0.32835 | 0.33835 | 0.01 |

1/3 = 0.33333… 每一列都乖乖躺在區間裡。數值證據齊了——但 ch04 教過：表再長也只是證據，不是證明。要結案，得走代數那條路。

## 代數路：一條公式管所有 n

n 不代數字，直接展開右和。需要一塊古董積木：平方和公式 Σ(k=1→n) k² = n(n+1)(2n+1)/6（古代已知的恆等式，可用歸納法驗證，本書不證）。

```text
Rₙ = Σ(k=1→n) f(k/n)·(1/n)
   = Σ(k=1→n) (k/n)²·(1/n)
   = (1/n³)·Σ(k=1→n) k²           ← 1/n³ 是公因數，提出去
   = (1/n³)·n(n+1)(2n+1)/6        ← 套平方和公式
   = (2n² + 3n + 1)/(6n²)         ← 約掉一個 n、展開分子
   = 1/3 + 1/(2n) + 1/(6n²)       ← 逐項除以 6n²
```

先對帳：n = 4 代回去，1/3 + 1/8 + 1/96 = 32/96 + 12/96 + 1/96 = 45/96 = 0.46875 ✓，與手算的 R₄ 一致。這條公式把表格的祕密全部亮出來：右和 = 答案 1/3，加上一塊 1/(2n) 的一階誤差，再加一小撮 1/(6n²)。取極限：

```text
lim(n→∞) Rₙ = lim(n→∞) [1/3 + 1/(2n) + 1/(6n²)] = 1/3
左和同理：Lₙ = Rₙ − 1/n = 1/3 − 1/(2n) + 1/(6n²) → 1/3
```

上下界收斂到同一點，夾擠閉合：**∫₀¹ x² dx = 1/3**。不是約等於，是等於——而且這次不需要雙重歸謬，「取極限」一個標準動作收尾。（這個動作憑什麼保證「夾到的那個點」真的存在——實數完備性——ch14 算總帳。）

記住 1/(2n) + 1/(6n²) 這份誤差解剖，梯形法那節要用它變一個漂亮的魔術。

順帶把名字記對。這台機器冠的是黎曼（Riemann）的姓，但「把積分定義成和的極限」在他之前就有：柯西（Cauchy）1823 年的講義已對連續函數這樣定義積分（2026-06 查證）。黎曼 1854 年的教授資格論文把定義推到一般——任意分割、任意取樣點（本書用等寬加端點的簡化版，夠用）——並追問「一個函數要多醜才會不可積」。耐人尋味的是那篇論文的主題其實是傅立葉級數（見 ch11）：是「任意波形拆成圓」引爆的收斂爭議，逼著數學家把積分本身打磨嚴格（同一篇論文裡還藏著 ch12 的重排定理）。積分的嚴格定義是被一場危機催生的——Part V 的伏筆，先按下。

## 同一題三算：第二算對帳

ch02 結尾立了約：「y = x² 從 0 到 1 的曲線下面積 = 1/3」這道題全書要算三次。第二算完成，正式對帳：

| | 第一算：窮盡法（ch02） | 第二算：黎曼和（本章） |
|---|---|---|
| 答案 | 1/3（弓形 4/3 換算而來） | 1/3 |
| 下界／上界 | 內接三角形逐層加密 | 左和／右和 |
| 誤差節奏 | 每層缺口 × 1/4 | 區間寬 = 1/n |
| 收尾 | 雙重歸謬，手工 | 取極限，標準零件 |
| 換一條曲線？ | 全部重新發明 | 同一條流水線 |

靈魂是同一個：用上下兩排算得出來的東西夾住算不動的東西，再把鉗口收緊。差別在最後一哩：阿基米德缺「極限」這個詞，每題都得用反證法手工封口；我們把封口做成了標準零件。ch02 說窮盡法不能量產——黎曼和把框架量產了：換函數、換區間，切、近似、加總、取極限，四步不變。

但誠實盤點：量產的只有**框架**。第 3 步的加總要收得攏，靠的是 Σk² 恰好有封閉公式。算 ∫₀¹ x³ dx？要 Σk³ 公式（還好，有）。sin 的面積？1/x 的呢？沒有現成的 Σ 公式讓你提出 1/n³ 這種好事。每換一個 f，加總那一步就是一場新的肉搏。框架通用、苦工逐題——記住這份累，它在全書劇情裡有戲份。

## ∫：記號的解剖

ch04 欠了一個故事，這裡還。1675 年 10 月 29 日，萊布尼茲（Leibniz）在手稿裡第一次寫下 ∫——拉丁文 summa（總和）的字首 s，拉長了寫；同年 11 月 11 日，∫ 與 dx 第一次連用；印上出版品則要等到 1686 年。「integral」這個詞反而不是他的：1690 年雅各布·白努利（Jacob Bernoulli）首次把它印出來，萊布尼茲自己偏愛 calculus summatorius（加總術），最後兩邊各讓一步——名字用白努利的、記號用萊布尼茲的（2026-06 查證）。中文的「積分」則出生於 1859 年：李善蘭與偉烈亞力合譯《代微積拾級》時所創。

把記號逐件拆開：

```text
∫₀¹ x² dx

∫    拉長的 S：「加總」——無限多片，全部收進來
₀ ¹  從哪裡加到哪裡（x 的下限與上限）
x²   每一片的高
dx   每一片無限細的寬（d 是 difference：x 的一小段變化）
```

整串唸出來：「x 從 0 走到 1，把高 x²、寬 dx 的無限細長條全部加總。」（界限不是數字時寫 ∫[a,b] f(x) dx，全書同此。）

這個記號是思想的化石：萊布尼茲真心把積分想成「無限多個無限細長條的和」，記號照著直覺刻。作為思考工具，這個圖像好用到三百年沒人捨得丟——你想「每一刻的 QPS 乘上一小段 dt，全部加起來」，方向、單位、形狀全部自動對齊。但嚴格身分要說清楚：在本書的定義裡，∫₀¹ x² dx 是「黎曼和的極限」的**整體記號**，dx 不是一個獨立存在的小數——與 ch04 對 dy/dx 的告誡同款：圖像合法，移項要有定理背書。無窮小本身後來真的拿到了嚴格的身分證（Robinson，見 ch14），但那是另一套地基。一句話，存檔備查。

## 積分是累積，面積只是一張臉

到目前為止都在說「面積」，但這個詞其實太小了。看單位：每個矩形是「高 × 寬」，高若是速度（m/s）、寬是時間（s），乘出來是距離（m）；高是 QPS、寬是秒，乘出來是請求數；高是 GB、寬是小時，乘出來是 GB-hours——帳單。加總與取極限都不改變單位，所以：

```text
距離　　 = ∫ 速度 dt          ← 里程表做的事
總請求數 = ∫ QPS dt           ← CloudWatch 的 Sum
本月帳單 = ∫ 單價·用量 dt     ← 計費器做的事
進水總量 = ∫ 流速 dt          ← 水表做的事
```

同一張圖、同一台四步機器，換的只是軸上的標籤。**積分的本體是累積（accumulation）：只要某個速率在變，它累積出的總量就是一個積分。**「曲線下的面積」只是這件事在 xy 平面上的一張臉——好畫、好教，但別把臉當本體。臉與本體的差距在 f 變負時最明顯：速度為負（倒車），∫ 速度 dt 老老實實扣里程，「負的累積」天經地義；只盯著「面積」想的人卻會卡住——面積怎麼會是負的？（教科書的規定是：x 軸下方的部分帶負號計入。用累積想，根本不需要這條規定。）

回頭看開場的 Fargate 帳單：計費器把月份切成一秒一片、每片記用量、全部加總——它字面上就是在算一個 n 極大的黎曼和，只是 AWS 替你選好了切片寬度。你不是「彷彿」在為積分付錢，你就是在為積分付錢。

## 梯形法：同樣的預算，多換好幾位數

把「誤差解剖」的存貨變現。實務上沒有人真的取極限——機器只能算有限的 n，於是問題變成工程問題：固定計算預算，怎麼把誤差壓到最低？

矩形的毛病是**系統性偏差**：f 遞增時，右矩形每一條都多收一塊（蓋過曲線的角），左矩形每一條都少收一塊。直覺版的記帳（嚴格版是數值分析的標準結果，本書不證）：每條多收的那一角近似一個三角形，底 1/n、高約「斜率 × 1/n」，一條虧 ~1/n²，n 條堆起來 ~1/n。這正是右和誤差 1/(2n) 的出身——偏差不會互相抵銷，只會堆積。

梯形法（trapezoidal rule）的點子小到不好意思：別用平頂，用斜頂——每條的頂改成弦（連接兩端點的直線），矩形變梯形，貼著曲線走。等價的說法更漂亮：**梯形和就是左和與右和的平均**。看它對誤差做了什麼：

```text
Rₙ = 1/3 + 1/(2n) + 1/(6n²)        ← 右和：多收 1/(2n)
Lₙ = 1/3 − 1/(2n) + 1/(6n²)        ← 左和：少收 1/(2n)
Tₙ = (Lₙ + Rₙ)/2 = 1/3 + 1/(6n²)   ← 平均：±1/(2n) 整項對銷
```

一階誤差是「方向相反、大小相等」的一對，平均一下互相註銷，活下來的只有 1/(6n²)——曲線與弦之間那一彎月牙，f 的彎曲量。誤差從 ~1/n 掉到 ~1/n²：n 每乘 10，矩形多得一位精度，梯形多得兩位。n = 100 時矩形誤差約 5×10⁻³、梯形 1.7×10⁻⁵——同樣一百次函數求值，白拿兩位數。一句話的直覺：矩形犯的是「看不見斜率」的錯，梯形把斜率吃進去了，只剩「看不見彎曲」的錯（再往前走，把彎曲也吃進去，就是 Simpson 法與整個求積家族——點到為止，本書不展開）。

你在數值程式庫、模擬器、監控後端裡遇到的「數值積分」，全是這條路上的東西：黎曼和是原型機，後面是一整場「同樣預算換更多位數」的軍備競賽。本章的 Python 小實驗會讓你親眼看到 1/n 與 1/n² 的差距。

## 微分與積分：至此是兩個故事

盤點你現在的庫存。Part II 給了你一台機器：微分——局部的，問「此刻多陡」，函數進、斜率出。本章給了你第二台：積分——全域的，問「累積了多少」，函數進、總量出。兩台機器都蓋在極限這塊地基上，但除此之外**互不相識**：一個在一點上越貼越近，一個在整段上越切越細；一個馴服了瞬間，一個馴服了無限加總。教科書把它們印進同一本書——到本章為止，純屬巧合。

同時記住這份帳：為了 ∫₀¹ x² dx 一個答案，我們動用了一整章——四步機器、Σk² 公式、代數展開、取極限。換 x³ 就得重來一輪，換 sin 直接卡關。框架通用、苦工逐題，黎曼和只解決了阿基米德困境的一半。

如果現在有人告訴你：這一整章的苦工可以縮成一行字，而且鑰匙不在積分這邊——藏在 ch04 你早就算好的那張導數表裡。你大概不會信：斜率關面積什麼事？很好，帶著這份不信去 ch08。那是全書的高潮，而你剛剛付清了入場費。

## 直覺的陷阱

| 錯誤直覺 | 它把你帶去哪條溝 | 自我察覺的徵兆 |
|---|---|---|
| 「積分＝面積」 | 把臉當本體：f 變負就講不下去，∫速度 dt 的倒車扣里程變成謎 | 你需要硬背「x 軸下方算負的」，而不是覺得它理所當然 |
| 「無限多條、每條寬 0，加起來是 0」（或 ∞ × 0 = 隨便） | 0 × ∞ 是未定式——ch04 的 0/0 同門師弟：資訊在趨近的路徑裡，不在形狀裡 | 你說出「最後每條的寬度變成 0」 |
| 「n 夠大，黎曼和就『是』積分了」 | 把極限當成過程的最後一項（ch03 的老病第三次發作） | 你以為 n = 10⁶ 算出來的數「就是」答案，講不出它附帶的保證區間 |
| 「dx 是個很小的數，可以拆下來單獨運算」 | 單層直覺無害，亂移項翻車——∫ f(x) dx 是一整個極限的名字 | 你移項 dx 時說不出哪個定理在背書（ch04 的 dy/dx 同款） |
| 「左和永遠是下界、右和永遠是上界」 | 這靠的是 f 遞增；遞減時整個反向，起伏的 f 兩個都不是界 | 你對 [0, 2π] 上的 sin 還在講「左和 ≤ 真值」 |
| 「切多一點就好，方法無所謂」 | 1/n 與 1/n² 的差距是數量級等級：同預算差好幾位精度 | 你的 10⁶ 條矩形輸給別人 10³ 條梯形，而你不知道為什麼 |

## 紙上推演

**題 1 [15 分鐘] ★** — 對 ∫₀¹ x dx 重演本章機器：n = 4，手算左和與右和（分數與小數都要），對照三角形面積 1/2 檢查夾擠；再把左右和平均，看看你得到什麼、解釋為什麼。

**題 2 [10 分鐘] ★★（口頭題）** — 對遞增的 f 解釋：(a) 為什麼每一條裡左矩形必然偏矮、右矩形必然偏高，因此 Lₙ ≤ 真值 ≤ Rₙ；(b) 為什麼 Rₙ − Lₙ = (f(b) − f(a))·(b − a)/n，不用算出任何一個和就知道；(c) 這給了你哪一種「事先可承諾」的誤差保證——用你寫壓測報告的語言講。

**題 3 [15 分鐘] ★★** — 把「本月帳單」完整翻譯成積分語言：某雲端資源的用量 u(t)（GB）隨 autoscaling 起伏，單價 p(t)（USD／GB-小時）隨 spot 市場浮動。(a) 寫出帳單的積分式並做單位檢查；(b) 說明為什麼「平均用量 × 平均單價 × 時數」這個偷懶算法一般是錯的；(c) 指出計費器實際算的東西在本章叫什麼名字。

### 推演解答

**題 1**　分割點 0、1/4、2/4、3/4、1，f(x) = x：

```text
L₄ = (1/4)·(0 + 1/4 + 2/4 + 3/4) = (1/4)·(6/4)  = 6/16  = 0.375
R₄ = (1/4)·(1/4 + 2/4 + 3/4 + 1) = (1/4)·(10/4) = 10/16 = 0.625
夾擠：0.375 ≤ 1/2 ≤ 0.625 ✓；區間寬 0.25 = (f(1) − f(0))/4 ✓
平均：(0.375 + 0.625)/2 = 0.5 ——一步命中精確值
```

平均命中不是巧合：左右和的平均就是梯形和，而 f(x) = x 是直線——梯形的斜頂與「曲線」完全重合，彎曲項為零，梯形法對一次函數**精確**（用本章誤差解剖的話說：±一階誤差對銷之後，剩下的項來自 f 的彎曲，直線沒有彎曲）。常見錯路：忘記每項都要乘寬度 1/4；左和手滑把 f(1) 也收進來變成五項。一般 n 的對照（自驗用）：Rₙ = (1/n²)·Σk = (n+1)/(2n) = 1/2 + 1/(2n)，Lₙ = 1/2 − 1/(2n)，兩側收斂到 1/2。

**題 2**　(a) 遞增的定義就是「右邊的點 f 值不小於左邊」：在第 k 條 [xₖ₋₁, xₖ] 裡，任何 x 都滿足 f(xₖ₋₁) ≤ f(x) ≤ f(xₖ)，所以這一條的真實面積被左、右矩形夾住；n 條不等式逐條相加，夾擠對總和成立。(b) 逐條相減：第 k 條貢獻 ((b−a)/n)·[f(xₖ) − f(xₖ₋₁)]，相鄰項望遠鏡式對銷，只剩頭尾 f(b) − f(a)。(c) 壓測語言：「我還沒開始算，就能承諾交付的上下界相距不超過 (f(b) − f(a))·(b − a)/n；你要求誤差小於多少，我就告訴你要切幾條。」——可以事先簽的 SLA，而不是「算完才知道準不準」。這正是 ch02 引理（誤差要多小有多小）的現代版，也是夾擠之所以是「保證」而不是「感覺」的原因。

**題 3**　(a) 帳單 = ∫[t₀, t₁] p(t)·u(t) dt。單位檢查：(USD/GB-小時) × GB × 小時 = USD ✓。(b) 「平均 × 平均」在 p 與 u 相關時失真：spot 單價在尖峰時段上漲，而你的用量恰好也在尖峰最大——貴的時刻配大用量、便宜的時刻配小用量，真實帳單高於「平均單價 × 平均用量 × 時數」（反向相關時則偏低）。逐時刻乘積的累積，不等於各自平均的乘積；只有 p、u 至少一個是常數時，乘法才退化成正確答案——長方形是積分的特例。(c) 計費器把月份切成固定寬度（如 1 秒）的片、逐片取樣相乘、加總——一個 n 已選定、不取極限的黎曼和（1 秒切片的誤差遠小於計費精度，所以停在有限 n 就夠了；但「它是哪個極限的近似」這件事，現在你說得出來了）。

### Python 小實驗

親眼看 1/n 與 1/n² 的差距（純標準函式庫）：

```python
# Riemann sum vs trapezoid rule for the integral of x^2 over [0, 1].
# Exact value: 1/3. Watch the right-sum error shrink like 1/(2n)
# while the trapezoid error shrinks like 1/(6n^2).
import math

def f(x):
    return x * x

EXACT = 1.0 / 3.0

print(f"{'n':>8}  {'right sum':>12}  {'error':>9}  {'trapezoid':>12}  {'error':>9}")
for n in (4, 10, 100, 10**4, 10**6):
    h = 1.0 / n
    right = math.fsum(f(k * h) for k in range(1, n + 1)) * h
    trap = right - (f(1.0) - f(0.0)) * h / 2   # = average of left and right sums
    print(f"{n:>8}  {right:12.8f}  {right - EXACT:9.1e}  {trap:12.8f}  {trap - EXACT:9.1e}")
```

預期輸出（CPython 3，雙精度；本書寫作時實跑複核）：

```text
       n     right sum      error     trapezoid      error
       4    0.46875000    1.4e-01    0.34375000    1.0e-02
      10    0.38500000    5.2e-02    0.33500000    1.7e-03
     100    0.33835000    5.0e-03    0.33335000    1.7e-05
   10000    0.33338334    5.0e-05    0.33333334    1.7e-09
 1000000    0.33333383    5.0e-07    0.33333333    1.7e-13
```

讀法：右和的誤差欄整齊得像教科書——n 每乘 10，誤差除 10（代數解剖早就預言了：誤差 ≈ 1/(2n)，所以 n = 10⁶ 時是 5×10⁻⁷）。梯形欄：n 每乘 10，誤差除 100（≈ 1/(6n²)，n = 10⁶ 時 1.7×10⁻¹³，已經貼到雙精度的地板附近）。同樣 10⁶ 次函數求值的預算，矩形給 7 位、梯形給 13 位——方法的差距不是快一點，是好幾個數量級。兩個註腳：加總用 math.fsum 是為了把浮點累加誤差這個變因隔離掉，確保 1.7×10⁻¹³ 量到的是方法本身的截斷誤差，不是運氣（ch04 的教訓：在 10⁶ 項的加總裡，浮點隨時可能回來攪局）；`trap` 那行用「右和 − 頭尾修正」計算，省一次迴圈——它代數上恆等於左右和的平均，值得你手驗一次。
