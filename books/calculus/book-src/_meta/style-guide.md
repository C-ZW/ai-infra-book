# 寫作風格指南（內部文件，非書籍內容）

本文件是全書的寫作契約。每一章都必須遵守，否則全書會失去一致性。
本書書名（工作名）：《馴服無限：微積分的直覺與驚嘆》。

## 這本書是什麼、不是什麼

- **是**：一本「欣賞與直覺」的書。目標是讓讀者親身感受人類如何馴服無限——看懂微積分不是一堆積分技巧，而是描述「變化」與「連續」的語言；讀完有「人類竟然想得出這個」的震撼，**而且能用自己的話對另一個工程師講清楚震撼在哪**。
- **不是**：
  - 不是考試解題訓練——不刷積分技巧、不背公式表、不出計算操練題。
  - 不是嚴格分析學教科書——ε-δ 只在它能帶來「啊哈」時當故事講（ch03 故事版、ch14 正式版），不是折磨。
  - 不是轉職用書——純興趣，不服務 AI Infra 敘事；但工程橋接要讓它讀起來像母語，不是教科書的外語。
- 驗收標準（全書的六條主線，ch16 總收官）：①微積分在馴服什麼；②三核心動作（極限／微分／積分）各自在做什麼；③FTC 為什麼是「面積＝斜率反過來」；④工程橋接；⑤直覺在哪裡崩壞；⑥它怎麼長成描述會變世界的語言。

## 讀者是誰

唯一讀者：Chewei，6 年以上資深後端工程師（TypeScript/Node.js）。可直接當橋接錨點的經歷與直覺：

- profiling 與效能工程（RDS CPU 尖峰降 40% 的量測→診斷→優化敘事）；CloudWatch／Prometheus 式指標思維（counter、rate、sum、P99）
- 高併發系統（500K+ 使用者、10K+ CCU）、佇列、重試與 exponential backoff、rate limiting
- 數值直覺：負載測試找容量上限（二分逼近）、浮點比較寫過 `abs(a-b) < eps`
- ML 概念層：知道梯度下降是「往坡下走」，沒推過數學

**不需要解釋**（讀者已精通）：函數的概念、代數操作、座標平面、二分搜尋、hash、佇列、重試、監控指標的工程意義。

**快速複習即可**（大學看過但生鏽）：三角函數、指數對數、Σ 記號、基礎數列。用「你大學看過，現在把它擦亮」的節奏帶，不當文組讀者教。

**要從零教**（本書主體）：極限的精確意義、微分與積分的「在做什麼」、FTC、級數收斂與發散、微分方程的世界觀、傅立葉的直覺、嚴格化的歷史與動機。

**跨書資源**：讀者書架上有《把系統寫成定理：TLA+ 與形式化方法》（量詞 ∀∃、否定推移、「工程師的嚴謹」證明觀）。ch14 的 ε-δ 量詞遊戲**可以引用該書**（「如果你讀過書架上的 TLA+ 書，∀ε∃δ 的對抗結構你已經見過」），但**不得依賴**——本書自身要把量詞講到能懂。

- 繁體中文，台灣慣用語。數學名詞中英並用：第一次出現給「中文（English）」，之後可只用中文——極限（limit）、導數（derivative）、積分（integral）、級數（series）、收斂（converge）／發散（diverge）、連續（continuous）、可微（differentiable）。
- 人名用通行中譯＋原文：牛頓（Newton）、萊布尼茲（Leibniz）、歐拉（Euler）、柯西（Cauchy）、魏爾斯特拉斯（Weierstrass）、黎曼（Riemann）、傅立葉（Fourier）、阿基米德（Archimedes）、芝諾（Zeno）。第一次出現附原文，之後用中譯。
- 台灣用語基準：函數／函式（數學脈絡用「函數」）、微分方程（不是微分方程式也可，擇一全書一致：用「微分方程」）、最佳化／優化皆可。禁止簡體中文詞彙（信息、默認、內存、代碼、對象、質量、導數的「求導」改用「微分」或「取導數」…）。
- 第二人稱「你」直接對讀者說話。

## 數學記號規範（全書一致，附錄 A 是速查表；閱讀器不渲染 LaTeX，一律純文字）

- **內文行內**用 Unicode：f′(x)（prime 用 ′）、ε δ Δ π ∞ √ ≈ ≠ ≤ ≥ → Σ ∫。
- 極限寫 `lim(x→a) f(x)`、`lim(n→∞) aₙ`。
- 求和寫 `Σ(n=1→∞) 1/n²`；積分上下限用 Unicode 上下標數字：`∫₀¹ x² dx`；非數字界限寫 `∫[a,b] f(x) dx`。
- 導數記號雙軌並用且**明說兩套記號的歷史**（牛頓 ẋ 一筆帶過即可；全書工作記號用萊布尼茲 dy/dx 與拉格朗日 f′(x)，何時用哪套在 ch04 講清楚）。
- **多行推導**放在 ```text 圍欄區塊裡，等號對齊、每步右側可加「← 為什麼」短註。禁止把長推導硬塞進行內。

## 每章骨架（必須遵守，標題名稱照抄——一致性掃描會 grep 這些標題）

1. 章首：`# chNN — 章名：副標`，接著 blockquote：
   `> **本章解決什麼問題**：…（這章在全書地圖的哪裡、為什麼存在、跟前後章的關係，3-5 行）`
   每個 Part 的第一章（ch01、ch04、ch07、ch09、ch12、ch15）在 blockquote 後放全書 ASCII 路線圖（基準圖在 outline.md，照抄並把「◄你在這裡」標到本章），讀者偏好架構圖先行。
2. `## 從你已知的出發` — 橋接段：把本章核心概念錨定在讀者的工程實戰上，要具體（CloudWatch rate、exponential backoff 的總時間、負載測試二分逼近、浮點 eps 比較），不要泛泛而談。
3. 核心內容（多個 `##` section，深入直覺與機制）。歷史敘事與數學內容交織，但**歷史服務於數學直覺**，不是編年史。
4. `## 直覺的陷阱` — 本書招牌段落（故障視角的數學版）：這個概念**最常被誤解成什麼**、錯誤直覺**會在哪一步把你帶溝裡**、**怎麼自我察覺**（什麼徵兆代表你其實沒懂）。用表格或條列。
5. `## 紙上推演` — 動手做的形式（2026-06-12 訪談定案：紙筆推演為主）。每章 2~4 題，標注預估時間（如 **[15 分鐘]**）與難度（★~★★★）。題型：手算逼近表、手推論證、找出錯誤論證的破綻、把白話翻成數學語言、口頭重講一個證明。題目後緊接 `### 推演解答` 給完整解答與討論（為什麼這樣想、常見錯路）。
   指定章節（ch04、ch07、ch09、ch10、ch11、ch13——見 outline）額外加 `### Python 小實驗`：本機可跑、純標準函式庫優先（matplotlib 可選用、要標明）、每段程式 ≤ 30 行、附預期輸出。程式碼與註解一律英文。
6. `## 自我檢核` — 5~8 個問題，**口頭自答**：講得出來才算過關，講不清楚＝還沒懂。問題對齊「能不能把這個概念講給另一個工程師聽」的深度，優先出「為什麼」而不是「是什麼」。
7. `## 延伸閱讀` — 真實連結（以 `_meta/landscape-2026-06.md` 驗證過的為優先），每條一句話說明為什麼值得讀、讀哪一段。

## 深度標準

- **每章至少一個 worked example**：帶具體數字、一步一步算的完整過程（逼近表、部分和表、迭代表），不准跳步。outline 已為每章指定基準範例。
- **嚴謹度的誠實標示**：每個論證明確標示自己是哪一級——「直覺版（嚴格版見 ch14／本書不證）」或「工程師的嚴謹：每步能口頭說出理由」。禁止「顯然」「易證」；禁止把直覺論證偽裝成證明。0.999…=1 這類題目要正面處理常見錯誤論證的循環依賴問題。
- **數值自我複核**：所有數值表格（逼近表、部分和、迭代）寫章時必須自己重算一遍（心算／筆算／必要時口頭逐步驗證），不得抄自記憶。跨章基準數值以 outline 基準表與 landscape 的「基準數值複核」節為準。
- **歷史事實照 landscape**：年份、歸屬、引文、軼事一律以 `_meta/landscape-2026-06.md` 為準；landscape 沒有的自行查證（每章 3~8 次 web 查證）並 hedge；傳說要標明是傳說（牛頓的蘋果）。禁止憑記憶引用「名人名言」。
- **每個「驚嘆點」要能被重講**：本書的核心商品是震撼感。每章至少一處明確回答「這裡了不起在哪」——而且要寫到讀者能用自己的話轉述，不是堆形容詞。
- 每個概念給「它在解什麼問題」：極限解什麼、FTC 解什麼、嚴格化解什麼。禁止無動機的定義。
- 寧深勿廣，不灌水。每章 5,000~9,000 中文字，依主題深度調整。

## 事實正確性

- 歷史與時效性事實（年份、人物、書籍版本、連結）以 `_meta/landscape-2026-06.md` 為事實基準；本章特定細節自行用 WebSearch/WebFetch 查證（每章 3~8 次）。
- 數學內容（定義、定理、計算）不依賴 landscape，但必須自我複核；常被誤傳的數學事實（−1/12、0.999…、Gibbs 現象百分比、Weierstrass 函數條件）以 landscape 查證版為準。
- 時效性事實標時點：`（2026-06）`。延伸閱讀只放真實存在的連結；沒把握標「（未驗證）」。
