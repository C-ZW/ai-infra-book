# ch14 — 背壓、rate limiting 與過載防禦：系統在洪水下怎麼不死

> **本章解決什麼問題**：Part IV 的第五個老問題——**過載**。前三章挖了交付（ch10）、一致（ch11）、並發（ch12）；這章問的是：**當進來的速度超過你處理的速度，會發生什麼？** 你的 FluentBit buffer 滿過（ch06）、SQS backlog 在結算尖峰堆過（ch04）、consumer 被洪水壓過——當年都靠直覺擋下來。本章把這些直覺組成一套**過載防禦的工程語言**：背壓、rate limiting、load shedding、circuit breaker、retry 的反作用。**這一章只講工程機制**——「為什麼 ρ→1 會爆炸」「雙穩態解長什麼樣」這些**數學**不在這、在《等待的數學》（queue），每次碰到就指過去。時間與重啟那一軸（buffer 在記憶體裡重啟即失）留給 ch13。

## 從你已知的出發

你處理過三次「進來的比處理得快」，三次都活下來了，但你大概沒把它們當成同一個問題。

**FluentBit 的 buffer**（ch06）。你自建那條 `input → parser → Lua filter → buffer → output(S3)` 的 pipeline 時碰過：**S3 寫入比 log 進來慢**、或 S3 偶爾抖一下寫不進去，buffer 就開始漲。你當時得決定——buffer 滿了，是**擋住**上游（讓 input 慢下來）、還是**丟掉**新 log？你選了某一邊（FluentBit 的 `Mem_Buf_Limit` / `storage.type` 在配這件事），但你大概沒意識到：**這個決定有個名字叫背壓（backpressure）——而且「擋」和「丟」之外還有第三個選項，那個最危險，叫「無界佇列」。**

**SQS 的 backlog**（ch04）。每季結算 10–30 萬筆，CronJob 一觸發任務一口氣灌進 SQS，你的 consumer 處理率固定、灌進來的卻是個**尖峰**。SQS 在這裡幫了你一個沒被感謝的忙——它是個**有界緩衝佇列**，把瞬間洪水攤平成 consumer 能慢慢消化的細流。但若那佇列是**無界**的、到達率持續超過處理率呢？backlog 無限長、**隊首等待時間趨近無限**。

**那次 consumer 被洪水壓住**。某次尖峰 consumer 處理不過來，下游（資料庫、某依賴服務）變慢甚至逾時。你大概加了重試——「逾時就重試嘛」。但反直覺的是：**你加的重試可能讓事情更糟**。下游已被壓垮，你重試等於對快死的人**再踹幾腳**——每個逾時請求變成兩、三個，流量不減反增，系統被自己的重試**卡死在壞平衡裡出不來**。

這三件事根子上是**同一個問題的三個面**：**到達率短暫或持續超過處理率，系統怎麼不死。** 你那三個直覺決定，正是過載防禦這套語言的基本動作。

但你可能沒想過——**這套防禦裡最危險的設計不是「丟東西」，而是「什麼都不丟」。** 一個沒有上界的佇列、一個無限的 buffer、一個「先存起來等等再處理」的善意設計，恰恰是把系統推向死亡的那條路。先說清楚為什麼。

## 背壓：下游慢了，壓力怎麼往上游傳

過載的本質是處理鏈上**某一環跟不上上一環的供給速度**。背壓（backpressure）就是**讓慢的下游能告訴上游「我跟不上了，你慢一點」，把壓力沿鏈往回傳**的機制。

沒有背壓會怎樣？想像水管：上游一直灌、下游出口堵住，水只能積在**中間的緩衝區**——而緩衝區是有限的記憶體。上游不知道下游堵了、繼續灌，緩衝區一直漲，最後**記憶體爆掉、行程被 OOM killer 殺掉**。背壓就是那根「告訴上游別再灌」的訊號。

當下游跟不上，系統面前永遠只有**三個選項**，沒有第四個：

```
   上游 ───灌資料───►  ┌─────────┐  ───處理───► 下游
                       │ 緩衝佇列 │
                       └────┬────┘
                            │ 下游跟不上時，這佇列怎麼辦？三選一：
            ┌──────────────┬──────────────┬──────────────┐
            │ ① 阻塞       │ ② 丟棄       │ ③ 無界增長   │
            │ 擋住上游     │ 滿了丟新的   │ 來者不拒     │
            │ 讓它等       │ 或丟舊的     │ 一直存       │
            ├──────────────┼──────────────┼──────────────┤
            │ 代價:上游卡  │ 代價:丟資料  │ 代價:延遲爆  │
            │ 壓力往上傳   │ 但系統活著   │ 記憶體爆死   │
            └──────────────┴──────────────┴──────────────┘
              ▲ 真正的背壓                    ▲ 最危險
```

**① 阻塞（block）**：緩衝區滿了就**擋住**上游、讓它停下來等，直到下游消化出空位。壓力一環環往上游傳——下游卡住、上游跟著卡、最源頭的生產者（接收請求的 socket）也慢下來。這是**最純粹的背壓**：把「我跟不上」的訊號一路傳回源頭。代價是上游被拖慢，但系統**不丟資料、不爆記憶體**。

**② 丟棄（drop）**：緩衝區滿了就**丟掉**——丟新來的（tail drop）或丟最舊的（head drop，視新舊誰更有價值）。代價是丟了資料，但系統**活著、延遲有界**。你的 FluentBit buffer 滿了丟 log 就是這個——丟幾條 log 比把整個 agent 撐爆好。

**③ 無界增長（unbounded）**：緩衝區**沒有上界**，來多少存多少。表面「最友善」——不擋、不丟、來者不拒。**這是三個選項裡最危險的**，下一節單獨講。先記住：**無界佇列不是「沒有背壓」，是「把背壓延後到記憶體爆炸那一刻才一次付清」。**

> **我以為 vs 實際**：你大概以為背壓是個要特別去實作的高級東西。實際上**背壓無所不在，你只是沒注意它有沒有被正確接上**。TCP 有背壓（滑動視窗滿了對方就停發）；SQS 其實是「有界佇列 + 消費端拉取」——consumer 沒拉訊息就留著、不會壓垮它。問題從不是「要不要背壓」，而是「**你那條鏈上有沒有哪一環偷偷選了選項 ③、把背壓訊號吃掉**」。

### Node stream 的 highWaterMark：背壓的具體化

你寫過的 Node stream 把背壓做成了一個**看得見的數字**——`highWaterMark`，它是 Writable stream 內部緩衝區的**警戒線**（警戒線、不是硬上限）。機制（2026-06 查證，Node.js 官方文件）：你呼叫 `writable.write(chunk)`，它**回傳 boolean**——回 `true` 表示緩衝區還沒到 `highWaterMark`、**可繼續寫**；回 `false` 表示已到或超過、意思是「**下游跟不上了你該停**」，你停下來**等 `'drain'` 事件**，緩衝區消化夠空時 stream 觸發 `'drain'`、你才繼續。

**`write()` 回的這個 boolean 就是背壓訊號本身**：下游慢 → 緩衝區漲過 highWaterMark → `write()` 回 false → 上游停下來等 → 壓力傳回上游。

`highWaterMark` 預設值是個你**很可能記成舊版**的數字：byte stream 的預設**在 Node 22 起從 16 KiB 調高到 64 KiB（65536 bytes）**（2026-06 查證——active LTS 是 Node 24「Krypton」，landscape §10；object mode 仍是 16 個物件）。記得「Node stream 預設 16K」的話，那是 Node 21 以前的事了。

最關鍵的陷阱是：**`highWaterMark` 是建議不是強制**。`write()` 回 false 後 **Node 不會阻止你繼續寫**；程式若**無視回傳值硬寫**，緩衝區無限漲——你親手把選項 ① 退化成選項 ③。這是 Node 串流最經典的記憶體洩漏：

```
# ❌ 無視背壓——快速來源灌進慢速目的地，buffer 無限漲
fastSource.on('data', (chunk) => {
    slowDest.write(chunk);   // 不看回傳值，下游慢也照寫 → buffer 爆
});

# ✅ 用 pipe / pipeline——它幫你接好背壓（write 回 false 就 pause 來源）
pipeline(fastSource, slowDest, (err) => { ... });
```

`pipe()` / `pipeline()` 是 Node 慣用法，正因為**它替你把背壓接好**——`write()` 回 false 時自動 `pause()` 來源、`'drain'` 時自動 `resume()`。你手動 `on('data')`+`write()` 時，背壓就**回到你頭上**，不接它它就消失。

## 為什麼「無界佇列」是最危險的設計

選項 ③ 是這一章你最該帶走的直覺——而它**反直覺到大多數人第一次都會選錯**。無界佇列看起來最善良：不擋、不丟、來多少收多少。問題出在一個殘酷的算術：**只要到達率（λ）哪怕只是「短暫地」超過處理率（μ），佇列等待時間就會漲，而且不會自己縮回去——除非 λ 降到 μ 以下、給系統時間追回欠的債。**

### Worked example：手追「短暫超載下延遲怎麼無限長」

用你結算 pipeline 的數字，但把 SQS 那個**有界**佇列換成假想的**無界**佇列。設 consumer 處理率 **μ = 100 筆/秒**、佇列無界，正常時 λ < 100、佇列空。CronJob 一觸發來了個尖峰：

```
時刻      到達率 λ        這 1 秒淨變化        佇列累積       隊首等待時間
─────────────────────────────────────────────────────────────────────
0–10s    300 筆/秒    +300 進、-100 出 = +200/s   到 10s: 2000 筆   2000/100 = 20 秒
10–20s   300 筆/秒    +200/s                       到 20s: 4000 筆   4000/100 = 40 秒
20–30s   300 筆/秒    +200/s                       到 30s: 6000 筆   60 秒
─────────────────────────────────────────────────────────────────────
30s      尖峰結束，λ 降回 50 筆/秒（< μ）
30–40s    50 筆/秒    +50 進、-100 出 = -50/s      到 40s: 5500 筆   55 秒
...                   每秒只能消化掉 50 筆積壓
排到第 30s 進來那筆全部清空，要 6000/50 = 120 秒之後
```

看清楚兩件事：

**第一，超載期間，隊首等待時間線性往上爬——只要 λ > μ 就一直漲、沒有上界。** 尖峰持續 30 秒，第 30 秒進來那筆前面排了 6000 筆、要等 60 秒；尖峰持續 5 分鐘，隊首就要等 10 分鐘。**佇列越長、最後進來的人等越久、沒有天花板**——這就是「延遲怎麼無限長」：**延遲的上界就是佇列長度的上界，佇列無界、所以延遲也無界。**

**第二，也最要命——債要還很久。** 尖峰第 30 秒結束（λ 降到 50 < μ），但系統**沒有立刻恢復**：欠了 6000 筆債、每秒只能還 50 筆（還要一邊處理新進的 50），要 **120 秒**才清完——**一個 30 秒尖峰造成 2 分鐘延遲後遺症**。這期間記憶體躺著 6000 筆未處理工作，尖峰更猛更久就大到**把記憶體吃光、OOM**。

> **這一段值得你停下來。** 它推翻一個直覺：「加大 buffer / 用無界佇列就能扛尖峰」。**不能。** 加大 buffer 只讓你**多撐幾秒、然後爆得更慘**（延遲漲更高、記憶體積更多、債欠更深）。無界佇列不解決過載，只是**把後果從「現在丟幾個請求」延後成「等一下整台一起死」**。真正的解不是「存更多」，是**讓 λ 降（限流、卸載）或讓 μ 升（擴容，但有延遲）**——在那之前你需要一個**有界**佇列、滿了就**丟**，用「丟一些」換「不死」。

> **數學在哪：** 為什麼 ρ=λ/μ 趨近 1 時等待會**爆炸性**（非線性、趨近無限）增長、為什麼「平均 ρ<1」也可能因**變異性**（突發、服務時間波動）排長隊——這些是排隊理論核心，完整在《等待的數學》（queue）：ρ→1 爆炸（M/M/1）、有界佇列丟棄機率（M/M/1/K）、變異性推高排隊（Kingman）各有專章。**本章只給「延遲 = 佇列長度 / 處理率、佇列無界則延遲無界」這個工程直覺。**

## Rate limiting：在門口限速——token bucket vs leaky bucket

背壓是「下游慢了往上游傳壓力」；rate limiting（限流）更主動——**在請求進門口就限速，根本不讓超量流量進系統**，是過載防禦的第一道閘門。兩個經典演算法 **token bucket**（令牌桶）和 **leaky bucket**（漏桶）常被混為一談，但它們在**對待突發（burst）**上有根本差別，這差別決定你該選哪個。

### token bucket：容忍突發

```
        令牌以固定速率滴進來（refill rate，如 5 個/秒）
                    │
                    ▼
            ┌───────────────┐
            │  ●●●●●●●●●●    │  桶有容量上限（capacity，如 10）
            │  令牌桶        │  滿了就溢出（多的令牌丟掉）
            └───────────────┘
                    │
   請求來 ─► 拿一個令牌 ─► 有令牌：放行（消耗 1 個）
                          └► 沒令牌：拒絕（或排隊等）
```

**令牌以固定速率補充**（refill rate）、桶有**容量上限**（capacity，滿了溢出），每個請求要**拿走一個令牌**才能通過、沒令牌就被拒。關鍵特性是**容忍突發**：令牌會**累積**，系統閒一陣子桶就攢滿，突然來一波時**攢的令牌讓這波瞬間全過**（最多「桶容量」個）。它限**長期平均速率**（=refill）、允許**短時突發**（最大=capacity）——正是大多數 API 限流想要的：使用者偶爾密集操作是正常的、只要長期不超配額就好。

### leaky bucket：整形成平滑

```
   請求來 ─► ┌───────────────┐
            │ ░░░░░░░░░░░░░  │  請求進佇列（漏桶）
            │  漏桶          │  桶滿了，新請求被丟棄
            └───────────────┘
                    │
                    ▼  以固定速率「漏出」（leak rate，如 5 個/秒）
              下游收到的永遠是平滑的固定速率流
```

請求進佇列（桶），桶**以固定速率漏出**（leak rate）給下游、桶滿了新請求被丟。關鍵特性是**把流量整形（shaping）成平滑固定速率**：無論進來多突發、**漏出的永遠是恆定速率**；它**不容忍突發**，突發會堆在桶裡被勻速放出（或桶滿被丟），下游看到一條**完全平滑**的流。

### 選哪個：保護的是「平均」還是「瞬時」（2026-06 查證，兩者語意穩定）

| | token bucket | leaky bucket |
|---|---|---|
| 限的是 | 長期平均速率，**容忍突發** | 瞬時輸出速率，**強制平滑** |
| 突發來時 | 攢的令牌讓它瞬間通過（≤ 容量），下游看到突發尖峰 | 堆在桶裡勻速放出 / 桶滿被丟，下游看到平滑流 |
| 適合 | **API 限流**（容忍使用者密集操作）、per-user 配額 | **流量整形**（餵一個只能吃固定速率的脆弱下游，如有硬性 QPS 上限的第三方 API） |

**判準一句話**：你要保護的下游**容不容得下突發**？容得下 → token bucket；容不下（瞬時尖峰就會打爆它）→ leaky bucket。大多數 API 限流選 token bucket——使用者的正常行為本來就帶突發，硬整形成平滑反而體驗差。

### 放在哪一層：邊緣 / 閘道 / 服務

限流是**分層**的，每層擋不同範圍：

- **邊緣（edge / CDN / WAF）**：擋最粗暴的——DDoS、單一 IP 的洪水。擋掉的流量**根本不進你的網路**，最省。
- **閘道（API Gateway / KrakenD）**：擋 per-API-key、per-user 配額（「免費用戶每分鐘 60 次」這種業務語意限流）。你用過的 **KrakenD**（無狀態 Go 閘道，landscape §16）就在這層；AWS **API Gateway** 有原生 throttling（帳號預設約 10,000 RPS burst，landscape §18），而 **ALB 本身沒有原生 throttling、要搭 WAF**（landscape §18）——這是條老兵記憶容易踩的線，別假設「擺個 ALB 就有限流」。
- **服務（應用層）**：擋這服務自己扛不住的（某個重 endpoint、某個下游依賴的保護性限流）。粒度最細，但請求已進到你家、成本最高。

原則：**越靠邊緣擋越省、越靠服務擋越精準**——粗暴流量靠邊緣、業務配額靠閘道、依賴保護靠服務。

### Worked example：token bucket 放行序列手算

設定：**桶容量 10、補充速率 5 個/秒、初始滿桶**。一波 **20 個請求**在 t=0 **瞬間**全部到達、之後無新請求。問哪些放行、哪些被拒、後續怎麼補。

```
初始：桶 = 10 個令牌（滿）。t=0 瞬間來 20 個請求。

t=0.0s   桶=10。20 個請求瞬間到。
         前 10 個各拿走 1 個令牌 → 放行 ✓（桶歸 0）
         第 11~20 個：桶空了 → 拒絕 ✗（10 個被拒）
         → 突發的前 10 個瞬間通過（這就是「容忍突發」——攢的令牌一次放）

此後桶以 5 個/秒補充（每 0.2 秒補 1 個令牌），被拒的第 11~20 個依序在
t=0.2、0.4、0.6 … 2.0s 各放行一個，到 t=2.0s 全部通過。

結果：前 10 個在 t=0 瞬間全過（突發容忍），剩 10 個以 5 個/秒在接下來 2 秒陸續放行。
      長期平均速率 = 5 個/秒（=refill rate），但瞬時容許 10 個的突發。
```

對照 **leaky bucket（漏速 5 個/秒、桶容量 10）**：20 個瞬間到，桶**最多裝 10 個**（其餘直接丟），以**固定 5 個/秒**勻速漏出——**沒有「前 10 個瞬間通過」這回事**，連第一個都要等漏桶輪到它。這就是兩者最直觀的差別：**token bucket 讓突發開頭瞬間通過，leaky bucket 把一切勻速化。**

## Load shedding：過載時主動丟低優先，保住核心

限流是「在門口按統一規則限速」；load shedding（負載卸載）更進一步——**系統已過載時，主動、有選擇地丟掉一部分請求，保住核心功能不死**。差別在「主動」與「有選擇」：限流是事先定好的閘門（每分鐘 60 次）、不管系統當下健不健康；load shedding 是**根據系統當前健康狀態動態決策**——偵測到自己快撐不住（佇列漲、延遲飆、CPU 滿載）就**開始丟、且優先丟低價值的**。

核心理念：**過載時丟一部分比全死好。** 被洪水淹沒的系統若什麼都想處理，結果是**所有請求都慢到逾時、一個都沒成功**（無界佇列的死法）。不如**主動丟一部分**讓剩下的**正常快速完成**——處理 70% 且都成功，遠勝收 100% 全部逾時。

「有選擇」是關鍵——先分優先級：**丟低優先保高優先**（背景批次 < 使用者互動、匿名 < 付費）、**丟讀保寫**（寧可讓人讀到稍舊快取也要保住寫入）、**丟新進保在途**（處理到一半的讓它完成、新進的才丟）。

回到你的系統：**FluentBit buffer 滿了丟 log**（ch06）就是一種 load shedding——log 相對低價值（丟幾條 access log 不影響業務），所以 buffer 撐不住時丟 log、保住 agent 不被撐爆、保住主業務不受可觀測性管線拖累。你當年用對了原則：**丟低價值的（log），保高價值的（業務本身）。**

> **load shedding 的觀測代價**（呼應五軸的「觀測」軸）：一旦開始丟，**必須讓「丟了多少、丟了誰」可觀測**——否則丟得無聲無息，你根本不知道系統正在卸載、使用者正在受影響。「靜默丟棄」的系統比「明確報錯」的更危險，因為你失去了「它壞了第一時間知道」的能力（ch15 要挖的）。**丟，但要大聲地丟。**

## Circuit breaker：強迫系統離開壞平衡

前面三個機制處理「流量太大」。circuit breaker（斷路器）處理另一種過載——**某個下游依賴壞了或變慢，而你還在不停呼叫它**。

問題場景：你呼叫的下游**掛了或嚴重變慢**（每個請求都逾時）。你若**繼續傻傻呼叫**，兩件壞事：(1) 每個請求卡在逾時上（比如 30 秒），執行緒/連線被占住、你自己也跟著慢甚至耗盡資源；(2) 你的重試**對半死的下游持續施壓**、讓它更難復原。**你和下游一起卡死在壞平衡裡，誰也出不來。**

circuit breaker 借電路斷路器的比喻：**偵測到下游持續失敗就「跳閘」——直接斷開、不再呼叫、快速失敗（fail fast），給下游喘息空檔，也保住自己的資源。** 它是個**三態有限狀態機**（2026-06 查證，closed/open/half-open 是業界標準三態，Resilience4j 等實作為準）：

```
   ┌──────────────────────────┐
   │ CLOSED（正常）           │
   │ 放行請求、統計失敗率     │
   └────────────┬─────────────┘
                │  失敗率超閾值 → 跳閘
   ┌────────────┴─────────────┐
   │ OPEN（跳閘）             │
   │ 全部拒絕、快速失敗       │
   └────────────┬─────────────┘
                │  冷卻時間到 → 試探
   ┌────────────┴─────────────┐
   │ HALF-OPEN（試探）        │
   │ 放行少量；成功→CLOSED    │
   │ 失敗→回 OPEN             │
   └──────────────────────────┘
```

- **CLOSED（閉合，正常）**：請求正常通過，斷路器**統計失敗率**。超過閾值（如「最近 100 次裡 50% 失敗」）→ **跳到 OPEN**。
- **OPEN（斷開，跳閘）**：**所有請求直接拒絕、快速失敗**，根本不呼叫下游——這**斬斷了那個施壓迴圈**：你不再踹半死的下游、它得以喘息，你也不再把執行緒卡在逾時上。等一段**冷卻時間**（如 30 秒）→ 跳到 HALF-OPEN。
- **HALF-OPEN（半開，試探）**：放行**少量**試探請求。成功 → 回 **CLOSED**；失敗 → 回 **OPEN** 再等一輪冷卻。

為什麼要 HALF-OPEN 中間態？因為從 OPEN 直接回 CLOSED 太魯莽——萬一下游還沒好，一口氣放行所有請求等於**又把它打趴一次**。HALF-OPEN 是**謹慎的試水溫**：只放幾個進去探，確認好了才全面恢復。

circuit breaker 該裝在「你呼叫一個可能壞掉/變慢的下游」的地方。**結算 consumer → 下游服務**（ch04）：consumer 寫 guild/player/inventory/mail 時某服務掛了，沒斷路器的話 consumer 會**對掛掉的服務不停重試、自己卡死、SQS backlog 暴漲**；裝個斷路器，那服務一跳閘 consumer 就快速失敗（訊息丟回佇列稍後重試或進 DLQ），不被一個壞下游拖垮全部。**FluentBit → S3**（ch06）：S3 抖動時與其每條 log 都卡在逾時上，不如跳閘、暫走 fallback（本地暫存）等 S3 恢復。

> **我以為 vs 實際**：你可能把斷路器當成「retry 的對立面」。其實它們**搭配**：斷路器決定**這下游現在值不值得呼叫**（OPEN 時連試都不試），retry 決定**值得呼叫時失敗了要不要再試**——斷路器是給 retry 加的**「別再踹死人」剎車**。Netflix 的 **Hystrix** 是這模式最有名的早期實作，但**早在 2018 年就進維護模式、不再積極開發**（2026-06 查證）——現在 JVM 生態標準是 **Resilience4j**。記憶裡斷路器=Hystrix 的話，是個該更新的老兵記憶。

## Retry 的反作用：重試怎麼把小尖峰放大成雪崩

這一章最反直覺、也最危險的機制：**retry（重試）——那個你以為是「容錯標配」的東西——在過載下會把系統推進一種叫 metastable failure（亞穩態故障）的死亡陷阱。**

你的直覺沒錯：單一請求偶發失敗、重試一下通常就好。問題出在**系統性過載**時——下游不是「偶發抖一下」而是「被壓垮了」，重試會做一件災難性的事：**放大流量**。

### retry storm：放大的數學身分

下游過載、開始逾時，每個請求有 50% 機率失敗，你設「失敗重試 3 次」。算一下這對下游的**實際流量**意味什麼：

```
正常時：100 個請求 → 下游收到 100 個。放大倍數 1×。

下游過載、50% 失敗、最多重試 3 次：
  第 1 輪：100 個請求 → 50 個失敗
  第 2 輪（重試）：50 個 → 25 個失敗
  第 3 輪（重試）：25 個 → 12 個失敗
  第 4 輪（重試）：12 個 → ...
  下游實際收到 ≈ 100 + 50 + 25 + 12 ≈ 187 個請求

→ 你的重試讓打在「已經過載的下游」上的流量幾乎翻倍。
   下游越慢、失敗越多、重試越多 → 流量越大 → 下游更慢……
```

那是個**正回饋迴圈**——不會自己收斂、只會自己**加速**。一個本來只是「下游慢了 200ms」的小擾動，被重試放大成「下游徹底崩潰、再也起不來」。這就叫 **retry storm（重試風暴）**。

### metastable failure：為什麼移除觸發因素也救不回來

retry storm 最可怕的地方，是它把系統卡進 **metastable failure（亞穩態故障）**。這概念由 **Bronson 等人在 HotOS '21 的論文 "Metastable Failures in Distributed Systems" 首次系統化**（2026-06 查證）——把「persistent congestion、overload、cascading failure、retry storm、death spiral」這些長得不一樣的故障**統一在同一框架下**。結構是三件事：**①壞狀態**（卡在高負載、高失敗、高重試的退化態）、**②觸發因素**（推進壞狀態的那一下——尖峰、下游抖動、一次部署）、**③自我維持的迴圈**（在 retry storm 裡就是「重試放大流量 → 下游持續過載 → 持續逾時 → 持續重試」）。

**這整章最該記住的一點：觸發因素消失了，系統也不會自己好。** 尖峰早過去、下游本該能喘口氣，但**③ 製造的流量已足以靠自己維持過載**——系統陷在壞平衡裡，**外力（手動重啟、清佇列、暫關重試）不來就一直死**。Bronson 等人因此主張：**根因應算在那個自我維持的迴圈（③）上、不是觸發因素（②）**——同一個壞狀態可被無數種觸發引發，**修迴圈才治本，修觸發只擋這一次。**

這正是 circuit breaker 重要的深層理由：**它是斬斷自我維持迴圈的剪刀。** OPEN 強制停止呼叫，等於**從外部把正回饋迴圈掐斷**、逼系統離開壞平衡——這就是本章標題「強迫系統離開壞平衡」的含義。

> **數學在哪：** 為什麼這迴圈有「**兩個**自洽平衡點」（健康低延遲態 vs 過載高延遲態）、為什麼會在兩態間**遲滯（hysteresis）**跳變、多大的擾動才把系統從健康態**推過山脊**、四種防禦各削掉哪一項——這些是雙穩態（bistability）的數學，完整在《等待的數學》（queue）**ch20**（有「一個 λ、兩個自洽狀態」的手算小模型）。**本章只到「正回饋迴圈→自我維持→需外力斬斷」這個工程直覺，數學一律指過去。**

### 退避 + jitter：為什麼兩個都不能少

既然純 retry 危險，那正確的 retry 長什麼樣？兩個元件，缺一不可：

**① 指數退避（exponential backoff）**：重試別立刻、也別固定間隔，而是**間隔越來越長**（1s、2s、4s、8s…，配一個上限 cap）。下游需要時間復原，你越退讓它越有機會喘氣。退避削掉的是**重試的頻率**——讓「下游慢 → 更多重試」這迴圈的後半段慢下來。

**② jitter（抖動 / 隨機化）**：在退避間隔上**加隨機**。這常被省略，但**省了它退避就廢一半**。為什麼（2026-06 查證，AWS Marc Brooker 的經典論證）：

```
問題：一個尖峰讓 1000 個 client 同時失敗。
   只有退避、沒有 jitter：
     它們全部「退避 1 秒」→ 1 秒後 1000 個同時重來 → 又一起打爆 →
     又全部「退避 2 秒」→ 2 秒後 1000 個又同時重來 → 再爆……
   → 退避沒有打散叢集，只是讓「同步的洪水」每隔越來越久來一次。
     重試請求在時間軸上仍然是一個個尖刺。

   退避 + jitter（每個 client 退避 1 秒 ± 隨機）：
     1000 個的重試被「抹開」散佈在一段時間裡 →
     從「一個 1000 的尖刺」變成「接近恆定速率的細流」→ 下游扛得住。
```

**jitter 的工作是「去相關（decorrelate）」**——把那些**因同一原因、在同一刻一起失敗、於是會在同一刻一起重試**的請求**打散到時間軸上**：沒有 jitter，所有 client 像被同一節拍器指揮、一波波同步衝擊；有了 jitter，各自隨機錯開、叢集被抹平成接近恆定的到達率。所以它降的不是「重試總量」（那是退避降的），而是重試的**「叢發性（burstiness）/ 相關性」**。**退避降頻率、jitter 降叢發**，兩個一起才把 retry 從「雪崩放大器」變回「安全的容錯手段」——AWS 各家 SDK 預設就帶退避+jitter，正因為這對組合是十幾年踩出來的血淚標配。

## 故障模式與防禦

過載這個老問題會怎麼壞、壞了長什麼樣、怎麼防。至少一個綁定你真實的系統。

| 故障 | 壞了長什麼樣（可觀測徵兆） | 怎麼防 |
|---|---|---|
| **無界佇列吃光記憶體**（某個 buffer / in-memory queue 沒設上界，**你的 FluentBit buffer 若不設 `Mem_Buf_Limit`**） | 記憶體單調上漲、延遲線性飆高、最後 OOM 被殺、重啟後在途資料全失（持久那一軸，見 ch13） | 任何佇列/buffer **必須有界**；滿了選「丟」或「擋」（背壓選項 ①②），**永遠別選無界**。FluentBit 設 `Mem_Buf_Limit` / 用 filesystem buffer |
| **Node stream 無視背壓**（`write()` 回 false 還硬寫，快來源灌慢目的地） | 行程記憶體緩慢漲、GC 壓力升、最終 OOM；壓測短時跑不出來、長跑才爆 | 用 `pipeline()`/`pipe()` 讓 Node 替你接背壓；手動 `write()` 必須檢查回傳值、false 就停等 `'drain'` |
| **retry storm / metastable failure**（下游過載時還在重試，**你那次 consumer 被洪水壓住**） | 觸發因素（尖峰）早過了系統卻不恢復；下游流量「莫名」維持高位；重啟/清佇列才好 | retry 必配**指數退避 + jitter**；加 **circuit breaker** 斬斷自我維持迴圈；對非冪等操作重試前先確認冪等（ch10） |
| **重試把流量放大成雪崩**（多層各自重試，放大倍數相乘） | 一個小抖動引發跨服務連鎖過載；每層日誌都顯示「下游逾時、正在重試」 | 限制重試次數與層數；**只在最該重試的那一層重試**，別每層都重試（放大倍數相乘）；配退避 + jitter |
| **限流放錯層 / 以為有其實沒有**（**假設「擺個 ALB 就有限流」**） | 洪水長驅直入打到服務層、服務被打爆，回頭查才發現 ALB 沒擋 | 認清各層能力：ALB **無**原生 throttling（要搭 WAF）、API Gateway/KrakenD 有（landscape §16/§18）；粗暴流量靠邊緣、業務配額靠閘道 |
| **靜默 load shedding**（過載丟請求但沒觀測，沒人知道在丟） | 使用者偶發失敗但監控一片綠、「沒有錯誤」卻有客訴；丟棄無紀錄 | 卸載**必須可觀測**——丟了多少、丟了誰要有 metric 與告警（ch15）；「丟，但要大聲地丟」 |
| **斷路器永遠半開不來 / 閾值設錯**（HALF-OPEN 試探一直失敗、或閾值太敏感頻繁跳閘） | 下游早恢復了斷路器還 OPEN（過度保守、可用性掉）；或抖一下就跳閘（過度敏感） | 閾值與冷卻時間要依下游特性調；試探請求數要夠判斷但別打爆下游；監控斷路器狀態轉換 |

最該記住兩條：**無界佇列/無視背壓**「壓測測不出、長跑才爆、爆就 OOM 整台死」，不能靠「沒復現」放心、要在設計層面保證每個佇列都有界；**metastable failure** 致命在「觸發因素消失也不會自己好」，防它不能只防觸發（只擋這一次）、要從機制上斬斷自我維持迴圈。過載故障的跨領域通論見《正常意外》（fail）。

## 紙上推演

### 推演一：手追「無界佇列在短暫超載下延遲怎麼無限長」 **[20 分鐘]** ★★

不准翻回前面，自己把那個算術走一遍，並回答它推翻了哪個直覺。

**問**：設一個 consumer 處理率 μ = 100 筆/秒、佇列**無界**。(a) 一個尖峰讓到達率 λ = 250 筆/秒、持續 20 秒。請逐 5 秒列出佇列累積量與「此刻進來那筆要等多久」。(b) 尖峰在第 20 秒結束、λ 降回 60 筆/秒（< μ）。系統**立刻**恢復正常嗎？把第 20 秒那筆積壓清完要多久？(c) 用一句話說明「延遲無界」的根源（提示：延遲和佇列長度是什麼關係、佇列的上界是什麼）。(d) 有人說「那我把佇列容量設大一點不就扛得住尖峰了」——駁倒他，說清楚加大佇列**換到了什麼、付出了什麼**。

### 推演二：token bucket 放行序列手算 **[15 分鐘]** ★★

設定：**桶容量 8、補充速率 4 個/秒、初始滿桶**。一波 **15 個請求**在 t=0 瞬間到達，之後無新請求；被拒的請求每 0.25 秒重試一次（時間對齊令牌補充）。

**問**：(a) t=0 瞬間，幾個放行、幾個被拒？為什麼是這個數字（哪個參數決定了瞬間放行量）？(b) 列出 t=0 之後每個被拒請求的放行時刻，直到 15 個全部通過。(c) 整體看，這 15 個請求的「長期平均通過速率」是多少？這個數字等於哪個參數？(d) 如果換成 **leaky bucket（漏速 4 個/秒、桶容量 8）**，同樣 15 個瞬間到，第一個請求的命運跟 token bucket 有什麼根本不同？

### 推演三：給一個過載場景選機制，並判斷「加大 buffer 救不救得了」 **[20 分鐘]** ★★★

場景：你的結算 consumer（ch04）在某次季末尖峰下，發現下游的 player 服務變慢（回應從 50ms 漲到 8 秒、開始逾時）。你的 consumer 設了「逾時就重試」，現在 SQS backlog 在暴漲、player 服務的流量「莫名」一直維持高位降不下來，連尖峰過了都不恢復。

**問**：(a) 這是什麼故障的典型徵兆？指出那個「自我維持的迴圈」具體是什麼。(b) 從本章五個機制（背壓、限流、卸載、circuit breaker、退避+jitter）裡，選出**最該先上**的一個來止血，說為什麼是它。(c) 有人提議「把 SQS 的 visibility timeout 調長、讓訊息多在佇列裡待著別急著重投，等下游恢復」——這算是「加大 buffer」嗎？它救得了還是拖更久？(d) 等止血了，要從根本防止再發生，你會把哪兩個機制做進 consumer？各防住迴圈的哪一段？

### 推演解答

**推演一解答。**

(a) μ=100、λ=250，每秒淨增 250−100 = **150 筆**：

```
t=5s:   佇列 = 150×5 = 750 筆    隊首等待 = 750/100 = 7.5 秒
t=10s:  佇列 = 1500 筆           等待 = 15 秒
t=15s:  佇列 = 2250 筆           等待 = 22.5 秒
t=20s:  佇列 = 3000 筆           等待 = 30 秒
```

等待時間**線性往上爬，沒有天花板**——尖峰持續越久，最後進來的人等越久。

(b) **不會立刻恢復**。第 20 秒積壓 3000 筆。尖峰後 λ=60、μ=100，每秒淨**消化** 40 筆，清完要 3000/40 = **75 秒**。一個 20 秒的尖峰，留下 75 秒的延遲後遺症——欠的債只能在「處理率 − 新到達率」這微薄差額下慢慢還。

(c) **延遲 ≈ 佇列長度 / 處理率**；佇列**無界**，故延遲**無界**。處理率固定，唯一能讓延遲爆炸的就是佇列長度，而無界佇列只要 λ>μ 就一直漲、沒有上限。

(d) 加大佇列**換到**多撐幾秒，**付出**延遲漲更高、記憶體積更多、債欠更深、爆得更慘——它**完全沒**解決「λ>μ」：λ>μ 時任何有限容量遲早填滿、任何無限容量遲早 OOM。真正的解是**讓 λ 降**（限流/卸載）或**讓 μ 升**（擴容，但有延遲），在那之前你要的是**有界**佇列、滿了就丟。（ρ→1 時等待為何爆炸性而非線性，見《等待的數學》queue。）

**推演二解答。**

(a) **8 個放行、7 個被拒**。瞬間放行量 = **桶容量**（初始滿桶 8 個令牌，前 8 個請求各拿 1 個，桶歸 0，第 9~15 個沒令牌被拒）。決定瞬間突發量的是**容量**，不是補充速率——這正是 token bucket「容忍突發」的來源。

(b) 補充速率 4 個/秒 = 每 0.25 秒補 1 個令牌，於是第 9~15 個請求依序在 t=0.25、0.50、0.75、1.00、1.25、1.50、1.75 秒各放行一個，到 t=1.75s 全部通過。

(c) 前 8 個瞬間過、後 7 個在 1.75 秒內陸續過。撇開初始那波突發，**長期平均通過速率 = 4 個/秒 = 補充速率（refill rate）**；token bucket 限的是長期平均、容忍的突發上限是容量。

(d) **根本不同**：leaky bucket 沒有「初始滿桶讓前 8 個瞬間過」這回事。15 個瞬間到、桶最多裝 8 個（其餘直接丟），以固定 4 個/秒勻速漏出——**連第一個都要等漏桶按節奏輪到它**，沒有任何瞬間放行。token bucket 讓突發開頭瞬間通過，leaky bucket 把一切勻速化。

**推演三解答。**

(a) 典型的 **retry storm / metastable failure**：三個招牌徵兆都齊了（流量「莫名」維持高位、尖峰過了不恢復、要外力才好）。**自我維持的迴圈**是：**player 慢 → consumer 逾時 → consumer 重試 → player 流量增加 → player 更慢 → 更多逾時……** 觸發因素（季末尖峰）早過了，但迴圈靠重試流量**自己維持**著過載——這就是 metastable failure 的定義性特徵。

(b) **最該先上 circuit breaker**。它是唯一能**直接斬斷自我維持迴圈**的機制——OPEN 強制停止呼叫 player，等於從外部把「重試→流量→更慢」掐斷，給 player 喘息空檔。退避+jitter 能**減弱**迴圈強度但不能立刻掐斷（還在重試，只是慢一點散一點）；限流/卸載擋的是**入口**流量，但病灶在 consumer 對 player 的**出口**呼叫，所以先上斷路器止血。

(c) **不是真正的解，會拖更久**。調長 visibility timeout 確實讓訊息晚點重投（某種「加大緩衝」），但它沒斬斷迴圈——只要 consumer 還在對慢掉的 player 發請求並重試，迴圈就在轉。它只把 backlog 爆發**延後**，代價是結算延遲更長，而 player 在延後期間不會自己好（迴圈還在），重投潮一來又是一輪風暴。跟推演一同理——**緩衝只延後後果，不解決「迴圈沒斷」這個根本**。

(d) 兩個機制：**① circuit breaker**——防住「consumer 持續踹半死的 player」（迴圈的施壓端），下游壞了快速失敗、別卡執行緒別放大流量。**② 重試的退避 + jitter**——防住「重試同步放大流量」（迴圈的放大端），退避降頻率、jitter 把重試打散成去相關平滑流。斷路器掐斷迴圈、退避+jitter 在閉合時也讓重試溫和。（前提：consumer 對 player 的寫入要冪等，重試才安全，見 ch10。）
