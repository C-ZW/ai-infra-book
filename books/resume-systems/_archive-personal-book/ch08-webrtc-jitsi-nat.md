# ch08 — 語音與視訊：WebRTC、Jitsi 與 NAT 穿透

> **本章解決什麼問題**：這是 Part III「回到 LetsTalk」的第二站。ch07 把你那套 WebSocket + Redis Pub/Sub 的即時通知挖到了「訊息扇出最容易掉在哪一格」——那是**伺服器替使用者轉訊息**的世界。這一章換一個更難的世界：當兩個使用者要**直接互通音訊與視訊**、繞過你的伺服器、媒體還是 30 fps 即時流時，事情就不一樣了。你當年評估了 Twilio / Agora，最後**因成本選 Jitsi** 做群組通話、用 Socket.io 做信令、設了 TURN/STUN 做 NAT 穿透。本章把這條決策挖到底：WebRTC 其實不是一個協定、信令為什麼規格刻意不定、為什麼 NAT 後的兩台機器不能直接連、symmetric NAT 下為什麼 STUN 會失敗退回 TURN relay、群組為什麼 mesh 不可行而 Jitsi Videobridge 是 SFU。最後還原你當年的選型——並校正一個現在已經站不住腳的老兵論點。

## 從你已知的出發

你在 LetsTalk 做過群組語音與視訊。這是一款隱私導向的 IM，500K+ 註冊、~3,200 並發連線、~10M 訊息/月。文字訊息那條路你已經自建好了（WebSocket + Redis Pub/Sub，見 ch07）；但「打電話」「開視訊會議」是完全不同的工程，你不可能用 WebSocket 傳即時影音——那是 WebRTC 的活。

當年你做了一輪選型：評估了 **Twilio** 的 Programmable Video、評估了 **Agora**，最後**因成本選了 Jitsi**——自架 Jitsi（Videobridge 那套）做群組通話，**Socket.io 當信令層**交換協商訊息，再自己架 **TURN/STUN**（coturn）做 NAT 穿透。你當年的理由很直接：Twilio、Agora 是商用 SDK，按用量計費，而你們是一款使用者規模不小、毛利又敏感的隱私 IM，每分鐘通話都要付錢給第三方，帳算不過來；Jitsi 開源、可以自架，授權費是零。

你大概還記得幾個操作層的細節：要設一台 STUN server 讓 client 找到自己的對外位址，還要設 TURN server 當「連不通時的中繼」，因為「有些使用者在很奇怪的網路後面，直連會失敗」。你也記得 Jitsi 的 Videobridge 是「群組通話的核心」，因為「一對一可以兩台直連，但多人就得有個伺服器在中間轉」。

這些你都做對了。但我想攤開幾個你當年**用直覺繞過、沒真正想穿**的問題：

**第一，WebRTC 到底是「一個協定」嗎？** 你大概把它當成一個東西——「瀏覽器之間直接傳影音的那個技術」。其實它是**一整套**：媒體怎麼加密傳（SRTP）、連線怎麼建立與穿透（ICE/DTLS）、雙方怎麼先講好「我支援什麼編碼、我的位址在哪」（SDP），以及——**信令層它根本不管，是你自己挑的**（你挑了 Socket.io）。把這四塊分清楚，後面所有故障才定位得到。

**第二，為什麼信令要你自己做？** 這不是 Jitsi 偷懶，是 WebRTC 規格**刻意不定**信令。為什麼一個這麼核心的東西，標準偏偏留白？

**第三，「連不通退 TURN」那個 TURN 到底在解什麼？** 你當年知道「symmetric NAT 下 STUN 會失敗、要退回 TURN relay」——但你大概沒細想：為什麼偏偏是 symmetric NAT 讓 STUN 失靈？TURN relay 一上場，媒體就不再是 P2P 直連、要繞一台中繼伺服器，這對你的成本與延遲意味著什麼？

**第四，當年那個「因成本選 Jitsi」的決定——對嗎？** 我先說結論：**對，但你今天會用來支撐它的某個理由，現在已經站不住腳了。** 你大概聽過「Twilio Video 反正要關了」這個說法。這件事有過、但**被官方撤回了**。後面我會還原這段，並重講一個今天仍然成立的取捨論述。

這一章就把這四個問題挖穿，最後手追一通 P2P 視訊從零建立的完整握手序列，再幫你把 Twilio / Agora / Jitsi 三家在四個維度上的位置重新排一次。

## WebRTC 不是一個協定，是一套——先把四塊分開

這是全章的地基，慢一點。你之所以後面會在「為什麼連不通」上卡住，往往是因為把這四塊混成一團。把它們拆開，每塊各管一件事：

```
            一通 WebRTC 連線（由上而下四塊）

┌────────────────────────────────────────────────────┐
│ 信令（你自己做）                                    │
│   offer/answer（SDP）＋ candidate                    │
│   Socket.io 在這層搬訊息・WebRTC 規格刻意不定       │
└────────────────────────────────────────────────────┘
┌────────────────────────────────────────────────────┐
│ 連線建立／穿透                                      │
│   ICE：蒐集 candidate、配對連通性                   │
│   STUN 找對外位址、TURN 當中繼                      │
└────────────────────────────────────────────────────┘
┌────────────────────────────────────────────────────┐
│ 傳輸安全                                            │
│   DTLS 握出金鑰（強制加密）                         │
└────────────────────────────────────────────────────┘
┌────────────────────────────────────────────────────┐
│ 媒體                                                │
│   SRTP 傳加密的音訊／視訊封包                       │
└────────────────────────────────────────────────────┘
```

由上而下，四塊各自的職責：

- **信令（signaling）**：在媒體連線**還沒建立**之前，雙方得先透過**某條已經存在的通道**互相告知「我支援哪些編碼、我的媒體位址候選有哪些」。這條通道就是信令——而 WebRTC 規格**不規定它長什麼樣**。你用 Socket.io 搬的，就是這一層的內容。
- **連線建立（ICE）**：雙方都躲在 NAT/防火牆後面，誰也不知道對方的真實可達位址。ICE（Interactive Connectivity Establishment，**RFC 8445**，obsoletes 5245，landscape §7）是整套「找出一條能通的路」的框架——它去蒐集各種**候選位址（candidate）**、兩兩配對測試連通性、挑一條最好的。STUN 與 TURN 是它手裡的兩個工具。
- **傳輸安全（DTLS）**：WebRTC **強制加密**，沒有「不加密」這個選項。DTLS 在 ICE 找到的路徑上握出金鑰。
- **媒體（SRTP）**：真正的音訊與視訊封包，用 DTLS 握出的金鑰加密後，以 SRTP（Secure RTP）傳輸。

> **我以為 vs 實際**：你大概把「WebRTC」當成一個會自己搞定一切的黑盒——`new RTCPeerConnection()`，影音就通了。實際上它是一套**分工明確的協定組合**，而且其中**最關鍵的一塊（信令）它根本不提供**。你之所以要寫 Socket.io 程式碼、要架 coturn，正是因為這套東西**故意只給你一半**：它把「媒體怎麼安全地直連」做好，把「兩個人怎麼先找到對方、講好參數」留給你。理解這個分工，是後面所有故障定位的前提——影音不通，你第一件事該問的是「斷在哪一塊？信令沒送到？ICE 沒配對成功？還是 DTLS 握手失敗？」

本章的核心是中間那兩塊：**信令**（為什麼你得自己做、做什麼）和 **ICE/STUN/TURN**（為什麼 NAT 後不能直連、為什麼有時要退回中繼）。最底兩塊（DTLS/SRTP 的密碼學內部、編解碼器內部）不是本章範圍——本書講機制與取捨，不展開演算法內部。

順帶把規格狀態釘死，免得你記成過時版本：**WebRTC 1.0 於 2021-01-26 成為 W3C Recommendation，至今仍是現行標準**；W3C 後續以 amended Recommendation 重新發布（最新版日期 2025-03-13），所以它**沒有被取代、持續有效**（landscape §5）。你 2019–2022 用的那套 WebRTC，協定面到今天是穩的——會腐的是周邊（Twilio 的產品狀態、coturn 版本），不是 WebRTC 本身。

## 為什麼信令要你自己做：規格的刻意留白

來回答「為什麼信令規格刻意不定」這個問題——這是本章最容易讓人「喔原來如此」的一段。

WebRTC 要建立連線，雙方必須先交換兩種東西：

1. **媒體能力與參數**：我支援 H.264 還是 VP9？音訊用 Opus 嗎？我的媒體串流長什麼樣？這些打包成一段叫 **SDP（Session Description Protocol）** 的文字。發起方產生一份 **offer**（SDP），接收方回一份 **answer**（SDP）——這就是 **offer/answer 模型**。
2. **網路位址候選（ICE candidate）**：我可能可以從這些位址被連到——本機網卡位址、經 STUN 探到的對外位址、經 TURN 拿到的中繼位址。

問題來了：要交換這些東西，你**需要一條雙向通道**。但 WebRTC 連線**此刻還沒建立**（你正是為了建立它才需要交換這些）。這是個雞生蛋問題——**建立媒體連線需要先交換參數，但交換參數需要一條已經存在的連線**。

WebRTC 的設計選擇是：**這條「先存在的通道」，我不規定，你自己挑。** 為什麼？因為每個應用本來就**早已有一條伺服器通道**了——你的 LetsTalk 本來就有 WebSocket/Socket.io 連著每個在線使用者（為了傳文字訊息）。讓 WebRTC 去硬性規定一套信令協定，等於強迫所有應用再多接一套東西；留白，反而讓你能**直接複用既有的即時通道**。這是一個漂亮的「不做決定」——把選擇權還給最了解自己場景的人。

所以你當年用 Socket.io 做信令，不是 hack，是**規格設計就預期你這麼做**（landscape §5、§21；官方文件與 codelab 也都用 Socket.io 當示範）。信令層搬的，就是上面那兩種東西的 JSON：

```
// 信令層（Socket.io）搬的訊息——示意，非可執行
A → server → B : { type: 'offer',  sdp: '<A 的 SDP offer>' }
B → server → A : { type: 'answer', sdp: '<B 的 SDP answer>' }
A → server → B : { type: 'candidate', candidate: '<A 的某個 ICE candidate>' }
B → server → A : { type: 'candidate', candidate: '<B 的某個 ICE candidate>' }
   …（candidate 邊蒐集邊送，trickle ICE，RFC 8838）
```

注意一個關鍵：**信令層只搬「協商用的後設資訊」，不碰真正的媒體**。offer/answer 和 candidate 走 Socket.io（也就是走你的伺服器）；但一旦協商完成、ICE 配對出一條路，**音訊與視訊封包就直接在兩個 client 之間流（P2P），不經過你的 Socket.io 伺服器**。這個分界至關重要——它決定了你的伺服器要扛多少流量（後面 mesh vs SFU 那節會看到，群組通話會打破這個「媒體不經伺服器」的美好假設）。

> **我以為 vs 實際**：你大概記得「Socket.io 是我做視訊的信令」，但可能沒清楚劃出**它搬什麼、不搬什麼**。它搬的是 SDP 與 candidate（協商用的控制訊息，量很小、只在通話建立的頭幾秒密集出現）；它**不搬**音訊視訊（那是 SRTP，P2P 直走）。如果你當年以為「視訊流量會壓垮我的 Socket.io 伺服器」而緊張——一對一通話不會，因為媒體根本不經過它。真正會讓伺服器扛媒體的是**群組通話用 SFU**，那是另一回事（下面講）。

## ICE / STUN / TURN：為什麼 NAT 後不能直連

現在挖第三個問題：為什麼兩台在 NAT 後的機器不能直接連，以及 STUN、TURN 各自在解什麼。

### 問題的根：NAT 讓你「對外的你」和「對內的你」不是同一個位址

你的手機在 LetsTalk 上開視訊，它的網路位址多半是個**私有 IP**（像 `192.168.1.23`）——這個位址在你家路由器內部有意義，在公網上**不可路由**。當你的封包出門，家用路由器的 **NAT（Network Address Translation）** 會把來源位址改寫成路由器的**公網 IP + 一個對外 port**，並記住「這個對外 port ↔ 內部那台機器的某個 port」的對應，好讓回來的封包找得到路。

對端也一樣躲在它自己的 NAT 後面。於是兩邊都面對同一個困境：**我只知道自己的私有 IP，但我得告訴對方一個它能連到我的位址——而那個位址，我自己根本不知道，因為它是我的 NAT 在出門那一刻才臨時分配的。** 你不能在 SDP 裡寫 `192.168.1.23`，對方連不到那個位址。

### STUN：幫你問「我從外面看起來是什麼位址」

**STUN（Session Traversal Utilities for NAT，RFC 8489，obsoletes 5389，landscape §7）** 解的就是這個。機制簡單到優雅：你架一台公網上的 STUN server，你的 client 往它送一個封包問「你看到我的來源位址是什麼？」。這個封包出門時會經過你的 NAT、被改寫成「公網 IP + 對外 port」，STUN server 收到時看到的就是這個**被 NAT 改寫後的位址**，它把這個位址原封不動回給你。

於是你的 client 學到了「我從外面看起來是 `203.0.113.7:51234`」——這叫 **server-reflexive candidate（srflx）**。把它放進 SDP 送給對方，對方就有機會直接往這個對外位址打洞連你。STUN 的成本極低：它只是「問一下對外位址」，封包量微乎其微，之後媒體仍然 P2P 直走，**伺服器不碰媒體**。你當年架 STUN 就是為了讓絕大多數使用者能 P2P 直連、把昂貴的中繼省下來。

### 為什麼 symmetric NAT 會讓 STUN 失靈

這是你當年知道「會發生、要退 TURN」但可能沒挖透的一段。

NAT 有很多種行為。關鍵差別在**它怎麼決定對外 port 的映射**：

- **大多數 NAT（cone 類）**：映射只看「來源內部位址」。也就是說，同一台內部機器的同一個內部 port，**不管要送到哪個目的地**，對外都用**同一個** port。這意味著你用 STUN 探到的 `203.0.113.7:51234`，對 STUN server 成立，對你的通話對端**也成立**——對方往這個位址打洞，能通。

- **symmetric NAT（對稱型）**：映射是**依目的地而異（destination-dependent）**。你送往 STUN server 用的對外 port，和你送往「通話對端」用的對外 port，**是兩個不同的 port**。於是悲劇發生了：你用 STUN 探到「我對外是 `:51234`」（那是專門給 STUN server 那條路的映射），把它寫進 SDP；但當對端真的往 `:51234` 打過來，這個 port 對「對端 → 你」這條路**根本沒有對應的映射**，NAT 直接丟棄。**STUN 探到的位址對真正的對端無效**——這就是 STUN 在 symmetric NAT 下失靈的精確原因（landscape §7）。

打個白話的比方：cone NAT 像是「不管誰來找我，門牌都是同一個」；symmetric NAT 像是「我給快遞公司的地址、和我給你的地址，是兩個不同的暫時門牌，你拿著給快遞的那個地址來找我，按了門鈴也沒人應」。STUN server 就是那個「快遞公司」，它探到的門牌只對它自己有效。

### TURN：直連不通時，繞一台中繼

當 STUN 失靈、雙方怎麼打洞都連不通，就只剩最後一招——**TURN（Traversal Using Relays around NAT，RFC 8656，obsoletes 5766，landscape §7）**。

TURN 的思路是放棄 P2P 直連，改成**繞一台公網上的中繼伺服器**：你和對端**都**連到這台 TURN server（連出去總是能通的，因為是你主動往公網一個固定位址發起），然後讓 TURN server 在中間**轉發**你們的媒體封包。你的 client 從 TURN server 拿到的中繼位址叫 **relay candidate**，寫進 SDP 給對方——對方往這個中繼位址送，TURN server 收到再轉給你。

這條路**一定能通**（這也是它叫「last resort」的原因），但代價實打實：

- **媒體不再 P2P，要繞中繼**——本來「伺服器不碰媒體」的美好假設破了。一通走 TURN relay 的視訊，**全部音訊視訊流量都過你的 TURN server**。
- **頻寬成本**：中繼意味著同一份媒體要進 TURN server 一次、出去一次，雙向流量都算你的伺服器頻寬。對一款 500K+ 使用者的 app，如果很大比例的通話走 relay，這筆頻寬帳很可觀——這正是為什麼你當年**自架 coturn** 而不是用第三方 TURN 服務（第三方 TURN 是按 GB 計費的，貴）。
- **延遲上升**：本來 A↔B 直線，現在變成 A→TURN→B，多一跳，延遲增加。

所以整個 ICE 的策略其實是一個**漏斗**：能用本機位址（host candidate）直連最好（同網段、或公網直連）；不行就用 STUN 探對外位址打洞（srflx，絕大多數 NAT 適用、便宜）；**再不行才退到 TURN relay**（一定通、但貴又慢）。你架 STUN 是為了讓漏斗的前段盡量接住流量，TURN 只接住那一小撮「怎麼都直連不通」的——而那一小撮的代表，就是 symmetric NAT 與嚴格防火牆後面的使用者。

coturn（landscape §8，2026 仍活躍維護、最新約 4.13.1／2026-06-15，**版本號易腐，引用標日期**）是事實上的開源 TURN/STUN server——它**一套就同時當 STUN 與 TURN**，所以你當年「設 TURN/STUN」其實多半是架一台 coturn 搞定兩件事。

> **我以為 vs 實際**：你大概把 STUN 和 TURN 當成「兩個都要架的 NAT 穿透元件」，平行地理解它們。實際上它們是**漏斗的兩層、解的是不同強度的問題**：STUN 只是「問一下對外位址」、媒體仍 P2P、近乎免費；TURN 是「直連徹底失敗時繞中繼」、媒體全走伺服器、要錢要延遲。把它們混為一談，你就會算錯成本——TURN 的用量（走 relay 的通話比例）才是你頻寬帳的大頭，STUN 幾乎不花錢。

## SFU vs MCU vs mesh：群組為什麼非要伺服器不可

前面講的 P2P 直連，是**一對一**的世界——媒體在兩個 client 之間直走，伺服器只搬信令。但你做的是**群組通話**。一旦人數 > 2，事情會爆炸，這就是 Jitsi Videobridge 存在的理由。

### mesh：每個人連每個人，N² 條流

最直覺的群組做法是 **mesh**：N 個人，每兩個人之間都建一條 P2P 連線。聽起來很「去中心化」很美，但算一下流量：

- 每個參與者要**把自己的上行視訊送給其他 N−1 個人**——也就是每人要編碼並上傳 **N−1 路**。
- 全場總連線數是 N(N−1)/2 條，總流數量級是 **N²**。

對你的群組 8 人場景，手算一下上行：

```
mesh，8 人：
  每人上行 = N−1 = 7 路（要分別送給其餘 7 人）
  全場上行流總數 = 8 × 7 = 56 路
  每個 client 都要同時編碼 7 份自己的視訊、上傳 7 份
```

問題不在伺服器（mesh 沒伺服器轉媒體）——**問題在每個 client 的上行頻寬與 CPU**。手機要同時編碼並上傳 7 路視訊，上行頻寬通常先爆（家用上行本來就比下行小），然後 CPU 與電量跟著爆。mesh 大約撐到 3–4 人就到頂（搜尋到的 2026 業界共識也是這個數）。**8 人 mesh 不可行**，不是伺服器扛不住，是每個使用者的裝置扛不住。

### MCU：伺服器把所有人混成一路——但很貴

另一個極端是 **MCU（Multipoint Control Unit）**：伺服器把所有人的視訊**解碼、混合（mixing）成一張合成畫面、再重新編碼**，每個參與者只收**一路**混好的流。client 端極輕（只收一路、只解一路），但——

伺服器要**即時解碼 N 路 + 混版面 + 重新編碼**，而且常常要為不同參與者的不同需求編出多種解析度。**轉碼（transcoding）是極吃 CPU 的工作**，人越多 CPU 需求越高。這就是 MCU 為什麼貴：它把成本從 client 端搬到了伺服器端，而且是搬到了**最昂貴的那種伺服器運算（即時轉碼）**上。對你這種要控成本的場景，MCU 是最糟的選擇。

### SFU：伺服器只轉發、不轉碼——Jitsi Videobridge 走這條

中間的甜蜜點是 **SFU（Selective Forwarding Unit）**，**Jitsi Videobridge（JVB）走的就是這條**（landscape §5：官方 repo 自稱「WebRTC compatible video router or SFU」）。

SFU 的關鍵字是**轉發不轉碼（forward, don't transcode）**：

- 每個參與者把自己的上行**只送一路**給 SFU（不是 mesh 的 N−1 路）。
- SFU **不解碼、不混版面、不重新編碼**——它只是個聰明的**路由器**：收到 A 的流，原封不動（選擇性地）轉發給其他想看 A 的人。
- 「選擇性（selective）」是它比笨轉發聰明的地方：它能依各接收端的頻寬與需求，決定轉發哪一路、哪個解析度（配合 simulcast，發送端送多種解析度讓 SFU 挑），但**它自己不做轉碼那種重活**。

手算同樣的 8 人，對照 mesh：

```
SFU（Jitsi Videobridge），8 人：
  每人上行 = 1 路（只送給 SFU）
  全場上行流總數 = 8 × 1 = 8 路   ← 對比 mesh 的 56 路
  client 端：上傳 1 路、下載至多 7 路（看版面）
  SFU：純轉發，不轉碼（CPU 遠低於 MCU）
```

每人上行從 mesh 的 7 路降到 **1 路**——這對手機的上行頻寬與電量是天差地別。代價是：**現在有一台伺服器在轉發媒體了**（不像一對一 P2P 那樣伺服器只搬信令）。這台伺服器就是 Jitsi Videobridge，也正是你當年「自架」要運維的東西。

把三者並排：

| 拓樸 | 媒體經誰 | 每人上行 | 伺服器負擔 | 適用 |
|---|---|---|---|---|
| **mesh** | P2P，不經伺服器 | N−1 路 | 無（但 client 爆） | ~2–4 人 |
| **MCU** | 伺服器混流轉碼 | 1 路 | 極高（即時轉碼） | 要單一合成流、貴 |
| **SFU** | 伺服器轉發不轉碼 | 1 路 | 中（純路由） | 群組通話主流（JVB） |

> **我以為 vs 實際**：你大概記得「群組要 SFU，因為一對一可以 P2P、多人要伺服器在中間」——對，但「為什麼非伺服器不可」的精確答案不是「多人需要一個協調者」，而是**mesh 的上行是 N−1 路、會先壓垮每個使用者的上行頻寬與裝置**。而 SFU 之所以是甜蜜點、不是 MCU，關鍵在**它轉發不轉碼**——它把「N 路上行」的負擔從每個 client 搬掉，又不像 MCU 那樣付出即時轉碼的天價 CPU。Jitsi Videobridge 是 SFU 這一句，背後就是這整套取捨。

## 選型還原：Twilio / Agora / Jitsi，與一個要校正的老兵記憶

現在回到你當年那個決定——以及那個今天已經站不住腳的理由。

### 先校正：Twilio Video「要關了」這個論點，已經被官方撤回

你大概在某個時點聽過、甚至自己講過：「Twilio Programmable Video 反正要 EOL 了，所以選它沒意義、不如自架 Jitsi。」這句話現在**不能再用了**，必須校正——這正是本書反覆強調的「老兵記憶會過時」的活教材。

事實的完整時間線（landscape §6，已用官方 changelog 核對）：

- **2023-12 → 2024-03**：Twilio 最早在 2023-12 就把 Programmable Video 列為 EOL（原訂約 2024-12 關閉），2024-03 再更新公告、把 EOL 日期延到 **2026-12-05**。兩次公告都指向「要關」。所以「Twilio Video 要關了」這個說法，**從 2023 年底到 2024-10 撤回之間曾經為真**。
- **2024-10-21**：Twilio **公開撤回**了這個決定。官方 changelog 原文：「we've reversed our earlier decision to retire Twilio Video in 2026」「Twilio Video will remain a standalone product」。
- **2026-06 現況**：Twilio Video **未關閉、仍是現役產品**，現有客戶「no action or change needed」。

所以如果你今天在面試或文件裡寫「我當年選 Jitsi 是因為 Twilio Video 要停運」——這個論點**已經不成立**了。更尷尬的是，就算當年你真的這樣想過（在 2023 年底到 2024-10 那個窗口裡，這甚至不算錯），用它當**今天**的決策理由也站不住，因為前提已被推翻。

要小心的是：這不代表你當年選 Jitsi 是錯的。**選擇對、但某一條支撐理由過時了**——這兩件事要分開。下面就重講一套**今天仍然成立**的取捨論述。

### 重講取捨：純成本、自架掌控、開源授權

把「Twilio 要關了」這條過時理由拿掉之後，你當年選 Jitsi 的**真正、且至今仍然成立**的理由是這三條：

1. **成本模型根本不同**。Twilio Programmable Video 與 Agora 都是**商用 SDK、按用量計費**（Agora 至今仍是活躍商用平台、pay-as-you-go，landscape §6）。對一款 500K+ 使用者、通話量可觀、毛利又敏感的隱私 IM，每一分鐘群組視訊都向第三方付費，會隨用量線性長大、難以控制。Jitsi 開源、自架，**沒有按量授權費**——你付的是固定的伺服器與頻寬成本，規模化後單位成本可壓得低很多。
2. **自架掌控（尤其對「隱私導向」這個定位）**。LetsTalk 是隱私導向 IM。把使用者的音訊視訊媒體交給第三方 SDK 路由，與「隱私導向」的品牌承諾有張力；**自架 Jitsi + 自架 coturn**，媒體路徑、relay 伺服器、資料落點都在你自己掌控下。這對你的定位是加分項，不只是成本問題。
3. **開源授權**。Jitsi 採 **Apache 2.0**（landscape §6），可自由自架、零授權費、可改。Twilio/Agora 是閉源商用服務，你只能用它給的 API、議它給的價。

而你當年選 Jitsi **放棄了什麼**，也要誠實列出——這是「被放棄的選項輸在哪」的反面：你放棄了「不用自己運維」。選商用 SDK，Twilio/Agora 替你扛了 SFU、TURN、跨地域節點、版本維護、擴容；選 Jitsi，這些**全部變成你的事**：你要自己運維 **Jitsi Videobridge（JVB）**（前面那台轉發媒體的 SFU 伺服器）、自己運維 **coturn**（TURN/STUN）、自己處理擴容與監控。對一個工程團隊不大的隱私 IM，這是一筆真實的長期運維負擔——但你判斷「省下的按量授權費 + 換來的掌控」值得，這個判斷在你的場景下站得住。

把四維取捨排成一張表（這也是紙上推演會用到的）：

| 維度 | Twilio Video | Agora | Jitsi（自架） |
|---|---|---|---|
| **成本** | 按用量計費、隨規模線性長 | 按用量計費（pay-as-you-go） | 零授權費；付伺服器＋頻寬（規模化後低） |
| **自運維** | 幾乎零（全託管） | 幾乎零（全託管） | **高**：自運維 JVB＋coturn＋擴容＋監控 |
| **品質/規模** | 成熟、全球節點、開箱即用 | 成熟、低延遲全球網路 | 看你自架功力；JVB 每實例約 100–300 人，靠協調層水平擴展 |
| **掌控/隱私** | 媒體經第三方 | 媒體經第三方 | **媒體路徑完全自有**（合隱私定位） |
| **授權** | 閉源商用 | 閉源商用 | **Apache 2.0 開源** |
| **2026 狀態** | 現役（2024-10 撤回 EOL） | 現役商用 | 由 8x8 維護、活躍（landscape §6） |

最後一行特別要記住：**三家在 2026 都是活著的**——Twilio Video 沒關（撤回了）、Agora 商用持續、Jitsi 由 8x8 活躍維護（Apache 2.0）。所以今天若要重做這個選型，三條路都還在，差別純粹在上面那五個維度的取捨，**不是「有沒有得選」**。

> **一個維護狀態的陷阱（landscape §6）**：判斷 Jitsi「是否還活著」時別只看它 GitHub 的 Releases 頁——jitsi-videobridge 的 tagged release 停在 2024-06，光看 release 頁會誤判已停更。實際活躍度要看 **commits**（master 分支 commit 到 2026-06，仍在改）。任何「用 release tag 判斷專案死活」的習慣都會在這裡騙到你。

## 故障模式與防禦

這套 WebRTC + Jitsi + coturn 會怎麼壞、壞了長什麼樣、怎麼防。至少一個綁定你真實的 LetsTalk 系統。

| 故障 | 壞了長什麼樣（可觀測徵兆） | 怎麼防 |
|---|---|---|
| **信令送丟，通話建不起來**（Socket.io 訊息掉了一則 offer/answer/candidate，**你的系統**） | 雙方都「在等對方」，連線狀態卡在 checking，媒體永遠不來；只發生在某些網路差的使用者 | 信令層要 ack／重傳關鍵訊息（Socket.io 有 acknowledgement）；trickle candidate 容許部分丟失（還有別條路），但 offer/answer 丟了通話必死，要重送 |
| **symmetric NAT / 嚴格防火牆下無 TURN，直連失敗**（只配了 STUN 沒配 TURN，**你的系統**） | 大多數人通話正常，但**某一群使用者**（特定電信/企業網路）100% 連不上、且只在他們那邊發生，難以重現 | **一定要配 TURN（coturn）**——STUN 只解 cone NAT，symmetric NAT 必須 relay；沒有 TURN 等於對這群使用者完全不可用 |
| **TURN relay 比例失控，頻寬帳爆掉** | coturn 流量與頻寬成本異常飆升；明明很多通話「本該能 P2P」卻都走了 relay | 監控「走 relay 的通話比例」這個 metric；比例異常高通常是 STUN 設錯、防火牆策略、或 candidate 蒐集出問題——relay 該是少數派 |
| **群組誤用 mesh，人一多就崩**（沒走 SFU、N 人 mesh） | 3–4 人還行，人一多每個 client 上行與 CPU 飆、畫面卡頓掉幀、手機發燙耗電 | 群組通話必須走 SFU（Jitsi Videobridge）；mesh 只適合 1 對 1 或極小群 |
| **JVB 單點過載**（自架 SFU，單實例參與者太多） | 某台 Videobridge CPU/頻寬打滿，上面所有會議一起卡；新會議無處可去 | JVB 每實例約 100–300 人，要靠協調層水平擴展（多 bridge）；監控每台 JVB 的負載並能調度新會議到空閒 bridge |
| **DTLS 握手失敗，ICE 通了媒體卻不來** | 連線狀態到 connected 後媒體仍不流；憑證/時鐘/中間設備干擾 | 把連線階段拆開觀測——分清是信令、ICE、還是 DTLS 斷的（呼應本章「四塊分開」）；別把所有「不通」當成同一種故障 |
| **老兵記憶誤導選型**（用「Twilio 要關了」當理由，**你的系統**） | 對外講選型故事時論點被當場戳破（Twilio Video 沒關）；決策可信度受損 | 時效性事實以 landscape 為錨、面試前重新查證；用「成本/掌控/授權」這套**不會過時**的論述，而非「對手要倒了」這種會腐的論述 |

注意這張表裡最陰險的兩個：**「沒配 TURN」** 和 **「relay 比例失控」**——它們都**只在一部分使用者身上發生**，而且你（開發者，多半在 cone NAT 的辦公網路）自己**永遠重現不出來**。你連得好好的，客訴卻一直來。這正是 ch02「觀測」軸的核心：你得有「走 relay 的通話比例」「各使用者連線成功率」這類 metric，否則 symmetric NAT 後那群人的痛苦，你第一時間根本不會知道。通用的「為什麼少數派故障最難抓」分類見《正常意外》（fail）。

## 紙上推演

### 推演一：手追兩個使用者建一通 P2P 視訊的完整握手 **[20 分鐘]** ★★

使用者 A 想跟 B 開一對一視訊。兩人各自在自家 NAT 後面，你的 LetsTalk 用 Socket.io 當信令、架了 coturn（同時提供 STUN 與 TURN）。

**問**：把這通視訊從「A 按下通話」到「媒體開始流」的完整握手序列**逐步**寫出來——(a) 信令層（Socket.io）依序搬了哪些訊息？(b) A 和 B 各自蒐集了哪幾類 ICE candidate（用 host / srflx / relay 命名）？STUN 在哪一步出場？(c) 如果兩邊都是普通 cone NAT，最後選哪條路、媒體經不經過你的伺服器？(d) 如果 A 在 symmetric NAT 後、srflx 連不通，序列怎麼變、最後媒體走哪、誰付頻寬？

### 推演二：估 8 人群組 mesh vs SFU 的上行流數 **[10 分鐘]** ★

LetsTalk 一個 8 人群組視訊。

**問**：(a) 用 mesh，每個人上行幾路？全場上行流總數幾路？(b) 用 SFU（Jitsi Videobridge），每個人上行幾路？全場幾路？(c) 哪個資源先成為 mesh 的瓶頸——伺服器還是 client？為什麼 8 人 mesh 不可行？(d) 換成 SFU 之後，瓶頸搬到哪裡去了、代價是什麼？

### 推演三：還原四維取捨表並判斷今天會不會改 **[15 分鐘]** ★★

不准翻前面那張表，自己重建一次 Twilio Video / Agora / Jitsi 在「成本、自運維、掌控/隱私、授權、2026 狀態」上的位置。

**問**：(a) 當年你選 Jitsi，這五個維度裡哪幾個是你選它的**正面理由**、哪個維度是你為此**付出的代價**？(b) 你曾用過的「Twilio 要關了」這個理由，今天哪裡錯了？(c) 如果 2026 的你重做這個選型、且 LetsTalk 規模與隱私定位不變，你會不會改選？把判斷講成一句可辯護的話。

### 推演解答

**推演一解答。**

(a) **信令層（Socket.io）依序搬的訊息**：
1. A 端 `createOffer()` 產生一份 SDP offer、`setLocalDescription()`，然後**經 Socket.io 把 offer 送給 B**（`{type:'offer', sdp:…}`，走你的伺服器）。
2. B 收到後 `setRemoteDescription(offer)`、`createAnswer()` 產生 SDP answer、`setLocalDescription()`，**經 Socket.io 把 answer 回送給 A**。
3. 在這之間（trickle ICE，RFC 8838），雙方**邊蒐集 candidate 邊經 Socket.io 互送** `{type:'candidate', …}`，不等全部蒐集完才送。

(b) **ICE candidate 的蒐集**：A、B 各自會蒐集——
- **host candidate**：本機網卡的私有位址（同網段或公網直連時可用）。
- **server-reflexive candidate（srflx）**：往 **coturn 的 STUN** 問「我對外是什麼位址」拿到的對外 IP:port。**STUN 在這一步出場**——只是探對外位址，封包量極小。
- **relay candidate**：往 **coturn 的 TURN** 要一個中繼位址（先準備著，直連失敗才用）。
每蒐集到一個就 trickle 給對方，雙方把對方的 candidate 兩兩配對、做連通性檢查（connectivity check）。

(c) **兩邊都 cone NAT**：srflx 配對能通——A 用 STUN 探到的對外位址，對 B 也有效，反之亦然，雙方打洞成功。最後選的是 **srflx ↔ srflx 這條 P2P 路徑**，**媒體（SRTP）直接在 A、B 之間流，不經過你的 Socket.io 伺服器、也不經過 coturn 的 relay**。你的伺服器只在頭幾秒搬了信令，之後完全不碰媒體——這是最省的理想情況。

(d) **A 在 symmetric NAT 後**：A 的 srflx 是「給 STUN server 那條路」的映射，對 B 無效（destination-dependent，前面機制那節）。B 往 A 的 srflx 打洞，NAT 丟棄、連不通。連通性檢查在所有「直連型」配對上都失敗，ICE 退到**最後的 relay candidate**：A 與 B 都連到 **coturn 的 TURN**，媒體改走 **A→TURN→B**。媒體此時**全程經過你的 coturn**，**頻寬由你（自架者）付**，且多一跳、延遲上升。這通就是會落進你「relay 比例」那個 metric 的通話。

**推演二解答。**

(a) **mesh**：每人上行 = N−1 = **7 路**（分別送給其餘 7 人）；全場上行流總數 = 8 × 7 = **56 路**（量級 N²）。

(b) **SFU（JVB）**：每人上行 = **1 路**（只送給 SFU）；全場上行流總數 = 8 × 1 = **8 路**。SFU 再把這些**轉發（不轉碼）**給需要的接收端。

(c) **mesh 的瓶頸先在 client，不在伺服器**（mesh 沒有伺服器轉媒體）。每個手機要**同時編碼並上傳 7 路**自己的視訊——**上行頻寬**通常先爆（家用上行小），接著 **CPU/電量**爆。所以 8 人 mesh 不可行不是「伺服器扛不住」，是**每個使用者的裝置扛不住**；mesh 大約 3–4 人到頂。

(d) **SFU 把瓶頸從「每個 client 的上行」搬到「一台伺服器」**——現在有 Jitsi Videobridge 在轉發媒體了。代價是：你得**自架並運維這台 SFU**（CPU、頻寬、擴容、單點過載風險，見故障表的「JVB 單點過載」）。但這個交換很划算：把「N 個使用者各自的 N−1 路上行重擔」換成「一台你能水平擴展、且只做轉發不轉碼的伺服器」。這正是你當年選 Jitsi（SFU）做群組的工程本質。

**推演三解答。**

(a) **正面理由**（選 Jitsi 的）：**成本**（零按量授權費，規模化後單位成本低）、**掌控/隱私**（媒體路徑完全自有，合隱私 IM 定位）、**授權**（Apache 2.0 可自架可改）。**付出的代價**：**自運維**——JVB + coturn + 擴容 + 監控全變你的事，這是選商用 SDK 本可省下的。

(b) **「Twilio 要關了」今天哪裡錯**：Twilio 確在 2024-03 宣布 Video 將 2026-12-05 EOL，但**2024-10-21 公開撤回**，Video 至今（2026-06）**仍是現役產品**（landscape §6）。所以用「對手要倒了」當選型理由——前提已被推翻、論點不成立。它是「老兵記憶會過時」的典型：曾經為真、現已失效。

(c) **2026 重做會不會改**：若規模與隱私定位不變，**多半不改、仍選自架 Jitsi**——因為支撐它的三條理由（成本模型、自架掌控/隱私、開源授權）**都沒過時、至今成立**；過時的只是「Twilio 要關了」這條我從來就不該倚賴的理由。一句可辯護的話：「我選 Jitsi 不是因為對手要倒，是因為在我這個規模與隱私定位下，自架的固定成本 + 媒體掌控，勝過商用 SDK 的按量授權費 + 便利——而這個取捨，把 Twilio 的產品狀態拿掉也依然成立。」**這句話的好處是它不會因為某家廠商的產品決策而失效。**

## 延伸閱讀

- **WebRTC connectivity（MDN）**（[MDN Web Docs](https://developer.mozilla.org/en-US/docs/Web/API/WebRTC_API/Connectivity)）：把 ICE、STUN、TURN、candidate 類型（host/srflx/relay）與 trickle ICE 講得最清楚的一篇入門權威；讀「ICE candidates」與「STUN/TURN」兩段，對照本章的漏斗模型。
- **Getting started with peer connections（webrtc.org）**（[webrtc.org](https://webrtc.org/getting-started/peer-connections)）：官方的 offer/answer 信令流程（createOffer → setLocalDescription → 經信令送 → setRemoteDescription → createAnswer），對照本章「信令搬什麼」。
- **RFC 8445 — ICE**（[datatracker](https://datatracker.ietf.org/doc/html/rfc8445)）：NAT 穿透整體框架的一手出處（obsoletes 5245，landscape §7）；不必全讀，看 candidate 蒐集與連通性檢查的概念段。
- **RFC 8489 — STUN**（[RFC Editor](https://www.rfc-editor.org/rfc/rfc8489.html)）／**RFC 8656 — TURN**（[datatracker](https://datatracker.ietf.org/doc/rfc8656/)）：STUN 與 TURN 的權威定義（分別 obsoletes 5389、5766，landscape §7）；TURN 那篇讀「為什麼需要 relay」的動機段，正是 symmetric NAT 失靈的官方版本。
- **Jitsi Videobridge（GitHub）**（[github.com/jitsi/jitsi-videobridge](https://github.com/jitsi/jitsi-videobridge)）：官方自述「WebRTC compatible video router or SFU」的一手出處（landscape §5）；判斷它死活看 **commits** 不看 releases（landscape §6 的陷阱）。
- **Twilio Video Will Remain a Standalone Product（Twilio changelog）**（[Twilio](https://www.twilio.com/en-us/changelog/-twilio-video-will-remain-a-standalone-product)）：2024-10-21 撤回 EOL 的官方原文——這是校正「Twilio Video 要關了」這個老兵記憶的一手證據（landscape §6）。
- **coturn（GitHub）**（[github.com/coturn/coturn](https://github.com/coturn/coturn)）：你當年自架的那台 TURN/STUN server，一套同時當 STUN 與 TURN（landscape §8）；版本號易腐，引用標日期。

### 本系統五軸體檢（WebRTC + Jitsi + coturn）

| 軸 | 這套系統的答案 | 代價 / 邊界 |
|---|---|---|
| **交付** | 媒體走 SRTP over UDP，**即時優先、不保證可靠**（丟封包就丟、不重傳，這對即時影音是對的——遲到的影格不如不要）；信令層（Socket.io）才需要可靠交付 offer/answer | 媒體本就容忍丟失；但**信令丟一則 offer/answer 通話就建不起來**，信令層要 ack/重送 |
| **持久** | 通話是**短暫狀態**，沒有持久化需求——掛斷即結束，沒有「重啟後續上一通」這種事 | JVB/coturn 重啟會中斷進行中的通話；無痛接續需另設計（一般直接重撥） |
| **分區** | P2P 直連時雙方各自一份媒體、無共享狀態；SFU 群組時 JVB 是媒體的匯流點，某台 bridge 分區會讓其上會議全斷 | 自架 SFU 把「群組的可用性」綁在 JVB 上——JVB 的分區域就是你的故障域 |
| **並發** | ICE 同時蒐集並測多條 candidate、選最優路徑；SFU 同時轉發多人多路——但**轉發不轉碼**，並發負擔遠低於 MCU | mesh 的並發負擔落在每個 client（N−1 路上行）；SFU 把它收斂到伺服器 |
| **觀測** | 最該盯的是「**走 relay 的通話比例**」「各使用者連線成功率」「每台 JVB 負載」——symmetric NAT 那群人的故障**你自己永遠重現不出來** | 沒有這些 metric，少數派使用者的不可用會靜默到只剩客訴；觀測必須是設計的一部分 |
