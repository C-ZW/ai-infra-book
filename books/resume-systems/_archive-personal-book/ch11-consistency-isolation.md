# ch11 — 一致性、隔離級別與複製延遲

> **本章解決什麼問題**：你調過 isolation level、處理過 deadlock、把讀導去 read replica——這三件事你都做過，但它們其實是同一個老問題的三個切面：**「誰，在什麼時候，看到什麼？」** 這是全書五個故障問句裡「分區／一致」那一軸的主場章。ch03 把鎖與交易挖到了單庫內部的並發機制，這章把鏡頭拉到「一致性」這條譜上——從最嚴格的線性一致，一路滑到最寬鬆的最終一致，沿路把隔離級別底下的 MVCC、那個你大概一直記錯的 PostgreSQL Repeatable Read、snapshot isolation 擋不住的 write skew、以及 read replica 複製延遲怎麼讓「讀己之寫」破裂，逐一挖開。它接 ch04 四服務最終一致、ch07 多裝置已讀同步留下的尾巴。分散式共識（Raft/Paxos）一句話指路 tla 書、CAP 的數學不展開、B-tree 內部一句話帶過。

## 從你已知的出發

你履歷上有三句話，當年你大概沒把它們連在一起想：

第一句，「MySQL/PostgreSQL 調校（isolation）」。在 NeoBards（Dynasty Warriors Mobile，500K+ 使用者、10K+ CCU）你動過隔離級別——多半是某段熱點寫入想降 gap lock，把那條連線從預設的 Repeatable Read 改成 Read Committed（這條線 ch03 已經鋪過）。你那時候是把它當一個「降鎖競爭的旋鈕」在轉，沒真正想過你轉的是一條**一致性與異常的光譜**：每往寬鬆走一格，你就放行了一類新的異常。

第二句，「deadlock」。ch03 那條「兩個交易以相反順序拿鎖 → 互等」的死結你解過，靠的是統一鎖獲取順序。但你大概沒看過資料庫**怎麼偵測**那個環、為什麼它要先等一秒才動手、選誰當犧牲者回滾。

第三句，「read replica」。你用 read replica 把讀流量從主庫分出去，降主庫負載——這是教科書級的水平擴展。但你大概踩過（或差點踩過）一個鬼故事：**使用者剛改完資料、頁面一刷新，改的東西不見了。** 不是 bug，是他那次讀剛好落到一個還沒追上主庫的 replica 上。你那時可能把它當成「偶發的詭異現象」糊弄過去了。

這三句話底下是同一個問題：**在一個有並發、有多版本、有複製延遲的系統裡，「現在的狀態」根本不是一個單一的、所有人都同意的東西。** 不同的交易、不同的副本、不同的時刻，看到的是不同版本。隔離級別、MVCC、複製延遲，全都是在回答「你這次讀，看到的是哪個版本？」

但你可能沒想過的是：**「一致性」不是一個你開或關的開關，是一條連續的譜。** 你以為你在「強一致」和「最終一致」之間二選一，其實中間還站著好幾級——線性一致、因果一致、讀己之寫……它們各自保證了什麼、放棄了什麼，而你當年那些決定（改隔離級別、導讀去 replica），每一個都是在這條譜上挑了一個點。這一章要把整條譜攤開，讓你看清楚自己當年站在哪、又不小心滑到了哪。

## 一致性是一條譜，不是一個開關

先把「一致性」這個被你濫用的詞拆乾淨。它在兩個完全不同的層面上被使用，老兵最容易把它們攪在一起：

- **交易隔離（isolation）的「一致性」**：單一資料庫內部，並發交易彼此能看到對方多少中間態。這是 ACID 的 I，由隔離級別管。下一節挖。
- **分散式複製（replication）的「一致性」**：多個副本之間，一筆寫入多快、以什麼順序變得對所有讀者可見。這是 CAP / PACELC 的 C。本節先談這個。

本節講的是後者這條譜。把它從最嚴格排到最寬鬆，你會看到一個清楚的階梯，每往下一格就便宜一點、也危險一點：

```
   嚴格 ┌────────────────────────────────────────────┐ 貴・慢・難擴展
        │ 線性一致（linearizable）                   │
        │  寫一旦完成，之後所有讀都看得到，           │
        │  全系統像只有一份資料、有個全域即時時鐘     │
        ├────────────────────────────────────────────┤
        │ 因果一致（causal）                         │
        │  有因果關係的操作，順序所有人一致；         │
        │  無因果關係的，誰先誰後可以各看各的         │
        ├────────────────────────────────────────────┤
        │ 讀己之寫（read-your-writes）               │
        │  你自己寫的，你之後一定讀得到；             │
        │  別人寫的，你不保證馬上看到                 │
        ├────────────────────────────────────────────┤
        │ 最終一致（eventual）                       │
        │  只要停止寫入，所有副本「最終」會收斂同值； │
        │  中間任何時刻可能各看各的、可能讀到舊值     │
   寬鬆 └────────────────────────────────────────────┘ 便宜・快・好擴展
```

逐級用白話過一遍，每一級配一個你的系統：

**線性一致（linearizability）**：最強。它假裝整個分散式系統只有一份資料、有一個全域的即時時鐘。一旦某個寫入「完成」（回了 ack），這之後發起的任何讀，**保證**看得到這個寫（或更新的值），不管讀的是哪台機器。你 NeoBards 的**單一 RDS 主庫**在不考慮 replica 時，大致就是這個保證——你寫進去 commit 了，下一個讀（從主庫）一定讀得到。代價：它需要副本之間協調（同步複製或共識），每個寫都要等協調完成，**慢、且在網路分區時得犧牲可用性**（這就是 CAP 那一刀，本章稍後談）。

**因果一致（causal consistency）**：弱一點，但常常夠用。它只保證**有因果關係**的操作順序對所有人一致。回到 ch07 你的即時通知：使用者 B 發了訊息「在嗎？」，使用者 A 回「在」——這兩則訊息有因果關係（A 的回覆「因為」看到了 B 的問句）。因果一致保證**沒有人會看到「在」出現在「在嗎？」之前**。但兩個**互不相干**的人各發各的訊息，誰先顯示誰後顯示，不同裝置上可以不一樣——這沒破壞因果，因為它們之間本來就沒有因果。因果一致比線性一致便宜（不需要全域時鐘），又擋掉了最讓人困惑的「果在因之前」的錯亂。

**讀己之寫（read-your-writes）**：再弱一點，但它守住一條使用者**體感上最在乎**的底線——**你自己剛寫的東西，你自己一定讀得到。** 別人寫的你不保證馬上看到無所謂，但「我剛改的設定怎麼不見了」是會讓使用者抓狂的。這一級正是 read replica 複製延遲最容易打破的那一格，本章最後一節整節在講它。

**最終一致（eventual consistency）**：最弱。它只保證一件事——**只要你停止寫入，所有副本「最終」（一段不保證多長的時間後）會收斂到同一個值。** 中間任何時刻，不同副本可以各看各的、可以讀到任意舊的版本。這正是 ch04 你那條結算 pipeline 跨 guild/player/inventory/mail 四服務的一致性模型：沒有分散式交易把四個服務框成原子，你靠的是「補償 + 重試 + 對帳」讓它們**最後**對得起來——中間那個「玩家收到 coins 但背包還沒加道具」的窗口，就是最終一致的「最終還沒到」的具體長相。

把這條譜記在心裡，你會發現一件解放的事：**你從來不是在「強一致」和「最終一致」之間做一次性的二選一。** 你是針對每一條讀寫路徑，在這條譜上挑一個點——結算 pipeline 挑最終一致（換吞吐與解耦），使用者改自己的個人資料挑讀己之寫（守住體感底線），扣錢轉帳挑線性一致（不能有半點含糊）。**同一個系統裡，不同路徑站在譜上不同的點，這才是真實世界的樣子。**

> 一句話收束：一致性的「強弱」，本質是**「你願意讓讀者看到多舊、多亂的版本」**。越強＝看到的越新越一致，但越貴越慢越難在分區下存活。這條譜的每一格，下面幾節會逐一對應到你動過的具體機制。

## 隔離級別：四級各防什麼異常

現在縮回單庫內部，談 ACID 的 I。隔離級別不是用「一致性多強」描述的，是用**「它防住了哪幾類異常」**反向定義的——SQL 標準（ANSI SQL-92）就是這樣訂的，列出三種異常，然後說「防住越多種＝隔離級別越高」。

三種經典異常，每一個配一個你會在 NeoBards 碰到的場景：

| 異常 | 白話 | NeoBards 場景 |
|---|---|---|
| **髒讀（dirty read）** | 你讀到了**別人還沒 commit** 的中間態，而對方後來 rollback 了——你讀到一個從未真正存在過的值 | 交易 A 正在發獎（coins +100 還沒 commit），交易 B 讀到了這個 +100，然後 A 失敗回滾——B 以為玩家有那筆錢，其實沒有 |
| **不可重複讀（non-repeatable read）** | 同一個交易裡，**同一行讀兩次，值不一樣**——因為中間別人 commit 了一筆改動 | 結算交易讀了玩家 coins = 500，做了點事，再讀一次變 600（別的交易中途加了錢 commit），同一交易內前後不一致 |
| **幻讀（phantom read）** | 同一個交易裡，**同一個範圍查詢跑兩次，行數變了**——因為中間別人 `INSERT` 了符合條件的新行 | 結算交易第一次數「公會排名前 10 的玩家」有 10 人，發到一半再數變 11 人（有人中途達標插進來了） |

四個隔離級別，就是用「擋住前幾種」來定義的（越往下擋越多、越嚴格）：

```
   級別                     髒讀     不可重複讀   幻讀
   ─────────────────────────────────────────────────
   Read Uncommitted         可能      可能        可能
   Read Committed           擋        可能        可能
   Repeatable Read          擋        擋          可能（標準如此，下節有大坑）
   Serializable             擋        擋          擋
   ─────────────────────────────────────────────────
   越往下：擋掉越多異常，但並發代價越高
```

`Read Uncommitted` 幾乎沒人用（連髒讀都不擋，等於沒隔離）。真正每天在用的是中間那兩級——`Read Committed` 和 `Repeatable Read`。而**這兩級正是兩大資料庫預設值不同、且藏著一個全書最容易記錯的對立事實的地方**。下一節整節在挖這個坑。

這裡先埋一個重點，下面會引爆：**上面這張表是 SQL 標準的「異常」框架，而這個框架本身有個漏洞——它只列了三種異常，但真實世界還有第四種它沒列、`Repeatable Read` 也擋不住的異常，叫 write skew。** 這就是為什麼「我升到 Repeatable Read 了應該很安全」會在某些場景下背叛你。先記住這個伏筆。

## 那個你一定記錯的坑：PostgreSQL 的 RR 不是你以為的 RR

這一節值得你停下來，因為你大概對它深信不疑、而它是錯的。landscape §15 把兩個極易被混為一談的對立事實並排——我直接把它們攤在你面前：

### 事實一：兩大資料庫的「預設」就不一樣

```
   ┌──────────────────────┬──────────────────────────┐
   │  PostgreSQL          │  MySQL / InnoDB          │
   ├──────────────────────┼──────────────────────────┤
   │  預設 Read Committed │  預設 Repeatable Read    │
   └──────────────────────┴──────────────────────────┘
```

**PostgreSQL 預設 Read Committed；MySQL/InnoDB 預設 Repeatable Read**（2026-06 查證，PostgreSQL 官方文件、MySQL 8.4 Reference Manual）。如果你 NeoBards 用 MySQL、又碰過 PG，這個預設差異本身就是個陷阱：同一段「沒指定隔離級別」的程式碼，搬到另一個庫上，**行為會變**。在 PG 上每條 statement 看到的是各自最新的快照（Read Committed），在 MySQL 上整個交易看到的是同一個快照（Repeatable Read）。這已經夠讓人記憶串線了，但真正的坑在下一條。

### 事實二：PG 的 Repeatable Read 其實是 Snapshot Isolation——不防 write skew

你大概以為「Repeatable Read 就是 SQL 標準那個 Repeatable Read，擋髒讀、擋不可重複讀、不擋幻讀」。對 PG 來說，這個理解**是錯的，而且錯的方向很危險**。

PostgreSQL 官方文件白紙黑字寫著（2026-06 查證）：**PG 的 Repeatable Read「是用一種學術界叫 Snapshot Isolation 的技術實作的」。** Snapshot Isolation（SI）的意思是：交易一開始就拍一張整個資料庫的**快照**，整個交易期間所有讀都看這張快照、看不到任何之後別人 commit 的改動。

這帶來一個比標準 RR **更強**的副作用——它連幻讀都順手擋掉了（因為你看的是固定快照，別人 INSERT 的新行根本不在你的快照裡）。聽起來棒透了，標準只要求 RR 擋兩種異常，PG 的 RR 白送你第三種。

**但 SI 有它自己的、SQL 標準那張表根本沒列的漏洞——它擋不住 write skew（寫偏序）。** PG 官方文件同樣明說：在 Repeatable Read 級別，**「Serialization Anomaly 仍是 Possible」**。這個「Serialization Anomaly」就是 write skew 的正式名字。換句話說：

> **PG 的 Repeatable Read 給你的，比 SQL 標準的 RR 多了「擋幻讀」，但少了「擋 write skew」——它根本不是標準定義的那個 RR，它是 Snapshot Isolation。**

write skew 是什麼、為什麼 SI 擋不住，下一節整節在講（它是本章最反直覺的機制）。這裡你只要先吞下這個校正：**「我升到 Repeatable Read 了，所以我的並發很安全」是一句危險的話。** 在 PG 上，RR 給你 snapshot isolation 的保證，而那個保證有一個 write skew 大小的洞。

### 對照：MySQL 的 RR 是另一種東西（locking + MVCC 混合）

把 PG 和 MySQL 的 RR 並排，你會看到它們**名字一樣、本質不同**——這正是 landscape §15 警告「最容易被混為一談的對立事實」：

```
   ┌────────────────────┬──────────────────────────────────────┐
   │  PG 的 RR          │  純 snapshot isolation               │
   │                    │  讀看固定快照、不上鎖                │
   │                    │  靠「快照裡沒有新行」擋幻讀          │
   │                    │  → 但擋不住 write skew               │
   ├────────────────────┼──────────────────────────────────────┤
   │  MySQL/InnoDB 的 RR│  locking + MVCC 混合                  │
   │                    │  一般讀走 MVCC 快照                  │
   │                    │  鎖定讀（FOR UPDATE）用 next-key lock│
   │                    │  （record + gap）上鎖擋別人插進範圍 │
   │                    │  → 靠 gap lock 擋幻讀               │
   └────────────────────┴──────────────────────────────────────┘
```

兩邊都叫 Repeatable Read、都「擋幻讀」，但**擋的手段本質不同**：PG 純靠「你看的是舊快照，新行不在裡面」（被動、不上鎖），MySQL 靠 **next-key lock**（record lock + 前方 gap lock）**主動上鎖**、不准別人 `INSERT` 進你掃描的範圍（這就是 ch03 那個「gap lock 卡住一個碰都沒碰同一行的 `INSERT`」的反直覺現象的來源）。

但有一件事**兩邊一樣**，而且是這節的關鍵收口：**MySQL 的 RR 同樣擋不住 write skew。** Jepsen 的 MySQL 8.0.34 分析（2026-06 查證）講得更狠：MySQL 的 RR **既不滿足標準 Repeatable Read、也不算 Snapshot Isolation**——Jepsen 明確否定「MySQL RR 等於 SI」這個常見說法，因為它比 SI 更弱，會出現 write skew（G2-item）、read skew（G-single）、甚至 lost update 這些 SI 本該擋掉的異常，而且在單節點、無故障下就照樣發生。所以結論是：

> **不管 PG 還是 MySQL，「升到 Repeatable Read」都擋不住 write skew。要擋 write skew，你得升到 Serializable。**

### 你的「我以為 vs 實際」

| 你大概記得的 | 實際上（2026-06 查證） |
|---|---|
| PG 和 MySQL 預設都差不多 | **PG 預設 Read Committed、MySQL 預設 Repeatable Read**——同一段沒指定隔離級別的碼，跨庫行為會變 |
| Repeatable Read 就是標準那個 RR（擋髒讀、擋不可重複讀、不擋幻讀） | **PG 的 RR 是 Snapshot Isolation**：連幻讀都擋，但 Serialization Anomaly（write skew）仍 Possible |
| 升到 Repeatable Read 我的並發就安全了 | RR **擋不住 write skew**（PG、MySQL 都一樣）；要擋它必須升 Serializable |
| PG 和 MySQL 的 RR 是同一個東西 | 名字一樣、本質不同：PG 純 snapshot（被動不上鎖），MySQL locking+MVCC（用 next-key lock 主動上鎖擋幻讀） |

## write skew：兩個交易各自合法，合起來違規

這是本章最值得你慢讀的一節，因為它顛覆「我用了快照所以每個交易看到的都是一致的，怎麼會錯」這個直覺。

### 機制：快照的盲點

snapshot isolation 給每個交易一張一致的快照——交易 A 看不到 B 沒 commit 的東西，B 也看不到 A 的。**問題就出在這個「互相看不到」。** 當兩個交易：

1. 各自**讀**了一組互相重疊的資料；
2. 各自**根據自己讀到的（舊）快照**做了決定；
3. 各自**寫**了不同的行（所以沒有寫-寫衝突，SI 不會擋）；
4. 但兩個寫**合起來**違反了一個橫跨它們讀集合的約束。

每個交易單獨看都合法（在它的快照裡，約束沒被破壞）；但因為它們**各自基於一個沒看到對方的快照**做決定，合起來就違規了。SI 擋的是「寫同一行」（lost update 那類），但 write skew **寫的是不同行**，所以 SI 的衝突偵測完全看不到它。

### 經典例子：醫院值班

最乾淨的例子（PostgreSQL 官方文件與學界都用它）：醫院規定**任何時刻至少要有一名醫生值班**。現在 Alice 和 Bob 兩名醫生都在值班，兩人都想請假。

```
   約束：on_call 的醫生數 ≥ 1（任何時刻至少 1 人值班）
   初始：Alice = on_call, Bob = on_call（共 2 人）

   時刻   交易 A（Alice 請假）              交易 B（Bob 請假）
   ─────────────────────────────────────────────────────────
   t0     BEGIN（拍快照：看到 2 人值班）
   t1                                       BEGIN（拍快照：也看到 2 人值班）
   t2     SELECT count(*) WHERE on_call
          → 2 人，「夠，扣我一個還剩 1」
   t3                                       SELECT count(*) WHERE on_call
                                            → 2 人，「夠，扣我一個還剩 1」
   t4     UPDATE Alice SET on_call=false
   t5                                       UPDATE Bob SET on_call=false
   t6     COMMIT  ✔                         （A 寫的是 Alice 那行）
   t7                                       COMMIT  ✔（B 寫的是 Bob 那行）
   ─────────────────────────────────────────────────────────
   結果：on_call 的醫生數 = 0  ✗  約束被違反，沒人值班
```

看清楚這四個關鍵：

1. **t2 和 t3 兩個交易各自讀到「2 人值班」**——因為它們的快照都拍在對方寫入之前，誰也沒看到對方正要請假。
2. **各自在自己的快照下，判斷「請我的假是合法的」**（扣一個還剩一個）——這個判斷在它各自的世界裡完全正確。
3. **A 寫 Alice 那行、B 寫 Bob 那行——兩個寫沒碰同一行**，所以 SI 的「寫-寫衝突偵測」完全沒觸發，兩個都順利 commit。
4. **合起來**，兩個人都請了假，值班人數變 0——約束被破壞，而**這個結果在任何「一個交易先跑完、另一個再跑」的串行執行下都不可能發生**（第二個跑的人會讀到「只剩 1 人」，於是不敢請假）。

這就是 write skew 的本質：**每個交易在自己的快照下都合法，但這組快照「整體」對應不到任何一個合法的串行順序。** 它擋不住，是因為 SI 的設計前提是「只要不寫同一行就沒衝突」——而 write skew 恰恰是「不寫同一行、卻有邏輯衝突」的反例。

### 你的系統裡的 write skew

別以為這只是醫院教科書例子。回到 NeoBards，任何「讀一個聚合、根據它決定要不要寫」的並發路徑都可能 write skew：

- **公會人數上限**：公會規定最多 50 人。兩個玩家同時申請加入（此刻 49 人），兩個交易各自讀「49 < 50，可以加」，各自 INSERT 自己那筆會員——兩筆寫不同行、SI 不擋，commit 後變 51 人，**超員**。
- **最後一件道具的限購**：活動限購、庫存只剩 1，兩個玩家同時下單，各自讀「庫存 1 > 0，可買」，各自寫自己的訂單——超賣。

這些在 ch03 你可能用「鎖」擋過（`SELECT ... FOR UPDATE` 把那一行鎖死），但如果你**只升隔離級別到 Repeatable Read 卻沒上鎖**，你以為安全了，其實 write skew 從門縫鑽進來。

### 怎麼真的擋住

三條路，代價遞增：

1. **手動上鎖**：`SELECT ... FOR UPDATE` 把你「讀來做決定」的那些行（或一個代表性的鎖行）顯式鎖住，把 write skew 退化成寫-寫衝突（有人得等）。便宜、可控，但要你**自己想到**哪裡需要鎖——漏一個就破功。
2. **物化衝突**：人為造一個兩邊都要寫的行（比如一個「公會人數計數器」行），讓兩個交易都去 `UPDATE` 它，把隱性的邏輯衝突變成顯性的寫-寫衝突。
3. **升到 Serializable（讓資料庫幫你想）**：下一節。讓資料庫自動偵測 write skew 這類危險結構並中止其中一個——代價是它比較貴、而且**你的應用必須準備好 retry**。

## Serializable：PG 的 SSI 與「你必須 retry」的代價

`Serializable` 是隔離級別的天花板——它保證並發交易的結果**等同於某個串行（一個接一個跑）的順序**。這正好就是 write skew 的解藥：既然串行下不可能 write skew，那「保證等同串行」就把 write skew 擋死了。

問題是「怎麼做到等同串行」。最笨的辦法是真的把所有交易串起來一個一個跑（或用粗粒度鎖逼近），那慢到不能用。PostgreSQL 用的是聰明得多的 **SSI（Serializable Snapshot Isolation）**（2026-06 查證，PostgreSQL 官方文件）：

> **SSI = 在 Snapshot Isolation 之上，加一層「危險結構偵測」。** 它讓交易照樣跑 snapshot isolation（讀快照、不互相阻塞、快），但同時用 **predicate locking** 追蹤每個交易的**讀寫依賴**——誰讀了誰可能寫的範圍。當它偵測到一組依賴形成了「可能導致非串行化結果」的危險結構（學術上叫 dangerous structure，兩條一進一出的 rw-依賴邊），它就在 commit 時**中止其中一個交易**，丟出錯誤：`could not serialize access due to read/write dependencies`。

幾個你必須記住的後果：

- **它樂觀、不上你的鎖**：SSI 不像 `SELECT FOR UPDATE` 那樣先鎖再做，它讓你先跑，事後偵測、必要時中止。所以正常路徑上它不增加阻塞——這是它相對「粗暴上鎖」的優勢。
- **first committer wins**：write skew 的兩個交易，先 commit 的那個贏，後 commit 的那個被中止回滾。
- **應用程式必須準備 retry**：被中止的交易**整個白做**，你的應用收到那個序列化失敗錯誤後，必須**重跑整個交易**。這不是可選的——**用 Serializable 而不寫 retry 迴圈，等於沒用**，因為偶發的中止會直接變成使用者看到的錯誤。（而 retry 又把你帶回 ch10：重試必須冪等，否則重跑一筆「發獎」會發兩次。一致性與交付在這裡咬尾巴。）
- **它比其他級別貴**：predicate locking 要追蹤讀寫依賴，有記憶體與 CPU 開銷；高衝突場景下中止率上升，retry 放大負載。

所以 Serializable 不是「最安全所以全開」的免費選項。它是一個**取捨**：你拿「自己想清楚每個 write skew 在哪、手動上鎖」的心智負擔，換成「讓資料庫自動擋、但我得接受偶發中止並寫好 retry」的運維負擔。對 write skew 多、又難一一手動上鎖的關鍵路徑（金流、限額），Serializable 划算；對單純的高吞吐寫入，它的中止 + retry 成本可能比你 ch03 那套「縮短持有時間 + 統一順序」還貴。

```
   隔離級別        擋的異常                      代價
   ──────────────────────────────────────────────────────────────
   Read Committed  髒讀                          最便宜，放行最多異常
   Repeatable Read 髒讀 + 不可重複讀             中等；PG/MySQL 都=SI，
   (= Snapshot Iso) （PG/MySQL 連幻讀也順手擋）    擋不住 write skew ←坑
   Serializable    全部，含 write skew           最貴；PG 用 SSI，
   (PG = SSI)                                     必須寫 retry、predicate
                                                  locking 有開銷
```

## MVCC：讀不擋寫、寫不擋讀，靠的是多版本

上面一直在用「快照」這個詞，現在把它底下的引擎拆開——**MVCC（Multi-Version Concurrency Control，多版本並發控制）**。PG 和 InnoDB 都用它，它是「snapshot isolation」能成立的物理基礎。

### 核心招式：不覆寫，留舊版本

傳統的鎖式並發，讀和寫會互相擋：你要讀一行、它正被寫，你得等。MVCC 用一個漂亮的招式繞開：**寫入時不直接覆蓋舊值，而是造一個新版本，舊版本留著。** 每個版本帶一個「哪個交易、什麼時候寫的」的標記。於是：

- **讀**：根據你交易的快照（一個「我能看到哪些交易的 commit」的判據），挑出**對你可見的那個版本**來讀——可能是當前最新的，也可能是一個舊版本（如果最新版是某個你看不到的、還沒 commit 或在你快照之後 commit 的交易寫的）。
- **寫**：造新版本。

結論就是那句你大概聽過、但沒想過為什麼的話：

> **MVCC 下，讀不擋寫、寫不擋讀。** 讀的人各自看自己快照裡的版本，寫的人造新版本——兩邊井水不犯河水，不用互等。（寫-寫之間還是要擋，兩個交易不能同時改同一行的最新版，這是 ch03 行鎖的事。）

這就是為什麼 ch03 你那些「降鎖競爭」的努力，主要是在跟**寫-寫**競爭搏鬥——因為 MVCC 已經幫你把讀-寫競爭消掉了。你那條 RDS 上排隊的交易，排的是寫鎖的隊，不是讀的隊。

### 代價：舊版本要有人清，長交易讓它清不掉

天下沒有白吃的多版本。舊版本不會自己消失——它得留到「沒有任何還活著的交易可能還要讀它」為止。清理舊版本：

- 在 **PostgreSQL** 是 **VACUUM**（autovacuum 在背景跑）。
- 在 **InnoDB** 是 **purge**（清 undo log 裡的舊版本）。

問題來了：**一個跑很久的長交易，會卡住清理。** 因為只要這個長交易還活著，它的快照就「可能還要讀」某些舊版本，於是這些舊版本**不能被清**。一個開了半小時忘了 commit 的交易（或一個慢吞吞的分析查詢），會讓整個資料庫的舊版本**堆積**：

- **PG 的表膨脹（bloat）**：dead tuple 清不掉，表和索引越長越肥，掃描越來越慢，磁碟越吃越多。
- **InnoDB 的 undo log 膨脹 / history list 變長**：舊版本鏈越拉越長，連帶讀舊版本要沿著版本鏈走更多步，越來越慢。

這直接呼應 ch03 那個「批次太大造出超大鎖 / 巨型交易」的警告——**長交易不只凍結鎖，還凍結 MVCC 的垃圾回收。** 你當年「拆交易、縮短持有時間」省的不只是鎖等待，還順手讓 VACUUM/purge 跟得上、表不膨脹。這是同一個美德的兩個受益面。

> **我以為 vs 實際**：你大概以為 MVCC 是「讀更快的優化」。它是，但它的真正身份是「讓快照隔離成立的機制」——而它的暗面是**舊版本的清理債務**，這筆債由長交易引爆，由 VACUUM/purge 償還。

## deadlock 偵測：等待圖找環、挑一個犧牲者回滾

ch03 你解過死結（兩個交易相反順序拿鎖 → 互等），靠統一鎖獲取順序**預防**。這節補上你沒看過的另一半：當預防失敗、死結真的發生了，資料庫**怎麼偵測並打破它**。

機制就一句話，但每個字都重要——**建一張等待圖（wait-for graph），找環，挑一個犧牲者回滾**：

1. **等待圖**：節點是交易，「A 在等 B 持有的鎖」就畫一條 A→B 的邊。
2. **找環**：從一個「拿不到鎖、開始等」的交易出發，沿著「它在等的鎖被誰握著、那個人又在等誰」一路走下去——如果走回起點（形成環），就是死結。ch03 那個 T1→T2→T1 的互等，在圖上就是一個兩節點的環。
3. **挑犧牲者回滾**：偵測到環，資料庫**只回滾環上的其中一個**交易（不是全部）——被選中的那個整個 rollback、釋放它的鎖，環就斷了，其他交易繼續跑。被回滾的交易由應用層收到錯誤後重試。

兩個你沒注意過的細節：

**為什麼 PG 要先等一秒才偵測？** PostgreSQL 的 `deadlock_timeout` **預設 1s**（2026-06 查證，PostgreSQL 官方文件）。意思是：一個交易拿不到鎖、開始等的時候，**不會立刻**去跑死結偵測——它先單純等 1 秒。因為**偵測本身不便宜**（要建圖、找環），而**絕大多數的「等鎖」是正常排隊、1 秒內對方就 commit 放鎖了，根本不是死結**。先等 1 秒，把「正常排隊」濾掉，只對「等超過 1 秒、很可能真有環」的情況才花力氣建圖。這是個漂亮的「樂觀延遲」設計：**別為了抓罕見的死結，懲罰常見的正常等待。**（對照 ch03：InnoDB 預設是發現等待立即跑死結偵測，並有 `innodb_lock_wait_timeout` 預設 50s 作為兜底超時——兩套機制取捨不同。）

**挑誰當犧牲者？** 通常挑「回滾代價較小」的那個（改動較少、做的工作較少的交易），讓白做的工作最少。ch03 提過 InnoDB 挑「改動較少者」，PG 也有自己的啟發式。但無論挑誰，**被選中的交易整個白做**——這就是死結的真實成本：不只當下那個錯誤，是一整個交易的 CPU + I/O 白燒，加上應用層重試的又一輪（ch03 那條「等待 → 重試 → 驚群」的 CPU 因果鏈，死結是其中一條支流）。

```
   deadlock 偵測流程
   ─────────────────────────────────────────────
   交易拿不到鎖、開始等
            │
            ▼
   ┌────────────────────┐
   │ 等 deadlock_timeout │  PG 預設 1s
   │ （別急著偵測，多數  │  （多數等待是正常排隊，
   │   等待是正常排隊）  │    對方很快放鎖）
   └─────────┬──────────┘
             ▼
   ┌────────────────────┐
   │ 建 wait-for graph   │  節點=交易，邊=「A 等 B」
   │ 找環                │
   └─────────┬──────────┘
        環？ │
      ┌──────┴──────┐
      ▼ 有環         ▼ 無環
   挑犧牲者回滾     繼續等（只是慢，不是死結）
   （挑代價小的）
      │
      ▼
   應用層收到錯誤 → retry（retry 須冪等，見 ch10）
```

## read replica 複製延遲：讀己之寫怎麼破裂

最後回到你履歷上的 read replica，把本章開頭那條譜的「讀己之寫」那一格，挖到它最容易破的地方。

### 機制：非同步複製就是「主庫不等 replica」

你把 read replica 加進來的動機很正當：主庫扛不住讀寫混合的壓力，把讀流量分到一個或多個唯讀副本上，主庫專心寫。問題出在**主庫和 replica 之間怎麼同步**。預設、也是最常見的，是**非同步複製（asynchronous replication）**：

```
   寫請求 ──► 主庫（primary）
                │ ① 本地 commit、立刻回 ack 給應用
                │   （主庫不等 replica，這就是「非同步」）
                │
                │ ② 把變更（PG 的 WAL / MySQL 的 binlog）
                │    在背景慢慢送給 replica
                ▼
            replica（背景套用變更，落後主庫幾毫秒到幾秒）
```

關鍵在 ①：**主庫本地 commit 完就立刻回 ack，根本不等 replica 收到、更不等 replica 套用完。** 變更是 ② 在背景慢慢流過去的。所以在「主庫已 commit」到「replica 套用完」之間，有一個**複製延遲（replication lag）**窗口——毫秒到秒，看負載與網路。在這個窗口裡，**replica 上是舊的**。

（對照：同步複製會讓主庫等 replica ack 才回——讀己之寫不破了，但每個寫都背上一次往返 replica 的延遲，且一個慢 replica 會卡住整個主庫不能寫。多數團隊試過同步複製後都關掉了。這又是一致性譜上的取捨：往線性一致靠，就得付這個延遲與可用性的代價。）

### 故障：讀己之寫破裂

把這個延遲窗口和「讀分流去 replica」湊在一起，就引爆讀己之寫破裂——本章開頭那個鬼故事的機制版：

```
   時刻   使用者操作                        系統內部
   ───────────────────────────────────────────────────────────
   t0     改暱稱「老王」→「Chewei」        寫 → 主庫，主庫 commit，回 ack ✔
   t1     （主庫已是「Chewei」，           複製延遲窗口：變更還在背景
           replica 還是「老王」）            往 replica 流，replica 尚未套用
   t2     頁面刷新，讀個人資料             讀 → 被負載均衡導去 replica
   t3     畫面顯示「老王」  ✗              replica 還沒追上，回了舊值
   ───────────────────────────────────────────────────────────
   使用者：「我明明改了，怎麼還是舊的？」→ 讀己之寫被打破
```

使用者剛**寫**完（t0），緊接著**讀**（t2），卻讀到自己改之前的值（t3）——因為他的讀落到一個還沒追上的 replica。這不是 bug、不是資料遺失（主庫是對的，replica 最終也會追上），是**「你把讀無腦導去 replica」這個架構決定，悄悄把系統從『讀己之寫』降級到了『最終一致』**。使用者體感上這是嚴重的——「我改的東西不見了」是會讓人懷疑系統壞掉的那種錯覺。

### 怎麼補：三條路

把讀己之寫補回來，標準有三招（代價遞增、適用範圍遞減）：

| 補法 | 機制 | 代價 / 適用 |
|---|---|---|
| **讀主庫（read-from-primary）** | 對「剛寫完很可能緊接著讀」的路徑，**強制讀主庫**，不走 replica | 簡單可靠，但把分流的好處還回去了——只該對少數關鍵路徑（剛改完的個人資料、剛下的單）這樣做，不能全部 |
| **sticky / 寫後一段時間讀主** | 使用者**寫入後的一段時間窗口內**（比如複製延遲的上界，幾秒），他的讀都釘在主庫；窗口過了再放回 replica | 比「永遠讀主」省，但要會話層記「這個使用者剛寫過」；窗口設多長是賭複製延遲上界，估錯就破 |
| **版本戳 / lag-aware 路由** | 寫入回傳一個位置戳（PG 的 LSN / MySQL 的 GTID）；讀時帶上它，路由層**只把讀導去『已追到該戳』的 replica**，否則讀主或等 | 最精準（讀己之寫嚴格成立），但要中介層（ProxySQL/pgpool/Vitess 之類）支援、要傳遞版本戳——複雜度最高 |

判準很清楚：**不是所有讀都需要讀己之寫。** 別人改的東西你晚幾秒看到無所謂（那本來就只是最終一致），但**你自己剛改的東西必須立刻看到**。所以正確的做法不是「為了保險全部讀主」（那等於沒用 replica），也不是「全部讀 replica」（那會破讀己之寫），而是**精準辨認哪些讀路徑緊跟在同一使用者的寫之後**，只對那些路徑補上上面三招之一。

### 回指你的兩個系統

這條複製延遲線，正好收口前面兩章埋的伏筆：

- **ch04 四服務最終一致**：結算 pipeline 的 guild/player/inventory/mail 之間的「最終一致」，跟這裡 replica 的「最終一致」是**同一個譜上同一格**——都是「中間有窗口、各看各的、最終收斂」。差別只在前者跨服務、後者跨副本。你現在可以把它們講成同一個故事：**沒有人等所有人都同步，系統就活在最終一致裡，你要嘛容忍那個窗口、要嘛花錢（同步、讀主、版本戳）把特定路徑拉回更強的一致。**
- **ch07 多裝置已讀同步**：使用者手機把一則訊息標已讀，平板上那則訊息**該不該、多快**變已讀？這就是讀己之寫 / 因果一致在使用者**自己的多個裝置**之間的版本。手機（寫）和平板（讀）是同一個人的兩條路徑——如果已讀狀態走非同步同步、平板讀到落後的副本，使用者會看到「我手機明明點開了，平板怎麼還顯示未讀」。一模一樣的破裂，只是換了個場景。Pub/Sub 即時推那一段（ch07）解決的是「在線時即時看到」，但離線補發 + 跨裝置收斂的一致性保證，落在的正是本章這條譜上。

## 故障模式與防禦

| # | 故障 | 壞了長什麼樣（可觀測徵兆） | 怎麼防 |
|---|---|---|---|
| **1** | **誤把 Repeatable Read 當安全網（write skew 鑽進來）** | 約束被悄悄違反但 DB 沒報任何錯：公會超員（51/50）、活動超賣、值班數歸零；查每個交易單獨看都合法、湊起來才違規 | 認清 PG/MySQL 的 RR 都=SI、擋不住 write skew；對「讀聚合再決定寫」的路徑用 `SELECT FOR UPDATE` 手動上鎖、或物化衝突、或升 Serializable（須寫 retry） |
| **2** | **讀己之寫破裂**（綁定你的真實系統） | 使用者「剛改的設定/暱稱刷新後變回舊的」「剛下的單在列表裡找不到」的客訴；主庫查是對的、replica 查是舊的——複製延遲窗口的指紋 | 辨認「同一使用者寫後緊接著讀」的路徑，對它讀主 / sticky 窗口 / 版本戳路由；監控複製延遲（PG 比 LSN、MySQL 看 `Seconds_Behind_Master`），延遲超閾值告警或自動把讀拉回主 |
| **3** | **長交易凍結 MVCC 清理 → 表膨脹 / undo 膨脹** | 表和索引持續變肥、掃描越來越慢、磁碟吃緊；PG 的 dead tuple 數狂飆、autovacuum 追不上；InnoDB history list length 長到嚇人 | 砍長交易持有時間（接 ch03）；別讓分析查詢/忘了 commit 的交易長開；監控最老交易年齡（PG 的 `xmin` age / `pg_stat_activity`）、history list length |
| **4** | **Serializable 沒寫 retry** | 偶發 `could not serialize access due to read/write dependencies` 直接冒到使用者面前變成錯誤 | 用 Serializable 的地方**必須**包 retry 迴圈；retry 必須冪等（接 ch10），否則重跑「發獎」會雙發 |
| **5** | **複製延遲尖峰下把讀全導 replica → 雪上加霜** | 主庫一波寫入高峰 → 複製延遲飆到數秒 → 此刻所有讀 replica 的人全看到大幅落後的舊資料，一致性窗口從毫秒級爆到秒級 | lag-aware 路由：延遲超閾值時自動把讀切回主或拒讀舊 replica；別在沒有延遲監控的情況下盲目把讀全導 replica |

這五個故障的共同訊號，不是「報錯」——是**「沒報錯但資料/體感不對」**。這正是一致性類故障最陰險的地方：鎖競爭會給你 1205/1213 的明確錯誤（ch03），但 write skew、讀己之寫破裂、表膨脹**靜悄悄地壞**，DB 一聲不吭。你第一時間怎麼知道？靠的不是錯誤日誌，是**約束檢查**（定期對帳跑「值班數 ≥ 1 嗎、人數 ≤ 50 嗎」）和**複製延遲監控**——這把你帶回全書五軸的最後一問「觀測」（ch15 挖穿）：一致性故障的觀測，得你主動設計去抓「該成立卻沒成立的不變量」，沒人會自動報給你。

## 五軸體檢：用語言回指系統

> Part IV 是把老問題從具體系統裡抽出來挖穿的橫切章，不像 Part II/III 的系統章那樣填一張五軸體檢小表。這裡改用五軸的語言，把本章挖的「一致性」這一軸，回扣到它在前面哪些系統上以什麼形態出現過。

本章整章在挖的，是五個故障問句裡的**第 3 軸「分區／一致：誰，在什麼時候，看到什麼？」**。它在你的系統上反覆現身，每次換一張臉：

- **NeoBards 單庫寫入路徑（ch03）**：一致＝隔離級別擋的那些異常。你轉隔離級別旋鈕時，是在這軸上挑「同一交易內看到多一致的快照」。本章補上你當年沒看到的坑——RR 不防 write skew。
- **NeoBards 結算 pipeline（ch04）**：一致＝四服務的最終一致。沒有分散式交易，這軸的答案是「中間有不一致窗口，靠補償收斂」——本章把它定位成一致性譜上的「最終一致」格，跟 read replica 是同一格。
- **NeoBards read replica（本章）**：一致＝讀己之寫能不能守住。非同步複製把這軸從「讀己之寫」悄悄降到「最終一致」，本章給了三招（讀主/sticky/版本戳）把特定路徑拉回去。
- **LetsTalk 多裝置已讀同步（ch07）**：一致＝同一使用者多裝置間的因果/讀己之寫。手機寫、平板讀，是讀己之寫在「自己的多個裝置」上的變形。

一句話收束本章在五軸上的位置：**「分區／一致」這一軸沒有正確答案，只有『你願意讓讀者看到多舊、多亂的版本』這個取捨——而你過去的每個系統，都已經在這條譜上選好了位置，只是你當年沒用『一致性譜』這個詞把它們講出來。** 本章給了你這個詞。

（想把並發正確性**證明**到滴水不漏——證明你的鎖策略不會 write skew、不會死結——那是形式化方法的活，見《把系統寫成定理：TLA+》。本章只到「機制與取捨」這一層。分散式多副本怎麼達成**強一致**的共識協定（Raft/Paxos）也不在本章——那是另一座山，同樣指路 tla 書。索引 B-tree 內部怎麼讓範圍查詢與 gap lock 成立，是另一個主題，一句話帶過：next-key lock 鎖的是 B-tree 索引上的記錄與其前方間隙。）

## 紙上推演

純觀念推演，不寫可執行碼；允許貼示意時序追邏輯。

### 推演一：手推一個 write skew 時序 **[15 分鐘] ★★**

NeoBards 公會規定**最多 50 人**。某個熱門公會此刻 **49 人**。兩個玩家 P 和 Q 幾乎同時申請加入，後端各起一個交易，都在 `Repeatable Read`（snapshot isolation）下跑：

```
   交易 A（P 加入）              交易 B（Q 加入）
   BEGIN                         BEGIN
   SELECT count(*) FROM member   SELECT count(*) FROM member
     WHERE guild = G              WHERE guild = G
   -- 拿到人數，判斷夠不夠       -- 拿到人數，判斷夠不夠
   INSERT member(P, G)           INSERT member(Q, G)
   COMMIT                        COMMIT
```

問：(a) 畫出讓公會變成 51 人的交錯時序；(b) 指出為什麼 snapshot isolation 的衝突偵測沒擋住；(c) 給三種擋法，各說它的代價。

#### 推演解答

**(a) 51 人的時序。**

```
   時刻   交易 A（P 加入）              交易 B（Q 加入）
   ───────────────────────────────────────────────────────
   t0     BEGIN（快照：49 人）
   t1                                   BEGIN（快照：也是 49 人）
   t2     SELECT count → 49
          「49 < 50，可以加」
   t3                                   SELECT count → 49
                                        「49 < 50，可以加」
   t4     INSERT member(P, G)
   t5                                   INSERT member(Q, G)
   t6     COMMIT ✔（寫的是 P 那行）
   t7                                   COMMIT ✔（寫的是 Q 那行）
   ───────────────────────────────────────────────────────
   結果：50 + P + Q = 51 人  ✗  超員
```

關鍵在 t2/t3：兩個交易的快照都拍在對方 INSERT 之前，**各自讀到 49**，各自獨立判斷「還有一個名額」——但那一個名額被兩個人同時當成自己的。

**(b) 為什麼 SI 沒擋。** snapshot isolation 的衝突偵測只看**寫-寫衝突**（兩個交易改同一行）。但 A 寫的是 P 那行、B 寫的是 Q 那行——**兩個 INSERT 碰的是不同行**，沒有寫-寫衝突。SI 的偵測器看不到「它們各自讀的 count 因為對方的寫而失效了」這層**讀-寫依賴**。這正是 write skew 的定義：邏輯上衝突（共享一個被讀的聚合）、物理上不衝突（寫不同行）。

**(c) 三種擋法與代價。**

1. **手動上鎖**：把「人數約束」物化成一把可鎖的東西——例如 `SELECT ... FROM guild WHERE id = G FOR UPDATE` 先把公會那行鎖住，再數人數、再 INSERT。代價：兩個交易在公會行上序列化（後到的等），加入公會這條路上失去部分並發；但便宜、直觀、可控。**前提是你得自己想到要鎖**——漏了就破。
2. **物化衝突計數器**：在 guild 表放一個 `member_count` 欄，兩個交易都 `UPDATE guild SET member_count = member_count + 1 WHERE id = G AND member_count < 50`，靠這個共同的寫把隱性衝突變顯性寫-寫衝突（且用 `WHERE member_count < 50` 讓超員的那次更新 0 行、據此回絕）。代價：要維護這個冗餘計數器、它本身成為熱點。
3. **升 Serializable（PG SSI）**：讓資料庫的 SSI 自動偵測這個讀寫依賴危險結構，在 commit 時中止後到的那個。代價：SSI 的 predicate locking 開銷 + **應用必須 retry 被中止的交易**（且 retry 須冪等）；高衝突的熱門公會下中止率高、retry 放大負載。

常見錯路：以為「我已經在 Repeatable Read 了，count 數出來是準的」。在 SI 下 count 是「你快照裡的準」，不是「commit 那一刻全域的準」——這兩個準在並發下不是同一個東西。

### 推演二：手追「寫主庫後立刻讀 replica」讀到舊值 **[12 分鐘] ★★**

使用者改了個人簡介，前端「樂觀更新」先把畫面改了，同時發寫請求；寫成功後，前端為了「跟伺服器對齊」立刻重新拉一次個人資料。後端：寫走主庫、讀走 read replica（非同步複製，當前複製延遲約 800ms）。

問：(a) 畫出使用者看到「簡介又變回舊的」的時序；(b) 為什麼前端的「樂觀更新」反而讓這個 bug 更難被開發者發現；(c) 用本章三招各補一次，說各自在這個場景下行不行。

#### 推演解答

**(a) 時序。**

```
   時刻    前端                          後端
   ────────────────────────────────────────────────────────
   t0      使用者送出新簡介
   t0+     樂觀更新：畫面先顯示「新簡介」
   t1                                    寫 → 主庫 commit ✔（主庫=新簡介）
   t2      收到寫成功，發起「重新拉資料」
   t3                                    讀 → 導去 replica
   t4                                    replica 還在 800ms 延遲窗口內，
                                         尚未套用這筆寫 → 回「舊簡介」
   t5      畫面用回傳值覆蓋：
          「新簡介」→ 顯示回「舊簡介」 ✗
   ────────────────────────────────────────────────────────
   使用者：剛改的東西，自己刷新後又變回去了
```

**(b) 為什麼樂觀更新讓 bug 更難抓。** 因為這個 bug **只在「寫成功後緊接著的那次讀」落到還沒追上的 replica 時**才現形——它是個時間競賽，複製延遲短的時候（負載低、本機測試環境往往延遲趨近 0）根本不重現。開發者本機測，replica 延遲幾乎是 0，t4 那次讀剛好讀到新值，一切正常；上了生產、負載一高複製延遲拉到幾百毫秒，才偶發。樂觀更新還火上加油：它讓「壞掉」表現為「畫面從新值閃回舊值」這種詭異的視覺跳動，而不是乾脆的錯誤，更容易被當成「前端偶發渲染問題」誤判。**這是一致性故障的典型指紋：不報錯、不重現、體感詭異。**

**(c) 三招各補。**

- **讀主**：把「寫成功後緊接的那次對齊讀」強制走主庫。在這個場景**完全有效**——主庫一定是新值。代價：這條對齊讀失去 replica 分流，但它只是偶發的單次讀，代價可忽略。**這招在這裡是最對的。**
- **sticky 窗口**：使用者寫後 N 秒內所有讀釘主庫。有效，但比「只把這次對齊讀導主」更重（窗口內所有讀都不分流），對「只需要這一次對齊讀準」的場景殺雞用牛刀。
- **版本戳路由**：寫回傳 LSN/GTID，對齊讀帶上它、只讀已追到的 replica。最精準，但需要中介層支援——在這個簡單場景下複雜度過高，是「對的但太重」。

判準：場景越窄（就這一次對齊讀），越該用最輕的「讀主」；場景越廣（使用者寫後一連串讀都要準），才升級到 sticky 或版本戳。

### 推演三：給異常判隔離級別、畫 deadlock 等待環、判讀分流打破誰 **[18 分鐘] ★★★**

三小題：

(a) 下面這個現象，在哪個隔離級別會出現？升到哪一級能擋？「結算交易第一次數『公會排名前 10 的玩家』有 10 人，發到一半再數變 11 人（中途有人達標插進來）」。

(b) 兩個交易 T1、T2 各要更新公會行 G 和玩家行 P，T1 先 G 後 P、T2 先 P 後 G，發生死結。畫等待環，指出資料庫該回滾誰、依據什麼。

(c) 某團隊為降主庫負載，把**所有** `SELECT` 都導去 read replica（含登入後立刻讀「我的待辦清單」、剛下單後讀「我的訂單」）。判斷這個決定打破了哪個系統的哪個一致性假設，舉一個會被使用者罵的具體場景。

#### 推演解答

**(a) 幻讀，升 Serializable（或在 PG/InnoDB 的 RR 下已被 SI/next-key lock 擋住）。** 「同一範圍查詢兩次、行數變多」是**幻讀**的教科書定義。按 SQL 標準，幻讀在 `Read Committed` 和標準 `Repeatable Read` 下都可能出現，要升到 `Serializable` 才保證擋。**但注意本章的校正**：PG 的 RR 是 snapshot isolation，第二次數的還是同一張快照、看不到中途插入的新行，所以**在 PG 的 RR 下這個幻讀其實已經不會出現**（你兩次數的都是快照裡的 10 人）；InnoDB 的 RR 則靠 next-key lock 鎖住範圍、擋掉中途 INSERT。所以正確答案是：標準框架說「要 Serializable」，但**實務上 PG/MySQL 的 RR 因為實作成 SI/加 gap lock，這個幻讀已被擋**——這恰恰反襯出它們的 RR「比標準強（擋幻讀）卻又有別的洞（不擋 write skew）」的雙面性。

**(b) 等待環與犧牲者。**

```
   時刻   T1（先 G 後 P）        T2（先 P 後 G）
   ────────────────────────────────────────────
   t0     鎖到 G（X）
   t1                            鎖到 P（X）
   t2     要鎖 P → 被 T2 握，等
   t3                            要鎖 G → 被 T1 握，等

   wait-for graph：
         T1 ──等P──► T2
          ▲           │
          └───等G─────┘     ← 環！T1→T2→T1
```

資料庫建這張 wait-for graph，發現 T1→T2→T1 的環，判定死結。**回滾誰**：挑「回滾代價較小」的那個（改動較少、做的工作較少的交易）當犧牲者——假設 T2 做的事較少就回滾 T2，釋放它握的 P，T1 拿到 P 繼續跑。依據是「讓白做的工作最少」的啟發式（ch03 InnoDB 挑「改動較少者」）。根治之道仍是 ch03 的統一順序（都先 G 後 P 或都先 P 後 G），讓環變鏈。

**(c) 打破讀己之寫，登入待辦/下單後讀單會被罵。** 把**所有**讀導去 replica，等於把整個系統的讀從「讀己之寫」降到「最終一致」。打破的是**「使用者自己剛寫的東西必須立刻讀得到」**這個假設。具體會被罵的場景：使用者**剛下單**，下單寫進主庫、回了成功，使用者興沖沖點「我的訂單」——這個讀落到還沒追上的 replica，**訂單列表裡找不到剛下的那筆單**。使用者合理懷疑「我的單沒下成功？」可能重複下單（又回到 ch10 冪等問題）。同理「剛新增的待辦刷新後不見」。**判準**：別人寫的晚幾秒看到無所謂（本就最終一致），但**同一使用者寫後緊接的讀**必須守讀己之寫——這些路徑不能無腦導 replica，要讀主/sticky/版本戳。團隊「全部導 replica」的錯，在於沒區分「讀別人的」和「讀自己剛寫的」這兩類讀。

## 延伸閱讀

- **PostgreSQL Documentation — Transaction Isolation（13.2）** <https://www.postgresql.org/docs/current/transaction-iso.html>：本章 §RR=SI、§write skew、§SSI 的一手依據。務必讀「Repeatable Read Isolation Level」（看它白紙黑字說這是 Snapshot Isolation、Serialization Anomaly 仍 Possible）與「Serializable Isolation Level」（SSI、predicate locking、必須 retry）兩段。
- **PostgreSQL Documentation — Lock Management（runtime-config-locks）** <https://www.postgresql.org/docs/current/runtime-config-locks.html>：`deadlock_timeout` 預設 1s 的官方說明與「為什麼先等再偵測」的取捨理由，本章 deadlock 那節的依據。
- **MySQL 8.4 Reference Manual — Transaction Isolation Levels（17.7.2.1）** <https://dev.mysql.com/doc/refman/8.4/en/innodb-transaction-isolation-levels.html>：InnoDB 預設 RR、各級別行為、與 next-key lock 的關係。對照 PG 看兩個 RR 的本質差異。
- **MySQL 8.4 Reference Manual — Phantom Rows（17.7.4）** <https://dev.mysql.com/doc/refman/8.4/en/innodb-next-key-locking.html>：InnoDB 怎麼用 next-key lock（record + gap）擋幻讀——這是 MySQL RR 和 PG RR「同名不同機制」的關鍵，也接 ch03 gap lock 那段。
- **Jepsen — MySQL 8.0.34** <https://jepsen.io/analyses/mysql-8.0.34>：實測指出 MySQL 的 RR **既不是標準 RR、也不是 Snapshot Isolation（比 SI 更弱）**，會出現 write skew（G2-item）、read skew、lost update 等多類異常。本章「MySQL RR 也擋不住 write skew」的獨立佐證（讀者注意：Jepsen 是嚴格的對抗性測試，結論比廠商文件更保守）。
- **Daniel Abadi — Consistency Tradeoffs in Modern Distributed Database System Design（PACELC, IEEE Computer 2012）** <https://www.cs.umd.edu/~abadi/papers/abadi-pacelc.pdf>：PACELC 原始論文。CAP 只談分區時的取捨，PACELC 補上「沒分區時，延遲 vs 一致」這條你每天都在面對的取捨——讀者把 read replica 同步/非同步的選擇對回 PACELC 的 E 段，會豁然開朗。（CAP 本身：Brewer 2000 conjecture、Gilbert-Lynch 2002 證明，見 landscape §23。）
- **Martin Kleppmann —《Designing Data-Intensive Applications》第 5、7、9 章** （書，非連結）：複製（ch5）、交易與隔離級別（ch7）、一致性與共識（ch9）的權威整合。本章挖的每一條（一致性譜、write skew、複製延遲、線性一致）這本書都有更深的版本；想把本章往「為什麼這樣設計」再推一層，讀它。
