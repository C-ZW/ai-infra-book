# ch17 — 快取與 Redis 內部：單執行緒的脾氣與快取的根本難題

> **本章解決什麼問題**：這是 Part IV「橫切的老問題」裡的快取章。你履歷上 Redis 出現過兩次——在 LetsTalk 當 Pub/Sub 扇出層（ch07 挖過），在多個系統當快取；你現職的多層 cache library（記憶體→S3→local 逐層降級）更是把「快取會掉，怎麼優雅降級」這件事做成了一個元件。本章不重講 ch07 的 Pub/Sub，而是挖 **Redis 本身的脾氣**：它為什麼快、為什麼是單執行緒、一個慢命令怎麼卡死所有人、它落地時（RDB/AOF）在權衡什麼、key 滿了它怎麼丟；以及更上層的——**快取這個概念本身的根本難題**（失效、雪崩/穿透/擊穿、與源不一致）。一致性的深掘留給 ch11，叢集分片與一致性雜湊不在本章（最多一句話）。

## 從你已知的出發

你用 Redis 用到不會多想。在 LetsTalk 它是通知扇出的 backplane（ch07）；在更多地方它是那個「先查 Redis，沒有再查 DB，查到了寫回 Redis」的快取層。你大概也在某次 RDS 壓力大的時候，把熱查詢的結果丟進 Redis，CPU 立刻鬆一口氣——這招太好用，好用到你從來沒問過「為什麼這招這麼有效，它的代價藏在哪」。

更值得停下來看的，是你現職在 Bahwan 做的那個**多層 cache library**：記憶體（in-process）→ S3 → local 逐層降級——**最快的層先查，每一層失效就退到下一層**（in-process Map 最快但小、隨行程重啟清空；S3 大、跨行程共享但慢；local 落地檔案是最後兜底）。這套能跑，代表你對「快取會掉、要有退路」有第一線直覺。但這直覺底下，藏著幾個你大概沒攤開講過的問題。

先預告一件你**很可能記成舊版**、本章最大的「老兵記憶校正」：你 2019–2024 認識的 Redis 是 **BSD 授權、版本停在 7.x** 的 Redis——**這兩件事現在都變了**（2026-06）：授權在 2024 改成 source-available、社群 fork 出了 **Valkey**；版本已經 8.x。這不是冷知識，它直接影響你今天要不要用「Redis」這名字、用哪個 fork、踩哪條授權——後面有一整節專門講。

但先把最核心的脾氣攤開。你大概知道「Redis 很快，因為它在記憶體裡、而且是單執行緒」。這句話對，但**它藏著一個你可能記反的點**：很多人聽到「Redis 6 之後支援多執行緒 I/O」，就以為「Redis 現在是多執行緒的、命令會平行跑了」。**錯。** Redis 的**命令執行永遠是單執行緒的**——這是設計本質，到 Redis 8 都沒變。多執行緒只分擔「網路 I/O」這件雜事，不分擔「執行你的命令」這件正事。這個區別，是理解「為什麼一個 `KEYS *` 能卡死所有人」的鑰匙。這一章就從這裡開挖。

## Redis 為什麼快——又為什麼故意選了單執行緒

先破除一個常見的因果倒置：**Redis 快不是因為單執行緒，是因為在記憶體裡；單執行緒是它「敢於」這麼設計的代價可控的選擇，不是它快的原因。**

把「為什麼快」拆成幾層，每一層的貢獻不一樣：

```
 Redis 的速度來源（由大到小）
 ┌──────────────────────────────────────────────────┐
 │ ① 資料全在記憶體（RAM）                          │
 │    → 沒有磁碟 seek、沒有 buffer pool 換頁        │
 │    → 這是最大的一塊，跟單/多執行緒無關           │
 ├──────────────────────────────────────────────────┤
 │ ② 簡單高效的資料結構（hash、skiplist、           │
 │    intset、ziplist/listpack…）＋ O(1) 雜湊表查找 │
 ├──────────────────────────────────────────────────┤
 │ ③ I/O 多工（epoll/kqueue）單執行緒撐上萬連線     │
 │    → 跟 Node event loop 同一招（見 ch12）        │
 ├──────────────────────────────────────────────────┤
 │ ④ 單執行緒命令執行 → 沒有鎖、沒有 context switch │
 │    → 不是「快的原因」，是「省掉慢的東西」        │
 └──────────────────────────────────────────────────┘
```

第④點才是「單執行緒」的真正意義，而它的好處是**反過來的**——它不是讓 Redis「跑得更快」，是讓 Redis「不必付多執行緒的稅」：

- **沒有鎖**：多執行緒共享資料結構，每次讀寫都要加鎖、解鎖，鎖競爭本身就是開銷（你在 ch03 為了降 RDS 鎖競爭折騰半天，最懂這個稅有多貴）。Redis 把「同一時刻只有一個命令在動資料」當成鐵律，於是**完全不需要鎖**——每個命令對資料的操作天生原子。`INCR`、`LPUSH`、`SETNX` 為什麼能當同步原語用？就是因為它們執行時不會被另一個命令插隊。
- **沒有 context switch、沒有 cache line 爭用**：單執行緒不會在 CPU 核心之間跳來跳去，CPU cache 命中率高。
- **行為可預測、好推理**：命令一條接一條跑，沒有交錯的並發 bug。這對一個被當成「全系統共享狀態」的東西來說，是巨大的工程簡化。

你大概注意到了——這跟 Node 的 event loop（ch12）是**同一套哲學**：單執行緒 + I/O 多工，用「並發但不平行」撐起大量連線，代價是「一個 CPU 密集的活會卡住所有人」。Redis 和 Node 是這套哲學的兩個近親。你對 event loop 卡死的直覺，原封不動可以搬到 Redis 上。

> **我以為 vs 實際**：你大概記得「Redis 6 加了多執行緒，所以現在是多執行緒的」。**精確的事實是**——Redis 6.0+ 加的是 **threaded I/O**，只把「從 socket 讀 bytes、parse 成命令、把 response 寫回 socket」這幾件**網路雜事**卸載到多執行緒；**真正執行命令（動你的資料）的那一步，從 Redis 1.0 到 Redis 8 都是、永遠是單一主執行緒、永遠不平行**（2026-06）。Redis 8 重寫了 I/O threading 實作、效能更好，但執行模型沒變。

而且有個更陰險的 nuance，連知道「有 io-threads」的人都常記錯：

> **易錯細節（2026-06）**：**即使在 Redis 8，`io-threads` 預設值仍是 `1`（等於關閉）**。你不手動在 `redis.conf` 設成 4–8 並重啟，threaded I/O 根本沒生效（`io-threads-do-reads` 預設也是 off，所以預設連讀都不卸載）。而且這個值**不能用 `CONFIG SET` 在執行期改**，一定要改設定檔重啟。所以「Redis 8 預設開了多執行緒 I/O」是錯的——正確說法是「Redis 8 強化了 I/O threading 的能力，但仍需手動啟用」。你如果沒動過設定，你的 Redis 就是徹頭徹尾的單執行緒。

把這一切合起來，一個關鍵後果浮現：**因為命令執行是單執行緒、一條一條跑，所以「執行一條慢命令的時間」= 所有其他命令必須排隊等待的時間。** 這就是下一節的主角。

## 一個慢命令怎麼卡死所有人

這是 Redis 最容易出事、也最值得你刻進肌肉記憶的一點。Redis 把所有命令排成一條隊，主執行緒從隊頭一條一條拿來執行。隊裡每一條命令，**獨佔主執行緒直到它做完**。沒有時間片輪轉、沒有搶佔——Redis 不會做到一半「先讓別人插個隊」。

所以一條 O(N) 甚至 O(N log N) 的命令，在大 key 上跑起來，就是一場災難。罪魁禍首通常是這幾個：

| 慢命令 | 為什麼慢 | 該用什麼代替 |
|---|---|---|
| `KEYS pattern` | O(N)，N = **整個 keyspace 的 key 數**，掃全部 key | `SCAN`（游標分批、每批 O(1)~O(小常數)，不阻塞） |
| `SORT`（大 list/set） | O(N + M log M)，資料大就久 | 預先排好存、或在應用層排 |
| `SMEMBERS` / `HGETALL` / `LRANGE 0 -1`（大集合） | O(N)，一次撈出整個大集合 | `SSCAN`/`HSCAN`/分頁 `LRANGE` |
| `FLUSHALL`/`FLUSHDB`（同步） | 一次清空、阻塞 | `FLUSHALL ASYNC`（背景釋放） |
| `DEL` 一個超大 key | O(N) 釋放記憶體 | `UNLINK`（背景回收） |

`KEYS *` 是最經典的殺手。它聽起來人畜無害（「列出所有 key 嘛」），但它要**遍歷整個 keyspace**——如果你的 Redis 裝了一千萬個 key，這一條命令就要在主執行緒上跑掉幾百毫秒甚至更久，**期間所有其他客戶端的所有命令全部凍結**。你那台同時當快取又當 Pub/Sub backplane 的 Redis，會在這幾百毫秒裡對全系統「裝死」。

> **我以為 vs 實際**：你大概在某個維運腳本、某個「臨時看一下有哪些 key」的 debug session 裡打過 `KEYS something*`。在你 dev 環境的幾百個 key 上，它瞬間回來，你以為它很快。**它不快——它的代價跟 key 總數成正比，只是你 dev 環境 key 少看不出來。** production 上一條 `KEYS *` 就是一次自殘。鐵律：**`KEYS` 只能出現在「絕對確定 keyspace 很小」或「這台 Redis 沒人在用」的場合，其他一律用 `SCAN`。**

### Worked example：一個 `KEYS *` 在 3,200 並發下阻塞 Redis

把它算到底。場景錨在你 LetsTalk 的規模：~3,200 並發連線，背後那台 Redis 同時扛通知 Pub/Sub 扇出 + 一些快取查詢。某個半夜，有人（也許就是當年的你）為了排查連到 production Redis 打了一條 `KEYS session:*`，而此刻 keyspace 裡有約 200 萬個 key。

先估這條命令本身的執行時間。Redis 遍歷 keyspace 大致是「掃過每個 key 做 pattern match」，吞吐量級在數百萬到上千萬 key/秒（取決於 CPU、key 長度）。保守抓 **500 萬 key/秒**：

```
 200 萬 key ÷ 500 萬 key/秒 ≈ 0.4 秒
```

**0.4 秒**——聽起來不長？對 Redis 是天文數字。把連鎖效應追出來：

```
 時間軸（主執行緒視角）
 ─────────────────────────────────────────────────────►
  t=0ms     KEYS session:*  進入執行，主執行緒被獨佔
  t=0~400ms 主執行緒「裝死」：
            ├─ 通知 Pub/Sub 的 PUBLISH → 全部排隊
            ├─ 快取查詢 GET/SET     → 全部排隊
            ├─ 心跳/連線管理        → 全部排隊
            └─ 新進命令持續堆積在隊裡
  t=400ms   KEYS 終於做完，主執行緒開始清積壓的隊
  t>400ms   積壓的命令一次湧出，延遲尾巴拖更久
```

在這 0.4 秒裡，假設你的系統平常每秒對 Redis 發約 **20,000 個命令**（3,200 連線 + 快取查詢，這是合理量級）。那麼這 0.4 秒堆積了：

```
 20,000 命令/秒 × 0.4 秒 ≈ 8,000 個命令卡在隊裡
```

這 8,000 個命令**全部至少多等 0.4 秒**（隊頭那批），加上 Redis 做完 `KEYS` 後還要花時間消化積壓，**尾延遲（p99/p999）會從平常的 < 1ms 飆到幾百毫秒**。對使用者端的觀感：通知突然集體延遲、某些請求 timeout、上層服務的連線池被「卡住的 Redis 呼叫」佔滿、連鎖觸發上游的 retry——而 retry 又放大流量（這就是 ch14 講的 retry storm / metastable failure 的起點）。

**最陰險的是觀測盲區**：Redis 的 CPU 在這 0.4 秒衝到 100%，但你的「Redis 連線數」「記憶體用量」這些常看的指標**毫無異常**。你盯著 dashboard 會一頭霧水——直到你去看 **`slowlog`**（Redis 內建的慢命令日誌，預設記錄超過 10ms 的命令），才會看到那條 `KEYS session:*` 孤零零地躺在那裡，旁邊標著 `412000 微秒`。

**防禦**：① 把 `KEYS`、`FLUSHALL`、`DEBUG` 等危險命令用 `rename-command` 在 production 改名或禁用（讓手滑的人打不出來）；② 監控 `slowlog`，把「有慢命令出現」做成告警；③ 對大集合操作一律用 `SCAN`/`SSCAN` 系列游標分批；④ 心理上接受——**這台 Redis 是全系統共享的單執行緒資源，任何人在上面跑一條慢命令，全系統陪葬**。它不是「你的」一台機器，是「大家的」一根獨木橋。

## 持久化：RDB 快照 vs AOF 指令日誌，與「當快取」和「當資料庫」的分野

Redis 在記憶體裡，但記憶體一斷電就沒了。所以它提供兩條落地（persistence）的路，而**你選哪條、怎麼設，取決於一個根本問題：這台 Redis 掉了資料，你能不能接受？**

這正好回扣你 ch09 的教訓——你當年把 setTimeout 換成 cron job queue，核心就是「把狀態從記憶體外部化到持久儲存，這樣重啟後還在」。Redis 當資料庫用時，面對的是一模一樣的問題，只是換到 Redis 自己身上。

**兩條路的機制：**

```
 RDB（Redis Database / 快照）
 ┌──────────────────────────────────────────────┐
 │ 每隔一段時間（或手動 BGSAVE），fork 出子行程，│
 │ 把「此刻整個記憶體」dump 成一個二進位檔。      │
 │ ▸ 點時間快照（point-in-time）                 │
 │ ▸ 檔案小、載入快 → 重啟恢復快                  │
 │ ▸ 兩次快照之間崩潰 → 丟掉這段全部資料          │
 │ ▸ fork() 在大資料集上會卡主執行緒幾 ms~1s      │
 └──────────────────────────────────────────────┘

 AOF（Append Only File / 指令日誌）
 ┌──────────────────────────────────────────────┐
 │ 把每一條「改變資料的命令」append 到一個 log。  │
 │ 重啟時重放整個 log 重建狀態。                  │
 │ ▸ 耐久度可調（看 fsync 策略，見下）            │
 │ ▸ log 會一直長 → 定期 rewrite 壓縮             │
 │ ▸ 重啟要重放 log → 比 RDB 載入慢               │
 └──────────────────────────────────────────────┘
```

AOF 的耐久度，全看那個 **`appendfsync`** 設定——它決定「命令寫進 log buffer 後，多久真正 `fsync` 到磁碟」：

| `appendfsync` | 行為 | 最壞資料損失 | 效能 |
|---|---|---|---|
| `always` | 每一條寫命令都 fsync 才回應 | 幾乎零 | **最慢**（每次寫都等磁碟） |
| `everysec`（預設） | 背景執行緒每秒 fsync 一次 | **最多約 1 秒**的寫入 | 高（fsync 不擋主執行緒） |
| `no` | 交給 OS 決定何時 flush（通常 ~30s） | 一個 OS flush 週期的資料 | 最快（≈ 無 fsync） |

這張表是整節的核心。`everysec` 是甜蜜點——背景執行緒做 fsync，主執行緒幾乎不被擋，代價是**崩潰時最多丟最近約 1 秒**的寫入（2026-06，官方文件原文：「you can only lose one second worth of writes」）。`always` 給你近乎零丟失，但每次寫都要等磁碟確認，**直接把 Redis 從記憶體速度拉回磁碟速度**——這對「Redis 圖快」的初衷是釜底抽薪。

> **關鍵 nuance**：`appendfsync always` **不等於** exactly 不丟。即使每條都 fsync，從「命令執行完」到「fsync 真正落地」之間仍有極窄的窗口；而且這只保證**已被 Redis 接收並執行**的命令，**不保證**「客戶端送出但還沒到 Redis」的命令。耐久度永遠是個程度問題，不是開關（這跟 ch10「exactly-once 是幻覺」是同一種誠實）。

**RDB 的暗礁是 `fork()`**：BGSAVE 要 fork 一個子行程來 dump，而 fork 在大資料集上**會卡住主執行緒幾毫秒到一秒**（要複製行程的 page table；資料越大越久）。所以「RDB 不阻塞」是半真半假的——dump 本身在子行程不阻塞，但**那一下 fork 會**。你那台共享 Redis 如果記憶體很大、又設了頻繁 BGSAVE，每次快照都會給全系統一個延遲尖刺。

**現在回到那個根本問題——「Redis 當快取」和「Redis 當資料庫」要不同設定：**

| 面向 | Redis 當「快取」 | Redis 當「資料庫 / 真相來源」 |
|---|---|---|
| 資料的源 | 在別處有源（DB 才是真相） | Redis 就是源，丟了補不回 |
| 持久化 | 可以不持久化（`appendonly no`、RDB 也可不開） | 必須持久化且夠耐久（AOF `everysec` 或 `always` + RDB 混合，重啟才能重建） |
| 重啟的意義 | 冷快取，慢慢回填、可接受 | 必須完整恢復，丟資料 = 事故 |
| eviction | 開 eviction（`allkeys-lru/lfu`），滿了淘汰舊的 | 通常 `noeviction`，滿了寧可報錯也不淘汰資料 |

**這就是為什麼同一個 Redis 軟體，兩種用途要兩套設定。** 當快取時，你**故意**容忍丟失——丟了就回去查 DB，重啟後冷快取慢慢回暖，沒人會死；甚至可以完全關掉持久化（`appendonly no` + 不設 RDB save 點）換取最高效能。當資料庫時（有人真把 Redis 當主儲存——leaderboard、session store、限流計數器），**丟資料就是丟業務**，必須開 AOF（至少 `everysec`，怕到丟一秒都不行就 `always`），通常還搭 RDB 做快速重啟，並設 `noeviction` 讓它「寧可寫不進去報錯，也不准淘汰任何一筆」。

回到你現職的多層 cache library——你那層用 Redis（如果有用）幾乎肯定是**當快取**：S3 / local 那兩層是更持久的退路，Redis 只是最快的前線，丟了往下退即可。所以你那層該配「可丟、開 eviction、輕持久化甚至不持久化」這一側。把它配成「資料庫」側（重持久化、noeviction），是在為一個本來就允許丟的東西付不必要的耐久稅、還可能因為 noeviction 在滿的時候直接報錯打爆前線。

## eviction：key 滿了 Redis 丟誰，以及「快取就是會掉」

Redis 在記憶體裡，記憶體有上限（`maxmemory`）。**滿了怎麼辦？** 這由 `maxmemory-policy` 決定，而你對這個設定的選擇，直接編碼了「這台 Redis 是快取還是資料庫」的態度。

八種核心策略（以 Redis 8.0 為基準）：

```
 maxmemory-policy 全家族
 ┌────────────────┬──────────────────────────────────┐
 │ noeviction     │ 預設。滿了就對寫命令回錯誤        │
 │ (預設)         │ (OOM command not allowed)，不淘汰 │
 ├────────────────┼──────────────────────────────────┤
 │ allkeys-lru    │ 所有 key 裡淘汰最久沒用的         │
 │ allkeys-lfu    │ 所有 key 裡淘汰最少用的           │
 │ allkeys-random │ 所有 key 裡隨機淘汰               │
 ├────────────────┼──────────────────────────────────┤
 │ volatile-lru   │ 只在「有設 TTL」的 key 裡淘汰     │
 │ volatile-lfu   │ 只在有 TTL 的 key 裡 LFU 淘汰     │
 │ volatile-ttl   │ 只在有 TTL 的 key 裡淘汰最快到期的│
 │ volatile-random│ 只在有 TTL 的 key 裡隨機淘汰      │
 └────────────────┴──────────────────────────────────┘
```

> **2026-06 補充**：Redis 8.6 起再加兩個「最近最少修改」策略 `volatile-lrm`／`allkeys-lrm`（Least Recently Modified——以「上次被**改寫**」而非「上次被**讀**」為準，適合寫密集場景），所以 8.6+ 共**十種**；下面的兩軸心智模型不變、只是多一個「準則」選項。

兩個軸：**「allkeys 還是 volatile」**（淘汰範圍是全部 key，還是只限有設過期時間的 key）× **「用什麼準則」**（LRU 最近最少用 / LFU 最不常用 / TTL 最快到期 / random）。

> **我以為 vs 實際**：你大概默認「Redis 滿了會自動淘汰舊的」。**預設不會！** Redis 的預設 `maxmemory-policy` 是 **`noeviction`**——滿了之後**寫命令直接回錯誤**，一個 key 都不淘汰。如果你把它當快取用、卻沒改成 `allkeys-lru`（或 lfu），有一天記憶體滿了，你會看到一片 `OOM command not allowed when used memory > 'maxmemory'`——快取突然「寫不進去」，所有 SET 失敗，而你還以為「快取嘛，滿了會自己擠掉舊的」。**這個預設值是為「當資料庫」設計的安全側（寧可報錯不准丟資料），不是為「當快取」**。當快取一定要主動改。

**LRU vs LFU 的選擇**：LRU（最近最少用）淘汰「最久沒被碰的」；LFU（最不常用，Redis 4.0+）淘汰「總體用得最少的」。差別在抗「一次性掃描污染」——如果有個批次作業一次性掃了一大票冷門 key，LRU 會把它們當成「剛用過」而保留、反而擠掉真正的熱 key；LFU 看「用的頻率」，不會被這種一次性掃描騙到。熱點明顯、有掃描型存取的快取，LFU 通常更穩。

但無論哪種策略，有一句心理建設必須刻進去：**快取就是會掉。** 不是「可能會掉」，是「設計上就會掉」——記憶體滿了 eviction 會掉、TTL 到期會掉、Redis 重啟（沒持久化或冷啟動）會掉、網路分區連不到也等於掉。**任何「假設快取裡一定有」的程式碼都是 bug。** 快取命中是 best-effort 的幸運，不是保證。你的程式必須對「快取 miss」有正確的退路（回源），而且回源這條路本身要扛得住「所有人同時 miss」的壓力——這就引出下一節快取的根本難題。

## 快取的根本難題：失效、雪崩/穿透/擊穿、與源不一致

有句被引用到爛的話（出自 Phil Karlton）：「電腦科學只有兩件難事——快取失效（cache invalidation）和命名。」這不是玩笑。**快取的本質是「在兩個地方放同一份資料的副本」，而只要有副本，就有「副本跟正本不同步」這個永恆問題。** 你的多層 cache library 有幾層，就有幾份副本，就有幾道不同步的縫。

把快取會壞的三種經典姿勢攤開——它們名字相近、機制不同、防禦也不同，最容易被混為一談：

```
 ┌───────────┬──────────────────────┬─────────────────────┐
 │ 故障      │ 機制                  │ 直白比喻            │
 ├───────────┼──────────────────────┼─────────────────────┤
 │ 穿透      │ 查「根本不存在」的    │ 一直問一個不存在的  │
 │penetration│ key → 快取永遠 miss   │ 人的電話，每次都翻  │
 │           │ → 每次都打到 DB       │ 完整本電話簿       │
 ├───────────┼──────────────────────┼─────────────────────┤
 │ 擊穿      │ 「單一熱 key」過期那  │ 一扇門突然消失，    │
 │breakdown  │ 一瞬，大量並發同時    │ 排隊的人瞬間全擠    │
 │(hot key)  │ miss → 同時打 DB      │ 向後面那扇門       │
 ├───────────┼──────────────────────┼─────────────────────┤
 │ 雪崩      │ 「大量 key 同時」過期 │ 整面牆同時倒，      │
 │avalanche  │ (或 Redis 整個掛) →   │ 後面的人全暴露在    │
 │           │ 大量 miss 同時打 DB   │ 火力下             │
 └───────────┴──────────────────────┴─────────────────────┘
```

**穿透（penetration）**：攻擊者（或 bug）一直查不存在的 key——`user:999999999`、`item:-1`。快取裡永遠沒有（因為 DB 裡也沒有，回源也回不出東西寫回快取），於是**每一次查詢都穿過快取直接砸 DB**。防禦：① **快取空值**——查不到也把「空」寫進快取（短 TTL），下次就擋在快取層；② **Bloom filter**——在快取前放一個布隆過濾器，「絕對不存在」的 key 直接拒絕，根本不查（代價是有極小機率誤判「可能存在」而放行，但不會誤判「不存在」）。

**擊穿（breakdown / hot key）**：某個**單一**超熱 key（首頁配置、熱門商品、當紅貼文）的 TTL 剛好到期的那一瞬間，成千上萬個並發請求**同時** miss、**同時**回源去重建這一個 key——DB 瞬間被同一個查詢打幾千次（這就是 thundering herd / 驚群）。防禦：① **互斥鎖（mutex / `SETNX`）**——只放一個請求去重建快取，其他請求短暫等待或回傳舊值；② **邏輯永不過期**——熱 key 不設真正的 TTL，而是在 value 裡記一個「邏輯過期時間」，背景非同步重建，前台永遠拿得到（哪怕略舊的）值。

**雪崩（avalanche）**：**大量** key 在**同一時刻**集體過期（常見原因：系統啟動時一次性批次預熱、給所有 key 設了一模一樣的 TTL），或者更極端——**Redis 整台掛了**，於是所有請求**同時**回源，DB 被瞬間流量壓垮、再連鎖把整個系統拖下水。防禦：① **TTL 加抖動（jitter）**——別給所有 key 設一樣的 TTL，在基準 TTL 上加一個隨機偏移（業界常見抓基準的 ±10~20%），讓過期時間打散、不要在同一秒集體到期；② **多層快取**——前面有 in-process 本地快取頂著，不會所有流量都落到同一層；③ **回源路徑加限流 / 斷路器**（ch14）——就算雪崩了，也別讓無限流量打死 DB，主動卸載一部分；④ **Redis 高可用**——sentinel / 副本，別讓「Redis 整台掛」成為單點。

### Worked example：設計多層 cache 一次「記憶體層失效」的降級路徑

把它落到你現職那個 **記憶體 → S3 → local** 的多層 cache library 上。設計一次「in-process 記憶體層失效」時的降級路徑，並決定它該發什麼觀測訊號。

先畫正常路徑與降級路徑：

```
 正常（記憶體命中）
 請求 ──► [記憶體層] 命中 ──► 回傳  (最快，< 1ms)

 記憶體 miss（重啟後冷快取 / eviction / TTL 到期）
 請求 ──► [記憶體層] miss
            │
            ▼
        [S3 層] ──┬─ 命中 ──► 回填記憶體層 ──► 回傳 (慢，數十~數百 ms)
                  │
                  └─ miss / S3 逾時 / S3 5xx
                        │
                        ▼
                   [local 落地層] ──┬─ 命中 ──► 回傳 (兜底，可能略舊)
                                    │
                                    └─ miss ──► 回源頭重算 / 回傳預設值
```

**每一層失效該往哪退、退化要不要被看見**，逐層拆：

1. **記憶體層失效（行程重啟、eviction、TTL）**：退到 S3。這是**預期內**的降級——記憶體層本來就會隨重啟清空。風險：行程剛重啟、記憶體全空時，**所有請求同時 miss → 同時打 S3**（這就是擊穿/雪崩搬到你的多層架構上）。防禦：回填記憶體層時用單飛（single-flight / mutex），同一個 key 只讓一個請求去 S3 取、其他等它回填；給記憶體層 TTL 加 jitter，避免重啟後一批 key 同時到期。

2. **S3 層失效（逾時、限流、5xx、整個 region 抖動）**：退到 local 落地層。**這裡是降級的關鍵縫**——S3 不像本地呼叫，它有網路延遲、有限流（503 SlowDown）、有逾時。**S3 呼叫必須有硬逾時**（不能無界等待，否則一個慢 S3 呼叫會把你的連線池/event loop 卡住，回到 ch12 的卡死），逾時就立刻退到 local，別傻等。

3. **local 層失效**：退到回源重算，或回傳一個安全的預設值（fail-safe）。這是最後一道防線。

**最重要、也最容易漏的——降級必須被觀測到**（呼應 ch15「壞了你第一時間怎麼知道」）：

> **降級是「靜默成功」，這正是它的危險**。多層 cache 的整個設計目的就是「某層掛了，使用者無感、系統照常回應」——**但「無感」對使用者是優點，對你是盲區**。如果 S3 層已經掛了三天、全靠 local 兜底，而你的 dashboard 一片綠（因為請求都成功回傳了），你根本不知道你早就跑在「最後一層兜底」上、離徹底崩潰只剩一步。**graceful degradation 最大的反模式，就是降級了卻沒有任何訊號。**

所以每一層降級都要發出**結構化的觀測訊號**，至少這幾個：

| 訊號 | 為什麼要 |
|---|---|
| **每層的命中率**（記憶體 / S3 / local 各自 hit rate） | 命中率分布右移（越來越多請求落到下層）= 上層在大量失效，是惡化的早期徵兆 |
| **降級事件計數**（「退到 S3」「退到 local」各發生幾次，帶層級標籤） | 直接量化「現在多少比例的請求在降級狀態跑」 |
| **「最深層命中」告警**（請求一路落到 local，或 local 也 miss 走回源） | 這代表前面所有層都失效了，是該被叫醒的事件 |
| **S3 層的延遲與錯誤率**（p99 latency、逾時率、5xx 率） | S3 是唯一的跨網路層，最可能慢/掛，要單獨盯 |

判準很簡單：**如果某一層默默掛了，你的告警會不會響？** 會響 → 這個降級是「優雅且可觀測」的；不會響 → 你做的不是 graceful degradation，是「優雅地隱藏故障直到它變成災難」。你當年那個 cache library 如果只做了降級邏輯、沒做降級訊號，那它差的就是這最後一塊——而這一塊才是把「能跑」變成「壞了你第一時間知道」的關鍵。

**與源不一致（回指 ch11）**：多層快取還有一個更安靜的問題——**每一層都是源的一份副本，每一份都可能過時**。記憶體層的 `config:homepage` 是 5 分鐘前的、S3 層是 1 小時前的、源剛剛改了——三份不一樣，使用者看到哪份取決於命中哪層。這是個徹頭徹尾的一致性問題：什麼時候要主動失效（invalidate）下游每一層？寫源頭時要不要連帶清快取（write-through / write-behind / cache-aside 的取捨）？這條線本身屬於 ch11（一致性、隔離級別與複製延遲）——這裡你只要記住：**多層快取用「更可能掉、可能更舊」換「更快、更能扛」，而「可能更舊」是你簽下這份合約時就接受的條款，不是 bug。** 哪些資料可以容忍略舊（首頁配置、商品列表）、哪些一秒都不能舊（帳戶餘額、庫存），決定了它能不能進你的快取。

## 老兵記憶校正：Redis 的授權之亂與 Valkey（2026-06）

這一節專門收你開頭那句「Redis 是 BSD、版本 7.x」的舊記憶。這是本章時效性最敏感、也最 load-bearing 的事實——它影響你今天的技術選型，不是冷知識。

把時間線拉直（2026-06 查證版）：

```
 Redis 授權與 Valkey 分裂時間線
 ──────────────────────────────────────────────────────►
 ~2024-03   Redis 7.2.4 仍是最後一個 BSD-3-Clause 版本
            │
            ▼  Redis Inc. 宣布改授權
 2024-03    BSD ──► 雙授權 SSPLv1 / RSALv2（source-available，
            │       非 OSI 開源，雲端轉售受限）
            │
            ├──► Linux Foundation 自 7.2.4 fork 出 Valkey
            │     （續用 BSD-3-Clause，多廠商治理）
            ▼
 2025-05    Redis 8 加回 AGPLv3 → 變成三授權（tri-license）
            （RSALv2 或 SSPLv1 或 AGPLv3，三選一；
             只有 AGPLv3 是 OSI 認可的開源）
            產品名變「Redis Open Source」，模組併進核心
            │
            ▼
 2026-06    現況：Redis 已 8.x（不是 7.x）；
            Valkey 走每年一大版、被主要 distro 與雲端
            （如 AWS ElastiCache）普遍採用
```

幾個必須講清楚的點：

- **授權的實質影響**：SSPL / RSAL **不是 OSI 認可的開源授權**，限制「把 Redis 當服務轉售」（針對雲廠商）；AGPLv3 雖是 OSI 開源，但是**強 copyleft**（把改過的 Redis 當網路服務提供就得開放對應源碼）。對你「在自己後端內部當快取」的絕大多數場景，這些授權**都不影響你**；但對「內嵌進產品、二次開發發佈」這類決定，授權選擇就變得 load-bearing。
- **Valkey 是什麼、為什麼重要**：社群對授權變動的回應——「你不開源，我們 fork 一個續開源的」。它從 7.2.4 fork、API 高度相容、**幾乎 drop-in**；加上 Linux Foundation 多廠商治理 + BSD，主要 distro 與雲端（AWS ElastiCache for Valkey、Google Memorystore for Valkey）已轉向它當預設。**所以你今天在 AWS 開「Redis 相容」的託管快取，預設給的很可能其實是 Valkey。**

> **引用注意（landscape §3b）**：寫到這裡有兩個容易出錯的細節要克制——① **不要綁死單一版本號**（「Redis 8.8.0」「Valkey 9.1.0」這種小版本高頻變動，講「Redis 已 8.x」「Valkey 走每年一大版」這種概括陳述就好）；② **distro 的精確版本號與廠商效能宣稱（「快 8%、省 20%、已遷數百萬節點」）多來自部落格而非官方逐一證實**，引用時用「主要 distro 與雲端已普遍轉向 Valkey」概括，別綁死數字。

**對你的實際意義**：你履歷寫「Redis」沒錯，但面試時若聊到授權或選型，你要能講出這條線——「我用過的 Redis 是 BSD 時代的，2024 之後它改了授權、社群 fork 了 Valkey，現在新專案要先想清楚用哪個、踩哪條授權」。這句話本身，就是「我不只會用，我跟得上它的演化」的證明。

## 故障模式與防禦

這台單執行緒、在記憶體裡、被全系統共享的 Redis，會怎麼壞、壞了長什麼樣、怎麼防。前三個尤其值得刻進肌肉記憶。

**故障 1：一個慢命令阻塞所有人（單執行緒的代價）。**
- **怎麼壞**：`KEYS *`、大 `SORT`、`SMEMBERS` 一個百萬成員的 set、同步 `FLUSHALL`、`DEL` 一個巨大 key——任一條在主執行緒上跑掉幾百毫秒，期間所有客戶端的所有命令全部凍結。
- **長什麼樣**：Redis CPU 瞬間衝 100%，但連線數/記憶體**毫無異常**；全系統的 Redis 呼叫尾延遲（p99/p999）集體飆升；上游連線池被卡住的呼叫佔滿、觸發 retry 放大。**slowlog 裡躺著那條兇手。**
- **怎麼防**：`SCAN`/`SSCAN`/`HSCAN` 取代所有 `KEYS`/大集合一次撈；`UNLINK` 取代大 key 的 `DEL`、`ASYNC` 取代同步 `FLUSH`；`rename-command` 在 production 禁掉危險命令；監控 `slowlog` 並對「出現慢命令」告警。**鐵律：這是大家的獨木橋，別在上面慢跑。**

**故障 2：記憶體滿了寫不進去（noeviction 預設的陷阱）。**
- **怎麼壞**：把 Redis 當快取、卻沒改 `maxmemory-policy`（預設 `noeviction`）。記憶體達 `maxmemory` 後，所有寫命令回 `OOM command not allowed`。
- **長什麼樣**：快取「突然寫不進去」、所有 `SET` 失敗、快取命中率崩塌（因為新值寫不進、舊值被讀爆）、回源流量暴增連帶打爆 DB。你還以為「快取滿了會自己擠舊的」。
- **怎麼防**：當快取用，**明確設 `maxmemory` + `allkeys-lru`（或 `allkeys-lfu`）**，讓它滿了主動淘汰；當資料庫用才保留 `noeviction`（並確保有持久化、別讓它寫滿）。**搞清楚這台是快取還是資料庫，再選 policy。**

**故障 3：快取雪崩/穿透/擊穿打爆 DB。**
- **怎麼壞**：大量 key 同時過期（一樣的 TTL）或 Redis 整台掛（雪崩）、單一熱 key 過期瞬間並發回源（擊穿）、一直查不存在的 key（穿透）——回源流量瞬間放大幾百上千倍砸向 DB。
- **長什麼樣**：DB CPU/連線數突然飆（你 ch03 那種 RDS 尖峰，但這次源頭在快取層）、快取命中率驟降、上游大面積逾時與 retry storm（ch14）。
- **怎麼防**：雪崩 → **TTL 加 jitter**（±10~20%）打散過期 + 回源加限流/斷路器 + Redis 高可用；擊穿 → 熱 key 用 **mutex/`SETNX` 單飛重建**或邏輯永不過期；穿透 → **快取空值（短 TTL）** 或 **Bloom filter** 擋掉不存在的 key。

**故障 4：把 Redis 當資料庫卻配成快取（持久化不足，重啟丟資料）。**
- **怎麼壞**：用 Redis 存 session / leaderboard / 限流計數當真相來源，卻沒開 AOF（或 `appendonly no`）。一次重啟/崩潰，記憶體裡的全沒了。
- **長什麼樣**：重啟後使用者集體登出、計分歸零、限流計數清空（瞬間放行洪水）。
- **怎麼防**：當資料庫一定開 AOF（`everysec` 起跳，怕丟一秒就 `always`）+ 搭 RDB 快速重啟 + `noeviction`。或者——**更誠實的選擇是別把 Redis 當主資料庫**，把真相放在真正的資料庫，Redis 只當前面那層快取。

**故障 5：BGSAVE 的 fork 尖刺（大資料集 + 頻繁快照）。**
- **怎麼壞**：記憶體很大、又設了頻繁 RDB save 點，每次 `BGSAVE` 的 `fork()` 卡住主執行緒幾百毫秒到一秒。
- **長什麼樣**：規律出現的延遲尖刺（跟 save 排程對得上），平常很快、每隔一陣子卡一下。
- **怎麼防**：調 save 排程、用 AOF + RDB 混合而非高頻 RDB、給 Redis 機器留足記憶體（fork 的 copy-on-write 在寫多時會放大記憶體用量）；雲端託管的 Valkey/Redis 通常已調好這塊。

## 紙上推演

### 推演題 1：判斷某用法該用 RDB、AOF、還是都不要 **[15 分鐘]** ★★

對下面三個用法，各判斷該用 RDB、AOF（哪種 `appendfsync`）、還是都不開，並說理由：

(a) **快取層**：先查 Redis、miss 就查 PostgreSQL 並回填。Redis 純加速、DB 是真相。
(b) **限流計數器**：用 `INCR` 做 API rate limiting，每個 key 短 TTL。
(c) **遊戲排行榜（leaderboard）**：用 sorted set 存全服玩家分數，Redis 就是這個排行榜的唯一儲存、沒有別的源。

### 推演題 2：畫快取雪崩的時序並設計防禦 **[20 分鐘]** ★★

某服務啟動時，一次性把 10 萬個熱 key 預熱進 Redis，全設 `TTL = 3600 秒`（一小時）。畫出：(a) 一小時後會發生什麼的時序；(b) 為什麼這是雪崩；(c) 用 TTL jitter 重新設計，並估「±10% jitter」把「同一秒到期的 key 數」從多少降到多少（粗估即可）。

### 推演題 3：找出「快取與 DB 不一致」會在你哪個系統悄悄發生 **[15 分鐘]** ★★★

在你的多層 cache library（記憶體 → S3 → local）裡，源頭資料剛被改了。指出：(a) 此刻三層各自可能是什麼狀態（新/舊）；(b) 使用者會看到哪份，取決於什麼；(c) 哪一類資料絕對不能進這個快取、哪一類可以容忍略舊，判準是什麼。

### 推演解答

**題 1 解答：**

(a) **快取層 → 可以都不開（或只開輕量 RDB）**。Redis 是快取、DB 才是真相，丟了回 DB 查就好。最高效能的選擇是 `appendonly no` + 不設 RDB save 點——重啟後冷快取慢慢回暖，可接受。若想「重啟後快取不要全空、避免冷啟動瞬間打爆 DB（雪崩）」，可開**輕量 RDB**（低頻快照）讓重啟後有個暖身的底，但不需要 AOF——你不在乎丟最近幾秒的快取寫入。`maxmemory-policy` 設 `allkeys-lru`/`lfu`。

(b) **限流計數器 → 通常不開持久化**。計數器短命（短 TTL），重啟丟掉只是「限流窗口重置、短暫多放行一點」，多數場景可接受；安全攸關的限流可考慮 `everysec` AOF 折衷。**常見錯路**：給限流計數器開 `always` AOF——每次 `INCR` 都 fsync，把這個高頻操作拖成磁碟速度，得不償失。

(c) **排行榜 → 必須 AOF，且 Redis 就是真相**。這裡 Redis 是**唯一儲存**，丟了排名就真的沒了（玩家炸鍋）。必須 `appendonly yes`；`everysec` 是合理起點（崩潰最多丟最近一秒的分數更新，通常可接受），若連一秒都不能丟（如涉及真金白銀的賽季獎勵結算）才上 `always`。再搭 RDB 做快速重啟（純 AOF 重放百萬筆 `ZADD` 重啟很慢）。`maxmemory-policy` 設 `noeviction`——這是真相資料，寧可報錯也不准淘汰任何玩家的分數。**這題的核心**：同一個 Redis 軟體，三種用法三套設定，分野全在一句話——**「這台 Redis 掉了資料，我能不能回去別處補？」** 能補（a/b）就往「快取側」配，不能補（c）就往「資料庫側」配。

**題 2 解答：**

(a) **時序**：

```
 t=0       啟動，10 萬 key 一次性寫入，全設 TTL=3600s
 t=0~3600  期間 Redis 全命中，DB 很閒，一切美好
 t≈3600    10 萬 key 幾乎「同一秒」集體到期
            │
            ▼
 t=3600+ε  下一波請求全部 miss（10 萬 key 同時沒了）
            → 全部同時回源 PostgreSQL
            → DB 瞬間被 10 萬個查詢砸中
            → DB CPU/連線數爆 → 慢 → 上游逾時 → retry
            → retry 放大流量 → 可能 metastable failure（ch14）
```

(b) **為什麼是雪崩**：因為這 10 萬 key 的過期時間**高度同步**（都在 t=0 寫入、都設一樣的 3600s TTL → 都在 t≈3600 同時到期）。「大量 key 同時失效 → 大量請求同時回源 → 源被瞬間壓垮」正是雪崩的定義。罪魁是「**一樣的 TTL + 一次性寫入**」這個組合。

(c) **用 jitter 重新設計**：把 TTL 從固定 `3600` 改成 `3600 + random(-360, +360)`（±10%，即每個 key 的 TTL 在 3240~3960 秒間隨機）。**估算**：原本 10 萬 key 擠在「同一秒」（假設原本全在 t=3600 那 1 秒內到期，10 萬 key/秒）。加 ±10% jitter 後，過期時間被均勻打散到 **720 秒寬的窗口**（3960−3240=720 秒）。若均勻分布：

```
 10 萬 key ÷ 720 秒 ≈ 每秒約 139 個 key 到期
```

從「**10 萬 key/秒**」降到「**~139 key/秒**」——回源尖峰被削掉約**700 倍**。DB 從「瞬間被砸 10 萬次」變成「每秒溫和地補幾百個」，完全扛得住。**進一步加固**：再配 mutex 單飛（同一 key 只放一個回源）防殘餘的局部擊穿、回源路徑加限流斷路器（ch14）兜底最壞情況、Redis 做高可用防「整台掛」型雪崩。**常見錯路**：以為「TTL 設長一點就好」——設長只是把雪崩往後推一小時，不解決「同時到期」這個根因；jitter 才是對症的，因為它打散的是**同步性**，不是時間長短。

**題 3 解答：**

(a) **三層各自的狀態**：源剛被改（新值 = V2，舊值 = V1）。此刻——記憶體層可能還是 **V1**（最近一次回填的，可能幾秒~幾分鐘前）；S3 層可能是**更舊的 V1 或 V0**（回填頻率更低、TTL 更長）；local 層可能是**最舊的**（兜底層，更新最不頻繁）。三層很可能是 V1 / V0 / V0 這種「全都不是 V2」的狀態——**沒有任何一層自動知道源變了**（除非你主動失效）。

(b) **使用者看到哪份，取決於命中哪層**：記憶體命中 → 看到 V1；記憶體 miss、S3 命中 → 看到 V0；一路到 local → 看到更舊的。**所以兩個使用者在同一時刻、查同一個 key，可能看到不同版本**——取決於各自的請求命中了哪一層、各層的 TTL 還剩多少。這是個徹頭徹尾的一致性問題（深掘見 ch11）：多副本 + 非同步更新 = 最終一致，**不是**強一致。要讓三層儘快收斂到 V2，得在「寫源頭」時**主動失效（invalidate）每一層**（或 write-through 同步寫穿），否則只能等各層 TTL 自然到期才會回填到 V2——那個延遲就是你的不一致窗口。

(c) **什麼能進、什麼不能**：判準是「**這份資料容忍多舊？**」——
- **可以容忍略舊 → 適合進多層快取**：首頁配置、商品列表/詳情、文章內容、排行榜（晚幾秒更新沒人死）、不常變的設定。這些「舊一點點」不會造成業務錯誤。
- **一秒都不能舊 → 絕對不能進（或只能進有強失效保證的單層）**：帳戶餘額、庫存數量、扣款/支付狀態、權限/封禁狀態（ch16 那個「封禁了 token 還有效」就是這類）。這些一旦讓使用者看到舊值，就是真金白銀的錯誤——餘額顯示成已花掉的、庫存顯示成已售完的、被封禁的人還能操作。

**這題的核心**：多層快取是用「**可能更舊**」換「**更快、更能扛**」的合約。你簽這份合約時，就接受了「使用者可能看到略舊的副本」這個條款。所以**進快取前先問「這份資料舊了會死人嗎」**——會死的留在源、不進快取（或只用有嚴格失效的單層強一致快取）；不會死的才放心進多層、享受速度。把會死的資料丟進「可能更舊」的多層快取，是把一致性故障埋進系統最深處、最難查的地方。

### 本系統五軸體檢

| 軸 | Redis（當快取）／多層 cache library 在這一軸的答案與代價 |
|---|---|
| **交付（delivery）** | 快取不是交付通道，但「快取命中」是 best-effort——miss 是常態（eviction/TTL/重啟/分區都會 miss）。任何「假設快取裡一定有」的程式碼都是 bug；正確姿勢是「miss 就回源」，且回源路徑要扛得住「所有人同時 miss」（雪崩/擊穿）。 |
| **持久（durability）** | 取決於設定：當快取**故意不持久**（丟了回源），重啟=冷快取可接受；當資料庫**必須** AOF（`everysec`/`always`）+ RDB。預設 `noeviction` 是「資料庫側」安全值，當快取必須改 `allkeys-lru/lfu`。分野判準：「掉了資料能不能回別處補？」 |
| **分區（consistency）** | 多層快取 = 多副本 = **最終一致**，不是強一致。各層 TTL 不同 → 同一 key 各層版本可能不同 → 不同使用者看到不同版本。寫源頭要主動失效每層（或 write-through）才收斂；會死人的資料（餘額/庫存/權限）不該進「可能更舊」的快取（深掘見 ch11）。 |
| **並發（concurrency）** | **單執行緒命令執行**——天生原子、無鎖（`INCR`/`SETNX` 可當同步原語），但一條慢命令（`KEYS *`/大 `SORT`）獨佔主執行緒、卡死所有人。io-threads 只分擔網路、不分擔執行（且預設關閉）。並發回源的擊穿用 mutex/單飛擋。 |
| **觀測（observability）** | 慢命令是隱形殺手——CPU 衝 100% 但連線/記憶體無異常，要靠 `slowlog` 抓兇手。多層 cache 的降級是**靜默成功**（最危險的盲區）：要發每層命中率、降級事件計數、「落到最深層」告警、S3 層 p99/逾時率——否則「降級了卻沒人知道」就是優雅地隱藏故障到變災難。 |
