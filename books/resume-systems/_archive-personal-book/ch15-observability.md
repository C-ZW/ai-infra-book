# ch15 — 觀測性：壞了你第一時間怎麼知道

> **本章解決什麼問題**：這是五個故障問句的第五軸——「觀測」——本身的專章。前面四章（交付、一致、並發、時間／背壓）都在問「它會怎麼壞」，這一章問的是收口的那句：**它壞的時候，你第一時間怎麼知道？** 這章把 metric / log / trace 三本柱的取捨講穿、把 ch06 你親手做的「從 log 算 metric」收口（cardinality 爆炸的帳就在這裡算）、把「為什麼看 p99 不看平均」這件你大概一直含糊帶過的事說清楚，最後對上一個你當年沒認真想過的問題：**告警設計**——為什麼「CPU 高」是個壞告警。尾延遲的數學（為什麼尾巴會那麼長）屬於《等待的數學》（queue），這裡只用它的結論；OpenTelemetry 只給一句定位，不教 API。

## 從你已知的出發

你的履歷上，「觀測」這一軸有一個非常具體的錨點：NeoBards 那條 **FluentBit + Lua pipeline，從 production log 萃取 API 回應時間 metric、輸出 S3 給 Nexon**（ch06）。那是你親手把「log 變 metric」這件事做了一遍——你選了邊緣聚合、選了 S3 當交接點、在 Lua filter 裡決定要抽哪些欄位、聚合成什麼。ch06 已經把那條管線挖穿了：百分位數不能跨機器平均、log 掉一行 metric 就少一個樣本、buffer 滿了會靜默丟。

但 ch06 是站在「你那條管線」裡面看的。這一章要把鏡頭拉遠，問一個更上層的問題：**你那條管線，在整個「觀測」這件事裡，站在哪？** 你做的是三本柱裡的 metric，而且是「從 log 反推 metric」這條特定的路。那 log 本身呢？trace 呢？什麼問題該用哪一本柱去查？你當年只做了 metric 那一柱（而且還是繞道從 log 做的），另外兩柱你其實沒搭——這對「事故時你查得到什麼」有很具體的後果。

再往下，還有一個你當年隱性回答過、但大概沒攤開想過的問題。你給 Nexon 的是「p95 回應時間」。為什麼是 p95、p99，而不是「平均回應時間」？你可能直覺知道「看平均會被騙」，但說不清楚被騙在哪。這一章要把它說清楚：**平均會把你最該看見的那群慘用戶，溶解在大多數爽用戶裡。**

最後，還有一個你在 NeoBards、LetsTalk 都一定設過、但設得對不對你大概沒回頭檢查過的東西：**告警**。你設過「CPU 超過 80% 就叫醒我」這種告警嗎？這一章會告訴你：那是一個**壞告警**——不是因為 CPU 不重要，而是因為它告的是「原因」不是「症狀」，而原因告警會在兩個方向上同時害你：半夜叫醒你但用戶根本沒事（誤報、告警疲勞），或者用戶已經慘了但 CPU 還沒高（漏報）。

> 但你可能沒想過的是：**觀測不是事後補的，是設計的一部分。** 一個系統「壞了你第一時間怎麼知道」，這個答案在你寫第一行程式時就已經被決定了——你有沒有留下足夠的痕跡，讓未來那個半夜被叫醒的你，能在不重現問題的前提下把它查清楚。這一章的真正主題，是把「觀測」從一個運維名詞，變成一個你設計系統時就要回答的故障問句。

這章的地圖：先把三本柱（metric / log / trace）的取捨講清楚、什麼問題該用哪本柱；再把 cardinality 爆炸的帳算給你看（收口 ch06）；接著正面拆「為什麼看 p99 不看平均」與 SLI/SLO/error budget；然後是告警設計（症狀 vs 原因）；最後分清「可觀測 vs 監控」這兩個常被混用的詞。

## 三本柱：metric、log、trace，什麼問題該用哪個

觀測性通常被拆成三本柱：metric、log、trace。ch06 已經給過它們的速寫，這裡要把它們當成**三種用不同代價換不同資訊的工具**，並且回答一個 ch06 沒回答的問題：**碰到一個具體事故，你該掏出哪一柱？**

先把三柱的取捨並排，這次把每一柱「能回答什麼問題、回答不了什麼問題」寫清楚：

| 本柱 | 它是什麼 | 擅長回答 | 回答不了 | 代價 |
|---|---|---|---|---|
| **Metric（指標）** | 預先聚合的數字時間序列（每分鐘 p95、每秒 QPS、錯誤率） | 「**有沒有**出事、**何時**開始、**多嚴重**」——趨勢、閾值、告警 | 「**為什麼**、**是哪一筆**」——細節在聚合時被壓掉了 | 便宜、查詢快、可長期保留；但**怕高基數**（按 user 拆就爆，下一節算帳） |
| **Log（日誌）** | 一行行帶時間戳的事件記錄 | 「**這一筆請求**發生了什麼、帶什麼參數、走哪條路、報什麼錯」——單點細節 | 「整體趨勢」要自己掃＋算（貴、慢）；高負載下還可能被丟（ch06） | 細節最高；但量大、貴、查詢慢、易遺漏 |
| **Trace（追蹤）** | 一個請求跨多個服務的因果鏈（span 樹） | 「這 800ms **花在哪個服務、哪一段**」——跨服務的延遲歸因與依賴關係 | 「**為什麼**那一段慢」要再下鑽到那段的 log；少埋一段鏈就斷 | 跨服務因果是別處給不了的；但要**全鏈路埋點**，少一段就斷，且通常要取樣（成本） |

把這張表變成一個可操作的判準——**碰到事故，三柱的分工是這樣**：

1. **metric 先報警、先定位「有沒有、何時、多嚴重」**：metric 是你的儀表板與告警的底層。它告訴你「結算服務的 p99 從 200ms 跳到 3s、錯誤率從 0.1% 升到 5%、從 02:14 開始」。它**不會**告訴你為什麼——它只是那個讓你知道「該起床查了」的東西。
2. **trace 接手「慢在哪一段、跨哪些服務」**：如果你的結算 pipeline 寫 guild → player → inventory → mail 四個服務（ch04），metric 說「整條慢了」，trace 才能告訴你「慢在 inventory 那一跳、它等了 2.8s」。trace 是**跨服務歸因**的唯一工具——沒有它，四個服務各自的 metric 都正常，但合起來的請求就是慢，你會卡在「到底是誰拖的」這個問題上瞎猜。
3. **log 收尾「那一筆到底發生了什麼」**：trace 指到 inventory 那一跳，你去翻 inventory 在那個 trace ID、那個時間點的 log，看到「DB 連線池耗盡、等了 2.8s 才拿到連線」。**log 是最後那一哩的細節**，但你得先有 metric 帶你到時間、trace 帶你到服務，否則在每秒幾千行的 log 海裡撈一筆，是大海撈針。

這套「**metric 定位 → trace 歸因 → log 細節**」的分工，是觀測性最實用的心智模型。它也立刻暴露了你當年的盲點：

> **你那條 NeoBards pipeline 只做了 metric，而且是繞道從 log 反推的 metric。** 你有 metric（能知道「有沒有、何時」），你有 raw log（能查「那一筆」），但你**沒有 trace**。所以當年若發生「某個 API 慢、但慢在哪個下游不清楚」的事故，你只能靠 log 的時間戳人工拼因果——這正是 trace 存在的理由。這不是說你當年做錯了（單體服務、或下游少時，沒 trace 也活得下去），而是：**你的觀測能力的邊界，就是你三柱裡缺的那一柱的邊界。**

### 三柱不是三套系統，是同一件事的三種視角（OTel 的一句話定位）

值得補一句現況：這三柱過去常是三套各自為政的系統（Prometheus 收 metric、ELK 收 log、Jaeger 收 trace），彼此對不上——metric 的尖峰跳到 log 海裡找不到對應那幾行，trace ID 串不起來。**OpenTelemetry（OTel）的存在意義，就是把這三柱（以及 2026 起 release candidate 的第四柱 continuous profiling）統一在一個 SDK、一個線上協定 OTLP、一套 semantic convention 之下**，讓「metric 異常 → 跳到同一個 trace ID → 撈到那幾行 log」這條路能串起來（2026-06：traces 規格穩定且有 LTS、metrics 資料模型穩定、logs 規格穩定但各語言實作成熟度不一）。本書不教 OTel 的 API（那是另一本書的事）；你只要知道它的定位是**三柱之間的共同骨架與關聯鍵（trace ID / resource）**，解決的就是上面那個「三套系統對不上」的痛。

## cardinality（基數）爆炸：metric 為什麼按 user 拆就死（收口 ch06）

ch06 留了一個預告：metric「怕高基數」，按 user 拆就爆。這一節把這個帳算清楚——因為它是你那條 pipeline 在 Lua filter 裡「決定聚合到哪些維度」時，隱性踩過的地雷。

**cardinality（基數）= 一個 metric 的不同標籤組合數 = 你要維護的時間序列（time series）數。** 這是理解 metric 成本的唯一公式。一個 metric 系統的成本，幾乎完全由**活躍時間序列的總數**決定，而時間序列數 = 各維度取值數的**乘積**。

乘積。這是關鍵字。維度是相乘不是相加的：

```
   時間序列數 = 維度A取值數 × 維度B取值數 × 維度C取值數 × …
```

來看 NeoBards 的數字。假設你有一個 metric「API 回應時間」，你想按幾個維度拆：

- **endpoint**：~50 條路徑
- **status**：~5 種（200/4xx/5xx 等）

到這裡：50 × 5 = **250 條時間序列**。完全沒問題——250 條序列對任何 metric 系統都是小數字。

現在來了一個看似無害的需求：「**能不能也按 user 拆，這樣我們能看每個用戶的體驗？**」NeoBards 有 **500K+ 使用者**。加上這個維度：

```
   50 endpoint × 5 status × 500,000 user = 125,000,000 條時間序列
```

**一億兩千五百萬條時間序列。** 從 250 條變成 1.25 億條——只因為加了一個維度。這就是 cardinality 爆炸：**高基數維度不是讓成本「增加」，是讓成本「乘以一個巨大的數字」。**

這個數字有多致命？把它換算成兩種真實後端的代價：

**換算一：Prometheus 這類記憶體型 TSDB。** 每條活躍時間序列在 head block 裡約佔 **3–4 KB 記憶體**（2026-06，含標籤、chunk、倒排索引）。1.25 億條 × 3.5 KB ≈ **437 GB 記憶體**——光是放序列的骨架，還沒存任何數值。Prometheus 的時間序列是常駐記憶體的，它不會優雅地換頁到磁碟，它會 **OOM 然後死掉**。所以這個維度不是「讓你的監控變貴」，是「讓你的監控伺服器當場崩潰」。

**換算二：CloudWatch 這類按 metric 計費的託管系統。** CloudWatch 的計費單位是「每個 metric name + namespace + 維度值的唯一組合算一個 metric」，**前 10,000 個每月每個 $0.30**（2026-06）。1.25 億個唯一組合 × $0.30 ≈ **每月 3,750 萬美元**（即使有階梯折扣，數量級也是天文數字）。你不會收到一個「cardinality 太高」的告警——你會收到一張帳單。

對比一下「按 user 拆」帶來的價值：你真的需要 500K 條「每個用戶的 p95」嗎？絕大多數時候不需要——你需要的是「整體 p95」「按 endpoint 的 p95」，外加「**出問題時，能下鑽到是哪些用戶受影響**」。而後面這個需求，**不該用 metric 解**——它正是 log / trace（高基數天生友善）該回答的問題。這就是三柱分工在 cardinality 上的鐵律：

> **低基數的維度（endpoint、status、region、服務名）放 metric；高基數的維度（user_id、request_id、session_id、guild_id）放 log / trace。** 把 user_id 塞進 metric 的標籤，是 cardinality 爆炸最常見的死法，而且它通常是「順手」加的——因為直覺上「多一個維度而已」，但維度是相乘的。

回頭看 ch06：你那條 pipeline 在 Lua filter 裡正規化 `/api/guild/5821/rewards` → `/api/guild/:id/rewards`（推演一那步），**那一步就是在控 cardinality**——你把 guild ID 這個高基數的東西從 endpoint 維度裡拿掉了，否則 ~5,000 公會 × 各路徑會讓你的 metric 維度直接膨脹。你當年做對了，但你大概沒意識到你做的那步的學名叫「cardinality 控制」，也沒意識到如果手一滑沒正規化，下游成本會是什麼數量級。

> **我以為 vs 實際**：
> **我以為**：metric 多加一個維度，成本只是多一點點。
> **實際**：維度是**相乘**的。加一個高基數維度（user_id）= 成本乘以那個維度的取值數（500K）。metric 的成本由「時間序列總數」決定，而時間序列數是各維度取值數的乘積——這讓「順手按 user 拆一下」變成讓監控系統 OOM 或讓帳單爆炸的單一最常見原因。

### 一個諷刺：高基數對 metric 是毒藥，卻是 observability 的全部意義

這裡藏著一個漂亮的張力，正好接到本章最後一節。你剛看到高基數是 metric 的死穴。但 Charity Majors 對「observability」的定義恰恰是：**能對你的系統問出任意新問題，而不必先預判要問什麼、不必改程式重新埋點**——而這份「問任意問題」的能力，**核心就是高基數**：能下鑽到「是哪個用戶、哪個 shopping cart、哪個 request ID 的那一筆」。

所以同一個東西（高基數）在兩個世界裡是相反的：

- 在**預先聚合的 metric** 世界裡，高基數是毒藥（每個取值組合是一條要常駐的時間序列）。
- 在**保留原始事件、查詢時才聚合**的 observability 世界裡，高基數是寶藏（它正是讓你能問「那一筆到底怎麼了」的東西）。

這不是矛盾，是兩種不同的資料模型。把它記住，因為它就是「監控 vs 可觀測」差別的技術根源——本章最後一節會收這條線。

## 為什麼看 p99/p999 不看平均：尾延遲

你給 Nexon 的是 p95，不是平均。為什麼？把這件事說透，因為它是「該觀測什麼」最違反直覺、也最常被做錯的一點。

**平均（mean）會說謊，因為它把少數慘用戶溶解在多數爽用戶裡。** 看一個具體例子。假設某 endpoint 一分鐘 1,000 個請求，延遲分布是：

- 990 個請求很快：每個 50ms。
- 10 個請求很慘：每個 5,000ms（5 秒——可能卡在鎖等待、GC、慢查詢、或某個下游逾時）。

算平均：(990 × 50 + 10 × 5000) / 1000 = (49,500 + 50,000) / 1000 = **99.5ms**。

你的儀表板顯示「平均延遲 99.5ms」——看起來**非常健康**，不到 100ms。但真相是：**有 1% 的請求等了 5 秒**。那 10 個用戶——可能是同一批高價值用戶、可能是某個特定操作（比如結算）的全部用戶——體驗到的是一個爛到會讓他們離開的服務。而你的平均值把這個事實**完全藏起來了**：50,000ms 的痛苦被 990 個快請求稀釋成了每個請求多 50ms 的錯覺。

現在看 p99（第 99 百分位，即「最慢的 1% 的門檻」）：把 1,000 個延遲排序，第 990 名就在「快」與「慘」的交界，p99 ≈ **5,000ms**。p99 直接把那 1% 的慘狀**端到你面前**——這正是你要的。

```
   延遲分布（1000 個請求）
   數量
    │
990 ┤█                         ← 990 個 @ 50ms（爽用戶）
    │█
    │█
    │█
    │█
    │█
    │█
    │█
 10 ┤█                      █  ← 10 個 @ 5000ms（慘用戶）
    └┴──────────────────────┴────► 延遲
     50ms                  5000ms
          ▲                    ▲
       平均=99.5ms          p99=5000ms
       （把慘用戶藏起來）   （把慘用戶端出來）
```

這就是**尾延遲（tail latency）**的核心：你的系統健不健康，由**尾巴**決定，不由中間決定。原因有兩個，一個關於用戶、一個關於數學：

1. **關於用戶**：尾巴上的那些慢請求，往往不是隨機散落的——它們常常**集中在同一批用戶或同一種操作**上（某個重查詢、某個倒楣命中冷快取的、某個 N+1 的）。對那批用戶，p99 才是他們的真實體驗；平均對他們毫無意義。更狠的是：一個用戶的一次操作常常要打很多次後端請求，**只要其中一次落在尾巴上，整個操作就慢**——所以「per-request 的 p99」會放大成「per-user 操作的更高比例慢」。
2. **關於數學**：在排隊系統裡，當利用率 ρ 接近 1，**尾巴會比中位數先爆炸、爆得更兇**——平均還溫和地往上爬時，p99 已經垂直飆升。為什麼尾巴對負載這麼敏感、ρ→1 時尾巴的數學形狀長怎樣，這是排隊理論的核心結果，**完整推導見《等待的數學》（queue）**。這章你要記住的結論是：**p99 是比平均更早、更靈敏的飽和警報**（Google SRE 的原話：量測 p99 回應時間能給你「飽和的極早期信號」）。

所以「看 p99 不看平均」不是一句口號，是一個有後果的設計決定。它也立刻回扣 ch06 的陷阱：**正因為 p99/p999 是你真正要看的東西，而百分位數又不能跨機器、跨視窗平均（ch06），所以你的觀測管線必須在邊緣保留可合併的結構（histogram / t-digest），絕不能在邊緣算好 p99 再去平均。** ch06 的「百分位不可合併」與本章的「要看 p99」是同一枚硬幣的兩面：**你最該看的那個量，恰好是最不能隨便聚合的那個量。**

### SLI / SLO / error budget：把「夠好」變成一個數字

知道要看 p99 之後，下一個問題是：**p99 要多低才算「夠好」？** 這就是 SLI / SLO / error budget 這套語言要回答的——它把「服務好不好」從一個吵不完的主觀爭論，變成一個能算、能告警、能拿去跟產品端談的數字。

- **SLI（Service Level Indicator，服務水準指標）**：一個量測服務好壞的**比例**。例如「回應時間 < 300ms 的請求佔比」「成功（非 5xx）請求佔比」。注意 SLI 是個比例（好事件 / 總事件），不是一個原始延遲值——這讓它能直接接到下面的 budget。
- **SLO（Service Level Objective，服務水準目標）**：對 SLI 設的**目標門檻**，含時間窗。例如「過去 30 天，99.9% 的請求回應時間 < 300ms」。SLO 是你跟自己（和產品端）的承諾。
- **error budget（錯誤預算）**：**1 − SLO**。如果 SLO 是 99.9%，你的 error budget 就是 0.1%。把它換算成具體數字：一個服務 30 天收 300 萬個請求，0.1% = **3,000 個請求**——這是你「**被允許**搞砸」的配額。用完了就該停下來修穩定性、別再推新功能；沒用完，代表你還有冒險的空間。

error budget 是這套語言裡最有威力的概念，因為它把「穩定 vs 速度」這個永恆的拉扯**變成一個非黑即白的數字**：預算還有，就放手做；預算燒光，就凍結。它也讓告警有了一個正確的底層——你不該在「錯誤率 > 0」時就告警（任何系統都有 0 不掉的錯誤），你該在「**燒預算的速度（burn rate）太快、照這速度會在窗期結束前燒光**」時告警。burn rate > 1 表示「照這速度，這個窗期會超出 SLO」；成熟的做法是**多窗多 burn-rate 告警**（一個快窗抓急性大故障、一個慢窗抓慢性滲漏），這是把 SLO 變成告警的正規方法（細節屬於 SRE 實務，本書點到為止、來源見延伸閱讀）。

> **這直接接到下一節的告警設計**：error budget / burn rate 告警是**症狀告警**的典範——它告的是「用戶正在體驗到的 SLO 違反」，不是「某個元件的內部狀態」。這就是好告警與壞告警的分水嶺。

## 故障模式與防禦

觀測這一軸的「故障」有它的特殊形狀：它不是某個系統壞掉，而是**觀測本身壞掉**——你以為你看得見，其實你瞎了，或者你被一堆假警報訓練成無視真警報。這一軸最招牌的故障全都集中在**告警設計**上：症狀告警 vs 原因告警，以及為什麼「CPU 高」是壞告警。這是你當年大概設得最隨意、但後果最直接（半夜會不會被叫醒）的部分。把告警分成兩種，這個分類會改變你設每一個告警的方式：

- **症狀告警（symptom-based）**：告「**用戶正在經歷的痛**」。p99 延遲超過 SLO、錯誤率超標、結算成功率掉了、可用性破了。它直接對應「有人正在受苦」。
- **原因告警（cause-based）**：告「**某個內部元件的狀態**」，這個狀態**可能**導致問題。CPU > 80%、記憶體 > 90%、連線池快滿、佇列積壓、磁碟快滿。它對應「某個東西看起來不太對」。

**鐵律：對症狀告警（叫醒人），用原因當診斷線索（不叫醒人）。** 為什麼「CPU 高」是個壞的**告警**（但是個好的**診斷指標**）？因為原因告警在兩個方向上同時害你：

1. **誤報（false positive）→ 告警疲勞**：CPU 衝到 90% 但**用戶根本沒事**——可能只是一個批次任務、一次正常的流量尖峰、一次 GC。你被半夜叫醒，爬起來一看，p99 正常、錯誤率正常、用戶沒抱怨。這種「狼來了」累積起來就是**告警疲勞（alert fatigue）**——當你的告警有一半是假的，你會開始無視所有告警，包括那個真的。**一個會誤報的告警比沒有告警更糟**，因為它訓練你忽略它。
2. **漏報（false negative）→ 真出事時它沒響**：反過來，**CPU 完全正常，但用戶慘了**——服務卡在一個下游逾時上、卡在鎖等待上、卡在一個沒佔 CPU 的 I/O wait 上（你的 event loop 被一個 await 卡住，ch12——CPU 可能很閒，但所有請求都在排隊）。用戶的 p99 飆到 10 秒，但你那個「CPU > 80%」的告警**安安靜靜**——因為這個故障不經過 CPU 這條路。原因告警只能抓「它預期的那個原因」，抓不到它沒預想到的原因。

把這兩個方向合起來看，原因告警的根本問題是：**原因和症狀之間不是一一對應的。** 一個原因（CPU 高）不一定導致症狀（用戶不一定慘）；一個症狀（用戶慘）可以由無數個你沒預想到的原因造成（CPU 根本不在其中）。所以：

> **症狀告警可靠的根本原因**：它直接量「用戶有沒有在受苦」，這個問題**只有一個答案**，且**與底層怎麼壞無關**。不管是 CPU、是鎖、是下游、是某個你這輩子沒見過的故障，只要它讓用戶的 p99 飆、錯誤率升，症狀告警就會響。它對「故障的形狀」是**不可知（agnostic）**的——這正是它能抓到「未知的未知」的原因。CPU 告警只能抓「CPU 這條已知的路」。

這也呼應 Google SRE 的「四個黃金信號」（latency、traffic、errors、saturation）——它們都是用戶側的症狀信號。SRE 書裡有一句話直接點破本章主題：**最常見的監控失敗不是漏掉信號，而是「告錯了每個信號的表現形式」——用平均延遲而不是 p99、用錯誤計數而不是錯誤率、用當前飽和度而不是飽和度的變化速度。** 這把本章前面三節（看 p99、看比例、看趨勢）和告警設計縫在了一起。

那原因告警（CPU、記憶體、佇列長度）就沒用了嗎？有用——**但它的位置是儀表板和診斷，不是叫人起床的告警**。症狀告警響了（p99 飆了），你爬起來看儀表板，這時 CPU、連線池、佇列積壓這些原因指標幫你**快速縮小是哪裡壞的**。Brendan Gregg 的 **USE 方法**（Utilization / Saturation / Errors，針對資源）和 Tom Wilkie 的 **RED 方法**（Rate / Errors / Duration，針對服務）就是這兩層的工具：RED 是服務層的症狀（拿來告警），USE 是資源層的診斷（拿來在症狀告警響後查根因）。**症狀告警把你叫醒，原因指標幫你查清——別把順序顛倒。**

把這條規則用回你的系統：你當年若設了「結算 consumer 的 SQS 佇列積壓 > 10,000 就告警」——這是個**原因告警**。佇列積壓不一定代表玩家有事（可能只是你在補一批歷史結算、刻意慢慢消）；而玩家真的沒收到獎（症狀）時，佇列可能根本不長（consumer 正常消，只是消錯了、發成 0 元、或冪等鍵撞掉了）。正確的告警是症狀層的——下一節 worked example 就來設計它。

## 紙上推演

### 推演一：手算 per-user 維度的 metric 在 500K 用戶下的基數帳 **[15 分鐘] ★★**

你有一個「API 回應時間」metric，目前按 `endpoint`（50 條）× `status`（5 種）拆。現在產品端要求「也按 user 拆，這樣能看每個用戶的延遲」，NeoBards 有 500K 使用者。請：(a) 算出加上 user 維度後的時間序列總數；(b) 換算成 Prometheus（每序列約 3.5 KB 記憶體）大概要多少記憶體；(c) 說明這個需求**真正想解的問題是什麼**，以及為什麼它不該用 metric 解、該用哪一柱。

### 推演二：為「結算 pipeline 發錯獎」設計三個 SLI **[20 分鐘] ★★★**

你的結算 pipeline（CronJob → SQS → 冪等 consumer → guild/player/inventory/mail，每季 10–30 萬筆）可能以一種陰險的方式壞掉：**它跑得「成功」，但發錯了**——發成 0 元、發給錯的人、漏發、重複發（冪等失效）。請設計**三個 SLI**，讓你能在這種「成功地發錯」第一時間發現。要求：每個 SLI 寫成一個比例（好事件 / 總事件），說明它各抓哪種錯，並指出為什麼光看「consumer 處理成功率」這個 metric 抓不到「發錯」。

### 推演三：判斷三個告警哪個是症狀、哪個是原因 **[15 分鐘] ★★**

下面三個告警，判斷各是症狀告警還是原因告警，哪些該拿來「半夜叫醒人」、哪些只該放儀表板當診斷，並各說一個它會**誤報**或**漏報**的具體場景：

1. 「Node 行程 CPU > 85% 持續 5 分鐘」
2. 「LetsTalk 訊息送達成功率（已送達 / 應送達）過去 10 分鐘 < 99%」
3. 「Redis 記憶體使用 > 90%」

### 推演解答

**推演一**：

(a) 時間序列數 = 50 × 5 × 500,000 = **1.25 億條**（維度相乘，不是相加）。

(b) 1.25 億 × 3.5 KB ≈ **437 GB 記憶體**——只放序列骨架（標籤＋索引＋chunk），還沒存任何數值。Prometheus 的序列常駐記憶體、不優雅換頁，這個量會直接讓它 **OOM 死掉**。若是 CloudWatch（每唯一組合前 10,000 個每月 $0.30），1.25 億組合是天文數字帳單。**結論：這個維度不是讓監控變貴，是讓監控崩潰。**

(c) 「按 user 拆」真正想解的問題是：「**出問題時，能下鑽到是哪些用戶受影響**」——這是個**事故時的下鑽需求**，不是個**常態聚合需求**。你不需要 500K 條常駐的「每用戶 p95」時間序列（99.99% 的時間沒人看）；你需要的是「整體 / 按 endpoint 的 p95」（metric，低基數）**外加**事故時能撈「哪些 user_id 在那段時間慢了」（log / trace，高基數天生友善）。**user_id 屬於 log/trace，不屬於 metric 標籤。** 正確回應產品端的方式不是「加 user 維度到 metric」，而是「metric 保持低基數監控整體健康，事故時用 log/trace 下鑽到具體用戶」。

**推演二**：三個 SLI（核心洞察：「處理成功率」只證明 consumer **沒崩**，不證明它**發對了**——要從「程式有沒有報錯」轉向「結果對不對」）：

1. **金額正確率**：`發出獎勵金額 = 應發金額的筆數 / 總結算筆數`。抓「發成 0 元 / 發錯金額」。實作上需要一個獨立的「應發」事實來源對帳（結算前就算好的預期值），比對實發。光看 consumer 成功率抓不到——consumer 把 0 元寫進 DB 也是「成功」。
2. **四服務一致率**：`四個服務（guild/player/inventory/mail）都成功落帳的筆數 / 總筆數`。抓「部分失敗」——寫了 player 沒寫 inventory（ch04 的最終一致破口）。consumer 對單一服務的寫入成功不代表跨四服務一致；要一個對帳作業去查「這筆在四個服務都齊了嗎」。
3. **去重正確率（無雙發）**：`恰好發一次的結算 ID 數 / 總結算 ID 數`，即「被處理超過一次且都真的落帳」的比例要趨近 0。抓「冪等失效導致重複發」（ch04 的 SQS 重投 + 冪等鍵沒擋住）。consumer 把同一筆處理兩次、兩次都「成功」——成功率 100%，但玩家收到兩份。

**為什麼「處理成功率」抓不到「發錯」**：處理成功率量的是「程式有沒有拋例外 / 有沒有 ack」，它對「寫進去的**值對不對**」完全盲目。發成 0 元、發給錯的人、發兩次——在 consumer 眼裡都可能是「成功處理」。要抓「成功地發錯」，SLI 必須建立在**對帳（reconciliation）**上：拿一個獨立的「應該是什麼」事實來源，比對「實際是什麼」。**這就是「觀測是設計的一部分」的具體例子——你得在設計 pipeline 時就留下「應發」這個事實來源，事後是補不出來的。**

**推演三**：

1. **「CPU > 85% 持續 5 分鐘」= 原因告警**，**不該叫醒人**，放儀表板當診斷。**誤報場景**：一個季度結算批次正在跑、CPU 衝高，但玩家請求的 p99 完全正常、沒人受苦——你被白叫醒。**漏報場景**：event loop 被一個 await 卡住（ch12），所有請求排隊、p99 飆到 10 秒，但 CPU 在 I/O wait、根本不高——這個告警安靜，用戶已經慘了。
2. **「訊息送達成功率 < 99%」= 症狀告警**，**該叫醒人**。它直接量用戶有沒有收到訊息（用戶側的痛），對「為什麼沒送達」不可知——不管是 WebSocket 斷、Redis Pub/Sub 扇出掉（ch07）、還是某台 WS server 掛了，只要送達率掉它就響。這是正確的「叫人起床」告警。（唯一要小心的是「應送達」的分母怎麼定義得可靠——這本身是個觀測設計題。）
3. **「Redis 記憶體 > 90%」= 原因告警**，**不該叫醒人**，放儀表板。**誤報場景**：Redis 設了 `allkeys-lru`、記憶體本來就會貼著上限跑（到頂就驅逐冷 key，這是正常運作不是故障，ch17），90% 是常態不是事故——天天誤報。**漏報場景**：記憶體只有 60%，但有個 `KEYS *` 把 Redis 單執行緒卡死（ch17），所有依賴 Redis 的請求都慢了，這個告警毫無反應。

## 自我檢核

口頭自答，講得出來才算過關：

1. metric / log / trace 三柱各擅長回答哪一類問題、回答不了哪一類？碰到「某 API 慢、但慢在哪個下游不清楚」的事故，三柱的查證順序是什麼？你當年的 NeoBards pipeline 缺了哪一柱、那道缺口讓你查不到什麼？
2. 為什麼「按 user 拆 metric」會讓 metric 系統 OOM 或帳單爆炸？把那個乘法帳大致算出來（50 × 5 × 500K）。這個需求真正想解的問題是什麼、該用哪一柱解？
3. 為什麼「高基數」對 metric 是毒藥、對 observability 卻是全部意義？這兩者差在哪種資料模型？
4. 用 990 個 50ms + 10 個 5000ms 的例子，講出「為什麼平均會說謊、p99 不會」。為什麼尾延遲（不是中位數）決定服務健不健康？
5. SLI / SLO / error budget 三者的關係？error budget = 1 − SLO 為什麼這麼有威力（它把什麼拉扯變成一個數字）？為什麼該告警在「burn rate」而不是「錯誤率 > 0」？
6. 為什麼「CPU > 80% 就告警」是壞告警？把它害你的兩個方向（誤報→告警疲勞、漏報→真出事它不響）各舉一個你系統裡的具體場景。
7. 症狀告警為什麼比原因告警可靠？「對故障的形狀不可知」是什麼意思、為什麼這讓它能抓到「未知的未知」？原因指標（CPU、佇列長度）正確的位置是哪裡？
8. 「監控」和「可觀測」差在哪（已知的未知 vs 未知的未知）？為什麼這個差別的技術根源就是「高基數」？

### 本軸（觀測）回指各系統

本章是「觀測」這一軸本身的專章，不對單一系統做五軸體檢，而是把這一軸用回前面各系統，補上它們當年的觀測盲點：

| 系統 | 觀測這一軸的答案與當年的盲點 |
|---|---|
| **FluentBit + Lua metric pipeline（ch06）** | 只做了 metric（且從 log 反推）、缺 trace；管線本身的存活、解析失敗率、邊緣算 p95 後平均的陷阱都需額外觀測，否則靜默變錯且「看起來變好」。 |
| **結算 pipeline（ch04）** | 「處理成功率」抓不到「成功地發錯」；需要建在對帳上的 SLI（金額正確、四服務一致、無雙發），且「應發」事實來源必須在設計時就留下。 |
| **即時通知（ch07）** | 正確的症狀告警是「送達成功率」（用戶側），不是「Redis 記憶體 / 連線數」（原因）；Pub/Sub fire-and-forget 讓「掉了沒」本身難觀測，分母（應送達）的定義是觀測設計題。 |
| **event loop（ch12）** | event loop 卡死時 CPU 可能很閒——所以「CPU 告警」漏報這類故障；要監測的是 event loop lag（症狀）而非 CPU（原因）。 |
| **共通教訓** | 症狀告警叫人、原因指標診斷；看 p99 不看平均；高基數放 log/trace 不放 metric；**觀測系統自己也要被觀測**（ch06 的 dead man's switch）。觀測不是事後補的，是寫第一行程式時就決定的事。 |
