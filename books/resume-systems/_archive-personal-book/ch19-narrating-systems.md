## 從你已知的出發

你的 plan.md 和你給自己定的轉職論點，都站在同一根支柱上：**「6 年資深後端、做過高併發與分散式系統。」** 這句話本身沒問題——你真的做過。問題在於，面試官不會因為你說了這句話就信你；他會問一個系統，然後看你**講得多深**。而你過去講這些系統的方式，多半是這個形狀：

> 「我在 NeoBards 把 RDS 尖峰 CPU 降了 40%，做法是重排 SQL、拆交易、批次化。」
> 「我做過一條結算 pipeline，CronJob 觸發、進 SQS、consumer 冪等處理。」
> 「我自建了一套即時通知系統，WebSocket 加 Redis Pub/Sub。」
> 「我評估了 Twilio 和 Agora，因為成本選了 Jitsi。」

這四句話**每一句都對**，而且每一句都是真的你做的事。但它們有一個共同的毛病——**它們是名詞的清單，不是理解的證明**。面試官聽到的是「這個人會用這些工具」，而不是「這個人懂這些工具底下在發生什麼」。在資深職缺的面試裡，這兩者天差地別：前者是初級到中級的訊號，後者才是你想傳達的「資深」。

更尷尬的是——你過去這樣講，**不是因為你不懂**，是因為你當年確實沒把它挖穿，所以講不出底下那一層。你做對了，但你說不清楚為什麼有效。讀完前面十八章之後，這件事變了：現在你知道那 40% 是鎖持有時間崩塌、不是算力；知道結算 pipeline 的冪等是架構前提、不是保險；知道 Pub/Sub 是門鈴、不是信箱；知道選 Jitsi 的真正理由是成本與掌控、不是「Twilio 要關了」。

**這一章要做的，就是把「你現在懂了」翻譯成「你講得出來」。** 但我要先講一句潑冷水的話，因為它決定了這整章的態度：

**講得出來，跟講得對，是兩件事。** 一個把過時事實講得很流暢的敘事，比一個結結巴巴但正確的敘事更危險——因為前者會在面試官追問時當場崩塌，而且崩塌得很難看（你不只答錯，還顯得「你以為你懂但其實沒查證」）。你大概沒想過的是：**面試級敘事的第一個要求不是「動聽」，是「禁得起追問」。** 這一章給你的模板，每一格都是為了「禁得起追問」而設計的。

## 敘事模板：從「我做了 X」到「禁得起追問的 X」

先把模板本身立起來。一個資深級的系統敘事，要把「我做了 X」這句光禿禿的話，升級成五個層次都填滿的結構：

```
   「我做了 X」 ← 名詞，初級訊號，到此為止會被追問打穿
        │
        │ 往下填五層
        ▼
   ① 情境與量測   為什麼要做？數字長什麼樣？（不是「感覺慢」是「CPU 90%」）
   ② 機制         X 底下實際怎麼運作？我動的是哪個機制、為什麼這樣有效？
   ③ 取捨         我放棄了 Y、Z——它們各輸在哪一軸？（成本/延遲/運維/故障域）
   ④ 故障與防禦   X 會怎麼壞？壞了長什麼樣？我怎麼防、或當年沒防到什麼？
   ⑤ 結果與誠實   數字怎麼動？哪些是我當年就懂、哪些是我最近才挖穿的？
```

這五層不是隨便排的，它對應全書的脊椎。①是量測（觀測軸的起點），②③是你前面每一章挖的機制與取捨，④是本書的招牌「故障模式與防禦」段落，⑤是「我以為 vs 實際」這條暗線的誠實版。**把任何一個系統套進這五層，你就從「報名詞」變成「展示理解」。**

幾條讓這個模板真正有殺傷力的規則：

**規則一：每個取捨都要說得出「被放棄的選項輸在哪」。** 這是全書反覆敲的釘子，在面試桌上是區分資深的最快訊號。「我選了 SQS 標準佇列」是名詞；「我選標準佇列而不是 FIFO，因為我不需要排序、FIFO 的吞吐上限對 30 萬筆是純損失、而它的 5 分鐘去重窗口擋不住我真正的重複來源（CronJob 重疊）」——這是理解。後者讓面試官知道你不是「會用 SQS」，是「知道 SQS 的每個選項各自的代價」。

**規則二：主動把故障端上桌，別等他問。** 大部分候選人講系統只講 happy path，等面試官問「那它怎麼壞」才開始想。你反過來——**主動說出這個系統會怎麼壞、你當年防了沒**。「這條 pipeline 最危險的不是雙發（那會被客訴、容易發現），是靜默少拿——冪等鍵粒度設錯會讓玩家跨季漏發、沒有錯誤日誌、只有對帳抓得到。」一個主動講故障模式的候選人，傳達的是「我設計系統時會先想它怎麼死」——這正是資深工程師的核心習慣。

**規則三：誠實標出「當年懂」與「最近才挖穿」的邊界。** 這條反直覺，但極其有效。你不需要假裝六年前的自己就全懂。事實上，**「我最近回頭把這個系統挖穿，發現我當年其實是憑直覺答對的，但講不清楚為什麼——現在我能講了」** 這句話，比假裝全程都懂更可信、更有說服力。它傳達兩個正面訊號：你有持續深掘的習慣（資深的成長性），以及你誠實（不吹）。面試官對「全都懂、從不犯錯」的敘事是有戒心的；對「我做對了但最近才真正理解為什麼」反而會點頭。

**規則四：每個時效性事實，面試前重新查證。** 這是規則裡最不浪漫、但會救你一命的一條。你的記憶停在 2019–2024，而很多事實已經變了——Twilio Video 沒關（撤回了）、Bull 已被 BullMQ 接替、Redis 改了授權還分裂出 Valkey、Clinic.js 停止積極維護、SQS 單則上限升到 1 MiB、Lambda 的 async payload 也升到 1 MB。**用一個過時事實當你敘事的支柱，就是給面試官一把當場拆穿你的刀。** 下面每個示範敘事，我都會標出那個「會腐的點」。

## 示範一：RDS CPU −40%（量測 → 診斷 → 鎖機制 → 優化 → 數據）

這是你最招牌的數字，也是最容易講成名詞清單的一個。先看反面教材，再看升級版。

**反面教材（名詞清單）**：

> 「NeoBards 的 RDS 尖峰 CPU 很高，我重排了 SQL、拆了交易、做了批次化，把 CPU 降了 40%。」

這段話的問題：它把「降 40%」歸因於三個動作，但**沒解釋為什麼這三個動作有效**。面試官追問一句「為什麼重排 SQL 順序會降 CPU？SQL 不是一樣的嗎？」你如果只能回「就是會比較快」，這段敘事就垮了。

**升級版（五層都填滿）**：

> **①情境與量測**：NeoBards 是一款 500K+ 使用者、10K+ CCU 的手遊。每到結算和活動尖峰，RDS（MySQL/InnoDB）的 CPU 就頂到 90% 以上、貼著天花板走平頂線。我從 CloudWatch 看到 CPU 尖峰，再用 slow query log 找出反覆出現的肇事 query。
>
> **②機制**：這裡是關鍵——我後來想清楚，那條 CPU 線**主要不是算力問題，是鎖競爭問題**。InnoDB 的行鎖要到交易 commit 才釋放，不是 update 完就放。所以一個交易只要中間有任何一步慢——比如包了一個外部 API 呼叫——它已經鎖住的那幾行就陪著一起被凍結那麼久。一個交易持有鎖的時間，等於所有想動那一行的其他交易被迫排隊的時間。尖峰時幾百個交易為了一個慢交易排隊，鎖一放開又一起驚群湧上來重新解析、重新掃描，加上很多交易等到 `innodb_lock_wait_timeout`（預設 50 秒）超時被回滾、應用層重試——CPU 就是被「等待後的驚群重啟 + 超時重試」雙重燒上去的，不是算術變多。
>
> **③取捨**：我當年的四個手段，其實是同一件事的四個切面——全都在縮短鎖持有時間。重排 SQL 順序是統一鎖獲取順序、根除死結（省掉回滾重試）；拆交易是把外部呼叫移出交易、讓鎖只持有幾毫秒；跨交易重排是讓最搶手的熱點行最晚拿、最早放;批次化是把 N 次往返壓成 1 次、讓那批行的鎖持有時間一起崩塌。而我**沒有選**「換更大的 RDS 機型」——因為 scale up 解的是算力問題，但鎖競爭是串行化瓶頸，加 CPU 核心並不能讓兩個交易同時握同一把行鎖。被放棄的那個選項輸在它解錯了問題。
>
> **④故障與防禦**：拆交易最危險——它直接攻擊持有時間，但它把原子性親手交了出去。能拆的是「兩步之間無守恆關係」的（比如更新登入時間和在線狀態）；不能裸拆的是「有借貸關係」的（轉帳、扣庫存建單）——那些拆下去會留下永久落地的中間態：玩家收到 coins 沒收到道具，而 DB 裡查不到任何錯誤，因為每個小交易自己都成功了。我當年沒爆，是因為我只拆了能拆的那些。
>
> **⑤結果與誠實**：尖峰 CPU 降 40%。但我要誠實講——當年我是憑工程直覺做對的，知道「拆交易降鎖競爭」「批次比逐列好」，但講不清楚底層為什麼。**我最近回頭把它挖穿，才真正理解那 40% 不是省了 40% 的算術，是關掉了一個「等待→重試→驚群」自我放大的浪費迴圈**——所以它的改善才會是非線性的、斷崖式的，而不是「快一點點」。

注意這段敘事的形狀：它從一個數字出發，往下挖到機制，回頭解釋每個動作為什麼有效，主動端出取捨和故障，最後誠實標出「最近才挖穿」的邊界。**面試官問什麼，你都有下一層。**

**這段敘事的會腐點**：`innodb_lock_wait_timeout` 預設值（50 秒）、InnoDB 預設隔離級別（Repeatable Read，跟 PostgreSQL 的 Read Committed 不同）這類數字面試前值得快速確認；它們相對穩定，但別憑「我記得」就斷言。

## 示範二：結算 pipeline（交付語意 + 冪等 + 最終一致）

這是你最能展示「分散式系統思維」的系統，也是 worked example 會完整鋪開的那個。這裡先給濃縮版，完整版在後面。

這個系統的完整面試敘事（六段展開 + 三個追問）是本章 worked example（推演一）的內容，這裡只給濃縮骨架、把細節留到那邊：

> **①情境**：每季把公會排名換算成獎勵——10–30 萬筆、跨 ~5,000 公會、每筆動 guild/player/inventory/mail 四個服務。架構 K8s CronJob → SQS → 冪等 consumer。
>
> **②機制**：核心是「為什麼一定要冪等」。SQS 標準佇列 at-least-once——它**架構上保證**重複投遞。機制在 visibility timeout:consumer 撈訊息後它變不可見一段時間、處理完才刪;只要 consumer 慢了或當機，SQS 在到期時重投——因為它**無法分辨「死了」和「只是慢」**，安全選擇永遠是再投一次。所以冪等不是保險，是這個架構的前提。
>
> **③取捨**：冪等鍵用業務身分（季+公會+玩家+獎勵類型）不用 SQS message ID——因為重複有**兩個獨立來源**:下游 SQS 重投、上游 CronJob 重疊。「檢查+寫入」必須原子，靠資料庫唯一約束（`INSERT ON CONFLICT DO NOTHING`），不能用 check-then-act。也評估過換 FIFO——但它的 exactly-once 只是 5 分鐘去重窗口、擋不住跨窗口的 CronJob 重疊，排序保證我不需要、吞吐上限是純損失。結論:冪等交給業務層、SQS 用最便宜的標準佇列。

四服務的部分失敗、重試/補償/對帳、為什麼「靜默少拿」比雙發更危險、以及「冪等是前提不是保險」的誠實收尾——這些在推演一完整展開，並標出三個追問的下探路徑。**這就是這個系統值得當 worked example 的原因:它的每一層都禁得起往下挖。**

## 示範三:通知系統（WebSocket + Pub/Sub 扇出 + 一個誠實的能力邊界）

這個系統最能展示「我知道我用的工具不是它聽起來的那個東西」——也就是全書「我以為 vs 實際」的精華。

**升級版骨架**：

> **①情境**：LetsTalk 是隱私導向 IM，主力市場含中國，而 FCM 在長城防火牆後連不上。我不能把「使用者收不收得到通知」押在一條對核心市場斷掉的鏈路上，所以自提自建——WebSocket 長連線 + Redis Pub/Sub 跨機扇出，~3,200 並發、~10M 訊息/月。
>
> **②機制**：兩個關鍵。第一，WebSocket 連線是 stateful 的，那條連線是活在某一台特定 server 記憶體裡的 socket，所以 sticky session 幾乎強制、跨機找人要靠 backplane——我用 Redis Pub/Sub。第二，也是最重要的——**Redis Pub/Sub 不是訊息佇列**。它是 fire-and-forget、at-most-once、零持久化:`PUBLISH` 只送給「此刻正連著」的訂閱者然後立刻忘記，訂閱者斷線 0.5 秒那期間的訊息永久消失，而且不報錯（`PUBLISH` 回 0 不是錯誤）。
>
> **③取捨與架構**：所以真正的對話訊息（10M/月）不能靠 Pub/Sub 保證。我的架構是**持久層 + 即時信號層分離**——訊息本體先寫 DB（真相來源），再用 Pub/Sub 推一個「你有新訊息」的即時信號。門鈴沒響（Pub/Sub 掉了），訊息還在 DB，使用者重連拉取照樣拿得到。我也評估過把 Pub/Sub 換成 Redis Streams（有持久化、at-least-once）——但即時門鈴這一層本來就允許丟（DB 兜底），硬塞 Streams 是用重工具解一個允許丟的問題，徒增記憶體與複雜度。
>
> **④故障與誠實的能力邊界**：這套系統有個我必須誠實講的邊界——**它解的是「App 開著、連線活著」時的在線即時推送，不是整個 FCM**。手機 App 切背景後，OS 會砍掉長連線,我的 Pub/Sub 推什麼都到不了。FCM 之所以不可替代，是因為它有三項作業系統特權：OS 級單一共用連線（省電）、系統級喚醒 Doze 中的裝置、雲端離線佇列存約 4 週。這三項是平台特權，我的應用層連線拿不到。所以我補的是「FCM 斷掉的那塊在線即時」，離線推送在中國得另接本地廠商通道。
>
> **⑤結果與誠實**：穩定跑了 10M 訊息/月。如果面試官問「那離線推送你怎麼辦」，我不會假裝自建涵蓋了一切——我會誠實說我補的是在線那一段，離線是另一條路。**這個邊界講清楚，比假裝全包更能展示我真的懂這套東西的能力與限制。**

注意⑤這個轉折——**主動暴露能力邊界，反而是強敘事**。一個能準確說出「我的方案不能做什麼、為什麼」的候選人，比一個把自建講得無所不能的候選人可信得多。

## 示範四：Jitsi 選型（成本 / 自架 / 授權，含一個必須校正的老兵記憶）

這個系統的價值不在技術多難，在於它是**「老兵記憶會過時、敘事必須查證」的活教材**。如果你講錯了，它會在面試桌上當場爆炸。

**這裡是全章最重要的一句話**：你大概聽過、甚至自己講過「我當年選 Jitsi 是因為 Twilio Video 反正要關了」。**這個論點現在不能再用了。**

事實的時間線（面試前務必重新查證，這正是「會腐的點」）：Twilio 確實在 2024-03 宣布 Programmable Video 將於 2026-12-05 EOL——所以「Twilio Video 要關了」這句話在 2024 年中曾經為真。但 Twilio 在 **2024-10-21 公開撤回**了這個決定，官方原文是「we've reversed our earlier decision to retire Twilio Video」「Twilio Video will remain a standalone product」。所以 2026-06 的現況是：**Twilio Video 沒關、仍是現役產品**（2026-06）。

如果你今天在面試裡說「我選 Jitsi 因為 Twilio 要停運」——面試官只要剛好知道這件事被撤回了,你的論點當場垮掉，而且垮得很難看：你不只用了一個錯的理由，還暴露了「你的技術認知停在過時的時間點、沒有重新查證的習慣」。對一個要展示「持續跟進技術現況」的候選人，這是致命傷。

但這**不代表你當年選錯了**。選擇對、某一條支撐理由過時了——這兩件事要分開。下面是**今天仍然成立、不會過時**的取捨論述：

> **①情境**：LetsTalk 要做群組語音視訊，這是 WebRTC 的活，不是 WebSocket。我評估了 Twilio Programmable Video、Agora，最後自架 Jitsi（Videobridge 那套 SFU）、Socket.io 做信令、自架 coturn 做 TURN/STUN。
>
> **②機制（展示我懂 WebRTC 不是黑盒）**：WebRTC 不是一個協定，是一套——媒體（SRTP）、傳輸（DTLS）、連線建立（ICE/STUN/TURN），而信令它刻意不定、留給我自己挑，所以我用既有的 Socket.io 通道搬 SDP offer/answer 和 candidate。群組為什麼非要伺服器不可？因為 mesh 的上行是 N−1 路，8 人就是每個手機要編碼上傳 7 路，先壓垮的是每個 client 的上行頻寬和裝置，不是伺服器。所以要 SFU——Jitsi Videobridge 走的就是 SFU：轉發不轉碼,每人上行降到 1 路，又不像 MCU 那樣付即時轉碼的天價 CPU。
>
> **③取捨（不會過時的三條理由 + 一條代價）**：我選 Jitsi 的真正理由是三條——**成本**（Twilio/Agora 都是商用 SDK 按用量計費、隨規模線性長大，Jitsi 開源自架零授權費）;**自架掌控**（隱私導向 IM 把媒體交給第三方與品牌承諾有張力,自架讓媒體路徑完全自有）;**開源授權**（Jitsi 是 Apache 2.0，可自架可改）。我為此**付出的代價**是自運維——JVB、coturn、擴容、監控全變我的事，這是選商用 SDK 本可省下的。
>
> **④故障與防禦**：最陰險的故障是「沒配 TURN」和「relay 比例失控」——它們只在一部分使用者身上發生（symmetric NAT 後那群），而我自己在 cone NAT 的辦公網路永遠重現不出來。symmetric NAT 下 STUN 探到的對外位址對真正的對端無效（映射 destination-dependent），所以必須退回 TURN relay——沒配 TURN 等於對那群人完全不可用。所以我得監控「走 relay 的通話比例」這個 metric。
>
> **⑤結果與誠實**：選 Jitsi 在我這個規模和隱私定位下站得住。而且我要特別講——**「Twilio 要關了」這個理由我不用了，因為它被官方撤回了**;我的論述靠成本、掌控、授權這三條，把 Twilio 的產品狀態拿掉也依然成立。一個不會因為某家廠商的產品決策而失效的選型理由,才是好理由。

注意⑤這個收尾的威力：**你主動承認一個曾經的理由過時了，並用更穩固的理由取代它**。這展示的不是「我犯過錯」，而是「我有重新審視自己論點、用查證取代記憶的習慣」——這恰恰是面試官想在資深候選人身上看到的。

## 面試官追問時，往哪一層下探

模板的五層也是一張**下探地圖**。面試官的追問幾乎總是沿著「現象 → 機制 → 取捨 → 故障」這條軸往下挖，你要預先知道每一層的答案藏在哪：

```
   面試官的問              你下探到哪一層            答案來源
 ─────────────────────────────────────────────────────────────
 「你做了什麼？」      →  現象層（量測+動作）    →  ①情境與量測
 「為什麼這樣有效？」  →  機制層                →  ②機制（前面各章）
 「為什麼不選 Y？」    →  取捨層                →  ③被放棄選項輸在哪
 「那它怎麼壞？」      →  故障層                →  ④故障模式與防禦
 「你怎麼知道它壞了？」→  觀測層                →  ④裡的可觀測徵兆
```

關鍵心法：**面試官每往下問一層，你就該已經站在那一層等他。** 一個只準備到「現象層」的候選人，問到「為什麼有效」就開始即興發揮、越講越虛;一個準備到「故障層」的候選人，問到任何一層都有結構化的答案。前面十八章就是你的下探庫存——每個系統你都做過完整的五軸體檢，所以無論他從哪個角度切，你都有下一層。

舉個具體的下探鏈，以 RDS 為例：

- 他問「你做了什麼」→ 你答「降 40%，重排 SQL、拆交易、批次化」。
- 他問「為什麼重排 SQL 順序會有差，query 不是一樣嗎」→ 你下探到機制：「兩個機制——統一鎖獲取順序根除死結（省回滾重試），熱點行延後鎖降等待結構。」
- 他問「那拆交易有什麼風險」→ 你下探到故障：「拆掉了原子性。能拆的是無守恆關係的，轉帳那種拆下去會留下永久落地的中間態。」
- 他問「你怎麼發現 CPU 是被鎖等待燒的，不是算力」→ 你下探到觀測：「只看 CloudWatch CPU% 會誤判成算力，要看 Performance Insights 的 `row_lock_wait` wait event,才分得出 CPU 是 on-CPU 算術還是等待後的重試。」

**每一問你都已經在下一層等他。** 這就是「機制級理解」在面試桌上的兌現方式。

## 誠實邊界：別假裝當年就全懂

這一節違反大部分人的面試本能——本能是「裝得越懂越好」。但對你這個情況，**誠實標出「當年懂」與「最近才挖穿」的邊界，是更強的敘事**，原因有三：

- **它更可信。** 聲稱六年前就把所有機制都想透了的人，面試官心裡是打問號的——那不符合真實的工程成長軌跡。真實軌跡是「當年憑直覺做對了很多、最近回頭把直覺補成理解」。這個軌跡本身就是「資深 + 持續成長」的證明。
- **它把兩個籌碼分開、且都不削弱。** 「我把 RDS 降了 40%」是「我做對了」的鐵籌碼;「我能講清楚那是鎖持有時間崩塌」是「我懂為什麼」的籌碼。承認這兩個是分兩個時間點拿到的，反而讓後者更可信（你沒假裝它一直都在）。
- **「我最近回頭把它挖穿，發現……」是強開場。** 它傳達:我會對舊作品做深度復盤、不滿足於「能跑就好」、在沒人要求下主動把理解推到機制層——全是資深特質。比起「我當年就……」，它讓面試官看到一個還在深掘的人，不是吃老本的人。

具體怎麼標這條邊界？給你幾個可直接用的句式：

- 「當年我是憑工程直覺做對的——我知道拆交易降鎖競爭，但講不清楚底層;最近我把它挖穿，才理解那 40% 是關掉了一個自我放大的浪費迴圈。」
- 「我當年說『保險起見讓 consumer 冪等』——現在我會重講這句話：冪等不是保險，是這個 at-least-once 架構的前提。」
- 「我以前大概心裡某個角落覺得『發進 Redis 就安全了』——後來才真正想清楚 Pub/Sub 是門鈴不是信箱，真正的不丟靠的是 DB 那層。」

注意這些句式的共同結構：**承認舊的（不那麼準確的）理解 → 講出新的（機制級的）理解**。這個「從 A 到 B」的位移，本身就是你想展示的東西。

唯一的禁區：**誠實不等於貶低自己當年的工作。** 別說「我當年其實亂搞」「我當年根本不懂」——你當年做對了，那是真本事。要說的是「我當年憑直覺做對了，現在我能解釋為什麼對」。前者是自我否定，後者是成長敘事。差別在語氣，但面試官聽得出來。

## 把 RDS 那個模板複製到每個系統

「RDS CPU −40%」之所以好用，不只因為它是個漂亮數字，更因為它**天生就是五層模板的完美範本**:有量測（CPU 90%）、診斷（slow query log）、機制（鎖持有時間）、優化（四手段）、數據（−40%）。這個「量測 → 診斷 → 機制 → 優化 → 數據」的結構**可以複製到每個系統**——哪怕那個系統沒有現成的漂亮數字。方法是對每個系統問同樣五個問題、逼出同樣五層：

| 系統 | 量測（從哪看到問題） | 機制（底下是什麼） | 優化/決策 | 數據/結果 |
|---|---|---|---|---|
| **RDS −40%** | CloudWatch CPU 貼天花板 | 鎖持有時間=等待時間 | 四手段縮短持有時間 | 尖峰 CPU −40% |
| **結算 pipeline** | 每季 10–30 萬筆要不重不漏 | SQS at-least-once 結構性重複 | 業務層冪等鍵 + 重試/補償/對帳 | 0 雙發、0 靜默少拿（對帳兜底） |
| **通知系統** | 10M 訊息/月不能掉 | Pub/Sub at-most-once 零持久化 | DB 持久層 + Pub/Sub 信號層分離 | 訊息不丟（DB 兜底）、即時可丟 |
| **Jitsi 選型** | 商用 SDK 按量計費隨規模長 | mesh N−1 路上行壓垮 client → SFU | 自架 SFU + coturn，零授權費 | 控成本、媒體自有、自運維為代價 |

注意右邊三個系統**不一定有「−40%」這種單一數字**——但它們都有「結果語句」：結算是「0 雙發 + 0 靜默少拿」，通知是「訊息不丟、即時允許丟」，Jitsi 是「控成本 + 媒體自有，代價是自運維」。**結果不一定是百分比，但一定要是一句能收尾的、可辯護的話。** 一個沒有結果層的敘事，聽起來像「我做了一堆事，然後呢？」——「然後」就是結果層。

這就是把 RDS 模板複製出去的真正價值：它逼你對每個系統都補齊「我這樣做，換來了什麼可量測/可辯護的結果」——而不是停在「我用了這些技術」。

## 故障模式與防禦

這一章談的是敘事，所以「故障」是**敘事本身會怎麼壞**——你在面試桌上，這套敘事可能在哪裡崩掉。每一個都綁定你真實要講的那幾個系統。

| # | 故障 | 壞了長什麼樣（可觀測徵兆） | 怎麼防 |
|---|---|---|---|
| **1** | **用過時事實當支柱**（「Twilio 要關了」、Bull 是現役、Redis 是 BSD/7.x、Clinic.js 是推薦工具，**你的系統**） | 面試官剛好知道現況，當場戳破;你不只答錯，還暴露「認知停在過時時間點、無查證習慣」 | 面試前對每個時效性事實（版本、產品狀態、授權、預設值）重新查證;用不會過時的論述（成本/機制/取捨）取代會腐的論述（對手要倒了/某版本最新） |
| **2** | **名詞堆疊、沒有機制**（「我用了 WebSocket + Redis Pub/Sub」就停住） | 追問「為什麼這樣有效」時開始即興發揮、越講越虛，被識破是「會用但不懂」 | 每個名詞都備好下一層機制;講系統時主動往②機制層走，別停在①現象層 |
| **3** | **只講 happy path，被故障問題打穿**（沒主動端出系統怎麼壞） | 問「那它怎麼壞」時才開始想，顯得沒設計過失敗;資深訊號丟失 | 主動把故障模式端上桌（規則二）;每個系統至少備好「最危險的那個故障 + 為什麼它最危險」 |
| **4** | **取捨講成「我選了 X」**（說不出被放棄選項輸在哪） | 追問「為什麼不用 Y」時答不出 Y 輸在哪一軸，決策顯得是「跟風」不是「判斷」 | 每個取捨都備好「Y、Z 各輸在哪個維度」（成本/延遲/運維/故障域）;這是區分資深最快的訊號 |
| **5** | **過度包裝、假裝全懂**（聲稱六年前就全想透了） | 面試官對「全懂從不犯錯」有戒心;一旦某個細節答錯，整段可信度連坐崩盤 | 誠實標「當年懂 vs 最近挖穿」的邊界（誠實邊界那節）;用「我最近回頭挖穿，發現……」當開場 |
| **6** | **數字記錯或誇大**（把 −40% 講成 −60%、3,200 並發講成 32,000） | 數字之間對不起來、或與後續細節矛盾;一個對不上的數字會讓所有數字都被懷疑 | 招牌數字以履歷與本書基準為準（500K、10K CCU、3,200 並發、10M/月、10–30 萬筆、−40%），一字不差，別臨場加碼 |

這張表裡最致命的是 #1（過時事實）——因為它不只讓你答錯，還傳達了一個你最不想傳達的訊號：「這個人的技術認知是凍結的。」對一個要展示「持續跟進現況」的轉職者，這比答不出某個機制細節嚴重得多。**敘事可以有不懂的地方（誠實標出來就好），但不能有「你以為對、其實過時」的地方**——前者是邊界，後者是地雷。

## 紙上推演

### 推演一：把「結算 pipeline」寫成完整面試敘事，並標出三個追問與下探路徑 **[30 分鐘] ★★★**

這是本章的招牌 worked example。請把結算 pipeline 寫成一段完整的面試敘事，依「情境 → 任務 → 機制決策 → 五軸取捨 → 故障防禦 → 結果」六段展開（這是把五層模板攤得更細的版本）。寫完後，**標出三個面試官最可能問的追問，並各寫出你的下探路徑**。

#### 推演解答

**完整面試敘事（口頭交付版，約 2.5 分鐘）：**

> **【情境】** NeoBards 是一款 500K+ 使用者、10K+ CCU 的手遊。每個賽季結束，要把公會排名換算成獎勵發給玩家——每季 10–30 萬筆，跨大約 5,000 個公會。一筆獎勵不是寫一張表就完事，它要動四個服務:guild（公會結算狀態）、player（貨幣點數）、inventory（背包道具）、mail（站內信）。
>
> **【任務】** 我要設計一條 pipeline，在這個量級下做到**不重不漏**——同一筆獎勵不能發兩次（玩家經濟會通膨），也不能漏發（玩家會客訴）。而且四個服務沒有分散式交易能包住。
>
> **【機制決策】** 架構是 K8s CronJob 觸發、拆任務進 SQS、一群冪等 consumer 處理。核心決策是冪等——因為 SQS 標準佇列是 at-least-once，它**架構上保證**會重複投遞。機制在 visibility timeout:consumer 撈訊息後它變不可見一段時間，處理完才刪;但只要 consumer 慢了或當機，SQS 在 timeout 到期時就重投——因為它**無法分辨「consumer 死了」和「只是慢」**，安全選擇永遠是再投一次。所以我用業務身分（季+公會+玩家+獎勵類型）當冪等鍵，靠資料庫唯一約束做原子去重，而不是 check-then-act——因為並發 consumer 下兩個都會通過 if-exists 然後都發獎。
>
> **【五軸取捨】** 交付：標準佇列 at-least-once、可能重複，靠冪等收斂成效果上剛好一次——我評估過換 FIFO，但它的 exactly-once 只是 5 分鐘去重窗口、擋不住跨 5 分鐘的 CronJob 重疊，而且排序保證我不需要、吞吐上限是純損失，所以冪等交給業務層、SQS 用最便宜的標準佇列。並發:多 consumer 同時處理同一筆是常態，正確性下放到資料庫的唯一約束。分區:四服務最終一致，沒有分散式交易，靠重試/補償/對帳收斂。
>
> **【故障防禦】** 部分失敗會留下不一致——寫了 player 沒寫 inventory。三層收斂:重試（暫時性失敗，不刪訊息讓它重投）、補償（永久業務失敗如背包滿，反向回退）、對帳（離線批次掃孤兒記錄）。最危險的不是雙發——雙發吵、會被客訴、容易發現;最危險的是**靜默少拿**:冪等鍵粒度設錯（漏了 season_id）會讓玩家跨季漏發，沒有錯誤日誌、沒人客訴，只有對帳抓得到。所以對帳不是事後補的補丁，是設計的一部分。
>
> **【結果與誠實】** 這條 pipeline 穩定跑了每季 10–30 萬筆。當年我說「保險起見讓 consumer 冪等」——**現在我會重講:冪等不是保險，是這個架構的前提**;而「at-least-once + 冪等」是分散式系統的預設安全帶。這個轉變是我最近回頭把它挖穿才講得清楚的。

**三個最可能的追問 + 下探路徑：**

**追問 A：「為什麼不直接用一個分散式交易把四個服務包起來，不就不用搞冪等和補償了？」**
→ 下探到**取捨層**:「兩階段提交（2PC）在這種跨服務、跨資料庫、要求高吞吐的場景下又慢又脆——它要一個協調者鎖住所有參與者直到全體確認，任何一個服務慢或掛，整個交易卡住，故障域被放大到所有服務。對 30 萬筆的吞吐和四個獨立服務，2PC 的可用性和延遲代價遠超它省下的複雜度。所以業界的標準解是放棄分散式交易、改用最終一致 + 冪等 + 補償（saga），用一點點中間不一致的窗口換可用性和吞吐。」

**追問 B：「冪等鍵的『檢查 + 寫入』你說要原子——具體怎麼做到？if-exists 不行那行嗎？」**
→ 下探到**機制層**:「if-exists 不行，因為它是 check-then-act——在『檢查』和『寫入』之間有一段空隙，並發的兩個 consumer 可能同時通過 `db.exists()`（都看到還沒做過），然後都 `apply_reward`，玩家還是雙發。正確做法是讓『宣告我做了這筆』本身成為原子操作:把 settlement_key 設成資料庫的唯一約束，`INSERT ... ON CONFLICT DO NOTHING`，由資料庫的並發控制（行鎖/索引唯一性）保證無論多少 consumer 同時搶，只有一個 insert 成功，其他撞唯一約束失敗、直接放行。先有約束，再談冪等——只靠應用層判斷在並發下等於沒做。」

**追問 C：「你說最危險的是靜默少拿——那你當年怎麼第一時間知道發生了靜默少拿？」**
→ 下探到**觀測層**:「這正是最難的部分，也是我當年最該補強的。靜默少拿沒有錯誤日誌、沒有客訴（玩家少拿通常不會大規模反映），所以不能靠被動等。要靠主動的對帳——定期掃一遍問『guild 標記為已結算的每一筆，player/inventory/mail 是不是都真的有對應記錄』，找出孤兒記錄。再加上 DLQ 深度告警（深度 > 0 即異常，DLQ 不是垃圾桶是收件匣）、和『結算 Job 該跑沒跑』的獨立告警。這些都是觀測作為設計一部分的體現——不是事後補的。」

**這題的核心**:一段好的面試敘事，不是把所有細節一口氣倒出來（那會超過 3 分鐘、聽眾抓不住重點），而是**主敘事控制在 2–3 分鐘、把細節壓在「追問庫存」裡**，等面試官往下問時再逐層放出。會的人都知道：面試官的追問不是攻擊，是給你展示深度的機會——你要做的是**確保每個追問下面，你都已經站在那一層等他**。

### 推演二：找出一段舊敘事裡「名詞堆疊但沒機制」的地方並補上 **[15 分鐘] ★★**

下面是一段（虛構但典型的）舊式敘事。請找出它「名詞堆疊但沒機制」的三個地方，各補上一層機制或取捨，讓它從「會用」升級成「懂」。

> 「我在 LetsTalk 自建了即時通知系統，用 WebSocket 加 Redis Pub/Sub 做訊息扇出，支援多裝置同步，~3,200 並發。後來還做了視訊，評估了 Twilio 和 Agora，因為成本選了 Jitsi，配了 TURN/STUN 做 NAT 穿透。」

#### 推演解答

三個「名詞堆疊」的點，各補一層：

**點一:「用 Redis Pub/Sub 做訊息扇出」——只有名詞，沒有它的能力邊界。**
→ 補機制:「這裡要講清楚 Pub/Sub 是 fire-and-forget、at-most-once、零持久化——它能做在線即時推送，但**不能做保證送達**:訂閱者斷線那一瞬間的訊息永久消失、不報錯。所以我的架構是持久層 + 即時信號層分離:訊息本體先寫 DB（真相來源），Pub/Sub 只推一個『你有新訊息』的信號。」——不補這層，面試官問「Pub/Sub 掉訊息怎麼辦」你就垮了。

**點二:「支援多裝置同步」——這是個成就宣稱，但沒講它怎麼成立、邊界在哪。**
→ 補機制:「多裝置同步 Pub/Sub 只覆蓋『兩端此刻都在線』那一種情況。平板一斷線，那期間的訊息它永久錯過——所以離線補發必須另走一條路，而且一定不能是 Pub/Sub:重連時帶最後收到的訊息 ID 向 DB 拉取補齊。Pub/Sub 只管在線即時那一段。」

**點三:「因為成本選了 Jitsi」——這是會過時的理由 + 沒講被放棄選項輸在哪。**
→ 補取捨 + 校正:「成本是真的——Twilio/Agora 是商用 SDK 按量計費隨規模線性長，Jitsi 開源自架零授權費;加上自架掌控（隱私 IM 媒體路徑自有）和 Apache 2.0 授權。代價是自運維 JVB + coturn。**特別要注意:別用『Twilio 要關了』當理由——那個 EOL 決定 2024-10 被官方撤回了，Video 現在仍是現役產品。**」

**常見錯路**:以為「補機制」就是「講更多技術名詞」。不是。補機制是**講清楚這個名詞底下實際怎麼運作、它的能力邊界在哪、被放棄的選項輸在哪**——是「為什麼」和「會怎麼壞」，不是「又用了什麼」。一段補對了的敘事，名詞數量可能還變少了，但每個名詞底下都站著一層理解。

### 推演三：為某系統準備三個「往下一層」的追問答案 **[20 分鐘] ★★**

挑通知系統，預判面試官會沿「現象 → 機制 → 取捨 → 故障」往下挖的三個追問，各寫好下探答案（不要寫成段落，寫成「他問 X → 我答下一層 Y」的對照）。

#### 推演解答

**追問 1（機制層）:「WebSocket 既然是長連線，那你多開幾台 server 不就水平擴展了嗎，為什麼還需要 Redis？」**
→ 下探:「因為 WebSocket 連線是 stateful 的——『使用者 B 的那條連線』是活在某一台特定 server 記憶體裡的 socket。當 A 從 server-1 進來一則給 B 的訊息、而 B 連在 server-2，server-1 手上根本沒有 B 的 socket。所以要一個跨機器的 backplane 把訊息送到正確那台——我用 Redis Pub/Sub 做這個。多開幾台不會自動解決『跨台找人』，那是我要自己解的工程問題，RFC 6455 本身完全不規範擴展。」

**追問 2（取捨層）:「你為什麼不用 Redis Streams？它有持久化，不是更安全嗎？」**
→ 下探:「Streams 確實有持久化、at-least-once、能 replay，能解決 Pub/Sub 斷線掉訊息的問題。但代價是更重——要管 consumer group、要 XACK、未確認訊息累積在 PEL、log 持久在記憶體要設 MAXLEN 修剪、而且 at-least-once 又把冪等問題帶回來。關鍵是:即時門鈴這一層**本來就允許丟**，因為 DB 兜底。用 Streams 解一個允許丟的問題，是用重工具增加記憶體和複雜度。Streams 的甜蜜點是『需要持久 + at-least-once + 多消費者協作』的場景，不是在線即時通知。」

**追問 3（故障/觀測層）:「Pub/Sub 掉訊息你說不報錯——那你怎麼知道它在掉訊息？」**
→ 下探:「這正是這套系統最弱的一軸。Pub/Sub 掉訊息是靜默的，`PUBLISH` 回 0 不是錯誤。所以監控不能靠錯誤日誌，要靠間接訊號:監控『重連拉取補發的訊息量』——如果重連時要從 DB 補回的訊息量異常高，就代表即時層在大量掉訊息;還要監控『真實活躍連線數』（用心跳驗證，因為幽靈連線會讓在線數虛高）和 `PUBLISH` 觸達 0 的比例。沒有這些 metric，掉訊息會靜默到只剩使用者體感『對話跳號』。」

**這題的核心**:準備追問答案的方法，是對每個系統把「現象 → 機制 → 取捨 → 故障 → 觀測」這條軸**預先走一遍**——你前面每章的五軸體檢就是這條軸的庫存。面試時你不知道他會問哪一層，但只要你每一層都備好了，無論他從哪切，你都有下一層在等他。**這就是「機制級理解」最直接的轉職紅利:它不是讓你多背幾個名詞，是讓你在任何追問深度上都站得住。**

## 自我檢核

口頭自答;講不清楚＝還沒真懂。

1. 「我做了 X」這句光禿禿的話，要升級成資深級敘事，要往下填哪五層？每一層分別在回答面試官的哪一種問題？
2. 為什麼說「禁得起追問」比「動聽」更重要？一段流暢但用了過時事實的敘事，會在哪裡、怎麼崩掉？
3. RDS −40% 的敘事，如果面試官追問「為什麼重排 SQL 順序會有差」，你往哪一層下探、答什麼？再追問「那拆交易有什麼風險」呢？
4. 結算 pipeline 的敘事，把「保險起見讓 consumer 冪等」重講成「這個架構的前提」——這句話完整要怎麼講？「at-least-once + 冪等」少了任一半會怎樣？
5. 通知系統的敘事，為什麼「主動承認自建只補了在線那一段、離線是另一條路」反而是強敘事？這展示了你的什麼？
6. Jitsi 選型,「Twilio 要關了」這個理由今天為什麼不能用（講出時間線）？取而代之、不會過時的三條理由 + 一條代價各是什麼？
7. 「誠實標出當年懂 vs 最近才挖穿的邊界」為什麼比「假裝全懂」更強？給一個你會用的句式。它的禁區是什麼（誠實不等於什麼）？
8. 把 RDS 那個「量測 → 診斷 → 機制 → 優化 → 數據」模板複製到一個沒有現成漂亮數字的系統（如通知系統）——它的「結果層」該講成哪一句可辯護的話？
