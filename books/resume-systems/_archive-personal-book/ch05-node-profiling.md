# ch05 — 負載測試與 Node 剖析：event loop、洩漏與 CPU 瓶頸

> **本章解決什麼問題**：你履歷上寫「用 Clinic.js 診斷記憶體洩漏與 CPU 瓶頸、寫負載測試情境」。這章把那句話往下挖到一個比工具更根本的東西——Node 的執行模型。為什麼在一台 8 核機器上，**一個慢函式**就能讓所有並發請求一起變慢？為什麼你的壓測「全綠」卻在 production 尖峰爆掉？這是 Part II「NeoBards 高併發戰場」的第三章，接在 RDS 鎖競爭（ch03）與結算 pipeline 交付語意（ch04）之後。event loop 相位的逐格細節與 GC 深論留給 ch12，排隊的數學留給《等待的數學》（queue）——這章只負責讓你看見「一台 Node 機器是怎麼被一行程式碼拖死的」。

## 從你已知的出發

當年在 NeoBards，這是個 client-managed 的基礎設施——你的程式碼跑在發行商 Nexon 那側的環境，500K+ 使用者、10K+ CCU。每一次大改動上線前，你要寫負載測試情境去模擬尖峰流量，看系統撐不撐得住；上線後出問題，你要用 Clinic.js 去診斷到底是記憶體在漏，還是 CPU 卡住了。

你大概記得那個畫面：跑一輪 `clinic doctor`，它給你一張時間軸圖，告訴你「這看起來像 event loop 阻塞」或「這看起來像記憶體問題」；然後你跑 `clinic flame`，得到一張火焰圖，自上而下找那條最寬的橫條——那就是吃掉 CPU 的兇手。你找到一個同步的重運算，把它改成非同步或丟到別處，尾延遲掉下來，問題解決。

這套流程你做得很順。但這章要問的，不是「Clinic.js 怎麼用」——那你早就會了。要問的是三件你當年**靠直覺繞過、但沒真正攤開**的事：

第一，**為什麼一個慢函式能拖垮整台機器**？在一個多執行緒的 Java 服務裡，一個慢請求只拖慢它自己那條執行緒；其他請求各走各的執行緒，互不相干。可是在 Node，你大概隱約知道「不要在主執行緒做重活」，但你說得清楚那個「主執行緒」到底卡的是什麼、為什麼會連坐所有並發請求嗎？

第二，**你的壓測情境，量到的是真實尖峰嗎**？你寫負載測試，模擬「每秒 N 個請求」，看 p95、p99。但有一個藏在壓測工具裡的陷阱：很多工具會在系統變慢的時候，**自己也跟著慢下來**——它等上一個請求回來才發下一個。結果系統越爛，它打的壓力越小，量出來的尾延遲反而越漂亮。這個陷阱有個名字，等下會講。

第三，這是個老兵記憶的活教材：**Clinic.js 現在已經不是你記得的那個現役工具了**。它官方 README 直接寫著自己「未在積極維護」（2026-06）。你 2022–2024 年依賴的那套東西，現在用之前得先知道它的狀態變了。

把這三件事攤開，你會發現它們其實是同一個底層機制的三張臉：Node 是**單執行緒**的，event loop 是那條唯一的執行緒，而你做過的所有 profiling 與壓測，本質上都是在問「這條唯一的執行緒，現在被誰占住了、占了多久」。

## Node 是單執行緒的——這句話的精確意思

先把「單執行緒」這句口號講準，因為它常被講錯。

說 Node「單執行緒」，**不是說整個行程只有一條 OS 執行緒**。Node 底下的 libuv 維護一個 thread pool（執行緒池），預設 4 條（`UV_THREADPOOL_SIZE`，2026-06 仍是 4），用來跑 `fs` 檔案操作、`dns.lookup`、async crypto（pbkdf2/scrypt 之類）、async zlib——這些東西在背景執行緒上跑，不卡你。網路 I/O 又是另一回事：它根本不走 thread pool，而是交給 OS 的非同步原語（Linux 的 epoll、macOS 的 kqueue、Windows 的 IOCP），在 event loop 自己那條執行緒上以非阻塞方式完成。

所以更精確的說法是：**你寫的 JavaScript，全部跑在唯一一條執行緒上——event loop 那條**。callback、Promise 的 `.then`、`async` 函式 `await` 之後的那段、Express 的 route handler、你解析 JSON 的那一行——這些**全都排隊在同一條執行緒上輪流執行**。libuv 的 thread pool 與 OS 的 epoll 只負責「等待 I/O 完成」這件不耗 CPU 的事；一旦 I/O 完成、要跑你的 callback 來處理結果，那個 callback **一定回到 event loop 這條執行緒上跑**。

這就是整章的地基。畫成圖：

```
            ┌──────────────────────────────────────────┐
            │      event loop（唯一執行你 JS 的執行緒）  │
            │  ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐        │
   請求 →   │  │ cb │ │ cb │ │ cb │ │ cb │ │ cb │ ...    │  → 回應
            │  └────┘ └────┘ └────┘ └────┘ └────┘        │
            │   一次只跑一個，跑完才換下一個，不搶不插隊   │
            └───────┬──────────────────────────┬─────────┘
                    │ 把「等 I/O」丟出去         │ I/O 好了就回來排隊
                    ▼                          ▲
            ┌───────────────┐         ┌─────────────────┐
            │ libuv 執行緒池 │         │ OS epoll/kqueue │
            │ （預設 4 條）  │         │  網路 I/O        │
            │ fs/dns/crypto │         │  不走執行緒池     │
            └───────────────┘         └─────────────────┘
```

關鍵在那條註解：**「一次只跑一個，跑完才換下一個，不搶不插隊」**。OS 執行緒之間是搶占式的——核心可以隨時暫停一條執行緒、切去跑另一條，所以一條卡住不影響別條。event loop 上的 callback **不是搶占式的**：一個 callback 開始跑，event loop 就**等它自己跑完、自己交還控制權**，才去跑下一個。沒有任何機制能在中途把它踢開。

這個設計換來的好處你很清楚：沒有共享狀態的鎖、沒有 race condition（在單一事件迴圈內）、用幾 KB 的記憶體就能掛住一條閒置連線（不像 thread-per-connection 每條連線吃一條執行緒的堆疊）——這就是 Node 能用一台機器扛住你 3,200 並發 WebSocket（見 ch07）的原因。但它的代價，正是這章的主題：**沒有搶占，就代表任何一個 callback 跑太久，後面所有人都得乾等。** event loop 並發的逐相位機制（timers → poll → check…）留給 ch12 挖穿；這章只需要這一條鐵律。

## event loop 被卡住：CPU 瓶頸的真身

現在來看「一個慢函式拖垮整台」具體怎麼發生。

假設你有個 Express 服務，正常每個請求做一點輕量的事（查 Redis、組個 JSON、回傳），單一請求 2ms 就回。1,000 RPS 在這台機器上輕鬆——event loop 一圈圈轉，每個 callback 都很短，排隊幾乎感覺不到。

現在有人在某個 hot path（每個請求都會走到的路徑）放進一行**同步的重運算**。最經典的就是 `JSON.parse` 一個大物件。Node 官方文件給過一個具體數字：把一個 2²¹ 大小的物件 `JSON.stringify` 成 50MB 字串、再 `JSON.parse` 回來，stringify 要 0.7 秒、parse 要 1.3 秒（2026-06，官方 *Don't Block the Event Loop* 文件）。`JSON.parse`／`JSON.stringify` 的成本是 O(n)，n 是輸入長度——輸入夠大，它就慢得驚人。

關鍵不是「這一行慢」，而是「這一行慢的時候，**event loop 上其他所有事都停了**」。它是同步的，沒有 `await` 把控制權交回去的時機；event loop 不搶占，所以它跑多久，event loop 就被釘在這裡多久。在這 200ms（假設一個中等大小的 payload）裡：

- 其他 999 個請求的 callback：**全部排在後面等，動不了**。
- 已經完成的 Redis 查詢、已經到達的網路封包：**callback 排在後面等，動不了**。
- 連你的健康檢查（health check）端點：**也在後面等**——於是負載均衡器以為這台機器死了，可能把它踢出去（誤判，見 ch13 的「慢 vs 死分不清」）。

這就是「CPU 瓶頸的真身」：它表面上是「CPU 高」，但你真正觀測到的災難不是 CPU 數字，而是**所有請求的尾延遲一起爆炸**——它們全被那一行同步運算堵在 event loop 的隊伍後面。Java 那種多執行緒模型裡，同樣一個 200ms 慢函式只拖慢**它自己那條執行緒**；在 Node，它讓**全部人**一起慢。這是兩種並發模型最根本的脾氣差異。

> **我以為 vs 實際。** 你當年大概記成「CPU 密集的工作會讓 Node 變慢」。精確版是：**讓 Node 變慢的不是「CPU 密集」本身，而是「同步且不讓出控制權的 CPU 密集」**。一個用 `worker_threads` 跑的 CPU 密集任務一點都不卡主執行緒；一個 1ms 的同步迴圈跑一萬次（總共幾秒）才會卡。判準不是「重不重」，是「**它有沒有在合理時間內把 event loop 還回來**」。這個判準，ch12 會用相位機制講到底。

### Worked example：200ms 同步處理放進 hot path，對 1,000 RPS 尾延遲的連鎖

來把這條連鎖一步步追到底，用真實數字。

**前提**：一台機器，目標 1,000 RPS（每秒 1,000 個請求，平均每 1ms 來一個）。正常處理每個請求 2ms。有人在 hot path 放進一個同步的 JSON 處理，平均耗時 **200ms**，每個請求都會走到。

**第一步：算這台機器還撐不撐得住這個吞吐。**
event loop 是單執行緒，所以它每秒能跑的「總工作時間」上限就是 1 秒。每個請求現在要占 event loop 200ms（純同步、不可中斷）。一秒鐘最多能跑完幾個？

```
1,000 ms ÷ 200 ms/req ≈ 5 req/s
```

**5。** 不是 1,000。你的目標是 1,000 RPS，但這條 event loop 一秒只能消化 5 個請求。剩下的 995 個全部堆進隊伍。這台機器的有效吞吐被那一行同步運算從 1,000 RPS 砍到 **5 RPS**——崩了 200 倍。

**第二步：算排隊延遲怎麼長。**
請求以 1,000/s 進來、5/s 出去。每一秒淨增 995 個積壓在隊伍裡。一個在第 1 秒末到達的請求，前面排了約 1,000 個請求，每個要 200ms：

```
等待時間 ≈ 1,000 個 × 200 ms = 200,000 ms = 200 秒
```

而且這個數字**每一秒都在變大**，因為到達率（1,000/s）遠超處理率（5/s）。這就是 ch14 要講的「無界佇列在到達率超過處理率時延遲無限增長」的具體長相——這裡的「佇列」不是什麼顯式的訊息佇列，**它就是 event loop 後面那條看不見的 callback 隊伍**。（為什麼 ρ→1 時延遲會這樣炸開的數學，見《等待的數學》queue 書；這裡只追工程現象。）

**第三步：對照觀測到的徵兆。**
這時你看監控會看到什麼？

- CPU：一條核心 100%（其他核心閒著——單執行緒只用得到一核）。這是你當年 `clinic doctor` 會標成「event loop blocked」的典型。
- 尾延遲：p50 還可能不難看（剛好排在前面的請求快），但 **p99 / p999 直接飛到天上**——這就是為什麼觀測要看 p99 不看平均（見 ch15）。
- event loop lag（事件迴圈延遲）：這是最直接的指標。正常 < 10ms，這時候會飆到數百 ms 甚至秒級。
- 健康檢查逾時 → 機器被踢出輪替 → 流量湧向其他機器 → 其他機器也卡 → **雪崩**。

**這就是「一個慢函式拖垮整台」的完整帳。** 注意：兇手只占了一行程式碼、200ms，但因為它在 event loop 上、因為 event loop 單執行緒不搶占、因為它在 hot path 每個請求都走——這三個條件疊起來，1,000 RPS 變 5 RPS。少了任何一個條件（丟去 worker、放冷路徑、改非同步分片），災難都不成立。

## Node 的記憶體洩漏：四個典型來源

CPU 卡住會立刻爆；記憶體洩漏是慢性病——它不會馬上殺死你，而是讓行程的記憶體用量**只增不減**，跑幾小時或幾天後撞到上限，被 OOM killer 砍掉、重啟，然後再來一輪。在你 3,200 並發、10M 訊息/月的 LetsTalk，或長時間掛著的 NeoBards 服務裡，這種「跑著跑著就吃光記憶體」的病最折磨人，因為重現它要時間。

JavaScript 有垃圾回收（GC），照理說你不用管記憶體。但 GC 的規則是：**只要還有任何一條引用鏈能從根（root）走到某個物件，這個物件就不會被回收。** 記憶體洩漏在 Node 幾乎都是同一個故事——**你以為某個東西該被丟掉了，但有一條你忘了的引用鏈還抓著它**。四個最典型的來源：

**① 閉包持有（closure capture）。** 一個閉包會抓住它定義時所在作用域的全部變數。如果這個閉包被長期保存（例如註冊成某個長壽物件的 callback），它抓住的那一整個作用域——包括那些你以為用完就沒了的大物件——就跟著它一起活下來，GC 動不了。常見變形：用 Promise 包了某個操作，但 resolve/reject 一直沒被呼叫，那條閉包鏈連同它捕獲的東西就卡在記憶體裡（2026-06，多篇實務分析指認此型）。

**② event listener 沒移除（這型最常見，下面 worked example 專門算它）。** 每次 `emitter.on('event', handler)` 都往 EventEmitter 的內部陣列**加一筆** handler；如果你只 `on` 不 `off`（或不用 `once`），這個陣列只增不減。每個 handler 又是個閉包，抓著一堆東西。Node 在單一 emitter 的 listener 超過預設 10 個時會印 `MaxListenersExceededWarning`——但**注意這只是一行 console 警告，它不會 throw、不會阻止你繼續加、也不會幫你清掉任何東西**（2026-06）。它是個提醒，不是防護。很多人把這警告靜音或把上限調高，等於把唯一的警報器關掉。

**③ 快取無上限（unbounded cache）。** 你寫個 memoization：`cache[key] = result`，每個沒見過的 key 就存一筆。看起來很合理——直到你發現 key 的空間是無限的（每個 userId、每個 requestId、每個訊息 ID 都是新 key）。這個 `cache` 物件只增不減，本質上就是個「永遠不過期、永遠不淘汰」的記憶體黑洞。快取**必須有淘汰策略**（LRU、TTL、大小上限）——一個沒有上限的快取不是快取，是洩漏（Redis 的 eviction 與這裡同一個道理，見 ch17）。

**④ 全域 Map / 陣列只增不減。** ③ 的近親。一個 module-level 的 `const sessions = new Map()`、`const pending = []`，你在連線建立／請求進來時往裡 `set`／`push`，卻在連線斷開／請求結束時忘了 `delete`／清掉。它掛在 module 作用域（一種 root）上，永遠不會被 GC。`Map` 尤其陰險，因為它的 key 是強引用——即使 key 物件在別處已經沒人用了，只要還在這個 `Map` 裡，它和它的 value 就都活著。（`WeakMap` 才用弱引用，但它不能列舉、用途受限。）

這四個有個共同的形狀：**「進」的時候你寫了，「出」的時候你忘了。** 加 listener／移 listener、放快取／淘汰快取、連線建立／斷開——洩漏永遠藏在那個你沒寫的「出」。

### Worked example：event listener 沒移除，在 3,200 並發下的增長帳

用你 LetsTalk 的真實規模來算這筆帳。

**情境**：即時通知系統，~3,200 並發 WebSocket 連線（見 ch07）。每條連線建立時，你在某個共用的事件來源（比方一個全域的 `notificationBus`，或 Redis client 的事件、或一個廣播 emitter）上註冊一個 listener，用來把訊息推給這條連線：

```
// 示意（追邏輯用，非可執行）
function onConnection(socket) {
  const handler = (msg) => socket.send(msg);   // 閉包抓住 socket
  notificationBus.on('broadcast', handler);     // 「進」：寫了
  // socket 斷線時呢？……忘了 notificationBus.off('broadcast', handler)
}
```

**bug**：連線斷開時沒有對應的 `notificationBus.off(...)`。

**算增長**。手機/平板上的 WebSocket 連線壽命不長——使用者切背景、進電梯、Wi-Fi 跳 4G、App 被系統回收，斷線重連是常態。假設一條連線平均**每 5 分鐘**斷一次重連一次（行動網路下這算保守）。

每次「重連」= 一次新的 `onConnection` = `notificationBus` 上**多一個 handler**；每次「斷線」**不移除**任何 handler。所以 `notificationBus` 的 listener 數量是純增的：

```
每分鐘新增 listener ≈ 3,200 連線 ÷ 5 分鐘 ≈ 640 個/分鐘
一小時      ≈ 640 × 60      = 38,400 個
八小時      ≈ 38,400 × 8     = 307,200 個
```

跑一個工作天，`notificationBus` 上掛了 **約 30 萬個 handler**，而真正活著的連線從頭到尾只有 3,200 個。多出來的 ~30 萬個全是**死連線留下的殭屍 listener**，每個都是一個閉包，抓著一個早就該被回收的 `socket` 物件（連同它的緩衝區、使用者上下文）。

**兩個疊加的災難**：

1. **記憶體**：30 萬個閉包 × 每個閉包抓住的 socket 相關物件（保守估每個幾 KB）= 數百 MB 到 GB 級的洩漏，**只增不減**，撐到記憶體上限被 OOM 砍掉重啟。
2. **CPU（被洩漏間接拖累）**：每次 `notificationBus.emit('broadcast', msg)`，EventEmitter 要**同步地、依序地**呼叫陣列裡**每一個** listener。本來該呼叫 3,200 個，現在要呼叫 30 萬個——其中 ~29.7 萬個是對著死 socket `send`（要嘛 throw、要嘛無聲失敗）。一次廣播從「跑 3,200 個 callback」變成「跑 30 萬個 callback」，而這**全部發生在 event loop 那條唯一執行緒上**。於是這個記憶體洩漏，最後以「event loop 被一次廣播卡住數百毫秒」的形式，變成上一節的 CPU 瓶頸。

**這就是 worked example 的重點**：在 Node，記憶體洩漏和 CPU 瓶頸不是兩個獨立的病——一個只增不減的 listener 陣列，同時是記憶體洩漏（閉包抓著死物件）**和** CPU 瓶頸（emit 時要遍歷整個陣列）。修法很小：連線斷開的 handler 補一行 `notificationBus.off('broadcast', handler)`，或用 `socket.once` / `AbortController` 隨連線生命週期自動清掉。**「進」寫了「出」也寫，洩漏就不成立。**

## 怎麼指認它：profiling 工具，與一個必須講清楚的老兵記憶校正

現在講你履歷上那句「用 Clinic.js 診斷」。先把最重要的事實放最前面，因為它直接影響你今天還能不能照當年那樣做。

> **⚠️ 老兵記憶校正（2026-06）：Clinic.js 已停止積極維護。** Clinic.js 由 NearForm 開發，但它官方 README 現在直接寫著：「Clinic.js is not being actively maintained. Due to its strong ties to Node.js internals, it may not work or the results you get may not be accurate.」最後一個正式 release 是 **v13.0.0（2023-06-28）**，距今約三年；repo 沒被正式封存（仍 public），但實質上是休眠/僅維護狀態。因為它跟 Node 內部綁很緊，**在新版 Node（active LTS 已是 Node 24，見下）上可能跑不動、或結果可能不準**。你 2022–2024 年依賴它沒問題，但今天要拿它當主力診斷工具之前，必須先知道它的狀態變了。這正是本書反覆強調的「老兵記憶會過時」最乾淨的例子。

把這件事講清楚後，Clinic.js 那套**心智模型**仍然完全有效——它的四個工具對應四類問題，這個對應關係是 Node 診斷的通用框架，換成別的工具也是這四類：

| Clinic.js 工具 | 看什麼 | 對應本章哪個病 |
|---|---|---|
| **Doctor** | 總覽：自動判斷「這像 event loop 阻塞 / I/O 問題 / GC 壓力 / 記憶體問題」 | 先分診，告訴你往哪個方向查 |
| **Flame**（火焰圖） | CPU：哪個函式吃掉最多 CPU 時間 | event loop 被卡住的 CPU 瓶頸 |
| **Bubbleprof** | 非同步流：時間花在哪些 async 操作之間 | 找「卡在等什麼」而非「卡在算什麼」 |
| **Heap Profiler** | 記憶體：什麼東西在堆上越積越多 | 四種記憶體洩漏 |

四個工具仍是 **Doctor / Flame / Bubbleprof / Heap Profiler**（這點沒變）。但今天該用什麼**現役**的工具來做同樣的事？

**現役替代（2026-06）**：

- **`node --cpu-prof`**：Node 內建（Node 12+），直接吐出 `.cpuprofile` 檔，丟進 Chrome DevTools 看。**零相依、官方維護、跟著 Node 一起更新**——這是最穩的選擇，因為它就是 Node 本身的一部分，不存在「跟 Node 版本不相容」的問題。
- **`node --prof`** + `--prof-process`：更底層的 V8 取樣分析器，輸出 tick log，自己處理成可讀報告。
- **Chrome DevTools Inspector**（`node --inspect`）：直接連上跑著的行程，做 CPU profile、抓 heap snapshot（記憶體洩漏的標準抓法——隔一段時間抓兩張快照，比對哪些物件數量只增不減，就是上面那四種洩漏的指紋）。
- **`0x`**（davidmarkclements/0x）：單指令產火焰圖的老牌工具，Clinic.js Flame 當年就是建在它上面。仍可用，但同樣留意它與最新 Node 的相容性。
- **Clinic.js 本身**：仍可裝、仍可用，但**標上「未積極維護、可能與新 Node 不相容」的風險標籤**再用，別當成第一推薦。

**順帶校正另一個版本記憶**：你用這些工具的環境也變了——2026-06 的 Node **active LTS 是 Node 24（codename "Krypton"，自 2025-10-28）**；Node 22（"Jod"）已降為 maintenance LTS。如果你腦中還停在「Node 18/20 是 LTS」，那也是該更新的記憶。（libuv 相位與 LTS 節奏的細節見 ch12。）

### flame graph 怎麼讀：自上而下找寬條

火焰圖是你當年找 CPU 兇手的主力，但它常被讀錯。把規則講準：

```
  level 0  ┌────────────────────────────────────┐
           │ root                               │
  level 1  ├──────────────────────┬─────────────┤
           │ handleRequest        │ other       │
  level 2  ├──────────┬───────────┤
           │ queryDB  │ parseBody │   ← parseBody 寬、底下無子＝self time 大
           └──────────┴───────────┘
            ↑ 在等 I/O，窄、不耗 CPU
```

讀法三條規則：

1. **寬 = 耗 CPU。** 每條橫條的**寬度**代表它（連同它呼叫的子函式）占了多少比例的取樣時間。越寬，吃的 CPU 越多。**先看寬度，不看高度**——高（堆疊深）不代表慢，只代表呼叫鏈長。
2. **找「寬、且底下沒有子條（或子條很窄）」的那一格。** 這代表時間花在**這個函式自己身上**（self time / 自身時間），不是花在它呼叫的別人身上。一個函式 total time 很大但 self time 很小，表示工作在它的子函式裡（往下找）；一個函式 self time 很大，**它本身就是兇手**——那行同步 `JSON.parse` 就會以「又寬、底下又沒東西」的形狀現身。
3. **顏色通常無意義**（多數實作只是為了區分相鄰格），別過度解讀。

當年你「自上而下找最寬的橫條」這個直覺是對的，但精確版是：**找寬、且 self time 大的那一格**——那才是真正在 event loop 上同步燒 CPU、把所有人堵在後面的兇手。

## 負載測試：為什麼你的「全綠」可能是假的

最後一塊，回到你履歷上的「寫負載測試情境」。你寫壓測是為了在上線前知道系統撐不撐得住尖峰。但壓測工具本身藏著一個會讓你「全綠卻在 production 爆掉」的陷阱。先把工具對齊（這裡有兩個版本/授權記憶要校正），再講陷阱。

### 工具：k6 與 Vegeta（2026-06）

- **k6**：正式名稱是 **Grafana k6**——Grafana Labs 於 2021 收購了它。Go 引擎、測試腳本用 JavaScript（現也支援 TypeScript）寫。**授權是 AGPLv3**（Grafana 系產品 2021 起從 Apache 2.0 改成 AGPLv3）。積極維護中。**記憶校正**：如果你記成「k6 是 Apache/MIT 授權」或只叫它「k6」，更新成「Grafana k6、AGPLv3」——AGPLv3 對「你把它包進自己對外提供的服務」有傳染性條款，授權這件事在公司情境下會被法務問到。
- **Vegeta**（tsenart/vegeta）：Go 寫的 HTTP 壓測工具，採**固定速率（constant request rate）**模型，可當 CLI 也可當 Go library，仍維護中（2026-06）。

兩者最關鍵的差別不是語言或授權，是**負載模型**——而這正是陷阱所在。

### 開放 vs 封閉負載模型，與「協調遺漏」

把兩種模型講清楚（數學深度——到達過程、為什麼 ρ→1 爆炸——指向《等待的數學》queue 書；這裡只講工程直覺）：

- **封閉模型（closed model）**：固定數量的「虛擬使用者」（VU），每個 VU 是**做完上一個請求、收到回應，才發下一個**。請求的發出**取決於系統的回應速度**。這像「N 個人排隊，前一個辦完才換下一個」。
- **開放模型（open model）**：請求**按一個固定的到達率獨立地產生**，不管前一個回來了沒。這像「不管櫃台多慢，客人照樣每秒進來 100 個」。**真實世界的流量是開放模型**——你的使用者不會因為伺服器變慢就有默契地少發請求。

陷阱叫**協調遺漏（coordinated omission）**，它幾乎只在 p99 以上現形，已經害過一些「壓測全綠」的團隊出 production 事故（2026-06）。機制是這樣：

> 在封閉模型下，當系統開始變慢、回應變久，壓測工具因為「要等上一個回來才發下一個」，**自己的發送速率也跟著降下來了**。系統越慢，它打的壓力越小。於是那些「本來該在系統最慢的時候發出、然後測到超長延遲」的請求，**根本沒被發出來**——它們被「協調著遺漏」了。結果：系統在尖峰其實爛得要命，但因為慢的時段請求變少、快的時段請求變多，**平均與尾延遲被算得異常漂亮**。你的壓測報告一片綠，production 一上真實尖峰就爆。

這就是「**固定速率/封閉式壓測會低估真實尖峰**」的根因：它讓壓力源跟著受測系統一起慢下來，等於偷偷把最該被量到的那段壞掉的尾巴砍掉了。

**所以選型判準**：要逼近真實尖峰、量準尾延遲，用**開放模型 + 固定到達率**——Vegeta 天生固定速率（刻意避開協調遺漏）；k6 要用 **constant-arrival-rate executor**（開放模型）而非預設的封閉式 VU 迴圈。**但 k6 的開放模型有但書**：它靠 `preAllocatedVUs` / `maxVUs` 撐開，VU 池在尖峰被榨乾時，k6 會回報 `dropped_iterations` 並**默默丟掉排隊的請求**——這等於部分地把協調遺漏又請回來。所以**「我用了開放模型」不等於「我量準了」**——還得確認 VU 池夠大、沒在偷丟請求。

> **我以為 vs 實際（壓測）。** 你當年大概覺得「壓測綠了 = 系統撐得住」。精確版是：**壓測綠了，可能只代表你的壓測工具在系統變慢時陪著它一起變慢**。一個能量到真實尖峰的壓測，必須用開放模型、固定到達率、且確認壓力源**沒有**因為系統變慢而縮手——並且永遠看 p99/p999，不看平均（平均是協調遺漏最愛藏屍的地方）。

## 故障模式與防禦

把這章的病攤成「會怎麼壞 / 壞了長什麼樣 / 怎麼防」。每個都至少能對回你做過的系統。

| 故障 | 會怎麼壞 | 壞了長什麼樣（可觀測徵兆） | 怎麼防 |
|---|---|---|---|
| **同步重運算卡 event loop** | 一行同步 CPU 密集（大 `JSON.parse`、同步 crypto、巨大迴圈）釘住唯一執行緒，所有並發請求一起排隊 | 單核 100%（其他核閒）、**event loop lag 飆到數百 ms**、p99/p999 爆炸但 p50 還行、健康檢查逾時被踢出輪替 | 量 event loop lag 當第一線指標；重活分片讓出（把大迴圈切段、相位間 yield）、丟 `worker_threads`、或改串流式解析；hot path 嚴禁同步重運算 |
| **event listener 洩漏**（綁你 LetsTalk 通知系統） | 連線重連時只 `on` 不 `off`，listener 陣列只增不減；既是記憶體洩漏（閉包抓死 socket）也是 CPU 瓶頸（emit 要遍歷整個陣列） | `MaxListenersExceededWarning`（但它只警告不阻止）、RSS 記憶體**只增不減**、跑數小時後 OOM 重啟、一次廣播延遲隨時間越來越長 | 連線生命週期配對 `on`/`off`，用 `once` / `AbortController` 自動清；別把警告靜音當解法 |
| **無上限快取 / 全域 Map 只增**（綁 NeoBards 長壽服務） | `cache[key]=` 或 `globalMap.set()` 沒有淘汰，key 空間無限 → 記憶體黑洞 | RSS 緩慢爬升、GC 越來越頻繁也回收不掉（old space 持續成長）、heap snapshot 比對顯示某類物件數量單調增 | 快取一律給 TTL / LRU / 大小上限；全域 Map 的每個 `set` 都要有對應的 `delete`；考慮 `WeakMap` |
| **壓測協調遺漏 → 低估尖峰** | 封閉式/固定速率壓測在系統變慢時自己也慢，砍掉最壞的尾巴，報告全綠 | 壓測 p99 漂亮但 production 一上真實尖峰就延遲爆炸／雪崩；壓測與線上尾延遲對不上 | 用開放模型 + 固定到達率（Vegeta 天生、k6 用 constant-arrival-rate）；確認 VU 池夠大、`dropped_iterations`=0；永遠看 p99/p999 |
| **用了停更工具卻不知道**（綁你的 Clinic.js 記憶） | Clinic.js 未積極維護（2026-06），在新版 Node 上可能跑不動或結果不準，你卻以為診斷可信 | 工具報錯、輸出空白/異常、或更糟——給出**看似合理但錯誤**的火焰圖，把你導向錯的兇手 | 改用 `node --cpu-prof` / `--prof` / Chrome DevTools inspector（跟著 Node 走、不會版本不相容）；要用 Clinic.js/0x 先確認與當前 Node 版本相容 |

通用的「複雜系統為什麼會崩」分類（雪崩、metastable failure 的跨領域通論）見《正常意外》（fail）；過載的數學見 queue 書。這裡的故障**綁定的是你這台 Node 機器具體會怎麼倒**。

## 紙上推演

**[15 分鐘] ★★ 找出會卡 event loop 的那一行，並說清楚為什麼。**
下面這段示意 code 在每個請求都會跑（hot path）。指出哪一行會卡住 event loop，並說明它卡住的時候，其他並發請求發生什麼事。

```
app.get('/report', async (req, res) => {
  const rows = await db.query('SELECT * FROM events WHERE ...'); // (A)
  const big = JSON.parse(req.body.payload);                       // (B) payload 可能很大
  const result = rows.filter(r => expensiveSyncCheck(r, big));    // (C) expensiveSyncCheck 純同步
  res.json(result);
});
```

### 推演解答

**會卡的是 (C)，潛在加上 (B)；(A) 不卡。**

- **(A) `await db.query`**：這是非同步 I/O。`await` 在這裡的意思是「把控制權**交還**給 event loop，等資料庫回來了再回到這行繼續」。在等待的這段時間，event loop **空出來去服務其他請求**——所以 (A) 完全不卡別人。這正是 Node 並發的精髓：等 I/O 的時間不浪費。
- **(B) `JSON.parse`**：這是**同步**的、O(n)。payload 小（幾 KB）時微不足道；payload 大（MB 級）時，它會同步占住 event loop 數十到數百 ms——這段時間沒有任何 `await` 讓它交還控制權，所以**所有其他請求乾等**。它「可能卡」，取決於輸入大小。這也是「相信所有輸入都可能異常大」的理由：一個沒有大小上限的 `req.body` 就是一顆定時炸彈。
- **(C) `rows.filter(... expensiveSyncCheck ...)`**：這是**最確定會卡**的一行。`.filter` 是同步的，它會把 `rows` 裡每一列都跑一次 `expensiveSyncCheck`，整個過程**沒有任何 `await`**。如果 `rows` 有上萬列、每次 check 又不便宜，這一行會把 event loop 釘住很久。`async` 函式這個外殼**完全幫不上忙**——`async` 只在遇到 `await` 時才讓出控制權，(C) 裡沒有 `await`，所以它從頭到尾獨占 event loop。

**常見錯路**：以為「函式是 `async` 的，所以裡面的東西都不阻塞」。錯。`async`／`await` 只在**真的遇到 `await` 一個非同步操作**時才讓出控制權；一段 `async` 函式裡的純同步迴圈（如 (C)）阻塞起來和普通同步 code 一模一樣。判準永遠是「**這段在跑的時候，有沒有 `await` 把 event loop 還回去**」，不是「外面包了 `async` 嗎」。

**修法方向**（不寫可執行碼）：(B) 加 payload 大小上限 + 考慮串流解析；(C) 若 `rows` 大，把 filter 分片（每處理一批 `await` 一個 `setImmediate` 讓出一輪）、或丟 `worker_threads`、或把過濾下推到 DB 的 `WHERE`（最好——根本別把上萬列拉進 Node）。

---

**[15 分鐘] ★★ 三個現象，哪個是洩漏、哪個是 GC 壓力、哪個是 event loop 阻塞？**
你看到三台機器各有一個徵兆，分別判斷它最可能是「記憶體洩漏 / GC 壓力 / event loop 阻塞」，並說你會用哪個工具確認。

- **甲**：RSS 記憶體在 6 小時內從 200MB 平穩爬到 1.8GB，從不下降；最後 OOM 重啟，重啟後又開始爬。
- **乙**：單核 CPU 貼著 100%，event loop lag 飆到 400ms，p99 延遲炸開，但記憶體用量穩定。
- **丙**：CPU 鋸齒狀（一陣高一陣低）、記憶體也鋸齒狀（漲到某線就被拉回），整體延遲偶爾出現規律的小尖刺，但不會 OOM。

### 推演解答

- **甲＝記憶體洩漏。** 招牌指紋是「**只增不減、重啟後重來**」。記憶體單調爬升、從不回落，代表有東西被一條引用鏈抓著、GC 回收不掉——四種來源（閉包/listener/無上限快取/全域 Map）之一。**確認**：用 Chrome DevTools inspector（`node --inspect`）隔一段時間抓兩張 heap snapshot，比對哪一類物件的數量單調增——那類物件就是洩漏的指紋（例如 30 萬個殭屍 listener 的閉包）。
- **乙＝event loop 阻塞。** 招牌指紋是「**單核 100%（其他核閒）+ event loop lag 高 + 記憶體穩定**」。記憶體穩定排除洩漏；單核滿載＋lag 高直指「有同步重運算釘住唯一執行緒」。**確認**：抓 CPU profile（`node --cpu-prof`）看火焰圖，找寬且 self time 大的那一格——那行同步運算會自己跳出來。
- **丙＝GC 壓力。** 招牌指紋是「**CPU 與記憶體都鋸齒狀、延遲規律小尖刺、但不 OOM**」。記憶體漲到某線被「拉回」= GC 真的有在回收（所以不是洩漏，洩漏拉不回來）；但拉回的動作（尤其 major GC）本身要花 CPU、且**會短暫暫停 event loop**（stop-the-world），於是 CPU 鋸齒、延遲出現規律小尖刺。病根通常是「製造太多短命物件」（每個請求 new 一堆大物件馬上丟），讓 GC 疲於奔命。**確認**：`clinic doctor`（或看 `--trace-gc` 輸出）會把它標成 GC 相關；GC 機制的深論留給 ch12。

**常見錯路**：把丙（GC 壓力）誤判成洩漏。差別在「記憶體**會不會被拉回**」——洩漏是單調爬升永不回頭，GC 壓力是鋸齒狀（漲了又被回收）。一個救法（給更多記憶體）只能拖延洩漏、卻可能加重 GC 壓力（heap 越大，major GC 暫停越久），方向相反，所以先分清病因再動手。

---

**[20 分鐘] ★★★ 設計一個能重現某尾延遲的壓測情境（敘述，不寫可執行碼）。**
production 上你觀察到：平時一切正常，但每天傍晚尖峰（流量約平時 5 倍）時，p99 延遲會從 80ms 跳到 4 秒，持續約十分鐘，CPU 其中一核貼 100%。你懷疑是某個 hot path 有同步重運算，但只在尖峰時資料量大才觸發。**設計一個壓測情境去重現並坐實這個尾延遲**，講清楚你用什麼負載模型、為什麼，以及怎麼避免協調遺漏騙了你。

### 推演解答

**目標先講清楚**：要重現的是「**高並發 + 大資料量**同時出現時的尾延遲爆炸」，而且要量**準**——不能讓壓測工具在系統變慢時陪著慢下來，否則就量不到那條 4 秒的尾巴。

**設計要點**：

1. **用開放模型 + 固定到達率，不用封閉式 VU 迴圈。** 理由就是協調遺漏：傍晚尖峰的真實流量是開放的（使用者不管伺服器多慢都照發請求）；如果壓測用封閉模型，系統一變慢壓測就少發請求，那條 4 秒尾巴會被「協調著遺漏」掉，你會得到一份漂亮的假報告。用 Vegeta（天生固定速率）或 k6 的 constant-arrival-rate executor。
2. **打到真實尖峰的到達率**：平時若是 200 RPS，尖峰 5 倍就設 **1,000 RPS 固定到達率**，持續至少十分鐘（對齊真實尖峰時長），讓積壓有時間長出來。
3. **重現「資料量大」這個觸發條件**：光打高 RPS 不夠——你懷疑是「尖峰時資料量大才觸發同步重運算」。所以壓測的 request payload / 查詢參數要刻意做成**會命中大資料路徑**的（大 payload、會回傳大量列的查詢條件），模擬尖峰時段那種「資料也變多」的真實狀況。只壓 RPS 不壓資料量，可能根本踩不到那行同步運算。
4. **確認壓力源沒偷工**：盯著 k6 的 `dropped_iterations` 必須是 0（否則 VU 池被榨乾、協調遺漏又回來了）；`preAllocatedVUs`/`maxVUs` 設得夠大。Vegeta 則確認它確實維持了設定的固定速率（看它的 rate 報告）。
5. **量對指標**：看 **p99 / p999**（不是平均——平均會被協調遺漏和大量快請求稀釋掉）；同時在受測機器上量 **event loop lag** 與**單核 CPU**。如果重現成功，你會看到：到達率撐住 1,000 RPS、event loop lag 在資料變大時飆起、單核 100%、p99 跳到秒級——三者同時出現就坐實了「同步重運算卡 event loop」這個假設。
6. **收尾：用 profiling 抓兇手。** 壓測重現延遲的當下，對受測行程跑 `node --cpu-prof`（或連 inspector），抓火焰圖找寬且 self time 大的那一格——那就是你要找的同步重運算。壓測負責「**穩定重現**」，profiling 負責「**指認兇手**」，兩者配套。

**常見錯路**：(a) 只壓 RPS 不壓資料量 → 踩不到觸發條件，重現不出來。(b) 用封閉模型 + 看平均延遲 → 協調遺漏 + 平均稀釋，雙重低估，得到假綠燈。(c) 沒盯 `dropped_iterations` → 以為自己用了開放模型，其實 VU 池早被榨乾在偷丟請求。這三個錯任一個，你都會「壓測重現不出 production 的問題」然後懷疑人生——而真正的問題是壓測設計，不是系統。

## 自我檢核

口頭自答，講得清楚才算過關：

1. 「Node 是單執行緒」這句話，精確來說是哪條執行緒單一？libuv thread pool（預設 4）和網路 I/O 各自跑在哪、為什麼它們不違反「你的 JS 跑在唯一執行緒上」？
2. 為什麼在 Node，一個 200ms 的同步函式能讓 1,000 RPS 變成 5 RPS，而同樣的函式在多執行緒的 Java 服務裡只會拖慢它自己那條執行緒？關鍵差別是哪兩個字（提示：搶占）？
3. 「CPU 密集就會卡 Node」這句話哪裡不精確？一個用 `worker_threads` 的 CPU 密集任務為什麼不卡？真正的判準是什麼？
4. 四種記憶體洩漏（閉包/listener/無上限快取/全域 Map）有什麼**共同的形狀**？為什麼說「洩漏永遠藏在那個你沒寫的『出』」？
5. 你 LetsTalk 通知系統那個「listener 只 on 不 off」的洩漏，為什麼會**同時**是記憶體洩漏和 CPU 瓶頸？一次廣播怎麼從跑 3,200 個 callback 變成跑 30 萬個？
6. 火焰圖上，你該先看寬度還是高度？「寬、但底下沒有子條」代表什麼（提示：self time）？為什麼「最高的那條」不一定是兇手？
7. 什麼是協調遺漏？為什麼封閉式/固定速率壓測會在系統變慢時**低估**尖峰尾延遲？你要怎麼設計壓測才不會被它騙（負載模型 + 看哪個百分位 + 盯哪個數字）？
8. （老兵記憶校正）Clinic.js 今天的維護狀態是什麼（2026-06）？如果今天要做同樣的 CPU/記憶體診斷，你會優先用什麼現役工具、為什麼那些工具比較不會踩到版本相容問題？

| 軸 | 這個系統的答案與代價 |
|---|---|
| **交付** | 不直接是交付問題；但 event loop 被卡住時，in-flight 的請求 callback 全堵在隊伍後面——「沒掉，只是無限延遲」，對上游（負載均衡器、客戶端逾時）等同於丟失。 |
| **持久** | 記憶體裡的狀態（未清的 listener、堆積的 callback 隊伍、未淘汰的快取）**重啟即失**——OOM 重啟會清掉洩漏，但也清掉所有 in-flight 工作；洩漏的「治標」其實就是定期被迫重啟（見 ch13 持久化要主動設計）。 |
| **分區** | 單機問題，不涉跨節點一致性；但一台機器被同步運算卡死、健康檢查逾時被踢出輪替時，會把壓力推給其他機器→連鎖（雪崩的起點）。 |
| **並發** | 本章核心。event loop 是**單執行緒非搶占並發**：沒有鎖、沒有 race（單迴圈內），但任何一個 callback 跑太久就讓全部人乾等——並發的強處與死穴是同一個設計（深掘見 ch12）。 |
