這一章不再深掘任何新東西。它的工作只有兩件：第一，把你走過的十八章地圖收攏，讓你看見這趟旅程的全貌；第二，把這套工具交到你手上——讓你能拿它去挖**這本書沒涵蓋的**系統（包括你現職 Bahwan 正在建的那些）。

## 這些底子怎麼接上 AI Infra：搭橋，不教 LLM

現在這座橋。你的 `plan.md` 和你的下一份工作目標都指向 AI Infrastructure——LLM serving、GPU、推理叢集。你可能擔心：我這六年的後端經驗，跟「跑大模型」是兩個世界吧？

**不是。** 這是這一章最想讓你帶走的判斷：**LLM serving 的底層問題，幾乎全是你這本書已經挖穿的老問題，只是換了一層皮。** 高併發、佇列、冪等、最終一致、背壓、觀測——這些不是「後端的」問題，它們是「任何要在不可靠硬體上、用有限資源、服務大量並發請求的系統」的問題。LLM serving 恰好就是這種系統，所以它撞上的是同一批老問題。

我給你三組具體對應。**注意：這只是搭橋——指出「你已經有的直覺對得上 AI Infra 的什麼問題」，不是教你 LLM。每一個對應的深度都在《從後端到 AI Infra》（aiib，書架上已完成）裡，這裡一律點到為止、不展開內容。**

**對應一：KV cache routing ≈ 你的 session affinity（sticky session）。**
你在 ch07 為了 WebSocket 長連線做過 sticky session——連線狀態綁在某台機器，「發給這個用戶」的請求必須路由回那台。LLM serving 有一個結構幾乎一樣的問題：一個多輪對話的 KV cache（模型對前文算過的中間狀態）很貴，你會希望同一個對話的後續請求**路由回快取著它的那台 GPU**，而不是每次重算。這就是「狀態綁定在某個節點、後續請求要找回那個節點」——和你的 session affinity 是同一個形狀的問題。你對「stateful 連線為什麼難水平擴展」的直覺，直接遷移過去（深度見 aiib）。

**對應二：continuous batching ≈ 你的 batch 取捨。**
你在 ch03 做過批次化——N 次逐列更新縮成 1 次，拿到雙重收益，但你也知道批次太大會反咬（長交易、鎖範圍擴大）。LLM serving 的 continuous batching 是同一道取捨的另一個化身：把多個推理請求湊成一批一起算（吞吐上升），但批次怎麼湊、什麼時候不等了先發——這是「吞吐 vs 延遲」的經典拉扯，正是你 ch14、ch03 反覆做過的判斷。你對「批多大才划算、批太大輸在哪」的直覺，直接遷移過去（深度見 aiib）。

**對應三：autoscaling GPU ≈ 你的過載防禦。**
你在 ch14 建過一整套過載防禦語言——背壓、rate limiting、load shedding、circuit breaker、退避＋jitter；你知道無界佇列會讓延遲無限長、retry storm 會放大流量。GPU 推理叢集面對流量尖峰的 autoscaling、排隊、過載卸載，是**同一套防禦語言**套在「GPU 比一般機器貴得多、冷啟動慢得多」的約束上——這約束讓取捨更尖銳，但問題的形狀沒變。你對「洪水來了怎麼不死」的整套直覺，直接遷移過去（深度見 aiib）。

把這三組對應收成一句話：**你不是要從零學 AI Infra，你是要把你已經有的五軸直覺，套到一個新的硬體約束（GPU 貴、KV cache 大、推理延遲敏感）上。** LLM serving 在五軸上同樣有它的答案——交付（請求會不會掉/重）、持久（節點掛了 in-flight 的推理）、分區（多副本一致）、並發（同一 GPU 上多請求搶資源）、觀測（推理延遲 p99 怎麼看）。這把尺照樣量得了它。aiib 那本書做的，就是把這把你已經會用的尺，對準 LLM serving 的每一格——所以你讀 aiib 不會是「學新東西」，會是「把舊直覺認出來」。

**這一段的邊界劃清楚：本書到此為止。** 我不講 PagedAttention 怎麼運作、不講 GPU 排程、不講 K8s GPU operator——那些是 aiib 的主場。本書的工作只是讓你**看見那座橋的兩端是同一批老問題**，這樣你跨過去的時候不是空手，是帶著六年直覺過去的。

## 本書沒挖到的：誠實的邊界與指路

一本誠實的書要說清楚自己**沒**做什麼，否則讀者會把「這本書沒講」誤讀成「這東西不重要」。本書是「你建過的系統往下挖一層」，所以有幾個重要的鄰居，本書**刻意不碰**，每一個我給你指一條路：

**分散式共識（Raft / Paxos）。** 本書處理了「最終一致」（ch04、ch11），但你會注意到一件事：你建過的系統**沒有一個真的需要強一致的分散式共識**。四服務最終一致靠補償對帳、SQS 靠 at-least-once＋冪等——你一直在用「弱一致＋冪等」這條更便宜的路繞過共識。這是對的選擇（共識很貴），但代價是你沒挖過「一群節點怎麼對一個值達成不可推翻的一致」這個問題（leader election、log replication、quorum）。**想挖穿它，去《把系統寫成定理：TLA+》（tla）**——它用形式化方法把並發正確性與共識協定攤開，是你「想證明這東西真的對」時該去的地方。

**排隊的數學。** 本書 ch14 講了背壓的工程機制（token bucket、load shedding、circuit breaker），但我一路把數學擋在門外——為什麼 ρ→1（利用率逼近 100%）時延遲會爆炸式成長、Little's Law、M/M/1、Kingman 公式、尾延遲為什麼那麼厚。你 ch05 寫負載測試、ch15 看 p99，骨子裡都在跟這些數學打交道，但你是憑直覺。**想把直覺換成公式，去《等待的數學：排隊理論》（queue）**——它會告訴你「五軸的代價到底多大」的量化答案。

**故障的跨領域通論。** 本書的「故障模式與防禦」段落**綁定你自己的系統**——當年那個 pipeline、那條連線、那個佇列。但複雜系統怎麼崩壞，有一套跨領域的通論（Normal Accidents、緊耦合與複雜互動、瑞士起司模型、為什麼大事故總是多個小故障疊起來的）。**想看你的系統故障放進更大的圖景，去《正常意外》（fail）**——它講的是「複雜系統為什麼註定會以你想不到的方式壞」這個通則。

**分散式時鐘。** ch13 講了「時鐘不可信、跨機器沒有共同的現在」，但我把 Lamport 時鐘、向量時鐘、因果排序這些**形式化工具**擋在門外（只點名沒展開）。你處理過的時間問題都是「單機時鐘漂移」「排程補跑」這種工程層面的；「在沒有全域時鐘的情況下怎麼判斷事件因果順序」這個更深的問題，本書沒挖。**想挖，同樣指向 tla 與分散式系統的經典文獻**（ch13 延伸閱讀有列）。

看出這四個鄰居的共同點了嗎？**它們都是「你的系統剛好沒逼你面對、所以你沒挖過」的那一層。** 你用最終一致繞過了共識、用直覺繞過了排隊數學、用「綁定自己系統」繞過了故障通論、用工程手段繞過了分散式時鐘理論。這不是疏漏——這是**好的工程判斷**（用更便宜的路解決夠用的問題）。但現在你知道那些更深的層在哪、什麼時候會需要、該去哪本書找。**知道自己不知道什麼、且知道去哪補**，本身就是資深的標誌。

## 如果只帶走三件事

十八章很長。如果三年後你忘了大半，我希望你還記得這三件——它們是這本書真正的脊椎，其他都是註腳。

**第一件：拿到任何系統，先做五軸體檢。**
交付會掉嗎會重嗎、持久重啟活得了嗎、分區誰看到不一致、並發誰跟誰打架、觀測壞了你怎麼知道。這五個問句是你面對**任何**系統（包括這本書沒涵蓋的、包括 AI Infra 的、包括你還沒遇到的）時的第一個反射。它不給你答案，它逼你問對問題——而問對問題是機制級理解的起點。一個你能在五軸上都說清楚取捨的系統，就是一個你真正懂的系統。

**第二件：at-least-once + 冪等是分散式系統的預設安全帶。**
因為 exactly-once *交付*是幻覺（丟一個 ack 就分不清沒送到還是 ack 丟了，ch10），所以在任何跨網路的通道上，你的預設假設應該是「訊息會重」，而你的預設防禦應該是「讓重複處理同一筆不會出錯」（冪等）。你的結算 pipeline、你的 delayed job 重啟、未來你碰到的任何訊息系統——都是這同一條安全帶。所謂的 exactly-once *處理*（SQS FIFO、Kafka EOS）不是魔法，它就是「at-least-once 交付 + 冪等去重」拼出來的。記住這條，你就不會再被「我用了 FIFO 是不是就不用冪等了」這類問題騙到。

**第三件：每個取捨都要說得出被放棄的選項輸在哪一維。**
這是全書對「懂」的最高標準，也是 ch19 面試敘事的骨架。「我選了 Jitsi」不算懂；「我選了 Jitsi，因為 Twilio/Agora 是商用按量計費、Jitsi 開源可自架零授權費，我用『自己要運維 JVB 和 TURN』換『成本可控加完全掌控』」——這才算懂。維度就那幾個：成本、延遲、運維、故障域、學習曲線、授權。一個你說不出「另一端輸在哪一維」的決定，代表你記的是**結論**不是**取捨**——而結論會過時（ch01 那條老兵記憶暗線），取捨的推理框架不會。

這三件事有個共同的形狀：**它們都不是知識，是反射。** 知識會過時（landscape 那整檔講的就是這個），反射不會。五軸體檢、冪等預設、取捨還原——這三個反射裝進你腦子之後，你拿它們去挖任何新系統都管用，包括下面這個。

**第三，這就是你的下一個挖穿目標。** 上面那張表的每一格代價欄，都是一個你可以往下挖的洞。比如「cache stampede 要不要加 single-flight 鎖」——這是個取捨（鎖的複雜度 vs 源被打爆的風險），照本書的標準，你該說得出「不加，輸在熱 key 過期瞬間源被 N 個請求打爆；加，輸在多了一層鎖的複雜度與它自己的故障模式」。你看，又回到了「帶走的第三件事」。

順帶把你 Bahwan 另一個系統也點一下當練習引子：你那個 **Lambda decorator framework**（class-validator DTO binding、宣告式參數注入、集中式例外映射，現已用於 30–50 個 handler、CI 在 API 未文件化時擋下 build）。對它做五軸體檢的角度會很不一樣——它本身是無狀態的（Lambda 的前提，ch18），所以「持久」軸該問的是「它依賴的 Lambda 冷啟動（landscape §17）對首次請求延遲的影響」，「並發」軸該問的是「decorator 裡有沒有跨請求共享的可變狀態（在 Lambda 同一個 execution environment 複用時會咬你）」。這個留給你當紙上推演。

### 推演一：五條主線口頭自答總測（全書驗收題）**[30 分鐘] ★★**

蓋住全書，對自己口頭回答 style-guide 那五條驗收主線。每條都要舉你**自己系統**的例子，不准只說抽象定義：

1. 對任一系統做五軸體檢，講出它在每一軸的答案與代價。
2. 為什麼 exactly-once *交付*是幻覺、exactly-once *處理*怎麼是「at-least-once + 冪等」拼出來的。
3. 還原你三個取捨（Jitsi vs Twilio/Agora、cron vs setTimeout、拆交易 vs 不拆），各講出被放棄選項輸在哪一維。
4. 對三個名詞各講一個「我以為 vs 實際」的落差（含至少一個是過時記憶、不是機制錯誤）。
5. 對任一系統，答「它壞的時候我第一時間怎麼知道」。

#### 推演解答

這是口頭過關制，沒有標準答案，但有**過關線**：每一條你都得能舉出自己系統的具體例子、且講到機制層（不是名詞層）。給你自評的判準：

- **第 1 條**過關長相：「LetsTalk 即時通知，交付＝在線 at-most-once（Pub/Sub）／離線另走持久路，代價是在線那段掉了不留痕跡；持久＝連線狀態在記憶體，重啟全體重連打 Redis……」——能逐軸帶出**答案＋代價**才算過。只說「交付就是訊息會不會掉」是背定義，不過關。
- **第 2 條**的機制核心是「丟一個 ack」：送方送完等 ack，ack 丟了它分不清是訊息沒到還是 ack 丟了，只能重送（at-least-once）或放棄（at-most-once），沒有第三條路（ch02/ch10）。講得出這個情境才算懂，說「因為網路不可靠」太籠統。
- **第 3 條**最容易在「Jitsi」上栽——如果你還說「因為 Twilio 要關了」，那你漏掉了 ch08 最關鍵的老兵記憶校正（Twilio 2024-10-21 撤回，landscape §6），這條不過關。正解是純成本／自架掌控／開源授權。
- **第 4 條**要包含至少一個**過時記憶**（曾經對、現在錯——如 Bull/Twilio/Node LTS/Redis 授權）和它跟**機制錯誤**（從來就不對——如 Pub/Sub 會排隊補送、JWT 登出即失效）的區別。混為一談就沒抓到 ch01 暗線的精髓。
- **第 5 條**如果你對某系統的答案是「等客訴」，誠實承認那是觀測軸的洞——**承認洞比假裝答滿更接近過關**（ch02 填表第 3 條規矩）。

哪一條卡住，回去翻它的主場章。這一題卡的數量，就是你這本書還沒讀透的章數。

### 推演二：把一個 AI Infra 概念對應回你的某個舊系統（橋接練習）**[20 分鐘] ★★**

從本章三組對應（KV cache routing ≈ session affinity、continuous batching ≈ batch 取捨、autoscaling GPU ≈ 過載防禦）中**挑一組**，用你**自己系統**的細節把這個類比講深一層：你舊系統的哪個具體決策／故障，對應到這個 AI Infra 問題的哪一面？並誠實指出這個類比**在哪裡會破**（類比都有極限）。注意：你只需要用你已有的直覺，**不需要懂 LLM 的內部**——破在哪不確定時，標「這要去 aiib 確認」。

#### 推演解答

示範一組（你挑別組同理）。**KV cache routing ≈ 你的 session affinity：**

對應的深處：你 ch07 的 WebSocket sticky session，核心問題是「狀態（連線）綁在某台機器，後續請求得路由回那台，否則狀態不在」。KV cache routing 的核心問題一模一樣——「狀態（多輪對話算過的 KV cache）綁在某張 GPU，後續請求最好路由回那張，否則要重算」。你對「為什麼 stateful 不能均勻飄在無狀態機器上」「sticky 沒做好同一個 session 散到多台會怎樣」的直覺，直接對得上。

類比**在哪裡破**（這才是高品質回答的關鍵）：

- 你的 session affinity，狀態不在就是**重連**（用戶體感是斷一下）；KV cache 不在是**重算**（延遲與算力成本，但功能上沒斷）——失效的「代價形態」不同。
- 你的連線狀態幾乎沒有「值不值得搬過去」的問題（連線就在那）；KV cache 有「要不要把它從一張 GPU 搬到另一張」的取捨（搬的成本 vs 重算的成本）——這一層你的舊系統沒有對應，**這裡要去 aiib 確認**。

自評判準：你有沒有（a）用自己系統的**具體細節**講對應，不是泛泛說「都是狀態綁定」；（b）誠實指出類比的**極限**並在不確定處標「去 aiib」。兩個都做到就過關。常見錯路：把類比講成「完全一樣」——任何類比都有破口，找出破口才證明你真的在用機制思考而不是套口號。

### 推演三：對 Bahwan 的 Lambda decorator framework 做五軸體檢＋寫「我接下來要挖穿的三個名詞」**[30 分鐘] ★★★**

兩部分。

**(a)** 照本章 cache library 的示範，對你 Bahwan 的 **Lambda decorator framework**（class-validator DTO binding、宣告式參數注入、集中式例外映射、30–50 handler、CI 擋未文件化 API）自己填一張五軸體檢表。提示：它無狀態（Lambda 前提），所以五軸的吃重點和有狀態系統很不一樣——重點在「它依賴的 Lambda 執行模型（冷啟動、execution environment 複用）怎麼從五軸咬它」。

**(b)** 寫下「我接下來要挖穿的三個名詞」清單——從你現職用到、但本書沒涵蓋、你自覺只有工作級熟練的名詞裡挑（例：NestJS 的 DI 容器內部、PostgreSQL partition 的查詢規劃、class-validator 的 reflection 機制、CloudFormation 的 drift detection……）。每個寫一句「我以為它怎麼運作」（封存起來，像 ch01 那樣，等你挖完回來對照）。

#### 推演解答

**(a) 參考體檢**（你的版本抓到結構即可）：

- **交付**：decorator framework 本身不是訊息通道，這軸大致不適用——但「集中式例外映射」碰到交付的邊緣：一個 handler 拋錯被映射成 HTTP 回應後，**上游（API Gateway/SQS 觸發）會不會因為這個錯而重投**？如果是 SQS 觸發的 Lambda，例外映射成「成功回應」會吃掉重試、映射成「失敗」會觸發重投（回到 ch10 冪等）——這格其實藏著一個你大概沒想過的交付決策。
- **持久**：無狀態是 Lambda 的前提（ch18），所以「進行中工作重啟會丟嗎」的答案是「每次 invocation 本來就是獨立的、沒有跨 invocation 的進行中狀態」——這軸 Lambda 幫你答了。但**冷啟動**（landscape §17，Node.js 不支援 SnapStart）會讓首次／擴容時的請求延遲尖峰，這是持久軸的鄰居（環境的建立成本）。
- **分區**：framework 層幾乎不碰——但如果 decorator 裡有「啟動時讀一次設定快取在 module scope」的東西，那它會跨 invocation 在同一個 execution environment 裡被複用，多個 warm 環境之間的設定可能不一致（分區的影子）。
- **並發**：**這是無狀態 framework 最該查的一軸**——decorator 或其依賴裡有沒有**跨請求共享的可變狀態**？Lambda 同一個 execution environment 會**序列**處理請求（單一 env 不並發），但會被**複用**，所以「上一個請求在 module-scope 變數留下的髒狀態」會洩漏到下一個請求。這是 Lambda 最經典的陷阱，正好落在並發軸（嚴格說是「跨請求狀態汙染」）。
- **觀測**：集中式例外映射是個觀測金礦也是個觀測陷阱——它讓錯誤格式統一（好觀測），但如果它把不同根因的錯誤都映射成同一個對外訊息，你會在日誌裡分不清是哪一種錯（壞觀測）。該問：映射前有沒有保留原始錯誤的結構化日誌？

看出來了嗎——即使是個「無狀態的小 framework」，五軸照樣逼出五個你大概沒從這角度想過的問題（尤其並發軸的跨請求狀態汙染、交付軸的例外映射影響重投）。**這就是這把尺的價值：它對任何系統都問得出好問題。**

**(b)** 這部分是開放的，過關判準同 ch01 推演一：「我以為」要寫**機制斷言**（可被證偽）不是寫**用途**。反例（沒對照價值）：「NestJS DI：我以為它是用來管理依賴的。」好的寫法：「NestJS DI 容器：我以為它在每次請求都重新建立 provider 實例。」（這是個可被打臉的機制斷言——預設其實是 singleton scope，跨請求複用）。挑三個你**最有把握**的名詞下手，因為「最有把握」的地方最可能藏著你沒挖過的機制——這正是 ch01 第一句話的意思：你寫得出名詞，不代表你懂它底下。封存它，這就是你這本書的續集——只是這次作者和讀者，都是你自己。
