## 從你已知的出發

當年那張 CloudWatch 圖你大概忘不掉：NeoBards（Dynasty Warriors Mobile，500K+ 使用者、10K+ CCU）的 RDS，每到結算或活動尖峰，CPU 利用率就頂到 90% 以上、貼著天花板走一條平頂線。你做了該做的事：把 slow query log 打開，找出尖峰時段那幾條反覆出現的 query，然後動手。最後你把尖峰 CPU 砍了 40%。履歷寫得乾淨俐落：「尖峰 RDS CPU 降 40%（profiling 驅動：重排 SQL、拆解交易降鎖競爭、批次化逐列更新）」。

你當年的四個手段，逐一回放是這樣的：

- **重排 SQL 執行順序**：同一個交易裡，幾條 `UPDATE` 的先後順序你動過。原本可能是「先更 inventory 再更 player」，你改成所有交易都「先 player 再 inventory」，或反過來——重點是**統一**。
- **拆解大交易**：一個包了五六步、跑好幾百毫秒的大交易，你把它切成幾個小交易。原本一個 `BEGIN…COMMIT` 框住的東西，變成兩三個各自 commit 的小段。
- **跨交易重排降熱點**：不只交易內部，連「哪個交易先碰哪一行」你也調過。某幾行（比如公會表的某個熱門公會那一列）大家都要動，你想辦法讓它們不要同時搶。
- **批次化逐列更新**：原本一個迴圈跑 100 次、每次一條 `UPDATE WHERE id = ?`，你改成一條 `UPDATE … WHERE id IN (…)` 或一次 `INSERT … ON DUPLICATE KEY UPDATE`，把 100 次往返壓成 1 次。

這四件事你都做對了。但你可能沒想過的是：**它們四個其實在解同一個問題，而那個問題不是「CPU 不夠」。** 你大概一直把它記成一個效能優化——「query 變快了所以 CPU 降了」。這個直覺有一半是錯的。RDS 那條貼天花板的 CPU 線，主要不是被「算術」推高的，是被**鎖等待**和**重複掃描**推高的。你那四個手段，沒有一個是讓 CPU 算得更快——它們全都是在**縮短交易持有鎖的時間**、**減少彼此互等**、**減少同一份資料被反覆掃**。

這一章要做的，就是把這條線拉出來：為什麼「交易持有鎖的時間」會變成「別人燒掉的 CPU」？為什麼「同樣的 query、不同的順序」差那麼多？拆交易降了鎖競爭，但你那一刀切下去，是不是也悄悄放棄了什麼？這些才是那 40% 底下真正在發生的事。

## 鎖競爭：別人等你的時間，是被你的交易長度決定的

先把一個你大概沒認真區分過的東西分清楚：**「鎖」不是一個東西，是好幾種granularity 疊在一起的協議。**

### 行鎖與意向鎖：兩層協議

你在 NeoBards 用的是 RDS for MySQL，引擎是 InnoDB。InnoDB 做的是**行級鎖（row-level lock）**——當一個交易要改某一行，它在那一行的索引記錄上掛一把**排他鎖（exclusive lock，X lock）**；要讀而且要鎖住（`SELECT … FOR UPDATE`）則可能是**共享鎖（shared lock，S lock）**。同一行上 X 鎖互斥、X 與 S 互斥、S 與 S 相容。這是你大概知道的部分。

你大概**沒**深想的是它上面那一層：**意向鎖（intention lock）**。InnoDB 在你掛行鎖之前，會先在**整張表**上掛一把意向鎖——要掛行的 X 鎖就先掛表的 IX（intention exclusive），要掛行的 S 鎖就先掛表的 IS（intention shared）。意向鎖之間幾乎全相容（IX 跟 IX 相容、IX 跟 IS 相容），它存在的唯一目的是：**讓「有沒有人正在鎖這張表裡的某些行」這個問題，不用掃過每一行就能回答**。

舉個你會碰到的場景：當某個操作要鎖**整張表**（比如某些 DDL、或顯式 `LOCK TABLES`），它得確認「現在沒有任何交易正握著這張表裡任何一行的鎖」。如果沒有意向鎖，它得逐行檢查——表有幾百萬行就掃幾百萬次。有了意向鎖，它只要看一眼表上「有沒有 IX/IS」就知道答案。意向鎖是一個**為了快速做相容性判斷而設計的索引**，不是用來擋你的——它幾乎從不造成你交易之間的等待。**真正讓你互等的，永遠是行鎖（和下一節的 gap lock）。**

把這層分清楚的好處是：當你 debug 鎖競爭時，意向鎖是雜訊，行鎖才是訊號。你看 `SHOW ENGINE INNODB STATUS` 時那一堆 `TABLE LOCK … IX` 不用管，要盯的是 `RECORD LOCKS … X` 那幾行。

### 持有時間就是別人的等待時間

這是整章的軸心，值得你停下來——因為你當年那四個手段，全都是這一句話的不同寫法：

> **一個交易持有某把行鎖的時間，等於所有想動那一行的其他交易被迫排隊等待的時間。**

InnoDB 的行鎖**直到交易 `COMMIT` 或 `ROLLBACK` 才釋放**（這是 two-phase locking 的「2PL」精神：鎖只增不減，到交易結束一次放完）。不是「`UPDATE` 那一行執行完就放」，是**整個交易結束才放**。這一點是老兵最容易記錯的地方之一——你大概隱約覺得「我 update 完那行就沒事了」，不是的，那把 X 鎖會一直掛到 commit。

所以一個交易長這樣：

```
BEGIN
  UPDATE guild  ...   ← 在 guild 那行掛 X 鎖
  （呼叫某個外部 API，等 200ms）
  UPDATE player ...   ← 在 player 那行掛 X 鎖
  （算一堆東西，又 50ms）
  UPDATE inventory ...
COMMIT                ← 此刻三把 X 鎖一起放開
```

這個交易從第一個 `UPDATE guild` 開始，guild 那行就被鎖住了；而它要到最後 `COMMIT` 才放。中間那 200ms 的外部 API 等待、那 50ms 的計算——**guild 那一行整段都鎖著，別人完全沒在用它，純等。** 任何其他想動 guild 那行的交易，全部在排隊。這就是你當年大交易最毒的地方：**交易裡只要有任何一步慢（外部呼叫、複雜計算、甚至只是 SQL 寫得爛多掃了幾萬行），它已經鎖住的所有行，都陪著一起被凍結那麼久。**

現在回頭看你那條貼天花板的 CPU 線。當 500 個交易都想動同一批熱門公會的那幾行、而每個交易都因為前面那個慢交易而排隊——它們在做什麼？大部分在**等鎖**（這部分其實是 CPU 在 sleep/wait，不直接燒 CPU），但更陰險的是：**每個被擋住的請求背後是一個沒釋放的連線、一個累積的 backlog；當鎖終於放開，幾百個積壓的交易同時湧上去搶，重新解析、重新規劃、重新掃描索引——這一波「驚群」式的同時重啟才是把 CPU 推上去的主力。** 加上很多被擋的交易最後 `innodb_lock_wait_timeout`（InnoDB 預設 **50 秒**，2026-06）到了被回滾，應用層重試，**同一份工作被做了兩遍、三遍**——每一次重試都是一輪完整的 parse + 掃描 + 鎖嘗試，CPU 就是這樣被「等待 + 重試」雙重燒上去的，不是因為算術變多了。

這就是為什麼**拆交易能降 CPU**：你把那個包了外部 API 的大交易拆開，外部 API 移到**任何鎖都不持有**的時候去呼叫，交易本身只剩「掛鎖 → 寫 → commit」這幾毫秒。持有時間從 250ms 砍到 3ms，別人的等待時間跟著砍掉 99%，驚群和重試一起消失。CPU 降下來不是因為你算得快了，是因為**你不再讓幾百個交易為了一個慢交易陪等與重試**。

### 為什麼「同樣的查詢、不同的順序」差很多

你當年「重排 SQL 執行順序」那一刀，表面上 query 一模一樣、只是先後不同，怎麼會有差？這裡有兩個完全不同的機制，別混為一談。

**機制一：死結預防（deadlock prevention）。** 考慮兩個交易，各要更新 player A 和 player B 兩行：

```
交易 T1：UPDATE player A → UPDATE player B
交易 T2：UPDATE player B → UPDATE player A
```

如果 T1 先鎖到 A、T2 先鎖到 B，然後 T1 要 B（被 T2 握著）、T2 要 A（被 T1 握著）——**互等，死結**。InnoDB 會偵測到（它在內部維護一張 wait-for graph，找到環就挑一個「改動較少的」交易當犧牲者回滾，回傳 error 1213，2026-06 查證）。被回滾的那個交易，剛剛做的一切白做，應用層收到錯誤後重試——**又是一輪 CPU**。

死結雖然 InnoDB 會自動解，但「解」的代價是**回滾 + 重試**，在高並發下這個代價會累積成可觀的 CPU。而**根治死結的標準手法，就是讓所有交易以同一個順序獲取鎖**：如果 T1 和 T2 都規定「先 A 再 B」（比如一律按 primary key 升序更新），那永遠不會出現「你握我要的、我握你要的」這種環——最壞情況只是後到的乖乖排隊等先到的，不會互鎖。你當年「重排 SQL 執行順序、統一順序」那一刀，多半就在做這件事：把死結出現的機率壓到零，省掉那些回滾 + 重試的 CPU。

**機制二：熱點行的等待結構。** 就算沒有死結，順序也影響等待。假設每個交易都要動「公會表的某熱門公會那一行（人人都搶）」和「自己玩家那一行（沒人搶）」。如果交易寫成「**先**鎖熱門公會行 → 做一堆事 → 再鎖玩家行 → commit」，那把全場唯一的瓶頸資源（熱門公會行）**在交易最早就鎖住、握最久**，所有人卡在這裡。但如果改成「**先**把不搶的玩家行、計算、外部呼叫全做完 → **最後一刻**才鎖熱門公會行、立刻 commit」，那把瓶頸資源的持有時間壓到最短——這就是「跨交易重排降熱點」的精髓：**讓最搶手的那把鎖，最晚拿、最早放。** 同樣的 query、同樣的工作量，只是把熱點行的 `UPDATE` 挪到交易尾巴，整個系統的吞吐就能翻倍，因為大家在熱點上排隊的時間結構變了。

### 你的「我以為 vs 實際」

| 你大概記得的 | 實際上 |
|---|---|
| 行鎖在 `UPDATE` 那行執行完就釋放 | 掛到**整個交易 commit/rollback** 才釋放（2PL），交易越長鎖越久 |
| CPU 高是因為 query 算得慢、要加 CPU/換機型 | 尖峰 CPU 主要被**鎖等待後的驚群重啟 + 超時重試**推高，是並發問題不是算力問題 |
| 重排 SQL 順序只是 cosmetic | 統一鎖獲取順序能**根除死結**（省回滾重試）、熱點行延後鎖能降等待——同一批 query 吞吐差數倍 |
| 意向鎖會擋我的交易 | 意向鎖幾乎全相容，只為快速判斷表級相容性而存在，**不是你互等的原因**——行鎖與 gap lock 才是 |

## InnoDB 的 Repeatable Read 與 gap lock：連「插入」都會卡

這一節是你最可能踩到、又最難自己看出來的坑，值得單獨挖，因為它顛覆一個你大概深信不疑的直覺：「我又沒改同一行，怎麼會卡？」

### 預設就不是你以為的那個

第一個老兵記憶校正：**MySQL/InnoDB 的預設隔離級別是 REPEATABLE READ**（不是 Read Committed）。這跟 **PostgreSQL 預設 Read Committed** 不一樣（2026-06 查證）。如果你當年也碰過 PG，這兩個的預設行為差很大，別把記憶串線了。（兩者 RR 的本質差異——PG 的 RR 其實是純 snapshot isolation、不上鎖也不防 write skew，MySQL 的 RR 是 locking + MVCC 混合——這條對立留到 ch11 並排講，這裡只用到 MySQL 這半。）

### next-key lock：record + 前方 gap

InnoDB 在 REPEATABLE READ 下，為了防止**幻讀（phantom read）**——也就是同一個交易裡兩次跑同樣的範圍查詢、第二次卻多出幾行——用的不是只鎖你掃到的那幾行，而是 **next-key lock**：

> **next-key lock = 索引記錄上的 record lock + 該記錄「前方那段 gap」上的 gap lock。**

gap lock（間隙鎖）鎖的不是任何一行，而是**索引上兩個現存值之間的「空隙」**。它的作用是純粹防禦性的（MySQL 官方原話：gap locks are "purely inhibitive"，2026-06 查證）——**它唯一的功能，就是禁止別人往這段空隙裡 `INSERT` 新行。**

舉個具體例子。假設 `player` 表的 `score` 欄有索引，現存值是 `10, 20, 30`。某個交易跑：

```sql
SELECT * FROM player WHERE score BETWEEN 15 AND 25 FOR UPDATE;
```

它掃到 `score = 20` 這一行（鎖住 record），但為了防止「幻讀」——防止別的交易插一個 `score = 18` 進來讓這個範圍查詢第二次跑時多一行——InnoDB 還會鎖住 **`10` 到 `20` 之間的 gap、`20` 到 `30` 之間的 gap**。結果就是：

> **另一個完全不相干的交易，想 `INSERT` 一個 `score = 18` 的新玩家——它沒有改任何現存行、沒有跟任何 record lock 衝突——卻會被卡住，因為 18 落在被 gap lock 鎖住的空隙裡。**

這就是那句反直覺的話：**在 InnoDB RR 下，`INSERT` 會被一個只是在跑範圍查詢的交易擋住，即使它們碰的根本不是同一行。** 機制細節：想插入的交易會先嘗試一把 **insert intention lock**（一種特殊的 gap lock），如果這段 gap 已被別人的 gap lock 鎖住，這個 insert 就得等——等到對方 commit 放鎖（2026-06 查證）。

### 這對你的 RDS 尖峰意味什麼

回到 NeoBards。結算或活動尖峰時，你有大量並發的「按範圍掃描 + 寫入」交易（比如「找出某公會排名在 X～Y 之間的玩家、發獎」這類）。如果這些交易用了範圍條件、又在 RR 下持有 next-key lock，那它們鎖住的**不只是掃到的行，還有一整段 gap**。這時候新玩家加入、新紀錄寫入這個 gap，全部排隊——**你會看到一堆 `INSERT` 莫名其妙地卡，而你盯著行鎖怎麼看都看不出衝突，因為衝突發生在「空隙」上、不在任何一行上。** 這種卡又會回到上一節那條因果鏈：等待 → 超時 → 重試 → CPU。

這就是為什麼「拆交易」和「縮短持有時間」在 InnoDB RR 下更重要：你持有 next-key lock 的時間越長，被你 gap lock 凍住、無法 `INSERT` 的範圍就被凍越久。**減少範圍掃描的鎖、或縮短持有時間，直接解放一整段空隙。** 一個常見的對策是針對這類純寫入熱點的連線改用 Read Committed（InnoDB 在 RC 下大幅減少 gap lock 的使用），但那要先確認你不需要 RR 的防幻讀保證——這是個取捨，不是免費午餐，細節 ch11 接棒。

## 交易拆解的雙面刃：你那一刀切掉了原子性

「拆交易降鎖競爭」是你四個手段裡最有效、也最危險的一個。有效，是因為它直接攻擊上面那條因果鏈的源頭（持有時間）。危險，是因為**交易的存在本來就是為了原子性——你把它拆開，等於把原子性親手交出去了。**

### 拆掉的到底是什麼

一個交易給你三個保證（ACID 裡的 A 和 I 在這最相關）：

- **原子性（Atomicity）**：交易裡所有寫入「要嘛全成功、要嘛全失敗」，沒有中間態落地。
- **隔離性（Isolation）**：交易執行中，別人看不到它的中間態。

當你把一個交易：

```
BEGIN
  UPDATE player    SET coins = coins + 100   -- 發獎
  UPDATE inventory SET items = items + 1     -- 發道具
COMMIT
```

拆成兩個各自 commit 的交易：

```
BEGIN; UPDATE player    ...; COMMIT;   -- 交易 1
BEGIN; UPDATE inventory ...; COMMIT;   -- 交易 2
```

你獲得了什麼：每個交易持有鎖的時間砍半，player 行和 inventory 行不再被同一把交易框綁在一起握到最後，鎖競爭下降。

你失去了什麼：**現在存在一個「中間態」——交易 1 commit 了、交易 2 還沒（或失敗了）。** 這個瞬間，玩家身上有 100 coins 但沒拿到道具。如果交易 2 因為任何原因失敗（連線斷、超時、行程被 OOM kill），**這個不一致會永久落地**：玩家白賺了 coins、或白丟了道具。原本的單一交易會把這種情況整個回滾，拆開之後沒有人幫你回滾交易 1 了。

### 什麼能拆、什麼不能：一條判準

不是所有交易都能拆。判準是問一句：

> **這兩步之間如果出現「只做了一半」的中間態，會不會造成無法接受的後果（守恆被破壞、錢/道具憑空生滅、約束違反）？**

- **能拆**：兩步之間在語意上**本就獨立**，或者中間態雖然存在但**可以被後續補償收斂**。例如「更新玩家最後登入時間」和「更新玩家在線狀態」——兩者沒有守恆關係，分開寫沒人會因此少拿東西。又例如批次發獎，每一筆獎勵彼此獨立，一筆失敗不影響其他筆的正確性（只要那一筆能重試）。
- **不能拆**：兩步之間有**守恆 / 借貸關係**，中間態違反一個必須恆真的約束。經典反例是「從 A 帳戶扣錢、往 B 帳戶加錢」——拆開後若第二步失敗，錢憑空消失，這是任何系統都不能接受的。又例如「扣庫存 + 建訂單」——拆開後若建訂單失敗，庫存白扣。

你當年能把那些交易拆開、而沒有引爆事故，是因為你（也許憑直覺）只拆了**能拆的那些**——多半是彼此獨立的批次寫入，或中間態可被結算 pipeline 的補償/對帳機制收斂的那些。**這正好就是 ch04 結算 pipeline 的故事**：當你不能靠單一交易保證原子性時，你改用「最終一致 + 冪等 + 補償」這套替代方案來收斂中間態。那條一致性接棒線，留到 ch11 把「強一致 ↔ 最終一致」整條譜挖穿。這裡你只要記住一句：**拆交易不是免費把鎖競爭變沒，是把「原子性」這個成本，從資料庫的交易機制，轉嫁到你應用層的補償邏輯上。** 如果你拆了卻沒補上補償邏輯，你只是把鎖競爭換成了資料不一致——那不是優化，是把炸彈埋深了一點。

## 批次化：N 次往返變 1 次的雙重收益，與批次太大的反咬

「批次化逐列更新」是四個手段裡最直觀、收益也最容易低估的一個——它同時砍掉兩種成本。

### 雙重收益：往返 + 鎖持有

假設你要更新 100 筆玩家資料。逐列版本：

```
for id in 100 個 id:
    UPDATE player SET ... WHERE id = ?   -- 一條 SQL = 一次網路往返
```

批次版本：

```
UPDATE player SET ... WHERE id IN (100 個 id)   -- 一條 SQL = 一次往返
-- 或 INSERT ... ON DUPLICATE KEY UPDATE 一次寫 100 行
```

**收益一（往返）**：應用層和 RDS 之間每條 SQL 都是一次網路 round-trip。NeoBards 跑在 AWS，應用層（ECS）和 RDS 之間就算同 region 同 AZ，一次往返的 RTT 算它 0.5～1ms。100 條就是 50～100ms 純粹花在「等網路來回」上——這段時間 CPU 在 idle，但**交易如果框住這 100 條，它就持有鎖持有了 50～100ms**。批次成 1 條，往返一次，省掉 99 次 RTT。

**收益二（鎖持有，這個更值錢）**：如果這 100 條更新在**同一個交易**裡（為了原子性），那逐列版本下，第一條 `UPDATE` 鎖到的那一行，要等到第 100 條跑完、交易 commit 才釋放——它被鎖了整整 100 次往返的時間（50～100ms）。而批次成 1 條後，這 100 行幾乎同時被鎖、幾乎同時釋放，**每一行的鎖持有時間從「100 次往返」壓縮到「1 次往返 + 一次寫入」的幾毫秒。** 在熱點行上，這個差異就是吞吐翻好幾倍的來源。這才是「批次化降 CPU」的主力——不是「資料庫執行 1 條比執行 100 條省 CPU」（其實寫入的工作量差不多），是**鎖持有時間崩塌帶來的鎖競爭崩塌**。

順帶一提，這也解釋了為什麼逐列 `UPDATE` 在迴圈裡跑特別毒：你在 ORM 裡寫一個 `for` 迴圈做 100 次 save，看起來很無辜，但如果外面包了交易，你就是在讓 100 行各自被鎖 100 次往返之久。這種「迴圈裡逐列寫」是 N+1 問題的孿生兄弟，老兵很容易視而不見。

### 批次太大的反咬

但批次不是越大越好，這是判準所在。把 100 萬行塞進一條 `UPDATE … WHERE id IN (一百萬個 id)` 或一個交易，會踩到三個反效果：

| 批次太大的反咬 | 機制 |
|---|---|
| **交易變長 → 鎖範圍擴大** | 一條鎖一百萬行的 `UPDATE`，這一百萬行（甚至加上 gap lock 鎖住的範圍）全程被鎖到 commit。你本來要解決鎖競爭，結果造了一把全表級的超大鎖，**反而擋住所有人**。 |
| **死結與超時風險上升** | 一個碰一百萬行的交易，跟別人的鎖重疊機率暴增；一旦超過 `innodb_lock_wait_timeout`（50s）或撞死結被選為犧牲者，**整批一百萬行的工作全部回滾重來**——你把「重試成本」放大了一百萬倍。 |
| **undo log / 複製延遲膨脹** | 巨型交易產生巨大的 undo log（MVCC 要保留舊版本給並發讀），還會在 binlog/複製鏈上造成一個巨大的事件，**拖慢 read replica 的複製延遲**（這條複製延遲線 ch11 接棒）。 |

所以正確的形狀是**分塊批次（chunked batch）**：不是「逐列 100 次」也不是「一次一百萬」，而是「每批 500～1000 行、each batch 各自 commit、批與批之間讓出」。這樣每一批的鎖持有時間夠短（不擋人）、又把往返和鎖競爭從逐列的 N 次壓到 N/批量大小 次。**批次大小是一個取捨旋鈕**：太小退化成逐列、太大造出超大鎖，甜蜜點要靠壓測找（這正是 ch05 負載測試要做的事）。

## 把四個手段收束成一句話

到這裡，你那四個手段其實是同一個原理的四個切面：

```
                    ┌─────────────────────────────────────┐
                    │  核心：縮短「持有鎖的時間」          │
                    │  ＝縮短「別人的等待時間」            │
                    │  ＝消滅「等待後的驚群 + 超時重試」    │
                    └──────────────┬──────────────────────┘
                                   │
        ┌──────────────┬───────────┼───────────┬──────────────┐
        ▼              ▼           ▼           ▼              ▼
  重排 SQL 順序   拆解大交易   跨交易重排   批次化逐列更新   （共同效果）
  統一鎖獲取序   把慢步驟移   熱點行最晚   N 次往返→1 次   等待↓ 重試↓
  →根除死結     出交易外     拿、最早放   鎖持有崩塌      →CPU 尖峰 -40%
```

那 40% 不是你讓 CPU 算得更快換來的——是你**讓 CPU 不再為了鎖等待後的驚群與重試而空轉**換來的。這也是為什麼當年如果你選擇「換更大的 RDS 機型」，效果會差很多：更大的機型給你更多算力，但**鎖競爭是個串行化瓶頸，加 CPU 核心並不能讓兩個交易同時握同一把行鎖**。被放棄的選項（scale up）輸在它解的是錯的問題——它以為瓶頸在算力，真瓶頸在並發協調。

## 故障模式與防禦

| # | 故障 | 壞了長什麼樣（可觀測徵兆） | 怎麼防 |
|---|---|---|---|
| **1** | **長交易凍結熱點行** | RDS CPU 貼天花板平頂；Performance Insights 上 `synch/cond/innodb/row_lock_wait` 等待事件竄高（2026-06 查證）；`SHOW ENGINE INNODB STATUS` 的 `TRANSACTIONS` 段看到一堆 `LOCK WAIT`；應用層大量 `Lock wait timeout exceeded; try restarting transaction`（error 1205） | 把外部呼叫/重計算移出交易；交易內只剩「鎖→寫→commit」；監控「交易平均持有時間」這個被你忽略的指標 |
| **2** | **死結風暴（互等順序）** | 應用層出現 error 1213 `Deadlock found when trying to get lock`；`SHOW ENGINE INNODB STATUS` 的 `LATEST DETECTED DEADLOCK` 段反覆刷新；CPU 因回滾+重試抖動 | **統一鎖獲取順序**（一律按 primary key 升序更新）根除環；不能統一時靠重試（但重試要冪等，見 ch10） |
| **3** | **gap lock 卡住 INSERT（最隱形）** | `INSERT` 莫名其妙超時，但你盯行鎖怎麼看都沒有跟現存行衝突；衝突發生在「空隙」上 | 縮短範圍掃描交易的持有時間；確認真需要 RR 防幻讀，否則該連線改 Read Committed（減少 gap lock）；機制細節（next-key、insert intention lock）見上節與 ch11 |
| **4** | **拆交易後中間態永久落地**（綁定你的真實系統） | 玩家「收到 coins 沒收到道具」「庫存扣了訂單沒建」這類不一致出現在客服回報，而 DB 裡查不到任何錯誤——因為每個小交易自己都成功了 | 只拆「能拆」的（無守恆關係 / 中間態可補償）；拆掉原子性後必須補上補償/對帳邏輯（ch04 的最終一致機制）；不能拆的（轉帳、扣庫存建單）保持單一交易 |
| **5** | **批次太大造出超大鎖 / 整批回滾** | 一個巨型 `UPDATE` 跑很久、期間整張表的並發全部卡死；中途超時或撞死結，**一百萬行的工作全部白做**；read replica 複製延遲突然飆高 | 改分塊批次（每批 500～1000 行各自 commit）；批次大小靠壓測找甜蜜點（ch05）；巨型維護操作排在離峰並切塊 |

這五個故障的共同訊號是「等待」——學會在 Performance Insights 上把「CPU 是被算術燒的」和「CPU 是被鎖等待後的重試燒的」分開（前者看 CPU on-CPU 時間、後者看 `row_lock_wait` 類 wait events），是你當年其實做到了、但沒能用語言講清楚的那個直覺。這一軸（觀測）是全書五個故障問句的最後一問，ch15 會把它挖穿。

## 五軸體檢：NeoBards 的 RDS 寫入路徑

按全書透鏡（五個故障問句，ch02 定義）給這個系統做一次體檢。這是 Part II 每章結尾的固定動作，附錄 B 會把所有系統並排。

| 軸 | 這個系統的答案 | 代價／何時會咬你 |
|---|---|---|
| **交付** | 不適用（這是 DB 寫入路徑、不是訊息系統）；但「應用層收到 1205/1213 後重試」就是一條隱性的交付問題——重試必須冪等 | 重試若不冪等，超時/死結回滾後的重試會重複發獎（接 ch04 結算冪等、ch10） |
| **持久** | 交易 commit 後落地（InnoDB redo log + fsync）；**未 commit 的工作（含拆交易後的中間態）行程掛了就消失** | 拆交易後若進程在兩個小交易之間死掉，中間態永久落地——這不是「丟失」是「不一致」 |
| **分區／一致** | 單庫內由交易+鎖保證；**一旦拆交易就放棄了原子性，跨步驟一致性外包給應用層補償**；read replica 非同步複製有延遲 | 拆掉的原子性沒補回（補償/對帳）→ 資料不一致；讀 replica 讀到舊值（ch11 接棒） |
| **並發** | InnoDB 行鎖 + 意向鎖 + RR 的 next-key/gap lock；**這是本章主場**——持有時間=等待時間，死結靠統一順序預防、靠 wait-for graph 偵測 | 長交易凍熱點行、相反順序死結、gap lock 卡 INSERT、批次太大造超大鎖——本章五個故障全在這軸 |
| **觀測** | CloudWatch CPU（你當年看的）+ Performance Insights 的 `row_lock_wait` wait event + `SHOW ENGINE INNODB STATUS` + `INNODB_LOCK_WAITS` | 只看 CPU% 會誤判成算力問題；要看 wait events 才看得出「CPU 被等待+重試燒掉」（ch15 挖穿） |

一句話總結：**這個系統最吃的是「並發」軸**——它的招牌數字（CPU -40%）本質是一次並發協調的優化，其他四軸都是被它牽動的配角。

## 紙上推演

### 推演一：標出鎖競爭點並重排降衝突 **[15 分鐘] ★★**

給你兩個並發交易，各要更新同一對玩家行（A 與 B 是兩個玩家的 primary key，A < B）：

```
T1: BEGIN; UPDATE player WHERE id=A; （計算 80ms）; UPDATE player WHERE id=B; COMMIT
T2: BEGIN; UPDATE player WHERE id=B; （計算 80ms）; UPDATE player WHERE id=A; COMMIT
```

問：(a) 畫出最壞情況下的等待環，說明死結怎麼形成；(b) 只改 SQL 順序、不改邏輯，怎麼讓死結不可能發生；(c) 改完之後，最壞情況下後到的交易要等多久？

#### 推演解答

**(a) 死結怎麼形成。** 時序展開：

```
時刻   T1                          T2
 t0    鎖到 A（X lock）
 t1                                鎖到 B（X lock）
 t2    （計算 80ms）               （計算 80ms）
 t3    要鎖 B → B 被 T2 握著，等
 t4                                要鎖 A → A 被 T1 握著，等
        ───────── 互等，環形成 ─────────
        T1 等 B（T2 握）、T2 等 A（T1 握）→ wait-for graph 出現環
        InnoDB 偵測到環，挑改動較少者回滾（假設 T2），回傳 error 1213
```

關鍵在 t0/t1：兩個交易**以相反順序**獲取鎖（T1 先 A、T2 先 B），於是各自握著對方下一步要的那把鎖。

**(b) 怎麼根除。** 讓兩個交易都**按 id 升序**獲取鎖（A < B，所以都「先 A 再 B」）：

```
T2 改成: BEGIN; UPDATE player WHERE id=A; （計算 80ms）; UPDATE player WHERE id=B; COMMIT
```

現在不可能出現環：誰先拿到 A，誰就能繼續往 B 走（B 此刻沒人能搶先，因為搶 B 之前一定要先過 A 這關）；後到的那個在 A 上就被擋住，乖乖排隊，不會「已經握著 B 反過來要 A」。**統一獲取順序把「環」變成「鏈」。**

**(c) 後到的要等多久。** 改完後是純排隊，沒有死結。最壞情況：T2 在 t0 之後一瞬間到，想鎖 A，但 A 被 T1 握著。T1 要握到 commit 才放 A——也就是 T1 整個交易時長（鎖 A + 計算 80ms + 鎖 B + commit）。所以 T2 大約要等 **80ms 多一點**。這也順帶說明為什麼「把那 80ms 計算移出交易」這麼重要：如果計算不在交易內，T1 持有 A 的時間從 80ms+ 崩到幾毫秒，T2 幾乎不用等。**死結預防（順序）和持有時間（拆交易）是兩個獨立的旋鈕，要一起轉。**

常見錯路：以為「InnoDB 會自動偵測死結，所以不用管順序」。會偵測，但代價是回滾 + 重試的 CPU，高並發下這個代價會累積成你那條 CPU 平頂線的一部分。自動偵測是安全網，不是免死金牌。

### 推演二：判斷「拆這個交易安全嗎」 **[10 分鐘] ★★**

下面三個交易，逐一判斷「能不能拆成兩個各自 commit 的小交易」，並指出拆了之後哪個中間態會出事：

```
交易甲: BEGIN; UPDATE player SET last_login=now(); UPDATE player SET online=true; COMMIT
交易乙: BEGIN; UPDATE wallet SET coins=coins-100 WHERE id=A; UPDATE wallet SET coins=coins+100 WHERE id=B; COMMIT
交易丙: BEGIN; UPDATE inventory SET stock=stock-1 WHERE item=X; INSERT INTO orders(item) VALUES(X); COMMIT
```

#### 推演解答

**甲：能拆。** `last_login` 和 `online` 之間沒有守恆關係，中間態（更新了登入時間、還沒更新在線狀態）在語意上完全合法、不違反任何必須恆真的約束。拆開唯一的「代價」是兩個欄位的更新不再瞬時一致，但沒有人會因此少拿東西。**判準命中「無守恆關係」→ 能拆。**

**乙：不能拆。** 這是經典轉帳。中間態是「A 扣了 100、B 還沒加 100」——此刻系統裡**憑空少了 100 coins**，破壞了「總額守恆」這個必須恆真的約束。若第二步失敗（連線斷、超時、進程被 kill），這個不一致永久落地，玩家的錢真的消失。**判準命中「守恆/借貸關係」→ 不能拆**，必須留在單一交易（或用 saga + 補償，但那是另一套複雜度，不是隨手拆）。

**丙：不能拆（或拆了必須補償）。** 「扣庫存 + 建訂單」之間有借貸關係：庫存扣了就代表「這一件貨已許諾給某張訂單」。中間態是「庫存扣了、訂單沒建」——這件貨憑空蒸發（扣了但沒對應訂單）。若 `INSERT orders` 失敗，庫存白扣、可能超賣的反面（賣少了）。**判準命中「守恆關係」→ 不能裸拆**；若一定要拆（比如建訂單很慢想移出交易），必須補上「訂單建失敗就把庫存補回」的補償邏輯（ch04 的故事）。

常見錯路：把「拆交易降鎖競爭」當成一個無腦能套用的優化。它只對「能拆的」有效；對乙、丙這種，拆下去不是優化，是製造資料損毀。**你當年沒爆，是因為你（也許憑直覺）只拆了甲這類。**

### 推演三：估「把交易持有時間砍半，CPU 大概怎麼動」 **[20 分鐘] ★★★**

當年尖峰時，假設熱點公會行上有大量交易排隊。簡化模型：每個交易在熱點行上的「持有時間」是 T，到達率是 λ（每秒多少個交易想動這行）。你做的優化把每個交易的持有時間從 T 砍到 T/2（拆交易 + 批次化的綜合效果）。問：CPU 尖峰為什麼會明顯下降？用「等待 → 重試」這條鏈定性推（不要求精確數學，數學在 queue 書）。

#### 推演解答

定性推理鏈：

1. **持有時間 T = 別人的等待時間。** 熱點行一次只能被一個交易握。在它被握的 T 時間內，所有想動它的後到交易全排隊。把 T 砍半，每個排隊者的等待期望直接減半。

2. **等待時間決定 backlog 深度。** 排隊越久，同一時刻「正在等這把鎖」的交易數越多（這就是 queue 書講的：服務時間下降→隊列長度下降，ρ 往下走）。backlog 從 N 個降到約 N/2。

3. **backlog 降低 → 驚群重啟的規模降低。** 鎖一放開，原本 N 個積壓交易同時湧上來搶、同時重新 parse + 規劃 + 掃描索引——這一波同時重啟是 CPU 尖峰的主力。backlog 砍半，這一波驚群的瞬間 CPU 衝高也跟著砍。

4. **等待時間降低 → 撞 `innodb_lock_wait_timeout`（50s）的比例降低 → 超時重試降低。** 原本等太久被超時回滾、應用層重試的那些交易（每次重試 = 一輪完整 CPU），現在因為等待變短、來得及拿到鎖，重試比例下降。**這是最直接的 CPU 節省**：每少一次超時重試，就少一輪完整的 parse + 掃描 + 鎖嘗試的 CPU 浪費。

5. **正回饋被打斷。** 高並發下這幾條會互相放大成 metastable failure（過載態自我維持，數學與雙穩態見 queue 書 ch20）：等待久→重試多→負載更高→等待更久。把 T 砍半，等於把系統從「正回饋的過載態」推回「穩定態」，CPU 不是線性降 50%，而是可能從「貼天花板的平頂」斷崖式掉回「有餘裕的曲線」——這就是為什麼你看到的是「尖峰 CPU -40%」這種非線性的大幅改善，而不是「快一點點」。

常見錯路：以為「持有時間砍半 → CPU 砍半」是線性的。不是。在過載態附近，系統對持有時間的變化是**高度非線性**的——因為等待、重試、驚群是互相放大的正回饋。砍掉源頭（持有時間）可能讓你跨過某個臨界點，從過載態整個塌回穩定態，改善遠超線性預期。這也是為什麼你的 40% 看起來「太好了」——因為它不是省了 40% 的算術，是關掉了一個自我放大的浪費迴圈。

## 自我檢核

口頭自答；講不清楚＝還沒真懂。

1. 為什麼說 NeoBards 那條貼天花板的 RDS CPU 線「主要不是算力問題」？把「等待 → 重試 → 驚群」這條因果鏈完整講一遍。
2. InnoDB 的行鎖什麼時候釋放？如果你回答「`UPDATE` 執行完」，你錯在哪、正確答案怎麼改變了「持有時間」這個概念？
3. 「同樣的 query、不同的順序」差很多——講出**兩個**不同的機制（死結預防 / 熱點行等待結構），它們解的是不一樣的問題。
4. 在 InnoDB 預設隔離級別下，一個只在跑範圍 `SELECT … FOR UPDATE` 的交易，為什麼能卡住一個碰都沒碰同一行的 `INSERT`？next-key lock 和 gap lock 各扮演什麼角色？
5. 給你一個交易，你怎麼一眼判斷「它能不能拆」？講出那條判準，並各舉一個能拆/不能拆的例子。
6. 拆交易「不是免費把鎖競爭變沒」——那個成本被轉嫁到哪裡去了？如果你拆了卻沒補上那一塊，會發生什麼？
7. 批次化的「雙重收益」是哪兩個？哪一個才是降 CPU 的主力、為什麼？批次太大會怎麼反咬你？
8. 當年若改成「換更大的 RDS 機型」而不是這四個手段，為什麼效果會差很多？scale up 解的是什麼問題、真瓶頸又是什麼？
