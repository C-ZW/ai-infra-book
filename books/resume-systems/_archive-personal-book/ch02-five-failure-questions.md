# ch02 — 五個故障問句：全書的透鏡

> **本章解決什麼問題**：ch01 把你的履歷攤成一張滿是名詞的地圖，並丟下一個尷尬的事實——你寫得出名詞，但你大概沒挖過它底下。這一章給你**一把固定的尺**。從現在起，每碰到一個你建過的系統，都用同一組五個問句去量它：交付、持久、分區、並發、觀測。這五個問句不是我發明的，是分散式系統四十年來反覆撞上的同一批老問題；你當年在 LetsTalk 和 NeoBards 都**隱性地**回答過它們，只是沒把答案攤開講。本章把這把尺定義清楚、示範怎麼用（拿 LetsTalk 即時通知和結算 pipeline 各走一遍），並先說清楚一件最重要的事：這五個問句**沒有正確答案，只有你場景下的取捨**。Part II 起的每個系統章結尾都會用這把尺做一次「五軸體檢」；附錄 B 是所有系統的體檢總表。

## 從你已知的出發

你應該還記得，2022 年你在 NeoBards 接手那條每季要發 10–30 萬筆獎勵的結算 pipeline 時，腦子裡其實沒有一張「我該擔心什麼」的清單。你是邊做邊撞——先發現 SQS 的訊息偶爾會被消費兩次（玩家收到兩份獎勵的客訴進來了），於是補了冪等鍵；後來又發現 CronJob 某次重疊觸發，兩個 job 同時在跑同一批；再後來四個服務（guild / player / inventory / mail）的寫入不總是同步，玩家公告欄說發了獎、背包卻還沒到。

每一個問題你最後都解掉了。但你解的方式是**反應式**的——壞了才補。你沒有一個「我在動手前就該逐項問過」的框架。

LetsTalk 那邊也一樣。你自建 WebSocket + Redis Pub/Sub 那套即時通知時，你知道 Pub/Sub「不太可靠」，所以離線訊息另走一條路；你知道 3,200 條長連線要 sticky 到固定機器；你知道某台 WS server 掛了那批連線要重連。這些你**都做對了**——但如果當年面試官問你「你這套系統在訊息交付上提供什麼保證」，你大概答不出一個乾淨的句子。你做對了，但你說不清楚為什麼對。

**但你可能沒想過的是**：你在這兩個系統上反覆撞的，其實是**同一批問題**，只是換了張臉。結算 pipeline 的「訊息被消費兩次」和通知系統的「Pub/Sub 訊息會掉」，是同一個軸（交付）的兩端；CronJob 的「重疊觸發」和兩個 consumer「搶同一筆」，是同一個軸（並發）。如果你當年手上有一張固定的問句清單，你不會邊做邊撞——你會在白板上花十分鐘把五個軸問一遍，然後**知道你要防什麼、放棄什麼**。

這一章就是要把那張清單給你。它有五格。

## 為什麼是這五個，而且只有這五個

先說清楚一件事，免得你把這當成又一套「最佳實踐口號」：這五個問句不是隨便湊的，它們對應分散式系統裡**四個你逃不掉的物理現實**，加上**一個你逃不掉的工程現實**。

物理現實是：

1. **網路會丟封包、會延遲、會亂序。** 你送出去的東西，不保證到、不保證只到一次、不保證按順序到。→ 這就是**交付**。
2. **行程會死、機器會重啟。** 活在記憶體裡的東西，重啟那一刻就沒了。→ 這就是**持久**。
3. **節點之間會斷線（網路分區），而且資料的副本之間需要時間才會一致。** 斷線或複製延遲期間，不同的人看到不同的真相。→ 這就是**分區／一致性**。
4. **多件事會同時發生。** 兩個請求、兩個 consumer、兩個 job 撞在同一筆資料上。→ 這就是**並發**。

這四個是「世界會怎麼背叛你」。第五個不一樣，它是「你怎麼知道世界背叛了你」：

5. **系統一定會壞，問題只是你多久才發現。** 觀測不是壞掉之後才補的東西，它決定了前四軸的故障是「五分鐘內告警」還是「玩家客訴三天後才浮上來」。→ 這就是**觀測**。

為什麼只有五個、不是十個？因為這五個是**正交**的——它們各自問一件其他四個問不到的事，而且任何一個分散式系統的故障，最後都能歸到這五格的某一格（或某幾格的組合）。你後面會發現，附錄 B 那張大表能容下你所有系統，靠的就是這五格夠寬又不重疊。

下面逐軸拆。每軸我給你四件事：**它在問什麼**、**為什麼這是個繞不開的老問題**、**你哪個系統最吃這一軸**、**答錯會出什麼事故**。

## 軸一 — 交付（delivery）：會掉嗎？會重嗎？順序到哪？

**它在問什麼。** 你把一個訊息／事件／請求送出去，三個子問題：它會**掉**嗎（at-most-once：可能丟，但絕不重複）？它會**重**嗎（at-least-once：絕不丟，但可能重複）？它的**順序**保證到哪（完全亂序／同一 key 內有序／全域有序）？

注意這裡有個你大概一直記反的事實：**「不掉」和「不重」你只能挑一個，不能都要**。原因很物理——送方送出訊息後，要嘛等收方回 ack，要嘛不等。等 ack：如果 ack 沒回來（網路丟了 ack），送方分不清是「訊息沒到」還是「訊息到了但 ack 丟了」，它只能**重送**（→ 可能重複，at-least-once）或**放棄**（→ 可能丟，at-most-once）。沒有第三條路。所謂的 exactly-once *交付* 是幻覺，你能買到的最好東西是「at-least-once 交付 + 收方冪等去重」拼出來的 exactly-once *處理*——這是 ch10 的主菜，這裡你先記住「交付保證三選二不可能」這個形狀。

**為什麼是繞不開的老問題。** 因為網路不可靠是分散式系統的定義性前提。只要你的訊息要跨越一條會丟封包的線（兩台機器之間、行程與佇列之間、瀏覽器與伺服器之間），交付問題就在那裡，你只是選擇了某個答案——很多時候是**預設幫你選的**，而你沒注意。

**你哪個系統最吃這一軸。** 結算 pipeline（ch04）。SQS Standard queue 是 **at-least-once、可能重複、best-effort 排序**（2026-06，AWS 官方明說標準佇列「在極少數情況下你可能收到一則已刪除訊息第二次」）。你那條 pipeline 的每一筆獎勵都跨越了 SQS 這條線，所以「會重」是**規格保證會發生的事**，不是 bug。你當年補的冪等鍵，本質上就是在交付這一軸上選了「at-least-once，靠冪等吃掉重複」。

對照另一端：LetsTalk 的 Redis Pub/Sub（ch07）是 **at-most-once、fire-and-forget、無持久化**（2026-06，官方：訊息只送給「發佈當下正連線」的訂閱者，離線者永久錯過、無 replay）。所以即時通知「會掉」也是規格保證的事——這就是為什麼你的離線訊息得另走一條路（拉取／DB），Pub/Sub 那條線**結構上做不到不掉**。

**答錯會出什麼事故。** 把 at-least-once 的通道當成「保證只送一次」用 → 玩家收到兩份獎勵、信用卡被扣兩次。把 at-most-once 的通道當成「保證送達」用 → 重要通知靜悄悄消失，沒人收到、沒人報錯（這是最陰的一種，因為它**不留痕跡**）。

## 軸二 — 持久（durability）：重啟會丟掉進行中的工作嗎？

**它在問什麼。** 此刻系統裡「正在進行但還沒完成」的工作——in-flight 的訊息、排程到一半的 job、還沒 flush 的 buffer、setTimeout 還沒觸發的計時器——如果這一秒行程被 kill、機器被換掉、佇列被清空，**這些工作會原地蒸發嗎**？

關鍵分界是：這個狀態活在**揮發性記憶體**裡，還是被**主動寫到了外部持久儲存**？記憶體裡的東西，重啟即失憶，沒有例外。持久不是「預設就有」的東西，它是你**主動付代價**（寫盤、fsync、外部化狀態）換來的。

**為什麼是繞不開的老問題。** 因為行程一定會重啟——部署、OOM、節點驅逐、機器故障。在雲端與 K8s 上，「行程被無預警換掉」不是異常，是**日常**（pod 隨時可能被 reschedule）。所以任何把重要狀態放在記憶體、又假設行程會一直活著的設計，都是在跟必然發生的事下注，而且你會輸。

**你哪個系統最吃這一軸。** 排程訊息（ch09）。你那個 setTimeout 版本是持久性的**反面教材**——計時器活在單一 Node 行程的記憶體裡，**重啟即失憶**。一個排在三天後發送的訊息，只要這三天內發生一次部署重啟，它就**靜悄悄消失**。這就是為什麼你當年只敢「限 15 分鐘延遲」：你不是技術上做不到更長，你是在跟「行程不可能 15 分鐘以上不重啟」這個現實妥協。你後來把它重構成 cron 驅動的 job queue，本質上就是把「何時該做什麼」這個狀態**外部化到持久儲存**（DB／Redis），於是重啟後它還在、任意時長、可重試。你那次重構，解的就是持久軸。

**答錯會出什麼事故。** 部署一次，一批排程訊息消失、一批 in-flight 的結算任務蒸發，而且——又是最陰的——**沒有錯誤日誌**，因為從程式的角度看，那些工作從來沒「失敗」過，它們只是隨著行程一起不存在了。你只會在很久以後從「為什麼這個玩家沒收到他該收的獎」這種客訴裡，反推出當時丟了東西。

## 軸三 — 分區／一致（partition / consistency）：誰在什麼時候看到什麼？

**它在問什麼。** 當資料有多份副本（read replica、跨服務各持一份、多台 WS server 各持一部分連線狀態），或當節點之間暫時斷線（網路分區）時：**在某一個時間點，不同的觀察者會不會看到不一致的真相**？剛寫進主庫的資料，去 replica 讀讀得到嗎？四個服務的狀態，玩家會不會看到「公告說發了獎、背包卻是空的」這種中間態？

**為什麼是繞不開的老問題——這裡得把 CAP 講準。** 這是全書最容易被講錯歸屬的一段，我特別小心地照查證版寫（landscape §23）：

- **CAP theorem 是 Eric Brewer 提出的**——他在 **2000 年 PODC keynote** 上以**猜想（conjecture）**形式提出。
- **形式化的證明是 Seth Gilbert 與 Nancy Lynch 在 2002 年做的**（論文〈Brewer's Conjecture and the Feasibility of Consistent, Available, Partition-Tolerant Web Services〉, ACM SIGACT News 2002）。**Brewer 提猜想、Gilbert-Lynch 給證明，這是兩組人、別混為一人。**
- CAP 的內容是：當發生**網路分區（P）**時，你只能在**一致（C）**與**可用（A）**之間二選一——分區期間，你要嘛拒絕服務以保證大家看到一致的值（犧牲 A），要嘛繼續服務但容許副本暫時不一致（犧牲 C）。

但 CAP 有個常被忽略的盲點：它只談「分區發生時」怎麼選，沒談「網路正常時」你還在付什麼代價。補上這塊的是 **PACELC，由 Daniel Abadi 提出**——**2010 年部落格初提、2012 年 IEEE Computer 論文**〈Consistency Tradeoffs in Modern Distributed Database System Design〉**形式化**。PACELC 說：**if Partition（分區時）→ 選 A 或 C；Else（正常運作時）→ 選 Latency（延遲）或 Consistency（一致）**。換句話說，就算網路一切正常，你想讓資料更一致（例如每次都讀主庫、或等所有副本同步完才回），就得付更多延遲；想更快，就得容忍讀到稍舊的值。這個取捨**永遠在那裡**，不是只有斷線時才出現。

**你哪個系統最吃這一軸。** 兩個都很吃，方式不同：

- 結算 pipeline 的**四服務最終一致**（ch04/ch11）：guild / player / inventory / mail 沒有分散式交易綁在一起，所以一定存在「公會發了獎（guild 已寫）、但背包還沒加（inventory 未寫）」的中間態。玩家在那個瞬間看到的是不一致的真相。你接受了這個——你選的是「最終一致」，代價是中間態可見，換來的是四個服務不必兩段提交鎖在一起。
- 通知系統的**多裝置同步**（ch07/ch11）：同一用戶手機＋平板兩端，「已讀」狀態哪一端先看到？這也是一致性問題。
- **read replica 的複製延遲**（你調過的 MySQL/PG read replica）：非同步複製下，「剛寫完主庫立刻去 replica 讀」可能讀不到自己剛寫的東西——**讀己之寫（read-your-writes）破裂**。這是 PACELC 的「Else→L or C」最日常的化身：你為了分攤讀流量把讀導去 replica（買了延遲與吞吐），代價就是一致性鬆了一格。

**答錯會出什麼事故。** 把最終一致的系統當強一致用 → 玩家截圖「公告發了獎、背包是空的」當作 bug 回報，你查了半天發現那只是兩個服務之間 200ms 的中間態。把 read replica 當主庫的即時鏡像用 → 使用者改了個人資料、馬上重整頁面，看到的還是舊資料，以為沒存成功，於是再存一次。

## 軸四 — 並發（concurrency）：兩件事同時來會打架嗎？

**它在問什麼。** 當兩件（或更多）事情**同時**發生——兩個請求改同一列、兩個 consumer 領到同一筆訊息、兩個 CronJob 重疊觸發跑同一批、兩個交易互相等對方的鎖——它們會**打架**嗎？會不會 race condition（誰先誰後決定結果，而你沒控制順序）？會不會 deadlock（A 等 B 的鎖、B 等 A 的鎖，互等到天荒地老）？

這一軸的核心是**共享資源 + 沒有強制的順序**。只要有兩個執行流碰同一份狀態，而你沒明確規定「誰先誰後」「誰擋誰」，結果就是不確定的，而不確定在生產環境約等於「偶爾出錯、很難重現」。

**為什麼是繞不開的老問題。** 因為你的系統**必須**並發才能撐住負載——3,200 條連線同時在線、上千 RPS 同時打進來、一批結算任務並行消費。並發是你要的效能，但效能的代價就是「同時性」帶來的打架可能。你不能靠「把所有東西排成一條線、一次只做一件」來迴避（那叫沒有並發，撐不住量）。所以你只能**管理**並發，不能消除它。

**你哪個系統最吃這一軸。** 三個層次：

- 你那次 **RDS CPU 降 40%**（ch03）骨子裡就是並發軸的勝利。你以為你在「優化 SQL」，其實你在做的是**降低鎖競爭**——拆解大交易（縮短持有鎖的時間 = 縮短別人等待的時間）、重排 SQL 執行順序（讓鎖獲取順序一致以避免互等的死結）、批次化逐列更新（N 次往返縮成 1 次，鎖範圍與往返雙重收益）。CPU 被推高，很大一部分不是純算力，是**鎖等待 + 衝突重試**堆出來的。這是並發軸最漂亮的一課：同樣的查詢、不同的順序，差很多。
- 結算 pipeline 的**兩個 consumer 搶同一筆**（ch04）：SQS visibility timeout 到期後，同一則訊息可能被第二個 consumer 領走，兩個 consumer 並發處理同一筆獎勵。這是並發軸與交付軸的交會點——也是為什麼冪等同時在防這兩軸。
- MySQL/InnoDB 預設的 **Repeatable Read + next-key lock（gap lock）**（ch03/ch11，landscape §15）：你可能沒意識到，在這個級別下「插入」也會被鎖卡住——next-key lock 不只鎖既有的列，還鎖列與列之間的**間隙**以擋住幻讀，於是兩個並發插入可能互卡。這是並發軸藏得最深的一個陷阱。

**答錯會出什麼事故。** Race：兩個請求同時把庫存從 1 減到 0，結果賣超（兩個人都買到最後一件）。Deadlock：兩個交易互等鎖，資料庫偵測到後犧牲其中一個回滾（你會在日誌看到 deadlock 然後某個操作莫名失敗）。鎖競爭失控：CPU 尖峰、所有請求一起變慢、看起來像「資料庫不夠力」，其實是大家在排隊等同一把鎖。

## 軸五 — 觀測（observability）：壞了你第一時間怎麼知道？

**它在問什麼。** 前面四軸都在問「系統會怎麼壞」。這一軸只問一件事，但它決定前四軸的故障有多痛：**當它壞了，你多久會知道，而且你知道的時候，手上有沒有足夠的線索定位到是哪一軸壞了？**

注意這一軸的特殊地位——它不是「系統的一個屬性」，它是**你能不能看見前四軸的能力**。一個交付軸出錯（訊息掉了）的系統，如果觀測做得好，五分鐘內告警、你立刻知道；如果觀測沒做，這個故障會以「三天後玩家客訴」的形式抵達你，而且你得花一天反推它是哪一軸。**同一個故障，觀測的好壞決定它是事件還是災難。**

**為什麼是繞不開的老問題。** 因為「系統會壞」是常態不是例外（前四軸保證了這點），所以唯一的問題是「壞了之後資訊以多快、多清晰的方式到你手上」。而且——這點很多人忽略——**觀測本身有成本**。你不能什麼都記、什麼都量。每多一個維度的 metric（per-user、per-endpoint）就多一批時間序列要存（基數爆炸，ch15）；log 記越細越貴、量大時還會掉。所以觀測也是個取捨軸：你在「看得清楚」和「成本可控」之間選。

**你哪個系統最吃這一軸。** 你的 **FluentBit + Lua 可觀測性 pipeline**（ch06）就是這一軸的親手實作——從 production log 萃取 API 回應時間 metric、輸出 S3 供發行商 Nexon 用。你當年做的正是「從 log 反推 metric」，這條路常見但有陷阱（log 掉了 metric 就錯、取樣會偏估 p99，ch06/ch15 收口）。

但這一軸最該被點名的是**結算 pipeline 上你當年觀測的空缺**：如果某次發錯獎（多發、漏發、發到錯的公會），你第一時間怎麼知道？如果答案是「等玩家客訴」，那這軸你當年其實沒認真答。發錯獎是交付軸／並發軸的故障，但**讓它從災難降級成事件的，是觀測軸**。

**答錯會出什麼事故。** 故障靜默累積——重複發獎跑了三天才被發現，已經發出去 5,000 筆、收不回來了；某台 WS server 的連線悄悄掉光、沒人注意，直到那批用戶集體回報「收不到通知」。觀測軸答錯的事故有個共同特徵：**它把前四軸任何一個小故障，放大成一個大事件**，因為時間站在故障那邊。

## 五軸彼此的張力：沒有正確答案，只有取捨

到這裡你可能想問：「那我五軸全都選最強的不就好了——不丟不重、絕對持久、強一致、零 race、全鏈觀測？」

不行。而且這正是這套透鏡最重要的一課：**五軸彼此拉扯，強化一軸常常要弱化另一軸**。給你三組最關鍵的張力：

**張力一（CAP/PACELC 的影子）：分區/一致 ↔ 可用/延遲。** 這是軸三內建的取捨，照 landscape §23 講準：分區發生時，一致與可用二選一（CAP，Brewer 提猜想／Gilbert-Lynch 證明）；網路正常時，一致與延遲二選一（PACELC，Abadi 提出）。你想讓四服務強一致（兩段提交、誰沒寫成功大家一起 rollback），代價是任一服務慢或掛全部都卡住（可用性塌）、而且每筆都更慢（延遲漲）。你當年選最終一致，就是用「中間態可見」這個一致性的弱化，換來四服務的可用與低延遲。**這是你做的取捨，不是你的疏忽。**

**張力二：交付保證越強 ↔ 越慢、越貴。** 軸一與效能的拉扯。at-most-once（Pub/Sub）最快——送了就忘，不等 ack、不存。at-least-once（SQS Standard）要存訊息、要 ack、要重投機制，慢一截但不丟。exactly-once *處理*（SQS FIFO 那種）還要維護去重窗、嚴格排序，於是有**吞吐上限**（2026-06，FIFO 預設每 partition 每秒 300 非批次／3,000 批次）。你即時通知選 Pub/Sub 的 at-most-once，不是因為你不懂更可靠的選項，是因為「即時」這個需求下，速度比「絕不掉一則在線通知」更重要——而你把真正不能掉的東西（離線訊息）放到另一條更慢更可靠的路。**對的系統，對的軸上選對的強度。**

**張力三：觀測看得越清 ↔ 成本越高。** 軸五與成本的拉扯。你想 per-user 追蹤每個人的延遲（看得最清），代價是 500K 用戶 × 每個維度 = 基數爆炸，metric 系統成本失控（ch15）。所以觀測永遠是「我願意為『早一點知道』付多少」的取捨，不是「越多越好」。

所以這把尺的用法**不是打分數**（沒有「五軸滿分的系統」這種東西），而是**逼你把每一軸的取捨講出來、講清楚被你放棄的那一端輸在哪**。一個你能在五軸上都說出「我選了這端，因為另一端在這個維度輸了」的系統，就是一個你真正懂的系統。這也正是 ch19 要把你每個系統講成面試級敘事時的骨架。

## 五軸體檢表：本書的招牌工具

從 Part II 起，每個系統章的結尾都會用下面這張表做一次「五軸體檢」。先把格式範本定下來，後面所有章共用同一個形狀（附錄 B 是把所有系統的表疊起來的總表）。每一格回答兩件事：**這個系統在這一軸的答案是什麼**、**這個答案的代價／它在什麼條件下會咬你**。

```
┌──────┬────────────────────────────┬────────────────────────────┐
│ 軸   │ 這個系統的答案             │ 代價／何時會咬你           │
├──────┼────────────────────────────┼────────────────────────────┤
│ 交付 │ at-most / at-least /       │ 選了這交付強度，放棄       │
│      │ exactly-once？順序？       │ 了什麼、何時掉或重         │
├──────┼────────────────────────────┼────────────────────────────┤
│ 持久 │ 進行中工作存哪？           │ 重啟那刻會丟什麼、         │
│      │ 重啟活得下來嗎？           │ 丟了有沒有痕跡             │
├──────┼────────────────────────────┼────────────────────────────┤
│ 分區 │ 一致性落在譜哪格？         │ 誰會在哪個瞬間看到         │
│      │ 斷線/延遲誰看到啥？        │ 不一致、讀己之寫破不破     │
├──────┼────────────────────────────┼────────────────────────────┤
│ 並發 │ 共享資源誰擋誰？           │ 同時來的兩件事在哪         │
│      │ 會 race／deadlock？        │ 打架、怎麼偵測與回滾       │
├──────┼────────────────────────────┼────────────────────────────┤
│ 觀測 │ 壞了第一時間怎麼知         │ 哪種故障會靜默、           │
│      │ 道？哪三個訊號告警？       │ 多久才會浮上來             │
└──────┴────────────────────────────┴────────────────────────────┘
```

填表三條規矩，後面每章都照辦：

1. **答案要具體到規格**，不要寫「應該還好」。寫「at-least-once（SQS Standard 規格保證會重複）」，不寫「大致可靠」。
2. **代價那欄一定要填**。一個沒有代價的答案，表示你還沒想清楚——五軸沒有免費的格子。
3. **空格也是答案**。如果某一軸你當年根本沒想過（很正常），就老實寫「未處理」並標出它何時會咬你。承認沒答，比假裝答過更有用。

## Worked example：對 LetsTalk 即時通知做一次完整五軸體檢

現在把這張表用滿一次。對象是你最熟的系統——LetsTalk 的即時通知（WebSocket + Redis Pub/Sub、~3,200 並發連線、~10M 訊息/月，補 FCM 在中國等市場不可用的缺口）。逐軸填，每格帶你當年真實的數字與決策。

```
┌──────┬──────────────────────────────────┬──────────────────────────────────┐
│ 軸   │ LetsTalk 即時通知的答案          │ 代價／何時會咬你                 │
├──────┼──────────────────────────────────┼──────────────────────────────────┤
│ 交付 │ 在線即時段：at-most-once         │ 訂閱者斷線那刻起，期間所         │
│      │ （Redis Pub/Sub，fire-and-       │ 有訊息永久消失、無 replay、      │
│      │ forget、無持久化）。離線         │ 不留痕跡。平板斷線那段的         │
│      │ 段：另走拉取／DB。               │ 通知，Pub/Sub 結構上救不回。     │
├──────┼──────────────────────────────────┼──────────────────────────────────┤
│ 持久 │ 連線狀態活在某台 WS              │ 那台 server 重啟／掛掉，         │

│ 分區 │ 多裝置（手機＋平板）狀態         │ 同一用戶兩端「已讀」會短         │
│      │ 最終一致；同步靠 Pub/Sub         │ 暫不同步；某端剛上線時可         │
│      │ 廣播＋各機過濾，非強一致。       │ 能短暫看到舊狀態，再被補上。     │
├──────┼──────────────────────────────────┼──────────────────────────────────┤

│      │ 機過濾出自己持有的連線。         │ 則同一用戶多連線散到多台。       │

│      │ 訊息吞吐量做粗略監控。           │ 知」不留任何錯誤——這軸最         │
│      │                                  │ 該補「在線訊息送達確認」         │
```

讀這張表，你會看到三件你當年**做對但沒說清楚**的事，現在能說清楚了：

- **交付軸你做了正確的「分流」**：把「即時但可掉」的在線通知放 Pub/Sub（at-most-once，要的是速度），把「不能掉」的離線訊息放另一條持久的路。你不是把整個系統建在不可靠的 Pub/Sub 上——你是**按需求把訊息分到不同交付強度的通道**。這是個漂亮的決策，但你當年大概沒用「at-most-once vs 持久通道」這組詞講過它。
- **持久軸的代價你接受了**：連線狀態放記憶體（重啟要全體重連），換來的是不必為每條連線做持久化的複雜度。對「即時通知」這種連線本來就會斷會重的場景，這個取捨對——但代價（重連風暴打 Redis）是真的，要靠軸五看著。
- **觀測軸是你當年的洞**：在線訊息掉了不留痕跡，這是 Pub/Sub at-most-once 在觀測軸上的直接後果。這格寫「未完整處理」是誠實的，而誠實地標出洞，比假裝五軸都答滿有用得多。

這就是一次完整體檢的樣子。注意它**沒有給分、沒有說這系統好不好**——它只是把五個取捨攤開，每個都連著一個你當年真實做過的決定。

## 另一遍：拿「結算一筆獎勵」走過五軸（破題版）

再快走一遍，這次不填整張表，而是讓「結算 pipeline 處理一筆公會獎勵」這件具體的事，依序撞上五個軸——你會看到五軸不是抽象分類，而是這一筆獎勵在旅途上會遇到的五道關卡。

一筆獎勵從 CronJob 觸發、拆成任務丟進 SQS、被冪等 consumer 領走、寫進 guild / player / inventory / mail。它沿路撞上：

- **交付**：SQS Standard 是 at-least-once——這一筆**可能被投兩次**（visibility timeout 到期、consumer 當機重啟）。會掉嗎？基本不會（at-least-once 保證不丟）。會重嗎？會，這是規格保證的。→ 所以你需要冪等鍵。
- **持久**：CronJob 觸發後、任務還沒全進 SQS 之前，如果這個觸發行程掛了，這批拆到一半的任務會怎樣？→ 這就是為什麼「把任務先落進 SQS」本身是一種持久化——進了 SQS 它就活在佇列裡（預設保留 4 天，2026-06），不在某個會死的行程記憶體裡。
- **分區/一致**：寫了 guild（公會公告「已發獎」）、還沒寫 inventory（背包還沒加）的那 200ms 裡，玩家看到的是不一致的真相。→ 你選了最終一致，接受這個中間態。
- **並發**：visibility timeout 到期後，第二個 consumer 領走同一筆，兩個 consumer 並發寫同一個玩家的背包。→ 冪等鍵在這裡也擋住了並發重複寫（交付軸與並發軸在這格交會，同一道防線擋兩軸）。
- **觀測**：如果這一筆因為某個 bug 發了兩份、或發到錯的公會，你第一時間怎麼知道？→ 這是你當年最該補強的一軸；發錯獎是交付/並發的故障，但讓它三天後才被發現的，是觀測的空缺。

走完你會發現一件事：**冪等鍵這一個設計，同時是交付軸（吃掉 at-least-once 的重複）和並發軸（擋掉兩個 consumer 的並發重複寫）的答案**。五軸不是互不相干的五張清單——它們在真實系統裡會交會，一道好的防線常常同時答好幾軸。這也是為什麼用五軸體檢系統時，你會反覆看到「這格的答案其實在解隔壁那格的問題」。

## 故障模式與防禦

這一章本身沒有一個「系統」會壞——它是工具。所以這一節講的是**用這把尺時你會犯的故障**，以及這把尺幫你抓到的、你當年真實系統上的故障。每一條都綁定你建過的東西。

| 故障 | 長什麼樣（徵兆） | 怎麼防 |
|---|---|---|
| **把交付強度記反**：以為 SQS Standard 「保證只送一次」、以為 Redis Pub/Sub 「保證送達」 | 結算偶爾雙發（記反了會重）；在線通知偶爾消失且無錯誤日誌（記反了會掉） | 對每條跨網路的通道，先查規格寫死的交付語意（landscape §1/§3），不靠記憶；標準佇列 = at-least-once、Pub/Sub = at-most-once，這是**規格保證**不是運氣 |
| **把記憶體狀態當持久**：setTimeout 排程、in-flight 任務、連線狀態放記憶體又假設行程不重啟 | 部署一次就丟一批排程訊息／in-flight 任務，**靜默無痕**，事後從客訴反推 | 持久軸體檢時，對每個「進行中的工作」問一句「這一秒 kill 行程它還在嗎」；要它在，就主動外部化到持久儲存（你 setTimeout→cron 那次幹的就是這個，ch09） |
| **把最終一致當強一致**：以為四服務的寫入是原子的、以為 read replica 是主庫即時鏡像 | 玩家截圖「公告發了獎、背包空的」當 bug 報；使用者改完資料重整看到舊值、以為沒存 | 分區軸體檢時明確標出「中間態可見窗口」有多長、讀己之寫破不破；該強一致的讀導回主庫，能容忍的才導 replica（ch11） |
| **把並發當不存在**：以為「同一筆訊息不會被處理兩次」「兩個請求不會撞同一列」 | 賣超（兩人買到最後一件）、deadlock 後操作莫名失敗、CPU 尖峰其實是鎖排隊 | 並發軸體檢時，對每個共享資源問「兩件事同時來誰擋誰」；用冪等鍵／唯一約束／鎖順序一致來管理（你 RDS -40% 那次就是這個，ch03） |
| **把觀測留到最後**：五軸只認真答前四個、觀測等壞了再補 | 任何前四軸的小故障都被放大成大事件，因為「三天後才發現」 | 觀測軸**和前四軸一起設計**，不是事後補；每個系統至少答「哪三個訊號能讓我第一時間知道它壞了」 |
| **用這把尺去打分數**：想找出「五軸滿分的系統」、或把某軸選了弱端當成「沒做好」 | 在不需要強一致的地方硬上兩段提交、在要速度的通道硬上 exactly-once，把系統做慢做貴 | 記住五軸是**取捨表不是計分卡**；每格只要能說出「我選這端，因為另一端在這維度輸了」就過關（這正是 ch19 面試敘事的骨架） |

誠實回答：對你建過的系統（結算 pipeline、即時通知、排程訊息任選），五軸裡哪一軸是你當年**最沒認真想過**的？寫出當時為什麼沒想（不是 bug，是注意力沒到那裡），以及如果重來你會怎麼補。

**推演 4 — 反駁「我五軸全選最強就好了」　[10 分鐘]　★★**
有人說：「取捨什麼取捨，我交付選 exactly-once、一致選強一致、並發全上鎖、觀測全鏈追蹤，五軸全拉滿不就沒事了。」用本章三組張力逐一反駁他——具體指出他每拉滿一軸，會在哪一軸或哪個維度（延遲、吞吐、成本、可用性）付出代價。

### 推演解答

**推演 1 解答。** cron/queue 版相對 setTimeout 版，**最劇烈改變的是持久軸**：舊版進行中的工作（待發訊息）活在單一行程記憶體、重啟即失憶（持久性零，只能限 15 分鐘賭行程不重啟）；新版把「何時發什麼」外部化到持久儲存（DB／Redis sorted set），重啟後還在、任意時長。代價那欄也變了——新版的代價是「重啟後 delayed job 可能**重跑一次**」（at-least-once，ch09/ch10），於是**交付軸**從「會掉（行程死了就沒了）」翻面成「會重（重啟後可能重投）」。這正是五軸彼此牽動的好例子：你修好持久軸（重啟存活），代價是交付軸冒出重複，於是你又需要冪等——一個軸的答案會把問題推到另一個軸。常見錯路：以為「重構成 queue 就五軸全好了」，忽略了它把故障從「丟失」換成了「重複」，而重複需要另一道防線。

**推演 2 解答。**
(a) **持久軸**為主。「重啟即失、靜默無痕」是持久軸故障的招牌特徵（記憶體狀態沒外部化）。它**不**主要是交付軸——交付軸談的是「跨通道的訊息會不會掉」，而這裡訊息根本還沒進任何通道、它隨行程一起蒸發。
(b) **交付軸**為主（at-least-once 的重複），但**與並發軸交會**——若是 SQS visibility timeout 到期後第二個 consumer 並發領走同一筆造成的雙發，那它同時是並發軸（兩個 consumer 搶同一筆）。一道冪等鍵同時答這兩軸。判斷要點：純粹「同一 consumer 因重投處理兩次」偏交付軸；「兩個 consumer 同時處理同一筆」就明確跨進並發軸。
(c) **分區/一致軸**。這是最終一致的中間態（guild 已寫、inventory 未寫的窗口），玩家在那幾百毫秒看到不一致真相。它**不是 bug**，是你選最終一致時就接受的代價——除非那個窗口長到不可接受，才需要縮短或改一致性模型。

**推演 3 解答。**（開放題，口頭過關制。）一個誠實的高品質回答通常會指向**觀測軸**——因為前四軸是「系統怎麼壞」，你做系統時自然會碰到（壞了你會去解）；但觀測是「你怎麼知道它壞了」，它不會主動逼你注意，要你**主動**設計。以結算 pipeline 為例：你當年大概把心力放在「讓它正確（冪等）」和「讓它一致（最終一致對帳）」，但「發錯獎我第一時間怎麼知道」這個觀測問題，很可能是等到出事才補的。如果你的答案是別的軸，也很好——重點是說得出「為什麼當時注意力沒到那裡」，那本身就是一次有價值的回顧。

**推演 4 解答。** 逐軸反駁：
- **交付選 exactly-once**：exactly-once *交付*根本不存在（丟一個 ack 就分不清沒送到還是 ack 丟了，ch10）；你能買到的 exactly-once *處理*要維護去重窗＋嚴格排序，代價是**吞吐上限**（FIFO 每 partition 每秒 300/3,000，2026-06）和**延遲**。即時通知上這麼搞，速度先死。
- **一致選強一致**（四服務兩段提交）：代價是**可用性塌**（任一服務慢或掛全部卡住，CAP 分區時 C 與 A 二選一）＋**每筆都更慢**（PACELC 正常時 C 與 L 二選一）。你那條 10–30 萬筆的 pipeline 會慢到跑不完一季結算。
- **並發全上鎖**：粗暴地把所有東西序列化＝**沒有並發**，撐不住 10K CCU／上千 RPS；而且鎖越多越容易 deadlock、鎖等待把 CPU 推高（你 RDS 那次正是反過來——**減**鎖競爭才降 40%）。
- **觀測全鏈追蹤**：per-user／全鏈 trace 在 500K 用戶下**基數爆炸**，metric 成本失控（ch15）。
結論：他每拉滿一軸，都在另一軸或某個維度（延遲／吞吐／可用／成本）付出可量化的代價。五軸是取捨表，不是計分卡——這就是本章的中心句。

## 自我檢核

口頭自答，講得清楚才算過關（講不清楚 = 還沒懂）：

1. 不看書，把五個故障問句一口氣講出來，每個用一句話說「它在問什麼」。
2. 為什麼「不掉」和「不重」只能挑一個？用「丟一個 ack」這個情境講給自己聽——這跟 exactly-once 交付為什麼是幻覺是同一件事嗎？
3. CAP 是誰提出、誰證明的？PACELC 是誰提出、它比 CAP 多補了什麼？（小心別把 Brewer 的猜想和 Gilbert-Lynch 的證明說成一人。）
4. 持久軸的核心判準是什麼？對你的 setTimeout 排程版，這個判準給出什麼答案、為什麼那逼出了「限 15 分鐘」？
5. 你的冪等鍵同時答了哪兩軸？為什麼一道防線能擋兩軸？這說明五軸之間是什麼關係？
6. 觀測軸為什麼地位特殊（它和前四軸不是同一種東西）？為什麼說「同一個故障，觀測好壞決定它是事件還是災難」？
7. 講出五軸三組張力中的任兩組，每組說「強化哪一軸會弱化哪一軸（或哪個維度）」。
8. 為什麼這把尺是「取捨表不是計分卡」？對 LetsTalk 即時通知的交付軸，你當年選 at-most-once（Pub/Sub）——這是疏忽還是取捨？說出被放棄的那一端輸在哪。

## 延伸閱讀

- **Gilbert & Lynch, "Brewer's Conjecture and the Feasibility of Consistent, Available, Partition-Tolerant Web Services"（ACM SIGACT News, 2002）** — CAP 的形式化證明原文。讀它確認歸屬（Brewer 提猜想、這兩位給證明），並看「分區時 C 與 A 不可兼得」是怎麼被嚴格證出來的。https://dl.acm.org/citation.cfm?id=564601
- **Daniel Abadi, "Consistency Tradeoffs in Modern Distributed Database System Design"（IEEE Computer, 2012）** — PACELC 的形式化論文。讀「Else→Latency or Consistency」這半邊——它補上了 CAP 沒談的「網路正常時你還在付什麼代價」，正是本章張力一的根。https://www.cs.umd.edu/~abadi/papers/abadi-pacelc.pdf
- **Amazon SQS — Standard vs FIFO 與 visibility timeout（官方文件，2026-06）** — 直接讀「at-least-once、可能重複」的官方措辭，當作交付軸「規格保證不是運氣」的第一手依據。https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/sqs-visibility-timeout.html
- **Kubernetes CronJob（官方文件，2026-06）** — 讀它白紙黑字寫「非 exactly-once、job 應為 idempotent」那段，當作持久軸與並發軸在排程上交會的標準案例。https://kubernetes.io/docs/concepts/workloads/controllers/cron-jobs/
- **Redis Pub/Sub（官方文件，2026-06）** — 確認 at-most-once、無持久化、離線者永久錯過。本章 LetsTalk 體檢的交付軸全靠這個事實，讀一手版本別靠記憶。https://redis.io/docs/latest/develop/pubsub/
- **《等待的數學：排隊理論》（queue 書）** — 當你想知道「五軸的代價多大」要量化時（過載為什麼爆炸、尾延遲、ρ→1）去那裡。本書只給五軸的工程取捨，數學在那本。
