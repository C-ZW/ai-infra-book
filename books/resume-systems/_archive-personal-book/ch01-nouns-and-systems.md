## 從你已知的出發

把你的履歷攤在桌上，用紅筆圈出每一個技術名詞。你會圈出很多。

LetsTalk 那段：WebSocket、Redis Pub/Sub、訊息扇出、多裝置同步、FCM、Twilio、Agora、Jitsi、Socket.io、TURN、STUN、NAT 穿透、setTimeout、cron、job queue。NeoBards 那段：RDS、CPU、交易、鎖、批次化、K8s CronJob、SQS、冪等 consumer、最終一致、guild/player/inventory/mail 四服務、FluentBit、Lua、S3、Clinic.js、memory leak、event loop、k6、Vegeta。再加上你的共通技能清單：MySQL、PostgreSQL、isolation level、deadlock、index、read replica、partition、Redis、ECS、Lambda、SNS、ALB、API Gateway、KrakenD、idempotency、最終一致、backpressure、rate limiting。

圈完，數一數。三十幾個。每一個你都不是聽說過而已——你**建過**它、線上跑過它、半夜被它叫醒過。這是你六年資歷的全部證據，也是你準備拿去換下一份工作的全部籌碼。

現在我要對做過這些事的你，問一個不太舒服的問題。

你的 LetsTalk 即時通知系統，撐了 ~3,200 並發連線、~10M 訊息/月。你會說它「用 Redis Pub/Sub 做訊息扇出」。這句話你講得很順。**但你能不能講出：當一個用戶的平板剛好斷線三秒，那三秒內發給他的訊息去了哪裡？** 不是「應該還在吧」，而是機制上、那一串 bytes 在 Redis 裡的命運。如果你卡住了——你不是不會用 Redis Pub/Sub，你是沒挖過它底下那一層。

這就是這本書的全部工作。你已經有了**工作級熟練**：寫得出名詞、接得起來、線上能跑。本書要把它往下推到**機制級理解**：它實際上怎麼運作、它在什麼條件下會壞、你當年那個決定其實在權衡什麼、底層為什麼長這個形狀。讀完之後，對每個系統你都該能口頭講出四句話——它怎麼運作、它怎麼壞、我當年的取捨、底下為什麼是這個形狀。

我不會從零教你 WebSocket 是什麼、SQS 怎麼用、交易怎麼開——你**建過**這些東西，重講一遍是侮辱。我要做的是站在「你當年做的那個版本」上，往下挖你**沒挖過**的那一層。

但你可能沒想過的是：往下挖第一鏟，挖出來的常常不是「更深的真相」，而是「你一直記錯的東西」。我們先用三個鉤子，立刻打你三巴掌。

## 三個立刻打臉的鉤子

這三個不是冷知識。它們是你履歷上的核心名詞，而你對它們的印象，很可能是錯的或過時的。每一個都先寫「你大概記得的版本」，再寫「它實際上的版本」。落差就是這本書的全部價值。

### 鉤子一：Redis Pub/Sub 不是訊息佇列

**你大概記得的版本**：Redis Pub/Sub 是個輕量的訊息系統，發布者 `PUBLISH`、訂閱者 `SUBSCRIBE`，訊息就送過去了。你拿它做 LetsTalk 的即時通知扇出，跑得很好，所以它「能送訊息」。

**它實際上的版本**：Redis Pub/Sub 是 **fire-and-forget、at-most-once、完全無持久化**（2026-06，見 landscape §3）。訊息只送給「發布那一瞬間正連著線的訂閱者」。一個訂閱者只要斷線——網路抖一下、行程重啟、GC 卡了 200ms——它在那段時間內被 `PUBLISH` 的所有訊息，**直接、永久、消失**。不排隊、不緩衝、不重送、沒有任何痕跡。Redis 連「它漏了一則」都不知道，因為它從來沒打算記得。

把這句話再讀一遍，因為它顛覆了「佇列」這個詞給你的安全感：**訊息佇列的核心承諾是「我先幫你存著，等你來拿」；Redis Pub/Sub 的承諾是「你不在我就丟掉」。** 這兩個是相反的東西。你拿 Pub/Sub 做「即時通知」是對的——在線那一段它快又便宜。但如果你心裡以為它順便保證了「送達」，那 ~10M 訊息/月裡，每一則發給離線裝置的訊息，都在你的盲區裡靜靜地掉了。要持久、要補發、要「離線的人重連後收得到」，那是另一個東西（Redis Streams，at-least-once、`XACK`、Pending Entries List），是另一條路（見 ch07、ch17）。

你當年的系統其實是對的——你**另外**走了一條離線補發的路，沒真的指望 Pub/Sub 送達。但你當時知道你在繞過它的缺陷，還是只是恰好做對了？這個差別，就是工作級熟練和機制級理解的差別。

### 鉤子二：SQS 標準佇列會重複，而且會亂序

**你大概記得的版本**：SQS 是 AWS 的訊息佇列，「保證送達」。你拿它接 NeoBards 結算 pipeline 的 consumer，每季 10–30 萬筆獎勵跑過去，沒出大事，所以「SQS 很可靠」。

**它實際上的版本**：SQS 標準佇列是 **at-least-once delivery——它保證至少送到一次，這意味著它明確允許送到很多次**；而且排序只是 best-effort，**不保證順序**（2026-06，見 landscape §1）。AWS 官方文件白紙黑字寫著：「你必須把你的應用設計成冪等的（idempotent），讓它重複處理同一則訊息時不會出錯。」這不是免責聲明，這是規格。當儲存某則訊息副本的某台伺服器在你刪除訊息時剛好不可用，那份副本就沒被刪掉，你下次 receive 時就會**再拿到它一次**。

所以你那條每季 10–30 萬筆的 pipeline，如果 0.1% 的訊息因為 visibility timeout 到期或 consumer 重啟而被重投，那就是**幾百筆**獎勵有機會被處理兩次。沒被雙發，不是因為 SQS 沒重投，而是因為你**寫了冪等 consumer**。你當年寫冪等是對的——但你寫它的時候，是因為你算清楚了 at-least-once 的數學，還是因為「大家都說 SQS 要冪等」？（這條 pipeline 的完整命運見 ch04，交付語意的形式化見 ch10。）

順帶把一個常見的安慰也戳破：「那我用 SQS FIFO 不就 exactly-once 了？」FIFO 確實提供 exactly-once **processing** 和嚴格排序——但那個「exactly-once」是靠一個只有 **5 分鐘的去重窗口**撐起來的，而且有吞吐上限（見 landscape §1）。它不是物理上送一次，它是「送很多次、但 5 分鐘內幫你去重」。這個區別在 ch10 會挖穿——因為 exactly-once *交付*，在分散式系統裡，是一個幻覺。

### 鉤子三：你登出了，但你的 JWT 還活著

**你大概記得的版本**：你接過 JWT 登入，使用者登出就「失效」了。token 嘛，登出就沒用了。

**它實際上的版本**：JWT 是 **stateless / self-contained** 的——它的有效性只由「簽章對不對」和「過期了沒」決定，伺服器驗證時根本不查資料庫（2026-06，見 landscape §14）。所以當你「登出」一個用戶，你做的通常只是把 token 從前端清掉。但那個 token 本身**還在使用者手上、簽章還是對的、還沒過期**。任何人拿著那個還沒過期的 token，伺服器照樣放行——因為伺服器無從知道「這個人登出了」，它不存任何狀態。

這代表：你以為「登出」「封禁」是即時的，實際上一個被你封禁的帳號，手上的 access token 在過期前（典型 15–60 分鐘）**還能繼續用**。要真正即時撤銷，你得加一張黑名單（denylist）讓每次驗證都去查——但那一查，就把 JWT「不用查資料庫」的整個賣點還回去了。這個兩難（無狀態 vs 可撤銷）就是 JWT 的核心張力，挖穿它是 ch16 的事。

## 這本書怎麼讀：橋接 → 往下挖一層 → 五軸體檢

每一章都走同一條路，因為你的學習方式就吃這套——架構先行，然後鑽進去。

**第一步：橋接你的真實系統。** 每章開頭把主題錨在你**具體做過的那個系統**上：哪個服務、什麼數字、當年你怎麼做的。不泛泛而談「分散式系統的交付問題」，而是「你那條每季 10–30 萬筆的結算 pipeline，當年你為什麼一定要讓 consumer 冪等」。橋接段假設你有完整工作經驗，直接從你當年的版本切入。

**第二步：往下挖一層。** 從「我做了什麼」挖到「為什麼有效 / 為什麼會壞 / 底下到底怎麼運作」。機制優先於名詞——先把資料怎麼流、誰跟誰握手、失敗時哪一步斷拆開，再回頭對照你當年的用法。需要時用 ASCII 圖追資料流、時序、握手、狀態機。

**第三步：五軸體檢。** 全書有一個唯一的透鏡——五個故障問句，ch02 會把它們正式定義。這裡先給你看一眼它們在問什麼（**只是預告，正式定義留 ch02，這裡不展開**）：

- **交付**：訊息會掉嗎？會重嗎？順序保證到哪？
- **持久**：行程重啟、機器當掉，正在進行的工作會丟嗎？
- **分區**：斷線、複製延遲、跨服務之下，誰在什麼時候看到什麼？
- **並發**：兩件事同時發生會打架嗎？會 deadlock / race 嗎？
- **觀測**：它壞的時候，你第一時間怎麼知道？

這五個是分散式系統反覆出現的五個老問題。你當年對每個系統都**隱性地**回答過它們——選了 SQS 就是回答了「交付」、寫了冪等就是回答了「重複」、設了 timeout 就是回答了「失敗偵測」。本書把這些隱性答案攤開、逐軸體檢。Part II/III 每個系統章結尾做一次小體檢，Part IV 把這五個老問題本身逐一挖穿，附錄 B 是全系統 × 五軸的總表。

還有兩條把全書讀法收緊的規則，先講明：

- **這本書不寫可執行程式碼。** 它是純觀念深掘。紙上推演會請你手追一條訊息的命運、畫一個 deadlock 的等待環、判斷一個取捨的對錯——用腦和紙，不用 IDE。允許貼示意性的 pseudo-code 來追邏輯，但不要求可執行。
- **本書有明確的邊界，越界一律指路、不展開。** 排隊／過載的**數學**（為什麼 ρ→1 會爆炸、Little's Law）在《等待的數學：排隊理論》（queue），本書只講工程機制；LLM serving、GPU 在《從後端到 AI Infra》（aiib），ch20 只搭橋；複雜系統故障的跨領域通論在《正常意外》（fail）；並發正確性的形式化證明在《把系統寫成定理：TLA+》（tla）。本書綁定的是**你自己的系統**，不是通論。

## Worked example：把履歷上一句話逐字拆開

來做一次完整的示範。拿你履歷上的一句話——就一句——把每個名詞底下藏的老問題挖出來，並標明它在哪一章被處理。這就是全書的縮影：一句你寫得很順的話，底下是一整張需要被挖穿的系統地圖。

那句話是：

> **「自建即時通知系統（WebSocket + Redis Pub/Sub），訊息扇出與多裝置同步，~3,200 並發連線、~10M 訊息/月，補上 FCM 在中國等市場不可用的缺口。」**

你寫這句話花了三十秒。我們花一節把它拆開。

**「WebSocket」** — 你記得它是「全雙工長連線，從 HTTP upgrade 升上來」。對。但你想過沒有：它是 **stateful、long-lived** 的連線——連線狀態綁在某一台機器上（landscape §22）。這代表你那 ~3,200 並發連線**不可能**均勻飄在一群無狀態機器上，你需要 **sticky session（session affinity）**，外加一個跨節點的 backplane。RFC 6455 定義了 frame、opcode、ping/pong 心跳，但它**不規範**你怎麼水平擴展——那是你的部署架構問題。→ 機制細節 **ch07**。

**「Redis Pub/Sub」** — 見鉤子一。它是扇出層：多台 WS server 各持一部分連線，「發給用戶 B」的訊息靠 Pub/Sub 廣播到所有機器、各機自己過濾出「B 在不在我這」。它快，但它 **at-most-once、無持久化**。這一個名詞底下，藏著「交付」這整根軸。→ 扇出機制 **ch07**，Redis 內幕（單執行緒、為什麼一個 `KEYS *` 阻塞所有人、持久化 RDB vs AOF、Valkey 授權事）**ch17**，交付語意形式化 **ch10**。

**「訊息扇出」** — 一則訊息要到達它該到的所有裝置。聽起來簡單，但「扇出放大」是個真問題：一個 10K 成員的熱門群發一則訊息，在 Pub/Sub 層會被放大成多少封內部廣播？這牽涉「並發」軸（廣播風暴對 Redis 的壓力）。→ 扇出拓樸與放大估算 **ch07**。

**「多裝置同步」** — 同一個用戶手機＋平板兩端，已讀狀態、離線補發。Pub/Sub 只管「在線即時」那一段；離線那一段（平板斷線時的那幾則）必須走另一條路（拉取 / Streams / DB）。這底下藏著「分區／一致性」軸——兩端在不同時刻看到不同狀態，這是讀己之寫、最終一致的問題。→ 補發路徑 **ch07**，一致性與複製延遲 **ch11**。

**「~3,200 並發連線」** — 這個數字不只是規模炫耀，它是「持久」和「並發」軸的壓力來源。每條連線是一份狀態（記憶體、心跳計時器、event listener）。如果你「每次連線就加一個 listener、斷線忘了移除」，在 3,200 並發下這就是一條穩定增長的記憶體洩漏。連線全跑在 Node 的單顆 event loop 上——一個沒讓出的同步重運算，會讓這 3,200 個人**一起**變慢。→ event loop 與洩漏 **ch05**，event loop 相位深潛 **ch12**。

**「~10M 訊息/月」** — 約每秒 4 則的平均，但平均會騙人——尖峰才是真相（這就是為什麼要看 p99 不看平均）。而且回到鉤子一：這 10M 裡每一則發給離線裝置的，在 Pub/Sub 下都可能靜靜地掉。「壞了你第一時間怎麼知道」——你當年有沒有一個指標能告訴你「今天掉了多少則」？沒有的話，那部分損失你永遠看不見。→ 觀測 **ch15**，尾延遲為什麼看 p99（數學指 queue 書）。

**「補 FCM 在中國等市場不可用的缺口」** — 你自建這套，是因為 FCM 在中國推不動。但「自建」意味著你**主動放棄**了 FCM 免費附送的東西：系統級喚醒、省電的離線佇列、作業系統幫你管的連線。你用 ~3,200 條自己維護的長連線，換來了「在中國能用」+「完全掌控」，代價是「離線推送、電量、運維」全壓在自己身上。這是一個**取捨**——而本書對每個取捨的要求是：說得出**被放棄的選項輸在哪一維**。→ FCM 缺口的取捨還原 **ch07**。

數一數：一句三十秒的話，拆出了 WebSocket、Redis Pub/Sub、扇出、多裝置同步、3,200 並發、10M 訊息、FCM 缺口——七個名詞，每一個底下都對應到 ch05、ch07、ch10、ch11、ch12、ch15、ch17 裡的一個老問題。**這就是這本書。** 你的整份履歷，就是這種句子的集合；本書的工作，是陪你把每一句都這樣拆穿。

## 故障模式與防禦

ch01 不深掘具體系統，所以這一節談的是「讀這本書本身的故障模式」——你帶著舊記憶讀技術書時，會怎麼把自己讀錯。每一條都綁到一個你真實系統的例子。

| 故障模式 | 壞了長什麼樣（徵兆） | 怎麼防 |
|---|---|---|
| **憑記憶確認事實**：把 2019–2024 記得的版本／狀態／預設值當「現在」 | 你流暢地說出「Twilio 要關了所以選 Jitsi」「Bull 是現役」「Node LTS 是 18」——而這三句在 2026-06 全部過時。面試官或同事一查就破，比「不知道」更傷信用 | 對任何版本號／產品狀態／授權／預設值，反射性問「這是查證的還是記得的」。本書全部錨在 landscape，標了「2026-06」。重大斷言要兩個獨立來源 |
| **把「我用過」當「我懂底下」**：工作級熟練偽裝成機制級理解 | 你講得出「Redis Pub/Sub 做扇出」，但答不出「平板斷線那三秒的訊息去哪了」。能用、不能解釋它怎麼壞 | 每個名詞逼自己回答「它怎麼壞、我當年怎麼防的」。講不出故障模式＝還沒懂（這正是 ch02 五軸體檢的用途） |
| **把實作行為當規格保證**：把「我那次 SQS 沒重投」當成「SQS 保證不重投」 | 你的結算 pipeline 沒出事，於是你心裡把 SQS 標準佇列記成「可靠不重複」——直到某次重投把一筆獎勵雙發 | 分清「規格保證的」「常見實作的行為」「我的推測」。SQS 標準佇列的規格是 at-least-once＝**允許**重複；你沒遇到不等於它不會 |
| **把取捨記成結論**：只記得「我選了 Jitsi」，忘了當年在權衡什麼 | 面試被追問「為什麼不用 Twilio」，你只能說「便宜」，說不出 Twilio／Agora 各輸在哪一維、今天會不會改選 | 每個取捨還原成「被放棄的選項輸在哪一維（成本／延遲／運維／故障域／授權）」。決策框架優先於「我選了 X」 |

## 紙上推演

### 推演一：圈名詞、寫「我以為」（全書貫穿題）**[20 分鐘] ★**

打開你自己的履歷（或 `Chewei_Chen_Resume.md`）。從上面圈出**至少十個**技術名詞。挑其中三個你最有把握的，各寫**一句**「我以為它怎麼運作」——憑現在的記憶寫，不准查。寫完封存起來，全書讀完回到 ch20 時拿出來對照。

#### 推演解答

這題沒有「標準答案」，它的價值在於**事後對照**——所以解答給的是「怎麼寫才有對照價值」和「怎麼自評」。

寫「我以為」時，要寫**機制**不是寫**用途**。反例（沒對照價值）：「Redis Pub/Sub：我以為它是用來做即時通知的。」這只是用途，無所謂對錯。好的寫法（有對照價值）：「Redis Pub/Sub：我以為訂閱者短暫斷線時，Redis 會幫它把訊息排隊，重連後補送。」——這是一個**可被證偽的機制斷言**，而它恰好是錯的（鉤子一已經破了它）。

自評時用一個標準：**這句話能不能被一個事實打臉？** 能被打臉的句子才值得寫下來對照。三個名詞裡，至少有一個你會在讀完後發現「我記的是過時版」（那是 landscape 那張老兵記憶表的料），至少有一個你會發現「我記的是用途、根本沒碰機制」。這兩種發現本身，就是這本書要給你的東西。常見錯路：把三個都寫成「它是什麼」的定義——那樣全書讀完你會發現「我寫的都對啊」，因為你根本沒賭任何機制上去。賭下去，才有得對照。

### 推演二：哪些「我以為」是過時記憶？**[15 分鐘] ★★**

下面五句「我以為」，**不准查資料**，憑你的直覺判斷：哪幾句是「機制理解錯誤」（從來就不對），哪幾句是「過時記憶」（曾經對、現在錯了）？

1. 「SQS 標準佇列保證訊息不重複。」
2. 「我選 Jitsi 是因為 Twilio 的視訊產品要停運了。」
3. 「JWT 使用者一登出，token 就立刻失效。」
4. 「Bull 是 Node 生態裡做 job queue 的現役主流套件。」
5. 「Node.js 現在的 active LTS 是 18 或 20。」

#### 推演解答

分類如下：

- **機制理解錯誤（從來就不對）**：第 1 句、第 3 句。SQS 標準佇列**從設計上就是** at-least-once、明確允許重複（landscape §1）——它任何一個版本都沒保證過不重複，這不是「以前對現在錯」，是你一直記錯。第 3 句同理：JWT **本質上**就是 stateless，登出不會讓還沒過期的 token 失效（landscape §14）——這是 JWT 的設計，不是哪一版改的。

- **過時記憶（曾經對、現在錯了）**：第 2、4、5 句。第 2 句在 **2024-03 到 2024-10 之間是對的**——Twilio 那段時間確實宣布了停運，所以「因為要停運而選 Jitsi」當時站得住腳；但 2024-10-21 撤回後就過時了（landscape §6）。第 4 句曾經完全正確，Bull 當年就是主流；但它已進維護模式，現役是 BullMQ（landscape §11）。第 5 句也曾經對，只是時間往前走了——2026-06 的 active LTS 是 Node 24（landscape §10）。

關鍵領悟：這兩類的**處理方式完全不同**。機制錯誤要靠「重新理解它的設計」修（讀 ch04/ch10、ch16）；過時記憶要靠「定期重新查證」修——而且它特別陰險，因為它**曾經是對的**，你對它有過成功經驗，記得格外牢。一個從沒對過的事實，你還會懷疑；一個曾經對過的事實，你會直接拿來用。這就是為什麼老兵的過時記憶比新手的無知更危險——它穿著「我有經驗」的外衣。常見錯路：把第 2 句也歸成「機制錯誤」。它不是——它在某個時間窗內是事實，這恰恰是「老化事實」最棘手的地方：對錯取決於日期。

### 推演三：拆你自己的一句話**[25 分鐘] ★★**

照本章 worked example 的方法，從你履歷上**另外挑一句話**（建議挑 NeoBards 那句「事件驅動結算 pipeline：K8s CronJob → SQS → 冪等 consumer，每季 10–30 萬筆獎勵、跨 ~5,000 公會，四服務最終一致」）。把每個名詞逐一拆出，各寫一句「它底下藏著哪個老問題」，並猜它對應到哪一章。

#### 推演解答

一個合格的拆解大概長這樣（你的版本只要抓到「名詞 → 老問題 → 章」這個結構就算過）：

- **K8s CronJob** → 它**不保證 exactly-once**：可能漏跑、可能重疊（landscape §19，官方明說「某些情況一個 job 可能被建立兩次或一次都不建」）。所以「每季結算」這個動作本身就可能跑兩次或跳過——這是「持久／時間」軸。→ ch04（pipeline 全貌）、ch13（時間與失敗）。
- **SQS** → at-least-once、可能重複又亂序（鉤子二）。「交付」軸。→ ch04、ch10、ch18。
- **冪等 consumer** → 它存在的**唯一理由**就是擋住上面 CronJob 和 SQS 帶來的重複。冪等鍵怎麼設計、去重窗口多大、擋得掉哪些擋不掉哪些。→ ch04、ch10。
- **每季 10–30 萬筆、跨 ~5,000 公會** → 規模決定了「0.1% 重投＝幾百筆雙發」的數量級，也決定了部分失敗的善後成本。
- **四服務最終一致（guild/player/inventory/mail）** → 沒有分散式交易，怎麼保證「發了獎、玩家收到、背包加了、信箱寄了」最後一致——補償、重試、對帳。「分區／一致性」軸。寫了 player 沒寫 inventory 怎麼收尾？→ ch04、ch11。

領悟：這一句話裡，**冪等 consumer 不是一個獨立功能，它是 CronJob 和 SQS 兩個「至少一次／不保證一次」缺陷的下游補丁**。當年你寫它，其實是在為上游兩個原語的交付語意買單。看清這個因果鏈——「為什麼非冪等不可」——就是從工作級熟練爬到機制級理解的那一步。常見錯路：把每個名詞當成獨立的功能模組各表一句，卻沒看出它們在「交付」這根軸上是同一個問題的不同段落。本書的整個 Part IV 就是在做這件事：把散在各系統裡的同一個老問題，抽出來挖穿。

口頭回答，講得出來才算過關。講不清楚＝這本書你還沒開始讀。
