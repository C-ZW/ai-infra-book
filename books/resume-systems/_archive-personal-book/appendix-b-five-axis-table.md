# 附錄 B — 五故障問句體檢總表

> 這是本書的招牌附錄。一張表，讓你一眼看完「我履歷上所有系統，在同一把尺下的全貌」。

全書脊椎是**五個故障問句**（ch02 定義、每章體檢、ch20 收官）。對任何一個你建過的系統，問同一組五個問題：

- **交付（delivery）**：訊息／事件會掉嗎？會重嗎？順序保證到哪？
- **持久（durability）**：行程重啟、機器當掉、佇列清空，會丟掉正在進行的工作嗎？
- **分區（partition／consistency）**：斷線、複製延遲、跨服務之下，誰在什麼時候看到什麼？
- **並發（concurrency）**：兩件事同時來會打架嗎？誰跟誰搶資源、會不會 race／deadlock？
- **觀測（observability）**：它壞的時候，你第一時間怎麼知道？

這五個問句**沒有正確答案，只有你場景下的取捨**（ch02）。所以這張表不是計分卡——**沒有任何一個系統在五軸上全綠**。它要攤開的是：每個系統在每一軸上選了哪一端、放棄了哪一端、那個放棄何時會咬你。

本附錄只收**書中實際體檢過**的系統：NeoBards 四案（ch03–ch06）、LetsTalk 三案（ch07–ch09）、雲端原語並排（ch18）。每個系統的五軸結論直接取自該章章末的「五軸體檢」小表，與章內一致；想看推導與代價細節，回該章。

下面先給一張**速覽總表**（每格一句話，掃過去抓全貌），再給每個系統的**逐軸展開**（一句話容不下的代價與邊界放這裡）。

## 逐系統展開

速覽表塞不下的代價、邊界與「我以為 vs 實際」，這裡逐系統補。每段結論與該章章末小表一致。

### NeoBards：RDS 寫入路徑（ch03）

招牌數字「尖峰 RDS CPU 降 40%」**本質是一次並發協調的優化**，不是算力問題——這個系統**最吃「並發」軸**，其他四軸都是被它牽動的配角。

- **交付**：不適用（這是 DB 寫入路徑、不是訊息系統）；但「應用層收到 1205（lock wait timeout）／1213（deadlock）後重試」是一條隱性交付問題——**重試必須冪等**，否則超時／死結回滾後的重試會重複發獎（接 ch04／ch10）。
- **持久**：交易 commit 後落地（InnoDB redo log ＋ fsync）；**未 commit 的工作（含拆交易後的中間態）行程掛了就消失**——拆交易後若行程在兩個小交易之間死掉，留下的不是「丟失」是「**不一致**」。
- **分區／一致**：單庫內由交易＋鎖保證；**一旦拆交易就放棄了原子性，跨步驟一致性外包給應用層補償**；read replica 非同步複製有延遲（讀己之寫可能破裂，ch11）。
- **並發**：本章主場。InnoDB 行鎖＋意向鎖＋RR 的 next-key／gap lock；**持有時間＝等待時間**，死結靠統一鎖獲取順序預防、靠 wait-for graph 偵測。本章五個故障（長交易凍熱點行、相反順序死結、gap lock 卡 INSERT、批次太大造超大鎖）全在這軸。
- **觀測**：只看 CloudWatch CPU% 會誤判成算力問題；要看 Performance Insights 的 `row_lock_wait` wait event ＋ `SHOW ENGINE INNODB STATUS`，才看得出「CPU 被鎖等待＋重試燒掉」（ch15 挖穿）。

### NeoBards：結算 pipeline（ch04）

每季 10–30 萬筆、跨 ~5,000 公會、寫 guild／player／inventory／mail 四服務。**冪等不是保險，是這個架構能成立的前提**。

- **交付**：SQS 標準佇列 at-least-once、可能重複、best-effort 排序；靠 `settlement_key`（季＋公會＋玩家＋獎勵類型）冪等收斂成「效果上剛好一次」。**重複是結構性的、消不掉，只能讓它無害。**
- **持久**：訊息持久在 SQS（最大保留 14 天）；去重記錄與業務寫入持久在各服務 DB；consumer 重啟不丟工作（訊息還在佇列）。**但 CronJob 觸發本身不持久**——漏跑那一刻沒有任何東西替你記得「該跑沒跑」。
- **分區**：無分散式交易；四服務最終一致，靠**重試＋補償＋對帳**收斂；中間有可見的不一致窗口（guild 已寫、inventory 未寫的 200ms）。強一致要的分散式交易在此場景太貴。
- **並發**：多 consumer 並發處理同一筆是常態（visibility timeout 重投造成）；靠唯一約束的**原子去重**防雙發。`check-then-act` 在此必爆——正確性下放到資料庫的並發控制（ch03 鎖機制）。
- **觀測**：對帳是主動發現不一致的眼睛；DLQ 深度、結算 Job「該跑沒跑」、player／settlement 筆數對帳都要告警。最危險的「**靜默少拿**」無錯誤日誌、無客訴，**只有對帳抓得到**——觀測必須是設計的一部分。

### NeoBards：負載測試與 Node 剖析（ch05）

「一台 Node 機器是怎麼被一行程式碼拖死的」。這個系統**最吃「並發」軸**——而它的並發是單執行緒非搶占並發，強處與死穴是同一個設計。

- **交付**：不直接是交付問題；但 event loop 被卡住時，in-flight 的請求 callback 全堵在隊伍後面——「**沒掉，只是無限延遲**」，對上游（負載均衡器、客戶端逾時）等同於丟失。
- **持久**：記憶體裡的狀態（未清的 listener、堆積的 callback 隊伍、未淘汰的快取）**重啟即失**——OOM 重啟會清掉洩漏，但也清掉所有 in-flight 工作；洩漏的「治標」其實就是定期被迫重啟（ch13）。
- **分區**：單機問題，不涉跨節點一致性；但一台機器被同步運算卡死、健康檢查逾時被踢出輪替時，會把壓力推給其他機器→連鎖（**雪崩的起點**）。
- **並發**：本章核心。event loop 是**單執行緒非搶占並發**：沒有鎖、沒有 race（單迴圈內），但任何一個 callback 跑太久就讓全部人乾等——並發的強處與死穴是同一個設計（深掘見 ch12）。
- **觀測**：第一線指標是 **event loop lag**（不是 CPU 數字本身）＋ p99／p999 ＋ RSS 趨勢（單調增＝洩漏 vs 鋸齒＝GC 壓力）。壓測要看 p99 不看平均，否則協調遺漏會騙你（ch15）。

### NeoBards：FluentBit + Lua 可觀測性管線（ch06）

「從 log 反推 metric」這條常見但有陷阱的路。最反直覺的一格在**分區**——百分位數不可合併。

- **交付**：log 行可能在 buffer 壓力下被丟（at-most-once 傾向）；丟一行＝metric 少一個樣本。代價：**metric 準確度被 log 完整度封頂，尖峰最不準**（而尖峰正是你最在乎 metric 的時候）。
- **持久**：預設記憶體 buffer，行程重啟／input 暫停期間 log 輪替即丟；改檔案系統 buffer（`storage.type filesystem`）才耐久。代價：耐久換磁碟 I/O。
- **分區**：跨多台邊緣各自聚合，**若送已算好的 p95 則無法正確合併**（百分位不可合併、不能跨機器跨視窗平均）；須送 histogram／t-digest 或原始分布。代價：可合併性 vs 頻寬。
- **並發**：各邊緣獨立處理自己的 log，無共享狀態、無鎖；但這也意味著跨機器的全域統計必須在下游靠可合併結構重建。
- **觀測**：這條管線**本身**就是觀測工具——但它自己壞了（行程掛、解析失敗、buffer 丟）多半**靜默且讓指標「變好看」**（丟掉的往往是壞樣本）。代價：必須對觀測管線本身加存活監控（dead man's switch）與解析失敗計數，否則盲點最大。

### LetsTalk：即時通知系統（ch07）

WebSocket + Redis Pub/Sub、~3,200 並發、~10M 訊息/月、補 FCM 在中國的缺口。**觀測是它最弱的一軸。**

- **交付**：Pub/Sub ＝ **at-most-once**：訂閱者斷線期間的訊息**靜默永久丟失**，`PUBLISH` 回 0 不報錯。靠「DB 持久化訊息本體 ＋ Pub/Sub 只當即時信號 ＋ 重連拉取補發」把交付保證從 Pub/Sub 移到 DB。
- **持久**：Pub/Sub **零持久化**——WS server 重啟、連線斷、Redis 重啟，in-flight 的推送全失憶。真相來源在 DB；連線本身不持久（接受重連是常態，做好重連恢復）。
- **分區**：多裝置（手機／平板）＋離線補發 ⟹ 各端看到的對話**最終一致**，靠重連拉取對齊；已讀同步是另一個一致性問題（ch11）。Pub/Sub 只保在線那一段的即時。
- **並發**：~3,200 並發長連線分散在多台 WS server，sticky session 釘住每條連線；跨機找人靠 Redis Pub/Sub 廣播。**扇出放大**（熱門群／全域 channel）是並發壓力的放大器——per-room channel／sharded Pub/Sub 防之。
- **觀測**：**最弱的一軸**：Pub/Sub 掉訊息是**靜默的**（`PUBLISH` 回 0 非錯誤），幽靈連線讓「在線數」虛高。要監控：真實活躍連線數（心跳驗證）、`PUBLISH` 觸達 0 的比例、重連拉取補發的訊息量（補太多＝即時層在大量掉訊息的徵兆）。

### LetsTalk：WebRTC + Jitsi + coturn 通話（ch08）

評估 Twilio／Agora、因成本選 Jitsi、Socket.io 信令、coturn 做 NAT 穿透。觀測軸最陰險——**少數派故障你自己永遠重現不出來**。

- **交付**：媒體走 SRTP over UDP，**即時優先、不保證可靠**（丟封包就丟、不重傳，這對即時影音是對的——遲到的影格不如不要）；信令層（Socket.io）才需要可靠交付 offer／answer。代價：媒體本就容忍丟失，但**信令丟一則 offer／answer 通話就建不起來**，信令層要 ack／重送。
- **持久**：通話是**短暫狀態**，沒有持久化需求——掛斷即結束，沒有「重啟後續上一通」這種事。JVB／coturn 重啟會中斷進行中的通話；無痛接續需另設計（一般直接重撥）。
- **分區**：P2P 直連時雙方各自一份媒體、無共享狀態；SFU 群組時 JVB 是媒體的匯流點，**某台 bridge 分區會讓其上所有會議全斷**——自架 SFU 把「群組的可用性」綁在 JVB 上，JVB 的分區域就是你的故障域。
- **並發**：ICE 同時蒐集並測多條 candidate、選最優路徑；SFU 同時轉發多人多路——但**轉發不轉碼**，並發負擔遠低於 MCU。mesh 的並發負擔落在每個 client（N−1 路上行）；SFU 把它收斂到伺服器。
- **觀測**：最該盯「**走 relay 的通話比例**」「各使用者連線成功率」「每台 JVB 負載」——symmetric NAT 那群人的故障**你（在 cone NAT 的辦公網路）自己永遠重現不出來**。沒有這些 metric，少數派使用者的不可用會靜默到只剩客訴。

### LetsTalk：排程訊息（ch09，setTimeout 版 → job queue 版）

「在會隨時重啟的行程裡，可靠地表達未來」。這個系統是**唯一一個書中對照了「重構前 vs 重構後」兩個版本**的案例——五軸的答案在重構中**整排翻面**，最戲劇性的是持久軸（0 → 存活）與觀測軸（0 → 可查），代價是交付軸從「會掉」翻成「會重」。

| 軸 | setTimeout 版 | job queue 版 |
|---|---|---|
| **交付** | 靜默掉失；超 24.8 天上限還會**提早誤發**（delay 被 clamp 成 1ms） | **at-least-once**（重啟／worker 被殺可能重複，需冪等償還，ch10） |
| **持久** | **0**——計時器活在單一行程記憶體、重啟即失憶、零補救 | 存活，**但前提是 Redis 開了 AOF**（預設 RDB 在 crash 時仍可能掉，ch17） |
| **分區** | 單機、不適用 | 狀態集中在 Redis（**新的單點與容量瓶頸**） |
| **並發** | timer 與 closure 堆積壓單一行程 heap，逼近 OOM | pending 移到 Redis sorted set（score＝執行時間），多 worker 靠 Redis 原子操作搶 job |
| **觀測** | **0**——pending 藏在 libuv heap，列都列不出來、壞了無感 | pending 在 Redis **可查、可數、可告警**（從「壞了才知道」變「主動可看」） |

> 一句話：這次重構表面是「setTimeout 換 Bull/BullMQ」，底下是把「未來該做的事」這種需要持久、可靠、可觀測的狀態，從**會隨時死掉的行程記憶體**搬到**行程外的持久儲存**。它解開了 15 分鐘的鐐銬，但沒有消滅問題——只是把持久性問題挪去 Redis（要開 AOF）、把可靠性的代價換成 at-least-once（要冪等兜底）。

### 雲端原語並排（ch18）

ch18 的主張：**它們不是可互換的積木——選一個原語，就是吞下它替你做好的整組五軸決定、連同它甩回給你的那幾軸**。各原語的逐軸結論已在上方速覽總表（SQS Standard／SQS FIFO／SNS／Lambda／ALB／API Gateway 六列），與 ch18 章末並排表一致，此處不重複；要看「每軸它保證什麼、又甩回什麼給你」的完整版，回 ch18。一句話收口：**沒有一個原語在五軸上全綠。**

## 橫向觀察：把整張表豎著讀

把表豎著一欄一欄讀（同一軸跨所有系統），會浮出三件在單一系統裡看不到的事。

### 一、最常被讀者系統忽略的，是觀測軸——而且它的忽略有個共同形狀

豎讀「觀測」這一欄，幾乎每個系統的觀測格都帶著同一個詞：**靜默**。

- 結算 pipeline 的「靜默少拿」無錯誤日誌、無客訴（ch04）。
- 即時通知的 Pub/Sub 掉訊息「`PUBLISH` 回 0 不是錯誤」（ch07）。
- WebRTC 的 symmetric NAT 那群人「你自己永遠重現不出來」（ch08）。
- setTimeout 排程「pending 列都列不出來、壞了無感」（ch09）。
- FluentBit 管線**自己**壞了「讓指標變好看」（ch06）——最陰險，因為沒人會去查一個變好的指標。

這不是巧合。前四軸（交付／持久／分區／並發）是「系統怎麼壞」——壞了你做系統時自然會撞到、會去解。**觀測是「你怎麼知道它壞了」，它不會主動逼你注意，要你主動設計**（ch02 的核心主張）。所以它是五軸裡最常被留到最後、甚至根本沒答的一軸。豎讀這一欄，等於把「你當年最沒認真想的那一軸」在所有系統上一次點名。

### 二、同一軸上，不同系統做了相反的取捨——這才是「沒有正確答案」的證據

五軸是取捨表不是計分卡，最直接的證據就是：**同一軸上，書中系統做了方向相反的選擇，而且各自都對。**

**交付軸**最戲劇：

- **即時通知**選 **at-most-once**（Pub/Sub fire-and-forget）——要的是速度，「即時」比「絕不掉一則在線通知」重要，掉了用 DB 兜（ch07）。
- **結算 pipeline／SQS Standard** 選 **at-least-once**（會重、靠冪等吃掉重複）——要的是不丟，每一筆獎勵不能漏，重複可以用冪等收斂（ch04／ch18）。
- 兩者**在同一軸選了相反端**，因為「一則在線通知」和「一筆遊戲幣」對「掉」與「重」的容忍度天差地別。對的系統，對的軸上選對的強度。

**持久軸**也對立：

- **WebRTC 通話**選「**不持久**」是對的——通話是短暫狀態，沒有「重啟後續上一通」這種需求（ch08）。
- **排程訊息**選「**必須持久**」也是對的——「3 天後發」這種未來工作丟了就是事故（ch09）。
- 同一軸，一個刻意不做持久化、一個拼命外部化狀態，差別只在「這個狀態值不值得活過重啟」。

**並發軸**的取捨方向也相反：

- **RDS 寫入路徑**的勝利是**減少**並發協調（拆交易、統一鎖序、降鎖競爭，ch03）——它在「序列化的代價」上往「少鎖」走。
- **Node event loop** 則是**擁抱**單執行緒並發（用非阻塞 I/O 撐上萬連線、根本沒有鎖，ch05）——它在「並發模型」上往「不要共享狀態的鎖」走。
- 一個在多執行緒世界裡精算鎖、一個在單執行緒世界裡躲開鎖，都是針對各自場景的正解。

把這些並排，你就懂了 ch02 那句中心句的份量：**這把尺的用法不是打分數，是逼你把每一軸的取捨講出來、講清楚被你放棄的那一端輸在哪。** 一個你能在五軸上都說出「我選了這端，因為另一端在這維度輸了」的系統，就是一個你真正懂的系統——這也正是 ch19 把每個系統講成面試級敘事的骨架。

### 三、一道防線常常同時答好幾軸——五軸會交會

最後一個豎讀才看得到的事實：**冪等**這一個設計，在表裡反覆出現，而且每次都同時站在好幾軸上：

- 在結算 pipeline，冪等鍵同時是**交付軸**（吃掉 at-least-once 的重複）和**並發軸**（擋掉兩個 consumer 的並發重複寫）的答案（ch04）。
- 在排程 job queue，冪等償還的是**持久軸**修好後冒出來的**交付軸**新債（重啟存活的代價是會重複，ch09）。
- 在 SQS／Lambda，冪等是所有 at-least-once 路徑的共同收尾（ch18）。

這說明五軸不是互不相干的五張清單——它們在真實系統裡會交會，**修好一軸常常把問題推到另一軸**（job queue 修好持久軸、把問題推給交付軸），而**一道好的防線常常同時答好幾軸**（冪等同時擋交付與並發）。這正是「at-least-once ＋ 冪等是分散式系統的預設安全帶」（ch02 驗收標準②、ch10 收官）為什麼是全書反覆出現的那條線——它是橫跨最多軸、最多系統的單一答案。
