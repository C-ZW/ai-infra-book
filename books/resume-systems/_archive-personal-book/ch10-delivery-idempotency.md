## 從你已知的出發

你在兩個系統裡都靠「冪等」兜過底。

NeoBards 的結算 pipeline（ch04）：CronJob → SQS → consumer，每季 10–30 萬筆獎勵。你特別讓 consumer 冪等，理由你當時大概是這麼說的——「SQS 可能重複投遞，保險起見做一下」。

LetsTalk 的排程訊息（ch09）：你把 setTimeout 換成 cron 驅動的 job queue，重啟存活。換完之後你發現一個新問題——job 重啟後**可能重跑一次**，於是你又回頭給它加了冪等。你當時大概也是同一句話：「重啟可能重跑，保險起見做一下」。

兩次都是同一個動作（加冪等），兩次都是同一句解釋（保險起見）。但「保險」這個詞透露了一件事：你當年把冪等當成**可加可不加的安全冗餘**——加了更安心，不加也許大多數時候沒事。

這一章我要把這個認知**整個翻過來**。冪等不是保險，是**前提**。在這兩個系統裡，「訊息會重」不是「有時候會發生的意外」，是**架構層面保證會發生的事**——不是 if，是 when。你不做冪等，不是「少了一層保險」，是「這個架構從第一天起就是壞的，只是還沒爆給你看」。

要說清楚為什麼，得先回答一個你大概從沒認真想過的問題：

**為什麼訊息系統不能就「剛好送一次」？AWS、Kafka 這些公司養了一堆博士，連「送一次」這種事都做不到？**

答案不是工程能力不夠。是**數學上不可能**——而且不可能的根源簡單到一句話就能講完：**網路會丟封包，連 ack（確認）也會被丟，於是送方永遠分不清「對方沒收到」和「對方收到了、但 ack 在回來的路上丟了」**。這兩種情況在送方看來**長得一模一樣**，而它必須在「再送一次」（可能造成重複）和「放棄」（可能造成丟失）之間二選一。沒有第三條路。

你大概也聽過 SQS FIFO 是「exactly-once」、Kafka 有「exactly-once semantics（EOS）」。這些都是真的——但它們是 exactly-once *processing*（處理），不是 exactly-once *delivery*（交付）。這兩個詞差一個字，差的卻是「真的做到了」和「在底下偷偷做了冪等、然後對你說做到了」的距離。這一章就把這個距離量給你看。

## 三種交付語意：精確定義與各自的代價

先把詞定死。一個訊息系統，送方（producer/sender）把訊息交給某個傳輸層（網路、broker、佇列），希望收方（consumer/receiver）拿到並處理。「交付語意」回答的是：**這則訊息最後被收方處理了幾次？** 只有三種可能的保證。

**at-most-once（至多一次）**：每則訊息**最多**被處理一次——可能零次（掉了），但**絕不會超過一次**。

- 怎麼做到：送方把訊息扔出去就不管了（fire-and-forget），不等 ack、不重送。收方收到就收到，沒收到就算了。
- 代價：**會掉訊息**。網路抖一下、收方剛好重啟，這則訊息就永遠消失，沒有人會發現、沒有人會重送。
- 你的系統裡誰是這個語意：**Redis Pub/Sub**（ch07、landscape §3）。它 fire-and-forget、無持久化，訂閱者斷線期間發的訊息**直接蒸發**。這就是為什麼它能做「即時通知」（在線的人立刻收到）但**不能**做「保證送達」（離線的人永遠錯過）。

**at-least-once（至少一次）**：每則訊息**至少**被處理一次——保證不掉，但**可能超過一次**（重複）。

- 怎麼做到：送方送出後**等 ack**；在 ack 回來之前，它就**重送**。收方處理完才回 ack。
- 代價：**會重複**。只要 ack 丟了（或來不及回），送方就會重送一則其實已經被處理過的訊息。
- 你的系統裡誰是這個語意：**SQS 標準佇列**（ch04、landscape §1）——at-least-once、可能重複、best-effort 排序。**Redis Streams**（consumer group + `XACK`，landscape §3）也是。你的**結算 pipeline 整條**站在這個語意上。

**exactly-once（恰好一次）**：每則訊息**恰好**被處理一次——不掉、不重。

- 這是大家都想要的。問題是：**作為「交付」保證，它不可能存在**。下一節證明。
- 你聽過的「exactly-once」（SQS FIFO、Kafka EOS）全部是 exactly-once *processing*，是用「at-least-once 交付 + 冪等/去重」組合出來的**幻覺**。再下一節拆穿。

把三者擺成一張表，看清楚它們在權衡什麼：

| 交付語意 | 保證 | 風險 | 機制 | 你的系統裡的代表 |
|---|---|---|---|---|
| at-most-once | 不重複 | **會掉** | 不重送、不等 ack | Redis Pub/Sub |
| at-least-once | 不丟失 | **會重複** | 等 ack、沒 ack 就重送 | SQS 標準、Redis Streams |
| exactly-once | 都不 | （交付層）**不可能** | —— | 沒有（FIFO/EOS 是 *processing* 幻覺） |

這張表的核心張力，是 at-most-once 和 at-least-once 之間那道**沒有中間地帶**的鴻溝。你**只能二選一**：要嘛容忍掉（at-most-once），要嘛容忍重（at-least-once）。沒有第三個選項說「都不容忍」——那就是 exactly-once，而它不存在。

> **我以為 vs 實際**：你大概記得「SQS 是訊息佇列，會保證把訊息送到」。一半對。「保證送到」是 at-least-once 沒錯，但你記漏了下半句——**它明確允許同一則送很多次**。at-least-once 的「至少」是雙面的：保證**至少一次**（不掉），代價是**可能多次**（重複）。你當年把它記成「可靠送達」，把重複那一面忘了——而重複恰恰是這一章的全部問題。

## 為什麼 exactly-once *交付* 不可能：一個丟失的 ack 就夠了

這一節是本章的心臟，慢一點。

我要證明的命題很具體：**在一個訊息可能丟失的網路上，沒有任何協定能保證「訊息被收方恰好處理一次」**。注意前提——「訊息可能丟失」。這不是什麼極端假設，這就是真實網路：封包會丟、連線會斷、機器會在最糟的時刻重啟。只要這個前提成立，exactly-once 交付就不可能。

證法是把它退化成一個經典問題：**兩將軍問題（Two Generals' Problem）**——這是電腦通訊史上第一個被**證明無解**的問題（2026-06 查證，下節給來源）。

### 兩將軍問題的退化版

兩支軍隊（將軍 A、將軍 B）分別在敵城兩側的山頭上。他們只有**同時**進攻才會贏；任何一方單獨進攻就會被殲滅。他們之間唯一的通訊方式是派信使穿過中間的山谷——而**山谷被敵人控制，信使可能被攔截（訊息可能丟）**。

問題：他們能不能透過互送信使，**確定地**約好「明天拂曉一起進攻」？

A 派信使送「明天拂曉進攻」。但 A 不能直接進攻——萬一信使被攔截，B 根本沒收到，A 單獨進攻就是送死。所以 A 必須等 B 回個 ack：「收到，我會進攻」。

但現在輪到 **B** 不敢動了：B 送出的那個 ack 也可能被攔截。如果 ack 被攔截，A 就沒收到確認、A 不會進攻——而 B 卻以為「我回了 ack，A 應該知道我要來」，於是 B 單獨進攻、送死。所以 **B 在進攻前，必須等 A 對「B 的 ack」再回一個 ack**：「我收到你的確認了」。

但這下輪到 **A** 不敢動了——A 對 B 的 ack，**也可能被攔截**……

看出來了嗎？**這是無限的**。無論送了多少回合，**最後送出那一封**的人，永遠不知道對方有沒有收到。每一個協定都可以被同一招破解：**攔截掉最後那一封 ack**，送它的人就陷入「我不知道對方知不知道」的不確定，不敢行動。

> 這就是兩將軍問題無解的核心：**任何有限的訊息序列，最後一封都可能丟，而送出最後一封的人無法確認對方收到**。沒有「最後一封的 ack」能逃過這個結構——因為那個 ack 自己又成了「新的最後一封」。（2026-06 查證：兩將軍問題是「第一個被證明 unsolvable 的電腦通訊問題」；注意一個常見的混淆——它和 FLP 不可能性定理[Fischer-Lynch-Paterson 1985，講的是非同步系統中即使只有一個行程會故障、共識也無法保證終止]是**兩個不同的結果**，雖然精神相通、常被一起提，但別把它們講成同一個定理。）

### 從兩將軍退化到「送一則訊息」

把將軍換成你的系統，問題一模一樣。送方 S（你的 producer）要把一則訊息交給收方 R（你的 consumer），希望它**恰好被處理一次**。

```
   S（送方）                            R（收方）
     │                                   │
     │   送出訊息 ─────────────────────► │
     │                                   │ 處理訊息（加 5000 金幣）
     │   ◄─────────────────────── 回 ack │
     │                                   │
     │  ┌────────────────────────────┐   │
     │  │ 但這個 ack 如果在這裡丟了？│   │
     │  └────────────────────────────┘   │
```

關鍵就在最後那個 ack。從 **S 的視角**看，「沒收到 ack」這件事有**兩種可能、而它分不清是哪一種**：

1. **R 根本沒收到訊息**（訊息在去程丟了）——這時 R 沒處理，S **應該重送**。
2. **R 收到並處理了，但 ack 在回程丟了**——這時 R 已經處理過，S **不該重送**（重送就重複）。

這兩種情況，在 S 看來**完全一樣**：都是「我送了、我沒收到 ack」。S 手上沒有任何資訊能區分它們。於是 S 被逼到一個二選一：

- **選擇重送**（為了不掉訊息）→ 如果真相是情況 2，R 就**處理了兩次**（重複）→ 這就是 **at-least-once**。
- **選擇不重送**（為了不重複）→ 如果真相是情況 1，R 就**一次都沒處理**（丟失）→ 這就是 **at-most-once**。

**沒有第三個選擇能同時避開重複和丟失**——因為要避開兩者，S 必須**知道** R 到底處理了沒，而那個「知道」需要一個保證送達的 ack，那個 ack 又需要它自己的 ack……無限回到兩將軍。

**這就是 exactly-once 交付不可能的全部證明**：一個可能丟失的 ack，就把「恰好一次」劈成了「至少一次」和「至多一次」兩個互斥選項。你只能挑一邊。

> **這一段值得你停下來。** 因為這是全書最反直覺、而你大概一直記錯的一件事。你以為「exactly-once 做不到」是因為太難、太貴、工程上沒實作好。不是。是**邏輯上不可能**——不是 NASA 砸再多錢也做不到的那種不可能，是「三角形不可能有四個角」的那種不可能。理解這一點，你才會明白為什麼底下要靠冪等——**因為交付層永遠給不了你 exactly-once，你只能在處理層自己造一個出來。**

## 所謂 exactly-once *processing*：一個用冪等造出來的幻覺

既然 exactly-once 交付不可能，那 SQS FIFO 和 Kafka EOS 的「exactly-once」是在賣什麼？

它們賣的是真貨，但賣的是**另一個東西**——exactly-once **processing**（處理效果），不是 exactly-once **delivery**（交付次數）。差別是：

- exactly-once **delivery**：訊息在線上**恰好被傳輸一次**。← 不可能。
- exactly-once **processing**：訊息可能**被傳輸很多次**，但**對系統狀態的效果，等同於只處理了一次**。← 可能，靠去重/冪等。

換句話說，所有號稱 exactly-once 的系統，底層都是**同一個把戲**：

> **at-least-once 交付（保證不掉、容忍重複）＋ 去重或冪等（讓重複無害）＝ exactly-once processing（效果上剛好一次）**

它們沒有違反兩將軍——它們**老老實實做 at-least-once**（送方重送、保證不掉），然後在**收方那一側做去重**，把重複的那些吃掉，讓你**看起來**像只處理了一次。「exactly-once」是這層去重的**行銷名字**，不是它真的在線上只傳一次。

### SQS FIFO 的把戲：5 分鐘去重窗口

SQS FIFO（landscape §1）的「exactly-once processing」靠的是 **`MessageDeduplicationId` ＋ 5 分鐘去重窗口（deduplication window）**（2026-06 查證，AWS 官方）。機制是這樣：

- 你送訊息時帶一個 `MessageDeduplicationId`（或開 content-based dedup，讓 SQS 用訊息內容的 SHA-256 當這個 ID）。
- 在 **5 分鐘**的窗口內，如果 SQS 已經收過一則相同 `MessageDeduplicationId` 的訊息，後來的重複會被**接受但不投遞**（acknowledged but not delivered）——靜靜吃掉。
- 重點：SQS **在訊息被收取、刪除之後仍持續追蹤這個 dedup ID**，所以窗口保護對已處理的訊息也成立。

看清楚了嗎？這就是 SQS **替你做的一層冪等**。它本質上跟你在結算 pipeline 裡用 `settlement_key` 做的事一模一樣——用一把鍵去重——只是它做在 broker 裡、用 broker 的時鐘窗口。它不是「線上只傳一次」，它是「傳很多次、但 5 分鐘內幫你濾掉重複的」。

### Kafka EOS 的把戲：序號 + 交易

Kafka 的 exactly-once semantics（2026-06 查證）更精緻，但骨子裡是同一招——只是把「冪等」做得更深：

- **冪等 producer**：Kafka 給每個 producer 一個 Producer ID（PID），對每個 partition 維護一個**遞增序號（sequence number）**。producer 重送時，broker 比對序號、發現重複就**只回 ack、不再寫一次**——這就是 broker 端對「producer 因 ack 丟失而重送」做的去重。
- **交易 + read_committed**：讓 producer **原子地**寫多個 partition（跨 partition 的兩階段提交），consumer 只讀已提交的資料。

每一層底下還是 at-least-once（producer 還是會因沒收到 ack 而重送），Kafka 只是在每層都加了「偵測並丟棄重複」。代價也擺著：**EOS 約增 2–5ms 延遲、降 10–20% 吞吐**（2026-06 常見量級，這是只開 idempotent producer 的量級；再加上跨 partition 的 transaction，吞吐降幅更大、視 message size 與 commit interval 可達 15–30%），所以連 Kafka 自己都建議——**關鍵流程才用 EOS，其他用 at-least-once 就好**。

> **我以為 vs 實際**：你大概以為 SQS FIFO 和 Kafka EOS 是「比標準佇列更高級的傳輸技術，真的做到了線上只傳一次」。實際上它們**沒有也不可能**做到線上只傳一次——它們做的是 **at-least-once 交付 + broker 端去重**，跟你在 consumer 裡手寫的冪等是同一類東西，只是搬進了 broker。理解這點有個直接的實戰後果（下節推演會用到）：**就算你用了 FIFO，你業務層的冪等也常常不能省**——因為 broker 的去重窗口是有限的、且只管「broker 到 consumer」這一段，管不了你下游跨服務重試時製造的重複。

## 冪等設計學：把「至少一次」收斂成「效果上剛好一次」

既然 exactly-once 交付給不了、而 broker 的去重又有它的邊界，那**正確性的最後一道防線，永遠在你自己的處理層**。這道防線就是**冪等（idempotency）**：讓「處理多次」和「處理一次」對系統狀態的效果**完全一樣**。

冪等的精確定義：一個操作 $f$ 是冪等的，若 $f(f(x)) = f(x)$——做兩次跟做一次結果相同。注意這不是「不執行第二次」，而是「執行第二次也不改變結果」。

### 天然冪等 vs 需要冪等鍵

有些操作**天生就冪等**，你不用做任何事；有些操作**天生不冪等**，你必須額外造一把鍵讓它冪等。分清楚這兩類，是冪等設計的第一步。

**天然冪等的操作**——「設定到某個絕對狀態」：

```
SET balance = 5000        # 設成 5000，做幾次都是 5000 ✓
PUT /user/993021 { ... }  # 把整個資源覆寫成這個值，做幾次都一樣 ✓
DELETE /session/abc       # 刪掉它，做幾次都是「沒了」✓（第二次刪一個已經沒的東西，結果還是「沒了」）
```

這類操作的共通點：它們**描述的是「最終該長什麼樣」（目標狀態），不是「做什麼增量改變」**。把帳戶設成 5000，重複一萬次帳戶還是 5000。HTTP 的 `PUT`、`DELETE` 在語意上就被設計成冪等（這是 REST 規範的一部分），`GET` 更是天然冪等（讀不改狀態）。

**天然不冪等的操作**——「做一個相對的增量改變」：

```
balance = balance + 5000   # 加 5000，做兩次就加了 10000 ✗
INSERT INTO mail (...)     # 寄一封信，做兩次就兩封信 ✗
publish_reward(player)     # 發一次獎，做兩次發兩份 ✗
```

這類操作的共通點：它們**描述的是「相對於現狀的增量」**——加錢、發獎、寄信、扣庫存。重複執行就是重複累加。你的結算 pipeline 裡每一步（player 加錢、inventory 加道具、mail 寄信）**全是這一類**。這也是為什麼那條 pipeline 非冪等不可——它做的全是不冪等的增量操作，而交付層保證會重投。

對天然不冪等的操作，要讓它冪等，唯一的辦法是給它**配一把冪等鍵（idempotency key）**：一個唯一標識「這筆工作」的字串，讓系統能判斷「這筆我做過沒」。

### 冪等鍵放哪：請求層 vs 資料層

冪等鍵可以放在兩個層次，而且它們**不是二選一，是分工**（2026-06 查證 Stripe/Postgres 的標準做法）：

**請求層（middleware/快取回應）**：client 帶一把 idempotency key 來，server 用它認出「這是同一個請求的重試」，把第一次的**回應**快取起來、第二次直接回放快取的回應。Stripe 的 `Idempotency-Key` header 就是這個。它擋的是**傳輸層的重複**（client 重試），讓重複在業務層無害。

**資料層（資料庫唯一約束）**：把冪等鍵做成資料庫某欄位的 **UNIQUE 約束**。這是**真正靠得住的那一層**——因為只有資料庫有足夠強的並發原語，能保證「在多個 worker 同時重試下，只有一個會成功寫入」。

為什麼非要資料層兜底？因為請求層的快取**擋不住並發 race**。看這個你一定見過、而且在壓測時看不出來、上線才爆的錯誤寫法：

```
# ❌ check-then-act：在並發下有 race window
if not db.exists(idempotency_key):    # 步驟 1：檢查「做過沒」
    apply_reward(player, 5000)         # 步驟 2：做（這中間有空隙）
    db.insert(idempotency_key)         # 步驟 3：記下「我做了」
```

問題在步驟 1 和步驟 2 之間的空隙。兩個並發 worker（在你的 pipeline 裡，這就是 SQS visibility timeout 重投造成的「A 還沒刪、B 又領走」場景，見 ch04）可能**同時**通過步驟 1 的檢查——**兩個都看到「還沒做過」**——然後**兩個都** `apply_reward`。玩家收到兩份。

而且——這點特別重要——**就算把資料庫隔離級別調到 Read Committed 也救不了**（2026-06 查證確認）：兩個 worker 各自的 `SELECT` 都看到「列不存在」，於是各自 `INSERT`，最後只有一個撞唯一約束失敗——但**另一個已經執行過副作用了**。隔離級別擋不住這種「先讀後寫」的競態（完整的隔離級別異常分析見 ch11）。

正確的寫法是把「宣告我做了這筆」變成一個**原子操作**，靠唯一約束本身去仲裁：

```
# ✅ 用唯一約束讓「宣告」原子化
try:
    db.insert(idempotency_key)         # idempotency_key 是 UNIQUE 欄位
    apply_reward(player, 5000)         # 只有 insert 成功者才執行
except UniqueConstraintViolation:
    return cached_response_or_skip()   # 別人已經做了，放行、不重做
# PostgreSQL 慣用寫法：INSERT ... ON CONFLICT DO NOTHING
```

唯一約束把「同一把鍵只能被宣告一次」下放到資料庫的並發控制（行鎖、索引唯一性，見 ch03 的鎖機制）——無論多少 worker 同時搶，**只有一個 insert 成功**，其餘全撞約束失敗。**先有約束，再談冪等**；只靠應用層的 `if exists`，在並發下等於沒做。

### 去重窗口的有限性：為什麼鍵不能永遠記著

這是冪等設計裡最容易被忽略、而代價最隱蔽的一點：**任何去重機制的「記憶」都是有限的**。

去重的本質是「記住我做過哪些鍵」。但這份記憶**不能無限長**——因為它要佔儲存、要佔索引、要被查詢。所以每個去重機制都有一個**窗口**，窗口外的記憶就被丟掉了：

- **SQS FIFO**：dedup 窗口 **5 分鐘**（landscape §1）。5 分鐘內的重複擋得住，**第 6 分鐘來的同一則訊息，SQS 已經忘了它、會當成新訊息投遞**。
- **Stripe idempotency key**：可在 **≥24 小時**後被清掉（2026-06 查證）；key 被清掉後重用同一把鍵，Stripe 會當成新請求處理。
- **PayPal**：依 API 而定，有的長達 **45 天**。

為什麼 SQS FIFO 只給 5 分鐘？因為 dedup 是 SQS 替**海量訊息**做的事——它不可能為每個佇列、每則訊息永久保存一份 dedup ID 的記錄（那是無限增長的儲存）。5 分鐘是「足以覆蓋絕大多數因 ack 丟失/網路抖動造成的短時重送」和「儲存成本可控」之間的工程折衷。它假設：**正常的重複都發生在很短的時間內**（producer 沒收到 ack，幾秒內就重送了）。

但這個假設有破口。考慮一個**跨越窗口**的重複：

```
T=0:00   producer 送訊息（dedup_id = X），SQS 記下 X
T=0:01   producer 沒收到 ack（ack 丟了），但這次它沒立刻重送
         （比方 producer 自己當機、重啟、過了幾分鐘才從某個持久佇列裡
           重新撈出這筆待送任務）
T=6:00   producer 重啟完，重送同一筆（dedup_id = X）
         → 但 SQS 的 5 分鐘窗口早過了、它已經忘了 X
         → SQS 把它當成全新訊息，投遞 → 重複！
```

窗外失效不是 bug，是**設計的邊界**。SQS FIFO 從沒承諾「永遠不重複」——它承諾的是「**5 分鐘內**不重複」。任何超過 5 分鐘的重送，它擋不住。

> **這就是為什麼 broker 的去重不能取代你業務層的冪等**：broker 的去重窗口有限（5 分鐘），而真實世界的重複可以遲到（producer 重啟、慢、跨服務的延遲重試都可能讓重複在幾小時後才出現）。你業務層用 `settlement_key` 做的去重，鍵記在**你自己的資料庫**裡，你可以讓它活一整季——窗口由你的業務需求決定，不是由 broker 的成本折衷決定。**broker 的 dedup 是第一道濾網，你的冪等是最後一道防線。**

## at-least-once + 冪等 vs at-most-once + 容忍丟失：決策框架

到這裡，全書最重要的一句斷言可以下了：

> **「at-least-once 交付 + 冪等」是分散式系統的預設安全帶。**

為什麼是「預設」？因為 exactly-once 交付不可能（你只能在 at-least-once 和 at-most-once 之間選），而兩者裡：

- **at-least-once + 冪等**：交付層保證**不掉**（失敗的有機會重試），冪等保證**重送無害**（成功的不會重做）。兩者合起來給你 exactly-once *processing*——**效果上剛好一次**。
- **at-most-once + 容忍丟失**：交付層**會掉**，你只能**接受掉的那些**。

第一條是預設，因為它對**絕大多數**業務操作（加錢、發獎、扣庫存、寫訂單）是**唯一可接受**的——這些操作掉一筆就是真金白銀的損失或客訴。但「預設」不等於「永遠」。有些場景，at-most-once + 容忍丟失反而是**對的選擇**：

| 維度 | at-least-once + 冪等 | at-most-once + 容忍丟失 |
|---|---|---|
| 保證 | 不掉（重試補上） | 不重（但會掉） |
| 你要付的工 | 設計冪等鍵、唯一約束、去重儲存 | 幾乎零成本（fire-and-forget） |
| 適合 | 不能掉的操作（錢、獎、訂單、庫存） | 掉了無所謂的操作（即時狀態推送） |
| 你的系統代表 | 結算 pipeline、排程訊息 | Redis Pub/Sub 即時通知 |

關鍵判準只有一個：**這則訊息掉了，會死人嗎？**

- **會**（少發一筆獎、漏一筆扣款）→ 你**不能**容忍丟失 → 走 at-least-once → 那你**必須**做冪等（不然會重複）→ 預設安全帶。
- **不會**（漏推一次「對方正在輸入…」的狀態、漏一幀即時位置更新）→ 你**可以**容忍丟失 → 走 at-most-once 換取零成本和零延遲。

你的兩個系統剛好示範了兩端。**結算**——一筆獎勵掉了，玩家就是少拿，會客訴、會破壞遊戲經濟——所以它**必須** at-least-once + 冪等。**Redis Pub/Sub 的即時通知**（ch07）——一次「正在輸入…」的狀態推送掉了，下一秒就有新的覆蓋它，掉了沒人在乎——所以它**可以** at-most-once、fire-and-forget，換取極低延遲和零持久化成本。

> **決策框架，一句話**：先問「掉得起嗎」。掉不起 → at-least-once → 冪等是**前提不是選項**。掉得起 → at-most-once 換成本與延遲。**沒有「我兩個都要」這個選項**——那是 exactly-once，它不存在。你當年在 Redis Pub/Sub 上選 at-most-once、在結算上選 at-least-once + 冪等，其實是**同一個決策框架的兩端**——只是你當年是憑直覺走到對的兩端，沒把這條判準說出口。

## 故障模式與防禦

交付與冪等這個老問題會怎麼壞、壞了長什麼樣、怎麼防。至少一個綁定你真實的系統。

| 故障 | 壞了長什麼樣（可觀測徵兆） | 怎麼防 |
|---|---|---|
| **把 at-least-once 當 exactly-once 用**（用了 SQS 標準佇列卻沒做冪等，**你的結算 pipeline 若不做冪等**） | 玩家貨幣翻倍、收到兩封結算信、客服收到「我怎麼有兩份」；對帳查出 player 餘額與 settlement 筆數對不上 | 認清交付層永遠不給 exactly-once；at-least-once 的下游**必須**有冪等。冪等鍵 = 業務身分（`settlement_key`），不是 broker 的 message ID |
| **冪等寫成 check-then-act**（並發兩個 consumer 同時通過 `if exists`） | 壓測時完全正常、上線高並發才偶發雙發；難以重現，因為要兩個 worker 卡在同一個微秒窗口 | 用資料庫 UNIQUE 約束 + `INSERT ON CONFLICT DO NOTHING` 讓宣告原子化；**禁止** check-then-act（Read Committed 也救不了它） |
| **以為 FIFO/EOS 就不用冪等**（broker dedup 窗口外的重複漏網，**你的系統**） | 大多數時候正常、偶爾在 producer 重啟/延遲重試後出現重複（重複間隔 > 5 分鐘）；很難對上 broker，因為 broker「自認」沒重投 | 業務層冪等不能省；broker dedup 是第一道濾網不是最後防線；鍵記在自己的 DB、窗口由業務定 |
| **冪等鍵粒度錯**（漏了 `season_id` 之類的區分維度，**你的結算 pipeline**） | 下一季同公會同玩家被當「做過」**靜默跳過**——玩家少拿、無錯誤日誌、無人客訴 | 鍵必須含**所有區分維度**；上線前用「同玩家連兩季」案例驗鍵不誤撞 |
| **以為冪等其實有隱藏副作用**（去重記錄擋住主寫入，但操作裡藏了個不冪等的副作用，如「每次都發一個外部 webhook / 寄一封信」） | 主資料只寫一次（看起來冪等），但 webhook/信被觸發多次——下游或玩家收到重複通知 | 把**所有副作用**（含外部呼叫）都納入冪等鍵保護；副作用也要綁同一把鍵去重，別只保護資料庫寫入 |
| **去重記錄寫了、業務沒寫**（insert 鍵後 consumer 當在 apply 之前） | 鍵在、事沒做；重投被當「做過」跳過——靜默少拿 | 盡量把去重記錄與業務寫入綁同一交易；跨服務做不到時靠對帳兜底（見 ch04） |
| **去重儲存無限增長**（為了「永久去重」把每把鍵都永久保存） | 去重表/索引無止境膨脹，查詢變慢、儲存爆、最終拖垮主庫 | 接受窗口有限：設一個夠長但有界的保留期（如一季），過期清理；窗口長度由「重複最遲可能多晚出現」決定 |

注意這張表裡最危險的兩類，跟 ch04 的教訓一致：一是**靜默少拿**（鍵粒度錯、去重記錄寫了業務沒寫）——沒日誌、沒客訴，只有對帳抓得到；二是**壓測測不出**（check-then-act 的並發 race）——它要剛好兩個 worker 撞在同一個微秒窗口才爆，所以壓測過了不代表安全，要靠**設計層面證明它原子**，不能靠測試「沒復現」就放心。通用的「靜默故障為什麼最致命」分類見《正常意外》（fail）。

## 紙上推演

### 推演一：手證「為什麼一個丟失的 ack 就讓 exactly-once 交付不可能」 **[20 分鐘]** ★★

不准翻回前面，自己把證明重講一遍。

**問**：(a) 設一個最簡單的場景——送方 S 送一則訊息給收方 R，希望 R 恰好處理一次。R 收到後回一個 ack。現在這個 ack 在回程**丟了**。從 S 的視角，「沒收到 ack」對應哪**兩種**它分不清的真相？(b) 對這兩種真相，S 各自「應該」怎麼做？這兩個「應該」一致嗎？(c) 為什麼 S 不能「再等一個更可靠的 ack 來區分這兩種情況」？(d) 把這個論證接回兩將軍——是哪一招破解了所有協定？

### 推演二：判斷「我們有了 SQS FIFO，是不是就不用冪等了」 **[15 分鐘]** ★★★

你的新同事說：「結算 pipeline 改用 SQS FIFO 吧，它是 exactly-once，這樣 consumer 的冪等就能拿掉了，省一堆唯一約束的麻煩。」

**問**：分四點駁或挺——(a) FIFO 的「exactly-once」精確是哪一種 once（交付還是處理）？它靠什麼機制、窗口多長？(b) 舉一個**具體的重複場景**，是 FIFO 的 dedup 窗口擋不住、但你業務層冪等擋得住的。(c) 就算 SQS 到 consumer 這段不重複了，consumer 寫四個服務時的部分失敗 + 重試會不會製造重複？FIFO 管得到嗎？(d) 一句話結論：FIFO 該不該換、冪等該不該留。

### 推演三：找出一個「以為冪等、其實不冪等」的操作 **[15 分鐘]** ★★

一段排程訊息（ch09）的處理邏輯，作者**以為**它是冪等的（重啟重跑無害）：

```
def process_scheduled_message(job):
    if db.exists(job.idempotency_key):   # 看起來有做去重
        return
    db.insert(job.idempotency_key)
    send_push_notification(job.user, job.text)   # ← 推播
    call_analytics_webhook(job.event)            # ← 外部分析 webhook
    db.update_job_status(job.id, "sent")
```

**問**：(a) 這段在**並發重跑**下冪等嗎？指出破口。(b) 就算把去重改成原子的唯一約束，這段裡有沒有**藏著的不冪等副作用**？在什麼時序下，主資料只寫一次、但某個副作用發生兩次？(c) 要讓它真正冪等（含副作用），你要改什麼？

### 推演解答

**推演一解答。**

(a) S 送了訊息、沒收到 ack，這對應兩種它分不清的真相：**(1) R 根本沒收到訊息**（訊息在去程就丟了，R 沒處理）；**(2) R 收到且處理了，但 ack 在回程丟了**（R 處理過了）。在 S 看來這兩種一模一樣——都是「我送了、沒回音」，S 手上沒有任何資訊能區分。

(b) 對真相 (1)，S **應該重送**（不然訊息掉了）。對真相 (2)，S **不該重送**（重送就重複）。這兩個「應該」**直接衝突**——同一個觀測（沒收到 ack）要求 S 做相反的事，而 S 不知道自己面對的是哪一個。它只能賭：賭重送（萬一是 (2) 就重複，這是 at-least-once）或賭不送（萬一是 (1) 就丟失，這是 at-most-once）。

(c) 因為「更可靠的 ack」**不存在**。任何 ack 都走同一個會丟封包的網路，它自己也可能丟。如果 S 要 R 對 ack 再回一個「ack 的 ack」來確認，那個「ack 的 ack」又可能丟、又需要它自己的 ack……這是無限遞迴，沒有底。**沒有任何有限的握手能逃出這個結構。**

(d) 破解所有協定的那一招是：**攔截掉最後送出的那一封訊息**。無論協定設計成幾個回合，總有「最後一封」，而送出最後一封的人**無法確認對方收到了它**——因為要確認，就需要對方再回一封，而那封又成了新的「最後一封」。兩將軍問題無解，根子就在這個「最後一封永遠沒有 ack」的結構，而 exactly-once 交付要求的「S 確知 R 恰好處理一次」正需要這樣一個不存在的、保證送達的最後確認。

**推演二解答。**

(a) FIFO 的是 **exactly-once *processing*（處理）**，不是交付。機制：`MessageDeduplicationId` + **5 分鐘**去重窗口（landscape §1）——5 分鐘內帶相同 dedup ID 的訊息只投一次。它本質上是 **SQS 替你在 broker 端做的一層冪等**，不是「線上只傳一次」。

(b) 具體場景：一個 producer 送了訊息（dedup ID = X），沒收到 ack，但它沒立刻重送——它自己當機了，重啟後從某個持久的待送佇列裡撈回這筆，**過了 6 分鐘**才重送。此時 SQS 的 5 分鐘窗口已過、已經忘了 X，於是把它當全新訊息投遞 → 重複。你業務層的 `settlement_key` 冪等記在自己的 DB、可以活一整季，擋得住這種跨窗口的遲到重複。

(c) **FIFO 完全管不到**。FIFO 只保證「SQS 到 consumer」這一段不重複；consumer 拿到訊息後寫 guild/player/inventory/mail 時若第三步失敗、整筆重投（ch04），guild 和 player 會被**再寫一次**——這個重複發生在 **SQS 之外、consumer 的業務邏輯裡**，FIFO 看都看不到。要擋它，只能靠業務層每一步都冪等。

(d) **一句話結論**：FIFO 不該（為了「省冪等」而）換——它的「exactly-once」只是 broker 端一層 5 分鐘的冪等，取代不了你業務層的冪等，而且結算根本不需要 FIFO 的嚴格排序、卻要付它的吞吐代價（見 ch04）。**冪等必須留**——它是擋下游跨服務重試重複、跨窗口遲到重複的唯一防線。

**推演三解答。**

(a) **不冪等**。破口在 `if db.exists()` 和 `db.insert()` 之間的 check-then-act race——兩個並發重跑的 worker 可能同時通過 `exists` 檢查（都看到「沒做過」），然後都往下執行。這跟推演的主旨一樣：**看起來有去重，但去重本身不是原子的**。

(b) **有藏著的不冪等副作用**——而且**就算把去重改成原子唯一約束也未必擋得住**。考慮這個時序：worker 成功 `insert` 了鍵、發了 `send_push_notification`、呼叫 `call_analytics_webhook`……然後**在 `update_job_status` 之前當機了**。鍵已經 insert 了沒錯，但如果你的冪等邏輯是「insert 成功才往下做、撞約束就跳過」，重跑時這把鍵已存在 → 被跳過 → 看起來安全。**但問題不在這裡**——問題在：如果你為了「失敗要能重試」而設計成「業務做完才 insert 鍵」（很多人這樣寫），那當機在「webhook 已呼叫、鍵還沒 insert」之間時，重跑會**再呼叫一次 webhook**。更普遍地：`send_push_notification` 和 `call_analytics_webhook` 是**外部副作用**，它們不受你資料庫交易保護——主資料（job status）可能只翻一次，但這兩個外部呼叫可能因為「在它們之後、鍵 commit 之前」的當機而被執行兩次。推播和 webhook 是不冪等的增量副作用（多發一次就多一則通知 / 多一筆分析事件）。

(c) 要真正冪等：**(1)** 用原子唯一約束取代 check-then-act；**(2)** 把去重記錄與「能綁交易的業務寫入」綁同一個資料庫交易（要嘛都成、要嘛都回滾）；**(3)** 對**無法綁交易的外部副作用**（推播、webhook），要嘛讓它們自己也冪等（帶上同一把 `idempotency_key` 給下游去重）、要嘛採 outbox 模式（先在同一交易裡寫一筆「待發送」記錄，再由另一個冪等的 relay 去實際發送，發送本身也帶鍵去重）。核心教訓：**冪等不能只保護資料庫寫入——每一個副作用（尤其外部呼叫）都要被同一把鍵保護**，否則它就是你表裡「以為冪等其實不冪等」那一行。

## 自我檢核

口頭自答，講得出來才算過關：

1. 三種交付語意（at-most/at-least/exactly-once）各自保證什麼、放棄什麼？為什麼 at-most-once 和 at-least-once 之間**沒有中間地帶**？
2. 用「一個丟失的 ack」把 exactly-once *交付*不可能講清楚——S 沒收到 ack 時，它分不清的那兩種真相是什麼？為什麼「再要一個更可靠的 ack」救不了它？
3. exactly-once *processing* 和 exactly-once *delivery* 差在哪？SQS FIFO 和 Kafka EOS「做到 exactly-once」的真相是什麼把戲（用「at-least-once + 去重」這個公式講）？
4. 哪些操作天然冪等、哪些需要冪等鍵？判準是什麼（絕對狀態 vs 相對增量）？你結算 pipeline 的每一步屬於哪一類？
5. 為什麼 check-then-act 在並發下不冪等、而且 Read Committed 也救不了？唯一約束修的是哪一個「同時」的破口？
6. 去重窗口為什麼**必須**有限？SQS FIFO 為什麼只給 5 分鐘？窗外失效會出什麼事，為什麼這代表 broker 的 dedup 取代不了你的業務冪等？
7. 「at-least-once + 冪等」為什麼是分散式系統的**預設**安全帶？什麼場景反而該選 at-most-once + 容忍丟失？判準那一句話是什麼？
8. 「以為冪等其實不冪等」最常見的破口是什麼（提示：藏在外部副作用裡）？你怎麼讓一個帶推播/webhook 的操作真正冪等？
