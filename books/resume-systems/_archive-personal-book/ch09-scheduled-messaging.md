# ch09 — 排程訊息重構：從 setTimeout 到 cron job queue

> **本章解決什麼問題**：這是 Part III「回到 LetsTalk」的最後一站，也是 Part III 收尾、把棒子交給 Part IV 橫切章的關鍵接點。ch07 的通知系統與 ch08 的視訊都在處理「**現在**」這件事——訊息此刻就要扇出、視訊此刻就要連通。這一章換一個時間軸：「**未來某個時刻**」要做的事。你當年那個「排程訊息」功能，第一版用 `setTimeout` 實作、不穩到只能限 **15 分鐘**延遲，後來重構成 cron 驅動的 job queue（任意時長、重啟存活）。表面上這是一次「換套件」的小重構；底下其實是你撞上了一個分散式系統裡反覆出現的老問題——**怎麼在一個會隨時重啟的行程裡，可靠地表達「未來」**。本章把這個問題挖到底，但只挖「為什麼 setTimeout 撐不住、job queue 怎麼補上」；它逼出的「重啟後可能重跑」回到冪等（見 ch10），「時鐘、時區、補跑」這些更深的時間問題留給 ch13。

## 從你已知的出發

LetsTalk，那個隱私導向的 IM——500K+ 註冊、~3,200 並發連線、~10M 訊息/月。某個版本你接了一個需求：使用者要能「排程訊息」，現在寫好、指定一個未來的時間發出去。生日祝福、提醒、定時公告，類似的東西。

你第一版怎麼做的，我大概猜得到，因為這是每個 Node 工程師第一次遇到「延遲執行」時最自然的反應：`setTimeout`。使用者排一則「30 分鐘後發」的訊息，你就 `setTimeout(() => sendMessage(msg), 30 * 60 * 1000)`，計時器掛在那裡，時間到了 callback 觸發、訊息送出。乾淨、直覺、零依賴。

然後它開始出問題。問題不是「偶爾沒發」，是**只要服務重啟，所有還沒到期的排程訊息就一起人間蒸發**。而服務會重啟——部署、crash 後 restart、滾動更新、節點被回收、OOM 被 kill。你大概很快發現，延遲越長、撐到發送之前遇到一次重啟的機率就越高。於是你做了一個**妥協**：把排程的延遲上限**砍到 15 分鐘**。理由很現實——15 分鐘內遇到部署重啟的機率夠低，低到使用者大概不會抱怨。

這個「15 分鐘」不是一個功能規格，是一個**跟不穩定性談判出來的數字**。你心裡清楚這不對：使用者想排「明天早上 9 點」，系統卻只讓他排「15 分鐘內」。所以你做了第二版——把「何時該做什麼」這個狀態從**行程記憶體**搬到**持久儲存**，用 cron 驅動的 job queue（當年八成是 Bull）來扛。重構後：任意時長、重啟存活、可重試。15 分鐘的鐐銬解開了。

你當年講得出這次重構的工程理由：「setTimeout 的計時器在記憶體裡，重啟就沒了；job queue 把它存到 Redis，所以重啟還在。」對，但這是**結論**，不是**機制**。我想跟你把底下這幾件你大概是憑直覺繞過去、沒真正攤開的事講清楚：

- `setTimeout` 撐不住，**不只是「重啟就沒了」這一條**。它在持久性、可靠性、記憶體、精確性**四個維度**上同時不及格——而「限 15 分鐘」這個妥協，其實是在用犧牲功能去買其中幾個維度的勉強堪用。
- 你以為 job queue「把狀態存到 Redis 所以重啟還在」。**一半對**。重啟後還在的前提是 **Redis 本身有設好持久化**——預設的 Redis 是會在 crash 時掉資料的（見 ch17 Redis 內幕）。你只是把「持久性」這個問題**從你的行程搬到了 Redis**，沒有讓它消失。如果你當年沒特別去確認 Redis 的持久化設定，你的「重啟存活」可能只是運氣好。
- 重構解開了一個鐐銬，但**套上了一個新的**：job queue 是 at-least-once 的——重啟、worker 被 kill、visibility 到期，同一則排程訊息**可能被發兩次**。你解決持久性的代價，是回到了 ch04 那條老安全帶：冪等（正式挖穿在 ch10）。

這一章就把「在會重啟的行程裡表達未來」這個問題拆到底——先解剖 `setTimeout` 為什麼是**架構級**的不及格而不是「調一調就好」，再看 job queue 用「把時間狀態外部化」這招怎麼逐項補上，最後算一筆帳：當你有 10 萬個 pending 排程在等的時候，兩種做法在記憶體上差多少。

## setTimeout 為什麼撐不住：不是一個 bug，是四個維度同時不及格

先把話說清楚：`setTimeout` 沒有壞。它在它被設計的場景裡（毫秒到秒級、行程生命週期內的延遲執行、retry 退避、debounce）是完全正確的工具。它撐不住的是**「跨越行程生命週期的、長時間的、必須可靠的」排程**——而「排一則明天發的訊息」剛好踩中這個它最不擅長的角落。

它在四個維度上同時不及格，我們一個一個拆。這四個維度不是我硬湊的——它們正好對到全書脊椎的故障問句（見 ch02）：持久、交付、（記憶體層面的）並發資源、以及一個 setTimeout 獨有的精確性問題。

### 維度一：持久性歸零——計時器活在單一行程的記憶體裡

這是最致命、也是你當年最直接撞到的一條。

`setTimeout` 回傳的那個 Timer 物件、它記著的「還剩多久觸發」「到期要呼叫哪個 callback」，**全部活在這個 Node 行程的記憶體裡**。更精確地說，它是 libuv 事件迴圈內部一個按到期時間排序的計時器堆（heap），這個堆隨行程而生、隨行程而死。callback 那個 closure 還抓著你的 `msg` 物件、抓著資料庫連線、抓著一切它需要的東西——全在 heap 上。

行程一沒了，這一切**一起沒了**。沒有任何地方記得「有一則訊息該在 14 分鐘後發」。不是「延遲發送」，是**徹底消失、無人知曉、無從補救**——使用者以為排好了，系統這邊連「曾經有這件事」的記錄都不存在。

而行程**一定會沒**。這不是「如果」，是「多久一次」：

| 重啟來源 | 頻率（典型後端服務） |
|---|---|
| 部署 / 滾動更新 | 每天到每週數次 |
| crash 後自動 restart（未捕捉例外、OOM kill） | 不可預測，但一定會發生 |
| 節點回收 / 縮容 / spot 中斷 | 視基礎設施，數天到數週 |
| 手動重啟（改設定、救火） | 隨時 |

把這張表跟「延遲時長」放一起看，你當年的妥協就很好懂了。假設你的服務平均每 8 小時部署一次（活躍開發期很常見），那麼：

- 排「15 分鐘後發」：要在這 15 分鐘內**剛好撞上**一次部署才會掉。機率粗估 15 分鐘 / 8 小時 ≈ **3%**。可以忍。
- 排「3 天後發」：72 小時內要躲過**所有**部署。3 天 / 8 小時 = 9 個部署窗口，每個都可能殺掉它。存活機率趨近於零——**3 天後那則訊息幾乎保證發不出去**。

> **我以為 vs 實際**：你大概一直把「15 分鐘上限」記成「一個保守的功能限制」。實際上它是一個**持久性等式的解**——你在「能容忍的掉訊機率」和「行程平均存活時間」之間解出來的那個延遲上限。它會隨著你部署得多頻繁而**自動變短**：團隊衝刺期一天部署五次，你那個 15 分鐘的安全假設就悄悄破了，掉訊率上升而你不會立刻知道（這就是 ch02 的觀測軸——一個靠運氣成立的數字，壞掉的時候不會有人告訴你）。

這一條的本質，就是全書脊椎的**持久性問題**（見 ch02）：行程重啟、機器當掉時，正在進行的工作會不會丟？`setTimeout` 的答案是：**全丟，且零補救**。持久性 = 0。

### 維度二：長延遲本身就不可靠——32-bit 的硬上限

就算行程奇蹟般不重啟，`setTimeout` 還有一個你大概**完全不知道**的硬限制，而且它的失敗方式極其惡毒。

Node 的 `setTimeout` 延遲參數會被轉成一個**有號 32-bit 整數**，上限是 **2,147,483,647 毫秒 ≈ 24.8 天**（2026-06，Node 官方 timers 文件）。問題不在於「超過 24.8 天不能排」——問題在於**超過上限時它怎麼壞**：

> Node 官方文件白紙黑字：當 delay 大於 2147483647、或小於 1、或是 NaN 時，**delay 會被設成 1**——也就是說，你排「30 天後發」，它不會報錯、不會等 30 天，而是**幾乎立刻就把訊息發出去**（1 毫秒後）。

停下來想一下這個失敗模式有多可怕。它**不丟例外**（你的 try/catch 攔不到）、**不拒絕**（API 回你成功）、**沒有任何警告**。使用者排「下個月生日發」，系統當場就把生日祝福發了。這是「安靜地做錯事」——在所有故障模式裡最難 debug 的一種，因為沒有任何訊號告訴你出事了，你只能等使用者來問「為什麼我的訊息提早一個月發出去了」。

你的 15 分鐘上限**順手擋掉了這個雷**（15 分鐘 = 90 萬毫秒，離 21 億遠得很），但這是**意外的副作用**，不是你當年想清楚的防禦。如果當初有人把上限放寬到「30 天」而不是換掉 setTimeout，這個 overflow 就會炸——而且炸得無聲無息。

這一條對應脊椎的**交付軸**：訊息會在錯的時間送出，等於一種交付保證的崩壞。

### 維度三：計時器堆積佔記憶體——pending 越多，行程越胖

第三條，把規模拉進來。

每一個 `setTimeout` 都是 libuv 計時器堆上的一個活物件，加上那個 closure 抓著的所有東西（`msg` 物件、可能還有引用到的服務實例）。一則排程訊息 = 一個常駐記憶體的 timer + 一坨 closure。

把這個乘上 LetsTalk 的規模。假設某個時段累積了大量未到期的排程——就算只是「全站使用者陸陸續續排的、還沒到時間的訊息」——這些 timer **全部同時活在一個 Node 行程的 heap 裡**。它們不是排隊等資源，是**每一個都實打實佔著記憶體、且不會釋放，直到到期**。pending 數量上去，行程記憶體單調上漲。

這跟 ch05 講的記憶體洩漏不一樣——洩漏是「該釋放的沒釋放」，這裡是「**該佔的就是會佔**，而且全壓在單一行程身上」。更糟的是它跟維度一形成惡性循環：pending 越多 → 記憶體越脹 → 越容易 OOM → 越容易被 kill → 一 kill **全部 pending 一起蒸發**。一個記憶體問題直接放大成一場資料損失。

我們在「紙上推演」會把這筆記憶體帳跟 Redis sorted set 並排算，你會看到差距有多大。

### 維度四：setTimeout 本就不精確——libuv 相位讓「準時」是奢望

最後一條，是 `setTimeout` 在它**最擅長的短延遲場景**裡都成立的一個事實，你可能也記得模糊：**它從來不保證「準時」，只保證「不早於」**。

機制在 ch12 會挖到相位級，這裡只取你需要的那一刀（landscape §10）：Node 的 event loop 是 libuv 的相位迴圈，順序是 **timers → pending → idle/prepare → poll → check → close**，一圈一圈轉。`setTimeout` 的 callback **只在迴圈轉到 timers 相位、且檢查到「該 timer 已到期」時才會被觸發**。

這意味著兩件事：

1. **下界被 clamp、且只是個門檻**——官方說法是「在指定時間之後**儘早**執行」。`setTimeout(fn, 0)` 不會是 0，最小延遲被 clamp；而且就算到期了，也得等迴圈轉回 timers 相位。
2. **上界完全沒有保證**——如果迴圈卡在 poll 相位處理一個慢 I/O，或者主執行緒被一段同步重運算佔住（正是 ch05 那種卡 event loop 的情況），timers 相位就遲遲輪不到，你的 callback 就**遲到**。timer 指定的是「之後可以執行的門檻」，不是「會在這個時間點執行」。

對「30 分鐘後發」這種粗粒度需求，遲個幾百毫秒甚至幾秒沒人在乎。但這條釘死了一件事：**`setTimeout` 從機制上就不是一個「準時的鬧鐘」**，它是一個「不早於某時、看迴圈忙不忙再說」的盡力而為。當你需要的是「在 T 時刻可靠地做某事」，你選錯了工具的**類別**，不是調錯了參數。

把四條收攏成一句話：

> **`setTimeout` 撐不住，不是因為它有 bug，是因為它把一個「未來該做的事」這種需要持久、需要可靠、需要可觀測的狀態，存在了一個會隨時消失、有硬上限、會吃記憶體、且本就不準的地方——單一行程的記憶體。「限 15 分鐘」不是修好它，是把功能砍到剛好讓這四個缺陷都不太會發作的那個甜蜜點。**

## job queue 怎麼解：把「未來」外部化到持久儲存

重構的核心動作，用一句話講完，就是**把「何時該做什麼」這個狀態從行程記憶體搬到行程外的持久儲存**。一旦這個狀態不再綁在某個會死的行程上，前面四個維度的問題**一次被連根拔掉大半**。

來看這個搬遷具體換掉了什麼。對照表：

| 維度 | setTimeout 版（狀態在行程記憶體） | job queue 版（狀態在 Redis） |
|---|---|---|
| 持久性 | 行程死 = 全丟、零補救 | 狀態在 Redis，行程死了重啟後**還在**（前提：Redis 有設持久化） |
| 長延遲 | 24.8 天硬上限、超過安靜出錯 | 用「執行時間」當排序鍵，**任意時長**（3 天、3 個月都行） |
| 記憶體 | 每個 pending 佔一個常駐 timer + closure 在單一行程 heap | pending 只是 Redis 裡一筆排序記錄，**不佔你的行程記憶體** |
| 精確性 | 看 event loop 忙不忙、盡力而為 | worker 輪詢到期、仍非毫秒級，但跟「3 天後發」的粒度完全匹配 |
| 重試 | callback 拋例外就沒了，無重試 | 處理失敗可重試（代價：可能重複，見下） |
| 可觀測 | 計時器在 heap 裡，**外面看不到有哪些 pending** | pending job 在 Redis 裡可查、可數、可監控 |

最後兩行其實是 job queue 最容易被忽略、但對你（重視故障視角的人）最該在意的好處：**可重試**和**可觀測**。setTimeout 版裡，你根本**沒辦法回答「現在有多少則訊息在等著發、最早的一則排在什麼時候」**——那些 timer 藏在 libuv 的 heap 裡，沒有 API 給你列出來。換成 job queue 後，「有哪些未來的工作在排隊」變成 Redis 裡一個你可以查詢、可以監控、可以告警的資料結構。從「壞了才知道」變成「可以主動看」（這正是 ch02 觀測軸要的）。

### Bull → BullMQ：一個必須校正的老兵記憶

在往下挖機制之前，先處理一個你**很可能記成舊版**的事實，因為它會直接影響你今天怎麼講這段經歷、甚至怎麼寫履歷。

你當年用的、現在大概還寫在履歷上的，是 **Bull**（OptimalBits 那個）。截至 2026-06，事實是這樣的（landscape §11）：

- **Bull（OptimalBits）已進入 legacy / maintenance 模式**——只收 bug fix、不加新功能。
- **BullMQ 是它的現役後繼者**，用 **TypeScript 重寫**，是會持續拿到新功能的那一個，最新版約 v5.71.0（2026-03）。
- 兩者由 **Taskforce.sh** 維護。2026 的明確建議是：新專案用 BullMQ；履歷上若還只列「Bull」，宜註明「已轉向 BullMQ」。

這裡有一個我要特別幫你**踩煞車**的點，因為它是個很容易被順口說錯的關聯：

> **不要說「BullMQ 屬於 Redis / 跟 Redis 是同一家」。** landscape §11 查過，找不到 Redis Ltd. 收購 Taskforce.sh 或 BullMQ 的任何證據。BullMQ **建立在 Redis 之上**（它用 Redis 當儲存），但這跟「歸屬 Redis」是兩回事。正確說法只有一句：**由 Taskforce.sh 維護、以 Redis 為儲存後端**。你面試時要是順口講成「BullMQ 是 Redis 的產品」，遇到懂的面試官會被抓包——而這完全可以避免。

為什麼這件事對你重要？因為這正是本書反覆強調的「老兵記憶會過時」的活教材。你 2019–2022 在 LetsTalk 用 Bull 的時候，Bull 還是主線、BullMQ 還很新。六年過去，主線換人了。你履歷上那個「Bull」如果不加註，今天讀起來就像「我用的是一個進維護模式的舊套件」——但只要改成「Bull（後遷 BullMQ）」，立刻變成「我跟著生態演進、知道現役該用哪個」。同一段經歷，差一個註腳，差很多。

下面講機制時我統一寫 **Bull/BullMQ**——它們的核心 delayed job 機制是同一套思路（Redis sorted set），差別在 API 與維護狀態。

### delayed job 怎麼實作：Redis sorted set 以執行時間為 score

現在挖機制。Bull/BullMQ 怎麼讓「3 天後發」這件事重啟存活、且記憶體高效？關鍵就一個資料結構：**Redis 的 sorted set（ZSET）**（2026-06，BullMQ 官方文件）。

sorted set 是 Redis 裡一種「每個成員都帶一個 score（分數），整個集合**按 score 排序**」的結構。Bull/BullMQ 的招式是：

1. 你排一則「在時刻 T 發送」的訊息，它**不會**掛一個 timer，而是往一個專門的 delayed sorted set 裡塞一筆記錄：**成員 = 這個 job 的資料（或它的 ID）、score = T 的 Unix timestamp（毫秒）**。
2. 整個 delayed set 因此**永遠按「該執行的時間」排好序**——最早該跑的在最前面。Redis 的 sorted set 插入是 O(log N)，所以就算裡面有幾萬筆，塞一筆也飛快。
3. 一個 **delayed-job checker / worker** 週期性地問 Redis：「**把所有 score ≤ 現在這個時間戳的成員給我**」（`ZRANGEBYSCORE` 這類查詢，配合 marker 機制決定下次該醒來的時間）。Redis 評估這個 sorted set，**只回傳已到期的那些**。
4. worker 把這些到期的 job 從 delayed set **搬到 waiting（待處理）佇列**，交給處理邏輯真正去發訊息。

排一則「在時刻 T 發送」的訊息：不掛 timer，往 delayed sorted set 塞一筆
（score = T 的 Unix timestamp、member = job 資料／ID），整個集合永遠按執行時間排序：

```
  Redis sorted set "delayed"（永遠按 score 排序，最早到期的在最前）
  ┌──────────────────────────────────────────────────┐
  │  score=1718760000000  member=job:msg-C  1 小時後  │ ◄ 最早
  │  score=1718780000000  member=job:msg-B  明早 9 點 │
  │  score=1718900000000  member=job:msg-A  3 天後    │ ◄ 最晚
  └────────────────────────┬─────────────────────────┘
                           │  checker 週期性問 Redis：
                           │  「score ≤ 現在的給我」（ZRANGEBYSCORE）
                           ▼
  ┌──────────────────────────────────────────────────┐
  │  到期的 job 被搬到 waiting 佇列                     │
  │  → worker 撈出來、真正發訊息                        │
  └──────────────────────────────────────────────────┘
```

對照一下 `setTimeout` 版，你會發現這招漂亮在哪：

- **不掛 timer**——所以沒有 libuv 計時器堆積，沒有 closure 抓著一坨東西常駐你的行程記憶體。pending 全在 Redis 那一筆 sorted set 記錄裡。
- **以時間為 score 排序**——所以 worker 不用「掃過所有 pending 看誰到期」，只要問 Redis 要「score ≤ 現在」的前段，O(log N) 定位。
- **worker 輪詢到期，而不是被 timer 喚醒**——這就是為什麼它能**重啟存活**：worker 死了重開，它做的第一件事就是再去問 Redis「有誰到期了」，而 Redis 裡那筆 sorted set 記錄**從來沒因為 worker 死掉而消失**。狀態在 Redis，不在 worker。

注意第三點裡有一個你大概沒留意的取捨：worker 是**輪詢（polling）**的——它每隔一段時間（BullMQ 預設的 delayed checker 約 5 秒一次，並用 marker 機制動態調整下次該醒來的時間）去看一眼。所以一則排「9:00:00 整發」的訊息，實際可能在 9:00:0X 才被撈出來處理。跟 setTimeout 一樣，**它也不是毫秒級準時**——但對「排程訊息」這種需求，幾秒的誤差完全在容忍範圍，這個取捨換來的是持久性與記憶體效率，划算得很。

## 故障模式與防禦

重構解開了 setTimeout 的鐐銬，但 job queue **不是免費的持久性**。它把問題搬了家，新家有新的故障模式——而且其中最危險的那個，正是你當年大概以為「換了套件就自動解決」、實際上只是被你**運氣好繞過去**的。這一段我們把它們攤開。

### 故障一：「重啟存活」是一句有前提的承諾——Redis 沒設持久化照樣掉資料

這是本章最該讓你停下來的一條，因為它直接打臉「把狀態存到 Redis 所以重啟還在」這個你大概一直深信不疑的結論。

job queue 把排程狀態寫進了 Redis。但 **Redis 自己也是個會掉資料的東西**。Redis 的持久化有兩種（深掘見 ch17）：RDB（週期性快照）和 AOF（append-only log，把每個寫指令記下來）。關鍵事實（2026-06，BullMQ 官方 going-to-production 文件）：

> 預設只開 RDB 快照的 Redis，在 crash 時**可能丟掉「上次快照之後」的所有資料**——也就是好幾分鐘的 pending job 一起沒。BullMQ 官方因此**建議生產環境開 AOF**（每秒 fsync 一次，crash 最多丟約 1 秒的資料）。

把這條跟你的重構放一起看，問題就很尖銳了：**如果你當年只是「把 Bull 接上一個 Redis」，沒有特別去確認那個 Redis 的持久化設定，那麼你的「重啟存活」可能只對「你的 Node 行程重啟」成立，對「Redis 重啟 / Redis 機器 crash」並不成立。** 你以為你解決了持久性，其實你只是把持久性問題**從你的行程搬到了 Redis**——而 Redis 預設的持久性也不是滿分。

| | setTimeout 版 | job queue 版（Redis 預設 RDB） | job queue 版（Redis 開 AOF） |
|---|---|---|---|
| Node 行程 crash | **全丟** | 存活 | 存活 |
| Redis crash | 不適用 | **丟上次快照後的 pending** | 最多丟約 1 秒 |
| 持久性是否「真的解決」 | 否 | **部分**（運氣好才滿分） | 是（接受 ~1s 視窗） |

> **我以為 vs 實際**：你以為「狀態搬到 Redis = 持久性解決」。實際上你把持久性問題**從『行程記憶體會不會丟』換成了『Redis 持久化設對沒』**。問題沒消失，只是換了個地方問。防禦很直接：**生產環境的 job queue Redis 一定要開 AOF**，並且把「Redis 持久化設定」當成這個功能的一部分去 review、去監控——而不是當成一個跟你無關的基礎設施細節。

### 故障二：at-least-once——重啟、worker 被 kill，同一則訊息可能發兩次

這是你解開 setTimeout 鐐銬時，**套上的那個新鐐銬**。

job queue 為了「不丟」，採的是 **at-least-once** 語意——它寧可重複也不漏。具體會在哪裡重複？

- **worker 處理到一半被 kill**：worker 撈了一個 job、正在發訊息，這時行程被 kill（部署、OOM）。它沒來得及把這個 job 標記成「完成」。BullMQ 的機制是把這種 job 視為 **stalled（卡住）**，等新 worker 上線後**自動重新處理**（預設等待約 30 秒，2026-06 BullMQ 文件）。如果它**第一次其實已經把訊息發出去了、只是還沒回報完成就被 kill**——那麼重新處理時，**訊息會被發第二次**。
- **重試**：處理失敗（比如發訊息的下游暫時掛了）配了重試，重試本身就是「再做一次」。

所以同一則排程訊息，在「worker 死於發送和回報之間」這個時間窗裡，**會被發兩次**。這不是 bug，是 at-least-once 的**設計選擇**——它選擇了「寧可重複也不漏」，而你（在做即時通訊、訊息掉了使用者會抱怨的場景）多半同意這個選擇。

但這就把你**送回了 ch04 那條老安全帶**：要嘛接受偶爾重複，要嘛讓「發送這則訊息」這個動作**冪等**——用排程訊息的某個穩定 ID（不是 job 的 message ID，理由跟 ch04 一樣：那只擋傳輸層的重複）去判斷「這則已經發過了就別再發」。

> **判斷題（留給你在「紙上推演」算）**：「delayed job 重啟後重跑了一次」是 bug 還是預期行為？答案取決於你有沒有做冪等。沒做 = bug（使用者收到兩則）；做了 = 預期且無害（第二次被冪等擋掉）。交付語意全論在 ch10，這裡只標出這條線。

### 故障三：錯過的排程要不要補跑——一個沒有「正確答案」的策略題

第三個故障，是時間軸特有的。

假設 worker / Redis / 整個服務**剛好在某則訊息該發的時刻當機了**，幾分鐘後恢復。那則訊息的執行時間已經**過去了**。現在它躺在 delayed set 裡，score（執行時間）已經小於現在——checker 一問「score ≤ 現在的給我」，它當然會被撈出來。問題是：**現在補發，對嗎？**

- 「生日 9:00 的祝福」延到 9:05 補發——大概可以。
- 「限時活動開始的廣播」活動已經結束了才補發——**反而製造困惑**，不如不發。
- 「每小時提醒」漏了 3 小時，恢復後**一口氣補發 3 則**——使用者被轟炸，這叫 thundering herd 式的補跑災難。

這跟 ch04 講的 **K8s CronJob 的 `startingDeadlineSeconds`** 是**同一個問題的兩個化身**（landscape §19）：「該在 T 跑的事，T 已經過了，還跑不跑？」CronJob 用 `startingDeadlineSeconds` 給你一個「過期多久就放棄」的窗口；job queue 沒有內建這個語意，得**你自己決定**。

防禦不是「補」或「不補」二選一，是**為每一類排程定義它的補跑語意**：

| 排程類型 | 過期後 | 理由 |
|---|---|---|
| 時間敏感、過時即無意義（活動廣播） | **丟棄**（設一個 deadline，過了就不發） | 補發製造困惑 |
| 時間不敏感（提醒、祝福） | **補發一次**（且只補最近一筆，不補積壓的全部） | 晚到比沒到好 |
| 週期性（每小時提醒） | **補一次、跳過積壓** | 避免恢復後一次轟炸 |

這個取捨沒有全域正確答案——它取決於**這則訊息的語意**，而語意是業務決定的，不是技術決定的。深掘（時鐘、時區、單調鐘 vs 牆鐘、補跑的形式化）留給 ch13；這裡你只要記住一件事：**「補跑策略」是排程系統的一個必須顯式設計的決策，不是一個可以靠預設值蒙混過去的細節**。預設不發或預設全補，都會在某類訊息上出事。

### 推演一：手追「排一則 3 天後發送的訊息」在兩版的命運 **[20 分鐘] ★★**

LetsTalk 的使用者在週一中午排了一則「週四中午發」的訊息（延遲 72 小時）。期間服務照常運作——平均每 8 小時部署一次（活躍期）。請分別追這則訊息在 **setTimeout 版** 和 **job queue 版** 的完整命運，並指出它在五個故障問句（交付/持久/分區/並發/觀測）上各自的答案。

提示：先算 72 小時裡有幾個部署窗口；再追每一版在「遇到一次重啟」那一刻發生什麼。

### 推演二：估 10 萬個 pending delayed job 的記憶體帳 **[15 分鐘] ★★**

某個時段，全站累積了 **10 萬個 pending 的排程訊息**（還沒到期）。請估算：

- **setTimeout 版**：這 10 萬個 timer + closure 大致佔多少記憶體、壓在哪裡、最壞會引發什麼。
- **job queue 版**：這 10 萬筆記錄在 Redis sorted set 裡大致佔多少、壓在哪裡。

提示：Redis sorted set 在元素多時用 skiplist + hashtable 編碼，**每個元素內部存兩份**（一份在 hash table、一份在 skiplist）。

### 推演三：列出 setTimeout 版的所有故障模式，逐一對應 job queue 怎麼擊破 **[15 分鐘] ★★**

把本章「四個維度」整理成一張表：每一行一個 setTimeout 的故障模式、它對應五軸的哪一軸、job queue 用什麼機制補上、以及**補上之後有沒有引入新的代價**。

### 推演四：設計「錯過的排程要不要補跑」的策略 **[15 分鐘] ★★★**

為 LetsTalk 的三類排程訊息各定一個補跑策略：(a) 生日祝福；(b) 限時活動開始廣播；(c) 每小時的「喝水提醒」。每類都要回答：過期多久內補？超過就丟還是發？補跑時要不要冪等？並說明你的策略**犧牲了什麼**。

### 推演解答

**推演一解答。** 先算窗口：72 小時 ÷ 8 小時/部署 = **9 個部署窗口**。也就是說這則訊息從排好到該發，要連續躲過 9 次可能殺死它的重啟。

- **setTimeout 版的命運**：使用者排下去的那一刻，Node 行程裡掛了一個延遲 ≈ 2.59 億毫秒的 timer。先別急——**這個延遲（72 小時 ≈ 2.59 億毫秒）還在 24.8 天上限內**，所以不會觸發 overflow 立刻發送的雷（如果排的是「30 天後」就會中那個雷）。timer 安靜地待在 libuv 計時器堆裡。然後第一個 8 小時到了，部署來了，行程重啟——**timer 連同那則訊息一起蒸發**。沒有例外、沒有 log、沒有任何記錄說「有一則訊息沒發成」。使用者週四中午沒收到訊息，系統這邊**毫不知情**。五軸體檢：交付＝靜默掉失；持久＝0（重啟即失憶）；分區＝不適用（單機）；並發＝timer 佔記憶體；觀測＝**0，這是最糟的一格——壞了完全無感**。命運：**這則訊息幾乎 100% 發不出去**（要躲過 9 次部署，機率趨近零）。
- **job queue 版的命運**：使用者排下去，Bull/BullMQ 往 Redis 的 delayed sorted set 塞一筆（score = 週四中午的 timestamp，member = job 資料）。**這筆記錄不在任何 Node 行程裡。** 接下來 9 次部署照常發生——每次 Node 行程重啟，但 Redis 那筆記錄**紋風不動**。worker 重啟後第一件事就是再去問 Redis「有誰到期了」。到了週四中午，checker 一問「score ≤ 現在」，這筆被撈出來、搬到 waiting、worker 發送。五軸體檢：交付＝at-least-once（可能重複，需冪等）；持久＝**存活**（前提 Redis 有 AOF）；分區＝狀態集中在 Redis（單點，見故障一）；並發＝多 worker 可搶同一 job，靠 Redis 原子操作協調；觀測＝**pending 可查可監控**。命運：**正常發出**（前提：Redis 持久化有設好）。

常見錯路：(1) 以為 setTimeout 版「3 天」會中 overflow 立刻發——不會，3 天在 24.8 天內，中雷的是 30 天那種；(2) 以為 job queue 版「100% 沒問題」——它的持久性掛在 Redis 的持久化設定上，沒開 AOF 時 Redis crash 一樣可能掉（故障一）。

**推演二解答。** 先講壓在哪——這是兩版**最本質**的差別。

- **setTimeout 版**：10 萬個 timer + 10 萬個 closure，**全部壓在單一 Node 行程的 heap 上**。一個 closure 抓著 `msg` 物件（訊息內容、收件人、排程時間）加上引用，保守每個算幾百 bytes 到 1–2 KB（看 msg 多大）。10 萬個粗估**幾十 MB 到上百 MB 量級**，全在**一個行程**裡。記憶體本身可能還扛得住，但致命的是**集中度**：它跟其他流量（3,200 並發連線、~10M 訊息/月）共用同一個 heap，一起逼近 OOM；一旦 OOM 被 kill，**10 萬個 pending 一起蒸發**——記憶體問題直接放大成 10 萬則訊息的資料損失。
- **job queue 版**：10 萬筆記錄在 **Redis 的 sorted set** 裡，壓在 **Redis 行程**（跟你的 Node 後端**不同行程、多半不同機器**）。Redis 在元素多時用 skiplist + hashtable 編碼，**每個元素內部存兩份**（hashtable 給 O(1) 查、skiplist 給 O(log N) 範圍查），粗估每個 (score, member) 約 **100–136 bytes + member 本身大小**（2026-06 社群估算；官方參考點：1,000 萬個約 20 bytes 成員的元素約 800 MB–1.2 GB）。10 萬筆（假設 member 是短 job ID）粗估**十幾到幾十 MB**。

關鍵差別**不在絕對數字**（兩邊都不算大），在三件事：(1) **壓在哪裡**——job queue 版把這份記憶體從「跟你的線上流量搶資源的 Node 行程」挪到了「專門幹這個的 Redis 行程」，你的 Node 後端 heap 乾淨了；(2) **可不可控**——Redis 那筆是一個你能查、能設 eviction、能監控大小的資料結構，setTimeout 那堆 timer 你連列都列不出來；(3) **死的時候會不會一起死**——Node 行程 OOM 不會帶走 Redis 裡的記錄，但會帶走所有 setTimeout timer。常見錯路：只比「誰佔的 MB 少」（其實差不多），漏掉「集中在會死的行程 vs 隔離在專責儲存」這個真正重要的維度。

**推演三解答**（範例表）：

| setTimeout 故障模式 | 對應軸 | job queue 怎麼補 | 新引入的代價 |
|---|---|---|---|
| 重啟即失憶（持久性 0） | 持久 | 狀態外部化到 Redis sorted set | 持久性改掛在 Redis 的 AOF 設定上（故障一） |
| 24.8 天硬上限、超過安靜立刻發 | 交付 | 用時間戳當 score，無時長上限 | 無（這條是純贏） |
| timer 堆積佔單一行程記憶體 | 並發/資源 | pending 移到 Redis，不佔 Node heap | Redis 成為新的單點與容量瓶頸 |
| 看 event loop 忙不忙、不準時 | 交付 | worker 輪詢（~5s 粒度），對粗粒度需求夠用 | 仍非毫秒級（但需求本不需要） |
| pending 看不見、無法監控 | 觀測 | pending 在 Redis 可查可告警 | 無（純贏） |
| callback 拋例外就沒了、無重試 | 交付 | 失敗可重試 | 重試 → at-least-once → 需冪等（故障二） |

要點：job queue **不是全贏**。「無時長上限」和「pending 可觀測」是純贏；但「持久性」只是換了個地方掛（Redis），「可重試」直接帶來「可能重複」這個必須用冪等去償還的債。誠實的敘述是：**重構解決了 setTimeout 的致命缺陷，代價是把問題挪進了一個你更能控制、但仍需要正確設定與冪等兜底的新位置。**

**推演四解答。**

- **(a) 生日祝福**：時間不敏感。策略——設一個寬鬆 deadline（比如「過期 6 小時內補發」），超過就丟（隔天補一個生日祝福很怪）。要冪等（避免重啟重發兩則）。**犧牲**：極端情況下（服務掛超過 6 小時）使用者收不到祝福，換來「不會在莫名其妙的時間補發」。
- **(b) 限時活動廣播**：時間極敏感、過時即有害。策略——設**很短**的 deadline（比如「過期 5 分鐘就丟棄，且記一筆告警」），絕不補發過期的。要冪等。**犧牲**：服務恰好在廣播時刻掛掉就**這次廣播沒了**（且要靠告警讓人知道去手動處理），換來「絕不在活動結束後發誤導訊息」。
- **(c) 每小時喝水提醒**：週期性。策略——**只補最近一筆、跳過所有積壓**（漏了 3 小時，恢復後只發 1 則「該喝水了」，不發 3 則）。要冪等。**犧牲**：漏掉的提醒次數，換來「使用者不會被恢復後的一波轟炸惹怒」（避免 thundering herd）。

共通原則：補跑策略**綁定訊息語意**、且每一類都需要冪等前提（因為補跑本質就是「再做一次」）。常見錯路：用一個全域「補/不補」開關套所有類型——對 (b) 補了會誤導，對 (c) 全補會轟炸。深掘（單調鐘 vs 牆鐘、時區、補跑的時間語意形式化）見 ch13。

## 自我檢核

| 軸 | setTimeout 版 | job queue 版 |
|---|---|---|
| 交付 | 靜默掉失；超上限還會提早誤發 | at-least-once（可能重複，需冪等償還，見 ch10） |
| 持久 | **0**——重啟即失憶、零補救 | 存活，**但前提是 Redis 開了 AOF**（沒設好仍可能掉，見 ch17） |
| 分區 | 單機、不適用 | 狀態集中在 Redis（新的單點與容量瓶頸） |
| 並發 | timer 與 closure 堆積壓單一行程 heap，逼近 OOM | pending 移到 Redis sorted set，多 worker 靠 Redis 原子操作搶 job |
| 觀測 | **0**——pending 藏在 libuv heap，列都列不出來、壞了無感 | pending 在 Redis 可查、可數、可告警（從「壞了才知道」變「主動可看」） |

> 一句話帶走：這次重構表面是「setTimeout 換 Bull/BullMQ」，底下是把「未來該做的事」這種需要持久、可靠、可觀測的狀態，從**會隨時死掉的行程記憶體**搬到**行程外的持久儲存**。它解開了 15 分鐘的鐐銬，但沒有消滅問題——只是把持久性問題挪去 Redis（要開 AOF）、把可靠性的代價換成 at-least-once（要冪等兜底，見 ch10）。「在會重啟的系統裡表達時間」這件事的更深一層——時鐘漂移、時區、補跑的時間語意——在 ch13 等你。
