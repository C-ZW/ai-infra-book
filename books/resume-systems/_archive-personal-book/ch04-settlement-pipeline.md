# ch04 — 結算 pipeline：交付語意與跨服務最終一致

> **本章解決什麼問題**：這是 Part II「回到 NeoBards」的第二站。ch03 把你那條「RDS CPU 降 40%」的優化挖到了鎖與交易的機制層——那是**單一資料庫內部**的並發問題。這一章往外走一步：當一筆工作要跨越**四個服務**、經過一條**非同步佇列**、由**可能當機重啟的 consumer** 處理時，你當年靠直覺搭起的那條「CronJob → SQS → 冪等 consumer」結算 pipeline，到底在防什麼？本章把它拆到兩個邊界——**交付保證**（訊息會掉嗎？會重嗎？）與**一致性邊界**（沒有分散式交易，四個服務怎麼最後對得起來？）。它逼出的「at-least-once + 冪等」這條安全帶，到 ch10 會被正式挖穿；它的最終一致問題，到 ch11 接棒。

## 從你已知的出發

你在 NeoBards 做過一條每季都要跑的結算 pipeline。Dynasty Warriors Mobile，500K+ 使用者、10K+ CCU，每到賽季結束就得把公會排名換算成獎勵發下去——每季 **10–30 萬筆**獎勵，跨 **~5,000 個公會**，而且一筆獎勵不是寫一張表就完事：它要動 **guild**（公會層的結算狀態）、**player**（玩家身上的貨幣與點數）、**inventory**（背包裡的道具）、**mail**（站內信，把「你拿到了什麼」寄給玩家）四個服務。

當年你的架構大概是這個形狀：一個 **K8s CronJob** 在賽季結束時被觸發，把整批結算拆成一筆一筆的任務丟進 **SQS**，然後一群 **consumer** 從佇列裡撈任務、逐筆處理、寫那四個服務。你還特別讓 consumer **冪等**——同一筆任務被處理兩次，結果跟處理一次一樣。

你當年為什麼這樣設計，多半講得出工程理由：用佇列把「觸發」和「處理」解耦，consumer 可以水平擴展、慢慢消化 30 萬筆而不會一次打爆下游；冪等是因為「佇列可能重複投遞，保險起見做一下」。這些都對。但我想問你一個你當年大概是**用直覺答對、卻沒真正攤開來想**的問題：

**為什麼那條 pipeline 一定需要冪等？不是「保險起見」，而是「不做就一定會出事」——出在哪一步、機率多大、出了會怎樣？**

還有一個更尷尬的：你大概記得「SQS 是訊息佇列，會保證把訊息送到」。一半對。SQS **標準佇列**的保證是 **at-least-once**——至少一次，意思是**它明確允許同一則訊息被送達多次**，而且**順序只是 best-effort**（見 ch18 收口 SQS 細節）。你那條 pipeline 不是「偶爾」會收到重複訊息，是**架構上保證**會收到。冪等不是保險，是這個架構能成立的**前提**。

這一章就把這條 pipeline 拆到底：訊息在哪一步會被複製、冪等鍵怎麼擋、四個服務在沒有分散式交易的情況下怎麼「最後對得起來」、以及最難看的——**寫了 player 沒寫 inventory** 的時候，這筆獎勵停在半路怎麼收尾。

## pipeline 全貌：一筆獎勵要走過的路

先把整條路畫出來。一筆公會獎勵從「賽季結束」到「玩家信箱收到」，要經過這些關卡：

```
                  ┌──────────────────────────┐
   賽季結束 ─────►│  K8s CronJob（觸發器）   │
                  │  每季跑一次・拆任務        │
                  └─────────────┬────────────┘
                                │ 把 10–30 萬筆拆成
                                │ 一筆一則訊息
                                ▼
                  ┌──────────────────────────┐
                  │  SQS 標準佇列             │
                  │  at-least-once・可能重複   │
                  │  可能亂序・visibility t/o  │
                  └─────────────┬────────────┘
                                │ consumer 撈訊息
                                │（可水平擴展）
                                ▼
                  ┌──────────────────────────┐
                  │  冪等 consumer            │
                  │  以結算 ID 去重           │
                  │  處理一筆獎勵              │
                  └─┬────┬────┬────┬─────────┘
                    │    │    │    │  逐一寫四個服務
            ┌───────┘    │    │    └───────┐
            ▼            ▼    ▼            ▼
        ┌───────┐  ┌────────┐ ┌─────────┐ ┌──────┐
        │ guild │  │ player │ │inventory│ │ mail │
        │ 結算  │  │ 貨幣   │ │ 道具    │ │ 站內信│
        └───────┘  └────────┘ └─────────┘ └──────┘
```

這張圖有三個關卡，每個關卡都在**全書脊椎的五個故障問句**（交付／持久／分區／並發／觀測，見 ch02）上有自己的答案，而且每個關卡都會**漏**：

1. **CronJob 觸發**——它**不保證 exactly-once**（landscape §19）。Kubernetes 官方白紙黑字寫著「在某些情況下，一個 job 可能被建立兩次，或一次都不建立」。所以「每季結算」這個動作本身，可能漏跑、可能重疊跑。
2. **SQS 佇列**——它是 at-least-once（landscape §1）。一則訊息可能被投遞多次，順序是 best-effort。
3. **consumer 寫四個服務**——這四次寫入**不在同一個交易裡**（它們是四個不同服務、多半是不同資料庫）。所以「四個都寫成功」這件事，沒有任何資料庫的原子性替你保證。

接下來逐一拆。

### 關卡一：CronJob 為什麼可能漏跑或重疊

你大概一直把 CronJob 當成「一個可靠的鬧鐘」——時間到了，它就會準時、剛好跑一次。這是老兵記憶裡最常見的一個簡化。

Kubernetes CronJob（landscape §19，`batch/v1`，自 v1.21 GA）的真相是：它由一個 controller 週期性地檢查「現在該不該建立一個新 Job」。這個 controller 自己會重啟、會延遲、會在 API server 壓力下慢半拍。官方文件對「錯過排程」的處理是這樣的：controller 計算自上次排程以來錯過了幾次，**超過 100 次就停止排程並記錄 error**（除非你設了 `startingDeadlineSeconds`，那就改用那個秒數窗口來算）。

對「每季結算」這種**低頻、高價值**的工作，這意味著兩個很實際的風險：

- **漏跑**：賽季結束那一刻，如果 controller 剛好在重啟、或節點壓力大導致觸發延遲，又超過了 `startingDeadlineSeconds` 窗口，這次結算**可能根本沒被建立**。對每天跑一次的 job，漏一次無所謂（明天補）；對**每季一次**的 job，漏一次就是一整季的玩家拿不到獎勵，而且你可能要等到玩家客訴才發現（這就是 ch02 說的「觀測」軸——它壞了你第一時間怎麼知道）。
- **重疊**：`concurrencyPolicy`（`Allow` / `Forbid` / `Replace`）決定「上一次還沒跑完，下一次時間又到了」怎麼辦。如果是 `Allow`（預設），你可能會有**兩個結算 Job 同時在拆任務**——同一批 10–30 萬筆獎勵被**塞進 SQS 兩次**。

第二點特別惡毒，因為它把「重複」這個問題**從 SQS 那一關往上游推**。就算 SQS 一則訊息只投一次，只要 CronJob 重疊跑了一次，每筆獎勵的訊息在佇列裡就有兩份。所以——

> **我以為 vs 實際**：你以為「CronJob 觸發 → 拆任務 → 入 SQS」這條上游是可靠的、重複只可能來自 SQS。實際上，**重複有兩個獨立的來源**：上游的 CronJob 重疊（同一批被拆兩次），和下游的 SQS 重投（同一則被送兩次）。冪等必須同時擋住這兩種。這也是為什麼冪等鍵不能用「SQS 的 message ID」——那只擋得了 SQS 那一關；要擋上游重複，鍵必須來自**結算這件事本身**（哪個賽季、哪個公會、哪個玩家、哪種獎勵），而不是來自傳輸層。

### 關卡二：SQS 為什麼一定會重投——visibility timeout 的機制

這是整章的核心機制，慢一點。

當你的 consumer 從 SQS 撈一則訊息（`ReceiveMessage`），訊息**不會**從佇列裡消失。它變成**對其他 consumer 暫時不可見**（in-flight），這段不可見的時間就是 **visibility timeout**（預設 30 秒，最大 12 小時，2026-06）。這個設計的意圖很單純：給你的 consumer 一段時間去處理；處理完了，你呼叫 `DeleteMessage` 把它真正刪掉。

問題出在「處理完了才刪」這個順序上。在 consumer 呼叫 `DeleteMessage` **之前**，只要發生下面任何一件事——

- consumer **當機**了（OOM、被 K8s 驅逐、節點掛掉）；
- 處理**太慢**，超過了 visibility timeout 還沒刪；
- 網路斷了，`DeleteMessage` 這個呼叫本身**沒送達**；

——SQS 就會在 visibility timeout 到期時，把這則訊息**重新變可見**，讓另一個 consumer 撈走。它**沒辦法分辨**「這個 consumer 死了」和「這個 consumer 只是慢、其實正在寫第三個服務」。對 SQS 來說，這兩種情況看起來一模一樣：時間到了、沒收到刪除指令、那就重投。

這就是 at-least-once 的根：**broker 無法區分「沒處理」和「處理了但 ack 丟了」，所以它的安全選擇永遠是「再投一次」**（這個「ack 丟失的不可區分性」是 exactly-once 交付不可能的根本原因，ch10 會用兩將軍問題的退化版正式證明）。

畫成時序，一筆訊息被投第二次的完整命運是這樣：

```
SQS                       consumer A              consumer B
 │                            │                       │
 │  ReceiveMessage ──────────►│                       │
 │  （visibility timeout 30s）│ 寫 guild ✓            │
 │                            │ 寫 player ✓           │
 │   ⏱ 30s 到期，A 還在寫    │ 寫 inventory…（慢）   │
 │  重新變可見 ──────────────────────────────────────►│
 │                            │ 寫 mail ✓             │ 又寫 guild ✓
 │  ◄── DeleteMessage（A）    │ DeleteMessage ✗       │ 又寫 player ✓ ← 雙發！
 │       （但訊息已被 B 領走）│                       │
 ▼                            ▼                       ▼
```

注意這裡有兩層災難：A 其實**處理成功了**（只是慢），但因為它超過了 visibility timeout，B 也把同一筆處理了一遍。如果四個服務的寫入**不是冪等的**，玩家就拿到了**兩份**獎勵——貨幣加兩次、道具兩份、兩封信。

> **我以為 vs 實際**：你大概記得 visibility timeout 的作用是「避免兩個 consumer 同時處理同一則訊息」。對，但它只在**處理時間 < timeout**時成立。一旦處理超時（或 consumer 慢、卡、當機），它**反而製造了重複**——它沒有任何機制確認 A「到底死了還是只是慢」，所以它的選擇永遠偏向重投。把 visibility timeout 調長能降低重投機率，但**永遠降不到零**，而且調太長會讓「真的當機」的訊息卡很久才被重試（landscape §1：建議值約 6 倍處理時間）。重複不是 bug，是這個交付語意的**結構性後果**。

## 冪等鍵：把「至少一次」收斂成「效果上剛好一次」

既然重複是結構性的、消不掉，那唯一的出路就是：**讓「處理多次」和「處理一次」的結果完全一樣**。這就是冪等（idempotency）。注意這個轉向——我們**不去消滅重複的交付**（那不可能），我們讓重複的交付**無害**。這正是「exactly-once *processing* 是 at-least-once *delivery* + 去重組合出來的幻覺」這句話的真身（ch10 會把這句話挖穿）。

### 冪等鍵怎麼設計

冪等的核心是一把**冪等鍵（idempotency key）**：一個能唯一標識「這筆工作」的字串，讓系統判斷「這筆我是不是已經做過了」。設計這把鍵有三個層次，缺一不可：

**第一層——鍵要來自工作本身，不來自傳輸層。** 前面說過了，不能用 SQS 的 message ID（那只擋 SQS 重投、擋不了 CronJob 重疊）。要用**結算這件事的天然身分**：

```
settlement_key = f"{season_id}:{guild_id}:{player_id}:{reward_type}"
# 例：2026Q2:guild_4821:player_993021:season_rank_gold
```

這把鍵的好處是：不管這筆獎勵的訊息被投幾次、來自上游哪一次重疊，它的鍵**永遠一樣**。同一個玩家、同一季、同一種獎勵，就是同一筆——這是業務語意決定的，不是傳輸決定的。

**第二層——「檢查」和「寫入」必須是同一個不可分割的動作。** 這是新手最容易踩、而且**在壓測時看不出來、上線才爆**的坑。直覺的寫法是：

```
# ❌ 錯誤：check-then-act，有 race window
if not db.exists(settlement_key):     # ← 檢查
    apply_reward(player_id, reward)    # ← 動作（這中間有空隙）
    db.insert(settlement_key)
```

問題在「檢查」和「寫入」之間有一段空隙。在這條 pipeline 裡，consumer 是**並發**的（前面時序圖裡 A 和 B 同時在跑同一筆，這是 ch02 的「並發」軸）。兩個 consumer 可能**同時**通過 `db.exists()` 檢查（兩個都看到「還沒做過」），然後**兩個都** `apply_reward`——玩家還是收到兩份。check-then-act 在並發下根本不冪等。

正確的做法是讓**檢查與宣告權成為一個原子操作**，交給資料庫的**唯一約束**兜底：

```
# ✅ 正確：用唯一約束讓「宣告我做了這筆」成為原子操作
try:
    db.insert(settlement_key)          # settlement_key 是 UNIQUE 欄位
    apply_reward(player_id, reward)    # 只有插入成功者才執行
except UniqueConstraintViolation:
    return                              # 別人已經做了，直接放行、什麼都不做
```

唯一約束是這裡的**最後一道、也是唯一靠得住的防線**。它把「同一把鍵只能被宣告一次」這件事下放到資料庫的並發控制（行鎖／索引唯一性，見 ch03 的鎖機制），由它保證**無論多少個 consumer 同時搶，只有一個 insert 會成功**，其他全部撞唯一約束失敗。PostgreSQL 的 `INSERT ... ON CONFLICT DO NOTHING` 是這個模式的慣用寫法。**先有約束，再談冪等**——只靠應用層的 `if exists` 判斷，在並發下等於沒做。

**第三層——去重記錄與業務寫入要在同一個交易裡（如果可能）。** 想像這個破口：你成功 `insert(settlement_key)`（宣告「我做了」），然後正要 `apply_reward` 時 consumer 當機了。下次重投，這筆鍵已存在 → 被當成「做過了」跳過 → **獎勵其實沒發**。鍵在、事沒做，這是比雙發更難抓的 bug（玩家少拿，通常沒人客訴，靜悄悄）。

理想解是把「寫去重記錄」和「業務寫入」**綁在同一個資料庫交易裡**（搜尋到的 2026 最佳實踐正是這句：record processed message IDs in the same database transaction as the business update）——要嘛兩個都成、要嘛兩個都回滾。但在這條 pipeline 裡有個殘酷的現實：**業務寫入跨了四個服務**，沒有一個交易能同時包住「去重記錄 + guild + player + inventory + mail」。這就把我們逼到下一節——跨服務的最終一致。

### 冪等鍵擋得了什麼、擋不了什麼

把帳算清楚，免得你以為一把鍵就萬事大吉：

| 重複來源 | 冪等鍵擋得了嗎 | 為什麼 |
|---|---|---|
| SQS visibility timeout 到期重投 | ✅ 擋得了 | 鍵一樣，第二次 insert 撞唯一約束 |
| CronJob 重疊跑、同批塞兩次 | ✅ 擋得了 | 鍵來自業務身分，不來自傳輸層 |
| consumer 當機後重投 | ✅ 擋得了 | 只要當機在 insert 之前或之後都被約束接住 |
| 同一玩家「該發兩種不同獎勵」 | ✅ 不誤殺 | 鍵含 `reward_type`，兩種獎勵是兩把鍵 |
| 業務邏輯本身發錯（算錯名次） | ❌ 擋不了 | 冪等只保證「同一筆只做一次」，不保證「這筆是對的」 |
| 鍵粒度設錯（漏了 `season_id`） | ❌ 反而災難 | 下一季同公會同玩家會被當「做過」直接跳過 |

最後兩行是重點：冪等鍵是**正確性的必要條件，不是充分條件**。它只回答「這筆做過沒」，不回答「這筆對不對」。鍵的**粒度**設計錯了（少放一個維度），會從「防重複」變成「跨季漏發」——這比沒做冪等還難發現。

## 跨四服務的最終一致：沒有分散式交易，怎麼最後對得起來

現在來到這條 pipeline 最深的那一層。consumer 處理一筆獎勵要寫四個服務：

```
寫 guild ✓ → 寫 player ✓ → 寫 inventory ✗（這裡掛了）→ 寫 mail（沒輪到）
```

你想用一個交易把四個都包起來，做不到——它們是四個獨立服務、各自的資料庫。**沒有分散式交易**（兩階段提交 2PC 在這種跨服務、跨資料庫、要求高吞吐的場景下又慢又脆，幾乎沒人用）。所以這四次寫入是**四個獨立的本地交易**，中間任何一步都可能失敗，留下一個**部分完成**的狀態：guild 結算了、player 加錢了，但 inventory 沒加道具、mail 沒寄信。

這就是 ch02「分區／一致性」軸的具體形態：**在某個時間窗口內，四個服務看到的世界是不一致的**——guild 以為發完了，inventory 卻什麼都沒收到。你能要的不是「永遠一致」（那要分散式交易），而是**最終一致（eventual consistency）**：允許中間有不一致的窗口，但保證**最後**四個服務會收斂到一致狀態。（最終一致與隔離級別、強一致譜系的完整討論在 ch11；這裡只講這條 pipeline 怎麼**工程上**做到收斂。）

收斂靠三件事：**重試、補償、對帳**。

### 重試：把「部分失敗」變成「重來一次」

最簡單的收斂機制，其實你已經有了——**讓整筆重來**。inventory 寫失敗時，consumer **不要** `DeleteMessage`。訊息會在 visibility timeout 到期後被重投，整筆獎勵被重新處理一遍。

但這裡藏著一個必須講清楚的依賴：**整筆重試之所以安全，正是因為前三步（guild、player）是冪等的**。重投時，guild 和 player 會被「再寫一次」——但因為它們冪等（同一把鍵，第二次撞唯一約束直接放行），所以重寫無害；只有真正還沒成功的 inventory 和 mail 會被補上。

```
第一次投遞：guild ✓  player ✓  inventory ✗  → 不刪，等重投
            （player 此時已經加了錢，但訊息沒刪）
第二次投遞：guild（冪等放行）player（冪等放行）inventory ✓ mail ✓ → 刪
```

**這就是「at-least-once + 冪等」這條安全帶的全部意義**：交付層保證「至少送到」（所以失敗的部分一定有機會重試），冪等保證「重送無害」（所以成功的部分不會被重做）。兩者缺一不可——只有 at-least-once 沒冪等，重試會雙發；只有冪等沒 at-least-once（即 at-most-once），失敗的部分永遠補不回來。這是分散式系統的**預設安全帶**，ch10 會把它確立為全書定理。

但整筆重試有它的代價與邊界：

- **每一步都必須冪等**，不只第一步。任何一步只要不冪等，重試就會在那一步造成重複。
- 如果失敗的是**第四步 mail**、而 mail 本身難以冪等（例如它每次都生成一個新的信件 ID），重投會寄兩封信。所以 mail 的冪等鍵也得綁回 `settlement_key`，不能用隨機信件 ID。
- 如果某一步**永久失敗**（不是暫時性錯誤，而是資料本身有問題，例如玩家帳號已刪除），它會**無限重試到撞上 SQS 的最大接收次數，進死信佇列（DLQ，見 ch18）**——這時就不是重試能解的，要靠下面的對帳和人工介入。

### 補償：當「重來」不可行，就「反做」

重試的前提是「失敗的步驟可以被重做」。但有些情況不行——比方說 player 加錢成功、但 inventory 因為「背包滿了」這種**業務規則**拒絕，而且重試多少次都會被拒。這時你不能無限重試，得**反向收尾**：把已經成功的步驟**撤銷**。這就是 **saga 模式**的補償交易（compensating transaction）：一個跨服務流程被拆成一串本地交易，每個本地交易都配一個「反做」的補償動作；中途失敗時，就反向觸發前面已成功步驟的補償，把系統退回乾淨狀態。

```
正向：guild結算 → player加錢 → inventory加道具 ✗（背包滿、補不了）
                                    │
                                    ▼  觸發補償，反向回退
補償：           player退錢 ◄── guild取消結算 ◄──┘
```

補償不是真正的 rollback（資料庫的 rollback 是把交易抹掉、彷彿沒發生過；補償是**另一筆正向交易**去抵銷前一筆的效果——player 退錢這件事本身會被記錄下來，不是消失）。這帶來兩個你當年大概沒細想的難處：

- **補償自己也會失敗**。「player 退錢」這步如果也掛了怎麼辦？所以補償動作**自己也必須冪等、也必須可重試**——補償流程是另一條 at-least-once + 冪等的小 pipeline。
- **補償有可見的中間態**。在「player 加了錢」到「player 退了錢」之間，玩家**真的短暫看到過那筆錢**。如果他手快花掉了，補償就更複雜了（要嘛允許負餘額、要嘛這個操作根本不該設計成可補償）。這就是為什麼**能用重試就別用補償**——補償是重試實在做不到時的下策。

> **決策框架**：什麼時候重試、什麼時候補償？判準是「失敗的步驟是不是**最終可成功**的」。暫時性失敗（網路抖、服務重啟、鎖等待）→ 重試（它遲早會成）。永久性業務失敗（背包滿、帳號刪除、規則拒絕）→ 補償或進 DLQ 人工處理（重試一萬次也不會成，只會浪費資源並可能引發 retry storm，見 ch14）。**分不清是哪一種，就是你 consumer 最該加觀測的地方**（ch02 觀測軸）。

### 對帳：承認前兩者都會漏，所以要兜底

這是整套機制裡最務實、也最常被工程師當年忽略的一環。重試會漏（永久失敗的進了 DLQ 就沒人管了）、補償會漏（補償自己也失敗了）。所以你需要一條**獨立於主流程的對帳（reconciliation）**：定期（比方每季結算跑完後）掃一遍，問一個簡單的問題——

> **「guild 標記為已結算的每一筆，player / inventory / mail 是不是都真的有對應的記錄？」**

對帳是一條離線的、批次的、**最終的安全網**。它不在乎中間發生過什麼，只在乎**結果對不對**：找出「guild 說發了、但 inventory 查無此筆」的孤兒記錄，然後補發或告警。它能抓到重試和補償都漏掉的長尾——那些卡在 DLQ、那些補償也失敗、那些因為某個你沒想到的 bug 而靜默不一致的筆數。

對帳之所以重要，是因為它對應 ch02 的**觀測軸**：**「結算發錯了，你第一時間怎麼知道？」** 答案不能是「等玩家客訴」。對帳就是你主動發現不一致的眼睛。一條好的結算 pipeline，對帳不是事後補的補丁，是設計的一部分。

把三者擺在一起看它們的分工：

| 機制 | 處理什麼 | 時機 | 漏掉的交給誰 |
|---|---|---|---|
| 重試 | 暫時性失敗（網路、慢、重啟） | 即時（visibility timeout 重投） | 永久失敗 → 補償 / DLQ |
| 補償 | 永久性業務失敗（規則拒絕） | 流程內反向回退 | 補償自己也失敗 → 對帳 |
| 對帳 | 前兩者都漏掉的長尾不一致 | 離線批次（事後掃描） | 人工介入 + 告警 |

## 故障模式與防禦

這條 pipeline 會怎麼壞、壞了長什麼樣、怎麼防。至少一個綁定你真實的 NeoBards 系統。

| 故障 | 壞了長什麼樣（可觀測徵兆） | 怎麼防 |
|---|---|---|
| **同一筆獎勵雙發**（SQS 重投 + consumer 非冪等，**你的系統**） | 玩家貨幣翻倍、收到兩封結算信、客服收到「我怎麼有兩份」的回報；對帳查出 player 餘額與 settlement 筆數對不上 | 以 `settlement_key`（季+公會+玩家+獎勵類型）為唯一約束，`INSERT ON CONFLICT DO NOTHING` 原子去重；**禁用 check-then-act** |
| **CronJob 漏跑**（觸發延遲超過 `startingDeadlineSeconds`，**你的系統**） | 整季玩家都沒收到獎勵、SQS 完全沒有流量、結算 dashboard 一片空白 | 監控「結算 Job 是否在預期時間建立」的 metric；對「每季一次」這種低頻高價值 job 加獨立的「該跑沒跑」告警，不靠 CronJob 自報 |
| **CronJob 重疊**（`concurrencyPolicy: Allow`，同批塞兩次） | SQS 訊息數暴增約一倍、consumer 工作量翻倍但因冪等不雙發、只是浪費算力 | 設 `concurrencyPolicy: Forbid`；同時依賴 `settlement_key` 冪等兜底（防禦不能只有一層） |
| **部分失敗滯留**（寫了 player 沒寫 inventory，未重投也未補償） | 玩家有錢沒道具、或有道具沒信；guild 標記已結算但下游查無記錄；**沒有對帳就永遠靜默** | 失敗時不 `DeleteMessage`（讓它重投）；永久失敗進 DLQ；上層加對帳掃孤兒記錄 |
| **冪等鍵粒度錯**（漏了 `season_id`） | 下一季同公會同玩家的獎勵被當「做過」**靜默跳過**——玩家少拿、無錯誤日誌、無人客訴 | 鍵必須含**所有區分維度**；上線前用「同玩家連兩季」的測試案例驗證鍵不會誤撞 |
| **DLQ 黑洞**（訊息進了死信佇列但沒人看） | 一批永久失敗的結算靜靜躺在 DLQ，玩家少拿、無人發現 | DLQ 深度本身就要告警（深度 > 0 即異常）；DLQ 不是垃圾桶，是「需要人看」的收件匣 |
| **去重記錄寫了、業務沒寫**（insert 後 consumer 當機在 apply 之前） | 鍵在、獎勵沒發；重投被當「做過」跳過——又是靜默少拿 | 盡量把去重記錄與業務寫入綁同一交易；跨服務做不到時，靠對帳兜底抓「鍵在但下游無記錄」 |

注意這張表裡最危險的不是「雙發」——雙發吵、會被客訴、容易發現。最危險的是那幾個**靜默少拿**（冪等鍵粒度錯、DLQ 黑洞、去重記錄寫了業務沒寫）：玩家少拿獎勵通常不會大規模客訴，沒有錯誤日誌，沒有告警——它們只能靠**對帳**這隻主動的眼睛抓出來。這正是 ch02 觀測軸的核心主張：**觀測不是事後補的，是設計的一部分**。通用的「靜默故障為什麼最致命」分類見《正常意外》（fail）。

## 紙上推演

### 推演一：手追一筆雙發的完整命運 **[20 分鐘]** ★★

賽季結束，CronJob 把 `2026Q2:guild_4821:player_993021:season_rank_gold` 這筆獎勵（5,000 金幣 + 1 件傳說武器 + 1 封信）拆成一則 SQS 訊息。consumer A 撈到它，visibility timeout 設 30 秒。A 寫完 guild、player（玩家此刻帳上多了 5,000 金幣），正在寫 inventory 時卡住了——下游 inventory 服務剛好在做一次慢查詢，A 等了 35 秒。

**問**：(a) 第 30 秒發生了什麼？(b) 如果 consumer **沒有**冪等，玩家最後帳上有多少金幣、幾件武器、幾封信？(c) 如果有冪等（`settlement_key` 唯一約束），同樣的時序下玩家最後拿到什麼？(d) 假設這季有 20 萬筆獎勵、每筆訊息有 0.1% 的機率因為這類超時而被重投一次，有幾筆會「進入雙發風險」？沒冪等的話幾筆真的雙發？

### 推演二：判斷「改用 SQS FIFO 能不能讓我不用自己做冪等」 **[15 分鐘]** ★★★

你的同事說：「既然 SQS 標準佇列會重複，我們改用 SQS FIFO 不就好了？FIFO 是 exactly-once，這樣 consumer 就不用自己做冪等了。」

**問**：這個論點對嗎？分三點回答——(a) FIFO 的「exactly-once」精確指什麼、它的去重靠什麼、窗口多長？(b) 就算 SQS 那一關不重複了，這條 pipeline 還有沒有別的重複來源？(c) 換 FIFO 要付什麼代價？最後給一句話結論：FIFO 該不該換、冪等該不該留。

### 推演三：畫「寫了兩個服務、第三個失敗」的補償流程 **[20 分鐘]** ★★

一筆獎勵的處理順序是 guild → player → inventory → mail。這次 guild ✓、player ✓，但 inventory 回報「玩家背包已滿，無法加入道具」——而且這是**業務規則**，重試一百次都會是同樣結果。

**問**：(a) 這該走重試還是補償？為什麼？(b) 畫出補償流程（哪些步驟要反做、反做的順序、每步反做失敗了怎麼辦）。(c) 指出補償流程裡一個「玩家可能看到的尷尬中間態」，並說它為什麼難處理。

### 推演解答

**推演一解答。**

(a) 第 30 秒，A 的 visibility timeout 到期。SQS **無法分辨** A 是死了還是只是慢——它只知道「30 秒到了、沒收到 `DeleteMessage`」，於是把這則訊息**重新變可見**。consumer B（或同一個 consumer 的下一輪輪詢）撈走它，開始**從頭**處理這筆獎勵：寫 guild、寫 player……

(b) **沒有冪等**：A 那邊還在（或剛寫完 inventory、mail，然後試圖 `DeleteMessage` 卻發現訊息已被 B 領走）。B 從頭跑一遍：guild 再結算一次、player **再加 5,000 金幣**、inventory **再加一件武器**、mail **再寄一封信**。最後玩家帳上：**10,000 金幣、2 件傳說武器、2 封信**。雙發。

(c) **有冪等**：B 從頭跑，第一件事是 `INSERT settlement_key`——但這把鍵在 A 第一次處理時（或 A 處理的某個點）已經插進去了，所以 B 的 insert **撞唯一約束失敗**，consumer 捕捉到 `UniqueConstraintViolation` 後**直接放行、什麼都不做**。玩家帳上維持：**5,000 金幣、1 件武器、1 封信**。（細節取決於去重記錄插在哪個時點——這正是推演的延伸思考：如果 insert 在四步**之前**做，B 會完全跳過、但要靠重試補上 A 可能沒寫完的 inventory；如果 insert 與業務寫入綁同一交易，最乾淨。）

(d) 20 萬筆 × 0.1% = **200 筆進入雙發風險**（被重投至少一次）。沒冪等的話，這 200 筆**全部真的雙發**——200 個玩家拿到雙倍獎勵。對一款 500K+ 使用者的遊戲，200 個玩家發雙倍貨幣，足以引發經濟系統失衡的客訴與通膨。有冪等的話，這 200 筆被重投但**零雙發**。這就是為什麼冪等在這個量級不是「保險」是「前提」——0.1% 聽起來很小，但乘上 20 萬就是 200 個真實受影響的玩家，而且這還只是「超時重投」一個來源，沒算 CronJob 重疊。

**推演二解答。**

(a) FIFO 的 exactly-once 是 **exactly-once *processing*，靠 5 分鐘的 deduplication window**（landscape §1）。它的機制是：在這 5 分鐘內，帶相同 `MessageDeduplicationId` 的訊息會被 SQS 視為重複、只投一次。注意兩個限制：它是「處理」層的去重不是「交付」層的魔法（它本質上就是 SQS 替你做了一層冪等），而且窗口**只有 5 分鐘**——超過 5 分鐘的重複（比方一則訊息卡在某處 10 分鐘後才重新出現）它就擋不住了。

(b) **還有別的重複來源**。就算 SQS FIFO 在 5 分鐘內不重複，這條 pipeline 還有：**CronJob 重疊**（上游同一批被拆兩次，可能間隔遠超 5 分鐘，FIFO dedup 窗口擋不到）；以及**超過 5 分鐘的延遲重投**。更根本的是，FIFO 只保證「SQS 到 consumer 這一段」不重複，但它**保證不了** consumer 寫四個服務時的部分失敗與重試——你還是需要重試，重試還是需要冪等。

(c) FIFO 的代價：**吞吐上限**（預設每 partition 每秒 300 非批次／3,000 批次，landscape §1）。你這季要塞 30 萬筆，FIFO 的嚴格排序與去重會成為吞吐瓶頸；而且**你根本不需要排序**——結算誰先誰後無所謂。為了不需要的排序保證，付吞吐的代價，不划算。

**一句話結論**：FIFO 不該換（你不需要排序、它的吞吐上限是純損失、5 分鐘窗口擋不住 CronJob 重疊）；冪等不該留——是**必須**留。FIFO 的 dedup 只是「SQS 內建的一層冪等」，它取代不了你**業務層**用 `settlement_key` 做的冪等，因為真正的重複來源（CronJob 重疊、跨服務重試）在 SQS 之外。**把冪等交給業務層，SQS 用最便宜的標準佇列就好。**

**推演三解答。**

(a) **走補償**。判準是「失敗的步驟是不是最終可成功的」——「背包已滿」是**業務規則**，不是暫時性故障，重試一百次都會被同樣拒絕。無限重試只會浪費資源、可能引發 retry storm（ch14），而且這筆永遠卡著。所以要反向收尾。

(b) 補償流程（反向、逆序）：

```
正向已完成：guild結算 ✓ → player加5000金幣 ✓ → inventory加武器 ✗（背包滿）
                                                       │
                                                       ▼ 觸發補償
補償（逆序）：  player扣回5000金幣 ◄── guild取消結算標記 ◄──┘
```

反做順序與正向相反（先補償最後成功的 player，再補償更早的 guild）。每步反做**自己也要冪等、可重試**——若「player 扣回」這步失敗，補償流程本身要能重投重做；若連補償都反覆失敗，最後落到**對帳 + 人工介入**（記一筆「需人工處理的不一致」，告警）。

(c) **尷尬中間態**：在「player 加了 5,000 金幣」到「player 扣回 5,000 金幣」之間，玩家帳上**真的短暫多了 5,000 金幣**，而且他可能**已經把這筆錢花掉了**（買了東西）。這時補償要扣回 5,000，帳就可能變負，或扣不回來。這難處理是因為**補償不是 rollback**——加錢這個動作在玩家視角是「真的發生過、且可被觀察與利用」的，不像資料庫交易回滾那樣彷彿沒發生。處理方式要嘛允許短暫負餘額、要嘛這類「會被立即消費的副作用」根本不該設計成「先做後補償」，而該設計成「全部成功才對玩家可見」（例如先寫一個 pending 狀態，四步全成才翻成 visible）。**這就是為什麼「能用重試就別用補償」**——補償一旦碰到「已對外可見且可被消費的副作用」，乾淨收尾就變得很難。

2. 用 visibility timeout 的機制解釋：為什麼 SQS「無法分辨 consumer 死了還是只是慢」，而這個無法分辨**直接導致**重投？
3. 為什麼 `if not exists: do()` 這種寫法在並發 consumer 下**不冪等**？唯一約束如何把它修好——修的是哪一個「同時」的破口？
4. 「at-least-once + 冪等」這條安全帶，少了任何一半會怎樣？（分別講「只有 at-least-once 沒冪等」和「只有冪等沒 at-least-once」各會出什麼事。）
5. 重試、補償、對帳三者的分工是什麼？什麼樣的失敗該走哪一個？分不清是哪種失敗時，你該在 consumer 哪裡加觀測？
6. 為什麼「靜默少拿」比「雙發」更危險？這條 pipeline 裡哪幾個故障會造成靜默少拿，它們只能靠什麼抓出來？
7. 把這條 pipeline 拿去做一次**五軸體檢**（交付／持久／分區／並發／觀測），每軸你的答案是什麼、代價是什麼？（見章末小表，先自己講一遍再對。）
8. 當年你說「保險起見讓 consumer 冪等」——現在你會怎麼重講這句話，讓它聽起來像「這個架構的前提」而不是「順手加的保險」？

## 延伸閱讀

- **Amazon SQS — Visibility timeout**（[AWS docs](https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/sqs-visibility-timeout.html)）：讀「at-least-once delivery」與「12 小時上限不因延長而重置」兩段——這是「SQS 一定會重投」的官方根據，也是 visibility timeout 機制的權威說明。
- **Amazon SQS — Preventing duplicate processing**（[AWS docs](https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/avoding-processing-duplicates-in-multiple-producer-consumer-system.html)）：AWS 自己怎麼建議在多生產者／多消費者下做去重——對照你的 `settlement_key` 設計。
- **Kubernetes — CronJob**（[k8s docs](https://kubernetes.io/docs/concepts/workloads/controllers/cron-jobs/)）：讀「CronJob limitations」那段——官方明說「可能建立兩次或一次都不建立」，並解釋 `concurrencyPolicy` 與 `startingDeadlineSeconds`。這是「CronJob 不保證 exactly-once」的一手出處（landscape §19）。
- **Saga choreography pattern**（[AWS Prescriptive Guidance](https://docs.aws.amazon.com/prescriptive-guidance/latest/cloud-design-patterns/saga-choreography.html)）：補償交易與跨服務最終一致的權威範式說明，對照本章「重試 / 補償 / 對帳」的工程拆解。
- **Saga pattern**（[Microsoft Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/patterns/saga)）：把「為什麼不能用一個分散式交易包住四個服務、所以拆成 saga」講得最清楚的一篇，含 orchestration vs choreography 取捨。

### 本系統五軸體檢（結算 pipeline）

| 軸 | 這條 pipeline 的答案 | 代價 / 邊界 |
|---|---|---|
| **交付** | SQS 標準佇列 at-least-once、可能重複、best-effort 排序；靠 `settlement_key` 冪等收斂成「效果上剛好一次」 | 重複是結構性的、消不掉，只能讓它無害；冪等是架構前提非保險 |
| **持久** | 訊息持久在 SQS（最大保留 14 天）；去重記錄與業務寫入持久在各服務 DB；consumer 重啟不丟工作（訊息還在佇列） | CronJob 觸發本身不持久——漏跑那一刻沒有任何東西替你記得「該跑沒跑」 |
| **分區** | 無分散式交易；四服務最終一致，靠重試 + 補償 + 對帳收斂；中間有可見的不一致窗口 | 部分失敗會留下短暫不一致；「強一致」要的分散式交易在此場景太貴 |
| **並發** | 多 consumer 並發處理同一筆是常態（visibility timeout 重投造成）；靠唯一約束的原子去重防雙發 | check-then-act 在此必爆；正確性下放到資料庫的並發控制（見 ch03 鎖機制） |
| **觀測** | 對帳是主動發現不一致的眼睛；DLQ 深度、結算 Job「該跑沒跑」、player/settlement 筆數對帳都要告警 | 最危險的「靜默少拿」無錯誤日誌、無客訴，**只有對帳抓得到**——觀測必須是設計的一部分 |
