# B · 一致性、複製與演進

本領域的另一半回答：當資料被**複製到多個副本、被水平切分到多台機器**時，讀到的值還能保證什麼？這個檔含七條互相咬合的條目——先給「一致性光譜」的全景（線性／因果／讀己之寫／最終一致），再展開讀寫複製延遲的實務後果、分片的一致性代價、最終一致下的補償與對帳語意，接著把 CAP／PACELC 與 ACID／BASE 兩組「保證模型框架」並排講清楚，最後落到一個高度實務的 owning 主題：**零停機 schema 遷移**。複製策略、consistent hashing、quorum、共識的**機制本身**在領域 M 深講，本檔只講它們投射到一致性與資料庫操作面的後果；對帳機制屬領域 L，本檔只談一致性模型那一面。

## 一致性光譜

### 定義與原理

「一致性（consistency）」是被嚴重多載的字。在複製系統裡，它問的是一件具體的事：**多個副本上，讀操作能看見哪些寫操作、以什麼順序看見**。這不是一個開關，而是一條從「最強、最貴」到「最弱、最便宜」的**光譜**。第一原理是 Lamport 與後續研究確立的層級：越往上，副本間需要越多協調（coordination），協調＝延遲＋在分區時犧牲可用性。

由強到弱排四個常被引用的點：

- **線性一致性（linearizability）**：每個操作看起來在它的呼叫與回傳之間的某個瞬間「原子地」生效，且這個全域順序與真實時間（real-time）一致。直覺：系統表現得像只有一份資料、所有人即時看到最新寫入。它是**單一物件**上最強的保證。
- **順序一致性（sequential consistency）**：所有行程看到的操作交錯順序相同，且各自保留自己的程式順序——但**不要求與真實時間一致**（你寫完，別人「之後」的讀不保證看到）。比線性弱一級。
- **因果一致性（causal consistency）**：有因果關係的操作（A 影響了 B）所有副本都以一致順序看見；互相獨立（concurrent）的操作各副本可看到不同順序。它捕捉「回覆必在原文之後出現」這類直覺，又不需全域協調，是「能在分區下保持可用」的最強模型。
- **最終一致性（eventual consistency）**：若停止寫入，所有副本**最終**會收斂到同一值——但沒說多久、過程中讀到什麼都可能。最弱、最便宜、可用性最高。

另有一組正交的、以「單一使用者視角」定義的**會話保證（session guarantees）**——讀己之寫、單調讀、單調寫、讀後寫——它們不要求全域強一致，只要求「對同一個 client 自己的動作前後一致」（見「複製延遲與讀己之寫」）。

### 解法空間

光譜上每個點對應一組實作機制：

- **線性一致**：靠共識協定（Raft／Paxos）讓一份 replicated log 全序提交，所有讀寫過 leader 或過 quorum 並校時（見領域 M）。代價最大。
- **順序／因果一致**：因果一致用**版本向量（version vector）／依賴追蹤**標記操作的因果先序，副本依此排序套用；不需全域時鐘。
- **讀己之寫等會話保證**：把同一 client 的讀導向主庫一小段時間、或用「讀至少新到我上次寫的版本」的版本門檻（read-your-writes token）。
- **最終一致**：無主複製（Dynamo 式）＋背景反熵（anti-entropy／gossip）＋衝突合併（LWW／CRDT），寫入只要求少數副本確認、其餘非同步補上（見領域 M、領域 L）。

「可調一致性（tunable consistency）」是現代資料庫的常態：同一個系統，不同操作可選不同強度。

### 各方案的保證與取捨

| 方案/做法 | 保證 | 適用場景 | 注意事項 |
|---|---|---|---|
| 線性一致 | 像單份資料、即時最新、與真實時間一致 | 分散式鎖、leader 選舉、餘額這種不能讀到舊值的單鍵 | 每次操作要過共識/quorum，延遲最高；分區時要犧牲可用性 |
| 順序/因果一致 | 因果順序一致、並發可分歧 | 留言串、社群動態、協作編輯 | 要追蹤因果依賴（版本向量），實作較重；不防並發衝突本身 |
| 讀己之寫（會話） | 同一 client 讀得到自己剛寫的 | 使用者更新個資/貼文後立刻自己看 | 只對「同一 client」成立，換裝置/換 session 可能破功 |
| 最終一致 | 無新寫入終會收斂 | 高可用、可容忍短暫舊值（追蹤數、瀏覽計數） | 收斂時間無上界；需衝突解決策略；讀到舊值要設計成不致命 |

### 何時需要

在光譜上選點的準則是：**這份資料讀到「不是最新值」會造成什麼？**

- **讀到舊值會破壞不變量或損失金錢** → 往線性一致靠（餘額、庫存、唯一性保留）。但別整個系統都拉滿，只對那幾個關鍵單鍵拉。
- **讀到舊值只是「慢半拍」、使用者能接受** → 最終一致足矣（按讚數、觀看數、推薦結果）。
- **使用者要立刻看到自己的動作、但別人慢半拍沒關係** → 讀己之寫這種會話保證最划算，避開全域強一致的成本。
- **需要「回覆不能出現在原文之前」這類順序，但不需即時** → 因果一致。

過度工程的典型：把整個 NoSQL 叢集設成 quorum 強讀只為了「保險」，犧牲了當初選它的可用性與延遲。

### 常見誤解與陷阱

- **把 consistency 一個字當成同一件事**——ACID 的 C（不變量不被破壞）、CAP 的 C（線性一致）、最終「一致」性，是**三個不同概念**。混用會讓討論失焦。
- **以為線性一致＝serializable**——前者是**單一物件**的即時性保證，後者是**多物件交易**的可序列化保證；兩者正交（見「隔離級別與異常」）。一個系統可以線性一致但不可序列化，反之亦然。
- **以為最終一致＝「很快就一致」**——「最終」沒有時間上界；分區或高負載時可長達數秒甚至更久。設計要能容忍這段窗口讀到舊值。
- **把會話保證當成全域保證**——讀己之寫只保證「你讀得到你自己的寫」，不保證別人讀得到你的寫，也不保證你讀得到別人的寫。

## 複製延遲與讀己之寫

### 定義與原理

把讀流量導到 read replica（讀副本）是擴展讀吞吐最常見的手段：主庫（primary）負責寫，多個副本非同步接收 WAL／binlog 並重放，承接讀。代價是**複製延遲（replication lag）**——副本永遠落後主庫一小段時間（毫秒到數秒，負載高時更久）。這直接打破一個使用者最在意的直覺：**「我剛寫的，自己應該讀得到。」** 寫入落在主庫、緊接著的讀卻打到還沒追上的副本，於是讀到舊值或 null，使用者以為「我的更新不見了」。

這就是**讀己之寫（read-your-writes，又稱 read-after-write）**保證要解決的問題：它不要求全域強一致，只要求「同一 client 讀得到自己剛提交的寫」。

### 解法空間

在「主寫、副本讀」架構下達成讀己之寫的辦法：

- **寫後讀主庫一小段時間（sticky-to-primary）**：使用者寫入後，把他接下來幾秒的讀都導回主庫。最簡單，但抵銷了部分讀擴展效益。
- **追蹤寫入版本/位置（read-your-writes token）**：記下這次寫提交後的複製位置（如 LSN／GTID），後續讀要求「副本至少追到這個位置才回應」，否則改打主庫或等待。精準但要傳遞 token。
- **單調讀的副本黏著（pin to one replica）**：把同一 client 黏在同一副本上，至少保證不會「前進又後退」（單調讀），但不保證讀得到主庫最新。
- **關鍵讀全走主庫**：對「寫完馬上要讀」的少數路徑直接標記為強讀、繞過副本。

### 各方案的保證與取捨

| 方案/做法 | 保證 | 適用場景 | 注意事項 |
|---|---|---|---|
| 寫後黏主庫 N 秒 | 該窗口內讀己之寫 | 寫完立刻自看（編輯個資後回看） | N 設太短仍可能 miss；主庫讀壓力回升 |
| 寫入位置 token | 精準讀己之寫 | 需嚴格、又想盡量用副本 | 要在請求間傳遞/儲存 token；跨服務難傳遞 |
| 黏同一副本 | 單調讀（不倒退） | 分頁瀏覽、列表翻頁不抖動 | 不保證看到主庫最新；該副本掛了要重新選 |
| 關鍵讀走主庫 | 該路徑線性一致 | 少數「絕不能舊」的讀 | 路徑分類錯就破功；主庫成瓶頸 |

### 何時需要

- **使用者會「寫完立刻自己看」** → 必須有讀己之寫，否則體驗像「資料丟了」。最典型：貼文、改設定、下單後看訂單。
- **資料只給別人看、寫者不馬上回看** → 不必特別處理，最終一致即可（如後台批次匯入的報表）。
- **跨裝置/跨 session 也要一致** → 單純的 client 黏著不夠（手機寫、電腦讀），要 token 或走主庫。

**Worked example**：某服務讀寫比 9:1，把讀導到 3 個副本後讀吞吐約 ×3。平均複製延遲 200 ms，但 p99 達 1.2 s。若使用者「改完暱稱→立刻刷新個人頁」平均間隔 800 ms，則約有一段尾巴（延遲 > 間隔的請求）會讀到舊暱稱——在 1.2 s 的 p99 下，這類「刷新比複製快」的情況並非罕見邊角。解法不必對全部讀做強一致：只要對「改完→自看」這一條路徑做寫後黏主庫 2 s（覆蓋 p99 的 1.2 s 有餘量），其餘 99% 讀仍享受副本擴展。一個窄路徑的特例，換掉「資料不見了」的客訴。

### 常見誤解與陷阱

- **以為加副本只賺不賠**——你買到讀吞吐，同時買進複製延遲這個新的一致性問題；不設計讀己之寫，就把延遲變成 bug。
- **把「平均延遲很低」當沒問題**——傷害發生在尾巴（p99／p999）。平均 50 ms 但 p99 2 s，受害的就是那批倒楣使用者，而且難以重現。
- **同步複製當成萬靈丹**——改成同步複製（等副本確認才回 client）能消除延遲，但寫延遲與失敗率上升、副本慢就拖垮寫入；通常只對極少數資料這麼做。
- **忽略「寫到主、讀到副本」的跨服務裂縫**——A 服務寫主庫、B 服務讀副本，token 傳不過去，讀己之寫在服務邊界就斷了。

## 分片 partition

### 定義與原理

當單機放不下資料或扛不住寫入，就把一張大表/一個資料集**水平切分（horizontal partitioning，即 sharding／分片）**到多台機器，每台只存一部分列。切分鍵（shard key）決定一筆資料落在哪個分片；路由層（router／coordinator／client-side logic）負責把請求送到正確的分片。第一原理：**分片把「規模」問題換成「跨分片協調」問題**——單分片內你還有資料庫原本的保證（交易、唯一性、線性讀），跨分片你幾乎全部失去，得自己補。

注意術語：本書統一用**分片（shard/sharding）**指這種「資料分散到多節點」的水平切分；**分區（partition）**保留給 CAP 語境裡的網路分區，與 Kafka 的 partition。

### 解法空間

切分鍵與路由策略決定一致性與熱點：

- **範圍分片（range）**：依鍵的區間切（A–M 在 shard1、N–Z 在 shard2）。範圍查詢友善，但易出**熱點**（如以時間為鍵，最新資料全擠在最後一個分片）。
- **雜湊分片（hash）**：對鍵取雜湊再分桶，分佈均勻、無範圍熱點，但範圍查詢要掃所有分片。重新分片時若用樸素 `hash % N`，N 一變幾乎所有鍵都要搬——**consistent hashing** 就是為了讓擴縮容只搬一小部分鍵（機制見領域 M）。
- **目錄/查表分片（directory）**：維護一張「鍵→分片」對照表，最靈活（可手動搬熱點），但對照表本身成為單點與一致性負擔。
- **地理分片（geo）**：按使用者所在區切，降延遲＋符合資料落地法規，代價是跨區操作昂貴。

### 各方案的保證與取捨

| 方案/做法 | 保證 | 適用場景 | 注意事項 |
|---|---|---|---|
| 範圍分片 | 範圍查詢落單分片、有序 | 時序、按帳號區間掃描 | 易熱點（尾端分片）；需監控與再平衡 |
| 雜湊分片 | 分佈均勻、無範圍熱點 | 點查為主、key 基數高 | 範圍查詢要 scatter-gather 所有分片 |
| consistent hashing | 擴縮容只搬約 1/N 的鍵 | 動態增減節點的快取/儲存 | 仍可能傾斜，需 virtual node 平衡（見領域 M） |
| 目錄分片 | 可任意搬鍵、處理熱點靈活 | 少數超大租戶需單獨安置 | 目錄是單點＋一致性熱點；查一次多一跳 |

### 何時需要

- **單機已到天花板**（容量、寫 IOPS、連線數）→ 才分片。分片前先窮盡讀副本、快取、索引最佳化、垂直升級——分片的營運複雜度極高，是最後手段。
- **跨分片交易/JOIN 是常態** → 重新檢討切分鍵；好的切分鍵讓**絕大多數操作落在單一分片**（如以 tenant_id 分片，單租戶操作天然不跨片）。
- **熱點集中在少數鍵** → 範圍分片要避開，或對熱鍵特殊處理。

**Worked example**：一張 20 億列的訂單表，單機寫入到頂。以 `user_id` 雜湊分到 16 個分片，平均每片 1.25 億列、寫壓力 ÷16。代價：原本一句 `SELECT ... WHERE user_id=?` 仍落單片（因為切分鍵就是 user_id），但「全站訂單總額」這種跨所有 user 的彙總，從一句 SQL 變成對 16 片各查一次再合併（scatter-gather）。若改用 `order_date` 範圍分片，最新月份的寫入會全擠在最後一片，16 片裡 15 片閒置——切分鍵選錯，分片帶來的是傾斜而非均衡。選 user_id 而非 order_date，差別就是「寫壓力 ÷16」與「寫壓力全壓一片」。

### 常見誤解與陷阱

- **太早分片**——分片是不可逆的複雜度躍升（跨片交易、再平衡、運維）。很多「需要分片」的系統其實只是缺索引或缺快取。
- **切分鍵選錯導致跨片操作變常態**——若最熱的查詢/交易頻繁跨多片，等於把分片的成本全付了、好處全沒拿到。切分鍵應對齊**最高頻、最在意延遲**的存取模式。
- **以為分片後還能用 DB 的唯一約束跨片保證唯一**——`UNIQUE` 只在單分片內有效；全域唯一要另設（如中央發號器或在切分鍵內含唯一性）。
- **用 `hash % N` 當長久方案**——擴容時 N 一變幾乎全表要搬，期間一致性與可用性都受創；要動態增減節點就上 consistent hashing（見領域 M）。

## 最終一致 · 補償 · 對帳

### 定義與原理

選了最終一致性（為了高可用＋低延遲），就接受了一個事實：**在收斂之前，系統會短暫處於「不一致」狀態**——兩個副本、或兩個服務的資料，在某個瞬間彼此矛盾。這不是 bug，是這個一致性模型的定義。問題不在「如何避免不一致」（避不掉），而在「如何讓系統在不一致的窗口裡仍正確運作、並保證最終收斂」。本條只談這個**一致性模型面**：補償（compensation）與對帳（reconciliation）作為「最終達成一致」的兩種手段在語意上意味什麼。對帳的**具體機制**（怎麼比對兩邊、漂移怎麼修）屬領域 L，本條只講它在一致性模型裡扮演的角色。

第一原理：當沒有跨資源的原子交易（跨服務、跨庫），你無法讓「兩件事一起成功或一起失敗」。於是改用**先各自做、事後修正**——前向用補償抵銷失敗的半成品，後向用對帳偵測並修補漂移。

### 解法空間

讓最終一致系統「最終真的一致」的手段：

- **補償交易（compensating transaction）**：無法回滾的跨服務操作，用一個「語意上的反向操作」抵銷（訂了位→取消訂位；扣了款→退款）。這是 saga 的核心（機制見 outbox／saga，領域 A）；本條只強調它是「達成最終一致的前向修正」。
- **對帳（reconciliation）**：定期把兩邊（或多邊）的真實狀態拉出來比對，找出漂移（drift）的記錄，依規則修正。它是「最終一致」這個承諾的**兌現機制與安全網**——不管中間漏了什麼，對帳保證偏差被發現並收斂（機制見對帳 reconciliation，領域 L）。
- **收斂式合併（convergent merge）**：副本各自接受寫入，靠 LWW／版本向量／CRDT 在合併時無衝突地收斂到同一值（見衝突解決，領域 L）。
- **讀修復（read repair）／反熵（anti-entropy）**：讀到副本不一致時順手修，或背景持續比對補齊（見領域 M）。

### 各方案的保證與取捨

| 方案/做法 | 保證 | 適用場景 | 注意事項 |
|---|---|---|---|
| 補償交易 | 失敗的跨服務操作被語意抵銷 | 訂位/扣款這類可語意反向的流程 | 補償本身可能失敗，要可重試/冪等；有「已讀到中間態」的窗口 |
| 對帳 | 漂移最終被偵測並修正 | 任何跨系統的金額/數量需可稽核 | 對帳週期＝最大不一致窗口；要有可信「真相源」 |
| 收斂合併（CRDT/LWW） | 並發寫無衝突收斂同值 | 計數器、購物車、協作狀態 | LWW 會默默丟資料；CRDT 限定資料型別、metadata 成本 |
| 讀修復/反熵 | 副本間差異被背景補齊 | 無主複製叢集 | 只修被讀到/被掃到的；冷資料漂移可長期潛伏 |

### 何時需要

- **跨服務/跨庫、又不能上分散式交易（2PC）** → 用補償＋對帳這對組合：補償管前向回滾、對帳當最後安全網。幾乎所有微服務金流都這樣。
- **資料涉及金額/數量、需可稽核** → 對帳是**非選項**，是底線。再強的一致性設計都可能有縫，對帳是兜底。
- **資料只是衍生/可重算** → 多數時候直接重算比對帳便宜，不必補償。

**Worked example**：一個下單流程跨「訂單服務」「庫存服務」「支付服務」三庫，無分散式交易。一天 10 萬筆訂單，假設每筆三步、每步失敗率 0.1%，則約 `100,000 × (1 − 0.999³) ≈ 100,000 × 0.3% ≈ 300` 筆會卡在「扣了庫存但沒扣款」或「扣了款但訂單沒建立」的中間態。前向靠 saga 補償自動回滾大部分；剩下補償也失敗的零頭（再 0.1%，約 0.3 筆/天）由每日對帳掃出來人工或自動修。沒有對帳，這 0.3 筆/天會無聲累積成帳目對不上的長期漂移；補償＋對帳兩層，把「300 筆異常/天」收斂到「最終全部一致」。

### 常見誤解與陷阱

- **以為補償＝rollback**——資料庫 rollback 是「當沒發生過」；補償是「再做一個反向操作」，中間態**對外可見過**（別人可能已讀到「已訂位」又看到「已取消」）。
- **沒有對帳就宣稱最終一致**——「最終一致」不會自己發生，它需要反熵/對帳這類機制兜底。少了它，不一致可能永久潛伏（尤其冷資料）。
- **對帳週期當免費**——對帳週期就是你的**最大不一致窗口**。一天對一次，最壞情況帳就錯一天；金流場景可能要近即時對帳。
- **LWW 默默丟資料卻沒人知道**——last-write-wins 在並發寫時用時間戳裁決，輸的那筆**直接消失**，且時鐘偏移會讓「贏家」不直觀（見衝突解決，領域 L）。

## CAP / PACELC

### 定義與原理

CAP 定理回答一個分散式系統避不掉的取捨。三個屬性：**一致性（Consistency，此處指線性一致）**、**可用性（Availability，每個請求都得到非錯誤回應）**、**分區容忍（Partition tolerance，網路被切開時系統仍運作）**。CAP 由 **Eric Brewer** 在 **2000 年 PODC keynote** 以 conjecture（猜想）提出，再由 **Seth Gilbert 與 Nancy Lynch 於 2002 年**以反證法形式化證明為定理（論文〈Brewer's Conjecture and the Feasibility of Consistent, Available, Partition-Tolerant Web Services〉，ACM SIGACT News）。

常被誤念成「三選二」。更準確的讀法：**網路分區（P）在分散式系統裡不是選項而是現實**（封包會丟、節點會失聯），所以真正的選擇只發生在**分區當下**——要嘛保一致性（拒絕在可能不一致的副本上回應，犧牲 A），要嘛保可用性（照常回應，可能回舊值，犧牲 C）。沒分區時 CAP 沒說話。

**PACELC** 補上 CAP 沒說的那一半。由 **Daniel Abadi** 提出（**2010 年部落格初提、2012 年 IEEE Computer 論文**〈Consistency Tradeoffs in Modern Distributed Database System Design: CAP is Only Part of the Story〉形式化）。它說：**if Partition（分區時）→ 在 Availability 與 Consistency 間選；Else（正常運作時）→ 在 Latency 與 Consistency 間選。** 關鍵洞見：即使沒有分區，強一致仍要付協調延遲的代價——你天天在 L 與 C 之間取捨，分區只是偶發的極端情形。

### 解法空間

把一個系統依 PACELC 歸類，就回答了它的取捨人格：

- **PC/EC**：分區時選一致（犧牲可用），正常時也選一致（犧牲延遲）。最重一致。例：傳統單主關聯式資料庫、用線性一致設定的系統。
- **PA/EL**：分區時選可用，正常時選低延遲。最重可用與速度。例：Dynamo 式無主、Cassandra 預設、許多最終一致 KV。
- **PA/EC**：分區時選可用，但正常時仍盡量一致。例：部分可調系統在「無分區就強讀、分區就降級」的配置。
- **PC/EL**：分區時選一致，正常時選低延遲（分區外不強求一致）。較少見，但可調系統能配出來。

實作上「選哪邊」由**寫入需多少副本確認（W）、讀取需讀多少副本（R）、與總副本數 N**等旋鈕決定（見 quorum，領域 M）。

### 各方案的保證與取捨

| 方案/工具 | PACELC 取向 | 保證 | 適用場景 | 注意事項 |
|---|---|---|---|---|
| 單主關聯式（強一致設定） | PC/EC | 線性一致、可序列化 | 金流、需強保證的核心交易 | 分區時主庫側不可寫；強讀有協調延遲 |
| Dynamo 式無主（如 Cassandra 預設） | PA/EL | 高可用、低延遲、最終一致 | 大規模寫入、可容忍舊值 | 並發衝突要合併；讀可能舊；需對帳兜底 |
| 可調 quorum（R+W>N） | 視 R/W 配置而定 | 配置出強讀或快讀 | 想按操作選強度 | 配錯（R+W≤N）會悄悄讀到舊值 |

> 提醒：表中「Cassandra 預設」「單主關聯式」是**取向示例**，非斷言某產品在所有配置下永遠如此；多數現代系統可調，PACELC 描述的是「預設/典型配置下的傾向」。

### 何時需要

- **設計或選型分散式儲存時，先問 PACELC 兩個問題**：分區時我要可用還是一致？正常時我願意為一致多付多少延遲？這比背「CAP 三選二」有用得多。
- **核心交易（錢、庫存、唯一性）** → PC/EC 那一格，寧可分區時拒絕服務也不讀錯。
- **大規模、可容忍舊值、要求永遠在線（社群計數、推薦、購物車）** → PA/EL，把一致性讓給可用與延遲，再用對帳兜底。

### 常見誤解與陷阱

- **把 CAP 念成「平時三選二、可任意放棄 P」**——分散式系統裡 P 不是你能放棄的選項，分區終會發生；真正的選擇是分區當下的 C vs A。能放棄 P 的只有單機。
- **混淆 CAP 的 C（線性一致）與 ACID 的 C（不變量）**——同字不同義（見「一致性光譜」）。
- **以為 CAP 涵蓋了全部取捨**——它只談分區時。日常 99.9% 時間沒分區，那段的 L vs C 取捨要靠 PACELC 才看得見。
- **把 Brewer 與 Gilbert-Lynch 混為一人/一事**——Brewer 提 conjecture（2000），Gilbert-Lynch 給 proof（2002）；PACELC 是 Abadi（2012 論文）。屬性別記錯。

- Abadi, "Consistency Tradeoffs in Modern Distributed Database System Design"（PACELC，IEEE Computer 2012）：https://www.cs.umd.edu/~abadi/papers/abadi-pacelc.pdf

## ACID vs BASE

### 定義與原理

ACID 與 BASE 是**兩套一致性哲學**，常被對立呈現——但它們不是「好壞」，而是光譜兩端，對應不同的取捨優先序（這個分類與 CAP／PACELC 同層，都在描述「你選什麼保證」）。

**ACID**（傳統交易資料庫的契約）：

- **Atomicity（原子性）**：交易要嘛全做、要嘛全不做。
- **Consistency（一致性）**：交易把資料庫從一個合法狀態帶到另一個合法狀態（不變量不被破壞）——注意這個 C 與 CAP 的 C 不同義。
- **Isolation（隔離性）**：並發交易彼此不互相干擾，效果如同某種序（見「隔離級別與異常」）。
- **Durability（持久性）**：一旦 commit，斷電也不丟。

**BASE** 是刻意取的反義詞（化學上酸 acid／鹼 base），由 **Eric Brewer** 在 CAP 脈絡下提出、Dan Pritchett 的〈BASE: An Acid Alternative〉（ACM Queue, 2008）廣為傳播：

- **Basically Available（基本可用）**：系統大多時候可用，分區/故障時可能降級但不整個倒。
- **Soft state（軟狀態）**：狀態可能隨時間自行改變（即使沒有新輸入），因為副本在背景收斂。
- **Eventual consistency（最終一致）**：最終會收斂，不保證即時。

第一原理：ACID 為**正確性**買單（用協調換強保證），BASE 為**可用性與規模**買單（放鬆一致性換高可用低延遲）。

### 解法空間

「選 ACID 還是 BASE」其實是「在系統不同部位各放多少」——現代系統幾乎都混用：

- **核心交易走 ACID**：用支援交易的資料庫（關聯式或支援多文件交易的 NoSQL），對錢、庫存、唯一性這類資料拉滿。
- **大規模/高可用部位走 BASE**：用最終一致儲存承接動態、計數、快取、衍生資料。
- **介於之間用會話保證/因果一致**：不要全域強一致的成本，但要「對使用者自己一致」（見「一致性光譜」）。
- **跨服務用補償＋對帳兜 BASE 的縫**：放鬆強一致後，用 saga 與對帳保證最終正確（見「最終一致 · 補償 · 對帳」）。

### 各方案的保證與取捨

| 方案/做法 | 保證 | 適用場景 | 注意事項 |
|---|---|---|---|
| ACID（交易資料庫） | 原子、隔離、即時一致、持久 | 金流、庫存、訂單、任何不變量神聖的核心 | 跨節點協調有延遲；極大規模/跨區擴展難、分區時要犧牲可用 |
| BASE（最終一致儲存） | 高可用、低延遲、最終收斂 | 計數、動態、購物車、快取、衍生資料 | 中間態可讀到舊值/矛盾；需衝突解決＋對帳；不可放金流核心 |
| 混合（多數真實系統） | 各部位各取所需 | 幾乎所有中大型系統 | 邊界要清楚：哪些資料 ACID、哪些 BASE，別讓 BASE 資料偷偷承載不變量 |

### 何時需要

- **資料承載「不能錯」的不變量**（餘額不為負、座位不重賣、唯一帳號）→ ACID，沒得商量。
- **資料是「慢半拍沒關係」的高頻動態**（讚數、瀏覽量、推薦、最近瀏覽）→ BASE，換可用與規模。
- **介於之間** → 不必非黑即白；用會話/因果一致取中間，或核心 ACID＋衍生 BASE 的混合。

**Worked example**：一個社群貼文系統。「發文」這個動作本身（建立貼文記錄、扣除每日發文額度）走 ACID 單筆交易——額度不能因並發而扣成負數，這是不變量。「貼文的按讚數」走 BASE——10 萬人同時按讚，若每個讚都進強一致交易，熱門貼文的那一行會變成所有寫入排隊的瓶頸（單行鎖爭用）；改成最終一致的計數（各節點本地累加、背景彙總），吞吐從「序列化的數百/秒」躍到「平行的數萬/秒」，代價是讚數短暫顯示舊值——沒人會因為讚數慢 200 ms 更新而受傷。同一個貼文系統，發文 ACID、計數 BASE，是教科書式的混合。

### 常見誤解與陷阱

- **把 ACID／BASE 當成「整個系統二選一」**——它是**每塊資料**的選擇，真實系統幾乎都混用。把它當全域開關會逼你在「全部很慢」與「全部不可靠」間做假選擇。
- **把 BASE 的 Consistency 與 ACID 的 C 混為一談**——前者是「最終收斂」，後者是「不變量不破」，差很遠。
- **讓 BASE 資料偷偷承載不變量**——例如用最終一致的計數器當「庫存」，並發超賣就發生了。不變量資料必須 ACID。
- **以為 NoSQL＝必然 BASE**——不少現代 NoSQL（如支援多文件 ACID 交易的文件庫）能提供 ACID；ACID/BASE 是配置與資料模型的選擇，不是「SQL vs NoSQL」的同義詞（見 SQL vs NoSQL，領域 O）。

## 零停機 schema 遷移

### 定義與原理

線上系統的資料庫結構（schema）總要演進：加欄位、改型別、拆表、改索引。難點在於**部署不是瞬間的**——新舊版本的應用程式碼會**同時**連著同一個資料庫運行一段時間（rolling 部署期）。若直接 `ALTER TABLE` 做破壞性變更（改名、改型別、刪欄位），舊碼會撞上它不認識的結構而崩潰，或新碼撞上還沒改的結構。零停機 schema 遷移的核心契約是：**任何一個時刻的 schema，都必須同時相容「正在退場的舊碼」與「正在進場的新碼」。**

本條是本書「資料庫結構變更」的 owning 條目；與「序列化與 schema 演進」（領域 A，講的是**線上協定／訊息格式**的向前後相容）分工：那條管「服務之間傳的訊息」，這條管「資料庫裡的表結構」，兩者都靠同一個「新舊並存期必須雙向相容」的原理，但作用面不同。

### 解法空間

達成零停機結構變更的標準做法，核心是 **expand-contract（擴張—收縮，又稱 parallel change）**，把一個破壞性變更拆成數個各自相容的小步：

- **Expand（擴張）**：先**只加不改**——加新欄位/新表，與舊結構並存。加 nullable 欄位、或在現代資料庫加帶靜態預設值的欄位，多為 metadata-only、不鎖全表（如 PostgreSQL 11+ 對非 volatile 預設的 `ADD COLUMN` 不重寫整表）。此時新舊碼都還能跑。
- **遷移期雙寫（dual-write）**：部署能「同時寫新欄位與舊欄位」的應用碼（或用 DB trigger 同步），確保新結構從此不再落後。
- **Backfill（回填）**：把歷史資料分批（典型每批 1k–10k 列、批間刻意 sleep 避免 I/O 飽和與長鎖）寫進新欄位，補齊雙寫之前的舊資料。
- **切讀**：確認新欄位已填滿、雙寫穩定後，把讀路徑切到新欄位，觀察一段時間。
- **Contract（收縮）**：等到**沒有任何碼還在讀寫舊欄位**（含 drain 掉舊版本的在途請求）後，才刪舊欄位。此時刪除多為 metadata-only、毫秒級。

其他配套：線上建索引（`CREATE INDEX CONCURRENTLY`／`ONLINE`／`INSTANT`，避免建索引鎖表）、影子表（shadow table，整表重建型變更用新表並雙寫再切換）、feature flag 控制切換時機。

### 各方案的保證與取捨

| 方案/做法 | 保證 | 適用場景 | 注意事項 |
|---|---|---|---|
| expand-contract（雙寫+backfill） | 全程相容新舊碼、零停機 | 加欄位、改型別、改名、拆表的主流做法 | 步驟多、跨多次部署；中途要保證雙寫不漏 |
| 應用層雙寫 | 應用碼控制、可觀測 | 多數遷移 | 漏寫某條路徑就資料分歧；直連 DB 的腳本繞過雙寫 |
| DB trigger 雙寫 | 連直連寫入都同步 | 有外部腳本/多寫入源 | trigger 增寫入開銷、隱性邏輯難維護 |
| 影子表 + 切換 | 整表重建型變更也不停機 | 大改（換主鍵、大規模型別變更） | 雙寫＋backfill＋一致性校驗成本高 |
| 直接 ALTER（破壞性一步） | 簡單 | 離線維護窗、或極小無流量表 | 線上跑會鎖表/讓舊碼崩潰，**非零停機** |

### 何時需要

- **服務不能停、且做 rolling 部署** → expand-contract 是預設做法，幾乎所有線上結構變更都該這樣拆。
- **表很小、或有維護窗、或無流量** → 直接 ALTER 即可，硬套 expand-contract 是 over-engineering。
- **變更涉及大表＋會鎖表的操作**（改型別、加 NOT NULL 帶 volatile 預設、重建主鍵）→ 一定要拆步＋線上索引＋分批，否則一句 ALTER 鎖住大表幾分鐘＝事實上的停機。

**Worked example**：要把使用者表的 `name`（單欄）拆成 `first_name`／`last_name`，表 5,000 萬列、全天有流量。直接做不行：刪 `name` 會讓舊碼崩、加 NOT NULL 新欄會鎖表。按 expand-contract：(1) **Expand**：加兩個 nullable 欄位 `first_name`／`last_name`（PG 11+ 為 metadata-only、毫秒級，不鎖表）。(2) **雙寫**：部署新碼，寫入時同時填 `name` 與拆分後的兩欄。(3) **Backfill**：背景每批 5,000 列、批間 sleep 100 ms 把舊資料拆填——5,000 萬列約 1 萬批，即使每批含 sleep 約 200 ms，總計約 `10,000 × 0.2s ≈ 2,000s ≈ 33 分鐘`跑完，全程不鎖表、不影響線上。(4) **切讀**：讀路徑改讀新欄，觀察數日。(5) **Contract**：確認無碼再用 `name`、舊版本實例全下線後，才 `DROP COLUMN name`（metadata-only）。把一個「會停機數分鐘」的破壞性變更，拆成五個各自零停機的小步。

### 常見誤解與陷阱

- **以為 `ADD COLUMN ... NOT NULL DEFAULT ...` 一定鎖表重寫**——要看資料庫版本與預設值是否 volatile：PG 11+ 對**非 volatile** 靜態預設是 metadata-only；但 volatile 預設（如每列算一個 UUID）仍會重寫整表。版本與預設型別決定生死，別一概而論（2026-06）。
- **跳過 backfill 校驗就切讀**——雙寫若有漏寫的路徑（某個直連 DB 的腳本、某條冷路徑），新欄會有缺口；切讀前要校驗新舊一致。
- **Contract 太早**——舊版本實例還沒完全下線、在途請求還在用舊欄位時就 drop，舊碼立刻崩。要等到確定無人再用。
- **backfill 一把梭不分批**——一句 `UPDATE` 掃 5,000 萬列會長時間持鎖＋撐爆 WAL/replication，等於自造停機。必須分批＋限速。
- **把它和線上協定相容搞混**——資料庫結構變更（本條）與服務間訊息格式演進（領域 A）是兩件事，各有各的相容規則，別用一邊的做法套另一邊。
