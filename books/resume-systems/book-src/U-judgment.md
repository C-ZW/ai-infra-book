# U · 工程判斷與決策

這個領域解的是「**怎麼選**」這件事本身：當一個技術問題擺出多種合理解法時，用什麼框架決定走哪條路、付什麼代價、留什麼退路。本檔含七條判斷條目——決策的可逆性、自建 vs 採用、簡單性 vs 過度工程、技術選型流程、錯誤處理哲學、AI 如何移動這些判斷的天平，以及量級估算的容量直覺。它是讀後面每一條技術概念時要戴的「判斷眼鏡」（與它對偶的「推動眼鏡」在領域 T）；技術概念本身的機制不在這裡講，這裡只講「面對那些機制，你怎麼下注」。

## 可逆 vs 不可逆決策（one-way / two-way doors）

### 定義與原理

決策的成本不只是「選錯的後果」，還包括「**反悔的代價**」。把這兩者拆開，就得到一個比「重要 vs 不重要」更有用的座標軸：**可逆 vs 不可逆**。Jeff Bezos 在 Amazon 2015 年致股東信裡用「單向門（one-way door）／雙向門（two-way door）」描述這件事——走過單向門，門就鎖上了，不喜歡也回不去，必須慢、必須謹慎、必須多方會診；雙向門則相反，走進去看一眼不對就退回來，可以快、可以由小團隊或個人直接拍板（2026-06 仍是業界廣用的決策語彙）。

第一原理是：**決策速度應該與可逆性成正比，而不是與重要性成正比**。組織變大後最常見的退化，是把所有決策都套上「重大決策」的厚重流程，連雙向門也開三次評審會，結果是緩慢、過度規避風險、不敢實驗。判斷一個決策落在哪一端，問三個問題就夠：撤銷它要花多少工程時間？撤銷它會不會破壞已經外溢出去的狀態（別人已經依賴的資料格式、已經發出去的 API 契約）？撤銷時有沒有資料會永久遺失？

### 解法空間

面對一個決策，先分類再決定投入多少審議成本：

- **判定可逆性**：列出「撤銷成本」——回滾要動幾個系統、有沒有不可逆的副作用（資料遷移、對外公開的契約、寫進別人快取的東西）。
- **對雙向門：快速試錯**。直接做最小版本、上線觀察、不對就退。把決策權下放到最接近問題的人，不堆審議流程。
- **對單向門：增加可逆性，把單向門改造成雙向門**。常見手段：feature flag（先暗度、可瞬間關掉）、灰階發布（canary，先放 1% 流量，見領域 Q）、雙寫過渡期（新舊並存、可切回，見領域 B 的零停機 schema 遷移）、抽象層隔離（把不可逆的選擇藏在介面後面，換掉時只動一處）。
- **對「偽單向門」：拆小**。很多看似不可逆的大決策，其實是一串小決策捆在一起；拆開後多數是雙向門，真正不可逆的只剩一兩個，集中審議那一兩個即可。
- **AI 維度**：AI 讓「先做一個能丟掉的原型」成本驟降——這把更多原本因「做了捨不得丟」而被當單向門的決策，變回雙向門（試了不對就刪、重來幾乎免費）。新陷阱是反過來：AI 一晚能生出一個跑得動的大型系統，讓團隊**誤以為**選型已驗證、實際是走進了沒人真正讀過、難以撤回的單向門（「能跑沒人懂」見本檔 AI 條目）。

### 各方案的保證與取捨

| 方案/做法 | 效果 | 適用場景 | 注意事項 |
|---|---|---|---|
| 快速試錯（不堆流程） | 決策延遲最低、學得最快 | 確認是雙向門 | 誤判可逆性時代價爆增；先驗證撤銷成本再快 |
| feature flag 包起來 | 把單向門變雙向門，可瞬間回退 | 行為類改動、漸進放量 | flag 本身是技術債，要有清除機制；長期堆積會變成隱性分支地獄 |
| 灰階 / canary | 小流量先驗證，限制爆炸半徑 | 上線風險高、可分流量 | 需要可觀測性支撐才看得出「對不對」（見領域 I） |
| 抽象層隔離 | 把不可逆選擇藏在介面後 | 預期未來可能要換的元件 | 過早抽象本身有成本（見本檔簡單性條目）；抽象選錯比沒抽象更難拆 |
| 拆小決策 | 把偽單向門拆成多數雙向門 | 看似巨大、不可逆的決策 | 拆解本身要花心力；別把真正的單向門也輕率拆掉 |

保證的本質：可逆性不是天生的，是**可以買的**——上面每一招都是「花一點前置成本，換取反悔權」。適用條件：當撤銷成本高且不確定性也高時，這筆保險最划算；當你很確定、或撤銷本來就便宜時，買保險就是浪費。

### 何時需要

需要鄭重對待可逆性的場景：資料格式 / 儲存引擎選擇（資料一旦寫進去，遷移是大工程）、對外公開的 API 契約（別人依賴後你不能說改就改，見領域 H）、跨團隊共用的核心抽象、難以撤回的組織級流程。這些是真單向門，慢一點、會診一點是對的。

反過來，over-engineering 的訊號是：**對雙向門也套用單向門的審議**——為一個改個內部函式名、加一個非公開端點、調一個快取 TTL 這種「五分鐘能改回來」的事開評審會。判準很簡單：先估撤銷成本，撤銷便宜就別審議。

### 常見誤解與陷阱

- **把「重要」當成「不可逆」**。一個決策可以同時很重要又很可逆（例如選哪個 CI 工具——影響每天工作，但要換隨時能換）。重要性決定要不要認真想，可逆性決定要不要慢。
- **把「不可逆」當成「不能犯錯」**。即使是單向門，目標也不是零失誤，而是把失誤的後果控制在可承受範圍——所以才有 canary、才有先小規模驗證。
- **忽略「外溢狀態」讓雙向門悄悄變單向門**。一個本來能回滾的部署，一旦它寫出了下游依賴的新資料、或客戶端快取了新行為，回滾就不再乾淨。任何會產生持久副作用的改動，可逆性都要重新評估。
- **誤把「沒人會反對」當成「可逆」**。共識高不代表好撤回；這兩件事無關。

## 自建 vs 採用與相依管理

### 定義與原理

每加一個相依（dependency），都是一筆**用「自己不寫」換「把一部分控制權交出去」的交易**。它替你省下開發與維護成本，代價是：你的系統現在會因為一個你沒寫、可能也沒讀過的東西而壞掉、變慢、或被攻擊。判斷的核心不是「自建好還是採用好」，而是「**這塊功能，是不是我的核心競爭力**」——是，傾向自建（你要能完全掌控、也願意長期養）；不是，傾向採用（別把心力花在不會讓你贏的地方）。

兩端各有一個失敗症候群。一端是 **NIH（Not Invented Here）**：什麼都自己寫，重造輪子、養一堆只有自己懂的內部框架、把時間花在已被解決的問題上。另一端是**過度相依**：一個小功能拉進一棵巨大的相依樹，npm 一個專案動輒上千個 transitive dependency，每一個都是潛在的故障點與攻擊面。真實案例：2016 年 left-pad 事件，一個只有十一行的套件被作者撤下，連鎖讓大量專案無法建置；2024 年 xz utils 後門（CVE-2024-3094），攻擊者花數年潛伏進一個冷門但被現代 Linux 發行版幾乎無所不在地間接依賴（透過 systemd → sshd）的壓縮庫，差點在 SSH 認證前植入遠端後門（2026-06 仍是供應鏈攻擊的標準警世案例）。

### 解法空間

決定要不要加一個相依，以及如何控管已加的相依：

- **核心 vs 非核心判準**：這塊是不是你的差異化來源？是→自建；否→採用成熟方案。
- **採用前的盡職調查**：維護活躍度（最近 commit、issue 回應、有沒有單一維護者風險）、授權相容性（見本領域對授權的注意）、相依樹大小（拉一個進來實際帶幾個 transitive）、攻擊面（它要不要網路 / 檔案 / 子行程權限）。
- **隔離相依**：用 adapter / 防腐層（anti-corruption layer）把第三方介面包在自己的介面後，讓「換掉它」只動一處（呼應本檔可逆性）。
- **鎖定與來源驗證**：lockfile 釘死版本與雜湊（避免同一版號內容被掉包）、相依漏洞掃描接進 CI（見領域 I）、必要時把關鍵相依 vendor 進自己的倉庫或內部 mirror。
- **「薄採用」**：採用標準協定 / 規格而非綁死某家實作（例如依賴 SQL、S3 API 這種有多家相容實作的介面），保留換供應商的退路。
- **AI 維度**：AI 把「自建」的成本拉低——以前因為太費工而選採用的小工具，現在可能自己生一個更划算、且完全可控（程式碼在你手上）。新陷阱：AI 生成的程式碼可能默默內嵌或建議過時 / 不存在 / 名稱被搶註（typosquatting）的套件，等於替你開了一道供應鏈側門，採信前要驗。

### 各方案的保證與取捨

| 方案/做法 | 效果 | 適用場景 | 注意事項 |
|---|---|---|---|
| 自建 | 完全掌控、無外部授權/供應鏈風險 | 核心競爭力、需求特殊、相依的維護風險高 | 你要永久維護它；別在非核心上做（NIH） |
| 採用成熟開源/商用 | 省開發、享社群修補與生態 | 非核心、問題已被解決、有活躍維護 | 供應鏈風險、授權風險、維護者跑路風險；要做盡職調查 |
| adapter 隔離後採用 | 享採用的省力又保留可換性 | 預期未來可能要換的第三方 | adapter 本身是成本；介面選錯一樣難換 |
| 薄採用（依賴標準/協定） | 換實作不換程式碼 | 有多家相容實作的領域 | 標準的「相容」常有方言差，邊角仍會綁住 |
| fork 並自維 | 取得掌控、可改 | 上游停更但你重度依賴 | 從此你扛全部維護，等於部分自建 |

保證的本質：相依省的是**你現在的時間**，賠上的是**未來的掌控權與一條額外的故障/攻擊路徑**。適用條件：在非核心領域、相依成熟且活躍時，這筆交易划算；在核心、或相依脆弱（單一維護者、停更、授權有雷）時，掌控權的價值會超過省下的工。

### 何時需要

傾向自建：這塊是你贏的關鍵、需求夠特殊讓現成方案都要大改、或可選的相依都有明顯維護/授權風險。傾向採用：標準化的橫切問題（序列化、加密、HTTP server、訊息佇列…這些自建幾乎一定虧）、團隊沒有專長也不該培養的領域、有成熟且活躍的方案可用。

over-engineering 的兩種形態都要警惕：在非核心上自建（NIH，把人力燒在不會讓你贏的地方），以及為了省幾行而拉進一棵巨樹（過度相依，買進一堆你不會用到卻要負責的攻擊面）。

### 常見誤解與陷阱

- **只看「省多少開發時間」，不看「日後維護與風險成本」**。相依的真實成本是整個生命週期的——升級、修漏洞、追上游破壞性變更、出事時 debug 一個你沒寫的東西。
- **忽略 transitive dependency**。你直接依賴五個，實際拉進來可能上千個；攻擊面與授權風險是整棵樹的，不是你直接挑的那幾個。
- **把「star 數多」當成「安全」**。熱門不等於有人在審安全（xz 案的庫並不冷門到沒人用，卻冷門到沒人盯維護權轉移）。要看的是**有效維護**，不是人氣。
- **授權踩雷**。把 copyleft（如 AGPL）的東西當無條件免費用，可能反過來綁住你自己的程式碼；商用授權的免費層也常有配額或用途限制。引入前確認授權與你的散布方式相容。
- **自建了卻不當核心養**。自建的前提是你願意長期維護它；寫完丟著沒人管的內部框架，比採用一個有社群的還糟。

## 簡單性 vs 過度工程

### 定義與原理

過度工程（over-engineering）是替**還沒出現、可能永遠不會出現**的需求預先付出複雜度。它的兩個經典形態：**過早抽象**（為了「將來可能要支援多種 X」先蓋一層通用抽象，結果只有一種 X，抽象憑空增加了理解與修改成本）與**過早最佳化**（為了「將來可能很慢」先寫複雜的快取/平行/微優化，結果瓶頸根本不在那）。對抗它的兩句口號：**YAGNI（You Aren't Gonna Need It）**——源於 Extreme Programming，Kent Beck 等人的主張是「真的需要時才實作，不要因為預見可能需要就先做」；以及廣為流傳的「**過早最佳化是萬惡之源**」——出自 Donald Knuth 1974 年〈Structured Programming with go to Statements〉（Knuth 把它歸給 Hoare，Hoare 又歸回 Knuth，出處其實成謎；2026-06 仍是最常被引用的工程箴言之一）。

但簡單性不是「永遠寫最笨的版本」。Knuth 那句完整是「我們該忽略小效率，約 97% 的時間：過早最佳化是萬惡之源；但別放過那關鍵的 3%」。真正的原則是：**複雜度要花在已被證實的需求上，不花在臆測的需求上**。簡單也不等於少寫程式碼——「簡單」指的是認知負擔低、容易理解與修改；有時為了真正的簡單反而要多寫一點（明確勝過聰明）。

### 解法空間

- **YAGNI / DTSTTCPW（do the simplest thing that could possibly work）**：先寫能解決當下需求的最直接版本，需求真的來了再演化。
- **延後抽象（rule of three）**：常見經驗法則是「同樣的東西出現第三次再抽象」——前兩次的重複比抽錯的抽象便宜，第三次你也才真的看清共通模式。
- **先量再優化**：最佳化前先 profiling / 壓測找出真瓶頸（見領域 I、S），對著數字優化那關鍵的 3%，別憑直覺。
- **可演化優於完備**：把系統設計成容易往後改（清楚的邊界、可逆的決策），而不是一開始就猜中所有未來需求。這比「先建好通用框架」更穩，因為你猜未來通常會猜錯。
- **明確優於隱晦**：用直白、好讀的寫法換低認知負擔；抗拒為了少幾行而上的「聰明」技巧。
- **AI 維度**：AI 讓「先寫簡單版、要改再重來」更划算（重寫便宜了，更不需要為了省未來的重寫而先過度設計）。但 AI 也擅長**一鍵生出過度工程**——憑一句「做個有彈性的方案」就吐出一堆抽象層、設定開關、外掛點，全是 YAGNI 該砍的東西。它生得快，不代表你該留下。

### 各方案的保證與取捨

| 方案/做法 | 效果 | 適用場景 | 注意事項 |
|---|---|---|---|
| YAGNI / 最簡版本 | 認知負擔最低、最快交付 | 需求未定、未來不確定 | 真有已知近期需求卻硬不做，是另一種短視 |
| rule of three 延後抽象 | 避免抽錯、用真實樣本抽象 | 重複才剛冒頭、模式未明 | 拖過頭會累積重複技術債；第三次出現要果斷抽 |
| 先 profiling 再優化 | 心力花在真瓶頸 | 效能問題 | 沒量測就優化＝賭；但有些瓶頸要壓力夠大才現形 |
| 預先建通用抽象 | 真有多變體時省事 | 變體已**確定**會出現且已知形狀 | 多數時候你猜錯形狀；抽象選錯比沒抽象更難拆 |

保證的本質：簡單性買的是**改變的自由**——程式碼越少、抽象越少、巧思越少，未來要改方向時要拆的東西就越少。過度工程則是**賭未來會長成你猜的樣子**，賭對省一點、賭錯賠上一整層沒人需要又難拆的複雜度。實務上未來多半不照劇本，所以這場賭注的期望值通常為負。

### 何時需要

該預先投入複雜度的少數場景：那個「未來需求」其實是**已知的近期確定需求**（不是臆測）、或那個抽象/最佳化**事後再加的成本極高**（例如資料模型、對外契約這種改起來要驚動很多下游的東西——這類偏單向門，前期想清楚是對的，見本檔可逆性條目）。

over-engineering 的訊號：抽象層只有一個實作、設定開關沒人切換過、為了「擴展性」加的外掛點從沒接過第二個外掛、優化的程式碼對著一個壓測證明不是瓶頸的地方。看到這些，砍掉就是淨賺。

### 常見誤解與陷阱

- **把 YAGNI 當成「不准做任何前瞻設計」**。YAGNI 反對的是「為臆測的需求預先實作」，不反對「為已知的近期需求留好演化空間」。已經明確要來的東西，先想是負責，不是過度工程。
- **把「簡單」等同「程式碼短」**。一行密不透風的聰明寫法，認知負擔可能遠高於五行直白的。簡單是給讀的人的，不是給行數的。
- **過早抽象的代價被低估**。錯的抽象比重複更貴：它把不該綁的東西綁在一起，每次要改其中一個變體都被另一個拖住，拆它比當初不抽象難得多。
- **「以後一定會慢」式的恐懼驅動最佳化**。在沒有量測、流量也還沒到的情況下，為臆想的效能問題寫複雜程式碼，多半是把簡單系統先搞慢（指 debug 變慢、改起來變慢）。
- **反向過頭——用 YAGNI 當藉口欠該還的債**。明知道下個 sprint 就要支援的東西硬說「還不需要」，是把短視包裝成紀律。判準永遠是「需求是已知近期的、還是臆測的」。

## 技術選型

### 定義與原理

技術選型是「在多個合理候選裡選一個，並承擔之後維護它的後果」的決策。它的特殊之處在於**資訊嚴重不對稱**：選的當下你掌握的是 demo、文件、benchmark 和別人的部落格——全是「順境資料」；而真正決定成敗的是逆境表現——它出事時好不好 debug、文件外的邊角、社群會不會幫你、以及最關鍵的**退場成本（cost of exit）**：選錯了要換掉它有多痛。很多技術選型的真正風險不在「它好不好」，而在「**它把你綁多深**」。

第一原理：選型應該**降低不確定性後再下注**，而不是憑 demo 和直覺賭。降低不確定性的工具是 PoC（proof of concept）和 spike——拿你**自己的真實場景**去打它，特別是打那些 demo 不會展示的逆境（高併發、故障、奇怪輸入、營運半夜出事）。同時要在選之前就想好退場：這個決策偏單向門還是雙向門（見本檔可逆性條目）。

### 解法空間

- **明確需求與約束先行**：先寫下硬需求（must-have）與軟偏好（nice-to-have），避免被「最酷的技術」帶跑。沒有需求清單的選型是選心情。
- **評估框架（decision matrix）**：列候選 × 評估維度（契合需求、維護活躍度、生態/社群、營運成本、團隊熟悉度、授權、退場成本…），各維度加權打分。它的價值不在分數本身，而在**逼你把隱性偏好攤開來討論**。
- **PoC / spike**：用真實（或擬真）場景做小型驗證，重點打逆境——壓到它的極限、模擬故障、看它出事時的可觀測性與 debug 體驗，而非只跑 happy path demo。
- **試生產（蹚雷區）**：在低風險的真實服務上先用，累積營運經驗，再決定要不要全面押注（呼應 canary 的精神，見領域 Q）。
- **評估退場成本**：選之前就問「要換掉它要動多少東西」——資料能不能搬出來、有沒有標準介面可隔離、它滲進你多少程式碼。退場成本高的，當單向門慎選。
- **AI 維度**：AI 能極快生出多個候選的 PoC 骨架、整理對照維度、爬文件做初步比較，壓縮評估的苦工。新陷阱：AI 給的對照常摻雜過時或編造的事實（版本、授權、預設值在它的訓練資料裡可能是舊的甚至錯的），任何時效性結論都要回官方來源驗（本書的時效事實都標 2026-06 並查官方，就是這個道理）。

### 各方案的保證與取捨

| 方案/做法 | 效果 | 適用場景 | 注意事項 |
|---|---|---|---|
| decision matrix 打分 | 把隱性偏好攤開、可討論可追溯 | 多候選、多利害關係人 | 權重與分數帶主觀，別假裝它客觀；用來引導討論非取代判斷 |
| PoC / spike | 用真實場景降不確定性 | 不確定它扛不扛得住你的場景 | 只打 happy path 等於沒打；要打逆境才有資訊量 |
| 試生產（小範圍先上） | 取得真實營運經驗 | 退場成本高、想先蹚雷 | 要有回退方案，別讓「試用」變成不知不覺的全押 |
| 跟既有技術（不選新的） | 零學習成本、團隊熟 | 新方案優勢不夠壓倒性時 | 熟悉度是真實價值，但別讓它變成拒絕一切新東西的藉口 |
| 選大廠/主流 | 生態大、人才好找、不易停更 | 求穩、團隊變動大 | 主流不等於最適配你的場景；仍要對需求驗 |

保證的本質：選型流程買的是**「下注前先把不確定性換成資訊」**。PoC 花的時間，是用來避免「全押之後才發現它在你的逆境下不行」這種昂貴錯誤的保險。適用條件：當退場成本高、且候選在你的真實場景下表現不明時，這筆前置投入最值；當決策可逆、或候選都明顯夠用時，過度評估本身就是 over-engineering。

### 何時需要

需要鄭重走完整流程（評估框架＋PoC＋評估退場）的：退場成本高的選擇（資料庫、訊息中介、核心框架、雲廠商鎖定——這些偏單向門）、團隊要長期依賴、或押錯影響面廣的決策。

可以從簡的：可逆、影響面小、或團隊已有強烈既有偏好且該偏好夠合理的場景（例如已經全棧用某語言，再選個同生態的小工具，不必為它開評估矩陣）。為一個雙向門的小工具跑三週 PoC，就是把流程用過頭。

### 常見誤解與陷阱

- **被 demo 和 benchmark 騙**。廠商的 benchmark 是在對它最有利的條件下跑的；demo 永遠走 happy path。要信的是你自己拿真實場景打出來的數字。
- **忽略退場成本，只比「功能」**。兩個功能相當的方案，退場成本可能差十倍——一個有標準介面隨時能換，一個讓你的程式碼到處是它的專有 API。選型要比的是「綁多深」，不只「現在好不好用」。
- **「履歷驅動開發」（resume-driven development）**：選一個技術只因為它熱門 / 想學 / 好看，而不是因為它最適配需求。新潮和適配是兩回事。
- **低估「無聊技術」的價值**。成熟、無聊、團隊熟的技術，故障模式已知、坑都有人踩過、半夜出事查得到解答——這種可預測性本身是巨大的價值，常勝過新技術那點性能或語法甜頭。
- **把選型當一次性事件**。選完不是結束；要持續觀察它在生產的表現，並保留「發現選錯時撤回」的機制（這又回到可逆性）。

## 錯誤處理哲學

### 定義與原理

每個會出錯的地方，都逼你回答一個哲學問題：**這個錯誤該讓誰知道、在哪一層處理、處理到什麼程度**。兩個極端立場：**fail-fast**——一偵測到不對就立刻、大聲地失敗，不帶著壞狀態繼續跑，把問題暴露在離源頭最近的地方；**韌性 / 容錯**——盡量讓系統在部分出錯時仍提供降級服務，不因為局部失敗就整體倒下。這不是二選一，而是**對不同錯誤該選不同立場**——關鍵是先把錯誤分類。

分類的核心是兩個維度：**可不可恢復**（暫時性如網路抖動、可重試 vs 永久性如格式錯誤、重試也沒用）與**是不是 bug**（程式設計錯誤如空指標、邏輯違例 vs 預期內的營運狀況如下游逾時、輸入不合法）。第一原理：**程式設計錯誤該 fail-fast**（它代表你的假設破了，藏起來只會讓壞狀態擴散、更難 debug——呼應「分散式運算的謬誤」：對網路錯誤幾乎不做處理的程式會悄悄壞掉）；**預期內的營運錯誤該被當成正常控制流處理**（重試、降級、或乾淨地回報給呼叫端）。把這兩類搞混——對 bug 寬容（吞掉例外裝沒事）或對營運狀況苛刻（一個下游慢就整個崩）——是兩個方向的災難。

### 解法空間

對一個錯誤，能做的事就這幾種，差別在「在哪一層做、為什麼」：

- **fail-fast（立刻失敗）**：偵測到違反不變式 / 不該發生的狀態就立刻爆，附足夠 context。適合 bug 與「繼續跑會擴散壞狀態」的情況。
- **重試**：對暫時性、可恢復的錯誤重試——但做法（退避、jitter、上限）見領域 A 的「重試、退避、jitter」，這裡只決定「**這個錯該不該重試**」（冪等嗎？是暫時性的嗎？重試會不會放大成 retry storm？見領域 E）。
- **傳播（propagate）**：自己處理不了就乾淨地往上拋，讓有足夠 context 的那一層決定——關鍵是**保留錯誤資訊、加上本層 context、不吞掉**。
- **降級 / fallback**：回退到次優結果（快取的舊值、預設值、簡化功能），讓使用者拿到「不完美但可用」而非錯誤——機制見領域 P 的「fallback · 降級」，這裡只決定「**這個錯值不值得降級而非直接失敗**」。
- **熔斷觸發**：當某依賴持續失敗，停止對它發請求以免拖垮自己——circuit breaker 的三態機制見領域 E，這裡只決定「哪些依賴的失敗該觸發熔斷」。
- **錯誤分類與建模**：用明確、有型別的錯誤（帶錯誤碼、訊息、足夠 debug 的 context）取代裸 `Error` / 字串；讓上層能依型別決定怎麼處理。
- **AI 維度**：AI 生成的程式碼有個系統性偏誤——傾向產出「看起來處理了錯誤」的程式碼（包個 try/catch、`return null`、log 一行就吞掉），happy path 漂亮、逆境靜默壞掉。Review AI 程式碼時，錯誤路徑是最該盯的：它有沒有把 bug 當營運狀況吞掉？有沒有把該傳播的錯誤靜默掉？

### 各方案的保證與取捨

| 方案/做法 | 效果 | 適用場景 | 注意事項 |
|---|---|---|---|
| fail-fast | 把問題暴露在離源頭最近處、不擴散壞狀態 | bug、不變式違反、繼續跑會更糟 | 對外部服務一律 fail-fast 會犧牲韌性；用在「自己這層的假設破了」 |
| 重試 | 吸收暫時性故障 | 暫時性、可恢復、且操作冪等 | 非冪等重試會重複副作用；無上限/無退避→retry storm（見領域 A、E） |
| 傳播 | 讓有 context 的層決定，不過早決策 | 本層資訊不足以正確處理 | 別吞、別裸拋——要保留原因鏈並加本層 context |
| 降級 / fallback | 局部失敗不變成整體失敗 | 有可接受的次優結果 | 降級要明示（別讓人以為拿到的是完整結果）；舊快取可能誤導 |
| 吞掉錯誤（反例） | （表面上）不中斷 | 幾乎沒有正當場景 | 把 bug 變成靜默資料損壞，是最危險的反模式 |

保證的本質：fail-fast 買的是**「壞狀態不擴散、問題好定位」**，代價是**可用性**（它真的會讓請求失敗）；韌性買的是**可用性**，代價是**複雜度與「掩蓋真實問題」的風險**（降級久了沒人發現底層壞了）。沒有普適最優——**你是在用可用性換可除錯性，或反過來**，選哪邊取決於這個錯誤是 bug 還是營運狀況、以及這條路徑對可用性多敏感。

### 何時需要

傾向 fail-fast：偵測到自己的不變式被違反、收到結構性非法的輸入（這是 bug 或攻擊，不是營運狀況）、繼續執行會寫出錯誤資料。**啟動期**尤其該 fail-fast——設定缺失、相依連不上，就該拒絕啟動而不是帶病上線（見領域 Q 的 config 管理）。

傾向韌性 / 降級：面對外部依賴的暫時性故障（網路、第三方 API、下游服務）、且該功能對可用性敏感、又存在可接受的次優結果。判準：**錯誤的根因在「我家」還是「外面」**——我家的（bug、壞狀態）fail-fast，外面的（暫時性、預期內）盡量吸收。對所有錯誤一律 fail-fast 是脆弱，對所有錯誤一律吞掉降級是埋雷，兩者都是 over-engineering 反面教材。

### 常見誤解與陷阱

- **吞掉例外（swallowing）**：`catch` 了什麼都不做、或只 log 一行就繼續。這把可恢復的「需要處理的錯誤」變成靜默的資料損壞——最難 debug 的一類 bug，因為現場已經被吃掉了。
- **裸 catch-all 蓋住 bug**。一個包山包海的 `catch (e)` 會連「空指標」這種程式 bug 一起吞掉，讓它偽裝成營運狀況。bug 該 fail-fast，別讓籠統的容錯把它藏起來。
- **錯誤訊息丟失 context**。往上拋時把原因鏈砍掉、或換成沒資訊的字串，到了頂層只剩「something went wrong」——等於放棄了半夜出事時的可除錯性。每往上一層都該疊加 context、保留原因。
- **把 fail-fast 和「沒有韌性」劃等號**。fail-fast 是「在正確的層、對正確的錯誤立刻失敗」，它和系統整體的韌性不衝突——一個元件 fail-fast，正是讓上層能乾淨地降級的前提。
- **重試非冪等操作**。把「重試」當成萬靈丹套在所有錯誤上，會對非冪等操作造成重複副作用、對持續性故障造成 retry storm。重試前先問冪等與暫時性（見領域 A、E）。
- **降級悄無聲息地長期掩蓋故障**。fallback 設計得太安靜，底層壞了好幾天沒人發現，因為使用者一直拿到「還行」的降級結果。降級必須可觀測——它觸發了就該有訊號（見領域 I）。

## AI 怎麼移動判斷的天平

### 定義與原理

前面幾條判斷——自建 vs 採用、簡單性 vs 過度工程、可逆性、選型——背後都有一組**隱性的成本假設**：寫程式碼貴、原型貴、抽象貴、重寫貴。AI 輔助開發改變的不是這些判斷的**框架**，而是框架裡的**輸入數字**。當「生產程式碼」「做原型」「大規模重構」的成本被壓低一個量級，每一條判斷的損益平衡點都會移動。這條條目不講任何工具或 model（那些會過時），只講**哪些天平被推動了、推向哪邊、以及推動本身帶來什麼新風險**——這是原理層的東西，工具再換代也不腐。

第一原理：**降低某件事的成本，會改變所有「為了省這件事而做的取捨」**。但有一個成本 AI 沒有降低、甚至升高了——**理解的成本**。AI 能讓你**擁有**一大堆能跑的程式碼，卻沒讓你**理解**它。判斷的天平因此多了一個新砝碼：「能跑沒人懂」的程式碼，是一種新型態的技術債與單向門。

### 解法空間

天平被推動後，對應的判斷與護欄要跟著調整：

- **自建 vs 採用的線往「自建」移**：以前因「太費工」而採用的小工具，現在自己生一個可能更划算又完全可控。對應做法：把 AI 生成的「自建」當成**要 review、要測試、要長期維護**的程式碼對待，而非免費午餐——掌控權的前提是有人真的懂它。
- **「過度工程」的判準不變，但要更警覺**：AI 讓寫複雜抽象變便宜，於是更容易**不知不覺**過度工程（它樂意憑一句話生出五層抽象）。對應做法：YAGNI 的紀律更重要，不是更不重要——便宜不是保留它的理由（見本檔簡單性條目）。
- **可逆決策更可逆**：丟棄式原型的成本驟降，更多決策能變回雙向門（試了刪掉重來幾乎免費）。對應做法：善用這點多做試錯，但別把「AI 一晚生出的能跑系統」誤當成「已驗證的選型」。
- **驗收取代撰寫，成為瓶頸**：當生成變快，**review、測試、理解**變成新的瓶頸與品質閘。對應做法：把判斷重心從「怎麼寫」移到「怎麼驗」——對 AI 產出要有明確的接受標準、要跑測試、要有人理解到能維護的程度（驗收與信任邊界的做法見領域 T 的 AI 協作條目）。
- **守住「理解門檻」**：把「至少要有一個人能在不靠 AI 的情況下解釋並修改這段程式碼」當成合併條件。理解不到這個程度的程式碼，等於走進了單向門。

### 各方案的保證與取捨

| 判斷維度 | AI 把天平推向 | 對應的工程做法 | 新陷阱 |
|---|---|---|---|
| 自建 vs 採用 | 更傾向自建（生成成本↓） | 把生成碼當須維護的自建對待 | 生成碼可能內嵌過時/不存在的相依，要驗 |
| 簡單 vs 過度工程 | 過度工程變更容易（憑一句話生一堆） | YAGNI 紀律更嚴、便宜≠該留 | 把「生得快」誤當「該保留」 |
| 可逆性 | 更多決策變雙向門（原型便宜） | 多試錯、丟棄式驗證 | 把「能跑」誤當「已驗證」走進單向門 |
| 撰寫 vs 驗收 | 瓶頸從寫移到驗 | 明確接受標準＋測試＋理解門檻 | review 變橡皮圖章、靜默吞錯（見錯誤處理條目）|

保證的本質：AI 買的是**生產速度**，可能賠上的是**理解深度與長期可維護性**。適用條件：當產出有人真正驗收、理解、並當成自己的程式碼維護時，速度是淨賺；當產出沒人懂、只因「能跑」就合併時，你是在用未來的可維護性換現在的速度——而那筆債利息很高。

### 何時需要

幾乎所有前述判斷都該重新校準輸入數字：原型/實驗類決策（成本降最多，最該善用）、非核心小工具的自建 vs 採用（天平明顯移動）、大規模機械式重構（AI 做苦工最划算，方向與驗收仍由人定，見領域 T）。

不該因為 AI 變便宜就放鬆的：核心競爭力的程式碼（仍要人深度掌控）、不可逆決策（生成快不代表驗證夠）、安全與正確性敏感路徑（生成碼的錯誤路徑尤其要盯）。把「AI 能做」當成「不需要判斷」，是這一波最大的 over-engineering 與失控來源。

### 常見誤解與陷阱

- **「能跑沒人懂」當成完成**。AI 生出通過測試的程式碼，不等於團隊理解它、能維護它。能跑是必要條件，不是充分條件；沒人懂的程式碼是會利滾利的技術債。
- **把 review 退化成橡皮圖章**。當程式碼是 AI 大量生出的，人很容易因為「看起來合理 + 測試綠」就放行。但 review 的價值正是在生成變便宜後更高——它是少數還在人手上的品質閘。
- **誤把「生成快」當「決策已驗證」**。一晚生出的可運行系統，只證明了「能組起來」，沒證明它在逆境、在你的真實負載下成立。選型仍需 PoC 打逆境（見本檔技術選型條目）。
- **生成碼悄悄引入過時或幻覺相依**。AI 的相依建議可能來自舊訓練資料，或根本是編造的套件名（被搶註成惡意套件的攻擊面）。任何 AI 給的相依與時效性事實都要回官方來源驗（見本檔自建 vs 採用條目）。
- **把錯誤處理交給 AI 的預設**。AI 傾向生出「表面處理了錯誤、實際靜默吞掉」的程式碼，逆境路徑是它最弱也最該被人盯的地方（見本檔錯誤處理哲學條目）。

## 量級估算與容量直覺

### 定義與原理

量級估算（back-of-the-envelope / napkin math）是**在不建系統的前提下，用幾個已知的硬數字，把「這個設計大概會撞到哪面牆」算到正確的數量級**。它的目的不是精確（差兩三倍無所謂），而是**及早排除明顯行不通的方案、及早發現「這裡需要 cache / 分片」的訊號**——把昂貴的「建完才發現撐不住」提前到「白板上五分鐘就知道」。

它倚賴一組刻在直覺裡的參考數字。經典的「Latency Numbers Every Programmer Should Know」量級（數量級仍成立，絕對值隨硬體演進，2026-06）：L1 cache 參考 ~0.5 ns、主記憶體參考 ~100 ns、SSD 隨機讀 4 KB ~150 µs（百微秒級）、同資料中心一次來回 ~0.5 ms、跨大陸來回 ~150 ms。換算成吞吐的天花板（2026-06 當代硬體量級）：單條 DDR5 記憶體頻寬約 48–64 GB/s（高速模組更高）；企業級 PCIe 5.0 NVMe SSD 隨機讀可達 250 萬–340 萬 IOPS（KIOXIA CM9 等 2024–2025 機種，高佇列深度下，來源見延伸閱讀），但這是裝置上限，真實應用受佇列深度、檔案系統、fsync 拖累常只摸到其零頭；10 GbE 網路理論上限 1.25 GB/s（10 Gbps ÷ 8）、實務持續吞吐約 0.9–1.18 GB/s。第一原理：**任何設計的吞吐都被它最先撞到的那面牆鎖住**——記憶體頻寬、磁碟 IOPS、網路頻寬、單核 CPU、或下游服務的處理率，五者取最小。估算就是找出那面牆在哪。

### 解法空間

- **拆成「每秒幾次操作 × 每次多大」**：把流量化成 QPS 與每次操作的資料量，分別撞到 IOPS 牆與頻寬牆。
- **記住一把錨點數字並做換算**：上面那組（記憶體 ns 級、SSD µs 級且 IOPS 百萬級、網路 GB/s 級、跨區 ms 級）。**90/10 與 2 的次方心算**：1 天 ≈ 86,400 s ≈ 10⁵ s（記成「一天約十萬秒」最好用）；2¹⁰≈10³、2²⁰≈10⁶、2³⁰≈10⁹。
- **找瓶頸（取最小天花板）**：把候選設計的每條路徑都對著對應的牆算一遍，最先撞到的就是瓶頸；其餘優化在它解掉前都沒意義。
- **算「要不要 cache」**：比較「打後端的成本（延遲/QPS/IOPS）×命中率能省多少」與「快取一致性的代價」。讀多寫少、熱點集中、可容忍些許 stale → cache 划算（機制見領域 G）。
- **算「要不要分片」**：當單機的某面牆（CPU、記憶體容量、單機 IOPS、單機連線數）擋住成長，就需要把負載切開（機制見領域 B 的分片、領域 M 的 consistent hashing）。估算告訴你「離牆還有多遠」。
- **AI 維度**：AI 能快速幫你列數字、跑換算、生估算草稿，省去查表。但它給的硬體數字常是舊訓練資料（某代 SSD、某代網路），數量級多半對、絕對值要存疑——估算重點本就在數量級，但別把它報的精確數字當權威，撞牆與否的邊界數字要自己驗。

**Worked example：一個讀取密集服務要不要加 cache、要不要分片**

假設一個服務每秒 50,000 次讀請求（50K QPS），每次讀要從資料庫撈一筆約 4 KB 的資料。

- **撞 IOPS 牆？** 若每次讀對應一次磁碟隨機讀：50K QPS → 需要 50,000 IOPS。對照企業級 NVMe 裝置上限（百萬級 IOPS）這看似輕鬆，**但**那是裝置裸測上限；真實 RDBMS 經查詢解析、鎖、MVCC、fsync 後，單機常只穩定供應數千到數萬量級的有效隨機讀。50K 已逼近單機現實上限 → **這是「需要 cache」的訊號**。
- **cache 能省多少？** 假設熱點集中、命中率 90%：打到資料庫的讀降為 50K × 10% = 5K QPS，落進單機舒適區。代價是接受快取與資料庫間的最終一致與失效複雜度（見領域 G）。**結論：加 cache 划算。**
- **撞頻寬牆？** 50K QPS × 4 KB = 200 MB/s。對照 10 GbE 上限（~1.25 GB/s）→ 用掉約 16%，**還沒撞牆**，頻寬不是這裡的瓶頸。
- **要不要分片？** 若這 50K 還會成長到 500K QPS，cache 後仍有 50K 打到資料庫，逼近甚至超過單機有效 IOPS → 此時**單機那面牆擋住成長，需要分片**把寫入/讀取分散到多台（見領域 B / M）。估算讓你在「白板上」就看出 50K 量級加 cache 能撐、500K 量級非分片不可——不必等真的塞爆才知道。

這串心算約一分鐘，省下的是「建完整套才發現單機資料庫撐不住」的數週返工。

### 各方案的保證與取捨

| 方案/做法 | 效果 | 適用場景 | 注意事項 |
|---|---|---|---|
| back-of-envelope 估算 | 數量級正確、五分鐘排除爛方案 | 設計初期、選型前、容量規劃 | 只保數量級，別當精算；錨點數字會隨硬體漂、要更新 |
| 找瓶頸（取最小天花板） | 心力集中在真正的牆 | 任何效能/容量判斷 | 撞牆順序會隨負載特性變（讀密集 vs 寫密集牆不同） |
| 估完再決定 cache | 避免無謂快取、也避免該快不快 | 讀密集、熱點集中 | 命中率是假設、要驗；一致性代價別漏算（見領域 G）|
| 估完再決定分片 | 避免過早分片、也避免太晚 | 單機某面牆擋住成長 | 分片是高成本單向門，估準「離牆多遠」再動（見領域 B/M）|

保證的本質：估算買的是**「用一分鐘的心算，換掉數週的盲目實作」**——它不保證設計對，只保證**明顯不對的設計被當場刷掉**。適用條件：在設計早期、數量級足以分勝負時最值；當方案都在同一數量級、差異要靠真實壓測才分得出時，估算就該讓位給 profiling（見領域 S）。

### 何時需要

該做估算的時刻：任何新設計動工前（先確認數量級可行）、選型時（這個方案的吞吐天花板夠不夠）、判斷要不要引入 cache / 分片 / 佇列這類複雜度時（先看離牆多遠，別憑感覺加）、容量規劃與成本估算時。

over-engineering 的訊號：在數量級明明還差很遠時就上分片、上多層快取、上複雜的非同步管線——估算會告訴你「單機還能撐十倍」，那這些複雜度就是替臆想的規模提前付帳（呼應本檔簡單性條目）。反過來，完全不估、憑感覺判斷「應該夠快 / 應該要 cache」，是把賭注下在直覺上——而人對量級的直覺，在跨好幾個數量級時經常錯得離譜。

### 常見誤解與陷阱

- **把裝置裸測上限當應用可得吞吐**。SSD 標稱百萬 IOPS，不代表你的資料庫能跑到百萬——中間隔著檔案系統、鎖、MVCC、fsync、查詢解析，真實值常是裸測的零頭。估「牆」要用**有效**吞吐，不是 spec sheet。
- **混淆 bit 與 byte**。網路常標 Gbps（bit），儲存與記憶體常標 GB/s（byte），差 8 倍。10 Gbps 是 1.25 GB/s 不是 10 GB/s——這個 8 倍誤差足以讓「撞不撞牆」的結論翻盤。
- **錨點數字過時還照用**。「Latency Numbers」的**數量級關係**（記憶體比磁碟快幾個數量級、跨區比同區慢幾個數量級）穩定，但**絕對值**隨硬體演進——SSD 早已不是當年的 SSD。引用絕對值要更新（本條的當代數字標 2026-06）。
- **忘記「最小天花板」原則**。算了 IOPS 撐得住就放心，卻沒算頻寬、CPU、或下游處理率——真正的瓶頸可能是你沒算的那面牆。要把每條路徑對每面牆都掃一遍，取最小。
- **追求精確而非數量級**。估算糾結在「是 47K 還是 52K QPS」毫無意義——它的價值在「是 5 萬還是 5 百萬」這種數量級判斷。要的是 10 的次方對，不是個位數對。
- **靜態估算忽略 ρ→1 的非線性**。把利用率推到接近 100% 時，延遲不是線性上升而是爆炸（排隊效應，直覺見領域 E）。估算「平均算得過來」不等於「尖峰扛得住」——要留餘裕，別把系統設計在貼著天花板跑。
