# K · 系統設計與解耦

這個領域回答的是：**怎麼讓系統的各部分「少知道彼此」——少知道對方在哪、用什麼寫、何時上線、內部資料長什麼樣——好讓任一部分能獨立變更、獨立失敗、獨立擴展，而不會牽一髮動全身。** 本檔含六個條目：解耦（拆解耦合的維度與各手段對應到哪一維）、耦合 vs 內聚（一體兩面的判準）、同步 vs 非同步通訊（呼叫的時間耦合）、事件驅動架構 EDA（用事件當通訊媒介，event notification vs event-carried state transfer）、Event sourcing 與 CQRS（狀態＝不可變事件序列、讀寫分離）、持久化工作流與 durable execution（把長流程的執行狀態本身持久化以便重試恢復）。脊椎：你想要哪一維的解耦（換來獨立演化、容錯、可擴展），你願意付什麼取捨（最終一致、除錯變難、間接層增加、難追蹤的全域流程）。與相鄰領域的邊界一句話：**這裡講「系統怎麼切、怎麼用事件接起來」的架構面；交付機制（outbox、saga 補償、訊息順序）由領域 A 深講，資料同步與 CDC 由領域 L 深講，一致性模型由領域 B 深講——本檔引用、不重講。**

## 解耦 decoupling

### 定義與原理

解耦要解的問題是：**當兩個元件之間的「假設」太多時，改動其中一個就會逼著另一個跟著改、跟著重新部署、甚至跟著失敗。** 耦合（coupling）就是「A 對 B 的假設量」——A 假設 B 在某個固定位址、用某種協定、永遠在線、回傳某個欄位結構、跟自己同步上線。每一條假設都是一條繩子；解耦就是有意識地剪掉某些繩子，換取「兩端能各自變動」的自由。

第一原理是：**耦合不是一個純量，而是多個正交維度**。把它們分開，才看得出「我加了一個訊息佇列」到底解了哪一條繩子、又沒解哪一條。常見的維度（沿用 Gregor Hohpe 與 Bobby Woolf 在 *Enterprise Integration Patterns* 的整理，並擴充）：

- **時間耦合（temporal）**：A 呼叫 B 時，B 必須當下就在、當下就回。同步 HTTP 呼叫是強時間耦合：B 掛了，A 這次呼叫就失敗。
- **空間耦合（spatial / location）**：A 必須知道 B 的具體位址（IP／host）。寫死 endpoint 是強空間耦合；服務發現（見領域 N）把它換成邏輯名稱。
- **同步性耦合（synchronization）**：A 發出請求後必須「卡住等」B 回來才能往下。這與時間耦合相關但不同——你可以時間解耦（B 不必當下在）卻仍同步等（A 卡著等回應）。
- **資料格式耦合（data / schema）**：A 必須知道 B 的內部資料結構。共享資料庫是最強的資料耦合：A 直接讀 B 的表，B 一改欄位 A 就裂。
- **實作耦合（implementation / technology）**：A 假設了 B 用什麼語言、什麼框架、什麼內部演算法。透過穩定的契約（API／訊息 schema）隔開，B 換實作 A 無感。

關鍵洞見：**不同的解耦手段，剪的是不同的繩子**。把「上訊息佇列」當成「萬用解耦」是最常見的誤判——佇列主要剪時間與同步性耦合，但若兩端共用同一個事件 schema，資料耦合一條都沒剪。

### 解法空間

各手段對應到它主要解的維度（一句機制）：

- **訊息佇列／非同步訊息**：A 把訊息丟進佇列就走，B 之後自己取。主要剪**時間＋同步性**耦合（B 掛了訊息留在佇列，A 不卡）。
- **服務發現／DNS／load balancer**：A 用邏輯名稱找 B，由中間層解析到實際位址。主要剪**空間**耦合（見領域 N）。
- **API 契約／schema registry／向後相容演進**：兩端只依賴「契約」而非「實作」。主要剪**資料格式＋實作**耦合（見領域 A 的序列化與 schema 演進）。
- **事件驅動（發布訂閱）**：發布者不認識訂閱者，只發事件到 topic。剪**空間（不知有誰在聽）＋時間＋同步性**（見事件驅動架構 EDA）。
- **API gateway／facade／反向代理**：在前面加一層，把後端的位址、協定、聚合細節藏起來。剪**空間＋部分實作**耦合（見領域 H）。
- **資料庫 per service（每服務私有庫）**：禁止跨服務直接讀彼此的表，只能走 API／事件。剪**資料格式**耦合（這是微服務最關鍵也最常被偷懶跳過的一條）。
- **介面／依賴反轉（DIP）**：程式碼層級，依賴抽象介面而非具體類別。剪**實作**耦合（單元層的解耦）。

### 各方案的保證與取捨

| 方案/做法 | 保證（剪哪條繩子） | 適用場景 | 注意事項 |
|---|---|---|---|
| 非同步訊息佇列 | 時間＋同步性解耦：B 暫時不在，A 照樣往下 | 工作可延後處理、可容忍最終完成、要吸收尖峰 | 換來最終一致與「訊息何時被處理」的不確定；下游要冪等（見領域 A）；除錯要追跨佇列的流程 |
| 服務發現 / DNS | 空間解耦：endpoint 變動 A 無感 | 動態擴縮、藍綠／滾動部署、多副本 | DNS 快取 TTL 會讓變更生效有延遲；發現層自己成為依賴（見領域 N） |
| API 契約 + 相容演進 | 資料格式＋實作解耦：B 換實作／加欄位 A 不裂 | 任何長期維護的服務間介面 | 契約本身要治理（破壞性變更要版本化）；過度寬鬆的契約＝沒解耦（見領域 A） |
| 事件驅動發布訂閱 | 空間＋時間＋同步性：發布者不知有誰在聽 | 一個變更要扇出給多個未知／可增的下游 | 全域流程變隱性、難一眼看出「誰觸發了誰」；事件 schema 仍是耦合點 |
| DB per service | 資料格式解耦的最強形式 | 真正要獨立演化、獨立擴展的服務 | 跨服務查詢／交易變難（要靠事件複製或 API 組合）；很多團隊偷懶共用庫，等於沒拆 |

一個帶數字的取捨。一個「使用者下單後寄確認信」的流程，原本是同步呼叫：下單 API 內直接呼叫郵件服務。郵件服務 SLA 是 **99.5%**（每月約 **3.6 小時** 不可用），下單 API 自己是 **99.9%**。同步串接後，下單成功率被郵件服務拖到 **約 99.4%**（0.999 × 0.995），因為郵件服務一掛、整個下單就失敗。把「寄信」改成丟一則訊息到佇列（剪掉時間＋同步性耦合）後，郵件服務掛掉只影響「信晚點才到」，下單本身回到 **99.9%**——這就是時間解耦用「信的即時性」換「下單的可用性」。但代價同時出現：現在「信到底寄出去沒」不再是同步可知，你得靠佇列的監控與 DLQ（見領域 A）才知道有沒有卡住。

### 何時需要

- **需要解耦**：兩個元件的變更／部署／失敗應該獨立，但現在被某條繩子綁死（改 B 的欄位 A 就裂、B 掛了 A 也掛、B 換位址 A 要改設定）。先定位「哪一維耦合在痛」，再選對應手段。
- **過度解耦是 over-engineering**：把一個本該同步、本該強一致的流程（如「扣庫存」必須當下成功才能確認訂單）硬拆成非同步事件，換來的最終一致反而製造對帳負擔與「訂單已成立但庫存沒扣」的窗口。**不是所有耦合都該剪**——強一致需求下的時間耦合是特性、不是缺陷。
- **判準**：問「這兩端真的需要獨立演化／獨立失敗嗎？」需要→剪對應維度的繩子；不需要（同一團隊、同一發布週期、同一交易邊界）→保持同步直連，省下間接層的複雜度。

### 常見誤解與陷阱

- **「加佇列＝解耦」**：佇列剪時間與同步性，但若兩端共用事件 schema，資料耦合一條沒少。改事件結構照樣兩邊一起裂。解耦要看「剪了哪一維」，不是看「有沒有中間件」。
- **把間接層當免費**：每加一層（佇列、gateway、發現層）都引入新的故障點、新的延遲、新的要監控的東西。解耦永遠是「用一種複雜度換另一種」——把「直接依賴」換成「對中間層的依賴＋對最終一致的處理」。
- **隱性耦合最危險**：共用資料庫、共用的「魔術字串」狀態值、跨服務假設對方的處理順序——這些不在 API 上，但綁得最死。真正的解耦要連這些檯面下的假設一起剪。
- **過早分散式化**：在還沒有獨立演化需求時就把單體拆成一堆非同步服務，得到的是分散式系統的全部痛（部分失敗、最終一致、難追蹤），卻沒有對應的好處。先單體、痛了再拆，通常比反過來省。

## 耦合 vs 內聚

### 定義與原理

這一條要解的問題是：**「該把什麼放在一起、什麼拆開」沒有絕對答案，但有一對互補的判準。** 耦合（coupling）＝**模組「之間」的依賴強度**，越低越好（改一個不必動另一個）。內聚（cohesion）＝**模組「內部」元素的相關程度**，越高越好（一個模組裡的東西都在服務同一個目的）。這對概念出自 Larry Constantine 與 Edward Yourdon 1970 年代的結構化設計，是「怎麼切模組／服務」最古老也最耐用的判準。

第一原理：**這兩個量必須一起看，單看一個會切錯**。只追求低耦合，可以把所有東西塞進一個巨大模組（內部沒有跨模組依賴，但內聚爛、什麼都混在一起）；只追求高內聚，可以把每個函式都拆成一個服務（每個都極專注，但彼此呼叫成網、耦合爆炸）。好的切法是**同時**：每個模組內部高內聚（職責單一），模組之間低耦合（依賴少而穩定）。

衡量的經典啟發是 **Constantine 的耦合光譜**（由弱到強：data → stamp → control → common → content coupling）與**內聚光譜**（由弱到強：coincidental → logical → temporal → procedural → communicational → functional cohesion）。實務上不必背全表，但要會分辨「兩個東西經常一起改」是內聚信號（該放一起）、「改 A 老是被迫改 B」是耦合信號（該拆開或穩定介面）。

### 解法空間

把「怎麼判斷邊界切對沒」拆成幾個可操作的視角：

- **變更軸線（axis of change）**：經常「一起改」的東西放一起（高內聚），很少一起改的拆開。一個檔／服務若每次需求變更都要動，可能塞了太多不相關職責。
- **領域邊界（DDD 的 bounded context）**：依業務語言／概念的自然接縫切，而非依技術層（不是「所有 controller 一個服務、所有 DAO 一個服務」）。
- **依賴方向**：穩定的東西被依賴、易變的東西去依賴別人（穩定依賴原則）。讓依賴箭頭指向不常變的核心。
- **連接點數量與型態**：模組對外暴露的介面越窄、越穩定，耦合越低。傳一整個物件進去（stamp coupling）比只傳需要的欄位（data coupling）耦合更強。
- **共享狀態**：共用可變狀態（common coupling，如全域變數、共享庫）是強耦合；改成各自持有、透過介面交換是弱化。

### 各方案的保證與取捨

| 方案/做法 | 效果 | 適用場景 | 注意事項 |
|---|---|---|---|
| 依「變更軸線」切 | 高內聚：常一起改的在一起，改動侷限 | 已能觀察到「哪些東西總是一起變」的成熟系統 | 早期需求不明時軸線會變，過早固化反而切錯 |
| 依領域邊界（bounded context）切 | 邊界對齊業務語言，耦合沿自然接縫最弱 | 業務概念清晰、團隊依領域分工 | 需要領域知識；技術人易誤按技術分層切，造成跨層耦合 |
| 窄而穩定的介面 | 弱化耦合到 data coupling（只傳必要欄位） | 任何跨模組／跨服務邊界 | 介面太細會囉嗦、太粗（傳大物件）又把內部結構洩漏出去 |
| 消除共享可變狀態 | 拆掉 common/content coupling（最強的耦合） | 多模組共用全域狀態或同一張表 | 改成訊息／事件交換會引入最終一致（見解耦） |

一個帶數字的場景。一個 200 個檔的單體，假設用工具量出「平均一次功能變更要動 12 個檔、分散在 5 個模組」。這是**低內聚＋高耦合**的徵狀：相關邏輯散落、改一處要連帶改四處。按變更軸線重切後，把「常一起改」的聚進同一模組，目標是「一次功能變更只動 1–2 個模組」。判準很具體：**如果你的 commit 歷史顯示某幾個檔總是一起出現在同一個 PR，它們多半該住在一起**——版本控制的共變（co-change）資料是內聚最誠實的證據，比任何架構圖都準。

### 何時需要

- **需要重新切**：每次小需求都觸發跨多模組的散彈式修改（低內聚徵狀）；或改一個模組老是弄壞另一個看似無關的（高耦合徵狀）。這兩個痛點是「邊界切錯」的直接信號。
- **別為了切而切**：一個內部高內聚、對外介面已經穩定的單體模組，沒必要因為「微服務比較潮」就拆成網路呼叫。拆開＝把行程內呼叫換成跨網路呼叫，引入部分失敗、延遲、序列化成本——只有當「獨立部署／獨立擴展／獨立團隊」的好處超過這些成本時才值得。
- **判準**：耦合與內聚是「診斷工具」不是「目標」。先看痛在哪（散彈修改？連鎖損壞？），再決定是把東西聚起來（提內聚）還是把繩子剪掉（降耦合）。

### 常見誤解與陷阱

- **「微服務＝低耦合」**：拆成服務若還共用資料庫、若事件 schema 緊綁、若部署必須同步，那是「分散式單體」——擁有單體的耦合，外加分散式的全部痛。服務邊界畫錯，比不拆還糟。
- **把技術分層當內聚**：「所有 repository 一層、所有 service 一層」是 logical cohesion（按種類分組），不是 functional cohesion（按目的分組）。改一個業務功能要橫切所有層，正是低內聚。
- **追求「零耦合」**：模組之間總要協作，耦合不可能歸零，目標是「弱且穩定的耦合」（data coupling、依賴穩定介面），不是消滅依賴。
- **內聚被「DRY 過頭」破壞**：為了不重複，把兩個「碰巧長得像但會各自演化」的東西強行抽成共用模組，製造了它們之間本不該有的耦合（共用模組一改兩邊都被迫改）。**有時刻意重複比錯誤抽象更解耦。**

## 同步 vs 非同步通訊

### 定義與原理

這一條要解的問題是：**「A 呼叫 B」這個動作，A 要不要卡住等 B 回來？** 同步（synchronous）通訊＝A 發出請求後**阻塞等待**回應，拿到結果才往下走（典型：HTTP request/response、gRPC unary、同步 RPC）。非同步（asynchronous）通訊＝A 發出請求／訊息後**立刻往下走**，不等回應；結果（若有）之後透過回呼、輪詢、或另一條訊息送回（典型：丟訊息到佇列、發事件、webhook 回呼）。

第一原理：**這個選擇決定了「失敗如何傳遞」與「兩端的命運是否綁在一起」**。同步呼叫下，B 的延遲就是 A 的延遲、B 的失敗就是 A 當下的失敗——兩端在這次呼叫期間命運共同體（強時間耦合，見解耦）。非同步下，A 把「請求」交給一個中介（佇列／事件匯流排），B 之後自己取；B 慢、B 掛，A 都不直接受影響，但代價是 A 不再當下知道結果，整個系統進入最終一致。

要釐清一個常被混淆的點：**「同步 vs 非同步」是「呼叫者要不要等」，與「程式碼用不用 async/await 語法」是兩回事**。Node.js 用 `await fetch()` 在語言層是非阻塞的（event loop 可去做別的，見領域 D），但在**通訊語意**上它仍是同步呼叫——這個邏輯流程卡在這裡等 HTTP 回應才能繼續。真正的非同步通訊是「發了訊息就不等這個邏輯流程的回應」。

### 解法空間

通訊的「同步性」其實是一條光譜，不只二選一：

- **純同步 request/response**：A 阻塞等 B 回應。機制：一來一回，A 直接拿到結果或錯誤。
- **同步但有逾時／斷路器**：仍等回應，但設上限與保護（見領域 P 的 timeout、領域 E 的 circuit breaker）。機制：等，但不無限等、不把失敗放大。
- **fire-and-forget 非同步**：A 發出就走，不要任何回應（如發一個「使用者登入了」事件）。機制：單向，A 不在乎結果。
- **非同步 + 回呼（callback / webhook）**：A 發請求附一個回呼位址，B 處理完反向呼叫 A。機制：結果之後反向送回（webhook 見領域 A）。
- **非同步 + 輪詢（polling / request-reply）**：A 發請求拿到一個 job id，之後週期性查「好了沒」。機制：A 主動拉結果。
- **非同步 request-reply over 佇列**：A 把請求丟到請求佇列、附一個 correlation id 與回覆佇列；B 處理完把結果丟回覆佇列。機制：用兩條佇列模擬「要回應但不阻塞」。

### 各方案的保證與取捨

| 方案/做法 | 保證 | 適用場景 | 注意事項 |
|---|---|---|---|
| 純同步 request/response | 即時拿到結果或錯誤；流程線性、最好懂、最好除錯 | 需要當下結果才能往下（查餘額、驗證、即時讀） | A 的延遲＝Σ 下游延遲；下游掛 A 當下失敗；同步鏈越長尾延遲越糟（見領域 S） |
| fire-and-forget 非同步 | A 不被 B 拖累；B 可吸收尖峰、可重試 | 通知、稽核、可延後的副作用、扇出 | A 不知結果；要靠監控／DLQ 才知有沒有卡（見領域 A）；最終一致 |
| 非同步 + webhook 回呼 | 不阻塞又能拿到結果；省輪詢 | 處理耗時、結果重要（金流、第三方批次） | 回呼可能丟失／重複／亂序，要簽章＋冪等（見領域 A 的 webhooks） |
| 非同步 + 輪詢 | 簡單、A 完全掌控節奏、無需暴露回呼端點 | A 在防火牆後、無法被回呼 | 輪詢間隔＝結果延遲；高頻輪詢空轉浪費；要設輪詢上限避免無限等 |

一個帶數字的取捨。一個 API 要呼叫三個下游：認證（p99 **20 ms**）、定價（p99 **50 ms**）、庫存（p99 **80 ms**）。若**同步串接**，A 的 p99 至少是三者相加再加自身——約 **150 ms** 起跳，且只要任一個下游在某次請求落到長尾，A 就被拖到長尾（尾延遲在串接下會放大，見領域 S）。若庫存查詢其實「不影響是否回應、只是預熱快取」，把它改成 **fire-and-forget 非同步**（丟個訊息讓背景去做），A 的同步路徑就只剩認證＋定價約 **70 ms**，砍掉近半。**判準很清楚：只有「這次回應真的需要的結果」才該同步等；其餘副作用都該推到非同步路徑。** 但別忘了代價——那筆庫存預熱現在是「最終會做」，若它失敗了，A 不會知道，要靠獨立監控補上。

### 何時需要

- **該同步**：呼叫者需要結果才能決定下一步（驗證、授權、讀取當前值、需要強一致地「現在就確認成功」）；或流程短、下游可靠、簡單性壓倒一切。
- **該非同步**：副作用可延後（通知、稽核、索引更新）；要吸收流量尖峰（佇列當緩衝）；要扇出給多個下游；下游慢或不穩、不想被它拖累可用性；流程本身就很長（見持久化工作流與 durable execution）。
- **判準**：對每個下游呼叫問「這次回應**需要**它的結果嗎？」需要→同步（並配逾時／斷路器）；不需要→非同步推走。不要因為「非同步比較快」就把需要強一致確認的步驟也推成最終一致。

### 常見誤解與陷阱

- **把 `async/await` 語法當成「非同步通訊」**：語言層非阻塞 ≠ 通訊語意非同步。`await` 一個 HTTP 呼叫，邏輯流程仍卡在這裡等回應、仍是同步通訊、仍承擔下游的延遲與失敗。
- **同步鏈過長**：A→B→C→D 全同步，A 的可用性＝各環節可用性相乘、延遲＝各環節相加、尾延遲被逐級放大。深同步呼叫鏈是脆弱性與長尾的主要來源（見領域 S 的 tail latency）。
- **非同步＝沒有錯誤處理**：fire-and-forget 不代表可以忽略失敗，只是失敗從「同步拋給呼叫者」變成「事後在佇列／DLQ 裡」。沒有 DLQ 與監控的非同步，是把錯誤掃到地毯下（見領域 A 的 DLQ）。
- **用非同步逃避強一致需求**：把「必須當下成功」的步驟（如扣款後才確認訂單）硬改非同步，製造「訂單已成立、款還沒扣」的不一致窗口。非同步換來的是可用性與吞吐，代價是放棄即時一致——別用在不能放棄的地方。

## 事件驅動架構 EDA

### 定義與原理

事件驅動架構（Event-Driven Architecture, EDA）要解的問題是：**怎麼讓「某件事發生了」這個事實，能被任意多個（包含未來才出現的）關心者知道，而事實的產生者不必認識任何一個關心者。** 核心轉變是把通訊媒介從「命令」（command：我叫你做某事，我知道你是誰）換成「事件」（event：某件事發生了，我廣播這個事實，誰要聽自便）。**事件是過去式、是已發生的事實**（`OrderPlaced`、`PaymentCaptured`），它陳述「發生過什麼」而不是「請你做什麼」——這個語法差別正是解耦的來源：產生者只負責「說發生了什麼」，怎麼反應是聽眾的事。

Martin Fowler 在〈What do you mean by 'Event-Driven'?〉（2017-01）裡把常被混為一談的「事件驅動」拆成**四種不同的模式**：**event notification**（事件通知）、**event-carried state transfer**（事件攜帶狀態傳遞）、**event sourcing**、**CQRS**。後兩者本檔另條深講（見 Event sourcing 與 CQRS）；這一條聚焦前兩者——它們是「用事件當通訊」最常見的兩種風格，差別在**事件裡裝多少資料**。

- **event notification（事件通知）**：事件只帶「某事發生了＋一個 id」，幾乎不帶資料。訂閱者收到後若需要細節，**回頭去呼叫來源**問。好處是事件小、來源資料庫仍是單一真相；壞處是製造了「回頭查」的反向流量與隱性耦合（訂閱者又得認識來源）。
- **event-carried state transfer（事件攜帶狀態傳遞）**：事件帶上訂閱者「會用到的全部資料」，訂閱者**不必回頭查**，自己存一份副本。好處是訂閱者徹底解耦於來源、效能好、來源負載低、可用性高（來源掛了訂閱者照樣運作）；壞處是**資料被複製、且引入最終一致**——訂閱者那份副本永遠晚於來源一點點（見領域 L 的 event-carried state transfer）。

### 解法空間

EDA 的設計選擇主要落在幾個軸上：

- **事件裝多少資料**：thin（只帶 id，event notification）↔ fat（帶完整狀態，event-carried state transfer）。中間有「帶部分常用欄位」的折衷。
- **傳輸基礎建設**：broker-based（Kafka／RabbitMQ／SNS+SQS 等中介，見領域 A）↔ brokerless。broker 決定持久性、順序、重放能力。
- **topic 粒度**：粗（一個 topic 一堆事件型別）↔ 細（一個事件型別一個 topic）。影響訂閱者的過濾負擔與耦合。
- **事件契約治理**：事件 schema 是公開契約，要向後相容演進（見領域 A 的序列化與 schema 演進）。事件一旦發出去就是「已發布的 API」。
- **訂閱模型**：competing consumers（一群 worker 分攤，每則只一個處理）↔ pub/sub fan-out（每個訂閱者都收到一份，見領域 A 的 SNS／consumer group）。

### 各方案的保證與取捨

| 方案/做法 | 保證 | 適用場景 | 注意事項 |
|---|---|---|---|
| event notification（thin event） | 事件小、來源是單一真相、不複製資料 | 事件頻繁、訂閱者只偶爾需要細節、資料敏感不宜廣播 | 訂閱者要「回頭查」→ 反向流量＋仍耦合來源；來源掛了訂閱者也卡；查到的可能已是更新後的值（時序坑） |
| event-carried state transfer（fat event） | 訂閱者完全自足、來源負載低、來源掛了仍可運作 | 訂閱者頻繁需要資料、要高可用、要跨服務複製狀態 | 資料複製＋最終一致；fat event 變大；事件 schema 變更影響面廣（見領域 L） |
| broker-based（Kafka 等） | 持久化、可重放、有序（partition 內）、解耦 | 要重放歷史、多下游、要保證不丟 | 運維 broker；順序只在 partition 內保證（見領域 A） |
| pub/sub fan-out | 一個事件扇出給 N 個未知／可增訂閱者 | 一個變更要驅動多個獨立反應 | 每個訂閱者各自要冪等；扇出放大下游負載（見領域 A 的 SNS） |

一個帶數字的取捨。一個訂單服務每秒產生 **1,000 筆** `OrderPlaced` 事件，有三個下游：出貨、發票、推薦。

- 用 **event notification**（事件只帶 `orderId`）：每筆事件約 **100 bytes**，三個下游各自要回頭呼叫訂單服務拿明細。下游負載：訂單服務每秒被多打 **3,000 次** 查詢（3 下游 × 1,000）——事件雖小，反向查詢流量是事件量的三倍，且訂單服務成了所有下游的同步依賴。
- 用 **event-carried state transfer**（事件帶完整訂單，約 **2 KB**）：頻寬從 100 KB/s 升到 2 MB/s，但**三個下游零回頭查**，訂單服務不再被下游拖累，且它掛了的時候三個下游照樣能處理手上的事件。

判準：**反向查詢的成本（流量＋耦合）vs 複製資料的成本（頻寬＋最終一致）**。下游越多、越頻繁需要資料，越偏向 fat event；事件超大或資料敏感、下游又很少真的需要細節，才用 thin event。

### 何時需要

- **需要 EDA**：一個變更要驅動多個、可能未來才增加的獨立反應（下單→出貨＋發票＋推薦＋通知…）；要時間／空間解耦；要能事後重放歷史驅動新功能。
- **EDA 是 over-engineering 的時候**：流程是線性的、反應者固定就一兩個、且需要強一致的當下確認。為一個「A 做完叫 B 做」的簡單同步流程套上事件匯流排，得到的是難以追蹤的隱性流程，沒有對應好處。
- **判準**：問「關心這件事的人是固定且少數、還是會增長且未知？」固定少數→直接呼叫可能更清楚；增長未知→事件解耦讓你加訂閱者不必動產生者。

### 常見誤解與陷阱

- **event 與 command 混用**：把 `SendEmail`（命令，我知道誰該做、期待它做）當事件發，等於偷渡了耦合——產生者其實知道下游、期待特定反應。真事件是 `OrderPlaced`（事實），誰反應、怎麼反應與產生者無關。命名用過去式事實能逼自己想清楚。
- **隱性流程難追蹤**：EDA 把「A 觸發 B 觸發 C」藏進事件流，沒有一個地方能一眼看出端到端流程。複雜的事件鏈會變成「沒人完整知道一筆訂單會觸發什麼」的迷宮——要靠分散式追蹤（見領域 I）與事件文件補回可見性。
- **event notification 的時序坑**：訂閱者收到「something changed, id=123」後回頭查，查到的可能已經是**再下一次更新後的值**（事件在途中，來源又被改了），導致處理的不是事件當下的狀態。fat event 沒這個問題（狀態就在事件裡）。
- **把事件 schema 當內部結構隨意改**：事件一發布就是公開契約，所有訂閱者依賴它。破壞性地改欄位＝悄悄弄裂所有下游。事件 schema 要當 API 治理、向後相容演進（見領域 A）。
- **以為 EDA 自動帶來最終一致以外的好處**：解耦、可擴展是真的，但你**買下了**最終一致、隱性流程、重複投遞（要冪等，見領域 A）這一整包代價。EDA 不是免費的解耦，是「用一組複雜度換另一組」。

## Event sourcing 與 CQRS

### 定義與原理

這一條要解的問題是：**傳統做法只存「現在的狀態」（帳戶餘額＝500），歷史被覆蓋掉、永遠回答不了「它怎麼變成 500 的、某個時間點是多少」。** Event sourcing（事件溯源）翻轉這件事：**不存當前狀態，而存導致這個狀態的「不可變事件序列」**（`Deposited(100)`、`Withdrew(30)`、`Deposited(430)`…），當前狀態是「把所有事件依序套用（fold/replay）算出來的結果」。事件是 append-only 的事實，永不修改、永不刪除——這是它與「存當前狀態再 UPDATE」的根本差別。CQRS（Command Query Responsibility Segregation，命令查詢職責分離）由 Greg Young 提出與命名（約 2006–2010），Martin Fowler 推廣：**把「改狀態的寫入模型（command）」與「讀資料的查詢模型（query）」拆成兩個分開的模型**，各自最佳化。兩者常一起出現但**互相獨立**：可以只用 CQRS 不用 event sourcing，也可以反過來；但**用 event sourcing 幾乎一定要配某種 CQRS**——因為「append-only 事件流」不適合直接查詢，得另建一個讀模型。

第一原理：

- **Event sourcing 的核心保證**：事件是不可變的真相，**完整的審計軌跡天然存在**（每次變更都留痕）、**可時光回溯**（replay 到任一時點的狀態）、**可重建任意讀模型**（同一份事件可投影成多種視圖）。代價：讀「當前狀態」要 replay（靠 snapshot 加速）、事件 schema 演進棘手（舊事件永遠在、得永遠看得懂）。
- **CQRS 的核心保證**：讀寫模型分開後，**寫端可為一致性／交易最佳化、讀端可為查詢／反正規化最佳化**，各自獨立擴展（讀多寫少時讀端可大量複製）。代價：兩個模型之間要同步，通常是**最終一致**（寫完到讀模型更新有延遲，見領域 B 的一致性光譜）。

### 解法空間

圍繞「狀態＝事件序列」與「讀寫分離」的具體做法：

- **純 CQRS（不溯源）**：寫模型仍存當前狀態（正規化的關聯表），但另維護一份為查詢最佳化的讀模型（反正規化視圖／物化視圖／搜尋索引），兩者透過事件或 CDC 同步。機制：一份權威狀態＋一份查詢副本。
- **Event sourcing + 投影（projection）讀模型**：事件流是唯一真相，背景把事件投影成各種讀模型（一個給列表頁、一個給詳情頁、一個給報表）。機制：事件 → 投影器 → 多個讀視圖。
- **snapshot 加速**：定期把「replay 到某版本的狀態」存成 snapshot，之後只需從 snapshot 往後 replay 少數事件。機制：用快照截斷 replay 長度。
- **事件投遞給讀端的管線**：寫端 append 事件後，怎麼可靠地讓讀端／投影器看到——常用 outbox（見領域 A）或從事件儲存的 log 訂閱、或 CDC（見領域 L）。機制：把事件可靠搬到讀模型。
- **事件儲存（event store）**：專用的 append-only 儲存（如 EventStoreDB／Kurrent），或用 Kafka 的 log、或關聯表當 events 表。機制：保證事件有序、不可變、可依 stream 讀回。

### 各方案的保證與取捨

| 方案/做法 | 保證 | 適用場景 | 注意事項 |
|---|---|---|---|
| 純 CQRS（讀寫分離、不溯源） | 讀寫各自最佳化與擴展；讀端可大量複製 | 讀遠多於寫、查詢型態與寫入型態差異大 | 讀模型最終一致（寫完讀不到自己剛寫的，要處理 read-your-writes，見領域 B）；兩模型同步成本 |
| Event sourcing + 投影 | 完整審計軌跡、可時光回溯、可重建任意讀視圖 | 法遵／審計要求高、需歷史分析、需多種視圖 | 讀當前狀態要 replay（靠 snapshot）；事件 schema 演進難（舊事件永存）；學習曲線陡 |
| snapshot 加速 | 把 replay 成本從 O(事件數) 降到 O(snapshot 後的事件數) | event-sourced 系統的長 stream | snapshot 也要隨事件 schema 演進；snapshot 是衍生物可重建、但別當真相 |
| 事件儲存（專用 event store） | append-only、有序、stream-级讀回、樂觀並發（版本檢查） | 認真做 event sourcing | 多一個要運維的儲存；或用既有 DB/Kafka 替代各有取捨 |

一個帶數字的場景。一個帳戶在 **3 年** 間累積了約 **5,000 筆** 交易事件。要算「當前餘額」，純 replay 要把 5,000 筆依序套用——若每筆套用 **0.02 ms**，約 **100 ms**，對高頻讀取太慢。引入 snapshot：每 **500 筆** 存一次當前狀態 snapshot，那麼讀「當前餘額」最多只需從最近的 snapshot 往後 replay **< 500 筆**（約 10 ms 以內）。而「審計」這件事在傳統「只存餘額」的系統裡幾乎做不到——你只有當前 500，問「2 年前的某天餘額多少」要靠另外維護的 log；在 event-sourced 系統裡，replay 到那天的事件即得，**審計能力是天生的、零額外成本**。這就是 event sourcing 的核心交換：**用「讀當前狀態變貴」換「歷史與審計變免費」。**

跨領域接點：把事件從寫端可靠搬到讀端／其他服務，常靠 **outbox**（與 append 同交易寫出，見領域 A）或 **CDC**（從事件表的 log 擷取變更，見領域 L 的 CDC）；讀寫模型之間的最終一致與「讀己之寫」屬一致性模型問題（見領域 B 的一致性光譜）。Kafka 的 log 結構天然適合當事件流（見領域 A 的 Kafka）。

### 何時需要

- **需要 event sourcing**：審計／法遵要求「每一次變更都要可追溯、不可竄改」（金融、醫療、帳務）；需要時光回溯或重放歷史驅動新分析；同一份資料要餵多種差異很大的視圖。
- **需要 CQRS（可不溯源）**：讀寫負載／型態嚴重不對稱（讀爆量、查詢複雜、寫入簡單），單一模型同時服務兩者很彆扭。
- **明確是 over-engineering**：簡單 CRUD、無審計需求、讀寫對稱、團隊不熟這套——硬上 event sourcing 會引入 replay、snapshot、事件演進、最終一致一整包複雜度，換來用不上的能力。**這是最常被濫用的兩個模式之一**：大多數系統用「存當前狀態＋一張審計表」就夠，不需要把整個狀態翻成事件流。
- **判準**：問「我真的需要『每次變更的完整不可變歷史』嗎？」是→考慮 event sourcing；問「讀寫負載／模型是否嚴重不對稱到單一模型很痛？」是→考慮 CQRS。兩個問題都否，就別碰。

### 常見誤解與陷阱

- **以為 CQRS 一定要配 event sourcing**：兩者獨立。純 CQRS（寫端存當前狀態＋讀端反正規化視圖）很常見也輕量，不必把狀態翻成事件流。把兩者綁死是過度複雜化的常見起點。
- **低估事件 schema 演進**：傳統表改欄位是一次性遷移；event-sourced 系統裡**舊事件永遠存在、永遠要看得懂**。改事件結構要靠 upcasting（讀舊事件時轉成新格式）等技巧，這是長期維護最痛的點。
- **忽略讀模型的最終一致**：CQRS 下寫完，讀模型更新有延遲，使用者「剛改完卻讀到舊值」。要嘛對關鍵讀走寫端、要嘛用 read-your-writes 技巧（見領域 B），不能假裝讀寫即時一致。
- **把 snapshot 當真相**：snapshot 是「事件 replay 的快取」，是衍生物、可從事件重建。若把 snapshot 當權威來源去改它，就破壞了「事件是唯一真相」的核心保證。
- **事件裡塞「命令意圖」或可變引用**：事件該是已發生的不可變事實（含當時的值），不該存「指向會變的外部資料的引用」——否則 replay 出來的歷史會隨外部資料變動而失真。事件要自足記錄「當時發生了什麼」。

## 持久化工作流與 durable execution

### 定義與原理

這一條要解的問題是：**一個跨越數秒、數天、甚至數月的長流程——下單 → 等付款 → 扣庫存 → 安排出貨 → 等簽收 → 結案——中間任一步崩潰（行程被殺、機器重啟、部署滾動），怎麼讓它「從斷點繼續」而不是從頭重來、也不是卡在半路？** 傳統做法是把流程狀態散在資料庫欄位裡（`order.status = 'awaiting_payment'`），靠一堆 cron／重試／補救 job 推進——這種「狀態機散在各處」的做法極易出現「卡在某狀態沒人推」「重試把已完成的步驟又做一遍」的洞。

Durable execution（持久化執行）翻轉這件事：**把「程式的執行狀態本身」持久化**。你用一般的程式碼寫流程（順序、迴圈、if、呼叫子步驟），但底層引擎把**每一步的輸入、輸出、決策**都記進一個持久化的事件歷史（history）。當行程崩潰後在別處重啟，引擎**重放（replay）這段歷史**，把你的函式「快轉」回崩潰前的那一行，已完成的步驟**不重做**（直接從歷史取回結果），從斷點繼續往下。對你的程式碼來說，它「感覺不到」自己崩潰過——這就是「durable」的意思：執行狀態像資料一樣持久。代表系統：**Temporal**（開源，MIT 授權，由 Uber Cadence 的原班人馬 Maxim Fateev 與 Samar Abbas 創立的 Temporal Technologies 維護，2026-06 持續活躍）、**AWS Step Functions**（全託管，用 ASL 定義狀態機）、Azure Durable Functions、Cadence 等。

第一原理：**它把「重試與恢復」從你的業務邏輯裡抽出來，下沉成平台保證**。你不再手寫「這步失敗了存個狀態、之後 cron 來重試」，而是寫一個「看起來像同步、實則可中斷可恢復」的函式，崩潰恢復由引擎負責。代價是你的工作流程式碼必須**deterministic（決定性）**——replay 時每次走的路徑要一致，所以不能直接在工作流裡 `new Date()`、`Math.random()`、直接打 I/O（這些要透過引擎提供的 activity／side-effect 封裝，讓結果被記進歷史、replay 時取回同一值）。

### 解法空間

讓長流程能跨崩潰恢復的辦法，由「手刻」到「平台化」：

- **手刻狀態機 + 持久化欄位**：用 DB 欄位記流程狀態，cron／worker 輪詢推進，每步自己處理重試與冪等。機制：狀態在資料、推進靠輪詢。
- **saga 編排（orchestration）**：一個協調者依序呼叫各步驟，失敗時逆序執行補償交易（compensating transaction）回滾已做的（見領域 A 的 outbox／saga）。機制：用「補償」抵銷而非「回滾」，因為跨服務無分散式交易。
- **durable execution 引擎（Temporal／Step Functions 等）**：把執行狀態（事件歷史）持久化，崩潰後 replay 回斷點。機制：持久化「程式走到哪、每步結果是什麼」，恢復＝重放歷史。
- **持久化計時器／長等待**：流程要「等 3 天」或「等外部事件（付款回呼）」時，引擎持久化這個等待，期間不佔執行資源，到時或事件到達才喚醒。機制：等待本身也是持久化狀態。

durable execution 與 saga 的關係常被搞混：**它們解的層次不同，且互補**。saga 是一個**模式**（怎麼在沒有分散式交易下維持跨服務一致性——靠補償），它不關心「協調者自己崩潰了怎麼辦」。durable execution 是一個**執行基礎建設**（怎麼讓任何長流程——包括一個 saga 的協調者——崩潰後可靠恢復）。實務上常見的組合是：**用 durable execution 引擎來實作 saga 的協調者**——這樣連「協調者執行到一半掛掉」都被引擎的 replay 兜住，補償邏輯本身也保證會被執行完。saga 的補償語意見領域 A 的 outbox／saga。

### 各方案的保證與取捨

| 方案/工具 | 保證 | 適用場景 | 注意事項 |
|---|---|---|---|
| 手刻狀態機 + cron 推進 | 無框架依賴、簡單流程下最直接 | 流程短、步驟少、團隊不想引入新平台 | 「卡在某狀態沒人推」「重試重做已完成步驟」靠你自己防；步驟一多狀態爆炸、難維護 |
| saga 編排 + 補償 | 跨服務最終一致：要嘛全做完、要嘛補償回滾 | 跨多服務、無分散式交易、可定義補償動作 | 補償邏輯難寫（有些動作無法真正撤銷，如「已寄出的信」）；協調者自身的可靠性要另解（見領域 A） |
| Temporal（durable execution） | 執行狀態持久化、崩潰自動 replay 回斷點、已完成步驟不重做、持久化長等待 | 長流程、多步驟、要可靠恢復、要可見性 | 工作流碼必須 deterministic（不可直接用時間／隨機／I/O）；要運維 Temporal 叢集或用 Cloud；學習曲線 |
| AWS Step Functions（Standard） | 全託管、Standard workflow 對狀態轉移提供 exactly-once 語意、最長可跑 **1 年**、可審計 | AWS 生態、不想自運維、流程可用 ASL 狀態機表達 | Standard 每次轉移有成本（高頻短流程貴）；Express workflow 改 at-least-once、最長 **5 分鐘**、需冪等（2026-06） |

一個帶數字的取捨。一個「下單後等付款，逾時 **30 分鐘** 未付則取消」的流程。

- **手刻**：下單時寫 `status=awaiting_payment`、`expire_at=now+30min`，再跑一個 cron 每分鐘掃「`expire_at < now` 且仍 awaiting」的訂單去取消。問題：cron 掛了一段時間 → 一批訂單該取消沒取消；cron 重疊跑 → 同一訂單可能被取消邏輯跑兩次（要冪等防）；流程一複雜（付款後還要扣庫存、排出貨…）狀態欄位與 cron 規則爆炸。
- **durable execution**：工作流碼直接寫「`await sleep('30min')`；醒來檢查付款狀態，未付就取消」。那個 30 分鐘的等待被引擎**持久化**——worker 在這 30 分鐘可以全部重啟、可以擴縮、可以部署新版，醒來時引擎 replay 把流程接回那一行繼續。一個流程實例的等待**不佔執行緒、不靠輪詢**。假設有 **10 萬筆** 在途訂單同時各自等 30 分鐘：手刻 cron 每分鐘要掃這 10 萬筆判斷誰到期；durable execution 則是 10 萬個持久化計時器，到期才精準喚醒對應實例，沒有全表掃描。**判準：流程越長、步驟越多、「等待」越久越多，手刻狀態機的維護與可靠性成本上升越快，durable execution 的「把恢復下沉成平台保證」越划算。**

### 何時需要

- **需要 durable execution**：多步驟、跨服務、含長等待（等付款、等審批、等外部回呼）的長流程；要求「中途崩潰能精確從斷點恢復、已完成步驟不重做」；流程邏輯複雜到散在 cron／狀態欄位裡已難維護。
- **明確是 over-engineering**：單步或兩三步的短同步流程、無長等待、無跨服務協調——套上 durable execution 引擎是用大砲打蚊子，多一個要運維／學習的平台。一個簡單的「呼叫 A 再呼叫 B」用同步呼叫＋逾時重試就夠。
- **判準**：問「這流程會不會跨越行程生命週期（含部署、重啟）、含分鐘／天級的等待、且崩潰後必須精確恢復？」三者都中→durable execution；只是個短請求→別碰。

### 常見誤解與陷阱

- **以為 durable execution＝saga 的替代品**：兩者不同層。saga 是「跨服務一致性怎麼維持（補償）」的模式；durable execution 是「長流程怎麼可靠執行與恢復」的基礎建設。**最佳實務常是用後者實作前者**——別把它們當二選一。
- **在工作流碼裡直接用時間／隨機／I/O**：durable execution 靠 replay 恢復，要求工作流碼 deterministic。直接 `new Date()`、`Math.random()`、直接打 HTTP，會讓 replay 時走的路與當初不同，狀態錯亂。這些副作用必須透過引擎的 activity／side-effect 封裝（結果記進歷史、replay 時取回同一值）。
- **把 activity 當成「保證只跑一次」**：durable execution 保證「工作流的編排邏輯」精確恢復，但**activity（實際的外部呼叫）本身通常是 at-least-once**——崩潰時機可能讓同一個 activity 被重試。activity 的副作用（扣款、發信）仍須冪等（見領域 A 的冪等）。Step Functions Standard 對「狀態轉移」是 exactly-once，但它呼叫的下游動作是否冪等仍是你的責任。
- **誤判 Step Functions 的工作流型別**：Standard 是 exactly-once、可跑到 1 年、適合非冪等關鍵流程；Express 是 at-least-once、上限 5 分鐘、為高吞吐短流程，套錯型別（拿 Express 跑非冪等的金流）會出重複執行（2026-06）。
- **以為「持久化執行」就不必想冪等與補償**：引擎兜住「協調邏輯的恢復」，但「已經對外造成的真實副作用」（信已寄、款已扣）它撤不回——那仍要靠冪等防重、靠補償邏輯回滾（有些動作根本無法撤銷，設計時就要避免）。
