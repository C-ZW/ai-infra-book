# E · 過載與流量控制

當需求超過系統當下能處理的量，問題不再是「怎麼算對」而是「怎麼決定誰被慢、誰被拒、誰被丟」——過載是一個關於**分配稀缺處理能力**的問題。本領域回答的是：在流量打超容量時，你想保住什麼保證（公平、低延遲、不雪崩），願意付什麼代價（拒絕一部分請求、丟資料、增加複雜度）。本檔含六個條目：背壓（讓壓力沿管線往上游傳）、rate limiting（主動限制進入速率，本書限流的 owning 條目）、load shedding（過載時主動丟低價值工作）、circuit breaker（對下游故障快速失敗，本書三態機制的 owning 條目）、retry storm 與 metastable failure（重試如何把小故障放大成持續性崩潰）、排隊直覺（為什麼利用率逼近 1 時延遲爆炸）。邊界：重試的退避與 jitter 做法在領域 A（本檔只講重試的「放大」效應）；通用 fallback 與降級在領域 P；timeout／deadline／budget 在領域 P；閘道層做限流的整合在領域 H（機制在本檔）。

## 背壓 backpressure

### 定義與原理

背壓指的是：當下游消費者處理不過來時，把「慢下來」這個訊號沿著管線**往上游傳遞**，讓生產者降速或暫停，而不是讓中間的緩衝無限長大。第一原理只有一條守恆律——**在任何穩定的管線裡，長期生產速率不可能超過最慢一段的處理速率**。若生產者不理會這條守恆律繼續灌，多出來的工作只能去三個地方：堆在一個會無限長大的佇列（直到記憶體耗盡）、被阻塞擋在門外（生產者被迫等待）、或被丟棄（部分工作永遠不做）。背壓就是「如何選擇並執行這三條路其中之一」的機制。

關鍵的觀念轉變：**一個沒有背壓的系統不是「沒有上限」，而是「把上限藏在記憶體耗盡那一刻」**——它會跑得好好的，直到某次流量尖峰讓無界佇列吃光記憶體、行程被 OOM killer 殺掉，從「慢一點」直接跳到「全死」。背壓把這個隱藏的懸崖換成一條可控的、漸進的減速曲線。

### 解法空間

處理「下游跟不上」的辦法本質上就是上面三條路，每條再細分：

- **有界佇列 + 阻塞（blocking / bounded buffer）**：佇列設一個固定上限，滿了之後 `put` 操作阻塞，生產者執行緒被擋住，自然降速。壓力透過「生產者被卡住」這件事往上傳。
- **有界佇列 + 拒絕（reject / fail-fast）**：佇列滿了直接回錯誤（如 HTTP 429 / 503），由更上游決定要重試還是放棄。把背壓變成一個明確的回應，而非隱性阻塞。
- **基於需求的拉取（demand-based pull）**：消費者主動告訴生產者「我現在還能收 N 筆」，生產者只送這麼多。Reactive Streams 規格就是把這個「需求訊號」做成一個非阻塞的控制通道（`request(n)`），所有緩衝都有界且由訂閱端控制。
- **取樣 / 丟棄最舊（sampling / drop-oldest）**：對「只要最新值」的串流（如即時報價、感測器讀數），滿了就丟掉舊的留新的，犧牲完整性換新鮮度與有界記憶體。
- **批次 / 合併（batch / coalesce）**：把多筆工作合併成一筆處理，提高下游每單位時間的有效吞吐，從另一端緩解壓力（見領域 S 的批次／coalescing）。

### 各方案的保證與取捨

| 方案/做法 | 保證 | 適用場景 | 注意事項 |
|---|---|---|---|
| 有界佇列 + 阻塞 | 記憶體有界；不丟資料 | 生產者可被安全地慢下來（同步管線、可降速的上游） | 阻塞會沿呼叫鏈往上傳，可能反過來卡住整個請求執行緒池；event-loop 模型裡阻塞是大忌 |
| 有界佇列 + 拒絕 | 記憶體有界；延遲有上界 | 對外 API、上游能處理 429／重試 | 把壓力推給上游，需上游有退避（否則變 retry storm，見 retry storm 條目）；丟掉的是「最新進來」的請求 |
| demand-based pull | 記憶體有界；端到端流量守恆 | 串流處理、Reactive 管線、gRPC streaming | 需整條鏈都支援拉取語意；單一不支援的環節就會在那裡退化成無界緩衝 |
| 丟棄最舊 / 取樣 | 記憶體有界；資料保新鮮 | 只在乎最新值的遙測／即時資料 | 會丟資料——不可用於每筆都要處理的場景（金流、訂單） |
| 批次 / 合併 | 提升有效吞吐 | 寫入可合併、下游有 per-call 固定成本 | 增加單筆延遲（要等湊批）；不解決根本的速率不匹配，只是把天花板抬高 |

worked example：一個 ingestion 服務每秒收 12,000 筆事件，但下游寫入只能消化 10,000 筆/秒。若用無界佇列，每秒淨積壓 2,000 筆——假設每筆事件物件約佔 1 KiB，10 分鐘就積壓 `2,000 × 600 = 1,200,000` 筆，約 1.1 GiB，足以在記憶體有限的容器裡觸發 OOM。換成上限 50,000 筆的有界佇列（約 50 MiB）+ 拒絕策略：佇列在約 `50,000 / 2,000 = 25` 秒內填滿，之後每秒拒絕約 2,000 筆（回 429），記憶體穩定在 50 MiB，由上游決定這 2,000 筆要退避重送還是丟。系統從「10 分鐘後猝死」變成「持續以 10,000/秒 穩定運作 + 明確拒絕超量」。

### 何時需要

- 任何「生產速率可能超過消費速率、且兩者解耦（中間有佇列）」的管線——這幾乎涵蓋所有訊息佇列消費者、串流處理、WebSocket 廣播扇出。
- 生產者與消費者速率天生不匹配且尖峰不可控（外部流量、上游批次倒灌）時，背壓是唯一能避免無界積壓的結構性手段。
- over-engineering 警訊：若整個管線是同步請求-回應、沒有中間佇列（如一個簡單的 HTTP handler 直接查 DB 回傳），背壓已經隱含在「連線數有限 + DB 連線池有界」裡，不必再額外造一層顯式背壓。先確認管線哪裡真的有「無界緩衝」風險，再針對那一點加保護。

### 常見誤解與陷阱

- **「加大佇列就解決了」**：加大佇列只是把懸崖往後挪，並沒有改變「生產 > 消費」這條不等式。佇列越大，積壓時的延遲越糟（一筆請求要排在前面所有人後面），最後不是更晚 OOM 就是延遲爆到等同逾時。佇列是緩衝尖峰用的，不是用來吸收持續性超載的。
- **阻塞式背壓在 event-loop 裡反咬一口**：在單執行緒 event-loop（見領域 D）裡，「阻塞 put」會卡住整個 loop，所有其他連線一起停擺——這時必須用非阻塞的拉取或拒絕，不能用阻塞。背壓策略要配合並發模型選。
- **背壓只防住了自己、沒通知上游**：在邊界拒絕（429）若上游不認得這個訊號、立刻無退避重試，背壓反而觸發 retry storm（見 retry storm 與 metastable failure）。有效的背壓是「拒絕 + 上游有退避」的組合，缺一不可。
- **AI 協作面**：用 AI 生成串流或管線的程式碼時，最常見的隱性缺陷是「示範碼用無界佇列或無 `request(n)` 的 push 模型」——能跑、demo 漂亮，但少了背壓這層在壓力下才會暴露的保護。驗收串流碼時要專門問一句「這條管線的緩衝在哪裡、有沒有界」。

## rate limiting

### 定義與原理

Rate limiting 是**主動限制單位時間內允許進入的請求數**，在請求真正消耗下游資源之前就把超量擋下。它與背壓的差別在「方向與時機」：背壓是被動的、事後的（下游滿了才回傳訊號）；rate limiting 是主動的、事前的（用一個預先設定的速率上限，在門口就決定放行或攔截）。第一原理：**處理能力是有限且可被預估的，那就把進入速率約束在這個能力之內，把「保護」從事後救火變成事前配額。** 限流回答的核心問題是「在這段時間窗內，我允許多少工作進來，多出來的怎麼辦」。

兩種經典演算法是這條問句的兩種答案，差別全在「容不容許突發（burst）」：

- **token bucket（令牌桶）**：以固定速率 `r` 往桶裡放令牌，桶有容量上限 `b`。每個請求消耗一個令牌；桶空了就限流。因為桶能**累積**令牌，閒置一陣子後能一次放行一批（最多 `b` 個），所以**容許突發**，長期平均速率受 `r` 約束。
- **leaky bucket（漏桶）**：請求進入一個固定容量的佇列，佇列以**固定速率**漏出（被處理）。輸出速率恆定、平滑，**不容許突發**；桶滿了就丟。它本質上就是「有界佇列 + 固定速率消費」。

### 解法空間

- **token bucket**：允許短突發、長期均速受控。最常見的 API 限流選擇（人類互動天然是突發式的）。
- **leaky bucket**：輸出絕對平滑、保護一個只能吃恆定速率的下游。代價是請求要排隊，增加延遲。
- **fixed window counter（固定窗計數）**：每個固定時間窗（如每分鐘）一個計數器，超過上限就拒。實作最簡單，但有「窗邊界突刺」——兩個相鄰窗的邊界各打滿，瞬間吃進兩倍配額。
- **sliding window log / sliding window counter（滑動窗）**：記錄每筆請求的時間戳（log）或用加權平滑（counter）來消除固定窗的邊界突刺，較精確但較吃記憶體／計算。
- **分散式限流（中央 vs 本地）**：當限流要跨多個節點共享一個全域配額時，計數狀態放哪裡是核心取捨：
  - **中央計數（centralized）**：所有節點對一個共享儲存（如 Redis，用 `INCR` + TTL 或 Lua 腳本做原子的 token bucket）讀寫計數。全域精確，但每個請求多一次網路往返、且那個共享儲存成為單點與熱點。
  - **本地計數（local / 配額分攤）**：把全域配額除以節點數，每個節點本地獨立限流（如全域 1,000 RPS、10 個節點，每節點本地限 100 RPS）。零協調、零額外延遲，但流量不均時會誤殺（某節點被導入較多流量就先到頂，而其他節點還有餘額）。
  - **折衷（本地 + 週期同步）**：本地先擋，背景週期性向中央回報/領取配額（如每秒同步一次剩餘額度），用「短暫的不精確」換掉每請求的協調成本。

### 各方案的保證與取捨

| 方案/做法 | 保證 | 適用場景 | 注意事項 |
|---|---|---|---|
| token bucket | 長期均速 ≤ r、容許突發 ≤ b | API 限流、人類互動、可吸收尖峰 | 突發大小 b 設太大等於沒限流瞬時；要同時想清楚 r 與 b |
| leaky bucket | 輸出恆定速率、絕對平滑 | 保護只能吃定速的下游、webhook 投遞 | 請求排隊增加延遲；桶滿丟請求；不容突發對 bursty 流量不友善 |
| fixed window | 實作最簡、O(1) 記憶體 | 粗粒度配額、容忍邊界突刺 | 窗邊界可吃進 2× 配額；對抗濫用不夠精確 |
| sliding window | 消除邊界突刺、較精確 | 需要平滑且精確的配額 | log 版吃記憶體（存每筆時間戳）、counter 版是近似 |
| 中央計數（Redis） | 全域精確、跨節點一致 | 嚴格全域配額（如付費方案額度） | 每請求一次往返；共享儲存是單點/熱點；需原子操作避免 race |
| 本地配額分攤 | 零協調、零額外延遲 | 節點間流量均勻、可容忍誤差 | 流量不均→提早誤殺或整體超額；節點數變動要重算配額 |
| 本地 + 週期同步 | 低延遲 + 近似全域精確 | 大規模、嚴格度可放寬一點 | 同步間隔內可短暫超額；實作較複雜 |

worked example（token bucket 手算）：設 `r = 100 tokens/s`、`b = 200`。系統閒置 3 秒後桶累滿到上限 200（不是 300，桶有容量上限）。此時來一波 250 筆突發：前 200 筆立刻放行（用光桶內令牌），剩 50 筆中，每 10 ms 補一個令牌（100/s），所以這 50 筆會在約 `50 × 10 ms = 500 ms` 內陸續放行或（若同時到達且不等待）被限流 50 筆。長期看，無論突發怎麼來，平均通過率被牢牢釘在 100/s。對比 leaky bucket（同樣定速 100/s、桶容量 200）：這 250 筆突發中 200 筆進佇列、50 筆立刻被丟，佇列裡的 200 筆以恆定 100/s 漏出，最後一筆要等約 2 秒——換來的是下游永遠只看到平穩的 100/s，沒有任何瞬間尖峰。

worked example（本地分攤的誤殺）：全域上限 1,000 RPS、10 節點、本地各限 100 RPS。若 L4 負載均衡用 hash 導流造成傾斜，某節點實際吃到 180 RPS——它會在 100 RPS 處開始回 429，誤殺 80 RPS，而此刻其他節點可能總共只用了 600 RPS、全域遠未到 1,000。這就是本地分攤用「省掉協調」換來的代價：**全域配額沒用滿，局部卻已在拒絕**。

### 何時需要

- 對外暴露的 API、可被濫用或被單一租戶打爆的端點 → 需要限流做配額與防濫用（常配合身分做 per-tenant 限流）。
- 下游是脆弱或昂貴資源（第三方 API 有自己的配額、DB 連線有限、付費外部服務按量計費）→ 用限流把自己的流量整形到下游能吃的速率，避免把下游打掛或超支。
- 選 token vs leaky：要保護的下游能吸收突發、且使用者體驗在乎「平時不卡」→ token bucket；下游只能吃恆定速率、或要絕對平滑的輸出（如對外投遞 webhook 不想被對方限流）→ leaky bucket。
- 選中央 vs 本地：配額是「硬合約」（付費方案額度，超了要負責）→ 中央精確計數；配額是「保護性軟限」（防雪崩，差個幾趴無所謂）→ 本地分攤夠用、且更便宜更快。
- over-engineering 警訊：內部、可信、流量可預測的服務間呼叫，往往用連線池上界 + 背壓就夠，不必再疊一層精算的滑動窗限流。

### 常見誤解與陷阱

- **「限流 = 背壓」**：兩者互補但不同。限流是事前用固定速率擋、不看下游真實狀態；背壓是事後依下游真實飽和度回壓。最穩的系統兩者都有：限流防可預期的配額濫用，背壓防無法預期的下游變慢。
- **fixed window 的邊界突刺被當成精確**：上限「每分鐘 100」，攻擊者在 0:59 打 100 筆、1:00 再打 100 筆，2 秒內通過 200 筆——固定窗在邊界允許 2× 瞬時。要精確就用滑動窗。
- **中央限流的計數操作不是原子的**：用「先讀計數、判斷、再寫回」三步在分散式下有 race（兩個節點同時讀到 99，都判定可放行，計數變 101 卻都過了）。必須用單一原子操作（Redis Lua 腳本或 `INCR` 配合 token bucket 的原子實作），把讀-判-寫收成一步。
- **回 429 卻沒給 `Retry-After`**：限流回應若不告訴客戶端「多久後再試」，客戶端只能瞎猜或立刻重試，把限流變成 retry storm 的觸發器。限流的回應要包含退避指引。
- **閘道層 vs 應用層**：限流可放在 API gateway（見領域 H）也可在應用內。閘道層擋得早、省下應用資源，但看不到應用層的細粒度語意（如「這個操作很貴」）；兩層常並存、各擋一種濫用。

## load shedding

### 定義與原理

Load shedding（卸載）是：當系統已經過載、無法服務所有進來的請求時，**主動、有選擇地丟棄一部分工作**，以保住對「剩下那部分」的服務品質。它和 rate limiting 看似都在「拒絕請求」，但觸發條件根本不同——rate limiting 依**預設的固定速率**擋（不管系統當下忙不忙），load shedding 依**系統當下的真實飽和度**擋（CPU、佇列深度、延遲超標才丟）。第一原理是一個殘酷但正確的取捨：**在過載時，「服務 100% 的請求但全部很慢甚至超時」遠不如「服務 80% 的請求且都快又正常、明確拒絕另外 20%」。** 因為前者往往連那個「全部」都保不住——延遲爆炸會引發逾時、重試、進而完全崩潰（見排隊直覺與 metastable failure）。

最關鍵的洞察來自一個反直覺現象：**過載時系統的有效吞吐（goodput，真正完成且有用的工作）會隨負載上升而下降**——因為越來越多 CPU 花在處理那些「最後會逾時、客戶端早已放棄」的請求上。Load shedding 的目的就是把這條下墜的 goodput 曲線拉平：寧可早早拒絕注定失敗的請求，把資源留給能成功的。

### 解法空間

- **基於佇列深度 / 等待時間丟（queue-based）**：當請求在進入佇列時，若佇列已超過閾值、或這筆請求預估等待時間已超過它的 deadline，直接丟掉——反正排到它時客戶端早就逾時了，做了也白做。
- **基於資源飽和度丟（resource-based）**：監控 CPU、記憶體、event-loop 延遲、in-flight 請求數，超過水位就開始拒絕新請求（回 503）。
- **依優先級 / 重要性丟（priority-based）**：把請求分級（如付費 vs 免費、寫 vs 讀、互動 vs 批次），過載時先丟低優先級的，保住高價值流量。
- **依成本丟（cost-based）**：丟掉昂貴的請求（大查詢、無快取的複雜操作）保住便宜的，用最少的犧牲換最多的存活請求數。
- **客戶端側卸載（client-side / deadline propagation）**：客戶端帶上 deadline，每一跳檢查「剩餘時間還夠不夠做完」，不夠就提早放棄、不送出注定逾時的請求（與領域 P 的 deadline/budget 呼應）。

### 各方案的保證與取捨

| 方案/做法 | 保證 | 適用場景 | 注意事項 |
|---|---|---|---|
| 佇列深度 / 等待時間丟 | 已逾時的請求不浪費資源 | 有明確佇列與 deadline 的管線 | 需準確估等待時間；閾值設太保守會誤丟可服務的請求 |
| 資源飽和度丟 | 系統不被推過崩潰點、延遲有界 | 通用過載保護 | 水位選錯（太高來不及、太低浪費容量）；要選對指標（event-loop 延遲常比 CPU 更早反映飽和） |
| 優先級丟 | 高價值流量在過載時存活 | 有清楚分級的多租戶/混合負載 | 需要請求攜帶優先級資訊；低優先級被持續餓死要有機制兜底 |
| 成本丟 | 用最小犧牲保最多請求 | 請求成本差異大 | 需能在執行前估成本；可能系統性地排擠某類用戶 |
| 客戶端 deadline 傳播 | 不送出注定失敗的請求、省全鏈資源 | 微服務多跳呼叫鏈 | 需全鏈支援 deadline 傳遞；時鐘/時區不一致會誤判（見領域 M） |

worked example（goodput 崩塌）：一個服務每秒能完成 1,000 個請求（每個耗 1 ms CPU，單核）。客戶端 deadline 是 100 ms。當進入流量升到 1,500 RPS、無 load shedding 時：佇列持續積壓，請求平均等待時間飆過 100 ms，於是**每個完成的請求其實客戶端早已逾時離開**——服務花滿 100% CPU 做的全是白工，有效 goodput 趨近 0。加上「進佇列時若預估等待 > 100 ms 就丟」的 load shedding 後：系統只收約 1,000 RPS 進來處理（其餘約 500 RPS 立刻回 503），這 1,000 個都在 deadline 內完成，goodput 穩在 1,000 RPS。同樣的 CPU，goodput 從約 0 回到 1,000——這就是「丟掉一部分」反而救活整體的算術。

### 何時需要

- 任何「流量尖峰會超過容量、且超過時延遲會爆炸到引發逾時」的線上服務——尤其當客戶端會在逾時後重試（把過載放大）時，load shedding 幾乎是必備。
- 多租戶或混合優先級系統，過載時必須保住關鍵流量（付費、健康檢查、控制面）→ 需要優先級式卸載。
- over-engineering 警訊：流量遠低於容量、或這是個離線批次系統（慢一點沒人在乎、沒有逾時客戶端）→ load shedding 是多餘的，背壓讓它自然降速就好。Load shedding 的價值專屬於「有逾時客戶端 + 會超容量」的線上場景。

### 常見誤解與陷阱

- **「過載時更努力處理」是反的**：直覺想「越忙越要把每個請求都做完」，但過載時把資源花在注定逾時的請求上，會把 goodput 推向 0、進而觸發 metastable failure。正確方向是**過載時更願意丟**。
- **健康檢查也被一起丟掉**：粗暴的「飽和就回 503」若連 liveness/readiness 探針也擋掉（見領域 P），會讓編排器以為服務死了而重啟它——本來只是過載，變成被殺。卸載要對控制面流量（健康檢查、優先級最高）網開一面。
- **丟棄訊號沒回饋給客戶端**：load shedding 回 503 卻不帶退避指引，客戶端立刻重試，等於把卸載掉的負載又加倍打回來——卸載必須配合客戶端退避（見 retry storm）才有效，否則只是空轉。
- **用錯飽和指標**：在 event-loop 模型裡（見領域 D），CPU 使用率可能還沒滿但 event-loop 已經延遲嚴重；用 CPU 當卸載觸發會太晚。要選「最早反映飽和」的指標。

## circuit breaker

### 定義與原理

Circuit breaker（斷路器）是一個包在「對外部依賴的呼叫」外面的狀態機：當下游故障率超過閾值時，它**主動切斷後續呼叫、立刻快速失敗**，而不是讓每個請求都去撞一個已知掛掉的下游、各自慢慢逾時。名字借自電路斷路器——偵測到異常就跳閘，保護整條電路。這個模式由 Michael Nygard 在《Release It!》（2007）提出。第一原理：**對一個已經故障的下游，繼續送請求過去不但救不了它，還會反過來拖垮自己**——每個呼叫都佔住一個連線/執行緒等到逾時，逾時佔用會耗盡呼叫方的資源池，把下游的局部故障傳染成呼叫方的全域故障（級聯失敗）。Circuit breaker 用「快速失敗」換掉「慢慢逾時」，把資源從注定失敗的呼叫上釋放出來。

它的核心是**三態狀態機**：

- **Closed（閉合，正常）**：呼叫正常通過，斷路器在背景統計失敗率。失敗率/連續失敗數超過閾值 → 跳到 Open。
- **Open（斷開，跳閘）**：所有呼叫**立刻失敗**（不真的去打下游），通常直接走 fallback。停留一段冷卻時間（reset timeout）後 → 跳到 Half-Open。
- **Half-Open（半開，試探）**：放**少量**試探請求通過，去探下游有沒有恢復。若這些試探成功 → 回到 Closed（恢復正常）；若失敗 → 退回 Open（再等一個冷卻週期）。

```
            失敗率超過閾值
   ┌──────────────────────────────┐
   │                              ▼
┌──────┐   冷卻時間到   ┌───────────┐   試探失敗   ┌──────┐
│Closed│ ◀──────────── │ Half-Open │ ───────────▶ │ Open │
│ 正常 │               │   試探    │              │ 跳閘 │
└──────┘ ◀──────────── └───────────┘              └──────┘
   ▲        試探成功                                   │
   │                                                  │
   └──────────────────────────────────────────────────┘
                   （Open 期間所有呼叫立即失敗）
```

### 解法空間

斷路器本身是一個機制，但「怎麼判斷該跳閘、跳閘後怎麼辦」有多種設計選項：

- **觸發條件**：連續失敗計數（連 N 次失敗就跳）vs 失敗率窗（滑動窗內失敗率 > X% 且樣本數足夠就跳）。後者對流量波動更穩健，但需要足夠樣本才有意義。
- **半開試探策略**：放單一試探 vs 放固定比例試探。試探太多會在下游還沒好時又一次打它；太少則恢復偵測慢。
- **跳閘後的行為**：直接回錯 / 回 fallback（快取舊值、預設值、降級回應）/ 排進佇列稍後重試（見領域 P 的 fallback·降級——斷路器是觸發降級的機制之一）。
- **粒度**：per-依賴（整個下游服務一個斷路器）vs per-端點/per-實例（更細，能隔離單一壞節點，但狀態更多）。
- **與重試/限流配合**：斷路器擋的是「對已知故障下游的呼叫」，配合 retry token bucket（成功補令牌、失敗扣令牌）能在「重試」與「不放大故障」間取得平衡（見 retry storm）。

### 各方案的保證與取捨

| 方案/做法 | 保證 | 適用場景 | 注意事項 |
|---|---|---|---|
| 連續失敗計數觸發 | 實作簡單、反應直接 | 流量穩定、失敗訊號清楚 | 偶發抖動（單筆超時）可能誤跳；對低流量不穩 |
| 失敗率窗觸發 | 對波動穩健、需足夠樣本 | 高流量、失敗率有意義 | 低流量時樣本不足、判斷失準；窗大小要調 |
| 半開放單一試探 | 恢復時對下游壓力最小 | 下游恢復後仍脆弱 | 恢復偵測較慢（一次只試一筆） |
| 半開放比例試探 | 恢復偵測快 | 下游恢復後能立刻承載 | 試探比例過高可能把剛恢復的下游又打掛 |
| per-依賴粒度 | 狀態少、好管理 | 下游是單一同質服務 | 無法隔離下游叢集中單一壞節點 |
| per-實例粒度 | 隔離單一壞節點 | 下游是多實例、故障常是單點 | 狀態爆炸；需與服務發現/LB 配合 |

worked example：服務 A 呼叫服務 B，B 突然全掛、每個呼叫要等 30 秒逾時才報錯。A 的請求執行緒池有 200 條。**無斷路器**時：每秒進來 100 個需呼叫 B 的請求，每個各佔一條執行緒等 30 秒——`100 × 30 = 3,000` 條執行緒需求遠超 200 條，約 2 秒內執行緒池就被耗盡，連那些**不呼叫 B** 的請求也拿不到執行緒而一起失敗，B 的局部故障傳染成 A 的全域癱瘓。**加斷路器**（連 20 次失敗跳閘、冷卻 10 秒）：跳閘後對 B 的呼叫立刻失敗（耗時約 0 ms）走 fallback，執行緒幾乎不被佔用，A 對其他功能照常服務；每 10 秒放一個試探探 B 是否恢復。級聯失敗被斷在 A 與 B 之間，沒往上傳。

### 何時需要

- 任何跨網路呼叫**外部依賴**、且該依賴故障時你不想被它拖垮的場景——尤其是同步呼叫鏈（A 等 B、B 等 C），一處故障最容易級聯。
- 下游故障時你有**可接受的 fallback**（快取舊值、降級回應、稍後重試）——斷路器跳閘後總得有個 Plan B，否則只是把「慢慢失敗」換成「快速失敗」、對使用者體驗未必更好。
- over-engineering 警訊：對同行程內的本地呼叫、或極可靠且故障時根本沒有 fallback 的關鍵依賴（如自己的主資料庫——它掛了你也沒救），加斷路器收益有限。斷路器的價值在「下游會獨立故障 + 你有降級空間」。

### 常見誤解與陷阱

- **冷卻時間設太短，斷路器在 Open/Half-Open 間抖動**：下游還沒真的恢復就放試探、失敗、又跳回 Open，反覆抖動既偵測不準也持續騷擾下游。冷卻時間要給下游真正喘息與恢復的時間。
- **半開放太多試探把剛恢復的下游又打死**：下游剛從過載恢復、承載力還很脆弱，此時一次放大量試探等於再來一波尖峰。試探要少、要漸進。
- **把斷路器當限流用**：斷路器依「下游健康度」開關（故障才跳），限流依「速率配額」開關（不管下游健不健康）。用斷路器做配額控制，會在下游明明健康時不該跳、在你只是想限速時跳不了。兩者解不同問題。
- **沒有 fallback 的斷路器只是把錯誤提早**：若跳閘後直接把錯誤往上拋而沒有降級路徑，使用者得到的還是失敗，只是快一點。斷路器要配 fallback 才完整（見領域 P）。
- **單機斷路器在叢集裡各自為政**：每個實例獨立統計失敗率，可能某些實例跳了某些沒跳，行為不一致。需要全域一致時要共享狀態，但那又引入協調成本——多數情況接受「各自判斷」即可，但要知道這個不一致存在。

## retry storm 與 metastable failure

### 定義與原理

Retry storm（重試風暴）是：當下游變慢或短暫故障，上游的重試機制把**一個失敗的請求放大成多個請求**，這些額外流量讓本就吃緊的下游更撐不住、失敗更多、於是觸發更多重試——一個正回饋迴圈把小故障滾成大災難。Metastable failure（亞穩定故障）是這現象的一般化與更深刻的版本：系統因為某個觸發事件（流量尖峰、一次部署、一次 GC 暫停）進入一個**過載狀態，而這個狀態會自我維持**——即使最初的觸發消失了，系統也卡在低 goodput 的壞狀態出不來，因為**有一個正回饋迴圈在持續餵養過載**。這個概念由 Bronson、Aghayev、Charapko、Zhu 在 HotOS 2021 的論文〈Metastable Failures in Distributed Systems〉中正式提出。

第一原理（這是本檔最重要的機制）：**重試的數學是乘法的，不是加法的。** 正常時重試是安全網（失敗的少數請求重送一下就成功）；但當下游整體變慢、大批請求開始逾時，每個逾時請求觸發 `k` 次重試，瞬間把下游流量乘上 `(1+k)` 倍——而下游正是因為流量太大才變慢的。於是「變慢 → 逾時 → 重試 → 流量更大 → 更慢」形成閉環。Metastable 的關鍵特徵是**遲滯（hysteresis）**：把流量降回觸發前的水準，系統**不會**自己恢復，因為正回饋迴圈（積壓的重試、被污染的快取、排滿的佇列）仍在自我餵養。要打破它通常得「比觸發點更猛地」降載——例如完全停止流量讓它排空，才能跳回好狀態。

### 解法空間

放大機制一旦理解，防禦就是「掐斷正回饋迴圈」的各種手段：

- **限制重試次數 + 退避 + jitter**：每個請求最多重試 `k` 次（k 小，如 1–2），重試間指數退避加隨機抖動，避免同步重試造成的尖峰（退避/jitter 的具體做法見領域 A，本檔強調的是「為什麼必須限制」）。
- **retry budget / retry token bucket**：給整個客戶端一個「重試預算」——成功的呼叫往桶裡滴少量令牌（如 0.1），每次重試扣一個令牌，桶空了就不再重試。這把重試率天然鎖在「成功流量的一個小比例」內，下游全掛時重試自動趨近 0（因為沒有成功來補令牌），從根本掐斷放大。
- **circuit breaker**：下游故障時直接跳閘、停止呼叫（見 circuit breaker），等於把「對已知故障下游的重試」整批掐掉。
- **load shedding**：過載時主動丟請求（見 load shedding），把系統推回好狀態的負載區間。
- **deduplication of work / 去重**：對「重複觸發同一工作」的迴圈（如多個請求重建同一快取），用 single-flight 合併，避免放大（見領域 G 的擊穿防護、領域 A 的去重）。
- **打破迴圈的「重啟級」手段**：metastable 一旦進入，常需要主動清空佇列、清快取、暫停入口流量讓系統排空，再緩慢放流量回來——因為遲滯讓它不會自己好。

### 各方案的保證與取捨

| 方案/做法 | 效果 | 適用場景 | 注意事項 |
|---|---|---|---|
| 限重試次數 + 退避 + jitter | 把放大倍數壓到 (1+k)、削平同步尖峰 | 所有有重試的客戶端（基本盤） | k 仍是乘數；多層呼叫鏈各自重試會層層相乘（見下） |
| retry token bucket | 下游全掛時重試自動趨 0 | 客戶端側、想要自適應重試 | 桶參數要調；正常抖動別把桶耗光 |
| circuit breaker | 整批掐掉對故障下游的呼叫 | 下游會整體故障 | 需 fallback；半開試探別又打爆（見 circuit breaker） |
| load shedding | 把負載推回好狀態區間 | 已過載、要主動降載 | 需配客戶端退避，否則丟掉的又被重試打回 |
| single-flight 去重 | 消除「重建同一工作」的放大 | 快取擊穿類重複工作 | 只解重複類放大，不解一般重試風暴 |
| 入口暫停 + 緩放（打破遲滯） | 把卡死的 metastable 拉回好狀態 | 已陷入亞穩定故障 | 是救火手段不是預防；放流量太快會立刻再陷入 |

worked example（多層相乘）：一條呼叫鏈 A → B → C，每層各自設「失敗重試 3 次」（即每個請求最多送 4 次）。當最底層 C 短暫過載開始逾時：C 對每個請求被打 4 次；但 B 也會對它送給 C 的每個請求重試，所以 B 因為 C 失敗而把自己對 C 的請求重送，乘上 4；A 又對 B 重試乘上 4——最壞情況下，使用者**一個**請求在最底層放大成 `4 × 4 × 4 = 64` 次對 C 的呼叫。C 本來只是稍微過載，瞬間被打 64 倍流量，徹底崩潰並把整條鏈拖進 metastable 狀態。正確做法：**只在最接近使用者或最有上下文的一層重試，其餘層不重試（fail-fast 往上傳）**，把總放大倍數從 64 壓回 4；再加 retry budget 讓 C 全掛時這 4 倍也自動收斂到 1。

### 何時需要

- 任何有重試 + 共享下游 + 流量可能尖峰的分散式系統——這幾乎是所有微服務架構，所以「防重試放大」應視為預設要考慮的，不是可選項。
- 系統曾經歷「故障已修但服務遲遲不恢復、要重啟/清快取才好」的事故 → 那很可能就是 metastable failure，需要從正回饋迴圈下手而非只修觸發點。
- over-engineering 警訊：單體、無重試、或下游不共享（每個請求打獨立資源）的系統，放大迴圈不成立，不必套用全套防禦。判準是「有沒有一個共享資源 + 一個會在它變慢時增加它負載的正回饋」。

### 常見誤解與陷阱

- **「加重試讓系統更可靠」是半真**：正常時對，過載時反——同一個重試把小故障放大成大故障。重試是把「常態的少量失敗」變好、把「過載時的大量失敗」變更糟。必須限次數 + 退避 + budget 才安全。
- **修了觸發點卻沒恢復**：metastable 的反直覺處在於——把流量降回故障前的水準，系統**不會**自己好（遲滯）。團隊常困惑「流量明明已經正常了為什麼還掛」，答案是正回饋迴圈還在自我餵養，得更猛地降載才能跳出。
- **多層各自重試**：每層獨立設重試看似各自合理，乘起來是指數放大（上面的 64 倍）。重試應集中在單一層，跨層用 deadline/budget 傳遞而非各自重試。
- **退避了但沒 jitter**：所有客戶端同時逾時、同時退避同樣的時間、於是同時重試——退避只是把尖峰整體往後挪，沒削平。必須加隨機 jitter 把重試攤開（見領域 A）。
- **把 metastable 當「容量不足」**：直覺想「加機器」，但若正回饋迴圈還在，加的容量也會被放大的重試吃掉、然後一起陷進去。先掐迴圈，再談擴容。

## 排隊直覺（utilization 與 ρ→1 爆炸）

### 定義與原理

這條回答一個每個過載決策底下都站著的問題：**為什麼系統在利用率（utilization）還沒到 100% 時，延遲就已經爆炸了？** 直覺上「CPU 用到 80% 應該還好、還有 20% 餘裕」，但實務上 80% 利用率的佇列延遲，可能已經是 50% 時的好幾倍。理解這個非線性，是所有限流、卸載、容量規劃的數學底座，所以本書在此把它講到「直覺夠用」、不外包。

用一個單一伺服器的排隊模型（排隊論裡稱 M/M/1：請求到達是 Poisson 隨機、服務時間是指數分布、單一伺服器）建立直覺。定義**利用率** `ρ = λ/μ`，其中 `λ` 是平均到達率、`μ` 是平均服務率（伺服器滿載時每秒能處理幾個）。`ρ` 就是「伺服器有多忙」的比例，必須 `ρ < 1` 系統才穩定（否則佇列無限長大）。這個模型給出一條招牌公式——**系統內平均請求數**：

```
E[N] = ρ / (1 − ρ)
```

關鍵全在分母那個 `(1 − ρ)`：當 `ρ` 逼近 1，分母逼近 0，`E[N]`（以及與它成正比的延遲）**爆炸式趨向無窮**。這不是線性的「越忙越慢」，而是雙曲線的「接近滿載時災難性地慢」。第一原理上的解釋是**變異性（variability）**：請求不是均勻到達的，到達有隨機叢集、服務時間有長有短。當伺服器幾乎全忙（ρ→1），任何一個瞬間叢集到達或一個慢請求，都沒有空閒容量去吸收，後面的人只能排隊等——而越接近滿載，「沒有空閒去吸收抖動」的情況越普遍，等待時間就越非線性地暴漲。**正是這個變異性讓你無法把利用率推到 100% 還指望低延遲。**

### 解法空間

「排隊直覺」不是一個要解的故障，而是一組要內化的判準——它告訴你過載防禦該把目標利用率設在哪、為什麼。對應的工程做法：

- **把目標利用率設在 70–80%、留意 (1−ρ) 的非線性**：不要把容量規劃做到「平均利用率 95%」就以為還有 5% 餘裕——那 5% 對應的延遲已經爆了。留 hysteresis 餘量。
- **降低變異性（讓 ρ 能更高仍穩）**：變異性是延遲爆炸的根因，所以平滑到達（限流/leaky bucket，見 rate limiting）、平滑服務時間（拆大請求、超時砍長尾、批次化）都能讓同一個 ρ 下延遲更低，或讓你安全地把 ρ 推更高。
- **增加伺服器數（M/M/c 的規模經濟）**：多伺服器共享一個佇列時，同樣的總利用率下延遲遠低於多個各自獨立的單伺服器——因為一個忙不過來時別的能接手，吸收抖動的能力更強。這就是「集中一個大池子比切成多個小池子」在排隊上更省的原因。
- **控制佇列長度（有界 + 卸載）**：既然 ρ→1 時佇列會炸，就用有界佇列 + load shedding 主動截斷，把「無限排隊」換成「明確拒絕」（見背壓、load shedding）。

### 各方案的保證與取捨

| 方案/做法 | 效果 | 適用場景 | 注意事項 |
|---|---|---|---|
| 目標利用率 70–80% | 留出吸收抖動的餘量、延遲可控 | 對延遲敏感的線上服務 | 浪費約 20–30% 名目容量；對成本敏感者心理上難接受 |
| 降低到達變異性（限流/平滑） | 同 ρ 下延遲更低、可安全推高 ρ | 流量突發、可整形 | 限流本身增加被擋請求的延遲；整形有成本 |
| 降低服務時間變異性（砍長尾/批次） | 縮短佇列、削長尾 | 服務時間分布長尾明顯 | 砍長尾請求＝丟一部分工作；批次增單筆延遲 |
| 增加伺服器（共享佇列 M/M/c） | 同總利用率下延遲大降（規模經濟） | 可水平擴展、共享佇列 | 共享佇列需負載均衡正確分派；不能分成各自獨立的小池 |
| 有界佇列 + load shedding | 把無限排隊換成明確拒絕 | ρ 可能逼近/超過 1 | 會丟請求；閾值要對（見 load shedding） |

worked example（ρ→1 爆炸的算術）：用 `E[N] = ρ/(1−ρ)` 直接代數字：

| 利用率 ρ | E[N]（系統內平均請求數） | 相對 ρ=0.5 的倍數 |
|---|---|---|
| 0.5 | 0.5 / 0.5 = 1 | 1× |
| 0.8 | 0.8 / 0.2 = 4 | 4× |
| 0.9 | 0.9 / 0.1 = 9 | 9× |
| 0.95 | 0.95 / 0.05 = 19 | 19× |
| 0.99 | 0.99 / 0.01 = 99 | 99× |

從 ρ=0.5 到 ρ=0.9，利用率只多了 80%（0.5→0.9），但系統內平均請求數（及與之成正比的平均延遲）漲了 **9 倍**；再從 0.9 推到 0.99，利用率只多了 10%，延遲卻再漲 11 倍（9→99）。這就是為什麼「把利用率從 90% 榨到 99% 省機器」是個延遲陷阱——名目上只多用 10% 容量，延遲代價是一個數量級。反過來看也解釋了 load shedding 的算術：在 ρ 逼近 1 時丟掉一小撮請求把 ρ 從 0.99 拉回 0.9，延遲就從 99 個排隊數掉回 9，**丟一點點換回一個數量級的延遲改善**。

（M/M/1 是理想化模型——真實系統的到達與服務分布更複雜，絕對數字會不同；但「分母 (1−ρ) 主導、接近滿載時超線性爆炸」這個定性結論對幾乎所有排隊系統都成立，這正是要記住的直覺。）

### 何時需要

- 做容量規劃、設定自動擴展（HPA）的目標利用率、或回答「這台機器還能再加多少流量」時——排隊直覺直接決定你把閾值設在哪（不是 95%，是 70–80%）。
- 解釋「為什麼系統明明 CPU 沒滿延遲卻很糟」這類事故時——答案往往是利用率已進入 (1−ρ) 的陡峭區。
- 決定「集中大池 vs 切多小池」「要不要做請求平滑」這類架構取捨時——M/M/c 的規模經濟與變異性論點是判準。
- over-engineering 警訊：流量遠低於容量（ρ 長期在 0.3 以下）的系統，排隊效應微不足道，不必為這個非線性做特別設計——但仍要知道它存在，以免某天流量長到 0.9 時措手不及。

### 常見誤解與陷阱

- **「利用率 80% 還有 20% 餘裕」的線性誤解**：餘裕不是線性的。80% 利用率對應的排隊延遲已是 50% 的 4 倍、是 99% 災難的前奏。利用率與延遲的關係是雙曲線，不能用「還剩幾趴」做線性外推。
- **追求 100% 利用率**：把利用率推到 100% 在排隊上等於 ρ→1，延遲趨向無窮、佇列無限長大。「機器跑滿才划算」對批次離線任務或許成立（沒有延遲要求），但對線上服務是災難。
- **忽略變異性**：兩個平均利用率都 80% 的系統，到達/服務變異性大的那個延遲糟得多。只看平均利用率、不看變異性（尤其長尾服務時間，見領域 S 的 tail latency）會嚴重低估排隊延遲。
- **把多個小池當一個大池估**：把流量切成 N 個各自獨立的小佇列，總延遲遠差於一個共享的大佇列（規模經濟反向）。分區/分片（見領域 B）在隔離性上有好處，但在排隊延遲上要付這個代價。
- **誤把這當「需要更多數學」**：實務上不需要解 M/M/1 的完整分布——記住 `ρ/(1−ρ)` 這條曲線的形狀、知道分母會在接近滿載時主導，就足夠做對所有過載與容量決策。直覺夠用，不必背完整排隊論。
