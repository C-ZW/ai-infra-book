# 寫作風格指南（內部文件，非書籍內容）

本文件是全書的寫作契約。每一條目都必須遵守，否則整本會失去一致性。
書名：《保證與取捨——後端與分散式系統的概念地圖》。

> **重大改版（2026-06-21）**：本書從「挖作者履歷系統的個人書」**整個重新定位**成「**給一般資深工程師的通用概念參考書**」。舊的 `chNN-*.md`（個人書 20 章）一律作廢、將被新的領域檔取代。寫作前務必理解這個轉向，別把舊書的個人化口吻帶進來。

## 這本書是什麼、不是什麼

- **是**：一本**通用的後端與分散式系統概念地圖**。對每一個概念/工具，回答四件事再加一個防雷：**它保證什麼／解什麼問題 → 有哪些解法（解法空間）→ 不同工具給什麼保證與坑（對照）→ 什麼時候需要這種保證 → 常見誤解與陷阱**。讀者讀完一條，腦中要有「這個問題的**整張選項地圖**與取捨」，不是單一答案。
- **不是**：
  - **不是個人書**。**完全不提任何特定作者的系統、公司、履歷、經歷**。沒有「你當年做的那個版本」「你的結算 pipeline」這種錨點。對象是**所有人**。
  - **不是糾錯書**。**禁用「我以為 vs 實際」「老兵記憶會過時」這類『預設讀者記錯』的裝置**（這是舊版被推翻的主因）。讀者是懂行的同儕，平視對話，不打臉。
  - **不是教學書**。讀者是**資深工程師**：不從零解釋什麼是資料庫、HTTP、交易。每個概念**從定義講起，但直接切深水區**，不鋪基礎。
  - **不是自足以外的書**。**不指向任何其他書**（不寫「數學見《等待的數學》」「故障見《正常意外》」）。需要的深度就在本書內把它講到夠、或點到為止用外部權威來源（RFC/論文/官方文件），但**不外包給作者的其他著作**。
  - **不含練習**。沒有「紙上推演／自我檢核」段。它是參考書，不是習題本。
- **全書脊椎（每條都在回答）**：保證（guarantee）與取捨（trade-off）。書名就是脊椎——每個概念底下都是「你想要什麼保證、你願意付什麼代價」。

## 讀者是誰

唯一假設：**一般資深後端／軟體工程師**（不特指任何人）。

- **預設已具備**：會寫程式、熟悉基本 web/HTTP/資料庫/並發/雲端的工作級使用。不要解釋這些是什麼。
- **本書提供的**：把每個概念**從定義講到機制、講到取捨、講到陷阱**的深度——資深讀者「會用但沒系統化想過底層與選項空間」的那一層。
- **語氣**：工程師對工程師、平視、有觀點。不諂媚、不打臉、不喊口號。

## 兩副閱讀透鏡（Part 0 的 T/U 前置存在的理由）

T（工程實踐與團隊推動）與 U（工程判斷與決策）放在最前面，是刻意的：讀者讀**後面每一條技術概念**時，要同時戴著兩副眼鏡——
- **U 的眼鏡**：我會怎麼**判斷/選**它？（建vs買、簡單性、退場成本…）
- **T 的眼鏡**：我會怎麼讓**團隊採用**它？

技術條目不必每條都明寫這兩副眼鏡，但全書的編排假設讀者已先讀過 T/U。

## 每條目的模板（六個 `###`＝五段內容＋延伸閱讀；必守，標題照抄，lint 會 grep）

每個領域檔 = 一個 `# <領域字母> — <領域名>`（h1）＋簡短前言，其下每個概念是一個 `## <概念名>`（h2）。每個 `##` 概念底下，依類型用下面**其中一組** `###` 小標：

**問題條目（`[問]`）**——標題照抄：
1. `### 定義與原理` — 它是什麼、解什麼問題、第一原理。
2. `### 解法空間` — 處理它的**所有**辦法（不只一種），條列＋一句機制。
3. `### 各方案的保證與取捨` — 把解法空間的各方案（工具／做法／變體）並排對照：各給什麼保證、各有什麼坑、各適合什麼（對照表）。**很多概念/判斷條目沒有「工具」——這一段就是「做法對照」，別硬湊工具或品牌。**
4. `### 何時需要` — 決策框架：什麼場景要這種保證、什麼場景是 over-engineering。
5. `### 常見誤解與陷阱` — 最容易誤解/踩雷的點。
6. `### 延伸閱讀` — 外部真實連結（RFC/論文/官方文件）。

**工具條目（`[工]`）**——標題照抄：
1. `### 是什麼與內部機制` — 它本身是什麼、底層怎麼運作。
2. `### 在哪些系統扮演什麼角色` — 用法地圖：它在 cache/lock/queue/… 各被用來幹嘛。
3. `### 保證與限制` — 它給的保證、它的硬限制。
4. `### 跟替代品的取捨` — 跟同類工具並排比（對照表）。
5. `### 常見誤解與陷阱`
6. `### 延伸閱讀`

兩組共用的固定錨點是 `### 常見誤解與陷阱` 與 `### 延伸閱讀`（每條都必有，lint 抓這兩個最穩）。

## AI 協作維度（只在 T/U，原理層）

- T 與 U 的每一條，在 `### 解法空間`／`### 常見誤解與陷阱` 裡**多問一句「AI 在這件事裡能做什麼、又帶來什麼新陷阱」**（例：技術債→AI codemod 能還也能製造「能跑沒人懂」的債）。
- T/U 另含 3 條**專門**講 AI 協作的條目（見 outline）。
- **鐵律：只寫原理層，不寫工具版本**——講委派/驗收/review gate/信任邊界/新故障模式，**不**講「用某 model／某 IDE」。AI 工具再換代也不腐。
- 技術領域（A–S）**不**強加 AI 維度（避免發散）；V 雖是軟性領域，但其「程式碼審查」「事故應變」等條目可在陷阱段點一句 AI 的影響，非必要。

## 深度標準

- **每條至少一個具體案例**（不准用任何特定作者的系統當例子）。**技術條目（A–S）帶真實數字的 worked example**——例：交付語意手算「10 萬則訊息、broker 重投率 0.1% → 約 100 筆重複」。**判斷/實踐/文化條目（T、U、V——V 全是軟性條目）用具體情境/案例代替數字**（如「一個三人小團隊要導入新 lint 規則」的推演），不硬湊數字。
- **允許不可執行的 pseudo-code／設定片段**追邏輯（資深讀者想看碼）；標清楚是示意、不保證可跑。程式碼與註解一律英文。
- **每條都要有「保證 + 適用條件 + 坑」**：禁止給「最佳實踐」口號而不講它在什麼條件下成立、什麼條件下騙你。這是本書與「背名詞」的根本差別。
- **決策框架優先於教條**：什麼時候用 A、什麼時候用 B——給判準，不給「就用 X」。
- **嚴謹度誠實標示**：分清「規格保證的」「常見實作的行為」「我的判斷/觀點」。別把某工具的行為當成所有同類都這樣。
- 對照表是本書招牌：工具 × 保證 × 注意事項。沒有「注意事項」欄的對照表是危險品。
- 寧深勿廣、不灌水。長度不限——該深就深，但每段都要有資訊量。

## 事實正確性

- 時效性事實（版本、產品狀態、授權、預設值、交付語意）標時點 `（2026-06）`，並以可查證的官方來源為準；寫條目時自行 WebSearch/WebFetch 查證（每條 3–8 次）。
- **延伸閱讀只放外部真實連結**（RFC / 論文 / 官方文件 / 權威技術文章）；**不指向作者的其他書**。沒把握的連結標「（未驗證）」或不收。
- 屬性與年份（CAP=Brewer/Gilbert-Lynch、PACELC=Abadi、RFC 編號…）查證後寫，禁憑記憶。

## 格式

- Markdown。檔案 `# 領域`（h1）→ 概念 `##`（h2）→ 五段 `###`（h3）。
- ASCII 對照表/圖（box-drawing）必須過 `check_diagrams.py`（CJK=2 欄；box 中間行含全形標點易讓右框漂一欄，用 Python 精算寬度自測）。對照表能用 Markdown pipe 表就用（不需過 check_diagrams、較好維護），只有需要畫機制/時序/狀態圖才用 ASCII box。
- 跨條目引用用「（見 <概念名>）」或「（見領域 X）」，不要寫死檔名路徑、不要寫死章號（領域會增刪）。
- 語言：繁體中文台灣用語，技術名詞保留英文。**禁簡體詞**（信息→訊息、默認→預設、緩存→快取、線程→執行緒、調用→呼叫、對象→物件、代碼、概率、方差…）。簡繁同形字（們/效/系/准/值/程）用 Python 逐字元比對、別用 shell grep 誤報。

## 平行產出一致性規範（27 個 agent 必守，避免發散）

- **「六段」澄清**：每條 6 個 `###`＝前五段內容（定義/解法空間/各方案保證取捨/何時需要/常見誤解陷阱）＋第六段 `### 延伸閱讀`。一律六個，別漏延伸閱讀。
- **主要術語對照表（全書統一，不得自創譯法）**：
  | English | 統一中文 | 備註 |
  |---|---|---|
  | partition | 分區 | 與「分片」不混用 |
  | shard / sharding | 分片 | |
  | replica / replication | 副本 / 複製 | |
  | consumer group | consumer group | 保留英文 |
  | worker thread | 工作執行緒 | |
  | event loop | event loop | 保留英文 |
  | idempotent / idempotency | 冪等 | |
  | backpressure | 背壓 | |
  | throughput / latency | 吞吐 / 延遲 | |
  | eventual consistency | 最終一致性 | |
  | queue | 佇列 | 一律「佇列」，勿用陸式同義詞 |
  | cache | 快取 | |
  | thread / process | 執行緒 / 行程 | |

  其餘術語第一次出現給「中文（English）」、之後擇一；拿不準保留英文。
- **條目長度（軟建議，非硬限）**：問題條目正文約 600–1500 中文字、工具條目 ≤2000；對照表/pseudo-code 不計入。深度需要可超過，但別灌水、也別短到一段帶過。
- **領域檔前言（2–4 句）**：(1) 本領域解什麼問題；(2) 條列本檔包含哪些條目；(3) 一句點明與相鄰領域的邊界（如 A 前言：「schema 協定相容在此、DB 結構變更在 B」）。
- **跨條目引用格式**：統一放句尾括號 `（見 <概念名>，領域 X）`；不插在句中。
- **對照表最小欄位**（欄名照抄、可增不可減）：`方案/工具 | 保證 | 適用場景 | 注意事項`。
- **時效性標注**：放句尾括號 `（2026-06）`，一段最多標一次。
- **`[工]` 對照深度**：同類工具列 2–4 個；本書已有專屬條目的工具，交叉引用、不重建對照表；無專屬條目的同類，簡表即可。
- **「光譜/分類型」問題條目**（如「一致性光譜」「SQL vs NoSQL」「ACID vs BASE」）：`### 解法空間` 改寫成「各選項/各維度的機制」、`### 何時需要` 改寫成「如何在光譜上選點的準則」——標題照抄、語意調整。
- **AI 專門條目**（T/U 的 AI 條目）：`### 定義與原理`＝AI 在這維度改變了什麼；`### 解法空間`＝對應的工程做法（review gate/信任邊界/驗收）；其餘段照常。

## 明確禁止（舊版的毛病，別重犯）

1. 提及任何特定作者的系統／公司／履歷／經歷（NeoBards/LetsTalk/Bahwan/結算 pipeline…一律不准出現）。
2. 第二人稱「對做過這件事的你」那種個人化口吻；改成中性的「你（泛指讀者）」或「我們」。
3. 「我以為 vs 實際」「老兵記憶會過時」「打臉/三巴掌」這類預設讀者記錯的糾錯裝置。
4. 指向作者的其他書（queue/aiib/fail/tla…）。需要深度就自足講或用外部來源。
5. 練習段（紙上推演/自我檢核）。
6. AI 維度寫成工具版本/品牌（只准原理層）。
