# T · 工程實踐與團隊推動（上）

這個領域回答的不是「哪個技術對」，而是「**怎麼讓一群人真的去用對的東西**」——一個慣例、一個工具、一條流程，技術上再正確，沒人採用就等於不存在。本檔含五個條目：推動採用的機制、RFC／設計文件流程、鋪路（paved road）、慣例的自動化強制、漸進式遷移與淘汰。它們共用同一條脊椎：你想要什麼保證（整個團隊照某個方式做事），你願意付什麼取捨（強制換來服從卻磨掉信任、鋪路換來自願採用卻得先養路）。與相鄰領域 U 的邊界一句話：**T＝決定要推某個東西之後，怎麼讓團隊採用它；U＝在那之前，怎麼判斷該不該採用它**（建 vs 買、簡單性、退場成本都在 U）。

## 推動工具/慣例的採用

### 定義與原理

「推動採用」要解的問題是：你（或一個小組）認為某個做法更好——換個 lint 規則、換個部署流程、換個 RPC 框架——但**團隊不會因為它「更好」就自動採用**。採用是一個行為改變問題，不是技術問題。第一原理有三條：

- **誘因（incentive）**：人會走對自己當下最有利的路。如果新做法讓「我這週的工作」更難交付，再好的長期理由都推不動。
- **轉換成本（switching cost）**：從舊習慣到新習慣之間，有一段「比現在更痛、但還沒拿到好處」的低谷。採用會不會發生，常常取決於這段低谷有多深、有沒有人幫忙填。
- **信任（trust）**：團隊願不願意賭你的判斷。信任是存量——它由過去推過的東西「後來證明是對的」累積而來，也會因為推過、然後爛尾、然後沒人收拾而被一次燒光。

把這三條合起來看：**採用的真正阻力不是「新東西不好」，而是「轉換成本由採用者付、好處卻歸於系統或未來」這個錯位**。推動的本質，就是想辦法把這個錯位補平——降低轉換成本、或把好處提前兌現給採用者、或用信任去借支那段低谷。

### 解法空間

讓團隊採用一個做法，大致有三種路線，外加一組通用槓桿：

- **Top-down（由上而下強制）**：管理層／架構組宣布「從某日起一律用 X」。機制：用權威壓過個人誘因。
- **Bottom-up（由下而上自然擴散）**：先在一兩個團隊做出明顯有效的範例，靠口碑與模仿擴散。機制：讓誘因自己對齊（別人看到你用 X 交付更快，自己跑來問）。
- **鋪路（paved road，讓對的做法＝最省力的路）**：不強制、也不純靠口碑，而是**主動把「照建議做」的那條路修到比任何其他路都好走**——文件、模板、預設設定、現成工具一次給足（這條獨立成一個條目，見 鋪路 paved road／golden path）。
- **通用槓桿**（可疊加在上面任一路線）：
  - **降轉換成本**：提供 codemod／自動遷移腳本、相容層、遷移期雙寫，讓採用者不用手動重寫（見 漸進式遷移與淘汰）。
  - **提前兌現好處**：讓採用者第一天就拿到看得見的甜頭（更快的 build、更少的 oncall、自動產生的樣板碼）。
  - **借信任**：找一個團隊信得過的同儕當早期採用者，讓「他用了說好」替你背書。
  - **設淘汰時程＋自動護欄**：宣布舊做法的退役日期，並用 CI gate 擋住新增的舊用法，避免「新人不知道、繼續用舊的」（見 慣例的自動化強制）。

> **AI 在這件事能做什麼、又帶來什麼新陷阱（原理層）**：AI 把「降轉換成本」這個槓桿變便宜了——過去要派人手寫的遷移腳本／codemod、把上千個呼叫點從舊 API 改到新 API，現在可以委派給 AI 大批生成，採用低谷被填平。但它也帶來一個新陷阱：**AI 能讓「採用」這個動作的成本趨近於零，於是團隊可能在還沒真正理解新做法之前就「被遷移完了」**——程式碼換成了新樣式，沒人說得清為什麼。推動者要把驗收（誰看懂、誰簽字）當成 review gate，而不是把「AI 改完、CI 綠了」當成採用成功（信任邊界的細節見領域 T 的 AI 協作條目）。

### 各方案的保證與取捨

| 方案/做法 | 效果 | 適用場景 | 注意事項 |
|---|---|---|---|
| Top-down 強制 | 一致性最強、最快達到「全員照做」 | 安全／合規／不可逆的單向決策（如棄用有漏洞的依賴）；組織小或權威清楚時 | 服從不等於認同；強推沒共識的東西會累積「合規但消極」的怨氣，下次更難推。**強制必須配自動化護欄**，否則人前照做、人後繞過 |
| Bottom-up 擴散 | 採用者是自願的、認同度高、阻力小 | 做法好壞「用了就看得出來」、有耐心等擴散時 | 慢，且可能擴散不完全（剩下的角落永遠不改）；缺乏「臨門一腳」會卡在 60% |
| 鋪路 paved road | 不靠強制就讓多數人走對路；可疊加少量強制收尾 | 中大型組織、有平台／工具團隊可以維護那條路時 | 路要持續養護，沒人維護的「曾經的最佳路」會比沒鋪更糟（人被引到一條已經壞掉的路上） |
| 派早期採用者借信任 | 用同儕背書突破「我憑什麼信你」 | 信任存量不足、或你是新加入的人 | 選錯背書人（大家不服的人）會反效果 |
| 設退役日期＋CI 擋舊用法 | 防止「採用」隨時間回滲倒退 | 任何想長期維持的慣例 | 退役日期若一再跳票，會訓練團隊「公告不必當真」 |

一個具體推演：**一個三人小團隊**要把「所有 PR 都加 commit message 規範」這件事推起來。Top-down 在三人團隊幾乎零成本——口頭講一句就到位，甚至不必寫文件。但同一件事換到**一個 50 人的工程組織**，口頭宣布會在第三個團隊就失真：有人沒聽到、有人聽到不認同、有人認同但忘了。這時純 top-down 會退化成「上面說了、下面各做各的」，必須改成「鋪路（給 commit template 與範例）＋自動化強制（commit-lint 擋不合規的 PR）＋一個受信任團隊先示範」三者疊加，才推得動。**團隊規模一變，能用的路線就變**——這是這個條目最核心的判準。

### 何時需要

- **需要刻意推動**：當「轉換成本由採用者付、好處歸於系統／未來」這個錯位明顯存在時（換框架、改流程、加約束）。錯位越大，越要主動補槓桿，不能指望自然發生。
- **不需要、甚至是 over-engineering**：當新做法對採用者「當下就更省力」時——這種東西會自己擴散，你硬搞一套推動計畫反而顯得官僚。也包括只影響一兩個人的內部偏好（用什麼編輯器），那不該被「推動」成全員慣例。
- **判準**：先問「採用者第一週是更痛還是更爽？」更爽的，鋪好路放著就好；更痛的，才需要算轉換成本、找信任、設護欄。

### 常見誤解與陷阱

- **「東西夠好就會自己被採用」**：最常見的誤解。好處歸於未來、痛苦歸於現在的東西，從不自己擴散。
- **把「強制」當「採用」**：CI 綠了、文件寫了、會開了，不等於團隊真的改了行為。沒有自動化護欄的強制，會在你不看的角落悄悄回退（見 慣例的自動化強制）。
- **一次推太多**：同時改五件事，團隊的轉換成本預算會爆，結果一件都沒落地。一次推一件、落地了再推下一件。
- **燒掉信任不補**：推一個東西、做到一半棄坑、留下半套爛攤子給別人收——信任是存量，這樣燒一次，下次你說「這次真的更好」就沒人信了。
- **忽略「最後 20%」**：bottom-up 擴散最容易卡在「大部分團隊改了、剩幾個死角永遠不改」。沒有退役日期＋護欄收尾，這 20% 會讓你永遠維護兩套。
- **AI 相關的新陷阱**：用 AI 大批改寫來「強推」採用時，很容易製造出**沒人理解的合規**——表面上全員都用了新樣式，實際上沒人能在出事時說清楚它為什麼這樣寫。把 AI 當降成本的工具沒問題，但別把「改完了」當「採用了」，採用的標準永遠包含「有人理解並能維護」。

## RFC／設計文件流程

### 定義與原理

RFC（Request for Comments）／設計文件流程，是把「一個有實質後果的技術決定」在**動手寫程式之前**寫成文件、公開徵求意見、留下決議的一套慣例。它解的問題是：**重要決定如果只活在某個人的腦子裡或某次走廊對話裡，就沒有可審查、可追溯、可挑戰的載體**——三個月後沒人記得為什麼這樣決定，半年後有人想改卻不知道當初放棄了哪些選項、為什麼放棄。

第一原理是把決策的**頻寬**從「同步、即時、少數人在場」拉成「**非同步、可留存、所有相關方都能參與**」。寫文件這個動作本身有副作用：把一個模糊的想法逼成精確的文字，常常在寫的過程中就自己發現了漏洞——這是 RFC 最被低估的價值，**很多 RFC 的好處在「寫」的時候就兌現了，不在「被批准」**。

要和它的近親分清楚：RFC／設計文件是**決策前**的提案與討論載體（多個選項、還在收意見）；ADR 是**決策後**的精簡紀錄（一個決定、它的脈絡與後果，見 架構決策紀錄 ADR，領域 T）。一份 RFC 最後常常濃縮出一條 ADR。

### 解法空間

「讓重要決定被記錄與審查」有一道輕重光譜，從幾乎零儀式到重流程：

- **走廊對話 / 口頭共識**：最輕，零留存。機制：靠在場者記憶。
- **PR 描述 / issue 討論**：把設計寫在程式碼提交的脈絡裡。機制：決策貼著程式碼，但分散、難回溯整體決定。
- **輕量 RFC（一頁設計文件）**：固定模板（問題／提案／替代方案／取捨／決議），開一個可評論的文件，設一個截止日期收斂。機制：非同步徵求意見＋留存。
- **重量 RFC（正式評審會 + 簽核）**：加上指定 reviewer、必經評審會、明確的 approve/reject 狀態。機制：用流程強制「重要決定必須被多方看過」。
- **ADR-only**：跳過提案討論，只在決定後補一條精簡紀錄。機制：最省力的留存，但失去「決策前公開徵意見」的好處。

> **AI 在這件事能做什麼、又帶來什麼新陷阱（原理層）**：AI 可以把寫 RFC 的門檻降低——草擬初稿、補齊「替代方案」一節、把口語想法整理成結構化文件，讓「寫文件太花時間所以不寫」這個藉口消失。新陷阱是：AI 草擬的 RFC 讀起來**結構完整、論證流暢，但可能掩蓋了作者其實還沒想清楚的地方**——以前一份寫得吃力的 RFC 會暴露作者的思考缺口，AI 順稿後缺口被填平了表面文字、卻沒被真正解決。Reviewer 的審查重點要從「文字是否清楚」上移到「**作者是否真的理解自己提的方案**」（這正是 RFC「寫的時候就兌現價值」那部分，被 AI 順稿削弱了）。

### 各方案的保證與取捨

| 方案/做法 | 效果 | 適用場景 | 注意事項 |
|---|---|---|---|
| 走廊對話 | 零成本、最快 | 可逆、影響小、改回來代價低的決定（two-way door，見領域 U） | 零留存；三個月後無人能重建「為什麼」；無法讓不在場的人參與 |
| PR / issue 討論 | 決策貼著程式碼、自然留存 | 範圍小、實作即決策的改動 | 整體設計散落各 PR，難看見全貌；大決定塞進 PR 描述沒人會回頭讀 |
| 輕量 RFC | 逼出精確思考＋非同步收意見＋留存，儀式適中 | 多數「值得想清楚但不必開會」的決定：新服務邊界、跨團隊介面、選型理由 | 需要有人主動收斂（設截止日期），否則文件變成永遠開放、永遠不結案的鬼魂 |
| 重量 RFC + 評審 | 一致性與把關最強，重大決定一定被多方看過 | 不可逆、跨多團隊、高成本的單向決策（換資料庫、定公司級協定） | 流程重、慢；濫用在小決定上會變成人人怕碰的官僚稅，逼大家退回走廊偷偷決定 |
| ADR-only | 留存成本最低 | 決定已成、只需記錄結論與後果 | 失去決策前公開徵意見的機會；容易變成「事後諸葛的單方宣告」 |

一個具體推演：**一個 50 人的工程組織**第一次導入 RFC 流程。最常見的失敗是「一步到位上重量流程」——要求每個技術決定都走指定 reviewer、評審會、簽核。結果是：小決定也被逼進流程，工程師覺得「光走流程比做事還久」，於是把真正的決定退回走廊、只把無關痛癢的東西寫成 RFC 應付，流程變成空轉的儀式。比較穩的做法是反過來：**先只要求「會影響其他團隊的決定」寫一頁輕量 RFC、設 72 小時評論窗、無異議即生效**，把儀式壓到最低、先讓人嘗到「寫下來後討論變清楚了」的甜頭，再視需要對「不可逆的大決定」加重流程。流程的重量要對齊**決定的可逆性**（見領域 U），不是一刀切。

### 何時需要

- **需要寫 RFC**：決定不可逆或退場成本高、影響跨越單一團隊、或「之後一定有人會問為什麼」時。可逆性越低、波及面越廣，越值得把它寫下來公開審。
- **不需要、是 over-engineering**：可逆、影響只在自己團隊、改回來幾乎零成本的決定。給這種決定套 RFC 是在繳官僚稅，還會稀釋大家對 RFC 的重視（「連這個都要寫，那 RFC 不值錢」）。
- **判準**：問「這個決定錯了，要回頭的代價多大？多少人會被它影響？半年後有人會不會問為什麼？」三個有一個重，就值得一頁 RFC。

### 常見誤解與陷阱

- **把 RFC 當審批關卡而非思考工具**：一旦團隊覺得 RFC 是「上面卡我」的關卡，就會退化成事後補件——先做完、再補一份漂亮文件走過場。RFC 的核心價值在動手前逼出思考，補件等於把價值全丟了。
- **替代方案一節造假**：很多 RFC 的「Alternatives Considered」是寫完決定後硬湊兩個明顯更差的選項來陪襯。真正有用的替代方案分析是誠實列出「這個其實也行、我們因為某個具體取捨才不選」。
- **永不結案**：開了文件、收了一堆評論、然後沒人按「決議」按鈕。懸而未決的 RFC 比沒有更糟——它讓大家以為「還在討論」而不敢動。每份 RFC 必須有 owner 與截止日期。
- **流程重到逼人繞過**：見上面 50 人組織的推演——過重的流程會把真正的決策趕回走廊，你只會看到應付用的 RFC。
- **AI 相關的新陷阱**：AI 順稿讓 RFC 表面完整度上升、但可能掩蓋作者的理解缺口。Reviewer 要小心「讀起來很順所以一定想清楚了」的錯覺，刻意去問文件沒明說的邊界與失敗模式，逼出作者真正的理解程度。

## 鋪路 paved road／golden path

### 定義與原理

鋪路（Netflix 的講法「paved road」／Spotify 的講法「golden path」）解的問題是：**在一個你不能、也不想對所有人下命令的組織裡，怎麼讓多數人自願走在對的做法上？** 答案是不靠強制，而是**把「照建議做」那條路修到比任何其他選擇都明顯省力**——一條 paved road 上鋪好了模板、預設設定、現成的 CI/CD、監控、log、權限樣板，新人沿著它走，幾分鐘就生出一個符合所有最佳實踐的服務；偏離它則要自己處理一堆瑣事。

第一原理：**人會走阻力最小的路**。強制是逆著這個天性硬掰（成本高、會反彈、要持續看守）；鋪路是順著它——你不禁止別人走野路，但你讓「對的路」變成阻力最小的那條，於是多數人自己就走上去了。它和強制的根本差異在於**自願 vs 服從**：鋪路換來的是高認同的自願採用，代價是你得先把路修好、而且持續養護。它常和少量強制收尾搭配（多數人靠鋪路自願上路，剩下死角用 CI gate 收口）。

「paved road」與「golden path」在 2026-06 業界基本同義、可互換；前者強調「鋪好的、有支援的路」，後者強調「那條被祝福的、推薦的路徑」（2026-06）。

### 解法空間

把「對的做法」變成「最省力的路」，可動的零件有：

- **腳手架／模板（scaffolding）**：一鍵生成一個新服務，CI/CD、Dockerfile、健康檢查、log 格式、監控接線全部預先配好。機制：把正確的起點變成預設起點。
- **預設即正確（secure/correct by default）**：團隊共用的 base library／base image 裡，把超時、重試、結構化 log、認證中介層設成預設值。機制：採用者什麼都不做就已經合規。
- **自助平台（internal developer platform）**：把「開一個資料庫／佇列／環境」做成自助按鈕，背後自動套好最佳設定。機制：消除「找人開、等審批」的摩擦。
- **黃金路徑文件（golden path docs）**：不是寫「所有可能性」，而是寫「**就照這條做**」的單一推薦路徑，附可複製貼上的範例。機制：消除選擇焦慮（quell rumor-driven development）。
- **少量強制收尾**：對偏離鋪路的用法用 CI gate／policy 擋住（見 慣例的自動化強制）。機制：補上純鋪路收不掉的死角。

> **AI 在這件事的影響（原理層）**：鋪路的核心是「降低照建議做的阻力」，而 AI 改變了阻力的來源——當生成樣板碼變得幾乎免費，「省下手寫樣板」這個傳統甜頭就不再稀有，鋪路的價值要上移到 AI 不便宜替代的部分：**被祝福的預設設定（哪個超時、哪個重試策略是對的）、被驗證過的整合方式、出事時有人支援**。新陷阱：AI 會樂於沿著任何路徑生成程式碼，包括野路——它不知道你的 paved road 在哪。若團隊的推薦做法沒有被寫成 AI 讀得到、用得上的形式（範例、模板、明確規則），AI 反而會成為「把人帶離鋪路」的力量，因為它總能順手生出一個能跑的非標準寫法。

### 各方案的保證與取捨

| 方案/做法 | 效果 | 適用場景 | 注意事項 |
|---|---|---|---|
| 腳手架／模板 | 新專案起點即合規，採用幾乎零摩擦 | 重複建同類服務／元件的組織 | 模板會腐——它凍結了「生成當天」的最佳實踐，沒人更新就把過時做法批量複製出去 |
| 預設即正確 | 採用者什麼都不做就合規，覆蓋面最廣 | 有共用 base lib／base image 的團隊 | base 一改影響所有人，升級要有相容策略；藏太多魔法會讓人不懂底層、出事查不動 |
| 自助平台 | 消除審批／等待摩擦，採用最順 | 中大型、有平台團隊可維護時 | 平台本身是要養的產品，沒人維護的自助平台會變成新的瓶頸 |
| 黃金路徑文件 | 最低成本，消除「該怎麼做」的選擇焦慮 | 任何規模，先寫一條推薦路徑 | 文件會過時；要綁定到實際模板／預設，否則文件說一套、模板做一套 |
| 鋪路＋少量強制收尾 | 多數自願上路、死角被收口 | 想長期維持一致性時 | 別把比例搞反——強制太多就退化成 top-down，失去鋪路的自願性 |

一個具體推演：**一個平台團隊**想統一公司「新建微服務」的做法。對照兩條路。**純強制**：發一份「微服務規範」文件，要求所有新服務照做，靠 code review 把關。結果是規範文件越寫越長、沒人讀完，每個團隊照自己的理解做出八個不一樣的「合規」服務，review 變成無止境的糾正。**鋪路**：做一個 `create-service` 腳手架，跑一行指令就生出一個已經配好 CI/CD、健康檢查、結構化 log、預設超時與重試、權限樣板的新服務骨架。工程師發現「照鋪路做」比「自己從零搭、還要查規範」省下一整天，於是自願都用它——一致性是順手得到的副產品，不是靠 review 硬掰出來的。差別在於：強制是把成本加在「不照做」上，鋪路是把好處加在「照做」上；後者順著人性、前者逆著。**但鋪路有它的硬成本**：那個腳手架是一個要持續維護的產品，最佳實踐變了要同步更新，否則它會變成「把過時做法一鍵複製到全公司」的反向放大器——這是純強制沒有的負擔。

### 何時需要

- **需要鋪路**：同一類事情被很多人重複做（建服務、開環境、接監控）、組織大到無法靠口頭傳遞、而你有（或能爭取到）持續維護那條路的人力時。重複度越高、組織越大，鋪路的投報率越高。
- **不需要、是 over-engineering**：那件事一年只做兩次、或團隊小到一句話就能傳遞共識。為了「只有三個人、一年建一個服務」的場景去搭一套腳手架＋自助平台，是把維護負擔變成比問題本身還大的新問題。
- **判準**：問「這條路會被走幾次？有沒有人能養它？」走的次數攤不平養護成本，就先別鋪、用一頁黃金路徑文件頂著。

### 常見誤解與陷阱

- **鋪了路卻不養**：最致命的陷阱。一條曾經是最佳實踐、現在已經過時或會壞的 paved road，比沒有鋪路更糟——你**主動**把所有人引導到一條壞掉的路上，而且因為它是「官方推薦」，出事時大家還以為自己沒做錯。鋪路是承諾持續維護，不是一次性發布。
- **把鋪路當強制的同義詞**：鋪路的力量在自願。一旦你開始用「不准走別條路」來補鋪路的不足，比例一過就退化成 top-down，失去了高認同這個最大好處。鋪路＋少量強制收尾可以，但主體必須是「對的路最好走」。
- **文件與實作脫節**：黃金路徑文件寫一套、實際模板做另一套。人會跟著實際能跑的東西走，不是跟著文件——文件必須綁定可執行的模板／預設，否則文件先死。
- **腳手架凍結最佳實踐**：模板生成的是「製作模板那天」的最佳實踐，之後被批量複製出去的所有服務都停在那一刻。沒有「模板更新後如何回推到已生成服務」的策略，鋪路會變成過時做法的放大器。
- **AI 相關的陷阱**：見上面——若推薦做法沒被寫成 AI 用得上的形式（模板、明確規則、可檢查的約束），AI 會順手把人帶離鋪路、生出能跑的非標準寫法。鋪路在 AI 時代要「對 AI 也可讀」。

## 慣例的自動化強制

### 定義與原理

「慣例的自動化強制」解的問題是：**靠人記憶與自律維持的慣例，會隨時間衰減為零**。一個團隊口頭約定「import 要排序」「函式要有型別標註」「commit message 要照格式」，第一週大家還記得，第三個月就鬆了，新人根本沒聽過。任何不被機器檢查的慣例，最終回到「各做各的」。自動化強制就是把慣例從「人的善意」搬到「機器的硬閘」上——讓不照做的程式碼**根本進不來**。

第一原理：**把人的判斷力從重複、機械、無趣的把關上釋放出來**。code review 裡如果 reviewer 花一半精力在抓「這裡少一個分號、那裡命名不對」，他就沒精力看真正重要的東西（邏輯、邊界、設計）。自動化把「機器查得了的」全部交給機器，人只看「機器查不了的」。第二原理是**一致性無感化**：當格式由 formatter 強制統一，沒人需要為「用幾格縮排」吵架，這類爭論直接消失。

要分清強制的不同層級：**linter** 抓潛在問題（可能的 bug、反模式）；**formatter** 統一格式（不爭對錯、只求一致）；**CI gate** 把前兩者設成「不過就不能合併」的硬閘；**codegen** 從單一真相生成程式碼，讓「手寫導致的不一致」根本不存在。

### 解法空間

把慣例變成機器強制，工具與強制點構成一張組合：

- **Formatter（格式化器）**：自動重排程式碼到統一風格。機制：消除格式爭論——不是「檢查你排對沒」，是「直接幫你排好」。
- **Linter（靜態檢查）**：掃出潛在 bug、反模式、風格違規並報錯。機制：在執行前用規則比對 AST。
- **Pre-commit hook（提交前掛勾）**：在 commit／push 前本機跑檢查。機制：在最早的時點攔下問題，回饋最快——但本機 hook 可被繞過（`--no-verify`），不能當唯一防線。
- **CI gate（持續整合關卡）**：在 PR 上跑檢查，不過就擋住合併。機制：唯一不可繞過的硬閘——它在伺服器端、不靠個人自律。
- **Codegen（程式碼生成）**：從 schema／IDL／單一真相生成 client、型別、樣板。機制：讓不一致從「要去檢查」變成「結構上不可能發生」。
- **Policy as code**：把組織級規則（依賴白名單、commit 格式、檔案結構）寫成可執行檢查掛進 CI。機制：把「規範文件」變成「會擋你的程式」。

> **AI 在這件事的影響（原理層）**：AI 大量生成程式碼後，自動化強制的角色從「幫人省把關力」升級成「**對 AI 產出的第一道、也是唯一不會累的防線**」——人 review 大量 AI 程式碼會疲勞、會放水，但 linter／CI gate 不會。某種意義上，自動化強制是讓「AI 生成」這件事可控的前提：你可以容忍 AI 寫很多碼，只要每一行都得過同一道機器閘。新陷阱有兩個：一是 AI 可能生成「**剛好騙過 linter**」的程式碼——通過所有靜態規則、邏輯卻是錯的，讓人誤以為「CI 綠了＝沒問題」；二是 AI 會學著繞過檢查（加一行 `# noqa`／`eslint-disable` 把警告壓掉），把「修正問題」偷換成「消音警告」。對策是把「禁止無理由的抑制註解」本身也變成一條被強制的規則。

### 各方案的保證與取捨

| 方案/做法 | 效果 | 適用場景 | 注意事項 |
|---|---|---|---|
| Formatter | 格式 100% 一致、零爭論、零 review 成本 | 任何團隊，越早導入越好 | 第一次套用會產生一個巨大 diff（要單獨一個 commit，別混進功能改動）；風格本身的選擇要一次定、別反覆改 |
| Linter | 自動抓潛在 bug／反模式 | 任何團隊 | 規則太嚴會淹沒在 false positive 裡、訓練大家無腦抑制；要挑「訊號強」的規則，逐步加嚴 |
| Pre-commit hook | 回饋最快（commit 前就攔） | 想早一點接到回饋時 | **可被 `--no-verify` 繞過**，只能當便利、不能當保證；真正的閘必須在 CI |
| CI gate | 唯一不可繞過的硬閘 | 任何想「保證」而非「鼓勵」的慣例 | 跑太慢會拖垮開發節奏，逼人想辦法跳過；要平衡嚴格度與速度 |
| Codegen | 消除整類「手寫不一致」 | 有單一真相（schema/IDL）可生成時 | 生成物別手改（會在下次生成被蓋掉）；生成器本身要納入版本控管與 review |
| Policy as code | 組織級規則自動化、可審計 | 中大型組織、有合規／安全約束時 | 規則即程式，本身會有 bug；過嚴的 policy 會逼出大量例外申請，變成新官僚 |

一個具體推演：**一個三人小團隊**要導入一條新 lint 規則（比如「禁止用 `any` 型別」）。最自然的做法是寫進 lint 設定、設成 error。但如果直接設成「CI 擋」，會立刻撞上一個現實：**既有程式碼裡早就有兩百個 `any`**，於是第一個 PR 就紅一片，沒人能合併任何東西，團隊只好把規則關掉——推動失敗。比較穩的做法是分三步走（這正是漸進式遷移在「規則」上的應用，見 漸進式遷移與淘汰）：第一步把新規則設成 **warning 而非 error**（看得到、不擋路）；第二步用工具一次清掉存量、或把存量檔案加進**豁免清單**（baseline），讓「新增的 `any`」報錯、「既有的」暫時放過；第三步逐步清豁免清單、直到可以把規則升成全域 error。**自動化強制的關鍵不在「設成 error」那一刻，而在「怎麼讓存量不擋住新規則上線」**——這是初次導入任何強制最容易翻車的地方。

### 何時需要

- **需要自動化強制**：當一個慣例「值得全員照做」且「機器查得了」時。這兩個條件都成立，就該把它從 review 與口頭約定搬進機器——人不該重複做機器做得更好的事。
- **不需要、是 over-engineering**：當規則「機器查不準」（需要人對上下文判斷的設計品味），或「違反它幾乎沒後果」（純偏好）時。硬把這類東西塞進 linter，只會製造大量 false positive，訓練團隊無腦抑制警告——最後連有意義的警告也一起被忽略。
- **判準**：問「機器查得準嗎？查錯的代價（false positive）大不大？」查得準、誤報少的，自動化；要靠上下文判斷的，留給人。

### 常見誤解與陷阱

- **把 pre-commit hook 當保證**：本機 hook 可被 `--no-verify` 繞過、可能沒裝、不同人環境不同。它是便利、不是保證。真正的閘必須在伺服器端的 CI——「在我機器上有跑」不是防線。
- **規則太嚴淹沒在 false positive**：一個誤報率高的 linter 規則，會訓練全團隊養成「看到警告就無腦加抑制註解」的反射，最後連真警告也被一起壓掉。寧可少而準，逐步加嚴。
- **存量擋住新規則**：見上面三人團隊的推演——不處理存量就把規則設成 error，第一個 PR 就紅一片、規則被迫關掉。新規則上線必須有 baseline／豁免存量的策略。
- **強制了格式卻沒人定義格式**：導入 formatter 卻為「用哪套風格」吵不完，等於沒導入。formatter 的價值在「一致」不在「對」——隨便選一套、定下來、別再改，比選「最好的」重要。
- **CI 太慢逼人繞過**：一道要跑二十分鐘的 CI gate，會逼開發者想盡辦法跳過或抱怨到把它關掉。強制的嚴格度要和執行速度平衡，慢到讓人痛恨的閘最終會被拆掉。
- **AI 相關的陷阱**：AI 生成的碼可能「剛好騙過 linter」（過靜態檢查、邏輯仍錯），讓人誤把「CI 綠」當「正確」；AI 也會學著用抑制註解消音警告而非真正修正。把「無理由的抑制註解」本身設成被擋的違規，是必要的對策。

## 漸進式遷移與淘汰

### 定義與原理

「漸進式遷移與淘汰」解的問題是：**你想把一個正在被使用、不能停的東西換掉**——一套舊系統、一條舊 API、一個舊流程、一條舊慣例——但「停下來、一次切換到新的、再開回去」（big-bang cutover）的風險高到不可接受：切換當下若出錯，沒有回頭路、沒有可比對的舊系統、blast radius 是全部。漸進式遷移的核心思想是**把一次性的大爆炸，拆成許多次可回退的小步**，讓新舊在一段時間內並存、流量逐步轉移、每一步都能觀測與回滾。

兩個經典模式構成這條的骨幹：

- **Strangler Fig（絞殺榕）**：Martin Fowler 命名（他原稱 "Strangler Application"，後改名 "Strangler Fig" 以淡化暴力意象，靈感來自他 2001 年在昆士蘭雨林觀察到的絞殺榕——藤蔓在宿主樹上生長、逐步包覆、最終取而代之）。做法：在舊系統外圍長出新系統，攔截請求、一塊一塊把功能搬到新系統，直到舊系統被「絞殺」殆盡才拆除。它對應「**用在系統**」的退役。
- **Expand-Contract（擴張-收縮，又名 Parallel Change，Martin Fowler 於 2014 記述並定名（技術最早由 Joshua Kerievsky 2006 提出））**：把一個破壞性變更拆成三段——**expand**（先加上新的、不動舊的，兩者並存）→ **migrate**（把使用方逐步切到新的）→ **contract**（確認沒人用舊的了，才移除）。它對應「**用在流程／介面／慣例**」的演進（一條 lint 規則的上線、一個設定鍵改名、一個欄位改型別，都可以走這三段）。

第一原理是同一條：**任何「不能停的東西要換掉」的安全做法，都是讓新舊並存一段時間、流量可控地轉移、每步可回退**——而不是賭一次切換不出錯。

### 解法空間

把「換掉不能停的東西」做成可控的漸進過程，手法有：

- **Strangler Fig + 請求攔截**：在舊系統前放一層路由／facade，按條件（路徑、租戶、百分比）把請求導向新或舊。機制：攔截點就是流量旋鈕，逐步擰過去。
- **Expand-Contract（parallel change）**：新舊並存的三段式——先擴張、再遷移、最後收縮。機制：永遠先有退路（舊的還在），確認後才拆退路。
- **雙寫（dual-write）/ 雙讀比對**：遷移期同時寫新舊兩邊、讀時比對結果，確認新的和舊的一致才切讀。機制：用「可比對的並存」買到信心（資料層的雙寫一致性細節屬資料同步範疇，見領域 L）。
- **Feature flag 漸進放量**：把切換做成可即時開關、可按比例放量的旗標。機制：把「切換」從一次部署降級成一個可隨時關掉的開關（canary／漸進放量的部署面見領域 Q）。
- **Baseline／豁免存量**：對流程／規則類的淘汰，先把存量違規凍結成 baseline、只擋新增，再逐步清 baseline。機制：讓新規則不被存量擋住上線（見 慣例的自動化強制）。
- **設退役日期＋擋新用法**：給舊東西一個明確 sunset 日期，並用 CI gate／deprecation 警告擋住新增的舊用法，防止淘汰倒退。機制：堵住「邊淘汰邊有人新增舊用法」的漏。

> **AI 在這件事的影響（原理層）**：遷移最苦的部分常是「把上千個呼叫點從舊樣式改到新樣式」這種大規模機械重複，AI／codemod 正好擅長這個——它能把 expand-contract 的 migrate 段、或 strangler 過程中的批量改寫，從「派人改幾週」壓到「生成＋審查幾天」（AI 輔助大規模變更的細節是領域 T 的獨立條目）。新陷阱：AI 讓「改」變便宜，會誘使人**跳過漸進、直接一次全改**——既然 AI 一晚上能改完所有呼叫點，何不一次切換？但遷移的風險從來不在「改的工作量」，而在「切換當下出錯沒有退路」。AI 降低了 migrate 的成本，沒有降低 big-bang 的風險；正確的用法是用 AI 加速每一個小步，而不是用它取消「分步」本身。

### 各方案的保證與取捨

| 方案/做法 | 效果 | 適用場景 | 注意事項 |
|---|---|---|---|
| Strangler Fig | 舊系統零停機、逐塊替換、隨時可停在中途 | 換掉「不能停、且夠大到無法一次重寫」的系統 | 並存期要同時養新舊兩套＋攔截層，複雜度與成本在過渡期翻倍；若沒有明確終點，會永遠卡在「新舊都還在」 |
| Expand-Contract | 破壞性變更零停機、永遠有退路、可回滾 | 介面／流程／慣例／schema 的演進 | 三段都要走完——最常見的失敗是做完 expand＋migrate 就忘了 contract，留下永久的舊殘骸 |
| 雙寫＋比對 | 用可比對的並存買到切換信心 | 資料正確性關鍵、切錯代價高時 | 雙寫本身有一致性問題（一邊成功一邊失敗）；比對與切換邏輯本身會出錯，要當成需被測試的程式 |
| Feature flag 放量 | 切換降級為可即時關掉的開關、可按比例 | 任何想「先放 1% 看看」的切換 | flag 是技術債——用完要清，否則 flag 堆積成新的複雜度泥潭 |
| Baseline 豁免存量 | 新規則不被存量擋住、可立即上線 | 流程／規則類的漸進收緊 | baseline 若永遠不清，等於規則永遠只管一半，存量問題被「合法化」了 |
| 退役日期＋擋新用法 | 防止淘汰倒退、給並存期一個終點 | 任何想真正退役舊東西的場合 | 退役日期一再跳票會訓練團隊「不必當真」，並存期無限延長 |

一個具體推演：要把一個**舊單體服務**的「使用者通知」功能搬到新服務。Big-bang 的做法是某天部署新服務、同時把單體裡的通知關掉、流量全切過去——一旦新服務有沒測到的問題，那一刻所有通知全掛、且沒有舊系統可立即切回，blast radius 是 100%。Strangler 的做法是：在單體前放一層路由（這個攔截點正是 Strangler 提供的），在攔截點上再疊一層**比例放量（canary，見領域 Q）**——先讓 **1% 的通知**走新服務、99% 仍走舊的；觀測新服務的錯誤率與延遲，沒問題就升到 5%、20%、50%、100%；過程中任一步出問題，路由旋鈕一擰就切回舊的，blast radius 始終被框在「當前放量比例」內。等 100% 都穩定走新服務、舊路徑一段時間零流量了，才進入 contract——把單體裡的舊通知程式碼刪掉。**這裡最容易被忽略的是最後一步**：很多遷移做到「流量全切過去」就宣告完成，舊程式碼留在原地沒人敢刪（怕還有人用），半年後變成「沒人懂、沒人敢動、但還在編譯」的殭屍——expand-contract 的 contract 段不是可選的收尾，是遷移真正結束的標誌。

### 何時需要

- **需要漸進式遷移**：當被換掉的東西**正在被使用、不能停**，且 big-bang 切換出錯的代價（無退路、blast radius 涵蓋全部流量）不可接受時。系統越關鍵、切換越不可逆，越要拆成可回退的小步。
- **不需要、是 over-engineering**：當東西沒人在用、或停機可接受、或範圍小到一次換完即使出錯也能五分鐘改回來時。給一個內部小工具的改名套上 strangler＋雙寫＋放量，是把五分鐘的事做成五週的工程。
- **判準**：問「一次切過去若出錯，回得來嗎？會炸到多少人？」回不來、或會炸到很多人，才需要漸進；回得來、炸得小，big-bang 反而更省。

### 常見誤解與陷阱

- **只做 expand＋migrate、忘了 contract**：最普遍的陷阱。新的加上了、流量切過去了，但舊的（舊欄位、舊 API、舊程式碼、舊 flag）沒人敢刪，永久殘留。遷移沒有 contract 段就沒真正結束——你只是把「一套系統」變成「永遠並存的兩套」，複雜度不減反增。
- **並存期沒有終點**：strangler 過程若沒有明確的退役日期與「擋新用法」的護欄，會卡在「新舊都還在、都有人用、都不敢動」的永久過渡態——這比換之前更糟。
- **以為 AI 讓你可以跳過分步**：AI 把「改的工作量」變便宜了，但遷移的風險在「切換當下沒退路」，那部分 AI 一點沒幫上。用 AI 加速每個小步，不是用它取消「分步」。
- **雙寫當成免費的保險**：雙寫本身有一致性問題（一邊寫成功、一邊失敗，兩邊就漂了），比對與切換邏輯本身也會有 bug。它是要被當作程式來測試與監控的東西，不是貼上去就安心的保險。
- **flag／baseline 堆積成新債**：feature flag 用完不清、豁免 baseline 永遠不收，會從「遷移工具」變成「新的技術債泥潭」。每個過渡裝置都要有「何時拆掉」的計畫。
