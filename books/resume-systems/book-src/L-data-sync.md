# L · 資料同步與整合

這個領域回答的是：**同一份資料以多個副本、或以多種型態，散在不只一個地方時，怎麼讓它們保持一致——而當它們不一致時，怎麼發現、怎麼修。** 本檔含六個條目：資料同步總覽（full／incremental、push／pull、batch／streaming 的座標軸）、CDC 變更資料擷取、dual-write 問題與 outbox、衝突解決（LWW／版本向量／CRDT）、對帳 reconciliation、跨區與異質庫同步。它們共用同一條脊椎：你想要兩邊資料一致到什麼程度（即時／最終／可容忍多少漂移），你願意付什麼取捨（延遲、寫入放大、複雜度、一份真相 vs 多份各自演化）。與相鄰領域的邊界一句話：**一致性「模型」面（最終一致是什麼、補償怎麼算）屬領域 B；本檔只講「機制」面——怎麼把變更搬過去、衝突怎麼判、兩邊不一致怎麼比對與修**。交付機制（outbox 怎麼運作、saga 補償）由領域 A 深講，本檔從資料同步角度引用、不重講。

## 資料同步總覽

### 定義與原理

資料同步要解的問題是：**一份資料的「來源版本」變了，怎麼讓「目的地版本」也跟著變，而且在可接受的時間內、用可接受的代價。** 「來源」與「目的地」可以是同一份資料的兩個副本（主庫→唯讀副本）、兩種儲存（DB→搜尋索引、DB→快取）、或兩個服務各自的私有庫。第一原理是：**同步永遠是「把『變更』從一端搬到另一端」**——差別只在你怎麼界定「變更」、什麼時候搬、搬多少。

任何同步方案都落在三條正交的座標軸上，先把這三軸講清楚，後面的 CDC、outbox、對帳都只是這三軸的具體取值：

- **full（全量）vs incremental（增量）**：每次搬「整份資料」還是只搬「上次同步後變動的部分」。全量簡單、無狀態、天生自我修復（每次都覆蓋成最新），但代價隨資料量線性成長；增量便宜、可即時，但要維護「上次同步到哪」的位置（watermark / cursor / log offset），且一旦這個位置算錯就會漏資料或重複。
- **push（推）vs pull（拉）**：來源端主動把變更推給目的地，還是目的地週期性來問來源「有什麼新的」。push 延遲低、但來源要知道有哪些下游、下游掛了要處理重試與背壓（見領域 E）；pull 解耦（來源不必認識下游）、下游自己控節奏，但有輪詢延遲、且高頻輪詢會空轉打爆來源。
- **batch（批次）vs streaming（串流）**：累積一批一起搬（每小時、每天一個 job），還是一筆變更發生就近即時搬一筆。batch 吞吐高、單位成本低、易重跑，但資料有「批次窗口」的陳舊度；streaming 新鮮度高，但要長期運行的管線、要處理亂序與背壓、運維面更重。

這三軸可任意組合：「full + pull + batch」＝每晚把整張表 dump 一份匯出（最樸素的 ETL）；「incremental + push + streaming」＝CDC 把每筆變更即時推給下游（見 CDC 變更資料擷取）。沒有哪一組永遠對——選哪一組，取決於資料量、可容忍的陳舊度、以及來源端能不能可靠地告訴你「什麼變了」。

### 解法空間

把「怎麼界定變更、什麼時候搬」拆開，常見做法有：

- **全量覆蓋（full snapshot replace）**：定期 dump 整份，目的地整批換掉。機制：不追蹤變更，用「覆蓋」換取「不必正確追蹤」。
- **時間戳輪詢（timestamp / high-watermark polling）**：來源表有 `updated_at`，目的地記住上次同步的最大時間戳，每輪只拉 `updated_at > watermark` 的列。機制：用「修改時間」當增量游標。
- **觸發器 / 應用層雙寫（trigger / app-level dual-write）**：在寫來源的同一條路徑上，順手把變更也寫一份給同步用（DB trigger 寫一張變更表、或應用程式同時寫兩處）。機制：在寫入點就分叉（dual-write 的一致性風險見 dual-write 問題與 outbox）。
- **日誌型擷取（log-based CDC）**：直接讀資料庫的交易日誌（WAL／binlog），把每筆 commit 的 row-level 變更解析出來。機制：把「資料庫已經為了持久化而寫的 log」當成變更來源（見 CDC 變更資料擷取）。
- **outbox / event-carried state transfer**：把「狀態變更」當事件寫進同一交易的 outbox 表，再可靠地投遞出去（見 dual-write 問題與 outbox）。

### 各方案的保證與取捨

| 方案/做法 | 保證 | 適用場景 | 注意事項 |
|---|---|---|---|
| 全量覆蓋 batch | 最終一定收斂到來源現值（每次覆蓋自我修復）；無「漏一筆」風險 | 資料量小、可容忍每日級陳舊、來源無可靠變更信號 | 成本隨資料量線性漲；覆蓋窗口內目的地是舊的；大表全量會壓垮來源與網路 |
| 時間戳輪詢增量 | 只搬變動列，便宜；實作門檻低 | 來源有可信 `updated_at`、刪除可用軟刪除標記 | **抓不到硬刪除**（DELETE 不留痕）；同一秒多筆更新易漏；時鐘回撥或批次未提交會漏抓；watermark 邊界要用「≥ 且去重」防漏 |
| 應用層雙寫 | 延遲低、即時 | 只有兩個目的地、且能接受偶發不一致 | 兩次寫非原子，中間崩潰就分叉（見 dual-write 問題與 outbox） |
| log-based CDC streaming | 抓得到所有變更含硬刪除、不漏序、對來源負擔低 | 需即時、需完整變更（含 DELETE）、資料量大 | 要運維長運管線；綁資料庫日誌格式；初始全量快照＋切換到 log 的接縫要處理 |
| outbox + 事件投遞 | 變更與事件同交易，至少一次投遞 | 跨服務、要把「狀態變更」當事件廣播 | 至少一次→下游要冪等（見領域 A） |

一個帶數字的權衡：一張 **5,000 萬列、每列約 1 KB** 的使用者表（約 50 GB）要同步到搜尋索引。**全量**每次搬 50 GB，若每晚跑一次、網路有效 100 MB/s，光傳輸就約 **500 秒**，且這 50 GB 對來源是一次重讀；資料新鮮度＝最多 24 小時陳舊。**增量輪詢**若每天實際變動 **0.5%＝25 萬列**，只搬約 250 MB、約 2.5 秒，新鮮度可降到輪詢間隔（如 1 分鐘）——但代價是你必須正確處理硬刪除與同秒衝突，否則索引會慢慢長出「來源已刪、索引還在」的鬼資料。**這就是同步的核心取捨：全量用成本換正確性（不會漏），增量用正確性的風險換成本與新鮮度。**

### 何時需要

- **需要 streaming／incremental**：資料量大到全量搬不動、或業務要求秒級新鮮度（搜尋、即時報表、跨服務狀態廣播）。
- **full batch 就夠、上 streaming 是 over-engineering**：資料量小（幾萬列）、下游能接受每日級陳舊、團隊沒有運維長運管線的人手。一個每晚重跑的全量 job 無狀態、好除錯、天生自我修復——別為了「即時」硬上 CDC 管線然後三個月後沒人會修。
- **判準**：先問「能不能容忍 N 小時陳舊」與「來源有沒有可靠的變更信號」。能容忍且資料小→full batch；不能容忍或資料大→往 incremental／streaming 走，並接受要處理刪除、亂序、watermark 的複雜度。

### 常見誤解與陷阱

- **「增量一定比全量好」**：增量便宜，但把「漏一筆」的風險轉嫁給你的 watermark 邏輯。全量雖貴，但「每次覆蓋」這個性質讓它天生自我修復——很多團隊低估了「永遠不會悄悄漂移」的價值。
- **用 `updated_at` 輪詢卻忘了刪除**：時間戳輪詢只看得到 INSERT／UPDATE，硬 DELETE 不留時間戳。目的地會累積「來源已不存在」的殭屍列，且這種漂移無聲無息。要嘛改軟刪除、要嘛改用 log-based CDC。
- **watermark 用 `>` 而非 `>=`＋去重**：用嚴格大於，邊界那一秒的列可能在你讀取的瞬間還沒提交，下一輪又被跳過。安全做法是 `>=` 取重疊窗口、下游靠主鍵去重（見 對帳 reconciliation 與領域 A 的去重）。
- **push 不管背壓**：來源端一有變更就猛推，下游消化不了就堆積、甚至拖垮來源。push 必須配有界佇列與背壓（見領域 E）。
- **把同步當「設定一次就忘」**：任何 incremental 管線都會在某次崩潰／重啟／schema 變更後悄悄落後或分叉。**沒有對帳的同步＝沒有保證的同步**（見 對帳 reconciliation）。

## CDC 變更資料擷取

### 定義與原理

CDC（Change Data Capture，變更資料擷取）要解的問題是：**怎麼可靠且即時地知道「資料庫裡哪一列、被改成了什麼」，而不必去輪詢整張表、也不漏掉任何變更（包含刪除）。** 第一原理很漂亮：**關聯式資料庫為了崩潰恢復，本來就把每一筆變更循序寫進交易日誌（PostgreSQL 的 WAL、MySQL 的 binlog）。這份 log 是事實上「所有變更的完整、有序、已提交的清單」**——CDC 做的就是把這份既有的 log 讀出來、解析成「INSERT/UPDATE/DELETE + 前後值」的串流，餵給下游。

對比之下，「查詢型 CDC」（query-based，即上一條的時間戳輪詢）是用 `SELECT ... WHERE updated_at > X` 去問資料庫「什麼變了」。它能用，但有三個天生缺陷：抓不到硬刪除、抓不到同一輪內的中間狀態（一列被改三次只看到最後一次）、且輪詢本身有負擔。**log-based CDC 沒有這三個缺陷**，因為它讀的是「事實發生的順序」而非「現在的快照」。

### 解法空間

- **log-based（日誌型）**：讀 WAL／binlog 解析變更。PostgreSQL 透過 logical decoding，輸出外掛常見有 **pgoutput**（PG 10+ 內建、預設、無需額外安裝）、**wal2json**（JSON 格式）、**decoderbufs**（Protobuf）；MySQL 讀 row-format binlog（2026-06）。代表工具 **Debezium**（開源、跑在 Kafka Connect 上，最新穩定版約 3.5.x（2026-06）、持續維護）。機制：訂閱 replication slot／binlog，把 commit 串流化。
- **query-based（查詢型 / 輪詢）**：靠 `updated_at` 游標輪詢（見 資料同步總覽）。機制：週期性 SELECT 增量列。
- **trigger-based（觸發器型）**：在表上掛 AFTER INSERT/UPDATE/DELETE trigger，把變更寫進一張審計表，下游讀審計表。機制：在 DB 層攔截寫入。
- **timestamp + tombstone**：輪詢 + 對刪除改用軟刪除（標 `deleted_at`）讓查詢型也抓得到刪除。機制：把刪除轉成可被輪詢看見的更新。

### 各方案的保證與取捨

| 方案/工具 | 保證 | 適用場景 | 注意事項 |
|---|---|---|---|
| log-based（Debezium 等） | 完整變更含硬刪除、嚴格保留 commit 順序、對來源負擔最低（讀 log 不掃表） | 即時、需完整變更串流、資料量大、要餵多個下游 | 需 replication slot（slot 未消費會讓 WAL 無限堆積、撐爆磁碟）；綁 DB 日誌格式；初始快照→log 切換的接縫；schema 變更要處理 |
| query-based 輪詢 | 實作最簡、無需 DB 特權 | 小資料、可容忍輪詢延遲、無硬刪除 | 抓不到硬刪除與中間狀態；輪詢有負擔；同秒/邊界易漏 |
| trigger-based | DB 層保證每筆寫入都被攔；可在 trigger 裡加工 | DB 不支援邏輯解碼、或需在擷取點做轉換 | trigger 拖慢每筆寫入（在交易關鍵路徑上）；trigger 邏輯難維護、易成隱形耦合 |
| 軟刪除 + 輪詢 | 讓便宜的輪詢也能抓到刪除 | 已用輪詢、只缺刪除這一塊 | 軟刪除列堆積要定期清理；應用層全程要記得帶 `WHERE deleted_at IS NULL` |

一個帶數字的場景：一張 **3,000 萬列**的訂單表要把變更同步到資料倉與搜尋索引兩個下游。用輪詢：每分鐘 `SELECT ... WHERE updated_at > X`，若索引非覆蓋、得掃描判斷，高峰每分鐘掃描成本可觀，且若一筆訂單在一分鐘內從 `pending→paid→shipped` 改三次，輪詢只會看到 `shipped`——中間兩個狀態對下游永遠不存在。用 log-based CDC：WAL 裡這三次 UPDATE 是三筆有序事件，下游拿到完整軌跡；對來源的額外負擔只是「多一個消費 replication slot 的連線」。**代價是那個 slot 必須一直被消費**——若 Debezium 掛了一天沒人發現，PostgreSQL 為了保留未消費的 WAL 會持續累積，磁碟可能在無預警下被填滿。這是 log-based CDC 最現實的運維陷阱。

### 何時需要

- **需要 log-based CDC**：要完整、有序、含刪除的變更串流；資料量大到不能輪詢掃表；要把同一份變更扇出給多個異質下游（搜尋、快取、資料倉、其他服務）。
- **輪詢或 trigger 就夠**：資料量小、能容忍延遲、沒有硬刪除需求、或 DB／權限不允許開 replication slot。在受管雲端資料庫上，logical replication 權限有時要特別申請，這也是選型考量。
- **過度工程警訊**：只有一個下游、且該下游能接受每日批次時，架一整套 Kafka + Debezium 是把運維複雜度提前付清卻沒對應收益。

### 常見誤解與陷阱

- **「CDC 給你 exactly-once」**：不會。CDC 串流本質是至少一次投遞——管線重啟、offset 重播都會讓同一筆變更出現兩次。下游必須冪等／去重（見領域 A）。
- **忽略 replication slot 的反壓代價**：未被消費的 slot 會釘住 WAL／binlog 不讓回收，磁碟被悄悄填滿。CDC 消費端的存活與落後量是必須監控的一級指標。
- **以為 CDC 會自動同步 schema**：CDC 搬的是資料變更，不是 DDL。來源加一欄、改型別，下游 schema 與消費邏輯要另外協調（協定相容見領域 A，DB 結構變更見領域 B）。
- **把 CDC 事件當「業務事件」**：CDC 給的是「`orders` 表第 N 列的 `status` 從 A 變 B」這種資料層事件，不是「訂單已出貨」這種業務語意事件。把資料表結構直接洩漏成跨服務契約，會讓下游緊耦合到你的表結構；要業務語意該用 outbox 發明確事件（見 dual-write 問題與 outbox）。
- **初始快照被忽略**：CDC 只給「從現在起」的變更，歷史資料要先做一次初始全量快照、再無縫接上 log 串流。這個接縫沒處理好會漏掉快照期間的變更。

## dual-write 問題與 outbox

### 定義與原理

dual-write（雙寫）問題是分散式系統最常見的隱性資料漏洞：**一個操作要「同時」更新兩個沒有共同交易邊界的系統——例如『寫資料庫』＋『發一則訊息到 broker』、或『寫主庫』＋『寫搜尋索引』——而這兩次寫入無法是原子的。** 第一原理很簡單也很殘酷：兩次獨立的網路/磁碟寫入之間，永遠存在一個「第一次成功、第二次還沒做」的時間窗，任何在這個窗內的崩潰都會讓兩端分叉。

具體拆解失敗排列：先寫 DB 成功、再發訊息——若發訊息前行程崩潰，DB 變了但下游永遠收不到（漏事件）。反過來先發訊息、再寫 DB——若寫 DB 失敗但訊息已發出，下游收到一個「其實沒發生」的事件（幽靈事件）。加上重試又可能造成重複。**沒有任何「兩次寫入的順序」能消除這個窗**——這是 dual-write 的本質：你想要兩個系統原子一致，但你手上只有兩個各自獨立的交易。

解法的核心洞見是：**把「兩次跨系統寫入」收斂成「一次本地交易」**。既然 DB 寫入本身是原子的，那就把「要通知下游的這件事」也寫進同一個 DB 交易裡——這就是 **outbox 模式**：在同一個本地交易內，既更新業務資料表、也往一張 `outbox` 表插一筆「待發送事件」。交易要嘛全成功（資料與事件都在），要嘛全回滾（兩者都沒有），分叉的窗被消滅。之後再由一個獨立的投遞器把 outbox 裡的事件可靠地搬到 broker。

> outbox 的投遞機制、relay 怎麼讀 outbox、與 saga（choreography vs orchestration）的補償邏輯，由領域 A 深講（見 outbox / saga，領域 A）。本條目只從「資料同步／一致性」角度切入：**為什麼非用它不可、以及它換來什麼語意**。

這裡值得獨立講的是 **event-carried state transfer（事件攜帶狀態傳遞）**：outbox 發出的事件，可以只攜帶「發生了什麼 + 主鍵」（event notification，下游要自己回來查狀態），也可以攜帶「這個實體的完整新狀態」（event-carried state transfer，下游不必回查、直接把狀態存成自己的本地副本）。後者讓下游服務能持有一份它需要的資料的本地複本、降低同步呼叫耦合——這正是資料同步的一種形態：**用事件流，把一個服務的狀態變更，持續同步成另一個服務的本地讀模型**（與 EDA／CQRS 的關係見領域 K）。

### 解法空間

- **outbox 表 + 投遞器**：業務寫入與「事件」同交易寫 outbox 表，獨立 relay 投遞。機制：用本地交易的原子性涵蓋「資料 + 待發事件」（見領域 A）。
- **log-based CDC 當投遞器**：不另設投遞器，直接用 CDC 讀 outbox 表（或直接讀業務表）的變更當事件源。機制：把 outbox 投遞外包給日誌擷取（見 CDC 變更資料擷取）。
- **listen-to-yourself / 事件先行**：先寫一筆事件（單一寫入），消費自己的事件去更新狀態。機制：把「狀態更新」也變成事件的下游，避免雙寫源頭。
- **2PC（兩階段提交）**：用 XA／分散式交易讓兩個系統真原子。機制：協調者跨資源協調 prepare/commit（取捨見領域 M）。
- **接受不一致 + 對帳**：明知會偶發分叉，靠週期性對帳偵測並修補（見 對帳 reconciliation）。機制：用事後修復替代事前原子。

### 各方案的保證與取捨

| 方案/做法 | 保證 | 適用場景 | 注意事項 |
|---|---|---|---|
| outbox 表 + relay | 資料與「待發事件」原子一致；事件至少一次投遞 | 寫 DB 同時要可靠通知下游／發事件的絕大多數場景 | 下游必須冪等（至少一次）；outbox 表要清理；多一個 relay 元件要運維（見領域 A） |
| CDC 讀 outbox/業務表 | 同上，且省掉自寫 relay | 已有 CDC 管線 | 直接讀業務表會把表結構洩漏成契約，建議仍走 outbox 表發明確事件 |
| 事件先行 / listen-to-yourself | 源頭只有一次寫入、無雙寫 | 系統可重塑成「狀態＝事件的投影」（event sourcing 取向） | 改變整個寫入模型、學習曲線陡（見領域 K） |
| 2PC | 跨兩系統真原子提交 | 兩端都支援 XA 且需強一致、低頻 | 同步阻塞、協調者單點、效能差、broker 多半不支援；分散式系統普遍避用（見領域 M） |
| 接受不一致 + 對帳 | 最終收斂（非即時） | 偶發分叉可容忍、有對帳兜底 | 不一致窗內下游是錯的；對帳本身要建（見 對帳 reconciliation） |

一個帶數字的推演：一條每天處理 **100 萬筆**訂單的路徑，要「寫訂單 DB ＋ 發一則 `order.created` 給通知服務」。若用天真雙寫，假設「寫 DB 成功後、發訊息前崩潰」的機率僅 **0.01%**（萬分之一），每天仍有約 **100 筆**訂單資料存在、但通知永遠不發——這 100 個使用者下了單卻收不到任何確認。換成 outbox：訂單與 `order.created` 同交易落地，崩潰只會讓整筆回滾（使用者會看到下單失敗、重試即可），絕不會出現「單成立但事件遺失」。代價是 relay 採至少一次投遞，下游通知服務要用冪等鍵去重，否則少數訂單可能收到兩封通知——**這正是 outbox 的語意取捨：把『可能遺失』換成『可能重複』，而重複是可以靠冪等消化的、遺失不行。**

### 何時需要

- **需要 outbox**：任何「寫本地 DB 且必須可靠地讓外部知道」的場景——發領域事件、同步到另一服務、餵搜尋索引/快取且不容遺失。這幾乎是事件驅動架構的預設地基。
- **不需要、是 over-engineering**：操作只動單一系統（沒有第二個寫入端）；或下游遺失可容忍且本就有全量對帳兜底（如純分析用、隔天重算）。為一個無關緊要的旁路事件架 outbox + relay 是過度設計。
- **判準**：問「第二次寫入若靜默失敗，會不會造成使用者可見的、不可接受的不一致？」會→outbox；不會→可接受裸雙寫 + 對帳。

### 常見誤解與陷阱

- **「先寫 DB 再發訊息就安全了」**：順序不能消除窗。任何「兩次獨立寫入」的排列都有崩潰點，差別只在分叉方向（漏 vs 幽靈）。
- **「outbox 給 exactly-once」**：outbox 給的是「資料與待發事件原子一致」＋「至少一次投遞」。下游仍會看到重複，仍需冪等（見領域 A）。
- **把 broker 的 publish ack 當交易保證**：訊息成功送進 broker，不代表它和你的 DB 寫入是原子的——publish 成功而 DB 交易最後回滾，就製造了幽靈事件。原子性只能靠「同一個本地交易」取得，不是靠 broker 的 ack。
- **直接拿業務表做 CDC 當事件源**：能動，但把資料表結構變成跨服務契約，下游就緊耦合到你的 schema，你連改個欄位名都會打到別人。要長期穩定的契約，發明確的 outbox 事件、而非洩漏表結構。
- **忘了 outbox 表會無限長**：投遞成功的列要定期清理／分區輪替，否則 outbox 表自己變成效能問題。

## 衝突解決

### 定義與原理

當同一份資料允許在多個地方各自被獨立更新（多主複製、離線編輯後再同步、跨區雙寫），就會出現**衝突**：兩個更新並發地（concurrent，互相沒看到對方）改了同一個值，同步到一起時系統得決定「最後算誰的」。衝突解決要解的就是這個問題——**而且要解到讓所有副本最終收斂到同一個確定結果，不論訊息以什麼順序、重複幾次到達。**

第一原理是區分「並發」與「有因果順序」。若更新 B 是在「看到了 A 之後」才做的，那 B 因果上晚於 A，沒有衝突——B 該贏。真正的衝突只發生在兩個更新**並發**（誰都沒看到誰）時。能不能正確判斷「並發 vs 因果先後」，是衝突解決品質的分水嶺：**只看牆鐘時間戳的方案無法區分這兩者**（時間戳大不代表因果上更晚），而版本向量/向量時鐘能精確判斷。收斂的數學保證最強的一端是 **CRDT**——它把資料結構設計成「合併操作天生滿足交換律、結合律、冪等」，於是不論順序、不論重複，所有副本必然收斂（Strong Eventual Consistency，強最終一致）。

### 解法空間

- **LWW（Last-Write-Wins，最後寫入者贏）**：給每個寫入一個時間戳，衝突時保留時間戳較大的。機制：用一個全序（通常是牆鐘）強行打破平手。Cassandra 預設即 LWW，以 microsecond 時間戳逐 cell 比較（2026-06）。
- **版本向量 / 向量時鐘（version vector / vector clock）**：每個副本維護一組 `{副本→計數}`，寫入時遞增自己的計數。比較兩個向量可精確判定「A 因果先於 B」「B 先於 A」還是「並發」。並發時不自動丟棄，而是保留多個版本（sibling）交給應用層或使用者解決。Dynamo（2007）用版本向量；其改良 **DVV（Dotted Version Vectors）** 避免 false conflict，用於 Riak（2026-06）。機制：追蹤因果，把真衝突暴露出來而非掩蓋。
- **CRDT（Conflict-free Replicated Data Type，無衝突複製資料型別）**：把資料設計成合併必然收斂的型別（計數器、集合、序列…）。由 Shapiro、Preguiça、Baquero、Zawirski 於 **2011** 形式化。機制：靠代數性質保證收斂，無需中央協調。
- **應用層合併（application-defined merge）**：把並發版本交給領域邏輯自訂合併（如購物車取聯集、餘額取相加）。機制：用業務語意決定怎麼合。
- **單主排序（avoid by single-writer）**：根本不讓並發發生——所有寫某個 key 的請求路由到單一主（見領域 M 的複製策略）。機制：用「不產生衝突」取代「解決衝突」。

CRDT 內部又分兩大流派，收斂語意不同：

- **state-based（CvRDT，convergent）**：副本間交換「整個狀態」，用一個 **merge** 函式合併。merge 必須是 **交換律、結合律、冪等**的「join」，狀態落在一個 **join-semilattice（聯結半格）** 上、且更新只會讓狀態單調「往上」走。因為 merge 冪等且可交換，訊息重複、亂序、重送都不影響最終結果——這讓它對不可靠網路（甚至 gossip／anti-entropy，見領域 M）極其寬容。代價是傳整個狀態較重（有 delta-CRDT 等優化）。
- **op-based（CmRDT，commutative）**：副本間只傳「操作」（如「+3」「加入元素 x」），每個副本重放收到的操作。要求並發操作**兩兩可交換**，但通常**要求投遞層保證每個操作恰好被送達一次、且滿足因果順序**（exactly-once、causal delivery）。傳輸量小，但對投遞通道的要求比 state-based 嚴。

一句話對照：**state-based 把「正確性」放進 merge 函式（半格＋冪等），對通道幾乎無要求；op-based 把一部分正確性外包給投遞層（恰好一次＋因果序），換取更小的傳輸量。**

### 各方案的保證與取捨

| 方案/做法 | 保證 | 適用場景 | 注意事項 |
|---|---|---|---|
| LWW（時間戳） | 確定收斂到「時間戳最大」者；實作最簡、無 sibling | 衝突罕見、丟掉並發的另一筆可接受（如 last-seen、設定旗標） | **靠牆鐘 → 時鐘偏移會讓「較晚的真實寫入」被較早的覆蓋＝靜默 lost update**；無法表達「兩筆都該保留」 |
| 版本向量 / 向量時鐘 | 精確區分並發 vs 因果先後；不靜默丟資料 | 需要正確判斷因果、願意讓應用層解 sibling（購物車類） | 向量大小隨副本數成長；sibling 解決邏輯要寫；客戶端要回傳 context |
| CRDT | 強最終一致：不論順序/重複必收斂、無需協調 | 多主/離線協作/計數器/集合/協同編輯 | 型別受限（要能塑成 CRDT）；某些語意（如「全域唯一約束」「餘額不可為負」）天生塞不進 CRDT；metadata 開銷 |
| 應用層合併 | 用業務語意得到「對的」合併結果 | 合併規則有明確領域意義時 | 要為每種資料寫合併邏輯；寫錯就靜默產生錯資料 |
| 單主排序避免衝突 | 根本不產生衝突，語意最簡單 | 能接受單主的可用性/延遲代價（跨區尤其） | 主故障要 failover；跨區寫要繞到主、延遲高（見領域 M） |

一個帶數字的 LWW 陷阱：兩個應用節點對同一個 key 寫入，節點 A 的時鐘比節點 B **快 50 ms**。使用者先在 B 寫了正確的新值（真實時間 `t`，但 B 打的時間戳是 `t`），50 ms 後在 A 寫了一個其實該被新值取代的舊操作（真實時間 `t+50ms`，A 打的時間戳卻是 `t+100ms`，因為 A 快）。LWW 比時間戳：`t+100ms > t`，於是 A 那筆「真實上較舊、但時間戳較大」的寫入贏了——**使用者剛存的正確值被靜默覆蓋成舊的，沒有任何錯誤、沒有任何日誌**。這就是為什麼「LWW + 牆鐘」在有並發寫的系統裡是個有牙的陷阱：它總會收斂，但可能收斂到錯的值。版本向量在這裡會判定兩者並發、保留 sibling，把「該保留誰」的決定權交還給應用而非交給不可靠的時鐘。

### 何時需要

- **需要顯式衝突解決**：採用多主／無主複製、或支援離線編輯後再同步、或跨區雙寫。只要「同一 key 能在多處並發被改」，就得選一種衝突策略。
- **不需要**：單主架構（所有寫某 key 都到同一主、由主定序）——這時根本沒有並發衝突要解，別為不存在的問題引入 CRDT 的複雜度。
- **選 LWW vs 版本向量 vs CRDT 的判準**：丟掉並發另一筆可接受且衝突罕見→LWW（最簡）；不能靜默丟資料、且能讓應用解 sibling→版本向量；要無人工介入的自動收斂、且資料能塑成 CRDT 型別→CRDT；有強約束（唯一、非負）塞不進 CRDT→退回單主排序。

### 常見誤解與陷阱

- **「LWW 安全因為它總能決出贏家」**：它總有贏家，但贏家可能是錯的。LWW 的「最後」是**時間戳的最後**，不是**真實因果的最後**；牆鐘偏移會讓它靜默丟掉本該保留的寫入。
- **以為向量時鐘自動解衝突**：版本向量只負責**偵測**衝突（判定並發），不負責**解決**。並發時它給你多個 sibling，怎麼合還是要 LWW／應用邏輯／使用者選——它的價值是「不靜默丟、把衝突誠實暴露」。
- **以為 CRDT 萬用**：CRDT 對「能塑成半格／可交換操作」的資料才成立。需要全域不變量（庫存不可超賣、餘額不可為負、用戶名全域唯一）的場景，CRDT 天生做不到——那類約束需要協調（共識／單主，見領域 M）。
- **混淆 state-based 與 op-based 的通道要求**：把 op-based CRDT 接到一個會重複投遞的通道上、卻沒保證恰好一次與因果序，會破壞收斂。state-based 對通道寬容（冪等 merge），op-based 不寬容——選錯流派配錯通道是隱性 bug。
- **把因果順序和接收順序搞混**：訊息「先到」不代表它因果上更早。衝突解決的正確性建立在因果關係上，不是 socket 的到達順序上。

## 對帳 reconciliation

### 定義與原理

對帳要解的問題是：**你有兩份本該一致的資料（主庫 vs 副本、DB vs 搜尋索引、你的帳 vs 對方的帳、來源 vs 同步出去的目的地），怎麼系統性地確認它們真的一致——而當它們不一致時，怎麼定位差在哪、怎麼把漂移修回去。** 第一原理是一句殘酷的事實：**任何長期運行的同步管線，最終都會漂移。** 崩潰、重啟、漏處理一則訊息、bug、手動改庫、schema 變更接縫——每一個都會在兩端之間留下一點差。**沒有對帳的同步，等於沒有保證的同步**：你以為一致，但沒有任何機制能告訴你何時不再一致。對帳就是這個「持續驗證 + 主動修復」的機制。

> 一致性「模型」面（最終一致是什麼、補償交易的語意）屬領域 B；本條目是**本書對帳機制的 owning 條目**，只講機制：**怎麼比對兩邊、怎麼定位差異、怎麼把漂移修回去**。

### 解法空間

對帳的核心是「用最少的比對成本找出兩邊的差」，做法按「比多細」與「比多省」排開：

- **全量逐筆比對（full row-by-row）**：兩邊各拉出來，逐主鍵比每個欄位。機制：最徹底、找得到一切差，但成本＝資料量。
- **整體 checksum / 計數比對**：先比「兩邊總列數」「某欄位的總和（金額類）」「整批的 hash」。機制：用聚合值快速判斷「有沒有差」，但不告訴你差在哪。
- **分段 checksum（chunked / bucketed hash）**：把 key 空間切成 N 段，每段算一個 hash，只比 hash；只有 hash 不同的段才往下鑽。機制：用分治把比對成本從「資料量」降到「差異量」。
- **Merkle tree（雜湊樹）**：把分段 checksum 做成樹——葉子是各段 hash，內部節點是子節點 hash 的 hash。先比 root hash，相同就整棵一致；不同就只往 hash 相異的子樹下鑽。機制：交換的資訊量正比於「差異量」而非「資料量」，Cassandra/DynamoDB 的 anti-entropy 即用此（2026-06）。
- **以 log/事件重放驗證**：拿來源的變更 log 重放一遍，比對目的地是否吻合。機制：用事實序列當基準。

找到差之後的**修復**做法：

- **以來源為準覆蓋（source-of-truth wins）**：認定一邊是真相，把另一邊差異列覆蓋成來源值。最常見、最簡單。
- **重放缺漏變更（replay）**：把目的地漏掉的那幾筆從來源重新投遞（需來源可重放，如 CDC offset 回播）。
- **產差異報表給人工裁決（金融類）**：金額對不上時不自動覆蓋，產出 break list 交人工查清原因再修。

### 各方案的保證與取捨

| 方案/做法 | 保證 | 適用場景 | 注意事項 |
|---|---|---|---|
| 全量逐筆比對 | 找得到所有欄位級差異，最徹底 | 資料量小、或週期性深度稽核、金額/合規 | 成本＝資料量；對線上庫是重讀，要排離峰或讀副本；兩邊「同一瞬間快照」難對齊（移動目標問題） |
| 整體 checksum / count | 極快判斷「有沒有差」 | 第一道哨兵、高頻輕量健檢 | 只說有沒有差、不說差在哪；hash 碰撞理論上可能（用強 hash 可忽略） |
| 分段 / Merkle tree | 比對成本正比於差異量、不正比資料量 | 大資料集要定期對帳、anti-entropy | 兩邊分段邊界與排序要一致；資料持續變動時 hash 是移動目標，要對齊快照點 |
| log 重放驗證 | 以事實序列為基準、可定位漏哪筆 | 有完整變更 log（CDC/event sourcing） | 需 log 可保留夠久、可回放 |
| 以來源為準覆蓋修復 | 修復後必收斂到來源 | 來源明確是唯一真相 | 會無條件蓋掉目的地的本地修正；若「目的地才是對的」就蓋錯方向 |
| 人工裁決差異報表 | 不盲目覆蓋、保留稽核軌跡 | 金額/合規/雙方各有帳本 | 慢、要人力；只適合 break 量可控 |

一個帶數字的場景：對帳一張主庫與唯讀副本各 **1,000 萬列**的表。**全量逐筆**要把雙邊各 1,000 萬列拉出比對，I/O 與記憶體都重，且在拉取的幾分鐘內資料還在變（移動目標）。改用 **Merkle tree**：把 key 空間切成譬如 1,024 段，雙邊各算 1,024 個段 hash（外加上層樹節點）。若實際只有 **3 段**裡共 **50 列**不一致，root hash 一比就知「有差」，往下鑽兩三層 hash 後鎖定那 3 段，最後只需逐筆比對那 3 段（約 3 萬列）就能定位出 50 筆差異列——**比對與傳輸的工作量正比於『差異所在的少數段』，而非 2,000 萬列總量**。代價是雙邊的分段邊界、排序規則、快照時間點必須對齊，否則 hash 會因「比的不是同一瞬間的同一批列」而全段不符，反而退化成全量。

### 何時需要

- **必須有對帳**：任何長運的 incremental／streaming 同步（CDC、雙寫、多主）、任何「兩個系統各記一份本該相等的帳」的場景（金額尤其——對不上是事故）。把對帳當同步管線的標配，不是可選加值。
- **可省略或極輕量**：full-overwrite 同步本身每次覆蓋＝自帶對帳（不會漂移）；或資料無關緊要、不一致無後果。
- **頻率判準**：金額類→每日強對帳 + 即時聚合哨兵；一般資料→定期 Merkle 對帳 + 持續的計數/checksum 哨兵。漂移的代價越高、對帳越頻繁越細。

### 常見誤解與陷阱

- **「同步管線沒報錯就代表一致」**：最危險的假設。漏一則訊息、消費端靜默重啟、一個 bug——都不會「報錯」，只會留下無聲的漂移。一致是要被**主動驗證**的，不是「沒出錯就有」。
- **比對「移動目標」**：線上資料在你拉取兩邊的過程中還在變，逐筆比會冒出一堆「假差異」（其實只是時間差）。對帳要對齊快照點（同一個一致性快照、或容許一個 grace window 把「正在途中的變更」排除）。
- **找到差就無腦以來源覆蓋**：若差異的成因是「目的地有一段本地修正」或「來源自己有 bug」，盲目以來源覆蓋會把對的蓋成錯的。修復前要先理解差異的**方向與成因**，金額類尤其要人工確認再修。
- **只比計數不比內容**：兩邊列數相同不代表內容相同（一邊多一筆 X、少一筆 Y，計數仍相等）。計數/總和是便宜哨兵，不能取代欄位級或 hash 級比對。
- **對帳本身沒有可觀測性**：對帳跑了沒、找到幾筆差、修了幾筆、還剩幾筆——這些必須是有告警的指標。一個「悄悄掛掉三週沒人發現」的對帳 job，比沒有對帳更危險（給了虛假的安心感）。

## 跨區與異質庫同步

### 定義與原理

這個條目把前面所有機制放進兩個最硬的現實約束：**(1) 跨地理區域（multi-region）——副本之間隔著幾十到上百毫秒的網路、且分區隨時可能發生；(2) 異質庫（heterogeneous）——來源與目的地是不同的資料庫（PostgreSQL ↔ MySQL、關聯式 ↔ 文件庫、OLTP ↔ 資料倉），型別系統、交易語意、甚至「一列長什麼樣」都不同。** 要解的問題是：在這兩個約束下，怎麼把資料同步過去、且想清楚你實際拿到的是什麼一致性。

第一原理是兩條物理／工程的不可繞過：**跨區的光速延遲是物理常數**（東京↔維吉尼亞往返就是上百毫秒，不是調參能消的），所以「跨區強同步寫」必然慢；而 **CAP 在跨區分區時逼你二選一**（分區期間要嘛拒寫保一致、要嘛放行寫接受分叉，理論面見領域 B）。異質庫則是「**沒有共同的交易、沒有共同的型別語意**」——你不能用一個 DB 交易跨兩個不同的庫，型別與 NULL／時間／精度語意還會在轉換中悄悄走樣。

### 解法空間

**跨區（multi-region）：**

- **單主跨區（single-leader, async replica in other regions）**：一個區當主、其他區是非同步唯讀副本。機制：寫都回主、讀可就近，跨區只承擔複製延遲。
- **多主跨區（multi-leader, active-active）**：每區都能寫，跨區非同步互相複製、靠衝突解決收斂。機制：用本地寫低延遲換「必然有並發衝突」（見 衝突解決）。
- **同步複製 / quorum 跨區**：寫要等跨區多數確認（如 R+W>N）。機制：用延遲換強一致（見領域 M 的 quorum、共識）。

**異質庫（heterogeneous）：**

- **CDC + 型別映射管線**：用 log-based CDC 從來源擷取變更，經一個轉換層做型別/編碼映射，寫入異質目的地。機制：把「變更」標準化成中介格式再落地（見 CDC 變更資料擷取）。
- **ETL / ELT 批次**：定期全量或增量抽取、轉換、載入。機制：批次 + 顯式轉換腳本。
- **雙寫到兩種庫**：應用層同時寫兩種庫（有 dual-write 風險，見 dual-write 問題與 outbox）。

注意：PostgreSQL 內建的 logical replication 是**同構**（PG→PG）方案，且**不複製 DDL／schema**（建表與後續 ALTER 都要兩端手動同步，2026-06）；要 PG↔MySQL 這種異質，得靠 Debezium 等 CDC 或專用遷移工具（如 AWS DMS）。

### 各方案的保證與取捨

| 方案/工具 | 保證 | 適用場景 | 注意事項 |
|---|---|---|---|
| 單主跨區 + 非同步副本 | 寫線性化於主；讀就近（但可能讀到舊值） | 讀多寫少、可接受跨區寫延遲、要單一真相 | 副本有複製延遲（讀己之寫問題，見領域 B）；主所在區故障要 failover、可能丟尚未複製的尾端寫入 |
| 多主 active-active | 本地寫低延遲、區故障仍可寫 | 各區要低延遲寫、能設計出可收斂的衝突策略 | **必然有並發衝突**，要選 LWW/版本向量/CRDT（見 衝突解決）；有強約束（唯一/非負）的資料不適合 |
| 跨區 quorum / 同步 | 跨區強一致 | 不容分叉的關鍵資料（少量、可付延遲） | 每次寫付跨區 RTT；分區時犧牲可用性（見領域 M） |
| CDC + 型別映射 | 即時、含刪除、解耦異質兩端 | PG↔MySQL、DB→資料倉/索引 | 型別/編碼/NULL/時間精度映射要顯式定義且測試；schema 變更要管 |
| ETL/ELT 批次 | 簡單、可重跑、轉換邏輯集中 | 可容忍批次陳舊、分析用 | 非即時；增量 ETL 仍有 watermark/刪除問題（見 資料同步總覽） |

一個帶數字的跨區推演：主在維吉尼亞（us-east），副本在東京（ap-northeast），跨區往返網路延遲約 **150 ms**（單程約 75 ms）。若採**單主非同步副本**，東京的使用者讀就近副本只需本地往返（幾 ms），但寫必須回到維吉尼亞主——一次寫的網路成本就是約 **150 ms RTT**，且東京副本落後主一個「複製延遲」（健康時數十至數百 ms，網路抖動時可達秒級），所以東京使用者「剛寫完、立刻讀東京副本」可能讀不到自己剛寫的（讀己之寫，見領域 B）。若改採**跨區 quorum 同步寫**（N=3 分三區、W=2），每次寫要等至少一個遠端區確認，寫延遲被釘在「最近的那個遠端區 RTT」量級——強一致拿到了，但每筆寫都付這個延遲。**這就是跨區的鐵律：你能買到低延遲讀、或低延遲寫、或強一致，但跨上百毫秒的距離，三者很難同時要。**

一個異質庫的隱性坑（帶具體值）：把 MySQL 的 `DATETIME`（無時區、秒或微秒精度）同步到 PostgreSQL 的 `timestamptz`（帶時區）。若轉換層沒顯式指定來源時區，一個存成 `2026-06-21 09:00:00` 的 MySQL 值，可能被當成 UTC、也可能被當成伺服器本地時區寫進 PG——同一筆資料在兩庫表示的「絕對時刻」差了好幾個小時，且這種錯**不會報錯**，只會在報表/排序時悄悄錯位（時間語意坑見領域 M）。異質同步的轉換層必須對型別/時區/編碼/NULL 語意逐一顯式定義並測試。

### 何時需要

- **需要 multi-region**：使用者分佈在多大洲要低延遲、或要區級容災（單區故障仍可服務）、或資料落地合規要求資料留在特定區。
- **需要異質同步**：OLTP 庫要餵分析倉/搜尋/快取；遷移期新舊庫並行（見領域 B 的零停機遷移）；不同服務選了不同最適儲存但要共享資料。
- **過度工程警訊**：使用者都在單一區、無容災硬需求時，硬上 active-active multi-region 是把「並發衝突 + 跨區運維」的巨大複雜度提前付清卻沒收益——單區 + 跨區唯讀副本（純災備）通常已足夠。
- **判準**：跨區先問「能不能單主就好（讀多寫少）」，能就別碰多主；異質先問「能不能就用 CDC + 明確映射」，別輕易裸雙寫到兩種庫。

### 常見誤解與陷阱

- **「加個跨區副本就有多區高可用」**：非同步副本只是災備，不是 active-active。主區掛了要 failover、且可能丟掉尚未複製的尾端寫入；把「有副本」當成「能無縫接管」是常見誤判。
- **低估跨區延遲是物理常數**：上百毫秒的跨洲 RTT 不是調 TCP 參數或加機器能消的。任何「每個請求都要跨區同步往返一次」的設計，延遲天花板就被釘死在那個 RTT。
- **異質同步只搬資料、不搬語意**：型別、時區、NULL、字元編碼、數值精度、自增主鍵語意在兩庫間都可能不同。「值搬過去了」不代表「意思一樣」——時區與精度走樣是最常見的靜默資料損壞。
- **以為 PG 邏輯複製能同步 schema**：PostgreSQL logical replication 不複製 DDL，加欄位/改型別要兩端手動同步、且通常要在訂閱端先加（additive 先行）才不會卡住複製（2026-06）。把它當「全自動同步含結構」會在第一次 schema 變更就出事。
- **多主卻沒想清楚衝突**：開了 active-active 卻沒顯式選衝突策略，等於把「最後算誰的」交給預設 LWW + 牆鐘——於是悄悄 lost update（見 衝突解決）。多主的第一個設計問題永遠是「衝突怎麼收斂」，不是「怎麼把資料複製過去」。
