# 保證與取捨——後端與分散式系統的概念地圖

這不是一本教你「某個工具怎麼用」的書，而是一張**概念地圖**：對後端與分散式系統裡反覆出現的每一個老問題，把「有哪些解法、各自給什麼**保證**、各自要付什麼**取捨**、什麼時候才需要這種保證」攤開成一整張選項地圖。

讀完一個條目，你腦中要有的不是單一答案，而是**這個問題的整張選項空間與取捨軸**。

## 這本書的脊椎：保證與取捨

書名就是脊椎。每一個概念底下，問的都是同一件事：

> **你想要什麼保證（guarantee）？你願意付什麼代價（trade-off）？**

at-least-once 保證「不漏」，代價是「可能重複」；快取保證「快」，代價是「可能讀到舊值」；共識保證「大家同意同一個值」，代價是「多一輪往返、少數派要等」。整本書就是在各種題目上反覆做這道選擇題。

## 寫給誰

寫給**一般資深後端／軟體工程師**。預設你會寫程式、熟悉 HTTP／資料庫／並發／雲端的工作級使用——所以本書不從零解釋這些是什麼，而是直接切進「機制、取捨、陷阱」這一層：你會用、但未必有系統化想過底層與選項空間的那一層。

語氣是工程師對工程師、平視、有觀點。不打高空、不喊口號。

## 兩副閱讀透鏡（先讀 Part 0）

Part 0 的兩個領域刻意放最前面，是讀後面每一條技術概念時要一直戴著的眼鏡：

- **U（工程判斷與決策）**：我會怎麼**判斷／選**這個東西？可逆嗎、自建還是採用、是不是過度工程、退場成本多高。
- **T（工程實踐與團隊推動）**：我會怎麼讓**團隊採用**它？誘因、轉換成本、鋪路、自動化強制、漸進遷移。

兩副透鏡都各自融入了「與 AI 協作」這個維度——但只談**原理層**（什麼交給 AI、怎麼設驗收與信任邊界、帶來哪些新故障模式），不綁任何工具版本或品牌，因為那一層腐得最快。

## 怎麼組織的

約 150 個概念，分成 22 個領域（以字母 A–V 標記），收進 8 個 Part：

| Part | 主題 | 領域 |
|---|---|---|
| 0 | 透鏡：怎麼判斷、怎麼推動 | T 實踐推動、U 判斷決策 |
| 1 | 訊息、交付與整合 | A 訊息交付與佇列 |
| 2 | 資料、一致性與儲存 | B 一致性與交易、O 儲存引擎、G 快取、L 資料同步 |
| 3 | 連線、網路與 API | N 網路協定、C 即時傳輸、H API 與閘道、F 身分與存取 |
| 4 | 並發、過載與韌性 | D 並發模型、E 過載控制、P 韌性容錯、S 效能延遲 |
| 5 | 分散式系統核心 | M 共識、複製、協調、時間 |
| 6 | 架構、雲端與發布 | K 解耦、J 雲端原語、Q 部署發布 |
| 7 | 觀測與營運文化 | I 可觀測性與品質、V 營運與事故文化 |

每個領域是側欄裡的一組檔；大領域（A、B、M、I、T）切成數檔，但同屬一個字母。

## 每個條目回答什麼

兩種條目，各有固定的六段骨架：

- **問題型**（一個老問題）：定義與原理 → 解法空間（不只一種辦法）→ 各方案的保證與取捨（對照表）→ 何時需要 → 常見誤解與陷阱 → 延伸閱讀。
- **工具型**（一個具體工具）：是什麼與內部機制 → 在哪些系統扮演什麼角色 → 保證與限制 → 跟替代品的取捨 → 常見誤解與陷阱 → 延伸閱讀。

招牌是**對照表**：方案 × 保證 × 注意事項。沒有「注意事項」那一欄的對照表是危險品——它會讓你以為某個選項沒有代價。

## 怎麼讀

- **先把 Part 0（T／U）讀過**，戴上兩副眼鏡。
- 之後可以**從任何一個 Part 切入**——每個條目自足，需要鄰近概念時用「（見 <概念名>，領域 X）」帶你過去。
- 想查某個保證怎麼選，翻**附錄 A（保證速查表）**；查名詞翻**附錄 B（術語表）**；想往外讀翻**附錄 C（延伸閱讀總地圖）**。

## 這本書不做的事

- **不含練習**。它是參考書，不是習題本。
- **不綁特定系統或公司**。所有案例都是通用情境。
- **自足，但會往外指**：需要的深度本書講到夠；延伸閱讀只放外部真實連結（RFC／論文／官方文件），不外包給別的讀物。
- **時效性事實標時點（2026-06）**。版本、授權、產品狀態這類會過期的事，都標了日期、查證了來源——但讀到時，請當它是「截至那個時點」的快照。
