# ch14 — 多屬性決策：當權衡不只一個維度

> **本章解決什麼問題**：前面十三章，那個帶不確定性的決策永遠只有一個維度的報酬——錢、效用、或一個後驗下的最適行動。但你真實做過的選型決策，從來沒有這麼乾淨：選一個資料庫，要同時看成本、延遲、可靠度、生態、鎖定風險。本章把單一效用推廣到多個彼此衝突的屬性——多屬性效用理論（MAUT）與加權加總法的直覺，以及它最危險的那個地方：權重看起來像客觀計算，其實是你先有結論再回填的價值判斷。這是 Part IV「有限理性」的第二站——上一章（ch13）講「要不要把分析算到底」，本章講「當報酬不是一個數而是一張表，怎麼把它壓回一個排序」。

## 從你已知的出發

你做過的最痛的決策，多半不是「要不要 rollback」這種一維的。一維的決策其實好做——你算個期望成本，比大小，完。真正讓你在 design doc 裡卡三天、在 RFC 底下吵二十條 comment 的，是這種：

**選一個資料庫。** Postgres 還是 DynamoDB？自架還是託管？候選有四個，每個在好幾條軸上各有死穴：A 便宜但延遲尾巴難看；B 延遲漂亮但每月帳單嚇人；C 生態最成熟但會把你鎖死在某家雲；D 樣樣中庸但社群小、出事沒人救。

你心裡很清楚這不是一個數能解決的。於是你做了一件幾乎所有資深工程師都做過的事：**開一張表。** 縱軸列候選，橫軸列你在乎的維度——成本、p99 延遲、可靠度、生態、鎖定風險——每一格打個分（1 到 5，或 1 到 10），最後乘上權重加起來，誰總分高選誰。

```text
       成本   延遲   可靠   生態   鎖定    加權總分
  A      5     2     3     4     5       ?
  B      2     5     4     3     4       ?
  C      3     4     5     5     1       ?
  D      4     3     3     2     3       ?
```

這張表有個名字。決策論叫它**加權加總值函數（additive value function）**，v = Σ wⱼ·vⱼ(xⱼ)——把每個屬性的分數 vⱼ 乘上權重 wⱼ 加起來。它正是本章的主角，也是 Keeney 與 Raiffa 1976 年那本《Decisions with Multiple Objectives》整本書在小心翼翼處理的東西。

我要先說一句煞風景的話，整章圍著它轉：**這張表最危險的地方，不是你算錯加法，而是它讓一個價值判斷穿上了計算的外衣。** 你以為你在「算」哪個資料庫好，其實你在用一組你自己也說不清從哪來的權重，把「我比較想選哪個」這件事包裝成「數字告訴我選哪個」。等一下你會看到，同樣合理的兩組權重，能讓結論直接翻轉——而當這發生時，那不是巧合，是個警訊。

## 從一個效用到一張表：問題長什麼樣

先把問題定義清楚。前面所有章節，一個結果 x 對應一個效用 u(x)：賭注贏 200 對應 √300，檢查治療對病人對應 −10。報酬是一個數，效用也是一個數，排序天經地義——數大的排前面。

多屬性問題的麻煩在於：一個選項不再對應一個數，而是對應一個**向量**。資料庫 A 不是「值 7 分」，而是「成本 5、延遲 2、可靠 3、生態 4、鎖定 5」這一串。問題來了——

**兩個向量，誰排前面？**

向量本身沒有天生的全序。如果 A 在每一個維度都不輸 B、且至少一維贏，那叫 A **支配（dominate）** B，這種好事不用算就知道選 A。但真實世界幾乎沒有支配——真實世界長的是這樣：A 便宜，B 快；你想要的兩件事剛好分散在不同選項上。這時候「誰比較好」這個問題**沒有客觀答案**，因為它取決於你願意拿多少延遲去換多少成本。

決策論對這件事的回答是：把向量壓成一個數。最簡單、也最常用的壓法，就是加權加總——

```text
每個屬性先各自正規化成一個可比的「分數」 vⱼ(xⱼ)
（例如把成本映射到 0~1：最貴的 = 0，最便宜的 = 1）

再給每個屬性一個權重 wⱼ（通常要求 Σwⱼ = 1）

合成：  v(選項) = w₁·v₁ + w₂·v₂ + … + wₙ·vₙ

最後比 v 的大小，誰高選誰。
```

這個動作很自然，自然到你從沒懷疑過它。但它偷偷塞進了兩個假設，而這兩個假設正是本章要拆的兩顆地雷：① 每個屬性可以**獨立**評分，一維的好壞跟別維的水準無關（這叫偏好獨立）；② 各維度可以**互相補償**——某維很爛可以被另一維很好補回來（這是加總法的本質）。先記住這兩句，後面每個陷阱都從它們長出來。

### 一個必要的提醒：值函數不是效用函數

這裡要慢一下，因為它正是 Keeney–Raiffa 小心區分、而科普常糊掉的地方。

前面 ch03 講的 u(x) 是**效用函數**，它的形狀（凹）編碼了你在**不確定下**的風險態度——凹代表面對賭局時風險趨避。但本章這張選型表，多半是在**確定**之下做的：你不是在賭「資料庫 A 有 50% 機率很便宜」，你大致知道 A 就是便宜、B 就是快。在確定下比較「強度」用的是**值函數（value function）** vⱼ，它編碼的是「偏好的強弱」而不是「對風險的態度」。

兩者在數學形狀上可能長得像，但問的問題不同：效用函數回答「面對這個賭該不該接」，值函數回答「在這幾個確定的選項裡偏好排序怎麼排、強度差多少」。本章談的多屬性合成，主場是值函數；當某個屬性本身帶不確定（例如「可靠度」其實是個機率分布），才需要把那一維升級回效用函數。本章為了把直覺講清楚，主要停在確定的值函數版——這也是 Keeney–Raiffa 的入門路徑。

## worked example：兩個資料庫，三個屬性，一組權重

把候選縮到兩個、屬性縮到三個，好讓我們能一步步把數字算完、不跳步。

情境：你要在兩個資料庫之間選一個，在乎三件事——**月成本、p99 延遲、可靠度**。原始數據是這樣：

```text
              月成本(USD)    p99延遲(ms)    可靠度(年故障次數)
  選項 X         1200           40              2
  選項 Y         2000           15              1

  X：便宜、但慢、且故障較多
  Y：貴、但快、且更穩
```

注意這是個沒有支配的局面：X 在成本贏，Y 在延遲和可靠贏。誰好，取決於你怎麼權衡。開始壓成一個數。

**第一步：正規化。** 三個屬性量綱完全不同（美元、毫秒、次數），不能直接加。先各自映射到 0~1，方向統一成「越大越好」（成本和延遲是越小越好，要反過來）。用最簡單的線性正規化——在這個兩選項的小局裡，每個維度的最好值給 1、最差值給 0：

```text
  成本（越小越好）：  X=1200 是較便宜 → v=1.00；Y=2000 是較貴 → v=0.00
  延遲（越小越好）：  Y=15  是較快   → v=1.00；X=40   是較慢 → v=0.00
  可靠（越小故障越好）：Y=1  是較穩  → v=1.00；X=2    是較不穩 → v=0.00

  正規化後的分數表：
              成本    延遲    可靠
       X       1.00    0.00    0.00
       Y       0.00    1.00    1.00
```

（這個「最好=1、最差=0」的兩點正規化故意選最簡單的版本，好讓算術透明。真實情況你可能用更細的映射，但那不改變本章要講的脆弱性——脆弱性來自權重，不來自正規化的精細度。）

**第二步：給一組權重。** 假設你問了團隊，大家覺得成本最重要，延遲次之，可靠再次之。你定了一組：

```text
  w(成本) = 0.5    w(延遲) = 0.3    w(可靠) = 0.2    （Σ = 1.0）
```

**第三步：加權加總。** 逐項乘起來、加總，手算：

```text
  v(X) = 0.5·1.00 + 0.3·0.00 + 0.2·0.00
       = 0.50     + 0        + 0
       = 0.50

  v(Y) = 0.5·0.00 + 0.3·1.00 + 0.2·1.00
       = 0        + 0.30      + 0.20
       = 0.50
```

複核一下：v(X)=0.5·1=0.50。v(Y)=0.3+0.2=0.50。**打平。** 0.50 對 0.50。

這個剛好打平是我故意挑的數字，因為它把整章的重點逼到台面上：你站在懸崖邊。結論「選哪個」此刻完全懸在權重上，一陣微風就吹過去了。

## 那陣微風：把權重動一點點

現在做一件你在真實 review 裡一定遇過的事——有人說「我覺得成本沒那麼關鍵吧，我們又不是新創，穩定比較重要」。於是權重微調，成本從 0.5 降到 0.45，那 0.05 挪給可靠：

```text
  新權重： w(成本)=0.45   w(延遲)=0.30   w(可靠)=0.25   （Σ 仍 = 1.0）

  v(X) = 0.45·1.00 + 0.30·0.00 + 0.25·0.00 = 0.45
  v(Y) = 0.45·0.00 + 0.30·1.00 + 0.25·1.00 = 0.55

  → Y 贏（0.55 > 0.45）
```

再做一次相反的微調——「不對，我們現在最該省錢」，成本拉到 0.55，從可靠那挪 0.05 過來：

```text
  新權重： w(成本)=0.55   w(延遲)=0.30   w(可靠)=0.15   （Σ = 1.0）

  v(X) = 0.55·1.00 + 0.30·0.00 + 0.15·0.00 = 0.55
  v(Y) = 0.55·0.00 + 0.30·1.00 + 0.15·1.00 = 0.45

  → X 贏（0.55 > 0.45）
```

看清楚發生了什麼。我**沒有改任何一格的評分**——成本、延遲、可靠的原始數據一個字都沒動。我只把成本的權重在 0.45 和 0.55 之間挪了 0.1，結論就從「選 Y」翻成「選 X」。而 0.45、0.5、0.55 這三個權重，**哪一個是對的？** 你說得出來嗎？沒有人說得出來。它們都「很合理」。

這就是加權加總法的脆弱核心，整章我最想讓你記住的一頁：

> **當結論可以被權重的微小、且同樣合理的改動翻轉時，這個決策的真正內容不在那張表裡——在你選權重的那一刻。表只是把你已經做的選擇重新排版成數字。**

工程師對這件事特別沒有抵抗力，因為那張表看起來太像我們平常信任的東西了：有數字、有公式、有總分、能排序。它有計算的**形式**，於是我們把它當計算的**權威**。但形式不等於權威。一個能被微風吹倒的結論，不是穩健的計算，是化了妝的偏好。

## 兩顆地雷拆給你看

前面說加總法偷塞了兩個假設。現在正面拆開。

### 地雷一：偏好獨立常常不成立

加權加總有個沉默的前提叫**偏好獨立（preferential independence）**：你對某一維的偏好排序，不受其他維水準的影響。具體說——「延遲從 40ms 降到 15ms 值多少」這個判斷，應該跟「成本是 1200 還是 2000」無關。只有當所有屬性兩兩之間都滿足這種獨立，加權加總才是合法的合成方式（這是 Keeney–Raiffa 1976 給的條件，嚴格證明本書不展開，指向延伸閱讀）。

問題是真實偏好常常不獨立。舉個你熟的例子：你對「多花錢買更低延遲」的意願，**取決於你現在的流量規模**。在低流量時，延遲從 40ms 降到 15ms 你根本無感，不值得多花一毛；但在你變成高頻交易那種場景，同樣這 25ms 的改善可能值天價。也就是說，「延遲這一維值多少」**跟另一個維度（規模、或成本檔位）糾纏在一起**——這正是偏好獨立被違反。

一旦違反，加權加總就**錯了**——不是「估得不準」，是用錯了模型。正確的合成需要乘法或多線性項（讓屬性之間能交互），那比加總複雜得多。但人們幾乎總是用加總，因為加總簡單、且大多數時候錯得不明顯。本章的誠實版是：**加總是個近似，當你感覺到「這兩維其實會互相影響」時，你的加權表已經在說謊了**，只是它的謊很有禮貌、不會報錯。

### 地雷二：不是所有東西都該被補償

加總法的第二個本質假設是**補償性**：一維很爛，可以被另一維很好補回來。v(Y)=0.55 裡，Y 在成本上拿 0，但靠延遲和可靠補回來，總分照樣贏。

可是有些屬性根本不該被補償。如果「鎖定風險」對你是個 **deal-breaker**——某個方案會讓你三年內無法搬遷、而這對公司是生存問題——那它不該只是表裡權重 0.1 的一格、可以被「便宜又快」高分淹過去。它應該是一道**門檻（hard constraint）**：不過這關，直接出局，連進加權表的資格都沒有。

加權加總會悄悄把 deal-breaker 稀釋成「只是一個分數低的維度」，然後讓其他高分把它補掉，給你一個總分很漂亮、但其實踩了地雷的選項。自我察覺的徵兆：當你看到加權表選出的贏家，心裡卻有個聲音說「可是那個我們真的不能接受」——那個聲音不是不理性，是它在提醒你：你把一個本該是約束的東西，誤放進了可補償的加總裡。**正確的做法是兩段式：先用硬門檻砍掉出局者，再對活下來的選項做加權權衡。**

## 邊際替代率：權衡到底是什麼

退一步看，多屬性決策的核心動作其實只有一個詞：**權衡（trade-off）**。而權衡的精確意義，是**邊際替代率（marginal rate of substitution, MRS）**——你願意拿多少 A 去換一單位 B，且維持同樣滿意。

在加權加總的框架裡，邊際替代率藏在權重比裡。如果成本權重 0.5、延遲權重 0.3，那（在正規化之後的尺度上）你隱含地說：「一單位正規化延遲改善，值 0.3/0.5 = 0.6 單位正規化成本改善」。權重比就是你的換算匯率。

這給了一個比「亂設權重」健康得多的設權重方法：**不要直接問「成本權重該多少」，那個問題沒有錨、人只能憑感覺亂喊。改問「我願意每月多付多少錢、換 p99 延遲降 10ms？」** 後者是個你的工程直覺答得出來的具體問題——比方說「降 10ms 我願意每月多付 200 美元」。把幾組這種具體換算問完，權重就被你的真實偏好反推出來、而不是憑空填上去。Keeney–Raiffa 的整套方法論，骨子裡就是逼你用這種「具體換算」去**測量**你的權衡，而不是讓你對著「成本權重 = ?」的空格瞎填一個好看的數。

換句話說，加權表的危險不在於用權重，而在於**權重的來源**。如果權重是從「我願意拿什麼換什麼」一個個具體換算反推出來的，它承載了真實偏好；如果權重是你盯著一個總分、反覆調到「Postgres 剛好勝出」為止——那你不是在計算，你是在用數字幫一個既定結論背書。後者極其常見，常見到我認為這是整個多屬性決策方法最該被點名的失敗模式。

## 何時加權法夠用，何時該誠實承認這是價值判斷

別矯枉過正。加權加總不是壞東西，它在對的場合很有用：

- **當結論穩健時**——你把權重在所有「合理範圍」裡掃一遍，贏家都不變，那這個決策其實不依賴你權重的細節，加權表幫你確認了「怎麼權衡都選這個」，這是真正的好消息，可以放心。
- **當它逼出對話時**——表最大的價值往往不是那個總分，而是填表的過程逼團隊把「我們到底在乎什麼、各在乎多少」攤開來吵清楚。爭論「延遲權重該 0.2 還 0.3」，本質是在爭論「我們是什麼樣的團隊、要服務什麼樣的負載」——那是該吵的、有營養的爭論。
- **當屬性真的大致獨立、且都可補償時**——選辦公室的飲料機這種低風險、屬性互不糾纏的決策，加總法夠用，別過度工程。

但當結論被微風吹倒（地雷在權重）、當屬性彼此糾纏（地雷在偏好獨立）、當有 deal-breaker 被稀釋（地雷在補償性）——這時最理性的動作，弔詭地，是**承認這不是一個計算題，而是一個價值判斷**。把那張表收起來，老實說「在我們現在的處境下，我認為穩定比省錢重要，所以我選 Y，不是因為它總分高，是因為我判斷現在該優先保穩」。這句話比一個 0.55 的總分誠實得多——它把判斷放回該放的地方（人的肩上），而不是藏進一個假裝客觀的數字裡。

這正好碰到全書那道中央張力的一個側面。規範理論說：把所有屬性正確地合成一個值、選最大的——這在無限資源、偏好真的獨立、權重真的可測的理想國裡是對的。描述現實說：人會開一張表、亂設權重、被總分綁架、把 deal-breaker 淹掉。本章的收束跟 ch13 同調——有限理性的智慧，不是把加權表算得更精，而是**知道這張表在哪些情況下會騙你，然後在那些情況下把判斷的責任老實扛回來**。表是工具，不是免責聲明。

## 這一章和脊椎的關係

提醒一句，免得你以為本章離主線遠了。脊椎那個賭注（+200, 0.5; −100, 0.5）和檢查線（盛行率 1%、後驗 16.7%），報酬都是**一維**的——錢、或效用、或一個最適行動。它們乾淨，是因為被設計成乾淨，好讓我們把期望值、效用、貝氏、展望理論一套套工具講透。

但你已經感覺到了：真實的「要不要換資料庫」「要不要遷雲」從來不是一維的。本章是脊椎之外的一塊必要補丁——它說：當報酬從一個數變成一張表，前十三章那套「比大小」就需要先過一道**合成**手續，而這道手續本身藏著可以翻轉結論的自由度。ch18 收官對帳時，脊椎賭注會被五種演算法各算一遍；本章提醒你的是，那五種演算法都預設了報酬是一維的——一旦不是，合成的權重就成了第六個、且最容易被人偷偷動手腳的旋鈕。

## 直覺的陷阱

| 陷阱 | 錯誤直覺帶你下溝的那一步 | 正確版 / 自我察覺 |
|---|---|---|
| **把加權評分表當「客觀計算」** | 看到有數字、公式、總分、排序，就把它當成「數字告訴我選哪個」的權威，忘了權重是人填的。 | 權重是**價值判斷**，不是測量值；常被「先有結論再回填」。徵兆：你調權重時眼睛在看總分、調到某個方案勝出才停手——那一刻你在背書、不在計算（守 landscape T11）。 |
| **以為結論被權重微調翻轉是「巧合」或「精度問題」** | 看到 0.45/0.55 的權重把結論翻掉，想「再調準一點就好」，於是花力氣去「精算」權重。 | 它不是精度問題，是**警訊**——這個決策的真正內容不在表裡、在你選權重那一刻。對權重不敏感才代表結論穩健；敏感代表你其實還沒決定，只是讓表替你假裝決定了。 |
| **以為所有屬性都可共量、可互相補償** | 把 deal-breaker（如三年無法搬遷的鎖定）當成「只是一個低分維度」，讓它被其他高分淹過去，選出總分漂亮卻踩雷的方案。 | 有些屬性是**硬門檻**不是可補償的維度。徵兆：贏家出爐你卻想「可是那個我們真的不能接受」——那聲音在提醒你把約束誤塞進了加總。正解：先門檻篩、再加權衡（兩段式）。 |
| **以為偏好獨立理所當然** | 預設「延遲值多少」跟「成本檔位」「流量規模」無關，安心用加總。 | 偏好獨立**常被違反**——某維的價值常跟另一維的水準糾纏。一旦糾纏，加總是用錯模型、不是估不準。徵兆：你感覺「這兩維其實會互相影響」時，加權表已經在禮貌地說謊。 |
| **直接對「權重該多少」瞎填** | 盯著「成本權重 = ?」的空格憑感覺填一個好看的數，沒有錨。 | 改問**具體換算**：「願意每月多付多少換延遲降 10ms？」——這你答得出來。權重從具體權衡反推、不是憑空填（MRS = 權重比）。 |

## 紙上推演

### 推演題

**題 1（★★，[20 分鐘]）——權重穩健性。**
三個候選 A、B、C 在三個屬性上的正規化分數（皆已正規化到 0~1，越大越好）如下：

```text
            屬性1    屬性2    屬性3
   A         0.9      0.3      0.5
   B         0.5      0.8      0.4
   C         0.4      0.5      0.9
```

(a) 用權重組 W₁ =（0.5, 0.3, 0.2）算三者的加權總分，排出第一名。
(b) 用權重組 W₂ =（0.3, 0.3, 0.4）重算，排出第一名。
(c) 結論在兩組權重下是否穩健？你會怎麼向團隊說明這個結果？

**題 2（★★，[15 分鐘]）——抓出被違反的偏好獨立。**
你在選日誌儲存方案，兩個屬性是「查詢延遲」和「儲存成本」。請說明：在「資料量很小」與「資料量是 PB 級」兩種情境下，你對「願意多花多少成本換更低查詢延遲」的判斷會不會一樣？由此指出加權加總的哪個假設被違反，以及為什麼這代表「不能直接用加總」。

**題 3（★★★，[25 分鐘]）——deal-breaker 與兩段式。**
四個雲端方案在三屬性（成本、延遲、合規）上正規化分數如下。其中「合規」是硬性要求：分數低於 0.5 代表無法通過你所在產業的法遵稽核，**直接出局**。

```text
            成本    延遲    合規
   P        0.9     0.8     0.3
   Q        0.6     0.7     0.9
   R        0.8     0.4     0.8
   S        0.5     0.9     0.6
```

(a) 若無視門檻，直接用權重（0.4, 0.4, 0.2）算四者加權總分，誰贏？
(b) 改用兩段式：先砍掉合規 < 0.5 的，再對活下來的用同一組權重排序，誰贏？
(c) 比較 (a)(b) 的贏家，說明「把 deal-breaker 當成可補償的維度」會犯什麼錯。

### 推演解答

**題 1 解答。**

(a) W₁ =（0.5, 0.3, 0.2），逐個手算：

```text
  v(A) = 0.5·0.9 + 0.3·0.3 + 0.2·0.5 = 0.45 + 0.09 + 0.10 = 0.64
  v(B) = 0.5·0.5 + 0.3·0.8 + 0.2·0.4 = 0.25 + 0.24 + 0.08 = 0.57
  v(C) = 0.5·0.4 + 0.3·0.5 + 0.2·0.9 = 0.20 + 0.15 + 0.18 = 0.53

  排序：A(0.64) > B(0.57) > C(0.53)。第一名 A。
```

複核：A 的三項 0.45+0.09+0.10，0.45+0.09=0.54，+0.10=0.64 ✓。

(b) W₂ =（0.3, 0.3, 0.4），重算：

```text
  v(A) = 0.3·0.9 + 0.3·0.3 + 0.4·0.5 = 0.27 + 0.09 + 0.20 = 0.56
  v(B) = 0.3·0.5 + 0.3·0.8 + 0.4·0.4 = 0.15 + 0.24 + 0.16 = 0.55
  v(C) = 0.3·0.4 + 0.3·0.5 + 0.4·0.9 = 0.12 + 0.15 + 0.36 = 0.63

  排序：C(0.63) > A(0.56) > B(0.55)。第一名 C。
```

複核：C 在 (b) 是 0.12+0.15+0.36，0.12+0.15=0.27，+0.36=0.63 ✓。

(c) **不穩健。** 第一名從 A 翻成 C——只因為把屬性 3 的權重從 0.2 提到 0.4（而屬性 3 正是 C 的強項、A 的弱項）。兩組權重都「很合理」，沒有哪組明顯錯。對團隊該說的話不是「數字算出來是 A」或「是 C」，而是：「**這個選擇對『我們有多在乎屬性 3』極度敏感——所以真正要先吵清楚的，是屬性 3 對我們到底多重要，而不是相信任何一個總分。**」常見錯路：看到 (a) 算出 A 就宣布「數據支持選 A」，完全沒做 (b) 的敏感度檢查——這是把化了妝的偏好當成客觀結論的典型現場。

**題 2 解答。**

不一樣。資料量很小時，查詢本來就快，把延遲再壓低幾乎無感、不值得為它多花成本——你對「成本換延遲」的意願很低。資料量到 PB 級時，慢查詢可能讓一個 dashboard 卡到沒人能用，這時你願意為降低延遲付出可觀的成本——意願高得多。

也就是說，**「查詢延遲這一維值多少」取決於另一個維度（資料量規模）的水準**。這正是**偏好獨立（preferential independence）被違反**：你對延遲的偏好強度不獨立於規模。

為什麼這代表不能直接用加總：加權加總 v = w₁·v₁ + w₂·v₂ 的數學形式裡，延遲那一項的貢獻 w₂·v₂ 是**固定**的，不隨其他維度變。但你的真實偏好裡，延遲的價值**會**隨規模變。模型結構和真實偏好的結構對不上——這不是權重沒調準的問題，是加總這個**函數形式**本身裝不下「交互作用」。要正確表達，得用帶交叉項的乘法或多線性合成（本書不展開，指向延伸閱讀）。常見錯路：以為「把延遲權重調大一點」就能修——調權重只能改那固定貢獻的大小，改不了「它該隨規模浮動」這件事。

**題 3 解答。**

(a) 無視門檻，權重（0.4, 0.4, 0.2），逐個算：

```text
  v(P) = 0.4·0.9 + 0.4·0.8 + 0.2·0.3 = 0.36 + 0.32 + 0.06 = 0.74
  v(Q) = 0.4·0.6 + 0.4·0.7 + 0.2·0.9 = 0.24 + 0.28 + 0.18 = 0.70
  v(R) = 0.4·0.8 + 0.4·0.4 + 0.2·0.8 = 0.32 + 0.16 + 0.16 = 0.64
  v(S) = 0.4·0.5 + 0.4·0.9 + 0.2·0.6 = 0.20 + 0.36 + 0.12 = 0.68

  排序：P(0.74) > Q(0.70) > S(0.68) > R(0.64)。贏家 P。
```

複核：P = 0.36+0.32+0.06，0.36+0.32=0.68，+0.06=0.74 ✓。

(b) 兩段式。先砍合規 < 0.5 的：P 的合規是 0.3 < 0.5 → **P 出局**。S 的合規是 0.6 ≥ 0.5、R 是 0.8、Q 是 0.9 → 都留下。對活下來的 Q、R、S 用同一組權重排序（分數沿用上面）：

```text
  存活者：Q(0.70)、S(0.68)、R(0.64)
  排序：Q(0.70) > S(0.68) > R(0.64)。贏家 Q。
```

(c) (a) 的贏家是 **P**，(b) 的贏家是 **Q**。差別致命：P 之所以在 (a) 拿第一，靠的是成本 0.9、延遲 0.8 兩個高分，把合規那個只有 0.3 的致命傷**補**了過去——合規只占權重 0.2，被另外兩維的高分淹沒。但合規 0.3 代表 **P 根本過不了法遵稽核、不該是個選項**。加權加總把一個本該「出局」的硬約束，稀釋成了「一個低分維度」，於是選出一個總分最高、卻在現實裡完全不能用的方案。這就是把 deal-breaker 當可補償維度的錯：**可補償性是加總法的內建假設，但硬約束的本質正是「不可被補償」**。正解永遠是兩段式——先用門檻砍出局者，再對合格者加權衡。常見錯路：把合規權重調到很大（例如 0.6）想讓 P 自動落選——這偶爾湊效，但本質仍錯，因為門檻的語意是「過/不過」的二元，不是「很重要的連續維度」；用大權重去模擬硬門檻，換一組數字就會再次穿幫。

口頭回答，講不清楚代表還沒懂：

1. 為什麼「兩個選項的屬性向量」沒有天生的排序？把它壓成一個數的動作，偷偷加進了哪兩個假設？
2. 加權加總表「看起來客觀」的錯覺從哪來？為什麼「形式像計算」不等於「權威是計算」？
3. 結論被權重的微小、合理改動翻轉——為什麼這是警訊而不是「再算準一點就好」？對權重穩健代表什麼、敏感又代表什麼？
4. 「偏好獨立」是什麼意思？舉一個你工作裡它被違反的例子，並說明為什麼違反後「加總」是用錯模型、不只是估不準。

8. 什麼情況下加權加總法是夠用甚至該用的、什麼情況下最理性的動作反而是「承認這是價值判斷、不是計算」？
