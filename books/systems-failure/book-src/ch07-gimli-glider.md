# ch07 — 金姆利滑翔機：把公斤當成磅，在四萬一千呎沒了油

## 從你已知的出發

你寫過這種 bug，我幾乎可以確定。某個值在系統裡流動——一筆金額、一個時間長度、一個資料量——它本身只是個 `number`，沒人記得它的單位。某個服務以為它是毫秒，另一個服務以為它是秒；某個 API 回的是 KiB，下游當成 bytes 在算。兩邊的程式碼**各自都對**：寫毫秒的那段邏輯沒錯，寫秒的那段也沒錯。錯的是它們之間那條沒人守的縫——**對「單位」的假設不一致**。

這就是為什麼你會用帶單位的型別（branded type、`Duration`、`Bytes`）把裸 `number` 包起來，讓 `addSeconds(ms)` 在編譯期就紅起來。這也是為什麼你在 CI 上掛了那條規則：未文件化的 API 變更會讓 build 失敗——因為你知道，**契約的縫是無聲的**，它不會在執行時拋例外，它會安安靜靜算出一個「看起來很合理」的錯誤答案，然後一路流到生產環境。

現在再加一層。你之所以敢讓一個 magic number 在系統裡跑，往往是因為**有一道自動校驗在背後兜底**：dashboard 上有個 gauge 即時顯示真實值，跟你算出來的對不上就會跳警報。那道 gauge 是你的安全網。

那麼,焦慮工程師的問題來了：**當那道自動校驗剛好壞了、你被迫退回手算的那一天，會發生什麼事？**

1983 年 7 月 23 日，一架嶄新的波音 767 給了你答案。它的油量錶（一個自動 gauge）那天故障了，地勤退回到「人工量油＋手算換算」。而那個手算，用錯了密度的單位——把該用的公斤密度，算成了磅。於是這架飛機在四萬一千呎的高空，引擎一具接一具熄火。**沒有人死**——這一章後半會告訴你為什麼。但前半，我們先看那條縫怎麼裂開的。

## 冷開場：四萬一千呎，引擎一具一具安靜下來

加拿大時間晚上八點剛過，安大略上空。Air Canada 143 航班，一架編號 C-GAUN 的波音 767-200，巡航在 FL410——四萬一千呎，比一般客機的巡航高度還高。機長 Bob Pearson 與副駕 Maurice Quintal 在駕駛艙裡，後面坐著 61 位乘客。航線是蒙特婁飛艾德蒙頓。

駕駛艙先響起的，是一個沒人想在巡航時聽到的警告音——左側引擎的燃油壓力低。Pearson 和 Quintal 的第一反應很合理：大概是燃油泵的問題。他們開始翻檢查表。

然後右側引擎也報了同樣的警告。

接著——更糟——左引擎的聲音變了，然後**沉默**。它熄火了。幾分鐘後，右引擎也停了。

這時駕駛艙裡發生了一件對飛行員來說極度不對勁的事：**安靜**。767 是雙引擎飛機，兩具都停了，意味著全部主動力消失。但同時，儀表板大半暗下去了——因為飛機的主電力來自引擎驅動的發電機。Pearson 面前剩下的，是幾個靠備援供電的基本儀表。

更要命的是 767 這代飛機高度依賴**液壓**來操縱舵面。沒有引擎，液壓泵也停了。理論上，這架飛機應該已經變成一塊不聽話的鐵。

但它沒有完全失控。機腹下方,一個叫做 **RAT（Ram Air Turbine，衝壓空氣渦輪）** 的小風車，在主動力喪失的瞬間**自動**彈了出來，靠飛機前進時的氣流轉動，提供了**有限的**液壓與基本飛控（[Wikipedia: Gimli Glider](https://en.wikipedia.org/wiki/Gimli_Glider)）。這是設計者埋的一道安全網——後面我們會回來談它，因為它正是這場災難「沒變成墳場」的硬體原因之一。

此刻 Pearson 手上是一架五十多噸、沒有動力、只剩最低限度操縱能力、正在從四萬一千呎往下掉的噴射客機。他要解的第一個問題不是「怎麼救」，而是更冷的那個：**為什麼會沒油？**

## 裝填：兩道孔洞，在加油那一刻就對齊了

要懂這場災難，得先懂它**早在飛機離地前就裝填好了**。觸發（雙引擎熄火）只是最後一根稻草；真正的炸藥，是地面上兩件各自無害的小事恰好疊在一起。

### 孔洞一：油量錶（FQIS）那天是壞的

767 的油量不是靠人盯著看的。它有一套叫 **FQIS（Fuel Quantity Indicator System，燃油量指示系統）** 的電腦，讀油箱裡感測器的訊號，換算成重量，顯示在駕駛艙的油量錶上。這是**自動的 gauge**——你的 dashboard 上那個即時顯示真實值的數字。

FQIS 設計成**雙通道**冗餘。但這裡有個關鍵的、反直覺的設計性質：**只要一個通道壞了，整套 FQIS 就罷工，油量錶直接變空白**（多源一致，見 [Wikipedia](https://en.wikipedia.org/wiki/Gimli_Glider)）。冗餘本來是要讓你「壞一個還能撐」，這裡卻是「壞一個就全倒」——這不是備援，這是把兩個通道串在一起的**偽冗餘**。

事情是這樣演的。在更早的一站（艾德蒙頓），一位技師發現油量錶空白，動手排查，發現**把第二通道停掉、只跑第一通道**，油量錶反而能正常顯示。他於是把那個通道的 circuit breaker（斷路器）拉開、貼上標籤標記「不可用」，油量錶恢復了。維修紀錄上寫著一行：`SERVICE CHK – FOUND FUEL QTY IND BLANK – FUEL QTY #2 C/B PULLED & TAGGED`（[failurology / Wikipedia](https://en.wikipedia.org/wiki/Gimli_Glider)）。飛機可以**依 MEL（Minimum Equipment List，最低裝備需求清單）放行**——規則允許在一個通道不可用、且改以人工量油的條件下繼續飛。

問題出在蒙特婁。另一位維修人員想進一步查這個故障，**把第二通道重新接回去**試試。結果他還沒來得及再把它停掉，就被叫去做別的事（去量油）——留下一個致命的狀態：那個 circuit breaker **標籤還貼著（看起來像「已拉開」）、實際上卻已經被接回**。於是 FQIS 又恢復成「雙通道都在跑」的故障態——**整套油量錶再次變成空白**（[Wikipedia](https://en.wikipedia.org/wiki/Gimli_Glider)）。

停在這裡，用焦慮工程師的眼睛看一眼：這就是**組態漂移（configuration drift）**。一個 feature flag，被一個人關掉、貼上「已關閉」的標籤，又被另一個人偷偷打開，但標籤沒改。下一個來看的人，**相信標籤，而不是相信實際狀態**。你在 ch16 三哩島會看到這個形狀的極端版——一盞燈顯示「我送出了關閉指令」，而不是「閥門真的關了」。這裡是它的前傳。

無論如何，結果是：這架飛機在油量錶全空白的狀態下，準備載客起飛。油量的唯一來源，只剩下**人工**。

### 孔洞二：人工量油，用錯了密度的單位

沒有油量錶，地勤改用一種古老可靠的方法：**浮油尺（dripstick / drip）**。從機翼下方的量油管讀出油箱裡的**油位**，查表換算成**體積（公升）**。這一步沒問題——他們量出油箱裡已經有約 7,682 公升的燃油。

接下來才是縫裂開的地方。飛行計畫要求的不是「公升」，是**重量（公斤）**——飛機關心的是載重和能量，不是體積。所以要把「公升」乘上一個**密度因子**換成「公斤」。

正確的因子是約 **0.803 kg/L**（每公升噴射燃油約 0.803 公斤）。

他們用的是 **1.77**。

1.77 是什麼？是同一種燃油的密度，但單位是 **lb/L（磅／公升）**。這個數字本身**完全正確**——只是它的單位是磅，不是公斤。而這正是 F9 的教科書定義：**一個各自都對的數字，跨過一條沒人守的單位縫，就變成災難**。

為什麼會錯用磅？因為這架 767 是 Air Canada **第一架全面採用公制單位的飛機**。在它之前，整個機隊、所有人的肌肉記憶、所有的換算表，都是英制——磅。1.77 lb/L 是他們用了一輩子的數字。突然來一架要用 kg/L 的飛機，那條換算習慣沒跟著換過來。

我們把那一刀算清楚（數字依 [Wikipedia](https://en.wikipedia.org/wiki/Gimli_Glider) 與 landscape，⚠️ 不同來源體積數字略有出入，此處取其一說明機制）：

- 他們大致是把當下油箱體積乘上 1.77，得到「以為是公斤、其實是磅」的重量讀數。把缺多少油也用同一個錯因子算。
- 算下來，他們以為飛機上有**約 22,300 公斤**燃油——剛好是飛這趟航程所需要的量。
- 實際上，那 22,300 是**磅**。22,300 磅 ≈ **10,100 公斤**。
- 也就是：他們以為加滿了，**實際只裝了需求的約 45%**——不到一半（landscape；[CBC](https://www.cbc.ca/news/canada/manitoba/gimli-glider-35-years-1.4756985)）。

這裡有一個最容易被忽略、也最該讓你脊背發涼的細節——**交叉確認失效了**。Pearson 和 Quintal 不是沒覆核。他們確實算了、也對了帳。問題是：**他們兩個人用的是同一個錯誤的因子**。

> 「兩個人都算過」**不等於**「校驗是獨立的」。當兩道校驗共用同一個錯誤前提，它們不會互相抵銷，只會**互相確認**——給你一種「我們很謹慎」的虛假安全感。

這是你做 code review 時最該怕的東西。兩個人 review 同一段用了錯誤常數的程式碼，都點了 approve——不是因為他們不細心，是因為他們**共用了同一份錯誤的心智模型**。獨立性，不是「人數」給的，是「來源不同」給的。

到這裡，瑞士起司的孔洞全部就位：FQIS 失效（潛伏）→ 退回人工（放大了出錯機會）→ 用錯密度單位（縫）→ 交叉確認共用同一錯誤（最後一道防線也破）。只差飛機飛到一半，把那不存在的另一半油「燒到見底」。

## 觸發與連鎖：當「以為夠」遇上「其實只有一半」

飛機起飛，一切正常。油量錶是空白的，但飛行計畫上的數字「對得起來」，沒人覺得有問題——這正是 F9 最陰險的地方：**它在執行時不報錯**。引擎照常燒油，飛機照常爬升、巡航。系統運行在一個「看起來健康」的狀態，而真實的油量正以兩倍於預期的速度逼近零。

巡航途中，駕駛艙裡的燃油壓力警告響起。然後是第二個。然後左引擎熄火，右引擎熄火。從「一切正常」到「雙發停車」，中間沒有緩衝、沒有時間餘裕——這就是 **F1 緊耦合**的味道：那個算錯的數字，沒有任何一層緩衝把它擋下來，一路直通到「油箱見底」這個物理終點。

而此刻，這場災難開始展露它的**第二張臉**。

## 餘波：它本來可以是墳場，但它不是

接下來發生的事，是這一整本書裡少數讓人想站起來鼓掌的段落。但請注意——**它能成功，靠的不是奇蹟，是設計餘裕＋人的技藝**。這正是 ch22 韌性工程的主題：韌性不是「不會壞」，是「**安全地壞**」。

**第一層餘裕：飛機本身能滑翔。** 一架沒有動力的 767 不是石頭，它是一架（很爛的）滑翔機。767 在最佳狀態下的滑翔比（glide ratio）約 20:1——每下降 1 單位高度，能往前飛 20 單位。沒了動力、構型不理想，Pearson 實際飛出來的滑翔比大約 **12:1**（[Wikipedia](https://en.wikipedia.org/wiki/Gimli_Glider)）。但「12:1」也意味著：從四萬一千呎，他**有距離可以飛、有時間可以想**。設計者沒有承諾「引擎永遠不會停」，他們承諾的是「就算停了，飛機還能滑」——這是**留了餘裕**。

**第二層餘裕：RAT 自動展開。** 前面說過的那個小風車，在主動力喪失的剎那自動跳出來，靠氣流發電、撐住最低限度的液壓與飛控。它給的能力很有限——但「有限」遠遠好過「零」。這是**優雅降級的硬體版**：主系統垮了，自動退到一個**功能更少、但還能操縱**的退化模式，而不是直接變磚。你的多層 cache library 在做的，正是同一件事——in-memory 掛了退到 S3，S3 掛了退到 local，每一層都比「整個服務 500」好。RAT 就是飛機的「退到下一層 cache」。

**第三層、也是決定性的一層：人。** 副駕 Quintal 算出他們飛不到原定的溫尼伯機場。他想起一個地方——**金姆利（Gimli）**，曼尼托巴省一座**已除役**的前加拿大皇家空軍基地。他知道這個地方，因為他**當年曾在那裡服役受訓**（[Wikipedia](https://en.wikipedia.org/wiki/Gimli_Glider)）。這不是運氣，這是**正確的人剛好帶著正確的領域知識在現場**。

而機長 Pearson 帶著另一項罕見資產：他是個**經驗豐富的滑翔機飛行員**。滑翔機沒有引擎，靠的全是能量管理——這正是此刻最需要的技能。當飛機高度太高、會衝過跑道時，Pearson 用了一個**民航客機幾乎不會用、但滑翔機飛行員的本能**動作：**forward slip（前向側滑）**——把方向舵踩向一邊、副翼壓向另一邊，讓飛機「斜著」飛，用機身去擋風、大幅增加阻力，**陡降而不增速**（[Wikipedia](https://en.wikipedia.org/wiki/Gimli_Glider)）。一般客機飛行員的訓練裡幾乎沒有這個動作；但對滑翔機飛行員來說，這是吃飯的傢伙。

他們以約 220 節速度下滑，無動力滑翔了**約 17 分鐘**。滑翔的距離各來源說法不一（約 100 至 180 多公里都有），**保守地說，超過 150 公里**（landscape ⚠️：滑降距離各源約 100–183 km）。

最後的麻煩：金姆利那條除役跑道當天**正在辦賽車活動**——跑道周邊滿是賽車、露營車和人。而飛機沒有液壓推下並鎖定前起落架，前輪沒鎖好。Pearson 把飛機放下，後輪先觸地，前輪一接地就**塌陷、縮回輪艙**，機鼻砸向地面、刮著跑道滑行——這個「故障」反而幫了忙：機鼻刮地產生的摩擦力，幫飛機更快停了下來，在衝進人群前停住（[Wikipedia](https://en.wikipedia.org/wiki/Gimli_Glider)）。

**機上 61 名乘客、8 名機組，無人死亡。** 幾位乘客在從後門逃生滑梯撤離時受了輕傷（因為機鼻著地、機尾翹高，滑梯不夠長）。地面上沒有人受傷。

加拿大政府委任 Justice George Lockhart 主持調查委員會（Board of Inquiry, 1985）。它的結論值得你抄下來貼在牆上：責任**不在單一機組或地勤的過失**，而在 Air Canada 的**「企業與設備層面的缺失」（corporate and equipment deficiencies）**——並建議全面從英制轉向 SI 公制。換句話說：官方調查**拒絕**「都是加油員算錯」這個簡單答案，它問的是更難的那個問題——**是什麼系統條件，讓這幾個人在那一刻幾乎不可能算對？**

## 故障解剖

### 潛伏條件 vs 觸發

**潛伏條件（早就裝填好的炸藥）**：

- FQIS 雙通道的「壞一個就全倒」設計，加上蒙特婁那次「標籤說已拉開、實際已接回」的組態漂移——油量錶在起飛前就是空白的（F8）。
- 機隊剛從英制改公制，1.77 lb/L 是所有人用了一輩子的肌肉記憶，0.803 kg/L 是個沒人習慣的新數字——**換算的縫，在制度層面早就存在**（F9 的土壤）。
- 加油重量計算**沒有帶單位的系統性防呆**，整個流程依賴人腦記得「這架要用公斤」。

**觸發（最後一根稻草）**：那趟加油，用 1.77 取代了 0.803。一個各自都對、合起來致命的數字，被乘了進去。

### 起司的孔洞

| 防禦層 | 本該擋住什麼 | 為什麼沒擋住 |
|---|---|---|
| FQIS 油量錶（自動 gauge） | 即時顯示真實油量、與計算對帳 | 雙通道「壞一個全倒」＋組態漂移（標籤騙人）→ 全空白，自動校驗整個消失 |
| MEL 放行規則 | 油量錶不可用時，靠人工量油補上 | 規則本身沒錯，但它把「最後一道防線」押在「人工手算」這條更脆弱的線上（F8 帶病放行） |
| 密度換算 | 把公升正確換成公斤 | 用了 lb/L（1.77）而非 kg/L（0.803）——單位的縫（F9）|
| 機組與地勤的交叉確認 | 一個人算錯，另一個人抓出來 | 兩邊**共用同一個錯誤因子**，校驗不獨立——「都算過」≠「獨立驗證」（F3 最後一個孔洞）|

任何**一層**沒破，這趟都不會沒油：油量錶正常 → 一眼看穿；用對 0.803 → 油就夠；交叉確認真的獨立（換個人、換個方法重算）→ 抓得到。**這就是為什麼「找出那個算錯的人」是錯的問法**——你要找的是「為什麼這四層全破了」。

### 對應模式

- **F9（介面與假設的縫）**——主角。1.77 vs 0.803，兩個各自正確的數字，對「單位」的假設不一致。與 ch08（飽和值被當真高度）、ch03（凱悅走道的吊桿改動）同屬「假設的縫」家族；下一章 ch08 是同一個 F9，但**沒有餘裕兜底**，於是直接墜毀。
- **F8（潛伏條件 vs 觸發）**——FQIS 故障與「帶病放行」是早就裝填好的潛伏條件，加油那一刀只是觸發。
- **F3（起司孔洞對齊）**——四層防禦各有破洞，平時錯開，那天恰好連成一線；尤其「交叉確認共用同一錯誤」這個孔洞，最值得記。
- **F1（緊耦合）**——算錯的數字沒有任何緩衝，直通「油箱見底」。
- **正面：graceful degradation（預告 ch22）**——RAT 自動展開、飛機能滑翔、人有滑翔技藝，是這場災難「安全地壞」的三道餘裕。

### 一句話形狀

**這場災難的形狀是：一個寫錯單位的 magic number，在唯一能抓它的自動校驗（油量錶）剛好失效、且人工複核又共用同一錯誤的那一天，沒被任何一層擋下來；而它沒有變成墳場，純粹是因為系統留了滑翔的餘裕、且現場剛好有一個會用這份餘裕的人。**

## 換成你的系統

你已經在防 F9 了。你用 branded type 把 `Seconds` 和 `Milliseconds` 分開，你在 CI 上掛 OpenAPI 同步檢查，你寫 contract test 確保上下游對欄位的假設一致。這些都是在補「單位／契約的縫」。很好。

但金姆利真正要扎你的，是更深一層的兩個假設：

**假設一：「反正有 dashboard 那個 gauge 會抓到對不上的值。」** 你之所以敢讓一個算出來的數字流到下游，常常是因為背後有道自動校驗在兜底——metric、alarm、reconciliation job。問題是：**那道校驗自己掛了的那天呢？** 金姆利的油量錶就是那道 gauge，它那天是空白的。你的對帳 job 如果默默失敗了（沒跑、跑了但 swallow 了 exception、alarm 因為某次「太吵」被人調成 warning），你會不會也在毫不知情的狀況下，退回到「相信計算結果」這條更脆弱的線？**問自己：我的自動校驗，失敗時是吵的（fail loud）還是安靜的（fail silent）？它自己掛了，誰會知道？**

**假設二：「我們有兩個人 review 過／兩套系統對帳過，所以是獨立的。」** 不一定。如果兩個 reviewer 看的是同一段用了錯誤常數的程式碼、共用同一份文件裡的同一個錯誤定義；如果你的「雙寫對帳」兩邊都從同一個算錯單位的上游取數——那你有的是**兩份相同的錯誤**，不是兩道獨立的防線。FQIS 的「雙通道」也是這個教訓的硬體版：兩個通道串在一起，壞一個就全倒，那不叫冗餘。**問自己：我以為的兩道防線，是不是同一份 bug 的兩個副本？**（這個「冗餘其實是同一份 bug」的陷阱，ch22 會正面講。）

而金姆利還給你一個**正面**的自問，這在這本書裡很罕見——大部分章節只教你怕。但這一章告訴你：**怕之外，要蓋餘裕。**

> 你的系統，**輸光的時候**，是直接 500、整個倒下；還是像那架 767 一樣，退到一個「功能變少、但還能操縱」的退化模式，撐到你（或 oncall）把它降落？

你的多層 cache library 已經是這個答案的一半——它會「退到下一層」而不是「整個爆掉」。金姆利告訴你另一半：光有降級路徑還不夠，**還得有人練過怎麼在降級模式下開飛機**。Pearson 的滑翔技藝、Quintal 對那座除役機場的記憶，不是出事那天臨時長出來的。**你的團隊，在自動化全部放棄、被迫手動接管的那一刻，有沒有那份「肌肉記憶」？** 你多久沒演練過「假設 dashboard 全黑、自動 failover 失效，純手動把服務救回來」這件事了？（這個「自動化把練習機會拿走、出事時人接不住」的反面，正是 ch05 法航 447 的主題——那一場，沒有 Pearson。）

## 紙上推演

### 推演題1 [15 分鐘] ★★

把「磅當公斤」這個 F9，對應到你自己遇過或怕過的**單位／型別假設事故**。挑一個：毫秒 vs 秒、bytes vs KiB、UTC vs 本地時間、含稅 vs 未稅金額。具體寫出：(a) 哪兩段「各自都對」的程式碼／服務，對這個單位的假設不一致；(b) 為什麼它**不會在執行時報錯**，而是算出一個「看起來合理」的錯值；(c) 你會在哪一層加什麼防呆（branded type？contract test？schema 上標注單位？），讓這個縫在**編譯期或部署前**就紅起來，而不是等生產環境見底。

#### 推演解答

關鍵是辨認出 F9 的指紋：**沒有 stack trace、沒有 5xx，只有一個合理的錯誤答案**。這正是它比 NullPointerException 危險的地方——後者會吵，前者沉默。好的答案會點出：裸 `number` 是元兇，因為它把單位資訊丟在型別系統之外、只活在「大家心照不宣」的約定裡。防呆的精神是**把單位從「人腦記憶」搬進「型別系統」或「契約檢查」**——讓機器替你守那條縫，因為人腦在換新機隊、趕進度、半夜值班時，記不住。金姆利的 Lockhart 委員會建議「全面改公制」，本質就是「**消滅那條縫本身**」（讓全系統只剩一種單位），這比「叫大家小心」強得多——對應到你的世界：與其在 code review 時提醒「記得這裡是毫秒」，不如讓 `Milliseconds` 型別讓 `addSeconds(ms)` 根本編不過。

### 推演題2 [20 分鐘] ★★★

**反事實推演（雙岔）。** 分別推演兩條岔路，並比較：

(a) **如果 FQIS 油量錶正常**（自動 gauge 沒壞），這趟還會沒油嗎？
(b) **如果換算用了帶單位的型別／流程**（不可能把 lb/L 餵進需要 kg/L 的欄位），但油量錶仍然故障，這趟還會沒油嗎？

#### 推演解答

(a) **幾乎不會。** 油量錶若正常，起飛前或巡航中，駕駛艙顯示的真實油量會與計算值嚴重不符（差一倍），任何一個人瞄一眼都會發現「不對，油怎麼這麼少」。自動校驗一旦在線，這個錯誤**幾分鐘內就會被抓到**。這說明 F8 的潛伏條件（gauge 失效）才是讓 F9 得以致命的前提——**單獨的單位錯誤，通常會被自動校驗擋下；是「校驗剛好也壞了」讓它穿透**。

(b) **也不會。** 如果換算流程從根本上不允許「拿磅的密度去算公斤」——例如量油表直接以 kg/L 標注、或計算工具強制單位——那麼即使油量錶全黑、純靠人工，算出來的也是對的。油就夠，引擎不會停。

**比較的價值**：兩條岔路各自獨立地能救命，這正是 F3 的反面證明——**多層防禦的意義就在於「任何一層完好就沒事」**。災難需要它們**同時**破洞。所以「根因是哪一個」是個假問題：油量錶失效和單位錯誤，**單獨任何一個都不足以釀災**，是它們對齊了才致命。

### 推演題3 [15 分鐘] ★★

**對帳你自己的「交叉確認共用同一錯誤」。** Pearson 和 Quintal 兩人都覆核了計算，卻同錯——因為共用同一個因子。在你的工程實踐裡，找出一處**「看起來有兩道防線、實際是同一份錯誤的兩個副本」**。提示方向：兩個 reviewer 看同一段程式碼／同一份規格；雙寫對帳兩邊取自同一個上游；主備節點跑完全相同的二進位（common-mode failure）；兩個 region 部署同一份有 bug 的設定。寫出：這個「假冗餘」在什麼觸發下會兩邊同時失效？

#### 推演解答

核心洞察：**獨立性來自「來源不同」，不來自「數量不同」。** 兩份相同的東西不提供冗餘，只提供「同時失效」。常見的真實版本：(1) 主備資料庫跑同一版有 bug 的 query——主庫被打掛，備庫接手後被同樣的 query 同樣打掛（你會在 ch13 雲端連鎖反覆看到這個形狀）；(2) 兩個 region 推同一份壞設定，「多區域部署」聽起來很安全，但同步推送讓它們**同時**倒（ch10 Cloudflare、ch13 CrowdStrike 都是這個）；(3) code review 兩人都漏掉同一個錯誤常數，因為他們共用了同一份過時的文件。對抗手段：讓校驗**來源異質**——換個人、換個方法、換個資料來源重算；備援用**不同實作**（而非同一份二進位的副本）；推送**分批／canary**而非全域同步。FQIS「雙通道串在一起、壞一個全倒」就是教科書級的反例——那不是冗餘，那是把兩個點綁成一個更大的單點。

### 推演題4 [25 分鐘] ★★★

**設計題（正面，預告 ch22）。** 金姆利之所以沒變成墳場，靠三道餘裕：飛機能滑翔（設計留了能量餘裕）、RAT 自動展開（優雅降級到退化模式）、人有滑翔技藝（手動接管的肌肉記憶）。挑一個你正在維運的系統，逐一對照這三道：(a) 它有沒有「滑翔餘裕」——主路徑全斷時，有沒有一條功能更少但還能服務的退化路徑？(b) 它的「RAT」是什麼——主動力喪失時，**自動**接管的最低限度機制（fallback、降級的唯讀模式、本地快取）是什麼？(c) 它的「Pearson」在哪——當自動化全部放棄、需要人手動接管時，你的團隊練過嗎？多久沒演練？

#### 推演解答

這題沒有單一正確答案，重點是**逼自己分清這三道是不同的東西、且常常只有前兩道**。最常見的盲點是 (c)：團隊蓋了漂亮的降級路徑（多層 cache、唯讀模式），但**從沒在「假設 dashboard 全黑」的前提下演練過手動接管**——於是真出事那天，降級路徑是有的，但沒人記得怎麼開那架「滑翔機」。這正是 ch05 法航 447 的悲劇（自動化把練習拿走了），也是 ch22 韌性工程的核心主張之一：**graceful degradation 是一半，chaos drill／手動接管演練是另一半**。好的答案會老實承認「我們的 (a)(b) 有，但 (c) 從沒認真演練」——這個誠實，比假裝三道都齊全有價值。順帶一提：(a) 的「滑翔餘裕」對應到你的容量規劃裡留的 headroom——而 ch09 Sleipner A 會告訴你，當那個餘裕被一個沒人細看的模型誤差悄悄吃掉 47% 時會怎樣。
