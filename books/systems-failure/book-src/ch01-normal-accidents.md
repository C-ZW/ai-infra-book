## 從你已知的出發

你寫過 postmortem。不是那種樣板填空，是真的在事後坐下來，把一條時間線一格一格拼回去的那種。

回想其中一次。對外那欄你可能寫了一句乾淨俐落的話——「根因：某次部署引入的迴歸」。但你自己心裡清楚，那句話是給管理層看的。真正發生的事，寫在下面那長長一串 contributing factors 裡：那個迴歸之所以能上線，是因為 review 的人剛好被另一件事拉走；它之所以能在線上撐了半小時沒被發現，是因為對應的那條 alarm 三個月前因為太吵被人調成了 warning；它之所以從一個服務的 5xx 滾成跨服務的雪崩，是因為上游的 retry 沒有退避、把下游的恢復窗口踩死了；而你之所以花了四十分鐘才定位，是因為那天值班的人不是寫這段 code 的人，runbook 又過期了。

把這五件事拿掉任何一件，那次 incident 都不會發生，或者至少不會那麼痛。沒有任何一件單獨來看是「錯」的——調 alarm 的人有他的理由，那條 alarm 確實每週誤報三次;沒退避的 retry 在九成的場景下是對的;runbook 過期是因為大家都很忙。它們各自無害。災難是它們**恰好同時對齊**那一刻才誕生的。

這就是本書要講的全部。你在你那台 10K+ CCU 的系統上、在那條 K8s CronJob→SQS→冪等消費者的結算 pipeline 上、在那個你帶著全團隊用的多層 cache library 上——已經反覆活過這個形狀了。你只是還沒發現，三十年前的核電廠、四十年前的橋樑、十年前蒸發近兆美元的股市，崩壞的時候是**同一個形狀**。

本書就是這件事的放大版：把你 postmortem 裡那串 contributing factors，跨領域、跨七十年、放大來看。我不教你開飛機、不教你運轉反應爐。我要給你的是一副眼鏡，讓你下次在「應該沒事吧」出口之前，認得出那是哪一號失敗模式正在悄悄裝填。

## 一個你認得的縮影：那個「單因」其實有五層

先不講大災難。先講一個你昨天可能就見過的東西。

凌晨兩點，PagerDuty 響了。某個核心 API 的 p99 latency 開始往上爬，接著 5xx 噴出來，幾分鐘內整個服務對外不可用。值班的人爬起來看 dashboard。

第一個被找到的「原因」很乾淨：**有人在前一天傍晚部署了一個版本，那個版本的某個查詢少了一個 index，慢查詢拖垮了連線池。** 寫進 postmortem，標題欄填「根因：缺少 index 的慢查詢」。回退、加 index、結案。

但如果你真的把那一夜拆開，你會看到不是一個洞，是一整排洞剛好連成一線：

- **那個慢查詢本身**：開發在本地測試時資料量只有幾千筆，查詢毫秒回應，看起來「應該沒事」。生產有幾千萬筆。這個洞叫「測試環境與生產的假設縫」。
- **CI 沒擋下來**：你們的 CI 會跑單元測試、會強制 OpenAPI 同步，但**沒有**對 SQL 做 explain plan 的把關。這個洞平時無所謂，因為大部分 PR 不碰熱路徑查詢。
- **連線池的設定**：max connections 設成一個三年前拍板、之後沒人動過的數字。它在正常負載下綽綽有餘，所以「沒人覺得需要重看」。慢查詢一來，連線被佔住不放，池子瞬間見底。
- **重試把局部變成全域**：上游服務看到下游超時，照預設重試三次。一個請求變三個，把已經見底的連線池踩得更死。這個 retry 在下游只是「偶爾抖一下」的場景下是救命的;在下游「持續性慢」的場景下是催命的。
- **alarm 沒在該叫的時候叫**：你們其實有一條「DB 連線池使用率」的 alarm，但它的閾值設在 95%，而且兩個月前因為一次容量演練誤報過、被人臨時調高又忘了調回來。等它真的叫，服務已經掛了八分鐘。

五個洞。把任何一個補上——本地用生產量級的資料測、CI 加 explain plan 把關、連線池有合理的上限與快速失敗、retry 有退避與熔斷、alarm 閾值正確——這次 incident 就被擋在某一層，不會穿透到使用者。

「缺少 index 的慢查詢」不是根因。它只是**最後一根稻草**，是那個剛好讓五排洞對齊的觸發事件。真正的問題不在那個查詢，在於**為什麼你有五層防禦，卻每一層都沒擋住**。

記住這個縮影的形狀。本書接下來二十一章，每一章都是這個形狀的放大版——只是把「服務掛了八分鐘」換成「346 條人命」「近兆美元蒸發」「半個網際網路斷線」。機制更致命，但解剖的方法一模一樣。

## 工具一：正常意外——有些系統,出事是統計上遲早的事

1979 年三哩島核電廠那場差點熔毀的事故（見 ch16）之後，一位耶魯的社會學家 Charles Perrow 被找去參與分析。他得出一個讓整個工程界不舒服的結論，寫成了 1984 年的《Normal Accidents: Living with High-Risk Technologies》。

Perrow 的核心主張是：在某一類系統裡，嚴重事故**不是異常，而是系統的正常產物**。不是「會不會」出事，是「何時、以哪種對齊方式」出事。他甚至替這種事故取了名字——**normal accident**（正常意外），或者 **system accident**（系統性意外）。「正常」不是說它常見，而是說它是這個系統的**內建性質**，是設計本身的必然衍生物，不是哪個人犯了錯才有的反常。

哪一類系統？Perrow 給了一張二維的地圖，兩個軸：

- **互動複雜度（interactive complexity）**：子系統之間有多少非線性、非預期、設計者當初根本沒想到的交互。低互動複雜度的系統像生產線——A 接 B 接 C，看得見、想得通。高互動複雜度的系統裡，A 會以你沒料到的方式影響到 F，而且只在某個特定狀態下才會發生，平時看不出來。Perrow 的話：簡單的原理以非線性的方式組合，會長出設計者**無法預見**的新性質。
- **緊耦合（tight coupling）**：一處失效有沒有緩衝、有沒有時間餘裕。緊耦合的系統裡,一個地方壞了會立刻、直接、沒有喘息地傳導下去;鬆耦合的系統有 buffer、有 slack、有「先停下來想一下」的空間。

Perrow 的判決是：**當一個系統同時落在「高互動複雜度」與「緊耦合」的那一格，正常意外就是不可避免的。** 用他自己的話：「如果互動複雜度與緊耦合這兩個系統特性必然會產生事故，我認為我們有理由稱之為正常意外、或系統性意外。」互動複雜度是**扳機**——它製造出沒人預見的交互;緊耦合是**放大器**——它讓那個交互在你反應過來之前就傳遍全身。

```
互動複雜度：低 ──────────► 高
┌────────────┬────────────┐
│ 水壩       │ 核電廠     │ 緊
│ 鐵路       │ 化工廠     │ 耦
│            │ 航太       │ 合  ◄── 正常意外的家
├────────────┼────────────┤
│ 製造業     │ 大學       │ 鬆
│ 郵局       │ 研發機構   │ 耦
│            │            │ 合
└────────────┴────────────┘
（依 Perrow 1984 的分類示意，非原圖精確座標）
```

現在把你的後端系統放上去。一個服務、一個資料庫、幾條 cron——大概落在偏左偏鬆的位置，出事多半看得懂、修得回來。但你那台 10K+ CCU、跨服務最終一致、有 SQS 佇列與冪等消費者、有自動 scaling 與自動 failover、跑在別人機房上的分散式系統呢？每加一個「自動幫你處理」的機制、每多一條跨服務的依賴、每砍掉一層 buffer 去追延遲，你就把這個系統往**右上角**推一點。

Perrow 的洞見對你意味著什麼：你引以為傲的那些「優化」——更緊的耦合換更低的延遲、更多的自動化換更少的人工——每一筆都在把你的系統搬向正常意外的家。這不是叫你別優化，是叫你**知道自己在往哪走**。

## 工具二：瑞士起司——災難需要孔洞同時對齊

Perrow 告訴你「哪些系統注定會出事」。James Reason 告訴你「出事的時候，事情長什麼樣」。

Reason 是英國的心理學家，1990 年的《Human Error》提出了那個後來被航空、醫療、核能全部採用的比喻——**Swiss-cheese model（瑞士起司模型）**。

想像你的系統有好幾層防禦，疊在一起，像一片片直立平行擺著的瑞士起司。每一片都有洞——沒有任何一層防禦是完美的，每一層都有它的弱點與盲區。平時這些洞**錯開**，所以一個危險想穿過去，會被某一層擋住。災難發生，是當**所有層的洞，在某一刻,恰好對齊成一條直線**——Reason 稱之為「事故機會的軌跡（a trajectory of accident opportunity）」,危險一路穿過每一層的洞，直達傷害。

回頭看你那個慢查詢 incident：本地測試是一層起司（洞：資料量不對）、CI 是一層（洞：不檢查 SQL）、連線池設定是一層（洞：上限過時）、retry 策略是一層（洞:沒退避）、alarm 是一層（洞:閾值錯）。五片起司,五個洞,平時錯開,那一夜對齊。一模一樣的圖。

Reason 還做了一個對本書至關重要的區分——洞分兩種：

- **active failure（主動失效）**：操作的人在事發當下犯的錯。看得見、好歸咎、最容易被寫進報告當「根因」。
- **latent condition（潛伏條件）**：早就存在、靜靜躺在系統裡的弱點，可能是幾個月前的一個設定、一個設計決策、一個被砍掉的測試。它可以休眠數年,直到某天有人踩中。

這個區分是焦慮工程師的命門。因為事後最容易看見的是 active failure——「那個值班的人少做了一步」——但真正讓災難成為可能的，幾乎總是那些早就裝填好、沒人注意的 latent condition。把調 alarm 閾值的人揪出來罵一頓，下一次同型事故照樣會來，因為**那五個洞還在**。

本書每一章的 `## 故障解剖` 段落，都會老老實實地把該災難的每一層起司列出來，逐層問：這層為什麼沒擋住？這是本書對「找單一根因」這個壞習慣最直接的反擊。

## 工具三：Cook 的清醒——複雜系統永遠在退化模式下運行

第三件工具不是一本書，是一篇只有幾頁、卻被無數 SRE 印出來貼在牆上的短文：Richard Cook 醫師 1998 年的〈How Complex Systems Fail〉，十八條對複雜系統失效本質的觀察。Cook 是麻醉科醫師兼安全研究者，他講的本來是手術室與病人安全，但每一條都精準命中你的分散式系統。挑幾條對你最致命的：

- **第 3 條:「Catastrophe requires multiple failures – single point failures are not enough.」**（災難需要多重失效——單點失效不足以釀災。）這就是瑞士起司的另一種說法,也是本書反覆敲打的那根釘子。
- **第 4 條:「Complex systems contain changing mixtures of failures latent within them.」**（複雜系統內部，永遠藏著一組不斷變化的潛伏失效。）你的系統**此刻**就帶著一堆 latent condition 在跑——某個過時的設定、某段沒人敢動的死碼、某條被靜音的 alarm。你不知道是哪些,你只知道一定有。
- **第 5 條:「Complex systems run in degraded mode.」**（複雜系統永遠運行在退化模式下。）這條最反直覺也最重要。你的系統從來不是「全綠」的——任何時刻都有某個節點不健康、某個 retry 在重試、某個降級在生效。系統能正常服務，**不是因為它沒有故障，而是因為它帶著一堆故障還能跑**。「一切正常」是一種錯覺;真相是「故障的數量目前還沒對齊到致命」。
- **第 7 條:「Post-accident attribution to a 'root cause' is fundamentally wrong.」**（事後歸因到一個「根因」，從根本上就是錯的。）Cook 說得很硬：因為災難需要多重失效，所以**不存在**一個單一的根因。「根因」是一個**事後建構**——是我們為了讓故事有個結尾、讓焦慮有個著落，回頭挑一個因素叫它「根因」。它滿足心理需求，不反映系統真相。
- **第 8 條:「Hindsight biases post-accident assessments of human performance.」**（後見之明會扭曲事後對人的評價。）事發後你**已經知道**結果了，於是回頭看當事人，會覺得「這不是很明顯嗎，他怎麼沒看出來」。但在當下，他面對的是資訊不全、訊號淹沒在噪音裡、十幾個同樣合理的可能性。本書有一條鐵律（ch02 會展開）:在嘲笑任何當事人「怎麼這麼蠢」之前,先問——**換成我,在那個時空、那個資訊量下,真的看得出來嗎?**

把這三件工具合起來，你就有了本書的完整讀法：Perrow 告訴你**哪些系統注定會出事**（高互動複雜度 × 緊耦合）;Reason 告訴你**出事時是多層孔洞對齊**，而且要分清潛伏與主動;Cook 告訴你**別找根因，要找「為什麼每一層都沒擋住」**，並且對「都是人為疏失」這個結論永遠保持懷疑。

## 失敗模式語言：本書的貨幣

工具給了你看的方法。但你需要一套**詞彙**，才能把每一場災難拆解、命名、互相對照——才能在讀完車諾比之後，一眼認出你昨天那個 incident 跟它共用了哪幾個零件。

這套詞彙就是本書的招牌：**失敗模式語言 F1–F14**。每一章的 `## 故障解剖` 都會把該災難對應到這些編號;全書累積，到終章（ch23）組裝成一份完整目錄。這是本書的「貨幣」——一種你可以拿來跟另一個工程師交易、彼此秒懂的通用語。先給你全套概覽（完整定義見附錄 B）:

| 編號 | 失敗模式 | 一句話 |
|---|---|---|
| F1 | 緊耦合 | 一處失效沒有緩衝、沒有時間餘裕，直接傳導下去 |
| F2 | 互動複雜度 | 子系統之間有非線性、非預期、設計者沒想到的交互 |
| F3 | 起司孔洞對齊 | 多層防禦各有破洞，平時錯開，出事那天恰好連成一線 |
| F4 | 安全系統即危險源 | 用來保護的機制本身造成事故 |
| F5 | 自動化的諷刺 | 自動化拿走了操作員平時的練習與狀況意識,出事時他接不住 |
| F6 | 偏差正常化／漂移 | 僥倖一次次被當成「沒事」,安全邊界被慢慢吃掉 |
| F7 | 級聯／雪崩 | 局部失效觸發連鎖,一路放大到全域 |
| F8 | 潛伏條件 vs 觸發 | 災難早已裝填好,觸發事件只是最後一根稻草 |
| F9 | 介面與假設的縫 | 兩個各自正確的子系統,對單位／契約／前提的假設不一致 |
| F10 | 單一故障點／單一感測器 | 一個點的失效就能拖垮全局,且沒有獨立的交叉驗證 |
| F11 | 監控盲區 | 該叫的警報沒叫、或被淹沒,操作員看到的世界與真實脫節 |
| F12 | 變更即觸發 | 部署、設定更新、程式重用、旗標切換——「動了它」就是觸發 |
| F13 | 生產壓力壓過安全 | 進度／成本／發射窗口壓過了「先停下來查清楚」 |
| F14 | 規模放大／單一栽培 | 同質化＋規模讓單點失效一次打中所有人 |

前三個 F1–F3 是這套語言的脊椎，剛好對應你剛學到的三件工具:F1 緊耦合與 F2 互動複雜度是 Perrow 的兩個軸;F3 起司孔洞對齊是 Reason 的模型。其餘十一個是這兩個母模式在不同場景下長出的具體面貌。本書接下來的每一章,都會點亮其中幾個。

這套語言的價值在於**可重用**。學會它,你就不再是一場一場孤立地讀災難;你會開始看見模式在跨時代重複——塔科馬海峽橋的 flutter（ch03）跟你壓測時遇過的自激震盪是同一個 F2;車諾比那顆按下去反而引爆的 AZ-5（ch17）跟法航 447 那個推頭才響的失速警告（ch05）共用 F4;CrowdStrike 一次推全球 850 萬台（ch13）跟 Cloudflare 一條 regex 同步部署（ch10）共用 F14。**一場災難就是一組 F 編號的特定組合。** 這就是本書讓你帶走的東西。

## 故障解剖

本章沒有歷史災難可解剖（那從 ch03 開始）。所以這一段,我們把上面那個慢查詢縮影當案例,示範本書每一章 `## 故障解剖` 會怎麼進行——這也是你讀完全書後該有的反射動作。

**潛伏條件 vs 觸發。**

| | 內容 |
|---|---|
| **早已裝填的潛伏條件** | 本地測試資料量級不對（縫已存在數月）；CI 不檢查 SQL plan（設計缺口）；連線池上限是三年前的過時數字；retry 無退避策略；alarm 閾值兩個月前被臨時調高沒調回 |
| **最後那根稻草（觸發）** | 一次引入慢查詢的部署——`F12 變更即觸發`。它本身平凡無奇，致命的是它恰好讓上面五個潛伏條件對齊 |

**起司的孔洞——每一層為什麼沒擋住:**

| 防禦層 | 它的洞 | 為什麼平時沒事 |
|---|---|---|
| 本地測試 | 資料量幾千筆 vs 生產幾千萬 | 多數 PR 不碰熱路徑查詢 |
| CI 把關 | 不對 SQL 做 explain plan | OpenAPI/單元測試擋住了大部分問題,這個缺口很少被踩 |
| 連線池 | 上限過時、無快速失敗 | 正常負載遠低於上限 |
| Retry 策略 | 無退避、無熔斷 | 下游「偶爾抖一下」時,重試是救命的 |
| Alarm | 閾值被調高沒調回 | 平時不誤報就沒人想起它 |

明確反駁「單一根因」:把「慢查詢」叫做根因,等於只盯著最後一片起司上的那個洞，假裝另外四個洞不存在。如果你的修復只是「回退這次部署」,你補的是觸發,不是系統——那五個洞還在，下一個能讓它們對齊的觸發遲早會來（可能是一次流量尖峰、一次依賴升級、一次設定變更）。

**對應模式**：這個縮影命中 `F8`（潛伏條件 vs 觸發——五個洞早就裝填好）、`F3`（起司孔洞對齊——五層全沒擋住）、`F12`（變更即觸發——部署是稻草）、`F1`（緊耦合——連線池見底直接傳導、retry 無 buffer）、`F11`（監控盲區——alarm 該叫沒叫）。五個 F,一場凌晨兩點的小事故。

**一句話總結**:這場災難的形狀是——**一個平凡的變更,恰好讓五層各自無害的潛伏孔洞對齊,在緊耦合下於八分鐘內穿透到使用者,而唯一能示警的監控正好被調瞎了。**

如果你能對你自己的 incident 寫出這樣一段——而不是「根因:某次部署」——你就已經學會了這本書要教的全部。剩下的二十一章,只是把這套手法,在比你的系統更古老、更致命、更被誤解的災難上,一遍遍練到成為本能。

## 換成你的系統

把鏡頭轉回你維運的那套系統。

你那個帶著全團隊用的**多層 cache library**(in-memory → S3 → local，含 graceful degradation)——它其實是本書正面教材的雛形(ch22 會把它放進從阿波羅到今天的韌性譜系)。一層壞了退到下一層，這正是「安全地壞」。但焦慮工程師要問的是另一個問題:**你的「graceful degradation」自己有沒有 latent condition?**

點名一個你「應該沒事吧」的假設:**「快取退化路徑我寫過、上線過,所以它能運作。」**

這個假設錯的時候會怎麼壞?想想 Cook 第 5 條——複雜系統永遠在退化模式下運行。你的 S3 層那條 fallback 路徑,平時根本不會被走到(因為 in-memory 幾乎總是命中)。它是一段**幾乎從不執行的 code**。於是:

- 半年前某次重構改了 S3 client 的初始化,沒人在退化路徑上測過,那條 path 現在其實會 throw——但你不知道,因為它從沒被走到過。這是一個正在你系統裡休眠的 latent condition(F8)。
- 真正需要它的那一刻,是 in-memory 與 S3 同時出問題的時候——也就是系統已經在壓力下、最不能再出錯的時候。緊耦合(F1)讓這個失效直接傳導,而你以為的「安全網」其實破了個洞。
- 更糟:如果這條 fallback 在大流量下被同時觸發,所有實例同時湧向 local 或同時 throw,你會得到一個 thundering herd——降級機制本身變成 F7 級聯的源頭。

可操作的自問,不是「要小心」,是這個:**你最後一次真的讓 in-memory 與 S3 同時失效、驗證 fallback 確實走通,是什麼時候?** 如果答案是「上線時測過一次」或「沒有,但邏輯看起來對」——那你的安全網就是一段沒人驗證過的死碼——休眠的程式碼一旦被某次部署或某個旗標喚醒,正是本書反覆出現的 F8 潛伏條件(見 ch19 Heartbleed 潛伏兩年)。退化路徑不演練,等於沒有。

## 紙上推演

### 推演題1 **[20 分鐘]** ★★

翻出你經手過的一次真實 incident（最好是你寫過或讀過 postmortem 的那種）。**用瑞士起司把它重畫**:列出當時存在的每一層防禦(測試、review、CI、監控、限流、熔斷、容量上限、值班流程……),逐層寫下它那天的「洞」是什麼、以及它平時為什麼沒事。然後回答:如果要把這次事故擋下來,哪些洞是「補一個就夠」、哪些是「補了也只是把對齊延後」?最後,把你當初寫的「根因」那一句,跟這張起司圖對照——那句話遺漏了幾層?

#### 推演解答

重點不在你畫得多漂亮，在於你會發現兩件事。

第一:你當初寫的「根因」幾乎一定只對應到最後一片起司上的那個洞(通常是觸發,F12 變更即觸發,或某個 active failure)。其他幾層的洞——那些 latent condition——往往根本沒進到「根因」那一欄,最多被塞在「contributing factors」裡草草帶過。這正是 Cook 第 7 條:根因是事後建構,它讓報告有個乾淨的結尾,但它系統性地隱藏了真正讓災難成為可能的潛伏層。

第二:當你問「補一個夠不夠」,你會分出兩類洞。有些洞補上就真的關掉一整條軌跡(例如「連線池加上快速失敗與合理上限」,直接讓慢查詢無法拖垮整池)。有些洞補了只是把下次對齊延後(例如「這條 alarm 閾值調回去」——但下一條被靜音的 alarm 還在排隊)。後者指向一個更深的問題:你的系統有沒有一個**製造 latent condition 的機制**(例如「alarm 太吵就調高、沒人追蹤、沒人調回」這個流程本身)?能看到這一層,你就從修 bug 升級到修系統了。

### 推演題2 **[15 分鐘]** ★★

拿 Perrow 的二維分類(互動複雜度 × 緊耦合),把你**目前負責的某個系統**放上座標。先誠實定位它現在在哪一格,再做一件事:列出你過去一年對它做的三個「改進」(加了自動 failover?砍了一層 buffer 去降延遲?多接了一個跨服務依賴?),逐一判斷每個改進把這個系統往座標的哪個方向推。

#### 推演解答

幾乎所有以「效能」「自動化」「可用性」為名的改進,都是把系統往**右上角**(高互動複雜度 × 緊耦合)推。

- 加**自動 failover / 自動 scaling**:增加互動複雜度(現在有一個會自己動作的子系統,它跟主系統有你沒完全想清楚的交互——例如 failover 在某個邊界條件下會 flap),通常也增加緊耦合(自動動作沒有人類的「先停下來想一下」那段 buffer)。
- **砍 buffer 降延遲**(減小佇列、縮短 timeout、拿掉中間 cache):直接增加緊耦合——你正是在拿掉「一處失效時的時間餘裕」。
- **多接一個跨服務依賴**:增加互動複雜度與緊耦合兩者,尤其如果新依賴造成隱藏的循環依賴(見 ch12 氣電互賴、ch13 AWS status page 依賴 S3)。

Perrow 的判決不是「別做這些」——是要你**知道每一筆改進都在抬高正常意外的機率**。這給你一個務實的判準:當你往右上角推一格,你有沒有**同時**補一道對應的防禦(canary、kill switch、熔斷、退避)?如果你只顧著往右上推、不補防禦,你是在主動為下一場正常意外裝填彈藥。終章(ch23)會把這變成一份可操作的清單。

### 推演題3 **[15 分鐘]** ★★★

用你自己的話,對一個資淺工程師講清楚:**為什麼「找出那個犯錯的人」是錯的問法?** 不能只說「不要怪人,要怪系統」這種口號——要講到他真的懂為什麼這在工程上是必要的、而不只是政治正確。

#### 推演解答

一個夠格的回答至少要踩到三個點。

一、**它在事實上就是錯的**(Cook 第 3、7 條):災難需要多重失效對齊,所以根本不存在一個「那個人」的單一根因。你揪出來的那個人,只是站在最後一片起司前面、最容易看見的那個。把他換掉,其他幾層的洞還在,同型事故照樣回來。

二、**後見之明在騙你**(Cook 第 8 條):你事後覺得「這明明很明顯」,是因為你已經知道結果了。當事人在當下面對的是訊號淹沒在噪音裡、十幾個同樣合理的可能、生產壓力催著他動作。把「人為疏失」當結論,等於用你的後見之明去審判他的當下——這不公平,而且學不到東西。真正該問的是:**是什麼系統條件,讓這個人在那一刻幾乎不可能做對?**

三、**這是最關鍵的工程理由——追究個人會讓你拿不到真相**(ch02 會深入 just culture):一旦你的文化是「出事就找人背鍋」,那麼下一次出事,沒有人會告訴你他到底做了什麼、看到了什麼、為什麼那樣判斷。你會失去事故資料,而事故資料是你預防下一次同型災難的唯一原料。blameless 不是仁慈,是**為了拿到真相而付的代價**。你自己 postmortem 範本裡那個「blameless」標籤,就是這個道理——它在保護的不是當事人的面子,是你下次的 uptime。

## 自我檢核

口頭回答以下問題;講得出來才算過關,優先講「為什麼」而不是背名詞。

1. 用一句話講出 Perrow 的核心主張:在什麼條件下,嚴重事故是「統計上遲早的常態」而非異常?那兩個條件分別在做什麼(誰是扳機、誰是放大器)?
2. 把瑞士起司模型講給一個沒聽過的人:洞代表什麼?災難在什麼時刻發生?latent condition 與 active failure 的差別為什麼重要?
3. Cook 說「複雜系統永遠運行在退化模式」。這對「我的 dashboard 現在全綠,所以一切正常」這句話意味著什麼?
4. 為什麼 Cook 說「事後歸因到一個根因,從根本上就是錯的」?「根因」滿足的是什麼需求、隱藏的是什麼真相?
5. 後見之明偏誤(hindsight bias)如何讓你誤判事故中的當事人?在嘲笑任何人「怎麼這麼蠢」之前,你該先問哪一個問題?
6. F1、F2、F3 分別對應到本章的哪件工具(Perrow 的兩個軸、Reason 的模型)?為什麼說它們是失敗模式語言的脊椎?
7. 拿你那個慢查詢縮影(或你自己的 incident):講出至少三層各自獨立的防禦,以及它們各自為什麼沒擋住。為什麼「回退那次部署」不算真正的修復?
8. 你過去一年對系統做的某個「改進」,把它往 Perrow 座標的哪個方向推了?你有沒有同時補上對應的防禦?

## 延伸閱讀

- **Charles Perrow,《Normal Accidents: Living with High-Risk Technologies》(1984)**——本書一號主張的源頭。互動複雜度 × 緊耦合的二維分類、以三哩島為原型發展的「正常意外」概念都出自這裡。讀它你會永遠改變對「高風險系統」的直覺:有些事故是設計的必然,不是誰的錯。
- **James Reason,《Human Error》(1990)**——瑞士起司模型的原典,以及 latent condition / active failure 的區分。航空、醫療、核能的安全文化幾乎都建立在這本書上。讀它學會「別只看最後犯錯的那隻手,要看整排早就破掉的起司」。
- **Richard I. Cook,〈How Complex Systems Fail〉(1998,十八條)**——幾頁長、終身受用。建議印出來貼在工位上。本章引的第 3、4、5、7、8 條只是開胃,十八條每一條都精準命中分散式系統。線上有多個免費全文版本(搜尋標題即可,如 how.complexsystems.fail)。
- (預告)本書接下來:ch02 給你一整套讀事故報告的方法(根因迷思、人為疏失的陷阱、just culture、漂移到失效);ch03 證明這些失敗模式早在電腦出現之前,就在橋樑與走道上演過(塔科馬海峽橋的 flutter、凱悅走道倒塌)。
