# ch10 — Cloudflare 的一條 regex：災難性回溯如何在數分鐘內拖垮半個網路

## 從你已知的出發

先講一件你大概沒寫成 incident、但心裡一直有個底的事。

你維運過吃使用者輸入的服務。某個欄位——一個 email 驗證、一個 URL 正規化、一段 log 解析、一個從 query string 抽欄位的 middleware——背後是一條 regex。它通過了測試，通過了 code review，上線跑了好幾個月，p99 漂亮。你大概沒特別想過它，因為它「很快」——對你餵給它的那些輸入而言。

但你心裡其實知道一件事：regex 在 Node 的 V8（用的是 Irregexp 這種回溯引擎）上，**最壞情況不是你測過的那些輸入**。你可能在某次 profiling、或某篇文章裡，瞄到過「ReDoS」這三個字母，知道有一類 regex 在特定輸入下會從毫秒級炸到秒級、甚至把整個 event loop 卡死。Node 是單執行緒事件迴圈——一條 regex 在熱路徑上跑爆，卡住的不是「這個請求」，是**這個 process 上的每一個請求**。你那套招牌的多層 cache library、那條冪等的 SQS consumer，全都共用同一個被卡死的 event loop。

你大概也做過「規則／設定的全量推送」。一個 feature flag、一條新的 rate-limit 規則、一份更新的 WAF 或 schema 設定——你按下去，它在幾秒內到達所有 ECS task。你信任這個動作，因為「它只是設定，不是程式碼」。

把這兩件事擺在一起：**一條沒設防的 regex，跑在熱路徑上、吃使用者輸入**，加上**一次推全球、沒有 canary、沒有 CPU 護欄的設定部署**。這就是一顆 CPU 炸彈，同時投到每一個節點。

2019 年 7 月 2 日，Cloudflare 把這兩件事同時做到了。差別只在於，它的「每一個節點」是分佈在全球、扛著當時相當一部分網際網路流量的代理網路。一條合法的 regex，27 分鐘內讓全球流量掉了約八成。

焦慮工程師的第一信條是「『應該沒事吧』是訃聞的第一句」。這一章的「應該沒事吧」特別陰險——因為它有兩層：「這條 regex 應該不會慢吧」（它平時真的不慢），加上「設定推全球應該沒事吧」（它平時真的沒事）。兩個各自合理的假設，恰好對齊那一天。

## 冷開場：13:42 UTC，PagerDuty 開始響

2019 年 7 月 2 日，世界協調時間（UTC）13:42。

一名 Cloudflare 工程師透過自動化流程，部署了一筆 WAF（Web Application Firewall，網頁應用程式防火牆）受管規則（Managed Rule）的更新。這是再日常不過的動作——WAF 規則就是這樣維護的，為了應對新出現的攻擊手法，規則庫需要能快速更新。這次更新的目的，是加強對某一類跨站腳本（XSS）攻擊的偵測。

部署完成。沒有編譯錯誤，沒有語法錯誤，所有自動化檢查都過了。

三分鐘後，13:45，PagerDuty 開始響。Cloudflare 的合成監控（synthetic monitoring）裡，一個專門測 WAF 的測試失敗了。緊接著，全球各地的邊緣節點開始回報同一件事：**CPU 衝到接近 100%，而且下不來**。

不是某一個資料中心。是全球。

對外的症狀來得又急又全面：客戶網站開始大量回傳 **502 Bad Gateway**。Cloudflare 的前線伺服器（front-line）還活著、還能接 TCP 連線，但它們把請求往後送給負責 HTTP/HTTPS 處理的程序時，那些程序的 CPU 已經被吃光、回不了話。於是前線只能對使用者吐 502。

全球流量驟降約 **80%**（這是 Cloudflare 自己事後給的數字，照 landscape）。Discord、Shopify、Notion、Glassdoor……以及大量把 Cloudflare 擺在自己網站前面的服務，在同一瞬間一起變得不可達。對當時的網際網路來說，這看起來像半個網路斷了線。

最初幾分鐘，Cloudflare 內部的第一個假設是：**我們是不是被打了？** 一次大規模、針對性的 DDoS 攻擊，能造成的徵狀正是這樣——CPU 飽和、流量被擠掉。這個誤判很自然，後面會看到它其實偷走了寶貴的幾分鐘。

但這不是攻擊。沒有任何外部敵人。把全球 CPU 吃光的，是 Cloudflare 自己三分鐘前部署的那條規則裡，一段合法的正規表達式。

## 裝填：兩道早就被吃掉的邊界

要看懂這顆炸彈為什麼一觸即發，得先看它在 7 月 2 日之前，已經被「裝填」好了哪些潛伏條件。觸發只是最後一根稻草——災難早就準備好了（這是 F8 潛伏條件 vs 觸發，ch02 已定義過）。

**潛伏條件一：底層引擎用回溯式匹配，對指數級回溯不設防。**

Cloudflare 當時那一代 WAF 跑在一套 Lua 架構上，正則匹配用的是 PCRE（Perl Compatible Regular Expressions）這類回溯式引擎。這不是 bug，是這類引擎的設計本質——它支援反向引用（backreference）、前後查找（lookahead/lookbehind）這些「真正的正規語言」做不到的功能，代價是放棄了「保證線性時間」這件事。一條結構不良的 regex，配上對的輸入，回溯步數會爆炸。引擎本身對此**沒有任何複雜度上限**。這個性質在它被選用的那一刻就存在了，只是平時沒人去踩。

**潛伏條件二：本來該兜底的 CPU 護欄，幾週前被人移走了。**

這是本案最關鍵、也最容易被略過的一層。在 7 月 2 日之前的「數週前」，Cloudflare 對 WAF 做過一次重構（refactoring）。諷刺的是，那次重構的目的，正是**讓 WAF 用更少的 CPU**。就在那次重構裡，一個本應限制「單一 regex 能用多少 CPU」的保護機制，被**誤刪了**（landscape 與官方 postmortem 一致）。

請停下來體會這個形狀。一個用來防「某條 regex 失控吃 CPU」的安全機制，在一次以「省 CPU」為目標的重構中被移除。它被拿掉的時候,什麼事都沒發生——因為當時沒有失控的 regex。它的價值是「在未來某天有條 regex 失控時，把傷害限制在這條 regex 上」。一個只在罕見情況下才有用的防禦，被移除時的代價是零、是隱形的，直到那個罕見情況真的來了。這正是焦慮工程師最怕的那種改動：**你拿掉了一張只在最壞情況下才會展開的安全網，而最壞情況平時不來，所以沒人會在 review 時喊停。**

到 7 月 2 日早上，這套系統的狀態是：一個對回溯不設防的引擎，加上一個剛剛被拆掉 CPU 護欄的執行環境。炸藥裝好了，引信還沒點。

## 觸發：那段 `.*(?:.*=.*)`

7 月 2 日 13:42 推出的那條新規則，包含這樣一段正規表達式（照 landscape 原文）：

```
(?:(?:\"|'|\]|\}|\\|\d|(?:nan|infinity|true|false|null|undefined|
symbol|math)|\`|\-|\+)+[)]*;?((?:\s|-|~|!|{}|\|\||\+)*.*(?:.*=.*)))
```

整條看起來很嚇人，但致命的不是前面那一長串列舉。致命的是末端那一小片：

```
.*(?:.*=.*)
```

把雜訊去掉，它的骨架是 `.*.*=.*`——**三個貪婪的 `.*`，其中兩個彼此重疊地去吃同一段文字**。問題就出在這個「重疊」。

這條規則的工作，是檢查 HTTP 請求的某些欄位裡有沒有可疑內容。也就是說，它直接吃**使用者輸入**，而且跑在每一個經過 Cloudflare 的請求的熱路徑上。對大多數短輸入，它跑得飛快。但只要輸入夠長、又恰好「比不中」（沒有要找的那個結構），回溯引擎就會陷進去。

### 為什麼回溯會炸——給看過自動機的你

你對 regex 與自動機有興趣，這段值得講清楚，因為它正是「災難性回溯」與「一個普通的慢查詢」之間的本質差別。

理論上，一條「乾淨」的正規表達式（沒有反向引用那些東西）對應一個 NFA（非確定有限自動機）。NFA 可以被編譯成 DFA，或用 Thompson 的方法模擬，在 **O(n)**（n 是輸入長度）時間內判定「中或不中」——這是正規語言的美好性質，輸入再長，時間也只是線性成長。如果 Cloudflare 用的是這種引擎（像 Go 的 RE2、Rust 的 regex crate），這一天根本不會發生。

但 PCRE 這類引擎不是這樣跑的。它用的是**回溯式**模擬：遇到 `.*` 這種貪婪量詞，它先讓 `.*` 吃掉它能吃的最多字元，然後試著匹配後面的部分；如果後面匹配失敗，它就**吐回一個字元**，再試一次；再失敗，再吐一個……這個「吐回去重試」的動作，就是 backtracking。

對單一個 `.*` 來說，這頂多是線性的回退。問題出在 `.*.*` 這種**重疊**結構：要在長度 n 的字串裡決定「第一個 `.*` 吃幾個字、第二個 `.*` 吃幾個字」，組合數是 O(n²) 量級；而在這條規則裡，因為末端要找的 `=` 在「比不中」的輸入裡根本不存在，引擎必須把**每一種拆法都試到底、全部失敗**，才能宣告「比不中」。每多一個重疊的貪婪量詞，可能的拆法又多乘一層；當量詞像 `(a+)+`、`(.*)*` 這樣**巢狀**起來時，路徑數會變成真正的指數級 **O(2ⁿ)**——每多一個字元，工作量大致翻倍。

Cloudflare 在 postmortem 裡用自己的引擎做了實測，數字很有說服力（照官方 postmortem）：

| 輸入 | 比對所需步數 |
|---|---|
| `x=x`（用 `.*.*=.*` 比） | 23 步 |
| `x=` 後接 20 個 `x` | 555 步 |
| 同樣 20 個 `x`，但用帶分號的變體 `.*.*=.*;` | 5,353 步 |

注意這個爆法：輸入只長了一點，步數從 23 跳到 555、再跳到 5,353。Cloudflare 那段 `.*(?:.*=.*)` 的重疊結構，在「比不中」的長輸入上正是這種超線性（super-linear）爆炸——嚴格說它更接近多項式量級的爆炸，而非教科書 `(a+)+` 那種純指數，但對一台正在處理真實流量、輸入長度不受控的 CPU 來說，多項式爆炸已經足夠致命（這點我區分清楚、不誇大成指數）。

關鍵是這跟「一個慢查詢」的差別。慢查詢是**對所有輸入都慢一點**——你會在 p99 上看到它、在容量規劃時算進去、它隨負載線性成長。災難性回溯是**對絕大多數輸入飛快、對特定一類輸入瞬間爆炸**：它不在你的 p99 裡，不在你的壓測場景裡（除非你恰好餵了那種輸入），它平時隱形，直到真實流量裡出現一個夠長、結構恰好的字串，把單一請求的 CPU 從微秒拉到把整顆核心吃滿。它不是「比較重的工作」，是「一個輸入引爆的局部無窮迴圈」。

## 連鎖：從一個請求，到全球八成流量消失

現在把潛伏條件和觸發接起來，看孔洞怎麼一個個被穿透。

13:42，新規則經 Cloudflare 的 **Quicksilver** 推送出去。Quicksilver 是 Cloudflare 自家的全球鍵值分發系統，能在**秒級**把設定散佈到全球每一個節點——postmortem 提到它平時每秒分發約 350 筆變更、全球分發的 p99 約 2.29 秒。這是一個了不起的工程成就：它讓 Cloudflare 能在幾秒內對全球同時生效一條防禦規則。

但這次，它幾秒內對全球同時生效的，是一顆 CPU 炸彈。**規則一次推全球，沒有分批、沒有金絲雀（canary）**——WAF 規則為了能快速反應新威脅，被設計成繞過分階段發布。於是這條規則不是先到 1% 節點、跑出異常、被擋下；它是在約兩秒內，同時抵達每一個節點（這是 F14 規模放大／monoculture：同質化＋同步推送，讓單點缺陷一次打中所有人）。

每個節點開始用這條 regex 比對流經它的真實請求。真實流量裡本來就有各種長字串——只要其中一個的結構讓那段 `.*(?:.*=.*)` 陷入回溯，處理它的那個程序的 CPU 就被一個請求吃滿。請求一個接一個來，CPU 釘在 100% 下不來。

這裡是**緊耦合**（F1）登場的地方。CPU 一旦被吃光，沒有任何緩衝、沒有時間餘裕去吸收它：負責 HTTP 處理的程序回不了話 → 前線伺服器等不到回應 → 對使用者吐 502。一個內部的 CPU 病態，沒有經過任何降級層、任何 bulkhead，直接變成外部使用者眼前的錯誤頁。本來該兜底的「單一 regex CPU 上限」護欄——本案最關鍵的那道起司孔洞——幾週前已經被移走了，所以這條失控的 regex 沒有任何上限把它的傷害限制在「這一條規則」上，它直接拖垮了整個 WAF 程序，連帶拖垮了正常流量的處理。

而**監控盲區**也插了一腳：最初幾分鐘，團隊的第一假設是 DDoS 攻擊。徵狀太像了——CPU 飽和、流量被擠。這個合理但錯誤的初始判斷，讓診斷往攻擊方向走了一段，才轉回「是不是我們自己剛推的東西」。在每一秒都是流量的場景裡，這幾分鐘很貴。

## 餘波：14:07 的 kill switch

13:45 PagerDuty 響。團隊排除攻擊、鎖定 WAF 是元凶，大約在 14:00。

止血的手段是一個全域的 **WAF kill switch**（官方稱 global WAF kill switch / global terminate）。14:02，有人提出動用它；14:07，WAF 在全球被關閉。

請注意這個決定的形狀——它正是 ch22 會講的「韌性」核心手段之一（見 ch22）。kill switch 不是「修好那條 regex」，是「先把整個出問題的子系統關掉，讓流量恢復，再從容處理壞掉的東西」。它承認了一件事：在全球 CPU 燒滿、每秒損失大量流量的當下，**最快的恢復不是修復，是切斷**。能安全地關掉 WAF（即使代價是這段時間沒有 WAF 防護），意味著 WAF 與「轉發流量」這兩件事在架構上是可分離的——這是一道刻意保留的逃生口。

14:07 WAF 關閉，14:09 CPU 與流量恢復正常。從觸發到恢復，**27 分鐘**（13:42–14:09 UTC，照 landscape）。隨後團隊移除壞規則、測試，14:52 把 WAF 重新全球啟用。

事後，Cloudflare 的 CTO John Graham-Cumming 在 7 月 12 日發了一篇詳盡到罕見的公開 postmortem，逐項坦承：Quicksilver 跳過了分階段發布、先前的重構誤移除了 CPU 保護、用回溯式引擎跑會吃使用者輸入的 regex。沒有甩鍋給「那個寫 regex 的人」——因為真正的問題從來不是那條 regex，是讓那條 regex 能造成這麼大傷害的整套系統條件。

無人傷亡。這是一場純粹的可用性事故。但它的形狀，值得你記很久。

## 故障解剖

### 潛伏條件 vs 觸發

- **潛伏條件**（早就裝填好的）：(1) 底層用 PCRE 回溯式引擎，對指數／超線性回溯本質不設防；(2) 幾週前一次「省 CPU」的重構中，限制單一 regex CPU 用量的護欄被誤刪——最後的資源兜底沒了；(3) WAF 規則被設計成經 Quicksilver 一次推全球、無金絲雀；(4) WAF 規則被當「設定／資料」而非「程式碼」對待，沒套用同等的效能測試與灰度發布。
- **觸發**（最後一根稻草）：7 月 2 日 13:42 部署的一條新 WAF 規則，內含 `.*(?:.*=.*)` 這段重疊貪婪量詞的 regex，對真實流量中的長輸入產生災難性回溯。

把觸發拿掉，潛伏條件還在，等下一條結構不良的 regex；把潛伏條件補上任一道（沒移除 CPU 護欄／用 RE2 類引擎／規則先 canary），同一條 regex 也炸不了全球。**這不是「一條爛 regex」的故事，是四道孔洞恰好對齊的故事。**

### 起司的孔洞

```
請求進來 ─┬─[防線1: regex 引擎複雜度保證]── 破洞: 用回溯式 PCRE，無上限
          │
          ├─[防線2: 單一 regex CPU 護欄]──── 破洞: 幾週前重構中被誤刪 ★本案核心
          │
          ├─[防線3: 分批/金絲雀發布]──────── 破洞: Quicksilver 一次推全球，無 canary
          │
          ├─[防線4: 規則=程式碼的測試紀律]── 破洞: 當設定看待，無效能/回溯測試
          │
          └─[防線5: 緊急切斷 (kill switch)]─ 唯一沒破: 14:07 動用，27 分止血
                                              ↓
                                        全球流量恢復
```

| 防線 | 為什麼沒擋住 |
|---|---|
| 引擎本身的複雜度保證 | 選了回溯式 PCRE（為了功能性），它對 catastrophic backtracking 不設防——這是引擎類型的本質取捨，不是 bug |
| 單一 regex 的 CPU 上限 | **本案核心孔洞**：幾週前以「省 CPU」為目標的重構誤移除了它，拿掉時無害、出事時致命 |
| 分批／金絲雀發布 | WAF 規則為求快速反應威脅，刻意繞過分階段發布，經 Quicksilver 秒級推全球 |
| 把規則當程式碼測試 | 規則被視為設定／資料，未跑效能或回溯複雜度檢查，缺陷沒在上線前被攔 |
| 緊急切斷 | **唯一沒破的一層**：kill switch 存在且可用，是它把 27 分鐘止住、而非 27 小時 |

注意這張表反駁了「單一根因」的衝動：把任何單一一層補起來，這場事故都不會以這個規模發生。能問的問題不是「誰寫了爛 regex」，是「**為什麼這條 regex 能一路穿透到全球每一顆 CPU**」。

### 對應模式

- **F2（互動複雜度）**：regex 引擎的指數／超線性回溯，是「子系統內部有設計者沒充分想到的非線性行為」的純粹範例——同一條規則，對 99.99% 輸入是線性快、對特定一類輸入是爆炸。慢得非線性、非預期，這就是 F2。
- **F12（變更即觸發）**：災難在 13:42 的一次部署被點燃。「動了它」就是觸發——不是流量變了、不是攻擊來了，是推了一筆設定。
- **F14（規模放大／monoculture）**：所有節點跑同一套規則、經 Quicksilver 同步生效，同質化＋規模讓單點缺陷一次打中全球。
- **F1（緊耦合）**：CPU 一滿，沒有緩衝、沒有降級層，直接變成使用者眼前的 502。內部病態零時差傳導成外部故障。

### 一句話形狀

**這場災難的形狀是：一條合法但會災難性回溯的 regex，跑在吃使用者輸入的熱路徑上、剛被拆掉 CPU 護欄、又一次同步推送到全球每個節點——於是一顆 CPU 炸彈在同一瞬間被投到所有節點，緊耦合讓它在 27 分鐘內變成全球八成流量消失。**

順帶把三件常被混為一談的事分清楚（照 landscape，這是本章必須校正的迷思）：

- **2019-07-02（本章）**：WAF regex 災難性回溯 → 全球 CPU 100% → 27 分鐘、流量掉約 80%。根因是 **CPU／regex**。
- **2020 的 BGP 路由事件**：另一回事，根因在路由設定／網路層，不是 CPU。
- **2025-11 的設定檔事件**：又是另一回事，根因在某份設定檔。

三者都是 Cloudflare、都是「變更即觸發」家族，但根因不同、機制不同，不要當成同一件事。本章只解剖 2019-07-02 這一場。

## 換成你的系統

你活在這個形狀的正中央。

**先點名一個你「應該沒事吧」的假設**：你那些跑在 Node 服務熱路徑上、吃使用者輸入的 regex 與 parser，**最壞情況是有界的**。

這個假設錯的時候會這樣壞：你的某條 email／URL／user-agent 驗證 regex，或某個解析使用者貼上來的字串的 parser，含一個你沒注意到的重疊或巢狀量詞（`(a+)+`、`.*.*`、`(\w+\s?)*` 這類）。它通過了所有測試——因為你的測試輸入都「比得中」、或都很短。某天，一個真實使用者（或一個知道 ReDoS 的攻擊者）送來一個長度幾十、結構恰好「比不中」的字串。處理那個請求的 V8 回溯引擎陷進去，把 event loop 卡住——而 Node 是單執行緒的，卡住的不是這一個請求，是**這個 process 上的所有請求**。你的 health check 開始逾時，HPA 以為這個 task 掛了、把它換掉、把流量導到別的 task——但別的 task 也在跑同一條 regex、也會被同樣的輸入打趴。你親手做的 graceful degradation 多層 cache 在這裡幫不上忙：它降級的是「下游慢／不可用」，不是「我自己的 CPU 被自己的 regex 吃光」。

**第二個假設**：你的規則／設定變更「只是設定，不是程式碼，所以不用 canary」。Cloudflare 的 WAF 規則就是「設定」，但它驅動執行期計算、會吃 CPU。你的 rate-limit 規則、feature flag、動態下發的 schema 或 validation 規則，只要它在執行期被「跑」，它就是程式碼，就該有跟程式碼一樣的灰度發布與資源護欄。

可操作的自問，三個：

1. **我的熱路徑上，有沒有跑在使用者輸入上的 regex／parser？** 把它們列出來。對每一條，問：它有沒有重疊或巢狀量詞？我有沒有對「比不中的長輸入」做過壓測？（工具層面：用 RE2／linear-time 引擎重寫高風險的，或對單條 regex 設執行逾時／輸入長度上限——這正是 Cloudflare 被拆掉的那道護欄。）
2. **我的規則／設定變更，是不是一次推全部？** 有沒有 canary／分批，讓壞東西先在 1% 流量上現形再被擋下？
3. **我有沒有 kill switch？** 當某個子系統（WAF、某條規則、某個 feature）開始燒 CPU，我能不能**在不重新部署的前提下，幾秒內把它整個關掉**，讓主流量恢復？Cloudflare 那 27 分鐘能止住，靠的就是這個。

最後一個橋接，回到本章與 ch18 的對照（見 ch18）：Morris 蠕蟲是程式**自我繁殖**的非同步級聯——它一台感染一台，像火苗蔓延。Cloudflare 這場不是自我繁殖，是**同步推送**——一次把同樣的東西放到所有節點，像在每張桌上同時點火。兩者的共通點是 monoculture（同質化讓單一缺陷通殺），但放大機制相反：一個靠時間蔓延，一個靠空間同步。你的世界裡兩種都有——retry storm／cache 踩踏是自我繁殖型（見 ch11 的金融版正回授），而「自動更新一次推全球」是同步型（見 ch13 的 CrowdStrike）。問自己：我的哪個機制是哪一型？它有沒有對應的剎車？

## 紙上推演

### 推演題1 **[15 分鐘]** ★★

口頭（或寫在紙上）向另一個工程師解釋：為什麼「災難性回溯」是一種與「普通慢查詢」**本質不同**的失效？請務必講到：(a) 乾淨的正規式對應 NFA／可線性時間判定，回溯引擎為什麼放棄了這個保證；(b) `.*.*=.*` 這種重疊量詞在「比不中」的長輸入上為什麼步數爆炸；(c) 為什麼這種 regex 不會出現在你的 p99 或壓測裡，卻能在生產環境瞬間吃滿一顆 CPU。

#### 推演解答

(a) 一條沒有反向引用／環視的正規式，理論上對應一個 NFA，可編譯成 DFA 或用 Thompson 模擬，在 O(n) 時間內判定中或不中——輸入再長，時間只線性成長。RE2、Rust regex crate 走這條路。PCRE／PCRE2、V8 Irregexp、Python `re`、Java 預設等走**回溯式**模擬：為了支援反向引用、lookahead/lookbehind 這些超出正規語言的功能，它們放棄了線性時間保證，改用「貪婪吃滿→失敗就吐回一個字元重試」的回溯。功能換來了最壞情況的不確定性。

(b) 對單一 `.*`，回溯是線性的。但 `.*.*` 是**重疊**的兩個貪婪量詞去分配同一段文字，要試「第一個吃幾個、第二個吃幾個」的所有拆法，是 O(n²) 量級；末端要找的 `=` 在「比不中」的輸入裡不存在，所以引擎必須把**每一種拆法都試到失敗**才能宣告比不中。Cloudflare 自己的實測：`x=x` 23 步、20 個 x 變成 555 步、帶分號變體 5,353 步——輸入只長一點，步數爆增。巢狀量詞（`(a+)+`）會更糟，達 O(2ⁿ) 真指數。

(c) p99 與壓測量的是「對你餵的那些輸入」的時間。災難性回溯對絕大多數輸入飛快、只對特定結構＋夠長的「比不中」輸入爆炸。除非你的壓測恰好包含那種對抗性輸入，否則它隱形。慢查詢是對所有輸入都慢一點（會進 p99、隨負載線性成長）；災難性回溯是「一個輸入引爆的局部無窮迴圈」，平時不存在，出現一次就把一顆核心釘死。

### 推演題2 **[20 分鐘]** ★★

反事實：四道潛伏孔洞——(i) 回溯式引擎、(ii) 被誤刪的單一 regex CPU 護欄、(iii) 無金絲雀的全球推送、(iv) 規則未當程式碼測試——如果**只補回其中一道**，這場 27 分鐘全球事故會變成什麼樣子？逐一推演。

#### 推演解答

- **只補回 (ii) CPU 護欄**：那條 regex 仍會災難性回溯，但每條 regex 的 CPU 用量被限住——失控的只是「這條規則」，它被 throttle 或跳過，WAF 其餘部分與正常流量繼續跑。可能有少量誤判或 WAF 部分降級，但**不會全球流量掉 80%**。這就是為什麼 landscape 把它標為「本案核心孔洞」：它是傷害的最後兜底。
- **只補回 (iii) 金絲雀**：規則先到 1% 節點。那 1% 的 CPU 立刻飆滿、合成監控報警、自動 rollback——壞規則在到達全球前就被擋下。**爆炸半徑從全球縮成 1%。** 代價是新威脅的防禦規則生效慢幾分鐘（這正是 Cloudflare 當初繞過分階段發布的理由，是個真實的取捨）。
- **只補回 (i) 線性時間引擎（RE2 類）**：這條 regex 根本不會災難性回溯——RE2 對任何輸入都 O(n)。代價是失去反向引用／環視等功能，且這是個大工程遷移。事故直接不發生。
- **只補回 (iv) 把規則當程式碼測試**：若上線前對規則跑 ReDoS／回溯複雜度檢查（如 `redos`-detector 類工具），這條規則可能在 CI 就被擋下。但這依賴檢查工具夠完整——是機率性的防線，不如前三道那麼確定。

結論：任何**單一**一道補回，都能把「27 分鐘全球」降一個數量級。災難需要四道同時破——這就是瑞士起司。

### 推演題3 **[15 分鐘]** ★★★（對帳）

把你維運過的一次真實 incident，對應到本章的 F 模式。具體做法：找一次「一個內部資源（CPU／記憶體／連線池／event loop）被某種輸入或某次變更打爆，然後沒有緩衝地傳導成外部故障」的事件。逐項問：它的觸發是不是一次變更（F12）？它有沒有同質化＋同步放大（F14）？內部病態到外部故障之間，有沒有任何緩衝／降級層，還是緊耦合直通（F1）？那個爆掉的子系統內部，有沒有設計者沒想到的非線性行為（F2）？

#### 推演解答

（開放題，自答。）一個常見對應：某次部署引入一段 N+1 query 或一個沒有上限的 batch，在特定資料分佈下讓 RDS CPU 飆滿（呼應你那次「尖峰 RDS CPU 降 40%」的反面——這次是讓它衝上去）。對帳：觸發＝部署（F12 ✔）；若該服務所有 task 跑同一份壞碼、HPA 把流量導去的新 task 也一樣會爆（F14 ✔，同質化）；DB CPU 滿→連線池耗盡→上游逾時→使用者看到錯誤，中間若無 circuit breaker／降級就是緊耦合直通（F1）；「特定資料分佈才爆、平時沒事」就是 F2 的非線性。能把一次自己的 incident 講成「哪幾道孔洞對齊」，而不是「都是某個 PR 的錯」，這題就過了。重點不是找到完美對應，是養成用孔洞語言、而非單一根因語言複述事故的習慣。

### 推演題4 **[20 分鐘]** ★★（設計題）

你要保護一個 Node 服務的熱路徑，上面有一條吃使用者輸入的 regex。請設計**三道**防線，並說清楚每一道各擋住本章孔洞清單裡的哪一段、它的代價是什麼。

#### 推演解答

一個合理的三道防線（深度防禦，對應四道孔洞）：

1. **輸入閘門 + regex 逾時／長度上限**（擋孔洞 ii）：對使用者輸入先做長度上限與字元集檢查；對高風險 regex，在能設逾時的執行環境（如 worker thread、或用支援 timeout 的引擎）裡跑，超時即放棄並回 4xx。這直接重建 Cloudflare 被拆掉的那道「單一 regex CPU 護欄」。代價：worker thread 有 IPC 開銷、長度上限可能擋掉極少數合法長輸入。
2. **換 linear-time 引擎重寫高風險 regex**（擋孔洞 i）：把熱路徑上、結構含重疊／巢狀量詞的 regex 用 RE2（Node 有 `re2` binding）重寫，獲得 O(n) 保證。代價：失去反向引用／環視；遷移與測試工。對「只需匹配、不需那些進階功能」的驗證型 regex，這是最乾淨的解。
3. **CI 階段的 ReDoS 靜態檢查 + 金絲雀部署**（擋孔洞 iii、iv）：CI 跑 ReDoS detector 攔明顯的災難性結構；規則／設定變更走 canary，先到一小部分流量、看 CPU 指標正常再放大。代價：CI 多一道（可能有偽陽性）、部署變慢。

排序原則：(1) 是最後兜底（一定要有，因為靜態檢查不可能完整）；(2) 性價比最高（從根上消除回溯可能）；(3) 防的是「下一條還沒寫的壞 regex」與「全球同步爆」。三道分別對應「限制傷害／消除成因／攔在上線前」——任一單獨都不夠，疊起來才是瑞士起司的正面用法（見 ch22）。

## 自我檢核

口頭自答，講得出來才算過關：

1. 為什麼說「這不是一條爛 regex 的故事」？把四道潛伏孔洞講出來，並說明拿掉觸發、災難仍裝填著。
2. 「災難性回溯」與「一個普通慢查詢」的本質差別是什麼？為什麼後者進得了你的 p99、前者進不了？
3. 用自動機的語言解釋：為什麼乾淨的正規式能線性時間判定，而 PCRE 這類回溯引擎放棄了這個保證、換到了什麼？
4. `.*.*=.*` 在「比不中」的長輸入上，回溯步數為什麼爆炸？它跟 `(a+)+` 那種純指數的差別在哪？
5. 為什麼「幾週前那次重構移除 CPU 護欄」在當下看起來完全無害，卻是本案的核心孔洞？這跟焦慮工程師對「拿掉只在最壞情況才有用的安全網」的警惕有什麼關係？
6. Quicksilver 一次推全球是個工程成就，也是放大器。為什麼同一個能力既是優點又是 F14？金絲雀在這裡擋住的是哪一段？
7. kill switch 為什麼能把 27 分鐘止住、而不是修 regex？它預設了系統的什麼性質（WAF 與轉發流量可分離）？這跟 ch22 的「安全地壞」是同一件事嗎？
8. 把 2019-07-02、2020 BGP、2025 設定檔三件 Cloudflare 事故分清楚：各自的根因家族是什麼？為什麼不能當成同一件事？

- **Cloudflare 官方 postmortem**：John Graham-Cumming, *Details of the Cloudflare outage on July 2, 2019*（2019-07-12）。<https://blog.cloudflare.com/details-of-the-cloudflare-outage-on-july-2-2019/> — 本章一手來源，罕見地誠實：逐項列出 regex 原文、`x=x` 23 步 / 5,353 步的實測、Quicksilver 跳過分階段發布、CPU 護欄被誤刪。一篇教科書級的 blameless postmortem，值得當範本讀。

- **OWASP ReDoS**：*Regular expression Denial of Service - ReDoS*。<https://owasp.org/www-community/attacks/Regular_expression_Denial_of_Service_-_ReDoS> — 把這場「自己打自己」的事故，連到「攻擊者能不能故意餵這種輸入打你」的安全視角。
