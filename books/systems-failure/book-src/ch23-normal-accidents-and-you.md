# ch23 — 正常意外與你：一個工程師該怕什麼，又該怎麼睡得著

> **本章解剖什麼**：這不是第二十二場災難，是把前面二十一場拆出來的那套刀法收進刀鞘。全書的核心商品——失敗模式語言 F1–F14——在這裡組裝成完整目錄：每一個模式配上它的定義、本書哪幾場真實災難命中了它、以及它藏在你後端／分散式世界裡的樣子。然後驗收四條全書主線（正常意外、瑞士起司、漂移、安全地壞），把「焦慮」這個你帶了二十一章的東西，翻譯成星期一早上能做的具體動作。最後給你一句誠實的結語：你消滅不了正常意外，你只能讓系統安全地壞、讓自己在孔洞對齊之前看見它。讀完，回到 ch01 那次凌晨兩點的 incident——你會發現你現在能說出它的每一片起司、每一個 F 編號，而那正是這本書要給你的全部。

## 從你已知的出發

回到開頭。ch01 我把你丟回那次凌晨兩點的 incident：PagerDuty 響了，某個核心 API 的 p99 latency 往上爬，5xx 噴出來，幾分鐘內對外不可用。當時我們在 postmortem 的「根因」欄填了一句乾淨的話——「缺少 index 的慢查詢」——然後在下面那串 contributing factors 裡，誠實地寫下了真正發生的事。

現在你讀完了二十一章，我要你再讀一次那段 postmortem，但這次帶著整套詞彙。

- 本地測試用的資料量級不對，所以慢查詢沒被抓到——這道縫已經開了幾個月。`F8`，潛伏條件。
- CI 不檢查 SQL 的 explain plan——一個設計上的缺口。又一片有洞的起司。
- 連線池上限是三年前拍板、之後沒人重看的數字，正常負載下綽綽有餘，所以「沒人覺得需要動它」——`F8` 疊著 `F6`（沒人重看＝被正常化的「它一直夠用」）。
- 上游 retry 沒有退避，把已經見底的連線池踩得更死——`F1`（緊耦合：失效直接傳導、retry 無 buffer）＋ `F7`（級聯：一個服務的 5xx 滾成跨服務雪崩）。
- 那條「DB 連線池使用率」的 alarm，閾值兩個月前因為誤報被人臨時調高、忘了調回——`F11`（監控盲區：該叫的警報沒叫）。
- 觸發它的，是前一天傍晚一個引入慢查詢的部署——`F12`（變更即觸發）。它本身平凡無奇，致命的是它恰好讓上面那幾個潛伏條件對齊。

`F8 + F3 + F12 + F1 + F11`。五個洞，一場凌晨兩點的小事故。把這串編號念給你聽，你應該已經有一種熟悉到發毛的感覺——因為它的形狀，跟三哩島那盞只顯示「指令」的指示燈（見 ch16）、跟車諾比那場為趕完成而逐一停用保護的測試（見 ch17）、跟 Cloudflare 那條一次推全球的 regex（見 ch10），是**同一個形狀**。差別只在規模、領域、和死了沒死人。

ch01 我說過：如果你能對自己的 incident 寫出「五片起司、五個洞」這樣一段，而不是「根因：某次部署」，你就已經學會了這本書要教的全部。現在，這一章把那套手法收口——把散在二十一章裡的 F 編號攤成一張總表，把「會講」升級成「星期一早上會做」。

把這張表念一遍，你會看到三件事。

第一，**沒有任何一場災難只命中一個 F**。挑戰者是 `F6+F13+F3+F8`，車諾比是 `F4+F13+F6+F8+F11+F2`，Heartbleed 是 `F9+F8+F14+F3`。這就是「單一根因是錯的問法」的具體證據——每一場都是一組編號的特定組合，不是一個編號。

第二，**同一個 F 在完全不同的領域反覆出現**。`F11`（監控盲區）把三哩島那盞指示燈（見 ch16）、法航 447 那套反直覺警報（見 ch05）、和你那條被調瞎的連線池 alarm（見 ch01）串成同一張臉：你看到的世界與真實世界脫節。`F14`（規模放大）把 1988 年的 Morris 蠕蟲（見 ch18）和 2024 年 CrowdStrike 的 850 萬台藍屏（見 ch13）串成同一個機制：同質化讓一份壞東西一次打中所有人。三十六年，同一個形狀。

第三，**F4 是這張表裡最反直覺、也最該記住的一行**。安全系統即危險源——你蓋來保護的東西，本身成了直接成因。MCAS 是為了讓操縱手感「像舊 737」而生的自動修正，它親手把兩架飛機壓向地面（見 ch04）；車諾比那顆寫著「緊急停止」的 AZ-5，按下去的瞬間反而引爆爐心（見 ch17）。對照組是阿波羅 11 的重啟保護（見 ch22）——同樣是保護機制，差別只在它失效時是 fail-safe 還是 fail-deadly。你系統裡每一個「為了安全而加」的機制——failover、看門狗、circuit breaker、autoscaling——都該被問一句：它在最壞情況下的行為，是救命還是點火？

## 換成你的系統：星期一早上做什麼不同的事

二十一章的「怕」，到這裡必須變成可操作的判斷力，否則它只是焦慮。下面這份清單不是「要小心」的口號，是星期一早上你能在 IDE、在 PR、在 oncall 排班表上做的具體動作。分三層：**事前**（在災難裝填前拆掉孔洞）、**事中**（孔洞對齊時別把事情弄得更糟）、**文化**（對抗那個一直在製造潛伏條件的機制）。

### 事前——在潛伏條件裝填好之前看見它

- **畫一張你某個服務的依賴圖，然後找循環依賴（F2／F13 雲端連鎖，見 ch13）。** 重點不是你架構圖上那層——循環依賴幾乎從不在你畫的層級。把你的**災難復原路徑**當成一條獨立的依賴鏈來畫：登入要 SSO、SSO 要網路、部署要拉 image、image registry 跑在你正要救的 cluster 上、oncall 要看 runbook、runbook 在你正掛掉的 wiki 上。問每一步：它依賴的東西，會不會就是我正在搶救的那個？Meta BGP 那次（見 ch13）的環，就藏在「修復要登入、登入要網路、網路是要修的東西」。
- **盤點你的 blast radius，問「一份壞東西能一次打中多少人」（F14，見 ch13、ch18）。** 列出你執行環境裡所有會**自我更新、而你對 rollout 節奏零控制**的東西：EDR、log shipper、service mesh sidecar、base image 的自動安全更新。對能控制的，要求廠商支援分階段；對不能控制的，至少在事故簿留一行「這是一個我無法 canary 的單點」。CrowdStrike 那次（見 ch13），就是一個你信任的安全 agent 全球同步推送、一次打中 850 萬台。
- **給每一個「為了安全而加」的機制裝 kill switch（F12，見 ch10）。** 一個不需重新部署、能立刻關掉某功能的開關。Cloudflare 那條 regex（見 ch10）能在 27 分鐘止血，靠的就是全球 kill switch。但記得：kill switch 自己不能變成 SPOF，而且要**演練**——沒演練過的 kill switch，等於沒有。
- **把假設寫成 contract test 與 sanity check（F9，見 ch07、ch08、ch19）。** 帶單位的型別（金姆利的磅當公斤，見 ch07，就是少了這個）；對任何上游傳來的數值做合理性閘門（Schiaparelli 的「負高度」，見 ch08，一行「高度不可能是負的」就擋下了）；對任何「宣告長度」比對「實際 payload」（Heartbleed，見 ch19）。你那條 CI 強制 OpenAPI 同步的規則，就是 contract test 的正面教材——把同樣的紀律推到數值邊界與單位上。
- **清除死碼、給「暫時」設過期日（F8，見 ch19、ch14）。** 你系統裡此刻就帶著一堆 latent condition：過時的設定、沒人敢動的死碼、被靜音的 alarm。你不知道是哪些，你只知道一定有。定期完整重啟驗證、把「暫時調高的閾值」設一個會在 sprint 結束時提醒你的過期日——別讓「暫時」變成「正常」。

### 事中——孔洞已經對齊時，別讓自己變成第六個洞

- **寫 blameless postmortem，而且檢查你的範本本身是不是孔洞（見 ch02、ch15）。** 你的範本裡有「root cause」這一欄嗎？如果有，它每一次都在溫柔地逼你的調查收斂到單一原因——把一張被穿透的網壓扁成一條線。改成複數「contributing factors」，並強制列出**至少三層各自獨立的防禦為什麼都沒擋住**。blameless 不是寫在範本頂端那行字，是寫在當事人**相不相信誠實是安全的**這件事上——一旦追究個人，你就再也拿不到真相，下一次同型事故無法預防。
- **永遠看實際狀態，不看意圖（F11，見 ch16、ch13）。** 三哩島那盞燈（見 ch16）顯示的是「我送了關閉指令」，不是「閥門真關了」——操作員據此用完美的邏輯把爐心推向熔毀。你的 dashboard 顯示「已部署」，是真在跑（actual），還是只是「指令已送出」（desired）？你的 alarm 依賴的 metric，會不會跟被監控的服務死在同一個 region？健康訊號和健康檢查的對象，必須有獨立的命運。
- **事先定義好 override 的優先權（F5，見 ch06）。** 于伯林根（見 ch06）的核心不是「俄國機組的錯」——兩台 TCAS 都完美運作、互補地一升一降，崩的是「衝突時誰優先沒有規定」。事後 ICAO 確立 TCAS RA 優先於 ATC，消除了那個致命模糊。回頭看你系統裡：人工 override 與自動機制可能衝突的地方，優先權定義了嗎？兩套監控對同一狀態給出矛盾結論時，誰贏？這個規則必須在出事**之前**就寫好，不能在凌晨兩點臨時裁決。

### 文化——對抗那個一直在製造孔洞的機制

事前事中都是補單一的洞。但 ch01 末尾我問過一個更深的問題：你的系統有沒有一個**製造 latent condition 的機制**？「alarm 太吵就調高、沒人追蹤、沒人調回」這個流程本身，就是一台孔洞製造機。

這就是偏差正常化（F6，見 ch14、ch15）的工程版，也是全書組織線的終點。挑戰者的 O-ring 侵蝕（見 ch14）從「異常」被一次次重新定義為「可接受」；十七年後哥倫比亞（見 ch15）證明了，寫得出好的 postmortem **不等於**學到教訓——文化不變，action items 會腐爛，同型事故會精準復發。

所以文化層只有一個動作，但它最難：**定期挑戰「為什麼這還沒爆」。** 列出你團隊三個「已知問題、暫時沒爆、大家習慣了」的東西——長期 flaky test、長期高但沒爆的錯誤率、那個「下次一定修」的循環依賴。對每一個問：這是「我們評估過、接受的風險」，還是「我們只是習慣了它沒爆」？這兩者的差別，就是挑戰者與哥倫比亞之間那十七年。翻出你團隊一年前某次 postmortem 的 action items，數數幾項真落實了——這個數字，就是你文化的體溫。

## 四條全書主線：驗收

讀到這裡，把全書貫穿的四條主線各自驗收一次。你應該能用自己的話對另一個工程師講清楚每一條。

**一、正常意外（Perrow，ch01 定調）。** 在「互動複雜度高 × 緊耦合」的系統裡，嚴重事故不是異常，是統計上遲早會發生的常態——不是「有沒有」，是「何時、以哪種對齊方式」。三哩島（見 ch16）是 Perrow 的原型案例；你那台 10K+ CCU、跑著 K8s CronJob→SQS→冪等消費者結算 pipeline 的系統，互動複雜度與耦合度都不低——所以對你而言，下一次 incident 也不是「會不會」，是「何時、哪幾片起司對齊」。這不是悲觀，是把運氣換成準備。

**二、瑞士起司／多重失效合謀（Reason／Cook，ch01 定調）。** 災難需要多層防禦的孔洞同時對齊；任何一層擋住就沒事。所以「找單一根因」是錯的問法——要找的是「為什麼每一層都沒擋住」。閃電崩盤（見 ch11）到今天兇手都還在爭，這件事本身就替這條主線做了活體示範：你最想找的那個「兇手」，可能根本不存在。Cook 還補了一句你該刻在牆上的話——**複雜系統永遠運行在退化模式**：你的系統此刻就帶著一堆 latent condition 在跑，它「正常運作」只是因為那些洞還沒對齊。

**三、漂移到失效／偏差正常化（Dekker／Vaughan，ch02 定義、ch14 主打）。** 系統不是某天「砰」一下壞掉，是一次次僥倖把異常變成常態，把安全邊界慢慢吃掉，直到某天孔洞對齊。挑戰者（見 ch14）、哥倫比亞（見 ch15）、德州電網（見 ch12，2011 年就被警告過的防寒）——這條線在組織裡最頑固、也最致命。多數調查報告的最終落點，都在這條線上，不在技術 bug 上。

**四、安全地壞（焦慮工程師的信條，ch22 提出、本章收束）。** 韌性不是「不會壞」，是安全地壞——graceful degradation、blast radius 控制、bulkhead、kill switch、冗餘的「多樣性」（不是複製同一份 bug）。阿波羅 11（見 ch22）的導引電腦過載時丟掉低優先工作、保住關鍵任務，穩穩落地；金姆利滑翔機（見 ch07）靠設計餘裕加人的技藝無動力迫降成功——這兩個正面案例，跟 Schiaparelli 那個壞數字直通推進器關閉（見 ch08）的差別，只在於系統選擇了優雅降級還是緊耦合直通。你那個帶著全團隊用的多層 cache library（in-memory → S3 → local，含 graceful degradation），就是這條主線的雛形——一層壞了退到下一層。本章把它從你已經在做的一件事，升級成你看待整個系統的眼鏡。

## 跨書的兩條線

兩件事點到為止，指向你書架上的另外兩本書，不在這裡展開。

**防禦＝不變式，災難＝安全性質被違反。** 本書講的「在哪一層加哪一道防禦」，在形式化方法裡有一個更精確的名字：你加的每一道 sanity check、每一個合理性閘門、每一條「override 優先權」，本質上都是一個**不變式（invariant）**——一個你斷言系統在任何狀態下都必須維持為真的性質。而每一場災難，都是一個**安全性質（safety property）被違反**的時刻：「高度不可能是負的」（Schiaparelli，見 ch08）就是一條沒被寫下、於是沒被檢查的安全性質。想把「我覺得這裡應該不會壞」變成「我能證明這裡不會違反這條性質」，那是《把系統寫成定理：TLA+ 與形式化方法》在做的事。

**為什麼「百年一遇」會在第三年發生。** 你在德州電網那章（見 ch12）讀到「2011 年就被警告過」——那場寒潮被當成百年一遇，於是防寒被免除。但「百年一遇」是一個關於機率的陳述，而機率最容易被誤用的地方，正是把「單一事件每年 1% 機率」錯當成「一百年內只會發生一次」。當你有夠多的獨立機會、夠長的時間、夠多同質的系統，尾端的罕見事件會比你的直覺**早得多**地到來——這也正是 `F14`（規模放大）在統計上的另一面。要把這個直覺校準到能算的程度，去讀《馴服隨機：機率與統計》。

## 誠實的結語

我不會用一個漂亮的收尾騙你。

你消滅不了正常意外。只要你的系統還是「互動複雜度高 × 緊耦合」——而任何一個值得做的分散式系統都是——嚴重事故就是它的內建性質，不是可以被根除的 bug。你蓋的每一道防禦，本身都可能變成下一個 `F4`；你加的每一層自動化，都可能拿走操作員出事時最需要的那塊狀況意識（`F5`）；你做的每一次「等價」重構，都可能像凱悅那根吊桿（見 ch03）一樣悄悄改了受力結構。

你能做的，只有兩件事，而這兩件事就是這本書的全部：

**讓系統安全地壞。** 不是追求「不會壞」——那是幻覺，是「應該沒事吧」的高級版。是讓它壞的時候有降級路徑、有 blast radius 邊界、有 kill switch、有獨立命運的監控。是讓下一次 incident 停在第三片起司，而不是穿透到使用者。

**讓自己在孔洞對齊之前看見它。** 這就是 F1–F14 這套語言的全部價值——它讓你在說出「應該沒事吧」的那一刻，能立刻指認：那是哪一號模式正在裝填。是 `F8`（一個被正常化的潛伏條件）？是 `F2`（一個你沒畫出來的循環依賴）？是 `F11`（一個你監控了意圖卻沒監控現實的地方）？看見它，你就有機會在它對齊成災難之前補上一片起司。

焦慮工程師的本能——「這裡會怎麼壞」——從來不是病。它是你最值得信任的工程直覺。這本書做的，只是給那份焦慮一套詞彙、一張地圖、和二十一個跨越七十年與多個領域的對照案例，讓它從一種讓你睡不著的情緒，變成一種讓你睡得著的判斷力。

你睡得著，不是因為你相信系統不會壞。是因為你已經畫好了依賴圖、設好了 blast radius、裝好了 kill switch、寫好了 override 優先權、而且你知道——下一個孔洞正在某處慢慢對齊，但這一次，你會在它連成一線之前看見它。

下一次凌晨兩點 PagerDuty 響起來的時候，你不會再只寫「根因：某次部署」。你會寫出那五片起司、那五個洞、那一串 F 編號——然後你會去補那個真正能關掉一整條軌跡的洞，而不只是那根稻草。

那就是這本書要給你的全部。

## 紙上推演

### 推演題1 — 用 F1–F14 對你目前負責的系統做一次孔洞盤點 **[60 分鐘] ★★★**

挑一個你此刻真正負責、且出事會痛的系統（不是玩具專案）。拿 F1–F14 總表當 checklist，逐一問：「這個模式，藏在我這個系統的哪裡？」對每一個 F，寫下一句具體的話——不是「我們有緊耦合風險」這種廢話，是「結算 pipeline 的 SQS 消費者直接同步呼叫下游定價服務，沒有 buffer，下游慢 500ms 就會讓消費者堆積、觸發 visibility timeout 重投、放大成 retry storm（`F1+F7`）」這種能畫出因果鏈的話。盤完一輪，把它們依「補上之後能關掉一整條軌跡」還是「補了只是把下次對齊延後」分兩類。

#### 推演解答

這題沒有標準答案，但有及格線。**及格的盤點，每個 F 都該命中至少一處具體的、你說得出觸發條件的東西**；如果某個 F 你寫不出任何一條，有兩種可能：(a) 你的系統真的在那個維度很乾淨（少見，但可能——例如一個純無狀態、無自動化決策的服務，`F5` 可能真的空）；(b) 更常見：那個維度有洞，但你還沒學會看見它。`F2`（互動複雜度）和 `F8`（潛伏條件）是最常被漏掉的兩個，因為它們不在任何一段程式碼裡——`F2` 在互動裡，`F8` 在「沒人重看的設定」裡。

分兩類的那一步是這題的真正價值。回想 ch01 末尾：有些洞補上就真的關掉一整條軌跡（「連線池加上快速失敗與合理上限」，直接讓慢查詢無法拖垮整池）；有些洞補了只是把下次對齊延後（「這條 alarm 閾值調回去」——但下一條被靜音的 alarm 還在排隊）。前者是修系統，後者是修 bug。一個成熟的盤點，會讓你看出哪幾個洞屬於同一個「孔洞製造機」（例如「alarm 太吵就調高、沒人追蹤」這個流程），那台機器才是你該優先拆的東西。

### 推演題2 — 把全書講給一個資淺工程師聽，一場災難＝一個模式 **[45 分鐘] ★★**

不准看書、不准看這張表。對著一個假想的資淺工程師（或真的找一個），用十到十五分鐘，把這本書講完。規則：**每一場災難只准用一句話帶過，且那句話必須點出它的主打 F 模式和它在後端世界的對應**。例如:「三哩島，一盞燈只顯示『我送了指令』不顯示『閥門真關了』——這是 `F11` 監控盲區，跟你 dashboard 顯示『已部署』其實只是『指令已送出』是同一件事。」講完二十一章，最後用三句話收尾：正常意外、瑞士起司、安全地壞。

#### 推演解答

這題在逼你做一件事：**檢驗你是不是真的把這套語言內化成了「貨幣」**。如果你能流暢地一場接一場講下去，每一場都自然落到一個 F 編號和一個你自己的系統對照，那你就過關了——這正是 style-guide 開頭說的「能用自己的話對另一個工程師講清楚」。

講的過程中，你大概會發現三件事。第一，**有些災難你會卡住**——通常是因為你記得故事但沒記住它的形狀（F 編號）。那些就是你該回去重讀「對應模式」block 的章。第二，**有些 F 你會重複用很多次**——`F8`、`F3`、`F11` 幾乎場場都有，這不是你講錯，這是真的：潛伏條件、孔洞對齊、監控盲區，本來就是複雜系統失效的共通骨架。第三，**講到後端對應時最能暴露理解深度**——「跟你 dashboard 顯示『已部署』是同一件事」這種句子，講得出來才代表你真的把跨領域的形狀對上了，而不只是記住了一堆空難故事。

收尾那三句話如果講得出來且不打結，這本書對你就成立了。

### 推演題3 — 寫下你星期一要改的三件事 **[30 分鐘] ★★**

不是「以後要更小心」。是三件**具體、可在這週內動手、且你說得出它擋住哪個 F** 的事。格式強制如下，每件一行：

> 動作：＿＿＿｜擋住：F＿＿｜代價：＿＿＿｜為什麼是這三件（性價比）：＿＿＿

#### 推演解答

範例（你的會不同，但形狀該長這樣）：

> 動作：給結算 pipeline 那個直連下游定價服務的同步呼叫，加一道 circuit breaker＋降級到「先入暫存佇列、稍後補算」｜擋住：F1＋F7｜代價：多一條程式碼路徑與最終一致性窗口、要處理補算冪等（你已經有冪等消費者的肌肉，這代價對你低）｜為什麼：它把「全有或全無」變成「降級服務」，直接消除最致命的崩法，性價比最高。

> 動作：把那條兩個月前被臨時調高、忘了調回的連線池 alarm 閾值改回，並在 IaC 裡給所有「臨時調整」加一個 sprint 結束時提醒的過期標記｜擋住：F11＋F6｜代價：幾乎為零，主要是建立流程｜為什麼：它不只補一個洞，是拆掉「調高就忘」這台孔洞製造機。

> 動作：畫一次災難復原路徑的依賴鏈（登入→部署→rollback→看 runbook），標出有沒有任一步依賴正在搶救的東西｜擋住：F2 隱藏循環依賴｜代價：一個下午｜為什麼：循環依賴藏在你沒畫的層級，不畫永遠看不見，而它在真出事時最致命。

解答的重點不在內容，在**紀律**：ch22 教過你「給一個現有系統，加哪三道防線性價比最高」——graceful degradation（擋 F1）、circuit breaker（擋 F7）、kill switch（擋 F12）通常是性價比前三名，因為它們各自關掉一整條軌跡、且彼此互補。如果你寫下的三件事裡有任何一件是「以後開會多討論」「大家要更注意」這種沒有動詞、沒有 F 編號的句子——刪掉它，它是焦慮，不是判斷力。

### 推演題4 — 反事實：挑一場災難，問「如果它有阿波羅的設計哲學會怎樣」 **[20 分鐘] ★★**

從本書挑一場最讓你印象深刻的災難。問：如果它的系統是用阿波羅 11 導引電腦（見 ch22）那套「優先權排程＋過載時丟棄低優先工作、保住關鍵任務」的哲學蓋的，這場災難會不會發生？會以什麼形狀發生（或不發生）？

#### 推演解答

最乾淨的對照是 Schiaparelli（見 ch08）——ch22 已經點明它們是同一個「異常輸入」場景的正反兩面。阿波羅的 AGC 收到雷達偷走的 CPU 負載（一個異常狀況）時，它沒有當機，也沒有盲目信任、繼續積分一個荒謬的結果——它的設計讓它**重啟、丟棄低優先工作、保住關鍵導引**。Schiaparelli 的 GNC 收到一個飽和的 IMU 值（也是異常狀況）時，沒有任何一層問「這合理嗎？高度不可能是負的」，就把它當真值積分、算出負高度、據此關掉推進器。

所以反事實的答案是：如果 Schiaparelli 有阿波羅的哲學，那道「sanity check／合理性閘門」就會在壞數字與「關掉推進器」這個不可逆決策之間插入一層緩衝（解耦 `F1`）——它會偵測到「這個輸入超出物理可能範圍」，降級到一個安全模式（例如保持當前姿態、等下一個有效讀數），而不是拿一個不可能的數字去做不可逆的事。災難不會發生，或至少不會以「自由落體墜毀」的形狀發生。

這題的領悟：阿波羅和 Schiaparelli 的差別**不在硬體可靠度，在設計哲學**——一個假設「輸入可能是錯的，所以要 fail-safe」，一個假設「輸入是對的，照單全收」。你系統裡每一個信任上游數值、未做合理性檢查的地方，都站在 Schiaparelli 那一邊。把它搬到阿波羅那一邊，只需要一行檢查——但前提是你先看見那個假設。

## 自我檢核

口頭自答，講得出來才算過關。

1. 不准看表——說出 F1 到 F14 的名字，並各舉一個本書命中它的主案。卡在哪幾個？那幾個就是你還沒內化的。
2. 為什麼說「沒有任何一場災難只命中一個 F」？這件事如何證明「單一根因是錯的問法」？
3. `F4`（安全系統即危險源）為什麼是這張表裡最反直覺、也最該記住的一行？用 MCAS（見 ch04）和 AZ-5（見 ch17）對照阿波羅的重啟保護（見 ch22）來答。
4. 「事前」清單裡，為什麼「畫災難復原路徑的依賴鏈」比「畫架構圖」更可能抓到循環依賴？（提示：循環依賴藏在哪一層）
5. 為什麼「看實際狀態而非意圖」是 `F11` 的核心防禦？三哩島那盞燈（見 ch16）和你 dashboard 上的「已部署」，共用哪個謊言？
6. 「我們有 blameless postmortem 文化」這句話，可能在哪兩個地方其實是假的？（提示：範本裡的某一欄、timeline 是誰寫的）
7. 為什麼說「你消滅不了正常意外，只能讓系統安全地壞」不是悲觀，而是把運氣換成準備？
8. 四條全書主線——正常意外、瑞士起司、漂移到失效、安全地壞——你能不能各用一句話，把它對應到你自己維運過的一次真實 incident？
