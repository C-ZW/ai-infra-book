# ch19 — Heartbleed：兩年沒人發現的一個缺漏邊界檢查

> **本章解剖什麼**：OpenSSL 在實作 TLS 的「心跳」（heartbeat）擴充時，相信了攻擊者宣告的長度欄位，卻從沒去比對「你宣告要回傳這麼多，可是你實際送來的 payload 有這麼多嗎」。少了這一行邊界檢查，伺服器會把緩衝區之後緊鄰的最多約 64 KB 行程記憶體一併回傳——裡頭可能就躺著私鑰、密碼、session cookie。這個缺陷 2012 年隨 OpenSSL 1.0.1 出貨，潛伏逾兩年沒人發現（F8），影響全球大半 HTTPS 伺服器（F14），而它最深的根因不在那一行 C 程式，而在組織面：撐起全網信任地基的 OpenSSL，當年只靠約四名開發者、其中僅一人全職、年捐款約兩千美元在維護（F14 單一栽培＋資源不足）。它和 Schiaparelli（見 ch08）是同一個家族——少了一行合理性／邊界檢查，整條鏈就鎖死——只是這次沒人死、卻要全世界輪換金鑰與憑證。它也預告了 ch20：你的信任邊界，其實延伸到每一個你 import、卻沒人在顧的東西。

## 從你已知的出發

你維運的服務跑在 AWS、ECS、Lambda 上，對外的每一條連線都是 TLS。使用者打到你的 ALB、你的服務之間互打、你連 RDS 連 S3——這些「s」全靠同一套東西在撐：一個會做 handshake、驗憑證、協商金鑰、把流量加密的密碼學函式庫。十之八九，那個函式庫的根，是 OpenSSL（或它的分支 BoringSSL/LibreSSL，但血緣都從這裡來）。它是你整個服務信任鏈的地基，地基的地基。你大概從來沒讀過它一行原始碼，也不需要——它「就在那裡」，跟空氣一樣。

現在停一下，誠實回答一個問題：這塊地基，是誰在顧？

如果你的直覺是「這麼重要的東西，總有個大公司、或一個資源充足的團隊在維護吧」——記住這個直覺，因為這就是本章要拆穿的那句**「應該沒事吧」**：一個被全世界用的相依，必然被好好維護。聽起來天經地義，對吧？用的人越多，盯的眼睛越多，bug 越快被抓到，Linus 定律（"given enough eyeballs, all bugs are shallow"）不是這樣講的嗎？

2014 年 4 月，這個直覺被砸碎了。撐起全球大半 HTTPS 的 OpenSSL，當時的維護團隊是**約四個人，其中只有一個全職**，靠**一年約兩千美元**的捐款在養。一個攻擊者可控的長度欄位，少了一行邊界檢查，潛伏了兩年多沒人發現。等到有人發現，全世界才驚覺：原來我們把信任的地基，蓋在幾個志工的業餘時間上。

我們來看這道縫是怎麼開的——以及為什麼它能開兩年。

## 冷開場：2014 年 4 月 7 日，一場同時被兩邊發現的洩漏

2014 年 4 月初，兩組人馬幾乎同時、各自獨立地，盯上了 OpenSSL 裡同一段程式碼。一邊是 Google 安全團隊的 Neel Mehta，另一邊是芬蘭資安公司 Codenomicon 的研究員。Google 的人先把問題回報給 OpenSSL 專案；Codenomicon 的人則做了一件影響深遠的事——他們給這個 bug 取了名字、做了 logo、註冊了 `heartbleed.com`。

這在當時不是慣例。多數 CVE 只有一個冷冰冰的編號（這個的編號是 **CVE-2014-0160**）。但 Codenomicon 直覺到這個東西的嚴重程度需要讓非技術的人也聽得懂——於是「Heartbleed（心淌血）」這個名字、那顆滴血的紅心，幾天內傳遍全世界的新聞頭條。一般人第一次因為一個 OpenSSL bug 而被叫去改密碼。

4 月 7 日，OpenSSL 發布安全公告與修補版本 **1.0.1g**，同一天揭露漏洞。然後全世界的維運人員開始了一個漫長的週末：升級 OpenSSL、重啟服務、**重新產生私鑰、重新簽發憑證、撤銷舊憑證、強制使用者改密碼**。CloudFlare 一家就撤銷並重新簽發了**逾十萬張憑證**。

為什麼一個「讀記憶體」的 bug，要勞動全世界輪換金鑰？因為這個 bug 洩的，可能正是金鑰本身。

## 裝填：心跳是什麼，縫埋在哪

要看懂這場災難，先得知道在那行缺失的檢查存在之前，有哪些孔洞早就安靜對齊在那裡。它們每一個單獨看都不離譜，這正是 F8（潛伏條件）的特徵——災難早就裝填完畢，只等有人按下觸發。

**先講心跳本身。** TLS（還有它的 UDP 版 DTLS）有一個叫 **heartbeat** 的擴充功能（定義在 RFC 6520）。它的用途很無辜：一條加密連線如果一陣子沒流量，雙方想確認「你還在嗎」，又不想付出重新 handshake 的代價，就送一個心跳請求。協定設計成這樣：請求方送出一個 **payload**（一段任意資料）加上一個 **payload_length**（宣告「我這段 payload 有多長」），對方收到後，把同樣的 payload 原封不動回送回來。就像 ping——你丟一句話過去，對方把同一句話喊回來，證明連線還活著。

聽起來人畜無害。問題出在「對方怎麼知道要回送多少 bytes」。

**潛伏條件一：伺服器選擇相信攻擊者宣告的長度。** OpenSSL 的伺服器端收到心跳請求後，要組一個回應封包，把 payload 抄回去。它抄多少？它讀請求裡那個 `payload_length` 欄位——也就是**請求方自己宣告的數字**——然後照這個數字，從記憶體裡複製這麼多 bytes 出來回送。用 C 的話講，大致就是一個 `memcpy`，長度參數直接吃了對方給的 `payload_length`。

這是整起事件的核心縫。`payload_length` 是**攻擊者完全可控的輸入**。伺服器把它當成可信的事實，照單全收。

**潛伏條件二：沒有人比對「宣告長度」和「實際長度」。** 一個誠實的心跳請求，宣告的 `payload_length` 應該等於它實際附上的 payload 大小。但協定本身、以及 OpenSSL 的實作，都**沒有強制這兩者相符**。攻擊者大可以送一個只有 1 byte 的 payload，卻在 `payload_length` 欄位填上 **65535（約 64 KB）**。伺服器不會起疑——它只看那個宣告的數字，然後忠實地從記憶體複製 64 KB 回送。

那多出來的、攻擊者根本沒送過來的 6 萬多個 bytes，從哪來？從那個 payload 緩衝區**之後緊鄰的行程記憶體**來。這就是 **buffer over-read（緩衝區越界讀取）**——不是寫壞記憶體，是**讀了不該讀的記憶體**。而一個處理 TLS 的行程，它的記憶體裡有什麼？剛剛別的使用者送來的密碼、session cookie、HTTP 請求內容——還有，運氣夠好的話，**伺服器自己的私鑰**。

**潛伏條件三：C 的手動記憶體管理沒有任何安全網。** 這段程式碼跑在 C 裡。沒有邊界檢查、沒有記憶體安全保證、沒有 ASAN 之類的工具在 production 盯著。一個越界讀取在 C 裡是**靜默成功**的——它不會 crash，不會丟例外，不會在 log 裡留下半個字。程式高高興興地把鄰居的記憶體當成自己的 payload 回送出去，一切「正常」。

**潛伏條件四，也是最深的一條：沒人有空盯這段碼。** 這道縫不是天上掉下來的。漏洞由開發者 Robin Seggelmann 在 **2011 年 12 月 31 日**（跨年夜）提交，審查者是 OpenSSL 的核心維護者 Stephen Henson。提交經過了 code review——**然後 review 沒抓到。** 這不是因為他們蠢。是因為這是個人畜無害的新功能、在跨年夜、由一個資源極度稀薄的團隊在審。記住 review 也是一層起司，這層起司那天也有洞。

這四個孔洞平時錯開、相安無事。它們安靜地等著，等了**兩年多**。

## 觸發：其實沒有「那根稻草」——這才是最可怕的地方

前面每一章的「觸發」段，總有一根明確的稻草：一次部署、一個飽和的感測器、一通電話、一次寒潮。Heartbleed 不一樣。它的觸發段，恐怖之處在於**沒有觸發事件**。

這個漏洞 2012 年 3 月隨 OpenSSL 1.0.1 出貨。從那一刻起，它就**一直可以被觸發**。任何人，對任何一台跑著 1.0.1 到 1.0.1f 的伺服器，送一個說謊的心跳請求，就能抽走 64 KB 記憶體。不需要登入、不需要特權、不需要任何前置條件。而且——**不留任何日誌痕跡**。一個正常的心跳請求和一個惡意的心跳請求，在伺服器看來幾乎沒有區別，它不會記錄「有人讀走了我的記憶體」。

所以這兩年裡，到底有沒有人在利用它？**沒有人能確定。** 這正是「不留痕跡」最毒的地方：你無法回頭去 log 裡查「我們被讀過嗎」。揭露之後，全世界只能假設**最壞情況**——凡是跑過脆弱版本的伺服器，私鑰都可能已經外洩，所以全部都得當作已洩漏來處理。輪換金鑰不是因為「確定被偷了」，是因為**你永遠無法證明沒被偷**。

如果說有什麼算是觸發，那就是 2014 年 4 月 7 日的「揭露」本身——但揭露不是漏洞被啟用的時刻，是**漏洞終於被看見**的時刻。它已經在那裡開了兩年。

## 連鎖：從一個說謊的長度，到全世界輪換金鑰

孔洞一個接一個被穿透。注意這條鏈和前面物理災難的差別——這裡的「連鎖」不是幾秒內的級聯，是一場以「兩年潛伏＋一次揭露＋全球同時應對」展開的慢動作雪崩。

**第一步，攻擊者送出說謊的心跳。** payload 給 1 byte，`payload_length` 宣告 64 KB。

**第二步，伺服器照宣告長度 memcpy，越界讀取。** 沒有任何一行檢查擋下「宣告 64 KB、實際只有 1 byte」這個矛盾。`memcpy` 從 payload 緩衝區之後一路複製 64 KB。

**第三步，鄰近記憶體被當成 payload 回送。** 那 64 KB 裡是什麼，是運氣——可能是別人的 HTTP 請求、session cookie、明文密碼，也可能是私鑰的片段。攻擊者收下，分析，然後**再送一個心跳，再抽 64 KB**。一次抽一點點，反覆抽，像用吸管一口一口吸乾一杯水。

**第四步，私鑰真的能被拼出來——這不是理論。** 揭露後幾天，CloudFlare 公開懸賞，問「真的能靠 Heartbleed 抽到伺服器私鑰嗎」——這在當時還被一些人認為是理論上的最壞情況、未必做得到。結果：**四名研究員各自獨立地、在十小時內就把私鑰抽出來了**。手法是反覆抽記憶體、在抽到的碎片裡找出 RSA 公鑰的質因數，再用兩個質因數反推私鑰。最壞情況被證實是真的。

**第五步，blast radius 撐開到全網。** 揭露當下的測量：約 **17%**（**約五十萬台**）受信任憑證機構認證的 HTTPS 伺服器，跑著脆弱版本的 OpenSSL。OpenSSL 不是某家公司的私有產品——它是無數系統共用的同一塊地基。所以這不是「一家公司中招」，是**同一個缺陷同時躺在半個網際網路底下**。這就是 F14（單一栽培／規模放大）：同質化＋規模，讓單點缺陷一次打中所有人。

**第六步，全球進入「假設一切已洩漏」的善後。** 因為無法證明沒被讀過，正確的應對是把所有脆弱伺服器的祕密**全部當作已外洩**：升級 OpenSSL 到 1.0.1g、重新產生私鑰、重新簽發並撤銷舊憑證、要使用者改密碼。CloudFlare 一家撤換逾十萬張憑證；全世界的維運團隊那個四月都在做同一件苦工。修補本身極簡單——1.0.1g 只是加上一行檢查：**如果宣告長度對不上實際 payload，就直接忽略這個心跳。** 一行。和 bug 一樣輕——一個少了，一場災難；一個補上，全部結束。

## 餘波：地基是誰在顧

技術上的故事到 1.0.1g 就講完了。但 Heartbleed 真正的遺產，不是那一行 `memcpy`，是它逼著全世界回答冷開場那個問題：**這塊地基，是誰在顧？**

答案讓人坐立難安。OpenSSL Software Foundation 的共同創辦人 Steve Marquess 在事後透露：OpenSSL 的核心開發者**約四個人，其中只有一個是全職**；專案一年收到的捐款**約兩千美元**。撐起全球大半 HTTPS 加密的這套程式，維護它的資源，少於很多人一個月的薪水。Marquess 說，這個專案至少該有**半打全職人手**，而不是一個。Heartbleed 之後捐款湧入，但也只湊到約九千美元——對於它扛的責任，仍是杯水車薪。

這就是開源世界的**公地悲劇（tragedy of the commons）**：一個東西被所有人使用、卻不屬於任何人，於是每個人都假設「別人會顧」，結果沒人真的在付錢養。用的人成千上萬，付錢／投人力的趨近於零。Linus 定律的前提是「足夠多的眼睛」——但 Heartbleed 證明，**「被很多人用」不等於「被很多人看」**。大家都在用，幾乎沒人在讀那段 C。

這場事故催生了它最重要的制度遺產：**Core Infrastructure Initiative（CII，核心基礎設施倡議）**，2014 年 4 月底由 Linux Foundation 牽頭成立。Amazon、Cisco、Dell、Facebook、Google、IBM、Intel、Microsoft 等科技巨頭各出資（多家各約十萬美元），湊出**約四百萬美元的基金**，專門去資助那些「全世界都依賴、卻長期被忽視、資源不足」的關鍵開源專案。OpenSSL 是第一批受助對象之一——CII 出錢讓它有了**兩名全職開發者**，並資助對 OpenSSL 的安全稽核。NTP、OpenSSH 等同樣脆弱的「無名地基」也陸續進了名單。

注意這個結構：**沒有一條結論是「Seggelmann 寫錯了一行 C」。** 那行確實漏了檢查，review 確實沒抓到——但把這場災難講成「某個工程師的疏失」，就錯失了全部的教訓。真正的根因是系統性的：一個關鍵到不能再關鍵的東西，被丟給幾個志工用業餘時間維護，沒有資源做嚴格的審查、測試與稽核。CII 修的不是那行 code，是**那個讓那行 code 能潛伏兩年的組織條件**。

## 故障解剖

### 潛伏條件 vs 觸發

| | 內容 |
|---|---|
| **潛伏條件（早就裝填好）** | (1) heartbeat 實作相信攻擊者宣告的 `payload_length`、照它 memcpy；(2) 沒有任何一處比對「宣告長度 vs 實際 payload 長度」；(3) C 的手動記憶體管理＋無記憶體安全防護，使越界讀取靜默成功、不 crash 不留 log；(4) **組織面**：OpenSSL 僅約四人、一名全職、年捐款約兩千美元在維護，code review 那層起司也因此有洞——這條是本案最深的潛伏條件。 |
| **觸發（最後一根稻草）** | **沒有傳統意義的觸發事件。** 漏洞 2012-03 隨 1.0.1 出貨後就一直可被任意人遠端觸發、不留痕跡，潛伏逾兩年。唯一算得上「事件」的是 2014-04-07 的揭露——但那是它被**看見**的時刻，不是被**啟用**的時刻。 |

關鍵在於：和前幾章不同，Heartbleed 沒有「那天恰好對齊」的戲劇性瞬間。它從出貨第一天就完整裝填、隨時可觸發。它教的不是「孔洞如何在某一刻對齊」，而是**「一個沒人盯的孔洞可以開多久而沒人發現」**——以及為什麼「不留痕跡」讓你連「有沒有出事」都查不出來。

### 起司的孔洞：每一層為什麼都沒擋住

```
一個說謊的長度的旅程：從一個欄位到全球輪換金鑰
（每一段 ↓ 旁邊是「本該擋住卻沒擋住」的那層起司）

  攻擊者送心跳：payload=1 byte，宣告 payload_length=64 KB
    ↓   孔洞1：伺服器相信攻擊者宣告的長度（沒擋住）
  照宣告長度 memcpy
    ↓   孔洞2：無「宣告長度 vs 實際長度」的邊界檢查（沒擋住）
  讀到 payload 緩衝區之後的鄰近記憶體
    ↓   孔洞3：C 無記憶體安全、越界讀取靜默成功（沒擋住）
  64 KB 鄰近記憶體被當 payload 回送（含私鑰/密碼/cookie）
    ↓   孔洞4：不留日誌，無法事後得知是否被讀（沒擋住）
  反覆抽取 → 拼出私鑰
    ↓   孔洞5：同一缺陷躺在約 50 萬台伺服器底下（沒擋住）
  全球同時暴露 → 假設一切已洩漏
    ↓
  輪換金鑰、重簽憑證、改密碼（全世界的苦工）

  ── 而這一切能潛伏兩年，因為： ──
  孔洞0（最深）：關鍵基礎設施由約 4 人、1 全職、年捐 ~$2,000 維護
                → code review 這層起司本身就缺人缺錢
```

| 防禦層 | 它本該擋住什麼 | 為什麼沒擋住 |
|---|---|---|
| 輸入驗證（邊界檢查） | 拒絕「宣告長度 ≠ 實際 payload」的矛盾請求 | **這層從來不存在**——直接信任攻擊者宣告的 `payload_length` 去 memcpy |
| 記憶體安全 | 讓越界讀取 crash/報錯而非靜默成功 | C 手動記憶體管理、無 ASAN/邊界保護，越界讀取無聲通過 |
| Code review | 在合併前抓到缺失的長度比對 | review 有做，但團隊資源極稀薄、新功能看似無害，那天這層起司有洞 |
| 可觀測性／稽核 | 偵測到異常的記憶體讀取行為 | 漏洞不留日誌痕跡，事後連「是否被利用」都無法查證 |
| 多樣性（避免單一栽培） | 讓單一缺陷不會一次打中所有人 | OpenSSL 是全網共用的同一塊地基，約 50 萬台同時脆弱（F14） |
| 組織資源 | 投入足夠人力做嚴格審查、測試、稽核 | 全球依賴、年捐約兩千美元、僅一名全職維護者（公地悲劇） |

明確反駁「單一根因」：你可以說根因是「少了一行邊界檢查」——但那行少了兩年沒人補，是因為沒人有資源去補、去審、去測。你可以說是「Seggelmann 寫錯了」——但 review 通過了、出貨了、跑了兩年，這不是一個人的失誤，是一整排防禦同時稀薄。真相是這幾層**每一層都有自己沒擋住的理由**，而最底下那層（孔洞0：沒人有空有錢顧）讓上面所有層都先天孱弱。補上任何一層——一行檢查、用記憶體安全語言、足夠的稽核資源——這個 bug 都不會活兩年。

### 對應的失敗模式

- **F9（介面與假設的縫）**：核心。heartbeat 協定的契約假設「宣告長度 = 實際 payload 長度」，但實作從不強制這個假設。攻擊者宣告的長度與實際送來的資料之間那道沒人守的縫，正是這場災難的入口。和 Schiaparelli 把飽和值當真（見 ch08）、和金姆利把磅當成公斤（見 ch07）是同一個病：兩邊對「這個數字代表什麼」的假設不一致，而中間沒有一層去裁決。
- **F8（潛伏條件 vs 觸發）**：缺失的邊界檢查、靜默的越界讀取、稀薄的 review——三個孔洞 2012 年就裝填好。沒有戲劇性觸發，它從出貨日就隨時可被啟用，潛伏逾兩年。本案是 F8 的極端形式：**潛伏期由「沒人盯」直接決定。**
- **F14（規模放大／單一栽培）**：核心。OpenSSL 的無所不在，讓一個缺陷一次躺進約 50 萬台伺服器底下。同質化＋規模＝單點打中所有人。這也是「孔洞0」的另一面——正因為大家都用同一塊地基，「該由誰維護」反而沒人扛。
- **F3（起司孔洞對齊）**：輸入驗證、記憶體安全、code review、稽核——這幾層防禦各有洞，平時靠「大概沒人會利用」僥倖錯開，揭露那天全部連成一線。沒有任何單一一層失效足以致災；是它們一起。

### 一句話形狀

**這場災難的形狀是：一個全世界都當成空氣在用、卻只有幾個志工用業餘時間在顧的信任地基，因為少了一行「你宣告的長度跟你實際送的對得上嗎」的邊界檢查，讓任何人都能無聲地一口一口吸走伺服器的記憶體——包括金鑰本身——而這個缺口能開兩年沒人發現，正是因為「大家都在用」從來不等於「有人在看」。**

## 換成你的系統

兩個層次，兩個你「應該沒事吧」的假設，都點名給你。

**第一層，技術的：你信任了上游宣告的長度／大小／數量，卻沒比對它跟實際內容。** 這個形狀你比想像中更熟。你解析一個 protobuf／一個自訂 binary 協定／一個 multipart upload，header 裡寫著「接下來這段是 N bytes」，你照 N 去讀／去配置緩衝區——你有沒有比對「實際真的有 N bytes 嗎」？你從 SQS 收一則訊息，裡頭一個欄位宣告「這個 batch 有 1000 筆」，你照 1000 去跑迴圈——實際陣列真的有 1000 筆，還是只有 3 筆而你讀過了界？Heartbleed 的核心不是「TLS 很難」，是**「相信對方宣告的 metadata，而不去對照實際的 payload」**——這是個跨語言、跨協定的反模式，和 Schiaparelli 把飽和值當真（見 ch08）、本書反覆出現的「缺一個 sanity／bounds check」是同一張臉。你天天在做的事裡，有幾處是「照 header 說的長度去動作，沒驗實際長度」？

**第二層，組織的，也是本章真正要你帶走的：你的關鍵相依裡，有哪些是「全世界都在用、其實沒人在顧」的單點？你的 SBOM 知道嗎？** 你 `npm install` 一個套件，它再 `dependencies` 一串你沒聽過的東西，那些東西又依賴更深的東西。你的服務最終跑在一棵幾百上千個節點的相依樹上，而其中**最關鍵的那幾個葉子，可能正是某個維護者在週末用愛發電的專案**。你預設它「被很多人用，所以很穩、很安全」——這就是 Heartbleed 砸碎的那個直覺：**「廣泛使用」是流行度，不是維護品質的保證。** 一個被一千萬個專案依賴、卻只有一個累壞的維護者的套件，不是「很安全」，是「一個極大的、沒人在顧的單點」（這條線延伸到供應鏈攻擊，見 ch20）。

焦慮工程師的版本，命名那個具體的未處理失敗案例：你的 production 此刻正信任著至少一個你叫不出名字、不知道誰在維護、上次安全稽核是什麼時候的相依。它平時沉默，因為還沒人去戳它。它出事的那天，會是某個你管不到的志工、在某個你沒讀過的檔案裡、漏了某一行檢查的那天——而你會跟 2014 年四月全世界的維運一樣，被迫假設「一切都已洩漏」，然後輪換你以為很安全的每一把鑰匙。

可操作的自問，不是「要關注供應鏈」這種廢話，是這三條具體的：

1. **你有 SBOM 嗎？** 你能不能在五分鐘內列出「我的服務直接與間接依賴了哪些套件、各是哪個版本」？如果不能，下一個 Heartbleed／Log4Shell（見 ch20）來的時候，你連「我中不中招」都答不出來——而那一天，分秒必爭。
2. **你的相依樹裡，最關鍵的三個是誰在維護？** 挑出「掛了你服務就掛」的那幾個（密碼學庫、序列化庫、你的 web framework 核心），去查：幾個維護者？多久沒更新？有沒有金主／基金會在後面？如果答案是「一個人、業餘、零資金」，你就找到了一個 Heartbleed 形狀的單點。
3. **你怎麼比對「宣告的長度」和「實際的內容」？** 在你自己解析外部輸入的每一處——header 宣告的 size、訊息宣告的 count、協定宣告的長度——你有沒有那一行「宣告值必須對得上實際值，否則拒絕」的檢查？還是你跟 2011 年那次 review 一樣，因為「這欄位應該不會說謊吧」就放它過去了？

## 紙上推演

### 推演題1 — 口頭重講：為什麼一個 missing bounds check 能潛伏兩年 **[15 分鐘] ★★**

不看書，向一個沒聽過這案子的工程師，用三分鐘講清楚兩件事：(a) 技術上，Heartbleed 是怎麼從「一個說謊的心跳請求」變成「讀走伺服器私鑰」的——必須講到「宣告長度 vs 實際長度」「越界讀取」「不留日誌」三個環節；(b) 為什麼這個缺陷能潛伏**逾兩年**沒被發現——而這個答案**不能**是「沒人夠聰明」。

#### 推演解答

(a) 合格的技術講述：心跳協定讓你送一段 payload＋宣告它多長，伺服器把同樣的 payload 回送。OpenSSL 相信你**宣告**的長度，照它 memcpy 回送，卻沒檢查你**實際**送了多少。攻擊者送 1 byte 卻宣告 64 KB，伺服器就把 payload 緩衝區之後的 64 KB 鄰近記憶體（可能含私鑰、密碼、cookie）一併回送——這是越界**讀取**（不是寫壞），在 C 裡靜默成功、不 crash、不留 log，所以可反覆抽取且無法事後查證。

(b) 合格的「為什麼潛伏兩年」答案必須落在**組織與系統條件**，不是個人能力：這段碼跑在一個**約四人、僅一人全職、年捐約兩千美元**的專案上；review 確實做了，但團隊資源稀薄、這又是個看似無害的新功能；漏洞**不留痕跡**，所以即使被利用過也不會浮現成「事故」去引人注意。換句話說——它能潛伏兩年，不是因為它藏得多巧，是因為**幾乎沒人有資源、有動機、有工具去盯它**。如果你的版本是「就是沒人發現而已」，你就漏掉了本章最重要的一層：「被很多人用」從不等於「被很多人看」（Linus 定律的前提沒被滿足）。

### 推演題2 — 反事實：補上不同的那層起司 **[20 分鐘] ★★**

考慮三個獨立的「假如」，分別問它能不能阻止 Heartbleed、以及它擋住／沒擋住哪些孔洞：(甲) 假如 heartbeat 實作裡有那一行邊界檢查（宣告長度必須對上實際 payload，否則忽略）；(乙) 假如 OpenSSL 是用記憶體安全的語言（如 Rust）寫的，越界讀取會 panic 而非靜默成功；(丙) 假如 OpenSSL 一直有 CII 那種等級的資源（半打全職人手＋定期安全稽核），但那行 code 還是照原樣寫出來了。哪一個最徹底？三者各對應修補哪一個孔洞？

#### 推演解答

(甲) 邊界檢查：直接堵死孔洞1、2。攻擊者宣告 64 KB 卻只送 1 byte，會被當成矛盾請求忽略，沒有越界讀取——這就是 1.0.1g 實際做的修補，一行解決。它擋住了這個**具體**的 bug，但沒改變「C 沒有記憶體安全」「下一個 memcpy 還可能漏檢」的體質。

(乙) 記憶體安全語言：補的是孔洞3。越界讀取會在執行期 panic／報錯，而非靜默回送鄰居記憶體——所以即使有人漏了那行檢查，傷害也會是「crash」而非「無聲洩漏私鑰」。這比 (甲) **更治本**，因為它把整**類**「越界讀取靜默成功」的 bug 變成可見的失敗（這正是近年用 Rust 重寫關鍵基礎設施的論據之一）。但它換來的是另一種失敗模式（panic = 可用性問題），且無法防住邏輯層的錯誤。

(丙) 資源充足：補的是孔洞0。但注意——**它不保證那行 bug 不被寫出來**，人會犯錯與資源無關。它改變的是 bug **被發現與修補的速度**：定期稽核、足夠的 review 眼睛、模糊測試（fuzzing），讓這個缺陷不可能潛伏兩年——很可能幾週、幾個月內就被抓到，blast radius 小得多。

最徹底的是**疊加**，而非單選：CII（丙）讓 fuzzing 與稽核更可能在出貨前抓到它；記憶體安全（乙）讓即使漏抓也不會靜默洩漏；邊界檢查（甲）是當下最便宜的止血。這正是瑞士起司的精神——你不需要補上每一層，但每多補一層，這個缺口能造成的傷害就指數下降。而本章的重點是：(丙) 是大家最容易忽略、卻決定「能潛伏多久」的那一層。

### 推演題3 — 對帳你自己的相依：找出你的 Heartbleed 形狀單點 **[30 分鐘] ★★★**

挑一個你實際維運的服務，做兩件事。(a) 試著產出（或盤點）它的相依清單——直接相依先列，能列出間接相依更好。(b) 從中挑出三個「掛了你服務就掛」的關鍵相依，對每一個查證：幾個活躍維護者？多久沒發版？背後有沒有公司／基金會在養？最近一次安全稽核（如果查得到）是什麼時候？然後評估：哪一個最像 Heartbleed 形狀的「全世界都用、其實沒人在顧」的單點？對那一個，你能採取的具體降風險動作是什麼？

#### 推演解答

這題沒有標準答案，但有合格標準。不合格的版本是「我應該多注意供應鏈」——那是焦慮不是工程。合格的對帳必須具體到「哪個套件、誰維護、什麼風險、做什麼」。一個夠好的例子長這樣：

「我的 NestJS 服務直接依賴約 40 個套件，`npm ls` 攤平後間接相依逾 800 個。挑三個關鍵的：(1) 我的 TLS 終止其實在 ALB，由 AWS 維護、資源充足，這個相對安心；(2) 我用某個小眾的序列化／驗證庫處理外部 webhook payload，去 GitHub 一看——**一個維護者、上次發版 14 個月前、零贊助**，這就是我的 Heartbleed 形狀單點：它直接吃外部輸入、又沒人在顧；(3) 我的 logging 庫雖然主流，但提醒我 Log4Shell（見 ch20）就是這類東西出的事。降風險動作：對 (2)，短期——把它包在我自己的輸入驗證層後面（先 sanity check 才交給它解析）、釘住版本並開 Dependabot 告警；中期——評估換成有金主／更活躍的替代品，或自己 fork 並補上我需要的檢查；長期——把『關鍵相依的維護健康度』納入定期 review，而不是等出事才查。」

注意這個回答的關鍵動作是「**先在我自己這層補上輸入驗證，不把信任全押在那個沒人顧的相依上**」——這正是 Heartbleed 與 ch20 的共同教訓：你的信任邊界要畫在「你能控制的那一層」，而不是「你 import 進來的那一層」。

## 自我檢核

口頭自答，講得出來才算過關：

1. heartbeat 的設計本意是無害的（確認連線還活著）。那麼危險的到底是 heartbeat 這個概念，還是某個具體的實作決定？請精確說出是哪一個決定。
2. 為什麼說 Heartbleed 是 buffer over-**read** 而不是 over-write？這個區別為什麼讓它「靜默成功、不留痕跡」、進而讓善後變成「假設一切已洩漏」？
3. 為什麼這個缺陷能潛伏逾兩年？把這個答案講成「沒人夠聰明」或「就是運氣不好」為什麼是錯的？Linus 定律（足夠多的眼睛）的前提，在這個案子裡哪裡沒被滿足？
4. 為什麼把根因定在「Seggelmann 寫錯了一行、review 沒抓到」是個會讓你錯失教訓的 root-cause 迷思？真正的「孔洞0」是什麼？
5. 1.0.1g 的修補只有一行（宣告長度對不上實際就忽略心跳）。既然修補這麼簡單，為什麼這仍是一場全球級災難而不是一個小 patch？（提示：F14、潛伏期、不留痕跡）
6. CII 修的不是那行 code，是什麼？為什麼說「給錢給人」補的是「能潛伏多久」這一層，而不是「會不會被寫出來」那一層？
7. 在你自己的系統裡，「相信上游宣告的長度／數量，卻沒比對實際內容」最可能出現在哪一處解析邏輯上？那裡缺的那一行檢查是什麼？
8. 你的關鍵相依裡，哪一個最像「全世界都用、其實沒人在顧」的單點？你的 SBOM 答得出「我中不中招」嗎？如果答不出，下一個 Heartbleed 來的那天你會怎麼樣？
