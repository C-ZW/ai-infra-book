## 從你已知的出發

你寫過一個全團隊都在用的多層 cache library：先查 in-memory，沒命中退到 S3，再退到 local，每一層都帶 graceful degradation。我要你先記住一件事——**那個東西本身就是這一章的主角之一**。你不是在讀一個與你無關的太空故事，你是在讀「你已經在做的事，三十年前在月球上空怎麼救了三條命」。

想想你那套 cache 真正在做什麼。當 in-memory 層失效，你的系統沒有「全有或全無」地崩掉——它退到下一層，吞下延遲，繼續服務。使用者也許慢了幾百毫秒，但他拿到了結果。這就是 graceful degradation 的本質：**壞掉的時候，不是停止服務，是降低服務**。你心裡有一條優先順序——「拿到資料」比「拿得快」重要，所以你寧可慢、不要斷。

現在把這條原則放大到一台 1969 年、記憶體以「字」計、時脈以 KHz 計的電腦上。它在月球上空一萬呎、燃料只夠再撐幾十秒、太空人的命壓在它身上的那一刻，做了和你的 cache library 同一件事：**過載了，就丟掉不重要的，保住要命的**。

這一章我會把你已經在做的事，放進一條從阿波羅到今天的譜系裡；然後誠實地告訴你，你還缺哪幾道防線（kill switch、混沌演練、依賴多樣性、canary）——因為焦慮工程師的信條不是「我有 graceful degradation 所以沒事」，而是「我有一道，但另外五道孔洞還開著」。

## 冷開場：1969-07-20，月球上空，"PROGRAM ALARM"

登月小艇 Eagle 進入動力下降（powered descent）約五分鐘。Armstrong 與 Aldrin 站在那台叫 Apollo Guidance Computer（AGC）的電腦前——它是人類第一台投入實戰的數位飛控電腦，由 MIT Instrumentation Laboratory 設計，記憶體小到今天一張賀卡的晶片都看不上眼。

然後 Armstrong 的聲音傳回休士頓：「Program alarm.」接著:「It's a 1202.」

地面控制室瞬間繃緊。1202 是什麼？沒人——包括正在下降的太空人——當下完全清楚。再幾秒，又一次。整個下降過程，AGC 一共跳出四次程式警報（三次 1202、一次 1201，照 landscape；部分史料記為五次⚠️）。每一次都像在問同一個問題：**要不要中止登月？**

中止登月（abort）意味著點燃上升引擎、拋掉下降段、放棄這次著陸——耗費了一個國家十年與數十億美元、全世界都在看的這一刻，可能因為一台電腦喊「我有狀況」而作廢。

但他們沒有中止。28 歲（一說 24 歲）的導引官 Steve Bales，與在他身後、口袋裡揣著一張小抄的 24 歲工程師 Jack Garman，在數秒內判定：**只要警報不是連續刷屏、導引資料仍然正常，就 GO。** Bales 回報那句後來進了歷史的話：「We're GO on that alarm.」

Eagle 繼續下降。1969-07-20，人類第一次踏上月球。

這裡是全書最重要的反轉：**1202 不是電腦壞了**。

## 裝填：AGC 為什麼會過載

要看懂這個正面案例，得先看清楚它「險」在哪——因為它的險，和前面二十章的災難長得一模一樣。

潛伏條件早就裝填好了。AGC 在動力下降階段本來就逼近運算負載上限——這是一台用盡了每一個週期的電腦，沒有多少餘裕。觸發是一個介面設定問題：Aldrin 把會合雷達（rendezvous radar）的開關放在某個位置，本意是「萬一要中止登月，好快速重新捕捉指揮艙」——一個完全合理的謹慎之舉。但會合雷達與 AGC 的電源相位不同步，使雷達不斷向 AGC 索取它根本不需要的計數週期，這叫 cycle steal（週期竊取）。它憑空吃掉約 13% 的 CPU 時間。

一台本就接近滿載的電腦，再被偷走 13%，就溢位了。AGC 在每個週期內排不下所有該做的工作，於是丟出警報：1202 是 "Executive overflow — no core sets"，1201 是 "Executive overflow — no VAC areas"——白話說，「我超載了，沒有可用的工作儲存區了」。

停在這裡，你看到的是一場標準的正常意外（見 ch01）的裝填：一個緊耦合的系統（F1）、一個沒人預期的子系統交互（會合雷達與導引電腦共享 CPU，F2）、一個合理操作意外觸發的潛伏條件（F8）。如果 AGC 是用前面二十章那些災難的設計哲學寫的，接下來就是熟悉的劇本——它會當機、會鎖死、會在最糟的時刻把一架好太空船丟回給來不及反應的人（這正是法航 447，見 ch05）。

但它沒有。這才是本章的全部重點。

## 為什麼它活了下來：優先權排程 + 重啟丟負載

AGC 的作業系統不是按固定時序跑死板排程的。它的 executive 由 Hal Laning 設計成**優先權導向、事件驅動的非同步排程器**——每個工作（job）帶一個優先權。下降導引、引擎控制、與太空人溝通的 DSKY 顯示，是最高優先；那些被雷達塞進來的雜務，是低優先。

關鍵在於：當 AGC 偵測到過載——它要不到 core set、要不到 VAC area——它**不會當機，也不會鎖死**。它做了一件當時極為前衛、今天我們叫 graceful degradation 的事：執行「軟重啟（software restart）」。重啟時，它丟棄低優先權、可重啟的工作，只重新排程並保住最高優先權的關鍵工作。MIT 的工程師 Don Eyles（寫了下降導引軟體的人）後來把這套機制叫 "restart protection"，形容它像「打進去一通假的炸彈威脅電話」——疏散非必要的人、保住核心。

於是每次 1201/1202 之後，AGC 都在毫秒級內重啟、繼續穩穩執行登月導引。被丟掉的是低優先的雷達雜務，導引、引擎、顯示**毫髮無傷**。太空人甚至沒有感覺到下降中斷過——對他們而言，數字一直在更新，飛行一直在繼續。

這就是優雅降級的精確定義，請你把它刻進腦子，因為它是全書收官的鑰匙：

> **韌性不是「不會壞」，是「壞的時候，主動犧牲非必要的，保住核心的」——而不是全有或全無地崩潰。**

再把它和你的 cache library 並排看：你的 in-memory 層「壞了」（cache miss），系統不崩，退到 S3，吞延遲、保結果。AGC 的低優先工作「排不下了」，系統不崩，丟掉雜務、保導引。**這是同一個形狀**。你每天在做的工程決策，和救了阿波羅 11 號的，是同一條原則。

還有一條同樣重要、但常被浪漫敘事蓋掉的線：**人的準備**。Garman 那張小抄不是運氣，是事前把「每一種程式警報代表什麼、哪些可以忽略」整理好的結果。Bales 與 Garman 能在數秒內判讀「這個警報不致命」，靠的是準備，不是英雄式的臨場直覺。這對你意味著：**你的 runbook、你的告警分級、你 oncall 時手邊那份「這個 alarm 響了該怎麼判斷」的文件——就是 Garman 的小抄**。系統的優雅降級若沒有人的「我知道這個降級是預期內的」配套，照樣會有人在恐慌中按下中止鍵。

## 對照組：為什麼 Apollo 活了，Schiaparelli 死了

把阿波羅 11 號和 Schiaparelli 火星著陸器（見 ch08）並排，是全書最乾淨的一組對照——因為它們的觸發幾乎一樣，結局卻完全相反。

| | 阿波羅 11 號（1969） | Schiaparelli（2016） |
|---|---|---|
| 異常 | 雷達偷 CPU、執行緒過載 | IMU 飽和、產生壞的姿態資料 |
| 壞數值 | 「排不下所有工作」 | 積分出「負高度」（物理上不可能） |
| 系統反應 | 偵測過載→軟重啟→丟低優先、保導引 | 壞數值直通決策→誤判已著陸→提前關推進器 |
| 有沒有合理性檢查 | 有：過載被當成「該降級」的訊號 | 沒有：「負高度」沒被任何一層擋下 |
| 緊耦合？ | 否：過載與「中止任務」之間有緩衝（降級層） | 是：壞數字直通推進器關閉，無緩衝 |
| 結局 | 成功著陸 | 自 3.7 km 高空墜毀、約 540 km/h 撞地 |

差別不在「誰的硬體比較好」。差別在**設計哲學**：

- Schiaparelli 信任了一個它不該信任的數值，把它直接餵給最終決策（F10 過度信任單一估計、F1 緊耦合直通）。一個飽和的感測器產生一個荒謬的數字（負高度），沒有任何一層問一句「這合理嗎？高度不可能是負的」，就觸發了關掉推進器。
- 阿波羅把過載當成一個**預期內、可降級的狀態**，而不是一個必須崩潰的錯誤。它的設計者假設「總有一天會過載」，並且為那一天寫好了優雅的反應。

這是焦慮工程師與樂觀工程師的分水嶺。樂觀工程師假設輸入是好的、負載在預期內、依賴會在。焦慮工程師假設輸入會是壞的、負載會爆、依賴會掛——然後為那一天**設計好降級路徑**。阿波羅的設計者是焦慮工程師。Schiaparelli 的某一層不是。

## 故障解剖：把 F1–F14 翻成緩解手段

本章的招牌不是解剖一場災難，而是**把全書累積的失敗模式語言整個翻過來**——每一個 F 模式，配一道緩解手段，並指出前面哪一章正是因為缺了它而崩。這就是 ch23 總目錄的骨架。

先反向解剖阿波羅 11 號這個正面案例——它「對在哪」：

- **潛伏條件 vs 觸發**：潛伏是「AGC 接近滿載、無多少餘裕」；觸發是「雷達 cycle steal 偷 13% CPU」。和前二十章一模一樣的裝填。差別在第三格——**系統對觸發的反應**：阿波羅選了降級，不是崩潰。
- **本可崩潰、卻沒崩的每一道設計**（把瑞士起司倒過來看——這次每一層都擋住了）：

| 防線 | 阿波羅怎麼做對 | 哪一章缺了它而崩 |
|---|---|---|
| 過載偵測 | AGC 主動偵測「要不到 core set」 | Morris 蠕蟲再感染失控前無過載煞車（見 ch18） |
| 優先權排程 | 關鍵工作有保證的優先權 | 2010 閃電崩盤（見 ch11）無價格/時間護欄，代理互相踩踏 |
| 重啟丟負載 | 軟重啟丟低優先、保核心 | Schiaparelli（見 ch08）壞數值直通、無降級 |
| 合理性把關 | 過載被當訊號、非致命錯誤 | Schiaparelli 對「負高度」無 sanity check |
| 人的準備 | Garman 小抄、Bales 快速判讀 | 三哩島（見 ch16）操作員被指示燈騙、無法判讀真實狀態 |

下面是全書的核心商品——**F1–F14 反面解方對照表**。讀法：左邊是失敗模式，中間是緩解手段，右邊是「前面哪一章因為缺它而崩」。

| F | 失敗模式 | 緩解手段 | 缺它而崩的案例 |
|---|---|---|---|
| F1 | 緊耦合 | **解耦＋緩衝（buffer/queue）、bulkhead 艙壁隔離** | Schiaparelli 壞數字直通推進器（見 ch08） |
| F2 | 互動複雜度 | **降低耦合面、明確契約、依賴圖盤點、混沌演練逼出非預期交互** | 車諾比正空泡係數的正回授（見 ch17）、雷達偷 CPU |
| F3 | 起司孔洞對齊 | **多層獨立防禦＋孔洞錯開（不要同因）、定期演練每一層** | 挑戰者雙 O-ring 同因失效（見 ch14） |
| F4 | 安全系統即危險源 | **失效要 fail-safe，不要 fail-deadly；保護機制本身要有「最壞情況下行為」的審查** | 車諾比 AZ-5 反而引爆（見 ch17）；阿波羅是反例——重啟保護真的保護了 |
| F5 | 自動化的諷刺 | **保留人的手動接管肌肉記憶、定期演練、清楚的狀況顯示** | 法航 447 自動化在最糟時刻丟回給機組（見 ch05） |
| F6 | 偏差正常化／漂移 | **重新檢視被正常化的告警、把「暫時」設過期日、定期挑戰「為什麼這還沒爆」** | 挑戰者 O-ring 侵蝕被逐步接受（見 ch14）、德州電網防寒（見 ch12） |
| F7 | 級聯／雪崩 | **circuit breaker 熔斷、backpressure 反壓、rate limit、隨機抖動打斷正回授** | 2010 閃電崩盤 hot-potato（見 ch11）、Morris 蠕蟲再感染（見 ch18） |
| F8 | 潛伏條件 vs 觸發 | **廢碼清除（dead code）、定期完整重啟驗證、把潛伏假設寫成測試** | 金姆利停飛前 FQIS 已失效的潛伏（見 ch07）、Heartbleed 潛伏兩年（見 ch19） |
| F9 | 介面與假設的縫 | **帶單位的型別、contract test、端到端驗證、sanity check 合理性閘門** | 金姆利磅當公斤（見 ch07）、Schiaparelli 負高度（見 ch08） |
| F10 | 單一故障點／單一感測器 | **冗餘的「多樣性」（diverse redundancy）、交叉驗證、多數決** | 737 MAX 單一 AoA 感測器（見 ch04） |
| F11 | 監控盲區 | **看實際狀態而非意圖、告警分級、告警不可被淹沒、儀表反映現實** | 三哩島指示燈顯示「指令」非「閥位」（見 ch16） |
| F12 | 變更即觸發 | **canary／分批滾動部署、自動 rollback、kill switch、變更複核** | Cloudflare regex 一次推全球（見 ch10）、CrowdStrike 無灰度（見 ch13） |
| F13 | 生產壓力壓過安全 | **明確的「停下來查」授權、blameless 文化、不以進度凌駕 gate** | 挑戰者發射前夜電話會議（見 ch14）、車諾比為完成測試停用保護（見 ch17） |
| F14 | 規模放大／單一栽培 | **依賴多樣性、分批 blast radius 控制、避免全域同步推送、隔離爆炸半徑** | CrowdStrike 850 萬台同時藍屏（見 ch13）、Morris 蠕蟲打通 Unix 同質生態（見 ch18） |

注意 F3、F10、F14 共用一個你必須吃透的概念，因為它是全書最反直覺的一道：**冗餘的「多樣性」**。

想想挑戰者號的雙 O-ring（見 ch14）:主環、副環是同一種橡膠、同一個接頭、暴露在同一個低溫下——讓主環失效的機制一字不差地也讓副環失效。兩道防線不是獨立的,是同一個失效模式的兩份副本,被同一顆子彈(低溫)同時打中。**複製一份有 bug 的東西,不叫冗餘,叫把同一個 bug 部署兩次。**

這也是 Cloudflare（見 ch10）與 CrowdStrike（見 ch13）的形狀：全球同步推送同一份內容，等於把同一顆炸彈同時投到每個節點——你有一萬台機器，但它們是一萬份相同的 bug。真正的冗餘要有**多樣性**：不同實作、不同版本、不同供應商、不同部署批次——這樣同一個觸發才不會同時打穿所有副本。阿波羅的優先權排程之所以有效，部分正是因為它沒有把「所有工作」當成同質的一坨，而是分了優先級、可以選擇性丟棄。

**這場「災難」的形狀是（反過來說）：一個緊耦合、互動複雜、本會過載的系統，因為設計者預設了「總會過載的那一天」並寫好了優雅降級——過載被當成可降級的訊號而非致命錯誤，於是孔洞沒有對齊，意外沒有變成災難。**

## 換成你的系統

現在把這套清單對準你自己的系統，點名你「應該沒事吧」的假設。

你有 graceful degradation（cache library），這是你的強項。但焦慮工程師不會停在強項上。我替你把最可能開著的孔洞列出來：

**假設一：「我的降級路徑會在真出事時生效。」** 你的 cache library 退到 S3 那條路，上一次真正在生產跑過是什麼時候？還是只在你寫它的時候測過一次？降級路徑是所有程式碼裡最少被執行、因此最可能悄悄壞掉的——它是你的「備用降落傘」，而備用降落傘最怕的就是「從沒在真實墜落中打開過」。這就是 chaos engineering（混沌工程）要解決的：**主動在生產（或近生產）注入故障，逼你的降級路徑真的跑一次**，趁你還醒著、還在看的時候，而不是凌晨三點 oncall 時。你做過嗎？

**假設二：「我的部署不會一次打中所有人。」** 你的服務跑在 ECS/Lambda 上，你會做藍綠/滾動部署——但那是不是 canary（金絲雀）？滾動部署只是「慢慢全換」，canary 是「先換 1%，盯著錯誤率，不對就自動 rollback，對才繼續」。Cloudflare 和 CrowdStrike 的差別不在它們不會滾動，而在它們把一份壞東西一次推給全世界、中間沒有一個「先放 1% 看看」的閘門。你的設定變更、feature flag 切換、規則更新，有沒有 canary？

**假設三：「真出事我來得及手動止血。」** 你有 kill switch 嗎——一個能在 30 秒內把某個失控功能、某條熱路徑、某個正在自我放大的迴圈整個關掉的開關？Cloudflare 那次靠全球 kill switch 關掉 WAF 才止住血（見 ch10）。2010 年閃電崩盤當時欠缺單一個股熔斷,自動賣壓正回授幾分鐘內蒸發近兆美元（見 ch11）。你那個自動 retry、自動 scaling、自動 failover——當它開始正回授、把系統往下推的時候（這是 retry storm/thundering herd 的形狀），你有沒有一個不需要重新部署、就能立刻掐斷它的開關？

**假設四：「我的冗餘是真的冗餘。」** 你的兩個 replica、兩個 region、兩條路徑——它們是不是跑同一份 image、同一個 bug、依賴同一個上游？挑戰者的雙 O-ring（見 ch14）教你的是：相同的副本對相同的觸發無效。你的「高可用」有沒有可能只是「把同一個 SPOF 部署了兩次」？

一個可操作的自問，給你帶走：**列出你系統裡每一個「自動採取激烈動作」的機制（autoscaling、failover、retry、circuit breaker、自動清理），問每一個：當它的輸入訊號是錯的時候，它會把系統推下懸崖，還是優雅降級？它有沒有 kill switch？** 這正是 737 MAX（見 ch04）的問題——MCAS 是一個「依賴單一錯誤輸入就採取激烈動作、且沒有 kill switch、飛行員還不知道它存在」的自動化。你的系統裡有幾個 MCAS？

## 紙上推演

### 推演題1 **[15 分鐘]** ★

口頭（對另一個工程師）講清楚：為什麼阿波羅 11 號的過載沒有釀災，而 Schiaparelli 的壞數字釀了災？要點名「優雅降級」與「合理性檢查」這兩個機制，並說明兩者觸發何其相似、結局何其相反。

#### 推演解答

兩者的觸發幾乎同構：都是一個子系統產生了系統「沒預期、超出正常範圍」的狀態——阿波羅是執行緒過載（排不下工作），Schiaparelli 是 IMU 飽和（產生負高度）。差別只在系統怎麼接這個異常：

- 阿波羅把「過載」當成一個**預期內、可降級的訊號**。它的 executive 偵測到要不到 core set，執行軟重啟，丟棄低優先工作、保住導引——過載與「任務失敗」之間有一層緩衝（降級機制）。
- Schiaparelli 把「負高度」當成**真值**直接餵給最終決策，沒有任何一層問「高度不可能是負的」。壞數值與「關掉推進器」之間**沒有緩衝**（F1 緊耦合 + F10 過度信任單一估計 + F9 缺合理性閘門）。

所以一句話總結：差別不在硬體、不在運氣，在**設計者有沒有為「總會出現的異常」預先寫好優雅的反應**。阿波羅的設計者假設「它總有一天會過載」並為那天設計；Schiaparelli 的某一層假設「IMU 給的數字都是對的」。前者是焦慮工程師，後者不是。把這條接回你的 cache library：cache miss 是你「預期內、可降級的訊號」，你退到下一層——你做的就是阿波羅做的事。

### 推演題2 **[25 分鐘]** ★★

拿你維運的一個真實系統，逐一對照本章的「安全地壞」清單。重點找三種洞：（a）哪裡是「全有或全無」、缺降級路徑；（b）哪裡缺 kill switch；（c）哪裡的冗餘其實只是同一份 bug 的副本（缺多樣性）。每一條寫下「它在什麼輸入/負載下會崩，以及崩成什麼樣」。

#### 推演解答

沒有標準答案，但合格的盤點要對每一條給出**具體的失敗劇本**，不能只寫「可能會有問題」。示範三條（以你的技術棧為例）：

- **(a) 全有或全無**：「結算 pipeline 的 idempotent consumer 若下游 RDS 不可用，目前會整批 retry 直到 SQS 訊息過期進 DLQ——沒有降級路徑（如先寫入暫存、稍後補算）。輸入：RDS 故障 10 分鐘。崩法：訊息堆積、DLQ 爆量、結算延遲，且 retry 本身加重 RDS 負載（正回授）。」
- **(b) 缺 kill switch**：「自建 Lambda decorator 框架的自動 retry，沒有一個全域開關能在不重新部署的情況下關掉某個正在 retry storm 的端點。輸入：某下游持續 5xx。崩法：retry 放大成 thundering herd，把下游與自己一起打趴，而我只能等重新部署生效。」
- **(c) 假冗餘**：「兩個 ECS task 跑同一份 image。輸入：image 裡有個會在特定 payload 下 OOM 的 bug。崩法：同一個 payload 打到哪個 task 都 OOM，兩個副本不提供任何獨立性——這是『把同一個 bug 部署兩次』的形狀（見 ch14 雙 O-ring 共因失效）。」

評分標準：你有沒有對每條寫出「哪個輸入/負載 + 崩成什麼樣」。寫不出具體劇本的那一條，就是你還沒真正想清楚的孔洞。

### 推演題3 **[20 分鐘]** ★★★

設計題：給你一個現有的、目前「全有或全無」的關鍵系統（自選），你只能加三道防線。你會加哪三道、各擋住哪個 F 模式、各自的代價（複雜度/延遲/成本）是什麼？論證為什麼是這三道而非別的。

#### 推演解答

合格答案要展現**取捨意識**——韌性不是免費的，每加一道防線都有代價，焦慮工程師要能說清楚「我為什麼先加這三道」。一個示範論證（以結算 pipeline 為例）：

1. **graceful degradation 降級路徑（擋 F1 緊耦合）**：RDS 不可用時，先寫入暫存佇列、稍後補算，而非整批失敗。代價：增加一條程式碼路徑與最終一致性窗口、要處理補算的冪等。為什麼第一：它把「全有或全無」變成「降級服務」，性價比最高——直接消除最致命的崩法。
2. **circuit breaker（擋 F7 級聯）**：下游連續失敗時跳開、停止 retry、快速失敗，避免 retry storm 正回授。代價：要調 threshold、half-open 探測；調太敏感會誤跳。為什麼第二：它防止「故障被自己放大」，且與第一道互補（熔斷後走降級路徑）。
3. **kill switch（擋 F12 變更即觸發 / 給人最後的手）**：一個不需重新部署、能立刻關掉某功能的開關（feature flag 已有，補上「緊急關閉」語意與演練）。代價：要維護開關狀態、避免它自己變成 SPOF。為什麼第三：當前兩道自動機制也誤判時，它是人的最後一道手動止血——Garman 的小抄與 Bales 的判讀的工程化。

關鍵在論證「為什麼這個順序」：先消除最致命的崩法（全有或全無），再防止放大（級聯），最後保留人的接管能力（kill switch）。三道彼此互補、覆蓋不同 F 模式，而不是三道都在防同一件事。

- **NASA Apollo 11 Lunar Surface Journal — Program Alarms 專頁**（https://www.hq.nasa.gov/alsj/a11/a11.1201-pa.html）：含 Bales、Garman、Hamilton 的第一手回憶與技術說明，是 1201/1202 事件的一級史料。要看「過載被優雅化解」的原始現場，從這裡開始。

- **David A. Mindell,《Digital Apollo: Human and Machine in Spaceflight》（MIT Press, 2008）**：把阿波羅飛控放進「人與自動化如何分工」的脈絡——正好是法航 447（見 ch05）與于伯林根（見 ch06）那條自動化諷刺線的正面對照。
- **Sidney Dekker,《Drift into Failure》**：本書「漂移到失效」的源頭（見 ch02、ch14）；讀完阿波羅的優雅降級，再回頭看「韌性是設計出來的、不是運氣」這個論點的理論底座。
