# ch18 — Morris 蠕蟲：第一次，整個網際網路停了下來

> **本章解剖什麼**：這是 Part VI「看不見的縫：安全與供應鏈」的開場。前五個 Part 的災難多半發生在一個組織內部——一架飛機、一座平台、一座反應爐、一個資料中心。從這裡開始，缺口不在你的程式裡，而在你**信任的東西**裡：你 import 的函式庫、你開著的預設設定、你跑出去的探測程式。Morris 蠕蟲（1988-11-02）是史上第一隻網際網路蠕蟲，主打 **F14（單一栽培／blast radius）**、**F8（潛伏條件 vs 觸發）**、**F4（安全系統即危險源）**、**F7（級聯）**。它最反直覺的地方：作者**沒有**想破壞任何東西——一個再感染機率參數設高了一點，就把一支善意程式變成了拖垮半個網際網路的 DoS。這正是「應該沒事吧」的標本。它與 ch10 的 Cloudflare regex（同步放大 vs 自我放大）成對，往後接 ch19 Heartbleed、ch20 供應鏈。

## 從你已知的出發

你寫過健康探測（health probe）。Kubernetes 的 readiness/liveness probe、你那條 FluentBit+Lua 的可觀測性 pipeline、跑在 CronJob 裡的對帳掃描——它們都有一個共同形狀：**一支以固定頻率、自動向別人發請求的程式，目的是「看看狀況」，本身不該造成傷害**。

但你一定看過監控反噬被監控對象。一個 liveness probe 的逾時設太短、間隔設太密，pod 在尖峰時稍微慢一拍就被判死、被重啟、重啟後冷啟動更慢、更容易再被判死——你親手把一個「只是想知道它還活著」的探測，變成了砍死它的劊子手。或者一條重試邏輯：上游慢了，下游重試，重試的流量讓上游更慢，更多重試湧入——這就是你複習過的 retry storm。再或者一隻爬蟲、一個背景 job，掃描頻率乘上併發數，把本來要保護的 RDS 打到 CPU 100%（你太熟這個數字了，只是上次是你**降**了 40%，這次是反過來）。

這些東西的危險不在「它做了什麼壞事」，而在**一個頻率參數、一個機率、一個間隔**。程式邏輯完全正確、意圖完全善良，但那個數字設錯一格，善意的探測就變成自我 DoS。

Morris 蠕蟲是這個形狀的祖宗。它的作者是一位聰明、無惡意的研究生，他想做的只是「估算一下網際網路有多大」。他甚至特地加了一段防止重複感染的邏輯——一段**安全機制**。問題出在那段安全機制裡一個分數：七分之一。

## 冷開場：1988 年 11 月 2 日晚上

那是個星期三的晚上。大約晚間八點半，一支程式從 MIT 的一台機器被放了出去——刻意選在 MIT，作者是康乃爾大學的研究生 Robert Tappan Morris，他想掩飾來源。（日期、釋出地點、作者身分照 landscape。）

那年的「網際網路」還很小：大約六萬台機器連在一起，主要是大學、研究機構、軍方的研究網路。沒有防火牆的概念，沒有入侵偵測，機器之間預設互相信任——因為連上來的，理論上都是「自己人」。

當晚稍後，加州大學 Berkeley 的一名學生在一封信裡寫下這句後來被反覆引用的話：「**We are currently under attack**（我們正在被攻擊）。」（此句廣見於事後報導，作為事件當晚氛圍的引述。）

但問題是，沒有人真的在攻擊。沒有人在偷資料、刪檔案、勒索。各地的系統管理員只看到一件事：他們的 VAX 和 Sun-3 機器**越來越慢**，慢到動不了。登入進去一看，process table 塞滿了同一支不明程式的副本，一個接一個 fork 出來。把機器重開機，負載在幾分鐘內又爬回頂點。Gene Spafford（後來寫出最權威剖析報告的人）就是這樣發現的：他重開了實驗室的機器，眼睜睜看著負載再次一路飆高。

接下來的二十四小時內，估計約六千台主機被感染——約當時連網機器的百分之十。（⚠️「約 6,000 台／約 10%」是當代估計、被廣泛引用，但精確數字不可考；landscape 註明此為估計區間，以「約一成的網際網路」這個量級理解較穩妥。）

這不是一場攻擊。這是一支善意的程式，因為一個參數設錯，把它接觸到的每一台機器都活活拖死。

## 裝填：那些早就敞開的門

要理解這場災難，先看那些**潛伏條件**——在 Morris 寫下第一行程式之前，這些門就已經開著了。蠕蟲沒有「攻破」任何精心設計的防線，它只是走進了一棟到處沒鎖門的房子。

**第一道門：`gets()` 與 fingerd 的緩衝區溢位。** finger 是一個查詢服務，你打 `finger someone@host`，對方的 `fingerd` 守護程式告訴你那個使用者是誰、上次何時登入。fingerd 用 C 的 `gets()` 把網路傳來的輸入讀進一個 512 bytes 的 stack 緩衝區——而 `gets()` **不檢查長度**。蠕蟲送一個遠超 512 bytes 的字串，溢出去的部分覆寫掉 stack 上的 return address，把它指向緩衝區裡夾帶的一小段機器碼，那段碼執行 `/bin/sh`。這就是經典的 stack buffer overflow：一個讀輸入的小工具，因為信任「輸入不會超過我預想的長度」，把整台機器的控制權交了出去。（機制照 landscape；本書不教 exploit 寫法，只取「沒有邊界檢查」這一面。）

**第二道門：sendmail 的 DEBUG 模式。** 當時許多系統的 sendmail（郵件傳輸程式）出貨即開啟 DEBUG 模式。DEBUG 模式允許遠端把「指令」當成收件人傳進來並執行——等於內建了一個遠端執行的後門。沒人關它，因為「方便除錯」「應該沒事吧，誰會去濫用」。

**第三道門：rsh／rexec 的傳遞信任與弱密碼。** Unix 機器之間用 `.rhosts`、`hosts.equiv` 建立互信：A 信任 B，B 上的使用者不必密碼就能登入 A。蠕蟲利用這層互信橫向移動，並對帳號做字典猜密碼。一旦攻下一台，它就能順著信任關係跳到下一批。

**第四道門，也是最致命的潛伏條件：單一栽培（monoculture）。** 整個早期網際網路跑著高度同質的 4.2／4.3 BSD Unix，在 VAX 和 Sun-3 上。**同一組漏洞，通殺幾乎所有機器。** 這不是某一台機器特別脆弱，是整片生態用同一張臉、犯同一組錯。一把鑰匙開所有的門——這正是 F14 單一栽培的定義。你在 ch10 看過 Cloudflare 把同一條 regex 一次推到全球每個節點，在 ch13 會看到 CrowdStrike 把同一個內容更新推到八百多萬台 Windows——形狀完全一樣：同質化讓單點失效一次打中所有人。Morris 蠕蟲是這個模式在 1988 年的初版。

這四道門，每一道單獨看都「應該沒事吧」。`gets()` 用了幾十年沒出事；sendmail debug 方便；互信省事；大家都跑 BSD 因為它好用。它們各自無害，但全部對齊在同一片同質的網路上——瑞士起司的孔洞早已排成一線，只等一支程式走過。

## 觸發：七分之一

Morris 的程式本身不含任何破壞性 payload。它不刪檔、不竄改、不竊取（這點 Spafford 的剖析講得很清楚）。它唯一做的事，就是**傳播自己**：攻進一台機器，找出它認識的其他機器，複製過去，再從那裡繼續。

如果只是這樣，它會是一個安靜的、最多被當成異常流量的東西。真正把網際網路拖垮的，是傳播控制邏輯裡一個分數。

Morris 想得很周到——他**知道**重複感染同一台機器是浪費、會引人注意。所以他加了一道防護：蠕蟲在感染前先問對方「你已經被感染了嗎？」如果對方說是，就不要再感染。

這正是一個**安全機制**。它的設計意圖是「溫和傳播、不要打擾已感染的機器」。

但 Morris 多想了一層。他擔心：如果系統管理員看穿了這個握手協定，寫一個假程式回答「我已經被感染了」，蠕蟲不就被騙得無法傳播了嗎？這對一支想「估算網路大小」的程式來說是致命的——它會被輕易免疫。

於是他做了一個決定:**即使對方回答「我已經被感染了」，仍然有七分之一（約 14%）的機率無視這個答案、照樣重新感染一次。** 這個七分之一，是用來對抗「假裝已感染」的詐騙的——是一道對付防禦者的反防禦。

問題是，這個機率設得太高了。

七分之一聽起來很小。但想想實際發生了什麼：一台熱門的機器（比如連線多、被很多主機認識的伺服器）會被很多其他被感染的機器**反覆**嘗試感染。每一次嘗試，都有六分之七的機率被「已感染」擋下——但有七分之一漏過去，又跑起一份新的蠕蟲行程。被很多來源不斷叩門的機器，七分之一乘上叩門的次數，很快就累積出大量並存的蠕蟲副本。

每一份副本都在 fork、在掃描、在嘗試傳播。process table 爆滿，記憶體耗盡，CPU 被一群自己的副本啃光——這在效果上就是一個 **fork bomb**。機器不是被「攻擊」癱瘓的，是被**自己身上跑著的、太多份的同一支善意程式**活活拖死的。

這就是觸發。不是 `gets()`，不是 sendmail debug——那些是裝填好的彈藥（傳播向量）。真正扣下扳機、把「溫和傳播」變成「自我 DoS」的，是那個七分之一的重感染率。一個分數。

（要破除一個流行的簡化說法：「Morris 蠕蟲就是 fingerd 緩衝區溢位。」不完整。緩衝區溢位只是它進門的方式之一，它同時用了 sendmail debug、rsh 互信、密碼猜測。真正把機器搞當的，是重複感染導致的行程爆量——是那個參數，不是那個漏洞。技術上它也是 worm 蠕蟲、不是 virus 病毒：它自我傳播、不需寄生在宿主檔上。）

## 連鎖：放大的方向

把 Morris 蠕蟲和你熟悉的級聯放在一起，會看到一個有趣的對照。

ch10 的 Cloudflare 是**同步放大**：一條 regex 被一次推送到全球每個節點，所有節點在同一瞬間 CPU 衝到 100%。災難是「中央一推、四方同死」。

Morris 蠕蟲是**自我放大**：沒有中央，沒有人推送任何東西。每一台被感染的機器，自己變成下一個感染源；它感染得越多，被它感染的機器又去感染更多——指數成長。而在單一機器內部，七分之一的重感染率讓副本數也指數累積。這是**兩層級聯疊在一起**：機器之間的橫向擴散（F7），加上單機內部的行程爆炸（F7 的微觀版）。

兩種放大的共同點是 F14：因為單一栽培，橫向擴散沒有任何「天然的防火牆」——下一台機器幾乎一定也是 BSD、也開著同樣的門。同質化讓蠕蟲在每一跳都暢通無阻。

止血靠的是人。Berkeley 的 Computer Systems Research Group 在發現後約十二小時內，就拼湊出阻止蠕蟲擴散的步驟（修補 sendmail、過濾、把受感染機器斷網清理）。但這是**手動**止血——靠一群熬夜的工程師逆向分析一支當時誰也沒見過的東西。對照 ch10 Cloudflare 有一個全域 kill switch 能在數十分鐘內關掉 WAF，1988 年沒有任何「關掉它」的開關。網際網路那時還沒有為「萬一有東西在我們之間自我複製」準備過任何東西。

## 餘波：一個分數，催生了一個產業

蠕蟲被清除後，帳單不是用人命算的（無傷亡），而是用**時間和信任**算的。

清除成本的估計差異極大：上訴法院的判決引用了「每台機器移除成本約 200 至 53,000 美元」這樣的範圍；Clifford Stoll 為美國 GAO 估計的整體經濟衝擊約在 10 萬到 1,000 萬美元之間。（⚠️ 這些都是估計區間、跨度很大，照 landscape 保守處理，不要釘死單一數字。）真正的代價是它讓所有人第一次意識到：**這張把大家連在一起的網，本身就是一個攻擊面**。

兩件事誕生於此：

**第一，CERT/CC（CERT Coordination Center）。** 1988 年 11 月，就在蠕蟲事件之後，美國 DARPA 委託卡內基美隆大學的 Software Engineering Institute（SEI）成立一個協調中心，負責在資安緊急事件中協調各方專家、並協助預防類似事件。這是世界上第一個正式的電腦緊急應變團隊，後來全球各地的 CERT 都是它的後代。換句話說，今天你收到的 CVE 通報、你追的安全公告體系，它的起點就是「有一支善意程式因為一個參數失控，沒有人知道該打電話給誰」。

**第二，《Computer Fraud and Abuse Act》（CFAA，電腦詐欺與濫用法）的首例定罪。** Morris 成為這條法律下第一個被定罪的人（United States v. Morris, 928 F.2d 504, 2d Cir. 1991）。他被判三年緩刑、400 小時社區服務、罰款約 10,050 美元。值得一提的後續：他後來成為 MIT 的教授、共同創辦了 Y Combinator——這個結局本身就是對「把當事人當蠢蛋審判」這種敘事的一記反駁。他不是壞人，也不蠢；他是一個聰明人，在一個資訊不全、沒人想過「自我複製程式會怎樣」的時空裡，把一個防禦用的機率設高了一格。換成你在 1988 年，未必看得出七分之一會炸。

## 故障解剖

### 潛伏條件 vs 觸發

| | 內容 |
|---|---|
| **潛伏條件**（早就裝填好） | `gets()` 無邊界檢查的 fingerd；sendmail 出貨即開 DEBUG 後門；rsh／rexec 傳遞信任＋弱密碼；整片網路高度同質的 BSD Unix（單一栽培）。這些門平時各自「沒事」，但合起來＝一組通殺所有機器的鑰匙。 |
| **觸發**（最後一根稻草） | 傳播控制邏輯裡的**七分之一重感染率**。它把「溫和傳播」變成單機行程爆量（fork bomb）＋網路指數擴散。**不是漏洞觸發了災難，是一個防禦參數的數值。** |

注意這個案子的關鍵翻案點：多數人記得的「漏洞」（緩衝區溢位）是**潛伏條件**，不是觸發；真正的觸發是一個**為了防禦而加、卻設得太強的安全機制**。

### 起司的孔洞——為什麼每一層都沒擋住

| 防禦層 | 為什麼沒擋住 |
|---|---|
| fingerd 的輸入處理 | 用 `gets()`，語言/函式庫層級就不設防——對「輸入不會超長」這個假設零防衛。 |
| sendmail 的遠端介面 | DEBUG 模式預設開啟，等於內建後門；沒人關，因為「方便、應該沒事」。 |
| 主機間的存取控制 | rsh 互信＋弱密碼，橫向移動幾乎無摩擦；信任邊界畫在「連得上的都是自己人」。 |
| 生態的多樣性 | **不存在**。單一栽培讓同一組漏洞通殺；沒有任何異質性當天然防火牆（F14）。 |
| 蠕蟲自己的「不重複感染」防護 | 存在，但被那七分之一的後門條款親手廢掉——**安全機制成了危險源**（F4）。 |
| 全網層級的止血開關 | 不存在。沒有 kill switch、沒有應變中心、沒人知道該打給誰（事後才催生 CERT）。 |

明確反駁「單一根因」：把這場災難歸給「緩衝區溢位」是錯的問法。緩衝區溢位若沒有單一栽培（F14）就只打中一小撮機器；單一栽培若沒有那個七分之一（F4／F8）就只是安靜傳播；安靜傳播若有全網 kill switch 就能及早止血。**是六道孔洞同時對齊**，才有 1988 年 11 月那一夜。

### 對應模式

- **F14（單一栽培／blast radius）**：本章主軸。高度同質的 BSD Unix 讓一組漏洞通殺幾乎所有機器，把單點失效放大到全網。
- **F8（潛伏條件 vs 觸發）**：傳播向量是早就裝填好的彈藥；那個七分之一才是扣扳機的。
- **F4（安全系統即危險源）**：「不重複感染」是一道防禦，但其中對抗詐騙用的七分之一反防禦，正是把蠕蟲變成 DoS 的直接成因——保護機制本身造成了災難。
- **F7（級聯／雪崩）**：兩層自我放大——機器間指數擴散＋單機內行程爆炸。

（與 ch10 Cloudflare 的對照值得記住：F14＋F7 在那裡是「同步放大」（一推全死），在這裡是「自我放大」（自我複製）。同一個 blast radius，兩種引爆方式。）

### 一句話形狀

**這場災難的形狀是：一個聰明人為了讓善意程式不被免疫，加了一道七分之一的反詐騙條款；在一片所有機器長得一模一樣的網路上，這個分數把「估算網路大小」變成了把網路自己拖死的指數級 DoS。**

## 換成你的系統

你活在這個形狀的正中央，而且不只一次。

**第一個藏身處：你的健康探測與背景掃描。** 你那條 FluentBit+Lua pipeline、你的 liveness/readiness probe、你跑對帳的 CronJob——它們的「頻率／間隔／併發數」就是 Morris 的那個七分之一。點名你那個「應該沒事吧」的假設：**「探測很輕量，多探幾次不會有事。」** 這個假設錯的時候會怎麼壞？當被探測的系統開始變慢（尖峰、GC、鎖競爭——你那次 RDS CPU 的老朋友），探測逾時、判它死、重啟它、冷啟動讓它更慢、更容易再逾時——你的監控變成了砍死被監控對象的劊子手。Morris 的機器不是被攻擊死的，是被太多份善意程式拖死的；你的 pod 也可以被自己的 liveness probe 探死。

**第二個藏身處：retry 與重試風暴。** 你複習過 retry storm。它的本質和 Morris 一模一樣：每個重試者各自「理性」（我只是想再試一次），但乘上規模和頻率，重試的流量讓上游更慢，更慢引發更多重試——自我放大的正回授。你的退避（backoff）有沒有抖動（jitter）？有沒有上限？有沒有 circuit breaker 在某個點說「別再試了」？沒有的話，你的重試邏輯裡就藏著一個七分之一。

**第三個藏身處：單一栽培。** 你的 ECS 服務跑的是不是同一個 base image、同一套依賴版本、同一份設定？同質化讓你的部署一致、好維運——但它也意味著一個有問題的 base image、一個有 bug 的共用 library（你那套全團隊採用的多層 cache library 就是候選），一次推上去，**所有實例同時中招**。這不是叫你去搞異質化（那會把維運變地獄），而是叫你誠實回答：當這個共用元件出事，我的 blast radius 是「一個服務」還是「全部」？（ch20 會把這條推到極致：你 `npm install` 的每個套件都是別人的 monoculture。）

可操作的自問，挑一個今天就去查：

- 我每一個對外發請求的背景程式（probe、scan、crawler、retry），它的**最壞情況流量**是多少？（頻率 × 併發 × 失敗時的放大倍數）被它打的系統撐得住嗎？
- 當被探測對象變慢，我的探測邏輯是**幫它**（退避、給它喘息）還是**踩它**（更密集地探、更快地判死重啟）？
- 我有沒有一個「七分之一」——某個為了某種邊角情況加上的參數，它在正常情況下無害，但在規模或壓力下會指數放大？

## 紙上推演

### 推演題1 [15 分鐘] ★★

口頭重講這場災難的因果鏈，但**刻意不准用「緩衝區溢位」當主因**。向一個只聽過「Morris 蠕蟲就是個 exploit」的同事解釋：為什麼真正的主因是一個七分之一的機率參數，而不是任何一個漏洞？把「潛伏條件」和「觸發」明確分開。

#### 推演解答

關鍵在分清彈藥和扳機。傳播向量（fingerd 的 `gets()` 溢位、sendmail DEBUG、rsh 互信）是**潛伏條件**——它們決定蠕蟲怎麼進門、能進多少門。但即使這些門全開，如果蠕蟲老實遵守「已感染就不再感染」，它頂多在每台機器上跑一份、安靜傳播，多半被當成異常流量處理掉。把機器拖死的是**單機內並存的大量副本**，而副本之所以爆量，是因為那個七分之一的重感染率——被很多來源反覆叩門的機器，七分之一乘上叩門次數，累積出 fork bomb。所以：漏洞決定了「能傳多廣」，那個分數決定了「會不會把宿主拖死」。災難等於兩者相乘，而致命的乘數是那個分數。再加一句翻案：那個七分之一還不是 bug，是 Morris **刻意**加的反詐騙防禦——一道安全機制（F4）。

### 推演題2 [20 分鐘] ★★★

**反事實**：假設 1988 年的網際網路不是清一色 BSD Unix，而是異質的——三分之一跑 BSD、三分之一跑某種 System V 變體、三分之一跑完全不同的作業系統，且 fingerd／sendmail 的實作各不相同。其餘條件不變（一樣有那個七分之一）。蠕蟲還會造成全網級災難嗎？這告訴你 F14 在整條因果鏈裡扮演什麼角色？

#### 推演解答

不會造成同等規模。蠕蟲的傳播向量是針對 BSD 特定實作寫的——特定版本 fingerd 的緩衝區佈局、特定 sendmail 的 DEBUG 行為。在異質生態裡，它攻下一台 BSD 後跳到下一台，有約三分之二的機率撞上一台它打不進去的機器。傳播的指數成長被「不相容」一次次截斷，擴散會迅速收斂而非爆炸。那個七分之一造成的單機行程爆量仍會發生在被感染的 BSD 機器上（局部仍會當），但**橫向級聯被生態多樣性當成天然防火牆擋住了**。這說明 F14 的角色：單一栽培不是觸發、也不是進門的方式，它是**放大器**——把「一小撮 BSD 機器當機」放大成「約一成的網際網路停擺」。你系統裡的同質化（同一 base image、同一共用 library）扮演的就是這個放大器：它決定一個缺陷的 blast radius 是局部還是全域。（注意：異質化的代價是維運複雜度暴增——這不是免費的防禦，ch22 會談「冗餘的多樣性」如何權衡。）

### 推演題3 [15 分鐘] ★★

**對帳**：回想你經手過的一次「監控／重試／掃描頻率不當而自我 DoS」的事件（或最接近的一次：liveness probe 探死 pod、retry storm、爬蟲打爆 RDS、CronJob 撞在一起）。把它對應到本章的 F 模式：哪個是你的「七分之一」（被設錯的參數）？哪個是你的「單一栽培」（讓它一次打中所有人的同質性）？你當時是怎麼止血的——有沒有 kill switch，還是像 1988 年那樣靠人手動斷？

#### 推演解答

（個人對帳題，無標準答案；以下是對齊框架。）你的「七分之一」通常是：probe 的 `timeoutSeconds`／`periodSeconds`、重試的次數上限或缺失的 backoff jitter、CronJob 的 schedule（多個 job 撞在整點）、爬蟲的併發度。你的「單一栽培」通常是：同一份設定推到所有實例、同一個共用 client library 的預設值、同一個下游被所有上游同時打。止血方式對照你的成熟度——理想是有 feature flag／kill switch 能即時關掉那個探測或降頻（對應 Cloudflare 的 WAF kill switch）；現實常常是「半夜被叫起來手動 scale down 或改設定重部署」（對應 1988 年的手動止血）。值得在這題誠實寫下：那個參數當初是誰設的、為什麼設成那樣、有沒有人想過它的最壞情況——通常答案是「沿用預設」或「當時覺得應該沒事」，這正是 F8 潛伏條件的生成方式。

### 推演題4 [25 分鐘] ★★★

**設計題**：你要上線一個新的背景掃描服務，定時對全公司數百個微服務發健康檢查並上報。在它變成「自己的 Morris 蠕蟲」之前，你會加哪幾道防線？針對每一道，說清楚它擋住因果鏈的**哪一段**，以及它的代價。

#### 推演解答

對照因果鏈逐段下防線：

1. **退避＋抖動（backoff + jitter）**——擋「被探測對象變慢→探測更密集→踩死它」這段正回授（F7 的核心）。當目標回應變慢或失敗，自動拉長間隔而非縮短，並加隨機抖動避免所有探測撞在同一瞬間（防 thundering herd）。代價：偵測延遲變大，目標真的死了你會晚一點知道。
2. **總流量上限／速率護欄**——擋「頻率 × 併發 × 失敗放大」的乘積失控（對應那個七分之一）。對整個掃描服務設一個全域的每秒請求上限，無論有多少目標、重試多少次都不突破。代價：目標很多時，單一目標的探測頻率被迫降低。
3. **circuit breaker**——擋級聯擴散。某個目標連續失敗就暫時停止探測它（開路），給它喘息，過一陣子再半開試探。代價：開路期間你對那個目標完全沒有可觀測性——要權衡「別踩它」與「別瞎」。
4. **kill switch（feature flag）**——這是 1988 年最缺的那一道。一個能在數秒內全域關掉整個掃描、或把頻率歸零的開關，不需要重新部署。擋的是「災難已經發生、要止血」這段——把「半夜手動斷網」變成「點一下」。代價：幾乎沒有，這是性價比最高的一道（呼應 ch22）。
5. **金絲雀／分批推送設定**——擋單一栽培放大（F14）。掃描頻率／規則的任何變更，先推一小批、觀察、再全推，不要一次把新設定送到所有掃描節點。代價：變更生效變慢。

一句總結：Morris 缺的是第 2 道（那個七分之一沒有上限）和第 4 道（沒有 kill switch），而 F14 讓缺這兩道的後果放大到全網。你的掃描服務只要這五道齊全，同樣的形狀就只會「安全地壞」。

## 自我檢核

口頭自答，講得出來才算過關：

1. 為什麼說這場災難的觸發是「一個機率參數」而不是「一個漏洞」？緩衝區溢位在因果鏈裡扮演什麼角色（潛伏 vs 觸發）？
2. 那個七分之一的重感染率，當初是為了解決什麼問題而加的？為什麼一個**防禦性**的設計反而成了災難的直接成因（這對應哪個 F 模式）？
3. 如果 1988 年的網際網路是異質的（多種作業系統），這場災難的規模會怎麼變？這說明 F14 單一栽培是「進門的方式」「扣扳機的東西」還是「放大器」？
4. 列出至少四道存在的防禦層，並說出每一道**為什麼沒擋住**。為什麼把這場災難歸給「fingerd 緩衝區溢位」是錯的問法？
5. Morris 蠕蟲的「自我放大」和 ch10 Cloudflare regex 的「同步放大」差在哪？兩者的共同 blast radius 機制是什麼？
6. 為什麼說「無傷亡、無破壞 payload」反而讓這場災難更值得後端工程師警惕，而不是更不值得？
7. 你系統裡哪一個背景程式（probe／retry／scan／crawler）的頻率或併發參數，是你的「七分之一」？它的最壞情況流量你算過嗎？
8. 你有沒有一個能在數秒內關掉某個自動探測／重試的 kill switch？如果沒有，下次它失控時你的止血方式是不是就是 1988 年的「手動斷」？
