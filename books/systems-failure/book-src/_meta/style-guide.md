# 寫作風格指南（內部文件，非書籍內容）

本文件是全書的寫作契約。每一章都必須遵守，否則全書會失去一致性。
本書書名（工作名）：《正常意外：複雜系統如何崩壞，以及一個工程師該怕什麼》。

## 這本書是什麼、不是什麼

- **是**：一本「欣賞與直覺」的**工程災難解剖書**，從**資深後端工程師的「焦慮工程師」視角**讀。每章一場真實災難當一篇可被複述因果的個案；目標是讓讀者看懂——大型災難**幾乎從不是單一原因、單一罪魁禍首**，而是**一群各自無害的小東西恰好對齊**的結果；複雜系統的失效不是異常，是系統在緊耦合與互動複雜度下的**內建性質**。讀完有「原來我那次 incident 的形狀，三十年前的核電廠早就演過一遍」「原來『應該沒事吧』就是訃聞的第一句」的通透感，**而且能用自己的話對另一個工程師講清楚「這場災難為什麼會發生、它跟我維運的系統是同一個形狀」**。
- **不是**：
  - 不是災難百科編年帳——不是「某年某處死了多少人」的條列；事件**服務於「失敗模式」的解剖邏輯**，不是為了湊時間軸（時間軸總表進附錄 A）。
  - 不是技術教科書——不教你寫核電廠控制軟體、不教你 SS7、不教你飛控律法；本書講的是「**這個系統為什麼會以沒人預見的方式崩壞、防禦在哪裡破洞、帳單長什麼樣**」，機制只講到能支撐這個判斷的深度。
  - **不是災難獵奇（disaster porn）**——死傷當人寫、不當奇觀；重點永遠是**系統機制**不是血腥細節；不堆煽情形容詞。
  - **不是事後諸葛的審判**——不嘲笑當事人「怎麼這麼蠢」。一律先講清楚他們當年面對的真實約束（資訊不全、生產壓力、介面假設），讓讀者理解「**換成我在那個時空、那個資訊量下，未必看得出來**」。
  - 不是轉職用書——純興趣，不服務 AI Infra 敘事；但工程橋接要讓它讀起來像母語，不是外語。
- **全書貫穿主張（核心商品，終章總收官驗收）**：
  1. **正常意外（Normal Accidents，Perrow）**：在「互動複雜度高 × 緊耦合」的系統裡，嚴重事故不是異常而是**統計上遲早會發生的常態**——不是「有沒有」，是「何時、以哪種對齊方式」。
  2. **瑞士起司（Reason）／多重失效的合謀（Cook）**：災難需要**多層防禦的孔洞同時對齊**；任何一層擋住就沒事。所以「找單一根因」是錯的問法——要找的是**為什麼每一層都沒擋住**。
  3. **安全系統即危險源 ＆ 自動化的諷刺**：用來保護的機制（interlock、MCAS、autopilot、看門狗）本身常常是事故的**直接成因**；自動化拿走的，往往正是操作員在出事時最需要的狀況意識。
  4. **漂移到失效（Dekker）／偏差正常化（normalization of deviance）**：系統不是某天「砰」一下壞掉，而是**一次次僥倖把異常變成常態**，把安全邊界慢慢吃掉，直到某天孔洞對齊。
  5. **焦慮工程師的信條**：「應該沒事吧」是訃聞的第一句。韌性不是「不會壞」，是**安全地壞**（graceful degradation、blast radius 控制、bulkhead）——這正是讀者那套多層 cache library 的 graceful degradation 在做的事。終章把這條從「怕」翻成「怎麼蓋」。

## 失敗模式語言（全書的累積產物——相當於本書的「貨幣」）

本書的招牌不是某一場災難，而是**一套可重用的「失敗模式」詞彙**。每章在 `## 故障解剖` 把該災難對應到下列模式編號；全書累積，終章組裝成完整目錄（附錄 B 是詞彙表）。**各章不得自創新編號或改名，要新增模式必須在 outline 登記。** 基準清單：

| 編號 | 失敗模式 | 一句話 |
|---|---|---|
| F1 | 緊耦合（tight coupling） | 一處失效沒有緩衝、沒有時間餘裕，直接傳導下去 |
| F2 | 互動複雜度（interactive complexity） | 子系統之間有非線性、非預期、設計者沒想到的交互 |
| F3 | 起司孔洞對齊（Swiss-cheese alignment） | 多層防禦各有破洞，平時錯開，出事那天恰好連成一線 |
| F4 | 安全系統即危險源 | 用來保護的機制（interlock/保護律法/看門狗）本身造成事故 |
| F5 | 自動化的諷刺（irony of automation） | 自動化拿走了操作員平時的練習與狀況意識，出事時他接不住 |
| F6 | 偏差正常化／漂移（normalization of deviance / drift） | 僥倖一次次被當成「沒事」，安全邊界被慢慢吃掉 |
| F7 | 級聯／雪崩（cascading failure） | 局部失效觸發連鎖，一路放大到全域 |
| F8 | 潛伏條件 vs 觸發（latent conditions vs trigger） | 災難早已裝填好，觸發事件只是最後一根稻草 |
| F9 | 介面與假設的縫（interface / assumption gap） | 兩個各自正確的子系統，對「單位／契約／前提」的假設不一致 |
| F10 | 單一故障點／單一感測器（SPOF） | 一個點的失效就能拖垮全局，且沒有獨立的交叉驗證 |
| F11 | 監控盲區（the alarm that never fired） | 該叫的警報沒叫、或被淹沒，操作員看到的世界與真實脫節 |
| F12 | 變更即觸發（change as trigger） | 部署、設定更新、程式重用、旗標切換——「動了它」就是觸發 |
| F13 | 生產壓力壓過安全（production pressure） | 進度／成本／發射窗口壓過了「先停下來查清楚」 |
| F14 | 規模放大／單一栽培（blast radius / monoculture） | 同質化＋規模讓單點失效一次打中所有人 |

## 每章骨架（必須遵守，標題名稱照抄——一致性掃描會 grep 這些標題）

1. 章首：`# chNN — 案件名：副標`，接著 blockquote：
   `> **本章解剖什麼**：…（這場災難在全書「失敗模式地圖」的哪裡、主打哪幾個 F 編號模式、跟前後章的對照關係，3–5 行）`
   每個 Part 的第一章在 blockquote 後放**全書 ASCII 路線圖**（基準圖在 outline.md，照抄並把「◄你在這裡」標到本 Part），讀者偏好架構圖先行。
2. `## 從你已知的出發` — 橋接段：把本章的失敗模式錨定在讀者**具體的**工程實戰上（點名場景：你那次 RDS CPU 降 40% 的 deadlock、SQS 冪等消費者、多層 cache 的 graceful degradation、一次有問題的部署/旗標、FluentBit 看板上沒叫的警報…），讓他在讀別人的災難前先「站在自己的系統上」。要具體，不要泛泛。
3. 故事主體（多個 `##` section）。建議節奏：**冷開場**（把讀者丟進事發那一刻的場景）→ **裝填**（潛伏條件：這套系統平時怎麼運作、哪些邊界早就被吃掉）→ **觸發**（那根稻草）→ **連鎖**（孔洞怎麼一個個被穿透、失效怎麼傳導）→ **餘波**（傷亡規模、調查、改了什麼）。敘事要好看，但**每個事實照 landscape，不憑記憶**；時序與機制要精確到工程師讀得懂「到底哪個變數、哪一行、哪個假設」。
4. `## 故障解剖` — **本書招牌段落（每章必有，名稱照抄）**：把這場災難拆成失敗模式語言。必含：
   - **潛伏條件 vs 觸發**：災難早就裝填好的是什麼，最後那根稻草是什麼。
   - **起司的孔洞**：列出存在的每一層防禦，以及它各自**為什麼沒擋住**（用條列或表格）。明確反駁「單一根因」。
   - **對應模式**：本章命中哪幾個 F 編號（引基準表），為什麼。
   - 一句話總結：「**這場災難的形狀是：＿＿**」。
5. `## 換成你的系統` — **橋接回去（每章必有，名稱照抄）**：同一個失敗形狀，**藏在讀者自己的後端／分散式世界的哪裡**？點名一個他「應該沒事吧」的假設，並說清楚那個假設錯的時候會怎麼壞（呼應焦慮工程師：命名具體的未處理失敗案例）。給一個可操作的自問或檢查點，不要只喊「要小心」。
6. `## 紙上推演` — 動手做的形式（純敘事書，無可執行 lab）。每章 2~4 題，標注預估時間（如 **[15 分鐘]**）與難度（★~★★★）。題型：**口頭重講**這場災難的因果鏈、**反事實**（如果某一層起司沒破洞會怎樣／如果沒有那個安全系統會不會反而沒事）、**找出**一個流行說法的史實破綻、**對帳**（把你維運過的某次 incident 對應到本書某個 F 模式）、**設計題**（你會在哪裡加哪一層防禦、它的代價是什麼）。題目用 `### 推演題N`，解答緊接 `#### 推演解答`（全書統一），給完整解答與討論。
7. `## 自我檢核` — 5~8 個問題，**口頭自答**：講得出來才算過關。優先出「為什麼每一層都沒擋住」「如果不這樣會怎樣」而不是「死了幾個人／哪一年」。
8. `## 延伸閱讀` — 真實連結（以 `_meta/landscape-2026-06.md` 驗證過的為優先），每條一句話說明為什麼值得讀。優先：**官方調查報告**（NTSB／CAIB／Rogers Commission／INSAG／NIST／SEC／FDA／Ariane Inquiry Board）、當事領域的經典（Perrow、Reason、Cook、Dekker、Leveson、Vaughan）、可信的事故技術分析。

## 深度標準

- **每章一個被講透的個案＝該災難本身**：把一場災難攤開到讀者能**複述其因果鏈**——具體的人、時間、技術約束、那個決策的代價、防禦為什麼逐層失效。要有細節（哪個變數溢位、哪個閥門卡住、哪通電話誰說了什麼），不准只給「於是就爆炸了」。outline 已為每章指定主打的 F 模式與必講的關鍵細節。
- **史實照 landscape，不准憑記憶**：日期、死傷、金額、技術根因、調查結論一律以 `_meta/landscape-2026-06.md` 為準；landscape 沒有的自行查證（每章 **3~8 次** WebSearch/WebFetch）並 hedge。
- **數字一律 hedge＋標來源**：死傷、金額（如 Knight 的 4.4 億）、影響規模（如 CrowdStrike 的機器數）這類數字最常被以訛傳訛——標為估計或給範圍、註明來源；landscape 標 ⚠️ 的更要保守。
- **迷思必須校正**：本書的可信度與趣味來自「大家都記錯的事」。凡 landscape 標出的流行誤解，**必須引用查證版而非流行版並明白點破**——例如：Tacoma Narrows **不是**單純共振而是 aeroelastic flutter；Knight Capital **不是**「一行 code」這麼簡單；「都是人為疏失」是 root-cause 迷思；「圖靈/某天才一個人」式的歸因。
- **每章必破「單一根因」的衝動**：明確指出至少兩到三層**各自獨立**的防禦失效。把「人為疏失」翻譯成「是什麼系統條件讓這個人在那一刻幾乎不可能做對」。
- **每章至少一處「工程師的領悟」**：本書核心商品是通透感與一種清醒的敬畏。每章至少一處回答「**這場災難對今天維運分散式系統的你意味著什麼**」，寫到讀者能用自己的話轉述，不是堆形容詞。
- **時效性誠實 hedge**：談近年事故（CrowdStrike 2024、SolarWinds、Log4Shell）標時點 `（2026-06）`，區分「調查已定論」與「仍有爭議／仍在訴訟」。
- 寧深勿廣，不灌水。每章 5,000~9,000 中文字，依案件深度調整。

## 事實正確性

- 一切硬事實（日期、死傷、金額、規模、技術根因、調查結論、連結）以 `_meta/landscape-2026-06.md` 為事實基準；本章特定細節自行查證（每章 3~8 次）。
- 與 landscape 衝突時：**以 landscape 為準**；若你查到 landscape 可能有誤，**在章末用註記標出疑點**（「⚠️ 待協調者複核：landscape 說 X，但來源 Y 說 Z」），不要自行改寫 landscape（一致性由協調者 P3/P4 統一處理；重大修正需兩個獨立來源）。
- 死傷／金額／影響規模類數字一律標估計或給範圍並 hedge。
- 延伸閱讀只放真實存在的連結；沒把握標「（未驗證）」。

## 格式

- Markdown。主要層級 `##`，次層 `###`。
- 圖：ASCII diagram，必須通過 `check_diagrams.py`（**CJK=2 欄寬規則**——中文字佔 2 欄、ASCII 佔 1 欄，框線要對齊）。本書圖的類型：
  - **全書失敗模式地圖**（Part 首章，基準圖在 outline.md）。
  - **瑞士起司圖**（多層防禦＋孔洞對齊，每章 `## 故障解剖` 視需要）。
  - **連鎖時序圖／級聯傳導圖**（觸發→連鎖，用 `→ ↓ │ ─`）。
  - **時間線片段**（裝填→觸發→餘波的關鍵時刻）。
  - 不追求精確比例，追求**因果關係清楚**。
- 跨章引用格式：`（見 chNN）`，不要寫死檔名路徑。跨章主題點到為止＋指向擁有它的章。

## 語氣

工程師對工程師。冷靜、直接、有觀點，底色是**焦慮工程師**：先講失敗模式再講全貌、把「應該沒事吧」的假設一個個點名拆穿。但克制——**面對死亡與傷者保持尊重**，不獵奇、不煽情、不喊「太慘了」式空洞感嘆；通透感與敬畏要用內容本身造成。允許冷幽默（針對系統與迷思，不針對死者）。歷史人物當人寫，不造神也不踩低：講清楚他們當年的真實約束，讓讀者理解「換成我也未必看得到」。對「這次不一樣」的技術樂觀保持清醒——看過正常意外反覆上演的人，知道下一個孔洞正在某處慢慢對齊。
