## 從你已知的出發

打開你某個專案的 repo，裡面大概有一個 `infra/` 或 `cdk/` 目錄，或者一份 CloudFormation 的 YAML。那份檔案裡，你**宣告**了一整套基礎設施：一個 VPC、幾個 subnet、一個 ECS service 跑你的 NestJS 容器、一個 RDS 的 PostgreSQL 實例、一條 SQS queue、一個 ALB 在前面分流、幾條 IAM policy 把權限綁好。你 `git push`，CI 跑起來，幾分鐘後——一整座資料中心的縮影就在 AWS 某個 region 裡長出來了。你沒有打電話給任何人、沒有簽採購單、沒有走進任何一棟掛著「機房重地」牌子的房間、沒有碰過一條網路線。

退一步看這件事有多荒謬。你剛剛用一份**文字檔**，調度了實體的伺服器、網路、儲存；而且這份文字檔在 Git 裡，可以 code review、可以 `git revert`、出事時可以 `git blame` 找出是哪次 commit 把 security group 開錯了。**你的基礎設施是程式碼。** 這句話今天聽起來像廢話，但它是這一章要拆開的整個世界觀——而它的歷史比你想的短得多，短到你職業生涯的長度就裝得下大半。

再看你天天踩的幾個坑，它們是同一個世界觀的洩漏帳單：

**第一個場景：Lambda 冷啟動。** 你把某個輕量的 webhook handler 做成 Lambda，平常沒流量。某天它被觸發，你看 trace，發現第一次呼叫慢了好幾百毫秒甚至上秒——後面的呼叫又正常了。你知道這叫 cold start：那個「無伺服器」的函式，底下其實**還是有伺服器**，只是平常被收走了；要跑你的碼，AWS 得先變出一個執行環境、把你的程式塞進去、初始化 runtime（如果是 JVM 那更慢），這段時間就漏了出來變成延遲。你用了一個叫「serverless」的抽象，而它在你最不希望的時候，把底下那台你以為不存在的伺服器**漏給你看**。

**第二個場景：供應商相依。** 你的系統用了 SQS、用了 DynamoDB、用了 Lambda、用了 CloudFormation。這些東西沒有一個是標準。哪天你想搬去另一家雲，你會發現你綁進去的不只是「一個訊息佇列」，而是「**AWS 的**訊息佇列、它特定的語意、它特定的 IAM 模型、它特定的計費方式」。你享受了 managed service 的方便，代價是你的系統長進了這家供應商的形狀裡——這叫 vendor lock-in，是雲抽象最安靜、也最貴的一筆帳。

**第三個場景：帳單字面意義上的帳單。** 月底 AWS 的帳單來了，比預期高。你去查，可能是某個忘了關的環境、某條跨 region 的流量、某個 NAT Gateway 默默計費、某個 S3 bucket 的 request 數量爆掉。雲把「擁有機房」的固定成本，換成了**按用量計費的浮動成本**——這在你需要彈性時是恩賜，在你沒看好時是個會自己長大的數字。

這三件事——冷啟動、鎖定、成本失控——不是 AWS 做壞了什麼。它們是你跟一個四十年來逐步成形的轉變簽的合約裡，寫在最下面那行小字。本章告訴你這份合約是怎麼簽成的：機器怎麼學會假裝自己是很多台、亞馬遜怎麼把自己的機房變成一行 API、以及一群工程師怎麼決定連基礎設施都要拉進版本控制裡管。

## 一台機器假裝成很多台：虛擬化的前史

雲不是憑空出現的。它底下站著一個更老的點子，老到 1960 年代——**虛擬化**。要懂雲為什麼能「按片租用伺服器」，得先懂機器是怎麼學會假裝的。

回到問題的源頭。1960 年代的大型主機（mainframe）貴得驚人，一台機器是一整個機構的核心資產，而它一次只能讓一個程式（或一批工作）獨佔。這浪費得心痛：機器在等 I/O 的時候閒著、一個使用者在用的時候別人只能排隊。IBM 的 Cambridge Scientific Center 想了個辦法——既然實體只有一台，那就讓它**假裝**成很多台。他們做的 **CP/CMS** 系統（CP-40 在 1960 年代中、CP-67 在 1967 年起進入日常生產使用）做到了一件當年很科幻的事：每個使用者登入，都得到一台**看起來完全屬於自己的虛擬 System/360**——有自己的記憶體、自己的磁碟、自己的作業系統，彼此互不干擾，但底下共享同一台實體機器。

停下來體會這個抽象。它在解什麼複雜度？**「一台昂貴的實體機器，怎麼安全地給很多人同時用」**——解法是**切出一台台假機器**，每台假機器裡的程式都以為自己獨佔了整台電腦。關鍵性質是它**透明**：程式不需要改寫、不需要知道自己跑在虛擬機裡，它對著一個跟真機器一模一樣的介面說話。這正是抽象的標準動作——藏起共享的真相，給上面一個獨佔的假象。

這個點子在大型主機世界活了幾十年，然後在 1990 年代末跳進了便宜得多的 x86 世界。**VMware**（1998 年成立、1999 年推出 Workstation）做成了一件當年被認為很難的事——x86 這架構當初根本沒考慮過要被虛擬化（有些指令讓「假裝」很難做得乾淨），VMware 用一套巧妙技巧繞了過去。2003 年劍橋的一群人發表 **Xen**（論文「Xen and the Art of Virtualization」）走另一條路——**paravirtualization**：不假裝得那麼徹底，而是要求客體作業系統「知道」自己在虛擬環境裡、配合著改一點，換來更好的效能。

為什麼這段前史對雲是必要的？因為雲的整個商業模式，建立在虛擬化這個能力上。亞馬遜不可能為每個租用 EC2 的客戶準備一台實體機器——那回到了「擁有機房」的老問題。雲能成立，正是因為一台實體伺服器可以**切成很多台虛擬伺服器**，賣給很多個互不相識的客戶，每個客戶都以為（在抽象層面也確實如此）自己租到了一台機器。**虛擬化把「一台機器」這個單位，從實體變成了可以任意切分、配置、計費的東西。** 沒有這一步，後面「按片租用伺服器」根本無從談起。

```text
        虛擬化：一台實體機器 → 很多台「假機器」

        ┌─────────┬─────────┬─────────┬─────────┐
        │ 虛擬機A │ 虛擬機B │ 虛擬機C │ 虛擬機D │   每個租戶以為自己獨佔一台
        ├─────────┴─────────┴─────────┴─────────┤
        │         Hypervisor（管理層）          │   負責「假裝」與隔離
        ├───────────────────────────────────────┤
        │            一台實體伺服器             │   真相：大家共享
        └───────────────────────────────────────┘

        抽象的標準動作：藏起共享的真相，給上面獨佔的假象
```

## 2006：亞馬遜把自己的機房變成一行 API

現在來到本章的個案——軟體史上最大的一次「把複雜度外包」的轉折。它不是某個實驗室裡的學術突破，而是一家網路書店為了解決自己內部的痛，順手把解法賣給了全世界。

**先講那個痛。** 2000 年代初，亞馬遜快速擴張，工程團隊一個接一個冒出來。但每個團隊在做任何真正有價值的事之前，都得先重蓋一套同樣的地基：要儲存就自己架儲存、要算就自己張羅運算、要對外服務就自己處理網路。Jeff Bezos 和經理們給這件事取了個傳神的名字——**「undifferentiated heavy lifting」**（沒有差異化的粗重活），抱怨它產出的是「**muck**」（爛泥、雜事）。據當年的說法（Bezos 在 2006 年的一場 MIT 演講等場合反覆講過），工程團隊把大約**七成**的時間、精力和金錢，花在這些任何專案都需要、卻無法讓你跟對手拉開差距的後端基礎工作上。

> ⚠️ 那個「70%」是 Bezos 演講裡反覆引用的說法，屬於他對內部觀察的概括，不是嚴格量測的統計數字；本書照他的措辭引用，但你應該把它讀成「絕大部分」的生動說法，而非精確比例。

這個觀察是整個雲的種子。如果**每一家**公司、每一個團隊都在重蓋同樣的爛泥地基，那麼有沒有可能——讓一家把這套地基做到極致的公司，把它的地基**當成服務開放出來**，讓所有人不必再各蓋一套？亞馬遜的內部已經被逼著在做這件事了：為了讓自家數百個團隊不要各自為政，他們早就開始把內部基礎設施標準化、自動化、用 API 封裝起來（這條「一切都是服務、團隊之間只透過 API 對話」的內部紀律，本身就是後來微服務的前身，見 ch17）。2003 年，Benjamin Black 和 Chris Pinkham 寫了一份給 Bezos 的文件，描繪一個願景：把亞馬遜的基礎設施徹底標準化、自動化，甚至——順帶一句——**可以把虛擬伺服器當成服務賣出去**。那句順帶的話，後來變成了一個比賣書大得多的生意。

**然後是那一跳。** 2006 年，Amazon Web Services 把內部地基對外開放成 API：

- **S3（Simple Storage Service，2006 年 3 月）**：儲存變成一個 API。你不再需要買磁碟、組 RAID、煩惱備援與容量規劃——你 `PUT` 一個物件進去、`GET` 出來，按你存了多少、傳了多少付費。
- **SQS（Simple Queue Service，2006 年稍後進入生產）**：訊息佇列變成一個 API——這正是你今天結算 pipeline 裡那條 queue 的祖宗。
- **EC2（Elastic Compute Cloud，2006 年 8 月公測）**：**運算**變成一個 API。這是最關鍵的一塊。你打一個 API 呼叫，幾分鐘後就有一台（虛擬）伺服器在跑，按小時計費；不用了就 `terminate`，計費停止。「買一台伺服器」這個動作——過去要採購、要等貨、要進機房、要接線、要設定，前後可能好幾週——被壓縮成**一行 API 呼叫、幾分鐘、按小時付錢**。

把這一跳的本質講白：**亞馬遜把「擁有並維運一座機房」這個巨大的複雜度，封裝成了一行 API 呼叫，然後按用量收費。** 這跟整本書的主旋律是同一件事——抽象把複雜度搬家，只是這次搬走的不是某個語言特性、某層程式結構，而是**整座資料中心**。你呼叫 `RunInstances`，底下那一整套「買機器、進機房、接網路、拉電力、換硬碟、做備援」的爛泥，全被搬去亞馬遜那邊、被它的虛擬化與維運團隊扛走，你看到的只剩一個乾淨的 API。

順帶一提詞本身的身世：「cloud computing」也是 2006 年前後才進入主流——Google 的 Eric Schmidt 在 2006 年 8 月一場搜尋引擎研討會（SES San Jose）上用它描述 Google 的服務，常被當作這詞普及的起點；但「誰最先造這詞」其實沒有定論（1997 年 NetCentric 公司就試圖註冊它的商標、後來放棄）。重點不在誰造詞，在 2006 這個時間點，雲從學術名詞變成了一門你訂得到、付得起的生意。

## CapEx → OpEx：所有權變租用，這不只是會計題

雲最常被講的好處是「省錢」，但這太淺，會讓你錯過它真正改變的東西：雲改變的不是成本的**多少**，是成本的**形狀**——而成本的形狀，反過來決定你能蓋什麼、敢蓋什麼。

用你熟悉的兩個詞來框：**CapEx（capital expenditure，資本支出）**與 **OpEx（operational expenditure，營運支出）**。

在雲之前，要上線一個服務，你得先**擁有**伺服器。擁有意味著一筆**資本支出**：估算尖峰要多少機器、一次把錢花下去買齊，加上機房、電力、冷氣、維運人力。這筆錢有兩個討厭的性質。第一，它是**預付**的——你得在賺到任何一塊錢之前，先把未來尖峰的硬體買好。第二，它逼你**賭一個容量數字**：買少了尖峰會掛、買多了大部分時間機器在曬太陽。而你幾乎一定買多——因為「服務掛掉」比「機器閒置」可怕得多，理性的選擇就是超買、長期養著一堆只為那幾天尖峰存在的閒置產能。

雲把這筆資本支出換成了**營運支出**：你不擁有機器，你**租用**；你不預付，你**按用量付**；你不賭容量，你**要多少開多少**。需求高峰來了就多開幾台 EC2（這就是你天天用的 autoscaling／HPA 背後的經濟學前提），峰過了就關掉、計費停止。「擁有」變成「租用」，固定成本變成浮動成本，預付的賭注變成隨需的水龍頭。

這個轉變的真正威力，不在帳面上省了多少，而在它**消滅了一道進入門檻**。在 CapEx 的世界裡，一個三人的新創想驗證一個點子，得先有一筆錢買機器、有人會架機房——光是地基就足以勸退大半數人。在 OpEx 的世界裡，同一個新創刷一張信用卡、`git push` 一份 CloudFormation，就有了跟大公司同一座資料中心等級的基礎設施，先付一個月幾十塊美金。**雲把「擁有基礎設施」這件曾經屬於有錢機構的特權，變成了按月付費的公共設施**——像你不會為了用電去自己蓋發電廠，你也不再為了上線一個服務去自己蓋機房。這是雲在商業上比技術上更深的影響：它重寫了「誰能蓋一個能撐住規模的系統」這個問題的答案。

但天下沒有白吃的午餐，這個轉變的帳單我們留到本章末的「複雜度搬去哪了」一起結。先看雲分層的另一個維度——你到底外包了多少。

## IaaS / PaaS / SaaS：你不用管的東西，一層層往上推

「把機房外包」不是一個全有全無的開關，而是一道**光譜**。你可以只外包最底層的硬體、自己管上面所有東西；也可以外包到連程式都不用寫、直接用人家做好的軟體。業界把這道光譜切成三層——IaaS、PaaS、SaaS——它們的分界，就是「**你不用管什麼**」這條線往上移了多少。（這套分類學大約在 2006–2011 年間成形，NIST 在 2011 年的雲定義文件 SP 800-145 把它正式定下來。）

用一個你會的方式理解這三層：把「跑一個服務需要操心的東西」從下到上疊成一疊——機房/硬體、虛擬化、作業系統、runtime、你的應用程式、你的資料——然後看每一層幫你接手到哪。

```text
        誰來管？（▓=供應商管  ░=你管）

                  自己機房   IaaS      PaaS      SaaS
        應用資料    ░         ░         ░         ▓
        應用程式    ░         ░         ░         ▓
        runtime     ░         ░         ▓         ▓
        作業系統    ░         ▓         ▓         ▓
        虛擬化      ░         ▓         ▓         ▓
        伺服器      ░         ▓         ▓         ▓
        儲存/網路   ░         ▓         ▓         ▓
        機房/電力   ░         ▓         ▓         ▓

        「你不用管」的線，一層層往上爬
```

- **IaaS（Infrastructure as a Service）**：供應商給你**虛擬化的硬體**——伺服器、儲存、網路。你拿到一台（虛擬）機器，作業系統以上全是你的事：自己裝 OS、裝 runtime、部署、打 patch、管監控。**EC2 是 IaaS 的原型**。它外包掉「買機器、進機房、接網路」這最底下的爛泥，但把機器交到你手上之後就放手了——給你最大的控制權，也留給你最多的活。

- **PaaS（Platform as a Service）**：再往上接一層——連作業系統、runtime、自動擴展、負載平衡都幫你弄好，你只管**丟程式進去**。你不再煩惱「OS 要不要升級」「要不要多開幾台」，推上一份程式碼，平台負責讓它跑起來、活著、撐得住流量。你的 **Lambda 是 PaaS 光譜上更極端的一種**——連「有幾台機器」都對你隱形了（這種形態業界叫 serverless，下一段單獨拆）。PaaS 外包掉「維運一個執行平台」的複雜度，代價是你被綁進這平台的形狀與限制裡。

- **SaaS（Software as a Service）**：最上面一層——連程式都不用你寫，直接用人家做好的軟體（你公司用的 Slack、Datadog 都是）。你管的只剩「放進去的資料」與設定。SaaS 把整個「自己做一套軟體」的複雜度都外包了，代價是你只能用人家給的功能、按人家的規則玩。

看出這道光譜的內在邏輯了嗎？**每往上一層，你外包掉一層複雜度、放棄一份控制權。** 這正是複雜度守恆在雲分層上的具體形狀：複雜度沒有消失，它被你**往上推給了供應商**，你交換到的是「不用管那一層」的輕鬆與交出控制權的代價。你不是在「消滅」維運作業系統的複雜度，是在決定「這層爛泥我要不要自己扛」。一個團隊選哪一格，本質是在回答：**我的差異化價值在哪一層以上？** 那層以下的，能外包就外包——因為那是 Bezos 說的 muck，扛它不會讓你贏。

值得單獨點出 serverless（你的 Lambda）這一格——它是這道光譜上「外包」推到極致的一種，把「伺服器」這個概念整個從你眼前移走：不配置機器、不管容量、沒流量時不付錢、流量來了自動長出執行環境。但這就是你開頭踩過的冷啟動的根：**「無伺服器」從來不是真的沒有伺服器，是伺服器被藏到了一個抽象後面**——而抽象，照本書一路講下來的鐵律，在邊緣總會漏。你享受了「沒流量不付錢」，就得接受「久沒流量第一發會慢」。這不是 bug，是這格抽象的定價（完整的帳單留到紙上推演拆）。

## Infrastructure as Code：把基礎設施拉回軟體工程

到這裡，雲已經把機房外包成了 API。但這留下一個新問題，而這個問題的答案，是本章標題的另一半。

問題是這樣的：雲給了你 API，但**人怎麼用這些 API**？最直覺的方式是——點。打開 AWS 的 web console，滑鼠點一點：開一台 EC2、建一個 security group、設一條 routing。能跑，但你很快會撞上一面熟悉的牆，而且這面牆你在前面好幾章都見過。

**手動點擊的痛，是「不可重現」。** 你在 console 上點了五十下、調好了一套能跑的環境。現在問你三個問題：第一，你能在另一個 region、另一個帳號，**一模一樣**地再點一次嗎？（你記得每一步嗎？）第二，三個月後出事了，你能查出「這套環境到底是怎麼變成現在這樣的、是誰在哪一天改了哪個設定」嗎？（console 不會幫你 `git blame`。）第三，你能讓另一個工程師 **review** 你要做的基礎設施變更、在它生效之前指出「你這個 security group 開太大了」嗎？（點擊沒有 diff、沒有 PR。）

這三個問題的答案在手動點擊的世界裡都是「不能」，而它們指向同一個診斷：**基礎設施的狀態，活在 console 裡、活在某個工程師的肌肉記憶裡、活在一堆沒人完整知道的點擊歷史裡——它沒有被當成軟體來管。** 而我們這本書講開源與 Git 時就講過（見 ch14），人類治理「多人協作、需要可重現、可審查、可回溯」這類複雜度，早就有一套成熟的工具：**版本控制下的程式碼**。

**Infrastructure as Code（IaC，基礎設施即程式碼）的核心主張只有一句：別再用滑鼠點基礎設施，把它寫成程式碼、放進 Git。** 你不在 console 上點出一台 EC2，你寫一段文字「我要一台這樣的 EC2」，提交進版本控制，讓一個工具去讀這段文字、把真實的基礎設施變成它描述的樣子。一旦基礎設施變成了文字檔，Git 那整套協作機器（見 ch14）就全部適用了：

| 手動點擊的世界 | IaC 的世界 |
|---|---|
| 環境怎麼來的——沒人完整知道 | 環境是一份檔案，讀檔就知道 |
| 重建環境——靠記憶重點一遍，幾乎一定不一樣 | 重建環境——跑同一份檔案，可重現 |
| 誰改了什麼——查不到 | `git log` / `git blame`，可回溯 |
| 變更要不要 review——口頭講講 | 開 PR、看 diff、code review，可審查 |
| 改錯了要復原——憑印象點回去 | `git revert`，可回滾 |

這條把基礎設施程式碼化的路比雲本身還老（「自動化伺服器設定」的需求在雲之前就有了），工具譜系是這樣長出來的：

- **CFEngine（1993）**：最早把「伺服器該長什麼樣」寫成宣告式設定、讓工具去**收斂**到那個狀態。它解的是 config drift——「一堆伺服器手動設定到後來各自漂移、沒人知道哪台跟哪台不一樣」的痛。
- **Puppet（2005）**、**Chef（2009）**：把這條路發揚光大，讓你用程式碼描述整個 fleet 的設定狀態。Puppet 偏宣告式（你說「要長成什麼樣」）、Chef 偏程序式（你寫「該怎麼一步步弄成那樣」）——這個「宣告 vs 程序」的張力等下會回來。
- **Terraform（2014，HashiCorp）**：把重心從「設定既有的機器」移到「**provision 基礎設施本身**」——用宣告式設定描述「我要哪些雲資源（機器、網路、資料庫、queue……）」，讓工具呼叫雲的 API 把它們變出來、並追蹤狀態。這正好接上了雲：既然雲把一切都變成了 API，就用程式碼去驅動這些 API。你用的 CloudFormation 是 AWS 自家的同類工具，CDK 則讓你用 TypeScript 寫它。

注意 IaC 裡反覆出現的一個關鍵字——**宣告式（declarative）**。你寫的不是「先開機器、再裝這個、再設那個」的一步步腳本，而是「**我要的最終狀態長這樣**」的描述，工具負責算出「從現在到那個狀態之間要做哪些事」。這你應該覺得眼熟——這跟 Codd 讓你用 SQL 宣告「我要什麼資料」而不是手寫存取路徑（見 ch13），是同一種抽象的衝動：**把「怎麼做」的複雜度，從你身上搬給引擎。** SQL 把它搬給查詢最佳化器，IaC 把它搬給 plan-and-apply 引擎。你說目標，引擎算路徑。這個血脈會在 Kubernetes 再出現一次（見 ch17）——雲時代治理複雜度的招數，反覆是同一個：**讓你描述要什麼，把怎麼做藏進引擎。**

順帶留一個對帳給紙上推演：你的 CloudFormation／CDK，某種意義上是「先設計、先把契約定下來」那個瀑布基因（見 ch11）在雲時代的合理回歸——只是這次的「設計」是活的、版本化的、可快速迭代的，不是 Royce 被誤讀成的那種一次定死的純線性。

## 複雜度搬去哪了

把這一章的帳結出來。雲與 IaC 這一跳，消滅了什麼負擔、又製造了什麼新負擔？

**它搬走的：** 「擁有並維運實體基礎設施」這整包複雜度，被外包給了雲供應商——買機器、進機房、接網路、容量規劃、硬體維護、備援，全都不再是你的事。資本支出變營運支出，所有權變租用，預付的容量賭注變隨需的水龍頭。手動點擊的不可重現，被 IaC 換成了版本控制下可重現、可審查、可回溯的程式碼。一個三人團隊因此能調度過去只有大機構養得起的基礎設施——這是雲最真實的禮物。

**它製造的（帳單在抽象洩漏時回來找你）：**

| 新負擔 | 它怎麼洩漏／反噬 | 一句話 |
|---|---|---|
| **抽象洩漏：底下還是有伺服器** | 冷啟動（serverless 的執行環境要現長）、底層硬體/網路的限制偶爾穿透上來 | 「無伺服器」不是沒伺服器，是伺服器被藏起來——藏不住時漏成延遲 |
| **網路分區與分散式現實** | 你的元件現在跨網路對話，網路會分區、會延遲、會丟包；「別人的機房」的故障你管不到也修不了 | 你把機房外包了，但沒外包掉分散式系統的物理（這是 ch17 的主場，見 ch17） |
| **供應商鎖定（vendor lock-in）** | 用了愈多 managed service，系統就長得愈像這家供應商的形狀，搬家成本愈高 | managed service 的方便，是用「綁進這家供應商」換來的 |
| **成本失控** | 按用量計費＝一個會自己長大的數字；忘了關的資源、沒看好的流量、誤設的規模，月底變成驚嚇 | OpEx 的彈性，反面是失控的彈性——省錢的工具也是燒錢的工具 |
| **信任問題：別人的電腦** | 你的資料、你的運算，跑在一家你看不見的公司的機器上——它的安全、它的可用性、它的營運決策，你只能信任，不能驗證 | 雲是「rent someone else's computer」，便利的代價是把信任外包 |

**這一跳的帳單，一句話：你把「擁有機房」的巨大複雜度搬給了別人，換來的是把自己的命運綁在別人的機器、別人的 API、別人的帳單上——而那台機器，你永遠看不見、控制不了、只能信任。** 雲沒有消滅基礎設施的複雜度，它把複雜度搬到了別人的資料中心，並在合約最下面那行小字裡，寫好了它漏回來找你的所有方式：冷啟動、分區、鎖定、帳單。這是複雜度守恆在這一章的形狀——也是你每天在 AWS 帳號裡，親手簽收的那份帳。

## 紙上推演

### 推演題一：用 IaaS/PaaS/SaaS 分層拆解你自己的系統 **[20 分鐘] ★★**

挑你維運過的一個真實系統（例如那套 ECS 上的 NestJS 微服務）。把它用到的每個元件，標到 IaaS/PaaS/SaaS 光譜上——哪些是你「拿到機器之後全自己管」（IaaS）、哪些是「丟程式進去就好」（PaaS/serverless）、哪些是「直接用人家做好的」（SaaS）。然後回答一個問題：**你把哪幾層複雜度外包了、自己刻意留下哪幾層，為什麼？** 試著用「我的差異化價值在哪一層以上」這個框架解釋你的選擇。

#### 推演解答

一個典型的拆法可能長這樣：你的 NestJS 容器跑在 **ECS** 上——ECS 幫你排程容器、管底層機器（如果用 Fargate 更是連機器都不管），這比純 EC2（IaaS）往 PaaS 移了一格，你不用自己裝 OS、打 patch，但你還掌握容器內的一切。你的 **RDS**（PostgreSQL）是 managed database——AWS 幫你管備份、patch、failover，你只管 schema 與 query；這是 PaaS 思維（外包掉「維運一個資料庫」的複雜度，但你仍寫自己的 SQL、調自己的 index）。你的 **SQS** 是 managed queue（同上）。你的 webhook handler 做成 **Lambda**＝serverless，外包到極致。而你監控用的 **Datadog/CloudWatch dashboard**、團隊溝通用的 **Slack**＝SaaS，你連程式都不寫。

關鍵的領悟在「為什麼這樣選」。你**沒有**自己架資料庫引擎、沒有自己寫訊息佇列、沒有自己做監控 SaaS——因為那些是 muck，你扛它不會讓你的 B2B 資料中介比對手強。你刻意自己留下的，是**業務邏輯那一層**（你的 NestJS service、你的 cache 策略、你的結算 pipeline 邏輯）——因為那才是你的差異化價值所在。這就是這道光譜的決策法則：**在你的差異化價值那一層以下，能外包就外包；在那一層以上，自己掌握。** 你做的每一個「用 managed service 還是自己刻」的決定，本質都是在回答「這層是不是 muck」。

### 推演題二：用「複雜度搬家」解釋 serverless 把什麼搬走、帳單是什麼 **[15 分鐘] ★★**

口頭（或寫下）向一個資淺工程師解釋：Lambda（serverless）相較於你自己在 EC2 上跑一個常駐 process，**把什麼複雜度搬走了**？而那筆搬走的複雜度，**以什麼形式漏回來變成帳單**？至少講出兩筆帳單。

#### 推演解答

**搬走的：** 在 EC2 上跑常駐 process，你得自己操心一整串東西——這台機器夠不夠力、要不要多開幾台應付尖峰、process 掛了誰把它拉起來、沒流量時這台機器還在燒錢、OS 要不要 patch。serverless 把這些全搬走了：容量規劃不用了（它自動長）、高可用不用你弄（它跨 AZ）、沒流量不付錢（它收掉執行環境）、機器維運消失（你根本看不到機器）。它把「擁有並照顧一個執行環境」的複雜度，徹底搬給了平台。

**漏回來的帳單（至少兩筆）：**

1. **冷啟動。** 它把機器藏起來的代價，是當機器需要為你重新長出來時（久沒呼叫、或突然要擴展），初始化的時間漏成了延遲。你拿「沒流量不付錢」換「久沒流量第一發慢」。
2. **可觀測性與除錯變難。** 沒有一台你能 SSH 進去的機器，沒有一個常駐 process 能掛 profiler、能看本機狀態。出事時你只剩 log 和 trace，要重現一個只在特定併發/冷啟動下出現的問題，比在一台你掌控的機器上難得多。
3.（加分）**執行模型的限制漏出來：** 最長執行時間、記憶體上限、無狀態假設、併發模型——這些平台的限制會在你的負載踩到邊界時穿透上來，逼你把程式寫成平台喜歡的形狀。

核心句：**serverless 不是消滅了「管伺服器」的複雜度，是把它搬到了平台後面——而抽象在邊緣總會漏，於是它以冷啟動、以除錯困難、以執行限制的形式回來找你。**

### 推演題三：你的 CloudFormation 算不算瀑布基因的回歸？ **[15 分鐘] ★★★**

ch11 講過瀑布「先把設計/契約完整定下來再動工」的基因（被 Royce 誤讀成純線性，但「先設計」本身在對的場景是合理的）。你寫 IaC（CloudFormation/CDK）時，是先把整套基礎設施**寫成一份完整規格**、再 apply 出去。這算不算那個「先設計」基因的回歸？它跟被誤讀的純瀑布，差在哪？

#### 推演解答

**算，而且是個漂亮的回歸——但帶著敏捷學到的教訓。**

相似的地方：IaC 確實是「先設計後執行」。你不是邊點 console 邊試（那是手工藝、是即興），你是先把「我要的基礎設施長什麼樣」完整地、明確地寫成一份規格（一份 template），讓它先被 review、先被 diff，然後才 `apply`。這正是瀑布最健康的那個基因——**在動真格之前，先把意圖寫清楚、讓它可被審查**。你在 greenfield 時先定 OpenAPI 契約再並行開發，是同一個基因（見 ch11、ch12）。

關鍵的差別，也是為什麼這是「健康的回歸」而非「純瀑布復活」：

- **這份設計是活的、版本化的、可快速迭代的。** 純瀑布被誤讀成「一次把需求定死、之後不准變」；IaC 的 template 在 Git 裡，改它就是開一個 PR、跑一次 plan、看 diff、apply——你一天可以迭代很多次。它把「先設計」的好處（明確、可審查、可重現）跟敏捷的好處（短迴圈、可快速修正）**合在了一起**。
- **plan/diff 給了你瀑布從來沒有的回饋。** `terraform plan` 或 CloudFormation 的 change set 會在你 apply 之前，告訴你「這次變更會動到哪些資源、會不會重建你的資料庫」——這是一個在「設計」與「執行」之間插入的、瀑布時代不存在的安全迴圈。

所以正確的講法是：**IaC 是「先設計」這個瀑布基因的回歸，但它學會了敏捷的教訓——把設計變成一份活的、版本化的、有快速回饋迴圈的程式碼**。它證明了「先設計」與「擁抱變化」從來不是對立的，對立的只是「把設計定死」與「擁抱變化」。這正好呼應全書的鐘擺：手工點擊（即興）↔ 寫成程式碼（先設計），這次回擺，帶著上次（敏捷）的教訓。

## 自我檢核

口頭自答，講得出因果才算過關：

1. 虛擬化在解什麼複雜度？為什麼說「沒有虛擬化就沒有雲的商業模式」？（提示：一台實體機器怎麼賣給很多個互不相識的客戶。）
2. 把 AWS 2006 那一跳用一句話講清楚——亞馬遜到底把**什麼**複雜度，封裝成了**什麼**？這跟前面十五章「抽象把複雜度搬家」是不是同一件事？
3. CapEx → OpEx 的轉變，為什麼說它的真正威力不在「省錢」而在「消滅了一道進入門檻」？舉一個你會的例子（三人新創 vs 大公司）。
4. IaaS/PaaS/SaaS 三層的分界線是什麼？用「你不用管什麼」與「你放棄了什麼控制權」兩個維度各講一遍。為什麼這道光譜是複雜度守恆的具體形狀？
5. serverless 的「無伺服器」是什麼意思的「無」？它把什麼藏起來了、藏不住時以什麼形式漏出來？
6. IaC 解的是「手動點擊」的哪三個痛（不可重現、不可回溯、不可審查）？為什麼說「把基礎設施寫成程式碼」就等於把 ch14 那套協作機器全部接過來用？
7. IaC 的「宣告式」跟 ch13 Codd 的 SQL、跟 ch17 將出現的 Kubernetes，共享的是哪一個抽象衝動？（提示：你說要什麼、引擎算怎麼做。）
8. 雲這一跳的帳單有五筆（冷啟動/分區/鎖定/成本/信任）——挑你親身踩過的兩筆，講清楚它是哪一層抽象在哪裡洩漏。

## 延伸閱讀

- **Benjamin Black,「EC2 Origins」（2009 部落格文，blog.b3k.us）**——EC2 共同發起人之一的第一手回憶：2003 年那份給 Bezos 的文件、把虛擬伺服器當服務賣的點子怎麼來的。讀它看「雲」如何從一家公司的內部痛點長出來，而非某個宏大藍圖。（建議搭配亞馬遜官方的早期客戶回顧 aboutamazon.com 的「the earliest AWS customers」一文。）
- **Jeff Bezos 談「undifferentiated heavy lifting / muck」（2006 前後多場演講）**——搜尋這個詞看 Bezos 怎麼用「爛泥」框出雲的核心商業邏輯。本章個案的源頭觀念；理解雲是「把所有人都要做的粗活做到極致再賣回去」這門生意。
- **NIST SP 800-145,「The NIST Definition of Cloud Computing」（2011，csrc.nist.gov）**——只有短短幾頁，是 IaaS/PaaS/SaaS 與雲特性（on-demand、measured service…）的權威定義。當你需要一個不被行銷詞汙染的分類學基準時讀它。
- **「Xen and the Art of Virtualization」（Barham 等，SOSP 2003）**——雲底層虛擬化的代表性論文。不必讀完，讀 introduction 體會「在 x86 上把一台機器乾淨地切成很多台」當年有多難——那份難，正是雲能成立的技術前提。
- **HashiCorp Terraform 文件的「What is Infrastructure as Code」段落（developer.hashicorp.com）**——當代 IaC 思維最清楚的官方陳述：宣告式、plan/apply、狀態管理。讀它把本章「基礎設施即程式碼」的世界觀，對應回你天天寫的 CloudFormation/CDK。
- **Joel Spolsky,「The Law of Leaky Abstractions」（2002，joelonsoftware.com）**——本章一直在還的「抽象洩漏」帳單的原典（全書到 ch19 才正式展開）。先讀它，你會在每一次 Lambda 冷啟動、每一張爆掉的 AWS 帳單裡，認出同一條定律。
