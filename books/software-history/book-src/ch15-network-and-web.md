# ch15 — 網路與 Web：軟體從盒子搬到伺服器

> **本章解決什麼問題**：到上一章為止，軟體還是一個「東西」——你把它編譯成一份可執行檔，刻進光碟、裝箱、寄到使用者手上，它在「他的機器」上跑。這一章講一次比任何新語言、新流程都更動搖根基的轉變：軟體的交付形態變了——它不再裝在使用者的機器上，而是跑在**我們的伺服器**上，使用者手上只剩一扇窗（瀏覽器）。撞的牆不是「程式太大」（ch05）也不是「需求會變」（ch12），而是**交付**這件事本身的成本與控制權：盒裝軟體一旦寄出去就改不動了，而世界不會停下來等你下一版光碟。Web 的開放標準（HTTP/HTML/URL，無專利、任何人可實作）把「網路」這層複雜度標準化，讓它成為史上最大的應用平台；而這一跳的帳單，正是你每天在付的那筆——軟體跑在你的伺服器上，於是「為所有人 24/7 可用」變成你的責任。雲（把伺服器再搬進別人的機房）是下一跳，留給 ch16。

## 從你已知的出發

你整個職業生涯，都站在一個你大概從沒當成「歷史事件」來想的前提上：**你寫的軟體不在使用者的電腦裡。**

把你做過的東西攤開看。你做的 B2B/B2C 資料中介服務，跑在 AWS ECS 上；對接你的那家企業客戶，他們的工程師從沒拿到過你的程式碼，他們拿到的是一份 OpenAPI 規格和一個 endpoint。你的 REST API 改了一版邏輯，當天部署上去，全世界打這支 API 的人下一個請求就吃到新版——你不需要寄任何東西給任何人，不需要任何人「安裝更新」。你那套 500K 註冊、尖峰 3,200 並發的 WebSocket 即時系統，使用者那端只是一個瀏覽器分頁（或一個 app 裡的 WebView），真正的狀態、真正的邏輯、真正的「程式」全在你的伺服器上；使用者的裝置只是一扇窗，負責把畫面畫出來、把點擊送回來。

這個安排你習以為常到不會去想它，但它在歷史上**相當晚近**才成立。在它之前，軟體業四十年的預設是反過來的：軟體是一份你**交付給對方、由對方執行**的製品；要讓使用者用上新功能，唯一的辦法是把新版本送到他手上——壓成軟碟、刻成光碟、裝箱、寄出，然後祈禱他真的會去安裝。

這個反轉——軟體從「裝在使用者機器的盒子」變成「跑在我們伺服器的服務」——是本章的主線。而它替你帶來的那筆帳單，你也每天在付：因為軟體跑在你的伺服器上，**所有人的可用性都是你的責任**。盒裝軟體商賣完光碟，使用者的電腦當機是使用者的事；你的服務掛了，是你半夜被 CloudWatch 叫醒。你做的所有事——多層 cache、read replica、HPA autoscaling、冪等消費者、那條 FluentBit pipeline——本質上都是在替「軟體搬到伺服器端」這次複雜度搬家還帳。你不是在維運一個產品，你是在替全世界的使用者**持有**那個產品的執行。

要看懂這筆帳怎麼來的，得回到兩件事：網路怎麼從一堆互不相通的機器變成一張標準化的網，以及 Web 怎麼用三個無專利的小東西，贏過當年所有財大氣粗的封閉平台。

## 第一層標準化：把「網路」這面牆抹平

在軟體能搬到伺服器之前，得先有一張能讓任何機器跟任何機器講話的網。這件事本身就是一場對複雜度的征服，而且征服的方式——**用一個標準把底層差異藏起來**——跟全書其他每一次抽象躍遷同形。

### ARPANET 與封包交換：別讓一根斷線拖垮整張網

1969 年底，ARPANET 接上頭四個節點（照 landscape）。它要解的本質是個複雜度問題：一堆異質、昂貴的電腦散在不同機構，你想讓它們互通。最直覺的做法是「電路交換」——像打電話，通訊前先在兩端建一條專屬線路，獨佔到講完。這個模型**脆弱又浪費**：線路上任一中繼點壞掉，整通連線就斷；就算你沒在傳資料，那條線也被你佔著。

ARPANET 押的是另一個賭注：**封包交換（packet switching）**。把訊息切成小封包，每個自帶目的地、獨立在網路裡找路；某條路斷了就繞別條走。關鍵性質是**沒有中央、沒有專屬線路**——智慧分散在每個節點，任何單點故障都不致命。你今天信賴的「網路大致總是通的」，根就在這裡。這已經是一次「複雜度搬家」：它把「保證送達」從「維持一條完美的專屬線路」搬成「在不可靠的網路上，靠重傳與繞路逼近可靠」——這筆帳會在很久以後以 TCP 重傳、亂序、壅塞控制的形式，漏回你的 production（接 ch19 抽象洩漏）。

### TCP/IP：一個讓「任何網」都能互連的協定

ARPANET 證明了封包交換可行，但它本身只是一張網。真正的問題更大：1970 年代冒出一堆**互不相通**的網路（ARPANET、衛星網、無線封包網……），各說各話。要怎麼讓「網路與網路」互連成一張「網際網路（inter-net）」？

1974 年 5 月，Vint Cerf 與 Bob Kahn 發表〈A Protocol for Packet Network Intercommunication〉（照 landscape），提出後來演化成 TCP/IP 的那套協定。它的核心洞見就是整個網際網路的設計哲學：**讓底層的各種網路保持笨，把互連的智慧放到端點。** 每個網路內部怎麼運作不管，只要能傳封包就好；不同網路之間靠 IP 這層共同語言定址、靠 TCP 這層在端點「把切碎的封包重組成有序、可靠的串流」。

這就是著名的 **end-to-end 原則**：網路本身只負責盡力把封包送過去（best-effort），可靠性、順序、重傳這些聰明事全部在兩端主機上做。你做後端時直覺依賴的那條分界——「網路會掉包、會亂序，所以我得在應用層處理冪等與重試」——正是 TCP/IP 這個五十年前的決策劃下的：網路被刻意設計成笨的、不保證的，聰明與責任被推到你這端。

1983 年 1 月 1 日，ARPANET 正式從舊的 NCP 協定切換到 TCP/IP（照 landscape，著名的 flag day）。值得記住的對比：這種「全網同一天硬切、不准回頭」的做法，在今天這個規模的網際網路上**完全不可能**——這本身就說明了當年的網有多小，以及一個標準一旦普及、複雜度漲起來後就再也搬不動（這主題會在本章下半場與 ch19 反覆出現）。

TCP/IP 做的事，用一句話總結：**它把「網路」這層複雜度標準化、藏到一個統一介面後面。** 從此，軟體不必管底層是光纖還是衛星、是哪家廠商的路由器——你只要會講 IP，你就在網上。這是「軟體搬到伺服器」的**前提條件**：沒有一張任何機器都能上的標準化網，就沒有「跑在遠端伺服器、人人可連」這回事。

```text
   電路交換（打電話模型）：建一條專屬線路，獨佔到講完
   ┌─────┐                          ┌─────┐
   │ A端 │ === 專屬線路（獨佔）==== │ B端 │   線斷＝整通斷、佔線＝浪費
   └─────┘                          └─────┘

   封包交換 + TCP/IP：訊息切成封包，各自找路，端點重組
   ┌─────┐                          ┌─────┐
   │ A端 │ -> 封包各自找路繞節點 -> │ B端 │   任一段斷＝繞路、端點重組
   └─────┘                          └─────┘
```

關於 TCP/IP 的協定機制細節（三向交握、壅塞視窗、封包格式），本書不展開——你需要時，延伸閱讀有原典。本章在意的只有一件事：**它把網路的複雜度標準化了，於是上面才長得出 Web。**

## Web：用三個無專利的小東西，造出史上最大的應用平台

網路通了，但網路不等於 Web。1980 年代末，網上能做的事（email、FTP 傳檔、Usenet、各種登入式服務）彼此割裂，沒有一個統一的方式去「指向並取得任何一份文件」。複雜度的牆在這裡：**資訊散落在無數不相通的孤島上，每座島有自己的存取方式、自己的位址規則。**

### Berners-Lee 的三件套：HTTP、HTML、URL

1989 年 3 月，CERN 的 Tim Berners-Lee 寫了提案〈Information Management: A Proposal〉（照 landscape）。他要解的是 CERN 自己的痛——一個龐大、人員流動快的研究機構，資訊散在各種不相容系統裡，新人來了找不到東西。他老闆在提案上批了句後來很有名的話，大意是「vague, but exciting（含糊，但令人興奮）」（廣為流傳，未逐字查證原件）。

他造出來的東西，是三個小到驚人的標準：

- **URL**：一個全球統一的定址法。`http://host/path` 這個格式讓你能用一個字串，唯一地指向網路上任何一份資源。這是把「資訊孤島各有各的位址」這個複雜度一舉抹平——所有東西都用同一種地址。
- **HTTP**：一個極簡的、無狀態的請求／回應協定。客戶端說「GET 這個 URL」，伺服器回「這是內容」。無狀態（stateless）是關鍵設計——每個請求自帶全部所需資訊、伺服器不必記住你上一次幹了什麼。你做 REST API、靠 token 在每個請求裡自證身分、把 session 狀態外推到 Redis 而非綁在某台機器上——你做的全部是 HTTP 這個「無狀態」基因的後代。它讓「水平擴展」成為可能：任何一台伺服器都能接任何一個請求，因為請求不依賴特定機器的記憶。
- **HTML**：一個描述文件結構、且能**內嵌指向其他文件的連結**的標記語言。超連結（hyperlink）是 Web 之所以是「網」的那條線——任何一份文件可以指向任何另一份，不需要對方同意、不需要中央目錄登記。

三件套合起來，第一次給了世界一個**統一的方式去發布、定址、取得、並互相連結任何文件**。1991 年 8 月 6 日，Berners-Lee 把 WWW 對外公開（照 landscape）。

### 決定性的一步：把它變成沒有人能擁有的東西

這裡是本章最關鍵、也最容易被略過的史實，請你慢讀。

Web 真正的勝負手，不是技術上多巧妙——三件套都簡單到近乎簡陋。真正決定一切的是一個**所有權決策**：**1993 年 4 月 30 日，CERN 正式宣告放棄 WWW 相關程式碼的所有智慧財產權，把它釋入公有領域（public domain），任何人都能免費使用、複製、修改、再散布，不收任何權利金。**（這份聲明的原文大意是：「CERN 放棄對此程式碼——無論原始碼或二進位形式——的一切智慧財產權，並允許任何人使用、複製、修改與再散布。」我查到 CERN 官方說法確認此日期與大意，逐字措辭以 CERN 原件為準。）

請注意這是**兩個不同的日子**，別搞混：1991 年 8 月是 WWW 軟體與概念「對外公開」；**1993 年 4 月 30 日才是 CERN 把它正式釋入公有領域、宣告無人可擁有**。前者是「能用了」，後者是「沒有人需要付錢或求許可就能實作它」——而軟體史的天平，是被後面這一天壓下去的。

為什麼這一步是決定性的？因為它意味著：**HTTP、HTML、URL 沒有專利、沒有授權費、沒有守門人。** 任何人——任何公司、任何學生、任何國家——都可以寫一個瀏覽器、架一台伺服器、發明一個新的 server-side 語言，而不必向任何人請求許可、不必付任何權利金、不會被告侵權。這個「無人可擁有」的性質，把 Web 從「一個好技術」變成了「一塊任何人都能在上面蓋東西的公共地基」。

你可以把它跟 ch14 的開源精神接起來看：那一章講軟體的**程式碼**所有權如何被重寫（從私有到公共財）；這一章講軟體的**平台標準**所有權如何被重寫。兩者是同一個主題的兩面——當一個東西「沒有人擁有」時，它反而能被「所有人建造」，於是它的成長不再受任何單一守門人的意願與商業利益限制。控制權的下放，換來的是爆炸性的、無需許可的（permissionless）創新。這是本書「複雜度／控制權的分配」這條暗線在網路時代的主場。

## 個案：開放標準如何擊敗圍牆花園

抽象講「開放贏過封閉」太廉價了。把它攤成一個有人、有時間、有代價的具體個案，你才會真的信。最乾淨的對照組不是別的，是一個今天幾乎被遺忘、但當年比 Web 還紅的競爭者：**Gopher**。

### Gopher：差一步就贏，輸在一個授權決定

1991 年，明尼蘇達大學推出了 Gopher——一個用選單階層組織、瀏覽全球文件的系統。它解的問題跟 Web 幾乎一樣，而且在早期**更受歡迎、更好用**：選單式的介面對當時的人來說比超連結直覺，部署也簡單。1993 年時，Gopher 的流量還遠大於 Web（照我查到的多份整理）。

然後明尼蘇達大學做了一個決定，親手把自己送進歷史的註腳。1993 年，校方宣布要**對商業使用者收取 Gopher 伺服器軟體的授權費**（據多份整理，校方的理由是要籌措開發經費，並類比 Unix 等軟體也收費）。費用其實不高，但傷害不在金額——在於那個訊號：**用 Gopher，你頭上就懸著一個守門人，他今天能收一點、明天能收更多、後天能改規則。** 工程師與機構的反應是理性的：大量 Gopher 部署停止開發、轉向，以避開這份不確定性。

對照之下，就在同一個 1993 年 4 月，CERN 把 Web 釋入公有領域、明說不收任何權利金。一邊是「免費，但我們保留收費的權利」，一邊是「無人可擁有，永遠免費」。差距是天壤之別。後來的數字像在演示這個原理：1993 年 Gopher 流量年成長約 997%，而 Web 的年成長率是離譜的數十萬個百分比（照我查到的整理，這類成長率數字屬估計、僅作量級示意）。明尼蘇達大學在 2000 年才把 Gopher 改成 GPL 開源——但木已成舟，遲了七年。

這個個案把本章的核心命題講得再清楚不過：**Web 贏 Gopher，技術不是主因，所有權結構才是。** 一個「保留收費權」的守門人，哪怕現在不收，光是「他有權收」這件事，就足以讓人不敢在上面下重注。開放標準的決定性，不在它今天免費，在它**結構上不可能有人收你過路費**。

### AOL 與圍牆花園：把同一個道理放大到消費端

把鏡頭從技術社群拉到一般消費者，同樣的劇本在 AOL（America Online）身上重演。1990 年代中後期，對多數美國家庭來說「上網」幾乎等於「上 AOL」。AOL 是經典的**圍牆花園（walled garden）**：精緻、好用、被策展過——它幫你把內容整理好、體驗顧好，代價是你**出不去**，你看到的是 AOL 決定讓你看到的，它的商業利益就是你的世界邊界（很多用戶甚至不知道牆外還有整個開放的 WWW，照我查到的整理）。一旦使用者嚐到開放 Web 那種「任何人都能發布、任何頁面都能連到任何頁面、沒人決定你能看什麼」的狂野，牆內那套「我們幫你選好」就從便利變成束縛；內容與創新在牆外爆炸性地長，而牆內受限於單一公司的策展意願。結局你知道：AOL 成了歷史名詞，開放 Web 接管了世界。

Gopher 與 AOL 是同一個道理的兩個尺度：**封閉平台的成長上限，是那個擁有它的人的意願與利益；開放標準沒有這個上限，因為沒有「那個人」。** 把複雜度與控制權集中在一個守門人手上，短期換來整潔與便利的體驗；但長期，那個集中點就是天花板。這是本書「集中↔分散」鐘擺（ch01 提出）在平台層的一次清楚擺動——而且請你記著它，因為到了 ch16 的雲、乃至今天各種平台，這個鐘擺會擺回來：當年贏在開放的 Web，後來又長出了新的、更精緻的圍牆花園。沒有哪一邊是終局。

## 從文件檢視器到應用平台：複雜度往伺服器端流

到這裡，Web 還只是個「把文件連起來」的東西。要走到你今天賴以維生的那個世界——軟體跑在伺服器、瀏覽器是窗口——還需要兩步：讓網頁能**動**，以及讓瀏覽器從「看文件的」變成「跑應用的」。

### LAMP 與動態網站：頁面不再是死的

最早的網頁是**靜態**的：伺服器上躺著一份寫死的 HTML 檔，誰來要都給同一份。這撐不起「你的帳戶餘額」「你的搜尋結果」這類**因人因時而異**的內容。於是出現**動態網站**——HTML 不是事先寫死，而是伺服器在收到請求那一刻，根據資料庫、根據你是誰，**即時產生**出來的。

承載這波的，是 1990 年代末成形、後來叫做 **LAMP** 的組合：**Linux**＋**Apache**（Web 伺服器）＋**MySQL**＋**PHP/Perl/Python**（server-side 語言）（照 landscape）。它在歷史上的分量不在任何單一元件多先進，而在它**整套都是開源、免費、無需許可**——這正是上一節那個「無人可擁有」的地基結出的果。一個學生、一間小公司，零授權成本就能架起服務全世界的動態網站；Web 的開放，往下傳染給了建在它上面的整套工具鏈。

對你來說，LAMP 是你那套 NestJS + PostgreSQL 後端的直系祖先。你寫的每一個「收到請求 → 查資料庫 → 組出回應」的 handler，做的就是當年 PHP 腳本做的事：**在伺服器上即時產生回應**。中間隔了三十年的抽象演進（框架、ORM、型別、容器），但那個基本動作——伺服器端動態生成——是 LAMP 時代定下來的。而它定下的同時也定下了那筆帳：**邏輯在伺服器跑，意味著伺服器得一直開著、一直能扛住流量。** 靜態檔案誰都能 cache、誰都能鏡像；動態生成則把計算的重擔牢牢釘在你的伺服器上。你做的多層 cache、read replica、autoscaling，全部是在替「動態生成」這個從 LAMP 開始的選擇，持續付運算與可用性的帳。

### Web 2.0 與 AJAX：瀏覽器變成應用程式的執行環境

最後一步，是瀏覽器本身的角色翻轉。

早期的 Web 互動模型很笨：你點任何東西，瀏覽器就向伺服器要一整張**新的頁面**、整頁重新載入、白屏一下、重畫。在這個模型裡，瀏覽器始終是個「文件檢視器」——它的工作是顯示伺服器送來的一份份文件。

2005 年 2 月，Jesse James Garrett 寫了一篇短文，給一個已經存在的技術組合起了個名字：**AJAX**（Asynchronous JavaScript and XML，照 landscape）。它的核心招數是：網頁裡的 JavaScript 可以**在背景**、不重新載入整頁的情況下，跟伺服器交換資料，然後**只更新頁面的一小塊**。Google Maps 可以讓你拖曳地圖、新區塊在背景悄悄載入而不白屏；Gmail 可以收到新信而不必你按重新整理。瀏覽器從「顯示一份份文件」，變成了「一個持續運行、會在背景跟伺服器對話的應用程式」。

差不多同時（名詞出於 2004 年 O'Reilly 與 MediaLive 的腦力激盪，定義文〈What Is Web 2.0〉發表於 2005 年，照 landscape），Tim O'Reilly 等人用 **Web 2.0** 這個詞替整波轉變命名：Web 從「人們閱讀的文件集」變成「人們使用的應用平台」、變成使用者自己生產內容、彼此互動的地方。

對你來說，這一步是你那套 WebSocket 即時系統的世界觀源頭。AJAX 打破了「一次請求一整頁」的桎梏，讓瀏覽器能持續、細碎地跟伺服器通訊；WebSocket 把它推到極致——一條一直開著的雙向通道，伺服器能主動推訊息給瀏覽器。你那套 500K 註冊、3,200 並發的即時系統能存在，前提就是「瀏覽器是個能跑常駐邏輯、能維持長連線的應用執行環境」——而這前提是 Web 2.0／AJAX 那幾年才落定的。在那之前，瀏覽器是文件檢視器，根本承載不了「即時」這個概念。

（至於這之後前端如何從散落的 jQuery 演化到 React、Vue 這類框架——那是另一條完整的故事線，本書不展開，一句帶過：前端框架是「把瀏覽器當應用平台」這個前提一旦成立後，為了管理瀏覽器端日益膨脹的應用複雜度而長出來的。）

### 交付形態的根本反轉：更新不再靠寄光碟

現在把這幾步合起來，那個本章開頭的主線就完整了。

軟體業頭四十年的交付模型是：開發一份軟體 → 壓成媒介（軟碟、光碟）→ 裝箱、鋪貨、寄出 → 使用者**安裝**到自己的機器 → 軟體在**他的機器**上跑。在這模型裡，你要修 bug 得發 patch、祈禱使用者去裝（多數不會），於是市面上同時跑著你軟體的 v1.2、v1.5、v2.0，你得全部支援（碎片化）；但相對地，使用者電腦當機、灌不起來、跟別的軟體衝突，那是**他的**問題——你交付完光碟，可用性的責任就轉給了使用者。

Web 把整個模型反轉。當軟體跑在**你的**伺服器上、使用者只用瀏覽器連過來：你要修 bug、上新功能，部署到伺服器即可，下個請求**所有人**同時吃到新版，沒有寄出、沒有安裝、沒有碎片化（你做的就是這個：REST API 改一版、當天部署、全世界下個請求就是新版）。但可用性的責任**整個搬到了你頭上**——使用者那端只剩一扇窗，他不再「持有」你的軟體，是你替他持有著它的執行；於是你的軟體掛了、慢了、被流量打爆了，全是你的事，而且是**所有使用者同時**的事。

這就是本章那筆複雜度帳單的全貌，也是 SaaS（software as a service）這個商業模式的技術根。一個常被當作里程碑的具體場景：2013 年 Adobe 宣告 Creative Suite 的 DVD 之死、全面轉向訂閱制的雲端交付（照我查到的整理）——那是「盒裝軟體」這個延續了三十年的形態，在一個大型消費軟體商身上正式落幕的象徵。軟體不再是你買來「擁有」一份拷貝的東西，而是你訂閱、由廠商在他的伺服器上「持續提供」給你的服務。

```text
   盒裝模型（~1990s）：軟體在使用者機器跑
   ┌────────┐   寄光碟、求人安裝   ┌──────────────┐
   │ 開發商 │─────────────────────►│ 使用者的機器 │   可用性＝使用者的事
   └────────┘                      └──────────────┘   改版＝寄新光碟

   伺服器模型（Web 之後）：軟體在你機器跑，客戶端只是窗口
   ┌─────────────┐   瀏覽器當窗口   ┌────────────┐
   │ 使用者（窗）│◄────────────────►│ 你的伺服器 │   可用性＝你的事
   └─────────────┘                  └────────────┘   改版＝部署一次、人人即得
```

你維運的痛——半夜被叫醒、為 SLA 負責、為每一個你從沒見過面的使用者的可用性扛責任——不是你倒楣，是這次交付形態反轉的**內建帳單**。軟體搬到伺服器，省掉了「寄光碟、求人安裝、支援一堆版本」的複雜度；但它把「替全世界持有這個軟體的執行、24/7 不能倒」這個全新的、巨大的複雜度，搬到了你身上。這就是複雜度守恆在交付形態上的演示：負擔沒有消失，它從使用者的機器，搬到了你的 on-call 排班表上。

## 複雜度搬去哪了

| 維度 | 搬家前（盒裝軟體時代） | 搬家後（軟體跑在伺服器） | 帳單／在哪洩漏 |
|---|---|---|---|
| 交付 | 壓媒介、裝箱、鋪貨、寄出；使用者自行安裝 | 部署到伺服器一次，所有人下個請求即得 | 你得 24/7 維運；部署成了高風險動作（一錯全體中招） |
| 更新 | 發 patch，祈禱使用者去裝；版本碎片化 | 統一版本，伺服器端說了算 | 你**不能**有「使用者還在用舊版」當藉口，鍋全是你的 |
| 可用性責任 | 使用者電腦的事，交付即轉移 | 全在你身上，且是「所有使用者同時」 | on-call、SLA、半夜被 CloudWatch 叫醒（你的日常） |
| 狀態與邏輯 | 在使用者機器上 | 在你的伺服器上；客戶端只是窗口 | 伺服器得一直開、扛得住流量；cache/replica/autoscaling 是這筆帳 |
| 平台控制權 | 各家專有格式、各自為政 | 開放標準（HTTP/HTML/URL，無專利） | 控制權下放換來 permissionless 創新；但也讓「標準一旦普及就改不動」 |
| 網路可靠性 | （單機，不涉網） | 靠 TCP 在不可靠網路上逼近可靠 | 重傳、亂序、壅塞、分區會漏回應用層（接 ch19） |

**這一跳的帳單一句話**：軟體搬到伺服器，消滅了「把軟體實體送到每個人手上、再求他安裝、再支援一堆版本」的交付複雜度；但它製造了一個對等巨大的新負擔——**你從此替全世界持有這個軟體的執行，所有人的可用性、效能、安全都是你一個人的 24/7 責任**。複雜度沒有消失，它從千千萬萬台使用者機器上的「安裝與相容」問題，凝聚成你機房裡的「永不停機」問題。而開放標準這一手，則是用「放棄擁有」換到「人人可建」——控制權的下放，是 Web 能長成史上最大平台的真正代價與真正紅利。

什麼情況下這個抽象會洩漏、反噬？當「軟體在遠端、網路總是通的、伺服器總是在的」這三個你習慣性假設的前提**不成立**時：網路分區了（你的 WebSocket 連線無預警斷開、重連風暴打爆伺服器）、伺服器扛不住了（尖峰流量、你那次靠調 cache 把 RDS CPU 降 40% 的場景）、底層的 TCP 漏了（重傳堆積、延遲尖刺）。每一次這類事故，都是「軟體搬到伺服器」這層抽象在邊緣漏出它藏起來的複雜度——而你之所以值錢，正因為漏的時候你跳得下去。

## 紙上推演

### 推演題一 **[15 分鐘]** ★★

用「複雜度搬到伺服器端」這個框架，向一個只用過盒裝軟體的人解釋：**為什麼 SaaS（如線上文件、你的 B2B API 服務）能持續、無痛地更新，而盒裝軟體（如當年的盒裝 Office）做不到？** 要求：(a) 講清楚兩種模型裡「軟體在誰的機器上跑」這個根本差異；(b) 說清楚這個差異如何決定了「更新」的難易；(c) 誠實講出 SaaS 為此付出的帳單是什麼，別把它講成只有好處。

#### 推演解答

(a) 根本差異在「軟體在誰的機器上跑」。盒裝軟體：你買一份拷貝、**安裝到自己的機器**，它在你的機器上執行；開發商手上沒有「正在執行中的那一份」。SaaS：軟體跑在**廠商的伺服器**上，你的瀏覽器只是一扇窗，負責顯示畫面、送回操作；你從不「持有」那份執行中的軟體。

(b) 這個差異直接決定更新難易。盒裝模型裡，「更新」意味著把新版本**送到每一台已安裝的使用者機器上**、再依賴使用者主動安裝——你管不到對方的機器，於是永遠有一堆人停在舊版，市面上同時跑著 v1、v2、v3，你得全部支援（版本碎片化）。SaaS 模型裡，「更新」只是**在你自己的伺服器上部署一次**——全世界用的都是那份在你伺服器上跑的執行，下個請求所有人同時是新版；沒有寄送、沒有安裝、沒有碎片化。差異的根，就在「執行中的那一份」在誰手上。

(c) 帳單：可用性責任整個搬到廠商頭上。盒裝模型裡，你交付完光碟，使用者電腦當機、裝不起來是**他的**問題；SaaS 模型裡，因為軟體在你的機器上跑，它掛了、慢了、被打爆了全是**你的**問題，而且是所有使用者**同時**遭殃。你換到了「人人即時更新」的便利，代價是簽下了「24/7 為全世界的可用性負責」這張賣身契——這正是 on-call、SLA、半夜被叫醒的由來。把這題講完整的關鍵，是不能只說 SaaS 的好處：它不是消滅了複雜度，是把「交付與相容」的複雜度，換成了「永不停機」的複雜度。

常見誤解：以為 SaaS 的優勢是「雲端很神奇」。不是——優勢純粹來自「執行那一份在廠商手上、使用者只連過來」這個結構，跟它跑在自家機房還是 AWS 無關（那是 ch16 的下一跳）。

### 推演題二 **[20 分鐘]** ★★★

解釋**開放標準為何擊敗圍牆花園**。要求：(a) 以 Gopher vs Web 為主軸，講清楚 Gopher 輸的關鍵**不是技術**而是 1993 年那個授權決定；(b) 點出「無人可擁有」與「免費」是兩回事，為什麼前者才是決定性的；(c) 把這個結論接回 ch14 開源的所有權主題——說清楚「開放標準擊敗圍牆花園」與「開源擊敗私有軟體」是不是同一個道理。

#### 推演解答

(a) Gopher 在早期其實**更受歡迎、更好用**：選單式介面對當時的人比超連結直覺，1993 年它的流量還遠大於 Web。它輸的轉捩點，是 1993 年明尼蘇達大學宣布要對商業使用者收 Gopher 伺服器軟體的授權費。費用其實不高，但傷害不在金額，在**訊號**：用 Gopher，你頭上就有一個守門人，他今天能收一點、明天能收更多、後天能改規則。理性的工程師與機構不敢在這種不確定性上下重注，於是紛紛轉向。對照之下，CERN 在同一年（1993-04-30）把 Web 釋入公有領域、明說永不收權利金。所以 Gopher 輸的不是技術較量，是**所有權結構**的較量——它有個守門人，Web 沒有。

(b) 「無人可擁有」≠「免費」，而前者才決定性。Gopher 在被收費前也是免費的；很多圍牆花園（早期 AOL 的某些服務）使用者也不直接付費。「免費」是一個**現狀**，一個擁有者隨時可以改變的現狀。「無人可擁有」是一個**結構性質**——CERN 放棄了智財權，意味著結構上**不存在任何人有權**向你收過路費或撤回許可。差別在於：在「免費但有主人」的平台上下注，你賭的是主人的善意會延續；在「無人可擁有」的標準上下注，你不需要賭任何人的善意，因為沒有那個人。正是這個「不需要信任任何守門人」的性質，讓無數人敢在 HTTP/HTML 上投入重本去蓋東西——permissionless（無需許可）的創新，只有在「無人可擁有」時才成立。

(c) 跟 ch14 是**同一個道理的兩面**。ch14 講軟體**程式碼**的所有權被重寫（開源：原始碼公共化，沒人能私有地鎖住它）；本章講軟體**平台標準**的所有權被重寫（Web 三件套釋入公有領域，沒人能私有地擁有這平台）。核心機制一致：**把「擁有權」拿掉，換來「人人可建」，於是成長不再受任何單一主人的意願與利益封頂。** 兩者背後是同一條暗線——複雜度與控制權的分配：把控制權集中在一個擁有者手上，換來短期整潔便利，但那集中點就是天花板；把它下放成「無人可擁有」，你失去整潔的掌控，換來無上限的、permissionless 的生長。

常見誤解：以為 Web 贏是因為超連結比選單「更先進」。技術差異在早期甚至是反過來的。真正的勝負手是所有權，不是介面。

### 推演題三 **[15 分鐘]** ★★

對帳：你那套 500K 註冊、3,200 並發的 **WebSocket 即時系統**，解的是哪一類「網路的複雜度」？要求把它拆到三層歷史地基上：(a) 它依賴 TCP/IP 解掉的是什麼；(b) 它依賴「Web 2.0／瀏覽器變應用平台」解掉的是什麼；(c) 它自己額外解的、前兩層沒給的是什麼。最後說一句：這套系統把複雜度搬到了哪裡、帳單是什麼。

#### 推演解答

(a) **依賴 TCP/IP 這層**：它解掉「在不可靠、會掉包的網路上，建立一條兩端有序、可靠的位元組串流」這件事。你的 WebSocket 底層就是一條 TCP 連線——封包亂序、遺失、繞路這些 TCP 在端點替你重組，應用層不必管。沒有它把「網路」標準化、提供可靠串流，你連「一條穩定的長連線」這個前提都沒有。

(b) **依賴 Web 2.0／瀏覽器變應用平台這層**：它解掉「瀏覽器只是文件檢視器、只能一次要一整頁」這個限制。WebSocket 要成立，瀏覽器得是個能跑常駐 JavaScript、維持背景長連線、收到推送時局部更新畫面的**應用執行環境**——這正是 AJAX／Web 2.0 那幾年才落定的角色翻轉給的。舊世界裡「即時」根本無處安放。

(c) **它自己額外解的**：前兩層給了你「一條可靠的長連線」和「一個能用它的瀏覽器」，但沒給你「**伺服器主動、即時地把狀態變化推給成千上萬個同時在線者**」這件事。HTTP 本身是無狀態請求／回應——客戶端問才答；你的系統要的是反過來、伺服器主動推、且要在 3,200 條並發連線間維持一致狀態（誰在哪個房間、誰該收到哪則訊息）。這得你自己搭：Redis Pub/Sub 在多台伺服器間廣播事件、處理連線狀態、重連風暴、backpressure。

一句話：這套系統把「即時協調大量同時在線者」的複雜度，從**不存在於前兩層的真空**，搬到了**你的伺服器叢集與 Redis Pub/Sub** 裡。帳單是：你得替每一條長連線維護狀態、得讓多台無狀態 Web 伺服器在 Redis 這個共享真相源上協調、得在斷線重連時不被自己的重連風暴打垮——這些全是「軟體在伺服器、客戶端只是窗口、而我要為即時性與全員可用性負責」這個本章核心轉變的、最尖銳的帳單形態。

## 自我檢核

口頭自答；講得出因果才算過關，講不清楚代表還沒懂。

1. 本章撞的牆，跟 ch05（程式太大）、ch12（需求會變）撞的牆，性質有什麼不同？為什麼說「交付形態」這件事本身會是一道複雜度的牆？
2. 封包交換相對於電路交換，把「可靠送達」這個複雜度從哪裡搬到了哪裡？TCP/IP 的 end-to-end 原則為什麼把「聰明」放在端點、把網路本身留笨？這跟你「在應用層處理冪等與重試」的直覺是什麼關係？
3. HTTP 的「無狀態」是個設計決策。它讓什麼變可能（提示：水平擴展）？你把 session 外推到 Redis、靠每個請求自證身分，是在還這個 1989 年的基因的哪一筆？
4. 1991 年 8 月與 1993 年 4 月 30 日，對 Web 來說各自是什麼日子？為什麼說軟體史的天平是被 1993 那天壓下去的、而不是 1991？
5. 為什麼 Gopher 輸給 Web 的關鍵不是技術而是 1993 年的授權決定？「無人可擁有」和「現在免費」差在哪、為什麼前者才決定性？
6. 「開放標準擊敗圍牆花園」與 ch14 的「開源擊敗私有軟體」，是同一個道理嗎？用「控制權的分配」這個框架講清楚它們的共同機制。
7. 把交付形態的反轉講給另一個工程師聽：盒裝模型裡「可用性是誰的責任」、伺服器模型裡又變成誰的？為什麼說你的 on-call 排班表就是這次複雜度搬家的帳單？
8. AJAX／Web 2.0 把瀏覽器從「文件檢視器」變成了「應用執行環境」——這一步為什麼是你那套 WebSocket 即時系統能存在的前提？在它之前，「即時」這個概念為什麼無處安放？

## 延伸閱讀

- **Vint Cerf & Bob Kahn, 〈A Protocol for Packet Network Intercommunication〉（IEEE Trans. Comm., 1974）**——TCP/IP 的原典。不必啃完數學，看它怎麼論證「讓底層網路保持笨、把互連智慧放到端點」這個 end-to-end 的核心決策——你「在應用層處理可靠性」的整套直覺，根就在這篇 1974 年的論文裡。
- **Tim Berners-Lee, 〈Information Management: A Proposal〉（CERN, 1989）**——Web 的起點提案，CERN 有原件掃描檔。看它要解的痛（一個人員流動快的大機構，資訊散在不相容系統裡找不到東西）多麼具體、多麼不像「要發明改變世界的東西」——這提醒你重大躍遷往往始於一個樸素的本地問題。配 CERN「30 years of a free and open Web」一頁讀 1993-04-30 釋入公有領域的那段。
- **W3C / CERN「The birth of the Web」（home.cern 的 birth-web 頁）**——CERN 官方的 Web 誕生時間軸，把 1989 提案、1991 公開、1993 公有領域三個日子分清楚的最可靠一手整理。本章最關鍵的「1991 ≠ 1993」就靠它釘死。
- **Gopher 興衰的歷史整理（如 MinnPost「The rise and fall of the Gopher protocol」, 2016；及 abortretry.fail 的 Gopher 長文）**——把本章「開放標準擊敗圍牆花園」個案的另一半補齊：一個差點贏、卻被一個授權決定親手葬送的競爭者。讀「1993 年授權費」那段，你會對「無人可擁有」的決定性有體感。（歷史整理，非一手檔案，年份與流量數字屬估計。）
- **Tim O'Reilly, 〈What Is Web 2.0〉（oreilly.com, 2005）**——「瀏覽器從文件檢視器變成應用平台」這波轉變的命名文。看它怎麼描述「Web as platform」——你今天視為理所當然的「在瀏覽器裡跑應用」這件事，在 2005 年還是個需要被命名、被論證的新觀念。
- **Jesse James Garrett, 〈Ajax: A New Approach to Web Applications〉（adaptivepath.com, 2005-02）**——AJAX 一詞的出處短文，很短。看它怎麼描述「背景非同步交換資料、只更新局部」如何打破「一次一整頁」的舊模型——這是你的 WebSocket 即時系統那條「持續、細碎的伺服器對話」之路的起點。
