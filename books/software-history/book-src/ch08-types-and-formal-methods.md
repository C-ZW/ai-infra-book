# ch08 — 型別與形式方法：用證明對抗 bug

> **本章解決什麼問題**：ch05 的結構化編程治控制流、ch06 的資訊隱藏治分解、ch07 的 OOP 治資料與行為的散落——三章都在「組織程式碼」這個層面對抗複雜度。這章換一個更激進的方向：能不能在程式**跑起來之前**，就把一整類錯誤排除掉？這撞的是**正確性的牆**——你怎麼知道一段程式是對的？兩個答案沿著同一種衝動排成光譜：弱的一端是**型別系統**（你天天用、便宜、被嫌不夠），強的一端是**形式驗證**（用數學證明、貴、被供在神壇）。本章告訴你它們是同一件事的不同強度，型別之爭（靜態 vs 動態）的真實 trade-off 是什麼，以及為什麼最強的那一端四十年來進不了主流、卻活在你想不到的地方。你從 JS 轉 TypeScript，就是在這條歷史上選了邊。

## 從你已知的出發

你被 `undefined is not a function` 坑過。不只一次。

那是 Node.js 後端最經典的死法之一：某個欄位在多數情況下都有值，於是整條呼叫鏈都假設它存在；直到某個邊界 case——一筆髒資料、一個漏掉的 await、一個上游沒回傳的欄位——它變成 `undefined`，然後 `.toUpperCase()` 或 `.map()` 在半夜的 production 炸開，CloudWatch 跳一個 stack trace，告訴你「在這一行，某個東西不是它該是的型別」。錯誤訊息精準地指出了**症狀發生的位置**，卻完全沒告訴你**那個 `undefined` 是從哪一層流進來的**——它可能在三個 service、五次 await 之前就已經是空的了，只是一路被當成有值傳遞，直到撞上一個真的要用它的地方才現形。

然後你做了一個選擇。你把專案從 JavaScript 遷到 TypeScript，打開 `tsconfig.json` 裡的 `strict`，其中包含一個叫 `strictNullChecks` 的開關。打開那一刻，編譯器開始對你嘮叨：這個值的型別是 `string | undefined`，你不能直接當 `string` 用，要嘛先檢查、要嘛標記它可能為空。一開始很煩，紅色波浪線鋪滿螢幕。但你很快發現，那些波浪線出現的位置，**正是過去半夜會炸的位置**——只是現在它在你打字的時候就炸，不在 production 炸。

你做的這個選擇，不是一個工具偏好。它是一場**始於 1965 年、爭論了六十年的戰爭**裡的一次選邊。你選了「在執行前就排除一類錯誤」這條路——這條路最弱的版本叫型別檢查（你每天 `tsc` 一次），最強的版本叫形式驗證（用數學證明程式絕無 bug，貴到只有航太和晶片公司付得起）。而那個逼你打開 `strictNullChecks` 的 `undefined`，它的源頭有名有姓：1965 年，一個叫 Tony Hoare 的人，為了「方便」，在一個語言裡加進了 null。四十多年後，他站上台公開道歉，說那是他的「十億美元錯誤」。

這章講的，就是這條從「用前後條件證明程式正確」的理想，一路降強度降到「你每天按的那個 `tsc`」的光譜——以及為什麼最理想的那一端，至今沒有、也可能永遠不會成為主流。

## 同一種衝動：在執行前就排除錯誤

先把這章的主軸釘死，後面所有東西都掛在它上面。

對抗 bug 有兩種根本不同的時機。一種是**事後**：把程式跑起來，餵它各種輸入，看它會不會壞——這是測試（testing），是 ch12 的主場。另一種是**事前**：不跑程式，光憑分析程式的「文本」就斷定它不會犯某一類錯——這是本章的主場。Dijkstra 在 ch05 留下那句被你引用過無數次的話，精確版是「Program testing can be used to show the presence of bugs, but never to show their absence!」（程式測試能證明 bug 存在，永遠無法證明 bug 不存在）。這句話的言下之意是一把鑰匙：**如果你想要「不存在」這種等級的保證，測試這條路天生走不到，你得換一條路——事前的、分析式的、最好是「證明」的路。**

這條「事前證明」的路，從理想到現實，是一道連續光譜。我把它畫成一條從強到弱的軸：

```text
「執行前就排除一類錯誤」的強度光譜

 強 ◄──────────────────────────────────────────────► 弱
 形式驗證          形式規格         型別系統        測試
 (full proof)     (spec/model)    (type check)   (runtime)
 證明「對」        描述「該怎樣」    證明「型別一致」 跑跑看
 ┌────────┐       ┌────────┐      ┌────────┐     ┌────────┐
 │seL4    │       │ TLA+   │      │ tsc    │     │ jest   │
 │CompCert│       │OpenAPI │      │ rustc  │     │  k6    │
 └────────┘       └────────┘      └────────┘     └────────┘
 貴到只有          中間地帶         你每天按       事後、只能
 航太/晶片付       (見書架TLA+書)   一次的那個     證明bug存在
   保證越強 → 成本越高、覆蓋的錯誤類別越窄、寫起來越痛
```

注意最右邊的測試被我放在光譜外、用虛線隔開——它不屬於「事前」這一族，但放在這裡是為了讓你看清對比：型別系統和形式驗證在光譜的**同一側**（事前、分析式、給的是「這類錯不會發生」的保證），它們的差別只是**強度**。型別檢查器說的是「你不會把 string 當 number 用、不會對 undefined 取屬性」；形式驗證說的是「這個程式在所有可能的輸入與排程下，都滿足這條規格」。後者強得多，但它們**想做的是同一件事**：把一類錯誤從「執行期的驚嚇」變成「執行前的拒絕」。

這就是本章要你帶走的第一個領悟：**靜態型別和形式驗證不是兩個領域，是同一種衝動的兩個強度檔位。** 你打開 `strictNullChecks`，和航太公司花三年證明一個微核心無 bug，動機完全一樣——都是不信任「跑跑看」，都想在程式碰到真實世界之前就把某類災難擋掉。差別只在你願意付多少成本、想擋掉多寬的災難。理解了這一點，這章剩下的就是沿著這條光譜走一遍：先看最強那端的理想（Hoare logic），再看它如何縮水成你每天用的型別檢查，最後算清楚為什麼最強那端進不了主流。

## Hoare logic（1969）：用「前後條件」證明程式正確的理想

光譜最強那一端的理論基礎，1969 年就奠定了。

那一年，Tony Hoare（全名 C. A. R. Hoare，後來受封爵士，圈內常稱 Tony Hoare）在 *CACM* 發表〈An Axiomatic Basis for Computer Programming〉。這篇論文做的事，用今天的話講，是給「程式正確」這個含糊的概念一個**數學上精確的定義**，精確到可以拿來證明。

他的核心發明是一個記法，今天叫 **Hoare triple（霍爾三元組）**，寫成 `{P} S {Q}`：

```text
Hoare triple：{P} S {Q}

  {P}          S            {Q}
 前置條件      程式碼        後置條件
 precondition  statement    postcondition
 ┌────────┐   ┌────────┐   ┌────────┐
 │執行前  │──►│ 一段   │──►│執行後  │
 │成立的  │   │ 程式   │   │保證成立│
 │假設    │   │        │   │的結論  │
 └────────┘   └────────┘   └────────┘
 讀法：若 P 在 S 執行前成立，則 S 執行後 Q 成立
```

讀法是：「**如果** P 在執行 S 之前為真，**那麼** S 執行完之後 Q 為真。」舉個土例子：`{x > 0} y := x + 1 {y > 1}`——如果進來時 x 大於 0，那 `y := x + 1` 跑完後，y 一定大於 1。這看起來無聊到可笑，但 Hoare 的洞見是：他為每一種程式結構（賦值、條件、迴圈、循序組合）都給了一條**推論規則**，讓你能像做幾何證明一樣，從小程式的三元組一步步組合出整支程式的三元組。最關鍵也最難的是迴圈——他引入了**迴圈不變式（loop invariant）**的概念：一個在迴圈每一輪前後都保持為真的條件，你只要證明它一直成立，就能推出迴圈結束後程式的性質。（這套思路建基於 Robert Floyd 1967 年對流程圖的類似工作，所以也叫 Floyd–Hoare logic；歸屬上 Hoare 自己也承認 Floyd 在先。）

停下來體會一下這件事的野心。Hoare 想要的不是「測了一百個 case 都過」，而是「我**證明**了：對所有滿足前置條件的輸入，這支程式都會給出滿足後置條件的結果」。這是數學等級的「不存在 bug」——正面回應了 Dijkstra 那句「測試證明不了不存在」的挑戰。如果這條路走得通，軟體就能像橋樑一樣被計算、被證明、被簽核，而不是被「跑跑看、出事再修」。

這個理想美得讓一整代理論計算機科學家投入畢生。它也**部分地**兌現了——只是兌現的地方，不是你以為的地方（後面〈形式驗證活在哪裡〉會講）。但對 1969 年的 Hoare 來說，這還只是純理論。真正讓「型別」這個概念走進每個工程師日常的，是另一條從 Hoare logic 縮水、降強度、換取「便宜到能自動跑」的支線——而諷刺的是，**也正是 Hoare 自己，在 1965 年，往程式裡埋了一個會反覆引爆的地雷**。我們先把這個地雷挖出來，因為它是整章最值得攤開的個案，也是你那個 `undefined` 的祖宗。

## 個案：null，那個十億美元的錯誤

時間倒回 1965 年。Hoare 當時在設計 ALGOL W——ALGOL 60（ch04 那個「遠超時代的語言」）的一個後續方言。他在做一件當年很前衛的事：為「物件參照（reference）」設計一套完整的型別系統，讓編譯器能檢查「你拿一個參照去用時，它指向的東西型別對不對」。這套型別檢查的目標，恰恰是**安全**——在編譯期擋掉一整類「用錯型別」的災難。這完全是本章光譜上「往強的一端走」的正派努力。

然後，他做了一個圖方便的決定。

參照要指向某個物件，但總有些時候，一個參照「還沒指向任何東西」——比如一個剛宣告、還沒賦值的變數。Hoare 需要一個值來表示「這裡暫時什麼都沒有」。他大可以設計一套機制，強迫每個參照在使用前必須先指向一個合法物件（型別系統明明有能力做到這件事）。但那比較麻煩。於是他選了最簡單的：引入一個特殊值 **null**，代表「指向空無」。任何參照型別都可以是 null。

用他自己 2009 年在台上的原話（大意，已對照錄影來源）：他「忍不住誘惑放進了一個 null 參照，**單純因為它太容易實作了**（simply because it was so easy to implement）」。

問題出在這個決定**從根上戳穿了他自己那套型別系統的保證**。型別系統本來承諾的是「一個宣告為 `Person` 的參照，一定指向一個 `Person`，你可以放心對它取 `.name`」。但 null 一旦被允許塞進任何參照型別，這個承諾就破了：現在一個 `Person` 型別的東西，可能其實是 null，你對它取 `.name` 就會在執行期爆炸。型別檢查器**看不出來**——在它眼裡那個變數的型別是 `Person`，完全合法；null 是型別系統的一個官方核可的後門，它合法地穿過了所有編譯期檢查，把炸彈延後到執行期。**一個為了安全而生的型別系統，被一個圖方便的特例打開了一道貫穿全身的裂縫。**

這道裂縫，順著程式語言的家譜，遺傳了下去。ALGOL 的後裔——C、C++、Java、C#、JavaScript、Python……幾乎每一個主流語言都繼承了某種形式的 null（C 的 null pointer、Java 的 `NullPointerException`、JS 的 `null` 和 `undefined`）。於是過去六十年，地球上每一個程式設計師，都在反覆償還 1965 年那個「太容易實作」的決定的利息：空指標解參照、`NullPointerException`、`undefined is not a function`——同一個地雷，換了無數個語言的外殼，在無數個半夜引爆。

2009 年，已經是計算機科學泰斗、拿過圖靈獎的 Hoare，在 QCon 的一場演講（標題就叫〈Null References: The Billion Dollar Mistake〉）上，公開為這件事道歉。他說他「稱它為我的十億美元錯誤」——粗估這幾十年來 null 造成的錯誤、漏洞、系統崩潰，加總起來的損失大概有十億美元等級。

> ⚠️ 一個你大概也記混的事，務必分清：這場「billion dollar mistake」的道歉，是 **2009 年的 QCon 演講**，**不是**他 1980 年的圖靈獎演說。圖靈獎那場演說標題是〈The Emperor's Old Clothes〉（皇帝的舊衣），講的是另一回事（軟體簡單性與他職涯的教訓）。網路上極多文章把「十億美元錯誤」繫到圖靈獎演說上，那是錯的。本書照查證版：1965 埋雷、2009 道歉、場合是 QCon London 2009。

這個個案攤開後，你該看見的不只是「null 很討厭」這種人盡皆知的牢騷，而是一個**結構性的教訓**：

一個為了「方便」而開的特例，如何變成一筆全球性的複雜度負擔。Hoare 當年面對的約束是真實的——他需要表達「空」，而強制初始化的方案在 1965 年的語言設計與編譯器技術下確實更費工。換成你在那個時空，趕著把語言做出來，大概也會選那條容易的路。這不是他笨，是一個**局部最優的便利決定，在六十年的尺度上累積成了全域的災難**。而你今天打開 `strictNullChecks`，做的正是反向工程：TypeScript 用 `T | null`、`T | undefined` 這種「可空型別」把 null 重新拉回型別系統的管轄之內——逼你在型別層面承認「這個東西可能是空的」，於是編譯器能在你取屬性前攔住你。**你那個 `strict` 開關，是在補 Hoare 1965 年那道裂縫。** 六十年後，型別系統終於把它當年漏掉的那個特例，收編回了編譯期。

## 從 Hoare logic 到型別檢查：理想如何縮水成你每天用的工具

現在把光譜接起來。Hoare logic 是最強的理想（證明任意性質），null 是它被自己戳的洞。那麼，介於「證明一切」和「什麼都不證」之間，有沒有一個甜蜜點——強到能擋掉一大類真實的 bug，又弱到能讓編譯器**自動、即時、零負擔**地檢查？

有。那就是**型別系統**。而讓它從「Algol 那種你得手寫每個型別」進化成「聰明到能自己推出型別」的關鍵一步，是 1970 年代的 ML 與 Hindley–Milner 型別推論。

ALGOL 已經有型別了，但你得把每個變數的型別老老實實寫出來（`int x; real y;`）。1970 年代，Robin Milner 在愛丁堡做一個叫 LCF 的定理證明器時，為它設計了一個元語言（meta-language），就叫 **ML**。ML 帶來一個漂亮的東西：**型別推論（type inference）**——你不必標註每個型別，編譯器能從你怎麼用一個值，**自己推導**出它的型別，而且推導出的型別檢查依然嚴格。背後的演算法後來叫 **Hindley–Milner**（型別系統由 J. Roger Hindley 先描述、Milner 獨立重新發現、Luis Damas 形式化證明，這個三方歸屬本身也是個「誰先發明」的小公案）。它是第一個有**多型型別推論**的實用系統。

為什麼這對本章重要？因為型別推論解決了型別系統最大的政治問題：**嫌麻煩**。動態語言的人嫌靜態型別「要寫一堆型別宣告、囉嗦、拖慢開發」，型別推論直接砍掉這個成本——你享受編譯期檢查的保護，卻幾乎不用多打字。這條血脈直接流到你身上：TypeScript 的型別推論（`const x = 5` 它自己知道 x 是 number，你不用寫）、Rust 的 `let` 推論、現代 C++ 的 `auto`，祖宗都是 1970 年代的 Hindley–Milner。你寫 TS 時那種「型別大多自己長出來、只在介面邊界手動標註」的舒服體驗，是五十年前在一個定理證明器的副產品裡誕生的。

現在可以講出本章第二個關鍵領悟，它來自型別理論家 Benjamin Pierce 在權威教科書《Types and Programming Languages》（2002）裡的一句定調（精確措辭）：型別系統是「**by far the most popular and best established lightweight formal methods**」——目前為止最普及、最成熟的**輕量級形式方法**。

讀慢一點：型別檢查**是一種形式方法**。它不是形式方法的對立面、不是「形式方法太難所以我們用型別湊合」——它就站在那條從 Hoare logic 延伸下來的光譜上，只是把強度調低、把成本壓到趨近於零。你每次按 `tsc`，跑的就是一次微型的、自動化的「程式性質證明」：編譯器在證明「你的程式裡，沒有任何地方把一個型別的值當成不相容的另一個型別來用」。這是個很窄的性質（它不證明你的業務邏輯對、不證明你的迴圈會終止），但它**便宜到你可以每次存檔都跑一遍、零思考成本**。

這就是甜蜜點的祕密：**型別系統用「能證明的性質很窄」換來了「便宜到能融進日常」**。Hoare logic 能證明深刻的性質，但貴到要人類專家手動推導；型別檢查只能證明膚淺的「型別一致」，卻全自動、即時、人人會用——於是它成了史上唯一真正普及到每個工程師日常的形式方法。你以為形式方法是學術象牙塔的東西，其實它最成功的化身，就是你 IDE 裡那條紅色波浪線。

## 靜態 vs 動態：型別之爭的真實 trade-off

既然型別檢查這麼好，為什麼還有一大票人——而且是很厲害的人——堅持用沒有靜態型別的動態語言（JavaScript、Python、Ruby、Clojure）？「型別之爭」吵了幾十年，網路上多半吵成宗教戰爭。我們把它還原成一筆**冷靜的工程帳**。

先講清楚兩邊到底在爭什麼。靜態型別（你 `tsc` 編譯期檢查）和動態型別（值在執行期才帶著型別跑）的差別，不是「有沒有型別」——JavaScript 當然有型別（`typeof` 一下就知道），差別是**型別錯誤在什麼時候被抓到**：

```text
同一個型別錯誤，在哪個時間點被抓到（時間由上往下流）

                  ┌──────────┐
   靜態型別在此抓 │ 你打字時 │ 動態型別：沒人管
                  ├──────────┤
   tsc / 紅波浪線 │  編譯期  │ 動態型別：沒有這步
                  ├──────────┤
        (已沒事)  │  執行期  │ 動態型別：半夜炸在這
                  └──────────┘
   靜態 = 把錯誤往上推、早抓     動態 = 留在執行期、開發當下沒摩擦
```

這張圖就是整場爭論的全部。靜態型別把型別錯誤**往左推**——推到你打字、編譯的當下；動態型別讓它**留在右邊**——一路放行到執行期。兩邊各自的帳是這樣：

| 維度 | 靜態型別（TypeScript） | 動態型別（JavaScript） |
|---|---|---|
| **何時抓到型別錯** | 編譯期，最早 | 執行期，最晚（或測試碰到才抓到） |
| **付出的成本** | 寫型別／標註介面／與型別系統搏鬥（雖然推論減輕了） | 幾乎零前期成本，打開就寫 |
| **回收的價值** | 一整類錯永不進 production；重構時編譯器當安全網；型別即文件 | 開發初期飛快、原型／探索性程式爽 |
| **最痛的時候** | 型別體操（複雜泛型搞到型別比邏輯難寫）、與動態資料邊界搏鬥 | 程式長大後，沒有編譯器擋著，重構像走鋼索、邊界 case 在 prod 炸 |

看出來了嗎——**這是 ch03 那把「效能 vs 生產力」鐘擺的變體。** ch03 講 FORTRAN 時，那場賭是「機器產生的碼能不能跟手寫的一樣快」，本質是「拿一點效能換生產力划不划算」。型別之爭是同一把鐘擺換了座標軸：拿的不是效能，是**早抓錯的安全**；換的不是生產力的全部，是**開發初期的速度與彈性**。靜態型別這邊押「前期多花一點、後期少還很多」；動態型別那邊押「前期快就是王道、錯誤用測試和紀律補」。

而且——這是關鍵——**沒有一邊是絕對對的，因為押注的回報取決於程式的壽命與規模。** 寫一個五十行的腳本、一個一次性的資料清洗、一個探索性的 notebook，動態型別的「零前期成本」幾乎穩賺，你根本活不到「重構時沒安全網」那一天。但寫一個會被五個人維護三年、跑在 production 上服務 50 萬使用者的後端——這正是你的場景——靜態型別那筆「前期多花、後期少還」的帳就壓倒性地划算：你那套全團隊採用的 cache library、那些跨 service 的 DTO，每一次重構若沒有編譯器當安全網，都是在 50 萬使用者面前走鋼索。**你從 JS 轉 TS，不是因為靜態型別「比較對」，是因為你的程式跨過了那條讓這筆帳翻盤的壽命×規模門檻。** 鐘擺沒有正確的一端，只有「對這個專案而言此刻該擺哪邊」。一個資深工程師的標誌，不是死守一邊，是知道自己的專案站在鐘擺的哪個位置、為什麼。

## 形式驗證活在哪裡：為什麼最強那端進不了主流

現在回到光譜最強的那一端——Hoare 1969 的理想：用數學證明程式完全正確。六十年過去，這個理想**沒有**成為主流。你寫後端從來不會「證明」你的 service 正確，你測試它。為什麼？這不是因為理想破滅了——它在某些地方活得好好的——而是一筆**成本帳**算下來的必然結果。

先看它活在哪裡。形式驗證（formal verification，用機器輔助的數學證明確認程式滿足規格）真實地、商業地、救命地運作著，但只在三類地方：

- **晶片**。1994 年 Intel 的 Pentium FDIV bug——浮點除法在極少數情況算錯——逼 Intel 認列約 4.75 億美元的損失把瑕疵晶片換掉。一個 bug 燒掉近五億美元，因為晶片**召回的成本是天文數字、且改不了已出貨的硬體**。這件事直接把形式驗證推進了硬體業：從此晶片大廠用形式方法證明浮點運算單元的正確性，因為證明的成本遠低於再來一次 FDIV。
- **航太與安全攸關系統**。航空軟體的認證標準 DO-178C 明文接受形式驗證作為證據；Airbus 用形式驗證取代部分安全攸關航電程式的單元測試。原因同上：**一個 bug 等於人命與墜機**，本質複雜度的代價極端到任何前期投入都划算。
- **作業系統核心與編譯器**。seL4 是第一個被完整形式驗證的微核心——數學證明它的實作符合規格、沒有特定類別的 bug；CompCert 是一個被證明「編譯出的機器碼忠實對應原始碼」的 C 編譯器，是商業產品，用在航太。

看出共通點了嗎？這三類全是**「錯一次就毀滅性、且修補成本極高或根本沒機會修」**的領域。在這些地方，Dijkstra 那句「測試證明不了 bug 不存在」是會死人的——你需要的就是「不存在」這個等級的保證，而形式驗證是唯一給得起的。

那為什麼它進不了你的後端、進不了主流的網頁與 app？成本帳，三筆：

第一，**寫證明的成本是天文數字**。形式驗證一個系統，往往要花比寫這個系統本身多好幾倍、甚至一個數量級的人力與時間，而且需要稀缺的形式方法專家。seL4 那種等級的驗證是以「人年」計的。你的 CRUD service 用幾天寫完，你不可能花幾個月去證明它。

第二，也是更深的一筆——**就算你證明了，你證明的是「程式符合規格」，不是「規格符合你真正要的東西」**。形式驗證把「程式對不對」這個問題，搬成了「規格對不對」這個問題。但寫出一份完全正確、毫無遺漏地捕捉了你真正意圖的規格，本身就極難——而且這個難，是 ch10 要講的 Brooks〈No Silver Bullet〉裡那個**本質複雜度（essential complexity）**：搞清楚「到底要建什麼」這件事，沒有任何工具能替你做。形式驗證能砍掉「實作有沒有忠於規格」這層偶然複雜度，但「規格有沒有忠於需求」這層本質複雜度，它一點忙都幫不上。你在 production 遇到的多數事故，根源往往不是「程式不符合規格」，而是「我們當初根本沒想到這個情況」——而那種 bug，再完美的證明也擋不住，因為它在規格層面就漏了。

第三，**你的世界本來就不確定**。形式驗證在「規格穩定、變更成本極高」的領域最划算（晶片下線就改不了、航電認證後不能亂動）。但你的後端天天在變——需求每週調整、API 每月演化、上游服務不在你掌控內。為一個下週就要改的 service 寫數學證明，等於為流沙澆灌混凝土。

所以形式驗證沒「失敗」，它只是**找到了自己成本划算的棲位**，並理性地待在那裡。這正是本書反覆出現的母題：**沒有銀彈**（ch10 會正式展開）。最強的正確性工具存在、有效、且不會普及——因為它砍的是偶然複雜度，而軟體最貴的那部分複雜度是本質的，砍不掉。形式驗證沒取代測試，不是因為它不夠好，是因為它和測試在對抗複雜度的不同層面工作：證明保證「實作忠於規格」，測試探查「規格之外我沒想到的現實」，而後者——「我沒想到」——是人類認知的本質限制，沒有工具能根除。

（光譜中間還有一檔，介於「完整證明」和「型別檢查」之間：**形式規格**——你不證明整支程式，但用一套精確的數學語言描述「系統該滿足什麼性質」，再用工具檢查模型。你書架上那本《把系統寫成定理：TLA+ 與形式化方法》就把這條路走到底——TLA+ 讓你對「並行系統會不會違反某條不變式」做窮舉檢查，特別適合你那種分散式、最終一致性、冪等消費者的場景。本書不依賴它，但若你想真的動手摸「事前證明」的中間檔位，那本書是入口。這裡點到為止。）

## 複雜度搬去哪了

這一跳（用型別系統／形式方法在執行前排除錯誤）搬動的複雜度，記成一筆帳：

| 項目 | 內容 |
|---|---|
| **消滅了什麼負擔** | 把「一整類錯誤」從執行期搬到執行前——型別檢查讓「型別不一致」這類 bug 永遠進不了 production；形式驗證讓「實作不符規格」這類 bug 在數學上被排除。最直接的：你不再在半夜 debug 一個三層之前就空掉的 `undefined`，編譯器在你打字時就攔住它 |
| **製造了什麼新負擔** | (1) 你得**前置**地把意圖寫成型別／規格——複雜度從「執行期 debug」搬到「開發期標註與設計」。(2) 型別系統本身會長成一門學問——型別體操、複雜泛型、與動態邊界（外部 API、JSON）搏鬥。(3) 形式驗證把「程式對不對」搬成「規格對不對」，而寫對規格是本質複雜度，更難。(4) 強度越高、成本越爆炸——光譜往左走一格，付出的人力是數量級跳升 |
| **在什麼情況下洩漏／反噬** | 型別系統的洩漏點：邊界。編譯期型別保證**止於程式邊界**——一旦資料從外部進來（API response、`JSON.parse`、`any`、資料庫的 raw row），型別系統就瞎了，你用 `as` 強制斷言的那一刻就在說謊，謊言會在執行期現形。形式驗證的洩漏點：規格與現實的縫——你證明了程式符合規格，但規格漏掉了真實世界的某個情況，證明擋不住那個 bug |
| **這一跳的帳單一句話** | **複雜度從「執行期的驚嚇」搬到了「開發期的紀律」**——你用「現在多想一點、多寫一點」換「將來少炸一點」；但你能擋掉的只到「型別一致／符合規格」為止，「我到底想要什麼」這層本質複雜度，光譜上沒有任何一檔工具砍得掉 |

最後一格是這章的核心，也是它和全書接榫的地方。型別與形式方法沒有消滅 bug，它們把「對抗 bug」的工作**從事後搬到了事前**——從執行期的偵錯，搬到開發期的標註、設計與證明。這是一筆好交易（早抓的錯便宜得多），但它有一條鐵打的天花板：**這些工具擋的都是「偶然複雜度」類的錯（型別、實作、語法），擋不掉「本質複雜度」類的錯（我們根本搞錯了要建什麼）。** 你那個 `strictNullChecks` 補好了 Hoare 1965 的洞，但它救不了「這個功能本來就不該這樣設計」這種根上的錯——那是 ch10 沒有銀彈、ch19 抽象洩漏要繼續算的帳。型別系統把 null 收編回編譯期，是這條光譜六十年來最普及的一次小勝；而最強那端的形式驗證，依然被供在航太與晶片的神壇上，理性地拒絕走下來——因為它知道，自己砍不掉那筆最貴的本質複雜度。

## 紙上推演

### 推演題一：用「複雜度搬家」拆解 `strictNullChecks` **[15 分鐘] ★★**

你打開 TypeScript 的 `strictNullChecks`。用本書的「複雜度搬家」框架精確回答：(a) 它把哪一類負擔，從哪個時間點，搬到了哪個時間點？(b) 你為這次搬家付的帳單具體是什麼（你打字時多了什麼麻煩）？(c) 把這件事接回 Hoare 1965——`strictNullChecks` 在「補」什麼？

#### 推演解答

(a) 它把「對可能為空的值取屬性」這一類錯誤，從**執行期**（半夜在 production 炸 `undefined is not a function`）搬到了**開發期**（你打字時的紅色波浪線）。本質是把一類 bug 沿著本章那條強度光譜，從「事後跑跑看才發現」往左推到「事前分析就拒絕」。

(b) 帳單：你不能再假設值都存在了。每個可能為空的值，型別變成 `T | undefined`，你被迫在用它之前**顯式處理空的情況**——加判斷、加 optional chaining（`?.`）、加 early return、或在型別層面證明它非空。打字當下多了摩擦，紅波浪線一開始鋪滿螢幕。但這摩擦的位置，精確地等於過去會炸的位置——你只是把 debug 的痛，預付成了打字的麻煩。

(c) `strictNullChecks` 在補 Hoare 1965 那道裂縫。Hoare 當年讓 null 能塞進任何參照型別，等於在型別系統上開了一個官方後門，讓「可能為空」這件事**逃出了型別系統的管轄**——編譯器看一個 `Person` 看不出它其實可能是 null。TypeScript 的可空型別（`T | null`、`T | undefined`）把 null **拉回型別系統的帳本**：強迫你在型別層面承認「這東西可能空」，於是編譯器重新能管它。一句話：六十年後，型別系統終於把當年漏掉的那個特例收編回了編譯期。常見誤解：以為 `strictNullChecks` 是「TypeScript 的一個 lint 嚴格度選項」——不，它是在修一個 1965 年的語言設計缺陷的繼承後果。

### 推演題二：為什麼形式驗證沒有取代測試 **[20 分鐘] ★★★**

形式驗證能用數學「證明」程式正確，比測試強得多。可是你的後端從來只測試、不證明，全世界絕大多數軟體也一樣。用**成本帳**和**本質 vs 偶然複雜度**論證：為什麼最強的正確性工具存在、有效，卻不會普及？（這題預告 ch10 的「沒有銀彈」。）

#### 推演解答

分三層論證。

第一層，**寫證明貴到數量級**。形式驗證一個系統的人力，常是寫它本身的數倍到一個數量級，且需要稀缺的形式方法專家（seL4 的驗證以人年計）。你幾天寫完的 CRUD service，不可能花幾個月證明——成本和回報不成比例。

第二層，也是更深的——**證明只把問題搬家，沒消滅**。形式驗證證明的是「程式符合規格」，它把「程式對不對」搬成了「規格對不對」。但寫一份完整、無遺漏地捕捉真正意圖的規格，本身極難，而且這個難是**本質複雜度**：搞清楚「到底要建什麼」沒有工具能代勞。你 production 上的多數事故，根源是「我們根本沒想到這情況」——這種 bug 在規格層面就漏了，再完美的證明也擋不住，因為證明只保證實作忠於規格，不保證規格忠於現實。

第三層，把它接到 Brooks（ch10 正式展開）：形式驗證砍的是**偶然複雜度**（實作有沒有忠於規格、型別對不對），砍不掉**本質複雜度**（理解問題本身、規格忠於需求）。而軟體最貴的複雜度是本質的。所以「沒有銀彈」：最強的工具也只能砍偶然複雜度那一半。

結論：形式驗證沒「失敗」，它找到了成本划算的棲位——錯一次就毀滅性、規格相對穩定的領域（晶片、航太、核心）——並理性地待在那裡。它和測試不是替代關係，是分工：證明保證「實作忠於規格」，測試探查「規格之外我沒想到的現實」。後者是人類認知的本質限制，無法被根除，所以測試永遠不會消失。常見誤解：以為「形式驗證沒普及＝它不夠成熟、再等幾年就行」。不是。是它砍不到本質複雜度那一半，這是原理上的天花板，不是技術成熟度問題。

### 推演題三（對帳）：你的 OpenAPI schema 算不算一種形式方法 **[15 分鐘] ★★**

你在 CI 裡強制 OpenAPI schema 同步——schema 一旦和實作不一致，CI 就紅。把這個機制放到本章那條「執行前就排除錯誤」的強度光譜上：(a) 它落在哪一檔，憑什麼說它是（或不是）形式方法？(b) 它擋掉了哪一類錯、擋不掉哪一類？(c) 它的「洩漏點」在哪？

#### 推演解答

(a) 它落在光譜中間偏弱、靠近「形式規格／型別系統」那一帶，**算一種輕量形式方法**。理由：OpenAPI schema 是一份**精確的、機器可檢查的規格**——它用一套形式語言描述「這個 API 的請求／回應該長什麼型別」。你在 CI 強制它與實作同步，等於用自動化工具**在執行前**檢查「實作符合這份規格」。這完全是本章的衝動（事前、分析式、排除一類錯）的低強度版，和 Pierce 說型別檢查是「最普及的輕量形式方法」是同一個道理——你只是把型別檢查從「函式邊界」擴大到了「服務邊界」。

(b) 擋掉的：「實作回傳的形狀和契約不一致」這類錯——欄位少了、型別變了、必填變選填，CI 在合併前就攔下，不會等到前端串接或下游消費者在執行期炸。也擋掉「文件與實作不同步」這個經典的人禍。擋不掉的：schema 本身**對不對**——如果你的 schema 設計就錯了（漏了一個狀態、把一個該必填的設成選填、語意理解錯了），CI 全綠，但 API 的行為依然不是使用者要的。這精確對應推演題二的第二層：它保證「實作忠於 schema」，不保證「schema 忠於需求」。

(c) 洩漏點和所有型別系統一樣，在**邊界**。OpenAPI 的保證止於「被它描述的那層介面」。一旦資料穿過邊界——你在 handler 裡 `as any`、上游傳來 schema 沒涵蓋的欄位、執行期的真實 payload 和宣告的型別不符（schema 是「宣稱」，執行期未必驗證每個 byte）——保證就漏了。而且它管的是**形狀**（型別、結構），管不了**業務不變式**（「折扣不能超過原價」這種跨欄位的語意規則）——那種約束要更強的形式規格（如 TLA+，見書架）或執行期驗證才擋得住。一句話：OpenAPI 是型別系統的服務間版本，它的力量和它的盲點，都和 `tsc` 完全同源——強在形狀、瞎在邊界與語意。

## 自我檢核

口頭自答，講得出因果才算過：

1. 為什麼說「靜態型別」和「形式驗證」是同一種衝動的不同強度，而不是兩個不同的領域？（提示：用那條強度光譜講——它們想做的同一件事是什麼？測試為什麼不在這條光譜上、或在哪一端？）
2. Hoare logic 的 `{P} S {Q}` 在表達什麼？它的野心（相對於測試）為什麼是對 Dijkstra「測試證明不了 bug 不存在」那句話的正面回應？
3. 把 null 那個個案完整講一遍：Hoare 1965 為什麼加 null、它為什麼從根上戳穿了他自己那套型別系統的保證、為什麼這個決定會遺傳給後世幾乎每個語言？
4. 「billion dollar mistake」是哪一場演講、哪一年？為什麼說「把它繫到圖靈獎演說」是錯的？（圖靈獎那場演說叫什麼、講的是別的事。）
5. Pierce 說型別系統是「最普及的輕量形式方法」——這句話為什麼重要？「型別檢查是一種形式方法」這個說法，改變了你怎麼看待你每天按的 `tsc`？
6. 靜態 vs 動態型別的真實 trade-off 是什麼？為什麼說它是 ch03「效能 vs 生產力」鐘擺的變體、為什麼沒有絕對正確的一邊？（用「壽命×規模門檻」解釋你自己從 JS 轉 TS 的決定。）
7. 形式驗證真實地活在哪三類領域？這三類有什麼共通點？為什麼它在那裡划算、在你的後端不划算？（要講出成本帳的三筆，特別是「證明只把問題搬成『規格對不對』」那筆。）
8. 為什麼形式驗證沒有、也不會取代測試？用「本質複雜度 vs 偶然複雜度」講——它砍得掉哪一半、砍不掉哪一半？（這題你該能把答案接到 ch10 沒有銀彈。）

## 延伸閱讀

- **Tony Hoare,〈Null References: The Billion Dollar Mistake〉, QCon 2009（InfoQ 有完整錄影）**（https://www.infoq.com/presentations/Null-References-The-Billion-Dollar-Mistake-Tony-Hoare/）。本章個案的一手出處，**務必看本人親口講**。重點看他怎麼描述 1965 年那個「太容易實作」的決定——你會發現一個圖靈獎得主的自我反省，比任何「null 很爛」的牢騷都有份量。順帶親耳確認：這是 2009 QCon，不是 1980 圖靈獎演說。
- **C. A. R. Hoare,〈An Axiomatic Basis for Computer Programming〉, *CACM* 1969**（https://dl.acm.org/doi/10.1145/363235.363259 ；經典 PDF 多處可得）。Hoare logic 的原典，只有幾頁。即使你不打算讀完那些推論規則，讀它的引言——感受 1969 年有人認真想「像證明數學定理一樣證明程式正確」這件事的野心，是理解整條光譜「強的那一端」的鑰匙。
- **Benjamin C. Pierce, *Types and Programming Languages*（TAPL）, MIT Press 2002——讀「Preface / 第 1 章 Introduction」**（https://www.cis.upenn.edu/~bcpierce/tapl/）。型別系統的權威教科書，但你**不必讀內文**（那是型別理論硬核，本書明言不展開）。只讀引言那幾頁，看他怎麼把型別系統定位成「最普及的輕量形式方法」——這幾頁就把本章的主軸講透了。System F、dependent types 想深入，這本是入口。
- **Gerwin Klein 等, seL4: Formal Verification of an OS Kernel（CACM 摘要版）**（https://cacm.acm.org/research/sel4-formal-verification-of-an-operating-system-kernel/）。想知道「形式驗證真的拿來驗一個能跑的系統」長什麼樣，看這篇。重點不是技術細節，是感受「為了證明一個微核心無 bug 要付出的人力規模」——你會立刻明白為什麼它進不了你的後端。
- **書架上的《把系統寫成定理：TLA+ 與形式化方法》**。本章光譜的「中間檔位」（形式規格／模型檢查）那本書走到底，特別針對你那種分散式、並行、最終一致性的場景——如果你想真的動手摸「事前證明」而不只是讀歷史，那是下一步。本書到此為止，不展開。
