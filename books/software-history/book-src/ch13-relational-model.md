## 從你已知的出發

你寫過的 SQL，這輩子大概數不清了。隨手一句：

```sql
SELECT u.name, COUNT(o.id) AS order_count
FROM users u
JOIN orders o ON o.user_id = u.id
WHERE u.created_at > '2026-01-01'
GROUP BY u.id
HAVING COUNT(o.id) > 5;
```

你寫這句的時候在想什麼？你在想**你要什麼**：建立在 2026 年後的使用者裡，下單超過五次的，給我名字和單數。你**沒有**在想：先掃 `users` 還是先掃 `orders`？走哪個 index？用 nested loop join 還是 hash join？要不要先把 `orders` 依 `user_id` 排序再 merge？這些「**怎麼拿**」的決定，你一個都沒下——你把它們整包丟給了資料庫引擎的**查詢最佳化器（query optimizer）**，它讀了你的 schema、看了 index、估了每張表的列數與資料分布，自己挑了一條執行計畫。

這件事有多奢侈，要等你親眼看過它的代價才會懂。回想你那個招牌戰績：某次尖峰時段 RDS 的 CPU 爆到頂，你下去 profiling，發現是一句查詢漏了 index，全表掃描。你做了什麼？你**加了一個 index**，CPU 降了 40%。注意這裡發生了什麼魔法——**你改了資料「怎麼被存取」（多了一個 B-tree），但你一行應用程式碼都沒動**。那句 `SELECT` 原封不動，它跑得快了一個數量級。應用程式根本不知道、也不需要知道底下多了個 index。

這個「改存取方式不用改程式」的能力，有個名字，叫**資料獨立性（data independence）**。它不是天上掉下來的。1970 年以前，這件事是**做不到的**——那時候，「怎麼存取」這件事是用手寫死在你程式裡的，你想換個存取路徑，就得改一堆程式。你今天能加 index、能調 isolation level、能設 read replica 把讀流量導走、能對大表做 partition——這些調校全都建立在一個前提上：**底層怎麼變，上層的查詢語意不變**。這個前提，是一個叫 Edgar F. Codd 的人在 1970 年用一篇論文掙來的。

而當你在某些場景放棄了 SQL、改用 Redis 或某種 NoSQL——比如你拿 Redis 當 cache 或做 Pub/Sub、用最終一致性處理 SQS 訊息——你其實是在做一個更深的決定：你**主動放棄了 Codd 那套引擎包辦一切的奢侈**，換取規模或延遲，自己把一致性的擔子扛回了應用層。這也不是新鮮事。它是 2009 年前後一整代工程師集體做過的同一個選擇，背後有一條定律（CAP）在逼著你選邊。

本章講的就是這條從「自己手寫存取路徑」到「引擎包辦」再到「為了規模又把一部分擔子要回來」的完整弧線。你天天站在這條弧線的某個點上，現在我們把它從頭走一遍。

## 前關聯時代：程式設計師是「領航員」

要懂 Codd 翻過的是哪面牆，得先看牆的另一邊長什麼樣。

1960 年代，企業開始把大量資料放進電腦——銀行帳戶、庫存、訂單、保單。當時的資料庫主要有兩種模型：**階層式（hierarchical）**，代表是 IBM 的 **IMS**（Information Management System，1960 年代末隨 Apollo 計畫而生），資料組成一棵樹；以及**網狀式（network）**，由 CODASYL committee 標準化（CODASYL 標準約 1971 年定案），資料組成一張用指標串起來的網。這兩種後來被合稱為**導航式資料庫（navigational database）**——因為存取資料的方式，是「導航」。

導航是什麼意思？意思是：你的程式想找一筆資料，得**自己沿著指標一個一個跳**。資料庫開啟時給你一個起點記錄，那筆記錄裡有指向其他記錄的指標；你要找的東西不在這筆，就順著指標跳到下一筆，再下一筆，直到找到為止。**存取路徑（access path）——先讀哪筆、沿哪條指標鏈走、走訪的迴圈怎麼寫——全部是程式設計師手寫在應用程式裡的。**

這個世界觀有一個再貼切不過的官方代言。1973 年，網狀資料庫的主要設計者 **Charles Bachman** 拿了圖靈獎，他的得獎演說標題就叫——**〈The Programmer as Navigator〉（程式設計師作為領航員，刊於 *CACM* 1973 年 11 月）**。Bachman 在台上熱情地描繪一個新世界：程式設計師應該把自己想成一個在資料的「空間」裡航行的領航員，從一筆記錄出發，熟練地在記錄之間穿梭、跳轉、定位。對 Bachman 來說，這是進步——程式設計師終於從「死記憶體位址」裡解放出來，能在更高層的記錄結構間導航了。

從今天回看，這個比喻精準到反諷。它精準地點出了那個時代的全部複雜度，然後把它當成榮耀。「領航員」是什麼意思？意思是**你得知道路**。你得知道資料是怎麼擺的、哪筆連到哪筆、走哪條路最快。這些知識——資料的物理結構與存取路徑——被焊死在你的應用程式裡。後果是一連串的脆弱：

- **存取路徑相依（access path dependence）**：你的程式寫死了「沿著這條指標鏈走」。哪天 DBA 為了效能改了資料的組織方式、調整了指標結構，你那段導航程式就**全錯了**，得重寫。
- **順序相依（ordering dependence）**：程式假設了記錄的物理排列順序。底層一換排法，邏輯就崩。
- **索引相依（indexing dependence）**：要加一個 index 或拿掉一個 index，可能逼著應用程式跟著改——因為程式知道、也依賴 index 的存在。

把這三條合起來看：**在前關聯時代，「資料怎麼存」和「程式怎麼寫」是死死綁在一起的。** 你動不了其中一個而不動另一個。這正是 ch06 Parnas 說的「資訊隱藏」在資料層的反面教材——導航式資料庫把「資料的物理結構」這個**最該被藏起來、最可能改變的決策**，大喇喇地暴露給每一支應用程式，於是每一次物理變更都漏成上層的災難。

用你的語言講：想像你的 ORM 不存在，連 SQL 都不存在。你要查「2026 年後下單超過五次的使用者」，你得**自己寫一個迴圈**：開啟使用者檔，拿到第一筆，檢查建立日期，是的話順著這個使用者的「訂單指標鏈」一筆一筆數他的訂單，數完超過五就收進結果，然後跳到下一個使用者……而且這條走訪路徑，你還得確保它走的是當前最快的那條，不然效能慘不忍睹。你不只要說「我要什麼」，你得從頭到尾規定「怎麼拿」。這就是 Bachman 口中那個值得驕傲的「領航」。Codd 看著這個世界，覺得這整件事根本就錯了。

## Codd 1970：把「怎麼拿」整包搬給引擎

Edgar F. Codd 是 IBM San Jose 研究中心的研究員，數學家出身。1970 年 6 月，他在 *CACM* 發表了 **〈A Relational Model of Data for Large Shared Data Banks〉**。這篇論文，我認為是軟體史上影響力最被「日用而不知」掩蓋的一篇——它太成功了，成功到它解決的問題今天的人根本想像不到曾經是個問題。

Codd 的核心洞見可以用一句話講：**把資料的「邏輯結構」和「物理結構」徹底切開，讓程式只面對邏輯結構。**

他的武器是數學裡的「關係（relation）」——白話講就是**表（table）**：列（row）是元組、欄（column）是屬性，沒有指標、沒有指向別筆記錄的鏈、沒有規定的順序。表與表之間的關聯不靠物理指標，而靠**值**（一張表的某欄值，去對應另一張表的某欄值——也就是你天天寫的 JOIN）。配套的是**關聯代數（relational algebra）**：selection、projection、join 這些集合運算，定義了「可以對表做哪些操作」，而且這些操作的定義**完全不提資料在磁碟上長什麼樣**。

但 Codd 真正的目標，論文裡寫得很白——不是「表」這個資料結構本身，而是**資料獨立性（data independence）**。他把它定義為「應用程式與終端活動，獨立於資料型別的增長與資料表示方式的變更」。他在論文裡明確點名要消滅前面那三種相依：**存取路徑相依、順序相依、索引相依**——這三條，正是導航式資料庫把程式設計師釘死在物理結構上的三根釘子。Codd 要把它們一次拔掉。

拔掉之後是什麼世界？是**宣告式（declarative）**的世界。這是整章的樞紐，值得停下來把它和你天天做的事對齊：

- **導航式（命令式）**：你告訴電腦**怎麼做**——開檔、拿第一筆、檢查、沿指標跳、迴圈、收集。你是領航員，你得知道路。
- **關聯式（宣告式）**：你告訴電腦**你要什麼**——「給我滿足這些條件的列」。**怎麼拿，是引擎的事，不是你的事。**

這個切換把一塊巨大的複雜度從**每一支應用程式**裡，搬到了**一個地方**——資料庫引擎。而引擎裡負責「決定怎麼拿」的那個元件，就是**查詢最佳化器**。它讀你的宣告式查詢，自己決定走哪個 index、用哪種 join 演算法、先過濾哪張表。你那句 `SELECT` 不再規定路徑，它只描述目標；路徑由最佳化器當下、根據當下的資料分布與可用 index，動態決定。

這也解釋了你那個「加 index 不用改程式」的魔法為什麼成立。因為你的查詢從一開始就**沒有**承諾任何存取路徑——它只說了要什麼。你加 index，等於給了最佳化器一條新路；它發現新路更快，就改走新路。你的查詢一個字都不用改，因為它本來就把「怎麼走」的決定權讓給了引擎。**這就是資料獨立性，這就是 Codd 1970 搬給你的奢侈。**

> **一個誠實的歷史補丁**：Codd 的論文一開始在 IBM 內部並不順利。IBM 當時的金雞母是階層式的 IMS，內部有龐大的既得利益。Bachman 陣營（導航派）和 Codd 陣營（關聯派）在 1970 年代初是真刀真槍辯論過的——導航派認為關聯模型「太慢、不切實際」，因為讓引擎自動決定路徑，怎麼可能比人手工調的路徑快？這個質疑你應該覺得似曾相識：它和 ch03 裡「編譯器產生的碼怎麼可能比手寫機器碼快」是**同一個質疑、同一條鐘擺**（效能 ↔ 生產力）。而答案也一樣：一開始確實慢，但當「自動的那一方」（最佳化器／編譯器）被打磨到夠好，生產力的勝利就壓倒了效能的疑慮。

## SQL 與 ACID：把勝利標準化

Codd 給了模型，但模型要落地，得有兩樣東西：一個讓人能用的**語言**，和一套讓人敢把錢存進去的**保證**。

**語言：SEQUEL → SQL（1974）。** Codd 的關聯代數是數學家的語言，一般程式設計師看了頭痛。IBM 的 **System R** 計畫（1970 年代中，第一個實作 Codd 模型的原型系統）需要一個給人用的查詢語言，於是 **Donald Chamberlin 與 Raymond Boyce 在 1974 年的 SIGFIDET 研討會發表了 SEQUEL（Structured English Query Language）**。設計目標很明確：用**英文關鍵字**（SELECT、FROM、WHERE、GROUP BY、ORDER BY）讓沒受過電腦訓練的人也能描述查詢。後來因為「SEQUEL」這個名字撞了別家商標，縮成了 **SQL**。（Boyce 在 1974 年就英年早逝，沒能看到自己參與的東西成為往後半世紀資料界的通用語。）

Chamberlin 後來在口述歷史裡，把當年那場路線之爭講得再清楚不過，大意是：Bachman 認為你該寫一支**程序式（procedural）**的程式，一步步告訴電腦怎麼找到答案；而 Codd 說「不，這全錯了，我們該用**宣告式（declarative）**的方法，讓電腦自己想辦法找到答案」。（這段是轉述其口述歷史的論點，非逐字引文。）SQL 就是把 Codd 這個「你說要什麼、電腦想怎麼拿」的哲學，包裝成一個人類能讀能寫的介面。你每天寫的 SQL，骨子裡就是這場 1970 年代路線之爭的勝方留下的遺產。

**保證：ACID（一詞 1983）。** 光能查還不夠。資料庫要存銀行帳戶，就得回答一個要命的問題：**一筆操作做到一半掛了會怎樣？** 兩個帳戶轉帳，扣了 A 的錢、還沒加到 B 的帳上，這時候斷電，錢去哪了？這類「交易（transaction）的可靠性」複雜度，需要被標準化地解決。

這裡有個歸屬要照史實講清楚：**ACID 這個縮寫是 Theo Härder 與 Andreas Reuter 在 1983 年的論文裡造的**（〈Principles of Transaction-Oriented Database Recovery〉）。但概念的奠基者是 **Jim Gray**——他在 1970 年代末就刻畫了交易的性質，命名了 **atomicity（原子性）、consistency（一致性）、durability（持久性）**，但**沒有包含 isolation（隔離性）**；isolation 是 Härder 與 Reuter 補進去湊成 ACID 四件套的。對你來說這四個字一點不陌生——你天天在跟它們的帳單打交道：

- **Atomicity**：轉帳要嘛全成、要嘛全不成，不會卡在中間。
- **Consistency**：交易前後資料庫都滿足約束（外鍵、唯一性…）。
- **Isolation**：併發的交易互不干擾——而你早就知道，這個「互不干擾」是有**等級**的。你調過 isolation level，你處理過 deadlock，你知道 `READ COMMITTED` 和 `SERIALIZABLE` 之間隔著效能與正確性的取捨。這一整片你熟悉的戰場，就是 ACID 的 I 在現實裡的展開。
- **Durability**：commit 了就不會因為當機而消失（靠 write-ahead log 之類的機制）。

所以「關聯模型」這個勝利，其實是三件事疊起來的：**Codd 的模型**（資料獨立性）＋**SQL**（人用得了的宣告式語言）＋**ACID**（交易可靠性的標準保證）。三者合體，造就了往後三十年資料庫的霸主地位，也造就了你「寫 SQL、調 index、設 isolation」這份工作的全部前提。

## 個案：同一個查詢，兩個時代

把抽象拉回地面。我們拿一個具體查詢，看它在兩個時代分別要你做什麼。任務：**找出 2026 年後註冊、且下單超過五次的使用者名字**。

**前關聯時代（導航式，命令式）**，你得在應用程式裡親手寫出全部存取邏輯，大意像這樣：

```text
開啟 USERS 檔，取得第一筆使用者記錄
迴圈 每一筆使用者:
    若 該使用者.註冊日 <= 2026-01-01: 跳下一筆
    訂單數 = 0
    沿著 該使用者 的「訂單指標鏈」走:        ← 存取路徑寫死在這
        每跳到一筆訂單記錄: 訂單數 += 1
    若 訂單數 > 5: 把 該使用者.名字 收進結果
    沿「下一位使用者」指標跳到下一筆          ← 走訪迴圈也寫死在這
回傳 結果
```

你看這段程式裡承載了多少「怎麼拿」的知識：你知道使用者之間靠指標串成鏈、知道每個使用者掛著一條訂單指標鏈、知道要怎麼走訪。這些知識一旦被 DBA 改動（比如為了效能把訂單改成另一種組織方式），**這整段都得重寫**。你是領航員，你扛著全部的路線圖。

**關聯時代（宣告式）**，同一個任務，本章開頭那句 SQL 就是全部：

```sql
SELECT u.name
FROM users u JOIN orders o ON o.user_id = u.id
WHERE u.created_at > '2026-01-01'
GROUP BY u.id HAVING COUNT(o.id) > 5;
```

差別不在「短」。差別在於：**這段裡沒有任何一個字在講「怎麼拿」。** 沒有迴圈、沒有指標、沒有走訪順序、沒有先掃哪張表。它只描述了結果該滿足的條件。「怎麼拿」整包被搬進了最佳化器——它會自己決定：是先用 `created_at` 的 index 過濾 `users` 再去 join `orders`，還是反過來；用 hash join 還是 nested loop；要不要先聚合。而且它的決定會**隨資料量變化**——表小的時候全表掃可能更快，表大了它會改走 index。你完全不用管。

這就是 Codd 搬走的複雜度，攤在你眼前：**前一個時代，「存取路徑」這塊複雜度乘上你所有的應用程式；關聯時代，它被收斂到一個地方（最佳化器），對每一支應用程式都隱形了。** 這是抽象 strand 教科書級的一次勝利——和 ch03 的編譯器同型：編譯器把「怎麼對機器說話」收進一個地方，最佳化器把「怎麼從磁碟拿資料」收進一個地方。兩者都是「把怎麼做外包，讓你只管要什麼」。

而這也正是 ch01 那個全書縮影 **ORM** 的故事的下半場。ORM 想做的事，是在 SQL 這層宣告式抽象**之上**再加一層——讓你連 SQL 都不用寫，操作物件就好。它把「物件 ↔ 表」的對應這塊複雜度也搬走了。但搬得乾不乾淨？這就要講帳單了。

## 複雜度搬去哪了

Codd 的勝利不是免費的。複雜度守恆——「怎麼拿」這塊複雜度沒有消失，它被搬進了引擎。大部分時候引擎扛得很好，你享受著資料獨立性的奢侈。但抽象總會洩漏（這條定律 ch19 會正式講），而關聯模型洩漏的地方，恰好就是你職業生涯花最多時間 debug 的那些地方。

| 這一跳 | 搬走了什麼複雜度 | 製造了什麼新複雜度（帳單在哪洩漏） |
|---|---|---|
| 導航式 → 關聯式（Codd 1970） | 「存取路徑、走訪迴圈」從每支程式裡消失，收進最佳化器；資料獨立性讓你改物理結構不改程式 | **最佳化器猜錯**：它靠統計（表列數、資料分布）估執行計畫，統計過期或資料傾斜時它選錯路，一句該毫秒回的查詢變成全表掃；你得學會讀 `EXPLAIN`、手動 hint、更新統計——**你被迫部分變回領航員** |
| 命令式 → 宣告式 SQL | 你只說要什麼，引擎決定怎麼拿 | **抽象不透明**：因為你不寫路徑，你也就「看不見」路徑——慢查詢的根因藏在你看不到的執行計畫裡，要 profiling 才挖得出來（你的 RDS CPU 降 40% 就是這種考古） |
| 表 ↔ 物件（ORM，建在 SQL 之上） | 你連 SQL 都不寫，操作物件就好 | **N+1 query**：你以為在操作物件，ORM 偷偷對每個物件發一句 SQL——拿 100 個使用者再各拿其訂單＝1+100 句查詢。這是 ch01 預告的全書縮影：ORM 把 SQL 的複雜度搬進框架，**帳單在你不知不覺發出 N 句查詢時回來找你**（這是抽象洩漏的標本，ch19 回收） |
| 單機 ACID → 規模 | （下一節）為了橫向擴展，放棄強一致 | **CAP 的帳單**：你得自己處理最終一致性、衝突解決、冪等——複雜度從引擎搬回了你的應用層 |

一句話總結這一跳的帳單：**Codd 把「怎麼拿資料」這塊複雜度從你的程式裡搬進了引擎，代價是當引擎猜錯、或當你疊太多層抽象（ORM）、或當規模逼引擎放棄保證時，那塊複雜度會以「慢查詢／N+1／一致性 bug」的形式漏回來找你——而且因為它本來被藏起來了，漏回來時特別難 debug。** 你值錢的地方，正在於抽象漏的時候你能跳下去讀執行計畫、能看穿 ORM 發了幾句 SQL。

## NoSQL 的反動：規模逼你把擔子要回來

關聯模型統治了三十年。然後在 2000 年代中後期，一群人開始**主動放棄**它的一部分。不是因為 Codd 錯了，而是因為他們撞上了一面 Codd 沒打算翻的牆：**規模**。

故事的主角是當年資料量爆炸的幾家公司。Google 要索引整個網路、Amazon 要扛全球購物車——它們的資料大到**單機放不下**，必須橫跨成千上萬台機器、跨越多個機房。而傳統關聯資料庫的 ACID 保證，是為「單機、或緊密耦合的少數機器」設計的；一旦你的資料散在地球兩端的機房，機房之間的網路隨時可能斷（network partition，這對你不陌生——你處理過分散式系統的網路分區），這時候「強一致性」和「可用性」就會打架。

技術源頭有兩篇論文，照史實要分清：**Google 的 Bigtable 論文（2006，OSDI）** 與 **Amazon 的 Dynamo 論文（2007，SOSP）**。這兩篇展示了「為了規模與可用性，可以放棄關聯模型與強一致性」的具體做法——Dynamo 尤其直白地擁抱最終一致性。它們是後來整場運動的技術藍本。

而逼著大家在「一致性」和「可用性」之間選邊的，是一條定律：**CAP**。順序要講對：**Eric Brewer 在 2000 年的 PODC 會議上提出 CAP 作為一個猜想（conjecture，他 1999 年就先講過）；2002 年由 Seth Gilbert 與 Nancy Lynch 證明成定理。** CAP 說的是：在一個會發生網路分區（Partition）的分散式系統裡，當分區真的發生時，你只能在**一致性（Consistency）**和**可用性（Availability）**之間二選一——分區兩邊要嘛拒絕服務以保持一致（選 C），要嘛繼續服務但可能讀到舊資料（選 A）。你不能在分區下同時要到 C 和 A。（CAP 的精確語意、以及它後來被 Brewer 自己澄清的諸多細節，分散式一致性的協定層面，本章點到為止——你書架上的 TLA+ 書把「分散式系統的正確性怎麼證」這條路走到底，這裡不展開。）

NoSQL 運動就是這個取捨的產品化。這裡有個**名字的陷阱**，照史實必須講清楚：

- **1998 年，Carlo Strozzi** 用過「NoSQL」這個詞，但指的是一個**仍然是關聯式、只是不用 SQL 介面**的東西。他自己後來說，2009 年那場運動既然根本背離了關聯模型，更該叫「NoREL」（not relational）才對。
- **2009 年**，「NoSQL」這個詞被**重新借用**：在一場由 Johan Oskarsson 在舊金山主辦的 meetup 上，**Eric Evans 提議**用這個詞來指「開源、分散式、非關聯」的那一批新資料庫。**這個 2009 的 NoSQL，和 1998 的 Strozzi NoSQL 是不同的所指**——前者背離關聯模型，後者只是換掉查詢語言。流行敘事常把兩者混為一談，別跟著錯。

「NoSQL」這個詞本身有點命名失敗（它聽起來像「反 SQL」，其實更準的意思是「不只是 SQL／非關聯」，後來很多人改讀成 "Not Only SQL"）。但它指向的轉變是真的：**鐘擺從「引擎包辦一切」擺回「應用自己處理一致性」。** 當你選了一個 eventual consistency 的 NoSQL，引擎不再替你保證「讀到的一定是最新的」——這塊一致性的複雜度，被退回給了你的應用層。於是你得自己做：冪等的寫入、衝突解決、版本向量、補償交易……這些你在 SQS 結算 pipeline 裡天天做的事，本質上就是在**手工補回引擎放棄掉的那部分一致性保證**。

把你的日常對到這條線上：你拿 **Redis** 當 cache 或 Pub/Sub，你就是在 CAP 上選了**可用性與低延遲**，放棄了「Redis 裡的值和資料庫永遠一致」——所以你得處理 cache 失效（cache invalidation，CS 兩大難題之一，ch01 提過）。你對某個寫多讀少、能容忍短暫不一致的負載選了某種 NoSQL，你就是在替那個負載做 Brewer 逼你做的那道選擇題。**你不是在用一個比 SQL「更新」的東西；你是在一條 1970 年就開始、2009 年回擺的鐘擺上，為這個負載挑了一邊站。**

> **鐘擺要誠實**：NoSQL 不是關聯模型的「進化終點」，正如關聯模型也沒「殺死」前面的一切。鐘擺擺回來過：2010 年代後期，**NewSQL**（如分散式 SQL 資料庫）試圖「在分散式規模下把 ACID 與 SQL 要回來」——也就是想兩邊都要。它有它的代價（這裡一句帶過，不展開）。重點是：沒有哪一邊是終局。你選 SQL 還是 NoSQL，永遠是看**這個負載**要的是引擎的保證、還是規模與可用性——這是個工程取捨，不是個信仰問題。能看穿這一點的工程師，不會把每個新資料庫都當革命，也不會死守單一答案。

## 紙上推演

### 推演 1：用資料獨立性解釋你的「降 40%」 **[15 分鐘]** ★

回到你那次經典戰績：尖峰時段 RDS CPU 爆頂，你加了一個 index，CPU 降 40%，**且一行應用程式碼都沒改**。用本章的概念，向一個資淺工程師解釋：為什麼你能「只改存取方式、不改程式」？這個能力的歷史名字是什麼？在 1970 年以前，同樣這件事你得付出什麼代價？

#### 推演解答

這個能力叫**資料獨立性**，是 Codd 1970 年關聯模型的核心目標。能成立的關鍵是：你的 SQL 查詢是**宣告式**的——它只說「要什麼」（滿足這些條件的列），**從來沒有承諾任何存取路徑**。所以當你加一個 index，等於給了查詢最佳化器一條新的、更快的路；它發現新路更快就改走新路，而你的查詢一個字都不用動，因為它本來就把「怎麼拿」的決定權讓給了引擎。

在 1970 年以前的導航式資料庫裡，這件事**做不到**。那時候存取路徑（走哪條指標鏈、怎麼走訪）是手寫在每一支應用程式裡的（存取路徑相依）。你想換一條更快的路徑，就得**回去改所有相關的應用程式碼**。換句話說，你那次「只動 DB、不動程式」的輕鬆，在前關聯時代是一場要動很多程式的重構。常見誤解是把「加 index 不用改程式」當成理所當然的天性——它不是天性，它是 Codd 用一篇論文掙來的、你每天免費繼承的奢侈。

### 推演 2：用 CAP 解釋一次你的選型 **[20 分鐘]** ★★

挑一個你真實做過的決定：某個場景你選了 Redis 或某種 NoSQL，而不是丟進關聯資料庫。用 CAP 的語言說清楚：在這個選擇裡，你**放棄了什麼**、**換到了什麼**？當網路分區真的發生時，你選的那個系統會偏向 C 還是 A？這個放棄，把哪塊複雜度退回給了你的應用層、你又是怎麼補的？

#### 推演解答

範例（以 Redis 當 cache 為例）：你選 Redis 是為了**低延遲與可用性**——讀取走記憶體、毫秒級回應，扛得住尖峰流量。你放棄的是「Redis 裡的值和後端資料庫**永遠一致**」這個保證：cache 裡的資料可能是舊的。對到 CAP，當「Redis 的值」與「DB 的真相」出現分歧（一種廣義的分區／不同步）時，你選擇了**繼續用可能過期的值服務（偏 A）**，而不是每次都回源確認（那會犧牲低延遲這個你選 Redis 的初衷）。

這個放棄把**一致性的複雜度退回給了你的應用層**——具體就是 **cache 失效（cache invalidation）**：你得決定 TTL、得在資料變更時主動失效或更新 cache、得處理「失效的瞬間有人讀到舊值」的窗口。這正是 ch01 點名的「CS 兩大難題之一」。關鍵領悟：**你選 NoSQL/cache 不是因為它「比較新」，而是因為這個負載的取捨點，落在了 CAP 逼你選的可用性那一側；而你省下的一致性保證，是引擎不幫你扛、你自己扛回來的擔子。** 一個資淺工程師常見的誤解是「Redis 比 MySQL 快所以更好」——錯，它不是更好，它是在一條鐘擺上選了不同的邊，而你為這個選擇付了「自己維護一致性」的帳。

### 推演 3：N+1 是哪一層抽象的洩漏 **[15 分鐘]** ★★

你一定踩過 N+1 query：拿一份清單，迴圈裡每個元素又各觸發一句查詢，1 變成 1+N。用本章「複雜度搬家／抽象洩漏」的框架回答：N+1 是**哪一層抽象**漏的？它漏的是 Codd 那層（SQL），還是更上面那層（ORM）？為什麼說它是「把搬走的複雜度漏回來」的標本？（這題接 ch19。）

#### 推演解答

N+1 漏的是 **ORM 這層**，不是 SQL 那層。要分清楚這個堆疊：Codd/SQL 把「怎麼拿資料」從你的程式搬進了引擎（宣告式）；ORM 又在 SQL **之上**疊了一層，把「物件 ↔ 表的對應、什麼時候發 SQL」也搬走，讓你以為自己只是在操作記憶體裡的物件。N+1 就是這層 ORM 抽象的洩漏：你寫 `users.forEach(u => u.orders)`，看起來在走訪物件，但 ORM 偷偷對**每一個** `u` 發了一句 `SELECT ... WHERE user_id = ?`——於是拿 100 個使用者各自的訂單，變成 1（拿使用者）＋100（拿訂單）句查詢。

為什麼是「搬走的複雜度漏回來」的標本？因為 ORM 搬走的那塊複雜度——「什麼時候、發幾句 SQL」——並沒有消失（複雜度守恆），它只是被藏到了框架裡。平常它藏得好好的，你享受著「操作物件就好」的方便；但當資料量一上來，它就以「莫名其妙發了 101 句查詢、效能塌掉」的形式漏回來找你。而且因為它本來被藏起來，你光看應用程式碼**看不出**這裡會發 N 句 SQL——你得開 query log 或 profiling 才抓得到。這完全符合 ch19 要講的 Joel Spolsky「抽象洩漏定律」：抽象藏得越多，漏的時候你越看不見根因。修法（eager loading / JOIN 一次拿完）本質上是**你部分接管了 ORM 放棄的「怎麼拿」決策**——又一次，抽象漏了，值錢的工程師跳下去補。

### 推演 4：反事實——如果 Codd 沒贏 **[20 分鐘]** ★★★

反事實推理：假設關聯模型在 1970 年代的路線之爭裡輸了，導航式資料庫（Bachman 那套）一路統治到今天。在這個平行世界裡，你今天的工作會有哪些**現在不存在的痛**？特別想想：read replica、加 index、換 ORM、甚至「換一個資料庫廠商」這些事，會變成什麼樣？這題幫你體會 Codd 那篇論文到底搬走了多大一塊複雜度。

#### 推演解答

在這個導航式統治的平行世界裡，最根本的差別是：**「資料怎麼存」和「程式怎麼寫」永遠綁死**（沒有資料獨立性）。一連串你今天視為理所當然的事都會變成大工程：

- **加 index**：今天你加個 index、應用程式無感。那個世界裡，因為程式可能依賴特定的存取路徑與索引結構（索引相依、存取路徑相依），加或改一個 index 可能逼著你回去改一堆導航程式碼。「只動 DB 不動程式」這種事不存在。
- **read replica / partition**：今天你把讀流量導到 replica、把大表 partition，應用層的查詢幾乎不用改（語意不變）。那個世界裡，資料的物理組織一變，導航路徑就變，應用程式得跟著重寫——水平擴展的成本高到嚇人。
- **換 ORM、換資料庫廠商**：今天換 ORM 或從 MySQL 遷 PostgreSQL 雖然麻煩，但有 SQL 這個（大致）共通的宣告式介面墊著。那個世界沒有 Codd 的標準化邏輯模型，每家導航式資料庫的存取 API 各長各的，你的程式焊死在某一家的物理模型上，遷移幾乎等於重寫。
- **臨時查詢（ad-hoc query）**：今天你想撈個沒預想過的報表，當場寫句 SQL 就好。那個世界裡，沒有預先鋪好指標路徑的查詢，可能得寫一支新程式去導航——「問資料一個新問題」的成本極高。

把這些痛加起來，你就摸到了 Codd 那篇論文的份量：**它搬走的不是「某個查詢」的複雜度，而是「資料的物理結構」與「所有應用程式」之間的那條死綁。** 一旦解綁，加 index、擴展、遷移、臨時查詢全都從「改程式的大工程」降級成「日常操作」。常見誤解是覺得關聯模型只是「把資料畫成表格」這種表面改變——不，它真正的革命是**資料獨立性**，是把整個產業從 Bachman 的「人人都是領航員、人人扛著路線圖」裡解放出來。你今天工作之所以能這麼輕鬆地動資料層，是因為這場仗在 1970 年代已經替你打贏了。

## 自我檢核

口頭自答，講得出因果才算過關：

1. 在 Codd 之前，「程式設計師作為領航員」是什麼意思？為什麼說導航式資料庫把「資料的物理結構」這個最該藏起來的決策暴露給了每一支應用程式？（接 ch06 資訊隱藏）
2. 「資料獨立性」到底獨立了什麼跟什麼？用你「加 index 不改程式」的經驗講清楚它為什麼成立——關鍵在於宣告式查詢從來沒承諾過什麼？
3. 為什麼說「宣告式 vs 命令式」「最佳化器 vs 手寫路徑」這場 1970 年代的爭論，和 ch03「編譯器產生的碼會不會比手寫慢」是同一條鐘擺？它們的答案為什麼也一樣？
4. ACID 的歸屬：哪些是 Jim Gray 命名的、哪一個是後來補進去湊成縮寫的、ACID 這個詞本身是誰在哪一年造的？（不能含糊成「Jim Gray 提出 ACID」）
5. CAP 在說什麼？為什麼順序是「Brewer 2000 猜想 → Gilbert-Lynch 2002 證明」？當網路分區發生時，你的系統「選 C」和「選 A」分別會有什麼可觀察的行為？
6. 1998 的 Strozzi NoSQL 和 2009 的 NoSQL 運動，所指有什麼根本不同？為什麼把兩者混為一談是史實錯誤？
7. NoSQL 的反動，把哪一塊複雜度從引擎退回給了應用層？你在 SQS pipeline 或 Redis cache 裡做的哪些事，本質上是在「手工補回引擎放棄的一致性保證」？
8. N+1 query 是哪一層抽象的洩漏（SQL 還是 ORM）？為什麼說它是「複雜度守恆」的標本——搬走的東西沒消失、只是藏起來、然後漏回來？

## 延伸閱讀

- **E. F. Codd, 〈A Relational Model of Data for Large Shared Data Banks〉, *CACM* 1970-06**（[dl.acm.org/doi/10.1145/362384.362685](https://dl.acm.org/doi/10.1145/362384.362685)）。本章的原典。讀它對「data independence」的定義，以及他點名要消滅的三種相依（ordering / indexing / access path）——你會發現整篇的目標不是「表」，是「解綁」。
- **Charles W. Bachman, 〈The Programmer as Navigator〉, 1973 圖靈獎演說, *CACM* 1973-11**（[people.csail.mit.edu/tdanford/6830papers/bachman-programmer-as-navigator.pdf](https://people.csail.mit.edu/tdanford/6830papers/bachman-programmer-as-navigator.pdf)）。讀 Codd 的對手怎麼**驕傲地**描述那個被 Codd 推翻的世界觀。把它和 Codd 的論文對讀，你會親眼看到一場範式轉移的兩邊。
- **Donald Chamberlin 口述歷史（Computer History Museum）**（[archive.computerhistory.org/resources/text/Oral_History/Chamberlin_Don/102702111.05.01.acc.pdf](http://archive.computerhistory.org/resources/text/Oral_History/Chamberlin_Don/102702111.05.01.acc.pdf)）。SEQUEL/SQL 的共同設計者親口講當年那場「程序式 vs 宣告式」的路線之爭，以及為什麼要用英文關鍵字。看歷史當事人怎麼回憶決策現場。
- **Eric Brewer, CAP 的回顧〈CAP Twelve Years Later: How the "Rules" Have Changed〉（2012, IEEE Computer）**（搜尋標題可得）。CAP 提出者本人十二年後的澄清——「三選二」其實是過度簡化，分區只在真的發生時才逼你選。讀它能讓你不把 CAP 講成口號。
- **Joel Spolsky, 〈The Law of Leaky Abstractions〉(2002)**（[joelonsoftware.com/2002/11/11/the-law-of-leaky-abstractions/](https://www.joelonsoftware.com/2002/11/11/the-law-of-leaky-abstractions/)）。本章的帳單（N+1、慢查詢）在這篇被升級成一條定律。ch19 會正式收它；先讀它對 SQL 抽象洩漏的那段。
- **書架上的《把系統寫成定理：TLA+ 與形式化方法》**。本章談 CAP、最終一致性都點到為止；那本書把「分散式系統的正確性怎麼用 invariant 證明」這條路走到底——當你想真正搞懂「為什麼分區下不能同時要 C 和 A」，去那裡。
