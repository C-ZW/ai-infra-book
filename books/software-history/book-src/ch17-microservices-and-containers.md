# ch17 — 微服務與容器：把單體拆散再封裝

> **本章解決什麼問題**：ch16 把機房搬給了別人（雲），但你租到的還是「機器」——一台台 VM，上面跑著你那個越長越大的單體。這章撞的是另一面牆：**不是機器的牆，是人的牆**。當一個 codebase 大到要幾十、上百個工程師一起改，他們會在同一份程式裡互相踩腳——你改的函式別人正在用，你想發版得等別人的功能也好了，整個系統被綁成一塊只能一起動的鐵板。微服務是對這面牆的回應，它的招數是把單體照業務能力切開、讓每個小團隊各自擁有、各自發版。但它翻牆的方式很特別：它**沒有消滅複雜度，它把複雜度從「程式內」搬到了「網路上」**——你拆掉的函式呼叫，變成了會逾時、會重試、會半途失敗的網路呼叫。容器（Docker）與 Kubernetes 則是這個分散式新世界的封裝層與作業系統。這章回收 ch06 的 Unix 組合哲學、ch10 與 ch13 的兩條伏線（Conway's Law 與宣告式），把它們在分散式尺度上一次兌現。維運這堆服務的工程化（DevOps/SRE）留給 ch18。

## 從你已知的出發

這一章你不是讀者，你是當事人。你每天上班做的事——AWS ECS 上的 TypeScript/NestJS 微服務、B2B/B2C 的資料中介——就是本章要拆解的那個歷史現象的 2026 年現場。所以這章的「橋接」不是把歷史翻譯成你聽得懂的話，而是反過來：把你天天在付的帳，還原成它在軟體史上的座標，讓你看清楚自己到底在跟哪一面複雜度的牆纏鬥。

先說你天天付的帳，逐筆列出來，因為它們就是這一章的主角。

**第一筆：網路的帳。** 你有一個服務要拿另一個服務的資料。在單體裡，這是一次函式呼叫——同一個 process、同一塊記憶體、奈秒級、要嘛成功要嘛拋例外，乾淨俐落。在你的微服務裡，這變成一次跨網路的呼叫：要序列化、要走 TCP、要反序列化，會逾時、會 503、會在你以為成功的那一刻悄悄半途失敗。於是你寫了 retry，retry 又逼你處理冪等（同一個請求被重送兩次不能扣兩次款），冪等又逼你想清楚什麼叫「同一個請求」。你做 SQS 的冪等消費者、處理服務間的最終一致性——這一整套你早就內化成肌肉記憶的東西，**沒有一樣是業務需求，全都是「把函式呼叫換成網路呼叫」這個決定逼出來的帳單**。

**第二筆：一致性的帳。** 單體裡有一個資料庫、一個交易（transaction），ACID 替你把「要嘛全做、要嘛全不做」保證好（見 ch13）。拆成微服務後，每個服務有自己的資料庫（這是微服務的定義特徵之一，不是你做錯了），於是一個跨服務的業務動作——下單、扣庫存、發通知——再也沒有一個交易能罩住它。你只能退而求其次，用最終一致性：先各自做、靠事件與補償慢慢收斂到一致。你維運的那些「對帳」「補償」「狀態機」，是這筆帳的還款憑證。

**第三筆：看不見的帳。** 單體出錯，一份 stack trace 看到底。你一個請求流經五個服務、其中第三個慢了，你得靠 trace ID 把這五段串起來才知道是誰拖累誰——你自建 FluentBit＋Lua 可觀測性 pipeline 的理由，一大半就是要把「被切碎散落在好幾個服務裡的真相」重新縫回來。單體時代這份複雜度根本不存在。

**還有一筆你可能沒意識到是帳的帳：邊界長得像組織圖。** 你切過的服務邊界，憑良心講，有多少是純技術判斷、有多少其實是照「哪塊歸哪個小組」長的？訂單組維護訂單服務、結算組維護結算服務，邊界就劃在兩個小組中間——這不是巧合，是 ch10 講過的 Conway's Law 在你的 AWS 帳號裡活生生上演。本章要把那條 ch10 只點到、答應留給這裡展開的伏線，正式兌現。

把這四筆帳排在一起，你會看到一個讓人不太舒服的真相：**這些帳沒有一筆是「微服務帶來的好處」，全部都是它的成本。** 那好處到哪去了？好處不在程式裡，在組織裡——它讓一百個工程師可以不互相踩腳。這就是本章要講透的核心：微服務是一筆用**技術複雜度**換**組織可擴展性**的交易，而且這筆交易的性質，1968 年 Conway 就替你說破了。

## 先看清楚撞的是哪面牆：單體不是壞架構

要看懂微服務在解什麼，得先公平地對待它要取代的東西——單體（monolith）。本書不寫鄉愁也不趕流行，所以這裡要先把一個被微服務行銷帶壞的觀念扳正：**單體不是一種錯誤，它在大多數情況下是對的起點。**

一個單體應用——所有功能編在同一個部署單元裡、共用一個資料庫——有一堆你拆成微服務後會痛失的好處：一次函式呼叫就能跨「模組」拿資料，沒有網路；一個交易就能保證跨功能的一致性，沒有最終一致性那套；要 debug 一個請求，一份 stack trace 走到底；要重構一個跨越好幾個功能的改動，IDE 幫你一次改完、編譯器替你檢查有沒有漏。對一個十人團隊、一個還在找產品方向的系統，單體幾乎一定是對的——它讓你跑得快，因為它沒有任何分散式的稅。

那單體什麼時候開始痛？**不是在它「程式碼變多」的時候，是在它「一起改它的人變多」的時候。** 這個區別是整章的樞紐，值得釘死。一個一百萬行、但只有八個人維護、彼此天天講話的單體，可能活得好好的。真正讓單體變成地獄的，是當維護它的人從八個變成八十個、分成十幾個小組、各有各的時程與目標時——這時候單體的「全部編在一起」從優點變成詛咒：

- **發版被綁死。** 你的功能改好了、測完了、想上線，但部署單元是整個單體——你得等其他十幾個小組的改動也都好了、整個系統一起測過，才能一起發版。一個團隊的節奏被全公司最慢的那個團隊拖住。
- **改動互相踩腳。** 你改一個共用的 util、一個共用的資料表 schema，可能在你完全不知情的情況下弄壞了另外三個小組的功能。共用的程式碼面積越大，這種「踩腳」越頻繁，於是大家越來越不敢動共用的東西，系統越來越僵。
- **協調成本爆炸。** 八十個人要在同一份程式上不衝突地前進，需要的溝通量——還記得 ch10 的 Brooks's Law 嗎——是 O(n²) 的。單體強迫一個大團隊維持高密度溝通，而人類維持不了那麼多溝通路徑。

看清楚了嗎？**這面牆不是技術牆，是組織牆。** 撞牆的不是 CPU、不是記憶體、不是程式碼行數，是「一大群人在同一塊程式上協調」這件事——這正是 ch10 Brooks 講的「人的複雜度」。微服務不是一個更好的程式架構（在很多技術指標上它更差），它是一個**讓大團隊能平行工作的組織架構**，只是長成了系統的形狀。把這句話放在心裡，後面所有的帳單你都會看得很清楚：微服務付出的全部技術代價，買的是同一樣東西——讓一群人不必再擠在同一塊程式上互相等待。

## SOA：微服務的前身，與它最重要的教訓

微服務不是 2014 年憑空蹦出來的點子。在它之前十幾年，業界已經試過一次「把單體拆成多個服務」的大規模實驗，叫 **SOA（Service-Oriented Architecture，服務導向架構）**。SOA 概念在 1990 年代末成形、2000 年代初中期紅遍企業 IT。它的核心主張跟微服務聽起來幾乎一樣：別把所有東西塞進一個巨石，把系統拆成一個個可重用的「服務」，服務之間透過標準化的方式溝通。

那 SOA 後來怎麼了？它在 2000 年代後期高調退潮，退潮的原因正是微服務最重要的一堂前車之鑑，必須講清楚——**SOA 的失敗不在它的原則，在它的實作，而失敗的核心是一個叫 ESB 的東西。**

SOA 的典型實作，會在所有服務中間擺一個 **ESB（Enterprise Service Bus，企業服務匯流排）**：一條粗壯的中央管線，負責所有服務之間的訊息路由、轉換、編排、甚至業務規則。它的賣點是「把整合邏輯集中管理」。但這個設計埋了一顆雷：**它把智慧放進了管線。** ESB 不是一根單純的水管，它是一個塞滿了路由規則、格式轉換、流程編排的龐然大物，而且通常是某家廠商的專有產品。

於是 ESB 製造了它本來要解決的問題的更糟版本：

- **它變成新的單點。** 所有服務的溝通都穿過 ESB，ESB 一慢、一掛，全公司的服務跟著遭殃——你拆出一堆服務，卻給它們造了一個共同的命門。
- **它變成新的瓶頸與新的單體。** 業務邏輯偷偷滲進 ESB 的編排規則裡，於是改一個跨服務流程，你得去動那條中央管線——那條管線變成一個誰都不敢碰、改一下牽連全公司的新單體，只是它長在中間而不是兩端。
- **它鎖死你。** ESB 多半是 IBM、Oracle、TIBCO 這類廠商的專有 stack，買了就很難換，整套架構被綁在供應商身上。

公允地說：SOA 的**原則**是對的（拆服務、定契約、解耦），失敗的是 ESB 這個**實作**——把協調的智慧集中到一條專有的中央管線上。這個教訓，微服務學得非常徹底，而且它學到的東西後來凝結成一句口號，是你理解微服務與 SOA 差別的鑰匙：**「smart endpoints and dumb pipes」（聰明的端點，笨的管線）**。把 ESB 的邏輯反過來——智慧放回各個服務（端點）自己身上，服務之間只用最笨、最簡單的管線溝通（一個 HTTP 呼叫、一則丟進佇列的訊息），管線不懂業務、不做編排、不可能變成瓶頸。

你看你自己的系統：服務之間就是 REST 呼叫或往 SQS 丟訊息，沒有一條「中央匯流排」幫你做路由與轉換。SQS 就是那根 dumb pipe——它只負責可靠地把訊息從 A 搬到 B，完全不懂訊息裡是什麼業務、也不替你編排流程；所有「收到這訊息該幹嘛」的智慧，都在你的 consumer（endpoint）裡。**你天天在實踐「smart endpoints, dumb pipes」，而這條原則的全部意義，是從 SOA 用 ESB 摔的那一跤裡長出來的。** 微服務某種程度上可以定義成「學會了 SOA 教訓的 SOA」。

```text
        SOA（ESB 版）：智慧在中間的管線
        服務A、服務B ──┐        ┌── 服務C、服務D
                       │        │
                       ▼        ▼
        ┌───────────────────────────────┐
        │  ESB：路由／轉換／編排／規則  │ ◄ 新單點、新單體、廠商鎖定
        └───────────────────────────────┘
        （所有服務的溝通都穿過中央管線，業務邏輯滲進管線變成新單體）

        微服務：智慧在端點，管線是笨的
        服務A ──HTTP/訊息──► 服務B    管線（HTTP、SQS）只搬運，
        服務C ──HTTP/訊息──► 服務D    不懂業務、不編排、不會變瓶頸
```

## 微服務：給「組織牆」的解法

把前面兩節接起來：撞的牆是組織牆（一大群人在同一塊程式上協調不過來），前人試過 SOA 但被 ESB 拖垮了。微服務就是在這兩個條件下長出來的答案。

「microservices」這個詞大約在 **2011–2012 年**於圈內成形，最廣為引用的定調文獻是 **James Lewis 與 Martin Fowler 於 2014 年 3 月發表的〈Microservices〉**一文；而最早的大規模實踐者裡，**Netflix**（Adrian Cockcroft 等人）常被點名——他們為了把服務搬上 AWS、撐住爆炸成長的串流流量，很早就走上了細粒度服務的路。要注意歸屬的分寸：Lewis 與 Fowler 是把這個已經在發生的實踐**命名與定調**，不是發明它；就像 Conway 命名了一個現象、Brooks 替它取名「Conway's Law」一樣，微服務也是先有實踐、後有定義文。

Lewis 與 Fowler 那篇文章列了微服務的一串特徵，我不打算逐條複述（那是規格書的工作），只挑出幾條能讓你看清「它在解組織牆」的：

- **按業務能力切（organized around business capabilities）。** 服務不照技術分層切（不是「UI 服務、邏輯服務、資料服務」），而是照業務能力切（「訂單、庫存、結算」各自一個服務，每個服務內含自己的 UI/邏輯/資料）。這直接呼應 ch06 Parnas 的資訊隱藏——照「會一起變的決定」切，不照「一筆資料流過的步驟」切（見 ch06）。切錯了，你就得到 ch06 警告過的分散式單體：一個需求進來要動三個服務、一起發版。
- **獨立部署（independent deployment）。** 這是微服務的命根子，也是它解組織牆的關鍵動作。每個服務是獨立的部署單元，你的服務改好了就能自己上線，不必等別人。發版被解綁，團隊的節奏就被解綁——這正是單體那條「發版被綁死」的帳，微服務付錢買回來的東西。
- **去中心化的資料管理（decentralized data management）。** 每個服務管自己的資料庫。這是微服務最貴的一條特徵——它正是你那筆「一致性的帳」的來源（沒有跨服務的 ACID 交易了），但它非要不可：如果兩個服務共用一個資料庫，它們就透過資料庫偷偷耦合了，你的「獨立部署」是假的——改一個 schema 還是會弄壞另一個服務。去中心化資料是「真獨立」的代價。
- **產品而非專案（products not projects）／你建它、你跑它。** 團隊擁有服務的**整個生命週期**，不是做完就丟給維運。這條的源頭比微服務更早——**Amazon CTO Werner Vogels 在 2006 年的一場 ACM Queue 訪談裡講過一句後來被反覆引用的話，大意是：傳統做法是「把軟體丟過開發與維運之間那道牆，然後就忘了它」；在 Amazon 不是這樣，「You build it, you run it」（你建它，你跑它）**（已查證此訪談與說法存在，非逐字引用全文）。這條把開發與維運的關係也改寫了——但那道「牆」怎麼被彌合、SRE 怎麼把維運工程化，是 ch18 的主題，這裡只取它對「團隊自治」的那一面。

把這幾條特徵串起來看，會浮現一個清晰的圖像：**微服務的每一條技術特徵，都對應著一塊組織自治。** 按業務能力切＝一個團隊擁有一塊業務；獨立部署＝這團隊自己決定何時上線；去中心化資料＝這團隊完全掌控自己的資料、不被別人的 schema 綁住；你建它你跑它＝這團隊對自己的服務負全責。微服務在做的事，本質上是把一個大團隊，沿著業務能力的線，切成一堆能各自全速前進的小團隊——而系統，忠實地長成了這個組織切法的形狀。

這就要把 ch10 那條伏線收回來了。

## Conway's Law 的兌現：微服務是組織結構的固化

ch10 把 Conway's Law 放上了檯面，並明說它的完整展開留給這裡（見 ch10）。現在兌現。

先重述那條定律的精確版本（措辭照 ch10 與 landscape）：**Melvin Conway 在 1968 年 4 月《Datamation》上那篇〈How Do Committees Invent?〉裡指出——任何設計系統的組織，最終產出的設計，其結構都會是這個組織溝通結構的翻版。**（這篇 1967 年先投《哈佛商業評論》被退稿；「Conway's Law」之名是後人冠的。）關鍵詞是「**溝通**結構」，不是「組織圖」——介面由溝通定義，兩塊模組要協作，負責它們的人就得溝通；溝通不順的兩組人，他們負責的兩塊就長不出緊密整合的介面。系統的接縫，會精準落在組織溝通的接縫上。

現在把它套到微服務上，你會發現一件事：**微服務不只是「受 Conway's Law 影響」，它根本就是 Conway's Law 的刻意執行。**

傳統上，Conway's Law 被當成一個**警告**——一條你最好提防、否則系統會被你的組織爛攤子帶歪的詛咒。微服務社群做了一件聰明（也有點狡猾）的事：他們把這條詛咒**反過來當工具用**。既然系統結構終究會複製組織結構，那與其抵抗，不如**先設計你想要的組織結構，讓它自動長出你想要的系統結構**。你想要一堆能獨立部署、互不阻塞的服務嗎？那就先建一堆能獨立決策、互不阻塞的小團隊，系統會自己長成那個樣子。

這個操作有個名字，叫 **Inverse Conway Maneuver（逆康威操作）**——一般歸給 ThoughtWorks 的 Jonny Leroy 與 Matt Simons（約 2010 年，Cutter IT Journal；這個歸屬與年份的來源較單一，姑且當作一個方便的標籤）。它的主張一句話：**與其調整系統去配合組織，不如先調整組織去長出你要的系統。** 微服務架構與「小而自治的跨職能團隊」這種組織形態，是天造地設的一對——前者是後者在系統上的投影。

到這裡，本章最重要的領悟可以講出來了，這是你下次跟另一個工程師解釋微服務時，能讓對方眼睛一亮的那句話：

> **微服務在技術上是一種部署架構，在本質上是一種組織架構。它真正切開的不是程式，是團隊；它把組織的溝通邊界，固化成了系統的網路邊界。**

這個領悟有個很實際的推論，直接打在你身上。回到本章開頭那筆「邊界長得像組織圖」的帳——現在你知道那**不是 bug，是 Conway's Law 在正常運作**。你切的服務邊界照團隊邊界長，是必然，不是失職。但這帶出一個更尖銳的問題（也是 ch10 預告過、留給這裡的那個推論）：**如果你的服務邊界切錯了，那往往不是技術問題，是偽裝成技術問題的組織問題。**

想清楚這個推論的力道。假設你有兩個服務，天天得一起改、一起發版（你的分散式單體，見 ch06）。直覺會說「這是我服務拆得不好，重新拆一下程式就好」。但 Conway 告訴你：如果負責這兩塊的是兩個獨立的小組，那麼**只要這兩個小組還是分開的，你怎麼重拆程式，系統都會再長回兩個糾纏的服務**——因為系統結構是組織結構的影子，你動的是影子，本體沒動。要真正修好這條邊界，你得先改組織：要嘛把這兩塊合給同一個團隊（讓它們合理地變回一個服務），要嘛把它們之間的依賴從組織層面真正解開。**想改系統結構而不動組織結構，是逆著 Conway's Law 推石頭。** 這就是為什麼很多「微服務重構」專案做了一輪又一輪、服務拆了又拆，痛卻一點沒減——他們在反覆整理影子，從沒碰過投影的本體。

```text
        Conway's Law 的兩個方向

        被動（詛咒）：組織爛 → 系統爛
        [溝通不順的組織]──忠實複製──►[糾纏的系統邊界]

        主動（工具）= Inverse Conway Maneuver：
        先設計組織 → 自動長出想要的系統
        [小而自治的跨職能團隊]──忠實複製──►[可獨立部署的微服務]

        關鍵領悟：服務邊界切錯，常是組織問題偽裝成技術問題。
        改影子（重拆程式）沒用，要改投影的本體（組織）。
```

## 容器：「在我機器上能跑」問題的終結

微服務解決了「一群人在一塊程式上踩腳」，但它立刻製造了一個新的後勤噩夢：**你現在不是部署一個應用，是部署幾十個。** 每個服務有自己的執行環境、自己的相依套件、自己的語言版本——這個服務要 Node 18、那個要 Python 3.11、另一個要某個特定版本的 system library。要把這幾十個東西可靠地部署到一堆機器上，你先得解決一個更古老、更惱人的問題，一個你一定聽過某人說過的問題：

> **「可是它在我機器上明明能跑啊。」**

這句話是一整個時代的軟體部署痛苦的縮影，而它的病根是**環境差異**。你開發機上的環境（OS 版本、裝了哪些 library、環境變數、某個你三個月前手動裝過早就忘了的相依套件）和測試機、和 production 機，永遠有細微的不同。程式在你機器上能跑，搬到別處就掛——不是程式錯了，是程式所依賴的那個「環境」沒跟著一起搬過去。在這個問題被根治之前，部署是一門充滿了「在 production 重現環境」的黑魔法，每次上線都像拆彈。

2013 年 3 月，這個問題迎來了它的終結者。**Solomon Hykes 在 PyCon 2013 的一場五分鐘 lightning talk〈The Future of Linux Containers〉上，第一次對公眾展示了 Docker。** 有意思的是，這場 talk 本來不是規劃中的產品發表——Docker 原本是 Hykes 的公司 dotCloud 內部用來支撐 PaaS 的工具，這場即興的展示意外地成了它的發表會，幾週後正式開源（年份與場合照 landscape）。

Docker 解環境差異的招數，說穿了極其優雅，而且——這是本章最值得你停下來品味的一個呼應——**它的核心思想，正是 ch02 那個七十年前的「程式即資料」。**

容器的做法是：把應用程式，連同它執行所需要的**整個環境**（OS 層的檔案系統、所有相依套件、設定、環境變數），一起打包成一個**不可變的（immutable）image**。這個 image 是一個靜態的、可被複製、可被傳輸、可被版本化的**檔案**——它就是資料。然後在任何裝了容器 runtime 的機器上，你把這份 image 載入、跑起來，它就變成一個運行中的容器（container）——這時它是程式。

停在這裡想一秒，這個形狀你太熟了。**image 是資料，跑起來是程式**——這跟 ch02 講的儲存程式（stored program）是同一個結構：程式以資料的形式存在，被載入後才成為運行的程式。你在 ch02 的橋接裡其實已經被預告過這件事了（見 ch02：「你部署的每個 container image 都是『資料』，跑起來才是『程式』」）。Docker 做的，是把這個 1945 年的概念，從「一條指令」放大到「一整個應用連同它的環境」這個尺度。「在我機器上能跑」之所以被根治，是因為**現在搬過去的不只是程式，是程式連同它的整個世界**——環境不再是部署目標機上那個飄忽不定、要你去重現的東西，它被凍進了 image 裡，跟程式焊死、一起搬家。環境差異這個複雜度，被一次性地搬走了。

這裡得把血脈交代清楚，免得你以為 Docker 憑空發明了容器。**它沒有。** 容器的底層機制——Linux 用來把一個 process 隔離起來、限制它能看到什麼、能用多少資源的能力——早就在 kernel 裡了：**cgroups（control groups）在 2007 年由 Google 主推進 Linux kernel**（原名 process containers），**LXC（Linux Containers）2008 年**在其上提供了較完整的容器封裝。Docker 真正的貢獻不是發明這些底層原語，而是**把它們包成了一個開發者用得下去的東西**：一個簡單的 image 格式、一套人話般的指令、一個能分享 image 的 registry（Docker Hub）。它把「組裝容器」這件原本要懂一堆 kernel 細節的苦差事，變成一行 `docker run`——這跟 ch03 編譯器把「對機器說話」外包出去、ch16 雲把機房外包出去是同一種動作：**把一個原本要專家才會的複雜操作，封裝成一個普通工程師能用的抽象。** Google 把容器原語放進了 kernel，Docker 把它變成了人人能用的工具。

## Kubernetes：分散式世界的作業系統

現在你手上有了幾十個打包好的容器 image。下一個問題立刻來了，而且它比前一個更難：**這幾十個容器，要在一堆機器（一個叢集，cluster）上跑起來，誰來決定哪個容器跑在哪台機器上？某台機器掛了，上面的容器怎麼自動搬到別台?某個服務流量大了，怎麼自動多開幾個 instance？這幾十個容器之間怎麼找到彼此、怎麼通訊？**

這一整類問題叫**容器編排（container orchestration）**，而它的事實標準答案，是 **Kubernetes（常簡寫 K8s）**——Google 於 **2014 年**宣布並開源，**2015 年捐給新成立的 CNCF（Cloud Native Computing Foundation）**作為首個種子專案（年份照 landscape）。

K8s 的血脈同樣值得交代，因為它解釋了為什麼 K8s 一出生就這麼成熟。它承自 Google 內部跑了十幾年的叢集管理系統 **Borg**（約 2003 年起）與其後繼 Omega——Google 早就在用這套東西調度它全公司的服務了，K8s 等於是把這十幾年「在地球上最大規模之一的叢集上調度容器」的經驗，萃取、開源出來。（一個有意思的細節：K8s 的三位創始者 Joe Beda、Brendan Burns、Craig McLuckie，其實都不是當年打造 Borg 核心的人，而是 Borg 的**使用者**——是被 Borg 服務過、深知其價值的人，把這份價值帶給了全世界。）

那為什麼說 K8s 是「分散式世界的作業系統」？這個類比不是修辭，是精準的。回想作業系統對單機做的事：它把一堆實體資源（CPU、記憶體、磁碟）抽象成一個資源池，讓你不必管「我的 process 該用哪顆 CPU 核心、放在哪段實體記憶體」——你只管說「我要跑這個 process」，OS 替你調度。**K8s 對一個叢集做的，是一模一樣的事，只是尺度從「一台機器的核心」放大到「一群機器」：** 它把幾十上百台機器抽象成一個資源池，你不必管「我這個容器該跑在哪台機器上」——你只管說「我要跑這個服務」，K8s 替你調度到叢集裡某台有空的機器上。它就是叢集這台「巨型電腦」的作業系統。

但 K8s 最重要的設計選擇，不是它調度容器的能力，而是它**怎麼讓你告訴它你要什麼**——而這裡，本章要回收 ch13 的另一條伏線：**宣告式（declarative）**。

ch13 講過 Codd 關聯模型的抽象勝利：SQL 是宣告式的——你說「我要什麼（WHERE 條件）」，引擎決定「怎麼拿（執行計畫）」（見 ch13）。K8s 用的是同一個世界觀，套在基礎設施上。你不寫一串命令式的腳本告訴 K8s「先在 3 號機開一個容器、再在 5 號機開一個、如果 3 號機掛了你就……」——那是命令式（imperative），你得把每一步、每一種意外都想到並寫出來。K8s 讓你做的是：寫一份**宣告式**的描述，說「我**想要**這個服務有 3 個 replica 一直跑著」（the desired state，期望狀態），然後 K8s 自己負責讓現實不斷地**收斂**到這個期望——這套機制叫 reconciliation loop（調和迴圈）：它持續比對「現在的實際狀態」與「你宣告的期望狀態」，發現有 replica 掛了（實際變成 2 個），它自動再開一個補回 3 個；發現你把期望改成 5 個，它自動多開 2 個。

```text
        宣告式 vs 命令式（K8s 的核心世界觀，回收 ch13）

        命令式：你寫每一步「怎麼做」，還得自己想到所有意外
        「在3號機開容器 → 失敗就重試 → 掛了就搬到5號機 → ……」
        （你扛全部的「怎麼做」與所有錯誤處理）

        宣告式：你只說「想要什麼」，引擎負責收斂到那個狀態
        宣告：「我要這服務一直有 3 個 replica」
                          │
                          ▼
              ┌───────────────────────┐
              │  reconciliation loop  │ 持續比對「實際 vs 期望」，
              │  實際=2 → 補1個到 3   │ 自動把現實拉回你宣告的樣子
              └───────────────────────┘
```

你做為一個天天打交道的人，可能對 K8s 的 YAML 又愛又恨，但請看清楚這個 YAML 在做的事的歷史座標：**它跟 ch16 的 Terraform（基礎設施即程式碼）、ch13 的 SQL，是同一條宣告式血脈的後裔。** 你宣告「期望的世界長什麼樣」，把「怎麼把現實變成那樣、以及一路上會出什麼錯該怎麼救」這整塊巨大的複雜度，搬給了一個引擎（K8s 的 control plane）。這是本書反覆出現的同一個動作——用宣告式把「怎麼做」的複雜度搬給引擎——只是這次搬走的是「在一個會死機、會網路分區的叢集上維持一堆服務活著」這份極其麻煩的複雜度。

（本書的範圍邊界在此要守住：K8s 怎麼操作、怎麼寫 YAML、它那些物件的細節，不是本書要教的——你正在學它，書架上的 AI Infra 書會帶你動手。service mesh 這類在 K8s 之上再加一層、把服務間通訊的橫切關注點抽出來的東西，這裡一句帶過：它是「smart endpoints」太重時，把一部分通訊智慧再外包出去的又一次抽象，同樣是搬家不是消滅。)

## 複雜度搬去哪了

微服務＋容器＋K8s 這一整跳，照例結一筆帳——而這一章的帳尤其值得你逐欄看，因為你每天在還它。

| 項目 | 內容 |
|---|---|
| **撞的牆** | 組織牆，不是技術牆：一大群人（幾十上百個工程師、十幾個團隊）在同一塊程式（單體）上協調不過來——發版被綁死、改動互相踩腳、溝通成本 O(n²) 爆炸（回收 ch10 Brooks）。單體不是壞架構，是「一起改它的人變多」時才變成詛咒 |

| **這一跳的帳單一句話** | **用「程式內的簡單」換「組織上的可擴展」——把函式呼叫的複雜度搬到網路上，買到的是讓一百個人不在同一塊程式上互相等待。** 帳單在你每一次處理逾時、冪等、最終一致性、跨服務 trace 時回來找你；而帳單合不合算，取決於你的牆到底是不是組織牆——人不夠多，這筆交易就是純虧 |

把這筆帳放進全書的脈絡：微服務是 ch06 Unix「小工具用 pipe 組合」哲學的分散式放大版（pipe 換成 message queue、小工具換成獨立部署的服務，見 ch06），是 ch16 雲（租到機器）之上「怎麼用這些機器跑一堆服務」的下一跳，骨子裡又是 ch10 Conway's Law 的兌現。而它把複雜度搬上網路所製造的那些帳——網路分區、分散式一致性、可觀測性——正是你 AI Infra 之路的核心地形：把一個大模型穩、快、省地 serve 出來，本質就是在一個分散式系統上跟這同一批複雜度搏鬥。這條線怎麼把維運本身也工程化（DevOps/SRE、平台工程），是 ch18 的事。

## 紙上推演

### 推演題一 **[15 分鐘]** ★★

用「複雜度從程式內搬到網路上」這個框架，列出你的微服務系統讓你**多付**的三筆帳：網路、一致性、可觀測性。每一筆都要說清楚：(a) 在單體裡這件事是怎樣的（簡單在哪）；(b) 拆成微服務後它變成怎樣（複雜在哪）；(c) 你具體用什麼機制在還這筆帳。要求落在你真實做過的東西上（SQS、冪等、最終一致性、FluentBit），不要泛泛談。

#### 推演解答

**網路帳。** (a) 單體裡，服務 A 拿服務 B 的資料是一次函式呼叫——同 process、同記憶體、奈秒級、要嘛回傳要嘛拋例外，沒有中間狀態。(b) 拆開後變成跨網路呼叫：要序列化／反序列化、走 TCP、會逾時、會 503、會在「我以為成功了」的瞬間半途失敗（請求送達但回應丟了）。函式呼叫的二元結果（成功/例外）變成了三元（成功/失敗/不知道）——「不知道」這個新狀態是全部麻煩的根。(c) 我用 retry 處理暫時性失敗，但 retry 把「不知道」變成「可能做兩次」，所以我必須讓消費端冪等（同一個 SQS 訊息被投遞兩次，業務結果只發生一次）——我的冪等消費者就是在還這筆帳：它存在的唯一理由，是網路把「要嘛成功要嘛失敗」變成了「還可能成功了但你不知道」。

**一致性帳。** (a) 單體裡一個跨功能的動作（下單→扣庫存→記帳）包在一個資料庫交易裡，ACID 保證要嘛全成要嘛全不成（見 ch13）。(b) 拆開後訂單、庫存、結算各有自己的資料庫（去中心化資料是微服務的定義特徵），於是再沒有一個交易能罩住這三步——其中一步成功、下一步失敗，系統會處於一個「半完成」的不一致狀態。(c) 我用最終一致性：每一步成功後發事件、下一步消費事件繼續推進，失敗就靠補償（compensating action）回退或重試，靠對帳機制兜底，讓系統**最終**收斂到一致而非**瞬間**一致。我那些狀態機與對帳 job，就是在還「沒有跨服務 ACID」這筆帳。

**可觀測性帳。** (a) 單體裡一個請求出錯，一份 stack trace 從入口走到出錯那行，因果一目了然。(b) 拆開後一個請求流經五個服務，每個服務各自寫各自的 log，「這個請求到底在哪一段慢了/錯了」的真相被切碎散落在五份 log 裡。(c) 我用 trace ID（一個請求進系統時生成、隨呼叫一路傳遞的 ID）把這五段重新串成一條 trace；我自建的 FluentBit＋Lua pipeline 就是在做這件「把散落的真相縫回一條因果鏈」的工作——這份複雜度在單體時代根本不存在，是拆服務憑空長出來的。

收束：三筆帳的共同病根是同一個——**函式呼叫變成了網路呼叫**。網路不可靠（生網路帳）、跨網路沒有共享交易（生一致性帳）、跨網路的因果被切斷（生可觀測性帳）。複雜度一點沒少，它只是從你看得見的程式裡，搬到了你看不見的服務之間那段網路裡。

### 推演題二 **[20 分鐘]** ★★★

用 Conway's Law 解剖你親手做過的一次服務拆分（或合併）。誠實回答：(a) 你當初給自己講的「技術理由」是什麼？(b) 拿 Conway 逼問：這條邊界有多少其實是照**團隊/組織邊界**長的？如果負責這兩塊的當初是同一個人/同一個三人小組，你還會把它切成兩個跨網路的服務嗎？(c) 判斷這是「技術驅動」還是「組織驅動」的邊界，以及它切得對不對。(d) 進階：如果你曾有一對「假微服務」（天天要一起改一起發），用 Inverse Conway 推論：要真正修好它，該動的是程式還是組織？

#### 推演解答

（以一個代表性的拆分為例，你可換成自己的真實案例。）

(a) 技術理由：「訂單」和「結算」是兩個不同的業務能力，職責不同、變動軸不同，該獨立部署、獨立擴展——聽起來是教科書級的乾淨理由。

(b) Conway 的逼問：訂單和結算本來就分屬兩個小組——不同的人、開不同的會、不同的 on-call 輪值、不同的 backlog。誠實地說，如果這兩塊功能當初是同一個三人小組維護的，我**幾乎確定**不會把它們切成兩個跨網路的服務——我會把它們做成同一個服務裡的兩個模組，省掉那條 API、省掉跨服務的最終一致性、省掉分散式 trace。把它切成兩個服務的，與其說是技術判斷，不如說是「人本來就分在兩組」這個組織事實的投影。

(c) 所以這條邊界**主要是組織驅動的**。切得對不對，要分兩面：如果這個組織劃分本身健康（兩塊業務確實該由兩個自治團隊各自擁有、各自演化），那 Conway 驅動的邊界反而是好的——系統忠實地配合了一個合理的組織結構，這正是 Inverse Conway 想要的。但如果這個組織劃分是歷史包袱（兩塊業務其實高度耦合、天天得一起改，當年分組只是因為人事或編制），那這條邊界就是一個**偽裝成技術決策的組織問題**：我付著分散式的全部稅，只為了複製一條本來就不該存在的組織接縫。

(d) 進階——假微服務的修法：假設訂單和結算就是那對「天天得一起改一起發」的假微服務。直覺會說「我把程式重拆一下就好」。但 Inverse Conway 告訴我：只要訂單組和結算組還是兩個分開的小組，我怎麼重拆程式，系統都會再長回兩個糾纏的服務——因為我動的是影子（系統結構），沒動本體（組織結構）。要真正修好，得先動組織：要嘛把這兩塊合給同一個團隊擁有（那它們合理地變回一個服務、一個資料庫、一個交易，痛就消失了），要嘛在組織層面真正切斷這兩塊的耦合（讓它們不再需要天天一起改）。**程式怎麼改都白搭，因為問題不在程式裡。** 這是本章最反直覺、也最值錢的一個推論：有些技術問題，正確的修法是改組織。

### 推演題三 **[10 分鐘]** ★★

解釋容器 image 與 ch02「儲存程式」的血緣。要求講清楚：(a) 為什麼說 image 是「資料」、跑起來的 container 是「程式」；(b) 這個「資料↔程式」的雙重身分，跟 ch02 的儲存程式概念是同一個結構嗎；(c) Docker 把這個 1945 年的概念放大到了什麼尺度，因而根治了什麼問題。

#### 推演解答

(a) 一個容器 image 是一個靜態的檔案——可以被複製、傳輸、推到 registry、打 tag 做版本控制。它不會自己做任何事，它就是一坨資料：把應用程式連同它的整個執行環境（檔案系統、相依套件、設定）凍在一起的一份快照。直到你在某台機器上把它載入、跑起來，它才變成一個運行中的 container——這時它是程式，是有狀態、在消耗 CPU 與記憶體、在處理請求的活物。

(b) 是同一個結構。ch02 講儲存程式的關鍵躍遷，就是「程式以資料的形式存在於記憶體裡，被載入後成為運行的程式」——程式與資料住在同一個記憶體、程式變成可被操作的資料（見 ch02）。容器是這個結構的放大版：image（資料）被載入成 container（程式）。「靜態時是資料、運行時是程式」這個雙重身分，1945 年就被 EDVAC 報告奠定了，容器只是在新尺度上重演它。

(c) Docker 把這個概念從「一條機器指令以資料形式存在」放大到了「**一整個應用連同它的整個環境**以資料形式存在」這個尺度。正因為搬過去的不再只是程式、而是程式連同它的整個世界（環境被凍進 image、跟程式焊死一起搬家），「在我機器上能跑、搬過去就掛」的環境差異問題才被根治——環境不再是部署目標機上那個飄忽、要你去重現的東西，它變成了 image 的一部分。這是 ch02「程式即資料」這個七十年前的前提，在你部署管線裡每天兌現一次。

## 自我檢核

口頭自答；講得出因果才算過關，講不清楚代表還沒懂。

1. 微服務撞的牆是「技術牆」還是「組織牆」？為什麼說「單體不是壞架構」、它到底在什麼條件下才從優點變詛咒？（提示：是「一起改它的人變多」，不是「程式碼變多」）
2. SOA 為什麼退潮？ESB 把智慧放在哪裡、因而製造了什麼問題？微服務的「smart endpoints, dumb pipes」是怎麼反過來學到這個教訓的——你的 SQS 是 dumb pipe 還是 smart pipe？
3. 為什麼說「微服務的每一條技術特徵都對應著一塊組織自治」？挑「獨立部署」和「去中心化資料管理」各講一遍它買到了什麼自治、又付了什麼代價。
4. 把 Conway's Law 的精確版本講出來（關鍵詞是「溝通結構」不是「組織圖」）。為什麼說微服務「不只受 Conway's Law 影響，根本是它的刻意執行」？Inverse Conway Maneuver 把這條詛咒反過來當什麼用？
5. 「服務邊界切錯往往是偽裝成技術問題的組織問題」——這句話什麼意思？如果你有一對天天得一起改一起發的假微服務，為什麼「重拆程式」往往沒用、該動的是什麼？
6. 容器怎麼根治「在我機器上能跑」？講清楚 image 的「資料↔程式」雙重身分跟 ch02 儲存程式的血緣。Docker 到底發明了什麼、沒發明什麼（cgroups/LXC 是誰的、Docker 的真貢獻是什麼）？
7. 為什麼說 K8s 是「分散式世界的作業系統」？它的宣告式（期望狀態＋reconciliation loop）跟 ch13 的 SQL、ch16 的 Terraform 是同一條什麼血脈？它把哪一份複雜度搬給了引擎？
8. 用一句話總結這一章的帳單：微服務用什麼換什麼？這筆交易在什麼情況下是純虧？

## 延伸閱讀

- **James Lewis & Martin Fowler, 〈Microservices〉（martinfowler.com, 2014-03）**——微服務最權威的定調文。不必背那九條特徵，重點看兩段：「Organized around Business Capabilities」那節怎麼用 Conway's Law 論證「為什麼按業務能力切贏過按技術分層切」，以及「Smart endpoints and dumb pipes」那節怎麼把矛頭對準 ESB。讀完你會發現它通篇都在講組織，不只是講技術。
- **Melvin Conway, 〈How Do Committees Invent?〉（Datamation, 1968-04）**——Conway's Law 原文，melconway.com 上作者本人放了全文。ch10 讀過的話這裡再讀一次，帶著「微服務是它的刻意執行」這個視角看，那句定律會讀出完全不同的力道：它不只是個警告，是個可以反過來用的工具。
- **Werner Vogels（訪談者 Jim Gray）, 〈A Conversation with Werner Vogels〉（ACM Queue, 2006）**——「You build it, you run it」的出處。看 Vogels 怎麼描述 Amazon 把服務的所有權整個交給建造它的團隊——這是「團隊自治」與「products not projects」的一手源頭，也預告了 ch18 要展開的、開發與維運那道牆怎麼被拆掉。
- **Solomon Hykes, 〈The Future of Linux Containers〉（PyCon 2013 lightning talk，YouTube/pyvideo 有錄影）**——五分鐘、意外成為 Docker 發表會的那場 talk。值得看的是它的**短**與**即興**：一個根治了整個產業部署痛苦的東西，是用五分鐘 demo 一個內部工具的形式登場的。歷史的轉折點常常不像轉折點。
- **Brendan Burns, Joe Beda, Kelsey Hightower, 《Kubernetes: Up and Running》**——若你要從「天天用 K8s」走向「理解 K8s 為何長這樣」，看它講宣告式 API 與 reconciliation loop 的部分就好（其餘操作細節你正在別處學）。重點是抓住「期望狀態 vs 實際狀態的持續收斂」這個跟 SQL、Terraform 同源的世界觀。
- **本書 ch06（模組化與 Unix）與 ch10（人月神話）**——這一章是那兩章的兌現。ch06 的「分散式單體」與 Parnas 資訊隱藏，是你切微服務邊界的判準；ch10 的 Conway's Law 與 Brooks's Law，是你理解「微服務在解組織牆」的地基。回頭重讀，會發現本章沒有一個新概念，全是舊概念在分散式尺度上長大。
