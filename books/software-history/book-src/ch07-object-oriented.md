# ch07 — 物件導向：把資料與行為綁在一起

## 從你已知的出發

你每天寫 NestJS。停下來看一眼你最熟的那個檔案結構：`OrderController` 接 HTTP、`OrderService` 放業務邏輯、`OrderModule` 把它們組裝起來，`constructor` 裡那一串 `private readonly xxxService` 是框架幫你注入（DI）的相依。你大概沒想過——你寫的這套東西，是 1960 年代一個為了跑模擬而生的語言，經過 Smalltalk 的理想、C++ 的妥協、Java 的企業化之後，傳到你手上的直系後代。NestJS 的官方文件第一句就說它「heavily inspired by Angular」，而 Angular 的 DI 與裝飾器（decorator）模型，骨子裡是 Java/Spring 那一脈企業級 OOP 的搬遷。你不是在用一個「現代框架」，你是在還一筆五十年的帳。

而且你已經在還早期 OOP 的另一筆帳，只是你可能沒意識到。回想你怎麼設計那套 service：你大概**不太用繼承**。你不會搞一個 `BaseService` 然後讓 `OrderService extends PaymentService extends BaseService` 疊三層深的繼承樹；你比較常的是定義一個 `interface`（`PaymentProvider`），讓 `StripeProvider`、`PaypalProvider` 各自 `implements` 它，再用 DI 把具體實作注進去。這個選擇——**用 interface 加組合，而不是用深繼承**——不是你的個人品味，是整個產業在 1990 到 2000 年代撞了一輪牆之後得出的集體教訓，後來被濃縮成一句口號：「組合優於繼承」。你天天在實踐它，卻多半沒聽過它從哪場災難裡長出來。

這章把這兩筆帳的身世補給你。先講 OOP 在解什麼複雜度（為什麼要把資料和行為綁一起），再講它的原意（Kay 想要的）和它落地後的樣子（你得到的 Java class 階層）為什麼差這麼多，最後算清楚繼承這把當年的銀彈，是怎麼變成今天「能不用就不用」的東西。

## OOP 在解什麼複雜度：資料與操作的離散

先把 OOP 出場前的痛說清楚，不然「把資料和行為綁一起」聽起來只是個沒來由的口號。

ch05、ch06 講的是「怎麼切大程式」：結構化編程治控制流、模組化治分解。但它們都沒正面處理一件事——**資料和操作資料的程式碼，在當年的主流結構化語言裡是分家的**。想像一個 1970 年代用 C 風格寫法的銀行系統：你有一個 `struct Account`（裡面是餘額、戶名、利率），這是「資料」；然後你有一堆散落在各檔案的函式 `deposit(account, amount)`、`withdraw(account, amount)`、`applyInterest(account)`，這是「操作」。資料和操作之間沒有任何語言層面的綁定關係——`account.balance` 是一個**裸露的欄位**，系統裡任何一段程式碼，只要拿得到那個 `struct` 的指標，都能直接 `account.balance = -999` 把它改成負數，繞過你所有的提款檢查。

這就是 OOP 要撞的牆，我把它叫做**資料與操作的離散**：

- **沒有人守門。** 餘額這個狀態，誰都能改，改的程式碼散在十幾個檔案。出了一筆不該存在的負餘額，你要 debug，得在整個 codebase 裡 grep 所有寫過 `.balance` 的地方——因為任何一處都可能是兇手。狀態沒有單一的看門人。
- **「能對這筆資料做什麼」沒有歸屬。** 新人接手，想知道一個 `Account` 合法的操作有哪些，沒地方查——操作散在各處的全域函式裡，跟資料本身沒有目錄式的關聯。
- **改資料結構＝全域地震。** 你想把 `balance` 從一個數字改成「主餘額＋凍結餘額」兩個欄位，所有直接讀寫 `.balance` 的程式碼全部要改——這正是 ch06 Parnas 警告的洩漏，只是發生在資料層。

OOP 的核心動作就一句話：**把資料和「合法操作那筆資料的行為」綁進同一個東西（物件），並且把資料藏起來，只准透過行為去碰。** 那個 `struct Account` 變成 `class Account`，`balance` 變成 private（外面碰不到），`deposit`／`withdraw` 變成這個 class 的 method（方法）——現在改餘額**只有一條路**：呼叫 method，而 method 裡有檢查。狀態有了單一看門人。

這個動作有個名字你天天用：**封裝**（encapsulation）。而你應該已經聞到熟悉的味道了——這不就是 ch06 的**資訊隱藏**嗎？

是的。這是本章第一個要釘死的血緣關係：**封裝是資訊隱藏落在「物件」這個粒度上的實作。** Parnas 1972 說「模組該藏住一個會變的設計決策」，那是檔案／模組層級的洞見。OOP 說「物件該藏住自己的資料，只露出方法」，是把同一個洞見做進語言的型別系統裡——把「祕密」的單位從「一個模組檔案」縮小到「一個物件」。ch06 是洞見，OOP 是它的另一條落地路線（Unix 用模組與 pipe 走，OOP 用 class 與 method 走）。你選 NestJS 的 `private readonly` 欄位、用 getter 而不讓外面直接改 state，做的就是 1972 年那個洞見的 2026 年版。

封裝解的是「資料沒人守門」。OOP 還配了第二件武器，解另一個痛：**多型**（polymorphism）。

## 多型：同一個動作，不同型別各自實作

封裝把守門人立起來了，但大系統還有另一種離散：**同一個概念上的動作，對不同型別要做不同的事，而呼叫方不該為此寫一堆 if-else。**

具體場景：你的系統要「結算一筆付款」，但付款方式有信用卡、有銀行轉帳、有點數折抵。結構化寫法是一個大 switch：

```text
結構化寫法：呼叫方扛著型別判斷

  function settle(payment) {
    switch (payment.type) {
      case 'card':     settleCard(payment);     break
      case 'transfer': settleTransfer(payment); break
      case 'points':   settlePoints(payment);   break
    }
  }
   ↑ 每加一種付款方式，這個 switch 就要改一次
     系統裡每一處「依型別分流」的地方都要同步改
```

問題在那個箭頭那行：每新增一種付款方式，你得回頭找出系統裡所有 `switch (payment.type)` 的地方，每一處都補一個 `case`。漏掉一處，就是一個沉默的 bug。型別判斷的責任落在**每一個呼叫方**身上，而呼叫方有很多個。

多型把這件事翻過來：**讓「該做什麼」歸屬到型別自己身上，呼叫方只管發出動作。** 你定義一個共同的介面 `Payment`，有一個 `settle()` 方法；`CardPayment`、`TransferPayment`、`PointsPayment` 各自實作自己版本的 `settle()`。呼叫方只寫 `payment.settle()`——它**不知道也不需要知道**手上這個 `payment` 具體是哪一種，執行期會自動分派（dispatch）到正確的實作：

```text
多型寫法：型別自己知道該怎麼做

  payment.settle()      ← 呼叫方只發動作，不問型別
        │
        ├─ CardPayment.settle()      各自實作
        ├─ TransferPayment.settle()
        └─ PointsPayment.settle()
   ↑ 新增一種付款方式 = 多一個 class、實作自己的 settle()
     呼叫方一行都不用改，那個 switch 消失了
```

這就是封裝之外，OOP 解的第二筆複雜度：**它把「依型別分流」的判斷，從散落的呼叫方收攏回型別自己身上。** 對你來說這應該極其眼熟——這正是你用 `interface PaymentProvider` 加 DI 在做的事。你定義介面、各家 provider 各自實作、呼叫方靠注入拿到具體實作而不寫型別判斷。你以為這是「框架的設計模式」，其實這是 OOP 1960 年代就埋下的兩個核心承諾之一。

封裝＋多型，就是 OOP 解的兩面複雜度：**資料有了單一看門人，分流判斷有了單一歸屬。** 大系統裡「狀態誰都能改、型別判斷散一地」這個離散，被收攏進了「物件」這個單位。

那麼這個好點子，是從哪來的？它的故事比「class 與繼承」深得多，也比業界記得的版本更值得講——因為原版和落地版差了一個本質。

## 源頭一：Simula 67——為了模擬，意外發明了 class

OOP 的 class 與 object 概念，不是某個人坐下來說「我要設計物件導向」憑空想出來的。它是兩個挪威人為了解一個很具體的工程問題——**模擬**——順手長出來的副產品。

1960 年代，Ole-Johan Dahl 與 Kristen Nygaard 在奧斯陸的 Norwegian Computing Center 要做離散事件模擬（discrete event simulation）：模擬港口的船、工廠的生產線、排隊系統這類「很多個各自有狀態、各自會動的東西，在時間軸上互相影響」的場景。他們先做了 Simula I（1960 年代初），再在 1967 年 5 月奧斯陸的 IFIP 模擬語言工作會議上提出 **Simula 67**——這是史上**第一個帶有 class、object、inheritance（繼承）、virtual method（虛擬方法）的語言**。Dahl 與 Nygaard 因為「為物件導向程式設計的出現奠定基礎性思想」，在 2001 年拿了圖靈獎。

關鍵在於——**class 是為了模擬而生的**。當你要模擬一個港口，你腦中的世界本來就是「一艘艘船」「一個個碼頭」，每個都是一個有自己狀態（位置、載重）、有自己行為（進港、卸貨）的獨立實體。Dahl 後來回憶，他是透過「想像一個 Algol 60 程式在執行期的記憶體長相」而得到 class 的點子——對有眼光的人來說，物件其實早就存在於每個正在跑的 Algol 程式裡了，只是被困在堆疊紀律（stack discipline）裡，需要把它們解放出來、讓它們能比一次函式呼叫活得更久。換句話說，class 不是一個抽象的哲學發明，而是**把「現實世界由一個個有狀態的實體組成」這件事，直接搬進程式語言**。一個 object 就是程式裡的一艘船。

這個出身決定了 OOP 後來的兩面性。好的一面：用「物件」建模，貼近人對世界的直覺（系統由一個個東西組成，東西有狀態有行為）。但也埋下了後來的麻煩——當大家把「程式裡的物件對應現實世界的東西」這個直覺推得太遠（「`Dog extends Animal`，因為狗是一種動物嘛」），就走向了繼承的濫用。這個我們等下算帳時再回來。

Simula 給了 class。但「object-oriented」這個詞、以及這個詞背後真正激進的願景，來自另一個人。而這裡，就是本章那個被走形的好點子的起點。

## 源頭二：Smalltalk 與 Kay——他要的是訊息傳遞，不是 class

1970 年代，在 Xerox PARC（全錄帕羅奧圖研究中心，那個發明了滑鼠、視窗、乙太網路、雷射印表機卻沒商業化的傳奇實驗室），Alan Kay、Dan Ingalls、Adele Goldberg 一群人做了 **Smalltalk**（經歷了 Smalltalk-72、-76，到 1980 年的 Smalltalk-80）。Alan Kay 就是造出「object-oriented」這個詞的人。

但這裡有本書最重要的「大家記錯的事」之一，必須照原文釘死。Kay 後來公開表示，他造這個詞造得有點後悔，因為大家抓錯了重點。在一封廣為流傳、被社群當作原始出處的 squeak-dev 郵件列表郵件裡（1998 年 10 月 10 日），他寫下這段（逐字引用）：

> "I'm sorry that I long ago coined the term 'objects' for this topic because it gets many people to focus on the lesser idea. The big idea is 'messaging'..."
>
> （大意是：我很抱歉很久以前為這個主題造了「物件」這個詞，因為它讓很多人把注意力放在比較次要的概念上。真正的大想法是「訊息傳遞」。）

> ⚠️ 這段引文的原始載體是單一的——就是上面那封 1998 squeak-dev 郵件（後續訪談多次重述，文字一致）。本書照此句引用、並標明它是 Kay「後來表示」的回顧性說法。另外在 OOPSLA 1997 的演講裡他也說過「I made up the term object-oriented, and I can tell you I did not have C++ in mind.」（我造了 object-oriented 這個詞，而我可以告訴你，我心裡想的絕不是 C++。）

那 Kay 心裡想的「訊息傳遞」到底是什麼？在同一封 1998 郵件裡，他直接點出他的模型是 **Internet**：

> "Think of the internet -- to live, it (a) has to allow many different kinds of ideas and realizations that are beyond any single standard and (b) to allow varying degrees of safe interoperability between these ideas."
>
> （大意是：想想 Internet——它要能活下去，必須一、容許超出任何單一標準的各種想法與實作；二、容許這些想法之間有不同程度的、安全的互通。）

Kay 的背景是生物學與數學，他反覆用的另一個比喻是**生物細胞**（這個細胞比喻散見於他多年來的訪談與 HOPL 等回顧文獻，非出自上面那封 1998 郵件，故此處標為他的一貫說法而非該郵件原文）。把這兩個比喻合起來，他的願景是這樣的：

**理想中的「物件」應該像一個細胞、或像 Internet 上的一台獨立電腦。** 它內部有自己完全私有、外界絕對碰不到的狀態（細胞膜把內外隔開）；它跟外界**唯一**的互動方式，是收發**訊息**（message）——你不能伸手進去改它的狀態，你只能「請求」它做某件事，由它自己決定怎麼回應、甚至要不要回應。系統的智慧不在每個物件內部有多複雜，而在**這些獨立單元之間怎麼互相發訊息**。Kay 自己的話：做出偉大且可成長的系統，關鍵更在於設計「模組之間怎麼溝通」，而不是「它們內部的屬性與行為該長怎樣」。

停在這裡，把這個願景和你的工程直覺對一下——你會發現一件很驚人的事：**Kay 1970 年代描述的「物件」，根本不像你寫的 Java/TypeScript class，它像你做的微服務。** 一個獨立部署、有自己私有資料庫（狀態外界碰不到）、只透過 API／訊息跟其他服務互動（不能伸手進去改別人的 DB）、彼此不知道對方內部怎麼實作的服務——那才是 Kay 要的「物件」。你的 SQS 訊息、你服務之間的非同步通訊，比你 class 裡的 method call 更接近 Kay 的訊息傳遞。

那為什麼我們今天講 OOP，想到的是 class、繼承、`new` 一個物件然後 `.呼叫方法`，而不是 Kay 那個「細胞互發訊息」的願景？因為一個好點子在規模化落地的路上，被複雜度改了形。

## 個案：Kay 想要的 vs 我們得到的

把這個走形攤開，是本章的 worked example。我們對照兩欄：Kay 的訊息傳遞願景，和業界實際落地的 class 中心 OOP。

| 面向 | Kay 想要的（訊息傳遞願景） | 我們得到的（class 中心 OOP） |
|---|---|---|
| 互動的本質 | 物件之間**發訊息**：請求一個獨立單元做某事，由它自己決定如何回應（late binding，執行期才決定） | **呼叫方法**：`obj.method()` 多半被理解成「直接執行 obj 上那段程式碼」，更像一次同步函式呼叫 |
| 物件像什麼 | 像細胞、像 Internet 上的一台機器——自治、狀態完全私有、唯一介面是訊息 | 像一個「裝了資料和函式的盒子」——封裝有做到，但自治與「唯一靠訊息」這層被弱化 |
| 系統智慧在哪 | 在**單元之間怎麼溝通**（訊息的設計） | 在**class 階層怎麼設計**（繼承樹、型別關係） |
| 核心機制 | messaging ＋ 極致的封裝 ＋ late binding | class ＋ inheritance（繼承反而成了焦點） |
| 最接近原意的現代產物 | actor model（Erlang／Akka）、微服務、你的 SQS pipeline | Java／C++／C# 的 class 階層、企業級框架 |

中間那一欄到右邊那一欄的滑動，不是誰背叛了誰，是**規模化落地時複雜度的自然選擇**。原因有幾層，都很現實：

**第一，效能與既有語言的引力。** Kay 的純訊息模型（每個動作都是一次動態分派的訊息）在 1970 年代的硬體上慢。當 OOP 要進入主流、要跟 C 那種貼著機器的效能語言競爭時，妥協幾乎是必然。Bjarne Stroustrup 從 1979 年開始做「C with Classes」，1983 年改名 **C++**——它的設計哲學就是「把 OOP 的好處加進 C，但**不為你不用的東西付出效能代價**（zero-overhead）」。這意味著 C++ 的「物件」是貼著 C 的記憶體模型實作的，方法呼叫盡量編譯成接近直接函式呼叫的東西——這跟 Kay「每次都是動態訊息」的願景在根上就不同。C++ 把 OOP 帶進了主流，代價是它帶進去的是「class 與繼承」這一面，不是「訊息傳遞」那一面。主流對 OOP 的第一印象，從此被 C++ 定義了。

**第二，Java 把這個 class 中心的版本工業化、全球化了。** 1995 年 5 月，Sun 的 James Gosling 團隊釋出 **Java 1.0**，帶著那句史上最成功的口號之一：**「Write Once, Run Anywhere」（WORA，一次編寫、到處執行）**。Java 的賭注是 **JVM**（Java Virtual Machine）：你的 Java 程式不編譯成某款 CPU 的機器碼，而是編譯成中介的 **bytecode**，由各平台上的 JVM 在執行期翻譯。於是同一份程式在 Windows、Solaris、後來的 Linux 上都能跑——這把 ch06 Unix「用 C 換可移植性」那條路又往上推了一層（從「原始碼可移植、要重編譯」推到「編譯產物可移植、不用重編譯」）。Java 趕上了 1995 年的 Web 浪潮（Netscape 瀏覽器支援 Java applet，網頁第一次能跑互動程式），加上 Sun 的市場推力，讓 Java 成為接下來二十年企業後端的預設語言。而 Java 教給整個世界的 OOP，是**徹底 class 中心的**：一切都得包在 class 裡（連 `main` 都得是某個 class 的 static method）、繼承是頭等公民、`interface` 與 `extends` 是你建模的主要工具。一整代工程師（包括教科書、大學課程）學到的「OOP＝class＋繼承＋多型」，是 Java 定義的，不是 Kay 定義的。

**第三，繼承這個機制，被當成了 OOP 的招牌賣點——而這是走形最深的一刀。** 接下來單獨講它，因為這是你天天在還的那筆帳。

工程師的領悟到這裡可以先收一句：**你今天用 NestJS 的 DI、用 interface 加組合、用非同步訊息串服務，其實在不知不覺中走回了 Kay 的原意——而不是你大學課本教的那套深繼承 class 階層。** 你不是在用「更現代的 OOP」，你是在繞過 OOP 落地時的那個走形，回到它本來想去的地方。下一節講清楚那個走形是怎麼回事。

## 繼承：從銀彈到「能不用就不用」

繼承（inheritance）一度是 OOP 最閃亮的賣點。它的承諾聽起來無懈可擊：**程式碼重用**。你有一個 `Animal` class，寫好了 `eat()`、`sleep()`；現在要做 `Dog`，不用重寫，直接 `Dog extends Animal`，免費繼承所有行為，再加上 `bark()` 就好。世界乾淨、整齊、像生物分類學。1990 年代的 OOP 教科書畫滿了這種繼承樹，把它當成軟體重用的銀彈。

然後產業大規模用了十幾年，撞出了一身傷。傷的名字叫**脆弱基底類別問題**（fragile base class problem）：

**當你 `extends` 一個 class，你不是「重用」它，你是把自己的命運和它的內部實作焊死了。** 子類別常常隱性地依賴父類別「內部怎麼做」——某個 method 內部呼叫了另一個 method、某幾個操作的先後順序。父類別的作者哪天為了修 bug 或加功能，改了一個看起來人畜無害的內部細節（改了某個 protected method 的行為、調了內部呼叫的順序），**所有繼承它的子類別可能就悄悄壞掉**，而且父類別作者根本不知道有誰在依賴那個細節。繼承把「父類別的內部實作」變成了子類別依賴的一部分——這恰恰**違反了封裝**。你以為繼承在重用，其實它在你父子之間開了一個後門，讓本該藏起來的實作細節洩漏給了子類別。這是 ch06 那個「不該洩漏的實作從窄介面漏出去」的問題，在繼承這個機制上的化身。

更麻煩的是繼承樹一深就僵。`is-a`（是一種）關係被推到極端會打架：經典的例子是 `Square extends Rectangle`（正方形是一種長方形嘛）——數學上對，但程式上災難：長方形可以獨立改寬和高，正方形不行（改寬必須同步改高），於是子類別沒辦法忠實履行父類別的契約。現實世界的分類學，硬塞進單一繼承樹，遲早爆炸。

產業學到的教訓，被 1994 年那本 OOP 聖經——**《Design Patterns: Elements of Reusable Object-Oriented Software》**（作者 Gamma、Helm、Johnson、Vlissides 四人，人稱 Gang of Four，GoF；1994 年 10 月在 OOPSLA 首發，版權頁標 1995）——濃縮成一句被印成斜體、強調再強調的原則：

> "Favor object composition over class inheritance."
>
> （偏好物件組合，勝過類別繼承。）

這就是你天天在用、卻可能沒查過出處的「組合優於繼承」。它的意思是：與其讓 `Car extends Engine`（車是一種引擎？根本不通），不如讓 `Car` **持有一個** `Engine`（`has-a`，組合）——車裡有一個引擎物件，車透過引擎的介面去用它，但不繼承它的實作。組合是 `has-a`（有一個），繼承是 `is-a`（是一種）；組合靠介面互動（封裝沒被打破），繼承靠共享實作（封裝被打破）。

把這條教訓接回你的日常，閉環就合上了：**你用 `interface PaymentProvider` 加 DI 注入具體實作，而不是 `StripeProvider extends BasePaymentService`，正是「組合優於繼承」的直接實踐。** DI 容器幫你做的，本質就是「把一個物件持有的相依，從外面組裝進去」——這是組合的自動化。你選 interface 而非深繼承樹，是在替 1990 年代那場繼承災難還帳，而你多半是被框架和社群慣例「養成」這個習慣的，沒親身踩過深繼承的坑。這章的價值就在這：讓你知道你手上這個「正確的習慣」是從哪場血戰裡長出來的，這樣你下次看到別人疊三層繼承時，講得出為什麼不該那樣。

順帶清掉一個常被笑的東西：所謂「企業級 OOP 過度設計」——`AbstractFactoryFactory`、`SimpleBeanFactoryAwareAspectInstantiationStrategy` 這種讓人發笑的長名字。這不是 OOP 本身的錯，是**把封裝與多型的工具用過了頭**：明明一個函式能解決的事，硬要包成五個 class 加三層抽象介面，美其名「為了未來的擴充」。這是 ch10「沒有銀彈」會講的偶然複雜度（accidental complexity）的活標本——你用抽象去防禦一個未必會來的未來，付出的是當下實實在在看不懂的複雜度。組合優於繼承的更深一層讀法是：**抽象要按需引入，不是預先囤積。**

（設計模式這個目錄本身——23 個 GoF 模式、Strategy／Observer／Factory 那一串——本書不展開，那是一本工具書的範圍；要學去讀 GoF 原書，本章延伸閱讀有列。本書關心的是「為什麼會有設計模式這種東西」：它們大多是在**繞過繼承的僵硬**、用組合與多型補 OOP 落地後留下的洞。型別系統那一面——`interface` 背後的型別理論、靜態 vs 動態之爭——留給 ch08；函數式陣營對「物件把狀態和行為綁一起」這件事本身的根本批評，留給 ch09。)

## 複雜度搬去哪了

這一跳（OOP：封裝＋多型，以 class 為主流落地形態）搬動的複雜度，記成一筆帳：

| 項目 | 內容 |
|---|---|
| **消滅了什麼負擔** | (1) 「狀態誰都能改、改它的程式碼散一地」——封裝把狀態關進物件，立了單一看門人，改 state 只有一條路（經過有檢查的 method）。(2) 「依型別分流的 switch 散落在每個呼叫方」——多型把判斷收攏回型別自己身上，新增型別不必改呼叫方。(3) 用「物件」建模貼近人對世界的直覺，降低「程式結構 vs 問題結構」的心智落差 |
| **製造了什麼新負擔** | (1) **繼承的脆弱**：`extends` 把子類別焊死在父類別的內部實作上，父類別一改、子類別暗壞——封裝被繼承自己打破（fragile base class）。(2) **class 階層成了新的複雜度沼澤**：深繼承樹、`is-a` 硬塞、為未來預先囤積抽象（企業級過度設計、`AbstractFactoryFactory` 之譏）。(3) **走形稅**：主流落地的 class 中心 OOP 弱化了 Kay 要的「自治＋訊息」，於是物件之間常常是緊耦合的同步呼叫，而非鬆耦合的訊息——這筆複雜度要到微服務／actor model 才被搬回原意 |
| **在什麼情況下洩漏／反噬** | 當你用繼承做「程式碼重用」（而非真正的 `is-a` 型別關係）時——父類別內部一變，繼承鏈悄悄崩，封裝形同虛設；當 class 階層被推到過深、抽象被預先囤積時，讀程式的人得在繼承樹裡爬上爬下才搞懂一個物件的真實行為（行為散在父類別、祖父類別……離散又回來了，只是換了個方向） |
| **這一跳的帳單一句話** | **OOP 把「資料與操作的離散」收攏進物件，但繼承又把「實作的離散」沿著繼承樹重新散開**——複雜度從「水平散落（資料和函式分家）」搬成了「垂直散落（行為藏在繼承鏈各層）」；而 Kay 想用「訊息傳遞」根治的耦合，在 class 中心的落地裡其實沒被根治，只是延後到了分散式時代（微服務）才被迫面對 |

最後一格是這章的核心，也是它在全書地圖上的位置：OOP 沒有消滅「離散」這個本質複雜度，它把離散**換了個方向搬**。封裝與多型收攏了「資料和操作分家」的水平離散，做得漂亮；但繼承這個落地時被當招牌的機制，又沿著繼承樹製造了「行為藏在哪一層」的垂直離散。而 Kay 那個真正想根治耦合的願景（自治單元＋訊息），被效能與既有語言的引力按住了二十年，直到複雜度大到一個程式裝不下、必須拆成分散式系統時，才以「微服務」「actor model」的形態重新浮現——那時你會發現，你在用 SQS 串服務時憑的直覺，是 1970 年代 Kay 就描述過的。好點子沒有死，它只是繞了四十年的遠路。

## 紙上推演

### 推演題一：為什麼 actor model／微服務比 class 繼承更接近 Kay 的原意 **[20 分鐘] ★★**

用 Kay 在 1998 squeak-dev 郵件裡的兩個比喻（Internet、以及他一貫用的生物細胞）口頭講清楚：(a) Kay 要的「物件」具備哪三個特徵？(b) 為什麼你的微服務（或 Erlang/Akka 的 actor）滿足這三個特徵，而一個典型的 Java `class` 物件不滿足？(c) 這對「我們對 OOP 的標準理解走形了多少」說明了什麼？

#### 推演解答

(a) Kay 的物件三特徵：**狀態完全私有**（像細胞膜把內外隔開——外界絕對碰不到內部 state）、**唯一介面是訊息**（你不能伸手改它，只能發訊息「請求」它做事，由它自己決定怎麼回應）、**自治**（像 Internet 上一台獨立電腦，late binding——執行期才決定怎麼回應，能容許超出單一標準的多樣實作互通）。核心一句：智慧在「單元之間怎麼溝通」，不在單元內部多複雜。

(b) **微服務滿足這三個**：每個服務有自己私有的資料庫（別的服務不能直連你的 DB，這是鐵律——一違反就是分散式單體）；服務之間只透過 API／訊息互動（你發一個 SQS 訊息或打一個 HTTP 請求，不能伸手進別人的記憶體）；每個服務獨立部署、獨立決定怎麼處理收到的請求（自治）。**典型 Java class 物件不滿足**：它的「私有」常被繼承、reflection、或同進程的直接引用穿透；`obj.method()` 多半是同步直接呼叫，不是「發訊息給一個自治單元、由它決定要不要回應」；物件之間常緊耦合，A 物件持有 B 物件的引用直接呼叫，沒有 Kay 要的那層鬆耦合訊息邊界。

(c) 說明走形很深：我們把 OOP 理解成「class＋繼承＋封裝」這個**語言內**的東西，而 Kay 要的是一個**像分散式系統**的東西。最接近他原意的產物（微服務、actor model）今天甚至**不被歸類為 OOP**——這正是「造詞人後悔造了那個詞」的具體後果：詞把大家的注意力鎖在了「物件（盒子）」這個次要概念，遮蔽了「訊息（溝通）」這個主要概念。常見誤解是以為 Kay 在「批評 OOP」；不是，他是在說大家把他的點子裡比較小的那一半當成了全部。

### 推演題二：找一個你看過或寫過的繼承濫用，說明組合會怎麼修 **[20 分鐘] ★★**

回想一個你親手寫過、或在 code review 看過的「用繼承做程式碼重用」的例子（常見長相：`extends` 一個 `BaseXxxService`／`BaseController` 只為了拿它幾個共用 method；或一條三層以上的繼承鏈）。(a) 指出它依賴了父類別的哪個內部實作，使它脆弱。(b) 用「組合優於繼承」改寫，說明改完後封裝為什麼回來了。(c) 把這個改動對應到你 NestJS 裡實際在用的機制。

#### 推演解答

(a) 典型脆弱點：`OrderService extends BaseService`，只為了用 `BaseService` 裡的 `this.logTransaction()` 和 `this.validateTenant()`。子類別其實隱性依賴了——比方說 `BaseService` 的 constructor 裡初始化了某個 `this.cache`，而子類別的某個 method 假設 `this.cache` 已經存在。父類別作者哪天把 cache 初始化挪到 lazy（第一次用才初始化），子類別就在某個時序下拿到 `undefined`——父類別作者完全不知道自己弄壞了誰。繼承把父類別的初始化順序變成了子類別的隱性契約。

(b) 組合改寫：不 `extends`，改成 `OrderService` 的 constructor **持有** `private readonly logger: TransactionLogger` 和 `private readonly tenantValidator: TenantValidator`，透過它們公開的介面去用（`this.logger.log(...)`）。封裝回來了，因為現在 `OrderService` 只依賴 `TransactionLogger` 的**公開介面**（一個明確的承諾），碰不到也不依賴它的內部實作——`TransactionLogger` 怎麼改內部，只要介面不變，`OrderService` 無感。`has-a` 把依賴限制在介面上，`is-a` 則讓依賴穿透到實作。

(c) 對應到 NestJS：你根本不用手動 `new TransactionLogger()`——你在 constructor 宣告 `private readonly logger: TransactionLogger`，DI 容器幫你注入。**DI 就是組合的自動化**：它把「一個物件需要持有哪些相依」從程式碼裡的 `new` 抽出來，由容器在外面組裝好注進去。所以你會發現：NestJS 整個框架的骨架（service 互相注入、provider 實作 interface）天生就在引導你走組合，深繼承在 NestJS 裡反而寫起來彆扭——框架把產業的教訓焊進了它的設計。你「自然就這樣寫」，是因為站在了還完帳的人的肩膀上。

### 推演題三：解釋「封裝」與 ch06「資訊隱藏」的血緣 **[15 分鐘] ★★**

口頭講清楚：(a) Parnas 1972 的資訊隱藏，原本講的是什麼粒度的東西？(b) OOP 的封裝把同一個洞見搬到了什麼粒度？(c) 為什麼說「封裝是資訊隱藏的一條落地路線」，而 Unix 的模組＋pipe 是「另一條」？

#### 推演解答

(a) Parnas 1972 講的是**模組／檔案粒度**：一個模組該藏住「一個很可能會變的設計決策」，只露出一個窄而穩的介面，把祕密關在裡面（ch06 的 KWIC：Line Storage 模組藏住「句子存記憶體還是磁碟」）。洞見是「邊界該包住會變的決定」。

(b) OOP 封裝把同一個洞見搬到**物件粒度**：一個物件藏住自己的資料（private state），只露出 method。把「祕密」的單位從一個模組檔案，縮小到一個物件實例。`Account` 物件藏住 `balance` 怎麼存、改它要過哪些檢查，外界只能透過 `deposit()`／`withdraw()` 碰——跟 Parnas 的 Line Storage 是同構的，只是粒度更細、做進了語言的型別系統。

(c) 「資訊隱藏」是**洞見**（該藏會變的決定），它不規定你用什麼語言機制落地。OOP 用「class 的 private＋method」這套語言機制落地它——這是一條路。Unix 用「獨立的小程式＋純文字 pipe」落地它——每個工具藏住自己怎麼做、只承諾讀寫文字，這是另一條路（ch06）。兩條路實作的是同一個 1972 年的洞見。常見誤解是把封裝當成 OOP 獨有的發明——不是，封裝是資訊隱藏的 OOP 方言；同一個意思，Unix 用 pipe 講、微服務用 API＋私有 DB 講（ch17）。看懂這條血緣，你就不會被「OOP vs 函數式 vs 微服務」這種陣營之爭迷惑——它們在「藏住會變的決定」這件事上其實同源，只是把線畫在不同粒度。

## 自我檢核

口頭自答，講得出因果才算過：

1. OOP 出場前，「資料與操作的離散」具體痛在哪？用銀行帳戶的 `balance` 這個裸露欄位舉例：為什麼「狀態沒有單一看門人」會讓 debug 一筆負餘額變成全 codebase grep？
2. 封裝解的是哪一面複雜度、多型解的是哪一面？用你的 `PaymentProvider`／那個付款 switch 的例子，講清楚多型把「依型別分流的判斷」從哪裡搬到了哪裡。
3. 封裝和 ch06 的資訊隱藏是什麼關係？粒度差在哪？為什麼說它們是「同一個洞見的兩條落地路線」？
4. Kay 那句「The big idea is 'messaging'」原意是什麼？他要的「物件」用 Internet／細胞比喻來說具備哪三個特徵？為什麼說你的微服務比你的 Java class 更接近他的願景？
5. 為什麼主流落地的 OOP 變成 class 中心、而不是 Kay 的訊息中心？把 C++（效能／C 的引力）和 Java（WORA／JVM／趕上 Web 與企業浪潮）兩個力道講出來。
6. 「脆弱基底類別問題」是什麼？為什麼說「繼承違反了封裝」？用一個父類別改了內部實作、子類別暗壞的場景說明。
7. 「組合優於繼承」（GoF 1994）在反對什麼、主張什麼？`is-a` 和 `has-a` 在「封裝有沒有被打破」這件事上差在哪？這條原則怎麼對應到你用 interface＋DI 而非深繼承？
8. 這章的複雜度帳單一句話是「OOP 把資料與操作的水平離散收攏進物件，繼承又把實作沿繼承樹垂直散開」——講清楚這兩個「離散」方向的不同，以及為什麼 Kay 想根治的耦合，要到微服務時代才被真正面對。

## 延伸閱讀

- **Alan Kay 的 1998 squeak-dev 郵件**（http://lists.squeakfoundation.org/pipermail/squeak-dev/1998-October/017019.html）。本章那句「The big idea is 'messaging'」的原始載體，也是他用 Internet 比喻講願景的地方。**一定要讀原文那幾段**——它短，但會徹底改變你對「OOP 是什麼」的預設理解。讀的時候對照你做微服務的直覺，你會起雞皮疙瘩。
- **Ole-Johan Dahl, 〈The Birth of Object Orientation: the Simula Languages〉, 2001**（https://www.mn.uio.no/ifi/english/about/ole-johan-dahl/bibliography/the-birth-of-object-orientation-the-simula-languages.pdf）。class 與 object 的源頭，由發明者本人回顧。看他怎麼從「模擬港口的船」這種具體需求、和「想像 Algol 程式執行期的記憶體長相」長出 class——你會懂為什麼 OOP 的出身決定了它後來「用物件對應現實世界東西」的直覺與陷阱。
- **Gamma, Helm, Johnson, Vlissides, 《Design Patterns: Elements of Reusable Object-Oriented Software》, 1994**（GoF）。「組合優於繼承」的出處。**不必整本啃**——讀它的〈Introduction〉那章，特別是把「Favor object composition over class inheritance」印成斜體的那一頁，以及它為什麼這樣主張的論證。其餘 23 個模式當工具書按需查即可（本書不展開模式目錄）。
- **Bjarne Stroustrup, 〈A History of C++〉, HOPL II, 1993**（https://www.stroustrup.com/hopl2.pdf）。C++ 作者親述「為什麼是 class with classes、為什麼 zero-overhead」。看「OOP 進主流要付什麼妥協」這件事最清楚的一手材料——它解釋了為什麼主流 OOP 是 class 中心而非訊息中心。
- **Steve Yegge, 〈Execution in the Kingdom of Nouns〉, 2006**（一篇廣傳的部落格長文；搜標題可得）。對「Java 把一切都硬塞進 class（名詞王國）」的著名吐槽。讀它感受一下「class 中心 OOP 被推到極致」的荒謬，再回頭對照 Kay 的訊息願景——你會更懂這章說的「走形」到底走到了哪。（觀點文，非史料，當作對照組讀。）
