# ch11 — 瀑布：一個被讀反的模型

> **本章解決什麼問題**：上一章（見 ch10）攤開了複雜度的人性面——加人不會線性加速、本質複雜度砍不掉。但人月神話是「診斷」，不是「處方」。1970 年代，大型國防與企業專案需要的不是更多警語，而是一套能落地的流程：怎麼把幾百人、跨好幾年的開發，組織成可預測、可稽核、可分工的東西。本章講方法 strand 在這個需求下長出的第一套重量級答案——以及它如何被一張圖綁架了五十年。瀑布撞的不是技術的牆，是**組織協調與課責**的牆；它用「先設計後實作的線性流程」翻越，帳單則是對需求不確定性的脆弱。而最諷刺的是：那篇被當成瀑布聖經的論文，作者本人畫那張圖是為了說「這樣不行」。本章是全書最重要的「大家讀反了」案例。

## 從你已知的出發

你對「瀑布」這個詞的第一反應，大概跟業界 90% 的人一樣：那是個笑話。一個老古板的、先寫三個月文件再寫程式、需求一變就全盤崩潰的恐龍流程，是敏捷站起來打倒的那個反派。你在某次 sprint retrospective 聽過有人嘲諷「這根本是瀑布」，全場心領神會地笑。

先把這個反射收起來，因為它讓你看不見一件你每天在做、而且做得對的事。

第一個場景：你開一個 greenfield 微服務專案，AWS ECS、TypeScript/NestJS、要當 B2B 與 B2C 之間的資料中介。你做的第一件事不是寫 business logic，是坐下來定 **OpenAPI 契約**。你把 endpoint、request/response schema、錯誤碼先談清楚、寫進 spec，CI 強制這份 spec 跟實作同步，前端團隊拿著它就能並行開工、後端拿著它各自實作。你沒有先寫程式再回頭補文件——你**先設計、後實作**，而且把「設計」固化成一份能被機器檢查的契約。

停一下。你剛剛做的，骨子裡就是瀑布的核心基因：**在動手寫第一行實作之前，先把介面與設計定下來、寫成文件、讓它成為協作的基準**。你不會覺得這是恐龍流程，你覺得這是基本的工程紀律——少了這份契約，三個團隊各寫各的，整合那天就是一場災難。

第二個場景：你做過強監管的 B2B 系統——資料中介常常碰到合約、稽核、資安審查。在那種場景裡，你親身感受過一種在純 B2C 玩法裡不存在的壓力：**有些變更不能「先上線再說、壞了再 rollback」**。它要先有文件、要有人簽核、要留下「誰在什麼時候基於什麼理由批准了這個改動」的軌跡。你那時可能覺得綁手綁腳，但你也隱約知道——在這個場景，這套東西不是純粹的官僚主義，它在解一個真實的問題。

這兩個場景——先定契約、需要簽核——是同一個東西的兩張臉，而那個東西就是本章要替你正名的「重量級流程」。本章要做三件事：第一，把瀑布的歷史真相挖出來，告訴你那篇 1970 年的論文**實際上**說了什麼（劇透：跟你以為的相反）；第二，講清楚重量級流程在解什麼複雜度、它的帳單是什麼；第三，給它一個公平的評價——它不是笑話，它是對一面真實的牆的合理回應，只是後來被用在了錯的地方。

## 那篇論文：一個被斷章取義五十年的活標本

我認為這是軟體史上最被冤枉的一篇論文，沒有之一。

**1970 年 8 月，Winston W. Royce 在 IEEE WESCON 會議上發表了一篇論文，題為「Managing the Development of Large Software Systems」**（管理大型軟體系統的開發）。這篇論文後來被全世界當成「瀑布模型」的奠基文獻、瀑布之父的宣言。問題是：Royce 在這篇論文裡**沒有**用過「waterfall」這個詞，**沒有**主張純線性的開發流程，而且——這是關鍵——他畫出那張著名的線性流程圖，是為了當作一個**有缺陷的反例**，緊接著就說「這樣行不通」。

讓我們照論文的實際結構走一遍，因為理解這篇論文怎麼被讀反，比記住任何結論都重要。

**第一步：他先畫了一張極簡的圖。** Royce 開場先給一個最樸素的兩步驟概念——分析（analysis），然後寫程式（coding）。他說對於小程式，這就夠了。然後他指出，大型系統光這兩步遠遠不夠，於是把它展開成一張更完整的、由上往下流的階段圖：系統需求 → 軟體需求 → 分析 → 程式設計 → 寫程式 → 測試 → 運轉。每一階段的產出餵給下一階段。**這張圖，就是後世截圖當聖經的那張「瀑布圖」。** 把它跟 Royce 緊接著主張的版本並排，你會一眼看出後人截走了什麼：

```text
    純線性（後人截走的那張圖）          Royce 主張的（帶回饋與迭代）
    ┌──────────┐                       ┌──────────┐
    │ 系統需求 │                       │ 系統需求 │◄─────┐
    └────┬─────┘                       └────┬─────┘      │
         ▼                                  ▼            │ 回饋
    ┌──────────┐                       ┌──────────┐      │ 修正
    │ 軟體需求 │                       │ 軟體需求 │◄───┐ │ 設計
    └────┬─────┘                       └────┬─────┘    │ │
         ▼                                  ▼          │ │
    ┌──────────┐                       ┌──────────┐    │ │
    │ 分析     │                       │ 分析     │◄─┐ │ │
    └────┬─────┘                       └────┬─────┘  │ │ │
         ▼                                  ▼        │ │ │
    ┌──────────┐                       ┌──────────┐  │ │ │
    │ 程式設計 │                       │ 程式設計 │  │ │ │
    └────┬─────┘                       └────┬─────┘  │ │ │
         ▼                                  ▼        │ │ │
    ┌──────────┐                       ┌──────────┐  │ │ │
    │ 寫程式   │                       │ 寫程式   │  │ │ │
    └────┬─────┘                       └────┬─────┘  │ │ │
         ▼                                  ▼        │ │ │
    ┌──────────┐                       ┌──────────┐  │ │ │
    │ 測試     │                       │ 測試 ────┼──┴─┴─┘
    └────┬─────┘                       └────┬─────┘
         ▼                                  ▼
    ┌──────────┐                       ┌──────────┐
    │ 運轉     │                       │ 運轉     │
    └──────────┘                       └──────────┘
    一路往下，不回頭            測試暴露的風險回饋去改前面（＝迭代）
    （Royce：risky and          外加「Do it twice」：先做縮小試作版
     invites failure）           走完全程再正式做
```

**第二步——這是所有人漏掉的一步——他立刻說這張圖有問題。** Royce 沒有停在那張漂亮的線性圖上歌頌它。他緊接著指出：這個純線性的模型，問題出在測試階段。因為測試是整個流程的**最後一步**，而很多風險（時序、I/O、資源極限這類只有跑起來才會暴露的問題）要到測試才會現形——可是這時候設計早已凍結，要改就得回頭大改前面所有階段，成本爆炸。他對這個純線性實作下的判決，原文是這一句（已查證，多個獨立來源確認逐字一致）：

> "I believe in this concept, but the implementation described above is risky and invites failure."

翻成你的話：**「我認同這個概念，但上面描述的這種（純線性）實作方式是有風險的、是在招致失敗。」** 注意他的精準——他不是否定「分階段」這個概念本身，他否定的是「一條路走到底、不准回頭」的那種天真實作。換句話說，那張被當成理想範本的圖，在原作者眼裡是**起點兼反例**，是「我們不能就這樣做」的示範。

**第三步：他提出五項修正，把這個有缺陷的起點改造成一個可行的東西。** Royce 沒有只破不立。他給了五條具體建議（以下是他論文裡的五個小節標題，照查證）：

```text
        Royce 1970 的五項修正
        （把純線性的「反例」改造成可行流程）
  ┌──────────────────────────────────────────────────┐
  │ 1. Program design comes first                    │
  │    程式設計先行——在分析定案前先做初步設計        │
  │ 2. Document the design                           │
  │    把設計寫成文件——讓它可溝通、可稽核            │
  │ 3. Do it twice                                   │
  │    做兩次——先建一個試作版（pilot）再正式做       │
  │ 4. Plan, control and monitor testing             │
  │    規劃、控制並監督測試——別把它推到最後賭        │
  │ 5. Involve the customer                          │
  │    讓客戶涉入——正式、深入、持續地參與            │
  └──────────────────────────────────────────────────┘
```

這五條裡，第三條「**Do it twice**」是壓垮「Royce 主張純瀑布」這個說法的決定性證據。他主張：在投入完整開發之前，**先做一個縮小版的試作（pilot model）跑一遍整個流程**，用它來提早暴露那些只有實作了才看得見的風險。他形容這個試作版（已查證，逐字）：

> "Note that it is simply the entire process done in miniature, to a time scale that is relatively small with respect to the overall effort."

**這是迭代。** 「先做一個小版本走完全程、學到東西、再正式做」——這正是後來敏捷、原型法、MVP 的精神內核。Royce 在 1970 年就寫進論文了。一個被全世界當成「反迭代、純線性」象徵的人，論文裡白紙黑字主張迭代。

第五條「**Involve the customer**」同樣反直覺。瀑布的刻板印象是「需求一次談定、之後客戶閉嘴等驗收」。Royce 說的恰恰相反——他主張客戶的涉入要「正式、深入、且持續」（formal, in-depth, and continuing）。他甚至寫過一句很有畫面的話（大意是「在需求定義與系統上線之間放手讓承包商自己跑，是在自找麻煩」，未逐字核對到精確措辭，故標為大意）。這跟敏捷宣言第三條「客戶協作重於合約談判」隔了三十一年遙遙呼應。

所以把整篇論文的真實主張收攏成一句：**Royce 把純線性流程當作一個危險的、會招致失敗的起點，然後主張用「先設計、寫文件、做兩次、嚴管測試、客戶持續涉入」把它改造成一個帶迭代與回饋的流程。** 他是純瀑布的**批判者**，不是它的傳道者。

| 流行說法（瀑布神話） | 史實（照查證） |
|---|---|
| Royce 1970 發明了瀑布模型 | 他畫的線性圖是**反例**；他下的判決是「risky and invites failure」 |
| Royce 主張純線性、不准回頭 | 他主張「Do it twice」（先做試作版）、要回饋環、要迭代 |
| Royce 創了「waterfall」這個詞 | 他**沒用過**這個詞；該詞是後人貼上去的 |
| 瀑布的客戶談定需求後就閉嘴 | Royce 主張客戶涉入要「正式、深入、持續」 |
| 那張線性圖是 Royce 的理想範本 | 那是他要你**別照做**的起點兼警示 |

## 那「waterfall」這個詞是誰貼的？

如果 Royce 沒說瀑布，那這個綁架了他五十年的詞是哪來的？

一般歸給 **Bell 與 Thayer，1976 年**在第二屆國際軟體工程會議（ICSE）發表的論文「Software Requirements: Are They Really a Problem?」（軟體需求：它們真的是問題嗎？）。他們在文中提到 Royce 的論文時，用了「the 'waterfall' of development activities」這個說法——把那張由上往下流的階段圖比喻成瀑布的水流，一級一級往下落、不回頭。注意這個時間差：**「waterfall」這個詞，比 Royce 的論文晚了整整六年才出現**，而且不是 Royce 自稱的，是別人讀他的圖讀出來的印象。

更糟的是後續。Bell 與 Thayer 只是順手用了個比喻，他們的論文重點是需求分析、不是在替整個開發流程背書。但這個生動的比喻流傳開來，而流傳的過程中發生了一件本書一再出現的事——**斷章取義**。後人取走了那張線性圖（因為它好畫、好教、好寫進政府採購標準），丟掉了 Royce 緊接在後的五項修正、丟掉了「Do it twice」、丟掉了「risky and invites failure」那句判決。一張示意「別這樣做」的圖，被裱起來掛在牆上當「該這樣做」的範本。

這裡有個更深的歷史機制值得點出，否則你會以為這只是一場單純的閱讀理解失誤。**為什麼是這張圖、而不是 Royce 的五項修正，被歷史選中？** 因為那張線性圖正好是大型採購官僚體系**想要**的東西。一張清楚的、階段分明的、每階段有明確產出物可以驗收付款的流程圖，完美契合國防部與大企業簽合約、分包、稽核、究責的需求。Royce 的「Do it twice」「客戶持續涉入」反而麻煩——它們意味著計畫會變、需求會調、付款里程碑不好定。組織不是讀錯了 Royce，組織是**選擇性地只聽見它想聽的那一半**。這比單純的誤讀更值得記住：一個想法被斷章取義五十年，往往不是因為大家不識字，而是因為被截走的那一半剛好服務了某種需求。

## 重量級流程在解什麼複雜度

把瀑布從「笑話」裡撈出來，你得先問一個本書每章都要問的問題：它撞的是哪一面複雜度的牆？

不是技術的牆。結構化編程撞的是「人腦裝不下控制流」（見 ch05），OOP 撞的是「資料與行為散落」（見 ch07）——那些是**技術複雜度**。瀑布與它背後的重量級流程撞的是另一種牆：**組織協調與課責的複雜度**。

想像你不是在帶一個五人 squad，而是在 1975 年管一個五百人、橫跨三年、要交付給美國國防部的軍事指揮系統。你面對的問題清單跟你今天的問題清單**根本不在同一個宇宙**：

- **分工**：五百個人不可能擠在同一個 Slack 頻道即興協調。你必須把工作切成能獨立指派、有明確介面的塊，而切塊的前提是先有一份大家都同意的整體設計。
- **可預測**：政府給你一筆固定預算、一個固定交付日，要求你現在就告訴他三年後會交出什麼、花多少錢。你不能說「我們先 sprint 兩週看看」。
- **可稽核**：出了事——飛彈系統算錯座標——要有人負責，要查得出「是哪個環節、誰的決定、基於什麼規格」。沒有文件軌跡，這整件事無法究責。
- **人員流動**：三年的專案，中途一定有人離職、有人加入。新人 ramp-up（見 ch10 的 Brooks's Law）的唯一指望，是有一份寫下來的設計能讓他讀懂這個系統，而不是去問某個「腦袋裡裝著全部」的人——那種人一走，知識就蒸發了。

看出來了嗎？**瀑布的每一個你討厭的特徵，都是對上面某一條的回應。** 先設計後實作，是為了能分工與預測；大量文件，是為了能稽核與抵抗人員流動；階段化的簽核關卡（phase gate），是為了能在每個里程碑究責與付款。這套流程不是某個老古板拍腦袋想出來折磨工程師的——它是一群人面對「在強課責、固定預算、大規模、長週期、高人員流動」的約束下，怎麼把軟體做出來的真實答案。

換成你在那個時空，你未必想得出更好的。

這套思路在 1980–90 年代被系統化、制度化，最具代表性的就是 **CMM（Capability Maturity Model，能力成熟度模型）**。它的脈絡值得知道，因為它把「為什麼需要重量級流程」這件事講得很白：1984 年，美國國防部出資成立了卡內基美隆大學的軟體工程研究所（SEI）；**Watts Humphrey**（從 IBM 退休後加入 SEI）在美國空軍的要求下，著手做一件具體的事——**國防部把軟體外包給一堆廠商，但它沒有任何工具能評估這些廠商「做軟體的能力到底靠不靠譜」**，於是 Humphrey 設計了一套模型，把一個組織的軟體開發過程成熟度分成五級（從第一級的「混亂、靠英雄」到第五級的「持續最佳化」）。CMM 後來演化成 CMMI，成了你接政府或大型企業案子時可能聽過的那串認證。

我不展開 CMM 的各等級細節（本書範圍邊界），但你要抓住它的**動機**：CMM 解的複雜度，跟瀑布是同一面牆——**當你要把關鍵任務外包給一個你不認識的組織，你需要一種方法去相信他們做事是可預測、可重複、可稽核的，而不是靠運氣和個別英雄。** 這是「可信任的協調」這個複雜度的制度化答案。你今天看一家廠商有沒有 SOC 2、ISO 27001 認證才敢把資料交給它——那股「我需要一個可稽核的信任憑證」的衝動，跟國防部當年要 CMM 是同一個。

## 帳單：重量級流程付出了什麼

公平歸公平，瀑布與重量級流程的帳單是真實而沉重的，而且這些帳單正是下一章敏捷反叛的全部理由（見 ch12，那是對重量級流程的正面反叛）。把帳算清楚：

**第一筆，也是最致命的一筆：對需求不確定性的脆弱。** 純線性瀑布有一個藏在骨子裡的假設——**需求在一開始就能被完整、正確地知道並凍結**。整個流程是建立在「先把要做什麼想清楚，之後就照著做」之上的。但軟體最難的部分恰恰是「搞清楚到底要建什麼」（這是 Brooks 說的本質複雜度，見 ch10），而這件事在真實世界裡**幾乎不可能在動工前就完全想清楚**。需求會變，因為客戶看到半成品才知道自己真正要的不是當初說的那個；市場會變；理解會深化。一旦需求變了，瀑布的成本曲線就反咬你——你要回頭改設計、改文件、改一切，而越往後改越貴。諷刺的是，Royce 的「Do it twice」正是為了治這個病，而後人偏偏把這帖藥丟了。

**第二筆：回饋太慢。** 在純瀑布裡，你要走到測試階段——可能是專案開始一年後——才第一次看到系統真的跑起來，才第一次拿到「這東西到底對不對」的回饋。一年的猜測無人驗證。等回饋來時，錯誤已經層層疊在彼此之上，修起來是天文數字。你今天靠 CI/CD 一天部署好幾次、靠 feature flag 灰度放量、靠監控即時看真實流量的反應——你享受的是**短回饋環**，而瀑布的原罪就是回饋環長到以年計。

**第三筆，最微妙的一筆：文件變成目的本身。** 重量級流程要文件，本意是讓文件**服務**於溝通、稽核、抵抗人員流動。但流程一旦制度化、一旦文件成為 phase gate 的驗收物、成為究責時的擋箭牌，它就會發生異化——**人開始為了通過審查而寫文件，而不是為了把事情想清楚而寫文件**。你大概見過那種沒人讀、跟實作早已脫節、純粹為了應付稽核而存在的設計文件。那一刻，文件從手段墮落成了目的，從減少複雜度的工具變成了增加複雜度的官僚負擔。這是所有重量級流程的通病：**為了治理複雜度而引入的流程，本身會長成一種新的複雜度。**

但——這是本章要你記住的公平話——**帳單沉重不等於這套流程是錯的。它只是有適用邊界。**

在某些領域，重流程不是錯，是必須。問問自己第二筆帳「回饋太慢」與第一筆帳「需求會變」的前提是什麼：它們假設**需求其實會變、而且變更成本不高、回饋拿得到**。但如果你在做的是：

- **航太飛控軟體**：需求由物理定律與適航法規定死，不會因為「使用者想要個深色模式」而變；而「先上線壞了再 rollback」在這裡的代價是飛機掉下來。
- **醫療器材韌體**：受 FDA 等監管，每個變更都要重新驗證、留軌跡；變更成本高到你**寧可前期多花十倍力氣想清楚**。
- **核電廠控制系統**：同上，需求穩定、變更成本是災難級。

在這些領域，**需求真的相對穩定、變更成本真的極高、且不允許「快速試錯」**。瀑布前期的「先把一切想清楚、寫成可稽核的文件、嚴格分階段驗證」——不是官僚主義，是工程倫理。你在強監管 B2B 場景隱約感受到的那股「有些地方真的需要文件與簽核」的直覺，就是這個邊界的縮影。同一套流程，用在需求飛速變動的消費級 App 上是災難，用在飛控上是救命。**問題從來不是瀑布本身好或壞，是有沒有用在它適用的牆上。** 把適用於航太的流程套到一個三人的 CRUD App，跟把適用於三人 CRUD 的「先上線再說」套到飛控上，是同一種錯誤的兩個方向。

這正是鐘擺隱喻的一次具體擺動（見 ch01）：手工↔自動、輕量↔重量這組張力，會隨著「課責壓力」與「變更速度」的不同而擺到不同位置。沒有一個位置永遠對。

## 複雜度搬去哪了

瀑布與重量級流程這一跳，照例結一筆帳——它消滅了什麼負擔，又製造了什麼新負擔。

| 項目 | 內容 |
|---|---|
| **撞的牆** | 組織協調與課責的複雜度：在強課責、固定預算、大規模（數百人）、長週期（數年）、高人員流動的約束下，怎麼把軟體做出來且做得可信 |
| **這次主導的 strand** | 純粹的方法 strand（流程/治理），與抽象 strand 幾乎無關——它管的是「怎麼蓋」而非「用什麼蓋」 |
| **消滅了什麼負擔** | 「靠英雄與口頭協調」的不可規模化負擔——用先設計後實作換來可分工、用文件換來可稽核與抗人員流動、用階段關卡換來可預測與可究責 |
| **複雜度搬到哪** | 從「執行時的混亂協調」搬到「**前期的設計與文件成本**」——複雜度沒消失，它被移到專案開頭，變成一大筆必須先付的規劃、文件與簽核成本 |
| **新製造的負擔** | (1) 對需求不確定性脆弱——整套流程賭「需求能在動工前凍結」，賭輸了改起來成本隨階段指數上升；(2) 回饋環長到以年計，錯誤層層堆疊才被發現；(3) 文件異化——制度化後人為通過審查而寫文件，文件從手段墮落成目的，反而變成新的官僚複雜度 |
| **會在哪洩漏／反噬** | 當它被用在**需求高速變動、變更成本低、能快速試錯**的領域（消費級軟體、新創探索期）時，前期凍結需求的假設徹底失效，整套流程從資產變成枷鎖——這正是 1990 年代末敏捷反叛的全部燃料（見 ch12）。反過來，當輕量流程被用在需求穩定、變更成本災難級的領域（飛控、醫療）時，缺乏文件與驗證又會反噬——抽象兩端都會洩漏 |
| **這一跳的帳單一句話** | 用「前期的設計與文件成本」換「大規模協作的可預測與可稽核」——複雜度從執行期搬到了規劃期，而一旦需求在動工後改變，這筆預付的成本就大半打了水漂 |

## 紙上推演

### 推演題一 **[15 分鐘]** ★★

「瀑布 vs 敏捷」常被講成一場善惡分明的對決——僵化過時的瀑布，被靈活進步的敏捷打倒。用 Royce 1970 論文的**真實主張**，論證這個對立有多大成分其實是個**稻草人**（straw man）。具體說：Royce 論文裡的哪些主張，其實跟後來敏捷宣言的主張是同一個方向？你的解釋裡要出現「Do it twice」與「客戶涉入」這兩個錨點。

#### 推演解答

這個對立的稻草人成分非常大，因為它把「瀑布」等同於「純線性、需求凍結、客戶閉嘴」這個**從來不是 Royce 主張、而是後人從一張圖誤讀出來的版本**，再讓敏捷去打這個稻草人。

對照 Royce 論文的真實主張：

- **「Do it twice」對上敏捷的迭代**：Royce 主張先做一個縮小版的試作（pilot model）走完整個流程、學到東西再正式做。這就是迭代、就是原型法、就是 MVP 的精神內核。敏捷的「短迭代、先交付能跑的小版本、從回饋中學」並不是推翻 Royce，而是把 Royce 這條被丟掉的建議**撿回來放大**。兩者在這一點上同向。

- **「客戶涉入」對上敏捷宣言第三條**：Royce 主張客戶的參與要「正式、深入、持續」。敏捷宣言第三條是「客戶協作重於合約談判」。隔了三十一年，講的是同一件事——別讓客戶談定需求後就消失，要他持續在場校正方向。又是同向。

所以真實的歷史不是「敏捷推翻瀑布」，而是「**敏捷推翻的是後人對 Royce 的誤讀版（純線性瀑布），而這個誤讀版恰恰也是 Royce 自己反對的**」。Royce、敏捷、與「純瀑布」三者的關係是：Royce 和敏捷站在一邊（都要迭代、要回饋、要客戶涉入），純瀑布的刻板印象站在另一邊——而這個刻板印象沒有真正的作者，它是一張圖被斷章取義的產物。

常見誤解要點破：以為「替瀑布平反」就是「敏捷錯了」。不是。敏捷對「純線性、需求凍結」這個東西的批判完全正確——只是那個東西本來就不該叫「Royce 的瀑布」。平反的是 Royce 這個人，不是純瀑布這套做法。

### 推演題二 **[10 分鐘]** ★★

找出你目前工作中，哪些環節其實是「瀑布式」的（先把設計/契約定下來、後實作）——而且這樣做是**對的**。挑一個具體環節，說清楚：它為什麼適合先設計後實作？如果反過來「先寫實作、再回頭補設計」會付出什麼代價？

#### 推演解答

最乾淨的例子就是本章開頭的 **OpenAPI 契約先行**。

為什麼這個環節適合先設計後實作：因為它是一個**多方協作的介面**。前端、後端、可能還有第三方串接，三方要並行開工。介面是他們之間的共享假設——如果不先把它定死，三方各自憑想像實作，整合那天就是 Brooks 說的 O(n²) 溝通成本總爆發（見 ch10）。先定契約，等於先把「我們之間怎麼對接」這個最容易出協調災難的點凍結下來，讓各方能在穩定的假設上獨立工作。這正是瀑布解的那面牆——大規模協作下的可分工——在小尺度上的合理殘留。

反過來「先實作再補契約」的代價：後端各 endpoint 各長各的、命名與錯誤碼不一致、前端只能等後端寫完才知道介面長怎樣（序列化而非並行，速度砍半），整合期才發現雙方對某個欄位的理解不同——而這時兩邊程式都寫好了，改任一邊都牽動一票已完成的程式碼，成本遠高於動工前在 spec 上改一行。這就是瀑布「越往後改越貴」的成本曲線，在你自己專案裡的微縮版。

關鍵領悟：「先設計後實作」不是瀑布的原罪，是它的**正確基因**。瀑布的錯不在「先設計」，在「**把整個系統的所有設計都凍結在最前面、且不准回頭**」。把設計先行用在「穩定的協作介面」上是對的（介面本就該穩定），用在「會隨理解深化而變的 business logic」上才是錯的。你的 CI 強制 OpenAPI 同步，是把瀑布基因用對了地方的範例。

### 推演題三 **[15 分鐘]** ★★★

反事實推理：**如果需求永遠不變、而且變更成本極高，敏捷還有優勢嗎？** 用這個極端假設，反推「敏捷優於瀑布」這個常見說法到底依賴哪些隱藏前提。然後用這個分析回答：為什麼航太、醫療這些領域至今仍大量採用重量級流程，這是不是它們「落後」？

#### 推演解答

如果需求永遠不變、且變更成本極高，那麼**敏捷的核心優勢會大幅消失，瀑布式的重流程反而更合理**。

先看敏捷的核心賭注是什麼：它賭「需求一定會變，所以與其花大力氣預測未來、不如用短迭代去**適應**變化」（這是 ch12 的核心，敏捷把賭注從「預測」改成「適應」）。這個賭注的價值，完全建立在「需求會變」這個前提上。把前提抽掉——假設需求真的永遠不變——那麼「擁抱變化」這個能力就無處可用，敏捷為了保持靈活而付出的成本（不預先做完整設計、接受重工、持續重構）反而變成純粹的浪費。在一個不會變的世界裡，前期一次把設計做到位、之後嚴格照做，是**最有效率**的，因為你不必為從不發生的變化保留彈性。

再加上「變更成本極高」這條：它讓「快速試錯、壞了再改」的敏捷打法直接變成自殺——因為這裡沒有便宜的「改」。當改一次的代價是飛機墜毀或病人受害時，「先上線再說」不是勇敢，是不負責任。此時前期窮盡一切、把需求與設計驗證到滴水不漏（即使慢、即使貴），才是唯一負責任的做法。

所以「敏捷優於瀑布」這個說法依賴的**隱藏前提**至少有三條：(1) 需求會變且無法預先完全知道；(2) 變更成本不高、能負擔重工與試錯；(3) 拿得到快速回饋（能上線、能觀測真實反應）。這三條在消費級軟體裡幾乎總是成立，所以敏捷在那裡幾乎總是對的——但它們不是宇宙定律。

航太、醫療大量採用重量級流程，**不是落後，是它們所在的牆不一樣**。在那些領域，上述三個前提全部翻轉：需求由物理與法規定死、相對穩定；變更成本是災難級；且不允許拿真實使用者試錯。在這組約束下，重流程是對的工具。把它們的選擇叫「落後」，等於用消費級軟體的世界觀去評判一個前提完全不同的世界——這恰恰是本章警告的那種錯誤：把一套流程的適用邊界忘掉，當成普世真理。**成熟的工程判斷不是「永遠敏捷」或「永遠瀑布」，是認得出自己站在哪一面牆前。**

## 自我檢核

口頭自答；講得出因果才算過關，講不清楚代表還沒懂。

1. Royce 1970 論文裡那張「瀑布圖」，在他自己的論證裡扮演什麼角色？他緊接著對純線性實作下了什麼判決（那句 risky and invites failure 的精確意思）？
2. Royce 提出的五項修正裡，哪一項最能證明「他主張迭代而非純線性」？為什麼「Do it twice」直接打臉「Royce 是純瀑布之父」這個說法？
3. 「waterfall」這個詞是誰、在哪一年貼上去的？它比 Royce 的論文晚了幾年？為什麼說組織是「選擇性地只聽見想聽的那一半」，而不只是單純讀錯？
4. 重量級流程（瀑布/CMM）撞的是哪一面複雜度的牆？為什麼這面牆跟結構化編程、OOP 撞的牆**性質不同**？（提示：技術複雜度 vs 組織課責複雜度）
5. 把瀑布「你討厭的三個特徵」（先設計後實作、大量文件、階段簽核）各自對應到它在解的一個組織問題（分工/稽核/抗人員流動/可預測）。
6. CMM 的真實動機是什麼？國防部為什麼需要它？這跟你今天要求廠商有 SOC 2 / ISO 27001 認證是同一股什麼衝動？
7. 重量級流程的三筆帳單是什麼？其中「文件異化」是怎麼發生的——文件本來服務於什麼、又怎麼墮落成目的本身？
8. 為什麼說「瀑布在航太/醫療不是落後」？「敏捷優於瀑布」這句話依賴哪三個隱藏前提？這三個前提在什麼領域不成立？

## 延伸閱讀

- **Royce, "Managing the Development of Large Software Systems"（Proc. IEEE WESCON, 1970）**——本章的核心一手文獻，全書最該親自讀的論文之一，約 11 頁，網路上有多份 PDF（如 praxisframework.org、cs.huji.ac.il 等）。**讀法**：讀到那張線性流程圖時先別停，往下讀他怎麼說它「risky and invites failure」、再讀五項修正特別是「Do it twice」那節——你會親眼看見一篇論文怎麼被截走一半。讀完你對「瀑布」這個詞的感覺會永久改變。
- **Bell & Thayer, "Software Requirements: Are They Really a Problem?"（Proc. 2nd ICSE, 1976）**——「waterfall」一詞的出處。不必通讀，找到它引用 Royce、稱之為「the 'waterfall' of development activities」那一句，看一個影響五十年的詞是多隨手地被造出來的。
- **David Wheeler, "The Waterfall Model"（dwheeler.com/essays/waterfall.html）**——對瀑布神話最冷靜可靠的二手整理之一。Wheeler 點明 Royce 稱自己的純線性版本為「simpler method」並明確「recommended against」它，並追溯瀑布如何被後來的標準（如 DOD-STD-2167）固化進政府採購。適合在讀完 Royce 原文後拿來對照、確認自己沒讀偏。
- **Watts Humphrey, "Managing the Software Process"（1989）**——CMM 思想的奠基書。本書不展開 CMM 等級細節，但若你想理解「為什麼有人會想把軟體開發過程分成五級成熟度」，這是源頭。讀「為什麼需要評估一個組織的過程成熟度」的動機部分即可，對應你今天看廠商認證的直覺。
- **Tobias Mayer / pragtob 等「reading the original Royce paper」系列部落格**——若你一時拿不到或讀不動原論文，這類「逐段帶你讀 Royce 原文」的文章是好的入口（搜尋關鍵字 "reading the original waterfall paper"）。挑有**逐字引用** Royce 原句的那幾篇讀，避開只給結論的。它們的共同主題正是本章主題：一篇主張迭代的論文，如何被讀成了它的反面。
