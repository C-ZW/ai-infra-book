# 寫作風格指南（內部文件，非書籍內容）

本文件是全書的寫作契約。每一章都必須遵守，否則全書會失去一致性。
本書書名（工作名）：《複雜度守恆：軟體發展史的工程師讀法》。

## 這本書是什麼、不是什麼

- **是**：一本「欣賞與直覺」的軟體史書，從**資深後端工程師的視角**讀。目標是讓讀者看懂——軟體七十年史不是一串產品與人名的編年帳，而是**人類反覆撞上「複雜度」這堵牆、再反覆翻過去**的故事；每一次翻越要嘛是一層新**抽象**（idea strand），要嘛是一套新**流程／方法**（method strand）。讀完有「原來我每天用的東西是這樣長出來的、原來這個坑前人早就踩過」的通透感，**而且能用自己的話對另一個工程師講清楚「我們為什麼會走到今天」**。
- **不是**：
  - 不是編年大事記——不是「某年某人發明某物」的條列；歷史事件**服務於思想與方法的演化邏輯**，不是為了湊時間軸（時間軸總表進附錄 A）。
  - 不是技術教科書——不教你寫 Lisp、不教你 OOP 語法、不教你架 K8s；本書講的是「**為什麼會有這東西、它在解什麼複雜度、它的代價是什麼**」，機制只講到能支撐這個判斷的深度。
  - 不是懷舊抒情——不堆「那個純真年代」式的鄉愁；歷史要冷靜、有觀點、可被另一個工程師檢驗。
  - 不是轉職用書——純興趣，不服務 AI Infra 敘事；但工程橋接要讓它讀起來像母語，不是外語。
- **全書貫穿主張（核心商品，ch21 總收官驗收）**：
  1. **複雜度守恆**：抽象不消滅複雜度，它把複雜度**搬家**——搬到別人看不見的地方（編譯器、runtime、雲、別人的團隊），代價在抽象洩漏時回來找你。
  2. **兩條 strand 交織**：「我們用什麼蓋（抽象）」與「我們怎麼蓋（方法）」是同一場對抗複雜度戰爭的兩面，全程互相觸發（結構化編程既是語言事件也是方法事件）。
  3. **鐘擺**：效能↔生產力、集中↔分散、單體↔分散式、手工↔自動——同一組張力反覆擺盪，每次回擺都帶著上次的教訓。
  4. **永恆的真話**：Brooks（沒有銀彈）、Conway（組織形狀=系統形狀）、Dijkstra（測試只能證明 bug 存在）——這些四十年前的話為什麼今天還準。

- 台灣用語基準：程式／程式碼（不用「代碼」）、函式／函數（程式脈絡用「函式」）、編譯器、直譯器、最佳化（也可「優化」，全書擇一：用「最佳化」）、記憶體（不用「內存」）、預設（不用「默認」）、物件（不用「對象」）、品質（不用「質量」）、資訊（不用「信息」）、相依（dependency）、容器（container）。**禁止簡體中文詞彙**。

## 每章骨架（必須遵守，標題名稱照抄——一致性掃描會 grep 這些標題）

1. 章首：`# chNN — 章名：副標`，接著 blockquote：
   `> **本章解決什麼問題**：…（這章在全書「對抗複雜度」地圖的哪裡、撞的是哪一面複雜度的牆、跟前後章的因果關係，3–5 行）`
   每個 Part 的第一章（ch01、ch04、ch07、ch10、ch13、ch16、ch19）在 blockquote 後放**全書 ASCII 路線圖**（基準圖在 outline.md，照抄並把「◄你在這裡」標到本章），讀者偏好架構圖先行。
2. `## 從你已知的出發` — 橋接段：把本章的核心問題錨定在讀者**具體的**工程實戰上（點名場景：你的 cache library、ECS 微服務、RDS CPU 降 40%、FluentBit pipeline、SQS 冪等消費者…），讓他在讀歷史前先「站在自己的經驗上」。要具體，不要泛泛。
3. 核心內容（多個 `##` section）。**歷史敘事與工程洞見交織，但歷史服務於「對抗複雜度」的邏輯**，不是編年史。每章明確點出：撞的是哪一面複雜度的牆、用抽象還是方法翻越、複雜度搬去了哪、代價是什麼。**兩條 strand（抽象/方法）哪一條主導要說清楚**。
4. `## 複雜度搬去哪了` — 本書招牌段落（每章必有，名稱照抄）：本章的躍遷**消滅了什麼負擔、又製造了什麼新負擔**；新抽象/新流程**在什麼情況下會洩漏／反噬**（對應讀者偏好的「故障視角」）；用一句話總結「這一跳的帳單是什麼」。用表格或條列。
5. `## 紙上推演` — 動手做的形式（純敘事書，無可執行 lab；2026-06-13 訪談定案）。每章 2~4 題，標注預估時間（如 **[15 分鐘]**）與難度（★~★★★）。題型：**口頭重講**一段歷史因果、**翻譯**（把一個現代工具對應回它的歷史原型）、**反事實推理**（如果沒有 X 會怎樣）、**找出**一個流行說法的史實破綻、**對帳**（把你維運過的某系統對應到本書某個歷史模式）。題目後緊接 `### 推演解答` 給完整解答與討論（為什麼這樣想、常見誤解）。
6. `## 自我檢核` — 5~8 個問題，**口頭自答**：講得出來才算過關，講不清楚＝還沒懂。問題對齊「能不能把這段歷史的**因果**講給另一個工程師聽」的深度，優先出「為什麼」與「如果不這樣會怎樣」而不是「是什麼／哪一年」。
7. `## 延伸閱讀` — 真實連結（以 `_meta/landscape-2026-06.md` 驗證過的為優先），每條一句話說明為什麼值得讀、讀哪一段／看哪一段。優先：原始論文、當事人的書/演講、可信的歷史整理。

## 深度標準

- **每章至少一個 worked example（歷史書的版本＝一個被講透的個案）**：不是計算題，而是**把一個關鍵事件／決策／論證攤開到讀者能複述其因果**——例如「Royce 1970 論文到底畫了什麼、為什麼後人讀反了」「Brooks 為什麼說加人會更慢，OS/360 發生了什麼」「Codd 的關聯模型把什麼複雜度搬走了」。要有具體細節（人、時間、技術約束、決策的代價），不准只給結論。outline 已為每章指定基準個案。
- **史實照 landscape，不准憑記憶**：年份、歸屬、引文、軼事一律以 `_meta/landscape-2026-06.md` 為準；landscape 沒有的自行查證（每章 **3~8 次** WebSearch/WebFetch）並 hedge。**傳說要明白標為傳說**（Hopper 的飛蛾 bug、Lovelace「第一位程式設計師」的歷史學爭議、Royce 瀑布誤讀、「Considered Harmful」標題是編輯下的、Kay「OOP 是訊息傳遞」的原意）——這些「大家記錯的事」是本書的趣味與可信度所在，**必須引用 landscape 的查證版而非流行版**。
- **名言要查證**：禁止憑記憶引用「名人名言」。引用前對照 landscape；landscape 沒有的自行查到原始出處，查不到原文就**改寫成「大意是」並標未逐字查證**，不偽造逐字引文。
- **誠實標示「這是傳說／這有爭議」**：歷史歸屬常被簡化（誰「發明」了 X）。凡有爭議的優先權／歸屬，明寫「歸屬有爭議」並給 landscape 的版本，不選邊站當定論。
- **每個躍遷都要回答「它在解什麼複雜度」**：禁止無動機的「然後出現了 X」。每個新抽象/新流程出場都要先講清楚**前一個狀態的痛**（撞了哪面牆），再講它怎麼翻越，再講帳單（複雜度搬去哪）。
- **每章至少一處明確的「工程師的領悟」**：本書核心商品是「通透感」。每章至少一處回答「**這段歷史對今天的你意味著什麼**」——而且寫到讀者能用自己的話轉述，不是堆形容詞。
- **時效性與前瞻誠實 hedge**：談 2023–2026 的當下（AI 輔助編程、平台工程等）一律標時點 `（2026-06）`，明確區分「已成定局的歷史」與「還在演化、判斷可能被推翻」的當下；不假裝預知未來。
- 寧深勿廣，不灌水。每章 5,000~9,000 中文字，依主題深度調整。

## 事實正確性

- 歷史與時效性事實（年份、人物、論文、引文、產品版本、連結）以 `_meta/landscape-2026-06.md` 為事實基準；本章特定細節自行用 WebSearch/WebFetch 查證（每章 3~8 次）。
- 與 landscape 衝突時：**以 landscape 為準**；若你查到 landscape 可能有誤，**在章末用註記標出疑點**（「⚠️ 待協調者複核：landscape 說 X，但來源 Y 說 Z」），不要自行改寫 landscape（一致性由協調者 P3/P4 統一處理；重大修正需兩個獨立來源）。
- 任何「使用量／市佔／LOC」類統計一律標為估計值並 hedge（這類數字最常被以訛傳訛）。
- 延伸閱讀只放真實存在的連結；沒把握標「（未驗證）」。

## 格式

- Markdown。主要層級 `##`，次層 `###`。
- 圖：ASCII diagram（全書地圖、演化譜系圖、strand 交織圖、時間軸片段），必須通過 `check_diagrams.py` 檢查（**CJK=2 欄寬規則**——中文字佔 2 欄、ASCII 佔 1 欄，框線要對齊）。譜系/演化圖用 `─ │ ┌ ┐ └ ┘ ├ → ↓` 繪製；不追求精確比例，追求關係清楚。
- 跨章引用格式：`（見 chNN）`，不要寫死檔名路徑。跨章主題點到為止＋指向擁有它的章，不重複展開（範圍邊界見 outline.md）。
- 表格用於：複雜度帳單對照、strand 對照、年代里程碑對照、流行說法 vs 史實對照。

## 語氣

工程師對工程師。冷靜、直接、有觀點（可以寫「我認為這是軟體史上最被低估的一篇論文」）。不行銷、不喊口號、不堆「太神奇了」式空洞感嘆——通透感要用內容本身造成，形容詞是結果不是手段。允許冷幽默與對「業界迷思」的吐槽（但吐槽要有史實根據）。歷史人物當人寫，不造神也不踩低：講清楚他們當年面對的真實約束，讓讀者理解「換成我在那個時空也未必想得到」。對今天的技術潮流保持歷史縱深的清醒——見過鐘擺擺過幾輪的人，不會把每個新詞都當革命。
