## Part I — 機器與第一層抽象（ch01–03）

### ch01 — 對抗複雜度：軟體史的一條主線（序章）
- **目標**：讀完能說出本書的讀法——軟體史是「反覆撞上複雜度、用抽象或方法翻越」的故事；能講出「複雜度守恆」與「兩條 strand」兩個全書工具；能畫出全書地圖。
- **必涵蓋**：開宗明義拋出核心主張（複雜度守恆、抽象只是搬家）；兩條 strand 的定義與它們如何交織；鐘擺隱喻（四組張力）先亮相；全書 ASCII 路線圖（基準圖首次出場）＋全書四條驗收主線（守恆、交織、鐘擺、永恆真話）；用一個讀者熟悉的當代例子當「全書縮影」——例如 ORM：它把 SQL 的複雜度搬到框架裡，N+1 query 與 leaky abstraction 是帳單回來找你（預告 ch13/ch19）。
- **不涵蓋**：任何具體歷史（從 ch02 開始）；任何結論性判斷（留給 ch21）。
- **橋接**：你的多層 cache library 全團隊採用——你已經在做「把複雜度搬到一個抽象後面讓別人不用想」這件事，也吃過 cache 失效（cache invalidation，CS 兩大難題之一）的帳單。本書就是這件事的七十年版。
- **個案（worked example）**：拿讀者自己的 cache library 當顯微鏡，逐項拆解「它消滅了什麼負擔／製造了什麼新負擔（失效、一致性、debug 變難）」，示範全書每章末「複雜度帳單」的記法。
- **紙上推演**：用「複雜度搬去哪了」拆解你維運過的一個抽象（自選：ORM、message queue、K8s）；把「鐘擺」四組張力各舉一個你親身經歷的例子；口頭講「為什麼說抽象不消滅複雜度只是搬家」給一個資淺工程師聽。

### ch02 — 儲存程式：當程式變成資料
- **目標**：能講清楚「儲存程式」為什麼是軟體史的第一次大躍遷（程式從接線變成可操作的資料）；能說出在它之前「寫程式」是什麼樣子；能擺正 Lovelace/圖靈/馮紐曼各自的真實貢獻與歷史學爭議。
- **必涵蓋**：Babbage 分析機與 Lovelace 的 Note G（Bernoulli 數）——「第一位程式設計師」標籤的歷史學爭議照 landscape，誠實呈現兩種觀點；圖靈 1936《On Computable Numbers》與「可計算」的邊界（圖靈機是思想實驗、不是機器）；ENIAC（1945–46）怎麼「編程」——接線與插板、ENIAC Six 女程式設計師；**儲存程式概念**：EDVAC 報告（1945，馮紐曼署名與 Eckert/Mauchly 的功勞爭議照 landscape）、首台運行機（Manchester Baby 1948／EDSAC 1949）；為什麼「程式與資料放同一個記憶體」是關鍵——程式變成可被程式操作的東西（編譯器、直譯器、後來的一切都靠這個前提）。
- **不涵蓋**：編譯器（ch03）；圖靈機的數學細節（指向延伸閱讀）；硬體電路細節；二戰密碼學史（一句話帶過）。
- **橋接**：你部署的每個 container image 都是「資料」，跑起來才是「程式」——code as data 是你天天依賴卻不會注意的前提；自我修改的部署腳本、用程式產生程式（codegen、你的 OpenAPI→client 生成）都站在 1945 這個概念上。
- **個案**：ENIAC 改算一道新題目要重接幾天的線 vs 儲存程式機器只要載入新指令——把「換工作 = 換資料 vs 換接線」對比攤開，講清楚這一跳省掉的是什麼複雜度。
- **紙上推演**：解釋「為什麼說圖靈機不是一台機器」給一個只聽過『圖靈』的人；反事實：如果程式與資料分屬不同記憶體（哈佛架構極端版），編譯器還可能出現嗎；找出「Lovelace 是第一位程式設計師」這句話需要加上哪些但書。

### ch03 — 編譯器的賭注：把「對機器說話」外包
- **目標**：能講出從機器碼→組語→高階語言的兩跳各自外包掉什麼；能重講 FORTRAN「自動編程」的賭注（機器產生的碼能不能跟人手寫的一樣快）——這是「效能 vs 生產力」鐘擺的第一次擺動；能擺正 Hopper「第一個編譯器」的歷史學分寸。
- **必涵蓋**：機器碼的痛（位址全手算、改一行牽動全部）；**組語與組譯器**：符號取代數字，第一次「不直接對機器說話」（Wilkes/EDSAC 的 initial orders）；**Grace Hopper 與 A-0（1952）**——「第一個編譯器」的說法照 landscape 校準（A-0 以現代定義更像 loader/linker）、FLOW-MATIC 對 COBOL 的影響；飛蛾 bug（1947）標明為流傳故事；**FORTRAN（1957，Backus，IBM）**：「automatic programming」的賭注、當年程式設計師對「編譯器寫的碼會不會慢」的真實懷疑、Backus 為什麼做它（程式設計成本已超過機器成本）；這一跳定調全書反覆出現的「效能↔生產力」鐘擺。
- **不涵蓋**：編譯器原理（lexer/parser/codegen 細節——指向延伸閱讀）；語言設計哲學（ch04）；型別（ch08）。
- **橋接**：你寫 TypeScript 由 tsc 編譯成 JS、再被 V8 JIT——你每天信任「機器產生的碼夠快」，這份信任始於 1957 的賭注；你選 NestJS（生產力）而非手寫 C（效能）就是在這條鐘擺上選邊。
- **個案**：FORTRAN 編譯器團隊為了讓產出碼夠快下的苦工（最佳化是 FORTRAN 被接受的關鍵）——「為了贏得『生產力不必犧牲效能』這場賭，他們做了什麼」。
- **紙上推演**：把「組語→高階語言」這一跳用你熟的「手寫 SQL→ORM」類比並指出類比在哪裡會破；口頭講 FORTRAN 的賭注與你選框架的取捨是同一條鐘擺；查證題：說清楚為什麼 Hopper 的 A-0「是不是第一個編譯器」取決於你怎麼定義編譯器。

## Part II — 語言的哲學與第一次危機（ch04–06）

### ch04 — Lisp 與 Algol：語言開始有哲學
- **目標**：能講出 Lisp（程式即資料）與 Algol（結構化、語言的語言）兩條血脈各自的世界觀，以及它們為什麼至今還在影響你用的語言；能說出 GC、遞迴、BNF 這些「現在理所當然」的東西當年是發明。
- **必涵蓋**：**LISP（McCarthy, 1958–60）**：程式即資料（homoiconicity）、遞迴、garbage collection 的誕生、AI 實驗室脈絡、eval 被 Steve Russell 手工編成機器碼的故事（照 landscape）；**ALGOL 60**：block structure、BNF（Backus-Naur Form）、「一個遠超時代的語言」（Hoare 名言照 landscape 查證）、它如何成為後世語言（C/Pascal/Java…）的共同祖先；**COBOL（1959–60）**：另一極——為商業可讀性設計、Hopper 的影響、為什麼它活了 60 年（不是因為好，是因為遷移成本＝複雜度搬不動）；三條路線（Lisp 的數學/AI、Algol 的結構、COBOL 的商業）定義了語言設計的價值光譜。
- **不涵蓋**：函數式編程的完整故事（ch09）；OOP（ch07）；型別系統（ch08）；語言語法教學。
- **橋接**：你的 JS 是「Algol 後裔（C 系語法）裝了 Lisp 的靈魂（first-class function、closure、GC）」——你天天在用兩條血脈的混血；你信賴 GC 不手動管記憶體，這個奢侈始於 Lisp。
- **個案**：garbage collection 的誕生——McCarthy 為了讓 Lisp 能自由配置 cons cell 而發明自動記憶體回收，把「記得 free」這個複雜度永久搬給 runtime（帳單：GC pause、你調過的 Node.js 堆積記憶體）。
- **紙上推演**：把「程式即資料」對應到你寫過的 codegen／用 AST 操作程式碼的經驗；解釋「為什麼 COBOL 還沒死」用複雜度搬不動來論證；反事實：如果 Algol 沒發明 BNF，描述語言語法會變多難。

### ch05 — 軟體危機與結構化編程：方法 strand 登場
- **目標**：能講清楚 1968 為什麼喊出「軟體危機」、它撞的是哪一面複雜度的牆（程式大到一個人腦裝不下）；能重講 Dijkstra 反 GOTO 的真正論點（不是「GOTO 邪惡」而是「人腦追不上跳來跳去的控制流」）；能說出這是方法 strand 第一次正式登場。
- **必涵蓋**：**1968 NATO 會議（Garmisch）**：software crisis 一詞、「software engineering」是刻意挑釁的命名（照 landscape）；當年的痛——大型系統（如後來的 OS/360）超支、延遲、不可靠；**Dijkstra「Go To Statement Considered Harmful」（1968）**：標題是編輯 Wirth 下的（照 landscape）、真正論點是「靜態程式文本與動態執行過程的差距」；結構化編程（Böhm-Jacopini 定理 1966：三種控制結構足夠）；Dijkstra 名言查證版（「測試只能證明 bug 存在、不能證明不存在」「簡單是可靠的前提」）；為什麼這同時是語言事件（控制結構）與方法事件（紀律）——兩條 strand 第一次明顯交織。
- **不涵蓋**：模組化（ch06）；形式方法的深入（ch08）；OOP（ch07）；具體語言語法。
- **橋接**：你做 code review 時要求別人別寫巢狀地獄、早 return、控制圈複雜度——你執行的就是結構化編程的紀律；你讀別人的 async 控制流追不上時的痛，正是 Dijkstra 說的「靜態文本 vs 動態過程」。
- **個案**：把一段 GOTO 義大利麵程式 vs 結構化版本攤開（ASCII 控制流圖），講清楚「為什麼後者人腦追得上」——不是美學，是人類工作記憶的容量限制。
- **紙上推演**：用「人腦工作記憶容量」解釋為什麼結構化編程是對複雜度的回應；找出「Dijkstra 說 GOTO 邪惡」這個流行說法錯在哪；對帳：你的 lint 規則（max-depth、complexity）哪幾條是結構化編程的後裔。

### ch06 — 模組化與 Unix：怎麼切一個大程式
- **目標**：能講出 Parnas「資訊隱藏」的原始洞見（模組該照「藏住什麼決策」切，不是照流程切）；能講出 Unix 哲學（小工具、組合、可移植）如何成為你今天微服務/pipeline 的祖宗；能區分「分解」的好壞標準。
- **必涵蓋**：**Parnas 1972《On the Criteria To Be Used in Decomposing Systems into Modules》**：資訊隱藏、模組邊界該包住「最可能改變的設計決策」——而不是照處理步驟切（這個洞見今天仍被違反）；separation of concerns（Dijkstra 用語）；**Unix（Thompson/Ritchie, 1969–71）**＋**C（~1972）**：可移植性（用高階語言寫 OS 是賭注）、**Unix 哲學**（McIlroy：寫只做一件事並做好的程式、讓程式協作——pipe 1973，引文照 landscape）；K&R（1978）；為什麼「小工具用 pipe 組合」是抽象與方法的雙重遺產。
- **不涵蓋**：OOP 的封裝（ch07，明寫 OOP 是資訊隱藏的另一條實作路線）；微服務（ch17，明寫是 Unix 哲學的分散式版）；作業系統內部原理。
- **橋接**：你切微服務邊界時在做 Parnas 的事——按「會一起改變的東西」分而不是按 CRUD 步驟分（切錯就是分散式單體）；你的 FluentBit+Lua pipeline 就是 Unix「小工具 pipe 組合」哲學的現代轉世。
- **個案**：Parnas 的 KWIC 索引例子——同一個程式用「照流程切」vs「照資訊隱藏切」兩種分解，當需求變時哪一種要改得少；把這個 1972 的例子對應到你切錯微服務邊界的痛。
- **紙上推演**：用資訊隱藏原則檢驗你切過的一個服務邊界（切對了嗎、藏住了哪個會變的決策）；解釋「Unix pipe」與「微服務 + message queue」在哲學上是同一件事、在複雜度上差在哪；反事實：如果 Unix 當年用組語而非 C 寫，可移植性與後續影響會怎樣。

## Part III — 抽象的大廈：物件、型別、函數（ch07–09）

### ch07 — 物件導向：把資料與行為綁在一起
- **目標**：能講出 OOP 在解什麼複雜度（大系統裡資料與操作散落各處）、它的原意（Kay 的訊息傳遞）與它後來被簡化成什麼（class/繼承）；能評價繼承為什麼從銀彈變成要小心的東西。
- **必涵蓋**：**Simula 67**（Dahl & Nygaard）：為模擬而生、class/object 概念的起源；**Smalltalk**（Kay/Ingalls/Goldberg, Xerox PARC, 1970s）：Kay 造詞「object-oriented」、**他後來說原意是訊息傳遞不是 class**（引文照 landscape）；**C++（Stroustrup, 1983 命名）**把 OOP 帶進主流效能語言、**Java（1995）**「write once run anywhere」與 JVM；OOP 解的複雜度＝封裝（資訊隱藏的另一條路，回收 ch06）＋多型；帳單：繼承的脆弱（fragile base class）、「組合優於繼承」的反省、企業級 OOP 的過度設計（AbstractFactoryFactory 之譏）。
- **不涵蓋**：設計模式目錄（一句話＋延伸閱讀 GoF）；型別系統（ch08）；FP 對 OOP 的批評（ch09）；語法教學。
- **橋接**：你寫 NestJS 的 service/controller/module、用 DI——這是 OOP 企業化的直系後代；你選「組合優於繼承」、用 interface 而非深繼承樹，是在還 OOP 早期的複雜度帳單。
- **個案**：Alan Kay 的訊息傳遞原意 vs 業界落地的 class 中心 OOP——把「Kay 想要的（像細胞間傳訊的生物系統）」與「我們得到的（Java 的 class 階層）」攤開，講清楚一個好點子在規模化落地時如何被複雜度改形。
- **紙上推演**：用「訊息傳遞」原意解釋為什麼 actor model／微服務比 class 繼承更接近 Kay 的願景；舉一個你看過/寫過的繼承濫用並說明組合會怎麼修；解釋「封裝」與 ch06 資訊隱藏的血緣。

### ch08 — 型別與形式方法：用證明對抗 bug
- **目標**：能講出「靜態型別」與「形式驗證」是同一種衝動（在執行前就排除一類錯誤）的不同強度；能擺正型別之爭（靜態 vs 動態）的真實 trade-off；能講出形式方法的理想為何沒成為主流、又在哪裡活著。
- **必涵蓋**：**Hoare logic（1969）**：用前後條件「證明」程式正確的理想；**Hoare 的「billion dollar mistake」**（null reference，ALGOL W 1965，引文照 landscape）；型別系統的演化：從 Algol 的型別到 **ML 與 Hindley-Milner 型別推論**（Milner, 1970s）——「型別檢查器是最普及的形式方法」；靜態 vs 動態之爭的真實 trade-off（早抓錯 vs 開發速度，回收 ch03 效能↔生產力鐘擺的變體）；形式驗證為什麼留在高成本領域（航太、晶片、協定）而非主流——成本帳；**一句話接書架上的 TLA+ 書**（不依賴）。
- **不涵蓋**：型別理論細節（System F、dependent types 指向延伸閱讀）；TLA+ 教學（書架另有書）；測試方法（ch12 的 TDD）。
- **橋接**：你從 JS 轉 TypeScript 就是在這條歷史上選邊——用編譯期型別換執行期驚嚇；你寫的 `strictNullChecks` 正是在補 Hoare 說的十億美元錯誤；你用 OpenAPI schema 當「輕量形式規格」強制 CI 同步。
- **個案**：null 的十億美元錯誤——Hoare 1965 為了「方便」加入 null reference，四十年後在台上道歉；把「一個圖方便的設計決策如何變成全球性的複雜度負擔」攤開，對應你被 `undefined is not a function` 坑過的每一次。
- **紙上推演**：用「複雜度搬家」解釋 TypeScript 把什麼負擔從執行期搬到開發期（帳單是什麼）；解釋為什麼形式驗證沒有取代測試（成本/本質複雜度，預告 ch10 沒有銀彈）；對帳：你的 OpenAPI/型別 schema 算不算一種形式方法。

### ch09 — 函數式的回歸：被冷落四十年的答案
- **目標**：能講出 FP 從 Lisp/lambda calculus 出發、被主流冷落、又在 2010 年代回歸的因果（多核心讓不可變變值錢）；能講出純函數/不可變如何治並行的複雜度；能把它接回你的冪等與最終一致性直覺。
- **必涵蓋**：**lambda calculus（Church, 1930s）**作為計算的另一個理論根；FP 的潛流（Lisp→ML→**Haskell, 1990**，惰性純函數）；為什麼主流冷落它四十年（硬體、心智模型、效能）；**回歸的觸發：多核心**——「免費的午餐結束了」（Herb Sutter, 2005，照 landscape）、時脈不再自動變快、平行化變成必修，而**不可變資料天生對並行友善**（沒有共享可變狀態就沒有 race condition）；map/reduce、純函數、不可變如何進入主流語言（你的 JS 的 `map/filter/reduce`、React 的不可變狀態、Rust 的 ownership 借鏡）。
- **不涵蓋**：monad/category theory 教學（一句話＋延伸閱讀）；Haskell 語法；並行理論細節（讀者書架上有 queueing 書談排隊、TLA+ 書談並行正確性——可一句話接，不依賴）。
- **橋接**：你靠冪等消費者、無共享狀態、最終一致性處理 SQS 訊息——你已經在用 FP 的核心直覺（不可變 + 純粹 = 可重試、可平行）；你的 SQS 結算 pipeline 之所以能安全重試，正因為它逼近純函數。
- **個案**：「免費的午餐結束了」（2005）——時脈撞牆、核心數開始長，於是「如何安全地用多核」變成全產業問題，而四十年前 Lisp 就給的答案（不可變）突然變值錢；把這個「老答案等到問題出現才被需要」的模式攤開（呼應 ch21 鐘擺）。
- **紙上推演**：用「無共享可變狀態」解釋為什麼你的冪等消費者可以安全平行重試；解釋「為什麼 FP 等到 2005 才紅」用硬體鐘擺論證；對帳：你的程式裡哪些地方其實是純函數、哪些是被可變狀態汙染的麻煩來源。

## Part IV — 流程的戰爭：方法 strand 的高潮（ch10–12）

### ch10 — 人月神話：複雜度的人性面
- **目標**：能重講 Brooks 的核心洞見（軟體的本質複雜度無法靠加人/加工具消除）；能講清楚 Brooks's Law、概念完整性、本質 vs 偶然複雜度（沒有銀彈）這三件全書級的分析工具；能講 Conway's Law 並預告它在微服務的回收。
- **必涵蓋**：**Brooks《人月神話》（1975）**＋ OS/360 的脈絡；**Brooks's Law**（加人到遲到的專案讓它更遲到——溝通成本 O(n²)、ramp-up 時間）；**概念完整性**（一致的設計願景比功能堆疊重要）；**《No Silver Bullet》（1986）**：**本質複雜度 vs 偶然複雜度**——工具只能砍偶然複雜度、本質複雜度砍不掉（這是全書反覆用的手術刀，回收到 ch08 形式方法、ch19 抽象洩漏、ch20 AI）；**Conway's Law（1968 Datamation 刊出；1967 投稿被退）**：系統結構＝組織溝通結構（照 landscape）；為什麼這些 1960–80 年代的話今天還準——它們講的是人與複雜度，不是技術。
- **不涵蓋**：瀑布（ch11）；敏捷（ch12）；具體估算方法；Conway 在微服務的展開（ch17）。
- **橋接**：你帶過 greenfield 專案、訂過團隊慣例——你親身驗過「加人不是線性加速」「沒有一致願景的 codebase 會分裂」；你的微服務邊界往往照團隊邊界長（Conway），切錯就是組織問題偽裝成技術問題。
- **個案**：OS/360 的延遲與 Brooks 的反省——把「為什麼加更多程式設計師反而更慢」用溝通路徑數 O(n²) 與新人 ramp-up 攤開算一遍（具體數字示意），講清楚這是數學不是管理失誤。
- **紙上推演**：用本質 vs 偶然複雜度分類你最近一個專案的難點（哪些是工具能救的、哪些不能）；用 O(n²) 溝通成本解釋一次你經歷的「加人沒加速」；預測：你切過的某個服務邊界是不是 Conway's Law 的產物。

### ch11 — 瀑布：一個被讀反的模型
- **目標**：能講出瀑布的歷史真相（Royce 1970 其實畫它是為了說「這樣不行、要迭代」）；能講出重量級流程（瀑布/CMM）在解什麼複雜度（協調大團隊、可稽核）、它的帳單是什麼；能公平評價它（不是純粹的笑話）。
- **必涵蓋**：**Royce 1970《Managing the Development of Large Software Systems》**：他畫出純瀑布圖是當作**有缺陷的起點**、主張迭代與原型（照 landscape，這是全書最重要的「大家讀反了」案例）；「waterfall」一詞與純瀑布的刻板印象是後人貼的（Bell & Thayer 1976 等）；重量級流程為什麼興起——大型國防/企業專案需要可預測、可稽核、可分工（**CMM/CMMI**, Humphrey, SEI）；它的帳單：對需求不確定性脆弱、回饋太慢、文件成為目的本身；公平話：在需求真的穩定、變更成本真的高（航太、醫療）的領域，重流程不是錯。
- **不涵蓋**：敏捷（ch12，明寫是對重量級的反叛）；具體 CMM 等級細節；專案管理工具。
- **橋接**：你做 greenfield 時也得先定 API 契約（OpenAPI）再並行開發——這是「先設計後實作」的瀑布基因的合理殘留；你在強監管 B2B 場景感受過「為什麼有些地方真的需要文件與簽核」。
- **個案**：Royce 論文的真實內容——逐段攤開他畫純瀑布圖之後緊接著說「這行不通，所以要…」，對比後世只截了那張圖當聖經；這是「一個想法被斷章取義五十年」的活標本。
- **紙上推演**：用 Royce 的原意解釋「瀑布 vs 敏捷」這個對立其實有多大成分是稻草人；找出你工作中哪些環節「瀑布式（先定契約）」其實是對的、為什麼；反事實：如果需求永遠不變，敏捷還有優勢嗎。

### ch12 — 敏捷：對重量級流程的反叛
- **目標**：能講出敏捷在反叛什麼（重量級流程對變化的脆弱）、它的核心賭注（用短迴圈擁抱變化而非預測變化）；能擺正敏捷的理想 vs 它被企業收編後的變形；能把 TDD/CI/重構接回你的日常。
- **必涵蓋**：**Agile Manifesto（2001, Snowbird）**：四個價值、十七人脈絡；前史——**XP（Kent Beck, 1996–99, Chrysler C3）**、**Scrum（Schwaber/Sutherland, ~1995；根在 Takeuchi & Nonaka 1986）**；配套實踐：**TDD（Beck）**、**重構（Fowler 1999）**、**持續整合（Booch 造詞、XP 推廣）**——它們如何把「擁抱變化」變成可操作的紀律；敏捷的複雜度策略＝把「需求不確定性」這個本質複雜度用短回饋迴圈馴服（不是消滅）；帳單：敏捷被企業收編成「Scrum 儀式劇場」、velocity 變成績效棒子、與原始宣言的背離。
- **不涵蓋**：DevOps（ch18，明寫是敏捷往維運的延伸）；具體 Scrum 儀式 how-to；專案管理工具。
- **橋接**：你做 CI 強制 OpenAPI 同步、寫測試、重構——你執行的是敏捷技術實踐的直系後代；你大概也經歷過「敏捷變成每天站會報進度」的儀式化走味。
- **個案**：XP 的 C3 專案——把「為什麼一群人在 1990 年代末決定反叛重流程、他們具體改了什麼（短迭代、結對、測試先行）」攤開，講清楚敏捷不是「不寫文件」而是「把賭注從預測改成適應」。
- **紙上推演**：用「擁抱變化 vs 預測變化」解釋敏捷與瀑布的根本分歧（不是文件多寡）；診斷你經歷過的一次「假敏捷」走味在哪、背離了宣言哪一條；對帳：你的 CI/CD pipeline 實現了敏捷宣言的哪些主張。

## Part V — 資料、開源與網路（ch13–15）

### ch13 — 關聯模型：把資料的複雜度搬給引擎
- **目標**：能講出 Codd 關聯模型的抽象勝利（用宣告式查詢把「怎麼存取」的複雜度搬給資料庫引擎）；能講出 SQL/ACID 的歷史與它們解的複雜度；能講 NoSQL 的反動是在還哪一筆帳（規模 vs 一致性，CAP）。
- **必涵蓋**：前關聯時代的痛（階層式/網狀資料庫，存取路徑寫死在程式裡）；**Codd 1970《A Relational Model of Data》**：關聯代數、**資料獨立性**——你說要什麼（宣告式）、引擎決定怎麼拿（查詢最佳化器）；System R / Ingres、**SQL（SEQUEL, Chamberlin & Boyce, 1974）**；**ACID（一詞 1983，概念源於 Jim Gray）**把交易的複雜度標準化；**NoSQL 反動（~2009；Google Bigtable 2006、Amazon Dynamo 2007）**＋**CAP（Brewer 2000 猜想、Gilbert-Lynch 2002 證明）**：規模逼迫放棄強一致——鐘擺從「引擎包辦一切」擺回「應用自己處理一致性」。
- **不涵蓋**：SQL 語法教學；查詢最佳化器內部（指向延伸閱讀）；分散式一致性協定細節（讀者書架上 TLA+/queueing 書可一句話接）；NewSQL 深入（一句話）。
- **橋接**：你天天寫 SQL、調 index、處理 isolation 與 deadlock、用 read replica 與 partition——你站在 Codd 的肩膀上享受「資料獨立性」這個奢侈；你選 Redis/NoSQL 處理某些負載就是在 CAP 上選邊、自己扛一致性。
- **個案**：宣告式的勝利——同一個查詢，前關聯時代要在程式裡手寫存取路徑與走訪迴圈，關聯時代只要一句 `SELECT … WHERE … JOIN`，引擎的最佳化器決定其餘；把「Codd 把什麼複雜度搬給引擎、帳單在哪（N+1、慢查詢、最佳化器猜錯）」攤開，回收 ch01 的 ORM 縮影。
- **紙上推演**：用「資料獨立性」解釋為什麼你能換 index 而不改應用程式碼（複雜度搬給了誰）；用 CAP 解釋你某次選 NoSQL/Redis 的取捨（放棄了什麼換了什麼）；對帳：N+1 query 是哪一層抽象的洩漏（接 ch19）。

### ch14 — 開源與 Git：重寫所有權與協作
- **目標**：能講出自由軟體/開源如何把軟體的「所有權與協作模式」重寫（從私有到公共財）；能講 Git 的分散式版控如何解協作的複雜度；能擺正「自由軟體 vs 開源」的理念差異。
- **必涵蓋**：**GNU（Stallman, 1983）＋GPL＋FSF（1985）**：「free as in freedom」、copyleft 的法律駭客；**Linux（Torvalds, 1991）**作為開源協作的證明；**《The Cathedral and the Bazaar》（Raymond, 1997）**：大教堂式 vs 市集式開發；**「open source」一詞（1998, OSI）**與它和「自由軟體」的理念分歧（務實 vs 理想）；版本控制的演化：CVS→SVN（集中式）→**Git（Torvalds, 2005, BitKeeper 爭議觸發、極短時間內成形——數天內自我託管、數週內接手 kernel；勿寫死「三週」）**：分散式、便宜分支、merge——把「多人協作的複雜度」用內容定址與 DAG 重新組織；GitHub 如何把開源變成社交。
- **不涵蓋**：Git 內部物件模型細節（指向延伸閱讀）；授權條款法律細節；具體 Git 指令教學（讀者已精通）。
- **橋接**：你每天 commit/branch/merge、用個人 GitHub 帳號累積足跡、靠開源 vLLM/llm-d 生態——你活在 Stallman 與 Torvalds 重寫的世界裡；你的 PR-based 協作就是市集模式的日常。
- **個案**：Git 的誕生——BitKeeper 撤回免費授權後，Torvalds 在極短時間內（一手 commit 紀錄：數天內自我託管、數週內接手 kernel）為了 Linux 核心的協作規模寫出 Git，核心設計（內容定址、便宜分支）如何直接回應「上千人併行改一個 codebase」的複雜度；把「協作複雜度如何逼出一個資料結構」攤開。
- **紙上推演**：用「複雜度搬家」解釋 Git 把什麼從集中式伺服器搬到每個人的本機（帳單是什麼）；解釋「自由軟體」與「開源」的理念差異為什麼不只是用詞；反事實：如果沒有便宜的分支與 merge，PR 工作流還可能存在嗎。

### ch15 — 網路與 Web：軟體從盒子搬到伺服器
- **目標**：能講出軟體交付形態的根本轉變（從「裝在使用者機器」到「跑在我們的伺服器」）如何改變一切；能講 Web 的開放標準如何成為史上最大的應用平台；能把這條線接到你的 B2B/B2C 服務。
- **必涵蓋**：底層：ARPANET（1969）、**TCP/IP（Cerf & Kahn, 1974 論文、1983 cutover）**——封包交換把網路的複雜度標準化；**WWW（Berners-Lee, CERN, 1989 提案／1991 公開）**：HTTP/HTML/URL 三件套、開放標準的決定性（沒有專利）；瀏覽器戰爭（Netscape vs IE）與標準的代價；**LAMP** 與動態網站；**Web 2.0（O'Reilly, ~2004）＋AJAX（一詞 2005）**：瀏覽器從文件檢視器變成應用平台；交付形態轉變的後果——更新不再靠寄光碟、複雜度從用戶端搬到伺服器端（你要為所有人維運）。
- **不涵蓋**：HTTP 協定細節（指向延伸閱讀）；前端框架史（一句話）；雲（ch16，明寫是「伺服器搬到別人機房」的下一跳）。
- **橋接**：你做的 B2B/B2C 資料中介、REST API、WebSocket 即時系統——全都站在「軟體跑在伺服器、用戶端只是窗口」這個轉變上；你維運的痛（為所有用戶的可用性負責）正是這次複雜度搬家的帳單。
- **個案**：開放標準的決定性——Web 贏過當年各種專有的線上服務（如 AOL 圍牆花園），關鍵是 HTTP/HTML 無專利、任何人可實作；把「為什麼開放標準擊敗封閉平台」與「複雜度/控制權的分配」攤開。
- **紙上推演**：用「複雜度搬到伺服器端」解釋為什麼 SaaS 能持續更新而盒裝軟體不能（帳單：你要 24/7 維運）；解釋開放標準為何擊敗圍牆花園（接 ch14 開源的所有權主題）；對帳：你的即時系統（WebSocket）解的是哪一類「網路的複雜度」。

## Part VI — 雲與分散式：複雜度再次搬家（ch16–18）

### ch16 — 從伺服器到雲：基礎設施變成程式碼
- **目標**：能講出雲的本質轉變（把機房的複雜度搬給別人、按用量計費）；能講 IaaS/PaaS/SaaS 的分層各搬走哪一層複雜度；能講 IaC 如何把基礎設施也納入軟體工程。
- **必涵蓋**：虛擬化前史（IBM CP/CMS 1960s、VMware 1998、Xen 2003）——一台機器假裝成多台；**AWS（S3/EC2, 2006）**：把資本支出變營運支出、把「買機器、進機房、接網路」的複雜度整包外包；**IaaS/PaaS/SaaS 分層**（各層把「你不用管什麼」往上推）；**Infrastructure as Code**（CFEngine 1993→Puppet 2005→Chef 2009→**Terraform 2014**）：基礎設施從手動點擊變成版本控制下的程式碼——可重現、可審查、可回溯；雲的帳單：抽象洩漏（網路分區、冷啟動、供應商鎖定）、成本失控、「別人的電腦」的信任問題。
- **不涵蓋**：特定雲服務 how-to（讀者已精通 ECS/Lambda/RDS）；容器（ch17）；微服務（ch17）；K8s（ch17）。
- **橋接**：你用 CloudFormation 定義基礎設施、用 ECS/Lambda/RDS/SQS——你完全活在「基礎設施即程式碼、機房是別人的」這個世界；你調過的冷啟動、踩過的供應商相依，就是雲抽象的洩漏帳單。
- **個案**：AWS 2006 的轉折——亞馬遜把自己的內部基礎設施對外開放成 API（S3/EC2），把「擁有並維運機房」這個巨大複雜度變成一行 API 呼叫；把「資本支出→營運支出、所有權→租用」的轉變與它的帳單（鎖定、信任、成本）攤開。
- **紙上推演**：用 IaaS/PaaS/SaaS 分層說明你的某個系統「把哪幾層複雜度外包、自己留哪幾層」；用「複雜度搬家」解釋 serverless（Lambda）把什麼搬走、帳單是什麼（冷啟動、可觀測性變難）；對帳：你的 CloudFormation 算不算 ch11「先設計」基因的回歸。

### ch17 — 微服務與容器：把單體拆散再封裝
- **目標**：能講出微服務在解什麼複雜度（大團隊在單體上互相踩腳）、它把複雜度搬去哪（從程式內搬到網路上）；能講容器/K8s 如何成為這個分散式世界的作業系統；能用 Conway's Law 解釋微服務的組織本質。
- **必涵蓋**：SOA 的前身與教訓；**微服務（一詞 ~2011–14；Lewis & Fowler 2014；Netflix 早期實踐）**：按業務能力切、獨立部署、團隊自治——解的是「大團隊在單體上協調」的複雜度；**容器：Docker（2013, Hykes）**＋cgroups/LXC 血脈（Google cgroups 2007）——把「在我機器上能跑」的複雜度封進 image（回收 ch02 code as data）；**Kubernetes（Google, 2014, 血脈 Borg）＋CNCF（2015）**：分散式系統的作業系統、宣告式調度（回收 ch13 宣告式）；**Conway's Law 回收（ch10）**：微服務本質是把組織結構固化成系統結構；帳單：複雜度從「程式內的函式呼叫」搬到「網路上的分散式系統」——你現在要面對網路分區、分散式交易、可觀測性、最終一致性（這正是讀者 AI Infra 之路的核心，一句話接 ch18 與書架 AI Infra 書）。
- **不涵蓋**：K8s 操作 how-to（讀者學習中、書架另有 AI Infra 書）；service mesh 細節（一句話）；分散式一致性協定（書架 TLA+ 書）；DevOps/SRE（ch18）。
- **橋接**：你做 AWS ECS 上的 TypeScript/NestJS 微服務、處理服務間的最終一致性與冪等——你天天在付微服務把複雜度搬上網路的帳單；你切的服務邊界往往照團隊邊界（Conway 活生生）。
- **個案**：「在我機器上能跑」問題的終結——Docker 如何把執行環境連同程式打包成不可變 image（呼應 ch02 程式即資料），把環境差異的複雜度一次搬走；再攤開它製造的新複雜度（image 體積、編排、網路），講清楚這是搬家不是消滅。
- **紙上推演**：用「複雜度從程式內搬到網路上」列出微服務讓你多付的三筆帳（網路、一致性、可觀測性）；用 Conway's Law 解釋你某次服務拆分的真正驅動力是技術還是組織；解釋容器 image 與 ch02 儲存程式的血緣。

### ch18 — DevOps 與 SRE：把維運變成工程
- **目標**：能講出 DevOps 在彌合什麼裂縫（開發丟給維運的牆）、SRE 如何把可靠度變成可量化工程（SLO/error budget）；能講平台工程為何作為對微服務複雜度的回應而興起；能把這條線接到你的可觀測性實戰與 AI Infra 之路。
- **必涵蓋**：開發 vs 維運的傳統對立（「丟過牆」）；**DevOps（一詞 ~2009, Debois, DevOpsDays Ghent）**＋**持續交付（Humble & Farley, 2010）**＋《The Phoenix Project》（2013）：把開發與維運融合、自動化部署流水線（敏捷往維運的延伸，回收 ch12）；**Google SRE（Treynor, ~2003；SRE book 2016）**：用軟體工程方法做維運、**SLO/SLI/error budget**——把「多可靠才夠」變成可談判的數字（可靠度的本質複雜度用工程紀律馴服，回收 ch10）；可觀測性（metrics/logs/traces）的興起；**平台工程（~2020s, 內部開發者平台, Team Topologies 2019）**作為對「每個團隊都要懂全套微服務維運」這個複雜度爆炸的回應——把維運複雜度再搬到一個平台抽象後面（鐘擺：又一次集中化）。
- **不涵蓋**：特定監控工具 how-to（讀者精通 CloudWatch/FluentBit）；K8s 細節（ch17）；具體 SLO 計算（一句話＋指向 SRE book）。
- **橋接**：你自建 FluentBit+Lua 可觀測性 pipeline（供 Nexon 用）、設計 metric、做 event-driven 結算——你已經是在實踐 SRE 的可觀測性與自動化；你的 AI Infra 目標（讓大模型跑得穩、快、省）本質就是 SRE+平台工程在 LLM serving 上的應用（一句話接書架 AI Infra 書）。
- **個案**：error budget 的發明——Google SRE 如何用「允許的不可靠額度」化解開發（要快上線）與維運（要穩）的永恆衝突，把一場立場之爭變成一個共享的數字；把「可靠度如何從信仰變成可談判的工程量」攤開。
- **紙上推演**：用 error budget 重新框架你經歷過的一次「上線 vs 穩定」衝突；解釋平台工程是 ch16 雲抽象之上的「再一層抽象」、它把什麼複雜度搬走（帳單是什麼）；對帳：你的 FluentBit pipeline 對應 SRE 三本柱（metrics/logs/traces）的哪一根。

## Part VII — 當下與回望（ch19–21）

### ch19 — 抽象會洩漏：鐘擺擺回來
- **目標**：能講出「所有重大抽象都會洩漏」這條定律（Joel 2002）與它為什麼是複雜度守恆的必然；能用全書累積的帳單盤點鐘擺的每一次回擺；能講出「為什麼我們週期性地回到底層」。
- **必涵蓋**：**Joel Spolsky《The Law of Leaky Abstractions》（2002）**：TCP/SQL/ORM/網路檔案——抽象在邊緣總會漏出底層細節（回收 ch13 N+1、ch16 雲的網路分區、ch08 型別）；**本質 vs 偶然複雜度回收（ch10）**：抽象只能藏偶然複雜度，本質複雜度終究漏回來；**鐘擺總盤點**——效能↔生產力（ch03/ch08）、集中↔分散（ch13 NoSQL、ch16 雲、ch18 平台工程）、單體↔微服務↔（近年「回到單體」的反思）、手工↔自動；為什麼資深工程師要懂底層——抽象洩漏時你得能跳下去；幾句被反覆驗證的老話（「過早最佳化是萬惡之源」Knuth 的完整脈絡照 landscape、「cache 失效與命名是兩大難題」、「worse is better」Gabriel）。
- **不涵蓋**：新的歷史事件（本章是回收與綜合）；AI（ch20）；最終總結（ch21）。
- **橋接**：你被 ORM 的 N+1、雲的冷啟動、TCP 的重傳坑過——每一次都是抽象洩漏；你之所以值錢正因為抽象漏的時候你能跳到底層（你的 profiling 與瓶頸診斷就是在抽象的裂縫裡工作）。
- **個案**：把全書的「複雜度帳單」彙整成一張大表（每章一筆：搬走了什麼、新增了什麼、在哪洩漏），讓讀者親眼看見「守恆」不是比喻而是反覆出現的事實；挑三筆讀者親身踩過的帳單深入。
- **紙上推演**：用「抽象洩漏」解釋你最近一次必須跳到底層 debug 的經歷（哪層漏了）；論證「過早最佳化是萬惡之源」這句話被濫用成什麼、Knuth 原意是什麼；盤點你的技術棧裡三個鐘擺，各自現在擺到哪一邊。

### ch20 — AI 輔助編程：第幾次「自動編程」的夢
- **目標**：能把 AI 輔助編程放進歷史座標（FORTRAN「自動編程」→4GL→低程式碼→Copilot 是同一個夢的反覆）；能用本質 vs 偶然複雜度判斷它能砍什麼、砍不掉什麼；能對「它是不是又一次抽象躍遷」給出有歷史縱深的、誠實 hedge 的判斷（2026-06）。
- **必涵蓋**：「自動編程」夢的歷史（回收 ch03 FORTRAN 當年也叫 automatic programming、4GL、CASE 工具、低程式碼/無程式碼的反覆承諾與反覆落空）；**程式碼補全→GitHub Copilot（預覽 2021／GA 2022，基於 OpenAI Codex）→agentic 編程工具（2023–2026，照 landscape，明確標時點與 hedge）**；用 Brooks 的手術刀分析：AI 砍的多是**偶然複雜度**（樣板、查 API、翻譯語言）、**本質複雜度（搞清楚到底要建什麼、跨系統的一致性）砍不掉**——所以「沒有銀彈」在 2026 仍是合理的預設懷疑；歷史視角的兩面判斷：這次和以往「自動編程」承諾有什麼**真的不同**（自然語言介面、規模）、又有什麼**似曾相識**（高估、把偶然當本質）；誠實聲明：這是還在演化的當下、本章的判斷可能被未來推翻（與「已成定局的歷史」嚴格區分）。
- **不涵蓋**：LLM/Transformer 內部機制（讀者書架上 AI Infra 書；本章只談「AI 作為寫程式的工具」）；特定產品功能比較；對未來的預言（明寫不預言）。

- **個案**：把「automatic programming」這個詞從 1957（FORTRAN）到 2026（AI agent）的反覆出現攤開成一條線，每一代都宣稱「讓寫程式更接近說人話」——講清楚哪些承諾兌現了（抽象確實升高了）、哪些是每代都犯的高估（以為能消滅本質複雜度）。
