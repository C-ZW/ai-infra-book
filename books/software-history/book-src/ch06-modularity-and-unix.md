# ch06 — 模組化與 Unix：怎麼切一個大程式

> **本章解決什麼問題**：ch05 的結構化編程治的是「一個函式內部」的複雜度——控制流亂跳，人腦追不上。但真正的大系統不是一個大函式，而是幾十、幾百個模組要拼在一起。切錯了，改一個需求要動十個檔；切對了，只動一個。這章撞的是**分解的牆**：一個大程式該照什麼準則切成模組？Parnas 1972 給的答案（照「會變的決策」切，不照流程切）至今仍被天天違反——你切出分散式單體那次就是。同一個洞見，Unix 用另一條路走給你看：把它做成「小工具＋pipe」的文化。這章是 ch07（OOP 封裝）、ch17（微服務）的共同祖先。

## 從你已知的出發

你切過微服務邊界，所以你知道這件事有多容易出錯。

回想你做那套 AWS ECS 上的 B2B/B2C 資料中介。最初的草圖大概是這樣切的：一個 service 管「接收訂單」、一個管「驗證」、一個管「計價」、一個管「寫帳」、一個管「通知」——照著一筆訂單**走過的流程**一刀一刀切下去。看起來乾淨，每個 service 一個動詞，畫成圖也漂亮。

然後需求變了。客戶要求計價邏輯要看「訂單來源通路」，而通路這個欄位是「接收訂單」那個 service 定義的。於是你得改接收、改計價、還要改它們中間那段共享的 DTO，三個 service 一起發版，CI 卡了一下午。再過一個月，又一個跨越流程步驟的需求進來，又是三四個 service 連動。你開始有種不對勁的感覺：這些「獨立」的 service 根本不獨立——它們得一起改、一起發、一起回滾。你拆出來的不是微服務，是一個被網路切開、但耦合一點沒少的**分散式單體**（distributed monolith）。最糟的版本：耦合還在，但你現在得多付網路延遲、序列化、分散式交易、跨服務 trace 的帳。

這個痛不是你發明的。1972 年，一個叫 David Parnas 的人寫了一篇論文，把「為什麼照流程切會痛、該照什麼切才不痛」講得清清楚楚。那篇論文我認為是軟體史上最被低估、也最常被違反的一篇——它的標題甚至沒提到後來最常用它的詞（「模組化」「資訊隱藏」業界都記得，論文本身很少人讀過）。這章先把 Parnas 的洞見攤開，再看 Unix 怎麼用一條完全不同的路——文化而非論文——把同一件事做成你今天的 pipeline。

## Parnas 1972：模組不是「流程的一段」，是「一個祕密」

先把場景放回 1972。那時「模組化」這個詞已經有人在喊了——大家都同意大程式要拆成小塊，理由很素樸：分工（一塊一個人寫）、可讀（一次只看一塊）、可測（一塊一塊測）。問題是**照什麼切**。當時的標準答案幾乎是條件反射：照流程切。程式要做的事是一連串步驟，那就一個步驟一個模組，資料像輸送帶一樣從第一個模組流到最後一個。這跟畫流程圖（flowchart）的直覺完全一致，自然到沒人覺得需要質疑。

Parnas 質疑了。他在 *CACM* 1972 年 12 月號發表〈On the Criteria To Be Used in Decomposing Systems into Modules〉，標題直譯就是「該用什麼準則來把系統分解成模組」——他攻擊的正是「準則」這件事：大家都在切，但沒人說得清切的準則是什麼，於是默認用了最壞的那個。

他的論證方式很工程師：不空談，給你看一個具體的小程式，用兩種切法各切一次，然後讓需求變一變，數哪種改得少。

### 個案：KWIC 索引的兩種切法

他選的例子是 **KWIC 索引**（Key Word in Context，關鍵詞索引）。功能很單純：給你一堆句子（每句是一串字），程式要對每句做「循環移位」——把第一個字搬到句尾，重複到每個字都當過開頭，產生這句的所有循環移位版本；最後把全部句子的全部循環移位**照字母排序**輸出。這是早年做書目索引的真實需求（讓每個關鍵詞都能當索引的開頭被查到）。

你不用管它的用途，只要看它怎麼被切。

**切法一：照流程切（Parnas 叫它 Modularization 1）。** 順著資料流，一刀一個步驟：

```text
切法一：照流程（會痛的那種）

  輸入        循環移位       排序          輸出
┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐
│Input │──►│Shift │──►│ Sort │──►│Output│
└──────┘   └──────┘   └──────┘   └──────┘
   每個模組讀上一個丟出來的資料、加工、丟給下一個
   所有模組共享「句子在記憶體裡長什麼樣」這個祕密
```

這切法的問題藏在最後一行：每個模組都得知道「資料在記憶體裡實際怎麼擺」——是一個大字元陣列？每個字前面放長度？句子之間用什麼分隔？因為要直接讀別人丟出來的 buffer，這個「記憶體佈局」的決定**漏給了每一個模組**。四個模組，四份對同一個格式的依賴。

**切法二：照祕密切（Modularization 2）。** Parnas 重切一次，這次每個模組包住「一個會變的設計決策」：

```text
切法二：照「藏住什麼決策」（不會痛的那種）

  Line Storage ── 藏住：句子到底存哪、怎麼存
  Input        ── 藏住：輸入資料的格式怎麼讀
  Circular Shift ─ 藏住：循環移位是先算好存起來、還是查的時候才算
  Alphabetizer ── 藏住：用哪種排序演算法
  Output       ── 藏住：輸出格式長怎樣
  Master Control ─ 藏住：這幾個模組誰先誰後的呼叫順序
```

注意切法二的模組**不對應任何一個流程步驟**。「Line Storage」不是流程裡的一步，它是一個祕密保管箱：別的模組要存字、取字，都透過它提供的函式（「給我第 i 句的第 j 個字」），永遠不直接碰底層的記憶體。循環移位模組甚至不真的「移位」——它只提供一個介面說「第 i 句循環移位 k 次後的內容是什麼」，至於背後是事先算好全部存起來、還是每次查才現算，是它一個人的祕密。

### 為什麼切法二改得少：數一筆帳

光看圖看不出差別，得讓需求變。Parnas 列了一串「很可能要改」的決策，挑最有殺傷力的一個：**「所有句子都放在記憶體裡」這個決定如果要改成「放在磁碟上」，要動幾個模組？**

（這在 1972 年是真問題：記憶體貴且小，資料一大就得放外存。換成今天，等價問題是「資料從單機記憶體改成放 Redis／放 S3」。）

| 需求變動 | 切法一（照流程）要改 | 切法二（照祕密）要改 |
|---|---|---|
| 句子從記憶體改放磁碟 | **全部四個模組**（每個都直接讀記憶體格式） | 只改 Line Storage 一個 |
| 改排序演算法（記憶體不夠，要外部排序） | 排序模組＋可能牽動共享格式 | 只改 Alphabetizer 一個 |
| 循環移位改成「不預存、查時現算」 | 改 Shift 模組＋下游讀它的人 | 只改 Circular Shift 一個 |
| 輸入格式改變 | Input＋共享格式假設 | 只改 Input 一個 |

差別不是「切法二比較整齊」這種美學。差別是**改一個需求要碰幾個檔**。切法一裡，「記憶體佈局」這個決定洩漏到每個模組，所以它一變，全部跟著變——這正是你那次三個 service 一起發版的 1972 年版本。切法二把每個會變的決定關進一個模組，需求變了，改動被**關在一個盒子裡**，不外溢。

Parnas 給這條準則一個名字：**資訊隱藏**（information hiding）。一個模組的價值不在於它「做了流程的哪一步」，而在於它**藏住了哪一個別人不該知道、而且很可能會變的決定**。模組的介面是它願意公開的承諾（「我提供取字、存字」），介面背後的實作是它的祕密（「我存在記憶體還是磁碟」）。只要承諾不變，祕密怎麼換都不影響別人。

這裡有個反直覺但關鍵的推論，值得你停一下：**好的模組邊界讓介面又小又抽象，而流程切法的介面又大又具體。** 切法一裡，模組之間傳的是「一大塊有特定格式的 buffer」——介面又寬又脆。切法二裡,模組之間傳的是「給我第 i 句第 j 個字」——介面又窄又穩。介面越抽象，藏住的祕密越多，能獨立變動的空間越大。

### 工程師的領悟：你切微服務邊界，做的就是這件事

把 1972 搬到你的 ECS。你最初照流程切（接收→驗證→計價→寫帳→通知），就是 Modularization 1。「訂單通路」這個欄位的定義、訂單在系統裡的資料形狀——這些「會變的決定」散落、洩漏在每個 service，所以它一變，三四個 service 連動。你拆的不是會變的決定，是流程的步驟。

Parnas 的準則翻譯成你的語言就一句話：**照「會一起改變的東西」切，不照「一筆資料流過的步驟」切。** 一個服務邊界該包住一個業務能力（business capability）裡那些**會一起變的決策**——通路規則變，最好只動一個服務。如果你發現某類需求一進來就要動三個服務、它們得一起發版，那不是「服務間正常協作」，那是 Parnas 在 1972 就警告過的：你照流程切了，把會變的決定洩漏到了每個服務。分散式單體的技術定義，就是「切法一被部署到了網路上」。

（這條準則 ch17 會以微服務的完整形態回收；ch07 會讓你看到 OOP 的「封裝」是實作資訊隱藏的**另一條路**——把祕密藏進物件而非藏進模組檔案。資訊隱藏是洞見，封裝與模組是兩種落地。）

順帶澄清一個常見混淆：資訊隱藏不等於「關注點分離」（separation of concerns）。後者這個詞是 Dijkstra 在 1974 年的 EWD447〈On the Role of Scientific Thought〉裡造的，講的是「一次只專注一個面向去想」的思考紀律，範圍比資訊隱藏更廣。資訊隱藏是關注點分離落到「模組邊界」上的一個具體準則。兩個都對，但別把它們當同義詞——一個是思考方法，一個是切割準則。

## Unix：同一個洞見，用文化而不是論文走

Parnas 用一篇嚴謹的論文論證「該怎麼切」。差不多同一時期，在 Bell Labs 的另一群人，用一個能跑的系統把同一件事做成了文化——而且他們多半不是讀了 Parnas 才這麼做的，是被現實逼出來的。這群人是 Ken Thompson、Dennis Ritchie，以及把哲學講得最精的 Doug McIlroy。

### 先看清楚他們撞的牆

1969 年 Thompson 在 Bell Labs 一台沒人要的舊 PDP-7 上開始寫 Unix（起點之一是他想玩一個叫 Space Travel 的程式）。當年寫作業系統的常識是：作業系統是效能最敏感的東西，必須用組合語言（assembly）一行一行手寫，貼著特定機器的指令集。代價你應該猜得到——這個 OS 跟這台機器**焊死**了。換一台硬體，整套重寫。1960 年代的軟體就是這樣一座座孤島，每座島綁一款機器。

Thompson 和 Ritchie 下了一個在當年看來很冒險的賭注：**用高階語言寫作業系統。** Ritchie 為此把一個叫 B 的語言加上型別與結構，做成了 **C**（約 1972）。1973 年，他們用 C 把 Unix 核心的大部分重寫了一遍。

這個賭注的賭資是效能（大家都怕編譯器產出的碼跑不過手寫組語——這正是 ch03 FORTRAN 那場「效能 vs 生產力」鐘擺的同一場賭，只是賭注更大：這次賭的是作業系統本身），贏面是**可移植性**（portability）。賭贏了：因為核心是 C 寫的，要搬到新硬體，理論上只要重寫一小塊機器相關的部分、其餘重新編譯即可，不必整套重來。1978 年 Unix 成功移植到一台跟 PDP-11 架構完全不同的 Interdata 8/32 上——這是「作業系統能跨硬體搬家」第一次被真正證明。在那之前，「可移植的作業系統」幾乎是個矛盾詞。

把它接回 Parnas：C 對 Unix 做的，正是資訊隱藏的最大規模示範——**「我們跑在哪款 CPU 上」這個會變的決定，被藏進了 C 編譯器和一小塊機器相關代碼裡。** 上層幾乎所有程式都不知道、也不需要知道底下是 PDP-11 還是 Interdata。換硬體，動的是那個被藏起來的盒子，不是整個系統。你今天信任「同一份 TypeScript 在你的 Mac、CI 的 Linux、ECS 的容器裡跑出一樣的結果」，這份信任的祖宗就是 1973 年那次 C 重寫。

### Unix 哲學：把「小」做成紀律

可移植性是底層的勝利，但 Unix 真正傳給你的遺產在上層：它的**哲學**。這套哲學最權威的文字版，是 McIlroy、E. N. Pinson、B. A. Tague 三人 1978 年在 *Bell System Technical Journal* 那期 Unix 專號的前言裡列的幾條原則。McIlroy 寫的原句是這樣（這裡逐字引用 1978 原文）：

- 「**Make each program do one thing well.** To do a new job, build afresh rather than complicate old programs by adding new 'features'.」（讓每個程式把一件事做好；要做新工作就重新蓋一個小的，別把舊程式塞滿新功能搞複雜。）
- 「**Expect the output of every program to become the input to another, as yet unknown, program.**」（預期每個程式的輸出，會成為另一個你還不知道的程式的輸入。）

> ⚠️ 校準一個你大概也記錯的事：網路上流傳的那個整齊三句版——「Write programs that do one thing and do it well. Write programs to work together. Write programs to handle text streams.」——**不是 McIlroy 1978 的逐字原文**，而是 Peter H. Salus 在 1994 年《A Quarter-Century of Unix》裡的歸納轉述。意思對，但若要逐字引用，請用上面 1978 的原句。本書照原句。

這兩句話為什麼是「方法」而不只是「品味」？因為第二句藏著整個系統的設計賭注：**如果每個程式的輸出都能餵給下一個程式，那程式之間就需要一種通用的接合劑。** Unix 的答案是：所有程式都讀寫純文字串流（text stream），用一個叫 **pipe** 的東西把一個程式的 stdout 直接接到下一個的 stdin。pipe 這個機制是 McIlroy 鼓吹多年、由 Thompson 在 1973 年實作進 Unix 的。

於是你得到一個組合爆炸式的好處：你不需要為每個新任務寫一個大程式，你寫很多個只做一件事的小程式，再用 pipe 把它們**像積木一樣串起來**：

```text
Unix 的接合劑：pipe

  cat access.log │ grep 500 │ awk '{print $7}' │ sort │ uniq -c

  ┌─────┐   ┌──────┐   ┌─────┐   ┌──────┐   ┌────────┐
  │ cat │──►│ grep │──►│ awk │──►│ sort │──►│ uniq-c │
  └─────┘   └──────┘   └─────┘   └──────┘   └────────┘
   出檔案    濾500行    取欄位    排序       去重計數
   每個工具不知道也不在乎前後是誰；接合劑是純文字
```

關鍵在那最後一行。`grep` 不知道它前面是 `cat` 還是別的東西，也不知道後面接誰；它只認一件事——從 stdin 讀文字、把符合的行寫到 stdout。這就是資訊隱藏的另一種長相：**每個工具藏住自己怎麼做，只承諾一個窄到不能再窄的介面（讀文字、寫文字）。** 介面窄到極致，所以任意兩個工具都能接——組合的自由度來自介面的克制。

這同時是抽象事件，也是方法事件（兩條 strand 又一次交織）。抽象面：純文字串流是一層通用介面，把「資料具體長怎樣」藏在每個工具內部。方法面：「寧可寫小工具再組合，不要寫大程式」是一條**文化紀律**——它約束的不是程式碼，是工程師的衝動（別把功能往一個程式裡塞）。1978 年那本 K&R（Kernighan & Ritchie 的《The C Programming Language》第 1 版）把 C 與這套品味一起傳播出去，讓「小而組合」成了整整幾代工程師的母語。

### 工程師的領悟：你的 FluentBit pipeline 是這套哲學的現代轉世

你自建過 FluentBit＋Lua 的可觀測性 pipeline。停下來看它的形狀：log 進來 → 一個 filter 解析 → 一個 Lua script 加工 → 一個 filter 改寫欄位 → 一個 output 送出去。每個 stage 只做一件小事，前一個的輸出是後一個的輸入，你靠把這些小 stage **接起來**而不是寫一個大程式，得到你要的處理流。

這不是巧合地「像」Unix——它**就是** Unix 哲學的直系後代。FluentBit 的 input/filter/output 是 Unix 小工具，它的 pipeline 設定是 pipe，Lua script 是你臨時插進管線的自訂小工具。你做這套東西時憑的工程直覺（「每個 stage 單純一點、用組合而非堆功能」），正是 McIlroy 1978 寫下的那兩句。你已經在實踐它，這章只是把它的身世補給你。

（同樣這套「小工具用 pipe 組合」的哲學，放大到分散式、把 pipe 換成 message queue、把工具換成獨立部署的服務，就是 ch17 的微服務——微服務本質是 Unix 哲學的分散式版。這裡先埋一句，不展開。）

|---|---|
| **消滅了什麼負擔** | 「改一個需求要動全系統」的牽一髮動全身——把會變的決定關進模組，改動被局部化；「每個任務寫一個大程式」的重複勞動——改成小工具組合，舊工具重用 |
| **製造了什麼新負擔** | (1) 切割本身變成一門難學的功夫——**「會變的決定」要靠預判，預判錯了照樣牽一髮動全身**，只是你以為自己切對了。(2) 模組／工具一多，**散在哪、誰呼叫誰、組合關係怎麼追**成了新複雜度（你的微服務「這個請求到底流經哪幾個服務」就是這筆帳）。(3) 介面是承諾，**承諾一旦公開就難改**——你被自己當初定的介面綁住 |
| **在什麼情況下洩漏／反噬** | 當你把祕密**藏錯了**（沒包住真正會變的那個決定），或當「不該洩漏的實作」從窄介面**漏出去**（呼叫方偷偷依賴了你介面背後的行為），資訊隱藏就失效，改動又開始外溢。微服務裡這叫「共享資料庫」「跨服務偷讀對方的表」——介面之外的後門 |
| **這一跳的帳單一句話** | **複雜度從「程式碼內部」搬到了「邊界的選擇」上**——你不再煩惱每個模組怎麼寫，改成煩惱該把線畫在哪；而畫線需要預測未來（什麼會變），這是本質複雜度，搬不掉 |

最後一格是這章的核心。Parnas 沒有消滅複雜度，他把它**搬到了一個更前期、更需要判斷力的位置**：分解的決策。寫程式的複雜度降了，但「該怎麼切」這個複雜度升了——而且它要求你預測「未來什麼會變」，這是出了名地難。所以「切錯邊界」這個痛，五十年來換了無數包裝（模組、物件、套件、服務），核心一直沒變：**你賭了什麼會變，賭錯就還債。**

## 紙上推演

### 推演題一：用資訊隱藏檢驗你切過的一個服務邊界 **[20 分鐘] ★★**

挑你親手切過的一個微服務（或一個模組／套件）。先寫下它對外的介面（公開的端點／函式）。然後問三個問題並寫下答案：(1) 這個服務**藏住了哪一個會變的決定**？(2) 過去一年，有沒有某類需求一進來就逼你「這個服務和另外幾個一起改、一起發版」？(3) 如果有，那個「一起變」的東西，本來該被關進哪一個服務、卻洩漏到了多個服務？

#### 推演解答

判斷標準只有一條：**改動有沒有被關在一個盒子裡。** 切對了的服務，多數需求變動只動它自己；它藏住的那個決定（某段業務規則、某個外部系統的接法、某種資料表示）變了，外面無感。切錯了的服務，你會在第 (2) 題列出一串「總是一起改」的服務——那串服務其實是**一個會變的決定被攤開切到了多處**。

常見誤解一：以為「一個 service 一個動詞／一個 CRUD 資源」就是切對了。不是。「動詞」是流程步驟，「CRUD 資源」是資料表——兩者都是 Modularization 1 的變形。問的不是「它做什麼」，是「它藏住哪個會變的決定」。常見誤解二：以為服務數量多＝模組化好。數量無關，**改動的局部性**才是。十個總是一起發版的服務，比一個單體更糟（你多付了網路與分散式的帳，耦合卻一點沒減）。

如果你第 (2) 題答得出一串名字——恭喜，你親手復現了 Parnas 1972 的 Modularization 1。這不丟臉，這是預設陷阱；丟臉的是知道了還繼續照流程切。

### 推演題二：Unix pipe 與「微服務＋message queue」在哲學上是同一件事、在複雜度上差在哪 **[15 分鐘] ★★**

口頭講清楚：(a) Unix pipe 把小工具串起來，和你用 SQS 把微服務串起來，在「哲學」上為什麼是同一件事？(b) 但在「複雜度帳單」上，後者比前者多付了哪些東西？

#### 推演解答

(a) **哲學同源**：兩者都是「小單元各做一件事、藏住自己怎麼做、只透過一個窄介面把輸出交給下一個」。pipe 的窄介面是純文字串流，message queue 的窄介面是訊息格式。`grep` 不在乎前後是誰、你的 idempotent consumer 不在乎訊息哪來——這份「彼此不知道對方內部」正是 McIlroy「expect the output of every program to become the input to another」放大到分散式的樣子。你寫 SQS pipeline 時的直覺和寫 `cat | grep | sort` 的直覺是同一個。

(b) **複雜度差在「接合劑跨了機器」**。Unix pipe 的兩個工具跑在同一台機器、同一個核心調度下：資料傳遞近乎瞬時且可靠，一個工具掛了整條 pipe 一起死（失敗是同步且明確的）。換成 SQS：接合劑變成**網路**，於是你繼承了一整套 pipe 沒有的帳——訊息可能重複送達（所以你需要冪等消費者）、可能延遲或亂序、下游掛了訊息得排隊重試、整條鏈路要靠 trace 才看得出資料流經哪裡。pipe 的失敗是「整條一起死」，message queue 的失敗是「某一段悄悄壞掉、其餘照跑」——後者難 debug 得多。哲學沒變，但**把接合劑從記憶體換成網路，你就買下了分散式系統的全部複雜度**（這正是 ch17 要算的大帳）。一句話：同一個好點子，搬到網路上要另外付一筆過路費。

### 推演題三（反事實）：如果 Unix 當年用組語而非 C 寫，後續會怎樣 **[15 分鐘] ★★★**

假設 1973 年 Thompson 和 Ritchie 沒賭那一把，繼續用 PDP-11 組合語言寫 Unix。推演：(a) 可移植性會怎樣？(b) Unix 後來的擴散（進大學、長出 BSD／Linux 這條線、變成今天伺服器與雲的底座）還可能發生嗎？(c) 這對「效能 vs 生產力」鐘擺說明了什麼？

#### 推演解答

(a) **可移植性基本歸零**。組語貼著 PDP-11 的指令集，換一款 CPU 等於整套核心重寫。1978 年那次搬到 Interdata 8/32 之所以可能，正因為核心是 C、機器相關的只剩一小塊。沒有 C，這次移植大概不會發生，Unix 會跟 PDP-11 一起變成一座孤島，命運跟著那款硬體走。

(b) **擴散幾乎不可能**。Unix 能進大學、被無數人讀原始碼學習、長出 BSD 再長出今天的 Linux 與整個雲端伺服器生態，前提是它能**被搬到各式各樣的機器上**。一個焊死在某款已停產硬體上的 OS，不會有這個未來。你今天 ECS 容器底下那個 Linux 核心，血緣上能追到 1973 那次賭贏——抽掉 C，這條線斷在起點。

(c) **這是鐘擺的關鍵一課**：當年「用高階語言寫 OS 會慢」的恐懼是真的（早期確實有效能代價），選 C 是往「生產力／可移植」那邊押。但這次押注的回報不是當下的開發速度，而是**幾十年的存活與擴散**——可移植性讓 Unix 活過了它出生的那款硬體。鐘擺往生產力擺的那些時刻，帳常常不在當下結，而在十年後以「還活著、還能搬家」的形式回報。效能是當下的事，可移植性是時間的事；Thompson 和 Ritchie 賭的是時間。

## 自我檢核

口頭自答，講得出因果才算過：

1. Parnas 說模組不該照流程切、該照「會變的決定」切——用 KWIC 的例子講清楚：照流程切時，「句子存記憶體還是磁碟」這個決定為什麼會逼你改全部四個模組？照資訊隱藏切時為什麼只改一個？
2. 「資訊隱藏」和「關注點分離」差在哪？哪個是思考方法、哪個是切割準則？separation of concerns 這個詞是誰造的、哪一年？（提示：不是 Parnas。）
3. 你那次切出「分散式單體」，用 Parnas 的話精確診斷：你藏錯了什麼、把哪個會變的決定洩漏到了多個服務？
4. Unix 用 C 重寫核心，賭的是什麼、賭資是什麼、贏面是什麼？用「資訊隱藏」解釋 C 替 Unix 藏住了哪個會變的決定。
5. McIlroy 1978 那兩句原則裡，「expect the output of every program to become the input to another」為什麼是整個 Unix 組合哲學的關鍵？它逼出了什麼機制（pipe）、為什麼接合劑非得是「純文字」這種窄介面不可？
6. 網路上那個整齊的「do one thing and do it well / work together / handle text streams」三句版——它的出處是哪裡、為什麼不能當成 McIlroy 1978 的逐字原文引用？
7. 你的 FluentBit pipeline 和 `cat | grep | sort` 在哲學上是同一件事——講清楚「同一件事」具體指什麼（每個單元藏住什麼、承諾什麼）。
8. 這章的複雜度帳單一句話是「複雜度從程式碼內部搬到了邊界的選擇上」——為什麼說「該怎麼切」是搬不掉的本質複雜度？（提示：它要求你預測未來什麼會變。）

## 延伸閱讀

- **David Parnas, 〈On the Criteria To Be Used in Decomposing Systems into Modules〉, *CACM* 1972**（https://dl.acm.org/doi/10.1145/361598.361623 ；全文鏡像 http://sunnyday.mit.edu/16.355/parnas-criteria.html）。本章的源頭，只有十幾頁，**一定要讀原文**。重點看他怎麼把 KWIC 用兩種切法各切一次、再列那串「likely to change」的決策——你會發現 2026 年切微服務的所有道理，1972 年都寫完了。
- **The morning paper（Adrian Colyer）對這篇的導讀**（https://blog.acolyer.org/2016/09/05/on-the-criteria-to-be-used-in-decomposing-systems-into-modules/）。如果原文的 1972 行文讀起來吃力，先看這篇現代工程師視角的拆解，再回頭讀原文。
- **Doug McIlroy 等, Unix Time-Sharing System Foreword, *Bell System Technical Journal* 1978**（哲學原則的逐字出處；亦見整理頁 https://en.wikipedia.org/wiki/Unix_philosophy）。看那幾條原則的**原句**，順手對照網路流傳的三句版差在哪——這是一手史料 vs 後人轉述的好練習。
- **Eric S. Raymond, 〈The Origins and History of Unix, 1969–1995〉, *The Art of Unix Programming*** 第 2 章（http://www.catb.org/~esr/writings/taoup/html/ch02s01.html）。把 Unix 的誕生、C 重寫、擴散講成連貫故事；對「可移植性為什麼是決定性勝利」這點寫得最清楚。
- **Kernighan & Ritchie, *The C Programming Language*, 1978**（俗稱 K&R）。不必當教科書讀，翻它的序與前幾章，感受「小而克制」這套品味是怎麼被一本薄書傳給幾代人的——這本書本身就是 Unix 哲學的文體示範。
