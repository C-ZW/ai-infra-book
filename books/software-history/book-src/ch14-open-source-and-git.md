# ch14 — 開源與 Git：重寫所有權與協作

> **本章解決什麼問題**：前面所有章節談的是「怎麼把軟體蓋好」——抽象 strand 蓋語言、型別、資料模型，方法 strand 蓋流程與紀律。但有兩個更底層的問題，到這一章才被掀開：軟體**屬於誰**，以及一群陌生人怎麼**一起改同一份程式碼而不互相踩死**。這不是技術問題，是制度問題與資料結構問題的合體。自由軟體運動把「軟體是私有財產」這個預設翻成「軟體可以是公共財」，用的不是程式而是一份授權條款——一個法律駭客。Git 則把「多人併行改一個 codebase」這面協作複雜度的牆，用內容定址與 DAG 重新組織，讓便宜分支與 merge 變成日常。本章接 ch06 的 Unix 哲學（Unix 是這個協作文化的搖籃）、回收 ch10 的 Conway's Law（協作結構決定系統結構），並替後面的雲與微服務（見 ch16、ch17）鋪路——因為你今天靠的整個開源生態，就是這一章寫下來的。

## 從你已知的出發

你今天做的每件事，幾乎都站在這一章重寫過的兩塊地基上，而你大概從沒想過它們本來可以是另一個樣子。

第一塊地基是**所有權**。你用個人 GitHub 帳號 C-ZW 累積足跡——你的 commit 歷史、你 star 過的 repo、你發過的 PR，是一份對外公開的工程履歷。你做 AI Infra 轉職，靠的是 vLLM、llm-d 這些開源專案的生態：你可以 clone 它們、讀它們的原始碼、改它們、把改回貢獻回去，不必付授權費、不必簽 NDA、不必問任何人同意。這件事你視為理所當然，但它在四十年前是不存在的世界。1980 年代初，主流的軟體是「黑盒子」：你買到的是編譯好的二進位，原始碼是商業機密，反編譯它、修改它、散布修改版，是違法的。你今天賴以維生的「讀別人的程式碼、改它、把它組進自己的系統」這整套工作方式，是一場運動把它從非法變成常態的。

第二塊地基是**協作**。你每天 commit、開 branch、merge、發 PR、review——這套流程順到你不會注意它的形狀。想一個場景：你的 greenfield 微服務團隊從三人長到十幾人，五個人同時改五個服務、各跑各的分支，你不會因此恐慌，因為「開分支」在你的心智模型裡是免費、無痛的——隨手開、隨手 merge、衝突了就解。這份「分支是免費的」的直覺正是 Git 帶來的；在 Git 之前，分支昂貴、危險、能不開就不開，merge 是讓人冒冷汗的高風險手術。你能輕鬆用 PR 工作流跟一群人協作，是因為有人在 2005 年很短的時間裡，把「多人併行改一份程式碼」這個問題，重新設計成一個資料結構。

這一章講的就是這兩塊地基怎麼被鋪下來：Stallman 用一份授權條款重寫所有權，Torvalds 用一個 DAG 重寫協作。它們是同一場革命的兩面——一面制度、一面工具——而你活在它們重寫過的世界裡，每天用、卻很少回頭看它本來的樣子。

## 所有權的革命：Stallman 與「free as in freedom」

要懂這場革命的力道，得先懂它反叛的對象。

1970 年代的早期電腦文化，軟體大多是「附帶的」——買硬體、軟體跟著來、原始碼通常也跟著來，因為大家都在同一個圈子裡互相改、互相用。Stallman（Richard Stallman）在 MIT 人工智慧實驗室就活在這種文化裡：程式碼共享，看到 bug 就改、改了就分享。但 1980 年代初這個世界崩了：商業軟體公司興起，軟體變成獨立販售、需要保護的商品——原始碼變成商業機密，授權條款明文禁止你修改、反編譯、散布。Stallman 撞上這道牆的著名軼事，是一台 Xerox 印表機的驅動：他想改它在卡紙時通知使用者，卻因為它是專有的而拿不到原始碼——一件以前五分鐘就能自己動手修的事，他被法律擋在門外。

對 Stallman 來說，這不是不方便，是**道德問題**：一個程式設計師被禁止幫助鄰居、被禁止理解自己在用的工具、被迫接受一個無法檢查的黑盒子——這在他眼裡是對使用者的剝奪。1983 年 9 月 27 日，他公開宣告 **GNU 計畫**（GNU 是「GNU's Not Unix」的遞迴縮寫，一個工程師才覺得好笑的命名），要從零打造一整套自由、相容 Unix 的作業系統，讓人不必再依賴任何專有軟體。1985 年他成立 **Free Software Foundation（FSF）** 把這件事制度化；同年 3 月已先在 *Dr. Dobb's Journal* 發表《GNU Manifesto》把理念寫成文件。

這裡有一個全書最容易被英語害到的詞要先擺正：**free**。中文常譯成「免費」，但 Stallman 反覆強調，他要的 free 是 **free as in freedom（自由），不是 free as in beer（免費）**。他有一句廣為流傳、精煉到變成口號的講法（大意是「想想言論自由，而不是免費啤酒」——"think free speech, not free beer"，未逐字查證確切首次出處）。自由軟體的「自由」指的是四項使用者的權利：自由地執行、自由地研究與修改（這需要原始碼）、自由地散布、自由地散布你的修改版。價格無關緊要——你可以賣自由軟體賺錢，自由指的是這四項權利不被剝奪。

```text
        free 的兩種意思（這個詞坑了二十年的中文翻譯）

        free as in beer（免費）         free as in freedom（自由）
        ──────────────────             ──────────────────────────
        重點：價格 = 0                  重點：四項使用者權利不被剝奪
        可以是黑盒子                    必須給原始碼（否則無法研究/修改）
        ✗ 不是 Stallman 要的            ✓ 這才是自由軟體的「free」
```

### copyleft：用著作權法反著打的法律駭客

光是把程式碼公開還不夠。Stallman 看穿一個漏洞：如果你只是把程式碼丟到公領域（public domain），任何人都可以拿走它、加一點東西、然後把整包重新包成**專有**軟體賣出去——使用者的自由在下一手就被剝奪了。公開不等於能持續自由。

他的反制是軟體史上最漂亮的一招法律駭客：**copyleft**。它的核心機制是用著作權法（copyright）來達成跟著作權相反的目的。一般的著作權說「保留所有權利，不准複製散布」；copyleft 反過來說「你可以複製、修改、散布——**但有一個條件**：你散布出去的衍生版本，必須用同樣的條款釋出，同樣保障下游使用者那四項自由」。它把「自由」變成一種會自我複製、無法被回收的屬性：任何碰了 copyleft 程式碼的衍生作品，都被這個條款黏住，不能變回專有。

這個機制的法律載體是 **GPL（GNU General Public License）**。這裡有一個本書必須替你擺正的高危事實——**GPL 的第一版（GPLv1）是 1989 年，不是 1985 年**。FSF 是 1985 成立，GNU Manifesto 是 1985 發表，但統一的 GPL 條款第一版要到 1989 才出現（GPLv2 在 1991、GPLv3 在 2007）。FSF 1985 與 GPLv1 1989 這兩個年份非常容易被混成同一年，網路上錯得到處都是；本書照查證版寫成兩個不同年份。

這個「衍生版本必須同樣開源」的條款，業界俗稱**病毒式（viral）授權**（帶貶意，是商業公司怕被「傳染」的觀點），自由軟體陣營則叫它**互惠式（reciprocal）**。不管叫什麼，工程後果是真實的：你要把某個 GPL 函式庫組進商業產品時，法務會跳出來問「這會不會逼我們整個 codebase 開源」——那焦慮的源頭就是 Stallman 1989 年寫下的這個自我複製條款。授權的法律細節本書不展開（指向延伸閱讀），你只需記住：**copyleft 不是技術，是制度設計；它用既有的法律武器，反向達成了一個法律本來要阻止的目的。** 這是「方法 strand」在所有權層面的一次出手——不是新抽象，是新規則。

## Linux：開源協作不只是理想，是能贏的工程

理念再漂亮，沒有作品撐腰就只是宣言。GNU 計畫到 1990 年代初，周邊工具（GCC、Emacs）都很出色，唯獨最核心的作業系統核心（kernel）遲遲做不出來。自由軟體運動需要一個證明：一群素未謀面、沒有薪水、沒有公司指揮的志願者，**真的能協作蓋出一個嚴肅的、能跑生產環境的大型系統**嗎？還是說認真的軟體終究得靠一家公司、一支領薪水的團隊、一套指揮鏈來蓋？

1991 年，芬蘭大學生 Torvalds（Linus Torvalds）出於興趣開始寫自己的作業系統核心，丟上網讓大家一起改——這就是 **Linux**（1992 年改用 GPL 授權，與 GNU 工具鏈結合成完整的自由作業系統）。接下來十年就是那個證明：Linux 在沒有中央公司、靠全球志願者用 email 寄 patch 協作的模式下，長成了今天跑在絕大多數伺服器、雲端、Android 底層的核心。你部署的每個 container、ECS 上的每個 task，底下幾乎都是 Linux。一群陌生人真的蓋出了能贏的東西。

這個證明的份量，在於它**反駁了一個關於協作複雜度的直覺**。直覺告訴你：人越多、越分散、越沒有統一指揮，協調成本就越爆炸（這正是 ch10 Brooks 講的 O(n²) 溝通成本）。照這個直覺，一個上千人、散布全球、沒有老闆的專案應該會在自身的混亂裡崩潰。Linux 沒有崩潰。它怎麼做到的？這個問題的答案，催生了本章下半場的兩個東西：一篇講「為什麼這種混亂反而有效」的文章，和一個讓這種混亂變得可管理的工具。

## 大教堂 vs 市集：Raymond 把開源的協作模式寫成理論

1997 年，Raymond（Eric S. Raymond）在一場 Linux 研討會（Linux Kongress, Würzburg）發表了《The Cathedral and the Bazaar》（大教堂與市集；1999 出書）。它做的事是把 Linux 現象**理論化**——回答「為什麼這種看似混亂的開發法行得通」。

Raymond 提出兩種開發模式的對比：

- **大教堂（Cathedral）**：軟體由一小群核心開發者，在封閉中精雕細琢，到成熟了才釋出。像蓋一座大教堂，有總設計師、有藍圖、由內而外控制。傳統商業軟體、甚至早期的 GNU 工具，多是這個模式。
- **市集（Bazaar）**：軟體在公開、喧鬧、看似無序的環境裡開發。「早發布、常發布」（release early, release often），讓盡可能多的人看到原始碼、回報 bug、提交修改。像一個吵雜的市集，沒有中央規劃，秩序從大量參與者的互動裡**湧現**出來。Linux 是這個模式的代表。

Raymond 從這個對比裡提煉出一句廣被引用的格言（大意是「眼睛夠多，所有 bug 都無所遁形」——"given enough eyeballs, all bugs are shallow"，他稱之為 Linus's Law，未逐字查證原始措辭）。它的論點是：在市集模式裡，因為原始碼對所有人公開、參與者夠多，每個 bug 總會有某個人剛好有對的背景一眼看穿——找 bug 這件難事被**平行化**了。

這裡要替 Raymond 加個但書：「眼睛夠多 bug 就淺」後來被現實多次打臉——最有名的是 2014 年 OpenSSL 的 Heartbleed，一個攸關全網安全的核心開源庫，原始碼公開多年，嚴重漏洞卻沒被「夠多的眼睛」抓到。準確的講法是：開源**讓**檢查成為可能，但不**保證**檢查真的發生（沒人付錢去看那段碼，眼睛再多也是零）。別把它讀成「開源所以一定安全」——那是另一種銀彈幻覺（回收 ch10）。

對你的意義很直接：**你每天的 PR-based 工作流，就是市集模式的縮小版日常。** 你開分支、推上去、發 PR、讓隊友 review、收到 comment、改、合進去——這套「公開地、增量地、讓多人看到並提交修改」的流程，正是 Raymond 1997 描述的那套協作哲學，被工具化、制度化之後，變成了今天每個團隊的標準動作。你不會用「市集」這個詞稱呼它，但你天天住在裡面。

## 自由軟體 vs 開源：一場用詞之爭，底下是理念分裂

到這裡有個你必須擺正的觀念，因為它常被當成同義詞混用，其實底下是一道真實的裂縫。

1998 年 2 月，在一場策略會議上（VA Linux 的辦公室，配合 Netscape 把瀏覽器原始碼開放這件大事），一群人決定要造一個新詞來推廣這套開發模式。「open source（開源）」這個詞由 **Christine Peterson** 提出；隨後 **OSI（Open Source Initiative）** 成立（Raymond 任首任主席，Bruce Perens 等共同創立）。

為什麼要造新詞？因為「free software」這個招牌有兩個問題：一是 free 的「免費」歧義一直在誤導人（前面講過），二是——更關鍵的——它帶著 Stallman 那套**道德訴求**，而道德訴求對企業是有威脅性的。你跟一家公司說「使用專有軟體在道德上是錯的、是剝奪使用者自由」，公司會把你當激進份子請出門。「open source」這個新詞是刻意去道德化的：它把重點從「使用者的自由是一種權利」搬到「公開原始碼的開發模式能做出更好、更可靠的軟體」——一個**務實的、可以對 CTO 推銷的**論點。

於是這裡形成了本書要替你擺正的核心分歧：**自由軟體（free software）與開源（open source）不只是同一件事的兩個名字，是兩種不同的世界觀。**

> **自由軟體 vs 開源：底下的理念分裂**
>
> | | 自由軟體（FSF/Stallman, 1985） | 開源（OSI, 1998） |
> |---|---|---|
> | 核心訴求 | **倫理**：軟體自由是使用者的權利 | **務實**：開放開發模式能做出更好的軟體 |
> | 「為什麼要開放」 | 因為剝奪使用者自由在道德上是錯的 | 因為它在工程上更有效、更可靠 |
> | 對企業的姿態 | 帶對抗性（專有軟體是不義的） | 友善（這對你的生意有好處） |
> | 招牌的用意 | 強調 freedom，哪怕嚇跑企業 | 刻意去道德化，方便對企業推銷 |
> | 對立的不是技術 | 是「軟體該屬於誰、為什麼」這個價值問題 | |

兩邊在「程式碼要公開、要能改、要能散布」這些**操作**上幾乎重疊，所以實務上常被合稱 FOSS（Free and Open Source Software）。但它們對「**為什麼**」的答案根本不同。Stallman 至今堅持用「free software」、拒絕被收編進「open source」這個去道德化的框架，不是文人相輕，是他認為一旦把運動的理由從「權利」降格成「好處」，那這個運動隨時可以因為「在某個情況下專有更划算」而被放棄——權利不能談判，好處可以。

這場分歧對今天的你不是純歷史：你選套件時看的授權——MIT/Apache 這種**寬鬆式（permissive）**（愛怎麼用就怎麼用、可組進閉源產品）vs GPL 這種 **copyleft**（碰了就得跟著開源）——正是這兩種世界觀的對峙（permissive 是務實派、copyleft 是理想派的武器）。你做技術選型時就在這條線上選邊，只是多半沒意識到自己在替一場 1998 年的理念之爭投票（這條帳本章末與推演題 2 再展開）。

## 個案：Git 的誕生——協作複雜度如何逼出一個資料結構

現在進入本章的個案，也是全書最乾淨的一個「複雜度逼出資料結構」的標本。前面說過，Linux 證明了上千人能協作蓋出大系統。但「能」不等於「不痛」——那個協作本身需要工具撐著，而這個工具的故事，把「協作複雜度如何具體地塑造一個資料結構」攤得清清楚楚。

### 前史：集中式版控的牆

先看 Git 之前的世界。你早就精通版控指令層，這裡只講形狀、不講操作。

早期的版本控制是**集中式（centralized）**的：CVS（1990 起流行）、後來的 SVN（Subversion，2000 起、1.0 於 2004）。它們的模型是有一台**中央伺服器**保管唯一的、權威的版本歷史；每個開發者從伺服器 checkout 一份工作副本，改完 commit 回伺服器。這個模型的麻煩，在「上千人併行改一個 codebase」的規模下會被放大成牆：

- **分支昂貴。** 開分支往往是重操作（早期 CVS 甚至用複製目錄硬幹），merge 更是惡夢。結果大家養成「能不開分支就不開」的習慣——跟你今天「隨手開分支」的直覺完全相反。
- **每個操作都要連伺服器。** commit、看歷史、diff 很多都得跟中央伺服器來回，伺服器一掛、網路一斷你就動不了。
- **權威集中、不適合大規模分散協作。** 上千個志願者不可能都有中央伺服器的 commit 權限；Linux 於是長期靠土法協作——大家用 email 互寄 patch、由維護者手動整合，能用但累、又不 scale。

### 引爆點：BitKeeper 撤回免費授權

2002 年起，Linux kernel 的開發改用一個叫 **BitKeeper** 的**分散式**版控工具——它是當時少見能讓分散式協作順暢運作的東西，但它是**專有軟體**。一個全世界最重要的自由軟體專案，核心協作工具竟是專有的，這在社群裡一直是個刺。BitKeeper 是它的公司 BitMover 出於善意，免費（注意，是 free as in beer）授權給 Linux 社群用的。

2005 年 4 月，這個脆弱的安排崩了。事件的導火線：另一位 kernel 開發者 Andrew Tridgell 著手對 BitKeeper 的協定做反向工程（想做一個能跟 BitKeeper repo 溝通的自由工具）。BitMover 認定這違反了授權精神，於是**撤回了給 Linux 社群的免費授權**。一夜之間，全世界最關鍵的軟體專案，失去了它的協作工具。

對 Torvalds 來說，市面上現有的選項都不合格：集中式的 CVS/SVN 撐不起 Linux 的規模與分散協作；其他分散式工具不是太慢就是設計不對他胃口。於是他做了一個典型 Torvalds 的決定——**自己寫一個**。

### 三週寫成：直接回應「上千人併行改一份程式碼」

接下來是本章最該被記住的部分。把可查證的時間軸攤開（照一手 commit 紀錄）：開發**從 2005 年 4 月 3 日開始**；4 月 7 日 Git 就能用自己管理自己的原始碼（self-hosting，第一個 commit 著名地自嘲為 "the information manager from hell"）；4 月 18 日完成第一次多分支 merge；到 6 月 16 日已實際接手管理 Linux kernel 2.6.12 的發布；7 月 26 日 Torvalds 把日常維護交棒給 Junio Hamano。

所以「三週寫成」這個流傳說法要稍微 hedge：更精確地說，**核心在幾天內就能自我管理、數週內就接手管理真正的 Linux kernel**——速度本身已經驚人。重點不在天數的精確，而在於：一個規模到上千人的協作複雜度問題，逼出了一個在數天到數週內成形、好到二十年後統治全行業的資料結構。

它為什麼能這麼快又這麼好？因為 Torvalds 沒把它當「版控軟體」設計，而是當**一個內容定址的檔案系統**設計——版控只是蓋在上面的薄薄一層。核心設計直接對準協作複雜度的痛點（內部物件模型細節本書不展開，指向延伸閱讀；這裡只講它**為什麼長這樣**）：

- **內容定址（content-addressable）。** Git 把每一塊內容（檔案、目錄樹、commit）用它**內容的雜湊值**（當年用 SHA-1）當作名字存起來。同樣的內容必然有同樣的名字，不同的內容必然有不同的名字。這帶來一個對分散協作至關重要的性質：**完整性是內建的**——任何一塊資料被竄改或損壞，它的雜湊就對不上，立刻被抓出來。當上千個互不信任、散布全球的人在交換程式碼時，這個性質讓你不必信任傳輸通道，只要雜湊對得上，內容就一定是對的。
- **歷史是一個 DAG（有向無環圖）。** 每個 commit 記錄它的父 commit（merge 時是多個父），於是整個版本歷史是一張節點指向祖先的有向無環圖，而不是一條線。這正是「分支」與「merge」在資料結構上的真身：分支只是圖上的一個指標，merge 就是讓一個 commit 認領多個父節點。在這個模型裡，**開分支幾乎零成本**（不過是多一個指標），merge 是圖的合併運算而非高風險手術。你今天「隨手開分支」的直覺，直接來自這個 DAG 設計。
- **分散式（distributed）：每個 clone 都是完整的。** 沒有「中央伺服器才有完整歷史」這回事——你 `git clone` 下來的，是**整個物件圖、完整的歷史**。每台機器都是一個完整、自足的 repo，可以離線 commit、離線看歷史、離線開分支。中央伺服器（如後來的 GitHub）變成一個**約定**而非**必須**——大家同意把某個 repo 當「官方」，但技術上沒有哪個 clone 比別的更權威。

```text
   集中式（CVS/SVN）             分散式（Git）

   ┌──────────────┐             每個 clone = 完整歷史的 DAG
   │ central repo │
   │ 唯一權威歷史 │             ○──○──○──○   ←── main
   └──────┬───────┘                    ＼
       ┌──┼──┐                          ○──○   ←── feature
       │  │  │                 開分支＝多一個指標、近乎零成本
      副本 副本 副本            merge＝讓一個 commit 認領多個父
   （凡事要連伺服器）            （每台機器都能離線 commit/分支/看歷史）
```

把這個個案的因果講透：**這不是天才憑空變出工具，而是一個非常具體的協作複雜度（上千人、分散全球、互不信任、要併行改一份 codebase、原本的工具一夜消失）逼出了一個非常具體的資料結構（內容定址 + DAG + 全量分散）。** 複雜度沒有被消滅，是被**重新組織**了——原本散落在「中央伺服器的協調」「email 寄 patch 的人工整合」「不敢開分支的保守」裡的協作摩擦，被收攏進一張可雜湊驗證的有向無環圖，讓機器去扛大部分協調。這是「複雜度搬家」最教科書的一次演示：從人的協調，搬到資料結構的設計。

## GitHub：把開源變成社交網路

Git 解決了「機器之間怎麼交換與整合歷史」，但它本身是個冷硬的工具——沒有網頁、沒有帳號、沒有「在哪裡放官方 repo」的答案。把 Git 從工程師的命令列工具，變成一個全球性的協作與社交平台的，是 **GitHub**（2008 年創立，公開上線於 2008 年 4 月 10 日；創辦人 Tom Preston-Werner、Chris Wanstrath 等）。

GitHub 做的事，是在 Git 這個分散式引擎外面包一層**集中的社交層**。注意這裡的鐘擺——Git 技術上徹底分散（每個 clone 平等），GitHub 卻在它上面重新引入一個事實上的中心（大家都把官方 repo 放這裡）。這不矛盾，是分工：底層協議分散（誰都能離線、能自架），上層協調集中（大家**自願**約定一個共同的會合點）。它把開源協作從「散落各處、靠郵件列表」收攏成「有一個地方能看到所有人在做什麼」。

GitHub 加上去的東西裡，有一個是它的發明、也徹底改變了你的日常：**pull request（PR）**。在純 Git 裡，把修改提議給別人是手動流程（推到某處、通知對方來拉）。GitHub 把它變成一個網頁上、可見、可討論的物件：你 fork 一個 repo（複製到你名下，也是把「每個 clone 平等」社交化的呈現）、改、開 PR，於是有一個頁面，上面是 diff、討論串、review 意見、CI 的綠勾紅叉。這套你天天做的動作，是 GitHub 把 Git 的分散式能力，包裝成一個任何人都能參與、旁觀、社交的儀式。

於是開源從「一種授權與協作模式」變成了「一種社交活動」。你的個人 GitHub 帳號 C-ZW 成了履歷——你 star 什麼、貢獻過哪些專案、commit 的密度，公開可見，成為一種工程聲望的貨幣；你做 AI Infra 轉職時讀 vLLM、llm-d 的原始碼、追它們的 issue 與 PR，整個入口都建立在 GitHub 把開源社交化這件事上。Stallman 在乎自由、Torvalds 在乎工具扛不扛得住 Linux 的規模，而 GitHub 在他們的地基上蓋了一個讓開源變得**好玩、可見、會上癮**的層——這正是開源在 2010 年代之後爆炸式擴張的一大推力。（這層社交化的帳單，下一段算。）

## 複雜度搬去哪了

這一章有兩筆帳要結：一筆是所有權（自由軟體/開源），一筆是協作（Git/GitHub）。

| 這一跳 | 消滅／搬走了什麼負擔 | 製造了什麼新負擔 | 什麼情況下洩漏／反噬 |
|---|---|---|---|
| **copyleft / 開源授權** | 軟體不再非得是黑盒子私產；你能讀、改、組裝別人的程式碼，不必重造輪子 | 授權相容性成了新複雜度：GPL/MIT/Apache 能不能混、會不會「傳染」、組進閉源產品合不合法 | 你把一個 copyleft 函式庫組進商業產品，法務發現整個 codebase 可能被要求開源；不同授權的相依互相衝突 |
| **「open source」去道德化** | 開源變得對企業友善，採用門檻降到最低，生態爆炸式擴張 | 運動的「為什麼」從權利降格成好處——當專有更划算時，理由隨時可被放棄 | 開源專案被大公司吸乾貢獻卻不回饋；「開放核心」「改授權」爭議（理想被務實稀釋） |
| **Git：協作搬進 DAG** | 分支與 merge 從高風險手術變成零成本日常；協作摩擦從人工協調搬給資料結構 | 一個你大多時候不必懂、但偶爾會漏的內部模型（物件、ref、rebase 改寫歷史） | rebase/force push 改寫共享歷史搞爛協作；衝突的 merge 還是要人腦解；歷史一旦複雜，工具的抽象就漏給你看底層 |
| **集中搬到本機（分散式）** | 離線可 commit/分支/看歷史；中央伺服器從「必須」降為「約定」 | 每個 clone 都扛全量歷史（大 repo 變肥）；「哪個是權威」從技術問題變成社交約定 | monorepo 大到 clone 要很久；沒有強制的中央權威，要靠流程紀律維持秩序 |
| **GitHub：開源社交化** | 開源從散落各處收攏成一個會合點；PR/fork 讓參與門檻降到點幾下 | 分散式的理想上重新長出一個事實壟斷的中心（鐘擺擺回集中） | 一個平台成了開源的單點依賴（它一掛、被收購、改政策，整個生態跟著抖） |

**這一跳的帳單，一句話：軟體的所有權從私有變成可公共，協作的複雜度從人的協調搬進了一張可驗證的有向無環圖——代價是你多扛了「授權相容性」這種制度複雜度，和「分散式理想最後又長回一個中心（GitHub）」這個鐘擺反噬。** 留意這裡的鐘擺（回收 ch01）：集中（CVS/SVN）→ 分散（Git）→ 又集中（GitHub）。技術上分散是進步，但人類協作終究需要一個共同的會合點，於是中心以另一種形式回來了——只是這次的中心是約定的、可替換的，而不是技術上不可少的。這個「分散了又自願集中」的模式，你在後面的雲（見 ch16）會再看到一次。

## 紙上推演

### 推演題 1 — 複雜度搬家：Git 把什麼從伺服器搬到了你的本機？ **[15 分鐘] ★★**

口頭講清楚：相較於 SVN 這類集中式版控，Git 把哪些東西從中央伺服器搬到了每個人的本機？這次搬家**消滅了什麼負擔、又新增了什麼負擔**？用「複雜度守恆」的框架回答，不要只說「Git 比較好」。

#### 推演解答

搬走的是「凡事都要連伺服器、凡事都要中央權威」這個負擔。集中式模型裡，完整的版本歷史只在伺服器上，所以 commit、看歷史、diff、開分支這些操作很多都得跟伺服器來回——伺服器掛了或網路斷了你就動彈不得，而且開分支/merge 在伺服器端是重操作，大家養成不敢開分支的習慣。Git 把**整個歷史（DAG 物件圖）搬到每一台 clone**：你的本機就是一個完整自足的 repo，可以離線 commit、離線開分支、離線看全部歷史，分支不過是圖上多一個指標，幾乎零成本。

但複雜度沒有消失，是搬了家、換了形狀，記下三筆新負擔：

1. **每個 clone 都扛全量歷史。** 方便的代價是肥——大型 monorepo 的歷史可能很大，clone 一次要很久、佔很多磁碟（這也是為什麼有 shallow clone、partial clone 這些後來的補丁）。
2. **「哪個是權威」從技術問題變成社交問題。** 集中式有天然的唯一真相（伺服器上那份）；Git 技術上每個 clone 平等，於是「哪個 repo 是官方的」不再由技術決定，得靠人的約定（這正是 GitHub 後來填的位）。
3. **一個你偶爾得跳下去懂的內部模型。** 平常你不必懂物件、ref、reflog；但當你 rebase 改寫歷史、force push 搞爛別人的分支、或要從一個爛掉的狀態救回來時，這個被藏起來的內部模型就漏給你看——這是抽象洩漏的標準場景（預告 ch19）。

**常見誤解**：以為「分散式」只是「不用伺服器」。不是——GitHub 證明了你還是會（自願）用一個中心。分散式的真正意義是「中央伺服器從技術上的必須，降格成社交上的約定」，這讓中心變得可替換（你能搬家、能自架），而不是讓中心消失。

### 推演題 2 — 自由軟體 vs 開源：為什麼這不只是用詞之爭？ **[20 分鐘] ★★★**

有人跟你說「free software 跟 open source 是同一件事，Stallman 只是在計較用詞」。請反駁他：講清楚這兩個詞底下是**兩種不同的世界觀**，以及這個分歧為什麼在你今天的技術選型（選授權）裡仍然活著。

#### 推演解答

它們在**操作**上幾乎重疊（都要求程式碼公開、可改、可散布），所以實務常合稱 FOSS——這是「只是用詞」這個誤解的來源。但它們對「**為什麼**要開放」的答案根本不同，而那才是分歧的本體：

- **自由軟體（Stallman/FSF, 1985）的答案是倫理的**：軟體自由是使用者的**權利**，剝奪它（用專有黑盒子綁住使用者）在道德上是錯的。開放不是手段，是目的本身。
- **開源（OSI, 1998）的答案是務實的**：開放的開發模式能做出**更好、更可靠**的軟體（市集模式、眾人之眼），所以值得做。「open source」這個詞 1998 年由 Christine Peterson 提出，刻意去道德化，就是為了把它變成一個能對企業 CTO 推銷的、關於「好處」的論點，而不是一個關於「權利」的、會嚇跑企業的道德指控。

為什麼這個分歧是真的、不是計較？因為**理由的性質決定了運動在壓力下會不會崩**。如果開放的理由是「它有好處」，那麼在某個「專有更划算」的情境下，這個理由就站不住——好處是可以被成本效益翻盤的。如果理由是「這是權利」，那它不隨情境動搖——權利不能拿來做成本效益分析。Stallman 拒絕被「open source」收編，正是怕運動的地基從不可談判的權利，被換成可談判的好處。

這個分歧在你今天的選型裡活生生：選 **permissive（MIT/Apache）** 還是 **copyleft（GPL）**，技術後果完全不同（前者能閉源商用、後者會「傳染」），而這個差別的根就是 1985（權利）與 1998（好處）那兩種世界觀。

**常見誤解**：以為 copyleft（GPL）比 permissive（MIT）「更開放」。不是「更」，是**不同方向**的開放——GPL 用強制條款保護下游的自由（限制了你閉源的自由，來保障使用者的自由），MIT 把幾乎所有自由都給你（包括閉源的自由）。哪個「更自由」取決於你問的是誰的自由——這正是兩種世界觀的核心差異，不是程度問題。

### 推演題 3 — 反事實：沒有便宜的分支與 merge，PR 工作流還可能存在嗎？ **[15 分鐘] ★★**

反事實推理：假設版本控制停在集中式時代（分支昂貴、merge 是高風險手術、沒有便宜的本地分支）。你今天的 PR-based 協作工作流——人人開 feature branch、發 PR、review、merge——還可能以現在這個形狀存在嗎？為什麼？

#### 推演解答

不可能以現在的形狀存在。整套 PR 工作流的前提，是**分支便宜到可以隨手開、merge 安全到可以天天做**。抽掉這個前提，三件事都崩：「人人開 feature branch」變成製造大量痛苦 merge 的奢侈品，大家會退回主線直接改；「先各做各的、最後再合」太危險，團隊會傾向在共享主線上頻繁同步而非長出分支；fork-and-PR 更無從談起，因為它整個建立在「每個 clone 都是完整平等的 repo」這個 Git 的分散式 DAG 性質上，而集中式模型裡根本沒有「平等的完整 clone」。

所以準確的因果鏈是：**Git 的便宜分支與安全 merge（來自 DAG 設計）→ 讓「隔離每個改動到分支、再合回」變成廉價安全的日常 → GitHub 把它包裝成 PR/fork 的社交儀式 → 才有你今天的協作工作流。** 抽掉第一環，後面整條鏈都不成立。PR 工作流不是憑空的最佳實踐，是被一個資料結構的性質「允許」出來的——工具能力先變，工作方式才跟著長出來（全書反覆的模式：抽象變了，方法跟著變）。

**常見誤解**：以為 PR 是一種「管理流程/方法論」，跟工具無關，換個版控也能照搬。錯——PR 工作流是 Git 的 DAG 性質**長出來**的，不是先有方法論再找工具實作。沒有便宜分支，這套方法論連想都不會被想出來。

5. 「自由軟體」與「開源」在操作上幾乎一樣，那它們**真正**的分歧在哪？為什麼這個分歧不是計較用詞、而是會決定運動在壓力下崩不崩？這個分歧怎麼活在你選 MIT vs GPL 的決定裡？
6. Git 誕生的觸發事件是什麼（BitKeeper 那段）？Torvalds 把「上千人併行改一份 codebase」這個協作複雜度，用哪三個核心設計（內容定址、DAG、全量分散）重新組織？各自解了什麼痛？
7. 用「複雜度守恆」講：Git 把協作複雜度搬去哪了？這次搬家的帳單是什麼（至少兩筆）？
8. GitHub 在 Git 的「徹底分散」上重新引入了一個「事實上的中心」，這是不是倒退？用鐘擺（分散↔集中）解釋為什麼這不矛盾。

## 延伸閱讀

- **Richard Stallman,《The GNU Manifesto》（1985）** — 自由軟體運動的創世文件，讀第一手的理念訴求（而非二手轉述）。看他怎麼把「分享程式碼」講成一個倫理立場。線上：gnu.org/gnu/manifesto。
- **Richard Stallman,《The GNU Project》** — 比 Manifesto 更完整的回顧，講 GNU 的緣起、free 的兩種意思、copyleft 的動機。要分清 FSF（1985）與 GPLv1（1989）兩個年份，讀這篇最清楚。線上：gnu.org/gnu/thegnuproject。
- **Eric S. Raymond,《The Cathedral and the Bazaar》（1997）** — 把 Linux 現象理論化的那篇。重點讀「大教堂 vs 市集」的對比與 Linus's Law；讀完記得自己補上 Heartbleed 這個反例的但書。線上：catb.org/~esr/writings/cathedral-bazaar。
- **Linus Torvalds, Git 最初的 commit 與 2005 年的 git mailing list 討論** — 想看「協作複雜度逼出資料結構」的第一手現場，去看 Git 倉庫裡那個 2005-04-07 自嘲 "the information manager from hell" 的初始 commit，與當時郵件列表上他解釋設計取捨的貼文。
- **Scott Chacon & Ben Straub,《Pro Git》（線上免費）** — 本章刻意沒展開的「Git 內部物件模型」在這裡有最清楚的講解。讀第 10 章「Git Internals」，你會看到內容定址、blob/tree/commit、DAG 在程式碼層面長什麼樣。線上：git-scm.com/book。
- **Christine Peterson,「How I coined the term 'open source'」（opensource.com, 2018）** — 「open source」這個詞 1998 年怎麼被造出來、為什麼要刻意去道德化的第一手回憶，理解自由軟體 vs 開源分歧的好入口。
