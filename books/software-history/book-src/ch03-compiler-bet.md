# ch03 — 編譯器的賭注：把「對機器說話」外包

> **本章解決什麼問題**：上一章（見 ch02）讓程式變成可被程式操作的資料，這一章就動手用程式操作程式——把「翻譯成機器聽得懂的話」這件苦工外包給另一支程式。我們撞的是「人腦追不上機器細節」這面牆：位址要手算、暫存器要手調、改一行牽動全表。翻越的方式是**抽象**（從機器碼到組語到高階語言，兩跳），但每一跳都有人懷疑「機器產生的碼會不會比人手寫的慢」。FORTRAN（1957）正面押了這個賭——這是全書「效能↔生產力」鐘擺的第一次擺動，後面每一章的取捨都在這條鐘擺上。

## 從你已知的出發

你每天寫 TypeScript，存檔，`tsc` 把它編成 JavaScript，丟給 Node 跑，V8 再把熱路徑 JIT 成機器碼。這條翻譯鏈有四五層，你一層都沒親手碰過，卻每天把產品壓在它身上。你信任「機器產生的碼夠快」——快到你從來不會打開 V8 吐出的組語去檢查它有沒有偷懶。

這份信任不是天生的，是 1957 年有一群人替你賭贏來的。

把鏡頭再拉近一點。你選 NestJS 而不是手寫一個 epoll loop 的 C 服務，理由是「開發速度、團隊好維護、生態完整」——你心裡有一筆帳：NestJS 的 decorator、DI、middleware 多繞的那幾層 overhead，換來的生產力划算。你那次把尖峰 RDS CPU 砍掉四成，靠的是改 query 與加 index，不是去重寫 V8。也就是說，你**預設**了「抽象層的效能損失在可接受範圍內，瓶頸在別處」。

這個預設就是本章的主題。「用更高的抽象換生產力，賭效能損失可接受」——這個賭，第一次有人正式押下注、而且賭贏，是 1957 年 IBM 那支寫 FORTRAN 編譯器的團隊。在他們之前，「機器產生的碼能不能跟人手寫一樣快」不是預設，是一個被資深程式設計師公開嘲笑的妄想。

這一章講的就是：這份妄想怎麼變成你每天的預設。

## 機器碼的痛：你是 CPU 的祕書

先把「沒有抽象」的世界講清楚，不然你感受不到後面每一跳省掉了什麼。

最底層，程式是一串數字。不是「看起來像數字的助記符」，是真的數字：這個位元組是「載入」的操作碼，接下來幾個位元組是要載入的記憶體位址，下一條是「相加」，再下一條是「跳到位址 0x4A2」。你不是在「寫邏輯」，你是在當 CPU 的祕書，替它把每一個動作的編號和位址都抄對。

這裡有三種痛，而且彼此會放大：

**第一，位址全靠手算。** 你想跳到某段程式，得知道它在記憶體第幾個字。這個位址不是你取的名字，是它**實際擺放的位置**。所以你得一邊寫一邊數：這支副程式從第 100 個字開始、佔 37 個字，那下一段就從 137 開始……你在心裡維護一張位址表，而這張表只存在你腦裡。

**第二，改一行牽動全表。** 真正要命的是這個。假設你在中間插入三條指令。後面所有東西的位址全部往後挪三格——而每一條「跳到位址 X」的指令裡，那個 X 都是寫死的數字。你得回頭把所有受影響的位址手動 +3。漏改一個，程式不是壞掉，是**安靜地跳到錯的地方**繼續跑。這是你在 production 最怕的那種 bug 的史前版本：沒有 stack trace，沒有 type error，機器忠實地執行了一個你沒打算下的指令。

**第三，暫存器要自己調度。** CPU 只有少數幾個暫存器（快速的工作記憶體）。哪個值該留在暫存器、哪個該先寫回記憶體騰位置、什麼時候再載回來——全是人腦排程。排得好程式快，排得差程式慢，而「排得好」需要對整支程式的資料流瞭如指掌。

你對這三種痛其實不陌生，只是你遇到的是它們的後代。手算位址＝你沒有變數名、只有 magic number 寫死的偏移量；改一行牽動全表＝你改了一個共用常數卻漏掉三處引用（還好你有編譯器報錯）；暫存器調度＝你在 hot path 上手動算 cache locality。差別在於：你只在**最痛的那一小段**做這種事，1950 年代的人**整支程式**都得這樣寫。

```text
機器碼世界裡，人腦扛的三筆負擔

  位址簿記 ──┐
  暫存器調度 ─┼──► 全在程式設計師腦中，沒有任何工具替你記
  資料流追蹤 ─┘
            一改全動，漏一個就安靜跑錯
```

關鍵不是「當年的人比較笨所以覺得痛」。是這些負擔的本質：它們都是**機器的細節**，跟你想算的東西（一道物理方程、一筆薪資）毫無關係，卻佔掉你一半以上的腦力。要翻過這面牆，方向只有一個——把機器的細節外包出去，讓某個東西替你記、替你算、替你調。

那個東西，就是另一支程式。能這樣做的前提，正是上一章的儲存程式：程式既然是資料，就能被別的程式讀進來、加工、產出。**用程式產生程式**——你做 OpenAPI schema 生 client、用 codegen 生 boilerplate，做的是同一件事，只是你站在這個前提上太久，久到不會注意它是個前提。

## 第一跳：組語——不再直接對機器說話

第一個外包很小，但方向決定了一切。

EDSAC（1949，Cambridge，Wilkes 的機器）上，一個叫 David Wheeler 的研究生寫了一套叫 **Initial Orders** 的東西。它讓你不用寫純二進位：每條指令的操作碼可以寫成一個助記字母（`A` 代表 add，剛好用字母 A 的位元樣式編碼），位址可以寫成十進位，而且——這是關鍵——你可以用**相對位址**，讓程式被載入時自動算出實際位置。整套東西只有 31 個字（words），卻同時做了後來「組譯器」（assembler）與「載入器」（loader）兩件事的雛形。它一般被認為是世界上第一個組譯器。

Wheeler 順手還奠基了副程式（subroutine）的概念（所謂 Wheeler Jump），隔兩年（1951）和 Wilkes、Gill 合寫了史上第一本程式設計教科書《The Preparation of Programs for an Electronic Digital Computer》，裡面就有副程式庫的想法。你今天 `import` 一個函式庫、不用自己重寫，這條線一路接到這裡。

組語外包掉了什麼？看那三筆負擔：

| 機器碼的負擔 | 組語怎麼外包 | 還沒外包的 |
|---|---|---|
| 位址手算 | 用符號名字代替數字，組譯器替你算實際位址 | — |
| 改一行牽動全表 | 你改的是符號，重新組譯時位址自動重算 | — |
| 暫存器調度 | **沒動**——還是你自己排 | 全部留著 |

這一跳的本質：你不再直接對機器說它的位址數字，你說**符號**，讓一支程式把符號翻成數字。但組語跟機器碼還是**一對一**的——一條組語指令對應一條機器指令。你說話的「詞彙」沒變，變的只是你不用記每個詞的編號。暫存器調度、資料流、演算法的每一步，仍然得你親手攤開成一條一條機器動作。組語讓你不用當 CPU 的「祕書」，但你還是得當 CPU 的「翻譯」，逐字逐句。

所以第一跳是個安全的賭：因為一對一，沒人會懷疑組譯器產生的碼比手寫慢——它產生的就是你寫的那些指令。真正的賭注在第二跳。

## 第二跳：高階語言——說人話，讓機器自己想辦法

第二跳要外包的，是組語沒碰的那塊：**怎麼把「一個想法」拆成一條一條機器動作**。

你想寫「把這個陣列的每個元素乘以二再加總」。組語裡，這是一個迴圈：設一個計數器、設一個位址指標、載入、相乘、相加、位址往前移、計數器減一、檢查歸零沒、沒歸零就跳回去。八九條指令，每一條你都得親手寫對。高階語言的賭注是：**讓你寫一行接近數學或人話的式子，由一支程式去拆成那八九條指令**。

這就跨過了一對一的界線。一行高階語言會展開成多條機器指令，而且**怎麼展開有好幾種選擇**。同一個迴圈，計數器放暫存器還是記憶體、檢查歸零的條件怎麼安排、要不要把不變的計算提到迴圈外——不同選擇跑起來快慢可以差很多。在組語裡這些選擇是**你**做的；在高階語言裡，這些選擇被外包給了編譯器。

於是那個要命的問題第一次浮現：**機器自己做的選擇，會不會比人手做的差？**

在這之前，有兩個重要的中間站，把「說人話」這個方向先試了水溫。

**Grace Hopper 與 A-0（1951–52）。** Hopper 在 UNIVAC I 上做了一套叫 A-0 的系統，常被稱為「第一個編譯器」。這個說法得校準。A-0 做的事，以現代定義來看，更接近**連結載入器**（linker / loader）：它把一串編了號的副程式按你給的序列串接、載入記憶體，而不是把一種高階語言翻譯成機器碼。所以「A-0 是不是第一個編譯器」這個問題，**答案完全取決於你怎麼定義「編譯器」**——如果「編譯器」指「自動把人寫的東西組裝成可執行程式的工具」，A-0 算；如果指「把高階語言語句翻成機器碼」，A-0 不太算，那個榮譽更接近 FORTRAN。本書的立場：誠實地說 A-0 是這條路上的重要先驅，但別用「第一個編譯器」這頂帽子蓋過定義的模糊。（這也是本書反覆要做的事：凡「誰第一個發明 X」的歸屬，先問「X 怎麼定義」。）

Hopper 後來（1955–59）做了 **FLOW-MATIC**（原名 B-0），第一個用英語化語句寫資料處理的編譯語言，直接影響了後來的 COBOL（見 ch04）。她真正的賭注不在「快」，在「**讓不是數學家的人也能寫程式**」——這條「可讀性、英語化」的血脈，跟 FORTRAN 的「給科學家算數值」是兩種不同的價值取向，ch04 會把這兩條血脈攤開。

順帶澄清一個你大概聽過的故事。1947 年，Harvard Mark II 的繼電器裡夾死了一隻蛾，工程團隊把牠貼進日誌，旁邊寫 "First actual case of bug being found"。這常被講成「Hopper 發明了 debug 這個詞」——**這是流傳的故事，不是詞源**。"bug"（小故障）和 "debug" 這兩個詞在那之前工程界早就在用了；那句日誌之所以好笑，正是因為他們在玩這個既有的雙關（這次真的是隻蟲）。Hopper 確實在場、也樂於講這個故事讓它流傳，但她沒有發明這個詞。歷史的趣味往往就在這種「比傳說無聊一點、但更準確」的版本裡。

## 個案：FORTRAN 團隊怎麼贏下這場賭

現在進入這一章真正的主角。

1953–54 年，IBM 的 John Backus 提了一個計畫。動機很實際，是一筆帳：**寫程式的成本已經追上、甚至超過機器本身的成本**。一個電腦中心，請程式設計師的錢往往不比那台機器便宜；機器時間有四分之一到一半花在 debug 上。Backus 看到的不是「程式設計師不夠聰明」，是「人力是瓶頸，而且會越來越貴」。這個經濟論證很重要——它讓 Backus 拿得到資源組隊（IBM 想賣更多機器，而機器賣不動的原因之一是程式難寫）。

注意這個論證的形狀，因為它會在本書反覆出現：**當某種資源變便宜、另一種變貴，抽象的方向就會轉向去省那個貴的。** 1954 年，機器相對變便宜了（704 把浮點運算搬進硬體，算得飛快），人變貴了。於是「多花一點機器時間、省下大量人時」第一次划算。這正是你選 NestJS 的同一套算盤，只是把「機器時間」換成「雲帳單」、把「人時」換成「團隊速度」。鐘擺往「生產力」這邊擺，是因為帳算得過來，不是因為誰比較有遠見。

但賭注就在這裡。Backus 的團隊很清楚一件事，這件事決定了 FORTRAN 的生死：

> 大意是：如果 FORTRAN 產生的機器碼，跑得比一個好程式設計師手寫的明顯慢，那再好寫也沒人會用。
> （這是 1957 年原始論文與 Backus 後來回顧反覆強調的核心約束；此處為轉述，非逐字引文。）

這不是杞人憂天。當年的程式設計師裡有一種近乎「祭司」的文化——他們以能榨出機器每一滴效能為傲，把「機器自動產生程式」視為對這門手藝的冒犯，公開表達敵意與嘲諷。一個普遍的信念是：**機器不可能做到人手寫的那些精巧、有創意的最佳化。** 在這種氛圍下，一個「好寫但慢」的編譯器只會證明懷疑者是對的，然後被丟進歷史的垃圾桶。

所以 FORTRAN 團隊把絕大部分力氣，不是花在「好寫」，是花在「**夠快**」。他們做的，是世界上第一個**最佳化編譯器**（optimizing compiler）。這場仗打了大約三年、約 18 個人年（person-years），整個編譯器約兩萬三千多條組語指令——以當年的尺度是巨大的工程。產品 1957 年 4 月出貨給 IBM 704 的用戶。

最值得攤開的，是他們**怎麼**讓產出的碼夠快。編譯器分成六個 section，最精彩的是第 4 與第 5 section 的配合：

```text
FORTRAN I 編譯器：兩個 section 怎麼合謀產生快碼

  你的程式
     │
     ▼
  [Section 4]  Lois Haibt 主導
     │  把程式切成 basic block（一段沒有分支進出的直線碼）
     │  用 Monte Carlo「假跑」一遍程式，估計每個 block
     │  多常被執行——熱的地方標記出來
     ▼
  [Section 5]  暫存器/索引配置
     │  把寶貴的索引暫存器，優先分給最熱的 block
     │  熱路徑上的變數留在暫存器，冷路徑才肯寫回記憶體
     ▼
  夠快的機器碼
```

停下來看清楚這件事的份量。**1957 年，這群人就在做「先估計程式哪裡熱、再把資源砸在熱的地方」。** Section 4 由 Lois Haibt 負責：她讓編譯器用蒙地卡羅（Monte Carlo）的方式「模擬執行」程式——遇到 `IF` 和分支，用加權的亂數決定走哪邊——藉此統計出哪些 basic block 是高頻熱區。Section 5 再拿這份熱度地圖，把數量有限的索引暫存器優先配置給最熱的迴圈。

你應該覺得這個套路很眼熟。**這就是 profile-guided optimization（PGO）的祖先**，比它正式成為主流早了幾十年。你今天做效能工程的核心動作——先 profiling（Clinic.js 找熱點）、再把優化力氣集中在那 3% 的瓶頸上——跟 1957 年 Haibt 做的事，是同一個思想：別平均用力，先量出熱在哪，再針對熱區下手。當年沒有 profiler 可跑（程式還沒寫完呢），所以她讓編譯器用統計**模擬**出一份熱度估計。換成今天的話：靜態 PGO 的史前版本。

賭贏了。FORTRAN 產生的碼，跑得**接近**手寫機器碼的水準——好到連團隊自己的人都驚訝。據說這個最佳化編譯器產生的碼，效率紀錄保持了大約二十年才被超越（這類「保持紀錄 N 年」的說法是業界流傳的讚譽，難以精確查證，當佐證、別當定論）。重點不在那個確切數字，在於：**懷疑者輸了。** 「機器產生的碼必然比人慢」這個信念，被一個能估熱度、配暫存器的編譯器當場推翻。

從這一刻起，鐘擺的位置被重新校準。不是「效能 vs 生產力，只能選一個」，而是「**在大多數情況下，你可以兩個都要**——把效能的苦工外包給一支夠聰明的編譯器，省下的人力遠大於損失的那點機器時間」。你今天能安心地不看 V8 產出的組語，就是站在這個被校準過的鐘擺上。

但請記住「大多數情況」這五個字。FORTRAN 沒有消滅效能的問題，它把效能的責任**搬家**了——從「每個程式設計師逐行負責」搬到「編譯器團隊集中負責」。這筆帳，下一節結。

## 複雜度搬去哪了

把這一章的兩跳結成一筆帳。

| | 消滅了什麼負擔 | 製造了什麼新負擔 | 在哪洩漏／反噬 |
|---|---|---|---|
| **第一跳：組語** | 位址手算、改一行牽動全表 | 多了一道組譯步驟；得信任組譯器算對位址 | 幾乎不漏（一對一，產出可預測） |
| **第二跳：高階語言** | 把「想法拆成機器動作」整包外包；暫存器調度、最佳化都交出去 | 你**看不到也控制不了**機器碼長怎樣；效能交給編譯器決定 | 編譯器選錯最佳化、或你的寫法讓它無法最佳化時，效能莫名變差，而你離產出的碼隔了好幾層 |

這一跳的帳單，一句話：**你用「不必懂機器碼」換來生產力，代價是當效能出問題時，你和真相之間多了一個你不完全理解的黑盒。**

這個帳單你天天在付，只是平常它不來收。它來收的時候長這樣：

- 你寫了一段 TypeScript，跑起來莫名其妙慢。問題不在你的邏輯，在 V8 因為你的物件形狀（hidden class）一直變，放棄了某個最佳化、把函式 deopt 回慢路徑。你得懂一點「編譯器/JIT 在背後怎麼決定要不要最佳化」，才追得到。這就是**抽象洩漏**（見 ch19）——平常編譯器替你藏好的機器細節，在效能的邊緣漏了回來。
- 你信任 `tsc` 的型別檢查，但 `tsc` 編出來的 JS 不帶任何型別保護，執行期一個 `any` 穿進來照樣炸。編譯器幫你擋的是一類錯，不是所有錯（型別這條線，見 ch08）。

注意一個全書會反覆出現的模式：**抽象不消滅複雜度，它把複雜度搬到一個你看不見的地方，並指定一群人專門替你扛。** 機器碼的最佳化複雜度沒有消失——它從「每個寫程式的人」搬到了「編譯器團隊」。對你是淨賺：你不用懂暫存器配置，FORTRAN 團隊（和今天 V8、LLVM 的人）替全世界懂了一次。代價是當那個黑盒在邊緣出錯，你得有能力暫時跳進去——這正是資深工程師值錢的地方（見 ch19）。

還有一個鐘擺的讀法，記著它，後面每一章都會用到。「效能↔生產力」不是一次性擺到「生產力」就停了。它一直在擺：

```text
效能 ◄─────────────────────────► 生產力
  手寫機器碼        FORTRAN(1957)        高階語言/GC語言
  你手寫 C 服務     你選 NestJS          你用 ORM 不寫 SQL

  每次往「生產力」擺，都靠「夠聰明的某個層替我扛效能」這個賭
  賭的前提：瓶頸不在這層、損失可接受——前提不成立時，鐘擺就回擺
```

什麼時候回擺？當「夠聰明的層」不夠聰明、或抽象的損失大到吃掉了生產力的好處時。你在 hot path 上手寫一段繞過 ORM 的原生 SQL，就是一次小回擺。整個產業也會回擺——FP 因為多核心回來（見 ch09）、有人從微服務退回單體（見 ch17、ch19）。每一次回擺都帶著上次的教訓。FORTRAN 是這條鐘擺有記載的第一次大擺動，方向定了調：只要那個賭算得過帳，人類就會一次又一次把更多細節外包出去。

## 紙上推演

### 推演一：手寫 SQL → ORM，這個類比在哪裡會破 [15 分鐘] ★★

把「組語 → 高階語言」這一跳，類比成你熟悉的「手寫 SQL → 用 ORM」。先講清楚類比成立的地方，再**主動找出**它會破的地方。

#### 推演解答

**成立的地方**：兩者都是「把一個低階、囉嗦、要懂底層細節的東西，外包給一支程式自動產生」。手寫 SQL ↔ 手寫一條條機器動作；ORM 的 `user.posts` ↔ 高階語言的一行式子。兩者都做了「一對多展開」：你寫一行 ORM，它可能展開成多條 SQL；你寫一行 FORTRAN，它展開成多條機器指令。而且兩者都引發**同一種懷疑**——「自動產生的會不會比手寫的差」。ORM 的 N+1 query，正是「自動產生的 SQL 比手寫的慢」這個古老懷疑的當代版本。

**會破的地方**（這才是重點）：

1. **FORTRAN 的賭最後贏了，ORM 的賭只贏了一半。** FORTRAN 的最佳化編譯器讓產出碼接近手寫；ORM 的查詢產生器在簡單情況夠好，但碰到複雜 join、N+1、需要特定 index hint 時，它產生的 SQL 經常明顯比手寫差。差別在於：機器碼的最佳化是個**封閉、可窮盡**的問題（暫存器就那幾個、指令就那些），編譯器團隊能集中解決一次；而 SQL 的最佳化牽涉資料分布、統計、執行計畫，跨出了 ORM 能掌握的邊界。所以你常常得「跳下去」手寫原生 SQL——這個動作在 FORTRAN 之後幾乎沒人對機器碼做了。
2. **層數與可見性不同。** 你能看到 ORM 產生的 SQL（打開 query log 就有），所以洩漏時你抓得到、改得動。FORTRAN 產生的機器碼你理論上看得到，但實務上沒人會去逐條改它——抽象封得更死。
3. **誰扛最佳化責任不同。** 編譯器的最佳化是全自動、你無法介入的；ORM 通常留了逃生口（raw query、query hint），等於它**承認**自己這個賭不一定贏，先給你留了回擺的門。

一個好的類比，價值往往在它破的地方——破的地方告訴你「這兩個抽象賭贏的程度不一樣」。

### 推演二：口頭講「FORTRAN 的賭」與「你選框架」是同一條鐘擺 [10 分鐘] ★

對著一個資淺工程師，用不超過三分鐘把這件事講清楚：1957 年 FORTRAN 團隊下的賭，跟你上週選 NestJS 而不是手寫 C 服務，是鐘擺上的同一個動作。不准用「抽象」「鐘擺」這種詞——用大白話。

#### 推演解答

一個可用的講法（你的版本只要命中因果即可）：

「1957 年以前，寫程式就是直接對機器下指令，每一步都你親手排，所以程式跑得快、但人累得半死、而且超容易出錯。後來有群人在 IBM 想：人越來越貴、機器越來越便宜，不如讓我寫一行接近數學的式子，由一支程式幫我翻成那一堆機器指令。當時一堆高手嘲笑這個想法，說機器翻出來的碼一定比我手寫的慢。結果這群人做了一個很會偷偷最佳化的翻譯程式——甚至會先猜程式哪裡最常跑、把力氣集中在那——翻出來的碼快到跟手寫差不多。於是大家發現：原來不用在『好寫』跟『夠快』之間二選一，多數時候可以兩個都要。

我上週選 NestJS 不手寫 C，做的是一模一樣的決定。手寫 C 會更快，但團隊維護成本高、開發慢。我賭的是：NestJS 那點 overhead 反正不是我們的瓶頸（瓶頸在資料庫），用它換來的開發速度划算。這個『用一點效能換生產力、賭損失可接受』的決定，1957 年那群人第一次正式下注。我只是那個賭的第六十幾年受益人。」

**常見誤解**：講的時候容易把它講成「進步論」——好像抽象越高越好、歷史單向前進。要避開。重點是**這是一筆帳，不是進步**：往生產力擺，是因為當下的成本結構讓帳算得過來；當瓶頸真的落在這一層（你手寫原生 SQL 那次），帳就反過來，你會往效能回擺。鐘擺，不是階梯。

### 推演三：A-0 是不是「第一個編譯器」，取決於你怎麼定義 [15 分鐘] ★★★

有人說 Grace Hopper 的 A-0（1952）是第一個編譯器，也有人說第一個編譯器是 FORTRAN（1957）。寫一段，說清楚**為什麼這個爭論的答案完全取決於「編譯器」怎麼定義**，並給出在哪個定義下誰算。

#### 推演解答

關鍵：「第一個 X」這種問題，幾乎都不是史實之爭，是**定義之爭**。把「編譯器」攤成兩種合理定義：

**定義 A：「自動把人寫的高階描述，組裝／轉換成可執行程式的工具」（寬定義）。** 在這個定義下，A-0 算得上——它把一串編了號的副程式按指定序列串接、載入成一支可跑的程式，確實自動化了「人手組裝程式」這件事。Hopper 自己用 "compiler" 這個詞，取的就是「compile＝把零件編組在一起」的本義。

**定義 B：「把一種高階語言的語句，翻譯成較低階語言（機器碼/組語）的工具」（現代定義）。** 在這個定義下，A-0 比較不算——它做的事更接近**連結載入器**（linker / loader），把預先寫好的副程式串接，而不是去「翻譯」一種語言的語法結構。符合這個定義的、第一個被廣泛使用的，是 FORTRAN：它把 FORTRAN 語言的式子翻成 IBM 704 的機器碼。

所以正確的講法不是「A-0 是／不是第一個編譯器」，而是：

> A-0 是這條路上不可抹滅的先驅，它示範了「程式可以自動組裝程式」；但若用今天「把高階語言翻成機器碼」的標準，它更接近連結載入器，那個意義下的第一個廣用編譯器是 FORTRAN。

**常見誤解一**：以為這是在「貶低 Hopper」。不是。誠實標明定義的模糊，比塞給她一頂尺寸不對的帽子更尊重她——她真正的、無爭議的貢獻是「英語化、讓更多人能寫程式」這條血脈（FLOW-MATIC → COBOL）。**常見誤解二**：以為一定有個「正確答案」。歷史的優先權之爭，多數時候沒有正確答案，只有「在哪個定義下成立」。本書遇到每一個「誰第一個發明 X」，都先問這句：你說的 X，是哪個定義？

## 自我檢核

口頭自答，講不清楚就是還沒懂：

1. 機器碼世界裡，程式設計師腦中扛的三筆負擔是什麼？為什麼「改一行牽動全表」比另外兩個更致命？
2. 組語（第一跳）跟高階語言（第二跳）外包掉的東西，本質差別在哪？為什麼第一跳沒人懷疑、第二跳才需要賭？（提示：一對一 vs 一對多）
3. Backus 為什麼做 FORTRAN？把那筆「人力 vs 機器」的帳講出來——並說清楚這筆帳跟你選 NestJS 的帳是同一個形狀。
4. FORTRAN 團隊把力氣主要花在「好寫」還是「夠快」？為什麼這個選擇決定了 FORTRAN 的生死？
5. Lois Haibt 的 Section 4 用 Monte Carlo 做了什麼？為什麼說這是 profile-guided optimization 的祖先、跟你用 Clinic.js 做效能工程是同一個思想？
6. 「A-0 是第一個編譯器」這句話要怎麼校準？為什麼答案取決於定義？
7. 用「複雜度搬家」講清楚：FORTRAN 消滅了什麼、製造了什麼、這個帳單什麼時候會來找你？
8. 「效能↔生產力」這條鐘擺，為什麼說它「不是階梯」？舉一個你自己回擺（往效能那邊）的真實例子。

## 延伸閱讀

- **John Backus,《The History of FORTRAN I, II, and III》**（HOPL, 1978）——當事人寫的一手史料，講團隊怎麼想、怎麼賭、怎麼做最佳化。讀它對「效能懷疑」與 Section 4/5 的描述。softwarepreservation.computerhistory.org 有全文 PDF。
- **Backus et al.,《The FORTRAN Automatic Coding System》**（1957 原始論文）——編譯器的原始技術描述，六個 section 的分工在此。要看 Monte Carlo 頻率估計與暫存器配置的原文讀這篇（Stanford 有掃描版）。
- **The New Stack,〈How John Backus' Fortran Beat the Machine Code 'Priesthood'〉**——把「祭司文化」的敵意與 Backus 如何贏過它講得生動，適合先讀建立畫面感（二手整理，史實以上面兩篇一手為準）。
- **EDSAC Initial Orders 相關**（Cambridge Computer Lab，Martin Richards 的 EDSAC 資料）——想看「第一個組譯器」長什麼樣、Wheeler 怎麼用 31 個字做出組譯＋載入，看這份。
- **編譯器原理本身**（lexer / parser / codegen 怎麼運作）不在本書範圍——若想補，經典是「龍書」（Aho et al.,《Compilers: Principles, Techniques, and Tools》），但那是另一條路，本書只關心「這個賭外包了什麼複雜度」。
