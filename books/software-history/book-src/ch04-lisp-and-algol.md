## 從你已知的出發

打開你任何一個 NestJS 專案，隨手挑一段你天天寫、卻從沒想過「這是誰發明的」的程式碼：

```ts
const handlers = routes.map((route) => {
  const cache = new Map();              // 你不會去 free 它
  return async (req) => cache.get(req.id) ?? await fetch(route, req);
});
```

這短短三行裡，住著兩個來自 1960 年的幽靈。

第一個幽靈是**那個傳進 `map` 的箭頭函式**——一個可以被當成值傳來傳去、還記得外層 `cache` 變數的東西（closure）。你大概覺得「函式是一種值」天經地義，就像你覺得水是濕的。它不是天經地義。它是 1958 年 McCarthy 為了把數學裡的函數搬進機器，刻意做的設計決定。你寫的每一個 callback、每一個 `.map().filter().reduce()`、每一個高階函式，靈魂都來自 Lisp。

第二個幽靈更隱形：**你建了一個 `Map`，卻從來不去回收它**。你信賴一件事——當這個 closure 沒人用了，那塊記憶體會自己消失，你不必寫 `free(cache)`，也不必擔心忘了寫。這份「不必記得 free」的奢侈，是這一章的個案主角。它有名字，叫 garbage collection，1959 年由 McCarthy 為了讓 Lisp 能自由配置記憶體而發明。而你調過的那次 Node.js 記憶體爆掉、盯著 `--max-old-space-size` 和 GC pause 看半天的下午——那就是這份奢侈的帳單，六十多年後寄到你信箱。

至於你那行 `route` 怎麼長成 `routes.map(...)` 這種「大括號分塊、用縮排表達結構、控制流一層套一層」的樣子——那是**第三條血脈**。C 系語法（`{ }`、`if/while`、巢狀區塊）不是 C 發明的，C 是繼承來的。源頭是 Algol 60，一個你大概沒寫過一行、卻決定了你每天敲的每個大括號的語言。

所以你的 JavaScript 是一場混血：**Algol 後裔的骨架（C 系語法、block structure），裝了 Lisp 的靈魂（first-class function、closure、GC）**。這一章要講的，就是這兩條血脈當年各自在賭什麼、為什麼至今還在你身上打架，以及為什麼它們旁邊那條看起來最土的血脈（COBOL）反而活得最久。

## 三條路線：語言第一次需要「相信什麼」

ch03 講過 FORTRAN（1957）的賭注：機器產生的碼能不能跟人手寫的一樣快。那是一場關於**效能**的賭。賭贏之後，門打開了——既然高階語言可行，那麼「高階」到底該長什麼樣？1958 到 1960 短短三年，三個答案幾乎同時出現，各自押在不同的價值上：

| 語言 | 年份 | 押注的價值 | 世界觀 | 留給你的遺產 |
|---|---|---|---|---|
| **LISP** | 構思 1958、論文 1960 | 數學的優雅、表達力 | 程式和資料是同一種東西 | first-class function、closure、GC、REPL、AST 操作 |
| **ALGOL 60** | 1960 報告 | 結構的嚴謹、可被精確描述 | 語言該有數學般精確的語法定義 | block structure、C/Pascal/Java 的語法骨架、BNF |
| **COBOL** | 規格 1959、正式 1960 | 商業的可讀、可維護 | 程式該像英文，讓非數學家也能讀 | 它本身——以及「遷移成本」這堂課 |

注意這張表的重點不是年份，是**它們第一次在「相信什麼」上分了岔**。FORTRAN 只想跑得快，沒空管哲學。這三個有空了。它們定義了往後六十年語言設計的價值光譜：一端是數學家的優雅（Lisp），一端是工程師的結構（Algol），一端是會計師的可讀（COBOL）。你今天選 TypeScript 而不選 Haskell、選 Go 而不選 Clojure，本質上還是在這條光譜上選邊。

## Lisp：當程式發現自己也是資料

先說背景，因為背景就是動機。1950 年代末，McCarthy（John McCarthy）在 MIT 搞的是**人工智慧**——他要讓機器做符號推理、下棋、證明定理。這類問題有個共同特徵：你事先不知道資料會長多大、什麼形狀。一棵推理的樹可能長三層也可能長三百層，一份符號運算式可能五個符號也可能五萬個。FORTRAN 那種「開頭宣告固定大小陣列」的世界對這種工作毫無用處——你怎麼可能事先知道要配多大？

所以 McCarthy 需要一種能**動態長出結構、自由連接、隨意拆開**的資料表示。他的答案是**list**（串列），由一格一格的 **cons cell** 串成——每格裝一個值、一個指向下一格的指標。需要更大就再配一格，需要分支就指出去。LISP 這名字就是 **LISt Processor**。

到這裡都還只是一種資料結構。真正的躍遷是下一步，而這一步是軟體史上最漂亮的一招之一。

### 程式即資料：homoiconicity

McCarthy 做了一個決定：**程式本身，也用 list 來寫**。

一個函式呼叫 `(+ 1 2)`，在 Lisp 裡就是一個三元素的 list：符號 `+`、數字 `1`、數字 `2`。它既是「把 1 和 2 相加」這段**程式**，又是「一個裝著三個東西的串列」這份**資料**。同一個東西，看你怎麼對待它。這個性質後來有個拗口的名字叫 **homoiconicity**（同像性，code as data），但概念極簡單：**在 Lisp 裡，程式和資料是同一種東西，所以程式可以像捏黏土一樣捏程式**。

這對你意味著什麼？意味著 Lisp 程式可以讀、改、生成另一段 Lisp 程式，因為「另一段程式」對它來說只是一個 list，跟它平常處理的任何 list 沒兩樣。你做過 codegen（你的 OpenAPI schema 自動生出 client）、你用過 AST 工具（Babel、TypeScript compiler API、ESLint 的 rule）去分析或改寫程式碼——你做這些事的時候，得先把原始碼**解析**成一棵抽象語法樹（AST），在那棵樹上動手腳，再**印**回字串。你是在「程式」和「資料」之間來回翻譯，因為在你的語言裡它們本來是兩種東西。

Lisp 不用翻譯。**Lisp 的原始碼本來就已經是那棵樹**。括號就是樹枝。這就是為什麼 Lisp 的 macro 系統威力大到嚇人——macro 不是字串替換（像 C 的前置處理器那種危險把戲），而是直接拿一段程式（一個 list）當輸入、回傳另一段程式（另一個 list）。你在 JavaScript 裡得架整套 Babel plugin 基礎設施才能做的事，Lisp 把它做成了語言的內建能力。

我認為這是理解 Lisp 為什麼六十年來一直被「重新發現」的關鍵：每隔一陣子就有人撞上「我需要程式來操作程式」的需求（template engine、ORM 的 query builder、React 的 JSX、各種 DSL），然後慢慢重新發明一個比較笨的 Lisp。Greenspun 那句著名的酸話「任何夠大的 C 或 Fortran 程式，都包含一個臨時、未明確規定、滿是 bug、跑得很慢、只實作了一半的 Common Lisp」——講的就是這件事。

### eval：當紙上的數學被人偷偷編成了機器碼

這是本章必講的故事，因為它把「程式即資料」這個抽象概念，變成了一個有血有肉的歷史時刻。

McCarthy 在 1960 那篇論文〈Recursive Functions of Symbolic Expressions and Their Computation by Machine, Part I〉裡，為了**證明** Lisp 的表達力夠強，寫了一個叫 `eval` 的函式：一個能讀進任何 Lisp 運算式、並算出它結果的 Lisp 函式。換句話說，他用 Lisp 寫了一個 Lisp 直譯器，當作一份數學上的存在性證明——「看，這語言強到可以描述自己」。在他心裡，這純粹是理論，是寫給人讀的，是黑板上的東西。

然後他的研究生 **Steve Russell**（史提夫·羅素）讀了這份論文，做了一件 McCarthy 沒料到的事：他發現紙上這個 `eval` 可以**直接手工編成 IBM 704 的機器碼**。McCarthy 自己後來的回憶大意是——羅素說「我來把這個 eval 寫成程式吧」，McCarthy 笑他「ho, ho，你把理論跟實務搞混了，這個 eval 是寫來讀的，不是拿來算的」。但羅素不管，真的把論文裡的 eval 一行一行編成機器碼、修掉 bug，然後宣布這是一個 Lisp 直譯器——而它確實就是。（此事為 McCarthy 本人多次轉述，史料一致。）

這個故事為什麼重要？因為它是「程式即資料」的活體示範。一個本來只是**寫在紙上描述計算的東西**（資料、規格、證明），被當成**程式**跑了起來。理論和實作之間那道牆，被 homoiconicity 這個設計直接捅穿了。順帶一提，這也是世界上第一個 REPL 的雛形——你在 Node.js 終端機敲 `node` 進去那個互動環境、即打即評估的體驗，血緣可以一路追到羅素 1959 年那次「不聽勸」。

## 個案：garbage collection 的誕生——把「記得 free」永久外包

現在來到本章的 worked example，也是你最有切膚之痛的一筆帳。

回到 cons cell。McCarthy 的 Lisp 要做符號推理，就得不停地配置新的 cons cell——每長出一個串列節點配一格，每做一次運算可能產生一堆中間結構。問題立刻來了：**這些 cell 用完之後怎麼辦？**

在 1959 年，標準答案是「程式設計師自己負責」。你配置（allocate）了記憶體，你就得記得在用完時歸還（free）。FORTRAN、組語、後來的 C，全都是這套規矩。但 McCarthy 的工作有個要命的特性：**串列結構會互相共享、互相指來指去**。同一個 cons cell 可能被三個不同的串列同時指到。這時候「誰該負責 free 它」根本算不清——你 free 早了，另外兩個串列就指向一塊被回收的廢墟（懸空指標）；你 free 晚了或忘了，記憶體就慢慢漏光。在一個會自動長出任意形狀結構的系統裡，要人腦去追蹤「每一格現在還有沒有人在用」，是不可能的任務。這就是 McCarthy 撞上的那面牆：**動態、共享的資料結構，讓手動記憶體管理在認知上破產**。

他翻越這面牆的方式，是發明 **garbage collection**：讓 runtime 自己定期掃描，找出「從程式還搆得到的」所有 cell（從根開始追指標，能走到的就是活的），其餘搆不到的就是垃圾，整批回收。程式設計師再也不必寫 `free`——你只管配置，回收是 runtime 的事。這個概念第一次被描述，就在 1960 那篇 Lisp 論文裡，只佔了一頁多，講的是後來人人都學過的 mark-and-sweep。

停下來體會一下這一跳的份量。**McCarthy 把「記得 free」這件事，從每一個程式設計師的腦袋裡，永久搬進了 runtime。** 這是你今天能寫出本章開頭那段「建個 Map 卻從不回收」程式碼的全部原因。你寫 JavaScript、Java、Python、Go——你信賴 GC，你的整個職業生涯幾乎沒手動 free 過記憶體，這份信賴的源頭就是 1959 年那群在 MIT 配置 cons cell 的人。

但這是一本講「複雜度守恆」的書。複雜度沒有消失，它搬家了。下一節結帳。

## Algol：語言的語言

如果說 Lisp 押的是「表達力」，Algol 60 押的是「精確」。而且 Algol 的影響力有個弔詭之處：**幾乎沒人用它寫過正式產品，它卻是你今天用的幾乎每個語言的祖宗**。

Algol（ALGOrithmic Language）60 是一個國際委員會的產物，1960 年在巴黎定案，編輯是 **Peter Naur**（彼得·諾爾）。它的目標很學術：做一個乾淨、通用、能精確描述演算法的語言。它帶來了兩樣東西，至今活在你每一行程式裡。

### Block structure：你的大括號從哪來

第一樣是 **block structure**——用 `begin ... end`（後來在 C 系變成 `{ ... }`）把一群陳述句框成一個區塊，區塊可以巢狀，變數有**語彙作用域**（lexical scope，在哪個區塊宣告就只在那個區塊裡有效）。

這聽起來平淡到不值一提，因為你從寫程式第一天起就活在這個世界裡。但在 Algol 之前，控制流主要靠 GOTO 跳來跳去（這條線會在 ch05 軟體危機那章爆發成大事），變數多半是全域的。Algol 說：不，程式有**結構**，結構可以巢狀，而且這個結構應該直接反映在你寫的程式長相上——縮排、巢狀、區塊。你 ESLint 設的 `max-depth`、你 code review 時嫌人家巢狀太深要求早 return——你管的那個「深度」，正是 Algol 在 1960 年發明的概念。你寫的每一對大括號，都是 Algol 的後代。C 從 Algol/BCPL 一脈繼承了它，C 又生了 C++、Java、JavaScript、Go……你的 NestJS 那層層 `{ }`，族譜直通巴黎那場會議。

### BNF：用一個語言去描述另一個語言

第二樣更深，也是 Algol 最被低估的貢獻：**BNF（Backus–Naur Form）**。

問題是這樣的：你要怎麼**精確地**告訴別人（和編譯器）「這個語言的語法長什麼樣」？在 Algol 之前，語言的語法都是用自然語言含混描述的——「一個運算式可以是一個數字，或者兩個運算式中間加個運算子」——這種描述充滿歧義，不同人實作出來的編譯器行為都不一樣。

BNF 是一套**用來描述語法的形式記號**：你用一組規則（grammar）精確定義「什麼樣的字串才是合法的程式」。它最初由 John Backus 為描述 Algol 提出、Peter Naur 在 Algol 60 報告裡改良採用，所以叫 Backus–Naur Form。這是史上第一次，一個程式語言的語法被用**數學般的嚴謹**寫下來，而不是靠自然語言比手畫腳。

這件事的份量你每天在吃它的紅利卻看不見它。任何 parser generator（你可能用過或聽過的 ANTLR、Yacc、PEG）、任何語言的官方語法規格、你 TypeScript compiler 背後那套把原始碼切成 token 再組成 AST 的機器——全都站在 BNF 上。「用一個形式系統去精確描述另一個語言的結構」這個想法，是 Algol 留給整個 CS 的工具。注意這裡的對稱性：Lisp 讓**程式**變成可操作的資料；Algol（透過 BNF）讓**語言的語法**變成可精確描述的形式對象。兩條血脈都在做「把某種原本含混、靠人腦掌握的東西，變成可被機器精確處理的形式」——只是下手的層次不同。

### Hoare 的悼詞

Algol 60 商業上不算成功（太學術、IBM 沒大力推、I/O 規格留白），但它在語言設計圈的地位近乎神話。最常被引的評價來自 **C. A. R. Hoare**（東尼·霍爾，後來發明 Quicksort、Hoare logic 的那位），在他 1980 年的圖靈獎演說〈The Emperor's Old Clothes〉（刊於 *CACM* 1981）裡論 Algol 60：

> 「Here is a language so far ahead of its time that it was not only an improvement on its predecessors but also on nearly all its successors.」
> （這是一個遠遠超越其時代的語言，它不只勝過它的前輩，也勝過它幾乎所有的後繼者。）

這句話之所以辛辣，在於「也勝過它幾乎所有的後繼者」——Hoare 是在說，後來那些更紅、更實用的語言（包括把 null 引進來、被他自己稱為「十億美元錯誤」的那些設計，見 ch08），有些地方反而**退步**了。一個遠超時代的設計，可能要等幾十年，後人才慢慢追回它已經想清楚的東西。這個「好點子提早出現、然後被市場繞過、幾十年後才被重新需要」的模式，你會在這本書裡反覆看到（函數式的回歸是最戲劇化的一次，見 ch09）。

## COBOL：複雜度搬不動的活標本

第三條路線，和前兩條的氣質完全相反。Lisp 是數學家寫給數學家的，Algol 是電腦科學家寫給電腦科學家的，**COBOL（COmmon Business-Oriented Language）是工程師寫給「不是工程師的人也要看得懂」的**。

COBOL 規格 1959 公布、1960 正式，出自 CODASYL 委員會。**Grace Hopper**（葛麗絲·霍普）透過她先前的 FLOW-MATIC 語言深刻影響了它的設計方向——注意是「深刻影響」，不是「唯一作者」，COBOL 是委員會的集體產物。它的核心信念是：**程式碼應該像英文一樣可讀**，讓商業領域的人（而不只是寫程式的人）能看懂、能稽核。所以你會看到 COBOL 寫成這樣：`MULTIPLY HOURS BY RATE GIVING GROSS-PAY.`——幾乎是一句英文。

在 1960 年，這是個合理的賭。當時程式設計師稀缺、商業電腦化正要起飛，「讓更多人能讀程式」直接對應到真金白銀的成本。Lisp 押表達力、Algol 押結構、COBOL 押**可讀與可維護**——三條路線各自押在語言設計價值光譜的一端，沒有誰天生比較蠢。

但 COBOL 在這本書裡的角色，不是它的設計，而是**它的壽命**。它活了六十多年。直到今天（2026-06），全球大量銀行、保險、政府的核心系統，底層還在跑 COBOL。

這裡要極度小心地處理數字。你會在網路上讀到各種驚人統計——「全球還有兩千多億行 COBOL 在跑」「43% 的銀行系統」「每天處理數兆美元交易」。**這些數字幾乎全部源自單一來源（多半輾轉自一份 Reuters 的估計），彼此還互相矛盾**（我查證時就同時看到「2,200 億行」和「8,000 億行」兩種版本）。所以本書的立場是：**COBOL 至今仍大量運行於金融與政府核心系統，這件事是真的；但任何具體行數、市佔、交易額都只是某個單一來源的估計，不可當確切事實引用**。重點不在數字多大，在於「它就是搬不動」這個現象本身。

那麼問題來了：COBOL 既不優雅（Lisp 笑它土）也不嚴謹（Algol 嫌它鬆），為什麼偏偏是它活得最久？

**答案就是這本書的核心主張：複雜度守恆，而 COBOL 系統裡的複雜度，搬不動。**

一個跑了四十年的銀行核心系統，它的價值早就不在那幾百萬行 COBOL 程式碼本身，而在**那些程式碼裡固化的業務邏輯**——四十年來無數監管規則、邊界案例、特殊處理、「為什麼這裡要這樣算」的決定，層層疊疊堆進了程式裡，而且**大多沒有寫進任何文件**。寫程式的人退休了、走了、過世了，知識只剩在程式碼裡。你想把它遷移到「現代」系統，等於要先把這四十年的隱性知識**逆向工程**出來——而你連需求是什麼都搞不清楚（這正是 ch10 會講的「本質複雜度」：搞清楚到底要建什麼，這部分的複雜度任何工具都砍不掉）。業界對大型 legacy 重寫的失敗率有各種估計，普遍偏高；澳洲一家銀行的核心遷移傳出耗時五年、花掉十億澳幣量級的成本（單一來源，數字僅供示意）。

所以 COBOL 不死，**不是因為它好，是因為遷移成本＝複雜度搬不動**。那些複雜度沒消失、也沒辦法搬到別的地方，它就卡在那裡，逼著一整個產業繼續餵養六十年前的語言。這是「複雜度守恆」最冷酷的版本：有時候複雜度不只是搬到別人看不見的地方，而是**重到誰也搬不動，只能繞著它蓋房子**。

你的職涯裡會遇到你自己的 COBOL——某個沒人敢動的服務、某段註解寫著「別碰，會壞，原因不明」的程式碼、某個遷移計畫年年列進 roadmap 又年年砍掉。當你遇到它，記得這不是哪個前人偷懶，這是複雜度守恆在小尺度上的重演。

## 複雜度搬去哪了

這一章其實搬了三次家，方向各不相同。把帳結清楚：

| 躍遷 | 消滅了什麼負擔 | 製造了什麼新負擔 | 在什麼情況下洩漏／反噬 | 這一跳的帳單 |
|---|---|---|---|---|
| **Lisp：程式即資料** | 「操作程式」不再需要在 code 與 data 兩種表示間翻譯（macro、REPL、自舉直譯器幾乎免費） | 一切都是括號的串列，可讀性與工具支援的代價；強大到容易自造方言、生態碎裂 | 當 macro 被濫用，程式變成「只有作者懂的私房語言」——彈性過頭反成複雜度 | 用「無結構的均質性」換來「無上限的表達力」 |
| **Lisp：garbage collection** | 程式設計師永遠不必再記得 `free`、不必再追懸空指標與記憶體洩漏 | 「何時回收」的決定權交給 runtime，你失去了對時機的控制 | GC pause（stop-the-world 停頓）、記憶體用量比手動管理高、在低延遲/即時場景會咬你 | 用「不可預測的延遲尖刺」換來「不必記得 free 的奢侈」——**這就是你那次 Node.js 堆積記憶體與 GC pause 的帳** |
| **Algol：block structure＋BNF** | 控制流不再是 GOTO 的義大利麵；語法不再靠自然語言含混描述 | 語言設計變成一門需要形式規格的嚴肅工程；門檻變高 | 規格與實作的落差（編譯器沒完全照 BNF 實作），以及過度結構化的儀式感 | 用「形式化的前期成本」換來「可精確描述、可自動解析的語言」 |
| **COBOL：複雜度搬不動** | （當年）讓非程式設計師也能讀程式、降低人力門檻 | 業務知識固化進程式碼、無文件、隨人員流失而失傳 | 六十年後遷移成本高到沒人敢動，整個產業被鎖死 | 用「四十年後的不可遷移性」換來「當年的可讀性」——複雜度重到搬不動 |

**這四筆帳裡，GC 那一筆是你最該記住的。** 它是「複雜度守恆」最乾淨的示範：手動 free 的負擔沒有消失，它從「你的腦袋」搬到了「runtime 的 GC 排程器」。大部分時候你享受著它（寫了六年程式幾乎沒 free 過記憶體），直到某個高負載的尖峰，GC 決定 stop-the-world、你的 p99 延遲突然爆了一根尖刺——這時帳單就回來找你了，你得跳進 V8 的堆積快照、調 GC 參數、改你的配置模式去減少垃圾產生。**抽象洩漏的時候，你得能跳到它底下**（這正是 ch19 的主題）。McCarthy 1959 年那個善意的決定，六十多年後變成你下午的 debug 工作——而這完全公平，因為那份奢侈你已經免費享受了六年。

## 紙上推演

### 推演題 1 — 把「程式即資料」對應回你的 codegen 經驗 **[20 分鐘] ★★**

你做過 OpenAPI schema → client 程式碼的自動生成，也用過 AST 工具（Babel／TS compiler API／ESLint rule）去分析或改寫程式碼。請口頭講清楚：（a）你做這些事時，原始碼在你手上經歷了哪幾種「形態」？（b）Lisp 的 homoiconicity 讓哪一步消失了？（c）為什麼這個差異讓 Lisp 的 macro 比你的 Babel plugin 更直接？

#### 推演解答

（a）在你的世界裡，原始碼經歷三種形態：**字串**（檔案裡的文字）→ **AST**（parser 把字串解析成的樹）→ 改完之後再**印回字串**（serialize/codegen）。你的工作永遠是「先把程式 parse 成資料，動完手腳再把資料 print 回程式」。OpenAPI codegen 也一樣：schema（資料）→ 你的生成器邏輯 → 吐出 client 程式碼（字串）。

（b）Lisp 讓 **parse 和 print 這兩步幾乎消失**。因為 Lisp 的原始碼**本來就已經是 list**（那棵 AST），不需要把字串解析成樹——括號就是樹。你拿到的程式，本來就是你平常處理的那種資料結構。

（c）你的 Babel plugin 要能跑，背後得有一整套基礎設施：lexer、parser、AST 節點型別定義、visitor pattern、再把改過的 AST 印回合法的 JavaScript。Lisp 的 macro 不需要這些——macro 就是一個「吃一段 list、吐一段 list」的普通函式，因為程式和資料是同一種東西。常見誤解是把 Lisp macro 想成 C 的 `#define` 那種字串替換；不是，C 的前置處理器在「字串」層次盲目貼字，不懂結構，所以危險；Lisp macro 在「結構（list/AST）」層次操作，安全得多。這就是 homoiconicity 的實際紅利：你在 JS 裡得自己搭一座橋（Babel）才能跨越的 code↔data 鴻溝，在 Lisp 裡根本不存在那條鴻溝。

### 推演題 2 — 用「複雜度搬不動」論證為什麼 COBOL 還沒死 **[15 分鐘] ★★**

有人對你說：「COBOL 又老又醜，這麼多年了銀行怎麼還不換掉？一定是那些老古板不思進取。」請用本書的「複雜度守恆／搬不動」框架，給一個比「老古板不思進取」更深的解釋。注意：論證裡如果要提到行數或金額，你必須怎麼說？

#### 推演解答

關鍵不在 COBOL 這個語言，在於**那些 COBOL 程式碼裡固化了什麼**。一個跑了四十年的核心系統，真正的資產不是程式碼，是四十年來層層堆進去的**業務邏輯**——監管規則、邊界案例、特殊處理、「為什麼這筆要這樣算」的無數決定，而且大多**沒有文件**，知識隨著原作者退休而流失。

要遷移，你得先把這四十年的隱性知識逆向工程出來——你連「需求到底是什麼」都搞不清楚。這部分的複雜度（搞清楚要建什麼）是**本質複雜度**（ch10 會正式定義），不是換個語言、換個框架就能砍掉的。所以複雜度不是搬到別處，是**重到搬不動**：它卡在原地，逼整個產業繞著它蓋房子。COBOL 不死不是因為它好，是因為**遷移成本＝複雜度搬不動**。

至於數字：你**必須** hedge。不能說「全球有 X 億行 COBOL」當定論，因為這類統計幾乎全來自單一來源、彼此矛盾（我查到的就有「2,200 億行」與「8,000 億行」兩種）。正確說法是「**據（單一來源）估計**，COBOL 仍大量運行於金融核心系統」——現象是真的，確切數字一律標為估計。這個自律本身就是一種專業：分得清「廣為流傳」和「經得起查證」。

### 推演題 3 — 反事實：如果 Algol 沒發明 BNF **[20 分鐘] ★★★**

假設歷史走了另一條路：Algol 60 沒有用 BNF，語言語法繼續像 1950 年代那樣用自然語言描述。請推演：（a）這會讓「實作一個編譯器」這件事變難在哪？（b）這對「同一個語言在不同機器上行為一致」有什麼影響？（c）把這個反事實接回本章主題——BNF 把哪一種複雜度從哪裡搬到了哪裡？

#### 推演解答

（a）沒有 BNF，語法只能靠自然語言描述（「一個運算式可以是一個數字，或兩個運算式中間加運算子」）。自然語言充滿歧義：邊界案例怎麼算？運算子優先順序到底誰先？巢狀深度有沒有上限？每個讀規格的人理解都不同，於是每個人實作出來的 parser 行為都不一樣。實作編譯器從「照著一套精確規則把 token 組成樹」退化成「猜原作者到底是什麼意思」。

（b）災難性影響：A 廠的編譯器和 B 廠的編譯器對同一段程式可能解析出不同結果，因為它們對含混的規格各有各的解讀。「同一個語言」名存實亡，每家實作都是方言。可移植性（ch03 講過這是高階語言的賣點）會嚴重打折。

（c）BNF 把「語法的歧義」這種複雜度，從**每個編譯器實作者各自的腦袋與猜測**，搬到了**一份所有人共享的、形式化的精確規格**。代價（帳單）是前期成本變高——你得先學會用形式記號精確描述語法，語言設計變成一門嚴肅工程而非隨手定義。但收益是：歧義被一次性消滅在規格層，下游所有實作者拿到的是同一份不會吵架的真相。這和 Lisp 用 homoiconicity 把「程式」變成可操作對象是同一種衝動的兩面——**把原本靠人腦含混掌握的東西，變成可被機器精確處理的形式**。

## 自我檢核

口頭回答，講得出因果才算過關（不是背年份）：

1. 為什麼說你的 JavaScript 是 Algol 和 Lisp 的混血？分別指出哪些東西來自哪一條血脈，並各舉一個你天天寫的具體例子。
2. 「程式即資料（homoiconicity）」具體是什麼意思？用 `(+ 1 2)` 同時是程式又是資料來解釋。它讓 Lisp 的 macro 比你的 Babel plugin 強在哪？
3. McCarthy 為什麼**需要**發明 garbage collection？（提示：從 cons cell 會共享、結構會動態長大講起，講到為什麼手動 free 在這種系統裡算不清。）
4. GC 是「複雜度守恆」的範例——手動 free 的負擔沒消失，它搬去哪了？帳單在什麼時候、以什麼形式回來找你？把它接回你調過的 Node.js GC pause。
5. BNF 解決了什麼問題？如果沒有它，「實作一個編譯器」和「語言的可移植性」會變難在哪？
6. Hoare 說 Algol 60「不只勝過前輩，也勝過它幾乎所有的後繼者」——這句話辛辣在哪？它暗示了一個本書反覆出現的模式，是哪個模式？
7. COBOL 又土又老，為什麼偏偏活得最久？用「複雜度搬不動」回答，而不是「銀行老古板」。當你被問到 COBOL 的行數或市佔時，你**必須**怎麼說、為什麼？
8. 這三條路線（Lisp 數學、Algol 結構、COBOL 商業）各押在「語言設計價值光譜」的哪一端？你今天選 TypeScript 而非 Haskell、選 Go 而非 Clojure，是在這條光譜上怎麼選的？

## 延伸閱讀

- **John McCarthy,〈Recursive Functions of Symbolic Expressions and Their Computation by Machine, Part I〉(*CACM*, 1960)** — Lisp 的原典，garbage collection 第一次被描述就在這裡（只佔一頁多）。原文 PDF：`http://jmc.stanford.edu/articles/lisp/lisp.pdf`。不必全讀，看它怎麼用 eval 證明語言能描述自己——那就是「程式即資料」的最初現場。
- **John McCarthy,〈History of Lisp〉(1979)** — McCarthy 本人回顧 Lisp 怎麼長出來，包括 Steve Russell 把 eval 編成機器碼那段的第一手敘述（Stanford 線上有原文）。想聽當事人講「我笑他把理論跟實務搞混」的人讀這篇。
- **C. A. R. Hoare,〈The Emperor's Old Clothes〉(1980 圖靈獎演說，刊 *CACM* 24(2), 1981)** — 「a language so far ahead of its time…」這句話的原始出處。整篇是一位大師對「簡單 vs 複雜」的反省，和本書「複雜度守恆」的主題直接共鳴，值得完整讀一遍。
- **ALGOL 60 Report (Naur 編, 1960)** — 想親眼看 BNF 第一次被用來描述一個真語言長什麼樣，翻這份報告的語法定義段落。歷史文件，當古蹟看，看「用形式記號描述語法」這件事在 1960 年是多新的東西。
- **Paul Graham,〈The Roots of Lisp〉** — 用現代讀者讀得懂的方式重講 McCarthy 1960 論文的核心（eval 的七個原始運算子）。如果原典太硬，從這篇進去（Graham 個人網站有 PDF）。
- **延伸（COBOL 壽命）**：IEEE *Spectrum*「COBOL programming shelf life」一類報導 — 看「為什麼這語言不死」的當代討論。讀的時候練本章教的自律：凡看到行數/金額統計，先問「這數字有幾個獨立來源」，多半只有一個。
