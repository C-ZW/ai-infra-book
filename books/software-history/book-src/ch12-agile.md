# ch12 — 敏捷：對重量級流程的反叛

> **本章解決什麼問題**：上一章（見 ch11）替瀑布翻案——重量級流程不是笑話，它在「需求穩定、變更昂貴、可稽核要緊」的世界裡是合理的。但軟體的世界大多不是那樣。1990 年代末，一群人撞上重量級流程的死穴：它把賭注全壓在「我們能在開工前把需求想清楚」上，而真實專案的需求一直在變。這面牆瀑布翻不過去——你沒辦法靠寫更厚的規格文件，把「客戶其實也不知道自己要什麼」這件事消滅掉。敏捷是對這面牆的回應，而它的招數不是更好的預測，是換一種賭法：用短回饋迴圈擁抱變化，而不是預測變化。本章是「方法 strand」的高潮收尾（接 ch10 人月神話、ch11 瀑布），也接住你每天在做的測試、CI、重構——它們是敏捷技術實踐的直系後代。敏捷往維運的延伸（DevOps/SRE）留給 ch18。

## 從你已知的出發

你每天都在實踐敏捷的技術核心，而且你大概也親眼看過敏捷的靈魂被掏空。這兩件事同時為真，正是本章要拆開的張力。

先說你在實踐的那一面。你做 greenfield 微服務時，CI 上有一條規則：OpenAPI schema 一改、client 不同步就擋下整個 pipeline。你寫這條規則不是為了好看，是為了讓「契約變了」這件事立刻、自動、無情地被抓出來，而不是等到三個禮拜後某個前端工程師在 production 才發現欄位對不上。你寫單元測試，不只是為了證明現在對，更是為了讓你**敢改**——有測試網接著，重構才不是賭命。你 review 時把一段醜但能跑的程式留言「先合進去、開個 ticket 之後重構」，你在做的是把「改善設計」變成一件可以分批、持續、安全進行的事，而不是攢到某個神話般的「之後會有時間重寫」。

這三件事——測試先行的安全網、CI 的即時回饋、持續重構——不是雜湊的工程習慣。它們是 1990 年代末一場明確反叛的技術遺產，那場反叛的核心主張只有一句：**既然你擋不住變化，就別再假裝能預測它；改成讓系統隨時都能安全地被改。** 你的 CI 強制同步、你的測試網、你的重構紀律，全都是這句話的可操作版本。

再說被掏空的那一面。你大概也經歷過另一種「敏捷」：每天早上站會，輪流報「我昨天做了什麼、今天要做什麼」，像在跟主管交代進度，沒人真的在「同步」什麼；sprint 結束算 velocity，然後 velocity 變成 PM 拿來壓你「上一季 40 分這一季怎麼只有 35」的棒子；retrospective 變成走過場，講完「下次會更好」沒人跟進。你心裡隱約知道哪裡不對，但說不清楚——明明每個儀式都照著做了，為什麼它感覺跟「敏捷」該有的樣子背道而馳？

本章給你說清楚的工具。那種走味不是你的錯覺——它有名字、有歷史，連當年寫下敏捷宣言的人自己都跳出來罵過。要看懂它，得先回到反叛的現場：一群人到底在反叛什麼、具體改了什麼、那個核心賭注是什麼。

## 反叛的對象：不是文件，是「先預測再執行」這個賭法

先把最常見的誤解清掉，因為它會讓你整章都讀偏。

流行的刻板印象是：「瀑布＝寫一堆文件，敏捷＝不寫文件、快速迭代。」這個對立是錯的，而且錯得很有破壞性——它讓無數團隊以為「我們不寫文件了，所以我們敏捷了」。敏捷宣言裡那句「**Working software over comprehensive documentation**」（可運作的軟體勝過詳盡的文件），關鍵字是 **over**，不是 **without**。宣言結尾那句話常被跳過、卻是整份文件的鑰匙（以下為原文）：

> "That is, while there is value in the items on the right, we value the items on the left more."

也就是說：右邊那些（流程、文件、合約、計畫）**有價值**，敏捷沒說它們是垃圾；只是當你必須在兩者間取捨時，左邊更值錢。把敏捷讀成「不寫文件」，跟把上一章 Royce 讀成「瀑布之父」是同一種病——把一個關於**權衡**的精緻主張，壓扁成一句非黑即白的口號。

那麼反叛的真正對象是什麼？是重量級流程背後那個賭法：**「我們可以在開工前，把需求、設計、計畫都想清楚，然後照計畫執行。」** 上一章說過，Royce 1970 自己畫純線性圖時就警告過它對需求變更脆弱、主張要迭代（見 ch11）；但 1980、1990 年代的企業與政府專案，把這套「先全部想清楚再做」用 CMM 這類成熟度模型制度化、規模化，變成一台龐大的、以文件和階段簽核為骨架的機器。這台機器有一個它無法消化的輸入：**變化**。

把痛攤開來看。重量級流程的生命週期大致是：需求分析（幾個月）→ 設計（幾個月）→ 實作（幾個月）→ 測試（幾個月）→ 上線。整條鏈的有效性，完全建立在一個假設上——**第一階段寫下的需求，到最後一階段還成立**。但真實世界裡：

- 客戶在看到第一版能跑的東西**之前**，往往講不清楚自己要什麼（他要看到才知道「不是這個」）。
- 市場、法規、競爭對手在這幾個月裡會動。
- 連團隊自己對問題的理解，都會在動手做的過程中才變深。

於是最致命的事發生了：在這個瀑布模型裡，「需求錯了」這個事實，要等到**最後**整合測試或上線時才會浮現——也就是在你已經把所有錢和時間都花掉之後。回饋迴圈長達數月甚至數年。等帳單回來，沉沒成本已經高到沒人敢喊停，於是大家含淚交付一個「照規格做對、但根本不是客戶要的」的系統。

這就是敏捷撞的牆，而它跟前幾章的牆性質不同。ch05 的牆是「程式大到一個人腦裝不下」（規模 vs 認知）；ch10 的牆是「軟體的本質複雜度砍不掉」（Brooks）。這一章的牆是**時間維度上的**：**世界在你交付之前就變了，而你的流程假設它不會變**。重量級流程不是不努力，是它把全部賭注押在一個會輸的假設上——押「未來可被預測」。

```text
        重量級流程的賭法：押「需求在開工時就能定死」
        ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐
        │ 需求 │─►│ 設計 │─►│ 實作 │─►│ 測試 │─►│ 上線 │
        └───┬──┘  └──────┘  └──────┘  └──────┘  └───┬──┘
            │                                        │
            └────────────────────────────────────────┘
        回饋要等到「上線」才回來（數月～數年）；等帳單回來，
        沉沒成本已高到不敢喊停
```

## 核心賭注：用短迴圈擁抱變化，而不是預測變化

如果問題是「未來無法被準確預測」，那解法的形狀就清楚了：**別再賭預測，改賭適應**。這是敏捷全部主張的根，其餘一切（迭代、測試、CI、重構）都是為了讓「適應」變得便宜又安全而長出來的工具。

敏捷宣言背後的十二條原則裡，第二條把這個賭注講得最白（原文）：

> "Welcome changing requirements, even late in development. Agile processes harness change for the customer's competitive advantage."

**歡迎變更的需求，即使是在開發後期。** 注意這句的攻擊性——它不是「容忍」變化、不是「管理」變化，是 **welcome**、是 **harness**（駕馭、為我所用）。重量級流程把「需求變更」當成要嚴防的災難（change control board、變更申請、影響評估），敏捷把它當成競爭優勢的來源。這是世界觀的一百八十度翻轉。

為什麼「短迴圈」能駕馭變化？因為它把那條長達數月的回饋迴圈，剁成一段段一兩週的小迴圈。每個迭代結束，你交出一塊**真的能跑**的軟體，拿給真實的人看，收到「不對、其實我要的是這個」的回饋——然後在下一個迭代修正。關鍵的數學在這裡：**錯誤的成本，正比於它被發現得多晚。** 在瀑布裡，一個需求理解錯誤潛伏數月才爆，炸掉的是整個專案；在兩週迭代裡，同一個錯誤兩週內就被一塊能跑的軟體照出來，你只損失了兩週。你沒有變得更會預測，你是讓「猜錯」的代價小到可以一直猜、一直修。

這裡要釘死一個觀念，因為它是本章最重要的領悟：**敏捷不是「不做計畫」，是「把賭注從預測改成適應」。** 它沒有取消計畫——它把「一份賭上全部、不容修改的長期計畫」，換成「一連串廉價、隨時可丟棄重來的短期計畫」。你還是規劃，只是你規劃的單位從「整個專案」縮到「下一個迭代」，並且接受每個迭代的結尾都可能推翻你對後面的假設。用你熟的語言講：重量級流程像一次性的大 batch、強一致、回滾代價極高的交易；敏捷像一串小的、冪等的、可重試的訊息——**每一則都小到失敗了也不痛，靠大量短迴圈逼近一個你開工時根本不知道的目標**。你做 SQS 結算 pipeline 時把大工作切成可重試小訊息，是同一個直覺：縮短回饋迴圈、壓低單次失敗代價。敏捷只是把這個你早已內化的原則，用在「整個產品開發流程」這個尺度上。

敏捷面對的是 Brooks 說的**本質複雜度**的一種——「需求不確定性」。它的策略不是消滅（沒有銀彈，見 ch10：沒有任何流程能讓客戶開工時就完全清楚自己要什麼），而是**馴服**——承認需求不確定是不可消除的本質，然後讓它以「可承受的小步損失」持續釋放，而不是「壓抑到最後一次性引爆」。這是方法 strand 的共同底色：方法不創造銀彈，只把不可消除的複雜度安排成人能承受的形狀。

## 前史：反叛不是憑空冒出來的（XP 與 Scrum）

2001 年那份宣言不是起點，是一場已經打了好幾年的仗的**停戰簽字**。在 Snowbird 之前，好幾條獨立的「輕量級流程」血脈各自演化，最重要的兩條是 XP 與 Scrum。

### Extreme Programming：把好實踐推到極致

Extreme Programming（XP，極限編程）的主角是 **Kent Beck**。名字裡的「extreme」是個工程笑話也是個主張：如果 code review 是好事，那就**一直**做（結對編程，pair programming，兩個人共用一個鍵盤、隨時互相 review）；如果測試是好事，那就在寫程式碼**之前**先寫測試（測試先行）；如果整合是好事，那就**一天整合很多次**（持續整合）。XP 的招數不是發明全新的東西，是把一堆「大家都知道是好習慣、但偶爾才做」的實踐，調到最大、變成持續不間斷的紀律。

XP 的鍛造場是一個具體的、有名有姓的專案：**克萊斯勒的 C3 薪資系統（Chrysler Comprehensive Compensation System）**。把這個個案攤開，比任何抽象定義都更能讓你看懂敏捷在反叛什麼。

### 個案：C3——一場反叛的解剖

C3 的目標是用一套新系統取代克萊斯勒好幾套老舊的薪資程式。專案大約 **1993 年**啟動（最初由 Tom Hadfield 主導，1994 年起用 Smalltalk 開發）。到 **1996 年** Kent Beck 被找進來時，系統已經做了好幾年，卻**連一張薪資單都還沒印出來過**——這是重量級開發的典型死法：投入巨大、進度看似在動（文件一份份產出、模組一個個「完成」），但離「能交付的真實價值」始終遙遠。

Beck 進來後做了什麼？他把整套 XP 實踐裝上去：

- **短迭代**：不再追求一次做完整個系統，而是切成小週期，每個週期結束都要有可展示的進展。
- **結對編程**：兩個人一台機器，知識即時擴散、缺陷即時被抓，沒有「某段程式只有一個人懂」的單點。
- **測試先行**：先寫測試定義「對」長什麼樣，再寫程式碼讓它通過（這後來被 Beck 整理成 TDD，下節談）。
- **現場客戶**（on-site customer）：一個真實的業務代表坐在團隊裡，隨時回答「這個 case 該怎麼算」——把那條數月長的需求回饋迴圈，縮短到「轉頭問一句」。

成果：**1997 年系統上線，發出大約九千到一萬人的薪資**（不同來源數字略有出入，未逐一逐字核對；都指向「絕大多數月薪員工」這個量級）。一個卡了好幾年、印不出半張薪資單的專案，在裝上短迭代與現場回饋後跑起來了。Beck 把這套經驗寫成 **《Extreme Programming Explained: Embrace Change》（1999）**——副標題 **Embrace Change**（擁抱變化）就是整章那個核心賭注的四個字版本。

但 C3 的結局，恰恰是本章後半「敏捷理想 vs 現實」最誠實的預演，必須照實講、不能只講成功的上半場。1997 上線後，C3 **再也沒有成功發布過下一個版本**，新開發在 1999 年停掉，專案在 **2000 年 2 月**（克萊斯勒與 Daimler-Benz 合併後）被取消。更刺的一個細節：上線沒幾個月，那個關鍵的「現場客戶」角色——XP 賴以縮短回饋迴圈的人——**因為過勞而離職，且找不到人接替**。

這個細節值得你停下來想。敏捷不是免費的午餐：它把「持續、即時的回饋」當燃料，而這份回饋需要真實的人持續付出（一個全心投入的客戶代表、一群願意天天結對與寫測試的工程師）。當組織只想要敏捷的「快」，卻不願持續供應它賴以運轉的「人的投入」，引擎就會空轉、燒乾。C3 既證明了 XP 能讓卡死的專案跑起來，也預告了它最脆弱的地方——而這個脆弱，正是後來「敏捷被收編」時第一個被犧牲掉的東西。

### Scrum：把節奏與角色制度化

另一條血脈是 **Scrum**，主角是 **Ken Schwaber 與 Jeff Sutherland**。如果 XP 偏重「工程師怎麼寫程式」的技術實踐，Scrum 偏重「團隊怎麼組織節奏」的管理框架——固定長度的迭代（sprint）、每天的同步、明確的角色。兩人在 **1995 年的 OOPSLA 會議**正式發表 Scrum（第一個 Scrum 團隊約在 1993–94）。

Scrum 這個名字的來歷，本身就濃縮了整章的主張，值得講清楚——**它不是縮寫**。它的根在一篇商管論文：**竹內弘高（Hiroaki Takeuchi）與野中郁次郎（Ikujiro Nonaka）1986 年發表在《哈佛商業評論》的〈The New New Product Development Game〉**。這兩位學者研究 Honda、Canon、Fuji-Xerox 等公司的產品開發，提出一個對比（以下為論文大意，非逐字引文）：傳統的開發像**接力賽（relay race）**——每一棒（部門/階段）跑完自己這段，把棒子交給下一棒，分工清楚但僵硬；他們觀察到的高效團隊則像**橄欖球（rugby）**——整支隊伍抱成一團、一起朝球門推進，球在隊員之間靈活地來回傳遞。「Scrum」正是橄欖球裡那個全隊擠在一起爭球的陣式。

看出這個隱喻在反叛什麼了嗎？**接力賽就是瀑布**——需求組跑完交給設計組，設計組跑完交給開發組，一棒交一棒、回不了頭。**橄欖球就是敏捷**——跨職能的一隊人一起、同時、持續地往目標推，需求、設計、實作交織進行而非依序交棒。Sutherland 後來把這篇論文的洞見，結合迭代增量開發、物件導向、經驗式流程控制，落地成軟體開發的 Scrum 框架（具體的 sprint/站會/角色 how-to 不在本書範圍，這裡只取它的歷史與哲學）。

```text
        竹內・野中 1986：兩種開發節奏

        接力賽（=瀑布）：一棒交一棒，回不了頭
        [需求]──棒──►[設計]──棒──►[開發]──棒──►[測試]

        橄欖球（=敏捷）：跨職能一隊，抱團一起推進
        ┌─────────────────────────────────────┐
        │  需求 ◄─► 設計 ◄─► 開發 ◄─► 測試    │
        └──────────────────┬──────────────────┘
                           │   整隊同時往前
                           ▼
                   球門（可交付的價值）
```

## 把「擁抱變化」變成可操作的紀律（TDD／重構／CI）

「擁抱變化」聽起來像一句勵志標語。它之所以沒淪為空話，是因為敏捷給了它三件具體的工程紀律當骨架。少了這三件，「歡迎變更」只是嘴上說說——因為一個你**不敢改**的系統，再怎麼歡迎變化也是假的。這三件事的共同目的只有一個：**讓系統隨時都能被安全地改動**。它們是你今天日常的直系祖先，值得各自釘清楚來歷。

**測試驅動開發（TDD）**。歸給 **Kent Beck**，但他自己說是「重新發現」而非發明——這個謙遜很重要，本書反覆出現（McCarthy 的 GC、Hopper 的編譯器都有歸屬該不該算「發明」的分寸問題）。TDD 的節奏是 red-green-refactor：先寫一個會失敗的測試（red，定義「對」是什麼）→ 寫最少的程式碼讓它通過（green）→ 在測試保護下重構（refactor）。TDD 真正搬走的複雜度不是「驗證正確」——是**讓你敢改**。一套覆蓋夠的測試，是一張隨時告訴你「你剛剛有沒有弄壞什麼」的安全網；沒有它，每次修改都是在沒繩索的情況下走鋼索，於是人會本能地**不敢動**舊程式碼，系統就僵化、腐爛。TDD 是「擁抱變化」的技術前提：你得先有網，才敢談擁抱。

**重構（Refactoring）**。系統化的論述歸給 **Martin Fowler 的《Refactoring》（1999）**（學術源頭可上溯到 William Opdyke 1992 的博士論文）。重構的定義很精確：**在不改變外部行為的前提下，改善內部結構**。為什麼這是敏捷的必需品？因為敏捷放棄了「開工時就把設計想到完美」——它接受你會邊做邊懂、設計會隨理解加深而需要調整。重構就是讓「持續調整設計」變成一件安全、日常、小步進行的事，而不是攢到某個傳說中的「大重寫」。它和 TDD 是一對：測試提供安全網，重構在網上改善結構。你 review 時說「先合進去、開 ticket 重構」，你執行的就是 Fowler 這套——把改善設計從「一次性豪賭」變成「持續償還的小額分期」。

**持續整合（CI）**。歸屬要小心，照 landscape：「CI」一詞**一般歸於 Grady Booch**（1990 年代，在他的物件導向分析與設計著作裡用到），**由 Beck 與 XP 發揚光大**——這個歸屬常被簡化成「XP 發明了 CI」，並不準確。CI 的痛點來自一個你太熟的災難：**整合地獄（integration hell）**。重量級流程裡，大家各自開發數週甚至數月，最後才把所有人的程式碼合在一起——然後發現它們互相衝突、誰也兜不攏，整合本身變成一個漫長痛苦、充滿驚嚇的階段。CI 的解法和敏捷的核心賭注完全同構：**別讓回饋迴圈變長**。一天整合很多次，每次只合一小塊，衝突在它還小、還好解的時候就被抓出來。你的 CI 強制 OpenAPI 同步、PR 一推就跑全套測試與 lint——這正是 Booch 命名、XP 發揚的那個主張，在 2026 年的成熟形態：**把「壞掉了」這個訊號的延遲，壓到接近零**。

三件紀律連起來看，是一台精巧的機器：

```text
        三件紀律，一個目的：讓系統隨時可被安全改動
        ┌────────────────┐
        │ TDD：先寫測試  │  造出「敢改」的安全網
        └───────┬────────┘
                ▼
        ┌────────────────┐
        │ 重構：在網上改 │  持續調整設計，不攢到大重寫
        └───────┬────────┘
                ▼
        ┌────────────────┐
        │ CI：頻繁整合   │  把「壞了」的回饋延遲壓到近零
        └───────┬────────┘
                ▼
            擁抱變化（變成可操作，而非口號）
```

這台機器就是「把賭注從預測改成適應」的物理實現。預測的賭法不需要這些——如果你相信開工就能想清楚一切，你不需要隨時能安全改動的能力。正因為敏捷賭的是「我會一直猜錯、一直要修」，它才**必須**有一套讓「修」變便宜又安全的工具。你天天在用這台機器，現在你知道它是為了押哪個賭注而造的了。

## 帳單：當反叛被收編成儀式劇場

到這裡為止都是敏捷的上半場——一個漂亮、有道理、被你日常驗證為真的賭注。但本書不寫鄉愁、也不寫行銷，必須誠實記下半場：**敏捷贏了，然後被它的勝利反噬。** 這是方法 strand 一個反覆出現的悲劇模式，值得攤開。

機制是這樣的。敏捷宣言的核心是**價值與原則**——個人與互動重於流程與工具、可運作的軟體重於文件、擁抱變化。但價值與原則很難賣、很難稽核、很難寫進顧問合約。於是市場做了它總會做的事：**把難賣的靈魂，換成好賣的軀殼**。「個人與互動重於流程與工具」這條，被一整套**流程與工具**（認證、儀式、看板軟體、教練顧問）給包裝起來販賣——這本身就是對宣言第一條價值的精準背叛。敏捷宣言反的是重量級流程，結果催生了一個新的重量級流程產業。

落到你身上的具體走味，你都見過：

- **站會變成進度匯報**。每日站會（daily stand-up）的原意是團隊**互相同步**、暴露阻礙、好讓大家自組織地排除障礙。被收編後，它變成輪流對著主管（或 Scrum Master 代理的主管之眼）報「昨天/今天/阻礙」三段式進度——**溝通的方向從「橫向同步」偷偷轉成了「縱向交代」**。一個本來服務團隊的儀式，變成服務管理層監控的工具。
- **velocity 變成績效棒子**。velocity（團隊每個迭代完成的工作量估計）的原意是**團隊自己**用來預測「下個 sprint 大概能吃多少」的內部規劃工具，是給團隊看的、會自然波動的估計。被收編後，它變成管理層拿來跨團隊比較、施壓、考核的 KPI——「為什麼這隊 velocity 比那隊低」。一旦估計值變成考核指標，它立刻被 Goodhart 定律反噬：工程師學會灌水點數，數字漂亮了、意義消失了。一個本來幫團隊誠實面對自己速度的工具，變成逼團隊對自己撒謊的棒子。
- **儀式照做、靈魂出走**。sprint planning、retrospective、grooming 一個不漏地開，但每個都變成走過場——retrospective 講完「下次會更好」沒人跟進，planning 變成把上面交辦的工作硬塞進固定 sprint。**形式被完整保留，目的被完整抽空**。

這不是我的吐槽，是當年寫下宣言的人自己的判決，而且分量極重——**簽署那份宣言的十七人裡，有人公開宣告它已死**。**Dave Thomas**（宣言十七位簽署人之一）2014 年寫了〈Agile is Dead (Long Live Agility)〉，大意是說「Agile」這個詞已經被顛覆到幾乎失去意義，如今所謂的敏捷社群，很大程度淪為顧問與廠商兜售服務和產品的場子（非逐字引文，已查證其立場與文章存在）。同為十七人之一的 **Martin Fowler**，在 2018 年 Agile Australia 的演講裡，把敏捷的現狀稱為一場 **travesty**（拙劣的諷刺），矛頭指向一個他命名為「**Agile Industrial Complex**（敏捷工業複合體）」的東西——一個把「方法」強加於人的產業，而這恰恰是宣言第一條價值（個人與互動 **over** 流程與工具）所反對的。

把這件事的諷刺講到底：**敏捷反叛的是「重量級流程把過程本身變成目的」，而被收編後的敏捷，自己變成了一套「把儀式本身變成目的」的重量級流程。** 鐘擺盪了一整圈，回到它出發時反對的東西旁邊。你經歷過的「假敏捷」，不是個別公司執行不力的偶發事故，是一個有名字、被原作者親自認證的系統性現象。看懂這個，你就有了診斷自己團隊的工具：問一句「這個儀式現在服務的是團隊的回饋，還是管理層的監控？」——答案會告訴你，你做的是敏捷，還是敏捷的標本。

## 複雜度搬去哪了

敏捷這一跳，照例結一筆帳——它消滅了什麼負擔，又製造了什麼新負擔。

| 項目 | 內容 |
|---|---|
| **撞的牆** | 時間維度的牆：世界（需求、市場、團隊自己的理解）在交付之前就變了，而重量級流程把全部賭注押在「未來可被預測、需求可在開工時定死」這個會輸的假設上 |
| **這次主導的 strand** | 方法 strand（流程方法的反叛與重組），但與抽象 strand 交織——TDD/重構/CI 把「適應」這個方法主張，沉澱成測試框架、CI 工具、IDE 重構功能等技術抽象 |
| **消滅了什麼負擔** | 「猜錯需求要等數月/數年、付出整個專案才發現」的巨額延遲成本；把一條超長回饋迴圈剁成一串短迴圈，讓單次猜錯的代價從「炸掉專案」降到「損失一個迭代」 |
| **複雜度搬到哪** | 「需求不確定性」這個本質複雜度沒被消滅（沒有銀彈，見 ch10），它被**搬成一筆持續支付的成本**——持續的回饋、持續的測試維護、持續的重構、持續的整合，由團隊每天小額償還，取代「最後一次性引爆」 |
| **新製造的負擔** | (1) 敏捷靠「人的持續投入」當燃料（投入的客戶代表、願意天天結對與寫測試的工程師），這份投入難以持續、易過勞（見 C3 現場客戶燒乾）；(2) 測試/CI 本身是要長期維護的資產，測試腐爛、CI 變慢會反過來拖累迭代；(3) 沒有了長期計畫的錨，弱組織容易滑進「無止境地對需求做反應、看不見全局方向」的另一種失控 |
| **會在哪洩漏／反噬** | 當組織只要敏捷的「快」、不願供應它賴以運轉的「人的投入與信任」時，敏捷退化成**儀式劇場**：站會變進度匯報、velocity 變績效棒子、儀式照做而靈魂出走——反叛重量級流程的敏捷，自己變成一套把儀式當目的的重量級流程（鐘擺盪回原點） |
| **這一跳的帳單一句話** | 用「持續的小額成本（回饋、測試、重構、整合）」換「猜錯需求的代價從炸掉專案降到損失一個迭代」——複雜度從「最後一次性引爆」搬成「每天分期償還」，而分期一旦停付（或被異化成考核儀式），帳單立刻以走味的形式回來 |

## 紙上推演

### 推演題一 **[15 分鐘]** ★★

用「擁抱變化 vs 預測變化」這組對立，向一個資淺工程師解釋敏捷與瀑布的**根本**分歧。要求：你的解釋**不准**用「文件多寡」或「快不快」當主要區別（那是表面症狀，不是病根），而要落在「兩者各自賭未來是不是可預測的」這個賭法差異上。並說清楚為什麼「敏捷＝不寫文件」是個有破壞性的誤讀。

#### 推演解答

根本分歧在賭法，不在文件量。**重量級流程賭「未來可被預測」**——它假設你能在開工前把需求、設計、計畫想清楚，然後照計畫執行；它的整個生命週期（需求→設計→實作→測試）的有效性，都建立在「第一階段寫下的需求，到最後一階段還成立」這個假設上。**敏捷賭「未來不可被準確預測」**——它承認客戶在看到能跑的東西之前講不清要什麼、市場與理解都會變，所以放棄預測，改用短迭代＋持續回饋來**適應**變化。

把賭法講清楚後，「文件多寡」與「快不快」就只是這個賭法差異的下游症狀：因為瀑布賭預測，它需要厚重的前期文件把預測固定下來、需要階段簽核確保大家照預測走；因為敏捷賭適應，它把規劃單位縮到一個迭代、把投入導向「能跑的軟體」這個最誠實的回饋來源。文件變少是**結果**，不是目的。

為什麼「敏捷＝不寫文件」有破壞性？因為它讓團隊以為「我們停掉文件就敏捷了」，完全錯過了真正的轉變（賭法）。宣言原文是「Working software **over** comprehensive documentation」——關鍵是 over（取捨時的優先序）不是 without（拋棄）；結尾還明說「右邊那些仍有價值，只是左邊更重要」。把它讀成「不寫文件」，跟把 Royce 讀成「瀑布之父」（見 ch11）是同一種病——把一個關於**權衡**的主張，壓扁成非黑即白的口號。

常見誤解點破：學徒最容易抓著「站會、sprint、不寫文件」這些**儀式與症狀**來定義敏捷，於是「照做儀式」就自認敏捷。反駁的關鍵是把定義從「做哪些儀式」搬到「押哪個賭注」——一個團隊就算開齊所有儀式，只要它骨子裡還在賭「需求開工就能定死」，它就還是瀑布。

### 推演題二 **[15 分鐘]** ★★★

診斷你親身經歷過的一次「假敏捷」走味。具體說清楚：(a) 哪個儀式或指標被異化了；(b) 它**原本**服務的對象/目的是什麼；(c) 被收編後它**實際**服務的對象/目的變成什麼；(d) 它背離了敏捷宣言四條價值或十二條原則裡的哪一條。試著至少診斷「站會變進度匯報」與「velocity 變績效棒子」其中一個，並對照宣言原文。

#### 推演解答

以「velocity 變績效棒子」為例（站會那題同理可套）：

(a) 被異化的指標：velocity（團隊每迭代完成的工作量估計）。

(b) 原本的對象/目的：**給團隊自己用**的內部規劃工具，用來預測「下個 sprint 大概能承接多少工作」。它本質是一個會自然波動的估計值，服務的是團隊的自我規劃。

(c) 收編後實際的對象/目的：變成**管理層**拿來跨團隊比較、施壓、考核的 KPI（「為什麼這隊比那隊低」「上季 40 這季怎麼掉到 35」）。服務對象從「團隊的規劃」偷換成「管理層的監控」。

(d) 背離哪一條：最直接背離第一條價值「**Individuals and interactions over processes and tools**」——把一個服務人與團隊互動的估計工具，異化成一個用流程/指標去管控人的棒子。它也踩到「**信任**團隊、給他們所需的支持」這條原則的反面：用估計值當考核棒，等於宣告不信任團隊會誠實規劃。

關鍵機制要講出來：一旦一個**估計值**被當成**考核指標**，它就觸發 Goodhart 定律——「當一個量度變成目標，它就不再是好的量度」。工程師理性的反應是灌水點數（同樣的工作估更多分），於是數字越來越漂亮、越來越沒意義。**這不是執行不力，是把工具用在它被設計來抵抗的方向上。**

延伸的領悟：診斷任何「假敏捷」都有一把通用鑰匙——問「這個儀式/指標現在服務的，是團隊的回饋與自組織，還是管理層的監控與考核？」方向一旦從橫向（團隊內同步）偷轉成縱向（向上交代），它就從敏捷的工具，變成了敏捷的標本。

### 推演題三 **[10 分鐘]** ★★

對帳：把你的 CI/CD pipeline 拆開，逐項對應它實現了敏捷的哪些主張。至少涵蓋：(1) CI 強制 OpenAPI 同步、PR 一推就跑全套測試與 lint；(2) 自動化測試套件；(3) 頻繁、小批量的部署。每一項說清楚它在押「把賭注從預測改成適應」這個核心賭注的哪一面。

#### 推演解答

- **CI 強制 OpenAPI 同步 + PR 跑全套測試/lint**：這是「持續整合」的直系後代（Booch 命名、XP 發揚）。它押的賭注是「**別讓回饋迴圈變長**」——把「契約壞了/測試掛了」這個訊號的延遲壓到接近零（推 PR 的當下就知道），對抗的是重量級流程的「整合地獄」（大家各自做數週、最後才合、然後兜不攏）。對應核心賭注的「適應」面：你不預測「大家寫的不會衝突」，你讓衝突在它還小的時候就被自動照出來。

- **自動化測試套件**：是 TDD 留下的安全網（就算你沒嚴格 red-green-refactor，測試的目的一致）。它押的是「**讓系統隨時能被安全改動**」——有網才敢改，敢改才談得上擁抱變化。它對應的不是「驗證正確」這麼淺，而是「讓修改的代價與恐懼降到可以一直修」。

- **頻繁、小批量部署**：把「交付」這條回饋迴圈也剁短——小批量上線，一旦出事，回滾範圍小、定位快（呼應你 SQS pipeline 把大工作切成可重試小訊息的直覺）。它押的是「**單次失敗的代價要小到可承受**」——你不賭「這次大版本不會出事」，你賭「就算出事，因為批量小，損失也有限」。

對帳的領悟：你的整條 pipeline，本質上是一台「把回饋迴圈縮短、把單次失敗代價壓低」的機器——而這正是敏捷核心賭注（用短迴圈適應、而非長計畫預測）的自動化、工程化形態。你可能從沒讀過敏捷宣言，但你的 CI/CD 每一條規則都在替它的核心主張背書。值得注意的反差：你在**技術實踐**這一層把敏捷做得很紮實（CI、測試、重構），但很多團隊正是在**流程儀式**那一層把敏捷做壞了（站會、velocity 走味）——這提示你，敏捷真正活著的地方，往往在 pipeline 裡，而不在會議室裡。

口頭自答；講得出因果才算過關，講不清楚代表還沒懂。

1. 敏捷反叛的真正對象是什麼？為什麼說「瀑布＝寫文件、敏捷＝不寫文件」這個對立是錯的、而且有破壞性？（提示：宣言裡的 over vs without、結尾那句保留條件）
2. 敏捷撞的那面牆，跟 ch05（規模 vs 認知）、ch10（本質複雜度）撞的牆，性質有什麼不同？為什麼這面牆瀑布翻不過去？
3. 用你自己的話講清楚敏捷的核心賭注——「用短迴圈擁抱變化，而不是預測變化」。為什麼「縮短回饋迴圈」能駕馭變化？這跟你把大工作切成可重試小訊息的直覺是同一件事嗎？
4. 把 C3 專案的故事講給另一個工程師聽：它卡在哪、Beck 裝上 XP 後改了什麼、為什麼最後仍然失敗（現場客戶過勞那個細節）。這個失敗預告了敏捷後來被收編時，第一個被犧牲的是什麼？

## 延伸閱讀

- **《Manifesto for Agile Software Development》（agilemanifesto.org，2001）**——四個價值＋十二條原則，全文不到一頁，必讀原文。重點看四句 over 後面那句保留條件「while there is value in the items on the right, we value the items on the left more」，以及十二原則第二條「Welcome changing requirements, even late in development」——整章的核心賭注就在這兩句裡。網站的 history.html 頁有 Snowbird 現場的第一手回憶。
- **Kent Beck, 《Extreme Programming Explained: Embrace Change》（1999）**——XP 的綱領，副標題就是本章核心四個字。不必通讀，看它怎麼論證「把好實踐推到極致」（為什麼整合好就一天整合很多次、測試好就先寫測試）；這是「把擁抱變化變成可操作紀律」的源頭文本。
- **Takeuchi & Nonaka, 〈The New New Product Development Game〉（Harvard Business Review, 1986）**——Scrum 的思想根。看「接力賽 vs 橄欖球」那組對比，你會發現敏捷對「跨職能團隊一起推進、而非依序交棒」的主張，1986 年就被兩位商管學者在汽車與相機工廠裡觀察到了——軟體業是後來才追上。
- **Martin Fowler, 《Refactoring》（1999；第 2 版 2018 改用 JavaScript 範例）**——重構的系統化論述。讀「為什麼重構」那幾章就好：理解「在不改變外部行為下持續改善內部結構」為什麼是敏捷的必需品（因為敏捷放棄了開工就設計完美、必須邊做邊改）。martinfowler.com 上有大量免費的 bliki 短文可先嚐。
- **Dave Thomas, 〈Agile is Dead (Long Live Agility)〉（2014，部落格文與演講）**——宣言十七位簽署人之一，親手宣告「Agile」一詞已被掏空。配 **Martin Fowler「The State of Agile Software in 2018」（Agile Australia 2018 keynote）**裡的「Agile Industrial Complex」一段一起看——兩位原作者親口診斷自己的孩子怎麼被收編，是本章「帳單」那節最有分量的一手佐證。
- **Winston Royce, 〈Managing the Development of Large Software Systems〉（1970）**——讀過 ch11 的話回頭再看一次：Royce 1970 就主張迭代與「do it twice」，敏捷某種意義上是把這個被埋沒三十年的主張，補完成一整套可操作的流程。瀑布與敏捷的對立，有很大成分是後人對 Royce 的誤讀製造出來的稻草人。
