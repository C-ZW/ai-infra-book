## 從你已知的出發

挑你這六年最難忘的三次 debug。我猜得出大概的形狀。

第一次：某個 API 莫名其妙地慢，p99 延遲爆了一根尖刺。你查程式邏輯，沒問題；查資料庫，也還好。最後你打開 ORM 的 query log，發現它在一個迴圈裡偷偷對每個物件各發了一句 SQL——拿 100 個使用者、再各拿他們的訂單，變成 1+100 句查詢。**N+1。** 你以為你在操作物件，底下其實是一百次往返。

第二次：某個 WebSocket 服務在尖峰時段湧入一波斷線重連。用戶端沒做退避（backoff），一斷線就同時搶著重連，重連後又各自立刻撈一輪初始化資料——一場教科書級的 thundering herd，把握手與 DB 查詢瞬間放大，連帶把 RDS 打到 CPU 滿載。你最後靠調 cache 策略擋掉那波重複查詢、把尖峰 RDS CPU 降了 40%——但在那之前，你得一層一層往下剝：先排除程式邏輯，再看連線池與 DB 執行計畫，最後跳到作業系統的 socket 層，去看 accept queue 是不是被瞬間湧入的握手塞爆。那一刻你不再是「寫應用的人」，你是「在每一層抽象的接縫之間找漏點的人」。

第三次：某個做成 Lambda 的輕量 handler，平常沒事，某次被觸發第一次呼叫慢了快一秒，後面又正常。你知道這叫 cold start——那個叫「serverless」的東西，底下**還是有伺服器**，只是平常被收走了，要跑你的碼得先變一個出來，這段時間就漏成了延遲。

這三次有一個共同點，而且這個共同點就是本章的全部：**每一次，都是一層你平常信任的抽象，在它的邊緣，把它本來替你藏好的底層複雜度漏了回來。** ORM 替你藏 SQL，邊緣漏出 N+1；WebSocket 那條「永遠在線」的連線抽象替你藏掉連線的脆弱，邊緣漏出重連風暴（底下還是一條會斷、要重新握手的 TCP 連線）；serverless 替你藏伺服器，邊緣漏出冷啟動。

而你值錢，恰恰是因為這些抽象會漏。若抽象永不洩漏，寫應用的人只要會用 API 就夠了，資深和資淺沒有差別。真實世界裡抽象一定會在邊緣漏底——漏的時候，能跳到底層、讀懂執行計畫、看穿 ORM 發了幾句 SQL、追到 TCP 重傳的人，就是把火滅掉的人。你的 profiling 與瓶頸診斷招牌，本質上都是**在抽象的裂縫裡工作**的能力。這一章要做的，是把這個你每天靠它吃飯的直覺，升級成一條你能講給別人聽的定律，並用全書的帳單證明它。

## Joel 的定律：所有不平凡的抽象都會洩漏

2002 年 11 月 11 日，Joel Spolsky 在他的部落格 *Joel on Software* 貼了一篇文章，標題叫 **《The Law of Leaky Abstractions》**。文章只提出一條定律，措辭乾淨到可以背下來：

> 「All non-trivial abstractions, to some degree, are leaky.」
> （所有不平凡的抽象，在某種程度上，都會洩漏。）

我把這句話的關鍵詞圈出來。**non-trivial**（不平凡）——平凡到沒藏任何東西的抽象不算，會藏東西的才算。**to some degree**（某種程度上）——不是「可能會」，是「都會」，差別只在程度。這不是一句吐槽，是一條他認為沒有例外的定律。

Joel 給的例子，你每一個都踩過：

- **TCP**：TCP 答應給你一條可靠的位元組串流，幫你把底下那個會丟包、會亂序、會壅塞的 IP 網路藏起來。但它藏不乾淨——當網路真的爛到一個程度（你的 Wi-Fi 訊號太差、跨洋線路壅塞），TCP 的「可靠」抽象就漏了：你感覺到的是卡頓、是逾時、是那根延遲尖刺。Joel 的原話大意是「TCP 試圖在不可靠的網路上提供一個完整的可靠抽象，但有時網路會從抽象底下漏上來，讓你感覺到它擋不住的東西」（非逐字，照原文精神轉述）。
- **SQL**：SQL 答應你「只要說要什麼、不用管怎麼拿」。但 Joel 指出，SQL 程式設計師終究得懂一點查詢計畫，否則寫不出效能可預測的程式碼——宣告式查詢這個抽象，在效能的邊緣漏出了底層的存取路徑。**這正是你的 RDS CPU 降 40% 那次幹的事**：你被迫部分變回領航員，去讀那張本該被藏起來的執行計畫（回收 ch13）。
- **網路檔案系統**：把遠端的檔案假裝成本機檔案（像 SMB／網路磁碟）。多數時候很好用，直到網路斷了、或某支假設「開檔很快」的舊程式碼跑在網路檔案上——本機檔案的抽象就漏出「這其實在網路另一端」的真相，慢得令人髮指，或者直接掛掉。

Joel 那篇文章還留下另一句被反覆引用的觀察，大意是：**「抽象幫我們省下做事的時間，但它沒省下學習的時間。」** 這句話是這條定律最實際的推論——你可以靠 ORM 不寫 SQL、靠 serverless 不管伺服器，但你**不能**因此不懂 SQL、不懂伺服器。因為抽象遲早會漏，漏的那一刻，你得有能力跳到它底下；而那個能力，只能靠你當初「明明可以不學」的時候去學來。一個只會用抽象、不懂抽象底下是什麼的工程師，在抽象正常運轉時跟資深的人沒兩樣，但在抽象洩漏的那一刻——也就是真正出事、真正需要人的那一刻——他無能為力。

這就是為什麼這本書一路逼你問「複雜度搬去哪了」。每一層抽象都把某塊複雜度搬到了你看不見的地方，而 Joel 的定律告訴你：**它沒搬走，只是搬到了一個會在邊緣漏回來的地方。** 你得知道它搬去哪了，才知道漏的時候要往哪裡跳。

## 為什麼洩漏是「守恆」的必然，不是工程沒做好

新手聽到「抽象會洩漏」，第一個反應常是：那是因為這個抽象做得不夠好吧？做得夠好不就不漏了？

不是。洩漏不是品質問題，是**結構問題**——它是「複雜度守恆」的直接推論，我用 ch10 的那把刀來證明給你看。

回收 ch10 Brooks 的《No Silver Bullet》（1986，注意是 1986 的獨立論文，不是 1975 原書）。Brooks 把軟體的複雜度切成兩塊：**偶然複雜度**（accidental，達成目的的手段所附帶的麻煩，可以被更好的抽象吸收、往下沉——對你這一層而言像是消失了）與**本質複雜度**（essential，問題本身固有的、任何工具都搬不掉的）。這把刀是全書反覆用的手術刀，這裡是它最鋒利的一次出場。

先把一個容易誤會的點說清楚：偶然複雜度被「吸收」，對站在上層的你來說是**真的變輕了**——這就是進步，是我們今天能管理比七十年前龐大數萬倍的系統的原因，別把「守恆」聽成「努力都白費」。但「對你這層消失」不等於「從宇宙消失」：它被搬進了編譯器、runtime、雲、別人的團隊，而那個接手的抽象，自己又在新的接縫上長出新的偶然複雜度。所以「守恆」講的不是一條熵不變的物理定律，而是一副眼鏡：**本質複雜度搬不掉、只能換地方頂著；偶然複雜度搬得掉，但每搬一次都在新邊緣留一道會漏的縫。沒有哪一跳是免費的——這才是「守恆」要你記住的事。**

現在看一個抽象在做什麼。一個抽象（ORM、TCP、雲、K8s）做的事，是把底下一坨複雜度包起來、給你一個乾淨的介面。它能成功包住的，是**偶然複雜度**——「怎麼拼出正確的 SQL 字串」「怎麼處理 TCP 的握手與重傳」「怎麼買機器進機房」這些「達成目的的麻煩手段」。這部分它確實替你扛了，而且扛得漂亮。

但它**包不住本質複雜度**。「這個查詢到底要碰多少資料、資料怎麼分布」是本質的——ORM 包不掉它，所以 N+1 漏回來。「這條網路到底有多不可靠」是本質的——TCP 包不掉它，所以壅塞時延遲漏回來。「這個運算到底要多少資源、什麼時候要」是本質的——serverless 包不掉它，所以冷啟動漏回來。

於是洩漏的位置是可以預測的：**抽象在『本質複雜度頂穿偶然複雜度的那道縫』洩漏。** 抽象能把偶然複雜度壓得很薄，但壓不到零；本質複雜度在底下頂著，總會在某個邊緣把那層薄膜頂破，漏上來。

把這個跟全書的主張接起來：複雜度守恆說「抽象不消滅複雜度，只把它搬家」。Joel 的定律是同一句話的另一面——**既然複雜度沒被消滅、只是被搬到抽象底下，那它就還在那裡；而還在那裡的東西，總有一天會從某個邊緣漏回來。** 不漏才奇怪。一個「永不洩漏的抽象」等於「真的消滅了複雜度的抽象」，而那違反守恆，不存在。

所以洩漏不是某個 ORM 寫得爛、某個雲做得差。再完美的 ORM 都會 N+1，再成熟的雲都有冷啟動。洩漏是抽象這個動作的本性。你能做的不是「找一個不漏的抽象」（沒有），而是**知道每個抽象會在哪裡漏、漏的時候跳得下去**。這是資深工程師的核心技能，也是這整本書想交給你的東西。

現在做這本書最該做、卻直到這一章才做得成的一件事：**把前面每一章末尾結的那筆帳，全部抄到同一張表上對齊。** 一筆一筆看，你只覺得「喔這個抽象有它的代價」；十七筆排在一起看，你會看見一個無法反駁的模式——**沒有哪一跳是憑空消滅複雜度：要嘛把偶然複雜度往下搬進工具（你這層變輕、底下變重），要嘛像 ch10 那樣指出有一塊本質複雜度根本搬不動；而搬家的過程，總在新的接縫上留下一個你認得的 debug 現場。** 這就是「守恆不是比喻、是反覆出現的事實」的證據。

| **ch17 微服務/容器** | 複雜度從「程式內」搬到「網路上」 | 大團隊在單體上的協調地獄、「在我機器上能跑」的環境差異沒了 | 函式呼叫變網路呼叫（逾時、半途失敗、最終一致性、要 trace 縫真相）；切錯邊界＝分散式單體；規模不對的早採用＝純付稅 |
| **ch18 DevOps/SRE** | 維運從「丟過牆」搬成共同擁有、用自動化吸收 | 上線從大事件變按鈕級非事件；可靠度從吵嗓門變可預算的數字 | CD/可觀測性是要維護的重資產（pipeline 腐爛、警報疲勞）；error budget 被當考核棒則異化；平台工程的 IDP 是**又一層抽象**，會洩漏 |

讀完這張表，把它倒過來讀一次——只讀最後一欄。N+1、GC pause、冷啟動、型別在邊界瞎掉、分散式單體、最佳化器猜錯、TCP 重傳……**這一整欄，就是你六年職業生涯的 debug 清單。** 它不是巧合。每一個你 debug 過的詭異現場，幾乎都能在這張表上找到對應的那一格——因為它們全都是同一件事：某層抽象在它的邊緣，把它替你藏好的複雜度漏了回來。

這張表還暴露了兩個更深的東西。

第一，**「往前搬」是個反覆出現的主旋律。** 看 ch08（型別把錯誤從執行期搬到執行前）、ch09（並行災難從執行期搬到設計期紀律）、ch05（控制流混亂搬成寫程式的紀律）——它們是同一個動作：**把問題從「跑的時候才爆」往前搬到「寫的時候就擋掉」。** 這是抽象與方法兩條 strand 反覆做的同一件事。而帳單也一致：往前搬不是免費的，代價是「必須時時維持那份紀律」，紀律一鬆，複雜度就從它被搬去的地方漏回來。

第二，**有一格是空的——ch10。** Brooks 那一章沒有「搬走了什麼」，因為他做的不是搬家，是**指出有一塊複雜度搬不走**。這一格的空白，是整張表的錨。它告訴你：其餘每一筆帳之所以都還有「在哪洩漏」這一欄，正是因為它們搬的都是偶然複雜度，而底下那塊 Brooks 指認的本質複雜度從不曾移動分毫——它就在那裡頂著，等著從每一層抽象的邊緣漏上來。Joel 的定律和 Brooks 的刀，講的是同一件事的兩端。

## 鐘擺總盤點：為什麼我們週期性回到底層

如果把這條路再拉遠一點看，你會看見比「逐章洩漏」更大的形狀：**鐘擺。** ch01 提出過四組反覆擺盪的張力，全書一路在回收。現在把它們的完整擺動軌跡盤點出來——你會發現所謂「新趨勢」，很多其實是上一輪的回擺。

```text
效能 ◄──────────────────────────────────────────► 生產力
  手寫機器碼   FORTRAN(1957)   高階語言/GC   ……   熱路徑手刻最佳化(回擺)
  集中 ◄──────────────────────────────────────────► 分散
  一台大主機   關聯DB引擎包辦   NoSQL放棄一致(2009)  雲   平台工程(再集中,回擺)
  單體 ◄──────────────────────────────────────────► 微服務
  一個process   SOA   微服務(2014)   ……   「回到單體」反思(2023+,回擺)
  手工 ◄──────────────────────────────────────────► 自動
  手動點機房   腳本   IaC(Terraform 2014)   ……   （下一輪在 ch20）
```

逐條讀這四條鐘擺，重點不是「現在擺到哪」，是**每次回擺都帶著上一輪的教訓**：

**效能↔生產力**（ch03 起）。FORTRAN 1957 是這條鐘擺有記載的第一次大擺動——往「生產力」擺，賭「夠聰明的編譯器替我扛效能」。從此每一次往生產力擺（高階語言、GC、ORM、雲），都靠同一個賭：瓶頸不在這層、損失可接受。賭贏就一路擺過去；賭輸（瓶頸真的落在這層）就回擺——你在 hot path 上手寫一段繞過 ORM 的原生 SQL，就是一次小回擺；你那次為了 RDS CPU 降 40% 去手動調 cache、讀執行計畫，也是。Knuth 1974 那句話（下一節細講）講的正是這條鐘擺的紀律：97% 的時候待在生產力那端別亂動，但要抓住關鍵 3% 回擺去手工最佳化的機會。

**集中↔分散**（ch13/16/18）。資料先是集中——關聯式引擎包辦一切（ch13）。然後 NoSQL 在 2009 年前後**主動放棄**一致性、往分散擺，因為規模逼的（CAP）。雲一方面是集中（運算集中到亞馬遜的機房）一方面是分散（你的服務跨網路）。然後平台工程（ch18）又把散在各團隊的維運複雜度**再次集中**搬進平台團隊——鐘擺回擺到集中，但帶著上一輪的教訓：這次集中的不是機器，是認知負荷。

**單體↔微服務**（ch17）。單體 → SOA → 微服務（2014 Lewis & Fowler 定義文）往「分散」擺到極致，然後——截至 2026 年，業界出現明顯的「回到單體」反思（modular monolith／有人叫 majestic monolith）。這裡要 hedge 得很小心：最常被引用的 Amazon Prime Video 2023 案例（一個團隊把影片品質分析的微服務合回單體、據報導大幅降本），當時就有人（含 AWS 前 VP Adrian Cockcroft）指出那其實更像「微服務的一次重構」而非真退回單體，而 DHH 倡議 majestic monolith 也早於這案例多年。所以準確的說法是：**這不是「微服務錯了」，是鐘擺帶著教訓回擺**——教訓正是 ch17 那筆帳：微服務的稅只在「牆是組織牆」時才划算，回擺修正的是「規模不對的早採用」，不是微服務本身。

**手工↔自動**（ch16）。維運從手動點機房 → 腳本 → IaC（Terraform 2014）一路往自動擺。這條鐘擺的下一次大擺動是 AI 輔助編程，那是 ch20 的主場，這裡不展開。

附帶一句給正要轉職 AI Infra 的你：這四條鐘擺在你要去的那個世界一條都沒缺席。GPU 程式從手寫 CUDA kernel（效能端）擺向 PyTorch 這類高階框架（生產力端）；推論服務從自己刻，擺向 vLLM 這類現成引擎。然後——當你撞上 GPU 記憶體不夠、或吞吐量卡死，框架那層「你不用管記憶體」的抽象就在邊緣漏了，你被迫跳回底層去算張量怎麼擺、KV cache 的記憶體怎麼配。那一刻，和你 debug N+1、追冷啟動是同一條規律：抽象在邊緣漏底，把你打回它底下。差別只在於，LLM serving 是這條七十年長河目前流到的最前緣——你未來的工作，就是站在那個浪頭上（這條線 ch21 收）。

看懂鐘擺，你就回答了本章標題那半句——**為什麼我們週期性地回到底層**。每一次往「更高的抽象／更分散／更自動」擺，都是在賭「底層的細節可以不管了」；而 Joel 的定律保證這個賭遲早會輸一次：抽象在邊緣漏底，底層細節頂穿上來，於是我們被迫回擺、回到底層去處理那個漏出來的東西。回擺不是退步，是還上一輪欠的帳。**見過鐘擺擺過幾輪的人，不會把每個新詞都當革命**——他會問：這是新東西，還是上一輪的回擺？它在賭哪個底層可以不管？那個賭，這次算得過帳嗎？

這正是資深工程師必須懂底層的原因——不是懷舊，是因為鐘擺保證你會被打回底層。抽象漏的那一刻、賭輸的那一刻，能跳下去的人值錢，跳不下去的人只能等別人來救。

## 幾句被反覆驗證的老話

回望的尾聲，收三句被引用到爛、卻幾乎每次都被引錯的老話。它們之所以四十年不過時，是因為它們講的是人與複雜度的關係，不是某個會過時的技術——而它們之所以被引錯，是因為斷章取義比讀原文容易。把它們的原意校準回來，本身就是這本書教你的事：分得清「廣為流傳」和「經得起查證」。

**「過早最佳化是萬惡之源」——Knuth，1974。** 這是被斷章取義最嚴重的一句。完整的脈絡是 Donald Knuth 在 1974 年的論文〈Structured Programming with go to Statements〉（刊於 *ACM Computing Surveys*）裡寫的，原文是：

> 「We should forget about small efficiencies, say about 97% of the time: premature optimization is the root of all evil. **Yet we should not pass up our opportunities in that critical 3%.**」
> （我們應該忘掉小的效率提升，大約 97% 的時候是這樣：過早最佳化是萬惡之源。**但我們不該放過那關鍵 3% 的機會。**）

注意那個被砍掉的後半句。流行版只留「過早最佳化是萬惡之源」，聽起來像「不要最佳化」——這是徹底的誤讀。Knuth 的原意是**有紀律的最佳化**：97% 的時候別為了小效率把程式搞醜，但要有本事辨認出那關鍵 3%、在那裡狠下功夫。這正是你的工作方式：你先用 NestJS/ORM 把東西做出來（待在生產力那端），再**用 profiling 找出那關鍵 3%**，只對它回擺去手動最佳化（RDS CPU 降 40% 那次）。你做的不是「不最佳化」，是 Knuth 真正說的那件事。一個歷史小註：Knuth 1989 年〈The Errors of TeX〉裡把這句稱為「Hoare's Dictum」、歸給 Tony Hoare，但 1974 原文並沒引 Hoare——所以引用時說「Knuth 1974，後來他自己歸給 Hoare」最穩。

**「電腦科學只有兩件難事：cache 失效與命名。」** 這句一般歸給 **Phil Karlton**（Netscape 的工程師），Martin Fowler 等人採此歸屬。但要老實說：它的**確切原始出處與年代無一手記載**——Fowler 自己說找不到滿意的原始 URL，網路上最早見於 Tim Bray 的部落格。所以正確的引用分寸是「一般歸於 Phil Karlton」，別給確切年份。撇開歸屬不談，它之所以準，是因為點名的兩件事都是**搬不掉的本質複雜度**：cache 失效（你的多層 cache library 半夜被 invalidation bug 叫起來那次，ch01 的招牌帳單）、命名（替一個會變的東西取個不會過時的名字，本質上是在預測未來會怎麼變，跟 ch06 切模組邊界是同一個難題）。後人愛加一句「……以及差一錯誤」湊成三件——那是後加的梗，不是原話。

**「Worse is Better」——Richard Gabriel，概念 1989、正式刊於 1991。** Richard P. Gabriel 在〈Lisp: Good News, Bad News, How to Win Big〉裡提出（核心想法 1989，1990 年 EuroPAL 會議公開講，其中〈The Rise of Worse is Better〉一節 1991 年刊於 *AI Expert* 後被廣傳）。他對比兩種哲學：「the right thing」（MIT 派，追求正確、完整、一致，哪怕複雜）和「worse is better」（New Jersey 派，寧可介面與實作都簡單，正確性與完整性可以打折）。他觀察到的反直覺事實是：**往往「worse」的那個贏**——Unix 和 C 比當年那些更「正確」的系統簡陋，卻活了下來、擴散開來，因為簡單的東西容易移植、容易上手、容易先跑起來占領生態再慢慢改好。這句話對你不抽象：它正是 ch14 開源擴散、ch15 Web 用無專利 HTML 打贏專有圍牆花園、乃至整個「先求有再求好」敏捷直覺的祖宗級表述。一個完美但沒人用得起來的抽象，輸給一個有缺陷但人人能上手的——這也是一種帳：把「做到完美」的複雜度，換成「先簡單地活下來」的生存力。

這三句話為什麼四十年還準？因為它們講的都不是技術，是**人與複雜度打交道的規律**：人腦的注意力該花在關鍵 3%（Knuth）、人最難的是預測未來與管理快取（Karlton）、人會選擇能先上手的而非最正確的（Gabriel）。技術會過時，這些規律不會——這正是 ch01 拋出、ch21 要收的第四條主線「永恆真話」的預演。

## 複雜度搬去哪了

這一章特別——它本身就在談洩漏，所以這一段不結某個技術的帳，而是結**洩漏定律本身的帳**。「抽象會洩漏」作為一個你從此會戴上的眼鏡，它消滅了什麼負擔、又製造了什麼？

| 項目 | 內容 |
|---|---|
| **消滅的負擔** | 「為什麼又是我在 debug 別人的抽象」這種挫敗的隨機感。一旦你看懂洩漏是守恆的必然、不是誰做壞了，你 debug 時就有了地圖：每層抽象「搬走了什麼／在哪洩漏」是可預測的（就是上面那張總帳表），出事時你知道往哪一層跳，而不是亂槍打鳥 |
| **製造的新負擔** | 這副眼鏡會逼你**對你依賴的每層抽象都欠一筆「學習債」**——Joel 說抽象省了做事的時間、沒省學習的時間。你用 ORM 就得懂 SQL、用雲就得懂伺服器、用 K8s 就得懂網路與排程。不能因為「平常用不到」就不學，因為洩漏專挑你沒準備的那層發生 |
| **它在哪洩漏（這條定律自己的洩漏）** | 當你把它當宿命論用時——「反正抽象都會漏，那就別抽象了、回去手刻一切」。這是誤讀。洩漏是必然，但抽象**仍然是淨賺**：你享受 ORM 的生產力六年，只在它漏的少數時候跳下去，這筆帳大多時候划算（ch04 GC 那筆帳就是這樣：免費享受六年，偶爾付一次 GC pause 的帳）。把「會洩漏」讀成「不該抽象」，跟把 Dijkstra 讀成「GOTO 邪惡」、把 Brooks 讀成「永遠不能加人」是同一種病——把一個關於權衡的觀察，僵化成一條全稱禁令 |
| **這一跳的帳單一句話** | 你用「對每層抽象欠一筆學習債、並隨時準備跳到底層」的負擔，換來了「在抽象洩漏的那一刻仍然有能力把火滅掉」的價值——而這個能力，正是資深工程師之所以是資深工程師的核心。洩漏不是 bug，是你這份工作存在的理由 |

把這一段接回全書：前十七章每一章都在示範「複雜度搬去哪了」，這一章告訴你**那些被搬走的複雜度後來怎麼了——它們從抽象的邊緣漏了回來，構成了你的 debug 人生**。守恆是「搬家」，洩漏是「搬過去的東西又漏回來」，兩句話是同一條定律的前半與後半。ch21 會把這條定律和另外三條主線（交織、鐘擺、永恆真話）一起收進終局的口試。

## 紙上推演

> 純敘事書，沒有可跑的 lab。下面每題請**口頭講出來**或寫下來，講得清楚才算過。先自己想，再看解答。

### 推演一：用「抽象洩漏」拆解你最近一次跳到底層 debug 的經歷 **[20 分鐘] ★★**

從你最近半年裡，挑一次「不得不跳到比平常更底層去 debug」的經歷（讀執行計畫、抓封包、看 GC 快照、追 K8s 事件……都行）。用本章的框架精確回答：(a) 是**哪一層抽象**漏了？(b) 它平常替你藏的是哪塊複雜度、這次漏出來的具體是什麼？(c) 對照本章那張總帳表，這次洩漏對應到哪一章的哪一格？(d) 你能跳下去解決它，靠的是哪個「本來可以不學、但你學了」的底層知識？

#### 推演解答

合格的回答要能把這四點串成一條因果，而不是只描述事件。示範（用 ORM N+1 那類經歷）：

(a) 漏的是 **ORM 這層抽象**（它建在 SQL 之上，SQL 又建在儲存引擎之上——一次漏穿了兩層）。

(b) ORM 平常替你藏的偶然複雜度是「怎麼把物件操作翻譯成正確的 SQL 字串」；它藏不住的本質複雜度是「這個操作到底要碰多少資料、發幾次往返」。這次漏出來的具體形態，是一個你以為「拿一個物件再拿它的關聯」的迴圈，底下被展開成 1+N 句查詢，往返次數隨資料量線性爆炸。

(c) 對照總帳表，這是 **ch13 那一列、「表↔物件（ORM）」那一格**：N+1 query，「你以為在操作物件，ORM 偷偷對每個物件發一句 SQL」。它也是 ch01 預告的全書縮影。

(d) 你能解，靠的是**你懂 SQL、懂一次 JOIN 和 N 次查詢的成本差異、懂怎麼讀 query log**——這些是「用 ORM 的人本來可以不學」的知識。如果你只會用 ORM、不懂它底下發了什麼，這個 bug 對你就是個無解的「莫名其妙變慢」。這正是 Joel 那句「抽象省了做事的時間、沒省學習的時間」的字面演出：你省下了寫 SQL 的時間，但沒省下懂 SQL 的必要。

評分重點：講得出 (d) 才算真懂。很多人能描述事件、甚至能修好，但說不清「我靠的是哪個底層知識」——說不清，下次換一層抽象漏、你就又是亂槍打鳥。

### 推演二：論證「過早最佳化是萬惡之源」被濫用成什麼、Knuth 原意是什麼 **[15 分鐘] ★★**

有同事在 code review 裡用「過早最佳化是萬惡之源」這句話，否決了你一個明顯合理的效能考量（比如你想給一條已知的熱路徑加 index）。請用 Knuth 1974 的**完整原文**反駁他，並說清楚這句話被砍掉的後半句改變了什麼。注意：引用時你該怎麼標年份與歸屬才嚴謹？

#### 推演解答

反駁的核心是把被砍掉的後半句補回去（完整原文見上一節）：流行版砍掉了「**Yet we should not pass up our opportunities in that critical 3%**」，於是「有紀律地分辨何時該最佳化」被偷換成「不要最佳化」。Knuth 主張的是兩件事的組合：97% 的時候別為了小效率把程式搞醜，**但要抓住那關鍵 3% 的機會**。這句話不是反最佳化，是**反「沒有證據就到處微最佳化」**。

回到 code review 現場：你給「**已知的熱路徑**」加 index，根本不是 Knuth 說的「過早」——你不是在非關鍵處瞎調，你是在辨認出來的關鍵 3% 動手。真正的過早最佳化是「還沒 profiling、沒任何證據說這是瓶頸，就憑感覺把程式扭曲成難讀的樣子去省幾個 cycle」。你的同事把這句話用反了：他用一句**主張要在關鍵處最佳化**的話，去否決了一個**正是在關鍵處最佳化**的提案。引用時給確切的論文與年份（「Knuth 1974〈Structured Programming with go to Statements〉」），不要只說「Knuth 說過」。

### 推演三：盤點你技術棧裡的三個鐘擺，各自現在擺到哪一邊 **[25 分鐘] ★★★**

從你現在維運的技術棧裡，挑三個本章談過的鐘擺（效能↔生產力、集中↔分散、單體↔微服務、手工↔自動，四選三）。對每一個：(a) 你目前**待在哪一邊**、為什麼選那邊？(b) 你選那邊時，**賭的是哪個底層可以不管**？(c) 如果那個賭哪天輸了（抽象在那裡漏了），你會被打回**哪一層底層**去處理？

#### 推演解答

沒有標準答案，但合格的回答每一條都要能講出「賭注」和「輸了會被打回哪」，不能只說「我用了 X」。示範三條：

**效能↔生產力**：(a) 我待在生產力那端——用 NestJS、用 ORM，因為團隊交付速度比那幾毫秒值錢。(b) 我賭的是「絕大多數路徑的瓶頸不在語言/框架/ORM 這層，而在 I/O 與資料庫」。(c) 賭輸時（某條熱路徑的瓶頸真的落在 ORM 生成的爛 SQL 上），我被打回 **SQL 與執行計畫那一層**——讀 `EXPLAIN`、手寫原生查詢繞過 ORM、加 index。這就是 Knuth 的關鍵 3%。

**單體↔微服務**：(a) 我待在微服務那端——服務按業務能力切、團隊自治。(b) 我賭的是「我的牆是組織牆，人夠多，分散式的稅付得划算」（ch17 的判準）。(c) 賭輸時（其實團隊沒那麼大、或邊界切錯成了分散式單體），我被打回**網路與分散式一致性那一層**——處理逾時、冪等、最終一致性對帳、跨服務 trace，而且因為根因是 Conway's Law（組織），重拆程式沒用，得改組織。這就是「回到單體」反思要修正的東西。

**手工↔自動**：(a) 我待在自動那端——用 CloudFormation/IaC 管基礎設施。(b) 我賭的是「宣告式的基礎設施定義能正確對應到實際開出來的資源、且供應商的抽象不漏」。(c) 賭輸時（某個 managed service 的底層行為穿透上來、或冷啟動咬人、或帳單失控），我被打回**雲供應商的底層語意那一層**——去讀那些本該被 IaC 藏起來的 service 細節、IAM 模型、計費模式。

評分重點：(b) 和 (c) 是這題的肉。能講清楚「我賭了什麼、輸了會掉進哪一層」，就證明你不是把抽象當魔法在用，而是清楚自己在做一筆有風險的交易——這正是本章要你戴上的眼鏡。

## 自我檢核

口頭自答，講得出來才算過關，講不清楚＝還沒懂。優先答「為什麼」與「如果不這樣會怎樣」。

1. 用一句話講出 Joel Spolsky 2002 那條定律的原文精神，並說清楚「non-trivial」和「to some degree」這兩個限定詞各自排除了什麼、強調了什麼。
2. 為什麼說「抽象洩漏」是「複雜度守恆」的**必然推論**，而不是工程沒做好？請用 Brooks 的本質/偶然複雜度這把刀來證明——洩漏發生在哪兩塊複雜度的交界？
3. 不看書，憑記憶說出全書總帳表裡**至少五筆**帳，每筆都講「搬走了什麼／在哪洩漏」。如果你發現自己記得的「在哪洩漏」全是你親手 debug 過的——這說明了什麼？
4. 為什麼總帳表裡 ch10 那一格是空的（沒有「搬走了什麼」）？這個空格為什麼反而是整張表的錨？
5. 「為什麼資深工程師要懂底層」——請用「鐘擺保證你會被打回底層」這個論點回答，而不是用「懂得多比較好」這種空話。
6. 挑一條鐘擺（如單體↔微服務），講出它歷史上擺過哪幾輪、最近一次回擺（「回到單體」反思）帶著上一輪什麼教訓。引用 Amazon Prime Video 2023 案例時，你必須加上哪個 hedge？
7. 「過早最佳化是萬惡之源」被砍掉的後半句是什麼？補回去之後，這句話的意思從「不要最佳化」變成了什麼？這跟你 profiling 後只動關鍵路徑的工作方式是什麼關係？
8. 「Worse is Better」為什麼能解釋「一個有缺陷但人人能上手的抽象，贏過一個完美但沒人用得起來的抽象」？舉一個本書前面章節的例子（提示：Web 怎麼贏圍牆花園、開源怎麼擴散）。

## 延伸閱讀

- **Joel Spolsky,《The Law of Leaky Abstractions》(2002-11-11)** — 本章的源頭，原文很短，半小時讀完。讀它怎麼用 TCP/SQL/網路檔案三個例子把「都會洩漏」講成一條定律，特別留意結尾那句「abstractions save us time working, but they don't save us time learning」——那是這條定律最實際的推論。連結：https://www.joelonsoftware.com/2002/11/11/the-law-of-leaky-abstractions/
- **Donald Knuth,〈Structured Programming with go to Statements〉(1974, *ACM Computing Surveys*)** — 「過早最佳化是萬惡之源」的原始出處。讀 p.268 那一段的**完整**上下文（含「97%」與「critical 3%」），你會發現你以為你懂的那句話，意思跟流行版相反。論文本身也是結構化編程辯論的一手史料（回收 ch05）。
- **Richard P. Gabriel,《The Rise of Worse is Better》(概念 1989／刊 1991)** — 「worse is better」的正式陳述。讀它對比「MIT 派（the right thing）」與「New Jersey 派（worse is better）」，理解為什麼「先簡單地活下來」常常打敗「做到完美」。免費全文：https://www.dreamsongs.com/WorseIsBetter.html
- **Martin Fowler,《TwoHardThings》(bliki)** — 關於「cache 失效與命名」這句話，Fowler 親自考據歸屬、坦承找不到一手出處的短文。讀它示範「廣為流傳 ≠ 經得起查證」的引用自律——這本身就是本書一直在教你的事。連結：https://martinfowler.com/bliki/TwoHardThings.html
- **Frederick Brooks,《No Silver Bullet》(1986)** — 本章那把「本質/偶然複雜度」的刀的原典（已在 ch10 詳讀）。回頭再讀一次，這次帶著「洩漏發生在兩塊複雜度的交界」這個本章視角，你會讀出 ch10 第一次讀時沒讀到的東西。
