# ch03 — 把事件組合起來：加法、餘事件、容斥

> **本章解決什麼問題**：ch02 給了你機率的三張臉與三條公理，但那是「單一事件」的層級。真實世界的問題幾乎都是組合的——「至少一台機器當機」「至少一個請求逾時」「兩人撞生日」，這些都不是單一事件，而是好幾個事件用「且」「或」「非」黏起來的複合體。本章教你把這些黏法算對，而且只用公理推得出來、不需要任何新假設。它也是 ch04（條件與獨立）的暖身：這裡你會第一次認真用到乘法，但**刻意先假設獨立、把獨立的微妙留給下一章**。本書最有名的反直覺案例之一——生日問題——在這裡完整破解。

## 從你已知的出發

你天天在算組合事件，只是沒把它寫成機率。

「這個 batch job 跑 200 個 task，每個失敗率 0.5%，整批至少掛一個的機率多少？」你直覺知道答案不是 0.5%，是高得多——但具體多高、為什麼，你大概沒算過。「兩個 AZ 的備援都掛掉」是交集；「至少一個健康檢查失敗」是聯集；「這次重試全部沒成功」是一連串失敗的乘法。這些黏法你每天在腦袋裡毛估，本章把它變成你能在白板上推給同事看的東西。

有一個體感你早就有了：**規模一大，罕見事件就變成常態**。單一請求逾時的機率小到可以忽略，但你一秒打十萬個請求，「這一秒內至少有一個逾時」幾乎是必然——這就是為什麼你的 dashboard 上 P99.9 永遠有東西在閃。尾端不是壞運氣，是算術。一個事件單次機率再小，只要你重複夠多次，「至少發生一次」就會悄悄爬過 50%、然後逼近 1。這章會給你一個公式，讓你能算出「重複幾次之後，這件罕見的事就由稀客變成常客」——而那個次數，通常比你猜的小得驚人。

還有一個反過來的體感：你以為兩個獨立備援「同時掛掉」的機率是兩個小機率相乘、小到不用擔心——這個乘法本章會教你算，但**它對不對，取決於那個「獨立」是真的還是你一廂情願**。本章先用「真獨立」把乘法練熟，下一章再來拆穿「假獨立」如何讓真實 outage 鑽你空子。

先校準一下：這章用到的組合計數（C(n,k)，n 取 k）你大學看過。本章只把它當成一個你已經認得的記號快速擦亮，**不開排列組合課**——那會把這本書變成考試解題訓練，那不是我們要的（細節真要補，見延伸閱讀）。

## 三個黏法：交集、聯集、餘事件

機率裡的事件就是樣本空間 Ω 的子集合，所以「組合事件」本質上就是集合運算。三個黏法，對應三個你早就會的邏輯詞：

- **交集（intersection）** A∩B —— 「A 且 B」，兩件事都發生。
- **聯集（union）** A∪B —— 「A 或 B」，至少一件發生（注意：機率裡的「或」是包含性的，A 發生、B 發生、兩個都發生，全算）。
- **餘事件（complement）** Aᶜ —— 「非 A」，A 沒發生。

餘事件最便宜，先講它，因為它是本章的主力武器。從公理 P(Ω)=1，而 A 與 Aᶜ 互斥、合起來就是整個 Ω，所以：

```text
P(Aᶜ) = 1 − P(A)      ← 一條線，但它是本章最常用的一招
```

這條看起來幼稚到不值得寫下來，但它是整章的引擎。原因很簡單：**很多問題的反面比正面好算太多**。「至少一個」攤開來算是一場惡夢（恰好一個、恰好兩個、恰好三個……全部加起來），但它的反面只有一種情形——「一個都沒有」。後面你會看到，生日問題和「至少一次逾時」都是被這條一行公式撬開的。

## 加法：互斥與不互斥，差在一個重疊

聯集 P(A∪B) 怎麼算？這裡是大家第一個踩的坑。

如果 A 和 B **互斥（mutually exclusive）**——不可能同時發生，A∩B=∅——那聯集就是單純相加：

```text
P(A∪B) = P(A) + P(B)      ← 只在互斥時成立
```

這正是 ch02 那條公理（互斥事件機率可加）。例：擲一顆骰子，「點數是 3」和「點數是 5」互斥，P(3 或 5)=1/6+1/6=2/6。乾淨俐落。

但大多數時候 A 和 B **不互斥**——它們有重疊。這時候你不能直接加，因為重疊的部分被你數了兩次。正確的式子是**容斥原理（inclusion-exclusion principle）**：

```text
P(A∪B) = P(A) + P(B) − P(A∩B)      ← 把重疊的部分扣回來一次
```

直覺是純粹的數數。想像你把 A 區域裡所有結果的機率加一遍、再把 B 區域加一遍——落在重疊 A∩B 裡的結果就被你加了兩次，所以要扣掉一次。多扣？不會，重疊只被多算了一次，所以只扣一次剛好。互斥只是 P(A∩B)=0 的特例，這兩條其實是同一條。

```text
        ┌───────────────┐
        │   A           │
        │       ┌───────┼───────┐
        │       │ A∩B   │       │
        │       │ (加了 │       │
        └───────┼  兩次)│   B   │
                │       │       │
                └───────┴───────┘
   P(A∪B) = P(A) + P(B) − P(A∩B)
            ↑      ↑      ↑
          整塊A  整塊B  扣掉被算兩次的中間
```

工程例子：一個請求可能「逾時」（A，機率 0.03）或「回 5xx」（B，機率 0.02），而有些逾時的請求最後也被記成 5xx（重疊，A∩B，機率 0.01）。「這個請求出事（逾時或 5xx）」的機率是 0.03+0.02−0.01=0.04，不是 0.05。你要是忘了扣那 0.01，就會系統性高估錯誤率——把同一批壞請求算了兩遍。

三個以上事件時容斥會長出更多項（加單個、扣兩兩重疊、補三個一起的重疊……正負交錯），但本書不展開那個一般式——你需要它的時候，會發現用餘事件繞過去通常更省力，下一節就是。

> **小史**：容斥這招通常歸給棣美弗（de Moivre），出現在他 1718 年的《The Doctrine of Chances》（史上第一本機率教科書）裡，他用它處理「巧合」類問題（兩副牌洗開有沒有同位置同張的經典題）。不過嚴格的一般公式更晚——達席爾瓦（Da Silva）1854、西爾維斯特（Sylvester）1883 才正式寫下，所以它有時被叫「Da Silva–Sylvester 公式」（2026-06 查證）。

## 「至少一個」幾乎都從反面算

現在把餘事件和乘法湊起來，解決工程裡最常出現的一類問題：**至少一個**。

直接算「至少一個 X 發生」是個陷阱，因為「至少一個」藏著一堆情形（恰好一個、恰好兩個、……）。但它的反面只有一種：**一個都沒發生**。所以：

```text
P(至少一個發生) = 1 − P(全都沒發生)
```

而「全都沒發生」如果各事件**獨立**，就是一串「沒發生」的機率連乘。先把「獨立」這個詞記在心上——本章一律假設它成立，**它何時成立、何時是你的一廂情願，是 ch04 的主題**。在那個假設下：

設有 n 個事件，每個發生機率都是 p，彼此獨立。每個「不發生」的機率是 (1−p)，n 個都不發生就是 (1−p)ⁿ，於是：

```text
P(至少一個發生) = 1 − (1−p)ⁿ
```

這條公式短，但它解釋了你整個職業生涯的尾端焦慮。先看一個你問過的問題：100 台機器，每台**當天**故障率 0.1%（p=0.001），整個機房「當天至少一台故障」的機率？

```text
P(至少一台壞) = 1 − (1 − 0.001)¹⁰⁰
             = 1 − 0.999¹⁰⁰
             ≈ 1 − 0.9048
             ≈ 0.0952 ≈ 9.5%
```

（0.999¹⁰⁰ 怎麼來的：ln(0.999)≈−0.0010005，乘以 100 得 −0.10005，取 exp 約 0.9048。下一節有更白話的近似法。）

單台 0.1% 你大概覺得「很穩」，但一百台擺在一起，「今天有一台出事」接近一成——一個月三十天，幾乎天天都有東西要你看。你的直覺低估它，是因為你盯著「單台 0.1%」這個小數字，卻沒在腦袋裡做那個乘冪。**罕見 × 規模 = 日常**，這就是 SRE 的數學宿命。

### Worked example：p=0.01 重複 n 次，何時越過一半

把上面那條公式當成一台「罕見事件變常客」的計算機。基準設定：某事件單次機率 p=0.01（百分之一，夠罕見了），獨立重複 n 次，問「至少發生一次」的機率 1−0.99ⁿ 在哪裡越過 0.5。算三個關鍵點：

```text
n        0.99ⁿ        1 − 0.99ⁿ      = P(至少一次)
────────────────────────────────────────────────
10       0.9044       0.0956          ≈ 9.6%
69       0.4999       0.5001          ≈ 50.0%  ← 剛好過半！
100      0.3660       0.6340          ≈ 63.4%
```

逐項驗算（用 ln(0.99)≈−0.0100503，再取 exp）：

```text
n=10:   −0.0100503 × 10  = −0.100503  → exp = 0.9044  → 1−0.9044 = 0.0956
n=69:   −0.0100503 × 69  = −0.693472  → exp = 0.4999  → 1−0.4999 = 0.5001
n=100:  −0.0100503 × 100 = −1.005034  → exp = 0.3660  → 1−0.3660 = 0.6340
```

看 n=69 那一行：一件單次只有 1% 機率的事，你只要重複 **69 次**，「它至少發生一次」就過半了。不是 50 次（那是把 1% 線性外推成 50% 的天真直覺，0.01×50=0.5），也不是 100 次（那會超過一半不少）。69 這個數字本身不重要，重要的是它比你猜的小——線性直覺（「1% 乘以次數」）在這裡會騙你，因為機率不會線性疊加，它是 1 減去一串連乘。

有個好用的速算：當 p 很小、n 不太大時，1−(1−p)ⁿ ≈ np（這是上式的一階近似，p² 以上的項小到可丟）。n=10 時 np=0.10，跟精確的 0.0956 很接近；但 n 越大近似越糟（n=100 時 np=1.0，但機率不可能是 1，精確值是 0.634）——因為近似忽略了「同一件事發生兩次以上」被重複算的修正。np 是上界，真值總在它下面。**這個 np 近似下一章還會以另一個面貌回來**（卜瓦松的 λ，見 ch10），先記著這個 √ 一樣會反覆出現的老朋友。

## 連續失敗的乘法：每次小機率，多次就不小

「至少一個」的反面是「全都失敗」，而「全都失敗」就是把失敗連乘。這個乘法本身也值得單獨看一眼，因為它解釋了兩個你熟悉的現象。

**現象一：多層快取全部 miss。** 三層快取，命中率分別 90%、80%、70%，一個請求「三層全部 miss、最後打到 DB」的機率是各層 miss 連乘：0.10×0.20×0.30=0.006，0.6%。每一層都不怎麼可靠（最差那層三成 miss），但疊三層之後，「全部漏掉」只剩 0.6%。乘法對你有利——這正是分層快取的意義，**用多個平庸的東西串出一個可靠的東西**。

**現象二：n 台各 p 故障，全部存活。** 反過來，n 台機器各自獨立故障率 p，「全部都活著」是 (1−p)ⁿ。注意這跟「至少一台壞」是同一枚硬幣的兩面：

```text
P(全活) = (1−p)ⁿ
P(至少一台壞) = 1 − (1−p)ⁿ
兩者相加 = 1      ← 餘事件，必然
```

乘法對你有利還是有害，取決於你站哪邊：你要「全部成功」（如冗餘投票要多數存活），乘法幫你把可靠性疊高；你怕「至少一個失敗」（如關鍵路徑上每一環都不能斷），乘法把風險疊高。同一個 (1−p)ⁿ，是祝福也是詛咒。

**但這整套乘法的地基是「獨立」。** 我必須在這裡踩一次剎車：上面每一個連乘都偷偷假設了各事件互不影響。三層快取若共用同一個被打爆的後端、n 台機器若插在同一條跳電的電源上，那「全部 miss」「全部存活」就不是連乘了——共享的故障源讓它們手牽手一起倒。這一章的乘法是在「真獨立」的理想國裡練的；ch04 會帶你出國，看看現實世界的相關性故障如何讓 (1−p)ⁿ 這個漂亮公式整個崩掉。**先學會理想版，才知道現實版差在哪。**

## 生日問題：23 個人就過半，到底為什麼

現在來破解本書最有名的反直覺案例。題目：一個房間裡 n 個人，「至少有兩人同一天生日」的機率多少？要多少人才會過半？

幾乎每個人第一次聽到都會猜一個大數字——183（365 的一半，感覺「要填滿一半的日子」）、甚至更多。**正確答案是 23。** 23 個人，至少兩人撞生日的機率就過半（≈50.73%）。這個落差大到它有個專門的名字叫「生日悖論」，雖然它一點都不悖論，只是你的直覺把問題問錯了。

直覺錯在哪，先講清楚，因為這是整題的關鍵：你心裡默默問的是「**要多少人才會有人跟『我』同一天生日**」——拿你一個人去比所有人。那確實要很多人（每個人只有 1/365 機會撞到你）。但題目問的是「**任意兩人**撞生日」——不是你對全體，是**所有人兩兩互比**。23 個人能湊出的配對數是 C(23,2)=23×22/2=**253 對**，每一對都是一次撞生日的機會。253 次機會，每次成功率約 1/365——這樣一想，過半就不奇怪了。你的直覺低估，是因為它把「253 對」誤聽成「22 個人對我」。

這個「253 對」是第一種視角，給你正確的數量級直覺。但它不是精確算法（這 253 對彼此不獨立，不能簡單把機率相乘），所以我們用第二種視角——**餘事件連乘**——來算出精確值。

### Worked example：餘事件連乘手算 P(無碰撞)

照本章主旋律：「至少兩人同生日」難算，算它的反面「所有人生日**都不同**」。假設一年 365 天、每天等可能、忽略閏日（這些假設讓題目乾淨，現實裡生日分布不均反而會讓碰撞**更**容易，所以 23 是個保守的門檻）。

讓 n 個人**一個一個走進房間**，每進來一個就要求他的生日跟前面所有人都不撞：

```text
第 1 人：隨便哪天都行                  → 365/365 = 1
第 2 人：要避開第 1 人的 1 天          → 364/365
第 3 人：要避開前 2 人的 2 天          → 363/365
第 4 人：要避開前 3 人的 3 天          → 362/365
  ⋮
第 23 人：要避開前 22 人的 22 天        → 343/365
```

所以 23 人「全部生日不同」的機率，是這 23 個分數的連乘：

```text
P(無碰撞) = 365/365 · 364/365 · 363/365 · … · 343/365
```

分子是從 365 往下連乘 23 個數（365×364×…×343），分母是 365²³。一項一項乘起來、看它怎麼往下掉（每一項都比 1 小一點，乘越多掉越多）：

```text
人數 n    新乘進的因子      累積 P(無碰撞)     P(至少一撞)=1−上一欄
──────────────────────────────────────────────────────────────
 1        365/365 = 1.0000    1.0000             0.0000
 2        364/365 = 0.99726   0.99726            0.00274
 3        363/365 = 0.99452   0.99180            0.00820
 5        361/365            ≈ 0.97286            0.02714
10        356/365            ≈ 0.88305            0.11695
20        346/365            ≈ 0.58856            0.41144
22        344/365            ≈ 0.52430            0.47570
23        343/365 = 0.93973   0.49270            0.50730   ◄ 過半就在這！
```

第 23 個人走進來，把累積機率從 0.5243 乘上 343/365≈0.9397，壓到 **0.4927**——這就是 landscape 給的基準值。它一旦掉到 0.5 以下，它的餘事件「至少一撞」就爬過 0.5，變成 **0.5073**。22 人時還是 0.4757（沒過半），23 人時 0.5073（過半），分水嶺就卡在這裡。

兩種視角現在對上了：**配對數 C(23,2)=253** 告訴你「為什麼這麼少人就夠」（253 次機會，不是 22 次）；**餘事件連乘** 給你精確的 0.4927／0.5073。一個給直覺、一個給數字，這正是本書要的「能算、也能講」。

交叉驗算一下這個 0.4927 合不合理。有個漂亮的近似：每一對不撞的機率是 364/365，253 對若「近似獨立」，全不撞約 (364/365)²⁵³。或者用更滑順的指數近似——每對撞生日機率約 1/365，253 對「至少一撞」用 1−(1−1/365)²⁵³ ≈ 1−exp(−253/365)=1−exp(−0.6932)≈1−0.4999=0.5001。近似值 0.5001 跟精確的 0.5073 同一個區，差在那些「不獨立」的零頭——足以確認我們沒算錯數量級。（精確值比近似略高一點點，本來就該如此。）

順帶把這題接回工程：這不是冷知識。雜湊碰撞、UUID 重複、分散式系統裡「兩個節點剛好選到同一個 ID」——全是生日問題。一個 n 位元的雜湊，你不需要產生 2ⁿ 個值才會碰撞，大約 2^(n/2) 個就有一半機率撞上（這就是密碼學裡的「生日攻擊」）。下次有人說「ID 空間有幾十億，不可能撞」，你心裡要響起 253 那個聲音：撞的不是「我對全體」，是「所有人兩兩」。

## 直覺的陷阱

組合事件是機率直覺崩壞的第一個高發區，因為「且／或／非」這三個日常詞，在機率裡的數值行為跟你的語感對不上。

| 錯誤直覺 | 它在哪一步把你帶溝裡 | 怎麼自我察覺 |
|---|---|---|
| 「A 或 B」直接把機率相加 | 忘了扣重疊 P(A∩B)，系統性高估聯集；極端時加出超過 1 的「機率」 | 算出來 >1，或你能想到「A 和 B 同時發生」的情境卻沒扣它 |
| 「至少一次」用 p×n 線性估 | 機率不會線性疊加；p×n 是上界，n 大就爆破 1 | 你的估計 >1（如 0.01×200=2），或你沒寫成 1−(1−p)ⁿ |
| 把小機率事件「重複很多次」當成還是很罕見 | 忽略了 1−(1−p)ⁿ 隨 n 爬升的事實；尾端被低估 | 你說「這麼小不可能發生」卻沒問「重複幾次」 |
| 生日問題：拿「我對全體」想，猜要 183 人 | 把「任意兩人撞」誤聽成「有人撞我」，漏掉 C(n,2) 那一大堆配對 | 你的答案接近 365/2，而沒算過配對數 |
| 連乘 (1−p)ⁿ 算「全部存活」卻沒檢查獨立 | 共享故障源讓事件相依，連乘嚴重低估「一起掛」 | 你用了乘法，卻說不出「為什麼這幾件事互不影響」（ch04 正解） |

最值得停十分鐘的是**生日問題那一格**，因為它示範了一個更通用的教訓：**你的直覺常常在偷偷簡化問題**。你以為你在回答「房間裡有人撞生日嗎」，其實你回答的是「有人撞**我**嗎」——這是兩個不同的問題，差了一個 C(n,2)。每次一個機率結果讓你覺得「不可能吧」，先別懷疑數學，先回去檢查：我是不是在心裡把問題換成了一個比較好想、但不一樣的問題？這個自我察覺的習慣，是這本書想送你的東西裡最值錢的一個。

還有一個微妙處（這裡先點，ch04 深講）：「互斥」和「獨立」是**完全不同、甚至相反**的兩件事，但中文語感容易把它們混成「沒關係」。互斥是「兩件事不能同時發生」——這恰恰是一種強烈的相依（知道 A 發生，就確定 B 沒發生）。獨立是「一件發生與否，不改變另一件的機率」。互斥的兩個正機率事件**一定不獨立**（這個你會在 ch04 的紙上推演親手證一遍）。本章你已經用過兩者：加法用互斥，連乘用獨立——別讓它們在你腦中黏成一團。

## 紙上推演

### 推演題一 [15 分鐘] ★★

200 個 task 的 batch job，每個 task 獨立失敗率 0.5%（p=0.005）。手算「整批至少掛一個」的機率。然後解釋：為什麼一個只有 0.5% 失敗率的 task，整批卻這麼容易出事？再用 np 近似估一次，比較兩個答案。

#### 推演解答

從反面算。「整批至少掛一個」的餘事件是「200 個全部成功」：

```text
P(全成功) = (1 − 0.005)²⁰⁰ = 0.995²⁰⁰
```

算 0.995²⁰⁰：ln(0.995)≈−0.0050125，乘以 200 得 −1.00251，取 exp≈0.3669。所以：

```text
P(至少掛一個) = 1 − 0.3669 ≈ 0.6331 ≈ 63.3%
```

**六成三**。一個失敗率只有 0.5% 的 task，跑 200 個，整批出事的機率超過六成——比丟硬幣還容易。為什麼直覺低估？因為你盯著「0.5% 好小」，卻沒做那個 200 次方。0.5% 的反面是 99.5% 成功，但 99.5% 連乘 200 次，每次都磨掉一點點，乘到後來只剩 37%——剩下的 63% 全是「至少一個壞了」。

np 近似：np=0.005×200=1.0。這給出「≈100%」的荒謬上界（機率不可能是 1），正好示範 np 近似在 np 接近或超過 1 時完全失效——它只在 np 遠小於 1 時堪用。精確值 63.3% 才是真的。**這題的工程教訓**：任何「對每個元素獨立施加小風險、再大量重複」的流程（批次寫入、大規模重試、fan-out 呼叫），你都要用 1−(1−p)ⁿ 來想整體，而不是看單元素的 p 就放心。

### 推演題二 [10 分鐘] ★★

用餘事件解「擲 4 次骰子，至少出現一個 6」。然後對照一個常被誤算的版本——有人會說「每次中 6 的機率 1/6，擲 4 次就是 4×1/6=4/6≈0.67」——指出這個算法錯在哪。

#### 推演解答

正解，從反面算。每擲一次「**沒**擲到 6」的機率是 5/6，四次獨立全沒中是 (5/6)⁴：

```text
(5/6)⁴ = 5⁴/6⁴ = 625/1296 ≈ 0.4823
P(至少一個 6) = 1 − 625/1296 = 671/1296 ≈ 0.5177
```

約 **51.8%**，略過半。

那個「4×1/6=0.67」的算法錯在哪？它在偷偷用**加法**算一個**非互斥**的聯集。「第一擲是 6」「第二擲是 6」……這四個事件**不互斥**——你完全可能兩擲都是 6。直接相加等於把「多次都中 6」的情形重複計算了好幾遍，所以高估（0.67 > 真值 0.518）。要用加法就得套完整容斥（加單個、扣兩兩重疊、補三個、扣四個一起），那會算到天荒地老；用餘事件一行解決。這就是為什麼本章一直唸「至少一個，先想反面」——正面要嘛要容斥的全套、要嘛要錯誤的天真加法，反面只有「一個都沒中」一種情形。

（歷史彩蛋：「擲 4 次至少一個 6」正是機率史的起點之一。17 世紀的賭徒德梅雷靠賭這個賺錢——51.8% 過半，長期穩賺。他後來把賭局換成「擲兩顆骰子 24 次至少出現一次雙 6」以為一樣賺，結果那個只有約 49.1%、長期穩賠，百思不解，去問了帕斯卡（Pascal）——這個困惑間接點燃了帕斯卡與費馬（Fermat）1654 年的通信，也就是 ch01 那封信。你剛算的這題，差點改寫了數學史。）

### 推演題三 [10 分鐘] ★

口頭題：你朋友賭生日問題要 183 個人才會過半（「365 的一半嘛」）。不准寫公式，用一句話的核心洞見＋一個比喻，講到他點頭。

#### 推演解答

模範答案要點（你要能用自己的話講出這兩點，不是背稿）：

1. **核心洞見**：「你問錯問題了。你心裡在算『要多少人才有人跟你同一天生日』——拿你一個去比所有人，那當然要很多人。但題目問的是『**任意兩個人**撞』，是大家**兩兩互比**。23 個人能配出 253 對，每一對都是一次撞的機會。253 次機會，過半就不奇怪了。」

2. **比喻**（擇一）：「就像房間裡握手——23 個人不是 22 次握手，是每兩人都握一次，253 次握手。撞生日跟握手一樣，算的是『對』不是『人』。」或者更工程的版本：「你 ID 空間有幾十億就以為不會撞？撞的不是某個新 ID 對上你指定的那個，是**所有已發的 ID 兩兩之間**——這就是為什麼雜湊只要 2^(n/2) 個值就有一半機率碰撞，不用 2ⁿ。」

常見錯路：很多人會想用連乘的公式去說服朋友，但對方眼神會死掉。這題的重點是**先修好他的直覺**（253 對 vs 22 人對我），數字（0.4927）反而是次要的。能把「他問錯問題」這件事一句話講清楚，才算過關。

## 自我檢核

口頭自答，講得出來才算懂：

1. 為什麼「至少一個」幾乎都從反面（1 減去全沒發生）算？正面直接算難在哪？
2. 容斥原理 P(A∪B)=P(A)+P(B)−P(A∩B) 裡那個「減去」在修正什麼？為什麼是「減一次」不是兩次？
3. 「互斥」和「獨立」差在哪？為什麼互斥的兩個正機率事件反而**不**獨立？（直覺講清楚即可，嚴格證明留 ch04）
4. 生日問題裡，C(23,2)=253 這個數字在直覺上扮演什麼角色？為什麼它能解釋「為何這麼少人就過半」？
5. 1−(1−p)ⁿ 這條公式，為什麼能解釋「規模一大、罕見事件變常態」？np 近似什麼時候堪用、什麼時候會騙你？
6. 「三層快取全部 miss 的機率是各層 miss 連乘」——這個連乘偷偷假設了什麼？如果假設不成立會怎樣？（這題故意逼你預習 ch04）
7. 為什麼「擲 4 次骰子至少一個 6」不能用 4×(1/6) 算？那個加法錯在哪？
