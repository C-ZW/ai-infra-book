# ch04 — 條件與獨立：最容易誤判的兩件事

> **本章解決什麼問題**：機率讓人類直覺翻車最慘的兩個地方，全在這一章——「在已知 B 的條件下看 A」到底改變了什麼（條件機率），以及「A 跟 B 互不影響」這句話的精確意思（獨立）。它接在 ch03「把事件組合起來」之後：上一章你學會把事件用交集、聯集、餘事件拼起來；這一章開始問「一個事件發生了，會不會改變另一個的機率」。它也是整個 Part II 貝氏的引信——本章只把 `P(A|B)` 和 `P(B|A)` 是兩回事這件事擺上檯面，真正的逆轉（貝氏定理）留給 ch05。順便埋兩顆雷：條件獨立 ≠ 邊際獨立（ch19 辛普森悖論會引爆），以及全機率公式（ch05 的分母就是它）。

## 從你已知的出發

你每天都在算條件機率，只是沒這樣叫它。

你開一個查詢：

```text
SELECT failure_rate FROM requests WHERE status = 'timeout';
```

這個 `WHERE` 子句做的事，跟條件機率做的事，一模一樣。它先把整個世界（所有請求）縮到一個子集裡（已經逾時的請求），然後在這個縮小的世界裡重新看故障率。你不是在問「請求逾時又故障的機率」，你是在問「**在已經逾時的前提下**，故障率多少」。分母換了——從「全部請求」換成「逾時的請求」。

這就是條件機率的全部。`P(故障 | 逾時)` 就是一句帶 WHERE 的查詢。

而獨立呢？獨立是另一件事，而且是你已經被它咬過的那件事。

你做高可用設計時都信一條教條：把備援放到不同的可用區（availability zone, AZ），這樣一個掛了還有另一個。背後的算盤是——兩個 AZ 各自的故障是**獨立**的，所以「兩個同時掛」的機率是兩個小機率相乘，小到可以忽略。99% × 99%，同時掛的機率 0.0001，可用度衝到 99.99%，四個九，漂亮。

然後某天它們真的同時掛了。事後檢討才發現，這兩個「獨立」的 AZ 共用同一條光纖、同一個控制平面、同一套 DNS、甚至同一個區域的電力。它們從來就不獨立。你以為自己在乘兩個小機率，其實你乘了一個小機率乘以「1」——因為一個掛的時候，另一個幾乎**必然**跟著掛。

這一章就是要把這兩件你天天在用、卻最容易誤判的事講清楚：條件機率到底改了什麼，獨立到底是什麼意思，以及為什麼「假獨立」是真實 outage 最常見的根因之一。我認為這一章的下半段，是整本書裡跟你工作扣得最緊的一頁。

## 條件機率：把世界縮到 B 裡再看 A

正式定義很短：

```text
P(A|B) = P(A∩B) / P(B)          ← 前提 P(B) > 0
         ─────────
唸法：「在 B 發生的條件下，A 的機率」
```

別急著背。先看它在幹嘛。

原本的樣本空間（sample space）是「所有可能發生的事」，總機率 1。現在我告訴你「B 確定發生了」。這句話把你的世界砍掉了一大塊——所有 B 沒發生的可能性，全部不算數了。你的新世界只剩下 B 這一塊。

但這個新世界的總機率得是 1 才能繼續當機率用，而 B 原本的機率只有 `P(B)`（小於 1）。所以你得把它**重新標準化**：把 B 裡面的每一塊機率，全部除以 `P(B)`，撐大回總和為 1。

`A∩B`（A 跟 B 同時發生）這一塊，在新世界裡的占比，就是 `P(A∩B) / P(B)`。這就是 `P(A|B)`。

```text
原本的世界                    縮到 B 之後的世界
┌──────────────────┐
│        A         │              ┌──────────┐
│    ┌─────┐       │   已知 B     │   A∩B    │  ← 這塊撐大成
│    │ A∩B│   B    │   ────────►  │  (其餘    │     新世界的全部
│    └─────┘       │   把世界      │   被砍掉) │     占比 = 
│                  │   縮到 B      └──────────┘     P(A∩B)/P(B)
└──────────────────┘              總機率重新算成 1
```

注意一件事：`P(A|B)` 跟 `A` 在原世界裡多大**沒有直接關係**，它只關心「在 B 裡面，A 占多少」。A 可能在整個世界裡很小，但只要它幾乎全擠在 B 裡，`P(A|B)` 就可以很大。這正是後面所有反直覺的源頭。

### 一個會錯的小例子：擲兩顆骰子

擲兩顆公正骰子。已知「點數和是 7」，問「其中至少一顆是 3」的機率。

別用感覺。把世界縮到「和為 7」。和為 7 的等可能結果有 6 種：

```text
和=7 的世界（6 種等可能）：
(1,6) (2,5) (3,4) (4,3) (5,2) (6,1)
            ───── ─────
            含 3 的有這 2 種
```

`P(至少一顆是 3 | 和=7) = 2/6 = 1/3`。

你的世界從 36 種縮到 6 種，然後在這 6 種裡數含 3 的。如果不縮世界、用整個 36 種去想，你會算錯——因為「和=7」這個條件已經把很多含 3 的組合（像 (3,3)、(3,5)）排除掉了。條件機率的威力跟陷阱，都在「縮世界」這一步。

## 乘法法則與機率樹：把條件機率倒過來用

把定義的等號兩邊乘以 `P(B)`，馬上得到**乘法法則（multiplication rule）**：

```text
P(A∩B) = P(B) · P(A|B)
       = P(A) · P(B|A)      ← 換個順序拆，結果一樣
```

這條公式的用處是：當你知道「先發生 B 的機率」和「B 之後 A 的機率」，就能拼出「兩件都發生」的機率。一步一步走流程的事情，全靠它。

而且這兩種拆法相等（`P(B)·P(A|B) = P(A)·P(B|A)`）這件小事，等號一搬，就是 ch05 的貝氏定理。先記著這個伏筆。

機率樹（probability tree）就是乘法法則的畫法。每往下走一層、走一條邊，就乘上那條邊的條件機率；走到底的那一片葉子，機率就是沿路所有邊相乘。

舉個你熟的：一個兩階段流程，先過驗證再寫入。驗證成功率 90%，**而且在驗證成功的前提下**，寫入成功率 95%。

```text
                  ┌── 驗證成功 (0.9) ──┬── 寫入成功 (0.95) → 0.9×0.95 = 0.855
   開始 ──────────┤                    └── 寫入失敗 (0.05) → 0.9×0.05 = 0.045
                  └── 驗證失敗 (0.1) ─────────────────────→ 0.1        = 0.100
                                                            ───────────────────
                                                            合計         = 1.000
```

「整個流程成功」= 0.855。注意第二層那個 0.95 是**條件機率** `P(寫入成功 | 驗證成功)`——它預設了驗證已經過。如果你天真地把它當成「寫入本身的成功率，跟驗證無關」，就是在偷偷假設兩階段獨立。多數時候這假設沒問題，但有時候不是：如果驗證和寫入都依賴同一個下游服務，那個服務一抖，兩階段會一起爛——這時候 `P(寫入成功 | 驗證成功)` 跟 `P(寫入成功)` 根本不是同一個數。哪裡藏了獨立假設，等下會專門挖。

## 獨立：被誤解最深的一個詞

工程師講「獨立」通常意思是「沒因果關係、互不影響、八竿子打不著」。這是日常語感，但機率的獨立是個**精確的數值條件**，跟你的因果直覺常常對不上。

正式定義：

```text
A 與 B 獨立  ⟺  P(A∩B) = P(A) · P(B)
```

就這樣。兩件事同時發生的機率，等於各自機率相乘——那它們就叫獨立。沒有「因果」「影響」這些字眼，它純粹是一條乘法等式成不成立。

這個定義有個更好懂的等價說法。把它代回條件機率：

```text
若 P(A∩B) = P(A)·P(B)，則

P(A|B) = P(A∩B)/P(B) = P(A)·P(B)/P(B) = P(A)
         ────────────────────────────────────
獨立 ⟺ P(A|B) = P(A)：知道 B 發生了，完全不改變你對 A 的判斷
```

**這才是獨立的靈魂**：B 這個資訊，對 A 的機率毫無作用。你本來覺得 A 有多少機率，知道了 B，還是那麼多。資訊量為零。

反過來，只要 `P(A|B) ≠ P(A)`——知道 B 之後你對 A 的判斷變了，哪怕只變一點點——它們就**不獨立**。獨立是個很嚴格的條件，不是「差不多沒關係」，是「分毫不差地沒影響」。現實世界裡真正獨立的事件其實很少；獨立通常是個**假設**，是你為了能把機率相乘而下的方便假設。它什麼時候站得住、什麼時候會背叛你，是這一章的重點。

## 獨立和互斥是相反的東西

這是本章第一個會把你帶溝裡的地方，值得停下來想清楚，因為它違反幾乎所有人的第一直覺。

很多人把「獨立」和「互斥（mutually exclusive）」當成近義詞——感覺都是「兩件事沒關係、分開的」。**錯得離譜。它們不但不一樣，方向還是相反的。**

先回憶互斥：A 和 B 互斥，意思是它們**不可能同時發生**，`P(A∩B) = 0`。擲一顆骰子，「出 3」和「出 5」互斥——出了 3 就不可能同時出 5。

現在問：互斥的兩件事獨立嗎？用定義檢查。假設 A、B 各自機率都大於 0（都有可能發生），而且互斥。

```text
互斥：           P(A∩B) = 0
若獨立應該：     P(A∩B) = P(A)·P(B)
兩者要同時成立： 0 = P(A)·P(B)
但 P(A)>0 且 P(B)>0  ⟹  P(A)·P(B) > 0  ≠ 0      ← 矛盾！
所以互斥（且各自機率為正）的兩事件，一定不獨立。
```

用條件機率的語感更直白：如果 A、B 互斥，那麼**一旦 B 發生，A 就絕對不可能發生**了——`P(A|B) = 0`。知道 B 發生這件事，把 A 的機率從原本的正數一口氣砍到 0。這資訊量大到爆表，這哪叫沒關係？這叫**高度相依**。

```text
              知道 B 發生後，A 的機率怎麼變
互斥：    P(A) > 0  ──知道 B──►  P(A|B) = 0     資訊量最大、極度相依
獨立：    P(A)      ──知道 B──►  P(A|B) = P(A)   資訊量為零、毫不相干
```

所以互斥是相依的一個**極端**，不是獨立的同義詞。一句話記住它：**互斥是「你死我活」，獨立是「井水不犯河水」**。你死我活的兩個人，關係能不緊密嗎？

這一條會在「紙上推演」裡讓你親手證一遍。它看起來像數學潔癖，但它其實是在訓練你一個習慣：別憑「感覺沒關係」就把機率相乘，去檢查那條乘法等式到底成不成立。

## 獨立假設為什麼這麼脆：共享的潛在原因

現在進到這一章跟你工作扣最緊、也最值錢的部分。

獨立是個假設。假設會背叛你。它背叛你的最常見方式，叫做**共享的潛在原因（confounder）**——兩件事看起來各走各的，其實背後被同一隻手牽著。

回到開頭那兩個 AZ 的備援。你的算盤是：

```text
備援 A 可用度 99%   →  故障率 P(A 掛) = 0.01
備援 B 可用度 99%   →  故障率 P(B 掛) = 0.01
假設獨立：
P(兩個同時掛) = P(A 掛)·P(B 掛) = 0.01 × 0.01 = 0.0001
整體可用度 = 1 − 0.0001 = 0.9999 = 99.99%   ← 四個九
```

四個九，老闆很高興，SLA 寫得很漂亮。問題是那個乘法——`P(A∩B) = P(A)·P(B)`——只有在 A、B 獨立時才成立。而它們不獨立。

它們不獨立，是因為背後有個你沒算進去的共享原因：**機房電力**。假設這個區域整片停電的機率是每年 1%，而一旦停電，A 和 B **兩個都必掛**（它們吃同一條電）。這個「停電」就是 confounder——一個同時牽動 A 和 B 的潛在原因。

現在重算。設停電事件為 P，`P(停電) = 0.01`，而且 `P(A 掛 | 停電) = 1`、`P(B 掛 | 停電) = 1`。除了停電，A、B 各自還有自己的獨立小毛病（硬碟、bug 之類），故障率還是各 0.01。用全機率的想法把「兩個同時掛」拆成兩種情況：

```text
P(兩個同時掛) 
  = P(停電) · P(兩掛|停電)  +  P(沒停電) · P(兩掛|沒停電)
  = 0.01    · 1            +  0.99      · (0.01 × 0.01)
  = 0.01                   +  0.99 × 0.0001
  = 0.01                   +  0.000099
  = 0.010099
                                           ← 沒停電時才回到「獨立相乘」那一項，
                                             但它被 0.01 這個共享項完全淹沒
```

整體可用度 = 1 − 0.010099 = 0.989901 ≈ **98.99%**，大概就是 **99%，兩個九**。

看清楚發生了什麼：

```text
            P(兩個同時掛)      可用度        幾個九
獨立假設：   0.0001           99.99%        四個九
共享電力：   0.010099         ≈ 98.99%      兩個九（約 99%）
            ────────────────────────────────────────────
            差了約 101 倍       你以為的「多兩個九」是幻覺
```

**你以為自己買了四個九，實際只有兩個九。** 加一個備援，可用度幾乎沒進步——因為決定命運的不是各自的小毛病，是那個你忘了算的共享原因。獨立假設在這裡值「兩個九」，而那兩個九是假的。

這不是我編的數學潔癖。Google 在 2010 年一篇研究自家分散式儲存可用度的論文（Ford 等人，OSDI 2010）發現：**如果不把節點故障的相關性算進去，你對可用度的估計通常會高估至少兩個數量級**（約 100 倍，2026-06 查證）。一百倍。你算出來的「五個九」可能其實是「三個九」。他們的結論寫得很重——故障之間的相關性，對「不可用」的貢獻，遠遠蓋過其他所有因素。換句話說：**在真實系統裡，殺死你的幾乎都是相關性故障，不是獨立故障。**

而這正是 2025 年 10 月 20 日那場 AWS us-east-1 大當機的劇本（2026-06 查證）。很多公司明明把服務做了多 AZ 備援，照樣全趴——因為那些「獨立」的 AZ 共用了 us-east-1 的控制平面、DNS 與 IAM 這些全域服務。底層一個監控網路負載平衡器健康狀態的子系統出事，順著 DNS 與 DynamoDB 端點擴散，多 AZ 的設計擋不住一個跨 AZ 的共享依賴。可用區做的事是「吸收區域內的實體故障」，它從來就不負責擋「邏輯上的共享依賴」。你以為你有兩個獨立的籃子，其實兩個籃子掛在同一根釘子上。

### 用條件機率把這件事說死

獨立失效這件事，用 `P(A|B)` 講最精準。在共享電力的模型裡，問一句：「已知 B 掛了，A 也掛的機率是多少？」

```text
P(B 掛) = P(停電) + P(沒停電)·P(B 自己掛) = 0.01 + 0.99×0.01 = 0.0199
P(A∩B 都掛) = 0.010099                （上面算過）

P(A 掛 | B 掛) = P(A∩B) / P(B) = 0.010099 / 0.0199 ≈ 0.507
                ────────────────────────────────────────
對照 P(A 掛) = 0.0199。
知道 B 掛了之後，A 也掛的機率從 2% 跳到 51%——暴增約 25 倍。
```

如果它們真的獨立，這兩個數字會相等（`P(A|B) = P(A)`）。它們差了 25 倍，這就是「不獨立」的數值鐵證。`P(A|B) ≠ P(A)`，獨立假設當場破產。B 掛掉這件事，帶著「八成是停電」的資訊，而停電會把 A 一起拖下水——這就是 confounder 怎麼把兩件「看起來獨立」的事偷偷綁在一起。

## 條件獨立 ≠ 邊際獨立（一顆留給 ch19 的雷）

還有一種更陰險的，這裡先埋雷、不引爆。

「獨立」其實有兩種，而它們可以**互相矛盾**：

- **邊際獨立（marginal independence）**：不看任何條件，A 和 B 整體上獨立。`P(A∩B) = P(A)·P(B)`。
- **條件獨立（conditional independence）**：在已知第三件事 C 的條件下，A 和 B 獨立。`P(A∩B|C) = P(A|C)·P(B|C)`。

直覺會告訴你這兩個應該差不多——獨立就獨立嘛，加不加條件能差多少？**它們可以完全相反。** 兩件事可以「整體看獨立、但分組看相依」，也可以「整體看相依、但一旦固定某個 confounder 就變獨立」。

上面那個 AZ 的例子其實就是後者的影子：A 和 B 在「不知道有沒有停電」時看起來高度相依（一個掛另一個常跟著掛）；但**一旦你固定了「沒停電」這個條件**，剩下的就只是各自的硬碟 bug，這時候 A、B 就獨立了（`0.01×0.01` 那一項）。是 confounder「停電」在製造相依；把它固定住，相依就消失。

當「整體」和「分組」給出方向相反的結論，那就是**辛普森悖論（Simpson's paradox）**——機率裡最讓人頭皮發麻的反轉。整本書我認為最違反直覺的一頁就在 ch19，那裡會用柏克萊研究所招生的真實資料，讓你看著同一份數據在合併時說「歧視女性」、分系後每一系都說「沒有」。那個悖論的引擎，就是這裡的「條件獨立 ≠ 邊際獨立」。現在你只要記住：**「A 和 B 獨立嗎」這個問題，沒有先講清楚「在什麼條件下」就無法回答。** 加一個條件，答案可能整個翻過來。

## 全機率公式：給 ch05 鋪的最後一塊磚

前面算「兩個同時掛」時，我偷偷用了一個招式：把一個事件按「停電 / 沒停電」拆成兩塊分開算，再加起來。那個招式有名字，叫**全機率公式（law of total probability）**，是 ch05 貝氏定理分母的真身。

它的想法很單純。如果你有一組互斥又窮盡的情況 `B₁, B₂, ..., Bₙ`（彼此不重疊、合起來蓋住整個世界——像「停電」和「沒停電」），那麼任何事件 A 的機率，都可以「分情況討論」再加總：

```text
P(A) = P(A|B₁)·P(B₁) + P(A|B₂)·P(B₂) + ... + P(A|Bₙ)·P(Bₙ)
       ─────────────────────────────────────────────────────
唸法：A 的總機率 = 在每種情況下 A 的機率，按該情況發生的機率加權平均
```

工程白話：**先把世界切成不重疊的幾塊，在每塊裡算 A 的機率，再按每塊的大小加權加起來。** 你算「整體錯誤率」時，如果把流量分成「來自快取 / 來自 DB」兩條路徑分別算錯誤率再加權，你用的就是全機率公式。

```text
                    ┌── B₁ 停電  (0.01) ── A|B₁ (1.00) ──► 0.01 × 1.00 = 0.0100
   兩個同時掛 A ────┤
                    └── B₂ 沒停電(0.99) ── A|B₂ (0.0001)─► 0.99 × 0.0001= 0.000099
                                                          ──────────────────────
                                                          P(A) 合計     = 0.010099
```

這就是剛剛那個 0.010099 的來歷，畫成樹而已。

為什麼這是貝氏的鋪墊？因為貝氏要算的是「**倒過來**」的問題——已知 A 發生了（兩個都掛了），回頭問「是停電造成的機率多大」`P(停電 | 兩掛)`。要算這個分數，分母正是 `P(A)` = `P(兩掛)`，而它就用全機率公式湊出來。`P(A|B)` 和 `P(B|A)` 是兩個天差地別的數字，把前者翻成後者的那把鑰匙，下一章登場。本章到此為止，只負責讓你深信一件事：**`P(A|B)` 和 `P(B|A)` 真的不一樣，別把它們搞混。**

差多少？舉個直覺例子收尾：`P(會講中文 | 是台灣人)` 很高（接近 1），但 `P(是台灣人 | 會講中文)` 沒那麼高（會講中文的人裡，很多不是台灣人）。同一對事件，兩個方向的條件機率可以差十萬八千里。誰當條件、誰被問，順序一換，答案就變。ch05 整章在處理這個翻轉，這裡先讓它扎進你腦子。

## 直覺的陷阱

| 你以為 | 實際上 | 怎麼自我察覺 |
|---|---|---|
| 「獨立」和「互斥」差不多，都是沒關係 | **相反**。互斥是 `P(A∩B)=0`，是高度相依（你死我活）；獨立是 `P(A∩B)=P(A)P(B)`，井水不犯河水 | 你脫口說「它們互斥所以獨立」「不會同時發生所以無關」——停。互斥的事知道一個就否決另一個，資訊量最大 |
| 把備援放不同機器／AZ 就獨立了，故障率直接相乘 | 只要共享任一隱藏資源（電力、光纖、控制平面、DNS、同一支 library、同一次部署），就**不**獨立，相乘嚴重低估同掛機率 | 問自己：「這兩個東西會不會因為**同一個原因**一起壞？」想得出任何一個共享原因，乘法就不能用 |
| `P(A\|B)` 跟 `P(B\|A)` 差不多 | 可以差幾個數量級（會中文 vs 是台灣人）。方向反了，分母就換了 | 你把「檢驗準確率」當成「陽性代表得病的機率」——這是 ch05 檢察官謬誤的種子，先學會聞到味道 |
| 算出 `P(A∩B)` 就直接用 `P(A)·P(B)` | 那是**獨立才能用**的捷徑，不是定義。`P(A∩B)=P(A)·P(B\|A)` 才是永遠成立的 | 你寫下乘法時有沒有先問「這兩個獨立嗎」？沒問就是預設了獨立，預設了就會被 confounder 咬 |
| 「A 跟 B 獨立」是個有絕對答案的問題 | 要看**在什麼條件下**。可以邊際相依、條件獨立，反之亦然（辛普森的引擎） | 有人問「這兩個指標獨立嗎」你能不假思索回答——小心，你大概忽略了某個沒被固定住的 confounder |

招牌謬誤點名：把「互不影響」當成「互斥」、或反過來。這是條件與獨立這一章最經典的翻車，連考卷上都年年有人栽。下次你想說「這兩件事獨立」之前，先做一個動作：在腦子裡問「知道其中一件發生了，我對另一件的判斷會不會改變？」——只要會改變，哪怕一點點，它們就不獨立。這個自問，比背公式有用一百倍。

## 紙上推演

> 全程紙筆，不寫程式。每題先自己推，再看解答。算機率最容易手滑，每個數值請至少用兩種方式對一次。

### 推演題 1：機率樹找出藏起來的獨立假設 **[15 分鐘]** ★★

一個請求要先過「身分驗證」再「寫入資料庫」才算成功。給你：驗證成功率 `P(驗證)=0.8`；在驗證成功的前提下，寫入成功率 `P(寫入|驗證)=0.9`。

(a) 畫出機率樹，算「整個請求成功」的機率。
(b) 老闆說「驗證 80% × 寫入 90% = 72%，所以成功率 72%」——他這樣算，**偷偷假設了什麼**？這個假設什麼情況下會破？
(c) 如果驗證和寫入其實都打同一個下游服務，那個服務有 5% 機率整個掛掉（掛了則兩步都失敗），這個 72% 是高估還是低估？定性說明即可。

#### 推演解答

(a) 樹長這樣：

```text
                  ┌── 驗證成功 (0.8) ─┬── 寫入成功 (0.9) → 0.8×0.9 = 0.72  ✓成功
   開始 ──────────┤                   └── 寫入失敗 (0.1) → 0.8×0.1 = 0.08
                  └── 驗證失敗 (0.2) ──────────────────► 0.2        = 0.20
                                                         ──────────────────
                                                         合計        = 1.00
```

「整個請求成功」只有一條路（兩步都成功）：`P(成功) = P(驗證)·P(寫入|驗證) = 0.8 × 0.9 = 0.72`。

對一下：失敗的機率 = 0.08 + 0.20 = 0.28，1 − 0.28 = 0.72，相符。

(b) 老闆的算式 `0.8 × 0.9` **數字上剛好對**——但只是因為題目給的本來就是條件機率 `P(寫入|驗證)=0.9`。他腦中想的卻是「寫入本身的成功率 90%，跟驗證沒關係」，也就是把 0.9 當成了**邊際**機率 `P(寫入)`，並假設兩步**獨立**。乘法法則 `P(A∩B)=P(A)·P(B|A)` 永遠成立，但 `P(A∩B)=P(A)·P(B)` 只有獨立才成立。這裡碰巧不出錯，是因為我們給的就是條件機率；換個情境，把邊際當條件用，就會錯。重點：**他的乘法藏了「獨立」這個沒講出口的假設。**

(c) **高估**。共享的下游服務就是 confounder。一旦它掛（5% 機率），兩步同時失敗，這是「一起壞」的相關性故障，原本的樹完全沒算進去。真實成功率會比 72% 低，因為多了一條「兩步因共享依賴一起爛」的失敗路徑。獨立模型看不見這條路，於是太樂觀——跟兩個 AZ 共享電力是同一個病。

### 推演題 2：證明「互斥 ⟹ 不獨立」 **[10 分鐘]** ★★

證明：若兩事件 A、B 互斥，且 `P(A)>0`、`P(B)>0`，則 A、B 一定**不**獨立。然後用一句白話講清楚為什麼這在直覺上也應該成立。

#### 推演解答

形式證明（每一步都能口頭講出理由）：

```text
前提：A、B 互斥        ⟹  A∩B = ∅      ⟹  P(A∩B) = 0
前提：P(A) > 0，P(B) > 0 ⟹  P(A)·P(B) > 0           （兩個正數相乘為正）

獨立的定義要求：P(A∩B) = P(A)·P(B)
代入即要求：    0      = P(A)·P(B)
但右邊 > 0，左邊 = 0，矛盾。
∴ 等式不成立 ⟹ A、B 不獨立。                            ∎
```

白話：A、B 互斥，代表**一旦 B 發生，A 就絕對不可能發生**，`P(A|B)=0`。本來 A 有正的機率，知道 B 之後直接歸零——這是天大的資訊，B 對 A 的影響大到把它一票否決。獨立的意思是「知道 B 對 A 毫無影響」（`P(A|B)=P(A)`），跟「一票否決」完全相反。所以互斥的兩件事不但不獨立，還是**相依的極端**。記法：互斥＝你死我活，獨立＝井水不犯河水；前者怎麼可能是後者。

附帶一個常見反問：「那 `P(A)=0` 的話呢？」如果允許機率為 0 的事件，數學上 `P(A∩B)=0=P(A)·P(B)` 會成立，它就「技術上獨立」了——但那是退化情形（一件根本不會發生的事跟誰都「獨立」），沒有實際意義。所以條件要寫明「各自機率為正」。

### 推演題 3：「獨立假設值多少個九」重算 **[20 分鐘]** ★★★

兩個備援各 99% 可用（各自故障率 0.01）。

(a) 假設兩者**獨立**，算整體可用度（要求兩個都掛才算整體掛）。
(b) 現在引入共享電力：停電機率 `P(停電)=0.01`，停電則兩者必掛；除停電外，兩者各自還有 0.01 的獨立故障。用全機率公式算 `P(兩個都掛)` 與整體可用度。
(c) 算 `P(A 掛 | B 掛)`，用它說明 (b) 裡 A、B 為何不獨立、以及這個數字怎麼當「不獨立」的鐵證。

#### 推演解答

(a) 獨立情形：

```text
P(兩個都掛) = P(A 掛)·P(B 掛) = 0.01 × 0.01 = 0.0001
整體可用度  = 1 − 0.0001 = 0.9999 = 99.99%          ← 四個九
```

(b) 共享電力，把「兩個都掛」按停電與否拆開（全機率公式）：

```text
P(兩掛) = P(停電)·P(兩掛|停電) + P(沒停電)·P(兩掛|沒停電)
        = 0.01  · 1          + 0.99    · (0.01 × 0.01)
        = 0.01              + 0.99 × 0.0001
        = 0.01              + 0.000099
        = 0.010099
整體可用度 = 1 − 0.010099 = 0.989901 ≈ 98.99%        ← 約兩個九
```

交叉驗算：共享項 0.01 把獨立項 0.000099 整個淹掉（差兩個數量級），所以可用度從四個九掉到約 99%。`0.010099 / 0.0001 ≈ 101`，相關性讓「兩掛」機率漲了約 101 倍——跟 Google 那篇論文「不算相關性會高估可用度約兩個數量級」完全同一個量級的故事。

(c) 條件機率當鐵證：

```text
P(B 掛) = P(停電) + P(沒停電)·P(B 自己掛) = 0.01 + 0.99×0.01 = 0.0199
P(A 掛 | B 掛) = P(A∩B) / P(B) = 0.010099 / 0.0199 ≈ 0.507

對照 P(A 掛) = 0.0199。
```

`P(A|B) ≈ 0.507` 遠大於 `P(A) ≈ 0.0199`（差約 25 倍）。獨立的定義要求這兩者相等，這裡天差地別，所以 A、B **不獨立**。直覺：B 掛了，多半是因為停電（`0.01/0.0199 ≈ 50%` 的機率 B 的掛是停電造成的），而停電會把 A 一起拖死，所以「B 掛」這個消息大幅推高了「A 也掛」的機率。**獨立假設在這裡值「四個九 → 兩個九」整整兩個九，而那兩個九全是假的。**

### 推演題 4：把你維運過的「假獨立」翻成機率語言 **[15 分鐘]** ★★

口頭＋紙筆題。回想一個你實際碰過、或合理想像得到的「以為獨立、其實共享故障源」的情境（例：兩個微服務跑在不同 pod 但同一個節點；主從資料庫在不同機器但同一個機櫃電源；多區部署但共用同一個 CDN／DNS）。

(a) 寫出你原本「假設獨立」時會怎麼相乘估同掛機率。
(b) 指出那個被你忽略的 confounder 是什麼。
(c) 用一句話估計這個假獨立大概讓你**低估了幾個數量級**的同掛風險，並說你怎麼自我察覺。

#### 推演解答

沒有標準答案，重點是你能把工程體感翻成機率語言。一個範例：

(a) 兩個微服務各自月故障率 1%，假設獨立 → `P(同月都掛) = 0.01² = 0.0001`，估「兩個一起掛」幾乎不可能。

(b) confounder：它們其實跑在同一個 K8s 節點上。節點本身（或其底層 VM、或那台機器的網卡）有 0.5% 的月故障率，節點一掛兩個 pod 一起死。

(c) 真實同掛機率 ≈ `0.005 + 0.99×0.0001 ≈ 0.005099`，跟你估的 0.0001 差了約 **50 倍**（一個半數量級）。自我察覺的觸發點：每次你準備把兩個可用度「相乘」之前，逼自己回答一句「**它們會不會因為同一個原因一起壞**」——只要答案不是斬釘截鐵的「不會」，那個乘法就有鬼，你低估的數量級大致就等於那個共享原因的故障率比獨立相乘項大多少。把「不同機器」當成「獨立」，是這一行最貴的免費假設。

## 自我檢核

口頭自答，講得清楚才算過。優先答「為什麼」。

1. 條件機率 `P(A|B) = P(A∩B)/P(B)` 那個分母為什麼是 `P(B)` 不是別的？用「把世界縮到 B、再重新標準化」講給人聽，不准只念公式。
2. 獨立的兩個等價定義（`P(A∩B)=P(A)P(B)` 和 `P(A|B)=P(A)`）為什麼是同一件事？哪一個更能講出「獨立的靈魂」，為什麼？
3. 為什麼「互斥」和「獨立」是相反的，不是近義詞？用「你死我活 vs 井水不犯河水」這組對比，並能說出「互斥且各自機率為正 ⟹ 不獨立」這條的證明骨架。
4. 兩個各 99% 可用的備援，獨立時是四個九、共享電力時掉到兩個九——這兩個九的落差，數學上是哪一項在作怪？用全機率公式那個拆解講出「共享項淹掉獨立項」。
5. `P(A|B)` 和 `P(B|A)` 為什麼可以差幾個數量級？舉一個你自己的例子（不准用書上的「會中文 vs 是台灣人」），並說這個混淆會在 ch05 變成什麼謬誤。
6. 「這兩個指標獨立嗎？」這個問題為什麼不夠完整、沒辦法直接回答？「邊際獨立」和「條件獨立」的差別，跟 ch19 辛普森悖論有什麼關係？
7. 下次你準備把兩個可用度相乘之前，你會先問自己哪一句話？為什麼這一句自問比記住任何公式都重要？
