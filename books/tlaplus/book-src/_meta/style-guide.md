# 寫作風格指南（內部文件，非書籍內容）

本文件是全書的寫作契約。每一章都必須遵守，否則全書會失去一致性。
本書書名（工作名）：《把系統寫成定理：TLA+ 與形式化方法》。

## 這本書是什麼、不是什麼

- **是**：一本「讀＋紙上推演」的書。讀者要能手寫不變量、手追狀態轉移、手驗歸納證明、精讀真實世界的 TLA+ spec，並深入到證明層（inductive invariant、refinement、TLAPS 的讀法）。
- **不是**：工具教學書。讀者**不會**安裝 TLA+ 工具鏈、不寫程式、不跑 TLC（2026-06-11 訪談確認）。書中仍要把 TLC、TLAPS、Apalache 的**原理**講清楚——理解機器怎麼做，但用紙筆代替鍵盤。
- 定位：**獨立興趣**，不服務 AI Infra 轉職敘事。橋接錨點是讀者的後端經驗本身，不需要繞到 GPU/LLM 話題（偶爾引用無妨，不依賴）。

## 讀者是誰

唯一讀者：Chewei，6 年以上資深後端工程師（TypeScript/Node.js、NestJS）。可直接當橋接錨點的經歷：

- Event-driven 結算 pipeline：K8s CronJobs → SQS → idempotent consumers，每季 10 萬~30 萬筆、at-least-once、最終一致性
- 高併發遊戲後端（500K+ 使用者、10K+ CCU）、WebSocket + Redis Pub/Sub 訊息系統（~3,200 並發連線）
- MySQL/PostgreSQL：交易、isolation level、deadlock、鎖競爭實戰；profiling 驅動讓 RDS CPU 尖峰降 40%
- 分散式模式的一線直覺：idempotency、最終一致性、backpressure、retry、rate limiting、佇列理論直覺

**不需要解釋**（讀者已精通）：queue、cache、DB 交易/鎖/isolation、idempotency、eventual consistency、race condition 的概念本身、K8s 基礎物件、HTTP/REST、重試與逾時的工程直覺。

**快速複習即可**（資工學士＋研究所選修演算法，看過但生鏽）：基礎離散數學的「樣子」——集合符號、量詞長相、歸納法的存在。要重教，但可以用「你大學看過，現在把它磨利」的節奏，不用當文組讀者教。

**要從零教**（接觸極少）：形式化規格的思維方式、時序邏輯、TLA 的動作邏輯、模型檢查原理、不變量證明技藝、refinement、定理證明器生態。

## 語言規範

- 繁體中文，台灣慣用語。技術名詞一律保留英文原文，不硬翻（invariant、safety、liveness、stuttering、refinement、fairness…第一次出現時用一句中文解釋）。已有公認譯名者中英並用：不變量（invariant）、互斥（mutual exclusion）、歸納法（induction）。
- 台灣用語基準：記憶體（不是内存）、佇列、叢集、程式碼、函式、伺服器、資料、網路、執行緒、行程（process，必要時直接用英文）、最佳化／優化皆可。禁止簡體中文詞彙與用法（信息、默認、內存、代碼、對象、質量…）。
- 第二人稱「你」直接對讀者說話。

## 數學符號規範（全書一致，附錄 A 是速查表）

- 內文與推演**一律用 Unicode 數學符號**：∧ ∨ ¬ ⇒ ≡ ∀ ∃ ∈ ⊆ ∪ ∩ □ ◇ ⤳（leads-to 可寫 ~>）、x′（prime 用 ′）。
- TLA+ 原始碼引用（精讀真實 spec 時）保留 ASCII 原文：`/\ \/ ~ => [] <> x'`。ch06 首次出現時給對照表，附錄 A 收全表。
- 狀態寫成 tuple：⟨pc = "cs", x = 1⟩。動作名用 UpperCamelCase（如 `SendPrepare`）。spec 片段用 code block 標 `tla`。
- 證明步驟編號用 ⟨1⟩1、⟨1⟩2…（TLAPS 層級式寫法），從 ch05 的手寫證明就開始養成，到 ch16 讀真 TLAPS 證明時無縫接軌。

## 每章骨架（必須遵守，標題名稱照抄——一致性掃描會 grep 這些標題）

1. 章首：`# chNN — 章名：副標`，接著 blockquote：
   `> **本章解決什麼問題**：…（這章在全書地圖的哪裡、為什麼存在、跟前後章的關係，3-5 行）`
   每個 Part 的第一章（ch01、ch03、ch06、ch10、ch14、ch17）在 blockquote 後放一張全書 ASCII 路線圖，標出「你在這裡」。讀者偏好架構圖先行：先全局、再細節。
2. `## 從你已知的出發` — 橋接段：把本章核心概念錨定在讀者的後端實戰上，要具體到實際經歷（結算 pipeline、idempotent consumer、DB deadlock、WebSocket 斷線重連），不要泛泛而談。例：invariant ≈ 你在 code review 守的「這個欄位永遠不會是負的」；fairness ≈ 你對 SQS「訊息終究會被消費」的假設；refinement ≈ 介面與實作的契約。
3. 核心內容（多個 `##` section，深入機制）。
4. `## 陷阱與防禦` — 本書招牌段落（故障模式視角的形式化版）：這個概念**怎麼被誤用**、證明**怎麼假通過**（vacuous truth、太弱的 invariant、忘了 fairness 導致 liveness 空洞地成立、模型太小漏掉反例…）、**怎麼自我察覺**。用表格或條列。
5. `## 紙上推演` — 取代動手做（2026-06-11 訪談：本書不動鍵盤）。每章 2~4 題，每題標注預估時間（如 **[15 分鐘]**）與難度（★~★★★）。題型：手寫/強化不變量、手追狀態轉移序列、手驗歸納步驟、找出 spec 的反例、改錯。題目後緊接 `### 推演解答` 給完整解答與討論（為什麼這樣想、常見的錯路）。
6. `## 自我檢核` — 5~8 個問題，**口頭自答**：講得出來才算過關，講不清楚＝還沒懂。問題要對齊「能不能把這個概念教給另一個工程師」的深度。
7. `## 延伸閱讀` — 真實連結（以 `_meta/landscape-2026-06.md` 驗證過的為優先），每條一句話說明為什麼值得讀、讀哪一段。

## 深度標準

- **每章至少一個 worked example**：完整手推過程、帶具體狀態與數字（狀態空間多大、第幾步出現反例、invariant 在哪一步被打破），一步一步走，不准跳步。
- **證明的嚴謹度基準**：「工程師的嚴謹」——每一步都能口頭說出理由；不要求形式系統的機械推導，但禁止「顯然」「易證」跳步。ch14–16 的證明章可以更嚴，並示範嚴謹度的光譜（敘述證明 → 層級證明 → 機器驗證）。
- 每個定理/論斷：要嘛在書中證（依上述基準），要嘛明寫「本書不證，見延伸閱讀」。不准假裝證過。
- 每個工具（TLC、TLAPS、Apalache…）都要回答「它在解什麼問題、原理是什麼、極限在哪」，禁止只羅列名稱。
- 每個方法都給決策框架：什麼時候值得形式化、什麼時候測試就夠、代價是什麼。
- Murphy 原則的形式化版：每個機制都要回答「它怎麼給你假安全感、你怎麼發現」。
- 寧深勿廣，不灌水。每章 5,000~10,000 中文字，依主題深度調整。

## 格式

- Markdown。主要層級 `##`，次層 `###`。
- 圖：ASCII diagram 為主（狀態機、狀態空間圖、全書地圖），必須通過 `check_diagrams.py` 的 CJK=2 欄寬檢查；流程可用 Mermaid。狀態空間圖是本書最常見的圖——畫節點（狀態）與邊（動作），小到能畫完的就畫完整。
- TLA+ 片段用 code block 標 `tla`；偽碼標 `text`。表格用於對照與決策框架。
- 跨章引用格式：`（見 ch07）`，不要寫死檔名路徑。
