# ch02 — 一切都是狀態機：state、behavior、invariant

> **本章解決什麼問題**：ch01 的結論是測試窮舉不完調度，最後一道防線是證明。但「證明系統正確」要成立，得先把「系統」變成一個數學上可以操作的東西——本章給出這個東西：狀態機。你會拿到全書通用的四個詞：狀態（state）、步（step）、行為（behavior）、規格（specification）＝允許行為的集合；並第一次見到貫穿全書的範例「結算系統 v0」與它的第一條不變量 `NoDoublePay`。ch03–05 負責磨利描述這套模型所需的數學，ch06 再把整套東西寫成 TLA+。

## 從你已知的出發

你其實天天在跟狀態機打交道，只是沒人逼你把詞說清楚。

- **DB 的一筆 row 在某個瞬間的值，就是一個狀態的片段。** 你 debug 線上事故時拼湊的那張全景——這筆訂單的 row、Redis 裡的鎖、佇列深度、consumer 的記憶體裡掛著什麼——把它們全部疊起來，那個完整快照才是本章說的「狀態」。
- **一筆 UPDATE 是一個步。** 交易 commit 前後各照一張快照，這一對「之前／之後」就是步的全部內容。
- **transaction log（binlog、WAL、event store）是一條行為。** 你做過 replay：從初始狀態出發，把事件一條一條套用，重建出現在的狀態。Event sourcing 與 Redux 的世界觀——「狀態＝初始狀態 fold 上事件序列」——就是狀態機觀點，你早就在寫了。
- **Code review 時你腦中跑的 what-if，是在窮舉行為。**「這裡如果被插隊呢？如果 consumer 在這行掛掉呢？」你審的從來不是「程式會做什麼」，而是「程式**可能**做什麼」。本章只是把這件你做了六年的事，從直覺升級成可以數、可以畫、可以證的東西。

ch01 的痛點是：行為太多，測試只能抽樣。本章的回應是：那就把「所有行為」本身定義成一個數學物件。物件定義清楚了，ch05 之後才有資格談「對所有行為證明」。

## 狀態、步、行為：三個定義

### 狀態（state）：變數的一組賦值

先選定一組變數，**狀態就是給每個變數各指定一個值的一組賦值（assignment）**。沒有別的了——不需要「目前執行到哪一行」這種特殊地位的東西；程式計數器要的話，自己也宣告成一個變數（ch08 會看到 PlusCal 翻譯出來的 `pc` 變數正是這樣）。

兩個立刻要釘住的推論：

1. **狀態是整體快照，不是單一變數。** 「queue 還剩什麼」不是狀態，「queue 還剩什麼＋consumer 手上有什麼＋帳本長什麼樣」合起來才是。併發 bug 幾乎都藏在「兩個變數各自看都合理、合在一起不合理」的組合裡——所以單位必須是整組賦值。
2. **所有可能的賦值組合構成「狀態空間」。** 注意這是「型別上寫得出來的全部組合」，跟「系統實際走得到的狀態」是兩回事——這個區分在本章後半是主角。

順帶把詞釘準：本書說「狀態機」，不是 regex 課本裡那種有限自動機，也不是 UML 裡畫給產品經理看的小圈圈圖。變數可以多個、值域可以無限大；「狀態機」在這裡只是「初始狀態＋轉移規則」這個極簡結構的名字。

### 步（step）：一對前後狀態

**步就是一對狀態 ⟨s, t⟩：之前是 s，之後是 t。** 哪些步被允許，由「動作規則」決定——每條規則說清楚兩件事：

- **前提**：在什麼樣的狀態 s 才可以走這條規則；
- **效果**：走完之後 t 的每一個變數是什麼。**每一個**變數——沒提到的變數不是「隨便」，而是「不變」。漏寫這件事是寫規格時最常見的 bug（ch06 會看到 TLA+ 用一個專門的寫法堵這個洞）。

橋接：一條動作規則就是一筆交易的 before/after 契約。前提 ≈ `WHERE` 條件與鎖的取得；效果 ≈ commit 之後資料庫的長相。

### 行為（behavior）：一條可以無限長的狀態序列

**行為就是一條狀態序列 s₀ → s₁ → s₂ → …**，其中 s₀ 是合法的初始狀態，而且每一對相鄰狀態 ⟨sᵢ, sᵢ₊₁⟩ 都是某條動作規則允許的步。

- 行為可以**有限**（系統做完了、或者就是停了），也可以**無限**（伺服器不關機）。形式化的慣例是把有限執行看成「之後永遠停在最後一個狀態」的無限序列——這個看起來無聊的小動作，之後會變成 TLA+ 最深思熟慮的設計之一（見 ch06）。
- 一台狀態機通常生出**很多條**行為：每個狀態可能有好幾條規則可走，每個分岔都是一條新的行為。你的 binlog 是其中一條被現實抽中的；狀態機是「所有可能的 binlog」的集合。

## 規格：允許行為的集合

現在可以把全書的核心視角一句話說完：

> **一個系統的規格（specification），就是它允許的行為的集合。**「系統正確」＝「實際發生的每一條行為，都落在這個集合裡」。

這句話翻轉了一個習慣：規格不是「程式會怎麼跑」的描述，而是「我允許世界怎麼演化」的立法。立法的對象不只你的程式——consumer 的程式碼、佇列的投遞、機器的當機、使用者的重試，全部寫進同一台狀態機的動作規則。你的 spec 裡沒寫的事，就是你宣告不可能發生的事；這個責任本章最後一節（抽象的選擇）會回來算帳。

### 非決定性：不是隨機，是「都可能」

一個狀態可以同時有好幾條規則可走。誰決定走哪條？**沒有人。** 規格只回答「允不允許」，不回答「多常發生」、也不附機率分布。這叫**非決定性（nondeterminism）**，它跟「隨機」是兩種世界觀：

- 隨機說：「這條路 0.1% 機率。」
- 非決定性說：「這條路**存在**。」

對驗證來說，非決定性才是對的工具，原因很工程：你要的保證是「**所有**允許的行為都不出事」，那麼調度器就必須當成對手而不是骰子——對手永遠挑最壞的那條路走。ch01 那個要 35 步才踩到的 DynamoDB bug（見 ch01）機率上低到測不到，但它**可達**；可達，乘上你的流量，就是遲早的事故報告。

這也預告了故障怎麼建模：consumer 會 crash，不是寫「以某機率失敗」，而是加一條「只要它活著，隨時允許走」的動作規則。crash 與重送正式進場是 v1 的事（見 ch08）；本章的 v0 先活在不會當機的童話裡——而且我們會誠實地標出童話的邊界。

## 結算系統 v0：把你的 pipeline 削到剩骨頭

全書的貫穿範例登場。原型是你維護過的東西：CronJob 把結算訊息丟進 SQS，idempotent consumer 撈出來入帳，每季十萬到三十萬筆。v0 把它削到只剩骨頭：

- **2 則訊息**：m1、m2。
- **1 個 consumer**（本章稍後加到 2 個）。
- **3 個變數**：

| 變數 | 值域 | 意義 |
|---|---|---|
| `queue` | {m1, m2} 的子集 | 還在佇列裡的訊息（集合，不記順序——這是個刻意的抽象，後面算帳） |
| `working` | idle、m1、m2 三選一 | consumer 手上正在處理的訊息；idle ＝空手 |
| `ledger` | 每則訊息對應 0 或 1 | 帳本：msgId → 是否已入帳（0 ＝未入帳、1 ＝已入帳，ch06 寫成 TLA+ 時用 FALSE/TRUE；↦ 讀作「對應到」） |

- **初始狀態**（恰好一個）：⟨queue = {m1, m2}, working = idle, ledger = [m1 ↦ 0, m2 ↦ 0]⟩。
- **2 條動作規則**：

| 規則 | 前提 | 效果 |
|---|---|---|
| `Fetch(m)` | m ∈ queue 且 working = idle | queue 移除 m；working 變成 m；ledger 不變 |
| `Settle(m)` | working = m | ledger[m] 改成 1；working 回到 idle；queue 不變 |

就這樣。沒有 crash、沒有重送、沒有 visibility timeout、入帳和「把訊息處理完」綁成原子的一步。v0 不是因為現實長這樣，而是因為**心智模型要先在最小的世界裡站穩**；每一項被省略的現實，都會在後面的章節以升級的形式還回來（v1 見 ch08）。

## 手畫 v0 的完整狀態空間（worked example）

現在動手。目標：把 v0（2 則訊息、1 個 consumer）**所有走得到的狀態**畫成一張圖——節點是狀態、邊是步。這張圖之後很多章要回來對帳，所以這裡一步不跳。

### 八個狀態，一個一個數

從初始狀態開始做廣度優先的窮舉：每拿到一個新狀態，就把兩條規則逐一比對前提，看能走去哪。

**第 0 層**：只有初始狀態，叫它 S0。

- S0 = ⟨queue = {m1, m2}, working = idle, ledger = [m1 ↦ 0, m2 ↦ 0]⟩
- 可走的規則：`Fetch(m1)`（m1 ∈ queue ✓、working = idle ✓）、`Fetch(m2)`（同理 ✓）。`Settle` 不可走：手上沒訊息。

**第 1 層**：S0 的兩條出邊各到一個新狀態。

- S1 = ⟨queue = {m2}, working = m1, ledger = [m1 ↦ 0, m2 ↦ 0]⟩ —— 由 `Fetch(m1)` 到達。
  可走：`Settle(m1)` ✓。`Fetch(m2)` 前提 m2 ∈ queue ✓ 但 working ≠ idle ✗，**不可走**——單一 consumer 一次只拿一則。
- S2 = ⟨queue = {m1}, working = m2, ledger = [m1 ↦ 0, m2 ↦ 0]⟩ —— 由 `Fetch(m2)` 到達。
  可走：只有 `Settle(m2)`，理由同上。

**第 2 層**：

- S3 = ⟨queue = {m2}, working = idle, ledger = [m1 ↦ 1, m2 ↦ 0]⟩ —— S1 經 `Settle(m1)`。
  可走：`Fetch(m2)` ✓（m2 ∈ queue、空手）。`Fetch(m1)` ✗：m1 ∉ queue（∉ ＝不屬於）。
- S4 = ⟨queue = {m1}, working = idle, ledger = [m1 ↦ 0, m2 ↦ 1]⟩ —— S2 經 `Settle(m2)`。
  可走：只有 `Fetch(m1)`。

**第 3 層**：

- S5 = ⟨queue = ∅, working = m2, ledger = [m1 ↦ 1, m2 ↦ 0]⟩ —— S3 經 `Fetch(m2)`。可走：只有 `Settle(m2)`。
- S6 = ⟨queue = ∅, working = m1, ledger = [m1 ↦ 0, m2 ↦ 1]⟩ —— S4 經 `Fetch(m1)`。可走：只有 `Settle(m1)`。

**第 4 層**：

- S7 = ⟨queue = ∅, working = idle, ledger = [m1 ↦ 1, m2 ↦ 1]⟩ —— S5 經 `Settle(m2)`、S6 經 `Settle(m1)`，兩條路在這裡會合。
  可走：沒有。queue 空了，`Fetch` 前提不成立；手是空的，`Settle` 前提不成立。系統停在這裡（「停」這件事在形式化裡怎麼安頓，ch06 有個漂亮的答案）。

沒有第 5 層。**可達狀態總數：8。**

整理成表（位置欄是個速記：每則訊息此刻在哪——Q ＝佇列、C ＝ consumer 手上、L ＝已入帳）：

| 代號 | (m1, m2) 位置 | queue | working | ledger |
|---|---|---|---|---|
| S0 | (Q, Q) | {m1, m2} | idle | [m1 ↦ 0, m2 ↦ 0] |
| S1 | (C, Q) | {m2} | m1 | [m1 ↦ 0, m2 ↦ 0] |
| S2 | (Q, C) | {m1} | m2 | [m1 ↦ 0, m2 ↦ 0] |
| S3 | (L, Q) | {m2} | idle | [m1 ↦ 1, m2 ↦ 0] |
| S4 | (Q, L) | {m1} | idle | [m1 ↦ 0, m2 ↦ 1] |
| S5 | (L, C) | ∅ | m2 | [m1 ↦ 1, m2 ↦ 0] |
| S6 | (C, L) | ∅ | m1 | [m1 ↦ 0, m2 ↦ 1] |
| S7 | (L, L) | ∅ | idle | [m1 ↦ 1, m2 ↦ 1] |

### 狀態空間圖

```
          ┌────────────────────────────┐
          │         S0  (Q, Q)         │
          └────┬───────────────────┬───┘
               │                   │
               ▼ Fetch(m1)         ▼ Fetch(m2)
        ┌──────┴─────┐      ┌──────┴─────┐
        │ S1  (C, Q) │      │ S2  (Q, C) │
        └──────┬─────┘      └──────┬─────┘
               │                   │
               ▼ Settle(m1)        ▼ Settle(m2)
        ┌──────┴─────┐      ┌──────┴─────┐
        │ S3  (L, Q) │      │ S4  (Q, L) │
        └──────┬─────┘      └──────┬─────┘
               │                   │
               ▼ Fetch(m2)         ▼ Fetch(m1)
        ┌──────┴─────┐      ┌──────┴─────┐
        │ S5  (L, C) │      │ S6  (C, L) │
        └──────┬─────┘      └──────┬─────┘
               │ Settle(m2)        │ Settle(m1)
               └─────────┬─────────┘
                         ▼
                 ┌───────┴────────┐
                 │   S7  (L, L)   │
                 └────────────────┘
```

**8 個狀態、8 條邊**，左右兩條歧路（先處理 m1 還是先處理 m2）在 S7 會合。從 S0 走到 S7 的完整路徑恰好 2 條——v0 的全部人生劇本。S0 那個分岔就是非決定性的長相：規格不挑邊，兩條都合法。

### 怎麼知道沒漏：封閉性檢查

「我畫完了」不是感覺問題，是可以機械檢查的主張：**對清單裡每一個狀態，列出它所有可走的規則，確認每條邊的落點都還在清單內。** 落點全都在裡面＝清單封閉＝再走一萬步也跳不出這 8 個——所以它就是全部的可達狀態。

| 狀態 | 出邊（規則 → 落點） | 出邊數 |
|---|---|---|
| S0 | Fetch(m1) → S1；Fetch(m2) → S2 | 2 |
| S1 | Settle(m1) → S3 | 1 |
| S2 | Settle(m2) → S4 | 1 |
| S3 | Fetch(m2) → S5 | 1 |
| S4 | Fetch(m1) → S6 | 1 |
| S5 | Settle(m2) → S7 | 1 |
| S6 | Settle(m1) → S7 | 1 |
| S7 | （無） | 0 |

合計 8 條邊，落點全數命中清單。✓ 你剛剛手動執行的，就是模型檢查器 TLC 的核心演算法——廣度優先搜尋加封閉性判定（原理見 ch09）。

### 48 與 8：型別空間 vs 可達空間

換個算法：不管規則，光看型別，v0 的變數能拼出幾種賦值？

- queue：{m1, m2} 的子集，4 種；
- working：3 種；
- ledger：每則訊息 0 或 1，2 × 2 ＝ 4 種。

4 × 3 × 4 ＝ **48 種賦值組合**。可達的只有 8 種——剩下 40 種是「寫得出來、走不到」的狀態。例如：

- ⟨queue = {m1}, working = m1, ledger = [m1 ↦ 0, m2 ↦ 0]⟩：m1 同時在佇列又在手上；
- ⟨queue = {m1}, working = idle, ledger = [m1 ↦ 1, m2 ↦ 0]⟩：m1 已入帳卻又躺回佇列——**雙重入帳的前夜**。

這 40 種不可達不是型別系統的功勞，是兩條動作規則一步一步「掙來的」秩序。把「掙來的」變成「證出來的」，是 ch05 的工作；在型別層面系統性地計算這種上界、跟本章的 8 對帳，是 ch04 的習題。

## 第二個 consumer 進場：狀態怎麼長

現實裡你不會只跑一個 consumer。加一個：working 從單格變成一張表 [c1 ↦ …, c2 ↦ …]，每格仍是 idle／m1／m2；`Fetch`、`Settle` 規則對每個 consumer 各自適用，前提多一條「同一則訊息不能同時被兩個 consumer 拿」（v0 的佇列取走即消失，這條自動成立）。位置速記擴成四種：Q、C1、C2、L。

老老實實一個一個數。按「已入帳幾則」分組：

**0 則已入帳**（m1、m2 各在 Q／C1／C2，扣掉「同一個 consumer 同時拿兩則」的 (C1,C1) 與 (C2,C2)，3 × 3 − 2 ＝ 7 個）：

| # | (m1, m2) | 出邊（規則 → 落點） | 出邊數 |
|---|---|---|---|
| T0 | (Q, Q) | c1 Fetch m1 → (C1,Q)；c2 Fetch m1 → (C2,Q)；c1 Fetch m2 → (Q,C1)；c2 Fetch m2 → (Q,C2) | 4 |
| T1 | (C1, Q) | c1 Settle m1 → (L,Q)；c2 Fetch m2 → (C1,C2) | 2 |
| T2 | (C2, Q) | c2 Settle m1 → (L,Q)；c1 Fetch m2 → (C2,C1) | 2 |
| T3 | (Q, C1) | c1 Settle m2 → (Q,L)；c2 Fetch m1 → (C2,C1) | 2 |
| T4 | (Q, C2) | c2 Settle m2 → (Q,L)；c1 Fetch m1 → (C1,C2) | 2 |
| T5 | (C1, C2) | c1 Settle m1 → (L,C2)；c2 Settle m2 → (C1,L) | 2 |
| T6 | (C2, C1) | c2 Settle m1 → (L,C1)；c1 Settle m2 → (C2,L) | 2 |

**1 則已入帳**（已入帳者固定在 L，另一則在 Q／C1／C2，3 種；誰先入帳 × 2，共 6 個）：

| # | (m1, m2) | 出邊 | 出邊數 |
|---|---|---|---|
| T7 | (L, Q) | c1 Fetch m2 → (L,C1)；c2 Fetch m2 → (L,C2) | 2 |
| T8 | (Q, L) | c1 Fetch m1 → (C1,L)；c2 Fetch m1 → (C2,L) | 2 |
| T9 | (L, C1) | c1 Settle m2 → (L,L) | 1 |
| T10 | (L, C2) | c2 Settle m2 → (L,L) | 1 |
| T11 | (C1, L) | c1 Settle m1 → (L,L) | 1 |
| T12 | (C2, L) | c2 Settle m1 → (L,L) | 1 |

**2 則已入帳**（1 個）：

| # | (m1, m2) | 出邊 | 出邊數 |
|---|---|---|---|
| T13 | (L, L) | （無） | 0 |

7 ＋ 6 ＋ 1 ＝ **14 個狀態**；出邊合計 4＋2×6＋2×2＋1×4＋0 ＝ **24 條邊**。每條邊的落點都在這 14 個裡——封閉性 ✓。

完整路徑（從 T0 走到 T13 的劇本數）也數得出來，倒著算：令 f(狀態) ＝ 從該狀態走到 T13 的路徑數，等於各出邊落點的 f 相加。

- f(T13) ＝ 1；
- f(T9) ＝ f(T10) ＝ f(T11) ＝ f(T12) ＝ 1（各只有一條邊通往 T13）；
- f(T7) ＝ f(T9) ＋ f(T10) ＝ 2；f(T8) ＝ 2；
- f(T5) ＝ f(T10) ＋ f(T11) ＝ 2；f(T6) ＝ f(T9) ＋ f(T12) ＝ 2；
- f(T1) ＝ f(T7) ＋ f(T5) ＝ 4；同理 f(T2) ＝ f(T3) ＝ f(T4) ＝ 4；
- f(T0) ＝ 4 ＋ 4 ＋ 4 ＋ 4 ＝ **16**。

對照表：

| | 1 consumer | 2 consumers | 倍率 |
|---|---|---|---|
| 狀態 | 8 | 14 | ×1.75 |
| 邊 | 8 | 24 | ×3 |
| 完整路徑（調度劇本） | 2 | 16 | ×8 |

注意哪一行長得最兇：**劇本數**。你要測試蓋住的正是劇本，不是狀態——這就是 ch01「調度爆炸」的數學臉孔。

再往外推一格。數法可以重用：把狀態按「幾則訊息被拿在手上」分組——沒人拿（每則訊息各在佇列或帳本，2 種）、恰一則被拿（選訊息 × 選 consumer × 其餘各 2 種）、兩則都被拿（兩則訊息分給兩個 consumer 的排法 × 其餘各 2 種）。先用它驗算一次 2 則訊息的情形：2² ＋ 2×2×2 ＋ 2×1 ＝ 4 ＋ 8 ＋ 2 ＝ 14 ✓。然後：

- **3 則訊息、2 個 consumer**：2³ ＋ 3×2×2² ＋ (3×2)×2 ＝ 8 ＋ 24 ＋ 12 ＝ **44**。
- **10 則訊息、2 個 consumer**：2¹⁰ ＋ 10×2×2⁹ ＋ (10×9)×2⁸ ＝ 1,024 ＋ 10,240 ＋ 23,040 ＝ **34,304**。

（每一種組合都真的走得到：想要哪個快照，就先讓該入帳的訊息一則一則走完 Fetch–Settle，再讓指定的 consumer 把指定的訊息拿在手上。）

10 則訊息——你真實 pipeline 一季量的十萬分之一——三萬四千個狀態。而這還是不會 crash、不會重送、帳本只有一個 bit 的童話世界。手畫到此為止是合理的極限；機器怎麼接手、以及為什麼連機器也會被淹沒，見 ch09。

## 不變量：在所有可達狀態都成立的述語

圖在手上了，可以開始對它**主張**些什麼。最重要的一類主張叫**不變量（invariant）**：

> 一個關於狀態的述語（predicate，「對單一狀態非真即假的判斷句」），如果在**每一個可達狀態**都為真，它就是這台狀態機的不變量。

三個例子，從無聊到值錢，全部拿 8 狀態圖當場驗：

1. **值域述語**：「working 永遠是 idle、m1、m2 三者之一。」對 S0–S7 逐一看 working 欄：idle, m1, m2, idle, idle, m2, m1, idle——✓。無聊，但它是每份規格的第一道防線（這類述語在 TLA+ 慣例裡有個固定角色，見 ch08），抓的是「變數被寫進鬼值」這種最低級也最常見的錯。
2. **恰好一處**：「每則訊息在任何時刻恰好處於佇列、手上、帳本三者之一。」看位置速記欄：八個狀態每則訊息都恰好一個字母——✓。這條就是把 48 砍成 8 的那股秩序本身。
3. **反例示範**：「佇列永遠非空。」S5 的 queue ＝ ∅——✗。**不是**不變量。注意判定方式：只要找到**一個**可達狀態打破它就出局；而「可達」要附證據（S0 → S1 → S3 → S5，每步都驗過規則）。

釘住量化範圍：不變量說的「永遠」，範圍是**全部可達狀態**——不是 48 種型別組合（那叫要求過頭，很多有用的不變量在不可達區根本不成立），也不是「我手追過的那幾條行為」（那叫測試）。

### NoDoublePay 首次登場

全書最重要的 safety 性質，現在正式命名：

> **`NoDoublePay`：同一個 msgId 至多入帳一次。**

字面上它談的是**整條行為的歷史**（「至多一次」是數整段過程中入帳動作的次數）。要在狀態圖上逐點檢查，得先把它翻成一條關於單一狀態的看門狗述語：

> 對每一則訊息 m：如果 ledger[m] ＝ 1，那麼 m ∉ queue，而且沒有任何 consumer 的手上是 m。

直覺連結：入帳只會發生在 `Settle`，`Settle` 的前提是 m 在手上，而要拿到手上得先從佇列 `Fetch`。所以只要「已入帳的訊息永遠不再出現於佇列或手上」在每個可達狀態都成立，第二次入帳就無路可走。對 8 個狀態逐一驗：位置速記裡標 L 的訊息，其所在狀態的 queue 與 working 都查無此訊息（S3、S4、S5、S6、S7 全數通過；S0–S2 沒有 L，前提落空，自動過關——這個「前提落空就算過」的邏輯細節先欠著，ch03 會正式跟它算帳）。**v0 滿足 `NoDoublePay`。** ✓

先別慶祝，誠實面對一件事：這個「成立」有兩種成分——

- 一部分是**設計的功勞**：取走即消失＋一次只拿一則，確實堵住了重複處理；
- 另一部分是**模型的盲區**：v0 根本沒有把「會導致雙重入帳的機制」建進來。沒有 redelivery、沒有 crash、入帳與移除綁成原子一步——失敗被建模成不可能，性質當然成立。

這是 Murphy 原則的形式化版第一課：**性質成立，有時是因為系統真的安全，有時只是因為你的模型說不出那種失敗。** 分辨方法：問「違反這條性質的狀態長什麼樣？我的模型畫得出來嗎？」v0 畫得出來（48 種組合裡就有「已入帳又在佇列」的賦值），只是不可達——所以 `NoDoublePay` 在 v0 是一條真的有內容、但還沒受過真正考驗的不變量。考驗在 v1：加入 crash 與 redelivery 之後（見 ch08），「已入帳的訊息又回到佇列」會變成貨真價實的可達狀態，到時候守住 `NoDoublePay` 的就不再是運氣，而是 dedup 表——以及一份要證明的功課（見 ch14）。

### safety 與 liveness：一句話的對比

`NoDoublePay` 屬於 **safety**：「壞事永遠不會發生」。它的特徵是違反的證據是**有限的**——行為走到第 k 步出現雙重入帳，當場定罪，後面不用看。

對照組是 **liveness**：「好事終究會發生」，例如「每則訊息終究會入帳」。違反它的證據永遠湊不齊有限版本——看到第一百萬步還沒入帳，你仍不能定罪，因為「再等等」永遠是合法辯護。

順手戳破一個直覺：v0 **並不**保證「每則訊息終究會入帳」。「系統停在 S1 從此不動」是一條完全合法的行為——沒有任何規則強迫 consumer 工作。你對 SQS「訊息總會被消費」的營運信心，其實是一條額外的公平性（fairness）假設，而不是動作規則的推論。這整套區分的正式定義、以及「終究」怎麼寫成數學，是 ch07 的主題；本章你只需要帶走這句對比：**safety 看得到犯罪現場，liveness 永遠在等待戈多。**

## 抽象的選擇：建模什麼、忽略什麼

v0 丟掉了很多現實。列出來，並且記帳：

| 被丟掉的現實 | 後果：哪些問題從此問不了 | 何時還債 |
|---|---|---|
| 金額與訊息內容 | 「入帳金額對不對」問不了；`NoDoublePay` 只關心次數，不關心錢數 | 不還——對本書的性質，金額自始無關 |
| 佇列順序（set 而非 sequence） | 「m1 比 m2 先到就先結」這類順序性質**連說都說不出來** | ch04 有一題專門算這筆帳 |
| consumer crash、redelivery | at-least-once 的核心風險整個不在模型裡 | v1（見 ch08） |
| 入帳與移除訊息的兩步性 | 「入帳成功、ack 之前掛掉」這個雙重入帳的經典開局不存在 | v1（見 ch08） |
| 時間與延遲 | 任何 SLA、timeout 性質問不了 | 不還——本書始終不建模時間量 |
| 真實訊息量（用 2 代表 N） | 「2 則沒事，2,000 則會不會有事」是個真問題 | ch09（small scope 的辯護與限度） |

從這張表提煉本章最後一個方法論：**先寫你要驗的性質，再決定模型留什麼。** 問 `NoDoublePay`，就必須保留訊息身分、入帳狀態、以及一切可能導致重複處理的機制；可以丟金額、順序、延遲。反過來，如果哪天要驗順序性，set 就得換成 sequence。模型不是現實的縮小版，是**針對特定問題的證詞**；抽象不是說謊，是控制謊言的範圍——而控制的手段，就是像上表那樣把每一筆省略寫成白紙黑字。這也是 ch01 那句「寫 spec 的最大價值是先想清楚」的具體操作版。

## 陷阱與防禦

| 陷阱 | 長相 | 防禦 |
|---|---|---|
| 把非決定性讀成機率 | 「這個交錯太刁鑽，現實不會發生」——然後它在第三季發生了 | 規格的世界只有可達／不可達。可達的壞狀態 × 夠大的流量 ＝ 排程中的事故。機率直覺只准用來排優先級，不准用來宣告安全 |
| 性質空洞成立 | 模型表達不出那種失敗，於是性質「通過」——v0 的 `NoDoublePay` 有一半是這樣成立的 | 替每條性質找出「最小違反狀態」：在型別空間裡畫出違反的長相。畫不出來＝性質對這個模型沒有牙齒，先補模型再談驗證 |
| 混淆型別空間與可達空間 | 拿 48 嚇人、或拿 8 自滿；在不可達區檢討不變量，或宣稱「我走過的路都沒事」 | 講不變量先講量化範圍：「所有**可達**狀態」。估規模時兩個都算——差距越大，代表動作規則扛的秩序越多，也代表越值得證明（見 ch05） |
| 窮舉漏狀態 | 畫了幾條順路就收工；漏一條邊，漏掉的可能是一整片大陸 | 封閉性檢查：清單裡每個狀態 × 每條規則，落點必須回到清單。這是機械工作，不是藝術——做不完就是還沒畫完 |
| 用「合理」代替「允許」 | 「consumer 不會拿著訊息不放」——模型裡它就是可以 | 判斷行為合法只看動作規則。常識要嘛寫成規則，要嘛閉嘴。你覺得「不會發生」的事，規格眼裡都是合法行為 |
| 為了好畫而裁模型 | 因為圖會爆炸所以不建 crash——於是驗了一個不存在的系統 | 順序不能反：先性質、後模型。裁掉的每一項記進「何時還債」表；圖太大是換工具的訊號（見 ch09），不是改題目的藉口 |

## 紙上推演

### 題 1：行為合不合法 **[10 分鐘]** ★

以下四條狀態序列（代號見 8 狀態表），逐一判斷是不是 v0（1 個 consumer）的合法行為。不合法的，指出第一個非法的步與原因。

- (a) S0 → S2 → S4 → S6 → S7
- (b) S0 → S3 → S5 → S7
- (c) S0 → S1 → S0 → S2 → S4 → S6 → S7
- (d) S0 → S1（序列到此結束）

### 推演解答

**(a) 合法。** 逐步驗，每步要核對三個變數：

1. S0 → S2：`Fetch(m2)`。前提：m2 ∈ {m1, m2} ✓、working ＝ idle ✓。效果：queue {m1, m2}→{m1} ✓、working idle→m2 ✓、ledger 不變 ✓。
2. S2 → S4：`Settle(m2)`。前提：working ＝ m2 ✓。效果：ledger[m2] 0→1 ✓、working 回 idle ✓、queue 不變（{m1}）✓。
3. S4 → S6：`Fetch(m1)`。前提 ✓；queue {m1}→∅ ✓、working idle→m1 ✓、ledger 不變 ✓。
4. S6 → S7：`Settle(m1)`。前提 ✓；ledger[m1] 0→1 ✓、working 回 idle ✓、queue 不變 ✓。

**(b) 不合法**，第 1 步就掛。S0 → S3 要求 queue 少了 m1、ledger[m1] 變 1、working 維持 idle。逐條規則比對：`Fetch(m1)` 會讓 working 變成 m1（不是 idle）✗；`Settle` 前提要求手上有訊息，S0 是空手 ✗。沒有任何單一規則能完成這個變化——「拿了就地入帳」是**兩步的合成**，不是一步。現實版教訓：你以為原子的操作其實是兩步時，中間就有縫可以被插隊（ch01 的 race 正是這個形狀）。

**(c) 不合法**，第 2 步（S1 → S0）非法。它要求 m1 從手上回到佇列；但 v0 的兩條規則裡，`Fetch` 只會從佇列拿走、`Settle` 根本不碰 queue。「放回去」這個動作不存在，所以這個步不被任何規則允許。（劇透：題 3 會把這個動作加進來，到時它就合法了——合法與否永遠是相對於規則集合說的。）

**(d) 合法。** 每一步（只有一步，S0 → S1，即 `Fetch(m1)`，驗法同 (a)）都合法，序列就是合法行為；v0 沒有任何規則強迫系統繼續走。停在 S1 手握 m1 直到宇宙熱寂，完全合規。如果這讓你不舒服——很好，那個不舒服的學名叫 liveness，工具在 ch07。

### 題 2：候選不變量的審判 **[15 分鐘]** ★★

對 v0 的 8 狀態圖逐一審判下列述語：是不變量的，說明你怎麼驗完；不是的，交出打破它的可達狀態＋一條從 S0 走到它的路徑。

- P1：「至少還有一則訊息未入帳。」
- P2：「不存在這樣的狀態：consumer 手上有訊息，而佇列同時還是滿的（＝ {m1, m2}）。」
- P3：「如果 m2 已入帳，那麼 m1 也已入帳。」（業務部的願望：m1 單比較早建立，應該先結。）

### 推演解答

**P1：不是不變量。** 反例 S7 ＝ ⟨∅, idle, [m1 ↦ 1, m2 ↦ 1]⟩，兩則都入帳了。可達性證據：S0 → S1（Fetch m1）→ S3（Settle m1）→ S5（Fetch m2）→ S7（Settle m2），四步各自驗過前提與效果。P1 把「過程中的常態」誤當「永恆的真理」——很多假不變量都是這樣寫出來的：它在你腦中預設的「進行中」畫面成立，在終局不成立。

**P2：是不變量。** 驗法：手上有訊息的狀態只有 S1、S2、S5、S6，它們的 queue 分別是 {m2}、{m1}、∅、∅——沒有一個是滿的 ✓；其餘四個狀態（S0、S3、S4、S7）手上沒訊息，述語的條件部分落空，不構成違反 ✓。八個全查完，成立。為什麼成立？因為 `Fetch` 把「從佇列移除」與「拿到手上」做成同一步——P2 實際上在偵測「拿了卻沒從佇列消失」這種 bug 形狀。記住這個影子：at-least-once 世界裡它有個名字叫 redelivery，v1 登場後這一帶的所有不變量都要重審（見 ch08）。

**P3：不是不變量。** 反例 S4 ＝ ⟨{m1}, idle, [m1 ↦ 0, m2 ↦ 1]⟩：m2 已入帳、m1 還沒。可達性證據：S0 → S2（Fetch m2，前提 m2 ∈ queue ✓、空手 ✓）→ S4（Settle m2 ✓）。兩步到案。深一層的教訓有兩個。其一，**指控一個述語不是不變量，標準格式是「可達反例＋路徑」**——這條路徑就是 ch09 反例 trace 的手工雛形。其二，P3 死因不在運氣，在抽象：v0 的佇列是集合，「m1 比較早」**在模型裡沒有這句話**；想立順序性質，得先把順序建回模型（見 ch04）。性質與模型必須門當戶對。

### 題 3：加上 consumer 重啟，重畫狀態圖 **[25 分鐘]** ★★★

給 v0（1 個 consumer）加第三條動作規則：

| 規則 | 前提 | 效果 |
|---|---|---|
| `Restart(m)` | working ＝ m | m 放回 queue；working 回 idle；ledger 不變 |

（情境：consumer 重啟，手上沒處理完的訊息回到佇列——SQS visibility timeout 到期的簡化版。）回答：

1. 重畫狀態空間圖（標出新邊）。可達狀態變多了嗎？用封閉性檢查支持你的答案。
2. 行為的集合發生了什麼**質變**？（提示：看長度。）
3. `NoDoublePay` 的看門狗述語還在所有可達狀態成立嗎？
4. 這個 `Restart` 模型跟真實的 consumer crash 差在哪——它看不見哪種事故？

### 推演解答

**1. 狀態數不變，仍是 8；新增 4 條邊。** 先找新邊：`Restart` 的前提是手上有訊息，逐一檢查手上有東西的狀態——

- S1（手持 m1，queue ＝ {m2}）：放回後 queue ＝ {m1, m2}、空手、帳本全 0 ＝ **S0**；
- S2（手持 m2）：放回後 ＝ **S0**；
- S5（手持 m2，queue ＝ ∅，m1 已入帳）：放回後 ⟨{m2}, idle, [m1 ↦ 1, m2 ↦ 0]⟩ ＝ **S3**；
- S6（手持 m1）：放回後 ＝ **S4**。

四條新邊的落點 S0、S0、S3、S4 全是舊面孔；其餘狀態（S0、S3、S4、S7）空手，`Restart` 不可走，不產生邊。原本的 8 條邊也原封不動。清單封閉 ✓——所以可達集合還是那 8 個。**加動作不一定加狀態**；但別把這當通則：新動作的落點只要有一個跳出清單，整張圖就要重新窮舉（這次只是運氣好）。

重畫的圖（▼ ＝ Fetch 往下、▲ ＝ Restart 往上；Settle 邊維持單向）：

```
          ┌────────────────────────────┐
          │         S0  (Q, Q)         │
          └─┬────▲──────────────┬────▲─┘
            │    │              │    │
            ▼    │              ▼    │
        ┌───┴────┴───┐      ┌───┴────┴───┐
        │ S1  (C, Q) │      │ S2  (Q, C) │
        └──────┬─────┘      └──────┬─────┘
               │                   │
               ▼ Settle(m1)        ▼ Settle(m2)
        ┌──────┴─────┐      ┌──────┴─────┐
        │ S3  (L, Q) │      │ S4  (Q, L) │
        └───┬────▲───┘      └───┬────▲───┘
            │    │              │    │
            ▼    │              ▼    │
        ┌───┴────┴───┐      ┌───┴────┴───┐
        │ S5  (L, C) │      │ S6  (C, L) │
        └──────┬─────┘      └──────┬─────┘
               │ Settle(m2)        │ Settle(m1)
               └─────────┬─────────┘
                         ▼
                 ┌───────┴────────┐
                 │   S7  (L, L)   │
                 └────────────────┘
```

（左上雙線：S0 ⇄ S1 之間 ▼ ＝ Fetch(m1)、▲ ＝ Restart(m1)；右上同理對 m2；中段 S3 ⇄ S5 是 Fetch(m2)／Restart(m2)、S4 ⇄ S6 是 Fetch(m1)／Restart(m1)。）

**2. 質變：圖上出現了環，於是出現了無限長的行為。** 原本的圖是一路向下的，所有行為最多 4 步就終結；現在 S0 → S1 → S0 → S1 → … 是合法行為——consumer 永遠在拿起與放下之間鬼打牆，永遠不入帳。「每則訊息終究會入帳」原本只是「沒人保證」，現在是「有具體的無限反例」。要排除這種行為，靠的不是加狀態，而是加公平性假設——ch07 的正題。

**3. 仍然成立。** 關鍵觀察：可達狀態集合沒變（第 1 小題已證），而看門狗述語是**關於狀態**的判斷——8 個狀態上它本來全數成立，現在還是那 8 個狀態，所以還是全數成立。加邊只可能透過「開出新的可達狀態」來打破不變量；這次一個新狀態都沒開出來。另外從機制看也合理：`Restart` 不碰 ledger，且它的前提是訊息在手上——而看門狗述語恰恰保證已入帳的訊息不會在手上，所以 `Restart` 永遠搬不動已入帳的訊息。

**4. 它看不見「入帳成功之後、ack 之前」的 crash。** 我們的 `Settle` 把「寫帳本」與「訊息從系統消失」綁成原子一步，`Restart` 只能發生在入帳前。真實世界裡 consumer 寫完 DB、還沒來得及刪訊息就掛掉——訊息帶著「已入帳」的事實回到佇列，下一個 consumer 撿起來再結一次——這正是雙重入帳的經典開局，而**本模型連表達它的詞彙都沒有**（陷阱表第 2 條：性質空洞成立）。把 `Settle` 拆成兩步、讓這隻幽靈進場、再用 dedup 表把它擋下來，就是 v1 的劇本（見 ch08）；證明擋得住，是 ch14 的戰役。

## 自我檢核

口頭回答，講得出來才算數；每一題的標準是「能教會另一個沒讀過本章的後端工程師」。

1. 用自己的話定義 state、step、behavior、規格，並各舉一個你維護過的系統裡的對應物（提示：快照、交易的 before/after、binlog、API 契約——最後一個為什麼不太對？）。
2. 「規格＝允許行為的集合」這個視角，跟「規格＝程式功能的描述」差在哪？它把誰寫進了模型、把什麼責任丟回給你？
3. 非決定性與隨機差在哪？為什麼驗證要用前者建模調度與故障？「這個 bug 機率太低」在這個世界觀裡是什麼地位？
4. 不變量的「永遠成立」量化在哪個集合上？「型別空間都成立」與「我追過的行為都成立」各差在哪裡、各自對應什麼錯覺？
5. v0 的 `NoDoublePay` 成立——這句話裡有多少是設計的功勞、多少是模型的盲區？你用什麼問題分辨這兩種成分？
6. safety 與 liveness 的一句話對比是什麼？為什麼違反前者的證據是有限的、違反後者的不是？「停在 S1 不動」打破了哪一邊？
7. 從 1 個 consumer 加到 2 個：狀態 8→14、邊 8→24、完整路徑 2→16。哪個數字跟「測試蓋不住」最直接相關？為什麼？
8. 挑一個你手上的系統：你會建模哪三個變數、丟掉哪五個細節、第一條想驗的不變量是什麼？丟掉的細節裡哪一個最可能讓那條不變量空洞成立？
