# ch04 — 集合、函數、關係：TLA+ 的原料

> **本章解決什麼問題**：ch03 給了你精確說話的文法——連接詞與量詞。但句子裡談論的「東西」本身：訊息、佇列、帳本、組態，還沒有精確的身分。本章把集合、函數、關係磨利成建模工具，並立起 TLA+ 的核心世界觀：**函數是一張總是查得到的表**。這是 Part II 數學地基的第二塊：ch05 要在上面做歸納證明，而 ch06 之後每一份 spec 的每個變數，裝的值全是本章的原料。

## 從你已知的出發

你每天的工作語言裡早就有這一章的所有東西，只是掛著別的名字：

| 你每天寫的 | 數學名字 | 本章符號 |
|---|---|---|
| `Set<T>`、SQL 的 DISTINCT 結果 | 集合（set） | {m1, m2}、∈、⊆ |
| `Map<K, V>`（且保證每個 key 都有值） | 函數（function） | [K → V]、f[k] |
| JSON object | record（定義域是欄位名的函數） | [state ↦ "paid"] |
| array | sequence（定義域是 1..n 的函數） | ⟨m1, m2⟩ |
| join table | 關係（relation） | R ⊆ S × T |
| feature flags 的所有組合 | 冪集（power set） | 𝒫(Flags) |

具體一點：你在結算 pipeline 裡維護的 dedup 表——`msgId → 處理狀態`——就是一張函數；SQS standard queue 的 at-least-once 讓「同一則訊息出現兩份」成為日常，這逼你天天面對「該用 set 還是 multiset」的問題，只是你從來不用這個詞；後台那十幾個 feature flag 的全部組合，就是一個冪集。

差別不在名字，在**保證**。`Map.get()` 可能回 `undefined`；LEFT JOIN 會生出 NULL；物件少個欄位要 optional chaining。你的工程直覺裡，「查表」天生是 partial 的——查不到是日常，所以你寫了滿手的 `?.` 與 `??`。數學世界把這件事整個翻過來：**函數對定義域裡的每個輸入都有值，沒有例外**。查不到不是「回 undefined」，而是你問了一個不合法的問題。這個翻轉乍看只是潔癖，但它正是規格語言能夠精確的原因：partial 的地方，就是規格含糊的地方。

回到 ch02 的心智模型：狀態＝給每個變數賦一個值。那「值」是什麼？在 TLA+ 的世界裡答案只有一種：**某個集合的元素**。佇列是一個集合、帳本是一張表、「誰在處理誰」是一個關係。把每個變數允許取值的集合（本書叫它這個變數的**值域**）寫精確，整個系統的狀態空間就有了邊界——本章結尾會替結算系統 v0 把這個邊界真的算出來。

## 集合：唯一的原料

集合是「一堆東西」，由**成員資格**唯一決定：給任何東西 x，「x ∈ S」（x 屬於 S）有明確的是或否。就這麼一條定義，兩個推論直接跟著來，而且都會咬人：

- **沒有順序**：{m1, m2} = {m2, m1}。
- **沒有重複**：{m1, m1} = {m1}。塞第二次，等於沒塞。

這兩條「失憶」性質是本章後半 multiset 討論的伏筆：對 at-least-once 的訊息，第二條會悄悄吞掉你最關心的資訊。

基本操作四件套，配上你熟的對應物：

- **∈ / ∉**（屬於／不屬於）：成員資格查詢，`set.has(x)`。
- **⊆**（包含於）：S ⊆ T ≜ S 的每個元素都在 T 裡。「整包都在範圍內」。
- **∪ / ∩ / \**（聯集／交集／差集）：合併、共同部分、扣掉。SQL 的 UNION／INTERSECT／EXCEPT。
- **=**：兩個集合相等 ⟺ 元素完全相同（順序、寫法、重複都不算數）。

一個馬上要立的警告：**∈ 和 ⊆ 是兩件事**。m1 ∈ Msg 說「m1 是 Msg 的成員」；{m1} ⊆ Msg 說「{m1} 這一包的每個元素都在 Msg 裡」。兩句都對，但 m1 ⊆ Msg 是另一回事（陷阱段細談）。口訣：∈ 問身分，⊆ 問範圍。

還有兩個你天天在寫、值得知道數學名字的操作（這裡用數學記法，TLA+ 寫法見 ch06 之後）：

- **篩選**：{x ∈ S : P(x)}——S 裡滿足 P 的那些。就是 `filter`，也是 SQL 的 WHERE。
- **映射**：{e(x) : x ∈ S}——把 S 的每個元素丟進表達式 e 得到的所有結果。就是 `map`，也是 SELECT 的表達式欄位。

### 冪集與笛卡兒積：組合爆炸的兩具引擎

**冪集** 𝒫(S)：S 的所有子集收成一個集合。拿 Flags ≜ {dark, billing, beta} 來手數一次：

```text
𝒫(Flags) = { {},
             {dark}, {billing}, {beta},
             {dark, billing}, {dark, beta}, {billing, beta},
             {dark, billing, beta} }
```

8 個，= 2³。公式 |𝒫(S)| = 2^|S| 的理由值得口頭講得出來：造一個子集，就是替**每個元素獨立做一次二選一**（進，或不進），n 個元素就是 2 × 2 × … × 2 共 n 次。你的服務有 10 個 feature flag？所有組態就是 𝒫(Flags)，共 1,024 種——你的測試矩陣不可能全蓋。ch01 用交錯數看過組合爆炸，這是它的代數面孔。

**笛卡兒積** S × T：所有「左邊取一個、右邊取一個」的有序對 ⟨s, t⟩。|S × T| = |S| × |T|，純乘法。它是本章最重要的計數引擎，因為**狀態空間天生就是笛卡兒積**：每個狀態＝每個變數各取值域裡一個值，所以「型別正確的狀態總數」就是各值域大小相乘——worked example 全靠它。

## Tuple 與 sequence：順序回來了

集合丟掉的兩樣東西——順序與重複——由 **tuple** 撿回來：⟨m1, m2⟩ 是有序的，⟨m1, m2⟩ ≠ ⟨m2, m1⟩；長度固定；允許重複，⟨m1, m1⟩ 是正經八百的東西。

**sequence** 是「長度不固定的 tuple」：⟨⟩、⟨m1⟩、⟨m2, m1, m1⟩ 都是 Msg 上的 sequence。你的 array 直覺幾乎全部成立，只有一條要記：這個世界的 sequence 是 **1-indexed**——第一個元素是第 1 格，不是第 0 格。

一個之後會回收的事實：S 上所有有限 sequence 的集合（記作 Seq(S)）是**無限的**——長度沒有上限。紙上數學毫不在意；想窮舉的人非常在意（見 ch09）。

tuple 和 sequence「其實是什麼」，等講完函數再揭曉——答案會把本章所有資料結構統一成一種東西。

## 函數：總是查得到的表

本章的主角。先把定義立正。一個函數 f 由兩件事完全決定：

1. **定義域**（domain）：一個集合，列出所有合法的輸入。
2. 對定義域裡**每一個** x，**恰好一個**值 f[x]。

就這樣。沒有第三件事：沒有程式碼、沒有執行過程、沒有副作用、沒有 undefined。函數就是一張表——左欄把定義域的元素列好列滿，右欄一格不空。結算系統的帳本就是一例：

```text
ledger:   m1 ↦ FALSE
          m2 ↦ TRUE
```

（套用寫成 f[x]，跟你的陣列下標同形；這也是之後 TLA+ 的寫法，現在先把手感養起來。）

「總是查得到的表」這個世界觀，有三個推論值得各給一段：

**一、查表不會失敗，但問錯問題沒人救你。**對定義域裡的 x，f[x] 必有值。對定義域**外**的 x 呢？f[x] 不是錯誤、不會拋例外、也不是 undefined——它是**無意義**的式子，值不明，但推理可以面不改色地繼續。Lamport 把這類式子叫 silly expression：「ledger[佇列] 」這種垃圾不會炸，只會默默讓你後面的結論全部作廢。防禦方式見陷阱段；現在先記住手感的差異：程式裡查錯 key 是 runtime 問題，數學裡查錯 key 是**你的問題**。

**二、「每個 key 都有值」從要維護的紀律，變成結構本身的保證。**在 TypeScript 裡，「dedup 表涵蓋所有 msgId」是你要靠初始化邏輯與 code review 守住的不變量；在數學裡，你宣告 ledger ∈ [Msg → BOOLEAN]，涵蓋性就是定義的一部分——不用維護，不會壞。寫 spec 時這是巨大的優點：**你被迫在建模當下回答「這張表到底涵蓋誰」**，而不是把這個問題留給 runtime。

**三、函數是值，不是行為。**一張表可以放進集合、存進變數、跟另一張表比較（f = g ⟺ 定義域相同且逐點相等）。它也可以無限大：「平方」就是一張定義域為自然數的無限長表格，紙上完全合法。在 TLA+ 裡，函數跟數字一樣是一級公民——可以賦給變數、可以當參數傳——語法見 ch06 與 ch08。

**更新＝抄表改一格。**數學函數是 immutable 的。「把 ledger 的 m1 改成 TRUE」的意思是：造一張**新表**，m1 那格填 TRUE，其他格照抄。你寫 reducer 時的 `{ ...state, m1: true }` 就是這個語意，一模一樣。ch06 的 x′（下一個狀態的值）會大量依賴這個觀念：狀態轉移不是「就地改」，是「舊表參照、新表登場」。

最後一個記號：**[S → T]** 表示「定義域是 S、值都落在 T」的**所有**函數收成的集合。它的大小：

|[S → T]| = |T|^|S|

理由跟冪集同款：定義域的每個元素**獨立**挑一個值，|S| 個元素各有 |T| 種挑法。拿 [Msg → BOOLEAN] 手數一次（|BOOLEAN| = 2、|Msg| = 2，所以應該有 2² = 4 張表）：

```text
f1:  m1 ↦ FALSE,  m2 ↦ FALSE
f2:  m1 ↦ FALSE,  m2 ↦ TRUE
f3:  m1 ↦ TRUE,   m2 ↦ FALSE
f4:  m1 ↦ TRUE,   m2 ↦ TRUE
```

四張，沒有第五張。注意指數的方向：**定義域在指數上**。|[S → T]| 和 |[T → S]| 一般不相等，背反了會把上界算錯一個數量級（紙上推演有練習）。

## Record：定義域是欄位名的函數

[id ↦ "p-001", state ↦ "paid"] 長得就像 JSON object，而它的數學身分是：**一張定義域為 {"id", "state"} 的函數**。r.state 只是 r["state"] 的縮寫——欄位存取就是查表。JSON 直覺幾乎全保留，外加一條升級：record 不會「少欄位」——定義域裡的欄位，格格有值。

「某型別的所有 record」也是一個集合，大小用乘法：欄位逐一獨立挑值。例如 [merchant: MerchantId, state: {pending, paid}] 的 record 共 |MerchantId| × 2 個。

現在揭曉剛剛賣的關子。**全家都是表**：

| 東西 | 其實是函數，定義域是…… |
|---|---|
| sequence ⟨a, b, c⟩ | 1..3（第 i 格 ↦ 第 i 個元素） |
| tuple ⟨s, t⟩ | 1..2 |
| record | 欄位名的集合 {"id", "state", …} |
| bag（下一節） | 元素的集合，值是「份數」 |

學會一種東西（函數），整個資料結構動物園就統一了。這是 TLA+ 數學的經濟學：概念極少，組合力極強。learntla 講得更直白：sequence 和 structure「只是語法糖」。

## 關係：join table 的數學名字

**關係**（relation）：R ⊆ S × T——「哪些配對成立」的完整清單。你資料庫裡的多對多 join table，例如 payout_msgs(payout_id, msg_id)，**就是** PayoutId × MsgId 的一個子集。這裡沒有類比修辭：逐字面相等。

三件值得帶走的事：

- **函數是特殊的關係**：每個 s 恰好配一個 t 的那種。你在 join table 上加 UNIQUE 約束，就是把關係收緊成函數——DDL 約束＝替關係加數學結構，同一個動作。
- **1-N、N-N** 這些你畫 ER 圖的詞，全是關係的形狀描述：N-N 是任意子集，1-N 是沿某個方向「至多一個」。
- **計數**：S × T 上的關係共 |𝒫(S × T)| = 2^(|S| × |T|) 個。3 × 4 = 12 格的勾選表，每格獨立打勾或不打：2¹² = 4,096 種。關係值域有多胖，這條公式先告訴你。

「誰在處理誰」——consumer 與訊息之間——天生是個關係。但若每個 consumer 同時至多處理一則訊息，這個關係沿 consumer 方向就是函數。worked example 替 working 變數挑值域時，用的正是這個觀察。

## Set、sequence 還是 bag？——at-least-once 的訊息該裝在哪

你的結算 pipeline 用 SQS standard queue：at-least-once、不保證順序（FIFO queue 是另一個產品、另一份保證）。建模「還沒處理完的訊息」時，三種容器擺在面前。先看各自的記憶力：

| 容器 | 記得什麼 | 忘記什麼 | 「第二份 m1 抵達」這步會怎樣 |
|---|---|---|---|
| set | 在不在 | 份數、順序 | queue ∪ {m1} ＝ 原樣——被靜默吞掉 |
| sequence | 份數＋順序 | （最少） | 真的多一份，而且有位置 |
| bag | 份數 | 順序 | m1 的計數 +1 |

**bag**（又叫 multiset，多重集）不需要新原料：它就是「元素 ↦ 正整數份數」的函數——又一張表。TLA+ 的標準模組 Bags 正是這樣編碼的：bag B 裡，e 有 B[e] 份。

接下來是誠實的部分。三個觀察：

**第一，set 不一定錯，但它錯起來無聲。**判準是你關心的性質。如果性質只問「這則訊息還可能被遞送嗎」——一個是非題——set 通常夠用：redelivery 可以建模成「處理過之後，訊息還能再被放回佇列」的動作，雙重處理的危險交錯依然存在於模型裡。但如果你下意識以為「set 自動去重＝幫我擋掉了重複」，那你不是建模了 dedup，是**免費假設**了 dedup——而現實的 standard queue 沒給你這個保證。模型比現實乾淨的地方，正是 bug 的藏身處。

**第二，sequence 白送你一個現實沒有的保證。**用 sequence 建模 standard queue，模型自動擁有全序。任何依賴「先進先出」才成立的性質，在這個模型裡會漂亮地通過驗證——而在生產環境沒有任何依據。這比「漏抓 bug」更陰險：**你驗證的是一個比現實更乖的系統**。方向感要建立起來：寧可讓模型比現實「更亂」（多出一些現實不會發生的行為——代價是多花力氣排除誤報），不要讓模型比現實「更乖」（少掉現實會發生的行為——漏掉真 bug，而且毫無徵兆）。

**第三，bag 最忠實，帳單也最貴。**份數沒有先天上界——重送可以一再發生——所以狀態空間直接變成無限。想窮舉就得人為設限（「至多重送 2 次」），而這個上限本身又是一條要寫明、要辯護的建模假設（small scope 的辯護與限度，見 ch09）。所以 bag 不是「永遠最對」，而是「性質真的依賴份數時，才值得付錢」：backpressure（in-flight 總數 ≤ K）、「同一訊息至多被遞送 N 次」這類計數性質，非它不可。

決策框架收成一句話：**先寫性質，再挑容器**。讓「性質需要分辨哪些狀態」決定容器，而不是讓容器的預設行為悄悄改寫你的性質。結算系統 v1 要正面處理 redelivery 與 dedup 表，它最後挑了什麼容器，見 ch08。

## 有限、無限，與「一切皆集合」

Msg 有限；自然數、Seq(S) 無限。紙上推演與證明不怕無限——ch05 與 ch14 的歸納法正是對付無限的武器；窮舉怕得要死（ch09）。替變數挑有限還是無限的值域，是「想交給模型檢查」與「想要一般性定理」之間真實的取捨，本書兩條路都會走。

最後補上一句承諾過的話：TLA+ 的數學底是 **ZF 集合論**（Zermelo–Fraenkel set theory），在這個世界裡**一切皆集合**——數字、tuple、函數、record，原則上都能編碼成集合。你永遠不需要看那些編碼，但這個事實有一個工程上的直接後果：**TLA+ 沒有型別系統**。「x 是布林值」不是編譯期的判定，而是 x ∈ BOOLEAN 這個普通命題——可真可假、要被檢查或證明。所以 {1, "a", {2}} 是完全合法的異質集合，沒有編譯器攔你；也所以之後每份 spec 的第一個不變量幾乎都是 TypeOK——「所有變數都落在宣告的值域裡」在這個世界要當**定理**來驗，而不是當型別來標（寫法見 ch08，機器怎麼驗見 ch09）。自由與風險，是同一枚硬幣。

## 完整推演：結算系統 v0 的值域與狀態空間上界

把本章的原料一次用上。v0 的口語規格（ch02 引入，safety 性質 `NoDoublePay`——同一 msgId 至多入帳一次——也在那裡登場）：兩則訊息 m1、m2 等著入帳；consumer 從佇列撿訊息、入帳；帳本記錄每個 msgId 是否已入帳。先一個 consumer，再加到兩個。

**第一步：固定常數集合。**

- Msg ≜ {m1, m2}
- Consumer ≜ {c1}（第五步加到 {c1, c2}）

**第二步：替每個變數挑值域。**這就是「用集合語言定義狀態型別」，每個決定都附理由：

- **queue**——還沒被撿走的訊息。v0 沒有 redelivery，每則訊息至多出現一份，set 不丟資訊（上一節的判準在這裡兌現：先確認動作與性質都不依賴份數，才有資格用 set）。值域：**𝒫(Msg)**。
- **working**——每個 consumer 手上正在處理哪則訊息。「沿 consumer 方向恰好一個值」——函數。手上沒事做也要能表示，值域加哨兵 idle，並要求 **idle ∉ Msg**（撞名陷阱見下節）。值域：**[Consumer → Msg ∪ {idle}]**。
- **ledger**——基準定義：msgId → 是否已入帳。一張涵蓋所有 msgId 的布林表。值域：**[Msg → BOOLEAN]**。

三條合起來，就是 v0 的型別述語（ch08 會叫它 TypeOK，而且它幾乎總是 spec 的第一個不變量）：

queue ∈ 𝒫(Msg) ∧ working ∈ [Consumer → Msg ∪ {idle}] ∧ ledger ∈ [Msg → BOOLEAN]

一個狀態＝三個變數各取值域中一個值。v0 裡一則訊息的一生：

```text
┌──────────┐  Fetch   ┌──────────────┐  Settle   ┌────────────────┐
│  queue   │ ───────► │   working    │ ────────► │ ledger[m]=TRUE │
│ 還在佇列 │          │ consumer 手上│           │     已入帳     │
└──────────┘          └──────────────┘           └────────────────┘
```

**第三步：逐個數值域大小，不跳步。**

- queue：|𝒫(Msg)| = 2² = **4**。逐一列出：{}、{m1}、{m2}、{m1, m2}。✓
- working（1 個 consumer）：|Msg ∪ {idle}| = 3，所以 |[{c1} → Msg ∪ {idle}]| = 3¹ = **3**。逐一列出：(c1 ↦ idle)、(c1 ↦ m1)、(c1 ↦ m2)。✓
- ledger：|[Msg → BOOLEAN]| = |BOOLEAN|^|Msg| = 2² = **4**。逐一列出：就是前面數過的 f1、f2、f3、f4 四張表。✓

**第四步：乘起來。**

狀態空間上界 = 4 × 3 × 4 = **48**。

這個 48 是什麼？是所有**型別正確**的狀態總數——三個值域的笛卡兒積大小。它**不是**可達狀態數。可達狀態是「從初始狀態出發、沿著動作一步一步走得到」的那些（ch02 正在一張一張手畫）；上界只看值域，完全不看動作。一句口訣，全書沿用：**值域給上界，動作決定可達**。

可達狀態遠少於 48。為什麼？因為 v0 的動作維護著一條守恆律：**每則訊息在任一時刻恰好在一個地方**——佇列裡、某個 consumer 手上、或已入帳。48 格裡大多數狀態違反它。例如 ⟨queue = {m1, m2}, working = (c1 ↦ m1), ledger = f1⟩：m1 同時在佇列裡又在 c1 手上——沒有任何動作序列能走到這裡。它型別正確，卻永遠不可達。可達狀態的精確張數與全圖，是 ch02 手畫圖的工作；兩邊的帳必須對得起來——那張圖的每一個節點，都得是這 48 格之一。

順帶一提：上界既然總是高估，算它幹嘛？三個實際用途。（a）**可行性試算**：動作都還沒設計就能算，30 秒告訴你這組參數值不值得手畫或窮舉；（b）**對帳工具**：任何聲稱的可達數若超過上界，必有人算錯；（c）**設計訊號**：值域選太胖，上界第一個告訴你。

**第五步：加到 2 個 consumer。**

只有 working 的值域變了：|[{c1, c2} → Msg ∪ {idle}]| = 3² = **9**（兩個 consumer 各自獨立挑 3 種）。上界 = 4 × 9 × 4 = **144**。加一個 consumer，上界乘一次 |Msg ∪ {idle}| = 3。

**第六步：把通式寫下來，感受指數。**

m 則訊息、c 個 consumer：

上界 = 2^m × (m+1)^c × 2^m = **4^m × (m+1)^c**

| m（訊息） | c（consumer） | 上界 |
|---|---|---|
| 2 | 1 | 16 × 3 = 48 |
| 2 | 2 | 16 × 9 = 144 |
| 3 | 2 | 64 × 16 = 1,024 |
| 4 | 3 | 256 × 125 = 32,000 |
| 10 | 4 | 1,048,576 × 14,641 ≈ 1.5 × 10¹⁰ |

ch01 用交錯數看過組合爆炸，這是它的值域版本：**每加一個變數，上界乘上整個值域的大小；每加一個元素，好幾個值域同時長胖**。這也是為什麼 Part III 之前我們把參數釘得這麼小——不是偷懶，是讓紙還夠用。

## 陷阱與防禦

本章的概念怎麼被誤用、驗證怎麼假通過、怎麼自我察覺：

| 陷阱 | 怎麼害你（往往無聲） | 防禦 |
|---|---|---|
| ∈ 與 ⊆ 混用 | m1 ⊆ Msg 在 ZF 裡甚至不報錯（一切皆集合，式子合法），只是值不明——推理照走，結論作廢 | 口訣：∈ 問身分、⊆ 問範圍。檢查兩側：∈ 左邊是元素；⊆ 兩邊都是集合 |
| 用 set 裝會重複的東西 | 第二份訊息被 ∪ 靜默吞掉；「佇列中每則訊息至多一份」之類的性質**由結構保證恆真**——驗了等於沒驗（vacuous truth 的資料結構版，ch03 的老朋友） | 對每個 set 變數問：「現實裡兩份能同時存在嗎？」能，就考慮 bag 或顯式計數 |
| 用 sequence 白拿順序 | 模型比現實乖：依賴 FIFO 的性質在模型裡通過，在生產環境沒有依據 | 問「系統真的承諾順序嗎」；性質不依賴順序就降級到 set／bag，讓模型保持「比現實亂」 |
| 定義域外查表 | f[x]（x ∉ 定義域）是 silly expression：不是錯誤、是垃圾，後續推理面不改色地繼續 | 每個函數寫下定義域；之後養成 TypeOK 習慣（ch08），讓檢查器抓（ch09） |
| 哨兵值撞名 | 若 idle ∈ Msg，「c1 沒事做」與「c1 在處理一則叫 idle 的訊息」是**同一個狀態**——狀態空間悄悄黏合，數錯、推錯都從這裡開始 | 哨兵與正常值取自不相交的集合；之後把 idle ∉ Msg 寫成 spec 的明文前提（ch08 的 ASSUME 預告） |
| 值域選太胖 | 上界爆炸；更糟的是胖值域裡大量「型別正確但概念荒謬」的狀態，會讓 ch14 找 inductive invariant 時多受很多苦 | 先算上界再定案；夠用 {0, 1, 2} 就不用自然數，夠用函數就不用關係 |

共同的後設教訓：這些陷阱**全都不報錯**。數學不會替你 lint；自我察覺的訊號是「一條性質怎麼改動作都不會破」（懷疑被結構保證，而不是被設計保證）與「一個式子兩側的東西種類對不上」（懷疑 silly expression）。

## 紙上推演

### 題 1：集合算術熱身 **[10 分鐘] ★**

設 S ≜ {a, b, c}、T ≜ {b, c, d}。手算：

1. S ∪ T、S ∩ T、S \ T、T \ S。
2. |S × T|、|𝒫(S)|，以及 S × T 上所有關係的個數。
3. |[S → T]| 與 |[T → S]|——一樣大嗎？若把 T 換成 {b, c, d, e} 呢？
4. 你的服務有 6 個 feature flag。所有組態幾種？「包含 dark_mode」的組態幾種？

### 推演解答

1. S ∪ T = {a, b, c, d}（重複只算一次）；S ∩ T = {b, c}；S \ T = {a}；T \ S = {d}。
2. |S × T| = 3 × 3 = 9。|𝒫(S)| = 2³ = 8：{}、{a}、{b}、{c}、{a,b}、{a,c}、{b,c}、{a,b,c}。關係＝S × T 的子集，所以共 |𝒫(S × T)| = 2⁹ = 512 個——一張 9 格勾選表，每格獨立打勾或不打。
3. |[S → T]| = |T|^|S| = 3³ = 27；|[T → S]| = |S|^|T| = 3³ = 27。一樣大是巧合（兩集合等大）。把 T 換成 4 個元素：|[S → T]| = 4³ = 64，|[T → S]| = 3⁴ = 81——不一樣了。**定義域在指數上**，方向背反，上界就錯。
4. 2⁶ = 64 種。包含 dark_mode：把它釘死為「在」，剩 5 個 flag 自由，2⁵ = 32 種。常見錯路是想用 64 去除以什麼——冪集計數永遠回到「每個元素獨立二選一」的結構去想，不要硬套排列組合公式。

### 題 2：把一張 DB schema 翻成 record 與關係 **[20 分鐘] ★★**

結算系統的周邊 schema：

```text
merchants(id PK, status)              -- status ∈ {active, frozen}
payouts(id PK, merchant_id FK, state) -- state ∈ {pending, paid}
payout_msgs(payout_id FK, msg_id FK)  -- 多對多，無其他欄位
```

用本章的原料寫出三張表的「型別」：每張表是什麼數學物件、值域是什麼。並回答：（a）PK 的唯一性在你的翻譯裡由什麼保證？（b）若 payouts 加一個 nullable 欄位 paid_at，怎麼建模？（c）若在 payout_msgs 上加 UNIQUE(msg_id)，數學上發生了什麼？

### 推演解答

先固定 id 的宇宙：MerchantId、PayoutId、MsgId 各是一個有限集合。

- **merchants ∈ [MerchantId → [status: {active, frozen}]]**——以 PK 為定義域的函數，值是「其餘欄位」的 record。注意我們**沒有**翻成「record 的集合」：若用 set of records，兩筆同 id、不同 status 的 record 可以同時共存，PK 唯一性就得另外當不變量去維護；翻成函數，唯一性是結構免費送的。若想表達「不是所有 MerchantId 都已註冊」，就讓定義域是 MerchantId 的某個子集——這是一個要寫明的建模決定。
- **payouts ∈ [PayoutId → [merchant: MerchantId, state: {pending, paid}]]**。外鍵約束變成值域宣告：merchant 欄位的值**只能**落在 MerchantId 裡。
- **payout_msgs ⊆ PayoutId × MsgId**——多對多、無其他欄位的 join table 就是關係本人，一個字都不用多寫。

（a）函數對定義域裡每個元素恰好給一個值——PK 唯一性由「函數」這個結構保證，零成本。（b）nullable＝值域加哨兵：paid_at 的值域取 Timestamp ∪ {none}，並要求 none ∉ Timestamp（哨兵撞名）。另一條路：把 paid_at 抽成一張只涵蓋「已付款 payout」的部分表。兩條都合法；後者多管理一個定義域，但不必發明 none。挑哪條，理由要寫下來。（c）UNIQUE(msg_id) 把關係沿 MsgId 方向收緊成「每個 msg 至多配一個 payout」——關係變成以 MsgId 一側為定義域的（部分）函數。你在 DDL 裡寫的每個約束，都是在替關係加數學結構；反過來說，關係的數學形狀，DB 沒有約束替你守。

### 題 3：set 建模佇列，到底丟了什麼 **[15 分鐘] ★★★**

v1 之前的思想實驗。把佇列建模成 set，動作兩個：Send 把訊息加進 queue；Deliver 把訊息從 queue 移除、交給 consumer。broker 是 at-least-once：同一則訊息可能被 Send 兩次。

（a）queue = {m1, m2} 時第二份 m1 抵達（Send(m1)），前後的 queue 值各是什麼？這一步在模型裡「看起來像」什麼？
（b）描述一個現實中可能發生、但這個模型**表示不出來**的情境（提示：兩份 m1 同時 in-flight）。
（c）寫一條在這個模型裡**恆真**、但對真實 standard queue **不成立**的性質，並說明為什麼這種「通過」最危險。
（d）改用 bag 後，（a）那一步變成什麼？代價是什麼？

### 推演解答

（a）前：{m1, m2}；後：{m1, m2} ∪ {m1} = {m1, m2}。**狀態完全沒變**——這一步在模型裡等於「什麼都沒發生」。重送事件被結構靜默吞掉，而且模型裡不留任何痕跡讓你發現它被吞掉。

（b）現實：佇列裡同時躺著兩份 m1；c1 取走一份開始處理，**另一份還在佇列裡**，隨後被 c2 取走——兩個 consumer 同時在處理 m1，`NoDoublePay` 岌岌可危。set 模型裡，Deliver(m1) 把 m1 從集合移除之後，佇列裡就**完全沒有** m1 了；「取走一份、還剩一份」這個中間狀態不存在於任何可達狀態。（模型還能不能間接製造雙重處理？能——如果你另外寫一個「重新入列」的動作。但那是你**手動補回**被 set 丟掉的行為，不是 set 替你保留的。誠實的建模者要分得清這兩件事。）

（c）例如：「任一時刻，queue 裡每則訊息至多一份」。在 set 模型裡這是**結構恆真**——動作隨便亂改都不可能違反，於是「驗證通過」。對真實的 standard queue，它就是假的。這種通過最危險，因為它長得跟真的驗證一模一樣：檢查跑了、綠燈亮了，但這條性質從頭到尾沒有檢查任何東西——ch03 的 vacuous truth 在資料結構層的轉世。自我察覺的訊號：一條性質如果「怎麼改動作都不會破」，要懷疑它是被**結構**保證的，不是被**設計**保證的。

（d）bag 版：m1 的份數 1 → 2——重送第一次在狀態裡留下痕跡；（b）的情境也能忠實表示（份數 2，Deliver 後份數 1，另一份照常可被遞送）。代價：份數沒有先天上界，狀態空間變無限；想窮舉就得設「至多重送 k 次」的人為上限，而 k 本身又是一條要寫明、要辯護的假設（ch09）。沒有免費的忠實——這正是「先寫性質，再挑容器」的理由。

## 自我檢核

口頭自答，講得出來才算過關：

1. 用 60 秒向同事解釋「函數＝總是查得到的表」，並說明它跟 `Map.get()` 可能回 undefined 的差別，為什麼對寫規格是優點而不是不便。
2. 口頭推導 |[S → T]| = |T|^|S|。接著心算：4 則訊息、3 個 consumer 時 working 的值域多大？（應得 5³ = 125——說得出 5 是誰、為什麼在 3 次方。）
3. record、tuple、sequence 各自「是定義域為什麼的函數」？三句話一口氣講完。
4. 佇列可能有重複訊息時，set、sequence、bag 各記得什麼、忘記什麼？「模型比現實乖」和「模型比現實亂」哪個更危險？為什麼？
5. 「值域給上界，動作決定可達」是什麼意思？上界明明總是高估，它的三個實際用途是什麼？
6. TLA+ 為什麼沒有型別系統？「x 是布林值」在一切皆集合的世界裡是什麼？這跟「TypeOK 是不變量而不是型別標註」有什麼關係？
7. 為什麼 |𝒫(S)| = 2^|S|？一個值域是 𝒫(S) 的變數，會讓狀態空間上界付出多大代價？
