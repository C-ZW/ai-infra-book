# ch18 — 形式化方法地圖與下一步

> **本章解決什麼問題**：全書最後一章，做三件收尾的事。第一，把 ch01 攤開的地圖收攏：那個結算夜的 bug 拿回來重走一遍，看看十七章之後的你跟當年差在哪裡。第二，誠實介紹 TLA+ 的鄰居——Alloy、P、Quint、Event-B——各自解什麼問題、什麼問題該換誰上場，以及設計級與程式碼級驗證的那條分界線。第三，也是對你的日常最實用的：把「沒有工具也帶得走」的輕量紅利打包——property-based testing、deterministic simulation testing、在 design doc 寫 invariant 的習慣。讀完這章，這本書就交還給你的工作了。

## 從你已知的出發

把 ch01 開場那一夜搬回來。季結算夜，監控撈出兩筆重複入帳：同一個 msgId、兩筆款項、相隔 11 毫秒。當時的你修好了它——檢查與寫入包進同一筆交易、msgId 上加 unique constraint——然後在事故報告的最後一節停筆很久，因為有個問題沒人敢大聲回答：**還有沒有別的調度會出事？**

現在換今天的你來寫那份報告。

**第一，你的眼睛先把程式拆成動作。**dedup 檢查一步、入帳一步——check 與 act 之間有一條縫。當年的你看到縫會皺眉；現在的你會把它寫下來：兩個 action、4 個時間槽，C(4,2) = 6 種交錯、4 種壞（ch01 推演題 1，你親手數過）。那 11 毫秒不再是靈異事件，是六條劇本裡編號可查的一條。

**第二，「同一 msgId 至多入帳一次」不再是 code review 的口頭禪。**它是 `NoDoublePay` ≜ ∀ m ∈ Msgs：ledger[m] ≤ 1——一條可反駁的述語（ch03）：要嘛舉出一條打破它的可達路徑，要嘛說明為什麼不存在。含糊地帶被量詞抽乾了。

**第三，修法不再是「包個交易，應該就好了」。**你知道交易為什麼有效：原子化把六條劇本砍成兩條，砍掉的剛好是全部的壞劇本（ch01 推演題 1(c)）。你也知道那句「應該」的全名：Credit 的原子性是 spec 裡最承重的**前提**，不是簡化（ch08 的抽象帳）——哪天有人把「查」與「寫」拆出交易，定理對你的系統就失效。下次 review 看到那種 PR，你攔得下來，而且理由說得出口。

**第四，那個沒人敢回答的問題，第一次有了全稱的答案。**基準參數下的窮舉你手推過（ch09 的 BFS）；任意訊息數、任意 consumer 數的版本，歸納不變量的證明你寫過（ch14）：dedup 只進不出、Credit 拿它當門票，Init 成立、每步保持，所以永遠成立。在模型涵蓋的世界裡——沒有，沒有別的調度會出事。

**第五，「在模型涵蓋的世界裡」這個限定詞，你不會吞掉。**dedup 表被 TTL 清掉、慢 consumer 的重複投遞、佇列服務自己抽風——v1 的抽象帳一筆一筆白紙黑字（ch08）。你簽的是哪份契約、契約建立在哪些前提上、前提破了會從哪裡先壞，一條一條說得出來。這是 Murphy 原則的形式化版：不是消滅不確定，是讓每一份安全感都有出處。

**第六，下一個系統你不會等到結算夜。**設計審查時 invariant 先寫好；寫不精確的地方，就是設計還沒收斂的地方（ch01 的藍圖論點、ch17 的 ROI 帳）。槓桿最大的時刻在第一行實作之前。

注意一件事：這六樣沒有一樣需要安裝任何工具。ch01 說這本書要做的，是把你的 code review 直覺接上數學，讓它從兩三步深走到 35 步深——你已經在 2PC 的 blocking scenario（ch11）、Paxos 的雙 proposer 競爭（ch12）上親手走過比當年深一個數量級的推演。工具可以之後再裝；眼睛已經換好了。

## 工具光譜：TLA+ 的鄰居們

ch01 的地圖用兩個軸把大陸劃開：驗哪個層次（設計 vs 程式碼）、怎麼驗（找反例 vs 寫證明）。TLA+ 佔的是「設計層」那一整排。但同一排還有幾個風格迥異的住戶——以下每個一段，只談定位與「什麼問題會讓你選它」，不教語法。

**Alloy：關係邏輯，結構優先。**Daniel Jackson（MIT）的 Alloy 把世界建模成「集合與關係」：signature 宣告原子的種類，關係把它們連起來，約束用關係邏輯寫。Analyzer 把問題翻成 SAT，在你指定的 small scope 內（每種 signature 至多幾個原子）窮舉所有結構，找到滿足（或違反）約束的**實例**畫成圖給你看。它的賭注是 small scope hypothesis 的結構版：設計層的錯，幾乎都能在很小的實例上現形（這個賭注的限度，ch09 討論過）。第 6 版起合併了 Electrum 路線的時序運算子，也能寫 behavior 了（2026-06 開頁查證，官方 model finding 引擎為 Kodkod）——但它的甜蜜點仍是結構，不是交錯。什麼問題會讓你選它：「在我們 org × 群組 × 個人覆寫三層權限規則疊加之下，**存不存在**一個使用者能同時通過 A 檢查又繞過 B？」——狀態的**形狀**比狀態的**演化**更複雜的問題：權限模型、schema 與遷移、組態空間、網路 ACL。一句話對比：TLA+ 問「會不會走到壞狀態」，Alloy 問「存不存在壞結構」。

**P：事件驅動，可執行。**P 把系統寫成一組互傳事件的 communicating state machines——你的微服務心智模型的形式化版，官網的自述就是「a state machine based programming language for formally modeling and specifying complex distributed systems」（2026-06 開頁查證）。跟 TLA+ 最大的差異是「可執行」：P 程式不只是紙上模型，配套的 checker 會系統性地探索訊息交錯與故障注入去找深層 bug。AWS 從 2019 年起把它納入主力工具箱、用在 S3 等旗艦服務（採用脈絡與工程帳見 ch17——這裡只談語言定位）。什麼問題會讓你選它：「我的系統就是一堆異步收發事件的服務，我要的不是一份文件型 spec，而是一個能直接跑、能餵故障的模型。」它把 spec 從文件變成可執行的測試資產——代價是你多了一份要跟實作同步的程式，ch17 的 spec 腐化問題照樣適用。

**Quint：TLA+ 語意的現代外衣。**Informal Systems 開發的規格語言，官網自介是「executable specification language for reliable systems」（2026-06 開頁查證；網址已遷至 quint.sh）。語意就是 TLA+ 那一套——state、action、不變量、時序性質——但語法長得像現代程式語言：有型別、有像樣的 CLI 與編輯器整合、錯誤訊息說人話；內建模擬器能隨機跑 trace，要做真正的模型檢查時接 Apalache 當後端（Apalache 對 TLA+ 與 Quint 都支援，如今由原核心開發者獨立維護，見 ch16）。什麼問題會讓你選它：「我想把這本書的思維帶進團隊，但我很確定同事看到 `/\` 跟 `ENABLED` 就會關掉視窗。」本書教你的每一個概念在 Quint 裡原樣成立——它換的是皮，不是骨。

**Event-B：精化驅動。**TLA+ 把 refinement 當成一個深刻的進階主題（ch15）；Event-B 把它當成整個方法的骨架：從一台極抽象的機器出發，一步一步精化到貼近實作，**每一步**精化都自動產生 proof obligations，配套的 Rodin 平台管理整條證據鏈。什麼問題會讓你選它：安全攸關等級的系統（鐵路聯鎖、航空），需要「從需求到實作每步都有證明」的完整證據鏈交給認證單位——歐洲安全關鍵產業的傳統路線（定位穩定；近年版本動態本書未逐一查證）。

**SPIN 與 NuSMV，一句話。**SPIN（Promela 語言、LTL 性質）是顯式狀態模型檢查的元老、協議驗證論文的常客；NuSMV 走符號路線（BDD），出身硬體驗證——文獻裡反覆出現的這兩個名字，是「TLC 的長輩們」。

收成一張表：

| 工具 | 核心抽象 | 檢查方式 | 什麼問題會讓你選它 |
|---|---|---|---|
| TLA+ | 狀態機＋時序邏輯 | 窮舉（TLC）／SMT（Apalache）／證明（TLAPS） | 併發 × 故障交錯下的協議設計 |
| Alloy | 集合與關係 | SAT、small scope 找實例 | 結構與規則：權限、schema、組態 |
| P | 通訊狀態機 | 可執行、系統性測試＋故障注入 | 事件驅動服務群，要能跑的模型 |
| Quint | TLA+ 語意、現代語法 | 模擬器＋Apalache | 想推 TLA+ 思維給吞不下語法的團隊 |
| Event-B | 逐步精化的機器 | 每步精化生證明義務（Rodin） | 安全攸關、要全程證據鏈 |

## 設計級與程式碼級：分界線上的三個路標

ch01 畫過那條線：規格層驗設計、程式碼層驗實作。三個路標讓你記得程式碼級長什麼樣子、貴成什麼樣子：

- **seL4**：一個 microkernel 的 C 實作，被 Isabelle/HOL 證明在功能上符合其抽象規格——作業系統核心等級的全程式碼驗證。人力以人年計。它是「做得到」的存在性證明，也是成本的刻度原器。
- **CompCert**：經過機器證明的 C 編譯器，定理是「編譯不改變程式語意」。它用的證明器是 Rocq——就是 Coq，2025 年 3 月隨 9.0 版完成改名；讀舊文獻時兩個名字是同一個東西。
- **Dafny**：把驗證內建進程式語言——函式上寫 pre/post-condition、迴圈寫 invariant，SMT 求解器在你打字時逐條檢查。它是程式碼級驗證最平易近人的入口；但驗的單位是函式與模組，不是分散式協議。

分界線的實務讀法（ch16 的嚴謹度光譜在這裡落地）：程式碼級比設計級貴一到兩個數量級。TLA+ 的賭注是「最要命的 bug 多半在設計層，殺掉它們的性價比最高」；程式碼級工具的賭注是「實作也會背叛設計」。兩個賭注都對，所以兩層不互斥——而夾在中間的「spec 與 code 是否一致」正是工業界的真實痛點，MongoDB 為此付過的學費見 ch17。你大概一輩子不會親手碰 seL4 等級的工程，但知道光譜的右端在哪裡，左端的每個取捨才有座標。

## 輕量形式化：沒有工具也帶得走的紅利

這一節是本章的實用核心。把 ch01 的投資光譜重畫一次，這回把左半邊放大：

```text
例子測試 ──► property-based testing ──► deterministic simulation ──► 模型檢查 ──► 證明
（一條執行）  （任意輸入、抽樣調度）     （任意故障、調度可重放）      （窮舉）     （全稱）
```

左邊三格不需要你學任何規格語言，但每一格都直接吃到這本書練出來的能力。

### Property-based testing：把 invariant 接上你的測試框架

example test 寫「給定輸入 X，期望輸出 Y」；property-based testing（PBT）寫「對**任意**輸入，性質 P 成立」，然後讓框架生成幾百組輸入轟炸實作，找到反例後自動 shrinking 到最小重現輸入。你的技術棧裡現成就有：fast-check（TypeScript）。

它跟這本書的關係是一句話：**寫 property 的能力＝寫 invariant 的能力，而你剛練完十七章。**好寫的性質形狀你全都見過：roundtrip（decode(encode(x)) = x）、守恆（轉帳前後總額不變）、冪等（f(f(x)) = f(x)——你的 idempotent consumer 的靈魂）、oracle 對照（新實作 vs 舊實作逐筆對答案）、結構不變量（排序後長度不變且有序）。

誠實的定位：PBT 仍是抽樣，ch01 那條「抽樣到不了全稱」的線它沒有跨過去；它對純函式銳利、對併發遲鈍——調度依然不是它能控制的輸入（本章 worked example 會把這個洞演給你看）。它做的是把 example test 的存在性陳述往全稱方向推了一大步：抽樣密度與無偏性遠勝手寫例子，而且性質本身就是文件。

### Deterministic simulation testing：把調度變成輸入

ch01 把測試的根本困境說成一句話：**調度是你控制不了的輸入。**deterministic simulation testing（DST）的回答不是窮舉它（那是 TLC 的路），而是把它**變成可控的輸入**：整個叢集——所有節點、網路、磁碟、時鐘——跑在同一個單執行緒模擬器裡，一切非決定性由一顆帶種子的偽亂數產生器供應。同一個種子，同一場災難，逐位元重放。

FoundationDB 是這條路的代表作。官方文件（2026-06 開頁查證）描述他們用自製的 actor 語言 Flow 把整個叢集塞進單執行緒行程，對網路、機器、資料中心三個層級注入故障——連線斷裂、機器降速、重開機、硬碟塞滿——並自估累計跑過約一兆 CPU 小時量級的模擬。文件裡那句結論值得抄在這裡：「it seems unlikely that we would have been able to build FoundationDB without this technology」。這條思路近年也被產品化成通用測試服務（如 Antithesis；其文件對 DST 有清楚的綜述，連結見延伸閱讀）。

跟模型檢查的對照：TLC 窮舉**小模型的全部**；DST 抽樣**真實實作的故障空間**，但每個樣本可重放、可除錯。一個驗設計、一個驗實作——DST 最吸引人的地方正是它直接驗你出貨的那份程式碼，沒有 spec 與 code 漂移的問題（ch17 的痛點）。代價是架構侵入性：I/O、時間、亂數全部要走可注入的介面，最好從第一天就這樣設計；事後改造一個到處 `Date.now()` 與裸 `fetch` 的 codebase，工程量近似重寫。順帶一提，Node.js 的單執行緒 event loop 給了你一點先天優勢——沒有搶占式執行緒交錯，把外部 I/O 與計時器 shim 掉之後，決定性模擬比多執行緒語言可行得多。

### 在 design doc 寫 invariant：零工具的形式化

光譜最左邊還有一格沒畫出來，因為它連測試框架都不要：**把 invariant 寫進 design doc 的習慣**。模板三段式，每條 invariant 三行：

1. **述語**：盡可能量化的一句話。不寫「系統應該不會重複入帳」，寫「∀ msgId：入帳次數 ≤ 1」。
2. **機制**：它靠什麼成立。「(msgId) 上的 unique constraint＋檢查與寫入同一交易」。
3. **破壞假設**：什麼事發生它就破。「dedup 紀錄的 TTL 短於最大重投遞窗口；或任何繞過該交易的補單腳本」。

寫的過程就是 ch01 的藍圖時刻：寫不出述語＝你還沒想清楚；寫不出機制＝你在祈禱；寫不出破壞假設＝你還沒做過 Murphy 模擬。配套的 design review 三問，從 ch06 與 ch08 的鐵律蒸餾而來：

- 這兩步之間，能插誰？
- crash 落在這條縫裡，會怎樣？
- 這個檢查用到的資訊，做檢查的那一方**真的看得到嗎**？

三問不需要任何符號，但每一問背後都是一章（動作的縫、故障建模、guard 的視力）。這就是「沒有工具也帶得走的思維紅利」的具體形狀——不是一種氣質，是一張可以印出來貼在螢幕邊的清單。

## AI × 形式化方法：截至 2026 年中的快照

先把限定詞釘死：本節是快照不是預言，每句話都該讀成「截至 2026 年中」。

**能的部分**：語法層面進展驚人。LLM 生成的 TLA+ spec 通過 parser、跑得動 TLC，已經不是新聞；函式層級的規格生成（pre/post-condition）有實際進展；針對「AI 做系統形式建模」的基準測試也開始出現（如 2025 年下半年起的 SysMoBench）。

**不能的部分**：語意是另一回事。有評測讓數十個模型生成數百份 TLA+ spec，**最佳模型的語意正確率仍停在個位數百分比的等級**，且與模型規模沒有明顯關係（截至 2026 年中的報告；本書未開原始論文核對精確數字，故不引）。失敗的形狀比數字更有教育意義：2026 年 5 月 ACM SIGOPS 部落格的一篇實測（Specula 團隊）讓 LLM 為 etcd 的 Raft 實作寫 spec——產出通過語法檢查、TLC 跑得動，但內容其實是照 Raft **論文**默寫的，與 etcd 程式碼的實際行為幾乎無關。它生成了「一份 Raft 的 spec」，不是「**這個系統**的 spec」。

所以主流架構是分工：**LLM 起草、符號工具把關**——生成的 spec 交給 TLC／Apalache 找反例，生成的證明步驟交給 TLAPS 後端逐條查證（LLM 引導證明分解的研究 2025 年底開始出現）。信任錨點還是三十年來那一個：機器查證的核心，不是模型的口才。

對你的含義，一句話：**生成變便宜的世界裡，驗收變值錢。**AI 生成的 spec 會得這本書教過的每一種病——vacuous 的性質、太弱的 invariant、替現實讀心的 guard、原子性切太粗——而且得病時更安靜，因為它語法漂亮、註解流暢。ch08 練的驗收技藝（手構一條應該被擋的 trace、檢查每條防線是否承壓過），從「寫 spec 的人的自律」升級成「審 spec 的人的武器」。這本書教的，恰好是審的那一半。

## 同一筆轉帳，三種眼睛（worked example）

最後一個 worked example，把三種思維放在同一道題上對照。題目刻意小：**兩人轉帳**。Alice 與 Bob 各有 100 元；一支 `transfer(src, dst, amt)`：檢查 src 餘額 ≥ amt，然後扣款、入帳。客戶端逾時重試，送出了兩筆一模一樣的請求：t1、t2 都是 alice → bob、70 元。要守的性質兩條：

- `NoOverdraft` ≜ ∀ a：bal[a] ≥ 0（餘額永不為負）
- `Conserve` ≜ bal[alice] ＋ bal[bob] ＝ 200（總額守恆）

### TLA+ 思維：先問「誰能動、怎麼動」

把 transfer 拆成動作，誠實保留實作裡 check 與 pay 之間的縫（約 15 行，**示意**——省略「檢查不過就放棄」的分支與部分宣告）：

```tla
\* 示意：重點在動作的拆法
CONSTANTS T                 \* 請求集合 {t1, t2}，皆為 alice -> bob、70 元
VARIABLES bal, pc           \* bal：帳戶餘額；pc[t]：請求 t 走到哪一步

Init == /\ bal = [a \in {"alice", "bob"} |-> 100]
        /\ pc  = [t \in T |-> "check"]

Check(t) == /\ pc[t] = "check"
            /\ bal["alice"] >= 70          \* 讀餘額、過檢查
            /\ pc' = [pc EXCEPT ![t] = "pay"]
            /\ UNCHANGED bal

Pay(t) == /\ pc[t] = "pay"                 \* 扣款＋入帳（同一步）
          /\ bal' = [bal EXCEPT !["alice"] = @ - 70, !["bob"] = @ + 70]
          /\ pc' = [pc EXCEPT ![t] = "done"]

Next == \E t \in T : Check(t) \/ Pay(t)

NoOverdraft == \A a \in {"alice", "bob"} : bal[a] >= 0
Conserve    == bal["alice"] + bal["bob"] = 200
```

手推反例，照全書的規矩不跳步：

| 狀態 | 動作 | bal[alice] | bal[bob] | pc[t1] | pc[t2] |
|---|---|---|---|---|---|
| s₀ | — | 100 | 100 | check | check |
| s₁ | Check(t1)：100 ≥ 70 ✓ | 100 | 100 | pay | check |
| s₂ | Check(t2)：100 ≥ 70 ✓ | 100 | 100 | pay | pay |
| s₃ | Pay(t1) | 30 | 170 | done | pay |
| s₄ | Pay(t2) | **−40** | 240 | done | done |

s₄ 打破 `NoOverdraft`——alice 用 100 元的餘額付掉了 140 元。安全的走法只有「一筆走完另一筆才 check」：t2 的 Check 會讀到 30 < 70，放棄。跟 ch01 的 counter 同構：check 寫成獨立動作，縫就在模型裡，TLC 式的窮舉必然撞到它。

再看一眼 `Conserve`：s₀ 到 s₄ 每一步都是「alice 減 70、bob 加 70」或不動，總和永遠 200——**全程綠燈**。一條太弱的性質對 bug 視而不見，而且驗得越勤、心裡越踏實。記住這個畫面，陷阱表會回收它。

### Alloy 思維：先問「什麼形狀的世界合法」

Alloy 的本能是先寫結構（約 15 行，**示意**——Alloy 風格、非可執行全文）：

```text
-- 示意：Alloy 風格
sig Acct { var bal: Int }
one sig Alice, Bob extends Acct {}

pred transfer[src, dst: Acct, amt: Int] {
  amt > 0 and src.bal >= amt            -- 檢查……
  src.bal' = src.bal.minus[amt]         -- ……與扣款、入帳
  dst.bal' = dst.bal.plus[amt]          -- 同在一步發生
  all a: Acct - src - dst | a.bal' = a.bal
}

fact Traces {
  always (some src, dst: Acct, amt: Int | transfer[src, dst, amt])
}

assert NoOverdraft { always all a: Acct | a.bal >= 0 }
check NoOverdraft for 2 but 8 Int, 10 steps    -- small scope：一切有界
```

注意慣用建模的後果：`transfer` 是**原子的一步**——check 與扣款之間沒有縫，於是上面那個 bug **不在這個模型裡**，Analyzer 會誠實地回報「scope 內找不到反例」。綠燈，但綠的是另一個系統。要抓交錯，得像 TLA+ 那樣手動拆出 pc——Alloy 6 寫得出來，但語言沒有在推你這麼做。這題還會先教你另一課：Alloy 的整數是固定位寬的二補數，預設位寬小到連 100 都裝不下，不加大就會無聲溢位繞回負數——你會先被「假反例」教育一頓。

那 Alloy 的主場在哪？把題目換掉：「帳戶有共同持有人、授權代理、凍結規則三組關係，**存不存在**一條授權鏈能把錢轉出凍結帳戶？」——這種結構疊加的問題，Alloy 的 small scope 實例搜尋就是最短路徑，TLA+ 反而要把關係硬翻成函數、自找麻煩。

### PBT 思維：先問「對任意輸入，什麼永遠成立」

PBT 直接轟炸**實作**（約 15 行，**示意**——fast-check 風格）：

```text
// 示意：fast-check 風格
const transferArb = fc.record({
  from: fc.constantFrom("alice", "bob"),
  to:   fc.constantFrom("alice", "bob"),
  amt:  fc.integer({ min: 1, max: 200 }),
});

fc.assert(fc.asyncProperty(
  fc.array(transferArb, { maxLength: 8 }),
  async (ts) => {
    const bank = newBank({ alice: 100, bob: 100 });
    await Promise.all(ts.map((t) => bank.transfer(t)));
    assert(bank.total() === 200);                  // Conserve
    assert(bank.balances().every((b) => b >= 0));  // NoOverdraft
  },
));
```

它做到一件 TLA+ 與 Alloy 都做不到的事：**跑的是你真正出貨的 transfer**，連你沒建模的部分（ORM 的怪癖、金額捨入、序列化）一起驗；amt 的邊界（0、負數、超大值）它抽得比你勤，找到反例還會 shrink 到最小。但兩個洞：其一，`Promise.all` 的交錯由 event loop 決定——調度仍然不是輸入，ch01 的困境原樣健在，這條 property 可能綠一整年；其二，性質太弱的風險照舊（只斷言 `Conserve` 的話，這支有 bug 的實作永遠全綠）。第一個洞的補法在光譜的下一格：把調度收編成輸入——那就是 DST（紙上推演題 3 會動手補）。

### 對照表

| | TLA+ 思維 | Alloy 思維 | PBT 思維 |
|---|---|---|---|
| 先問什麼 | 誰能動、怎麼動（轉移） | 什麼形狀的世界合法（關係） | 對任意輸入什麼永遠成立（性質） |
| 驗的對象 | 設計 | 設計（結構面） | 實作本身 |
| 量詞 | scope 內全稱（窮舉路徑） | scope 內全稱（窮舉結構） | 抽樣 |
| 這題抓到什麼 | Check 與 Pay 之間的插隊，s₄ 反例 | 位寬溢位這類值域怪事；交錯預設不在鏡頭裡 | 實作層 bug、輸入邊界；交錯靠運氣 |
| 這題漏什麼 | 實作是否照 spec；參數只能取小 | 慣用建模把 race 原子化掉 | 調度不可控；抽不到的角落 |
| 產出 | 反例 trace | 反例實例（一張結構圖） | 最小失敗輸入（shrinking） |

三雙眼睛沒有誰對誰錯——它們在問三個不同的問題。方法地圖的正確用法從來不是選一個信仰，是看清手上的問題該先問哪一個。

## 如果哪天想動手

先把話說明白：這本書不碰鍵盤是刻意的——紙上推演逼你把每一步想穿，而不是讓工具替你想。但十八章下來，「想親眼看 TLC 打出反例」是健康的癮。從你的現狀出發，路徑長這樣：

## 陷阱與防禦

方法地圖層級的故障模式——這些坑不在某個工具裡，在「選方法」這件事本身：

| 陷阱 | 它怎麼給你假安全感 | 防禦 |
|---|---|---|
| 工具收藏家 | 地圖看熟了、書籤收齊了，卻從沒對自己的系統問過一條精確的問題——「知道」的滿足感冒充了「用過」 | 地圖要反著用：從手上的問題出發選格子，不是從工具出發找題目。檢驗法：你說得出自己系統裡一個「該用 Alloy 的問題」嗎？說不出來，地圖還沒長在你身上 |
| 銀彈遷移 | 「改用 P／Quint／DST 就安全了」——可是每個方法驗的對象不同，換工具不會把你沒問過的問題自動問出來 | 先寫性質清單，再選工具。性質清單是不變的資產，工具是可換的零件 |
| PBT 性質套套邏輯 | property 用實作同一套邏輯算期望值（自己對自己），或斷言太弱——`Conserve` 對 overdraft 全盲，幾百萬筆輸入全綠 | mutation 思維：故意把實作弄壞一處，property 必須變紅；不紅，這條性質沒在承壓。這是 ch08「防線承壓測試」的 PBT 版 |
| generator 抽不到的角落 | 輸入生成器分布偏掉：永遠抽不到並發衝突、空集合、邊界金額——「任意輸入」其實是「常見輸入」 | 審 generator 如審 guard：列出你最怕的五種輸入，確認生成器真的會生出它們，極端 case 用權重拉高 |
| DST 的決定性破口 | 模擬器外偷用了真時鐘、真亂數、未 shim 的 I/O——同種子不再同結果，重放安靜地失效；你以為在做 DST，其實在做普通 fuzzing | 把「同種子 ⇒ 同 trace」本身寫成 CI 檢查的性質；review 把 `Date.now()`、裸亂數、未注入的 I/O 當 lint 錯誤抓 |
| AI 生成的 spec 直接收貨 | 語法過、TLC 綠、註解流暢——但它可能是照論文默寫的通用協議，不是你的系統；vacuous 性質與讀心 guard 一樣不會自己報錯 | ch08 的驗收技全套用上：手構一條它應該擋下的 trace 餵進去；逐條性質問「承壓過嗎」；逐個故障動作審 guard 的視力。生成省下的時間，全數投進驗收，帳才是平的 |

## 紙上推演

**第 1 題 [15 分鐘] ★★ —— 三個場景，選方法**

為下列場景各選一個主力方法、說出判準、給一個第二選擇：

(a) 權限模型重構：報表系統的存取規則疊了三層（org 繼承、群組、個人覆寫），重構前要確認「不存在一個使用者能看到別人的薪資報表」。
(b) 跨服務退款 saga：訂單、金流、庫存三個服務，事件驅動、有補償流程，要保證「crash 與重送交錯下，不會退兩次、也不會卡在退一半」。
(c) 計費函式庫重寫：純函式（輸入帳單明細、輸出含稅總額），跑了四年的舊版要換新版，兩版將並行三個月。

### 推演解答

(a) **Alloy**。狀態的形狀比演化複雜：三層規則是關係的疊加，問題本身就是 ∃ 量詞——「存不存在一個 user、一條授權鏈」。small scope 夠用：這種壞結構幾乎都能在 3 user × 3 群組的小宇宙現形。第二選擇：寫腳本對小宇宙暴力窮舉——可行，但你其實在手刻 Alloy（ch01 那個 AWS S3 團隊差點手刻模型檢查器的故事，換了個主角）。TLA+ 在這題不順手：沒有「演化」可言，硬寫等於把關係翻成函數自找麻煩。

(b) **TLA+（或 P）**。故障 × 併發交錯、錢、不可逆——ch17 的判準全中。先把兩條需求分類：「不會退兩次」是 safety（跟 `NoDoublePay` 同一個形狀）；「不會卡在退一半」是 **liveness**——要 fairness 才有意義（ch07），光是在白板上把這兩條分開、各自寫清楚依賴什麼，就已經值回票價。若團隊想要可執行模型、直接對事件流注入故障，P 是工程化選項。第二選擇：DST——前提是三個服務都在你家、I/O 可以 shim。

(c) **PBT ＋ oracle 對照**。純函式、有現成 oracle（舊版）、輸入空間大而調度空間為零——ch01 判準表「測試就夠」那欄全中，形式化方法一條都不中。主力 property 一條：∀ 帳單 x：new(x) = old(x)；輔以守恆與單調性質。shrinking 會把不一致縮到最小帳單。第二選擇：邊界 example test 補刀（0 元、負項、捨入邊界）。這題的教訓最重要：**方法地圖的第一個用途，是告訴你什麼時候不需要它。**

**第 2 題 [20 分鐘] ★★ —— 幫 design doc 補三條 invariant**

虛構 design doc（優惠券服務）節錄：「行銷活動發放優惠券。每個使用者每檔活動限領一張；全檔活動預算上限 10,000 張；使用者領到的券在活動期間內隨時可以核銷。發放走 API，核銷走另一個服務，兩邊都有重試。」

用三段式（述語／機制／破壞假設）補三條 invariant；其中一條其實根本不是 invariant——指出是哪條、它是什麼、需要什麼才成立。

### 推演解答

第一條——限領一張。**述語**：∀ u, c：|issued(u, c)| ≤ 1。**機制**：(userId, campaignId) 複合 unique constraint，發放檢查與寫入同一交易。**破壞假設**：客服補發腳本不走這條路；或「換綁手機重領」需求讓 userId 不再是穩定鍵。

第二條——預算上限。**述語**：∀ c：|issued(c)| ≤ 10000。**機制**：原子遞減的額度計數器，guard ≥ 0 與發券同一交易。**破壞假設**：計數器與發券落在兩個服務、兩筆交易——check-then-act 的縫原樣回歸（ch01）；另一個方向也要寫：重試在「扣了額度、寫券失敗」後不歸還額度，會反向破壞（少發）——同一條 invariant 的兩側都得設防。

第三條——隨時可核銷。動筆寫述語就會發現不對勁：「∀ 已發放的券：核銷服務**終會**認得它」——這不是 invariant，是 **liveness**（◇ 的形狀）。它不能被任何有限前綴反駁（ch07 的 Alpern–Schneider 直覺），任何時刻你都可以說「再等等就同步到了」。要它成立，靠的不是 unique constraint，而是「發放 → 核銷服務的資料同步終會完成」這種 fairness 級的營運假設（同步管道健康、補償任務真的會跑、會重試到成功）。design doc 最常見的病就是把 liveness 寫得彷彿一條 DB constraint 就能保證——認出它、把「靠什麼終會發生」寫清楚，就是 ch07 在你日常工作裡的兌現。

**第 3 題 [15 分鐘] ★★ —— 給 PBT 那雙眼睛補洞**

worked example 的 PBT property，在多數執行環境連跑一萬次也抓不到 alice = −40。

(a) 說出兩個原因。
(b) 不引入任何新工具、只改測試程式，給一個讓壞調度可被抽到的辦法。
(c) 這個辦法把測試往光譜的哪一格推了半步？差的那半步是什麼？

### 推演解答

(a) 其一：**調度不可控**。`Promise.all` 把交錯交給 event loop；若 transfer 實作在 check 與 debit 之間沒有 await 點，兩筆請求在單執行緒上根本不會交錯——壞調度的機率不是低，是**零**；就算有 await 點，交錯分布也壓在無害區（ch01 的「數學上 90% 壞、物理上 99.99% 抽到好的」）。其二：**性質可能太弱**——若只斷言 `Conserve`，這支有 bug 的實作永遠綠燈：每筆轉帳一減一加，總和不動如山。

(b) 把非決定性收編成輸入：給測試版的 DB client 加一個 schedule 參數，依序決定哪個請求的查詢先返回（在 await 縫隙注入可控的讓步），然後讓 fc **連 schedule 一起生成**——property 從「∀ 轉帳序列」升級成「∀ 轉帳序列 × ∀ 調度」。失敗時 shrinking 連調度一起縮到最小重現。

(c) 往 **DST** 推了半步：調度從環境變成了輸入，同種子可重放。差的那半步是**範圍**——你只 shim 了一個 DB client；FoundationDB 路線 shim 的是整個世界（網路、磁碟、時鐘、行程生死）。但方向完全相同，而「把非決定性變成輸入」這一手，就是整條輕量光譜的靈魂。
