## 從你已知的出發

你每天都在寫邏輯，只是用的是程式語法：

- **guard clause 就是 ⇒**。`if (!order.isCommitted) return;` 之後的每一行程式碼，隱含的正確性主張是「isCommitted ⇒ 這段處理是安全的」。被 guard 擋掉的路徑你從不覺得需要負責——這個「前件不成立就不關我的事」的直覺，正是邏輯蘊涵的定義。
- **`[].every(...)` 回 `true` 就是 vacuous truth**。Node.js 裡 `[].every(x => x.paid)` 永遠回 `true`：空陣列上的「全部都……」無條件成立。你大概踩過一次：上游查詢條件寫錯、回了空陣列，下游「全部驗證通過」的綠燈亮著，資料卻根本沒被檢查過。
- **SQL 的 `NOT EXISTS` 改寫就是量詞推移**。「每張訂單都有對帳記錄」直接寫 SQL 很難下手；你實際寫的是「不存在沒有對帳記錄的訂單」——`WHERE NOT EXISTS (...)`。∀ 與 ¬∃¬ 的互換，你的手指早就會了，本章只是給它名字和規則。
- **code review 的爭論常常是量詞之爭**。「所有 retry 都會被 dedup 擋住」對上「存在一條 retry 路徑繞過 dedup」——前者是 ∀，後者是它的否定 ∃¬。吵直覺沒有結果；把兩句話寫成式子，戰場立刻清楚：找到一條繞過的路徑就分出勝負，找不到就得說明為什麼找不到。

所以本章的目標不是教你新東西，而是把這些散落的直覺接上同一套語法，並且練到三件事變成反射動作：把白話翻成邏輯式、把邏輯式翻回白話、否定一條規則得到「bug 的形狀」。

## 命題：能判定真假的句子

命題（proposition）：一個非真即假的宣告句。看四個例子就知道邊界在哪：

- 「m1 已入帳」——在一個給定的系統狀態下非真即假：**是**命題。
- 「佇列深度超過 1000 就該加 consumer」——這是規範性意見，不是事實宣告：不是命題。
- 「這段程式碼很乾淨」——沒有公認判準：不是命題。
- 「x > 3」——x 沒給值之前無法判定：還不是命題，是**述語**（本章後半的主角）。

第一個例子值得多看一眼：「m1 已入帳」的真假取決於你問的是哪個狀態。在 ch02 的語言裡，狀態是變數的一組賦值；本章前半段先把狀態凍結住——想像你對著某一瞬間的系統快照說話，一句話在這個快照上非真即假，它就是命題。同一句話在不同狀態下真值不同，這正是 invariant 在處理的事（每個可達狀態上都為真）；但那是把本章的工具套到 ch02 的模型上之後的事，工具本身先不用管狀態。

順帶聲明本章刻意不教的東西：自然演繹、推理規則表這類形式證明系統，本書全程不碰。我們的標準是「工程師的嚴謹」——每一步都能口頭說出理由、能用真值表或具體例子驗證，但不玩符號系統的機械推導。之後需要管理證明結構時（ch05、ch14），用的是層級式寫法，不是推理規則的名字。

## 五個連接詞與真值表

命題是原子，連接詞把原子接成分子。全書內文一律用 Unicode 符號（TLA+ 原始碼裡的 ASCII 寫法，對照表見 ch06）：

| 符號 | 讀法 | 白話 |
|---|---|---|
| ∧ | and | 兩者都成立 |
| ∨ | or | 至少一個成立（可以同時） |
| ¬ | not | 不成立 |
| ⇒ | implies | 若…則…（小心，下一節整節都在講它） |
| ≡ | 等價 | 兩邊真值永遠相同 |

五個連接詞的完整定義就是一張真值表（T＝真、F＝假）：

| p | q | ¬p | p ∧ q | p ∨ q | p ⇒ q | p ≡ q |
|---|---|---|---|---|---|---|
| T | T | F | T | T | T | T |
| T | F | F | F | T | F | F |
| F | T | T | F | T | T | F |
| F | F | T | F | F | T | T |

三個提醒：

1. ∨ 是 inclusive or：兩邊同時成立也算成立。白話的「或」常常暗示二選一；需求若真的是「恰好一個」，要明寫成 (p ∨ q) ∧ ¬(p ∧ q)。
2. ≡ 是「真值表完全相同」。要驗證兩條式子等價，列出所有真值組合、逐列比對即可——純機械，不需要靈感。n 個命題變數就是 2ⁿ 列，這也是你第一次摸到狀態爆炸的邊（見 ch01 的組合爆炸，ch09 會正面處理）。
3. ∧ 與 ∨ 各自滿足交換律、結合律，彼此有分配律，還有 De Morgan 律 ¬(p ∧ q) ≡ ¬p ∨ ¬q 與 ¬(p ∨ q) ≡ ¬p ∧ ¬q。全部可用真值表驗證，其中兩條留在「紙上推演」給你親手做。

## ⇒：最重要、也最會騙人的連接詞

把 ⇒ 的兩列單獨拿出來看：

| p | q | p ⇒ q | 怎麼讀 |
|---|---|---|---|
| T | T | T | 前件成立、後件做到：履約 |
| T | F | **F** | 前件成立、後件沒做到：**唯一的違約世界** |
| F | T | T | 前件不成立：契約自動履行 |
| F | F | T | 前件不成立：契約自動履行 |

「若 p 則 q」的日常語感帶著因果和時間：先有 p、後有 q、p 導致 q。邏輯的 ⇒ 三個都不是。p ⇒ q 只是一個關於真值組合的宣告：「不會出現 p 真而 q 假」。跟因果無關、跟先後無關；唯一讓它為假的世界是第二列。

三種讀法，挑你順手的：

1. **契約讀法**：p ⇒ q 是一紙契約，只有「p 成立而 q 沒做到」算違約；p 沒成立，契約自動算履行。
2. **guard 讀法**：q 只在 p 的範圍內被檢查；p 為假的狀態直接放行——正是 `if (!p) return;` 的語意。
3. **排除讀法**：p ⇒ q ≡ ¬(p ∧ ¬q)：我只排除一種世界，其他都接受。

### vacuous truth：前件為假，整句為真

第三、四列值得單獨開一節，因為它是這本書後半段一個真實事故模式的根。

當 p 恆假——或在你實際碰到的所有情況下為假——p ⇒ q 恆真，**不管 q 是什麼**。「所有獨角獸都會寫 TLA+」是真的，因為沒有獨角獸。這叫空洞地為真（vacuously true）。

工程現場的屍體展示：

- `[].every(x => x.paid)` 回 `true`：空集合上的「全部都」無條件成立。
- 監控主張「每個 5xx 都觸發了 alert」，在今天沒有任何 5xx 的系統上為真。綠燈。但它完全沒告訴你 alert pipeline 是好的——可能 alert 根本沒接上，你要等第一個 5xx 才會知道。
- 測試斷言「所有被處理的訊息都恰好入帳一次」，在 consumer 掛掉、什麼都沒處理的環境裡：全數通過。
- 寫 spec 時，一條 invariant 的前件用了永遠不可能滿足的條件（打錯字、或對系統的誤解），檢查器回報 invariant 成立——它當然成立，它什麼都沒檢查。

關鍵的反直覺點一句話：**「P 為真」與「P 檢查過東西」是兩回事**。空洞為真的性質給你的安全感是純粹的假貨，而且它不會自己暴露——空轉的綠燈跟真正驗證過的綠燈長得一模一樣，你必須主動拆穿。

防禦動作，現在就養成：寫下任何 p ⇒ q 形式的主張之後，**為前件找一個 witness**——舉出一個具體的、真實會發生的情況讓 p 為真。舉不出來，這條主張就在空轉，要嘛改寫、要嘛承認它沒驗證任何東西。本書後面凡是寫 invariant，這個動作都應該是反射。

把話說在前頭：**這個陷阱在 ch14 會回來咬人**。歸納證明裡，歸納假設是整個歸納步的前件；如果它強到沒有任何狀態能滿足，歸納步會「證明成功」——空洞地。你會看到一個每一步都通過、結論卻毫無價值的完整案例。先把警報器裝好。

### 逆否命題，以及它的兩個冒牌貨

從真值表可以驗證（或用排除讀法直接看出）：

- p ⇒ q **≡** ¬q ⇒ ¬p（逆否，contrapositive）——同一條主張的兩種說法。
- p ⇒ q **不等價於** q ⇒ p（converse）。
- p ⇒ q **不等價於** ¬p ⇒ ¬q（inverse）。

工程例：「部署成功 ⇒ 健康檢查通過」等價於「健康檢查沒過 ⇒ 部署沒成功」。但「健康檢查通過 ⇒ 部署成功」是另一條主張——健康檢查可能太弱，這條可以是假的。你每天的排查推理其實常走逆否：alert 響了（¬q），所以系統不在正常路徑上（¬p）。會混淆的是冒牌貨：「沒有 alert，所以沒問題」走的是 inverse，邏輯上不成立——alert 沒接上時就是沒有 alert。

## 述語與量詞：把「哪些東西」說清楚

述語（predicate）：帶洞的命題。P(x) 在 x 填入具體值之前沒有真假；填進去就變成命題。工程對照：回傳 boolean 的純函式——`isPaid(msgId)` 本身不是真也不是假，`isPaid('m1')` 在給定狀態下才是。

量詞把「填洞」從「給一個值」升級成「掃一整袋值」：

- ∀ m ∈ Msgs : P(m)——袋子裡**每一個** m 都滿足 P。
- ∃ m ∈ Msgs : P(m)——袋子裡**至少一個** m 滿足 P。

「袋子」（像 Msgs 這種範圍）的正式名稱是集合，細節是 ch04 的事；本章只需要「一袋可以逐一檢視的東西」這個直覺。另外，本書一律寫**有界量詞**——∀ x ∈ S 明寫範圍。無界的「對宇宙中一切 x」在 TLA+ 的世界幾乎不出現，而且容易讓你忘記問「範圍到底是誰」——後面會看到，範圍不明是翻譯錯誤的頭號來源。

袋子有限時，量詞可以攤開：

∀ m ∈ {m1, m2} : Paid(m) ≡ Paid(m1) ∧ Paid(m2)
∃ m ∈ {m1, m2} : Paid(m) ≡ Paid(m1) ∨ Paid(m2)

這個攤開讓量詞的性質都能從連接詞繼承，包括下一節的 De Morgan。它也順手解釋了空袋子的行為：範圍為空時，∀ 攤開成「零個條件的 ∧」——恆真（`[].every`）；∃ 攤開成「零個條件的 ∨」——恆假（`[].some`）。JavaScript 在這件事上跟邏輯學家站同一邊。

### bound 與 free：誰的變數誰負責

在 ∀ m ∈ Msgs : PayCount(m) ≤ 1 裡，m 是**約束變數**（bound variable）：它是量詞自己的迴圈變數，對外不可見，整條式子的真假不取決於「外面某個叫 m 的東西」。把 PayCount(m) ≤ 1 單獨抽出來，m 就是**自由變數**（free variable）：式子的真假取決於誰來填 m。

工程對照很直接：bound variable ≈ `msgs.every(m => ...)` 的 callback 參數，scope 只在 callback 裡，外面沒有 m 這個東西、改名也不影響語意；free variable ≈ 從外層閉包抓進來的變數，值由外面決定。

判讀規則：一條式子是不是命題，看它有沒有 free variable——全部 bound（或根本沒變數）就是命題；有 free 的是述語。這個區分在 ch02 的圖景下有個漂亮的應用：invariant 裡的狀態變數（ledger 之類）嚴格說是 free 的，由「目前狀態」這個外層 scope 填值——所以同一條 invariant 在不同狀態下有不同真值，而「invariant 成立」的完整主張是「在每個可達狀態填值後都為真」。

### 量詞順序：∀∃ 與 ∃∀ 差一個世界

- ∀ m ∈ Msgs : ∃ c ∈ Consumers : Handles(c, m)——每則訊息都有 consumer 處理它；**誰處理可以各自不同**。
- ∃ c ∈ Consumers : ∀ m ∈ Msgs : Handles(c, m)——存在**一個** consumer，獨自處理了所有訊息。

第二條強得多（它成立時第一條必成立，反之不然）。交換 ∀∃ 順序是翻譯白話時最常見的錯誤，因為中文天生兩可：「有 consumer 處理每則訊息」——你讀讀看，兩種解釋都通。防禦：碰到「有」和「每」同時出現的句子，把兩種順序都寫出來、各自翻回白話，問哪一句才是需求的本意。同類量詞相鄰可以交換（∀∀、∃∃），異類不行——這是肌肉記憶等級的規則。

## 否定推移：機械地求出 bug 的形狀

先說為什麼否定值得單獨一節。在這本書裡，**找反例＝證明否定式可滿足**。你主張「每個可達狀態都滿足 Inv」；模型檢查器（ch09）做的事，是在可達狀態裡搜尋讓 ¬Inv 成立的狀態。¬Inv 長什麼樣，決定你知不知道自己在找什麼——而 ¬Inv 不需要靈感，它是機械推導出來的。

推移規則一張表（每條都能用真值表或有限攤開驗證；量詞那兩條就是 De Morgan 的量詞版）：

| 原式 | 等價式 | 名字 |
|---|---|---|
| ¬¬p | p | 雙重否定 |
| ¬(p ∧ q) | ¬p ∨ ¬q | De Morgan |
| ¬(p ∨ q) | ¬p ∧ ¬q | De Morgan |
| ¬(p ⇒ q) | p ∧ ¬q | **反例的形狀** |
| ¬(∀ x ∈ S : P(x)) | ∃ x ∈ S : ¬P(x) | 量詞推移 |
| ¬(∃ x ∈ S : P(x)) | ∀ x ∈ S : ¬P(x) | 量詞推移 |

兩條值得駐足。

**¬(p ⇒ q) ≡ p ∧ ¬q**。蘊涵的否定不是另一個蘊涵，是一個 ∧：「前件成立**而且**後件失敗」。反例必須同時呈現兩件事——這正是一份合格 bug report 的格式：重現條件（p 成立）加錯誤現象（q 沒發生）。順手拆穿常見錯誤：把 ¬(p ⇒ q) 寫成 p ⇒ ¬q 或 ¬p ⇒ ¬q 都是錯的——前者在 p 為假時空洞為真，後者根本是另一條主張，真值表立判。

**¬∀ ≡ ∃¬**。「並非全部都好」＝「存在一個壞的」。推翻一條全稱主張只需要一個證人（witness）。這就是為什麼反例 trace 殺傷力這麼大（ch09）：它是一個具體的證人，不容辯駁；而你要守住 ∀，就得對每一個成員負責。

推移演算法本身純機械：把 ¬ 從最外層往內推，每經過一層就查一次表——∀ 翻成 ∃、∃ 翻成 ∀、∧ 翻成 ∨、∨ 翻成 ∧、⇒ 拆成 ∧，直到所有 ¬ 都貼在原子述語上（原子層級的否定通常能再化簡，例如 ¬(n ≤ 1) 在自然數上就是 n ≥ 2）。推完之後做一次抽查：挑一組具體值，代入原式和結果式，真值必須相反——這是給推導上的保險。

## 把白話規則翻成邏輯式：方法論與 worked example

累積到這裡，工具齊了。翻譯的步驟固定五步：

1. **先找量詞範圍**：這句話在對「哪一袋東西」說話？袋子是誰、有限還是會長大？
2. **名詞變述語**：每個性質一個帶參數的 boolean。
3. **接上連接詞**：注意中文的「的」常常藏著 ⇒——「已發放的獎勵」＝「若它已發放」。
4. **否定它**：機械推移，讀出 bug 的形狀。
5. **兩頭 sanity check**：原式在無聊狀態（初始、空袋）下真值合理嗎？有沒有空洞？否定式描述的是你真正想抓的 bug 嗎？

為什麼第一步是量詞範圍：範圍錯，後面全錯。「所有訊息」是這一季的、目前 in-flight 的、還是含未來重送的副本？範圍變，真值就變，而白話從來不明說。

### Worked example：NoDoublePay 的形狀

設定沿用 ch02 的結算系統 v0：訊息 Msgs = {m1, m2}，consumer 從佇列取訊息入帳，ledger 記錄哪些 msgId 已入帳。要翻譯的業務規則，白話版：

> 已發放的獎勵不會再發。

一步一步來，不跳步。

**步驟 1：量詞範圍。**「已發放的獎勵」在對哪些東西宣告？這個系統裡獎勵跟著訊息走，每則訊息對應一筆入帳，所以範圍是 Msgs。這句話對**每一則**訊息都成立（沒說哪則可以例外），所以最外層是 ∀ m ∈ Msgs。

**步驟 2：名詞變述語。**「已發放」「不會再發」合起來談的其實是**次數**：發過一次之後，次數不能變成二。引入述語的原料 PayCount(m)：到目前為止 m 被入帳的累計次數。注意，這是個觀察值——ch02 的 ledger 只記「有沒有入帳」（boolean），沒記次數。在紙上我們可以自由談 PayCount，數學不在乎它有沒有被實作；但若要讓機器檢查它，狀態就得記得它。「invariant 需要的資訊必須存在於狀態裡」這個觀察先放在口袋，ch14 會把它升級成一門技藝（輔助述語）。

**步驟 3：接連接詞。**「不會再發」＝次數至多 1。定義：

NoDoublePay：∀ m ∈ Msgs : PayCount(m) ≤ 1

停一下，比對另一種更貼字面的翻法。「已發放的」聽起來像個前件，所以也有人會寫：

候選 B：∀ m ∈ Msgs : Paid(m) ⇒ PayCount(m) = 1，其中 Paid(m) 定義為 PayCount(m) ≥ 1

攤開檢查兩者是否等價：對任一 m，若 PayCount(m) = 0，候選 B 的前件為假、整條空洞為真，而 0 ≤ 1 也為真——兩邊一致；若 PayCount(m) ≥ 1，候選 B 要求恰為 1，NoDoublePay 要求至多 1，配上「至少 1」也是恰為 1——兩邊一致。所以等價。但第一種寫法更好：它不需要讀者看穿一層 vacuous truth 才能確認初始狀態合法。**能不用 ⇒ 就不用**，是寫 invariant 的品味——每多一個 ⇒，就多一個需要驗 witness 的點。

**步驟 4：否定。**機械推移：

```text
  ¬NoDoublePay
≡ ¬(∀ m ∈ Msgs : PayCount(m) ≤ 1)
≡ ∃ m ∈ Msgs : ¬(PayCount(m) ≤ 1)      -- ¬∀ 翻成 ∃¬
≡ ∃ m ∈ Msgs : PayCount(m) ≥ 2         -- 次數是自然數
```

讀出來：「**存在一則訊息被入帳兩次以上**」。這就是 double pay 這個 bug 的形狀——一句話、零歧義、可以拿著去搜尋。

**步驟 5：兩頭 sanity check。**原式在初始狀態（什麼都沒入帳，PayCount 全為 0）：0 ≤ 1 為真，而且不是空洞真——它真實檢查了每則訊息的計數。否定式描述的情境正是結算 pipeline 最怕的事故。兩頭都過，翻譯收工。

最後示範「找反例＝證明否定式可滿足」是什麼意思。手追一條 at-least-once 世界裡的路徑（這個版本的 consumer 不查 dedup）：

| 狀態 | 發生的事 | PayCount(m1) | PayCount(m2) | NoDoublePay |
|---|---|---|---|---|
| s0 | 初始 | 0 | 0 | 真 |
| s1 | consumer 取 m1、入帳 | 1 | 0 | 真 |
| s2 | m1 重送（at-least-once）、consumer 未查重、再入帳 | 2 | 0 | **假** |

在 s2 取證人 m ≔ m1：PayCount(m1) = 2 ≥ 2，∃ 式成立，¬NoDoublePay 為真。這條從 s0 走到 s2 的路徑，就是「否定式在某個可達狀態上可滿足」的構造性證明——也就是一條反例 trace。ch09 你會看到 TLC 做的正是這件事，而且保證交出最短的那條。

補一句誠實話：白話裡的「**再**發」其實藏著時間——「先發了一次、之後又發」。本章的工具只能對單一狀態說話，所以我們把歷史壓進 PayCount 這個計數裡，抓到的是結果（至多一次）。要直接對「步」說話（「這一步若 m 已發放，就不會再入帳 m」）得等 ch06 的動作邏輯；要對「永遠、終究」說話得等 ch07 的時序邏輯。知道自己手上工具的邊界在哪，也是精確的一部分。

## 陷阱與防禦

本章每個工具都有它給假安全感的方式。一張表收齊，左欄是事故，右欄是你的自救動作：

| 陷阱 | 它怎麼給你假安全感 | 怎麼自我察覺 |
|---|---|---|
| vacuous truth | 前件不可滿足（打錯字、條件過強、情境還沒發生過），p ⇒ q 恆真；invariant 綠燈，但什麼都沒檢查 | 為前件找 witness：舉出一個真實會發生的情況讓前件為真，舉不出來就改寫。這個警報器 ch14 會再用一次 |
| 把 ⇒ 讀成因果或時序 | 「p ⇒ q」被讀成「p 導致 q」或「p 之後 q」，把需要時序邏輯（見 ch07）的主張誤寫成單狀態述語，自以為涵蓋了 | 自問：這條主張談的是「同一瞬間的真值關係」還是「先後發生」？句子裡出現「之後」「終究」「再也不」→ 本章工具寫不了，標記下來等 ch07 |
| 量詞順序錯置 | ∃c∀m 與 ∀m∃c 差一個世界；中文白話兩可，翻錯的那條往往更弱、更容易「驗證通過」 | 兩種順序都寫出來、各自翻回白話，對照需求問哪句才是本意 |
| 否定只推一半 | ¬(p ∧ q) 寫成 ¬p ∧ ¬q、推 ¬∀ 忘了翻量詞——得到錯的 bug 形狀，去找一個不存在的 bug，或漏掉真的 | 全程查表推移，不心算；推完拿一組具體值代回原式與結果式，真值必須相反 |
| 量詞範圍含混 | 「所有訊息」沒說是哪一袋：含不含重送副本？含不含未來才進來的？範圍縮小，∀ 變容易成立 | 翻譯第一步先寫下範圍集合的名字與內容物；範圍會隨時間長大的，特別標記（這正是 ch02 的 behavior 在管的事） |
| 空袋子邊界 | ∀ 在空範圍上恆真：「所有 prepared 的交易都 commit 了」在沒有 prepared 交易時成立 | 對每條 ∀ 主張問一次：「範圍為空時它為真，是我要的嗎？」通常是，但要「知情地是」 |

共同的主旋律：**綠燈有兩種**——驗證過的，和空轉的。邏輯不會替你分辨，分辨是你的工作。

## 紙上推演

**題 1 [15 分鐘] ★ 真值表驗證等價式**

用完整真值表驗證以下兩條等價，並各用一句白話說明它們為什麼理當如此：

(a) p ⇒ q ≡ ¬p ∨ q
(b) ¬(p ∧ q) ≡ ¬p ∨ ¬q

### 推演解答

(a) 四列攤開：

| p | q | ¬p | ¬p ∨ q | p ⇒ q |
|---|---|---|---|---|
| T | T | F | T | T |
| T | F | F | F | F |
| F | T | T | T | T |
| F | F | T | T | T |

第四、五欄逐列相同，等價成立。白話：「若 p 則 q」唯一排除的世界是「p 真 q 假」；「要嘛 p 不成立、要嘛 q 成立」排除的也正是那個世界。工程版：`if (p && !q) throw` 跟 `assert(!p || q)` 是同一條檢查。這條等價也是推導的潤滑劑——遇到 ⇒ 推不動時，先把它換成 ¬p ∨ q，世界會變簡單。

(b) 同樣四列：

| p | q | p ∧ q | ¬(p ∧ q) | ¬p ∨ ¬q |
|---|---|---|---|---|
| T | T | T | F | F |
| T | F | F | T | T |
| F | T | F | T | T |
| F | F | F | T | T |

第四、五欄相同，等價成立。白話：「並非兩者都成立」＝「至少一個不成立」。常見錯誤是把右邊寫成 ¬p ∧ ¬q（「兩個都不成立」）——那是 ¬(p ∨ q) 的等價式，第二、三列就會被抓出來。

**題 2 [20 分鐘] ★★ 中⇄邏輯互翻三題**

(a) 中→邏輯：「每筆已 commit 的訂單都有 ledger 記錄。」（量詞範圍自己定，要明寫。）
(b) 中→邏輯：「要嘛至少有一個 consumer 是空閒的，要嘛佇列是空的。」
(c) 邏輯→中：∀ c ∈ Consumers : Busy(c) ⇒ (∃ m ∈ Msgs : AssignedTo(m, c))

### 推演解答

(a) 範圍：Orders ＝ 本結算期的所有訂單。述語：Committed(o)、HasLedger(o)。

∀ o ∈ Orders : Committed(o) ⇒ HasLedger(o)

「已 commit 的」是中文的「的」藏起來的前件。witness check：committed 的訂單真實存在，所以不空洞。順手否定一次：∃ o ∈ Orders : Committed(o) ∧ ¬HasLedger(o)——對帳作業每天在找的就是這個 ∃ 的證人；SQL 寫出來正是 `NOT EXISTS` 的形狀：查詢回空 ⇔ 原式成立。

(b) 述語：Idle(c)、QueueEmpty。

(∃ c ∈ Consumers : Idle(c)) ∨ QueueEmpty

兩個討論點。第一，白話的「要嘛…要嘛…」這裡是 inclusive or：佇列空、同時又有人閒著，不算違規；若需求真是恰好一邊，必須明寫排他。第二，否定它：(∀ c ∈ Consumers : ¬Idle(c)) ∧ ¬QueueEmpty——「所有 consumer 都在忙，而且佇列非空」，這是 backpressure 告警的形狀。一條健康主張的否定式往往就是你該設的 alert 條件。

(c) 白話：「每個忙碌中的 consumer 都至少被指派了一則訊息。」重點在 ∃ 住在 ∀ 裡面：每個 consumer 各有各的訊息，不要求同一則。常見錯翻是「有一則訊息被指派給所有忙碌的 consumer」——那是 ∃ m ∀ c，順序翻錯差一個世界。另外注意這條主張容忍空閒的 consumer（前件為假、放行），它防的是「殭屍忙碌」：標記為忙、手上卻沒有任何工作。

**題 3 [15 分鐘] ★★ 否定推移（巢狀量詞）**

求下式的否定範式（把 ¬ 全部推到原子述語上），把結果翻成白話，並說出它是哪種 bug 的形狀：

∀ m ∈ Msgs : Delivered(m) ⇒ (∃ c ∈ Consumers : Processed(c, m))

### 推演解答

逐層推移，每步只動一層：

```text
  ¬(∀ m ∈ Msgs : Delivered(m) ⇒ (∃ c ∈ Consumers : Processed(c, m)))
≡ ∃ m ∈ Msgs : ¬(Delivered(m) ⇒ (∃ c ∈ Consumers : Processed(c, m)))
                                        -- ¬∀ 翻成 ∃¬
≡ ∃ m ∈ Msgs : Delivered(m) ∧ ¬(∃ c ∈ Consumers : Processed(c, m))
                                        -- ¬(p ⇒ q) ≡ p ∧ ¬q
≡ ∃ m ∈ Msgs : Delivered(m) ∧ (∀ c ∈ Consumers : ¬Processed(c, m))
                                        -- ¬∃ 翻成 ∀¬
```

白話：「存在一則已投遞的訊息，**所有** consumer 都沒處理它。」——訊息黑洞（掉單）的形狀。三層結構各自翻面：∀ 變 ∃、⇒ 變 ∧、∃ 變 ∀。最容易犯的錯在第二步：把 ⇒ 的否定寫成 Delivered(m) ⇒ ¬(∃ c : …)——這句話會空洞地放過所有「未投遞」的訊息沒錯，但它根本不是否定式：拿「m 未投遞」的情況代入抽查，原式為真、它也為真，真值沒有相反，立刻穿幫。

**題 4 [10 分鐘] ★★ 拆穿空洞的 invariant**

你接手一份 spec，裡面有一條三個月來從沒亮過紅燈的 invariant（Refunded(m)：m 已退款）：

∀ m ∈ Msgs : Refunded(m) ⇒ PayCount(m) = 0

問：(1) 它到目前為止證明了什麼？(2) 怎麼判定它是不是在空轉？(3) 若真在空轉，怎麼處置？

### 推演解答

(1) 如果這三個月沒有任何訊息走過退款流程（Refunded 恆假），這條 invariant 一直是空洞為真——它證明的唯一一件事是「沒有東西被退款過」，後件 PayCount(m) = 0 從未被檢查。三個月的綠燈，資訊量是零。

(2) witness check：找一個 Refunded(m) 為真的實例——查資料、查可達狀態，任何形式都行。找得到，invariant 真的驗過東西；找不到，它在空轉。記住「invariant 通過」與「invariant 驗過東西」是兩個獨立的問題，第二個問題沒有任何工具會自動替你問。

(3) 兩種情況。若退款流程在系統裡存在、只是模型沒涵蓋到，就把場景補進模型，並用一招確認場景真的被走到：故意寫一條錯的主張（例如 ∀ m ∈ Msgs : ¬Refunded(m)），它被打破才表示退款狀態真的可達——「讓反例替你做展示」這招 ch08 的 DieHard 會正式登場。若退款根本不在這個模型的範圍內，誠實的處置是把這條 invariant 連同「本模型不含退款」的註記一起移除或標示——留著一條永遠空轉的 invariant，是給下一個讀 spec 的人埋假安全感。

## 自我檢核

口頭回答，講得出來才算過：

1. p ⇒ q 在 p 為假時為什麼定義為真？用契約或 guard clause 的語言給一個工程上說得通的理由，不准回答「定義就是這樣」。
2. 什麼是 vacuous truth？舉一個你自己系統裡「綠燈但什麼都沒檢查」的真實例子，並說出拆穿它的具體動作。
3. ¬(p ⇒ q) 等價於什麼？為什麼說它是「bug report 的格式」？
4. ∀ m ∃ c 與 ∃ c ∀ m 差在哪？哪條強？用訊息與 consumer 造一對例句示範。
5. 怎麼判斷一條式子是命題還是述語？bound／free variable 跟這個判斷有什麼關係？
6. 空集合上的 ∀ 與 ∃ 各取什麼真值？為什麼說 JavaScript 的 `[].every` 與 `[].some` 跟邏輯學家站同一邊？
7. 把白話規則翻成邏輯式的五個步驟是什麼？為什麼第一步是找量詞範圍？
8. 「找反例」在邏輯上等於對哪條式子做什麼事？這跟 ch09 模型檢查器的工作有什麼關係？
