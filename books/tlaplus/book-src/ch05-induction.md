# ch05 — 歸納法：證明「永遠」的唯一辦法

> **本章解決什麼問題**：ch02 把 invariant（不變量）定義成「所有可達狀態都成立的述語」，但可達狀態往往多到畫不完，甚至無限多——逐一檢查不是辦法。本章交給你處理「永遠」的唯一有限手段：歸納法——起點成立、每步保持，就永遠成立。全書最重要的單一概念 **inductive invariant（歸納不變量）** 在此正式登場，全書沿用的層級證明格式也從本章開始養成。Part V 的整個證明工程（ch14–16）都站在這一章上面。

## 從你已知的出發

你其實天天在做歸納證明，只是沒這樣叫它。

第一個例子：code review 時你守著「`balance` 永遠不會是負的」。你怎麼說服自己？你不會（也不可能）列舉這個帳戶未來所有可能的請求序列。你做的是兩件有限的事：一、開戶時 balance = 0，不是負的；二、逐條檢查每一段會動到 balance 的程式碼，確認它在扣款前都驗過餘額。然後你就敢說「永遠」。

第二個例子：你相信 production 的 DB schema 跟 repo 裡的 migration 資料夾一致。這份信心不是來自每天 diff 一次，而是一條遞推鏈：最初的 schema 由第一份 migration 建立；之後每一次變更都只透過 migration 進行。起點對、每一步保持，所以永遠對。也注意這份信心有多脆弱：只要有人 SSH 進去手改一次 schema——繞過「每一步」——整條鏈應聲而斷，而且斷在哪你不知道。

第三個例子：binary search 的迴圈不變量（loop invariant）：「目標若存在，必在 [lo, hi) 內」。進迴圈前成立、每圈保持、迴圈結束時拿它收割結論。

三個例子是同一個邏輯形狀：**用「起點」加「每步保持」兩件有限的事，換一個無限的「永遠」**。本章把這個形狀磨成一把能承重的工具——承「結算系統永不重複入帳」這種重量——並給它配一套你能寫、別人能查的證明格式。

## 把「永遠」拆成兩件有限的事

先回到數學的原型。你大學的離散數學看過它，現在把它磨利。**數學歸納法**（mathematical induction）：對自然數上的述語 P，若

1. **基底**（base case）：P(0) 成立；
2. **歸納步**（inductive step）：∀ n ∈ ℕ：P(n) ⇒ P(n+1)；

則 ∀ n ∈ ℕ：P(n)。

白話就是推骨牌：第 0 張倒了；任何一張倒了都會推倒下一張；所以每一張都會倒。注意兩個條件各自都是有限可檢查的：基底是一次具體驗算；歸納步是一條**對任意 n 的蘊涵**——你不必知道 n 是多少，只要論證「假設 P(n) 成立」能推出 P(n+1)。那個「假設 P(n) 成立」叫**歸納假設**（induction hypothesis），它是歸納步裡免費的彈藥。

歸納原理本身要不要證？不用——在標準的自然數公理系統（Peano）裡它是公理，等於「自然數是什麼」這個定義的一部分。本書不展開；你只需要記得它不是經驗法則，是地基。

### 層級證明：本書的證明格式

從這一章起，本書所有證明都用同一種寫法。先看一個你熟的小定理怎麼穿上它：對所有 n ∈ ℕ，0 + 1 + 2 + ⋯ + n = n(n+1)/2。

令 P(n) ≜ 「0 + 1 + ⋯ + n = n(n+1)/2」。

⟨1⟩1. 基底 P(0)：左邊＝0（空手起步，只有 0 這一項），右邊＝0·1/2＝0，相等。
⟨1⟩2. 歸納步：對任意 n ∈ ℕ，P(n) ⇒ P(n+1)。
  ⟨2⟩1. 假設 P(n)。（歸納假設）
  ⟨2⟩2. 0 + ⋯ + (n+1) = (0 + ⋯ + n) + (n+1) = n(n+1)/2 + (n+1)。理由：拆出最後一項，再用 ⟨2⟩1 替換括號。
  ⟨2⟩3. n(n+1)/2 + (n+1) = (n+1)(n+2)/2。理由：提出公因式 n+1，剩 n/2 + 1 = (n+2)/2。
  ⟨2⟩4. Q.E.D. ⟨2⟩2 接 ⟨2⟩3 就是 P(n+1) 的等式。
⟨1⟩3. Q.E.D. 由 ⟨1⟩1、⟨1⟩2 與數學歸納法。

怎麼讀（也怎麼寫）這種證明：

- **編號是 ⟨層⟩序**：⟨1⟩1、⟨1⟩2 是最外層的步驟；某一步若一行理由講不完，就往下開一層專屬於它的 ⟨2⟩1、⟨2⟩2……依此類推，⟨3⟩ 永遠隸屬於它上面最近的某個 ⟨2⟩ 步。
- **每層以 Q.E.D. 步收尾**：宣告「本層的目標已由前面諸步湊齊」，並指名用了哪幾步。這一步沒有新內容，它是裝配說明。
- **每個不再往下展開的步，附一行理由**：用了哪個假設、哪個定義、前面哪一步。寫不出這一行，代表這一步你還沒真懂——這是對自己的測謊器。
- **讀法是可調深度的**：先只讀 ⟨1⟩ 層，那是證明的骨架；哪一步可疑，就鑽進它的下一層。信任可以分層發放。

這個格式抄自 Lamport。他的觀察是：散文式證明擅長把跳步藏進流暢的句子裡，而層級結構讓每個跳步都變成肉眼可見的「少一層」，無所遁形（見延伸閱讀）。本書從這裡一路用到 ch16——到時你會看到 TLAPS 用幾乎一模一樣的層級格式讓機器逐步查證，你現在養成的手感屆時無縫接軌。也先說好本書的嚴謹度基準：每一步要能口頭講出理由，禁止用「這是顯然的」打發任何一步；但不要求機械式的形式推導——那是機器的工作（見 ch16）。

### 強歸納：當「前一步」不夠用

有些論證在證 P(n) 時需要的不是 P(n−1)，而是更早的某些祖先。**強歸納**（strong induction）允許你假設 P 在所有更小的數上都成立：

若 ∀ n ∈ ℕ：（∀ k < n：P(k)）⇒ P(n)，則 ∀ n ∈ ℕ：P(n)。

注意基底藏在裡面：n = 0 時「所有 k < 0」是空集合，前件空洞成立（ch03 的 vacuous truth），所以你等於要無條件證出 P(0)。

經典例子：每個整數 n ≥ 2 都能寫成質數的乘積。

⟨1⟩1. 任取 n ≥ 2，假設每個 2 ≤ k < n 都可分解為質數乘積。（強歸納假設）
⟨1⟩2. 情況一：n 是質數。n 自己就是長度一的乘積。
⟨1⟩3. 情況二：n 不是質數。依定義 n = a·b，其中 2 ≤ a < n、2 ≤ b < n。由 ⟨1⟩1，a 與 b 各自可分解；接起來就是 n 的分解。
⟨1⟩4. Q.E.D. 兩種情況蓋住所有 n ≥ 2。

一般歸納在 ⟨1⟩3 幫不上忙：你需要的是 P(a) 與 P(b)，不是 P(n−1)。這個形狀你熟——quicksort 遞迴到的不是「規模減一」的子問題，而是任意更小的分割；強歸納就是這種遞迴的證明影子。最後去魅一句：強歸納不是新公理——令 Q(n) ≜「∀ k ≤ n：P(k)」，對 Q 做一般歸納就得到強歸納，它只是方便的包裝。

### 結構歸納：對形狀做歸納

資料結構是「歸納定義」出來的：list 要嘛是空、要嘛是一個元素接上一個更小的 list；二元樹要嘛是一片葉、要嘛是一個節點掛兩棵更小的子樹。**結構歸納**（structural induction）順著定義的形狀走：每種基底構造直接驗，每種組合構造假設零件成立、證組合成立。

小例：二元樹（每個內部節點恰有兩棵子樹）滿足「葉節點數 L(t) ＝ 內部節點數 I(t) + 1」。

⟨1⟩1. 基底：t 是單一葉節點。L(t) = 1、I(t) = 0，1 = 0 + 1。
⟨1⟩2. 組合：t 是一個內部節點掛上子樹 l 與 r；假設 l、r 都滿足性質。
  ⟨2⟩1. L(t) = L(l) + L(r) = (I(l) + 1) + (I(r) + 1)。理由：葉全在子樹裡；對 l、r 各用一次歸納假設。
  ⟨2⟩2. I(t) = I(l) + I(r) + 1。理由：內部節點＝兩棵子樹的，加上根自己。
  ⟨2⟩3. Q.E.D. 由 ⟨2⟩1 與 ⟨2⟩2：L(t) = I(t) − 1 + 2 = I(t) + 1。
⟨1⟩3. Q.E.D. 二元樹只有這兩種構造方式。

你每天寫的遞迴函式就是結構歸納的程式版：pattern match 的 case 數＝歸納的 case 數，遞迴呼叫＝引用歸納假設。本書後面常對歸納定義的東西做證明——sequence（ch04）、執行的有限前綴（下一節馬上用到）都是。

## 狀態機上的歸納：不變量規則

回到 ch02 的語言。**執行**：從某個初始狀態出發的有限狀態序列，相鄰兩個狀態之間都是某個動作允許的一步。狀態**可達**：存在一條以它收尾的執行。invariant：在所有可達狀態上都成立的述語。麻煩在於「所有可達狀態」沒辦法逐一檢查——本章等下會遇到無限大的狀態空間，而 ch02 的結算系統光加一個 consumer 就讓你畫圖畫到手痠。

但注意：「可達」這個概念本身就是歸納定義的——0 步可達的是初始狀態；n+1 步可達的是「從某個 n 步可達的狀態再走一步」。對歸納定義的東西，用歸納法。這次被歸納的不是自然數本身，而是**抵達一個狀態所需的步數**：

**定理（不變量規則）**：給定狀態機與狀態述語 Ind。若

-（A）**起點**：每個初始狀態都滿足 Ind；
-（B）**封閉**：對任何滿足 Ind 的狀態 s，任何動作把 s 帶到的任何狀態 t 都滿足 Ind；

則每個可達狀態都滿足 Ind。

證明（對步數做歸納；令 P(n) ≜「所有 n 步可達的狀態都滿足 Ind」）：

⟨1⟩1. P(0)：0 步可達的狀態就是初始狀態，由（A）成立。
⟨1⟩2. 對任意 n：P(n) ⇒ P(n+1)。
  ⟨2⟩1. 任取一個 n+1 步可達的狀態 t，以及一條見證它的長度 n+1 的執行。
  ⟨2⟩2. 把這條執行的最後一步拆下來：存在狀態 s 與一步 s → t，而去掉最後一步剩下的前綴是長度 n 的執行，所以 s 是 n 步可達的。
  ⟨2⟩3. s 滿足 Ind。理由：歸納假設 P(n) 套在 ⟨2⟩2 的 s 上。
  ⟨2⟩4. t 滿足 Ind。理由：（B）套在 ⟨2⟩2 的那一步與 ⟨2⟩3。
  ⟨2⟩5. Q.E.D. t 是任取的。
⟨1⟩3. Q.E.D. 由 ⟨1⟩1、⟨1⟩2 與數學歸納法，∀ n：P(n)；而「可達」就是「存在某個有限的 n 使其 n 步可達」，故所有可達狀態滿足 Ind。

之後全書把這條規則念成口訣：**起點成立、每步保持，所以永遠成立**。（ch06 會把「起點」與「每一步」寫成 TLA+ 的 Init 與 Next——名字先記著，語法到時再說。）

三件事值得停下來想清楚。

**第一，為什麼對自然數的歸納就足以覆蓋「永遠」？** 行為可以無限長，但行為裡的每個狀態都出現在某個有限位置 n。「永遠成立」＝「在每個有限位置都成立」，歸納恰好逐位置覆蓋。這其實是 safety 性質的本色：它要壞，必定在某個有限時刻壞給你看（ch07 會把這句話變成 safety 的正式定義）。反過來，「某事終將發生」型的性質（liveness，例如 `AllPaid`）不是這個形狀——歸納法管不到它，那是另一套工具的事（見 ch07，證明面 ch14 淺談）。

**第二，為什麼歸納步只需要「前一個狀態」？** 因為狀態機的定義就是：下一步只取決於目前狀態，不取決於你怎麼來的。歷史不影響未來——除非歷史被記錄在狀態裡。對照強歸納很有意思：自然數上你可以免費引用全部歷史，狀態機上你只有眼前這一格的記憶。推論：當你的性質需要「歷史」才撐得起來（例如「這則訊息曾經被領走過」），你就得把那段歷史編碼進述語、甚至加變數去記——這是 ch14「輔助述語」的伏筆。

**第三，「唯一辦法」不是修辭。** 我認為可以站得住：程式驗證的 inductive assertion 方法（Lamport 在 Teaching Concurrency 裡把它歸給 Floyd 與 Hoare）是它；TLC 的可達性窮舉（ch09）是把同一個遞推在有限實例上機械地跑到收斂；Apalache 的歸納模式（ch16）是把條件（B）丟給 SMT 檢查。所有已知能承重的 safety 論證，拆開都是不變量規則穿不同的衣服。差別只在「誰來執行歸納步、在多大的範圍執行」。

## Inductive invariant：「為真」與「可證」是兩回事

現在給本章的主角正式名分。

**定義**：述語 Ind 是狀態機的 **inductive invariant（歸納不變量）**，當且僅當它滿足不變量規則的兩個前提——（A）所有初始狀態滿足 Ind；（B）從**任何**滿足 Ind 的狀態走任何一步，仍滿足 Ind。

對照 ch02：invariant 是「所有**可達**狀態都成立」的述語。兩個定義的差異全在量詞範圍：條件（B）量化的是「所有滿足 Ind 的狀態」——**包括那些實際執行永遠到不了的狀態**。這一字之差是本章、也是整個 Part V 的戲肉：

- inductive ⇒ invariant。不變量規則保證。
- invariant ⇏ inductive。一個述語可以在每個可達狀態都為真，卻在某個「不可達、但滿足它」的狀態上一步跨出去。

畫成圖（這張圖值得你記住一輩子）：

```text
┌─ 滿足目標性質 S 的狀態 ────────────────────┐
│                                            │
│  ┌─ 滿足 Ind 的狀態（每步都封閉在內）─┐    │
│  │                                    │    │
│  │   ┌─ 可達狀態 ─────────┐           │    │
│  │   │  初始─►s1─►s2─►... │           │    │
│  │   └────────────────────┘           │    │
│  └────────────────────────────────────┘    │
│                                            │
│  s（S 為真、Ind 為假）──走一步─────────────┼──► 性質 S 破了
└────────────────────────────────────────────┘
```

讀法：目標性質 S 為真，意思只是「可達狀態都落在 S 圈內」。但 S 圈裡可能住著像 s 這樣的幽靈——不可達、卻滿足 S——而它一步就能跨出圈外。歸納步面對的正是整個 S 圈（它分不出誰可達），所以會在 s 身上卡死。Ind 圈則不同：它對「圈內任何一步」封閉，所以歸納推得動；只要它同時包住可達集合、又縮在 S 圈內，S 就被證明了。

於是，證「目標性質 S 是 invariant」的標準路線是**三個義務**：

1. **起點**：所有初始狀態滿足 Ind；
2. **封閉**：Ind 走任何一步還在 Ind 內；
3. **蘊涵**：Ind ⇒ S。

運氣好時 Ind 就是 S 自己（這時叫 S 本身 inductive）；更常見的是 S 不 inductive，你得在 S 上疊幾條輔助知識才封得起來。疊的動作叫**強化**（strengthening）。

最後是日常工作流。歸納步證不動時，你手上會有一個具體的**壞前狀態**：滿足 Ind、卻一步跨出去的狀態。下一個動作永遠是同一個問題——**它可達嗎？**

- 可達 → 你的性質根本是假的。從初始狀態找一條走到它的執行，接上那要命的一步，就是反例。
- 不可達 → 性質可能仍為真，只是述語太弱、管不住幽靈。把「它為什麼不可達」的理由寫成新述語疊進 Ind，重證。

證、卡、讀卡點、判可達、強化或舉反例——這個迴圈就是證 safety 的日常。本章接下來讓你完整走兩遍；至於怎麼又快又準地讀出「該強化什麼」，那門技藝深論留給 ch14。

## 完整手推：ping-pong 機的彎路

你維護過 WebSocket 訊息系統，對 heartbeat 不陌生：client 發 ping，server 回 pong，嚴格一來一回——收到 pong 之前不發下一個 ping。把它削成最小的狀態機：

- 變數：turn ∈ {A, B}（現在輪到誰動：A＝client 可發 ping，B＝server 該回 pong）；p ∈ ℕ（累計發出的 ping 數）；q ∈ ℕ（累計回覆的 pong 數）。
- 初始狀態：⟨turn = A, p = 0, q = 0⟩。
- 動作：
  - Ping：前提 turn = A。效果：p 加 1，turn 變 B。
  - Pong：前提 turn = B。效果：q 加 1，turn 變 A。

可達狀態長這樣（圖中縮寫 (turn, p, q)）：

```text
(A,0,0) ─Ping─► (B,1,0) ─Pong─► (A,1,1) ─Ping─► (B,2,1) ─Pong─► (A,2,2) ─Ping─► ...
```

p 與 q 沒有上限，狀態空間是**無限的**：手畫畫不完，ch09 的 TLC 想窮舉也得先砍出有限範圍。要對這台機器說「永遠」，只剩證明一條路——正好拿它練第一次完整的歸納。

要證的性質，從 heartbeat 的營運直覺翻過來：

Gap ≜ q ≤ p ∧ p ≤ q + 1

白話：pong 不會超發（回覆數不超過請求數），且 ping 至多領先一個（嚴格一來一回）。掃一眼上面的鏈，每個可達狀態都滿足——Gap 是 invariant。現在來證。

### 第一次嘗試：直接拿 Gap 做歸納

⟨1⟩1. 起點：初始狀態 p = q = 0，0 ≤ 0 且 0 ≤ 0 + 1，成立。
⟨1⟩2. 封閉：設步前狀態滿足 Gap，分動作討論。
  ⟨2⟩1. 情況 Ping（前提 turn = A）：步後 p 的值是 p + 1，q 不變。要驗 q ≤ p + 1 與 p + 1 ≤ q + 1。
    前半：由歸納假設 q ≤ p ≤ p + 1，成立。
    後半：p + 1 ≤ q + 1 等價於 p ≤ q；但歸納假設只給 q ≤ p ≤ q + 1，**容許 p = q + 1**。若步前 p = q + 1，步後 p 的值是 q + 2 > q + 1。**卡住。**

卡住不是世界末日，是資訊。照流程辦事：把卡點具體化成壞前狀態——⟨turn = A, p = 1, q = 0⟩。檢查：它滿足 Gap（0 ≤ 1 ≤ 0 + 1），Ping 的前提成立，走一步到 ⟨turn = B, p = 2, q = 0⟩，Gap 的第二條破了（2 > 0 + 1）。貨真價實的壞前狀態。

接著問標準問題：**⟨turn = A, p = 1, q = 0⟩ 可達嗎？** 看可達鏈：turn = A 的可達狀態是 (A,0,0)、(A,1,1)、(A,2,2)……每個都是 p = q。「輪到 client 發、卻已經領先一個」的狀態，真實執行裡到不了。上一節那張圖裡的幽靈 s，在這裡有了名字。結論：Gap 為真（invariant），但**不 inductive**。

### 讀懂失敗：缺的是哪句話

卡點會告訴你缺什麼。歸納假設不知道「輪到誰」跟「領先多少」是綁在一起的：真實執行裡，輪到 A 時 ping 與 pong 打平，輪到 B 時 ping 恰好領先一。Gap 把兩種情況糊成一個範圍 [q, q+1]，丟掉了「哪個值對應哪個 turn」的資訊。把這句話寫回述語——強化：

Ind ≜ （turn = A ⇒ p = q）∧（turn = B ⇒ p = q + 1）

先驗收蘊涵：Ind ⇒ Gap？turn 只有 A、B 兩種值，兩個 case 分別給 p = q 與 p = q + 1，都落在 q ≤ p ≤ q + 1 內。成立。

### 第二次嘗試：證 Ind

⟨1⟩1. 起點：初始狀態 turn = A 且 p = 0 = q，第一條成立；第二條前件為假，空洞成立。（ch03 的 vacuous truth 這次站在你這邊——前件假的蘊涵不欠你任何東西。）
⟨1⟩2. 封閉：設步前狀態滿足 Ind，分動作討論。
  ⟨2⟩1. 情況 Ping。前提 turn = A，故 Ind 第一條的前件為真：p = q。步後：p 的值是 p + 1 = q + 1，q 不變，turn = B。驗步後的 Ind：第一條前件假（turn 已是 B），空洞成立；第二條要求步後的 p 值等於 q + 1，由 p + 1 = q + 1 成立。
  ⟨2⟩2. 情況 Pong。前提 turn = B，故 Ind 第二條給 p = q + 1。步後：q 的值是 q + 1 = p，p 不變，turn = A。驗步後的 Ind：第一條要求 p 等於步後的 q 值，由 p = q + 1 成立；第二條前件假，空洞成立。
  ⟨2⟩3. Q.E.D. 動作只有 Ping、Pong 兩個，⟨2⟩1、⟨2⟩2 蓋住所有步。
⟨1⟩3. 蘊涵：Ind ⇒ Gap（上面驗過）。
⟨1⟩4. Q.E.D. 由 ⟨1⟩1、⟨1⟩2 與不變量規則，所有可達狀態滿足 Ind；再由 ⟨1⟩3，Gap 是 invariant。∎

### 三個值得帶走的觀察

**更強的主張反而更好證。** Gap 弱，證不動；Ind 強，一次就過。不神祕：要證的東西變強，歸納假設同步變強——火力跟負擔一起上升，而上升的火力恰好補上卡點。數學課把這招叫「加強歸納假設」；在狀態機世界它有專名：強化 invariant。什麼時候強化、強化多少，是一門手藝（見 ch14）。

**Ind 是怎麼找到的？** 這裡兩條路殊途同歸：從卡點反讀（缺「turn 與 p − q 的綁定」），或直接盯著可達鏈找規律（A 時打平、B 時領先一）。小系統兩條路都通；大系統的系統化方法見 ch14。

**這個例子裡 Ind 恰好把可達集合一字不差圈了出來**：滿足 Ind 的狀態就是 ⟨A, k, k⟩ 與 ⟨B, k+1, k⟩（k ∈ ℕ），正是那條鏈。這是小例子的奢侈，不是要求——inductive invariant 只需要三個義務成立，圈得比可達集合鬆完全合法。下一節就是圈得鬆的例子。

## 手證結算系統 v0 的 NoDoublePay

熱身結束，上正菜：對 ch02 的結算系統 v0，手證一條真實世界重量級的性質——`NoDoublePay`。

### 把 v0 說成狀態機

參數照 ch02 的基準：訊息兩則，Msgs = {m1, m2}；consumer 從一個加到兩個——這裡直接寫 Consumers = {c1, c2}，而且你馬上會看到，證明根本不在乎這兩個集合裡有幾個元素。三個變數（「函數＝查表」的讀法見 ch04）：

- queue：尚未被領取的訊息集合；值是 Msgs 的子集合。
- working：每個 consumer 手上的訊息；working[c] ∈ Msgs ∪ {"idle"}，哨兵值 "idle" 表示空手。
- ledger：每個 msgId 是否已入帳；ledger[m] ∈ {未入帳, 已入帳}。

初始狀態：queue = {m1, m2}，所有 working[c] = "idle"，所有 ledger[m] = 未入帳。

動作（v0 的世界裡沒有 crash、沒有 redelivery——那些是 v1 的現實，見 ch08）：

- Fetch(c, m)：前提 working[c] = "idle" 且 m ∈ queue。效果：把 m 從 queue 移除，working[c] 變為 m。**領取與出隊是同一步**——這是 v0 的原子性假設，先記著，最後要對帳。
- Settle(c)：前提 working[c] = m（某個 m ∈ Msgs）。效果：ledger[m] 變為已入帳，working[c] 變回 "idle"。

一則訊息在 v0 的一生：

```text
queue ──Fetch(c,m)──► working[c] ──Settle(c)──► ledger[m] = 已入帳（從此離開系統）
```

### NoDoublePay 的狀態述語長相

ch02 的直覺版是「同一 msgId 至多入帳一次」。這裡有個誠實的小尷尬：ledger[m] 是布林——「再入帳一次」只是把已入帳蓋寫成已入帳，**狀態上看不出差別**（ch04 說過：建模時丟掉的資訊，性質就無法談論它）。所以把「至多一次」翻成狀態語言，要說的其實是：一旦入帳，這則訊息就徹底離開系統，再也沒有任何動作能對它觸發第二次入帳——

NoDoublePay ≜ ∀ m ∈ Msgs：ledger[m] = 已入帳 ⇒ （m ∉ queue ∧ ∀ c ∈ Consumers：working[c] ≠ m）

只要它在每個可達狀態都成立，「第二次 Settle 同一則訊息」就永遠不可能發生：Settle 需要有人手上拿著 m，而這個述語說已入帳的 m 不在任何人手上、也回不了佇列。

### 先讓它卡一次

直接拿 NoDoublePay 做歸納。起點沒問題：全部未入帳，每條蘊涵前件都是假的，空洞成立。封閉呢？看 Settle：

設步前滿足 NoDoublePay，consumer c 拿著 m，執行 Settle(c)。步後 ledger[m] = 已入帳——述語對 m 的前件**從假變真**，步前空洞成立的那條蘊涵，步後突然要求三件事：m ∉ queue；c 自己放手了（成立，working[c] 變 "idle"）；**其他** consumer 手上也都沒有 m。前兩件有著落，第三件呢？歸納假設一聲不吭：步前 ledger[m] 還是未入帳，NoDoublePay 對 m 什麼都沒說。卡住。

具體的壞前狀態，例如：

s ≜ ⟨queue = {m1}, working = (c1 ↦ m1, c2 ↦ "idle"), ledger = (m1 ↦ 未入帳, m2 ↦ 已入帳)⟩

檢查 s 滿足 NoDoublePay：m1 未入帳，蘊涵空洞成立；m2 已入帳且不在佇列、不在任何手上，成立。但 s 裡 m1 同時在佇列**和** c1 手上——Settle(c1) 之後 ledger[m1] = 已入帳而 m1 還躺在 queue 裡，述語應聲而破。（另一型壞前狀態：working[c1] = working[c2] = m1，兩隻手同時拿著同一則訊息。）

照流程問：這些狀態可達嗎？直覺說不可達——v0 的 Fetch 是原子的，一則訊息出了隊就不會還在隊裡，更不會被拿兩次。但「直覺上」三個字不能寫進證明。這個直覺有個正式的名字：**守恆**。把它寫成述語。

### 守恆強化與完整證明

定義每則訊息在系統裡的存量：

copies(m) ≜ （若 m ∈ queue 則 1，否則 0）＋ |{c ∈ Consumers : working[c] = m}|

強化後的述語（本章暫稱 IndPay）：

IndPay ≜ ∀ m ∈ Msgs：copies(m) ≤ 1 ∧ （ledger[m] = 已入帳 ⇒ copies(m) = 0）

讀法：第一條是守恆——全系統裡一則訊息至多存在一份，像一張支票不該有影本；第二條就是 NoDoublePay 換個寫法——copies(m) = 0 展開正是「不在 queue、不在任何手上」。

**定理**：v0 的每個可達狀態都滿足 NoDoublePay。

⟨1⟩1. 起點：初始狀態滿足 IndPay。
  任取 m：m ∈ queue 貢獻 1、無人持有，copies(m) = 1 ≤ 1；ledger[m] = 未入帳，第二條前件假，空洞成立。
⟨1⟩2. 封閉：設步前狀態滿足 IndPay，分動作討論。
  ⟨2⟩1. 情況 Fetch(c, m)（前提：working[c] = "idle"、m ∈ queue）。
    ⟨3⟩1. 步前 ledger[m] = 未入帳。理由：若已入帳，IndPay 第二條給 copies(m) = 0；但 m ∈ queue 使 copies(m) ≥ 1，矛盾。
    ⟨3⟩2. 步前無人持有 m。理由：m ∈ queue 已貢獻 1，IndPay 第一條說總量 ≤ 1，故各 working 的貢獻為 0。
    ⟨3⟩3. 步後 copies(m) = 1。理由：m 離開 queue（−1）、working[c] 變為 m（+1），其餘貢獻照 ⟨3⟩2 為 0。
    ⟨3⟩4. 步後 IndPay 對 m 成立。理由：1 ≤ 1；Fetch 不動 ledger，由 ⟨3⟩1 步後 ledger[m] 仍未入帳，第二條空洞成立。
    ⟨3⟩5. 步後 IndPay 對每個 m̃ ≠ m 成立。理由：queue 只少了 m，唯一變動的手 working[c] 從 "idle" 變 m，ledger 沒動——copies(m̃) 與 ledger[m̃] 都與步前相同，照歸納假設成立。
    ⟨3⟩6. Q.E.D. ⟨3⟩4 與 ⟨3⟩5 合起來蓋住所有訊息。
  ⟨2⟩2. 情況 Settle(c)（前提：working[c] = m）。
    ⟨3⟩1. 步前 copies(m) = 1，且 m ∉ queue、除 c 外無人持有 m。理由：working[c] = m 已貢獻 1，IndPay 第一條說總量 ≤ 1，故其餘貢獻全為 0。
    ⟨3⟩2. 步後 copies(m) = 0。理由：working[c] 變回 "idle"（−1）；queue 與其他手沒動，而 ⟨3⟩1 說它們本來就不持有 m。
    ⟨3⟩3. 步後 IndPay 對 m 成立。理由：0 ≤ 1；第二條前件為真（ledger[m] 變已入帳），要求 copies(m) = 0，由 ⟨3⟩2 成立。
    ⟨3⟩4. 步後 IndPay 對每個 m̃ ≠ m 成立。理由：queue 沒動，唯一變動的手從 m 變 "idle"，ledger 只改了 m 那一格——m̃ 的存量與入帳狀態都與步前相同。
    ⟨3⟩5. Q.E.D.
  ⟨2⟩3. Q.E.D. v0 只有 Fetch 與 Settle 兩類動作，⟨2⟩1、⟨2⟩2 蓋住所有步。
⟨1⟩3. 蘊涵：IndPay ⇒ NoDoublePay。理由：第二條的 copies(m) = 0 展開即「m ∉ queue ∧ ∀ c：working[c] ≠ m」。
⟨1⟩4. Q.E.D. 由 ⟨1⟩1、⟨1⟩2 與不變量規則，所有可達狀態滿足 IndPay；由 ⟨1⟩3，NoDoublePay 是 invariant。∎

### 這份證明買到了什麼、又欠著什麼

**買到的：規模無關。** 整份證明沒有任何一步用到「訊息有兩則」或「consumer 有兩個」——量詞全是 ∀ m、∀ c，案例分析按動作而不是按參數。同一份證明對 2 則訊息、2 個 consumer 成立，對每季 30 萬筆、200 個 consumer 也成立。對照一下：ch02 手畫的狀態空間圖只驗了一組參數；ch09 的 TLC 也只能窮舉有限組。歸納證明一次買斷所有規模——這就是本章標題的份量。

**順帶一提：IndPay 圈得比可達集合鬆。** 例如「queue 空、大家空手、ledger 全是未入帳」滿足 IndPay，卻是真實執行裡到不了的狀態（訊息不會憑空消失）。沒關係：這個狀態一步也走不動（兩個動作的前提都不成立），封閉性對它免費成立。inductive invariant 不必精確刻畫可達集合，夠封閉、夠強就好。（它走不動是 deadlock？safety 不管這個——「不出事」與「走得下去」是兩種性質，後者見 ch07。）

**欠著的：兩個原子性假設。** 一、Fetch 的原子性。若把「讀訊息」與「出隊」拆成兩步——或者更現實地，佇列服務的 visibility timeout 到期、把處理中的訊息重新投遞——copies(m) 會出現 2，IndPay 第一條當場陣亡。二、Settle 的原子性。入帳與放手同一步完成；若中間可以 crash，「入了帳卻還拿著訊息」的狀態就回來了。v1（ch08）會把 crash 與 redelivery 加回模型，到時 `NoDoublePay` 的擔子落到 dedup 表上，證明得整個重練——那是 ch14 的高峰戲。換句話說：這份證明證的是「v0 這個模型」永不重複入帳，不是「你的 production」永不重複入帳。模型與現實的差距，永遠是你自己的責任（ch02 的老話）。

## 陷阱與防禦

| 陷阱 | 長相 | 防禦／自我察覺 |
|---|---|---|
| 把「跑很久沒壞」當歸納 | 基底驗了、又觀察了一萬步，宣稱永遠 | 一萬步是一萬個 case，不是歸納步。問自己：我的論證對「任意」滿足述語的狀態成立，還是只對我看過的狀態成立？ |
| 歸納步偷用可達性 | 證明裡寫「這種狀態實際上不會發生，略過」 | 「不會發生」正是你要證的結論——循環論證。歸納步只准用三樣東西：歸納假設、動作的前提、動作的效果。看到「實際上」三個字就該警鈴大作 |
| 誤讀失敗的歸納步 | 一卡住就宣告性質是假的；或反過來硬說「差不多證完了」 | 把卡點具體化成壞前狀態，判可達：可達→真反例；不可達→強化。兩條路都有明確的下一步，「差不多」沒有 |
| 三個義務缺一 | 只證了封閉、忘了起點（假述語的封閉性完美無瑕）；或忘了驗蘊涵（證了一個跟目標無關的東西） | 起點、封閉、蘊涵逐項打勾；再拿一個具體可達狀態代入 Ind，確認它不是空話 |
| 漏動作 case | 「這個動作跟這條性質無關」於是跳過 | 無關也要寫一行理由（它不動哪些變數、為什麼存量不變）。動作 × 述語條款畫成矩陣逐格打勾——v0 證明裡的 ⟨3⟩5 就是在做這件事 |
| 空洞前件的假安全感 | 述語全是 p ⇒ q 形，歸納步都靠前件為假空洞通過 | 問：有沒有可達狀態真的讓前件為真過？沒有的話這條述語什麼都沒說（ch03 的 vacuous truth；ch14 回收這個坑） |
| 把 invariant 與 inductive invariant 當同義詞 | 看到性質為真就直接逐動作驗封閉，卡住後懷疑人生 | 為真是「可達狀態上的事實」；inductive 是「述語區域的封閉性」。先有心理準備：有意義的目標性質通常**不是** inductive（ch14） |

## 紙上推演

**題 1 [15 分鐘] ★ — 奇數和。** 用本章的層級格式證明：對所有 n ≥ 1，1 + 3 + 5 + ⋯ + (2n − 1) = n²。要求：明確寫出 P(n)、基底、歸納步，每個不展開的步附一行理由。

### 推演解答

令 P(n) ≜「前 n 個奇數的和等於 n²」。

⟨1⟩1. 基底 P(1)：左邊＝1，右邊＝1²＝1，相等。
⟨1⟩2. 歸納步：對任意 n ≥ 1，P(n) ⇒ P(n+1)。
  ⟨2⟩1. 假設 P(n)：1 + 3 + ⋯ + (2n − 1) = n²。（歸納假設）
  ⟨2⟩2. 前 n+1 個奇數的和＝（前 n 個奇數的和）＋（2(n+1) − 1）＝ n² + 2n + 1。理由：第 n+1 個奇數是 2(n+1) − 1 = 2n + 1；和的前段用 ⟨2⟩1 替換。
  ⟨2⟩3. n² + 2n + 1 = (n + 1)²。理由：展開 (n+1)²。
  ⟨2⟩4. Q.E.D. ⟨2⟩2 接 ⟨2⟩3 即 P(n+1)。
⟨1⟩3. Q.E.D. 由 ⟨1⟩1、⟨1⟩2 與數學歸納法。

常見錯路：在歸納步裡把 P(n+1) 的等式「假設成立」，左右同減 2n + 1 變回 P(n)——方向反了，這是從結論推前提。正確姿勢永遠是從左邊出發、用歸納假設替換、整理成右邊。另一個小坑：第 n+1 個奇數是 2n + 1 不是 2n − 1——通項跟基底都值得代個小數字驗算（n = 2：1 + 3 = 4 = 2²）。

**題 2 [20 分鐘] ★★ — 請求與回應。** 一台計數器機：變數 sent, acked ∈ ℕ，初始皆為 0。動作 Send：無前提，效果 sent 加 1。動作 Ack：前提 acked < sent，效果 acked 加 1。證明 acked ≤ sent 是 inductive invariant。（這題的目標自己就 inductive，不需強化——請完整走完三個義務，並留意每一步的彈藥從哪來。）

### 推演解答

⟨1⟩1. 起點：acked = 0 ≤ 0 = sent。
⟨1⟩2. 封閉：設步前 acked ≤ sent，分動作討論。
  ⟨2⟩1. 情況 Send：步後 sent 的值是 sent + 1，acked 不變。acked ≤ sent < sent + 1。理由：歸納假設，加上加一不會變小。
  ⟨2⟩2. 情況 Ack：前提 acked < sent。步後 acked 的值是 acked + 1，sent 不變。由前提得 acked + 1 ≤ sent。
  ⟨2⟩3. Q.E.D. 動作只有兩個。
⟨1⟩3. 蘊涵：目標就是 acked ≤ sent 自己，免費成立。
⟨1⟩4. Q.E.D. 由不變量規則。

兩個值得注意的點。其一：⟨2⟩2 的關鍵彈藥不是歸納假設，而是**動作的前提**（acked < sent）——guard 常常才是歸納步過關的主力，這正是「歸納步只准用歸納假設、前提、效果三樣」清單裡前提佔一席的原因。其二：想像拿掉 Ack 的前提（允許超收 ack）——封閉立刻破，壞前狀態 acked = sent 一步變成 acked > sent，而且這次它**可達**（初始狀態就是）：性質直接是假的。同一個證明骨架，讓你一眼看出修哪裡會塌。

**題 3 [25 分鐘] ★★★ — 會卡的那種。** 一台批次計數器：變數 x ∈ ℕ，初始 x = 0。動作 Step：前提 x < 10，效果 x 加 2。動作 Reset：前提 x = 10，效果 x 歸 0。目標性質：x ≠ 7。
（a）直接拿 x ≠ 7 做歸納，找出卡住的位置，寫出一個具體的壞前狀態，並判斷它是否可達。
（b）提出一個強化述語，完成三個義務的完整層級證明。

### 推演解答

（a）起點：0 ≠ 7，成立。封閉：情況 Step——步前 x ≠ 7 且 x < 10，步後 x + 2 ≠ 7？歸納假設擋不住 x = 5：5 ≠ 7 成立、前提 5 < 10 成立，步後 x = 7。卡住，壞前狀態是 x = 5。可達嗎？從 0 出發每步加 2、滿 10 歸零，看得到的值是 0, 2, 4, 6, 8, 10, 0, …全是偶數，5 到不了。所以 x ≠ 7 為真但不 inductive。注意：「全是偶數」此刻只是觀察，還不是定理——它正是（b）要證的東西。

（b）把那句觀察升級成述語：Even ≜「x 是偶數」。

⟨1⟩1. 起點：x = 0，偶數。
⟨1⟩2. 封閉：設步前 x 是偶數，分動作討論。
  ⟨2⟩1. 情況 Step：步後 x + 2。偶數加 2 仍是偶數。
  ⟨2⟩2. 情況 Reset：步後 x = 0，偶數。
  ⟨2⟩3. Q.E.D. 動作只有兩個。
⟨1⟩3. 蘊涵：x 是偶數 ⇒ x ≠ 7。理由：7 是奇數。
⟨1⟩4. Q.E.D. 由不變量規則，x ≠ 7 是 invariant。

討論：強化不必「剛剛好」。Even ∧ x ≤ 10 也是 inductive（值得自己驗一次：Step 的歸納步在哪裡用到前提 x < 10），而且把可達集合圈得更緊；但對「x ≠ 7」這個目標，Even 一條就夠承重。強化到多強才夠、會不會強過頭（強到連起點都不成立），是 ch14 的正題。這題的迷你版教訓：**強化述語往往就是你解釋「壞前狀態為什麼不可達」時，說出口的那句話**——把口頭理由升級成述語，是這門手藝的起點。

## 自我檢核

口頭回答，講得出來才算過關：

1. 用兩句話向另一個工程師解釋「invariant」與「inductive invariant」的差別，並說清楚為什麼前者不保證能直接做歸納。
2. 不變量規則的三個義務是哪三個？對每一個，舉一個「少了它會假通過」的具體例子。
3. 歸納步卡住時，標準流程是什麼？「壞前狀態可達」與「不可達」各對應什麼結論與下一步動作？
4. 為什麼歸納法配得上「證明永遠的唯一辦法」這個標題？ch09 的 TLC 窮舉算不算第二條路？
5. 層級證明裡 ⟨2⟩3 這個編號怎麼讀？Q.E.D. 步的義務是什麼？為什麼說這個格式讓跳步無所遁形？
6. 歸納步必須照顧不可達的幽靈狀態，聽起來像浪費力氣——它換來了什麼？（提示：精確算出可達集合有多貴？）
7. v0 的證明在哪些步驟依賴「Fetch 是原子的」？如果領取與出隊拆成兩步，IndPay 的哪一條最先陣亡、會在哪個動作的歸納步現形？

## 延伸閱讀

- Leslie Lamport, “How to Write a 21st Century Proof”（2011；後刊於 *Journal of Fixed Point Theory and Applications*, 2012）：<https://lamport.azurewebsites.net/pubs/proof.pdf> — 層級證明格式的原典。讀前半即可：為什麼散文證明擅長藏錯、層級結構怎麼讓每一步無所遁形。本章的 ⟨1⟩⟨2⟩ 編號就是它的手寫版，ch16 的 TLAPS 是它的機器版。
- Leslie Lamport, “Teaching Concurrency”（2009）：<https://lamport.azurewebsites.net/pubs/teaching-concurrency.pdf> — 兩頁的小論文，核心立場：不變量是理解併發系統的鑰匙，而多數工程師沒被教過這樣思考。文末有一個比 ping-pong 機略難的小演算法邀你練手——讀完本章正好接得住。
- Lorin Hochstein, “Inductive invariants”（Surfing Complexity, 2018-12）：<https://surfingcomplexity.blog/2018/12/27/inductive-invariants/> — 用一個 decrement 計數器小例把「為真但不 inductive」講得又短又準，還親手解了 Teaching Concurrency 的習題。適合當本章的第二視角複習。
- Hillel Wayne, “Writing an Invariant”（learntla.com）：<https://learntla.com/core/invariants.html> — invariant 在 TLA+ 工具脈絡下的入門寫法（量詞與 ⇒ 的用法都會再見面）。注意它談的是 invariant 本身、不談 inductive invariant：在 TLC 的工作流程裡你只需要寫出性質，可達性窮舉由機器代勞（見 ch09）；要走證明這條路，inductive 的功夫才回到你手上——也就是本章與 ch14 的內容。
