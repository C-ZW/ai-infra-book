# ch17 — 工程師的科學態度：把全書搬回你的系統

> **本章解決什麼問題**：前面十六章，你一直在讀「別人怎麼做科學、科學怎麼防自欺、科學怎麼失靈又怎麼修補」。ch16 把所有失敗模式收成一張檢查表，那是你讀**別人**的科學宣稱時用的。本章把方向掉轉一百八十度——不是讀別人的科學，是做**你自己**的工程判斷。全書講的那一整套防自欺習慣，哪些是可攜的、能直接搬進你的 post-mortem、技術選型、效能優化、容量規劃？這一章把它們打包成一個工程師版的「科學循環」，並誠實處理一個關鍵問題：科學態度什麼時候會過頭——不是每個決定都值得做 RCT，怎麼分清哪些賭注大到值得認真防自欺、哪些不值得。這是 Part V 的倒數第二章，也是 ch18 收官總對帳前的最後一次「把工具拿在手上用」。

## 從你已知的出發

你其實已經是個科學家了，只是沒人這樣跟你說過，而且你做得時好時壞。

回想你上一次認真做的效能優化。比方說那個經典場景：某個 API 的 P99 延遲長期卡在 800ms，你懷疑是 RDS 上一條沒走索引的查詢在拖。你做了什麼？如果你是個還能保住飯碗的工程師，你**不會**直接「我覺得是這條 query」就上去加個索引、改完宣布搞定。你會：先拉 profiling、把慢查詢日誌攤開、確認那條 query 真的佔了延遲的大頭（**量測**）；形成一個明確的猜測「這條全表掃描是主因」（**假說**）；推出一個可被打臉的預測「如果是它，加上索引後 P99 應該掉到 200ms 以下，而且 RDS CPU 會跟著降」（**可被推翻的預測**）；在 staging 上只改這一個索引、別的不動，跑同一批壓測（**對照**）；看到 P99 真的掉到 180ms、CPU 從 70% 降到 40%（**證據**）；然後——這一步最常被跳過——你不會只跑一次就宣布勝利，你會多跑幾輪、換幾個時段，確認不是某次剛好流量低（**複製**）；最後你想搞清楚**為什麼**降這麼多，是索引讓掃描行數從百萬降到幾百，這個機制講得通（**機制**），你才敢把結論寫進 PR 描述、敢說「真的修好了，不會復發」（**結論**）。

把這串動作的名字連起來看：

```text
量測 → 假說 → 可被推翻的預測 → 對照 → 證據 → 複製 → 機制 → 結論
```

這就是科學循環。你做得好的那幾次，是因為你（多半是無意識地）走完了大半圈；你做砸的那幾次——「重啟就好了，關 ticket」「改了三個東西指標動了，反正動了就好」「跑一次綠就上線」——是因為你在某一步偷懶，而**每一步偷懶都對應前面某一章講過的一種自欺**。本章不是要教你新東西，是要把你工程直覺裡這套循環**顯式化**，給每一步貼上「它在防哪種自欺」的標籤，讓你下次偷懶時心裡會「咯噔」一下。

說得更直白：前十六章是把科學拆開給你看零件；這一章是把零件裝回去，裝進**你的**系統。中央張力在這裡換了個說法——**科學態度不是某種你偶爾切換進去的「研究模式」，是一套你對自己的判斷保持誠實的日常衛生習慣。** 它的全部目的還是那句老話：有系統地把自己被騙的機會壓到最低。只是這次被騙的人是你，騙你的人也是你。

## 科學態度的可攜核心：八個你帶得走的習慣

把全書濃縮成「遇事我該對自己做的自我盤問」，大概是這八條。它們不是流程步驟（流程下一節講），是**態度**——是你在做任何技術判斷時，腦子裡該常駐的八個聲音。每一條後面我標了它的出處章，這樣你知道它不是我隨口編的格言，是前面論證過的東西。

```text
工程師的科學態度：八個可攜習慣（每條註明出處與它防的自欺）

  1. 先問「我怎麼知道」                    （ch01 起手式）
     在相信任何結論前，先問它的證據是什麼、品質如何。
     防的自欺：軼事當數據、印象當事實。

  2. 警惕「我親眼看到」                    （ch02 理論負載）
     你看到的已經被你的指標選擇、閾值、預期 baseline 塑造過。
     防的自欺：以為自己看到的是「裸事實」。

  3. 要求可被推翻的預測                     （ch04 可證偽）
     好假說要能說「如果我對，你會看到 X；看不到 X 我就錯了」。
     防的自欺：怎樣都能自圓其說的「萬能解釋」。

  4. 做對照、控制變因                       （ch05 對照引擎）
     一次只改一個你關心的東西，留個比較基準。
     防的自欺：「同時改了三個、指標動了」的歸因錯亂。

  5. 相關不等於因果，先抓混淆變因           （ch06 因果）
     兩件事一起動，先問有沒有第三個原因同時推著它倆。
     防的自欺：把巧合或共因當成因果。

  6. 一次結果不算數，複製再下結論           （ch09/ch13 複製）
     僥倖會偽裝成效果；多跑幾次、換條件，濾掉僥倖。
     防的自欺：單次顯著、flaky 的綠燈、偷看資料就收手。

  7. 區分「有效果」與「有機制」              （ch14 機制）
     「重啟就好了」是有效果無機制——你不知道它會不會復發。
     防的自欺：黑箱有效當成真懂、把碰巧當成搞定。

  8. 對自己的客觀性保持懷疑                  （ch11 動機性推理）
     你最容易放過的，是支持你既有結論的爛證據。
     防的自欺：「我很客觀」這句話本身（見下文陷阱）。
```

我認為這八條裡，**第 1 條和第 8 條是地基，其餘六條是工具**。第 1 條「我怎麼知道」是所有防自欺的起點——它把你從「相信」切換到「檢查」。第 8 條「對自己的客觀性保持懷疑」是所有防自欺的天花板——因為前面七條工具，每一條都可以被你的動機性推理偷偷繞過（你會「不小心」挑對自己有利的對照、會「剛好」在達標時停止複製）。承認自己會這樣，是用好其他七條的前提。

這八條沒有一條是我發明的。它們是塞麥爾維斯（Ignaz Semmelweis）案被全書十層透鏡照過之後，留在你手上的結晶。

## 工程師的科學循環：不是中學那個五步驟

現在把態度組裝成流程。你大概在國中自然課看過一張「科學方法五步驟」流程圖：觀察 → 假設 → 實驗 → 分析 → 結論，畫成一條從上到下的直線，彷彿做科學就是照表操課走一遍就交差。

**那張圖是錯的，或至少是誤導的。** 全書十六章如果有教你一件事，那就是：科學不是一條直線跑完的管線，是一個**會回頭、會卡住、會推翻自己重來**的迭代迴圈。把它畫成工程師版，長這樣：

```text
工程師的科學循環（迭代，不是一次性管線；箭頭往下走，
                    虛線是「否證成功就回上游」的回頭路）

   ┌──────────┐
   │ 1. 觀察  │   量測：指標、log、profiling
   └────┬─────┘   （ch01：先有系統性比較，不靠印象）
        ▼
   ┌──────────┐
   │ 2. 假說  │   溯因：跳到最能解釋症狀的根因
   └────┬─────┘   （ch03：這是 IBE，不是歸納也非演繹）
        ▼
   ┌──────────┐
   │ 3. 可被  │   演繹：「如果是它，我應該看到 X」
   │  推翻的  │   （ch04：說不出反例的假說是壞假說）
   │  預測    │
   └────┬─────┘
        ▼
   ┌──────────┐
   │ 4. 實驗  │   對照：只改一個變因、留比較基準
   └────┬─────┘   （ch05/ch06：切斷混淆，別歸因錯亂）
        ▼
   ┌──────────┐
   │ 5. 更新  │   貝氏：證據抬高還是壓低這個假說？
   └────┬─────┘   （ch07：強證據＝假說真才會出現的觀察）
        ▼ ····► 預測沒中／證據壓低？回第 2 步換假說
        ▼        （否證成功，這是好事，不是失敗）
   ┌──────────┐
   │ 6. 複製  │   複製：多跑幾次、換條件，濾掉僥倖
   └────┬─────┘   （ch09/ch13：一次綠不算數）
        ▼ ····► 複製失敗？回第 4 步，這是 flaky
        ▼
   ┌──────────┐
   │ 7. 機制  │   問「為什麼」：有效果還要有機制
   └────┬─────┘   （ch14：黑箱有效不等於真懂）
        ▼
   ┌──────────┐
   │ 8. 結論  │   暫定結論：永遠帶著「在測過的範圍內」
   └──────────┘   （ch10：科學的真是暫定收據，不是鐵證）
```

注意這張圖跟中學那張的三個根本差別，每一個都是全書某章的回收：

- **它有回頭箭頭。** 預測沒中、複製失敗，你回到上游換假說、重做實驗——而且這**不是失敗，是循環正常運作**（ch04：否證成功是科學的標誌；ch10：科學結論本就可修正）。中學那張圖讓你以為「結論」是終點；真實的循環裡，結論只是「目前還沒被否證的暫定收據」，隨時可能因為一個新觀察被踢回第 2 步。
- **它把「假說從哪來」獨立成一步（溯因）。** 中學圖直接從觀察跳到假設，彷彿假設會自己長出來。ch03 戳破了這個——你做 root cause 時「跳到最像的嫌疑犯」那一刻是**溯因（IBE）**，是創造性的一躍，不是機械推導。承認這一步是溯因，你才會記得「最像的嫌疑犯不一定是真兇」，才會回頭做可否證的檢驗。
- **它把「複製」和「機制」獨立成步驟。** 中學圖在「實驗→結論」之間什麼都沒有，於是訓練出一堆「跑一次成功就結案」的人。真實循環裡，一次成功和可複製是兩回事（ch09/ch13），有效果和有機制是兩回事（ch14）——塞麥爾維斯的悲劇正是栽在「有效果、被複製過、卻說不出機制」這個縫裡。

這張循環圖就是本章的脊樑。它不是要你每次改一行程式都跑完八步——那叫教條，下文「科學態度何時過頭」會處理。它是一張**地圖**，讓你在任何技術判斷裡都能定位：我現在在第幾步？我跳過了哪一步？那一步本來防的是哪種自欺？

## 「強意見，弱持有」：可證偽態度的工程版

有一句在工程圈流傳很廣的話，叫「**強意見，弱持有（strong opinions, weakly held）**」——這句話常被歸給科技預測家保羅・薩弗（Paul Saffo），由未來研究所（Institute for the Future）推廣，後來被組織行為學者 Bob Sutton 等人帶進科技圈（出處與確切歸屬有不同說法，這裡 hedge，2026-06）。它的意思是：要敢於形成清楚、強硬的判斷（不要當那種「都有道理啦」的和稀泥的人），但要隨時準備好在證據出現時放掉它。

剝掉勵志包裝，這句話**就是 ch04 可證偽態度的個人版**：

```text
「強意見，弱持有」拆成科學態度

  「強意見」      = 敢做出明確、冒風險的判斷
                   （ch04：好假說要冒風險、要可被推翻，
                    不是怎樣都對的廢話）
                   工程版：「我賭是這條 query 拖垮 P99」
                   ——明確到可以被一次量測打臉

  「弱持有」      = 證據一翻盤就放掉，不護航、不找補
                   （ch11：對抗動機性推理——你最捨不得放的，
                    往往是你投入最多、最想對的那個結論）
                   工程版：加了索引 P99 沒降？立刻回第 2 步換假說，
                   不去硬掰「一定是快取沒熱」來保住原判斷
```

但這句話有個**著名的被濫用方式**，得誠實標出來（2026-06 工程圈早有批評）：很多人用「強意見、弱持有」當**逃避責任的擋箭牌**——平常嘴最硬、最不聽人勸（強意見），出事了就「啊我本來就說是弱持有嘛」一句話卸責（假裝弱持有）。真正的「弱持有」不是嘴上說說，是**事先講好什麼證據會讓你改變主意**。這跟可證偽一模一樣：一個假說可不可證偽，看的不是它聽起來多開放，是它**事先指定了什麼觀察能推翻它**。所以這句格言要可操作，得加一句工程版的補丁：

> 強意見、弱持有——但「弱持有」必須事先寫下：「如果我看到 ＿＿，我就承認錯了。」 寫不出那個 ＿＿，你的「弱持有」就是假的，你的意見其實是 ch04 講的「怎樣都對」。

這個補丁把一句容易淪為自我安慰的格言，釘回了可證偽的硬地面。

## 脊椎透鏡：如果塞麥爾維斯是你團隊的工程師

最後一次（在 ch18 總對帳前）戴上透鏡，回看脊椎的塞麥爾維斯洗手案——這次換個玩法。前面每一章都問「當時的他做對／做錯了什麼」，本章問一個假設性的問題：**如果塞麥爾維斯今天是你團隊的後端工程師，他這次「洗手降死亡率」的調查，在 code review／incident review 裡會拿到什麼評價？**

```text
塞麥爾維斯的 incident review（用工程師的科學循環逐步打分）

  循環步驟        他做了什麼                          評價
  ─────────────────────────────────────────────────────────
  1 觀察/量測     兩病房死亡率系統性比較               ✅ 漂亮
                  （98.4‰ vs 36.2‰，不靠軼事）         （ch01：教科書級的量測）

  2 假說/溯因     從 Kolletschka 之死跳到               ✅ 漂亮
                  「屍體微粒經手傳染」                  （ch03：IBE 的範本）

  3 可被推翻      「如果是它，洗手後死亡率該降」         ✅ 漂亮
    的預測        ——明確、可被打臉                     （ch04：冒了風險）

  4 對照/實驗     單雙日交替分配 ≈ 準隨機對照；          ✅ 大致漂亮
                  洗手令 = 介入                        （ch05：缺雙盲，但對照硬）

  5 證據更新      洗手後死亡率確實劇降                  ✅ 強證據
                  （1846 約 13% → 1848 約 1.27%）       （ch07：後驗大幅抬高）

  6 複製          調職後在別處也重現了降幅              ✅ 有複製
                                                       （ch13：濾掉了僥倖）

  7 機制          說不出「為什麼」                      ❌ 卡死在這裡
                  （無細菌理論，Pasteur/Koch 之後才有） （ch14：被否決的關鍵）

  8 結論/溝通     發表慢、文筆與性情不利說服            ❌ 致命傷
                  把否決他的人罵成殺人兇手              （ch08：制度溝通全盤皆輸）
```

把這張表唸一遍，你會發現一件讓人發涼的事：**塞麥爾維斯這次 incident review，技術調查部分（第 1–6 步）幾乎是滿分的——比你我大多數真實 incident 做得都好。** 他輸在第 7 步（機制）和第 8 步（結論的溝通與制度說服）。

這對一個工程師有極其具體的教訓：

- **第 7 步的教訓**：你修好一個生產問題，指標漂亮、複製過了——但如果你說不出**為什麼**（「重啟就好了，不知道為啥」），你其實只走到第 6 步。你敢不敢關 ticket？塞麥爾維斯敢，結果他被質疑「你連為什麼有效都說不清，憑什麼要我們改流程」。**機制是你從「碰巧有效」升級到「我真的搞懂了、它不會復發」的那張門票**（ch14 也誠實講過：有些東西先有效、後懂機制是常態，黑箱有效不該被輕視——但你至少要**知道自己缺機制**，把它標成風險，而不是假裝搞定了）。
- **第 8 步的教訓**：這是塞麥爾維斯最被低估、也最該被工程師記住的一課。他的科學做得近乎完美，**卻死在溝通與制度**——他不發表、發表了文筆又差、被質疑就破口大罵。一個 incident 調查做得再漂亮，如果你寫不出讓 reviewer 信服的 post-mortem、如果你把不同意你的人當敵人，你的正確結論**一樣會被否決**。科學的最後一哩不是「我對」，是「我能讓這個分工信任網裡的其他人，基於可檢驗的證據，理性地接受我對」（ch08 社群、ch15 證詞）。塞麥爾維斯是「對的科學、輸的溝通」的永恆紀念碑。

**守住誠實版**（全書紅線）：別因為這張表前六步給了滿分，就把他重新捧成「發現細菌的英雄」。他沒有細菌理論、他缺機制、他被否決、他憂鬱而終。他是一個**做對了大半科學循環、卻栽在機制與制度溝通上**的工程師——而這恰恰比「孤膽英雄」的版本，對你我有用一百倍。因為你我每天面對的，正是「技術上我可能對，但我說得清機制、講得服團隊嗎」這個真實難題。

## 科學態度何時會過頭：賭注 vs 成本

到這裡我得踩一下煞車，否則本章會變成另一種教條——「凡事都要走完八步循環、凡事都要 RCT」。**那是錯的，而且是一種偽裝成嚴謹的不嚴謹。**

科學態度是有成本的：做對照要時間、複製要重跑、追機制要深挖。對一個**賭注很小、容易回滾**的決定，硬套全套循環是浪費，甚至是逃避決策的拖延。判準很簡單，就一條軸：

```text
這個決定值得多少防自欺成本？——看賭注 × 不可逆性

  ┌─ 賭注大 + 難回滾  →  走完整循環，要對照、要複製、要機制
  │
  │   例：資料庫 schema 大遷移、計費邏輯改動、
  │       不可逆的資料刪除、影響全站的架構決策
  │   → 值得 A/B test、值得漸進式放量、值得預註冊式紀錄、
  │     值得追到機制才上
  │
  ├─ 賭注小 + 易回滾  →  快速試、看結果、錯了回滾就好
  │
  │   例：調一個 UI 文案、改個 log level、
  │       試一個容易切回的快取參數
  │   → 別開 RCT，直接上 feature flag 試，
  │     壞了關掉。做對照的成本 > 猜錯的成本
  │
  └─ 校準＝把嚴謹花在賭注大、不可逆、難察覺自欺的地方
```

這條判準本身就很科學——它是**把資源花在最可能騙到你、且騙到你代價最大的地方**。對一個五分鐘能回滾的文案改動做嚴格對照，跟對一個不可逆的計費遷移「憑感覺就上」，是同一種錯誤的兩面：**前者是把防自欺成本花錯地方，後者是該防的時候沒防。** 成熟的科學態度不是「永遠最嚴謹」，是**校準**——在賭注大、不可逆、難察覺自欺的地方加倍嚴謹，在賭注小、可回滾、反饋快的地方果斷快試。

> 一個容易被忽略的不對稱（焦慮工程師的提醒）：「易回滾」這個判斷本身常常是**自欺的重災區**。你以為某個改動「壞了關掉就好」，但它可能已經寫髒了資料、已經讓下游快取了錯誤狀態、已經觸發了不可逆的外部副作用（發出去的通知、扣掉的款）。**在把一個決定歸到「賭注小、易回滾」那一格之前，先用 ch16 的檢查表問一句：它真的可逆嗎？我憑什麼相信它可逆？** 把「可逆」當成不證自明的免死金牌，是這條判準最危險的漏洞——這也正是 ch02 理論負載的回聲：你「看到」的「這個改動很安全」，已經被你「希望它安全」的預期塗過色了。

這條「賭注 vs 成本」的校準，把全書從「科學態度真好，凡事都該如此」這種廉價結論，拉回到一個工程師真正用得上的位置：**防自欺是一種要分配的資源，不是一種要無差別崇拜的姿態。** 賭注大小決定你該花多少。決策的賭注與期望效用怎麼正式權衡，是另一本書的主場（見《決策與理性》談賭注大小／期望效用的那章）；本章只給「大賭注加嚴、小賭注快試」這個工程師夠用的直覺。

## 直覺的陷阱

把全書搬回自己系統時，最常見的翻車不在「不懂科學態度」，而在「自以為已經有了」。守住下面的正確版。

| 陷阱（你會這樣想） | 它在哪一步帶你下溝 | 正確版 / 怎麼自我察覺 |
|---|---|---|
| **「我很客觀／我是數據驅動的」**（最危險） | 你以為宣告客觀就等於做到客觀，於是放下戒心——而戒心正是客觀的唯一來源 | 「我很客觀」這句話本身是警訊（ch11）。真正在防自欺的人不會宣告自己客觀，會宣告「我這裡可能在騙自己，我設了哪些閘來抓自己」。自我察覺：當你聽到自己（或別人）說「我純粹看數據」，立刻問「哪些數據被你略過了、為什麼略過」 |
| **「我跑過實驗了 / 我做了 A/B test，所以我嚴謹」** | 你把「做了實驗」這個動作當成嚴謹本身，忽略實驗可能偷看、可能 p-hack、可能根本沒對照 | 做實驗 ≠ 做對實驗（ch05/ch09）。問：有沒有偷看著達標就停？有沒有同時看了十個指標挑一個顯著的？對照組是不是只差你關心的那一個變因？「做了實驗」是必要不是充分 |
| **「科學循環＝每件事都走完八步」** | 你把校準誤當教條，對小賭注決定也硬套全套，於是用嚴謹當拖延的藉口 | 嚴謹是要校準的（上一節）。賭注小、易回滾的事果斷快試；把防自欺成本花在賭注大、不可逆、難察覺的地方。對五分鐘能回滾的事做 RCT，跟對不可逆遷移憑感覺上，是同一種不嚴謹 |
| **「強意見、弱持有——我很開放」** | 你用「弱持有」當卸責擋箭牌：平常最嘴硬，出事就說「我本來就弱持有嘛」 | 弱持有必須事先寫下「什麼證據會讓我改變主意」（ch04 可證偽）。寫不出那個觸發條件，你的「弱持有」是假的、你的意見其實怎樣都對 |
| **「我有 root cause 了，所以結案」** | 你把「找到一個說得通的解釋」當成「找到真正的根因」，忘了溯因不保真 | 你的 root cause 是溯因出來的最佳候選，不是被證明的真兇（ch03）。問：有沒有競爭解釋我沒排除？我有沒有做可否證的驗證（不是只有「這樣解釋得通」）？「解釋得通」是最低標，不是結案線 |

具名陷阱，展開最危險的那個——**「我很客觀」自證式幻覺**。為什麼它最危險？因為它是個**自我關閉的迴路**：你越相信自己客觀，就越不去檢查自己的偏心，於是越容易被偏心帶著走，而每一次被帶著走你都會解釋成「我只是客觀地跟著數據」。塞麥爾維斯案裡那些否決他的醫師，沒有一個覺得自己不客觀——他們真心相信自己是依據主流醫學知識在做專業判斷。**動機性推理最毒的地方，就是它讓你感覺不到它**（心理機制見《思考的陷阱》談確認偏誤／動機性推理的那章；本書講的是制度與習慣怎麼防它）。怎麼自我察覺？把「我很客觀」這個念頭本身當成觸發器：每次它冒出來，強迫自己回答一個具體問題——「如果我錯了，我會在哪裡先看到？我有沒有真的去那裡看？」 一個真在防自欺的工程師，手上永遠有一張「我可能在哪裡騙自己」的清單；一個宣告自己客觀的工程師，手上是空的。**這張清單空不空，比你的職級、比你多聰明，更能預測你會不會栽。**

## 紙上推演

下面三題，純紙筆。先自己想，再讀解答。

### 第 1 題（★★）**[20 分鐘]**：把一次效能優化重述成完整的科學循環

挑一次你真正做過的效能優化（沒有的話用這個：某 API 的 P99 從 800ms 經你調整後降到 200ms，你當時加了一個資料庫索引）。把它逐步映射到本章的八步循環（觀察→假說→可推翻的預測→實驗→證據更新→複製→機制→結論），並在每一步標出「這一步在堵哪一種自欺」。然後誠實回答：你當時**實際上**跳過了哪一步？那一步本來防的是什麼？

### 推演解答

以「加索引降 P99」為例，完整循環長這樣：

```text
步驟          你做的事                        堵的自欺
─────────────────────────────────────────────────────────
1 觀察        拉 profiling，確認某 query        印象當事實（ch01）
              佔了 P99 的大頭，不靠「我覺得」
2 假說        溯因：這條全表掃描是主因          ——（這是生成步，不堵自欺，
              （從症狀跳到最像的嫌疑犯）         但要記得它不保真，ch03）
3 可推翻預測  「加索引後 P99 應降到 <200ms，     怎樣都對的萬能解釋（ch04）
              且 RDS CPU 跟著降」——可被打臉
4 實驗        staging 上只加這一個索引、         歸因錯亂——同時改多項
              別的不動，跑同一批壓測             指標動了不知是誰（ch05/ch06）
5 證據更新    P99 真降到 180ms、CPU 70%→40%     ——強證據抬高後驗（ch07）
6 複製        換時段、多跑幾輪，確認不是          僥倖／單次顯著（ch09/ch13）
              某次剛好流量低
7 機制        索引讓掃描行數從百萬降到幾百，      黑箱有效當真懂（ch14）
              講得通「為什麼」降這麼多
8 結論        暫定結論：「在測過的流量形狀下，    把暫定當鐵證（ch10）——
              這個索引把 P99 壓到 200ms 內」     留著「沒測過的尖峰」這條小字
```

**最常被跳過的兩步，幾乎都是第 4 步和第 6 步。**

- 跳過第 4 步（對照）：你常常是「加了索引、順手又調了連線池、還改了個超時設定」，然後 P99 降了，你歸功給索引——但你**同時改了三個東西**，到底哪個讓指標動的？這是 ch05/ch06 的歸因錯亂。防法：一次只改一個，或至少分開上線、分開量測。
- 跳過第 6 步（複製）：你「在 staging 跑一次降了就上線」，沒換時段、沒多跑——萬一那次剛好是低流量時段，你看到的「降」可能有一半是 flaky。這是 ch09/ch13 的單次結果不算數。防法：多跑幾輪、換條件。

**常見錯路**：以為「我有量測、有上線後看指標，就是科學的」。量測和事後看指標只是第 1 步和第 5 步——真正把這變成科學的，是中間那個**可被推翻的預測（第 3 步）**和**對照（第 4 步）**。少了它們，你只是在「改了 → 動了 → 宣布是我改的功勞」，這跟塞麥爾維斯時代「放血了 → 病人好了 → 放血有效」是同構的——少了對照，你分不清是你的介入有效，還是它本來就會好（回歸均值，ch06）。

### 第 2 題（★★★）**[25 分鐘]**：找出你跳過的那一步、判斷哪些決定值得做對照

(a) 拿你手上一個**正在進行**的技術判斷（例如：要不要把某服務從 REST 改成 gRPC、要不要換個快取策略、某個慢的批次任務該怎麼優化）。把它套進八步循環，找出你**還沒做**的那一步，並說清楚那一步本來防的是哪種自欺。

(b) 然後做校準判斷：這個決定的賭注有多大、有多難回滾？根據「賭注 × 不可逆性」，它值得走完整循環嗎，還是該快速試？給出你的判斷和理由。

### 推演解答

這題沒有標準答案（每個人手上的判斷不同），但好的解答要展現兩個能力，我用「REST 改 gRPC」當示範。

**(a) 找出跳過的步驟。** 假設你現在卡在「我覺得 gRPC 會更快（假說），所以想全面改」。對照八步循環，你大概跳過了第 3、4、6 步：

- 跳過第 3 步（可推翻的預測）：你說「gRPC 會更快」——更快多少？哪個指標？什麼條件下？這話**怎樣都對**（總能事後找個它快的場景）。誠實版要寫成「在我們這個 payload 大小與 QPS 下，gRPC 的 P99 該比 REST 低至少 X%，否則不值得改」——這才可被打臉（ch04）。
- 跳過第 4 步（對照）：你打算「改完一起上、看整體指標」——但你同時會換掉序列化、連線複用、可能還順手改了重試邏輯。指標動了你歸因不到 gRPC（ch05）。防法：先在一個邊界清楚的小服務上、只換協定、別的不動，做對照量測。
- 跳過第 6 步（複製）：一次 benchmark 漂亮就決定全面改——沒換負載形狀、沒測尾延遲在高併發下的表現（ch13）。

**(b) 校準判斷。** 「全面把 REST 改 gRPC」這個決定：賭注**大**（牽動所有呼叫方、客戶端、可能的 infra）、回滾**難**（改了協定，呼叫方都得跟著改回去，不是關個 flag 的事）。落在「賭注大 + 難回滾」那一格——**值得走完整循環**：值得先在一個小服務上做乾淨對照（只換協定）、值得複製驗證（多種負載）、值得追機制（到底是序列化省了、還是連線複用省了，這決定了它在你別的服務上會不會一樣有效）。

對比一下：如果你的決定是「把某個快取 TTL 從 60 秒調到 300 秒、用 feature flag 控制、隨時能切回」——賭注**小**、回滾**易**（關 flag 即可，且不寫髒資料）。這就**不值得**開全套循環，直接小流量試、看命中率與資料新鮮度、不對勁就切回。

**常見錯路一**：對所有事一視同仁地嚴謹，結果連調個 TTL 都要寫實驗計畫，把自己拖死，最後反而因為「太累」而在真正的大決定上偷懶。**常見錯路二**（更危險）：把「我以為易回滾」當成真易回滾。先用上一節的提醒檢查——這個改動真的可逆嗎？它會不會寫髒資料、汙染下游快取、觸發不可逆副作用？「TTL 調長」看似可逆，但若這 300 秒內下游基於過期資料做了不可逆的動作（發了錯的通知、扣錯了款），它就不可逆了。**把「易回滾」當不證自明的免死金牌，是這題最隱蔽的陷阱**（ch02 理論負載＋ch16 檢查表）。

### 第 3 題（★★）**[15 分鐘]**：拆穿一句「弱持有」的假開放

你的資深同事在架構評審裡說：「我對用 Kafka 是強意見、弱持有的——有更好的證據我隨時改。」 但你觀察到：過去半年每次有人質疑這個選擇，他都能當場找出一套理由反駁，從沒真的動搖過。請問：(a) 他的「弱持有」是真的還是假的？怎麼判斷？(b) 要怎麼把他的「弱持有」變成可檢驗的、而不是卸責的擋箭牌？

### 推演解答

**(a) 高度可疑是假的，但要的不是看他「有沒有改過主意」，而是看他能不能回答一個問題。** 「從沒動搖」本身不直接證明他不開放——也許 Kafka 真的一直是對的選擇，那不動搖是合理的。真正的判準是 ch04 的可證偽性：**他能不能事先說出「什麼證據會讓我放棄 Kafka」？** 如果他答得出具體的、可觀察的觸發條件（「如果我們的訊息量半年內掉到每秒一千以下、或運維成本超過某個門檻、或團隊三個月沒人能上手，我就該換回更簡單的方案」），那他的「弱持有」是真的——他只是還沒看到那些條件。如果他答的是一堆模糊的「看情況啦」「有更好的我就換」卻說不出**怎樣算更好、好到哪就換**，那他的「弱持有」是假的：他的意見其實**怎樣都對**，因為任何質疑他都能臨場編個理由反駁掉，這正是 ch04 講的「不冒風險的萬能解釋」，也是 ch11 動機性推理的標準長相（他真心相信自己開放，卻系統性地護航既有結論）。

**(b) 把它釘到可檢驗的地面。** 方法就是逼出那個事先的觸發條件，並把它寫進文件：

```text
把「弱持有」變可檢驗

  假的弱持有：「有更好的證據我就改」
              （怎樣都對，事後任何結果都能解釋）
       ↓ 逼問
  真的弱持有：事先寫下「放棄 Kafka 的觸發條件」——
    + 量化門檻：訊息量、運維成本、團隊上手時間…
    + 可觀察：這些指標現在多少、半年後回來複查
    + 對稱：也寫下「什麼證據會讓我更堅定用它」
       ↓
  半年後拿這張表回來對：條件中了沒？
    中了卻不改 = 假持有暴露；
    沒中 = 不動搖是合理的，不是固執
```

**這跟預註冊（ch13）是同一個動作**：事先凍結「什麼結果會讓我下什麼結論」，堵掉「事後挑理由護航既有立場」這個漏洞。**常見錯路**：以為「弱持有」是個性問題（他人太固執）——不是，它是個**制度／紀錄問題**。再固執的人，只要事先把觸發條件白紙黑字寫下、約定到期複查，他的固執就無處藏身；再開放的人，沒寫下觸發條件，他的開放也只是自我感覺。把態度問題轉成可檢驗的紀錄，這正是全書「科學靠制度而非靠個人美德防自欺」的微縮版。

口頭自答，講得出來才算過：

1. **為什麼「中學那張科學方法五步驟流程圖」是誤導的？** 工程師的科學循環跟它最關鍵的三個差別是什麼？（提示：回頭箭頭、溯因獨立成步、複製與機制獨立成步。）
2. 八個可攜習慣裡，為什麼「先問我怎麼知道」和「對自己的客觀性保持懷疑」是地基、其餘六條是工具？少了這兩條地基，那六條工具會怎麼被繞過？
3. 「強意見、弱持有」要怎麼樣才不是一句自我安慰的格言？「弱持有」必須事先寫下什麼，它才跟 ch04 的可證偽接得上？
4. 為什麼「我很客觀／我是數據驅動的」這句話本身是個警訊？一個真在防自欺的工程師，手上會有什麼、宣告客觀的工程師手上是空的？
5. 如果塞麥爾維斯今天是你同事，他這次調查的技術部分（觀察到複製）評價如何？他栽在哪兩步？這兩步對你做 incident review 各有什麼具體教訓？

## 延伸閱讀

- **Wikipedia, "Rubber duck debugging"**（https://en.wikipedia.org/wiki/Rubber_duck_debugging）：本章「把你的調查過程顯式化、逐句講出來才會發現破綻」的最小可攜版。橡皮鴨除錯之所以有效，正是因為它逼你從「我覺得是這樣」切換到「我得把每一步講清楚」——那一切換就是 ch01 的「先問我怎麼知道」在最小尺度上運作。讀它對「為何說出口就抓到 bug」的解釋。
- **Paul Saffo / Bob Sutton 對「strong opinions, weakly held」的原始討論**（如 Bob Sutton 的部落格與訪談，網路上多處轉述，出處與確切歸屬有不同說法，2026-06）：本章「強意見、弱持有」的源頭。但**更該讀的是對它的批評**——搜尋「strong opinions weakly held is bad」（如 commoncog.com 那篇）會找到對「這句話常被當卸責擋箭牌」的反思，正好對應本章「弱持有必須事先寫下觸發條件」那個補丁。讀兩面，別只記勵志版。
- **Carl Hempel,《Philosophy of Natural Science》(1966)**：全書脊椎塞麥爾維斯案的科哲課本源頭（landscape 已驗證）。本章把它從「讀別人的案例」翻轉成「如果他是你同事」——回頭讀 Hempel 怎麼用這個案例講「假設檢定的基本步驟」，你會發現他拆的步驟，跟本章的工程師科學循環是同一套骨架，只是換了詞彙。
- 跨書：決定值得花多少防自欺成本（賭注大小與期望效用的正式權衡）——在**《決策與理性》談賭注大小／期望效用的那章**；本章只給「大賭注加嚴、小賭注快試」的工程師直覺，正式的權衡框架在那裡。「我很客觀」為何是幻覺、動機性推理的心理機制——在**《思考的陷阱》談確認偏誤／動機性推理的那章**；本章講的是怎麼用習慣與紀錄防它，不是它的心理成因。
- 跨章：本章是全書工具的組裝場，每個習慣的完整論證在它的出處章（可證偽 ch04、對照 ch05、因果 ch06、貝氏 ch07、機制 ch14）。下一章 ch18 會用脊椎塞麥爾維斯案把這一切做最後總對帳——本章的「如果他是你同事」那張評分表，正是 ch18 十層對帳的暖身。
