# ch06 — 相關不是因果：混淆變因與它的工程既視感

> **本章解決什麼問題**：ch05 立起了科學的引擎——對照實驗，並說了「隨機化」是二十世紀最被低估的發明之一。但它沒回答一個更前面的問題：**為什麼我們需要對照？到底是哪一種自欺，逼得人類發明出隨機分組這種麻煩東西？** 答案是一個你每天在 dashboard 上都會撞到的幽靈——**兩件事一起動，不代表一件造成另一件**。本章專講相關（correlation）為什麼推不出因果（causation）：混淆變因（confounder）、反向因果、純巧合這三個陷阱，它們的結構，以及為什麼 ch05 的隨機化恰恰是切斷混淆的利器。這是 Part II「設計的防線」的中段——ch05 給工具，本章給「工具在防的那個鬼」。

## 從你已知的出發

你上線了一版新快取。隔天看 dashboard，錯誤率掉了。你在 standup 上說「新快取讓錯誤率降了 30%」，大家鼓掌，ticket 關掉。

停一下。你怎麼知道是快取？

你會的，因為你被這種事咬過。那天你同時還做了別的事：上線時間正好卡在週五晚上、流量比平日低；同一個 PR 裡順手把一個逾時時間從 1 秒調到 3 秒；而且運維那邊昨天剛把一台一直在抖的機器換掉。錯誤率掉了 30%，**到底是哪一個**？快取？流量低？逾時改了？換機器？還是這四個的某種組合？

這就是本章的全部內容，只是換了件衣服。你觀察到兩件事一起動了——「上了快取」和「錯誤率降」——你的大腦**立刻**想把前者當成後者的原因。但「一起動」（相關）和「一個造成另一個」（因果）是兩回事，中間隔著一條深溝，而這條溝裡住著三種專門騙你的東西。

工程師對這件事其實有很好的直覺，比一般人好得多。你天天在跟它搏鬥：

- **相關性故障**的反面——CloudWatch 上兩條曲線一起飆，你不會馬上說 A 害了 B，你會先懷疑「是不是有個共同的上游把兩個都拖下水了」。那個「共同的上游」，就是本章的主角，**混淆變因**。
- **「部署後 P99 升高」**——你被教過不要馬上甩鍋給部署，因為部署時間常常正好撞上流量尖峰。流量尖峰同時造成「你部署了」（你挑離峰部署但有時撞上）和「P99 升高」，它就是混淆變因。
- 做 root cause 時你最怕的那句「**會不會只是碰巧**」——那是純巧合在敲門。

本章要做的，是把你這份模糊的維運直覺**磨成一把能講出來的刀**：相關推不出因果，到底是哪三種結構在搞鬼；為什麼 ch05 的隨機分組能一刀切斷其中最陰險的那種；以及——回到脊椎——塞麥爾維斯當年是怎麼把「洗手與死亡率下降的相關」，一步步論證成「洗手**導致**死亡率下降」的，而他又躲過了哪些混淆質疑。

## 相關推不出因果：三種陷阱，逐一拆開

先把話說清楚，免得矯枉過正。「相關不是因果」這句口號被講爛了，爛到很多人以為它的意思是「相關沒用」。**不是。** 相關是因果的線索，沒有相關，連嫌疑犯都湊不齊。問題不在相關沒用，問題在**從「相關」跳到「因果」這一步，有三個地方會讓你摔下去**。

```text
你看到的：X 和 Y 一起動（相關）
你想跳到的：X 造成 Y（因果）

  這一跳，有三種摔法：

  陷阱一  共同原因（confounder）
          有個第三者 Z，同時造成 X 和 Y。
          X 和 Y 一起動，但彼此沒有因果關係，
          它們只是同一個爸爸生的。
                Z
              ↙   ↘
            X       Y        ← X、Y 相關，但 X 不導致 Y

  陷阱二  反向因果（reverse causation）
          確實有因果，但方向反了——是 Y 造成 X，
          不是 X 造成 Y。
            X  ←  Y          ← 你以為 X→Y，其實 Y→X

  陷阱三  純巧合（coincidence）
          根本沒有任何因果關係，
          兩條曲線只是剛好一起動。
          資料一多、變數一多，這種「假相關」必然出現。
            X       Y        ← 之間什麼線都沒有
```

逐一來。

**陷阱一：共同原因（混淆變因）。** 這是三者裡最重要、最陰險、也是科學方法花最大力氣去防的一種，所以本章後面整節都在講它。先給直覺：兩件事一起動，常常不是因為一個推了另一個，而是因為**背後有第三個東西同時推了它們兩個**。

教科書級的例子，冰淇淋。某城市的資料顯示：冰淇淋銷量越高，溺水死亡人數越多，相關性扎實得很（2026-06：這是統計教材的標準範例，數字隨教材而異，當示意看）。難道吃冰淇淋會讓人溺水？當然不是。背後有個第三者：**氣溫**。天氣熱，大家買冰淇淋；天氣熱，大家去游泳，溺水跟著多。氣溫同時推高了冰淇淋銷量和溺水數，這兩者本身**沒有任何因果關係**，它們只是同一個夏天的兩個孩子。氣溫就是混淆變因。

把它翻成你的語言：「**部署後 P99 升高**」——你以為「部署 → P99 升」，但其實可能是「**週五晚上的流量尖峰**」同時造成了「你剛好那時在部署」和「P99 升高」。流量尖峰是混淆變因。你甩鍋給部署，真兇逍遙法外，下次離峰部署你還會再被咬一次。

**陷阱二：反向因果。** 因果是真的，但箭頭反了。經典的工程版：你發現「**重啟服務的機器，故障率特別高**」，於是得出「重啟導致故障」，差點下令禁止重啟。但想想：是不是**因為機器先出了故障，運維才去重啟它**？是故障導致重啟，不是重啟導致故障。箭頭反了。

醫學裡也滿地都是：「常去醫院的人死亡率高，所以醫院害死人」——其實是病得重的人才常跑醫院。反向因果的陰險在於，它**真的有因果**，只是方向錯，所以你做再多「確認 X 和 Y 真的相關」的功課都救不了你——X 和 Y 確實有因果關係，你只是搞反了誰是因。

**陷阱三：純巧合。** 沒有任何機制，兩條線就是剛好一起動。這在大數據時代特別危險，因為它是**數學上的必然**：你手上的指標越多、看的時間越長，總會有兩條曲線的形狀長得很像。有人專門收集這種「假相關（spurious correlation）」當笑話——美國某州的離婚率和某種食品的人均消費量逐年高度相關之類（2026-06：Tyler Vigen 的 *Spurious Correlations* 是這類例子的著名收藏，當趣味看）。

巧合的工程既視感：你 dashboard 上 30 個指標，總有兩個今天一起往上飄。你盯著它們說「欸這兩個是不是有關係」——多半沒有，是你看的東西太多了。這其實是 ch09 要講的「多重比較」的近親：**找的地方夠多，假訊號必然冒出來**。

這三個陷阱裡，反向因果靠「想清楚時序、想清楚誰先誰後」常常能擋掉；純巧合靠「要求一個說得通的機制、要求能複製」能擋掉（這是 ch14 機制、ch13 複製的伏筆）。**最難擋的是混淆變因**——它既有真實的統計相關，又有看似合理的故事，唯一能乾淨切斷它的，是 ch05 那個你可能還沒完全領會其威力的東西：隨機化。下一節專講它。

## 混淆變因：為什麼隨機分組是一把手術刀

把混淆變因的結構再釘死一次，因為這是本章——也是整個「為什麼需要對照實驗」——的核心。

```text
混淆變因的標準結構

           混淆變因 Z
          （共同原因）
         ↙           ↘
       X  ─ ─ ─ ─ ─ ─  Y
           (觀察到相關)

  你想知道的因果：X → Y 是否存在？
  你真正看到的　：Z 同時推動 X 和 Y，造成「假因果」相關。

  關鍵：只要 Z 存在且你沒控制它，
        X 和 Y 就會相關——即使 X 對 Y 毫無影響。
```

混淆變因之所以是三個陷阱裡最陰險的，在於它**同時滿足你判斷因果的兩個直覺檢查**：(1) X 和 Y 真的有統計相關（不是巧合）；(2) 「X 造成 Y」聽起來有道理（不像storks 送子那麼荒謬）。冰淇淋例子之所以是教學範例而不是真實災難，正因為「吃冰淇淋導致溺水」荒謬到你會停下來想。真正咬死人的混淆，是那些**故事講得通**的——「用了我們 SDK 的 App 留存率更高」、「上了快取錯誤率就降」——這些你不會本能起疑，於是直接信了。

現在來看 ch05 那個隨機化為什麼是手術刀。回到留存率的例子：

你觀察到「**裝了你們家 SDK 的 App，30 天留存率比沒裝的高 15%**」。你想說「我們 SDK 提升留存」。但混淆變因在哪？想想**誰會選擇裝你的 SDK**——多半是那些本來就比較認真、比較有資源、產品本來就比較好的團隊。「團隊認真程度」這個 Z，同時造成了「會去裝 SDK」（X）和「留存率高」（Y）。你的 SDK 可能一點用都沒有，相關照樣存在。

要切斷它，有兩條路。

**笨路（觀察法）：** 把你能想到的混淆變因一個一個「控制」住——只比較團隊規模相近、產品類別相同、融資階段一致的 App。這叫「分層」或「調整」。問題是：**你只能控制你想得到的混淆變因。** 那些你沒想到的（團隊的技術品味？產品經理的偏執程度？），照樣在背後搗亂。你永遠不知道自己漏了哪個。這就是觀察性研究的根本軟肋——也是 ch09 要講的，為什麼觀察數據做出來的結論複製率那麼慘。

**聰明路（隨機化）：** 不要讓團隊自己選裝不裝。**你來隨機指定**——丟硬幣，正面這批 App 裝 SDK，反面那批不裝。一旦是隨機分的，「團隊認真程度」這個混淆變因，在兩組之間會被**平均掉**：認真的團隊和混水摸魚的團隊，在裝 SDK 組和不裝組裡的比例會大致一樣。不只「團隊認真程度」，**所有你想得到和想不到的混淆變因，全部一起被平均掉**。這就是隨機化的魔法，也是 ch05 說它被低估的原因：

```text
隨機化為什麼能切斷「所有」混淆變因（包括你沒想到的）

  觀察法（誰自己選裝 SDK）：
    認真團隊 ─────────► 多半裝 SDK   ─┐
    混水團隊 ─────────► 多半不裝      ├─ 兩組「認真度」失衡
                                      │   → 留存率差，分不清是
                                      │     SDK 還是認真度造成的
                                     ─┘

  隨機法（丟硬幣決定誰裝）：
    認真團隊 ─┬─► 一半裝 SDK         ─┐
              └─► 一半不裝            │
    混水團隊 ─┬─► 一半裝 SDK         ├─ 兩組「認真度」平均相同
              └─► 一半不裝            │   → 留存率若仍有差，
                                      │     只剩 SDK 一個可能的解釋
                                     ─┘

  關鍵差別：隨機化不需要你「知道」混淆變因是什麼。
            它把已知的、未知的、你做夢都想不到的混淆，
            一起在兩組之間平均掉。這是它無可取代的地方。
```

我認為這是整本書最該停下來想十分鐘的一頁之一。觀察法只能擋你想得到的鬼，隨機化把所有鬼一起擋掉——**包括那些你這輩子都不會想到的鬼**。這就是為什麼隨機對照試驗（RCT）被當成因果證據的金標準，也是為什麼你的 A/B test（隨機分流使用者）比「上線後看指標有沒有動」（沒分流，全是混淆）強一個數量級。

順帶埋一個 ch07 的線：隨機化不是說做完一次就保證兩組完全一樣。樣本小的時候，運氣不好，硬幣可能把認真團隊大多分到一邊去。隨機化保證的是「**平均而言**平衡」和「失衡的程度可以被機率量化」——而「怎麼量化這個機率」是《馴服隨機》的活兒（見《馴服隨機》談隨機變數與獨立的那章），本書不展開，只用結論。

## Bradford Hill：當你不能做 RCT 時，怎麼小心地往因果靠

說了半天隨機化最強，但你心裡清楚——**很多時候你根本不能做 RCT**。

你不能為了證明「抽菸致癌」，隨機指定一萬個人去抽三十年菸（不道德）。你不能為了證明「某個架構決策拖垮了系統」，把生產環境隨機分一半去跑爛架構（不可能）。塞麥爾維斯也沒辦法真的設計一個乾淨的 RCT——他手上是一個已經在運轉的醫院。

那當沒有 RCT、只有觀察相關時，怎麼**負責任地**評估「這個相關有多可能是因果」？1965 年，統計學家布拉德福・希爾（Austin Bradford Hill）在一場演講裡給了一套思考框架，後世通稱「希爾判準（Bradford Hill criteria）」。

這裡要立刻誠實標一件事，因為它常被講錯（2026-06，已查證）：**希爾本人不叫它們「判準（criteria）」，他叫「視角（viewpoints）」**——「我們在喊因果之前，應該從這九個視角去審視一個關聯」。他還明白說過：這九個視角**沒有一個能單獨提供無可爭辯的因果證據，也沒有一個是必要條件**。所以把它們當成一張「打勾就算因果」的檢查表，是對它最常見的誤用。它們是「**讓因果更可信或更可疑的思考角度**」，不是判決書。

九個視角我不全展開（完整清單留給延伸閱讀），挑工程師最用得上、塞麥爾維斯案最相關的三個：

```text
Bradford Hill 三個最有用的視角（不是清單，是思考角度）

  ① 時序（temporality）—— 唯一接近「必要」的一條
     因必須在果之前。X 若在 Y 之後才發生，X 不可能是 Y 的因。
     工程版：你說「上了快取所以錯誤率降」，先確認——
             錯誤率是「上快取之後」才降的，
             還是上之前就已經在降了？（很多「效果」是上線前就開始的趨勢）

  ② 劑量反應（biological gradient / dose-response）
     劑量越大、效果越強，因果可信度提高。
     工程版：把快取命中率從 50% 拉到 90%，錯誤率降得更多嗎？
             如果「給多給少效果一樣」，反而可疑（像是別的東西在作用）。

  ③ 可逆／實驗（experiment）
     拿掉因，果跟著消失，因果可信度大增。
     工程版：把快取關掉，錯誤率彈回去嗎？
             這是你能做的最接近「介入」的檢查——
             它正是 ch05 對照實驗的精神：能開能關才好歸因。
```

注意這三個視角的共通點：它們都在**降低混淆變因的嫌疑**，但沒有一個能像隨機化那樣**徹底切斷**它。時序對了，混淆變因可能也滿足時序（氣溫確實在冰淇淋和溺水之前升高）；劑量反應有了，混淆變因也可能呈現劑量關係。希爾自己最清楚這點——這就是他堅持叫「視角」不叫「判準」的原因。**它們提高可信度，不提供保證。** 工程師該記住的一句話：當你不能隨機分流時，Bradford Hill 是你退而求其次的盡責清單，但「盡責」不等於「確定」。

## 脊椎透鏡：塞麥爾維斯怎麼把相關論證成因果

現在換上本章的透鏡，重看脊椎的塞麥爾維斯案（這是全書貫穿的 1847 年維也納洗手案，現在換「因果 vs 相關」透鏡看）。

塞麥爾維斯手上有一個扎實的**相關**：頒布氯化石灰洗手令之後，第一病房的產褥熱死亡率劇降（誠實的對照數字見下）。但「洗手之後死亡率降了」是相關。他憑什麼說是「洗手**導致**死亡率降」，而不是某個混淆變因？

先把誠實的數字擺出來，因為這本身就是本章該守的紀律（死亡率數字以全書基準為準，兩套表述別混用）：

```text
塞麥爾維斯案的誠實對照（別用挑選過的數字誇大效果）

  最乾淨的對照（同院同期兩病房，每千人版，1840–46 平均）：
    第一病房（醫師、醫學生接生）   98.4‰
    第二病房（助產士接生）         36.2‰   ← 約 2.7 倍差
    兩病房病人按「單雙日交替」分配 ≈ 天然的準隨機分組

  介入前後（全年 vs 全年，百分比版）：
    1846 第一病房全年   約 13%
    1848 第一病房全年   約 1.27%（同年第二病房 1.33%，兩病房拉平）

  ⚠ 不要用「18.27% → 1.27%」這種講法。
    18.27% 是「1847 年 4 月單月」（洗手令 5 月中才下、接近峰值），
    不是 1847 全年。拿單月峰值對比 1848 全年 = 用挑選過的切片誇大效果。
    （這個「連防自欺脊椎都可能被挑數字誇大」的諷刺，ch09 會專門講。）
```

數字守住了，回到因果論證。塞麥爾維斯能把這個相關推向因果，靠的正是這一章講的幾把工具——他當時沒有這些名詞，但做的就是這些事：

```text
塞麥爾維斯如何把「相關」論證成「因果」（本章透鏡逐欄拆解）

  工具               他手上的證據                     擋掉的陷阱
  ─────────────────────────────────────────────────────────────
  準隨機對照         單雙日交替分病房，               大幅削弱混淆變因
  （ch05）           病人無從自選 → 兩病房病人         （兩病房的病人
                     條件大致可比                       條件被「分」平了）

  時序               洗手令在前，死亡率驟降在後         反向因果
  （Hill 視角①）     （不是死亡率先降才開始洗手）       （不是「先變好
                                                         才去洗手」）

  劑量/可逆          洗手執行嚴，死亡率低；             削弱純巧合
  （Hill 視角②③）   鬆懈時回彈 → 開關對得上            （介入可開可關，
                                                         效果跟著動）

  機制候選           Kolletschka 之死溯因出的           給因果一個故事
  （ch03 溯因）      「屍體微粒經手傳染」               （但這故事不完整，
                                                         見下）
```

看清楚了——按本章的標準，塞麥爾維斯的因果論證其實**相當扎實**：他有接近隨機的對照（單雙日交替，削弱混淆）、時序對（洗手在前、死亡率降在後，排除反向因果）、介入可逆（鬆懈就回彈，排除巧合）。如果今天有人拿這套證據來說服你某個快取真的有效，你會信。

那他為什麼還是被否決了二十年？

這就是脊椎案最痛、也最重要的一刀——**他的因果論證在「對照」這一軸幾乎滿分，卻栽在另一軸：機制**。當時可能的混淆質疑（病房通風不同？病人本來體質不同？醫學生比較粗暴？）他大多能回應，但他說不清「屍體微粒到底**為什麼**會致病」——細菌致病說要到他死後，巴斯德（Louis Pasteur）和科赫（Robert Koch）在 1860–80 年代才立起來。在反對者眼裡，一個「相關很強、時序也對、但講不出機制」的宣稱，**永遠可以被懷疑成某個還沒被發現的混淆變因**。

這是本章留給你的最深一層：**對照能切斷你想得到的混淆，但「沒有機制」會讓人永遠懷疑還有一個你沒切斷的混淆。** 切斷已知混淆（對照）和解釋為什麼有效（機制）是兩件事，缺了後者，前者的說服力會被打折。塞麥爾維斯缺的那塊機制，是 ch14 的主角；他明明對照做得漂亮卻被整個醫界否決，那是 ch08「科學是一群人」的主角。本章只負責讓你看清：**他不是輸在相關不夠強、也不是輸在沒切混淆——他輸在沒有機制這條退路被人留著。**

## 一句話的 do：觀察和介入是兩種不同的問題

最後埋一個你之後可能會撞到的詞，一句話帶過。

統計學家朱迪亞・珀爾（Judea Pearl）在 1990 年代發展了一套「因果推論」的形式語言，核心區分是：**「觀察到 X」和「介入去設定 X」是兩個根本不同的問題**。他用 `P(Y | do(X))`（讀作「對 X 動手」）來表示後者，跟一般條件機率 `P(Y | X)`（單純觀察到 X）區分開——你做 A/B test「隨機指定一批人裝 SDK」就是 `do`，你「看哪些人自己裝了 SDK」就只是觀察。本章整章其實都在講這兩者的鴻溝，珀爾給了它一套可計算的數學。**這套形式理論（do-calculus、因果圖、後門準則）本書不展開**，有興趣往因果推論的專書去（見延伸閱讀）——你只要先記住那個直覺：**「看到」和「動手」是兩回事，混淆變因正是讓這兩者裂開的那道縫。**

## 直覺的陷阱

本章的概念是「想當然耳」的重災區——人類的大腦天生是台「找因果」的機器，看到兩件事一起發生，幾乎是反射性地腦補一條箭頭。以下是最容易把你帶溝裡的幾個。

| 陷阱 | 錯誤直覺 | 它在哪一步害你 | 怎麼自我察覺 |
|---|---|---|---|
| **把相關當因果** | 「A 和 B 一起動，A 應該造成了 B」 | 跳過「有沒有第三者 Z 同時推兩個」這一問 | 問自己：「有沒有一個東西，能同時解釋 A 和 B？」想不出來才往因果靠 |
| **「相關沒用」的矯枉過正** | 「既然相關不是因果，那相關都是噪音」 | 把線索當垃圾丟掉 | 相關是嫌疑犯名單，沒它連查都沒得查；它有用，只是不能直接定罪 |
| **Hill 視角當打勾清單** | 「九條中了七條，所以是因果」 | 把「思考角度」當「判決標準」 | 記住希爾原話：沒有一條是必要或充分的；混淆變因可能讓你中很多條 |
| **控制了我想到的，就乾淨了** | 「我把規模、地區都調整了，剩下的就是真因果」 | 漏掉「你沒想到的混淆變因」 | 觀察法只能擋已知的鬼；要擋未知的鬼，只有隨機化 |
| **反向因果看不出來** | 「X 和 Y 強相關，X 一定是因」 | 沒問「會不會是 Y 造成 X」 | 對每個因果宣稱反過來念一遍：「會不會其實是果先發生？」 |

**具名陷阱：「控制了混淆變因」的假乾淨。** 這是最危險的一個，因為它**偽裝成嚴謹**。一個研究者（或你）在觀察數據上「調整」了年齡、性別、地區、收入……列了長長一串控制變因，看起來無懈可擊。但這份嚴謹有個藏起來的前提：**你控制的，只能是你想得到、而且量得到的東西。** 任何一個你沒想到的混淆變因（或想到了但沒數據的），照樣在背後把相關撐起來。

這就是為什麼「我控制了一堆變因，所以這是因果」這句話，比「我做了 RCT，所以這是因果」**弱一個檔次**——前者擋的是「你想到的鬼」，後者擋的是「所有鬼」。經典的教學例子是「搬到佛羅里達的人較多得阿茲海默症，所以佛州氣候致病」（2026-06：這是統計教材常用的示意例，當教學素材看）——真正的混淆是**年齡**：退休的老人才搬去佛州，老人本來就較易失智。如果你的控制變因清單裡漏了年齡，你那套「我調整了氣候、濕度、飲食」的嚴謹，全是白工。自我察覺的方法只有一個：**每次你說「我已經控制了混淆變因」，立刻補一句「我沒控制到、甚至沒想到的，會是什麼？」** 答不出來，就還沒乾淨。

## 紙上推演

以下題目純紙筆推演，不需要跑任何程式。請先自己想，再看解答。

---

**推演題 1（★★，[15 分鐘]）：揪出留存率宣稱裡的混淆變因。**

某家分析平台宣稱：「**使用我們 SDK 的 App，30 天留存率平均比不使用的高 18%。所以接入我們 SDK 能提升留存。**」

(a) 指出最可能的混淆變因（提示：誰會「選擇」去裝這個 SDK？）。
(b) 畫出這個宣稱的因果結構圖（用本章的 Z→X、Z→Y 形式），標明哪條箭頭是平台希望你相信的、哪條是實際在起作用的。
(c) 設計一個能切斷這個混淆的對照，說明關鍵在哪一步。

---

**推演題 2（★★★，[20 分鐘]）：用 Bradford Hill 視角體檢一個你信過的因果宣稱。**

挑一個你過去在工作中相信過、並據以做了決策的因果宣稱（例如「加了這層快取所以延遲降了」「換了這個 GC 參數所以停頓變短了」）。用本章講的三個 Hill 視角——**時序、劑量反應、可逆/實驗**——逐一審視它。

(a) 這三個視角各自，你當時手上的證據滿足嗎？
(b) 對於你「沒滿足」或「沒檢查」的那些視角，提出一個能補上的具體檢查。
(c) 誠實回答：就算這三個視角全滿足，你能排除所有混淆變因嗎？為什麼不能？

---

**推演題 3（★★，[12 分鐘]）：分辨三種陷阱。**

下面三個「相關」各屬於本章哪一種陷阱（混淆／反向因果／純巧合）？說明理由。

1. 「我們發現：被 on-call 工程師手動重啟過的服務，後續一週故障率特別高。」
2. 「過去半年，公司的咖啡豆採購量和線上轉換率高度相關。」
3. 「資料顯示：寫了較多單元測試的專案，線上 bug 較少。」

---

### 推演解答

**第 1 題：**

(a) 最可能的混淆變因是「**團隊本身的品質／資源水準**」——把它叫做 Z。本來就認真、有資源、產品本來就好的團隊，(i) 比較會去評估並接入第三方 SDK，(ii) 留存率本來就高。Z 同時造成「裝 SDK」和「高留存」。

(b)
```text
              Z = 團隊品質/資源
            ↙                    ↘
   X = 裝了 SDK  ─ ─ ─ ─ ─ ─ ─  Y = 高留存
                  (觀察到 +18% 相關)

   平台希望你相信：X ──────► Y   （SDK 提升留存）
   實際在起作用的：Z 同時推 X 和 Y（高留存其實來自團隊本身）
                   X→Y 可能根本是 0
```

(c) 切斷混淆的乾淨對照＝**隨機分流**：在一批條件可比的 App 裡，**由你隨機指定**哪些裝 SDK、哪些不裝（而不是讓團隊自選）。關鍵在「**不讓團隊自己選**」這一步——一旦是隨機分的，「團隊品質」這個 Z 在兩組之間被平均掉，若還剩留存差，才可歸因於 SDK。常見錯路：以為「找一批沒裝 SDK 的 App 來比」就夠了——不夠，因為那批「沒裝的」在團隊品質上系統性地跟「裝了的」不同，混淆還在。

**第 2 題：**

這題沒有標準答案，重點在**審視的紀律**，不在結論。一份及格的解答長這樣：

(a) 假設宣稱是「加了快取所以延遲降了」。
- **時序**：延遲是「上快取之後」才降的，還是上之前就在降的趨勢？——很多人沒檢查，直接看「上線後延遲低」就信了，但延遲可能是流量自然回落造成的既有趨勢。
- **劑量反應**：快取命中率高的時段，延遲降得更多嗎？如果「命中率高低延遲都一樣」，反而可疑。
- **可逆/實驗**：把快取關掉（或 A/B 分流），延遲彈回去嗎？這是最強的一條，但也最少人做。

(b) 沒滿足的補法：例如時序沒查，就去調出上線前後各兩週的延遲曲線，看降是不是真的卡在上線那一刻；可逆沒驗，就做一次漸進式放量（分流一部分流量），繞過快取，直接對比。

(c) **不能排除所有混淆。** 因為這三個視角再漂亮，混淆變因都可能同時滿足它們——例如「上線那天也正好換了一批更快的機器」，這個混淆變因同樣會呈現「時序對（換機器在前、延遲降在後）」「可逆（換回舊機器延遲就回升）」。Hill 視角提高可信度，但只有隨機化能徹底切斷未知混淆，而你這裡沒做隨機化。

**第 3 題：**

1. **反向因果**。不是「重啟導致故障」，而是「**服務先出問題，on-call 才去重啟它**」。故障在前，重啟在後，箭頭是 故障→重啟，你看到的是它的尾巴。常見錯路：真的去推「禁止重啟」，結果什麼也沒改善，因為動錯了方向。
2. **純巧合（或極弱的混淆）**。咖啡豆採購量和轉換率之間，找不到一個說得通的因果機制，也很難想出一個合理的共同原因（硬要扯「公司擴張」當 Z 也很牽強）。半年的兩條曲線剛好一起動，多半是巧合——指標看夠多，總有兩條對得上（ch09 多重比較的近親）。判別關鍵：**講不出機制、也講不出合理的共同原因**，就高度懷疑是巧合。
3. **混淆變因**。「寫較多測試」和「bug 較少」之間，很可能有個共同原因 Z＝「**團隊的工程紀律/資深程度**」——紀律好的團隊既寫較多測試、也寫出較少 bug 的程式。寫測試本身或許有用，但這個觀察相關**高估**了它的因果效果，因為一部分「bug 少」其實是團隊素質直接造成的，跟測試無關。要切斷，得在「紀律相近」的團隊間比較，或更好地——隨機指定某些任務「必須先寫測試」、某些不要求，再比 bug 率。

## 自我檢核

口頭自答，講得出來才算過關：

1. 為什麼「相關不是因果」**不**等於「相關沒用」？相關在因果推論裡扮演什麼角色？
2. 混淆變因、反向因果、純巧合，三者的結構各長什麼樣？為什麼說混淆變因是其中最陰險的？
3. 為什麼隨機化能切斷「**所有**」混淆變因，而「在觀察數據上控制變因」只能切斷一部分？這個差別為什麼那麼重要？
4. 為什麼說「我已經控制了一堆混淆變因」這句話，比「我做了 RCT」弱一個檔次？
5. Bradford Hill 為什麼堅持叫「視角」不叫「判準」？把它當打勾清單會出什麼錯？
6. 塞麥爾維斯的因果論證在「對照」這一軸其實很強，那他為什麼還是被否決？「沒有機制」如何讓人永遠能懷疑還有一個沒切斷的混淆？
7. 「觀察到 X」和「介入去設定 X」差在哪？為什麼你的 A/B test 屬於後者，而「看哪些使用者自己用了某功能」屬於前者？

## 延伸閱讀

- **Bradford Hill, A. B. (1965). "The Environment and Disease: Association or Causation?"** —— 九個「視角」的原始演講。讀它你會驚訝於希爾自己有多謹慎：他反覆強調這些不是判準、沒有一個是必要或充分的。維基的「Bradford Hill criteria」條目（https://en.wikipedia.org/wiki/Bradford_Hill_criteria）有九條的清楚整理與「不是 criteria 是 viewpoints」的澄清。
- **Judea Pearl & Dana Mackenzie, *The Book of Why*（《因果革命》，2018）** —— 想真正搞懂「觀察 vs 介入」`P(Y|do(X))` 的鴻溝、混淆變因與後門準則，這是最好讀的入門。本章那句「一句話的 do」展開成一整本書。
- **Tyler Vigen, *Spurious Correlations*（spuriouscorrelations.com）** —— 純巧合假相關的著名收藏，看完你對「兩條曲線一起動」會永久免疫。當笑話讀，但教學效果一流。
- **混淆變因與相關係數「怎麼算」** —— 相關係數 ρ 的定義、為什麼「相關 = 0」不等於「獨立」，機率機制見《馴服隨機》談相關與獨立的那章；本書只用結論、不推公式。
- **隨機化的統計威力「為什麼成立」** —— 「隨機分組讓混淆變因平均掉」這件事的機率基礎，見《馴服隨機》談隨機變數與獨立的那章。
- **下一步**：對照與混淆都講完了，但「證據如何**一點一點**抬高或壓低你對某假設的信任」還沒處理——那是 ch07「證據如何更新信念」（貝氏確證）的活兒；而「為什麼沒有機制就永遠有人能懷疑你」會在 ch14 收。
