# ch01 — 機械的證明：形式系統是什麼

> **本章解決什麼問題**：在我們能談「一個系統證不出自己的真話」之前，得先把「系統」「證明」這兩個詞釘死成精確的東西。本章把形式系統（formal system）拆成三件套——語法、公理、推理規則——讓你看清「證明」其實是一台不懂數學、只會按規則搬符號的機器跑出來的有限符號串。這是全書的地基：脊椎句 G 還沒出場，但它將來指涉的那個「本系統 F」，就是本章定義的這台機器。記住一句話——**可證（provable）是純語法、可機械檢查的**，後面整本書都在拿它跟「真」對撞。

全書有六條主線會反覆回來，先記個印象，每一條這本書都會親手交給你：

1. **真 ≠ 可證**——真理可以溢出證明。
2. **形式系統＝機械符號遊戲**——公理加規則就是一台搬符號的機器。
3. **自我指涉不是噱頭，是引擎**——說謊者悖論改一個字就變成不完備武器。
4. **兩條定理有精確的三個條件**（一致、可遞迴公理化、夠強表達算術），缺一就不成立。
5. **停機問題是孿生子**——圖靈用「會不會停」重演同一齣戲。
6. **正確邊界 vs 濫用**——定理講的是形式系統內的可證性，不是人生哲理。

本章只負責第二條，順手替第一條鋪路。我們先把那台機器組裝起來。

## 從你已知的出發

你每天都在跟一台「不懂語意、只查語法」的機器打交道：型別檢查器（type checker）。

寫一段 TypeScript，按下儲存，編譯器在毫秒內告訴你 `const x: number = "hello"` 不合法。它做這個判斷時，**完全不知道** `"hello"` 是一句問候、`number` 是你拿來算購物車金額的東西。它只是拿著一組規則——「`string` 不是 `number` 的子型別」——對著你的程式碼的語法結構（抽象語法樹，AST）逐節點比對。比對不過，就拒絕。它不理解，它只搬符號。

這就是形式系統最核心的直覺，而且你早就內化了：

- **編譯器接受一段程式 ≈ 系統「證得出」某命題；編譯器拒絕 ≈ 系統「證不出」。** 一段通過型別檢查的程式，是一份「它型別正確」的機械可驗證的證明。
- **型別＝命題，程式＝證明。** 這不是比喻，是一個真定理，叫 Curry–Howard 對應（Curry–Howard correspondence；Curry 1934 觀察到、Howard 1969 寫成手稿、1980 才正式發表）。一個型別 `A → B` 可以讀成命題「A 則 B」，一個具有該型別的函式就是這個命題的一份證明。本書點到這裡為止、不深推——但它告訴你「證明是一種機械可檢查的結構物」這件事，工程師其實比數學家更早從鍵盤上學會。
- **語法是一套可機械列舉的規則。** 你寫過 BNF 文法（Backus–Naur Form）、寫過 parser；「哪些字串是合法的程式」由文法機械地決定，跟「這程式做的事對不對」是兩回事。

把這幾個直覺握緊。形式系統就是把它們抽到最乾淨：一套符號、一套「哪些符號串合法」的語法、一個起點（公理）、幾條改寫規則（推理規則）。「證明」就是從起點出發、每步合規則、抵達目標的一串符號串。**整個過程一台不懂數學的機器就能跑、也能驗。** 這台機器後面會被希爾伯特當成救命稻草（ch02），又被哥德爾證明它救不了自己（ch07–08）。我們先把它組好。

## 形式系統的三件套

一個形式系統，剝到底，就三樣東西。我故意先講「它在解決什麼問題」，再給形狀——本書不給無動機的定義。

它要解決的問題是：**怎麼把「推理」這件原本住在人腦裡、靠「我覺得這樣推很合理」運作的事，改造成一台機器查得了的東西？** 答案是把意義整個抽掉，只留下符號的形狀和搬動符號的規則。於是「合不合理」這個會吵架的判斷，被換成「合不合規則」這個能機械核對的判斷。

三件套如下：

```text
形式系統 = 三個盒子，輸出「定理」

  ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
  │ 語法盒       │   │ 公理盒       │   │ 規則盒       │
  │              │   │              │   │              │
  │ 哪些符號串   │   │ 一開始免費   │   │ 從手上符號串 │
  │ 算「合法」   │   │ 給的那幾條   │   │ 生出新符號串 │
  │ 公式         │   │ 合法公式     │   │（改寫規則）  │
  └──────┬───────┘   └──────┬───────┘   └──────┬───────┘
         │                  │                  │
         └─────────┬────────┴─────────┬────────┘
                   │                  │
                   v                  v
            從公理出發、每步套一條規則，走到的每個
            符號串 = 一條「定理」(theorem)
                   │
                   v
        這一整串符號串（含每一步）= 一份「證明」(proof)
```

逐個拆：

**① 語法（syntax）。** 先有一張字母表（alphabet）——一組基本符號，例如 `0`、`=`、`+`、`(`、`)`、變數記號。再有一組造句規則，告訴你哪些符號串是「合法公式（well-formed formula）」。`0 = 0` 是合法公式；`= = 0 (` 不是，就像 `int int = ;` 過不了 parser。注意：合不合法是**純形狀**的判斷，跟真假、跟有沒有意義都無關。`0 = 1` 是個漂漂亮亮的合法公式，它只是（在我們心裡的算術意義下）為假——但系統根本不在乎假不假，那是 ch03 的事。

**② 公理（axiom）。** 一開始就無條件給你的幾條合法公式，當作推理的起點。你不用證它們，它們是地基。在算術系統裡，公理會是像「`x + 0 = x`」「不存在比 0 還小的數」這類東西（皮亞諾算術 PA 的公理，ch02 細談）。**關鍵**：公理本身也只是符號串，系統不「相信」它們、也不「理解」它們，只是把它們放進「手上已經有的合法公式」這個袋子裡。

**③ 推理規則（inference rule）。** 從你手上已有的公式，機械地生出新公式的改寫規則。最有名的一條叫離斷規則（modus ponens）：手上若有 `φ` 也有 `φ → ψ`，就可以寫下 `ψ`。注意這條規則怎麼運作——它**只看符號的形狀**：看到一個東西、又看到「那個東西 → 另一個東西」，就把箭頭右邊那個抄下來。它不問 `φ` 是什麼意思、不問它真不真。一台 `grep` 加 `sed` 等級的機器就能套。

組起來，「證明」與「定理」就有了精確的定義：

```text
證明（proof）：一串有限長的合法公式  s₁, s₂, …, sₙ
   其中每一個 sᵢ 要嘛是公理，
                要嘛是由前面某些 sⱼ（j < i）套一條推理規則得到的。

定理（theorem）：上面那串的最後一句 sₙ。
   （只要存在這樣一串，sₙ 就是定理；它「可證」。）

記法：F ⊢ φ  讀作「F 證明 φ」「φ 在系統 F 中可證」。
      F ⊬ φ  讀作「F 證不出 φ」。
```

三件事值得你停下來盯著看，因為全書都建在上面：

- **「證明」是一個物件，不是一個動作。** 它就是一串符號串，可以寫在紙上、可以存成檔案、可以（ch05）編碼成一個數字。它不是「我被說服了」這種心理狀態。
- **「可證」是純語法的。** 「F ⊢ φ 嗎？」這個問句，原則上不需要任何理解力來回答——只需要檢查「有沒有一串合法公式，最後一句是 φ，且每步合規則」。
- **這串東西可機械檢查。** 這是本章的核心收穫，下一節單獨講。

## 給我一份證明，一台機器就能驗它對不對

請務必把「**找**證明」和「**驗**證明」這兩件事分開——這是後面很多誤解的根源。

**找**一份證明很難。可能要靈感、要運氣、要試很多條路、甚至（ch07 會說）對某些真句子根本找不到。

但**驗**一份已經攤在你面前的證明，是純機械、保證有限步會結束的：

```text
驗證程序 VERIFY(宣稱的證明 s₁ … sₙ, 目標句 φ)：

  for i = 1 to n:
      檢查 sᵢ 是不是合法公式            ← 純語法比對（語法盒）
      檢查 sᵢ 是不是下列之一：
          (a) 一條公理                  ← 查公理盒，比對符號串
          (b) 由某些 sⱼ (j<i) 套某條規則 ← 規則盒，比對形狀
      若兩項都不過 → 回傳「這份證明無效」
  最後檢查 sₙ 是不是就是 φ
  全部通過 → 回傳「有效：這確實是 φ 的一份證明」
```

這段東西你讀起來像不像一個 linter、一個 schema validator、或一個區塊鏈節點在驗交易？正是。它逐項比對形狀，每步都是局部、有限、不需要理解的。一台「完全不懂算術在說什麼」的機器，照樣能斬釘截鐵地告訴你「這 47 行符號串確實是一份合法證明」。

這就是「**形式系統＝符號遊戲**」這句話的硬核意思，也是全書第二條主線。哥德爾後來的整個天才之舉（ch05），正是抓住「驗證證明是個純機械、純算術可表達的程序」這一點——因為純機械、純語法的東西，可以被編碼成數字、可以變成系統「談得了」的算術對象。換句話說：正因為證明檢查蠢到一台機器就能做，系統才有機會（透過編碼）回頭談論它自己的證明。地基的「機械性」不是缺點，是後面所有戲法的著力點。讀過書架上《把系統寫成定理》TLA+ 那本的話，這個「規格＋機械檢查器」的味道你熟——TLA+ 的 model checker 也是不理解你系統的語意，只機械地搜狀態。

> 嚴謹度標示：本節是**工程師的嚴謹**——上面每一步你都能口頭講出理由（為什麼有限、為什麼局部可檢查）。它不是哥德爾 1931 論文裡那 45 條原始遞迴函數的形式化建構（那個本書不展開，指向延伸閱讀）；它是那件事的直覺骨架，而骨架對本書夠用。

## Worked example：MIU 系統，一台真的在搬符號的機器

抽象講完，來真跑一台。我借霍夫斯塔特（Douglas Hofstadter）在《哥德爾、艾雪、巴赫》（*Gödel, Escher, Bach*，常簡稱 GEB）裡設計的 MIU 系統——他造它正是為了讓人親手摸到形式系統的機械性（2026-06 查證：MIU 系統出自 GEB，是教科書級的形式系統入門玩具）。

**字母表**：只有三個符號 `M`、`I`、`U`。合法公式＝任意由這三個符號組成的串（語法極簡）。

**公理**：只有一條，`MI`。

**推理規則**（四條，全是純形狀改寫；x、y 代表任意子串）：

```text
規則 R1：若手上的串以 I 結尾，可在尾巴加一個 U。      …  xI      → xIU
規則 R2：若手上的串是 Mx（M 後面接子串 x），可寫 Mxx。  …  Mx     → Mxx
規則 R3：串中若出現連續三個 I（III），可換成一個 U。    …  xIIIy  → xUy
規則 R4：串中若出現連續兩個 U（UU），可整段刪掉。       …  xUUy   → xy
```

注意這四條規則沒有一條提到「意義」。`M`、`I`、`U` 不代表任何東西。整個系統就是一個搬字母的遊戲。現在我手動證一條定理——也就是從公理 `MI` 出發、每步套一條規則、走到某個目標串。我要走到 `MUIU`：

```text
證明（一串合法公式，每步註明套了哪條規則、為什麼合法）：

  s₁ : MI        ← 公理（免費起點）
  s₂ : MII       ← 對 s₁ 套 R2：MI 是 Mx（x = I），寫 Mxx = MII        ← 把 I 複製一份
  s₃ : MIIII     ← 對 s₂ 套 R2：MII 是 Mx（x = II），寫 Mxx = MIIII    ← 再複製一次，現在有四個 I
  s₄ : MUI       ← 對 s₃ 套 R3：MIIII 裡 IIII 含 III，前三個 I→U，得 M U I ← 注意只換最前面那組 III
  s₅ : MUIU      ← 對 s₄ 套 R1：MUI 以 I 結尾，尾巴加 U，得 MUIU         ← 達標

  最後一句 s₅ = MUIU，正是目標 ⇒ MUIU 是 MIU 系統的定理（可證）。
  記作：MIU ⊢ MUIU
```

現在做關鍵的事——**驗**這份證明，扮演那台不懂事的機器。我逐行只問「合不合規則」，完全不問「這串字母是什麼意思」（它本來就沒意思）：

```text
VERIFY 這份宣稱的證明：

  s₁ = MI    : 是公理嗎？ 是。                                  ✓
  s₂ = MII   : 由 s₁ 套規則來嗎？ MI →(R2,x=I)→ MII，形狀對。    ✓
  s₃ = MIIII : 由 s₂ 套規則來嗎？ MII →(R2,x=II)→ MIIII，對。    ✓
  s₄ = MUI   : 由 s₃ 套規則來嗎？ MIIII 含 III →(R3)→ MUI，對。  ✓
  s₅ = MUIU  : 由 s₄ 套規則來嗎？ MUI 以 I 結尾 →(R1)→ MUIU，對。✓
  s₅ 等於目標 MUIU 嗎？ 是。                                    ✓
  ⇒ 有效。這 5 行確實是 MUIU 的一份合法證明。
```

整個 VERIFY 沒有一步動用到「理解」。它就是字串比對。這正是「可證是純語法、可機械檢查的」的活生生樣本。

順帶一個會在 ch07 大放異彩的伏筆：霍夫斯塔特出的著名難題是「**能不能從 `MI` 走到 `MU`？**」答案是**不能**——`MU` 不是這系統的定理。但你**沒辦法只待在系統裡、靠套規則套到天荒地老來發現這件事**（你套規則只會生出一個個定理，永遠碰不到「`MU` 證不出來」這個結論）。要看出 `MU` 證不出，得**跳到系統外面**，注意到一個規則永遠保持的不變量：串裡 `I` 的個數，在這四條規則下永遠不會變成 3 的倍數（R2 把 I 數翻倍、R3 一次拿掉 3 個，從非 3 倍數出發永遠回不到 3 倍數，而 `MU` 需要 I 數為 0＝3 的倍數）。**「系統內證不出，但站到系統外用對的眼光就看得出真相」——這個結構，正是哥德爾不完備定理的雛形。** 先記著這個味道，ch07 會把它放大成全書的高潮。

## 一致：為什麼「不矛盾」是系統的命根子

組好機器，得馬上提一件後面（尤其 ch08）會變成主角的事：**一致（consistent）**。

一個系統叫**一致**，意思是：它**不會同時證出某句 φ 和它的否定 ¬φ**。反過來，若一個系統能同時證出 `φ` 和 `¬φ`，它就叫**不一致（inconsistent）**。

聽起來只是「別自相矛盾」的常識，但對形式系統來說，不一致是個**核災級**的故障，原因是一條叫**爆炸原理（principle of explosion，拉丁文 ex falso quodlibet，「從假推出一切」）**的東西：

```text
爆炸原理：一旦系統證出某個 φ 同時又證出 ¬φ，
          那麼系統就能證出「任何」句子 ψ —— 對的、錯的、毫不相干的，全都證得出來。

      直覺（一步步走）：
        已有 φ              （假設拿到了）
        已有 ¬φ             （假設也拿到了）
        由 φ 可推 (φ ∨ ψ)   ← 「或」只要一邊真就行，ψ 隨便填什麼都行
        由 (φ ∨ ψ) 和 ¬φ
            推得 ψ          ← φ 那條路被 ¬φ 堵死了，只剩 ψ 這條路
        ⇒ 任意 ψ 都可證
```

對工程師這個直覺極好懂：**一個自相矛盾的規格能「滿足」任何需求**。如果你的型別系統裡某個地方能同時推出 `T` 和 `¬T`，那它就能型別檢查通過任何東西——`number` 是 `string`、`string` 是 `never`、什麼都行。這種「過於聽話」不是強大，是徹底壞掉：一個什麼都證得出來的系統，等於什麼都沒證——「可證」這個詞在裡面失去了全部資訊量。你 review 過自相矛盾的需求文件就懂：規格一旦自打嘴巴，它就「同意」任何實作，於是它對實作不再有任何約束力。

所以**一致是底線**。一個有用的形式系統，最起碼的要求是它一致——它證出來的東西才值得當一回事。這就是為什麼哥德爾第二定理（ch08）那麼痛：它會告訴你，一個夠強又一致的系統，連「我自己不會爆炸」這件最起碼的保命符，都沒辦法靠自己的規則證出來。一致是命根子，而你摸不到自己的命根子。

## 哥德爾的標靶：Principia Mathematica

最後給本章一個歷史錨點，順便交代哥德爾 1931 論文標題裡那個名字。

二十世紀初，數學界經歷了一場地基危機——集合論裡冒出羅素悖論（Russell's paradox，ch04 會用到它的近親）這類自相矛盾，讓人開始擔心：整座數學大廈會不會其實建在流沙上？羅素（Bertrand Russell）和懷海德（Alfred North Whitehead）的回應是一套雄心驚人的工程：**把全部數學裝進一個形式系統**。他們合寫了三大卷的《數學原理》（*Principia Mathematica*，常簡稱 PM；三卷分別於 1910、1912、1913 出版，2026-06 查證），想從純邏輯的公理與規則出發，一步步機械地推導出全部數學真理——著名的是書裡推到第幾百頁才證出「1 + 1 = 2」。

PM 就是「把三件套開到極致、想裝下整個數學」的代表作。它正是哥德爾 1931 那篇論文標題點名的對象——〈論《數學原理》及相關系統中形式上不可判定的命題 I〉。哥德爾要證的，不是「PM 有個小 bug」，而是更狠的：**任何一個像 PM 這樣夠強、又一致、規則寫得出來的系統，都必然關得住一個它自己證不出、卻又千真萬確的真句子。** 不是 PM 不夠好，是「夠好就會這樣」。

但要走到那一刀，得先看清楚希爾伯特究竟想從這台機器要到什麼——那是 ch02。

## 直覺的陷阱

本章建立的是最基本的詞彙，最常見的誤解就藏在「把日常義偷偷裝回技術詞」這個動作裡。先抓兩個具名陷阱。

| 陷阱 | 錯誤直覺帶你下溝的那一步 | 正確版（怎麼自我察覺） |
|---|---|---|
| **「公理就是『顯然為真』的東西」** | 你把「公理」理解成「不證自明的真理」，於是預設系統「知道」公理是真的、系統在「相信」什麼。 | 公理在形式系統裡**只是無條件當起點的合法符號串**，系統不理解也不相信它。真假是 ch03 的語意層的事，跟系統內的「可證」是兩條線。察覺法：問自己「我剛剛是不是偷偷把『真』塞進『公理』裡？」——系統不需要真，只需要規則。 |
| **「『證明』就是說服我的論證」** | 把形式證明等同於日常那種「講到你被說服」的論證，於是覺得證明需要理解、需要洞見、是主觀的。 | 形式證明是一個**有限的符號串物件**，可被一台不懂數學的機器逐步核對。找它可能要洞見，但它本身與驗它都不靠洞見。察覺法：問「這份『證明』能不能寫成檔案、丟給一個只會比對形狀的程式去驗？」能，才是本書說的證明。 |

還有一個更隱蔽、值得獨立點名的陷阱——**把「合法（合語法）」聽成「為真」**。`0 = 1` 是個完美合法的公式，系統甚至可能在某種設定下證出它（如果系統不一致的話）。「合法」只說它形狀對、能參與符號遊戲；「可證」說它從公理走得到；「真」（ch03 才登場）說它在我們心裡那個算術世界裡成立。這是三件不同的事，本章你只需要先把前兩件——合法 vs 可證——跟「真」隔開。後面整本書都在量這幾條線之間的縫。

## 紙上推演

以下純紙筆，不需要任何程式。MIU 系統的規則就在上面 Worked example，回頭對照即可。

### 題目一（★，10 分鐘）：手走一條 MIU 證明

從公理 `MI` 出發，用四條規則證出 `MIIU`。寫出完整證明串（每行一個符號串，註明套了哪條規則），然後扮演 VERIFY 機器逐行驗一遍。

### 題目二（★★，15 分鐘）：為什麼一台不懂數學的機器能驗證明？

不准用「因為規則很簡單」這種話。用你自己的話，向一個沒讀過本章的後端同事解釋：為什麼「驗證一份證明」原則上不需要任何理解力、保證有限步會結束？提示：把它跟你熟的某個東西（linter、schema validator、parser）對上，並點出「找證明」和「驗證明」難度為何天差地別。

### 題目三（★★，15 分鐘）：合法 / 可證 / 真，三條線

在 MIU 系統裡判斷下面三句各屬哪一類（合法但你不知可不可證 / 可證 / 在系統內無從談「真假」），並說明為什麼「真」這個詞在 MIU 系統裡根本無處安放：

（a）`MU`　（b）`MIU`　（c）`IMU`

### 推演解答

**題目一。** 一條可行的證明（不唯一）：

```text
  s₁ : MI     ← 公理
  s₂ : MII    ← R2（MI 是 Mx，x=I，寫 Mxx=MII）        ← 複製 I
  s₃ : MIIU   ← R1（MII 以 I 結尾，尾加 U）             ← 等等，這是 MIIU 嗎？
```

停——這裡要小心，常見錯路就在這。`MII` 以 `I` 結尾，套 R1 加 `U`，得到的是 `MIIU`，正是目標。所以其實兩步就到：

```text
  s₁ : MI    ← 公理
  s₂ : MII   ← R2（x=I）：MI → MII
  s₃ : MIIU  ← R1：MII 以 I 結尾，加 U → MIIU   ← 達標
  ⇒ MIU ⊢ MIIU
```

VERIFY：s₁ 是公理 ✓；s₂ 由 s₁ 套 R2，形狀對 ✓；s₃ 由 s₂ 套 R1（結尾是 I），形狀對 ✓；s₃＝目標 ✓ ⇒ 有效。**常見錯路**：(1) 想套 R1 卻在串不以 `I` 結尾時硬加 `U`——規則的前提條件（結尾必須是 `I`）是規則的一部分，前提不滿足就不准套，這正是「純形狀」的嚴格性。(2) 以為步數越多越「嚴謹」——形式證明不比長短，最短的合法串照樣是合法證明。

**題目二。** 要點答案：因為「驗證明」只是對一份**已經給定的有限符號串**做**逐行局部比對**——每一行只問兩件事：它是不是公理（查表）、它是不是由前面某幾行套某條規則得來（比對形狀）。這兩件事都是純語法的字串比對，不涉及這些符號「代表什麼」；而串是有限長的，行數有限，所以保證有限步停。它跟 linter / schema validator / parser 是同一類東西：拿規則對著結構逐節點核對，過或不過。**「找」與「驗」的難度天差地別**：驗是給定一份答案核對它對不對（有限、機械、保證停）；找是要在「所有可能的符號串序列」這個無限大的空間裡撈出一條能抵達目標的——可能要洞見、可能撈很久、ch07 甚至會說對某些真句子根本撈不到。一句話：證明檢查是 P 等級的無腦活，證明搜尋是大海撈針。（這個「驗易、找難」的不對稱，正是 ch05 編碼能成立、ch09 停機問題能咬住的關鍵。）

**題目三。**

- （a）`MU`：合法公式（三個合法符號組成），但它**不可證**——如本章伏筆所述，I 的個數在四條規則下永遠保持「非 3 的倍數」這個不變量，而 `MU` 的 I 數是 0（＝3 的倍數），所以從 `MI`（I 數＝1）永遠走不到。注意：「我們知道它證不出」這個判斷是**站在系統外**用不變量看出來的，不是在系統內套規則套出來的。
- （b）`MIU`：合法，且**可證**（`MI →R1→ MIU`，一步）。
- （c）`IMU`：**合法但不可證**——依本章語法定義「任意由 M/I/U 組成的串都是合法公式」，`IMU` 三個合法符號組成，形狀上是合法公式。但規則 R1–R4 沒有一條能把開頭的 `M` 移走或在非 `M` 開頭的位置生出新串，而公理 `MI` 以 `M` 開頭，所以系統可達到的每一個串都以 `M` 開頭。`IMU` 以 `I` 開頭，永遠走不到，故**不可證**。

至於**「真」**：MIU 系統**沒有給任何符號意義**，`M`、`I`、`U` 不指涉自然界任何東西，所以「`MU` 真不真」這個問句在 MIU 裡**沒有意義可問**——沒有一個「MIU 世界」讓我們去核對。這正是本章要你先隔開的：合法（形狀）、可證（走得到）是系統內就有的；「真」需要一個系統外的「意義／模型」才談得起來，那是 ch03 把算術系統接到標準模型 ℕ 之後才登場的事。MIU 的純粹（沒意義）反而讓你看清：可證，從來不等於真。

## 自我檢核

口頭自答，講得出來才算過關：

1. 把形式系統的三件套（語法、公理、推理規則）各用一句話講給同事聽，並各舉一個你工作中的對應物。
2. 「證明是一個物件，不是一個動作」——這句話什麼意思？為什麼這個觀點對後面（ch05 編碼）很重要？
3. 為什麼「**驗**一份證明」是純機械、保證有限步停的，而「**找**一份證明」可能要洞見甚至找不到？
4. 「可證（provable）」和「真（true）」為什麼是兩件事？在 MIU 這種沒給符號意義的系統裡，為什麼「真」根本無處安放？
5. 什麼叫「一致（consistent）」？爆炸原理為什麼讓「不一致」變成核災級故障、而不只是個小瑕疵？拿你 review 過自相矛盾的規格的經驗講講。
6. 為什麼說「形式系統＝符號遊戲」這件事——系統蠢到不懂語意——反而是哥德爾後面所有戲法的著力點，而不是缺點？
7. MIU 的 `MU` 證不出來，但你怎麼知道它證不出來？這個「站到系統外才看得出」的動作，跟哥德爾不完備在味道上哪裡像？
