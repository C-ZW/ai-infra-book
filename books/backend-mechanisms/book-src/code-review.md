# 程式碼審查：審什麼、它在保護什麼

兩個工程師為了同一個 pull request 卡了三天。作者覺得審查者在雞蛋裡挑骨頭——「這個變數叫 `data` 還是 `payload` 有差嗎」；審查者覺得作者沒抓到重點——「這段在高併發下會 race，你卻只想討論命名」。他們都以為程式碼審查是在做同一件事，其實一個在審「對不對」，一個在審「往後好不好維護」。最後 PR 被一句「LGTM」放行，race 沒人提，命名改了三輪——審查發生了，但它該保護的東西，一樣也沒保護到。

這一幕之所以常見，是因為「程式碼審查在幹嘛」這個問題，多數團隊從沒講清楚過。它表面上是一道流程——變更合進主幹前，由作者以外的人讀一遍——但它真正在保護什麼、用什麼代價換什麼保證，往往沒人說得明白。把這件事想透，比背一套「審查最佳實踐」有用得多：因為一旦你知道它在保護什麼，你就知道什麼時候它在空轉、什麼時候它正在替你擋下未來的一場事故。

## 直覺會說審查是來抓 bug 的——資料不這麼說

直覺上，審查的價值就是「多一雙眼睛幫你找 bug」。這個直覺不算錯，但它把審查最弱的功能當成了主功能。

有資料可以拆這個誤會。Microsoft 在 2015 年發表的一份內部審查研究（Czerwonka 等人，〈Code Reviews Do Not Find Bugs〉）手動分類了審查留言，發現一個常被引用的數字：審查留言中，**直接和缺陷有關的只約 15%**，其餘絕大多數談的是可維護性、可讀性、命名、設計方向。Google 在 2018 年那份分析九百萬筆審查紀錄的研究（Sadowski 等人）則從另一個角度佐證了同一件事：工程師導入審查的首要動機是可理解性與可維護性、教育與把關，缺陷偵測並非主軸。也就是說，當工程師真的坐下來審一段 code，他們**自發地**把大部分注意力放在「這以後好不好讀、好不好改」，而不是「這裡有沒有 bug」。

那「找 bug」這件事誰做得更好？是測試。把幾十年的軟體工程研究攤開看，會發現一個穩定的分工：程式碼審查（人讀）對缺陷的偵測率大約落在 **55%–60%**，而各類測試——單元、功能、整合——大約在 **25%–45%**。乍看審查贏了，但這個比較會誤導：兩者抓的根本不是同一批 bug。測試擅長抓「行為對不對」——給定輸入有沒有給出預期輸出，而且抓完之後能**自動化、反覆執行、永不疲勞**；審查擅長抓的是測試**寫不出斷言**的東西——這個抽象選錯了、這個邊界沒人想到、這段以後沒人看得懂。一個 race condition，測試可能十次有九次跑綠（它只在特定排程下才現形），但一個有經驗的審查者掃過共享狀態與缺失的鎖，當場就能指出來。

所以正確的結論不是「審查比測試會抓 bug」，而是**兩者互補、各擋一類**。把審查當成抓 bug 的主力，你會用錯它，也會失望——你會拿「這次審查抓到幾個 bug」去衡量它，然後得出「審查沒什麼用」的結論，因為它本來就不是靠這個吃飯的。

## 那它到底在保護什麼

把抓 bug 從主位上移開，審查真正守的四樣東西才浮得出來。它們有一個共同點：**都是「未來才會兌現、出事時才發現沒做」的保證**——這正是它們容易被輕忽、又最不該被輕忽的原因。

**第一，它在對抗「只有一個人懂」。** 審查強制至少兩個人讀懂每一段進入系統的變更。這聽起來像副作用，其實是審查最硬的價值。一段沒人審過的 code，它的設計意圖、它為什麼這樣寫而不那樣寫、它有哪些隱藏前提，全都只活在作者一個人的腦子裡。那個人請長假、轉組、離職的那一天，這段 code 就變成系統裡的一塊黑箱——能跑，但沒人敢碰。審查是團隊對抗這種「知識孤島」的主要保險：每一次審查，都把一塊原本只有一個人懂的領土，至少擴散給了第二個人。

**第二，它在替「未來要改這段 code 的人」把關。** 作者寫的時候腦子裡是「怎麼讓它跑起來」；審查者該站的位置是「半年後有人要改它時，他讀得懂嗎、改得動嗎」。一個糟糕的命名、一個過度聰明的抽象、一段把三件事揉在一起的函式——它們今天都能跑，bug 一個都沒有，但每一個都在替系統累積「以後難改」的債。這種債測試攔不下來（它跑得綠綠的），只有一個站在未來視角的人讀得出來。前面那份研究裡審查留言以可維護性為大宗，不是工程師閒著沒事挑剔，而是他們的注意力**自然地**流向了這個測試照不到的角落。

**第三，它在變更還小的時候攔下方向性的錯。** 一個設計走偏了的變更，越晚發現越貴——它一旦長進系統、長出依賴它的下游，重構的成本就以倍數膨脹。審查是把「這個方向對嗎」這個問題，問在它最便宜的時刻：code 還只有幾十行、還沒人依賴它、改起來只是改一個 PR 而不是改半個系統。

**第四，它是一道社會性的閘——而且這道閘在你按下「送出審查」之前就生效了。** 知道有人會逐行讀你的 code，你下筆時自然更謹慎：那個你本來想之後再補的測試、那個你想偷懶跳過的錯誤處理、那段你自己都覺得醜的 hack，你會在開 PR 前就先收拾掉。這個「被看見」的預期本身就在提升品質，而且它發生在審查者讀到第一行之前。這也是為什麼「反正沒人會仔細看」的團隊文化一旦形成，code 品質會崩得特別快——閘形同虛設，連那層自律都沒了。

把四樣收成一句：**審查的保證不是「這段 code 沒有 bug」，而是「沒有任何變更是只有一個人知道、且沒有第二雙眼睛覺得它往後可維護」。** 記住這句話，你就有了一把尺去判斷一次審查是真的發生了，還是只是蓋了個橡皮章。

## 代價在哪：注意力與 Lead Time

審查不是免費的，它的代價具體而尖銳：**它花的是審查者的注意力，和變更上線的時間（Lead Time）。** 而這兩樣都是稀缺資源——尤其注意力，它不只稀缺，還會**隨著要讀的量增加而急速貶值**。這一點是理解審查為何常常失效的關鍵，值得用真實數字看透。

最經典的一組數據來自 SmartBear 在 Cisco 做的審查研究——他們花了十個月，分析了 **2,500 次審查、涵蓋 320 萬行 code**。這份研究把「審查者一次能有效讀多少」量化了出來，結論非常一致：

- **規模**：當一次審查的變更在 **200–400 行**以內時，缺陷偵測效果最好；一旦超過 400 行，偵測率就開始陡降。
- **速度**：審查速度慢於每小時約 **300–400 行**的審查者，缺陷偵測能力高於平均；一旦快過每小時約 450 行，有 **87%** 的情況偵測密度低於平均——讀太快，等於沒讀。
- **持續時間**：人的審查效力在連續 **60–90 分鐘**後急遽下滑；在這個時間窗內審 200–400 行，能達到約 **70%–90%** 的缺陷發現率。

把這幾個數字串成一個 worked example，你會看到審查的經濟學。假設一個 PR 是 800 行。理論上審查者該以低於每小時 400 行的速度讀，也就是**至少花兩小時**才讀得仔細。但兩小時遠超過那個 60–90 分鐘的有效窗——真實會發生的是：審查者讀到第 300 行注意力已經開始渙散，第 500 行起進入「掃過去、看起來沒事」的模式，最後那 300 行幾乎是沒被讀的。於是這個 800 行的 PR，實際得到的有效審查，可能還不如一個 150 行的 PR——後者落在最佳區間裡，被從頭到尾認真讀完。

這就導出審查最反直覺、也最重要的一條規律：**PR 越大，審查品質越差，而且是斷崖式地差。** 同一份研究系的後續資料給了更刺眼的對照——小 PR（百行以下）的缺陷偵測率可達 **87%**，而超過千行的 PR 掉到約 **28%**。一個 1,000 行的 PR 拿到的有效意見，往往比一個 50 行的還少，因為審查者讀不動就放水，直接 approve。在更大規模的 PR 樣本裡也看得到同樣的影子：約**一半的 PR 沒有收到任何留言**就被批准。「有人審過」這四個字，在大 PR 上常常是個假象。

所以「小批量」這件事，不只是為了部署安全（小變更出錯時 blast radius 小、好 rollback），它根本是**讓審查這個機制能運作的物理前提**。一個團隊如果一邊要求「所有變更都要審」、一邊放任 PR 動輒上千行，它買到的只是審查的**形式**，沒買到審查的**保證**——閘在那裡，但門是開著的。

## 把機器能做的丟給機器

理解了「審查者的注意力是會貶值的稀缺資源」，下一步的設計就順理成章：**凡是機器能判斷的，都不該佔用人的注意力。**

格式、縮排、引號、import 排序、明顯的 lint 違規、測試有沒有跑過、覆蓋率有沒有掉——這些東西全都可以自動化（交給 formatter、linter、CI gate），而且**必須**自動化。原因不只是省時間，更是省**頻寬與情緒**：一個審查者如果把注意力花在「你這裡少了個空格」，他用掉的是那 60–90 分鐘有效窗裡的一部分，而那部分本該留給「這個抽象對嗎」「這個邊界你想過嗎」這種機器永遠看不出的事。更糟的是，格式爭議特別容易擦槍走火變成個人偏好之爭（tab 還是 space 能吵一整個下午），消耗的是團隊的耐心。

把這層想清楚，「自動化先行、人審其餘」就不是一句口號，而是一個明確的分工原則：**讓 CI 把所有有客觀對錯的東西擋在人類審查之前，人類只看那些沒有客觀對錯、需要判斷的東西。** 一個常見的失敗模式正好相反——團隊有了 CI，卻把它當成「綠了就能合」，於是 review 退化成橡皮章；或者反過來，沒有 CI，於是人類審查者被淹沒在瑣事裡，等讀到真正重要的設計問題時注意力已經耗光了。CI 綠燈和人類審查保護的是**不同的東西**：前者保證「沒有客觀錯誤」，後者保證「有人理解、且覺得它可維護」——少了哪個都不完整。

## 一個容易被忽略的非顯然事實：審查是雙向的

關於審查，有一個多數團隊默會接受、卻很少明說的扭曲：把審查當成**資深者卡資淺者的關卡**。資淺的人寫 code、資深的人審、資深的人點頭才能過——權力是單向的。

這個模型錯在它浪費了審查一半的價值。回到審查真正保護的那四樣東西，會發現它們**沒有一樣是單向的**。知識擴散是雙向的：資淺者審資深者的 code，同樣能把那塊領土的知識擴散給多一個人，而且對資淺者來說，逐行讀懂一段資深者寫的、設計成熟的 code，是 codebase 裡最好的學習材料之一——比讀文件有效得多。社會性的閘也是雙向的：當資深者知道自己的 PR 也會被資淺的同事讀，他下筆同樣會更在意可讀性（畢竟得寫到讓人讀得懂）。甚至「方向性錯誤的早期攔截」也是雙向的——一個剛進來、還沒被既有假設綁住的人，常常能問出「我們為什麼非得這樣做」這種資深者已經問不出口的問題。

把審查設計成單向的權力關卡，是把一個雙向的學習與防護機制，硬閹成一個守門儀式。它會帶來兩個可預期的後果：資淺者學不到該學的（因為他們從不審、只被審），資深者的 code 進了系統卻沒被任何人真正讀過（因為「沒人有資格審我」）。**最危險的黑箱，往往正是團隊裡最資深那個人寫的、誰也不敢審的那塊 code。**

## 回饋本身是一個機制——而且它的失敗模式很具體

審查的軟性核心——怎麼給意見、怎麼收意見——常被當成「人際技巧」這種模糊的東西帶過。但如果把它當成一個機制來看，它的保證與失敗模式其實很清楚。

審查回饋要保護的，是**團隊長期的心理安全**——也就是「人們願不願意誠實地把不確定、把半成品攤出來被看」這件事。這個保證一旦破了，整個審查機制會跟著空轉：人們開始把 PR 拆得又小又保守只求不被挑剔、開始在審查裡防衛而非討論、開始為了不被批評而不去碰困難但該碰的改動。而破壞它的，往往就是一句措辭。

「這寫得很爛」和「這段在高併發下會 race，建議改用 CAS（見〈樂觀與悲觀並發控制〉）」是天差地別的兩句話。它們可能指向同一個技術問題，但前者攻擊的是人、後者針對的是 code，且還教了對方一件事。這不是禮貌問題，是機制問題：一句針對人的評論，扣掉的是團隊下一次的誠實；一句針對 code、附帶理由的評論，存進的是團隊的集體知識。

幾個具體的、可操作的分界，把這個機制講到能在腦中重放：

- **評論 code、不評論人。** 主詞永遠是「這段邏輯」「這個函式」，不是「你」。
- **區分「必須改」與「建議」。** 把個人偏好（nit，nitpick）和真正的阻擋問題（blocking）明確標開，別把「我比較喜歡這樣寫」偽裝成硬性要求——這是審查者最容易濫用的權力，也是作者最容易被惹毛的點。
- **解釋「為什麼」。** 一句「改成 X」是命令，一句「改成 X，因為現在這樣在 Y 情況下會 Z」是教學。後者讓回饋能複利，前者只能消費一次。
- **收的一方：把意見當成對 code 的、不是對人格的。** 不同意時用討論而非防衛回應——審查者花了他寶貴的 60 分鐘有效窗讀你的 code，那是在幫你，哪怕他說錯了。

這些不是禮儀守則，是在維護那個讓審查機制能持續運作的前提。一個措辭尖刻的團隊，會在統計上收到更小、更保守、更不誠實的 PR——機制還在跑，但它保護的東西已經漏光了。

## AI 進來之後，審查更該守的反而是老東西

當一部分 code 開始由 AI 生成、一部分審查的初篩也交給 AI 來做，審查這個機制的形狀會被推一把。但推的方向常被誤判，值得在原理層講清楚。

AI 確實能承接審查裡很多苦工：指出明顯的問題、生成 diff 的摘要、解釋一段陌生 code 在幹嘛、做第一輪的掃描。這些都減輕人類審查者的負擔——本質上，它把更多「機器能判斷的」從人的注意力裡移出去，和前面那條「自動化先行」是同一個方向的延伸。這是好事。

但有兩個原理層的新陷阱，恰恰打在審查本來就最該守的地方。

其一，**AI 生成的 code 會大量製造「能跑、但沒有任何活人真正理解」的變更。** 這正是審查第一條保證——對抗知識孤島——要對付的東西，只是換了個更刁鑽的形態：以前的黑箱至少作者本人懂，AI 生成的黑箱可能**連作者都沒完全懂**（他只是看它跑綠了就送出來）。所以面對 AI 生成的變更，人類審查者反而更該追問那個最樸素的問題：「我們**懂**這在做什麼嗎？」——審查的重心從「挑錯」前移到「確認有人真的理解」。

其二，**把審查整個外包給 AI，會把審查最核心的功能抽掉。** 如果一個 PR 由 AI 寫、由 AI 審、人只負責點 approve，那麼「至少有一個活人讀懂了這段變更」這個保證——審查存在的第一理由——就從來沒被兌現過。AI 審查能補強細節、能加速，但它不能替代「讓**人**理解這段 code、達成知識擴散」這件事，因為知識擴散的終點必須是團隊裡的人腦，不是模型。

換句話說，AI 改變了審查的**苦工**，但沒有改變審查的**目的**——如果說有什麼改變，是它讓那個目的（確認有活人真懂）變得**更**重要，因為製造「能跑沒人懂」的黑箱，從來沒有這麼容易過。

## 為什麼是這個形狀

退一步看，程式碼審查的整個樣貌——為什麼要小 PR、為什麼把格式丟給機器、為什麼措辭這麼要緊、為什麼它不該是單向的權力——都是從一個樸素的事實裡長出來的：**它保護的東西，全都是「未來才兌現、出事才發現沒做」的保證**，而它用來保護這些的唯一工具，是**一種會疲勞、會貶值、會被措辭影響的人類注意力**。

正因為它守的是未來的可維護性與知識的不孤立，而不是當下的對錯，所以你不能拿「這次抓到幾個 bug」去衡量它——那是測試的指標。正因為它的工具是會疲勞的注意力，所以 PR 必須小、瑣事必須自動化、有效窗只有那 60–90 分鐘——這些不是風格偏好，是注意力這個資源的物理。正因為它的保證建立在「人願意誠實攤出 code 被看」之上，所以措辭與雙向，不是軟性的加分項，而是讓機制不空轉的硬前提。

下次你看到一個 PR 被秒批「LGTM」，你會知道該問的不是「審查者夠不夠認真」，而是「這次審查，到底有沒有兌現它該兌現的那個保證——有第二個活人，真的讀懂了這段變更，並且覺得它往後可維護嗎」。如果沒有，那道閘就只是擺著好看；如果有，它正在替你擋下一場半年後才會爆、爆了沒人改得動的事故。
