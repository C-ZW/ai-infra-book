# SQL 與 NoSQL：資料模型各擅其場

兩個團隊在同一場架構評審上吵起來了。一個說「這明明就是一張 `orders` 表加幾個外鍵，幹嘛上 Cassandra」；另一個說「我們一天要寫進三十億筆事件，PostgreSQL 早就被寫爆了」。雙方都對——但他們不是在爭同一件事。前者腦中是「資料之間的關係要怎麼正確地表達」，後者腦中是「這個寫入量在單機上扛不住」。「SQL vs NoSQL」之所以是一場吵不完的架，正是因為這個名字把至少三件互不相干的事——查詢語言、資料模型、擴展與一致性取捨——綁成一團，讓兩個討論不同問題的人，用同一組詞互相否定。

要把這團解開，得先承認一個不太順耳的事實：**「NoSQL」不是一種資料庫，它是「不是關係模型的那一大堆東西」的補集。** 文件庫、鍵值庫、寬欄庫、圖庫、時序庫，它們之間的差異，往往比它們各自跟關係庫的差異還大。把 MongoDB 跟 Cassandra 放進同一個「NoSQL」籃子裡，就像把腳踏車跟貨櫃輪船放進「不是汽車」這個類別——技術上沒錯，但這個類別什麼也沒告訴你。所以這一章不打算回答「SQL 還是 NoSQL」，那是個假問題。真正能拿來做決定的問題只有一個：**你的資料長什麼形狀、你會怎麼查它、你願意為擴展放棄什麼。**

## 關係模型到底買到了什麼

要看懂 NoSQL 各家族「放棄了什麼」，得先看清楚關係模型當年「保證了什麼」——因為每一種 NoSQL，本質上都是拿掉關係模型的某一根支柱，換取別的東西。

1970 年 Edgar Codd 在 CACM 上發表〈A Relational Model of Data for Large Shared Data Banks〉（Comm. ACM 13(6):377–387），這篇論文的革命性不在「用表格存資料」——那之前就有——而在一個更深的主張：**把「資料怎麼被查」和「資料怎麼被存」徹底解耦。** 在 Codd 之前的階層式、網狀資料庫裡，你存資料時就得把查詢路徑寫死成指標鏈，想換個查法就得重新組織儲存。Codd 說，不對，資料應該存成一組數學上的關係（relation，也就是表），查詢用宣告式的代數去描述「我要什麼」，而不是「怎麼一步步走到它」。**你先把資料正規化地存好，查法可以事後再想，甚至想無數種。** 這個「先存好、查法之後想」的自由度，是關係模型最值錢、也最容易被視為理所當然的東西。

這個自由度具體由三根支柱撐起來：

第一根是**固定 schema 與正規化**。每張表的欄位形狀是宣告好的、資料庫會強制執行；正規化（把重複的事實拆出去、用外鍵指過去）保證「同一個事實只存一份」。客戶的地址改了，你改一個地方，所有指向它的訂單自動看到新值——不會有十筆訂單各存一份過期地址在那裡互相矛盾。

第二根是**宣告式 join**。因為事實被拆散在多張表，查的時候要靠 join 把它們在查詢時即時拼回來。`SELECT ... FROM orders JOIN customers ON ...` 你只說要拼什麼，planner 去決定用 hash join 還是 nested loop、走哪個索引。這是「事後任意查法」的引擎——你能把任何幾張表，沿任何外鍵關係，臨時組合出一個從沒預想過的報表。

第三根是**跨列跨表的 ACID 交易**。一筆轉帳要從 A 扣、往 B 加，這兩個動作要嘛一起成功、要嘛一起失敗，中間沒有任何人看得到「A 已扣、B 未加」的破碎狀態。這個保證不限於單一資料列，它跨越任意多列、多表（ACID 與隔離級別的機制細節，見〈一致性與交易的保證模型〉與〈隔離級別與異常〉）。

這三根支柱合起來，是一個極其慷慨的承諾：**你不必預先知道會怎麼查，存好就行；資料的一致性由資料庫替你扛。** 接下來要講的每一種 NoSQL，都是在某個壓力下，發現這個慷慨的承諾太貴、或根本給不出來，於是抽掉其中一根，換成自己更需要的東西。

## 第一原理：先問怎麼查，再決定怎麼存

關係模型給你「先存好、查法之後想」的奢侈。NoSQL 世界裡，這句話幾乎完全反過來：**你得先想清楚怎麼查，才能決定怎麼存。**

這不是 NoSQL 設計者懶，而是「事後任意查法」這個能力本身有代價。join 要快，得維護一堆索引；跨表 ACID 要成立，得有鎖或 MVCC 在背後協調；正規化省了空間，卻讓每次查詢都要把拆散的事實重新拼回來。當你的資料量大到要橫跨幾十台機器，這些代價會放大成致命傷：兩張表如果分片在不同機器上，一次 join 就變成一次跨網路的分散式查詢，慢到不能用。所以多數 NoSQL 的設計哲學是：**與其在查詢時花大力氣把事實拼回來，不如在寫入時就把它們擺成查詢要的樣子。** 這就是為什麼用 Cassandra 或 DynamoDB，第一步永遠是「列出你所有的查詢」，而不是「畫實體關係圖」。你不是在建模資料，你是在建模存取。

帶著這個第一原理，我們逐一看各家族抽掉了哪根支柱、換到了什麼。

## 文件庫：把聚合就近放在一起

關係模型把一個訂單拆成 `orders`、`order_items`、`shipping_address` 好幾張表，是因為它信奉「同一事實只存一份」。但很多時候，你存取資料的單位根本就是「一整個訂單」——你很少單獨去查「所有人的所有 order_item」，你查的是「這一筆訂單連同它的明細和地址」。文件庫（MongoDB、Couchbase 是代表）的洞見就是：**如果存取的單位是一個聚合（aggregate），那就把這個聚合整個存成一份巢狀文件，別拆。**

一筆訂單在 MongoDB 裡是一份 JSON-like 的文件：訂單欄位在最外層，明細是一個內嵌陣列，地址是一個內嵌物件，全部躺在同一份文件裡。查這筆訂單＝一次讀取拿到整份，不需要 join。寫這筆訂單＝更新一份文件，而**單一文件的更新是原子的**——這是文件庫保留下來的那一小塊交易保證：聚合邊界內，你有原子性。

它抽掉的支柱是**固定 schema**。文件庫不在寫入時強制欄位形狀，這帶來真正的彈性：今天的訂單多一個 `coupon_code` 欄位，舊訂單沒有也無妨，不用跑 schema migration。但這個彈性是有代價的，而且代價藏得很深——schema 並沒有消失，它只是**從資料庫搬進了你的應用程式碼**。資料庫不再保證 `price` 一定是數字，於是某一版程式不小心寫進了字串 `"19.99"`，資料庫照單全收，三個月後另一段程式拿它去做數值比較，當場炸開。這就是文件庫最典型的故障模式：**「隱形 schema 漂移」**。schema 從來沒被取消，它只是不再有人替你把關，於是漂移悄悄累積，直到讀路徑上某個假設破掉。

文件庫的第二個陷阱在「跨文件」。單文件原子更新很美好，但一旦你的操作要同時改兩份文件——比如「把一個 item 從訂單 A 搬到訂單 B」——你就離開了那塊原子性的庇護。MongoDB 從 4.0（replica set）和 4.2（sharded cluster）起確實提供了多文件 ACID 交易，底層由 WiredTiger 的 snapshot isolation 與 MVCC 支撐，sharded 場景則靠兩階段提交協調各分片（2026-06 查證）。但官方一致的建議是：**能用單文件原子更新解決的，就別開多文件交易。** 原因不只是「比較貴」這麼籠統。WiredTiger 用的是樂觀並發控制：兩個交易若改到同一份文件，後到的會收到 `WT_ROLLBACK`、整個交易被中止，然後你的應用得自己抓 `TransientTransactionError`、退避重試。高競爭下，這個「中止—重試」迴圈會把吞吐拖垮，還會因為長交易把舊版本釘在快取裡而拖累別人（這也是為什麼預設交易壽命被限制在 60 秒）。多文件交易在文件庫裡是「有，但你應該把它當作不得已的逃生口，而不是日常工具」——這跟它在關係庫裡是麵包與奶油的地位，剛好相反。

所以文件庫真正的甜蜜點很明確：**聚合根清楚、幾乎不跨聚合查、欄位多變**的場景。一旦你發現自己在應用層手寫一堆「先查 A 文件、拿到 id 再查 B 文件、在記憶體裡拼起來」的程式碼，那是個訊號——你正在用應用程式重新發明 join，而那比讓資料庫做 join 更慢、更容易出錯、還沒有交易保護。

## 鍵值庫：把查法燒進 key

如果說文件庫保留了「在文件內部還能查」的能力，鍵值庫（KV）則把這個能力幾乎砍到見骨。它的模型簡單到只剩一句話：`key → value`，value 是一坨不透明的位元組，資料庫對它的內容一無所知。給 key，O(1) 拿到 value；沒有 key，你什麼也查不到。

這種近乎殘暴的簡化，換來的是極致的速度與最容易的水平擴展——因為「給 key 拿 value」這個操作天生可以靠 key 的雜湊均勻散到任意多台機器上，沒有任何跨機器的協調需求（怎麼散、節點進出時怎麼少搬資料，見〈consistent hashing〉）。這讓 KV 成為快取、session 儲存、feature flag 這類「查法單一、要快、要能擴」場景的天選之人（Redis 的內部機制見〈Redis 的內部〉）。

但 KV 把「先想好怎麼查」這個第一原理推到了極端：**你的查詢路徑被完全燒死在 key 的設計裡。** 你想「依使用者列出所有 session」？對不起，如果 key 是 `session:{session_id}`，你根本沒有「依使用者查」這條路——除非你額外維護一個 `user:{user_id}:sessions → [session_id, ...]` 的反向索引，而那個索引得你自己在每次寫入時手動維護、自己保證它跟主資料一致。KV 把關係庫替你做的所有索引工作，原封不動丟回給你。

DynamoDB 是這個家族裡有趣的混血。它名義上是 KV/文件混合，但它真正教給人的，是「先想查法」這條原則被貫徹到底會長什麼樣。DynamoDB 的主鍵可以是單一 partition key，也可以是 partition key + sort key 的複合鍵；同一個 partition key 下的所有 item 構成一個 item collection、依 sort key 排序，而 Query 操作一次只能掃一個 item collection。這個限制聽起來綁手綁腳，卻逼出了著名的 **single-table design**：把使用者、訂單、地址全塞進同一張表，用帶前綴的 sort key（`PROFILE`、`ORDER#2026-04-26#abc`、`ADDR#home`）區分，讓「查一個使用者的 profile 加最近訂單」變成「掃一個 item collection 的一段」。要支援主鍵之外的查法，就靠 Global Secondary Index（GSI）——本質是 DynamoDB 替你維護的、另一種 key 排布的資料副本；一張表預設最多 20 個 GSI（2026-06 查證）。

DynamoDB 設計裡有個值得單獨點出的精巧機制：**sparse index（稀疏索引）**。GSI 只會索引「有那個 GSI partition key 屬性」的 item。所以如果你只在「狀態為 PENDING」的訂單上才寫入某個屬性，那個 GSI 就只含 PENDING 訂單——查「所有待處理訂單」變成直接掃這個小小的稀疏索引，不必掃全表再過濾。這是「先想查法、再設計 key」這條原則開花結果的樣子：查詢效率不是事後靠 planner 變出來的，是你在設計 key 結構時親手雕進去的。代價也很誠實——你每多一種查法，就多維護一份資料副本，多一份寫入成本與儲存成本。關係庫把這個成本藏在 planner 和索引裡讓你忘記它存在；KV 家族把帳單直接攤在你面前。

## 寬欄庫：用寫多份換線性擴展

寬欄庫（Cassandra、HBase、ScyllaDB）是「先想查法」這條原則最極端、也最反直覺的體現。它的模型是 `(partition key, clustering key) → 一列欄位`，底層是 LSM 引擎（為什麼 LSM 適合寫密集，見〈LSM-tree 與 B-tree〉），為「海量寫入 + 線性水平擴展」而生。它最著名的特性是**沒有 join**——而且這不是「暫時還沒做」，是設計上刻意不做。

為什麼一個資料庫會刻意不支援 join？因為 join 在一個橫跨幾百台機器的叢集上，意味著跨網路的資料搬運與協調，那會直接摧毀「線性擴展」這個它最想要的性質。Cassandra 的哲學是：**寧可在寫入時把資料按查詢需要的樣子擺好，也不要在讀取時花任何跨節點的力氣。** 具體做法就是反正規化到底——一張表只服務一種查詢。

這裡有個能手算的代價，把寬欄庫的取捨標得清清楚楚。假設一個電商要同時支援兩種查詢：「依使用者查他的所有訂單」和「依商品查所有相關訂單」。在關係庫裡，這是一張 `orders` 表加兩個索引（一個建在 `user_id`、一個建在 `product_id`），訂單本身只存一份。在寬欄庫裡，沒有 join、也沒有那種「一份資料供多種查法」的次級索引能高效跨分區查，於是你得建兩張表：`orders_by_user`（partition key = user_id）和 `orders_by_product`（partition key = product_id），**把每一筆訂單同時寫進兩張表**。

把數字帶進去：10 萬筆訂單，在關係庫裡是磁碟上 10 萬列資料 + 兩個索引結構。在寬欄庫裡是 **20 萬列**——同一筆訂單存了兩份。寫入吞吐要乘二（每筆訂單寫兩次）、儲存成本要乘二。如果之後產品又要「依倉庫查訂單」，那就是第三張表、第三份副本、30 萬列。這就是寬欄庫的定價單：**它用寫放大與儲存成本，買「讀路徑零 join、且能線性加機器」**。當你的寫入量大到 PostgreSQL 單機扛不住、而查詢路徑又屈指可數時，這筆交易划算到無可取代；當你的查詢路徑多變、又沒那麼大的量時，你會在維護一堆「同一筆資料的 N 份副本」上被慢慢凌遲。

寬欄庫還抽掉了一根你可能沒注意到的支柱：**強一致性預設不見了**。Cassandra 是無主（leaderless）架構，每次讀寫你都得指定 consistency level——`ONE`、`QUORUM`、`ALL` 等等。`QUORUM` 意思是「過半副本（n/2 + 1）回應才算數」；只有當讀的 quorum 與寫的 quorum 有交集（R + W > N）時，你才讀得到最新寫入（這個鴿巢原理，見〈quorum〉）。換句話說，一致性在這裡不是資料庫替你保證的常數，而是**你每次操作親手在「一致性」與「延遲／可用性」之間調的旋鈕**。

那如果寬欄庫裡真的需要「比較並交換」式的強一致——比如「這個 username 沒被註冊過才能寫入」——怎麼辦？Cassandra 提供 Lightweight Transaction（LWT），底層是 Paxos 共識（共識的代價見〈共識〉）。但它的標價很嚇人：一次 LWT 大約要 **4 個網路往返**（普通寫入只要 1 個），延遲是普通寫入的 4–10 倍（這是 Paxos v1 的經典數字；Cassandra 4.1 起的 Paxos v2 把往返大致砍半、LWT 效能約改善 50%，但要設 `paxos_variant: v2` 才啟用，且即便如此 LWT 仍遠貴於普通寫入，2026-06 查證）。這個數字本身就是一句設計警語：**在一個為「便宜地大量寫」而生的系統裡，每一次強一致操作都貴得像在懲罰你。** 它的存在是讓你在極少數真的需要線性一致的地方有路可走，而不是讓你把它當日常用。如果你發現自己每筆寫入都想包個 LWT，那不是 Cassandra 的問題，是你選錯了資料庫。

## 圖庫與時序庫：當「關係本身」或「時間本身」是主角

還有兩個家族，是為了兩種關係模型表達起來特別彆扭的資料形狀而生的。

**圖庫**（Neo4j、Amazon Neptune）服務的是「多跳關係遍歷」。想想「找出我朋友的朋友裡，跟我有共同興趣但還不是我朋友的人」——在關係庫裡，這是一連串自我 join，每多一跳就多一層 join，三四跳之後 SQL 寫到自己都看不懂、planner 也跑不動。圖庫把**節點和邊都當成一等公民**，遍歷是原生操作：從一個節點沿著邊走到鄰居，是 O(1) 的指標跳轉，不是一次 join。它擅長的正是關係庫最痛的地方——深度關聯查詢（社交網路、權限鏈、詐欺偵測裡的關聯圖）。代價在另一頭：**圖天生難分片**。關係庫裡你可以按 user_id 把資料切開，但一張圖裡「哪些節點該放同一台機器」沒有顯然的切法——任何切法都會有大量的邊橫跨切割線，而跨機器的邊就是跨網路的遍歷。超大圖的分區至今仍是個沒有漂亮答案的公開難題。

**時序庫**（TimescaleDB、InfluxDB、Prometheus）服務的是「時間戳 + 值」這種資料，查詢幾乎都是時間範圍上的聚合（「過去一小時這個指標的 p99」）。它針對 append-only 寫入和時間範圍掃描做了深度最佳化，還常自帶 retention 和降採樣（資料怎麼老去與過期，見〈資料的生命週期〉）。它的甜蜜點和它的陷阱是同一枚硬幣的兩面：**低基數標籤 + 高頻時間點**是它的主場，但如果你把高基數的維度（比如把 `user_id` 當 tag）塞進去，索引的基數會爆炸式增長、直接拖垮它（這個 cardinality 爆炸的機制，見〈cardinality 爆炸〉）。時序庫對「非時間維度的查詢」也很弱——它是為「沿時間軸聚合」而生的，你要它「找出某個特定值的所有來源」，它會很吃力。

這兩個家族提醒我們一件事：NoSQL 各家族不是「關係庫的擴展版」，而是**為某一種特定的資料形狀，把整個引擎重新長一遍**。選它們，是因為你的資料「本來就是一張圖」或「本來就是一條時間序列」，而不是因為關係庫不夠快。

## 收斂：界線正在模糊

到這裡，把各家族並排看一眼會有幫助——但記住，這張表是前面敘事的收尾，不是它的骨架。每一格背後的「為什麼」，前面都講過了。

| 家族 | 抽掉的支柱 / 換到的東西 | 甜蜜點 | 最典型的故障模式 |
|---|---|---|---|
| 關係（PostgreSQL / MySQL） | （基準）固定 schema + join + 跨表 ACID | 關係複雜、查法多變、需強交易 | 水平擴展要自己做；跨分片 join / 交易困難 |
| 文件（MongoDB） | 抽掉固定 schema，換 schema 彈性 + 聚合就近 | 聚合根清楚、欄位多變 | 隱形 schema 漂移；跨文件交易昂貴 |
| 鍵值（Redis / DynamoDB） | 抽掉查詢結構，換 O(1) + 易擴展 | 快取、session、查法單一 | 查法燒死在 key；反向索引要自己維護 |
| 寬欄（Cassandra） | 抽掉 join + 強一致預設，換線性擴展 | 寫密集、查法少而固定 | 一表服務一查、寫多份；LWT 貴 4–10 倍 |
| 圖（Neo4j） | 重做引擎讓邊成一等公民 | 多跳關係遍歷 | 超大圖難分片 |
| 時序（TimescaleDB / InfluxDB） | 重做引擎針對時間軸 | 低基數標籤 + 高頻時間點 | 高基數 tag 撐爆索引；非時間查詢弱 |

但 2026 年再看這張對立的圖，有個越來越明顯的趨勢：**界線正在模糊。** 最有力的例子是 PostgreSQL 的 `JSONB`。它讓你在一張關係表裡，把某些欄位存成二進位 JSON 文件——既能用 GIN 索引高效查文件內部的欄位，又完全活在關係庫的 ACID 交易和 join 能力之下。於是「核心欄位用結構化的關係 column，多變的 metadata 塞進一個 `JSONB` column」成了 2026 最常見、通常也是最務實的模式——你拿到了文件庫的彈性，卻沒放棄關係庫的任何一根支柱。反方向也在發生：MongoDB 補上了多文件 ACID 交易，Cassandra 補上了 LWT，許多 NewSQL 系統則想同時要關係模型和水平擴展。

這個收斂趨勢不會讓選型問題消失，但它確實改變了問題的形狀：**「我需要彈性，所以得放棄關係庫」這個十年前的二分法，今天往往不成立**——你可能只需要在 PostgreSQL 裡開一個 `JSONB` 欄位。真正逼你跳出關係庫的，不再是「schema 彈性」這種已經被吸收進來的需求，而是那些關係模型架構上給不出來的東西：**單機扛不住的寫入量、天生就是圖或時間序列的資料形狀、必須線性加機器才付得起的規模。**

## 為什麼是這個形狀

退一步看整片地景，所有這些家族其實都站在同一道張力的不同位置上：**你想把多少工作放在寫入時、多少放在讀取時。**

關係模型把賭注押在讀取時的彈性上——正規化地、緊湊地存一份，把重組事實的工作留給查詢時的 join 與 planner。它換來的是「事後任意查法」這份奢侈，付出的是「join 與跨表交易在大規模下昂貴、難分片」這筆帳。NoSQL 各家族則大多把賭注押在寫入時——寫入時就把資料擺成查詢要的樣子，甚至為不同查法寫好幾份，讓讀取時零協調、可線性擴展。它們換來的是規模與速度，付出的是彈性、是「先想好怎麼查」的紀律、是自己維護一致性的負擔。

沒有哪一邊更先進。這就是為什麼成熟系統裡 polyglot persistence（多種資料庫並用）是常態而非異端：交易性的核心資料躺在 PostgreSQL，熱資料快取在 Redis，指標在時序庫，搜尋索引在別處——讓每種資料形狀，住進為它而生的引擎。代價是每多一種庫，就多一份維運、多一條要對齊的資料同步管線、多一處可能漂移的一致性（這些跨庫同步的難題，見〈資料同步的全景〉）。

所以那場架構評審上的兩個團隊，吵的從來不是「SQL 還是 NoSQL」。他們真正該攤在桌上的問題只有三個：**這份資料天生是什麼形狀？我會用哪幾種固定的方式查它？我願意為了擴展，放棄關係庫的哪一根支柱？** 把這三個問題回答清楚，選型自己就浮出來了——而「NoSQL 比較潮、比較快」這種話，從來就不在這三個問題的答案裡。
