# 寫作風格指南《機制之下》（內部文件，非書籍內容）

本文件是全書的寫作契約。每一章都必須遵守。
書名：**《機制之下——後端與分散式系統如何運作、為何如此》**。

> **這本書的由來**：它是姊妹書 `rsys`（《保證與取捨——後端與分散式系統的概念地圖》）的**深讀版**。`rsys` 是速查／概念地圖（每條目固定六段＋對照表）；本書把**同一批主題重寫成可以坐下來讀的敘事章**。`rsys` 原封不動保留，兩書並存互補。**寫本書時，rsys 的對應條目是「已查證的底子」——拿來加深、敘事化，不是抄。**

## 這本書是什麼、不是什麼

- **是**：一本**能讀的後端與分散式系統技術書**。每一章帶讀者**真正讀懂一個機制**——它解什麼問題、底下怎麼運作、買到什麼保證、付什麼代價、在哪裡會壞、為什麼是這個形狀。讀完一章，讀者腦中是一條**理解的脈絡**，不是一格一格可掃描的條目。標竿是 DDIA（《Designing Data-Intensive Applications》）那種「讀得下去、讀完真的懂」的章。
- **不是**：
  - **不是速查表**。**不要 `rsys` 那套固定六段骨架**（定義與原理／解法空間／各方案的保證與取捨／何時需要／常見誤解與陷阱／延伸閱讀）。**不要把對照表當一章的骨幹。** 條列、對照表、pseudo-code 都可以用，但它們是敘事裡的工具，不是章的脊椎。一章如果讀起來像「定義→列點→列點」，就是失敗。
  - **不是個人書**。**完全不提任何特定作者的系統、公司、履歷、經歷**。對象是所有讀者。
  - **不是糾錯書**。**禁用「你以為 vs 實際」「老兵記憶會過時」這類預設讀者記錯的裝置**。讀者是懂行的同儕，平視對話。
  - **不是教學書（入門）**。讀者是資深工程師：不從零解釋什麼是資料庫、HTTP、交易、執行緒。每章從一個**有張力的問題或場景**切入，直接進深水區。
  - **不指向任何其他書**（包含姊妹書 `rsys`、作者其他著作）。需要的深度就在本章內講夠，或點到外部權威來源（RFC／論文／官方文件）。**章與章之間可互相引用**（見下方「章內引用」），但**不得指向本書以外的任何書**。
  - **不含練習**。沒有「紙上推演／自我檢核／動手做」段。它是一本書，不是習題本。

## 全書脊椎

和 `rsys` 同一條脊椎：**你想要什麼保證（guarantee），你願意付什麼取捨（trade-off）。** 差別在**呈現方式**——`rsys` 用對照表把取捨攤平成格子；本書用敘事讓取捨**作為「故事裡的後果」浮現**：先看到天真做法在哪裡破掉，機制怎麼補，補了之後又欠下什麼新的債。

## 讀者是誰

一般資深後端／軟體工程師（約 **3–8 年**實戰）：

- **預設已具備**：會寫程式，HTTP／資料庫／交易／並發／雲端用到工作級。不要解釋這些是什麼。
- **不預設**：讀過分散式系統理論（不假設讀過 Paxos／FLP／向量時鐘）。本書補的正是「**會用、但底下只有模糊直覺、沒系統化想透**」的那一層——把它講到機制級。
- **不寫給**：完全的新手，或分散式系統研究者。
- **語氣**：工程師對工程師、平視、有觀點。不諂媚、不打臉、不喊口號。

## 每一章的形狀（與 `rsys` 最大的差異，務必守）

**沒有固定模板。** 但一章好的技術敘事，通常會走過下面這幾個「動作」——順序與比重隨主題調整，這是**節奏感**，不是填空格：

1. **開場：一個具體的場景或問題。** 讓讀者立刻看到「沒有這個機制會出什麼事」。是張力開頭，不是定義開頭。（例：共識章從「兩台機器同時自認 leader、各自接受寫入，資料分岔了」開場，而不是「共識的定義是…」。）
2. **把問題講清楚：為什麼這件事難。** 第一原理是什麼？天真的做法在哪裡破掉？這一步建立讀者對「為什麼需要接下來那套機制」的飢渴。
3. **機制登場：解法怎麼運作，一步步講透。** 這是全章的肉。允許 pseudo-code、時序圖、手算數字。**講到讓讀者能在腦中重放這個機制**，不是貼名詞。
4. **取捨浮現：解法不是免費的。** 它買到什麼保證、付出什麼代價——織進敘事，不是另起一段對照表。有多個方案時，用敘事帶讀者走過「為什麼會演化出這幾種選擇、各自在什麼壓力下勝出」。
5. **故障模式：這個機制怎麼壞、在什麼邊界騙你。** 同樣織進敘事，當成「故事的轉折」而非條列警語。
6. **收束：為何是這個形狀。** 回到第一原理，讓讀者帶走「理解」而非「條目」。

**明確不要**：
- 不要六個固定 `###`。小標（`##`）是有機的、為這一章的敘事服務、數量不拘、命名自由（用能推進敘事的句子，不要用「定義與原理」這種模板名）。
- 不要把對照表當骨幹。需要並排比較時，先用敘事講清楚「為什麼要比、各自的取捨從何而來」，表格只是收尾的速查。
- 不要「定義 → 列點 → 列點」的速查表形狀。如果某段可以原封不動貼進 `rsys`，它就太乾了。

**章的結尾**：每章以一個簡短的 `## 延伸閱讀` 收尾，放 3–8 條**外部真實連結**（RFC／論文／官方文件／權威技術文章）。這是全書唯一的固定段（P3 lint 靠它定位）。**不得指向任何其他書。**

## 深度標準

- **深度＝把這個主題講透所需的長度。** 不准湊字數，也不准為了「像本書」而稀釋。consensus 樣章那種份量是「**需要時可以多深**」的示範，不是每章硬下限。health-check、graceful shutdown 這類主題該短就短（一個機制講透可能就 800–1500 字）；consensus、MVCC、背壓這類主題該長就長。**每一段都要推進理解**——重複、鋪陳、口號式總結一律刪。
- **深度的操作型標準：讀完一章，讀者要能「向另一個工程師把這個機制的細節講清楚」**（2026-06-21 校準）——不只知道名詞與結論，而是能解釋它**怎麼運作、為何如此、在哪會壞**。沒到這個程度就是不夠深。具體做到：把機制講到讀者能在腦中**重放**；主動帶出幾個**非顯然的邊界情況**並講透——如樣章 consensus 的「過半交集為何保證換屆不丟資料」「讀 leader 也可能讀到舊值」「fsync 說謊會怎樣」「成員變更為何危險」。但**深 ≠ 長**：edge case 點關鍵的幾個、不窮舉清單；徹底是指「每個被講的點都見底」，不是「把所有點都列一遍」。`consensus.md` 是這個徹底度的標竿。
- **技術章至少一個帶真實數字、手算到底的 worked example**，織進敘事（像 rsys 那樣的算式，但不要孤立成一段，要當敘事的證據）。例：共識延遲下限「5 節點 Raft、單程 1 ms、fsync 5 ms → 一次提交約 7 ms」。
- **判斷／文化類主題**（Part 0 的 T/U、Part 7 的 V）用**具體情境／案例**代替數字（如「一個三人小團隊要導入新 lint 規則」的推演），不硬湊數字。
- **每個機制都要講「保證 + 成立條件 + 怎麼壞」**：禁止給「最佳實踐」口號而不講它在什麼條件下成立、什麼條件下騙你。
- **決策框架優先於教條**：什麼時候用 A、什麼時候用 B——給判準。
- **嚴謹度誠實標示**：分清「規格保證的」「常見實作的行為」「作者的判斷」。別把某工具的行為當成所有同類都這樣。
- **允許不可執行的 pseudo-code／設定片段**追邏輯（資深讀者想看碼）；標清楚是示意。程式碼與註解一律英文。

## 沿用 `rsys` 的硬規則（紅線，違反即重寫）

1. **不提任何特定作者的系統／公司／履歷／經歷**（NeoBards／LetsTalk／Bahwan／結算 pipeline… 一律不准出現）。
2. **禁糾錯口吻**（「你以為 vs 實際」「老兵記憶會過時」「打臉」）。中性的「你（泛指讀者）」「我們」可用。
3. **不指向本書以外的任何書**（含 `rsys`、queue/aiib/fail/tla… 作者其他著作）。延伸閱讀只放外部真實連結。
4. **無練習段**（紙上推演／自我檢核／動手做）。
5. **AI 維度只寫原理層**（委派／驗收／review gate／信任邊界／新故障模式），**不寫任何 AI 工具版本或品牌**。只有 Part 0 的 T/U 章碰 AI；技術章（A–S、M）不強加 AI 維度。
6. 程式碼／設定／註解／log 一律**英文**；書內容與閱讀器 UI 用**繁體中文（台灣用語）**。

## 章節切法與合併

- **預設：一個 `rsys` 主題＝一章。** 章清單與合併決定見 `outline.md`。
- **可合併**：太薄、或關係太緊、拆開反而不自然的主題，合成一章（如交付語意三選一＋exactly-once 的真相→一章；at-least-once＋冪等＋去重→一章「重複是常態」）。合併後該章要有**一條完整的敘事線**串起被併的子題，不是兩篇拼一起。
- 每章一個檔＝reader 一個 doc，sidebar 顯示章的 h1 標題。檔名用英文 slug（不帶章號；閱讀順序與 Part 分組由 `web/book.config.json` 定，章號於打包時才加）。

## 章內引用與去重（owning 章）

- 同一個機制只在 **owning 章**深講一次；別章需要它時，**給讀者足夠跟上的最小說明，再用「（見〈章名〉）」指向 owning 章**，不重新推導整套機制。owning 表見 `outline.md`。
  - 深讀書比 `rsys` 寬鬆一點：別章可以為了敘事完整，**簡述**該機制（讓本章自足好讀），但**不要重講整套推導**——那是 owning 章的事。判準：讀者不必跳章也能讀懂本章的主線，但想深挖某個被引用的機制時，知道去哪一章。
- 引用格式：句尾括號 **（見〈章名〉）**，用章的標題、**不要寫死檔名或章號**（章號會在打包時才定、會變）。

## 格式

- Markdown。每章 `# <章標題>`（h1，就是這一章的名字）→ 有機的 `##` 小節（命名自由、為敘事服務）。
- ASCII 機制圖／時序圖／狀態圖用 box-drawing 時，必須過 `check_diagrams.py`（CJK=2 欄；box 中間行含全形標點易讓右框漂一欄，用 Python 精算寬度自測）。**單純的並排比較優先用 Markdown pipe 表**（不需過 check_diagrams、好維護），只有要畫機制／時序／狀態圖才用 ASCII box。
- 語言：繁體中文台灣用語，技術名詞保留英文。
