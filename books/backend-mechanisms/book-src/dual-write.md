# dual-write 問題與 event-carried state transfer

一段再普通不過的程式碼，藏著分散式系統裡最常見、也最難被發現的資料漏洞。設想一個下單的處理函式，它要做兩件事：把訂單寫進資料庫，然後發一則 `order.created` 訊息到 broker，好讓通知服務、庫存服務、報表服務都知道「有一筆新訂單」。寫起來大概像這樣：

```
async function placeOrder(order) {
  await db.insert("orders", order);          // write #1
  await broker.publish("order.created", order); // write #2
}
```

看起來無懈可擊。資料進了庫、事件發了出去，兩邊都拿到了這筆訂單。團隊跑了三個月，一切正常。直到某天有人發現：資料庫裡有一萬筆訂單，通知服務只處理過九千九百筆——有一百個使用者下了單、付了錢、訂單明明在庫裡，卻從來沒收到任何確認，下游的庫存也沒扣。沒有任何錯誤日誌，沒有任何告警。這就是 **dual-write（雙寫）問題**：一段沒有任何 bug 的程式碼，正在悄悄地、持續地讓兩份本該一致的資料分叉。

## 為什麼這兩行湊不成一件事

問題的根，是一個常被忽略的事實：**這兩次寫入，落在兩個沒有共同交易邊界的系統上。** `db.insert` 是資料庫的一次交易，`broker.publish` 是對 broker 的一次網路呼叫——它們各自原子，但它們之間沒有任何東西能保證「要嘛都成功、要嘛都不發生」。中間永遠存在一個窗：第一次寫入已經落地、第二次還沒做。任何在這個窗內的崩潰——行程被 OOM kill、機器斷電、容器被調度器收走、broker 剛好不可用——都會讓兩端對不起來。

把失敗的排列攤開看，會發現這不是「順序沒排好」能修的：

**先寫 DB、再發訊息**（上面那段碼）。若在 `insert` 成功之後、`publish` 之前崩潰：訂單在庫裡，但事件永遠不發。下游永遠不知道這筆訂單存在——**漏事件（lost event）**。這正是那一百筆訂單消失的去向。

**反過來，先發訊息、再寫 DB**。若 `publish` 成功之後、`insert` 失敗或崩潰：下游收到一個 `order.created`，興沖沖地去扣庫存、寄信、記帳——但這筆訂單在資料庫裡根本不存在，因為那次寫入回滾了。下游對著一個從未發生的事實做了一連串動作——**幽靈事件（phantom event）**。

你可能會想：那加上重試呢？發訊息失敗就重發，總能補上吧。重試確實能補住「漏」，但它馬上帶來新的麻煩：如果 `publish` 其實已經成功、只是 ack 在回程丟了，你的重試會讓同一筆事件發兩次——**重複**。於是你在「漏」「幽靈」「重複」三者之間打轉，怎麼排列順序、怎麼加重試，都只是在改變分叉的方向，消不掉那個窗。

這才是 dual-write 真正的殘酷之處：**它不是一個你寫錯了、改對就好的 bug，而是「想用兩個各自獨立的交易去達成一個原子效果」這件事在物理上做不到的後果。** 兩次獨立的寫入之間一定有一個時間間隙，而崩潰可以發生在任何一個瞬間。

更糟的是它的隱蔽性。這種分叉不會報錯——兩次寫入「各自」都成功了，沒有任何一方覺得出了問題。它留下的是無聲的漂移：資料庫和搜尋索引慢慢對不上、訂單和通知慢慢差幾筆。正如 Kleppmann 在那篇著名的〈為什麼雙寫是個壞主意〉裡說的，你通常不會當下發現，而是半年後才驚覺兩邊兜不攏——那時已經漂了幾千筆，根本不知道從哪一筆開始錯。一個會立刻爆炸的 bug 是仁慈的；dual-write 給你的是一個假裝一切正常、實則持續滲漏的系統。

## 兩個獨立交易湊不出原子性，那就只留一個交易

既然「兩次跨系統寫入」的窗無法消除，唯一的出路是不要有兩次跨系統寫入。核心洞見只有一句：**把『兩次跨系統寫入』收斂成『一次本地交易』。**

資料庫的本地交易本來就是原子的——同一個交易裡寫兩張表，要嘛兩張都成、要嘛兩張都回滾，這是資料庫早就替你保證好的。那麼，與其在交易之外去呼叫 broker，不如把「我要通知下游這件事」也當成一筆資料，寫進**同一個資料庫交易**裡：

```
BEGIN;
  INSERT INTO orders        (...) VALUES (...);   -- business state
  INSERT INTO outbox_events (...) VALUES (...);   -- "I owe the world an event"
COMMIT;
```

現在那個致命的窗不見了。崩潰若發生在 commit 之前，兩筆 INSERT 一起回滾——訂單沒成立、事件也沒記，使用者看到下單失敗、重試即可，**不會出現「訂單成立但事件遺失」這種兩端分叉**。崩潰若發生在 commit 之後，兩筆都在，事件這筆「欠條」穩穩地躺在 `outbox_events` 表裡，等著被一個獨立的投遞器（relay）讀出來、真正送到 broker。資料與「待發事件」的原子一致，就這樣靠資料庫本來就有的交易性質拿到了。這套把「待發事件」寫進同庫的設計就是 **outbox（發件匣）模式**。

outbox 本身的全套機制——relay 怎麼輪詢這張表、投遞成功怎麼標記、它換來的「至少一次投遞」語意又給下游帶來什麼義務——是另一個主題的核心，這裡只需要知道它**為什麼非用不可、以及它把問題搬到了哪裡**（投遞器與交付保證的細節見〈跨服務的交付一致〉）。

值得在這裡釘死的是它的語意取捨。outbox **不會**給你「事件恰好送達一次」。relay 在投遞之後、標記之前若崩潰，重啟後它會重送同一筆——所以 outbox 的投遞是**至少一次**。換句話說，outbox 做的是一筆乾淨的交換：**把『可能遺失』換成『可能重複』。** 這是個好交易，因為這兩種錯的可修復性天差地別——重複可以靠下游的冪等（用一個冪等鍵把重複的事件吞掉，見〈重複是常態〉）安靜地消化掉；而遺失是不可逆的，一筆事件一旦沒發，沒有任何下游機制能無中生有地把它變回來。dual-write 最可怕的失敗模式恰恰是「遺失」，outbox 把它換成了一個你有辦法處理的失敗模式。

## 算一筆帳：那個「萬分之一」其實一點都不小

抽象地說「有個崩潰窗」很容易被當成可以忽略的邊角。給它一個數字，感覺就不一樣了。

一條每天處理 **100 萬筆**訂單的路徑，走天真雙寫。假設「寫 DB 成功之後、發訊息之前剛好崩潰」這件事的機率只有 **0.01%**——萬分之一，聽起來小到像不會發生。但乘上每天 100 萬筆：

```
1,000,000 orders/day x 0.0001 = 100 orders/day
```

**每天約 100 筆**訂單，資料在庫、通知永不發出。一週就是 700 個使用者付了錢卻收不到任何確認，下游庫存少扣了 700 筆。而且——這是最扎心的地方——**沒有任何錯誤被記錄下來**，因為每一次崩潰時，那唯一一次成功的寫入（INSERT）本身是正常的；崩潰打斷的是「下一步」，而沒有「下一步」會去抱怨自己沒被執行。你不會在日誌裡看到 100 行 error，你會在三個月後的某次對帳裡看到一個對不上的總數。

換成 outbox，這個 0.01% 的窗作用在完全不同的地方。崩潰若打斷的是「commit 之前」，整筆交易回滾——使用者看到下單失敗，重試，乾淨。崩潰若打斷的是 relay 的投遞，最壞結果是同一筆 `order.created` 被送兩次——下游用冪等鍵一比對，把第二筆丟掉。**100 筆「靜默遺失」變成了最多 100 筆「被去重吃掉的重複」**，後者使用者完全無感。同樣的萬分之一崩潰率，作用在 outbox 上只是噪音，作用在裸雙寫上是每天 100 起靜默事故。這個對比就是 outbox 存在的全部理由。

## 那些看起來能繞過、其實不能的捷徑

dual-write 之所以反覆出現，是因為每個遇到它的人都會先想到幾條捷徑，而這幾條捷徑各有各的破法。把它們講透，才算真懂這個問題。

**「把順序排好就行了吧？」** 不行。前面已經拆過：任何「兩次獨立寫入」的排列都有崩潰點，調換順序只是把分叉方向從「漏事件」換成「幽靈事件」，兩個都是病。

**「broker 回了 ack，不就代表發成功、跟 DB 一致了嗎？」** 這是最迷惑人的一條。`publish` 成功只證明「訊息進了 broker」，它和你的 DB 交易之間**仍然沒有共同的原子邊界**。考慮這個順序：你先 publish、broker 回了 ack，然後才要 commit DB 交易——結果 commit 失敗回滾了。broker 那邊木已成舟，一個 `order.created` 已經發出去了，但訂單在庫裡根本不存在。你拿到了一個漂亮的 ack，也拿到了一個幽靈事件。**原子性只能來自「同一個本地交易」，不可能來自一個外部系統的回執。**

**「那讓 broker 加入分散式交易，用 2PC 把 DB 和 broker 綁成一個原子提交？」** 理論上對，這正是兩階段提交要解的事（它的阻塞代價見〈2PC 與分散式交易〉）。但現實裡這條路在訊息系統上幾乎走不通：絕大多數 broker 不參與 XA 交易，而且 2PC 帶來同步阻塞與協調者單點，是分散式系統普遍躲著走的東西。值得一提的是，Kafka 社群確實有一個提案（KIP-939）想讓 producer 能參與外部協調的兩階段提交、為「原子雙寫」鋪路——但截至 2026 年中，這項能力**仍未正式釋出**，相關的公開 API 在 2025 年底因實作未完成而被回退（2026-06）。所以「讓 broker 當交易參與者」在今天還不是一條能依賴的逃生路；outbox 之所以是主流答案，正因為它**只用你手上早就有的東西**——一個本地資料庫交易——就解決了問題，不需要任何系統去支援跨資源的原子提交。

**「我先發事件、再消費自己的事件去更新狀態，不就只有一次寫入了嗎？」** 這是個有名字的做法，叫 **listen-to-yourself（聽自己）**：源頭只 publish 一筆事件，自己的狀態更新也變成這筆事件的下游消費者。它確實把「源頭的雙寫」消掉了——源頭只有一次寫入（publish）。但它把一致性問題挪了個位置而非解決：發出事件到自己消費並落地之間有延遲，這段時間裡你自己的資料庫**還沒有**這筆訂單，任何同步讀都讀不到；而且 publish 這一步本身又回到了「broker 寫成功但後續處理失敗」的老問題。它在某些把狀態徹底重塑成事件投影的架構（event sourcing 取向）裡站得住，但作為「outbox 的輕量替代」，它常常只是把難處藏進了另一個角落，而非真的更簡單。

**「偶爾漏一兩筆，事後對帳補回來不行嗎？」** 在某些場景可以——如果第二次寫入的下游是個純分析旁路、遺失幾筆無關緊要、而且本來就有全量對帳每晚兜底（對帳機制見〈對帳〉）。判準很乾脆：**問自己「第二次寫入若靜默失敗，會不會造成使用者可見的、不可接受的不一致？」** 會（漏了訂單通知、漏扣了庫存、漏更新了搜尋索引導致賣出已下架商品）——上 outbox；不會（只是某個延遲幾小時也無所謂的報表少了一筆）——裸雙寫加對帳也許就夠，硬上 outbox + relay 反而是給一個無關緊要的旁路扛了一套要運維的元件。

## 從「通知一件事」到「同步一份狀態」

outbox 解決了「事件可靠地發出去」這半邊。但事件發出去之後，**事件裡到底裝什麼**，會徹底改變下游系統的形狀——這也是為什麼這個主題會落在「資料同步」這個範疇裡，而不只是「訊息傳遞」。

退一步看，從來源到下游的事件可以裝兩種不同份量的東西，這兩種選擇背後是一條完整的耦合光譜（兩種事件風格的完整討論見〈事件驅動架構〉，這裡只取它和資料同步直接相關的那一面）。一種事件只攜帶「發生了什麼 + 是哪個實體」——`order.created`，帶一個 `orderId`，沒了。下游收到後知道「有事發生」，但要知道這筆訂單的細節（金額、品項、收件地址），得**回頭去問來源服務**。這是 event notification（事件通知）：事件很瘦，下游和來源之間是一條「我會回來查你」的同步依賴。

另一種事件把**這個實體變更後的完整狀態**整包裝進去——`order.created` 裡帶著這筆訂單的全部欄位。下游收到後不必回問任何人，直接把這份狀態**存進自己的本地資料庫**，當成自己的一份唯讀副本。這就是 **event-carried state transfer（事件攜帶狀態傳遞）**。

從資料同步的角度看，這第二種選擇是一件相當漂亮的事：**一條事件流，持續地把一個服務的狀態變更，同步成另一個服務裡的本地讀模型。** 通知服務不再需要在每次要寄信時打電話去問訂單服務「這筆訂單長怎樣」——它手上就有一份隨著事件流不斷更新的訂單副本。這帶來幾個具體的後果，每一個都是「保證換取代價」的小交易：

- **下游與來源解耦到能各自獨立存活。** 訂單服務半夜掛了，通知服務照樣能用它本地那份副本繼續寄信、繼續回應查詢——因為它不必在請求路徑上同步呼叫訂單服務。event notification 做不到這點：來源一掛，下游那條「回來查你」的依賴就斷了。
- **讀取變快、變便宜。** 下游讀自己的本地表，沒有跨服務的網路往返。一個要聚合好幾個上游資料的頁面，若每個欄位都得即時去問對應的服務，延遲是各服務 RTT 的總和；若每個上游都用 event-carried state transfer 餵一份本地副本過來，這個頁面只查自己的庫。
- **代價是資料冗餘與「副本必然落後」。** 同一份訂單資料現在存在於訂單服務和通知服務兩個地方——你用儲存空間和「資料有多份」的複雜度，換來了解耦和讀取速度。而且這份本地副本是**最終一致**的：事件在路上要花時間，下游副本永遠落後來源一個傳播延遲。「剛在訂單服務改了地址、立刻去通知服務看」可能看到舊地址。這不是 bug，是 event-carried state transfer 用即時一致性換獨立性的必然結果。
- **每個下游各留一份副本，意味著每個下游都得自己消化重複。** 別忘了上游的 outbox 是至少一次投遞——同一筆 `order.created` 可能到兩次。下游若天真地「收到就覆蓋本地副本」，碰到重複還好（覆蓋同樣的值是冪等的）；但若事件帶的是「餘額 +100」這種增量語意，重複套用就會多加一次。所以 event-carried state transfer 的下游，最穩的做法是讓事件攜帶**完整狀態的快照**（直接覆蓋）而非增量，並帶上版本/序號好識別與丟棄過期或重複的事件。

把這兩種風格放在一起，取捨就清楚了——它其實是同一條「下游要不要持有自己的副本」的光譜的兩端：

| | event notification | event-carried state transfer |
|---|---|---|
| 事件裡裝什麼 | 「發生了什麼」+ 主鍵 | 變更後的完整狀態 |
| 下游要不要回查來源 | 要（同步呼叫） | 不要（讀本地副本） |
| 耦合度 | 較高（執行期依賴來源存活） | 較低（來源掛了仍能運作） |
| 代價 | 多一次回查的延遲與來源負載 | 資料冗餘、副本最終一致、要處理重複 |

選哪一種，回到的還是那條全書的脊椎：你想要什麼保證（下游能不能在來源不可用時繼續工作、讀取要多快），你願意付什麼代價（資料存幾份、能不能忍受副本落後、誰來消化重複）。

## 為什麼是這個形狀

把整章收束成一條線：dual-write 的所有解法，都是從一個無法迴避的物理事實裡長出來的——**你沒辦法用兩個各自獨立的交易，去達成一個跨系統的原子效果，因為它們之間永遠有一個崩潰可以鑽進去的窗。**

正因如此，你不去消除那「第二次跨系統寫入」，而是把它**降級成一筆寫進同一個本地交易的資料**——讓資料庫本來就有的原子性替你扛住一致性，這就是 outbox。正因為投遞器無法保證恰好一次，你接受「至少一次」、把不可修復的「遺失」換成可修復的「重複」，再用冪等在下游把重複吞掉。而當你決定讓事件不只通知、還攜帶完整狀態時，你就把一個「訊息傳遞」問題悄悄變成了一個「資料同步」問題：一條事件流，把一個服務的狀態，持續複製成另一個服務手裡的一份本地真相。

下次你在某個服務裡看到一張 `outbox_events` 表、或看到一個下游服務存著一份本不屬於它的資料副本，你會知道那不是誰過度設計——那是有人認真對待了「兩行看似無害的寫入湊不成一件事」這個事實，並且決定不讓那一百筆訂單，在三個月後的對帳裡才被人發現。
