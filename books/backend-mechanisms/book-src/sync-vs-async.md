# 同步與非同步通訊

凌晨的值班頁面亮起：下單成功率跌到 99.4%，使用者開始抱怨「按了結帳就轉圈、最後跳錯誤」。你翻 trace，發現失敗的請求全卡在同一個地方——下單 API 內部呼叫郵件服務寄確認信。郵件服務正在做一次例行重啟，整批請求在那裡逾時。

奇怪的是，使用者根本不在乎那封信晚幾秒到。他們在乎的是「訂單到底成立了沒」。而訂單其實早就寫進資料庫了——只是因為下單這段程式碼在 `await sendConfirmationEmail()` 那一行卡死，整個請求被郵件服務的故障一起拖下水，最後吐回 500，連帶讓使用者以為訂單失敗、又按了一次。

這一幕的根因不是郵件服務不穩，而是一個更早、更安靜的決定：**寄信這件事，被寫成了「下單必須等它回來才算完成」。** 把「A 呼叫 B」做成同步還是非同步，看起來只是寫法差異，實際上是在決定一件大事——**當 B 出問題時，A 會跟著倒下，還是能撐住。** 這一章要把這個決定講透：它到底改變了什麼、在哪幾個地方會以非顯然的方式咬你。

## 「等不等」這個問題，比你想的更基本

先把詞釘死。**同步（synchronous）通訊**是：A 發出請求後**阻塞等待**，拿到回應（或錯誤）才往下走。典型是 HTTP request/response、gRPC 的 unary call、一次同步 RPC——你呼叫，你等，你拿到結果，你繼續。**非同步（asynchronous）通訊**是：A 把請求或訊息交出去後**立刻往下走**，不等這次的結果；若之後需要結果，靠回呼、輪詢、或另一條訊息把它送回來。典型是丟一則訊息進佇列、發一個事件、登記一個 webhook 回呼。

這個區分聽起來像在講「快或慢」，但它真正決定的是兩件更底層的事：**失敗怎麼傳遞，以及兩端的命運綁不綁在一起。**

同步呼叫底下，A 和 B 在這次呼叫期間是命運共同體。B 的延遲，原封不動變成 A 的延遲——A 在那行程式碼上動彈不得，B 慢 80 毫秒，A 這次就慢 80 毫秒。B 的失敗，原封不動變成 A 當下的失敗——B 回 500 或逾時，A 手上立刻多了一個要處理的錯誤。這是一種很強的耦合：時間上的耦合（B 必須此刻就在、此刻就回）疊加同步性的耦合（A 必須卡著等）。它的好處是這份耦合換來了**確定性**：A 往下走的那一刻，它已經確切知道 B 成功了還是失敗了。流程是線性的，最好讀、最好除錯——出事時你沿著呼叫堆疊一路往下看就行。

非同步把這份命運共同體拆開。A 把請求交給一個中介（佇列、事件匯流排），就當自己的事做完了，繼續往下。B 之後自己從中介取走、慢慢處理。現在 B 慢、B 掛，A 都不直接受影響——訊息安安靜靜躺在佇列裡等 B 回來。A 換到了**獨立性**：它的可用性不再被 B 拖累。但它付出的代價是**確定性**：A 往下走的那一刻，它**不知道**B 後來成功了沒有。那個結果，要嘛它不在乎（fire-and-forget），要嘛得另闢蹊徑繞回來。

回到開場。把「寄信」從同步呼叫改成「丟一則訊息到佇列」，下單這段程式碼就不再卡在郵件服務上——訊息丟進去，下單立刻回成功，郵件服務之後自己取訊息去寄。郵件服務重啟那十分鐘，訊息就堆在佇列裡等它回來，使用者的訂單照樣秒成。我們用「信的即時性」換到了「下單的可用性」。這筆交換值不值，是這一章後半要算的帳；但先看清楚交換的是什麼——**同步買確定性、賠獨立性；非同步買獨立性、賠確定性。** 沒有哪個免費。

## 兩層「非同步」很容易被當成同一件事：`async/await` 不是非同步通訊

這裡有兩層「非同步」很容易被當成同一件事：**「同步 vs 非同步通訊」和「程式碼用不用 `async`/`await` 語法」是兩回事**，而且它們的「非同步」指的根本不是同一層東西。

看這行 Node.js：

```js
const balance = await fetchBalance(userId);  // HTTP call to balance-service
if (balance < amount) throw new InsufficientFundsError();
```

在**語言執行**這一層，`await` 確實是非阻塞的。它不會卡死整個執行緒——event loop 在等 HTTP 回應的這段時間，會跑去處理別的請求、別的 callback，這正是單執行緒的 Node 能同時服務上千連線的原因（這套機制本身值得單獨講，見〈event loop：單執行緒怎麼同時做很多事〉）。

但在**通訊語意**這一層，這段程式碼是徹徹底底的**同步**。這個邏輯流程——「查餘額 → 拿結果 → 判斷夠不夠」——卡在 `await` 那一行，非得等 balance-service 回來才能往下。它承擔 balance-service 的全部延遲，也承擔它的全部失敗：balance-service 逾時，這個 `await` 就拋例外，下面的判斷根本走不到。從「失敗怎麼傳遞、命運綁不綁在一起」的角度看，它和一個老式的阻塞式 RPC 沒有任何區別。

兩個「非同步」說的是不同的問題。語言層的非同步問的是「**這條執行緒在等的時候，能不能去做別的事**」——它優化的是單機的資源利用率。通訊層的非同步問的是「**這個邏輯流程要不要等對方的結果**」——它決定的是兩個元件的耦合。你完全可以「語言層非阻塞、通訊層同步」（上面這段就是），也可以反過來。把這兩層混為一談，會得到「我都用了 `await`，所以我的服務之間就解耦了」這個站不住的結論。真正的非同步通訊，是這個邏輯流程**發出去之後就不等這次的結果**，繼續往下，結果之後再說。

gRPC 把這兩層攤得很清楚。它的 API 在多數語言裡同時提供 synchronous 和 asynchronous 兩種風味：synchronous stub 的呼叫「阻塞到伺服器回應抵達」，是最貼近「一次本地函式呼叫」這個抽象的形態；asynchronous 風味讓你「不阻塞當前執行緒就發起 RPC」，因為「網路本質上就是非同步的」。但注意——這裡的 sync/async **還是語言層的**「執行緒等不等」，不是通訊層的「邏輯要不要結果」。一個 unary gRPC call，無論你用 blocking stub 還是 async stub 去發，**通訊語意上都是 request/response、都要那個回應才算完成**。async stub 只是讓你在等的時候能做別的，不是讓你「不要這次的回應」。

## 不是二選一，是一條光譜

把「同步 vs 非同步」想成非黑即白的開關，會錯過中間最有用的那些選擇。真實世界裡，「要不要等、怎麼拿結果」是一條連續的光譜：

- **純同步 request/response**：A 阻塞等 B 回應，直接拿到結果或錯誤。最簡單、最好懂。
- **同步但有逾時與斷路保護**：仍然等，但等有上限——設逾時，避免無限等（見〈timeout、deadline 與 budget：把等待變成有界〉）；連續失敗就跳開、不再硬打（見〈circuit breaker：三態與半開試探〉）。這是**生產環境裡同步呼叫該有的樣子**，純裸的同步呼叫只存在於教科書。
- **fire-and-forget 非同步**：A 發出就走，根本不要回應。發一個「使用者登入了」事件、丟一筆稽核紀錄——A 不在乎誰收到、何時處理。
- **非同步 + 回呼（webhook）**：A 發請求時附上一個回呼位址，B 處理完反向呼叫 A。適合「處理很慢、但結果很重要」——金流批次、第三方非同步 API（webhook 的交付、重試、簽章與冪等另有講究，見〈Webhooks：伺服器對伺服器的回呼〉）。
- **非同步 + 輪詢**：A 發請求拿到一個 job id，之後週期性問「好了沒」。適合 A 在防火牆後、無法被回呼的情境——A 完全掌控節奏，代價是「輪詢間隔＝結果延遲」，且高頻空輪會浪費。
- **非同步 request-reply over 佇列**：要結果、又不想阻塞，就用兩條佇列模擬一次「對話」。

最後這個值得停下來看，因為它把「非同步通訊怎麼還能拿到回應」這件事的機制講透了。Hohpe 與 Woolf 在《Enterprise Integration Patterns》裡把它拆成兩個小零件。第一個是 **return address（回覆位址）**：A 在送出的請求訊息**標頭**裡，附上「回覆請丟到這條佇列」。為什麼放標頭？因為這是傳輸層的元資料，不是 B 要處理的業務內容——B 只是讀出來、知道該往哪回，不該把它和訊息正文混在一起。這樣 B 不必預先寫死「回覆送去哪」，問訊息就知道，連回覆通道都解耦了。第二個是 **correlation identifier（關聯識別碼）**：A 給每個請求一個唯一 id 放進標頭，B 處理完把這個 id**原封不動**抄進回覆訊息。

第二個零件是關鍵，因為非同步打破了一個同步世界裡免費的東西——**順序對應**。同步呼叫裡，「哪個回應對應哪個請求」從來不是問題：你在這一行等，回來的就是這一行要的。但 A 同時發了一百個非同步請求、回覆陸續從回覆佇列冒出來、而且**順序完全是亂的**（誰先處理完誰先回），A 怎麼知道這條回覆是當初一百個請求裡的哪一個？靠 correlation id。A 維護一張「id → 還在等這個回覆的那個邏輯流程（或 promise）」的對照表，每收到一條回覆，就用裡面的 id 去查表、喚醒對的那個等待者。B 全程不在乎這個 id 是什麼意思——它只負責「抄回去」，這個 id 純粹是給 A 認領用的。

這就是非同步 request-reply 的全部祕密：**同步呼叫用「執行緒卡在哪一行」隱式地記住了對應關係，非同步把這份關係外顯成一個必須自己管理的 id**。你享受了不阻塞，就得自己扛起這套對帳。漏了 correlation id，回覆就認不了領；漏了 return address，B 就不知道往哪回——這兩個在同步世界裡白送的東西，到了非同步就得親手補上。

## 把帳算清楚：同步鏈條怎麼吃掉你的可用性

現在來算這一章承諾的那筆帳，看「同步買確定性、賠獨立性」具體賠多少。

回到開場那個下單流程。下單 API 自己的可用性是 **99.9%**（以 30 天月估算，每月約 43 分鐘不可用）。它同步呼叫的郵件服務，SLA 是 **99.5%**（每月約 3.6 小時不可用）。當「寄信」被寫成同步、且寄信失敗就讓整個下單失敗時，下單的成功率是兩者相乘：

```
0.999 × 0.995 ≈ 0.994
```

下單的有效可用性被拖到 **99.4%**——每月不可用時間從 43 分鐘暴增到約 4.3 小時。多出來的近 4 小時，**全是郵件服務的故障借同步這條繩子傳染過來的**。而最諷刺的是：使用者根本不需要那封信此刻寄成，他們只需要訂單成立。我們用一個無關緊要的副作用，賠掉了核心流程一個數量級的可用性。

這就是同步鏈條的數學本質：**串接的同步呼叫，可用性是各環節相乘、延遲是各環節相加。** 鏈條越長，這兩個都越糟，而且是非線性地糟。一條 A→B→C→D 全同步的呼叫鏈，A 的可用性是四個數字連乘——每多串一個 99.9% 的依賴，整體就再掉一點；A 的延遲是四段相加，任何一段抖一下，A 就跟著抖。深層的同步呼叫鏈，是脆弱性與長尾延遲最主要的來源。

把「寄信」剪成非同步之後，這條乘法就斷了。下單只依賴「把訊息成功丟進佇列」——佇列本身通常是高可用的叢集，可用性遠高於郵件服務，而且訊息一旦入列，郵件服務後續掛多久都不影響下單已經回的那個「成功」。下單回到它自己的 99.9%。我們花「信會晚一點到、且偶爾要靠監控才知道有沒有卡住」這個代價，買回了核心流程的可用性。**判準很乾淨：只有「這次回應真正需要的結果」才該同步等；其餘的副作用，全部推到非同步路徑去。**

## 延遲那一側：尾巴會在串接裡放大

可用性是一條乘法，延遲是另一條——而且延遲的故事在「尾巴」上更陰險。

設想一個 API 要同步呼叫三個下游：認證（p99 **20 ms**）、定價（p99 **50 ms**）、庫存（p99 **80 ms**）。天真的演算法是三者相加再加自身，p99 大約 **150 ms** 起跳。但真正的問題不在這個總和，而在於**尾巴會放大**。

「p99 = 20ms」的意思是：認證有 1% 的請求會比 20 ms 慢。三個下游若獨立，一個請求要「三個都落在各自的 p99 之內」的機率是 0.99 × 0.99 × 0.99 ≈ 0.97——也就是說，**約 3% 的請求會至少踩中一個下游的長尾**，被那個慢的拖到尾巴上去。下游越多，這個機率漲得越快：串十個各自 p99 1% 的下游，「至少踩中一個慢的」機率逼近 1 − 0.99¹⁰ ≈ 10%。你的系統 p99 不是由任何單一下游決定的，**是由它們之中最慢的那個決定的**。這正是 Dean 與 Barroso 在《The Tail at Scale》裡點破的事：在中等規模無關緊要的偶發高延遲，到了大規模的扇出之下，會反過來主宰整體服務的表現。同步串接，就是把每個下游的尾巴一條條疊上來。

怎麼救？回到那個判準——問每個下游：「**這次回應，真的需要它的結果嗎？**」認證要（沒通過就不能往下）、定價要（要算出價格回給使用者）。但假設庫存那一步其實只是「順便預熱一下快取」，不影響這次要不要回應、回什麼——那它就不該待在同步路徑上。把它改成 fire-and-forget（丟個訊息讓背景去做），同步路徑就只剩認證＋定價，p99 從 150 ms 砍到約 **70 ms**，少了一半多。而且少的不只是平均——你還把庫存那條最長的尾巴，整個從關鍵路徑上摘掉了。

但別忘了帳的另一面。那筆庫存預熱現在是「最終會做」——如果它失敗了，**A 不會知道**。在同步世界裡，庫存呼叫失敗會當場拋給 A、A 能立刻反應；推成非同步之後，這個失敗從「同步拋給呼叫者」變成了「事後躺在佇列或死信佇列裡」。非同步不是讓錯誤消失，是把錯誤**移到別的時空**——你必須在那個時空（佇列監控、死信佇列、見〈死信與毒訊息：DLQ〉）擺好接住它的網。**沒有監控與死信佇列的非同步，是把錯誤掃到地毯下面**，看起來乾淨，其實只是你還沒發現它。

## 非同步偷不走的東西：強一致的當下確認

非同步這麼好，是不是能把所有同步呼叫都改掉？不能。有一類東西，非同步**從原理上**就給不了，硬改就是製造 bug。

那就是**需要「此刻就確定地知道結果」才能往下決定的步驟**。最典型的是「扣款成功了，才能確認訂單成立」。如果你把扣款改成非同步——下單時丟個「請扣款」的訊息就回使用者「訂單成立」，扣款在背景慢慢做——你就憑空造出了一個危險的窗口：**訂單已經對使用者顯示成立，但款其實還沒扣，甚至可能扣失敗（餘額不足、卡片過期）。** 這段「已承諾、未兌現」的不一致，要嘛你之後得撤銷已經成立的訂單（把難堪丟給使用者），要嘛你得跑一整套對帳與補償流程去收拾（見〈跨服務的交付一致：outbox 與 saga〉）。你用非同步換來的吞吐與可用性，在這裡換的是「正確性」——而正確性不該拿來換。

所以「該同步還是該非同步」的判準，最終落在一個問題上：**這次回應，需不需要對方的結果來做下一步的決定？**

- **需要 → 同步**（並一定要配逾時與斷路保護，把等待變成有界）。查餘額才能判斷夠不夠扣、驗證 token 才能決定放不放行、扣款成功才能確認訂單——這些是「拿到結果才能往下」的步驟，它們的時間耦合不是缺陷，是**業務正確性的要求**。
- **不需要 → 非同步推走**。通知、稽核、索引更新、快取預熱、扇出給多個下游、吸收流量尖峰——這些副作用「最終做完」就行，沒道理讓它們綁架核心流程的可用性與延遲。

特別要警惕一個反向的錯誤：**因為「非同步比較快、比較潮」，就把需要強一致確認的步驟也推成最終一致。** 非同步換來的永遠是「可用性與吞吐」，付出的永遠是「即時一致性」——這筆交換在通知、稽核這種地方划算到不行，在扣款、扣庫存這種「必須當下成立」的地方則是災難。工具的甜頭不該蓋過業務的硬約束。

## 一個同步世界免費、非同步世界要命的細節：結果其實是「三態」

最後講一個真正非顯然、卻會在生產裡反覆咬人的邊界——它連純同步也躲不掉，只是非同步把它放得更大。

我們直覺裡，一次呼叫的結果是二元的：成功，或失敗。但網路上的呼叫，結果其實是**三態**：成功、明確失敗、以及——**不知道**。

gRPC 的文件把這件事講得很坦白：呼叫的成敗，「client 和 server 各自獨立、在本地做出判定，而兩邊的結論可能對不上」。設想一個 unary call，server **已經處理完、也改了狀態**，回應正要送回的路上，client 端的 deadline 先到了——client 收到 `DEADLINE_EXCEEDED`，認定「失敗」。但 server 那邊明明**成功**了。這不是 bug，這是分散式呼叫的本質：**逾時只能告訴你「我沒在我等的窗口內收到回應」，它永遠無法告訴你「對方到底做了沒」。** 那個「不知道」是真實存在的第三態，不是你哪裡寫錯了。

這個三態在同步世界已經夠麻煩。你 `await` 一個扣款 API，逾時了——你**不能**就當它失敗、直接放使用者走，因為款可能真的扣了；你也**不能**就當它成功，因為可能真沒扣。你卡在「不知道」裡，唯一的出路是**帶著同一個冪等鍵去重試**（見〈重複是常態：冪等與去重〉）：如果上次其實扣成功了，重試會被冪等擋掉、回給你那次的結果；如果上次其實沒扣，重試會補上。冪等，就是用來把「不知道」這個第三態，安全地收斂回「成功」的。

到了非同步，這個三態被放得更大、也藏得更深。你發了一則訊息進佇列——它真的入列了嗎？發送 ack 在回程丟了，你以為沒發成功、重發了一次，於是下游收到**兩則**。下游處理完一則訊息——它的 ack 在回程丟了，broker 以為它沒處理、把訊息**再投一次**。非同步系統裡到處是這種「做了但對方不知道」的縫，這正是為什麼「至少一次投遞＋冪等去重」會成為非同步訊息的標準姿勢，而不是錦上添花的選配（見〈三種交付語意〉）。同步呼叫至少把這個三態壓縮在一次呼叫的窗口裡、當場逼你面對；非同步把它撒進了整條管線的每一個交接點。**你選非同步買到了獨立性，但這份獨立性的底價，就是你必須把每一個交接點都設計成「重複一次也不出錯」。**

## 為什麼是這個形狀

退一步看，「同步還是非同步」從來不是一個風格選擇，它是在回答一個更根本的問題：**這次互動裡，我需不需要對方「此刻就確定的結果」？**

需要，你就得接受同步的代價——A 的命運綁上 B，B 的延遲變你的延遲、B 的失敗變你當下的失敗，可用性連乘、尾巴疊加。你付這個代價，換的是「往下走的那一刻，我確切知道發生了什麼」這份確定性，而有些決定（扣款、授權、讀當前值）非有這份確定性不可。

不需要，你就能把繩子剪開——A 把事情交給中介就走，B 慢、B 掛都不再拖垮 A。但獨立性不是白給的：你失去了「當下就知道結果」，於是得自己補回那套機制——correlation id 來認領亂序回來的回覆，死信佇列與監控來接住事後才浮現的失敗，冪等來收斂那個「做了沒做不知道」的第三態。

同步和非同步，是同一道張力——**確定性 vs 獨立性**——的兩個極端，中間是一整條光譜，每一檔都是這道張力的不同配比。下次你在一段程式碼前猶豫「這個呼叫該不該 `await` 它的業務結果」，別問「哪個比較快」，問「**這次回應，真的需要對方的結果嗎**」。答案會把你放到光譜上對的那一格——也會告訴你，剪開那條繩子之後，你還欠下了哪幾筆得親手償還的債。
