# 鎖與死結：行鎖、gap、next-key 與 wait-for graph

兩個轉帳請求在同一毫秒落到資料庫。第一個要把錢從帳戶 1 轉到帳戶 2，第二個方向相反，從 2 轉回 1。兩段程式碼長得幾乎一樣，都是先扣一邊、再加一邊，看起來人畜無害——直到某天監控冒出一行 `Deadlock found when trying to get lock; try restarting transaction`，其中一筆轉帳被資料庫一刀砍掉、整個交易回滾，使用者那邊看到的是一個莫名其妙的失敗。

沒有人寫錯任何一行 SQL。兩筆交易各自都完全合法，問題出在它們**交錯**的方式。要看懂這行錯誤從哪來、為什麼是資料庫主動殺掉其中一個、以及怎樣才能讓它不再發生，得先回到一個更基本的問題：當多個交易同時要改同一份資料，引擎到底拿什麼把它們排開。

## 為什麼需要鎖：MVCC 擋不住的那一半

並發控制有兩台引擎在跑。多數現代 OLTP 資料庫的主力是 MVCC——寫不覆蓋舊值，而是產生新版本，讀去找對自己可見的舊版本，於是讀者和寫者不必互相等待（這套機制怎麼運作，見〈MVCC：讓讀不擋寫〉）。MVCC 解決的是**讀寫互擋**：一個長報表在跑的時候，寫入照樣進得來，因為它們動的是不同的版本。

但 MVCC 解不了**寫寫衝突**。兩個交易要同時改同一列，引擎不可能讓它們各寫各的新版本然後各自以為成功——那樣其中一筆更新就憑空消失了（lost update）。所以同一列的並發寫，最終一定得**串起來**：第二個 `UPDATE` 必須等第一個 commit 或 rollback，才知道自己該在哪個版本上接著改。這個「等」，就是鎖。

鎖的本質樸素到近乎廢話：在一個資源上掛一個標記，要動它的交易得先拿到相容的鎖，拿不到就排隊。寫鎖（exclusive，X）彼此互斥，一列上同時只能有一個交易持有；讀鎖（shared，S）彼此相容，但和寫鎖互斥。把「鎖什麼、鎖多久」這兩個旋鈕轉一轉，就調出了不同的隔離級別（隔離級別到底在防哪些異常，見〈隔離級別與異常〉）。鎖真正的麻煩，不在它擋住誰——那正是它的工作——而在它們會**互相等成一個環**。

## 鎖的粒度：從鎖一張表到鎖一個間隙

引擎能鎖的東西有粗有細。最粗的是表鎖，一鎖整張表別人全部停擺，簡單但並發等於零；細一點是頁鎖，鎖住一個資料頁；OLTP 真正倚賴的是**行鎖（row lock）**，只鎖住要動的那幾列，讓不相干的列照樣並發。

但這裡有個容易被略過的事實，它是後面所有 InnoDB 死結的根源：**InnoDB 的行鎖鎖的不是「列」，而是「索引記錄」。** 引擎沒有「鎖住第幾列」這種概念，它只會鎖索引上的條目。`UPDATE ... WHERE id = 5` 之所以「鎖住列 5」，是因為它在主鍵索引上找到 `id=5` 這筆索引記錄、鎖住它。如果一張表連個索引都沒有，InnoDB 會偷偷建一個隱藏的叢集索引，拿它來上鎖——鎖永遠落在索引上，沒有例外。記住這一點，「為什麼一個只改一列的 `UPDATE` 會跟別人死結」才講得通。

光有「鎖住一筆記錄」還不夠擋住所有並發異常。考慮幻讀（phantom）：交易 T1 跑 `SELECT ... WHERE age > 30`，讀到三列；它還在進行中，T2 插入了一筆 `age = 35` 的新列並 commit；T1 再跑同一個查詢，這次讀到四列。憑空多出來的那一列就是幻影。要擋住它，光鎖住「已經存在的三列」沒用——問題出在**還不存在、但符合條件的那些列被插進來了**。你沒辦法鎖一個還不存在的東西。

InnoDB 的答案是去鎖**空隙**。它有三種鎖，疊起來正好把這個洞補上：

- **record lock（記錄鎖）**：鎖一筆索引記錄本身。`WHERE id = 10 FOR UPDATE` 鎖住 `id=10` 那筆記錄，別人改不了、刪不了它。
- **gap lock（間隙鎖）**：鎖住兩筆索引記錄**之間的空隙**——注意，是空隙，不是任何一筆真實記錄。它的唯一作用，按官方文件的原話，是「**purely inhibitive**」：它**只阻止別人往這個間隙插入新列**，不擋讀、不擋對既有列的修改。`WHERE c1 BETWEEN 10 AND 20 FOR UPDATE` 會讓別人插不進一個 `c1=15`。
- **next-key lock**：record lock 加上**它前面那段 gap** 的組合，鎖住「這筆記錄 + 記錄之前的空隙」。

把這三種疊起來看就懂了：InnoDB 在 Repeatable Read 級別下做範圍掃描時，**預設對掃到的每一筆都上 next-key lock**。於是「範圍內的所有列 + 範圍內的所有間隙」被一整片鎖住，別人既改不了既有的列、也插不進新的列——幻讀被連根擋死。這正是 MySQL 的 RR 和 PostgreSQL 的 RR 走上岔路的地方：PostgreSQL 用純快照（snapshot）讓幻影根本不出現在你的視野裡，InnoDB 用 next-key lock 把間隙**鎖起來**不准別人動。一個靠「看不見」，一個靠「鎖得住」（兩者哲學的全面對照，見〈PostgreSQL 與 MySQL：兩種隔離哲學〉）。

## gap lock 的脾氣：它比你想的安靜，也比你想的危險

gap lock 有兩個違反直覺的性質，不知道的話會在兩個相反的方向上踩坑。

第一個方向是「它比你以為的更**寬容**」。gap lock 彼此**不衝突**——官方文件明說「shared 和 exclusive 的 gap lock 沒有區別，它們互不衝突」。兩個交易可以同時持有**同一個間隙**的 gap lock，誰也不擋誰。更進一步，真正的插入並不直接撞 gap lock，而是先取一種叫 **insert intention lock**（插入意圖鎖）的特殊 gap lock。它的設計很聰明：兩個交易要往同一個間隙的**不同位置**插入（一個插 5、一個插 6，間隙是 4 到 7），它們不必互相等待，因為插出來的列不衝突。要不是這個機制，所有並發 `INSERT` 落在同一段範圍時就會排成一條長龍。

第二個方向是「它比你以為的更**危險**」，而這正是 InnoDB 隱性死結的溫床。`DELETE FROM t WHERE status = 'pending'` 在 RR 下，會對 `status` 索引上所有 `pending` 的記錄、以及它們之間和前後的**所有間隙**上 next-key lock。如果這個範圍很寬，你等於鎖住了一大片誰也插不進去的區域——一個完全不相干、只是想插入一筆 `status='pending'` 新工作的交易，會卡在這片 gap 上動彈不得。當 `WHERE` 條件**命不中索引**時更糟：InnoDB 沒得選，只能退化成掃更大的範圍、鎖更多的間隙，甚至接近鎖整張表。一條看似只刪幾列的語句，實際上凍結了一整段索引。

那有沒有辦法讓 gap lock 不要亂張嘴？有，而且這是一個漂亮的非顯然邊界：**當你用唯一索引精確查一筆唯一的列時，gap lock 會被關掉。** 官方文件說得很死——`SELECT * FROM child WHERE id = 100`（`id` 是唯一索引）只會對 `id=100` 那筆上 record lock，**不上 gap lock**，因為唯一索引保證了「等於 100 的列至多一筆」，根本沒有「會不會多冒出一筆 id=100」的幻讀風險，鎖間隙是多餘的。但這個豁免有個藏在括號裡的陷阱：如果你查的是一個**多欄唯一索引**、`WHERE` 卻只給了其中一部分欄，或者那個欄根本沒索引、或只有非唯一索引——那麼「等於這個值的列只有一筆」就不再成立，gap lock **照樣回來**。還有一個更隱蔽的前提：豁免只在那筆唯一的列**真的存在**時成立；如果 `id=100` 根本查不到，InnoDB 反而會在它**該在**的那個間隙上加 gap lock——因為這時它要擋的正是「有人趁隙插進一筆 `id=100`」的幻讀。唯一索引上的等值查詢是鎖範圍的最小情況；只要偏離一點點——多欄只給一半、走錯索引、甚至查了個不存在的值——鎖就重新撐開。

## 死結：當等待連成一個環

現在回到開場那兩筆反向轉帳，慢動作重播一次。

T1 先跑 `UPDATE acct SET bal = bal - 100 WHERE id = 1`，在 `id=1` 上拿到 X 鎖。幾乎同時，T2 跑 `UPDATE acct SET bal = bal - 100 WHERE id = 2`，在 `id=2` 上拿到 X 鎖。到這一步天下太平，它們鎖的是不同的列。

接著 T1 走到第二步 `... WHERE id = 2`——它要鎖 `id=2`，但 `id=2` 在 T2 手上，於是 T1 開始等。同一瞬間 T2 也走到它的第二步 `... WHERE id = 1`——它要鎖 `id=1`，而 `id=1` 在 T1 手上，T2 也開始等。

```
   wait-for graph (deadlock = a cycle)

   +-----------------+          +-----------------+
   |       T1        |--wait--> |       T2        |
   |  holds lock(1)  |          |  holds lock(2)  |
   |  wants lock(2)  | <--wait--|  wants lock(1)  |
   +-----------------+          +-----------------+
        cycle  T1 -> T2 -> T1  : engine aborts one victim
```

兩個交易各自抱著對方要的鎖、又都在等對方放手，誰也不肯先退。這就是死結——不是誰寫錯了程式，而是兩個合法的執行序，以一個**循環依賴**的順序撞在一起。沒有外力介入的話，它們會永遠等下去。

引擎怎麼知道死結發生了？它在內部維護一張 **wait-for graph（等待圖）**：節點是交易，「T1 在等 T2 持有的鎖」就畫一條 `T1 → T2` 的邊。死結的數學定義乾淨利落——**圖裡出現環，就是死結**。InnoDB 預設開著死結偵測（`innodb_deadlock_detect = ON`），每當一個交易要進入等待，它就順著等待邊往下找，看會不會繞回自己。一找到環，它不會傻等，而是**主動選一個受害者、把它整個回滾**，斬斷這條環，讓剩下的交易繼續跑。被斬的那個交易，收到的就是開場那行 `Deadlock found`。

選誰當受害者不是隨機的。InnoDB 的原則是**挑代價小的下手**——按官方文件，「交易的大小由它 insert/update/delete 的列數決定」，它傾向回滾那個改動列數較少的交易，因為回滾它要撤銷的工作最少。所以在兩筆規模相當的轉帳裡，誰被殺有點看運氣；但如果一邊是改了一列的小交易、另一邊是改了上萬列的批次作業，被犧牲的通常是那筆小的。這也帶出一個實務含意：被殺的是**交易**，不是語句，整個交易的工作全部白做，應用端必須準備好**從頭重試**。

## 兩種偵測哲學：畫圖找環，還是等到逾時

「圖裡找環」不是唯一的死結對策。把所有手段攤開，本質是**主動偵測**和**被動逾時**兩條路。

PostgreSQL 和 InnoDB 預設都走**偵測**這條：維護 wait-for graph、找環、殺受害者。但 PostgreSQL 有一個常被誤解的細節，藏在 `deadlock_timeout` 這個參數名裡。這個名字會誤導：直覺會把它讀成「等鎖等這麼久就放棄」，但它不是放棄的門檻。官方文件的原話是：它是「在一個鎖上**等多久，才開始去檢查**有沒有死結」。檢查死結（跑一遍找環的演算法）本身不便宜，所以 PostgreSQL 樂觀地賭「死結很罕見」，先安靜等一段時間（**預設 1 秒**），等超過了才花力氣去畫圖找環。換句話說，`deadlock_timeout` 調的是**偵測的延遲**，不是放棄的門檻。把它調大，省下的是無謂的死結檢查開銷，代價是真死結要更久才被發現和解開。絕大多數正常的鎖等待在這 1 秒內就放掉了，根本不會觸發檢查——這正是設計的用意。

另一條路是**逾時放棄**，不畫任何圖：等一個鎖超過某個時限就直接放棄這次嘗試。InnoDB 的 `innodb_lock_wait_timeout`（**預設 50 秒**）就是這個角色——當死結偵測被關掉時，它退回來當安全網；即使偵測開著，它也兜住那些不構成環、但就是遲遲拿不到的「長等待」。逾時這條路實作簡單、不用算圖，但兩頭不討好：它會**誤殺**——一個只是慢、並沒有死結的交易，等滿逾時就被砍；它也**反應遲鈍**——一個真死結，得乾等滿整個逾時才解開。這個差距大到是量級之分：純逾時下，受害交易最壞要卡滿 `innodb_lock_wait_timeout`，預設就是 50 秒（50,000 毫秒）才被放掉；而偵測一旦發現新的等待邊，只要沿著等待圖走一趟找環就能斷案，這趟遍歷的節點數是當下活躍交易數的等級，通常微秒到次毫秒就完成，兩者差了約五個數量級（InnoDB 在偵測開啟時就是這樣靠找環秒解，逾時只是後備，所以 50 秒這個值平時不太會痛）。逾時值因此極難調：調短了誤殺正常的慢交易，調長了真死結拖很久。這也是為什麼成熟引擎寧可花力氣維護 wait-for graph，把逾時降格成後備。

## 怎麼不再死結：一行排序約定的威力

死結偵測能讓系統**從死結中活下來**，但它不能讓死結**不發生**——每次發生都有一個交易被犧牲、要重試，使用者那端就是一次失敗或一次卡頓。真正的解法是從源頭讓那個環**無法形成**。

回到兩筆反向轉帳。死結之所以成環，是因為 T1 按 `1 → 2` 的順序拿鎖，T2 按 `2 → 1` 的順序拿鎖，兩個方向相反，於是頭尾咬在一起。如果我們強制**所有交易都按同一個固定順序拿鎖**——比方說永遠先鎖 `id` 較小的那一列——環就再也連不起來。

具體做：兩筆轉帳在動任何一列之前，都先把要碰的 id 排序，永遠從小的開始鎖。

```sql
-- both transfers lock rows in ascending id order, no matter the direction
-- transfer money between account `a` and account `b`
SELECT * FROM acct WHERE id IN (LEAST(:a, :b), GREATEST(:a, :b))
  ORDER BY id
  FOR UPDATE;     -- acquire X locks in a globally consistent order
-- now do the debit and credit; the lock order can never form a cycle
```

現在 T1 和 T2 都會先去搶 `id=1`，搶到的那個接著從容拿 `id=2`，搶輸的那個在 `id=1` 上排隊等——這是一條**直線**的等待，不是環。等待圖裡只有 `T2 → T1` 一條邊，永遠繞不回去。一個拿不到鎖的交易會等，但不會死結；它頂多慢一點，不會被殺、不用重試。**固定加鎖順序**是死結最徹底的預防——它從根本上讓 wait-for graph 不可能成環。代價是紀律：全應用都得遵守同一個排序約定，而當「要鎖哪些列」是執行時才動態算出來的，排序就沒那麼好套。

排序之外還有兩條減壓路徑，方向不同但都有效。一是**縮短持鎖時間**：鎖從拿到到 commit 為止一直握著，所以交易裡夾任何慢動作——等一個 HTTP 回應、等使用者在前端按確認、跑一段長運算——都是在拉長別人等你的時間、放大撞死結的窗口。一個健康的寫交易應該短到「拿鎖、改值、commit」一氣呵成。二是把高衝突的更新挪去用**樂觀並發控制**，讀的時候不上鎖、只在寫回時用版本檢查擋住過期的寫（這套和悲觀鎖的取捨，見〈樂觀與悲觀並發控制〉）——衝突率低時，這比讓大家排隊拿鎖划算得多。

## 死結不是 bug，是要被「重試」掉的東西

關於死結，最該扭轉的一個認知是：**它不是程式邏輯的錯誤，而是高並發下的正常物理現象。** 只要有多個交易、各自鎖多個資源、而且拿鎖順序可能交錯，死結在數學上就可能發生。指望「寫得夠仔細就不會死結」，等於指望兩條在十字路口的車永遠不會同時搶進去——只要沒有號誌（固定順序），總有撞上的時候。

所以正確的姿態不是消滅死結的可能性（除非你能對全系統強制加鎖順序），而是**讓被犧牲的交易能安全地從頭重試**。這對交易本身提出一個要求：它必須是可以整個重放而不出錯的——這通常意味著它得是**冪等**的，或至少整個包在交易語意裡，回滾後重來不會留下半截副作用（冪等到底是什麼、冪等鍵怎麼設計，見〈重複是常態：冪等與去重〉）。一個寫對的寫入路徑，外層應該有一圈「捕捉死結錯誤碼 → 退一步 → 重試這個交易」的邏輯，重試幾次仍失敗才真的往上拋。把死結當成「偶爾要重試一次」的常規成本來設計，而不是當成「不該出現的 bug」來消滅，才是和這個機制和平共處的方式。

有幾個藏得很深的死結來源，值得單獨點名，因為它們會在「我明明只動了一列」的情況下冒出來：

- **走次級索引的更新**。`UPDATE ... WHERE email = ?`（`email` 上有非唯一索引）可能在次級索引和主鍵上**各加一把鎖**，加鎖的先後和另一個走不同索引的交易反序相遇，就成環。一個 `UPDATE` 看著只動一列，鎖其實落在兩個索引上的兩個位置。
- **外鍵檢查的隱形鎖**。插入或更新一筆帶外鍵的子表記錄時，InnoDB 會默默對**父表**那筆被參照的記錄加共享鎖，確保它在交易期間不被刪。這把你看不見的鎖，常是「兩個操作不同子表的交易卻死結」的元兇。
- **gap lock 串起不相干的插入**。前面說過的，RR 下一個寬範圍的 `DELETE` 或 `UPDATE` 鎖住一大片 gap，把幾個原本毫不相干、只是想往這片範圍插入的交易，硬是串進同一個等待環裡。

## 為什麼是這個形狀

把這一章倒過來看，鎖與死結的整個樣貌，是從一個無法迴避的事實裡長出來的：**同一份資料的並發寫，最終必須被串成某個順序，而「串」就意味著「等」，「等」一旦交叉就成環。**

正因為寫寫衝突不能靠多版本各退一步解決，所以一定要有鎖。正因為要擋的不只是「既有的列被同時改」、還有「符合條件的新列被插進來」，所以鎖不能只鎖記錄、還得鎖記錄之間的**間隙**——於是有了 gap lock 和 next-key lock 這套鎖「不存在之物」的精巧設計。正因為多把鎖按不同順序拿就會循環依賴，所以死結是並發的內生現象，引擎只能用 wait-for graph 找環、犧牲一個來解套，而不能保證它不發生。也正因為偵測只能事後補救、無法事前杜絕，真正的根治才落在應用這一側——一個固定的加鎖順序，讓環在數學上無法成形。

鎖不是資料庫強加給你的累贅，而是「多人同時改一份資料、又不能弄丟任何一筆」這個需求被逼到牆角後，唯一站得住的形狀。下次看到那行 `Deadlock found`，你會知道它不是在報告一個 bug——它是在告訴你，引擎剛剛替你斬斷了一個本來會永遠卡住的環，現在輪到你的重試邏輯接手了。
