# LSM-tree 與 B-tree：寫放大換讀放大

把一筆 100 byte 的更新寫進資料庫——`UPDATE users SET last_seen = now() WHERE id = 42`——磁碟上實際被寫下去的，有多少位元組？

直覺會說「差不多 100 byte 吧，再加一點索引開銷」。真實的答案是：可能是 16 KB，也可能是好幾百 byte 被反覆重寫五六次累積成上千 byte。哪一種，取決於這台資料庫底層用的是 B-tree 還是 LSM-tree。同一筆邏輯上微不足道的更新，落到磁碟上會膨脹成幾十倍甚至上百倍的物理寫入——這個膨脹倍率叫**寫放大（write amplification）**，而它不是 bug，是儲存引擎為了「把資料維持成可以快速查詢的形狀」必須付出的代價。

兩大流派付這個代價的方式截然不同，而且方向相反：一個在你寫的當下就把帳結清，一個把帳賒到背景慢慢還。理解這個差異——以及它連帶換來的**讀放大**與**空間放大**——你就握住了「為什麼有些資料庫天生適合寫密集、有些天生適合讀密集」這條看似玄學的選型直覺底下的硬機制。

## 問題的根：磁碟喜歡順序，資料要求有序

退到第一原理。一個儲存引擎要同時滿足兩個彼此拉扯的需求。

第一，資料在邏輯上**必須是有序的**，至少要能高效地依鍵查找與範圍掃描。`WHERE id = 42` 要能不掃全表就定位到那一列；`WHERE created_at BETWEEN ...` 要能順著時間連續讀一段。沒有某種有序結構，每次查詢都是全表掃描，O(N) 不可規模化。

第二，**磁碟對順序寫和隨機寫的偏好天差地遠**。傳統機械硬碟上，順序寫不用移動磁頭、不用等碟片轉到對的位置，比隨機寫快上兩三個數量級。SSD 沒有磁頭，但它有自己的偏好：它以 page 為讀寫單位、以遠大於 page 的 block 為抹除單位，隨機的小寫會引發寫前抹除、引發內部的搬移整理（這本身就是 SSD 控制器層的另一重寫放大），還會啃蝕快閃記憶體有限的抹寫壽命。無論哪種介質，「把寫入變成一條順序的、大塊的流」都是性能上的大勝。

矛盾就在這裡：**「維持有序」傾向於就地修改——你想把新值放到它在排序中該在的位置，而那個位置幾乎一定是磁碟上某個隨機的點。但「順序寫」要求你別到處亂跳、只往尾巴追加。** 一個引擎不可能同時把「資料永遠就地有序」和「寫入永遠是順序追加」都做到——這兩個目標在物理上互斥。

於是分岔出現了。B-tree 選擇站在「就地維護有序」這邊，吞下隨機寫的代價；LSM-tree 選擇站在「永遠順序追加」這邊，把「維持有序」這件事延後、外包給背景。這一個選擇，決定了它們後面所有的脾氣。

## B-tree：在寫的當下把帳結清

B-tree（實務上幾乎都是它的變體 B+tree）是一棵又寬又矮、永遠保持平衡的搜尋樹。它的設計初衷——1972 年 Bayer 與 McCreight 那篇〈Organization and Maintenance of Large Ordered Indexes〉——就是為了「在以 block 為單位讀寫的磁碟上，用最少的磁碟存取量維護一個大型有序索引」。這句話裡每個字都還在今天的資料庫裡活著。

它的形狀直接對應磁碟的物理：樹的每個節點是一個固定大小的 **page**（PostgreSQL 預設 8 KB，InnoDB 預設 16 KB），剛好對齊磁碟的 block I/O 單位。一個 page 能塞進幾百到上千個鍵，所以樹的扇出（fanout）極高、樹高極矮。對一個有十億列的表，樹高也不過三四層——意思是任何一筆查詢，從根節點走到葉節點，只需要三四次磁碟讀取就命中。查詢路徑長度約 `log_B(N)`，B 是每頁的鍵數，這個對數的底大到讓樹幾乎永遠是矮的。

讀，因此乾淨利落：一條從根到葉的路徑，命中即得，而且命中的就是最新值——因為值是就地更新的，葉子裡那個 `id=42` 的位置永遠只有一份、永遠是最新的。讀延遲穩定、可預測，範圍掃描更是漂亮（B+tree 的葉節點之間用雙向鏈結串起來，順著鏈一路掃就是有序的範圍讀）。這是 B-tree 的甜蜜點。

代價藏在寫。要更新 `id=42` 這一列，引擎得：從根走到葉，找到那個 page，在 page 裡就地改掉那幾個 byte，然後**把整個 page 刷回磁碟**。注意最後這一步——磁碟的最小刷盤單位是 page，你改了 100 byte，但落盤的是整個 16 KB。這就是 B-tree 寫放大的第一個來源，而且它不講道理地殘酷：

一筆 100 byte 的更新，落在一個 16 KB 的 InnoDB page，最小寫入單位是那 16 KB，**頁層級的寫放大就是 16384 ÷ 100 ≈ 164 倍**。

你只想改一個時間戳，磁碟卻寫了一整頁。如果你的工作負載是「對一張大表做大量分散的小筆更新」——每筆都打到不同的 page——那每一筆都是一次「為了 100 byte 寫掉 16 KB」的隨機寫。這就是為什麼 B-tree 在高頻小寫的工作負載下會撞牆：不是 CPU 不夠，是磁碟被一頁一頁的隨機刷盤頂死了。

而且這還沒完。B-tree 維持平衡是有動作的：當一個 page 被塞滿、再插不進新鍵，它得**分裂（page split）**——把這個 page 一切為二，各搬一半的鍵到新 page，還要更新父節點的指標。分裂意味著更多的 page 寫入，還會在磁碟上留下沒填滿的 page、造成碎片（fragmentation），讓本來緊湊的樹漸漸鬆散、讀效率緩慢下滑（這也是為什麼 B-tree 系統需要定期 `REINDEX` 或 `OPTIMIZE TABLE` 來重整碎片）。

這裡有個常被忽略的細節值得停一下：B-tree 系統的「寫」其實是兩段。你不能只是改了 page 就回報成功——萬一改到一半斷電，那個 16 KB 的 page 會是個半新半舊的爛狀態。所以幾乎所有 B-tree 引擎前面都擋著一道預寫日誌（WAL，見〈WAL：先寫日誌，再改資料〉）：變更先以順序追加的方式寫進日誌、落穩，**那一刻才算 committed**，髒掉的 page 可以稍後再從容刷回。所以 B-tree 的世界裡其實已經偷偷藏了一條「順序寫」的捷徑來拿持久性——它不是純粹的就地更新，而是「順序日誌拿 durability + 就地更新維護查詢結構」的組合。記住這一點，因為 LSM-tree 做的事，某種意義上就是把這條捷徑推到極致、讓整個資料結構本身都長成日誌的樣子。

## LSM-tree：永遠只追加，整理賒給背景

LSM-tree（Log-Structured Merge tree，O'Neil 等人 1996 年提出）做了一個近乎激進的決定：**寫入永遠不就地更新，只往前追加。** 它把 B-tree 那條「順序日誌」的捷徑從「拿來保命的附屬品」升格成「整個引擎的主結構」。

一筆寫入的旅程是這樣的：

```
write path:
  1. append to WAL on disk (sequential, for crash recovery)
  2. insert into memtable  (in-memory sorted structure, e.g. skiplist)
  3. when memtable is full -> flush as one immutable SSTable on disk
                              (already sorted; a single sequential write)
  4. background compaction merges SSTables, drops shadowed old values
```

新值先寫進記憶體裡一個有序結構（memtable，通常是 skiplist 或紅黑樹），這是純記憶體操作、極快。memtable 滿了，整個被**一次性、順序地**刷成磁碟上一個不可變的有序檔——這個檔叫 **SSTable（Sorted String Table）**，它一旦寫下就永不修改。注意「一次性、順序地」這幾個字：刷一個 memtable 是一次大塊的順序寫，不是一頁一頁的隨機刷盤。LSM-tree 把 B-tree 那些散落各處的隨機小寫，攢在記憶體裡排好序，然後變成一次漂亮的順序大寫。這就是它寫吞吐高的根。

但這裡立刻冒出一個問題：如果寫入永不就地更新，那麼當我第二次、第三次更新 `id=42` 時，發生了什麼？答案是——**舊值沒有被刪掉，新值只是被追加在某個更新的 SSTable 裡。** 同一個鍵的不同版本，散落在好幾個不同時間刷下來的 SSTable 中。刪除更詭異：你不能「就地把它移除」，因為 SSTable 不可變，於是刪除被實作成寫一個特殊的標記——**墓碑（tombstone）**，一筆「這個鍵已死」的記錄，同樣是追加上去的。

於是讀，變成了麻煩事。要查 `id=42` 的當前值，引擎不能像 B-tree 那樣一條路徑到底，它得**從最新的 SSTable 往最舊的找，直到撞到第一個關於這個鍵的記錄**——那個記錄可能是最新的值，也可能是一個墓碑（代表它被刪了）。最壞情況下，它得逐個 SSTable 查過去。這就是 **LSM 的讀放大**：一次邏輯讀，可能變成對多個檔案的多次物理查找。這正是 B-tree 那個「一條路徑、命中即最新」的乾淨讀路徑，在 LSM 世界裡的反面。

LSM 用兩個工具來止血讀放大。一是 **bloom filter**：每個 SSTable 配一個機率性的位元集合，能極快地回答「這個鍵**一定不在**這個檔裡」（它可能偽陽、但永不偽陰）。查一個鍵時，bloom filter 讓引擎跳過絕大多數根本不含它的 SSTable，只去真正可能命中的那幾個裡找。二是**每個 SSTable 內部的稀疏索引**，讓在單一檔裡的定位也是對數級而非掃描。但要記住 bloom filter 有個關鍵的盲區——它只對「點查單一鍵」有效，**對範圍掃描完全幫不上忙**：範圍查詢得問遍每一層，因為你不知道區間裡的鍵分散在哪些檔。LSM 的範圍掃描天生比點查吃力，這是它的另一個非顯然的軟肋。

## compaction：背景裡那場永不結束的整理

如果只是不停追加，SSTable 會無限增多、舊版本與墓碑越積越厚、讀放大失控、磁碟塞爆。LSM 必須有一個機制把這些攤平——這就是 **compaction（壓實）**。它是 LSM 的心臟，也是它最容易被低估的地方。

compaction 是一個背景行程，持續地把多個 SSTable 讀進來、合併（像 merge sort 那樣合併有序檔）、丟掉被覆蓋的舊版本和已過期的墓碑、寫出更少、更大、更乾淨的新 SSTable，然後刪掉舊的。它做兩件事：壓低讀放大（檔變少、版本變單一），以及真正回收空間（舊版本和墓碑被清掉）。

而「怎麼合併」本身又是一個取捨旋鈕，分出兩大策略，這是 LSM 內部第二層的「寫放大換讀放大」：

**Leveled compaction（分層）**把資料組織成大小逐層遞增的層級（L0、L1、L2…，每層通常比上一層大 10 倍），而且**關鍵約束是：同一層內的 SSTable 之間鍵不重疊**。這意味著查一個鍵，每一層最多只需要查一個 SSTable——讀放大低、且可預測。空間放大也低（同一鍵的版本不會在同層重複）。代價在寫：要維持「層內不重疊」這個不變量，當資料從上層往下層 compaction 時，得把它和下層所有重疊的部分讀出來、合併、寫回去。一筆資料在它的一生裡，可能被從 L0 一路重寫到最底層，每下一層都被讀出再寫回一次。

這個寫放大可以算個量級出來。以 RocksDB 的預設層級放大係數 10、一個典型的多層配置為例，一筆資料寫進去，加總各層的重寫——照官方 Tuning Guide 的拆法，大致是 `1（memtable flush 到 L0）+ 2（L0→L1，因 L1 與 L0 同量級，得讀出再寫回）+ 10（L1→L2）+ 10（L2→L3）+ 10（L3→L4）= 33`——也就是說，**你寫 1 GB 的邏輯資料，磁碟上可能實際被寫了約 33 GB**（WAL 那條順序寫是另一筆持久性開銷，不算進這個 compaction 內部的寫放大）。每往下一層，因為要和那層大 10 倍的資料合併重疊區段，重寫的量就被那個係數放大一次。這是 leveled compaction 為「低讀放大」開出的價碼。

**Size-tiered compaction（分桶）**反過來：它把差不多大小的 SSTable 攢成一桶，攢夠數量就整桶合併成一個更大的檔，如此層層往上。它不維持「同層不重疊」的約束，所以寫的時候不必反覆和下層合併、寫放大低得多。代價是同一個鍵的不同版本可能散在多個桶裡，讀要查更多檔、讀放大高，而且合併前所有舊版本都還佔著磁碟、**空間放大也高**（最壞情況下，一場大合併進行時，磁碟上得同時容得下舊檔和正在寫的新檔，瞬間空間需求翻倍）。

所以同一個 LSM 引擎，光是換 compaction 策略，就是換掉一整組「讀放大／寫放大／空間放大」的配比。這是個第一級的調校旋鈕，不是某個可以忽略的常數。現代引擎甚至把它做成連續可調的——Cassandra 5.0 的 Unified Compaction Strategy（UCS，2026-06）就用一個密度參數，讓你在 leveled 和 tiered 之間平滑滑動，依工作負載從寫密集調到讀密集，多數情況不必再做破壞性的策略遷移。

這裡藏著一個 LSM 最關鍵、也最常被忽略的故障模式：**compaction 不是免費的背景工作。** 它和前台的讀寫請求爭搶同一份 CPU 和磁碟 I/O。寫入越猛，compaction 越追不上，未合併的 SSTable 越堆越多（L0 堆積），讀放大跟著惡化——這叫 **write stall**。更糟的是它的延遲特性：平時相安無事，但一場大的 compaction 啟動時，會週期性地搶走 I/O，造成讀延遲的尖峰。這就是為什麼 LSM 系統的 p99、p999 尾延遲往往比 B-tree 更難看、更毛躁——平均延遲很漂亮，但偶爾有一筆讀正好撞上 compaction 的 I/O 風暴。把 compaction 當成「設定好就不用管的背景雜務」，是 LSM 上線後最常踩的坑。

## 一個反直覺的結論：小筆高頻更新，LSM 反而寫得少

現在把兩邊的寫放大擺在一起算，會撞出一個讓很多人愣住的結論。

回到開頭那筆 100 byte 的更新。在 B-tree 上，它最小要刷一個 16 KB 的 page，寫放大約 **164 倍**——而且是隨機 I/O。在一個 leveled LSM 上，這 100 byte 先進 memtable（免費），最終被 compaction 一路重寫，寫放大約 **33 倍**——而且全是順序 I/O。

33 比 164，LSM 不只放大倍率更低，而且它放大出來的是磁碟更喜歡的順序寫。（這兩個倍率能這樣直接相減，前提是更新夠小、兩邊的分母都接近那 100 byte——一旦 value 變大、分母口徑岔開，這個對比就不再成立，下一節會看到天平如何反轉。）所以對「大量分散的小筆更新」這種工作負載——時序資料、計數器、事件流、IoT 感測讀數——**LSM 整體的物理寫入量反而比 B-tree 少得多。** 這就是為什麼這類寫密集系統幾乎清一色用 LSM 底層：RocksDB、Cassandra、TiKV、CockroachDB、ScyllaDB（2026-06：RocksDB 最新穩定版約 11.0.4／2026-04，Cassandra 5.0 帶上了 UCS 與 BTI 新 SSTable 格式）。

但這個結論有它成立的**條件**，越界就翻車——這正是「LSM 一定比 B-tree 寫得快」這個流行說法不可靠的地方。B-tree 的 164 倍，分子是固定的 page 大小、分母是你那筆更新的大小。如果你的更新不是 100 byte 而是 8 KB 的大 value 呢？那 B-tree 的頁層級寫放大就降到 2 倍左右，而 LSM 的 compaction 放大不會跟著縮——LSM 的 33 倍是**對整筆 value 反覆重寫**算出來的，value 越大，LSM 要在各層之間搬運的位元組就越多。**所以鍵小、value 大時，天平可能反過來倒向 B-tree。** 「LSM 比較快」只在「寫放大由 B-tree 的頁粒度主導」這個前提下成立，也就是「小筆、高頻、分散」的更新。一旦 value 變大、或讀路徑被 compaction 尾延遲拖累，這個前提就不成立了。

## 那些會在邊界上騙你的地方

把這套機制講到能在腦中重放，得認得幾個它會反咬你的非顯然邊界。

**「LSM 刪除立刻釋放空間」——錯得很危險。** LSM 的刪除只是寫一個墓碑，原始資料還躺在某個 SSTable 裡，要等到一場「同時涵蓋了那筆資料和它的墓碑」的 compaction 把兩者一起清掉，空間才真正回收。在那之前，刪除不但沒省空間，反而**多佔了一個墓碑的空間**。

而墓碑本身會反過來拖慢讀。這是時序資料庫和高刪除／高 TTL 過期工作負載的經典陷阱：想像你對一個分區反覆寫入又刪除，累積了成千上萬個墓碑。現在來一筆範圍查詢——引擎為了確認「這個區間裡到底還剩什麼活著」，**必須讀過所有這些墓碑、把它們和舊值逐一比對核銷**，哪怕最後回給你的是空集合。讀一堆「已經死掉的資料」來確認它們真的死了，這個成本是實打實的。Cassandra 把這個風險量化進了引擎：一次查詢若掃過超過 **100,000 個墓碑，會直接拋 `TombstoneOverwhelmingException` 中止**——它寧可讓查詢失敗，也不讓你在不知情下被墓碑活活拖垮。而墓碑之所以不能一寫下去就清掉，是因為分散式副本之間需要時間把這筆刪除傳播到所有節點（Cassandra 的 `gc_grace_seconds` 預設整整 10 天），太早清掉墓碑會讓某個還沒收到刪除的副本在下次同步時把「已刪資料」復活——這就是所謂的 zombie data。墓碑是 LSM 為「不可變 + 分散式」這兩個前提付的稅。

**「B-tree 沒有寫放大」——也是錯覺。** 頁粒度本身就是寫放大的源頭（那個 164 倍），page split 造成的額外寫入和碎片又在上面加碼。沒有哪個維護有序結構的引擎能逃過寫放大，差別只在它把放大藏在哪裡：B-tree 藏在 page 粒度，LSM 藏在 compaction 的重寫。

**「讀就是讀」——但兩邊讀到的代價結構不同。** 順帶說一個 B-tree 自己的非顯然讀放大：InnoDB 的主鍵索引是 clustered 的——葉節點直接存整列資料、按主鍵排序。但它的**次級索引（secondary index）葉節點存的不是整列，而是「索引鍵 + 主鍵值」**。所以用次級索引查一筆完整的列，是**兩次 B+tree 查找**：先在次級索引樹裡找到主鍵，再拿這個主鍵回到 clustered 索引樹裡撈出整列。一次邏輯查詢、兩趟樹遍歷。這提醒我們，「讀放大」不是 LSM 的專利——任何索引結構都有它自己的讀放大形狀，只是長得不一樣。

**「compaction／checkpoint 設好就不用管」——它們是活的旋鈕。** LSM 的 compaction 速度、B-tree 的 checkpoint 頻率，都直接影響尾延遲和崩潰恢復時間，調不好就是週期性的延遲尖峰或開機後漫長的日誌重放。

## 為什麼是這個形狀

退一步看，這整個分裂——B-tree 對 LSM、寫放大對讀放大——都是從本章開頭那個無法迴避的物理矛盾裡長出來的：**資料要求有序，磁碟偏好順序寫，而「永遠就地有序」和「永遠順序追加」這兩件事在物理上不能同時做到。**

B-tree 選擇在你寫的當下就把帳結清——立刻把新值就地放進它在排序中的正確位置，於是讀的時候一切井然、一條路徑命中即最新；代價是寫的當下要付隨機 I/O 和整頁刷盤。LSM 選擇把帳賒到背景——寫的當下只管順序追加、把整理外包給 compaction，於是寫吞吐極高、對 SSD 友善；代價是讀的時候要在散落的版本裡翻找，而且永遠有一個背景行程在和你的前台請求搶資源。

沒有哪一邊是「對的」。它們是同一個物理約束下，把成本推往不同時刻、不同維度的兩種選擇：**B-tree 把成本付在寫入的此刻，LSM 把它延後並攤平到背景。** 你選哪一個，本質上是在回答一個關於你工作負載的問題——我的瓶頸是寫吞吐撞牆，還是我需要可預測的低讀延遲？是大量分散的小筆更新，還是讀多寫少配上大量範圍掃描？答對了，引擎的脾氣就站在你這邊；答錯了，你會在某個流量高峰，眼睜睜看著寫入被隨機刷盤頂死，或看著 p99 被一場 compaction 風暴撕開。

所以下次看到一個資料庫在寫密集負載下從容自若、卻在範圍查詢上偶有毛躁的尾延遲，你會知道那不是它沒調好——那是 LSM 在兌現它出生時就簽下的合約：把寫放大壓到最低，代價是讀放大和那台永不停歇的背景整理機。而當你看到另一個資料庫讀延遲穩如磐石、卻在小筆高頻寫入下吃力——那是 B-tree 在誠實地、在你按下 commit 的那一刻，就把整理有序的帳，一頁一頁地結清了。
