# 對帳：怎麼確認兩邊一致、漂移怎麼修

某個週二早上，一個工程師被叫醒：客服說有使用者在搜尋頁找不到一筆三天前就上架的商品，但商品明明在資料庫裡好端端地躺著。他翻日誌——CDC 管線沒報錯，consumer 沒重啟，broker 沒丟訊息，所有儀表板都是綠的。他手動把那筆資料重推一次到索引，問題消失。然後他做了一件讓他更睡不著的事：寫了個小腳本，把資料庫的商品總數和搜尋索引的文件總數拉出來比。數字差了 **47**。

不是 47 萬，是 47。在幾百萬筆裡，47 筆鬼資料——資料庫有、索引沒有，或反過來。沒有人知道它們是什麼時候、為什麼漏的。沒有任何一個系統「壞掉」過。這 47 筆，是一條跑了八個月、看起來一直很健康的同步管線，悄悄漏下來的沉積物。

這一章講的就是怎麼面對這個事實——以及一句聽起來像悲觀主義、其實是工程現實的話：**任何長期運行的同步管線，最終都會漂移。** 對帳（reconciliation）這套機制，存在的理由就是把「我以為兩邊一致」變成「我**驗證過**兩邊一致」，並在它們不一致時，能找出差在哪、把它修回去。

## 為什麼「沒報錯」是一種幻覺

先把問題講難。直覺上，一條同步管線如果一路沒拋例外、沒掉訊息、consumer lag 是零，那兩邊就該一致。這個直覺錯在哪？

錯在它假設「所有的不一致都會以錯誤的形式現身」。但漂移的成因，絕大多數是**靜默**的：

- consumer 在處理一則訊息的中途被 OOM kill，offset 已經 commit、但那筆寫入還沒落到目的地——這在 at-least-once 的世界裡甚至不算 bug，它只是「那則訊息恰好掉進了 ack 與 apply 之間的窗」。
- 某個人為了救火，半夜直接 `UPDATE` 了一筆生產資料，繞過了整條應用層寫入路徑——同步管線根本不知道這筆變更發生過。
- 一次 schema 變更，目的地的某個欄位映射在切換的接縫上吞掉了幾秒鐘的事件。
- 一個邊界條件的 bug：某種特殊字元的商品名讓索引寫入靜默失敗、但回了 200。

這些全都不會點亮任何一盞紅燈。它們的共同點是：**錯誤的不是「過程」，而是「結果」。** 過程順順利利地跑完了，只是結果和來源對不上。一個只監控「過程有沒有出錯」的系統，對這種漂移是全盲的——它在問「我有沒有摔倒」，而你真正該問的是「我到了沒到對的地方」。

這就是對帳的第一原理：**一致性是一種要被「主動驗證」的性質，不是「沒出錯就自動擁有」的性質。** 一條沒有對帳的同步管線，不是「一致的管線」，而是「你不知道它一不一致的管線」——這兩件事的差別，就是那 47 筆鬼資料。

## 天真的對帳，以及它為什麼撐不住

知道要對帳之後，最直覺的做法是：把兩邊全部拉出來，逐筆比。來源一千萬列、目的地一千萬列，照主鍵排好，一行一行對每個欄位。

這做法的好處是它**找得到一切**——任何一個欄位的任何一點差異都逃不掉。壞處是它的成本等於資料量本身。一千萬列雙邊各拉一次，I/O、記憶體、網路全都吃滿；對一個還在線上服務的庫，這是一次沉重的重讀。更糟的是，它在這幾分鐘的拉取過程裡，資料**還在變**。你比到第 300 萬列時，第 50 萬列那筆可能已經被使用者改過了——於是你比的根本不是「同一個瞬間的兩份資料」，而是「來源在 T1 的快照」對「目的地在 T2 的快照」。它們本來就該不一樣。這叫**移動目標（moving target）**問題，等一下會看到它如何反咬整個對帳的正確性。

所以全量逐筆比對在小資料、或低頻的深度稽核（金額、合規）場景有它的位置，但你不可能拿它當「每小時跑一次」的健檢。要把對帳變成可以高頻、輕量、常態跑的東西，得換一個思路：**別把功夫花在「比所有資料」，把功夫花在「找出哪裡不一樣」。**

## 用聚合值當哨兵：便宜，但會騙你

第一步降本，是不比內容、先比**聚合值**。兩邊各算一個 `COUNT(*)`、或某個金額欄位的 `SUM`、或整批資料的一個 hash，比這幾個數字。一千萬列的比對，瞬間縮成比一個數字。

這是一道極其便宜的哨兵，適合高頻跑。但它有一個必須講透的陷阱，否則你會被它給的安心感害死：**聚合值相等，不代表內容相等。**

看一個具體的反例。來源有一筆 `id=8801` 的訂單、沒有 `id=9902`；目的地恰好相反——有 `9902`、沒有 `8801`。兩邊都是一千萬列整，`COUNT(*)` 一模一樣，哨兵亮綠燈。但這兩份資料**根本不同**：一邊缺一筆、另一邊多一筆，兩個錯誤剛好在「計數」這個維度上互相抵銷了。這叫**抵銷誤差（offsetting errors）**。如果你比的是 `SUM(amount)`，只要那兩筆金額碰巧相等，連金額總和都騙得過去。

所以計數和總和是「有沒有差」的快速哨兵，**永遠不能**升格成「一不一致」的判準。要抓抵銷誤差，你得比的不是「有幾筆、加起來多少」，而是「每一筆的內容指紋」。整批資料的 hash 就強得多——它對「內容」敏感，`8801` 換成 `9902` 一定會讓 hash 變號。但整批一個 hash 又退回了老問題：它告訴你「有差」，卻**完全不告訴你差在哪**。一千萬列裡有一筆不對，root hash 就翻了，然後呢？你還是得從頭找。

我們需要的，是一個既對內容敏感、又能把「差異定位」的成本壓到正比於「差異量」而非「資料量」的結構。這就是 Merkle tree 登場的地方。

## Merkle tree：讓比對成本正比於差異，而非資料量

把「整批一個 hash」這個想法做成一棵樹，魔法就出現了。

做法是這樣：把主鍵空間切成許多連續的段（bucket），每一段算一個 hash，這些段 hash 是樹的**葉子**。然後每兩個（或多個）相鄰葉子的 hash，再 hash 一次，成為它們的父節點；父節點再兩兩 hash 成祖父節點……一路往上收斂到唯一一個 **root hash**。這棵樹的關鍵性質是：**任何一個葉子裡的任何一筆資料變了，它的 hash 變，連帶它這一路往上到 root 的所有祖先 hash 全都變；但跟它無關的那些子樹，hash 一個都不動。**

於是比對兩邊變成一場「沿著 hash 不同的路往下鑽」的遊戲：

```
                 root
                /    \
          H(L,R)      H(L,R)        <- 先比 root: 不同 -> 往下
          /    \      /    \
       h0      h1   h2      h3      <- 比這層: 只有 h2 不同 -> 只往 h2 鑽
      ////    ////  ////    ////
     leaves  leaves ...    ...      <- 最終鎖定 h2 底下那幾個葉子
```

先比兩邊的 root hash。相同——恭喜，整棵樹、整批資料保證一致，**一個 hash 的傳輸就結束了**。不同——往下比它的兩個子節點 hash，哪個子樹的 hash 相同就整個剪掉不看，哪個不同就繼續往那邊鑽。一層一層下去，最後只會落在「真的含有差異」的那少數幾個葉子上。你只需要把那幾個葉子對應的實際資料逐筆比一遍，就能精確列出差異列。**整個過程交換的資訊量，正比於差異所在的少數段，而不是資料總量。**

這正是 Amazon Dynamo 論文裡描述的 anti-entropy 機制，也是 Apache Cassandra 的 repair 在做的事（2026-06）。Cassandra 的實作用一棵深度 15 的緊湊樹——2^15 = 32,768 個葉子——一棵樹蓋住一個 token range，葉子小到能快速在節點間傳輸比對。

## 一次手算：50 筆差異，要比多少資料

把這個「正比於差異」講成數字才有重量。

設想對帳一張主庫與唯讀副本各 **一千萬列**的表。把主鍵空間切成 **1,024 段**（一棵約 10 層的二元 Merkle tree，2^10 = 1,024 個葉子），平均每段約 **9,766 列**。每一側各算 1,024 個葉子 hash，加上 1,023 個上層內部節點 hash，每側約 2,048 個 hash 值（雙邊合計約 4,096）。

現在假設真實的漂移是：**3 個段裡，散落著總共 50 筆不一致的列。**

對帳怎麼跑？先比 root，兩個 32 位或 64 位的數字一比——不同。往下走，每一層只需要交換「hash 不同的那條路上的節點」。在一棵 10 層樹裡，要定位 3 個有差的葉子，最壞情況下需要交換的內部 hash 數量，是這 3 條從 root 到葉的路徑覆蓋到的節點——每下鑽一層、對 hash 不同的節點交換它的兩個子節點 hash，3 條最深 10 層的路徑最壞約 3 × 10 × 2 ≈ 60 個（路徑在頂層重疊會更少）——量級在**幾十個 hash** 上下，遠不到上百。鎖定那 3 個葉子後，把這 3 段對應的實際資料拉出來逐筆比：3 × 9,766 ≈ **29,300 列**。

對照全量逐筆：要拉 **2,000 萬列**（雙邊各一千萬）做比對。

兩者一比：Merkle 的工作量是「幾十個 hash 的網路往返 + 約 3 萬列的逐筆比對」，全量是「2,000 萬列的拉取與比對」。後者是前者的數百倍。而且這個比值會隨資料量惡化——資料漲到一億列，全量成本漲十倍，但只要差異還是那 50 筆，Merkle 的成本幾乎**不變**（只是樹多一兩層、葉子段大一點）。**這就是 Merkle tree 的核心承諾：對帳成本脫鉤於資料量、掛鉤於漂移量。** 一個健康的系統漂移量趨近於零，於是對帳幾乎免費——你可以高頻地跑，而它平常什麼都不傳，只在真有差時才花力氣。

代價在哪？在於**兩邊的「分段」必須完全對齊**：同樣的主鍵邊界、同樣的排序規則（collation！一邊用 `utf8_general_ci`、另一邊用 binary 排序，同一批列會落進不同的段）、同樣的 hash 函式。任何一處不齊，root hash 必然不同，而且往下鑽會發現**每一段都不一樣**——這時 Merkle tree 不但沒幫你省事，還退化成全量比對外加一堆假差異。Merkle 對帳第一次接上時最常見的故障，不是真有漂移，而是兩邊的分段或排序根本沒對齊。

## 移動目標：對帳最陰險的假差異來源

現在回到那個一直埋著的伏筆——**移動目標**。它是對帳裡最容易產生「幽靈差異」的根源，值得單獨講透，因為它的錯覺極強。

問題是這樣：你在 T1 時刻去算來源那 1,024 個葉子 hash，花了 8 秒；算到一半，使用者改了第 500 段裡的一筆資料。你在 T1+8s 算目的地的 hash 時，那筆變更可能已經透過同步管線到了目的地、也可能還在路上。於是第 500 段的 hash 兩邊不一致——**但這不是漂移，這是時間差**。那筆資料正在被正常地同步，只是你恰好在它「半路上」的時候給兩邊各拍了一張照。

如果你天真地把這種段也當成「差異」去修，會發生一件荒謬的事：你會去「修復」一筆其實沒壞、只是正在路上的資料，甚至可能用一個你以為的「正確值」覆蓋掉一個更新的、真正正確的值。對帳本該是安全網，結果它自己變成了製造漂移的源頭。

要堵住移動目標，思路是讓「兩邊比的，是同一個邏輯時間點的同一批資料」。有幾種做法，從重到輕：

一是**對齊一致性快照**。在來源開一個一致性快照（PostgreSQL 的 repeatable read / snapshot，或邏輯複製啟動時讀的那個一致快照），記下它對應的複製位點（LSN / binlog position），然後等目的地的同步追上**同一個位點**，再各自在那個凍結的視圖上算 hash。這樣兩邊算的字面上就是同一個 commit 邊界的資料，時間差被消滅。

二是更巧妙的、`pt-table-checksum`（Percona Toolkit）採用的招數——它示範了一個漂亮的「讓問題自己消失」的設計，值得拆開看。它**不在兩邊各自獨立算 hash**，而是只在主庫上跑分段的 checksum 查詢，把這個 checksum 查詢本身**當成一筆 statement 寫進 binlog**，讓它順著複製鏈流到每個副本上**重新執行**。因為這個 checksum 查詢在複製串流裡有固定的位置，它在主庫和副本上「看到的資料」，是複製順序裡**同一個點**的資料——主庫先算完、副本在自己重放到那一點時才算。兩邊的 checksum 被寫進一張 `percona.checksums` 表，事後再用一句 `SELECT` 比這張表就知道哪個 chunk 不一致。移動目標問題在這裡不是被「處理」掉的，是被設計成**根本不會發生**——比對的時間點由複製串流本身釘死，而不是由兩個各自看錶的對帳程式去湊。

（這也是為什麼 `pt-table-checksum` 要求 checksum 寫入以 STATEMENT 格式複製、而非 ROW：要的就是讓副本**重算**一遍、而不是直接收主庫算好的結果——重算才能驗證副本自己的資料對不對。）

`pt-table-checksum` 還順手解了另一個現實問題：它把每張表切成許多 chunk、每個 chunk 的目標執行時間約 0.5 秒，並用一個指數衰減的加權平均動態調整 chunk 大小——若主庫一忙、或某個副本複製落後太多，它就**自己暫停**等對方追上。對帳本身是會給線上系統加負載的，一個不懂節流、不看背壓的對帳 job，會在系統最忙的時候火上澆油。

## 找到差之後：修，是一個方向問題

對帳的前半是「發現差異」，後半是「修復」。而修復這一步，藏著一個比技術更需要判斷力的陷阱：**你要往哪個方向覆蓋？**

最常見、最簡單的修法是「以來源為準」（source-of-truth wins）：認定來源是唯一真相，把目的地的差異列無條件覆蓋成來源的值。對「DB → 搜尋索引」「主庫 → 唯讀副本」這種有明確上下游的場景，這通常是對的——索引和副本本來就該是來源的投影，它們的值沒有獨立存在的意義，蓋掉就蓋掉。

但「以來源為準」內建了一個危險假設：**差異的成因，一定是目的地落後了。** 萬一不是呢？萬一這次的差異，是來源自己被某個 bug 寫進了髒資料、而目的地反而是對的？或者目的地那筆值，其實是一段刻意的本地修正（某個下游服務對它的本地讀模型做了補丁）？這時你一覆蓋，就把「對的」蓋成「錯的」，而且因為對帳「成功收斂了」，你還會以為自己修好了——你親手製造了一筆漂移，並蓋上了「已對帳」的綠章。

所以成熟的對帳，在覆蓋之前會先問「差異的方向與成因是什麼」。這也是為什麼**金額類**的對帳幾乎從不自動覆蓋：兩邊帳對不上，不是拿一邊去蓋另一邊就算了，而是產出一張**差異報表（break list）**，交給人先查清楚每一筆 break 的成因——是漏記、重複、還是時間差——再決定怎麼修。金融場景裡「對不上」本身就是一級事件，盲目自動覆蓋等於銷毀證據。

如果來源本身有一份**可重放的變更 log**（CDC 的 offset、event sourcing 的事件序列），還有第三種更乾淨的修法：不直接覆蓋目的地的當前值，而是把目的地漏掉的那幾筆變更，從 log 的對應位置**重新投遞**一遍。這比「拿來源現值硬蓋」更忠於事實——它修的是「漏了哪幾筆事件」，而不是「現在的值差多少」，因此能正確處理「目的地漏了中間一個狀態」這種逐筆覆蓋會錯過的情況。

## 對帳器自己也會悄悄死掉

最後一個轉折，也是最容易被忽略的：**對帳 job 自己，也是一條會靜默失敗的管線。**

想想這個情境：你建好了 Merkle 對帳，每小時跑一次，跑了三個月一切太平，於是大家都忘了它的存在。某次部署改了資料庫連線設定，對帳 job 從此每次啟動就拋例外、直接退出——但它的失敗沒有人盯，cron 不在乎子程序的 exit code，於是它**安靜地掛了三週**。這三週裡，真正的漂移在背後慢慢累積，而你的團隊比沒有對帳時更危險——因為大家都「知道有對帳兜底」，於是更放心地不去管同步管線了。一個掛掉沒人發現的對帳器，提供的不是保障，是**虛假的安心感**，它比完全沒有對帳更糟。

所以對帳機制要完整，它的**運作本身**必須是可觀測的，而且觀測的維度不是「有沒有報錯」（我們開頭就講過那是幻覺），而是這幾個只有對帳器自己知道的數字，每一個都該是有告警的指標：

- 對帳**跑了沒**——上一次成功完成是什麼時候？超過預期間隔還沒跑完，本身就是警報。
- 這一輪**找到幾筆差**——差異數突然從個位數跳到上千，意味著某條管線出事了，這是比管線自身的錯誤日誌更早、更可靠的領先指標。
- 找到的差**修了幾筆、還剩幾筆**——持續修不掉的「頑固差異」，往往指向一個結構性的 bug（某類資料永遠同步失敗），而不是偶發漏訊。

把這三個數字接上告警，對帳才從「一個會悄悄死掉的腳本」變成「一個系統真正一致性的儀表」。它的價值不在某一次修好了 47 筆鬼資料，而在於：從此你對「兩邊一致」這件事的信心，有了一個持續、自我驗證的來源，而不是一句沒人敢打包票的「應該沒問題吧」。

## 為什麼是這個形狀

退一步看，對帳的整個樣貌，都是從一個無法迴避的事實裡長出來的：**任何長運的同步，最終都會漂移；而漂移是靜默的。**

正因為漂移會發生，你不能只在出錯時補救，得**主動、週期性地驗證**——所以要有對帳這道工序。正因為驗證全量太貴、且資料是移動目標，你得用 Merkle tree 把成本壓到正比於漂移量、用複製串流或一致性快照把比對釘在同一個時間點——所以對帳長出了「分層 hash + 對齊快照」這個形狀。正因為修復是個方向問題、蓋錯方向會把對的弄壞，所以它要先判方向、金額類要人工裁決。又正因為對帳器自己也會靜默死掉，所以它必須監控自己的存活與發現量——否則它只是又一條沒人驗證的管線。

對帳不是同步管線的「可選加值」，它是讓「最終一致」這四個字真的成立的那一半。沒有它，你擁有的不是一個最終會一致的系統，而是一個你永遠無法確認到底一不一致的系統——直到某個週二早上，客服打來電話。
