# 連線池調校

凌晨的告警是這樣開始的：流量突然往上爬，自動擴容很盡責地把應用實例從 5 個拉到 30 個——這正是擴容該做的事。然後資料庫開始大面積拒絕連線，錯誤日誌裡刷出整片 `FATAL: sorry, too many clients already`，連帶把那些**已經連上**的請求也拖慢，因為資料庫忙著處理一千條連線的排程。詭異的是，你盯著資料庫的 CPU，它**並不忙**——使用率不到一半。明明加了機器、明明 CPU 還有餘裕，系統卻在「流量最大、最需要它撐住」的那一刻整片塌掉。

這一幕的兇手不是流量，是**連線數**——一個多數人憑直覺設、而那個直覺幾乎總是錯的旋鈕。連線池本身的機制（為什麼要池化、借出與歸還的語意、怎麼驗活、leak 怎麼發生）在別處講透了（見〈連線池〉）；這一章假設池已經就位，只問一件事：**那幾個旋鈕——池大小、各種逾時、連線壽命——到底該轉到哪裡，又該盯著什麼指標才知道自己轉對了。** 這聽起來像調參瑣事，但它底下藏著一個違反直覺的事實，而沒看穿那個事實的人，遲早會親手把上面那場凌晨事故做出來。

## 池不是越大越快

幾乎每個第一次調連線池的人，都會犯同一個錯：覺得「連線多 = 能同時處理的請求多 = 快」，於是把池開大。應用想要 200 的並發？那就把池設成 200，讓每個請求都有連線可借，誰也不用等。

這個直覺錯在它把「應用端的並發需求」和「資料庫的有效並發能力」當成了同一件事。它們不是。

把鏡頭推到資料庫內部。一台資料庫真正能**同時推進**的查詢數，受限的是它的物理資源——CPU 核心數、磁碟的並行 I/O 能力。一顆 4 核的 CPU，無論你塞給它 10 條連線還是 1000 條，任一瞬間真正在跑的也就那 4 個（再加上等 I/O 時可以讓出來的那些）。當你給它 1000 條連線、每條都帶著一個活躍查詢，多出來的 996 個查詢並沒有變快——它們只是被推進資料庫**內部的排隊**：搶 CPU 時間片、搶鎖、塞滿排程器、製造大量 context switch。每一次 context switch 都是純粹的開銷，CPU 花在「切換誰該跑」上的時間，是從「真正跑查詢」裡偷來的。

結果是一條凹下去的曲線：吞吐隨連線數上升，到某個點達到頂峰，之後**不升反降**——連線再加，吞吐反而往下掉、延遲往上爬。HikariCP 的作者引用過一組 PostgreSQL 的壓測：在那台機器上，TPS 大約在 **50 條連線**附近就攤平、不再上升，再往上加只是徒增排程與切換的開銷、把曲線推向下坡。也就是說，那台機器最快的池大小，遠比多數工程師憑「我有幾百個並發」估出來的數字要小。

這就是為什麼有一條看起來小得不像話的經驗公式。HikariCP 官方那篇〈About Pool Sizing〉給的起點是：

```
connections = (core_count * 2) + effective_spindle_count
```

`core_count` 用的是**資料庫伺服器**的實體核心數（不是應用伺服器、不是 vCPU），`effective_spindle_count` 大致是「資料能不能並行從磁碟撈」的衡量——資料若**全在記憶體快取**裡，這項趨近 **0**；隨著快取命中率下降、查詢得真的去打磁碟，它就往實際的磁碟數靠近（傳統機械碟一軸算一個 spindle，SSD 的並行度高、行為不同，官方坦言這條公式對 SSD 的校準還沒有定論，實務上多半當它接近 1）。

代進去算一台 4 核、SSD、資料大致快取得住的資料庫：`(4 × 2) + 1 = 9`。湊個整數叫它 **10**。文中還補了一個讓人愣住的數字：就這 9–10 條連線，那台機器能輕鬆扛住約 3000 個前端使用者、跑到約 6000 TPS。一台 4 核機的最佳池大小是個位數——這跟「我有兩百個並發所以開兩百」的直覺差了一個數量級，而那個數量級的差距，正是開場那場凌晨事故的溫床。

這裡要釘死一件事，因為它常被誤讀：**這個公式不是讓連線「夠用」的下限，而是讓吞吐「最高」的最佳值。** 開到比它大，不是「多一點保險」，而是主動把自己推下那條凹曲線的右半邊。連線池大小是一個有最佳點的凹函數，不是一個越大越安全的滑桿。

## 為什麼那台 CPU 還閒著系統卻塌了

回到開場那個違和感：資料庫拒絕連線、請求變慢，可是 CPU 不忙。現在它說得通了。

當應用端開出去的連線數遠超過資料庫能有效推進的量，瓶頸根本不在 CPU——CPU 在等。等什麼？等鎖、等 I/O、等 context switch 的開銷散去。海量的連線把資料庫的注意力從「跑查詢」分散到「管理連線與排程」上，於是你看到一個經典的假象：**資源利用率不高，系統卻已經過載。** 光盯 CPU 的人會徹底誤判——他會以為「CPU 還有空，問題一定在別處」，然後往錯的方向找半天。

而連線數會塌穿臨界值，幾乎總是因為第二個被忽略的事實。

## 池大小要乘上實例數

連線池是**每個應用實例各開一個**的。你在設定檔裡寫的那個 `maximumPoolSize`，是**單一實例**的上限。資料庫真正看到的連線總數，是另一筆帳：

```
DB 看到的連線數 = 每實例池大小 × 實例數
```

這個乘法是所有連線池事故的數學核心，而它之所以陰險，是因為**你調的那個旋鈕（單實例池大小）和真正撞牆的那個量（總連線數）之間，隔著一個會自己變動的乘數**——實例數，由自動擴容決定，由流量決定，不由你決定。你在開發機上、在只有一兩個實例的測試環境裡，永遠摸不到那面牆。它只在生產環境、在流量大到觸發大規模擴容的那一刻才出現——也就是最糟的時刻。

把開場那場事故的數字補上。每個實例 HikariCP `maximumPoolSize = 20`（一個看起來很保守、很合理的值），自動擴容把實例從 5 個拉到 30 個。資料庫的 `max_connections = 100`（PostgreSQL 的預設值就是 100，其中還有 3 個保留給超級使用者做維運），但這不重要，重要的是乘法：

```
5 個實例  × 20 = 100   ← 剛好頂到 max_connections=100，已經危險
30 個實例 × 20 = 600   ← 是上限的 6 倍
```

擴容到 30 個實例，這 30 個池一起伸手，要的是 **600** 條連線，資料庫只給得出 100。多出來的全部被 `too many clients` 拒於門外。最毒的是時序：這發生在「流量大到需要擴容」的瞬間，正是系統最不能掛的時候，擴容這個本該救命的動作反而成了壓垮資料庫的最後一根稻草——加機器這個直覺上以為會更穩的動作，物理上卻是把資料庫淹死了。

要把這道題解對，得倒過來算。先定出**全體實例加起來**對資料庫的有效並發應該是多少——假設這台資料庫是 8 核 SSD，那麼按公式，它的甜蜜點大約是 `(8 × 2) + 1 = 17` 條真正在跑的連線。注意這個 17 是**整個叢集共享**的預算，不是每個實例的。如果高峰會有 30 個實例，把這 17 條平分下去，每實例連 1 條都分不到（17 ÷ 30 還不到 1）。而連線池小於 1 是沒有意義的——每個實例至少得有一條連線才跑得動，現實中為了服務自己的並發往往想要 2–4 條。可是 30 個實例 × 哪怕只有 2 條，就是 60 條，早已是甜蜜點 17 的三倍多、直直衝向 `max_connections`。換句話說，多實例直連根本湊不出一個既「每個實例夠用」又「總數安全」的池大小：低到安全就低到沒用，高到夠用就高到撞牆。一邊是「每個實例都想要足夠的連線好服務自己的並發」，一邊是「資料庫只認得總數、且總數有硬上限」——這兩個願望在多實例下直接打架，沒有中間地帶，而打架的結果，就是逼出了下一個機制。

## 在資料庫前面擺一個 pooler

當實例數會自己長大、每個實例的直覺需求又湊不出一個小到安全的池，直連這條路就走死了。出路是在應用和資料庫**中間**插一層：連線池中介（pooler），最常見的是 PgBouncer。

它的作用是把「多」收斂成「少」。應用端那 600 條連線全部打到 PgBouncer，PgBouncer 自己只對真正的資料庫維持一小撮連線（比如 20 條），把那 600 個客戶端**多工**到這 20 條上。資料庫永遠只看到 20 條，無論前面接了多少個應用實例、開了多大的池。乘法那道題就此被拆掉——資料庫看到的連線數不再隨實例數線性暴漲，而是被 pooler 封在一個固定的天花板下。

但這個收斂不是免費的，代價藏在 PgBouncer 的**池化模式**裡，而選錯模式會以最難 debug 的方式咬人。PgBouncer 有三種模式，差別在「一條資料庫連線什麼時候被還回池子、可以給下一個客戶端用」：

| 模式 | 連線何時釋放回池 | 收斂力 | 代價 |
|---|---|---|---|
| session | 客戶端斷線時 | 最弱 | 連線整段被一個客戶端霸佔，省不了多少 |
| transaction | 每個交易結束時 | 最強 | session 狀態不跨交易保留 |
| statement | 每個語句結束時 | 最激進 | 禁止跨多語句的交易 |

PgBouncer 的**預設是 session 模式**——而 session 模式幾乎收斂不了什麼。它要等客戶端**整個斷線**才把連線還回池，意思是一個長壽的應用連線會一直霸佔著一條資料庫連線，跟直連差不了多少。真正能把上千客戶端壓成數十條連線的，是 **transaction 模式**：連線只在「一個交易進行中」才綁定給某個客戶端，交易一 commit 就立刻還回池給別人用。因為大多數時間連線是閒著的（交易很短、思考時間很長），transaction 模式能用極少的真實連線服務極多的客戶端。`default_pool_size` 預設是 **20**，`max_client_conn` 預設是 **100**（這兩個數字會隨版本演進，引用時對一下當前官方文件）。

代價就埋在「交易一結束連線就被別人拿走」這句話裡。**session 狀態不跨交易保留。** 一條資料庫連線本來有很多「黏在這條連線上」的狀態——`SET` 設的 session 變數、`PREPARE` 出來的具名 prepared statement、session 級的 advisory lock、`LISTEN` 註冊的通知、`WITH HOLD` 游標、session 範圍的暫存表。在 session 模式下這些都安然無恙，因為連線整段屬於你。但在 transaction 模式下，你這個交易拿到的可能是 A 連線，下個交易拿到的是 B 連線——**上一個交易設的 session 狀態，在下一個交易裡憑空消失**，因為它根本是另一條物理連線。

這個坑特別陰，是因為它**沉默、且間歇**。一個依賴 prepared statement 或 advisory lock 的應用，在 session 模式或直連時跑得好好的，一換到 transaction 模式的 pooler 後面，會開始**隨機**出現「prepared statement 不存在」或鎖行為錯亂——不是每次都錯，是「剛好下個交易落在另一條連線上」時才錯，於是它在低負載測試時可能完全不發作，上了線、連線復用變頻繁，才開始零星冒出來。許多 ORM 預設大量使用 prepared statement，正是這個衝突最常見的引爆點：以 Prisma 為例，它用具名 prepared statement，放在 transaction 模式的 pooler 後面就需要特別處理——要嘛在連線字串標記讓它對 pooler 相容、要嘛靠 PgBouncer 對「協定層 prepared statement」的追蹤（1.21.0 起加入；1.24.0（2025-01）起 `max_prepared_statements` 預設已是 200、等於預設開啟，較舊版本才需手動設成非 0——引用時對著當前官方文件確認此預設，2026-06）。看不見這層的人，會花一整個下午追一個「只在生產偶發、本機怎麼都重現不出來」的鬼。

順帶一提，serverless 把整個問題推到極致，也讓 pooler 從「選項」變成「幾乎必需」。每個函式實例各自帶一個連線池，而 serverless 平台會在流量來時瞬間並發拉起成百上千個函式環境——這等於把「池大小 × 實例數」裡的實例數放大到極不可控，而且每個環境的生命週期很短、連線剛建好可能就被回收。一個直連資料庫的 serverless 函式，在流量尖峰時打爆 `max_connections` 幾乎是必然，這也是 Prisma 這類 ORM 在 serverless 場景一律建議前置 PgBouncer 或託管 pooler 的根本原因。

## 逾時與壽命：剩下那幾個旋鈕

池大小決定了「能借出多少」，但連線池還有另一組旋鈕，管的是「借不到時等多久」和「一條連線能活多久」。預設值往往為「最大相容性」而設，套到線上請求路徑常常太鬆。

最該動的是**取得連線的逾時**。當池滿了、所有連線都借出去了，下一個來借的請求會怎樣？它會**等**——等到有人歸還，或等到逾時。HikariCP 的 `connectionTimeout` 預設是 **30,000 毫秒，整整 30 秒**。設想一下這 30 秒在線上意味著什麼：池一旦滿了，新進來的每個請求都會卡在那裡等最多 30 秒才失敗。但這些請求往往背後各佔著一條應用執行緒（或一個事件迴圈的 in-flight 槽位），它們不是安靜地等——它們在**堆積**。前端的逾時可能只有 5 秒、10 秒，使用者早就收到錯誤、可能已經重試了，而你的應用還在傻等那條永遠借不到的連線、執行緒一條條被佔死，新請求連進都進不來。一個本該「快速失敗、把壓力擋在門口」的時刻，被預設的 30 秒拖成了「請求在內部無聲堆積到雪崩」。所以線上路徑常把它調到 **1–5 秒**：池滿就快速失敗，讓過載**立刻可見**、可以往上游回壓或降級，而不是悄悄積在底層（這正是把池滿當成一種背壓訊號來用——見〈背壓〉）。

第二組是**連線壽命**。連線不是建一次就能永遠用的——資料庫端、中間的防火牆或負載均衡器，都可能在連線**閒置一段時間後默默把它關掉**，而池這邊毫不知情，還以為這條連線活著躺在池裡。等到某個倒楣的請求把這條「殭屍連線」借出去、發出查詢，才在查詢中途撞上 `connection reset`——一個看似隨機、其實源於壽命沒管好的錯誤。

HikariCP 用兩個旋鈕防這件事。`maxLifetime`（預設 **1,800,000 毫秒，30 分鐘**）是一條連線的最長壽命，到期就主動回收重建，讓連線在「被對端默默掐掉」之前先體面退休。鐵律是：**`maxLifetime` 必須比下游（資料庫或防火牆）的閒置超時更短**，留出餘量，這樣連線總是由池主動汰換、而不是被對端從背後關掉。另一個 `idleTimeout`（預設 **600,000 毫秒，10 分鐘**）管的是閒置連線何時縮回 `minimumIdle`，省離峰資源。這兩個旋鈕都有最小值約束（`maxLifetime` 不得低於 30 秒，`idleTimeout` 不得低於 10 秒），調太短會讓連線頻繁重建、把建連線那筆昂貴的握手成本（TCP + TLS + 認證）又重新背上身——壽命管理本身也是個取捨，不是越短越安全。

還有一個常被忽略、但對開場那種冷啟動瞬間很關鍵的選擇：**啟動就把池建滿**。HikariCP 預設 `minimumIdle` 等於 `maximumPoolSize`，也就是預設行為就是維持一個固定大小的滿池。把 min 設得跟 max 一樣（或夠大），意思是行程一啟動就付清所有建連線的成本，而不是把這筆成本轉嫁給「第一個倒楣撞上空池的真實請求」——後者會在那個請求的延遲裡突然多出幾十毫秒的現場建連線時間（這跟冷啟動的暖機是同一個道理，見〈冷啟動與暖機〉）。

值得留意的是，這些「該手動收緊預設值」的旋鈕，前提是你直連、自己管池。換到別的生態，預設哲學會變：Prisma ORM 早期版本的預設池大小是一條跟 CPU 綁定的公式（`物理核心數 × 2 + 1`，與 HikariCP 同源的直覺），但在 **Prisma 7（2025-11）** 改成了一個跨資料庫統一的固定預設 **10**——調參的入口、甚至預設值本身都隨工具版本在變，所以任何「預設值是多少」的記憶都該對著當前官方文件再確認一次（2026-06）。

## 沒有指標就是閉著眼睛調

到這裡，所有旋鈕都講過了——但有一個前提一直沒說破：**你怎麼知道自己轉對了？** 你不知道，除非你在看正確的指標。而前面那個「CPU 還閒著系統卻塌了」的假象，已經劇透了答案：**池的問題，盯 CPU 是看不出來的。**

池耗盡的時候，CPU 很可能無比清閒——因為大家都不在跑查詢，大家都在**排隊等連線**。一個只盯 CPU、記憶體、QPS 這些常規儀表的人，會在連線池已經滿載、請求已經在借用佇列裡堆成山的時候，看著一片綠燈的 dashboard 一臉茫然。連線池有它自己的一組生命徵象，必須單獨盯：

- **active / idle / waiting**：借出去的、閒置的、**正在等連線**的數量。`waiting` 持續大於 0，是池太小（或有 leak）最直接的證據——有請求拿不到連線在排隊。
- **borrow 等待時間**：從「伸手借」到「真的拿到」之間的耗時。這段時間是**直接加進請求延遲**的純等待，它悄悄上升，往往比任何其他指標更早預告「池快不夠了」。
- **取得逾時次數 / 池耗盡事件**：直接撞到 `connectionTimeout` 的次數。一旦非 0，就是池已經滿到開始丟請求。
- **DB 端連線數 / `max_connections` 的比值**：盯著前面那道乘法有沒有逼近硬上限。這個比值爬向 1，就是擴容快要把資料庫打死的早期警報。

這幾個指標和 CPU 的關係是**互補、甚至背離**的——池出問題時 CPU 反而閒，正因如此你不能用 CPU 代替它們。沒有 `waiting` 和取得逾時這兩個數字，你對連線池的所有調整都是盲猜：你不知道池太小還是太大，不知道調逾時有沒有救到請求，不知道下一次擴容會不會撞穿上限。**有了它們，連線池調校才從「憑感覺轉旋鈕」變成「看著儀表收斂到那個凹函數的最高點」。**

## 為何是這個形狀

退一步看，連線池調校的整套邏輯，其實都從一個被反覆忽略的事實裡長出來：**連線不是免費的並發，而是一種稀缺、共享、有硬上限的資源——而它的上限不在你調的那一端，在資料庫那一端。**

正因為資料庫能有效並行的查詢數受限於物理核心，所以池有個遠比直覺小的最佳值、開大反而更慢——那條凹曲線。正因為每個實例各開一個池、實例數又會自己擴張，所以真正撞牆的是「池大小 × 實例數」這個你不直接控制的乘積——那道乘法。正因為這道乘法在多實例下湊不出一個既夠用又安全的數，所以要在資料庫前擺 pooler 把「多」收斂成「少」——而收斂的代價是 session 狀態的取捨。正因為連線會被對端默默掐死、池滿時請求會無聲堆積，所以逾時要收緊、壽命要管住。而正因為這一切的症狀都不在 CPU 上顯形，所以你非得盯池自己的那組指標不可。

連線池調校看起來是一堆瑣碎的數字旋鈕，但每一個旋鈕的正確方向，都是同一句話的不同說法：**別把應用想要的並發，誤當成資料庫扛得住的並發。** 開場那場凌晨事故——加了機器、CPU 還閒、資料庫卻被連線淹死——不是哪個旋鈕設錯了一格，而是沒看穿這句話的人，遲早會用「越大越好」的直覺，親手把它做出來。
