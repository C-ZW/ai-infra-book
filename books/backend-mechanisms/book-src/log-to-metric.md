# 從 log 算 metric 的陷阱：取樣、遺漏與視窗

凌晨三點，某個服務的延遲告警響了。值班的人打開儀表板想確認規模，順手在日誌系統裡敲了一行查詢：過去五分鐘有多少筆 5xx？日誌系統忠實地回了一個數字——`30`。看起來不像大事，於是這條告警被降級成「明早再看」。但真相是那五分鐘裡發生了三千次失敗；日誌系統回的 `30`，是被一道一個月前為了省成本而設下的 1% 取樣率，悄悄縮小了一百倍的影子。沒有人記得那道取樣率存在，更沒有人記得它會在「最需要數據」的那一刻反過來騙人。

這一幕的根源，是一個看起來再合理不過的工程捷徑：**既然每個請求都已經寫了一行 log，何必再額外埋一個 counter？** 直接在日誌系統裡 `count by (status)`、對 `latency_ms` 欄位算個 p99，metric 不就「免費」長出來了嗎。少維護一套 pipeline、少在程式碼裡埋一堆點，這個誘惑非常真實，而且在小流量下它確實划算。問題在於，這個捷徑把一件事偷偷綁死了：**你算出來的 metric，正確性從此取決於 log 的完整性**——而 log 偏偏是三種遙測訊號裡，最容易被取樣、被丟、被截斷、被算錯視窗的那一種。這一章要把這條捷徑底下的三個結構性陷阱拆開：取樣（sampling）讓樣本不再代表全體、遺漏（loss）讓你在最關鍵的時刻數據最殘缺、視窗（windowing）讓時間本身變得不可信。看懂它們，你就會明白為什麼一句話可以總結：**要拿去告警、定 SLO、做容量決策的數字，不要從 log 算。**

## 先看清楚：metric 和 log 到底差在哪

要理解陷阱，得先把兩種訊號的本質擺正。一個 metric——具體說，一個 counter——是「每發生一次事件就原子地 +1」的累計值。它的關鍵性質是**無損且與流量解耦**：不論這條 time series 背後是一秒十次還是一秒十萬次請求，它在記憶體裡永遠只是一串 `(timestamp, value)`，而且**每一次事件都被算進去了**，一個都不會掉。counter 不認識「取樣」這個詞——它要嘛 +1，要嘛沒被呼叫到，沒有中間地帶。

一條 log 則相反，它是一筆離散的、帶時間戳、欄位完整的事件紀錄。它資訊量最高、最不失真——你可以事後從它身上撈出任何當初沒想到要看的維度。但代價是它**量最大、最貴**，於是在高流量下幾乎一定要對它動刀：取樣、設 buffer、丟棄、截斷。從 log 算 metric，本質上就是**拿一個「為了便宜而被刻意弄殘的訊號」，去重建一個「本來要求精確」的訊號**。這個方向是逆水行舟的。

把這個成本差距放進具體數字就更清楚了。一個每秒一千請求的服務，若每筆寫一條約 1 KB 的 access log，一天是 `1000 × 86400 × 1 KB ≈ 86 GB` 純 access log。同樣這些流量，做成 metric 只要一條 `http_requests_total` counter 加幾條 histogram，一天的時序資料是 **MB 等級**。差了四到五個數量級。正是這個差距逼出了一條鐵律：**metric 永遠開全量，log 與 trace 幾乎一定要取樣**。而一旦 log 被取樣，從它算出來的 metric 就繼承了取樣的所有毛病。下面三節，就是這些毛病各自的長相。

## 陷阱一：取樣——你算的是樣本，不是全體

取樣的數學陷阱，要分成兩個層次看，因為它們的可修復程度天差地遠。

**第一層是絕對量的低估，這層可以修。** 假設日誌系統設了 1% 的 head sampling（每一百條進來的 log 隨機留一條）。實際發生了三百萬筆請求，你的 log 裡只留下約三萬筆。如果你直接 `count` 這三萬筆就當成「總請求數」，你低估了一百倍。修法很直接：取樣時把取樣率記在每條 log 上，聚合時用 `1/rate` 加權還原——OpenTelemetry 在 trace 取樣規格裡把這個權重叫做 **adjusted count**（原為 span-to-metrics 設計，同理可套到 log 紀錄上），一條取樣率 1% 的紀錄，它的 adjusted count 就是 `1/0.01 = 100`，代表「我這一條，背後站著一百條」。把三萬筆各乘以一百，你又得回三百萬。只要你**記得**乘、而且取樣是均勻的，總量級就能救回來。

危險在於，這個「記得乘」恰恰是最容易漏的一步——它不在查詢語言的預設行為裡。`count()` 這個函式天真地數行數，它不知道每行背後站著多少被丟掉的兄弟。開場那一幕的 `30`，就是少了這一乘。而且就算你乘對了總量，下一層的陷阱也不是乘法能解的。

**第二層是分位數與長尾的失真，這層修不回來。** 這是取樣最陰險的地方，值得用真實數字手算到底。

設想一個每秒一萬請求的服務，log 設 1% head sampling。我們想算「過去五分鐘的 p99 延遲」。五分鐘是三百秒，總請求數約 `10000 × 300 = 3,000,000` 筆。p99 的定義，是最慢的那 1%——也就是約 `3,000,000 × 0.01 = 30,000` 筆**最值得你關心的慢請求**。現在問一個關鍵問題：這三萬筆慢請求，在 1% 取樣後，期望會有幾筆留在 log 裡？

```
慢請求總數（最慢的 1%）   = 3,000,000 × 0.01 = 30,000
取樣後期望留下的慢請求數  = 30,000 × 0.01     = 300
```

三百筆。聽起來好像還夠？但這是針對「整個慢尾巴」的數字。真正定義 p99 的，是那批請求**最快的那一端的邊界值**；而你真正在乎的、會讓使用者尖叫的 p99.9——千分之一的尾巴——情況更糟：

```
p99.9 對應的請求數        = 3,000,000 × 0.001 = 3,000
取樣後期望留下            = 3,000 × 0.01      = 30
```

只剩三十筆樣本去估計一個千分之一分位數。三十個樣本的分位數估計，變異大到幾乎是擲骰子——這一刻的 log 給你 p99.9 是 1.1 秒，下一個五分鐘可能跳到 2.3 秒，不是因為系統變了，而是因為你的尺本身在抖。**這就是「總量可加權修、長尾分位數修不回來」的精確形狀**：加權還原是個確定性的乘法，它能把「三萬筆代表三百萬筆」這件事補正回來；但分位數的精度取決於**尾端真實落進樣本的點數**，而尾端事件被低取樣率命中的機率本來就低，乘以權重並不會無中生有地補出那些根本沒被記錄下來的點。

這裡藏著一個更深的不對稱：head sampling 在請求**剛開始、還不知道它會不會慢**的時候就擲骰決定去留，所以它對快請求和慢請求一視同仁地丟。但慢請求和錯誤請求本來就是少數，一視同仁地丟，等於系統性地把你最想看的那批事件丟得最徹底。這正是 tail sampling（等請求結束、看到它確實慢或錯了再決定留）存在的理由——但 tail sampling 換來的精準是有代價的：它必須緩衝整條請求的所有紀錄直到結束、同一條 trace 的 span 還得路由到同一個收集器實例，記憶體與架構成本都高。而且就算用了 tail sampling 把錯誤全留下來，你也**換不回**準確的總流量——當你的取樣策略是「只留 error=true 的紀錄」，你就再也算不出整體的每秒請求數了。取樣這把刀，無論從哪一頭下，都會切掉某種你需要的東西。

## 陷阱二：遺漏——數據在你最需要時最殘缺

取樣是你**主動**設下的縮減，至少你知道它存在、知道倍率。遺漏更陰險：它是 pipeline 在壓力下**被動、靜默**地丟資料，沒有人按下按鈕，也沒有一條 log 記著「這裡丟了一萬條」。

要理解它怎麼發生，得看一條 log 從產生到落地要走的路。應用程式寫出一行 log，它先進到本地採集 agent 的記憶體 buffer，agent 再批次往下游（日誌後端、物件儲存）送。這條路上每一段都有有限的容量，而當下游跟不上——網路慢了、後端在做 compaction、儲存在限流——buffer 就會漲。漲到頂之後會發生什麼，取決於這個 agent 怎麼設計。

以一個被廣泛使用的採集器為例，看它的官方行為（2026-06）。當記憶體 buffer 達到設定的上限（`mem_buf_limit`）時，它的第一反應**不是丟**，而是**暫停**對應的輸入——「stop buffering data from that source plugin until enough buffered chunks are flushed」，並回呼一個 `pause` callback 讓輸入端停止追加。聽起來很安全，資料沒丟，只是停了。但「暫停輸入」這四個字底下藏著真正的損失：當 agent 暫停去讀一個正在被寫入、而且還在輪替（rotate）的 log 檔時，應用程式並不會跟著停下來等它——應用照常以每秒上萬行的速度寫、檔案照常輪替、舊檔照常被刪。等 agent 從暫停中恢復、回頭再讀，**那些在暫停期間被寫進去又隨輪替消失的行，已經不在了**。沒有任何一層會為它們留一條墓碑。

另一條丟法在輸出端。當把 log 緩衝到磁碟、而某個輸出目的地的佇列漲到 `storage.total_limit_size` 上限時，官方行為是「the oldest chunk from its queue for that logical output destination will be discarded to make room for new data」——**丟最舊的那塊，給新資料騰位置**。這是個合理的設計選擇（與其讓整條 pipeline 卡死，不如犧牲最舊的、保住即時性），但對「從 log 算 metric」的人來說它是個無聲的洞：你聚合的那個時間視窗裡，可能整整一塊 chunk 的事件被默默丟了。

現在把這兩種遺漏，和「它們**什麼時候**最容易發生」疊在一起，陷阱的全貌才浮現：**log pipeline 最容易爆掉的時刻，正是系統出事、log 量暴漲的時刻**。一個正常時跑得好好的 buffer，會在一次故障引發 error log 海嘯時瞬間被灌滿——而那正是你最需要準確數據去判斷「炸到什麼程度」的時刻。於是你從 log 算出來的「事故期間請求數」往往莫名其妙地**偏低**，因為造成事故的那波流量，有一部分根本沒能寫進你正在查詢的儲存裡。數據在你最不需要它的平靜時刻完整無缺，在你最需要它的災難時刻殘缺——這是從 log 算 metric 最致命的失敗模式，因為它讓你在事故中**低估事故本身**。

這裡有個收尾的對照：一個 counter 不會有這種問題。它在程式行程內、在事件發生的當下就 `+1` 了，這次自增發生在你寫 log 之前，不經過任何會被背壓暫停的 buffer。就算後來上報 metric 的網路也塞住了，下一次成功上報時送出的是**累計值**——counter 的單調遞增天生抗丟：中間漏報幾次，恢復後一個累計值就把缺口補平了（拉取式系統會把這個遞增折算回速率）。同樣一場 log 海嘯，counter 照樣每一筆都算進，因為它根本不在那條會被淹沒的資料路徑上。

## 陷阱三：視窗——時間本身就不可信

就算取樣和遺漏都被你完美處理了——全量保留、零丟失——從 log 算 metric 還剩最後一個陷阱，而它和資料量完全無關，純粹關於**時間**。

「過去一分鐘的 QPS」這句話裡，藏著一個沒被言明的問題：你說的「一分鐘」，是按哪個時間戳切的？一條 log 身上其實有兩個時間：**事件發生的時間（event time）**——請求真正進來的那一刻；以及**它被寫入儲存的時間（ingestion time）**——它穿過 buffer、網路、批次上傳，終於落進日誌後端的那一刻。這兩個時間之間，隔著整條 pipeline 的延遲，從毫秒到（在背壓時）數十秒不等。

如果你的聚合按 ingestion time 切視窗——這常常是預設，因為後端只能按「我何時收到」來分桶——那麼一筆在 12:00:59 發生、但因為 agent 剛從暫停恢復、拖到 12:01:30 才被寫入的請求，會被算進 12:01 那一分鐘，而不是它真正屬於的 12:00。後果是兩個相鄰視窗一個莫名其妙地凹下去、一個莫名其妙地凸起來。當 pipeline 延遲穩定時這還只是個固定偏移，但 pipeline 延遲在**負載變化時最不穩定**，於是你的「每分鐘 QPS」會在流量高峰時無端抖動——而你分不清那抖動是真實的流量波動，還是 pipeline 延遲在搗亂。

要算對，就得按 event time 切。但這立刻引出一個無解的兩難：**遲到的事件（late arrival）什麼時候算數？** 一筆 12:00 發生、12:01:30 才到的 log，當你在 12:01:05 算「12:00 那一分鐘的 QPS」時，它還沒到，於是被漏算；可你又不能無限期地等——總得在某個時刻說「12:00 這一分鐘我封板了，後面再來的不算」。串流處理領域給這條封板線起了個名字叫 **watermark（水位線）**：一條在 event time 上、跟在「目前見過的最大 event time」後面一段距離的移動門檻，它代表一個聲明——「我相信所有時間戳早於水位線的事件都已經到齊了」。水位線一旦越過某個視窗邊界，那個視窗就結算、不再接受更晚到的資料。

於是這成了一個**準確性換即時性**的取捨，而且沒有免費的解：水位線拖得越靠後（等越久），你抓進的遲到事件越多、QPS 越準，但你得到結果的時間也越晚；水位線收得越緊（等越短），你越快拿到數字，但越多遲到的事件被判出局、被永久丟棄。一個 counter 又一次繞過了整個問題——它在事件發生當下、在那台機器上就 `+1` 了，根本沒有「事件時間 vs 寫入時間」這條縫，也就不需要水位線去縫合它。從 log 算 metric，等於是把一個本可以在源頭就解決的計時問題，硬生生推遲到下游、再用一套複雜的水位線機制去近似地補救。

順帶一提一個常被低估的時間陷阱：時區。如果產生 log 的機器、寫入的 pipeline、查詢的儀表板對「現在幾點」的理解不一致（有的用 UTC、有的用本地時間、有的還趕上日光節約時間切換），那麼你的視窗邊界會整段平移甚至重疊，而這種錯不會報錯，只會讓某個小時的數字看起來「就是怪怪的」。時間從來不是一個老實的維度。

## 還有一種更隱形的漏：解析悄悄失敗

前面三個陷阱都假設 log 至少被正確地解讀了。但從 log 算 metric 還依賴一個脆弱的前提：**聚合查詢能正確地從每條 log 裡解析出它要的欄位**。這個前提會以一種特別安靜的方式破掉。

設想某次發版，工程師順手改了 log 的格式——把 `status=500` 改成了 `status_code: 500`，或者把延遲欄位從毫秒換成秒。負責聚合的那條正則表達式或 schema 沒跟著改，於是它在新格式的 log 上**匹配不到**，那一批 log 被靜默地跳過、不計入。儀表板上你會看到的，是「流量在發版那一刻突然下降了一截」——而你的第一反應幾乎一定是「是不是真的掉流量了？」，於是你開始查負載均衡、查上游、查網路，查了半天才（如果運氣好）發現流量根本沒掉，只是**算不出來了**。一個聚合腳本的靜默解析失敗，偽裝成一場根本不存在的流量下跌，這比明確報錯危險得多，因為它把你的注意力引向完全錯誤的方向。

這也是「結構化 log＋嚴格 schema」相對於「自由文字 log＋事後正則」的真正價值所在：當每條 log 都被強制帶上型別固定的 `latency_ms`、`status` 欄位，格式漂移會在寫入端就被擋下或被明確標記，而不是在下游某條正則裡無聲地漏掉。但這只是把問題前移，並沒有消滅「log 的契約一旦變動、所有下游聚合都得跟上」這個根本的脆弱性。

## 那 log 聚合到底什麼時候該用

把上面三個半陷阱攤開，很容易讀成「永遠別從 log 算 metric」——但那是錯的結論，會讓你錯過這條路真正的價值。真正的判準，是分清**哪些數字錯了會出人命，哪些數字粗糙一點無妨**。

要拿去**告警、定 SLO、做容量規劃**的數字，一律走埋點 metric。理由前面都鋪好了：這些數字錯了會直接誤導決策——一個被取樣縮水一百倍的錯誤率會讓你睡過一場真實事故，一個被遺漏壓低的事故期請求數會讓你低估災難規模，一個被視窗抖動污染的 QPS 會讓你的容量模型建在流沙上。SLO 尤其不能建在 log 上：SLO 是要拿去和人談判、拿去觸發「凍結發版」這種有牙齒的政策的承諾，把它建在一個會被取樣、會被靜默丟棄、會因格式漂移而漏算的來源上，等於把承諾寫在水面上。

但反過來，log 聚合有一塊 metric 永遠搆不到的領地：**你事前沒想到要看的維度**。metric 的代價是你必須在寫程式碼時就決定好要埋哪些維度——而真實的事故總是來自你沒預料到的角度。「為什麼只有用某個特定 API 版本、且來自某個地區、且帳號建立於上個月的使用者會踩到這個 bug？」——你不可能為這種臨時拼出來的三維交集預先埋一個 counter（就算想，把這些高基數維度塞進 metric 標籤本身就是另一場災難）。這時候，那些欄位完整、可以即席任意切分的 log 就是唯一能回答問題的東西。log 聚合在這裡不是「廉價的 metric 替代品」，而是**探索性分析的正主**——它的價值在於靈活，不在於精確。

還有一塊中間地帶：流量小到 log 全量保留毫無壓力時，從 log 算 metric 是完全合理的省事，沒必要為它再起一套埋點。三個陷阱裡，取樣（你沒設取樣）和遺漏（量小到 buffer 永遠不會滿）在這個規模下基本不存在，只剩視窗問題還在，而視窗問題在低流量、低 pipeline 延遲下也溫和得多。**這些陷阱全都是「規模」逼出來的**——它們在你流量長大、開始為 log 設取樣率、buffer 開始在高峰期吃緊的那一刻才一起醒來。一個成熟團隊常見的做法是兩條路並行：關鍵指標用埋點 metric 當作真相來源，同時保留 log 聚合當作便宜的旁路校驗，定期比對兩者的差距——當哪天它們開始顯著漂移，那本身就是一個信號：要嘛取樣率變了，要嘛 pipeline 在悄悄丟東西。

## 為何是這個形狀

退一步看，這三個陷阱不是三件無關的倒楣事，它們全都源自同一個根本事實：**從 log 算 metric，是在拿一個「為了便宜而被設計成可丟、可取樣、可延遲」的訊號，去支撐一個「要求精確、要求完整、要求即時」的用途。** 取樣是為了省儲存而主動弄殘代表性；遺漏是為了在過載時自保而被動犧牲完整性；視窗的扭曲是為了應付一條有延遲的 pipeline 而不得不在「等多久」和「準多少」之間妥協。每一個陷阱，都是 log 這個訊號為了它本來的目的（廉價地保存高保真的個別事件）所付出的、轉嫁到「聚合」這個它本來不擅長的用途上的代價。

而一個 counter 之所以全程繞開這三個陷阱，也是同一個道理反過來看：它從一出生就是為「精確計數」設計的——在事件發生的當下、在源頭的行程裡就 `+1`，不取樣（它根本沒有「樣本」的概念）、不經過會被背壓淹沒的緩衝路徑、不需要事後對齊時間。它放棄的恰恰是 log 的長處：你沒法事後從一個 counter 裡撈出「那一秒到底是哪些使用者觸發的」，因為那些細節在 `+1` 的瞬間就被聚合掉了。

所以這從來不是「metric 比 log 好」或反之，而是**兩種訊號各自為不同的真相而生**：metric 為「現在系統健康嗎、趨勢往哪走」這種要準的問題而生，log 為「這一筆到底發生了什麼」這種要細的問題而生。從 log 算 metric 的陷阱，本質上是混淆了這條分工線——想用一個為「細」而生的訊號去回答一個要「準」的問題。下次你想在日誌系統裡敲一行 `count` 當成 metric 用之前，先問自己一句：這個數字錯一百倍，我會不會睡過一場事故？如果會，它就不該從 log 來。
