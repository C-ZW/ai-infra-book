# Lambda：冷啟動、SnapStart、payload 與併發

凌晨三點，一個平常每秒幾十筆的訂單流，因為一檔限時促銷，在三十秒內被推上每秒幾千筆。你的服務跑在 Lambda 上，理論上「自動擴縮、彈性無限」——你不必半夜被叫起來加機器。但監控上開始冒出兩種你沒預期的東西：一批請求的延遲突然飆到一秒以上，另一批請求直接收到 `429 TooManyRequestsException` 被拒。函式碼一行沒改、下游資料庫看起來也還活著，可是「無限彈性」就是沒有把這波流量接住。

這一章要拆的，就是這個「明明該無限彈性、卻在尖峰被擋住」的落差是怎麼來的。Lambda 把「何時開機器、開幾台、跑在哪」這些調度決策從你手裡接走，換來免運維與縮到零的省錢；但它沒有把分散式系統的硬約束一起接走，只是換了一種長相還給你。要把 Lambda 用對、用到能在腦中預測它何時會卡住，你得看懂三件互相糾纏的事：一個請求進來時底層發生了什麼（執行環境與冷啟動）、它怎麼決定要不要、能不能多開一台（併發與擴縮），以及它在邊界上會怎麼悄悄騙你（payload 上限、狀態殘留、限流的兩種源頭）。

## 一個請求進來，底下到底發生了什麼

先把最小的事實釘住（以下談的都是 Lambda 預設計算型）：Lambda 執行你的函式的單位，是一個叫 **execution environment** 的東西——一個受隔離的輕量虛擬機（底層是 Firecracker microVM）。它不是「一個容器服務無數請求」，而是一個 environment 在**任一瞬間只服務一個** invocation。這句話是理解 Lambda 一切行為的地基：要同時處理 N 個請求，平台就得有 N 個 environment 同時活著。在預設計算型下沒有「一個行程裡開很多執行緒併發服務」這回事——併發等於環境數，不是執行緒數。（2025-12 起新增的 Lambda Managed Instances 是例外：它跑在你帳號的 EC2 上、一個 environment 可同時處理多個 invocation，本章後面所有推導都針對預設計算型、不涵蓋它。）

當一個請求到來、而手上沒有空閒的 environment 可用，平台得從頭造一個。這條「從無到有」的路徑，就是**冷啟動（cold start）**，它分成幾個清楚的階段：

```
cold start path (no warm environment available):

  find / create microVM   ->   download code   ->   start runtime
        ->   run Init code (top-level, outside handler)   ->   enter handler
  [-------------------- platform's job --------------------][- your code -]
```

最後那兩步——啟動 runtime、跑你 handler 外的頂層初始化程式（讀設定、建 SDK client、連資料庫）——是你能影響的部分，也是冷啟動延遲的大頭。等這個 environment 被造好、初始化跑完，它服務完這個請求後不會立刻消失：平台會把它保留一段時間，下一個同函式的請求進來就**重用**它，跳過上面整條路徑，直接進 handler。這就是**暖啟動（warm start）**，快上一兩個數量級。閒置夠久沒人用，平台才回收它。

冷啟動的通用成因——為什麼第一個請求總是慢、JIT 與連線建立的代價、暖機策略——是一套跨平台的通用機制，本章不重講（見〈冷啟動與暖機〉）。Lambda 在這之上有一個它**獨有**的減速手段，值得單獨講透，因為它的省時來自一個會反咬你的設計。

## SnapStart：把初始化的結果凍起來，連同它的副作用

冷啟動慢，大頭在「跑 Init」。那有沒有辦法只跑一次 Init，之後所有冷啟動都跳過它？**SnapStart** 就是這個想法的具體實作，但它選的實作方式決定了它最深的那個陷阱。

機制是這樣的：當你發布一個函式版本時，平台不是把程式碼放著等請求，而是**當場先把 Init 跑完**，然後對這個「初始化已完成」的 microVM 的整個記憶體拍一張加密快照（Firecracker microVM snapshot）。之後每一次冷啟動，不再從頭跑 runtime + Init，而是直接**從這張快照還原**記憶體，落地就已經是「Init 跑完的狀態」，省掉整段最貴的初始化。官方對它的招牌數字是「啟動最高約 10x 快、可壓到 sub-second」（實測去除的 init latency 視 runtime 與初始化重量而定，常落在七八成以上），且 Java 託管 runtime 用它不額外收費（這點和另一條救冷啟動的路 provisioned concurrency 不同——後者預留環境是要持續計費的）。

聽起來像免費的午餐，直到你看清一個字：**還原**。快照是在**發布那一刻拍的，一張**；之後成千上萬次冷啟動，全部從**同一張**快照還原。這意味著任何「在 Init 期間產生、且本該每次不一樣」的東西，被凍進快照後，會在每個還原出來的環境裡**一模一樣**。

把這個抽象講到能在腦中重放，看三個真實的後果：

- **隨機種子被凍住。** 如果你在 Init 時播下一個亂數種子、之後用它產生「唯一」的 token 或 ID，那麼每個從快照還原的環境都帶著**同一顆種子**——它們會吐出同一串「隨機」值。這不是理論，是 SnapStart 官方文件專門列出的 uniqueness 問題。
- **連線被凍住。** Init 時開好的資料庫連線，被快照記下來；還原後那個連線描述符指向的是**發布當下**那條 TCP 連線，早就不存在或已被對端關閉。第一次用它就報錯。
- **時間戳被凍住。** Init 時 `now()` 算出來的「啟動時間」，在三天後還原的環境裡仍是三天前那一刻。任何拿它當「快取新鮮度」基準的邏輯都會錯亂。

平台給的解法，是讓你在快照前後插手。runtime hook `beforeCheckpoint`（拍快照前跑）讓你把不該進快照的唯一性內容**先清掉**；`afterRestore`（還原後跑）讓你把它們**重建**——重新播種、重開連線、重取時間戳。這套 hook 來自開源的 CRaC（Coordinated Restore at Checkpoint）專案。判準很乾淨：凡是「每個環境該不一樣、或會隨時間失效」的東西，都得移到 handler 內生成、或在 `afterRestore` 裡重建，絕不能讓它在 Init 裡定型後被凍進快照。

哪些 runtime 能用 SnapStart？這是一個會悄悄過時的事實，得釘準（2026-06，官方）：**Java（11 及以上）、Python（3.12 及以上）、.NET（8 及以上）** 支援；其中 Python 與 .NET 於 2024-11 GA。**Node.js 不支援、Ruby 不支援**，OS-only runtime、container image 也都不支援。Node 之所以不在優先支援名單，部分原因是它的冷啟動本就相對低，救它的邊際效益小——但這也意味著，押注 SnapStart 來救一個 Node 函式的冷啟動，會完全落空。另外 SnapStart 和 provisioned concurrency **不能同時用在同一個函式版本**（兩者都在解冷啟動、機制互斥），也不支援 EFS 與大於 512 MB 的 ephemeral storage。

| 救冷啟動的手段 | 機制 | 成本 | 適用 |
|---|---|---|---|
| Provisioned concurrency | 預先初始化 N 個環境、常駐待命 | 預留即計費 | 延遲敏感、尖峰可預測 |
| SnapStart | 發布時拍快照、冷啟動從快照還原 | 僅 Java 託管免費；Python/.NET 有快照快取＋還原費用 | Java/Python/.NET、能處理 uniqueness |
| 純優化 Init | 縮小 bundle、延後非必要初始化 | 無 | 任何 runtime、第一道防線 |

## 併發不是 RPS：尖峰被擋住的第一個源頭

現在回到開場那波尖峰。要看懂它為什麼被擋，得先把一個最常被混用的等式講死：**併發（concurrency）不是每秒請求數（RPS）。** 併發是「此刻同時在飛的請求數」，而把它和流量、延遲串起來的，是一條 Lambda 官方寫進文件的簡單公式：

```
concurrency = (average requests per second) × (average duration in seconds)
```

這條公式是 Little's law 在 serverless 上的長相，也是 Lambda 容量直覺的全部起點。手算一遍就懂它為什麼是陷阱的源頭。

假設一個函式平均每個請求跑 200 ms（即 0.2 秒），尖峰時觀測到 5,000 req/s。需要多少併發？

```
concurrency = 5,000 req/s × 0.2 s = 1,000
```

剛好頂到帳號預設的併發上限（**1,000，per region，可申請調高**）。換個演算法更直觀：一個環境 200 ms 跑完一個請求，等於一秒能處理 5 個；要消化 5,000 req/s，就需要 1,000 個環境同時開著。兩種演算法殊途同歸。

關鍵的非顯然之處在這裡：**併發需求 = 流量 × 延遲，所以下游一變慢，所需併發是成倍漲的。** 同樣 5,000 req/s，若下游資料庫變慢、平均延遲從 200 ms 漲到 800 ms：

```
concurrency = 5,000 req/s × 0.8 s = 4,000
```

需求瞬間翻成四倍、遠遠衝破 1,000 上限，後續請求收到 `429` 被限流。注意這裡的因果——**它不是被流量打爆的，是被延遲打爆的。** 流量一個沒多，只因為每個請求佔住環境的時間變長，併發需求就翻倍。這是 Lambda 容量規劃最反直覺、也最容易在凌晨三點咬你的一點：你盯著 RPS 覺得還很安全，卻沒注意到下游 p99 悄悄爬高，等限流冒出來時，根因在兩跳之外。

## 還有第二道天花板：每秒請求數本身有上限

就算你把併發上限調得很高，還有一道很多人沒讀到的牆。Lambda 在帳號層另外強制一條 **RPS 上限 = 帳號併發上限 × 10**。預設併發 1,000，於是你的函式們合計最多吃 **10,000 req/s**——和有沒有用滿併發無關。

這對「跑得很快」的短函式是個隱形殺手。看一個官方文件就拿來示範的場景：一個函式平均只跑 50 ms，尖峰 20,000 req/s。算併發：

```
concurrency = 20,000 req/s × 0.05 s = 1,000
```

併發只要 1,000，看起來預設上限就夠了。但 RPS 上限是 10,000——你的 20,000 req/s 有一半根本進不來，照樣限流。**這個限流不是因為環境不夠，是因為請求速率本身撞牆。** 要解它，得把帳號併發上限調到 2,000（連帶把 RPS 上限抬到 20,000），而不是只盯著併發數字。所以對 sub-100ms 的快函式，你必須同時檢查兩條線：併發夠不夠、RPS 上限夠不夠——它們是兩道獨立的閘門，任一道先撞到都會限流。

（順帶澄清一個由此衍生的誤讀：這條 RPS 上限**不**等於「每個環境只能處理 10 req/s」。平台是在帳號層總量上算併發與 RPS 兩個配額，不是在單一環境上設速率——別把這條總量規則錯算成單環境吞吐。）

## 就算配額夠，也不是想開多少就能瞬間開多少

第三道閘門更隱蔽：**擴縮是有速率的。** 平台為了防止對突發流量過度反應，限制了「多快能造出新環境」。這個 **concurrency scaling rate** 是**每個函式、每個 region：每 10 秒最多新增 1,000 個執行環境（或每 10 秒多接 10,000 req/s，先到先算）**。

這條速率上限和前面兩條配額是不同維度的東西：配額是「總共能開到多少」，速率是「往那個總量爬得多快」。把開場那波「30 秒內從幾十 req/s 衝到幾千」放進這個框架就清楚了——就算你的帳號併發上限有 5,000，一個函式也沒辦法在第一個 10 秒裡就生出 5,000 個環境；它每 10 秒只能多長 1,000 個。在爬升追不上流量的那段窗口裡，多出來的請求要嘛排隊冷啟動（延遲飆高），要嘛被限流（`429`）——這正好對上開場監控裡同時冒出的那兩種症狀。它們不是兩個 bug，是同一個「擴縮速率追不上瞬時尖峰」的兩種表現。

好消息是這條速率是**函式級**的：每個函式各自獨立地以這個速率擴縮，一個函式爬升不會吃掉另一個的額度。要對抗陡尖峰，正解是事先把該預熱的環境用 provisioned concurrency 鋪好，讓尖峰來時不必現場從零爬。

## 三種限流，長得一樣、根因不同

把上面三道閘門收在一起看，你會得到一個對 Lambda 維運極實用的心智模型：你看到的 `429`，根因可能是三個完全不同的地方，而它們的表面症狀一模一樣。

| 撞到的牆 | 觸發條件 | 解法 |
|---|---|---|
| 帳號併發上限 | 同時在飛的請求數 ≥ 1,000（預設） | 調高併發配額；或縮短下游延遲降低併發需求 |
| RPS 上限（併發×10） | 每秒請求數 ≥ 併發上限 × 10 | 調高併發配額（連帶抬高 RPS）；對快函式尤其要查 |
| 擴縮速率 | 10 秒內需要新增 > 1,000 個環境 | provisioned concurrency 預熱、削平爬升斜率 |

診斷的關鍵問句是：限流發生時，併發數到頂了嗎？沒到頂卻在限流——那多半是 RPS 上限（函式跑很快）或擴縮速率（尖峰來得太陡）在作怪，光調併發配額是調不到點上的。能在腦中分清這三者，你才有辦法對著監控說出「這波是被擴縮速率卡住、不是配額不夠」這種話。

## payload 的兩個上限，和一個悄悄變過的數字

換一個維度的邊界：請求帶進來、結果送出去的資料量。Lambda 對 payload 有兩個**不同**的上限，混用會出事：

- **同步呼叫（synchronous）**：request 與 response 各自上限 **6 MB**。API Gateway 或 ALB 把 HTTP 請求轉成同步 invoke 時走的就是這條。
- **非同步呼叫（asynchronous）**：上限 **1 MB**——但這個數字 **2025-10-24 才從 256 KB 提升上來**（官方公告）。

最後這條是本主題裡最容易悄悄過時、也最容易釀錯的一個。很長一段時間裡，async payload 上限就是 256 KB，無數系統的截斷與校驗邏輯都是照 256 KB 寫死的。世界在 2025-10-24 變了，但那些舊邏輯不會自己更新——一段照 256 KB 設計、預期會把超大訊息擋下來的校驗，現在會在 256 KB 到 1 MB 這個區間**放行**原本以為擋得住的訊息。這不是「誰記錯了」，是一個典型的「當年寫對、世界後來變了」型的老化邊界，值得在依賴這個數字的地方主動標上日期。

這個 1 MB 還藏著一個容易被忽略的計費細節：async invoke 的前 256 KB 算 1 個 request，超過 256 KB 的部分**每 64 KB 多算一個 request**，一路到 1 MB。所以把 payload 從 256 KB 撐到 1 MB，不只是「能塞更多」，計費上等於一次 invoke 最多被算成多個 request——payload 變大不是免費的。

還有一個跨服務的陷阱：**這條上限只是 Lambda 自己的。** 你的事件常常是別的服務推進來的，而每一個上游各有各的上限——SNS、SQS、EventBridge 的單則訊息上限是它們自己的數字。整條鏈路真正的 payload 天花板，是**鏈路上最小的那個上限**，不是 Lambda 的 1 MB。當 payload 接近任何一段的上限時，正解是把大資料體放進物件儲存、事件裡只傳一個引用（見〈物件儲存〉），讓事件本身保持輕。

## 「無狀態」是個半真半假的承諾

最後一個會咬人的邊界，藏在一句大家都會背的話裡：「Lambda 是無狀態的。」這話**對一半**——它在語意上要求你的函式無狀態，但它的執行模型在實作上**會留下狀態**，而這個落差是一條真實的資料外洩路徑。

關鍵還是那個「environment 會被重用」。warm start 跳過 Init，重用的是**同一個 environment 的同一份記憶體**——這代表 handler 外的全域變數、`/tmp` 裡的檔案，會**跨 invocation 殘留**。如果你天真地把上一個請求的使用者資料快取在一個全域變數裡，下一個請求（很可能是**另一個使用者**）打到同一個被重用的環境，就讀得到上一個人的資料。這不是假想，是真實系統裡出過的隱私事故。

所以無狀態的檢驗標準不是「我有沒有寫 `global`」，而是「把這個環境殺掉換一個，行為會不會變」。連線池、模組級快取、`/tmp` 的累積檔案都是狀態。規則因此很簡單：全域只放**和請求無關、無狀態**的東西（SDK client、設定、唯讀查表）；任何屬於「這一個請求 / 這一個使用者」的資料，都不准離開 handler 的作用域。把這層想清楚，serverless 為什麼鐵了心要無狀態，也就順理成章——平台保留了「隨時把你的環境回收、搬遷、增減」的權利，正是靠這個權利換來縮到零與自動擴縮；你交出「狀態留在記憶體」的便利，換來「環境可被任意替換」的彈性，這是一筆寫在合約裡的明確交易。狀態管理的完整原理（為什麼 process 記憶體一死即蒸發、該外置到哪）本章不展開（見〈無狀態與有狀態〉），這裡只點 Lambda 把這條鐵律以「環境殘留」的形式具體還給了你。

順著這個邏輯還能預判一個經典的連鎖故障：每個新環境各自開自己的資料庫連線，前面算出的 400 個併發就是瞬間 400 條連線。冷啟動風暴一來，環境數陡增、連線數跟著陡增，可以一口氣打爆資料庫的連線上限——這也是為什麼 serverless 連資料庫幾乎總得走外部連線池、而不是讓每個環境直連（見〈連線池〉）。

## 為什麼是這個形狀

退一步看，Lambda 所有讓你意外的行為，都從一個核心交易長出來：**你把「何時開機器、開幾台、開在哪」的調度權，整個交給了平台。**

正因為你交出了調度權，平台才能替你縮到零、替你自動擴縮——這是你買到的東西。但同一個交易的另一面是：平台得用**它的**規則來保護自己不被你的尖峰拖垮，於是有了併發配額（防你吃光整個 region）、RPS 上限（防短函式用請求數淹沒系統）、擴縮速率（防它對突發過度反應）這三道你繞不開的閘門。冷啟動，是「縮到零」這個省錢承諾的必然代價——沒有常駐的環境，第一個請求就得現造。SnapStart 想抹掉這個代價，於是用「凍結一張初始化快照」的方式去抄捷徑，而凍結這個動作本身，又必然帶來「唯一性內容被複製」的新債。連「無狀態」都不是平台白送的保證，而是它「隨時換掉你的環境」這個權利的副作用——你得自己守住作用域，才不會被環境的重用反咬。

所以下次你看到一波尖峰，一邊在飆冷啟動延遲、一邊在吐 `429`，你會知道那不是「彈性壞了」。那是這筆交易在如實兌現它的條款：你要的是免運維與縮到零，平台給你的，就是一個有配額、有速率、會冷啟動、會留下殘影的彈性——它的每一個邊界，都是「把調度權交出去」這件事，在不同維度上的回聲。
