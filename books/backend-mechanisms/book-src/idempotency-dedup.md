# 重複是常態：冪等與去重

一筆扣款訊息送進了佇列，消費者把它取出來、呼叫支付閘道扣了 100 元、然後正要回 ack——就在這一瞬間，這台 worker 被 OOM killer 收掉了。佇列那頭沒等到 ack，逾時之後它做了一件完全合理、也完全符合契約的事：把同一筆訊息重新投遞給另一台 worker。第二台 worker 不知道前面發生過什麼，它老老實實地又呼叫了一次支付閘道。使用者的帳上，現在有兩筆 100 元。

沒有人寫錯程式。佇列遵守了它承諾的「at-least-once（至少一次）」語意，worker 也只是照規矩處理訊息。問題出在一個更深的地方：只要你選擇了「絕不丟失訊息」，你就同時簽下了「可能重複處理」這份合約的另一半。重複不是 bug，不是哪裡配置錯了，而是這個世界運作方式的直接後果——**在會丟封包、會逾時、會重啟的系統裡，重複是常態，不是異常。** 這一章要講的，就是當重複無可避免時，你怎麼讓「處理了很多次」在效果上等同於「處理了一次」。

## 為什麼你躲不掉重複

先把重複的來源攤開，因為很多人以為重複只有一種，於是只防住了一種。重複至少從三個獨立的地方鑽進來。

第一種是**broker 重投**，也就是開場那一幕。消費者其實已經把事情做完了，但它回給 broker 的那個 ack 在回程網路上丟了；broker 等不到 ack，逾時之後判定「這筆沒處理成功」，重新投遞。注意這裡的關鍵不對稱：broker 完全無法區分「消費者沒做完」和「消費者做完了但 ack 丟了」——這兩種情況從 broker 那端看出去一模一樣，都是「我沒收到 ack」。它只能選擇相信最壞的情況並重投，否則就違反了「至少一次」。

第二種是**生產者重試**。生產者把訊息送給 broker、等 broker 回一個「我收下了」的確認，這個確認也走在同一張不可靠的網路上、也會丟。確認一丟，生產者面對的是和 broker 對稱的兩難：broker 到底收到沒有？它若不重送、訊息可能就此遺失；它若重送、broker 可能真的收了兩筆。選了「絕不遺失」，就意味著 broker 的佇列裡躺著兩筆內容相同的訊息——這時連 broker 自己都不知道它們是重複的。

第三種最隱蔽，是**消費者中途崩潰再被接手**。一個 worker 處理到一半（已經扣了款，還沒寫「已處理」標記）就崩了，訊息回到佇列、被另一個 worker 接走，從頭再做一遍前半段。這種重複不來自網路，而來自「處理一筆訊息」這件事本身不是原子的——它由好幾個步驟組成，崩潰可能卡在任何兩步之間。

這三種來源有一個共同的根，值得記死：**只要你想要「不丟」，你就無法同時要「不重」。** 這是訊息傳遞的一個基本取捨（交付語意的全景見〈三種交付語意〉），ack 會丟這件事讓它沒有第三條路。既然多數有狀態的業務系統都選了「不丟」，重複就成了你必須在消費端親手處理的義務，而不是某個能在 broker 設定裡關掉的選項。

## 重複本身不可怕，可怕的是它「留下痕跡」

退一步問：重複處理一筆訊息，到底有什麼壞處？答案完全取決於這個操作做了什麼。

考慮兩個操作。一個是 `SET status = 'shipped'`——把某張訂單的狀態設成「已出貨」。把這個操作做一次、做五次、做一百次，訂單的狀態都是「已出貨」，系統的最終狀態一模一樣。另一個是 `balance = balance - 100`——從餘額裡扣 100。做一次扣 100，做兩次扣 200，做五次扣 500。第一個操作「重放不留痕跡」，第二個「每一次都疊加」。

這個區別有個精確的數學名字：**冪等（idempotency）**。一個操作 `f` 是冪等的，意思是 `f(f(x)) = f(x)`——把它套用在某個狀態上一次，和連續套用很多次，得到的狀態相同。`SET status = 'shipped'` 冪等，因為它設定的是一個**目標狀態**，重放只是把狀態再次設到同一個值。`balance = balance - 100` 不冪等，因為它做的是**相對當前狀態的增量**，每多做一次，當前狀態就被推得更遠。

把這個性質想透，會發現它和「報不報錯」無關。一個冪等的操作重放時可能回給你一個不同的 HTTP 狀態碼——第一次 `DELETE /orders/7` 回 `200`、第二次回 `404`（因為已經刪掉了）——但兩次之後伺服器的狀態是一致的：那張訂單就是不存在。冪等講的是**狀態的淨效果**，不是回應的位元組。這也是為什麼 HTTP 的方法語意（RFC 9110）把冪等定義成「多次相同請求對伺服器的預期效果，和單次請求相同」，並明文規定 `GET`、`HEAD`、`OPTIONS`、`TRACE`、`PUT`、`DELETE` 是冪等的，而 `POST` 和 `PATCH` 不是。`PUT /users/42 {name:'Alice'}` 設定整個資源到一個確定的值，重放不變；`POST /charges` 每次都新建一筆收費，重放就是多扣一次款。

於是壓制重複的第一條、也是最乾淨的一條路浮現了：**如果你能把操作寫成「設定目標狀態」而不是「在當前狀態上累加」，重複就自動失去了傷害力**，你連去重都不必做。能做到這一步的場景比直覺以為的多——「把餘額算成事件序列的總和再 `SET`」就比「在餘額上 `-= 100`」更耐重放，因為前者重算多少次都收斂到同一個數。但有兩類操作天生改不成冪等：純粹的累加（計數器、扣庫存），以及**對外的副作用**（呼叫支付閘道、寄一封信、發一張券）。你沒辦法把「呼叫第三方 API 扣款」重寫成「設定目標狀態」，因為那一次呼叫本身就是不可收回的事實。對這兩類操作，需要更主動的機制。

## 不冪等的操作，怎麼裝成冪等的

當操作本身改不動，通用的解法是把「去重的責任」從操作內部搬到操作外圍，用一把**冪等鍵（idempotency key）**當作識別。這是這套機制的核心，值得一步步看清楚它怎麼運作、又為什麼必須是這個形狀。

直覺很簡單：呼叫端為「同一個邏輯操作」生成一個唯一的字串，跟著請求一起送過去；被呼叫端用這把鍵記住「這個操作我做過了，結果是這個」，下次再看到同一把鍵，就直接把上次的結果回放出來，不再真正執行。這把「天生不冪等的端點」包裝成了「對外冪等的端點」——這正是 Stripe 等支付 API 採用的做法，也是 IETF 正在標準化的 `Idempotency-Key` HTTP 標頭（最新為 draft-07，2025-10 發布、HTTPAPI 工作組仍在推進中；截至 2026-06 該版本已過期、尚未成為 RFC）所規範的東西。

但魔鬼藏在四個生命週期階段裡，每一階段都有一個非做對不可的細節：

```
# illustration, not runnable
1. CREATE   client generates a key (e.g. a V4 UUID), binds it to ONE
            logical operation, puts it in the Idempotency-Key header.

2. CLAIM    server sees the key for the first time: atomically INSERT a
            "in-progress" record (a UNIQUE constraint blocks concurrent
            re-entry), then start processing.

3. STORE    on completion: persist (status_code, response_body) under the
            key. Store BOTH success AND failure outcomes.

4. EXPIRE   after a TTL, drop the key. Within TTL -> replay stored result;
            outside TTL -> treat as a brand-new request.
```

第一個細節在 **CLAIM 必須原子**。設想兩個帶著同一把鍵的請求幾乎同時抵達伺服器——使用者連點了兩下、或客戶端的重試和原請求賽跑。如果伺服器的邏輯是「先查這把鍵存不存在，不存在就處理」，那麼兩個請求可能**都**查到「不存在」，於是**都**開始處理，扣了兩次款。冪等鍵自己有了 race condition。正解是讓「查」與「寫」合成單一原子動作：用資料庫的唯一約束，第一個 `INSERT` 成功的請求贏得處理權，第二個撞約束失敗，被擋在門外。是資料庫在替你做這個仲裁，而不是你的應用程式碼去上鎖。

第二個細節在 **STORE 必須連失敗一起存**。一個很自然但會出事的想法是「只有成功才值得記住」。問題在於：假設第一次處理走到一半、業務邏輯回了 `500`（也許下游暫時抽風），如果你因為「它失敗了」而不存任何記錄，那麼帶同鍵的重試會把整段業務邏輯**重跑一遍**——而這次可能成功了。聽起來像好事，但它打破了冪等鍵的核心承諾：「同一把鍵，永遠得到同一個結果」。更糟的是，你無法確定第一次那個 `500` 是「真的什麼都沒做」還是「做了一半、副作用已經發生、只是回應失敗」。Stripe 的設計是**成功與失敗都存**，包括 `500`：同一把鍵重送，一律回放第一次的結果，連業務邏輯都不重跑。這條規則看似違反直覺（誰想把錯誤也快取起來？），但它是唯一能讓「重試」這件事有確定語意的做法。

第三個細節在 **server 必須校驗參數一致**。同一把冪等鍵，但這次的請求 body 變了——比如金額從 100 改成了 1000——這是個危險訊號，幾乎一定是 client 的 bug（把鍵重用到了不同的操作上）。正確的伺服器會比對這次的參數和第一次的是否相同，不同就直接報錯，而不是糊里糊塗地把舊結果回放、或執行新的金額。Stripe 正是這麼做的：冪等層比對進來的參數和原始請求，不一致就拒絕。

第四個細節在 **TTL 必須涵蓋最壞的重投窗口**。冪等鍵不能永久存下去，否則去重表會無限膨脹，所以要設過期。Stripe 的記錄至少保留 24 小時。但 TTL 設多長不是隨便挑的——它必須長過你系統裡「重複最晚會多晚到」的那個窗口。如果一筆毒訊息進了死信佇列（見〈死信與毒訊息：DLQ〉），三天後才被人工重放，而你的去重 TTL 只有一小時，那麼這筆「其實見過」的訊息會被當成全新的處理。TTL 太短，等於在去重視窗的盡頭開了一道後門，重複會準時從那裡溜進來。

## 一場誤會：誰該記住這把鍵

把這套機制講到這裡，最容易被搞混的不是任何一個技術細節，而是**責任的分工**。冪等鍵是一支需要兩個人配合才能成立的舞——呼叫端和被呼叫端各管一半，任何一邊鬆手，整個機制就破。

呼叫端（client）的責任只有一件，但這件事最反直覺：**同一個邏輯操作的所有重試，必須用同一把鍵。** 最容易掉進去的陷阱是把鍵綁在「每次發送請求」這個動作上——在每次送出前才生成。如果重試時生成一把新鍵，被呼叫端會把它當成一個全新的操作，於是你精心設計的去重完全失效，照樣扣兩次款。鍵必須在「決定要做這件事」的那一刻生成一次，然後在這個操作的整個生命週期裡（包括所有重試）牢牢沿用。反過來，不同的邏輯操作必須用不同的鍵——同一張訂單的「建立」和「取消」是兩件事，共用一把鍵會讓第二個操作被誤判為第一個的重放而被吞掉。

被呼叫端（server）的責任，就是前一節那四個生命週期階段：用這把鍵去重、原子地 CLAIM、儲存成功與失敗的結果、設好 TTL、校驗參數。

把兩邊擺在一起看，就懂了為什麼這個機制必須是這個形狀：**只有呼叫端知道「這兩次請求是不是同一個邏輯操作」。** broker 不知道（它看到的是兩個 byte 序列）、server 也不知道（它看到的是兩個獨立的 HTTP 請求）。唯一握有「這是同一件事的重試」這個資訊的，是那個決定要做這件事的呼叫端。所以冪等鍵必須**由呼叫端生成並攜帶**，去重的判斷才有依據。這也解釋了一個常見的困惑：「為什麼不讓伺服器自己對請求內容取雜湊去重就好？」因為內容相同不代表是重試（兩筆獨立的、金額剛好都是 100 的扣款，內容可能一樣但是兩件不同的事），內容不同也不代表不是重試（同一個操作，重試時帶了不同的 trace id）。意圖只在呼叫端那裡，去重鍵就必須從那裡來。

## 去重：在操作外面攔住重複

冪等鍵是「讓 server 端點對外冪等」的一種做法。但更廣義地說，當你無法或不想改動操作本身，你可以在操作**前面**架一道閘，問一句「這筆我見過嗎」——見過就跳過。這道閘就是**去重（dedup）**。

冪等和去重常被當成同義詞，但它們其實是兩個正交的東西：冪等改的是**操作本身的數學性質**（讓 `f(f(x)) = f(x)`），去重不碰操作、而是在外圍**攔截重複的輸入**。你可以用去重讓一個本質上不冪等的操作對外表現得像只執行了一次——操作還是那個會累加的操作，只是重複的訊息根本沒機會抵達它。

任何去重方案，不管實作多花俏，都得回答三個問題，這三個問題構成了去重的三條軸：

| 軸 | 它決定什麼 | 常見選擇 |
|---|---|---|
| key（去重鍵） | 用什麼判斷兩筆是「同一筆」 | 訊息 id、業務自然鍵（order_id）、內容雜湊（SHA-256 of body） |
| window（去重視窗） | 「見過」這件事記多久 | 永久、固定 TTL、滑動視窗 |
| store（去重儲存） | 「見過的鍵」存在哪 | 記憶體、Redis、關聯式 DB 的唯一約束 |

這三條軸不是學術分類，每一條選錯都對應一個真實的故障。

**key 選錯**會兩頭出事。選得太粗（比如只用「使用者 id」當去重鍵），不同的操作會被誤判成同一筆而被誤殺；選得太細（比如把一個不該影響身分的欄位也算進去），真正的重複會逃過去重。內容雜湊式的去重特別容易在這裡踩雷：Amazon SQS FIFO 佇列的 content-based deduplication 對 message body 取 SHA-256 當去重鍵，但**不涵蓋 message attributes**（2026-06）。如果兩筆訊息 body 完全相同、只有 attribute 不同（例如各自帶了不同的 trace id），SQS 會算出相同的雜湊、把第二筆當成重複擋掉——而它其實是另一筆該被處理的訊息。內容雜湊去重要先確認雜湊涵蓋了所有「足以區分兩筆訊息」的欄位。

**store 選錯**會讓去重在最需要它的時候失憶。把「見過的鍵」存在行程的記憶體裡，看起來最快，但程式一重啟、節點一崩潰，這份記憶就清空了——而崩潰重啟正是重複最容易湧現的時刻，於是重啟後的第一波重複全部漏網。多節點時更糟：A 節點去重過的鍵，B 節點的記憶體裡根本沒有，同一筆訊息打到不同節點就各去各的、形同沒去。去重的 store 必須是**共享且持久**的，這就是為什麼實務上要麼用 Redis（跨節點共享、能設 TTL），要麼用關聯式資料庫的唯一約束（持久、且天然原子）。

**window 選錯**就是前面講過的 TTL 陷阱：視窗短於最壞重投間隔，重複會準時從視窗的盡頭溜進來。

## 去重最經典的那個 race，和它的正解

把去重講到機制級，必須直面它最容易寫錯的地方——一個幾乎每個人第一次都會踩的並發陷阱。

天真的去重邏輯長這樣：先 `SELECT` 查去重表裡有沒有這把鍵，沒有的話再 `INSERT` 進去並處理。讀起來毫無問題，但它在並發下會破。設想兩個 worker 拿到同一筆重複訊息、幾乎同時執行：

```
worker A                        worker B
  SELECT key -> not found
                                  SELECT key -> not found
  INSERT key
  process (charge $100)
                                  INSERT key   (depends on schema!)
                                  process (charge $100)   <-- duplicate!
```

兩個 worker 都在各自的 `SELECT` 那一刻查到「沒見過」，於是都認為自己是第一個、都往下處理。問題的根源是「查」和「寫」之間有一道縫，第二個 worker 的查發生在第一個 worker 的寫之前。去重本身有了 race condition——你想用去重擋住重複，結果去重自己被重複擊穿。

正解是**取消那道縫**，把「查與寫」壓成一個不可分割的原子動作，讓資料庫的原子性替你做仲裁。在 PostgreSQL 裡是一句 `INSERT ... ON CONFLICT DO NOTHING RETURNING ...`：

```sql
-- atomic claim: only the first inserter gets a row back
INSERT INTO processed_ids (key) VALUES ($1)
ON CONFLICT (key) DO NOTHING
RETURNING key;
```

關鍵在 `RETURNING` 的回傳值：第一個插進去的 worker 會拿到那一列（代表「我是第一個，我來處理」），後到的 worker 因為撞上唯一約束、`ON CONFLICT DO NOTHING` 讓它什麼也沒插，`RETURNING` 回空集合（代表「已經有人處理了，我跳過、直接 ack」）。整個「判斷我是不是第一個」的決定，由資料庫的唯一約束在一句 SQL 裡原子地完成，應用層完全不需要自己上鎖。在 Redis 上對應的是 `SET key value NX`——`NX` 表示「只在 key 不存在時才設成功」，回傳值直接告訴你是不是你設成功的；是，才處理。

這裡有個值得內化的觀念：**正確的去重，不是「先檢查再行動」（check-then-act），而是「行動本身就是檢查」。** 你不是先問「能不能進」再進，而是直接嘗試「佔位」，佔到了就代表你是第一個。這個從「先查後做」翻轉成「做的動作本身帶著仲裁」的思路，是並發正確性的一個通用模式，不只用在去重上（樂觀並發控制用版本號做同一件事，見〈樂觀與悲觀並發控制〉）。

## 把整條鏈算一遍

抽象講完，把一個真實的數字算到底，看看這些選擇在量級上意味著什麼。

一條推播管線每天處理 10 萬則訊息，佇列的重投率是 0.1%——也就是約有 `100,000 × 0.1% = 100` 筆訊息會在某一天被投遞兩次。這 100 筆重複該怎麼辦，**完全取決於這條管線在做什麼副作用**，而這正是「要不要去重、做到多嚴」的判準。

如果這條管線是「發放優惠券」，那 100 筆重複就是 100 張多發的券——真金白銀的損失。這時去重不是選項而是必須，而且最好用最可靠的那一招：如果每張券對應一個自然唯一鍵（比如 `(user_id, campaign_id)`），就直接在資料表上加 `UNIQUE(user_id, campaign_id)`。第二、三次發券的 `INSERT` 撞上約束失敗，消費者把這個「unique violation」解讀為「這張券已經發過了」、ack 掉這筆訊息——注意這一步：撞約束的錯必須被當成「成功（已處理）」吞掉，而不是當成失敗去重試，否則它會永遠撞、永遠重試、變成另一種毒訊息。零去重表、零鎖，靠資料庫一個唯一約束就擋掉了 at-least-once 的全部重複。這是去重最省力也最可靠的形態，前提是你有一個天然的唯一鍵可用。

但如果這條管線是「推送一則『你有新通知』的提醒」，那 100 則重複提醒幾乎無感——使用者頂多覺得手機多震了一下。為這種場景架一張去重表、扛它的儲存與 TTL 成本，是把工程預算花在沒人會痛的地方。同一個 0.1% 的重投率、同一個 100 筆，要不要去重的答案卻完全相反。所以面對重複，最有用的不是「怎麼去重」，而是先問一句：**「這筆重複處理一次，誰會痛？」** 沒人痛，就讓它重複，把省下的延遲、儲存與複雜度拿去守那些真正會痛的路徑。

## broker 說它幫你去重了，你還要做嗎

有些 broker 會宣稱自己提供去重甚至「exactly-once」，這很容易讓人鬆一口氣：「那我消費端就不用管了吧？」這個鬆懈往往是事故的起點，值得把 broker 去重的**邊界**看清楚。

broker 的去重是真的，但它有兩道你必須記住的圍籬。第一道是**視窗**：SQS FIFO 對同一個 `MessageDeduplicationId`，只在 **5 分鐘**的去重視窗內保證去重（2026-06）。5 分鐘內的重複它替你擋掉，超過 5 分鐘的同一個 dedup id，它會當成新訊息投遞。前面講的 DLQ 隔幾小時人工重放，就妥妥落在這道圍籬之外。第二道是**邊界**：broker 的去重只在它自己的世界裡成立。Kafka 的冪等生產者（`enable.idempotence`，Kafka 3.0 起預設開啟）給每個 producer 一個 PID、對每個 partition 維護一個單調遞增的序號，broker 只接受「序號剛好比上一筆大 1」的訊息，藉此擋掉生產者重試造成的重複——這套機制乾淨漂亮，但它有兩個刺：producer 一旦**重啟**就拿到新的 PID、序號歸零，跨重啟的重複它認不出來；而且它只防 broker 端的寫入重複，**完全管不到你消費這筆訊息之後對外做的副作用**。

把這兩道圍籬內化成一句話：**broker 去重是第一層紗窗，不是最後一道防線。** 你的消費者一旦「收到訊息 → 呼叫外部支付 API → 然後才 ack」，那個外部呼叫就站在 broker 所有保證的邊界之外了——消費者可能在呼叫之後、ack 之前崩潰，broker 看不到那次外部副作用發生過，照樣重投。所以即使坐在一個號稱 exactly-once 的 broker 上，只要你的副作用落在 broker 之外（幾乎總是如此），跨越它視窗或邊界的那些重複，仍然要消費端用冪等鍵或去重自己擋（為什麼 exactly-once delivery 在原理上不可能、只能靠 at-least-once 加去重合成，見〈三種交付語意〉）。

## 為什麼是這個形狀

退一步看，這一整套機制——冪等、冪等鍵、去重表、唯一約束——都是從同一個無法迴避的事實裡長出來的：**你想要「不丟」，就必須接受「會重」；而既然會重，總得有人在某個地方把重複收乾淨。**

收的地方有講究。最理想的情況，操作本身就冪等（設定目標狀態），重複從一開始就無害，什麼都不必做。退一步，操作有天然的唯一鍵，就讓資料庫的唯一約束在一句原子的 `INSERT` 裡替你仲裁，連去重表都省了。再退一步，操作既不冪等又沒有自然鍵、還對外有副作用，才需要動用冪等鍵——讓唯一握有「這是同一件事」這個意圖的呼叫端生成並攜帶一把鍵，讓被呼叫端原子地 CLAIM、連失敗一起 STORE、設好涵蓋最壞重投窗的 TTL。每一層都比上一層費工，所以你從最省的那層開始選，能在更前面收乾淨就別拖到後面。

而所有這些機制都共享同一個底層紀律：**正確的去重不是「先檢查再行動」，而是讓「行動本身就是那個檢查」。** 你不去問「我能不能處理這筆」，你直接去搶那個「我是第一個」的位子——搶到了才動手。先查後做留下的那道縫，正是重複擊穿去重的地方；把縫消滅在一個原子動作裡，去重才扛得住並發。

下次你看到一個支付 API 要你帶 `Idempotency-Key`，或一段消費者程式在處理前先 `INSERT ... ON CONFLICT`，你會知道那不是多此一舉的防禦性編程——那是它在兌現「不丟」這個承諾時，必然欠下的另一半債，正在被人認真地還掉。重複是常態；讓重複不留痕跡，才是本事。
