# 分頁：offset 與 cursor/keyset、深分頁

某個 API 上線時毫無破綻。某個列表端點，背後一張 `events` 表，使用者一頁 20 筆地往下翻，`p50` 在個位數毫秒，`p99` 也漂亮。半年後，這張表長到五千萬列，告警開始零星地響：同一個端點，`p99` 偶爾衝到一兩秒。翻去看慢查詢，發現都集中在「翻得很深」的那些請求——有人（通常是爬蟲、或某個重試卡住的 client）一路翻到第幾千頁。同一句 SQL，第 1 頁飛快，第 5000 頁卻在資料庫裡磨了一秒。

這不是退化，是這個寫法**從第一天就埋好**的時間炸彈，只是要等資料量長到一定程度才引爆。要看懂它，得先承認分頁這件事比它看起來難——難不在「把結果切成一頁一頁」，而在**底下那份資料是活的**：你翻頁的同時，有人在插入、有人在刪除，於是「第 2 頁」這個概念本身就在腳下漂移。所有分頁機制的分歧，歸根到底都是在回答同一個問題：**用什麼當「接著上次往下」的座標？**

## 天真的座標：跳過 N 筆

最直覺的分頁，是把「第幾頁」翻譯成「跳過前面幾筆」：

```sql
SELECT * FROM events
ORDER BY created_at DESC
LIMIT 20 OFFSET 199980;   -- 第 10,000 頁，每頁 20 筆
```

`OFFSET 199980` 讀起來像是「直接跳到第 199,981 筆」，彷彿資料庫能憑空定位到那裡。但資料庫做不到——它沒有「第 199,981 筆在哪」的捷徑。它能做的，是**沿著 `created_at` 的順序，從頭一筆一筆往下數，數到第 199,980 筆為止，全部讀出來、再全部丟掉**，然後才開始收集你真正要的那 20 筆。

這就是炸彈的引信：`OFFSET` 的成本和它的數值**成正比**。第 1 頁 `OFFSET 0`，不丟任何東西；第 10,000 頁要先讀進近二十萬列、排好序、再原封不動扔掉 199,980 列，只為了拿後面 20 列。即使 `created_at` 上有索引，索引也只幫你按順序走，省不掉「走過並丟棄前面那一大段」的工。頁碼越深，被丟掉的部分越大，延遲就越往上爬——這正是 **深分頁（deep pagination）** 問題。一個五千萬列的表，深處的單頁延遲從個位數毫秒漲到數百毫秒甚至秒級，是這個機制的必然，不是 bug。

offset 還有第二個、更隱蔽的毛病，和效能無關，和**正確性**有關。回想資料是活的：第 1–20 名屬第 1 頁、第 21–40 名屬第 2 頁，而名次是在你查每一頁的當下才重新算的。你看完第 1 頁、正要翻第 2 頁的這幾秒之間，如果有人在列表最頂端插入了一筆新 event，那麼所有列一起往後擠一名——原本第 20 名（第 1 頁最後一筆、你**已經看過**）被擠成第 21 名，而你的第 2 頁是 `OFFSET 20`，正好從第 21 名開始撈，於是把這筆**重複看一次**。反過來，如果頂端刪掉一筆，所有列往前遞補一名——原本排在第 21 名、本該是第 2 頁第 1 筆、你**還沒看過**的那一列，遞補成第 20 名而落進了第 1 頁的範圍；你的第 2 頁從新的第 21 名（原本的第 22 名）開始，那筆原第 21 名就**永遠不會出現在任何一頁**，你**漏掉一筆**。

問題的根源很乾淨：offset 用「位置序號」當座標，而位置序號會隨著資料增刪而**重新編號**。你昨天記下的「跳過前 20 筆」，今天指的已經不是同一批 20 筆了。座標漂了，頁與頁之間就會漏與重。

那 offset 一無是處嗎？不。它買到一個 cursor 永遠給不了的能力：**跳到任意頁碼**。`OFFSET 199980` 雖然慢，但它真的能讓你「直接去第 10,000 頁」，也能算出「第 7 頁」「上一頁」這種需要絕對頁碼的東西。一個後台管理介面、總筆數只有幾千筆、使用者想看「第 3 頁／共 17 頁」並隨意跳——這種場景 offset 又快又簡單，硬換成別的反而是過度設計。offset 的問題從來不是「它錯」，而是「把它用在一張會長大的大表、又做成對外、又被翻得很深」。

## 換一個不會漂的座標

要根治深分頁與漏重，得把座標從「第幾筆」換成**資料本身的值**。不要再問「跳過前面 N 筆」，改問「給我排在**這一筆之後**的那些」——而「這一筆」用它的排序鍵值來指認，不是用它的序號。這就是 **keyset 分頁**，也叫 **cursor 分頁** 或 **seek method（尋位法）**。

假設我們照 `created_at` 由新到舊翻，上一頁最後一筆的時間戳是 `2026-06-20 14:03:00`。下一頁不再是 `OFFSET 20`，而是：

```sql
SELECT * FROM events
WHERE created_at < '2026-06-20 14:03:00'
ORDER BY created_at DESC
LIMIT 20;
```

差別是質變的。`WHERE created_at < ...` 配上 `created_at` 索引，資料庫能直接在索引上**定位（seek）**到那個時間點，然後順著索引往下取 20 筆就收工——它**完全不碰**前面那一大段資料。第 1 頁這樣做、第 10,000 頁也這樣做，成本一模一樣：一次 index seek 加 20 筆讀取。延遲不再隨頁碼增長，而是**恆定**的。前面那個「第 5000 頁磨一秒」的場景，換成 keyset 後是個位數毫秒，與翻到第幾頁無關。深分頁問題就此消失，因為「深」這個概念本身被取消了——每一頁都是從一個明確的值開始的「下一段」，沒有「跳過多深」可言。

漏與重也一併解決。座標現在是 `created_at = '2026-06-20 14:03:00'` 這個**具體的值**，不是「第 20 個位置」。中間有人插入或刪除，都不會改變這個值的含義——`< '2026-06-20 14:03:00'` 永遠指同一條邊界線。新插入的 event 時間戳比它新，落在這條線**上方**、不進這一頁；被刪掉的也不會讓邊界平移。座標釘在資料上，資料怎麼增刪，邊界紋風不動。

代價也很直接，而且要老實認帳：**keyset 不能跳到任意頁碼。** 它只知道「比某個值更舊的下一段」，不知道「第 10,000 頁從哪個值開始」——要知道那個值，你得先一頁頁翻到那裡。所以 keyset 天生只服務「下一頁／上一頁」式的順序翻動：無限下拉、活動流、時間線、給程式消費的 feed。這些場景本來就沒有人會手動跳到第 1 萬頁，失去跳頁能力多半不痛。但若需求真的要「輸入頁碼直接跳」，就得承認 keyset 不適合，回去用 offset，別硬把 cursor 包裝成假頁碼去騙自己。

## 那個會在邊界吃掉一筆的陷阱

到這裡 keyset 看起來很美，但它藏著一個**不踩到不會痛、踩到很難查**的坑，值得整段拆開講透——因為它正是 keyset 從「看懂」到「做對」之間最容易摔的一階。

問題出在 `created_at` **不是唯一的**。時間戳的精度有限，高吞吐下，同一毫秒（甚至同一微秒）湧進好幾筆 event 是常態。假設這 20 筆裡，最後一筆的 `created_at` 是 `14:03:00.000`，而緊接在它**後面**、本該屬於下一頁第一筆的那筆，`created_at` **也是** `14:03:00.000`——兩筆撞在同一個時間戳上。

現在看下一頁的條件 `WHERE created_at < '14:03:00.000'`。那筆和邊界**相等**的 event，因為是嚴格小於、不含等於，被排除在外——它**永遠不會出現在任何一頁裡**。你靜悄悄地漏掉了一筆，而且因為它夾在頁與頁的接縫處，肉眼翻列表根本看不出來。反過來，若你為了不漏而把條件放寬成 `<=`，那筆又會**同時出現在這一頁的尾巴和下一頁的開頭**，變成重複。`<` 漏一筆、`<=` 重一筆，兩邊都不對——根因是**用一個不唯一的鍵當座標，邊界本身就是模糊的**。

解法是給排序鍵補一個**唯一的 tie-breaker（破平手鍵）**，通常就是主鍵 `id`。把排序與座標都升級成複合的 `(created_at, id)`：

```sql
SELECT * FROM events
WHERE (created_at, id) < ('2026-06-20 14:03:00.000', 8801234)
ORDER BY created_at DESC, id DESC
LIMIT 20;
```

這裡 `(created_at, id) < (...)` 是 SQL 的 **row value（列值）比較**，PostgreSQL 與 MySQL 都支援，語意是**字典序**：先比 `created_at`，只有在 `created_at` 相等時才用 `id` 來定勝負。展開來看，它精確等價於：

```sql
WHERE created_at < '2026-06-20 14:03:00.000'
   OR (created_at = '2026-06-20 14:03:00.000' AND id < 8801234)
```

加上 `id` 之後，那兩筆撞在同一毫秒的 event，被 `id` 拉開成兩個不同的座標點：邊界從「時間戳 `14:03:00.000`」這條會壓到多筆的粗線，收細成「`(14:03:00.000, 8801234)` 這一個點」，精確地落在某兩筆之間。同一時間戳的多筆現在有了確定的先後，邊界穿過它們時不會再漏、也不會再重。**只要排序鍵能對到唯一一行，keyset 的邊界就是乾淨的；不唯一，就一定要補 tie-breaker，沒有例外。**

這裡還有一個只在某些資料庫上現形的暗角，值得資深讀者留意，而且兩大資料庫的脾氣正好相反。理論上 row value 比較和展開的 `AND`／`OR` 形式**結果**完全等價，但**查詢計畫**不見得等價。PostgreSQL 的優化器懂得把原生的 `(created_at, id) < (...)` 直接對應到 `(created_at, id)` 複合索引上的一次範圍掃描，所以在 PG 上原生 row value 寫法又乾淨又快。MySQL 卻**反過來**：直到 8.4，它的優化器對原生 row-constructor 不等式**不一定**走範圍掃描，反而可能退化成掃過整段索引甚至全表（見官方〈Row Constructor Expression Optimization〉與 Bug #111952），而展開成 `created_at < ... OR (created_at = ... AND id < ...)` 的**非建構式**形式才能完整吃到索引前綴、走成 index range——MySQL 官方文件正是建議「避免把 row constructor 和 `AND`／`OR` 混用」「改寫成等價的非建構式表達式才能更完整地用到索引」。也就是說，明明邏輯一樣的兩種寫法，效能可能差一截，而且**哪種快還因庫而異**：沒有一種「可攜最佳寫法」——寫 keyset 時務必對所用的資料庫實際 `EXPLAIN`／看執行計畫，PG 偏好原生 row value、MySQL 上若退化就改用 `AND`／`OR` 展開式。座標選對了還不夠，得讓優化器**也看懂**你選的座標。

## 把座標藏起來：opaque cursor

到目前為止，「下一頁」需要 client 帶回 `created_at` 和 `id` 兩個值。如果這是一個對外的公開 API，把這兩個內部欄位**裸露**給 client 是個長期負債：client 會開始依賴「cursor 就是時間戳加 id」這個事實，自己拼湊 cursor、甚至猜測別人的 cursor；而你哪天想把排序從 `created_at` 換成別的、或加一個欄位進座標，所有依賴舊格式的 client 就全破了——你的內部分頁細節，悄悄變成了不能改的公開契約。

慣例的解法是 **opaque cursor（不透明游標）**：把 `(created_at, id)` 這組座標——往往連同排序方向、套用的 filter 一起——序列化、base64 編碼成一個沒有語意的字串，回給 client：

```
eyJjcmVhdGVkX2F0IjoiMjAyNi0wNi0yMFQxNDowMzowMC4wMDBaIiwiaWQiOjg4MDEyMzR9
```

client 對這串東西**唯一能做的事，就是下次原封不動帶回來**。它不該、也（被鼓勵地）無法解讀裡面是什麼。GraphQL 的 **Cursor Connections 規格**——也就是 Relay 風格的 `edges`／`cursor`／`pageInfo`——把這件事訂成了事實標準：cursor 被明定為一個 **opaque string**，「除了它的值以外沒有任何意義」，並建議用 base64 編碼**正是為了**讓人別想去依賴它的格式。回應裡的 `pageInfo` 帶 `hasNextPage` 和 `endCursor`，client 拿 `endCursor` 當下一次的 `after` 參數，完全不必知道座標長什麼樣。

opaque 買到的是**演進的自由**：cursor 的內部格式是你的私事，今天裝 `(created_at, id)`、明天想多塞一個分片編號或排序方向的雜湊，都不影響 client——它只是把那串 base64 收下再吐回來。代價是 client **失去了自行構造 cursor 的能力**（這正是目的），以及你得處理 cursor **失效**的情形：如果 cursor 編碼了排序欄位，而 client 拿著「按時間排序」的舊 cursor 卻請求「按金額排序」，這個座標就對不上了，server 得能偵測到並乾脆地回一個錯誤，而不是裝作沒事、回出一頁順序錯亂的資料。需要的話，cursor 裡還可以加一段簽章或雜湊來防 client 竄改——畢竟一旦它被當成不可猜測的 token，就別讓人能偽造它指到別處去。

## 為什麼 COUNT 也是一個陷阱

分頁的 UI 常常想顯示「第 3 頁 / 共 417 頁」，而那個「417」要靠 `SELECT COUNT(*)` 算出總筆數再除以每頁大小。在小表上這沒什麼，但在那張五千萬列的大表上，**精確的 `COUNT(*)` 本身可能比查那一頁資料還貴**——它得掃過符合條件的所有列才能數出總數，而你要的那 20 筆只是一次 seek。於是會出現一個荒謬的局面：使用者每翻一頁，後端跑兩個查詢，便宜的那個（取 20 筆）個位數毫秒，貴的那個（數總數）幾百毫秒，慢的拖累快的。

這也是 keyset 的世界觀更誠實的地方：它**根本不提供總頁數**，因為它連「現在是第幾頁」都不知道。它只回答一個便宜得多的問題——「還有沒有下一頁？」而這個問題有個漂亮的小技巧：要拿每頁 20 筆，就**取 21 筆**。如果真的取到了 21 筆，代表後面還有，`hasNextPage = true`，把第 21 筆丟掉、只回前 20 筆；如果只取到 20 筆或更少，代表這就是最後一頁。多取一筆，就用一次同樣的 seek 換到了「有無下一頁」，完全不需要另跑一次 `COUNT`。

如果產品**非要**一個總數不可，務實的做法是承認它只能是近似值：用資料庫的統計資訊估個量級（PostgreSQL 的 `reltuples`、或 `EXPLAIN` 的列數估計），或快取一個定期更新的近似總數，而不是每翻一頁都要求一個分毫不差、且隨時在變的精確值。在一張高頻寫入的表上，「精確總數」這個概念本身就有點虛——你 `COUNT` 完的那一刻，它已經變了。

## 當分頁要求「翻完之前這份結果不准變」

keyset 解決了「邊界穩定」，但它解決的是**單筆**的漏與重——某筆 event 不會因為鄰居增刪而被跳過或重看。它沒解決、也解不了另一類更強的需求：**整輪翻頁期間，整份結果集要像被凍結了一樣，從頭到尾是同一個快照。**

這種需求出現在匯出、對帳、報表這類場景：你要把符合條件的五百萬筆全部撈出來落地，過程中如果底層資料一直在動，你撈到的就是一份「東邊新、西邊舊」的拼貼，可能對不上總帳。keyset 雖然每一頁邊界穩，但翻第 1 頁和翻第 1 萬頁之間隔了幾分鐘，這幾分鐘裡新插入又符合條件的資料，照樣會被你後面的頁撈進來——你要的是「開始那一刻的世界」，keyset 給的是「每一刻都最新」。

這時要的是**快照式分頁**，由儲存層提供一個固定的時間切面。搜尋引擎這邊的演化很能說明問題。Elasticsearch 早年用 **scroll API** 來解這件事：開一個 scroll，就在 server 端凍結一份搜尋上下文，之後一路翻都讀同一份快照。但 scroll 有個沉重的代價——它把狀態**留在 server 上**，每個 scroll 都吃 server 的記憶體與資源，並發一多就壓力山大。所以現在（2026-06）官方對深分頁的建議已轉向 **`search_after` 配 PIT（Point In Time）**：PIT 凍結一個輕量的時間切面、回一個 PIT id，`search_after` 則是不折不扣的 keyset——帶上一頁最後一筆的 sort values 往下取，**本身無狀態**。兩者合起來，既有 scroll 的快照一致性，又把翻頁的位置交給 client 用 keyset 攜帶、不在 server 端囤積上下文。scroll 至今沒被正式廢棄，但已**不再推薦**用於深分頁，只留給匯出、重建索引這類整批掃描的場景。

順帶一提搜尋引擎為什麼對深分頁格外敏感：它的資料散在多個分片上，要回「全域第 199,981 名到 200,000 名」，每個分片都得各自把**前 from + size 筆**排好、送回協調節點再做一次合併排序。記憶體與耗時和 `from + size` 成正比，而且是**每個分片乘上去**。這就是 Elasticsearch 預設 `index.max_result_window = 10000`、`from + size` 一超過一萬就直接報錯擋下的原因——它不是小氣，是替你擋住一個會把資料節點打到 OOM 的查詢。`search_after` 之所以能繞過這道牆，正因為它是 seek 而非 skip：每個分片只需從「上一頁最後那筆」的 sort values 之後取一段，不必為了第 1 萬頁去排前一萬筆。同一個道理，在搜尋引擎裡又演了一遍。

## 為什麼分頁是這個形狀

把這一章收束成一句話：**分頁的全部分歧，都是在「用什麼當座標」上的取捨。**

用**位置序號**當座標（offset），你買到「跳任意頁」的自由，付出的是兩筆債——成本隨深度線性上升，以及座標會隨資料增刪而重新編號、害你在頁的接縫漏與重。這兩筆債在小表、淺頁、後台場景小到可以忽略，所以 offset 在那裡是對的、簡單的選擇；它們在大表、深頁、對外 feed 上會放大成線上事故，所以那裡要換座標。

用**資料的值**當座標（keyset／cursor），你買到恆定的單頁成本和穩定的邊界，付出的是「只能順著翻、不能跳頁」，以及必須老實面對兩件事——排序鍵不唯一就一定要補 tie-breaker，否則邊界會悄悄吃掉或重複一筆；想對外就把座標包成 opaque cursor，把演進的自由留給自己、把猜測的衝動擋在 client 外。

而當需求從「每一頁穩」升級到「整份結果凍結」，連 keyset 都不夠，得請儲存層給一個快照——這也是同一條光譜往更強一致那端再走一步，代價是 server 端要為那個切面付出狀態或資源。

所以下次你寫下 `LIMIT 20 OFFSET ...`，可以停一秒問自己：這張表會長大嗎？這個端點會被翻深嗎？它是給人手點、還是給程式拉？答案會直接告訴你，該用哪一種座標——以及，那顆要等半年才引爆的炸彈，到底有沒有被你提前拆掉。
