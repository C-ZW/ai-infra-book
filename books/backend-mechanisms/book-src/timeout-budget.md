# timeout、deadline 與 budget：把等待變成有界

凌晨三點，一個本來該在 200 毫秒內回覆的 API 端點開始堆積。不是回錯，是不回——請求進來，然後什麼都沒發生。打開監控，你會看到一條詭異的曲線：錯誤率是零，延遲圖卻沒有 p99，因為那些請求根本還沒「完成」到能被計入延遲分布。它們卡在某個地方，握著一條執行緒、一條連線、一塊在堆上等著被填滿的緩衝區，安安靜靜地不死也不活。再過幾分鐘，執行緒池見底，連那些本來毫秒級就能回的健康請求也排不進來。整個服務不是「變慢」，是「掛起」——而你的告警裡，沒有任何一條規則叫做「太多請求永遠不回來」。

這一幕的根源簡單到近乎荒謬：某個跨越行程邊界的呼叫，**沒有等待的上限**。

## 沒有上限的等待，不是慢，是資源洩漏

先把問題講難。一個行程內的函式呼叫，最壞情況是無窮迴圈，CPU 燒滿，至少你看得到它在動。但任何**跨越行程邊界**的呼叫——一個 TCP socket 上的 `recv`、一次資料庫查詢、一個對子行程的 wait、一把分散式檔案鎖——它的最壞情況不是「很慢」，而是「永遠」。對端可能崩潰了、可能網路把回程封包吞了、可能對方還活著但卡在它自己的某個下游上。從你這一端看出去，這些情況**長得一模一樣**：你發了請求，然後什麼都沒回來。

關鍵在於，一個阻塞在 socket 上的呼叫並不是「閒著」。它正握著一整串資源不放：

- 一條執行緒（或一個 async task 的 stack frame），在它回傳之前不會被回收；
- 一條連線池裡的連線，別人借不走；
- 一塊接收緩衝區、一個請求物件、它一路引用到的所有東西，全都不能被 GC；
- 在更高層，它可能還佔著一個 semaphore、一個 in-flight 計數的名額。

於是「一個慢下游」會透過這串被凍結的資源，把壓力一路往上傳。下游慢 → 上游的呼叫卡住 → 卡住的呼叫佔滿執行緒池 → 上游自己也開始無法回應它的上游。這不是「下游壞了」這麼單純，而是**下游的慢，透過沒有上限的等待，變成了上游的死**。逾時要解的，正是這件事：給每一次「把命交到別人手上」的等待，劃一條硬線——過了線，我就放棄、回一個明確的錯誤、把手上的資源全部還回去。

所以這裡有一個容易被講反的第一原理：**逾時的本質不是「猜對方多久會回」，而是「我願意為這個呼叫凍結多少資源」。** 你設 timeout = 2 秒，真正的意思不是「我賭它 2 秒內會回」，而是「我最多願意讓這條執行緒、這條連線，為這一個呼叫被綁住 2 秒；超過，這些資源對我活下去更重要」。一旦這樣想，後面很多反直覺的取捨就順了。

## 一個分散式系統裡無法迴避的真相

往下挖一層，逾時其實坐在一個更冷酷的事實上：**在一個訊息延遲沒有上界的網路裡，你無法區分「對方死了」和「對方只是很慢」。**

一個沉默的節點，可能是已經當機、可能是正在做一個很重的計算、可能是它的回應已經發出但塞在某個交換器的佇列裡。這三種情況該有的反應完全不同——但你手上唯一的偵測工具，就是「等一段時間，看它回不回」。換句話說，**逾時是分散式系統裡唯一的失敗偵測器**，而它先天測不準：它只能告訴你「對方沒在我設的窗口內回應」，永遠無法證明「對方死了」。設短了，你會把一個只是稍微慢一點的健康節點誤判成死亡，徒增重試與抖動；設長了，一個真的死掉的節點會把你的資源凍結到天荒地老（這個「慢與死分不開」的根本侷限本身值得單獨深究，見〈失敗偵測：逾時是唯一手段、慢與死分不開〉）。

理解了這一點，你就懂了為什麼「timeout 設大一點比較安全」是徹底反了的直覺。timeout 越大，一個慢依賴能凍結你的資源越多、越久。一個設成 30 秒的 timeout，意思是「我願意讓一個可能早就死了的對端，把我的一條執行緒綁架半分鐘」。在過載的瞬間，正是這種「寬容」把執行緒池一口氣抽乾。timeout 該設在「正常 p99 之上留一點餘裕」——比如正常 p99 是 120 毫秒，你設 300 毫秒，容忍偶發的尾巴，但絕不縱容一個卡死的呼叫——而不是「設大到絕不誤殺」。**寧可偶爾誤殺一個慢請求並重試，也不要讓死掉的呼叫無限期吃住資源。**

## 連線、讀取、整體：一個 timeout 不夠用

天真的做法是給一個呼叫設「一個」timeout。但一次 HTTP 呼叫其實是好幾個階段串起來的，把它們混成一個值，會在錯誤的地方用錯誤的耐心。

最該分開的是**建立連線**和**等待回應**這兩段：

- **connect timeout**：建立 TCP 連線（含 TLS 握手）要花多久。這一段該設得**很短**——毫秒到一兩秒級。連不上一台機器，通常代表它死了、網路斷了、或 DNS 解錯了，這些是「快速失敗、換一台」的訊號，沒有理由耐心等。一個健康的同資料中心連線握手是個位數毫秒；如果 connect 都要好幾秒，等下去也多半是徒勞。
- **read / response timeout**：連上之後，等對方把回應算完、送回來要多久。這一段要按**業務的 p99** 設，因為它真的取決於對方要做多少事——一個複雜報表查詢花 800 毫秒是正常的，不該被一個短 timeout 砍掉。

很多 HTTP client 預設只給你一個「總 timeout」，把這兩段揉在一起。後果是：你為了容忍那個慢報表查詢，被迫把總 timeout 設成 2 秒；於是當某台後端機器真的死了、連線根本建不起來時，你的呼叫也要傻等 2 秒才放棄，而不是在 200 毫秒內就快速失敗、轉去重試別台。**連線階段該秒級失敗，回應階段該按業務容忍，兩者量級差很多，混成一個值總是在遷就其中一邊。**

除了這兩個，成熟的 client 還會分出 **idle / socket timeout**（連線建好、開始傳輸後，兩次收到資料之間最多能空多久——防的是「連上了、回了一半、然後卡住」這種半死不活的連線）。階段分得越細，你就越能對「在哪一步卡住」做出對的反應。

## 從 timeout 到 deadline：為什麼要換一種記時方式

現在把場景放大到微服務。一個對外端點 G，收到請求後要依序呼叫內部服務 A，A 回來後再呼叫 B。每個呼叫你都老實設了 timeout = 1000 毫秒。看起來很安全——每一跳都有上限。

但每一跳的 timeout 是**獨立重新計時**的。某次請求裡，A 慢吞吞用了 950 毫秒才回（沒逾時，剛好壓線），接著 G 呼叫 B，B 又用了 900 毫秒（也沒逾時）。兩個呼叫各自都「健康」，監控上不會亮任何紅燈——但這個使用者實際等了 950 + 900 = 1850 毫秒，幾乎是你對外承諾的兩倍。沒有任何一層覺得自己錯了，錯誤就這樣從每一層的縫隙裡漏了出去。

問題出在 timeout 的語意：它是一個**相對時長**，「從現在起最多等 1000 毫秒」，而「現在」對每一跳都是新的。每一層都從零開始給自己一整份耐心，串起來自然會累加爆掉。

修正的關鍵，是把記時方式從「相對時長」換成「絕對時間點」：

- **timeout（逾時）** ＝ 相對：「從現在起最多等 1000 毫秒」。它是 socket / library 層的設定，每一跳重新計時，天生不會跨跳累加。
- **deadline（截止點）** ＝ 絕對：「不管轉了幾手，最晚到 12:00:03.250 為止」。它描述的是這**整個請求**的死線，而不是某一跳的耐心。

deadline 的威力在於它**可以一路往下游傳**。端點 G 在入口處算出 deadline = now + 1000ms，呼叫 A 時把這個 deadline 帶下去；A 在 950 毫秒後回來，此刻離 deadline 只剩 50 毫秒；G 呼叫 B 時，把「剩 50 毫秒」當成給 B 的 timeout 傳下去。B 一看只剩 50 毫秒、自知做不完，要嘛立刻放棄回一個明確的「逾時」，要嘛盡力在 50 毫秒內回——無論如何，**整個請求準時失敗，而不是悄悄超時一倍**。同一筆截止點被所有層共享，過期的請求會在鏈條的每一節一起放棄，沒有人會去算一個早就沒人要的結果。

把這兩種記時擺在一起，差別一目了然：

| | 固定 per-call timeout | deadline 傳遞 |
|---|---|---|
| 記的是 | 每一跳的相對耐心 | 整個請求的絕對死線 |
| 跨跳行為 | 各跳獨立、會累加 | 共享同一條線、一起放棄 |
| 深鏈總時間 | 各層相加，可能遠超承諾 | 被入口的 deadline 封頂 |
| 過期請求 | 每層各自還在傻等 | 全鏈路一起放棄、不浪費算力 |
| 成本 | 零，每個 client 都會 | 需框架支援＋下游檢查 deadline |

## gRPC 怎麼把這件事做對：grpc-timeout 與時鐘偏移

deadline 傳遞聽起來很美，但有一個魔鬼藏在「絕對時間點」這四個字裡：**G 機器上的「12:00:03.250」和 B 機器上的「12:00:03.250」不是同一個瞬間。** 兩台機器的牆鐘各跑各的，差個幾十毫秒是家常便飯，差到幾秒也不稀奇。如果 G 把它本地時鐘算出的絕對時間點原封不動丟給 B，B 拿自己的時鐘去比，得到的「還剩多少」可能整個錯位——B 的鐘快了 80 毫秒，它會以為死線早就過了，把一個還來得及的請求直接砍掉；慢了，又會多等出不該等的時間。

gRPC 的做法值得記下來，因為它優雅地繞開了整個時鐘同步的泥潭：**它在概念上用 deadline，但在線上傳的是 timeout。** 具體說，gRPC client 內部維護一個絕對 deadline；當它要對下游發 RPC 時，它把 deadline **換算回「從現在起還剩多少時長」**——一個相對值——再把這個相對值寫進一個叫 `grpc-timeout` 的 HTTP/2 header 傳出去。下游收到的不是「某個絕對時刻」，而是「你還有 X 時長」，於是下游用**自己的**時鐘從收到的那一刻開始倒數即可，完全不必和上游的鐘對齊。

```
G 機器（deadline = T_end，本地時鐘）
  發 RPC 給 A 前：remaining = T_end - now_G
  在 wire 上送：grpc-timeout: <remaining>      <- 相對時長，不是絕對時刻
        |
        v
A 機器（自己的時鐘，與 G 不必同步）
  收到 header，立刻換算回本地 deadline：
        deadline_A = now_A + remaining
  發 RPC 給 B 前：remaining' = deadline_A - now_A   <- 已扣掉 A 花掉的時間
  在 wire 上送：grpc-timeout: <remaining'>
```

每一跳都「把絕對死線換成相對剩餘時長」送出、收到後「再用本地時鐘換回絕對死線」，於是**clock skew 在換算中被自然抵消**——傳的永遠是「還剩多久」這種相對量，而相對量不在乎你倆的鐘指向幾點。途中每經一手，扣掉的就是那一手實際花掉的處理時間，剩餘額度單調遞減。

`grpc-timeout` header 的線上格式也設計得很省：一個最多 8 位數的十進位整數，後面跟一個單位字母——`H`（時）、`M`（分）、`S`（秒）、`m`（毫秒）、`u`（微秒）、`n`（奈秒）。所以「還剩 50 毫秒」就是 `grpc-timeout: 50m`，「還剩 1 秒」就是 `grpc-timeout: 1S`。下游若在死線內沒做完，回的是 gRPC 狀態碼 **4（`DEADLINE_EXCEEDED`）**——一個全鏈路都認得的「時間到了」訊號。

這套機制不是每個 gRPC 實作都預設開。**Go 與 Java 預設就會自動把入站 RPC 的 deadline 傳遞給出站 RPC；C++ 等則需要顯式開啟**（2026-06）。還有一個更基本、也更危險的預設：**gRPC client 本身預設不設任何 deadline**——官方文件原話是「By default, gRPC does not set a deadline」，意思是一個沒人替它設死線的呼叫，可以「effectively forever」地等下去。換句話說，前面那整套精巧的傳遞機制，要有人在入口處先放下第一個 deadline 才會啟動；忘了設，整條鏈就退回「可能永遠等」的最壞情況。

## budget：把一份時間預算切給多個下游

deadline 解決了「跨跳不累加」，但還有另一個維度的問題：當一個端點要**扇出**呼叫多個下游時，這些下游該怎麼分這份總時間？

假設端點 G 對外承諾 p99 ≤ 1000 毫秒。它內部要呼叫 A、B、C 三個下游才能拼出回應。天真的做法是三個都設 timeout = 1000 毫秒——但如果它們是**串行**的，最壞情況就是 3000 毫秒，三倍超支。就算它們是並行的、總時間取最慢那個，你也還欠著序列化、反序列化、排隊、GC 暫停這些「膠水時間」沒算進去。

**budget（時間預算）** 就是把這份對外承諾，**主動拆分**給各段，讓它們相加（或取最大，視串並行而定）之後仍然守得住承諾。對串行的 A→B→C，你可能這樣拆：

```
對外承諾 p99 = 1000 ms，內部串行呼叫 A -> B -> C
  A: 350 ms
  B: 350 ms
  C: 200 ms
  留白（序列化、排隊、GC、網路餘裕）: 100 ms
  ---------------------------------
  合計 1000 ms
```

每一段拿到的不是「全部的 1000 毫秒」，而是它在這份預算裡分到的那一格。拆分有兩條鐵律。第一，**一定要留白**——序列化、跨執行緒排隊、一次不巧的 GC 暫停，這些都要花時間，把預算榨到一毫秒不剩，意味著任何一點抖動都會讓你破線。第二，**某一段超支會吃掉別段的額度**——如果用 deadline 傳遞來實作這份 budget（這正是 deadline 的自然用法），A 多花了 100 毫秒，B 和 C 拿到的剩餘時間就真的少了 100 毫秒，預算是一個會被前面的人花掉的共享池，不是各自獨立的配額。

budget 還有一個常被忽略、但同樣關鍵的親戚：**retry budget（重試預算）**。它管的不是時間怎麼分，而是**重試能佔總請求量的多大比例**。這引出下一個更隱蔽的陷阱。

## 沒對齊的多層 timeout，會讓重試全成白工

逾時和重試（重試的退避與 jitter 做法見〈重試、退避與 jitter〉）湊在一起時，有一個非常容易踩、踩了又很難看出來的坑：**多層 timeout 沒有對齊。**

設想兩層：外層端點對使用者設了 timeout = 1000 毫秒；它呼叫的內層服務設了 timeout = 800 毫秒，而且內層在失敗時會重試 2 次。直覺上「內層有重試，更穩」，對吧？算一下就知道不對。內層第一次嘗試最壞等 800 毫秒——但外層在 1000 毫秒就放棄了。於是第一次嘗試還沒逾時、或剛逾時要啟動第二次重試的那一刻，**外層早已對使用者回了錯、掛斷了連線**。內層後面那兩次重試，是在為一個沒有人還在等的請求賣力——純粹的白工，而且更糟的是，它們還在持續對那個本來就慢、可能正過載的下游加壓，把火上澆油。

正確的關係是一條可以背下來的不等式：**外層 timeout ≥ 內層 timeout ×（1 ＋ 內層重試次數）**，再加上重試之間的退避時間。內層要試 3 次（1 次原始＋2 次重試）、每次上限 800 毫秒，那它最壞要 2400 毫秒以上才跑得完一輪完整的重試——外層若只給 1000 毫秒，內層的重試機制根本沒有時間落地，等於形同虛設。要嘛把外層的耐心放大到容得下內層的完整重試，要嘛把內層的重試次數／單次 timeout 收斂到外層的預算之內，兩者必須是同一盤帳上協調過的數字，不能各設各的。

這也是為什麼 retry budget 重要。Google 的 SRE 實務裡有一個漂亮的量化：在每個下游 client 上限制「重試請求佔總請求的比例不超過 10%」。沒有這個限制時，一個系統性的故障會讓重試把請求量放大到約 **3 倍**（每個請求失敗就重試到上限）；加上 10% 的 retry budget 後，一般情況下的請求量增長被壓到約 **1.1 倍**。差別是「故障時流量暴增三倍、把搖搖欲墜的系統徹底壓垮」與「故障時流量只多一成、系統還撐得住」。逾時設定若不和重試一起放進同一份預算裡算，你就是在親手埋一個重試風暴的雷（這個正回饋放大如何把系統推進無法自癒的亞穩態，見〈retry storm 與 metastable failure〉）。

## 放棄等待，不等於對方停手

最後一個邊界，是逾時機制裡最常被漏掉的半邊。client 端的 timeout 觸發時，它做的事情只有一件：**「我不等了。」** 它把自己這條執行緒、這條連線釋放掉，回一個錯誤給上游。但它預設**不會**告訴對端「我已經不要這個結果了」——對端很可能還在埋頭把那個已經沒人要的計算跑完，繼續燒它的 CPU、佔它的記憶體、壓它本來就吃緊的資源。

在正常情況這只是一點浪費；在過載的瞬間，這點浪費是致命的。想像下游已經慢到讓上游紛紛逾時——此刻下游手上堆積的，全是「上游已經放棄、但下游還在算」的殭屍請求。它把本就稀缺的算力，全砸在沒有人會收的結果上，於是它更慢，於是更多上游逾時，於是更多殭屍堆積。這是一個會自我強化的死亡螺旋，而打斷它的關鍵，是讓「我不等了」能變成一個傳得到對端的訊號。

這就是**取消（cancellation）** 要補的洞。逾時除了「不再等」，最好還能向下游送一個取消訊號，讓下游也停手：gRPC 有原生的 cancellation（client 取消會透過 HTTP/2 的 `RST_STREAM` 傳到 server 端）、HTTP 可以靠斷開連線讓 server 偵測到 client 走了、Go 裡則是把 `context.Context` 一路傳下去、上游 `cancel()` 一呼叫，所有監聽 `ctx.Done()` 的下游一起收手。把 deadline 用 context 串起來時，這件事是順帶完成的——deadline 一到，context 自動進入 done 狀態，整條鏈上聽著它的人同時停手。沒有取消傳遞的逾時，只擋住了上游的資源洩漏，下游那半邊的洩漏仍然敞著。

## 為何是這個形狀

退一步看，這一整套機制——逾時、deadline 傳遞、budget 拆分、取消——其實全是從同一個無法迴避的事實裡長出來的：**在一個延遲沒有上界的網路上，等待若沒有界限，就會把「別人的慢」變成「自己的死」。**

正因為你分不清沉默是慢還是死，所以你只能武斷地劃一條線，過了就放棄——這是 timeout。正因為一個請求會轉很多手、每手各自計時會讓總時間失控，所以你需要一條被所有層共享、且能抵消時鐘偏移的絕對死線——這是 deadline 傳遞。正因為一個端點要把有限的時間分給多個下游，所以你得把承諾拆成預算、還要留白——這是 budget。正因為「我放棄」和「對方停手」是兩件事，而過載時後者才是真正的止血點，所以你需要把放棄變成一個傳得到對端的訊號——這是 cancellation。

每一塊都不是憑空的工程慣例，而是「把無界的等待變成有界」這個目標，被各種真實的失敗——資源凍結、延遲累加、時鐘偏移、重試風暴、殭屍計算——逐一逼出來的、唯一站得住的形狀。下次你在一個 client 上猶豫該不該設 timeout、設多少，記住那個問題從來不是「對方多久會回」，而是「當對方不回時，我要不要把整個系統的命，押在它身上」。
