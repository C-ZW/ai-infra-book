# 重試、退避與 jitter

凌晨三點，一個下游服務抖了一下——也許是一次 GC 暫停、一次主從切換、一次短暫的網路丟包。它在大約半秒內恢復了。但在那半秒裡，有一千個 client 同時對它發了請求、同時收到逾時。每個 client 都做了那件「最合理」的事：再試一次。

如果這一千個 client 都立刻重試，下游在剛要爬起來的那一刻，被同一批請求又砸了一輪。它再次倒下，這次倒得更徹底——因為現在不只有新進來的正常流量，還疊上了一千個失敗請求的重試。重試又逾時，逾時又觸發下一輪重試。原本半秒就該自癒的小抖動，被重試這個「善意」的機制，親手養成了一場持續幾十分鐘、要人介入才能止血的全面停擺。

重試是後端工程裡最樸素的一個動作，樸素到幾乎不像個「機制」——失敗了，再試一次，有什麼好講的？但上面這一幕說明了它的雙面性：重試是對抗暫時性失敗最有效的手段，同時也是把局部小故障放大成系統級災難最常見的引信。這一章要把這件看似一行程式碼就能解決的事拆開，看清楚它在什麼條件下救你、在什麼條件下殺你，以及退避（backoff）與 jitter 這兩個修正，到底各自堵住了哪一個漏洞。

## 重試前要先回答的兩個問題

在談「怎麼重試」之前，得先擋住一個更根本的衝動：不是所有失敗都該重試。盲目重試一個本來就不該重試的請求，比不重試還糟——它不只浪費，還可能製造副作用、或在對端最虛弱時追加無意義的負載。

第一個問題是**這個操作能不能安全地重試**。重試的前提是「再做一次的結果，和只做一次一樣」——也就是冪等（idempotency 的定義與冪等鍵生命週期，見〈重複是常態〉）。讀取天生冪等，重試一個 `GET` 沒有任何代價。但寫入不是。設想一個「扣款」請求：client 送出後，請求其實成功了，只是回應在回程丟了。client 沒收到 200，判定失敗，重試——於是同一筆款被扣了兩次。問題的根源在於一個逾時**無法區分**「請求沒送到」和「請求送到了、處理成功了、但回應沒回來」。從 client 這端看出去，這兩種情況長得一模一樣，但前者重試是對的、後者重試是災難。所以非冪等的寫入要能安全重試，唯一的辦法是讓它**變得冪等**：client 為這筆操作生一個唯一的冪等鍵帶上，server 端認鍵去重，第二次帶同一把鍵進來時直接回上次的結果、不重做。沒有這層保護，「重試」這個詞對寫入而言就是「重複副作用」的同義詞。

第二個問題是**這個錯誤值不值得重試**。失敗有兩種本質不同的長相。一種是暫時性的（transient）：對端剛好在 GC、網路抖一下、限流回了 429、伺服器回了 503——這些「世界的狀態」過一會兒會變，再試一次有機會成功。另一種是持久性的：請求本身就是錯的，少了必填欄位、權限不足、資源不存在——這些對端回 4xx，世界的狀態再怎麼變，你送同樣的錯誤請求還是同樣的錯。重試一個 400，只是用一模一樣的方式重複犯錯，浪費雙方的資源還拖長使用者等待。

所以一條清醒的重試規則，第一步永遠是分類錯誤碼。粗略地說，可重試的是逾時、連線被拒、5xx（伺服器端的暫時故障）、429（被限流，但對端通常會明說等多久）；不可重試的是大部分 4xx（請求本身的問題）。這個分類不是裝飾——它決定了你的重試到底是在幫對端爭取喘息時間，還是在對端的傷口上反覆撒鹽。退避與 jitter 是「確定值得重試」之後才登場的事，它們讓重試變得溫和，但救不了一個本來就不該發出的重試。

## 立刻重試為什麼最毒

假設我們已經確認：操作冪等、錯誤是暫時性的，該重試。最天真的做法是失敗了就馬上再送。看起來無害，卻恰恰是最危險的一種。

關鍵在於：暫時性失敗最常見的成因，是對端**暫時過載或暫時不可用**。而「立刻重試」做的事，是在對端最虛弱、最需要喘息的那一刻，立刻把同一個請求再丟回去。這不是在等它恢復，這是在阻止它恢復。一個過載的服務需要的是負載下降一段時間、把積壓的工作消化掉；立刻重試卻保證了負載**不會**下降——每個失敗的請求都即刻變成一個新的請求。

更糟的是這會形成一個**正回饋迴圈（positive feedback loop）**，而正回饋迴圈是工程裡最該警惕的結構。對端慢 → 請求逾時 → client 重試 → 負載增加 → 對端更慢 → 更多逾時 → 更多重試。每一圈都讓下一圈更嚴重。Google SRE 的書裡有個很具體的描述：在這種迴圈下，請求量會像滾雪球一樣長——第一秒 100 QPS 的重試，下一秒變 200，再下一秒 300，疊在原本就有上萬 QPS 的正常流量之上。

這就引出了一個比「停機」更陰險的失敗形態：**metastable failure（亞穩態故障）**。一般的故障，觸發原因消失後系統就會自己恢復——拔掉引信，炸彈就不響了。但 metastable failure 不是：那個半秒的抖動（觸發原因）早就過去了，下游其實有能力處理正常流量，可是系統被自己的重試流量鎖死在一個「持續失敗」的穩定狀態裡，靠自己永遠爬不出來。你會看到一個詭異的現象：監控顯示底層依賴明明已經健康了，但整個系統還在持續地噴錯，而且你找不到任何「正在發生」的觸發原因——因為觸發原因早就結束了，現在維持故障的，是系統自己對失敗的反應。要打破它，往往得人為地把流量整個切掉、讓重試風暴自然衰減，再慢慢放回來。重試做法的整個演化，本質上都是為了不掉進這個坑。

## 第一個修正：退避

第一個補丁很直覺：別立刻重試，每次失敗後等久一點。這就是退避。最常用的形態是**指數退避（exponential backoff）**——等待時間隨重試次數指數成長：

```
delay = base * 2 ^ attempt
```

`base` 是基礎間隔（比如 100 毫秒），`attempt` 是第幾次重試。於是間隔變成 100ms、200ms、400ms、800ms……每失敗一次就翻倍。為什麼是指數而不是線性（固定每次 +100ms）？因為你對「對端要多久才好」一無所知。如果線性增長，而對端需要的恢復時間比你預期長，你會用密集的重試持續騷擾它很久；指數增長則讓壓力**隨失敗次數快速衰減**——對端越是持續不好，你退得越遠、給它越多空間，這正是你想要的自適應行為。

指數成長會把間隔推到很離譜的數字（第 10 次就是 100ms × 2^10 ≈ 102 秒），所以實務上一定要加一個上限 `cap`，封住單次等待不超過某個值（比如 20 秒）：

```
delay = min(cap, base * 2 ^ attempt)
```

退避堵住了「立刻重試在對端傷口上撒鹽」這個漏洞。對單一 client 而言，它的重試時間軸被拉開了，不再是密集地猛戳。但它有一個致命的盲點，而這個盲點，正是開場那場災難的真正成因。

## 退避堵不住的洞：同步的尖峰

回到開場：一千個 client 在「同一毫秒」因為同一個下游抖動而同時逾時。現在它們全都用指數退避，`base = 1s`。

問題來了——它們失敗的時刻相同，退避的公式也相同，於是它們算出來的下次重試時刻**也完全相同**。第 1 秒，一千個請求同時砸過來；如果這輪又失敗，第 2 秒，一千個再一起來；然後第 4 秒、第 8 秒。退避確實把每個 client 自己的重試拉疏了，但它沒有把**不同 client 之間**的重試錯開。下游看到的不是一條平緩的細流，而是一系列整齊的、千請求齊發的尖峰。對一個正在掙扎著想恢復的服務來說，它面對的不是「平均負載降低了」，而是「每隔幾秒被一千個請求同步轟炸一次」——它在尖峰之間根本沒有足夠連續的喘息來消化積壓。這就是經典的 **thundering herd（驚群）**：退避是同步的，所以群體還是一起行動。

問題的數學本質是：退避只在**時間軸**上做了處理（把單一 client 的重試拉開），卻沒在**機率分布**上做處理（把群體的重試時刻打散）。所有 client 共享同一個確定性的公式，相同的輸入（失敗時刻）就產生相同的輸出（重試時刻）。要打散群體，得引入隨機性。

## 第二個修正：jitter

jitter（抖動）就是那個隨機性。它的思路簡單到近乎狡猾：既然「大家算出同一個時刻」是病根，那就讓每個 client 在自己的退避窗內，**隨機**挑一個時刻重試。

最直接、也是 AWS 公開推薦為預設的做法叫 **full jitter**——把確定性的退避值整個當成上界，在 `0` 到那個上界之間均勻隨機取樣：

```
sleep = random(0, min(cap, base * 2 ^ attempt))
```

差別有多大？同樣那一千個 client、`base = 1s`、第一次重試。無 jitter 時，一千個請求精準地擠在第 1 秒這一個點上。換成 full jitter 後，每個 client 的等待時間在 0 到 1 秒之間均勻分布——平均約五百個 client 落在前半秒、其餘攤到後半秒，而且在這一秒內任何一個瞬間，期望只有一兩個請求落下。同樣的重試總量（一千次），下游看到的從「某一刻一千個的尖峰」變成了「每毫秒一兩個的細流」。重試的**總工作量沒變**，但對端的存活與否，完全取決於這些工作是擠成一團砸下來、還是攤平成連續可消化的流。AWS 的模擬把這件事量化過：在多個 client 競爭的情境下，加上 full jitter 後，達成全部成功所需的**總呼叫次數比無 jitter 的指數退避少了一半以上**，完成時間也更短——也就是說 jitter 不只對伺服器更友善，連 client 自己也更快做完，是雙贏，不是取捨。

這裡有個直覺上的反轉值得停一下：加了隨機性，**整體**反而更有秩序了。單看一個 client，它的重試時刻變得不可預測；但正是這種個體層面的不可預測，換來了群體層面的平滑。把它想成過馬路——一群人若都聽同一個號誌、同時起步，人行道某一瞬間會被擠爆；若每個人各自隨機地起步，人流反而均勻。jitter 用「打散同步」這一招，把退避漏掉的那半個問題補上了。

## jitter 不只一種，差在收斂速度

full jitter 把區間從 0 起算，意味著偶爾會抽到一個非常短的 sleep（接近 0）。在某些情境下你不希望重試間隔太短——抽到接近 0 等於幾乎立刻重試，又有點回到「立刻重試」的味道。於是有了 **equal jitter**：保留一半的確定性退避當下限，另一半才隨機：

```
sleep = min(cap, base * 2 ^ attempt) / 2 + random(0, min(cap, base * 2 ^ attempt) / 2)
```

這樣既攤平了尖峰，又保證每次至少等了退避值的一半，不會抽到太短。代價是分散度比 full jitter 略低——因為一半被釘死了，隨機能打散的只剩另一半。

還有一種 **decorrelated jitter**，它的上界不跟著「第幾次重試」走，而是跟著「上一次實際睡了多久」走：

```
sleep = min(cap, random(base, prev_sleep * 3))
```

`prev_sleep` 是上一次的睡眠時間。它讓退避有機會更快地往上爬（上界能擴張到上次的三倍），在對端持續壞掉時更快地退到安全距離，又因為每次都重新隨機而保持分散。AWS 的模擬裡，full jitter 做的總工作量最少（對伺服器最友善），decorrelated jitter 完成得稍快一點但做的工作稍多——這就是兩者的取捨：你更在乎峰值伺服器負載，選 full jitter；你更在乎自己這端的延遲、想更快爬出谷底，選 decorrelated。equal jitter 在兩個軸上都略遜，多半是個折衷選項。對絕大多數服務，full jitter 是最簡單、行為最好預測的預設，沒有特殊理由就用它。

順帶一提，jitter 不必是非黑即白的「整段隨機」。gRPC 的內建重試策略走的是另一種輕量版本：在每次計算出的指數退避值上，加一個 **±20% 的隨機抖動**。這已經足夠錯開大量 client 的同步重試了。重點不在哪個公式最神，而在於**有沒有引入足以打散同步的隨機性**——一點點就比完全沒有好非常多。

## 真正會殺死系統的不是單層重試，是層層相乘

到這裡，單一 client 的重試已經被馴服得相當好了：分類錯誤、指數退避、加 jitter。但有一個放大效應，藏在系統的**結構**裡，而不是任何單一 client 的邏輯裡，它比 thundering herd 更隱蔽、也更致命。

問題出在重試的**層層疊加**。設想一個請求的呼叫鏈有五層深：使用者打到 API gateway，gateway 呼叫 service A，A 呼叫 service B，B 呼叫 service C，C 最後查資料庫。每一層的工程師都很「負責任」地加了重試：失敗就重試 3 次（也就是每層最多嘗試 3 次）。每一層單獨看都合情合理。

但把它們乘起來看。最底層的資料庫慢了一下，C 對它的查詢失敗。C 重試 3 次。這 3 次對 C 的上游 B 而言，可能只是 C 的一次調用——但如果 B 也在 C 失敗後重試 3 次，那 B 這一層就會驅動 C 跑 3 次，每次 C 又對資料庫打 3 次，已經是 9 次了。再往上，A 重試 3 次，gateway 重試 3 次。倍率是**相乘**的：

```
3 (gateway) × 3 (A) × 3 (B) × 3 (C) × 3 (db query) = 3^5 = 243
```

一個使用者的請求，在最底層的資料庫上放大成了 **243 次**查詢。資料庫本來只是慢了一下、需要一點喘息，現在卻要承受兩百多倍的負載——它不可能恢復，只會被這個放大效應徹底壓垮。退避和 jitter 在這裡幫不上忙：它們讓每一層的重試變得溫和，但溫和的 243 倍還是 243 倍。

這個放大效應的根因是**多層各自為政地重試**，而解法是一條架構級的紀律：**重試應該集中在某一層，而不是每一層都做**。最常見的選擇是只在最接近使用者的邊緣（如 gateway 或 client SDK）重試，底下的內部服務一律不重試、失敗就快速把錯誤往上拋（fail-fast）。這樣總放大倍率就回到那一層的次數本身，而不是各層相乘。另一種思路是讓重試的決定權跟著請求走：請求 metadata 裡帶一個「已重試次數」的計數，到上限後任何一層都不准再重試——Google SRE 描述過這種 per-request 的重試預算，計數從 0 起、每次重試遞增、到 2 就鎖死，從根上掐斷相乘。無論哪種，核心都是同一句話：在系統設計時就明確規定「誰負責重試」，而不是讓每個元件各自貼心地加一層。

## 把重試本身關進預算裡

退避和 jitter 處理「重試的時機」，重試集中化處理「重試的位置」，還剩最後一個維度：**重試的總量**。即使在單一層、即使有退避有 jitter，如果失敗是大面積且持續的，重試流量本身就足以拖垮對端。這時需要一個全局的剎車。

最簡單的剎車是兩個硬上限。一是**最大重試次數**——重試 N 次還不成就放棄，別無限重試下去。二是**總時間預算（deadline）**，這個比次數上限更重要也更常被忽略：你設「最多重 5 次」，但 5 次指數退避累積起來可能要好幾十秒，早就超過了上游能等的時間。所以真正該管的是一個牆鐘總預算——「這個操作從頭到尾最多花 3 秒，超過就放棄」，重試只能在這個預算內進行（這個總預算怎麼跨呼叫鏈傳遞、怎麼隨層遞減，是 timeout 與 deadline 自己的主題，見〈timeout、deadline 與 budget〉）。沒有總預算的重試，等於把使用者的等待時間交給「對端到底壞多久」去決定。

但這兩個都是 per-request 的上限，擋不住「很多請求同時都在重試」的洪流。針對這個，Google SRE 和 AWS 不約而同地用了一個更聰明的機制：**retry budget（重試預算）**，把整個 client 的重試量限制成成功流量的一個小比例。具體做法常是一個 token bucket——每次請求**成功**就往桶裡滴一小份 token（比如 0.1 個），每次**重試**就從桶裡扣一個（1 個），桶空了就不准再重試。

這個機制的妙處在於它**自適應**：當系統健康、絕大多數請求都成功時，桶被成功流量不斷補滿，零星的失敗有充足的預算去重試——這正是你要的，少數抖動該重試。但當下游大面積故障、成功率暴跌時，桶幾乎得不到補充、迅速見底，於是重試被自動掐斷，client 退化成「失敗就直接放棄」。這恰恰是大故障時你要的行為——對端都倒了，重試只會火上澆油，這時候停止重試反而是慈悲。Google SRE 給的經驗值是把每個 client 的重試比例壓在 **10%** 以內（重試請求佔總請求不超過一成）。一個有趣的數字感：他們的分析顯示，沒有預算時請求量可能放大到接近 3 倍，而套上 10% 的重試比例上限後，一般情況的放大被壓到只剩約 1.1 倍。retry budget 把「重試」從一個無界的、會自我放大的行為，變成了一個有界的、會在最危險時自動關閉的行為。gRPC 的 retry throttling 走的也是同一條 token bucket 路：成功補 token、失敗扣 token，桶低於門檻就節流掉重試。

## 那些藏在邊角的陷阱

主線講完了，但重試這個機制真正會在生產環境咬你的，往往是幾個不那麼顯眼的邊角。每一個都值得在腦中重放一遍。

**對端已經明說了等多久，別用自己的公式蓋過它。** 當對端回 429 或 503，它常常附帶一個 `Retry-After` 標頭（RFC 9110 §10.2.3 定義，值可以是秒數、也可以是一個絕對的 HTTP-date）。這是對端在直接告訴你「我需要 N 秒，N 秒內別來」。這個來自源頭的資訊，比你 client 端任何退避演算法的猜測都準確。如果你的退避算出 2 秒、但對端說 30 秒，你卻照自己的 2 秒回來，等於無視它的求救。正確的做法是：`Retry-After` 在時，以它為準（仍可在它之上疊一點 jitter，避免到點時大家又同步回來）；沒有它時，才退回自己的指數退避。

**冪等鍵在重試之間必須保持同一把。** 前面說非冪等寫入要靠冪等鍵兜住重試，但有個隱蔽的失誤：如果你在「每次重試時」重新生成冪等鍵，那這層保護就形同虛設——server 看到的是兩把不同的鍵，當成兩筆獨立操作，照樣重複執行。冪等鍵必須在**第一次嘗試前**就生成、並在這個邏輯操作的所有重試裡保持不變。鍵的生命週期要綁定「這筆業務操作」，而不是「這次網路嘗試」。

**重試讓亂序與重複變成常態，下游要扛得住。** 一旦引入重試，「同一筆請求被處理兩次」和「較晚發出的請求反而較早到達」就從異常變成了日常。前者要靠冪等與去重兜住；後者在乎順序的場景（如狀態機式的事件流）會出問題（訊息順序怎麼保，見〈訊息順序〉）。設計時要假設「我的每一個請求都可能被對端收到不只一次、且不保證順序」，而不是把重複和亂序當成罕見的意外。

**逾時設太短，會把一個正在成功的請求殺掉再重試。** 重試依賴逾時來判定失敗，但如果逾時閾值設得比正常的處理時間還短，你會在請求其實**正在順利處理**的途中就判它死刑、然後重試——對端被迫同時處理原始請求和重試請求，負載憑空翻倍，而且這完全是你自己造成的。逾時要設在正常延遲分布的尾巴之外（比如 p99 之上留足餘裕），否則重試機制會把健康的系統當成故障來「搶救」，越救越糟。

**不可重試的錯誤被當成可重試，是另一種浪費。** 把錯誤碼分類做反了，比如把一個結構性的 400 當成暫時故障去重試，就是用指數退避這套溫和的機制，去溫和地、反覆地撞同一面牆。退避和 jitter 讓這個浪費變得不顯眼（因為間隔被拉開了、不會造成尖峰），但它依然是浪費——耗著資源、拖長使用者等待，最後還是同樣的失敗。分類錯誤是重試的第一道閘，這道閘漏了，後面再漂亮的退避都是在精緻地做無用功。

## 為什麼是這個形狀

退一步看，重試這套機制的整個形狀——退避、jitter、集中化、預算——都是從一個樸素事實裡逼出來的：**重試是一個正回饋的動作**。失敗會觸發更多請求，而更多請求在對端虛弱時會製造更多失敗。任何正回饋迴圈，只要沒有阻尼，最終都會失控。

於是每一個修正，都對應著給這個迴圈裝上一道阻尼。退避是時間上的阻尼——別在傷口上立刻撒鹽，給對端隨失敗加深而加長的喘息。jitter 是統計上的阻尼——別讓群體同步行動，把尖峰攤成細流。重試集中化是結構上的阻尼——別讓倍率在呼叫鏈裡相乘，把它鎖在一層。retry budget 是總量上的阻尼——在系統大面積故障、迴圈最危險時，自動把重試整個關掉。

把這四道阻尼合起來，你會發現它們指向同一個設計哲學：**一個好的重試系統，最重要的不是它在小故障時多努力地重試，而是它在大故障時多果斷地放棄。** 小抖動時，溫和地、錯開地重幾次，幫對端撐過去；大故障時，迅速地縮手、讓重試流量自然衰減，給對端留出靠自己爬起來的空間。最危險的重試系統，恰恰是那個「永遠不放棄」的——它會在對端最需要喘息時，用無窮的善意把它活活壓死，再把這份善意沿著呼叫鏈相乘成一場誰也救不了的雪崩。重試的智慧，一半在於知道什麼時候該再試一次，另一半——也是更難的一半——在於知道什麼時候該停下來。
