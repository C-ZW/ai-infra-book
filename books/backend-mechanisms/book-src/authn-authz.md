# 認證與授權：AuthN、AuthZ 與 OIDC 的位置

一個再普通不過的 bug 報告：使用者 A 反映，他在訂單頁面把網址裡的數字從 `/orders/4821` 改成 `/orders/4822`，就看到了別人的訂單——金額、收件地址、電話，全都在。後端沒報任何錯，回 200，資料乾乾淨淨地吐了出來。

值得停下來想：這個系統**沒有壞**。它的登入是好的——A 確實是用自己的帳號密碼登進來的，session 是有效的，JWT 簽章驗得過，`exp` 還沒到。每一道「你是誰」的檢查都通過了。漏掉的是另一件事：沒有人問過「`4822` 這張單，到底是不是 A 的」。系統把「你已經登入了」默默當成了「你有權看這張單」，而這兩句話之間，隔著一整個被跳過的判斷。

這個跳過，是後端最常見、也最昂貴的一類漏洞。OWASP 把它叫 Broken Access Control，2021 年它從第五名躍升到 Top 10 第一名，取代了盤踞榜首多年的注入攻擊；那次調查裡，它的測試覆蓋率（94% 的應用都被檢測過是否有此類缺陷）與累計出現次數（逾 31.8 萬次）都排全榜第一，正是它登頂的原因；而到了最新的 OWASP Top 10:2025（2026-06），它依然穩坐第一名——這個位置一坐就是兩個版本，本身就說明了問題之頑固。而這一整類問題的根，幾乎都可以收斂成同一句話：**有人把「認證」和「授權」當成了同一件事。** 這一章要做的，就是把這兩件事徹底拆開——拆到你能向另一個工程師講清楚它們為什麼正交、在哪裡會被混為一談、以及一個叫 OIDC 的東西到底補的是這兩塊裡的哪一塊。

## 兩個正交的問題

先把名詞釘死，因為混淆往往從名詞開始。

**認證（authentication，常縮寫 AuthN）回答的是「你是誰」。** 它驗證一個身分主張為真——這個請求，確實來自它聲稱的那個 principal（人、服務、或裝置），而不是別人冒充的。認證的輸出，是一個**已被驗證的身分**：這是 user 4821，這是 `payment-service`，這是某張裝置憑證。

**授權（authorization，常縮寫 AuthZ）回答的是「你能做什麼」。** 在已知身分（或已知是匿名）的前提下，判斷某個具體操作對某個具體資源，是不是被允許。授權的輸出，是一個二元判定：allow 或 deny。

這兩個問題的關係，是工程上少見的「順序固定、但職責完全獨立」。順序固定，是因為授權通常需要先知道身分才能判斷——你得先知道請求來自誰，才能問「這個誰能不能做這件事」。但職責獨立，是因為**知道你是誰，完全沒告訴我你能做什麼**。這兩句話之間沒有任何邏輯蘊含關係，它們只是恰好常常被擺在一起執行。

正交到什麼程度？看兩個極端就懂了。一個系統可以**只認證、不授權**：登入了就什麼都能做，沒有任何權限分級（很多內部小工具就是這樣，一個 `is_logged_in` 走天下）。反過來，一個系統也可以**只授權、不認證**：對匿名請求做 rate limiting、用 IP allowlist 擋掉非辦公室網段、給未登入訪客開放唯讀的公開頁面——這些都是不問「你是誰」、純粹基於非身分屬性做的存取決策。兩件事可以各自單獨存在，這正是它們正交的鐵證。

開場那個 bug，本質上就是系統**做了認證、跳過了授權**。它驗了 A 是 A，卻沒驗「這張單屬不屬於 A」。而把這兩件事綁死，是後端最頑固的一種耦合來源——它甚至常常一開始看起來像個優點。

## 綁死它們，會在哪裡反噬你

考慮一個非常自然、非常常見的設計：登入流程裡，順手把「這個人是不是管理員」也一起判了。登入成功，回一個帶 `is_admin: true` 的 session，之後前端看到這個 flag 就把管理選單顯示出來。乾淨、直覺、少寫一次查詢。

問題藏在「順手」這兩個字裡。你把一個**授權判斷**（是不是管理員）塞進了**認證流程**（登入），於是這兩件事在程式碼層面焊在了一起。半年後，需求來了：要加一種「唯讀管理員」——能看後台所有資料，但不能改任何東西。這個需求在乾淨的模型裡只是多一條授權規則，但在你這個焊死的系統裡，它逼你回去動登入碼，因為「是不是管理員」這個二元 flag 根本表達不了「是管理員、但只能讀」這種細分。你得拆掉 `is_admin`、重新設計 session 結構、重新測整條登入路徑——只因為當初省了一次查詢。

這還是好的情況，至少它只是難改。更危險的反噬，是**安全層面的**。當「進得來」和「能做事」在心智模型裡是同一件事，工程師會不自覺地在某個進入點做完登入檢查後，就認為「後面都是自己人了」。於是內部 API 不再做授權、批次任務直接信任呼叫方、管理端點只靠「你得先登入」來保護。這種「城牆外是敵人、城牆內都是朋友」的假設，在現代架構下基本上是錯的——一旦有任何一個合法使用者起了壞心（或任何一個合法帳號被盜），他已經在城牆「內」了，而城牆內什麼授權檢查都沒有。

開場那個改網址的攻擊，就是這個假設的直接後果。攻擊者不需要突破認證——他用的是自己**真實、合法**的帳號。他只是利用了「系統以為登入＝有權」這個漏洞，去碰了不屬於他的資源。這類漏洞有個名字叫 **IDOR（Insecure Direct Object Reference，不安全的直接物件參照）**：當資源用使用者可控的識別碼（`order_id`、`user_id`、`file_id`）來引用，而後端沒有驗證「這個已認證的請求者，是否被授權碰這個特定物件」時，把 `4821` 改成 `4822` 就成了一把萬能鑰匙。IDOR 是 Broken Access Control 底下最典型的一種，而 Broken Access Control 之所以能登上 OWASP 榜首，正是因為「認證做了、授權漏了」這個錯誤太容易犯、太難在 code review 裡一眼看出來——畢竟那段程式碼**看起來**有在做檢查，只是檢查的是錯的那一半。

從這裡能逼出一條幾乎沒有例外的原則：**每一個敏感操作，都要獨立做一次授權檢查，而且這次檢查必須包含「這個資源是不是這個請求者的」。** 不能靠「他進得來，所以是自己人」來省略。認證在請求的入口處做一次就夠了（驗明身分）；授權必須在**每一個**碰到受保護資源的地方重做，因為每個資源、每個操作的「能不能」都不一樣。

順帶釘死一個相關的陷阱：**前端的權限控制不是授權。** 把管理按鈕在 UI 上藏起來、把唯讀使用者的編輯框設成 disabled，這些全都只是 UX——它讓介面更友善，但攻擊者根本不看你的介面，他直接對 API 發請求。任何真正的授權判斷，都必須在伺服器端、在請求實際抵達資源的那一刻做。前端的隱藏，價值是「別讓使用者點到他不能用的東西」，零安全價值。把它當成防線，等於沒有防線。

## 一個請求走過兩道關卡

把這兩件事放回一個請求的生命週期裡看，它們的先後與分工就立體了。設想一個請求 `DELETE /orders/4822`，帶著一個 token：

```
incoming request: DELETE /orders/4822   (token attached)

  ── gate 1: authentication (who are you?) ──────────────
     verify token signature / lookup session
     -> identity established: user_id = 4821
     fail -> 401 Unauthorized   (we don't know who you are)

  ── gate 2: authorization (may you do this?) ───────────
     load order 4822
     check: does order 4822 belong to user 4821?  -> NO
     check: does user 4821 have role/permission to delete? 
     fail -> 403 Forbidden      (we know who you are; you may not)

  -> only if BOTH pass: perform the delete
```

這張圖裡藏著一個被很多人忽略、卻把兩道關卡的分工講得最清楚的細節：**HTTP 狀態碼 401 和 403，恰好對應認證和授權兩道關卡。** `401 Unauthorized`（這個名字其實取錯了，它講的是 authentication）意思是「**我不知道你是誰**，請出示有效憑證」；`403 Forbidden` 意思是「**我知道你是誰，但你不能做這件事**」。401 是第一道關卡沒過，403 是第一道過了、第二道沒過。一個 API 如果在該回 403 的時候回 401，往往就洩漏了它把這兩件事搞混了。

這張圖也讓「資源層級授權」這件事變得具體。注意 gate 2 裡那兩道檢查是分開的：一道是「order 4822 是不是屬於 4821」（**所有權／關係**），另一道是「4821 有沒有刪除的權限」（**角色／能力**）。開場的 IDOR，漏的就是第一道——系統很可能有做第二道（檢查角色），但忘了問所有權。這也說明為什麼授權不能只是個全域的「這個角色能不能刪訂單」的查表：很多授權判斷的答案，取決於**這個請求者和這個特定資源之間的關係**，而那是查不到全域角色表裡去的。授權怎麼建模、用角色還是用屬性、所有權這種「資源相關」的條件該怎麼表達，本身是一門需要展開的學問（見〈RBAC 與 ABAC：兩種授權模型〉）——這裡只要記住：授權的輸入裡，常常必須包含「資源」這一維。

## 認證怎麼做：從有狀態到無狀態的張力

把鏡頭拉回認證這一側。「驗明你是誰」聽起來簡單，但它底下有一條貫穿整個現代後端的張力線，值得看清楚——因為授權的諸多困難，往往是認證的設計選擇逼出來的。

最古早、也最直覺的做法是**有狀態的工作階段（session）**：使用者用密碼登入，伺服器在自己這邊建一筆 session 記錄、發一個不透明的 session id（通常存在 cookie 裡），之後每個請求帶這個 id 回來，伺服器拿 id 去查那筆記錄，確認身分。這個模型的好處是**伺服器完全掌控**——它手上有那筆記錄，想讓某個 session 立刻失效，刪掉記錄即可，撤銷是即時的。代價是那筆記錄得存在某個地方：單機時放記憶體就好，但一旦水平擴展到多台機器，每台都得能查到同一筆 session，於是你被迫引入共享的 session store（Redis 之類）或 sticky session，這給擴展添了一層相依。

另一條路是**無狀態的 token**：登入後，伺服器發一個**自包含、帶簽章**的 token（最常見的形式就是 JWT），token 本身就攜帶了身分資訊，並用簽章保證沒被竄改。之後每個請求帶著它，伺服器**本地驗簽**就能確認身分，不必查任何 store。這個模型對水平擴展近乎完美——任何一台機器、甚至邊緣節點，只要有驗簽用的金鑰就能獨立驗證身分，無需共享狀態。token 的結構、用對稱還是非對稱簽章、金鑰怎麼透過 JWKS 散佈，是另一個需要單獨講透的機制（見〈JWT 與簽章：結構、HS256/RS256 與 JWKS〉）。

但無狀態的便利，欠下了一筆精準的債：**伺服器手上不再有「這個身分還算不算數」的開關。** session 模型裡，撤銷就是刪一筆記錄；token 模型裡，token 一旦簽發，在它過期之前都驗得過——即使背後的使用者已經被停權、密碼已經洩漏、權限已經被收回。本地驗簽不查 store，正意味著本地驗簽也看不到「這個人已經被踢了」這件事。這條「無狀態 vs 即時撤銷」的張力，是整個 token 體系最核心的取捨，怎麼用短 TTL 加 refresh、或退回查 denylist 來緩解它，本身是一整套機制（見〈token 的生命週期與撤銷〉）。在這一章，我們只需要記住它的形狀：**認證模型選擇了無狀態，就同時選擇了「身分的撤銷不再即時」這個後果**——而這個後果，會反過來影響你能對授權做出什麼假設（一個帶著舊權限的 token，可能在使用者被降權後還活著一陣子）。

認證還有第三條重要的路：**把它整個委派出去**。你不自己驗密碼，而是把「你是誰」這個問題交給一個專門的身分提供者（identity provider，IdP），自己只負責信任它的判斷。這就是 SSO（單一登入）和「用某某帳號登入」背後的模型，自己不碰、不存使用者密碼，認證的安全責任、MFA、稽核全集中到 IdP 那一處。好處明顯，代價是你**強相依**那個 IdP——IdP 掛了，等於全員登不進來。而這條委派之路，正好把我們帶到那個最常被誤解的東西：OIDC。

## OIDC 落在哪一塊：認證，不是授權

到這裡，這一章最容易被搞混、也最值得講透的一點來了：**OAuth 2.0 和 OIDC，一個是授權、一個是認證，它們補的是上面這兩塊裡完全不同的那一塊。** 把它們搞混，不只是名詞潔癖的問題——它會直接導致一類真實的安全漏洞。

先說 OAuth 2.0（RFC 6749）。它的全名是「OAuth 2.0 **Authorization** Framework」，它解的問題是**委派授權（delegated authorization）**：讓一個第三方 App，在使用者同意下，拿到一個範圍受限的 access token，去**代表使用者**存取某個資源服務上的東西——而**完全不需要拿到使用者的密碼**。經典場景是「授權某個排版工具讀取你雲端硬碟裡的特定資料夾」：你不會把雲端硬碟的密碼給那個工具，你只給它一個只能讀那個資料夾、隨時可撤、會過期的 token。OAuth 的全部價值，就在「讓第三方替你做事、但別給它你的鑰匙」。

關鍵在於——**OAuth 本身不回答「使用者是誰」。** access token 證明的是「持有者被授權去存取某個資源」，它不是、也不打算是一個身分證明。這不是 OAuth 的缺陷，是它的設計範圍：它是授權框架，認證根本不在它的職責裡。RFC 6749 也好、後續的安全 BCP 也好，反覆強調的就是這條界線。

於是危險的誤用就誕生了：很多人想「我都拿到 access token 了，那我拿它去呼叫資源服務的 userinfo 端點，問出使用者是誰，不就完成登入了嗎？」這個推理錯在一個致命的地方——**access token 不保證是發給「你這個 client」的。** 它可能是使用者授權給**另一個** App 的 token，被惡意（或無意）拿到了你這裡。你拿一個不是為你簽發的 token 去推導身分，等於接受了一張不是給你的入場券。這正是 OAuth 安全文獻裡反覆警告的 **confused deputy（混淆代理）** 類攻擊的溫床：token 沒有被綁定到「打算使用它的那一方」，於是一方的 token 被另一方接受，身分就被冒充了。「用 access token 推使用者身分」是把授權憑證當認證憑證用，這個混淆本身就是漏洞。

**OIDC（OpenID Connect Core 1.0）正是為了堵這個洞而生的。** 它不是 OAuth 的競爭者或替代品，而是**疊在 OAuth 2.0 之上的一個身分層（identity layer）**——它把 OAuth 缺的那塊「認證」補了回來。OIDC 的官方定位就是一句話：「a simple identity layer on top of the OAuth 2.0 protocol」。機制上，client 在發起 OAuth 授權請求時，多帶一個 `openid` 的 scope；這時 authorization server（在 OIDC 的語境裡改稱 OpenID Provider，OP）除了照常發 access token，還會**額外**發一個叫 **ID Token** 的東西。

ID Token 和 access token 的分工，是理解 OIDC 的核心，值得並排釘死：

| | access token（OAuth 給的） | ID Token（OIDC 多給的） |
|---|---|---|
| 回答的問題 | 你能做什麼（授權） | 你是誰（認證） |
| 給誰用 | resource server（拿去判要不要放行存取） | client 自己（拿去確認使用者身分） |
| 內容 | 通常是 scope、可能不透明 | 使用者的身分 claim（`sub`、何時/如何完成認證等） |
| 形式 | 可以是不透明字串或 JWT | **必須**是簽章過的 JWT |
| 誰該驗它 | resource server | 收到它的那個 client（relying party） |

一句話記法：**OAuth 給你 access token（去做事的授權憑證）、OIDC 給你 ID Token（你是誰的認證結果）。** 一個對著資源服務說「放他進來」，一個對著你的 App 說「他就是 user 4821」。

ID Token 為什麼非得是**簽章的 JWT**，而且為什麼這個簽章在 OIDC 裡格外要緊？因為它要解的，正是上面 access token 解不了的那個信任問題。OIDC 規格明文要求 ID Token 必須以 JWS 簽章，且 OP **必須**支援 `RS256`（非對稱簽章），預設演算法也是 `RS256`。非對稱簽章在這裡是刻意的：OP 用私鑰簽，任何 client（relying party）用公開的公鑰就能驗——這讓「一處簽發、多方驗證」成立，而 client 永遠碰不到能偽造身分斷言的私鑰（為什麼非對稱在這種「一處簽、多方驗」的情境下是決定性的，見〈JWT 與簽章〉）。

但光是簽章對了還不夠，這是 OIDC 最容易被實作者漏掉的一層。收到 ID Token 的 client **必須**逐一檢查幾個 claim，否則簽章驗過了也可能被騙：

- **`iss`（issuer）**：簽發者是不是你信任的那個 OP？簽章只證明「token 沒被改」，不證明「是你信的人簽的」——你得明確比對 issuer。
- **`aud`（audience）**：這張 token 的受眾，**必須**包含你這個 client 的 `client_id`。這一條，正是堵死前面那個 confused deputy 漏洞的關鍵——它確保「這張身分斷言是專門簽給我的」，而不是簽給別人、被輾轉送到我這裡的。OAuth 的 access token 不強制這種受眾綁定，ID Token 強制，差別就在這裡。
- **`nonce`**（前三條 `iss`/`aud`/`exp` 是規格的無條件 MUST，這條則是強烈建議的防重放實務——且一旦 ID Token 帶了 `nonce`，client 就**必須**驗它）：發起登入時，client 生成一個一次性隨機值，塞進授權請求；OP 把它原樣放進 ID Token。client 收回來時比對：這張 ID Token 的 `nonce`，是不是我這次登入發出去的那個？這道檢查防的是**重放攻擊**——攔截者就算錄下一張舊的、合法的 ID Token 重送，它的 `nonce` 也對不上當前這次登入，會被拒。
- **`exp`**：過期了沒。

把這幾條串起來看，OIDC 的整個設計就有了內在的一致性：它之所以要**獨立**發一個 ID Token、要它**必須簽章**、要 client **必須驗 `aud` 和 `nonce`**，全都是為了回答 OAuth 的 access token 回答不了、也不該回答的那個問題——「眼前這個請求，背後到底是不是它聲稱的那個人，而且這個答案是不是專門、且新鮮地，講給我聽的」。少驗任何一條，這個保證就漏一個洞。

## 為什麼是這個形狀

退一步看整章。認證和授權之所以被分成兩件事，不是術語潔癖，而是因為「你是誰」和「你能做什麼」在資訊上根本不蘊含彼此——一個系統可以知道你是誰卻不讓你做任何事，也可以不知道你是誰卻給你做一些事。把它們綁在一起，省下的是一次查詢，欠下的是「進得來就是自己人」這個會在某個 IDOR、某個越權操作上連本帶利反噬的假設。所以正確的形狀是：**認證在入口驗明身分一次，授權在每個資源、每個操作前獨立判斷一次，且判斷裡必須容得下「這個資源屬不屬於這個人」這一維。**

而 OAuth 和 OIDC 的分工，是這條「認證／授權正交」原則在跨組織、委派場景下的一次精準落地。OAuth 把「授權」這塊做到極致——讓你能在不交出密碼的前提下，把有限的權力委派出去；它**刻意**不碰認證，因為那是另一個問題。OIDC 看見了「但我有時候就是想知道使用者是誰」這個真實需求，於是不去污染 OAuth 的授權語意，而是疊一層上去、用一個受眾綁定、簽章、防重放的 ID Token，把認證乾乾淨淨地補回來。兩者各管一塊、互不越界——這個分層，本身就是「認證歸認證、授權歸授權」這條原則最漂亮的一次體現。

下次你在 code review 裡看到一段「使用者已登入，所以放行」的邏輯，你會本能地多問一句：登入只證明了他是誰，那「他能不能碰**這一個**資源」，是在哪裡、被誰、用什麼判斷的？這個多問的習慣，就是這一章想留下的東西。
