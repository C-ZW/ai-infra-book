# 物件儲存：S3 與大 payload 的交接

設想一條再尋常不過的管線。使用者上傳一張原圖，前端服務收下、丟一則訊息進佇列，後端的工作者把圖取出來、裁切、轉檔、產縮圖。第一版你寫得很直覺：訊息裡就帶著那張圖的 bytes。本機測試一路綠燈，3 MB 的圖跑得飛快。

上線那天它在第一個真實上傳就炸了。佇列回給你一個 400：訊息太大。你查了一下 Amazon SQS 的上限——單則訊息最大 1 MiB（2025-08 從 256 KiB 上調，2026-06），而使用者那張單眼相機的原圖是 8 MB。於是你做了第二版「修補」：把圖切成 8 片、每片塞一則訊息、消費端再拼回來。這版能跑，但它把一個本來簡單的問題變成了一個分散式拼圖——8 片要保證都到、保證順序、有一片重送了要去重、有一片永遠不來要怎麼辦。你在訊息系統裡，徒手重新發明了一套檔案分塊傳輸協定，而且做得比任何現成的都爛。

問題的根不在 SQS 的上限定得太小，而在**你把大 bytes 跟「這件事發生了」這個事實，硬塞進了同一個通道**。訊息系統、資料庫、RPC——這些通道被設計來搬「小而頻繁、需要排序與交付保證」的東西。一筆訊息的價值在它攜帶的**意義**（「ID 為 4271 的圖該被處理了」），不在它攜帶的**位元組**。把幾 MB 的 bytes 灌進去，你是在用一個為毫秒級協調而生的系統，去做它最不擅長的事：搬大塊、不常變、只需要被原樣取回的資料。

這一章談的，就是這些大塊位元組該去哪，以及把它們搬到那裡之後，交接處會長出哪些非顯然的裂縫。

## 大 payload 想要的，和訊息系統能給的，根本是兩回事

退到第一原理。一塊資料對儲存系統的要求，可以拆成幾個彼此獨立、甚至互相拉扯的維度：它有多大、改多頻繁、要被讀寫的延遲多低、要多耐久、要不要檔案系統那種「seek 到某個位置改幾個 byte」的語意。一張上傳的原圖在這幾個維度上的畫像很清楚：**大**（MB 到 GB）、**幾乎不改**（傳上來就定了，後續只會被讀）、**延遲不敏感**（晚個幾十毫秒拿到沒人在意）、**要極耐久**（使用者的資料丟了是事故）、**不需要部分更新**（沒有人會去改原圖的第 100 個 byte）。

把這幅畫像跟訊息系統、跟區塊磁碟、跟記憶體快取的長處對一對，會發現它落在一塊很特別的空地上——沒有任何一個「快」的儲存層是為它優化的。訊息系統優化的是排序與交付，不是容量；記憶體快取（見〈Redis 的內部〉）優化的是微秒級延遲，貴且易失，拿來放 GB 級的圖是奢侈又危險；區塊磁碟優化的是低延遲隨機讀寫與檔案系統語意，但它綁在單一機器、單一可用區、容量有上限。

物件儲存就是為這塊空地長出來的形狀。它的設計取捨幾乎是上述每一個「快」系統的鏡像：**用毫秒到數十毫秒的網路延遲，換近乎無限的容量、極低的單位成本、跨可用區的極高耐久。** 它不假裝自己快，它賭你要存的這類資料根本不在乎那幾十毫秒。S3（Amazon Simple Storage Service）是這個形狀最廣為人知的實作，後面就用它當主角，但底下的取捨——不可變物件、HTTP 端點、冗餘換耐久——對任何物件儲存（GCS、Azure Blob、Cloudflare R2、自架的 MinIO）都成立。

## 物件不是檔案：扁平命名空間與「整個重寫」

第一個要從腦中清掉的錯誤模型，是把 S3 想成一個掛在某處的硬碟。它不是。它沒有目錄樹、沒有 inode、沒有「打開檔案、seek、寫幾個 byte、關檔」這套 POSIX 語意。它是一個 **HTTP 端點**：你對它 `PUT` 一個物件、`GET` 一個物件、`DELETE` 一個物件，如此而已。

物件（object）是三件東西綁在一起：一串 bytes、一份 metadata（content-type、自訂標頭、ETag…）、一個 key。bucket 是一個**扁平**的命名空間，裡面就是一堆 key→object 的映射。你寫一個 key 叫 `users/4271/avatar.png`，那個 `/` 不是目錄分隔符——它只是 key 字串裡的普通字元。S3 沒有「`users/` 這個資料夾」這種東西，它只有一個叫 `users/4271/avatar.png` 的 key。控制台 UI 把共同前綴渲染成資料夾的樣子，那是給人看的幻覺，底層沒有任何樹狀結構。

這個「扁平」帶來一個常被忽略的後果：**`LIST` 是分頁掃描，不是查詢。** 你不能對 bucket 說「給我所有 size 大於 10 MB 的物件」，也不能指望 `LIST` 廉價地排序。它就是按 key 的字典序、一頁一千個地翻。把 S3 當資料庫用、靠 `LIST` 來查資料，會在物件數量上去之後變得又慢又貴。

第二個要清掉的模型，是「就地部分更新」。物件是**不可變**的——更準確說，S3 沒有提供「改物件的第 N 個 byte」這種操作。你想改一個 1 GB 物件裡的一個 byte，唯一的辦法是把整個 1 GB 重新 `PUT` 上去覆蓋掉舊的。這聽起來浪費，但它正是物件儲存能做到那麼便宜、那麼耐久的原因之一：物件一旦寫定就不動，系統可以放心地把它切片、編碼、散到多個地方，不必處理「就地修改一小段」這種會破壞冗餘佈局的麻煩操作。如果你的資料需要頻繁就地改幾個 byte，那它根本不該放物件儲存——它該放區塊磁碟或資料庫。

## 冗餘換耐久：11 個 9 是什麼意思，又不是什麼意思

物件儲存最招牌的數字是 S3 設計上的耐久度（durability）：**99.999999999%，11 個 9**（2026-06）。AWS 給的直觀換算是：如果你存 1,000 萬個物件，平均大約**每一萬年**才預期丟掉一個。

這個數字怎麼來的？核心機制是**抹除碼（erasure coding）加跨可用區冗餘**。物件不是簡單地存三份完整副本——那太貴。它被切成資料片，再用 Reed-Solomon 類的編碼算出額外的校驗片，這些片**散在單一 region 內至少 3 個可用區（AZ）**。抹除碼的數學性質是：只要任意「足夠多」的片還在，原始資料就能被重算出來。於是即使一整個 AZ 因為斷電、火災、網路隔離而整批消失，剩下兩個 AZ 的片仍足以還原每一個物件。11 個 9 不是靠運氣，是靠「任何單一故障域的損失都在編碼的容錯範圍內」這個設計算出來的。

但這裡有一個**極容易被混為一談、後果卻很實際**的區別，值得釘死：**耐久度（durability）不等於可用度（availability）。**

- **耐久度**回答的是「資料會不會永久消失」。11 個 9 說的是這件事——你的 bytes 幾乎不可能丟。
- **可用度**回答的是「此刻你能不能取到它」。S3 Standard 的設計可用度是 **99.99%**，而 SLA 的第一階退費門檻是 **99.9%**（2026-06）——AWS 的字面承諾其實是「commercially reasonable efforts」、跌破 99.9% 才開始退 service credit，並非保證至少 99.9% 的硬性 uptime。但用 99.9% 當對照，它仍是個比設計值低得多的數字：99.9% 一年大約對應到**接近九小時**的不可用。

兩者差了八個數量級，因為它們是兩種完全不同的承諾。一個 region 級或 AZ 級的事件期間，你的物件**一個 byte 都沒丟**（耐久度守住了），但你**就是暫時 `GET` 不到它**（可用度破了）。如果你的系統在關鍵路徑上同步依賴 S3——比如每個使用者請求都要先去 S3 撈一塊資料才能回應——那麼 S3 的可用度就是你的可用度上限，而那是 99.9%，不是 11 個 9。把耐久度的數字拿來安慰「服務不會中斷」，是把「資料不會丟」偷換成「資料永遠拿得到」，這兩件事在一次 AZ 事件裡會清楚地分開。

順帶一提，早年 S3 為人詬病的「最終一致性」（`PUT` 完馬上 `GET` 可能讀到舊版或讀不到）已經不再是限制：S3 現在對所有 API 操作提供**強讀後寫一致性（strong read-after-write consistency）**（2026-06）——`PUT` 成功回來之後，立刻 `GET` 必定讀到那個新版本。這移除了過去一個很惱人的坑（寫完得自旋等待一致），但它**只保證單一物件**的讀後寫一致；它沒有、也不打算給你「同時原子地更新兩個物件」這種跨物件保證。這一點稍後會回來咬人。

## 大物件怎麼進去：單次 PUT 的天花板與 multipart 的拼裝

把一個物件放進 S3，最簡單的路是單次 `PUT`。但單次 `PUT` 有個硬上限：**5 GB**（2026-06）。超過這個大小，你**必須**改用 **multipart upload**。

multipart 的機制本身就是一堂「怎麼可靠地搬大塊資料」的課，值得看清楚它每一步在防什麼。流程是這樣：

```
1. CreateMultipartUpload  -> S3 回一個 uploadId（這次上傳的代號）
2. UploadPart × N         -> 把物件切成 N 片、各自 PUT，每片帶 partNumber
                            S3 對每片回一個 ETag
3. CompleteMultipartUpload-> 你把 (partNumber, ETag) 清單交回去
                            S3 按 partNumber 排序、拼成最終物件
```

每一步都在解一個具體的麻煩：

- **切片讓失敗變得便宜。** 一個 50 GB 的單次上傳，在第 49 GB 斷線，整件事重來——這在不穩的網路上幾乎不可能成功。切成片之後，斷的只是那一片，重傳那一片就好。片可以**平行**上傳，吃滿頻寬；也可以斷點續傳。
- **partNumber 解決順序。** 片是亂序、平行傳上去的，但最終物件的位元組順序由 `partNumber` 決定，不由到達順序決定。S3 在 `Complete` 那一步才按編號拼裝。
- **ETag 解決完整性。** 每片傳完，S3 回一個 ETag（通常是該片內容的 MD5）。`Complete` 時你把 `(partNumber, ETag)` 一起交回去，等於對 S3 說「我這片的內容指紋是這個」。如果中途某片被截斷或損壞，指紋對不上，`Complete` 會失敗——而不是默默拼出一個壞物件。

片的大小有規矩：**5 MiB 到 5 GiB**，最多 **10,000 片**，而且——這是個容易踩的細節——**只有最後一片能小於 5 MiB**。這個規矩的檢查時機很陰險：如果你天真地用固定 4 MiB 切片，每一片的 `UploadPart` 都會**回 200 成功**，看起來一切順利；直到最後 `CompleteMultipartUpload` 那一刻，S3 才一次性回 `EntityTooSmall`、整筆上傳作廢。錯誤不在你以為的地方爆，而是憋到拼裝那步才爆。

這裡藏著一個非顯然、而且最近才變得更微妙的邊界——但要把方向搞對。S3 在 2025-12 把**單一物件的最大尺寸從 5 TB 上調到 50 TB**（2026-06）。你若去翻 multipart 的官方限制表，會看到「multipart 拼裝出的最大物件」寫的是 **48.8 TiB**——這正是 `10,000 片 × 5 GiB/片` 的上限。容易被誤讀成「multipart 拼不到 50 TB」，但換算一下就反過來了：48.8 TiB 換成十進位是 **≈ 53.7 TB**，比 50 TB 的物件上限**還高**；而 50 TB 換成二進位是 45.47 TiB，用標準的 5 GiB 分片只需 `⌈50 TB ÷ 5 GiB⌉ ≈ 9,314 片`，**穩穩落在 1 萬片之內**。所以傳統 multipart 配標準分片就足以拼滿 50 TB，不需要任何「更大的片」或別的上傳路徑（5 GiB 本來就是片大小的硬上限，沒得再大）。真正的非顯然點不在這裡，而在另一條線上：**「物件最大 50 TB」和「單次 `PUT` 最大 5 GB」是兩個不同的上限**——前者是物件的儲存天花板，後者是單一請求能塞的量，任何超過 5 GB 的物件都**必須**走 multipart。引用上限時把這兩個分清楚，否則容量規劃會在邊緣處對不上。

還有一個多數人不知道、但出帳單時會發現的坑：**沒有 `Complete` 也沒有中止的 multipart 上傳，那些已經傳上去的片會一直留著、一直計費。** 上傳到一半的程式崩了、沒有走 `CompleteMultipartUpload` 也沒有 `AbortMultipartUpload`，那些孤兒片不屬於任何「完成的物件」，`LIST` 物件時看不到它們，但它們佔著儲存、算著錢。標準做法是設一條 lifecycle 規則自動清掉「N 天未完成的 multipart 上傳」——這是物件儲存版本的記憶體洩漏，只是洩的是錢。

## claim-check：訊息只帶鑰匙，不帶貨

回到開場那條卡住的圖片管線。正解叫 **claim-check（取物牌）模式**：大 payload 放 S3，訊息只帶一個指向它的 key。消費端拿到訊息、看到 key、自己去 S3 把貨取回來。就像寄物櫃——你手上那張小牌子不是大衣本身，是「去 14 號櫃取大衣」的憑證。

把這個換算成數字，省下的東西大到值得手算一遍。假設這條管線每天處理 **100 萬則**圖片任務，每張原圖平均 **3 MB**。

把圖塞進訊息（先假設用一個支援大訊息的系統，因為 SQS 的 1 MiB 上限根本擋住 3 MB）：佇列每天要扛

```
1,000,000 則 × 3 MB/則 = 3,000,000 MB ≈ 3 TB / 日
```

3 TB 的資料每天流過 broker、佔著它的儲存與記憶體、在每次投遞與重試時被搬來搬去。broker 不是為當大檔倉庫設計的，這個量會把它的成本與壓力推到很難看的地方。

改成 claim-check：原圖 `PUT` 進 S3（那 3 TB/日 落在按 GB 計、單價極低的物件儲存上，是它最擅長的活），訊息只帶一個 S3 key 加一點 metadata，撐死 **100 bytes**。佇列每天扛的變成

```
1,000,000 則 × 100 bytes/則 = 100,000,000 bytes ≈ 100 MB / 日
```

佇列流量從 3 TB/日 砍到 100 MB/日，**降了約三萬倍**。訊息系統回去做它擅長的事——搬小而有意義的協調訊息；大 bytes 去物件儲存待著。

當然，沒有免費的午餐，claim-check 也欠下兩筆債：

- **多一次往返。** 消費端每則訊息多一次 S3 `GET`（約數十毫秒）才能拿到真正的 payload。對一條批次處理的管線這完全不痛；但如果這是個延遲敏感的同步路徑，這次額外往返要算進預算。
- **生命週期的責任分裂了。** 訊息和它指向的物件現在是**兩份分開的狀態**，而且——關鍵——它們之間沒有跨系統的原子性。任務失敗、訊息進了死信佇列（見〈死信與毒訊息：DLQ〉），那顆 S3 物件不會自己消失。沒有一條 lifecycle 規則去回收這些孤兒物件，它們會無聲地累積成一筆越滾越大的儲存帳單。前面 multipart 孤兒片的坑在這裡換了個樣子再出現：**只要「指標」和「資料」分屬兩個系統，清理就永遠是你自己的責任，沒有人會替你做。**

這第二筆債其實是前面埋的伏筆兌現了：S3 給你單一物件的強一致，但**不給你跨物件、更不給你跨系統的原子性**。「寫了訊息、也寫了 S3 物件」這兩件事無法被包進一個交易裡同時成立或同時失敗。一邊成功、另一邊失敗的窗口永遠存在，而彌合這個窗口——確保最終兩邊一致、孤兒被回收——是應用層自己的工作。

## 把上傳的活外包出去：pre-signed URL 與一個會咬人的過期語意

claim-check 還有一個更進階、也更常被用錯的變形：**讓客戶端直接把大檔傳進 S3，後端連 bytes 都不碰。**

天真的做法是讓使用者把檔案上傳到你的 API 伺服器，伺服器再轉手 `PUT` 到 S3。但這讓你的伺服器變成一個純粹的位元組搬運工——一個 500 MB 的影片要先全部流過你的服務、佔著它的頻寬與記憶體，再吐到 S3。伺服器在這裡沒有創造任何價值，只是當了個昂貴的水管。

**pre-signed URL（預簽網址）** 把這根水管拆掉。後端用自己的憑證，產生一個帶簽章、有時限、限定操作（比如「只能 `PUT` 到這個 key」）的 URL，把它交給客戶端。客戶端拿著這個 URL **直接** `PUT` 到 S3，bytes 完全不經過你的伺服器。後端的角色縮回它該有的樣子：發放權限、決定 key、之後（透過 S3 事件通知或客戶端回報）知道「傳好了」。這是 claim-check 的鏡像——上傳方向也讓資料走旁路，控制平面（你的服務）只管「誰可以傳、傳到哪」，資料平面（bytes）直接走 S3。

但 pre-signed URL 有一個過期語意，幾乎每個第一次用的人都會誤解，而且它在生產環境裡會以「大檔上傳到一半失敗」的形式咬人。URL 的有效期：用控制台產的，1 分鐘到 12 小時；用 SDK／CLI 產的，最長 **7 天**（2026-06）。問題不在數字，在**過期是在什麼時刻被檢查的**：

**S3 是在 HTTP 請求「開始」的那一刻檢查過期，不是在傳輸的全程。** 這切出兩個方向相反的非顯然後果：

- 一個客戶端在過期**前一秒**開始下載一個大檔，傳到一半過期時間到了——**下載會繼續、會成功**，因為請求在過期前就已開始。
- 但如果連線中途斷了，客戶端想在過期**之後**重連續傳——**這次會失敗**，因為新的請求是在過期之後才發起的。

對一個在弱網下傳幾 GB、會斷線重連好幾次的客戶端來說，這個語意意味著：你給的有效期不只要涵蓋「一次順利傳完的時間」，還要涵蓋「斷了好幾次、來回重試直到傳完的總時長」。把過期設成「估計上傳時間剛剛好」是個陷阱——第一次斷線後的重連就會落在過期之後，整個上傳功虧一簣。還有一個更隱蔽的上限：如果這個 pre-signed URL 是用某個 IAM 角色的**臨時憑證**簽出來的，那麼**無論你把過期設多長，URL 都不可能活得比那組臨時憑證本身更久**——憑證的 session 一過期，URL 就跟著失效，哪怕你寫了 7 天。簽發長效 URL 時，底層憑證的壽命才是真正的天花板。

## 它什麼時候是錯的形狀

物件儲存的取捨一旦看清楚，它的反面用法也就一目了然了。把這些「不該用」的情形擺出來，等於從另一個角度再確認一次它的形狀。

- **當高頻小 KV 用。** 每一個 `PUT`／`GET` 都是一次計費的 API 請求。幾百萬個微小物件，請求費與 `LIST` 成本會遠超你對「儲存很便宜」的直覺。海量小資料該打包成大物件，或乾脆放 KV／資料庫。物件儲存的單位**儲存**成本極低，但它的**每請求**成本不是零。
- **當需要就地部分更新的東西用。** 改幾個 byte 就得重寫整個物件。一份會被頻繁小幅修改的資料放這裡，等於每次改動都複製一遍全文。那是區塊磁碟或資料庫的活。
- **當低延遲熱路徑用。** 數十毫秒的首位元組延遲，對一條一秒要回應上千次的熱路徑是災難。那層要快取（見〈快取策略〉）或記憶體儲存頂著，物件儲存待在後面當冷的、大的、耐久的真相層。
- **當跨物件交易用。** 沒有「同時原子更新兩個物件」。需要多個物件一致變更，那個一致性得你在應用層自己織出來——通常是靠版本化、冪等寫（見〈重複是常態〉）、或一個外部的協調記錄，而不是指望 S3 給你一個 `BEGIN…COMMIT`。

## 為什麼是這個形狀

把整章收束成一句話：**物件儲存是「大塊、不可變、要極耐久、不在乎那幾十毫秒」這幅資料畫像，被推到極致後唯一站得住的形狀。**

它放棄檔案系統語意（沒有目錄、沒有就地更新），換來的是可以放心地把每個物件切片、抹除碼、散到三個可用區——這才有 11 個 9 的耐久。它放棄低延遲（HTTP 級而非記憶體級），換來的是近乎無限的容量與極低的單位成本。它放棄跨物件原子性，換來的是每個物件操作彼此獨立、可以海量平行、不必協調。每一個「它做不到的事」，背後都站著一個「正因為不做這件事，它才能把那件事做到極致」的取捨。

而它與訊息系統、與你的 API 伺服器之間那道 claim-check 的交接線，本質上是同一個取捨的延伸：**讓大 bytes 走它自己的、為大 bytes 優化的旁路，讓協調訊息與控制平面只攜帶意義、不攜帶重量。** 訊息系統搬「這件事發生了」，物件儲存搬「事情牽涉的那一大塊資料」，兩者各司其職、在 key 這個小小的取物牌上交接。下次你看到一條管線的訊息裡只躺著一個 S3 key，你會知道那不是偷懶——那是有人想清楚了，把重量卸在了該卸的地方，只讓鑰匙在快通道裡流動。
