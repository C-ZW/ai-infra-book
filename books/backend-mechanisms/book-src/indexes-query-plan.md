# 索引與查詢計畫：B-tree、覆蓋索引與 planner

一張一千萬列的訂單表，一條再普通不過的查詢：

```sql
SELECT id, amount FROM orders
WHERE user_id = 42 AND status = 'paid'
ORDER BY created_at DESC
LIMIT 20;
```

沒有索引時，這條 SQL 會逼資料庫做一件蠢事：把一千萬列**逐一**讀進記憶體、逐一比對 `user_id` 與 `status`、把符合的全部撈出來排序、最後丟掉只留 20 列。使用者要的是螢幕上最近 20 筆訂單，引擎卻為此翻遍了整本帳簿。在一台磁碟每秒能讀幾萬個頁的機器上，這條查詢要讀掉表的全部資料頁，動輒幾十萬到上百萬個頁、幾百毫秒到幾秒；而它本該是個位數毫秒的事。

中間差的那三、四個數量級，全部濃縮在一個詞裡：**索引**。但索引不是「建了就快」的魔法開關——它是一筆你**預先付清、且每次寫入都要續繳**的帳，買到的是讀取時不必掃全表的權利。要把這筆帳算清楚，得先看懂底下那棵樹的形狀，再看懂查詢優化器（planner）憑什麼決定要不要用它。這兩件事——資料結構與決策——合起來，才是「為什麼那條查詢有時飛快、有時退化成全表掃描」的完整答案。

## 為什麼掃全表這麼貴：問題出在「沒有順序」

把問題講難一點。為什麼一條 `WHERE user_id = 42` 非得掃一千萬列？因為**資料在表裡是按寫入順序堆的，不是按 `user_id` 排的**。`user_id = 42` 的列可能散落在表的任何角落——第 3 列一個、第 9,000,001 列一個、中間還散著一堆。引擎事先不知道它們在哪，唯一保證找全的辦法就是每一列都看一遍。這叫 **sequential scan（seq scan / 全表掃描）**，它的成本和表的大小成線性正比，和你最後要的列數毫無關係：撈 20 列和撈 200 萬列，掃的頁數一樣多。

天真的修法是「把表按 `user_id` 排序存」。但你只有一份實體資料，一旦按 `user_id` 排好，就沒法同時按 `created_at` 排、按 `status` 排、按 `amount` 排——每多一種查詢角度，就想要另一種物理順序，而一份資料只能有一種。索引解的正是這個矛盾：**它不動原表，而是另外維護一份「只存某幾欄＋一個指回原列的指標」的、永遠排好序的副結構。** 你想按哪個維度快速查找，就為那個維度建一份這樣的有序副本。代價昭然：每多一份索引，就多一份要佔空間、且每次 `INSERT`／`UPDATE`／`DELETE` 都得同步更新的東西。索引的全部哲學，就是這筆「空間與寫入成本，換讀取速度」的交易。

## 那棵樹為什麼是這個形狀：B-tree 的矮

「排好序的副本」聽起來像一個排序陣列——二分搜尋不就好了？問題在於資料庫的資料不在記憶體、在磁碟（或 SSD），而磁碟的存取單位是**頁（page，典型 8 KB 或 16 KB）**，不是單筆記錄。一個排序陣列要插入一筆新值，得把後面所有元素往後挪——在百萬列規模上，每次寫入搬動半個結構，是災難。資料庫要的是一個「既保持有序、又能便宜地局部插入、且把磁碟讀取次數壓到最低」的結構。這就是 **B-tree**（實務上幾乎都是它的變體 **B+tree**）勝出的地方。

B+tree 是一棵平衡的多叉搜尋樹。它的關鍵不在「樹」，而在**極高的分支因子（fanout）**：每個內部節點正好是一個磁碟頁的大小，一頁能塞下幾百個鍵與子節點指標。fanout 高，樹就矮得驚人——而樹高，正好等於一次查找要讀的磁碟頁數。

把這件事手算到底。假設一頁能容納的 fanout 約 **數百**（一個 8 KB 頁裝幾百個「鍵＋指標」是典型值）。取保守的 fanout = 200：

- 1 層（根）能指向 200 個葉頁；
- 2 層能涵蓋 200 × 200 = 40,000 個葉頁；
- 3 層能涵蓋 200³ = 8,000,000 個葉頁；
- 4 層能涵蓋 200⁴ = 1.6 × 10⁹ 個葉頁。

每個葉頁又裝幾百筆索引項。換算下來，**一千萬列的索引，樹高只有 3 到 4 層**——這不是巧合，是 fanout 在幾百量級時的數學必然。意思是：要從一千萬列裡定位某個 `user_id`，引擎從根頁出發，讀根頁、選一個子頁、讀那個子頁、再選、再讀……**總共只讀 3 到 4 個頁**就到了葉子，而不是掃一萬個頁。這就是那三、四個數量級的來源。表越大，這個優勢越誇張：列數翻 100 倍，seq scan 的成本翻 100 倍，而 B+tree 的樹高頂多多一層。對數成長對線性成長，碾壓是時間問題。

B+tree 還有一個常被忽略、卻決定它能不能當「萬用預設」的設計：**所有真實資料都在葉節點，而葉節點之間用指標串成一條有序的雙向鏈。** 內部節點只放「路標」（鍵與往下的指標），不放資料。這個安排讓 B+tree 同時擅長四件事：等值查找（`= 42`，往下走到葉子）、範圍查找（`BETWEEN`，定位到範圍起點後，沿葉子鏈往右掃即可，不必每筆重新從根找）、前綴查找、以及**有序輸出**——因為葉子本身就是排好序的，`ORDER BY` 想要的順序，索引天生就有，可以省掉一次排序。這四件事一個結構全包，正是 B+tree 成為關聯式資料庫預設索引的原因。它不是最快的等值查找結構（純等值場景，hash 索引更快），但它是**最不挑食**的那個。

順帶一提，B+tree 把「讀很快」買到手的代價，藏在「就地維護一棵排好序的樹」這件事本身——每次寫入都可能觸發頁分裂、隨機改寫某個葉頁。這跟另一種主流引擎用「先順序追加、之後再合併」來換取寫入吞吐的取向，是兩條相反的路（見〈LSM-tree 與 B-tree〉）。本章只談 B-tree 這一側：它買到的是讀的便利，續繳的是寫的成本。

## 一次查詢實際讀了幾個頁：把開場那條 SQL 算到底

回到開場那條查詢，現在把「有沒有索引」的差距用頁數算清楚。

無索引時，引擎做 seq scan：假設這張表的資料佔 **8 GB**、頁大小 8 KB，那就是約 **一百萬個頁**全部讀過，再對撈出的（可能幾千筆 `user_id=42` 的）列做記憶體排序，最後 `LIMIT 20`。讀一百萬個頁，是這條查詢的底價。

現在建一個**複合索引**，把三個條件一次照顧到：

```sql
CREATE INDEX idx_orders_lookup
  ON orders (user_id, status, created_at DESC);
```

這個順序不是隨便排的——它是整章最容易出錯、也最該講透的設計決定，下一節專門拆。先看它怎麼讓查詢飛起來。索引按 `(user_id, status, created_at)` 排好序，意味著所有 `user_id=42 AND status='paid'` 的索引項，在葉子層是**連續的一段**，而且這段內部已經按 `created_at DESC` 排好。於是引擎的動作變成：

1. 從根頁往下走到葉子，定位 `(42, 'paid')` 那段的起點——讀 **3 到 4 個頁**（樹高）；
2. 沿葉子鏈往右，取前 20 筆就停——這 20 筆落在連續的少數幾個葉頁裡，讀 **1 到 2 個頁**；
3. `ORDER BY created_at DESC` 不必排序，因為索引早就是這個順序；`LIMIT 20` 一到就停手，不多讀一筆。

讀的頁數從**一百萬**掉到**個位數**。這就是「三到四個數量級」的具體現場——不是修辭，是把樹高乘出來的結果。

但這裡藏著一個非顯然的細節，一個常被當成終點、其實還沒結束的環節。索引項裡只存了被索引的欄（`user_id`、`status`、`created_at`）和一個指回原列的指標；查詢卻還要 `id` 和 `amount`。所以引擎拿著那 20 個指標，還得**回到原表**，去讀那 20 列所在的資料頁，把 `id` 和 `amount` 撈出來。這個動作叫**回表**（PostgreSQL 的 heap fetch、MySQL 的 bookmark lookup / 二次查找）。這 20 次回表是 20 次**隨機** I/O——指標指向的 20 列可能散在表的 20 個不同頁，每次都是一跳。在這個例子裡 20 次無傷大雅，但若查詢要回表的是兩萬列、且分散在兩萬個頁，回表的隨機 I/O 可能比省下來的還貴——這正是優化器有時**寧可不用索引、直接 seq scan** 的原因之一（順序掃一百萬個頁，可能比隨機跳兩萬次還快，因為順序 I/O 遠便宜於隨機 I/O）。索引不是免費的捷徑，它的划算與否，取決於「定位的精準度」和「回表的代價」之間的拉鋸。

## 欄序就是設計：最左前綴與它的非顯然破口

複合索引 `(user_id, status, created_at)` 的威力全繫於一條規則，這條規則也是索引設計裡最常被搞砸的地方：**最左前綴（leftmost prefix）**。

索引是按 `(user_id, status, created_at)` 這個**複合鍵的字典序**排的。想像電話簿按「姓、名」排：你能瞬間查到「姓陳的所有人」，也能查到「姓陳、名大文的人」，但你**沒法**用這本電話簿快速查「所有名叫大文的人」——因為名叫大文的、姓什麼的都有，散落在整本簿子裡，這本簿子對「只給名」這個查詢毫無幫助。複合索引一模一樣：`(user_id, status, created_at)` 能服務 `WHERE user_id=?`、`WHERE user_id=? AND status=?`、`WHERE user_id=? AND status=? AND created_at>?`，但**不能**只靠它服務 `WHERE status='paid'`——因為 `status` 不是最左欄，所有 `status='paid'` 的項按 `user_id` 散落各處，索引給不出一段連續區間。

這推導出一條鐵律：**欄序就是設計，排錯等於白建。** 把 `(status, user_id)` 反過來建，就服務不了「只給 user_id」的查詢；而 `status` 只有 `'paid'`／`'pending'`／`'cancelled'` 寥寥幾種值（低選擇性），把它放最左，等於一上來就只能把搜尋範圍縮到三分之一，遠不如先用高選擇性的 `user_id` 把範圍縮到某個人的訂單。經驗法則因此是：**等值條件的欄放前面、範圍條件的欄放後面、選擇性高的放更左。** 範圍欄之所以要殿後，是因為一旦某欄用了範圍（`created_at > ?`），它後面的欄在索引裡就不再保證有序——範圍掃過的那一段裡，更後面的欄是亂的，索引的「已排序」紅利到範圍欄為止。

這條規則有一個**非顯然的破口**，PostgreSQL 18（2025-09 GA）才正式補上，值得單獨點出來，因為它顛覆了「不是最左前綴就一定用不上」的舊直覺。PostgreSQL 18 引入了 **skip scan**：當複合索引的前導欄**基數很低**（比如 `status` 只有三種值），而你的查詢只約束了後面的欄時，planner 可以「替你把前導欄的每一個可能值都試一遍」——對 `status` 的三個值，各做一次後段欄的小範圍查找，再跳到下一個值。等於用「對低基數前導欄逐值跳躍」換取「沒有最左前綴也能用上索引」。這不是廢除最左前綴規則，而是在「前導欄值很少」這個特例下繞過它（MySQL 8.0.13+ 也有類似的 skip scan，其設計類比於更早、針對 `GROUP BY`／`DISTINCT` 的 loose index scan）。它的存在恰恰反證了規則的本質：最左前綴之所以是限制，是因為前導欄一旦值多到沒法逐一枚舉，跳躍就退化成全掃；skip scan 只在前導欄值少到可枚舉時才划算。

## 覆蓋索引：連回表都省掉

前面那 20 次回表的隨機 I/O，有沒有辦法乾脆消滅？有，這就是**覆蓋索引（covering index）**——讓查詢需要的**每一欄**都待在索引裡，引擎讀完索引就湊齊了全部資料，**根本不必回原表**。這種掃描叫 **index-only scan**。

開場那條查詢要的是 `id` 和 `amount`，而這兩欄都不在原索引裡——PostgreSQL 的 B-tree 索引項葉子存的是「索引鍵值＋一個指回 heap 的物理指標（TID）」，那個 TID 是一筆列在表中的位置，**不帶**主鍵 `id` 的值（這和 InnoDB 不同：InnoDB 的 secondary index 才把主鍵欄當最右欄一起存進去）。所以要 index-only，得把查詢需要、但不在搜尋鍵裡的 `amount` 和 `id` 都「掛」進索引：

```sql
CREATE INDEX idx_orders_cover
  ON orders (user_id, status, created_at DESC) INCLUDE (amount, id);
```

PostgreSQL 的 `INCLUDE` 子句把 `amount`、`id` 存進索引的葉節點，**但不參與排序**（它們不是搜尋鍵、不影響樹的結構，只是搭便車存著供取值）。現在引擎沿葉子鏈取那 20 筆時，`user_id`、`status`、`created_at`、`id`、`amount` 全都在手邊，那 20 次回表的隨機 I/O——消失了。對一條每秒被打數千次的熱查詢，省掉的回表 I/O 是實打實的延遲與磁碟負載下降。

代價同樣誠實：索引變大（多存了 `amount`、`id`，葉子更胖、樹可能更高一點），寫入更貴（`amount` 一改，索引也得改），而且**查詢欄位集一變就失效**——哪天前端多顯示一個 `currency` 欄，這個覆蓋索引就不再「覆蓋」，又回到要回表。覆蓋索引是為**特定一條固定的熱查詢**量身打造的，不是通用加速。

這裡有一個幾乎所有人都會踩、且只有 PostgreSQL 用戶會遇到的**非顯然陷阱**：在 PostgreSQL 裡，「index-only scan 不必回表」這個承諾**是有條件的**，條件是那一列所在的資料頁在 **visibility map（可見性映射）** 上被標記為「全部可見」。為什麼需要這個？因為 PostgreSQL 用 MVCC，一筆資料可能存在多個版本，索引項裡**不帶版本可見性資訊**——引擎光看索引，無法判斷「這一列對當前交易到底可不可見、是不是已被某個未提交交易刪了」。visibility map 是一個極小的位元圖，為每個資料頁記一個位：若這個頁上所有列都老到「對所有交易都可見」，這個位就是 1，引擎可以信任索引、不回表。但這個位**由 VACUUM 設定**——一個剛被大量更新過、VACUUM 還沒跑到的頁，它的位是 0，於是即使你建了完美的覆蓋索引，引擎**仍得回表**去查那一列的可見性，「免回表」的承諾當場打折。

這個陷阱的非顯然之處在於：它把兩個看似無關的機制綁在了一起。你的覆蓋索引效益，竟然取決於 MVCC 的垃圾回收節奏（見〈MVCC：讓讀不擋寫〉）。一張寫入極頻繁、VACUUM 永遠追不上的表，上面的 index-only scan 會頻繁退化成普通 index scan，而 `EXPLAIN` 裡會看到 `Heap Fetches` 這個數字居高不下——它正是在量化：有多少「理論上免掉」的回表，其實仍落了回去。

值得一提的是，這個特定陷阱是 PostgreSQL「舊版本留在主表、靠 VACUUM 回收」這條 MVCC 路線的副作用；MySQL InnoDB 走的是「最新版本就地、舊版本進 undo log」的另一條路，它的覆蓋索引（secondary index 本就帶主鍵欄）不需要 visibility map 這道關卡，但換來的是別的取捨。同一個「覆蓋索引」概念，在兩種 MVCC 實作下，連「免回表的條件」都不一樣。

## planner 不是執行你寫的 SQL，是執行它選的計畫

到這裡有個容易被跳過的真相值得擺到檯面：**你建了索引，不代表查詢會用它。** SQL 是宣告式的——你描述「想要什麼結果」，不指定「怎麼取」。決定怎麼取的，是引擎裡的**查詢優化器（query planner / optimizer）**。它面對你的 SQL，會列舉一堆可能的執行方案（用索引 A？用索引 B？兩個索引取交集？直接 seq scan？這個 join 先做還是後做、用 nested loop 還是 hash join？），給每個方案**估一個成本**，然後挑最便宜的那個去跑。`EXPLAIN` 給你看它選了什麼計畫，`EXPLAIN ANALYZE` 會真的跑一遍、把估計值和實際值並排給你看。

它憑什麼估成本？憑**統計資訊**。PostgreSQL 由 `ANALYZE` 命令（autovacuum 會自動觸發）對每張表取樣，把「這欄大約有多少種不同的值（n_distinct）」「最常見的值是哪幾個、各佔多少比例（most common values）」「值的分佈長什麼樣（histogram）」這些統計，存進系統表 `pg_statistic`。預設的取樣精細度由 `default_statistics_target` 控制，**預設值 100**（每欄最多取 100 個直方圖區間與最常見值，2026-06）。planner 拿這些統計來估「`WHERE user_id=42 AND status='paid'` 大概會命中幾列」——這個估計叫**選擇性（selectivity）估計**，它是整個成本模型的地基。估得準，就選對計畫；估歪了，全盤皆錯。

於是「查詢有時飛快、有時退化成全表掃描」這件事，根源往往不在索引本身，而在統計：

**統計過期。** 昨天 `ANALYZE` 時這張表還是一千列，今天灌進了一千萬列卻還沒重新取樣，planner 仍以為它只有一千列——於是它覺得「反正才一千列，seq scan 也很快，何必用索引」，選了全表掃描，而實際上要掃一千萬列。同一條 SQL，昨天毫秒級、今天好幾秒，常常就是這個。修法是 `ANALYZE`（或讓 autovacuum 跟上）。

**選擇性被低估或高估。** 若某欄值的分佈極度傾斜（99% 的列 `status='paid'`，剩下 1% 散在其他值），而 planner 用了過於粗糙的均勻假設，它可能以為 `status='shipped'` 會命中很多列、決定 seq scan，實際上只有寥寥幾列、本該走索引。提高那一欄的 `default_statistics_target`、讓直方圖更細，常能救回來。

**索引明明存在，卻被條件本身廢掉。** 這是最隱蔽的一類，因為它和統計無關，純粹是查詢寫法讓索引「用不上」：

- 在被索引的欄上**套了函數或運算**：`WHERE lower(email) = 'a@b.com'` 對 `email` 上的索引無效——因為索引存的是 `email` 的原值、排的是原值的序，不是 `lower(email)` 的序，引擎沒法在一棵「按原值排序」的樹上對「函數結果」做有序查找。（要救：建一個 `lower(email)` 的**表達式索引**。）
- **隱式型別轉換**：`WHERE phone = 12345`，而 `phone` 是字串欄——引擎得把每一列的 `phone` 轉成數字再比，等於對每列套了個轉換函數，索引同樣作廢。
- **前綴萬用的 `LIKE`**：`WHERE name LIKE '%son'` 用不上 B-tree 索引，因為 B-tree 是按「從左到右的字典序」排的，你卻要找「結尾是 son」的——這跟「電話簿按姓排，你卻要找名字結尾是某字的人」一樣，順序幫不上忙。`LIKE 'john%'`（前綴固定）則用得上，因為那是個前綴範圍查找——但在 PostgreSQL 有個前提：只有 C locale 下，預設 operator class 的索引才會自動服務這種前綴比對；非 C locale（ICU／glibc 這些常見預設）下，得用 `text_pattern_ops` 這類 operator class 另建索引才行。
- **低選擇性條件**：`WHERE is_deleted = false`，而半數列都符合——planner 算了算，發現「走索引定位一半的列、再回表」比「直接 seq scan」還貴（因為回表那麼多列的隨機 I/O 太兇），於是理性地**選擇不用索引**。這不是 bug，是它算對了。

這四種裡前三種是「索引被廢」，最後一種是「索引在但不划算」，但它們在 `EXPLAIN` 裡長得一樣：都是 `Seq Scan` 而不是 `Index Scan`。**這就是為什麼「我建了索引」這句話本身不算數，得看 `EXPLAIN` 才算數。** planner 的選擇是一個成本估算的結果，不是你建索引的義務回報。

## 為什麼是這個形狀

退一步看，索引與查詢計畫這一整套機制，是從一個無法迴避的張力裡長出來的：**一份實體資料只能有一種物理順序，但查詢想從很多個維度快速切入。**

正因如此，才需要在原表之外，為每個想快速查找的維度，另外維護一份排好序的副結構——這就是索引，以及它「空間與寫入換讀取」的本質代價。正因為這份副結構住在以頁為單位、隨機存取昂貴的磁碟上，它才長成 B+tree 那種「高 fanout、矮到只有三四層、葉子串成有序鏈」的形狀——把磁碟讀取次數壓到對數級，順便一併支援等值、範圍與有序輸出。正因為複合鍵只能按一個固定的字典序排，才有最左前綴這條規則，以及它在「前導欄值少」時被 skip scan 鬆綁的例外。正因為索引項不帶可見性資訊，PostgreSQL 的 index-only scan 才得靠 visibility map 兜底，把覆蓋索引的效益悄悄綁上了 VACUUM 的節奏。也正因為 SQL 是宣告式的，「用不用索引、用哪個」這個決定才落到 planner 手上——而 planner 只是個拿著取樣統計做成本估算的理性機器，統計一過期、查詢一寫歪，它就會「正確地」選出一個糟糕的計畫。

所以下次你看到一條 SQL 莫名其妙地慢，第一個動作不該是「再加一個索引」，而該是 `EXPLAIN ANALYZE`：看看 planner 到底選了什麼計畫、它估的列數和實際差多遠、`Heap Fetches` 是不是高得反常、那個你以為會被用上的索引為什麼躺在那裡沒人理。索引不是你下達的命令，是你提供給 planner 的一個選項；它用不用、什麼時候用，是它根據一份可能已經過期的世界觀，算出來的決定。讀懂這條因果鏈，你就從「會建索引」走到了「知道索引為什麼有時不靈」——而後者，才是這個機制真正難的地方。
