# 自建還是採用：相依的長期成本

2016 年 3 月某個下午，世界各地的 JavaScript 工程師同時看到自己的建置失敗了。Babel 編不過、Webpack 跑不動、CI 一片紅，錯誤訊息指向一個你大概從沒注意過的套件：`left-pad`。它的全部內容是十一行——把字串左邊補上空白或零，補到指定長度為止。一個你五分鐘能自己寫完、甚至不會多想一秒的功能。

那天它消失了。作者因為一場命名權糾紛，一氣之下把自己在 npm 上的全部套件都撤了下來，其中包括 `left-pad`。它本身沒人直接依賴，但 Babel 依賴它、無數工具又依賴 Babel，於是這十一行像抽掉一塊不起眼的積木，讓上面整座塔垮了。npm 大約兩小時後手動還原了那個版本，並從此收緊了撤回規則：套件只要還有別人依賴它，作者就再也撤不掉。但那兩小時已經把一件事攤在所有人面前——**你的系統能不能建置，取決於一個你沒寫、沒讀過、甚至不知道它存在的東西，是否還願意待在那裡。**

這就是「自建還是採用」這個問題真正的形狀。它表面上是一道效率題：這塊功能，自己寫划算，還是拉個現成的套件划算？但 `left-pad` 揭穿的是，**「拉一個現成的進來」從來不是「省下寫它的時間」這麼乾淨的一筆交易**。你省下的是現在的工，換來的是一條你無法完全掌控的長期相依——它會在你睡覺時改版、會在某天被作者撤掉、會悄悄拉進另外幾百個你從沒選過的東西、會成為攻擊者潛入你系統的一道側門。這一章要把這筆交易的兩端都算清楚：你到底用「不寫它」換到了什麼，又賠進去了什麼。

## 為什麼這是一筆難算的帳

天真的演算法是這樣的：自建要花兩週寫、之後要維護；採用一個成熟套件只要十分鐘接進來。兩週 vs 十分鐘，採用完勝，還用想嗎？

這個演算法錯在它只看了交易的**買入價**，沒看**持有成本**，更沒看**退場成本**。一個相依的真實開銷攤在它整個生命週期上，而且其中大部分發生在你接進來「之後」：

它會老。你接進來時是 v3.2，半年後安全公告說 v3.2 有個遠端執行漏洞、要升到 v5；但 v5 改了 API，你得跟著改一輪自己的程式碼。它會跟著上游的破壞性變更走，而你沒有投票權。

它會壞，而且壞在你看不懂的地方。出事時你 debug 的不是自己寫的邏輯，是一個陌生人三年前寫的、文件只覆蓋 happy path 的程式碼。你得先讀懂它，才能修它周邊的東西——而你當初選它，正是為了「不必懂它」。

它不只是「它」。你直接挑的是一個套件，但它背後拖著一整棵相依樹。npm 生態裡，一個尋常專案間接拉進的相依動輒**數十到數百個**（確切數字隨統計方法與抽樣差很大，但量級一致地嚇人）——你親手挑的可能只有五個，其餘那一大票是「你依賴的東西所依賴的東西」，遞迴下去。這整棵樹的每一個節點，都是一個潛在的故障點、一份潛在的授權義務、一塊潛在的攻擊面。你為這整棵樹負責，但你只認得最上面那幾片葉子。

它可能反過來咬你。這一點最反直覺，但也最致命，下一節單獨講。

所以這筆帳真正在問的不是「現在哪個便宜」，而是「**把整個生命週期、整棵相依樹、整個攻擊面都算進去之後，哪個便宜**」。而要算清這筆帳，得先想清楚一個更前置的問題：這塊功能，到底是不是你的核心。

## 唯一真正的判準：這是不是你贏的地方

把「自建 vs 採用」拆成一條條技術指標去比，很容易迷失。真正能把多數情況一刀切開的判準只有一個：**這塊功能，是不是你的核心競爭力——是不是「你能不能贏」取決於它做得好不好的地方。**

是核心，就傾向自建。不是因為自建在技術上更優，而是因為核心是你必須能**完全掌控、且願意長期養**的東西。你的差異化來自它，你就不能把它的演進節奏、它的故障模式、它的天花板，交到一個外部維護者手上。你需要能隨時把它改成只有你的場景才需要的樣子，需要在它半夜出事時有人真的懂它的每一行。這份掌控權的代價是你得永久維護它——但對核心而言，這份維護本身就是你在投資自己贏的能力。

不是核心，就傾向採用。序列化、加密、HTTP server、訊息佇列、壓縮、JSON 解析——這些是被解決過千百遍的橫切問題，自建幾乎一定虧。你在這上面寫的每一行，都是把人力燒在不會讓你贏的地方，而且多半寫得比那些被千萬人用過、踩過、修過的成熟方案更爛、更不安全。把這些交出去，省下的心力才能投到真正讓你贏的核心上。

這個判準之所以有力，是因為它把問題從「哪個技術上更好」這種沒有普適答案的爭論，轉成「**這塊值不值得我親自掌控**」這種有答案的判斷。而兩端各有一個失敗症候群，認得它們，你就能在偏離判準時及時警覺。

一端是 **NIH（Not Invented Here，非我所創）**：什麼都自己寫。重造已經滾圓了的輪子，養一堆只有內部人懂的私有框架，把寶貴的工程時間花在早就被解決的問題上——還美其名曰「掌控」。NIH 的代價隱蔽，因為它消耗的是機會成本：你在自建一個爛掉的 ORM 的那三個月，本該拿去打磨真正讓你贏的東西。

另一端是**過度相依**：為了省幾行，拉進一棵巨樹。`left-pad` 是這一端的紀念碑——為了不寫十一行字串補白，把整個建置流程的命運，交給一個十一行套件的作者心情。每多接一個相依，你就多買進一堆你不會用到、卻要負責的故障點與攻擊面。

兩端都是 over-engineering 的不同臉孔：一個在不該自建的地方自建，一個在不該依賴的地方依賴。判準——「這是不是核心」——同時管住兩端。

## 相依會反過來咬你：供應鏈這道側門

前面說相依「可能反過來咬你」。把這句話講透，得看一個讓整個產業背脊發涼的案例。

2024 年初，一位 Microsoft 的工程師 Andres Freund 在調查一個瑣碎的效能問題——他的 SSH 登入慢了大約 500 毫秒。一般人會聳聳肩，他卻順著這條線索一路挖，最後挖出了 **CVE-2024-3094**：一個被植入 `xz utils`（一個壓縮函式庫）的後門。

這個後門的可怕，不在技術本身，而在它的**潛伏路徑**。攻擊者（化名 Jia Tan）不是駭進去的——他花了將近兩年，以「熱心貢獻者」的身分慢慢經營這個冷門但關鍵的開源專案，回 issue、修 bug、累積信譽，直到原本就分身乏術的維護者把維護權交給了他。拿到權限後，他在發布的 tarball 裡藏進了惡意碼（注意：是發布的壓縮檔，不是 git 倉庫裡看得到的原始碼——這讓在 GitHub 上 review 程式碼的人看不出端倪）。這段惡意碼會在編譯時改寫 `liblzma`，而 `liblzma` 透過 `systemd` 被 `sshd` 間接載入，於是後門最終能在 SSH 完成身分驗證**之前**就攔截並執行攻擊者送來的指令——一個遠端、認證前的完整控制權。

把這條鏈攤開看：`xz` 是個壓縮庫，幾乎沒有人「直接」依賴它。但它被現代 Linux 發行版透過 `systemd → sshd` 的鏈路幾乎無所不在地**間接**依賴著。一個你的服務清單上根本不會出現的冷門套件，差一點就成了從你的 SSH 守門員背後捅進來的刀。它沒被大規模利用，純粹是因為那位工程師多管了一個 500 毫秒的閒事。

`xz` 教的這一課是：**你的攻擊面是整棵相依樹，不是你直接挑的那幾個**。而攻擊者很清楚這一點——他們不會去攻你嚴防死守的核心，他們會去找那棵樹上最深、最冷門、最沒人盯著維護權轉移的那個節點。

這不是孤例，而是一種正在工業化的攻擊模式。2025 年 9 月，一個叫 **Shai-Hulud** 的自我複製蠕蟲掃過 npm，是 npm 史上第一次成功的「自動傳播」攻擊：它先用釣魚騙到某個維護者的發布 token，然後用這個 token 找出該維護者名下的**所有**套件，把惡意碼注進去、發布新版本，再從感染這些套件的機器上偷出更多 token，指數式地擴散——一個套件的維護者中招，他名下幾十個套件連帶淪陷，這幾十個套件的下游又連帶暴露。受影響的套件數以百計。

供應鏈攻擊的本質，是**信任的遞移**。你信任你直接依賴的套件，於是你也被迫信任它依賴的東西、以及維護那些東西的每一個人、以及他們的帳號安全。這條信任鏈有多長，你的攻擊面就有多寬——而你對這條鏈的絕大部分，毫無可見性，也毫無控制權。

## 採用之前：把盡職調查做完

既然每個相依都是一筆把控制權交出去的交易，那麼在簽約之前——也就是 `npm install` 那一刻之前——你該像盡職調查一樁併購那樣審視它。幾個非看不可的維度：

**有效維護，不是人氣。** star 數多不等於安全。`xz` 那個被植入後門的庫，並不冷門到沒人用——它冷門到沒人盯它的維護權轉移。要看的是「最近的 commit 多新、issue 有沒有人回、release 節奏穩不穩、會不會明天就停更」，尤其是**有沒有單一維護者風險**——一個累垮的、無償的、獨自扛著一個被千萬人依賴的庫的維護者，本身就是一個故障點。

這一點有個經典的悲劇註腳。2014 年的 **Heartbleed**——OpenSSL 那個讓全球無數網站洩漏記憶體內容（含密碼、私鑰）的漏洞——背後是這樣一個事實：OpenSSL 被半個網際網路依賴，但在事發前，它一年只收到約 2,000 美元的捐款，核心程式碼長期幾乎只靠一個全職的人在維護，他一個人審了那 45 萬行程式碼裡超過一半的更新。事後 Linux 基金會才成立 Core Infrastructure Initiative 去資助這類「大家都在用、卻幾乎沒人在養」的關鍵基礎設施。**你依賴的東西有沒有人在好好維護，和它有多熱門，是兩件無關的事。**

**相依樹的實際大小。** 別只看你直接拉的那個，要看它實際帶進來幾個 transitive dependency。一個「輕量」的工具，拆開可能是一棵幾百個節點的樹。每個節點都進了你的攻擊面和授權義務範圍。

**授權相容性。** 這是另一種「反過來咬你」。把一個 copyleft 授權（特別是 **AGPLv3**）的東西當無條件免費用，可能反過來綁住你自己的程式碼。AGPLv3 的關鍵在於它的第 13 條堵掉了 GPL 的「SaaS 漏洞」：一般 GPL 的 copyleft 義務由「散布（distribution）」觸發，而你把軟體當服務跑、只透過網路給人用、從不把二進位檔交出去，就繞過了散布——AGPL 把觸發點改成「**讓人透過網路使用你修改過的版本**」。也就是說，如果你改了一個 AGPL 的元件、又把它當線上服務提供，你就有義務把對應的原始碼開放給那些使用者。對一家靠閉源服務賺錢的公司，這可能是不可接受的。（精確地說：這義務由「**修改**」觸發；未經修改地使用通常不觸發第 13 條——但「什麼算修改」的邊界，正是你該在引入前和法務確認、而不是事後才發現的地方。）商用授權的「免費層」同理，常有配額、用途或部署方式的限制，越過了就要付費或就違約。

**攻擊面權限。** 這個套件要不要網路、要不要讀寫檔案系統、會不會在安裝時跑 script、要不要開子行程？一個只做字串處理的庫卻要求網路權限，是個該停下來的訊號。

盡職調查不是官僚流程，它是你在「把控制權交出去」之前，唯一一次便宜地看清楚「交給的是誰、交出去多少」的機會。跳過它，你就是在閉著眼睛簽一份你沒讀過的長期合約。

## 算一筆退場帳：薄採用 vs 厚採用

盡職調查管的是「要不要接」。但相依管理還有一個被嚴重低估的維度：**就算決定採用，你能不能保留「日後換掉它」的退路。** 這個退路的成本，叫退場成本（cost of exit），而它常常比採用本身的決策更要命——因為很多相依不是「壞了才痛」，而是「想換才發現換不掉」。

用一個具體的對照把這筆帳算出來。假設你要為服務加上物件儲存，把使用者上傳的檔案存到某家雲端供應商。有兩種接法。

**厚採用**：你圖快，直接把那家供應商的 SDK 撒得到處都是。上傳邏輯在 controller 裡直接呼叫它的 `client.uploadObject(...)`、下載在另一處直接 `client.getObject(...)`、產生預簽 URL 又在第三處、生命週期設定在第四處……假設散落在 47 個檔案、共 120 處呼叫，每一處都用了那家 SDK 特有的參數結構、錯誤型別、回傳格式。

**薄採用**：你先花半天，定義一個自己的介面——

```typescript
// your own seam, not the vendor's
interface ObjectStore {
  put(key: string, body: Buffer): Promise<void>;
  get(key: string): Promise<Buffer>;
  presignGet(key: string, ttlSeconds: number): Promise<string>;
}
```

——然後寫一個 adapter，把那家供應商的 SDK 包在這個介面**後面**。你全部的業務程式碼只認得 `ObjectStore`，那 120 處呼叫變成呼叫你自己的介面，供應商的 SDK 只出現在那一個 adapter 檔裡。這層包裝有個正式名字：**防腐層（anti-corruption layer）**——它的作用是不讓第三方的概念、命名、怪癖滲進你的核心，把「外面的世界」和「你的世界」之間架一道翻譯閘。

現在算退場帳。某天你要換供應商——可能因為漲價、可能因為它出了區域性故障、可能因為合規要求資料落地別處。

- 厚採用下，你得改 120 處呼叫，每一處都要把舊 SDK 的參數結構、錯誤處理、回傳格式，翻譯成新供應商的——分散在 47 個檔案裡，每一處都是一次出錯的機會，整個改動橫跨好幾個團隊、要重新測一輪全部上傳下載路徑。
- 薄採用下，你寫一個新的 adapter（實作同一個 `ObjectStore` 介面），改一行設定把舊 adapter 換成新的。業務程式碼一行都不用動，因為它從頭到尾只認得那個介面。

半天的前置投入，換來把一場跨 47 檔、120 處、多團隊的大遷移，壓縮成「寫一個新 adapter + 改一行設定」。這就是退場成本的槓桿：**薄採用買的是「換實作不換程式碼」的權利。**

更進一步的薄採用，是連介面都對齊一個**標準**而非單一實作。物件儲存有事實標準的 API（多家供應商相容），訊息佇列、SQL、快取協定也都有多家相容實作——依賴「這類東西的標準形狀」而非「某一家的專有 API」，你的退路天生就寬。但要誠實：標準的「相容」幾乎總有方言差，邊角的功能、效能特性、一致性語意常綁著實作走，所以薄採用降低退場成本，不等於消滅它。它把一場手術變成換零件，但零件的接縫處仍可能咬手。

這也呼應一個更普遍的設計原則：可逆性是可以買的。一個本來像「單向門」的選擇（選了某家供應商、深陷其中、回不了頭），透過防腐層的隔離，被改造成「雙向門」（試了不對，換個 adapter 就退回來）。adapter 本身是成本——它是一層你要維護的間接、一個你得設計對的介面（介面選錯，一樣難換）——但對「預期未來可能要換」的相依，這筆保費通常很划算。

## 接進來之後：把這棵樹釘住、盯住

決定採用、也做了隔離，事情還沒完。相依是活的——它會自己改版、它的相依樹會在你不注意時長出新枝、樹上某個節點某天會冒出一個 CVE。接進來之後的控管，是把這筆持有成本壓到可承受的關鍵。

**用 lockfile 釘死版本與雜湊。** 只寫 `"^3.2.0"` 這種範圍版本，意味著下次安裝可能裝到 `3.7.0`——一個你沒測過的版本，可能引入了行為差異甚至惡意碼。lockfile（`package-lock.json`、`yarn.lock`、`Cargo.lock` 之類）把整棵樹的每個節點釘到精確版本，並記下內容雜湊。雜湊這一筆尤其重要：它防的是「同一個版本號，內容被掉包」——版本號 `3.2.0` 沒變，但 registry 上那個 `3.2.0` 的 tarball 被換成了帶後門的，雜湊對不上，安裝就會失敗。沒有 lockfile，你每次 CI build 拉到的可能是不同的相依樹，「在我機器上能跑」就成了沒有意義的句子。

**把漏洞掃描接進 CI。** 既然你的攻擊面是整棵樹，你就需要持續地對整棵樹比對已知漏洞資料庫——每次有新的 CVE 揭露、或你的樹長出新節點，都該被掃到。這類掃描（SCA，software composition analysis）的價值不在「找到一個漏洞」這個瞬間，而在它把「我依賴的東西安不安全」從一次性的盲信，變成持續的、自動化的監看。

**知道你到底依賴了什麼。** 你能不能在五分鐘內，列出你生產系統裡每一個（含 transitive 的）相依、它們的版本、它們的授權？如果不能，那 `xz` 那種「你清單上根本不會出現的套件」就有藏身之處。這份清單有個名字叫 SBOM（software bill of materials，軟體物料清單）——它的存在意義，是讓「下一個 CVE 公告出來時，我能不能在十分鐘內知道自己有沒有中」這個問題有答案。

**對最關鍵的相依，考慮自己留一份。** 對那種「一旦消失或被掉包就要命」的核心相依，可以 vendor 進自己的倉庫、或架內部 mirror——`left-pad` 那種「作者一撤就垮」的劇本，前提正是你的 build 在每次安裝時都得去公開 registry 把它拉下來。把關鍵相依放進自己掌控的範圍，等於買一份「它在上游消失時我還能 build」的保險。

這些動作合起來，是把「採用」這筆交易裡那條失控的尾巴——版本漂移、樹的擴張、攻擊面的暗角——重新拉回到你能看見、能管的範圍。它們不消除相依的風險，但讓風險從「不可見、不可控」變成「可見、可控」，而這個轉變本身，就是相依管理的全部價值。

## 當天平被推動：成本變了，判斷的輸入就變了

前面整套判斷——核心 vs 非核心、盡職調查、隔離、釘住——它的框架是穩的，但框架裡有一個隱性的輸入數字：**自建很貴。** 正因為自己寫一個小工具要花掉可觀的工，許多「其實不是核心、但也沒大到該採用整個框架」的灰色地帶，過去都倒向了採用——拉個套件總比花一週自己寫便宜。

當「生產一段能跑的程式碼」的成本被壓低一個量級，這條判斷線就會往「自建」那邊移。以前因為太費工而採用的小工具，現在自己生一個可能更划算、而且程式碼完全在你手上、沒有外部維護者跑路風險、沒有那棵幾十上百個節點的相依樹、沒有供應鏈側門。對非核心的小工具，這是個真實的、值得善用的轉變。

但這裡藏著一個新陷阱，而且它正好踩在這一章的痛點上。「自建成本下降」不等於「自建沒成本」——它**只降了寫的成本，沒降理解與維護的成本**。一段快速生成出來的「自建」程式碼，如果沒有人真正讀懂它、測過它、能在它半夜出事時修它，那它就不是「掌控」，而是「能跑但沒人懂」——一種新型態的、利滾利的技術債。掌控權的前提，從來是有人真的懂這東西；繞過了這個前提的「自建」，只是把外部相依的失控，換成了內部黑盒的失控。

還有一個更直接的陷阱：快速生成的程式碼，可能默默替你**建議或內嵌了相依**——而這些相依可能是過時的、或根本不存在的、甚至是名稱被惡意搶註的（typosquatting：把 `lodahs` 偽裝成 `lodash`，賭你不會細看）。等於是在你以為自己「自建、可控」的時候，從另一個方向替你開了一道供應鏈側門。所以這一章講的盡職調查、釘住、掃描，不但沒過時，反而更該嚴格地套用在每一個被替你選進來的相依上——不論那個「替你選」的，是某個 transitive dependency 的作者，還是一段生成出來的程式碼。

## 為什麼是這個形狀

退一步看，「自建還是採用」之所以難，不是因為缺少正確答案，而是因為**它根本不是一道能一次算完的效率題**。

它難，是因為一個相依的成本不在你接它的那一刻結帳，而是攤在它整個生命週期上、攤在你看不見的整棵相依樹上、攤在一條你無法控制的信任鏈上——所以你不能只比「現在哪個便宜」，得比「把未來都算進去之後哪個便宜」。

它需要那個「是不是核心」的判準，是因為唯有「值不值得我親自永久掌控」這個問題有答案，而「哪個技術上更好」沒有——核心你必須能完全掌控、願意長期養，非核心你該果斷交出去、別把心力燒在不會讓你贏的地方。

它需要盡職調查、防腐層、lockfile、漏洞掃描這一整套，是因為「把控制權交出去」這個動作本身無法避免——只要你採用，你就交出了一部分掌控權——你能做的，是讓這份交出去的控制權**可見、可換、可監看**：看清楚交給的是誰，保留把它換掉的退路，持續盯著它會不會反過來咬你。

`left-pad` 那兩小時、`xz` 那 500 毫秒、Heartbleed 那一年 2,000 美元，講的是同一件事：**你的系統，建立在無數你沒寫、沒讀、不認識的人的善意與警覺之上。** 採用一個相依，就是把一塊地基的維護權，託付給這條鏈上的某個陌生人。這筆託付多數時候很划算——你確實不該自己寫加密、自己寫壓縮、自己寫一切。但它是託付，不是免費；是合約，不是禮物。「自建還是採用」這個問題之所以長成這個形狀，正是因為它逼你在每一次 `install` 之前，誠實地回答：這塊地基，我願意託付給誰、託付多少、出事時我有沒有退路。
