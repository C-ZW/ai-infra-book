# 測試策略：unit、integration、contract、e2e

一支綠燈的測試套件，是工程師對自己系統下的賭注：它說「我相信這次改動沒有破壞既有行為，可以送出去」。問題是這個賭注的**賠率**幾乎從不被認真計算。一個跑了三千個 unit test、全綠、覆蓋率 92% 的服務，可能在上線第一個小時就因為某個欄位被供應端悄悄改名而崩潰；而另一個只有幾百個測試、卻把真實資料庫和服務邊界都釘住的套件，可能一年都不出事。同樣是「綠燈」，買到的信心天差地遠。

所以測試策略真正的問題不是「要不要測」——這沒人反對——而是一個資源分配問題：**你有一筆有限的預算（工程師的時間、CI 的機器分鐘、每次 push 後等紅綠燈的耐心），要把它鋪到哪幾層測試上，才能用最低的成本買到最高、而且不是假的信心。** 每一層測試都在「真實度」和「速度與穩定」這條軸上佔據不同的位置，而這條軸上沒有免費的位置：越接近真實的東西，跑起來越慢、越容易無端變紅、紅了之後越難定位是哪一環壞掉。把這條軸看清楚，整個策略就立得起來。

## 四層，其實是同一條軸上的四個座標

把一筆寫入流程攤開：一個 HTTP handler 收到請求，呼叫一個計算折扣的純函式，把結果寫進 PostgreSQL，再發一則訊息給下游的庫存服務。要驗證這條路徑是對的，你可以從四個不同的「縮放層級」下手，而這四個層級不是四種不同的技術，是**你願意讓多少真實的東西參與這次測試**。

**unit** 把鏡頭推到最近：只測那個折扣函式。資料庫、訊息佇列、HTTP 全部用 mock 或 stub 替掉，函式收到一組輸入、吐出一個輸出，你斷言它對。這種測試**快到可以每存檔就跑一次**——沒有 I/O、沒有網路、沒有容器啟動，幾百個 unit test 在一秒內跑完是正常的。它定位精準：紅了就是那個函式錯，不會是別的。但它的信心邊界也畫得最死——它**只證明了那個函式在「它所 mock 出來的那組依賴行為」下是對的**，對「這些部件兜起來能不能跑」一無所知。

**integration** 把鏡頭拉遠一格：讓程式碼配上**真實的**資料庫、真實的快取、真實的佇列一起跑。現代做法通常是用 testcontainers 這類工具，在測試開始時用 Docker 起一個**真的** PostgreSQL（不是 in-memory 的假貨、不是 SQLite 冒充），測完即丟，每條 CI pipeline 各自隔離一套、互不污染（這正是 testcontainers 的賣點：throwaway、ephemeral 的真實服務，多語言皆有）。它買到的是 unit 永遠買不到的東西——**接線是對的**：你的 SQL 語法真的能在那個版本的 PostgreSQL 上跑、你的 ORM 真的把那個 enum 序列化成資料庫認得的值、那個交易隔離級別下的行為真的如你所想。代價是慢（起一個容器要數秒到數十秒）、需要環境、而且當它紅的時候，定位範圍變大了。

**e2e（端到端）**把鏡頭拉到最遠：把整條真實系統串起來——真的前端、真的閘道、真的所有後端服務、真的資料庫——然後模擬一個使用者真的會走的旅程（登入 → 加入購物車 → 結帳 → 收到確認）。它的真實度最高，因此信心最強：**如果這個測試綠了，這條使用者旅程在生產環境幾乎一定能走通。** 但它把所有缺點也一併推到極致：最慢（整條系統要拉起來、要等網路、要等非同步傳播完成）、最脆（任何一環的任何抖動都讓它變紅）、定位最差（紅了你常常不知道是哪個服務、哪一跳壞的，得從一條二十幾步的旅程裡逆向追凶）。

**contract（契約）**不在這條從近到遠的縮放軸的延伸線上——它是這條軸**斜出去的一條岔路**，專門解一個前三層都解不好的縫，值得單獨講透。

## 為什麼需要金字塔這個形狀

把這四層的成本與信心並排，一個結構就浮現了：越往上（越接近 e2e），每個測試**買到的信心越高、但付出的成本（時間、穩定性、定位難度）也越高**；越往下（越接近 unit），單價越便宜、但單個測試買到的信心越薄。

於是經典的**測試金字塔**給出一個直覺的分配：底層 unit 鋪很多（便宜、快、定位準，當作你的第一道密網），中層 integration 適量（驗接線，貴一點），頂層 e2e 只留少數幾條最關鍵的黃金路徑（昂貴，只買最值得買的信心）。這個形狀最早由 Mike Cohn 在 2009 年的《Succeeding with Agile》提出，後來由 Martin Fowler 的〈The Practical Test Pyramid〉發揚成今天的通用語彙。

金字塔的反面，業界給了一個很傳神的名字：**冰淇淋甜筒反模式**——上面一大坨 e2e、底下只剩一小撮 unit。為什麼這個形狀會把團隊拖垮，值得用數字算一次，因為它的崩壞不是線性的，是**相乘**的。

設想一條 e2e 旅程有 25 個步驟，每一步單獨看都很穩——成功率 99.8%，相當可靠了。但整條旅程要全綠，得**每一步都不出錯**，所以整體成功率是 0.998 自乘 25 次：

```
P(整條綠) = 0.998 ^ 25 ≈ 0.951
```

也就是說，即使每一步都有 99.8% 的可靠度，這條 25 步的 e2e 仍有約 **5% 的機率無端變紅**——不是因為你的程式碼錯了，而是因為某一步剛好撞上一次網路抖動、一次容器還沒暖好、一次非同步訊息晚到了 200 毫秒。現在把 e2e 的數量堆到 200 條，每條都是 25 步、各有 5% 的 flaky 率，那麼**每一次 CI 跑下來，期望有 200 × 0.05 = 10 條測試是紅的，而其中沒有一條對應到真正的 bug**。

這就是冰淇淋甜筒致命的地方。當一個套件每次跑都紅十條、而十條裡通常零條是真問題，團隊會學到一個理性但災難性的反應：**紅了就 retry，retry 兩三次總會綠**。一旦這個習慣養成，測試套件就**死了**——它再也無法區分「真的壞了」和「又抖了一下」，因為人們已經訓練自己無視它的紅燈。一個被習慣性 retry 到綠的 e2e 套件，提供的信心是零，甚至是負的（它讓你**以為**有人在守門）。金字塔之所以要把 e2e 壓到很少，不是因為 e2e 不好，而是因為 e2e 的 flaky 率會隨數量相乘放大，多到一個量級就會反過來吞掉整套測試的可信度。

## flaky 不是天氣，是 bug

順著上面的算術，要點破一個普遍的壞習慣：把 flaky 測試當成「背景噪音」、當成「今天天氣不好」來忍。

一個偶發變紅的測試，背後幾乎總有一個**真實的非決定性來源**：一個沒等夠的非同步操作（測試在資料還沒寫進去就去讀）、一個共用的測試資料庫被另一條平行測試動了手腳、一個依賴牆鐘時間的斷言在跨午夜時翻車、一個依賴 map 迭代順序的斷言在某次剛好換了順序。這些都不是「運氣」，是真實的競態或順序假設沒被守住——而且**同樣的競態，很可能也活在你的生產程式碼裡**。一個 flaky 測試在對你大喊「這裡有個時序假設不成立」，你卻把它當噪音 retry 掉了。

所以 flaky 要當 bug 修，不是當天氣忍：要嘛找出那個非決定性來源把它釘死（補上正確的等待條件、隔離共用狀態、把時間注入成可控的依賴），要嘛把這個測試從套件裡拿掉、別讓它繼續腐蝕大家對紅燈的信任。一個會說謊的告警，比沒有告警更糟——測試套件也一樣。

## 金字塔不是法律：當重心該往上移

金字塔常被當成鐵律，但它其實是**對某一類系統**的最佳解——重邏輯的系統。如果你的服務核心是一堆演算法、狀態機、計費規則、邊界條件，那麼「邏輯」才是 bug 的主要棲息地，unit test 的高密度密網就該鋪在那裡，金字塔的標準形狀完全正確。

但有一大類服務不長這樣：薄邏輯、重 I/O 的 CRUD 服務。它的程式碼掰開看沒幾行**自己的**邏輯——大部分工作是「收請求、查資料庫、塞進另一個格式、發出去」。對這種服務寫大量 unit test，你會發現自己在 mock 掉幾乎所有真正會出錯的東西（資料庫、序列化、外部呼叫），剩下被測的那點純邏輯薄到不值一測——你買到的是**假信心**：unit 全綠，但凡是會真的壞的接線一個都沒驗到。

針對這種系統，Kent C. Dodds 提出了**測試獎盃（testing trophy）**：把測試重心從 unit 那層**上移到 integration**。獎盃的論點很實在——「你的測試越像軟體真正被使用的方式，它給你的信心就越高」，而對一個薄邏輯服務來說，「真正被使用的方式」就是帶著真實資料庫跑一遍。獎盃也比金字塔多畫了一層放在最底下：**static**（型別檢查、linter），主張型別系統在你打字的當下就免費攔下一整類錯誤（typo、型別不符、null 解參考），是 CP 值最高的「測試」。

金字塔和獎盃不是對立的兩派要你選邊站，而是**同一條成本/信心軸上，對不同 I/O 比重的系統給出的不同重心**。重邏輯 → 重心在 unit（金字塔）；薄邏輯重 I/O → 重心在 integration（獎盃）。誰對，取決於你的系統把複雜度藏在哪。教條地照搬任何一個形狀，都是把答案抄來而沒做題。

順帶戳破一個和形狀糾纏在一起的迷思：**覆蓋率不等於信心**。行覆蓋率（line coverage）只回答一件事——「這一行在某個測試裡被執行過」，它**完全不保證**有任何斷言去驗證那行的結果是對的。一個跑遍所有程式碼、但斷言全是 `expect(result).toBeDefined()` 的套件，可以堂堂正正拿到 100% 覆蓋率，卻一個真 bug 都攔不下。80% 覆蓋率但每條路徑都驗了失敗分支的套件，遠勝 95% 覆蓋率但全是沒斷言的 happy path。追覆蓋率數字本身，是把手段誤當成目的——這也是 AI 大量生成測試時最容易掉進去的坑：覆蓋率漲得很漂亮，因為生出來的多是測 getter/setter、斷言寫成永真的湊數測試，數字升了、真實信心沒升，review 要盯的是斷言驗了什麼、不是行數蓋了多少。

## contract：填補 mock 與 e2e 之間那道致命的縫

現在來看那條斜出去的岔路，因為它解的問題，是微服務世界裡最隱蔽、也最常見的爆炸方式。

設想服務 A 要呼叫服務 B 的 API。在 A 的 unit test 裡，你把 B 整個 mock 掉——你寫一個假的 B，讓它回傳「A 預期 B 會回的東西」，然後測 A 在這個假回應下行為正確。一切很美好，A 全綠。問題是：**你那個 mock，是你對 B 行為的「假設」，而假設會悄悄過期。** 某天 B 的團隊把回應裡的 `userId` 改名成 `user_id`、把一個原本必有的欄位改成可選、或把某個錯誤的 HTTP 狀態碼從 400 改成 422——B 自己的測試全綠（它測的是 B 新的、正確的行為），A 自己的測試也全綠（A 還在對著那個**過期的 mock** 測）。兩邊各自一片綠，但它們對 API 的理解已經分岔了。這個分岔**只有在 A、B 真的兜在一起時才會炸**——也就是上線後。這就是 **mock 漂移**，微服務裡最典型的「各自綠燈、合起來爆炸」。

那為什麼不乾脆全跑 e2e，讓 A、B 真的兜在一起測？因為 e2e 的成本會**組合爆炸**。把這件事算到底，你就懂為什麼 contract 是這個縫的甜蜜點。

設想三個微服務串成一條鏈 A → B → C，每個服務各有 5 個還在線上、彼此要相容的版本（漸進式發布、未完成下線的舊版本，這在生產裡是常態）。要用 e2e 驗證「任意版本組合兜起來都對」，要驗的組合數是：

```
5 (A) × 5 (B) × 5 (C) = 125 種組合
```

每一種組合都得把三個服務都拉起來跑一遍完整旅程，假設一次 8 分鐘，125 種就是約 **1000 分鐘**的 CI 時間——而且前面算過，每條 25 步的 e2e 還各帶約 5% 的 flaky 率。這個數字會隨服務數量**指數成長**：再加一個有 5 版本的服務 D，組合就從 125 跳到 625。e2e 在這個維度上是**先天不可擴展**的。

**contract testing 把這個 125 的組合爆炸，拆成兩個可以線性處理的邊界。** 它的核心洞見是：A → B → C 這條鏈會不會出事，不取決於「125 種端到端組合」，只取決於**兩個介面**——A↔B 這個邊界、B↔C 這個邊界——各自是否相容。如果每個邊界都被獨立地釘住，整條鏈就是對的，不需要把整條鏈拉起來證明。

最主流的做法是 **consumer-driven contract（CDC，消費者驅動契約）**，事實標準工具是 **Pact**（2026-06）。它的機制值得一步步看清楚，因為這是它跟「在 e2e 裡兜起來」最本質的不同——**它讓雙方永遠不必同時在場**：

```
1. 消費端 A 寫一個「契約測試」：在 A 自己的 CI 裡，
   對著一個 Pact 提供的 mock server 發出 A 真正會發的請求，
   並宣告 A 期待收到的回應形狀（哪些欄位、什麼型別、什麼狀態碼）。
2. 這個測試跑完，Pact 產出一份 "pact file"——一個 JSON 文件，
   逐筆記下「A 會發這些請求、期待這些回應」。這份檔案就是契約。
3. 契約被推上一個 Pact Broker（契約登記中心）。
4. 供應端 B 在 B 自己的 CI 裡，從 Broker 抓下這份契約,
   把裡面每一筆「A 期待的請求」重放到「真實運行的 B」身上,
   驗證 B 真的回得出 A 期待的那種回應。
5. 若 B 改了 API 破壞了 A 的期待（漏了欄位、改了型別),
   B 的 CI 在第 4 步就直接紅——在合併前、在上線前。
```

關鍵在第 1 步和第 4 步**發生在不同的地方、不同的時間**：A 跑自己的 CI 時 B 根本沒上場（B 被一個由契約生成的 mock 替代），B 跑自己的 CI 時 A 也沒上場（A 只留下一份靜態的契約檔）。**兩邊各自獨立 CI、各自獨立部署，卻仍能在「B 不小心破壞了 A 的真實假設」的那一刻被打紅。** 這正好把前面 mock 漂移的洞堵死了——A 的 mock 不再是 A 單方面的猜測，而是一份 B 必須在自己的 CI 裡親自兌現的承諾。契約成了 mock 漂移的剎車：mock 還是 mock，但它背後有一份 B 簽了字、且 B 每次改動都要重新驗證的合約撐著。

## 契約測試怎麼壞、以及一個容易混淆的變體

契約測試解決一個真問題，但它本身也有幾個非顯然的邊界，講透了才算真懂這個機制。

第一個，也是最常被誤解的：**契約只驗「雙方對 API 形狀與語意的假設一致」，它不驗端到端的業務正確性。** 一份綠燈的契約保證「B 回得出 A 期待的欄位結構」，但它**不保證**「B 算出來的折扣金額在業務上是對的」、不保證「A 拿到回應後的後續流程跑得通」。把契約當成 e2e 的替代品，是它最危險的誤用——它填的是 unit 和 e2e 之間那道「介面假設」的縫，不是 e2e 那塊「整條旅程都對」的地。兩者互補，誰也取代不了誰：你仍然需要少數幾條 e2e 守黃金路徑，只是不再需要 125 條。

第二個邊界藏在治理裡：**契約會漂移，而且維護一個 Pact Broker 是有成本的。** 當你有幾十個服務、上百份契約，「現在這個版本的 A，可以安全部署到正在跑著 B-v3、C-v5 的 production 嗎？」這個問題會變得難以心算。Pact 對此的答案是一個叫 **matrix（矩陣）** 的東西——記錄「每一對消費端版本 × 供應端版本」是否驗證通過——以及一個叫 **can-i-deploy** 的工具：部署前先問 Broker「我這個版本，跟目前所有環境裡正在跑的、會跟我互動的服務版本，全都驗證過相容了嗎？」得到綠燈才放行。沒有這層治理，光有契約檔本身，會在版本一多時退化成又一堆沒人維護的測試。契約測試買到的「獨立部署的信心」，是要靠 Broker 這套版本記帳機制持續餵養的，不是裝個 Pact library 就免費到手。

第三個，是一個近年才分化出來、極容易和 CDC 混為一談的變體：**bi-directional contract testing（BDCT，雙向契約測試）**。傳統 CDC 是「消費端先寫期待、生成契約，供應端去兌現」——它要求供應端在自己的 CI 裡跑契約驗證、需要雙方都接入這套流程。BDCT 走的是另一條路：供應端**獨立地**發布一份它的 OpenAPI 規格（描述它實際提供什麼），消費端**獨立地**發布一份它依賴的子集，然後由一個能做**靜態規格比對**的 Broker 去判定這兩份規格相不相容——供應端完全不必跑任何契約測試，只要交出它本來就該維護的 API 規格。（這裡要分清：BDCT 是 **PactFlow**（SmartBear 的商用 Broker）的專屬能力，前面那套消費端跑、供應端兌現的 CDC 流程才是開源 Pact Broker 支援的；自架的開源 Pact Broker 目前（2026-06）不做 BDCT 的靜態規格比對。）這在「供應端已經有一套完整的 OpenAPI、不想為了消費端再加一套驗證流程」或「跨團隊溝通不順、拿不到供應端內部細節」時特別合用。兩者解的是同一個「不必兜起來就驗介面相容」的問題，但分工不同：CDC 要供應端主動兌現消費端的期待，BDCT 只要雙方各自交規格、讓 Broker 當裁判。把 BDCT 的省力誤當成「CDC 過時了」，會選錯工具——它們是針對「雙方協作緊密度」這個變數的兩個座標，不是新舊替代。

## 不是只有這四層：兩種驗「測試本身」的測試

到目前為止談的都是「用測試驗程式碼」。但有兩類技術跳出這個框，值得知道它們存在、以及各自釘住的是什麼角度，因為它們補的是前面四層都看不見的盲區。

**property-based testing（性質測試）**翻轉了「測試」的寫法。一般測試是你手寫幾個案例：「輸入 3，期待 9」。性質測試讓你改為宣告一條**對所有合法輸入都該成立的性質**——「對任何字串 s，`decode(encode(s))` 都應等於 s」、「對任何數列，排序後的結果長度不變、且每個相鄰元素前 ≤ 後」——然後框架自動生成成百上千組輸入去**找反例**。它特別擅長那些案例你想不全的地方：解析器、編解碼器、純函式的邊界。它的代價是「想出一條好的性質」比「寫一個案例」難得多；好處是當它找到反例時，會自動把反例**收縮（shrink）**到最小的那個——不是丟給你一個 5000 字的亂數字串，而是收斂到「就是空字串會炸」這種一眼看穿的最小案例。

**mutation testing（突變測試）**測的不是你的程式碼，而是**你的測試本身有沒有用**。它的手法很賊：故意把你的程式碼改壞——把 `<` 改成 `<=`、把 `&&` 改成 `||`、把 `return x` 改成 `return 0`，每改一處叫一個 mutant——然後跑你的測試套件，看它**抓不抓得到**。如果你的測試在程式碼被改壞之後**依然全綠**，那這個 mutant「存活」了，意味著你那行覆蓋率底下根本沒有有效的斷言在守——這直接戳破了覆蓋率的謊言：覆蓋率說「這行被執行過」，mutation testing 說「這行被改壞了卻沒人發現」。它是覆蓋率數字的測謊機。代價是**極慢**（每個 mutant 都要重跑一次相關測試，n 個 mutant 就是 n 倍），所以它不進每次 CI，而是當作定期的「測試套件健康體檢」偶爾跑一次。

這兩者和金字塔不在同一個維度：金字塔分的是「讓多少真實東西參與」，這兩者分的是「從哪個角度逼出 bug」——一個用海量自動輸入逼，一個用改壞程式碼逼。它們是策略工具箱裡的特化武器，不是日常每層都要鋪的東西。

## 為什麼是這個形狀

退一步看，整個測試策略的樣貌，都是從一個無法迴避的張力裡長出來的：**真實度和成本，在這條軸上永遠同向移動。** 你想要的信心（這東西在生產真的能跑）來自真實度，而真實度必然帶著速度、穩定、定位這三項成本一起漲。沒有任何一層測試能同時又快又穩又真實又好定位——如果有，我們只寫那一種就好了。

正因如此，理性的策略不是「全押最真實的那層」（那是冰淇淋甜筒，會被 flaky 的相乘放大吞掉），也不是「全押最便宜的那層」（那是一堆 mock 出來的假信心，會在 mock 漂移那天集體破功）。理性的策略是**在這條軸上鋪一個有層次的組合**：用大量便宜的下層測試密密地網住邏輯，用適量的 integration 釘住接線，用斜出去的 contract 在每個服務邊界釘住介面假設、把組合爆炸拆成線性，最後只留幾條最貴的 e2e 去守那幾條最不能斷的黃金路徑。

每一層都在用一種特定的成本，買一種特定、且邊界清楚的信心。把它們的成本與信心邊界看清楚，你就不會再問「覆蓋率夠不夠」這種把手段當目的的問題，而會問那個真正的問題：**我這次想送出去的這個改動，最可能在哪裡壞、我有沒有用最便宜的那層測試把那個地方釘住了。** 一支綠燈套件的價值，從來不在它有多綠，而在它紅的時候，紅得有沒有道理。
