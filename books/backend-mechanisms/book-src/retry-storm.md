# retry storm 與 metastable failure

凌晨三點，一個資料庫副本因為例行的 GC 暫停卡了 800 毫秒。這在正常的日子裡什麼事也不會發生——幾筆查詢慢了一下，逾時的少數請求被客戶端重送一次，成功，沒人會注意到。但這一晚不一樣：流量正好在尖峰，那 800 毫秒讓一批查詢同時越過了逾時門檻，於是它們同時觸發重試。重試打在那台還沒從 GC 喘過氣的副本上，把它推得更慢，更多查詢逾時，更多重試湧入——十幾秒內，一台原本只是打了個盹的機器，被它自己服務的客戶端活活壓垮。

最詭異的部分在後面。值班工程師很快定位到那次 GC，調整了參數、確認它不會再發生。可是服務**沒有恢復**。流量明明已經降回故障前的水準，CPU 卻還是滿的，延遲還是爆的，錯誤率還是高得嚇人。團隊盯著監控板困惑了二十分鐘：「觸發的那件事已經修好了，為什麼系統還在地上？」最後是有人把入口流量完全掐斷、讓系統把積壓排空、再緩慢放流量回來，它才終於站起來。

這一幕裡藏著兩個彼此糾纏、卻常被混為一談的東西。一個是 **retry storm（重試風暴）**：重試把一個失敗放大成許多失敗的正回饋。另一個更深、更反直覺：**metastable failure（亞穩定故障）**——系統被某個觸發事件踹進一個「會自我維持的壞狀態」，即使觸發早就消失，它也卡在那裡出不來。前者是後者最常見的引擎。要真正讀懂這一晚為什麼會發生、又為什麼修了觸發點還是好不了，得把這兩件事分開、一層層拆開看。

## 重試的數學是乘法的

先把退避、jitter、退避上限這些「怎麼重試才對」的具體做法放到一邊（那是重試本身的主題，見〈重試、退避與 jitter〉）。這一章只盯著一個更底層、也更危險的性質：**重試對下游負載的影響是乘法的，不是加法的。**

正常運作時，重試是一張溫和的安全網。少數請求因為偶發抖動失敗了，重送一次就成功，下游幾乎感覺不到差別——因為「失敗的請求」只是總流量裡的一小撮，補送那一小撮，負載只多了幾個百分點。重試在這個區間裡是純粹的好事：它把使用者看到的錯誤率往下壓，而代價小到可以忽略。

問題出在另一個區間。當下游**整體**變慢——一次 GC、一次部署、一次相依服務抖動——大批請求會在同一時間越過逾時門檻。注意這個「同時」：它們不是零星地、分散地失敗，而是成群地失敗。如果每個失敗的請求觸發 `k` 次重試，那麼在這一瞬間，下游收到的流量被乘上了 `(1 + k)` 倍。而下游正是**因為流量太大才變慢的**。於是一條閉環合上了：

```
下游變慢  ->  大批請求逾時  ->  每個逾時觸發 k 次重試
   ^                                      |
   |                                      v
流量更大  <-----------------------  下游收到 (1+k)x 流量
```

這就是為什麼同一個重試機制，在兩個區間裡是兩種完全相反的東西。在「常態的少量失敗」區間，它讓系統更可靠；在「過載的大量失敗」區間，它把小故障放大成大故障。「加重試讓系統更穩」這句話只對了一半——它對的是前半個世界，錯的是後半個。一個沒想清楚這件事的系統，等於在自己最脆弱的時刻，給自己裝了一個流量放大器。

## 放大為什麼會層層相乘

把單層的 `(1 + k)` 倍放到真實的微服務拓樸裡，事情會更糟，因為放大會**沿著呼叫鏈相乘**。

設想一條再普通不過的呼叫鏈：使用者打到服務 A，A 呼叫 B，B 呼叫 C，C 查資料庫。一個看似人畜無害的「預設」散佈在每一層——每層都各自配了「失敗就重試 3 次」（也就是每個請求最多送 4 次）。平時這完全合理，每一層的作者都覺得自己在做對的事。

現在最底層的 C 短暫過載，開始逾時。算一下一個使用者請求會在 C 身上放大成幾次呼叫：

- C 自己對每個進來的請求被重試，乘 4。
- B 看到它送給 C 的請求失敗了，於是把「B→C 的這次呼叫」整個重試——B 的每一次嘗試又各自在 C 身上展開最多 4 次，所以再乘 4。
- A 看到 B 失敗，重試 B，又乘 4。

最壞情況下，使用者按下去的**一個**請求，在最底層變成 `4 × 4 × 4 = 64` 次打在 C 上的呼叫。C 本來只是稍微過載——也許只超出容量 20%——卻被瞬間灌進 64 倍的重試流量，沒有任何系統能扛住這種乘數。它徹底崩潰，並把整條鏈拖下水。

這個 64 不是危言聳聽的極端值，它就是「每層各自重試」這個在程式碼審查裡看起來無比合理的決定，乘起來的必然結果。每一層的工程師都只看到自己那一層的 `4`，沒有人看到那個乘積。修法也清楚得近乎冷酷：**重試只能發生在一個地方**——通常是最接近使用者、或最有上下文判斷「這次失敗值不值得重試」的那一層；其餘各層遇到失敗就直接 fail-fast 往上拋，把判斷權留給那唯一一層。這一刀下去，總放大倍數從 64 壓回 4。跨層之間要傳遞的不是「各自重試」，而是 deadline 或剩餘預算（見〈timeout、deadline 與 budget〉），讓下游知道「這個請求還剩多少時間值得做」，而不是各自悶頭重送。

## 把放大鎖死：retry budget

把放大倍數從 64 壓回 4 之後，那個 4 仍然是個乘數——在 C **完全掛掉**的時候，每個請求還是會白白多送 3 次，等於在屍體上補刀。我們想要的是一種更聰明的重試：常態下大方地重試（因為那是安全網），下游一旦真的撐不住就**自動把重試收斂到趨近於零**。這正是 **retry budget（重試預算，也叫 retry token bucket）** 在做的事。

機制本身樸素到優雅。給客戶端對某個下游維護一個令牌桶：

- 每一次**成功**的呼叫，往桶裡滴入一小撮令牌（例如 0.1 個）。
- 每一次**重試**，從桶裡扣掉一個令牌。
- 桶空了，就**不再重試**——失敗直接往上拋。

這個設計的妙處，全在它讓重試率自動綁定到「成功流量」上。常態下絕大多數呼叫成功，桶被穩定地餵飽，偶發失敗想重試隨時有令牌可用，安全網功能完好。可是當下游整體故障、成功率崩到接近零，補令牌的來源也跟著乾涸——桶很快見底，重試率自動趨近零。換句話說，**正是在最該停止重試的時刻（下游全掛、重試只會火上加油），預算機制讓重試自己停了下來**，不需要任何人去判斷「現在該不該關掉重試」。它把「重試的數學」從一個失控的乘法，重新鎖回「不超過成功流量的一個小比例」這個有界的範圍。

這不是紙上談兵的設計。gRPC 的 retry 規格（proposal A6）就把它做成標準的 `retryThrottling` 設定，欄位只有兩個（2026-06 查證自官方文件）：

```json
"retryThrottling": {
  "maxTokens": 10,
  "tokenRatio": 0.1
}
```

客戶端對每個 server 維護一個 `token_count`，初始為 `maxTokens`。**每次可重試失敗的 RPC 扣 1，每次成功的 RPC 加 `tokenRatio`**（規格上只把「以可重試／non-fatal 狀態碼失敗、或收到 pushback 要求別重試」的 RPC 計為失敗，硬失敗不扣）。關鍵在一個刻意設計的門檻：**當 `token_count` 掉到 `maxTokens` 的一半（含半線）或以下，重試就被暫停**，直到令牌補回來。注意這裡的不對稱——失敗一次扣的（1）遠大於成功一次補的（0.1），所以一旦失敗開始成串地出現，桶會迅速失血、掉破半線、把重試整個關掉；而要重新開啟重試，得累積一長串成功把令牌補回半線以上。這個不對稱不是隨意挑的，它就是「壞起來要快、好起來要穩」這條防雪崩直覺的數字化身。

## 退避了，卻沒有 jitter

退避（backoff）是另一個常被誤以為「裝了就安全」的東西。它確實必要——讓重試一次比一次間隔更久，給下游喘息——但單靠退避，會留下一個隱蔽的破口。

問題在於那批請求是**同時**逾時的。如果它們退避同樣的時間（比如「失敗後等 1 秒再試」），那麼它們會在 1 秒後**同時**重試。退避只是把那道重試的尖峰整體往後挪了 1 秒，尖峰本身的高度一點沒削平。下游在 T+1 秒迎來和 T 時刻一樣猛的一波同步衝擊，於是再次被打趴、再次集體逾時、再在 T+2 秒同步撞回來——退避反而把混亂變得有節奏地週期化了。這個現象有個傳神的名字：thundering herd（驚群）。

補法是給退避時間加上**隨機 jitter**：每個客戶端退避的不是固定的 1 秒，而是「0 到某個上限之間隨機取一個值」。本來擠在同一個時間點的那一大坨重試，被隨機地攤平到整段視窗裡，下游收到的從「一道道同步的牆」變成「一條平穩的細流」。AWS 的工程實踐把這條講得很白：full jitter 把重試均勻散佈到整個退避視窗，讓服務看到的是穩定的涓流而非同步的尖峰。退避決定「重試的平均速率」，jitter 決定「重試攤不攤得開」——兩者解的是不同的問題，少了任何一個都不完整。

## 真正反直覺的部分：metastable failure

到這裡，retry storm 還只是一個「放大」的故事：流量被乘大、系統被壓垮。如果故事到此為止，那麼修法應該很簡單——移除觸發點（修好那次 GC），流量降回常態，系統就該自己復原。

但凌晨三點那一幕告訴我們：**它不會。** 這才是真正需要花力氣理解的地方，也是 Bronson 等人在 2021 年 HotOS 那篇〈Metastable Failures in Distributed Systems〉、以及隨後 OSDI 2022〈Metastable Failures in the Wild〉所釘下的核心概念。

關鍵是把一次故障拆成兩個獨立的角色：

- **觸發（trigger）**：把系統推進壞狀態的那個事件。它可能是 load-spiking（流量突然衝高）或 capacity-decreasing（一台機器掛了、容量突然變小）——這次是後者（GC 讓一台副本暫時失能）。觸發通常是短暫的。
- **sustaining effect（維持效應）**：讓系統**留在**壞狀態的那個正回饋迴圈。在這一晚，它就是重試風暴——延遲高 → 逾時 → 重試 → 負載更高 → 延遲更高，一個自我餵養的閉環。

這個拆分解開了那個二十分鐘的困惑。觸發和維持效應是**兩件不同的事**：你修好了觸發（GC 不會再來），但維持效應（重試迴圈）已經接管，它不需要原來的觸發也能自己轉下去。系統的「病因」不是那次 GC，而是那個迴圈——GC 只是按下開關的手指，手指早就移開了，但開關已經卡在「開」。

論文把這件事描述成系統的三種狀態。常態下系統在一個**穩定狀態**運作，負載低、有餘裕，偶發抖動會被自然吸收掉。但當利用率被推高，系統會滑進一個**脆弱狀態（vulnerable state）**：表面上還在正常服務，但已經沒有餘裕去吸收任何衝擊——這時一個夠大的觸發，就會把它踹進**metastable failure 狀態**，正回饋迴圈全面啟動，系統穩定地、持續地待在低 goodput 的壞狀態裡。「metastable」這個詞借自物理：它是一種**亞**穩定——看似停在一個窪地裡不動，但那是個高位的窪地，要跳回真正的谷底，得先翻過一道牆。

## hysteresis：為什麼降回原點沒用

那道牆，就是這整章最值得記死的概念：**hysteresis（遲滯）**。

遲滯的意思是，把系統推進壞狀態所需的負載，和把它拉出壞狀態所需的負載，**不是同一個點**。想像一條負載對狀態的曲線：當負載往上爬、越過某個臨界點 `L_trigger`，系統翻進 metastable 故障。但此刻你把負載降回 `L_trigger` 以下、甚至降回故障發生前的水準，系統**不會**翻回來——因為維持效應（積壓的重試、被打爆的佇列、被污染的快取、滿載的連線池）仍在自我餵養，它把系統死死按在壞狀態裡。要真正復原，你得把負載降到一個**更低**的點 `L_recover`，低到讓那個正回饋迴圈再也餵不飽自己、終於斷掉。`L_recover` 和 `L_trigger` 之間那段差距，就是遲滯，也就是那道牆的高度。

```
            L_recover        L_trigger
                |                |
  good state    |   bistable     |   metastable state
  <-------------+----------------+------------------->
                ^                ^
       降到這裡才彈回好狀態   升過這裡才掉進壞狀態
```

這就是為什麼「流量明明已經正常了，系統卻還在地上」。團隊直覺以為負載對狀態是條單行道——升上去會壞，那降回來就會好。但有遲滯的系統不是單行道，它有記憶：它記得自己正在掙扎，而那份掙扎本身（重試、積壓）就是讓它繼續掙扎的燃料。降回 `L_trigger` 落在那段中間的雙穩區裡，好壞兩個狀態都穩定，系統會固執地待在它現在所在的那個（壞的）。

於是復原手段也跟著反直覺：你不能溫和地等它自己好，你得**比觸發點更猛地**降載——把入口流量完全掐斷或大幅 load shedding（見〈load shedding：主動丟掉一部分流量〉），讓佇列徹底排空、讓重試找不到東西可放、讓快取有機會重建，把系統強行壓到 `L_recover` 之下，它才會「啪」地一聲彈回好狀態。然後再緩慢地、一點一點地把流量放回來。放太快會立刻把它重新推過 `L_trigger`，前功盡棄。凌晨那位工程師掐斷流量、排空、緩放——他不是在亂試，他是在翻那道遲滯的牆，儘管他當下未必說得出這個詞。

## goodput 歸零：忙得要死卻什麼也沒做成

維持效應為什麼這麼難打破？因為陷進去之後，系統會把資源花在一種特別惡毒的浪費上——做**注定沒有用的工作**。

設想客戶端的逾時設成 100 毫秒。系統過載後，請求在佇列裡排隊的時間早就超過 100 毫秒，意思是：**等這個請求終於排到、被處理完，發出它的客戶端早在好幾十毫秒前就逾時離開、甚至已經重送了新的一份。** 系統吭哧吭哧算完的這個結果，沒有任何人在等它了——它是純粹的白工。更糟的是，那個逾時離開的客戶端送來的「新的一份」，又排進了同一條已經滿到溢出的佇列，繼續排隊、繼續注定逾時、繼續變成白工。

這裡要分清兩個常被混用的詞。**throughput（吞吐）**是系統每秒處理完的請求數——過載時這個數字可能還很漂亮，CPU 滿載、機器很忙。**goodput（有效吞吐）**是其中「真的有人在等、真的有用」的那部分——過載時它崩向零。系統忙得要死，goodput 卻趨近於零，因為它 100% 的力氣都花在處理已經沒人要的屍體請求上。這個 goodput 的崩塌，正是維持效應的燃料：它讓系統永遠騰不出餘裕去追上、去把佇列排空，因為每一分算力都被白工吃掉了。

看清了這一點，前面那些防禦的真正作用就浮現了。**load shedding 在進佇列時就丟掉「預估等待已超過 deadline」的請求**——因為做了也是白工，不如連算都別算，把算力留給還來得及的。它不是在拒絕服務，它是在拒絕做白工，好讓 goodput 從零爬回來（goodput 崩塌與卸載的算術，見〈load shedding：主動丟掉一部分流量〉）。**circuit breaker 在偵測到下游故障率過高時直接跳閘**（見〈circuit breaker：三態與半開試探〉），把對一個已知掛掉的下游的所有重試整批掐斷，不再餵養那個迴圈。這些機制看似各自獨立，骨子裡都在做同一件事：**找到那個正回饋迴圈，把它斷掉。**

## 加機器為什麼救不了

陷進 metastable 故障時，最自然、也最致命的直覺反應是：「容量不夠嘛，加機器。」

這個直覺在「單純容量不足」的世界裡是對的，但 metastable 故障不是那個世界。問題不在容量的絕對值，而在那個還在轉的正回饋迴圈。你新加的容量一上線，會立刻被放大的重試流量吃掉——還記得那個 64 倍嗎？只要迴圈沒斷，新機器面對的不是真實負載，而是**被放大過的**負載，它撐不了多久也會被拖進同一個壞狀態，然後你的故障域只是從幾台變成更多台，一起躺平。

正確的次序是反過來的：**先掐斷迴圈，再談擴容。** 先 load shed、先跳閘、先把重試壓下去，讓系統脫離 metastable 狀態、goodput 回升；等它站穩在好狀態了，這時若真的是容量不足，加機器才有意義——因為此刻新容量面對的是真實負載，不是被乘大的幻影。把次序搞反，是事故覆盤裡反覆出現的、代價高昂的錯。

順帶一個同源的陷阱：**重啟**。陷進去之後重啟服務常常「看起來有效」——但那往往不是因為重啟修好了什麼，而是因為重啟的那段時間恰好等同於一次徹底的 load shedding（行程不在、流量被擋在外面、佇列被清空），把系統意外地壓到了 `L_recover` 之下。理解了遲滯，你會知道真正起作用的是「清空 + 緩放」這個動作，而不是「重啟」這個儀式——下次你可以更精準、傷害更小地做到同一件事，而不必把整個行程殺掉重來。

## 一個藏在重試語意裡的陷阱

最後一個非顯然的邊界，藏在「逾時就重試」這個動作本身的語意裡，它和 retry storm 的放大效應疊在一起會特別陰險。

逾時**不等於失敗**。當你的請求逾時，下游可能根本沒收到（那重試是對的），但也可能**已經收到、已經做完、甚至已經把成功的回應送出來了，只是回應在回程上慢了或丟了**——這種情況下，你以為的「失敗」其實是一次成功，而你的重試是在讓同一件事**再做一次**。如果那件事不是冪等的（見〈重複是常態：冪等與去重〉），這次重試就是一筆重複的扣款、一封重複的信、一個重複的訂單。

把這個語意放回 retry storm 的脈絡裡，它的破壞力被放大了：在過載時，逾時大量發生，而其中相當一部分其實是「下游做完了、只是慢」的偽失敗。你的重試不只在放大下游的**負載**，還在放大下游的**重複工作**——它得花額外的力氣去做那些它其實已經做過的事，這讓它更慢，又觸發更多逾時、更多重試。冪等與去重在這裡不是一個可有可無的「資料正確性」考量，它是 retry storm 防禦的一部分：唯有當重試是冪等的，你才敢在過載時安心地讓它重送，而不必擔心「重送本身」又給下游加了一層重複的負擔。一個不冪等的系統，連重試都不敢放心開——而不敢開重試，又會在常態的偶發失敗時失去那張安全網。這兩難正是「重試」與「冪等」必須成對出現的原因。

## 為何是這個形狀

退一步看，retry storm 和 metastable failure 的整個樣貌，都是從一條樸素的不等式長出來的：**在一個有共享下游、又有正回饋迴圈的系統裡，局部的慢會餵養更多的慢。**

正因為放大是乘法的、還會沿呼叫鏈相乘，所以重試必須限次、必須集中在單一層、必須用預算把它鎖回成功流量的一個小比例——這些不是繁文縟節，是在跟那個乘數搏鬥。正因為退避只挪動尖峰而不削平它，所以非加 jitter 不可。正因為維持效應一旦接管就不需要原觸發、且有遲滯，所以修觸發點不會讓系統復原，你得更猛地降載去翻那道牆。正因為過載時系統會把算力燒在沒人要的白工上、goodput 崩向零，所以 load shedding 與 circuit breaker 的真正使命不是「拒絕服務」，而是「斷掉那個迴圈、把算力還給有用的工作」。正因為加機器只會餵養還在轉的迴圈，所以次序永遠是先掐迴圈、再談容量。

這些防禦看起來是一籮筐互不相干的招式——限次、退避、jitter、retry budget、斷路器、卸載、冪等——但它們其實都在回答同一個問題：**這個系統裡，有沒有一個會在它變慢時反而增加它負載的正回饋迴圈？如果有，它在哪，怎麼把它斷掉？** 凌晨三點那台只是打了個盹的副本，最後不是被流量壓垮的，是被它自己服務的客戶端的善意——那些「失敗了就再試一次」的善意——壓垮的。理解 metastable failure，本質上就是學會在系統的善意裡，認出那個會反噬的迴圈。
