# 跨服務的交付一致：outbox 與 saga

一筆下單請求進來，你的服務做了兩件再自然不過的事：在自己的資料庫裡把訂單狀態從 `pending` 改成 `paid`，然後往訊息佇列丟一則 `OrderPaid` 事件，讓下游的物流、通知、對帳服務各自接手。程式碼讀起來像這樣：

```
db.update("UPDATE orders SET status='paid' WHERE id=?", orderId)
broker.publish("OrderPaid", orderId)
```

兩行，先後執行，看起來無懈可擊。但這兩行之間有一道你看不見的裂縫——它們分屬兩個**獨立的系統**，一個是你的資料庫，一個是 broker，**沒有任何一個交易能同時罩住它們**。於是只要在這兩行之間的某個瞬間，行程崩潰、網路斷掉、broker 剛好不可用，你就會掉進兩種互斥的災難其中之一。

第一行成功、第二行還沒跑，行程就死了：資料庫裡訂單已經是 `paid`，但 `OrderPaid` 事件**從來沒有發出去**。物流不知道要出貨，通知不知道要寄信，對帳那邊看到一筆「付了錢卻沒有任何後續動作」的孤兒訂單。錢收了，貨永遠不會動。

把順序倒過來也不會更好。先發事件、再改資料庫：事件出去了，下游開始出貨了，結果 `UPDATE` 因為一個約束衝突或連線中斷而失敗——現在你的資料庫說這筆訂單還是 `pending`，但全世界其他服務都以為它已經付清、貨都在路上了。一個從未真正發生的事件，污染了整條下游。

這就是 **dual-write 問題**：一筆業務需要同時改動「自己的狀態」和「外界的世界」，而這兩次寫入無法被綁成一個原子操作。它不是一個邊角 case，它是事件驅動架構發件端的**結構性裂縫**——任何「改完 DB 再發訊息」的服務都站在這道裂縫上，只是大多數時候運氣好，那兩行之間的窗口太短，崩潰沒剛好落在那裡。但「大多數時候沒事」不是一個保證，它只是一個還沒被觸發的故障。

## 為什麼不能用一個交易把它們綁起來

第一個會冒出來的念頭是：那就用一個分散式交易把資料庫和 broker 一起提交不就好了？兩階段提交（2PC）正是為這個設計的——一個協調者先讓所有參與者 `prepare`，全部說好之後再一起 `commit`，要嘛全成、要嘛全不成。

問題是這條路在現實裡幾乎走不通，原因有兩層。第一層是**能力**：絕大多數現代 broker 根本不實作 XA 交易協定。Kafka 的線協定裡沒有 XA 的 prepare/commit 概念，你想把一次 publish 掛進一個跨資源的兩階段交易裡，它接不住。第二層更根本，是**代價**：2PC 是一個阻塞協定，協調者在整個交易期間握著所有參與者的鎖，任何一個參與者卡住或崩潰，其他人就得跟著乾等——你等於把資料庫的可用性綁定到 broker 的可用性上，兩個系統的故障機率不再獨立，而是相乘疊加。為了讓兩次寫入原子化，你買來一個更脆、更慢、更容易整條一起倒下的系統。這個取捨在絕大多數場景都不划算（2PC 的阻塞代價值得單獨細看，見〈2PC 與分散式交易：原子提交與它的阻塞代價〉）。

所以方向得反過來想。既然「跨兩個系統的原子提交」這麼貴又這麼脆，那就**別跨系統**——把問題從「兩個資源的交易」硬生生壓回「一個資源的交易」。這就是 outbox 的全部洞見。

## outbox：把訊息當成一筆資料，和狀態寫進同一個交易

關鍵的轉念是：你**真的需要**在那一刻把訊息送到 broker 嗎？不需要。你真正需要保證的，只是「訂單改成 paid」和「OrderPaid 這則訊息該被送出」這兩件事**同生共死**——要嘛都記下來，要嘛都沒發生。至於訊息**何時**真正離開你的機器去到 broker，可以晚一點、可以非同步。

於是 outbox 這樣做：在你的資料庫裡建一張 `outbox` 表。處理下單時，把業務狀態的更新，和「我要發一則 OrderPaid 事件」這個意圖，**寫進同一個本地交易**：

```
BEGIN;
  UPDATE orders SET status='paid' WHERE id=?;
  INSERT INTO outbox (id, topic, payload, created_at, published)
    VALUES (?, 'OrderPaid', ?, now(), false);
COMMIT;
```

現在那道裂縫不見了。這兩個 `INSERT`/`UPDATE` 落在**同一個資料庫**的同一個交易裡，由資料庫本身的 ACID 原子性兜底——交易提交，兩者都在；交易回滾，兩者都不在。**訊息被記下，若且唯若業務狀態被改**。dual-write 的兩個獨立寫入，被收斂成了一個資源內部的單一原子寫入。微服務社群把這句話講得很乾淨：「訊息被送出，若且唯若資料庫交易提交。」

但訊息現在只是躺在 `outbox` 表裡，還沒到 broker。誰把它送出去？一個獨立的角色，叫 **message relay**（投遞器）。relay 不參與業務交易，它只做一件事：把 `outbox` 裡 `published=false` 的訊息撈出來，投到 broker，投成功了再把它標記為已發。業務寫入和真正的投遞，被解耦成兩個獨立的階段——前者保證「不丟」，後者負責「送達」。

## relay 怎麼把訊息撈出去：兩條路

relay 有兩種長法，差別在「它怎麼知道 outbox 裡有新訊息」。

**第一條是輪詢（polling publisher）。** relay 開一個迴圈，定期去 `SELECT ... WHERE published=false ORDER BY created_at LIMIT n`，撈一批沒發過的，逐筆投到 broker，投成功就 `UPDATE ... SET published=true`（或直接刪掉）。它的好處是**樸素到不依賴任何資料庫特性**——任何能跑 SQL 的庫都能這樣搞，沒有額外的基礎設施要維運。代價是延遲：訊息得等到下一次輪詢才會被看見，輪詢間隔就是你的尾延遲下限。而且每一次輪詢都是一次查詢，輪詢越密、DB 壓力越大，這之間有個你得自己拿捏的取捨。

輪詢還藏著一個並發陷阱，值得拆開看。如果你為了吞吐跑了**多個 relay 實例**，它們會同時 `SELECT` 到同一批 `published=false` 的訊息，然後各自投一遍——同一則訊息被送了好幾次。要堵這個洞，標準做法是讓撈取的 SQL 帶上 `FOR UPDATE SKIP LOCKED`：每個 relay 在交易裡鎖住自己撈到的那幾列，`SKIP LOCKED` 讓別的 relay 自動跳過已被鎖的列去撈下一批。於是多個 relay 可以安全地平行跑，各認領各的訊息，不會踩到同一則。沒有這道鎖，多實例的 outbox relay 反而成了重複的放大器。

**第二條是 log tailing（交易日誌尾隨），也就是 CDC。** 與其反覆去問資料庫「有新的嗎」，不如直接讀資料庫的預寫日誌（write-ahead log，見〈WAL：先寫日誌，再改資料〉）——每一筆 commit 都會在 WAL 留下痕跡，一個像 Debezium 這樣的工具尾隨這條日誌，把 `outbox` 表的每一筆 `INSERT` 即時變成一道事件流，推到 broker。它幾乎沒有輪詢延遲，也不對資料庫加查詢壓力，因為它讀的是本來就要寫的日誌。代價是你得架一套 CDC 管線並為它的運維負責（CDC 的機制細節，見〈CDC：從資料庫變更長出事件流〉）。

這裡有個關於**順序**的微妙保證，值得講清楚。CDC 尾隨 WAL，WAL 的順序就是交易提交的順序，所以 CDC **能保證 outbox 訊息照「被寫進去的順序」流出**。但這個順序只保到 broker 的入口——一旦進了 broker，順序只在**單一 partition 內**成立（訊息順序的完整機理，見〈訊息順序：FIFO 與 partition key〉）。如果你的訊息散到多個 partition，跨 partition 之間就沒有全域順序了。所以即使用 CDC，你仍得用 partition key（例如以 `order_id` 雜湊）把同一筆業務實體的事件釘到同一個 partition，順序才在「真正重要的粒度」上被保住。

兩條路的選擇其實很好判：先用輪詢。它簡單、可觀測、沒有額外的基礎設施債，足以應付絕大多數吞吐。只有當你真的撞到「每秒上千則事件」或「延遲 SLA 卡得很死」的牆，才值得把 CDC 那套維運成本扛起來。

## outbox 買到的是「不丟」，不是「只一次」

這是 outbox 最常被誤讀的地方，得用最直白的話釘死：**outbox 給你的是 at-least-once（至少一次），不是 exactly-once（恰好一次）。**

看 relay 的那個動作序列就懂為什麼。relay 把訊息投到 broker、broker 回了 ack，relay 接著要去 `UPDATE outbox SET published=true`——如果 relay 偏偏在「broker 已收到」和「published 已標記」這兩步**之間**崩潰了呢？訊息明明已經送出去了，但 outbox 裡那一列還是 `false`。relay 重啟，再撈一次，看到這列還沒標記，於是**再投一次**。同一則訊息，下游收到兩遍。

這個窗口無法消除，因為「broker 收到」和「我記下它收到了」又是兩個系統、又是一次 dual-write——outbox 把業務側的 dual-write 解掉了，但 relay 投遞這一段，本質上仍是一次跨系統操作，仍然只能做到 at-least-once。你沒有把不可能變可能，你只是把不一致從「可能丟訊息」搬到了「可能重複訊息」——而重複，是一個遠比丟失溫和、可治的病。

治法是把「去重」的責任明確交給下游：每則事件帶一個穩定的唯一 ID，消費端在真正處理前先查一張去重表（常稱 inbox），這個 ID 見過就直接回 ack、不重做（冪等與去重的完整機制，見〈重複是常態：冪等與去重〉）。outbox 負責「狀態改了就一定有訊息出去」，下游的冪等負責「同一則訊息來幾次都只生效一次」，兩者合起來，才湊出端到端「每筆業務剛好被處理一次」的**效果**——而這正是 exactly-once 在現實裡唯一站得住的形態：傳輸層 at-least-once，加上處理層去重。任何宣稱「我用了 outbox 所以 exactly-once」的說法，都漏掉了下游那半邊。

還有一個 outbox 自己的維運故障模式：那張表只進不出。每筆業務都往裡塞一列，relay 即使標記了 `published=true` 也只是改個欄位、列還在。不清理，這張表會無限膨脹，索引變慢、`SELECT ... WHERE published=false` 越掃越久、最終拖垮整個資料庫。所以 outbox 一定要配一個清理策略——已投遞超過 N 天的列定期刪除，或乾脆投遞成功即刪。一張沒人清的 outbox 表，是一顆定時膨脹的炸彈。

## 一筆訂單跨三個服務：當 outbox 不夠用

outbox 解的是**一個服務內部**「改 DB ＋ 發訊息」的一致。但真實的下單往往不止一個服務。設想這筆訂單要走三步，分屬三個團隊、三個資料庫：庫存服務扣庫存、支付服務扣餘額、物流服務建運單。這三步**沒有共同的資料庫**，所以沒有任何一個本地交易能罩住它們——outbox 在這裡無能為力，它只能保證每個服務各自「改 DB ＋ 發事件」一致，但管不了「三步要嘛全成、要嘛全不成」這個跨服務的原子性。

而我們剛剛已經論證過，跨服務的真原子提交（2PC）太貴太脆。那怎麼辦？答案是**放棄原子提交，改用 saga**。

saga 的想法樸素得近乎狡猾：既然不能讓三步「同時提交」，那就讓它們**依序各自提交**——每一步都是一個獨立、立即生效的本地交易。庫存扣了就真的扣了、支付扣了就真的扣了。然後，為**每一步**事先準備一個**補償交易（compensating transaction）**：一個語意上能抵銷它的反向操作。一旦走到某一步失敗，就把已經成功的前面那些步驟，**逆序**地各自補償回去，達到「語意上的回滾」。

saga 這個詞和這套機制，出自 Garcia-Molina 與 Salem 1987 年發表在 SIGMOD 的論文〈Sagas〉（pp. 249-259）。他們當年要解的其實是單機資料庫的長交易問題——一個長交易把資源鎖太久、拖垮其他短交易，於是他們提出把長交易拆成一串能交錯執行的小交易，並保證「要嘛全部小交易成功完成，要嘛跑補償交易來修補一個部分執行」。微服務時代把這個老想法重新挖出來，因為它描述的正是「沒有全域交易、只能靠一連串本地交易加補償」的處境。

## 補償不是 rollback：一個必須想清楚的區別

這裡有一個直覺會誤導人的地方：很容易把補償當成 rollback。它們**根本不是一回事**，這個區別是 saga 全部複雜度的源頭。

資料庫的 rollback 是把交易還沒提交的修改**整個丟棄**，世界像那筆交易從未發生過——因為它本來就還沒對外可見。但 saga 的每一步**已經提交、已經對外生效**了。庫存真的扣了、其他請求可能已經看到了扣減後的庫存、支付真的從帳上劃走了錢。你不可能「假裝它沒發生」，你只能**再做一筆反向的操作去抵銷它**：還庫存、退款。

這個差別在某些步驟上會變得很尖銳。如果其中一步是「寄一封訂單確認信給客戶」，而後面的步驟失敗了——你**收不回那封信**。信已經在客戶的收件匣裡了。補償在這裡不可能是「刪掉那封信」，只能是「再寄一封取消通知」。所以設計補償的正確思路，永遠是問**「業務上怎麼抵銷這件事」**，而不是「技術上怎麼回到上一個狀態」。有些副作用沒有乾淨的抵銷，只有將就的彌補，這是 saga 必須正視的現實。

## 把補償手算一遍

把那筆訂單走到底，看補償具體怎麼運轉。三步：扣庫存、扣餘額、建運單。前兩步都成功了，第三步建運單時，物流服務回了 5xx、重試也耗盡。orchestrator 判定這筆 saga 失敗，開始補償。

補償**逆序**執行——後做的先補。所以順序是：先補「建運單」，再補「扣餘額」，最後補「扣庫存」：

```
forward:   扣庫存 ✓ → 扣餘額 ✓ → 建運單 ✗ (5xx, retries exhausted)
                                      │
compensate (逆序):  ← 還庫存  ← 退餘額  ← 取消運單
```

第一個補償「取消運單」有意思：建運單那一步根本沒成功，所以這個補償其實是個 no-op——但它**不能假設**自己是 no-op，它得冪等地去確認「這張運單不存在或已取消」，因為在某些競態下，運單可能其實建到一半。接著「退餘額」把劃走的錢退回、「還庫存」把扣掉的數量加回。整筆 saga 在語意上「回滾」了，雖然資料庫層面它走的是一連串各自提交的本地交易（兩步前進真的提交了、第三步前進失敗未提交，再加三步補償），而非一次原子回退。

這裡有個非顯然、卻致命的邊界：**補償本身也是分散式呼叫，它也會逾時、也會失敗、也會被重投。** 「退餘額」這個補償，如果投遞了兩次怎麼辦？如果它不冪等，客戶就被退了兩次款——你在試圖修復一個不一致時，製造了另一個更糟的不一致。所以**補償操作必須帶冪等鍵**，重複補償只生效一次。一個沒做冪等的補償，會在重試的世界裡把超扣變成超退。saga 的補償路徑，往往比正常路徑更需要被測試，因為它只在出錯時才跑、平時藏著、出事時又是壓力最大的時刻。

還有一個產品層面、容易被工程師忽略的真相：在補償跑完之前，系統存在一個**暫態的不一致視窗**。從「扣餘額成功」到「還庫存完成」之間的那幾百毫秒甚至幾秒，世界處於「庫存已扣、款已退、運單未建」這種任何單一服務看都「不對」的中間態。這個視窗是 saga 用最終一致性換可用性的本質代價，它不是 bug——但它必須被產品、被對帳邏輯（見〈對帳：怎麼確認兩邊一致、漂移怎麼修〉）**明確承認並處理**，而不是假裝不存在。

## 誰來指揮這串步驟：編舞還是編排

saga 把長交易拆成一串本地交易加補償，但「誰決定下一步是什麼、誰在失敗時觸發補償」這件事，有兩種截然不同的組織方式。

**choreography（編舞）：沒有指揮，靠事件接龍。** 每個服務只管「我消費什麼事件、做完發什麼事件」。庫存服務消費 `OrderCreated`、扣完庫存發 `StockReserved`；支付服務消費 `StockReserved`、扣完款發 `PaymentCompleted`；物流服務消費 `PaymentCompleted`、建完運單發 `OrderShipped`。失敗時，反向的事件（`PaymentFailed`）回頭觸發上游的補償。沒有任何一個地方「知道整個流程」，流程**藏在事件之間的因果鏈裡**。好處是極度解耦、沒有中央單點，每個團隊只關心自己的進出。

但 choreography 有個會隨步驟數惡化的病：**整個業務流程沒有一個地方看得見。** 想知道「下單到底走到哪了、為什麼卡住」，你得在腦中把散落在各服務的事件訂閱關係拼回一張圖。步驟少（兩三步）時還好，步驟一多，這張隱形的圖就變成一盤義大利麵——事件 A 觸發 B 觸發 C，某人不小心讓 C 又發了個 A 監聽的事件，繞成一個沒人察覺的迴圈，訊息在系統裡無限打轉。choreography 的去中心，在複雜流程下會變成「沒有人為全局負責」。

**orchestration（編排）：一個中央指揮明確下令。** 一個 orchestrator（編排器）持有整個流程的腳本：呼叫庫存服務、等回應、成功就呼叫支付服務、成功就呼叫物流服務，任何一步失敗就反向觸發補償。流程**集中、顯式、可監控**——你想知道一筆 saga 走到哪，去問 orchestrator 就好，它就是那張圖。代價是 orchestrator 成了一個新的耦合點和單點：所有流程邏輯往它身上集中，它自己的高可用得你負責，它掛了整類 saga 就停擺。

選擇的判準是流程的複雜度。步驟少、耦合鬆、各團隊獨立演進——choreography 的去中心很合適。步驟多、補償組合複雜、需要審計和可觀測性——把流程顯式化、集中到 orchestrator 才管得住。

值得一提的是 orchestration 的一個工程化演進：**durable execution**。手刻一個 orchestrator，你得自己處理「它在某一步崩潰、重啟後怎麼知道流程走到哪、哪些步驟做過了、補償該從哪接續」——這本身就是一個棘手的狀態持久化問題。Temporal、AWS Step Functions 這類持久化工作流引擎，把 orchestrator 的狀態、重試、補償續跑這些機制全接管了：流程以程式碼描述，引擎保證即使行程崩潰，工作流也能從中斷處精確地恢復，而不需要你自己維護那套「我做到哪了」的記錄。它本質上是 orchestration 把「orchestrator 自己怎麼不丟狀態」這個難題外包給了專門的引擎（這類引擎的機制，見〈durable execution：Temporal、Step Functions 與長流程〉）。

## 最該記住的取捨：什麼時候根本不該用 saga

講完機制，最重要的判斷反而是反向的：**很多時候你根本不需要 saga。**

saga 的全部成本——最終一致性、補償邏輯的複雜度、暫態不一致視窗、補償的冪等要求、流程的可觀測性負擔——只在一種情況下值得付：那筆業務交易**真的**跨越了多個不共享資料庫的服務，沒有任何單一本地交易能罩住它。如果你的三步操作其實都打同一個資料庫，那就用**一個本地 ACID 交易**把它們一起提交，乾淨、原子、零暫態——別把一個交易能解決的事，拆成三個服務的 saga 來證明自己懂分散式系統。「為了微服務而微服務」，把本可一個 `BEGIN...COMMIT` 搞定的邏輯硬拆成跨服務 saga，買來的全是真實的複雜度成本，賣掉的是本來免費的強一致。

這兩個機制其實在回答同一個問題的兩個尺度。outbox 問的是：**在一個服務內，怎麼讓「改狀態」和「告訴外界」同生共死？** saga 問的是：**在多個服務間，怎麼讓一串無法原子提交的步驟，要嘛一起前進、要嘛一起退回？** 它們常常一起出現——saga 的每一步，內部往往就用 outbox 來保證「我這步改了狀態，就一定會發出讓下一步啟動的事件」。

而它們形狀的根源是同一句話：**真正的跨系統原子性太貴，貴到你買不起，於是只能退而求其次。** outbox 退的這一步是「把跨系統的原子寫，壓回單一資源的原子寫，剩下的不一致用下游冪等去兜」；saga 退的這一步是「把跨服務的原子提交，換成一串本地提交加補償，剩下的暫態不一致用最終一致性去兜」。兩者都不是在追求完美的一致，而是在「完美一致買不起」的前提下，把不一致**收窄到一個可控、可治、可觀測的形狀**——重複而非丟失，暫態而非永久不一致。理解了這層退讓，你就理解了為什麼這兩個機制長成現在這個樣子：它們不是某種優雅理論的推論，而是工程師在 2PC 的帳單面前，認賠殺出的一條活路。
