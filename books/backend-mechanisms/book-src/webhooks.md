# Webhooks：伺服器對伺服器的回呼

你接了金流。使用者按下付款、跳轉到信用卡頁、輸入卡號、回到你的網站——然後呢？你怎麼知道這筆錢真的進來了？

最直覺的做法是輪詢（polling）：使用者一回來，你就拿著訂單號去問金流商「這筆付了沒？」沒付就過幾秒再問一次。這做法能跑，但很快就會難看起來。付款授權可能要幾秒、可能要幾十秒（3DS 驗證、銀行那端排隊），你不知道該等多久；問太密集是浪費，問太鬆使用者就盯著一個轉圈圈的畫面。更糟的是離線事件——一筆訂閱在三天後續扣失敗、一個退款在隔週由人工核可——這些事件發生時，根本沒有使用者坐在瀏覽器前可以「一回來就問」。你得有一個背景排程器，永無止境地對所有「可能還會變」的訂單反覆探問。輪詢的成本與延遲，本質上是在用「問的頻率」去逼近「事件發生的時刻」，而這兩者永遠對不齊。

Webhook 把這個關係整個反轉過來。與其你一直問「發生了沒」，不如讓金流商在事件**真的發生的那一刻**，主動對你預先登記的一個 URL 發一個 HTTP POST，把事件內容塞在 body 裡推給你。`payment_intent.succeeded`、`charge.refunded`、`customer.subscription.deleted`——對方知道發生了什麼，由對方來敲你的門。延遲從「輪詢間隔」壓縮到「事件發生到一個 HTTP 往返」，而且你不必再養那個沒完沒了的探問迴圈。這就是 webhook：**伺服器對伺服器的回呼**，是 server→client 推送家族裡的一支，只是這裡的「client」是你的另一台伺服器，不是瀏覽器（瀏覽器端的即時推送是另一條路，見〈server→client 推送的全景：推還是拉〉）。

## 反轉之後，難的地方全換了一批

把「拉」換成「推」，省掉了輪詢的麻煩，但也把問題的重心整個搬了家。輪詢時，主動權在你：你決定問誰、何時問、信任誰的回答——因為**是你發起對一個已知端點的請求**。Webhook 把主動權交給了對方，於是冒出四個輪詢時根本不存在的問題，而且每一個都不容易。

第一，**這個 POST 真的是金流商送的嗎？** 你的 webhook 端點掛在公開網路上。它的 URL 不是什麼祕密——它會出現在你的設定頁、可能被記進對方的 log、可能在某次截圖裡外洩。任何人只要知道這個 URL，就能對它發一個長得像 `payment_intent.succeeded` 的 POST。如果你的程式碼天真地相信「收到這個事件＝這筆付了」，攻擊者就能憑空「付款成功」，騙你出貨。輪詢時你不會有這個問題，因為回應來自你主動連上的那台伺服器；推送時，你必須有辦法證明送來的人就是你以為的那一位。

第二，**送不到怎麼辦？** 你的伺服器會重啟、會部署、會偶爾 5xx，會在尖峰時來不及回應。事件發生的那一刻，你家恰好不在——這個事件就這樣丟了嗎？對一筆「付款成功」而言，丟掉意味著客人付了錢卻沒拿到東西。所以推送方**不能只送一次**就算了；它必須在你沒成功收下時，過一陣子再試、再試、再試。

第三，一旦對方會重送，**重複就成了常態**。而重送最陰險的情況不是「你真的沒收到」，而是「你收到了、也處理了、但你回應的那個 200 在回程上塞住或逾時了」。對方那端看到的是「沒收到成功確認」，於是合理地重送同一個事件。同一筆付款，你處理了兩次：出貨兩次、寄了兩封確認信、入帳兩次。推送方無法分辨「沒送到」和「送到了但確認沒回來」——這跟分散式系統裡那個更普遍的真相同根同源：**送出請求卻沒收到回應時，你無法區分是請求丟了還是回應丟了**（這個區分不開的窘境，在訊息交付裡被講到底，見〈三種交付語意：at-most、at-least、exactly-once〉）。

第四，**你被迫接住別人的節奏。** 對方可能在某個熱門時段把幾千個事件一股腦推給你；可能在你下游慢的時候，因為你回得慢而判你逾時、開始重送、讓你雪上加霜。你接收端的處理速度、逾時門檻、去重能力，全都要假設「對面是一個會重投、可能爆量、不等你慢慢來的 at-least-once 投遞系統」來設計。

把這四件事擺在一起看，webhook 的真相就清楚了：**它本質上是「別人家的 at-least-once 投遞，打到你家來」。** 你不擁有那個投遞系統、不能改它的重試策略，但你接收的這一端，必須當成在實作一個 at-least-once 消費者來做——驗來源、容重投、保冪等、快速確認。下面就一件一件拆。

## 證明「是你送的」：HMAC 簽章

端點公開、誰都能 POST，所以第一道、也是最不能省的防線是**驗證來源**。做法是雙方共享一個祕密（webhook secret），推送方用這個 secret 對 payload 算一個訊息驗證碼（HMAC），把結果放在 HTTP header 裡；你收到後用同一個 secret 對收到的 body 重算一次，比對兩者是否相同。算得出來，就證明對方手上有那個 secret——而那個 secret 只有你和推送方知道。

為什麼是 HMAC，而不是「附一個密碼在 header 裡」？因為單純附一個共享密碼，只證明「送的人知道密碼」，不證明「這個 body 沒被竄改」。HMAC 把 secret 和**整個 payload**一起餵進雜湊函數，輸出的標籤同時綁定了「來源」與「內容」——改動 body 的任何一個位元組，重算出來的標籤就對不上。它也不是加密：payload 本身仍是明文（webhook body 多半就是一段 JSON，誰攔到都看得懂），HMAC 提供的是**真實性與完整性**，不是機密性。要機密性，靠的是底下那層 TLS（見〈TLS 與 mTLS：握手與憑證管理〉），不是 HMAC。

實務上各家的 header 格式不同，但骨架一致。看 Stripe 的（2026-06）：它在 `Stripe-Signature` header 裡放 `t=1699...,v1=5257a8...`，`t` 是 Stripe 產生簽章時的 Unix 秒數時間戳，`v1` 是當前的 HMAC-SHA256 簽章。關鍵在簽章**簽的是什麼**：Stripe 不是直接對 body 算 HMAC，而是先把時間戳、一個句點、和原始 body 串成一個 `signed_payload` 字串——

```
signed_payload = "{timestamp}" + "." + "{raw_request_body}"
expected_sig   = HMAC_SHA256(endpoint_secret, signed_payload)
```

——再對這個串算 HMAC-SHA256。把時間戳一起簽進去，是為了下一節要講的防重放；而你驗證時，必須**自己也用同一個時間戳 + 句點 + body 重新組出 signed_payload**，否則永遠對不上。GitHub 走的是更直白的版本：header 叫 `X-Hub-Signature-256`、值是 `sha256=` 開頭接 HMAC-SHA256 的十六進位摘要，直接對 raw body 算、不含時間戳。Standard Webhooks 這個近年成形的規範（由 Svix 發起，OpenAI、Resend、Polar 等採用）則把三件事都標準化成三個 header——`webhook-id`、`webhook-timestamp`、`webhook-signature`——簽的內容是 `{id}.{timestamp}.{body}`，簽章寫成 `v1,` 後接 base64。格式各異，但你看得出共通的形狀：**secret + 時間戳 + 唯一 ID + raw body，一起進 HMAC。**

這裡有一個會反覆絆倒人的細節，值得用力講透：**簽章必須對原始 raw bytes 算，不能對「解析後再重新序列化」的物件算。** 你的 web 框架很可能有一個 body parser 中介層，自動把進來的 JSON 解析成一個物件交給你的 handler。問題是 HMAC 對位元組敏感到極點——重新把物件序列化成 JSON 時，多一個空白、key 的順序變了、一個非 ASCII 字元的 Unicode 正規化不同、數字 `1.0` 被寫成 `1`，產生的位元組就和對方簽的那一份**不一樣**，HMAC 立刻對不上。更糟的是，很多框架的 body parser 會**消費掉**原始輸入流，等你想拿 raw body 時它已經沒了。所以正確的接法是：在 parse 之前就先攔下並保存 raw body（許多框架提供 `verify` callback 或 raw-body 選項），用那份原始位元組去驗簽，驗過了再去用解析好的物件。這不是潔癖，是 HMAC 的數學本性逼出來的硬約束。

最後一個小而致命的點：**比對簽章要用常數時間比對，不能用 `==`。** 一般的字串相等比對是「逐字元比、遇到第一個不同就回 false」——這意味著比對花的時間，洩漏了「你猜的簽章前綴對了幾個字元」。攻擊者反覆送猜測、量回應時間的微小差異，理論上能一個字元一個字元地把簽章磨出來（時序側信道攻擊）。所以要用 `hmac.compare_digest`（Python）、`crypto.timingSafeEqual`（Node）這類**無論對錯都花一樣時間**的比對函數。GitHub 的官方文件就明說「never use a plain `==` operator」。多數官方 SDK 的 `constructEvent` 已經把驗簽、組 signed_payload、常數時間比對都包好了——能用官方 SDK 就用，自己手刻是這些坑的溫床。

## 防重放：簽章驗對了還不夠

假設攻擊者攔截到了一個**合法**的 webhook 請求——連同它那個正確的簽章。它沒有 secret、改不動 body，但它能原封不動地**重送**這個請求。簽章是真的、body 是真的，你驗證會通過。如果這是一筆 `account.credited`（帳戶入帳），攻擊者就能靠重送同一個合法請求，讓你重複入帳。光驗簽章擋不住這種**重放攻擊（replay attack）**，因為被重放的東西本身就是合法的。

這就是為什麼前面要把**時間戳簽進去**。請求裡帶一個時間戳、且這個時間戳也在 HMAC 的保護範圍內（改不動），於是你可以多加一道檢查：**只接受時間戳離「現在」夠近的請求。** Stripe 的官方 SDK 預設容忍 5 分鐘（300 秒）——時間戳比現在早超過 5 分鐘的，一律拒收。攻擊者就算攔到合法請求，過了這個窗口再重放就無效；想在窗口內重放，他得跟你比快，門檻陡然升高。對特別敏感的端點，你可以把這個窗口收得更短。

但容忍窗不是越短越好，這裡有個真實的取捨。網路會延遲、推送方和你的時鐘不會完美對齊、重送本來就會晚到——窗口設得太緊，會把**正常但延遲到達**的合法請求也誤殺掉。所以這道防線有兩個前提：一是你的伺服器時鐘要靠 NTP 對準（時鐘漂移直接吃掉你的容忍預算，時間本身的坑見〈時間與日期：UTC、epoch、timezone 與 DST 的應用層坑〉）；二是窗口大小是一個**安全 vs 容錯**的刻意決定，不是越小越安全。5 分鐘是一個被廣泛驗證過的折衷。

要注意防重放和下一節要講的冪等去重**不是同一件事**，雖然常被混淆。容忍窗擋的是「攻擊者**惡意**重放一個舊請求」；它擋不住「推送方**善意**地在 5 分鐘內合法重投同一個事件」——那個重投帶著新鮮的時間戳，會堂堂正正通過容忍窗檢查。善意重投造成的重複，得靠冪等去重來收。

## 重投是承諾，不是意外：用 event ID 去重

把推送方的視角想一遍，就明白重複為什麼是設計的一部分、而非偶發的 bug。推送方對你發 POST，要嘛收到 2xx（Stripe 明說只認 2xx 為成功，且要你在跑任何可能逾時的複雜邏輯**之前**就先回 2xx），要嘛收到非 2xx 或根本逾時。後者它一律當作「沒送達」，於是排程重送。Stripe 在正式環境會帶著指數退避**重試長達三天**（退避與 jitter 的機制本身見〈重試、退避與 jitter〉）。這對你是好事——你部署期間掉線的事件，三天內會自己補回來，不會丟。但代價就是：**只要你曾經回得慢或回得不順，同一個事件你就會再收到一次。** 對推送方而言，「至少送達一次」是它能給的最強承諾；exactly-once 投遞在這個模型下根本不存在。

所以接收端必須假設**每個事件都可能來不只一次**，並讓「重複處理同一個事件」不造成任何額外副作用。手段是去重：每個 webhook 事件都帶一個全域唯一的 event ID（Stripe 的 `evt_...`、Standard Webhooks 的 `webhook-id`）。你維護一張去重表，記下「已處理過的 event ID」；handler 一進來，先拿 event ID 查表——見過了，就**直接回 2xx、什麼都不做**；沒見過，才真正處理、並把這個 ID 記進表。讓重複操作「做幾次效果都一樣」，正是冪等（idempotency）的核心，這套機制——冪等鍵的生命週期、去重表存哪、TTL 該多長、查表與寫入之間的競態怎麼收——在它的 owning 章講到底（見〈重複是常態：冪等與去重〉）。對 webhook 接收端來說，記住一條操作型結論就夠了：**event ID 是天然的冪等鍵，去重是 at-least-once 接收端的義務，不是選配。**

## 把它算一遍：50,000 個支付事件

抽象的「快點回 2xx、別同步做重活」聽起來像建議；用數字走一遍，它就變成不得不然。

設想一個支付 webhook 端點，每天收 50,000 個事件，推送方對逾時與非 2xx 做 at-least-once 重投，逾時門檻設在 10 秒。你的 handler 老老實實地**同步**做完所有事才回應：寫資料庫、寄一封確認信、更新快取——平均耗時 800 毫秒。平常沒事。但某個下午，寄信的 SMTP 服務變慢，你的 handler 卡在「寄確認信」這一步，整個請求拖過了 10 秒。推送方等不到回應，判定失敗，重送。

假設這天有 0.5% 的事件因為這類慢而觸發重投——250 個。這 250 個裡，有相當一部分（保守估一半，125 個）其實**第一次就已經處理成功了**，只是你回應慢、對方沒收到那個 2xx。於是這 125 筆支付，每一筆都被你的 handler 跑了**第二遍**：寫了第二次資料庫、寄了第二封確認信。如果這個端點背後還連著入帳邏輯，那就是 125 筆重複入帳——一場對帳惡夢（兩邊帳對不齊時怎麼查、怎麼修，見〈對帳：怎麼確認兩邊一致、漂移怎麼修〉）。

正確的形狀是把「確認收下」和「真正幹活」拆開。handler 一進來，先用 event ID 查去重表（一次索引查詢，約莫個位數毫秒）；沒見過，就把事件落地——寫進去重表、丟進一個內部佇列——然後**立刻回 2xx**。整個同步路徑壓到約 20 毫秒，遠在 10 秒逾時之內，推送方每次都及時收到成功確認，重投率掉回趨近零。寄確認信、更新快取這些重活，由背景 worker 從佇列裡慢慢消化（背景處理時，下游跟不上又是另一個故事，見〈背壓：當下游跟不上〉）。而那少數真的因為網路抖動而重投來的事件，命中去重表，直接回 2xx——**零重複副作用**。

這裡藏著一個必須講透的邊界，否則「先落地再回 2xx」會變成「先回 2xx 再落地」的災難：**回 2xx 之前，你必須已經把這個事件持久化到一個可靠的地方**（去重表寫成功、或佇列確認入列）。順序若反過來——先回 2xx 安撫對方，再去寫 DB，結果寫 DB 失敗了——你就製造了最壞的結果：**事件永久遺失**。對方收到了 2xx，認為大功告成、再也不會重送；而你這邊根本沒存下來。Webhook 接收的黃金律是：**先可靠落地，再回 2xx；2xx 是一句「我已經安全收下、你不用再送了」的承諾，不能空口許下。** 落地成功、背景處理卻失敗，那是你自己佇列裡的事，可以重試、可以進死信佇列（見〈死信與毒訊息：DLQ〉），與「要不要請對方重送」無關。

## 順序，與一個容易被忽略的接收端責任

Webhook 還有一件事和直覺相反：**你不該假設事件按發生順序到達。** 同一個訂閱，`customer.subscription.updated` 和 `customer.subscription.deleted` 兩個事件，可能因為網路、因為其中一個觸發了重投、因為推送方內部本就不保證順序，而以顛倒的次序敲你的門。如果你的程式碼是「收到 updated 就把狀態設成 active、收到 deleted 就設成 cancelled」，而 deleted 先到、updated 後到，你最後會錯誤地停在 active。

防禦的辦法不是去要求對方保證順序（多數 webhook 機制本就不保證，要嚴格有序、可重放的事件流，訊息佇列或 event streaming 才是對的工具），而是**讓你的處理對亂序免疫**：每個事件帶一個版本號或事件本身的時間戳，handler 在套用前先比對「這個事件描述的狀態，是不是比我手上已知的更新」——更舊的就丟掉。這把「順序」從一個你無法控制的傳輸保證，轉成一個你能自己掌握的應用層判斷。順序保證為什麼那麼貴、為什麼通常只能在單一序列化點上成立，是另一個主題（見〈訊息順序：FIFO 與 partition key〉）；對 webhook 接收端，記住「假設亂序、用版本收斂」這條防線就夠了。

## 當你是推送方：別讓你的 webhook 成為攻擊跳板

到目前為止都站在接收端。但如果換你當提供方——你讓客戶登記一個 URL，你在事件發生時對那個 URL 發 POST——有一個獨特的、容易被完全忽略的危險：**你正在讓外部使用者指定一個 URL，然後由你的伺服器去主動連它。** 這正是伺服器端請求偽造（SSRF, Server-Side Request Forgery）的標準入口。

惡意客戶可以把 webhook URL 填成 `http://169.254.169.254/...`（雲端的 metadata 端點，可能吐出你的臨時憑證）、或 `http://localhost:port/...`（你內網的管理介面）、或某個你內部才連得到的服務位址。你的伺服器一旦「貼心地」對這些位址發出 POST，就等於替攻擊者打了一個他自己打不到的內網請求，回應內容若再被記進 log 或回吐給對方，就是一次資訊外洩。所以當推送方有一條必修的防線：**對客戶登記的 URL 做出站側驗證**——解析它的目標 IP，拒絕私有網段（10.0.0.0/8、172.16/12、192.168/16）、loopback（127.0.0.0/8 整段，不只 `localhost` 一個名字，還有 `0.0.0.0`）、link-local（169.254/16）等內部位址——而且**IPv6 的對應網段（`::1`、`fc00::/7`、`fe80::/10`）一樣要擋**，只擋 IPv4 私網卻漏掉 IPv6，是 SSRF allowlist 實作裡很常見的真實破口；還要小心 DNS rebinding（驗證時解析到公網 IP、真正連線時又解析到內網的時間差攻擊）。這條和 webhook 的功能無關，純是「讓外人決定你連哪裡」這個動作天生帶的風險（更完整的清單見 OWASP 的 SSRF 防護指南，列在延伸閱讀）。

推送方該扛的另外兩件事，前面已從接收端的鏡像看過：你要對非 2xx 與逾時做帶退避的重投（並設一個放棄門檻、把最終送不到的事件落進 DLQ 或標記給人看），以及你要支援 **secret 輪替**。Secret 不可能永遠不換——它會在某次設定外洩、某個離職的人手上、某份被誤推上 git 的設定檔裡曝光。但如果你只支援「一個 secret」，輪替的瞬間就是個死結：你一改 secret，所有還在用舊 secret 驗證的接收端立刻全部驗不過。正確的設計是**讓新舊兩個 secret 在一段過渡窗內同時有效**——這段期間，你用新 secret 簽、但接收端拿新舊任一個都驗得過；等所有接收端都換上新 secret，再把舊的停用。這跟憑證、token 的輪替是同一個母題：任何共享祕密的更換，都要有一段新舊並存的重疊窗，不能瞬時切換（token 的撤銷與輪替見〈token 的生命週期與撤銷〉）。

## 為什麼是這個形狀

退一步看，webhook 的每一道機制，都是從「把拉換成推」之後、主動權交給對方所必然衍生的後果裡長出來的。

因為端點公開、誰都能敲門，所以非有 **HMAC 簽章**不可——你得有辦法證明送門的人手上有那個只有你倆知道的祕密。因為合法請求可以被原樣重放，所以簽章裡要綁**時間戳**、要設**容忍窗**。因為對方只能承諾「至少送一次」、且分不清「沒送到」和「確認沒回來」，所以重投是常態，所以你非有 **event ID 去重**不可。因為對方會因你回得慢而重投、甚至判你逾時，所以你得**先可靠落地、立刻回 2xx、重活丟背景**——而那個「先落地」的順序，是丟不丟事件的分水嶺。因為傳輸不保證順序，所以你得在應用層**用版本收斂**而不是信任到達次序。而當你站到推送方那一側，「讓外人指定一個 URL 由我去連」這個動作本身，又開了一道 SSRF 的門需要堵。

把這些連起來，webhook 不是一個鬆散的「對方 POST 給你、你收下」的約定，而是一整套**把別人家的 at-least-once 投遞，安全地落地到你家**的協定。它看起來簡單——不就是一個 HTTP 端點嗎——但它簡單的外表底下，藏著分散式投遞的每一個老問題：來源信任、重放、重複、順序、可靠落地。下次你接一個第三方的 webhook，在 handler 第一行寫下「先驗簽、再查去重表」之前，你會知道那兩行不是樣板，而是這個機制全部重量的支點。
