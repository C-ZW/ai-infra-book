# IaC：宣告式、漂移與冪等 apply

凌晨兩點，生產環境的某個 load balancer 開始把流量打回 503。值班的人翻到根因：某顆 EC2 沒掛上 target group。再往上追，三週前有人為了救一個臨時的火，到 console 手動把這顆機器從某個 security group 挪了出去，事後忘了講。沒人記得這件事發生過，因為它從來沒寫在任何地方——它只存在於「那天那個人的手」和「雲端後台的當前狀態」之間。

這台機器是一個**雪花伺服器（snowflake server）**：每一片都獨一無二、由一連串無人記錄的手動操作雕出來，壞了重建不出來，因為沒人說得清它「應該」長什麼樣。把這個現象放大到三十台機器、五個環境，你得到的是一個沒有人能完整描述、也沒有人敢動的系統。每次部署都是在賭——賭你記得的狀態，剛好等於真實的狀態。

IaC（Infrastructure as Code）這套機制存在的理由，就是把那句「應該長什麼樣」從某個人的腦袋裡，搬到一份可以版控、可以 review、可以重放的**程式碼**裡，然後讓一個工具負責把真實世界收斂到那份程式碼。聽起來只是「把指令寫進檔案」這麼樸素的一件事，但要真正做對，得先想清楚一個比想像中深的問題：**當你描述「應該長什麼樣」、而現實「現在長這樣」時，工具要怎麼算出「該動哪些手」，而且算錯了、跑兩次了，都不會把事情搞砸。**

## 命令式的天真做法，與它在哪裡破掉

最直覺的自動化，是把那些手動步驟寫成一段腳本：建一台 VM、裝套件、開 443、掛磁碟、加進 security group。這是**命令式（imperative）**——你寫的是一串**動作的序列**，腳本忠實地一條條執行。

它的第一個破綻，是**不冪等**。你跑一次，它建了一台 VM；你因為網路中斷在第三步掛了、重跑一次，它**又建了一台 VM**——因為腳本只知道「執行 create」這個動作，不知道「這個東西已經存在了」。命令式腳本要做到「重跑安全」，得自己在每一步前面塞一堆 `if exists then skip`，而這些檢查正是最容易寫漏、寫錯的地方。一個沒處理好的重試，就在雲端帳單上留下一排你不記得建過的孤兒資源。

第二個破綻更隱蔽：**它沒有「目標」的概念，只有「動作」的概念**。腳本能告訴你它做了什麼，但回答不了「現在的基礎設施，跟它應該有的樣子，差在哪」。你想把某顆磁碟從 100 GB 改成 200 GB，命令式的你得自己知道「現在是 100」、自己寫出「resize 到 200」這個動作；你想刪掉一個不再需要的 security group rule，得自己記得它曾經被加過、再寫一條對應的刪除。基礎設施的**當前形狀**，從來不在腳本裡——它在你的記憶裡，而記憶會漂移。

宣告式（declarative）IaC 的整個設計，就是針對這兩個破綻。你不再寫「做哪些動作」，而是寫**「最終要長什麼樣」**：我要一台 `t3.medium`、開 443、掛一顆 200 GB 的磁碟、屬於這個 security group。你描述的是**期望狀態（desired state）**，至於「從現在到那個樣子要動哪些手」——那是工具的工作，不是你的。

這個轉換把問題的重心整個挪了位置。命令式問「我接下來該做什麼動作」；宣告式問「現實和我要的目標，差在哪，怎麼把差距補平」。而要回答後者，工具必須先有辦法知道「現實現在是什麼樣子」——這就帶出了宣告式 IaC 真正的核心機器：**state**。

## state：一份「我上次管到的世界」的快照

宣告式工具要把現實收斂到程式碼，得先知道現實長什麼樣。最天真的辦法是每次都去雲端把所有東西撈一遍、跟程式碼比對。但這行不通，原因很實際：你的程式碼說「我要一台 t3.medium」，雲端上有五十台 t3.medium——**哪一台是「我這份程式碼管的那一台」？** 程式碼裡的一個邏輯資源（你給它取名叫 `web_server`），怎麼對應到雲端那個有著一長串隨機 ID（`i-0a1b2c3d…`）的真實實體？

這就是 **state（狀態檔）** 要解的問題：它是一份**映射表**，記錄「我程式碼裡這個叫 `web_server` 的邏輯資源，對應到雲端上 ID 為 `i-0a1b2c3d` 的那個真實物件，而且我上次看到它的時候，它的屬性是這些」。state 是工具對「我管轄的這塊世界」的**記憶**。沒有它，工具每次都得重新猜哪個資源是哪個，而宣告式的整套收斂邏輯就無從談起。

有了 state，一次 `plan`（Terraform 的術語，OpenTofu 同）就是一場**三方比對**，這是理解宣告式 IaC 最關鍵的一張圖：

```
            terraform plan / apply
   +------------------+   +------------------+   +------------------+
   |   CONFIG (.tf)   |   | STATE (.tfstate) |   |    REAL INFRA    |
   |    你要的樣子    |   |   我記得的樣子   |   |    現在的樣子    |
   +------------------+   +------------------+   +------------------+
            |                       |                       |
            |    refresh: 用 real infra 更新 state          |
            |                       +<----------------------+
            |                       |
            +------- diff ----------+   算出「要動哪些手」
                    |
               execution plan (create/update/delete/replace)
```

`plan` 跑的時候，**第一步永遠是 refresh**：拿著 state 裡記的那些 ID，去雲端逐一查「這些資源現在的真實屬性是什麼」，把答案更新回 state（在記憶體裡）。refresh 之後，工具手上就有了兩份對齊的東西——**現實的當前樣子**（剛刷新的 state）和**你要的樣子**（config）。兩者一 diff，差異就是「要動哪些手」：config 有、現實沒有 → create；兩邊都有但屬性不同 → update；現實有、config 沒有了 → delete。這份差異清單就是 execution plan，它把抽象的「收斂到目標」變成一張具體、可讀、可審的動作表。

值得停下來看清 state 的雙重身分，因為它同時是這套機制的力量來源和最大的要害。它是力量來源，因為有了「我上次看到的樣子」，工具才有辦法回答「現實有沒有被人偷偷動過」——這正是漂移偵測的全部根據。它是要害，因為一旦 state 和現實對不上（state 損壞、被兩個人同時寫、記了一個已被手動刪除的資源），工具對世界的認知就錯了，而它後面所有的 `plan` 都建立在這個錯誤的認知上，會算出一連串荒謬的動作。state 是宣告式 IaC 的記憶，而一個記錯了的系統，比一個沒有記憶的系統更危險。

## 漂移：現實偷偷偏離了你寫下的樣子

**漂移（drift）** 就是開場那台掉了 security group 的機器：**真實基礎設施偏離了程式碼宣告的狀態**。有人到 console 手改、某個別的系統動了一個資源、一次手動的緊急修復沒有回填進程式碼——這時「程式碼說的」和「現實的」就分了岔。

漂移之所以能被偵測，全靠 state 那份「我上次看到的樣子」。`plan` 的 refresh 一旦把現實的真實屬性刷回來，跟 state 裡記的舊屬性一比，對不上的地方就是漂移。Terraform/OpenTofu 提供一個專門的 `plan -refresh-only`：它只做這件事——把 state 跟現實對齊、把差異**報給你看**，但**不提出任何要改動基礎設施的動作**。它回答的問題是純粹的「有人動過我管的東西嗎、動了哪些」，而不是「我該怎麼把它改回去」。（順帶一提，老的 `terraform refresh` 子指令已被棄用，因為它會**直接覆寫 state**而不先給你看差異——這正是 `-refresh-only` 要修正的危險：先看，再決定。）

漂移偵測有一個容易被忽略、但很要命的邊界：**`plan` 只看得到 state 裡已經有的資源。** 如果有人不是「改了」一個受管資源，而是**全新手動建了**一台 EC2、一條 security group rule——這個東西從來沒進過 state，工具對它的存在**一無所知**。它不會出現在任何 `plan` 的 diff 裡，不會被報成漂移，因為漂移的定義是「受管資源偏離了宣告」，而這個東西根本不在受管的範圍內。它是一個 IaC 看不見的影子資源，靜靜地存在於你的雲端帳單和攻擊面上。要把它納管，得反向 `import`——把這個現存的真實物件「認領」進 state，IaC 才開始對它有認知。**「我用 IaC 管基礎設施」從來不等於「我所有的基礎設施都被 IaC 管著」**，這兩者之間的縫隙，正是漂移最陰險的藏身處。

偵測到漂移之後，怎麼處理是一個有取捨的決定，而不是只有一個正解：

- **拉回程式碼（revert）**：下一次 `apply` 時，把現實改回程式碼宣告的樣子。這是宣告式的預設姿態——程式碼是真相，現實服從程式碼。代價是：那次手動修改如果是**有意義的緊急修復**，這一拉就把救火的成果無聲地抹掉了（下一節會看到這怎麼傷人）。
- **吸收進程式碼（align）**：承認手改是對的，把它**回填**進 config，讓程式碼追上現實。
- **忽略（ignore）**：有些欄位本來就會被外部系統合法地改（如某個自動擴縮控制器調整的 replica 數），用 `ignore_changes` 之類的機制告訴工具「這個欄位的漂移不要管」。

實務的趨勢（2026-06）是從「只偵測、發告警」走向「定期自動矯正」——但自動矯正本身就是一把雙面刃，因為它意味著 IaC 會在你沒盯著的時候，把現實拉回程式碼。這個威力，正是下一個陷阱的來源。

## 一次「無聲覆蓋」是怎麼發生的

把上面的機制接起來，跑一遍最常見、也最會傷人的一個故事，你就懂了為什麼漂移不只是個技術名詞。

週二下午，一個服務在生產環境扛不住流量。值班工程師到 console 手動把某個 auto scaling group 的容量從 4 台臨時拉到 12 台，火滅了，鬆一口氣。但他沒有把這個 `12` 回填進 Terraform 的 config——config 裡那個資源還寫著 `desired_capacity = 4`。

現在 state 裡記的是「我上次管的時候是 4」，現實是「12」，config 是「4」。**漂移已經產生**，只是沒人去看。

兩天後，週四，另一個工程師為了一件完全不相干的事（改一個 tag）跑了 `apply`。`apply` 一如往常先 refresh——它去雲端一查，發現這個 ASG 現在是 `12`，跟 state 記的 `4` 對不上，於是把**記憶體裡**的 state 更新成 `12`（注意：這時還沒寫回磁碟上的 `.tfstate`——refresh 的結果要到 apply 真的成功才會被持久化）。接著它做 diff：現實（剛刷新成 `12`）對上 config（`desired_capacity = 4`）。從工具的角度看，這是一個再清楚不過的漂移：**現實是 12，但你的程式碼說要 4。** 它忠實地把這個差異寫進 plan：`desired_capacity: 12 -> 4`。

如果這位工程師沒有逐行讀 plan（他只是來改個 tag，誰會細看一個 ASG 的容量？），他按下 yes，`apply` 就把容量從 12 砍回 4。週二那場救火的成果，在一個跟它八竿子打不著的部署裡，被**無聲地覆蓋**了。幾分鐘後，服務又開始扛不住——而這次沒人知道為什麼，因為「把容量改回 4」這個動作，從來沒有人**打算**做。它是宣告式收斂的一個忠實後果：你寫的 config 還是 `4`，工具只是盡責地讓現實服從了它。

這個故事裡沒有任何一個元件壞掉。state 正確地記錄、refresh 正確地偵測、diff 正確地計算、apply 正確地收斂。傷害來自一個**人類流程的縫隙**：手動修復沒有回填。而宣告式 IaC 的力量——「程式碼是真相，現實必須服從」——在這個縫隙上，恰好變成了它的危險。這也是為什麼**「手改了就一定要回填，否則別手改」**不是一句道德勸說，而是這套機制能不能信任的前提。它還解釋了為什麼那張 `plan` 的 diff 必須有人**真的讀懂**：plan 不只是「我要做的事」的預告，它同時是「現實偷偷變成什麼樣」的體檢報告。閉著眼睛按 yes，等於放棄了宣告式設計最值錢的那道安全閘。

## state 是共享資源，所以它需要一把鎖

前面說 state 是要害，最尖銳的一面在於：**它是一份被多人共享、且會被並發修改的資源。** 任何共享可變狀態都會有並發問題，state 也不例外。

設想兩個工程師同時對同一份 state 跑 `apply`。兩邊都 refresh、都讀到「現實是這樣」、各自算出自己的 diff、各自開始改雲端、然後**各自把更新後的 state 寫回去**。後寫的那一份會蓋掉先寫的——於是 state 裡丟失了一部分剛剛真實發生的變更。從此 state 對「現實是什麼」的認知就**錯了**：它記得的世界，跟雲端的真實世界對不上。而前面說過，後面所有的 `plan` 都建立在 state 之上——一個記錯的 state，會讓接下來每一次 `plan` 都算出基於錯誤前提的、可能極具破壞性的動作（比如它以為某個資源不存在，於是要 create 一個其實已經存在的東西，或者反過來要 delete 一個還在服役的東西）。

解法是給 state 上**鎖**：一個 `apply` 開始前先「取得鎖」，沒拿到鎖就拒絕動手、報 `Error acquiring the state lock`，做完才「釋放鎖」。這是把「讀-改-寫 state」這串操作變成**互斥**的臨界區，本質上跟一個分散式鎖要解的問題同構——只有一個 writer 能在任一時刻持有對 state 的寫入權。

機制細節在這幾年有個值得一提的演進（2026-06）。早年 Terraform 的 S3 backend 沒有自己的鎖，得**外掛一張 DynamoDB 表**：用 DynamoDB 的條件寫入（conditional write）來實作「同一個 key 只有一個 writer 能成功寫入」這個互斥原語。後來 S3 自己支援了條件寫入（put-if-absent 風格：靠 `If-None-Match`，只在物件不存在時才寫入成功），於是 S3 backend 加上了**原生鎖**：設 `use_lockfile = true`，它會在 state 檔旁邊寫一個 `.tflock` 物件，靠 S3 的條件寫入保證同一時刻只有一個持有者。這個原生鎖自 Terraform 1.11（2025-02）起 GA，而原本的 DynamoDB 鎖則被標記為棄用、將在未來版本移除。不管底層用哪個，要解的問題從頭到尾是同一個：**state 是共享可變狀態，並發寫它會壞，所以要一把鎖把寫入序列化。**

鎖也有它自己的故障模式：一個 `apply` 跑到一半，工程師的筆電斷網、行程被 kill——鎖**沒被釋放**，卡在那裡，後面的人全部拿不到鎖、動彈不得。這時得用 `force-unlock` 手動把它撬開。但 `force-unlock` 是一個危險動作：它的前提是「你**確定**沒有別的程序還握著這把鎖、還在改雲端」。如果你誤判了——對方其實還活著、還在 apply 到一半——你撬開鎖、自己也開始 apply，兩個程序就同時在改同一塊基礎設施、寫同一份 state，回到了一開始要避免的並發災難。所以 `force-unlock` 之前，第一件事永遠是**搞清楚這把鎖到底是誰、什麼時候鎖的、那個程序還活著嗎**，而不是看到「拿不到鎖」就反射性地強解。

順帶一提 state 的另一個常被忽略的安全面：state 檔裡會以**明文**存下資源的屬性，而這些屬性裡可能藏著密碼、金鑰、token（因為某些資源建立時這些東西就是它的屬性）。一份 state 放進沒加密的 bucket、或不小心 commit 進 git，就是一次洩密。比較新的機制（Terraform 1.10 的 ephemeral values、1.11 的 write-only arguments，2025-02 GA）能讓部分敏感欄位**只在記憶體裡流轉、不寫進 state**——但它需要 provider 適配，舊資源與未適配的 provider 仍會把秘密明文落進 state，所以「state 要加密、要管權限」依然是不能省的防線。state 既是 IaC 的記憶，也是它的保險箱——而保險箱沒上鎖、沒加密，比沒有保險箱更糟。

## 冪等 apply：為什麼跑兩次跟跑一次結果一樣

回到那個讓宣告式 IaC 能被信任的核心性質：**冪等（idempotent）**——同一份 config，`apply` 一次和 `apply` 五次，結果都收斂到同一個狀態，不會重複建資源、不會越改越亂。冪等的完整定義在訊息交付的脈絡裡深講（見〈重複是常態：冪等與去重〉），這裡只看它在 IaC 裡**怎麼長出來的**，因為這正是宣告式相對命令式最關鍵的優勢。

冪等不是工具額外加的一道保險，它是**「比對現實再決定動作」這個架構的自然結果**。命令式腳本不冪等，因為它無腦執行「create」這個動作——跑兩次就建兩個。宣告式 `apply` 冪等，因為它每次都先看現實：

第一次 `apply`，config 說要一台 t3.medium，refresh 後發現現實沒有 → diff 算出 `create` → 建了，state 記下它。第二次 `apply`，**同一份 config**，refresh 後發現現實**已經有**一台 t3.medium 而且屬性跟 config 一致 → diff 算出來是**空的**（no changes）→ 什麼都不做。動作的有無，取決於「現實和目標的差距」，而不是「我跑了幾次指令」。差距為零，動作就為零。這就是冪等：操作的結果只跟「目標狀態」綁定，跟「執行次數」無關。

理解了這一點，你就懂了為什麼宣告式 IaC 敢讓你「反覆 apply 同一份 config 而不怕」，也懂了 GitOps 這類工具（一個常駐 controller 不停地把叢集現狀往 Git 裡的宣告收斂）為什麼能成立——它本質上就是**把冪等的 apply 排成一個無窮迴圈**：每隔一段時間 diff 一次，有差距就補、沒差距就閒著。如果 apply 不冪等，這個迴圈第二圈就會把一切搞砸。冪等是宣告式收斂能變成「持續收斂」的前提。

## 冪等的縫隙：宣告式不會自動讓一切都冪等

但這裡有個關鍵的、極容易被高估的邊界：**宣告式工具的冪等，只覆蓋它「建模得到」的那些資源；對它模型之外的副作用，它什麼保證都不給。**

宣告式之所以能冪等，是因為對每一種它支援的資源（一台 VM、一條 DNS record、一個 bucket），背後的 provider 都精確地知道「這個東西現在的狀態怎麼讀、有哪些屬性、怎麼比對、怎麼增量修改」。冪等性建立在這個「provider 完整理解這種資源」的前提上。

可一旦你在 IaC 裡塞進一段**它無法建模的副作用**，前提就破了。最典型的是 `local-exec` / `remote-exec` 這類 provisioner：它本質上就是讓你在 apply 過程中跑一段任意 shell。而 shell **是命令式的**——它執行動作，工具完全不知道這段 shell 幹了什麼、改了什麼、能不能安全重跑。如果那段 shell 不冪等（比如它往某個外部系統 POST 了一筆資料、或往一個檔案 append 了一行），那麼**這一塊的冪等性就漏了**：apply 兩次，這段 shell 就跑兩次，外部系統就多了一筆、檔案就多了一行——儘管你周圍所有「正規」的資源都乖乖地冪等著。

這是一個很細、但會反覆咬人的真相：**「我用的是宣告式工具」不等於「我的 apply 整體是冪等的」。** 你的 apply 的冪等性，是它管轄的所有部分的冪等性的**交集**——只要你混進一塊命令式的、工具看不透的副作用，整體的冪等保證就在那一塊上破洞。冪等不是你選了宣告式工具就自動拿到的徽章，它是一個需要你守住邊界、不往裡面偷渡命令式副作用才能維持的性質。

## 為什麼有時候「改」其實是「砍掉重建」

還有一個藏在宣告式 diff 底下、不看清會出事的機制。`plan` 的動作不只 create / update / delete 三種——在這三種之外，diff 還會算出第四種，而且它最危險：**replace（替換）**。

不是所有屬性都能「就地修改」。雲端 API 對某些屬性是**不可變的**——比如一台 VM 的可用區、一個資源的某些底層 ID。當你在 config 裡改了一個這樣的「不可變屬性」，工具沒辦法 update（API 不允許），它唯一能做到「讓現實符合新 config」的辦法，是**把舊的整個 destroy 掉、再 create 一個帶新屬性的**。在 plan 的輸出裡，這會顯示成那個刺眼的 `-/+ destroy and then create replacement` 或 `# forces replacement`。

問題在於**順序**。預設行為是**先 destroy、再 create**——而這中間有一段時間，這個資源**根本不存在**。如果這個資源正在服役（一個還在接流量的 load balancer、一個正在被引用的 security group），這段「先拆後建」的空窗就是一次實打實的停機，甚至可能因為「別的資源還引用著這個正在被拆的東西」而直接 apply 失敗。

修正的機制叫 `create_before_destroy`：明確告訴工具「**先把新的建好、確認 OK，再拆舊的**」，把空窗消掉，達成零停機替換。但它不是免費的——它有自己的脾氣，而這些脾氣本身就是值得記住的 edge case：

- **命名衝突**：先建後拆意味著有一瞬間「新舊兩個同時存在」。如果這種資源要求名字全域唯一（很多資源都是），新的建立會因為「名字已被舊的佔用」而失敗。解法是不用固定名字，改用 `name_prefix` 讓工具自動生成不衝突的名字，讓新舊能短暫共存。
- **依賴的連鎖傳染**：如果資源 A 設了 `create_before_destroy`、而 A 依賴 B，工具會把這個行為**強制傳染給 B**——否則「先建新 A」時，新 A 要依賴的 B 還是舊的、可能已經要被拆了，邏輯就矛盾了。這個連鎖可能比你預期的傳得更遠，牽動一整串你沒打算重建的資源。
- **deposed 狀態**：在「新的已建好、舊的還沒拆完」的中間時刻，state 必須同時記著兩個——舊的被標成 `deposed`（待廢）。如果 apply 恰好在這個中間態崩掉，你會留下一個卡在 state 裡的 deposed 物件，得手動清理。

這整套東西的存在，呼應了一個更大的設計哲學——**不可變基礎設施（immutable infrastructure）**：與其去「修改」一台跑著的機器（這正是雪花伺服器的成因），不如每次變更都**建一台全新的、替換掉舊的**。`create_before_destroy` 就是把這個哲學落實到單一資源層級的機制。它買到的是「乾淨、可重現、沒有累積的手動修改」，付出的是「替換的協調成本」——而上面那幾個 edge case，正是這個協調成本的具體長相。

## 同一個機制，不只活在 Terraform 裡

值得把鏡頭拉遠一點，因為「宣告期望狀態 + 工具持續收斂」這個模式，遠不只是某個 IaC 工具的特色——它是現代基礎設施的**共同骨架**，認出它，你會在很多地方看到同一套機器在運轉。

Kubernetes 本身就是一個巨大的宣告式收斂系統：你 apply 一份 manifest 宣告「我要 3 個 replica」，一群 controller 跑著「觀察現狀 → 比對期望 → 採取行動」的迴圈，把現實往你的宣告收斂（它的核心物件與控制迴圈見〈Kubernetes：Pod、Service、HPA 與調度〉）。而 `kubectl apply` 本身也得解跟 Terraform 一模一樣的冪等問題：怎麼讓「反覆 apply 同一份 manifest」收斂到同一個狀態、而不會把別人（如某個 autoscaler）合法改過的欄位無腦覆蓋掉？

它的答案叫 **server-side apply**（自 Kubernetes 1.22 GA，2026-06 為穩定機制）。早期 `kubectl` 用一個叫 `last-applied-configuration` 的 annotation 來記「我上次 apply 了什麼」，靠它做三方合併——這跟 Terraform 用 state 記「我上次看到什麼」是同一個思路：**你需要一份「我上次的版本」當參照，才能算出這次該動哪些欄位、又不去碰別人管的欄位。** server-side apply 把這個機制搬到 API server 上，用一個 `managedFields` 結構**逐欄位地記錄「每個欄位現在歸哪個 field manager 管」**。當兩個寫入者想改同一個欄位，它會偵測到衝突並擋下來，而不是讓後到的無聲覆蓋先到的。

注意這裡的回響：Kubernetes 用 `managedFields` 解的「兩個 manager 搶同一欄位、別讓後者無聲蓋掉前者」，跟前面那個「週二手改容量、週四被 apply 覆蓋」的故事，是**同一個問題的兩種長相**。一邊是 IaC 與人類手改之間的衝突，一邊是兩個自動控制器之間的衝突。宣告式收斂的力量——「現實服從宣告」——在多個寫入者並存時，總會遇到「誰的宣告算數」這道題；不同的系統用不同的機制（Terraform 靠人讀 plan + state 鎖、Kubernetes 靠 managedFields + 衝突偵測）來回答它，但題目是同一道。

## 為什麼是這個形狀

退一步看，宣告式 IaC 的整副樣貌，都是從一個樸素的需求裡長出來的：**讓「基礎設施應該長什麼樣」這件事，有一個唯一的、可版控的、可重放的真相來源，而不是散落在某些人的手和記憶裡。**

正因為真相要在程式碼裡，所以你描述的是**目標**而非動作——這逼出了宣告式。正因為工具要從目標反推動作，它必須知道現實長什麼樣——這逼出了 **state**。正因為它每次都先比對現實再決定動作，它天然**冪等**，敢讓你反覆 apply、也讓 GitOps 那種持續收斂的閉迴路得以成立。正因為 state 是被共享、被並發修改的共同記憶，它需要**鎖**，也需要加密。正因為現實會被程式碼之外的手偷偷改動，這套機制必須能偵測**漂移**——而漂移偵測的全部根據，又恰好是 state 那份「我上次看到的樣子」。每一個零件都不是任意的工程選擇，而是「把基礎設施變成可信任的程式碼」這個目標，被一路逼到底之後，必然會長出來的形狀。

它的危險也是同一個源頭的另一面。「現實服從程式碼」的力量，在「手改沒回填」的縫隙上會變成無聲的覆蓋；「state 是記憶」的好處，在 state 被寫壞或記錯時會放大成系統性的災難；「宣告式自動冪等」的承諾，在你偷渡命令式副作用時會默默破洞。這些不是 bug，是這個形狀的影子——理解它們，你才能在凌晨兩點看到那張 `plan` 的 diff 時，分得出哪些是「我要的收斂」，哪些是「我不小心要抹掉的、某個人週二的救火」。
