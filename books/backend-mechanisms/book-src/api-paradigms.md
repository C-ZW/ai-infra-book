# REST、GraphQL 與 gRPC：三種 API 範式

一個訂單詳情頁，要顯示「這筆訂單、下單的買家是誰、訂單裡三個品項的縮圖」。後端把資料拆得很乾淨：訂單一張表、使用者一張表、商品一張表，各自有獨立的 HTTP 端點。手機 App 打開這個頁面，於是發生了這樣的事——先 `GET /orders/1` 拿到訂單，從回應裡讀出 `buyer_id` 和三個 `product_id`，再 `GET /users/9` 拿買家名字，接著 `GET /products/{id}` 三次拿縮圖。五次往返，而且後三次得等第一次回來、解析出 id 才能發。在辦公室的 Wi-Fi 上，這五次加起來幾十毫秒，沒人在意。換到地鐵裡、單程延遲 200 毫秒的行動網路上，這個頁面要轉小圈圈轉超過一秒，而且使用者真正想看的資料其實早就準備好了，只是被「介面的形狀」拖住了。

這一秒，不是資料庫慢，不是序列化慢，是 **API 範式**選錯了。同一批資料、同一個後端，換一種介面風格，這五次往返可以塌成一次。但那一次往返會買來別的麻煩。這一章要講的，就是 REST、GraphQL 與 gRPC 這三種風格各自在解什麼問題、底下怎麼運作、以及它們各自欠下什麼債——因為**它們不是同一條軸上比快慢的三個競品，而是針對三個不同問題優化出來的三種形狀**。把它們擺成「誰比較好」的對決，是理解這件事最快的錯路。

## 先把問題的軸找出來

要看懂這三者為什麼長得不一樣，得先承認：一個 API 不是一個東西，而是好幾個決定疊在一起。把這些決定拆成幾根正交的軸，三者就會各自落在不同的格子裡，而不是擠在「好／壞」這一條線上。

第一根軸是**資料的形狀由誰決定**。你呼叫一次 API，回來的那團資料，它的欄位是誰說了算？是 server 在端點裡寫死的、是 client 在請求裡點菜的、還是雙方在一份 schema 裡事先約好的？這根軸幾乎決定了其他一切。

第二根軸是**契約有多硬**。「契約」指的是「呼叫方和被呼叫方對『有哪些方法、每個方法吃什麼吐什麼』的共同認定」。這份認定可以硬到編譯期就擋下不合的呼叫，也可以軟到只是一份寫在文件裡、沒人強制你遵守的描述。硬契約買到「不合的呼叫根本編不過」，代價是兩端得綁在同一份 schema 上一起演進。

第三根軸是**傳輸有多省**。同一筆資料，在線上是用人看得懂的 JSON 文字傳、還是壓成機器才讀得懂的二進位？省下來的位元組要夠多、呼叫要夠頻繁，這個省才划算；對一個一天被打幾百次的內部小服務，省那幾百位元組換來「不能用 `curl` 直接試、出錯時看不懂封包」的代價，幾乎一定不值。

第四根軸最容易被忽略，卻常常是事後最痛的：**這個範式跟它腳下的基礎設施合不合得來**。HTTP 這一層有一整套免費的東西——CDN 快取、瀏覽器快取、反向代理、各種看得懂 HTTP 狀態碼的監控和告警。一個範式如果順著 HTTP 的紋理長，這些東西白送；如果逆著長，這些東西就一個個失效，你得在應用層自己把它們重新發明一遍。

REST、GraphQL、gRPC 之所以是三種範式而不是三個版本，正是因為它們在這四根軸上做了**不同方向**的賭注。接下來一個一個看它們賭了什麼、贏了什麼、又欠了什麼。

## REST：順著 HTTP 的紋理長

REST 不是一個協定，是一組架構約束。Roy Fielding 在 2000 年的博士論文裡提出它時，描述的是「網路化軟體系統該滿足哪些約束才能長得像 Web 一樣可擴展」——client-server、無狀態、可快取、分層、統一介面，以及那條最常被跳過的 code-on-demand。其中 Fielding 自己點名「統一介面」是 REST 區別於其他風格的**核心**特徵。但對天天寫後端的人來說，REST 真正交付的工程價值，不在 URL 長得漂不漂亮，而在三件**可以被機器驗證**的東西。

第一件是**方法語意**。HTTP 的動詞不是裝飾，它們各自帶著規格層級的承諾：GET 是安全的（safe，不該有副作用）也是冪等的（重送結果一致），PUT 和 DELETE 是冪等的，POST 不是。這幾個字背後的約束，RFC 9110 寫得清清楚楚。它直接決定一件每天都會遇到的事——**逾時之後能不能重試**。一個 GET 逾時了，你閉著眼睛重送都行；一個 POST 逾時了，你不知道它到底有沒有在 server 那邊生效，重送可能就建了兩筆訂單。所以「把建立資源做成 POST、把查詢做成 GET」不是慣例潔癖，是在替整條鏈路上的重試邏輯定錨。反過來，把刪除做成 `GET /delete?id=1` 這種事為什麼危險，也是同一個道理：GET 被規格承諾為安全的，於是瀏覽器的預抓取、搜尋引擎的爬蟲、各種中間層的重試，都會「以為這只是讀一下」而放心地打它——結果一個爬蟲爬過你的後台，資料就被刪光了。

第二件是**版本相容**。介面一旦有了它管不動的 client——別的團隊的前端、第三方、市面上還沒更新的舊版 App——它就進入了「不能說改就改」的契約狀態。

第三件是**契約描述**。OpenAPI（前身是 Swagger）把端點、參數、schema、錯誤碼寫成一份機器可讀的 YAML 或 JSON，讓 client SDK、mock server、契約測試都從同一份真相生出來。但這裡有個保證的邊界值得記死：**OpenAPI 只是「描述」，不是「強制」。** 它保證「有一份契約」，不保證「實作真的符合契約」——後者得靠契約測試把生成的契約拿去比對真實回應，才補得上。一份漂亮但跟實作對不上的 OpenAPI，比沒有更危險，因為它讓所有下游都相信了一個謊。

REST 在四根軸上的賭注很清楚：資料形狀由 server 的端點決定（你打哪個端點，回什麼欄位是固定的）；契約偏軟（靠 OpenAPI 描述，不強制）；傳輸用 JSON 文字，肥但人看得懂；而它最大的贏面在第四根軸——**它順著 HTTP 的紋理長，所以基礎設施白送**。一個 `GET /products/42` 天然就能被 CDN、被瀏覽器、被反向代理快取，因為它就是一個標準的、可快取的 HTTP GET。這件事 GraphQL 和 gRPC 都得拿東西來換。

REST 的痛點，正是開場那一幕。資源拆得乾淨、每個端點形狀固定，於是「一個畫面要好幾種資源」時，你要嘛被迫多打幾次（under-fetch，欄位不夠得追加呼叫），要嘛某個端點為了「以防萬一」回了一大坨用不到的欄位（over-fetch）。端點的形狀是 server 定的，client 只能照單全收。GraphQL 就是衝著這個痛點來的。

## GraphQL：把點菜權交給 client

GraphQL 把第一根軸整個翻轉過來：**資料的形狀，改由 client 在請求裡宣告。** 它只開一個端點（通常是 `POST /graphql`），client 送上去的不是「打哪個資源」，而是一段 query——一棵描述「我要哪些欄位、欄位底下還要哪些欄位」的樹。server 照著這棵樹的形狀，把資料填回去。開場那五次往返，在 GraphQL 下塌成一次：一個 query 同時要訂單、要訂單底下的買家名、要訂單底下三個品項的縮圖 URL，而且明確只要這幾個欄位——訂單那 50 個內部欄位一個都不沾。一次往返，payload 還更小。

這套機制能成立，靠的是一份**強型別的 schema**。server 用 schema 定義「有哪些型別、每個型別有哪些欄位、欄位是什麼型別、哪些是查詢入口」。client 的 query 會被拿來對著這份 schema 驗證——你問了一個不存在的欄位，請求在執行前就被擋下。所以 GraphQL 在「契約硬度」這根軸上的位置很微妙：**schema 本身是硬的（型別是強制的），但「這次要哪些欄位」是 client 動態決定的（軟的）**。它不像 gRPC 那樣編譯期就把方法簽章釘死，但也遠比「一份沒人強制的 OpenAPI」要硬。

代價從哪裡冒出來？三個地方，每一個都不顯然，而且都得在 server 端防。

**第一個是 N+1，藏在「照著樹填資料」這個動作裡。** 設想一個 query 拉了 100 筆訂單，每筆訂單底下都要「買家名字」。GraphQL 的執行引擎是按欄位逐個解析（resolve）的：它先解出 100 筆訂單，然後對「買家名字」這個欄位，**一筆一筆**地去解——於是它對 user 表發了 100 次「查這一個 user」的查詢。加上最初拉訂單那一次，就是 1 + 100 次資料庫往返。畫面上看起來只是「一個 query」，底下卻是 101 次 DB 查詢。這不是 GraphQL 的 bug，是它「逐欄位解析」這個執行模型的天然後果。解法是 dataloader 這類批次合併器：它不立刻去查，而是在同一個執行 tick 內把所有「要查 user」的請求攢起來，攢成一批 `WHERE id IN (9, 12, 7, ...)`，一次查回來，再分發給各個欄位。把 1 + N 壓回 1 + 1。但這個防線是**你得主動裝的**——沒裝，N+1 就在那裡，而且通常是上線跑順了、資料長大了、某個 query 被前端改深了一層之後，才在某個午後把 DB 打到冒煙（這個放大機制本身值得單獨講，見〈N+1 查詢〉）。

**第二個是快取，這是 GraphQL 跟基礎設施逆著長的代價。** REST 的 `GET /products/42` 天然可被 HTTP 快取；GraphQL 預設是 `POST /graphql`，body 裡塞著一段每次都可能不同的 query——HTTP 這一層完全沒辦法快取一個 POST，因為它看不進 body、不知道兩個請求是不是要同一個東西。於是 CDN、反向代理、瀏覽器快取，對 GraphQL 預設**整排失效**。你不是不用快取，是得在應用層把它重新發明：欄位級快取、或 persisted query（把固定的 query 事先存在 server，client 只送一個雜湊 id）。GraphQL-over-HTTP 這份規格（目前仍是 draft）正是在補這道裂縫——它規定 server **可以**接受 GET 來跑查詢，而一旦查詢能用 GET 帶著走，瀏覽器快取和 CDN 就重新接得上了；搭配 persisted query，client 送的是一個短短的 SHA-256 雜湊而非整段 query，既縮小請求、又讓 CDN 有個穩定的 key 可以快取。換句話說，GraphQL 要把 REST 白送的快取拿回來，得繞一大圈、補一套機制。

順帶提一個同源的細節：GraphQL 早年有個惡名，叫「**永遠回 200**」。因為它跑在單一 POST 端點上、把錯誤塞進回應 body 的 `errors` 欄位，於是即使某個欄位解析失敗，HTTP 狀態碼還是 200。這會把所有「靠狀態碼判斷成敗」的東西——LB 的健康判斷、監控的錯誤率、client 的重試邏輯——全部騙過去（這跟 REST 那條「別用 200 包失敗」是同一個陷阱的兩種長相）。GraphQL-over-HTTP 規格新增的 `application/graphql-response+json` 這個 content type，就是要讓 server 能名正言順地回真正的 4xx/5xx，把這個歷史包袱拆掉。

**第三個是 DoS 面，藏在「client 能點任意形狀」這個自由裡。** 既然 client 能宣告要哪些欄位、能巢狀多深，一個惡意（或只是寫錯）的深層巢狀 query 就能放大成天量查詢——「文章的作者的所有文章的所有評論的作者的所有文章……」這樣套下去，一個請求就能讓 server 解析出指數級的工作量。REST 的端點形狀是 server 釘死的，沒有這個面；GraphQL 把形狀的決定權交給 client，就得自己長出防線：限制 query 的深度、算 query 的複雜度上限、或乾脆只准跑 persisted query（白名單）。**開放卻不設複雜度上限的 GraphQL 端點，是一個現成的 DoS 入口**，這不是可選的加固，是必備的。

GraphQL 的賭注因此很清晰：它在第一根軸上贏得徹底（client 精準點菜、一次往返聚合多個資源），代價是把第四根軸（跟基礎設施的契合）整個讓出去——快取、錯誤碼、防濫用，三樣 REST 白送的東西，它都得在應用層自己重建。這筆交易什麼時候划算？當你有**多種前端、對同一批資料的欄位需求差異很大**（web 要 20 個欄位、手錶只要 3 個），或**一個畫面天然要聚合好幾個後端**的時候。反過來，一個只有單一 web 前端、以 CRUD 為主的內部系統硬上 GraphQL，換來的是 schema 維護、複雜度限制、快取自建這一串成本，卻完全沒吃到「多端差異化取欄」的好處——這時 REST 更簡單（這種「為了將來可能的彈性、預支現在的複雜度」的取捨，見〈簡單性與過度工程：YAGNI 的兩面〉）。

## gRPC：兩端先在一份 schema 上對好暗號

REST 和 GraphQL 都還站在「人看得懂的文字、瀏覽器友善」這一側。gRPC 走到了另一側，因為它優化的根本不是「對外給瀏覽器用」這個場景，而是**服務跟服務之間**那種高頻、低延遲、頻寬敏感的呼叫。

它的核心決定落在第二根和第三根軸上，而且都押到極端。**契約硬度**：gRPC 用 Protobuf 當 IDL（介面定義語言），你在一份 `.proto` 檔裡寫死「有哪些 service、每個 service 有哪些方法、每個方法吃什麼 message、吐什麼 message」，然後用工具從這份 `.proto` 生成**雙邊的 stub**——client 端生一份呼叫用的 stub，server 端生一份實作用的骨架。於是「呼叫一個遠端方法」在你的程式裡長得就像呼叫一個本地函式，型別由編譯器替你檢查。這是三者裡最硬的契約：方法簽章對不上，**根本編不過**，錯誤在編譯期就被攔下，輪不到上線時才炸（這份 `.proto` 怎麼在不破壞舊 client 的前提下演進——加欄位、留欄位號、別重用刪掉的號——是另一套學問，見〈序列化與 schema 演進〉）。

**傳輸效率**：gRPC 把資料壓成 Protobuf 的二進位格式，跑在 HTTP/2 上。這個「省」省到什麼程度，可以手算到底，而且小到位元組。看 Protobuf 怎麼把一個 message 編上線。

假設一個 message 只有一個欄位 `x`，欄位號是 1，型別 int32，值是 150。Protobuf 不傳欄位名「x」，它只傳一個**標籤位元組（tag，由欄位號和型別算出）**加上**值**——這裡標籤是 `0x08`，值 150 用變長的 varint 編碼成兩個位元組 `0x96 0x01`，整個 message 上線就是三個位元組：`08 96 01`（tag／varint 這套 wire format 怎麼從欄位號算出來、為什麼這樣設計，是 owning 章的事，見〈序列化與 schema 演進〉）。

把同一份資料寫成 JSON：`{"x":150}`，光是大括號、引號、欄位名、冒號就吃掉一堆位元組，整串是 9 個 ASCII 字元、9 個位元組。再加一個欄位 `y=3`，JSON 的 `{"x":150,"y":3}` 是 15 個位元組；Protobuf 是 `08 96 01 10 03`，5 個位元組（`0x10` 是欄位 2 的標籤、`0x03` 是值 3）。**15 對 5，三倍。** 而且這個差距會隨著欄位變多、欄位名變長而拉得更開——JSON 每一筆都得把欄位名重抄一遍，Protobuf 只送一個標籤位元組。這裡藏著一個實務上的細節：欄位號 1 到 15 的標籤只佔一個位元組，欄位號 16 起就要兩個位元組（因為 `16 << 3 = 128`，超過一個 varint 位元組裝得下的範圍）——所以 Protobuf 的慣例是把**最常出現的欄位編成 1 到 15**，把那一個位元組省在刀口上。

但這個「省」的價值，得**量級夠大才顯現**。對一個一天被打幾百次的低頻呼叫，省下來的那幾百個位元組，會被「不能用 `curl` 直接戳、出錯時 `tcpdump` 抓到一團二進位看不懂、要維護 stub 生成的建置流程」這些工程成本整個蓋過。Protobuf 省頻寬的回報，要在「服務間每秒幾萬次呼叫」這種量級下才結算得出來。

gRPC 還白送一樣 REST 和 GraphQL 都沒有的東西：**原生的 streaming**。因為它跑在 HTTP/2 上，而 HTTP/2 的一條連線天生能跑多條獨立的雙向 stream，gRPC 順勢開出四種方法形狀。最普通的是 **unary**——送一個請求、收一個回應，就是一次普通的遠端呼叫。然後是 **server-streaming**：client 送一個請求，server 回一**串**訊息，像訂閱一個持續吐資料的源。反過來是 **client-streaming**：client 連續送一串訊息（比如上傳一大批指標）、server 最後回一個總結。最自由的是 **bidirectional streaming**：兩邊各自開一條 stream、互不阻塞地隨意收發，server 可以收一條回一條、也可以等收完一批再一起回。這四種裡，後三種在 REST 的請求-回應模型裡是擠不進去的——你得另外搬出 WebSocket 或 SSE 才做得到（那些是另一個主題，見〈SSE、long-poll 與 WebTransport〉）。

代價呢？就是它跟瀏覽器**逆著長**，而且這個逆是物理性的、繞不開。gRPC 倚賴 HTTP/2 的底層 framing——特別是 **trailers**（在回應主體之後才送的一段 metadata，gRPC 拿它傳呼叫狀態）。問題是，瀏覽器給 JavaScript 用的 `fetch`、`XMLHttpRequest` 這些 API，**根本不暴露 HTTP/2 的 trailers 和 stream 的底層控制**，而且短期內看不到會暴露的跡象。於是瀏覽器沒辦法直接講純 gRPC。補法是 **gRPC-Web**：瀏覽器講一種改過的、不需要 trailers 的協定，中間擺一個 proxy（Envoy 是內建支援、最常見的選擇）把它即時翻譯成後端的純 gRPC。這就是為什麼 gRPC 幾乎只出現在後端對後端的內網裡——一旦要直接面對瀏覽器，你就得多養一個翻譯層，那層的延遲和運維，常常把 gRPC 省下來的好處又吃回去一截。

## 同一個訂單頁，三種範式各自會怎麼跑

把開場那個訂單詳情頁放回來，讓三種範式各跑一遍，取捨就具體了。

REST 下，若資源端點各自獨立，client 要 `GET /orders/1`、`GET /users/9`，再加三個 `GET /products/{id}`，共五次往返；後三次還得等第一次回來、解析出 id 才能發。在 200 ms RTT 的行動網路上、若這些呼叫有序列依賴，那就是肉眼可感的等待——但每一個 GET 都能被 CDN 和瀏覽器快取，第二次打開同一頁，那些縮圖端點可能根本不碰後端。

GraphQL 下，一個 query 把訂單、買家名、三個縮圖 URL 一次取回，只取真正用到的欄位，單次往返、payload 更小。但 server 端得當心：這個 query 若被改成「拉 100 筆訂單、每筆各要買家」，沒裝 dataloader 就是 1 + 100 次 DB 查詢；而且這次往返是 POST，CDN 那層幫不上忙，要快取得自己在應用層做。

gRPC 在這個場景反而**最不合適**——它是給後端內部用的，瀏覽器和手機 App 直接講 gRPC 要過 gRPC-Web proxy，省下的那點 payload 換不回多養一層的成本。但把鏡頭推到後端內部：當這個「訂單服務」要去問「使用者服務」「商品服務」拿資料時，那幾跳**服務間**的呼叫，正是 gRPC 的主場——強型別、二進位省頻寬、低延遲，每秒成千上萬次地跑。

於是真正成熟的系統，答案常常不是三選一，而是**三個都用、各守一段**：對外那層給瀏覽器和第三方的，用 REST 吃 CDN 快取與通用性；一個 BFF（backend-for-frontend）聚合層，用 GraphQL 替差異很大的多種前端精準組裝資料；而後端服務彼此之間，用 gRPC 高效地互打。它們不互斥，因為它們本來就在解三個不同位置的問題。

## 為什麼是這三種形狀

退一步看，這三種範式不是三個人對「API 該長怎樣」的三種品味，而是四根軸上**三組不同的賭注**，各自押在不同的壓力下勝出。

REST 押的是「順著 HTTP 的紋理長」。它把資料形狀的決定權留在 server、契約做得偏軟、傳輸用人看得懂的 JSON——換來的是整個 HTTP 生態的快取、可讀、通用性全部白送。它的痛點（over/under-fetch、多次往返）是這個賭注的必然找零。

GraphQL 押的是「把形狀的決定權交給 client」。它一次解決了 over/under-fetch，讓多端各取所需、讓一次往返聚合多源——代價是它跟 HTTP 基礎設施逆著長，快取、錯誤碼、防濫用三樣都得在應用層自己重建，還多背了 N+1 這個藏在執行模型裡的陷阱。

gRPC 押的是「兩端先在一份 schema 上把暗號對死」。它把契約硬度和傳輸效率推到極端，換來編譯期型別安全、二進位的頻寬節省、和原生的雙向 streaming——代價是它放棄了「對瀏覽器友善」「人能直接讀」這一整側，於是只在後端內網、高頻呼叫的量級下才結算得出回報。

理解了這三組賭注，你就不會再問「哪個最好」。你會問的是另一組問題：這個介面是給誰用的——瀏覽器、第三方，還是後端自己？它跟基礎設施是順著長還是逆著長，逆著長要補哪幾套東西？省下來的那點頻寬，在它的呼叫量級下結算得出來嗎？答案會把你直接指向其中一種——或者，更常見地，指向「在系統的不同位置，三種都用上」。
