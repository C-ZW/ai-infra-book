# RBAC 與 ABAC：兩種授權模型

認證之後，請求帶著一個已驗明的身分走進來——「這是 user 8472，他是真的」（認證怎麼證明這點，見〈認證與授權〉）。接著系統要回答一個看似簡單的問題：**他能不能做這件事？** 點下「刪除這份合約」的那一瞬間，後端必須在幾毫秒內給出 allow 或 deny。

天真的做法是把答案直接寫進程式裡。`if user.is_admin: allow`。一行解決。然後產品要一個「能改文件但不能刪」的角色，於是變成 `if user.is_admin or user.is_editor`。再來要「行銷部的 editor 只能碰行銷的文件」，於是 `if user.is_admin or (user.is_editor and doc.dept == user.dept)`。再來是「但週末不准匯出」「但機密等級高於自己的不能讀」「但這份文件的擁有者授權過的協作者例外」。每加一條規則，這串條件就長出一節，散落在十幾個 endpoint 裡，沒有人能回答「到底現在誰能刪這份合約」這種稽核必問的問題。

授權模型——RBAC、ABAC，以及它們之間的光譜——存在的全部理由，就是把這串失控的 `if` **從業務程式碼裡抽出來，變成一個能集中管理、能稽核、能演化的東西**。它們不是兩個競爭的「正確答案」，而是同一條取捨軸上的兩個位置：一端是**能一眼看清「誰能做什麼」但表達力有限**，另一端是**什麼條件都寫得出來但你很難說清「誰能做什麼」**。理解這兩種模型，本質上是理解這條軸——以及為什麼大多數真實系統最後落在中間。

## 把權限從使用者身上挪開：角色這層間接

RBAC（role-based access control，角色式存取控制）的核心動作，是在「使用者」和「權限」之間插入一層**角色**。不是「user 8472 能 `delete_contract`」，而是「user 8472 是 `legal-manager`」「`legal-manager` 這個角色能 `delete_contract`」。

這層間接看起來只是多繞一步，但它買到的東西很實在。權限不再黏在個別使用者身上，而是黏在一個**有業務意義的命名集合**上。新人報到，分配 `legal-manager` 角色，他立刻擁有那個角色該有的一切；離職，撤掉角色，權限一次清乾淨。法務部門整體要多一個「能簽核退款」的能力，你改一次 `legal-manager` 角色的權限集，所有掛這角色的人同步生效——而不是去一個個使用者身上補。**使用者與權限之間是多對多，直接連會爆炸成 m×n 條關係；插入角色後從乘積級降到加總級（單一橋接角色的理想情形就是 m + n 條）**，這是這層間接最樸素的價值。

RBAC 不是某家廠商的發明，而是一個被反覆形式化、最後寫進國家標準的模型。它在 1992 年由 David Ferraiolo 與 Rick Kuhn 形式化；2000 年，這個模型與 Sandhu 等人 1996 年的框架整併成統一的 NIST RBAC 模型；2004 年 2 月 11 日成為美國國家標準 **ANSI/INCITS 359-2004**，2012 年修訂為 359-2012（並於 2022 年重新確認）。標準把 RBAC 拆成四個堆疊的元件，理解它們就理解了 RBAC 的全貌。

最底層是 **Core RBAC**：使用者、角色、權限，加上兩組指派關係（user-role assignment、permission-role assignment），以及一個常被忽略的元素——**session（工作階段）**。session 是這裡第一個非顯然的細節：標準裡使用者不是「擁有」角色就自動帶著全部權限走，而是在一次 session 裡**啟用（activate）**他被指派角色的一個子集。這讓最小權限原則（least privilege）有了落腳處——一個同時是 `developer` 和 `db-admin` 的人，做日常開發時可以只啟用 `developer`，需要動資料庫時才啟用 `db-admin`，把高權限的曝險窗口收到最小。很多輕量 RBAC 實作直接跳過 session、把「指派」當「生效」，這在功能上能跑，但丟掉的正是這個收斂曝險的能力。

往上是 **Hierarchical RBAC**：角色之間可以繼承。`senior-engineer` 繼承 `engineer` 的全部權限再加幾條，你不必把基層權限在每個資深角色裡重抄一遍。這對應真實組織的層級結構，省掉大量重複。

最上層是 **Constrained RBAC**，加入**職責分離（separation of duty, SoD）**約束——這是 RBAC 真正展現它「可稽核」價值的地方。靜態職責分離（SSD）禁止同一個人同時被指派兩個互斥角色：例如一個人不能既是「付款申請人」又是「付款核准人」，否則他能自己申請、自己核准、把錢轉給自己。動態職責分離（DSD）更鬆一點——你可以**擁有**兩個衝突角色，但**不能在同一個 session 裡同時啟用**。SoD 是合規稽核（SOX、各種金融法規）反覆要求的東西，而 RBAC 能把它表達成一條清楚的、可被機器檢查的約束。能乾淨地寫出「這兩個權力不能落在同一人身上」，是 ABAC 那種純規則模型反而做起來彆扭的事。

## RBAC 的甜蜜點，與它撞牆的地方

RBAC 最被低估的優點是**可枚舉**。因為權限是「使用者 → 角色 → 權限」這條固定的靜態鏈，你可以**離線**回答稽核最愛問的兩個問題：「誰能做 X？」（找出含該權限的所有角色，再找出掛這些角色的所有人）和「這個人能做哪些事？」（展開他的角色）。這兩個查詢不需要知道任何「請求當下的狀況」，純粹是圖上的展開。對一個要過審計、要回答監管機構「哪些員工有權存取客戶資料」的系統，這個性質幾乎是不可替代的。

但 RBAC 有一個會撞牆的地方，而且撞得很有規律。**角色是全域的、靜態的——它無法表達「跟請求當下的資料有關」的條件。** 「使用者只能讀**自己**部門的文件」這句話，角色表達不了，因為「自己的部門」要到請求進來、看到是哪個使用者、要碰哪份文件時才算得出來。RBAC 唯一能硬塞的辦法，是為每個部門做一個角色：`editor-marketing`、`editor-finance`、`editor-legal`……

這就是 **role explosion（角色爆炸）** 的起點，而它的數學是無情的。角色數隨「正交的權限維度」**相乘**膨脹。算一個具體的例子：假設你有 3 種操作（read / write / delete）、5 個部門、再加「自己建立的 vs 全部」2 種範圍。要用純角色覆蓋每一種組合，你需要 3 × 5 × 2 = **30 個角色**。現在加入第四個正交維度——「僅上班時間」——角色數直接翻倍到 **60**。再加一個「機密等級 ≥ 自己」的維度，又翻幾倍。每多一個維度，你不是加幾個角色，而是**乘上**那個維度的基數。

```
RBAC：角色數 ∝ 維度基數的乘積
  operations × departments × scope × time × clearance ...
       3     ×      5      ×   2   ×  ...  ×    ...

ABAC：規則數 ∝ 維度的加總
  rule(operation) + rule(dept match) + rule(time) + rule(clearance) ...
```

撐到幾十、幾百個角色時，RBAC 原本最大的優點——可稽核——反而崩了：沒有人能說清 `editor-marketing-readonly-businesshours` 和 `editor-marketing-fulltime` 到底差在哪、誰該有哪個。**角色爆炸不是「再多建幾個角色」能解的問題，它是模型選錯的症狀**——你真正需要的，是把那些「跟資料、跟環境有關」的條件，從角色裡抽出來，變成請求當下才求值的東西。

## 把判斷搬到請求當下：屬性這條路

ABAC（attribute-based access control，屬性式存取控制）就是這個抽法的極致。它不問「你有沒有某個角色」，而是在請求進來的那一刻，對一條**規則（policy）**求值，規則裡用的全是**屬性**：

- **主體屬性**：誰在發請求——部門、職級、所屬團隊、機密許可等級。
- **資源屬性**：他要碰什麼——這份文件的擁有者、它的機密等級、它屬於哪個專案。
- **環境屬性**：在什麼情境下——現在幾點、來源 IP 在不在公司網段、用的是不是受管裝置。

於是前面那條「使用者只能讀自己部門的文件」，在 ABAC 裡就是一條規則：`permit if subject.dept == resource.dept`。「上班時間才能匯出」是 `permit if action == "export" and time in business_hours`。這些規則**隨維度相加**而不是相乘——加一個條件就多一條（或多一個合取項），不會引發爆炸。NIST 在 2014 年的 **SP 800-162** 把 ABAC 形式化定義為：透過對主體、物件、操作、環境的屬性求值一套 policy 來決定授權。它權威定義裡那句最關鍵的話是——**改變屬性值，就改變了授權結果，不必去動任何「使用者-角色」或「角色-權限」的指派**。授權變成動態的、組合式的。

代價在哪？代價正好是 RBAC 的優點的反面：**ABAC 很難回答「到底誰能做 X」**。因為答案取決於請求當下的環境屬性——同一個人，在公司網段內、上班時間能匯出，在家、半夜就不行。「誰能匯出」這個問題沒有一個靜態的答案，它是「在什麼條件下、誰能匯出」。你不能像 RBAC 那樣離線展開一張權限表交給稽核員。同樣難的是**測試**：屬性組合是笛卡爾積，規則集一大，就有覆蓋不到的死角——某一組罕見的屬性組合意外被 allow，而你根本沒測到那一格。一條 ABAC 規則寫錯一個比較符，可能要到某個特定使用者在某個特定時間碰某份特定文件時才爆出來。

還有一個更實務的痛：**「為什麼被拒？」很難回答。** RBAC 下，被拒就是「你沒有帶這個權限的角色」，一句話講完。ABAC 下，一條請求可能要過好幾條 policy、求值十幾個屬性，最後 deny——是哪個屬性、哪條規則擋的？沒有好的工具，這是個偵錯惡夢。這也是為什麼成熟的 ABAC 系統都要配上**決策日誌**（記下每次判斷用了哪些屬性、命中哪條規則）和**預設 deny**的兜底（沒有任何規則明確 allow 就拒絕，絕不 fail-open）。

## 規則住在哪裡：把授權判斷拉出應用

ABAC 把判斷變成「對規則求值」，這自然帶出一個架構問題：**這些規則住在哪、由誰求值？** 如果規則還是散在各個服務的程式碼裡，那不過是把 role explosion 換成了 rule explosion，沒解決集中管理的問題。

業界對這題的標準答案，是把授權判斷**從應用裡拉出來**，變成一個獨立的決策層。XACML（OASIS 的 eXtensible Access Control Markup Language，3.0 是現行 OASIS 標準）給出的是這套架構的經典詞彙，至今仍是理解它的最好框架——即使 XACML 本身那套 XML policy 語言已經很少人新採用了：

- **PEP（Policy Enforcement Point，執行點）**：擋在資源前面的關卡。它攔截請求、把它包成一個授權問題、丟給決策者、然後**執行**回傳的 allow/deny。它只管攔截與執行，不懂規則。
- **PDP（Policy Decision Point，決策點）**：真正的大腦。它拿到請求、載入相關 policy、求值，吐出 permit 或 deny。
- **PIP（Policy Information Point，資訊點）**：PDP 求值時若缺屬性（「這個使用者的部門是什麼」「這份文件的機密等級多少」），向 PIP 要。
- **PAP（Policy Administration Point，管理點）**：人在這裡撰寫、管理 policy。

這套 PEP/PDP/PIP/PAP 的拆分是 ABAC 落地的骨架，它的價值是**把 policy 與業務程式碼徹底分離**——policy 變成可以獨立撰寫、版本控制、審核、測試的工件，而不是埋在 `if` 裡的字面量。但它也暴露了 ABAC 的一個故障模式：**PDP 是一個新的同步相依**。每個敏感操作前都要問一次 PDP，如果 PDP 在獨立行程、甚至獨立服務，這就是請求路徑上多出來的一跳——它的延遲加進你的延遲，它的可用性綁住你的可用性。實務上的解法是把 PDP 當 sidecar 跑在每台機器上、policy 與屬性盡量本地快取，讓求值不必跨網路；但快取又帶來屬性新鮮度的問題——使用者剛被降權，PDP 的本地快取若還是舊的部門/許可等級，授權就會用過期的資訊放行。**授權決策的正確性，受限於它讀到的屬性有多新**，這是把判斷搬到請求當下必然欠下的債。

XACML 那套笨重的 XML 語言今天已大多被更輕的工具取代。**OPA（Open Policy Agent，2021 年 1 月成為 CNCF 畢業專案）** 用 Rego 這個宣告式語言寫 policy，常以 sidecar 形式跑、做雲原生與 API 層的通用授權判斷；**AWS Cedar**（2023 年開源）是一個專為授權設計、能同時表達 RBAC 與 ABAC、且**可被形式化分析**（能自動證明「這兩條 policy 是否等價」「這條 policy 是否永遠拒絕某類請求」）的語言。它們是同一個 PEP/PDP 架構在 2026 年的長相——名字換了，骨架沒變。

## 大多數系統落在中間

把 RBAC 和 ABAC 擺成對立的兩端，是教學上的方便，卻是工程上的誤導。**真實世界裡幾乎沒有純 ABAC 的系統，最常見的形態是「RBAC 當骨架、屬性當微調」**——NIST 自己在〈Adding Attributes to Role-Based Access Control〉(Kuhn, Coyne, Weil, 2010) 裡就明確推薦這條混合路線：用角色框住「最大權限範圍」（保留 RBAC 好稽核、能快速估算一個人風險曝險的優點），再用少量屬性條件去收窄，**避免為了涵蓋每一種偶發情境而把角色炸開**。

具體長這樣：你仍然有 `editor` 這個粗粒度角色——它回答「這個人大致上能編輯文件」，稽核員看一眼就懂。然後你不為每個部門做一個 editor，而是讓 `editor` 帶一個屬性條件：`editor` 角色 **且** `resource.dept == subject.dept`。角色管「你屬於哪一類」，屬性管「在這類裡，你具體碰得到哪些」。前者可枚舉、好稽核；後者細粒度、不爆炸。兩邊各取所長。

這個混合也解釋了授權模型其實是一條**光譜**，不是兩個點。從最原始到最有表達力：

| 模型 | 判斷依據 | 強處 | 撞牆處 |
|---|---|---|---|
| ACL | 資源直接列「誰能做什麼」 | 最直觀、零學習成本 | 規模化即崩，難回答「某人能碰哪些」 |
| RBAC | 使用者 → 角色 → 權限 | 可離線枚舉「誰能做 X」、易稽核、能表達 SoD | 撐不起資源/環境相關條件，導致角色爆炸 |
| RBAC + 屬性 | 角色為主、少量屬性微調 | 保留可稽核、補上動態條件 | 屬性條件散落各處會難維護，需收斂 |
| ABAC | 對任意屬性求值 policy | 任意維度、細粒度、規則集中可改 | 「誰能做 X」難枚舉、規則難窮舉測試、難解釋 deny |
| ReBAC | 主體與資源之間的關係圖 | 關係主導場景的天然表達 | 需維護關係圖，查詢變圖遍歷 |

最後那一行——**ReBAC（relationship-based access control）**——值得單獨點出，因為有一大類授權其實兩種主流模型都表達不好：權限由**關係**定義。「文件擁有者的同團隊成員可讀」「資料夾的編輯者，對其下所有子文件都是編輯者」——這些既不是角色（不是全域的身分類別），也不容易拆成獨立屬性（它是主體與資源之間、可能多層傳遞的連結）。Google 在 2019 年 USENIX ATC 發表的 **Zanzibar** 論文（Google 內部給 Drive、YouTube、Cloud 等幾乎所有產品授權的系統）把這條路推成了顯學：把授權建模成一張**關係元組（relationship tuple）**的圖——`doc:readme#viewer@user:8472`、`group:eng#member@user:8472`——授權判斷變成「從這個使用者出發，沿著關係邊，能不能走到這個資源的這個權限」的圖可達性查詢。受它啟發的開源實作（如 CNCF 的 OpenFGA）讓這條路普及到 Google 以外。ReBAC 在協作、檔案共享、社交圖這類「權限天生是關係」的領域，比硬塞 RBAC 或 ABAC 都乾淨——但代價是你得維護那張關係圖的一致性，而原本一次屬性比較的判斷，現在可能變成一趟圖遍歷。

## 一個能讓你選錯的判準，與一個會反咬的習慣

要在這條光譜上選位置，有一個出奇好用的判準：**數一數你的授權決策依賴幾個維度，以及其中有沒有「跟請求當下的資料/環境有關」的。**

- 維度只有一個、就是「職務」（admin / editor / viewer / billing），而且權限對齊組織結構——**RBAC**，別過度設計。在這裡上 ABAC 是拿規則引擎去解一個查表就夠的問題。
- 開始出現 `editor-but-only-marketing`、`viewer-except-finance` 這種角色——這是 **role explosion 的警報**，代表你撞牆了。往 RBAC + 屬性走。
- 決策本質上依賴資源屬性或環境（「只能存取自己部門的」「機密等級夠才能讀」「只在公司網段內可匯出」）——RBAC 表達不了，需要 **ABAC**。
- 權限由關係定義（擁有者、協作者、團隊成員、繼承的資料夾權限）——**ReBAC**。

關鍵不是「哪個模型先進」，而是「我的授權有幾個維度、它們是靜態的還是依賴當下的」。**ABAC 不是 RBAC 的升級版**——它在低維場景下是純粹的過度工程，把一個查表問題包裝成難測、難稽核、難解釋的規則引擎。

最後一個會反咬的習慣，跨所有模型都成立，值得記死：**永遠把授權綁到「能力」，不要綁到「角色名字」或「身分」。** 把業務判斷寫成 `if role == "manager"`，看起來最直接，卻是給未來埋的雷——當「manager」這個角色要改名、要拆成兩個、要把某項權力下放給別的角色時，你得回去翻遍程式碼裡每一個寫死了 `"manager"` 的地方。正確的形狀是 `if can("approve_refund")`——程式只問「這個請求有沒有核准退款的**能力**」，而「哪些角色、在什麼屬性條件下擁有這個能力」是授權層的事，可以獨立演化。**角色和屬性是權限的來源，不是判斷的對象；程式該問的永遠是能力。** 守住這條，無論你後來從 RBAC 滑到 RBAC+屬性、再滑到 ABAC，業務程式碼都不必跟著動——因為它從頭到尾只問了一個它真正關心的問題：這個人，能不能做這件事。

## 為什麼是這個形狀

退一步看，這兩種模型——以及它們之間的光譜——都是從同一個張力裡長出來的：**你想要「一眼看清誰能做什麼」（可稽核、可枚舉），同時又想要「表達任意細的條件」（細粒度、跟資料與環境有關）——而這兩者天生互斥。**

越靜態、越可枚舉的模型（ACL、RBAC），就越答不出「跟當下狀況有關」的問題；越動態、越有表達力的模型（ABAC、ReBAC），就越難回答「到底誰能做 X」。RBAC 把判斷**前置**成靜態的角色指派，換來可稽核，付出表達力；ABAC 把判斷**推遲**到請求當下對屬性求值，換來表達力，付出可稽核與可測試性。中間那條混合路線之所以是大多數系統的歸宿，正是因為它在這條軸上找了個不偏激的點：用角色框住大局保住稽核，用屬性收窄細節保住精度。

下次你在一個系統裡看到「角色」和「policy 規則」同時存在、有人問為什麼不統一成一種，你會知道那通常不是設計沒收斂，而是它在這條軸上**刻意**站在了中間——用 RBAC 的可枚舉守住「誰大致能做什麼」這個稽核必答的問題，再用屬性補上那些角色撐不住、卻又非有不可的動態條件。授權模型的全部工程智慧，就在於認清你沒法兩端通吃，然後為你這個系統，挑一個誠實的位置。
