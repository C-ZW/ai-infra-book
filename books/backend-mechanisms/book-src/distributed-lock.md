# 分散式鎖：Redlock、fencing token 與「鎖其實是租約」

兩個 worker 同時醒來，都打算處理同一個訂單的退款。它們跑在不同機器上、共用同一個資料庫，而退款這件事一旦做兩次，就是真的把錢退兩次出去。你想擋住這件事，第一個念頭幾乎是反射——「加一把鎖」。單機上這再自然不過：一個 `mutex`、一段 `synchronized`，作業系統替你保證同一時間只有一個執行緒進得了臨界區。你想要把那個熟悉的保證原封不動搬到跨機器的世界。

問題是，那個保證的根基在跨機器時整個垮掉了。單機 mutex 之所以可靠，是因為持有它的執行緒和管理它的核心活在**同一塊記憶體、同一個生死共同體**裡：執行緒一旦死了，它的整個行程連同那把鎖一起消失，核心立刻知道，不會有「鬼魂還握著鎖」這種事。跨機器沒有這塊共享記憶體，也沒有這個共同體。鎖必須由一個**外部協調者**（Redis、ZooKeeper、etcd、一張資料庫表）發放，而拿鎖的行程和發鎖的協調者之間，隔著一條會延遲、會丟包、會分區的網路。這條網路一插進來，整個故事就變了。

## 為什麼一插進網路，鎖就不再是鎖

把單機 mutex 拆開看，它其實默默依賴一件事：**持有者一旦不再持有，這個事實是「即時且確定」地被所有人知道的**。執行緒釋放鎖、或執行緒崩潰，核心當下就知道鎖空了。跨機器版本想複製這個，立刻撞上一道牆——協調者**無法可靠地知道遠端的持有者還活著沒有**。

持有者可能崩潰了，可能只是很慢，可能網路把它和協調者隔開了。從協調者這端看出去，這三種情況長得一模一樣：都是「那個 client 不再跟我說話了」。它分不出「慢」和「死」——這是整個分散式世界共有的先天侷限，逾時是唯一能用的偵測手段，而逾時永遠只能告訴你「它沒在我等的窗口內回應」，證明不了「它死了」（這個侷限本身值得單獨講透，見〈失敗偵測：逾時是唯一手段、慢與死分不開〉）。

這道牆逼出一個無法迴避的設計決定。如果協調者**永遠等**持有者主動還鎖，那只要持有者崩潰一次、再也不回來，這把鎖就永久卡死，後面所有人陪葬。為了不被一個死掉的持有者拖垮，鎖**必須**帶一個過期時間：拿鎖時講好「這把鎖租給你 30 秒，到時間我自動收回」。換句話說——

**分散式鎖在物理上不可能是「鎖」，它只能是一紙「租約」（lease）。** 你從沒真正「擁有」它，你只是「租到了它一段時間」。

這個從「鎖」到「租約」的退讓不是實作偷懶，是被網路的本性逼出來的唯一活路。而它一被逼出來，就在系統裡撕開一道**靠鎖本身永遠補不起來**的裂縫。

## 那道補不起來的裂縫

裂縫的形狀是這樣的：**租約是按時間到期的，但持有者並不會在租約到期的那一刻收到通知說「你的鎖沒了」。** 時間到了，協調者那邊單方面把鎖收回、發給下一個人；可持有者自己這邊，可能正卡在一個它毫不知情的暫停裡，醒來後仍然天真地以為自己握著鎖。

把它放慢成一條時間線，用具體數字走一遍。Client A 在 `t=0` 拿到鎖，TTL 設 30 秒。

```
t=0     A acquires lock, lease TTL = 30s
t=5     A enters a 40s stop-the-world GC pause  (or OS swap-out, or
        network black hole — A is frozen, unaware)
t=30    lease expires; coordinator hands the lock to B
t=31    B acquires the lock, starts writing to the resource
t=45    A wakes up from GC. It never saw the expiry.
        A still believes it holds the lock — and writes to the resource.
        ── now A and B are BOTH in the critical section ──
```

最致命的細節在 `t=45`：A 醒來時，它手上那個「我持有鎖」的判斷，是 `t=5` 凍結前的記憶。在它的主觀時間裡，一秒都沒過。它不會去重新確認，因為它根本不知道自己睡了 40 秒。於是 A 和 B 同時動同一筆資料——退款退兩次、檔案寫花、庫存扣穿。

這裡要看清一件反直覺的事：**這道裂縫，再貴、再可靠的鎖服務都堵不住。** 換成跑共識的 ZooKeeper、etcd，它們能保證「同一時刻全世界只發出一張有效租約」——可它們管不到 A 醒來後**還拿著一張過期的舊憑證去敲資源的門**。GC 暫停發生在 client 的行程裡，協調者看不見；A 的「我以為我持有」這個誤判，活在 A 自己的記憶體裡，不在協調者的視野中。鎖服務的職責止於「發租約、管到期」，止於資源那道門之外。問題出在門內。

所以結論很硬：**只靠鎖，你永遠擋不住「過期持有者的遲到寫入」。** 要堵這道裂縫，得換一個地方下手——不是把鎖做得更可靠，而是讓**資源那道門自己會驗票**。

## fencing token：把正確性從鎖那裡搬到資源這裡

Martin Kleppmann 在 2016 年那篇〈How to do distributed locking〉裡，給了這道裂縫一個乾淨利落的解法，叫 **fencing token（柵欄記號）**。核心只有一句話：**每次發鎖，連帶發一個單調遞增的整數；每次寫資源，都把這個整數帶上；資源端記住它見過的最大值，拒絕任何比它小的。**

回到剛才那條時間線，把 fencing token 疊上去：

```
t=0     A acquires lease, gets fencing token = 33
t=5     A freezes in GC
t=30    lease expires
t=31    B acquires lease, gets fencing token = 34  (always increases)
t=32    B writes to storage with token 34
        → storage records "highest token seen = 34"
t=45    A wakes, writes to storage with its stale token 33
        → storage sees 33 < 34, REJECTS the write
        ── A is fenced off. Only one writer's data lands. ──
```

A 那筆遲到的寫入，不是被鎖擋下的——鎖早在 `t=30` 就跟 A 沒關係了。它是被**資源端那個「33 比我見過的 34 小」的判斷**擋下的。妙處在於：資源端**完全不需要知道**有沒有鎖、誰持有鎖、租約過沒過期。它只認一條死規則——token 只能往上、不能往下。一個拿著過期租約的遲到者，它的 token 必然停在拿鎖那一刻的舊值，而在它沉睡時，世界已經把 token 推高了。它一出手就被自己的舊號碼出賣。

這就是 fencing 的全部精髓，也是為什麼這道防線**非得疊在資源端不可**。鎖服務能做的，是保證「發出去的 token 嚴格遞增」；但「**檢查** token、**拒絕**倒退的寫入」這個動作，只有真正執行寫入的那一端做得了。把這兩件事拆開，正確性的重心就從「易碎的、靠時間判生死的鎖服務」轉移到了「資源自己手上那把可靠的、純比大小的尺」。Kleppmann 把它講得很直白：**這要求 storage server 主動扮演檢查 token 的角色，主動拒絕任何 token 倒退的寫入。** 只發 token 而資源不檢查，等於沒做——那只是一個沒人看的號碼牌。

token 不一定要是新造的計數器。一個關鍵的實務觀察是：**很多協調系統本來就在發單調遞增的數字，你直接拿來用就好。** ZooKeeper 的每筆交易都帶一個全域單調遞增的 64-bit `zxid`，你拿鎖時建立的那個 znode 的 `zxid` 天生就是一個高品質的 fencing token；etcd 的每次修改都配一個全域單調遞增的 `revision`，那個 `revision`（或某個 key 的 `mod_revision`）同樣能直接當 token 用。這不是巧合——一個系統若能保證「全域單調遞增的版本號」，它本身就已經在解共識等級的問題了，而那正是「可靠遞增」這件事的難處所在。

## 難就難在「可靠地遞增」

「發一個遞增的整數」聽起來像最簡單的事——一個 `counter++` 而已。但放回分散式的脈絡，它恰恰是整個機制裡最硬的一塊，也是它把不同的鎖方案分出高下的那條分水嶺。

天真的做法：在某一台 Redis 上放個計數器，每次發鎖 `INCR` 一下。問題立刻來——**那台 Redis 一掛，計數器就沒了**。你說那放多台、每台各一個計數器？那它們會各走各的、彼此對不齊，你拿到的數字根本不再單調。Kleppmann 把這層點破得很透：要在一個會故障的分散式系統裡造出一個**保證全域單調遞增**的數字，本身就**需要一套共識演算法**。

這句話是理解整個分散式鎖版圖的鑰匙。它意味著：fencing token 不是隨便哪個鎖都附送得起的贈品——**能發出可靠 fencing token 的能力，和能跑共識的能力，是同一件事。** ZooKeeper 的 `zxid`、etcd 的 `revision` 之所以能當 token，正因為它們底下就跑著共識協定（ZooKeeper 的 Zab、etcd 的 Raft），那個遞增的數字是共識的副產品（共識為何能扛住這種全域唯一一致的決定，見〈共識：讓一群會當機的機器同意一件事〉）。反過來，一個**不跑共識**的鎖方案，就算它在「互斥」這件事上做得再花俏，也**生不出**可靠的 fencing token——這正是接下來 Redlock 故事的引信。

## Redlock：想用 Redis 把單點補成多數決

先看最樸素的 Redis 鎖。它其實相當漂亮，就一行：

```
SET resource_name <random_value> NX PX 30000
```

`NX` 是「只在 key 不存在時才設」，`PX 30000` 是「30 秒後自動過期」。這一行把「搶鎖」做成了一個原子操作——同一時刻只有一個 client 的 `SET` 會成功，因為 `NX` 擋掉了後到的。`<random_value>` 是每個 client 每次拿鎖時自己生的一個唯一隨機值，它的用處在**解鎖**那一刻才顯出來：你不能無條件 `DEL`，否則會出這種事——你的鎖在 `t=30` 過期了，B 在 `t=31` 拿到新鎖，你在 `t=32` 才慢吞吞地 `DEL`，一刀砍掉的是**B 的鎖**。所以解鎖必須是「先比對 value 是不是我當初寫的那個隨機值，是才刪」，而且這個「比對再刪」必須原子——要嘛用一段 Lua 腳本，要嘛用 Redis 8.4 起提供的 `DELEX key IFEQ <random_value>` 一次做完。

```lua
-- atomic compare-and-delete: only release the lock if it's still mine
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

這個單實例鎖快、簡單，但有個刺眼的弱點：**那台 Redis 是單點。** 它一掛，鎖服務就全沒了。為了去掉這個單點，Redis 作者 antirez 設計了 **Redlock**——把鎖架在 **N 台彼此獨立**的 Redis（典型 N=5，互不複製、互不知道彼此存在）上，用**多數決**取代單點。

Redlock 拿鎖的流程，每一步都有它的理由：

```
1. T1 = now()  (milliseconds)
2. try SET ... NX PX on all 5 instances, in parallel,
   each with a SHORT per-request timeout (e.g. 5–50ms, far below TTL)
3. T2 = now();  elapsed = T2 - T1
4. lock is ACQUIRED only if BOTH hold:
     (a) succeeded on a majority  (>= N/2+1 = 3 of 5)
     (b) elapsed < TTL
5. effective validity = TTL - elapsed - clock_drift_allowance
   (drift allowance ~ 1–2% of TTL, for inter-machine clock skew)
   on failure → unlock ALL instances (even ones you think you missed),
                wait a random delay, retry
```

第 2 步那個「每台用很短的逾時」是為了不讓一台慢吞吞或掛掉的 Redis 把整輪拿鎖卡死——湊不到多數就趕快放棄重來。第 4 步的兩個條件缺一不可：拿到多數（這樣同一時刻不可能有第二個 client 也拿到多數，鴿巢原理保證任兩個多數集必有交集，見〈quorum：R + W > N 的鴿巢原理〉），而且**整輪耗時要遠小於 TTL**——因為你拿鎖本身花掉的時間，是從租約裡實打實扣掉的。第 5 步把這件事算進去：你真正能安全使用的有效租期，是「名目 TTL 減掉拿鎖耗時、再減掉一點時鐘漂移的餘裕（典型抓 TTL 的 1–2%）」。第 5 步失敗時要去**所有**實例解鎖、包括你以為沒設成功的那些（因為你的 `SET` 可能其實到了、只是回應在回程丟了），然後等一個**隨機**延遲再重試——隨機是為了打散互相搶鎖的 client，避免它們步調一致、永遠卡在對撞。

Redlock 確實補掉了單點：掛兩台、剩三台還能運作。它在工程上是個聰明的設計。但它沒能、也不可能補掉前面那道更底層的裂縫——而這正是它引爆一場著名爭論的地方。

## 那場著名的爭論：Redlock 到底安不安全

2016 年，Kleppmann 寫了那篇文章，把矛頭直接對準 Redlock，理由分兩層。

**第一層，也是最致命的：Redlock 生不出 fencing token。** 回到上一節的論證——可靠遞增需要共識，而 Redlock 底下那 N 台 Redis **彼此獨立、不跑共識**，它們之間沒有任何協定能產出一個「保證每次拿鎖都比上次大」的全域數字。有人會說「那就用拿鎖時的隨機 value 當 token 嘛」——不行，隨機值只能比對「相不相等」，**沒有大小順序**，資源端無法判斷「33 比 34 舊」。沒有單調性，就沒有 fencing，就堵不住那道 GC 暫停的裂縫。Kleppmann 的結論很冷：就算 Redlock 演算法本身完美無瑕，**它仍然不安全**，因為它先天交不出資源端驗票需要的那張遞增號碼牌。

**第二層：Redlock 對時間和時鐘做了危險的假設。** Redlock 的多數決安全性，悄悄依賴「各台 Redis 的時鐘以接近的速率前進」。但時鐘會跳——NTP 校正、VM 遷移、虛擬化暫停，都可能讓某台的時鐘**突然往前蹦**，於是那台上的鎖提早過期。Kleppmann 給的場景是：client 1 在 A、B、C 拿到鎖；C 的時鐘往前一跳，C 上的鎖提早失效；client 2 趁機在 C、D、E 拿到鎖——**兩個 client 同時握著各自的「多數」**。Redis 判斷 key 過期用的是 `gettimeofday` 而非單調鐘，而 `gettimeofday` 的手冊明寫它會「不連續地跳動」。一旦這些「網路延遲有界、行程暫停有界、時鐘誤差有界」的同步假設被現實打破，Redlock 的安全性就可能破功，在某個 client 的租約還沒真正過期時，就把鎖發給了另一個。

antirez 隔天就回了一篇〈Is Redlock safe?〉，逐點反駁。他並不接受「Redlock 只配當效率工具」這個定位——他為 Redlock 的正確性辯護，主張它在需要互斥時仍然可用，認為 Kleppmann 的攻擊建立在「時鐘無界跳動、行程無界暫停」這類太極端的假設上。但這場交火真正留下來的，是它逼出的那條分界——而那條分界，其實是 **Kleppmann** 在原文裡就劃下的：**鎖若只是效率最佳化（偶爾撞上只浪費一點 CPU、不壞資料），便宜的 Redis 鎖甚至 Redlock 又快又夠用；若要絕對正確（同時兩個寫入者就會壞資料），才該動用 ZooKeeper、etcd 這種跑共識的系統。** 很多場景下，鎖只是一個**最佳化**——避免兩個 worker 重複跑同一個昂貴的計算、避免同一個 cron 在兩台機器上各跑一次——這種「撞上也只浪費點 CPU」的場合，Redlock 完全夠用。

這場爭論之所以值得每個做這行的人讀一遍，不在於誰贏，而在於它逼出了一條**你選鎖之前必須先想清楚的分界**——而那條線，比任何一個鎖的實作細節都重要。

## 那條分界：你的鎖是為效率，還是為正確性

Kleppmann 那篇文章真正的禮物，是把所有「我需要一把分散式鎖」的需求劈成截然不同的兩半：

- **效率鎖（efficiency lock）**：鎖只是省力。失手了，兩個節點重複做了同一件事，後果是「多花一點成本」——多算一次、多寄一封信、多打一次無關痛癢的 API。資料**不會壞**。
- **正確性鎖（correctness lock）**：鎖在守資料的完整性。失手了，兩個節點同時動同一筆資料，後果是「檔案寫花、資料遺失、永久不一致、給病人錯誤劑量的藥」——**真正的損害**。

這兩半要的東西天差地別，而把它們混為一談，正是分散式鎖事故的源頭。

如果你的鎖是**效率鎖**，那條 GC 暫停的裂縫對你無關痛癢——偶爾兩個 worker 撞上，浪費點算力而已。這時你**不需要** fencing token，一把單實例 Redis 鎖（甚至 Redlock）綽綽有餘，去追求共識級的保證反而是拿大砲打蚊子，還白白多引入一個可用性瓶頸。

如果你的鎖是**正確性鎖**，那麼——記死這句——**光有鎖，無論多貴的鎖，都不夠。** 你**必須**有 fencing token，而且資源端**必須**真的去檢查它。而既然可靠的 fencing token 需要共識，你就該用一個本來就跑共識、本來就會吐單調版本號的系統（ZooKeeper 的 `zxid`、etcd 的 `revision`），把那個版本號一路帶到資源端去驗。鎖本身在這裡只負責**降低**衝突的機率（讓大多數時候只有一個人在臨界區，省下白工），而**消除**衝突造成的損害，是 fencing 在資源端兜底的。鎖管效率，fencing 管正確性——這兩件事在正確性鎖的場景裡，是**兩道分開的防線**，缺了第二道，第一道遲早被一次 GC 暫停穿透。

順帶一提，這也說明了為什麼 fencing 這個機制不只屬於「鎖」。任何「同一時刻應該只有一個寫入者」的場景——比如選出來的那個唯一 leader——都吃同一道裂縫：舊 leader 可能在被廢黜後、還沒意識到自己已經出局時，繼續發號施令。解法同樣是 fencing：給每一任 leader 一個遞增的 epoch / term，資源端只認最新那個（leader 端怎麼把舊王者擋在門外，見〈leader election：選出唯一的寫入點〉）。「鎖」和「leader」表面是兩個題目，底下是同一道裂縫、同一道柵欄。

## 但更多時候，你根本不需要鎖

把這條分界推到底，會浮出一個更省事的結論：**真正需要分散式鎖的場景，比直覺以為的少得多。** 在掏出鎖之前，先過三道篩子：

**一、資源能不能自己做條件寫入？** 如果那筆資料的儲存端支援「帶版本號的條件更新」（CAS、樂觀並發、`UPDATE ... WHERE version = ?`、HTTP 的 `If-Match` ETag），那你壓根不需要鎖。讓兩個 worker 都去搶著寫，誰的版本號對誰就贏，輸的那個拿到衝突、退回重試就好。這其實就是 fencing 的近親——把「誰能寫」的裁決下放到資源端的條件檢查（樂觀並發的完整機制見〈樂觀與悲觀並發控制：CAS、版本與 ETag〉）。

**二、能不能把同一實體的操作都路由到同一個地方序列化？** 如果你能讓「同一個訂單的所有操作」永遠落到同一個 partition、同一個 consumer 上按順序處理（靠 partition key，見〈訊息順序：FIFO 與 partition key〉），那它們天生就不會並發，互斥是免費贈送的，連鎖都不必出場。

**三、偶爾重複執行會不會壞？** 如果不會壞（效率鎖的情形），用最便宜的單實例 Redis 鎖；如果會壞（正確性鎖），那就不是「要不要鎖」的問題，而是「**這個操作本身夠不夠冪等**」的問題——把操作設計成「做幾次結果都一樣」，重複就傷不了你（冪等怎麼從根上化解重複，見〈重複是常態：冪等與去重〉）。

只有當這三道篩子全都漏接——資源無法條件寫入、操作無法路由序列化、而且重複真的會損壞資料——你才真正落到「需要一把帶 fencing token 的強分散式鎖」的格子裡。而即便落到這格，鎖也從不是單獨上場：它和資源端的 fencing 檢查、和操作的冪等性，**一起**才兜得住。把分散式鎖當成並發問題的第一選擇，通常是過度設計，還順手給系統添了一個新的單點與延遲瓶頸。

## 為什麼是這個形狀

退一步看，分散式鎖的整副樣貌，都是從章首那一個無法迴避的事實裡長出來的：**跨機器的協調者，分不清遠端的持有者是慢、是死、還是只是失聯。**

正因為分不清，協調者不敢無限等一個可能已死的持有者——所以鎖必須帶 TTL，於是「鎖」在物理上只能是「租約」。正因為它是租約、按時間單方面到期，持有者就可能在毫不知情的暫停後，攥著一張過期的舊票醒來——於是那道 GC 暫停的裂縫被打開，而它**長在 client 的記憶體裡、不在協調者的視野中**，再可靠的鎖服務都搆不著。正因為鎖服務搆不著門內，唯一堵得住的辦法就是讓資源那道門**自己驗票**——於是有了 fencing token，把正確性的重心從易碎的鎖搬到資源端那把只認大小的尺上。又因為「可靠遞增」這件事本身就需要共識，能不能發 token 就成了「跑不跑共識」的分水嶺，也就劃出了 Redlock 那場爭論的兩岸：效率這邊，便宜的鎖夠用；正確性那邊，非共識＋fencing 不可。

所以「鎖其實是租約」這句話，不是一個聰明的修辭，而是這整個機制的地基。一旦你接受了它——接受你從未真正持有什麼、只是租到了一段隨時可能被悄悄收回的時間——後面的一切，從 TTL、到 fencing、到「先問問自己是不是根本不需要鎖」，就都成了同一個事實合乎邏輯的推論。下次你伸手去拿一把分散式鎖，真正該問的不是「哪個鎖最可靠」，而是：**我這把鎖，是為效率，還是為正確性？如果是後者，守住正確性的那道柵欄，在資源那邊架好了沒有？**
