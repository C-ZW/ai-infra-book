# 漸進式遷移與淘汰：strangler 與 expand-contract

有一類工作，技術上你早就知道該怎麼做，難的卻是「怎麼開始、怎麼收尾」。你面前是一套還在線上、每天跑著真實流量的東西——一個老單體、一條公開 API、一個被上千處呼叫的函式簽章、一個埋在設定檔深處的鍵——它有問題，你想換掉它。新的設計你都想好了。然後你卡住了，因為唯一一個你會的動作是「停下來、把舊的拔掉、把新的接上、再開回去」，而這個動作對一個不能停的東西來說，風險高到沒人敢按下去。

這種「大爆炸切換」（big-bang cutover）的可怕之處，不在於它一定會錯，而在於它**錯了沒有退路**。你在某個維護窗口把開關一扳，新系統接管全部流量。如果它在那一刻暴露出一個測試沒覆蓋到的問題——一個你連想都沒想過的 edge case——那麼此刻所有使用者都中標，而你想退回去時才發現：舊系統已經被你關掉了，狀態可能已經被新系統改過了，你沒有一個乾淨的、可以立刻切回去的舊世界。blast radius 是 100%，而回滾按鈕不存在。

漸進式遷移這整套思路，就是為了把這個「一次性、不可回退」的賭注，拆解成另一種形狀的工作。它的第一原理只有一句話：**讓新舊在一段時間內並存，流量可控地一點一點轉移，每一步都能觀測、都能退回去。** 你不再賭一次切換不出錯；你把切換攤成幾十次小切換，每一次只動一小塊、只影響一小撮流量，出錯了就把那一小塊退回來。風險沒有消失，但它被切碎、被框住、被你逐口吞下，而不是一口氣砸在臉上。

這條路上有兩個經典模式，分別管兩種「不能停的東西」。一個叫 strangler fig，管的是**整套系統**的退役；一個叫 expand-contract（又名 parallel change），管的是**一個介面、一條流程、一個慣例**的演進。它們是同一個第一原理在兩種尺度上的長相，值得分開看清楚。

## 絞殺榕：在舊系統外圍長出新系統

Strangler fig（絞殺榕）這個名字，比多數工程術語更值得知道它的來歷，因為來歷本身就解釋了機制。Martin Fowler 在 2001 年到澳洲昆士蘭的雨林，看到一種藤蔓植物：它的種子落在宿主樹高處的枝椏縫裡發芽，向下伸出氣根，一路長到地面紮根、向上展開樹冠搶陽光，慢慢地把宿主樹整個包覆起來。許多年後，藤蔓自己長成了一棵能獨立存活的樹，而被它包住的宿主樹枯死，只在中間留下一個與藤蔓同形的空殼。Fowler 原本把這個遷移模式叫「Strangler Application」（2004），後來在 2019 年 4 月改名為「Strangler Fig Application」，刻意淡化「絞殺」那層暴力意味、強調它其實是「新東西沿著舊東西長出來」的生長過程。

機制和那株藤蔓一模一樣。你**不去碰那棵宿主樹的內部**——舊系統繼續跑、繼續服務、一行都不動。你在它的外圍長出新系統，並在新舊之間放一層**攔截層**（facade／proxy／路由）。所有進來的請求先到這層攔截層，由它決定：這個請求該送給舊系統，還是已經搬好的新系統。一開始，攔截層把幾乎所有請求都導向舊的；隨著你一塊一塊把功能搬到新系統，攔截層把越來越多請求導向新的。對外面的 client 來說，介面從頭到尾沒變過，它根本不知道底下正在換引擎。

```
phase 1: facade introduced, nearly all traffic to legacy
   client ---> [ facade ] ---> legacy   (almost all)
                       \-----> new       (a thin slice)

phase 2: features moved over slice by slice
   client ---> [ facade ] ---> legacy   (shrinking)
                       \-----> new       (growing)

phase 3: legacy at zero traffic, ready to retire
   client ---> [ facade ] ---> new       (all)

phase 4: facade removed, client talks to new directly
   client ----------------> new
```

這四個相位裡，藏著整個模式的重量。攔截層是那根**流量旋鈕**：你能搬多細的功能、能多快回退，全看你能在哪個粒度上攔截並改寫路由。能按路徑（`/orders/*` 走新的、其餘走舊）攔，你就能一個 endpoint 一個 endpoint 地搬；能按租戶、按使用者 ID 的 hash、按百分比攔，你就能先讓 1% 的流量去試水。出了問題，旋鈕往回擰一格，那一小塊立刻切回舊系統，blast radius 始終被框在「當前這一格」之內。這就是漸進式遷移把 100% 的賭注切成許多 1% 賭注的具體做法。

## 攔截層不是免費的：anti-corruption layer 與兩個系統的對話

天真地想，攔截層只是個 if-else：看請求長什麼樣，決定轉發給誰。但只要遷移稍微深一點，你會撞上一個沒那麼顯然的問題：**並存期間，新舊兩個系統往往不是互不相干的兩條平行線，而是要彼此呼叫的。**

設想你已經把「訂單」搬到新系統，但「庫存」還留在舊單體裡。新系統處理一筆訂單時，得去問舊系統「這個商品還有貨嗎」。反過來，舊單體裡某段還沒搬的邏輯，可能也需要回頭呼叫已經搬到新系統的「訂單」。於是並存期不是「新的歸新的、舊的歸舊的」這麼乾淨，而是一張兩邊互相牽連的網。

這裡的陷阱是：如果讓新系統直接照舊系統的資料格式、舊系統的語意去對話，舊世界的那些彆扭設計——奇怪的欄位命名、隱含的狀態約定、為了相容某個古早需求而留的特例——就會**滲進**你本想乾乾淨淨重寫的新系統，把它重新汙染成舊系統的形狀。你費力重寫，結果新系統內部又長出了一套舊邏輯的影子。

擋這個滲透的東西叫 **anti-corruption layer（防腐層）**：在新系統與舊系統之間夾一層轉譯器，把舊系統的概念、資料格式、呼叫語意翻譯成新系統的語言，反之亦然。新系統內部只說自己的話，所有與舊世界的髒對話都被隔離在這層轉譯器裡。它的代價是你得多寫、多維護一層純粹「為了過渡而存在」的程式碼；它的回報是新系統的設計不被舊語意綁架。這層轉譯器和攔截層一樣，都是**過渡架構**——它們不是最終產品的一部分，是為了讓遷移安全進行而臨時搭起來的鷹架，遷移結束後該拆。

攔截層自己也有它的故障模式，而且不容忽視：**它現在站在所有流量的必經之路上。** 原本請求直達舊系統，現在每一個請求都要先過這層 facade。如果它掛了，新舊兩邊都連不到——你為了降低遷移風險引進的東西，反而成了一個全新的單點故障。如果它慢了，每個請求都多吃一份延遲。所以這層攔截器本身得當成關鍵基礎設施來對待：要有自己的健康檢查、要能水平擴展、它的可用性必須高於它所保護的任何一個後端。把遷移的安全性建立在一個比後端還脆弱的代理上，是把風險從一處挪到另一處，不是消除。

## 一筆手算：為什麼漸進的回退代價遠小於大爆炸

漸進式遷移「比較安全」是一句容易認同、卻容易停留在口號的話。把它換算成具體的數字，才看得到安全到底買在哪裡。

設想要把一個舊單體的「使用者通知」功能搬到新服務，這個功能每分鐘處理 10,000 筆通知。先算大爆炸的代價。某天你部署新服務、同時關掉單體裡的通知、把流量 100% 切過去。假設新服務藏著一個只在真實流量下才會炸的問題，而你從切換到「發現不對、決定回滾、把舊路徑重新接回來並確認穩定」總共花了 15 分鐘——這在半夜、在還要先搞清楚發生什麼事的情況下，已經是樂觀估計。這 15 分鐘裡，每一筆通知都受影響：

```
big-bang blast = 10,000 notifications/min x 15 min x 100% = 150,000 notifications affected
```

15 萬筆通知出問題，而且因為舊路徑已經被你關掉、狀態可能已被新服務動過，「重新接回舊的」本身就是一場有風險的手術——回滾不是按個按鈕，是另一次未經演練的切換。

現在算 strangler + 漸進放量的版本。你在攔截層上做**比例放量**（canary，部署面的放量機制見〈blue-green、canary 與 rolling〉）：先讓 1% 的通知走新服務、99% 仍走舊的。同樣那個問題，現在只打在 1% 的流量上；而且因為你正盯著這 1% 的錯誤率和延遲，發現它的時間從 15 分鐘縮短到——保守說——2 分鐘，回退又只是把旋鈕從 1% 擰回 0%、瞬間生效、舊路徑根本沒關過：

```
canary blast = 10,000 notifications/min x 2 min x 1% = 200 notifications affected
```

15 萬對 200，差了 750 倍。這個倍數不是魔法，它是兩個因子的乘積：**放量比例**把「同時受影響的流量」從 100% 壓到 1%，**早期偵測**把「受影響的時間」從 15 分鐘壓到 2 分鐘，因為你只盯著一小塊、訊號清楚、且舊世界完好無損隨時可回。確認 1% 穩了，再升到 5%、20%、50%、100%，每一級都重複「放量 → 觀測 → 沒事才往上」這個循環。整段遷移裡，你暴露在風險下的，永遠只是「當前放量比例 × 偵測到問題的那一小段時間」，而不是「全部流量 × 你來得及反應的全部時間」。漸進買到的安全，量化出來就是這兩個因子各自被壓了一到兩個數量級。

## Expand-contract：當你換的不是系統、而是一個介面

Strangler 管的是「一整套系統」這種大塊頭。但更日常的遷移其實小得多、也密集得多：把一個函式多加一個參數、把一個 API 欄位改名、把一個設定鍵換成新格式、把一張表的某欄改型別。這些東西同樣不能停——它們正被上千處程式碼、被外部 client、被線上資料倚賴著——但對它們套一整層 facade 顯然太重。

這類「破壞性介面變更」的安全做法，是另一個模式：**expand-contract**，Danilo Sato 在 2014 年於 Martin Fowler 的 bliki 上記述、定名為 parallel change（這個技術更早由 Joshua Kerievsky 在 2006 年作為一種重構策略記述）。它的核心洞見是：一個破壞性變更之所以破壞，是因為你想**同時**改掉「提供方」和「使用方」——把舊的拔掉的同一瞬間，要求所有使用方都已經切到新的。只要這兩件事必須同時發生，你就回到了大爆炸。expand-contract 的全部技巧，就是把這「同時」拆成**三個可以分開部署、之間永遠相容**的階段：

- **expand（擴張）**：先**只加新的、完全不動舊的**。新舊兩個版本並存，提供方同時支援兩者。這一步部署完，沒有任何使用方受影響——因為舊的還在原地，新的只是多出來的選項。
- **migrate（遷移）**：把使用方**一個一個**從舊版切到新版。這是最長的一段，尤其當使用方是你管不到的外部 client 時，可能要等上幾個月。關鍵是它能**逐個進行**，每切一個都是獨立、可回退的小步。
- **contract（收縮）**：確認**再也沒有人用舊版了**，才把舊的移除，讓介面只剩新版。

這三段之間的每一個中間狀態，都是新舊並存、彼此相容的——這正是它零停機、永遠有退路的根源。你永遠先有退路（舊的還在），確認沒人用了，才拆退路。

舉一個能在腦中重放的具體例子：一個服務對外的 JSON 回應，要把 `name` 這個欄位拆成 `first_name` 與 `last_name`。大爆炸的做法是某次部署直接把 `name` 換掉——那一刻，所有還在讀 `name` 的舊 client 全部解析失敗。expand-contract 的做法是：

1. **expand**——伺服器的回應**同時**輸出 `name` 和 `first_name`／`last_name` 三個欄位。舊 client 讀 `name`、毫無感覺；新 client 可以開始讀新欄位。這次部署對任何人都無害。
2. **migrate**——逐個把 client 改成讀新欄位。你管得到的內部 client 自己改；管不到的外部 client，給它們時間，並用監控觀察「還有多少請求方在讀舊的 `name`」（比如在回應這個欄位的程式碼路徑上埋一個計數器）。這個計數器是 contract 階段能不能安全進行的唯一依據。
3. **contract**——當監控顯示 `name` 的讀取量歸零、且穩定維持一段時間後，才把 `name` 從回應裡拿掉。

注意這裡的三段，**每一段都是一次獨立的部署，每一段之間系統都處於完全可用、可回退的狀態**。expand 部署完若發現新欄位算錯了，回退它不影響任何讀 `name` 的人；migrate 切某個 client 出錯，只回退那個 client。你把一個「會打爆所有 client」的破壞性變更，拆成了一串「每一步都不破壞任何人」的相容變更。

順帶一提，同一套 expand-contract 的三段邏輯，落到**資料庫 schema** 上有它自己一整套講究——加欄位、雙寫、回填（backfill）歷史資料、切讀、再砍舊欄位，每一步在大表上的鎖與效能都要算——那是它自己的深水區（見〈零停機 schema 遷移：expand-contract 與 backfill〉）。這裡要看清的是更上層的東西：expand-contract 不只是資料庫技巧，它是**任何破壞性介面變更**的通用安全形狀，函式簽章、API 欄位、設定格式、訊息 schema 都適用。

## 同一個原理，在程式碼內部的長相：branch by abstraction

strangler 在系統邊界放 facade，expand-contract 在介面層讓新舊並存。如果要換的東西在這兩者之間——比如某個被程式碼裡幾百處直接呼叫的核心元件，一個老舊的支付供應商客戶端、一套自己刻的快取存取層——你既不好在它外面套 HTTP facade（它不是一個獨立服務），又不只是一個欄位（它是散落各處的呼叫）。

這時的做法叫 **branch by abstraction（藉抽象分支）**：先在那個元件前面引進一層**抽象介面**，把散落各處的直接呼叫，一個一個改成透過這層抽象去呼叫——這一步本身不改任何行為，只是把「直接依賴實作」換成「依賴介面」。等所有呼叫點都走抽象了，你在抽象底下實作一個新版本，讓新舊兩個實作並存，用一個開關（feature flag）決定走哪個。然後就是熟悉的劇本：漸進放量、觀測、確認新的沒問題、把舊實作刪掉、最後連那層為過渡而生的抽象都可以視情況拆掉。

它和前兩者是同一個基因：**先讓新舊並存、可切換，再可控地把流量挪過去，最後清掉舊的。** 差別只在「並存」這件事發生在哪一層——系統邊界（strangler）、介面定義（expand-contract）、還是程式碼內部的抽象（branch by abstraction）。看穿這一點，你就不會把它們當成三個要分別背的招式，而是同一個第一原理在不同尺度上的自然投影。

## 最容易被忘記的那一步：contract

到目前為止講的都是怎麼安全地把新的長出來、把流量挪過去。但這些模式真正最常翻車的地方，不是 expand、不是 migrate，而是**最後那一步 contract——也就是把舊的真正刪掉**。

這是個違反直覺的觀察，值得停下來想。一旦流量 100% 走新系統、新欄位、新實作，遷移**看起來**就完成了：使用者沒感覺、監控一片綠、新功能正常。於是團隊宣告勝利、轉去做下一件事。舊系統、舊欄位、舊 API、舊 flag、那層 facade——全留在原地，沒人敢刪。為什麼不敢？因為「萬一還有人在用呢」，因為刪掉是要冒風險的而留著不痛不癢，因為當初寫它的人已經離開、沒人完全確定它還連著什麼。半年後，這些殘骸變成一種特殊的負債：**沒人懂、沒人敢動、但還在編譯、還在跑、還在占據每一個讀程式碼的人的注意力**。

不刪舊的，代價比想像中大。直覺上遷移把「一套系統」換成了「一套更好的系統」，只要 contract 沒做，實際得到的是「永遠並存的兩套系統」——兩套都要部署、都要監控、都要在出安全漏洞時打補丁、都要在新人讀程式碼時解釋「這個為什麼還在」。複雜度不減反增。strangler 若沒走到「拆掉舊系統與 facade」，你只是多養了一棵樹；expand-contract 若停在 migrate、忘了 contract，那個本該被刪的 `name` 欄位會在你的 API 裡留存到天荒地老，每個新 client 的開發者都得困惑一次「我到底該讀 `name` 還是 `first_name`」。

而 contract 之所以難，有一個**真實的技術理由**，不只是人的怠惰：**contract 是這整套模式裡唯一通常不可回退的一步。** 回頭看 strangler 的資料庫遷移——在你把舊表、舊欄位、舊同步流程都還留著的時候，你隨時能切回舊系統；但一旦你執行了 contract、把那些舊物件刪掉，要再切回去就不是擰旋鈕了，而是得把刪掉的東西重建、把這段期間的資料變更重放，風險陡增。所以 contract 必須是**刻意的、最後的、確認過沒有回頭需求才做的一步**，而它的「確認」必須建立在**證據**上，不是建立在「應該沒人用了吧」的感覺上：你需要那個讀取計數器歸零、需要那個 endpoint 的存取日誌一段時間沒有命中、需要主動去問過所有已知的下游。沒有這個證據，contract 要嘛永遠不敢做（殭屍殘留），要嘛魯莽地做了然後炸掉某個你不知道的下游。

讓 contract 真的發生，靠的不是紀律，是**機制**：給舊東西一個明確的退役日期（sunset date），並用護欄擋住「邊淘汰邊有人新增舊用法」這個會讓並存期無限延長的漏——比如在 CI 裡加一條規則，擋住任何**新增**的舊 API 呼叫、舊欄位讀取、舊 flag 引用，讓存量逐步歸零而不會被新的用法回補（這種「把慣例變成機器強制的閘」屬於慣例自動化的範疇，見〈鋪路：讓對的做法最省力〉）。退役日期配上「擋新用法」的閘，並存期才有一個會收斂的終點；少了任何一個，遷移就會卡在「新舊都還在、都有人用、都不敢動」的永久過渡態——這比遷移之前更糟，因為你現在維護的是兩個系統的總和加上它們之間那層鷹架。

## 並存期的暗礁：雙寫不是免費的保險

漸進式遷移有一個被廣泛使用、也被廣泛低估其危險的工具：**雙寫（dual-write）**。當你要切換的是有狀態的東西——一個資料儲存、一份索引——並存期裡你常常會「同時寫新舊兩邊」，這樣無論讀哪邊都有資料，切讀的時候才平滑。聽起來像個穩妥的保險：兩邊都寫了，還能出什麼錯？

問題在於，**「寫兩個地方」這個動作本身不是原子的**。你先寫舊的、再寫新的，這中間有一個窗口：如果寫完舊的、程式崩了或網路斷了、新的沒寫成，兩邊就**漂了**——舊有、新沒有。反過來先寫新的也一樣。沒有跨兩個儲存的交易把它們綁在一起（而跨異質儲存的原子提交本身是分散式系統裡最貴的問題之一），雙寫就是一個**持續產生不一致的來源**，而且這種不一致是悄悄累積的：每一次失敗的半邊寫，都在兩份資料間留下一個微小的差異，平時沒人發現，等你切讀切到新的那一刻，才以「某些資料莫名其妙不見了／不對了」的形式集體爆出來。

所以雙寫不是「貼上去就安心」的保險，它是一段**需要被當成程式來測試、來監控的邏輯**：你得有辦法持續比對新舊兩邊是否一致（雙讀比對），得在發現漂移時能修，得清楚知道在不一致時以哪邊為準。雙寫期間真正買到信心的，不是「我寫了兩邊」，而是「我**持續驗證**兩邊一致、且漂移時我能發現並修」——這套「確認兩邊一致、漂移怎麼修」的機制有它自己的深度（見〈對帳：怎麼確認兩邊一致、漂移怎麼修〉）。把雙寫當成不需要驗證的保險，是並存期最常見、也最隱蔽的暗礁：看起來像兩份一樣的資料，實際上是兩份從沒比對過、正在悄悄分岔的資料。

## 何時不該漸進

把這套模式講得這麼周全，容易讓人覺得「遷移就該永遠漸進」。但漸進是有成本的，而且不低：並存期你要同時養新舊兩套、要寫攔截層或防腐層或抽象層、要管雙寫一致性、要追蹤退役進度——複雜度在過渡期是**翻倍**的。這份成本只有在「大爆炸的風險不可接受」時才划算。

所以判準要反過來問：**一次切過去若出錯，回得來嗎？會炸到多少人？** 如果被換掉的東西沒人在用、或停機完全可接受、或範圍小到一次換完即使出錯也能在五分鐘內改回來——那麼大爆炸反而是對的：給一個內部小工具的改名套上 strangler + 雙寫 + 漸進放量，是把五分鐘的事做成五週的工程，是另一種形式的過度設計。漸進式遷移的整套機械，是為了馴服「無退路、blast radius 涵蓋全部」這個具體的恐懼而存在的；當這個恐懼不在場時，它的成本就成了純粹的浪費。

值得多想一層的，是 AI 對這個判斷的影響——但要小心它影響的是哪一部分。遷移裡最苦、最耗人的，往往是 migrate 那一段的機械重複：把上千個呼叫點從舊樣式改到新樣式。這種大批改寫正是 AI 與 codemod 擅長的，它能把這段從「派人改幾週」壓到「生成加審查幾天」。誘惑也隨之而來：既然一個晚上就能把所有呼叫點全改完，何不一次切換、跳過那些麻煩的小步？這裡藏著一個要看穿的混淆——**AI 降低的是「改的工作量」，沒有降低「切換當下出錯沒退路」的風險**。遷移的危險從來不在改有多累，而在那一刻新世界出錯時舊世界還在不在。AI 讓你能更快地走完每一個小步，但它沒有、也不可能取消「分步」本身的價值。正確的用法是用它加速漸進，不是用它合理化大爆炸。

## 為什麼是這個形狀

退一步看，strangler、expand-contract、branch by abstraction，這些模式的整個樣貌，都是從同一個無法迴避的事實裡長出來的：**一個正在被使用、不能停的東西，你沒有辦法在一個瞬間安全地把它換掉，因為那個瞬間若出錯，你沒有一個完好的舊世界可以退回去。**

正因如此，這些模式都堅持讓新舊**並存一段時間**——並存就是隨時可用的退路。正因如此，它們都在新舊之間放一個**可控的旋鈕**（facade 的路由、介面的新舊欄位、抽象底下的 flag），把「一次全切」拆成「一格一格地挪」，讓任何一步的失敗都被框在一小塊裡、都能擰回去。正因如此，它們都有一個容易被忘記、卻不可省略的 contract 階段——因為遷移的終點不是「流量切過去了」，而是「舊的被乾淨地拆掉了」，否則你只是把賭注換成了永久的雙重負擔。

這套形狀不是某種工程美學上的講究，而是「換掉一個不能停的東西」這個問題，被「沒有安全的瞬間切換」這個物理事實逼到牆角後，唯一站得住的解。下次你看到一個遷移計畫裡寫著「facade」「並存期」「放量比例」「退役日期」這些字眼，你會知道那不是把簡單的事複雜化——那是有人認真算過大爆炸那一刻的 blast radius，決定不去賭它。
