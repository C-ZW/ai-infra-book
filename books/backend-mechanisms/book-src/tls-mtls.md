# TLS 與 mTLS：握手與憑證管理

凌晨四點，監控把一整個服務群的可用性畫成一條直墜到零的線。沒有部署、沒有流量尖峰、資料庫好端端的。打開任何一個端點，瀏覽器回的是 `NET::ERR_CERT_DATE_INVALID`；用 `curl` 打，看到的是 `certificate has expired`。原因簡單到令人發窘：某張憑證在前一刻過了效期，於是每一個 client 在 TLS 握手的驗證那一步直接掛斷，連請求都還沒送出去。

這個事故有一個值得玩味的形狀——它**沉默地醞釀、瞬間引爆**。憑證的剩餘天數從 90 倒數到 0，這段期間沒有任何徵兆，服務的延遲、錯誤率、CPU 全都正常；直到歸零那一刻，所有 HTTPS 連線同時失敗，沒有半死狀態、沒有降級曲線，是一道斷崖。要理解為什麼一個「忘了續簽」的小疏忽能造成這種乾淨俐落的全域當機，得先搞懂 TLS 到底在你看不見的地方做了哪些事——它如何在一個被假設為敵意的網路上，憑空建立起「對方是誰、訊息沒被偷看、沒被改過」這三件事的信心。

## 在敵意的網路上，要憑空證明三件事

把問題放到第一原理。兩個行程之間隔著一段公開網路，你必須假設這段網路上有一個能**看、能改、能假冒**的對手坐在中間。在這個前提下，一條裸 TCP 連線什麼都保證不了：對手可以讀走你傳的每個位元組（竊聽）、可以把你傳的內容改掉再轉送（竄改）、可以乾脆冒充對端跟你對話（冒充）。TLS（Transport Layer Security）要在 TCP 之上補回來的，正是針對這三種攻擊的三個保證：

- **機密性（confidentiality）**：對手就算錄下整段流量，也解不出明文。
- **完整性（integrity）**：對手若改動任何一個位元組，接收端會發現並拒收。
- **身分驗證（authentication）**：你連上的真的是預期的那一方，不是中間插進來的冒充者。

這三件事彼此糾纏，缺一不可，而且順序很要命。最容易被忽略的陷阱是：**光有加密，不足以安全。** 設想你和對手做了一次漂亮的金鑰交換、之後全程加密——但如果那個「對手」本身就是中間人，你加密給的是他、他解密後再加密轉給真正的伺服器，你拿到的機密性是假的，因為你從一開始就把通道建給了錯的人。所以三個保證裡，**身分驗證是地基**：先確認對端是誰，加密才有意義。這也是為什麼 TLS 握手的核心戲份，不是「怎麼加密」，而是「怎麼在還沒有共享祕密的情況下，既確認對方身分、又談出一把只有雙方知道的金鑰」。

而「怎麼確認對方身分」這件事，把問題又往下推了一層：你和一台素未謀面的伺服器，憑什麼相信它就是它聲稱的那個網域？這就是憑證與憑證鏈要回答的問題，也是整個 TLS 體系裡最容易在凌晨四點咬人的部分。先看握手，再看信任。

## 一個來回就開始加密：TLS 1.3 的握手

當前的現役主力是 **TLS 1.3（RFC 8446，2018-08 定案）**。它相對前一代 TLS 1.2 最大的改進，是把建立連線的握手從兩個來回（2-RTT）壓到**一個來回（1-RTT）**，同時砍掉了一大票不安全的舊式金鑰交換、強制每條連線都具備前向保密。要理解這 1-RTT 是怎麼擠出來的，得看 client 在第一個封包裡就玩了一手「投機」。

在 TLS 1.2 裡，雙方得先互相打招呼、商量好要用哪種金鑰交換演算法，**然後**才開始交換金鑰材料——兩個來回。TLS 1.3 的洞見是：與其先問再做，不如**賭一把先做**。client 在第一個訊息 `ClientHello` 裡，除了列出自己支援的參數，還**直接附上一份臨時的 Diffie-Hellman 公開金鑰**（放在 `key_share` 擴充裡），等於先把自己這半邊的金鑰材料投機性地丟了出去。伺服器收到後，挑一組雙方都支援的參數，回一個 `ServerHello`、附上**它自己那半邊**的金鑰公開值。到這一刻，雙方手上各有「自己的私鑰 + 對方的公鑰」——靠 Diffie-Hellman 的數學，兩邊**各自獨立算出同一把共享祕密**，而這把祕密從未在線路上出現過。對手就算錄下整段 `ClientHello` 和 `ServerHello`，也只看到兩個公開值，算不出那把共享金鑰。

把這個來回攤平來看：

```
TLS 1.3 handshake (1-RTT)

  client                                  server
    |                                       |
    |  ClientHello                          |
    |    + key_share (client DH pubkey)     |
    |  ------------------------------------>|
    |                                       |  pick params, compute
    |                                       |  shared secret
    |          ServerHello                  |
    |            + key_share (server pubkey)|
    |          {Certificate}                |
    |          {CertificateVerify}          |
    |          {Finished}                   |
    |  <------------------------------------|
    |  verify cert chain, verify signature  |
    |                                       |
    |  {Finished}  + application data       |
    |  ------------------------------------>|
    |                                       |
   (braces {} = already encrypted)
```

注意第二個訊息（伺服器回來那一束）裡，`ServerHello` 之後的東西**已經是加密的了**——因為共享祕密在伺服器算出公鑰那一刻就成立了，之後的憑證、簽章全走加密通道。這帶來一個常被忽略的好處：TLS 1.3 把伺服器憑證藏進了加密段，竊聽者連對方出示的是哪張憑證都看不到。

伺服器回的那束訊息裡，三樣東西扛起身分驗證：`Certificate` 是伺服器出示的憑證（裡頭有它的網域名和公鑰）；`CertificateVerify` 是伺服器**用憑證對應的私鑰**，對到目前為止的握手內容簽的一個簽章；`Finished` 是對整段握手的完整性校驗。client 收到後做兩件事的合一驗證：第一，這張憑證能不能順著信任鏈一路驗回我信任的根（下一節講）；第二，`CertificateVerify` 這個簽章能不能用憑證裡的公鑰驗過。**這一步是整個身分驗證的關鍵**——它證明對端不只是「拿了一張憑證」，而是**真的持有那張憑證對應的私鑰**。憑證是公開的，誰都能抄一份；但只有私鑰的持有者才簽得出那個對得上的簽章。一個冒充者就算偷到了伺服器的憑證檔，沒有私鑰，照樣簽不出 `CertificateVerify`，握手就在這裡破功。

這裡還埋著前向保密（forward secrecy）的機制。TLS 1.3 強制每條連線都用**臨時的（ephemeral）** Diffie-Hellman 金鑰——每次握手生一把新的、用完即丟。它的意義是：就算對手錄下了你今天所有的加密流量、並在三年後設法拿到了伺服器的長期私鑰，他**依然解不開**那些舊流量。因為加密用的是當時那把臨時金鑰，而那把金鑰早已從記憶體裡消失、從未寫進任何地方。長期私鑰只用來**簽** `CertificateVerify`（證明身分），不用來**加密**——身分的私鑰和加密的金鑰被刻意分開，這就是為什麼私鑰外洩不等於歷史流量全裸。這個「身分用的鑰匙」和「加密用的鑰匙」分家的設計，正是前向保密的全部祕密。

## 信任從哪裡來：憑證鏈與那個地基

握手裡那句「這張憑證能不能驗回我信任的根」，把問題推到了 TLS 真正的隱形主體：**你憑什麼相信一張憑證？**

伺服器出示的憑證，本質是一份聲明：「我是 `example.com`，我的公鑰是這串。」但聲明誰都能寫。讓這份聲明有份量的，是上面有一個**憑證頒發機構（Certificate Authority, CA）的數位簽章**——CA 用它的私鑰簽了這張憑證，等於背書「我查過了，這串公鑰確實屬於 `example.com` 的持有者」。於是問題遞迴了一層：你又憑什麼相信這個 CA？

答案是一條**信任鏈（certificate chain）**，從伺服器憑證一路往上指向一個你**預先就信任**的根。實務上鏈通常是三層：最底下是伺服器憑證（也叫 leaf 或 end-entity 憑證），它由一個**中間 CA（intermediate CA）** 簽發，中間 CA 又由一個**根 CA（root CA）** 簽發，根 CA 是自簽的、是鏈的終點。client 驗證的方式是逐級往上爬：用中間 CA 的公鑰驗伺服器憑證上的簽章，再用根 CA 的公鑰驗中間 CA 上的簽章，直到爬到一個**它本來就信任**的根為止。

那「本來就信任」從何而來？這是整個體系的地基所在：你的作業系統和瀏覽器內建了一份**根憑證信任庫（trust store）**，裡頭預載了數百個被審核過的根 CA 公鑰。信任鏈最終必須爬到這份信任庫裡的某一個根，驗證才算通過。這份信任庫，就是整個公開網路 TLS 信任的源頭——所有的信心，最後都壓在「我作業系統裡那份根清單沒被動過手腳」這個假設上。

```
certificate chain validation

  +-------------------+   signed by    +-------------------+   signed by    +----------------+
  | leaf cert         | -------------> | intermediate CA   | -------------> | root CA        |
  | CN = example.com  |                | (issuer)          |                | (self-signed)  |
  +-------------------+                +-------------------+                +----------------+
        presented by server               sent by server                   in client's
                                                                            trust store
        client walks up: verify each signature with the issuer's pubkey,
        until it reaches a root it already trusts.
```

這個鏈式結構解了一個現實問題：為什麼不讓根 CA 直接簽每一張伺服器憑證？因為根 CA 的私鑰是整個體系最高價值的目標，一旦外洩、它簽過的一切全部崩塌。所以根 CA 的私鑰被鎖在離線的硬體裡、極少動用，日常的簽發工作交給中間 CA。中間 CA 萬一出事，可以單獨撤銷、換一個，不必動到根。這也帶出一個**最常見、最容易被誤判的部署故障**：伺服器設定時，運維只裝了 leaf 憑證、**忘了一併提供中間 CA 憑證**。後果很狡猾——在自己的開發機上測常常是「好的」，因為開發機的系統碰巧快取過那張中間憑證、補得回來；但換一個乾淨的 client、一支沒快取的手機、一個嚴格的程式庫，它爬到 leaf 之後找不到中間那一環，鏈就斷了，握手失敗。這種「在我這裡明明好的」的不一致，根源就在於**憑證鏈的完整性是伺服器的責任**：伺服器必須把 leaf 加上所有中間憑證一起送出，不能賭 client 自己補得齊。

## 那個會在凌晨四點咬你的數字：效期

現在回到開場那場斷崖式當機。憑證除了「我是誰、公鑰是這串」，還帶一個**效期（validity period）**——一個明確的「生效日」和「到期日」。client 在驗證時會檢查當前時間是否落在這個區間內，**過期的憑證一律拒收**，沒有寬限、沒有降級。這個設計本身是對的：效期是一道安全閘，限制了一張被盜私鑰能被濫用多久。但它也意味著，**每一張憑證都是一顆設定好的定時炸彈**，唯一的拆彈方式是趕在引爆前換一張新的。

這件事在 2026 年變得格外要緊，因為這個「定時」的時間正在被業界**系統性地縮短**。公開 CA 簽發的憑證最長效期，過去長達 398 天，如今走在一條明確的遞減時間表上（由 CA/Browser Forum 在 2025-04 通過的 SC-081v3 確立，2026-06 查證）：

| 期間 | 最長效期 | 網域驗證資料可重用期 |
|---|---|---|
| 至 2026-03-14 | 398 天 | 398 天 |
| 2026-03-15 起 | 200 天 | 200 天 |
| 2027-03-15 起 | 100 天 | 100 天 |
| 2029-03-15 起 | 47 天 | 10 天 |

方向只有一個：越來越短。背後的邏輯是「用更短的效期取代笨重的撤銷機制」——既然憑證活不久，一張被盜的憑證能造成的損害窗口本來就小，撤銷的需求隨之下降。Let's Encrypt 這類自動化 CA 走得更前面：它的預設憑證效期目前是 90 天，但官方已宣告（2025-12）規劃分階段把預設降到 **45 天**（時間表：2027-02 先過渡到 64 天，2028-02 再降到 45 天），並另外提供短至 6 天的選項。

這個趨勢把一個結論釘死：**憑證輪換（rotation）必須是自動的，靠人記得去續簽是在賭命。** 當效期是 398 天時，一年手動換一次還勉強撐得住；當它掉到 47 天、甚至 6 天，任何依賴人工的流程都注定在某一輪漏掉。這正是 ACME 協定（Let's Encrypt 用的自動化簽發協定）存在的理由——它讓「證明你擁有這個網域 → 拿到新憑證 → 裝上去 → 重載」這整條鏈變成一個無人值守的迴圈，cron 在到期前若干天自動跑一輪。

但自動化本身也會壞，而且壞得一樣沉默。把開場那場事故的數字算到底：假設一張 Let's Encrypt 90 天憑證，續簽 cron 設定在「到期前 30 天」觸發。理想情況下，第 60 天 cron 跑、換上新憑證、剩餘天數跳回 90，永遠續得上。但假設某次部署改動把那個 cron 弄壞了——也許是換了排程系統、也許是憑證存放路徑變了、也許是 ACME client 的 API token 過期。從這一刻起，**剩餘天數開始真正地倒數，而且沒有任何東西在喊**：第 60 天該續沒續，系統毫無反應；第 70 天、第 80 天，一切正常；直到**第 90 天的那一秒**，憑證過期，下一個來敲門的 client 在握手驗證效期時看到當前時間已超過到期日，握斷連線。然後是第二個 client、第三個——全部，同時。從「續簽壞掉」到「服務歸零」之間，隔了整整 **30 天的寂靜**，而沒有人在那 30 天裡收到任何信號。

這就是為什麼成熟的做法是**不去監控「續簽 cron 有沒有跑成功」，而是直接監控「憑證本身的剩餘天數」**——把它當成一個一級的服務指標（SLI），對「剩餘 < 14 天」設告警。前者監控的是手段（cron），會被各種你想不到的方式繞過；後者監控的是**真正要保證的那個狀態**（憑證還新鮮），無論輪換是用什麼機制、是不是壞了，這個數字都不會騙你。它把一個「沉默到爆炸」的故障，提前 14 天變成一個有人會看到的告警。

## 換一個方向問身分：mTLS

到目前為止講的都是**單向**的 TLS：伺服器出示憑證、client 驗證它，但 client 自己是誰、伺服器並不在意。這對公開網站完全合理——你連上銀行網站，你要確認它真的是銀行（不然就把密碼送給了釣魚站），但銀行不靠 TLS 憑證來認你是誰，它靠的是之後你輸入的帳號密碼。身分驗證在這裡是**不對稱**的：只有一方需要用憑證證明自己。

但有一類場景，這個不對稱不夠用：**服務對服務的呼叫**。在一個內部由幾十上百個微服務組成的系統裡，當服務 A 呼叫服務 B，B 該不該無條件相信「來敲門的就是 A」？在零信任（zero trust）的安全模型下，答案是不該——網路內部一樣可能有被攻陷的節點在冒充。這就需要**雙向**的身分驗證：A 連 B 時驗 B 的憑證，**B 同時也要驗 A 的憑證**。這就是 mTLS（mutual TLS，雙向 TLS）。

機制上，mTLS 不是另一個協定，就是同一套 TLS 握手，多走一步。還記得單向握手裡伺服器回的那束訊息嗎？在 mTLS 裡，伺服器會在那束裡多塞一個 `CertificateRequest`，明確要求 client 也出示憑證。client 收到後，在自己的回應裡補上自己的 `Certificate` 和一個 `CertificateVerify`（用 client 私鑰簽的）——和伺服器證明自己身分的方式**完全對稱**。伺服器驗這個 client 簽章和 client 憑證鏈，驗過了，雙方才都確信對端是誰。整個過程的密碼學基礎和單向一模一樣，只是「誰需要證明身分」從一方變成了雙方。

mTLS 帶來的身分保證，比常見的替代方案（如 API key、bearer token）強在一個地方：它證明的是**對端持有某把私鑰**，而私鑰可以做到永不離開它所在的機器。一個 API key 是一串字串，會被印進日誌、貼進設定檔、在行程之間傳遞，任何一個經手點都可能洩漏；而一把綁定在某台機器上的 client 私鑰，理論上從不被傳輸，偽造它的難度高出一個量級。

但這裡有一個**必須講清楚、否則會釀成安全幻覺的邊界**：mTLS 證明的是「**身分**」，不是「**授權**」。當伺服器驗過了 client 的憑證，它知道的只是「對端是 `service-a`，持有一張我信任的 CA 簽發的憑證」——它**不知道** `service-a` 該不該存取它正在請求的那個資源。把「持有有效憑證」當成「有權做這件事」，是一個經典的混淆：身分驗證（AuthN，你是誰）和授權（AuthZ，你能做什麼）是兩件分開的事，mTLS 只解決前者（授權的機制是另一個主題，見〈認證與授權：AuthN、AuthZ 與 OIDC 的位置〉）。一個有效但被授予了過多權限的 client 憑證，和一把萬能鑰匙沒有區別——驗證它真實無誤，不代表它該被放行去做任何事。

## mTLS 的真正成本，在憑證本身

mTLS 聽起來只是「多驗一邊」，但它的代價不在握手那多出的幾個位元組，而在它**讓需要被管理的憑證數量翻了一倍還不止**。

單向 TLS 下，你管的是少數幾張對外的伺服器憑證。一上 mTLS，**每一個服務實例都需要一張自己的 client 憑證**——而這些憑證和公開的伺服器憑證一樣有效期、一樣會過期、一樣需要輪換。前面講的那個「凌晨四點的定時炸彈」，現在不是一兩顆，而是**每個服務、每個實例各揣著一顆**。更尖銳的是，內部 mTLS 為了安全，通常刻意用**極短效期**的憑證——不是 90 天，而是小時級甚至分鐘級。短效期是好事（一張被盜的憑證很快自然失效，等於用過期取代了笨重的撤銷機制），但它把輪換的頻率推到了任何人工流程都絕對跟不上的程度：一張只活一小時的憑證，意味著每個實例每小時都要重新簽發、分發、裝載一次。

這就是為什麼**「裸著上 mTLS」幾乎總是個壞主意**。手動管理幾十個服務、每個實例每小時輪換一次的短效期憑證，是一場注定失敗的人力戰爭——漏掉任何一輪，那個實例就在下一刻握不了手、被踢出通訊。讓 mTLS 在實務上可行的，是把整套憑證生命週期**自動化**：一個內部的私有 CA（不需要、也不該用公開 CA 來簽內部服務憑證——內部憑證的信任只在自家叢集內成立），自動為每個新起的實例簽發短效期憑證、在過期前自動重簽、自動分發。這套自動化通常不是手刻的，而是下沉到一層專門的基礎設施裡，由它透明地替每個服務處理 mTLS 的建立與憑證輪換，應用程式幾乎無感（這層基礎設施本身是另一個主題，見〈Service mesh：把網路關注點下沉到 sidecar〉）。

所以判斷要不要上 mTLS 的真正提問，不是「需不需要服務間加密」，而是「**有沒有那套自動化來扛住憑證輪換**」。在一個只有三五個互信服務、沒有合規壓力的小系統裡硬上裸 mTLS，運維負擔幾乎一定超過它買到的安全收益——更務實的順序是先把單向 TLS 與自動續簽做穩，等規模和治理痛點長到需要、且有自動化撐腰時，mTLS 才回本。

## 別在最後一哩把保證丟掉：終結位置

最後一個值得想透的取捨，藏在一個部署細節裡：**TLS 在哪裡被「終結」（terminate）？**

絕大多數真實系統不會讓 TLS 一路加密到最終那個應用程式。常見的做法是在邊緣放一個負載均衡器或反向代理（見〈負載均衡：L4 與 L7、演算法與健康檢查〉），讓它**終結 TLS**——也就是在這裡解密，把加密流量變回明文，再轉送給後面的應用。這麼做有實際好處：應用程式不必各自處理加密的 CPU 開銷和憑證管理，集中在一處統一處理；L7 負載均衡也只有解密後才能看到 HTTP 內容、據以做路由。

但這裡有一個常被輕輕放過、實則牴觸零信任的陷阱：**終結點之後的那一段，是明文。** 「TLS 終結在負載均衡器」這句話的完整含義是——從 client 到負載均衡器是加密的，但從負載均衡器到後端應用那一段內網流量，是裸的。如果你的威脅模型假設「內網是可信的」，這沒問題；但零信任的整個前提正是**不假設內網可信**——一旦對手在內網裡站穩腳跟，他就能在終結點之後竊聽到所有解密後的流量，前面那段精心設計的加密形同虛設，因為保證在最後一哩被丟掉了。

所以終結位置是個**明確的取捨**，不是一個有標準答案的最佳實踐：邊緣終結換來的是簡單和省 CPU，代價是內網明文；要堅守零信任，就得讓內網那段也加密——要嘛讓 TLS 一路穿透到後端（passthrough），要嘛在終結後對內網流量重新加密一次（re-encrypt，這往往就是 mTLS 在服務間落地的形態）。判準回到威脅模型：你到底信不信任你的內網？這個問題的答案，決定了那條「明文的最後一哩」能不能存在。

## 為什麼是這個形狀

退一步看，TLS 的整個樣貌——握手、憑證鏈、效期、雙向驗證、終結位置——都是從同一個無法迴避的前提長出來的：**網路上的對手能看、能改、能假冒，所以信任不能被預設，只能被證明。**

正因為對手能假冒，你不能光靠「對方說它是誰」，得要它**用私鑰簽出一個只有真身簽得出的證明**——這是 `CertificateVerify`。正因為「對方自稱可信」不夠，信任得有一個源頭，於是有了從 leaf 一路驗回作業系統信任庫的**憑證鏈**。正因為一把私鑰可能在未來某天被盜，效期把每張憑證的壽命掐短、把損害窗口關小——而短效期反過來逼出了自動輪換，因為人跟不上機器的節奏。正因為內部節點一樣可能被攻陷，零信任要求連服務之間都互相驗身分，於是單向 TLS 長出了 **mTLS** 這個對稱的變體。每一個看似繁瑣的環節，都不是密碼學家的潔癖，而是「在敵意網路上憑空建立信任」這個問題被逼到牆角後，留下來的唯一站得住的形狀。

下次當一個服務在凌晨四點因為一張過期憑證集體斷線，你會知道那不是 TLS 壞了——恰恰相反，那是它在固執地兌現承諾：寧可拒絕一個身分已經無法被擔保的連線，也絕不在一條無法證明對端是誰的通道上，把你的資料交出去。
