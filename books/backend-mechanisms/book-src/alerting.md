# 告警設計：症狀 vs 原因、告警疲勞

凌晨三點，值班的人被吵醒。手機上是一條告警：`node-7 CPU > 85% for 5m`。他半睡半醒地連進去看，CPU 確實高——因為一個每天這個時間跑的批次任務正在轉檔，跑完就好了，使用者一個都沒受影響。他把告警靜音，倒回去睡。這是這週第四次了。

問題不在那一條告警。問題在於，三週後的某個凌晨，真正的災難來了——資料庫連線池耗盡，每一筆寫入都在排隊、逾時、回 500。但這時候值班的人對手機震動的反應，已經被訓練成「先靜音，明天再看」。那條真正要命的 page 響了，他半夢半醒地滑掉，繼續睡。等到有人發現的時候，事故已經延誤了四十分鐘。

殺死他的不是那一條重要的告警漏掉了，而是前面那幾十條不重要的告警，把他對告警的信任**磨光了**。告警設計這件事，表面上是在調門檻、寫規則，骨子裡是在管理一個比 CPU、比延遲都更稀缺的資源：**一個人在被吵醒時，還願意認真對待告警的次數。** 這一章講的就是：告警這套機制到底在解什麼問題、為什麼天真的做法必然走向上面那一幕、以及怎麼把它設計成「響起來就一定值得起床」。

## 告警的本質：把「需要人」的判斷推給人

先把問題講清楚。監控系統收集了海量的訊號——每秒幾千條 metric、每分鐘幾 GB 的 log。這些訊號絕大多數時候只是靜靜地躺在儀表板上，沒有人看。告警是這套機制裡唯一**主動**的部分：它在某個條件成立時，把一個人從他正在做的事（睡覺、寫程式、吃飯）裡**拽出來**，告訴他「現在需要你」。

這個「拽出來」的動作，代價遠比它看起來高。它打斷的不只是當下那幾分鐘，而是一個人的專注、睡眠、以及——最關鍵的——他對「告警＝真的有事」這個等式的信任。每一條誤報都在這個等式上鑿一個小洞，鑿到某個臨界點，人就不再相信告警了。所以告警設計的第一原理不是「怎麼不漏掉任何問題」，而是**怎麼讓每一次「拽出來」都對得起它的代價**。

這裡有一個無法迴避的取捨，它定義了整個問題的形狀。任何告警規則本質上是一個分類器：它把系統的狀態分成「需要人介入」和「不需要」，靠的是一條門檻。而這條門檻有兩種錯法：

- **漏報（false negative）**：真的出事了，告警卻沒響。值班的人睡得很安穩，使用者在哀嚎。
- **誤報（false positive）**：沒出事，告警卻響了。值班的人被吵醒，連進去發現什麼事都沒有。

把門檻調鬆——比如「CPU > 60% 就告」——你會抓到更多真問題，但也會被一堆無害的尖峰吵死，誤報暴增。把門檻調緊——「CPU > 95% 持續十分鐘才告」——誤報少了，但某些真問題會溜過去，漏報變多。**這兩種錯誤是此消彼長的，沒有一個門檻能同時把它們都壓到零。** 這不是調參技巧不夠好，這是分類問題的數學本質。

於是真正的問題變成：既然兩種錯都消不掉，該往哪邊偏？直覺會說「寧可誤報也不要漏報，多告總比少告好」。這個直覺，正是把值班的人推向那個凌晨三點滑掉真告警的深淵的元兇。要看清為什麼，得先理解告警疲勞是怎麼運作的。

## 告警疲勞：為什麼「多告總比漏告好」是錯的

告警疲勞（alert fatigue）不是一種情緒，它是一個可以推演的機制。

假設一個值班輪班的人，在被告警吵醒後，能保持「有急迫感、認真排查」的次數是有限的——一天也許三到五次，超過之後，無論是生理上的疲倦還是心理上的麻木，他對下一條告警的反應都會鈍化。這不是他不專業，是人腦對重複的、大多無意義的刺激的自然反應，和你對冷氣運轉聲的「聽而不聞」是同一回事。

現在把誤報率代進去。假設一個告警系統一個月觸發八百次，事後復盤，其中只有四十次對應真正需要人介入的事件——誤報率 95%。值班的人很快會在統計上學到一件事：**一條告警響起來，有 95% 的機率是假的。** 在這個機率下，最划算的策略不是「每次都認真查」，而是「先靜音，等它自己好或等別人喊」。他不是懶，他是在 95% 誤報率下做了**理性**的選擇。

而災難就埋在這裡：那 5% 的真告警，和那 95% 的假告警，在手機上長得**一模一樣**——同樣的震動、同樣的紅色、同樣半夜兩點。當人已經養成「先靜音」的條件反射，他靜音的不只是雜訊，連同那 5% 的命也一起靜音了。

這就是為什麼「多告總比漏告好」是錯的。它假設告警之間是獨立的——多設一條，只是多一次提醒，不影響別的。但告警之間**不獨立**：每一條雜訊告警都在消耗同一個有限資源（人的注意力與信任），消耗到一定程度，它會把真告警一起拖下水。**淹沒在雜訊裡的關鍵告警，等於一次實質的漏報。** 於是「多告」這個本意在減少漏報的動作，反過來用一種更隱蔽的方式製造漏報——把雜訊堆到真告警被一起忽略。

所以衡量一個告警系統好壞的，從來不是「覆蓋率」（它能抓到多少種問題），而是**訊噪比**——響起來的告警裡，有多少比例真的值得起床。一個覆蓋率 100% 但訊噪比 5% 的系統，比一個覆蓋率 80% 但訊噪比 90% 的系統危險得多，因為前者會親手訓練值班的人去無視它。

那麼，雜訊是從哪裡來的？絕大多數來自一個看似合理、實則致命的習慣：對「原因」告警。

## 症狀 vs 原因：對的告警對應的是使用者的痛

這是整個告警設計裡最重要的一個區分，它的源頭是 Google 一位 SRE（Rob Ewaschuk）在內部流傳的一份文件〈My Philosophy on Alerting〉，後來成了 Google SRE 書裡監控那一章的骨幹。核心只有一句話：**對症狀告警，不要對原因告警。**

什麼是症狀、什麼是原因？拿開場那一幕對比就清楚了：

- **症狀（symptom）**：使用者**直接體驗到**的壞——請求失敗了、請求變慢了、頁面打不開、資料是錯的。症狀回答的是「使用者現在痛不痛」。
- **原因（cause）**：系統**內部某個零件**的異常——CPU 高、記憶體快滿、某個 queue 堆積、某台機器的磁碟 I/O 飆高、連線池快用完。原因回答的是「哪裡可能不對勁」。

為什麼要對症狀告警、而不是原因？有三個層層遞進的理由。

**第一，原因不一定導致使用者受影響，但告警卻照響。** CPU 飆到 90%，可能是一個批次任務在跑，使用者毫無感覺；記憶體用到 85%，可能正是設計如此（快取本來就該把記憶體吃滿）。對這些原因告警，你會收到大量「零件數字超標、但沒人受影響」的誤報——這正是凌晨三點那條 CPU 告警的來源。原因告警是**誤報工廠**，因為「零件忙」和「使用者痛」之間，根本不是一一對應的。

**第二，原因有千百種，而且會隨架構變；症狀少而穩定。** 一個服務可能掛掉的原因，你今天能想到的也許有二十種：DB 連線池滿、下游逾時、記憶體洩漏、磁碟寫滿、憑證過期、某個第三方 API 變慢……每一種你都設一條告警，於是你有了二十條會隨架構漂移的規則（換了資料庫、加了快取層，半數規則就過時了）。但這二十種原因，最終表現在使用者那端的症狀，往往就那麼幾種：**請求失敗率上升、延遲分位數上升。** 對這兩個症狀告警，你用兩條穩定的規則，涵蓋了那二十種原因——**包括你今天還沒想到、明年才會出現的那幾種。** 這是症狀告警最強大的地方：它對未知的故障模式天生免疫，因為無論底下壞的是什麼零件，只要它讓使用者痛，症狀就會動。

**第三，症狀直接定義了「該不該起床」。** 一條告警存在的唯一理由，是它響了之後有人需要立刻做點什麼。使用者正在大量收到 500——這毫無疑問需要立刻處理。CPU 90%——需要嗎？不知道，得看它有沒有造成使用者受影響，而那正是症狀告警會告訴你的事。**用症狀當 page 的判準，等於把「該不該吵醒人」這個決定，錨定在唯一真正重要的東西上：使用者的體驗。**

那原因訊號就沒用了嗎？當然有用——但它的角色是**診斷**，不是**告警**。症狀告訴你「出事了、快起床」，原因告訴你「事出在哪、從這裡查」。正確的分工是：症狀告警負責 page 人，而 CPU、記憶體、queue 長度這些原因訊號全部進儀表板，等症狀告警把人叫醒之後，他打開儀表板，用這些原因訊號快速定位根因。原因訊號從「半夜吵醒你的東西」降級成「被吵醒後幫你查案的東西」——這一個降級，就能砍掉絕大部分的告警雜訊。

這個原則有一個重要的例外，值得單獨點出，否則它會被誤用成教條。

## 有前導期的原因：唯一值得提前告的零件問題

不是所有原因都該被降級成純診斷訊號。有一類原因有資格提前告警，它們的共同特徵是：**有明確的前導期，而且不處理就一定會變成症狀。**

最典型的是「磁碟還有三十分鐘就寫滿」。這是一個原因（零件狀態），此刻使用者還毫無感覺，症狀告警一片寧靜。但磁碟一旦真的寫滿，後果是災難性的、而且往往不可逆——資料庫拒絕寫入、log 寫不進去、服務整個僵住。如果你死守「只對症狀告警」，那你會等到磁碟真的滿了、使用者開始收到錯誤，才被叫醒——而那時候你只剩零分鐘可以反應。

這類原因之所以配得上提前告，是因為它把「等到症狀出現再反應」這個策略的代價放大到不可接受。判準有兩條，必須同時成立：一是**可預測**——你能算出「以目前的成長速度，X 分鐘後會撞牆」（磁碟用量、憑證到期日、license 配額，都屬此類）；二是**撞牆即災難**——一旦真的發生，恢復成本遠高於提前介入的成本。

但即使是這類值得提前告的原因，它的緊急程度也和症狀不同。磁碟三十分鐘後滿，你有三十分鐘——這該開一張**工單**、在上班時間從容處理，而不是半夜把人 page 起來。把告警的**嚴重程度**和它的**緊急程度**分開，是降低疲勞的另一把鑰匙：不是所有告警都該 page。

## 告警的三個檔位與降噪

一條告警響了，「需要人做什麼、多急」差異巨大。把所有告警都用同一種方式（半夜震動手機）送達，本身就是製造疲勞。成熟的告警系統至少分三個檔位：

- **Page**：立刻吵醒人。保留給「使用者正在受影響、每一分鐘都在累積損失」的症狀。這個檔位要**極度吝嗇**——它直接決定值班的人能不能睡。
- **Ticket（工單）**：上班時間看。給「不處理會出事，但還有幾小時到幾天緩衝」的事——磁碟容量預警、緩慢的 error budget 漏血、需要排程的維護。
- **Log-only**：只記錄，不主動通知。給「事後排查可能用得上、但當下不需要任何人知道」的事件。

光有檔位還不夠，告警還需要一層**降噪**，否則瞬間的抖動、或一個根因引發的告警風暴，都會製造一堆假 page。降噪有兩種不同性質的手段：一種是**去抖（debounce）**，針對時間維度上的瞬間抖動；另一種是**關聯與聚合**，針對多條告警之間的拓樸依賴。幾個常見的機制：

**for duration（持續才告）**：不是「錯誤率一超標就告」，而是「錯誤率超標**持續五分鐘**才告」。這濾掉了那些一秒就自己恢復的瞬間尖峰。代價要清楚：拉長這個視窗，等於**延後**了真實事件的告警——你用「晚五分鐘知道真問題」換「不被瞬間抖動吵醒」，這個權衡要根據事件的損失速率來定。

**依賴抑制（dependency inhibition）**：如果資料庫掛了，那麼依賴它的二十個服務會**同時**全部開始告警——但根因只有一個。依賴抑制讓你定義「上游掛了就抑制下游的告警」，把一場二十條的告警風暴收斂成一條指向根因的 page。風險也要說清楚：抑制規則寫錯了，會把真正獨立的問題一起吞掉，於是你在某次事故裡少了一條本該響的告警。

**分組（grouping）**：同一個事件觸發的多條告警（同一個部署導致五個 endpoint 同時變慢）合併成一則通知，而不是連發五條把人的手機震到沒電。

這些機制都在做同一件事：在「真問題」和「人的注意力」之間，加一層過濾，讓只有夠真、夠急、夠濃縮的訊號才穿透到人那裡。但這些都還是戰術。要把告警的訊噪比從根本上拉高，需要一個能直接量化「使用者正在受影響」的訊號——這就是 error budget burn rate 的舞台。

## 對 error budget 燒穿告警：把症狀變成一個可調的旋鈕

症狀告警最好的落地形式，是對 **SLO 的 error budget 燒穿**告警（SLO 與 error budget 的機制見〈SLI、SLO 與 error budget〉，這裡只取它作為告警訊號的部分）。

先把概念對齊到剛好夠用。SLO 是你對自己的可靠性承諾，比如「過去 30 天，99.9% 的請求是好的」。那允許壞掉的 0.1%，就是你的 **error budget**——這個月可以「花掉」的失敗額度。**Burn rate（燃燒率）** 衡量你**正在以多快的速度花掉這個額度**：burn rate ＝ 1 表示你正好以「剛好一個月燒完」的速度在花；burn rate ＝ 10 表示你燒得比預算快十倍，照這速度三天就花光整月的額度。它的演算法就是把某個觀測窗口內的實際失敗比例除以允許的失敗額度——`burn rate = (窗口內 bad / total) / (1 − SLO)`；99.9% SLO 下，失敗率 1.44% 就對應 burn rate 14.4，這也是底下那個門檻表能直接落成一條查詢的接點。

為什麼 burn rate 是個好的告警訊號？因為它把「使用者受影響的程度」濃縮成了**一個數字**，而且這個數字自帶**流量歸一化**——它是比率，不會因為半夜流量整體下滑就跟著漂移（這正是「QPS < 100 就告」這種靜態門檻天天在半夜誤報的毛病：被監控的量本身有日夜週期，絕對門檻就注定週期性虛驚）。更妙的是，這個數字直接對應急迫性：燒得越快，越該立刻起床。

不過「比率」這層歸一化只擋掉了**門檻隨流量漂移**這一種誤報，擋不掉另一種：**樣本太少時比率本身會劇烈跳動**。半夜某個一小時的窗口裡若只有兩個請求、剛好一個因為隨機網路抖動失敗，錯誤率就是 50%，對 99.9% SLO 而言 burn rate 衝到 500，遠超任何 page 門檻——可是它實際只燒掉了一個請求的預算，對整月幾乎毫無影響。所以實務落地時，burn rate 規則幾乎都會再串一個**最小錯誤數／最小流量**的條件（例如「窗口內失敗數也要超過某個絕對值才算數」），把這種低樣本的統計噪音擋在 page 之外。比率解決了流量的**週期性**，但流量的**絕對稀少**得靠這個額外閘門補上。

問題是，單一一個 burn rate 門檻調不好。調成「burn rate > 1 就 page」，太敏感——任何一個小波動讓你短暫地燒快了一點，就半夜把人吵醒，疲勞回來了。調成「burn rate > 100 才 page」，太遲鈍——一個雖然不那麼劇烈、但持續漏血的故障（比如 burn rate 穩定在 5，一週就把預算燒穿），會在門檻底下悄悄把你的整月預算耗光，等你發現已經太遲。一個固定門檻沒辦法同時抓住「急性大出血」和「慢性漏血」這兩種形態完全不同的故障。

Google SRE 的解法是**多視窗多燃燒率（multi-window multi-burn-rate）**告警，它優雅地同時解決了「夠敏感抓急、夠沉得住氣不被波動騙、又不漏掉慢性漏血」這三個看似矛盾的要求。我們把它的數字算到底。

## 把多視窗燃燒率算到底

設定：SLO 是 99.9% 成功率，視窗 30 天。error budget 就是 0.1% 的請求可以失敗。

第一個要算清的數字是那個著名的 **14.4**。Google SRE Workbook 推薦的第一檔 page 條件是「在 1 小時內燒掉 2% 的月度預算」。把它換算成 burn rate：一個月（30 天）有 720 小時，如果你以 burn rate ＝ 1 的速度燒，1 小時應該燒掉 `1 / 720 ≈ 0.139%` 的預算。現在你要在 1 小時內燒掉 **2%**，那燒得有多快？

```
burn rate = (預算燒掉的比例) / (時間佔整月的比例)
          = 2% / (1 hour / 720 hours)
          = 0.02 / (1 / 720)
          = 0.02 × 720
          = 14.4
```

所以「1 小時燒掉 2% 預算」等價於「burn rate ＝ 14.4」。這個數字不是拍腦袋來的，它是「在 1 小時這個夠快的反應窗口內，燒掉一個夠顯著（2%）的預算份額」反推出來的速率。Workbook 的三檔推薦長這樣：

| 嚴重度 | 長視窗 | 短視窗 | burn rate | 觸發時已燒掉的月預算 |
|---|---|---|---|---|
| Page | 1 hour | 5 min | 14.4 | 2% |
| Page | 6 hours | 30 min | 6 | 5% |
| Ticket | 3 days | 6 hours | 1 | 10% |

看出設計的巧思了嗎？三檔對應三種故障形態。**14.4 那一檔抓急性大出血**——短短一小時就燒掉 2%，這是劇烈故障，必須立刻 page。**6 那一檔抓中速漏血**——一小時的高燒可能是個尖峰，但如果六小時內穩定地燒掉了 5%，這是個沒在自己恢復的真問題，也該 page。**1 那一檔抓慢性漏血**——burn rate 只有 1（正好預算速度）聽起來無害，但若它持續三天、累積燒掉 10% 的月預算，這代表有個低強度但長期的問題在啃你的可靠性，值得開張工單慢慢查，不必半夜起床。

那「多視窗」裡的**短視窗**（5 分鐘、30 分鐘）是幹嘛的？它是用來**確認故障還在持續、好讓你能快速關閉告警**的。考慮這個 edge case：錯誤在某一小時的前十分鐘大爆發、然後自己恢復了。只看 1 小時的長視窗，這十分鐘的爆發會讓「過去 1 小時」的平均 burn rate 維持在高點將近一小時——於是故障早就沒了，告警還在響，又是一條雜訊。加上「過去 5 分鐘的 burn rate 也必須超標」這個**同時成立**的條件後，一旦真實故障停止，短視窗的 burn rate 五分鐘內就掉下來，告警隨之自動消解。長視窗負責**抗抖動**（要燒夠久才認定是真問題），短視窗負責**快速復原確認**（故障一停就閉警），兩者用 AND 串起來，才是「多視窗」的全部意義。

Workbook 給的經驗法則是短視窗取長視窗的 1/12——1 小時配 5 分鐘、6 小時配 30 分鐘，正是這個比例。

## 把它接回那個凌晨三點

現在用這套東西，重寫開場那一幕。

假設那個團隊原本有 120 條告警規則，其中約 90 條是原因告警（CPU > 80%、記憶體 > 85%、各種 queue > 1000……），而且每一條都設成 page。一個月統計：總觸發 800 次，事後只有約 40 次對應真正需要人介入的事件——誤報率 95%。值班的人三週後養成「先靜音」的習慣，於是某次真正的資料庫連線池耗盡告警（一個真症狀：寫入大量逾時、500 暴增）也被條件反射靜音，事故延誤四十分鐘才有人處理。

改法不是去「優化」那 90 條原因告警的門檻——那是在打磨一個本就不該存在的東西。改法是**結構性的**：

把那 90 條原因告警的 **page 屬性全部拿掉**，降級成儀表板和工單。CPU、記憶體、queue 長度從此不再半夜吵人，它們乖乖待在儀表板上，等真症狀把人叫醒後當診斷線索用。

然後只保留**少數幾條症狀告警**——對使用者面向的 SLI（成功率、p99 延遲）做多視窗多燃燒率告警。整個服務也許只需要那麼六條 page 級規則。

效果可以估算。原本每月約 800 次觸發、其中 ~95% 是原因告警的雜訊；砍掉原因告警的 page 後，剩下的 page 只來自症狀燒穿，而症狀燒穿本身就是真問題的代名詞——加上多視窗去抖濾掉瞬間波動，**每月 page 量從數百次降到個位數。** 更關鍵的是質變：現在每一條 page 響起來，都對應「使用者正在受影響」。值班的人重新學到的統計規律變成「告警響＝真有事」，那條「先靜音」的條件反射就再也養不起來了。當下次資料庫連線池真的耗盡時，那條 page 在一個安靜了很久的手機上響起，它會被當一回事。

注意這裡發生的事：我們沒有讓系統「監控得更少」——CPU、記憶體那些訊號一個都沒丟，它們還在儀表板上，排障時照用。我們只是把它們從「主動拽人起床」的角色，挪回「被動供人查案」的角色。**告警變少了，可觀測性一點沒減。** 這兩件事能分開，正是整個重新設計的支點。

## 幾個會反咬你的邊角

把上面的原則落地時，有幾個非顯然的坑，每一個都見過真實系統在上面栽跟頭。

**靜態門檻撞上週期性波動。** 「QPS 低於 100 就告」這種規則，在白天尖峰時段沒問題，但半夜本來流量就低，於是它天天在凌晨誤報。任何基於**絕對值**的門檻，只要被監控的量本身有日夜或週末的週期，就注定週期性誤報。兩條出路：要嘛改用相對基準（和上週同一時刻比，跌超過 X% 才告），要嘛乾脆別對流量絕對值告警、改對 SLI 燒穿告警——burn rate 是比率，自帶流量歸一化，半夜流量整體偏低它也不會跟著漂移（但別忘了前面那個反向陷阱：流量低到只剩零星幾個請求時，比率反而會被單一隨機失敗放大，所以燒穿規則仍要配一個最小錯誤數閘門）。

**告警沒有 runbook，等於只完成了一半。** 一條 page 把人叫醒，他第一個念頭是「這響了我現在該做什麼」。如果答案是「不知道，得從頭查起」，那這條告警的價值大打折扣——它製造了急迫感，卻沒提供方向。每一條 page 級告警都該連到一份 runbook：這個症狀通常意味著什麼、先看哪幾個儀表板、有哪些已知的快速處置。沒有 runbook 的 page，是把「思考」這個最耗時的步驟全壓在一個剛被吵醒、腦子還沒開機的人身上。

**抑制規則本身會悄悄漏報。** 依賴抑制很好用，但它是一把雙面刃。你定義「上游 A 掛了就抑制下游 B、C、D 的告警」，是假設 B、C、D 的問題都源於 A。但如果某次 B 是因為**自己的**獨立問題掛了，恰好 A 也在同時出了狀況——B 的告警會被 A 的抑制規則一起吞掉，於是現場看起來只有一個根因，實際上有兩個，其中一個被抑制規則悄悄吞掉了。抑制是在「告警風暴」和「漏掉獨立故障」之間做取捨，規則越激進，漏報的風險越高。

**降噪降過頭，會把真問題也降掉。** for duration 拉太長、抑制太激進、門檻太高——這些降低疲勞的手段，每一個都在把刻度往「漏報」那邊推。告警設計不是單向地「越安靜越好」，而是在**訊噪比**這個指標上找最佳點：既不能讓雜訊淹沒真告警，也不能為了安靜而把真告警一起靜音。判斷一條告警該不該存在，永遠回到同一個問題：**它響了，有一個人需要立刻做一件具體的事嗎？** 答得出來，留著；答不出來——那它不該 page，頂多進儀表板。

## 為什麼是這個形狀

退一步看，告警設計的整個樣貌，都是從一個約束裡長出來的：**真正稀缺的不是訊號，是人在被打斷時還願意認真對待告警的次數。**

正因為這個資源稀缺，你不能對每個內部零件的異常都告警——那會在使用者根本沒事的時候反覆消耗它，於是要「對症狀、不對原因」。正因為它稀缺，你要把告警分成 page / ticket / log 三檔，只有「使用者正在受影響」才配得上最貴的那一檔 page。正因為它稀缺，你要用多視窗燃燒率去抖，確保響起來的每一條都夠真、夠急、不是瞬間波動的虛驚。連那些看似「越多越安全」的覆蓋率本能，都得讓位給訊噪比——因為一條淹在雜訊裡的告警，和一條根本不存在的告警，後果是一樣的。

好的告警系統因此是**吝嗇**的。它一個月只響幾次，但每一次響，值班的人都會立刻坐起來——因為他從長期的經驗裡學到，這東西從不說謊。這種吝嗇不是功能上的偷工減料，而是對那個稀缺資源的精心守護。一個吵但大多是雜訊的告警系統，比沒有告警還危險，因為它一邊給你「有在看」的安全假象，一邊悄悄把唯一真正在看的那個人，訓練成對它視而不見。
