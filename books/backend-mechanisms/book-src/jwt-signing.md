# JWT 與簽章：結構、HS256/RS256 與 JWKS

一個請求落在你的服務上，標頭裡夾著一行看起來像亂碼的字串：`eyJhbGci...`。這個服務沒有共享的 session store，它甚至可能跑在離使用者最近的某個邊緣節點上，跟簽發這個身分的中央認證服務隔著半個地球。但它得在幾微秒內回答一個沉重的問題：**這個請求到底是誰發的？** 它不能為了問這個去打一趟資料庫——那會在每個請求上加一次網路往返，把「無狀態、可水平擴展」這件事整個賠掉。

它需要的，是一張能**自己證明自己**的通行證：一張只要在本地檢查就知道真偽、不必回頭問任何人的票。JWT（JSON Web Token，RFC 7519）就是這張票，而讓它「自己證明自己」的那塊核心機制，是**簽章**。這一章要把這張票拆開——它的三段結構、底下的簽章到底在保證什麼、為什麼會分裂成「同一把鑰匙」與「一對鑰匙」兩條路線、以及一套叫 JWKS 的基建怎麼讓「一處簽、多方驗」這件事在不停機的前提下成立。我們也會看到，這套機制最致命的幾個破口，全都來自同一個誤會：把「簽章」當成了「加密」，或把「一把 key」當成了一個概念。

## 為什麼不能只是「夾帶身分」

先把問題講難。最天真的無狀態做法是這樣：使用者登入後，伺服器發一張票，票上明白寫著 `user_id=42, role=admin`。之後每個請求帶著這張票來，服務讀一讀票面，就知道你是 42 號、是管理員，不必查 store。

這在十秒內就會崩。因為這張票經過的每一站——瀏覽器、代理、CDN、使用者自己的開發者工具——任何一方都能把 `user_id=42` 改成 `user_id=1`，把 `role=admin` 從無到有加上去。**沒有任何東西阻止持票人改寫票面。** 一張可以被任意竄改的身分證明，等於沒有證明。

問題的本質，於是收斂成一個密碼學上有明確答案的需求：**完整性（integrity）與來源（origin）**。拿到一張票，驗證方要能確定兩件事——這張票確實是某個我信任的簽發者發的（來源），而且從簽發到現在一個位元組都沒被動過（完整性）。注意這裡**不**包含第三件事：機密性。票面內容是不是只有我看得到？JWT 的答案是「不」——而這個「不」日後會變成一個反覆被踩的坑，我們留到後面。

達成「完整性＋來源」而不必把驗證方也變成可信第三方的工具，就是**簽章**。簽章的數學承諾很乾脆：簽的人用某個祕密產生一段簽章，附在資料後面；驗的人能確認「這段資料配這段簽章，確實出自握有那個祕密的人之手」，而且資料只要改動一個位元，簽章立刻對不上。把這個承諾套到票上，竄改 `user_id` 的攻擊就死了——你能改票面，但你算不出配得上新票面的新簽章，因為你沒有那個祕密。

## 三段式結構：兩段明文，一段印章

JWT 最常見的形態是一個 JWS（JSON Web Signature，RFC 7515）——三段 base64url 編碼的字串，用兩個句點串起來：

```
header . payload . signature        # 示意，不可執行
eyJhbGciOiJSUzI1NiIsImtpZCI6IjJhIn0 . eyJzdWIiOiI0MiIsImV4cCI6MTcw... . NHVaYe26...
```

把它解開來看，三段各司其職：

- **header**：一個 JSON 物件，最重要的欄位是 `alg`（這張票用哪個簽章演算法，例如 `HS256`、`RS256`、`ES256`），通常還有 `typ`（值是 `JWT`）和 `kid`（key id，指向「用哪一把鑰匙簽的」，等講 JWKS 時會用到）。
- **payload**：又叫 claims，也是一個 JSON 物件，裝的是這張票宣告的事實。RFC 7519 定義了一組**註冊 claim**：`iss`（issuer，誰簽的）、`sub`（subject，這張票是關於誰的）、`exp`（expiration，到期時間，以 epoch 秒計）、`iat`（issued-at，簽發時間）、`aud`（audience，這張票是發給誰用的）、`jti`（這張票的唯一 id）。除此之外你可以塞自訂 claim。
- **signature**：對前兩段算出來的印章。

這裡藏著第一個**非顯然、但理解一切的細節**：簽章到底簽在什麼東西上？直覺會說「簽 payload 那個 JSON 物件」。錯。RFC 7515 把簽章的輸入（JWS Signing Input）定義得很死：

```
ASCII( BASE64URL(header) || '.' || BASE64URL(payload) )
```

也就是說，**被簽的不是原始的 JSON，而是「已經 base64url 編碼、用句點接起來的那兩段字串」本身**。為什麼這個區別重要？因為 JSON 是有歧義的序列化格式——同一個邏輯物件，欄位順序不同、空白不同、Unicode 跳脫不同，可以序列化成許多種不同的位元組。如果簽的是「邏輯物件」，那驗證方就得先把 payload 反序列化、再用「相同規則」重新序列化一遍才能比對，而任何兩端序列化規則的細微不一致，都會變成一個簽章對不上的玄學 bug，或更糟——一個讓攻擊者能塞進不同位元組卻通過驗證的破口。RFC 把簽章釘在「收到的那串確切位元組」上，整個歧義空間就消失了：驗證方拿到 token，連反序列化都不必，直接對 `parts[0] + "." + parts[1]` 這串原文驗簽，對上了才去解析 claim。**先驗印章、再讀內容**，順序不能反。

還有一件必須講白的事：**base64url 不是加密，是編碼。** payload 那段 `eyJzdWIi...` 看起來像亂碼，但它只是把 JSON 做了 URL-safe 的 base64 轉換——任何人貼進任何一個 base64 解碼器，`{"sub":"42","exp":...}` 就原形畢露。簽章保護的是「不能改」，**不是**「不能看」。這兩段是公開的明文，只有第三段印章是密碼學產物。

## 驗證一張票，到底做了哪幾件事

把驗證流程在腦中放慢，會看到它其實是一串必須**全部通過**的關卡——少做任何一關都是一個洞：

```
parts = token.split(".")                       # 必須剛好 3 段，否則直接拒
header = json(base64url_decode(parts[0]))

assert header.alg in EXPECTED_ALGS             # 鎖死演算法！不信 token 自報（見下文）
key = resolve_key(header.kid)                   # 從 JWKS 或本地金鑰庫挑出對應的鑰

signing_input = parts[0] + "." + parts[1]
assert verify_sig(signing_input, parts[2], key) # 對「原文兩段」驗印章

claims = json(base64url_decode(parts[1]))       # 印章過了，才解析 claim
assert claims.exp > now()                       # 沒過期
assert claims.iss == EXPECTED_ISS               # 是我信任的簽發者
assert EXPECTED_AUD in claims.aud               # 這張票確實是發給我的
# 全過 → 信任 claims，不查任何 store
```

關鍵在最後一行的註解：**整個過程沒碰任何資料庫。** 驗簽只用到一把本地（或快取下來的）金鑰，加上幾個字串比對。這就是 JWT 能無狀態、能水平擴展、能在邊緣節點上驗證的全部祕密——也是它**難以即時撤銷**的全部根源（撤銷的取捨是另一個機制的主場，見〈token 的生命週期與撤銷〉）。便利與不可撤銷是同一枚硬幣的兩面：正因為驗證方不必回頭問簽發者「這張票還算不算數」，簽發者也就失去了喊停的開關。

注意驗簽通過**只**代表「沒被改、出自可信簽發者」。它**不**代表這張票沒過期、不代表它是發給你這個服務的。`exp`、`iss`、`aud` 這些 claim 得逐一檢查——印章對了，是相信票面的**前提**，不是終點。只驗印章不驗 claim，等於放過一張簽得好好的、但早該作廢或根本是發給別人的票。

## 分水嶺：同一把鑰匙，還是一對鑰匙

到這裡，「簽章」還是個黑盒。打開它，會看到一個影響整個系統架構的分岔：簽和驗，用的是**同一把**祕密，還是**一對不同**的鑰匙？這個選擇不是密碼學品味問題，它直接決定了你的**信任邊界**長什麼樣。

**對稱簽章**走的是 `HS256`——HMAC（hash-based message authentication code）配 SHA-256。它只有一把祕密，一個 shared secret。簽的時候用它，驗的時候**也**用它。這條路又快又簡單，沒有公私鑰那套數學開銷。但它有一個無法迴避的後果：**能驗的人，就能簽。** 驗證方手上握的那把 secret，跟簽發方手上的是同一把；驗證方完全有能力自己偽造一張任何內容的票，蓋上一個完美的印章。在「簽和驗是同一方」或「同一個信任域裡少數幾個互信服務」的場景，這沒問題——反正大家本來就互信。但你每多一個驗證方，就多一份這把「能偽造一切」的祕密散落在外，外洩面隨之擴大。

**非對稱簽章**走的是 `RS256`（RSASSA-PKCS1-v1_5 配 SHA-256）或 `ES256`（ECDSA 在 P-256 曲線上配 SHA-256），它把鑰匙拆成一對：**私鑰用來簽，公鑰用來驗**，兩把不同，而且從公鑰反推不出私鑰。這個不對稱性帶來一個決定性的性質——**握有公鑰的人，只能驗，不能簽。** 於是你可以把「簽的能力」死死鎖在一處（私鑰只待在簽發方的保險庫裡），同時把「驗的能力」自由散佈到任意多的下游服務、甚至完全不受你控制的第三方（公鑰本來就設計成可以公開）。任何一個下游被攻破，攻擊者拿到的也只是一把公鑰——他能驗票，但偽造不出新票。

這就是分水嶺的第一原理：**對稱簽章把「偽造能力」複製給每一個驗證方；非對稱簽章把「偽造能力」收斂在唯一一處。** 一旦你的票是「身分服務簽發、被一堆下游 API 各自驗證」，這個差別就是天與地——用 HS256，你得把那把能偽造一切的 secret 發給每個下游，其中任何一個被滲透（或某個工程師不小心把它 commit 進了公開 repo），整條身分鏈就此淪陷；用 RS256/ES256，下游只拿到公鑰，攻破再多個也偽造不出一張假票。這也是為什麼 OIDC 的 ID Token——一張「身分提供者簽、眾多依賴方各自驗、甚至跨組織」的票——規格預設要求支援 RS256：把驗證能力安全地交到你不完全信任的多方手裡，非對稱是唯一站得住的形狀。（規格也留了一扇窄門：當依賴方是持有 `client_secret` 的機密 client、簽發與驗證實質上仍是「共享一個祕密的兩方」時，ID Token 也允許用 HS256 對稱簽。）

## 把代價算出來：一張票要付多少位元組

非對稱不是免費的，它的代價可以手算到底，而且這個帳一旦攤開，ES256 為什麼漸漸贏過 RS256 就一目了然。

三種演算法的印章大小差距是數量級的。HS256 的印章是 SHA-256 的輸出，固定 **32 bytes**。RS256 的印章大小等於 RSA 金鑰長度——用業界下限的 RSA-2048，印章就是 **256 bytes**，整整是 HS256 的八倍。ES256 在 P-256 上的印章是兩個各 32 bytes 的值（r 和 s）併起來，**64 bytes**——只有 RS256 的四分之一，安全強度卻相當（P-256 的 128-bit 安全等級，大約等同 RSA-3072，比 RSA-2048 還略強一截）。

這 192 bytes 的差距（256 vs 64）聽起來不痛不癢，直到你把它乘上流量。設想一個閘道每秒驗 50,000 個請求，每個請求的標頭都背著一張 token。光是印章那一段，RS256 與 ES256 的差是：

```
每請求多出的位元組:  256 - 64 = 192 bytes
每秒多出的入站流量:  192 bytes × 50,000 = 9.6 MB/s ≈ 77 Mbps
```

每秒近 80 Mbps 的頻寬，純粹耗在「同一份安全保證的更大號印章」上，一年到頭、每一站轉發都在付。再算上公鑰本身的大小（RSA-2048 公鑰約 256 bytes，P-256 公鑰約 64 bytes），以及行動端與物聯網裝置上 ECDSA 簽驗運算更省電的事實，現代部署在非對稱這條路上越來越常直接選 ES256。RS256 的地位更多是歷史與相容性——老系統、老函式庫對 RSA 的支援最齊全，所以它仍是「最不會出錯」的預設，但不是「最省」的選擇。

（順帶把帳算完整：這只是印章。如果你還往 payload 裡塞了完整的角色清單、權限明細、使用者偏好，一張 token 輕易就上 1–2 KB，那才是頻寬的大頭。所以慣例是票上只放身分與粗粒度的範圍，細粒度授權去查快取的授權服務，別讓每個請求都背著一本權限手冊跑。）

## JWKS：公鑰怎麼送到驗證方手上，又怎麼換

非對稱解決了「驗證方不能偽造」，但立刻冒出一個物流問題：那把公鑰，要怎麼送到散佈各地、甚至屬於別人的驗證方手上？而且——金鑰總要輪替（rotation），舊鑰用久了該換新的，**換的那一刻怎麼不讓正在飛的票全部驗不過、造成一次全站登出？**

把公鑰寫死（hardcode）在每個驗證服務的設定裡，是能跑，但一輪替就是惡夢：你得同時改幾十個服務的設定、重新部署，而且新舊交接的那個瞬間幾乎必然有票卡在縫裡。JWKS（JSON Web Key Set，RFC 7517）就是為了消掉這個惡夢而生的基建。

機制很直接：簽發方把自己的**公鑰**，以一份 JSON 發布在一個固定的 HTTPS 端點上，慣例路徑是 `/.well-known/jwks.json`。這份 JSON 的骨架是一個 `keys` 陣列，裡頭每把鑰是一個 JWK（JSON Web Key），每把都帶一個 `kid`：

```json
{
  "keys": [
    { "kid": "2a", "kty": "RSA", "alg": "RS256", "use": "sig", "n": "...", "e": "AQAB" },
    { "kid": "3b", "kty": "EC",  "alg": "ES256", "use": "sig", "crv": "P-256", "x": "...", "y": "..." }
  ]
}
```

回頭看 token 的 header——裡頭那個 `kid` 此刻就有了用處。簽發方簽票時，在 header 寫上「我這次用的是 `kid=2a` 那把」；驗證方收到票，讀出 `kid=2a`，到（已快取的）JWKS 裡撈出對應的公鑰來驗。這個 `kid` 對應關係，正是讓**輪替不必停機**的關鍵。輪替的舞步是這樣跳的：

1. 簽發方生成一把新鑰（`kid=3b`），先把它**加進** JWKS——此時 JWKS 同時有舊鑰 `2a` 和新鑰 `3b`，但簽發方還在繼續用舊鑰 `2a` 簽。
2. 等所有驗證方都重新抓過 JWKS、快取裡同時有了兩把鑰之後，簽發方才切換到用新鑰 `3b` 簽新票。
3. 此刻，舊票（header 寫 `kid=2a`）和新票（`kid=3b`）都驗得過——因為兩把公鑰都還在 JWKS 裡。
4. 等舊票全數自然過期（最長一個 token TTL）之後，簽發方才把舊鑰 `2a` 從 JWKS 移除。

整套過程沒有任何一張正在飛的票會驗不過，沒有一次停機，驗證方也不必改一行設定——它們只是在下次抓 JWKS 時自動學到了新鑰。這套「先加新、再切簽、後撤舊」的重疊期，就是無停機輪替的全部精髓。

但這份便利不是白拿的，它欠下兩筆要還的債。第一筆是**快取策略**：驗證方不能每驗一張票就去打一次 JWKS URL——那等於把無狀態驗證的好處賠光，還把自己的可用性綁死在那個端點上（JWKS 端點一掛，全體驗證失敗）。所以公鑰必須快取。但快取又不能太久——萬一快取了一天，而簽發方因為私鑰外洩緊急輪替了金鑰，你這邊還抱著舊鑰傻驗一整天。務實的解法是一個組合：給公鑰設一個合理的快取 TTL，**外加**一條「遇到不認得的 `kid` 就主動去重抓一次 JWKS」的規則——這條規則讓新鑰能在輪替後立刻被學到，而不必等快取自然過期。第二筆債是**信任這個端點本身**：既然你打算拿 JWKS 上的公鑰來判斷「誰是可信簽發者」，那這個 JWKS URL 就絕不能被掉包。它必須走 TLS、必須是寫死在設定裡你確知的那個 issuer 的端點——如果攻擊者能讓你去抓**他的** JWKS，他就能塞進**他的**公鑰，然後用配對的私鑰簽出任意身分，你還會一路綠燈放行。

## 它會怎麼騙你

JWT 的故障模式有個共同的氣味：它們幾乎都不是密碼學被破解（HMAC、RSA、ECDSA 本身都很硬），而是**驗證方在「該由誰決定用哪個演算法、哪把鑰」這件事上信錯了人**。把這幾個破口看透，比記住任何「最佳實踐」清單都有用。

**`alg: none` ——讓票自己宣布「我不用簽」。** JWT 的 header 裡有個 `alg` 欄位，問題是：這個欄位是**攻擊者能改的票面內容**。早年有些函式庫天真地照著 `alg` 說的去做——於是攻擊者把 `alg` 改成 `none`、把第三段印章直接刪空，函式庫一看「喔，這張票宣告自己不需要簽章」，就放行了。這等於讓嫌犯自己決定要不要被搜身。防法是一句話：**驗證方必須在伺服器端寫死它接受的演算法**，絕不讀 token 自報的 `alg` 來決定怎麼驗。前面驗證流程裡那行 `assert header.alg in EXPECTED_ALGS`，守的就是這道門。

**演算法混淆（RS256 → HS256 confusion）——同一個破口的高級版，也最隱蔽。** 設想一個服務本來用 RS256 驗票，手上有簽發方的**公鑰**（公開的，誰都拿得到）。攻擊者做一件陰險的事：他拿到那把公鑰，把一張票的 `alg` 從 `RS256` 改成 `HS256`，然後**把那把公鑰當成 HMAC 的 secret**，對票面算出一個 HS256 印章。現在問題來了——如果驗證函式庫也是「照 `alg` 說的辦」，它看到 `HS256`，就會去拿「驗證用的金鑰」當作 HMAC secret 來驗；而它驗證用的金鑰，正是那把公鑰。於是攻擊者用公鑰簽、驗證方用公鑰驗，印章完美吻合，一張偽造的票就這樣通過了。這個攻擊的根因，是把「對稱演算法的密鑰」和「非對稱演算法的公鑰」當成了一個可以互換的概念——它們在程式裡都是「那個 key 變數」，但語意天差地別：一個是必須保密的祕密，一個是公開散佈的資料。防法仍是同一條：**驗證方鎖死演算法類別**，設定成「我只接受 RS256」的服務，碰到宣告 `HS256` 的票就該當場拒絕，根本不給它「拿公鑰當 secret 驗」的機會。針對這類問題的系統性對策，RFC 8725（JSON Web Token Best Current Practices，2020）的核心建議就是：把每一把金鑰**綁定到唯一一個演算法**，永遠不信 token header 裡的 `alg`。（它的後續修訂 `draft-ietf-oauth-rfc8725bis` 把規則從 12 條擴充到 15 條，但截至 2026-06 仍在 IETF draft 階段、尚未取代 RFC 8725，別當已定案的標準引用。）

**「JWT 是加密的，所以我把敏感資料放進去。」** 回到最前面埋的那個坑。簽章保證完整性與來源，**不**保證機密性——payload 是 base64url 明文，貼進任何解碼器就一覽無遺。把密碼、信用卡號、個資塞進 JWT payload，等於把它們印在一張防偽但透明的卡片上：別人改不了，但人人看得到。真要機密性，得用 JWE（JSON Web Encryption）另外加密那一層，那是完全不同的機制。

**「HS256 嘛，secret 隨便設一個就好。」** HS256 的全部安全性，繫於那把 secret 的熵。用一個短的、像密碼一樣可猜的字串（`secret`、`mykey123`）當 HS256 的 secret，等於沒簽——攻擊者可以拿到一張你簽的票，離線地對著它暴力嘗試各種 secret，因為驗證的數學他完全公開、不必跟你互動。一旦試中，他就能簽任意票。HS256 的 secret 必須是高熵的隨機值（夠長的隨機位元組），這不是建議，是這套機制能成立的前提。

**時鐘的相依。** `exp` 和 `nbf`（not-before）的判斷，靠的是**驗證方自己的時鐘**。如果簽發方和驗證方的時鐘飄移了幾秒，一張剛簽好的票可能在某些驗證方那裡被當成「還沒生效」或「已經過期」而被拒。慣例是留一個小小的時鐘容差（clock skew，通常幾十秒到一兩分鐘）來吸收這種飄移——但這個容差也是雙面的：開太大，等於變相延長了每張票的有效期，讓本該過期的票多活一陣子。這是一個沒有完美值、只能在「容忍時鐘飄移」與「讓過期及時生效」之間拿捏的旋鈕。

## 為什麼是這個形狀

退一步看，JWT 與它底下的簽章，整個樣貌都是從一個原始的需求裡長出來的：**讓一張票自己證明自己，好讓驗證方不必回頭問任何人。**

正因為要「不必回頭問」，它必須是自包含的——身分與時效全寫在票面上，於是有了那一串 claim。正因為票面是公開、可被任何中間人讀到改寫的，光寫上身分不夠，必須蓋一個改不動的印章——於是有了簽章，而且簽的是「收到的那串確切位元組」、不是有歧義的 JSON。正因為「能驗的人是否該能簽」這個問題在分散式場景裡答案是「不該」，簽章分裂成了對稱與非對稱兩條路——前者快而互信、後者把偽造能力鎖在一處。正因為非對稱要把公鑰散佈到多方、又要能換鑰而不停機，才長出了 JWKS 這套帶 `kid` 的、可重疊輪替的公鑰發布基建。

而它所有最危險的破口，又都指回同一個薄弱點：這套機制把「用哪個演算法、哪把鑰」的線索寫在了攻擊者搆得到的票面上。所以整套防禦的脊椎只有一句——**演算法和金鑰的決定權，必須牢牢握在驗證方手裡，永遠不交給票自己宣告。** 看懂這一句，你就看懂了 `alg: none`、看懂了演算法混淆、也看懂了為什麼「先驗印章、再讀內容，且只信你預期的演算法」是這張小小通行證身上，最不能讓步的紀律。
