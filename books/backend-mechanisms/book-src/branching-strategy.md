# 分支與發布策略：trunk-based vs git-flow

兩個工程師，星期一早上各自從 `main` 切出一條分支。一個去重構訂單服務的金流模組，一個去把整個結帳流程從同步改成事件驅動。各自埋頭兩週，各自的分支綠燈、測試通過、PR 看起來乾乾淨淨。到了要合回主幹那一天，災難不在任何一條分支裡——它在兩條分支的**交集**：兩個人都動了那個共用的 `OrderService`，一個把方法改成同步回傳，一個把它拆成發事件、不回傳。Git 的三方合併在這裡舉手投降，吐出一片 `<<<<<<<` 衝突標記，而真正棘手的是：**沒有任何一方的測試覆蓋了「兩邊合在一起之後」的行為**。各自都對，合起來壞掉，而且要到合併那一刻才會知道。

這就是分支策略真正在管的東西。表面上它是一份「誰可以 push 到哪、什麼時候開 PR」的 Git 慣例，骨子裡它決定的是一個更底層的量：**團隊多久整合一次彼此的工作**。而整合的頻率，幾乎單手決定了上面那場合併地獄會多痛、壞掉的變更要多久才被發現、以及團隊敢不敢一天部署十次。把這條線看清楚，trunk-based 和 git-flow 之間那場看似宗教戰爭的爭論，就會塌縮成一個冷靜的工程問題。

## 為什麼分支拖得越久越痛：發散是會複利的

先把問題講難。直覺會說：「分支開久一點怕什麼？最後 merge 一次不就好了？」這個直覺錯在它把合併的成本當成**線性**的——以為兩週的分支就是兩天分支的七倍痛。實際上，合併衝突的成本隨分支存活時間**超線性**增長，原因是兩條分支不是各自靜止的，它們是兩條**同時在動**的軌跡。

把它想成兩個人在一張白板上各自畫圖。如果每隔幾分鐘就互相對一次、把對方畫的併進來，每次只要調和幾筆。但如果各畫兩小時再來合，問題不是「線多了」，而是**結構漂移了**——你重新命名的那個函式，對方在他那邊呼叫了三十次；你刪掉的那個欄位，對方基於它長出了一整套邏輯。衝突的數量大致正比於「兩條分支各自改動的範圍」的乘積，而改動範圍又隨時間累積。兩個 O(t) 相乘，痛苦就是 O(t²)。這就是為什麼「分支多活一倍，合併不是多痛一倍而是多痛數倍」。

更陰險的是第二層成本：**語意衝突（semantic conflict）**。Git 的文字合併只看行有沒有撞在一起。但開場那個例子裡，兩個人改的是**不同的行**——一個改了 `OrderService` 的定義，一個改了某個遠在另一個檔案、呼叫它的地方。Git 會「乾淨地」合併成功、沒有半個衝突標記，然後在執行期爆炸：呼叫端期待一個回傳值，定義端已經不回傳了。文字層面的衝突 Git 幫你擋；語意層面的衝突它**完全看不見**，而它隨分支發散得越遠越多。分支拖得越久，你累積的不只是會被標出來的衝突，還有一整批**不會被標出來、要等執行才現形**的地雷。

於是「分支策略」這個問題，第一原理其實非常窄：**你願意讓彼此的工作發散到多遠，才強制整合一次？** 答案的兩個極端，長成了兩種完全不同的工作流。

## 一個極端：把整合推遲到最後的 git-flow

2010 年，Vincent Driessen 在一篇後來被引用到爛的部落格文章裡，提出了一套精緻的分支模型，後世稱為 git-flow（2026-06）。它的形狀是「多條長命分支各司其職」：

- `main`（原文叫 `master`）：永遠只放**已發行**的版本，每個 commit 對應一個正式 release，平時不動。
- `develop`：開發的主線，功能完成後先匯流到這裡。
- `feature/*`：每個功能一條，從 `develop` 切出、做完合回 `develop`。
- `release/*`：準備發版時從 `develop` 切出，在上面做最後的穩定化（改 bug、調版號），完成後同時合進 `main` 和 `develop`。
- `hotfix/*`：線上版本出緊急 bug 時，直接從 `main` 切出修補，修完合回 `main` 和 `develop`。

```
feature/* ──→ develop ──→ release/* ──→ main ──→ (tagged release v1.4.0)
                  (release/* 完成後同時合回 main 與 develop)
                  hotfix/*: 從 main 切出、修完合回 main 與 develop
```

看懂這套設計的關鍵，是看懂**它為什麼存在**。git-flow 不是給「一個持續運行、只有一個線上版本的網路服務」設計的——它是給**有明確版本、且必須同時維護多個在野版本**的軟體設計的。想像你出貨一套桌面軟體或一個 SDK：客戶手上跑著 v2.3、v3.1、v4.0 三個版本，你不能逼所有人升級。當 v3.1 爆出一個安全漏洞，你需要一條 `hotfix` 分支，從「v3.1 當時發布的那個 commit」切出來、只修那個洞、再發一個 v3.1.1，而完全不把 v4.0 還沒發布的新功能帶進去。`main` 上一個個 tag 標著「哪個 commit 是哪個對外版本」，`release` 分支讓你在發版前還能繼續在 `develop` 上開發下一版——這整套長命分支的複雜度，買到的是**對「多版本並存」這個現實的乾淨管理**。

代價就是第一原理算給我們看的那筆帳：`feature` 分支動輒活幾天到幾週，`develop` 到 `release` 的合併是一次性的大整合。git-flow 把整合**刻意推遲**了，於是它把開場那場合併地獄當成常態來承受——這對「一版一版發行」的軟體是合理的取捨，因為你本來就不頻繁整合、不頻繁發布。但把這套搬到一個一天想部署好幾次的網路服務上，你付了多分支的全部複雜度，卻**沒有任何多版本要維護**——`hotfix` 和 `release` 分支對一個「線上永遠只有一個版本」的服務根本沒有用武之地。值得一提的是，連 Driessen 本人後來都在原文頂端加了一段反思：如果你的團隊在做持續交付，他建議改用更簡單的工作流（如 GitHub Flow），別硬把 git-flow 套上去——他的原話是「web apps are typically continuously delivered, not rolled back, and you don't have to support multiple versions of the software running in the wild」，並補一句「panaceas don't exist」（2026-06）。設計者親口劃出了適用邊界。

## 另一個極端：頻繁到痛不起來的 trunk-based

如果合併的痛是 O(t²)，最直接的解法不是把合併變聰明，而是**讓 t 永遠很小**。這就是 trunk-based development 的全部直覺：每個人每天至少一次把自己的變更整合進那條唯一的共享主幹（trunk / `main`），分支只活幾小時、絕不長期偏離。

這裡有一個常被忽略的點：**trunk-based 並不是「持續整合（CI）」的一種工作流，它幾乎就是 CI 這個詞最初的意思**。CI 最早指的不是「跑測試的那台機器」——「CI server」是後來才有的詞——而是指**工程師持續地把工作整合到一起**這個行為本身。trunk-based 把這件事推到極致：把每個人的工作流向同一個整合點，而且做得很頻繁，頻繁到每次只差幾十行，O(t²) 裡的 t 小到平方也不痛。發散還沒來得及累積成結構漂移，就被下一次整合抹平了。

具體怎麼運作？trunk-based 不是一種形狀，而是一條光譜，依團隊規模分成兩種風格：

- **小團隊**：直接 commit 到 `main`，或開一條只活幾小時的微分支，當天合回。
- **較大團隊**：用短命的 `feature` 分支，由一個人或一對 pair 擁有，但一樣**至少每天**合回 trunk 一次——重點是「短命」，不是「沒有分支」。

落到主流實踐，最常見的版本是 **GitHub Flow**：`main` 永遠保持可部署狀態，功能走短命分支加 PR，merge 進 `main` 後即可部署。它是 trunk-based 的輕量版，介於嚴格 trunk-based 與 git-flow 之間，是多數網路服務的合理預設。需要更多環境結構的團隊則用 **GitLab Flow**，在 GitHub Flow 之上加 environment 或 release 分支。

但這套立刻撞上一個尖銳的矛盾，而這個矛盾正是理解 trunk-based 的核心。

## 矛盾：功能還沒做完，怎麼能天天合進主幹？

「每天合回主幹」聽起來很美，直到你問一個顯而易見的問題：**一個要做兩週的大功能，怎麼可能每天合一次？合進去的是半成品，主幹不就壞了、不就沒法部署了嗎？**

這個矛盾如果無解，trunk-based 就只是個天真的口號。它的解法，是 trunk-based 真正的技術前提：**把「程式碼合併」和「功能對使用者開啟」這兩件事，從根本上解耦。** 機制叫 feature flag（功能旗標，又稱 feature toggle）。

它的運作意外地簡單。你把未完成的功能寫好、合進主幹，但用一個執行期的開關把它的入口包起來：

```
if (featureFlags.isEnabled("new-checkout-flow", user)) {
  return newCheckoutFlow(order);   // merged but dormant in production
} else {
  return legacyCheckoutFlow(order); // what users actually hit, for now
}
```

`new-checkout-flow` 這個 flag 預設關著。於是 `newCheckoutFlow` 的程式碼**確實已經合進主幹、已經部署到生產環境**——但它是一段**潛伏的程式碼（latent code）**，沒有任何使用者走得到它。你可以每天把這個尚未完成的功能多寫一點、多合一點進主幹，主幹始終是綠的、始終可部署，因為對使用者而言什麼都沒變。等功能真的做完、測試到位，你不需要任何一次「大合併」——只要把 flag 打開。要是打開後出事，把 flag 關掉就回滾了，連重新部署都不必。

這就是 Martin Fowler 那句「decouple deployment from release」的具體模樣：**部署（deploy）是把程式碼放上生產環境，發布（release）是把功能開給使用者，這兩件事被 flag 切成了兩個獨立的時刻。** 沒有這個解耦，trunk-based 的「每天整合」就是空話——你要嘛被迫等功能完整才敢合（那就退化回長命分支、退回 git-flow 的世界），要嘛把半成品推上線砸到使用者。feature flag 不是 trunk-based 的可選配件，它是讓「頻繁整合」這件事在邏輯上**成立**的那塊地基。

值得停下來看一個非顯然的後果：flag 一旦能控制「誰看得到新功能」，它就不只是個開關，而是一根可以**漸進旋轉**的旋鈕——先開給 1% 的使用者、再 10%、再全開。這條路自然長進金絲雀發布（canary）的領地（實際怎麼安全地把流量導過去是部署策略的事，見〈blue-green、canary 與 rolling〉）。但 flag 自己也欠下了債，這一點下面會算。

## 把帳算清楚：一場真實的整合對撞

抽象講夠了，用開場那兩條分支把成本手算到底，對比兩種策略下它會長成什麼樣。

設定：團隊 6 人，每人每天約產出 200 行有效變更。兩個功能各需約 10 個工作天。我們關心的是**合併衝突的暴露時機與規模**。

**git-flow 的版本。** 兩條 `feature` 分支各自活 10 天才合回 `develop`。第一條先合，乾淨。第二條合的時候，要面對第一條這 10 天裡改動的全部表面——約 200 行/天 × 10 天 = 2000 行的改動範圍，加上自己這側也是 2000 行。文字衝突大致正比於兩側改動範圍的重疊；更糟的是語意衝突——`OrderService` 的契約在這 10 天裡被第一條分支改了，第二條分支的 2000 行全都建立在**舊契約**上，毫不知情。合併那天，工程師面對的是一個「需要重新理解對方兩週工作、才改得對」的大泥團，而 CI 直到此刻才第一次看到「兩邊合起來」的樣子。發現得最晚、規模最大、最難改——這正是 O(t²) 那條曲線的尾端。

**trunk-based 的版本。** 同樣兩個功能，但每人每天合回主幹。第二條分支的工程師在第一條改動 `OrderService` 契約的**當天傍晚**，就會在自己拉取最新主幹、跑 CI 時撞上問題——此時對方只改了約 200 行，自己也只多寫了約 200 行。衝突的規模小一個數量級，而且**他立刻知道契約變了**，可以馬上跟對方對齊，而不是兩週後對著 2000 行考古。同一場對撞，被切成了十次微不足道的小調和，而不是一次毀滅性的大爆炸。

數字的差別不在「總共要調和多少」——兩種策略下，兩個功能的本質衝突量級相當。差別在**什麼時候付、一次付多少**。git-flow 把成本累積成期末一次付清，且因為 O(t²)，期末那一筆比十次小額加起來**還貴**；trunk-based 把它攤成十次小額分期，每次小到幾乎無感。這也正是 DORA／《Accelerate》研究反覆量到的：高效能團隊的特徵之一，是**程式庫裡同時存活的分支「三條或更少」、且開發者至少每天把工作合進主幹一次**（2026-06）。研究把 trunk-based development 列為高效能交付的關鍵技術實踐之一，背後就是這條「小批量整合、早暴露、快回饋」的力學。

## 取捨的真相：trunk-based 把賭注押在哪

到這裡 trunk-based 像是完勝。但每個機制都要問它「在什麼條件下成立、什麼條件下騙你」，trunk-based 的成立條件相當苛刻，看不清它你會把好處當成免費的。

trunk-based 用「強制頻繁整合」換來「小衝突、快回饋」，但這個交換有一個**沉默的前提**：它把全部賭注押在**自動化測試**上。想清楚這條因果鏈——既然每個人每天都把變更合進那條「永遠要保持可部署」的主幹，那麼**唯一**擋在「某人的爛變更」和「主幹變成壞的、所有人都被卡住」之間的東西，就是合併前那道自動化測試閘。git-flow 還有 `develop`、`release` 分支當緩衝，壞變更先污染的是緩衝區；trunk-based 沒有緩衝區，主幹就是前線。測試不夠強，trunk-based 不會給你「小衝突」的好處，只會給你「一條頻繁壞掉、所有人輪流踩雷」的主幹——比 git-flow 還糟。

所以 trunk-based 對紀律的要求其實**比** git-flow 更高，這違反很多人的直覺。它要求：

- 一套快又可信的測試與 CI gate（慢測試會逼人跳過頻繁整合，整條鏈就斷了）。
- 真正的小 PR（PR 一大，審查者讀不動就放水蓋章，品質閘形同虛設——這是程式碼審查最反直覺的規律，見〈程式碼審查〉）。
- 紀律地使用 feature flag 來藏未完成的工作。

少了任何一條，「trunk-based」就只是「沒有分支、亂 push」的美化說法。它不是更鬆，是把規矩從「合併時的人工把關」前移成了「每次提交的自動化把關」。

而 feature flag 這塊地基，自己也會塌。flag 是有重量的——每一個 flag 都在程式碼裡留下一條 `if/else` 的隱藏分支，於是程式的**實際行為路徑數量隨 flag 數量指數成長**：10 個各自獨立的 boolean flag，理論上就是 2¹⁰ = 1024 種行為組合，沒有人測得完、也沒有人推理得清。一個早該「全開」或「全關」的 flag 忘了清掉，就等於系統裡永遠埋著一條沒人記得的隱藏路徑，哪天被某個邊角條件意外觸發，就是一樁查不出根因的詭異 bug。更危險的是把 flag 設反——一個本該關著的半成品功能，因為設定打錯而對使用者開了，等於**沒人按發布鈕就偷偷上線了未完成功能**。所以 flag 必須有生命週期：功能穩定後就刪掉 flag、讓程式碼收斂回單一路徑。**trunk-based 用 feature flag 還掉了「長命分支」這筆債，但 flag 本身是另一種債，只是這種債更可控、更可見**——前提是你真的去管理它的生命週期，而不是讓它無限累積。

## 怎麼選：你發布的是「一個服務」還是「一個個版本」

把兩端的力學攤開後，選擇判準收斂成一個出奇簡單的問題：**你是「持續把同一個服務推上線」，還是「一版一版地發行、且要支援舊版」？**

- 如果你跑的是 SaaS、網路服務，線上永遠只有一個版本、想頻繁安全地部署——trunk-based 系（嚴格 trunk-based 或 GitHub Flow）幾乎總是更好。它的整個形狀都和「頻繁、低風險地部署」同向。
- 如果你出貨的是套裝軟體、SDK、需同時維護多個已發行版本、有正式的 release cycle——git-flow 那套 `release`/`hotfix` 長命分支**才有存在的理由**。你確實需要「在維護 v3 的同時為 v2 出修補」這個能力，而那正是 git-flow 唯一真正擅長的事。

最常見的錯，是把 git-flow 當成「正統」「專業」的預設，套在一個持續交付的網路服務上。這是教科書級的過度工程：你扛下了多長命分支的全部協調複雜度，卻換不到任何多版本維護的好處——因為你根本沒有多個版本要維護。連設計者都明說了這條邊界，沿用它的唯一正當理由是「我真的有多版本要養」，而不是「大家都這樣」。

反過來也有一個對稱的陷阱，值得一提：**以為換成 trunk-based 就會自動變快。** 分支策略只是交付前置時間（lead time）裡的一環。如果你真正的瓶頸是「測試跑四十分鐘」或「部署要人工排程排隊」，光把 git-flow 改成 trunk-based 不會讓你變快——它甚至會在測試不足時讓主幹更頻繁地壞掉，因為你抽掉了長命分支這層緩衝，卻沒補上 trunk-based 賴以成立的測試強度。分支模型是放大器，不是引擎：它把你既有的工程紀律（測試、小批量、CI）放大成交付速度，但它本身不產生紀律。

## 為什麼是這個形狀

退一步看，這兩種策略不是兩種「風格偏好」，而是同一個底層變數——**整合頻率**——被兩種完全不同的發布現實拉到了兩端。

一端是「一個個版本、要養舊版」的世界。在那裡，整合本來就稀疏（你不頻繁發版），多版本並存是真實需求，於是長命分支的複雜度買得到對應的價值，git-flow 的形狀就長了出來。另一端是「一個服務、持續上線」的世界。在那裡，O(t²) 那條合併成本曲線是頭號敵人，唯一的解法是把 t 壓到最小、把整合做到最頻繁，於是 trunk-based 的形狀長了出來——而它一旦把整合頻率推到「每天」，就必然逼出 feature flag 來解開「沒做完怎麼合」的死結，也必然把全部安全性押到自動化測試這道唯一的閘上。

所以沒有「正確的分支策略」，只有「和你的發布模型對齊的分支策略」。當你看到一個持續部署的團隊在 `develop` 和 `release` 分支間打合併衝突的泥巴仗，或一個出貨多版本 SDK 的團隊試圖用 feature flag 模擬版本隔離，你看到的都是同一個錯誤：分支策略和它服務的發布現實脫了鉤。先回答「我發布的是一個服務還是一個個版本」，分支該長什麼樣，就不再是信仰問題，而是這個答案的必然推論。
