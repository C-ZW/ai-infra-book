# 冷啟動與暖機：第一個請求為什麼那麼慢

某個服務上線得很順。負載測試漂亮、p50 三十毫秒、CI 綠燈。然後它進了生產，凌晨三點被一個自動擴容事件叫醒：流量突然爬升，平台拉起一批新的實例去接，於是儀表板上冒出一條很醜的尾巴——p99 從六十毫秒跳到八百毫秒，維持了幾秒鐘，然後又掉回去。沒有任何程式碼變慢，沒有任何查詢退化，DB 一切如常。但確實有一群使用者，在那幾秒裡，每個人的請求都慢了一個數量級。

把鏡頭推近到那條尾巴裡的某一個請求，你會看到它落在一個**剛剛被創造出來的環境**上。它是這個行程接到的第一個請求。在它之前，這台機器什麼都還沒做過：runtime 剛啟動、相依剛載入、連線池是空的、JIT 還沒看過任何一行熱路徑、所有快取都是冷的。這個請求不只在做它自己的工作，它還順手替後面所有請求把整個環境**付了一次清**。它慢，不是因為它特別，而是因為它倒楣地排在第一個。

這就是冷啟動（cold start）。它不是一個 bug、不是一次退化，而是一種**狀態**——某個執行單位第一次服務請求時，必須先付一筆只在「冷」狀態才付得到的一次性成本。它的反面是暖機（warmup）：在真正的流量到達之前，主動把這筆成本**先付掉**，讓第一個真實請求落在一個已經熱好的環境上。整章要回答的就是這筆成本到底由哪些東西組成、為什麼幾乎所有高效能系統都注定要付它、以及在什麼情況下你該花力氣去消滅它、什麼情況下花這力氣只是燒錢。

## 為什麼第一次注定特別貴：攤平的代價

冷啟動不是某個平台的設計失誤，它是一個更深的工程取捨的影子。把它講透，得先看清一件事：**幾乎所有讓「之後每一次」變快的手段，都是用「第一次」變慢換來的。**

這句話聽起來像一句格言，但它其實是字面成立的機制。你建一個連線池，是為了讓之後每一次查詢都省下 TCP 加 TLS 握手——代價是池子在某個時刻得被建滿，那個時刻的請求要等握手。你讓 JIT 編譯器去最佳化熱路徑，是為了讓之後每一次呼叫都跑機器碼——代價是前幾千次呼叫得跑沒最佳化的慢版本，外加編譯本身佔的 CPU。你把相依預載進記憶體、把設定解析好、把 ORM 的 metadata 建起來，全都是同一個交易：**把重複的工作做一次、存起來，之後白拿。** 而「做一次」這個動作，必然發生在某個第一次。冷啟動就是這些「第一次」全部疊在同一個請求頭上時的總和。

把這筆總和拆開，它分三層，由外而內疊上去。

**最外面是環境層。** 一個新的容器、行程或函式環境要被排程到某台機器上、把映像拉下來、解壓、把 runtime 進程啟動起來。這層在 serverless 與自動擴容的場景最顯眼，因為那正是「無中生有一個新環境」的時刻。一個被縮到零的服務突然來了流量，平台得從頭生一個環境出來；一個 pod 在尖峰被 HPA 拉起，新 pod 從拉映像開始。

**中間是行程層。** 環境起來了，但你的程式還沒準備好做事：相依要被載入並連結、設定檔要讀進來解析、連線池要對著資料庫和下游一條一條握手、in-process 的快取要被預載。這一層是你自己的程式碼直接造成的，也因此是你最有控制權的一層。

**最裡面是執行體層，也是最違反直覺的一層。** 就算環境和行程都備好了，你的程式**第一次跑**和**跑了一萬次之後**的速度，可能差好幾倍——因為現代的高階執行體都靠**即時編譯（JIT）加上分層**來達到峰值效能，而分層需要時間去爬。這一層值得單獨拆開看，因為它是「程式碼一樣就跑得一樣快」這個直覺最容易失準的地方——程式碼一個字沒變，速度卻能差好幾倍。

## 程式碼一樣，為什麼跑起來不一樣快

拿 JVM 的 HotSpot 來看，因為它的分層編譯把這件事攤得最清楚。

一個方法第一次被呼叫時，HotSpot **不編譯它**——它用直譯器（interpreter）一行一行地跑 bytecode，慢，但啟動零成本。同時它在後台默默數這個方法被呼叫了幾次、迴圈轉了幾圈，並蒐集 profiling 資訊。當呼叫次數越過一個門檻（預設 `Tier3CompileThreshold` 約 2000 次），HotSpot 把這個方法交給 C1 編譯器編成機器碼，但這版機器碼還帶著 profiling 的鉤子，繼續觀察。等次數再越過一個更高的門檻（預設 `Tier4CompileThreshold` 約 15000 次），它才把方法交給 C2 做重度最佳化——內聯、逃逸分析、依據觀察到的行為做的投機最佳化——產出真正快的那一版（2026-06）。

關鍵在於：**在跨過這些門檻之前，你的熱路徑跑的是慢版本。** 一個剛啟動的 JVM 服務，前幾千個請求落在直譯或 C1 上，比它穩態時慢得多。這不是「機器還沒暖」這種模糊的感覺，它有確切的機制和確切的門檻在那裡。

V8（Node.js 底下的引擎）是同一個故事的另一種口音。它的程式碼先被編成 Ignition 直譯器跑的 bytecode；同時 Ignition 一邊跑一邊記錄回饋——property 怎麼查的、函式怎麼呼叫的、物件長什麼形狀。隨著一段程式碼變熱，它會被往上抬：先到 Sparkplug 這個 baseline 編譯器（比直譯快上三五成），再到 Maglev 這個快速的中階最佳化編譯器（2023 年的 Chrome 117 才加進來的一層），最後才到 TurboFan 這個追求峰值的頂層最佳化器。同樣地，一段函式要被執行夠多次、累積夠多回饋，才會爬到最快的那一層（2026-06）。冷的 Node 行程，前幾批請求都還在底層待著。

這一層的反直覺之處在於：**它沒有任何「環境」的成分。** 就算你把環境保溫到永遠不冷、連線池建到滿、相依全載好，只要這是一個**剛 fork 出來、還沒服務過真實流量**的行程，它的 JIT 仍然是冷的。你能消滅前兩層，但這一層必須靠「真的去跑」才能暖——這就是為什麼後面會講的「主動暖機請求」不能只打健康檢查端點：health check 不會觸發你的業務熱路徑，JIT 就暖不到該暖的地方。

## 一個函式環境的一生

要看冷啟動最赤裸的樣子，serverless 是最好的標本，因為它把「環境的生與死」變成了一個你看得見、甚至要付費的東西。以 AWS Lambda 為例，它的執行環境（execution environment）生命週期被切成幾個明確的相位，官方把它畫成 Init、Invoke、Shutdown 三段（2026-06）。

冷啟動的全部重量都壓在 **Init** 這一段。當一個請求來了、而手邊沒有暖的環境可用時，平台得從頭備一個：下載你的程式碼、把環境起來，然後跑你的 Init。這個 Init 相位內部又做三件事——啟動所有 extension、bootstrap runtime、跑你寫在 handler 外面的那段靜態初始化碼（`Function init`）。這三件做完，環境才算就緒，才會進 Invoke 去真正跑你的 handler。

這裡藏著一個非顯然的硬約束：**Init 相位有 10 秒的上限。** 三件事若沒在 10 秒內全部完成，平台不會無限等——它會中止這次 Init，並在第一次真正 invoke 的時候改用你設定的 function timeout 重跑一次初始化（2026-06）。這意味著如果你把一大堆同步 I/O、一堆相依載入、一堆暖機塞進 Init，你不只是讓冷啟動變慢、變貴（自 2025-08 起，按需函式的 Init 時間也計入 billed duration、按執行時間收費，不再免費），你還在賭一個會在某天靜默地咬你一口的計時器：某次相依拉得慢一點、某次下游握手卡一下，Init 就可能撞線。（用 provisioned concurrency、SnapStart 或 Managed Instances 時這個 10 秒上限不適用，初始化可以跑更久——這本身就透露了「預先初始化」和「按需冷啟」在平台眼裡是兩種不同的東西。）

Invoke 跑完，環境不會立刻死。平台會把它**凍結（freeze）**起來留一段時間，賭等下還有同一個函式的請求進來——真來了就**解凍（thaw）**重用，這就是暖啟動（warm start）。這個凍結重用的機制，正是冷啟動之所以「通常只發生在不到 1% 的請求」的原因（2026-06）：絕大多數請求落在已經熱好、被重用的環境上，只有「無暖環境可用」那一瞬間——擴容、長時間沒流量後的第一發、定期回收後的新環境——才會付冷啟動。

而這個重用機制還有一個極其實用的副作用，它直接決定了你該怎麼寫初始化碼：**寫在 handler 外面（global scope）的物件，在環境被重用時會留著、不會重跑。** 你在 handler 外面建的那個 DB client、那個 SDK client、那條連線，第一次冷啟動時付一次建立成本，之後同一個環境的每一次 invoke 都白拿。這就是為什麼幾乎每一份 Lambda 的最佳實踐都在喊同一句話：**把昂貴的、可重用的初始化放到 handler 外面去。** 放對地方，你只在冷啟動付一次；放錯地方（塞進 handler 裡），你把一次性成本變成了**每次都付**——這比冷啟動本身糟糕得多，因為它連暖請求都拖累。

反過來，handler 外面的東西在環境被重用時殘留，也是一把雙面刃：任何「只屬於這一次請求」的狀態都不該放到 global——它會洩漏到下一個落在同一環境的請求上。一個用 global 變數暫存「當前使用者」的函式，會在重用時把上一個使用者的殘影帶給下一個人。冷啟動的攤平省下了重複初始化，但它換來的是「環境有記憶」這個你必須隨時記在心裡的前提。

## 量一筆帳：保溫到底買到了什麼

把上面的機制收束成一個能手算的決定。假設一個函式：冷啟動的 init 要 800 毫秒，熱呼叫只要 40 毫秒，平均 QPS 是 4，但這個服務每天部署 3 次，而且平台每隔一段時間就回收一次閒置環境。問題是：要不要花錢保溫？

先算不保溫時的傷害。每一個新被創造出來的環境，它的第一個請求要付的不是 40 毫秒，而是 800 加上那一次的處理，約 840 毫秒——比熱請求多付了 800 毫秒。這些冷請求散落在哪裡？部署後所有舊環境作廢、新環境一個個冷起；回收後重新拉起；尖峰擴容時一批新環境同時冷起。它們的**絕對數量不多**——也許全天請求裡只有不到 1% 是冷的。但這就是陷阱所在：**p99 量的正是尾巴。** 只要冷啟動的比例超過 1%，p99 這個百分位就會直接落進冷請求的區間，於是你的 p99 從 40 毫秒被拉到逼近 800 毫秒——儘管 99% 的請求其實又快又好。一個健康的中位數，配一條被冷啟動主導的長尾。

現在把保溫打開：預留 2 個常駐環境，讓它們永遠不被回收、永遠是熱的。算一下這夠不夠：每個環境一次只服務一個請求，熱請求佔用它 40 毫秒，平均 4 QPS 之下，同一瞬間正在被處理的請求數平均是 4 × 0.04 = 0.16 個（offered load）。要同時需要超過 2 個環境（也就是逼出一個冷環境），得碰上並發瞬間衝到 3 個以上——以這個負載來算，落在大約 0.1% 以下的尾端機率裡。於是常態流量幾乎全落在這兩個熱環境上，冷啟動只在「突發尖峰超過保溫量」時才零星出現，比例壓到 0.1% 以下。p99 從接近 800 毫秒掉回 60 毫秒左右。

代價呢？這 2 個環境，**無論有沒有流量，你都在付費**。這就是整個決定的本質，它從來不是「要不要消滅冷啟動」這種技術問題，而是一筆赤裸的交換：**「兩個環境的常駐月費」對「p99 從 800 降到 60」，值不值。** 對一個有硬 p99 SLO 的同步請求路徑，這通常值；對一個沒人會盯著單次延遲的離線批次 worker，這就是純燒錢。冷啟動的存在不是要你消滅它，而是要你**先量、再決定**——量出「冷啟動佔請求的比例」和「冷熱延遲差」這兩個數字，兩個都小就別碰。

順帶把一個常見的誤會擋掉：保溫不等於「永遠不冷」。保溫量是固定的兩個、十個、一百個環境；只要尖峰的並發超過這個數，超出的那部分流量**還是**落在現生的冷環境上。保溫降低的是冷啟動的**頻率**，不是它的**可能性**。把保溫當成「買了就絕不冷」，會在某次超出保溫量的大尖峰時，被那條你以為已經砍掉的尾巴反咬。

## 消滅冷啟動的幾條路，與它們各自欠的債

知道了成本由哪三層組成，對策其實就是「針對某一層、用某種代價把它的成本搬走或抹平」。值得把這幾條路擺在一起看，因為它們各自買到的東西不同、欠的債也不同。

最直接的是**保溫**——讓環境根本不被回收。serverless 用 provisioned 或預留的環境，長壽服務用「最小副本數不縮到零」。它買的是「環境層加行程層」的冷成本直接消失、p99 平穩；它欠的是「你為沒在用的容量付錢」，而且如上所述，尖峰超過保溫量的部分仍會冷。

第二條是**預初始化快照**。它的洞見很漂亮：既然 init 是純粹的、可重現的計算，為什麼每個新環境都要從頭重跑一遍？乾脆在啟動後、服務前，把 init 跑完，然後把整個記憶體和磁碟狀態存成一份快照；之後每個新環境**從快照恢復**，而不是重跑 init。AWS Lambda 的 SnapStart 正是這條路——它在你發佈函式版本時就跑完 Init、存下加密快照，之後新環境從快照解凍，能砍掉很大一塊 init 與 JIT 暖機的延遲。截至 2026-06，SnapStart 支援 **Java 11+、Python 3.12+、.NET 8+**，但**不支援 Node.js、Ruby、OS-only runtime 與 container image**（Python 與 .NET 是 2024-11 才 GA 的）。Lambda 特有的這套機制細節屬於它自己的章（見〈Lambda：冷啟動、SnapStart、payload 與併發〉），這裡只取它的通用教訓。

而這個通用教訓裡藏著快照這條路最隱蔽的債：**從同一份快照恢復出來的多個環境，會共用快照拍下那一刻的所有「唯一性狀態」。** 快照裡那顆隨機數種子、那個快取下來的當前時間、那條開到一半的連線——它們在被拍下時是某個特定值，恢復出來的每一個環境都帶著**同一個**值。如果你用啟動時的隨機種子去生 ID，那麼從同一快照恢復的一群環境會生出**重複的 ID**；如果你在 init 期開了一條 DB 連線，快照裡那條連線恢復後可能早已被對端關掉、是一條死連線。所以快照這條路要求你在「恢復鉤子」裡把這些唯一性狀態**重新生成**——重新搖種子、重建連線、重發暫存 ID。攤平省下了重複的 init，但它複製的不只是你想複製的初始化結果，還有你不想複製的那些「本該每次不同」的東西。

第三條是**主動暖機請求加 readiness gate**。部署一個新實例後，先別把它放進負載均衡的輪轉，而是對它打一批合成請求，把熱路徑、連線、JIT 全跑熱，再用 readiness probe 告訴負載均衡「這台現在可以收流量了」（probe 的語意見〈health check：liveness 與 readiness〉）。它買的是「第一個真實請求」的冷峰被你自己用假請求吸收掉；它欠的是這些合成請求**必須打到真實的業務熱路徑**——前面講過，JIT 是靠真的執行去暖的，所以暖機請求若只打 `/health` 之類的端點，根本沒碰到你的業務碼、沒觸發真正的 DB 查詢、沒讓熱方法越過編譯門檻，等於沒暖。暖機沒蓋到的程式碼路徑，第一次被真實流量走到時仍然是冷的。

第四條更像衛生習慣而非技術：**把惰性初始化收斂回啟動期。** 很多冷成本是被「第一次用到才建」的惰性初始化偷偷推遲的——它沒有消失，只是被轉嫁給了某個倒楣的真實請求，而且是隨機的某一個。連線池就是最典型的例子：如果你讓池子等第一個查詢來才現連，那第一個查詢就要等握手；如果你在啟動時就把池子建到 min connections 那麼滿，這筆成本就乖乖待在 init 階段、不會跳出來咬某個使用者（連線池本身的機制見〈連線池：為什麼不每次都重開連線〉）。原則很簡單：**能在 init 期做完的，就別留到請求路徑上做。**

## AOT 不是萬靈丹：把暖機移到 build 時的取捨

最後一條路最徹底，也最值得單獨講，因為它最容易被當成「冷啟動的終極答案」而誤用。那就是 **AOT（ahead-of-time）編譯**——既然 JIT 暖機要靠執行時一次次爬門檻，那乾脆在 build 階段就把程式碼直接編成原生機器碼，跳過整個 runtime 暖機。GraalVM native image 是這條路在 JVM 世界的代表。

它的效果是戲劇性的。一個冷啟動要五秒多的 Spring Boot 服務，編成 native image 後啟動可以掉到幾百毫秒等級，記憶體佔用也大幅下降（2026-06）。對「短壽、高擴容、要極速啟動」的場景——尖峰要在幾秒內拉起幾十個新實例、每個實例可能只活幾分鐘——這幾乎是換了一個物種。

但它欠的債，恰恰藏在它的優點背面，而且很多人不知道它存在：**AOT 放棄了 JIT 在長跑時的峰值最佳化。** JIT 慢，是因為它要先觀察你的程式**實際上**怎麼跑——哪條分支幾乎總是成立、哪個型別幾乎總是出現、哪個呼叫值得內聯——然後依據這些**運行時觀察**做投機最佳化。AOT 在 build 時編譯，它**看不到**運行時的行為，只能保守地編。結果就是：native image 啟動快得驚人，但它的**穩態吞吐**通常比一個暖好的 JIT 低一截——業界的觀察大致落在約 10–25% 之間，取決於工作負載；用上 PGO（後述）能把這個差距縮小，但通常仍補不平（2026-06）。

於是出現一個漂亮的對稱：**JIT 用「慢啟動」換「高穩態」，AOT 用「快啟動」換「低穩態」。** 哪個划算，完全取決於你的實例活多久。一個會跑好幾個小時、處理上億請求的長壽服務，JIT 那幾千次的暖機成本被攤到幾乎為零，而它換來的峰值吞吐會贏 native image 一整天——對它用 AOT 是省了根本不痛的啟動、卻賠上了一直在痛的吞吐。反過來，一個活幾分鐘就被回收的短命實例，根本等不到 JIT 暖完就死了，它一輩子都在付暖機稅卻從沒享受到峰值——對它，AOT 把暖機成本前移到 build 時、是純賺。

（這條路另外還有它自己的工程麻煩：native image 的閉世界假設讓反射、動態代理、執行時載入類別這些東西需要額外設定才能用，build 也更慢更複雜。業界也有像 PGO 這類手段，先在 JVM 上跑一輪蒐集 profile、再餵回 build 去縮小穩態吞吐的差距——但那是在用更多的 build 期工夫去贖回一部分被放棄的最佳化，本質上仍是同一個取捨的微調。）

把這條路講透的意義，不在於記住「GraalVM 快」，而在於看穿一個更普遍的形狀：**冷啟動的每一種解法，幾乎都是在「啟動成本」和「穩態成本」之間搬東西。** 你沒辦法兩邊都不付——你只能選擇在哪裡付、用什麼幣別付。

## 為什麼是這個形狀

退一步看，冷啟動之所以無法被「修掉」，是因為它根本不是一個缺陷，而是一個**取捨投下的影子**。

現代執行體之所以快，靠的是把重複的工作做一次、存起來、之後白拿：相依載一次、連線建一次、JIT 觀察並編譯一次、快取填一次。這套「攤平」哲學贏在「之後的每一次」——但「做一次」這個動作必須發生在某個第一次，而那個第一次，就成了冷啟動。你能做的不是讓它消失，而是**決定誰來付、付在哪裡、付多少**：保溫是讓平台替你一直墊著（拿錢換）；快照是把它做一次存起來給後面所有環境用（拿複製的唯一性風險換）；暖機是讓假請求替真請求先付（拿暖對地方的細心換）；AOT 是把它整個移到 build 時（拿穩態吞吐換）。

也正因為它是狀態而非 bug，它的受害分布是有規律、可預測的：高擴容比、低 QPS、突發尖峰、頻繁部署的系統受害最深，因為它們一直在製造「新環境的第一次」；而穩定高流量、行程長壽的系統幾乎感覺不到它，因為它們的環境幾乎永遠是熱的、攤平的好處被吃滿。這條規律本身就是你的決策框架：在動手保溫、上快照、寫暖機之前，先問這個系統是不是真的常在製造「第一次」——如果不是，最省力的冷啟動對策，是承認它根本不值得對付。

下次你在儀表板上看到那條只在擴容瞬間冒出來、幾秒後又消失的醜尾巴，你會知道那不是有什麼壞了。那是一批剛被生出來的環境，正在替後面所有請求，把整個世界熱起來。
