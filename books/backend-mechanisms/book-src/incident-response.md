# 事故應變與無咎事後檢討

凌晨三點十七分，有人的手機開始響。錯誤率在三分鐘內從 0.2% 衝到 41%，付款 API 大面積回 500，告警把值班工程師從睡夢裡撈出來。他揉著眼睛連上 VPN，盯著一片飄紅的儀表板——資料庫連線數頂滿、上游服務在 timeout、Slack 上同時有三個人開始問「是不是我剛上的那版？」。

接下來的四十分鐘，會決定這件事是「一次被妥善處理、事後讓系統變強的事故」，還是「一場越救越亂、修完還會再犯的災難」。差別不在那位值班工程師有多聰明，而在他背後有沒有一套**機制**：誰先知道、誰來指揮、第一步看哪份文件、怎麼決定「現在該止血還是該查清楚」，以及——最關鍵的——事後怎麼把這四十分鐘的混亂，榨成下次不會再踩的學習。

這一章講的就是這套機制。它表面上是「文化」「流程」這些聽起來軟綿綿的詞，但底下其實有非常硬的工程邏輯：在一個複雜系統裡，**故障是常態而非例外**，所以你需要的不是「保證不出事」（做不到），而是「出事時的應變是有結構的、可重複的、而且每次都讓系統更難壞」。把這件事做對，可靠性就從「靠某幾個英雄半夜救火」變成一種**組織能力**；做錯，你會年復一年地在同一個坑裡跌倒，每次都換一個新人來摔。

## 為什麼「找出是誰的錯」是條死路

先把最直覺、也最常見的天真做法講清楚，因為它錯得很有說服力。

事故發生了，老闆問：「為什麼會這樣？」最自然的答案是順著時間軸往回追，追到那個「按下去就爆了」的動作，然後說：「喔，是小王半夜跑了一個沒測過的 migration。」根因找到了，責任歸屬了，給小王一個教訓，叫大家「以後小心一點」，結案。

這套邏輯的吸引力在於它**乾淨**——有一個明確的人、一個明確的動作、一個明確的教訓。但它幾乎在每一個環節都是錯的，而且錯的方式很隱蔽。

問題出在它問錯了問題。真正該問的不是「小王為什麼這麼粗心」，而是：**為什麼這個系統，允許一個凌晨三點、疲憊的、手上資訊不完整的人，用一個指令就造成全面中斷，而且過程中沒有任何一道護欄攔住他？** 為什麼那個 migration 可以不經 review 就跑進生產？為什麼沒有一個 dry-run 機制？為什麼權限大到一個人就能做這件事？為什麼監控沒在它真正失控前就示警？小王按下 enter 那一刻，是一長串系統缺陷的**最後一環**，不是根因。把他當根因，等於修掉了鏈條末端那一個可見的人，卻把上游那一整排隱形的地雷原封不動留在原地——下次換另一個人、在另一個疲憊的深夜，照樣踩爆。

這個視角不是工程界發明的，它借自安全科學裡一個叫 **Just Culture（公正文化）**的概念，源頭可追到 Sidney Dekker 對航空、醫療這類高風險領域事故的研究。Dekker 的核心觀察是：每一個事故都有兩個版本的故事。**「第一個故事」**是事後諸葛的版本——你已經知道結局了，於是回頭看，當事人的每一步都顯得愚蠢可避免（「他怎麼會沒注意到那個警告？」）。**「第二個故事」**則是去重建當事人**在當下**握有什麼資訊、面對什麼壓力、為什麼那個錯誤決定在當時看起來是合理的。事故調查唯一有價值的，是第二個故事——因為它揭露的是系統如何把「合理的決定」引向「災難的結果」，而這才是下次能修的東西。

工程界把這套思想落地得最有影響力的版本，是後來擔任 Etsy CTO 的 John Allspaw，於 2012 年發表的〈Blameless PostMortems and a Just Culture〉一文（2026-06）。它至今仍是 SRE 領域被引用最多的文章之一，原因正是它把一個抽象的安全科學概念，翻譯成了工程團隊能照做的具體實踐。

## 無咎不是「沒人負責」，而是一筆精算過的交易

「無咎」（blameless）這個詞最容易被誤讀。它聽起來像「出了事大家互相安慰一下、沒人需要負責」，於是務實的工程主管會本能地抗拒：那豈不是縱容犯錯？

無咎的真正定義要精確得多，而且它其實是一筆**冷靜的、利益導向的交易**，跟仁慈無關。

它保證的是這麼一件事：**誠實揭露細節的人，不會因為這份誠實而受罰。** 注意它保證的對象——不是「犯錯沒有後果」，而是「說真話沒有懲罰」。這兩者天差地別。

為什麼要做這個保證？因為它換來的是一樣對組織極其昂貴、卻又無法強買的東西：**資訊的完整性。** 想清楚這個因果鏈就懂了。事故的真實機制，幾乎總是只有當事人最清楚——他知道自己當時看到什麼、心裡怎麼想、為什麼跳過了那一步檢查。這些是事故調查最有價值的原料。但這些原料有一個致命的特性：**只能自願交出，無法強取。** 一旦工程師預期自己會因為說出真相而被究責，理性的反應就是隱藏、淡化、把「我當時根本沒注意到監控在叫」說成「監控系統的告警閾值設得不夠靈敏」——把責任從自己身上挪開。每一次這樣的修飾，組織就少拿到一塊真實的拼圖，於是它永遠看不清事故的完整機制，下次照樣踩。

所以無咎是一筆精算過的交易：**組織用「不懲罰個人」這個讓步，去交換「拿到事故的完整真相」這個它別無他法取得的資產。** Google 的 SRE 實踐裡有一句話把這個機制說得很白——「一個瀰漫究責氣氛的環境，會養出一種把事故和問題掃到地毯下的文化，反而給組織帶來更大的風險」（2026-06）。掃到地毯下的那些問題，不會消失，只會在你看不見的地方繼續累積，直到某天以更大的規模引爆。

但這裡有個關鍵的、常被忽略的邊界：**無咎不等於無責任（accountability）。** 這是整套機制最容易被做歪的地方。無咎針對的是**人**——不懲罰、不羞辱、不秋後算帳；責任針對的是**組織對改善的承諾**——每一個事後檢討都必須產出有明確負責人、有期限的行動項，組織必須真的去把那些系統缺陷修掉。如果一個團隊把「無咎」做成「大家檢討完互相拍拍肩膀就散會、什麼都沒改」，那它拿到了無咎的成本（不能罰人）卻沒拿到無咎的收益（用真相換來的改善）——這是兩頭落空，比有咎還糟。無咎是「不罰人，但要修系統」，缺了後半句它就退化成一場昂貴的團體治療。

還有一個常被忽略的前提：無咎是一個**自上而下的承諾，不是一條寫進 wiki 的規則。** 你可以在文件裡宣告「我們是無咎文化」，但只要發生過**一次**——主管在某場檢討會上，當著眾人的面，把矛頭指向某個工程師——所有人都會立刻學會：原來說真話是危險的。一次背叛，就摧毀掉之前所有人建立的誠實。所以無咎的成立條件非常苛刻：它需要高層真心背書、並且在最想罵人的那個當下忍住。這也是為什麼它是「文化」而非「流程」——流程可以靠工具強制，文化只能靠每一次選擇來證明。

## 事故進行時：把「協調」和「動手」拆開

理解了事後檢討的哲學，回到事故**正在燒**的那四十分鐘。這裡的機制核心是一個反直覺、但被無數次救火驗證過的原則：**在一場像樣的事故裡，「指揮」和「動手」必須是兩個分開的角色。**

先看不分開會怎樣。回到開場那個場景：付款 API 在噴 500。沒有結構的版本是這樣的——三、四個工程師同時連進去，每個人都有自己的猜測，A 在重啟服務、B 在改設定回滾、C 在查資料庫，**他們互相不知道對方在做什麼**。A 剛重啟完，B 的回滾把狀態又改回去；C 發現的線索沒人聽見，因為大家都在自己的終端機裡埋頭苦幹。更糟的是，那個最懂這套系統的資深工程師，一邊在敲指令救火，一邊還要回 Slack 上老闆、客服、其他團隊湧進來的十幾條「現在怎樣了？」——他的注意力被撕成碎片，救火的手停下來打字，打字的時候腦子裡的救火思路又斷了。狀態越救越亂，而且沒有人能說清楚「我們現在到底試過了什麼、還沒試什麼」。

這套混亂的解法，工程界沒有重新發明，而是直接借了一套被大規模災難驗證了半世紀的體系——**Incident Command System（ICS，事故指揮系統）**。它的起源很具體：1970 年代加州一連串毀滅性的野火，事後檢討發現，造成最大損失的往往**不是資源不足，而是協調失靈**——消防、警察、各級政府的隊伍各自為政、通訊不通、指揮重疊。於是一個跨機構的計畫（FIRESCOPE）發展出 ICS 這套標準化的指揮架構，後來成為全美緊急應變的通用體系。科技公司（PagerDuty 把它整理成一套公開的事故應變文件，影響很廣）發現，半夜的線上事故和野火現場有著驚人相似的結構問題，於是把這套體系搬了過來。

它的核心是幾個**角色**，而不是幾個人——同一個事故裡角色數量隨規模伸縮，但職責邊界是清楚的：

```
                 +------------------------+
                 |  Incident Commander    |  decides & coordinates
                 |  (does NOT type fixes) |  owns the incident
                 +-----------+------------+
                             |
          +------------------+------------------+
          |                  |                  |
   +------v------+   +-------v-------+   +------v------+
   | Operations  |   | Comms Lead    |   |   Scribe    |
   | (hands-on)  |   | (status out)  |   | (timeline)  |
   +-------------+   +---------------+   +-------------+
```

最反直覺、也最重要的角色是 **Incident Commander（IC，事故指揮）**。IC 的職責是**讓事故往解決的方向推進**——做決策、分派任務、聽取各路專家的輸入然後拍板——但他**不親手敲修復指令**。這條「不動手」的紀律是整套機制的靈魂。一個顯而易見的反對是：「那 IC 不就是個只會出一張嘴、不懂技術的協調者？」恰恰相反，這個設計揭露了一個被長期低估的事實：**在一場多人、高壓、資訊飛速更新的事故裡，「維持一張全局的、所有人共享的當前狀態圖」本身就是一份滿載的全職工作。** 誰試過了什麼、結論是什麼、現在的假設是哪個、下一步賭哪邊——這張圖如果沒有一個專人持續維護，它就會碎成每個人腦中各自的、互相矛盾的版本，而那正是混亂的源頭。IC 不需要是技術最強的人（PagerDuty 的文件甚至明說 IC 不需要深厚技術知識）；他需要的是能在資訊洪流裡保持清醒、快速決策、並且在動手的人之間維持秩序的能力。一個有趣的副作用：在事故進行時，IC 是這通電話裡**最高的決策者，無論他平時的職級是什麼**——這個臨時的權威，正是為了讓決策能快、能集中、不被職級政治拖慢。

把 IC 從「動手」裡解放出來，連帶就騰出了另外兩個角色。**Communications Lead** 專責對內外同步狀態（每隔固定時間更新一次，讓老闆和客服不必每五分鐘戳一次救火的人）；**Operations** 才是真正動手改系統的人，而且因為有 IC 在維持全局、協調誰做什麼，他們不會再互相覆蓋。事故越大、跨的團隊越多，這套角色分工的價值越高。

但這套機制本身也有它的成本與邊界，誠實地講：**對一個小事故硬套完整的 IC 體系，是純粹的儀式負擔。** 一個只影響內部工具、五分鐘就能修好的小毛病，你不需要指定指揮、不需要溝通主管、不需要記錄員——三個人在 Slack 裡喊一聲、一個人去修、修好說一聲，就夠了。所以這套機制需要一個**觸發門檻**，而那個門檻，通常就是事故分級。

## 分級：把「該驚動多少人」變成一個查表動作

事故分級（severity levels，常寫成 SEV1、SEV2、SEV3…）表面上像是給事故「貼標籤」，但它的真正功能是一個**觸發器**：分級這個動作的瞬間，就決定了「要喚醒多少人、多快、用什麼管道、要不要啟動完整的 IC 體系」。

為什麼需要這個機制？因為它解決一個兩難。如果沒有分級，每個告警都用同一種強度應變，你會落入兩個極端的其中一個：要嘛每個小毛病都驚動全公司（很快大家就對告警麻木，狼來了喊多了，真出大事時沒人理會），要嘛大事故被某個人當成小事默默處理、拖到無可挽回才升級。分級就是用一把客觀的尺，把**應變的強度**和**事故的嚴重度**對齊。

具體的層級定義各家不同，但骨架類似。借 PagerDuty 那套公開定義來看（2026-06）：**SEV1** 是系統處於危急狀態、大量使用者正在受影響、或有資料外洩這類需要驚動高層、甚至對外公告的等級；**SEV2** 是嚴重影響許多使用者使用產品能力、但還沒到 SEV1 那種「全公司動起來」的程度；**SEV3 以下**是需要處理、但不影響使用者核心使用的小問題。注意這裡的判準刻意綁在**使用者影響**上，而不是「工程師覺得這個 bug 有多難」——一個技術上很棘手但沒人受影響的問題，分級應該低；一個技術上很蠢但讓所有人付不了款的問題，分級必須高。

這裡藏著分級機制最常壞掉的地方，值得講透。**分級的標準必須客觀、可量化，否則它會被人性扭曲。** 想像一下：如果分級靠主觀判斷，那麼一個剛上線就出包、心裡發虛的工程師，會傾向把自己捅的事故**往低報**（「應該還好吧，我先默默修」），好讓它不那麼顯眼；而一個想證明自己團隊很重要、或想叫醒某個一直不理他的上游團隊的人，會傾向**往高報**。兩個方向的扭曲都讓分級失去意義。對策是把分級綁在可量測的訊號上——錯誤率超過多少、受影響使用者數超過多少、SLA 是否已被打破——讓「這是 SEV1 還是 SEV2」盡可能變成一個**查表**而非辯論的動作。當然，沒有任何分級表能涵蓋所有情境，總會有需要人判斷的灰色地帶；但把能客觀的部分客觀化，已經能擋掉絕大多數的扭曲。

## Runbook：把專家半夜的腦子，預先寫下來

事故進行時還有一個機制值得單獨講，因為它直接攻擊一個非常具體的痛點：**為什麼每次半夜出事，都得等那個唯一懂的人爬起來？**

對「已知會發生」的那類事故——憑證過期、磁碟寫滿、主從切換、某個下游依賴掛了該怎麼降級——你其實**第二次遇到時不該重新推理一遍**。第一次解決它的人，腦中已經跑過了一遍完整的診斷與修復路徑。Runbook（也叫 playbook）的機制，就是把那條路徑**事先寫成步驟**，讓下一個半夜被叫醒的 on-call，不必在腦子最不清醒的時候重新發明它。

Runbook 的本質，是把一個專家的**隱性知識顯性化**——把「只活在某個資深工程師腦子裡」的處理流程，搬到一份所有人都讀得到的文件上。它買到的東西有兩樣：**快**（已知事故的處理時間大幅縮短）和**去英雄化**（不再依賴某個特定的人剛好醒著、剛好在線）。

但 runbook 有一個必須講清楚的、相當陰險的故障模式：**它會腐化，而過時的 runbook 比沒有 runbook 更危險。** 邏輯是這樣的——系統在持續變化，某個指令名改了、某個服務搬了家、某個步驟的前提不再成立，但寫好的 runbook 不會自己更新。於是某個半夜，一個半睡半醒的 on-call，**完全信任**地照著一份已經不成立的步驟操作——而「照著一份看起來權威、實則錯誤的指南操作」，可能比「沒有指南、被迫自己小心思考」造成更大的破壞。沒有 runbook 時人會警覺地慢下來；有一份過時 runbook 時人會放心地快下去，直接衝進坑裡。所以 runbook 不是寫完就一勞永逸的資產，它是一筆**持續的維護負債**——必須綁進系統的變更流程（改系統時連帶檢查相關 runbook），或定期實際演練校驗，否則它會在你最需要它的那個深夜背叛你。

## 事故的時間軸：MTTR 不是一個數字，是四段路

事故落幕後，幾乎每個團隊都會盯上一個指標：**MTTR（Mean Time To Resolve，平均修復時間）**——從事故開始到服務恢復，平均花多久。它直覺、好懂、老闆愛看。但如果你只盯這一個總數，你會被它誤導，因為 **MTTR 是好幾段性質完全不同的時間黏在一起的總和**，而改善的施力點藏在它的分解裡。

把事故的時間軸拆開看（用一組業界常見的細分指標來標記每一段，2026-06）：

```
故障   告警        有人開始     找到原因    服務
發生   觸發        處理         /止血       恢復
 |------|------------|------------|-----------|
 |<MTTD>|            |            |           |
 |      |<-- MTTA -->|            |           |
 |      |            |<--- 調查 + 修復 --->   |
 |<--------------------- MTTR --------------->|
```

- **MTTD（Mean Time To Detect）**：從故障真正發生，到監控系統發出告警、有人意識到出事了。這一段量的是你的**監控有多敏銳**。如果這段很長，意思是系統壞了半天你還蒙在鼓裡——往往是靠使用者來客訴你才知道，那是最難堪的偵測方式。
- **MTTA（Mean Time To Acknowledge）**：從告警響起，到有個活人真正開始處理。這段量的是你的**值班與升級機制有多順**——告警有沒有送對人、那個人有沒有在睡、升級鏈會不會卡住。
- **調查 + 修復**：有人開始查，到找出原因並讓服務恢復。這段量的是**診斷的難易**與**修復手段是否就緒**（有沒有快速回滾、有沒有 feature flag 能一鍵關掉壞功能）。

把這個分解握在手上，一個看似籠統的「我們事故修得太慢」就能被精準定位。做個具體的推演：假設某團隊統計過去一季的事故，平均 MTTR 是 **90 分鐘**，老闆嫌太慢。拆開一看：MTTD 平均 **5 分鐘**（監控很靈，幾乎一壞就知道），MTTA 平均 **3 分鐘**（值班很到位，馬上有人接手），但「調查 + 修復」吃掉了 **82 分鐘**。這個分解立刻把改善方向釘死了：問題**不在**監控、**不在**值班——往那兩個方向砸資源（買更貴的監控、加更多值班人手）是浪費。82 分鐘全花在「查不出原因」或「查到了但回滾很慢」上。於是真正該投資的是**可診斷性**（更好的 trace、更清楚的 log）和**修復槓桿**（一鍵回滾、feature flag）。反過來，如果同樣 90 分鐘裡，MTTD 占了 60 分鐘（壞了一小時才發現），那再快的回滾也救不了你——你該去修的是監控覆蓋的盲區。**同一個 90 分鐘，分解不同，藥方完全相反。** 這就是為什麼「只看 MTTR 一個總數」是不夠的：它把該分開治療的病灶熬成了一鍋看不出成分的湯。

但這裡要插一句誠實的提醒，避免從一個極端掉進另一個。把 MTTR（或它的任何分段）當成**唯一**目標，本身也是個陷阱。一味追求「修得快」，會誘導團隊養成一個壞習慣：**草草止血、跳過真正理解。** 服務恢復了，MTTR 數字漂亮了，但根因從沒查清楚——於是同一個事故下個月、下下個月反覆回來，每次都「很快修好」，每次都沒真的解決。這裡要分清兩件常被混為一談的事：**緩解（mitigate）**是讓使用者止血、讓服務先恢復；**理解（understand）**是搞清楚它為什麼壞、怎麼讓它永遠不再這樣壞。兩件都要做，而且順序通常是先緩解（止血優先，使用者在流血）、後理解（事故落幕、頭腦冷靜時才挖根因）。MTTR 只量得到前者。一個只優化 MTTR 的團隊，會變成一支非常擅長止血、卻永遠在止同一道血的隊伍。

## 為什麼「唯一根因」是個美麗的謊言

事後檢討要挖根因，最流行的工具是 **Five Whys（五個為什麼）**——對著問題連問五次「為什麼」，一層層往下挖到根。它對某些事故很好用，尤其是那種**單一主因清晰**的故障：為什麼掛了？磁碟滿了。為什麼滿了？log 沒輪替。為什麼沒輪替？設定漏了。為什麼漏了？沒有預設模板。為什麼沒模板？——挖到一個可修的系統缺陷，漂亮收工。

但 Five Whys 有一個它自己藏不住的預設，而這個預設對複雜系統往往是假的：**它假設根因是「一條鏈」，只要順著往回拉就能拉到那唯一的源頭。** 真實的大事故幾乎從不長這樣。一場真正嚴重的事故，典型的結構是**多個本來無害的小問題，在一個特別不巧的時機同時湊齊**——

- 一個埋了很久、平時根本碰不到的程式缺陷（觸發條件）；
- 加上一道本該攔住它、卻剛好缺席或失效的護欄（缺失的防禦）；
- 加上監控恰好沒覆蓋到這個中間狀態，於是它惡化了一陣子才被發現（延遲的偵測）；
- 也許再加上當天某個依賴剛好在做維護、某個快取剛好冷掉——

這幾件事**任何一件單獨發生都不會出事**，是它們的**合流**造成了災難。對這種事故硬套 Five Whys、執意挖出「那唯一的根因」，會逼你在好幾個同等重要的成因裡，挑一個當代罪羔羊，然後修掉它、宣布結案——而留下的另外三、四個成因，全數原封不動。你修掉了一個誘因，卻留下了一整片地雷。

所以成熟的事後檢討會在這裡換工具：對多因耦合的事故，用**因果圖**（畫出所有貢獻因素及其關係）取代線性的 Five Whys，目標從「找出那一個根因」變成「找出所有可以被加固的環節」。判準很簡單：如果這個事故的故事可以用一句「因為 X，所以 Y，所以 Z」說完，Five Whys 夠用；如果你發現自己在「主因到底是 A 還是 B」上爭論不休，那答案多半是**都是**，你需要的是一張圖，不是一條鏈。

## 行動項：檢討會最常死在這裡

把事後檢討的整套流程在腦中重放一遍：事故落幕→產出時間軸（什麼時間發生了什麼，這也是為什麼進行時要有人當 Scribe 記錄）→盤點影響範圍→做根因分析→然後，產出**行動項（action items）**。

最後這一步，是整個檢討的**唯一產出物**，也是它最常死掉的地方。

事後檢討最常見的失敗，不是時間軸畫得不夠細、不是根因挖得不夠深——而是**行動項寫完之後，就再也沒人管了。** 檢討會開得情真意切，大家都點頭，文件寫得漂漂亮亮，列了八條「我們應該……」，然後文件被歸檔，所有人回去做手上的功能開發，那八條行動項靜靜地躺在 wiki 裡腐爛。三個月後同類事故再次發生，有人翻出舊文件，發現當初的行動項一條都沒做。這時候，整場無咎檢討就退化成了一場**昂貴的儀式**——花了真相、花了會議時間，什麼都沒換到。

要讓行動項真正閉環，機制上有幾個硬要求。第一，每一條行動項都必須有**明確的負責人和期限**——「我們應該加強監控」是一句空話，沒有人會去做一句沒有主詞的話；「@某人，在下個 sprint 內為這個資料庫的連線數加上告警」才是一個能被追蹤、能被驗收的承諾（Google SRE 的實踐把這點講得很死：行動項要 specific、要 owned、要有期限、要被追蹤，2026-06）。第二，這些行動項必須被**接回團隊正常的工作排程**——進到 backlog、進到 sprint，跟其他功能需求一起排優先序、一起被 review，而不是活在一份和日常工作隔離的「事故文件」裡。隔離的清單會被遺忘，混進日常工作流的任務才會被做掉。把行動項分成兩類也很有用：**緩解性的**（修掉這個具體的缺口）和**預防性的**（解決這一**類**問題，讓整個類別的事故都不再發生）——後者才是事後檢討真正的高價值產出，因為它修的不是這一次的事故，是未來一整批還沒發生的事故。

還有一個容易被忽略、卻是「免費學習機會」的東西：**near miss（虛驚一場）。** 很多團隊只對造成實際損失的大事故做檢討，但那些「差一點就出大事、但僥倖沒事」的小插曲——某個告警響了但剛好有人在、某個資源差幾個百分點就爆但流量剛好回落——其實是系統在用**最低的成本**告訴你它哪裡脆弱。等到它真的釀成 SEV1 才學，學費就貴多了。對 near miss 的重視程度，某種意義上是一個團隊事故文化成熟度的試金石：成熟的團隊會主動去檢討那些「沒出事但本來會」的時刻，把脆弱點在便宜的時候就補掉。

## 強度要配規模：別給內部小工具配一支消防隊

把上面這一整套機制——分級、IC 體系、runbook、強制檢討、行動項追蹤——擺在一起，很容易產生一個誤會：以為這是每個團隊都該全套照搬的「正確做法」。不是。這套機制的強度必須**匹配你的規模和風險**，套錯了就是過度工程。

光譜的一端：一個一人或小團隊維護的、低流量的內部工具。出事了影響幾個內部使用者，等上班再修也無妨。給它配上完整的 SEV 分級、Incident Commander、Communications Lead、強制的 blameless postmortem——這是拿消防隊的編制去處理廚房裡一支冒煙的吐司機。對這種情境，一份簡單的 on-call 聯絡方式、加上幾條最常見故障的 runbook，就完全夠了。

光譜的另一端：一個有 SLA 承諾、有付費使用者、橫跨多個團隊的服務。在這裡，**「沒有事故流程」本身就是你最大的風險**——因為下一次大故障一定會來，而到時候如果沒有預先約定好的分級、指揮、升級路徑，你就會回到本章開頭那個「三個人各自連進去越救越亂」的場景，只是賭注大得多。

那判準是什麼？有一個很實用的提問可以幫你定位：**問自己「下一次半夜大故障，誰會先知道、誰來指揮、第一步該看哪份文件？」** 如果你能清楚回答這三個問題，你的事故機制大致到位了。如果答案是「看誰運氣好剛好醒著、看誰先衝進去、然後現場隨機應變」——那就是該開始建立流程的訊號了。

## 為什麼是這個形狀

退一步看，事故應變這套機制的整個樣貌，都是從一個無法迴避的前提裡長出來的：**在一個夠複雜的系統裡，故障不是「會不會」的問題，而是「何時、以何種組合」的問題。** 你無法靠「更小心」「更努力」來消除事故——複雜系統的失敗是湧現的、是多因合流的，不是某個人不夠認真造成的。

一旦接受了這個前提，這套機制的每一個形狀就都變得必然。正因為故障必然發生，所以你需要的不是「防止出事」的承諾，而是「出事時應變有結構」的能力——於是有了分級、有了 IC 體系，把混亂收編成可重複的協調。正因為人不是根因、系統才是，所以追究個人是條死路、無咎才能換到真相——於是有了 blameless postmortem，用「不罰人」去買「拿到完整機制」。正因為真實的大事故是多因耦合，所以「唯一根因」是個美麗的謊言——於是要用因果圖去盤點所有能加固的環節。正因為檢討的價值全在改善、而改善只存在於被追蹤的行動項裡——所以那份帶負責人、帶期限、被接回排程的清單，才是整套流程真正的產出，其餘都是過程。

把這一切串起來，你會發現它在做的其實是同一件事：**把可靠性，從「靠某幾個英雄在半夜的運氣與毅力」，變成一種可以被設計、被重複、被持續改善的組織能力。** 事故無法避免，但「每一次事故都讓系統變得更難壞」——這件事，是可以靠機制保證的。一個團隊處理事故的方式，最終決定的不是它會不會出事（一定會），而是它出同一種事的次數，會隨時間遞減還是原地打轉。
