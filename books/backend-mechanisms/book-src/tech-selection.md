# 技術選型：評估、PoC 與退場成本

桌上攤著三個候選。一個是團隊都沒用過、但 benchmark 漂亮、GitHub 星星很多的新訊息中介；一個是大廠的託管服務，文件齊全、demo 跑起來十分鐘就通；一個是團隊已經用熟的老東西，無聊、夠用、沒有任何讓人興奮的點。你要替接下來三年的系統挑一個。

問題出在你手上的資訊。你能看到的——demo、官方文件、廠商的 benchmark、別人寫的「我們如何用 X 撐起千萬流量」的部落格——全都是**順境資料**。它們無一例外地展示這個技術在最舒服的條件下、被最懂它的人操作、跑著最配合它的工作負載時的樣子。而真正決定這次選型成敗的，是你**看不到**的那一面：它半夜出事時好不好 debug、文件沒寫的那些邊角行為、你撞到怪 bug 時社群會不會理你、以及最容易被整桌人忽略的那一項——**選錯了要把它換掉有多痛**。

技術選型的難，不在於「哪個技術更好」。多數時候候選們在 happy path 上都夠好，不然不會進到桌上。難在於：**你要在資訊嚴重不對稱的情況下，下一個之後很難反悔的注。** 這一章談的就是怎麼在下注前，把「不確定」儘量換成「資訊」，以及怎麼估算那個幾乎所有人都低估的數字——退場成本。

## 為什麼這件事這麼難：你比的是看不見的那一半

把「選 X 還是 Y」這個問題拆開，會發現它由兩堆變數組成，而這兩堆的可見度天差地別。

第一堆是**順境變數**：功能清單對不對得上需求、benchmark 數字多漂亮、API 設計順不順手、文件全不全。這些可見、可量、可比，於是評估很容易**只**圍著它們轉——做張表，把功能打勾，把 benchmark 抄進去，分數高的勝出。

但決定一個技術長期是賺是賠的，幾乎都是第二堆**逆境變數**：它在高併發下會不會雪崩、某個節點掛掉時行為是否如文件所說、給你的錯誤訊息能不能定位問題、版本升級會不會破壞相容、出冷門問題時 Stack Overflow 和 issue tracker 有沒有答案。這堆變數有個共同點——**demo 和 benchmark 永遠不會展示它們**。廠商的 benchmark 是在對它最有利的硬體、最配合的負載、調到最優的參數下跑出來的；demo 走的永遠是設計好的 happy path。你拿這些去比，等於只比了冰山水面上的那一角。

Dan McKinley 在〈Choose Boring Technology〉裡點破了這件事的核心：選一個無聊技術的好處，不只是「能力被充分理解」，更關鍵的是「**它的故障模式被充分理解（their failure modes are well understood）**」。一個被千萬人用了十年的資料庫，它會怎麼壞、壞了長什麼樣、怎麼救，都已經有人踩過、寫成文章、變成你搜得到的答案。一個剛紅起來的新技術，能力或許更強、語法或許更甜，但它的故障模式對全世界——包括它的作者——都還是一團「未知的未知（unknown unknowns）」。你選它，就是替自己簽下一批還沒被任何人發現、只能由你在某個生產事故裡第一個踩到的雷。

這就帶出第三個、也是最隱蔽的維度：**退場成本（cost of exit）**。前兩堆變數至少還能在事前討論，退場成本卻常常整桌沒人提——因為在選型的興奮時刻，沒人想討論「萬一這玩意兒不行，我們要怎麼擺脫它」。但這恰恰是選型真正的風險所在。兩個功能旗鼓相當的方案，把你綁住的深度可能差十倍：一個躲在標準介面後面、隨時能換掉，另一個會把它的專有 API 滲進你程式碼的每個角落、把你的資料鎖進只有它讀得懂的格式。選錯前者，是一週的事；選錯後者，是一場為期半年、動到幾十個服務的遷移戰爭。**選型真正在比的，是「它把你綁多深」，而不只是「它現在好不好用」。**

天真的選型——做張功能對照表、挑 benchmark 贏家、被 demo 打動——之所以反覆釀成昂貴的錯誤，正是因為它把全部注意力放在可見的那一半，對逆境變數靠猜、對退場成本視而不見。要做對，得用兩件工具把這兩件事從「猜」變成「知道」。

## 評估矩陣：它的價值不在分數，在逼你攤牌

第一件工具是**評估矩陣（decision matrix）**：把候選列成欄、把評估維度列成列，每格打分、各維度加權，最後加總比高下。

```
                weight  Cand-A  Cand-B  Cand-C
fit to needs      0.30      4       5       3
ops/maintenance   0.20      3       5       4
ecosystem         0.15      5       4       2
team familiarity  0.15      2       3       5
licensing         0.10      5       3       5
cost of exit      0.10      2       2       5     <- 常被漏掉的一列
                          ----    ----    ----
weighted total            3.55    4.05    3.75
```

這張表會吐出一個數字，而那個數字幾乎是它**最不重要**的產物。把評估矩陣當成「客觀計算機」是個誤用——權重是你定的、分數是你打的，主觀性從頭滲到尾。給某一維度多 0.05 的權重、給某候選多打一分，排名就翻盤。把這張表當成替你「算」出答案的客觀依據，等於把既有的偏好洗成了看似客觀的小數點。

它真正的價值，在於**逼團隊把隱性的偏好攤到桌面上吵**。當有人堅持選那個新技術，矩陣會逼他回答：是哪個維度讓它贏的？是「契合需求」還是「我想學它」？當資深的人憑直覺反對，矩陣會逼他把直覺翻譯成某幾列的分數，讓別人能反駁。沒有這張表，選型會退化成會議室裡誰嗓門大、誰資歷深、誰對新技術更興奮——隱性偏好在暗處主導決策卻無人能挑戰。有了它，每一個偏好都被釘在某一列上，可被指認、可被討論、可被推翻。

所以那一列「cost of exit」非放不可，而且要給它真實的權重。少了它，矩陣會系統性地偏袒「現在最好用」的候選——那個 demo 最炫、功能打勾最多、卻可能把你綁得最死的。把退場成本擺進去，等於強迫整桌在興奮的時刻就直視「萬一錯了怎麼辦」。

但矩陣有個它自己解不了的致命弱點：**那些分數是哪來的？** 「ops/maintenance 打 5」「高併發下表現打 4」——這些恰恰是逆境變數，是 demo 和文件不會告訴你的。如果這些分數是抄廠商 benchmark、讀部落格、憑感覺填的，那你只是把順境資料包裝成了一張看起來很嚴謹的表。要讓這些格子裡填進**真實**的數字，你得自己去把那個技術逼到逆境裡——這就是第二件、也是更重要的工具。

## PoC 與 spike：拿你自己的逆境去打它

**PoC（proof of concept）／spike** 的精神可以一句話講完：**不要拿廠商的 happy path 來評估，拿你自己的逆境來打。**

spike 這個詞來自敏捷實務，指一段**有時限（time-boxed）的研究活動**——通常幾小時到幾天，目的不是產出正式程式碼，而是**用實驗把一個技術風險或未知，換成一個可以下決策的答案**。它的產物常常是一個用完即丟的原型加一份「它在我們的場景下到底行不行」的結論。時限是 spike 的靈魂：沒有時限，「研究一下這個技術」會膨脹成一個沒有終點的黑洞；釘死「我們花三天，回答這三個具體問題」，它才是評估工具而不是又一個拖延。

而 PoC 的價值，**全部**取決於你拿什麼去打它。這是最多人做錯的地方：他們照著官方 quickstart 跑一遍，看到 demo 通了，就在矩陣上給「能用」打高分。這等於沒打——你只是重現了廠商已經替你驗證過的 happy path，沒得到任何**新**資訊。真正有資訊量的 PoC，打的是廠商**不會**展示的逆境：

- **把它壓到極限**：用接近你尖峰負載的流量打它，看它在哪裡開始退化、退化得優雅還是雪崩。注意，要算清楚你的尖峰量級該打多大——這個量級估算本身是另一門功課（見〈信封背面：量級估算與容量直覺〉），但 PoC 要打的就是那個估出來的逆境負載，不是廠商 demo 那種玩具流量。
- **模擬故障**：殺掉一個節點、拔掉網路、灌進畸形輸入，看它的行為是否真如文件所說。文件寫「自動 failover」和你親眼看到它 failover 時丟了幾筆資料、卡了幾秒，是兩回事。
- **看它出事時的可觀測性**：故意製造一個錯誤，然後問自己——它給的錯誤訊息能讓我定位問題嗎？還是只丟一句 `internal error`？半夜出事時，這個 debug 體驗就是你的生死線。

只跑 happy path 的 PoC，和讀一篇部落格的資訊量一樣多——約等於零。**PoC 的全部意義，在於主動去碰那些 demo 刻意繞開的暗角。**

對退場成本特別高、或牽連特別廣的候選，光做 PoC 還不夠，下一步是**試生產（trial run）**：在一個低風險的真實服務上先用起來，累積幾週真正的營運經驗——半夜的告警、真實流量的長尾、團隊操作它的順手程度——再決定要不要全面押注。這跟金絲雀發布是同一種精神：先放一小部分真實流量去蹚雷，限制萬一錯了的爆炸半徑（金絲雀的機制本身見〈blue-green、canary 與 rolling〉）。關鍵是試生產**必須**配一條乾淨的回退路；少了它，「試用」會在不知不覺間變成「已經全押、想退也退不掉」——那就違背了試生產的初衷。

## 退場成本怎麼估：一個手算的例子

退場成本是這一章最該講透的東西，因為它最常被低估，而且**可以**事前估算到一個有用的數量級——別把它當成模糊的「換起來應該很麻煩」，把它拆成具體的工。

設想一個團隊在替核心服務選資料儲存，兩個候選功能上旗鼓相當：

- **候選 A**：一個用標準 SQL、走標準 wire protocol 的關聯式資料庫。資料能用標準工具 dump 出來，查詢是標準 SQL，應用層透過一個薄薄的 repository 介面存取它。
- **候選 B**：一個專有的託管資料庫，有自己的查詢語言、自己的資料模型、自己的 SDK，效能在 demo 裡確實更亮眼。

兩年後假設要換掉它，把退場成本拆成四塊工來估：

1. **資料搬遷**——把資料從舊的搬到新的。A 走標準 dump/restore，假設 2 人週。B 的資料鎖在專有格式裡，得寫一支自訂的匯出轉換程式、處理它特有的型別與索引語意，假設 8 人週。
2. **應用層改寫**——把所有碰到這個儲存的程式碼改掉。A 的存取都收斂在一個 repository 介面後面（一個抽象層把第三方藏起來，使得「換掉它」只動一處——這正是退場成本的關鍵變數），假設改一處、3 人週。B 的專有查詢語言和 SDK 呼叫**散落**在三十幾個服務裡，每一處都要重寫、重測，假設 20 人週。
3. **雙寫過渡**——遷移期間新舊並存、雙寫比對、漸進式切換，兩者都需要，各假設 5 人週。
4. **團隊重新學習＋風險緩衝**——A 換到另一個標準 SQL 庫，學習成本低，假設 2 人週；B 累積了兩年的專有 know-how 要作廢、新方案要重新摸熟，假設 6 人週。

把帳算出來：

```
                       Cand-A     Cand-B
data migration          2 pw       8 pw
app rewrite             3 pw      20 pw
dual-write transition   5 pw       5 pw
relearn + buffer        2 pw       6 pw
                       ------     ------
cost of exit           12 pw      39 pw
```

A 的退場成本約 12 人週，B 約 39 人週——**差超過三倍**。注意這個差距的來源：絕大部分不在「資料量大小」這種規模因素，而在兩個**選型當下就埋下**的結構決定——資料是不是鎖在專有格式、存取是不是被一個抽象層收斂在一處。這兩件事都是你在選型那一刻、用一點前置紀律（堅持標準介面、堅持把第三方藏在 repository 後面）就能大幅左右的。**退場成本不是技術的固有屬性，有很大一部分是你怎麼用它決定的。**

這個算術也直接告訴你選型該慎重到什麼程度。退場成本 12 人週的決定，偏向「雙向門」——錯了，認賠 12 人週退回來，不算世界末日，可以快一點決定、小團隊就能拍板。退場成本 39 人週、還牽動三十幾個服務的決定，是扎扎實實的「單向門」——錯了，是一季的遷移戰爭加上期間的風險。決策該慎重到什麼程度，不取決於這個技術多重要，而取決於**反悔的代價**有多高（可逆與不可逆決策的完整框架見〈兩扇門：可逆與不可逆決策〉）。一分鐘把退場成本拆成這四塊工算一遍，就能在白板上分辨出眼前是雙向門還是單向門，而不必等兩年後真要換時才發現自己當初走進的是哪一種。

值得補一句：估出 39 人週這個數字，常常會反過來改變選型本身。如果 B 的效能優勢只在某些非核心場景才用得上，卻要扛三倍的退場風險，那這筆帳很可能根本不划算——把退場成本算清楚，本身就是在替「要不要為了一點效能甜頭把自己綁死」這個問題提供彈藥。

## 它會怎麼騙你

技術選型有幾個反覆讓聰明團隊翻車的陷阱，每一個都值得認出來。

**被 benchmark 和 demo 騙。** 這是最根本的一個，前面反覆在講：廠商的 benchmark 跑在對它最有利的條件下，demo 永遠是 happy path。把這些順境資料當成評估依據，等於讓對手出考卷。唯一能信的，是你自己拿真實場景、真實逆境打出來的數字。看到一個漂亮的 benchmark，正確的反應不是「哇好快」，而是「它在什麼硬體、什麼負載、什麼參數下跑的？我的場景對得上嗎？」

**只比功能，不比退場成本。** 兩個方案功能對照表打勾數一樣，看起來等價，退場成本卻可能差三倍——前面那個 12 vs 39 人週就是活生生的例子。只比「現在能做什麼」而不問「綁多深」，是把選型最大的長期風險直接漏掉。功能是當下的事，退場成本是未來兩年的事，後者的權重往往該更高。

**履歷驅動開發（resume-driven development）。** 這個反模式有正式的學術刻畫——2021 年 ICSE-SEIS 的一篇論文把它定義為招募雙方對「熱門／被炒作的技術」的過度強調，諷刺的是雙方私下都覺得這些技術的實際經驗價值沒那麼高。落到選型上，它的長相是：團隊先看上一個很酷的新技術，再回頭替它找一個能用上的理由——而不是從需求出發去找最適配的工具。新潮和適配是兩回事；當「我想學它／它能寫進履歷」偷偷變成選型的真實動機，矩陣裡那些分數就全是替既定結論服務的裝飾。對抗它的方法，正是前面那張逼人攤牌的矩陣——它會把「fit to needs 其實只有 3 分」這件事赤裸裸地擺出來，讓「但我就是想用它」無處遁形。

**低估「無聊技術」的價值。** 成熟、無聊、團隊用熟的技術有一種不性感卻極其值錢的特質：它的故障模式已經被全世界踩過、寫成文章、變成你半夜搜得到的答案。McKinley 那句「故障模式被充分理解」不是在說無聊技術更強，而是在說它的**未知的未知更少**——你不會在某個生產事故裡，成為全世界第一個踩到某個雷的人。這種可預測性本身就是巨大的價值，常常勝過新技術那點效能或語法上的甜頭。一個團隊能不能在凌晨三點救活一個系統，跟它用的技術有多潮幾乎無關，跟「這個故障有沒有人踩過、查得到解法」高度相關。

**把選型當成一次性事件。** 簽完字不是結束。技術會老化、需求會漂移、當初的假設會被現實推翻——PoC 當時驗過的逆境表現，可能在流量量級上漲三倍後完全變樣。選型真正的紀律，是**選完繼續觀察它在生產的真實表現，並且預先保留「發現選錯時撤回」的機制**。這又繞回退場成本：當初為了壓低退場成本而堅持的那些事——標準介面、抽象層隔離、資料用可搬遷的格式——不只是省未來遷移的工，更是讓「持續監看、必要時撤回」這件事在現實中真的可行的前提。一個把自己綁死、退不掉的選型，等於放棄了事後糾錯的權利。

**過度評估也是一種陷阱。** 反過來說，不是每個選型都該跑三週 PoC、開評估矩陣、拉全團隊會診。一個可逆、影響面小、團隊已有合理既有偏好的小工具——比如已經全棧用某語言、再選個同生態的小套件——為它跑完整流程就是把紀律用過了頭，那本身是另一種形式的過度工程（見〈簡單性與過度工程：YAGNI 的兩面〉）。評估的投入該與退場成本成正比：退場便宜的快選，退場昂貴的才值得慢慢來。

## 為什麼是這個形狀

退一步看，整套選型方法——評估矩陣、PoC、試生產、估退場成本——都是從同一個無法迴避的事實裡長出來的：**你必須在資訊嚴重不對稱的情況下，下一個之後很難反悔的注。**

正因為資訊不對稱、你能看到的只有順境的那一半，所以才需要 PoC——主動拿逆境去打，把「猜」換成「知道」。正因為下注前的不確定能用前置投入換成資訊，所以才有那條原則：**選型該在降低不確定性後再下注，而不是憑 demo 和直覺賭。** 正因為這個注很難反悔，所以才要在選之前就把退場成本拆成具體的工算清楚，認出眼前是雙向門還是單向門——而那張逼人攤牌的矩陣，存在的唯一理由，是把暗處的偏好拖到光下，讓「下注」這件事接受同儕的審視而不是某個人的興奮。

所以一個好的選型，從來不是「選出最強的那個」。它是一連串把不確定性換成資訊、把反悔成本壓到可承受、把隱性偏好攤開受檢的動作。最強的技術，如果你只在它的 happy path 上見過它、又把自己綁得退不出來，那它在你某個凌晨三點的生產事故裡，會變成最昂貴的那個。選型要選的，是那個你**真正逼問過、也留得住退路**的——哪怕它無聊得讓人提不起勁。
