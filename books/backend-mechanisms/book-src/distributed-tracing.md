# 分散式追蹤：trace context 傳播、span 與取樣

某個午後，一條使用者下單的請求被回報「很慢」。你打開監控，路徑是 gateway → orders → payments → DB 四個服務，端到端 p99 是 800 毫秒。你一個一個服務點進去看儀表板，結果令人發毛：**每個服務都顯示自己很健康**——gateway 的 p99 是 220ms、orders 是 180ms、payments 是 240ms、DB 是 150ms，沒有任何一個超過你訂的門檻。四個團隊在群組裡互相看著，每個人都能拿出自己服務「綠燈」的截圖，沒有人是兇手。但請求確實慢了 800ms，它躲在某個地方。

這是一場羅生門，而它的根源是一個結構性的盲點：**metric 和 log 都是「以服務為單位」蒐集的，沒有任何訊號是「以請求為單位」跨越服務邊界的。** payments 服務的 p99 是 240ms，這個數字是把它收到的**所有**請求混在一起算出來的——其中絕大多數是健康的快請求，把那少數幾筆慢請求的尾巴稀釋掉了。你那一筆倒楣的請求，可能在 payments 裡花了 620ms，但它只是 payments 海量請求裡的一筆，淹沒在分位數的聚合裡看不見。每個服務的視角都是局部的、被聚合過的、對「單一請求的命運」失明。

分散式追蹤（distributed tracing）這套機制存在的唯一理由，就是把這個盲點補上：給每一筆請求一個身分，讓它穿越所有服務時都被同一個標識串起來，最後能把「這一筆請求在每一跳各花了多久」攤平成一張圖。它回答的是 metric 與 log 都先天答不了的問題——**這一次慢，是慢在哪一跳。**

## 為什麼這件事不能靠 log 拼出來

直覺上你會想：每個服務不是都有 log 嗎？把四個服務的 log 撈出來，按時間戳對一對，不就知道哪一跳慢了？

試一次就知道為什麼不行。第一，**你不知道要對哪幾條 log。** payments 在那 800ms 的窗口裡可能處理了幾百筆請求、寫了幾千行 log，哪一行屬於「你那一筆」？時間戳對不出來——多筆請求在同一毫秒並行進來是常態，光憑時間戳你無法把 gateway 的某一行和 payments 的某一行認定成同一筆請求的兩段。第二，**就算你猜對了，log 行之間也沒有因果關係。** 你看到 payments 寫了一行 "calling risk API"，但你不知道這個呼叫是不是由你追的那筆 gateway 請求觸發的，也不知道它後面那行 "risk API returned" 之間隔了多久、那段時間算誰的。log 是一地散落的事件，沒有把它們串成「一棵請求樹」的線。

問題的第一原理因此浮現：**要重建一筆請求的因果路徑，你需要一個在請求第一次進入系統時就生成、並且能完整地隨請求傳遍所有服務的全域標識。** 沒有這個標識，下游服務產生的每一段紀錄都是孤兒——你有一堆碎片，卻沒有把它們認親、排序、嵌套的依據。追蹤的全部工程，圍繞的就是「怎麼生出這個標識、怎麼讓它一路傳下去不斷、怎麼用它把碎片拼回一棵樹」。

## 三個概念：trace、span、context propagation

把這棵樹的零件先擺清楚。

一次端到端請求的完整紀錄叫一個 **trace**，由一個全域唯一的 `trace_id` 標識。從使用者點下按鈕到拿到回應之間，無論這筆請求扇出到多少個服務、觸發多少次 DB 查詢，它們全部共享同一個 `trace_id`——這是把所有碎片認親的鑰匙。

trace 樹裡的每個節點是一個 **span**：一段**有起止時間的工作**。一次 RPC 是一個 span、一次 DB 查詢是一個 span、一段值得單獨計時的內部計算也可以是一個 span。每個 span 攜帶這幾樣東西：自己的 `span_id`、指向呼叫它的上游的 `parent_span_id`、一對開始與結束時間戳，外加一組 attributes（HTTP method 與 status、DB statement、錯誤碼之類的鍵值標註）。`parent_span_id` 是把 span 串成樹的關節——順著每個 span 往上指的 parent，整棵「誰呼叫了誰」的拓樸就長出來了；而每個 span 的時間區間嵌套在它 parent 的區間之內，「誰花了多久、誰在等誰」也就一目了然。

最關鍵、也最容易出事的是第三件：**context propagation（context 傳播）**。trace 樹的零件分散在 N 個不同的行程裡，要把它們拼成一棵樹，每個服務在往下游發請求的當下，必須把「我是哪個 trace、我這個 span 的 id 是什麼」**塞進請求裡傳下去**。下游服務收到後，從中讀出 `trace_id`（沿用同一個）和上游的 `span_id`（拿來當自己新 span 的 `parent_span_id`），於是新生成的 span 就正確地掛在了樹上。對 HTTP 來說，這個「塞進請求」的載體就是 header。

第一原理因此很尖銳：**沒有 context 傳播，每個服務的 span 都是孤兒。** 你可以在每個服務裡勤勤懇懇地產生 span，但只要傳播沒做，這些 span 各自帶著不同的隨機 `trace_id`、彼此不知道對方存在——你有一堆 span，卻拼不成任何一棵樹，等於沒有 trace。傳播鏈是這整套機制的命脈，而它的脆弱之處在於：**只要斷一環，那一環之後的因果就全丟了。**

## traceparent：一行 header 的解剖

要讓不同語言、不同廠商寫的服務能互相接力傳遞 context，傳的格式必須是大家都認得的標準，否則 Go 寫的 gateway 塞的 header，Python 寫的 orders 讀不懂，鏈子在第一跳就斷。W3C 把這個格式標準化成 **Trace Context**，核心是一個叫 `traceparent` 的 HTTP header（這份規格的現行 Recommendation 於 2021-11-23 重新發布，初版早在 2020-02 即成為 Recommendation；強化 trace-id 隨機性的 Level 2 在 2026-06 仍停留在 Candidate Recommendation Draft 階段，但它不改 header 的結構）。

這個 header 長這樣，四個欄位用連字號分開：

```
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
             └┬┘ └──────────────┬──────────────┘ └───────┬──────┘ └┬┘
           version          trace-id (16 bytes)     parent-id    flags
            (2 hex)            (32 hex digits)     (span-id,8B)  (2 hex)
```

`version` 目前固定是 `00`，留給未來版本演進。`trace-id` 是 16 bytes、寫成 32 個十六進位字元，就是那把貫穿全程的鑰匙。`parent-id` 是發出這個 header 的服務「當前 span」的 `span_id`（8 bytes / 16 hex）——下游會拿它當自己 span 的 parent。最後的 `flags` 是 8 個 bit，目前只用了最低位一個：**sampled flag**。它的值（`01` 表示要採樣、`00` 表示不採）扛著一個遠比看起來重要的決定，我們稍後會看到它為什麼是整套取樣機制的關鍵。

傳播的動作於是收斂成一個簡單的循環：服務 A 處理請求時開一個 span，往下游 B 發請求前，把自己的 `trace_id` 和這個 span 的 `span_id` 編進 `traceparent` header 一起送出；B 收到後解析 header，沿用同一個 `trace_id`、把 A 的 `span_id` 設成自己新 span 的 parent，繼續往 C 傳同樣的編碼。一條 header 在每一跳被「讀出、改寫 parent-id、再送出」，trace 樹就沿著請求的真實路徑長出來了。除了 `traceparent`，標準還有一個 `tracestate` header 承載各廠商的私有資料；另外有一份獨立的 W3C 規格 baggage，透過自己專屬的 `baggage` header（而非塞進 `tracestate`）傳遞應用層業務鍵值，如 user_id（OpenTelemetry 內建它的 propagator）。但讓樹得以成形的最小骨架，就是上面那行 `traceparent`。

## 把羅生門攤平：一次手算

回到開場那筆 800ms 的請求。接上追蹤、讓這條 trace 的每個 span 都被記錄之後，我們把它攤開來看。每個 span 標出它的起止與自身耗時（self time，扣掉等待子 span 的時間）：

```
trace_id = 4bf9...4736   end-to-end ≈ 800 ms

gateway   [■···························································]  span 自身 20 ms
  └ orders  [·■·········································]              span 自身 30 ms
      └ payments  [··████████████████████████████████··]            span 自身 620 ms ← 瓶頸
          └ DB query  [········■········]                           span 自身 80 ms
```

把各跳的自身耗時相加：gateway 20 + orders 30 + payments 620 + DB 80 = **750ms**。它和端到端的 800ms 還差約 50ms——這 50ms 不是哪個服務算錯了，而是落在 span 沒涵蓋的縫隙裡：跨服務的網路傳輸、請求與回應的序列化反序列化、各服務收到請求到開出第一個 span 之間的那點排程延遲。這個「各 span 相加 ≠ 端到端」的差額本身就是一條診斷線索——差額大得異常，往往意味著有未被 instrument 的工作（某個沒開 span 的中介層、某段沒被計入的佇列等待）藏在縫隙裡。

但主角已經無所遁形：**payments 的自身耗時 620ms，吃掉了整筆請求的四分之三以上。** 點開 payments 那個 span 的 attributes，看到它對一個外部風控 API 的呼叫——那次呼叫沒設好逾時，外部 API 抖了一下，payments 就乾等了 600 多毫秒。

關鍵在於：payments 服務的儀表板為什麼一直顯示綠燈 p99 240ms？因為那條 trace 是 payments 處理的**成千上萬筆請求裡的一筆**。絕大多數請求的風控呼叫是快的，p99 被它們拉到 240ms；你這筆 620ms 的請求，是落在 p999 甚至更尾端的長尾裡的個案，被聚合的分位數徹底藏住了。**metric 給你的是「群體的統計」，trace 給你的是「個體的命運」**——羅生門之所以成為羅生門，正是因為大家手上都只有前者。一旦有了後者，兇手只需要一眼。

## 取樣：你不可能留下每一筆

攤平那筆慢請求很美好，但這裡藏著追蹤最現實的代價：**span 很貴。** 每一筆請求若橫跨 4 個服務、每個服務開兩三個 span，一筆請求就產生十幾個 span，每個 span 帶著時間戳、attributes、parent 關係，是一筆不小的結構化資料。一個每秒一萬請求的系統，全量記錄就是每秒十幾萬個 span——儲存、傳輸、查詢的成本會迅速失控。所以幾乎所有上規模的系統都得**取樣**：只留下一部分 trace，丟掉其餘的。

問題是「留哪些、丟哪些」這個決定要在**什麼時候**做，而這個時機點，劈出了取樣機制的兩條根本不同的路。

第一條叫 **head-based sampling（頭部取樣）**：在請求剛進入系統、root span 產生的那一刻就擲骰子決定。比方說設 10% 取樣率，root 服務擲一次骰，命中就把整條 trace 標記為「要留」，沒命中就標「不留」。這個決定**就編碼在前面那個 `traceparent` 的 sampled flag 裡**——root 把 flag 設成 `01` 或 `00`，隨請求傳遍全鏈，每個下游服務看到 flag 就知道「這條 trace 要不要記錄我這段」，於是整條鏈的取樣決定天然一致、不會出現「半條 trace 被留、半條被丟」的破碎樹。head-based 的好處全在這個「一開始就定、無狀態、隨 header 擴散」上：成本固定可預測、不增加延遲、不需要任何中央緩衝。

但 head-based 有一個致命的、且非常違反直覺的缺陷，值得講透：**它在還不知道這筆請求會不會出事之前，就決定了要不要記錄它。** 你最想看的 trace 是哪些？是出錯的、是慢的——但這些恰恰是極少數。錯誤率 0.1% 的系統，一千筆裡才一筆錯；而 10% 的隨機取樣，留下那一筆錯誤 trace 的機率只有 10%。換句話說，**九成的錯誤 trace 在它出錯之前，就已經被骰子判了丟棄。** 等你事後發現某筆請求出了錯、想回頭看它的 trace，它的 span 根本從來沒被記錄過——這不是「資料還在、只是難找」，而是資料**從不存在**。head-based 取樣的這個性質，是許多人第一次踩到時最錯愕的：取樣率寫 10% 不只是「省九成成本」，它同時是「主動扔掉九成你最想要的證據」。

## 尾部取樣：等看完全貌再決定

要修掉這個缺陷，唯一的辦法是把取樣決定**往後挪**——挪到能看見這筆請求結局的時候。這就是 **tail-based sampling（尾部取樣）**：不在請求開頭擲骰，而是**等整條 trace 完成、所有 span 都到齊、看到它有沒有出錯、夠不夠慢之後**，才決定留不留。於是你可以訂出聰明的政策：「latency > 500ms 的全留、有 error status 的全留、其餘的快樂正常請求只隨機留 10%」。那筆 620ms 的慢請求？100% 被保留。一筆 500 錯誤？100% 被保留。而海量的健康快請求，仍然只留一成控成本。tail-based 精準地把預算花在最有診斷價值的 trace 上——這正是它存在的全部理由。

但這份精準不是免費的，它的代價直接寫在「要等 trace 完成」這個前提裡。要在 trace 結束時根據它的全貌做決定，蒐集端（collector）就**必須把這條 trace 的所有 span 緩衝在記憶體裡，一直留到判斷它已經完成**——而「一條 trace 何時算完成」本身沒有明確信號，實務上只能等一個逾時窗口（比方說「root span 之後 N 秒內沒有新 span 進來就當它完成」）。這帶來兩個結構性的負擔。其一是**記憶體**：在那個窗口內，所有進行中的 trace 的所有 span 都堆在 collector 的記憶體裡，高吞吐下這是一大塊持續佔用的空間。其二，也是更刁鑽的一個，是**路由約束**：同一條 trace 的 span 可能來自不同服務、打到不同的 collector 實例，但 tail 取樣要看「整條 trace 的全貌」才能決策——所以**同一個 `trace_id` 的所有 span 必須被路由到同一個 collector 實例**。這逼著你在 collector 前面再加一層「按 `trace_id` 做一致性路由」的負載均衡（trace-aware load balancing），架構複雜度陡增。

兩條路的取捨於是清晰了，它不是「誰比較好」，而是「規模到了沒、痛點是什麼」：

| | head-based（頭部） | tail-based（尾部） |
|---|---|---|
| 決定時機 | root span 產生時 | 整條 trace 完成後 |
| 狀態需求 | 無狀態（編碼進 header flag） | 須緩衝整條 trace 在記憶體 |
| 成本 / 延遲 | 固定、低、可預測 | 記憶體貴、需 trace-aware 路由 |
| 能否留住錯誤/長尾 | 否（大機率漏掉） | 能（按 error/latency 精準留） |
| 適用 | 高吞吐、要的是代表性樣本 | 排障價值高、規模到了、預算夠 |

實務上的演化路徑幾乎總是：**先 head-based。** 它便宜、簡單、不需要特殊基礎設施，對「我只想要一批有代表性的 trace 看看系統長相」這個需求綽綽有餘。等到「老是在事後發現某筆錯誤想看 trace、卻發現它根本沒被記錄」這件事反覆發生、變成真正的痛點，再去評估上 tail-based。一開始就鋪 tail-based 是典型的過早最佳化——它那層 trace-aware 路由與記憶體緩衝的複雜度，只有在規模真的到了、且錯誤 trace 的可得性真的值錢時才划算。

## 它會在哪裡悄悄騙你

追蹤這套機制有幾個失敗模式，它們的共同特徵是**不會報錯、只會讓你的 trace 樹缺一塊或長歪**，比一個顯式的崩潰更難察覺。

最常見、也最隱蔽的是**傳播鏈悄悄斷掉**。前面說過，傳播鏈斷一環，那一環之後的因果就全丟。它最容易斷在哪裡？斷在**非同步邊界**。當請求路徑從同步的 HTTP 呼叫，轉成「丟一則訊息進佇列、稍後被另一個服務消費」時，那個 `traceparent` header 不會自動跟著訊息走——HTTP 的傳播是 instrumentation library 包住 HTTP client、在送出請求時自動注入 header 的，但訊息佇列是另一套傳輸、沒有這層自動攔截，你必須**手動把 trace context 塞進訊息的 metadata**，消費端再手動取出來接續。漏了這一步，trace 就在佇列那裡斷成兩棵互不相識的孤樹：生產端的一棵到「送出訊息」為止，消費端的一棵憑空冒出一個沒有 parent 的新 root。你的 trace 看起來「正常」，只是再也串不起跨佇列的那段因果——而跨佇列往往正是延遲最容易累積的地方。同理，任何一個沒做 instrument 的老服務、任何一條非標準協定的內部通道，都是傳播鏈潛在的斷點。

第二個陷阱埋在取樣裡，前面點到、這裡釘死：**head-based 取樣不能讓你事後補看錯誤 trace。** 這值得當成一條獨立的教訓記住，因為它違反「資料應該還在某處」的本能。head-based 在請求一開始就把「不記錄」這個決定編進了 flag，整條鏈都遵從它不寫 span——所以當你事後發現某筆出錯、想回頭調它的 trace，那些 span 從來沒有被寫下來過。錯誤 trace 的「事後可得性」是 tail-based 獨有的能力，不是換個查詢條件就能從 head-based 的資料裡撈出來的。

第三個是把**取樣後的 trace 數量當成流量指標**。trace 是被取樣過的——你資料庫裡有十萬條 trace，不代表系統處理了十萬筆請求，那只是命中取樣的那一部分。想知道真實請求數，要用全量、不取樣的 counter metric，絕不能去數 trace。同理，想從取樣後的 trace 算統計量（平均延遲、分位數），必須記下取樣率、用 `1/rate` 加權還原，否則總量會被系統性地低估——而這個「取樣後計數要加權還原、長尾分位數即使加權也修不回來」的陷阱，在所有被取樣的訊號上都成立（見〈從 log 算 metric 的陷阱〉）。

第四個是 **span 粒度的兩端極端**。把每個函式呼叫都開成一個 span，trace 樹會爆炸成幾百個節點、又貴又沒人讀得下去；但整個服務只開一個粗大的 span，你又看不出它內部到底卡在哪一段。粒度要對齊「有意義的工作單元」——一次 RPC、一次 DB query、一段關鍵的同步計算值得各自一個 span，而一個三行的純函式不值得。這沒有機械規則，是個需要拿捏的工程判斷：問自己「如果這一段慢了，我會想單獨看到它嗎」，會，就開 span；不會，就讓它併進 parent。

最後一個，也是最容易讓人對著 trace 圖困惑半天的，是 **clock skew（時鐘偏移）扭曲 span 時間**。每個 span 的起止時間戳，取自**它所在那台機器的本地時鐘**。但跨機器的時鐘從不完美同步——A 機器的時鐘可能比 B 機器快了幾毫秒。於是你會在 trace 圖上看到匪夷所思的畫面：一個子 span 的開始時間**早於**它 parent span 的開始時間，或者某段呈現出負的延遲。這不是 bug，是兩台機器的時鐘各說各話的結果。成熟的追蹤後端通常會用「子 span 必然在 parent 之內」這類因果約束去**修正顯示**，把明顯違反因果的時間戳拉回合理範圍，但底層那組原始時間戳本身仍然不可全信——當你看到接近 0 或為負的跨機器 span 延遲時，先懷疑時鐘，而不是懷疑物理（時鐘為何無法精確同步、邏輯排序怎麼補救，是另一個大主題，見〈邏輯時鐘與排序〉）。

## 為什麼是這個形狀

退一步看，分散式追蹤的整個樣貌，都是從開場那場羅生門裡長出來的。

因為 metric 與 log 都以服務為單位、對單一請求的命運失明，所以追蹤必須引入一個**以請求為單位、跨越所有服務邊界**的標識——於是有了 `trace_id` 和它的 span 樹。因為這個標識要穿過 N 個獨立行程才能把樹拼起來，所以它必須能隨請求**可靠地傳播**——於是有了 `traceparent` 這個跨廠商的標準 header，以及「傳播鏈斷一環就丟一段因果」這條命脈般的脆弱性。因為全量記錄太貴，所以必須取樣——而「在請求開頭擲骰」與「等請求結局再決定」這兩個時機，劈出了 head-based 的便宜無狀態與 tail-based 的精準昂貴，兩者的取捨不過是「你願意為了留住那些最有價值的錯誤 trace，付多少記憶體與架構複雜度」。

下次你盯著一條攤平的 trace、一眼看出 800ms 裡有 620ms 卡在 payments 對某個外部 API 的呼叫上時，你看到的不只是一張瀑布圖。你看到的是一套機制在兌現它唯一的承諾：在一個沒有任何單一服務能看見全局的系統裡，把一筆請求走過的每一步，重新縫合成一個完整、有序、可以被一個人讀懂的故事。
