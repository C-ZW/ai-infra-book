# 複製延遲與讀己之寫

使用者把暱稱從「老王」改成「王大明」，按下儲存，畫面轉一圈，回到個人頁——上面寫著「老王」。他皺眉，重新整理一次，這次對了。沒有任何錯誤、沒有任何例外、log 裡乾乾淨淨：那筆 `UPDATE` 確確實實成功了，資料庫裡躺著的就是「王大明」。但使用者剛剛親眼看到自己的修改「消失」了一瞬間，而且這件事每隔幾次就重演一次，永遠抓不到穩定的重現步驟。

這不是 bug，至少不是程式碼層面的 bug。它是一個架構決定在使用者眼前的投影：為了扛住讀流量，這個系統把讀請求分散到了好幾個**讀副本（read replica）**上，而那次倒楣的重新整理，恰好打到了一個還沒追上主庫的副本。要看懂這一瞬間為什麼會發生、為什麼難以根除、又為什麼不能靠「把延遲調到零」來解決，得先看清楚讀副本到底買到了什麼、又欠下了什麼。

## 為什麼會有讀副本，以及它的代價

絕大多數線上系統的讀寫比是嚴重傾斜的——讀遠多於寫，常常是 10:1、甚至更高。寫一筆貼文，可能被讀上千次。於是擴展讀吞吐成了最先撞上的牆，而最直接的解法是：讓一台主庫（primary）負責所有寫入，再掛上幾個副本，副本不斷把主庫的變更**重放（replay）**到自己身上，然後把讀流量分到這些副本去。三個副本，讀吞吐粗略地翻三倍，而且加副本比重新設計資料模型便宜得多。

這套架構的迷人之處在於它幾乎不需要改應用程式的邏輯——寫照樣寫主庫，讀換個連線字串指向副本就好。但「副本怎麼追上主庫」這件事，藏著整章的所有麻煩。主庫把每一筆變更寫進它的預寫日誌（PostgreSQL 的 WAL、MySQL 的 binlog；機制見〈WAL：先寫日誌，再改資料〉），副本訂閱這份日誌、按順序重放。關鍵在於：**這個重放預設是非同步的**——主庫不會等副本確認「我也寫好了」才回覆 client 成功，它寫完自己的、回覆 client，然後副本在背景慢慢追。

非同步是刻意的選擇，不是疏忽。如果主庫每筆寫入都要等所有副本確認，那副本的網路往返與磁碟速度就直接掛在了寫入的關鍵路徑上，寫延遲會被最慢的那個副本拖死，而且任何一個副本卡住就連帶卡住整個系統的寫入——你本來是為了擴展才加副本，結果加副本反而讓寫變慢、變脆。所以預設大家都選非同步：寫入快、副本掛了也不影響主庫接單。

代價就是那個無法消滅的時間差：**複製延遲（replication lag）**。副本永遠落後主庫一小段——健康時是毫秒級，但在主庫寫入尖峰、副本重放跟不上、或副本忙著跑一個大查詢時，這個落後可以拉長到數秒甚至更久。延遲不是常數，它是一條有長尾的分布。

## 延遲怎麼變成 bug：三種會話保證

複製延遲本身只是個物理事實，它要變成使用者眼中的 bug，得搭配一個很自然、卻在「主寫副讀」架構下不再成立的直覺。把這些直覺攤開，會發現它們其實是三條獨立的保證，各自會被延遲以不同方式打破。

第一條，也是開場那一幕，叫**讀己之寫（read-your-writes，又稱 read-after-write）**：「我剛寫的東西，我自己應該讀得到。」使用者寫入落在主庫，緊接著的讀卻打到一個還沒重放那筆寫的副本，於是讀到舊值——對使用者而言，他的修改「不見了」。注意這條保證的範圍很窄：它只要求**同一個使用者讀得到他自己剛寫的東西**，完全不要求別人也立刻看得到。這正是它便宜的原因——我們待會會看到，這個「只對自己」的限縮讓它能用遠比全域強一致便宜的手段達成。

第二條叫**單調讀（monotonic reads）**：「我看過新版本後，不該再看到舊版本。」設想使用者連續兩次讀同一份資料，第一次打到一個追得比較前的副本、看到了最新留言；第二次打到一個落後的副本、那則留言又消失了。資料像是在時間裡倒退。單調讀保證的是「只會前進、不會倒退」——它不保證你讀到的是最新的，只保證你不會看到比上次更舊的。

第三條叫**寫後讀（writes-follow-reads，或寫隨讀）**：你基於讀到的內容做了一筆寫（看到一則貼文、回覆它），那麼任何看得到你這筆回覆的人，也應該看得到你回覆的那則原文。否則會出現「回覆出現在原文之前」這種因果錯亂。

這三條保證最早由 Terry 等人在 1994 年的論文 "Session Guarantees for Weakly Consistent Replicated Data" 中系統化（連同第四條「單調寫」），合稱**會話保證（session guarantees）**。它們的共同特徵是：都不要求全域強一致，只要求「對單一使用者的會話視角」前後自洽。這個定位很重要——它告訴你，在一個最終一致（見〈一致性光譜〉）的系統裡，你不必把一致性整個拉滿，往往只要補上這幾條「對使用者自己一致」的窄保證，就能把絕大多數「資料不見了」的客訴擋掉，而成本遠低於對所有讀都走強一致。

## 把延遲算成 bug 的機率

「延遲偶爾會讓使用者讀到舊值」這句話太模糊，沒法拿來做工程決策。把它變成數字，才知道該不該管、該管到什麼程度。

設想一個服務，讀寫比 9:1，把讀導到 3 個副本後讀吞吐大約乘三。複製延遲在健康狀態下平均 200 ms，但它有長尾：p99 達到 1.2 秒——也就是說每 100 次讀，約有 1 次副本落後超過 1.2 秒。這個尾巴正是麻煩所在。

現在看那個最危險的使用者行為模式：「改完暱稱 → 立刻刷新個人頁」。從按下儲存到那次刷新落到副本上的讀，假設平均間隔 800 ms。讀己之寫會不會破，取決於一個簡單的賽跑：**這次刷新打到的副本，重放速度有沒有跟上這 800 ms 的間隔。** 如果副本的複製延遲小於 800 ms，它早已重放完那筆寫，使用者讀到新暱稱、相安無事；如果延遲大於 800 ms，副本還沒追上，使用者就讀到舊暱稱。

所以受傷的，是那批「複製延遲 > 使用者間隔」的請求。平均延遲 200 ms 遠小於 800 ms，看起來很安全——但別被平均騙了。傷害不發生在平均，發生在尾巴：在 p99 達 1.2 秒的分布下，「副本比使用者慢」的情況落在那條 200 ms 到 1.2 秒以上的尾巴裡，絕非罕見邊角。粗估之下，每天若有十萬次這種「改完即看」的操作，就算只有百分之幾落進壞掉的尾巴，也是每天數千次「我的修改不見了」——而且因為它取決於「刷新恰好打到哪個副本、那個副本當下落後多少」，它**無法穩定重現**，工程師照著客訴的步驟做十次有九次都是對的，於是這類 bug 特別容易被歸成「使用者看錯了」而擱置。

關鍵的設計轉折在這裡：**你不需要對全部的讀做強一致。** 真正會傷人的只有「改完→自看」這一條窄路徑。只要對這一條路徑特別處理——比如待會要講的「寫後黏主庫」，把窗口設成 2 秒（覆蓋 1.2 秒的 p99 還有餘量）——其餘 99% 的讀仍舊安穩地享受副本擴展。用一個窄路徑的特例，換掉一整類「資料不見了」的客訴，這個性價比是這整章最值得帶走的判斷。

## 怎麼補：四種手段，從鈍到精

知道了傷害集中在窄路徑，補法就有了層次。從最粗暴到最精巧，大致是四種，各自買到不同強度的保證、付不同的代價。

**第一種：寫後黏主庫一小段時間（sticky-to-primary）。** 使用者寫入後，把他接下來幾秒鐘的讀**全部導回主庫**。實作上常見的是寫入成功後，在一個共享的快取（如 Redis）裡記下 `user:42 → read-primary until T+2s` 這樣的標記，期間這個使用者的讀繞過副本、直接打主庫。主庫永遠是最新的，所以這段窗口內讀己之寫鐵定成立。它的好處是簡單、不需要在請求之間傳遞任何位置資訊；代價是抵銷了一部分讀擴展的效益——本來想把讀分出去，現在又有一撮讀回流到主庫。窗口長度是個取捨：設太短（比如 200 ms）可能還沒覆蓋到 p99 的延遲就放行了、讀己之寫漏接；設太長則主庫讀壓力回升。實務上把它對齊複製延遲的 p99（甚至 p999）再加一點餘量，是合理的起點。

**第二種：追蹤寫入位置，要求副本至少追到那裡（read-your-writes token）。** 這是最精準的一種，值得拆開看它怎麼運作，因為它把「讀己之寫」這個抽象保證，化成了一個可以手動實作的具體協定。

每一筆寫入在主庫提交時，都對應日誌上的一個**位置**——PostgreSQL 叫它 LSN（Log Sequence Number），MySQL 用 GTID（全域交易識別碼）。這個位置是單調遞增的，可以理解成「資料庫歷史的進度條」。協定是這樣的：

```
write path:
  1. client writes to primary
  2. primary commits, returns the write's log position (LSN / GTID)
  3. client stores this position as a "read-your-writes token"

read path:
  4. client sends a read, carrying the token
  5. router picks a replica, checks: has this replica replayed >= token?
        - yes -> serve the read from this replica (it has your write)
        - no  -> either wait for it to catch up, or fall back to primary
```

它的精髓在第 5 步那個比較。副本隨時知道自己重放到了哪個位置，路由層拿這個位置和 token 一比，就能判斷「這個副本身上有沒有使用者那筆寫」。有，就放心從副本讀；沒有，就退而求其次（等它追上、或改打主庫）。比起第一種「無腦黏主庫」，這種做法只在副本**真的還沒追上**時才回退，平常依然吃副本的擴展紅利，所以更省。

資料庫近年甚至把這個「等副本追上」內建成了原語。MySQL 8 有 `WAIT_FOR_EXECUTED_GTID_SET(gtid_set, timeout)`——在副本上呼叫它，會阻塞到副本重放完指定的 GTID 集合為止（timeout 預設 0，意思是無限等待；給了 timeout 就在那麼多秒後放棄並回傳 1 表示逾時，2026-06）。PostgreSQL 19 引入了 `WAIT FOR LSN`，效果類似——在副本 session 上阻塞等到重放越過某個 LSN 才繼續，等待由資料庫負責、不必應用層自己輪詢。值得一提的是這個功能在 PostgreSQL 17、18 都嘗試過、但都因為快照相關的死結問題在正式發布前被撤回，19 才落地（19 在 2026-06 仍是 beta 階段，未驗證其最終穩定版行為）。有了這類原語，第 5 步的「等待」就不必在應用層手刻輪詢。

這個 token 法精準歸精準，卻有個會在系統邊界悄悄斷掉的弱點，待會單獨講。

**第三種：把同一使用者黏在同一個副本上（pin to one replica）。** 這個手段瞄準的不是讀己之寫，而是**單調讀**。如果一個使用者的所有讀都打到固定那一個副本，那這個副本的重放只會前進不會後退，使用者自然不會看到「新版本又變回舊版本」的倒退現象。它便宜、不需要 token，但要清楚它的邊界：它**只給單調讀，不給讀己之寫**——你黏住的那個副本可能整體就比主庫慢，使用者還是可能讀不到自己剛寫的。而且那個副本一旦掛了、流量要重新分配，黏著就斷了，單調性也跟著斷一次。它適合「列表翻頁不要抖動」這類只在意「不倒退」的場景。

**第四種：少數關鍵讀直接走主庫。** 對那些「寫完馬上要讀、且絕不能讀到舊值」的路徑，乾脆標記成強讀、繞過所有副本。這是把問題消滅在源頭——主庫永遠最新。代價是這些讀完全沒享受到副本擴展，所以這條路徑必須**少**；一旦因為路徑分類沒做好、把大量讀都標成強讀，主庫就重新成為瓶頸，等於白加了副本。

四種手段不是互斥的選項，而是一套可以混搭的工具：絕大多數讀走便宜的副本、不做任何特殊處理；「改完即看」的窄路徑走第一或第二種；翻頁瀏覽走第三種；極少數「絕不能舊」的讀走第四種。

| 手段 | 買到的保證 | 主要代價 |
|---|---|---|
| 寫後黏主庫 N 秒 | 窗口內讀己之寫 | 主庫讀壓回升；N 設太短會漏接 |
| 寫入位置 token | 精準讀己之寫 | 要在請求間傳遞 token；跨服務難傳 |
| 黏同一副本 | 單調讀（不倒退） | 不保證讀到主庫最新；副本掛了就斷 |
| 關鍵讀走主庫 | 該路徑線性一致 | 路徑分類錯就破功；主庫成瓶頸 |

## 幾個會在邊角咬人的地方

機制講清楚了，真正會讓人栽跟頭的是幾個不那麼顯然的邊界。把它們一一講透，才算真的握住這個主題。

**「把延遲調到零」這條路通常走不通，或代價比 bug 還大。** 一個很自然的念頭是：既然非同步複製造成延遲、延遲造成 bug，那改成**同步複製**——主庫等副本確認才回 client——延遲不就沒了嗎？確實沒了，但你把成本搬到了更糟的位置。同步複製等於把副本的網路往返與磁碟 fsync 掛回寫入的關鍵路徑，寫延遲上升、寫入的失敗率上升（副本慢或掛，寫就跟著慢或失敗），你當初選非同步要躲的那些問題全回來了。而且這裡有個常被忽略的層次：同步複製本身還分等第。以 PostgreSQL 為例，預設根本不對副本做同步——但這不是因為 commit 模式本身非同步：`synchronous_commit` 預設值是 `on`，它只保證本機 WAL flush 到磁碟。真正讓它「不等任何副本」的是另一個參數 `synchronous_standby_names` 預設為空（沒有指定任何同步 standby）。要它等副本，得先填上 standby 名單；就算開了同步，設成 `remote_write` 只保證副本**收到並寫進 OS** 了、不保證**重放套用**了——也就是說副本確認了，但你去那個副本讀，還是可能讀不到！要真正保證「副本上讀得到」，得設到最嚴的 `remote_apply`（等副本把 WAL **重放套用**完才回），而這是最慢的一檔。所以「改成同步就解決了」這句話，既要付寫入變慢的全額，還得設對到 `remote_apply` 這一檔才真的給你讀己之寫——多數系統權衡下來，寧可只對那條窄路徑做特殊處理，也不願意讓所有寫入都付這個稅。

**token 法會在服務邊界悄悄斷掉。** 第二種手段在單一服務內很乾淨，但微服務一拆就出問題：A 服務寫主庫、拿到了 LSN/GTID，但接下來讀的是 B 服務，而 B 服務讀的是它**自己那套**副本、用的是它自己的位置座標系——A 拿到的 token 對 B 毫無意義，讀己之寫在服務的接縫處就斷了。要跨服務維持這條保證，得把這個位置資訊當成一種**因果上下文**，隨著請求一路傳遞下去（這已經很接近邏輯時鐘那套思路了，見〈邏輯時鐘與排序〉)，工程成本陡升。很多「在單體裡好好的讀己之寫，拆成微服務後莫名其妙開始漏」的怪事，根子就在這條傳不過去的 token。

**多筆連續寫，token 要對齊「最後一筆」。** 如果使用者連著做了三筆寫，你拿來當門檻的 token 必須是**最後一筆**（也就是位置最大那筆）的位置。拿錯了——比如只記了第一筆的 LSN——副本追到第一筆就被判定「夠新了」放行，但其實還差後兩筆，使用者照樣讀到中間態。位置是單調遞增的，所以規則很簡單：永遠用最大的那個。但「永遠記最新」這件事在有快取、有重試、有並發的真實程式裡，出乎意料地容易記錯。

**主庫故障切換（failover）會讓位置座標系跳變。** 當主庫掛掉、一個副本被提升為新主庫，日誌的位置編號可能不連續——PostgreSQL 會進入新的 timeline、MySQL 的 GTID 雖然設計上跨故障切換仍唯一（這正是 GTID 比舊式 binlog file+position 優越的地方），但你手上那個指向舊主庫的 token，在新主庫的世界裡未必還能直接拿來比較。任何依賴「位置可比較」的讀己之寫方案，都必須想清楚故障切換後這個假設還成不成立，否則切換後的那一小段時間，讀己之寫會集體破功。

**加副本是買吞吐、同時買進一個新的一致性問題。** 最該記住的一句話也許是：讀副本不是只賺不賠的免費午餐。你用它買到了讀吞吐，但同一筆交易裡，你也買進了「複製延遲」這個原本不存在的一致性難題。如果加了副本卻不為讀己之寫、單調讀做任何設計，等於是把一個物理延遲，直接放任它變成使用者眼中時隱時現、無法重現的 bug。決定加副本的那一刻，就該同時決定哪些讀路徑需要哪一種會話保證——這兩個決定是同一個決定的兩面。

## 為何是這個形狀

退一步看，這整章的所有機制，都是從一個拒絕被消滅的事實裡長出來的：**只要讀和寫落在不同的物理副本上，而副本之間的同步不是瞬間的，「我剛寫的我自己讀得到」這個最樸素的直覺就不再免費。**

正因為強行讓它免費（同步複製到 `remote_apply`）要付的稅太重——寫入變慢、變脆、被最慢的副本綁架——所以實務上幾乎所有系統都選了非同步、接受了延遲，再回過頭來只在**真正會傷人的窄路徑**上，用「黏主庫」「位置 token」「黏副本」這些**局部、廉價**的補丁，把使用者最在意的那幾條會話保證撐起來。會話保證之所以是「對單一使用者前後自洽」這個看似偏小的定位，不是因為它不夠野心，而是因為它精準地切中了「使用者真正會察覺到、會抱怨」的那一小塊，從而避開了全域強一致的昂貴。

所以下次你看到一個系統「平常好好的，偶爾使用者的修改閃一下不見、刷新又回來了」，你會知道那多半不是哪段程式碼錯了，而是一個讀請求恰好落到了還沒追上的副本上——而修它的正確姿勢，不是去追求把延遲歸零，是去問：**哪幾條路徑的使用者，真的會在副本追上來之前就回頭看？** 把答案數出來，只補那幾條，其餘的，讓延遲安靜地待在它該在的地方。
