# 背壓：當下游跟不上

凌晨的監控圖長這樣：服務跑得好好的，CPU 平穩、錯誤率掛零、延遲漂亮，所有綠燈。然後某一刻，一條看起來無害的曲線——行程的常駐記憶體——開始以幾乎是直線的斜率往上爬。沒有報錯，沒有逾時，吞吐數字甚至還很體面。直到記憶體撞上容器上限，OOM killer 出手，行程在毫無預警下被一槍打死，重啟，幾分鐘後又開始往上爬。

這幅圖的詭異之處在於：系統從頭到尾「看起來沒壞」。它不是慢慢劣化的，它是在懸崖邊上勻速散步，然後一腳踏空。如果你去翻那段時間的程式碼，多半找不到一行明顯的 bug——找到的會是一個佇列，一個 channel，一個 `for await` 迴圈，某個地方有人把工作往一個容器裡塞，而那個容器**沒有上限**。

這一章講的就是那個容器該有的上限，以及上限滿了之後該發生什麼事。這件事有個名字叫**背壓（backpressure）**：當下游消費者處理不過來，要有一條路把「慢下來」這個訊號沿著管線往上游傳回去，逼生產者降速、暫停、或被擋住——而不是讓中間的緩衝默默長大，直到把記憶體吃光。

## 一條守恆律，三個出口

把問題剝到第一原理，背壓底下只有一條鐵律，而且它簡單到近乎廢話：

> 在任何穩定運行的管線裡，長期的生產速率不可能超過最慢那一段的處理速率。

這條守恆律沒有例外，也無法用任何聰明的工程手段繞過——它和「水不能往上流」是同一個等級的東西。如果生產者長期以每秒 12,000 筆的速度往管線裡灌，而最慢的那一段每秒只能消化 10,000 筆，那麼每秒就有 2,000 筆「無處可去」。問題從來不是「會不會塞」，而是**那 2,000 筆去了哪裡**。

它只有三個出口，沒有第四個：

1. **堆起來**——多出來的工作排進一個佇列等著。如果這個佇列沒有上限，它會一直長大，把「處理不完」這件事的代價轉化成「記憶體被吃光」。這就是開場那幅圖。
2. **擋住**——讓生產者停下來。生產者想塞東西進管線時被卡住、被迫等待，於是它自然降速到下游能跟上的節奏。
3. **丟掉**——多出來的工作直接被丟棄，永遠不做。

背壓不是「第四個出口」，它是**有意識地在這三條路之間做選擇、並把選擇執行到底的那套機制**。一個沒有背壓的系統不是「沒有上限」——它是**把上限偷偷藏在了記憶體耗盡那一刻**。它會跑得好好的，給你一切正常的假象，直到某次流量尖峰讓那個無界佇列把記憶體吃乾，然後從「慢一點」毫無過渡地跳到「全死」。背壓做的事，本質上是把這個藏起來的、災難性的懸崖，換成一條**可控的、漸進的減速曲線**：寧可現在就慢下來或拒絕一部分，也不要在某個無法預測的時刻整個崩掉。

值得停下來看清楚這個取捨的形狀，因為它是全章的脊椎：**無界緩衝買到的是「平時看起來很順、尖峰時災難性崩潰」；背壓買到的是「平時可能偶爾被擋或被拒、但永遠不會猝死」。** 你付出的是一點點日常的不順，換回的是「不會在最糟的時刻被 OOM」的保證。對線上系統來說，這幾乎總是划算的交易。

## 把懸崖算出來

抽象的守恆律不如一個帶數字的場景有說服力。把開場那個 ingestion 服務拿來算到底。

它每秒收進 12,000 筆事件，下游（某個批次寫入）每秒只能消化 10,000 筆。假設每筆事件在記憶體裡的物件大約佔 1 KiB（一個帶幾個欄位的 JSON 解析後物件，這個量級很現實）。

**先看無界佇列的版本。** 每秒淨積壓 `12,000 − 10,000 = 2,000` 筆。記憶體的增長是線性的：

```
積壓筆數(t) = 2,000 × t      (t 以秒計)
佔用記憶體(t) = 2,000 × t × 1 KiB
```

代進去：1 分鐘積壓 12 萬筆、約 117 MiB；10 分鐘積壓 120 萬筆、約 1.1 GiB。在一個記憶體上限 1 GiB 的容器裡，這條線會在大約第 9 分鐘撞牆，行程被 OOM 殺掉。注意這整整 9 分鐘裡，**所有對外指標都是健康的**——它一直以 10,000/秒 的速度在出貨，延遲只是緩慢變長（後進來的事件要排在前面那一大坨後面），沒有任何一個請求報錯。監控不會在第 3 分鐘就告訴你「要爆了」，因為從吞吐和錯誤率看，什麼壞事都還沒發生。這正是無界佇列最陰險的地方：它把警訊壓到最後一刻才一次引爆。

**換成有界佇列 + 拒絕。** 給佇列一個 50,000 筆的上限（約 49 MiB，對任何容器都是零頭）。佇列從空到滿，需要 `50,000 / 2,000 = 25` 秒填滿。25 秒後，佇列維持在 50,000 筆滿載，每秒新進的 12,000 筆裡，10,000 筆能進去補上被消費掉的空位，剩下的 2,000 筆碰到滿佇列、被拒絕（回一個 HTTP 429 或 503，把球踢回上游）。

結果是：記憶體穩定釘在約 49 MiB，永遠不再長大；系統持續以 10,000/秒 穩定出貨，不會猝死；而那每秒 2,000 筆被拒絕的事件，變成一個**明確的、上游看得見的訊號**——上游可以選擇退避後重送、降速、或在它自己那端做取捨。系統的行為從「10 分鐘後無預警暴斃」變成了「持續穩定運作 + 對超量的部分誠實地說不」。

這個對比裡藏著一個常被誤解的點，值得拆穿：**把佇列加大，不是背壓。** 假設有人看到佇列滿了在拒絕，覺得「那我把上限調到 50 萬筆不就好了」。算一下：50 萬筆的佇列只是把填滿時間從 25 秒延到 `500,000 / 2,000 = 250` 秒，約 4 分鐘——懸崖往後挪了，但生產 > 消費這條不等式一個字都沒變，佇列照樣會滿，而且這次積壓的這約 488 MiB 全變成延遲：一筆新進來的事件要排在前面 50 萬筆後面，等 `500,000 / 10,000 = 50` 秒才被處理，而它的上游可能 30 秒前就逾時放棄了。**佇列是用來吸收短暫尖峰的，不是用來吸收持續性超載的。** 加大佇列只在「尖峰是一陣一陣、平均速率其實跟得上」時有用；面對「平均速率就是跟不上」，它唯一的作用是讓你更晚死、死得更難看。

## 壓力怎麼往上傳：四種傳法

知道了「有界 + 在三個出口之間選」是答案，真正的工程問題是：這個「慢下來」的訊號**具體怎麼從下游傳回生產者**？這裡有幾種本質不同的傳法，各自買到不同的保證、各自在不同的並發模型裡會反咬一口。

**第一種：阻塞（blocking）。** 佇列設一個固定上限，滿了之後，生產者呼叫 `put`（或往 channel 寫）的那個動作直接**卡住、不返回**，直到佇列騰出空位。壓力透過「生產者被擋在那裡動不了」這件事往上傳——它被卡住，就不會再去拉更多上游的工作，於是上游也跟著慢下來，一路傳到源頭。這是最直接、最不丟資料的傳法：沒有東西被拒絕、沒有東西被丟，只是整條鏈以最慢那段的節奏前進。Java 的 `BlockingQueue`、Go 的有緩衝 channel（寫滿即阻塞）都是這個模型。

**第二種：拒絕（reject / fail-fast）。** 佇列滿了不阻塞，直接回一個錯誤——對外是 HTTP 429（Too Many Requests）或 503，對內可能是一個 `QueueFullError`。把「慢下來」變成一個**明確的回應**，讓更上游自己決定要退避重送還是放棄。這是 ingestion 那個 worked example 用的策略。它的好處是延遲有上界（沒有人會被無限期卡住），代價是把問題推給了上游——而這就埋下一個之後要拆的雷：如果上游收到 429 卻無腦地立刻重試，背壓不但沒幫到忙，還可能引爆重試風暴（見〈retry storm 與 metastable failure〉）。

**第三種：基於需求的拉取（demand-based pull）。** 前兩種都是「生產者推、滿了再處理」，這一種反過來：**消費者主動告訴生產者「我現在還能收 N 筆」，生產者只送這麼多，一筆不多。** 需求（demand）成了一個顯式的、可流動的數字，沿著訂閱關係從下游流向上游。這是最乾淨的背壓——因為**所有緩衝天生有界，而且界限由消費者掌握**，生產者根本沒有機會「推太多」，壓力不是「事後回壓」而是「事前不給」。Reactive Streams 規格（下一節細談）、gRPC streaming、TCP 本身，都是這個模型的不同化身。

**第四種：丟棄最舊 / 取樣（drop-oldest / sampling）。** 對某些串流，「完整」根本不是目標，「新鮮」才是——即時報價、感測器讀數、遊戲狀態同步，你要的永遠是最新值，舊值過時就是垃圾。這時佇列滿了最聰明的做法是**丟掉最舊的、留下最新的**，用犧牲完整性換取有界記憶體和最低延遲。但這條路有個鋒利的邊界：它**只對「可以丟」的資料成立**。金流交易、訂單、每一筆都要處理的事件，絕對不能用丟棄當背壓——那不是背壓，那是資料遺失。選背壓策略的第一個問題永遠是：**這條管線上的工作，能不能丟？能丟到什麼程度？**

這四種不是「哪個最好」，而是「在什麼約束下用哪個」。把它們的取捨並排：

| 傳法 | 買到的保證 | 適用 | 反咬一口的地方 |
|---|---|---|---|
| 阻塞 | 記憶體有界、不丟資料 | 生產者可被安全地卡住的同步管線 | 阻塞會沿呼叫鏈往上傳染，在 event-loop 裡是大忌（下節） |
| 拒絕 (429/503) | 記憶體有界、延遲有上界 | 對外 API、上游能退避 | 上游若無退避地重試 → retry storm |
| 需求拉取 (`request(n)`) | 記憶體有界、端到端流量守恆 | 串流處理、Reactive、gRPC streaming | 整條鏈得都支援拉取，一個環節不支援就破功（下節） |
| 丟棄最舊 / 取樣 | 記憶體有界、資料保新鮮 | 只在乎最新值的遙測 | 會丟資料，金流/訂單禁用 |

## 阻塞式背壓的甜蜜陷阱

阻塞看起來是四種裡最省心的：設個有界佇列、滿了就卡住、不丟不拒、整條鏈自動同步到最慢的節奏。在一個「每個請求一條執行緒」的傳統模型裡，它確實大致沒問題——卡住的是處理那筆工作的那一條執行緒，別的執行緒繼續跑。

但把同一招原封不動搬進**單執行緒的 event-loop**（Node.js、或任何 reactor 模型），它會從救星變成兇手，而且是那種「demo 跑得好好的、上線在尖峰時把整個服務一次掐死」的兇手。

原因是 event-loop 的核心假設：**那唯一的一條執行緒永遠不准被阻塞。** 它靠著「不停地在大量連線之間快速切換、每個只碰一下就走」來製造「同時服務上萬連線」的假象（這個機制本身見〈event loop：單執行緒怎麼同時做很多事〉）。一旦你在這條執行緒上呼叫了一個會「卡住等佇列騰空」的阻塞 `put`，你卡住的不是「當前這個請求」——你卡住的是**整條 event-loop**。在它被卡住的這段時間裡，其他**所有**連線一起停擺：健康檢查超時、正在進行的請求全部凍結、新連線無人接受。下游一慢，本來只該影響一條管線，結果透過這個阻塞點，瞬間變成全服務癱瘓。

所以背壓策略不能脫離並發模型來選。**在 event-loop 裡，背壓必須是非阻塞的**——要嘛用拒絕（立刻回 429，執行緒碰一下就走），要嘛用需求拉取（用一個非阻塞的訊號控制流量）。Node.js 的 stream 把這套做進了協定裡，值得當成「非阻塞背壓長什麼樣」的標本看一眼。

當你對一個 writable stream 呼叫 `write(chunk)`，它的回傳值不是「寫成功了沒」，而是一個背壓訊號：**回傳 `false` 的意思是「我內部的緩衝已經超過 highWaterMark 這條水位線了，你該停手了」。** 注意這筆 `chunk` **還是被收下了**(不會丟)——`false` 不是拒絕，是一句「這筆收了，但別再給我更多，等我喊你」。生產者該做的，是停止寫入、掛一個 `'drain'` 事件的監聽器，等下游把緩衝消化到水位線以下、`stream` 主動發出 `'drain'`，再恢復寫入：

```js
// non-blocking backpressure, the Node.js way
function writeWithBackpressure(writable, chunk, done) {
  const canContinue = writable.write(chunk);
  if (!canContinue) {
    // buffer is over highWaterMark — stop, wait to be told it's drained
    writable.once('drain', done);
  } else {
    done();
  }
}
```

整套協定沒有任何一處阻塞：`write()` 立刻返回（`false` 或 `true`），event-loop 一刻都沒被卡住，壓力是透過「生產者自願掛起、等 `drain` 再續」這個非阻塞的握手往上傳的。那條 highWaterMark 水位線就是佇列的上限（byte stream 預設 64 KiB——這是 Node.js 22 起的值，更早的版本是 16 KiB；objectMode 預設 16 個物件），`write()` 回 `false` 就是「滿了」的訊號。順帶一提，`pipe()` 和 `pipeline()` 之所以該優先用，正是因為它們把這整套 `write`/`drain` 的握手自動接好了——手動接最容易漏掉的，就是「`write()` 回 `false` 了卻沒等 `drain` 繼續硬塞」，那一行之差，就把非阻塞背壓退化回了開場那個無界增長的記憶體炸彈。

## 需求拉取：把背壓做成協定

拒絕和阻塞都是「事後」的——下游滿了才回壓。最徹底的背壓是把它做成**事前**的：讓需求變成一個顯式的、由消費者發出的訊號，生產者**只在收到需求時才生產、且絕不超過需求量**。這就是 Reactive Streams 規格的全部精神。

Reactive Streams 用四個介面把這件事釘死（Publisher、Subscriber、Subscription、Processor），核心動作在 `Subscription` 上的一個方法：`request(n)`。流程是這樣的——`Subscriber` 訂閱後，不是被動等著被灌，而是主動呼叫 `subscription.request(n)`，宣告「我現在準備好處理 n 筆」。`Publisher` 收到這個信用額度，**至多送 n 筆**過來，送完就停，直到 `Subscriber` 處理完、再 `request` 更多。整個過程沒有任何阻塞——`request(n)` 和 `onNext` 都是非阻塞的回呼；背壓不是靠「卡住誰」，而是靠「沒拿到信用就一筆都不送」。

這個設計的漂亮之處，是它讓**整條鏈的緩衝天生有界，而且界限由最下游的消費者一路反向控制**。消費者只 `request` 它當下能吃的量，這個量沿著鏈往上游傳遞，每一段都只生產下游要的量。生產者快、消費者慢？不要緊——生產者拿不到信用，自然就停在那裡，壓力守恆地傳到了源頭，沒有任何一段需要無界緩衝來吸收速差。

但這種優雅有一個**全有全無的前提**，而且這是需求拉取最容易栽的地方：**整條鏈必須每一個環節都遵守 `request(n)` 語意。只要有任何一個環節是「無視需求、來多少推多少」的 push，整條鏈的背壓就在那一點破功。** 想像一條 `源 → A → B → 慢消費者` 的鏈，A 和 B 都老實實作了拉取，但 A 的上游「源」是個第三方 SDK，它不認識 `request`，連上就用 push 模式把資料往 A 身上砸。這時 A 為了不丟資料，只能把砸過來、但下游還沒 `request` 的部分**自己緩衝起來**——而這個緩衝，因為源不理會背壓，是**無界的**。整條鏈精心設計的有界性，被這一個不合作的環節徹底瓦解；記憶體還是會在 A 這裡爆掉，只是爆點換了個位置。這個教訓很硬：**背壓是一條鏈的性質，不是一個元件的性質。一條鏈的背壓強度，等於它最弱（最不合作）那一環的背壓強度。** 評估一條串流管線安不安全，要問的不是「我有沒有做背壓」，而是「這條鏈上每一個 hop，包括那些我沒寫的第三方環節，有沒有一個是無界 push 的」。

## 背壓會穿過網路：你早就在用它

到這裡為止，背壓聽起來像是「程式內部、佇列與 channel 之間」的事。但有一個事實會改變你對它的整個觀感：**背壓不只在行程內傳，它會穿過網路，而且 TCP 本身就是一套背壓機制。** 你寫的每一個網路服務，底下早就跑著背壓，只是你沒意識到。

TCP 的接收端有一個**接收視窗（receive window,rwnd）**——它在每一個 ACK 封包裡，告訴發送端「我的接收緩衝區現在還剩多少空間，你最多再送這麼多 bytes 過來」。發送端被這個數字約束，絕不送超過接收端宣告的容量。如果接收方的應用程式讀取得慢（緩衝區塞滿了還沒被 `read` 走），接收視窗就會縮小，一路縮到 0；發送端看到視窗 0，就**停止發送**，卡在那裡等接收端騰出空間、宣告一個更大的視窗。這不就是需求拉取嗎？接收端用視窗大小，顯式地告訴發送端「我現在能收多少」——這正是 `request(n)`，只是它叫 rwnd，而且做在傳輸層。

(這裡要劃清一條常被混淆的界線：TCP 的**流量控制（flow control，即 rwnd）** 和**壅塞控制（congestion control，即 cwnd）** 是兩回事。前者保護的是**接收端**不被發送端灌爆——這才是背壓；後者保護的是**中間的網路**不被灌爆。實際送出量取兩者的最小值。背壓對應的是 rwnd 這一半。)

這件事的深刻後果是：**只要你的管線最終落在一個阻塞式的 socket 寫入上，你其實免費繼承了 TCP 的背壓。** 一個老派的同步伺服器，從 socket `read` 一筆、處理、往另一個 socket `write` 一筆——當下游消費端讀得慢，它的 TCP rwnd 縮小，你這端的 `write` 就會阻塞，於是你的 `read` 也跟著停，壓力一路經由 TCP 視窗反向傳回了**最源頭的那個客戶端**。這套人沒寫一行背壓邏輯，卻有完整的端到端背壓——因為它老老實實地靠在阻塞 socket 上，而阻塞 socket 後面站著 TCP 視窗。

往上一層，HTTP/2 和 gRPC 把同樣的把戲在應用層又做了一遍，而且**做得更必要**。HTTP/2 在單一條 TCP 連線上多工（multiplex）許多 streams——但這帶來一個新問題：如果只靠底層那一個連線層級的 TCP 視窗，一個**應用層讀得慢**的 stream 會把共用的接收緩衝佔滿、吃光整條連線的視窗，連累其他所有 stream 一起被餓死。這是個流量控制粒度／公平性的問題。所以 HTTP/2 自己加了**一層 stream 級的流量控制**：每個 stream 和整條連線各有獨立的視窗（初始值 65,535 bytes），把各 stream 的水位彼此隔離開，接收端用 `WINDOW_UPDATE` frame 顯式地「補充信用」——「這個 stream 我又能多收這麼多了」。這又是 `request(n)`，化身成 `WINDOW_UPDATE`。（這層 stream 級流量控制解的是「公平性」，別把它和 HTTP/2 另一個有名的毛病——transport 層的 head-of-line blocking——搞混：後者是**封包遺失**造成的，一個 TCP segment 掉了，後面所有 stream 的資料都得卡著等重傳，那是流量控制管不到的問題，HTTP/2 始終沒能根除，要到 HTTP/3 改用 QUIC 的 per-stream loss recovery 才解；見〈HTTP/1.1、HTTP/2、HTTP/3〉。）gRPC 直接坐在 HTTP/2 上，於是 gRPC streaming 的背壓就是 HTTP/2 視窗的背壓：當你做 server streaming、客戶端讀得慢，客戶端的視窗不再補充，server 端的傳送就會停滯。gRPC 的 stub 把這個底層訊號暴露成 `isReady()`/`onReadyHandler`——server 在 `onNext` 之前該檢查 `isReady()`,`false` 就表示「視窗滿了、慢客戶端跟不上，先別送」。無視這個訊號、在 `isReady()` 已經 `false` 時還硬塞，資料就會堆在 server 端的記憶體裡——又是同一個無界緩衝的炸彈，只是換了個門面。

看穿這一層，背壓就從「一個要額外實作的功能」變成了「一個貫穿所有層的同一個機制」:TCP 的 rwnd、HTTP/2 的 `WINDOW_UPDATE`、Reactive Streams 的 `request(n)`、Node stream 的 `drain`——**它們是同一件事在不同高度的四張臉**。都是消費端對生產端說同一句話：**「我現在最多能收這麼多，別超過。」**

## 背壓不是限流，別把兩者搞混

背壓常和限流（rate limiting，見〈rate limiting〉那章深講）被當成一回事，因為兩者在過載時都「擋住了一些請求」。但它們是兩個方向相反的機制，混為一談會讓你在該用 A 的地方用了 B。

差別在**訊號從哪來、何時決定**。限流是**主動的、事前的**：你預先設一個固定速率上限（每秒 1,000 筆），在門口就按這個配額決定放行或攔截，**完全不看下游當下忙不忙**——哪怕下游此刻很閒，超過配額照樣擋。背壓是**被動的、事後的**：沒有預設的數字，擋不擋取決於**下游此刻真實的飽和度**——佇列滿了、視窗到 0 了、`isReady()` 變 `false` 了，才回壓。一個是「我規定你每秒只能進這麼多」，一個是「我下游現在塞住了，你等等」。

這個區別有實際後果。限流防的是**可預期的濫用**——某個租戶想打爆你、某個 API 要按配額計費，你用一個和下游狀態無關的固定數字把它框住。背壓防的是**不可預期的變慢**——下游突然 GC 暫停、突然一個依賴變慢、突然流量正常但某段卡了，這些你事先不知道速率該設多少，只能靠「下游真的塞了才回壓」來應對。所以最穩的系統**兩者都有，各司其職**：限流在前門用固定配額擋掉可預期的洪水，背壓在管線內部依真實飽和度應對不可預期的速差。把背壓當限流用（「我加個固定速率就不會塞了吧」），會在下游莫名其妙變慢、但流量還在配額內時失守；把限流當背壓用（「下游塞了我臨時調低速率」），則永遠慢半拍、且調不準。

## 什麼時候不需要它

背壓是個好東西，但「好東西」最危險的用法是到處硬加。有一類管線，背壓**已經隱含在結構裡了**，再疊一層顯式背壓只是徒增複雜度。

判準很簡單：**問這條管線上，有沒有任何一個「無界緩衝」?** 沒有，就不需要額外的背壓。

最常見的「天生有背壓」的結構，是純同步的請求-回應：一個 HTTP handler 收到請求、直接查資料庫、回傳，中間**沒有佇列**。這種管線的背壓藏在兩個有界資源裡——伺服器的並發連線數有上限，資料庫連線池有上限（見〈連線池〉）。當下游（資料庫）變慢，連線池的連線被佔住、很快耗盡，新請求拿不到連線就被擋住或拒絕，壓力自動透過「連線池見底」傳回了源頭。這裡的連線池**就是**那個有界佇列，背壓**就是**「連線池滿了申請者被擋住」。你不需要再造一層，事實上再造一層往往是把同一件事做兩遍。

真正需要顯式背壓的，是**生產與消費解耦、中間夾著一個可能無界的緩衝**的場合：訊息佇列的消費者、串流處理管線、WebSocket 的廣播扇出、任何「一端往 channel 推、另一端慢慢消化」的結構。這些地方的緩衝如果沒人管上限，就是開場那幅記憶體曲線。所以加背壓之前，先誠實地找出「這條管線的緩衝到底在哪裡、它有沒有界」——找到那個真正無界的點，把保護加在那一點上，而不是憑「背壓是好東西」的直覺到處撒。

## 為何是這個形狀

退一步看，背壓的整個樣貌，都是從那條開頭的守恆律裡長出來的：**長期生產速率不可能超過最慢一段的處理速率。**

正因為這條律無法繞過，多出來的工作只有三個出口——堆、擋、丟——所以背壓的本質永遠是「在這三者間做一個誠實的選擇」，而不是幻想存在一個「都不選」的免費方案。正因為「都不選」實際上等於「偷偷選了堆，而且是無界地堆」，所以無界緩衝才那麼陰險：它把代價從「現在付一點」遞延成「將來一次付清」，用一段虛假的歲月靜好，換一場毫無預警的猝死。背壓做的，就是把這筆遞延的、會利滾利的債，換成當下就結清的小額帳單——平時偶爾被擋、被拒、被取樣，換掉尖峰時的 OOM。

正因為「慢下來」這個訊號必須真的傳到源頭才有用，背壓才必須是一條**完整的鏈**：任何一個不合作的環節——一個阻塞了 event-loop 的 `put`、一個無視 `request(n)` 的 push 源、一個沒等 `drain` 就硬寫的迴圈——都會在那一點讓壓力洩漏成無界增長。也正因為這個訊號的需求是普遍的，它才會在每一層被重新發明一遍：TCP 的視窗、HTTP/2 的 `WINDOW_UPDATE`、Reactive 的 `request(n)`、stream 的 `drain`，全是同一句「我現在最多能收這麼多」在不同高度的回聲。

下次你看到一個服務在尖峰時開始回 429、或一個 stream 在 `write()` 上回 `false`、或一個 gRPC server 的 `isReady()` 翻成 `false`，你會知道那不是它壞了——那是它在拒絕走上開場那條記憶體曲線。它在用一點點當下的不順，兌現一個承諾：寧可現在對你說「等一下」，也不要在某個你猝不及防的凌晨，一腳踏空。
