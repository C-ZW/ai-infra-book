# 錯誤處理哲學：fail-fast、傳播、重試還是降級

凌晨三點，一個結帳請求進來。它要呼叫四個下游：庫存服務確認貨還在、定價服務算出金額、風控服務檢查這張卡有沒有被盜刷、最後寫一筆訂單進資料庫。前三個都回了，第四個——資料庫——這一刻連線逾時了。你的程式碼站在這個分岔口上，手裡握著一個 error 物件，必須當場決定一件事：**接下來怎麼辦。**

選項看起來很多。可以立刻把整個請求打掉、回 500 給使用者（fail-fast）。可以原地重試幾次，賭那只是一次網路抖動（重試）。可以把這個 error 往上拋，讓更外層的某個 handler 去決定（傳播）。可以退而求其次，把訂單寫進一個本地佇列、先回使用者「已收到，處理中」，稍後補（降級）。也可以——這是最誘人、也最危險的一條——`catch` 住它、log 一行 `order failed`，然後**假裝什麼都沒發生**，讓請求繼續往下走，回一個 200。

這五條路會把系統帶到五個截然不同的未來。而真正麻煩的是：**沒有一條對所有錯誤都對。** 同一句 `catch (e)`，對某一類錯誤是救命，對另一類是埋雷。錯誤處理之所以是「哲學」而不是「技巧」，正因為它逼你先回答一個比「怎麼處理」更前面的問題——**這個錯誤，到底是哪一種錯誤。**

## 天真的做法：一視同仁

最自然、也最常見的天真做法，是把所有錯誤當成同一種東西處理。最極端的版本是「全都 try/catch 包起來，不要讓程式掛掉」——這背後有一個聽起來很有道理的直覺：**程式不崩，使用者體驗就好。** 於是每一段可能出錯的程式碼外面都裹一層：

```js
try {
  await chargeCard(order);
} catch (e) {
  logger.warn("charge failed", e);
  // ... 然後呢？繼續往下走
}
```

這段程式碼在 demo 裡跑得很漂亮。問題是 `chargeCard` 失敗有兩種完全不同的原因，而這個 `catch` 把它們碾成了同一團：

第一種，`chargeCard` 失敗是因為**支付閘道暫時連不上**——一次網路抖動，五秒後就好了。這是一個**預期之內的營運狀況**。網路會抖、下游會慢、第三方會偶爾 503，這些不是 bug，是分散式系統的日常天氣。對這種錯誤，吞掉它確實「不會崩」，但代價是這筆訂單**沒收到錢卻可能繼續往下出貨**。

第二種，`chargeCard` 失敗是因為傳進去的 `order.amount` 是 `undefined`——上游某個欄位算錯了，是一個**程式 bug**，一個「不該發生」的狀態。對這種錯誤，吞掉它是災難中的災難：你不但沒崩，還**帶著一個壞掉的訂單繼續跑**，把這個 bug 的後果擴散到下游每一個系統。等到三天後有人發現對帳對不平、回頭追查時，現場早就被那句 `logger.warn` 吃掉了——你只剩一行沒有 stack、沒有 context 的 warning，連 bug 在哪都不知道。

這就是天真做法的根本錯誤：**它假設所有錯誤的正確反應都一樣。** 但「支付閘道暫時連不上」和「金額是 undefined」是兩種在本質上對立的事件，混為一談，必然對其中一種做錯。要不就是對營運狀況太苛刻（一個下游慢就讓整個請求崩），要不就是對 bug 太寬容（把該大聲尖叫的程式錯誤靜靜吞掉）。錯誤處理的第一步，從來不是決定「怎麼處理」，而是**把錯誤分類**。

## 兩條切割線

把錯誤分清楚，靠的是兩條互相垂直的切割線。它們是這整章的座標軸，後面所有決定都掛在上面。

第一條線：**這個錯誤可不可恢復？** 一端是**暫時性（transient）**錯誤——它現在不行，但本質上「過一陣子、或換個節點、或重試一次」就可能成功：網路抖動、連線池暫時耗盡、下游 503、樂觀鎖衝突。另一端是**永久性（permanent）**錯誤——重試一百次也是同樣的結果：請求格式非法、認證失敗、業務規則違反（餘額不足）、資源不存在。這條線決定「**重試有沒有意義**」：對暫時性錯誤重試是吸收故障，對永久性錯誤重試是浪費資源、甚至把一次失敗放大成一場 retry storm（見〈retry storm 與 metastable failure〉）。

第二條線：**這個錯誤是不是 bug？** 一端是**程式錯誤（programming error）**——空指標、陣列越界、不變式被違反、`switch` 漏了一個 case。它代表「**我對世界的某個假設破了**」。另一端是**預期內的營運狀況（operational condition）**——下游逾時、輸入不合法、第三方限流。它代表「**世界如常運作，只是這一刻不順**」。這條線決定「**該不該把錯誤暴露出來、還是該把它當控制流處理**」。

這兩條線交叉，給出四個象限，而它們的正確反應南轅北轍：

| | 暫時性（可恢復） | 永久性（不可恢復） |
|---|---|---|
| **是 bug（程式錯誤）** | 罕見且可疑：是 bug 偽裝成偶發 | fail-fast，大聲爆，附完整 context |
| **不是 bug（營運狀況）** | 重試 / 降級 / 熔斷 | 乾淨地回報給呼叫端（4xx 類） |

整章接下來要做的，就是把每個象限該走的那條路講透——以及，更重要的，講清楚**為什麼**是那條路。

## 程式錯誤：為什麼要 fail-fast

從最違反直覺的那一格開始：bug 該怎麼處理。常見的反應是「bug 當然要處理掉、不要讓它崩」，但正確答案恰恰相反——**程式錯誤應該 fail-fast：一偵測到不對，就立刻、大聲地失敗，停在離源頭最近的地方。**

「fail-fast」這個詞作為一條明確的工程主張，出自 Jim Shore 2004 年發在 IEEE Software 的一篇短文（這個專欄當時由 Martin Fowler 編）。它的核心論點很反直覺：**一個會立刻崩潰的系統，往往比一個努力撐著不崩的系統更穩定、更好維護。** 為什麼？

關鍵在於「壞狀態」會傳染。回到那個 `order.amount === undefined`。這是一個訊號，它在告訴你：上游某處有一段邏輯算錯了。如果你在這一刻 fail-fast——立刻拋出一個帶著 `order.id`、帶著完整 stack trace 的錯誤——你就把這個 bug 釘死在了它**最接近成因的地方**。看 log 的人會看到「就是這筆訂單、就是這個欄位、在這個函式裡是 undefined」，幾分鐘就能定位。

但如果你不 fail-fast，而是讓這個 `undefined` 繼續往下流呢？它會被傳進折扣計算，算出一個 `NaN`；`NaN` 被寫進訂單；訂單被送進結算管線；結算把 `NaN` 當成 0 處理，出了一張金額為零的發票；發票進了財報。三個禮拜後，一個會計發現帳對不平，開了一張 ticket。現在你要 debug 的不是「`amount` 為什麼是 undefined」，而是「為什麼三週前某張發票是零」——而那個 undefined 早就被五層處理碾過、面目全非了。**bug 沒有消失，只是被搬到了離成因很遠、且現場已被破壞的地方。** 這就是「壞狀態擴散」：你用「不崩」換來的，是「壞得更深、更難查」。

fail-fast 的本質，是一筆划算的交易：**用一次當下的、吵鬧的、容易定位的失敗，換掉一次未來的、安靜的、難以追溯的資料損壞。** 對 bug 而言，這筆交易幾乎永遠值得——因為 bug 代表你的假設已經破了，在一個假設破掉的狀態上繼續計算，每多算一步，產出的就是更多錯誤資料。

這裡有個操作上的關鍵，常被忽略：fail-fast 不只是「拋出例外」，而是「**拋出一個帶足夠 context 的、有型別的例外**」。一個 `throw new Error("something went wrong")` 幾乎和吞掉它一樣糟——它在錯誤的層失敗了，卻沒留下任何能 debug 的線索。好的 fail-fast 應該是 `throw new InvalidOrderError(order.id, "amount is undefined", { field: "amount", upstream: "pricing-svc" })`：明確的型別讓上層能依型別決定怎麼處理，附帶的 context 讓凌晨三點被叫起來的人不必重現問題就能讀懂發生了什麼。**fail-fast 的價值一半在「快」，一半在「失敗的那一刻留下了什麼」。**

值得借鏡的是 Erlang 把這個哲學推到極致的「let it crash」。Joe Armstrong 的論點是：與其在每一個可能出錯的地方寫防禦性程式碼、試圖讓單一行程在任何情況下都不死，不如讓出問題的行程**乾脆地死掉**，由一個獨立的 supervisor（監督者）來偵測它的死亡並重啟它。錯誤處理的責任，從「出錯的那個行程自己硬撐」轉移到了「一個結構上獨立、狀態乾淨的監督者」。這背後是同一個洞見：**在一個已經出錯、狀態可能已經污染的行程裡試圖『修復並繼續』，比讓它死掉、從一個已知乾淨的狀態重來，更危險。** fail-fast 不是放棄韌性——它正是**把韌性建立在『元件會乾淨地失敗』這個前提之上**。一個 supervisor 之所以能可靠地重啟 worker，正因為 worker 答應「出事我就乾淨地死，不留爛攤子」。

## 啟動期：最該 fail-fast 的時刻

有一個時刻，fail-fast 的紀律最該被守住、卻最常被破壞——**行程啟動的那一刻。**

設想一個服務啟動時要讀十個環境變數、連上資料庫、連上 Redis、載入一份設定檔。其中一個環境變數——比方說資料庫密碼——沒設。這時有兩種寫法。一種是 fail-fast：偵測到密碼缺失，**立刻拒絕啟動**，印一行清楚的錯誤後 `process.exit(1)`。另一種是「容錯」：log 一個 warning「DB password not set, continuing」，然後讓服務起來。

第二種寫法的災難在於：它把一個「啟動就該爆、十秒內就能發現」的設定錯誤，延後成了「**第一個真的要寫資料庫的請求**才會爆」的執行期錯誤。而那個請求可能在部署半小時後、在尖峰時段、帶著一個真實使用者的真實交易才到來。更糟的是，這個服務通過了 health check（它「活著」）、被掛進了負載均衡（見〈health check〉）、開始接收流量——然後對每一個碰到資料庫的請求回 500。你把一個本可在 CI 或部署瞬間就攔下的問題，變成了一場生產事故。

啟動期是 fail-fast 的黃金時刻，因為這時系統還沒有對外承諾任何事、還沒有處理任何真實流量、爆炸半徑趨近於零。一個帶病啟動的行程，是在用「看起來啟動成功了」這個假象，把炸彈的引信拉到未來、交到使用者手上。原則很簡單：**啟動期該驗的（設定、相依、憑證），全部在啟動期驗完；缺一不可的東西不齊，就拒絕啟動，而不是帶病上線**（設定與機密的管理見〈config 與機密管理〉）。

## 營運狀況：重試、降級、熔斷

跨過 bug 那條線，到了另一半——**預期內的營運錯誤**。對它們，fail-fast 反而是錯的。下游服務偶爾 503 不是你的 bug，把整個請求打掉只是把下游的一次小抖動，放大成你這邊的一次完整失敗。這一半的世界，目標從「暴露問題」反轉成「**吸收波動、維持可用**」。能做的事有三種，差別在於這個錯誤的性質。

**重試**，對付的是暫時性的、會自己好的錯誤。一次網路丟包、一次連線池瞬間耗盡、一次樂觀鎖衝突——重試一次往往就過了。但重試是一把有後座力的槍，扣扳機前必須先問兩個問題。第一，**這個操作冪等嗎？** 如果 `chargeCard` 失敗是因為「請求送到了、扣款成功了、但回應在回程丟了」，那麼天真地重試會**扣兩次款**。重試一個非冪等操作，是在用「修復一次失敗」的名義製造一次重複副作用——所以重試和冪等是綁在一起的一對（冪等的機制見〈重複是常態〉，重試的退避、jitter、上限做法見〈重試、退避與 jitter〉）。第二，**這個錯誤真的是暫時性的嗎？** 對一個永久性錯誤（餘額不足、請求非法）重試,不只徒勞，還會在下游已經掛掉時，把 N 個 client 的盲目重試疊加成一場 retry storm，把一個本來會自己恢復的下游徹底壓死（見〈retry storm 與 metastable failure〉）。**重試是給「可恢復且冪等」的錯誤的特權，不是萬靈丹。**

**降級（fallback）**，對付的是「失敗了，但有一個還能接受的次優答案」的情況。定價服務掛了，但你手上有它五分鐘前的快取值——回那個略舊的價格，比回一個 500 好。推薦服務掛了，但你有一份「熱門商品」的靜態清單——回那份通用清單，比讓整個首頁開天窗好。降級買到的是可用性：**局部的失敗，不變成整體的失敗。** 但降級有一個沉默的代價，這個代價兇險到值得單獨點明——**降級會掩蓋故障。** 如果你的 fallback 設計得太安靜，定價服務可能已經掛了三天，而沒有任何人發現，因為使用者一直拿到「還行」的快取舊價。系統的儀表板一片綠，因為「沒有人收到錯誤」——錯誤被 fallback 悄悄吸收了。**這是 fail-fast 那條線在容錯這一側的回聲：吞掉 bug 會讓 bug 隱形，安靜的降級會讓故障隱形。** 所以一條鐵律：**降級必須可觀測**——它每觸發一次，就該發出一個訊號（一個 metric、一個降級計數），讓「我們正在降級服務」這件事本身變成一個被監控的狀態，而不是一個沒人知道的祕密（fallback 的機制見〈fallback 與降級〉，可觀測性見〈三本柱〉）。

**熔斷（circuit breaker）**，對付的是「某個下游不是抖一下，而是持續地、大面積地掛掉」。這時連重試都是有害的——你每多打一個必然失敗的請求，都在白白佔用自己的執行緒、連線、逾時等待，把資源耗在一個已知會失敗的方向上。Michael Nygard 在 *Release It!*（2007）裡把這個機制命名為 circuit breaker：當對某依賴的失敗率超過門檻，就「跳閘」——之後的請求**立刻失敗，根本不去呼叫那個垂死的下游**。注意這裡發生了一件耐人尋味的事：**熔斷器其實是一種 fail-fast。** 它對「一個已知掛掉的下游」立刻失敗，正是為了不讓「等待逾時」這件事燒光自己的資源、把下游的故障**級聯**成自己的故障（cascading failure）。fail-fast 和容錯在這裡握手了：對個別請求快速失敗，恰恰是保全整體系統可用性的手段（三態與半開試探的機制見〈circuit breaker〉）。

## 傳播：在哪一層做決定

到這裡，所有「處理動作」（fail-fast、重試、降級、熔斷）都講過了。但有一個更基礎的問題藏在它們底下：**這些決定，該由哪一層來做？**

設想結帳請求的呼叫鏈：`HTTP handler → OrderService → PaymentClient → HTTP 客戶端`。最底層的 HTTP 客戶端收到一個 connection reset。它該怎麼辦？它**什麼都不該決定**——它根本沒有足夠的資訊去決定。它不知道這個請求是不是冪等的、能不能重試、有沒有 fallback、使用者願不願意等。它知道的只有一件事：「連線斷了」。所以它唯一正確的動作是**乾淨地往上傳播**——把這個錯誤如實往上拋。

「往上拋」聽起來簡單，但這裡藏著錯誤處理最常見的兩個反模式，它們是一體兩面。

第一個是**吞掉（swallowing）**：`catch (e) {}`，或者 `catch (e) { log(e); }` 然後若無其事繼續。這把一個「需要被處理的錯誤」變成了「一個沒人知道發生過的錯誤」。前面講 bug 時已經見過它最毒的形態——它把可恢復的失敗變成靜默的資料損壞。值得再強調一次它為什麼是錯誤處理裡**最危險**的反模式：所有其他錯誤都至少留下了一個失敗的訊號，唯獨被吞掉的錯誤連訊號都沒有。你不是在 debug 一個錯誤，你是在 debug 一個「看起來成功了、其實沒有」的幽靈。

第二個是**傳播時丟失 context**：`catch (e) { throw new Error("payment failed"); }`。這比吞掉好一點——至少錯誤還在往上走——但它砍斷了**原因鏈**。底層那個 `connection reset to gateway.acme.com:443`、那個 stack trace，全沒了。傳到最頂層時，你的 log 裡只剩一句乾巴巴的 `payment failed`，一個凌晨三點被叫醒的人面對它，等於從零開始查案。

正確的傳播是**包裝（wrap）而非取代**：每往上一層，都疊加上「我這一層知道、而下層不知道」的 context，同時保留下層的原因。底層說「連線斷了」，`PaymentClient` 包一層「對 gateway 的扣款呼叫失敗了，原因是：連線斷了」，`OrderService` 再包一層「訂單 #12345 的付款步驟失敗了，原因是：對 gateway 的扣款呼叫失敗，原因是：連線斷了」。到了最頂層的 handler，它終於**握有了完整的資訊**——它知道是哪筆訂單、卡在哪個步驟、根因是什麼——於是它**有資格做決定**：這個錯誤該回 503 給使用者並重試，還是該回 400 說輸入非法，還是該降級。

這就是傳播的全部意義：**錯誤應該一路往上冒泡，直到抵達那個『擁有足夠 context 能正確處理它』的層，由那一層做決定。** 底層的職責是「**忠實地報告，不要自作主張**」——不吞、不亂改、保留原因、疊加 context。把「決定」留給有資訊的人，把「報告」留給沒資訊的人，這是分層錯誤處理的根本紀律。

## 一個 worked example：四個下游的結帳請求

把所有切割線和動作收束到開場那個結帳請求上，逐個下游走一遍，看每一個錯誤分別落進哪個象限、走哪條路。假設 SLA 要求這個 API 的 p99 在 800 ms 內，每個下游的正常回應約 50 ms。

**庫存服務回 409 Conflict（庫存不足）。** 分類：永久性、非 bug（這是預期內的業務狀況——東西就是賣完了）。重試無意義（再問一次還是缺貨），降級無意義（不能假裝有貨）。正確路徑：**乾淨地傳播成一個對使用者的 4xx**——「此商品缺貨」。這不是錯誤處理的失敗，這正是錯誤處理的成功：一個營運狀況被乾淨地當成控制流回報了。

**定價服務逾時（500 ms 沒回應）。** 分類：暫時性、非 bug。先看能不能重試——但這裡 budget 很緊：光定價這次逾時就花掉 500 ms，離 800 ms 的天花板只剩 300 ms（還沒扣掉前面庫存呼叫的時間，實際剩餘更緊），重試一次（最壞再 500 ms）會直接撞破 SLA（timeout 與 budget 的傳遞見〈timeout、deadline 與 budget〉）。所以這裡更該走**降級**：用定價服務五分鐘前的快取值，並**發出一個降級 metric**。使用者拿到一個可能略舊但可用的價格，請求活了下來，而「我們降級了定價」這件事被記錄在案，不是一個祕密。

**風控服務回應一個我們沒見過的 enum 值 `RISK_LEVEL_UNKNOWN`。** 分類：這是一個 bug 訊號——要嘛是風控服務發布了新版、加了我們還沒處理的狀態，要嘛是序列化出了問題。它不該被 `default:` 分支默默當成「低風險」放行（那等於把一個未知狀態當成安全狀態，是把 bug 當營運狀況吞掉的典型）。正確路徑：**fail-fast**——對這一筆請求拋出一個帶 context 的錯誤（「unknown risk level from risk-svc: RISK_LEVEL_UNKNOWN, order #12345」），保守地拒絕這筆交易，並讓這個錯誤大聲到能觸發告警。寧可擋下一筆可疑交易，不可放行一筆我們其實不理解其風險的交易。

**資料庫寫入逾時（開場那一刻）。** 分類：暫時性、非 bug。但這裡有個致命的細節——**寫入是否冪等？** 如果這個 `INSERT` 沒有冪等保護（沒有唯一鍵、沒有冪等鍵），那麼「逾時」是最兇險的一種失敗：你**不知道**那筆寫入到底成功了沒有（可能寫成功了，只是 ack 在回程丟了）。盲目重試會寫出兩筆訂單。所以正確路徑取決於前置設計：若這個寫入有冪等鍵保護（見〈重複是常態〉），重試是安全的；若沒有，就**不能重試**，只能傳播成一個失敗、並依賴使用者或上游的「沒收到成功就重試 + 冪等」來兜住——而這恰恰反過來證明了，錯誤處理哲學和系統的冪等設計是同一件事的兩面：**你能對一個逾時做出的反應，被你當初有沒有把這個操作設計成冪等死死框住。**

把這四個放在一起看，最重要的領悟不是「每個都有對的做法」，而是：**四個錯誤,沒有任何兩個走同一條路。** 409 傳播、逾時降級、未知 enum fail-fast、DB 逾時看冪等再定——而它們在天真做法裡，全都會被那一句 `catch (e) { log(e); }` 碾成同一團。錯誤處理的功夫，九成在分類，一成在動作。

## 為什麼是這個形狀

退一步看，整套錯誤處理哲學，是從一個無法迴避的張力裡長出來的：**你永遠在用兩種好東西互相交換——可除錯性（debuggability）與可用性（availability）。**

fail-fast 買的是可除錯性。它讓問題暴露在離源頭最近、現場最完整的地方，代價是它**真的會讓請求失敗**——它犧牲了這一次的可用性，換來「壞狀態不擴散、根因好定位」。容錯（重試、降級、熔斷）買的是可用性。它讓系統在部分零件失靈時依然能提供服務，代價是複雜度，以及一個更陰險的代價——它**會掩蓋真實的問題**，讓底層的故障在一片「看起來還行」的降級結果下無聲地腐爛。

這兩者沒有普世最優解，因為它們交換的是兩種在不同錯誤上價值不同的東西。而決定該往哪邊倒的，正是那兩條切割線：**錯誤的根因在『我家』還是『外面』。** 我家的——bug、被違反的不變式、啟動期缺的設定、未知的狀態——fail-fast，因為對自己的錯誤,可除錯性遠比「再撐一下」值錢。外面的——下游的暫時性故障、第三方的限流、預期內的業務拒絕——盡量吸收，因為別人的一次抖動不值得拖垮你整個請求。

所以那句最誘人的「全都 try/catch 包起來不要崩」之所以錯，不是因為它太想要可用性，而是因為它**用一個動作回答了兩個相反的問題**。它對 bug 給了「再撐一下」，而 bug 最不需要的就是再撐一下；它對營運狀況也只給了「吞掉」，而營運狀況需要的是有結構的重試與降級。對所有錯誤一律 fail-fast 是脆弱，對所有錯誤一律吞掉是埋雷——兩者都是同一個錯誤的兩種長相：**拒絕先分類，就急著統一處理。**

回到凌晨三點，那個手裡握著資料庫逾時 error 的程式碼。它真正該做的第一件事，從來不是「決定怎麼處理」，而是先問清楚：你是誰？你是一次會自己好的網路抖動，還是一個寫了一半、狀態未知的危險窗口？你的根因在我這層，還是在資料庫那頭？——把這個問題回答對了，該 fail-fast 還是該重試、該傳播還是該降級，答案自己就浮出來了。錯誤處理哲學的全部，就是**逼自己在動手之前，先把錯誤看清楚**。
