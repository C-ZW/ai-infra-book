# 批次、coalescing 與 debounce

一段程式碼，從開發機搬到生產，邏輯一個字沒改，延遲卻多了兩個數量級。你打開 profiler，沒有慢查詢、沒有鎖競爭、CPU 也沒滿——時間全花在「等」上面。一個請求發了五十次資料庫往返，每一次都不慢，但五十個 round-trip 串起來，就是一百毫秒。或者反過來：一個搜尋框，使用者打一個字你查一次，他敲十個字你就把後端打了十次，前九次的結果還沒回來就被作廢。

這三種病，根子是同一個：**你在用很多次小操作，做本來該用少數幾次大操作完成的事。** 治法也是同一個動作——把多個小操作合併成較少的大操作。但合併的「觸發時機」有三種根本不同的語意，對應三個常被混用、混用就出錯的名詞：**批次（batching）**、**coalescing**、**debounce**。它們長得像，做的事卻不是一回事；選錯一個，你不是省了成本，而是悄悄吃掉了不該吃的資料。這一章把這三者拆開，講清楚每一個底下在做什麼、買到什麼、在哪裡會咬你。

## 為什麼「次數」本身就是成本

先把問題講難。直覺上，一次操作的成本應該正比於它搬的資料量——傳一筆和傳一百筆，差一百倍，合理。但真實系統裡幾乎不是這樣。每一次操作，無論搬多少資料，都背著一筆**與 payload 大小幾乎無關的固定成本**：

- 一次資料庫操作，要付網路 RTT、連線取得、查詢規劃（parse + plan）、結果序列化；若是寫入，還多一次 commit 的 fsync。做一筆和做一百筆，這些固定開銷一模一樣。
- 一次外部 API 呼叫，要付 TCP 握手（若沒復用連線）、TLS、HTTP header、對端的 routing 與 auth。body 裡放一個 id 還是一百個 id，這層開銷不變。
- 一次訊息 ack、一次磁碟 flush、一次系統呼叫——每一個都有它的入場費。

固定成本的存在，意味著操作的「次數」本身就是一條獨立的成本軸，和「資料量」那條軸正交。當每次操作的固定成本佔比很高（payload 很小、但開銷很重），把 N 次操作合併成 1 次，等於把那筆固定開銷除以 N。這就是這一整章所有機制的第一原理——**固定成本攤平**。再加上一個變體：有時候那 N 次操作根本是**重複的同一件工作**，合併就不只是攤平、而是直接消去 N−1 份白做的活，這是第二原理——**重複工作去除**。

天真的做法在哪裡破掉？它假設「一次操作的成本 ≈ 0」，於是放心地在迴圈裡發查詢、對每個事件各打一次後端、對使用者每個按鍵各觸發一次。在開發機上這個假設近似成立——本機資料庫 RTT 趨近零、N 又小，所以「秒回」。上了線，RTT 變成幾毫秒、N 從五變五百、後端還有負載，同一段碼慢上百倍，而且病根藏在 ORM 的屬性存取或某個 `for` 迴圈裡，從程式碼一眼看不出來。三種合併機制，就是在不同的觸發語意下，把這個被破壞的假設補回來。

## 批次：主動湊一車再發

批次是三者裡最直白的：**累積到「夠多」或「等夠久」，就把累積的這一整批一次處理掉。** 關鍵字是「主動」——批次是你刻意去湊量，為了攤平固定成本，哪怕這些操作之間毫無關係。

把它講透，先看一個手算到底的例子。你要把 10,000 筆事件寫進資料庫。

逐筆寫法：每筆一次 `INSERT`、一次 round-trip、一次 commit。設這筆固定成本是 1 ms/筆（網路往返加 commit，對小 payload 這已經很樂觀），純資料傳輸忽略不計。總時間 = 10,000 × 1 ms = **10 秒**。

批次寫法：每 500 筆湊成一批，發一次 `INSERT ... VALUES (...),(...),...`、跟一次 commit。批數 = 10,000 / 500 = 20 批。每批的固定成本還是那 1 ms，但這次它被 500 筆分攤；再算上這一批比較大、傳輸多花 1 ms，每批約 2 ms。總時間 = 20 × 2 ms = **40 毫秒**。

同樣的 10,000 筆、同一份結果，10 秒變 40 毫秒，差 250 倍。差距全來自把「commit 次數」從 10,000 砍到 20——固定成本被批量除掉了。而且 N 越大，逐筆寫法越慘、批次的優勢越誇張，因為固定成本那條軸是線性長的，批次把它的斜率壓平了。

但這 250 倍不是白拿的，它欠下三筆債，每一筆都得講清楚：

**第一筆債：延遲。** 「累積到 500 筆才發」意味著第 1 筆事件要等到第 500 筆來了才被寫出去。如果事件來得慢，第 1 筆可能要乾等很久。所以批次幾乎從來不能只用「湊滿 N 筆」這一個觸發條件——它必須是 **size-or-time**：累積到 N 筆，**或者**等了 T 毫秒，**先到的那個**就觸發。這兩個旋鈕一起調：size 控制吞吐上限，time 控制延遲上限（也就是「湊不滿也別讓第一筆等超過 T」）。

這個 size-or-time 窗不是學院概念，它就長在你每天用的東西裡。Kafka producer 的 `batch.size`（預設 16384 bytes）配 `linger.ms`，正是這對旋鈕：累積資料到 `batch.size` 就立刻送、否則最多 linger `linger.ms` 毫秒等更多訊息湊進來，哪個先到算哪個。值得一提的是 `linger.ms` 的預設值在 Kafka 4.0 從沿用已久的 **0 ms 改成了 5 ms**（2026-06）——官方等於是公開承認：用一點點刻意引入的延遲去換批次效率，在真實的網路 I/O 模型下，往往連端到端延遲都更低，而不只是吞吐更高。這是個反直覺但重要的點：**主動延遲一下下，整體反而更快**，因為省掉的 per-batch 開銷通常遠大於你等的那幾毫秒。

**第二筆債：失敗半徑放大。** 逐筆寫，一筆失敗就只壞一筆。批次寫，一批 500 筆若違反某個約束（unique 衝突、外鍵不存在），整批 `INSERT` 可能一起回滾——一顆老鼠屎壞了一整批。批量越大，單批失敗時要重做的成本越高、波及的筆數越多。所以批次的設計不能只想著「攤平」，還得想「壞了怎麼辦」：批要能**拆批重試**（整批失敗時對半切、二分找出毒訊息那一筆），單筆的副作用要**冪等**（拆批重試時同一筆可能被送兩次，見〈重複是常態〉），這樣重試才不會把已成功的部分又做一遍。

**第三筆債：批太大反而是另一種病。** 你可能想，那我把批開到 50,000 筆一次寫完不是更省？不行。一個 50,000 筆的 `INSERT` 是一個巨大的單一交易：它持有鎖的時間極長（擋住別人）、佔的記憶體與 WAL 極大、若失敗整批 5 萬筆全部重來。批量是個有最佳點的旋鈕，不是越大越好——它在「攤平固定成本的收益」和「單批的風險與資源佔用」之間取點。太小攤不平，太大放大失敗半徑、拖長鎖、撐爆交易。

## per-tick 批次：把延遲債還掉的那個變體

上面那筆「延遲債」——為了湊量得等——聽起來是批次的宿命。但有一個極漂亮的變體幾乎把這筆債還到了零：**per-tick 批次**，最有名的實作是 GraphQL 生態裡的 DataLoader。

它解的是一個具體的痛：GraphQL 的 resolver 對一個清單裡的每個元素獨立解析欄位，於是「列出 50 筆訂單、每筆顯示客戶名」會變成 1 次查訂單 + 50 次各查一個客戶 = 51 次往返的 N+1 災難（這個病本身見〈N+1 查詢〉）。你想批次合併這 50 次查詢，但你不能設一個固定的等待窗——使用者在等這個 HTTP 回應，你不能為了湊批讓他多等 100 ms。

DataLoader 的巧思是利用 event loop 的結構（見〈event loop〉）。它不設時間窗，而是攔截**同一個 event loop tick 裡**發生的所有 `load(id)` 呼叫，把它們收進一個佇列；然後用 `process.nextTick`（精確地說是 `enqueuePostPromiseJob`）排一個工作，在**下一個 tick** 把整個佇列的 key 合併成單一一次 `load([id1, id2, ..., idN])` 後端呼叫。關鍵在於：JavaScript 是單執行緒的，當前 tick 內那些同步、平行發出的 `load` 呼叫，它們之間根本沒有真實的時間流逝——下一個 tick「馬上」就到。所以這個合併幾乎**不引入任何延遲**，卻把 1+N 壓成了 1+1。

要在腦中重放它的運作，得釘住三個非顯然的契約，每一個錯了都會出微妙的 bug：

**批次函式的回傳必須與 key 嚴格等長、同序。** 你交給 DataLoader 一個 batch function，它收到 `[id1, id2, id3]`，必須回一個長度恰好為 3、且第 i 個值對應第 i 個 key 的陣列。但資料庫的 `WHERE id IN (1,2,3)` **不保證**按你給的順序回、也不保證每個 id 都有一行（有的 id 查無資料）。所以 batch function 內部幾乎一定要先把 DB 回來的結果**依 key 重新排序、缺的補上 `null` 或 `Error`**。漏了這一步，使用者 A 會拿到使用者 B 的資料——這是用 DataLoader 最容易踩、也最危險的坑，因為它靜默地對錯人。

**快取是 request-scoped，不能跨請求重用。** DataLoader 預設開快取（記憶化同一個 tick／同一個請求內對同一個 key 的重複 `load`），但這個快取的設計壽命就是「一個請求」。它的標準用法是**每個請求進來時 new 一個新的 DataLoader 實例**，請求結束就丟。如果你貪圖方便把一個 DataLoader 實例做成全域單例跨請求共用，它會把第一個請求查到的舊資料，餵給後面所有請求——而且因為它「看起來快取命中、很快」，你還會以為這是優化。

**只合併同一個 tick 內的呼叫。** 如果你的兩個 `load` 因為中間 `await` 了別的東西而落在不同 tick，它們不會被合併。所以「在巢狀 resolver 的每一層都要套 loader」：A→B→C 三層關聯，只在 A→B 加 loader、B→C 那層仍然逐筆，N+1 只解了一半。

DataLoader 也允許你把那個「下一個 tick 就發」的排程器換掉（`batchScheduleFn`），改成「等 100 ms 再發」——這時它就退化回了普通的 size-or-time 批次，重新欠下延遲債。預設的 per-tick 排程之所以是預設，正是因為它在「不加延遲」和「最大化合併」之間，靠 event loop 的結構拿到了一個近乎免費的甜蜜點。`maxBatchSize` 預設是 Infinity——它不主動限批量，因為它賭的是「一個 tick 內的 load 數量天然有界」，這跟 size-or-time 批次「主動湊大批」的哲學剛好相反。

## coalescing：撞在一起的同一件事，只做一次

批次是主動湊量，湊的可以是毫不相干的操作。coalescing 是另一回事：它是**被動去重**——當多個請求恰好在同一段時間內、要的是**完全相同的那一件工作**，就只讓一個去真的做，其餘的掛在它身上、共享它的結果。它攤平的不是固定成本，而是直接消去那 N−1 份**重複的工作本身**。

最經典的場景是快取擊穿（cache breakdown / cache stampede，也叫 thundering herd）。一個熱門的 key 在快取裡過期的那一瞬間，假設有 1,000 個並行請求同時來讀它。天真的 cache-aside 邏輯是「快取沒有？那我去後端撈、撈完寫回快取」——於是這 1,000 個請求**每一個**都發現快取空了，**每一個**都衝去後端撈同一份資料。後端瞬間被 1,000 個一模一樣的昂貴查詢打爆，而其中 999 個是純粹浪費——它們撈的是同一個東西（這個失效災難的全貌見〈快取的失效災難〉）。

coalescing 的解法叫 **single-flight**：維護一張「進行中請求表」，以 key 為索引。第一個請求發現表裡沒有這個 key 的進行中工作，它就登記自己、真的衝去後端做；接下來的 999 個請求發現表裡已經有人在做這個 key 了，它們**不再各自衝後端**，而是掛在第一個請求的 future／channel 上等。第一個做完，把結果同時交給所有掛著的人，再從表裡清掉這個 key。1,000 次後端呼叫，coalesce 成 1 次。

Go 的 `golang.org/x/sync/singleflight` 是這個機制的標準實作，它的 API 把語意講得很乾淨。`Do(key, fn)` 保證「對同一個 key，同一時間只有一次 `fn` 在執行；重複的呼叫者會阻塞、等原本那次完成、拿到同一份結果」。它回傳一個 `shared bool`，明白地告訴你「這個結果是不是被分給了多個呼叫者」——這對觀測很有用，你能量出 coalescing 實際救了多少重複工作。它還有兩個值得知道的方法：`DoChan` 是不阻塞的版本，回一個 channel 讓你 `select`；`Forget(key)` 主動把某個 key 從進行中表裡踢掉，讓後續呼叫重新去做、而不是繼續等那個可能已經出問題的 in-flight 呼叫。

`Forget` 的存在，正指向 single-flight 最陰險的故障模式，這個必須講透：**被選去做事的那個請求，如果 hang 住了，掛在它身上的全部一起 hang。** coalescing 把所有人的命運綁到了那「第一個」請求上——這是它省工作的代價，也是它的單點。設想第一個請求去後端撈資料，後端正好慢、或網路卡住、或它根本死了，那麼掛在它後面等的 999 個請求，會**一起**卡死，直到那第一個逾時或返回。coalescing 看起來像純粹的優化，但它同時悄悄製造了一個「命運共同體」的故障放大器。所以正確的 single-flight 必須：

- **給那個 in-flight 的工作設逾時**（見〈timeout、deadline 與 budget〉），不能讓它無界地拖著所有人。第一個請求等不到後端，要在有界時間內失敗，把等待的人放出來。
- **失敗的結果絕不寫進快取**。如果第一個請求撈失敗了，卻把這個失敗（空值、錯誤）寫回快取，那等於把一次失敗放大成「接下來一段時間所有人都讀到失敗」——快取擊穿的另一種長相。失敗就讓大家各自 fallback（見〈fallback 與降級〉），別污染快取。
- **只合併「同 key 且時間上重疊」的請求**。single-flight 不是去重快取，它只在「正在進行」這個窗口內收斂；一個請求做完、表清掉之後，下一個同 key 請求是新的一次 flight。它去的是「同時撞在一起」的重複，不是「先後出現」的重複（後者是快取的工作）。

值得停下來看一眼 coalescing 和批次的本質差別，因為混淆它倆會選錯工具。批次合併的是**不同的**工作（10,000 筆不同的事件），目的是攤平固定成本，所以它要主動「湊」、會引入等待。coalescing 合併的是**相同的**工作（1,000 個都要同一個 key），目的是消去重複，所以它「不湊」、撞上才合、理想上零延遲。一個是「把不同的東西打包省運費」，一個是「同一個東西別買一千次」。

## debounce：等它安定，只認最後一次

第三種合併的觸發語意又完全不同。debounce 處理的是**高頻、連續、且只有最終狀態才有意義**的觸發源——它的規則是：**每次觸發都重置一個計時器，只有當觸發停止、安靜超過一段約定時間之後，才真正執行一次。**

場景：一個搜尋輸入框，使用者打字搜尋。天真做法是 `onChange` 每觸發一次就查一次後端。使用者打「distributed」這個字，你就發了 11 次查詢（d, di, dis, dist...），前 10 次的結果在使用者還沒打完時就已經作廢，純粹是噪音，還白白把後端打了 10 次。debounce 的處理是：每個按鍵都把「真正去查」這件事往後推 300 ms 並重置；只要使用者還在連續打字（間隔小於 300 ms），查詢就一直被推遲、不執行；直到他停手 300 ms，才用**最後**那個完整的字串查一次。11 次觸發，debounce 成 1 次有意義的查詢。

debounce 常和它的雙胞胎 **throttle** 搞混，這兩個選錯，症狀是「該動的時候沒動」或「不該一直動卻一直動」，必須分清：

- **debounce**：等安定。連續觸發期間**完全不執行**，只在停止觸發後執行一次。它關心的是「最終狀態」——中間過程全部丟掉。搜尋輸入、視窗 resize 完才重排、自動儲存（停止編輯才存）、檔案監看（一批檔案變動完才重建）都是 debounce。
- **throttle**：限頻率。在固定的時間窗內**最多執行一次**，不管觸發多密集——它不等安定，而是保證「每隔 T 一定動一次」。捲動時更新進度條、限制對下游的呼叫頻率、滑鼠移動取樣都是 throttle。

一個漂亮的視角是：**throttle 其實就是「帶上限的 debounce」**。在 Lodash 的實作裡，`throttle` 字面上就是用 `debounce` 加一個 `maxWait` 參數做出來的。這直接點出了 debounce 自己最大的故障模式——**無上限的 debounce 可能永遠不執行。** 如果觸發源持續不斷地、間隔總是小於你的 debounce 窗（使用者一直打字停不下來、事件流永不安定），那麼計時器被無限重置，「真正執行」這件事被無限往後推，搜尋永遠不發生。所以生產級的 debounce 幾乎都要配一個 `maxWait`：保證「就算你一直觸發，最遲每隔 maxWait 也一定強制執行一次」——而這，恰好就是 throttle 的行為。debounce 和 throttle 不是對立的兩個東西，是同一個旋鈕的兩端，maxWait 是把前者拉向後者的那根桿。

但 debounce 有一個紅線級的誤用，比上面的故障模式更該記死：**debounce 會主動丟掉中間的觸發。** 這對「只認最終狀態」的場景是特性，對「每一次觸發都有獨立意義、都必須被處理」的場景就是**資料遺失**。把 debounce 套到「每一筆付款事件」「每一筆下單」「每一條必須落地的訊息」上，等於主動把中間那些事件吃掉——使用者連續下了三單，debounce 只處理了最後一單，前兩單憑空消失。判準很簡單：問自己「中間那些觸發，丟掉行不行？」搜尋輸入丟得起（中間的半截字串本來就沒用），交易事件丟不起。丟不起的，就絕對不能 debounce——該用佇列把每一筆都收下來（哪怕慢慢處理），而不是用 debounce 把它們抹掉。

throttle 也有對稱的版本：它在每個窗口內**丟棄**多餘的觸發。如果那些被丟的觸發不能丟，throttle 同樣不適用，得改用佇列承接。合併機制省成本的前提，永遠是「被合併掉的那些，真的可以被合併」——攤平不同的工作、去重相同的工作、丟棄無意義的中間態，這三個前提各自只在它對應的場景成立。

## 一個藏在 TCP 裡的合併，以及它的著名陷阱

合併不是應用層的專利，它一路滲到協定底層，而且底層那個合併出過一個經典到值得每個後端工程師知道的事故，因為它把「兩個各自合理的合併機制撞在一起會出鬼」這件事演示得淋漓盡致。

TCP 裡有個 **Nagle 演算法**：為了避免把一堆只裝一個位元組的小封包（payload 1 byte、header 40 bytes，效率低到荒謬的「tinygram」）灑滿網路，Nagle 規定——當你有資料還沒被對方 ACK 確認時，先別急著送下一小段，把後續的小寫入**攢起來**，等湊夠一個 MSS 或等到前面的資料被 ACK 了再一起送。這就是一個合併：用「攢一下」去換「少發幾個低效小封包」，本質上和應用層批次同源。

問題出在它和另一個各自也很合理的機制——**delayed ACK**——撞在一起。delayed ACK 是接收端的省事策略：收到資料先別急著回 ACK，稍等一下（Linux 上 `TCP_DELACK_MIN` 是 40 ms、上限 `TCP_DELACK_MAX` 200 ms），看看能不能把 ACK 捎在自己要回的資料裡一起送，省一個純 ACK 封包。

把這兩個合併放一起，會卡出一個經典的**互相等待死結**：發送端因為 Nagle，要等前面資料的 ACK 才肯送下一段；接收端因為 delayed ACK，要等更多資料或計時器到期才肯送 ACK。於是雙方乾瞪眼——發送端等 ACK、接收端等資料——直到接收端的 40 ms delayed-ACK 計時器超時、被迫送出 ACK，發送端才解凍。結果就是一個請求平白多了 **40 毫秒**（最壞可達數百毫秒）的延遲，而你的程式碼、你的查詢、你的網路頻寬，**全都沒有問題**。這個延遲不是任何一方的 bug——是兩個各自正確的合併機制，在特定的「寫-寫-讀」流量模式下，把彼此的等待條件鎖死了。

解法是 `TCP_NODELAY`——關掉 Nagle，小封包來了立刻送、不攢。對於「小訊息、低延遲優先」的場景（互動式終端、RPC／gRPC 小請求、資料庫協定客戶端），這幾乎是預設要開的。這個故事的寓意超出 TCP 本身：**合併機制省成本，靠的是「攢一下」的延遲；而當系統裡有多個各自在攢的環節，它們的延遲會疊加、甚至會像 Nagle × delayed-ACK 這樣互相鎖死。** 你在應用層加了批次、傳輸層有 Nagle 在攢、下游又有它自己的緩衝——這些「為了效率而引入的等待」一旦串起來，端到端延遲可能遠超你對任何單一環節的預期。每加一層合併，都要問：它攢的那點延遲，會和別處的延遲怎麼疊？

## 為什麼是這三種形狀

退一步看，這三個機制不是三個任意的工具，而是「把小操作合併成大操作」這個單一意圖，在三種不同的觸發語意下分岔出來的三條路——而觸發語意之所以分成這三種，是因為「被合併掉的東西」有三種不同的身分。

當被合併的是**一批彼此不同、但都得做**的操作，你要的是攤平它們共有的固定成本，於是你**主動湊量**——這是批次，它的代價是為了湊而等的那段延遲，它的旋鈕是 size 和 time 兩個上限。當被合併的是**一群恰好撞在一起、其實是同一件**的工作，你要的是消去那些重複，於是你**讓第一個去做、其餘搭便車**——這是 coalescing，它的代價是所有人的命運綁在第一個身上，它的紅線是給那第一個設逾時、別讓失敗污染快取。當被合併的是**一串連續抖動、只有最終態才算數**的觸發，你要的是抑制中間的噪音，於是你**等它安定、只認最後一次**——這是 debounce，它的代價是中間態被丟棄，它的紅線是別把它套到每一筆都不能丟的事件上。

三者可以疊加：一個系統完全可以一邊用 single-flight 去重熱 key、一邊把去重後的真實查詢攢成批次、再對使用者的輸入做 debounce。但每疊一層，你就多攢了一段延遲、多接受了一條「被合併掉的東西真的可以被合併」的前提。合併從來不是免費的優化——它是拿延遲、拿失敗半徑、拿「中間態」去換固定成本的省下。看清楚你換掉的是哪一樣，才知道這筆交易在你的場景裡划不划算。
