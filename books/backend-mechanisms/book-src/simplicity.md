# 簡單性與過度工程：YAGNI 的兩面

兩個工程師在白板前盯著同一個需求：「做一個寄通知的功能，現在只要寄 email。」第一個人三十分鐘寫完——一個函式，收件人、主旨、內文，呼叫 SMTP，結束。第二個人皺眉，說：「以後一定也要寄簡訊、推播、Slack 吧？我先抽一個 `NotificationChannel` 介面，加一個 provider registry，再用一張設定表決定哪種事件走哪些 channel，這樣將來加一種只要實作一個 class。」一週後，第二個版本上線了：三層抽象、一個外掛機制、一份 YAML 設定，而它實際在做的事，和第一個人那三十分鐘的函式一模一樣——寄 email。簡訊那天始終沒來；半年後產品轉向，連 email 通知都被砍了。

這兩個版本都「能跑」，但它們在系統裡的重量天差地別。第一個是一塊可以隨手搬動的磚；第二個是一座為了一輛永遠不會來的卡車而蓋的橋。這一章要講的，就是怎麼分辨這兩者——以及為什麼這個判斷比它看起來難得多。

## 過度工程的本質：替還沒出現的需求付帳

先把詞定義精準。**過度工程（over-engineering）不是「寫得太好」，而是替一個還沒出現、甚至可能永遠不會出現的需求，預先付出複雜度。** 它幾乎總是以兩種面貌登場。

一種是**過早抽象**：因為「將來可能要支援多種 X」，先蓋一層通用的抽象層，結果到頭來只有一種 X。上面那個 notification 例子就是典型——介面、registry、外掛點全都是為了「多種 channel」這個臆測，而臆測沒兌現，留下的是憑空多出來的、每個讀程式碼的人都得先穿過的一層。

另一種是**過早最佳化**：因為「將來可能很慢」，先寫複雜的快取、平行化、或某段位元級的微調，結果系統真正的瓶頸根本不在那裡。你把一段本來三行就講完的邏輯，寫成了帶手寫快取失效、帶併發控制的兩百行，而那段程式碼一天被呼叫十次、每次零點幾毫秒。

對抗這兩者的，是兩句在工程圈流傳了幾十年的話。一句是 **YAGNI——You Aren't Gonna Need It**，源自 1990 年代末 Kent Beck 在 Chrysler C3 專案上提出的 Extreme Programming：當同事 Chet Hendrickson 一條條列出「系統將來會需要的能力」時，Beck 對每一條的回答都是「你不會需要它」，後來寫進了 1999 年的《Extreme Programming Explained》。另一句更老、也更常被引用：**「過早最佳化是萬惡之源」**，出自 Donald Knuth 1974 年發表在《ACM Computing Surveys》上的〈Structured Programming with go to Statements〉。

但這兩句話都長期被斷章取義，而被砍掉的那半句，恰恰是這一章真正的重點。

## Knuth 沒被引用的那半句

「過早最佳化是萬惡之源」幾乎成了拒絕一切效能考量的擋箭牌。可是 Knuth 那段話的完整版，講的是一個比口號精細得多的判斷：

> 程式設計師浪費了大量時間，去擔心程式裡不關鍵部分的速度，而這些對效率的嘗試，一旦把除錯與維護算進來，其實是強烈的負面影響。我們該忽略小的效率，約 **97% 的時間**：過早最佳化是萬惡之源。**但我們不該放過那關鍵的 3% 的機會。**

關鍵在後半句。Knuth 不是說「別最佳化」，他說的是「**別在錯的地方最佳化，但一定要在對的地方最佳化**」。97% 的程式碼不值得你為它的速度操心，可是有 3% 是熱路徑、是真正決定系統會不會被拖垮的地方——那 3% 你不只可以、而且應該錙銖必較。

而怎麼認出那 3%，本身就是這條紀律的另一半：**用測量，不用猜。** 拿 profiler 或壓測對著代表性負載跑一遍，看延遲真正堆在哪、哪段被呼叫得最兇，讓真實數據指出熱路徑，再動手最佳化那裡——而不是憑直覺認定某段「看起來很慢」。這和抽象側的紀律是同一個形狀：抽象要等到第三個真實樣本才校準得準，最佳化也要等到測量證實了瓶頸才下手。兩邊都是同一句話——**先有真實的證據，再付複雜度**；差別只在一邊的證據是重複出現的需求，另一邊的證據是 profiling 上凸起的那一段。

於是「簡單性」的原則浮現出它真正的形狀，不是「永遠寫最笨的版本」，而是：**複雜度要花在已被證實的需求上，不花在臆測的需求上。** 那段被快取糾纏的兩百行，問題從來不是「它最佳化了」，而是「它在沒有任何證據顯示這裡是瓶頸的情況下就最佳化了」。判斷的分水嶺不在「複不複雜」，而在「這份複雜度對應的是一個**已知的事實**，還是一個**臆測的未來**」。

這也順手澄清一個常見的混淆：**簡單不等於程式碼短。** 一行密不透風、塞滿三元運算子與位元技巧的「聰明」寫法，認知負擔可能遠高於五行直白的版本。簡單指的是**認知負擔低、容易理解與修改**——它是給未來要讀這段程式碼的人的，不是給行數計數器的。有時候為了真正的簡單，你反而得多寫幾行，把隱晦換成明確。明確勝過聰明。

## 為什麼這個賭注的期望值通常為負

把過度工程拆開來看，它本質上是一場賭局：**你賭未來會長成你現在猜的樣子。** 賭對了，省下一點將來的重做；賭錯了，賠上一整層沒人需要、又難以拆除的複雜度。要判斷該不該下這個注，得先看清楚兩件事：賠率有多差，以及輸了之後賠的到底是什麼。

賠率為什麼差？因為**人對未來需求的猜測，準確率出奇地低。** 預期將來要支援簡訊，結果產品方向轉了；預期某張表會長到上億筆需要分片，結果它停在十萬筆；預期這個 API 會有三種呼叫方，結果永遠只有一種。需求不照劇本走，是常態而非例外。一場賭局，當你贏的機率本來就低、而輸的代價又遠大於贏的回報時，它的期望值自然為負。

那輸了之後，賠的具體是什麼？Martin Fowler 把一個「臆測功能」的成本拆成四筆帳，值得逐筆看，因為其中兩筆是大多數人沒算進去的：

- **建造成本（cost of build）**：你花在分析、寫、測那個沒人要的功能上的時間。這筆最直觀，也最常被當成唯一的一筆。
- **延遲成本（cost of delay）**：你花在這個臆測功能上的時間，本來可以拿去做一個**已知該做**的功能。那個真功能晚兩個月上線、晚兩個月開始產生價值——這是看不見、卻真實的機會成本。
- **負荷成本（cost of carry）**：這才是最毒的一筆。臆測功能加進系統的那層複雜度，會讓**之後每一次修改與除錯**都更費勁。它不是付一次就結束，而是像利息一樣，每一個碰到它的工程師、每一次要繞過它的改動，都在持續付費。前面那座為不存在的卡車蓋的橋，從蓋好那天起，每個過橋的人都得多走一段。
- **修復成本（cost of repair）**：當你終於真的需要那個功能時，往往發現當初猜的形狀是錯的——於是你得先把舊的拆掉或繞開，再蓋對的。早做反而比晚做更貴。

把這四筆帳擺在一起，過度工程的賬本就清楚了：**你用「現在多付建造成本＋持續付負荷成本」，去買一張「未來可能省一點重做」的彩券，而這張彩券中獎率很低。** 簡單性買到的，正好相反——它買的是**改變方向的自由**。程式碼越少、抽象越少、巧思越少，未來要轉向時要拆的東西就越少。在一個你大機率猜錯未來的世界裡，保留轉向的自由，幾乎總是比押注某個特定未來更值錢。

## 錯的抽象，比重複更貴

過度工程裡最反直覺、也最該講透的一點，是**過早抽象的代價，常常高於它想消除的重複。** 這違反很多人從小被灌輸的 DRY（Don't Repeat Yourself）直覺——重複是壞的、抽象是好的，看到兩段像的程式碼就該合併。但這個直覺在「時機未到」時會反咬一口。

把過程慢動作播一遍。工程師 A 看到兩處相似的程式碼，抽出一個共用函式，乾淨俐落。過一陣子，需求 B 來了，它幾乎一樣、但有一點不同——於是有人給那個共用函式加一個參數、加一個 `if`。再過一陣子需求 C、需求 D，每來一個就再加一個參數、一個分支。最後那個當初「乾淨」的抽象，變成一團佈滿條件判斷的怪物：它對外宣稱自己是「一個通用的東西」，內部卻在用一堆旗標假裝自己是好幾個不同的東西。每次你要改其中一個變體的行為，都得小心翼翼確認不會弄壞被它綁在一起的另外三個變體。

這就是 Sandi Metz 那句被反覆引用的話的由來——**「重複，遠比錯的抽象便宜」**。重複的程式碼至少是誠實的：兩段相似但獨立的程式碼，你改一段不會驚動另一段，它們各自說自己的話。而錯的抽象是有毒的：它把本來不該綁在一起的東西綁在了一起，讓每個變體都被其他變體拖住。

更殘酷的是修復方向。當你發現抽象錯了，最快的前進方式是**往回走**——把抽象拆掉，將被抽走的程式碼**內聯（inline）回每一個呼叫點**，讓每個呼叫者只留下它自己真正需要的那幾行。Metz 說得精準：「當抽象是錯的，最快的前進方向是後退。這不是撤退，是換個更好的方向前進。」可是這條退路在實務上往往走不了——因為那個錯抽象已經被十幾處依賴、被測試纏住、被新人當成「既有設計」尊重，於是大家寧可繼續往那團怪物上疊參數，而不敢動它的根。錯的抽象一旦長進系統，拆它比當初不抽象難得多。

那什麼時候才該抽象？一條流傳已久的經驗法則是**三的法則（rule of three）**——由 Don Roberts 提出、經 Martin Fowler 在《Refactoring》裡推廣：「第一次做某件事，就直接做；第二次做類似的事，你會對重複皺眉，但還是把它複製一遍；第三次做類似的事，你才重構。」為什麼要等到第三次？因為**前兩次的重複，比一個抽錯的抽象便宜；而且要到第三個樣本，你才真正看清楚這幾段程式碼的「共通模式」到底長什麼樣。** 只看兩個點，你能畫出無數條曲線穿過它們；有了第三個點，曲線的形狀才開始被釘住。抽象需要足夠的真實樣本來校準，而不是憑兩個點就外推。

## 簡單不是逃避，分清臆測與已知

到這裡很容易把 YAGNI 讀成「永遠不要做任何前瞻設計」——這是另一個方向的災難，而且 Fowler 本人對此講得最清楚：**YAGNI 只適用於「為臆測功能而預先建造的能力」，它不適用於「讓軟體更容易修改」的努力。** 重構、自動化測試、清楚的模組邊界、把不可逆的選擇藏在介面後面——這些不是過度工程，恰恰是 YAGNI 能成立的**前提**。因為你打算「需求真的來了再演化」，你才更需要一個容易演化的底子。簡單性的賭注（賭你猜不準未來）和可演化性的投資（讓猜錯時改得動）是同一枚硬幣的兩面，不是對立的。

所以真正的判斷，從來不是「複雜 vs 簡單」，而是**「這個需求是臆測的，還是已知近期確定的」**。看兩個情境就懂這條線劃在哪。

**情境一：一個內部報表服務，目前只服務自家後台。** 有人提議「先把它做成多租戶、加上租戶隔離、計費計量、配額管理，將來要對外賣 SaaS 就現成」。問下去：對外賣的決定拍板了嗎？沒有，只是「老闆提過一句」。客戶是誰、計費怎麼算、隔離要到哪個層級，全都沒有答案。這是**純臆測**——你連那個未來的形狀都還看不清，現在蓋的多租戶架構幾乎一定是錯的形狀（回到修復成本那筆帳）。判斷：YAGNI，砍掉，先把單租戶的報表做好做穩。等對外賣真的拍板、第一個外部客戶真的出現，你會用真實的需求重做一個對的版本，而那比現在憑空猜一個快、也對。

**情境二：同一個服務，但這次要決定「報表的資料格式」——存進資料庫、也要回傳給前端。** 有人說「先用最簡單的格式，將來不夠再改」。但這裡要停一下：資料格式是**寫進儲存、外溢給下游**的東西。一旦線上累積了幾個月的資料、前端依賴了某個欄位結構，要改格式就得做資料遷移、得通知所有下游、得處理新舊並存——這是一道**單向門**（見〈兩扇門：可逆與不可逆決策〉），撤銷成本極高。對這種事，前期多想一步、把格式設計得能向後相容（留擴充欄位、用明確的版本標記），**不是過度工程，是負責任**。

兩個情境的差別，不在「複雜度高低」，而在**那份複雜度對應的是不是一個事後再加成本極高的單向門**。臆測的、可逆的，用 YAGNI 砍；已知近期的、或不可逆的，前期想清楚。Knuth 的「關鍵 3%」、Fowler 的「不適用於易修改性」、單向門的不可逆性——講的其實是同一件事：**簡單性的紀律，是把複雜度從「臆測且可逆」的地方撤走，集中投到「已證實或不可逆」的地方。** 它從來不是一刀切的「少做」，而是「把有限的複雜度預算，花在刀口上」。

## 怎麼認出已經過度工程的系統

事後諸葛容易，難的是當下認出來。幾個相當可靠的訊號，看到就值得停下來問一句「這真的需要嗎」：

- **抽象層底下只有一個實作。** `PaymentGateway` 介面後面只接著一個 `StripeGateway`，而且接了兩年都沒出現第二個——那層介面在替一個從未到來的第二實作買保險。
- **設定開關從沒被切換過。** 一個 feature flag、一個 `if config.useNewAlgorithm`，自上線那天起就固定是同一個值，沒人動過——它只是把死的常數，偽裝成了活的選擇，徒增一條沒人走的分支。
- **為「擴展性」加的外掛點，從沒接過第二個外掛。** plugin 機制、event hook、strategy registry，設計時想著「將來方便擴充」，結果擴充點上永遠只掛著當初那一個。
- **最佳化的程式碼，對著一個壓測證明不是瓶頸的地方。** 那段帶手寫快取的複雜邏輯，profiling 顯示它佔總延遲的 0.3%——它在 Knuth 說的 97% 裡，卻被當成那 3% 來伺候。

這些訊號的共通點，是它們都在為一個**沒有兌現的「將來」**服務。看到它們，砍掉通常是淨賺——你移除的不只是幾行程式碼，是每個後來者都得繞過的那份理解成本。

但要小心同一把尺的反面，別把 YAGNI 變成偷懶的藉口。明知道下個 sprint 就要支援的東西，硬說「現在還不需要」而不留演化空間，那不是紀律，是把短視包裝成原則。也別把「簡單」當成「可以不寫測試、不處理錯誤、不劃清邊界」——那些不是複雜度，是品質，砍掉它們不會讓系統更簡單，只會讓它更脆。判準永遠回到那條線：**你省掉的，是一個臆測的未來，還是一個已知的責任？**

## 為什麼是這個形狀：複雜系統從簡單系統長出來

退一步看，簡單性之所以是個值得守的紀律，根子上有一條觀察，John Gall 在 1975 年的《Systemantics》裡講得最透——後來被稱作 **Gall's Law**：

> 一個能運作的複雜系統，無一例外，都是從一個能運作的簡單系統演化來的。一個從零設計的複雜系統，永遠不會運作，也無法靠打補丁讓它運作——你必須從一個能運作的簡單系統重新開始。

這句話為整章提供了第一原理。過早抽象、過早最佳化、那座為不存在的卡車蓋的橋，本質上都是在嘗試 Gall 說「永遠不會運作」的那件事：**從零設計一個複雜系統，去迎合一個你還沒真正理解的未來。** 而你之所以還沒理解那個未來，正是因為它還沒來——你手上沒有真實的需求、真實的負載、真實的第三個樣本來校準你的設計。你猜的形狀，幾乎註定是錯的形狀。

簡單性不是審美偏好，也不是懶惰的託辭。它是對「我猜不準未來」這個事實的誠實回應。先做一個能運作的簡單版本，讓它在真實世界裡跑、暴露出真實的壓力點、累積出真實的樣本——然後**順著真實的需求**，一步步把複雜度加在被證實該加的地方。這條路長出來的複雜系統會運作，因為它的每一分複雜度都是被現實逼出來的，不是憑空想像出來的。

所以下次當你站在白板前，面對那句「以後一定也會需要……」時，真正該問的不是「這個功能複不複雜」，而是「**這個『以後』，是我已經看見的，還是我猜的**」。看見的，留好演化的空間；猜的，先放下。簡單，不是因為你做不出複雜的東西，而是因為你誠實地承認——關於未來，你知道得比想像中少。
