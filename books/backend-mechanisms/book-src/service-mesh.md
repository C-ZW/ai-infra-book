# Service mesh：把網路關注點下沉到 sidecar

設想一個正在長大的系統。一開始只有三個服務，彼此用 HTTP 互相呼叫，相安無事。你在訂單服務裡寫了一段很合理的客戶端邏輯：每次呼叫庫存服務，設一個 800 毫秒的逾時、失敗重試兩次、加上指數退避與 jitter。這段碼不長，工作得很好。

問題從第四個服務開始。它是另一組人用 Go 寫的，他們也呼叫庫存服務，也寫了重試——但忘了設逾時。第五個服務是 Python 寫的，重試了，退避卻是固定間隔、沒有 jitter。等到系統長到二十個服務、三種語言，「服務怎麼呼叫另一個服務」這件事已經有二十份**互不一致**的實作散落各處：有的逾時設 800ms、有的設 30s、有的根本沒設；有的重試只對冪等請求開、有的見錯就重試。沒有人有惡意，每一份單看都合理，但合在一起，這個系統對「網路出錯時該怎麼辦」這個問題，給出了二十個不同的答案。

更糟的是，這些邏輯**和業務邏輯纏在一起**。庫存服務的重試碼，就寫在處理訂單的函式旁邊。哪天你想要全公司統一「對下游一律先試 mTLS、憑證每小時輪換一次」，你得去改二十個服務、三種語言、發二十次版。在這套散落的世界裡，任何一條橫切的網路策略，都是一次跨團隊、跨語言的大遷徙。

Service mesh 想解的，就是這件事：**把「服務之間怎麼通訊」這層橫切關注點，從每個應用程式裡抽出來，下沉成一層獨立的、由平台統一管理的基礎設施。** 它的賭注是——mTLS、重試、逾時、熔斷、負載均衡、流量路由、可觀測性，這些東西本質上和你的業務無關，不該由每個服務、每種語言各刻一遍。

## 為什麼函式庫補不了這個洞

最直覺的修法，是別讓每個團隊各寫各的，而是寫一個**共用函式庫**：把逾時、重試、mTLS、負載均衡全包進去，每個服務引入它、照規矩呼叫下游。Google 的 gRPC、Netflix 早年的 Hystrix/Ribbon 走的就是這條路，而且在單一語言、紀律好的組織裡，它運作得相當好——少一層代理、沒有額外的網路跳轉、延遲最低。

但函式庫這條路有兩道天生的牆，規模一上來就撞得鼻青臉腫。

**第一道牆是語言。** 函式庫是綁語言的。你用 Java 寫了一套完美的 mesh 函式庫，那 Go 團隊、Python 團隊、Rust 團隊呢？要嘛每種語言各維護一份（於是「統一策略」又碎成 N 份，繞回原點），要嘛逼所有人用同一種語言（在一個有歷史的組織裡近乎不可能）。微服務的賣點之一就是各團隊能選自己的技術棧，而函式庫式的治理，恰恰和這個賣點對著幹。

**第二道牆是升級。** 假設你發現重試邏輯有個 bug，會在某種錯誤碼下無限重試。修好了——現在你得讓**每一個**引入這個函式庫的服務升級版本、重新編譯、重新部署。在一個有上百個服務、各自有自己發版節奏的系統裡，「讓所有人同時升到新版函式庫」是一場永遠跑不完的馬拉松。你想推一條新的網路策略，它的生效速度，等於你最慢的那個團隊願意發版的速度。

退一步看這兩道牆，會發現它們指向同一個結構性問題：**只要網路邏輯還活在應用程式的行程裡，它就繼承了應用程式的所有耦合**——綁語言、綁編譯、綁發版週期。要徹底解開，唯一的辦法是把它**搬出行程**，放到一個和應用程式語言無關、能被獨立升級的地方。這就是 sidecar 的全部動機。

## 機制登場：在每個 Pod 旁邊塞一個代理

Sidecar 模式的核心動作只有一句話：**在每個應用程式實例旁邊，放一個獨立的網路代理，把這個實例所有進出的流量，都先穿過這個代理。**

在 Kubernetes 的世界裡，這個「旁邊」非常具體——同一個 Pod 裡，除了你的應用容器，再注入一個代理容器（最常見的是 **Envoy**，一個 CNCF 的高效能 L7 代理）。同一個 Pod 共享同一個網路 namespace，於是可以用 iptables 規則（ambient 模式變的是代理層級——從每 Pod 一個全功能 L7 代理改成每節點一個輕量 L4 代理，後面會講；攔截本身仍走 iptables/Geneve，1.18 起另可選 eBPF）把這個 Pod 的所有 inbound 與 outbound 流量，悄悄重導向到 Envoy：

```
            Pod A                                Pod B
  +-----------------------+            +-----------------------+
  |  app container        |            |  app container        |
  |    | calls B          |            |    ^ receives         |
  |    v (localhost)      |            |    | (localhost)      |
  |  +-----------------+  |   mTLS     |  +-----------------+  |
  |  | Envoy sidecar   |==|============|=>| Envoy sidecar   |  |
  |  +-----------------+  |  encrypted |  +-----------------+  |
  +-----------------------+            +-----------------------+
       ^                                        ^
       |  xDS (config push)                     |  xDS
       +------------------+   +------------------+
                          |   |
                     +----------------+
                     | control plane  |   (e.g. istiod)
                     +----------------+
```

從應用程式的角度看，它什麼都沒變——它以為自己在直接呼叫庫存服務，程式碼裡還是 `http://inventory-service/...`。但實際上這個請求一出 socket，就被 iptables 攔下、交給同 Pod 的 Envoy。Envoy 才是真正做事的那個：它替應用查出庫存服務現在有哪些健康實例、挑一台、跟對方 Pod 的 Envoy 建立 mTLS 加密通道、套用逾時與重試策略、記錄這次呼叫的 metric 與 trace span。對端的 Envoy 收到後解密、轉交給它隔壁的應用容器。應用程式自始至終，**一行網路治理碼都沒寫**。

這就是 mesh 的兩層架構，值得把名字釘清楚：

- **資料面（data plane）**：真正攔截並處理每一筆流量的那些代理（每個 Pod 的 Envoy）。它在資料路徑上，每個請求都經過它。
- **控制面（control plane）**：它**不碰**任何業務流量，只負責一件事——把設定下發給所有資料面代理。誰有哪些健康端點、路由規則是什麼、用哪張憑證、套什麼策略，全由控制面算好、推給每個 Envoy。Istio 的控制面叫 **istiod**。

把資料面和控制面分開，是理解 mesh 為什麼能成立的關鍵。**策略是集中決定的（控制面），但執行是分散進行的（每個 sidecar）。** 你在控制面改一條規則——比如「對 payment 服務的呼叫逾時改成 500ms」——控制面把新設定推給所有相關的 Envoy，幾秒內全叢集生效，而**沒有任何一個應用程式需要重新編譯或發版**。函式庫那道「升級牆」，就是這樣被繞過去的：被升級的是 Envoy 與它的設定，不是你的應用。

## 控制面怎麼把設定餵給代理：xDS

控制面到資料面之間，靠的是一組叫 **xDS** 的協定（x Discovery Service 的縮寫，x 是萬用字）。它的精髓是 Envoy 的設定**不是寫死在啟動檔裡、改一次要重啟一次**，而是**運行中由控制面動態推送、即時生效、不需重啟**。這對一個流量代理至關重要——後端實例隨時在擴縮、漂移、死掉重生，代理手上那份「誰還活著」的名單必須能秒級更新。

xDS 不是一個 API，而是一族，各管一塊設定：

- **LDS（Listener Discovery Service）**：Envoy 該監聽哪些 port、每個 port 上掛什麼 filter 鏈。
- **RDS（Route Discovery Service）**：收到一個請求後，照什麼規則路由（依 path、header、權重…）。
- **CDS（Cluster Discovery Service）**：有哪些上游服務群（cluster），各自的負載均衡演算法、熔斷門檻是什麼。
- **EDS（Endpoint Discovery Service）**：每個 cluster 背後具體有哪些 IP:port 的健康實例。這是變動最頻繁的一塊——一個 Pod 死了，控制面就透過 EDS 把它從清單拔掉，秒級推給所有相關代理。
- **SDS（Secret Discovery Service）**：憑證與私鑰怎麼發。短效期的 mTLS 憑證就靠它持續輪換，整個過程應用程式無感。

把這幾塊拼起來看，你會發現 mesh 其實順手解決了一個老問題：服務發現。在沒有 mesh 的世界，「inventory-service 現在跑在哪些 IP 上」這個問題，要嘛靠 DNS（但 DNS 的 TTL 快取讓死掉的實例遲遲下不了線），要嘛靠應用程式自己去查 registry。有了 xDS，控制面把「誰是健康端點」這份名單**主動即時推**給每個 sidecar，失效窗口從 DNS 那種分鐘級，降到秒級。這份服務發現的能力，是 mesh 內建的副產品，而不是另外搭的東西（DNS 與 registry 那一整套權衡見〈DNS 與服務發現〉）。

## 零信任的身分：mTLS 與 SPIFFE 識別碼

Mesh 最常被當成「上車理由」的功能，是**自動 mTLS**——服務之間的每一條連線都雙向驗證身分、全程加密，而且應用程式完全不用碰憑證（單向 TLS 與 mTLS 的握手與憑證權衡見〈TLS 與 mTLS〉）。這件事在沒有 mesh 時是運維的惡夢：你得替每個服務簽發 client 憑證、分發、在到期前輪換，漏掉任何一個服務的輪換，它就在某個半夜憑證過期、整片連線報錯。

Mesh 把這套自動化到近乎隱形，靠的是一個叫 **SPIFFE** 的工作負載身分標準。它的核心觀念是：每個工作負載都該有一個**密碼學上可驗證的身分**，而不是靠寫死的 API key 或共享密鑰。在 mesh 裡，控制面的內建 CA 會替每個工作負載簽發一張短效期的 X.509 憑證（稱為 **SVID**，SPIFFE Verifiable Identity Document），把身分編進憑證的 SAN 欄位，長這樣：

```
spiffe://<trust-domain>/ns/<namespace>/sa/<service-account>
```

比如 `spiffe://cluster.local/ns/prod/sa/order-service`。當訂單服務的 Envoy 要連庫存服務，雙方各自出示自己的 SVID、各自驗對方的：庫存服務這端不只確認「對方持有一張我信任的 CA 簽的憑證」，還能讀出對方的 SPIFFE 識別碼**就是** `order-service`，再對照授權策略決定放不放行。身分、加密、授權，全織在這一次握手裡，而且——這是重點——**憑證壽命很短，由 SDS 持續自動輪換**（Istio 預設 24 小時、約 80% 壽命處自動換新；搭 SPIRE 可低到 1 小時）。

短效期憑證這個設計，順手化解了 TLS 世界裡最棘手的問題之一：撤銷。傳統 PKI 要撤銷一張被洩漏的憑證，得靠 CRL（笨重的撤銷清單）或 OCSP（線上查詢，有延遲與隱私代價）。Mesh 的答案是釜底抽薪——**憑證活得夠短，壞了等它自然過期就好**。一張只活一天、甚至只活一小時的憑證，根本沒有「需要緊急撤銷」的窗口。這是「用過期取代撤銷」的思路，而它之所以可行，正是因為輪換被自動化到了應用無感的程度；要人工每天、每小時換一次憑證，這個方案一秒鐘都活不下去。

## 算一筆 sidecar 的帳

Sidecar 模式聽起來很美，但「每個 Pod 旁邊塞一個代理」這句話，藏著一筆必須算清楚的帳。它不是免費的，而且代價會隨規模線性放大。

拿一個具體的中型系統來算：50 個服務、每個平均跑 3 個 Pod，總共 150 個 Pod。Sidecar 模式下，每個 Pod 都多一個 Envoy。問題是一個 Envoy 要吃多少資源？這裡有個常被低估的事實——Envoy 是個**通用** L7 代理，功能全、也因此不輕。實測中一個 Envoy sidecar 的記憶體常落在 **50 MB 起跳、隨設定與服務數量成長到 100 MB 以上**；早年的 benchmark 甚至量到單一 Envoy 峰值約 150 MB。取一個保守的中間值，每個 Envoy 算 60 MB：

```
150 Pods × 60 MB/Envoy ≈ 9 GB
```

**這 9 GB 記憶體，完全沒有跑任何業務邏輯**——它純粹是「讓服務之間能安全、可控地通訊」的稅。在資源就是錢的雲端帳單上，這是一筆看得見的開銷，而且它和你的 Pod 數綁死：你水平擴展應用到 300 個 Pod，這筆稅就翻倍到 18 GB。

延遲也有一筆帳。原本 A 直接呼叫 B 是一跳；現在變成 A → A的Envoy → B的Envoy → B，**每次服務間呼叫多穿過兩個代理**（出口一個、入口一個）。每個 Envoy 的處理通常在數百微秒到約 1 毫秒之間，疊起來一次呼叫多個 1～2 ms 不等。對單次呼叫這不算什麼，但在一條跨越五六個服務的請求鏈上，這些代理延遲會累加，而且 Envoy 的尾延遲（p99）在高負載下可能比平均高出好幾毫秒——這正是長尾延遲的一個被忽略的來源。

值得一提的是，不是所有 mesh 都付一樣的稅。Linkerd 走了另一條路：它不用通用的 Envoy，而是自己用 Rust 寫了一個**只為 sidecar 這一個用途特化**的微代理 linkerd2-proxy。Linkerd 官方 2021-05 的一份 benchmark 裡，Linkerd 代理的峰值記憶體平均約 **18 MB**，而 Istio 的 Envoy 約 **155 MB**——差了將近一個數量級。代價是 Linkerd 放棄了 Envoy 那套無所不包的通用性（它做不了 Envoy 能做的所有花式流量操作），換來輕量。這個取捨本身就很能說明問題：sidecar 的成本，很大一部分來自「用一個通用代理去做一件專用的事」。

## Mesh 自己會怎麼壞

到這裡 mesh 看起來像個淨贏：抽掉散落的網路碼、統一策略、自動 mTLS。但一個成熟的判斷必須看見它的另一面——**mesh 不會讓系統自動更可靠，它只是把可靠性問題集中化了，而集中化是一把雙面刃**。它引入了好幾個只有 mesh 才會有的新故障模式，每一個都值得單獨看清楚。

**啟動順序的競態。** 你的應用容器和它的 Envoy sidecar 在同一個 Pod 裡，但它們是兩個獨立的容器、各自啟動，誰先 ready 沒有保證。如果應用先起來、開始往外打請求，而 Envoy 還沒 ready——iptables 已經把流量導向 Envoy 了，但 Envoy 還沒準備好轉發——這些請求就**憑空失敗**。應用程式對此一無所知，它只看到莫名其妙的連線錯誤。關閉時是反過來的鏡像災難：如果 Envoy 先死，應用最後那幾個還沒送完的請求，會發現出口的代理已經不在了。這個耦合讓 graceful shutdown 變複雜——你不能只考慮自己的應用怎麼好好收尾，還得考慮和你共生的那個代理的生死順序（行程怎麼好好關掉見〈graceful shutdown〉）。

**雙重重試疊成風暴。** 假設你的應用程式碼裡還留著舊的重試邏輯（畢竟它是漸進式上 mesh 的），而你又在 mesh 層開了重試。現在一次失敗的呼叫，會被應用重試 3 次，每一次又被 mesh 重試 3 次——3 × 3 = 9 次請求打向一個已經在掙扎的下游。在多層服務鏈上，這種重試的乘積會像滾雪球一樣放大，把一個本來能自己恢復的瞬時故障，推成一場壓垮整片下游的重試風暴（這種放大效應的機制見〈retry storm 與 metastable failure〉）。Mesh 的重試很好用，但它**必須和應用層的重試協調**——通常的紀律是：重試只在一個地方開（要嘛應用、要嘛 mesh，不要兩邊都開），而且只對冪等請求開。

**控制面是新的單點。** 控制面雖然不在資料路徑上（它掛了，已經推下去的設定還在，現有流量照跑），但它一掛，整個叢集就**凍結在當下的設定**——新的端點推不下去、憑證輪換停擺、路由規則改不動。如果控制面長時間掛著，短效期憑證會一張張到期而沒有新的補上，最後 mesh 內的服務會開始彼此**握手失敗**——那個「短效期憑證」的優點，在控制面失聯時反過來變成催命符。

**它把一次升級變成全叢集事件。** 在函式庫的世界，升級網路邏輯是慢，但波及範圍（blast radius）是一個服務一個服務的。在 mesh 的世界，升級 Envoy 或控制面，是一次**牽動全叢集每一條連線**的變更。集中化的代價就在這裡：好處集中，風險也集中。一次有 bug 的 mesh 升級，能在幾分鐘內讓所有服務同時出問題。

還有一個常見的範疇混淆值得澄清：**mesh 管的是東西向流量（服務之間，內部互相呼叫），不是南北向流量（外部使用者進到系統）。** 後者是 API gateway 的工作。兩者職責不同、常常並存——gateway 守在邊界做認證、限流、把外部請求路由進來；mesh 在內部讓這些服務彼此安全通訊（API gateway 在邊界承擔什麼見〈API gateway 的角色〉）。把 mesh 當成 gateway、或反過來，是設計上很常見的混淆。

## Sidecar 太重，於是有了 ambient

前面那筆「150 個 Pod 就吃掉 9 GB」的帳，不是抽象的抱怨——它是 mesh 演化的直接推力。當社群把這筆帳算清楚，一個很自然的問題浮現了：**為什麼每個 Pod 都要一個全功能的 L7 代理？** 大部分服務其實只需要 mTLS 加密和基本的遙測（L4 層的事），真正用到 Envoy 那套華麗 L7 流量操作的，只是其中一小撮。為了那一小撮，卻讓每個 Pod 都扛一個 60 MB 的 Envoy，比例嚴重失衡。

這個觀察催生了 **ambient（無 sidecar）模式**。它把 mesh 的功能拆成兩層，按需取用：

- **節點級的 L4 代理（ztunnel）**：每個**節點**一個（不是每個 Pod 一個），用 Rust 寫的輕量代理，只做 L4 的事——mTLS 加密、L4 授權、基本遙測。所有 Pod 的基礎安全通訊都走它。因為從「每 Pod 一個」變成「每節點一個」，代理數量從 Pod 數降到節點數，記憶體足跡塌縮——官方宣稱可省 90% 以上的 per-workload 代理記憶體（實際省多少依工作負載而異）。
- **按需的 L7 代理（waypoint）**：只有真的需要進階流量管理（依 header 做金絲雀、複雜路由、L7 授權）的服務，才為它配一個 Envoy waypoint。waypoint 是 **per-namespace 或 per-service-account 共享**的 Envoy deployment、跑在應用 Pod 之外、可獨立擴縮升級（不是每個服務各黏一個），所以一個 namespace 的多個服務能共用同一個 waypoint。用得到才付這個代價，用不到的服務一毛不花。

回到剛才那筆帳：150 個 Pod、分佈在 10 個節點上。Sidecar 模式要 150 個 Envoy；ambient 模式只要 10 個 ztunnel（加上少數真的需要 L7 的服務各擺一個 waypoint）。代理數量從 150 砍到約十幾個，這就是 ambient 存在的全部動機——**把「人人都付的全功能稅」改成「基礎安全大家共享、進階功能按需付費」**。

代價當然有。ztunnel/waypoint 是一套和 sidecar 不同的心智模型，運維要重新理解流量怎麼流、出問題時去哪裡看。而且因為 L4 與 L7 拆到了不同的代理、不同的層級，一條請求的路徑變得更曲折，排查時要在腦中重建的跳數更多。Istio 的 ambient 模式在 **v1.24（2024-11）正式 GA / Stable**（2026-02 的 1.29 把 ambient 的跨叢集、跨網路多叢集支援推進到 **Beta**——尚未生產就緒，單網路多叢集仍是 alpha）；它代表的不是「sidecar 錯了」，而是「同一個問題在資源壓力下演化出的更精算的形狀」。

## 為什麼是這個形狀

退一步看，service mesh 的整個樣貌，都是從一句話裡長出來的：**服務之間怎麼通訊，是一個橫切所有服務、卻又和任何單一服務的業務無關的關注點。**

正因為它橫切所有服務，把它留在每個應用裡各寫一遍，必然漂移、不一致、難治理——所以要抽出來。正因為它要跨越所有語言、又要能獨立於應用的發版週期被升級，唯一能擺它的地方，就是一個語言無關、可獨立升級的**行程外代理**——所以是 sidecar。正因為策略要統一、執行要分散，所以分成集中的控制面與分散的資料面，靠 xDS 把設定即時推下去。正因為它在資料路徑上多插了一層，它就既買到了統一治理與自動 mTLS，也欠下了資源、延遲、與一整族新故障模式的債——而 ambient 模式，正是社群在把這筆債算清楚後，對「能不能少付一點」這個問題的回答。

Mesh 不是一套憑空長出來的時髦架構。它是「網路治理邏輯不該繼承應用程式的耦合」這個樸素的洞察，被推到極致後唯一站得住的形狀。下次你看到一個 Pod 裡除了應用容器還躺著一個 Envoy，你會知道那不是多餘的累贅——那是你的服務間通訊邏輯，終於從二十份散落的、各說各話的實作，收斂成了一份。
