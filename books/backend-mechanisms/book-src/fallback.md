# fallback 與降級：壞掉時退到哪

商品頁載入的瞬間，後端同步打了七個服務：商品本體、庫存、價格、評論摘要、運費試算、「你可能也喜歡」的個人化推薦、還有右下角那個「最近瀏覽」。其中六個回得又快又準，只有推薦服務這天卡住了——它平常 p99 是 50 毫秒，今晚不知為何抖到兩秒。

如果這個商品頁的程式碼老老實實地「等所有依賴都回來才渲染」，結果是：一個本來該秒開的頁面，被一個**最不重要**的方塊拖到兩秒；推薦服務再壞一點、直接超時拋錯，整個商品頁跟著回 500——使用者連商品都看不到、更別說下單了。一個邊角功能的故障，升級成了整條購買路徑的故障。

這裡缺的那塊機制，就是 fallback（後備／降級）。它要回答的問題只有一句：**當主路徑失敗或不可用時，我端給使用者什麼？** 不是「怎麼讓它不失敗」——那是 timeout、retry、circuit breaker 的事；fallback 接手的是失敗**已經發生之後**的世界，它的信念是一句樸素到容易被忽略的話：**部分可用，遠勝於完全不可用。** 給不出最新最準的推薦時，給一個稍舊、稍粗、非個人化的熱門榜，幾乎總比給一個 500 強。

## 它和容錯家族其他成員的分工

fallback 很容易和它的鄰居混在一起，所以先把界線劃清楚。一次外部呼叫，從發出到處理結果，沿路有好幾道機制各管一段：

- **timeout** 決定「我願意等多久才放棄」——它把無界的等待變成有界的失敗（見〈timeout、deadline 與 budget：把等待變成有界〉）。
- **circuit breaker** 決定「要不要連試都不試、直接快速失敗」——當某個依賴連續壞了一陣子，它跳開，後續呼叫不再真的打過去，省下白等的時間（三態與半開試探的機制見〈circuit breaker：三態與半開試探〉）。
- **bulkhead** 決定「這個故障最多能佔走多少資源」——把資源切艙，讓一個慢依賴塞滿自己那艙就好、別蔓延（見〈bulkhead：艙壁隔離〉）。
- **fallback** 決定「放棄／失敗／被拒之後，到底端什麼給使用者」。

前三者都在處理「怎麼偵測故障、怎麼框住故障」；fallback 是唯一一個在處理「故障已成定局，怎麼還能交付一點價值」的。它們是接力，不是替代——一個成熟的呼叫往往是：timeout 設了 80 毫秒、circuit breaker 在背後監看失敗率、bulkhead 限制並發，而當以上任何一道判定「這次不行」時，控制權交給 fallback，由它決定退到哪。

值得記住「fallback 由誰觸發」這件事，因為它決定了你的降級邏輯掛在哪：最常見的三個觸發點是**逾時**（等不到，超過 deadline）、**例外**（呼叫直接拋錯）、以及**斷路器跳開或艙滿被拒**（根本沒真的打出去）。同一段 fallback 程式碼，可能被這三種情況裡的任何一種喚醒——所以它不能假設「主路徑一定真的試過了」。

## 降級不是開關，是一道斜坡

把「降級」想成一個布林開關（要嘛全功能、要嘛全掛）是新手最常見的心智模型，而它是錯的。好的降級設計（graceful degradation）是把「失敗」攤成一條**從完整功能緩緩滑落到最小可用**的斜坡，而不是一道非黑即白的懸崖。

Google SRE 在《Addressing Cascading Failures》裡給的定義很精確：降級是「**減少需要做的工作量**」，而不只是把請求丟掉。他們舉的例子——搜尋在過載時只查記憶體快取裡的一小部分語料、或改用一個比較不準但比較快的排序演算法；新聞服務在容量吃緊時，不再附上配圖和小地圖，只回純文字。每一步都還在服務，只是服務得更省、更粗。

具體到一個讀路徑，這條斜坡通常長這樣，由貴到便宜、由準到粗：

```
full        -> 即時計算的個人化結果（最準、最貴）
degraded-1  -> 回上次成功的快取值（稍舊，但近似個人化）
degraded-2  -> 回一個全站通用的靜態結果（不個人化，但秒開）
degraded-3  -> 回一個保守的預設／空殼（功能縮水，但頁面不崩）
fail-loud   -> 連預設都給不出時，回明確錯誤（最後手段）
```

設計的功夫，全在「想清楚每一級退到哪、退了之後使用者損失什麼」。常見的退路有這幾種，各有各的脾氣：

**回快取／陳舊資料。** 主資料源掛了，就回上一次成功時快取下來的值。即使過期，「稍舊的答案」通常比「沒有答案」好。這條退路有個 HTTP 層面的標準化版本值得知道：RFC 5861 定義的 `stale-if-error`——當快取在回源時撞到錯誤（5xx、連線中斷、DNS 失敗），它**可以**直接拿一份已過期的快取回應頂上，而不是把硬錯誤甩給使用者。它的雙生兄弟 `stale-while-revalidate` 則是「先回過期的、同時在背景悄悄更新」，主要在藏延遲。這兩個指令被主流 CDN 廣泛支援，本質上就是把「壞掉時退回陳舊版本」這件事寫進了快取協定。

**回預設／靜態值。** 給一個保守的常數。風控分數算不出來時預設「需人工複審」、推薦清單回「全站熱門 Top 10」、配置讀不到時回內建的安全預設。

**功能降級。** 關掉非核心功能，保住核心。推薦服務掛了就不顯示「猜你喜歡」，但商品和下單照常——這正是開場那個商品頁該做的。

**降低品質。** 回低解析度、少欄位、近似結果，代替完整精確的版本。

**切備援來源。** 主供應商掛了，切到次要供應商——可能更貴、更慢，但能用。

## 一個 80 毫秒的數字，怎麼救活整個商品頁

回到開場。把降級織進那個被推薦服務拖累的商品頁，數字會說話。

推薦呼叫平常 p99 = 50 毫秒，但偶爾抖到 2000 毫秒。我們對它設一個 `timeout = 80ms`：給它在正常 p99 之上一點餘裕（50ms 之上留 30ms），超過就放棄。超過 80ms 沒回，就走 fallback——回那個「全站熱門 Top 10」、每 5 分鐘刷新一次的靜態榜。

來看這一招買到了什麼。商品頁對外的延遲，本來是被它**最慢的那個下游**決定的——七個並行呼叫，整頁要等最慢的回來才渲染，所以推薦一抖到 2000ms，整頁 p99 就跟著爆到 2000ms。加了 80ms timeout + fallback 之後，推薦這條路徑對整頁延遲的貢獻被**硬封頂在 80ms**：推薦在 80ms 內回（含健康時的 ≤50ms）就用真結果，超過 80ms 沒回就在第 80ms 果斷放棄、改用熱門榜。整頁延遲是七條路徑取最大值，所以只要其餘六個依賴本來就快（≤80ms），推薦這條被封頂後，整頁 p99 就不再被下游的長尾牽著走，而是被自己設的那個 80ms 上限封住。

代價是什麼？是一部分使用者在推薦抖動的那幾分鐘裡，看到的不是個人化清單而是通用榜單——「稍舊、非個人化，但秒開」。這就是 fallback 的核心交換式：**用一點結果品質，換延遲的穩定性與頁面的可用性。** 對推薦這種「次等正確完全可接受」的功能，這筆交易幾乎穩賺。

值得停下來看清一個常被忽略的細節：那 80ms 是 timeout，不是「推薦服務變快了」。推薦服務那邊**還在繼續算**那個沒人要的個人化結果——你只是不等它了。所以理想情況下，timeout 觸發 fallback 的同時，最好也向下游送一個取消訊號（gRPC cancellation、斷開連線），讓它別繼續燒 CPU 去算一個已經被丟棄的答案（見〈timeout、deadline 與 budget：把等待變成有界〉）。否則在過載時，這些「沒人要的計算」會堆積成壓垮下游的最後一根稻草。

## 降級會在最壞的時刻悄悄放大負載

這是 fallback 最反直覺、也最危險的故障模式——它恰恰在「以為自己已經加對了安全網」的那一刻反咬一口。

設想主資料源掛了，於是**所有**讀請求都觸發 fallback，全部湧向那個熱門榜的快取。平常這個快取每秒只被讀幾百次（cache miss 才回源），現在主源一倒，它瞬間要扛下原本由主源承接的**全部**流量。如果這個 fallback 路徑的容量沒有按「主路徑全倒」的最壞情況來規劃，它會被這波湧入壓垮——於是你的安全網自己也破了，只是把雪崩換了個方向。

Google SRE 把這個陷阱講得很白。在分層架構裡，當主後端不可用、流量改走次要後端時，「如果整個系統已經過載，主到次的轉發會增加、給系統添上**更多**負載」——因為在最缺容量的那一刻，你還得額外付出「解析請求、發給次要後端、等它回來」的成本。降級路徑不是免費的退路，它在最壞的時刻消耗的，恰恰是你最缺的那點資源。

所以一條鐵律：**降級路徑的容量，要能扛住「主路徑 100% 倒下」時湧過來的全部流量。** 如果你的 fallback 是回一個快取值，那個快取要能吃下原本由主源承接的 QPS；如果是切備援供應商，那個備援的容量和你的主流量要在一個量級，否則切過去只是換它先死。

這也是為什麼 fallback 和 circuit breaker 是天生的搭檔。斷路器跳開後，後續呼叫**根本不真的打主路徑**，而是立刻走 fallback——這恰好削掉了「每個請求都先去捅一下已死的主源、再轉 fallback」那層重複負載。沒有斷路器的純 fallback，等於在主源每次失敗前都先白白打它一次，既慢又給垂死的主源繼續加壓（半開試探如何在主源恢復後自動切回，見〈circuit breaker：三態與半開試探〉）。

## 有些東西，寧可硬報錯也不准降級

到目前為止 fallback 聽起來像是純粹的好東西——壞了就退一步，總比死透好。但有一整類操作，對它們降級是**製造資料錯誤的捷徑**，正解恰恰相反：明確失敗，把錯誤大聲拋出去（fail loud）。

界線在於：這個操作能不能容忍「次等正確」？

- 推薦、搜尋補全、個性化、儀表板、非交易型展示——這些回個稍舊或近似的結果，使用者頂多覺得「不夠準」，不會出事。**fallback 的主場。**
- 扣款、轉帳、下單、配額扣減、權限判定、唯一性檢查——這些只有「對」和「錯」，沒有「差不多對」。對「這筆轉帳成功了嗎」這個問題，回一個「降級的猜測」比回一個明確錯誤糟糕**得多**：你把一個「不確定」偽裝成了一個「成功」，下游照著這個假成功繼續走，資料就此分岔。

對這類強一致、涉及金錢與唯一性的寫操作，正解是 **fail loud**：回一個清楚的錯誤，把「要不要重試、要不要人工介入」的決定權交還給上游，而不是自己擅自編一個「看起來成功」的結果蒙混過去。降級在這裡是反模式。

安全與授權路徑上，這個取捨有個專門的名字：**fail-open 還是 fail-closed**。當鑑權服務本身掛了、你判不出這個請求該不該放行時，你有兩個選擇：fail-open（判不出就放行，賭可用性）或 fail-closed（判不出就拒絕，賭安全）。對絕大多數授權決策，正解是 **fail-closed**——寧可錯殺一個合法請求（使用者重試一下），也不能在鑑權系統故障的窗口裡放行一個本該被擋的請求。一個「壞了就全放行」的鑑權器，等於給攻擊者一個「只要能弄垮鑑權服務、就能暢通無阻」的後門。這裡的「降級」如果降錯方向，代價是一道敞開的大門，而不是一個稍舊的推薦。

所以 fallback 設計的第一個問題永遠不是「退到哪」，而是「**這個操作到底准不准降級**」。准，才談退路；不准，就老實 fail loud。

## 沉默的降級，比錯誤更危險

假設你把上面都做對了：降級分了級、容量規劃了、寫操作 fail loud 了。還有一個故障模式在等你，而且它特別陰險——因為它**看起來一切正常**。

降級成功了。使用者看到的是熱門榜而不是個人化推薦，但他大概也沒太在意，照樣下單了。監控面板綠油油的——HTTP 200、延遲正常、錯誤率為零。於是沒有人知道：推薦服務其實已經死了好幾天，整個站一直在悄悄跑降級模式。直到某天連那個熱門榜的快取也過期、連 fallback 都給不出東西了，才轟一聲全面爆掉——而這時故障已經積累了好幾天的勢能。

問題的根源是：**fallback 在設計上就是用來「把故障藏起來」的**——它的整個價值就在於讓使用者感覺不到後端壞了。但「使用者感覺不到」絕不能等於「營運的人也感覺不到」。所以一條配套鐵律：

**每一次走 fallback，都必須被計數、被觀測。** fallback rate（降級率）是一個一級指標：它一旦從接近零跳起來，就該告警——不是等到 fallback 也撐不住才知道，而是在主路徑剛開始壞、降級剛開始接管的那一刻就知道。降級必須是**顯眼的**，不是無聲的。Google SRE 對此的說法是「監控並在太多伺服器進入降級模式時告警」——降級該是個罕見事件（通常意味著容量規劃失誤或意外的流量轉移），而不是日常背景噪音。

這個陷阱還有一個更隱蔽的變體：**把錯誤吞掉、回一個空結果，當作降級。** 主源掛了，於是 `catch` 區塊裡 `return []`，上游收到一個空陣列，以為「這個使用者真的沒有推薦」。這比拋錯**更危險**——因為一個明確的錯誤至少會被當成異常處理、會留下痕跡；而一個被偽裝成「正常的空狀態」的故障，會被當成真相一路傳下去。降級的結果**必須標明自己是降級的**（一個 header、一個欄位、一個 metric），不能讓它和真實結果長得一模一樣。

## 沒人走的那條路，早就壞了

還有最後一個故障模式，它不在故障發生時暴露，而在你最需要它時暴露：**fallback 路徑自己也會失敗，而它平時沒人走，所以沒人知道它早就壞了。**

那個備援供應商，你上次真的切過去是什麼時候？那個降級快取，它的格式還和當前的程式碼相容嗎？那段 `catch` 裡的 fallback 邏輯，從寫完那天起跑過幾次？答案往往是令人不安的「不知道」。Google SRE 有一句話精準地刺中這點：「**你從不走的那條程式碼路徑，（往往）就是不能用的那條。**」備援平時不接流量，於是它的配置會漂移、依賴會過期、容量會被別的東西悄悄借走——等到主路徑真的倒下、你滿懷信心切過去時，才發現安全網本身早已是一張破網。

這意味著 fallback 不是「寫完就放著」的東西。它需要被**定期演練**：主動注入故障（殺掉主源、讓推薦服務超時），驗證降級路徑真的會被觸發、真的端得出東西、容量真的扛得住。沒被測試過的 fallback，是一張看似張著、其實早已破掉的安全網——而它的破，偏偏只會在最糟的時刻才被發現。

順帶一提，這也是為什麼 circuit breaker 的 half-open 半開試探如此重要：它強制系統定期「派一個探子去試試主源活了沒」，主源一恢復就自動切回。沒有這個自動回正機制，一個切到備援的系統很可能就**永遠跑在次等模式**——主源早就好了，但因為沒人去試、沒人切回，整個站長期跑在更貴或更差的降級路徑上，直到有人偶然發現（見〈circuit breaker：三態與半開試探〉）。

## 一個歷史註腳：從「框架替你做」到「你自己想清楚」

降級這套思路在工程實務上有個清楚的演化痕跡，值得知道，因為它解釋了今天的工具長什麼樣。

2010 年代中期，Netflix 的 **Hystrix** 把 timeout、circuit breaker、bulkhead、fallback 打包成一個庫，用 `getFallback()` 這樣的鉤子讓你掛降級邏輯，紅極一時。但 Netflix 在 **2018 年 11 月**把 Hystrix 移入維護模式（最後一個版本是 1.5.18），不再積極開發。接棒的 **Resilience4j** 換了個哲學：不再像 Hystrix 那樣替每個呼叫開執行緒池，而是用輕量的裝飾器（decorator）包住既有方法，讓你用 `fallbackMethod` 顯式指定降級函式（2026-06）。

這段演化的重點不在工具名字，而在它揭示的一件事：**降級邏輯沒辦法被框架完全替你想好。** 框架能幫你的是「機制」——在哪觸發、怎麼切、怎麼統計失敗率；但「退到哪」「退了之後使用者損失什麼」「這個操作准不准降級」這幾個真正要命的問題，永遠是業務判斷，得由懂這個功能的人一個一個想清楚。Hystrix 的 `getFallback()` 和 Resilience4j 的 `fallbackMethod` 都只是給你一個掛鉤子的地方；鉤子上掛什麼，是你的事。

## 為什麼是這個形狀

退一步看，fallback 的整套樣貌，是從一個樸素的不對稱裡長出來的：**對使用者而言，一個稍舊、稍粗、功能縮水的答案，幾乎總比一個 500 好——但前提是這個操作能容忍「次等正確」。**

正因為「部分可用 > 完全不可用」，所以失敗時要退而不是死——這是 fallback 存在的理由。正因為退路在最壞時刻也要消耗資源，所以它的容量必須按「主路徑全倒」來規劃，否則就是把雪崩換個方向。正因為它的本職就是「把故障藏起來」，所以它必須配上響亮的可觀測性，否則藏住的故障會積累成更大的災難。正因為平時沒人走的路會悄悄壞掉，所以它必須被定期演練。而正因為有一整類操作根本不能容忍「次等正確」，所以 fallback 的第一個問題永遠是「准不准降級」，金流與授權的正解是 fail loud、fail closed，不是退讓。

降級不是「壞掉時隨便給點東西」的權宜之計，而是一個被想清楚的承諾：在故障發生時，這個系統**已經預先決定好**它會退到哪、退了之後還能交付什麼、以及哪些東西它寧可拒絕也絕不含糊。下次你看到一個服務在依賴掛掉時，端出一份明顯稍舊但秒開的內容、同時在監控上亮起一盞降級的燈——那不是它在湊合，那是它在兌現一個事先想好的退路。
