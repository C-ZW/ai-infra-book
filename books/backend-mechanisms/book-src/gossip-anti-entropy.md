# gossip 與 anti-entropy

想像一個橫跨三個機房、一千台節點的叢集。每一刻都有節點在加入、有節點在被替換、有節點因為一張壞掉的網卡靜靜地死去。現在你得回答一個聽起來無辜的問題：**此刻，誰還活著？**

最直覺的做法，是擺一台中央伺服器當花名冊。每台節點每秒對它報一次平安，它掌握全局的存活視圖，誰要問「叢集現在有哪些成員」就問它。這個設計在十台節點時無懈可擊，在一千台節點時會以兩種方式同時崩潰。第一，那台中央伺服器自己一掛，整個叢集瞬間失明——沒有人知道誰活著，這是一個刺眼的單點故障。第二，就算它不掛，一千台節點每秒對它打一次心跳，它要扛一千個連線、一千次寫入；節點數再翻一個數量級，它就先被自己負責偵測的那群節點給壓垮了。**你想要一份全局的成員視圖，卻又不想讓任何一台機器為此成為瓶頸或單點。** 這個張力，就是 gossip 協定存在的全部理由。

## 把問題講難：去掉中央之後，狀態怎麼擴散

去掉中央伺服器，問題立刻變得棘手。沒有人掌握全局，每台節點只認得一部分對等節點。當 A 節點觀察到「D 死了」這條新資訊，它要怎麼讓另外九百九十九台都知道？

天真做法一：**A 廣播給所有人。** 這等於把中央伺服器的負擔分散成「每個有新消息的節點都對全網廣播一次」——一千台節點同時各自廣播，瞬間製造 N² 量級的訊息風暴，網路先死。而且只要有一個封包在路上丟了，那台節點就永遠錯過這條消息，沒有任何補救。

天真做法二：**串成一條鏈，A 告訴 B、B 告訴 C，一個傳一個。** 訊息量是省了，但擴散是線性的——傳到第一千台要一千跳；中間任何一台掛掉，鏈就斷了，後面的人全部失聯。

問題的本質是：在一個**沒有中央、訊息會丟、節點會死**的網路上，怎麼讓一條資訊**可靠地、快速地、且不靠任何單點**傳遍所有人。這看起來像是要同時要到「快」「省」「容錯」三樣彼此打架的東西。1987 年，Xerox PARC 的 Alan Demers 等人在一篇談「怎麼維護複製資料庫」的論文裡，給了一個出人意料的答案：**別去設計傳播路徑，讓資訊像流行病一樣自己擴散。**

## 流言怎麼傳：隨機、週期、指數

gossip（流言／八卦協定，又稱 epidemic protocol，疫情式協定）的規則簡單到近乎隨便。每台節點都做同一件事，週期性地、各做各的：

```
every T seconds, each node does:
  pick a few random peers from its member list
  exchange state with them (who's alive, latest config, version vectors...)
  merge whatever is newer into its own view
```

沒有協調者，沒有固定路徑，沒有誰負責通知誰。每台節點每隔一個固定週期（典型一秒），從自己認得的成員裡**隨機**挑幾個對等節點，交換彼此的狀態，把對方比較新的部分併進來。一個剛收到新消息的節點，下一輪就成了新的傳播源，再去隨機感染別人——就像一個剛被傳染的人，會在他自己的社交圈裡繼續散播病毒。

這個「被感染者再去感染別人」的結構，正是它快得驚人的原因。第一輪，只有源頭一個節點知道；它感染幾個，第二輪知道的人翻倍；這些人又各自感染，第三輪再翻倍。知情者數量大致是**指數成長**，於是傳遍 N 個節點所需的回合數，跟 **log N** 成正比，而不是 N。

這個 log 不是隨口說的，可以拿真實數字算到底。回到那個一千台節點的叢集。一條「D 死了」的消息要傳遍全網，需要的回合數約與 log₂(N) 成正比：

```
log2(1000) ≈ 9.97  →  about 10 rounds
```

也就是大約 **10 個回合**。若每個回合的週期是 1 秒，這條成員變更在大約 **10 秒內**就擴散到全部一千台節點。把節點數翻成一百萬呢？log₂(1,000,000) ≈ 20——只要約 20 個回合、20 秒。節點數翻了一千倍，擴散時間只翻一倍。這就是對數的威力，也是為什麼 gossip 能撐到數千、數萬節點而中央伺服器撐不到。

更關鍵的是這 10 秒**買到了容錯**。整個過程裡，就算有一批節點正在當機、有一堆封包在路上遺失，消息照樣傳得到——因為每台節點每輪都挑**不同的**隨機對端，一條路不通自然會走另一條。沒有任何一台機器是非它不可的關卡。冗餘不是設計出來的，是隨機性免費附帶的。

> 補一個常被忽略的精確值：理論上 push-pull 式 gossip 在完全圖上把一條消息傳遍全網，期望回合數主階是 **log₃ N**，再加上一個量級遠小於 log N 的後期 pull 收尾項，整體偏差只有**加性常數**（±O(1)）——這個精確到常數的結果是 Doerr 與 Kostrygin 的分析（2017），建立在 Karp 等人 push-pull 的奠基研究之上。它比單純 log₂ 還快一點。實務上我們記「O(log N) 回合」這個量級就夠用，重點是它隨節點數**對數**成長，不是線性。

## push、pull 與 push-pull：擴散到一半時誰更划算

「交換狀態」這四個字底下，藏著一個影響收斂速度的選擇：**A 找上 B 的時候，是把自己的新消息推給 B（push）、還是問 B 有什麼新的（pull）、還是兩件事一起做（push-pull）？**

這三者的差別，在擴散的不同階段會反轉。**感染初期**，知情的人少、不知情的人多：這時 push 划算——一個知情者隨便推給一個對端，幾乎一定推給了一個還不知道的人，每次都有效。但到了**感染後期**，絕大多數人都已經知道了：這時 push 開始空轉——一個知情者推給的對端，多半也早就知道了，這次傳播完全白費。反過來，pull 在後期更划算——一個還沒被感染的少數派，主動去問任何一個對端，對方幾乎一定已經知道、能把消息給它。

於是 **push-pull** 結合兩者，在每個階段都不浪費：兩個節點碰面時，互相把對方缺的補齊。它收斂最快，也是多數實作（包括 Cassandra）的選擇。代價是每次交換的訊息稍大、邏輯稍複雜——但對 gossip 這種背景流量而言，這點開銷換來更快的收斂，幾乎總是值得。

Cassandra 的 gossip 把 push-pull 做成一個很像 TCP 三次握手的三段交換，值得拿來看一眼，因為它把「只傳差異」這件事做得很乾淨：

```
Cassandra gossip, three messages (digest = node + generation + heartbeat version):

  initiator              peer
     |   SYN  (my digests)   |   "here's a summary of every node I know,
     | --------------------> |    with its generation + heartbeat version"
     |                       |
     |   ACK (diff + req)    |   peer compares, replies with: the states it
     | <-------------------- |    has that you lack, AND digests of what it
     |                       |    wants from you
     |   ACK2 (states)       |
     | --------------------> |   "here are the states you asked for"
```

第一段 SYN，發起者送的不是完整狀態，而是每台已知節點的**摘要**（digest）——節點 ID、generation（這台節點某次啟動的紀元號）、heartbeat version（它報過幾次平安的版本號）。對端拿這份摘要跟自己的比對，第二段 ACK 只回「你缺的那些狀態」加上「我想跟你要的摘要」。第三段 ACK2 補上對方要的狀態。整個過程**只傳兩邊真正有差異的部分**，不傳已經一致的狀態——一千台節點的成員表絕大多數時候是相同的，每秒拿整份去交換會把頻寬燒光。generation 加 heartbeat version 這對版本號，就是用來快速判斷「同一台節點，誰手上的資訊比較新」的依據：generation 大的代表是更晚一次的啟動（壓過舊紀元的殘留資訊），同 generation 下 heartbeat version 大的代表更新。

## 兩種機制：散播熱消息 vs 對帳全狀態

到這裡有個容易混淆的地方需要拆開。Demers 那篇論文其實提出了**兩種不同的 gossip 機制**，它們解決的是不同的問題，實務上要搭配著用。

**第一種叫 rumor mongering（謠言傳播）**，也就是前面講的那套——只散播「最近的更新」這種**熱**消息。一條 rumor 被隨機轉發給幾個對端，傳著傳著，如果一個知情者連續遇到好幾個「早就知道了」的對端，它就判斷這條消息已經傳夠了、把它標記成冷的、停止傳播。這很像八卦的生命週期：一條新聞剛爆出來時人人搶著講，講到大家都聽膩了、你跟誰講對方都說「我知道了」，這條八卦就自然死了。rumor mongering 快、省頻寬，是 O(log N) 回合擴散的主力。

但它有一個無法迴避的裂縫：**機率性漏網。** 一條 rumor 在被判定為「冷」、停止傳播之前，有極小的機率某幾個倒楣的節點剛好沒被任何傳播鏈碰到。對單一一條消息，這個機率小到可以忽略；但一個長期運行的系統，每天有成千上萬條更新在 gossip，這些「極小機率」會累積成持續的**漂移**（drift）——總有一小撮節點的狀態跟全網對不上，而且因為 rumor 已經冷掉了，沒有任何機制會去修它。

這就是**第二種機制 anti-entropy（反熵）** 要補的洞。「熵」在這裡借的是熱力學的隱喻：副本之間的不一致就是系統的「熵」，anti-entropy 就是持續把這個熵往下壓的後台工作。它的做法跟 rumor mongering 正好相反——**不管消息冷熱，週期性地隨機挑一個對端，比對雙方的完整狀態，把所有差異調和一致。** rumor mongering 只追著「最近發生了什麼」跑，anti-entropy 則定期做一次「我們兩個現在到底有沒有完全一樣」的全面體檢，把任何漏網、任何累積的漂移**最終**修平。

兩者的分工於是很清楚：rumor mongering 負責**快**（新消息幾秒內傳開），anti-entropy 負責**對**（保證最終沒有任何漂移殘留）。Demers 論文的核心論點之一，正是這兩者必須搭配——只跑 rumor mongering 不跑 anti-entropy，漂移會無聲累積；只跑 anti-entropy 不跑 rumor mongering，新消息的擴散會慢到不堪用。

## anti-entropy 的頻寬難題：別把整份狀態掃一遍

anti-entropy 聽起來簡單——比對兩邊的完整狀態、傳差異就好。但「比對完整狀態」這件事本身有個尷尬的成本：如果兩個節點各持有一千萬筆資料副本，要找出哪幾筆不一樣，難道要把一千萬筆逐筆傳給對方比對嗎？那這個「降熵」的後台工作，自己就先把頻寬燒光了。絕大多數時候兩邊其實**幾乎一致**，只差零星幾筆——為了找出那幾筆而傳輸整份資料，是荒謬的浪費。

解法是 **Merkle tree（雜湊樹）**，Dynamo 與 Cassandra 的 anti-entropy 都靠它。它的構造是一棵雜湊樹：葉節點是每一筆（或每一小段 key）資料值的雜湊，每個父節點是它所有子節點雜湊再拼起來的雜湊，一路往上,**根節點的雜湊濃縮了整段 key range 的全部內容**。

關鍵在這棵樹怎麼讓你「只比差異」：

```
Merkle tree anti-entropy between two replicas of the same key range:

  compare roots
     |
     +-- roots equal?  --> whole range in sync, done. (one hash exchanged!)
     |
     +-- roots differ? --> recurse into children, compare each subtree's hash
                            |
                            +-- subtree hashes equal --> skip, untouched
                            +-- subtree hashes differ --> descend further
                                                          ...down to the
                                                          divergent leaves
```

兩個副本先只交換**根雜湊**。若根雜湊相同——這段 key range 完全一致，一筆資料都不用傳，整次對帳就靠交換**一個雜湊值**結案。若根雜湊不同，代表底下某處有差異，於是往下走一層、比對左右子樹的雜湊：一致的子樹整棵跳過（底下不可能有差異），只順著**不一致的那條路徑**遞迴下探，最後定位到真正分歧的那幾片葉子，只傳那幾筆。

於是 anti-entropy 的傳輸量，從「跟資料總量成正比」變成「跟**分歧量**成正比」。兩邊完全一致時，成本趨近於零（一個根雜湊）；只差幾筆時，也只付定位那幾筆所需的 log 量級的雜湊比對。這就是為什麼 anti-entropy 可以開著當低頻後台工作而不拖垮系統——它在絕大多數「其實沒事」的情況下幾乎免費。對應到資料對帳那一面的「怎麼確認兩邊一致、漂移怎麼修」，Merkle tree 正是把漂移最終修平的底層引擎（見〈對帳〉）。

## 把 gossip 拿去偵測故障：SWIM 的設計

gossip 最廣為人知的用途，是散播「成員與故障狀態」——也就是回答開頭那個問題：誰還活著。但這裡有個微妙的轉折：要 gossip「誰死了」，你得先**偵測**誰死了，而偵測本身又不能變回那個「每台對每台心跳」的 N² 災難。2002 年 Das、Gupta、Motivala 的 **SWIM** 協定，把這件事做漂亮了，值得拆開講透，因為它的每個設計決定都在堵一個具體的坑。

SWIM 的第一個聰明處，是把「偵測」和「散播」拆開。傳統做法用全互相心跳（all-to-all heartbeat）來偵測，訊息量隨節點數平方成長。SWIM 改成：每台節點每個週期**只隨機挑一個對端 ping 一下**，不論叢集多大，每台節點每輪的偵測流量是固定的常數。成員變更的散播則完全交給 gossip，而且巧妙地**搭在 ping/ack 的封包上捎帶**（piggyback），不另開訊息。偵測是常數成本、散播是 log 成本，整個協定的訊息複雜度被壓到隨節點數線性，而非平方。

第二個聰明處，是不**直接**根據「ping 不到」就宣判死亡。設想 A ping B、沒收到 ack。這背後可能是 B 真的死了，但也可能只是這一個封包在路上丟了、或 B 正卡在一次 GC 暫停裡——直接判 B 死，是把一次網路抖動誤判成一場葬禮。SWIM 的對策是**間接探測**（indirect ping）：A ping 不到 B，不馬上下結論，而是**隨機請另外 k 台節點各自代替 A 去 ping B**。只要這 k 台裡有任何一台能 ping 到 B 並把 ack 轉回來，就證明問題出在 A↔B 這條特定的網路路徑上，B 本身是活的。唯有 A 直接 ping 與這 k 台間接 ping**全都**失敗，B 才被升級為可疑。間接探測就是用「換幾條不同的網路路徑再試」來把「單一路徑抖動」和「節點真的死了」區分開。

第三個聰明處，是 **suspect（可疑）這個中間狀態，搭配 incarnation number（化身號）來允許「死而復生的澄清」**。被間接探測也判失敗的節點，不會立刻被標成 dead、踢出叢集，而是先標成 **suspect**，並把這個懷疑 gossip 出去。這給了被冤枉的節點一個申訴的窗口：每台節點維護一個只能由自己遞增的 incarnation number；當 B 從別人的 gossip 裡聽到「有人懷疑我死了」，它**把自己的 incarnation number 加一、廣播一條「我還活著」（alive/refute）**。因為這個號碼只有 B 自己能增、且更大的號碼壓過小的，全網收到後就知道這是 B 本人最新的、權威的澄清，於是撤銷對它的懷疑。只有在 suspect 狀態持續一段時限、B 始終沒有用更大的 incarnation 來反駁，它才終於被確認為 dead 並 gossip 出去。

這三層——常數成本偵測、間接探測防誤判、suspect＋incarnation 給澄清窗口——疊起來，才讓「用 gossip 管成員」在數千節點規模下既便宜又不至於動不動就誤殺活著的節點。Consul、Serf、Nomad 底下的 memberlist 函式庫，跑的正是 SWIM（HashiCorp 還在其上疊了名為 Lifeguard 的擴充來進一步壓低誤判率）；Cassandra 的成員管理也是同一套 gossip 思路。

## 一個藏在 SWIM 細節裡的非顯然陷阱：隨機 ping 不保證每個人都被探到

這裡有個直覺會錯的地方，值得單獨點出來。前面說「每輪隨機挑一個對端 ping」，聽起來每台節點遲早都會被探到。但**純隨機**有個惱人的後果：在任意有限時間窗口內，某台倒楣的節點可能**剛好一直沒被任何人挑中** ping——它若這時死了，會有很長一段時間沒人發現。隨機帶來的「平均上很快」並不保證「每一台的偵測都有時間上界」。

SWIM 論文（4.3 節）對此的修正很細膩：把 ping 對端的選擇從「純隨機」改成**輪詢（round-robin）**——每台節點把自己的成員清單排成一輪，依序一個一個 ping，**等整輪每個成員都被 ping 過一遍之後，才把清單洗牌、開始下一輪**。如此一來，任一節點在「一整輪」的時間內**保證**至少被探一次，偵測時間有了確定的上界（這個性質叫 time-bounded completeness）。洗牌則確保探測順序不固定、不會總是同樣的配對。這是一個典型的「隨機看似夠用、但要時間保證就得引入結構」的設計取捨——把純隨機換成「打散順序的輪詢」，用一點點協調換來「沒有人會被無限期忽略」的硬保證。

## gossip 在哪裡騙你：它是「最終」一致，不是即時

gossip 漂亮，但它的保證有明確的邊界，踩過去就會被它騙。最根本的一條：**gossip 提供的是機率性的最終一致，不是即時一致。**

這意味著任何「需要立刻全網一致」的邏輯，都不能建在 gossip 上。前面算過,一千台叢集裡一條成員變更要約 10 秒才擴散完。在這 10 秒的窗口裡，不同節點對「D 還活著嗎」會有**不一致的看法**——有些已經知道 D 死了，有些還沒收到消息。如果你的系統需要在某個瞬間對「叢集成員到底是哪些」做出一個**全網公認、不容兩種答案**的判定（譬如要選出唯一的 leader、要判定某個分散式鎖現在歸誰），gossip 的這個收斂延遲就是致命的——這類「必須當下強一致」的決定，得交給共識（見〈共識〉），不能交給 gossip。

更尖銳的故障模式出現在**網路分區**時。叢集被切成兩半、兩側無法互通，於是兩側各自 gossip，各自演化出一份「誰活著」的視圖——左邊認為右邊那些節點全死了，右邊認為左邊全死了。**兩份成員視圖在分區期間是矛盾的，而 gossip 本身沒有任何機制阻止這種矛盾。** 這不是 bug，恰恰是 gossip 用一致性換來可用性的代價：它選擇「分區時兩邊都繼續運作、各自有一份可能過時的視圖」，而不是「停下來等一個全網一致的答案」。如果你需要的恰恰是後者——分區時絕不能有兩個 leader、絕不能腦裂——那你需要的工具是共識，不是 gossip。把 gossip 的成員視圖當成強一致的真相來源，正是這個誤用的核心。

故障偵測的調參也藏著一個對立的張力。把「判定 suspect」的門檻調得越敏感（少等幾個週期、間接探測的 k 取小一點），故障偵測得越快，但在尖峰流量、CPU 吃緊、訊息處理變慢的時候，越容易把一票**其實活著、只是暫時慢**的節點誤標成 suspect 甚至 dead——而誤標一片節點 dead 會觸發不必要的資料重平衡、流量重導，反過來讓系統更慢、製造更多誤判，形成惡性循環。反過來把門檻調得越保守，誤判少了，但真正死掉的節點要拖很久才被發現。**偵測速度與誤判率是一組此消彼長、沒有最佳解的取捨**，只能根據叢集的網路品質與容忍度去調——這也正是 HashiCorp 在 SWIM 之上做 Lifeguard 那一層的動機：在節點自己「感覺被冤枉」時放慢對它的判定，把誤判率壓下去而不犧牲太多偵測速度。

## 為什麼是這個形狀

退一步看，gossip 的整個樣貌，都是從一個拒絕妥協的需求裡長出來的：**要在沒有中央、節點會死、訊息會丟的大叢集裡傳播狀態，而且不准有任何單點。**

正因為不准有單點，它不能信任任何一台機器去當花名冊或傳播中樞——所以每台節點地位對等、各自隨機找對端。正因為要在不可靠的網路上保證消息傳得到，它不能靠固定路徑——所以靠隨機性製造冗餘，一條路不通自然走另一條。正因為要快，它讓被感染者再去感染別人——所以擴散是指數的、回合數是對數的。正因為 rumor 會有極小機率漏網、會累積漂移，它得有一個 anti-entropy 的後台去全面對帳——而對帳又不能把頻寬燒光，所以用 Merkle tree 讓「一致時幾乎免費、只為分歧付費」。每一層都是前一層欠下的債被下一個機制接手償還。

但這條設計路線是有代價的，而且這個代價無法用更聰明的工程消除——它是用「即時一致」換來的「規模與可用性」。gossip 給你的是一份**最終會對、但此刻可能還沒對齊、分區時甚至會矛盾**的全局視圖。當你的系統能接受「秒級收斂、偶有短暫不一致」，gossip 讓你輕鬆撐到數千、數萬節點而沒有任何瓶頸；當你的系統需要「此刻就要一個不容兩種答案的權威判定」，gossip 就不是答案——那是共識守的那扇門。看懂這條分界，你就懂了為什麼一個成熟的大型叢集，往往讓 gossip 去管那海量、可以慢慢趨同的成員與狀態，而把少數幾件「絕不容含糊」的協調決定，留給旁邊那個更貴、更慢、但絕不矛盾的共識中樞。
