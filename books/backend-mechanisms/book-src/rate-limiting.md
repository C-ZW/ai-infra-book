# rate limiting：token bucket、leaky bucket 與分散式限流

設想一個再尋常不過的早晨。你的 API 平時每秒接幾百個請求，後端綽綽有餘。然後某個合作方的整合程式出了 bug——它在一個迴圈裡忘了等回應，開始以每秒上萬次的頻率重打同一個端點。你的服務沒有崩，但它開始幫這一個失控的客戶端做白工：連線池被佔滿、資料庫排起長隊、其他正常使用者的請求開始逾時。一個下游的小錯誤，正在透過你的服務放大成一場全域的災難。

你需要的不是「等系統撐不住時才反應」，而是在門口就立一個牌子：**這段時間內，我最多讓你進這麼多，多的請回頭。** 這就是 rate limiting（限流）。它聽起來像一句廢話——「請求太多就擋掉」——但魔鬼全藏在「太多」這兩個字怎麼定義、計數狀態放在哪裡、以及在一個有十台機器的叢集裡這個「太多」要怎麼算才不自相矛盾。這一章把限流講透：兩個經典演算法各自在容不容許突發上做的取捨、它們其實是同一枚硬幣的兩面、以及當你從一台機器擴展到一整個叢集時，那個看似簡單的計數器會怎麼背叛你。

## 為什麼「擋掉太多」這件事不簡單

先把問題講難。限流要回答的第一原理問題是：**在這段時間窗內，我允許多少工作進來，多出來的怎麼辦。** 天真的答案是「數一數這分鐘進來幾個，超過 100 就拒」。這個答案有個藏得很深的破綻，我們待會兒會看到它讓你在窗的邊界瞬間吃進兩倍流量。

但更根本的張力在於「突發」。人類使用一個 API 從來不是均勻的。一個使用者開啟頁面，前端可能在 200 毫秒內並發打十個請求，然後安靜三十秒。如果你的限流規則是死板的「每秒最多 3 個」，這個完全正常的開頁行為就會被你誤殺一半。可是反過來，如果你為了容忍這種突發而把上限拉到「每秒 10 個」，那個失控的客戶端就能持續以每秒 10 個的速度灌爆你——你想擋的長期濫用反而漏了。

所以限流真正要拿捏的，是一個雙參數的問題：**長期的平均速率該壓在哪裡，以及瞬間能容許多大的一陣突發。** 把這兩件事分開想，是理解所有限流演算法的鑰匙。兩個經典演算法——token bucket 與 leaky bucket——本質上就是對「容不容許突發」這個問題給出的兩種相反答案。

順帶先劃清一條界線，因為這兩件事極常被混為一談：限流（rate limiting）和背壓（backpressure，見〈背壓〉）是互補但方向相反的兩個機制。背壓是**被動、事後**的——下游真的滿了，才把「慢下來」的訊號沿管線往上游傳。限流是**主動、事前**的——你用一個預先設定的速率上限，在請求還沒碰到任何下游資源之前，就在門口決定放行或攔截。背壓問的是「下游現在到底忙不忙」，限流問的是「不管下游忙不忙，這個來源這段時間配額用完了沒」。最穩的系統兩者都有：限流擋可預期的配額濫用，背壓防無法預期的下游變慢。這一章只講限流；計數狀態如何在叢集間同步是它獨有的難題，也是本章的重頭戲。

## token bucket：用一桶可累積的令牌容許突發

token bucket（令牌桶）的機制小到可以一句話講完，但它的行為值得在腦中放慢重放一遍。

想像一個桶，容量上限是 `b` 個令牌。有一個固定的補充速率 `r`，系統以每秒 `r` 個的速度往桶裡放令牌——但桶滿了就溢出、不再累積。每來一個請求，就從桶裡拿走一個令牌；桶裡還有令牌就放行，桶空了就限流（拒絕或讓它等）。

機制的全部精妙都在「桶能累積」這四個字。當系統閒置一陣子，令牌會一路補到桶滿（最多 `b` 個）。這時候突然來一陣突發，前 `b` 個請求可以**瞬間**全部放行——因為它們花的是之前閒置時攢下的令牌。突發過後，桶空了，後續請求就只能以補充速率 `r` 的節奏一個一個放行。於是 token bucket 給出的保證是兩層的：**長期平均速率被牢牢釘在 `r`，但瞬間容許一陣最大為 `b` 的突發。** 參數 `r` 管長期、`b` 管突發，剛好對上前一節那個雙參數問題。

來手算一遍，把數字織進去看清楚。設補充速率 `r = 100 tokens/s`、桶容量 `b = 200`。系統閒置了 3 秒——注意這裡有個容易算錯的地方：3 秒以 100/s 的速率本該補 300 個令牌，但桶**有容量上限**，所以它只會累到 200 就停，那多出來的 100 個令牌憑空蒸發了。現在來一波 250 筆同時到達的突發：

- 前 200 筆立刻放行，用光桶裡攢的 200 個令牌。
- 剩下 50 筆，此刻桶空了，只能等補充。令牌以 100/s 補充，等於每 10 毫秒補一個。如果這 50 筆願意排隊等，它們會在約 `50 × 10 ms = 500 ms` 內陸續放行；如果它們不等（典型的 API 限流就是不等、直接拒），這 50 筆當場被限流、回 429。

長期看，無論突發怎麼來，這個桶的通過率被釘死在 100/s——`b` 只決定「一口氣能吞多大一陣」，`r` 決定「長跑的配速」。這就是為什麼 token bucket 是對外 API 限流壓倒性的預設選擇：它天然吻合人類那種「一陣一陣、間有空檔」的使用節奏。

這裡藏著一個非顯然的陷阱，值得單獨點出來：**把 `b` 設得太大，等於在瞬時尺度上根本沒有限流。** 如果你為了「對使用者友善」把桶容量設成 `b = 100000`，那麼一個攻擊者只要先讓系統閒置攢滿桶，就能一口氣放進十萬個請求——這一瞬間的洪峰可能就足以打垮你的下游，而你的「長期平均速率」統計上還很漂亮。`r` 和 `b` 必須**一起**想清楚：`r` 對應你下游能長期承受的速率，`b` 對應你下游能一口氣吸收的尖峰。只調一個是常見的設定錯誤。

## leaky bucket：把輸出磨成一條平直的線

leaky bucket（漏桶）是另一個極端。它最容易引起混淆，因為「漏桶」這個名字底下其實藏著**兩種不同的東西**——分清楚這兩種，是這一節真正的價值，也是很多人講不清楚的地方。

第一種，把它當成一個**佇列（as a queue）**。請求進入一個固定容量的佇列，佇列底部有個洞，以**固定速率**漏出（被處理）。桶滿了，新來的請求就溢出、被丟掉。這個版本的關鍵是：被接受的請求會被**延遲**——一陣突發進來會被攤平、一個一個間隔著送出去，輸出端看到的是一條絕對平直的恆定速率線，沒有任何瞬間尖峰。代價也很清楚：請求要排隊、延遲增加；桶滿了就丟。它本質上就是「有界佇列 + 固定速率消費」，目的是把 bursty 的輸入整形（shape）成 smooth 的輸出。

第二種，把它當成一個**計量器（as a meter）**。這裡沒有真的佇列、沒有人被延遲。你維護一個虛擬的「水位」計數器：每來一個請求，水位加一；水位以固定速率隨時間流逝而下降。如果一個請求會讓水位**溢出**容量上限，這個請求就被判為不合規、當場拒絕（不是排隊，是直接擋掉）。

這兩種的差別不是文字遊戲：**as a queue 會延遲請求（整形），as a meter 會拒絕請求（管制）。** 一個讓你等，一個讓你滾。當人們說「leaky bucket 不容許突發、輸出絕對平滑」，他們講的是 queue 版本；當人們說「leaky bucket 和 token bucket 其實是對偶的」，他們講的是 meter 版本。把這兩個混在一起，整段討論就會打結。

選 leaky bucket（queue 版）而非 token bucket 的場景很具體：當你的下游**只能吃恆定速率**、一點突發都受不了的時候。最典型的是你要對外投遞 webhook（見〈Webhooks〉），對方可能對你做了嚴格的限流，你寧可自己這端把投遞磨成平穩的每秒 N 筆，也不要一陣突發打過去被對方擋掉。token bucket 容許突發的優點，在這種「下游零容忍突發」的場景反而是缺點。

## 其實是同一枚硬幣：meter 版漏桶與令牌桶的對偶

把上一節的 meter 版漏桶和 token bucket 並排放，會看到一件漂亮的事：**它們在數學上是同一個東西，只是從相反方向描述。**

token bucket 是「令牌往上加、請求把它消耗掉」；meter 版 leaky bucket 是「水位往上加、時間把它漏掉」。一個追蹤「還剩多少額度」，一個追蹤「已經用掉多少」——而「剩餘 = 容量 − 已用」。同一個閒置攢額度、突發消耗額度、長期受固定速率約束的行為，換個座標就從一個變成另一個。理解了這個對偶，你就不會再把「到底該用哪個」當成一個玄學問題：對於「容許突發、超量就拒絕」這個語意，token bucket 和 meter 版 leaky bucket 給的是**一模一樣的**放行決定。

這個對偶有一個極實用的副產品，叫 **GCRA（Generic Cell Rate Algorithm，通用信元速率演算法）**。它源自 ATM 電信網路的流量管制（「cell」指的是 ATM 的信元），但它是今天高效限流器底下最常見的引擎之一。GCRA 的洞察是：你根本不需要真的存「現在桶裡有幾個令牌」這個會隨時間變化的數字，你只需要存**一個時間戳**——「理論上下一個合規請求最早該在什麼時候到達」，稱為 TAT（Theoretical Arrival Time，理論到達時間）。

它的運作可以在腦中重放：設每個請求之間理想的間隔是 `T = 1/r`（emission interval，發射間隔），再給一個容許提早的寬容量 `τ`（tolerance，對應 token bucket 裡能攢的突發額度）。一個請求在實際時刻 `t_a` 到達時，判斷規則只有一條比較：

- 若 `t_a ≤ TAT − τ`，代表它來得太早、超出了寬容量 → **拒絕**，TAT 不動。
- 否則（即 `t_a > TAT − τ`）→ **放行**，並把 TAT 更新為 `max(t_a, TAT) + T`（從現在或原本的理論時間往後推一個發射間隔）。

整個演算法只讀寫一個時間戳，沒有「隨時間補令牌」這種需要持續模擬的動作——閒置時它完全不耗 CPU，因為時間自己會流逝，TAT 和現在的差距自然反映出攢了多少額度。這就是為什麼 GCRA 在需要極省記憶體、極省計算的大規模限流（每個 key 只存一個 8 bytes 時間戳）裡如此受歡迎。它不是第三種演算法，而是 token bucket 那個對偶在工程上最精煉的化身。

## 固定窗的邊界突刺，與滑動窗怎麼補

前面欠下的那筆帳——天真的「數一數這分鐘進來幾個」——現在來還。這種做法叫 **fixed window counter（固定窗計數）**：每個固定時間窗（比如每分鐘）配一個計數器，請求進來就 `+1`，超過上限就拒，窗一翻計數器歸零。它的好處是實作極簡、O(1) 記憶體，只要一個整數。

它的破綻藏在窗的**邊界**。假設上限是「每分鐘 100」。一個攻擊者在 `0:59`（第一個窗的最後一秒）打進 100 筆——合規，因為這個窗還沒滿。然後在 `1:00`（第二個窗的第一秒）再打進 100 筆——也合規，因為新窗的計數器剛歸零。結果是：在橫跨兩窗邊界的**短短 2 秒內**，通過了 200 筆，是你名目上限的兩倍。固定窗在任何一個窗內都守規矩，但它在邊界上對「真正的滑動 60 秒」這個直覺撒了謊。對於只是粗粒度防呆的場景，這個 2 倍突刺無傷大雅；但若你的上限是要拿來對抗濫用或當作硬性付費配額，這個破綻會被人精準地利用。

要堵住它，就要讓「窗」真的跟著當下時刻滑動。最精確的做法是 **sliding window log（滑動窗日誌）**：記下**每一筆**請求的時間戳，每次新請求進來時，把超過 60 秒的舊時間戳清掉，數一數剩幾筆，未滿上限才放行並記下這一筆。它給的是逐筆精確的滑動窗、完全沒有邊界突刺——代價是每個 key 要存一串時間戳，高流量下記憶體會膨脹。

實務上更常見的折衷是 **sliding window counter（滑動窗計數，近似版）**，它用上一個窗的計數**按比例加權**來逼近真正的滑動窗，既擋住邊界突刺、又只要存兩個整數。Cloudflare 公開過它的公式與一個漂亮的具體例子：上限設每分鐘 50，當前這一分鐘已經過了 15 秒、收到 18 筆，上一整分鐘收了 42 筆，那麼估算的速率是

```
rate = previous_count × (1 − elapsed_fraction) + current_count
     = 42 × ((60 − 15) / 60) + 18
     = 42 × 0.75 + 18
     = 49.5
```

49.5 < 50，所以這一筆放行。直覺是：當前窗才走了 15 秒，所以「過去這整整 60 秒」裡有 45 秒落在上一個窗，就把上一個窗的 42 筆乘上 0.75 權重算進來。它假設上一個窗內請求是均勻分布的，所以是近似——但 Cloudflare 在 4 億個請求上的實測顯示平均誤差約 6%、只有 0.003% 的請求被誤判，在生產上精度足夠，而省下的記憶體相當可觀。這就是為什麼大規模限流器普遍選這個近似版，而不是逐筆精確的 log 版。

## 一台機器到一個叢集：計數器在哪裡

到目前為止，所有演算法都假設那個桶、那個計數器、那個 TAT 時間戳，住在**一台機器**的記憶體裡。一旦你的服務水平擴展成十台機器、流量被負載均衡器（見〈負載均衡〉）打散到各台，限流真正的難題才登場：**那個本該全域唯一的配額計數，現在到底放在哪裡？** 這正是本章的核心，也是限流從一個小演算法升級成一個分散式問題的地方。

有三條路，每條都在「精確」和「成本」之間站在不同的位置。

**第一條：本地計數、配額分攤。** 最省事的做法是把全域配額除以機器數，每台機器在本地獨立限流。全域要限 1,000 RPS、有 10 台機器，那每台就本地限 100 RPS，各算各的、零協調、零額外延遲。它的破綻是流量**不均**時會誤殺。負載均衡若用某種 hash 導流造成傾斜，某一台實際吃到 180 RPS——它會在自己的 100 RPS 處就開始回 429，誤殺掉 80 RPS，而**此刻其他機器可能總共才用了 600 RPS、全域離 1,000 還遠得很**。這就是本地分攤用「省掉協調」換來的代價：全域配額沒用滿，局部卻已經在拒絕。而且機器數一變動（自動擴縮容加減了幾台），每台的配額就得重算，否則總和會偏離全域目標。

**第二條：中央計數。** 把計數狀態集中放到一個共享儲存——最常見的就是 Redis——所有機器對同一個 key 讀寫計數。它給你全域精確：十台機器看到的是同一個真相，配額一分不多一分不少。代價是每個請求多一次網路往返去問 Redis，而且那個 Redis 變成了單點與熱點——所有流量的限流決定都壓在它身上。

中央計數這條路上有一個**極其容易踩、且在分散式下幾乎必然出事**的陷阱，必須單獨講透：**「先讀、再判斷、再寫回」這三步不是原子的。** 想像兩台機器幾乎同時處理請求，計數現在是 99、上限 100。機器 A 讀到 99，判斷「未滿，可放行」；機器 B 在 A 寫回之前也讀到 99，同樣判斷「未滿，可放行」。兩個都放行了，然後各自把計數寫成 100——可是實際上過了兩個請求，計數卻只加了一次，真實值該是 101。這是一個經典的 race condition（見〈race condition 與原子性〉）：你的限流器在高並發下會**系統性地超發**，而且越並發漏得越多。

堵法是把「讀-判-寫」收成**一個不可分割的原子操作**。在 Redis 上有兩種標準做法。簡單的計數型限流可以直接用 `INCR`——它回傳遞增後的值，第一次遞增時順手 `EXPIRE` 設個窗的 TTL，靠「先加再看回傳值有沒有超」把判斷併進那一次原子遞增裡。但要注意 `INCR` 與 `EXPIRE` 是兩個指令、合起來並不原子——若程序在 `INCR` 成功後、`EXPIRE` 送達前崩潰（或 `EXPIRE` 封包遺失），這個 key 就永遠不過期、計數永不歸零，等它一旦達到上限，這個來源就被永久卡死。穩當的做法是連這個 TTL 設定也一起包進 Lua（或用帶 `NX`／`EX` 的單一指令先把 key 連同過期時間建好、再遞增），讓「遞增＋首次設過期」也成為一步。而 token bucket 或 GCRA 這種需要「讀出當前令牌數、按流逝時間補、判斷、扣減、寫回」多步邏輯的，單一指令做不到——這時用 Redis 的 **Lua 腳本**：整段腳本在 Redis 裡原子執行、中途不會被別的指令插隊，於是那一串「讀-算-判-寫」對外看起來就是一步。這也是為什麼幾乎所有正經的 Redis 限流器都是一段 Lua 腳本，而不是在應用程式裡 `GET` 完再 `SET`——後者那個看似無害的兩步，在十台機器並發時就是個會漏配額的洞。（在 Redis Cluster 下還有一層約束：一段腳本碰到的所有 key 必須落在同一個 slot，所以單 key 的限流演算法天然安全，跨多 key 的要用 hash tag 把它們綁到同一個 slot。）

**第三條：本地擋 + 週期同步。** 前兩條的折衷。每台機器先用本地的桶擋掉明顯超量的請求（省掉大部分往返），背景再週期性地（比如每秒一次）向中央回報自己用了多少、領取剩餘的全域配額。它用「同步間隔內短暫的不精確」換掉了每個請求都要協調的成本——代價是在那個間隔內，叢集整體有可能短暫地略微超發。實作比前兩者都複雜，但在「規模大到中央計數扛不住每請求往返、但又不能容忍本地分攤那麼粗」的場景，這是常見的工程解。

怎麼選這三條，取決於你的配額是哪一種。如果配額是**硬合約**——比如付費方案「每月 100 萬次呼叫」的額度，超了要跟客戶算錢——那你需要中央精確計數，差一筆都不行。如果配額是**保護性軟限**——純粹為了防止某個來源打垮你、差個幾趴根本無所謂——那本地分攤就夠了，而且更便宜更快，何苦為了防雪崩去扛一個 Redis 單點。把保護性軟限做成中央精確計數，是一種常見的過度工程：你為了一個「差幾趴沒人在乎」的目標，引入了一個新的單點、一次額外往返、和一整套原子操作的複雜度。

## 被拒絕之後：429 與一個常被漏掉的 Retry-After

限流的最後一哩，是「擋下來之後怎麼跟客戶端說」——這一步做不好，限流會反過來變成壓垮你的那根稻草。

被限流的請求，標準回應是 HTTP **429 Too Many Requests**（由 RFC 6585 於 2012 年定義）。但光回 429 是不夠的，而且這個不夠之處藏著一個會自我惡化的迴圈。設想客戶端收到一個光禿禿的 429、沒有任何「該等多久」的指引——它只能瞎猜，而多數天真的客戶端會做最糟的事：**立刻重試。** 立刻重試的請求又被限流、又回 429、又立刻重試……你的限流器本來是要保護系統的，現在卻變成了一台製造重試風暴的引擎（見〈retry storm 與 metastable failure〉）。

破法是在 429 裡帶上 `Retry-After` 標頭——這是 HTTP 標準的一部分（RFC 9110 第 10.2.3 節定義其語意），值可以是「幾秒後再試」或一個明確的時間點。它把「該等多久」這個只有伺服器算得出來的資訊（你的桶什麼時候會補出下一個令牌？GCRA 的 `TAT − τ` 落在哪一刻？）直接告訴客戶端，客戶端就不必瞎猜、也沒有藉口立刻重打。一個帶 `Retry-After` 的 429 和一個不帶的 429，在「會不會引發重試風暴」上是天壤之別。限流的回應不只是「拒絕」，它得是「拒絕 + 明確的退避指引」——少了後半句，你只是把過載換了個形式而已。

## 放在哪一層：閘道還是應用

最後一個常見的取捨：限流該放在 API gateway（見〈API gateway 的角色〉）還是放在應用程式內部？這不是二選一，而是兩層常常並存、各擋一種濫用。

放在閘道層，擋得**早**——請求還沒進到你的應用、還沒消耗任何應用層資源（連線、執行緒、記憶體）就被攔下，省力。但閘道看到的資訊粗：它知道來源 IP、API key、端點路徑，卻不知道應用層的細粒度語意——比如「這個特定操作其實很貴、該用更嚴的配額」「這個請求會觸發一個無快取的複雜查詢」。放在應用層，擋得**晚**（請求已經進來了、佔了點資源），但它能根據業務語意做精細的、per-operation 甚至 per-tenant 的限流。

實務上的分工通常是：閘道層擋掉粗暴的、量大的、明顯異常的洪峰（純看速率與來源），應用層補上那些需要懂業務才擋得準的細活。兩層各守一段，而不是指望單一一層把所有濫用都認出來。

## 為什麼是這個形狀

退一步看，限流的整個樣貌都是從一個張力裡長出來的：**你的下游能承受的處理能力是有限且大致可估的，而上游的流量既不均勻、也未必善意。**

正因為流量天生一陣一陣，所以限流不能只管平均速率，得把「長期配速 `r`」和「瞬間突發 `b`」拆成兩個旋鈕——這就是 token bucket 的形狀。正因為有些下游一點突發都吃不下，所以還需要一個能把輸出磨平的 leaky-bucket-as-queue。正因為這兩種需求其實是同一道數學的兩面，所以才有 GCRA 這種「只存一個時間戳」的精煉化身。正因為天真的固定窗會在邊界撒謊，所以才演化出滑動窗。而當這一切從一台機器擴展到一個叢集，那個本該唯一的計數器立刻逼你在「全域精確但有單點與往返」和「本地省事但會誤殺或超發」之間選邊站，並且逼你把「讀-判-寫」收成一個原子操作，否則高並發會讓你的配額靜靜地漏掉。最後，連「拒絕」本身都不是終點——一個不告訴客戶端何時再試的 429，會把你精心設計的防線變成重試風暴的火種。

限流不是一道「擋掉太多請求」的單行指令，而是一連串「想要什麼保證、願意付什麼代價」的選擇：要不要容許突發、配額要多精確、計數放哪裡、被拒之後怎麼引導。把這幾個選擇想清楚，你就能向任何一個工程師講明白，為什麼你的限流器長成它現在這個樣子。
