# durable execution：Temporal、Step Functions 與長流程

一筆訂單的人生可以很長。使用者按下結帳，系統要保留庫存、向第三方請款、等對方非同步回傳結果、扣庫存、通知倉庫揀貨、等倉庫掃描出貨、寄出確認信、若三十分鐘沒付款就釋放保留——這條線從幾秒到幾天都有可能，中間還夾著好幾段「等別人」。現在把這條線寫成程式。最自然的寫法是一個函式：

```
async function placeOrder(order) {
  reserveInventory(order)
  const payment = await chargeCard(order)   // 第三方，可能要等幾秒到幾分鐘
  if (!payment.ok) { releaseInventory(order); return fail() }
  deductInventory(order)
  notifyWarehouse(order)
  await waitForShipment(order)               // 等倉庫掃描，可能幾小時到幾天
  sendConfirmationEmail(order)
}
```

這段程式碼讀起來很清楚，問題只有一個：它**活不過一次重啟**。`chargeCard` 還在等第三方回應時，部署滾動把這個行程換掉了；或者機器當機、OOM killer 把它殺了。函式的呼叫堆疊、區域變數、「我走到第幾行了」——全部在記憶體裡，行程一死就蒸發。重新起來的新行程不知道這筆訂單已經請過款，要嘛從頭重跑（請第二次款），要嘛根本沒人接手（訂單永遠卡在半路）。一個跨越行程生命週期的長流程，和「程式的執行狀態只活在記憶體」這件事，天生互斥。

durable execution 這套機制要解的，就是這個互斥：**讓一段普通的程式碼，在它執行的中途崩潰、被殺、被換版之後，能從斷掉的那一行繼續，而不是從頭重來、也不是卡死。** 它要把「程式走到哪、每一步拿到什麼結果」這件原本只活在記憶體裡的東西，變得像資料一樣持久（durable）。

## 天真做法：把流程狀態塞進資料庫欄位

在 durable execution 這個詞變成主流之前，每個團隊都自己發明過一套土法。最常見的版本是：**把流程的狀態塞進資料庫的一個欄位，再用一堆背景 job 去推它。**

訂單表加一欄 `status`，取值在 `reserved` → `charging` → `paid` → `deducted` → `shipping` → `done` 之間遷移。每寫完一步就 `UPDATE order SET status=...`。然後跑一個 cron，每分鐘掃一遍：看到 `charging` 卡太久的去查請款結果、看到 `paid` 的去扣庫存、看到 `expire_at` 過了還沒付的去取消。流程不再活在某個行程的記憶體裡，而是攤在資料庫的一張表上，靠輪詢一格一格往前挪。

這個做法能跑，但它有兩個一旦規模上來就會反咬的洞，值得看清楚，因為理解它們，才理解 durable execution 到底買了什麼。

**第一個洞：卡住沒人推。** 流程的推進力完全來自那個 cron。cron 掛了一段時間、或它掃的 query 因為表變大而變慢、或某個狀態根本沒寫對應的推進規則——一批訂單就會靜靜地卡在某個中間狀態，沒有任何人會主動發現，直到客訴進來。「狀態機散在 cron 規則裡」的本質問題是：**沒有一個地方，能讓你回答「這筆訂單現在到底走到哪、為什麼不動了」。** 狀態欄位只告訴你它停在哪一格，不告訴你它怎麼來的、卡在等什麼。

**第二個洞：重做已完成的步驟。** cron 為了不漏，往往會重疊跑（上一輪還沒掃完，下一輪又啟動了），或者多個實例同時跑。同一筆 `charging` 的訂單，可能同時被兩個 cron 實例撈到，各自去查請款、各自往下推。如果推進動作不是冪等的，你就請了兩次款、扣了兩次庫存。於是你被迫在每一步都手寫冪等與去重——這是對的、但它把本該簡單的「依序做幾件事」變成處處要防重入的雷區。

退一步看，這套土法其實是在**手刻一個極簡的工作流引擎**：狀態欄位是它的執行歷史，cron 是它的排程器，你寫的冪等檢查是它的恢復邏輯。只是這個自製引擎沒有人專門維護，每個團隊重刻一遍、每次都漏掉幾個邊角。durable execution 就是把這個你本來要手刻的東西，抽出來、做對、下沉成平台保證。

## 機制登場：持久化的，是「執行本身」

durable execution 的核心轉念，一句話講完：**它持久化的不是「流程的狀態」，而是「程式的執行過程」。**

差別在哪？土法持久化的是 `status='paid'` 這種**結果快照**——它記得你現在在哪，但忘了你是怎麼來的。durable execution 持久化的是一份**事件歷史（event history）**：一條 append-only 的日誌，逐筆記下「工作流啟動，輸入是 X」「排程了 activity `chargeCard`」「`chargeCard` 完成，回傳 `{ok:true, txn:'abc'}`」「啟動了一個 30 分鐘的計時器」「計時器觸發」……流程每做一個有外部後果的決定，引擎就往這份歷史追加一筆。

有了這份歷史，恢復就有了著力點。當行程崩潰、工作流在另一台 worker 上重新被拉起，引擎做的事叫**重放（replay）**：它從頭把你的工作流函式再跑一遍，但每跑到一個「該呼叫外部步驟」的點，它不真的去呼叫，而是**從事件歷史裡取出當初那一步的結果**，直接餵回去。於是你的函式被「快轉」——`reserveInventory` 不重做（歷史裡有它的結果）、`chargeCard` 不重做（歷史裡有 `txn:'abc'`）——一路快轉到崩潰前的那一行，然後從那裡，第一次真正地往下執行。

```
event history（append-only，每筆都持久化）：

  1  WorkflowStarted          input = order#1234
  2  ActivityScheduled        reserveInventory
  3  ActivityCompleted        reserveInventory -> ok
  4  ActivityScheduled        chargeCard
  5  ActivityCompleted        chargeCard -> {ok, txn:abc}
  6  TimerStarted             30m
  --- 行程在此崩潰 ---
  --- 在另一台 worker 重啟，從事件 1 開始 replay ---
  replay: reserveInventory  → 不呼叫，從事件 3 取回 ok
  replay: chargeCard        → 不呼叫，從事件 5 取回 {ok, txn:abc}
  replay: 30m timer         → 從事件 6 知道計時器已在跑、不重新起一個
  → 快轉到崩潰前那一行（sleep 處）；計時器若還沒到期就在此等剩下的時間，
    到期（歷史多一筆 TimerFired）才往下第一次執行
```

對你寫的那段程式碼來說，它**感覺不到自己崩潰過**。它就是一個從上到下、`await` 一步等一步的普通函式；崩潰與恢復發生在它腳下的引擎層，被 replay 完全藏了起來。這就是「durable」的字面意思——執行狀態像資料一樣持久，跨得過行程的生死。

## 為什麼工作流碼必須是「決定性」的

replay 這個機制，悄悄地對你的程式碼提了一個嚴格的要求，而這個要求是整套機制最容易被誤用、也最該講透的地方：**工作流函式必須是決定性的（deterministic）——同樣的輸入、同樣的歷史，每次重放都必須走出完全一樣的步驟序列。**

為什麼非得如此？因為 replay 的整個邏輯，建立在「函式這次重跑時做的決定，和當初第一次跑時做的決定，逐一對得起來」之上。引擎重跑你的函式，期待第 4 步排程的是 `chargeCard`，於是去歷史的第 4、5 筆取它的結果。如果你的函式這次重跑時，因為某種隨機性，第 4 步排程的卻是 `sendEmail`——歷史對不上了，引擎不知道該把哪個結果餵回哪一步，整個恢復就地崩潰（Temporal 會報 non-deterministic error 讓這次 replay 失敗，工作流就此卡住推不動——worker 會反覆重試這個失敗的工作流任務、流程不回應 query，直到有人介入修正——而不是悄悄給你一個錯亂的狀態）。

於是有一類在普通程式裡天經地義的操作，在工作流碼裡是禁忌：

- **讀牆上時鐘**：`new Date()`、`time.Now()`——每次重放都給不同的值，路徑就可能分岔。
- **取隨機數**：`Math.random()`、`rand.Int()`——同理，replay 時生出不同的數，決定就變了。
- **直接打 I/O**：直接發 HTTP、讀寫資料庫、讀環境變數——這些的回應在 replay 當下和原本那次不會一樣，而且 replay 本來就不該真的再打一次外部世界。

這些操作不是不能做，而是**不能在工作流碼裡裸著做**。它們必須被推進一個叫 **activity** 的封裝裡。activity 是「真正接觸外部世界的那一步」——請款、寄信、查庫存都是 activity。引擎執行 activity 時，把它的**回傳結果記進事件歷史**；於是 replay 時，這個結果就從歷史取回，而不是重新去問外部世界。換句話說：**所有不決定性的東西，都要先被執行一次、把結果釘進歷史，之後 replay 就有一個固定的答案可取。** 連「現在幾點」「擲個骰子」這種事，引擎也提供了決定性的版本——它在第一次執行時取一次值、記進歷史，replay 時回放同一個值（Temporal 的 `workflow.Now()`、`workflow.Sleep()` 走的就是這條路：時間從工作流上下文讀、與歷史對齊，計時器是歷史裡的一筆事件、replay 時不會重新等）。

這條 determinism 約束，是 durable execution 「免費的可恢復性」背後那張帳單。你換來了「不必手寫狀態機與重試」，代價是「工作流碼要守一套不那麼直覺的紀律」。多數踩雷都踩在這裡：一個工程師在工作流裡順手寫了 `if (Date.now() % 2)`，本機測試一切正常，直到某次生產環境的 worker 重啟觸發 replay，整條流程詭異地報出 non-determinism——而它在崩潰前明明跑得好好的。

## 一份歷史會無限長：Continue-As-New

事件歷史是 append-only 的，這帶來一個必然的後果：**一個跑很久、步驟很多的工作流，它的歷史會無止境地長下去。** 而歷史不能無限大——每次 replay 都要把整份歷史重放一遍，歷史越長、恢復越慢；而且它得整個載進記憶體。

Temporal 給這份歷史劃了一條硬線（2026-06）：**單一工作流的事件歷史上限是 51,200 筆事件或 50 MB，在達到 10,240 筆事件或 10 MB 時就會發出警告。** 一個天真寫成「無限迴圈、每圈做一點事」的工作流——比如一個每分鐘輪詢一次、跑好幾個月的監控流程——會穩穩地撞上這條線，然後死掉。

解法叫 **Continue-As-New**：工作流主動把「現在這一輪」結束，**原子地**開一個全新的工作流實例接手，沿用同一個 workflow ID，但帶著一份**全新的、空的事件歷史**，只把必要的狀態（目前進度、累計值）當作新實例的輸入傳過去。對外看，它還是同一個邏輯流程在跑；對內，它週期性地把背在身上的歷史包袱卸掉、重新開始累積。這個「原子地結束舊的、開始新的」很關鍵——它避免了你手動「關掉再開一個」之間的那條縫，在那條縫裡崩潰會讓流程消失或重複。

這條 51,200 的硬線，是 durable execution 一個很實在的設計約束：**它擅長的是「有限步驟、會跑很久（因為夾著長等待）」的流程，不是「無限步驟」的流程。** 把一個本質上永不停止、不斷累積事件的東西硬塞進一個工作流，遲早撞牆；該用 Continue-As-New 切段，或者根本就不該用工作流來表達。

## 一個帶數字的取捨：等三十分鐘該不該付出一個執行緒

回到開場那個「下單後逾時三十分鐘未付款則取消」的需求，用土法和 durable execution 各算一遍，差別會具體起來。

**土法**：下單時寫 `status='awaiting_payment'`、`expire_at = now + 30min`，再跑一個 cron，每分鐘掃一次「`expire_at < now` 且仍 `awaiting_payment`」的訂單去取消。假設系統裡同時有 **10 萬筆**在途訂單各自等著三十分鐘。那麼這個 cron **每分鐘**都要對訂單表跑一次輪詢查詢，從這 10 萬筆裡撈出剛好到期的那幾筆。就算 `(status, expire_at)` 上有複合索引、查詢只命中到期的那幾筆，**這個每分鐘一次的輪詢本身就是固定開銷**——連線、查詢計畫、索引 I/O 都得付，而且查得到期者的精度被綁死在「每分鐘一次」的輪詢節奏上；一旦少了那個剛好的索引，它就退化成在 10 萬筆裡硬掃。cron 掛了五分鐘，這五分鐘該取消的訂單全部延誤；cron 重疊跑，同一筆到期訂單可能被兩個實例各取消一次，於是取消邏輯又得自己冪等。

**durable execution**：工作流碼直接寫 `await sleep('30min'); if (!paid) cancel()`。那個三十分鐘的等待，不是一個佔著執行緒空轉的 `Thread.sleep`，而是被引擎**持久化成事件歷史裡的一個計時器**。在這三十分鐘裡，沒有任何執行緒、任何記憶體被這個工作流佔用——worker 可以全部重啟、可以擴縮、可以部署三次新版本。三十分鐘到了，引擎精準地喚醒這一個工作流實例，replay 把它接回 `sleep` 的下一行繼續。10 萬筆在途訂單，就是 **10 萬個持久化的計時器**，到期才各自被精準喚醒，**你的業務資料庫上沒有任何一次輪詢掃描**——（精確地說，輪詢沒有消失，而是下沉成引擎內部對它自己的計時器佇列的管理，這個成本被攤平到平台層、不再壓在你的訂單表上）。

把這兩條算式並排，判準就浮出來了：

| | 手刻狀態機 + cron | durable execution |
|---|---|---|
| 「等 30 分鐘」的代價 | 每分鐘對 10 萬筆所在的表輪詢一次找到期者（精度綁死輪詢節奏） | 10 萬個持久化計時器，到期才精準喚醒，業務 DB 零輪詢 |
| 推進靠什麼 | 外部 cron 輪詢（會掛、會重疊） | 引擎在到期/事件到達時精準觸發 |
| 卡住會怎樣 | 靜默卡死，沒人發現 | 流程有歷史可查，看得出卡在等什麼 |
| 重試/冪等 | 每一步自己手寫防重 | 編排邏輯的恢復由引擎兜，activity 仍需冪等 |
| 成本臨界 | 步驟少、等待短時最划算 | 步驟多、等待長、實例多時越划算 |

數字告訴你的不是「durable execution 永遠比較好」，而是：**流程越長、步驟越多、「等待」越久越多、在途實例越多，手刻狀態機的維護與可靠性成本上升得越快，而把恢復下沉成平台保證就越划算。** 反過來，一個只有兩三步、沒有長等待的短流程，套上一整座工作流引擎，是用大砲打蚊子——多一個要運維、要學習的平台，換來你根本用不上的恢復能力。一個「呼叫 A 再呼叫 B」的同步流程，配個逾時加重試就夠了。

## 兩種長相：自帶程式碼的 Temporal，與託管狀態機的 Step Functions

durable execution 不是單一產品，而是一類機制，市面上兩個主流代表走的是相當不同的形態，理解它們的差異，比記住名字重要。

**Temporal**（開源，MIT 授權，由 Uber 內部 Cadence 專案的原班人馬創立的 Temporal Technologies 維護，2026-06 持續活躍）走的是「**工作流即程式碼**」路線。你用 Go、Java、TypeScript、Python 等一般語言寫工作流函式，前面講的事件歷史、replay、determinism 約束、Continue-As-New 全部就是它的機制。它的好處是表達力——迴圈、條件、複雜分支都用你熟悉的語言寫；它的代價是你要運維一個 Temporal 叢集（或付費用 Temporal Cloud），而且要守那套 determinism 紀律。

**AWS Step Functions** 走的是「**工作流即託管狀態機**」路線。你不寫程式碼，而是用一種叫 ASL（Amazon States Language）的 JSON 把流程描述成一台狀態機——每個狀態做一件事（呼叫一個 Lambda、等一段時間、做個選擇），狀態之間用轉移連起來。它是全託管的，沒有叢集要你運維。它的取捨藏在兩種工作流型別的對比裡，而這個對比正是用對它的關鍵：

| | Standard | Express |
|---|---|---|
| 最長執行時間 | 1 年 | 5 分鐘 |
| 執行語意 | exactly-once | 非同步：at-least-once；**同步：at-most-once** |
| 狀態是否跨轉移持久化 | 是（每次轉移都存） | 否 |
| 計費 | 按狀態轉移次數 | 按執行次數 × 時長 × 記憶體 |
| 適合 | 長流程、非冪等關鍵步驟（金流） | 高吞吐、短、可冪等（資料轉換） |

這張表裡有幾個容易被忽略、但會在生產上咬人的細節。

**第一**，Standard 的 exactly-once 指的是「**工作流的狀態轉移**不會被執行超過一次」——它的執行狀態在每次轉移之間持久化，所以崩潰恢復不會讓你重跑已經過的狀態。但這個 exactly-once **不延伸到它呼叫的下游動作**。如果某個狀態呼叫的 Lambda 自己內部對外請了款，而那次呼叫因為網路問題重試了，請款這個真實副作用照樣可能發生兩次——exactly-once 保護的是編排層，不是你的業務副作用。下游動作的冪等，永遠是你的責任（冪等的完整機制見〈重複是常態〉）。

**第二**，Express 不只是「Standard 的便宜短版」，它的語意整個換了一套：非同步 Express 是 at-least-once（執行狀態不跨轉移持久化，崩潰可能讓它整個重跑），而**同步** Express 是 at-most-once（出錯就出錯、不重啟）。把一個非冪等的金流流程拿 Express 跑，等於主動放棄了 Standard 那個「不重複執行」的保證——這是個換錯型別就會悄悄出重複的陷阱。

**第三**，計費模型決定了它們的甜蜜區互斥。Standard 按**狀態轉移次數**計費，所以一個每秒觸發幾千次、每次只有三四步的高頻短流程，用 Standard 會被轉移次數的帳單壓垮——那正是 Express 按執行計費要接的場景。反過來，一個一年才幾千次、但每次跑好幾天的長流程，Express 的五分鐘上限根本裝不下，只能用 Standard。

Temporal 也有它自己的「歷史會爆」問題（前面那條 51,200 事件的硬線），Step Functions 則用 90 天的執行歷史保留（可申請縮短到 30 天以符合法遵）來界定它記多久。兩者用不同的方式，撞上同一道物理：**持久化執行的代價，是你得為「記住每一步」付出儲存與重放的成本，這個成本不可能是零。**

## 它和 saga 是兩層，不是兩個選項

durable execution 最常見的一個誤解，是把它當成 **saga** 的競爭者，二選一。這個誤解值得拆掉，因為它們其實解的是不同層次的問題，而且常常一起用。

saga 是一個**模式**：在沒有跨服務分散式交易的世界裡，怎麼維持「一串跨服務的步驟，要嘛全做完、要嘛把已做的逆序補償掉」的一致性。它的核心是補償交易（compensating transaction）——既然你不能 ROLLBACK 一個已經發生在別的服務裡的動作，那就執行一個語意上抵銷它的動作（請款的補償是退款，扣庫存的補償是補回庫存）。saga 回答的是「**跨服務一致性怎麼維持**」（補償機制的完整討論見〈跨服務的交付一致〉）。

但 saga 這個模式本身，有一個它不回答的問題：**那個依序呼叫各步驟、在失敗時逆序跑補償的「協調者」，自己崩潰了怎麼辦？** 如果協調者在跑到一半時行程死了，誰來確保剩下的補償會被執行完？saga 模式對此沉默——它假設有某種可靠的執行載體在跑協調邏輯。

而 durable execution 正是這個「可靠的執行載體」。它回答的是「**任何長流程——包括一個 saga 的協調者——崩潰後怎麼可靠地恢復執行**」。於是兩者最自然的關係不是二選一，而是**疊起來**：用 durable execution 引擎來實作 saga 的協調者。這樣一來，連「協調者執行補償補到一半掛掉」這種最棘手的情況，都被引擎的 replay 兜住——補償邏輯本身也保證會被跑完。saga 提供「怎麼維持一致性」的模式，durable execution 提供「讓這個模式的協調者可靠運行」的地基。

這也順帶解掉另一個誤解：**「用了 durable execution 就不必想冪等與補償了。」** 不對。引擎兜住的是「編排邏輯的恢復」——它保證你的協調函式會精確地從斷點繼續。但它撤不回**已經對外造成的真實副作用**：信寄出去了、款扣下去了，這些活在外部世界、不在引擎的歷史裡。所以 activity 的副作用仍要冪等（崩潰的時機可能讓同一個 activity 被重試），不可逆的補償（已寄出的信收不回來）仍要在設計時就避免。durable execution 讓「協調」變可靠，但它從來沒承諾、也無法承諾讓「外部世界的副作用」變可逆。

## 為何是這個形狀

退一步看，durable execution 的整個樣貌，都是從開場那個矛盾長出來的：**一個跨越行程生命週期的長流程，和「執行狀態只活在記憶體」這件事，無法共存。**

正因如此，它要把執行過程本身——而不只是結果快照——持久化成一份事件歷史，這樣才有東西可以拿來恢復。正因為恢復靠的是把函式重放、逐步對齊歷史，它才被迫要求工作流碼決定性，把所有不決定的東西（時間、隨機、I/O）都趕進 activity、把結果釘進歷史。正因為歷史會無限長而重放不能無限慢，它才需要 Continue-As-New 來週期性卸下包袱。正因為它擅長的是「有限步驟、夾著長等待」，那個三十分鐘的等待才能變成一個不佔執行緒的持久化計時器，而不是一個空轉的 sleep。

兩種主流長相——Temporal 的「工作流即程式碼」與 Step Functions 的「工作流即託管狀態機」——是同一個機制在不同取捨下的兩張臉：一個給你語言的表達力、要你扛運維與紀律；一個給你託管的省心、要你接受狀態機的表達框與型別選錯就換掉語意的陷阱。而它和 saga 疊在一起時，最終拼出的圖是這樣的：saga 說「跨服務怎麼保持一致」，durable execution 說「讓這套協調，崩了也跑得完」。

下次你看到一個長流程的工作流碼裡，某一行寫著 `await sleep('3 days')`，而它背後的 worker 在這三天裡重啟了十幾次、換了五個版本，醒來時流程卻分毫不差地接著往下走——你會知道那不是魔法，那是有人把「程式走到哪」這件事，老老實實地當成資料存了下來。
