# 事件驅動架構：event notification 與 event-carried state transfer

一筆訂單成立的瞬間，有多少東西想知道這件事？倉庫要備貨、發票系統要開立、推薦引擎要更新使用者畫像、通知服務要寄信、風控要重算額度、資料倉儲要記一筆。半年後產品又加了忠誠點數、加了詐騙偵測、加了履約時效報表——每一個都想在訂單成立的那一刻被叫醒。

如果訂單服務用最直接的方式滿足這些需求，它的下單流程裡會長出一串呼叫：`callWarehouse()`、`callBilling()`、`callRecommendation()`、`callNotification()`……每加一個關心者，就在這個本該只管「把訂單寫進資料庫」的函式裡塞一行。訂單服務漸漸變成一個它根本不該是的東西——一個知道全公司每個下游、知道它們的位址、知道它們的 API、並且要為它們任何一個掛掉負責的中樞。哪天推薦服務回應慢了 500 毫秒，使用者下單就跟著慢 500 毫秒；哪天通知服務掛了，下單跟著失敗。下單這個核心動作的可用性，被一群「只是想旁聽」的下游給綁架了。

事件驅動架構（Event-Driven Architecture, EDA）要翻轉的，正是這個「產生者必須認識每一個關心者」的關係。它的核心動作只有一個句法上的轉換：訂單服務不再說「我叫你去做某事」（command），而是說「某件事發生了」（event）——`OrderPlaced`，一個過去式的事實。它把這個事實廣播出去，誰要聽自便。產生者不知道、也不需要知道，此刻有幾個下游在聽、它們各自要做什麼。

這個轉換聽起來只是換了個動詞，但它把耦合的方向整個倒過來了，也因此欠下一整包新的債。這一章要把這件事講透——尤其是「事件裡到底該裝多少資料」這個看似瑣碎、實則決定整個架構性格的選擇。

## 為什麼「過去式」這麼關鍵

先把 command 和 event 的差別釘死，因為這是 EDA 一切好處的源頭，也是最常被偷渡掉的地方。

`PlaceOrder`（下單）是 command：祈使句、未來式、有明確的對象——我送這個訊息，是因為我**知道**有個訂單服務該處理它、而且我**期待**它去處理。`OrderPlaced`（訂單已成立）是 event：陳述句、過去式、沒有對象——我只是宣告一個已經發生的事實，至於誰要對這個事實做出反應、做什麼反應，與我無關。

差別不在語法美感，在於**誰握著「接下來該發生什麼」的知識**。command 的世界裡，這份知識握在發送端：它知道下一步是誰、要做什麼。event 的世界裡，這份知識被移到了接收端：訂單服務只負責「誠實地說發生了什麼」，「該怎麼反應」完全是聽眾自己的事。正因為知識搬了家，產生者才能對下游一無所知——它不可能去呼叫一個它根本不認識的服務。

這就是為什麼命名要逼自己用過去式。一旦你發現自己想發一個叫 `SendConfirmationEmail` 的「事件」，那其實是 command 偽裝的：你心裡清楚有個通知服務該寄信、你期待它寄。這種偽事件會把耦合悄悄偷渡回來——產生者又開始認識下游、又開始期待特定反應，只是披了一層 pub/sub 的皮。真正的事件是 `OrderPlaced`，它不關心有沒有人寄信；寄不寄信是通知服務聽到這個事實後自己的決定。用過去式命名，等於強迫自己每次都回答一個問題：我是在陳述事實，還是在偷偷下命令？

值得分清楚的是：**讓事件不關心對象的，不是 broker，是命名與語意。** 你完全可以把一個 command 丟上 pub/sub channel，享受到時間解耦的好處（接收端不在線，訊息留著等它）——但只要那個訊息本質上是「我期待某個特定下游做某件特定事」，topology 上的耦合就還在，產生者心裡仍綁著那個下游。Broker 給你的是基礎建設，過去式的事實語意給你的才是真正的解耦。這兩件事常被混為一談。

## 事件裡裝多少資料：一個架構分岔

產生者廣播了 `OrderPlaced`。一個下游——比如發票服務——收到了。它要開發票，需要知道訂單金額、品項、買家稅務資訊。問題來了：**這些資料，從哪來？**

這個問題的答案有兩條路，而你選哪一條，會決定整個系統的負載長相、可用性、一致性模型，乃至於哪天會在什麼地方壞掉。Martin Fowler 在〈What do you mean by 'Event-Driven'?〉裡把常被混為一談的「事件驅動」拆成四種模式，其中兩種正是這個問題的兩種答案：**event notification（事件通知）** 和 **event-carried state transfer（事件攜帶狀態傳遞）**。另外兩種——event sourcing 和 CQRS——是更大的命題（見〈Event sourcing 與 CQRS〉），這一章專心把前兩種講到見底。

差別只有一句話：**事件裡裝多少資料。** 但這一句話展開來，是兩種性格南轅北轍的架構。

### 路一：薄事件，回頭查（event notification）

事件只帶最精瘦的東西：`{ "type": "OrderPlaced", "orderId": "ord_8a3f", "at": "2026-06-22T10:15:00Z" }`。一個 id，一個時間戳，幾十個 bytes。發票服務收到後，自己**回頭呼叫訂單服務**：`GET /orders/ord_8a3f`，把開發票需要的明細撈回來。

Fowler 對這種模式的描述很精準：事件「往往只帶一些 id 資訊和一個回指發送端的連結」（"often just some id information and a link back to the sender"），而且「來源系統其實不太在乎回應，往往根本不期待任何回答」（"The source system doesn't really care much about the response"）。

這條路的吸引力在於**單純與單一真相**。事件小，網路便宜；訂單的權威資料只有一份，就在訂單服務的資料庫裡，沒有任何副本要維護、要對帳。每個下游需要什麼，自己回頭按需求查，要什麼撈什麼。

### 路二：胖事件，自己存（event-carried state transfer）

事件帶上下游會用到的**完整狀態**：

```json
{
  "type": "OrderPlaced",
  "orderId": "ord_8a3f",
  "buyer": { "id": "u_271", "taxId": "...", "address": "..." },
  "items": [ { "sku": "...", "qty": 2, "price": 1200 } ],
  "total": 2400,
  "at": "2026-06-22T10:15:00Z"
}
```

發票服務收到後，**不必回頭查任何東西**——它要的明細就在事件裡。更進一步，它通常會把自己關心的那部分**存成一份本地副本**，這樣連未來查詢、產報表都不必再去碰訂單服務。

Fowler 列了這條路的四個好處，每一個都直指前面那個「下單被旁聽者綁架」的痛：**更強的解耦**、**降低來源負載**（"We don't have to worry about load on the customer system to satisfy queries from all the consumer systems"）、**更低的延遲**（"there's no remote call required"）、以及最關鍵的**更高的可用性**——「接收系統在來源系統不可用時仍能運作」（"the recipient systems can function if the customer system becomes unavailable"）。訂單服務掛了，發票服務照樣能拿手上事件裡的資料開發票，因為它根本不需要訂單服務在線。

而代價，Fowler 也說得很白：「明顯的壞處是有一大堆資料被搬來搬去、一大堆副本」（"lots of data schlepped around and lots of copies"）。資料被複製了，於是你買下了**最終一致性**——下游那份副本，永遠比來源晚那麼一點點。

## 把這個選擇算給你看

兩條路的差別不是哲學，是可以量化的負載。把它擺到一個具體場景上手算一遍，你就再也不會憑感覺選。

訂單服務每秒產生 **1,000 筆** `OrderPlaced`，有三個下游在聽：出貨、發票、推薦。

**走 event notification（薄事件）**，每筆事件約 100 bytes，事件本身的頻寬可以忽略不計——每秒 1,000 × 100 bytes = 約 100 KB/s，微不足道。但三個下游每收到一筆，各自要回頭查一次明細。於是訂單服務每秒要承受的反查流量是：

```
3 下游 × 1,000 事件/s = 3,000 次/s 的明細查詢，全打在訂單服務上
```

事件雖小，**反查流量是事件量的三倍**，而且這 3,000 次/s 是同步的、即時的——訂單服務成了三個下游的共同同步依賴。它一慢，三個下游跟著卡；它一掛，三個下游全部開不了發票、出不了貨。表面上用事件把流程解耦了，實際上在「回頭查」這條反向路徑上，同步耦合又被補了回來，只是換了個方向——事件的扇出是非同步的，緊接著的反查卻是同步的，兩者一加一減，淨耦合幾乎沒減多少。

**走 event-carried state transfer（胖事件）**，每筆事件帶完整訂單約 2 KB。事件頻寬從 100 KB/s 升到：

```
1,000 事件/s × 2 KB = 2 MB/s
```

頻寬漲了 20 倍。但回頭查的流量是 **零**。訂單服務發完事件就解脫了，它不再被任何下游拖累；它掛掉的那段時間，三個下游照樣能消化手上佇列裡的事件、照樣運作。

這兩個數字——**「3,000 次/s 反查」對上「2 MB/s 頻寬」**——就是這個選擇的全部。判準因此很清楚：**下游越多、越頻繁需要資料，反查的成本（同步流量＋耦合＋共同故障點）就漲得越凶，胖事件越划算；反過來，事件本身就很大、或下游其實很少真的需要細節、或資料敏感不宜廣播，才偏向薄事件。** 注意這裡有個容易被忽略的方向性：反查成本隨**下游數量**線性放大（每多一個下游，反查流量就多一份），而胖事件的頻寬成本主要隨**事件大小**走，與下游數量幾乎無關（broker 扇出，產生者只發一次）。下游一多，天平幾乎必然倒向胖事件。

## 薄事件那個沒人提的時序坑

event notification 還藏著一個比「反查流量」更陰險的問題，它不是效能問題，是**正確性**問題，而且只在特定時序下現形，平時跑得好好的。

回頭看那個動作：下游收到「`orderId=ord_8a3f` 變了」，然後回頭查 `GET /orders/ord_8a3f`。它默默假設了一件事——**查回來的，就是觸發這個事件當下的那個狀態**。但這個假設是錯的。

設想這串時序。10:15:00，訂單從 `pending` 變成 `paid`，發出事件 A（"order ord_8a3f changed"）。10:15:02，事件 A 還在 broker 裡排隊、還沒送到下游，這時訂單又從 `paid` 變成 `shipped`，發出事件 B。10:15:03，下游終於收到事件 A，回頭查 `GET /orders/ord_8a3f`——查回來的是什麼？是 `shipped`。

下游本來在處理「付款」這個事件，結果手上拿到的卻是「已出貨」的狀態。它對 A 做出的反應，用的是 B 之後的資料。如果它的邏輯是「訂單付款後寄收據」，它可能在一個已經出貨的訂單上跑付款邏輯，狀態錯亂。事件說的是「過去某一刻發生了什麼」，但回頭查拿到的永遠是「此刻的最新值」——這兩者之間，只要來源又被改過，就對不上了。事件越頻繁、reorder 或延遲越常見，這個坑踩得越勤。

胖事件天生沒有這個問題：事件 A 裡裝的就是 10:15:00 那一刻的完整狀態（`paid`），下游處理 A 時看到的就是 A 當下的真相，不會被後來的 B 污染。**「狀態就在事件裡」不只是省一次網路往返，它把事件和它描述的那個時刻焊死在一起，杜絕了「事實在途中、世界已改變」這類時序錯位。** 這是薄事件換單純所付出的、最容易被低估的代價。

## 胖事件的兩個邊角：順序與大小

但別把胖事件當銀彈，它有自己的兩個尖角，都得正面講清楚，否則你會在生產環境被它扎到。

**第一個尖角是順序。** 既然下游靠一連串事件來更新自己那份副本，那這些事件的**到達順序**就攸關副本的正確性。`AddressChanged(A→B)` 後面跟著 `AddressChanged(B→C)`，如果下游先收到後者、再收到前者，它的副本會停在 B，永遠錯。胖事件本質上是在用事件流複製狀態，而狀態複製對順序敏感。所以 event-carried state transfer 幾乎總要搭配「同一個實體的事件保持有序」——典型做法是用實體 id 當 partition key，把同一筆訂單的所有事件釘進同一個 partition，靠 partition 內的順序保證來兜（順序怎麼保證、partition key 怎麼選，見〈訊息順序：FIFO 與 partition key〉）。下游還得對重複投遞免疫，因為 at-least-once 投遞下同一個事件可能來兩次（冪等的定義與做法見〈重複是常態：冪等與去重〉）。這裡又能看到「全量快照」的好處：胖事件帶的若是某 key 的完整最新狀態，重複投遞同一筆只是把副本覆寫成同一個值，天然冪等；唯有帶「增量變更」（如「餘額 +100」）時，重複才會把變更重複套用、才真正需要額外去重。

有一個讓「順序」這件事變簡單的優雅手法值得一提：如果你的事件帶的是**某個 key 的完整最新狀態**（而非「增量變更」），那麼把它放進一個 **log compaction（日誌壓縮）** 的 topic——Kafka 支援這個——broker 會在背景逐步壓掉同一 key 的舊值、至少保留每個 key 的最新一筆（compaction 是非同步的背景過程，保證「至少留最新值」，不保證任一時刻只剩一筆）。這樣一個新上線的下游，只要從頭把這個 compacted topic 重播一遍，就能重建出每個實體的當前狀態，不必寫任何 backfill 腳本、不必去敲來源系統的門。胖事件帶的是「全量快照」而非「差分」這個選擇，在這裡直接換來了「狀態可被重建」的能力——這也是為什麼 event-carried state transfer 偏好帶完整狀態，而不是只帶 changed 欄位。

**第二個尖角是大小，而它會撞上你用的 broker 的硬上限。** 胖事件天生比薄事件大，而訊息中介軟體對單則訊息有體積天花板。一個具體的、容易在 2026 年踩到的數字（2026-06）：Amazon SNS 的單則訊息上限仍是 **256 KB**；它扇出去的 Amazon SQS 在 2025-08 才把上限從 256 KiB 提升到 **1 MiB**。也就是說，如果你用 SNS→SQS 這套標準扇出來搬胖事件，**瓶頸卡在 SNS 那 256 KB**，不是 SQS 的 1 MiB。一個塞了完整訂單、完整買家資料、完整品項清單的胖事件，在大訂單上很可能就頂破 256 KB。超過了怎麼辦？標準解法是 **claim-check 模式**：把肥大的 payload 丟進物件儲存（S3），事件裡只放一個指標（一個 S3 key），下游拿指標去 S3 取——這其實是讓胖事件在「太胖」時退化成一種受控的薄事件（見〈物件儲存：S3 與大 payload 的交接〉）。換句話說，胖事件能胖到哪裡，是有物理上限的，撞到上限時你被迫退回一種「半薄」的折衷。

順帶一提，胖事件還有個非技術但同樣硬的邊界：**把完整狀態廣播出去，意味著敏感欄位（稅號、地址、個資）被複製到每一個訂閱者那裡。** 薄事件天然把資料留在來源、只有真正需要的下游才查得到；胖事件則是把資料攤給所有聽眾。在有資料治理或法遵要求的場景，這個「廣播即擴散」的性質，有時候直接否決了胖事件這個選項——不是因為它慢或貴，是因為它把資料送到了不該存放的地方。

## 解耦從來不是免費的，它只是換了一種耦合

走到這裡，EDA 的好處清單很誘人：下單動作不再被旁聽者綁架、加新下游不必動產生者、來源掛了下游照活。但「事件驅動 = 鬆耦合」這個等號，下得太草率了。

Gregor Hohpe（《Enterprise Integration Patterns》的作者）有一個很冷靜的觀察，值得每個要上 EDA 的人記住：**讓你解耦的，是 channel 的類型（pub/sub、非同步），不是「事件」這個語意本身**；而且——這是關鍵——**「在某一個維度上享受解耦，會讓你在另一個維度上更耦合」**（"taking advantage of one dimension of decoupling exposes you more to another dimension of coupling"）。

具體說，EDA 漂亮地剪掉了三條繩子：時間耦合（接收端不必當下在線）、空間耦合（產生者不必知道下游在哪）、以及 topology 耦合（加一個訂閱者，發送端零改動）。但它沒剪掉、甚至加重了另外兩條：

一是 **schema 耦合**。事件一旦發布，它的結構就是一份**公開的 API**——所有訂閱者都依賴它。事件看起來像是產生者內部的東西、欄位名似乎能隨手改，但每一個下游都按那個欄位名解析。破壞性地改事件結構，等於悄悄弄裂所有可能根本不知道存在的下游。所以事件 schema 必須當公開契約來治理，向前向後相容地演進（見〈序列化與 schema 演進〉）。這裡有個諷刺的反向關係：解耦得越徹底（產生者越不認識下游），schema 改錯的後果越難預料——連會弄裂誰都不知道。

二是 **observability（可觀測性）的耦合**，或者說，可觀測性的**喪失**。Hohpe 直接點名：你「可能根本不知道系統結構正在如何演變」。同步呼叫的世界裡，「A 呼叫 B 呼叫 C」這條流程白紙黑字寫在程式碼裡，你讀 code 就看得到。EDA 把這條流程拆散、藏進了事件流：訂單服務發了個 `OrderPlaced`，然後呢？誰被叫醒、它又發了什麼事件、又叫醒了誰——**沒有任何一個地方能一眼看出端到端的流程**。Fowler 對此的警告一針見血：當「真的有一條跨越多個事件通知的邏輯流程」時，問題在於「你很難看出這條流程，因為它不顯現在任何程式文字裡」（"it can be hard to see such a flow as it's not explicit in any program text"）。

這就是 EDA 最深的那筆債：**它用「可見的耦合」換來了「不可見的耦合」。** 直接呼叫是看得見的依賴——醜，但你知道它在那。事件流是看不見的依賴——優雅，但一筆訂單到底會觸發什麼連鎖反應，可能沒有任何一個人完整知道。一個成熟的事件驅動系統，必須靠分散式追蹤（把 trace context 隨事件一路傳下去，見〈分散式追蹤：trace context 傳播、span 與取樣〉）和事件文件，把這份被解耦解掉的可見性，**人為地補回來**。否則你蓋的不是一個鬆耦合的系統，是一座沒人有地圖的迷宮。

## 那到底什麼時候該用、什麼時候別碰

把上面所有的取捨收束成一個判斷。EDA 不是越多越好的東西，它是一個「用一組複雜度換另一組」的交易，得算清楚划不划算。

**它真正發光的場景**有一個共同特徵：**關心某件事的人，是會增長且未知的。** 下單要驅動出貨、發票、推薦、通知、點數、風控——而且這個清單半年後還會變長。如果你用直接呼叫，每加一個下游就要改一次訂單服務、重新部署、重新測試；用事件，產生者一次都不用動，新下游自己訂閱就好。當「未來會冒出多少關心者」這件事本身不確定時，事件驅動讓你把產生者和這份不確定性隔開——這是它最值錢的地方，也是「下單不被旁聽者綁架」這句話的全部含義。

**它是過度設計的場景**也有共同特徵：**流程是線性的、關心者固定就一兩個、且需要當下的強一致確認。** 「A 做完就叫 B 做，永遠只有 B」這種流程，套上事件匯流排，你得到的是一條看不見的隱性流程、加上最終一致、加上重複投遞要處理——換來的解耦你根本用不到，因為你壓根沒有「未知下游」要應付。尤其當這一步需要**當下就確認成功**（扣庫存必須現在就成、才能確認訂單），把它硬拆成「發個事件、最終會扣」，你買來的是「訂單已成立但庫存還沒扣」的不一致窗口和對帳負擔——這時候，那條同步的時間耦合不是缺陷，是**特性**：你要的就是「現在、立刻、確定地」知道結果（同步與非同步的權衡，見〈同步與非同步通訊〉）。

判準濃縮成一個問題：**關心這件事的人，是固定且少數，還是會增長且未知？** 固定少數，直接呼叫往往更清楚、更好除錯；增長未知，事件解耦讓你加訂閱者而不必驚動產生者。

而一旦你選了 event 這條路，下一個立刻要回答的問題就是這一章的主軸：**事件裡裝多少資料？** 薄事件（event notification）省頻寬、守單一真相，但欠下反查流量、共同故障點、和那個惡名昭彰的時序坑；胖事件（event-carried state transfer）買到真正的解耦、來源高可用、和時間正確性，但付出資料複製、最終一致、順序敏感、體積上限。沒有哪個對、哪個錯——只有「你這個系統的下游有多少、多頻繁要資料、資料多敏感、能容忍多少不一致」算出來的那個答案。

## 為什麼是這個形狀

退一步看，事件驅動架構的整個樣貌，都從一個簡單的拒絕長出來：**產生者拒絕認識它的下游。**

正因為它拒絕認識下游，它只能廣播「事實」而非下達「命令」——所以事件必須是過去式，否則就是偷渡命令。正因為下游不能假設來源永遠在線，才有了「把狀態裝進事件、自己存一份」的胖事件路線，換來來源掛了也能活。正因為產生者不知道誰在聽，schema 一改就可能弄裂一群看不見的下游，所以事件 schema 得當公開契約治理。也正因為流程被拆散藏進了事件流，沒人能一眼看完整條鏈，所以你得用追蹤把可見性人工補回來。

每一個特性，都是「產生者不認識下游」這個核心決定的直接後果——包括它的好處，也包括它的債。EDA 不是一套任意的工程時尚；它是當你真心想讓系統的各部分「少知道彼此」時，被逼出來的那個形狀，連同它所有的甜頭和苦頭，一起打包奉送。

下次你看到一個 `OrderPlaced` 事件在系統裡靜靜地扇出，沒有任何一行程式碼指明它會驚動誰，你會知道那不是混亂——那是產生者在兌現一個承諾：我只負責誠實地說發生了什麼，至於這個世界要怎麼回應，不歸我管。
