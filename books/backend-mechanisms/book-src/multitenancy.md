# 多租戶隔離：row、schema、db-per-tenant

某個禮拜五，一位客戶的工程師寫信來，附了一張截圖：他在自己帳號的報表頁，看到一筆不屬於他的訂單——金額、收件人、商品全是別家公司的。他沒做錯任何事，就是點了個再普通不過的「匯出本月訂單」。問題出在你這邊：那份報表用的是一條手寫的 SQL，為了 join 一張統計表，那條 query 漏掉了 `WHERE tenant_id = ?`。在一張裝了所有客戶資料的共用表上，少一個過濾條件，就是把整個資料庫攤在一個外人面前。

這就是多租戶隔離要解的問題，而且它的殘酷之處在於：**洩漏的代價是離散的、災難性的，而不是漸進的。** 效能差一點，使用者抱怨；但跨租戶資料洩漏，是要發合規通報、要登上新聞、可能讓你失去整個客戶群的那種事故。一套服務多個彼此獨立、互不信任的客戶（tenant）、共用同一份程式碼與基礎設施——這是 SaaS 的經濟基礎，省下的就是「不必為每個客戶各養一套系統」的成本。但省下來的每一分，都是用「把不同客戶的資料放得更近」換來的，而越近，就越容易在某個你沒注意的縫隙裡漏出去。

## 一條軸線：隔離強度換資源效率

先把問題的形狀看清楚。多租戶的所有設計選擇，本質上都落在同一條軸線上：**一端是「每個租戶完全獨立、互不共用任何東西」，另一端是「所有租戶擠在同一份資源裡」。** 越靠獨立那端，隔離越硬、爆炸半徑越小、越好搬遷與合規，但你要為每個租戶複製一份基礎設施，成本與營運負擔隨租戶數線性膨脹；越靠共用那端，資源利用率越高、開新租戶越快，但任何一個租戶的失誤或暴衝，潛在地波及所有人。

這條軸線上有三個被反覆發現、業界給了固定名字的點。AWS 的 SaaS 隔離白皮書把它們叫做 **silo（穀倉）／bridge（橋接）／pool（共池）**；資料庫的語境裡，它們對應到三種把租戶資料擺進儲存的方式：

- **row-level（共池）**：所有租戶的資料同在一張表裡，靠每一列上的一個 `tenant_id` 欄區分。一張 `orders` 表裝下所有人的訂單。
- **schema-per-tenant（橋接）**：同一個資料庫實例裡，每個租戶一套自己的 schema（一組同名的表）。租戶 A 有 `tenant_a.orders`，租戶 B 有 `tenant_b.orders`。
- **db-per-tenant（穀倉）**：每個租戶一個完全獨立的資料庫，甚至獨立的實例。

這三者不是「好壞」之分，而是同一條取捨軸上的三個刻度。理解它們的關鍵，不是背下「哪個隔離強」，而是看懂**每一種的隔離邊界畫在哪、由誰來守、在什麼壓力下會破**。

## row-level：最省，也最容易出人命

共池模型是最誘人的：一套表結構、一套 migration、一個連線池，開一個新租戶只是 `INSERT` 一列。一萬個小租戶共用一張表，硬碟、記憶體、連線全部攤平共享，資源利用率拉到最高。代價全部集中在一個地方——**租戶之間沒有任何物理邊界，唯一的邊界是每一條 query 裡那個 `WHERE tenant_id = ?`。**

把這句話的份量掂清楚：在共池模型裡，租戶隔離不是資料庫的性質，而是**你寫的每一行 query 的性質**。資料庫不知道什麼叫租戶，它只看到一張大表。隔離成立的條件，是「所有碰這張表的程式碼，每一條，都正確地帶上了租戶過濾」。這是一個全稱命題——只要有一條漏了，隔離就破了，而且破得無聲無息：那條漏掉過濾的 query 不會報錯、不會變慢，它只是安靜地回傳了別人的資料。開場那封禮拜五的信，就是這個命題被一條手寫報表 SQL 證偽的瞬間。

ORM 讓這件事更危險，不是更安全。ORM 替你自動帶上租戶條件的前提，是你乖乖走它的 query builder；但只要有人為了效能寫了一段 raw SQL、用了一個 ORM 沒攔到的 lazy load、或在一個 join 的子查詢裡忘了延續租戶條件，那條路徑就繞過了所有自動防護。隔離的強度，等於那條最弱的、最少人看的程式碼路徑的強度。

### 把守門人從人腦搬進引擎：RLS

既然「靠每個工程師記得帶 `WHERE`」是一個遲早會失守的防線，自然的下一步是：**讓資料庫自己強制過濾，無論應用怎麼寫 query。** PostgreSQL 的 Row-Level Security（RLS，列級安全）正是做這件事。你在表上開啟 RLS、綁一條 policy：

```sql
ALTER TABLE orders ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON orders
  USING (tenant_id = current_setting('app.tenant_id')::bigint);
```

開啟之後，任何對 `orders` 的 `SELECT`、`UPDATE`、`DELETE`，引擎都會自動在你的 query 上**再 AND 一個** policy 條件。應用送來一條 `SELECT * FROM orders`，引擎實際執行的是 `SELECT * FROM orders WHERE tenant_id = current_setting('app.tenant_id')::bigint`。就算那條手寫報表 SQL 忘了帶租戶過濾，引擎也會替它補上——隔離的判定權，從散落各處的應用程式碼，收回到資料庫這一個地方。

聽起來像銀彈，但 RLS 有兩個會讓人栽跟頭的邊角，兩個都不顯然，都得講透。

**第一個：表的擁有者預設繞過 RLS。** 這是最容易讓人「設了 RLS 卻等於沒設」的陷阱。`ENABLE ROW LEVEL SECURITY` 只對「一般角色」生效；**表的 owner，以及任何有 superuser 或 `BYPASSRLS` 屬性的角色，預設直接無視 policy**。而很多應用——尤其是用 migration 工具建表的——線上連的就是建表那個 owner 角色。於是你興沖沖地開了 RLS，policy 卻對你的應用連線完全不起作用，跨租戶照樣看光光。要堵住這個洞，必須額外下一道：

```sql
ALTER TABLE orders FORCE ROW LEVEL SECURITY;
```

`FORCE` 才讓 policy 連 owner 都管。但即使如此，superuser 與 `BYPASSRLS` 角色仍然繞得過去——所以線上應用**絕不能用 superuser 連線**。「擁有者後門」是 RLS 隔離邊界上一道沉默的裂縫：引擎看似在守，但用 owner 角色連線的應用，恰好站在被豁免的那一側。

**第二個，也是更陰險的：policy 靠 session 變數認租戶，而 session 變數會在連線池裡跨租戶洩漏。** 上面那條 policy 讀的是 `current_setting('app.tenant_id')`——一個 session 層級的設定值，得在每次處理請求前由應用設好：「這次請求是租戶 42 的」。問題出在 `SET` 和 `SET LOCAL` 的差別，而這個差別在連線池下會放大成一個資料洩漏漏洞。

看清楚這條時序。假設前面架了 PgBouncer 走 **transaction pooling**（交易層池化）——這是高並發 web 應用幾乎必選的模式，因為它能讓少數幾條實際 DB 連線服務成千上萬個客戶端（後面會算這筆帳）。在這個模式下，**一條實體連線只在一個交易期間屬於某個客戶端，交易一結束就被丟回池裡給下一個人用**：

```
連線 #7 的一生（transaction pooling）：

  T1: client A 借走 → SET app.tenant_id = 'A' → SELECT ... → COMMIT → 還回池
  T2: client B 借走 → (沒重設) → SELECT ...        → COMMIT → 還回池
                         ▲
                         └─ #7 上還殘留著 'A'！client B 讀到了租戶 A 的資料
```

如果應用用的是 `SET app.tenant_id = 'A'`（session 級），這個設定會**活過整個 session 的生命週期**。但在 transaction pooling 下，「一個 session」橫跨許多個不同客戶端——連線 #7 處理完 A 的交易、被還回池、下一個被分到 #7 的是 B，而 #7 上還黏著 `app.tenant_id = 'A'`。B 的 query 套上 policy，過濾出來的是**租戶 A** 的列。RLS 不但沒擋住洩漏，還親手把別人的資料端給了 B。

正解是用 `SET LOCAL`：

```sql
BEGIN;
  SET LOCAL app.tenant_id = '42';   -- 只在這個交易內有效
  SELECT * FROM orders;             -- policy 用的是 '42'
COMMIT;                              -- 交易結束，app.tenant_id 自動失效
```

`SET LOCAL` 把租戶上下文綁死在**單一交易**的邊界內，交易一結束它就消失，連線還回池時是乾淨的，下一個客戶端絕不會繼承到上一個的租戶身分。這個「`SET` 還是 `SET LOCAL`」的選擇，表面上是個無關痛癢的關鍵字差別，實際上是「RLS 多租戶在連線池下成不成立」的分水嶺——而它之所以難抓，正因為在沒有池化、或低並發測試環境裡，這個 bug **完全不會出現**：每個 session 一條專屬連線時，殘留的 session 變數從來不會被別人撿到。它只在「高並發 ＋ transaction pooling ＋ 用了 `SET` 而非 `SET LOCAL`」三者齊備時，才會在生產環境隨機地、間歇地把租戶 A 的資料漏給租戶 B。這是那種會讓你在凌晨三點對著日誌發抖的 bug：它不可重現、不報錯，只是偶爾把隔離邊界悄悄抹掉。

還有一個藏在 `current_setting` 用法裡的、同樣攸關隔離的選擇：上面 policy 用的是**單參數**形式 `current_setting('app.tenant_id')`，而它在變數從未被設定時（某條路徑漏了 `SET LOCAL`、或 `BEGIN` 後還沒設）會直接**拋錯**（`unrecognized configuration parameter`）——這其實是好事，它讓「忘了設租戶」**fail-closed**：整條 query 立刻報錯、馬上被發現。陷阱在於改用雙參數形式 `current_setting('app.tenant_id', true)`：它在變數未設時回傳 `NULL`，於是 policy 條件變成 `tenant_id = NULL`（永遠為假）、靜默回傳一個**空集**。空集不會洩漏別人的資料，但它把「漏設租戶」從一個會炸、會被立刻抓到的錯誤，變成一個不報錯、悄悄少了資料、難 debug 的無聲故障。RLS 的安全性不只取決於變數一定被設，還取決於**設不到時要炸、而不是靜默**——所以這裡偏偏該選那個「會拋錯」的單參數形式。

### 共池還有一個你刪不掉的問題：吵鬧的鄰居

就算隔離守住了，共池還有一個結構性的軟肋——**noisy neighbor（吵鬧的鄰居）**。所有租戶共用同一批表、同一個 buffer pool、同一組 CPU。一個大客戶半夜跑了個全表掃描，或一條沒命中索引的報表查詢，把 buffer pool 裡的熱頁全洗掉、把 I/O 打滿——所有其他租戶的查詢跟著一起慢。在共池模型裡，你**沒有任何資料庫層的工具**去限制單一租戶的資源用量；隔離的是「能看到誰的資料」，從來沒隔離「能吃掉多少資源」。

這也是為什麼共池天然適合「**多而小、且體量相近**」的租戶——一群誰也撐不爆系統的小客戶。一旦租戶之間體量差距拉大（5000 個小租戶配 1 個吃掉半個資料庫的巨獸），共池就開始反噬：巨獸拖慢所有人，而你又因為大家擠在同一張表裡，連「把巨獸單獨挪走」都做不到。

## schema-per-tenant：邏輯隔離，與一道你撞得到的牆

往隔離強的方向走一格，就是每個租戶一套 schema。`tenant_a.orders` 和 `tenant_b.orders` 是兩張**不同的表**，物理上分開存放；連線時用 `SET search_path TO tenant_a`（PostgreSQL）或 `USE tenant_a`（MySQL）切到對的租戶，之後的 query 寫 `SELECT * FROM orders` 就只會碰到那個 schema 的表。「漏一個 `WHERE`」的風險消失了——因為根本不存在一張裝著所有人的大表；過濾從「每條 query 的責任」變成「連線時選對 schema 的責任」，而後者只需要設定一次、且設錯時通常是看不到自己的資料（會立刻被發現），而不是悄悄看到別人的（永遠不會被發現）。它還順帶買到了「按租戶客製」「按租戶單獨備份」的能力——某個大客戶想多一個欄位、想單獨匯出，schema 邊界讓這些變得自然。

但 schema-per-tenant 有一道幾乎所有人第一次都會低估的牆：**它撐不住「無上限增長的租戶數」。** 原因藏在資料庫的系統目錄（catalog）裡。

PostgreSQL 裡每一張表、每一個索引、每一個 sequence、每一個 constraint，都在共享的系統目錄（`pg_class`、`pg_attribute` 等）裡佔一列——**無論它屬於哪個 schema**。schema 是個邏輯命名空間，它並沒有給每個租戶一份獨立的目錄。於是算一筆帳：假設每個租戶 50 張表、每張表平均 5 個索引，那一個租戶就往 `pg_class` 塞進約 50 + 250 = 300 列。1000 個租戶就是 30 萬列，1 萬個租戶就是 300 萬列——全部擠在同一份共享目錄裡。

這份目錄不是裝飾品，它是查詢規劃器（planner）每次規劃 query 都要查的工作集。當 `pg_class` 膨脹到百萬列等級，連鎖反應就來了：planner 變慢、catalog 的快取命中率下降、`pg_dump` 因為要遍歷整個目錄而拖成幾小時、每開一條新連線（要載入 catalog 資訊）都變慢。這些都不是「資料量大」造成的——就算每個 schema 只有幾列資料，光是**表的數量**就足以壓垮目錄。schema-per-tenant 的甜蜜區是「幾十到幾百、最多一兩千個租戶」；把它套在「想長成幾萬個租戶的 SaaS」上，你不是會慢慢變慢，而是會在某個租戶數附近撞牆，而且這堵牆和你存了多少業務資料無關。

另一個常被忘記的成本：**migration 要對每個 schema 各跑一次。** 加一個欄位，在共池裡是一條 `ALTER TABLE`；在 schema-per-tenant 裡是 N 條，N 等於租戶數。1000 個租戶就是 1000 次 DDL，每一次都可能鎖表、都可能有一個失敗需要回頭補。零停機的結構變更（這本身是門學問，見〈零停機 schema 遷移〉）在乘上 N 倍 schema 之後，難度也跟著乘上去。

## db-per-tenant：最硬的牆，最貴的牆

軸線的另一端：每個租戶一個完全獨立的資料庫，甚至獨立的實例。這是隔離的極致——租戶之間連系統目錄都不共享，noisy neighbor 從根本上消失（各自的資源是各自的），爆炸半徑縮到單一租戶（一個庫掛了只影響一個客戶），按租戶備份、按租戶調效能、把某個客戶的資料整個搬到指定地理區域以滿足資料主權要求，全都變成「對那一個庫操作」這麼乾淨。對「少而大、強監管」的客戶——金融、醫療、有資料落地（data residency）法規要求的——這種隔離不是奢侈，是入場券。

代價同樣極致，而且有一個特別容易被低估的具體形式：**連線數。** 每一個資料庫都需要自己的一套連線池。算一筆：1000 個租戶 × 每庫只開 10 條連線 = 1 萬條連線。而 PostgreSQL 的 `max_connections` 預設值是 **100**（2026-06），就算調高，每一條連線在 PostgreSQL 裡都是一個獨立的後端行程、吃掉好幾 MB 記憶體，幾千條連線就能把一台資料庫伺服器的記憶體壓垮。你不可能讓一台實例扛 1000 個庫各自的連線池。

這就把 db-per-tenant 的真實成本暴露出來：它幾乎一定要搭多個資料庫實例（把 1000 個庫分散到多台機器），外加一個路由層——請求進來，先查「這個租戶住在哪個實例的哪個庫」，再把連線導過去。它和共池的成本差距，往往是一兩個數量級。「再開一個庫」聽起來便宜，實際上你買的是「再開一整套要監控、要備份、要打補丁、要管連線的基礎設施」。隔離最強，營運負擔也最重，沒有白吃的午餐。

## 一個真實的取捨：把三種模型混著用

現在把三種模型放回那條軸線上，用一個具體的場景把取捨手算出來——因為現實裡幾乎沒有人純用單一模型，**成熟的做法是混合（bridge）**，而混合的判準正是從這筆帳裡長出來的。

設想一個 SaaS：**5000 個小租戶 ＋ 20 個大企業客戶**。先看兩個極端方案各自怎麼爆掉。

**全部 db-per-tenant**：5000 + 20 = 5020 個資料庫。連線池 ×5020、migration 每次 ×5020、監控與備份 ×5020。那 5000 個小租戶大多數時間幾乎沒流量，卻各自佔著一個庫的固定開銷——記憶體、連線、維護。資源利用率低到荒謬，營運成本爆炸。為了 20 個真正需要硬隔離的客戶，你給 5000 個不需要的也套上了最貴的方案。

**全部 row-level 共池**：5020 個租戶全擠進同一批表。那 20 個大客戶的查詢和 5000 個小租戶搶同一個 buffer pool——一個大客戶的全表掃描就能拖慢所有人（noisy neighbor 沒處躲）。更麻煩的是合規：大客戶幾乎一定會要求「我的資料要能單獨匯出、單獨刪除、落在指定區域」，而在共池下，「把租戶 X 的資料全撈出來」意味著在一張幾十億列的大表裡按 `tenant_id` 掃——慢、痛、且永遠擔心漏撈或多撈。

務實解一眼就看出來了：**按租戶體量分層。** 5000 個小租戶走 row-level 共池（加上 RLS 兜底，把「漏 `WHERE`」的風險交給引擎），20 個大客戶各自 db-per-tenant（拿到硬隔離、資料主權、獨立調效能）。現在再算 migration 那筆帳：加一個欄位，要跑的是 **1 次（共池那張表）＋ 20 次（20 個獨立庫）= 21 次**——而不是全 db-per-tenant 的 5020 次，也避開了全共池下大客戶合規操作的地獄。同一個產品、同一份程式碼，靠「把不同體量的租戶放在軸線的不同刻度上」，同時拿到了小租戶的效率和大客戶的隔離。

這個 21 對 5020 的對比，就是混合模型的全部道理：**隔離強度不該是全系統一刀切的決定，而該是按租戶分配的決定。** 讓每個租戶住在「剛好夠它的隔離等級」上，不多付、也不少守。

## 一個從第一天就得定、之後極難改的決定

最後要講一件最容易被推遲、卻最不該推遲的事：**多租戶模型是地基決策，不是「先上線、之後再說」的功能。**

具體到一個技術細節上就看得很清楚：在共池模型裡，`tenant_id` 必須從第一天就進**每一張表、每一個索引**——而且在每個複合索引裡，它必須是**最左欄**。原因是 B-tree 複合索引的最左前綴規則（這背後的機制屬於索引那一章的範疇，見〈索引與查詢計畫〉）：一個 `(tenant_id, created_at, status)` 的索引能服務「給定租戶、按時間範圍、再篩狀態」的查詢，因為 `tenant_id` 在最左、planner 能先用它把搜尋範圍夾到單一租戶的那一段。但如果你當初建的是 `(created_at, status)`、`tenant_id` 沒進去或不在最左，那麼「查租戶 42 上個月的已付訂單」就無法靠索引夾到租戶 42——planner 只能掃過**所有租戶**符合時間與狀態的列，再逐列丟掉不屬於 42 的。租戶越多，這條查詢越慢，而且是隨總租戶數退化，不是隨單租戶資料量。

這意味著：如果 `tenant_id` 不是從一開始就織進 schema 與每一個索引，事後補是一場大手術——要改表結構、重建幾乎所有索引、回填歷史資料、還要在不停機的前提下做完。而在三種模型之間切換（比如把一個長成巨獸的共池租戶單獨拆成 db-per-tenant），同樣是搬遷級的工程，不是改一行設定。多租戶模型一旦選定、資料按它的形狀堆積起來，這個決定就接近不可逆（這類「兩扇門」的判斷本身值得單獨想透，見〈兩扇門：可逆與不可逆決策〉）。正因為改起來這麼貴，它才必須在還沒有任何資料、改動成本最低的第一天，被當成一級設計決策慎重地定下來——而不是等到第 5000 個租戶、或那封禮拜五的信來了，才發現地基歪了。

## 為何是這個形狀

退一步看，多租戶隔離的整套樣貌，都是從一個無法迴避的張力裡長出來的：**你想用共用基礎設施服務很多客戶來省成本，但這些客戶彼此不信任、且任何一次串戶都是災難級事故。** 省成本要靠靠近、靠共享；安全要靠分開、靠邊界。沒有一個點同時最省又最安全，所以才會有一整條軸線，而不是一個標準答案。

row、schema、db-per-tenant 不是三個選項，而是這條張力被解到不同強度時，邊界自然落腳的三個位置——邊界畫在每條 query 裡（最省、最脆）、畫在 schema 命名空間上（折中、但撞目錄的牆）、還是畫在獨立的資料庫上（最硬、最貴）。RLS 是想在共池的便宜上補回一點硬度，把守門人從人腦搬進引擎；混合模型是承認「同一條軸線上，不同租戶該站不同的點」。每一種選擇都在回答同一個問題：**這個邊界由誰來守，它在什麼壓力下會破，破了你賠得起嗎。**

下次你在設計一套要服務多個客戶的系統，第一個該問的不是「用哪個資料庫」，而是「我的租戶隔離邊界畫在哪裡、由誰強制、最壞情況下一個租戶能波及多少別人」。把這個問題在第一天想透，因為它是少數幾個「事後幾乎改不動」的地基決定之一——而它守的，是那封禮拜五的信永遠不會被寄出。
