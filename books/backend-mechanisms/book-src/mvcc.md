# MVCC：讓讀不擋寫

一張報表跑了三分鐘。它在掃一張一千萬列的訂單表，逐頁累加當天的營收。就在這三分鐘裡，線上的交易系統照常運作——成千上萬筆新訂單寫進來、舊訂單被改狀態、退款把某些列的金額抹掉。

問題來了：這份報表該看到什麼？

如果它看到「掃描當下每一頁的最新狀態」，那麼它就是一份自相矛盾的報表——它在第一頁讀到的某筆訂單，可能在它讀到第八百萬頁時已經被退款改掉了，於是這份「當天營收」是某個從未真實存在過的時刻的拼貼：一半是三分鐘前的世界、一半是現在的世界。更糟的是，如果讀取要靠鎖來保證一致——讀一列就鎖一列、不讓人改——那這三分鐘裡，整張表的寫入全都得排在這份報表後面。一個唯讀的查詢，把整個寫入路徑凍住了。

這就是並發控制最古老的張力：**讀和寫想同時碰同一份資料，而你既要讀到一個一致的世界、又不想讓讀把寫擋死、也不想讓寫把讀擋死。** 純粹用鎖解這個問題，總會在某一端付出代價——要嘛讀鎖住寫，要嘛讀看到一個正在你眼前變形的世界。MVCC（Multi-Version Concurrency Control，多版本並發控制）是現代 OLTP 資料庫——PostgreSQL、MySQL 的 InnoDB、Oracle、SQL Server 的 RCSI——幾乎一致採用的解法，而它的核心想法，簡單到近乎作弊。

## 不覆蓋，而是堆疊

純鎖路線的根本困境，是它把資料當成一塊**只有一個當前值**的記憶體：要改它，就得先確保沒有人正在看；要看它，就得先確保沒有人正在改。一塊記憶體只能有一個值，於是讀和寫天生要爭奪它。

MVCC 把這個前提整個拆掉。它的一句話是：**寫入時不就地覆蓋舊值，而是寫一個新版本；讀取時不去看「當前值」，而是去找「對我這個交易可見」的那個版本。**

一旦資料變成「同一列、多個並存的版本」，讀和寫就不再爭奪同一塊記憶體了。寫者在堆上添一層新版本，舊版本原封不動留著；讀者拿著自己的「視角」，去舊版本堆裡挑出該由它看到的那一個。兩條路徑解耦——寫者不必等讀者讀完，因為它根本沒動讀者正在看的那個版本；讀者不必擋住寫者，因為它看的是一張已經凝固的快照。PostgreSQL 的官方文件把這個性質寫成一句近乎口號的話：**reading never blocks writing and writing never blocks reading**（讀永遠不擋寫、寫永遠不擋讀）。

那份跑三分鐘的報表，於是有了乾淨的答案：它在開始的那一刻，拿到一張**快照（snapshot）**——一個「對我而言，世界停在這一瞬間」的視角。接下來三分鐘，無論多少新訂單寫進來、多少列被改，它眼裡的世界都不動。它讀到的是一致的、某個確定時刻的全表，而它一根寫鎖都沒上，旁邊的線上交易一筆都沒被它擋住。

但「快照」這個詞太輕巧了。一張快照不可能真的去複製整張一千萬列的表——那太貴。它必須是某種**便宜的、能在每一列上即時判斷「這個版本你看不看得到」的規則**。MVCC 的全部精巧，就藏在這個判斷裡。要真的把它講清楚，得鑽進一列資料的骨頭裡，看看一個「版本」到底長什麼樣。

## 一個版本的身分證：xmin 與 xmax

在 PostgreSQL 裡，每一個列版本（一個 tuple）除了你定義的那些欄位，還在表頭偷偷帶了兩個系統欄位，它們是整套 MVCC 的支點：

- **xmin**：創造出這個版本的那筆交易的 ID。
- **xmax**：刪除或取代了這個版本的那筆交易的 ID。如果這個版本還活著、沒被誰取代，xmax 是 0。

交易 ID（XID）是一個單調遞增的整數，每開一筆交易就發一個更大的號。所以 xmin 和 xmax 不只是兩個標記，它們是兩個**時間戳**——它們把「這個版本存在於哪一段交易區間」釘死了：它從 xmin 那筆交易出生，到 xmax 那筆交易死去。一個版本的一生，就是 `[xmin, xmax)` 這段半開區間。

於是 `INSERT`、`UPDATE`、`DELETE` 在 MVCC 底下，全都被翻譯成對 xmin/xmax 的操作，沒有一個是「就地改值」：

```
INSERT (txn 100):                            -- write a new version
  v1: [xmin=100, xmax=0  ] data=A

UPDATE A -> B (txn 105):                      -- old dies, new is born
  v1: [xmin=100, xmax=105] data=A            -- stamp xmax=105 on the old one
  v2: [xmin=105, xmax=0  ] data=B            -- a brand-new version

DELETE (txn 110):                            -- only stamps xmax, data stays
  v2: [xmin=105, xmax=110] data=B
```

注意 `UPDATE` 不是「修改」——它是一次 `DELETE` 加一次 `INSERT`：把舊版本的 xmax 蓋上自己的交易 ID（宣告「我這筆交易終結了它」），再寫一個全新的版本帶上自己的 xmin。也注意 `DELETE` 沒有真的把資料抹掉——它只是在版本上蓋了一個 xmax，那行死掉的資料還躺在原地。這兩件事都不是實作上的偷懶，它們是 MVCC 能成立的前提：舊版本必須留著，因為**可能還有別的交易，它的視角裡那個舊版本還活著、還該被看見**。

## 快照：你帶在身上的一把尺

現在來解開那張「快照」到底是什麼。它不是資料的拷貝，而是一把**判斷版本可見性的尺**。在 PostgreSQL 裡，一張快照本質上是三樣東西：

- **xmin**：此刻所有「還在進行中」的交易裡，最小的那個 ID。比它更小的交易，要嘛早已 commit、要嘛早已 abort，總之都有定論了。
- **xmax**：下一個還沒分配出去的交易 ID。大於等於它的交易，在我這張快照誕生時根本還不存在。
- **進行中清單（xip list）**：在 xmin 和 xmax 之間，那些此刻仍未 commit 的交易 ID 列表。

有了這把尺，引擎掃到任何一個列版本時，就能用一套機械的規則判斷「這個版本，對拿著這張快照的我來說，可不可見」。直覺上只有一句話：**一個版本可見，若且唯若「生它的那筆交易，在我這張快照成形時已經 commit 了」，而且「殺它的那筆交易，在我這張快照成形時還沒 commit（或根本沒發生）」。** 把它落到 xmin/xmax 的比對上：

```
判斷一個版本 [xmin=C, xmax=D] 對我的快照是否可見：

  生者 C 必須對我「已成定局且已提交」：
    - C 不在我的進行中清單裡（不是還沒 commit 的）
    - C < 我快照的 xmax（不是我之後才開始的交易）
    - 且 C 確實是 committed（不是 aborted）
  → 通過：這個版本「已經出生」

  殺者 D 必須對我「尚未生效」：
    - D = 0（沒人殺它），或
    - D 還在我的進行中清單裡（殺它的交易我看不到它 commit），或
    - D >= 我快照的 xmax（殺它的交易在我之後才開始），或
    - D 其實是 aborted（殺它的交易回滾了，這次刪除/取代根本沒生效）
  → 通過：這個版本「對我還沒死」

  兩個條件都通過 → 可見
```

有一條凌駕上面所有比對的例外：**交易永遠看得到自己尚未 commit 的寫入**——自己這筆交易生的版本對自己可見（即使生者交易還在進行中），自己殺掉的版本對自己立即不可見。沒有這條，你就讀不到自己剛 `INSERT` 進去、還沒 commit 的那一列了。

這套規則的美妙之處，是它把「隔離級別」這個聽起來很抽象的東西，化約成一個**極其具體的工程問題：你這張快照，是什麼時候拍下來的？**

- **Read Committed（讀已提交）**：每執行一條 SQL 語句，就**重新拍一張快照**。所以同一筆交易裡，你連續跑兩次同樣的 `SELECT`，中間若有別人 commit 了改動，第二次就會看到——因為第二條語句用的是一張更新的尺。
- **Repeatable Read / Snapshot Isolation（可重複讀）**：整筆交易在**第一條實際查詢／寫入語句**執行時拍一張快照（不是 `BEGIN` 當下——`BEGIN` 之後、第一條語句之前若有別人 commit，這筆交易仍會看到），之後所有語句共用它。於是無論外界怎麼變，你這筆交易眼裡的世界從頭到尾凝固——這正是「可重複讀」這名字的由來：同一個查詢，跑幾次都一樣。

同一套版本鏈、同一套可見性比對，只是換了「快照何時成形」這一個旋鈕，就長出兩個不同的隔離級別。這就是為什麼說 MVCC 是現代資料庫實作隔離的主力引擎——隔離級別不是一堆獨立的特例，而是「快照時機」這一個參數的不同取值。（更弱與更強的隔離級別、以及 snapshot isolation 仍然防不住的 write skew，屬於另一個主題，見〈隔離級別與異常〉。）

## InnoDB 走的是另一條路：把舊版本藏進 undo log

到這裡你可能以為 MVCC 就是「舊版本和新版本一起堆在表裡」。PostgreSQL 確實是這樣——舊版本就留在 heap（資料堆）裡，原地不動，等著後面清理。但這只是 MVCC 的一種長相。MySQL 的 InnoDB 用同一個想法，卻把舊版本藏到了完全不同的地方，而這個差異會一路滲透到效能與運維的每個角落，值得拆開看。

InnoDB 給每一列也偷偷加了系統欄位，三個：

- **DB_TRX_ID**（6 bytes）：最後一筆改動這列的交易 ID——角色類似 PostgreSQL 的 xmin，差別在它是**就地覆寫**成最新改動者（PostgreSQL 的 xmin 則烙死在每個版本上不變），舊版本的生者 ID 留在 undo 記錄裡。
- **DB_ROLL_PTR**（7 bytes）：一個「回滾指標」，指向 undo log 裡的一筆記錄。
- **DB_ROW_ID**（6 bytes）：在沒有主鍵時，InnoDB 自動生成的隱藏列 ID。

關鍵是第二個。InnoDB 的資料頁裡，**只存最新版本**——它就地把列改成新值。那舊版本去哪了？它被寫進一個叫 **undo log（回滾日誌）**的地方，而 DB_ROLL_PTR 就是一根指向它的指標。需要讀舊版本時，引擎拿著當前的最新版本，順著 DB_ROLL_PTR 往 undo log 裡回溯，一步步「反向套用」那些改動，把列**重建**成它當年的樣子。

兩種路線的對比，可以攤成一張表：

| | PostgreSQL（heap 內多版本） | InnoDB（undo log 回溯） |
|---|---|---|
| 最新版本放哪 | heap 裡（與舊版本並排） | 就地寫在資料頁 |
| 舊版本放哪 | 留在 heap 裡，標記過期 | 寫進 undo log，靠 roll pointer 串成鏈 |
| 讀最新版本 | 要在 heap 裡跳過散落的死版本 | 直接讀，最快 |
| 讀舊版本 | 直接讀那個舊 tuple | 沿 undo 鏈一筆筆反推、重建 |
| 舊版本怎麼清 | 背景 VACUUM 回收 | 背景 purge 丟棄 undo 記錄 |
| 主要代價 | 表膨脹（bloat） | 長交易撐長 undo 鏈、讀舊版本變慢 |

這張表底下藏著一個漂亮的對稱：兩條路線在**讀最新版本**和**讀舊版本**之間做了相反的取捨。InnoDB 把最新版本擺在最順手的位置（就地讀，最快），代價是讀舊版本得沿 undo 鏈一路反推；PostgreSQL 把所有版本平等地攤在 heap 裡，讀哪個版本都是一次直接存取，代價是 heap 會被死版本撐胖、得靠 VACUUM 來瘦身。

InnoDB 對應 PostgreSQL「快照」的東西叫 **read view（讀視圖）**，原理一模一樣：記下「此刻有哪些交易還沒 commit」，據此判斷沿 undo 鏈回溯時，該停在哪個版本。而它的隔離級別差異，也同樣只是 read view 的成形時機之別——InnoDB 官方文件講得很明白：**Repeatable Read（它的預設）下，一整筆交易共用「第一次讀」時建立的那一張 read view；Read Committed 下，每一次一致性讀都建立一張新的 read view。** 一字不差地，這和 PostgreSQL 的快照時機是同一個旋鈕。換了引擎、換了舊版本的藏身處，MVCC 的靈魂沒變。

## 舊版本不會自己消失，這是 MVCC 全部麻煩的根源

到這裡，MVCC 看起來幾乎是免費的午餐：讀寫互不阻塞、隔離級別優雅統一。但有一筆帳一直沒結：那些被 `UPDATE` 和 `DELETE` 留下來的舊版本，誰來清？

答案是——**不是即時清的，是背景慢慢清的，而且只有在「確定再也沒有任何交易需要它」之後才敢清。** 這一句話，是 MVCC 從天堂掉回人間的地方。

想清楚為什麼不能即時清：一個版本被 `UPDATE` 蓋上 xmax 的那一刻，它對「新開的交易」確實死了——但對那份還在跑、快照停在三分鐘前的報表來說，它**還活著、還必須被看見**。引擎根本不能在 `UPDATE` 的當下就把舊版本回收，否則那份報表掃到這一列時，會發現它要看的版本已經被抹掉了。所以回收的判準必須是保守的：**只有當「最老的、還在進行中的交易」都已經越過了這個舊版本的死亡時刻，這個版本才再也不可能被任何人看見，才能安全回收。** PostgreSQL 靠背景的 VACUUM 做這件事，InnoDB 靠背景的 purge 執行緒丟棄 undo 記錄——兩者的回收機制與調校是各自獨立的大主題，這裡只需抓住那個共同的命門：**回收的進度，被「最老的活躍交易」卡著。**

於是一個本來毫不起眼的東西，成了 MVCC 系統最危險的單點：**一筆開著不關的長交易。**

設想一個連線開了交易，跑了一條 `SELECT`，然後……忘了 commit。或者一支 ETL 程式開了一筆交易跑一小時的大匯出。或者一個連線池裡的連線進了 `idle in transaction` 狀態卡死。在這一小時裡，這筆老交易的快照，把「最老活躍交易」這個水位線死死釘在一小時前。結果是：**這一小時內，全庫所有被改過的列，它們的舊版本一個都不能回收**——因為引擎不敢保證那個老交易不會去看它們。

把這件事手算到底，威力才看得清楚。一張一千萬列、每列 200 bytes 的表，磁碟上約 2 GB。某個連線開了交易忘了關，就這麼掛著。這一小時裡，應用對這張表做了一輪全表 `UPDATE`，每一列都被改了一次。在 PostgreSQL 下會發生什麼？

每一列的 `UPDATE` 都生出一個新版本，而舊版本因為那個老交易還在、無法被 VACUUM 回收。於是一千萬個舊版本全部滯留在 heap 裡——表從 2 GB **膨脹到約 4 GB**。而且這膨脹不會在老交易結束的瞬間消失：VACUUM 之後能把那些死 tuple 占的空間標記為可重用，但檔案通常不會自動縮回去（要真正還給作業系統得 `VACUUM FULL`，那會鎖表重寫）。在 InnoDB 下，對應的症狀是 undo log 的「history list length」一路飆升、purge 執行緒怎麼追都追不上，舊版本的鏈越掛越長，讀舊版本越來越慢。

這就帶出 MVCC 運維最反直覺、也最該記死的一條：**最便宜、最有效的 MVCC 調校，不是加大硬碟、不是調 VACUUM 參數，而是「別讓交易開著不關」。** 表膨脹的成因九成是長交易或回收跟不上，加硬碟只是把爆炸延後。要救火，第一個該查的不是磁碟，而是「現在最老的那筆活躍交易開了多久」（PostgreSQL 看 `pg_stat_activity`、InnoDB 看 history list length）。

## 幾個會反咬你的邊界

把 MVCC 講到這裡，已經能在腦中重放它了。但真正能跟人講清楚這個機制的人，得知道它在哪幾個地方會悄悄騙你——這幾個邊界都不顯然，卻每一個都是線上事故的常客。

**「MVCC 等於沒有鎖」——只對讀成立。** 這是最普遍的誤會。MVCC 消除的是**讀寫互擋**：讀不擋寫、寫不擋讀。但它**完全沒有消除寫寫互擋**。兩筆交易同時 `UPDATE` 同一列，第二筆照樣得排隊等第一筆 commit 或 rollback——這是行鎖（row lock）在起作用，和 MVCC 是兩套機制（鎖與死結是另一個主題，見〈鎖與死結〉）。MVCC 讓「讀」這件事便宜到近乎免費，但「改同一列」這件事，本質上仍然是串行的。把「讀不擋寫」誤推成「寫也不會卡」，是並發 bug 的常見起點。

**「`SELECT` 只是讀，無傷大雅」——一個只讀的長交易，照樣是殺手。** 前面那個撐爆表的老交易，可以完完全全是一筆唯讀交易——它一個位元組都沒寫。它的殺傷力不來自它寫了什麼，而來自它**持有一張舊快照**這個事實本身：只要它還活著，它的快照就釘住了回收水位線，逼著引擎保留一大票舊版本「以防它要看」。唯讀不等於無害。一份開在 Repeatable Read、跑了兩小時的分析查詢，對線上庫的膨脹壓力，和一筆瘋狂寫入的交易一樣致命。

**「rollback 一定很貴」——在 PostgreSQL 裡，rollback 反而便宜。** 直覺會說「撤銷一堆改動」聽起來很重，但這個直覺在兩種引擎裡指向相反的方向。PostgreSQL 是 append 式的：`UPDATE` 早就把新版本寫進 heap 了，rollback 不需要「搬回去」——它只要把這些新版本的「生者交易」標記為 aborted，這些版本就自動對所有人不可見了，幾乎是零成本。反過來，InnoDB 是 undo 式的：rollback 一筆改了百萬列的大交易，得沿著 undo log 一筆筆把每一列**反向重建**回原樣，這可能比當初的 `UPDATE` 還慢。同一個詞「rollback」，在兩種 MVCC 實作下的代價直覺，要整個反過來。

**那個沉默的計時炸彈：交易 ID 會用完。** 這是 MVCC 最深、也最少人知道的邊界。PostgreSQL 的交易 ID 是 **32 位元整數**——大約 42 億個號。它不是無限的。整套可見性比對的基礎，是「比較交易 ID 的大小」（生我的交易在你之前還之後？）；可一旦這個 32 位元計數器繞回頭（wraparound），大小比較就會錯亂——一個非常老的版本，它的 xmin 是個很小的數字，繞回之後會突然看起來「比現在還新」，於是被每一筆新交易判定為「來自未來、不可見」。一整張表的歷史資料，可能在某個臨界點集體**憑空消失**。

PostgreSQL 防這件事的手段，本身就是 MVCC 機制的一部分：對那些「老到不可能再有人在乎它出生於哪一刻」的版本，VACUUM 會把它們的 xmin **凍結（freeze）**——換成一個特殊的 `FrozenXID`，這個值不參與一般的大小比較，永遠被當成「比所有正常交易都老」，於是它怎麼繞回都安全。但凍結要靠 VACUUM 去跑——而你已經知道，**長交易會卡住 VACUUM**。於是一條陰險的因果鏈浮現了：一筆忘了關的長交易，不只撐胖你的表，它還悄悄阻止著凍結的推進，把整個叢集往「交易 ID 耗盡」的懸崖邊推。真到了臨界點，PostgreSQL 為了不讓資料損毀，會強迫自己進入單使用者模式拒絕一切連線，逼你手動跑一場可能要數小時的 VACUUM。一個「忘了 commit」的連線，最壞能演變成一次全庫停機——這就是 MVCC 那筆「舊版本不會自己消失」的帳，攤到最極端時的樣子。

## 一個讓 MVCC 不那麼貴的巧思：HOT

MVCC 還欠著一筆隱藏的帳，值得補上，因為它揭示了「append 式 MVCC」最痛的地方，以及工程上怎麼把它補回來。

PostgreSQL 那套「`UPDATE` = 寫一個全新版本」有個討厭的連鎖反應：新版本是一個**物理位置不同的新 tuple**，於是這張表上的**每一個索引**，原則上都得新增一筆指向新位置的條目。如果一張表有五個索引，改一列不碰索引欄位的資料，卻要連帶寫五筆索引條目、未來再清五筆死索引條目。對改動頻繁的表，索引的膨脹常常比 heap 本身的膨脹還凶。

PostgreSQL 的解法叫 **HOT（Heap-Only Tuple，僅堆內元組）**，它的條件與收益都很精準：**當這次 `UPDATE` 沒有改到任何被索引的欄位，而且舊版本所在的那一頁還有空間塞下新版本時**，新版本就直接寫在同一頁，並用一根頁內指標（t_ctid）把舊版本接到新版本上，串成一條鏈。此時——**所有索引一筆都不用動**，因為索引條目仍然指向那條鏈的入口，引擎查索引找到入口後，沿著鏈往前走幾步就能落到當前的活版本。無論這一列被 HOT 更新多少次，索引都不會長大一分。

HOT 不是萬靈丹，它的兩個前提很現實：改到索引欄位（比如更新了 `status` 而 `status` 上有索引）就用不上；那一頁滿了塞不下新版本，也用不上（這也是為什麼寫密集的表常刻意把 `fillfactor` 調低、預留頁內空間給未來的 HOT 更新）。但它精準地命中了 OLTP 最常見的一種寫入——「改幾個沒被索引的欄位」——並把 append 式 MVCC 在這個場景下的索引膨脹幾乎清零。它是個提醒：MVCC 的成本不是鐵板一塊，引擎一直在邊角上做精算，把那筆「不覆蓋、而是堆疊」的帳，能省一點是一點。

## 為什麼是這個形狀

退一步看，MVCC 的整個樣貌，都是從一個拒絕妥協的需求裡長出來的：**讀不該擋寫，寫不該擋讀，而讀又必須看到一個一致的世界。**

純鎖路線給不出這個——它的世界裡資料只有一個當前值，讀和寫注定要爭。MVCC 的破局，是接受「一列可以同時有很多個版本」這個看似浪費的前提，用空間換來並發：寫者只管在堆上添新版本，讀者只管拿著自己的快照去挑該看的版本，兩者再也不必互相等待。隔離級別於是不再是一堆特例，而坍縮成「你的快照什麼時候拍」這一個旋鈕；無論引擎把舊版本留在 heap 還是塞進 undo log，這個內核都不變。

而它要付的代價，也全寫在那個前提裡：舊版本不會自己消失，得有人在背景慢慢清，而清理的進度被「最老的活躍交易」卡著——所以一筆忘了關的交易能撐胖你的表，極端時甚至能把交易 ID 推到耗盡。MVCC 不是免費的，它只是把賬從「讀寫互相阻塞的延遲」換成了「舊版本的空間與回收壓力」。

下次你看到一個監控面板上 `idle in transaction` 的告警、或一張表莫名其妙地膨脹、或一份報表在三分鐘的掃描裡始終看到一個凝固而一致的世界——你會知道，這些全是同一個機制的不同側面：那個「不覆蓋、而是堆疊」的決定，買到的自由，與欠下的債。
