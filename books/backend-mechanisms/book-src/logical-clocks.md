# 邏輯時鐘與排序：Lamport、向量時鐘、HLC 與 TrueTime

兩個機房，相距幾千公里。倫敦那台機器在 `14:00:00.000` 收到一筆「把帳戶餘額設成 100」的寫入；幾百毫秒後，新加坡那台收到「把同一個帳戶設成 200」。後來要對帳，你打開兩筆紀錄一看——倫敦那筆的 timestamp 是 `14:00:00.000`，新加坡那筆居然是 `13:59:59.880`。照 timestamp 排，較晚發生的寫入反而「比較早」。你照著時間先後重放這兩筆，得到的最終值是 100，但實際上世界裡最後發生的是「設成 200」。

沒有人改錯時間。兩台機器都在跑 NTP、都自認準到毫秒。問題出在更根本的地方：**在分散式系統裡，根本不存在一個所有節點都同意的「現在幾點」。** 每台機器有它自己的石英晶體，以略微不同的速率振盪；NTP 會週期性地把它們往標準時間拉，而「拉」這個動作本身——尤其是當本地鐘走太快、需要被往回校正時——會讓牆鐘（wall clock）**往回跳**。（小偏差 NTP 用 slew 緩慢調速、不回跳；但偏差超過門檻時改用 step 瞬間校正，這一跳就可能往回。）一台剛被 NTP 回拉了 120 毫秒的機器，它打出來的 timestamp 會比 100 毫秒前那筆還小。你拿這種會倒退的數字去比大小、去判斷「誰先誰後」，得到的因果關係隨時可能是錯的。

這一章要解的，就是這個看似無害、實則埋著無數隱性 bug 的問題：**在沒有可信全域時鐘的世界裡，怎麼判斷兩個事件誰先發生。** 答案不是「把時鐘調得更準」——那條路有物理上限，永遠關不上最後那道縫。答案是換一個問法。

## 你要的其實不是時間，是因果

退一步問：當你說「事件 A 在事件 B 之前」，你到底在乎什麼？

絕大多數時候，你在乎的不是「A 發生在物理時間軸上更早的某一點」，而是一件更具體的事：**B 有沒有可能受到 A 的影響。** 如果有人先把訂單建立（A），然後才把它取消（B），你需要系統知道「取消」是衝著「建立」來的，不能反過來先處理取消、再處理一筆它根本不知道的建立。你要的是**因果順序**，不是日曆上的時刻。

1978 年 Leslie Lamport 在〈Time, Clocks, and the Ordering of Events in a Distributed System〉（CACM 21(7):558–565，這篇論文後來拿了 2000 年的 Dijkstra 獎）把這個直覺釘成了一個精確的定義，叫做 **happens-before**，記作 `→`。它只認三種「A 確實能影響 B」的情況：

1. **同一個行程內，A 在 B 之前發生**——程式是循序執行的，先跑的那行確實能影響後跑的那行。
2. **A 是「送出一則訊息」，B 是「收到那則訊息」**——你不可能收到一則還沒送出的訊息，所以送必然在收之前。
3. **遞移**：如果 `A → C` 而且 `C → B`，那麼 `A → B`。因果可以串成鏈。

關鍵在於這個定義**沒有提到任何時鐘**。它純粹建立在「行程內的順序」和「訊息的收送」這兩件事上，而這兩件事，每個節點都能在本地百分之百確定地觀察到，不需要任何全域時間。

更關鍵的是它的反面。如果兩個事件之間**找不到**任何一條 happens-before 的鏈——A 影響不到 B，B 也影響不到 A——那它們就是**並行的（concurrent）**。並行不是「同時發生」的意思，它的意思更強也更謙虛：**這兩個事件之間，客觀上不存在先後。** 倫敦那筆「設成 100」和新加坡那筆「設成 200」，如果在它們各自發生時，誰都還沒看到對方，那它們就是並行的——硬要說誰先誰後，是在問一個沒有答案的問題。你能做的只有**人為地訂一條規則**去打破這個平手（例如「節點 ID 小的算前面」），但要清楚那是一條任意的約定，不是在還原什麼真相。

理解了這一點，整章後面的所有機制都只是同一個目標的不同精度：**用一個能在節點間傳遞的計數器，去捕捉 happens-before 這個偏序（partial order），而不去碰那個會騙人的牆鐘。**

## Lamport 時鐘：一個整數能走多遠

最小的可行方案，是給每個行程配一個整數計數器 `C`，初始為 0。規則只有三條，簡單到可以背：

```
on local event:        C = C + 1
on send(msg):          C = C + 1;  attach C to msg
on receive(msg):       C = max(C_local, C_msg) + 1
```

本地做任何事，計數器加一。送訊息時，把當前的 `C` 跟著訊息一起捎出去。收訊息時——這是整個機制的靈魂——把本地的 `C` 和訊息帶來的 `C` 取較大值，再加一。

那個 `max` 是在做什麼？它是在說：「我現在知道，對方在送出這則訊息時，至少已經數到 `C_msg` 了；既然我收到了這則訊息，我這邊的『邏輯時間』就不能落後於它。」這一步把因果關係焊進了計數器——只要 `A → B`，沿著那條 happens-before 的鏈，每一步要嘛加一、要嘛取 max 再加一，`C` 只增不減，所以**保證 `C(A) < C(B)`**。

我們手算一遍。三個行程 P1、P2、P3，計數器都從 0 開始：

```
P1: [e1 C=1] [e4 send m, C=2] ----m---->
                                        \
P2:               [e2 C=1]               \-->[e3 recv m, C=max(1,2)+1=3]
                                                       ----send m2---->
P3:                                                                    [e5 recv m2, ...]
```

P1 做了本地事件 e1，`C` 變 1，送出 `m` 這個動作本身就是事件 e4——送出前先加一，所以 e4 的 `C=2`，`m` 也帶著 `C=2` 出門。P2 先自己悶頭做了 e2，`C=1`；接著收到 `m`，做 `max(C_local=1, C_msg=2)+1`，得到 `C=3`——它一收到就「知道」自己的邏輯時間至少要跳到 2 之後。e1 確實 `→` e3（透過訊息 m），而 `C(e1)=1 < C(e3)=3`，對上了。

到這裡 Lamport 時鐘看起來很完美。但它有一個**致命的不對稱**，而這個不對稱正是它和後面所有機制的分水嶺：**`C(A) < C(B)` 並不能反推出 `A → B`。**

看上面的 e1（`C=1`）和 e2（`C=1`）。它們的計數器一樣大，但這兩個事件之間沒有任何因果關係——它們是並行的。再看 e2（`C=1`）和 e4（`C=2`）：`C(e2) < C(e4)`，但 e2 在 P2、e4 在 P1，兩者之間同樣沒有 happens-before 鏈，它們也是並行的。計數器的大小關係，**只能單向地反映因果，不能反過來證明因果**。看到 `C(A) < C(B)`，你唯一能確定的是「B 不可能在 A 之前發生」（因為若 `B → A` 就會有 `C(B) < C(A)`，矛盾），但你分不出到底是 `A → B`，還是兩者並行。

這個侷限不是 bug，是純量（單一整數）的數學天花板。你想把一個多維的偏序——「誰能影響誰」這張關係網——壓進一個一維的整數軸，必然會壓丟資訊：不同分支上的並行事件，被擠到了同一條線上，再也分不開。

那 Lamport 時鐘還有什麼用？它能給你一個**全序（total order）**。把計數器當主鍵、節點 ID 當 tie-breaker（`C` 相同就比 ID），任意兩個事件都能排出唯一的先後。這個全序的可貴之處在於：它**相容於**因果——只要真的 `A → B`，這個全序裡 A 一定排在 B 前面；它只是在「本來並行、沒有客觀先後」的那些事件對上，硬塞了一個一致但任意的順序。這正是「全序廣播（total order broadcast）」這類需求要的東西：所有節點按同一個順序處理同一批訊息，至於那些並行訊息誰先誰後並不重要，重要的是**大家都同意同一個順序**。要這個、而且不在乎偵測並行，Lamport 時鐘就夠了，而且它只花一個整數。

## 向量時鐘：把丟掉的那一維拿回來

但有些場景，「分不出並行」恰恰是不能接受的。

設想一個多副本的鍵值儲存：同一個 key 在兩個副本上各被改了一次。如果這兩次修改是並行的（兩個客戶端互不知情，同時改了），那是一個**寫入衝突**，系統必須察覺它、保留兩個版本、交給上層去解（衝突解決見〈衝突解決〉）。但如果其中一次修改是看著另一次的結果才下的（`A → B`），那就不是衝突，B 直接覆蓋 A 即可。**「衝突」和「覆蓋」的差別，就是「並行」和「因果在前」的差別**——而這正是 Lamport 時鐘分不出來的那件事。

向量時鐘（vector clock）把丟掉的那一維拿了回來。它的代價是：不再用一個整數，而是用一個**向量**——長度等於節點數，每個節點佔一格。每個行程只增自己那一格，收訊息時**逐格取 max**：

```
on local event at process i:   V[i] = V[i] + 1
on send(msg):                  V[i] = V[i] + 1;  attach whole V to msg
on receive(msg):               for each k: V[k] = max(V_local[k], V_msg[k])
                               then V[i] = V[i] + 1
```

直覺上，`V[k]` 記的是「就我所知，行程 k 至今做過幾件事」。逐格取 max，等於把「我知道的」和「訊息告訴我的」對每個來源各自合併——它保留了每條因果鏈的獨立進度，而不是像 Lamport 那樣把它們全壓進一個數。

向量之間怎麼比？定義 `V(A) ≤ V(B)`，若且唯若**每一格** `V(A)[k] ≤ V(B)[k]`。於是有三種互斥的結果：

- `V(A) < V(B)`（每格 ≤ 且至少一格 <）：**`A → B`**，A 因果在前。
- `V(B) < V(A)`：**`B → A`**，B 因果在前。
- 兩者**互不 ≤**（A 某格較大、B 另一格較大）：**並行**。

注意這裡的關鍵升級：和 Lamport 不同，向量時鐘是**雙向**的——`V(A) < V(B)` 這個關係**等價於** `A → B`，反推也成立。代價就是那個「互不 ≤」的第三種結果，它精確地對應「並行」，而這正是純量怎麼也擠不出來的那一維。

手算一個衝突偵測。三個節點 P1、P2、P3，向量初始 `[0,0,0]`：

```
P1: [1,0,0] --m-->                          [2,0,0]
                  \
P2:                \--> [1,1,0] --m2--> [1,2,0]
P3:                                                  [0,0,1]
```

P1 做一件事到 `[1,0,0]`，送 `m` 給 P2。P2 收到後逐格 max 再增自己那格，到 `[1,1,0]`——這一格清楚記著「我看過 P1 的第 1 件事」。後來 P1 又自己做了一件事到 `[2,0,0]`（P1 從沒收過任何訊息，無從得知 P2 做過事，P2 那格只能是 0）。現在比 `[2,0,0]`（P1 的後續事件）和 `[1,2,0]`（P2 的後續事件）：P1 那格 `2 > 1`，但 P2 那格 `0 < 2`——**互不 ≤，並行**。它們確實是各做各的、誰也沒看到對方的後一步。再看 P3 的 `[0,0,1]`：跟前面每一個都互不 ≤，它從頭到尾沒跟任何人通訊，跟所有事件都並行。向量時鐘把這張「誰看過誰」的關係網完整地保了下來。

這個能力不是免費的。向量的長度等於節點數，每傳一則訊息都得把整個向量捎上——**節點越多，每筆元資料越胖**，這在動輒上千節點的系統裡會痛。更陰險的是節點的**動態增減**：節點來來去去，那些離開的節點留下的舊格子要不要清、什麼時候清才安全，是一個沒有漂亮答案的垃圾回收問題（清早了會誤判因果，清晚了向量無限長）。所以實務上向量時鐘很少用在「全體節點」這個尺度，而是用在範圍受控的地方——典型是**版本向量（version vector）**：它是向量時鐘的特例，每一格對應一個**資料副本**而非一個行程，專門用來在副本間偵測寫入衝突。Amazon 的 Dynamo 就是用這套機制來追蹤每個 key 的因果依賴、揪出需要協調的並行寫入——值得一提的是，Dynamo 論文自己把它叫做 vector clock，但因為每一格對應的是一個副本（而非任意行程），它在技術上其實正是版本向量；這也是這個機制常被點名術語用得不夠精確的地方。範圍從「所有節點」縮到「這個 key 的幾個副本」，向量就短了，動態增減的問題也緩和了。

到這裡，純邏輯時鐘的兩端都齊了：Lamport 用一個整數換來「相容因果的全序」，向量時鐘用一個向量換來「精確的並行偵測」。但兩者都有一個共同的、在某些場景下很要命的缺陷——**它們和真實時間徹底脫鉤**。一個 Lamport 計數器是 `8273`、一個向量是 `[5,2,9]`，這些數字無法告訴你「這件事大概發生在哪個下午」。你沒辦法拿它們去回答「給我過去一小時的所有事件」，沒辦法做基於時間的資料保留，沒辦法跟外部世界（log、人類、其他系統的 timestamp）對齊。下一步要補的，正是這道縫。

## HLC：讓邏輯時鐘長得像真的時間

純邏輯時鐘和牆鐘各有各的好，也各有各的破。牆鐘貼近真實時間、人看得懂、能做時間範圍查詢，但它會往回跳、跨機器有偏差，不能用來判因果。邏輯時鐘能可靠判因果、永遠單調，但它是一串和真實時間脫鉤的抽象數字。

混合邏輯時鐘（Hybrid Logical Clock，HLC，Kulkarni、Demirbas 等人 OPODIS 2014 論文〈Logical Physical Clocks and Consistent Snapshots in Globally Distributed Databases〉）的想法，是把兩者的優點縫在一起：用一個 tuple `(pt, c)`，`pt` 是一段貼著本地物理時鐘走的「物理部分」，`c` 是一個小小的邏輯計數器，專門用來在物理時間「卡住沒動」時，仍然維持單調與因果。

更新規則的精神，是**永遠不讓 HLC 倒退**，同時盡量讓它貼著真實時間：

```
on local or send event:
    pt' = max(pt_last, wall_clock_now())
    if pt' == pt_last:  c = c + 1          # physical time didn't advance, bump logical
    else:               c = 0              # physical time moved, reset logical
    (pt, c) = (pt', c)

on receive(msg with (pt_m, c_m)):
    pt' = max(pt_last, pt_m, wall_clock_now())
    if pt' == pt_last == pt_m:  c = max(c_last, c_m) + 1
    elif pt' == pt_last:        c = c_last + 1
    elif pt' == pt_m:           c = c_m + 1
    else:                       c = 0
    (pt, c) = (pt', c)
```

別被分支嚇到，骨幹很單純：**取「上次的 HLC、收到的 HLC、本地牆鐘」三者中物理部分最大的那個當新的 `pt`**；只有當這個 `pt` 跟上一次一模一樣（物理時間沒前進，或被往回校了）時，才靠 `c` 加一來維持單調，否則 `c` 歸零。

這個設計買到了三件事，每件都對得起前面的鋪陳：

- **單調**：因為總是取 max，HLC 的 `(pt, c)` 字典序永遠只增不減，即使本地牆鐘被 NTP 往回校，`pt` 也不會跟著倒退（它被 `max(pt_last, ...)` 鎖住了），`c` 會接手往前走。前面那個「NTP 回跳導致 timestamp 倒退」的開場災難，在 HLC 下消失了。
- **捕捉因果**：收訊息時把對方的 `(pt_m, c_m)` 納入 max，跟 Lamport 同一個道理——`A → B` 必然 `HLC(A) < HLC(B)`。
- **貼近真實時間、塞得進 64 位元**：`pt` 一直被本地牆鐘往前拉，所以 HLC 和真實時間的差距有上界——它**超前真實牆鐘不會超過時鐘間的偏移上界**。而且 `c` 的成長受到嚴格約束（只在物理時間沒動時才加），實作上 `pt` 用約 48 位元、`c` 用約 16 位元，整個塞進一個 64 位元整數，傳輸成本跟一個普通 timestamp 一樣，遠比向量時鐘便宜。

這就是為什麼現代分散式資料庫幾乎清一色用 HLC 來給交易排序：CockroachDB 用它（下一節細看），MongoDB 用它實作 cluster time、達成因果一致（causal consistency）。

但 HLC 有一個必須講透的邊界，否則就會被誤用：**`c` 那一部分讓 HLC 可能略微超前真實牆鐘，所以 HLC 是排序工具，不是計時工具。** 它的 `pt` 看起來像個 timestamp，誘惑你把它當成「這件事發生的精確時刻」對外承諾——別這麼做。如果一段時間內物理時鐘沒前進、`c` 一路 bump，HLC 的 `pt` 會停在某個值，而它代表的「邏輯時間」其實已經前進了好幾步；反過來，繁忙時 HLC 可能略微跑在真實牆鐘前頭。它保證的是「單調」和「因果」，從來沒保證「等於物理現實」。把 HLC 的 `pt` 當作 SLA 裡承諾的精確 timestamp、或拿去做需要真實時間精度的計費，是把一個排序工具誤當計時工具用。

順帶一個很容易被略過、卻在「敵意環境」下要命的邊界：HLC 收訊息時會 `max` 對方帶來的 `pt`。如果某個客戶端（或被入侵的節點）謊報一個遠在未來的 `pt`，整個叢集的 HLC 會被這個謊言一口氣拖到未來，而且因為單調、再也回不來。MongoDB 正是因此給 cluster time 加了一層**簽章（signed cluster time）**——伺服器只接受帶有合法簽章的邏輯時間，讓一個惡意客戶端無法把全叢集的時鐘推到天荒地老。這提醒我們：HLC 的「永不倒退」是雙面刃，它讓你扛得住 NTP 回跳，但也意味著任何一次被污染的前進都洗不掉。

## CockroachDB 怎麼用 HLC，與「不確定區間」這道縫

把 HLC 放進一個真要保證可序列化交易的資料庫，會撞上一個 HLC 自己解不掉的問題，值得拆開看——因為它把「時鐘偏移從來沒有真正消失、只是被管理」這件事講得最清楚。

CockroachDB 用 HLC 給每筆交易蓋時間戳，並用時間戳的大小來決定 MVCC 版本的可見性（讓讀不擋寫的多版本機制，見〈MVCC〉）。表面上 HLC 給了單調與因果，看起來夠了。但有一個情況 HLC 答不出來：節點 A 在它的 HLC 時間 `t` 發起一筆讀，碰到一筆寫，這筆寫的時間戳是 `t + 30ms`。問題是——這筆寫到底是「在我這筆讀**之後**才真正發生」（那我不該看到它），還是「其實在我之前就發生了，只是寫它的那台機器時鐘**快了 30 毫秒**，才打出一個比我大的時間戳」（那我**應該**看到它，否則就讀漏了一筆因果在我之前的寫）？

光看兩個時間戳，分不出來。這正是時鐘偏移的幽靈——它沒被 HLC 消滅，只是換了個地方冒出來。CockroachDB 的處理方式是**承認偏移有一個上界，並把它顯式建模成一段「不確定區間（uncertainty interval）」**：每筆讀帶著 `[t, t + max_offset]` 這段窗口，預設 `max_offset` 是 **500 毫秒**（2026-06；可調，多區域部署官方甚至建議降到 250ms）。讀到時間戳落在這段窗口內的值時，系統**不敢**斷定它在過去還是未來，於是觸發一次 **uncertainty restart**：把交易的讀時間戳往前推過這個可疑值，重跑——這一次，那些原本可疑的值就確定在過去了，因果不會被讀漏。

這套機制有個前提：**所有節點的時鐘偏移真的不超過 `max_offset`**。萬一某台機器的時鐘飄超過這個上界，前面那套「不確定區間」的保證就破了——它可能打出一個落在別人窗口**之外**、卻其實因果在前的時間戳，被當成「確定在過去/未來」而排錯。CockroachDB 為此設了一道殘酷但必要的自保：當一個節點偵測到自己的時鐘和半數以上節點的偏差超過 `max_offset` 的 **80%**（預設就是 400ms），它會**立刻自殺（spontaneously shut down）**，寧可少一個節點，也不讓一台時鐘失準的機器留在叢集裡破壞一致性。

這就是 HLC 的誠實之處，也是它的代價：它沒有假裝時鐘是準的，但它把「時鐘有多不準」這個假設攤在桌上，變成一個你必須維護（靠 NTP/PTP）、一旦違反就會出錯的**運維契約**。`max_offset` 設大了，不確定區間就寬，uncertainty restart 變多、寫入延遲變高、效能變差；設小了，對時鐘同步的要求就嚴，越容易踩到 80% 那條自殺線。這裡沒有最佳值，只有取捨——而這個取捨，本質上就是「你願意花多少力氣讓時鐘同步」換「你能容忍多少 restart」。

## TrueTime：花錢把不確定性壓到能等過去

CockroachDB 把不確定性管理成一段最大 500 毫秒的區間，靠重試兜住它。Google Spanner 走了相反方向的極端：**與其容忍不確定，不如花硬體把它壓到小得可以直接等過去。**

Spanner 的 TrueTime API 有一個誠實到反直覺的設計：你問它「現在幾點」，它**不回一個值，回一個區間** `[earliest, latest]`，並且向你保證「真正的絕對時間，一定落在這個區間裡」。區間的半寬叫 ε（epsilon）。一般的時鐘 API 假裝自己準，給你一個點，把誤差藏起來騙你；TrueTime 反過來，把誤差**顯式**地交到你手上——它不告訴你確切時刻，但它保證真相不會跑出這個界。

要把 ε 壓得夠小，得砸硬體：Google 在每個機房部署 **GPS 接收器和原子鐘**兩套獨立的時間源（互為備援、防單一源故障），讓 ε 維持在很小的範圍。最初的 Spanner 論文裡 ε 典型約數毫秒（常被引用為 < 4ms，實際會在每次和時間主機同步後從約 1ms 漲到下次同步前約 7ms 之間擺盪）；到了 2023 年，Google 報告 TrueTime 在 p99 已能做到不確定性**小於 1 毫秒**。這是純軟體、靠 NTP 的系統摸不到的數量級——CockroachDB 之所以用 500ms 那麼寬的 `max_offset`，正是因為它假設你跑在沒有 GPS 原子鐘的普通機房。

有了一個小而**有保證上界**的 ε，Spanner 能買到一般系統買不起的東西——**外部一致性（external consistency）**，也就是最強的那種保證：如果交易 T1 在真實世界裡完成後，T2 才開始，那麼 T1 的 commit 時間戳一定小於 T2 的。注意這個保證連結了**真實世界的時間先後**，而不只是系統內部的某種一致順序——它意味著「銀行先存了款再嘗試提款，任何讀者都不會看到提款在餘額不足下成功」這種跟物理時間綁定的承諾。

它怎麼用 ε 做到？靠一個叫 **commit-wait** 的動作。一筆寫入交易選定一個 commit 時間戳 `s`（取自 `TT.now().latest`，即區間的上界）之後，**不立刻對外宣告提交**，而是刻意等到 `TT.now().earliest > s`——也就是等到「現在的絕對時間，確定已經越過了 `s`」——才真正讓這筆交易可見。等多久？大約 **2ε**：因為要先讓 `s` 大過當下區間的上界，再等真實時間走過 `s`。等過這 2ε，TrueTime 就向你保證：任何在這筆交易**之後**才開始的交易，它拿到的時間戳一定更大。先後關係被時間戳忠實地反映了，外部一致性成立。

這裡有一個常被漏掉、但很解釋現實效能的細節：**commit-wait 在多數情況下其實不用真的等。** 一筆 Spanner 寫入本來就要跑一輪 Paxos 共識（讓副本們同意這筆 log，見〈共識〉），而這輪 Paxos 的網路往返時間，常常**已經比 2ε 還長**。等 Paxos 跑完，那 2ε 早就在背景流逝掉了，commit-wait 那一刻往往發現「時間已經夠了」，無需額外停等。所以 commit-wait 的成本是「2ε 和 Paxos 延遲取 max、再扣掉重疊」，而不是死板地每筆加 2ε——當 ε 被壓到 < 1ms、而跨區 Paxos 動輒數毫秒到數十毫秒時，這個 wait 在實務上常常趨近於零。

把整條取捨鏈收束成一句：**你想要的保證越強，付的代價越具體。** 要外部一致性（這是 HLC 給不了的——HLC 只保證單調與因果，不保證時間戳忠實反映真實世界先後），你就得把時鐘的不確定性壓到能「等過去」的程度，而把 ε 壓到 < 4ms 甚至 < 1ms，需要 GPS 加原子鐘的專屬硬體——這在自架機房裡幾乎不可得，是 Spanner 這類全託管系統才提供得起的奢侈品。CockroachDB 用便宜的 NTP 加 500ms 區間加 uncertainty restart，買到的是較弱但夠用的保證；Spanner 用昂貴的硬體加 commit-wait，買到的是最強的外部一致性。同一個問題——「怎麼在不確定的時鐘下排序」——兩種預算、兩種答案。

## 該用哪一個，與一個反覆出現的誤用

把這四種機制攤在一條「精度 vs 成本」的軸上，選擇其實很清楚——關鍵是先問清楚自己到底要什麼保證：

| 機制 | 保證 | 成本 | 什麼時候用 |
|---|---|---|---|
| Lamport timestamp | `A→B ⇒ C(A)<C(B)`；可得任意全序 | 一個整數 | 只要「一致的全序」、不需偵測並行（如全序廣播） |
| 向量時鐘 / 版本向量 | `V(A)<V(B) ⇔ A→B`；可偵測並行 | 大小隨節點/副本數線性成長 | 要分辨「衝突」與「覆蓋」（衝突偵測、因果一致） |
| HLC `(pt,c)` | 單調、捕捉因果、貼近牆鐘、64-bit | 受時鐘偏移上界約束；非真實時間 | 分散式資料庫排序交易、一致性快照，無原子鐘可用 |
| TrueTime + commit-wait | 外部一致性（反映真實世界先後） | GPS＋原子鐘硬體；每筆寫付至多約 2ε | 全球跨區、需要最強一致、且付得起硬體 |

最該記住的一條判準是：**絕大多數系統不需要爬到這張表的下端。** 把「事件用牆鐘排序」這個會出錯的習慣，換成「量間隔用單調鐘（monotonic clock）、判因果用邏輯時鐘」，就解決了九成的問題。需要在副本間偵測寫入衝突，向量／版本向量就夠。真的要 Spanner 等級的外部一致性、願意吞硬體與延遲成本的，是極少數——而很多自以為需要的場景，其實最終一致加因果排序就綽綽有餘。

最後釘死那個反覆出現、代價又高的誤用——它其實就是這一章開場那一幕的根源：**拿牆鐘（`Date.now()`、`System.currentTimeMillis()`）去比兩個跨機器事件的先後。** 牆鐘會被 NTP 往回校、跨機器有偏差，這個比較隨時會給你錯誤的因果，而且錯得隱蔽——它平常都對，只在某次 NTP 校正、某次 VM 遷移、某次跨機房延遲尖峰時悄悄錯一次，留下一筆「不可能發生」的資料讓你半夜對帳對到懷疑人生。判先後用邏輯時鐘，量「過了多久」用單調鐘，把牆鐘留給它唯一適合的工作：對人顯示一個大概的時刻。這兩件事——「排序」和「計時」——在分散式系統裡是兩個必須分開的問題，而這一章從頭到尾，講的都是為什麼第一件事不能用第二件事的工具去做。

## 為何是這個形狀

退一步看，從 Lamport 的一個整數，到 TrueTime 的 GPS 原子鐘，這四種機制其實是同一條原理被推到不同預算下的產物：**因為沒有可信的全域時鐘，所以你不能問「絕對幾點」，只能問「誰能影響誰」。**

Lamport 用最小的代價——一個只增不減、收訊息時取 max 的整數——捕捉了這個因果偏序，但把它壓進一維，丟掉了並行的資訊。向量時鐘多花一個隨節點數成長的向量，把那一維贖了回來，換來精確的衝突偵測。HLC 看出純邏輯數字和真實世界脫鉤的不便，把牆鐘縫進來，在不犧牲單調與因果的前提下讓時間戳「看起來像時間」、塞得進 64 位元，於是成了現代分散式資料庫的預設。而 TrueTime 走到盡頭：它不假裝時鐘準，而是誠實地把不確定性量化成一個有上界的區間，再花硬體把這個區間壓到小得可以「等過去」，換來那個其他機制都給不起的外部一致性。

每一步都是同一個取捨在不同位置上的兌現：你想要的保證越強——從「相容因果的全序」，到「偵測並行」，到「貼近真實時間」，到「忠實反映真實世界先後」——你付的代價就越具體，從多一個整數，到多一個向量，到一個運維契約，到一整套 GPS 原子鐘。分散式系統裡沒有免費的時間。你看到的每一個時鐘設計，都是某個人在「我需要多強的排序」和「我願意付多少」之間，劃下的一條線。
