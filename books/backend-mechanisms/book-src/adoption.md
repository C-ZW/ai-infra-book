# 推動採用：誘因、轉換成本與信任

你寫了一條更好的 lint 規則。它能擋掉一整類已經咬過你們三次的 bug，設定只有八行，你在自己的服務上跑了兩週、零誤報。你把它發到團隊頻道，附上一段清楚的理由，按下送出，心想這種東西總不需要推銷吧——它客觀上更好，誰會反對？

三個月後你回頭看，採用率停在大概一半。有兩個服務照做了，有一個服務的人說「下個 sprint 一定加」然後再也沒提，剩下的那幾個倉庫連設定檔都沒動過。沒有人公開反對你，沒有人說這規則不好。它只是……沒有發生。

這是推動採用這件事最反直覺、也最讓工程師受挫的地方：**一個做法在技術上正確，和它真的被一群人採用，是兩個幾乎不相關的問題。** 這看起來像一個技術問題（這規則對不對），底下其實是一個行為改變問題（怎麼讓一群各有各的當下壓力的人，改掉一個已經能用的舊習慣）。把這兩件事混為一談，是所有「明明更好卻推不動」的故事的共同起點。

這一章要把「採用」當成一個有結構的問題來拆——它有自己的第一原理、自己的故障模式、自己會在哪裡騙你。拆透之後你會發現，「東西夠好就會自己被採用」這句話不只是樂觀，它在多數情況下是**錯的**，而且錯得有規律可循。

## 為什麼「更好」推不動：成本與好處的錯位

先把「一個做法為什麼會被採用，或不被採用」這件事拆到底。退到第一原理，只有三股力量在拉扯。

第一股是**誘因（incentive）**。人會走對自己當下最有利的那條路——不是長期最有利，是當下。如果採用你的新做法，讓「我這週要交付的那個功能」變得更難做、更慢出、更容易出錯，那麼無論你給的長期理由多漂亮，它在這個人的當週決策裡都是負分。誘因不是「他懂不懂道理」，是「這件事此刻對他是順風還是逆風」。

第二股是**轉換成本（switching cost）**。從舊習慣到新習慣之間，總有一段山谷：你已經離開了熟悉的舊做法，但還沒拿到新做法的好處，中間是純粹的痛——要重讀文件、要改既有的程式、要在不熟的工具上犯幾次新手錯誤。這段山谷有多深、要趟多久，往往就決定了採用會不會發生。一個技術上只好一點點、但轉換成本很高的東西，理性的人會選擇不動——他不是看不出它更好，是算出來「不值得」。

第三股，也是最容易被工程師低估的，是**信任（trust）**。團隊願不願意賭你的判斷？這不是一次性的問題，而是一個**存量**。它由你過去推過的東西「後來證明是對的」一點一點累積起來；也會因為你推過一個東西、做到一半棄坑、留下半套爛攤子給別人收拾，而被**一次燒光**。信任低的時候，連一個客觀上很好的提議都會被打折——因為大家賭的不是「這個提議好不好」，是「你說好的東西，過去有沒有真的好」。

把這三股力量擺在一起，會浮現出推動採用真正的難處所在。它不是「新東西不好」，新東西通常真的更好。難處是一個**結構性的錯位**：

> 轉換成本由**採用者**在**現在**付，好處卻歸於**系統**或**未來**。

採用者要趟那段山谷、付那個當週的逆風；而新做法的好處——更少的 bug、更好維護的程式碼、未來某天的某個人少踩一個坑——分散在系統層面、攤在未來的時間軸上，**不落在那個要付出成本的人身上**。一個人被要求現在掏錢，買一張別人未來才會用到的票。這個錯位不修平，再好的東西也只會卡住。理解了這一點，後面所有的招數其實都是同一件事的不同做法：**想辦法把這個錯位補平**——要嘛降低採用者付的成本，要嘛把好處提前兌現到採用者身上，要嘛用信任去借支那段山谷，讓採用者先付出、相信好處會來。

## 三條路線：強制、擴散、鋪路

知道了要補平錯位，接下來是怎麼補。讓一群人採用一個做法，骨子裡只有三條路線，它們的差別不在「用什麼工具」，而在**靠什麼力量讓人改變行為**。

**第一條是由上而下的強制（top-down）。** 管理層或架構組宣布：從某日起，一律用 X。它的機制是**用權威壓過個人誘因**——你當下不想改沒關係，這不是選項。強制的好處是快、是一致：它是唯一能在短時間內逼近「全員照做」的路線。它的代價藏在一個容易被忽略的區別裡——**服從不等於認同**。人前照做、人後繞過，是強制最典型的失敗形態。一個被強推、但團隊心裡不服的做法，會在你沒看的角落悄悄回滲：有人偷偷保留舊寫法、有人把檢查關掉、新人不知道為什麼要這樣做於是照抄了一個錯誤的理解。所以強制有一條鐵律：**強制必須配自動化護欄**，把「不照做」變成機器層面根本進不來，否則你買到的只是表面的服從，底下照舊各做各的（讓對的做法自動成為阻力最小的路，與用機器把守邊界，本身是一整套機制，見〈鋪路：讓對的做法最省力〉）。

**第二條是由下而上的擴散（bottom-up）。** 你不宣布、不強制，你先在一兩個地方把新做法做出明顯有效的範例，然後靠口碑和模仿自己長出去。它的機制和強制相反——不是壓過誘因，而是**讓誘因自己對齊**：別人看到你用了 X 之後交付更快、oncall 更少，他不需要你勸，他自己會跑來問怎麼用。擴散換來的採用是自願的，認同度高，阻力小。代價是**慢**，而且常常擴散不完全——這一點重要到值得單獨講。

擴散的失敗幾乎都長一個樣：卡在某個百分比上不去。這不是巧合，創新擴散的研究早就描出了這條曲線的形狀。Everett Rogers 在 1962 年的《Diffusion of Innovations》裡，把一個群體對新事物的採用者分成五類——創新者、早期採用者、早期多數、晚期多數、落後者——並指出採用隨時間是一條 S 形曲線。後來 Geoffrey Moore 在《Crossing the Chasm》裡點出這條曲線上有一道**鴻溝**：早期採用者（願意嘗鮮、容忍粗糙的人）和早期多數（務實、要看到別人先成功才肯動的人）之間，存在一個**信任的斷層**。每一群人傾向相信「跟自己同類的人」的背書，而不是相信前一群人。早期採用者覺得「酷、我試試」的東西，務實的早期多數會冷眼看著問：「有跟我一樣的人，在跟我一樣的處境下，靠它真的成功了嗎？」

這就是為什麼純擴散會卡住。你在團隊裡最愛嘗鮮的那兩個人那裡，採用毫不費力——他們本來就喜歡新東西。但越往後，剩下的人越務實、越保守，他們要的不是「這東西很酷」，是「有人在我的處境下、用它把我這種活幹成了」。如果你拿不出一個和他們夠像的成功案例去跨過那道信任斷層，擴散就停在早期採用者那一小撮人，剩下的大多數永遠不動。開頭那個 lint 規則停在一半，多半就是卡在這裡：愛乾淨的人加了，務實的人在等一個「跟我的服務一樣複雜的服務，加了它沒出事」的證據，而你一直沒給。

**第三條是鋪路（paved road）。** 它既不靠權威壓人，也不純靠口碑自然發生，而是主動把「照建議做」那條路，修到比任何其他選擇都明顯省力——模板、預設設定、現成工具一次給足，沿著它走幾分鐘就生出一個符合所有最佳實踐的東西，偏離它則要自己處理一堆瑣事。它順著「人會走阻力最小的路」這個天性，把好處加在「照做」上，而不是把成本加在「不照做」上。鋪路是補平那個錯位最優雅的一招——它直接把轉換成本壓到趨近於零。但它有自己的硬成本（那條路要持續養護，鋪了不養比沒鋪更糟），整套機制是另一個故事（見〈鋪路：讓對的做法最省力〉）。

這三條路線不是互斥的選擇題，現實裡最穩的推動往往是**疊加**：鋪好路讓多數人自願上路，找一個受信任的同儕先示範跨過信任鴻溝，最後用少量強制收口那幾個怎麼都不動的死角。

## 同一件事，團隊一變大就換一條路

上面三條路線聽起來像可以挑，但有一個變數會直接決定哪條路活、哪條路死，而它常被忽略：**團隊規模**。同一件要推的事，在三個人的團隊和五十個人的組織裡，能用的路線完全不同。把這個推演走一遍，比任何抽象原則都清楚。

設想要推的事很小：所有 PR 的 commit message 都照一個固定格式寫（type、scope、描述，類似 Conventional Commits 那套規格）。

在**一個三人團隊**裡，這件事的成本幾乎是零。你在站會上講一句「以後 commit 照這個格式」，三個人都聽到了、都認同了（因為他們都信得過你，三個人的信任存量是滿的，你們天天一起寫碼），下次 review 看到沒照做的順手提一句就改了。你甚至不必寫文件——top-down 在這個規模下幾乎沒有成本，因為「宣布」這個動作的頻寬足夠覆蓋整個團隊，訊息不會在傳遞中失真，信任也不需要額外去掙。

把**完全同一件事**搬到一個**五十人的工程組織**，每一個前提都塌了。你的「宣布」在傳到第三個團隊時就失真了：有人根本沒看到那則訊息；有人看到了但不認同（他覺得現有格式也沒問題，憑什麼改）；有人認同但下週就忘了；新加入的人從來不知道有這回事，照著倉庫裡看到的舊 commit 學了個錯誤的範例。口頭的 top-down 在這個規模下會退化成「上面說了、下面各做各的」——看似推動了一個慣例，實際上只是製造了一份沒人遵守的公告。

要在五十人規模真的推起這件小事，你得換成疊加的打法：先**鋪路**（給一個 commit 範本和幾個可複製貼上的範例，把「該怎麼寫」的選擇焦慮消掉）；再上**自動化強制**（一個 commit-lint 掛在 CI 上，格式不對的 PR 根本合不進來——這是唯一不靠個人記憶與自律的防線）；再找**一個受信任的團隊先示範**，讓「他們用了，沒影響交付速度」替你跨過那道信任鴻溝。

這裡的教訓不是「大組織比較麻煩」這種廢話，而是一個能拿來判斷的原則：**團隊規模一變，能用的路線就跟著變，因為規模直接改變了那三股力量的強弱**。規模大了，「宣布」的頻寬不夠覆蓋所有人（誘因傳不到），共同協作的時間變少（信任存量稀薄），而既有的存量（一堆不照格式的舊 commit、一群習慣了舊做法的人）變成需要專門處理的轉換成本。你不能拿三人團隊那套零儀式的打法去推五十人組織的慣例，反過來也一樣——在三個人的團隊裡架一套 CI gate 加範本加示範團隊，是把一句話能解決的事做成一個工程專案，是另一種形式的過度設計。**推動的力道要對齊團隊的規模**，這是這整件事最實用的一條判準。

## 什麼時候根本不需要推

把上面的機制反過來用，能回答一個同樣重要、卻很少被問的問題：**什麼時候你根本不該花力氣去「推動」？** 推動是有成本的——你的時間、你的信任存量、團隊被打擾的注意力。把這些花在不需要的地方，本身就是一種浪費，甚至會反噬。

判準回到那個錯位。推動之所以必要，是因為「轉換成本由採用者付、好處歸於系統或未來」這個錯位存在——你得花力氣去補它。那麼反過來，**當這個錯位不存在時，推動就是多餘的，甚至有害。**

最清楚的訊號是這個問題：**採用者的第一週，是更痛還是更爽？**

如果一個新做法讓採用者**當下就更省力**——第一天用就比舊做法快、比舊做法少踩坑、立刻拿到看得見的甜頭——那它根本不在那個錯位裡。好處就落在付出成本的人身上，誘因天然對齊。這種東西你不必推，它會自己擴散：一個人用了爽，會主動講給隔壁聽，隔壁試了也爽，於是它像好消息一樣自己傳開。對這種東西架一整套推動計畫——開會、發文件、設里程碑、追進度——只會顯得官僚，甚至引起反感（「這麼好用的東西還要你逼我用？」）。你要做的只是把它放在一個容易被看見、容易被拿來用的地方，然後讓它自己走。

只有當答案是「第一週更痛」時——採用者要先趟一段山谷才拿得到好處——你才真正面對那個錯位，才需要動用前面那些槓桿：算清楚轉換成本有多深、找信任去借支那段山谷、設護欄防止回滲。**先分清這個東西落在哪一邊，再決定要不要推、推多重**，比一上來就規劃推動計畫重要得多。很多推動的力氣，是花在那些本來放著就會自己長起來的東西上——那不是推動，是擾民。

## 它會怎麼騙你

推動採用是一個直覺特別容易出錯的領域，因為它的失敗大多是**安靜的**——沒有報錯、沒有當機，只是某個你以為發生了的改變，其實沒發生。幾個最常見、也最危險的錯覺：

**「東西夠好就會自己被採用。」** 這是最根本的誤解，前面整章都在拆它。好處歸於未來、痛苦歸於現在的東西，從不自己擴散——它違反了採用者當下的誘因。會自己擴散的，只有那些「第一週就更爽」的東西；其餘的都需要刻意補平錯位。把「它客觀上更好」當成「它會被採用」的理由，是推動失敗最常見的源頭。

**把「強制」當成「採用」。** CI 加了規則、文件寫了、會也開了、公告也發了——這些都不等於團隊真的改了行為。一個沒有自動化護欄的強制，會在你看不到的角落悄悄回退：有人本機把 hook 繞過去，有人在新服務裡又用了舊寫法，因為沒有任何一道機器層面的閘真正擋著。「公告」「文件」「會議」量的是你做了多少推動的**動作**，不是團隊改變了多少**行為**——這兩者之間的落差，正是大多數「以為推成了、其實沒有」的故事發生的地方。採用的唯一證據是行為改變，而不是你下達了多少指令。

**一次推太多。** 一個團隊在任何時候，能消化的轉換成本是有預算的。你同時推五件事——換 lint、換部署流程、換 RPC 框架、改 commit 格式、上新的監控——這個預算會瞬間爆掉，結果不是五件都推成，而是五件都卡住、一件都沒落地，因為每一件都在搶同一份稀缺的「願意現在承受改變之痛」的額度。穩的做法是一次推一件，落地了、穩了，再推下一件。推動是序列的，不是平行的。

**燒掉信任不補。** 信任是存量，而存量最殘酷的性質是：**累積很慢、清零很快**。你花一年推成三個東西、每個後來都證明是對的，慢慢攢起一點「他說好的通常真的好」的信任；然後你推第四個，做到一半失去興趣，棄坑，留下一個半套的、誰也不敢動也不敢刪的爛攤子給別人收。這一次棄坑，會把前面攢的信任大半燒掉——下一次你說「這次真的更好、相信我」，再也沒人願意賭。所以推動有一條紀律：**不要推你沒打算負責到底的東西**。每一次有頭無尾的推動，都是在預支你未來所有推動的成功率。

**忽略最後那兩成。** 由下而上的擴散最容易卡在「大部分人改了、剩幾個死角永遠不改」——就是前面那道信任鴻溝沒跨過去的那群務實派。如果你不給並存狀態設一個終點（一個明確的退役日期，加上一道擋住新增舊用法的護欄），這剩下的兩成會逼你**永遠同時維護新舊兩套**。而維護兩套的成本，常常高到把當初換新的好處整個吃掉——看似升級了，實際只是多養了一套東西。「推到八成」在很多時候比「沒推」更糟，因為它讓你付了遷移的成本，卻沒拿到「只剩一套」的好處（怎麼在不停機下把舊的逐步絞殺掉、給並存期一個真正的終點，是另一套機制，見〈漸進式遷移與淘汰：strangler 與 expand-contract〉）。

## AI 怎麼動了這幾股力量

把 AI 放進這張圖，它並沒有改寫上面的第一原理，但它**大幅改變了其中一股力量的強弱**——它讓「降低轉換成本」這個槓桿變得極其便宜。

過去，補平那個錯位最費力的一招，是有人手寫遷移腳本、把上千個呼叫點從舊 API 一個個改到新 API，這往往要派人改上好幾週，是轉換成本山谷裡最深的一段。現在這類大規模、機械、重複的改寫，可以大批委派出去生成，那段山谷被填得幾乎看不見了。從「降轉換成本」的角度，這是好事：許多原本因為遷移工作量太大而推不動的採用，現在門檻低到可以動了。

但這裡藏著一個新的、安靜的陷阱，而且它正好打在這一章的要害上。**AI 能讓「採用」這個動作的成本趨近於零，於是團隊可能在還沒真正理解新做法之前，就已經「被遷移完了」。** 程式碼換成了新樣式，CI 也綠了，從每一個外在指標看，採用都成功了——除了一件事：沒有人說得清楚為什麼要這樣寫、它在哪裡會壞、出事時該從哪裡查。你買到的是一種**沒有理解的合規**：表面上全員都用了新做法，實際上沒有任何一個人，在凌晨三點那個東西出問題時，能講清楚它的脾氣。

這把「採用算不算成功」的標準逼出了它真正的內核。回到開頭：採用從來不是一個技術動作，是一個讓人**真正改變**的過程。而「真正改變」的標準裡，一直都默默包含著「有人理解、並且能維護」這一條——只是在改寫很貴的年代，這一條被「能完成改寫」順帶保證了（願意費那麼大力氣改完上千個呼叫點的人，過程中總會理解它）。AI 把改寫變便宜之後，這個順帶的保證斷了：完成改寫不再需要理解。於是「有人理解並能維護」這一條，得被**單獨拎出來、刻意守住**。

守的方式，是把驗收當成一道真正的關卡（review gate），而不是把「AI 改完了、CI 綠了」當成採用成功。驗收問的不是「程式碼換了沒」，是「有沒有一個人，看懂了這個改動、能在它出事時負責」。誰看懂了、誰簽的字——這才是採用真正落地的證據（委派、驗收與信任邊界本身是一整套工作流，這裡只點到它和「採用」的交界）。AI 沒有改變「採用的標準包含理解」這件事，它只是把「完成」和「理解」這兩件原本綁在一起的事，拆開了——而推動者的工作，是別讓它們真的散掉。

## 為什麼是這個形狀

退一步看，這一整套——三股力量、三條路線、團隊規模這個變數、那些安靜的故障模式——全都是從同一個無法迴避的事實裡長出來的：**採用是一個行為改變問題，不是技術問題；而行為改變，被那個「成本在現在、由個人付，好處在未來、歸於系統」的錯位牢牢卡著。**

正因為這個錯位真實存在，「東西夠好就會被採用」才會反覆失靈——好不好從來不是癥結，錯位才是。正因為要補平這個錯位，才演化出三條路線：強制是用權威硬壓過個人當下的誘因，擴散是耐心等誘因自己對齊、卻會卡在信任的鴻溝上，鋪路是直接把採用者要付的成本壓到趨近於零。正因為這三股力量的強弱會隨團隊規模劇烈變化，同一件事在三人和五十人那裡才需要完全不同的打法。正因為採用的成功只能用「行為真的改變了」來度量、而不能用「我下了多少指令」來度量，那些「以為推成了、其實沒有」的失敗才會一片安靜、難以察覺。

所以下次你發出一個客觀上更好的提議、卻看著它停在採用率一半的地方不動時，你會知道那不是因為你的東西不好，也不是因為團隊不講道理。那是因為你在要求一群人，現在就掏出成本，去買一張別人未來才會用到的票——而推動採用這門手藝的全部，就是想盡辦法，讓這筆交易對眼前這個人也划算起來。
