# 兩扇門：可逆與不可逆決策

一個三人小組要選資料庫。爭論卡了兩週，會開了三場，PoC 跑了兩個，最後選了 PostgreSQL。同一個禮拜，這個小組還做了另一個決定：把使用者上傳的圖片直接存進那張 `users` 表的一個 `bytea` 欄位——「先這樣，反正能跑」，三分鐘拍板，沒人覺得需要討論。

一年後回頭看，這兩個決定的重量恰好顛倒了。資料庫選 PostgreSQL 還是 MySQL，事後幾乎沒人再想起——兩者都撐住了，真要換也不是不能換。但那個三分鐘拍板的「圖片塞進 `bytea`」，變成了搬不動的山：表膨脹到幾百 GB、備份要跑一整夜、`SELECT *` 不小心就把幾百 MB 二進位資料拉進記憶體，而此刻已經有十幾個查詢、三個下游報表、兩個快取層默默依賴著那張表的形狀。把圖片挪到物件儲存（見〈物件儲存〉）這件事，要動的不再是一個欄位，是一整圈長出來的依賴。

把心力花在資料庫選型、卻對「圖片存哪」隨手放行——這個小組犯的不是「沒認真想」的錯，而是**把力氣花錯了地方**。他們用「這個決定重不重要」當作要不要慎重的標準，而那是錯的標準。

## 反悔的代價，不是選錯的代價

決策的成本，我們直覺上只算一半：**選錯的後果**。但還有另一半幾乎總是被忽略——**反悔的代價**。把這兩者拆開，會得到一條比「重要 vs 不重要」有用得多的座標軸。

Jeff Bezos 在 Amazon 2015 年致股東信裡，用一個物理意象把這條軸講透了：有些決定是**單向門（one-way door）**，走過去門就在身後鎖上，發現不對也回不來，你只能活在門後的世界；另一些是**雙向門（two-way door）**，推門進去看一眼，不喜歡就退回來，幾乎不留痕跡。信裡他還給了一個更精確的對應——單向門是 Type 1 決策，必須慢、必須謹慎、必須多方會診；雙向門是 Type 2，應該**由判斷力好的個人或小組快速拍板**。這套語彙到 2026 年仍是業界通用的決策詞彙，因為它抓到了一件事：**該不該慢，取決於門是哪一種，不取決於門後的房間有多大。**

於是第一原理浮出來了，而且它和直覺相反：

> 決策的審議成本，應該與**可逆性**成正比，而不是與**重要性**成正比。

回到開頭那個小組。資料庫選型是一扇**雙向門**——它確實重要（每天的開發體驗都受它影響），但只要你沒把自己鎖死在某家的專有 SQL 方言、沒讓程式碼到處是它獨有的 API，要換是一場有計畫的遷移，不是不歸路。而「圖片塞進 `bytea`」是一扇**單向門**——它看起來不重要（就一個欄位嘛），但一旦資料寫進去、一旦下游開始依賴那張表，你就被鎖在門後了。**重要決定了「要不要認真想」，可逆決定了「要不要慢」**，這是兩條獨立的軸，把它們混為一談，就會在雙向門前開三場評審會、在單向門前三分鐘拍板。

組織長大以後最常見的退化，正是把這條軸壓扁成一條：所有決策都套上「重大決策」的厚重流程，連雙向門也要走簽核、開評審、等三個 reviewer。Bezos 那封信裡點名的就是這個病——它的症狀是緩慢、過度規避風險、不敢實驗，而病根是「用 Type 1 的流程去處理 Type 2 的決策」。

## 怎麼判斷一扇門是哪一種

「可逆」聽起來主觀，其實可以問出三個相當具體的問題，把一扇門照出原形。**撤銷它要花多少工程時間？撤銷它會不會破壞已經外溢出去的狀態？撤銷時有沒有資料會永久遺失？**

第一個問題是直接成本：回滾要動幾個系統、改幾行設定、跑多久。第二、三個問題才是真正分出單向門的關鍵——它們問的是**這個決定有沒有產生持久的、別人開始依賴的副作用**。一個改內部函式名的提交，撤銷就是 `git revert`，零外溢、零資料損失，是純粹的雙向門。一個「把 API 回應欄位從 `snake_case` 改成 `camelCase`」的提交，技術上一樣是 `git revert`，但只要它上線過、只要有一個外部 client 已經依賴了新格式，回滾就會反過來打破那個 client——外溢狀態把一扇看似雙向的門，悄悄焊死成了單向。

這就帶出可逆性最反直覺的一點：**門的種類不是天生的、固定的，它會隨時間和外溢而改變。** 同一個部署，在沒人依賴它的產出時是雙向門；一旦它寫出了下游消費的新資料格式、或客戶端快取了它的新行為，那扇門就在你不注意時關上了。任何會產生持久副作用的改動——寫進資料庫的、發給外部的、被別人快取的——可逆性都得**重新評估**，不能用「它剛上線時是雙向門」來免除後來的審議。

也要小心三個冒充者。其一是**把「重要」當成「不可逆」**：選哪個 CI 工具影響每天工作、很重要，但要換隨時能換、極度可逆——它配得上你認真想，但不配讓你慢。其二是**把「不可逆」當成「不能犯錯」**：單向門的目標從來不是零失誤，而是把失誤的後果框在可承受範圍內，這正是後面要講的那些手段的用武之地。其三是**把「沒人反對」當成「可逆」**：共識高和好不好撤回，是兩件毫不相關的事——一個全票通過的不可逆決策，撤起來和一個有爭議的一樣痛。

## 對雙向門：不要堆流程

判定一扇門是雙向的，最划算的事就是**什麼都別多做**——直接做最小版本、上線觀察、不對就退，把決策權下放到最接近問題的人手上。為一個「調個內部快取 TTL」「加一個非公開端點」「改個變數名」這種五分鐘能改回來的事開評審會，不是謹慎，是把審議成本花在不會回本的地方。

判準乾淨利落：**先估撤銷成本，撤銷便宜就別審議。** 雙向門的價值，有一半在於你**敢快**——快地試、快地學、快地在錯的方向上付出最小代價就掉頭。把雙向門也用單向門的流程伺候，等於主動放棄這個價值。

真正需要動腦的，是另一扇門。

## 對單向門：把它改造成雙向門

面對一扇真正的單向門，最弱的回應是「那我們慢慢來、多開幾場會」。多開會只是讓你**更慢地走進同一扇關死的門**——它降低了選錯的機率，卻完全沒有改變「選錯了就回不來」這個結構。

更強的回應是換一個動詞：不是「更謹慎地穿過單向門」，而是**動手把這扇單向門改造成雙向門**。可逆性不是某些決策天生有、某些天生沒有的屬性——它是**可以買的**。每一招改造，本質都是「花一點前置成本，換取一份反悔權」。

最常用的一招是 **feature flag（功能旗標）**。把新行為包在一個開關後面，預設關閉，讓它先「暗度」進生產環境——程式碼上線了，但行為還沒對任何人生效。確認沒問題再逐步打開；一旦出事，把開關撥回去就是瞬間回退，不需要重新部署、不需要 `git revert`、不需要等 CI。它把「上線一個新功能」這個原本要靠回滾整個部署才能撤銷的單向動作，變成了「撥一個開關」的雙向動作。

但 feature flag **本身就是一筆技術債**，而且是會利滾利的那種，這是它最容易被忽略的代價。一個 flag 在程式碼裡製造了一個分岔：`if flag_on { 新路徑 } else { 舊路徑 }`。功能完全推全、舊路徑再也不會被走之後，如果這個 flag 沒被清掉，幾件事會慢慢發生。第一，那條 `else` 裡的死程式碼還在，但沒人再記得它為什麼存在、敢不敢刪——它變成一塊認知地雷。第二，更陰險的是**意外回退**：假設你用 flag 控制 UI 改版，新 UI 推全了卻沒清 flag，哪天有人手滑、或 flag 服務拉取失敗 fallback 到預設值，整個產品會無聲地退回那個你以為早就埋葬了的舊 UI。第三，flag 會互相纏繞——五個沒清的 flag 就是 2⁵ = 32 種理論組合，其中絕大多數從沒被測過，卻都是「理論上可達」的狀態。

所以 flag 的紀律不在「怎麼加」，而在「**怎麼保證它會被清掉**」。成熟團隊把 flag 當成**有到期日的短命產物**來治理：建立時就指定 owner 和預期壽命，把「清掉這個 flag」寫進功能的「完成定義」（definition of done）——就像你不會交一段沒測試的程式碼，也不該交一個沒有退場計畫的 flag。規模再大一點，就把清理自動化：Uber 為此開源了一個叫 **Piranha** 的工具，靜態分析程式碼的 AST，找出已經推全的 flag，自動生成「刪掉這個 flag、刪掉因此變不可達的程式碼、刪掉相關測試」的 diff——他們靠它在 Android／iOS 程式碼庫裡清掉了約**兩千個**過期 flag。需要這種規模的工具本身就是個訊號：flag 債如果放著不管，會堆到只能靠機器來鏟。

值得補一個常被混為一談的區分：不是所有 flag 都該死。Martin Fowler 網站上 Pete Hodgson 那篇 toggle 分類把 toggle 分成幾類，壽命天差地別。**release toggle**（控制新功能上線的）是短命的，用完就該清——它就是上面講的那種「改造單向門用的」可逆性工具。但 **ops toggle**（高負載時降級用的，本質是 circuit breaker 的近親，見〈circuit breaker：三態與半開試探〉）和 **permission toggle**（按方案分級放行功能、實作 freemium 的）是**刻意長命**的，可能存在好幾年。把 release toggle 留成永久、或把 permission toggle 當技術債急著刪，都是看錯了它的種類。判斷一個 flag 該不該清，先問它是哪一類。

feature flag 不是唯一的改造手段，它只是最即時的那個。換一個壓力場景，勝出的招數也不同：

- **灰階發布（canary）**：不直接把單向門推給所有人，而是先放 1% 流量進去看反應，框住「萬一錯了」的爆炸半徑（機制見〈blue-green、canary 與 rolling〉）。它買的是「在小範圍內試錯」的可逆性——錯了只傷到 1%，而且這 1% 通常能在影響擴大前就把問題照出來。但 canary 有個沉默的前提：你得**看得見**那 1% 出沒出事——沒有足夠的可觀測性，canary 只是把同一個災難先發給了一小群人，然後你照樣在指標上看不出來。
- **雙寫過渡期**：要換掉一個資料儲存或改一個 schema，不是「停機、切換、祈禱」，而是讓新舊並存一段時間、同時寫兩邊、隨時能切回舊的，等驗證夠了再退役舊的（這正是零停機遷移的核心，見〈零停機 schema 遷移：expand-contract 與 backfill〉）。它把一個「切過去就回不來」的單向切換，拆成了一串「每一步都能退回前一步」的小雙向門。
- **抽象層隔離**：把那個不可逆的選擇藏在一個介面後面。要依賴某個第三方訊息佇列，就先讓自己的程式碼只認一個自家定義的介面，把那家的專有 API 關在介面之內——將來要換，理想上只動介面後面那一處實作。它買的是「換掉它時，波及面被框在一個地方」的可逆性。但這一招有它自己的陷阱：抽象本身有成本，而且**抽象選錯比沒抽象更難拆**——你猜的那層介面如果框錯了形狀，反而把不該綁的東西綁死了。所以它只在「你相當確定未來真的要換」時才划算，不該對每個依賴都先包一層（過度抽象的代價見〈簡單性與過度工程：YAGNI 的兩面〉）。

## 最被低估的一招：拆小

前面那些手段都是「替整扇單向門加裝反悔機制」。但有一招更釜底抽薪，也最常被跳過：**先檢查這扇門是不是真的只有一扇。**

很多看似巨大、不可逆的決策，其實是**一串小決策被捆在一起**，整捆貼上了「重大架構決策」的標籤。「我們要不要遷到微服務」聽起來是一扇通往無法回頭的單向門，但拆開看，它其實是幾十個獨立的小決定：先抽出哪個服務、用什麼協定通訊、資料怎麼切、先遷哪條流量……這裡面**絕大多數是雙向門**——先抽一個邊緣服務出來、不行就併回去，幾乎沒有外溢。真正不可逆的，可能只剩一兩個：比如「資料庫一旦按服務邊界切開、各自演化 schema，就很難再合回單一交易」。

拆小的威力在於它**重新分配了審議預算**。原本你要對整捆三十個決定都套上單向門的慎重，現在你可以對二十八個雙向門快速試錯、把全部的會診火力集中在那真正不可逆的一兩個上。這比「整捆慢慢來」高效得多——後者是把九成的力氣浪費在本來就能反悔的事情上。

對應地，這也是 over-engineering 的一個典型訊號：**對偽單向門整捆慎重，因為沒人花力氣去拆。** 把一個其實能拆成多數雙向門的決策當成鐵板一塊地慎重對待，看起來很負責，實則是把流程成本花在了二十八扇本來可以快走的門上。

## 哪些是真的單向門

講了這麼多「把單向門改成雙向門」，得誠實承認：**有些門就是真的單向，而且改造的成本高到不值得。** 對這些，慢一點、多會診一點，是對的。

這類門有個共同的形狀——它們產生的副作用**外溢到了你控制不到的地方**，而那正是「撤銷成本」三問裡最致命的兩問（破壞外溢狀態、永久遺失資料）：

- **資料格式與儲存引擎的選擇**：資料一旦以某種格式寫進去，遷移就是一場大工程——你不只要改讀寫的程式碼，還要把已經躺在那裡的存量資料一筆筆搬過去、轉換、校驗，而搬遷期間系統還在持續寫入。開頭那個 `bytea` 就是這種門。
- **對外公開的 API 契約**：你一旦發布了一個 API 的形狀，依賴它的就不只是你自己了——那些 client 不在你的程式碼庫裡、不歸你部署、甚至你不知道有誰在用。改這個契約，等於替一群你看不見的人決定他們的程式碼要不要壞掉（API 的版本化正是為了在這扇門上裝一點可逆性，見〈REST 設計、版本化與 OpenAPI 契約〉）。
- **跨團隊共用的核心抽象**：一旦三個團隊的程式碼都長在你定義的那個介面上，改它就要協調三個團隊的節奏。

這些是真單向門，慢是合理的。但要注意，「真單向門」和「不可改造」不是一回事——上面提到的版本化、雙寫過渡，正是在這些最硬的門上**仍然能擠出一點可逆性**的手段。只是對它們，可逆性買得比較貴，而且常常買不到全額。

## 為何是這個形狀

退一步看，整套關於「兩扇門」的思考，是從一個樸素的觀察裡長出來的：**我們本能地用「這件事重不重要」來決定要花多少力氣去想，但決定撤銷成本的，是另一回事。**

正因為撤銷成本和重要性是兩條獨立的軸，所以才需要刻意把它們分開看——否則就會在重要但可逆的事上空轉、在瑣碎但不可逆的事上失守。正因為可逆性不是天生的、是可以花前置成本買的，所以面對單向門，聰明的回應不是「更慢地穿過」，而是「動手把它改造成雙向門」——feature flag、canary、雙寫、抽象隔離，每一招都是在賣一份反悔權。正因為很多單向門其實是小門捆成的一束，所以拆小幾乎總是值得先試。而正因為有些門真的單向、外溢到你控制不到的地方，所以對那幾扇——資料格式、對外契約、共用抽象——會診和緩慢才有了它真正的位置。

可逆性的整個價值，最後收束成一句話：**它讓你有資格快。** 一個能隨時反悔的決定，不需要在事前想到所有可能；它把「想清楚」的負擔，從昂貴的事前審議，挪到了便宜的事後觀察。把這份自由用好，意味著你大部分時間都在快速地穿過雙向門、從錯誤裡便宜地學習，只在那少數真正關死的門前，才停下來、慢下來、把人叫齊。而那少數門之所以值得你慢，恰恰是因為你在其餘所有門上都快得起來。
