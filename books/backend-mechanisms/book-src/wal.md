# WAL：先寫日誌，再改資料

設想一個再普通不過的更新：一筆轉帳要把 A 帳戶減 100、B 帳戶加 100。這兩個值落在磁碟上的兩個不同位置——可能是兩張表、兩個資料頁，甚至兩個不同的磁碟區塊。資料庫照辦：改了 A 的頁、正準備改 B 的頁，這一瞬間，機房跳電。

重啟之後，磁碟上是什麼狀態？A 已經被扣了 100，B 卻沒收到。錢憑空蒸發了。更糟的版本是：連 A 那一頁都只寫了一半——一個 8 KiB 的頁，作業系統把它拆成兩次 I/O 寫，第一半是新資料、第二半還是舊資料，於是磁碟上躺著一個前後矛盾、語意上根本不存在的「半頁」。沒有任何純靠「就地改資料頁」的辦法能在斷電後把這個爛攤子收拾乾淨，因為災難恰恰發生在「改了一部分、還沒改完」的中間態，而磁碟上沒有留下任何線索告訴你「我本來想做什麼、做到哪了」。

這就是所有持久化儲存引擎都得先回答的問題：**在一個隨時可能斷電、可能被 kill -9、可能寫到一半就死掉的世界裡，怎麼保證「提交了的交易，崩潰後一定還在；沒提交完的交易，崩潰後一定像沒發生過」？** 答案，幾乎清一色是同一招——預寫日誌（Write-Ahead Logging，WAL）。PostgreSQL 有 WAL，InnoDB 有 redo log，RocksDB 有 WAL，SQLite 有 WAL 模式。名字各異，骨子裡是同一條鐵律，同一套力學。這一章要把這條鐵律拆到你能在腦中重放：它為什麼非這樣不可、它買到什麼、又在哪些邊角會悄悄背叛你。

## 為什麼「直接小心地改資料」救不了你

先把天真做法逼到牆角。一個直覺的反應是：那我改資料時小心一點不就好了？比如「先改 A、確認落盤，再改 B」。問題是，無論你把這串就地寫入排得多仔細，**斷電可以發生在任意兩個指令之間**。你能保證的頂多是「A 先於 B」，但你永遠無法保證「A 和 B 要嘛都成、要嘛都不成」——因為原子性需要的是「一個不可分割的瞬間」，而你手上只有一連串可被打斷的離散寫入。

更根本的麻煩藏在硬體粒度裡。資料庫眼中的最小更新單位可能只是改一個整數欄位的 8 個位元組，但磁碟（以及檔案系統、作業系統 page cache）能保證原子寫入的單位是一整個頁。PostgreSQL 的頁是 8 KiB。當引擎要求「把這 8 KiB 寫下去」，作業系統完全有權把它拆成多次底層 I/O；只要在拆開的中途斷電，磁碟上就是一個**撕裂頁（torn page）**——一半新、一半舊，混在一起。這種頁裡的資料是憑空捏造出來的，行級的「我把欄位 x 從 3 改成 5」那種紀錄根本無從修復它，因為連這一頁的物理結構都壞了。

於是真正的難點不是「怎麼把資料寫對」，而是兩件更刁鑽的事：**怎麼讓一組散落各處的修改變成一個原子的「全有或全無」**，以及**怎麼在崩潰後知道「我本來打算做什麼、做到哪、哪些該補完、哪些該抹掉」**。要回答後者，你需要在動手改資料之前，先在某個地方留下一份**意圖的紀錄**。這份紀錄就是日誌，而「先寫它、再改資料」這個順序，就是整個機制的名字。

## 那條鐵律：日誌先落穩，資料頁才准動

WAL 的核心可以濃縮成一句話：**在把任何資料頁的變更寫到它最終的位置之前，先把「這個變更是什麼」以順序追加的方式寫進一個日誌，並讓這條日誌真正落穩（durable）。** 「預寫」這兩個字就是字面意思——日誌寫在資料頁之前，never the other way around。

為什麼這個順序能解決原子性？關鍵的轉換是：**它把「同時原子地改好幾個地方」這個做不到的事，換成了「原子地往一個地方追加一條紀錄」這個做得到的事。** 提交一筆交易，不再等於「A、B 兩個資料頁都安全落盤」，而是縮小成一個單點動作——把一條 `COMMIT` 紀錄追加到日誌尾端並 fsync 落穩。這一刻之前，交易不算數；這一刻之後，交易板上釘釘。真正的資料頁（A 和 B）可以稍後再從容刷回它們的最終位置，那不影響「這筆交易已經提交」這個事實，因為事實已經被日誌記下了。

崩潰之後，恢復流程靠這份日誌把世界扳回一致。已經有 `COMMIT` 紀錄、但變更還沒刷進資料頁的——**redo（重做）**，照日誌補上。寫了一半資料頁、但交易其實沒提交（日誌裡沒有 `COMMIT`）的——**undo（回滾）**，照日誌把它撤回。一來一回，資料庫被帶回到「崩潰前最後一個一致狀態」：所有提交過的都在，所有沒提交完的都像沒發生。這就是 WAL 撐起 ACID 裡 **D（durability，持久性）與 A（atomicity，原子性）** 的物理基礎。

這裡還有一個常被忽略、卻是整套機制效能命脈的第一原理：**順序寫遠快於隨機寫。** 一筆交易可能東改一頁、西改一頁，這些頁散落在磁碟各處，逐一刷回是一堆隨機 I/O——對機械硬碟是磁頭來回亂跳的災難，對 SSD 也意味著更多的寫放大與更差的合併。WAL 把這「一堆隨機寫」換成「一條順序追加」：所有交易的變更，不管它們最終要落到哪，都先排隊接到同一條日誌的尾巴上，這是磁碟最擅長的存取模式。所以 WAL 不只是為了崩潰安全——它順手把寫入路徑變快了。慢的從來不是「寫日誌」這件事，慢的是「每一筆都強制等磁碟確認落盤」那一下，而那是另一個可以單獨調的旋鈕，我們等下會碰到。

## 在腦中跑一遍：一筆交易的生與死

把抽象的鐵律換成可以重放的動作。一筆交易從進來到提交，日誌與資料頁的舞步大致是這樣：

```
1. txn 開始，要改 page A 和 page B
2. 改 page A：先寫一條 redo log record 到 WAL buffer（"page A: x 3->5"）
              再到記憶體裡（buffer pool / page cache）就地改 A 的副本
3. 改 page B：同樣先寫 log record，再改記憶體裡的 B
4. COMMIT：把一條 COMMIT record 追加到 WAL，並 fsync——
           等磁碟確認這段日誌真的落穩了，才回覆 client「成功」
5. （此刻磁碟上：WAL 有完整紀錄含 COMMIT；資料頁 A/B 可能還只在記憶體）
6. checkpoint（稍後、背景）：把記憶體裡改過的髒頁（dirty pages）
           批次刷回它們在磁碟上的最終位置，並在 WAL 標一個 checkpoint 點
```

注意第 4 步和第 6 步之間那個刻意的時間差。提交的那一刻，被強制落盤的**只有日誌**，資料頁 A、B 很可能還只待在記憶體的緩衝池裡，是「髒的」。這完全沒問題——日誌已經保證了它們的命運。把髒頁真正寫回最終位置，是 checkpoint 的工作，它在背景慢慢做、批次做、合併著做，把隨機寫的痛攤平。

恢復時，引擎從最後一個 checkpoint 點開始往後讀日誌（checkpoint 之前的都已確定落盤、不必管）。對每一條 redo record，它檢查對應資料頁是否已經反映了這個變更（靠頁裡記的版本號），沒反映的就補做。讀到沒有對應 `COMMIT` 的交易，就用 undo 資訊回滾。這套「掃一遍 + 重做 + 回滾」的恢復骨架，正是 1992 年 IBM 的 C. Mohan 等人提出的 **ARIES** 演算法的核心，幾乎所有主流資料庫的崩潰恢復都是它或它的直系後裔。ARIES 有個聽起來有點反直覺的設計準則叫 **repeating history（重演歷史）**：redo 階段會無差別地把日誌裡所有變更重做一遍——**包括那些最終會被回滾的交易的變更**——先把資料庫精確地還原到「崩潰前一瞬間」的模樣，然後才在 undo 階段把未提交的交易撤掉。為什麼要先重做注定要被撤的東西？因為這樣 redo 階段不必判斷誰會被回滾、只要無腦照日誌前進，邏輯簡單到不會出錯；複雜的「誰該活誰該死」全部推遲到 undo，且 undo 本身也寫日誌（補償紀錄，compensation log record，CLR），於是連「恢復做到一半又崩潰」都能再恢復。這種對「恢復過程自己也可能崩潰」的偏執，正是 ARIES 耐用的原因。

要讓「這條 redo record 是否已經反映在這頁上」這個判斷能做，日誌與資料頁之間需要一把共同的尺。這把尺就是 **LSN（Log Sequence Number，日誌序號）**：一個只增不減的序號，標記每一條日誌紀錄在日誌流裡的位置。每一個資料頁裡也記著「我最後套用到第幾號 LSN」。恢復時的判斷就變得乾淨俐落——日誌讀到 LSN=500 的紀錄，發現對應頁上記的是 LSN=480，表示這頁還沒套用過這筆變更，redo；若頁上已是 LSN=520，表示這變更早已落盤，跳過。InnoDB 的 redo log 正是這樣以 LSN 串起日誌與資料頁、並用 LSN 標記 checkpoint 推進到哪。

## 記什麼進日誌：物理、邏輯，與折衷

日誌裡到底寫什麼，是個真實的設計抉擇，三條路各有取捨。

**物理日誌（physical logging）** 記「位元組層級的結果」——直接記下改完之後那一頁（或那幾個 byte）的影像。好處是 redo 極其簡單：把影像蓋上去就對了，不需要任何上下文。壞處是日誌很肥，改一個整數可能要記一整頁。

**邏輯日誌（logical logging）** 記「做了什麼操作」——例如「在 orders 表插入這一列」。日誌很瘦，但 redo 變得危險：要重放「插入一列」，引擎得知道當時的索引結構、頁分裂狀態……重放的結果必須和當初一模一樣，任何不確定性都會讓兩邊分岔。

實務上主流引擎走的是折衷的 **生理日誌（physiological logging）**：**頁與頁之間用物理定位（明確指出是哪一頁），頁內部用邏輯描述（在這頁的這個位置做這個改動）。** InnoDB 的 redo log 正是這一路。這樣既不必把整頁塞進日誌、又避開了純邏輯重放對全域結構的依賴——redo 時只需把某一頁讀進來、套用「頁內這個小操作」，自足、確定、可重放。

純物理日誌還有個躲不掉的場景，回到開頭那個撕裂頁。生理日誌假設「被 redo 的那一頁本身是完整的」，可一旦發生 torn page，那一頁的物理結構已經壞了，頁內邏輯操作無從套起。怎麼辦？PostgreSQL 的解法是 `full_page_writes`（預設開）：**每個資料頁在一次 checkpoint 之後第一次被改動時，把整頁的完整影像（不是增量）寫進 WAL。** 之後對同一頁的改動才回到便宜的生理日誌。恢復時若懷疑某頁被撕裂，直接拿日誌裡那份完整頁影像覆蓋上去、完全不去看磁碟上那個可能半新半舊的爛頁。代價是 WAL 變肥——每次 checkpoint 後第一批寫入都要扛整頁影像，這也是為什麼把 checkpoint 間隔調得太密反而會放大 WAL 流量。這是一個典型的 WAL 內部取捨：用日誌空間，買對撕裂頁的免疫。

## 那個會吃掉你資料的字：fsync

到這裡，WAL 的邏輯看似滴水不漏。但整套機制的安全性，**完全押在一個假設上**：當引擎說「日誌落穩了、可以回覆 client 成功」時，那段日誌**真的**已經躺在斷電也不會丟的穩定儲存上。這個假設，比多數人以為的脆弱得多。

問題出在 `write()` 和 `fsync()` 是兩回事。`write()` 只是把資料交給作業系統的 page cache——快，但那還在記憶體裡，斷電就沒。要真正落盤，得呼叫 `fsync()`，它**應該**阻塞到「這個檔案所有未落盤的髒頁都確實刷進持久儲存」才回。WAL 的提交路徑因此必然包含一次 fsync：寫完 `COMMIT` 紀錄、fsync、確認回來，才敢告訴 client 成功。少了這次 fsync，你的「持久性」是假的——名義上提交了，實際只在一塊斷電即逝的記憶體裡。

第一個陷阱是有人為了快，主動把這次 fsync 拿掉或放寬。這在某些場景是合理的取捨（下一節談），但被無意中帶上 production 就是事故：開發環境圖快關了 durable fsync，忘了改回來，平時毫無徵兆，直到某次斷電才發現「提交成功」的交易成批消失。

第二個陷阱更陰險：**fsync 會說謊。** 整條路徑上的多個環節都可能在 fsync 上撒謊——磁碟自己的 write cache 收到資料就回報「好了」，其實還沒寫進碟片；某些虛擬化層、某些雲端區塊儲存、某些消費級 SSD 的韌體，為了好看的效能數字，會在資料還在易失快取裡時就讓 fsync 提早回傳。對這些裝置，WAL 的整個證明就地失效：引擎以為日誌落穩了、回覆 client 成功了，斷電卻把那段日誌連同它保證的交易一起帶走。這類「資料神祕消失」事故的根因，往往不在資料庫程式碼，而在它腳下那塊會撒謊的儲存。

第三個陷阱是 2018 年震動整個資料庫圈的 **fsyncgate**。PostgreSQL（以及幾乎所有用緩衝 I/O 的資料庫）原本假設「fsync 回傳成功，代表上次成功 fsync 以來的所有寫入都安全落盤了」。但 Linux 核心的實際行為是：當一個緩衝寫入因底層硬體錯誤而失敗，某些檔案系統會**丟棄**那些髒頁、並把它們標記成 clean——於是下一次 fsync 看到「沒有髒頁了」，回報成功，而那筆失敗的寫入已經人間蒸發、永遠不會被重試。更要命的是 fsync 回報的那一個錯誤可能只被某個行程消費掉，其他行程再也看不到。這個 bug 在 PostgreSQL 裡潛伏了二十年。修法簡單到近乎認輸：**fsync 一旦回報失敗，直接 PANIC（讓整個資料庫崩潰）**，逼一次完整的崩潰恢復、從 WAL 重來，而不是樂觀地假設「再試一次就好」。PostgreSQL 12 把這定為預設、並回頭修補到更舊的版本。這件事的教訓不是「PostgreSQL 不可靠」，而是**持久性是一條跨越資料庫、檔案系統、作業系統、磁碟韌體的端到端鏈條，任何一環說謊，上層所有漂亮的證明都歸零。**

## fsync 的代價，與 group commit 這記漂亮的攤平

既然每次提交都要 fsync，而 fsync 本質上要等一次物理落盤，它就成了寫入吞吐的天花板。算一筆帳就看得很清楚。

一顆需要真正等待落盤的 SSD，單次 durable fsync 大約要 1 毫秒（這數字隨硬體變動很大，但量級夠用）。如果每一筆交易提交都各自獨立 fsync 一次，那麼單一執行緒的提交吞吐上限就是：

```
1 commit / 1 ms = 1,000 commits/s
```

一千筆每秒。對一個高並發的 OLTP 系統，這個天花板低到荒謬——而且瓶頸根本不是 CPU、不是記憶體頻寬，純粹是那一次次「等磁碟」的 fsync。

解法是 **group commit（組提交）**，一個簡單到優雅的觀察：fsync 是「把這個檔案當下所有髒資料一次刷穩」，它的成本和「這次刷了多少資料」幾乎無關——刷 1 筆交易的日誌要 1 ms，刷 100 筆並發交易**攢在一起**的日誌，還是大約 1 ms。既然如此，為什麼讓它們排隊一個一個 fsync？讓在同一個短窗口（約 1 ms）內到達的所有提交，把它們的 `COMMIT` 紀錄一起寫進 WAL buffer，然後**共用同一次 fsync**。那一次 fsync 回來，這 100 筆同時被宣告持久、同時回覆各自的 client 成功：

```
無 group commit：  txn1 fsync(1ms) → txn2 fsync(1ms) → ... 序列化
                   100 筆 ≈ 100 ms，吞吐 ≈ 1,000/s

有 group commit：  txn1..txn100 攢進同一個 WAL buffer
                   → 一次 fsync(1ms) → 100 筆全部 durable
                   100 筆 ≈ 1 ms，吞吐 ≈ 100,000/s
```

同一顆磁碟、同樣的 1 ms fsync，有效提交吞吐從一千躍到十萬。代價只有一個：單筆交易的延遲略增——它得多等幾百微秒，讓更多同伴擠進這一批。這是個幾乎穩賺的交換：用「每筆多等一點點延遲」買「整體吞吐放大兩個數量級」。這就是為什麼高並發資料庫離不開 group commit：它讓那道「fsync 是物理、躲不掉」的牆，被一群交易合力攤平成一道矮坎。MySQL 在開了 binlog 時還把這套玩得更精緻——把 redo log 與 binlog 的多次 sync 也跨一組交易合併，進一步省下每筆的同步次數。

## 旋鈕：把多少持久性換成多少吞吐

WAL 最有意思的地方，是它把「持久性」做成了一個**可調的連續旋鈕**，而不是一個 on/off 開關。這個旋鈕的兩端，在主流引擎裡都有對應的參數，值得並排看清楚它們各自把你放在風險光譜的哪一格。

PostgreSQL 的旋鈕是 `synchronous_commit`，預設 `on`。MySQL/InnoDB 的旋鈕是 `innodb_flush_log_at_trx_commit`，預設 `1`。兩者解的是同一個問題——「提交時，日誌要落穩到什麼程度才回覆 client 成功」——但刻度切法略有不同：

| 設定 | 提交時的行為 | 崩潰會丟什麼 |
|---|---|---|
| InnoDB `=1`（預設） | 每次提交寫 redo log 並 fsync | 零丟失（磁碟誠實的前提下） |
| InnoDB `=2` | 每次提交寫到 OS page cache，但不 fsync；每秒才 fsync 一次 | 行程崩潰不丟（OS 還在）；**整台機器斷電**會丟最後約 1 秒 |
| InnoDB `=0` | 每秒才把 log buffer 寫出並 fsync；提交當下什麼都不做 | 連 mysqld 崩潰都可能丟最後約 1 秒 |
| PG `synchronous_commit=on`（預設） | 提交等本地 WAL flush 到持久儲存 | 零丟失（磁碟誠實的前提下） |
| PG `synchronous_commit=off` | 提交不等 WAL flush，稍後才落盤 | 崩潰丟最後一小段已「提交」的交易，**但資料庫仍一致**（不會壞，只是丟尾巴） |

最關鍵、也最容易被搞混的一點藏在 PostgreSQL 的 `off` 模式裡：**關掉 synchronous_commit 並不會破壞資料庫的一致性，它只會讓你丟掉崩潰前最後一小段交易。** 為什麼？因為 WAL 紀錄的相對順序、以及「沒有 COMMIT 就回滾」的恢復邏輯完全不變——丟掉的是「日誌尾巴上那幾筆還沒來得及 flush 的紀錄」，而少了尾巴的日誌重放出來的，依然是一個前後一致的資料庫，只是它停在稍早一點的時間點。這跟「關掉 full_page_writes」是兩種完全不同性質的危險：後者一旦遇上撕裂頁會弄出一個**損壞**的資料庫，前者最壞只是**丟掉幾筆**。把這兩種「放寬」混為一談，是調這些旋鈕時最危險的誤解。

那什麼時候該往「快」的那端調？判準是**這份資料丟掉最後一秒，代價有多大**。金流、訂單、不可重算的交易——留在預設的零丟失端，那一次次 fsync 的延遲是你買「斷電不丟錢」付的合理保費。反過來，如果這份資料本來就可重算、或可從上游重放（快取的物化結果、衍生視圖、能從訊息系統重放出來的計數），那麼為它付 per-commit fsync 的代價，是花了持久性的錢卻買了一個你根本不需要的保證——這時把旋鈕往 InnoDB `=2` 或 PG `off` 調，換來的吞吐是實打實的。重點是：**這是一個明確的、該寫進文件、該被團隊知道的取捨，而不是某人某天為了讓 benchmark 好看隨手改掉的設定。** 一個放寬了 fsync 的 production 資料庫，在那一秒的容忍度被寫清楚之前，是一顆沉默的定時炸彈。

## 不只是防崩潰：那條日誌是系統的真相

WAL 寫到這裡，你會發現它意外地長出了第二條命。它本來只是為了崩潰恢復而存在的「私房帳」，卻碰巧具備一個極有價值的性質：**它是這個資料庫上所有變更的、嚴格有序的、完整的流水帳。** 每一筆改動，按它真正發生的順序，一條不漏地記在那裡。一旦你意識到這點，WAL 就從「崩潰時才讀的東西」變成了「平時就能訂閱的事件流」。

複製就是第一個受益者。要讓一個副本跟上主庫，最乾淨的辦法不是去比對兩邊的資料、找差異，而是**把主庫的 WAL 串流給副本，讓它照樣重放一遍**——重放同一串有序日誌，副本自然收斂到和主庫一致的狀態。PostgreSQL 的物理/邏輯複製、MySQL 的 binlog 複製，本質都是「把日誌寄給下游、下游重放」。同一條日誌，既是崩潰恢復的依據，也是複製的傳輸協定。

再往外一步是 CDC（Change Data Capture）：同樣訂閱這條日誌，把資料庫的每一筆變更轉成事件，餵給下游的搜尋索引、資料倉、快取失效管線。WAL 之所以是 CDC 最理想的來源，正因為它有那兩個別處難得的保證——**完整（沒有任何提交了的變更會漏掉）** 與 **有序（事件順序就是交易真正發生的順序）**。這也是為什麼「先寫日誌」這個原本純粹防崩潰的設計，會在資料整合的世界裡反覆出現：當你需要一份「絕對可靠、嚴格有序的變更紀錄」，你會發現你需要的東西，儲存引擎為了不丟資料，早就替你記好了。

當然，WAL 終究是 WAL，它有它管不到的邊界，而這條邊界值得記死：**WAL 保證的是「把你下的指令，原子且持久地做下去」，它不保證你下的指令是對的。** 一筆 `DELETE FROM users` 漏了 `WHERE`,WAL 會無比忠實、無比持久地把這場災難記下來、執行下去、複製出去——它防的是崩潰，不是 bug。把資料還原回那個 `DELETE` 之前，不是 WAL 的工作，那是備份與 point-in-time recovery 的領域。WAL 給你的是「機器層級的不背叛」：斷了電、被 kill、磁碟抽掉，提交過的東西還在；它從不承諾保護你不被自己的邏輯所傷。

## 為什麼是這個形狀

退一步看，WAL 的每一個部件，都是被同一個無法迴避的物理事實逼出來的：**你沒有辦法原子地、同時地改好磁碟上散落各處的多個地方，而災難偏偏最愛在「改了一半」的瞬間降臨。**

正因為「同時改多處」做不到，所以把它換成「原子地追加一條紀錄」——這給了你 WAL 的核心。正因為追加比四處亂改快得多，所以順序日誌順手把寫入變快了，慢的只剩那一次 fsync。正因為那一次 fsync 是物理、躲不掉，所以演化出 group commit 把它攤平、演化出一排旋鈕讓你按「丟一秒值多少」去買賣持久性。正因為頁可能被撕裂，所以有了 full-page image 這道額外的保險。正因為這條日誌天生完整且有序，所以它又被複製和 CDC 借去當真相之源。而所有這一切的地基——那個讓整座大廈成立或崩塌的單點——是 fsync 是否誠實：它若說了真話，WAL 給你斷電不丟的鐵保證；它若說謊，上面這套精巧的力學會悄無聲息地、在某次斷電時，把你以為已經安全的交易一起帶走。

下次你看到一個資料庫在提交路徑上卡在 fsync、看到它啟動時花幾秒「replaying WAL」、看到 DBA 為了 `synchronous_commit` 的設定爭論——你會知道那都不是偶然的工程細節，而是「先寫日誌、再改資料」這條鐵律，在不同的壓力下，各自長出來的、唯一站得住的形狀。
