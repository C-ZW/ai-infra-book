# HTTP/1.1、HTTP/2、HTTP/3：keep-alive、多工與 QUIC

打開瀏覽器的開發者工具，看一個現代網頁載入時的瀑布圖（waterfall）。一個首頁可能要抓三、四十個資源：HTML、幾支 CSS、十幾張圖、一堆字型、追蹤腳本。每一條都是一個 HTTP 請求。問題來了：這幾十個請求，到底是怎麼擠進有限的幾條連線、又不讓彼此卡死的？

這個問題聽起來像是前端的事，但它其實是一個非常純粹的傳輸層難題，而且這個難題的三次不同解法，恰好就是 HTTP/1.1、HTTP/2、HTTP/3 的全部劇情。三個世代不是「越新越快」這麼簡單——它們是同一個問題被推到三個不同的層，每往下推一層，就解掉前一層解不掉的某種卡頓，同時又欠下一筆新的債。要真正讀懂它們，得先看清楚這個問題長什麼樣，以及它為什麼這麼頑固。

## 一個請求要付的隱形成本

先把鏡頭拉到最底層：發一個 HTTP 請求，在連上線之前要先付什麼？

一條 HTTPS 連線的成立，要先做完 TCP 三向交握（一個 RTT），再做完 TLS 交握。以現役主力 TLS 1.3（RFC 8446）為例，握手壓到了一個 RTT。兩者疊起來，**一條全新的加密連線，在你能送出第一個位元組之前，已經燒掉了大約兩個 RTT**。如果你的使用者在跨國連線、RTT 是 150 毫秒，那就是還沒開始傳資料，先付掉 300 毫秒。

這個成本本身不是 HTTP 的錯——它是 TCP 與 TLS 的分層帳單。但 HTTP 早期的設計把它放大到荒謬的地步：最原始的 HTTP/1.0，每抓一個資源就開一條新連線、抓完就關。一個頁面四十個資源，就是四十次「兩個 RTT 的開場費」串在一起。連線的建立成本，遠遠蓋過了真正傳輸那幾 KB 資料的成本。

於是第一個、也是最直觀的優化登場：**別關連線，留著重用。**

## HTTP/1.1：keep-alive 省下交握，但留了一條單行道

HTTP/1.1（其訊息語法與連線管理的當前規範是 2022 年的 RFC 9112，接續更早的 RFC 7230、再更早的 RFC 2616）把 **keep-alive**（persistent connection，持久連線）變成預設。一條 TCP 連線建立後不立刻關閉，可以連續送好幾個請求，攤掉那筆交握開場費。這是巨大的進步——四十個資源不必再付四十次開場費。

但 keep-alive 留下了一個關鍵的限制，這個限制是理解後面兩個世代的起點：**同一條連線上，請求是嚴格序列化的。** 你送出請求 A，必須等 A 的回應**完整回來**，才能在這條連線上送請求 B。連線是被重用了，但它是一條**單行道**——一次只能有一個請求在飛。

```
HTTP/1.1 一條連線上的請求（嚴格序列化）：

  request A  ───────────►
                          (server 處理 A)
             ◄─────────── response A
  request B  ───────────►        ← B 必須等 A 的回應完整回來才能發
                          (server 處理 B)
             ◄─────────── response B
```

規格上其實有一個叫 **pipelining** 的東西，允許你不等 A 回來就先把 B、C 都送出去。但它幾乎沒人用，原因很實際：server **必須照請求的順序回應**。就算 B 早就處理好了，只要 A 還沒生出回應，B 的回應就得在後面排隊等。一個慢的 A 會把後面所有人都堵住——這就是**隊頭阻塞（head-of-line blocking）**，而且它發生在**應用層**。pipelining 把問題從「等回應才發下一個」搬到「回應被迫排隊」，沒解決根本，還引進了一堆 proxy 相容性的雷，所以實務上幾乎所有實作都把它關掉了。

那 HTTP/1.1 的世界裡，瀏覽器是怎麼讓四十個資源看起來像「同時」載入的？答案是**蠻力**：對同一個 host，**開好幾條連線並行**。主流瀏覽器的慣例是每個 host 大約六條。六條單行道並排，就能讓六個請求同時在飛。但這也是它的天花板——四十個資源、六條連線，得分成大約七批先後跑完，而且每多開一條連線，就多付一次兩個 RTT 的開場費、多吃一份 server 的連線資源。

把這條死路看清楚，就會發現問題的真正形狀：**我們需要的是「一條連線上同時跑多個請求、彼此不互相等待」。** keep-alive 給了「一條連線跑多個請求」，但沒給「不互相等待」。補上後半句，就是 HTTP/2 的全部使命。

## HTTP/2：把一條連線切成許多不互相等待的 stream

HTTP/2（RFC 9113，2022 年發布，接替 2015 年的 RFC 7540）的核心發明是**多工（multiplexing）**。它在一條 TCP 連線之上，憑空造出許多條**獨立編號的 stream（串流）**。每個請求/回應對應一條 stream，而真正在連線上傳輸的，是被切碎、打上 stream 編號的 **frame（訊框）**。

關鍵在這裡：不同 stream 的 frame 可以**交錯**傳輸。stream 1 的一段、stream 3 的一段、stream 1 的另一段、stream 5 的一段……它們在同一條 TCP 連線上像拉鍊一樣交織著流動，接收端再按 stream 編號把它們重新組裝回各自的訊息。

```
HTTP/2 一條 TCP 連線上 frame 交錯（多工）：

  TCP byte stream: [S1][S3][S1][S5][S3][S1][S5] ...
                     │   │   │   │   │   │   │
   reassemble ──►  stream 1: ████  ████  █
                   stream 3: ██    ██
                   stream 5: ██          ██
```

這一下就把 HTTP/1.1 的應用層隊頭阻塞給消了。stream 3 的回應慢，不會擋住 stream 1 和 stream 5——它們是邏輯上獨立的，各自的 frame 自顧自地流。瀏覽器不再需要對一個 host 開六條連線，**一條連線就夠**，幾十個請求一次全部發出去同時在飛。RFC 9113 建議的併發 stream 數下限是 100，所以一條連線同時跑上百個請求是稀鬆平常的事。

附帶還解了一個 HTTP/1.1 的浪費：header 重複。同一個網站連抓四十個資源，每個請求都帶著幾乎一樣的一大坨 header（`User-Agent`、`Cookie`、`Accept`…），明文重送四十遍。HTTP/2 用 **HPACK** 壓縮 header——雙方各維護一張動態表（dynamic table），重複出現的 header 欄位後面只要送一個索引號就好。對 header 肥大的 API 流量，這省下的頻寬相當可觀。

HTTP/2 還引進過一個叫 **server push** 的功能：server 可以在 client 還沒要求之前，主動把它「猜」client 接下來會需要的資源推過去。聽起來很美，實際上很糟——server 幾乎無法準確猜中 client 的快取裡已經有什麼，於是經常把使用者早就有的東西重推一遍，白白浪費頻寬。它複雜、效益差，使用率低到統計上幾乎看不見，最終被主流瀏覽器陸續放棄（Chrome 自 106 版起預設關閉、Firefox 在 132 版即 2024-10 移除），HTTP/3 也不再沿用。需要「提示 client 預先載入資源」的場景，現在改用 `103 Early Hints` 這種更可控的機制。記住這段：HTTP/2 不是只有勝利，server push 是它賭錯、又退回來的一步。

到這裡，HTTP/2 看起來圓滿了：一條連線、多工、不互相等待。但它有一個藏得很深的破綻，要在特定的網路條件下才會現形——而那個破綻，恰恰是它「只用一條 TCP 連線」這個漂亮設計的直接後果。

## HTTP/2 沒解掉的：隊頭阻塞只是搬了一層

把問題講到底：HTTP/2 消除的是**應用層**的隊頭阻塞。但它的所有 stream，仍然共用**同一條 TCP 連線**。而 TCP，根本不知道 stream 是什麼東西。

從 TCP 的視角，它收到的不是「stream 1 的 frame、stream 3 的 frame」，而是**一條連續、有序、不可分割的位元組流**。TCP 對上層的承諾是「我交付給你的位元組，順序和對端送出的一模一樣，中間不會有洞」。為了兌現這個承諾，**只要中間有一個封包遺失，TCP 就會把它後面所有已經收到的封包全部扣在緩衝區裡，不交給上層**——一直等到那個丟失的封包被重傳、補上那個洞，才把後面累積的一整串按序放行。

現在把這兩件事疊起來看那個破綻：假設一條 HTTP/2 連線上正在交錯傳 stream 1、3、5 的 frame。某個封包丟了，而那個封包**剛好只裝著 stream 3 的資料**。從應用的角度，stream 1 和 stream 5 的資料明明都完整到齊了，理應可以馬上交付。但 TCP 不知道這件事——它只看到「位元組流中間有個洞」，於是把那個洞**後面**的所有位元組（包括已經到齊的 stream 1、stream 5 的部分）全部扣住，等 stream 3 那個封包重傳回來填洞。

```
TCP 不認得 stream，遺失一個封包就扣住整條連線：

  在途封包:  [S1][S3][S5][S1] [ X ] [S5][S1][S3] ...
                              ↑ 這個封包裝著 S3 的資料，丟了
                              └─ TCP 把它後面全部扣住等重傳
                                 即使 S1、S5 的後續資料已經到了，也不交付
```

於是一個 stream 的丟包，卡住了**這條連線上所有 stream**。隊頭阻塞沒有消失，它只是從**應用層搬到了傳輸層**。在資料中心內部、丟包率趨近於零的網路裡，這個破綻幾乎不會發作——封包不丟，就沒有洞要等。但在行動網路、跨國長途、丟包率偏高的環境裡，它會狠狠咬一口：你以為多工讓所有請求並行了，結果一個丟包讓它們集體暫停。

這是一個極其值得記在腦中重放的因果鏈：**HTTP/2 用「單一 TCP 連線」換來了乾淨的多工，但也正因為綁死在單一 TCP 連線上，它繼承了 TCP「全有序交付」這個承諾的副作用。** 要根治，就不能再讓多個 stream 共用一條「對丟包一視同仁」的 TCP 連線。問題被推到了傳輸層本身——而 TCP 是動不了的，它躺在每一台路由器、每一個作業系統核心裡幾十年了。

## HTTP/3：既然動不了 TCP，就在 UDP 上重造一個傳輸層

HTTP/3（RFC 9114，2022 年成為 Standards Track）的解法乾脆而激進：**不再用 TCP。** 它把傳輸層換成 **QUIC**（RFC 9000，2021 年發布），而 QUIC 跑在 **UDP** 上。

為什麼是 UDP？因為 UDP 幾乎什麼都不保證——它只負責把一包資料盡力送到對面，不保證順序、不保證不丟、不做重傳。這聽起來像退步，但它正是 QUIC 需要的**一塊乾淨的畫布**。TCP 之所以動不了，是因為它的「全有序交付」邏輯燒死在核心裡；而 UDP 上面什麼都沒有，QUIC 就能在使用者態（user space）**重新實作一套傳輸層**，並且這次，讓它**認得 stream**。

這是整個 HTTP/3 最關鍵的一招。QUIC 自己內建了多工——stream 是 QUIC 這一層的原生概念，不是疊在不認得它的傳輸層之上的幻覺。於是 QUIC 可以做 TCP 永遠做不到的事：**每個 stream 獨立做遺失復原（loss recovery）**。一個封包丟了，QUIC 知道它屬於哪個 stream，於是它**只扣住那一個 stream** 等重傳，其餘 stream 的資料照常交付給上層。

回到剛才那個場景：stream 3 的封包丟了。在 HTTP/3 裡，stream 1 和 stream 5 的資料只要到齊就立刻放行，完全不受 stream 3 那個洞的牽連。**傳輸層的隊頭阻塞，到這裡才算真正被根治。** 這不是「又快一點」，而是把那條從 HTTP/1.1 一路追到 HTTP/2 都沒抓到的隊頭阻塞，逼到了最後一個藏身的角落並消滅它。

QUIC 既然把傳輸層整個重寫，順手還收了幾個 TCP 時代的老帳：

**握手合併，連線更快。** TCP 與 TLS 在傳統堆疊裡是兩層、要分兩次握手（前面算過約兩個 RTT）。QUIC 把傳輸握手與 TLS 1.3 的加密握手**合二為一**，新連線只要**一個 RTT** 就能開始傳資料。對曾經連過的對端，甚至能做 **0-RTT**——把第一個請求直接塞進握手的第一包送出去，連那一個 RTT 都省了。

**連線遷移（connection migration），換網不斷線。** TCP 連線是用「來源 IP + port、目的 IP + port」這個四元組來識別的——你的手機從 Wi-Fi 切到行動網路，IP 一變，這個四元組就變了，TCP 連線當場死亡，得整個重來。QUIC 不靠 IP 識別連線，它給每條連線一個獨立的 **connection ID**。IP 變了沒關係，只要 connection ID 還在，server 就認得「這還是同一條連線」（server 會先用一次輕量的 path validation 確認新路徑可達、來源未被偽造，再把連線遷過去），握手不必重做，下載不會中斷。對在地鐵、電梯、街上不斷切換網路的行動使用者，這是看得見的體驗差異。

## 一個頁面、三種世代，把時間手算出來

把上面的機制換算成具體數字，差異會更清楚。設想一個頁面要抓 30 個小資源，到 server 的 RTT 是 100 毫秒，並且假設這些資源都很小、傳輸時間相對 RTT 可忽略，連線都已建立好（先不算開場費，只看「發完所有請求」要幾個 RTT）。

**HTTP/1.1**：瀏覽器對這個 host 最多開 6 條連線，每條一次只能有一個請求在飛。30 個資源分到 6 條連線上，每條要依序跑 5 個。一個請求一去一回是一個 RTT，所以每條連線要 5 個 RTT 才能把分到的 5 個請求跑完——30 個資源**至少 5 × 100 = 500 毫秒**才全部完成。

**HTTP/2**：一條連線、30 個 stream，30 個請求**一次全部發出**，理論上**一個 RTT（100 毫秒）** 內全部在飛、回應陸續回來。從 500 毫秒到 100 毫秒，這就是多工的威力。

但現在加一個現實條件：這條 TCP 連線在傳輸中途丟了**一個**封包，重傳需要一個 RTT。在 HTTP/2 下，這 100 毫秒的重傳等待期間，因為 TCP 把洞後面的位元組全扣住，**30 個 stream 全部停擺**——大家一起多等這 100 毫秒。在 HTTP/3 下，同樣丟一個封包，**只有踩到那個封包的那一個 stream 停**，其餘 29 個照常完成。丟包率越高、網路越差，這個差距被放大得越明顯。這就是為什麼 HTTP/3 的好處在行動端與跨國場景才真正兌現，在零丟包的機房內網裡幾乎量不出來。

## 沒有免費的午餐：QUIC 也欠下了新的債

把傳輸層搬到 UDP、搬進使用者態，解掉了隊頭阻塞，但每一個好處背後都有對應的代價，講透它們才算真懂這個機制會在哪裡咬人。

**0-RTT 不是免費的，它對重放攻擊天生不設防。** 0-RTT 的第一包資料是用「上次連線記住的金鑰」加密的，它**沒有前向保密、也無法防止被重放**。一個攻擊者只要側錄下那個 0-RTT 封包，原封不動重送給 server，server 沒辦法分辨這是不是同一個請求的第二次。如果那個請求是「轉帳 100 元」這種**非冪等**操作，重放就等於被執行兩次。所以規範明確限制：**0-RTT 只能用於冪等請求**（見〈重複是常態〉），server 對不該被重放的請求要回 `425 Too Early`，逼 client 改走完整握手。把 0-RTT 當成「無腦加速所有請求」的開關，是會出事的。

**header 壓縮也得跟著重寫，而且做了妥協。** HTTP/2 的 HPACK 有一個內建假設：所有 frame 在連線上是**全域有序**的，所以動態表裡「第 N 個 header」這種引用永遠指得準。但 HTTP/3 的 stream 是亂序到達的——如果 stream 5 的 header 引用了「動態表第 3 項」，而定義第 3 項的那個 stream 2 的封包還沒到，stream 5 就得停下來等它。這等於把剛剛千辛萬苦消掉的隊頭阻塞，從壓縮表這個後門又放了回來。HTTP/3 因此把 HPACK 換成了重新設計的 **QPACK**：它讓編碼端能**自己權衡**——要嘛保守一點、少用會造成跨 stream 依賴的引用（壓縮率低一點，但不會被別的 stream 卡住），要嘛激進一點追求壓縮率（但可能引入一點隊頭阻塞）。這個「壓縮率 vs 隊頭阻塞」的旋鈕，是 QUIC 為了根治傳輸層阻塞而不得不在另一處留下的張力。

**UDP 在真實網路裡常被當二等公民。** 幾十年來，企業防火牆、NAT、各種中間設備都是圍繞 TCP 調校的，不少網路環境直接封掉 UDP/443，或對 UDP 流量限速、不友善。所以幾乎所有 HTTP/3 部署都得準備一條**退回 TCP（HTTP/2）的後路**：client 先試 QUIC，發現走不通就退回去。HTTP/3 的能力通常是透過 `Alt-Svc` header 宣告的——你的第一個請求其實還是走 HTTP/2，server 在回應裡告訴你「我也支援 HTTP/3，下次來試試這個 UDP 端口」，client 才在後續連線升級上去。

**加密搬進使用者態，CPU 帳單變貴。** TCP + TLS 的處理有大量成熟的核心優化、甚至硬體卸載（offload）。QUIC 跑在使用者態、每個封包的加解密都在應用程式裡做，在同樣的吞吐下通常更吃 CPU。對一個流量巨大的 server，這筆 CPU 成本是實打實的，不能假裝它不存在。

## 那到底該用哪一個

把這些取捨收束成判準，而不是「越新越好」的口號：

對**面向公網、使用者大量在行動端或跨國、丟包率偏高**的流量，HTTP/3 的每 stream 獨立復原與連線遷移有實質價值——而且這通常不必你自己折騰，CDN 邊緣多半已經替你開好 HTTP/3，並準備好退回 HTTP/2 的後路。

對**資料中心內部、服務對服務、低丟包的可信網路**，HTTP/2 的多工就足夠了。HTTP/3 在這裡解的傳輸層隊頭阻塞幾乎不會發作（封包根本不丟），白白多付使用者態加密的 CPU 帳單並不划算。這也正是 gRPC 預設跑 HTTP/2 的原因（見〈gRPC 與 Protobuf〉）——它的主戰場是低丟包的內網。

對**低頻呼叫、後台批次、需要用純文字 debug** 的場景，HTTP/1.1 反而最省心：協定簡單、人眼可讀、proxy 相容性最高，沒必要為了「上新版」而上。

| 世代 | 隊頭阻塞在哪一層 | 多工方式 | 新連線握手 | 最適場景 |
|---|---|---|---|---|
| HTTP/1.1 | 應用層（單行道、序列化） | 每 host 開多條連線模擬 | TCP + TLS ≈ 2-RTT | 簡單 REST、debug、低頻 |
| HTTP/2 | 傳輸層（TCP 全有序的副作用） | 單連線多 stream | TCP + TLS ≈ 2-RTT | 低丟包內網、微服務、gRPC |
| HTTP/3 (QUIC) | 無傳輸層隊頭阻塞 | QUIC 原生 stream，獨立復原 | 1-RTT，復用 0-RTT | 行動、跨國、高丟包、CDN 邊緣 |

最後要避開的一個陷阱：**HTTP/2「單一連線」這個設計，碰上 L4 負載均衡會反咬一口。** L4 LB 以「連線」為分配單位，一條長命的 HTTP/2/gRPC 連線一旦被釘到某台後端，這條連線上後續成千上萬個請求就全去那一台，看起來連線數均衡、實際請求量嚴重失衡。要在多工協定上拿到請求級的均衡，得靠 L7 或 client-side 負載均衡（見〈負載均衡〉）。多工省了連線，卻把均衡的難題挪到了別處——這又是一筆換來的、不會憑空消失的債。

## 為什麼是這個形狀

退一步看，這三個世代不是三套互不相干的協定，而是**同一個問題被逐層逼問的記錄**：怎麼在一條（或幾條）底層連線上，塞進許多個請求，而不讓它們互相卡死。

HTTP/1.1 用 keep-alive 解掉了「每個請求都重開連線」的浪費，但留下一條序列化的單行道，隊頭阻塞卡在應用層。HTTP/2 用多工把單行道變成許多條並行的 stream，消掉了應用層的阻塞——代價是它把所有 stream 綁在一條 TCP 連線上，於是繼承了 TCP「全有序交付」的承諾，把隊頭阻塞偷偷搬到了傳輸層。HTTP/3 看穿了「只要還躺在 TCP 上就躲不掉這個承諾」，索性在 UDP 上重造一個認得 stream 的傳輸層，讓每個 stream 獨立復原，才把隊頭阻塞逼到最後的角落消滅——代價是 0-RTT 的重放風險、QPACK 的壓縮妥協、UDP 的部署摩擦、使用者態加密的 CPU 帳單。

每一代都解掉了上一代的卡頓，每一代也都欠下了新的債。沒有哪一個是「終極答案」，只有「在你這個網路條件下，哪一筆債最便宜」。下次你在 DevTools 裡看到某個請求標著 `h2` 或 `h3`，你看到的不是一個版本號，而是一整條「隊頭阻塞被逐層往下趕」的故事——以及為了趕走它，這一層各自付出了什麼。
