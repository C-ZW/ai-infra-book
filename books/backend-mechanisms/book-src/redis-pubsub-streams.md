# Redis Pub/Sub 與 Streams：交付語意的取捨

有一個常見的時刻，會讓人第一次認真去分辨 Redis 這兩套訊息機制的差別。團隊做了一個跨節點的即時功能——比方說一個在線狀態廣播，或一套讓多台 web server 同步清快取的失效通知。手邊正好有 Redis，於是很自然地寫下 `PUBLISH` 跟 `SUBSCRIBE`，幾行就跑起來，demo 漂亮，上線。

然後某天，運維滾動更新了一批訂閱端。每個實例重啟大概花了八秒。重啟期間，發布端照常 `PUBLISH`，一秒兩百則。等實例回來、重新 `SUBSCRIBE`，一切看起來又正常了——但那八秒裡發出的大約一千六百則訊息，對這個實例來說**永遠不存在過**。沒有錯誤、沒有重試、沒有 backlog，連一行 log 都沒有。重連之後它只收得到「重連之後」的訊息，斷線那段是一個無聲的洞。

如果那些訊息是「使用者上線了」這種可丟的信號，這個洞無傷大雅——反正下一次心跳又會把狀態補回來。但如果裡面混進了一則「某筆訂單要開始處理」，那它就是一筆憑空蒸發的工作，而且你不會知道它蒸發了。**Redis 在「把訊息從一個地方送到另一個地方」這件事上提供了兩套機制，它們的交付語意天差地別**，而選錯的代價，正是這種查不到、重現不了、只在某次部署窗口裡悄悄漏掉幾筆的故障。這一章要把這兩套機制拆到你能對另一個工程師講清楚：它們底下各自怎麼運作、為什麼一個註定會丟、另一個用什麼換到了不丟，以及那個「不丟」其實藏了多少要你自己扛的責任。

## Pub/Sub 的本質：一條沒有記憶的管線

先看會丟的那一套。Pub/Sub 的心智模型可以濃縮成一句話：**它是一條即時的、沒有任何記憶的廣播管線。** `PUBLISH channel msg` 進來，Redis 做的事極其簡單——查一下「此刻有哪些連線訂閱了這個 channel」，把訊息逐一寫進那些連線的輸出緩衝、推出去，然後就忘了這件事。訊息沒有被存進任何地方，channel 不是一個會累積東西的佇列，它只是一張「誰在聽」的對照表。

這個設計帶來一個直接的後果，也是它全部問題的根源：**訊息只送給「發布的那一瞬間，剛好連著且已訂閱」的人。** 三種人會錯過它——發布前還沒訂閱的、發布時剛好斷線的、以及發布後才來訂閱的。對這三種人，訊息不是「晚點補給你」，而是從來沒有為他們存在過。沒有 replay，沒有歷史，沒有任何「我讀到哪了」的游標，因為根本沒有東西可以記游標。這就是教科書說的 **at-most-once（至多一次）**：每則訊息頂多被你收到一次，也可能一次都收不到，而你無從分辨「沒收到」和「沒發生」。

把它跟 Redis 的持久化機制對照，會更清楚這條管線有多「輕」。Redis 的 RDB 快照和 AOF 日誌（持久化的細節是 Redis 內部那章的事，見〈Redis 的內部：單執行緒、I/O threads、持久化與多重角色〉）會把資料結構刷到磁碟、會把主節點的寫入複製給副本。但 **Pub/Sub 的訊息兩者都不沾**——它不寫進 RDB、不進 AOF、也不複製給副本。一則 `PUBLISH` 在主節點上扇出完就消失了，副本那邊根本不知道發生過。這不是配置疏漏，是它的定義：一個不被持久化、不被複製、不被確認的東西，本來就沒有「之後」可言。

那它換到了什麼？換到了**幾乎為零的成本與最低的延遲**。沒有寫磁碟、沒有等副本、沒有維護任何狀態，發布端 `PUBLISH` 完幾乎立刻返回，訂閱端在訊息產生的同一個事件迴圈裡就拿到。對於「此刻的狀態信號」——某個節點要所有同伴立刻清掉某個 local 快取、某個使用者的在線狀態變了、一個 WebSocket 叢集要把訊息扇出到別台機器上的連線（這類 backplane 用法見〈WebSocket：frame、ping-pong 與水平擴展〉）——這正是你要的語意。這些訊息的價值就在「現在」，沒人在聽就代表此刻沒人需要，丟了也無所謂，因為下一個信號很快會覆蓋它。為這種訊息去買持久化，是替一個不存在的問題付管理費。

## 一個更隱蔽的丟法：連著也會被丟掉

到這裡，多數人對 Pub/Sub 的理解是「離線會漏」。但有個更陰險的丟法，連「訂閱端一直好好連著」都救不了你，值得單獨拆開——因為它推翻了一個很自然的假設：「只要我不斷線，就一則都不會少」。

問題出在那個「輸出緩衝」。Redis 把訊息推給訂閱端時，是寫進每條連線各自的 output buffer，再由作業系統慢慢送出去。如果某個訂閱端**消費得比訊息來得慢**——它的 CPU 卡住了、它在做一件耗時的事、或者單純發布速率突然飆高——它的 output buffer 就會越積越多。Redis 不會無限替它墊著：它對 Pub/Sub 連線設了一道 `client-output-buffer-limit`，預設值是 `32mb 8mb 60`，意思是**硬上限 32 MB**（一超過立刻砍連線），**軟上限 8 MB 持續 60 秒**（緩衝在 8 MB 以上賴著超過一分鐘，也砍）。把這道門檻換算成時間，它有多近就一目了然了：假設每則訊息 1 KB、發布速率 2000 則/秒，而某個訂閱端完全停止消費，它的 output buffer 就以每秒 1 KB × 2000 = 2 MB 的速度漲——32 MB 的硬上限大約 16 秒就被擊穿（`32 MB ÷ 2 MB/s ≈ 16 s`），8 MB 的軟上限更是 4 秒就到、再賴 60 秒一樣被砍。也就是說，一個 GC 卡頓十幾秒、或一段同步阻塞處理，就足以讓 Redis 把這個訂閱端踢掉。

砍掉之後會怎樣？這個慢訂閱端的連線被 Redis 主動關閉。它的 client library 八成會自動重連、重新 `SUBSCRIBE`，然後——你已經知道接下來會發生什麼了——**從被砍到重連回來的這整段，全部訊息照樣是那個無聲的洞**。它從頭到尾沒有「自己斷線」的意識，它以為自己一直在聽，但 Redis 早就把它踢了又讓它默默回來。一個訂閱端只要偶爾慢一下、buffer 漲過門檻，就會週期性地丟一批訊息，而監控上看到的只是「偶爾有連線重置」這種容易被忽略的雜訊。

這件事之所以重要，是因為它揭示了 Pub/Sub 「at-most-once」背後真正的工程含意：**Redis 為了保護自己（不讓一個慢消費者把記憶體吃爆），寧可丟掉你的訊息，也不會替你把訊息留著等你。** 這個取捨是對的——一個記憶體型的廣播系統，本來就不該為了某個拖後腿的訂閱端而把整個實例拖垮——但它意味著「不丟訊息」這個保證，Pub/Sub 從頭到尾就沒打算給你，連在最理想的「連線從未中斷」假設下都沒給。

## Streams：把訊息變成一筆會留下來的記錄

如果你要的是「這則訊息不能丟」，就得換一套底層觀念：**別把訊息當成轉瞬即逝的信號，把它當成一筆要被記下來的記錄。** 這正是 Redis Streams 做的事。

Stream 不是一張「誰在聽」的對照表，而是一個 **append-only 的日誌**。`XADD mystream * field value` 把一則訊息追加到日誌尾端，Redis 回你一個單調遞增的訊息 ID（預設是 `<毫秒時間戳>-<序號>`，例如 `1718000000000-0`）。這則訊息此刻**真的存在於 Redis 的資料結構裡**——它會進 RDB 快照、會寫 AOF、會被複製給副本，和你存的任何一個 hash、sorted set 同等待遇。這是它和 Pub/Sub 最根本的分水嶺：Pub/Sub 的訊息是「事件發生的副作用」，Stream 的訊息是「資料」。沒人在「發布的那一刻」聽，一點關係都沒有，因為它就躺在那裡，誰都能晚點來讀。

最樸素的讀法是 `XREAD`：你給它一個起始 ID，它把那之後的訊息給你，讀到哪、下次從哪開始，全由你自己記。這已經比 Pub/Sub 強——至少有歷史、能重讀——但它把「進度」這件事整個丟回給你。一旦你想要「多個 worker 分攤同一條流」「某個 worker 掛了它沒讀完的活要有人接手」這種真正的工作佇列語意，光靠 `XREAD` 是湊不出來的。真正讓 Streams 變成「Redis 內的迷你 Kafka」的，是 **consumer group**。

## consumer group：at-least-once 是怎麼長出來的

consumer group 是這一章的肉，值得一步步在腦中重放，因為「不丟」這個保證的全部機關都在這裡。

你先用 `XGROUP CREATE mystream mygroup $` 在一條 stream 上建一個 group。group 是一個共享的消費實體：組裡可以有很多個 consumer（用名字區分，如 `worker-1`、`worker-2`），Redis 保證**同一則訊息只會被分派給組內的一個 consumer**，於是多個 worker 自然地把一條流的工作量分攤掉，彼此不重工。不同的 group 之間則完全獨立、各讀各的同一份日誌——這就是「多訂閱者」與「重放」的來源。

關鍵的機關在讀取這一步。consumer 用 `XREADGROUP GROUP mygroup worker-1 COUNT 10 STREAMS mystream >` 來拿訊息，那個特殊的 `>` ID 意思是「給我這個 group 從沒有人領過的新訊息」。Redis 把訊息交給 `worker-1` 的同時，**做了一件 Pub/Sub 從不做的事：它把這則訊息的 ID 記進一個叫 PEL（Pending Entries List，待處理清單）的結構裡**，標記為「已經發出去給 worker-1，但還沒有人確認處理完」。

訊息進了 PEL，就不算交差。consumer 把活幹完之後，**必須**呼叫 `XACK mystream mygroup <id>`，明確告訴 Redis「這則我處理好了」。`XACK` 一到，這筆 pending entry 才從 PEL 移除、真正落幕。整個生命週期是這樣的：

```
XADD      -> 訊息寫入 append-only log（持久化、可被複製）
XREADGROUP > -> 分派給某 consumer，同時記進 PEL（pending = 已發、未確認）
  ... consumer 處理 ...
XACK      -> 從 PEL 移除，這則訊息對這個 group 才算「完成」
```

現在來看為什麼這套機制能撐住「不丟」。假設 `worker-1` 在 `XREADGROUP` 之後、`XACK` 之前**崩潰了**——它領了訊息但活沒幹完。在 Pub/Sub 的世界，這則訊息早就推出去消失了，沒救。但在 Streams，這則訊息的 ID**還躺在 PEL 裡**，標記著「已分派給 worker-1、未確認」。它不會自己消失，也不會自動退回給別人，但它**可被找回**：`worker-1` 重啟後，可以用 `XREADGROUP` 帶一個明確的 ID `0`（而不是 `>`）去讀「我自己 PEL 裡還沒 ack 的舊帳」，把上次沒做完的重做一遍；或者如果 `worker-1` 再也不回來了，**別的 consumer 可以把它的 pending 訊息接管過來**——`XAUTOCLAIM`（或更底層的 `XCLAIM`）會把閒置超過某個時間（`min-idle-time`）的 pending 訊息，轉移所有權給一個健在的 consumer 去處理。

把這條鏈接起來，就是 **at-least-once（至少一次）**：一則訊息一旦寫進 stream，就會被持有它的 group 處理**至少一次**——要嘛被 `XACK`，要嘛留在 PEL 等著被重讀或被接管，總之不會在「領了卻沒做完」的縫隙裡蒸發。回到開頭那個八秒、一千六百則的場景：如果當初用的是 Streams + consumer group，那一千六百則訊息全都安穩地躺在日誌裡，worker 重啟回來、從上次 ack 過的位置之後繼續 `XREADGROUP`，一則不漏。Pub/Sub 用「會丟」換「零狀態」，Streams 反過來——它用「要記 PEL、要 ack、要管接管」這一堆狀態，換到了「不丟」。

## 「至少一次」這四個字，每一個字都要還債

但這裡有個極容易被一句「Streams 是可靠的」帶過去、卻會在生產上反咬你的真相：**at-least-once 的字面意思就是「至少」一次，不是「剛好」一次。** 那條保證鏈的另一端，欠下的債得你自己還。

最直接的後果是**重複**。設想 `worker-1` 已經把訊息處理完了——錢扣了、信寄了、外部 API 也呼叫了——但就在要 `XACK` 的前一刻它崩了，或者那個 `XACK` 的網路封包在路上丟了。從 Redis 的角度，這則訊息**從未被確認**，它老老實實留在 PEL 裡。於是過一陣子，`XAUTOCLAIM` 把它接管給 `worker-2`，`worker-2` 盡責地把它**再處理一次**——又扣一次錢、又寄一封信。Streams 信守了「至少一次」的承諾，而那「多出來的一次」正是這個承諾的代價。

這就是為什麼任何用 at-least-once 機制的消費端，**處理邏輯必須冪等**——同一則訊息處理兩次、三次，效果要和處理一次一樣（冪等怎麼設計、冪等鍵的生命週期，是另一個主題，見〈重複是常態：冪等與去重〉）。Streams 給你的不是「exactly-once」，它從機制上做不到；它給你的是「不會少，但可能多」，把「多出來的那次怎麼變無害」這個責任，明明白白地交到你手上。（Redis 8.6 起 `XADD` 加了 producer 端的冪等寫入去重（`IDMP`／`IDMPAUTO`，靠 producer-id ＋ idempotent-id 比對），但那解的是「producer 重試把**同一筆 `XADD` 寫了兩次**」這個 production 側的問題，不是「同一筆訊息被 consumer **處理**兩次」——消費端的冪等仍得你自己做。）誰要是把 Streams 當成 exactly-once 來用、消費端沒做冪等，那次接管重投就是一筆重複的副作用，而且同樣難查——因為大部分時候接管不會發生，它只在 worker 恰好在 ack 前掛掉的窄窗口裡咬你。

第二筆債更隱蔽，藏在 PEL 自己身上。PEL 是「已發未確認」的清單，它有一個沉默的前提：**訊息一旦被 ack 才會離開，沒被 ack 的會一直堆著。** 如果某個 consumer 因為 bug 卡死了、或者它的處理邏輯有個分支永遠到不了 `XACK`，那些它領走的訊息會**無限期地賴在 PEL 裡**——不會自己消失，也不會自動被重投給別人，除非有人主動跑 `XAUTOCLAIM` 去接管。一個沒人盯著的 PEL 會悄悄漲大，它是 Streams 的一種隱形背壓（背壓的整套機制見〈背壓：當下游跟不上〉）：上游一直 `XADD`，下游一直領、卻一直不 ack，PEL 就成了一個只進不出的黑洞。**運維 Streams 的人必須監控 PEL 的積壓**（用 `XPENDING` 看），它漲起來就是「有 consumer 領了活沒幹完」的信號。

Streams 還順手在 PEL 裡藏了一個很有用的東西：**每則 pending 訊息帶一個投遞計數**。每次它被 `XCLAIM`/`XAUTOCLAIM` 重新接管，這個計數就加一。於是一則「每個 consumer 領了都會讓它崩潰」的毒訊息（poison message），它的投遞計數會越爬越高——你可以靠監控這個計數把這種訊息揪出來，超過某個門檻就把它丟進死信佇列（DLQ 的機制見〈死信與毒訊息：DLQ〉），而不是讓它一個接一個地搞垮你的 worker。這個計數是 Pub/Sub 連做夢都給不了的——你連訊息發出去之後的命運都不知道，遑論數它被誰處理失敗了幾次。

最後一筆容易被忘的債：**stream 的日誌不會自己變短。** `XADD` 預設只管追加、不管修剪，一條被狂寫的 stream 會無上限地長大，最後把 Redis 的記憶體吃光。你得自己用 `XADD mystream MAXLEN ~ 100000 * ...`（近似修剪，那個 `~` 是讓 Redis 為了效率不必精確砍到剛好 10 萬筆）或定期 `XTRIM` 來控制它的長度。Pub/Sub 不留東西，所以沒有這個問題；Streams 留東西，所以「留多久、留多少」就成了你要回答的問題。持久化的代價，從來不只是磁碟。

## 為什麼有人還是選那條會丟的路

看到這裡，一個自然的反應是：既然 Streams 不丟、Pub/Sub 會丟，那 Pub/Sub 是不是個該被淘汰的劣等品？恰恰相反——**很多時候，那個「會丟」才是你真正想要的語意，而 Streams 的「不丟」是你不需要、卻要付管理費的負擔。**

判準其實只有一句話：**這則訊息代表的是「此刻的一個信號」，還是「一件必須被做掉的工作」？**

如果是前者——清快取的失效廣播、在線狀態、即時通知的扇出、WebSocket 跨節點的 backplane——那麼「沒人在聽就丟掉」恰恰是正確的。一個快取失效信號，如果發出去的那一刻沒有節點在聽，代表那一刻沒有節點有那份快取需要清，丟了完全合理；就算某個節點剛好錯過，它的 local 快取 TTL 到期後自然會重讀新值，不一致是有界且自癒的（多層快取裡用 Pub/Sub 廣播失效、以及它「會漏」的後果，見〈快取策略：cache-aside、write-through、write-behind〉）。對這種訊息硬上 Streams，等於替「丟了也無所謂的東西」扛起 PEL 監控、ack、修剪、冪等這一整套責任——付出全是真的，買到的保證你根本用不上。

如果是後者——一筆要處理的訂單、一個要寄的通知、一件不能因為 worker 重啟就消失的活——那麼把它放 Pub/Sub 就是在賭「消費端永遠在線、永遠不慢、永遠不會被 buffer limit 砍」。這個賭，在第一次滾動更新、第一次 GC 卡頓、第一次流量尖峰時就會輸，而且輸得無聲無息。

值得補一筆的是 Pub/Sub 在 cluster 模式下的一個演化，因為它常被誤解成「修好了可靠性」。傳統 Pub/Sub 在 Redis Cluster 裡有個擴展瓶頸：為了讓訂閱端連到任一節點都能收到訊息，每則 `PUBLISH` 都得透過 cluster bus 廣播到**所有**節點，節點一多，這個內部廣播就成了負擔。Redis 7.0 加入的 **sharded Pub/Sub**（用 `SPUBLISH`/`SSUBSCRIBE`）把 channel 依 slot 分片，訊息只在負責那個 slot 的 shard 內部傳播，省掉了全叢集廣播。但要講清楚的是：**它改的是擴展性，不是交付語意**——sharded Pub/Sub 依然是 at-most-once，依然會在離線、晚訂閱、buffer 超限時丟訊息。它讓 Pub/Sub 在大叢集裡更能撐，卻沒有、也不打算把它變成可靠交付。把 sharded Pub/Sub 當成「可靠版 Pub/Sub」，是又一個會在某次部署窗口裡咬人的誤解。

同樣的陷阱有個常見的變體：**keyspace notifications（鍵空間通知）**。Redis 可以在某個 key 被改、被刪、過期時發出一個事件，聽起來像個天然的「資料變更事件源」。但它**底層就是 Pub/Sub**——所以它繼承了 Pub/Sub 的一切：at-most-once、會漏、訂閱端離線或太慢的期間發生的所有 key 事件全部消失。拿它來做「即時刷新某個畫面」這種可丟的事沒問題；拿它來當「絕不能漏掉任何一筆資料變更」的可靠捕獲機制，就是把一個會漏的東西架在一個不能漏的需求上。要可靠地捕捉資料庫或 key 的每一次變更，那是 CDC 一類機制的事（見〈CDC：從資料庫變更長出事件流〉），不是 keyspace notifications 能擔的。

## 兩條路放在一起看

把這一整章收進一張表，不是因為表能取代上面的敘事，而是因為當你真的站在「這個功能該用哪個」的岔路口時，需要一眼掃過去確認自己沒漏掉哪個取捨：

| | Pub/Sub | Streams + consumer group |
|---|---|---|
| 交付語意 | at-most-once（可能一則都收不到） | at-least-once（不會少，但可能多） |
| 訊息留存 | 不留，發完即忘 | append-only log，持久化＋複製 |
| 離線/晚到的消費者 | 永久錯過那段 | 重連後從上次位置續讀，一則不漏 |
| 慢消費者 | buffer 超限被砍、靜默丟訊息 | 訊息留在 stream/PEL，不丟 |
| 進度追蹤 | 無 | group 記到哪、PEL 記未確認 |
| 重複 | 不會（因為根本不重投） | 會（接管重投），消費端須冪等 |
| 要你扛的責任 | 幾乎沒有 | ack、監控 PEL 積壓、修剪日誌、冪等 |
| 適合 | 此刻的信號、可丟、最低延遲 | 不能丟的工作、要重試、多 worker 分攤 |

這張表真正想說的，其實是同一條脊椎的兩端：**你想要什麼保證，你願意付什麼代價。** Pub/Sub 把「不丟」整個放掉，換回零狀態、最低延遲、零維護負擔；Streams 把「不丟」買進來，代價是 ack 的紀律、PEL 的監控、日誌的修剪，以及最關鍵的——一個你必須在消費端親手建好的冪等防線，去吸收那「至少一次」裡多出來的次數。

## 為什麼是這個形狀

退一步看，這兩套機制不是 Redis 隨手提供的兩個選項，它們是「訊息」這個詞兩種截然不同的解讀，各自被推到極致的樣子。

Pub/Sub 把訊息理解成**一個當下的事件**——一道閃過的信號，它的全部意義就在「現在」這一刻被聽見。一個只關乎當下的東西，沒有理由被存下來、被複製、被確認；於是它自然長成一條沒有記憶的管線，輕到幾乎沒有成本，也輕到沒有任何挽回的餘地。它「會丟」不是因為做得不好，而是因為它根本沒有「之後」可以丟。

Streams 把訊息理解成**一筆被記下來的資料**——一個事實，發生過就該留著，誰都能晚點來讀、來確認、來分攤。一個要被留著的東西，自然就得進磁碟、進副本、有 ID、有「誰讀到哪」的游標、有「誰領了還沒交差」的待辦清單；於是它長成一個帶 PEL 的 append-only log，買到了「不丟」，也就此背上了所有持久化系統都逃不掉的負擔：狀態要管、積壓要看、空間要清、重複要防。

所以下次你在 Redis 上要把一則訊息從一處送到另一處，真正要先問清楚的，從來不是「用 `PUBLISH` 還是 `XADD`」，而是那句更底層的問題：**這則訊息要是沒人在聽，它該消失，還是該等？** 答案決定了你選哪一套，也決定了你接下來要為它扛起哪一堆——或哪一堆都不必扛。
