# event loop：單執行緒怎麼同時做很多事

打開一台跑著 Node.js 的 API 伺服器，用 `top` 看它。它正穩穩服務著八千條同時連著的 WebSocket、每秒回掉幾千個 HTTP 請求，而它只佔著**一條**執行緒。沒有執行緒池在背後悄悄開了八千條 worker，沒有每來一個連線就 fork 一個行程——就是一條 main thread，一個人扛下全部。

這件事第一次看到時是反直覺的。我們對「同時做很多事」的本能想像是「多請一些人手」：來一個請求派一條執行緒去伺候，人多好辦事。一台機器同時要照顧八千個連線，腦中浮現的是八千條執行緒各據一方。但 thread-per-connection 這條路有它的天花板——每條 OS 執行緒光堆疊就要吃掉約 0.5 到 8 MB 記憶體，八千條就是好幾 GB 只花在「站著等」上；而且這八千條多半時間什麼都沒做，就掛在自己那個 `read()` 上等對端送資料來，作業系統卻還得在它們之間排程、切換、存還暫存器，把 CPU 大把耗在管理執行緒本身（這條路的算術與 C10K 的細節屬於另一個主題，見〈阻塞與非阻塞 I/O〉）。

event loop 走的是完全相反的賭注。它賭的是：這八千個連線裡，**任一瞬間真正需要 CPU 動手的，少之又少**。絕大多數連線此刻都在等——等對端送下一個封包、等資料庫回查詢、等磁碟把檔案讀完。既然大家都在等，那為什麼要為「等」這件事各養一條執行緒？把所有的等待集中交給作業系統去盯，一條執行緒只在「真的有事要做」的那一刻醒來把事做掉——這就夠了。要理解 event loop，得先把這個賭注的底細看清楚：它賭工作負載的瓶頸在 I/O 等待、不在 CPU 計算，而這個賭注一旦賭錯，整套機制會以最難看的方式崩掉。

## 一條執行緒的紀律：絕不停下來等

event loop 的全部魔法可以濃縮成一條鐵律：**永遠不要在這條執行緒上同步地停下來等任何東西。**

把這條律拆開看就懂它有多嚴格。傳統寫法裡，你呼叫 `const row = db.query(sql)`，這一行會卡住——程式停在這裡，直到資料庫回了結果才往下走。在 thread-per-connection 的世界這沒問題，反正卡住的只是這條連線專屬的執行緒，別人有別人的執行緒。但在 event loop 的世界，只有一條執行緒，你一旦卡住它，**所有八千個連線同時凍結**。所以這裡不能有「卡住等結果」這回事。

取而代之的是一套「登記、返回、之後再回來」的節奏。你發起一個會等待的操作（送出 DB 查詢、發起一個 HTTP 呼叫、開始讀一個檔案），但你不站在原地等它完成——你**登記一個 callback**（「等結果回來了，請執行這段」），然後立刻返回，把執行緒讓出去處理別的連線。那個操作交給作業系統的非同步機制去盯著。等資料真的就緒了，作業系統通知回來，event loop 找出對應的那個 callback、把它排進待辦，等輪到它就執行。

於是一條執行緒的時間軸長這樣：醒著的每一刻，它要嘛在跑某個已就緒事件的 callback（純 CPU 動作，很快），要嘛在「沒事可做」時把自己掛起、交給作業系統去等下一個事件就緒。它從不為單一一個連線的等待停留——所有的等待被攤平、疊在一起，由作業系統統一盯著。八千個連線的等待重疊成一個等待，這就是單執行緒能服務海量並發的根。

這裡藏著一個常被忽略、卻是 event loop 最大的隱性紅利：**因為任一瞬間只有一個 callback 在動共享狀態，runtime 內天生沒有資料競爭。** 你不需要鎖、不需要 atomic、不用擔心兩個 callback 同時改同一個變數改出半成品——它們根本不可能「同時」，event loop 一次只跑一個，跑完一個才跑下一個。多執行緒模型用「真平行」換來「你得自己管同步」的那些苦，event loop 一概免了（並發與平行是兩回事，這個區分本身值得單獨講，見〈並發不是平行〉）。當然天下沒有白吃的午餐，這個紅利的代價待會就會以最痛的形式出現。

## 那條執行緒醒著的時候在做什麼：phase 循環

「event loop 不斷輪詢有沒有就緒的事件」這句話太籠統了。真要在腦中重放它，得看清那條執行緒一輪到底走了哪些步驟。我們用 Node.js 底層的 libuv 來看——它是這套機制被實作得最廣為人知的版本（2026-06 的 Node active LTS 是 Node 24「Krypton」）。

libuv 把事件迴圈的一輪切成幾個固定的**相位（phase）**，每一輪依固定順序走過它們，走完一輪再從頭來。順序是這樣的：

```text
   (one iteration of the loop; not executable)
   ┌───────────────────────────────────────────────┐
┌─>│  timers  (due setTimeout / setInterval)       │
│  ├───────────────────────────────────────────────┤
│  │  pending callbacks  (deferred system cbs)     │
│  ├───────────────────────────────────────────────┤
│  │  idle / prepare  (internal only)              │    incoming:
│  ├───────────────────────────────────────────────┤
│  │  poll  (wait for & take I/O; may BLOCK here)  │<── sockets,
│  ├───────────────────────────────────────────────┤
│  │  check  (setImmediate callbacks)              │    data, etc.
│  ├───────────────────────────────────────────────┤
└──┤  close callbacks  (socket 'close', etc.)      │
   └───────────────────────────────────────────────┘
```

每一輪，這條執行緒：先看 **timers** phase——有哪些 `setTimeout`／`setInterval` 已經到期了，把它們的 callback 跑掉；接著 **pending callbacks** 處理一些被延後到這輪的系統層 callback；中間的 **idle/prepare** 是 libuv 內部自用、我們碰不到；然後是全場的核心 **poll**；之後 **check** phase 跑所有 `setImmediate` 排的 callback；最後 **close callbacks** 處理像 socket 關閉這類善後事件。

整套機制的重心，全壓在 **poll** 這一個 phase 上。poll 才是那條執行緒「掛起來等作業系統」的地方——它呼叫作業系統的事件通知 API（Linux 上是 `epoll`、macOS/BSD 是 `kqueue`，這兩個是「就緒通知」——告訴你某個 fd 可讀可寫了，你再自己去 `read`；Windows 的 IOCP 走的是另一套「完成通知」模型——作業系統替你把 I/O 做完、資料已在你的緩衝區裡才通知你），一口氣問作業系統「我盯著的這幾千個 fd 裡，現在哪些有動靜了？」。如果有就緒的，把對應的 I/O callback 一個個跑掉；如果一個都還沒就緒，這條執行緒就**真的在這裡阻塞睡著**，把 CPU 整個讓出去，直到作業系統喚醒它說「有事了」。

「在 poll 阻塞」聽起來和「絕不停下來等」自相矛盾，其實正相反——這是整套設計的精髓。它**不為任何單一連線停留**，但它會為「全體連線中，下一個就緒的那個」一起停留。一條執行緒阻塞一次，等的是八千個 fd 裡最先有動靜的那一個，而不是八千次各等各的。把八千個分散的等待，合併成一次集中的等待，這就是 epoll/kqueue 替 event loop 扛起的活（這幾種就緒通知 API 的差別，是〈阻塞與非阻塞 I/O〉的主題）。

## 它睡多久才醒：一個容易被略過的細節

poll 不會無止境地睡——它睡多久，是**算**出來的，而這套計算方式藏著一個只有把機制重放到底才看得見的設計。

在進 poll 之前，libuv 會先算一個 timeout：去看 timer 堆裡**最近一個還沒到期的 timer 還有多久到期**。假設下一個 `setTimeout` 是 50 毫秒後要觸發，那 poll 最多就只能睡 50 毫秒——因為睡過頭，那個 timer 就遲到了。所以 poll 的阻塞時間，被「最近的 timer」這條繩子拴著上限。如果一個 timer 都沒有，而且確實有 I/O 在等，poll 就可以一路睡到「有 I/O 就緒」才醒；但如果連 I/O 都沒在等、又沒 timer、整個 loop 沒事可幹了，timeout 算出來是 0，poll 不阻塞、直接過——這正是一個沒有任何掛起工作的 Node 行程會乾淨俐落地退出，而不是傻站著空轉的原因。

這也順手解釋了一個經常被誤解的行為：`setTimeout(fn, 0)` **不會立刻執行**。它被排進 timers phase，而 timers phase 一輪只走過一次——你的 `fn` 得等執行緒繞完這一圈、下一輪走到 timers 才被跑。而且即使你寫 `0`，Node 也會把延遲**無條件夾到最小 1 毫秒**（瀏覽器則是另一套規則：只有在 `setTimeout` 巢狀達第 5 層以上時才強制夾到最小 4 毫秒，第一層的 `setTimeout(fn, 0)` 不受此限——別把瀏覽器那個有條件的 4 毫秒記成無條件的）。所以「`setTimeout(fn, 0)` 馬上跑」這個直覺從兩個方向同時破：它不在當下、它甚至不真的是 0。如果你要的是「盡快、但先讓出控制權給正在等的 I/O」，正確工具是 `setImmediate`——它排在 check phase，會在這一輪的 poll 把 I/O 處理完之後緊接著跑，而不是繞到下一輪的 timers。

## phase 之間的插隊者：microtask 與一條餓死 I/O 的路

到這裡 phase 循環看起來很整齊，但真實的執行序裡還有一群**不屬於任何 phase**的插隊者，它們才是 event loop 最危險的角落。

那就是 **microtask**：在 Node 裡有兩種，`process.nextTick` 排的 callback，以及 Promise 的 `.then`/`await` 接續。它們的排程規則和六個 phase 完全不同——microtask **不等到某個 phase**，而是在「**每一個 callback 跑完、要進下一個之前**」就被整個排空。也就是說，不管你現在在哪個 phase，只要當前這個 callback 一執行完，event loop 會先停下來把累積的 microtask 全部清空（`nextTick` 佇列優先於 Promise 佇列），清乾淨了才繼續往下走。

這個「先於一切下一步」的特性，多數時候是好事——它讓「緊接著當前邏輯的收尾」能在任何 I/O 插進來之前先做完。但它藏著 event loop 整套設計裡最陰險的一個故障模式。設想這段碼：

```js
// not executable — illustrates microtask starvation
function spin() {
  process.nextTick(spin);  // re-queues itself, forever
}
spin();
```

每跑完一次 `spin`，它又往 microtask 佇列塞了一個 `spin`。event loop 的規矩是「microtask 沒清空，不准進下一步」——可這個佇列**永遠清不空**，因為每清一個就生一個。於是那條執行緒被困死在排空 microtask 的迴圈裡，**永遠走不到 poll phase**。結果是：所有 I/O 完全停擺，新連線進不來、已就緒的資料沒人讀、連 timer 都觸發不了——整個伺服器看起來「活著」（行程沒崩、CPU 100%），卻對外完全沒反應。用無界遞迴的 Promise 鏈也是一樣的死法。這是 event loop 最該記死的一個坑：**microtask 的優先權是用「可能餓死 I/O」換來的**。

值得停下來體會這裡的對稱性：同樣是「優先」，在 thread 模型裡一段死迴圈頂多吃滿它那一條執行緒、別的執行緒還能跑；在 event loop 裡，一段不肯讓出的 microtask 鏈會**連坐整個行程**。單執行緒「無資料競爭」的紅利，和「一個工作卡住就全卡住」的詛咒，是同一枚硬幣的兩面。

## 賭注賭錯的時候：CPU 重活的凍結

現在來看那個從一開始就埋著的代價。event loop 賭「大家都在等 I/O」，可如果某個 callback 不是在等、而是在**算**呢？

設想一個 API，某條路徑上要同步解析一個 50 MB 的 JSON 字串。`JSON.parse` 是純 CPU 工作，它不會 `await`、不會讓出，它就在那條 main thread 上**埋頭算**。假設這一算要 800 毫秒。把這 800 毫秒放進 event loop 的時間軸看：在這整整 800 毫秒裡，那條執行緒被這一個 callback 獨佔，它走不到 poll、處理不了任何就緒的 I/O、跑不了任何別人的 callback。

把數字鋪開算一遍，後果有多難看就清楚了。假設這個服務平常 QPS 是 100、p99 延遲約 20 毫秒，跑得好好的。某一刻有一個請求觸發了那段 800 毫秒的 `JSON.parse`。在它埋頭算的這 800 毫秒裡，按 QPS 100 估算，大約有 `100 × 0.8 = 80` 個其他請求陸續到達——它們的資料其實早就躺在 socket 上就緒了，但**沒有人去讀**，因為唯一的那條執行緒正卡在算 JSON。這 80 個請求只能排隊乾等。等那 800 毫秒結束、執行緒終於回到 poll，它們才被一次處理掉——但它們每一個的延遲，都憑空多了最多 800 毫秒。p99 從 20 毫秒被一個請求拖到 800 毫秒以上，而監控上你會看到一件詭異的事：CPU 沒爆、記憶體沒爆、那段 `JSON.parse` 本身「只是慢了一點」，但**整個服務的尾延遲集體爆炸**。元兇不是負載太高，是一個沒讓出的同步 CPU 區段，掐住了所有人的咽喉。

這就是 event loop 賭錯時的樣子。它的高並發完全建立在「callback 都很短、很快讓出」這個前提上；一旦某個 callback 變成 CPU 重活，單執行緒從紅利變成單點。這裡也順手破掉一個常見誤解：**`async` 並不能救你**。`async`/`await` 只保證「等 I/O 的時候讓出」，它對函式裡那段純同步的 CPU 計算毫無辦法——`await` 不會魔法般地把一個 800 毫秒的 `for` 迴圈切開。一個 `async function` 裡若有一段同步重活，它照樣把 event loop 凍得死死的。

解法不是「讓 event loop 跑快一點」（單執行緒就那麼快），而是把這塊計算**搬離這條執行緒**——丟到 worker thread 上去算，讓 main thread 繼續服務 I/O（這套外送機制是〈thread pool 與 worker threads〉的主題）。event loop 的設計信條從頭到尾沒變過：這條執行緒上，只准放「短到可以一口氣跑完」的工作。

## 那條被偷偷用掉的執行緒池

把「event loop 是單執行緒」這句話記得太死，會在一個地方踢到鐵板。事情是這樣的：作業系統並不是對所有 I/O 都提供漂亮的非同步介面。網路 I/O 有 epoll/kqueue 撐著、走得很乾淨；但有一類操作——**檔案系統**、**DNS 名稱解析（`dns.lookup`/`getaddrinfo`）**、**部分非同步密碼學運算（`pbkdf2`、`scrypt` 之類）**、**所有非同步 zlib 壓縮**——在主流平台上沒有真正好用的非同步原語。

libuv 對這類操作的處理方式是：偷偷在背後開一個**執行緒池**（預設 **4** 條，可用 `UV_THREADPOOL_SIZE` 調到最多 1024），把這些「沒有非同步原語」的操作丟進去，由池裡的工作執行緒**同步地**做完，做完再把結果丟回 main thread 的 event loop，當成一個就緒事件來處理。從你的 JS 看出去，`fs.readFile` 是非阻塞的；但它的非阻塞，是靠那 4 條背景執行緒替你扛下了那個同步的 `read`。

這個「池只有 4 條」是個容易悄悄咬人的數字。假設你的 API 每個請求都做一次 `crypto.pbkdf2` 雜湊密碼，單次約 100 毫秒。pbkdf2 走的是 libuv 執行緒池。某一刻同時湧進 8 個請求：前 4 個各搶到一條池執行緒、並行各算 100 毫秒；**後 4 個搶不到，只能排隊**，得等前 4 個騰出執行緒才開始算。於是第 5 到第 8 個請求的雜湊延遲憑空翻倍，從 100 毫秒變成約 200 毫秒。詭異之處在於：你的 JS 一行都沒卡 event loop、main thread 好端端的，服務卻就是慢了——因為瓶頸在那個你看不見的 4 條執行緒池被佔滿了。把 `UV_THREADPOOL_SIZE` 調到 8（且機器有 8 核可用），8 個請求就能同時做完、延遲回到約 100 毫秒。這是「Node 在高並發雜湊或大量檔案操作下突然變慢」最常見、也最難一眼看穿的真兇。

這裡還埋著一個尤其反直覺的邊角，值得單獨拎出來：**DNS 查詢未必走網路那條乾淨的路**。Node 裡 `dns.lookup`（也是 `http` 模組連線時預設用的那個）底層呼叫的是 `getaddrinfo`，這是一個**阻塞的 libc 呼叫**，所以它走的是那 4 條執行緒池——不是 epoll。於是一個對外發大量請求的服務，可能在某次下游 DNS 變慢時，被一批 `dns.lookup` 悄悄塞滿那 4 條池子，連帶把同樣靠這個池的檔案讀取、密碼雜湊一起拖慢，而你盯著網路指標怎麼看都看不出問題。相對地，`dns.resolve` 系列走的是 c-ares、真正在網路上用非阻塞 socket 解析、**不碰執行緒池**——兩個名字像、底下卻是兩條完全不同的路。把「DNS 是網路操作所以一定走 epoll」當理所當然，就會在這裡判斷失準。

所以「event loop 是單執行緒」這句話精確的講法是：**JS 的執行、event loop 本身的運轉，是單執行緒；但 runtime 在背後為那些沒有非同步原語的操作，藏了一個小小的執行緒池。** 網路 I/O 不碰這個池（走 epoll/kqueue），檔案、DNS lookup、async crypto、async zlib 才碰。記住這個分界，才不會在排查「Node 莫名變慢」時找錯方向。

## 為什麼是這個形狀

退一步看，event loop 的整套樣貌，都是從一個賭注、與一條紀律裡長出來的。

賭注是：**大量的連線，任一瞬間真正要 CPU 動手的極少，絕大多數都在等。** 正因如此，為「等」各養一條執行緒是浪費——所以把所有等待集中交給作業系統，用一條執行緒、一次 `epoll_wait` 照看上萬個 fd。紀律是：**這條執行緒絕不為任何單一工作停下來等。** 正因如此，所有會等待的操作都改成「登記 callback、立刻返回」；正因如此，那條執行緒醒著時只做「跑就緒事件的 callback」這件短活，睡著時把 CPU 整個讓給作業系統。

這個形狀買到的，是用極少的記憶體、零鎖、無資料競爭，服務海量並發 I/O 的能力——這正是 Nginx、Redis、Node 不約而同選它的理由。但它賣出去的，是對「每個工作都短、都肯讓出」的徹底依賴：一段不讓出的 CPU 重活會凍結全場，一條不收斂的 microtask 鏈會餓死 I/O，一個你忘了它存在的 4 條執行緒池會在高並發下成為隱形瓶頸。它的強大和它的脆弱，是同一個決定的正反面——把所有並發壓進一條執行緒，於是既得到了「沒有平行就沒有競爭」的乾淨，也背上了「一處卡住就處處卡住」的連坐。

下次你看到一個 Node 服務 CPU 沒滿、記憶體沒爆，尾延遲卻無端飆高，你會知道該往哪裡看：不是去加機器，而是去找那個忘了讓出的同步區段、那條停不下來的 microtask、或那個被悄悄佔滿的執行緒池。event loop 不是一套任意的工程慣例，而是「用一條執行緒服務海量等待」這個目標，被推到極致後，唯一站得住、也必須付出對應代價的那個形狀。
