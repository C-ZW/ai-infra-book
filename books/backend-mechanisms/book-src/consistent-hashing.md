# consistent hashing：節點進出時少搬一點資料

你有一個十台機器的快取叢集，前面用最直白的辦法決定每個 key 落在哪台：`node = hash(key) % 10`。它跑得又快又均勻，沒人有意見——直到某個週五下午，流量漲了，你決定加第十一台機器。你把設定從 `% 10` 改成 `% 11`，重新部署。

接下來那幾分鐘，監控圖上發生的事會讓你記很久。快取命中率不是「掉一點」，而是幾乎**垂直墜地**。原本九成以上的請求在快取裡就回掉了，現在絕大多數變成 miss，全部回頭去打後端資料庫。後端的 CPU 與連線數瞬間衝頂，延遲拉長，有些查詢開始逾時，逾時又觸發重試，重試把後端壓得更死。你只是加了一台機器想讓系統更快，結果差點把它打垮。

問題的根源是一行看似無辜的算式。`% N` 這個做法有一個你平常看不到、一旦 N 改變就會跳出來咬你的性質：**改變 N，幾乎每一個 key 的歸屬都會變。** 這一章要講的，就是一套專門為了「讓節點進出時，盡量少搬一點資料」而設計的機制——consistent hashing。它不是什麼花俏的最佳化，而是任何「節點集合會動、但資料／連線又黏在特定節點上」的系統，遲早都得面對的基本功。

## 為什麼 `% N` 一改就全盤皆動

先把天真做法為什麼破掉講清楚，因為理解了它的破法，才會理解後面那套環為什麼長那個樣子。

`hash(key) % N` 把每個 key 對應到 `0` 到 `N−1` 之間的一個桶。它的均勻性沒有問題——只要 hash 函數夠好，key 會大致平均散在所有桶裡。它的致命傷在於：**桶的編號是由「除以 N 的餘數」決定的，而 N 一變，整個餘數系統就重排了。**

具體看一個 key。假設 `hash(key)` 算出來是 `1003`。在 `% 10` 之下它落在桶 `3`；換成 `% 11`，`1003 mod 11 = 2`，它跳到桶 `2`。再看 `hash = 1004`：`% 10` 是桶 `4`，`% 11` 是桶 `3`。幾乎每一個 key 的餘數都變了，因為「除以 10 的餘數」和「除以 11 的餘數」之間沒有任何結構性的關聯——它們是兩套各自獨立的分桶規則。

這不是偶發，是數學上的必然。當你從 `% N` 換到 `% (N+1)`，一個 key 要能「留在原地」，必須剛好滿足 `hash mod N == hash mod (N+1)`，而這對隨機分布的 hash 值來說是小機率事件。粗略估，從 `% 10` 到 `% 11`，**大約只有十分之一的 key 留在原桶，其餘九成左右全部搬家**；更一般地，移除一台機器時，大約 `(N−1)/N` 比例的 key 會被重新指派。對一個無狀態的純計算服務，重映射只是「下次請求換一台處理」，無傷大雅。但對快取，重映射意味著 key 跑到一台沒有它的機器上——cache miss；對有狀態的儲存，重映射意味著那筆資料**實際上得從舊節點搬到新節點**——一場資料遷移風暴。

所以真正的需求不是「把 key 平均分散」——`% N` 早就做到了。真正的需求多了一個維度：**當 N 從 4 變 5、從 5 變 4，被迫改變歸屬的 key 要盡可能少。** 這個性質有個名字叫 consistent（一致性，指的是「函數的值域改變時，函數本身的變化最小」，不是資料一致性那個 consistency）。`% N` 在這個維度上是最差的——它幾乎把所有 key 都動了。

## 把節點和 key 放到同一個環上

1997 年，MIT 的 David Karger 等人在一篇原本是為了解決 web 快取「熱點」問題的 STOC 論文裡，給了這個問題一個漂亮到近乎簡單的答案。核心只有一句話：**不要讓 key 的歸屬取決於「節點的數量」，而要取決於「節點在一個固定空間裡的位置」。**

做法是這樣。想像一個首尾相接的環，環上的位置是 `0` 到 `2³²−1`（用一個 32 位元的 hash 輸出空間，繞一圈再回到 0）。現在做兩件事：

1. **把每個節點雜湊到環上。** 拿節點的識別字（IP、主機名、節點 ID）去算 hash，得到的值就是它在環上的位置。十台機器，就在環上釘下十個點。
2. **把每個 key 也雜湊到同一個環上。** 用同一個 hash 函數，算出 key 在環上的位置。

然後規則只有一條：**一個 key 從它自己的位置出發，沿著環順時針走，遇到的第一個節點，就是它的歸屬。**

```
 hash ring（順時針；key 歸屬於它順時針遇到的第一個 node）

                  0 / 2^32
                     |
            N_a *    |    * N_b
                \    |    /
                 \   |   /  <- key k1 lands here,
                  \  |  /      walks clockwise -> belongs to N_b
        N_d *------( ring )------* N_c
                  /  |  \
                 /   |   \
                /    |    \
            N_e *    |    * N_f
                     |
```

關鍵的轉變在這裡：**key 的歸屬不再是「算一個 0~N−1 的編號」，而是「在環上找順時針最近的節點」。** 為什麼這個小小的改寫，就解決了全盤皆動的問題？

看加一台機器會發生什麼事。你把新節點 `N_g` 雜湊到環上某個位置，它插進了 `N_b` 和 `N_c` 之間。現在環上的其他位置——`N_a`、`N_b`、`N_d`、`N_e` 全都**原地不動**，它們各自負責的那段弧也原封不動。唯一受影響的，是「原本順時針會走到 `N_c`、但現在會先撞上 `N_g`」的那一小段 key——也就是 `N_b` 到 `N_g` 之間的那段。這段 key 的歸屬從 `N_c` 改成 `N_g`，需要搬家；**環上其餘所有 key，歸屬完全不變。**

移除一台機器是對稱的：`N_c` 下線，它原本負責的那段弧，順理成章地併給它順時針的下一個節點。只有 `N_c` 那段 key 受影響，其餘不動。

這就是 consistent hashing 的全部魔法——把「歸屬」從一個會隨 N 重排的編號系統（餘數），換成一個固定空間裡的相對位置（環上順時針最近）。節點進出時，受擾動的只有環上局部的一段，而不是整個系統。

實作上「找順時針第一個節點」並不是真的繞環走一圈：把所有節點（之後還有 vnode）的位置存成一個排序好的陣列或 TreeMap，對 key 的 hash 做一次二分搜尋，取「第一個 ≥ 它的位置」（ceiling，找不到就環繞回開頭那個），單次查詢就是 O(log V)（V 是環上的點數）。

## 到底少搬了多少：一個算到底的對照

「少搬一點」聽起來很好，但工程上要的是數字。把開場那個快取叢集具體化來算。

設你有 **1,000 萬個 key**、目前 **4 個節點**，要加到 5 個。

**先看 `% N`。** 從 `% 4` 換到 `% 5`，一個 key 要留在原節點，得滿足 `hash mod 4 == hash mod 5`。對均勻分布的 hash 值，這個機率約是 `1/5`——也就是只有約兩成的 key 留下，**約 800 萬個 key（80%）改變歸屬**。對快取，這 800 萬筆下一次被存取時全是 miss，全部回打後端。這就是開場那場墜崖。

**再看 consistent hashing。** 加第 5 個節點時，新節點只會「切走」環上它落點前方那一段，把這段 key 從原本的鄰居手上接過來。理論上，把第 5 個節點加進一個已有 4 個節點的環，期望被移動的 key 比例正好是 **`1/(N+1) = 1/5 = 20%`**——也就是約 **200 萬個 key** 搬家，其餘 800 萬筆歸屬不變、快取照樣命中。

```
加第 5 個節點時，搬動的 key 量：

  % N            consistent hashing
  ----           ------------------
  ~800 萬 (80%)   ~200 萬 (20%)
  miss 風暴       只擾動環上一段
```

兩者差了四倍。換算成快取命中率，`% N` 讓你瞬間掉到一兩成命中，consistent hashing 只讓你掉到八成。對後端而言，前者是「突然要扛五倍流量」，後者是「多扛兩成」——一個可能讓你連環當機，一個你大概不會注意到。

順帶把這個 `1/(N+1)` 的直覺講透，因為它常被含糊地寫成「搬 K/N」。當環上已經均勻散著 N 個節點、你再丟進第 N+1 個，這個新節點隨機落在某處，它接管的是「它到順時針前一個鄰居之間」那段弧。整個環的長度被 N+1 個點切成 N+1 段，平均每段是全環的 `1/(N+1)`，而新節點正好接管其中一段的 key——所以期望搬動量是總 key 數的 `1/(N+1)`。從 4 加到 5 是 `1/5`，從 99 加到 100 是 `1/100`。**節點越多，加一台的擾動越小**——這正是大型叢集能持續滾動擴縮而不引發快取雪崩的根本原因。

## 環自己並不均勻：virtual node 的必要

到這裡好像太美好了，但有個陷阱藏在「節點隨機落在環上」這句話裡。把節點雜湊到環上是**隨機**的，而隨機不等於均勻。

設想只有 3 個節點。你把它們雜湊到環上，運氣不好的話，三個點可能擠在環的某一小段，留下一大段弧沒有節點。那段弧上的所有 key，順時針走很遠才撞到節點——於是某一台機器負責了環上 60% 的弧長，扛了六成流量，另外兩台閒著。節點越少，這種傾斜（skew）越嚴重。consistent hashing 解決了「搬動量」，卻在「負載均勻」上開了一個新洞。

更糟的是移除節點時的**負載突刺**。基本環上每個節點只有一個點，當 `N_c` 下線，它負責的**整段**弧全部順延給它順時針的單一鄰居 `N_d`。`N_d` 的負載可能瞬間翻倍——它本來扛自己那段，現在還要扛 `N_c` 那段。一個節點故障，把壓力全壓到另一個特定節點上，這正是把局部故障放大成連鎖故障的經典路徑。

解法是一個簡單但關鍵的轉折：**virtual node（虛擬節點，vnode）**。不要讓每個實體節點在環上只占一個點，而是讓它占**很多個**點——用「節點 ID + 序號」去雜湊，一個實體節點可以在環上撒下 100 到 200 個虛擬點，散布在環的各處。

```
基本環（每節點 1 點）          環 + vnode（每節點 N 個點，打散）
  N_a 占一大段弧                 N_a 的點散在環上 100 處
  -> 分布傾斜                    -> 每個點只負責一小段
  -> 移除時整段壓給單一鄰居      -> 每段都很小、且鄰居來自不同節點
```

vnode 一次解決兩個問題。**負載均勻**：因為每個實體節點變成環上一百多個小段的集合，由大數法則，這些小段加起來會非常接近 `1/N` 的均分——傾斜被抹平了。**移除時負載分散**：一個實體節點下線，它的一百多個小段各自順延給**環上不同位置的鄰居**，而這些鄰居分屬不同的實體節點。於是離開的負載被打散到所有剩餘節點，而不是全壓給單一一台——突刺變成平緩的均攤。

vnode 的數量是個有具體後果的旋鈕，不是隨便填。vnode 越多，環上的分布越均勻、傾斜越小：Dynamo 論文的實測給了一組常被引用的量級——用大約 **100 個 vnode**，每台機器的負載落在平均值上下約 **10%** 以內；加到約 **200 個 vnode**，可再收斂到約 **5%** 以內（這些是實測示意值，精確常數依度量方式與叢集規模而異）。太少（每節點個位數）仍然明顯傾斜；但也不是越多越好——每個 vnode 都是環上一筆要維護、要在節點間交換的 metadata，太多會吃記憶體、拖慢叢集狀態的傳播與重平衡。這個取捨真實到 Cassandra 在 4.0 版把 vnode 的預設值從 256 大幅調降到 16（2026-06）——因為實務上發現過高的 vnode 數讓重平衡的成本成超線性成長，反而成了負擔。所以「多少個 vnode」沒有萬用答案，是「均勻度 vs metadata 開銷」之間，依叢集規模拿捏的一條線。

## 環救不了的那種不均：熱 key

vnode 把負載抹平了，但這裡有一個非常容易誤判的邊界，值得單獨點出來，因為它讓很多人對 consistent hashing 抱了錯誤的期待。

consistent hashing（即使配上充足的 vnode）均勻分布的是 **key 空間**——它保證「不同的 key」會大致平均散到各節點。它**完全管不到「流量在 key 之間的分布」**。如果你的工作負載裡有一個超級熱的 key——某個爆紅商品的庫存、某個全站都在查的設定、某個瘋傳貼文的計數器——那麼這個 key 經過 hash 永遠落在環上同一個固定位置，永遠歸屬同一個節點。不管你撒多少 vnode、環多均勻，這一個 key 的所有流量都集中打在那一台機器上。它會被打爆，而旁邊的機器很閒。

這是一個原理性的邊界，不是調參能解的：環解的是「key 怎麼分配到節點」，解不了「某些 key 天生就比別人燙」。要對付熱 key 得換工具——把熱 key 拆成多個子 key 分散（`counter:1`、`counter:2`…）、把熱 key 複製到多個節點讓讀取分流、或用「有上限的負載」變體（bounded-load consistent hashing：給每個節點設一個負載天花板，某個節點被熱 key 撐到上限時，溢出的請求順延到環上下一個節點）。但這些都是在 consistent hashing 之外**額外**疊的機制。把「我用了 consistent hashing 所以負載一定均勻」當成定論，會在第一個爆紅事件時破功。

## 不是只有環這一種：同一個目標的幾條路

環是 consistent hashing 最廣為人知的形態，但「節點進出時少搬資料」這個目標，還有幾條設計各異的路，各自在不同壓力下勝出。理解它們，才算真的懂這個問題的全貌。

**rendezvous hashing（會合雜湊，又稱 HRW，highest random weight）**走的是完全不同的思路。它不維護任何環。對一個 key，它把這個 key 跟**每一個節點**配對算一次 hash——`hash(key, node_1)`、`hash(key, node_2)`…——然後把 key 指派給算出來**分數最高**的那個節點。它的 consistent 性質從哪來？當某個節點 `node_j` 下線，只有「原本最高分恰好是 `node_j`」的那些 key 受影響，它們改投「第二高分」的節點；其餘 key 的最高分節點根本沒變，歸屬不動。它不需要 vnode 就天生均勻分布（因為對每個 key 都獨立評分），也不需要維護環狀態。代價很直接：查一個 key 要對**所有 N 個節點**各算一次 hash，是 O(N)；節點數成千上萬時，這個 O(N) 就慢了。所以 rendezvous 適合節點數中等、又想避開 vnode 那套複雜度的場景。它由 Thaler 與 Ravishankar 在 1990 年代中期提出——1996 年的技術報告早於 Karger 環的 1997 STOC 約一年，與環約略同期、同樣經典。

**jump consistent hash（Google，Lamping & Veach）**把省記憶體推到極致。它是一個只有幾行、**完全不需要儲存任何狀態**的演算法，用 O(ln N) 的時間把一個 key 直接映射到 `[0, N)` 的某個桶，分布極均勻、加桶時也只搬 `1/N` 的 key。它的限制非常具體：**桶只能從尾端增減**——你可以把節點數從 N 加到 N+1，或從 N 減到 N−1，但**不能任意移除中間某個編號的節點**。這讓它不適合「任一節點都可能隨機故障」的場景，卻非常適合「節點是連續編號的 shard、只在尾端擴縮」這種儲存應用。它在「沒有環、沒有 metadata、純算術」這個維度上無人能及。

| 做法 | 加減節點搬動量 | 均勻度來源 | 查一個 key 的成本 | 代價／限制 |
|---|---|---|---|---|
| `hash(key) % N` | 幾乎全部（~80%+） | 天生均勻 | O(1) | N 一變就全盤重映射，會擴縮的叢集禁用 |
| 基本環（無 vnode） | 只搬一段（K/N 量級） | 不均勻、節點少時嚴重傾斜 | O(log V) 查環 | 移除時負載全壓單一鄰居 |
| 環 + vnode | 只搬一段、且均攤到所有節點 | vnode 數夠多才均勻 | O(log V) 查環 | vnode 是 metadata，過多吃記憶體拖重平衡 |
| rendezvous（HRW） | 只搬「最高分是它」的 key | 天生均勻、免 vnode | O(N) 對每節點評分 | 節點極多時慢 |
| jump consistent hash | 只搬 1/N | 天生均勻 | O(ln N) | 桶只能尾端增減，不能移除中間節點 |

落到現實（2026-06），環＋vnode 是分散式 KV 與快取叢集的主流——Amazon 的 Dynamo、Apache Cassandra、各家 memcached client 都走這條路；rendezvous 與 jump hash 則在「節點數適中、要避開環的維護成本」或「純尾端擴縮的 shard」這類更窄的場景裡出場。它們不是互相取代，而是「節點進出時少搬資料」這個問題在不同約束下的不同最佳解。

## 它在哪裡會壞、什麼時候根本不該用

把這套機制講到能在腦中重放之後，最後一段留給「它的脾氣」——哪些假設一旦不成立，它就會悄悄背叛你。

**最常見的誤用，是把它套在不需要它的地方。** consistent hashing 解的是一個很具體的問題：節點集合**會動**，而且資料／連線**黏在特定節點上**（快取、sticky 路由、有狀態分片）。這兩個條件缺一個，它就是徒增複雜度。如果你的服務是**無狀態**的——任一台都能處理任一請求，後端資料庫才是真相來源——那你根本不在乎請求落到哪台，一個簡單的 round-robin 負載均衡（見〈負載均衡：L4 與 L7、演算法與健康檢查〉）就夠了，硬套一個環只是給自己找維護負擔。如果節點數**永遠固定不變**，那 `% N` 的全盤重映射問題根本不會發生，`% N` 反而更簡單更快。consistent hashing 的價值完全建立在「節點會進出」這個前提上；前提不在，它買到的東西就不存在。

**第二個陷阱，是以為環本身就均勻。** 前面講過了，但它值得當成獨立的故障模式記住：沒有足夠的 vnode，節點少時環上的分布會嚴重傾斜，某台機器可能默默扛了四成流量。如果你看程式碼裡寫了 consistent hashing 就安心地假設負載均勻，沒去確認 vnode 數量夠不夠，那你可能正運行在一個一台過熱、其餘閒置的叢集上而不自知。均勻來自 vnode（或 rendezvous 那種對每節點獨立評分），不來自「環」這個結構本身。

**第三個，是熱 key。** 同樣前面說過，但它是最容易讓人對這個機制失望的邊界：環均勻分布的是 key 空間，不是流量。一個超熱的 key 會把它落到的那台節點打爆，環無能為力——這不是 bug，是這個機制的設計範圍之外。

**還有一個更隱蔽的：所有客戶端必須對「環長什麼樣」有一致的看法。** consistent hashing 的正確性建立在「大家用同一個 hash 函數、看到同一份節點清單、因此對同一個 key 算出同一個歸屬」。但節點是會進出的，這份「節點清單」本身就是一份需要在叢集裡傳播、且可能短暫不一致的分散式狀態。當一個節點剛加入、有些客戶端已經更新了環、有些還沒，同一個 key 在這個過渡窗口裡可能被不同客戶端算到不同的節點——一邊去寫新節點，一邊去讀舊節點。這份「成員視圖怎麼最終趨於一致」本身是另一套機制的工作（見〈gossip 與 anti-entropy〉），但你要知道：consistent hashing 給你「歸屬函數穩定」，**沒有**白送你「所有人此刻都同意當前的節點清單」。後者要另外掙。

退一步看，consistent hashing 的整個形狀，都是從開場那行 `% N` 的破法裡長出來的。因為「歸屬取決於節點數量」會在數量改變時全盤重排，所以它把歸屬改成「取決於固定空間裡的相對位置」——環因此而生。因為隨機落點不均勻、移除時會把負載全壓給單一鄰居，所以有了 vnode 把每個節點打散成環上的許多小段。因為環解的是 key 空間而非流量，所以熱 key 仍要另尋他法。它不是一個漂亮的資料結構炫技，而是「讓一個會擴縮的有狀態系統，在每次節點進出時只付最小的搬遷代價」這個務實需求，被逼出來的、剛好夠用的形狀。下次你看到一個快取叢集能無聲無息地滾動換機、命中率只抖一下就回穩，你會知道底下擋著的，正是這個環。
