# 衝突解決：LWW、版本向量與 CRDT

兩個人在飛機上各自編輯同一份共享筆記。一個把標題從「草稿」改成「Q3 提案」，另一個把同一行改成「第三季規劃」。兩台裝置都離線，各自把改動存進本地副本，都以為自己改的是最新版。飛機落地，兩台手機重新連網，把各自的版本推回伺服器——現在伺服器手上有同一個欄位的兩個值，而且**沒有任何時間順序能告訴它哪個才算數**。它們不是一前一後發生的，是平行發生的：誰都沒看到誰。

這就是衝突。它不是 bug，是一個你**主動允許**了某種架構之後必然要面對的後果：只要同一份資料可以在多個地方被獨立更新——多主複製、離線編輯後再同步、跨區雙寫——你就買下了「兩個並發更新撞在一起」這件事，而且系統得替你決定「最後算誰的」。更麻煩的是，它不能只是「挑一個」就算了：所有副本，不論訊息以什麼順序、重複幾次到達，都得**最終收斂到同一個確定的結果**。否則你不是有衝突，你是有兩份永遠對不齊的資料。

這一章要做的，是把「怎麼讓並發更新收斂」這件事拆到機制級。我們會看到三種答案——最後寫入者贏（LWW）、版本向量、CRDT——它們不是三個並列的選項，而是同一個問題被逼到不同程度時，逐步長硬的回答。每往下一層，你買到更強的收斂保證，付出更貴的代價。

## 先把「並發」這個詞講死

衝突解決的全部品質，押在一個判斷上：**你能不能分清「兩個更新是並發的」還是「一個因果上晚於另一個」。**

這兩件事看起來像，其實天差地遠。假設更新 A 把餘額設成 100，更新 B 把它設成 120。如果 B 是某個人**看到了 A 之後**才做的決定——他知道餘額是 100，決定改成 120——那 B 因果上晚於 A，根本沒有衝突：B 就該贏，它是基於最新狀態做的。但如果 A 和 B 是兩個人各自看著舊餘額 80、彼此沒看到對方就分別改的，那這才是真正的衝突：誰都沒比誰「更新」，系統憑什麼挑一個丟一個？

用 Leslie Lamport 在 1978 年給的「happens-before」關係來說：如果一連串訊息傳遞能讓 A 的效果傳達到 B 發生之前，A 就 happens-before B，兩者有因果順序；如果**兩個方向都傳不到**——A 影響不到 B，B 也影響不到 A——它們就是並發（concurrent）。真正需要解的衝突，只發生在並發的那一刻。一個因果上有先後的更新對，根本不該被當成衝突來「解」，直接讓晚的贏就對了。

於是衝突解決機制的第一道分水嶺就清楚了：**它有沒有能力區分並發和因果先後？** 只用一個時間戳排序的方案，沒有這個能力——它把所有更新硬塞進一條全序，看起來解決了衝突，其實是把「並發」這個概念整個碾平、假裝它不存在。能精確判斷並發的方案（版本向量、向量時鐘）則相反：它把真正的並發**暴露出來**，逼你正面處理，而不是讓某個值悄悄消失。理解了這條分水嶺，下面三種機制的取捨就全是它的推論。

## LWW：用一個時間戳碾平一切

最便宜、也最常見的答案是：給每個寫入打一個時間戳，衝突時保留時間戳大的那個。Last-Write-Wins，最後寫入者贏。

它的吸引力是實打實的。它**無狀態**——你不必記住「這個值之前是誰、什麼因果鏈改出來的」，只要比一個數字。它**不留 sibling**——任何時刻一個 key 就一個值，讀取端拿到的永遠是單一答案，不必寫合併邏輯。它在不可靠網路上也**天然冪等**：同一筆寫入重送幾次、亂序到達，比的都是同一個時間戳，結果不變。對「衝突罕見、而且丟掉並發的另一筆可以接受」的資料——某個 feature flag 的開關、使用者最後上線時間、一個無關緊要的偏好設定——LWW 簡單到沒有理由不用。

Cassandra 就是把 LWW 做進骨子裡的典型（2026-06）。它給每個寫入一個**微秒級時間戳**，衝突時保留時間戳最大的；而且粒度細到**逐 cell（逐欄位）**——同一列裡的不同欄位，各自帶自己的時間戳、各自獨立比較。時間戳來自 client 提供的時鐘，或在 client 沒給時由協調節點的時鐘補上。

但這個設計的甜頭和它的牙，是同一顆。LWW 的「最後」，是**時間戳的最後**，不是**真實因果的最後**。這兩者只在時鐘完美同步時才重合，而時鐘從來不完美同步。

## 一個會靜默吃掉你資料的時鐘

把 LWW 的陷阱手算一遍，你就再也不會把它當「安全的預設」。

兩個應用節點對同一個 key 寫入，節點 A 的時鐘比節點 B **快了 50 毫秒**——這在沒有嚴格時鐘同步的真實系統裡是家常便飯。事件這樣展開：

```
real time   actor    operation            stamped time (microseconds since epoch)
---------   -----    -----------------    ----------------------------------------
t = 0 ms    node A   write "stale-v1"     1_000_050_000   (A's clock = real + 50ms)
t = 20 ms   node B   write "correct-v2"   1_000_020_000   (B's clock = real time)
```

真實世界裡，A 上那筆**較舊**的值在 real time 0 先寫下，20 毫秒**後**使用者才在 B 上存了取而代之的正確新值（real time 20ms）。按真實時間與因果，B 那筆**較晚發生、也較新**，本該贏。但時間戳說的是另一個故事：A 的時鐘快了 50ms，它在 real time 0 打出的戳是 `1_000_050_000`；B 的時鐘準，它在 real time 20ms 打出的戳只有 `1_000_020_000`——physically 較晚的 B，stamp 反而比 physically 較早的 A **小**了 30,000 微秒。

LWW 只會做一件事：比這兩個數字，`1_000_050_000 > 1_000_020_000`，A 贏。於是**使用者剛在 B 上存好的正確新值，被 A 那筆時間戳較大、真實上卻較舊的寫入靜默覆蓋了**。沒有報錯，沒有日誌，沒有任何訊號。資料就這麼丟了——這是教科書級的 lost update，而且是無聲的那種。注意這裡時鐘偏移是**翻轉勝負**的那隻手：若兩台時鐘完美同步，A 的戳會是 `1_000_000_000`、小於 B 的 `1_000_020_000`，B 正確地贏；正是 A 多出來的那 50ms，把 stamp 順序拗成了真實順序的反面。

50 毫秒不是極端假設。NTP 同步在良好網路下能把時鐘對齊到幾毫秒內，但虛擬機暫停、GC 停頓、跨資料中心、或單純某台機器的 NTP 飄了，把偏差推到幾十甚至上百毫秒，都不是罕見事。只要偏差量級**超過**「兩個並發寫入之間的真實時間差」，LWW 就會挑錯贏家。它**永遠會收斂**——所有副本最後都同意是 A 那個值——但它收斂到了**錯的值**。收斂和正確是兩回事，LWW 只保證前者。

這就是為什麼「LWW + 牆鐘」在有並發寫入的系統裡是個有牙的陷阱：它的可靠是假的，它只是把「丟資料」這件事藏進了「總是有個贏家」的表象底下。

## 連平手都不安好心

LWW 還有一個更隱蔽的角落：**兩個寫入的時間戳剛好相等時，怎麼辦？**

微秒級時間戳要撞在一起，機率很低但不是零——尤其當 client 自己提供時間戳、或多個寫入由同一台協調節點在同一微秒內蓋章時。Cassandra 對這種平手的處理是：**比較 cell 的值本身，位元組序較大的那個值贏**（2026-06）。這個 tie-break 規則是確定性的——叢集裡每個副本都套同一條規則，所以至少能保證「所有副本對平手挑同一個贏家」，不會因平手而發散。

聽起來合理，直到你撞上一個它的副作用（CASSANDRA-14323 記錄的正是這個）。因為 Cassandra 的衝突解決是**逐 cell** 的，平手 tie-break 也是逐 cell 套用。設想兩筆 INSERT 打進同一個主鍵、時間戳相同，但寫的是不同的整列：

```
write 1 (ts=T):  (pk=1, col_a="z",  col_b="a")
write 2 (ts=T):  (pk=1, col_a="a",  col_b="z")

每個 cell 各自 tie-break（位元組序較大者贏）：
  col_a:  "z" vs "a"  → "z" 較大 → write 1 的 "z" 贏
  col_b:  "a" vs "z"  → "z" 較大 → write 2 的 "z" 贏

合成的最終列：(pk=1, col_a="z", col_b="z")
```

關鍵在最後那一列：`(col_a="z", col_b="z")` **既不等於 write 1（"z","a"）、也不等於 write 2（"a","z"）**——它是個從來沒有任何人寫過的、由 tie-break 規則**從兩筆寫入各挑一欄拼出來的縫合怪**。col_a 來自 write 1、col_b 來自 write 2，最終那列是一個原子性上根本不存在的幻影。對 LWW「挑一個贏家」的直覺來說這很違反預期：直覺會把它想成「平手時整列挑一筆」，但逐 cell 的設計下，它是各 cell 各挑各的。這不是 Cassandra 寫錯了，是「逐 cell LWW」這個設計選擇推到平手邊界時的必然長相——它把「列」當成一堆獨立 cell 的集合，而不是一個原子單位。

順帶一提，Cassandra 的 COUNTER 欄位是個例外：它**不走 LWW**，而是把並發的增量**相加合併**——這其實已經悄悄是下面要講的 CRDT 思路了。一個只能加的計數器，並發的「+3」和「+5」沒有「誰覆蓋誰」的問題，它們就該都算數、合成 +8。Cassandra 在計數器這個特例上，選擇了收斂而非覆蓋。

## 版本向量：不挑贏家，先把衝突誠實攤開

LWW 的根本問題是它**太急著決定**——它在還分不清「並發還是因果先後」的時候，就用一個不可靠的時鐘硬選了贏家。版本向量（version vector）反過來：它先把「這到底是不是真衝突」判準確了，是因果先後就自動讓晚的贏，是真並發就**兩個都留著**，把「該保留誰」的決定權交還給更懂這份資料語意的人。

機制是這樣的。每個副本維護一組計數器，形如 `{副本 → 計數}`。一個副本每接受一次本地寫入，就把**自己那一格**的計數加一。每個值都附帶它被寫下時的那份向量當作「因果脈絡」。要判斷兩個版本 X 和 Y 的關係，逐格比較它們的向量：

- 如果 X 的每一格都 **≥** Y 的對應格、且至少有一格嚴格大於——那 X 看過 Y 知道的所有事、還多知道一些，X 因果上**晚於** Y，X 贏，Y 可以丟。
- 反過來則 Y 晚於 X。
- 如果 X 有某格比 Y 大、Y 又有另一格比 X 大——**兩個方向都不包含對方**，它們就是並發。這時版本向量**拒絕**自作主張丟掉任何一個，把兩個值都保留成 sibling。

舉個三步就看懂的例子。一個 key 從 `{A:0, B:0}` 起步。副本 A 接到一次寫入，記為 `{A:1, B:0}`，值是 `v1`。這份 `v1` 同步到了 B，B 在它的基礎上又寫一次，記為 `{A:1, B:1}`，值 `v2`——`{A:1,B:1}` 每格都 ≥ `{A:1,B:0}`，所以 `v2` 因果上晚於 `v1`，`v1` 被乾淨地取代，沒有衝突。但若在 `v1` 同步過去之前，B 就**獨立地**對原值寫了一筆 `{A:0, B:1}` 的 `v2'`，那 `{A:1,B:0}`（v1）和 `{A:0,B:1}`（v2'）互不包含——A 那格 v1 大、B 那格 v2' 大——判定並發，**兩個都留**。系統不會假裝其中一個沒發生過。

這正是 Amazon Dynamo 在 2007 年那篇影響深遠的論文裡選的路：用版本向量追因果，並發時把多個版本一起回給 client，讓**應用層**決定怎麼合。對購物車這種資料，合併規則甚至很自然——兩個並發版本的購物車，取商品的**聯集**，使用者頂多看到一兩件他以為刪掉的東西又冒出來，但**絕不會有「加進購物車的東西憑空消失」**。Dynamo 寧可錯在「多保留」，不錯在「靜默丟失」——對電商，少賣一單比丟一單嚴重得多。

## 版本向量也不是免費午餐

但版本向量買到的「不靜默丟資料」，要付三筆帳，每一筆都實打實。

**第一，向量會隨副本數膨脹。** 每個值都拖著一份 `{副本 → 計數}`，副本越多，這份 metadata 越大。在副本集合穩定的系統裡還好，但若「寫入者」是大量的 client（每個 client 都往向量裡加一格），向量可能長到比資料本身還大。

**第二，sibling 不會自己消失，得有人收。** 版本向量只負責**偵測**衝突——它告訴你「這兩個是並發的」，但**怎麼合**它一個字都不管。並發時它丟給你兩個 sibling，最終還是得有 LWW、有應用邏輯、或有使用者出面挑一個或合一個。如果你開了多版本卻沒寫合併邏輯，sibling 會越積越多——這就是 Riak 圈子裡惡名昭彰的 **sibling explosion（sibling 爆炸）**：一個頻繁更新的 key，若 client 每次寫入都不帶上「我看到的因果脈絡」（causal context），系統無從判斷新寫入是要覆蓋舊版本還是又一個並發分支，只能一律當並發保留，sibling 數量失控膨脹。

Riak 對這個痛點的回應，本身就是個好故事。它後來讓新建的 bucket type 預設改用 **DVV（Dotted Version Vectors，帶點版本向量）**：給每個值額外標一個「dot」——標出**具體是哪一次更新造出了這個值**。有了 dot，重複的值能被精確識別並移除，sibling 的數量被壓到**正比於真正並發的更新數**，而不是無上界地膨脹（2026-06，Riak 2.0+ 推薦 DVV，且新建的 bucket type 預設 `allow_mult=true`）。這修掉的是普通版本向量的一個 false conflict 毛病——它有時會把其實不衝突的更新誤判成並發，DVV 用更精細的因果記號把這個誤判堵掉。

**第三，client 得乖乖回傳 context。** 版本向量要能判因果，靠的是 client 讀取時拿到向量、寫回時把它帶上。哪個 client 圖省事不帶，系統就失去了判斷它「看過什麼」的依據，只能保守地當並發——這又餵養了 sibling explosion。順帶一個 Riak 的硬規則：`allow_mult`（保留 sibling）和 `last_write_wins`（時間戳覆蓋）**兩個同時設成 true 是未定義行為**，不能混用——你得明確選一條路，要嘛保留並發、要嘛時間戳碾平，不能既要又要。

所以版本向量沒有「解決」衝突，它把衝突從「悄悄被時鐘吃掉」變成「明擺在應用面前，逼你處理」。這是進步——誠實的衝突遠勝靜默的丟失——但它把複雜度從基礎設施轉嫁給了應用程式。

## CRDT：把衝突解決設計進資料結構本身

LWW 用時鐘碾平衝突，版本向量把衝突攤開讓人來收。還有沒有第三條路——**讓衝突在數學上根本不可能不收斂，且不需要任何中央協調、不需要任何人來「解」？** 有，這就是 CRDT。

CRDT（Conflict-free Replicated Data Type，無衝突複製資料型別）由 Marc Shapiro、Nuno Preguiça、Carlos Baquero、Marek Zawirski 於 **2011 年**形式化。它的核心洞見是把整個問題**翻面**：不要事後去解衝突，而是把資料結構設計成「合併操作天生滿足某些代數性質」，於是不論更新以什麼順序、重複幾次到達，所有副本**必然**收斂到同一狀態。它換來的保證有個專名——**Strong Eventual Consistency（強最終一致，SEC）**：不只「最終會收斂」，而是**收到同一組更新的任意兩個副本，狀態必然相同**，收斂是即時的、無需等待某個全域協調點。

讓這件事成立的代數性質，是合併函式（merge）必須同時是：

- **交換律（commutative）**：`merge(a, b) = merge(b, a)`——順序不影響結果。
- **結合律（associative）**：`merge(merge(a, b), c) = merge(a, merge(b, c))`——怎麼分組不影響結果。
- **冪等（idempotent）**：`merge(a, a) = a`——同一個東西合併幾次都一樣。

這三條合起來意味著什麼？意味著訊息**亂序**沒關係（交換律）、**分批**沒關係（結合律）、**重複**沒關係（冪等）。而亂序、分批、重複，恰恰就是不可靠網路會對你做的全部壞事。把它們全部「免疫」掉之後，副本之間怎麼交換狀態、用 gossip 還是用什麼破爛通道，都不再影響最終結果——這就是 CRDT 對通道極度寬容的根源。

## 一個會數數的 CRDT，手算給你看

抽象說完，看一個最簡單的 CRDT——**G-Counter（grow-only counter，只增計數器）**——怎麼在三個副本上靠「取 max」收斂。場景：三台伺服器各自統計同一個貼文的按讚數，彼此非同步地用 gossip 對帳。

關鍵設計：計數器**不是一個數字**，而是一份 `{副本 → 該副本貢獻的增量}`。每台伺服器只增加**自己那一格**；某 key 的「總讚數」是所有格子相加。合併兩份 G-Counter 的規則簡單到不可思議：**每一格各取兩邊的較大值**。

```
初始：三個副本都是 {A:0, B:0, C:0}，總數 0

各自接收本地按讚（彼此還沒同步）：
  replica A 收到 3 個讚 → {A:3, B:0, C:0}
  replica B 收到 5 個讚 → {A:0, B:5, C:0}
  replica C 收到 2 個讚 → {A:0, B:0, C:2}

第一輪 gossip：A 把狀態送給 B，B merge（逐格取 max）：
  merge({A:0,B:5,C:0}, {A:3,B:0,C:0}) = {A:3, B:5, C:0}   → B 現在算出 8

同時 C 把狀態送給 A，A merge：
  merge({A:3,B:0,C:0}, {A:0,B:0,C:2}) = {A:3, B:0, C:2}   → A 現在算出 5

此刻三個副本「總數」各不相同：A=5, B=8, C=2 —— 還沒收斂

繼續讓狀態到處流（順序、重複都無所謂）：
  最終每個副本都看過 {A:3}, {B:5}, {C:2} 三筆貢獻
  逐格取 max → {A:3, B:5, C:2}  →  總數 3+5+2 = 10
```

收斂到 10，而且**不論這些 gossip 訊息以什麼順序到、重送幾遍**，結果都是 10。為什麼「取 max」這麼神？因為每一格只增不減，是個單調往上爬的數；取 max 就是「保留每個副本貢獻過的最高水位」。同一筆狀態重複收到？max 不變（冪等）。先收 B 再收 C 還是反過來？兩格互不干擾，結果一樣（交換律、結合律）。三條性質齊備，收斂是數學保證，不是運氣。

這也順手解釋了為什麼**不能**用「一個普通整數、各副本各自 +1、合併時相加」來做——那樣同一筆增量被 gossip 重送一次就會被重複加，1 個讚變成 2 個。把「每個副本的貢獻」拆開、用 max 合併，正是為了讓重複到達變成無害。要支援減少（按讚又取消），就用 **PN-Counter**：兩個 G-Counter，一個記所有增量（P）、一個記所有減量（N），當前值是 `sum(P) − sum(N)`——減量也只增不減地累加，同樣靠 max 收斂。

## 當「刪除」遇上「新增」：OR-Set 的非顯然選擇

計數器好辦，因為加法天生可交換。集合就微妙了，難點全在**刪除**。

設想一個共享標籤集合。副本 A 加入元素 `x`，副本 B 並發地**刪除** `x`（它本地有 `x`，決定移掉）。兩邊同步，merge 該怎麼判？`x` 到底在不在集合裡？這沒有「自然正確」的答案——它是個**語意選擇**。而最樸素的做法（集合就一份成員清單，加入=塞進去、刪除=拿出來）在這裡會直接壞掉：合併兩份清單若用聯集，刪除就永遠無效（被另一邊的存在蓋過）；若用交集，加入就永遠無效。普通的集合操作不滿足那三條代數性質，硬合就是發散。

**OR-Set（Observed-Remove Set，已觀察刪除集合）**給的答案是「**add-wins**（新增者勝）」，而它達成的手法是關鍵的非顯然之處：**每次「加入」都生成一個全域唯一的 tag（標籤），刪除時只能移除「你當下看得到的那些 tag」。** 一個元素是否在集合裡，看它「目前還活著的 tag 集合」非空與否。

回到剛才那一幕，慢動作重放：

```
replica A:  add x  →  x 帶 tag a1       （x 的活 tag: {a1}）
replica B:  本地已有 x（tag 集 {b1}），remove x
            → 只移除 B 當下看得到的 tag：{b1}

merge 兩邊：
  x 的所有 add tag = {a1, b1}
  被刪除（觀察到）的 tag = {b1}
  存活 tag = {a1, b1} − {b1} = {a1}  ≠ 空集
  → x 仍在集合裡（A 的新增贏了 B 的刪除）
```

`x` 活下來，因為 B 的刪除只能殺掉它**當時觀察到的** tag（`b1`），而 A 那次新增造的 `a1` 是 B 根本沒看過的——一個你沒觀察到的新增，你刪不掉它。這就是 add-wins：**並發的「加」與「刪」撞在一起時，加贏**。這個選擇對「協作編輯」「共享清單」這類場景通常是對的——一個人剛加進來的東西，不該被另一個沒看到這次新增的人的刪除動作抹掉。

代價藏在那些 tag 裡。刪除一個元素，不能真的把它的資料丟掉——你得**記住「這個 tag 被刪過」**（tombstone，墓碑），否則一個遲到的、攜帶舊 tag 的同步訊息會讓已刪的元素「詐屍」復活。於是 OR-Set 會累積 tombstone，這份只增的墓碑 metadata 是要花錢的。實務的緩解是**等到所有副本都確認看過某次刪除後，再壓實（compact）掉對應的 tombstone**——但「確認所有副本都看過」本身又需要一輪協調或夠長的等待。CRDT 把衝突從邏輯層趕走了，metadata 的開銷卻是它收的過路費。

## 兩種傳法：state-based 與 op-based

CRDT 內部還分兩大流派，差別在副本之間**傳什麼**，而這個差別直接決定了它對通道的要求——這是 owning 衝突解決必須講透的一層。

**state-based（CvRDT，convergent）**：副本之間交換**整個狀態**，用前面說的 merge 函式合併。merge 是個滿足交換、結合、冪等的「join」運算，狀態落在一個 **join-semilattice（聯結半格）** 上、更新只讓狀態單調「往上」走。前面 G-Counter 取 max、OR-Set 取 tag 聯集，都是 state-based。它的好處正是那三條性質帶來的：訊息重複、亂序、重送一概無害——所以它對通道幾乎**零要求**，扔給 gossip、anti-entropy 這種「最終會送到、但不保證順序也不保證不重複」的破爛通道也照樣收斂。代價是傳整個狀態較重；後來的 **delta-CRDT** 用「只傳狀態增量」來省頻寬，但保留同樣的數學保證。

**op-based（CmRDT，commutative）**：副本之間只傳**操作本身**（「+3」「加入帶 tag a1 的 x」），每個副本把收到的操作重放一遍。它要求並發操作**兩兩可交換**，傳輸量小得多——但天下沒有白吃的午餐，省下來的傳輸量是用**對通道的嚴格要求**換的：op-based 通常要求投遞層保證每個操作**恰好被送達一次、且滿足因果順序**（exactly-once、causal delivery）。為什麼？因為操作多半**不**冪等——一個「+3」重放兩次就變成 +6，一個「加入 x」可能還好但「移除某 tag」重放就可能亂套。state-based 的冪等 merge 把「重複」吸收掉了，op-based 沒有這層保護，只能要求通道別重複、別亂序。

一句話收束這個取捨：**state-based 把正確性全部放進 merge 函式（半格＋冪等），對通道幾乎無要求；op-based 把一部分正確性外包給投遞層（恰好一次＋因果序），換取更小的傳輸量。** 這也帶出一個隱性 bug 的常見來源——把一個 op-based CRDT 接到一個會重複投遞、不保證因果序的通道上，它會**悄悄不收斂**，而且因為 CRDT「理論上保證收斂」的名聲，這種錯特別容易被信任、特別難被發現。選對流派、配對通道，是用 CRDT 的真功夫。

## CRDT 不是萬靈丹，它有它塞不進去的形狀

CRDT 的收斂保證強得誘人，強到容易讓人想把它套到一切多副本場景上。但它有一道天生的牆：**它只能表達那些「合併操作能滿足交換、結合、冪等」的語意。** 凡是需要**全域不變量（global invariant）**的約束，CRDT 結構性地做不到。

最清楚的例子是「餘額不可為負」。設想一個帳戶餘額 100，用某種 CRDT 在兩個副本上維護。副本 A 並發地扣 80，副本 B 也並發地扣 80——各自看本地餘額都是 100，扣 80 後剩 20，都合法。但合併之後呢？兩筆扣款都得算數（CRDT 不會丟更新），合起來扣了 160，餘額變成 **−60**。每一步本地看都沒違規，合併後全域不變量被突破了。CRDT 沒有任何辦法在合併時「拒絕」其中一筆——它的本事是讓一切收斂，不是讓某些操作失敗。「庫存不可超賣」「用戶名全域唯一」是同一類問題：它們都要求一個**跨所有副本的全域約束在每個瞬間都成立**，而這種「拒絕某些並發組合」的能力，CRDT 的代數結構裡根本沒有位置放它。

這類約束需要的不是收斂，是**協調**——某個能在「准許這筆扣款之前，先確認沒有別人並發地把餘額用光」的全域序點。那是共識、是單主排序的活（見〈共識：讓一群會當機的機器同意一件事〉、〈複製策略：單主、多主、無主〉）。把要強約束的資料硬塞進 CRDT，你會得到一個「永遠收斂、但收斂到一個違反業務規則的狀態」的系統——比不收斂更難察覺，因為它看起來一切正常。

## 為什麼是這幾個形狀

退一步看這三種機制，它們不是三個任意的工程選項，而是「並發更新如何收斂」這個問題被逼到不同嚴格程度時，依次長出的硬骨頭。

**LWW** 賭的是「衝突罕見、丟一筆無所謂」，於是它敢用一個時鐘把一切碾成全序——最便宜、最簡單，但它分不清並發和因果，會在時鐘偏移時靜默吃掉本該保留的寫入。當「丟資料不可接受」時，你被迫往上爬一層。**版本向量**換來了「精確區分並發、絕不靜默丟」，但它只負責偵測、不負責解決，把合併的複雜度和 sibling 的開銷轉嫁給應用層。當「連讓人來收 sibling 都嫌貴、要全自動無協調收斂」時，你再爬一層。**CRDT** 用代數性質把收斂變成數學保證、把通道要求降到最低，但它只能表達能塞進半格的語意，碰到全域不變量就撞牆，得退回去找協調。

所以它們不是「哪個最好」的競賽，而是一條取捨的階梯：你願意為「不丟資料」付多少 metadata、為「自動收斂」付多少結構限制、為「強約束」付多少協調延遲。沒有一層是免費的——LWW 用正確性的風險換簡單，版本向量用 metadata 和應用複雜度換誠實，CRDT 用表達力的限制換無協調的收斂。

而所有這一切的源頭，都回到最開頭那個被講死的詞：**並發**。只要你允許同一份資料在多處被獨立更新，你就製造了並發；只要有並發，你就得在某處付出代價去把它收斂回單一真相。這三種機制，是人類面對「兩個沒看到彼此的更新撞在一起」這道題，至今最站得住的三種答案——而選哪一個，取決於你那份資料到底經不經得起被一個快了 50 毫秒的時鐘悄悄改寫。
