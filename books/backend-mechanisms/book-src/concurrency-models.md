# 並發模型比較：thread、event-loop、actor、coroutine、CSP

先看一個你大概見過很多次的瞬間。同一份需求——「一台機器要同時服務上萬個連線，每個連線大半時間在等資料庫或外部 API 回來」——交給三個不同的團隊，你會收到三份長得完全不一樣的程式碼。一份開了一池執行緒，每個連線一條，程式碼是線性的 `read` 然後 `write`；一份是單執行緒的事件迴圈，滿屏的 `async`／`await` 與回呼；還有一份每個連線丟一個 goroutine，看起來像阻塞式同步寫法，但底下是另一回事。三份都能跑，三份的記憶體佔用、CPU 利用、崩潰時的脾氣卻差到一個量級。

差別不在誰寫得好，而在他們選了不同的**並發模型**。並發模型不是某個函式庫，而是一個語言或 runtime 對三個根本問題的整套回答：**並發的單元是什麼？這些單元之間怎麼通訊？切換由誰決定？** 主流的五個答案——thread、event-loop、actor、coroutine、CSP——不是五種隨意的風味，而是同一個老問題在不同壓力下逼出來的不同妥協。這一章要做的，是把這五個模型拆到你能向另一個工程師講清楚「為什麼換個模型，同一個負載的資源消耗會差一百倍」，以及「為什麼換模型救不了你真正的瓶頸」。

並發本身為什麼需要，以及它和平行的區別，是另一個主題（見〈並發不是平行：兩個常被混用的詞〉）。這裡的前提很短：**並發是「同時推進很多件事」的結構，平行是「同一瞬間真的多個核心一起跑」的執行**。五個模型在「並發結構」這條軸上各有姿態，在「要不要平行」這條正交軸上又各有立場——把這兩條軸分清楚，五個模型才不會糊成一團。

## 兩條軸，把五個模型釘在座標上

把五個模型攤平比較之前，先建立座標系，否則它們看起來只是五個並列的名詞。真正區分它們的，是兩條彼此正交的軸。

**第一條軸：通訊靠共享記憶體，還是靠傳訊息。** 兩個並發單元要協作，總得交換狀態。一種做法是讓它們**共用同一塊記憶體**——A 寫一個變數、B 讀同一個變數，省事、零拷貝，但只要兩者可能同時碰它，就有了 race condition，得靠鎖或原子指令把臨界區圍起來（共享狀態的競態怎麼長出來、怎麼補，是另一個主題，見〈race condition 與原子性：跨 DB、cache、runtime〉）。另一種做法是**誰都不碰別人的記憶體，只互相寄訊息**——A 把資料複製一份送進一個信箱或管道，B 從那頭收。沒有共享，就沒有資料競爭；代價是訊息要拷貝、要排隊、要面對「信箱滿了怎麼辦」。

**第二條軸：切換由 OS 搶佔，還是靠單元自己協作讓出。** 一個並發單元跑著跑著，總得停下來讓別人跑。**搶佔式（preemptive）**是 OS 用時鐘中斷強行打斷你：每個工作以為自己獨佔 CPU，被切走時毫不知情，公平性由核心兜底，但每次切換要保存與還原暫存器、汙染快取，代價在微秒等級。**協作式（cooperative）**是工作跑到一個「我現在沒事做了」的點（等 I/O、`await`、`yield`）主動交出控制權，runtime 在單一執行緒上順勢推進下一個；切換輕到接近一次函式呼叫，但有一個致命前提——**只要有一個工作賴著不讓（跑進一個沒有讓出點的死迴圈），整條線就一起凍住**。

五個模型，就是這兩條軸上的五個落點：

| 模型 | 並發單元 | 通訊 | 切換 | 代表 runtime |
|---|---|---|---|---|
| thread | OS 執行緒 | 共享記憶體＋鎖 | OS 搶佔 | Java、C++、POSIX threads |
| event-loop | callback / Promise | 共享單執行緒記憶體 | 協作（I/O 讓出） | Node.js、Nginx、Redis |
| actor | actor（私有狀態） | 非同步訊息（mailbox） | runtime 排程 | Erlang/Elixir、Akka/Pekko |
| coroutine | 協程 / goroutine | 視語言（channel 或共享） | 協作（Go 另有 SIGURG 搶佔兜底） | Go、Kotlin、Python asyncio |
| CSP | 行程（如 goroutine） | 同步 channel | 協作＋搶佔 | Go、occam |

這張表是地圖，不是結論。接下來逐一走進每個落點，看它在那個位置上**買到什麼保證、欠下什麼債**——因為那才是你真正要在腦中重放的東西。

## thread：最貼近硬體，也最容易割傷自己

執行緒模型是最老、也最直白的一個：並發單元就是 OS 執行緒，它們**共用同一個位址空間**，OS 用搶佔式排程在多核上把它們攤開跑。它最貼近硬體，給你的是**真平行**——四核機器上四條執行緒可以真的同時前進，這是 event-loop 在單執行緒下永遠拿不到的。對 CPU 密集、需要榨乾每一顆核心的工作，這是天然的家。

它的麻煩全部來自那個「共用同一塊記憶體」。一旦兩條執行緒可能同時碰同一個變數，你就得自己用鎖把臨界區圍起來——而鎖是 bug 的溫床。鎖太粗，並發度掉回近乎序列化；鎖太細，你開始面對死結（A 持鎖 1 等鎖 2、B 持鎖 2 等鎖 1，兩邊永遠互等）、活鎖、優先權反轉這一整個動物園。更陰險的是，這類 bug 取決於執行緒交錯的**精確時序**——它在你的測試機上一萬次都對，上線在某個負載尖峰下錯一次，且幾乎無法穩定重現。

還有一筆常被低估的成本：**執行緒本身很貴**。一條 OS 執行緒預設要配一塊堆疊，平台執行緒典型約 1 MB（可調，但很少有人調到更小）。要服務一萬個連線、每連線一條執行緒，光堆疊就吃掉約 10 GB，再加上一萬條執行緒讓 OS 排程器在它們之間切換的開銷——這就是經典的 C10K 問題撞牆的地方。執行緒模型的甜蜜區，是**連線數有限（數百量級）、每個工作邏輯複雜、且真的需要多核算力**的場景；連線數一上萬、且大多在閒等，它會被自己的重量壓垮。

值得記住的是，「執行緒貴」這個前提在 2026 年正被一個有趣的發展鬆動。Java 在 Java 21（2023-09 釋出）把 **virtual threads（Project Loom）** 帶成正式特性，到 2026 已是成熟可用：你照舊寫 `new Thread()` 那套線性阻塞風格的程式碼，但 JVM 把成千上萬個 virtual thread **多工到一小池真正的 OS 執行緒（carrier thread）上**——當一個 virtual thread 卡在 I/O，JVM 把它從 carrier 上卸下、換另一個上來，carrier 這條真 OS 執行緒永不阻塞。這正是 coroutine 那套 M:N 多工的思路，繞了一圈回到了當年發明 thread-per-request 的語言。它沒有改寫「共享記憶體＋鎖」這條軸（virtual thread 之間仍共享記憶體、仍需要同步），但把「執行緒貴到連線數一高就 OOM」這個歷史包袱拆掉了——這條軸的取捨，正在我們眼前移動。

## event-loop：用「不平行」換「不會資料競爭」

把執行緒模型的痛點倒過來想，就得到 event-loop。它的賭注是：**既然大量連線多半時間在閒等，那我根本不需要多條執行緒——一條就夠**。一條主執行緒不斷輪詢「有沒有就緒的事件」，有就跑對應的回呼，沒有就把 CPU 讓給 OS 去等。所有會等待的操作都登記一個回呼後立刻返回，等資料就緒 OS 通知，迴圈再回頭跑那個回呼。event loop 本身的相位細節（libuv 的 timers、poll、check…）與它怎麼把阻塞操作委派出去，是另一個主題（見〈event loop：單執行緒怎麼同時做很多事〉）；這裡只取它在模型光譜上的位置。

它落在座標系的這個點：**共享記憶體，但協作式切換、且只有一條執行緒在跑**。這個組合送出一份 thread 模型夢寐以求的紅利——**沒有資料競爭**。因為任一瞬間只有一個回呼在動，臨界區天然互斥，你寫一段「讀計數器、加一、寫回」不必上任何鎖，中間不可能有別的回呼插進來。這是 event-loop 最被低估的隱性好處：它不是「也能避免」競態，而是在模型層面**讓你寫的單一回呼天生不可能被別人切開**。

但這份「不平行」的保證有刺。第一，它**只用一條執行緒**，所以一個請求只要做了一段不讓出的同步 CPU 重活，整條線就凍在那裡——此刻在線的所有連線一起被延遲，p99 瞬間爆炸。event loop 不怕「等」，最怕「算」；CPU 重活得外送到別處（見〈thread pool 與 worker threads：CPU 密集怎麼辦〉），否則模型本身保護不了你。第二，「不平行」意味著一條 event loop 只吃得到一顆核心；要用滿多核，得開多個行程，每個行程一條 event loop，這又把「跨行程怎麼共享連線狀態」的問題請了回來。

還有一個常被誤判的死結。很多人以為「event-loop 無鎖所以不可能死結」——傳統的鎖死結確實沒有，但兩個 async 流程互等對方的 Promise 照樣會永遠卡住：A 在 `await` B 產出的結果，B 又在 `await` A 產出的結果，誰都不會先完成。沒有鎖，不代表沒有「互相等待的環」；死結的本質是依賴成環，鎖只是製造環的其中一種方式。

## actor：每個單元抱著自己的狀態，只靠寄信溝通

actor 模型從另一個方向解決「共享記憶體會割傷人」這件事——它**直接廢掉共享**。並發單元叫 actor，每個 actor 私藏自己的狀態，**誰都不能直接讀寫別人的狀態**；要互動，只能往對方的 mailbox（信箱）丟一封非同步訊息。actor 一次處理一封信，處理時可以改自己的狀態、可以送信給別的 actor、可以生出新的 actor。因為狀態被一個 actor 獨佔、且它一次只處理一封信，它內部根本沒有並發——**競態被結構性地消滅了，不是靠鎖壓制的**。

這個設計的歷史比多數人以為的更早。actor 模型由 Carl Hewitt、Peter Bishop 與 Richard Steiger 在 **1973** 年的論文〈A Universal Modular ACTOR Formalism for Artificial Intelligence〉提出，後由 Hewitt 的學生 **Gul Agha 在 1986** 年的著作《Actors: A Model of Concurrent Computation in Distributed Systems》完整形式化。它最成功的工程化身是 Erlang，以及建在 BEAM 虛擬機上的 Elixir。

actor 真正的殺手鐧不在「沒有競態」，而在**容錯**。因為 actor 之間沒有共享狀態，一個 actor 崩潰**不會弄髒任何別人**——它的爛狀態都鎖在自己身上。於是 Erlang 世界發展出一套違反多數人直覺的哲學：**let it crash（讓它崩潰）**。不要在每個 actor 裡塞滿防禦性的 try-catch 去硬撐一個已經進入未知狀態的流程，而是讓它乾脆崩掉，由一個**監督樹（supervision tree）**裡的上層監督者偵測到子 actor 死亡，按預設策略把它重啟回一個乾淨的初始狀態。「把壞掉的東西重啟成乾淨的」比「把壞掉的東西修回正確的」可靠得多——這套思路讓 Erlang 系統做出了電信級的可用性。而且因為通訊一律是「往某個位址寄訊息」，那個位址在本機還是在另一台機器，對程式碼幾乎沒差——actor 模型**天生對分散式友善**，這是共享記憶體模型給不出的。

代價也都長在「寄信」這個動作上。訊息要拷貝、要走 mailbox，比起直接讀一個共享變數，**延遲更高**。mailbox 預設無界時，一個生產過快、消費過慢的 actor 會讓信件無止盡堆積，吃光記憶體——這正是背壓要解的問題（見〈背壓：當下游跟不上〉）。而全非同步的訊息流也讓**除錯更難**：沒有一條清晰的同步呼叫堆疊讓你回溯「這個錯是誰引發的」，你面對的是一張訊息亂飛的圖。

這裡有一個 2026 年必須講清楚的現實，否則會給讀者過時的地圖。JVM 世界最知名的 actor 框架 **Akka**，已經**不再是開源**了：Lightbend 在 **2022-09** 把 Akka 從 Apache 2.0 改成 **Business Source License（BSL）**——可自由用於開發與測試，但年營收超過門檻的公司要**production 使用得付商業授權**（BSL 授權的版本三年後回退為 Apache 2.0）。社群隨即 fork 出 **Apache Pekko**（基於 Akka 2.6.x、採 Apache 2.0、2023-07 釋出 1.0），目前在 Apache 基金會下維護。所以若你在 JVM 上要 actor 模型、又要避開商業授權，2026 年的選項是 Pekko 而非新版 Akka；Erlang/Elixir 那條路（BEAM）則一直是開源、且 actor 是語言的一等公民，不是外掛框架。

## coroutine：寫起來像阻塞，跑起來不卡執行緒

coroutine（協程）瞄準的是另一組讓人左右為難的取捨：event-loop 高效但回呼風格難寫、thread 好寫（線性程式碼）但太貴。協程想兩邊都要——**程式碼長得像同步阻塞的線性風格，執行時卻不真的卡住執行緒**。

它怎麼做到？協程是一種可以**在中途暫停、之後從暫停點接著跑**的函式。當協程跑到一個會等待的點（等 I/O、`await`、`yield`），它不阻塞底層執行緒，而是把自己掛起、把控制權交還給 runtime；runtime 在同一條（或少數幾條）OS 執行緒上挑下一個就緒的協程接著跑。等那個 I/O 好了，原協程從掛起的那一行**無縫續跑**。於是你寫的是 `data := fetch(); use(data)` 這種一眼看懂的線性碼，編譯器與 runtime 在背後把它拆成「掛起—讓出—恢復」的狀態機。可讀性是 thread 的，效率是 event-loop 的。

代表是 Go 的 goroutine、Kotlin 的 coroutine、Python 的 asyncio。其中 Go 的排程值得拆開看，因為它示範了協程模型怎麼把幾個取捨同時吃下來。Go 用的是 **M:N（GMP）排程**：把海量的 **G**（goroutine）多工到少量的 **M**（OS 執行緒）上，中間隔一層 **P**（邏輯處理器，數量決定真正能平行的 goroutine 數）。`GOMAXPROCS`——也就是 P 的數量——預設等於可用 CPU 核心數；而從 **Go 1.25** 起這個預設變成 **cgroup 感知**：行程跑在有 CPU limit 的容器裡時，預設取「核心數」與「CPU limit」中較小的那個，避免在 Kubernetes 裡誤以為自己有整台機器的核心、結果被 cgroup 節流到頻繁暫停。

協程的軟肋藏在那個「協作式讓出」裡。協作式的天生風險是**一個不讓出的工作卡死全場**——若一個協程跑進一個既不等 I/O、又沒有任何讓出點的純 CPU 死迴圈，它會霸佔底層執行緒、餓死同一條執行緒上的其他協程。Go 用一個漂亮的手段補上這個洞：**訊號式非同步搶佔**。從 **Go 1.14** 起，一個叫 sysmon 的背景監控執行緒會盯著每個 goroutine，發現某個 goroutine 連續跑超過約 **10 毫秒**還沒讓出，就對它所在的 OS 執行緒送一個 **SIGURG** 訊號，強制把它打斷、切走。（選 SIGURG 是因為它不干擾既有的除錯器與其他訊號用途。）這把純協作式「跑死迴圈就卡死排程」的洞補成了「協作為主、必要時搶佔兜底」的混合模型——這也是為什麼 Go 在「高並發 ＋ 夾雜可觀的 CPU 計算」這個 event-loop 和純 thread 都不甜的中間地帶特別好用。

另一個常被當成口號的軟肋是「協程裡的阻塞系統呼叫」。協程在**非阻塞**操作上才能優雅讓出；一旦它呼叫一個**真阻塞**的 C 函式庫或系統呼叫（runtime 沒辦法在那個點插手讓出），這條協程就會佔住底層的 OS 執行緒不放。Go runtime 多半會察覺並補開一條新的 M 來救場，讓其他 goroutine 繼續跑，但這條協程在那段期間就**不是免費讓出**了——「協程永遠不卡執行緒」是個需要加星號的說法。

## CSP：把 channel 變成第一公民

CSP 和 actor 是表親——都靠傳訊息、都迴避共享記憶體——但它們的**一等公民不同**，這個差別細到常被混為一談，卻決定了兩種程式碼長得完全不一樣。

CSP 由 Tony Hoare 在 **1978** 年的論文〈Communicating Sequential Processes〉（CACM 21, 1978）提出。它的核心信條被 Rob Pike 的 Go 諺語濃縮成一句口號：「**Do not communicate by sharing memory; instead, share memory by communicating（不要用共享記憶體來通訊，要用通訊來共享記憶體）**。」差別在這裡：actor 的一等公民是**行為者**——你把訊息送給「某個具名的 actor」的 mailbox，是非同步的，丟完就走，actor 自己慢慢從信箱取。CSP 的一等公民是**channel（通道）**——你把值送進「一個具名的 channel」，**不指定誰來收**，任何在那個 channel 那頭等著收的行程都可能拿到。在經典 CSP 裡，這個交換是**同步交會（rendezvous）**：送方和收方必須同時到位才完成這次傳遞，沒人在等的話送方就停在那裡——這天然帶來背壓。

值得釐清一個歷史細節，免得把這條脈絡講歪：Hoare 1978 年的原始 CSP 其實是讓**具名的行程**之間直接同步收送（行程 A 寫 `B!x` 送給行程 B、B 寫 `A?y` 收），**channel 作為第一公民、可以被當值傳來傳去**，是後來 occam 等語言、以及 Go 把它發揚的形態。我們今天說「CSP 風格」，多半指的是這個被 Go 普及的「goroutine ＋ channel」版本，而不是 1978 年論文的字面語法。

CSP 風格最閃光的地方是**管線**。把一串處理階段（讀取→解析→轉換→寫出）各自做成一個行程，用 channel 把它們串起來，每段只管「從上游 channel 收、往下游 channel 送」。如果 channel 是**有界**的，背壓就自然成立：下游慢了，它的輸入 channel 塞滿，上游送不進去就被迫停下來等——整條管線自動把速度對齊到最慢的那一段，不需要你額外寫任何流量控制。這是生產者—消費者問題最乾淨的一種表達。

它的陷阱也都圍著 channel：**無界 channel 會悄悄丟掉背壓**。一旦你給 channel 一個很大的緩衝（或無界），上游就不再被下游頂住，可以一路狂送，訊息在緩衝裡越積越多——你以為自己在做 CSP，其實退化成了一個會撐爆記憶體的無界佇列。還有 `select`（同時在多個 channel 上等）的邏輯出了名地容易寫錯，漏掉一個 case 或忘了關閉 channel，就是 deadlock 或 goroutine 洩漏。

這裡得補一個關於 Go 的精確說法，否則容易過度簡化：**Go 提供 channel（CSP 風），但也提供 `sync.Mutex` 和共享記憶體（thread 風），實務上兩者常混用**。Go 的並發**不等於純 CSP**——channel 是它的招牌，但語言本身並不禁止你用鎖保護共享狀態，很多高效能 Go 程式正是混著用的。把「Go ＝ CSP」當定論，會錯過 Go 並發模型真正的彈性。

## 同一個負載，三種模型，差一個量級

抽象講完，把它釘到一個帶數字的具體場景上，這些取捨才會從腦中變成肌肉記憶。

假設你要在一台機器上服務 **50,000 個並發長連線**——想像一個聊天或推播伺服器，每個連線大半時間只是掛著、偶爾收發一小段資料。三種模型分別會付出什麼？

**thread-per-connection。** 每個連線一條 OS 執行緒，每條執行緒堆疊預設約 1 MB。光堆疊就是 `50,000 × 1 MB ≈ 50 GB`——還沒算上五萬條執行緒讓 OS 排程器在它們之間切換、汙染快取的開銷。這個方案在這個量級上根本不成立，記憶體先 OOM。

**event-loop（如 Node）。** 一條主執行緒，靠 epoll/kqueue 一次照看五萬個 fd，這五萬個連線在核心眼裡只是一張就緒列表。記憶體以**數百 MB** 計，比 thread 方案省了兩個數量級。代價是它只吃得到一顆核心，而且只要任何一個請求做了一段沒讓出的 CPU 重活，這五萬個連線會**一起**凍住。

**goroutine（Go）。** 每個 goroutine 初始堆疊僅約 **2 KB**（按需成長，不夠時 runtime 配一塊更大的連續堆疊、把內容搬過去），五萬個約從 **100 MB** 起跳。而且 GMP 排程把這些 goroutine 攤到所有核心上跑——拿得到多核，又有 SIGURG 搶佔兜底，不怕單一 goroutine 卡死全場。

把三個數字並排：

```
50,000 concurrent connections, stack memory (lower bound):

  thread-per-conn   ~50 GB   (50000 * 1 MB)    multi-core, ctx-switch storm, OOM risk
  event-loop        ~hundreds of MB             1 core only, CPU-heavy stalls all
  goroutine         ~100 MB  (50000 * 2 KB+)    multi-core, preemption-safe
```

從約 50 GB 到約 100 MB，是**五百倍**的記憶體差距；從單核到多核，又是一條正交的維度。同一個負載，僅僅換並發模型，資源消耗和擴展性就跨越一個量級——這就是為什麼模型選型不是品味問題，而是會直接決定一台機器扛得住多少連線的工程決策。

但這個 worked example 也藏著一個必須立刻補上的反面教訓。

## 模型救不了你選錯的瓶頸

上面那個漂亮的對比，前提是這個負載的瓶頸在**等待**（I/O 密集）。如果你把場景換成「每個請求要做 200 毫秒的純 CPU 計算」，再怎麼換並發模型都救不了你——因為瓶頸根本不在「同時推進很多件事」的結構上，而在「算力不夠」這件事上。event-loop 會被第一個請求的計算卡死全線；goroutine 雖然能攤到多核，但核心就那幾顆，五萬個 CPU 重活還是得排隊；換成 actor 也一樣，actor 處理那封信時照樣要燒那 200 毫秒的 CPU。

這是並發模型最常被誤用的地方：**把「換個並發模型」當成效能問題的萬靈丹**。並發模型決定的是「並發結構長什麼樣、省不省資源、好不好寫、崩潰時的脾氣」——它**不增加算力**。在動手換模型之前，唯一要先回答的問題是：我的瓶頸在「等」還是在「算」？瓶頸在等，換成更省資源的並發結構（event-loop、coroutine）立竿見影；瓶頸在算，你需要的是更多核心、更快的演算法、或把計算移出請求路徑，跟選哪個並發模型沒有關係（這條「先分清等與算」的判準，正是並發與平行之別的實務後果，見〈並發不是平行：兩個常被混用的詞〉）。

順帶拆掉另外兩個常見的混淆。其一，「**goroutine 是免費的**」——每個初始 2 KB 確實便宜，但不代表可以無限開。開幾百萬個仍會吃掉可觀的記憶體與排程成本；更隱蔽的是，如果每個 goroutine 各自持有一個資料庫連線，你會在連線數遠未到「記憶體上限」之前就先耗盡連線池（連線為什麼不能無限開、池怎麼成為瓶頸，見〈連線池：為什麼不每次都重開連線〉）。輕量不等於無限。其二，「**actor 和 CSP 是同一回事**」——它們確實都傳訊息、都避共享，但前面已經拆過：actor 送給具名的行為者（非同步、有 mailbox），CSP 送進具名的 channel（不指定誰收、經典形態是同步交會）。一個以「誰」為中心，一個以「管道」為中心，寫出來的程式碼骨架完全不同。

## 為什麼是這幾個形狀

退一步看，五個模型不是五種任意的設計品味，而是同一個問題——「**多個並發單元怎麼協作而不互相踩爛**」——在那兩條軸上窮舉出的幾個穩定解。

順著「通訊」這條軸，你只有兩個根本選擇：要嘛**共享記憶體**（快、零拷貝，但得自己用鎖管好競態——thread、event-loop 都在這一側），要嘛**傳訊息**（沒有競態，但要付拷貝與排隊的代價——actor、CSP 都在這一側）。順著「切換」這條軸，也只有兩個根本選擇：要嘛**搶佔**（公平、不怕賴皮，但切換貴——thread），要嘛**協作**（切換極輕，但一個不讓出就卡死它所在的那條執行緒——單執行緒模型即全場，多執行緒協程則餓死同一條 carrier 上的協程；event-loop、coroutine、CSP）。而 coroutine 之所以在當代特別受歡迎，正因為它站在一個聰明的中間點：用協作式切換拿到輕量並發與線性可讀性，再用搶佔（Go 的 SIGURG）兜住協作式那個「賴皮卡死」的致命洞——它不是發明了第三條軸，而是把兩條軸上各取所長的妥協做到了平衡。

所以下次你看到一份程式碼選了某個並發模型，你看到的不是一個品味標籤，而是一組已經做完的取捨：它在那兩條軸上站的位置，決定了它無鎖還是要鎖、省記憶體還是吃記憶體、怕不怕一個請求卡死全場、容不容易在崩潰時自癒、好不好往分散式長。模型沒有最好的，只有「在你這個負載的瓶頸與壓力下，哪組取捨划算」。而要做這個判斷，你得先看清自己站在哪兩條軸的哪個落點上——這正是把五個名詞拆回兩條軸之後，真正帶得走的東西。
