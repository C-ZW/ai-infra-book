# 並發不是平行：兩個常被混用的詞

設想一個再普通不過的決定。你的服務每秒要處理幾千個請求，每個請求大半時間卡在等資料庫回應，p99 延遲開始爬升。有人在 retro 上提議：「機器升級成 16 核，把執行緒開多一點，吞吐就上去了。」聽起來天經地義——更多核心、更多執行緒、更多吞吐。三個月後帳單翻倍，p99 紋風不動。

問題不在那台機器，而在那句話悄悄把兩件不同的事黏成了一件：它假設「同時處理很多請求」這個需求，要靠「同時在很多核心上跑」來滿足。前者是**並發（concurrency）**，後者是**平行（parallelism）**，而這個服務的瓶頸——「等資料庫」——既不是核心數能解的，也根本不需要平行。把這兩個詞當同義詞，是並發架構幾乎所有誤判的共同起點：選錯執行模型、買錯硬體、在不需要鎖的地方加鎖、在需要平行的地方狂開協程。

這一章要做的事很單純，卻是後面所有並發主題的地基：把這兩個詞**徹底分開**，分到你之後看任何一段並發程式碼，都能立刻問出對的問題——這裡要的是結構，還是執行？瓶頸在等待，還是在計算？

## 一個關於咖啡店的場景，和一句經常被引錯的話

Rob Pike 在 2012 年 Heroku 的 Waza 研討會上講過一場後來被無數次引用的 talk，標題就是〈Concurrency is not Parallelism〉。他用一句話釘住了核心：**「concurrency is about dealing with lots of things at once; parallelism is about doing lots of things at once」**——並發是「**同時在應付**很多事」，平行是「**同時在做**很多事」。投影片上他寫得更精煉：concurrency is about structure（結構），parallelism is about execution（執行）。

這句話之所以難懂，是因為中文「同時」把兩種完全不同的「同時」壓成了一個詞。換個場景就清楚了。一個咖啡店店員獨自顧店：客人 A 點了手沖，他開始注水悶蒸，趁悶蒸的 30 秒空檔轉身收銀、接客人 B 的點單、把客人 C 的杯子遞出去，再回頭繼續注水。從門外看，這個店員「同時在服務三個客人」——但任何一個瞬間，他的手只在做一件事。他做的是**並發**：把工作拆成可以交錯推進的片段，在某件事「卡住等待」（悶蒸、等機器、等出杯）的空檔，切去推進別的事。一個人，零平行，卻能同時應付一整排客人。

**平行**是另一回事：你雇了第二個、第三個店員，三杯咖啡在三雙手上**真的同時**被沖煮。這需要的不是「巧妙地交錯」，而是「真的有三雙手」——也就是硬體。

兩者的關係不是「一個是另一個的進階版」，而是**正交**：它們是兩根獨立的軸，可以任意組合。

| | 不平行（單核） | 平行（多核） |
|---|---|---|
| **不並發**（單一工作流） | 一個店員沖一杯，沖完才做下一件 | 一個人把一張大圖切四塊、四核同時 resize（同一件事被切開） |
| **並發**（多個交錯的工作流） | 一個店員交錯應付一排客人（event loop） | 三個店員各自交錯應付各自一排客人（多核 + 多工作流） |

四個格子都成立，這正是重點。一個單核心、單執行緒的 event loop（深入機制見〈event loop：單執行緒怎麼同時做很多事〉）是**高度並發但零平行**：它在 I/O 等待的空檔切去處理別的連線，看起來同時服務上千條連線，但任一瞬間 CPU 上只有一個 callback 在跑。反過來，一段把陣列切成四塊、丟到四核同時運算的純計算程式碼，是**平行但不並發**——那只是同一件事被機械地切開，沒有「多個獨立、會互相等待、需要交錯」的工作流，也就沒有要被結構化的並發。

## 為什麼非得把它們分開：因為它們各自解的問題不一樣

把兩個詞分清楚不是術語潔癖，而是因為**它們對應到兩種不同的瓶頸，用錯了藥就治不好病**。

第一原理只有一句：**平行需要硬體（多核心），並發只需要一種「能在工作之間切換」的機制。** 你能在單核心上寫出極度並發的程式（靠協作式切換），也能寫出完全不並發卻吃滿八核的程式（resize 一張巨圖）。並發是你**寫程式的方式**，平行是執行時**恰好擁有的資源**。

從這條原理直接長出一個判準，它比任何框架選型指南都重要——**先問瓶頸在哪一邊**：

- 瓶頸在 **I/O 等待**（呼叫 DB、外部 API、讀檔、等網路）：你要的是**並發**，不是平行。在等待的空檔塞進別的工作就夠了，多買核心幫不上忙——CPU 大半時間在閒等，再多核心也是閒等。
- 瓶頸在 **CPU 計算**（壓縮、加密、影像處理、大量 JSON 解析）：你要的是**平行**。純粹的並發結構救不了一個吃滿單核的計算，你得真的把工作攤到多個核心上跑（CPU 重活怎麼搬離主執行緒，見〈thread pool 與 worker threads：CPU 密集怎麼辦〉）。

開場那個服務的病根，現在看得一清二楚：它的瓶頸是「等資料庫」，是 I/O。它需要的是把等待重疊起來的**並發**——一條執行緒在等 A 請求的 DB 回應時，去推進 B、C、D 的工作。升級到 16 核給它的是**平行**能力，而它的瓶頸根本不在計算，於是那 16 個核心絕大多數時間都坐在那裡，陪著它一起等資料庫。錢花在了問題的另一側。

## 把瓶頸算給你看：同一個並發結構，瓶頸換邊，結論相反

抽象的「瓶頸在哪邊」可以變成具體的數字，而且這個算術會逼出一個反直覺的結論。

假設一個服務處理每個請求要做兩件事：等資料庫約 **20 ms**（I/O），然後在本地算約 **1 ms**（CPU，解析、組裝回應）。我們只有一條執行緒、一顆核心，用 event loop 把請求並發起來。問：單核理論吞吐上限是多少？

關鍵在於「等 DB 的 20 ms」期間，這條執行緒**不必空等**——它可以切去處理別的請求的那 1 ms CPU 工作。所以真正佔住這顆核心、無法重疊的，只有每個請求那 1 ms 的計算。一秒鐘有 1000 ms，每個請求消耗 1 ms 的 CPU：

```
單核吞吐上限 ≈ 1000 ms / 1 ms per req ≈ 1000 req/s
```

瓶頸是那 1 ms 的 CPU，**不是**那 20 ms 的等待。等待是「免費」的——它被重疊掉了。這就是為什麼一條 Node 執行緒能服務上千個慢請求：慢的是 I/O，而 I/O 不佔 CPU。

現在把瓶頸換到另一邊。同一個服務、同一個 event loop 結構，但每個請求改成**算 20 ms（CPU）+ 等 1 ms（I/O）**：

```
單核吞吐上限 ≈ 1000 ms / 20 ms per req ≈ 50 req/s
```

吞吐從 1000 掉到 50，掉了 20 倍——**而程式碼的並發結構一個字都沒改**。差別只在瓶頸從 I/O 那 20 ms 搬到了 CPU 那 20 ms。CPU 工作無法被重疊：這條執行緒算 A 的 20 ms 時，它真的什麼別的都做不了，B、C、D 全部排隊乾等。這時，而且只有這時，加核心（平行）才有意義——把計算攤到 16 核，理論上能逼近 50 × 16 ≈ 800 req/s。

同一個並發結構，瓶頸在 I/O 時加核心毫無幫助、在 CPU 時加核心是唯一出路。這就是為什麼「升級核心數」不是一句安全的通用建議——它只對其中一種瓶頸有效，而你得先知道自己是哪一種。

## 平行的天花板：為什麼加核心的回報會塌掉

就算你確認了瓶頸在 CPU、決定上平行，還有一個常被忽略的硬限制：**平行的加速比不是線性的，因為幾乎沒有任何工作是 100% 可平行的。**

把這件事算到底的是 Amdahl's Law（Gene Amdahl，1967）。它說：如果一個工作裡有比例 *f* 的部分**只能序列執行**（無論如何切不開——讀取輸入、最後的彙整、必須照順序的那一段），用 *P* 個處理器能拿到的最大加速比是：

```
              1
speedup(P) = ───────────────
             f + (1 − f) / P
```

直覺是：可平行的那 (1 − *f*) 切成 *P* 份、各自加速，但序列的那 *f* 怎麼加核心都不會變快，它就杵在那裡當下限。

代真實數字進去。假設某個批次計算有 **10% 是序列的**（*f* = 0.1，比如最後要把所有分片結果合併成一份報表，這步只能一個核心做），其餘 90% 完美可平行。你給它 4 個核心：

```
speedup(4) = 1 / (0.1 + 0.9/4) = 1 / (0.1 + 0.225) = 1 / 0.325 ≈ 3.08x
```

4 個核心只換到 3.08 倍——已經有 23% 的算力被那 10% 的序列段吃掉了。更殘酷的是把核心開到無限大：

```
speedup(∞) = 1 / 0.1 = 10x
```

無論你堆多少核心，這個工作最多只能快 10 倍——因為那 10% 的序列段永遠在那裡，它是天花板。把核心從 4 開到 16，加速比也只從 3.08 爬到約 6.4，回報已經明顯在塌。

這條定律收束了平行的全部浪漫：**平行的回報受限於你工作裡最不可平行的那一段。** 它解釋了為什麼「開更多執行緒/協程一定更快」是錯的——超過某個點，多開的單元搶的是同一批核心，序列段卻紋絲不動，吞吐不升反降（多開協程為何反而變慢，深入見〈並發模型比較〉）。也解釋了一個架構直覺：**與其堆核心去壓榨那條序列段，不如先想辦法縮小 *f* 本身**——把序列的彙整改成可平行的、把「必須照順序」的約束鬆掉。降低 *f* 比增加 *P* 划算得多。

## 「誰決定何時切換」：並發底下的三種切換機制

把並發講到能「在腦中重放」，就得拆開那句輕飄飄的「在工作之間切換」——切換到底是**誰**觸發的？這個問題的三種答案，劃出了所有並發執行模型的版圖。

**搶佔式（preemptive）——OS 替你切。** 作業系統用時鐘中斷，每隔幾毫秒強制把當前執行緒踢下 CPU、換上另一條。每個工作都以為自己獨佔 CPU，切換對程式碼完全透明——你不必在程式裡寫任何「該讓出了」的提示。代價是切換要存下整組暫存器、可能換掉一整批快取，一次 context switch 大約微秒級；而且因為任何時刻都可能被打斷，多執行緒共享記憶體時就有了 race condition（共享狀態怎麼出錯、怎麼修，見〈race condition 與原子性：跨 DB、cache、runtime〉）。OS 執行緒與行程走的就是這條路。

**協作式（cooperative）——工作自己讓。** 切換只發生在工作**主動交出控制權**的時刻：碰到「要等 I/O」或顯式的 `await`／`yield`。runtime 在單一執行緒上，趁這些讓出點輪流推進各個工作。切換極輕（差不多就是一次函式呼叫），也因為「只在我選擇的點被打斷」，同一條執行緒上不會有兩個工作真的同時碰共享狀態——這是 event loop **天生沒有資料競爭**的隱性紅利。但別把它讀成「單執行緒＝並發安全」：沒有資料競爭不等於沒有並發 bug——跨 `await`／`yield` 讓出點的交錯仍可能造成 check-then-act 型的邏輯競爭（讀到某值、讓出、別人在這空檔改了它、回來沿用舊值），只是它不是記憶體層級的 data race。但它有一個致命前提：**每個工作都得守規矩、肯讓出**。一個工作若進了一段沒有 `await` 的死迴圈、或一段純 CPU 的長計算，它就霸著那條執行緒不放，**其餘所有工作一起凍結**。event loop 與協程走這條路。

**混合式（M:N）——runtime 替你排，還補上搶佔。** 純協作式有個阿基里斯腱：一個不讓出的工作能卡死全場。Go 的排程器是這條路最成熟的例子，值得看清楚它怎麼補洞。它把海量的 goroutine（G）多工到少量 OS 執行緒（M）上，由一個邏輯處理器（P，數量等於 `GOMAXPROCS`）來牽線——程式碼寫得像同步阻塞，執行時卻是非阻塞的協作式切換，輕到一個 goroutine 初始堆疊只要約 **2 KB**（按需成長），開幾十萬個也不過幾百 MB。

但 Go 不只靠協作。Go 1.14 起加入了**訊號式非同步搶佔**：一個背景監控執行緒（sysmon）盯著每個 goroutine，發現某個 goroutine 連續霸著處理器超過約 **10 ms** 還不讓出（runtime 裡的 `forcePreemptNS` 常數），就對它所在的 OS 執行緒送一個 `SIGURG` 訊號，硬把它打斷、換下別人。這一手把「純協作式會被死迴圈卡死」的洞補上了——你寫的 goroutine 就算進了一段不讓出的 CPU 迴圈，sysmon 也會在 10 ms 後強制把它請下台。協作式的輕量，加上搶佔式的兜底，這正是 Go 的並發在高並發 CPU 混合負載下好用的原因。

值得單獨點出的是，**「開幾個 OS 執行緒／設多少 `GOMAXPROCS`／event loop 配幾顆 worker」這個決定，跟你用哪種切換機制是兩件正交的事**。前者決定你拿幾顆核心（平行度），後者決定你的工作怎麼交錯（並發結構）。把這兩個旋鈕當成一個來轉，又是一輪誤判的開始。

## 一個藏在容器裡、會悄悄打爆並發架構的邊界

理論上「平行度由核心數封頂」很乾淨，但這條線在現代部署環境裡有一道隱蔽的裂縫，值得專門講透——因為它害過太多人。

很多 runtime 探測「我有幾顆核心」的方式，是去問作業系統「這台機器有幾顆 CPU」。在裸機上這沒問題。但在容器裡，這個答案會**說謊**。一個跑在 64 核宿主機上的容器，就算被 Kubernetes 的 cgroup CPU limit 限制成「最多只能用 2 顆核心的算力」，傳統的探測方式仍會回報「我看到 64 顆核心」。

後果很具體。一個 Go 程式若把 `GOMAXPROCS` 設成偵測到的 64，它會建立 64 個邏輯處理器、試圖讓 64 個 goroutine 真的平行跑——但 cgroup 只給它 2 核的 CPU 配額。於是它在一個排程週期裡瘋狂排程 64 條工作、很快用光配額，接著被 cgroup **throttle（節流）**：核心被強制凍結，直到下個配額週期才解凍。結果不是更快，而是 p99 延遲被這種週期性的「凍結—解凍」狠狠拉長。「以為自己有 64 核」的並發結構，撞上「實際只有 2 核」的平行現實，撞出來的是更糟的延遲。

這個坑普遍到 Go 在 **1.25 版（2025）把 `GOMAXPROCS` 預設改成 cgroup-aware**：在 Linux 上，runtime 會讀取 cgroup 的 CPU bandwidth limit，若它低於實體核心數，就用較小的那個當預設；而且因為編排系統可能在運行中動態調整 CPU limit，runtime 還會週期性重新檢查、自動跟著調。這個改動本身就是「並發 ≠ 平行」這條原理的一個註腳：你的並發結構想開多少工作是一回事，runtime 實際能拿到多少平行度，得問那個真正發配額的人——在容器世界裡，那個人是 cgroup，不是 `/proc/cpuinfo`。

## 為什麼是這個形狀

退一步看，這兩個詞之所以非分開不可，是因為它們**根植於兩種不同的稀缺**。並發要解的稀缺是「等待」——大量工作彼此獨立、各自會卡在 I/O 上，你需要一種結構，讓一個工作等待的空檔不被浪費，拿去推進別的工作。平行要解的稀缺是「算力」——一件大計算超過單核能負荷，你需要真的把它攤到多顆核心上同時算。一個關於**結構**，一個關於**資源**。

把它們黏在一起，會在兩個方向上犯錯：以為加核心能治好等待病（開場那個翻倍帳單的服務），或以為巧妙的並發結構能變出算力（在純 CPU 瓶頸上猛調 async，多一層排程開銷卻一點沒變快）。分開之後，每個並發決策都還原成兩個清楚的子問題：我的工作之間需不需要交錯（並發結構選什麼——event loop、執行緒、協程、actor，見〈並發模型比較〉），以及我需不需要、能不能拿到真正的同時執行（平行度——幾顆核心、容器給不給、Amdahl 的天花板在哪）。

下次再聽到「升級核心數就會更快」，你會先停下來問那個唯一要緊的問題：瓶頸在等待，還是在計算？答案是前者，再多核心也只是陪著一起等；是後者，才輪到 Amdahl 來告訴你天花板有多高。並發與平行不是同一個詞的兩種說法，而是你面對「同時做很多事」時，必須分開回答的兩個問題。
