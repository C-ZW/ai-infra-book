# WebRTC：媒體、傳輸與信令

兩個人想在瀏覽器裡通話。直覺上這應該是即時傳輸裡最單純的一種——比起聊天室那種要把一條訊息扇出給五千個人的場景，這裡只有兩端、一條音訊、一條視訊，把位元組從 A 送到 B 而已。但真的動手做，會撞上一連串在後端世界裡幾乎沒遇過的牆。

第一個問題：A 和 B 根本不知道彼此在哪裡。兩人都坐在家用路由器後面，對外只有一個被 NAT 共用的公開 IP，內網的 `192.168.x.x` 對方完全連不到。在傳統後端架構裡這不是問題——大家都連到同一台有公開位址的伺服器，伺服器當交會點。但語音視訊如果每一個位元組都繞經你的伺服器，一場 1080p 通話單向就要好幾 Mbps，一萬場併發就是幾十 Gbps 的出口頻寬，你會為了「兩個人講話」付一筆和通話量等比成長的流量帳。所以這裡真正想要的是**讓 A 和 B 直接連線、媒體不經你的伺服器**——可是他們又彼此看不見。

第二個問題：就算連上了，這條連線要傳的不是請求-回應式的資料，而是**對延遲極度敏感、對偶爾丟一兩個封包卻無所謂**的即時媒體。視訊會議裡，一個遲到 500 毫秒才到的畫格還不如直接丟掉——它已經沒有意義，硬要重傳只會把後面的畫面一起卡住。TCP 那套「保證每個位元組有序不丟、丟了就停下來重傳」的脾氣，剛好和即時媒體要的東西相反。

WebRTC 就是為了同時解掉這兩道牆而存在的一整套標準。它不是單一協定，而是一個由 W3C 與 IETF 共同維護的協定集合——W3C 那份《WebRTC: Real-Time Communication in Browsers》自 2021-01-26 成為 Recommendation，2025-03-13 又發布納入修訂的更新版，至今仍是現行標準。理解它的訣竅，是把它拆成三塊互相獨立、各自解一個問題的責任：**媒體**（怎麼編碼、加密、收送音視訊）、**傳輸**（兩個 NAT 後的端怎麼找到彼此、建立連線）、**信令**（連線建立前，兩端怎麼先交換「我是誰、我能解哪些編碼、我在哪些位址」這些後設資訊）。這三塊的邊界劃在哪、為什麼劃在那裡，就是這一章要講透的事。

## 信令：標準刻意不管的那一塊

先講最反直覺的一塊：WebRTC **刻意不規範信令**。

想連線，A 至少得先把「我支援 H.264 和 VP8、我的音訊用 Opus、我猜我的公開位址大概是這幾個」這串描述送給 B，B 也得回一份自己的。這個「交換連線描述」的過程叫信令。一個自然的疑問是：既然 WebRTC 連 NAT 穿透這種髒活都標準化了，為什麼不順手把信令也定下來？

答案是一個漂亮的關注點分離決定。信令要走哪條路、用什麼協定、訊息怎麼可靠送達、怎麼處理兩端同時發 offer 的衝突（glare），這些**和媒體本身無關**，而且每個應用的偏好天差地遠——有的想複用既有的 SIP 基礎設施，有的只想用一條 WebSocket，有的甚至用兩台手機互相掃 QR code。把信令焊死在標準裡，等於強迫所有人接受同一套訊息傳輸，反而綁手綁腳。所以 WebRTC 的設計哲學是：**標準只管媒體平面（media plane），信令平面（signaling plane）留給應用自己決定。**

但「不規範信令的傳輸」不等於「不規範信令的內容」。要交換的那份連線描述，格式是標準化的——它叫 **SDP（Session Description Protocol）**，一段文字，描述媒體軌、編碼能力、加密指紋、候選位址。交換的流程也是標準的 **offer/answer 模型**（其語意由 RFC 3264 定義，在 WebRTC 裡由 JSEP——JavaScript Session Establishment Protocol，原為 RFC 8829，已由 RFC 9429 取代（obsoletes，2024-04）——規範瀏覽器該怎麼產生與套用這些描述）：一端 `createOffer()` 產生 offer SDP，透過你自選的信令通道送給對端，對端 `createAnswer()` 回一份 answer，雙方各自把這兩份描述套進自己的 `RTCPeerConnection`，連線協商就有了共同基礎。

```
   A  ── offer SDP ──►  〔你的信令通道：WebSocket / SIP / 掃 QR…〕  ──►  B
   A  ◄─ answer SDP ──  〔同一條通道〕  ◄──────────────────────────────  B
        之後雙方各自繼續經這條通道補送 ICE candidate（trickle ICE）
```

這裡藏著第一個容易翻車的邊界：**信令通道的安全是你自己的責任，不在 WebRTC 的保護傘下。** 後面會講到媒體有強制加密，但 SDP 在你的信令通道上裸奔。如果那條 WebSocket 沒上 TLS，SDP 裡的編碼能力、candidate 位址、甚至用來綁定加密的指紋都會被中間人看到甚至竄改。媒體再怎麼端到端加密，信令被竄改一樣能把連線導去攻擊者控制的中繼。WebRTC 把信令交給你，連同信令的安全責任也一起交給了你。

## 傳輸：在 NAT 的迷宮裡找一條通路

信令把兩端的描述湊到一起之後，真正難的傳輸問題才登場：A 和 B 都在 NAT 後面，**沒有任何一個位址是對方能直接打到的**。

把 NAT 想清楚。當 A 的電腦（內網 `192.168.1.5:50000`）對外發一個封包，家用路由器會把來源改寫成它自己的公開位址（比如 `203.0.113.7:62000`），並記下「凡是回到 `62000` 的封包，轉回給內網的 `192.168.1.5:50000`」這條映射。問題是 A 自己**根本不知道** `203.0.113.7:62000` 這個對外的長相——它只看得到自己的內網位址。而 B 要連 A，需要的恰恰是這個對外位址。兩端都被困在自己的內網視角裡，這就是 NAT 穿透的核心難題。

WebRTC 用 **ICE（Interactive Connectivity Establishment，RFC 8445，取代了舊的 RFC 5245）** 這套框架來破局。ICE 的思路不是「算出唯一正確的位址」，而是**蒐集所有可能的位址、然後全部試一遍，挑一條通的**。每一個可能的位址叫一個 candidate，分三類：

- **host candidate**：自己的內網位址（`192.168.1.5:50000`）。同一區網的兩端靠這個就能直連。
- **server-reflexive candidate**：經 NAT 映射後的對外位址。這個要靠 **STUN（Session Traversal Utilities for NAT，RFC 8489，取代 RFC 5389）** 取得——A 對一台公開的 STUN 伺服器發一個請求，STUN 伺服器回信告訴 A：「我看到你這個封包的來源是 `203.0.113.7:62000`」。A 這才知道自己經 NAT 後的對外長相，把它當成一個 candidate 交給對方試。STUN 的全部工作就這麼簡單——當一面鏡子，照出你自己在 NAT 外的樣子。
- **relay candidate**：當直連怎麼都打不通時，退而求其次，走一台中繼伺服器轉發。這要靠 **TURN（Traversal Using Relays around NAT，RFC 8656，取代 RFC 5766）**。

ICE 把雙方各自蒐集到的所有 candidate 兩兩配對，對每一對做連通性檢查（互發 STUN binding 封包看打不打得通），選出一條能通、且優先序最高的路徑——優先序通常是「直連 host > STUN 反射直連 > TURN 中繼」，因為越靠前的越省延遲、越省你的伺服器頻寬。整個過程是邊蒐集邊試的，candidate 不必一次到齊，所以實務上信令交換完 offer/answer 之後，雙方還會持續經信令通道補送後來才發現的 candidate（trickle ICE）。

### 為什麼非得有 TURN：symmetric NAT 這道牆

STUN 看起來已經解決問題了——照出對外位址，對方照著打就好。那 TURN 為什麼非存在不可？

關鍵在一種叫 **symmetric NAT** 的 NAT 行為。普通的 NAT，A 對外的映射只看來源——只要 A 從 `192.168.1.5:50000` 出去，不管目的地是誰，對外都映射成同一個 `203.0.113.7:62000`。這種情況下 STUN 照出來的位址是有用的：A 對 STUN 伺服器探到 `62000`，B 照著 `62000` 打也能進來。

但 symmetric NAT 的映射是**依目的地而異的**：A 連 STUN 伺服器時映射成 `62000`，A 連 B 時 NAT 卻會開一個全新的對外埠 `63000`。於是 A 經由 STUN 探到的 `62000`，對真正的對端 B **完全不適用**——B 照著 `62000` 打，封包會被 NAT 丟棄，因為那條映射只認 STUN 伺服器、不認 B。兩端如果都在 symmetric NAT 後面，STUN 的反射位址就成了一張過期的地址，怎麼配對都連不通。

這時唯一的辦法，就是雙方都改連同一台有公開位址、雙方都打得到的中繼——**TURN relay**。A 把媒體送給 TURN 伺服器，TURN 轉給 B；B 回的媒體也經 TURN 轉回 A。直連的夢碎了，媒體繞了一圈經過你的伺服器，那筆你本想省掉的頻寬帳又回來了。

這就是為什麼**生產級的 WebRTC 必須部署 TURN**，這是最常被漏掉的一塊。開發時和測試對象多半在同一個寬鬆的網路環境，光靠 STUN 就能直連，於是 STUN 看起來就夠了；一上線碰到企業防火牆、行動網路的 carrier-grade NAT（這類幾乎清一色是 symmetric NAT），一大批使用者直接連不上。實務上有相當比例的連線——視網路環境，常見估計落在一到兩成、在受管制的企業或行動網路裡會更高——最終得走 TURN 中繼。沒有 TURN 兜底，這些使用者對你的應用而言就是「打不通電話」。開源世界裡的 **coturn**（2026-06 仍活躍維護，最新約 4.13.1）是事實上的標準 TURN/STUN 伺服器選擇。

```
直連可能（host 或 STUN 反射）：
   A ───────────── 媒體直送 ─────────────── B      你的伺服器不碰媒體

直連失敗（symmetric NAT），走 TURN：
   A ──── 媒體 ────> [ TURN relay ] <──── 媒體 ──── B
                      (你出頻寬)                      你為這條連線付中繼流量
```

## 媒體：加密是強制的，可靠性是可選的

連線通了，接下來談真正要傳的東西。WebRTC 在媒體這塊定下兩個強硬的立場，剛好對應開場那兩道牆。

第一個立場：**加密不可關閉。** 音視訊走 **SRTP（Secure RTP）**，金鑰交換用 **DTLS-SRTP**——兩端先跑一次 DTLS 握手（在 UDP 上的 TLS）協商出 SRTP 金鑰，之後所有 RTP 媒體封包都用這把金鑰加密。WebRTC 規格明文禁止裸的、未加密的 RTP，連 NULL 加密的 cipher suite 都禁止協商。這在後端世界是少見的態度——多數協定把加密當成「建議開啟」的選項，WebRTC 直接把它焊死成不可繞過的地基。理由也合理：這是會在瀏覽器裡、在任意網路上、傳輸別人臉和聲音的東西，把加密當可選等於把隱私當可選。而 DTLS-SRTP 把金鑰交換做在端到端的握手裡，意味著即使媒體被迫經過 TURN 中繼，中繼伺服器看到的也只是密文——它轉發位元組，但解不開內容。

但這道防線有個前提，正好接回前面那句「信令被竄改一樣能把連線導去攻擊者控制的中繼」。WebRTC 的 DTLS 握手用的是**自簽憑證**、沒有 PKI/CA，所以握手本身證明不了對端是誰——它只能保證「跟我握手的那個人，從頭到尾是同一個」。真正把「跟我握手的人」綁定到「信令裡宣告的那個人」的，是 SDP 裡的 `a=fingerprint`（憑證指紋）：只要信令通道完整、沒被竄改，fingerprint 就鎖死了合法的 DTLS 對端。反過來，攻擊者一旦能竄改信令、換掉這條 fingerprint，就能讓你的端去和它握手——端到端加密被無聲地接管，而媒體層的強制加密一點忙都幫不上。這就是為什麼信令通道的完整性，是整個媒體安全的根。

第二個立場，正好相反：**可靠性是可選的，而且預設選「不可靠」。** 這對應開場第二道牆——即時媒體寧可丟也不要等。SRTP 跑在 UDP 上，丟了的封包預設不重傳；編解碼器（視訊有 VP8、VP9、AV1、H.264，音訊有 Opus）被設計成能容忍一定程度的丟包，靠後續畫格自我修復，而不是卡住等那個遲到的封包。這是一個和 TCP 哲學完全顛倒的取捨：TCP 用「絕不丟、但可能卡」服務檔案傳輸，WebRTC 用「可能丟、但絕不卡」服務即時媒體。

但 WebRTC 不只能傳媒體。`RTCDataChannel` 讓兩端傳**任意資料**——檔案、遊戲狀態、聊天文字，走的是 **SCTP over DTLS**（同樣建在那條已經打通、已經加密的 ICE+DTLS 傳輸上）。SCTP 的妙處在於它的可靠性與順序性是**逐通道可配置**的：你可以開一條「完全可靠、保證有序」的通道傳檔案（行為像 TCP），也可以開一條「不可靠、無序」的通道傳遊戲裡每幀的座標（行為像 UDP，過時的座標丟了無妨，下一幀馬上覆蓋）。同一條底層連線，上面能同時跑可靠和不可靠的通道。這讓 WebRTC 遠不只是「視訊通話技術」——它是一條打通了 NAT、強制加密、可調可靠性的通用 P2P 管道。

## 多人會議：把拓樸畫對，頻寬才不會爆

到目前為止講的都是兩個人。一旦變成多人會議，一個全新的問題冒出來：媒體該怎麼在 N 個人之間流動？這個選擇——拓樸——直接決定了系統能撐多少人、誰來付頻寬和 CPU 的帳。有三種選法，值得用真實數字算清楚它們的差別。

假設一場會議，每個人的視訊流是 **1 Mbps**。

**第一種，mesh（網狀，純 P2P，無媒體伺服器）。** 每個人和其他所有人各建一條 WebRTC 連線，把自己的流直接送給每一個人。這是 WebRTC「P2P」精神最純粹的延伸，沒有任何媒體伺服器。問題出在上傳：N 個人時，每個人要把自己那一份流**各自上傳 N−1 份**給其他每一個人。4 個人時，每人上傳 3 Mbps、下載 3 Mbps，家用上行勉強扛得住。但這個成本隨人數線性爆炸——

```
mesh 每人上傳頻寬（每流 1 Mbps）：
   人數 N    每人上傳     每人下載
     3        2 Mbps      2 Mbps
     4        3 Mbps      3 Mbps
     8        7 Mbps      7 Mbps
    10        9 Mbps      9 Mbps
```

到 10 個人，每人要同時上傳 9 Mbps——絕大多數家用網路的上行根本沒這個量，連線直接塞爆。mesh 沒有伺服器成本的優雅，是用「把成本全壓在每個 client 的上行」換來的，所以它只適合極小群，大約 4 到 5 人封頂。

**第二種，SFU（Selective Forwarding Unit，選擇性轉發單元）。** 加一台伺服器，但它**不解碼、不混流**，只做一件事：收每個人**一份**上傳流，再**選擇性地轉發**給其他人。關鍵的轉折在上傳側——每個人不管會議多少人，**只上傳一份**自己的流給 SFU（1 Mbps），由 SFU 負責複製轉發。下載側仍然是 N−1 份（你要看到其他每個人）。

用同樣 4 人、1 Mbps 算：每人上傳從 mesh 的 3 Mbps 掉到 **1 Mbps**，且這個 1 Mbps **不隨人數成長**——10 人會議每人上傳還是 1 Mbps。代價轉移到了 SFU：它要進 4 份（4 Mbps），出 4 人各收 3 份（4 × 3 = 12 Mbps）。但因為 SFU 只轉發、不碰編解碼，它的 CPU 負擔很輕，主要吃的是頻寬。這個「每人上傳成本固定為 1 份，與人數無關」的性質，就是 SFU 能撐起幾十上百人會議、成為現代多人視訊主流架構的全部原因。開源世界的 Jitsi Videobridge（JVB）就是一台 SFU。

**第三種，MCU（Multipoint Control Unit，多點控制單元）。** 伺服器走到最重的一端：把所有人的流**解碼、混成一個合成畫面、再重新編碼**成單一一路，送給每個人。於是每個人上傳 1 份、**下載也只 1 份**（那路合成畫面），client 端的頻寬和算力都壓到最低——這對頻寬極受限或算力很弱的端（老舊裝置、低階機上盒）是救命的。代價是伺服器要為每一路會議做即時轉碼，CPU 隨會議數和人數線性暴漲，而且解碼再編碼這一趟會引入額外的延遲。

把三者並排，取捨就清晰了：

| 拓樸 | 每人上傳 | 每人下載 | 伺服器負擔 | 適用 |
|---|---|---|---|---|
| mesh | N−1 份 | N−1 份 | 無（純 P2P） | 1 對 1、極小群（≤4–5 人） |
| SFU | **1 份**（固定） | N−1 份 | 頻寬重、CPU 輕（只轉發） | 多人會議主流（5–100+ 人） |
| MCU | 1 份 | **1 份**（合成） | CPU 極重（即時轉碼） | client 端頻寬/算力極受限 |

選錯拓樸，等於把成本壓在錯誤的地方：用 mesh 撐多人，把帳壓爆使用者的上行；用 MCU 服務一般會議，把帳壓爆你的伺服器 CPU。多數要擴展的多人場景，答案是 SFU——它在「client 上傳成本」這個最容易先爆的維度上做到了與人數脫鉤。

## 一個容易混淆的陷阱：SFU 不解碼，MCU 才解碼

SFU 和 MCU 常被混為一談，但它們在伺服器上做的事根本不同，而這個不同決定了你的成本會壓在頻寬還是 CPU。

SFU **不跑編解碼**——它不解碼壓縮的視訊位元流、也不重新編碼，這是它和 MCU 最根本的分野。但「不跑 codec」不等於「什麼都不解開」：標準部署裡 DTLS-SRTP 的加密是**逐 hop（每條連線各自一把金鑰）**的，所以 SFU 其實會終結 DTLS、把 SRTP 媒體**解密成明文**——它得讀得到 RTP header 才能改寫 sequence number/SSRC、才能配合分層編碼（simulcast/SVC）按各接收者的網路狀況挑不同畫質的那一層轉發——然後再用對每個接收端各自協商的金鑰**重新加密**送出。也就是說，預設情況下 SFU 看得到明文媒體；要讓它連明文都看不到，得在上面再疊一層端到端加密（WebRTC Insertable Streams / Encoded Transform ＋ SFrame，2026 瀏覽器支援仍不齊，Chromium 完整、Safari/Firefox 部分）。但因為始終不碰 codec，它的 CPU 幾乎只花在搬封包與收發加解密上，一台機器能轉發的路數仍很高；它出口的總頻寬是「每個接收者一份」，所以頻寬才是它的瓶頸。SFU 還有一個 MCU 給不了的好處：因為保留了每一路的原始獨立流，client 端可以自由排版面（誰大誰小、誰釘選）。

MCU **要把媒體解開來**。它得解碼每一路、在像素層面混成一張合成畫面、再重新編碼——這是真正的計算密集工作，每一路會議都得燒一份轉碼的 CPU。換來的是 client 端只需處理一路流。也因為它在伺服器端就把畫面焊成一張了，client 失去了自由排版面的彈性。

一句話記牢：**SFU 壓你的頻寬帳，MCU 壓你的 CPU 帳。** 知道自己的瓶頸在哪一邊，才知道該往哪邊選。

## WebRTC 是 P2P——但 P2P 不等於零伺服器

WebRTC 最深入人心、也最容易誤導人的一個簡化，是「它是點對點的，所以不需要伺服器」。把它逐塊拆開，會發現這句話幾乎在每一塊都不成立：

- **信令一定要伺服器。** 兩端在建立連線前彼此看不見，那份 offer/answer SDP 總得有人居中傳遞——這台信令伺服器跑不掉。
- **NAT 穿透多半要伺服器。** STUN 伺服器幫你照出對外位址；直連失敗時 TURN 伺服器中繼全部媒體。symmetric NAT 後的使用者，沒有 TURN 就連不上。
- **多人幾乎一定要伺服器。** mesh 撐不了幾個人，多人會議要 SFU 或 MCU——這是一塊有狀態、吃頻寬或 CPU 的伺服器基建。

WebRTC 真正省掉的，只是「**直連成功時的那條媒體路徑**」——而且僅在那個幸運情況下。它不是「零伺服器」，而是「在條件允許時，把最重的那條媒體流量從伺服器卸載到 P2P 直連」。把它當成完全不需要後端的技術，會在 NAT 穿透和多人這兩個地方準時翻車。

## 為什麼是這個形狀

退一步看，WebRTC 的整套樣貌——三塊責任的切分、媒體的兩個強硬立場、多人時非得選拓樸——全是從開場那兩道牆逼出來的。

因為要**把媒體從伺服器卸載到 P2P**，才有了 ICE/STUN/TURN 這整套在 NAT 迷宮裡找路的機制；因為直連不保證成功，才必須留一條 TURN 中繼的後路，於是「P2P」永遠帶著一條伺服器尾巴。因為傳的是**即時媒體**，才會做出「可能丟、但絕不卡」這個和 TCP 顛倒的取捨，把可靠性下放成逐通道可選；因為是在開放網路上傳別人的臉和聲音，才把加密焊死成不可關閉的地基。因為信令的偏好千差萬別、又和媒體本身無關，才**刻意不規範它**，只標準化 SDP 的內容與 offer/answer 的流程，把傳輸與安全的責任連同自由一起交還給你。而當人數從 2 變成 N，「每個位元組從哪流到哪」這個問題沒有單一最優解，才分裂出 mesh、SFU、MCU 三種拓樸，各自在「壓 client 上行、壓伺服器頻寬、壓伺服器 CPU」之間選一個位置站定。

WebRTC 看起來零碎、看起來牽涉一堆首字母縮寫，但它不是一堆協定的隨意堆疊。它是「讓兩個躲在 NAT 後面的端，在開放網路上即時、加密、低延遲地直接交換媒體」這個被推到極限的需求，被現實的每一道牆——位址不可見、媒體不能等、隱私不能讓、人數會成長——依次塑形之後，唯一站得住的形狀。
