# thread pool 與 worker threads：CPU 密集怎麼辦

一個跑得好好的 Node.js 服務，p99 延遲長期穩在 20 毫秒。某天產品加了一個功能：上傳大頭照後，伺服器要把圖縮成三種尺寸。程式碼很直觀——收到請求，呼叫一個同步的影像縮放函式，算個一兩百毫秒，回傳結果。本機測起來沒問題，圖確實縮好了。上線之後，監控開始尖叫：不只是上傳的人變慢，**所有人**都變慢了。一個查餘額的請求、一個讀設定的請求，本來 5 毫秒回完的，現在動輒卡上一兩百毫秒。明明只有少數人在上傳照片，怎麼會全線中毒？

這一章要回答的就是這個問題，以及它的解法為什麼長成現在這個樣子。答案的核心是一句聽起來很反常識的話：**在單執行緒的 event loop 模型裡，「讓計算變快」幫不了你，你要做的是把計算搬離那條執行緒。** 而「搬去哪、怎麼搬、搬的時候付什麼代價」，就是 thread pool 與 worker threads 這兩套機制的全部內容。

## 為什麼一個人的 CPU 重活會凍住所有人

要看懂這場災難，得先記住 event loop 的一條鐵律：**主執行緒永遠不該同步地等任何東西**（event loop 怎麼用單執行緒服務海量連線，是另一個主題，見〈event loop：單執行緒怎麼同時做很多事〉）。一條主執行緒不斷輪詢「有沒有就緒的事件」，有就跑對應的 callback，跑完馬上回去輪詢下一個。整套並發的假象，建立在「每個 callback 都很快跑完、馬上把執行緒還回來」這個前提上。等 I/O 不佔執行緒——那是登記一個 callback 就返回、由作業系統在資料就緒時通知。所以一條執行緒交錯服務上千個連線，靠的是它們大半時間都在閒等，輪流借用那條執行緒的瞬間。

CPU 重活打破的正是這個前提。影像縮放不是「等」，是「算」——它要 CPU 連續忙上一兩百毫秒，中間沒有任何 `await`、沒有任何讓出點。在這一兩百毫秒裡，event loop 那條唯一的執行緒被這個請求**整個佔住**了：它跑不到輪詢、撿不起任何別的 callback、回不了任何別人的請求。其他連線不是「比較慢」，是**完全凍結**，排在那個計算後面乾等。等計算終於做完、執行緒還回來，後面積壓的請求才一個個解凍。

這就是為什麼少數人的 CPU 重活會讓全線中毒。把它量化一下會更刺眼。設想一個 QPS 100 的服務，平常每個請求 20 毫秒回完。某個請求在主執行緒上跑了一次同步的 `JSON.parse` 解一個 50 MB 的字串，耗時約 800 毫秒。在這 800 毫秒裡，主執行緒做不了別的事——若此刻有 100 個連線在線，它們全部被推遲約 800 毫秒。你的 p99 延遲從 20 毫秒被一個請求拉到 800 毫秒以上，而 CPU 使用率看起來甚至沒爆（單核可能才 100%，七個核在閒著）。監控上是一片詭異的「延遲爆炸但資源沒滿」，因為瓶頸不是算力不夠，是**所有人擠在同一條執行緒上**。

這裡有個必須先釘死的誤解：**`async` 救不了 CPU。** 很多人以為把函式標成 `async`、在前面加個 `await`，計算就「非同步」了、就不卡了。不會。`async`／`await` 的全部本事是「等 I/O 時讓出執行緒」；它對函式裡那段**同步的 CPU 區段**毫無作用。`await` 不會魔法般切開一個跑了 200 毫秒的 `for` 迴圈——迴圈跑的時候，執行緒就是被佔著。把一個純計算函式包成 `async`，只是多繞一層 Promise 排程，計算照樣在主執行緒上一口氣跑完，照樣凍住全線。要解決 CPU 密集，得從「換個寫法」升級到「換個執行緒」。

## 兩個「別的執行緒」：libuv 偷偷管的，跟你自己開的

「把計算搬離主執行緒」聽起來簡單，但在 Node 的世界裡，「別的執行緒」其實有兩種，它們解的問題不同、你對它們的掌控權也不同。把這兩種搞混，是這個主題最常見的迷路點。

第一種是 **libuv 的 thread pool**。Node 底層的 libuv 內建了一個工作執行緒池——預設 **4 條**（可由環境變數 `UV_THREADPOOL_SIZE` 在啟動時調整，絕對上限是 **1024**，這個上限在 libuv 1.30.0 從原本的 128 提高上來）。這個池是 **libuv 替你管的、你插不上手**。它存在的理由是：有些操作在作業系統層面**沒有**好用的非同步原語，逼得 libuv 只能找一條工作執行緒去同步地做、做完再把結果丟回主執行緒的迴圈。哪些操作走這條路？libuv 官方文件列出的是**檔案系統操作**與 `getaddrinfo`／`getnameinfo`（也就是 `dns.lookup` 的 DNS 解析）；Node.js 在這之上、於 C++ binding 層把幾類沒有 async OS 原語的 CPU 操作也掛進同一個池——async crypto（`pbkdf2`、`scrypt` 等）與所有 async zlib 壓縮（這份清單記在 Node.js 官方的「Don't Block the Event Loop」指南裡，不在 libuv 文件中）。

這裡藏著一個讓無數人栽跟頭的反直覺事實：**網路 I/O 不走 thread pool。** 你或許會以為「DNS 是網路操作，所以也走 epoll」——錯了。`dns.lookup` 走的是 libuv 那 4 條執行緒，而真正的網路 socket 讀寫（TCP/UDP）走的是作業系統的就緒通知（Linux 的 `epoll`、macOS 的 `kqueue`），在 event loop 自己的執行緒上完成、完全不碰 thread pool。差別的根源是：socket 有 `epoll` 這種「一次問一萬個 fd 哪些就緒」的非同步原語，檔案讀寫與 `getaddrinfo` 沒有（POSIX 沒給檔案 I/O 一個真正好用的 readiness 模型），於是後者只能用「開執行緒同步做」來偽裝成非阻塞。記住這條分界，下一節那個排隊陷阱才看得懂。

第二種是 **worker threads**（Node 的 `worker_threads` 模組，官方標記為 Stable）。這是**你自己明確開出來的**獨立 JS 執行緒。關鍵在於：每個 worker 都是一個獨立的 **V8 isolate**——它有自己的 JS 堆、自己的 microtask 佇列、自己的 event loop。這意味著兩個 worker 真的能**平行跑 JavaScript**，各佔一顆核。你把 CPU 重活（影像縮放、加密、大批資料解析聚合）丟進 worker，主執行緒只負責「派工、收結果」，自己那條執行緒就解放出來繼續服務 I/O 了。

這兩者的分界要刻在腦子裡：**libuv thread pool 是框架替你管的、給 C/C++ 層的 fs/dns/crypto 用，你不能往裡丟任意 JS；worker threads 是你自己開的、跑你自己的 JS 計算。** 開場那個影像縮放災難，如果縮放是純 JS 算的，libuv thread pool 一點忙都幫不上（它不收 JS），你需要的是 worker threads。而為什麼非得開新 isolate 才能平行跑 JS？因為這是第一原理：**在單一 V8 isolate 內，JavaScript 永遠是單執行緒的。** 要讓兩段 JS 真的同時在兩顆核上跑，唯一的路就是讓它們待在不同的 isolate 裡。沒有捷徑，`async` 不行、callback 不行、Promise.all 也不行——它們全都還在同一條執行緒上排隊。

## 那 4 條執行緒怎麼悄悄變成你的瓶頸

先看一個只關於 libuv thread pool 的故障，因為它最隱蔽、也最常被誤診成「Node 莫名其妙變慢」。

設想一個 API，每個請求都要做一次 `crypto.pbkdf2` 來雜湊密碼，假設單次約 100 毫秒。`pbkdf2` 是 async crypto，走 libuv 那 4 條執行緒。平常流量低，沒人察覺。某個瞬間同時湧入 **8 個**登入請求，會發生什麼？

池只有 4 條執行緒。前 4 個請求各搶到一條，並行開算，各約 100 毫秒。第 5 到第 8 個請求呢？它們的 `pbkdf2` 任務被丟進池，但**沒有空閒執行緒**——它們只能在佇列裡等。等前 4 個算完、騰出執行緒，後 4 個才接著算，又是約 100 毫秒。於是第 5 到第 8 個請求的雜湊延遲，從應有的 100 毫秒**憑空翻倍**到約 200 毫秒。

```
UV_THREADPOOL_SIZE = 4，8 個 pbkdf2 同時到（每個約 100 ms）

  pool slot 1:  [req1 ===100ms===][req5 ===100ms===]
  pool slot 2:  [req2 ===100ms===][req6 ===100ms===]
  pool slot 3:  [req3 ===100ms===][req7 ===100ms===]
  pool slot 4:  [req4 ===100ms===][req8 ===100ms===]
                |<--- 100 ms --->|<--- 100 ms --->|
                 req1-4 done       req5-8 done (~200ms)
```

這個故障最陰險的地方在於：它跟「event loop 被你的 JS 卡住」**毫無關係**。你的主執行緒清白得很，event loop 跑得飛快，CPU 也許還有閒。慢的根源是那個你平常根本看不見的 4 條執行緒的池被佔滿了，後來者在排隊。很多「Node 在高並發雜湊／大量檔案讀取／海量 DNS 查詢下突然變慢」的案子，真兇都是這個。而它還會跟別的東西**搶**池——你的 `pbkdf2`、你的 `fs.readFile`、你的 `dns.lookup`，全擠在同一個全域的 4 條執行緒上（libuv 文件明確說這個池是 global、跨所有 event loop 共享），彼此排隊。

解法是把 `UV_THREADPOOL_SIZE` 調大，比如調到 8——前提是你**真有 8 顆以上的核**可用。這時 8 個 `pbkdf2` 能同時開算，延遲回到約 100 毫秒。但這裡又埋著一個反直覺的陷阱：**把池調得比可用核心數還大，毫無幫助、甚至有害。** 假設你只有 4 顆核，卻把池設成 16，那 16 條執行緒會去搶那 4 顆核，作業系統得在它們之間不停切換（context switch）。`pbkdf2` 是純 CPU 工作，16 條執行緒搶 4 顆核，總算力沒變、還多付了切換成本，整體只會更慢。調大池解的是「並發的慢操作數量 > 4，被排隊卡住」這個問題，它**從來不會讓單一操作本身變快**——一次 `pbkdf2` 還是 100 毫秒，你只是讓更多個能同時跑。核心數才是真正的天花板。

## worker threads：把 JS 計算搬出去，但搬不是免費的

現在回到開場那個影像縮放——純 JS 算的 CPU 重活，libuv 池幫不上，得用 worker threads。機制本身不難想像：主執行緒維護一組（或幾個）worker，收到要縮放的請求時，把圖的資料 `postMessage` 給某個 worker，worker 在自己的 isolate 裡算完，再 `postMessage` 把結果送回來。主執行緒在這整段時間裡是自由的，照常服務別的請求。全線凍結的問題消失了。

但「搬」這個動作本身有成本，而且成本藏在兩個地方，不認清就會把一個問題換成另一個問題。

**第一個成本是啟動。** 開一個 worker 不是免費的——它要起一個全新的 V8 isolate，這是個相對重的操作，量級在**數十毫秒**，而且每個 worker 還帶著大約 **10 MB** 的記憶體開銷（自己的堆、自己的 runtime）。這直接決定了一件事：**你不該每個請求都臨時開一個 worker 再關掉。** 那樣的話，光是開關 worker 的開銷就可能比計算本身還貴，而且高並發下你會瞬間開出成百上千個 isolate，記憶體爆掉。正確的形狀是**池化**——啟動時就建好一組固定數量的 worker（通常對齊 CPU 核心數），讓它們常駐，請求來了就派給空閒的那個，算完歸還、不關掉。這正是為什麼生產上幾乎沒人裸用 `worker_threads`，而是用 Piscina 這類 worker pool 函式庫——它把「池化、派工、排隊、逾時、結果回傳」這套包好了。這個演化路徑跟連線池一模一樣：昂貴的東西重用、不要每次重建（連線池的機制是另一個主題，見〈連線池：為什麼不每次都重開連線〉）。

**第二個成本是資料搬運。** worker 各自有獨立的 isolate、記憶體不共享，所以主執行緒跟 worker 之間傳資料，不是傳個指標就好。`postMessage` 預設走的是 **structured clone（結構化複製）**——它把你傳的物件**整個複製一份**到對方的 isolate。傳個小設定物件無所謂，但如果你要傳一張 50 MB 的圖進去、再把結果傳回來，這一來一回兩次深複製的成本可能就吃掉了你並行計算省下來的時間。Node 給了兩條繞過複製的路：

- **transfer（轉移所有權）**：對 `ArrayBuffer` 這類，你可以把它列進 `postMessage` 的 transfer list，這時底層記憶體是**移交**過去、不是複製——零拷貝，但代價是**轉移之後，發送端那一側就再也不能用它了**（記憶體所有權整個搬走，原本的 handle 變成不可用）。適合「算完就不需要原資料」的單向交接。
- **`SharedArrayBuffer`（共享記憶體）**：這塊記憶體**兩邊的 isolate 同時看得到**，誰都能讀寫，完全不複製。它不能放進 transfer list（因為它本來就是共享的，沒有「所有權」要轉移）。代價是：一旦多個執行緒能同時讀寫同一塊記憶體，你就把 race condition 從別處請了回來——而 worker 之間沒有鎖可用，要安全協調得靠 `Atomics` 提供的原子操作（共享可變狀態下的競態與原子性，是另一個主題，見〈race condition 與原子性：跨 DB、cache、runtime〉）。

判斷要不要搬、怎麼搬，有一條乾淨的判準：**這段計算會在主執行緒上跑超過幾毫秒嗎？** 低於 1 毫秒的小計算（解個小 JSON、做個簡單轉換），別動它——外送的序列化與排程開銷比計算本身還大，搬了反而虧。數十到數百毫秒的（影像縮放、加密、大批資料聚合），用 worker pool，把它從主執行緒挪開值回票價。秒級或可以延後的（產報表、批次 ETL），那根本不該待在請求路徑上，丟進佇列由背景 worker 機群非同步處理（這又引入交付語意與重試的問題，是訊息那一塊的主題，此處不展開）。

## 一個常被搞錯的岔路：worker threads 不是用來擴展連線的

worker threads 解的是「**單一請求內**的 CPU 重活」。它**不是**用來擴展「每秒能處理多少個網路請求」的——這是一個高頻誤配，值得單獨點出來。

要榨乾一台多核機器的網路吞吐，用的是 **cluster**（或更外層的多行程部署）：fork 出多個完整的 Node 行程，每個有自己的 event loop、各吃一顆核，靠作業系統把連線分散到這些行程上。它擴展的是「同時處理的連線數／請求數」。worker threads 擴展的是「一個請求內那段擋路的計算」。官方文件把這條線劃得很清楚：worker threads 適合 CPU 密集的 JavaScript 運算，對 I/O 密集的工作幫不上多少忙——因為 Node 內建的非同步 I/O 本來就比 worker 更有效率（網路 I/O 走 epoll，本來就不佔執行緒，根本沒有 CPU 要平行）。

兩個量綱不一樣：一個是「我這條請求裡有一坨算不完的東西擋住了主執行緒」（→ worker threads，把那坨搬走），一個是「我這台 16 核的機器只用了 1 核在接請求」（→ cluster，多開幾個行程吃滿核）。把 worker threads 拿去當 cluster 用，你會發現連線吞吐根本沒上去——因為網路 I/O 從一開始就不是 CPU 瓶頸，平行化一個本來就不忙的東西，當然沒有意義。

順帶收掉幾個還在路上的誤解。**「開越多 worker 越快」**——不對，worker 數量一旦超過 CPU 核心數，它們就互相搶核、徒增 context switch，吞吐反而往下掉，對齊核心數（或核心數 ±1）通常最甜。**「worker 之間記憶體共享很便宜」**——只有 `SharedArrayBuffer` 是真零拷貝，普通物件走 `postMessage` 是 structured clone 深複製，傳大物件有實打實的序列化成本。**「worker 算完就高枕無憂」**——如果結果送回主執行緒後，你又在主執行緒上同步地 `JSON.stringify` 一個巨大的結果物件，那等於白搬：你把計算移走了，卻把序列化這個 CPU 重活留在了主執行緒上，全線照樣凍。搬，要連著結果處理一起想清楚搬到哪為止。

## 為什麼是這個形狀

退一步看，這整套機制——libuv 那個你碰不到的 4 條池、加上你自己開的 worker isolate——全都是從一個無法迴避的事實裡長出來的：**event loop 用「整個模型只有一條執行緒在跑 JS」換來了「沒有資料競爭、極省資源、海量並發 I/O」這份大紅利，而這份紅利的前提是沒有人霸佔那條執行緒。** CPU 重活恰恰是霸佔者。

於是解法不可能是「讓 event loop 變快」——它已經夠快了，問題從來不在速度，在「獨佔」。解法只能是**把霸佔者請出去**：作業系統層面沒有非同步原語的那些操作（fs、dns、crypto、zlib），libuv 默默用一個小執行緒池替你扛走，扛走得有限（4 條），所以池會排隊、所以網路 I/O 不屑於用它（epoll 更好）。而你自己的 JS 計算，因為 V8 單一 isolate 鎖死了單執行緒，只能靠開新 isolate 來真正平行——這就是 worker threads，它買到「真平行跑 JS」，付出「開銷數十毫秒、每個 10 MB、跨 isolate 傳資料要複製或顯式轉移」的代價，所以要池化、要小心搬資料。

下次你看到一個 Node 服務在某個功能上線後 p99 集體爆炸、CPU 卻沒滿，你會知道往哪裡看：不是去調快那段計算，而是去找出**是誰在主執行緒上算了太久**，然後決定該把它交給 libuv 那 4 條（如果它本來就是 fs/crypto），還是搬進一個 worker（如果它是你自己的 JS）。這個機制的全部形狀，就是「單執行緒的紅利」與「CPU 重活的詛咒」之間，那條精心劃出的疏散路線。
