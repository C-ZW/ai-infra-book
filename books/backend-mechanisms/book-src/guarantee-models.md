# 一致性與交易的保證模型：ACID、BASE、CAP、PACELC

兩個工程師在白板前吵了二十分鐘，誰也說服不了誰。一個堅持「我們的系統有保證一致性」——他指的是資料庫每筆交易都不會破壞「餘額不為負」這條規則。另一個搖頭：「不對，我們明明是最終一致的，副本之間會落後好幾秒」——他指的是多台機器讀到的值可能不一樣。第三個人插嘴：「分散式系統本來就只能三選二，要一致性就得放棄可用性」——他指的又是第三種東西。

三個人嘴裡都是「consistency」這個字，腦子裡卻是三件**毫不相干**的事。這不是誰程度差，而是這個詞被產業界**嚴重多載**了。ACID 的 C、CAP 的 C、最終「一致」性的一致——它們碰巧共用一個英文單字，意思卻南轅北轍。任何關於資料保證的討論，只要沒先把這三個 C 拆開，就注定各說各話。

這一章要做的，正是把這團糾纏拆乾淨。ACID、BASE、CAP、PACELC 這四個縮寫，常被擺在一起當成「四個要背的名詞」，但它們其實在回答**不同層次**的問題：交易在單機上保證什麼（ACID）、放棄這些保證換來什麼（BASE）、網路被切開時你被迫做什麼選擇（CAP）、以及——這是最少人講清楚的——網路**好端端**的時候你每天在付什麼代價（PACELC）。把這四個放進同一個故事，你會看到它們不是四張要記的卡片，而是同一條主軸上的四個刻度：**你想要哪一種「一致」、你願意為它付多少延遲與可用性。**

## 第一個 C：交易把資料庫從一個合法狀態帶到另一個

從最老、也最沒爭議的那個 C 開始。當你寫下一筆銀行轉帳——A 扣 100、B 加 100——你預設這兩步要嘛一起成功、要嘛一起不發生。你不會接受「A 扣了但 B 沒加」這種中間態被持久化下來。這個直覺，就是 ACID 在保護的東西。

ACID 是傳統交易資料庫給你的契約，四個字母各管一件事，值得逐一看清它們到底保證什麼、又靠什麼成立：

- **Atomicity（原子性）**：一筆交易裡的所有操作，要嘛全做、要嘛全不做。轉帳轉到一半當機，重啟後不該看到「扣了沒加」。它靠的是 undo log——交易未提交就崩潰，重啟時依 log 回滾到彷彿沒發生過。
- **Consistency（一致性）**：交易把資料庫從一個**合法狀態**帶到另一個**合法狀態**——所有宣告過的不變量（約束、外鍵、唯一性、`CHECK`、觸發器）在交易前後都成立。注意：這個 C 其實是 A、I、D 加上你自己定義的約束**共同**撐出來的結果，而不是一個獨立機制。**它就是那個會跟另外兩個 C 撞名、害人吵架的字。**
- **Isolation（隔離性）**：並發執行的交易，效果要像它們是按某個順序一筆一筆跑的，彼此不互相干擾。這是四個字母裡最微妙的一個——「隔離」其實有好幾個強度等級，弱一點的等級會放行一些反直覺的異常（這套等級與異常本身值得單獨一章，見〈隔離級別與異常〉）。
- **Durability（持久性）**：一旦回覆 client「commit 成功」，這筆資料就算斷電也不會丟。它靠的是先把變更寫進預寫日誌並 fsync 落盤，才回覆成功（見〈WAL：先寫日誌，再改資料〉）。

這四條撐起一個極強的承諾：你可以把一段並發的、可能中途失敗的複雜操作，當成一個**不可分割、不被干擾、寫下去就算數**的整體來推理。資料庫幾十年的工程，很大一部分就是在便宜地兌現這個承諾。

但這裡藏著本章第一個非顯然的點，先釘住它，後面會反覆用到：**ACID 的 I（隔離）和你以為的「最強一致」不是同一件事。** 最強的隔離等級叫 serializable（可序列化），它保證的是「多筆交易的效果等於某個序列執行」。而後面要講的 CAP 的 C——linearizability（線性一致）——保證的是「**單一物件**上的讀寫，看起來在真實時間的某一瞬間原子地生效」。一個管的是**多物件交易的可序列化**，一個管的是**單物件的即時性**。Herlihy 與 Wing 在 1990 年定義 linearizability 時就點明：它本質上是 **strict serializability（線性一致＋可序列化合起來）的特例——當交易退化成對單一物件的單一操作**。這也正好解釋了它為什麼又跟（一般的）serializability **正交**：兩者各管一軸（一個管單物件即時性、一個管多物件可序列化），合起來才是 strict serializability，所以你可以只有其中一軸——一個系統可以線性一致卻不可序列化，也可以可序列化卻不線性一致。光是 ACID 內部，「一致」這個概念就已經分岔成兩條路了。

## 第二個 C 與它的反義詞：BASE 是怎麼長出來的

ACID 這麼好，為什麼有人要放棄它？

因為它有一個沉默的前提：**所有資料都在一個地方，或至少在一個能被單一交易協調器統一指揮的地方。** 一旦你的資料被複製到多個副本、切分到多台機器、甚至散落在不同的地理區域，ACID 那套「把所有相關操作鎖在一個原子交易裡」的做法就開始變得昂貴，甚至不可能——跨機器的原子提交需要分散式交易協定，而那是出了名的慢與脆（見〈2PC 與分散式交易〉）。

2000 年代中期，大規模網路服務撞上了這堵牆。它們的資料量與流量遠超單機，又必須**永遠在線**——購物車不能因為某個資料中心斷線就掛掉。於是一批人做了一個哲學上的反向選擇，並刻意取了一個跟 ACID（酸）對立的化學名：**BASE（鹼）**。這個對立的命名可追溯到 Eric Brewer 1990 年代末、2000 年 PODC keynote 一帶的工作，而真正把它系統化論述、在產業界推廣開來的，是 Dan Pritchett 2008 年的〈BASE: An Acid Alternative〉（ACM Queue）：

- **Basically Available（基本可用）**：系統絕大多數時候都能回應，故障或分區時可能降級（回舊值、回部分結果），但**不整個倒下**。
- **Soft state（軟狀態）**：系統的狀態可能**自己**隨時間改變，即使沒有新的外部輸入——因為副本在背景互相收斂、覆寫。ACID 的世界裡狀態只在交易裡變，BASE 的世界裡狀態是「軟」的。
- **Eventual consistency（最終一致）**：如果停止寫入，所有副本**最終**會收斂到同一個值。沒說多久，過程中讀到什麼都有可能。

這就是第二個 C 登場的地方——而且它和 ACID 的 C 幾乎沒有關係。ACID 的 C 是「不變量在每筆交易後都成立」，是一個**瞬時、強硬**的承諾；最終一致的「一致」是「副本們最後會收斂」，是一個**關於未來、且沒有時限**的承諾。把這兩個 C 混為一談，是白板前那場爭吵的根源之一。

關鍵在於理解 ACID 和 BASE **不是好壞之分，而是取捨優先序的兩端**。ACID 為**正確性**買單：用協調（鎖、日誌、共識）換來強硬的即時保證，代價是延遲與擴展性。BASE 為**可用性與規模**買單：放鬆即時一致，換來高可用、低延遲、近乎無限的水平擴展，代價是你得自己面對「中間態被讀到」這件事。沒有哪一端永遠對。

而真正成熟的系統，從來不在這兩端之間二選一——它在**系統的不同部位放不同的選擇**。

## 一個貼文系統，同時是 ACID 又是 BASE

把這件事講具體。設想一個社群貼文系統，看它如何在同一個產品裡同時用上兩種哲學。

「發一篇文」這個動作走 ACID。它要在單筆交易裡做兩件事：寫入貼文記錄、把使用者今天的發文額度減一。這裡有一條神聖的不變量——**每日發文額度不能被扣成負數**。如果兩個請求並發進來，各自讀到「還剩 1 則額度」，然後各自扣減、各自放行，這個使用者就發了兩篇、額度卻只夠一篇。要擋住這種事，這筆操作必須是原子且隔離的：兩個並發交易裡，只有一個能成功扣到最後那一格，另一個必須看到額度已耗盡而被拒。這就是 ACID 的 A 與 I 在保護一條業務不變量——也就是 ACID 的 C。

「這篇文有幾個讚」走 BASE。設想一篇爆紅貼文，十萬人在幾秒內同時按讚。如果每個讚都進一筆強一致交易去 `UPDATE posts SET likes = likes + 1 WHERE id = ?`，那一行 `likes` 就變成所有寫入排隊爭搶的單點——每個交易都要鎖住那一行、加一、解鎖，下一個才能進。寫入被**序列化**了，這一行的吞吐被單行鎖爭用死死壓在每秒數百筆的量級。

換成 BASE 怎麼做？讓計數**不必即時精確**。各個應用節點在本地各自累加自己收到的讚（或寫進一個分散的計數結構），背景再把這些零散的增量彙總、收斂成總數。沒有任何一行被全域鎖住，每個節點平行地加自己的，吞吐輕鬆躍到每秒數萬筆。代價是什麼？某個瞬間你刷新頁面，看到的讚數可能比真實值少了數千（每秒數萬筆的流入、慢一兩百毫秒才追上，差的就是這段時間累積的量），但那點落差沒人在乎。

問問自己：有人會因為「讚數慢 200 毫秒更新」而受到實質傷害嗎？不會。但有人會因為「發文額度被扣成負數、或座位被重複賣出」而受傷嗎？會，而且是真金白銀的傷。**這就是判準**——資料承載的是不能破的不變量，就上 ACID；資料是「慢半拍沒人在乎」的高頻動態，就上 BASE。同一個貼文系統，發文 ACID、計數 BASE，不是矛盾，是教科書式的正確設計。

這裡藏著一個要命的反模式，值得單獨警告：**別讓 BASE 資料偷偷承載不變量。** 最常見的版本是拿一個最終一致的計數器當「庫存」。它平時看起來工作得很好，直到某個瞬間兩個請求都讀到「還剩 1 件」、都判斷「可以賣」、於是超賣。最終一致的計數器永遠不該守護「不能為負」這種不變量——那是 ACID 的活，交錯了就等著在某個高並發的夜晚出事。

## 第三個 C：當網路被切開，你被迫選邊

前兩個 C 都還在「單機 vs 多機」的層次打轉。第三個 C 把問題推到分散式系統的核心：**當機器之間的網路斷了，你還能保證什麼？**

這就是 **CAP 定理**。它由 Eric Brewer 在 2000 年的 PODC keynote 上以**猜想（conjecture）**提出，兩年後由 MIT 的 Seth Gilbert 與 Nancy Lynch 在 2002 年以反證法**形式化證明**為定理（論文〈Brewer's Conjecture and the Feasibility of Consistent, Available, Partition-Tolerant Web Services〉，ACM SIGACT News）。三個屬性：

- **Consistency（一致性）**：這裡指的是 **linearizability（線性一致）**——所有節點看起來共享同一份資料，任何讀取都拿到最近一次寫入的值。**注意這個 C 又跟前兩個不同**：它不是 ACID 的「不變量」，也不是「最終會收斂」，而是「**即時、單物件、與真實時間一致**」的最強讀寫保證。
- **Availability（可用性）**：Gilbert 與 Lynch 給的精確定義是——**每個被非故障節點收到的請求，都必須得到回應**，但不保證這個回應含有最新的資料。注意這跟工程上講的「高可用」不是一回事：它要求的是「就算這個節點被完全孤立，它也得回個答案」。
- **Partition tolerance（分區容忍）**：當網路把節點切成互不相通的幾群（partition），系統仍能繼續運作。

CAP 最常被念成「三選二」，而這個念法**幾乎是錯的**，是所有 CAP 誤解的源頭。問題出在它把 P 當成一個你能主動放棄的選項。但在一個真正的分散式系統裡，**網路分區不是你能選擇要不要的——它是會發生的物理現實。** 封包會丟、交換器會壞、跨區光纖會被挖斷，分區遲早來。能「放棄 P」的只有單機系統（根本沒有節點間網路可斷）。

所以正確的讀法是：**P 不是選項，是前提。真正的選擇只在分區發生的那一刻浮現，而且只在 C 和 A 之間二選一。** 當網路把你的副本切成兩半、它們無法互相確認誰有最新資料時，你面前只有兩條路：

- **保 C，犧牲 A**：拒絕在「可能不是最新」的副本上回應。少數派那側的節點寧可回錯誤、不服務，也不冒險回一個過時的值。代價是這段期間部分使用者連不上——你選擇了「寧可不可用，也不給錯答案」。
- **保 A，犧牲 C**：照常回應，接受回出去的可能是分區前的舊值。每個節點都還活著、都回答，代價是兩側的資料在分區期間各自漂移、不再一致——你選擇了「寧可給舊答案，也不讓使用者吃閉門羹」。

沒有第三條路。在分區的那一刻，**你不可能同時保證「每個節點都回應」又「每個回應都是最新」**——因為被切到另一側的最新寫入，孤立節點根本看不到。這就是 Gilbert-Lynch 證明的硬核：它用反證法說明，假設一個系統同時是 C 和 A，構造一個分區，就一定能逼出一次「回應了、但回的是舊值」的矛盾。

一個務必釘死的屬性別記錯：**CAP 是 Brewer 的猜想（2000）、Gilbert-Lynch 的證明（2002），別把提出者和證明者混為一人。** 而且如同前面反覆強調的——CAP 的 C 是 linearizability，跟 ACID 的 C（不變量）、最終一致的「一致」是**三個不同的概念**，只是不幸共用一個字。

## CAP 沒說的那一半：沒分區的時候你在付什麼

CAP 有一個巨大的盲區，而它正是 CAP 在實務上最容易誤導人的地方：**CAP 只在分區發生時才開口說話。** 可是分區是罕見的——一個健康的系統，99.9% 以上的時間網路都是好的。那剩下這 99.9% 的時間，CAP 什麼都沒告訴你。

於是問題來了：網路好端端的時候，強一致是免費的嗎？

不是。這就是 **PACELC** 要補上的那一半。它由 Daniel Abadi 提出（2010 年部落格初提，2012 年在 IEEE Computer 的論文〈Consistency Tradeoffs in Modern Distributed Database System Design: CAP is Only Part of the Story〉裡正式形式化）。這個縮寫本身就是它的完整陳述，讀法是：

```
PACELC = if (P) then (A or C) else (L or C)

  if Partition         -->  choose Availability or Consistency
  else (no partition)  -->  choose Latency or Consistency
```

前半段 `P → A or C` 就是 CAP，沒有新東西。**真正的貢獻是後半段 `E → L or C`**：即使沒有任何分區，只要你想要強一致，就得付**協調延遲（latency）**的代價。

為什麼？因為「線性一致」這個保證——讓所有節點看起來共享同一份即時資料——本身就需要副本之間互相溝通、確認。設想一筆寫入要保證「之後任何節點的讀都看得到」，它就不能只寫一個副本就回覆成功；它得等到足夠多的副本都確認收到（過半，或某個 quorum，見〈quorum：R + W > N 的鴿巢原理〉），才能宣告成功。這一輪「等副本確認」是要花時間的——一次跨副本往返，在同機房是亞毫秒級，跨可用區（同區域）通常是個位數毫秒、多半落在一兩毫秒，跨地理區可能是上百毫秒。**強一致的價格，就是這一輪往返，而且你在沒有任何分區、一切正常的時候，每一筆都在付。**

PACELC 的洞見因此很尖銳：CAP 讓你以為一致性的代價只在罕見的分區時刻才需要面對，而 PACELC 戳破這個錯覺——**你天天、每一筆操作，都在 L 和 C 之間做取捨，分區只是那個取捨被推到極端（變成 A vs C）的偶發情形。** 一個只懂 CAP 的人會問「分區時我要 C 還是 A」；一個懂 PACELC 的人會多問一句、而且是更常用到的那句：「平時我願意為強一致多付多少延遲？」

## 用 PACELC 給系統做人格分類

PACELC 的好用之處，在於它把一個系統的取捨「人格」壓縮成兩個字母：分區時選什麼（P 後面跟 A 或 C）、正常時選什麼（E 後面跟 L 或 C）。四種組合，各自對應一類典型系統：

| PACELC 類型 | 分區時 | 正常時 | 取捨人格 | 典型代表 |
|---|---|---|---|---|
| **PC/EC** | 選一致（犧牲可用） | 選一致（犧牲延遲） | 最重一致，兩種情況都不讓步 | 傳統單主關聯式資料庫、設成線性一致的系統 |
| **PA/EL** | 選可用 | 選低延遲 | 最重可用與速度，願意吃舊值 | Dynamo 式無主、Cassandra 預設配置 |
| **PA/EC** | 選可用 | 選一致 | 平時盡量一致，分區時才讓步給可用 | MongoDB 預設配置、部分可調系統的「無分區強讀」配置 |
| **PC/EL** | 選一致 | 選低延遲 | 較少見，可調系統能配出來 | Yahoo! PNUTS、可調系統「分區時嚴守一致、平時不強求」的配置 |

讀這張表有個鐵則，否則就會把它讀成又一張要背的速查卡。表裡寫「Cassandra 預設」「單主關聯式」，指的是這些系統在**預設或典型配置下的傾向**，不是斷言某個產品在所有配置下永遠如此。現代資料庫絕大多數是**可調（tunable）**的——同一個 Cassandra，你可以把某次讀寫設成 `QUORUM` 換到比 `ONE` 強得多的一致性（讀取保證涵蓋上一次成功寫入的某個副本），也可以設成 `ONE` 換到最低延遲。PACELC 描述的是「這個系統把旋鈕擺在哪個預設位置、它的設計重心偏向哪邊」，而不是一個不可改變的物理屬性。

那個旋鈕具體是什麼？在 quorum 系統裡，就是三個數字：寫入要幾個副本確認（W）、讀取要讀幾個副本（R）、總共幾個副本（N）。只要 **R + W > N**，讀寫的 quorum 就一定有交集——你讀的那組副本裡，保證至少有一個在**上一次成功完成的寫入**的 W 組裡，於是讀得到那次寫入的值（這個鴿巢交集的機制，在〈quorum：R + W > N 的鴿巢原理〉裡深講）。把 R、W 調大，你買到更強的一致、付出延遲；調小，你買到低延遲、冒讀到舊值的風險。PACELC 的每一個字母，最後都落到這幾個你能真的去改的數字上。

但這裡得釘住一條容易踩過頭的界線，否則就把 quorum 一致誤當成本章定義的 CAP 之 C：**R + W > N 的交集只保證讀到「上一次成功完成的寫入」，並不等於 linearizability。** 並行寫入、部分寫入沒有 rollback、read-repair 的時機與時鐘偏移，都可能讓兩個讀者在同一瞬間看到不同的值——quorum 給的是「讀得到某次已完成的寫入」，不是「即時、單物件、與真實時間一致」的最強保證。要在 Dynamo 式系統上拿到真正的單物件線性一致，得另外動用單鍵共識（如 Cassandra 的 Lightweight Transactions，底層走 Paxos / `SERIAL`）。換句話說，調大 quorum 買到的是「強得多的一致」，而不是免費的 CAP 之 C。

這裡也有個悄悄出錯的陷阱：如果你把 R + W 配成 **小於等於 N**，讀寫 quorum 可能不相交，你就會在**沒有任何錯誤訊息**的情況下讀到舊值。系統不會報錯——它只是默默地給你一個過時的答案。配錯 quorum 是那種「測試時看起來一切正常、上線後在某個並發時刻悄悄回錯資料」的 bug。

## 把四個 C 並排，看清它們各管什麼

繞了一圈，現在可以把開場那場白板爭吵徹底解開了。四個框架嘴裡的「一致」，其實是四個各自獨立的概念，釘死它們的分野，以後再聽到誰說「一致性」，你會本能地反問「你說的是哪個 C」：

| 出處 | 「一致」指的是 | 範圍 | 一句話 |
|---|---|---|---|
| **ACID 的 C** | 不變量在每筆交易前後都成立 | 單機交易內 | 約束沒被破壞（由 A/I/D + 你的約束共同撐出） |
| **CAP 的 C** | linearizability：即時讀到最新寫入 | 單一物件、跨副本 | 像只有一份資料、與真實時間一致 |
| **最終一致的「一致」** | 副本最終會收斂到同一值 | 多副本、關於未來 | 停止寫入後總會收斂，但沒說多久 |
| **serializable（ACID 的 I 最強級）** | 多筆交易效果等於某個序列執行 | 多物件交易 | 可序列化，與 linearizability 正交 |

最後一行特別容易和 CAP 的 C 黏在一起，再強調一次它們的分野：**linearizability 管「單一物件的讀寫即時性」，serializability 管「多筆交易的可序列化」。** 一個資料庫可以對每個單鍵都線性一致、卻允許多筆交易交錯出不可序列化的結果（例如 write skew——兩筆交易各自讀了對方即將改的資料、各自基於舊快照做決定，合起來破壞了一條跨列的不變量）；反過來，一個系統也可以可序列化、卻不對外保證每次讀都拿到掛鐘意義上的最新值。把這兩個混為一談，會讓你對「我的資料庫到底保證了什麼」產生根本性的誤判。

這也是為什麼前面說 ACID、BASE、CAP、PACELC 不是四張平行的卡片。它們在回答**不同層次**的問題：ACID 問「單機交易給我什麼」，BASE 問「放掉這些換來什麼」，CAP 問「分區時我被迫選什麼」，PACELC 問「平時我每筆在付什麼」。串起它們的，自始至終是同一條主軸——**你要哪一種一致、願意付多少延遲與可用性。**

## 為什麼是這個形狀

退一步看，這四個框架的存在，都是被同一個無法迴避的物理事實逼出來的：**資料一旦離開單一節點、複製到多個地方，「現在的真相是什麼」就不再有一個唯一、即時、免費的答案。**

在單機上，記憶體與磁碟給你一個天然的「現在」——寫下去，下一個讀就看得到，ACID 在這個基礎上把一組操作打包成原子整體，代價可控。可一旦資料散到多台機器，光速與封包延遲就介入了：任何「讓所有副本同意當前值」的動作都需要溝通，而溝通要時間、要對方還活著、要網路還通。於是強一致從「免費的預設」變成「要付費的選項」——平時付延遲（PACELC 的 E），分區時付可用性（CAP 的 P）。BASE 則是乾脆不買這個強一致，改用「最終收斂 + 自己兜底」換取規模與永遠在線。

四個縮寫，四個不同的提問角度，但底下是同一個真相在不同光照下的投影。而那個害三個工程師在白板前吵了二十分鐘的「consistency」，之所以同時指涉不變量、即時性、與最終收斂三件事，正是因為這個詞被迫去描述「在多副本世界裡，真相的不同面向」——每一個面向，都是你可以單獨選擇、單獨付費的。下次再有人說「我們要保證一致性」，你知道該先問的不是「怎麼做」，而是：**你說的，是哪一個 C？**
