# 為什麼分散式這麼難：partial failure 與失敗模型

寫單機程式久了，會養出一個沉在骨子裡、平常不會注意到的假設：**一個函式呼叫，要嘛回傳結果，要嘛拋例外，要嘛整個行程死掉。** 它不會「回傳到一半人就走了」。你呼叫 `db.save(order)`，最壞的情況是它丟一個 exception 給你接，或者整台機器當掉、連同你的 process、你的 try-catch、你的整個世界一起消失。無論哪種，事後你都能說清楚「這次到底成了沒有」。在單機裡，失敗是一件 **all-or-nothing** 的事：要嘛大家都活著，要嘛大家一起死。你從來不必處理「記憶體的左半邊還在跑、右半邊已經死了、而左半邊對此一無所知」這種事。

把同一行 `db.save(order)` 搬到網路另一頭——資料庫是另一台機器，中間隔著交換器、網卡、幾段光纖——這個讓你舒服了十年的假設**就地崩塌**。現在 save 可以用一種單機裡根本不存在的方式失敗：你送出了請求，然後**等不到回應**。不是 exception，不是 crash，就只是……安靜。你的 process 好端端活著，對方那台機器可能也好端端活著，但你們之間那條線出了狀況，而你被困在「不知道剛才那筆訂單到底存進去沒有」的迷霧裡。這個迷霧，就是分散式系統一切困難的源頭，它有個名字：**部分失敗（partial failure）**。

這一章不講任何具體的協定。它要把這層迷霧本身講透——為什麼它在原則上無法被驅散、它會以哪幾種形狀出現、工程上我們是靠什麼姿勢與它共存。後面那些看起來繞了一大圈、付出昂貴代價的機制（共識、租約、fencing、逾時、冪等重試），它們繞的每一個圈，都是在繞這同一團迷霧。先看清楚敵人長什麼樣，後面那些防禦才不會像一堆無謂的儀式。

## 那個等不到的回應，背後藏著幾種真相

把「`save` 送出去了、回應沒回來」這件事放到顯微鏡下。從你這端看出去，畫面只有一個：沉默。但這個沉默底下，可能是至少四種**截然不同**的真相：

1. 請求在路上就丟了，對方**根本沒收到**，訂單沒存。
2. 對方收到了、正在處理，回應**還在生**，訂單即將存進去。
3. 對方**處理完了、訂單已經存好**，但那聲「成功」的回應在回程路上丟了。
4. 對方那台機器**整個當機了**，訂單八成沒存（也可能存了一半）。

這四種真相，要你做的事完全相反。情況 (1) 你該重送；情況 (3) 你**絕對不能**重送（否則同一筆訂單會存兩次）；情況 (2) 你該再等等；情況 (4) 你該放棄這台、改打別台。可是——這就是 partial failure 全部的惡意所在——**從你這端，這四種情況長得一模一樣，都只是「等不到回應」。** 你手上沒有任何資訊能把它們區分開。

你唯一握著的工具，是逾時（timeout）：等夠久了還沒回應，就「當作」它失敗了。但 timeout 從不告訴你真相，它只告訴你「它沒在我願意等的這段窗口內回應」。一個被你判定逾時、當作死掉的節點，可能下一毫秒就把回應送來了——它只是慢，不是死。**「慢」和「死」在一個延遲沒有上界的網路裡，從外部觀察，是無法區分的兩件事。** 這句話是整個分散式系統理論的地基，值得停在這裡多看兩眼。它之所以無法區分，是因為要區分「慢」和「死」，你得知道「最多會慢到多久」——超過這個上界還沒回應，才能斷定是死。但真實網路給不出這個上界：一個封包可能因為某台中繼路由器的佇列塞滿而延遲幾秒，一個 process 可能因為一次 stop-the-world 的垃圾回收、一次 page fault、甚至一個 `SIGSTOP` 而**整個凍結數秒鐘**，期間它對外界完全沉默、和死了沒有任何區別——然後若無其事地醒來，繼續執行它凍結前的那行程式碼。沒有任何有限的 timeout 能保證「等到這麼久還沒回，它就一定是真死」。

## 連兩個人都沒辦法可靠地約好一件事

你可能會想：好吧，回應會丟，那我讓對方**確認**收到不就好了？我送請求、它回 ack、確認這筆事成了。但這條看似顯然的補救，藏著一個更深、更古老的不可能性，把它攤開來看，才會真正死心。

設想最小的協作場景：兩個將軍，分駐在敵城兩側的山頭上，中間隔著敵人控制的山谷。他們要約好「黎明同時進攻」——**只有同時進攻才會贏，任何一方單獨進攻都會全軍覆沒。** 唯一的通訊方式是派傳令兵穿過山谷，而傳令兵**可能被攔截**（訊息可能遺失）。問題來了：他們有辦法可靠地約定一個共同的進攻時刻嗎？

將軍 A 送出「黎明進攻」。但他不敢就這樣進攻——萬一傳令兵被抓了、B 根本沒收到呢？所以他得等 B 回一個「收到，我也黎明進攻」。可是 B 送出這個確認後，B 同樣不敢進攻——萬一**這個確認**在回程被攔截、A 沒收到、A 以為計畫沒談成而按兵不動呢？那 B 單獨進攻就是送死。於是 B 也需要 A 確認「我收到你的確認了」。而 A 送出這第三封信之後……你看出來了：**每一封信都需要下一封信來確認它送達，而那封確認信本身又可能遺失、又需要再下一封。** 這個確認鏈沒有盡頭。無論他們交換了多少封信，**最後送出那封信的人，永遠無法確定它有沒有送達**，於是永遠無法確定對方是否與自己處在同一個共識上。

這就是 **Two Generals' Problem（兩將軍問題）**——Akkoyunlu、Ekanadham、Huber 在 1975 年首次給出它的不可能性證明，Jim Gray 在 1978 年替它取了這個名字。它證明的事冷酷而徹底：**在一個訊息可能遺失的通道上，兩方無法在有限步內保證達成共識。** 注意這裡只有兩個節點、只有訊息遺失這一種故障——連節點當機都還沒算進來——就已經不可能了。partial failure 不是「工程上很難處理」的麻煩，它在最乾淨的數學模型裡就已經是一道無法翻越的牆。我們後面所有的機制，都不是在「解決」它（它無解），而是在它劃下的牆內，盡可能把不確定性壓到一個能接受的程度。

## 失敗的光譜：節點可以用幾種方式壞掉

partial failure 之所以難纏，還有第二層：節點壞掉的「方式」不只一種，而是一道由溫和到惡毒的光譜。你的協定設計成能扛住哪一層，決定了它的成本與它真正買到的安全。把這道光譜看清楚，是選對工具的前提——很多過度設計，源自把節點想得比它實際上更壞。

由弱到強，這道光譜的主幹是一條乾淨的包含鏈——crash ⊂ omission ⊂ timing ⊂ Byzantine，外層真的涵蓋內層所有的壞法（一個崩潰的節點等於漏掉之後所有訊息，一個漏訊息的節點是一種退化的拜占庭行為）。下表把這條主幹攤開，另外插進兩個實務上躲不掉、但嚴格說不在這條鏈上的常客——fail-stop 是 crash 的「可偵測特例」（更友善，不是更弱的一層），crash-recovery 則是另一根正交的軸（崩潰後會不會重啟），擺進來是為了實務直覺、不是因為它們在同一條包含鏈上：

| 失敗模型 | 節點會怎麼壞 | 你能假設什麼 |
|---|---|---|
| fail-stop | 停掉，而且**讓別人知道它停了** | 沉默＝它真的停了，可安全接手 |
| crash-stop | 停掉就不再參與，但別人**得自己推斷** | 沒回應，但不確定是停了還是慢 |
| crash-recovery | 崩潰後**會重啟回來**，記憶體狀態全失 | 沉默的節點稍後可能帶著舊認知復活 |
| omission | 偶爾**漏掉**某些訊息（收或送） | 沒收到不代表沒送、送了不代表會到 |
| timing | 回應**太晚或太早**，超出時序假設 | 慢到像死，破壞所有靠 timeout 的判斷 |
| Byzantine | **任意行為**：說謊、送矛盾訊息、被攻陷 | 什麼都不能假設，連它的回答都可能是假的 |

最左邊的 **fail-stop** 是理論上最友善的世界：節點要嘛正確運作，要嘛乾淨地停掉，而且**停的時候會留下一個明確的訊號**讓其他人知道「我死了」。在這個模型裡 partial failure 其實不太痛——沉默就等於死亡，你可以放心接手。問題是，**現實幾乎不給你 fail-stop**。當機的機器不會臨終前廣播遺言；它就是突然不說話了，而你分不清那是死亡還是一次特別長的沉默。

往右一格的 **crash-stop** 才是大多數人腦中「機器掛了」的樣子：它停了、不再參與，但**沒有人通知你**，你得自己從沉默裡推斷。再往右，**crash-recovery** 是真實後端的常態，也是最容易被忽略而埋雷的一層：那台你以為已經死透、已經把它的工作接手過來的機器，**會重啟回來**。它重啟時記憶體裡的一切都沒了（只剩之前 fsync 到磁碟的部分），帶著一個過時的世界觀重新加入叢集——它可能還以為自己是 leader，還想繼續執行它崩潰前那一刻的動作。一個只設計來扛 crash-stop、沒料到「死人會詐屍」的系統，會在這裡被狠狠咬一口。這也是為什麼嚴肅的協定都要給每一輪協作蓋一個遞增的世代編號（term 或 epoch）：一個帶著舊編號詐屍回來的節點，它的指令一出口就會被認出是上一輩子的、當場駁回。

中間的 **omission** 和 **timing** 常被初學者跳過，但它們恰恰是 partial failure 的本體。omission 是「訊息有時候會漏」——你送了它沒到，或它送了你沒收到，不是節點死了，只是這一則蒸發了。timing 是「回應來得太晚」——節點沒死、訊息也沒丟，只是慢到超過了你的 timeout，於是你**誤判**它死了。前面那個凍結數秒的 GC 暫停，就是一次教科書級的 timing failure：節點完全健康，只是慢得像死人，而任何靠「等夠久＝死了」做判斷的機制，都會在這一刻被它騙過去。

最右邊的 **Byzantine（拜占庭）** 是最黑暗的模型：節點可能**任意地壞**——回傳偽造的資料、對不同的詢問者給出互相矛盾的答案、甚至被攻陷後主動配合攻擊者。要在這種模型下達成一致，代價陡升：Lamport、Shostak、Pease 在 1982 年證明，要容忍 m 個會說謊的節點，你至少需要 **n ≥ 3m + 1** 個節點（光靠口信的話）。容忍一個叛徒，就得有四個節點；容忍兩個，得七個（這條 3m+1 下界是口信模型的結果，若節點能用不可偽造的簽章背書訊息，需要的節點數可以更省）。這個三倍以上的成本，只有在「參與者真的可能互相欺騙」時才付得值得——跨組織的去中心化帳本、公開的區塊鏈。而在自家機房裡，所有節點都是你自己開的，它們只會當機、不會說謊；在這裡上 Byzantine 容錯，是請了一支反間諜部隊來防一個根本不存在的內鬼，昂貴而荒謬。一個常見的誤判是：某個節點因為 bug 回了錯資料，就驚呼「這是 Byzantine！」要上 BFT。不是的——bug 造成的錯誤不是一個有策略的對手，它用 checksum、用測試、用對帳（見〈對帳〉）去抓，比 BFT 便宜幾個數量級。

## 把失敗模型選對，能買到多少安全

光譜不是用來背的，它是一個設計旋鈕：**你假設節點最壞會壞到哪一層，就決定了你的協定有多貴、又真正擋住了什麼。** 假設得太樂觀（以為是 fail-stop，結果是 crash-recovery），系統會在詐屍節點手上腦裂；假設得太悲觀（以為要防 Byzantine，其實只會 crash），你付了三倍成本買了用不到的保險。把這個旋鈕轉對，本身就是一筆能算出來的帳。

算一筆給你看。設想一個 3 節點的叢集，每個節點的可用性是 95%（也就是每個節點獨立地、有 5% 的時間是掛的——這是個刻意偏高的故障率，好讓數字明顯）。現在比較兩種設計姿態。

第一種姿態，把系統設計成 **fail-stop 串聯**：任何一個節點掛了，整個服務就不可用。這其實是很多天真的多節點系統的隱性假設——「每個元件都得活著，服務才活」。它的整體可用性是三個節點同時都活的機率：

```
P(service up) = 0.95 × 0.95 × 0.95 = 0.857...   ≈ 85.7%
```

三台 95% 的機器串起來，整體反而掉到 85.7%——**多加機器，可用性不升反降**，因為你給了失敗更多的著力點。這是把 partial failure 當 all-or-nothing 來對待的下場。

第二種姿態，承認 partial failure、把系統設計成 **多數存活即可服務**（只要 3 台裡有 2 台活著，服務就繼續）。現在「不可用」意味著同時有 2 台或 3 台掛掉。算這個機率：

```
P(恰好 2 台掛) = C(3,2) × 0.05² × 0.95 = 3 × 0.0025 × 0.95 = 0.007125
P(3 台全掛)   = 0.05³                  = 0.000125
P(service down) = 0.007125 + 0.000125 = 0.00725  ≈ 0.73%
P(service up)   = 1 − 0.00725          ≈ 99.28%
```

同樣是這三台機器、同樣的硬體故障率，只是把失敗模型的假設從「全活才行」換成「多數活就行」，整體可用性從 **85.7% 跳到 99.28%**——把停機時間從一年約 52 天壓到約 2.6 天。這就是「擁抱 partial failure」具體值多少錢：你沒換更好的硬體，你只是不再假裝失敗是 all-or-nothing，而是設計成讓系統在一部分零件壞掉時繼續運轉。後面那些昂貴的共識協定，本質上就是在實現「多數存活即可進展」這個性質——它們貴，但它們把上面那個 85.7% 變成 99.28% 的魔法，是真的。

不過這筆帳有個前提，正是 partial failure 的回馬槍：「多數存活即可服務」要成立，你得**真的能判斷誰活著、誰是多數**——而我們開頭就證明了，你判斷不準。一個慢節點會被誤判為死，一個被網路孤立的少數派會以為全世界只剩自己。所以「多數存活即進展」不能只靠數人頭，它必須建立在一套能在「分不清慢與死」的前提下**仍然安全**的協定之上。這套協定怎麼把過半變成不會腦裂的安全保證，是共識的核心工作（見〈共識：讓一群會當機的機器同意一件事〉），這裡只需記住：99.28% 那個漂亮數字，不是免費送的，它要用一個能扛住誤判的協定去掙。

## 當迷霧釀成真正的災難：腦裂

把前面所有的線索擰成一股，看它在現實裡造成的最經典災難。partial failure 最危險的時刻，是它**對稱地**欺騙了兩邊。

設想一個系統靠網路連著兩個機房，中間那條線突然斷了 43 秒——只是斷，機器全都好端端活著。從**左邊機房**看出去，右邊沉默了：它收不到右邊的任何訊息，於是合理地推斷「右邊死了，我得接管」。從**右邊機房**看出去，畫面對稱地一樣：左邊沉默了，它也合理地推斷「左邊死了，我得接管」。兩邊都沒做錯任何一步推理——它們只是都被那 43 秒的沉默騙了，各自以為自己是唯一的倖存者。於是兩邊都把自己升為主、都開始接受寫入。同一份資料，在兩個機房分頭長出兩條互相矛盾的歷史。這就是 **split-brain（腦裂）**。

這不是假想。2018 年 10 月 21 日，一次例行的網路維護讓某個大型程式碼托管平台的東岸網路樞紐與東岸主資料中心之間斷線了 **43 秒**。就在這 43 秒裡，負責管理資料庫主從關係的協調器（它跑著 Raft）按部就班地做了一件「正確」的事：它判定原主所在那側（東岸主資料中心）失聯，於是讓另一側（西岸機房）湊出 quorum（法定數）、把寫入切換了過去。問題是原主那側**並沒有死**——在那 43 秒裡，它仍在接受寫入，而這些寫入還來不及複製到另一側。等網路恢復，兩邊各自握著一批對方沒有的資料，誰也不是誰的超集。自動化已經沒法收拾這種分岔——和好兩條矛盾的歷史需要人來判斷哪筆該留、哪筆該丟。那 43 秒的 partial failure，最終換來了**超過 24 小時**的服務降級。一次幾十秒的沉默，因為被自動故障切換當真、在原主還活著時就把寫入切走，放大成了一整天的災難。

腦裂之所以是 partial failure 的「終極形態」，因為它把這一章所有的主題一次性收齊了：「分不清慢與死」（那 43 秒到底是斷線還是對方真死？分不出）、「兩將軍」（斷線時兩邊無法交換任何訊息來協調出一個共同決定）、「失敗模型選擇」（系統假設了一個能自動切換的模型，卻沒處理好「被切掉的那個其實還活著、還在寫」）。防住腦裂，不能靠「讓網路別斷」（它一定會斷），只能靠在協定層面下一條鐵律：**任何時刻，最多只有一邊能取得繼續寫入的權力**。怎麼保證這條鐵律——靠過半交集讓少數派湊不到 quorum、靠 fencing token 讓被廢黜的舊主即使詐屍、它的寫入也會被下游認出是過時的而拒絕——是共識與分散式鎖那幾章的主戲（見〈共識：讓一群會當機的機器同意一件事〉、〈分散式鎖：Redlock、fencing token 與「鎖其實是租約」〉）。這一章只需要你**看見那個洞**：只要系統會在「以為對方死了」時做出接管動作，而那個判斷可能錯，split-brain 的門就是開著的。

## 與不確定性共存的三種基本姿勢

partial failure 無法消除，於是工程上真正的問題從「怎麼讓它別發生」變成「**在它必然發生的前提下，我用什麼姿勢活下去**」。後面整本書的韌性機制，幾乎都是這三種基本姿勢的變奏。先在這裡把姿勢本身講清楚，後面那些機制就不再是孤立的招式，而是同一套心法的展開。

**第一，把「沒回應」當成「可能成功也可能失敗」，於是重試——但重試的前提是冪等。** 既然你分不清那個沉默是「請求沒到」還是「成功了但回應丟了」，最務實的做法是當作前者、重送一次。但這裡有個致命的細節：萬一真相其實是後者（對方已經成功了），你這一重送，就讓同一筆操作執行了兩次。對「查詢餘額」這種操作重複執行無傷，但對「扣款」「發貨」「建立訂單」，重複就是事故。所以**重試與冪等是一對連體嬰**：你不能只擁抱「partial failure 要重試」而忘了「重試必須冪等」，否則你是在用一個故障去製造另一個故障。冪等怎麼設計、冪等鍵的生命週期怎麼管，是另一章的主題（見〈重複是常態：冪等與去重〉）；這裡的重點是看清它為什麼**非有不可**——它不是錦上添花，它是「沉默有歧義」這個事實逼出來的必然。

**第二，把「等待」變成有界的，於是逾時——但逾時只是猜測，不是真相。** 你不能無限期地等一個回應（那會讓整條呼叫鏈跟著一起卡死，把一個節點的慢拖成整個系統的死），所以你必須設一個 timeout。但要清醒地記住：timeout 觸發時，你並沒有得知「它死了」，你只是**決定不再等了**。這個決定可能是錯的——對方可能在你放棄後的下一毫秒回應。所以任何「逾時後就採取行動」的設計，那個行動都必須能承受「我猜錯了、它其實還活著」的後果。把等待設成有界、把逾時與重試和退避組織成一套不會互相踩踏的節奏，是 timeout 與重試那幾章的工作；而「逾時是偵測故障唯一的、且根本不可靠的手段」這個更深的侷限，值得單獨一章看清（見〈失敗偵測：逾時是唯一手段、慢與死分不開〉）。

**第三，絕不單憑一個節點的判斷下不可逆的決定，於是要過半同意。** 一個節點看到的「對方沉默」，可能只是它自己被網路孤立了的錯覺（它以為全世界死了，其實是它自己掉線了）。所以任何「會造成不可逆後果」的決定——誰當 leader、要不要把某個節點踢出叢集、要不要把寫入切到另一邊——都不能讓任何單一節點說了算，而要湊到一個**多數**的同意。多數的妙處在於它的鴿巢性質：任兩個多數必有交集，於是不可能同時存在兩個互相矛盾、又都自認合法的多數——這正是堵死腦裂的數學閘門。這套「過半才算數」如何撐起整個共識協定的安全性，是共識那一章的核心（見〈共識：讓一群會當機的機器同意一件事〉）。

這三種姿勢——冪等重試、有界逾時、過半決議——不是三個彼此獨立的技巧，而是同一個認知的三個面向：**你身處的世界資訊先天不完整，你永遠無法確定遠方那台機器此刻的真實狀態，所以你的每一個動作，都必須在「我可能判斷錯了」的前提下仍然是安全的。**

## 為什麼分散式系統長成這個樣子

退一步，把整章收束成一句話：分散式系統的全部困難，都從一個無法迴避的事實裡長出來——**在一個會延遲、會丟訊息、會獨立壞掉的世界裡，你無法確知遠方一個沉默的節點此刻是慢、是死、還是只是訊息卡在路上。**

正因為分不清慢與死，你不敢無限期等待，於是有了逾時；正因為逾時只是猜測，你不敢單憑它接管，於是接管動作必須能承受猜錯；正因為一個節點的判斷可能只是它自己掉線的錯覺，你不敢讓任何單點獨斷，於是有了過半同意；正因為沉默有歧義、回應會丟，你不敢假設「送出去就成了」，於是有了冪等與重試；正因為兩邊可能對稱地誤判彼此已死，你不敢讓系統在分區時兩邊都繼續寫，於是有了「最多一邊能寫」的鐵律。

這些機制看上去各自繁複、各有各的術語，但它們全是同一道牆下的不同支撐——那道在 1975 年的兩將軍、1985 年的 FLP 不可能性裡就已經被數學鎖死的牆。分散式系統的設計者，從來不是在試圖打穿這道牆（打不穿），而是在牆內，用冗餘、用多數、用持久化、用世代編號，一寸一寸地把「我不確定遠方發生了什麼」這團迷霧，壓縮到一個能讓帳目對得起來、讓使用者察覺不到的程度。當你下次看到一個系統在網路分區時寧可拒絕服務也不肯兩邊都寫、看到它對每筆寫入都要求過半確認、看到它在重啟後謹慎地拒絕自己上輩子發出的指令——你會知道，那不是它過度小心，那是它在誠實地面對一件單機程式設計師可以一輩子假裝不存在、而分散式系統一秒鐘都逃不掉的事：**你那台機器，並不孤獨地活在一個 all-or-nothing 的世界裡。**
