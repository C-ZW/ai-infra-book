# 樂觀與悲觀並發控制：CAS、版本與 ETag

兩個人同時打開同一張表單。畫面上是一份產品說明，定價 100 元、庫存 10 件。第一個人把定價改成 120，按下儲存。第二個人沒看到這個變動——他的瀏覽器裡還是 100——他把庫存改成 8，也按下儲存。兩個請求都成功了，伺服器都回 200。然後第一個人重新整理頁面，發現自己改的 120 不見了，又變回 100。

沒有人收到錯誤。沒有任何一筆寫入「失敗」。但有一筆更新被另一筆悄悄蓋掉了，而且蓋掉它的人根本不知道那裡曾經有過別的值。這就是 **lost update（更新遺失）**——並發控制要解的核心問題，也是它存在的全部理由。它不是當機、不是逾時、不是任何會讓你半夜被叫起來的爆炸；它更陰險，因為它**靜悄悄地**把資料寫錯，沒有任何訊號。

要看清這件事為什麼難，得先把那兩個請求拆開，用慢動作重放一次。

## 讀—改—寫之間的那道縫

第二個人的操作，在後端其實是三個分開的步驟：**讀**出現在的庫存（10）、在腦中或程式裡**改**成新值（8）、把新值**寫**回去。第一個人也是三步：讀出定價（100）、改成 120、寫回去。問題不在任何單一步驟，而在三步**不是原子的**——讀和寫之間有一道時間縫，別人可以在這道縫裡插進來。

```
time ->
  client 1:  READ price=100 ............................. WRITE price=120
  client 2:  ........ READ stock=10 ... WRITE stock=8
```

如果兩個人改的是同一個欄位，這道縫的後果最清楚。但即使他們改的是不同欄位（一個改價、一個改庫存），只要後端是「讀整列、改記憶體裡的物件、寫整列回去」這種常見的 ORM 模式，第二個人寫回去的那一整列裡，帶的是**他讀到時的舊定價 100**——於是他無辜地把第一個人的 120 覆蓋掉了。lost update 不需要兩個人改同一格才會發生，只要他們的「寫回」是基於各自過時的快照。

最天真的修法是「寫之前再讀一次比對」：寫之前先 `SELECT` 看看值還是不是我當初讀到的那個。但這只是把那道縫往後挪了一點——你「再讀一次」和「寫回去」之間，仍然有一道新的縫，別人照樣能插進來。問題的本質是：**檢查和寫入必須是同一個不可分割的動作**，而不是兩個分開、中間能被打斷的動作。任何「先檢查、再行動」的並發邏輯，只要檢查和行動之間有縫，就有這個漏洞。這是貫穿整章的第一原理。

從這裡開始，路分成兩條。一條是**在縫出現之前就把它焊死**：動資料前先把它鎖住，別人想碰得排隊等。這是悲觀。另一條是**讓縫存在，但在寫的瞬間原子地檢查它有沒有被別人動過**：沒動才寫，動了就拒絕。這是樂觀。兩條路解的是同一個 lost update，賭的卻是相反的東西。

## 悲觀：先鎖住，誰也別想插隊

悲觀並發控制（pessimistic concurrency control）的假設藏在名字裡：它**悲觀地認為衝突會發生**，所以乾脆在你開始讀之前，就把要動的資料鎖起來。在關聯式資料庫裡這通常長這樣：

```sql
BEGIN;
SELECT price, stock FROM products WHERE id = 42 FOR UPDATE;  -- 上行鎖
-- ... 在這中間，任何別的交易想 FOR UPDATE 同一列，都會卡住等
UPDATE products SET price = 120 WHERE id = 42;
COMMIT;  -- 釋放鎖
```

`FOR UPDATE` 在那一列上掛了一把寫鎖。第二個人的交易如果也想 `SELECT ... FOR UPDATE` 同一列，它會**阻塞**——停在那裡，直到第一個人 commit 或 rollback 把鎖放掉，才輪到它。等它終於拿到鎖、再讀一次，讀到的已經是 120 了。讀—改—寫的那道縫被消滅了，因為在第一個人放手之前，第二個人連讀都讀不進去。

悲觀鎖的保證很硬：**只要你進場拿到了鎖，這段期間這份資料就是你獨佔的，不可能有 lost update。** 它把並發強行串行化成排隊。代價也很直接——排隊就是等待，而等待會放大成三種具體的痛：

第一，**吞吐被鎖的持有時間綁死**。一列同時只有一個交易能持寫鎖，其他人全在排隊。如果持鎖的交易裡夾了一個慢動作——等一個外部 API 回應、等使用者在前端慢慢填表單——那麼這把鎖會被它一直攥著，後面排隊的全部跟著卡。所以有一條鐵律：**持鎖的交易要短，中間絕不能穿插網路呼叫或等使用者輸入。** 把鎖的範圍縮到「讀—改—寫」這幾毫秒，而不是覆蓋整個使用者互動的幾秒甚至幾分鐘。

第二，**死結**。一旦一個交易要鎖多列，而不同交易以不同順序去鎖，就可能繞成一個環：A 持列 1 等列 2，B 持列 2 等列 1，誰也不放，誰也不進。資料庫引擎會偵測到這個環、選一個交易中止掉。死結的成因、等待圖、與固定加鎖順序這類解法，是鎖機制本身的深水區（見〈鎖與死結：行鎖、gap、next-key 與 wait-for graph〉）；這裡只需要記住：悲觀鎖一旦跨多列，就把死結這個新故障模式請進了門。

第三、也是最根本的限制——**鎖出不了交易的邊界，更出不了單機的邊界**。`FOR UPDATE` 的鎖活在一個資料庫交易裡，交易一結束鎖就沒了。但很多並發場景的「讀」和「寫」根本不在同一個交易、甚至不在同一個請求裡。一個使用者在網頁上打開編輯表單（讀），填了五分鐘才按儲存（寫）——你不可能讓一個資料庫交易、攥著一把行鎖，橫跨這五分鐘的使用者發呆時間。HTTP 是無狀態的，請求和請求之間沒有一個可以掛住鎖的地方。悲觀鎖在「讀和寫之間隔著一個無狀態邊界」的場景裡，根本無從施力。

而那，恰好是樂觀控制的主場。

## 樂觀：讓縫存在，但在寫的瞬間驗一次

樂觀並發控制（optimistic concurrency control，OCC）反過來賭：**衝突很罕見，所以不值得為了防它而讓所有人排隊。** 它讓你自由地讀、自由地改，全程不上任何鎖；只在最後寫回去的那一瞬間，原子地問一句話：「我讀到現在，這份資料有沒有被別人動過？」沒動，寫就成功；動了，寫就失敗，由你決定要重來還是放棄。

關鍵在於怎麼讓那句「有沒有被別人動過」的檢查和寫入變成**一個**不可分割的動作——也就是把前面那道縫真正焊死。答案是一個古老的原語：**compare-and-swap（CAS，比較並交換）**。CAS 的語意是「只有當這個位置現在的值還等於我預期的舊值時，才把它換成新值」，而且比較和交換在硬體或引擎層面是原子的，中間插不進任何東西。所有樂觀控制——從 CPU 的 lock-free 資料結構，到資料庫的版本欄，到 HTTP 的 ETag——骨子裡都是 CAS。

在資料庫裡，最常見的 CAS 實作是給表加一個 **version 欄**。每次更新，把 version 加一；寫的時候，把「我讀到的 version」塞進 `WHERE`：

```sql
-- 讀：拿到 price 與當前 version
SELECT price, version FROM products WHERE id = 42;   -- 假設讀到 price=100, version=7

-- 改—寫：條件更新，只有 version 還是 7 才寫得進去
UPDATE products
SET price = 120, version = 8
WHERE id = 42 AND version = 7;
```

這條 `UPDATE` 就是一次 CAS。`WHERE id = 42 AND version = 7` 是「比較」，`SET ... version = 8` 是「交換」，整條 SQL 在引擎裡原子執行。如果在你讀到 7、寫回去之間，別人已經把這列改過、version 推到了 8，那麼你這條 `WHERE version = 7` 會**命中零列**——更新什麼都沒做。資料庫不會報錯，它只是告訴你 **affected rows = 0**。

這個 affected rows = 0 就是樂觀控制全部的安全性所在，也是它最容易出 bug 的地方。引擎不會替你把「改了零列」當成失敗——對 SQL 來說，`UPDATE` 改了零列是完全合法、完全成功的執行。**安全性掛在你的程式有沒有去檢查那個數字上**。一個只寫了 `db.execute(update_sql)` 卻從不看回傳 affected rows 的程式，會在 version 不符、零列被改的時候，若無其事地往下走、回給使用者「儲存成功」——lost update 原封不動地發生了，只是這次連覆蓋都沒覆蓋成,使用者以為存進去的東西根本沒進去。樂觀控制的整個機制，可以被一行「忘了檢查回傳值」的程式徹底架空。

## 一個庫存欄位，手算到底

把這套放進一個帶數字的場景，看它在低衝突下怎麼乾淨地工作、在高衝突下怎麼自爆。

一個賣手作商品的小店,某商品 `stock = 10`,一天大概賣個位數,兩筆訂單剛好同時打到同一件商品的機率很低。用 version OCC:

兩筆訂單 T1、T2 幾乎同時到，兩邊都先讀到 `stock = 10, version = 7`。

T1 先寫:`UPDATE items SET stock = 9, version = 8 WHERE id = ? AND version = 7`。此刻 version 還是 7,命中 1 列,成功,version 變 8。

T2 隨後寫:`UPDATE items SET stock = 9, version = 8 WHERE id = ? AND version = 7`。但 version 已經是 8 了,`WHERE version = 7` 命中 **0 列**。T2 的程式看到 affected rows = 0,知道「我讀到的快照過期了」,於是**重讀**——這次讀到 `stock = 9, version = 8`——再算一次(9 − 1 = 8),重寫:`... SET stock = 8, version = 9 WHERE id = ? AND version = 8`,成功。

最終 `stock = 8`,兩筆訂單都正確地各扣了一件。整個過程**沒有任何一把鎖**,T2 只多付了「撞一次、重試一次」的成本。在衝突罕見時,絕大多數請求一次就過,這個成本攤平後幾乎是零——這正是樂觀控制在讀多寫少、衝突低的場景裡吞吐最高的原因。

現在把同一個欄位、同一段程式,丟進一場秒殺:`stock = 10`,但同一瞬間有 5000 個人搶。每個請求都讀到某個 version、算出新庫存、嘗試 CAS 寫回。但 10 件商品意味著只有 10 次 CAS 能成功,**光是第一輪就有約 4990 次撞上 version 不符、affected rows = 0**。這些失敗的請求會怎麼做?重讀、重算、重試——然後大機率又撞上、又失敗、又重試。每多一輪重試,失敗次數又疊上一層,真正燒掉的 CAS 嘗試遠不止這 4990 次。CPU 不是燒在做生意上,而是燒在這場「讀—改—寫—失敗—重來」的空轉裡。衝突率一高,樂觀控制就退化成一台重試風暴製造機,反覆消耗資源卻幾乎不前進。

同一個欄位、同一段邏輯,衝突率從「一天幾筆」變成「一瞬間五千筆」,正確答案就從樂觀翻轉成悲觀。秒殺這種熱點場景,改用悲觀 `FOR UPDATE` 讓請求老老實實排隊(排隊雖慢,但不空轉)是一條路——但這條路也有自己的上限:每個阻塞在那把行鎖上的交易都攥著一條資料庫連線,數千個交易同時卡在同一列上,會先把連線池(通常只有幾十到幾百條)抽乾,新請求連連線都拿不到、外加鎖等待逾時的雪崩,系統照樣崩,只是死因從「重試燒 CPU」換成「連線池耗盡」。所以真正的秒殺多半走第三條路:乾脆把搶量擋在資料庫之外——用一個 Redis 原子 `DECR` 先把「還有沒有貨」這件事在記憶體裡決勝負,只放贏家進到資料庫。**並發控制沒有「哪個比較快」的絕對答案,只有「衝突率對上策略」的配對。** 樂觀把代價放在「重試」,悲觀把代價放在「等待」;衝突低,重試幾乎免費;衝突高,等待反而便宜。選錯邊,兩者都會把系統拖垮,只是死法不同。

## 為什麼非得是 version,不能只比值

這裡藏著一個非顯然、但一旦踩到就極難查的陷阱:為什麼要特地加一個 version 欄、每次 +1?直接拿欄位的**值**本身來做 CAS——「只有當 stock 還等於我讀到的 10 時才寫」——不行嗎?

大多數時候行,但有一種情況會騙過你,它有個名字叫 **ABA 問題**。設想你讀到 `stock = 10`,正準備做「只有 stock 還是 10 才寫」的條件更新。在你動作之前,另外兩筆操作飛快地發生了:有人賣掉一件(stock 變 9),又有人退貨補回一件(stock 變回 10)。現在你的條件更新跑了——它看到 stock **還是** 10,於是判定「沒人動過」,放行。但事實是動過兩次,狀態已經不是你以為的那個了。值繞了一圈回到原點(A → B → A),而純粹比值的 CAS **看不見這趟旅程**。

version 欄正是用來堵這個的。它**單調遞增、只增不減、絕不重設**,所以哪怕值兜了一圈回到 10,version 也已經從 7 走到了 9——你那條 `WHERE version = 7` 立刻識破:這份資料動過。version 把「值有沒有變」這個會被 ABA 愚弄的問題,換成了「有沒有人在我之後寫過」這個誠實得多的問題。

也因此,version 欄的兩條設計戒律不能破。**第一,只增不減,而且永不重設。** 如果你圖省事用了個小整數、讓它在某個上限後回繞歸零,那麼一個被延遲夠久的請求,理論上可能撞上 version 繞回它當初讀到的那個數字,ABA 又從窗戶爬回來了——高寫入量的熱表用 `BIGINT` 而非 `INT`,正是為了把這個繞回的天文數字推到永遠不會發生。**第二,rollback 時 version 不該被「還原」成更小的值**——它記的是「這列被嘗試寫過幾次」的歷史,不是「成功寫過幾次」的計數,單調性壓倒一切。任何讓 version 變小或重複的捷徑,都是在親手把 ABA 的後門打開。

## ETag:把 CAS 搬到 HTTP 上

回到開場那個「打開表單、五分鐘後才儲存」的場景。讀(GET 表單)和寫(PUT/PATCH 儲存)隔著一個無狀態的 HTTP 邊界、隔著五分鐘的使用者發呆——悲觀鎖在這裡完全使不上力,你不可能攥著一把資料庫行鎖等使用者五分鐘。這正是樂觀控制的主場,而 HTTP 早就把這套機制標準化了,就是 **ETag ＋ conditional request**。

機制和 version 欄一模一樣,只是換了層皮:

```
# 客戶端讀資源,伺服器在回應裡帶上這個版本的指紋
GET /products/42
  -> 200 OK
     ETag: "v7-a1b2c3"        # 這份表示的版本指紋
     { "price": 100, "stock": 10 }

# 五分鐘後,客戶端寫回,帶上它當初讀到的 ETag
PUT /products/42
  If-Match: "v7-a1b2c3"       # 「只有伺服器上現在還是這個版本,才接受我的寫」
  { "price": 120, "stock": 10 }

# 情況 A:期間沒人改過 -> 伺服器接受
  -> 200 OK
     ETag: "v8-d4e5f6"        # 新版本指紋

# 情況 B:期間別人改過,當前 ETag 已不是 "v7-a1b2c3" -> 拒絕
  -> 412 Precondition Failed
```

`ETag` 是伺服器給這份資源當前表示打的一枚版本指紋(可以是內容雜湊,也可以就是後端那個 version 欄的值)。`If-Match` 是客戶端說的條件:「只有當伺服器上這份資源**現在**的 ETag 還等於我帶來的這個,才執行我的寫。」伺服器拿客戶端帶來的 ETag 和當前 ETag 比——相符就執行、回新 ETag;不符就回 **412 Precondition Failed**,一個位元組都不改。這就是 CAS:`If-Match` 是「比較」,寫入是「交換」,伺服器原子地完成這次比較與交換,RFC 9110 把它定義為標準的條件請求語意。412 在這套機制裡不是錯誤,是設計——它是伺服器在說「你手上的版本過期了,先去重讀」,等同於資料庫那個 affected rows = 0。

這層機制有幾個非顯然的邊角,每一個沒對齊都會讓樂觀控制悄悄失效:

**`If-Match` 只認強驗證器(strong validator)。** ETag 分強弱兩種,弱的長這樣:`W/"abc"`,前綴一個 `W/`。弱 ETag 只保證「語意上等價」(兩份表示在使用者看來一樣,即使位元組不完全相同——例如只差一個壓縮方式或一個時間戳註解);強 ETag 保證「位元組層面完全相同」。RFC 9110 規定 `If-Match` 用**強比較**——它只接受強 ETag,而且要求兩邊位元組一致。原因很直白:你要防的是 lost update,賭的是「這份資料一個位元組都沒被動過」,弱比較那種「看起來差不多就算同一個」的寬鬆,在這裡正是漏洞。所以拿弱 ETag 去做並發控制,等於沒做。

**客戶端不回傳 ETag,整套機制就是空的。** 伺服器發了 ETag,客戶端寫回時若不帶 `If-Match`,那就是一個無條件覆蓋——伺服器不知道這個寫是基於哪個版本,只能照單全收,lost update 照常發生。樂觀控制的安全性,一半在伺服器、一半在客戶端的配合上。問題是伺服器無從強迫客戶端守規矩——除非它**主動拒絕所有沒帶 `If-Match` 的寫**。HTTP 為此準備了 **428 Precondition Required**:伺服器回 428,等於說「我這個資源不收無條件的寫,請你帶上 `If-Match` 再來」。這把「客戶端可能忘了帶條件」這個靜默漏洞,變成一個顯式的、寫操作前就被擋下的錯誤。不過 RFC 也誠實地標明:428 是可選的,客戶端不能反過來「依賴伺服器會回 428」來保證自己沒漏掉條件——責任仍在寫入方主動帶上 `If-Match`。

**`If-Match: *` 是另一種語意,別和版本檢查混用。** 星號的意思是「只要這個資源**存在**(任何版本)就執行」,它防的不是 lost update,而是「別在資源已經不存在時還去更新它」這類存在性條件。要做版本檢查,得帶**具體的 ETag 值**,不是星號。

## 同一套模式,在分散式 KV 裡換了名字

一旦你認出「讀一個版本、原子地條件寫回」這個骨架,就會發現它在各種儲存裡反覆出現,只是 API 名字不同。它們全是同一個 CAS,只是把「version 欄」這件事用各自的原語表達:

| 系統 | CAS 的長相 | 「版本」是什麼 | 衝突時你拿到什麼 |
|---|---|---|---|
| SQL（version 欄） | `UPDATE ... WHERE id=? AND version=?` | 自己維護的 version 欄 | affected rows = 0 |
| SQL（系統欄位） | 用 `xmin`(PG) / `rowversion`(SQL Server) 當版本 | 引擎自動標記的列版本（xmin 是最後寫入交易的 txid；rowversion 是單調遞增計數器） | affected rows = 0 |
| HTTP | `If-Match: <etag>` + 寫 | 資源的 ETag | 412 Precondition Failed |
| Redis | `WATCH key` ... `MULTI`/`EXEC` | 被 WATCH 的鍵有沒有被改 | `EXEC` 回 nil（交易整批作廢） |
| DynamoDB | 帶 `ConditionExpression` 的條件寫 | 你在條件裡指定的欄位/版本 | `ConditionalCheckFailedException` |
| etcd | `Txn` 比 `mod_revision` | 鍵的修訂版號 | 交易回「比較為 false」、走 failure 分支 |

這裡的「系統欄位」一行要小心一個前後不一致的陷阱:`rowversion` 是資料庫層級單調遞增、絕不回繞、永不重設的 8-byte 計數器,完全符合前面立的兩條 version 戒律;但 PG 的 `xmin` 是 32-bit 交易 ID,會在約 40 億筆交易後 wraparound 回繞,並被 vacuum freeze 後改成「對所有交易永遠可見」、原始 xmin 不再被讀。換句話說,它恰好牴觸前面那條「version 千萬不能回繞、不能重設」的戒律。對絕大多數負載 xmin 夠用(EF Core/Npgsql 的 `IsRowVersion` 就映到它),但超高交易量、或需要嚴格單調性的熱表,該用專屬的 `BIGINT` version 欄,而不是借 xmin。

值得停下來看的是 Redis 的 `WATCH`/`MULTI`/`EXEC`,因為它把樂觀的精神演得最透。你先 `WATCH` 一個鍵,然後讀它的值、在客戶端算出新值、用 `MULTI` 把要做的寫排成一個交易,最後 `EXEC`。Redis 在 `EXEC` 的瞬間檢查:從你 `WATCH` 到現在,這個鍵有沒有被任何別的客戶端改過?**沒改,整批執行;改了,整批作廢,`EXEC` 直接回 nil**——一個寫都不做。它沒有鎖任何東西(`WATCH` 不阻塞別人寫),只是在提交的那一刻驗一次有沒有被插隊,然後讓你自己決定要不要從頭再來。這就是 OCC 的純粹形態,和那條 `UPDATE ... WHERE version=?` 是同一個靈魂。

DynamoDB 的條件寫和 etcd 的 `Txn` 也一樣:前者讓你在 `PutItem`/`UpdateItem` 上掛一個 `ConditionExpression`(「只有當這個欄位還是某值時才寫」),條件不符就拋 `ConditionalCheckFailedException`;後者讓你在交易裡比較鍵的 `mod_revision`(每次修改自增的全域修訂號,本質就是一個由叢集統一維護的 version),比中才走成功分支。換個儲存、換個錯誤名稱,機制紋絲不動。

但這張表也藏著樂觀控制最硬的邊界:**這些 CAS 全都只保護「單一個鍵/單一列」的原子性**。`WHERE version = ?` 守的是那一列;`If-Match` 守的是那一個資源;Redis 的 `WATCH` 守的是那幾個被 watch 的鍵;DynamoDB 的條件寫守的是那一個 item。一旦你的不變量是**跨多列**的——「這兩個共用帳戶的餘額相加不得為負」這種——單欄 CAS 就守不住了。兩筆交易可以各自原子地、合法地更新**不同的列**(一個扣 A、一個扣 B),每一筆的 version 檢查都通過,合起來卻打破了那個橫跨兩列的不變量。這類「沒有任何單一列被兩人同時寫,所以逐列的版本檢查抓不到」的異常,屬於交易隔離級別要處理的範疇(見〈隔離級別與異常:dirty、phantom 到 write-skew〉),CAS 救不了。樂觀控制是**單點**的守衛,不是跨點一致性的解法。

## 別把樂觀控制錯當成去重

最後釐清一個極常見、也極要命的混淆:樂觀並發控制和**冪等**不是同一件事,雖然它們在重試的場景裡經常並肩出現。

冪等解的是「同一個操作被執行兩次,結果要和執行一次一樣」(定義與冪等鍵的生命週期見〈重複是常態:冪等與去重〉)。樂觀控制解的是「基於過期快照的寫不能覆蓋掉新值」。一個防的是**重複**,一個防的是**過期覆蓋**——是兩個正交的問題。

它們纏在一起,是因為當一個樂觀寫因 version 不符而失敗、你決定重試時,這次重試本身也得是安全的——而保證重試安全的,正是冪等。但這只是說「重試樂觀寫時需要用到冪等」,不代表兩者可以互相替代。具體的反例:一個操作可以完美冪等(同一個冪等鍵送十次只生效一次),卻照樣會 lost update——因為冪等只管「這個請求別重複生效」,完全不管「這個請求帶來的值是不是基於過期快照」。反過來,version 欄能擋住過期覆蓋,卻擋不住「同一筆全新的、帶著新冪等鍵的請求被網路重送兩次」。把樂觀控制當去重、或把冪等當並發控制,都會在你以為被保護的地方留一個洞。

## 為什麼是這個形狀

退一步看,整章其實只是一個問題的兩個答案。問題是:**讀和寫之間那道縫,怎麼焊死。** 悲觀的答案是「進場前就鎖死,別人別想插進來」;樂觀的答案是「讓縫開著,但在寫的瞬間原子地驗一次有沒有被插隊」。一個賭衝突常見、寧可讓人排隊;一個賭衝突罕見、寧可偶爾重試。

它們之所以是兩種、而不是一種,是因為「衝突率」這個變數本身是兩極的,沒有一個策略能在兩極都贏。衝突低的時候,悲觀讓所有人為了那個幾乎不會發生的衝突白白排隊,純屬浪費;衝突高的時候,樂觀讓所有人陷進讀—改—寫的空轉風暴,純屬內耗。所以這個領域長成了兩套並存的機制,而真正的工程判斷不在「哪個更好」,而在「**我這份資料,被併發寫的機率有多高,我的讀和寫之間隔著什麼邊界**」——衝突高就悲觀排隊,衝突低就樂觀重試,讀寫之間隔著無狀態的 HTTP 就只能樂觀、用 ETag。

而無論選哪邊,核心都是同一個古老的原語:**檢查和寫入必須是一個不可分割的動作**。悲觀用鎖讓這件事在獨佔下成立,樂觀用 CAS 讓這件事在一條 `WHERE version = ?`、一個 `If-Match`、一次 `WATCH`/`EXEC` 裡原子地成立。下次你寫下一條 `UPDATE` 卻沒去看它改了幾列,或發了一個 ETag 卻沒人帶 `If-Match` 回來——你就是在那道焊死的縫上,又悄悄敲開了一條,而它不會報錯,只會在某個深夜把一筆更新無聲地吃掉。
