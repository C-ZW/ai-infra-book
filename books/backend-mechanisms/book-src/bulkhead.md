# bulkhead：艙壁隔離

一個服務同時做兩件事：接「下單」，也接「查歷史訂單」。兩條路徑跑在同一個行程裡，共用同一個工作執行緒池——這是再正常不過的設計，沒人會為兩個 endpoint 各起一個服務。某天下午，撐著「查歷史」的那個報表型資料庫慢了下來，每筆查詢從平常的 80 毫秒變成 4 秒。沒有人改過下單的程式碼，下單的資料庫也好好的，但下單開始大量超時，使用者付不了款。

這一幕的詭異之處在於：**壞掉的東西和受害的東西沒有任何直接關係。** 慢的是報表庫，倒下的是結帳。中間沒有共用的資料表、沒有共用的下游服務，唯一的交集是那個誰都沒多想的執行緒池。bulkhead（艙壁隔離）這套機制，整個就是為了讓上面這種「不相干的故障互相牽連」變得不可能。要把它講透，得先看清楚一件事：那個共用的池子，為什麼會把局部的慢，放大成全域的死。

## 共用資源是故障傳播的管道

把那個執行緒池的內部放慢看。假設它有 200 條工作執行緒，這是這個行程同時能「正在處理」的請求上限——第 201 個請求進來，要嘛排隊等，要嘛被拒。平時下單和查歷史各佔一部分，井水不犯河水。

現在報表庫慢到 4 秒。一條工作執行緒接下一個查歷史請求後，會卡在那筆查詢的 socket 上**整整 4 秒**——這段時間它不是在「處理」，是在「等」，但對執行緒池來說沒有區別：這條執行緒被佔住了，不能拿去做別的事。關鍵的問題是，這種「被佔住」的執行緒會累積多快。

這裡可以用 Little's Law 把它算死。一個穩態的系統裡，「同時在處理中的請求數」等於「請求到達率」乘上「每筆處理時間」（L = λ × W）。假設查歷史的入站速率是每秒 60 筆，平常每筆 80 毫秒，那它穩態只佔用 60 × 0.08 = 4.8 條執行緒——零頭，沒人會注意到。但當每筆變成 4 秒，同樣的 60 筆/秒到達率下，它要佔用的執行緒變成 60 × 4 = **240 條**。

240 超過了整個池子的 200。也就是說，光是查歷史這一條路徑，就足以把整個執行緒池吸乾，還不夠。下單請求進來時，池子裡一條空閒執行緒都不剩——它們全卡在報表庫的 socket 上等那 4 秒。下單於是開始排隊、超時、失敗。報表庫的慢，透過「共用執行緒池」這條看不見的管道，淹沒了結帳。

```
                shared pool: 200 threads
   +-------------------------------------------------+
   |  checkout req --> []  []  []  ... (no thread!)  |
   |                                                 |
   |  history req  --> [####  240 wanted  ####] >200 |
   |                   all blocked on slow report DB |
   +-------------------------------------------------+
   one dependency's slowness drains the whole pool
```

這就是 bulkhead 想堵的第一原理：**任何被多種工作共用的有界資源，都是故障的傳播管道。** 執行緒池是最典型的一個，但同樣的劇本會在連線池、記憶體、檔案描述子、甚至一個全域的 rate limiter 計數器上重演。只要 A 和 B 共用一個會被耗盡的東西，A 把它耗盡，B 就跟著死——哪怕 B 自己一切正常。Michael Nygard 在 2007 年的《Release It!》裡把這個觀察整理成一個穩定性模式，名字借自造船：船體用一道道水密隔艙壁（bulkhead）切成獨立的艙，一個艙破洞進水，水被框在那個艙裡，漫不到別的艙，船因此不沉。軟體裡的「進水」就是資源耗盡，而隔艙壁，就是把那個共用的池子切開。

## 切開池子：兩種隔離，兩種代價

最直接的解法呼之欲出：別讓下單和查歷史共用一個池。給查歷史一個獨立的、上限 50 的池，給下單留它自己的 150。現在報表庫再慢，查歷史最多也只能佔滿它那 50 條執行緒——第 51 個查歷史請求直接被拒（這時該接一個降級結果，見〈fallback 與降級〉），而下單那 150 條一條都動不到。局部的慢被框死在它自己的艙裡。

但「切開池子」這件事，實作上有兩條路，它們的隔離強度與代價差很多，這個差別是 bulkhead 最容易被講錯的地方。

第一條路是**獨立執行緒池**（thread-pool bulkhead）。給查歷史真的開一組專屬的工作執行緒。當一個請求要打報表庫時，它不在原本接收請求的執行緒上跑，而是被丟進這組專屬執行緒裡執行。報表庫卡 4 秒，卡的是這組專屬執行緒，原本那條接請求的執行緒早就回去接下一個了。這種隔離最徹底——**連「阻塞」本身都被框住了**，因為阻塞發生在另一組執行緒上。

第二條路是**信號量**（semaphore bulkhead）。不開任何新執行緒，只用一個計數器：規定「同時最多 50 個請求在打報表庫」，每進一個計數加一、每完成一個減一，到 50 就立刻拒絕第 51 個。它輕量到幾乎沒有額外成本——沒有多餘的執行緒、沒有 context switch。但它有個關鍵的、常被忽略的限制：**請求還是在原本那條接收它的執行緒上跑。**

這個限制要命在哪？回到報表庫卡 4 秒的場景。信號量限制了「最多 50 個併發查歷史」，聽起來像是把傷害上限框在 50 了。但那 50 個請求，是卡在**接收請求用的那個主執行緒池**裡的 50 條執行緒上——信號量只擋住了第 51 個，前 50 個照樣在主池裡一人佔住一條執行緒，空等 4 秒。如果主池只有 200 條，這 50 條就是實打實地從下單那邊借走的。信號量管住了「併發數」，但管不住「這些併發各自佔住了主池的一條執行緒、而且整整阻塞 4 秒」。

所以這兩者的分水嶺，是一個很乾淨的判準：

| | 執行緒池隔離 | 信號量隔離 |
|---|---|---|
| 呼叫跑在哪 | 獨立的專屬執行緒 | 原本接收請求的執行緒 |
| 能否隔離「阻塞」 | 能（阻塞被框在專屬池） | **不能**（阻塞發生在主池） |
| 成本 | 多一組執行緒＋context switch | 幾乎零（只是一個計數器） |
| 適合 | 會長時間阻塞的同步呼叫 | 非阻塞／極短的呼叫 |

判準濃縮成一句：**如果被保護的呼叫可能長時間阻塞，而你真正想隔離的就是這個阻塞，那只有獨立執行緒池做得到；信號量在這種場景下幾乎等於沒裝。** 反過來，如果呼叫本身不阻塞（例如它走非同步 I/O、執行緒一發出請求就回來了），或快到根本來不及堆積，那信號量足夠，而且它省下了一整組執行緒的開銷。

這個分類不是憑空來的。Netflix 的 Hystrix 把這兩種策略並列成 `THREAD` 與 `SEMAPHORE` 兩種 isolation strategy，後來的 Resilience4j 也分成（信號量式的）`SemaphoreBulkhead` 與執行緒池式的 `FixedThreadPoolBulkhead`（Spring 整合層慣以 `ThreadPoolBulkhead` 稱之）——順帶一提，Hystrix 自 2018 年起已進入維護模式、不再主動開發，社群與 Netflix 自己都把 Resilience4j 當作現役後繼（2026-06）。但工具會換，這個「執行緒池 vs 信號量、能不能隔離阻塞」的分野不會變，它是從「阻塞發生在哪條執行緒上」這個物理事實長出來的。

## 艙不只切執行緒

執行緒池是最容易講清楚的艙，但「共用資源是傳播管道」這個原理，會在好幾個層次上重演，每一層都對應一種艙。把它們擺在一起看，bulkhead 的全貌才完整。

**連線池**是僅次於執行緒池的常見傳播管道。一個服務對同一個資料庫開一個有 30 條連線的池，租戶 A 跑了一個沒走索引的全表掃描，每個查詢佔住一條連線好幾秒，30 條連線很快被 A 佔光——租戶 B、C 的請求拿不到連線，明明它們的查詢一毫秒就能跑完。解法同樣是切艙：給不同租戶、或不同重要性的查詢開獨立的連線池。但這裡藏著一個 bulkhead 最常見的失誤——**只切了上層、忘了下層共用**。你把應用層的執行緒池切成三個艙，自以為隔離了，結果這三個艙的請求最後都連到**同一個資料庫連線池**，或同一個下游服務實例。底層那個共用點沒切，故障照樣從那裡穿透。隔離要一路切到**真正的瓶頸資源**為止，切到一半的艙壁等於沒有艙壁——水會從你沒堵的那道縫漫過去。

再往外一層是**行程／實例隔離**。把高風險或高優先的工作負載，拆成獨立的 deployment、跑在獨立的節點上，CPU 和記憶體層級完全不共享。這是最強的艙——一個艙的工作把整台機器的 CPU 燒光、把記憶體吃到 OOM，也波及不到另一個艙，因為它們根本不在同一台機器上。代價也最直白：機器數量、運維複雜度、以及最低的資源利用率（每台機器都得各自留 buffer）。

而 bulkhead 在實務上最有價值的用法，往往不是按「依賴」切，而是按**工作負載的性質**切：線上請求一個艙、背景批次一個艙；付費租戶一個艙、免費租戶一個艙；關鍵路徑一個艙、次要功能一個艙。這擋的是一個有專門名字的問題——**noisy neighbor（吵鬧的鄰居）**：一個免費租戶寫了個瘋狂輪詢的腳本，不該有能力餓死付費租戶的請求。把它們分到不同的艙，就是在資源層面畫一條「你的混亂止於此」的線。

## 隔離不是免費的：你買到什麼、付出什麼

到這裡得把那個被反覆暗示的取捨講白。切艙的代價，是**資源利用率下降**，而且是必然下降，不是調得好就能避免。

道理很簡單：共用一個池子時，下單和查歷史可以互相借用對方暫時用不到的執行緒，整個池子能跑到很滿。一旦切成 150 + 50 兩個艙，這種互相借用就被禁止了——某個瞬間下單只用了 100 條、剩 50 條閒著，而查歷史正好爆量、50 條不夠用，這時那 50 條閒置的下單執行緒**不能**借給查歷史，因為艙壁就是用來禁止這種跨界的。你刻意製造了一部分「明明閒著卻不能用」的容量。

這正是 bulkhead 的取捨本質，而且它是一筆划算的交易：**你用一部分恆定的閒置容量，買到「壞一塊不壞全部」的保證。** 平時你為這個保證付一點點利用率的租金；故障來臨時，這份保險兌現——下單在報表庫崩潰時毫髮無傷。願不願意付這筆租金，取決於「壞全部」對你有多貴。對結帳這種一停就直接損失營收的路徑，這筆租金便宜到不值得猶豫。

也正因為有租金，bulkhead 並非到處都該用。一個**單一同質**的工作負載——只做一件事、只有一個下游、流量小到資源從不接近上限——切艙只會白白增加閒置和調參的複雜度，買不到任何東西，因為根本沒有「另一塊」需要被保護。bulkhead 該出手的場景非常明確：**一個行程裡混跑了重要性或延遲特性差很多的多種工作，而它們共用著會被耗盡的資源。** 線上 vs 批次、付費 vs 免費、關鍵 vs 次要——當這幾條線存在時，艙壁就值得畫。

## 艙的邊界會怎麼騙你

bulkhead 的失敗模式，幾乎都來自「以為自己隔離了，其實沒有」。把幾個最隱蔽的講透。

**用信號量去擋一個會阻塞的呼叫**——前面算過，但值得當成獨立的陷阱記住，因為它太容易發生。有人讀到「bulkhead 能防止故障蔓延」，順手用最輕量的信號量實作，配了個上限 50，自我感覺良好。但被保護的是一個同步的、會卡 4 秒的資料庫呼叫。信號量只擋住第 51 個，前 50 個照樣在主執行緒池裡一人佔一條、卡滿 4 秒——主池被借走的那 50 條，足以拖垮其他路徑。**這種配置不是「隔離弱一點」，而是幾乎沒有隔離**：它框住了一個你並不在乎的維度（併發計數），放過了你真正怕的那個維度（阻塞佔住主池執行緒）。

**艙切太細**，是另一個方向的失誤。看到 bulkhead 的好處，有人給每一個下游依賴都配一個專屬執行緒池——十個依賴十個池。結果池數膨脹，每個池都得留 buffer 應付突發，這些 buffer 加起來是一大筆恆定閒置；更糟的是調參維度爆炸，十個池的大小要分別調、分別監控。隔離有明顯的邊際效益遞減：第一刀（把關鍵和非關鍵切開）價值最高，切到第十刀通常只剩成本。正確的切法是按**故障的爆炸半徑**分艙，不是按依賴的數量分艙——問「哪些東西一起壞我可以接受、哪些絕對不能互相牽連」，沿著那條線切，而不是機械地一個依賴一道牆。

**把 bulkhead 當成 circuit breaker 的替代品**，則是一種概念上的混淆。這兩者常被放在一起講（Nygard 的同一批穩定性模式裡都有），但它們管的是不同的事，互補而非互替。bulkhead 管的是「故障**能佔用多少資源**」——它把傷害框在一個艙裡，但艙裡的請求還是會一個個地慢慢超時、慢慢失敗，只是失敗被關在艙內。circuit breaker 管的是「偵測到故障後要不要**還繼續嘗試**」——它跳開之後，請求連試都不試、直接快速失敗，不再去敲那個已知會卡的門（三態與半開試探的機制見〈circuit breaker：三態與半開試探〉）。一個是「壞了之後傷害的容器邊界」，一個是「壞了之後乾脆別碰的開關」。實務上它們幾乎總是疊在一起用：bulkhead 在外圈限制資源佔用，circuit breaker 在內圈在故障時直接斷掉，兩者再加上對每個呼叫的 timeout（任何跨行程邊界的呼叫都該有等待上限，見〈timeout、deadline 與 budget〉），構成一組互相補位的防線。

**艙滿了卻沒接降級**，是最後一個容易留下的尾巴。艙壁正常運作、第 51 個查歷史請求被乾脆地拒絕了——這在資源保護上完全正確，下單因此活了下來。但對那個被拒的使用者來說，他看到的是一個硬邦邦的錯誤。隔離成功地保住了系統，卻沒人替被擋在艙外的請求想一個次等但能用的答案。被艙壁拒絕，應該是觸發降級的時機之一：給一個快取的舊結果、一個靜態的預設、一個「稍後再試」的明確訊息（怎麼端出次等正確的結果，見〈fallback 與降級〉）。隔離決定「誰被擋下」，降級決定「被擋下的人拿到什麼」，兩件事要接起來，使用者體驗才完整。

## 為什麼是這個形狀

退一步看，bulkhead 的整個樣貌都從一個樸素的事實長出來：**只要兩種工作共用一個會被耗盡的資源，其中一種就有能力獨自把它耗盡，連帶拖死另一種——哪怕那另一種自己毫無問題。**

於是這套機制的每一個選擇都有了理由。它要切開共用的池，因為共用就是傳播的管道。它分執行緒池與信號量兩種切法，因為「阻塞發生在哪條執行緒上」這個物理事實，決定了你能不能真的把阻塞框住。它必然犧牲一點利用率，因為禁止跨艙借用是隔離的定義，不是可以調掉的副作用。它按爆炸半徑切而非按依賴數切，因為艙壁的價值在於框住「不能互相牽連的東西」，而不在於牆的數量。

bulkhead 不保證任何一塊不會壞——報表庫該慢還是會慢，它那一艙該滿還是會滿。它只保證一件事：**壞的那一塊壞下去的時候，不會順手把好的那一塊一起拖下水。** 在一個任何單一依賴隨時都可能變慢的世界裡，這個「把傷害關進一個艙」的承諾，常常就是「一個下午結帳全掛」和「一條次要查詢短暫降級、其他一切照常」之間的全部差別。船不會因為不破洞而安全，船是因為破了洞水也漫不過那道艙壁，才不沉。
