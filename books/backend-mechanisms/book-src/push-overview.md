# server→client 推送的全景：推還是拉

一個交易看板上線當天，產品經理盯著螢幕問了一個聽起來無辜的問題：「為什麼價格跳動要等三秒才反映？」工程師打開程式碼，看到答案躺在那裡——前端每三秒發一次 `GET /prices`，把整張表重抓一遍。延遲不是哪裡卡住了，是設計本身：client 每三秒「問一次」，所以最壞情況下，一筆在你剛問完那一刻發生的價格變動，要等下一輪、整整三秒後才被看見。

把輪詢間隔縮短到一秒，延遲是好看了，但伺服器的請求量瞬間漲了三倍——而且其中絕大多數請求的回應是「沒有新東西」。看板上一千個使用者、每秒各問一次，就是每秒一千次請求打在後端，只為了在偶爾有變動時不漏掉它。你花了三倍的力氣，換來的大部分是空轉。

這就是「推送」這整個主題的起點。它要解的問題其實只有一句話：**有些事 server 知道、client 不知道，而且 client 不知道「該什麼時候問」。** 新訊息到了、訂單狀態變了、價格跳動了——這些事件發生在 server 端，但需要它的人在 client 端。怎麼讓 server 端發生的事，在「可接受的延遲內」抵達 client？這一章不深入任何一種具體傳輸的內部機制（那些各有專章），而是把整張選項地圖攤開，講清楚一件事：**所有的方案，都在同一條軸上選點，而那條軸是物理逼出來的。**

## 為什麼 HTTP 天生只會拉

要看清這條軸，得先回到 HTTP 最原始的形狀。HTTP 是一個請求-回應協定：client 開一條連線、送一個 request，server 回一個 response，然後——按照這個模型的本意——這次交換就結束了。連線是 **client 開的**，回應是 client **問出來的**。server 在沒有被問的時候，手上沒有任何一條「能寫回去」的管道。它就算現在就知道價格變了，也沒有地方可以主動把這件事推給 client。

這不是 HTTP 的缺陷，而是它的設計前提。它假設的世界是「瀏覽器去抓文件」：使用者點一個連結，瀏覽器去要那個頁面，要到了就顯示。在這個世界裡，「誰決定傳輸時機」永遠是 client——我想看了，我才去要。

推送要的恰恰是把這件事**倒過來**：讓 server 決定時機。事件什麼時候發生，server 最清楚，所以該由 server 決定什麼時候送。而 HTTP 的原始模型給不了這個。於是所有推送方案，歸根結柢只有兩條出路，沒有第三條：

- 要嘛**維持一條長連線**，讓 server 在這條一直開著的管道上，隨時能往 client 寫；
- 要嘛**用反覆地拉，去模擬推**——client 一遍又一遍地問，把延遲壓到「下一次問」的間隔之內。

理解了這個分岔，下面整張選項地圖就不再是一堆要背的名詞，而是同一個問題的連續變奏。它們全部落在一條軸上：**一端是延遲低、另一端是連線成本低，你不可能同時把兩端都拿滿。**

## 從「最像拉」到「最像推」：一條連續光譜

把這條軸畫出來，從最像拉的一端走到最像推的一端，會經過幾個有名字的點。它們不是彼此孤立的技術，而是同一條光譜上的刻度。

**短輪詢（short polling）** 在最靠近「拉」的那一端，就是看板那個天真做法：client 每隔固定間隔發一次普通 HTTP 請求問「有沒有新東西」。它的好處是極其簡單——沒有長連線、server 完全無狀態、任何一台後端都能處理任何一個請求，要水平擴展就無腦加機器。它的代價也一目了然：延遲的上界就是你的輪詢間隔，而你為了壓低這個上界，得付出大量「沒有新東西」的空請求。延遲和浪費，在短輪詢裡是直接互換的——你想要延遲減半，請求量就翻倍。

**長輪詢（long polling）** 往「推」的方向挪了一步，做了一個聰明的把戲：client 一樣發一個普通 HTTP 請求，但 server **收到後不立刻回**，而是把這個請求「掛」在那裡——hold 住——直到真的有新事件發生（或者到了一個逾時上限）才回應。client 一收到回應，立刻再發下一個請求接上。它用「掛起一個請求」這件事，模擬了「server 在事件發生時主動送一次」。

長輪詢的延遲比短輪詢好得多：事件一發生，那個被掛起的請求立刻就回，client 幾乎是即時收到，不必再等一個完整的輪詢間隔。而且它走的是標準 HTTP 請求，能穿過幾乎所有的 proxy 和防火牆——這是它在很多受限環境裡仍然是退路（fallback）的原因。但它有一個常被誤會的地方：**長輪詢不是「沒有輪詢」。** 它仍然是一連串的 HTTP 請求，只是每個請求被 hold 住而已。在高頻事件下，這個「hold 一下就回、回了再發」的循環會退化成連環的短輪詢——事件密集到每次掛起幾乎立刻就被觸發，連線建立的成本一個都沒省下來。而且 server 端要同時 hold 住成千上萬個掛起的請求，每一個都佔著一份資源。

再往「推」走，就跨進了長連線的世界。**Server-Sent Events（SSE）** 是其中最克制的一種：一條長連線、`text/event-stream` 的內容型別，server 在這條連線上**單向**地、持續地寫一筆又一筆事件給 client。瀏覽器原生的 `EventSource` 會自動解析這些事件，斷線時還會自動重連。它只能 server→client 單向推（client 要送東西回去，照樣走普通 HTTP 請求），但對「我只需要把事情推給你」的場景，這個單向恰恰是它的省事之處（內部格式與重連續傳的細節見〈SSE、long-poll 與 WebTransport〉）。

**WebSocket** 站在「推」這一端的中心：握手升級之後，這條連線變成全雙工，兩端都能隨時送 frame，延遲最低、每筆訊息的額外開銷也最小。它是雙向高頻互動——聊天、協作、遊戲、即時下單——的主力（frame 結構、ping-pong 心跳與水平擴展見〈WebSocket〉）。

光譜最遠端是 **WebTransport**，建在 HTTP/3（QUIC）之上的新傳輸，能在一條 session 裡開多條獨立的 stream、外加不可靠的 datagram，彼此不互相阻塞。它解的是更深一層的問題——連 TCP 層的隊頭阻塞都繞過去（這個細節下一節會碰到，完整對照見〈SSE、long-poll 與 WebTransport〉）。

還有一條岔出主光譜的路：**第三方推送通道**。行動裝置的 APNs（Apple）、FCM（Google），以及瀏覽器的 Web Push——它們把「維持一條連線」這件苦差事整個外包給平台。你的 server 不再自己養連線，而是把訊息丟給推送服務，由作業系統層級維護的那條常駐連線負責送達。這在「App 已經被使用者切到背景、甚至被系統殺掉」時是唯一能送到的辦法——你自己的連線早就斷了，但作業系統的推送通道還活著。代價是你交出了控制權：送不送得到、什麼時候送到，由平台和它的標準說了算（IETF 的 Web Push 走 RFC 8030，瀏覽器端的 Push API 在 2026-06 是 W3C Working Draft，且把宣告式 Web Push 併入了核心規格）。

## 那條軸上的數字：股價看板手算

抽象的「延遲 vs 連線成本」放到具體數字上才有重量。回到開場那個看板，把它算到底。

設定：一千個 client，每檔股票平均每兩秒更新一次價格。

走**短輪詢**、要把延遲壓到一秒，意味著每個 client 每秒問一次：一千個 client × 每秒一次 = **每秒一千次請求**。而股票兩秒才變一次，所以任何一秒裡，大約有一半的請求問下去得到的是「跟你上次問的一樣，沒有新東西」。換句話說，這一千 req/s 裡，**約莫一半是純粹的空轉**——後端認認真真地處理請求、查資料、組回應，只為了回一句「沒變」。你想把延遲從一秒再壓到半秒？請求量立刻變成兩千 req/s，空轉的比例還更高，因為更新頻率沒變、你只是問得更勤。

換成**事件驅動**的長連線（SSE 或 WebSocket）：一千條常駐連線各自掛著，server 只在價格**真的變動**時才往對應的連線上推一筆。出口的推送量等於實際的變動量——每檔每兩秒變一次、一千個訂閱，大約是每秒五百筆推送。注意這個數字的性質完全不同：短輪詢的一千 req/s 是「不管有沒有事都要付」的固定成本，而長連線的五百筆/秒是「有多少事就推多少」的變動成本。事件稀疏時它幾乎免費，事件密集時它也只付它該付的。而且延遲從「最壞一秒」降到了亞秒級——變動發生的那一刻就推出去，不必等下一輪。

那長連線的成本跑到哪去了？它沒有消失，只是換了形態。短輪詢付的是 **CPU 與請求處理**（每個空請求都要走完整條後端路徑），長連線付的是**常駐連線本身的記憶體與檔案描述子**：一千條連線就是一千份連線物件、一千份緩衝區，一直佔著，哪怕此刻一個事件都沒有。這就是那條軸的真面目——你不是在「好」和「壞」之間選，而是在「為延遲付固定的請求成本」和「為連線付固定的駐留成本」之間選。哪個划算，取決於你的事件有多頻繁、延遲要求有多嚴。事件罕見、延遲容忍秒級，短輪詢的空轉成本低到可以忽略，這時硬上 WebSocket 養一堆閒置連線反而是過度工程；事件頻繁、延遲要求嚴，長連線的駐留成本被密集的事件攤薄，就完勝輪詢。

## 隊頭阻塞：為什麼 SSE 要跑在 HTTP/2 上

長連線省了空轉，但它把一個新問題請進了門，而且這個問題藏得很深：**瀏覽器對「同一個網域能同時開幾條連線」有上限。**

在 HTTP/1.1 下，主流瀏覽器對同一個網域的並行連線數上限**通常是 6**（這是 WHATWG 的 SSE 規格都明文提醒過的麻煩）。一條 SSE 連線會一直開著不還——這正是它的本意。於是一個算術問題浮現：使用者開了 6 個分頁、每個分頁各開一條到同網域的 SSE 連線，這個網域的 6 條連線配額就被佔滿了。第 7 個分頁裡的**任何** HTTP 請求——載入一張圖片、呼叫一個 API、抓一份 CSS——全都會被卡住，在佇列裡乾等，因為沒有連線可用了。使用者的體驗是「網站莫名其妙卡死」，而原因藏在一個跟卡死的那個分頁八竿子打不著的地方：另外六個分頁各自佔著一條一直不還的長連線。

修法是把 SSE 跑在 **HTTP/2** 上。HTTP/2 在**一條** TCP 連線裡做多工（multiplexing）：所有的請求和 SSE 串流都變成這條連線裡的一條條獨立 **stream**，共用同一條底層連線。HTTP/2 規格建議伺服器對並行 stream 數的上限不低於 **100**（RFC 9113 §6.5.2 的 SHOULD），主流實作普遍以 100 上下為預設。6 個分頁的 SSE 此時只佔 6 條 stream，離 100 還遠得很，那個「第 7 個分頁卡死」的問題就消失了。這就是「SSE 強烈建議部署在 HTTP/2 之上」這句話背後的具體理由——不是性能調優的錦上添花，而是避開一個會讓使用者多開幾個分頁就翻車的硬限制。

但這裡有一個更深的轉折，值得講透，因為它解釋了 WebTransport 為什麼存在。HTTP/2 把多工搬進了一條 TCP 連線，**解掉了 HTTP 層的隊頭阻塞**（head-of-line blocking）——不同的 stream 在 HTTP 的眼裡互不相干、誰也不擋誰。可是它沒辦法解掉 **TCP 層**的隊頭阻塞。TCP 是一條保證「有序、不丟」的位元組流：如果這條連線上的某一個封包在網路上掉了，TCP 的接收端就**必須**等它重傳補上，才能把後面已經到達的位元組交給應用層——哪怕後面那些位元組屬於另一條完全無關的 stream。一個封包掉了，連線上所有的 stream 一起卡住等它。HTTP/2 把多條邏輯通道塞進一條 TCP 連線，反而讓這條連線的任何一次丟包，變成了**所有** stream 的共同災難。

這正是 HTTP/3（QUIC）要解的。QUIC 不跑在 TCP 上，它在 UDP 之上自己實作了多工，並且讓**每一條 stream 各自獨立地排序與重傳**——一條 stream 掉了封包，只有那條 stream 等，其他 stream 照走不誤。WebTransport 建在這個基礎上，於是它能在一條 session 裡開多條真正互不阻塞的 stream，外加可以容忍丟包、無序、追求極低延遲的 datagram。對一個延遲敏感、又能容忍偶爾丟一兩個封包的串流場景（即時遊戲狀態、即時媒體的訊號通道），這個「不被別人的丟包拖累」是它相對 WebSocket 的關鍵勝場。2026-06 它已經是 Baseline（Safari 26.4 在 2026-03 跟上後，主流瀏覽器到齊），但 W3C 文件仍是 Working Draft、IETF 的 over-HTTP/3 對應草案仍在 Working Group Last Call（draft-15，2026-06 datatracker 仍標 In WG Last Call），意思是「能用了，但標準仍可能有破壞性變動」，生產上要先想好沒有 HTTP/3 時退回什麼。

## 推送不等於可靠投遞

關於這整套機制，有一個誤解最危險，也最常見：**以為「推送通道」等於「可靠投遞」。** 它不是。所有這些連線——SSE、WebSocket、長輪詢——給的保證都只有一句：**「你連著的時候，我送得到。」** 至於你沒連著的那段時間發生的事件，它一個字都不保證。

把這件事的後果想清楚。一個使用者的手機進了電梯，網路斷了三十秒。這三十秒裡，server 推給他的每一筆訊息，都打在一條已經斷掉的連線上——對 server 而言，它「送出去了」；對使用者而言，這些訊息**從未存在**。連線恢復、`EventSource` 自動重連上來，使用者看到的是「此刻之後」的新訊息，中間那三十秒是一個黑洞。底層的 TCP 在連線存活時保證不丟不亂序，但連線一斷，這個保證跟著連線一起死了；TCP 從來不負責「跨越一次斷線」。

所以「不漏訊息」這件事，**必須在連線這一層之上自己補**。常見的補法有兩種思路。一是**斷點續傳**：SSE 的設計裡，每筆事件可以帶一個 `id`，client 重連時會把上次收到的 id 放進 `Last-Event-ID` 請求標頭送回來，server 據此從斷點之後重放——但這有一個沉默的前提：**server 真的保存了事件歷史、而且能依 id 回放**。如果 server 沒實作這段，`Last-Event-ID` 送回去也沒人理會，重連照樣從「此刻」開始，那三十秒照漏。斷點續傳是一個你必須兩端都實作出來的能力，不是連上去就自動有的。二是**重連後補拉一次快照**：client 一重連，先主動拉一次「現在的完整狀態」，把斷線期間錯過的變化一次補齊，再開始接收增量推送——這把「推」和「拉」結合起來，用一次拉兜住推漏掉的洞。

順帶說一個重連自己帶來的坑。SSE 斷線後會自動重連，預設的重連間隔在規格裡只說是「實作自訂、大約幾秒」（瀏覽器普遍用三秒左右，可以被 server 用 `retry:` 欄位覆寫）。而長輪詢的「重連」更微妙：因為它本質是一連串請求，在某些情況下，同一個 client 可能有**多個請求同時在飛**（前一個還沒處理完、client 已經發了下一個），如果 server 端的事件交付沒有用 id 去重，就可能把同一筆事件送兩次。「連線會斷、會重連」不是一個偶發的異常，而是長連線世界裡的**常態**——任何不為重連設計的推送系統，都會在某次斷線時悄悄丟訊息或重複訊息。要做到不漏不重，得在這層之上加序號與去重（去重與冪等的完整機制見〈重複是常態：冪等與去重〉）。

還有一個會悄悄殺死連線的東西：**中間設備的逾時**。client 和 server 之間往往隔著 proxy、負載均衡器、各種網路中介，這些設備常常對「閒置太久」的連線設逾時——一條 SSE 或 WebSocket 連線如果太久沒有任何位元組流動，可能被某個中間設備在六十秒後**靜默地**砍掉，而兩端都還以為自己連著。server 繼續往一條已死的連線寫，client 繼續等永遠不會來的訊息。防這個的辦法是**應用層心跳**：SSE 可以週期性送一行註解（`:` 開頭的空事件）讓連線上始終有位元組流動，WebSocket 則送 ping frame。心跳同時扮演兩個角色——保活（讓中間設備別把連線當死的砍掉）和死亡偵測（對端如果不回應，你能及時知道這條連線其實已經斷了，而不是傻等）。

## 怎麼選：把判斷收束成幾條線

走完整條光譜，選擇其實收束成幾個清楚的判準，而每一條都是前面那條「延遲 vs 連線成本」軸的直接推論。

**更新罕見、延遲容忍秒級以上**——短輪詢就夠了。空轉成本低到可以忽略，硬上長連線去養一堆閒置連線是過度工程。一個每三十秒查一次狀態的後台儀表，犯不著為它建一套 WebSocket 基建。

**只需要 server 單向推**——SSE 通常勝過 WebSocket。它走純 HTTP、瀏覽器原生自動重連、不必處理升級握手、不必自己實作雙向協定的那一半。很多被直覺歸類為「需要 WebSocket」的場景，其實 client→server 那個方向用普通 HTTP 請求就夠了（送出去不需要即時回應），真正需要即時的只有 server→client——這正是 SSE 的甜蜜點。把這種場景硬塞給 WebSocket，是白白多做了雙向協定的功。

**需要 client→server 也即時**——這才輪到 WebSocket。雙向、高頻、低延遲，frame 開銷小，這是它的本場。

**能容忍丟包、但要極低延遲，而且基礎設施跑得起 HTTP/3**——考慮 WebTransport，用它的多 stream 不互阻與 datagram。但記得它的標準成熟度還是現階段的代價，要備好降級路徑。

**行動端的背景推播**——用平台推送（APNs/FCM/Web Push）。這不是「也可以」，而是「只能」：App 被切到背景或被系統殺掉之後，你自己維持的任何連線都已經死了，唯一還能把訊息送到的，是作業系統層級那條由平台維護的常駐通道。

## 為什麼是這個形狀

退一步看，這整張地圖的形狀，是被 HTTP 一個原始的前提逼出來的：**連線是 client 開的，所以時機天生由 client 決定。** 推送要的是把時機交還給 server，而你只有兩種辦法搶回它——維持一條長連線讓 server 隨時能寫，或者讓 client 拉得夠勤去逼近「即時」。

所有的方案都活在這兩條出路撐開的那條軸上。短輪詢站在「拉」的一端，用空轉換無狀態的簡單；長輪詢往中間挪，用掛起的請求模擬一次推；SSE、WebSocket 站在「推」的一端，用常駐連線的記憶體換低延遲；WebTransport 再往前一步，連 TCP 的隊頭阻塞都要繞過去。它們之間沒有一個是「最好的」——只有「在你這個事件頻率、這個延遲要求、這個連線規模下最划算的」。

而無論你選哪一個，有一件事不會因為連上去就免費送你：**跨越斷線的可靠性。** 推送通道只承諾「連著時送得到」，那道斷線造成的黑洞，永遠得靠連線之上的續傳或補拉自己去填。看清楚這條軸、和它收不進去的那個洞，這張看似雜亂的選項地圖，就變成了一個只有兩三步的決策。
