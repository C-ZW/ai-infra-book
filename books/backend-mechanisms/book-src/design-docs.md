# 把決定寫下來：RFC、設計文件與 ADR

半年後的一個下午，有人翻到你寫的某段程式碼，問了一個簡單的問題：「這裡為什麼用樂觀鎖而不是直接上行鎖？」沒有人記得。寫它的人換了組，當時在白板上爭論過的兩個方案、那個讓大家最後選了這條路的關鍵顧慮，全都蒸發了。剩下的只有程式碼本身——它精確地記錄了**做了什麼**，卻對**為什麼**保持沉默。於是這個問題只有兩種下場：要嘛沒人敢動那段碼（因為不知道改了會踩到當初避開的什麼坑），要嘛有人自信地把它改成「更好」的樣子，然後在三個月後重新發現了當年放棄某方案的那個理由——通常是以一次生產事故的形式。

這就是這一章要解的問題。它表面上是「寫文件」這件聽起來無聊的事，底下卻是一個相當尖銳的工程問題：**一個有實質後果的技術決定，如果只活在某個人的腦子裡、或某次走廊對話、或某條已經沉到水底的 Slack 訊息裡，它就沒有可審查、可追溯、可挑戰的載體。** 決定本身會隨人員流動而衰減為零，但它造成的後果——那段樂觀鎖的碼——會留下來，變成一個沒有出處的既成事實。RFC、設計文件、ADR 這一整套慣例，都是在對抗這個衰減：把決定從易失的人腦記憶，搬到持久的、可被未來的人讀到的文字上。

但如果只把它理解成「為了後人留檔」，就漏掉了一半，而且是更有價值的那一半。

## 第一個價值，在你動筆的當下就兌現了

把一個技術想法寫成文件，會發生一件不寫就不會發生的事：**模糊的直覺被逼成精確的文字。**

腦子裡的方案總是比它實際成立的程度看起來更完整。你「覺得」用事件驅動解耦這兩個服務是對的，這個感覺在腦中是一團溫暖的確定性。但當你坐下來，要寫出「事件的 schema 長什麼樣、誰是生產者、消費失敗時重試還是進死信、消費端怎麼保證冪等、舊事件格式怎麼演進」——你會在寫到第三點時卡住，因為你發現自己根本沒想過消費失敗的情形。那團溫暖的確定性，是被「沒寫出來」這件事保護著的。寫，就是把保護層撕掉。

這是 RFC 與設計文件最被低估的價值：**很多文件的好處，在「寫」的時候就兌現了，根本不在「被批准」。** 一份你寫到一半就自己發現方案有洞、於是改了方向的 RFC，即使從來沒有第二個人讀過，它也已經完成了最重要的工作。寫作是一種強迫思考——它不允許你在「我大概知道怎麼做」的舒適區停留，它逼你把每個「大概」展開成具體的句子，而展開的過程就是發現缺口的過程。

這個觀察有一個直接的、容易被忽視的推論：**評審一份設計文件時，真正該審的不是文字是否流暢，而是作者是否真的想清楚了。** 一份論證滑順、結構完整、讀起來毫無滯礙的文件，可能恰恰是因為作者迴避了所有困難的地方——他只寫了想清楚的部分，把沒想清楚的悄悄繞過去了。一個好的 reviewer 會刻意去戳文件**沒有明說**的東西：失敗時會怎樣？這個 schema 半年後要加欄位怎麼辦？你說的「效能更好」，好多少、在什麼負載下量過？這些問題的答案不在文件裡，正是因為作者還沒想到那裡。把審查的重心從「這句話通不通順」上移到「作者的理解到哪裡為止」，才接得住寫作這個機制真正要保護的東西。

## 一道從零儀式到重流程的光譜

知道了「要把決定寫下來」之後，下一個問題立刻出現：寫多重？每個決定都寫一份正式文件、開評審會、要人簽核？還是隨手記一句就好？這裡沒有單一答案，因為決定本身的份量天差地別——把「變數要不要叫 `userId` 還是 `uid`」和「整間公司的主資料庫從 PostgreSQL 換成什麼」用同一套流程處理，無論往哪邊靠都是災難。

所以實務上存在的是一道**輕重光譜**，每一檔都在「儀式成本」和「把關強度」之間取一個不同的平衡：

最輕的一端是**走廊對話**——零儀式、零留存，靠在場的人記得。它快到幾乎沒有成本，代價是三個月後沒有任何東西能重建「為什麼」，而且不在場的人完全無法參與。再往上一點是**把設計寫在 PR 描述或 issue 討論裡**——決策貼著程式碼、自然留存，但整體設計會散落在十幾個 PR 裡，沒有人能看見全貌；一個大決定塞進某個 PR 的描述欄，等於是寫了等於沒寫，沒人會回頭去那裡讀它。

光譜的中段，是大多數「值得想清楚但不必興師動眾」的決定該待的地方——**輕量 RFC**：一頁固定模板（問題是什麼／提案／考慮過的替代方案／取捨／決議），開一個任何人都能留言的文件，設一個明確的截止日期讓討論收斂。它的機制是把決策的頻寬從「同步、即時、少數人在場」拉成「**非同步、可留存、所有相關方都能參與**」——你不必把所有利害關係人約進同一間會議室的同一個小時，他們可以在自己方便的時候讀、想、留意見。

再往重的一端走，是**正式 RFC 加評審會與簽核**：指定 reviewer、必經評審、明確的 approve/reject 狀態。它的把關最強，代價是流程最重、最慢。這套適合的是那些**不可逆、跨多個團隊、出錯代價極高**的決定——換資料庫、定一個公司級的服務間協定、選一個未來五年都要綁著的雲端供應商。

這道光譜的關鍵判準，是讓**流程的重量對齊決定的可逆性與波及面**。三個問題就夠用了：這個決定錯了，要回頭的代價多大？它會影響幾個團隊？半年後會不會有人來問為什麼？三個裡有一個夠重，就值得一頁正經的 RFC；三個都輕，走廊講一句然後在 commit message 裡帶一筆就夠了，硬要它走 RFC 流程反而是浪費。

## 一個流程怎麼會自己殺死自己

光譜聽起來很清楚，但有一個非顯然的失敗模式，幾乎每個第一次導入 RFC 流程的組織都會踩——而且踩法很反直覺：**用力過猛的流程，會把它本來想保護的決定趕到看不見的地方。**

想像一個五十人的工程組織，決定「以後重要技術決定都要走 RFC」。一個很自然、看起來很負責任的做法是一步到位上重流程：每個技術決定都要指定 reviewer、進評審會、走簽核。然後會發生什麼？工程師很快算出一筆帳——光是把一個小決定推過這套流程的時間，比實際把它做掉還長。於是理性的反應是**繞過**：真正的決定退回到走廊和私訊裡偷偷做掉，只把那些無關痛癢、不做也沒差的東西寫成 RFC 來應付流程。

結果是最糟的一種：流程空轉成儀式，重要決定反而比導入流程**之前**更不透明了——因為現在大家學會了「真正的事不要寫進那個會卡我的系統」。流程不是中立的；它太重，就會主動把行為推離它想捕捉的目標。

穩當的導入方向恰好相反，是**從最輕開始、讓人先嘗到甜頭**：一開始只要求「會影響其他團隊的決定」寫一頁輕量 RFC，設 72 小時評論窗，無異議即生效。把儀式壓到幾乎沒有，先讓人經驗到「寫下來之後，討論真的變清楚了、扯皮真的變少了」這件事本身的好處。等這個習慣長出來、大家認同它有用之後，再針對「真正不可逆的大決定」選擇性地加重流程。流程的重量是**長出來的**，不是一開始就壓上去的。

這裡藏著一個更普遍的道理：**任何靠人遵守的慣例，如果遵守它的成本高於不遵守，它最終會被繞過，而你甚至看不到繞過發生**——因為繞過的人不會來告訴你。RFC 流程的設計，本質上是在做一個成本工程：把「寫下來」的阻力壓到低於「不寫下來」的長期痛苦，讓人**自願**走進這套流程。這和把對的做法鋪成最省力的路（見〈鋪路：讓對的做法最省力〉）是同一個思路的兩面。

## 替代方案那一節，是整份文件的良心

如果只能保留 RFC 模板裡的一節，該保留哪一節？不是「提案」，也不是「決議」——是**「考慮過的替代方案」（Alternatives Considered）**。

理由要從這一節最常見的造假講起。大量 RFC 的替代方案一節，是**寫完決定之後**硬湊出來的：作者心裡早就定了用方案 A，然後為了讓文件「看起來有比較過」，隨手塞兩個明顯更差的選項（方案 B「太慢」、方案 C「太複雜」）來陪襯。這種替代方案分析是純粹的裝飾——它沒有揭露任何取捨，只是給一個已成的決定披上「經過審慎評估」的外衣。一個有經驗的 reviewer 一眼就能認出這種陪襯：那兩個替代方案弱得可疑，弱到你會懷疑作者根本沒認真考慮過它們。

真正有價值的替代方案分析長得完全不同。它誠實地寫出「**這個方案其實也行，我們因為某個具體的取捨才不選它**」。比如:「方案 B（同步呼叫）其實更簡單、更好除錯，我們不選它，是因為在預估的尖峰流量下，它會讓這個請求的 p99 延遲被下游服務的延遲完全綁死，而我們有一個 200ms 的 SLO 要守。」這樣寫，揭露的是真正的決策邊界——它告訴未來的讀者：如果有一天那個 200ms 的 SLO 放寬了，或者下游變快了，方案 B 是可以回頭考慮的。

這就是為什麼這一節是整份文件的良心。一個決定真正的價值，不在「選了什麼」，而在「**在什麼條件下、為了什麼取捨、放棄了什麼**」。選了什麼，看程式碼就知道；放棄了什麼、為什麼放棄，只有這一節記得。半年後那個想把樂觀鎖改掉的人，他真正需要的，正是當初那份「我們考慮過行鎖，但因為這張表的寫衝突其實很少、而行鎖在這個熱點上會造成排隊」的誠實記錄。沒有它，他就只能重新踩一遍那個坑。

## RFC 講「為什麼這樣決定」，ADR 講「我們確實決定了」

到這裡為止，我們一直在談 RFC 與設計文件——它們是**決策前**的提案與討論載體：多個選項還攤在桌上、討論還在進行、結論還沒鎖定。但有一個近親，解的是稍微不同的問題，常常被和 RFC 混為一談：**架構決策紀錄（Architecture Decision Record，ADR）。**

ADR 是**決策後**的精簡紀錄。它不是用來討論的，討論已經結束了；它是用來**釘住**一個已成的決定，連同它的脈絡與後果。這個概念與它的標準形態，來自 Michael Nygard 2011 年的一篇短文〈Documenting Architecture Decisions〉，模板小到只有五欄，而且十幾年來幾乎沒變過：

| 欄位 | 內容 |
|---|---|
| Title | 一個短名詞片語，附遞增編號，如「ADR-0007: Use optimistic locking for order updates」 |
| Status | proposed / accepted / deprecated / superseded by ADR-NNNN |
| Context | 當時在角力的各種力量（技術的、組織的、時程的），中性陳述、不粉飾 |
| Decision | 我們實際決定做什麼，用主動語態完整句子寫 |
| Consequences | 套用這個決定之後的處境——好的壞的都列在同一節 |

模板簡單到近乎簡陋，但每一欄的設計都有用意，其中兩個機制特別值得拆開看，因為它們決定了 ADR 能不能真正可信。

第一個是 **Consequences 把好處和代價放在同一節**。這不是排版偷懶，是刻意的逼供。當你被要求在「決定」的正下方、同一個標題裡，寫出這個決定的負面後果，你就很難再寫出那種「這個選擇毫無代價、純粹是更好」的虛假文件。它強迫作者承認：我們選了樂觀鎖，好處是讀不擋寫、熱點不排隊；代價是寫衝突時要靠應用層重試，而且如果衝突其實很頻繁，重試風暴會反過來咬我們。把代價和好處並列，文件就不能說謊。

## 不可變，才可信

第二個機制更微妙，是 ADR 整套價值的地基：**一份 ADR 一旦被接受（accepted），它的內文——Context 與 Decision——就不再修改。** 唯一會動的是 Status 這個元欄位：當這個決定後來被推翻，它的 Status 會被翻成「superseded by ADR-NNNN」指向接班的那份，但決定本身的文字一個字都不動。

這聽起來違反直覺——決定變了，難道不該回去把文件改成新的樣子嗎？恰恰不該。設想你三年前寫了 ADR-0007「用樂觀鎖」，今天發現這個決定錯了，要改成悲觀鎖。如果你回去把 0007 直接改成「用悲觀鎖」，會發生一件糟糕的事：**歷史被抹掉了。** 任何讀到改過的 0007 的人,都會以為「我們從一開始就用悲觀鎖、而且有充分理由」——當初為什麼選樂觀鎖、後來為什麼推翻、推翻的理由是什麼，全部消失。更糟的是，下一個工程師可能會在不知道這段歷史的情況下，重新提議「我們改用樂觀鎖吧，讀不擋寫多好」，然後把你們三年來學到的教訓再交一次學費。

正確的做法是：**舊的 ADR 永遠不動，只把它的 Status 改成「superseded by ADR-0023」，然後寫一份全新的 ADR-0023 記錄新決定、以及為什麼推翻舊的。** 編號單調遞增、永不重用，即使某個 ADR 被廢棄了，它的號碼也不會讓給別人。於是這一串 ADR 連起來，就成了一條**決策的地質層**——你能順著它讀出整個系統的架構是怎麼一步步演化成今天這樣的，每一層為什麼被下一層覆蓋。

不可變正是這條地質層可信的全部原因。一份可以被隨意回頭修改的紀錄，記載的是「現在的人希望當初是怎麼決定的」，而不是「當初到底怎麼決定的」——它會被現在的認知不斷地、無聲地重寫，最後變成一個美化過的、自我合理化的版本。而一份不可變的紀錄，記的是當時的真相，包括當時的無知和後來被證明錯誤的判斷。**正是「不能改」這個約束，讓這份紀錄值得信任**——它不會偷偷地把過去改成符合現在的樣子。

一份 RFC 和一份 ADR 之間常有的關係是：一份充分討論過的 RFC，最後**濃縮**出一條或幾條 ADR。RFC 是過程——多個選項、來回的爭論、被否決的方向，全部攤開；ADR 是結晶——把那個過程沉澱成「我們最終決定了什麼、在什麼脈絡下、付什麼代價」的一頁。前者厚、易失（讀過就過去了），後者薄、持久（要被未來反覆查閱）。

## 「徵求意見」這四個字的重量

值得在收尾前停下來看一眼 RFC 這個縮寫本身：**Request for Comments——徵求意見。** 這個名字不是隨便取的，它來自網際網路標準制定的傳統，而那個傳統對「決定該怎麼做出來」有一個相當深刻、也相當反直覺的立場。

IETF（制定網際網路協定的組織）有一句廣為流傳的信條，出自 David Clark 1992 年的一場演講：「我們拒絕：國王、總統與投票。我們信仰：粗略共識與可運行的程式碼（rough consensus and running code）。」這句話的前半段是在說，重要的技術決定**不靠多數表決**——他們甚至刻意用「哼聲（humming）」而不是舉手計票，就是為了避免把它變成一場數人頭的勝負。後來 IETF 把這套思路寫成了 RFC 7282〈On Consensus and Humming in the IETF〉，專門解釋什麼叫「粗略共識」。

為什麼不投票？因為投票會製造輸家。一個 51% 對 49% 通過的技術決定，意味著有將近一半的人帶著沒被回應的顧慮被輾過去了——而那些顧慮往往是真的技術問題，不會因為票數輸了就消失，它們會在六個月後變成生產事故回來找你。粗略共識的核心精神是:**不是「多數同意就過」，而是「所有被認真提出的反對意見，都已經被聽見並處理過」**——不一定要讓每個人都滿意（那叫全體一致，太貴也太慢），但每一個有技術根據的異議都不能被無視地碾過去。

這正是 RFC 這套機制最深的一層用意。它表面上是「把決定寫下來給後人看」，但「徵求意見」這個動作本身，是在**主動製造一個讓異議浮現、被聽見、被處理的場合**。那個在你腦中覺得方案完美無缺、但有個你沒考慮到的失敗模式的同事，需要一個低成本的管道把他的顧慮說出來——而一份開放留言的 RFC，正是那個管道。把決定寫下來、公開徵意見，買到的不只是未來的可追溯性，還有當下的**一次集體糾錯的機會**，趁它還只是文件、改起來幾乎免費的時候。

## 為什麼是這個形狀

退一步看，RFC、設計文件、ADR 這一整套東西，全都是從同一個樸素到容易被忽視的事實裡長出來的：**程式碼忠實地記錄了「做了什麼」，卻對「為什麼」徹底沉默；而「為什麼」恰恰是隨人員流動衰減得最快、未來又最需要的那部分知識。**

正因為「為什麼」會衰減，所以要把它**寫下來**，趁寫的人還記得。正因為寫的當下會逼出思考、暴露缺口，所以這套機制最大的價值反而在動筆的那一刻、不在批准的那一刻。正因為不同決定的份量天差地別，所以流程要分輕重、對齊可逆性，而不是一刀切——一刀切要嘛把大決定放掉、要嘛把小決定的流程逼到沒人遵守。正因為一個決定真正的價值在「放棄了什麼、為什麼放棄」，所以替代方案那一節是整份文件的良心。正因為被現在不斷改寫的歷史不值得信任，所以 ADR 一旦接受就不可變、只能被新的取代。正因為被表決碾過去的異議不會消失、只會延後爆發，所以「徵求意見」要主動把異議請出來、趁早處理。

這套慣例不是某種官僚的儀式，也不是「成熟團隊應該要有的東西」這種空洞的最佳實踐。它是一群人在「決定會消失、但後果會留下」這個結構性困境前，唯一站得住的應對：在決定還熱、人還記得、改還便宜的那個短暫窗口裡，把它**逼成文字、攤開徵意見、釘成不可變的紀錄**——這樣半年後那個翻到你那段樂觀鎖、問「為什麼」的人，得到的就不是沉默，而是一份當年的、誠實的回答。
