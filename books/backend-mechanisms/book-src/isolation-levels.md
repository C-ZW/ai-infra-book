# 隔離級別與異常：dirty、phantom 到 write-skew

設想一個排班系統，規則只有一條：任何時刻，急診室至少要留一個醫生值班。今晚 Alice 和 Bob 都值班，兩人都想請假回家。系統的檢查邏輯很直白——請假前先數一下「現在還有幾個人值班」，只要算出來 ≥ 2，就准。

兩人幾乎同時點下「請假」。Alice 的交易讀到「值班人數 = 2」，通過檢查，把自己標成休假。Bob 的交易也讀到「值班人數 = 2」（他讀的那一瞬間，Alice 還沒提交），通過檢查，把自己也標成休假。兩筆交易各自看都合法——「我請假時明明還有兩個人」——兩筆都成功提交。然後急診室一個醫生都不剩。

詭異的地方在於：**沒有任何一列被兩個交易同時寫。** Alice 改的是 Alice 那列，Bob 改的是 Bob 那列，它們井水不犯河水。任何「偵測同一列被併發改寫」的機制——行鎖、版本欄位、CAS——在這裡全部沉默，因為衝突根本不在「同一列」上，而在一個**跨列的不變量**上：兩列的「值班總數」這個聚合，被兩筆各自合法的寫，聯手推下了懸崖。

這就是 **write skew（寫偏序）**，而它之所以能逃過所有看似嚴密的防線，根源要回到一個更基本的問題：當多個交易同時碰同一份資料，「隔離」到底是什麼意思、又是用什麼定義的。這一章就從這裡開始。

## 隔離是用「禁止哪些壞事」定義的，不是用「鎖什麼」

ACID 的 I 是 isolation。它承諾的理想很乾淨：每個交易執行時，**像是**整個資料庫只有它一個人在用，別人的併發操作對它不可見。這個理想有個精確的名字叫 **serializable（可序列化）**——一批併發交易跑出來的結果，必須等同於「把它們排成某個串列順序、一個接一個跑完」的結果。注意這裡只要求**存在某個**串列順序能解釋觀察到的結果，不要求是哪一個；只要找得到一個說得通的順序，就算可序列化。

問題是完整的 serializable 很貴——它逼著併發交易在任何可能互相干擾的地方排隊或重來，吞吐會掉。於是現實裡的資料庫提供**較弱的級別**，讓你用一部分隔離性換並發。但「較弱」要怎麼精確定義？

這裡是理解整套機制的第一個關鍵轉折，也是最反直覺的一點：**SQL 標準的隔離級別不是用「這個級別會鎖什麼、用什麼演算法」定義的，而是用「這個級別容許哪些壞現象發生」定義的。** 標準刻意不規定實作——它不在乎你是用鎖、用多版本、還是用什麼魔法——它只規定一條紅線清單：在這個級別，下面這幾種讀寫交錯的「異常（anomaly）」**不准出現**，至於其他的，悉聽尊便。

這個「用禁止的現象來定義」的設計，是 1992 年的 ANSI SQL-92 標準為了讓不同廠商的資料庫能用同一套語言描述隔離強度而定的（更早 1970 年代的 System R 那套是用「鎖的等級」定義的，正是 SQL-92 想擺脫的綁定）。它的好處是實作中立；它的代價，我們很快會看到，是這套現象清單本身漏洞百出，定義出來的階梯並不是它看起來的那條直線。

先把那幾個經典異常一個個講清楚，因為它們是後面一切的詞彙。

## 三種經典異常，與它們各自漏掉什麼

**dirty read（髒讀）** 是最粗暴的一種：你讀到了另一個**還沒提交**的交易寫進去的值。對方寫了 `balance = 500`，你讀到 500，然後對方 rollback——它那筆寫從未真正發生，但你已經拿著一個**從未存在過**的值在做決定。髒讀的危險不只是「值可能變」，而是「值可能根本是幻覺」。要堵它，引擎只要保證「你只看得到已提交的資料」即可，這幾乎是所有實用級別的最低門檻。

**non-repeatable read（不可重複讀）** 細緻一級：你在同一個交易裡，讀**同一列**兩次，兩次的值不一樣。中間有別人改了那列並提交了。你的交易明明還沒結束，腳下的資料卻已經動了。對需要「交易內看到的世界保持穩定」的邏輯——先算一個值、晚點再依它做決定——這是致命的，因為你第二次讀到的，已經不是你第一次據以決策的那個世界。

**phantom read（幻讀）** 和不可重複讀長得像，但差別微妙且重要。不可重複讀是「一列的值變了」；幻讀是「**符合條件的列集合變了**」。你在交易裡跑 `SELECT count(*) FROM shifts WHERE on_call = true` 得到 2，做了點事，再跑同一句，得到 3——不是哪一列的值被改，而是別人**插入**（或刪除）了一列，讓你那個範圍查詢的結果集多（少）了一個成員。差別之所以要緊，是因為防住「列被改」和防住「列被插」是兩件技術上不同的事：前者只要盯住已存在的列，後者得盯住「還不存在、但未來可能落進這個範圍」的位置——這是個本質上更難守的承諾，我們待會看 MySQL 怎麼用鎖把整個範圍連縫隙一起鎖死來做到。

SQL 標準就用「容許／禁止這三種裡的哪幾種」排出四個級別。從最弱到最強，每升一級多禁止一種：

| 級別 | dirty read | non-repeatable read | phantom read |
|---|---|---|---|
| Read Uncommitted | 容許 | 容許 | 容許 |
| Read Committed | 禁止 | 容許 | 容許 |
| Repeatable Read | 禁止 | 禁止 | 容許 |
| Serializable | 禁止 | 禁止 | 禁止 |

看起來很整齊：一條筆直的階梯，每爬一階就更安全一點，爬到頂（Serializable）就萬無一失。這張三異常的表是隔離級別最廣為人知的圖像，也恰恰是它最會誤導人的地方——**因為這三種異常根本沒蓋全。** 開場那個排班災難，就不在這張表的任何一格裡。

## 階梯的謊言：snapshot isolation 不在這條線上

要看穿這張表的破綻，得先知道現實裡絕大多數資料庫的「Repeatable Read」是怎麼實作的。它們不是真的去逐列上讀鎖、防止別人改動，那太貴。它們用的是**多版本**：寫不覆蓋舊值，而是生出一個新版本；每個交易在開始時（或每條語句開始時）拿一個一致的「快照」，整個交易期間就看那個時間點凍結的世界，別人後來改了什麼、提交了什麼，你的快照一概看不見（多版本的可見性判斷怎麼運作，是它自己的一章，見〈MVCC：讓讀不擋寫〉）。

這套「交易看一個凍結快照」的做法，學名叫 **snapshot isolation（快照隔離）**，1995 年 Berenson、Bernstein、Gray 等人那篇《A Critique of ANSI SQL Isolation Levels》才把它正式命名並分析。它有個非常好的性質：因為你整個交易都看同一個快照，**non-repeatable read 不可能發生**（同一列你永遠讀到快照裡那個版本），而且——這是重點——連 **phantom read 也不會發生**（別人新插的列不在你的快照裡，你那個範圍查詢跑幾次結果都一樣）。

於是廠商就把 snapshot isolation 掛上「Repeatable Read」這個招牌賣。PostgreSQL 的官方文件講得毫不遮掩：它的 Repeatable Read「是用學界與某些資料庫產品稱為 snapshot isolation 的技術實作的」，而且明說這個實作**連 phantom read 都防住**——比 SQL 標準要求的 Repeatable Read 還強。標準只規定「至少要禁止哪些」，提供更強的保證不違規。

聽起來是好消息：我買 Repeatable Read，賺到了標準沒承諾的防幻讀。但天下沒有白賺的保證。snapshot isolation 換來防幻讀的同時，**漏掉了一整類標準那三個異常從未描述過的東西**——正是 write skew。Berenson 那篇論文的結論一針見血，也是這一章的樞紐：snapshot isolation **不是**「比 Repeatable Read 強一點、比 Serializable 弱一點」的某個中間階——它和**以鎖實作的** Repeatable Read（那種靠對讀過的列上長效讀鎖、因而擋得住 write skew、卻擋不住幻讀的 RR）是**incomparable（不可比較的）**。（對照上一段：若只看 SQL 標準那張三異常的表、字面只要求禁 P1／P2，SI 其實比它強——SI 額外防了幻讀；不可比較講的是 SI 與鎖式 RR 這兩種「都比標準字面強、但各強在不同地方」的實作之間。）

「不可比較」這四個字值得停下來咀嚼。它的意思是：snapshot isolation 防住了一些 Repeatable Read 不防的東西（幻讀），同時又漏掉了一些別的東西（write skew 那類跨列依賴的異常）。兩者各有對方沒有的洞，誰也不包含誰。那條看起來筆直的、一階比一階安全的階梯，在這裡**斷掉了**——看起來像往上爬了一級到「Repeatable Read」，其實是橫跨到了另一條線上，那條線堵住了幻讀，卻在腳下開了一個標準清單裡根本沒畫出來的新坑。

這就是為什麼開場那個排班災難能發生在一個「設定成 Repeatable Read、看起來很安全」的資料庫上。它防住了髒讀、防住了不可重複讀、甚至防住了幻讀——但它**就是不防 write skew**，而那張三異常的表，從頭到尾沒有一欄叫 write skew。

## 把 write skew 一步步重放

抽象的「不可比較」講完了，把它落到具體數字上，才看得清坑長什麼樣。換一個比排班更容易手算的版本：兩個共用帳戶 A 和 B，銀行的規則是「兩戶總餘額不得為負」。初始 A = 100、B = 100，總額 200，健康。

現在兩筆交易幾乎同時進來，都在 snapshot isolation 下執行：

```
時間軸（兩交易各自看自己開始時的快照）

T1: 從 A 領 150                      T2: 從 B 領 150
─────────────────────────────       ─────────────────────────────
t1  BEGIN (快照: A=100, B=100)
t2                                   BEGIN (快照: A=100, B=100)
t3  讀 A=100, B=100 → 總額 200
t4  檢查: 領完 A=-50, A+B=50 ≥ 0 ✓
t5                                   讀 A=100, B=100 → 總額 200
t6                                   檢查: 領完 B=-50, A+B=50 ≥ 0 ✓
t7  UPDATE A = -50
t8                                   UPDATE B = -50
t9  COMMIT
t10                                  COMMIT

結果: A = -50, B = -50, 總額 = -100   ← 不變量被打破
```

逐格重放，看清楚每一步**為什麼合法**。T1 在 t3 讀到的是它自己快照裡的世界：A=100、B=100。它在 t4 做的檢查完全正確——「我領走 150，A 變成 -50，但 A+B 還有 50，沒違規」。T2 在 t5 讀的也是**它自己**快照裡的世界，而它的快照同樣是 A=100、B=100（t2 開始時，T1 還沒提交，所以 T2 看不到 T1 任何動作）。T2 的檢查在它的視角下也完全正確。

關鍵在 t7 和 t8：T1 寫的是 A 這一列，T2 寫的是 B 這一列。**它們寫的不是同一列。** snapshot isolation 在提交時會做一件事——檢查「我這次交易寫過的列，有沒有被別的並發交易也寫過並先提交了」（這叫 first-committer-wins，是它防 lost update 的機制；PostgreSQL 的具體實作因在 update 當下就上行鎖、衝突在提交時報 `could not serialize access due to concurrent update`，文件稱之 first-updater-wins，本質是同一道寫集合衝突檢查）。T1 寫了 A，T2 寫了 B，兩個寫集合**沒有交集**，所以這個檢查在 t9 和 t10 都順利通過，兩筆都成功提交。

災難的全部祕密就在這裡：**snapshot isolation 只盯「寫－寫衝突」（同一列被兩個交易都寫），但 write skew 是「讀－寫衝突」——T1 讀了 B（用來算總額）、然後 T2 寫了 B；T2 讀了 A、然後 T1 寫了 A。** 每個交易都建立在「對方那列的舊值」上做決策，而對方緊接著把那個值改掉了。它們各自讀的東西被對方寫廢了，但因為「讀了什麼」不在寫集合裡，first-committer-wins 那道檢查根本看不到這種交叉。版本欄位、CAS、行鎖——這些守的全是「同一列的寫－寫」，自然也全部失效。

凡是「**先查一個聚合條件、再據此寫一列**」的邏輯，都是 write skew 的溫床，遠不只帳戶：庫存「還有貨才扣」、會議室「沒人訂才占」、唯一性「查不到才插入」（先 `SELECT` 確認某 email 不存在、再 `INSERT`，兩個併發請求都查到「不存在」於是都插入）。它們的共同骨架都是：讀一個橫跨多列或多次查詢的條件，據此寫，而那個條件被另一筆併發交易在你讀完之後悄悄改變了。

## 怎麼真正堵住它：兩種防線、兩種代價

知道病根是「讀－寫交叉沒人管」，解法就清楚了：要嘛**讓引擎去追蹤這些讀寫依賴**，要嘛**把跨列的讀變成一個有形的寫鎖、逼出衝突**。這對應兩條真正能防 write skew 的路。

第一條是升到貨真價實的 **Serializable**。PostgreSQL 走的是 **SSI（Serializable Snapshot Isolation）**：它在 snapshot isolation 之上，額外去追蹤交易之間的讀寫依賴——誰讀了什麼、之後誰寫了那個東西——當它偵測到一組依賴構成「會導致無法序列化的危險環」（理論上是讀寫依賴圖裡出現了特定形狀的環），就在提交時把其中一個交易中止，回報一個錯誤：

```
ERROR:  could not serialize access due to read/write dependencies among transactions
```

SQLSTATE 是 `40001`。回到帳戶的例子：在 Serializable 下，引擎會注意到 T1 讀了 B 而 T2 寫了 B、T2 讀了 A 而 T1 寫了 A，這個交叉正是危險環，於是中止其中一筆。被中止的交易拿到 `40001`，**應用程式必須重跑整個交易**。這是 SSI 的靈魂，也是它最常被忽略的代價：它不像鎖那樣讓你等，而是讓你**重來**——你的程式碼裡每一個 Serializable 交易外面，都得包一層「收到 40001 就從頭重試」的迴圈。沒有那層迴圈，「升到 Serializable」的淨效果就是把資料錯誤換成了使用者眼中的隨機報錯。SSI 把「正確性」這個保證，明碼標價成「你要會處理中止重試」這個義務。

第二條路，是不改全域隔離級別，而在那條危險的讀上，**把隱性的讀變成顯性的鎖**——`SELECT ... FOR UPDATE`。回到排班：請假前不要用普通的 `SELECT count(*)` 去數值班人數，而是 `SELECT ... FOR UPDATE` 把「目前值班的那些列」實際鎖住。這麼一來，Alice 的交易鎖住了那兩列，Bob 的交易要讀同一批列時就被擋住、得等 Alice 提交或回滾後才能繼續——等他終於讀到時，看到的是「只剩一人值班」的新事實，他的檢查就會擋下他的請假。`FOR UPDATE` 的作用，是把原本「無形的、引擎管不到的讀」變成「有形的、會和別的寫互斥的鎖」，**人為地把 write skew 那個藏起來的讀－寫衝突，物化（materialize）成一個看得見的寫－寫衝突**——而寫－寫衝突，正是所有引擎本來就會處理的東西。代價是你退回了悲觀鎖的世界：要承受持鎖期間的等待，以及多列加鎖時的死結風險（鎖、死結與加鎖順序是另一套機制，見〈鎖與死結：行鎖、gap、next-key 與 wait-for graph〉）。

兩條路是兩種哲學：SSI 是樂觀的——先放手讓大家跑，提交時才驗、發現危險就中止重來，賭的是衝突少、重試成本低；`FOR UPDATE` 是悲觀的——進場前先鎖、把衝突擋在門外，賭的是衝突多、重試會更貴（樂觀與悲觀這對哲學，見〈樂觀與悲觀並發控制：CAS、版本與 ETag〉）。選哪條，取決於那條業務的衝突率，沒有絕對答案。

## 同名不同義：「Repeatable Read」是個壞契約

到這裡，最該被打碎的一個假設是：**「隔離級別的名字是個可靠的契約」。** 它不是。同一個級別名，在不同資料庫裡保證的東西可以差很遠，因為標準只規定「至少要禁止哪些異常」，沒規定「不准多禁止」也沒規定「實作怎麼做」，留白被各家用不同方式填了。

PostgreSQL 的 Repeatable Read 是 snapshot isolation：強到連幻讀都防，但不防 write skew。它沒有用鎖去守範圍，而是靠那個凍結的快照——別人插的列不在你快照裡，你自然看不見幻影。

MySQL 的 InnoDB 走的是完全不同的路。它的預設級別就是 Repeatable Read，但它防幻讀的手段不是快照，而是**鎖**——具體說是 **next-key lock**。當你在 Repeatable Read 下用範圍條件做 locking read，InnoDB 不只鎖住命中的那幾列，還鎖住「**列與列之間的縫隙**」（gap lock）——record lock 加上它前面那段 gap，合起來就是 next-key lock。鎖住縫隙的意義是：別人**沒辦法往這個範圍裡插入新列**，因為要插的那個位置已經被你的 gap lock 占住了。沒有插入，就沒有幻影。這是用「先把整個範圍連縫隙鎖死」這種悲觀手段，達到和 PostgreSQL 快照同樣「防幻讀」的效果，但代價完全不同——gap lock 鎖住的範圍常比直覺大得多，是 InnoDB 一類隱性死結的來源。

更要命的是，InnoDB 的 Repeatable Read 不只實作和 PostgreSQL 不同，**保證強度也不同**。Jepsen 在 2023 年對 MySQL 8.0.34 的測試裡發現，InnoDB 的 Repeatable Read **連 snapshot isolation 都不滿足**——在單一健康節點、正常併發負載下就能重現 lost update、read skew（G-single）、write skew（G2-item）這些異常。也就是說，PostgreSQL 的 RR 至少穩穩地是 snapshot isolation（強、只漏 write skew），而 MySQL 的 RR 比 snapshot isolation 還要弱、漏的更多。同樣寫著「Repeatable Read」，一邊漏一個坑，另一邊漏一籮筐。

| | PostgreSQL Repeatable Read | MySQL InnoDB Repeatable Read |
|---|---|---|
| 實作機制 | snapshot isolation（純多版本快照） | 多版本一致讀 ＋ next-key lock |
| 防幻讀的手段 | 別人插的列不在快照裡 | gap lock 鎖住範圍縫隙、擋住插入 |
| 實際保證 | 是 snapshot isolation；防幻讀、不防 write skew | 比 snapshot isolation 還弱（Jepsen：有 lost update、read skew、write skew） |
| 防 write skew 要怎麼做 | 升 Serializable（SSI）或 `SELECT ... FOR UPDATE` | 同上；不能假設 RR 幫你擋住 |

這張表的教訓不是「哪個資料庫比較好」，而是：**把應用從一個資料庫搬到另一個、只改了連線字串，你的併發正確性不會自動跟過去。** 預設級別不同（PG 是 Read Committed、MySQL 是 Repeatable Read）、同名級別的真實語意不同、防異常的機制不同——這些差異全部藏在「Repeatable Read」這四個一模一樣的字底下。名字是壞契約，唯一可靠的契約是「這個級別在這個資料庫上，實測會放掉哪些異常」。

## 為什麼最高級別不是免費的萬靈丹

讀到這裡，一個很自然的念頭是：那我全部都設成 Serializable 不就好了，一勞永逸，連 write skew 都不用想？

這個念頭值得認真對待，但它低估了 Serializable 的代價，而且那代價的**形狀**和你以為的不一樣。Serializable 不是一個「打開就更安全」的開關，它是把**問題的形態換掉了**——從「資料可能悄悄錯」換成「交易可能隨時被打斷」。

在 PostgreSQL 的 SSI 下，這個代價是中止與重試：衝突一密，中止率就上去，你那層重試迴圈會被反覆觸發，每次重試都是把整個交易從頭再跑一遍的白工。在低衝突的場景，SSI 的開銷其實接近 snapshot isolation——它平時只是默默追蹤依賴，沒事不中止誰，所以「Serializable 一定很慢」這句話本身也是個需要先量過衝突率才能下的判斷，不是定律。但在高衝突的熱點上，中止重試的成本會像滾雪球，遠超過老老實實上鎖排隊。

在 MySQL 的 Serializable 下，代價是另一種形狀：在 autocommit 關閉（即顯式交易）下，它把普通的 `SELECT` 隱式升級成 `SELECT ... FOR SHARE`、給每個讀都加上共享鎖（autocommit 開著時，單獨一句 `SELECT` 自成一筆交易、仍是不加鎖的一致讀）。讀本來是不互相擋的，現在交易裡的讀也要排隊了，併發直接塌下來。沒有中止重試的麻煩，換來的是純粹的鎖等待與更高的死結機率。

所以「升到最高級別」從來不是免費的。它要麼讓你付重試的工（PG），要麼讓你付鎖等待的並發（MySQL）。**正確的姿勢不是把全庫設成 Serializable，而是認清「這條業務有沒有跨列／跨範圍的不變量、且兩筆交易真的可能同時打它」**——有，才為那條路徑付 Serializable 或 `FOR UPDATE` 的代價；沒有，Read Committed 對絕大多數「讀單列、寫單列、彼此不相干」的 OLTP 操作就夠用，升級隔離只是白白付並發成本。隔離級別是一把要按業務逐處權衡的精細刻度，不是一個全域拉到底就萬事大吉的旋鈕。

## 為什麼是這個形狀

退一步看整套機制的輪廓。隔離級別之所以長成「用一張容許／禁止異常的表」這個樣子，是因為完整的 serializable 太貴，而工程上需要一個能描述「我願意放掉哪些隔離性、換多少並發」的共同語言——用「禁止哪些可觀察的壞現象」來定義級別，正好做到實作中立、廠商可各自發揮。

但這個設計的代價，是那張現象清單先天不完整。SQL-92 那幾個異常其實是把「以鎖實作」的資料庫會漏的東西反過來描述（Berenson 那篇就指出，這幾個現象不過是鎖協議的變裝），框得住鎖會漏的東西；可是當多版本快照成為主流，一整類**讀－寫交叉**的異常（write skew 是其代表）從清單的縫隙裡漏了出去——它們不是清單上任何一格，所以一個老老實實「禁止了清單上所有該禁止的異常」的 snapshot isolation，依然會在排班、在共用帳戶、在庫存與唯一性檢查上爆掉。那條看似筆直的階梯，在 snapshot isolation 這裡橫岔成了另一條線；「不可比較」不是學術上的吹毛求疵，而是會在生產環境吃掉你不變量的真實裂縫。

而真正的 Serializable 之所以貴，是因為要堵住這個縫，引擎不是去追蹤每筆交易「讀了什麼、那東西之後被誰寫了」這張依賴圖（SSI、付重試），就是把每個範圍讀都升級成鎖（next-key、共享鎖，付並發）。沒有第三條免費的路——「在併發下讓每個交易都像獨佔資料庫」這個承諾，要嘛用樂觀的中止重試買、要嘛用悲觀的鎖等待買，兩者都不便宜。

所以下次你在一個設成 Repeatable Read 的資料庫上，看到一個「先查條件再寫入」的邏輯，請不要被那個名字安撫。問一句：這裡有沒有一個跨列的不變量，會被兩筆各自合法的寫聯手打破？如果有，那張三異常的表救不了你——你需要的是 Serializable 配一層重試迴圈，或一句把無形的讀變成有形鎖的 `SELECT ... FOR UPDATE`。隔離級別不是一條安全到不安全的直線，而是一張有缺口的地圖；認得出缺口在哪，才是真正讀懂了它。
