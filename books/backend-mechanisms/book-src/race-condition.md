# race condition 與原子性：跨 DB、cache、runtime

一個庫存只剩一件的商品，兩個人同一秒按下「結帳」。程式碼長得無辜：先 `SELECT stock FROM items WHERE id = 42`，讀到 1；判斷 `if stock > 0`，通過；再 `UPDATE items SET stock = stock - 1`，扣成 0；最後建一張訂單。在開發機上，你怎麼點都對——因為你一次只點得了一下。上線後某個促銷的尖峰，兩個請求幾乎同時抵達：請求 A 讀到 stock=1，還沒來得及扣，請求 B 也讀到 stock=1。兩個都通過了 `if stock > 0`，兩個都建了訂單，庫存最後是 0 或 -1，但你賣出了**兩件**只有一件的東西。

這個 bug 有一個討厭的性質：它在測試裡幾乎不出現，在 code review 裡看起來完全正確，只在真實流量的某個時序縫隙裡才偶爾露面，然後在你查 log 時又消失。它是後端最常見、也最容易被「在我機器上沒事」這句話放過的一類錯誤。這一章要做的，是把這個「縫隙」講到你能在腦中一格一格重放它、看清它為什麼會在三個完全不同的層次（runtime 內、跨快取、跨資料庫）反覆長出來，以及每一層各自用什麼把它堵死。

## 那道縫隙到底是什麼

把上面那筆「賣超」拆成最小的動作，它其實是一個結構：**讀 → 改 → 寫（read-modify-write）**，中間夾著一個判斷。問題不在任何單一步驟，而在於這三步**不是一個不可分割的整體**——在「讀」和「寫」之間，存在一段時間，這段時間裡別人可以插進來，基於同樣的舊值做出同樣的決定。

```text
（示意，非可執行——stock 初值 = 1）
時間 ───────────────────────────────────────────►
A:  read 1 ──► 1>0? yes ──────────────► write 0   建單
B:        read 1 ──► 1>0? yes ──────────────► write 0   建單
                                          兩張單都成立，賣超
```

這就是 **race condition（競態）** 的本質：多個執行單元並發存取同一份共享狀態，最終結果取決於它們交錯執行的**時序**，於是出現「有時對、有時錯」的不確定行為。它的孿生兄弟叫 **lost update（更新遺失）**——兩個 `+1` 只生效了一次，因為兩者都讀到改之前的舊值。庫存賣超，本質上就是一次 lost update 加上一個基於舊值的判斷。

對付它的解藥，名字叫 **原子性（atomicity）**：一個操作要嘛完整發生、要嘛完全不發生，中途**沒有任何其他單元能看到半成品、或在它中間插隊**。注意這裡的「原子」和交易（ACID）裡那個 A 是同一個詞但範圍不同——交易的原子性管的是「一組寫要嘛全成功要嘛全回滾」，而我們這裡要的，是更細的「這個讀-改-寫不可被打斷」。把那道縫隙消滅掉，競態就無處藏身。

問題是，「把讀-改-寫變成不可打斷」這件事，在不同的層次有完全不同的工具，而且有一個極容易誤判的前提：**很多人以為只有多執行緒程式才需要操心這件事**。這個誤會值得先拆掉，因為它決定了你會不會在自以為安全的地方留下縫隙。

## 三個層次：競態不挑語言

原子性不是「多執行緒專屬」的麻煩。同一道縫隙，會在三個彼此獨立的層次上各長一次，而你的程式很可能同時暴露在三者之下：

- **runtime 內**：同一個行程裡，多個執行緒（或多個交錯的 async 流程）共享同一塊記憶體。
- **跨 cache**：多個應用實例，打同一個 Redis（或 Memcached）。
- **跨 DB**：多個資料庫交易，改同一行。

最反直覺的是中間那層的根源。你可能寫的是「單執行緒」的 Node 服務，runtime 內確實不會有兩條執行緒同時動同一個變數——但這**不是免死金牌**。只要你的服務跑了多個實例打同一個 Redis 或 DB，競態就只是從 runtime 搬到了資料層；而且即使在單一行程內，只要程式碼裡有 `await`，縫隙照樣存在。

看一段「單執行緒所以安全」的典型錯覺：

```js
// 看起來安全，其實不是 —— 每個 await 都是一道縫
async function reserveSeat(seatId) {
  const seat = await db.getSeat(seatId);   // ① 讀（讓出）
  if (seat.taken) throw new Error('taken');
  await db.markTaken(seatId);              // ② 寫（讓出）
}
```

JavaScript 是單執行緒沒錯，但 `async/await` 是一種**協作式多工**：執行只在 `await` 那一刻把控制權交還給 event loop（event loop 怎麼運作見〈event loop：單執行緒怎麼同時做很多事〉）。在 ① 的 `await` 期間，event loop 可以去推進**另一個** `reserveSeat` 呼叫，讓它也跑到 ①、也讀到 `taken === false`。兩個流程於是都通過了檢查、都去 ②。沒有兩條 OS 執行緒、沒有共享記憶體的資料競爭，但 race condition 一模一樣地發生了——因為 `await` 把臨界區（critical section）從中間剖開了。**「單執行緒」消滅的是 CPU 指令層的交錯，不是邏輯流程層的交錯。** 只要你的「讀」和「寫」之間隔著一個 `await`，那道縫就在。

這給了我們一個統一的視角看完整章：競態的根源永遠是「讀-改-寫被某種讓出點切開」，差別只在切開它的是什麼——OS 的搶佔式排程（多執行緒）、event loop 的協作讓出（async）、還是「多個獨立行程各跑各的」（跨實例）。對應地，把它縫回去的工具也分三層。我們從最有效、卻最常被跳過的一招看起。

## 第一招：別讓讀-改-寫離開資料層

對付競態，工程師的第一反應常常是「加鎖」。但很多時候，最乾淨的解法是**根本不要在應用層做讀-改-寫**——把整個操作壓縮成資料庫或快取能一條指令原子完成的形式，讓那個本來就單執行緒、或本來就會上行鎖的引擎替你保證原子性。

回到開頭的計數器。天真版是「讀回來、在應用層 +1、寫回去」，三步分離，縫隙巨大。原子版只有一句：

```sql
UPDATE items SET stock = stock - 1 WHERE id = 42 AND stock > 0;
```

這一句的威力，值得拆開看 PostgreSQL 在預設的 READ COMMITTED 隔離級別下到底做了什麼。當請求 A 的這句 `UPDATE` 開始執行，它會對 id=42 那一行上一個**行寫鎖**。此刻請求 B 的同一句 `UPDATE` 想改同一行，會被擋住——它**不報錯、不跳過，就停在那裡等** A 的交易結束。等 A 提交，B 被喚醒，這裡是關鍵：PostgreSQL 不會拿 B 當初看到的舊快照硬寫，而是**重新讀取那一行被 A 改過後的最新值**，再重新評估 `WHERE` 條件。如果 A 已經把 stock 從 1 扣成 0，B 重評 `stock > 0` 時就**不成立**了，這一句更新到 0 行、扣減不發生。兩個請求，只有一個扣得掉，賣超被堵死。

這裡有一個非顯然、但 load-bearing 的細節：**READ COMMITTED 之所以能擋住這種 lost update，靠的不是隔離級別本身，而是「單一 `UPDATE` 語句在等鎖之後會重新讀最新行、重評 WHERE」這個行為。** 同樣是 READ COMMITTED，如果你改成「先 `SELECT stock`、應用層判斷、再 `UPDATE`」這種跨語句的讀-改-寫，這個重評就**不會**幫你——A 提交後，B 的應用層手上還是那個過時的 stock=1，照樣會做出錯誤判斷。所以不是「用了交易就安全」，而是「把判斷和寫**塞進同一句 SQL** 才安全」。把 `stock > 0` 從應用層的 `if` 搬進 SQL 的 `WHERE`，是這個解法的靈魂。

快取層有完全對等的一招。用 Redis 做跨實例計數時，天真版同樣是 `GET` 回來、應用層加、`SET` 回去，10 個實例一起這樣幹就是一場災難。原子版是一條指令：

```text
INCR rate:user:42        # 單一指令，Redis 保證原子，必定精確 +1
```

Redis 的指令**執行**是單執行緒的——所有 client 的指令在伺服器端排成一條隊，一次跑一個、跑完才跑下一個（Redis 6 之後加入的 threaded I/O 只是把網路讀寫卸載到別的執行緒，指令執行本身永遠不平行；見〈Redis 的內部〉）。所以 `INCR` 天生不可分割：1000 個並發請求打進來，無論時序怎麼交錯，計數一定精確地從 1 走到 1000，一次不多一次不少。

這一招的代價要說清楚，否則就成了口號：它**只適用於那些能用單一原子指令表達的更新**。計數、餘額增減、條件扣減——可以。但「讀出一個 JSON、改其中三個欄位、按某個跨欄位規則決定第四個欄位、再寫回」這種多步邏輯，沒有一條現成指令裝得下。而且它把競爭推到了**同一行/同一個 key 的熱點**上：所有人都搶改 id=42 那一行，行鎖會排成長隊，這一行就是你的吞吐瓶頸（要散熱得靠分片計數，把一個熱 key 拆成 N 個分片再加總，那是另一個取捨）。原子指令是這場戰爭裡最鋒利、但也最窄的武器——能用的時候優先用，用不了的時候才動更重的傢伙。

## 第二招：偵測衝突，撞了再重來（樂觀）

當更新邏輯複雜到塞不進一條指令，你被迫把它讀進應用層處理。這時有兩條路：要嘛賭「衝突很少發生」，先動手做、提交時才檢查有沒有被人插隊（樂觀）；要嘛假設「衝突常發生」，動手前先把門鎖上（悲觀）。先看樂觀，因為它在多數真實場景（衝突其實罕見）裡更划算。

**樂觀並發控制（optimistic concurrency control）** 的核心是一個版本印記。給每一行加一個 `version` 欄位（或用 PostgreSQL 內建的 `xmin`、HTTP 世界的 ETag，本質都是「這份資料的當前版本指紋」）。流程是：

1. 讀出資料，連同它當下的 `version`（比如 7）。
2. 在應用層慢慢算你要的新值，愛算多久算多久。
3. 寫回時帶上條件：`UPDATE ... SET ..., version = 8 WHERE id = ? AND version = 7`。
4. 看影響行數。改到 1 行——沒人動過，你贏了。改到 **0 行**——在你計算的這段時間裡，有人已經把 version 推到了 8，你手上的是舊值，這次更新作廢，**回到第 1 步重來**。

它的精髓是把「有沒有被插隊」這件事，變成一個**可偵測的事實**：`WHERE version = 7` 沒中，就是鐵證。它不上任何鎖，所以衝突不發生時零等待、零阻塞——這正是它在「寫少、衝突罕見」場景下勝出的原因。

Redis 在 8.4 之前，這一招要靠 `WATCH`/`MULTI`/`EXEC` 來做，機制和上面一模一樣：`WATCH key` 盯住一個 key，接著 `MULTI` 把你要做的寫排進交易，`EXEC` 提交時 Redis 檢查——**只要被 WATCH 的 key 在 WATCH 之後被任何 client 改過，整個交易就被放棄，`EXEC` 回傳 nil**，你得自己偵測這個 nil 並重來。這就是 Redis 版的樂觀鎖：不是真的鎖住，而是「提交時發現被動過就整批作廢」。Redis 8.4（2026-06）之後多了更省事的選擇——`SET key val IFEQ oldval` 這類原生 compare-and-set 指令，一條指令完成「值還是我讀到的那個才寫」，連 `WATCH`/`MULTI`/`EXEC` 三句的來回都省了（對單一字串的 CAS 而言最快）。但底層哲學沒變：先讀、後寫、寫時驗證「中間沒被人動過」。

樂觀鎖有一個非顯然、卻常被忽視的故障模式：**它在高衝突下會反過來咬你**。如果某個熱門資源（秒殺商品、全域配額）同時被幾百個請求搶改，那麼絕大多數請求都會在第 3 步發現 `version` 對不上、退回去重試，重試又再次撞車……CPU 和延遲全燒在「重試風暴」上，吞吐可能比老老實實排隊還慘。樂觀鎖的「樂觀」是有前提的——它賭的是衝突罕見。賭錯了，它就從最便宜的方案變成最貴的。

還有一個更隱蔽的陷阱：**重試必須冪等。** 樂觀鎖失敗後你會重跑整段邏輯，如果這段邏輯帶副作用——多發一封通知信、多打一次外部扣款 API——那麼每次重試都會把那個副作用再做一遍，你把一個「lost update」的 bug，換成了一個「重複執行」的 bug（冪等怎麼設計、冪等鍵的生命週期，見〈重複是常態：冪等與去重〉）。重試路徑上的副作用，是樂觀並發控制最容易踩穿的地板。

## 第三招：先鎖再改（悲觀）

如果衝突是常態，或者你的讀-改-寫過程中要做一些**沒法重跑**的事，賭「不會撞」就太冒險了。這時改用悲觀策略：動手之前，先把那一行鎖住，逼其他人排隊等你做完。

在 SQL 裡這通常是 `SELECT ... FOR UPDATE`：

```sql
BEGIN;
SELECT balance FROM accounts WHERE id = 42 FOR UPDATE;  -- 鎖住這一行
-- 在應用層做任意複雜的計算 ...
UPDATE accounts SET balance = ? WHERE id = 42;
COMMIT;  -- 鎖在這裡釋放
```

`FOR UPDATE` 在 `SELECT` 的當下就對選中的行上寫鎖，把「讀」和後面的「寫」用一把鎖**框成一個不可被插隊的整體**。其他交易想 `FOR UPDATE` 同一行，會停在那裡等，直到你 `COMMIT` 或 `ROLLBACK`。它解決了樂觀鎖的重試問題——沒有撞車、沒有重做、副作用只發生一次——代價是**等待**：在你持鎖的整段時間裡，所有想碰這行的人都被堵著，吞吐被你的臨界區長度直接限制。

悲觀鎖帶來兩個必須講透的故障模式。

第一個是**死結（deadlock）**。交易 A 鎖了行 1、想再鎖行 2；交易 B 鎖了行 2、想再鎖行 1。兩個互相等對方手上的鎖，永遠等不到。資料庫不會讓它們永遠卡著——PostgreSQL 有一個 `deadlock_timeout`（預設 **1 秒**），交易等鎖等超過這個時間，引擎就跑一次死結偵測，發現環就**主動中止其中一個交易**，回一個「deadlock detected」的錯。所以用悲觀鎖時，你的應用程式必須準備好接住這個錯並重試。一個實務上的防身術：讓所有交易**以固定順序**取鎖（永遠先鎖 id 小的那行），環就構不成，死結從根上消失（鎖、gap lock、next-key 與 wait-for graph 的完整機制見〈鎖與死結〉）。

第二個更陰險，叫 **write skew（寫偏斜）**，它專門騙過「我用了 snapshot isolation，應該很安全吧」的直覺。設想一條業務規則：值班醫生不能少於 1 人。兩位醫生 Alice 和 Bob 都在值班，兩人同時想請假。各自的交易都先 `SELECT count(*) FROM doctors WHERE on_call = true`——在 snapshot isolation 下，兩個交易讀的是**同一個快照**，都看到 count = 2，都通過了「≥ 1」的檢查；然後 Alice 的交易把自己設成不值班，Bob 的交易把自己設成不值班。兩個寫**改的是不同的行**，所以彼此不衝突、不互相阻塞、都順利提交。結果：值班人數變成 0，業務規則被破壞，而**整個過程沒有任何一次鎖衝突、沒有任何一筆寫被拒絕**。

write skew 之所以是 race condition 家族裡最難抓的一隻，正在於它的「讀」和「寫」**碰的不是同一批資料**：檢查讀的是一個跨行的聚合（總值班人數），動手改的是你自己那一行。snapshot isolation 只保證「你寫的行沒被別人同時寫」，但管不住「你讀來做判斷的那些行，在你提交前被別人改了、使你的判斷失效」。要堵它，要嘛升到 `SERIALIZABLE`——PostgreSQL 用 SSI（Serializable Snapshot Isolation）追蹤交易間的讀寫依賴，偵測到這種危險結構時中止其一、要你重試；要嘛在悲觀路線上**對你檢查所依賴的行顯式上鎖**（`SELECT ... FOR UPDATE` 把那些醫生的行鎖住，逼兩個請假交易序列化）。隔離級別、各種異常與 write skew 的完整光譜（見〈隔離級別與異常：dirty、phantom 到 write-skew〉）是另一條更長的線；這裡要記住的是：**「我開了交易、還用了 snapshot」絕不等於「我擋得住所有競態」**。預設隔離級別擋不住 lost update，snapshot isolation 擋不住 write skew，每一種保證都有它管不到的縫。

## 把競態縫進複雜邏輯：Redis 的 Lua

原子指令太窄、樂觀鎖怕重試、悲觀鎖怕死結——當你需要在快取層做一段**多步、帶條件分支**的更新，而且要它整段原子，Redis 還有一張底牌：Lua 腳本。

把整段邏輯寫成一個 Lua 腳本交給 `EVAL`，Redis 會把**整個腳本當成一個指令**來跑——因為指令執行是單執行緒的，腳本從第一行跑到最後一行的全程，**沒有任何其他 client 的指令能插進來**。這給了你 `WATCH`/`MULTI`/`EXEC` 給不了的東西：腳本裡可以有 `if`、可以根據前一個 `GET` 的結果決定下一步 `SET` 什麼，整段讀-判斷-寫渾然一體、不可分割。一個典型用途是「分散式限流」的滑動窗口：讀當前計數、判斷有沒有超過閾值、沒超就 +1 並回 OK、超了就回拒絕——這四步要嘛一起發生、要嘛一起不發生，Lua 腳本正好裝得下。

但這張底牌有一個與 Redis 單執行緒模型直接綁死的鋒利邊緣：**腳本跑的時候，整個 Redis 被你獨佔。** 既然指令執行單執行緒、而你的腳本算一個指令，那麼你的腳本跑多久，其他**所有** client 的**所有**請求就被卡多久。原子性的代價，是這段時間 Redis 對全世界停止服務。所以 Lua 腳本裡**絕對不能放慢操作**——不要在腳本裡 `KEYS *` 掃全庫、不要跑一個會遍歷百萬元素的迴圈。一個寫歪的 Lua 腳本能讓整台 Redis 凍住，連帶把所有依賴它的服務一起拖垮。它買到的「整段原子」和它索取的「獨佔全局」是同一枚硬幣的兩面：腳本必須短到接近瞬間完成，這個原子性才付得起。

## 一個算到底的例子：限流計數器為什麼會漏

把這一章的判斷壓到一個帶數字、跑得出結論的場景上。你要做「每個使用者每秒最多 100 個請求」的限流，10 個應用實例共用一個 Redis 計數器。某個使用者在某一秒被打進來 **1000** 個請求，平均散到 10 個實例上。

**天真版（應用層讀-改-寫）**：每個請求做三步——`GET rate:user:42` 讀回當前計數、在 Node 裡 `count + 1`、`SET rate:user:42 <new>` 寫回。這三步不是原子的，1000 個請求的這 1000 組「讀-改-寫」彼此瘋狂交錯。設想最壞的時序：幾十個請求在差不多的瞬間都 `GET` 到同一個值（比如都讀到 50），各自算出 51，再各自把 51 寫回去——幾十次本該累加的 `+1`，最後只在計數器上留下了一個 51。大量更新互相覆蓋（典型的 lost update），這一秒結束時計數器的最終值可能只爬到兩三百，遠低於真實的 1000。於是你的限流閾值 100 形同虛設：計數器以為才到 250，閘門一路放行，實際放進去的請求遠遠超過 100。**限流器在最需要它的高並發瞬間，恰恰失效。**

**原子版（Redis `INCR`）**：每個請求只做一件事——`INCR rate:user:42`。Redis 單執行緒把這 1000 個 `INCR` 排成一條隊，一個一個原子地執行，計數器精確地從 1 數到 1000，一次不漏。第 101 個 `INCR` 回傳 101，應用層一看超過 100，準確擋下；之後的請求全被擋。閾值 100 被嚴格執行，誤差為零。

同一個限流邏輯、同一份流量，差別只在「讀-改-寫留在應用層」還是「壓成資料層的一條原子指令」——前者在尖峰時把上限放大了三四倍，後者分毫不差。這就是這一章所有招數共同指向的那句話的具體重量：**競態的解法，幾乎都是設法讓那道「讀和寫之間的縫」不存在**，無論你是把它壓進一條指令、用版本號偵測它、用鎖框住它、還是用單執行緒的腳本把它整段吞下。

## 為什麼是這個形狀

退一步看，這一章所有看似零散的工具——原子 `UPDATE`、`INCR`、版本號、`SELECT FOR UPDATE`、`WATCH`、Lua 腳本、SSI——其實都在回答同一個問題的不同變體：**怎麼讓一段「讀-改-寫」在被讓出點切開的世界裡，依然表現得像不可分割。**

它們的分歧，全來自兩個維度的取捨。一是**衝突有多頻繁**：罕見就用樂觀（偵測 + 重試，零等待但怕重試風暴），頻繁就用悲觀（先鎖，序列化但怕死結與排隊）。二是**邏輯有多複雜**：簡單到能塞進一條指令就壓成原子指令（最快最窄），複雜到要多步條件就升級到腳本或顯式鎖（最廣最重）。沒有一個是「最佳實踐」可以無腦套用——選錯維度，樂觀鎖會在熱點上重試到崩潰，悲觀鎖會在低衝突場景白白排隊，原子指令會因為裝不下你的邏輯而留下縫。

而貫穿全章那個最該帶走的判斷，是關於「我已經安全了」的幻覺有多容易產生。單執行緒不安全——`await` 會切開臨界區。開了交易不安全——預設隔離級別擋不住 lost update。用了 snapshot 不安全——它擋不住 write skew。每一層都有一個聽起來像保證、其實有縫的承諾，而競態這種 bug 的殘忍之處，是它平時隱形、只在真實流量的某個時序縫隙裡現形一次，賣超一件庫存、放行一批超額請求、讓值班表上一個人都不剩。把每一個「應該沒事吧」翻譯成「這裡的讀和寫之間有沒有縫、誰能插進來、插進來會怎樣」——這個翻譯動作，就是原子性思維的全部。
