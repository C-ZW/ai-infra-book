# 快取策略：cache-aside、write-through、write-behind

一個讀多寫少的服務，某天資料庫的 CPU 開始貼著 90% 跑。慢查詢日誌裡沒有特別離譜的語句，只是同一批熱門查詢——首頁的商品列表、某個使用者的個人檔案——被重複打了千萬次。你做了所有工程師都會做的第一件事：在資料庫前面放一個 Redis，把熱資料快取起來。DB 的負載應聲掉到 20%，延遲從幾十毫秒降到亞毫秒。看起來是免費的勝利。

但你其實沒有「加一層快取」這麼簡單。你做的是一個更深的決定：**讓系統裡同一份資料，從此有兩個來源——一個快但可能過時，一個慢但是真相。** 從這一刻起，每一次寫入都要回答一個以前不存在的問題：快取裡那份副本，什麼時候、由誰、用什麼順序去更新或作廢？答錯了，使用者就會在某個你想不到的時刻讀到一筆早就被改掉的舊資料，而且因為快取「看起來在正常運作」，你很難察覺。

快取策略這個主題，表面上是在問「cache-aside、write-through 還是 write-behind」，骨子裡只在回答一個問題：**當真相有了兩份拷貝，你願意接受哪一種它們對不上的方式。** 沒有哪個策略消滅了這個裂縫——它們只是把裂縫挪到不同的地方、用不同的代價去掩蓋它。要選對，得先看清楚這道裂縫長在哪。

## 為什麼「兩份狀態」必然會分叉

把問題剝到第一原理。快取（cache）和底層儲存（資料庫）是兩個**獨立的系統**：它們有各自的記憶體、各自的崩潰時機、各自的寫入確認。任何一次寫入，只要它需要同時影響這兩個系統，就有一個無法消滅的事實——**這兩次更新不是同一個原子動作**。中間永遠存在一個窗口，在那個窗口裡，一個已經更新、另一個還沒。

這不是實作不夠仔細的問題，是分散式的本質。你寫快取、再寫 DB，中間行程可能崩潰；你寫 DB、再刪快取，中間網路可能抖一下讓刪除遲到；就算兩個寫都成功了，兩個並發的寫入者也可能以相反的順序抵達這兩個系統，讓快取停在 A 的值、DB 停在 B 的值。只要你拒絕為「快取＋DB」這對組合付出一個真正的分散式交易（兩階段提交，貴到沒人會為快取這麼做，見〈2PC 與分散式交易：原子提交與它的阻塞代價〉），這個窗口就不可能關掉。

所以「選快取策略」這句話的真實意思是：**你沒辦法讓裂縫消失，只能選它出現在讀路徑還是寫路徑、出現時系統是慢一點還是舊一點、出現後是自己癒合還是要你手動修。** 接下來三種策略，正是對這道題的三種不同回答。

## cache-aside：把填快取的責任交給應用程式

最常見、也該當預設的策略，是 cache-aside（旁路快取，又叫 look-aside）。它的核心立場很直白：**快取只是 DB 旁邊一份可丟棄的副本，DB 永遠是唯一的真相來源，而搬運資料這件事由應用程式自己負責。** 快取本身對 DB 一無所知，它就是個 key-value 盒子。

讀路徑是這樣走的：

```
read(key):
  v = cache.get(key)
  if v is not None:          # cache hit
      return v
  v = db.read(key)           # cache miss
  cache.set(key, v, ttl=T)   # back-fill，並設一個過期時間
  return v
```

第一次讀某個 key 一定是 miss，要付一趟 DB 的延遲，並順手把結果回填（back-fill）進快取；之後的讀就直接命中。這叫**惰性載入（lazy loading）**——只有真的被讀到的資料才會進快取，沒人讀的冷資料永遠不佔記憶體。那個 TTL 很關鍵：它是這套機制唯一的「保底癒合」手段。就算別的地方搞砸了讓快取卡在舊值，TTL 一到，這個 key 被清掉，下次讀就重新從 DB 拉到正確的值。**TTL 把「永久不一致」降級成「最多舊 T 秒」**，這是 cache-aside 能容忍各種小競態的底氣。

寫路徑才是真正分勝負的地方，而它有兩個看似都對、實則差很多的選擇。

### 寫入時：刪除，而不是更新

直覺上，寫入時最自然的做法是「更新 DB，順便把快取也更新成新值」——省掉一次未來的 miss，多好。但 cache-aside 的標準做法偏偏不是更新，而是**刪除（invalidate）**快取裡那一項，讓下一次讀去 DB 重新填。

```
write(key, new_value):
  db.write(key, new_value)   # 先寫真相來源
  cache.delete(key)          # 再作廢快取，不是 cache.set
```

為什麼寧可多付一次 miss 也要刪？因為「更新快取」會打開一個更糟的、會**永久毒化**快取的競態。設想兩個並發寫：寫入者 X 要把值改成 `v1`，寫入者 Y 要改成 `v2`。它們各自寫 DB、各自更新快取，但這是四個獨立的步驟，排程可以讓它們交錯成這樣——X 先寫 DB（DB=v1），Y 後寫 DB（DB=v2，Y 是最終勝者），然後 Y 先更新快取（cache=v2），X 才慢吞吞地更新快取（cache=v1）。結局是 **DB=v2、cache=v1，而且這個錯誤不會自己消失**，它會一直賴到 TTL 過期為止。你用「省一次 miss」換來了「快取與 DB 的最終值相反」。

刪除就乾淨得多：無論誰先誰後刪，刪完就是空的，下一次讀一定回 DB 拿到當前的真相再回填。**「更新」會在快取裡留下一個可能錯的值，「刪除」只留下一個空格——空格不會說謊。** 這就是 cache-aside 把寫路徑的一致性責任，收斂到「讀時回填」這條唯一路徑上的設計哲學：寫只負責清掉舊的，填永遠由讀來做，於是並發的填只會填出同一個（當前的）值。

### 順序：先寫 DB，再刪快取

第二個看似無關緊要、其實扛著正確性的細節是：**這兩步的順序不能反。** 必須先寫 DB、再刪快取，反過來「先刪快取、再寫 DB」會打開另一個窗口。

把「先刪快取、再寫 DB」放慢看：你先刪了快取（現在是空的），正準備寫 DB——就在這個瞬間，一個並發的讀請求進來，它 miss、去 DB 讀到了**還沒被你更新的舊值**，並把這個舊值回填進快取。然後你的 DB 寫才完成。現在 DB 是新值、快取是剛被那個讀回填的舊值，又一次卡到 TTL 過期。

先寫 DB、再刪快取則沒有這個問題：等你去刪快取時，DB 已經是新值了，任何在這之後 miss 的讀都會從 DB 讀到新值。Microsoft 在 Azure 的 cache-aside 文件裡把這個順序講得很死——「先更新資料儲存，再移除快取項；如果你先移除快取，會有一個小窗口讓客戶端讀到尚未更新的舊資料並把它填回去」。這不是建議，是這套機制能不能成立的前提。

### 連先寫 DB 再刪，都還有一條更細的縫

就算你嚴守「先寫 DB、再刪快取」，仍有一個機率極低、但在熱鍵上一天能命中幾次的競態，值得知道它存在，因為它揭示了「兩份狀態」的代價是降不到零的：

一個讀請求 A 進來，cache miss，它去 DB 讀到當前值 `v1`。**就在 A 拿到 `v1`、還沒回填快取的這個空檔裡**，一個寫請求把 DB 改成 `v2` 並刪掉了快取（此刻快取本來就是空的，刪了也沒事）。然後 A 才終於把它手上那個過時的 `v1` 回填進快取。結局又是 DB=v2、cache=v1，撐到 TTL。

注意這個競態的成因和前面不同：它要求 A 的「讀 DB」發生在寫之前、而 A 的「回填」發生在寫的刪除之後——讀的兩個動作被一次寫「夾」住了。它比前兩種罕見得多（需要回填恰好被延遲到刪除之後），但在每秒幾千次讀的超熱鍵上，機率乘以次數仍會時不時中一次。要進一步壓低，做法是回填時用 `SET key value NX`（只在 key 不存在時才寫，避免覆蓋掉別人新填的）配短 TTL，或在寫路徑用**延遲雙刪**（更 DB → 刪快取 → 等個幾百毫秒 → 再刪一次，把那個慢回填的舊值補刀掉）。但要說清楚：這些都只是**降低**機率，不是消滅。只要快取和 DB 是兩份狀態、寫不是一個原子動作，這條縫就在那裡。承認它、用 TTL 兜住它的最壞後果，比假裝它不存在要安全。

cache-aside 把裂縫放在哪？放在**寫之後的一小段時間窗口**，最壞後果是「讀到最多舊 T 秒的值」，而且它自己會癒合（TTL 到期或下次寫刪除）。對絕大多數讀多寫少、能容忍秒級舊讀的業務，這是最划算的一筆交易——簡單、爆炸半徑小、壞了也會自己好。所以它是預設。

## write-through：寫的時候就把快取一起寫對

如果你的場景偏偏不能容忍「寫完馬上讀卻讀到舊值」——使用者改了個人資料，下一個畫面立刻要顯示新的——cache-aside 那個「寫只刪、讀才填」的設計就有點尷尬：寫完快取是空的，緊接著的讀必然 miss、要繞一趟 DB。write-through（寫穿）換了個立場：**寫入時就同步地把快取和 DB 都寫成新值，兩個都成功才回應客戶端。**

```
write(key, new_value):
  cache.set(key, new_value)   # 同步寫快取
  db.write(key, new_value)    # 同步寫 DB
  # 兩個都成功，才回 OK
```

買到的保證很實在：**只要寫成功了，快取裡就是新鮮且命中的**，後續的讀不必繞 DB，讀延遲穩定地低。它特別適合「寫完極可能立刻被讀」的資料，省掉了 cache-aside 那次 miss 的回填往返。

代價也很直白，分兩層。第一層是**延遲**：每次寫都要等快取和 DB 兩段都完成，寫延遲變成兩者之和，哪一邊慢都會拖累寫吞吐。第二層更隱蔽，是個一致性陷阱：**write-through 並不保證快取和 DB 一致，它只保證「讀命中時新鮮」。** 因為「寫快取」和「寫 DB」這兩步本身又不是原子的——如果寫快取成功、寫 DB 卻失敗（或反過來），而你沒有一個讓兩段一起成功或一起回滾的機制，這兩份狀態就地分叉。很多人把 write-through 當成「一致性方案」，其實它把同一道「兩份狀態不原子」的裂縫，從 cache-aside 的讀後窗口，搬到了寫的兩段之間。要真的讓這兩段最終一致，得靠 outbox 那類「把兩個寫變成可追蹤、會重試直到對齊」的機制（見〈跨服務的交付一致：outbox 與 saga〉），而那又是另一筆複雜度的帳。

還有一個常被忽略的浪費：write-through 會把**每一筆寫過的資料都灌進快取**，包括那些寫完之後再也沒人讀的冷資料。它們白白佔著記憶體、把真正的熱資料擠出去。所以 write-through 經常和 cache-aside 的惰性載入搭配——寫穿保證熱寫資料新鮮，惰性載入保證冷資料不污染快取——而不是單獨用。

## write-behind：先認帳，再慢慢還

第三種策略把「快」推到極致。write-behind（回寫，又叫 write-back）的立場是：**寫入只更新快取就立刻回應客戶端，至於把資料刷回 DB，交給背景非同步去做。**

```
write(key, new_value):
  cache.set(key, new_value)   # 只寫快取
  mark_dirty(key)             # 標記為待刷回
  return OK                   # 立刻回應，不等 DB

# 背景 flusher 週期性地把 dirty 項批次刷回 DB
```

寫延遲被壓到最低——只是一次記憶體操作。但 write-behind 真正的威力不在單筆延遲，而在它解鎖了兩個 cache-aside 和 write-through 都做不到的優化：**合併（coalescing）與批次（batching）。**

想像一個計數器或一份頻繁變動的聚合資料，一秒內被改了 50 次。在 write-through 下，這是 50 次同步 DB 寫。在 write-behind 下，這 50 次只改了快取裡同一個 key，背景 flusher 在刷回時看到的只是它**最終的那一個值**——50 次寫被「合併」成 1 次 DB 寫。再加上批次：flusher 可以把這一秒內所有 dirty 的不同 key 攢成一筆事務一起寫回，把 N 次往返變成 1 次。對寫吞吐是瓶頸、且同一份資料被反覆改寫的場景（計數、metrics、排行榜、購物車這類可累積的聚合），write-behind 能把 DB 的寫負載砍掉一兩個數量級。

來把這筆帳算到底。設一個熱門計數器，峰值每秒被 `INCR` 1,000 次。write-through 路線下，這是每秒 1,000 次同步 DB 寫，每次假設 2 ms，這一個 key 就要吃掉 DB 大量的寫 IOPS、而且每個請求都得等那 2 ms。換成 write-behind：1,000 次 `INCR` 全打在快取的同一個記憶體計數器上（微秒級，請求立刻返回），背景 flusher 每秒醒來一次，看到的是這個 key **淨增了 1,000** 這一個事實，於是只發**一筆** DB 寫把計數器加 1,000。**每秒 1,000 次 DB 寫 → 每秒 1 次**，降到千分之一，而且使用者那端的寫延遲從 2 ms 掉到微秒級。這就是合併的數字。

但這份性能不是白拿的，它的代價尖銳到決定了 write-behind 能用在哪、不能用在哪：**在背景 flusher 把資料刷回 DB 之前，那些「已經回 OK 給客戶端」的寫，只活在快取的記憶體裡。如果快取在這個窗口崩潰，這些寫就永久消失了——而客戶端早已收到成功回應。** 你對使用者撒了一個善意的謊：「寫好了」，但那筆寫此刻還沒有任何持久的落腳處。所以 write-behind 鐵律般地只能用在「崩潰丟掉最近幾秒的寫是可以接受的」資料上——可重算的計數、可重建的聚合、丟一點也無傷大雅的 metrics。**金流、訂單、任何「丟了就是事故」的資料，絕不能走 write-behind。** 把回寫快取的易失記憶體當成持久儲存，是一個會在某次斷電後變成資料遺失報告的設計。

除了丟資料的窗口，write-behind 還欠下一串非顯然的債：刷回失敗了怎麼辦（DB 暫時掛了，dirty 資料越積越多，記憶體會不會先爆？）、刷回的順序要不要保證（如果同一個 key 的多次寫被合併還好，但跨 key 的依賴呢）、背景刷回的速率跟不上前台寫入時的背壓（見〈背壓：當下游跟不上〉）。這些都是「先認帳、再還錢」這個模式天生的併發症。write-behind 不是「更快的 write-through」，它是一個用「可控的資料遺失風險」去換吞吐的、完全不同的承諾。

## 三種策略，一張裂縫地圖

把三者並排，會發現它們其實是同一道「兩份狀態不原子」的裂縫，被放在了三個不同的位置：

| 策略 | 寫路徑做什麼 | 裂縫在哪、最壞後果 | 適合 |
|---|---|---|---|
| cache-aside | 寫 DB，刪快取 | 寫後一小段窗口讀到舊值；TTL 自癒 | 讀多寫少、可容忍秒級舊讀（預設首選） |
| write-through | 同步寫快取＋DB | 兩段寫之間若一邊失敗則分叉；寫延遲＝兩段之和 | 寫後立刻要讀到新值、要求讀己之寫 |
| write-behind | 只寫快取，背景批次刷 DB | 刷回前崩潰則永久丟最近的寫 | 高頻寫、資料可重算、能容忍丟最近幾秒 |

還有兩個邊角策略值得一句帶過。**read-through** 和 cache-aside 幾乎是同一回事，差別只在「miss 時去 DB 回填」這段邏輯收進了快取元件／client library 裡，應用程式只管對快取讀——代價是快取元件得懂 DB 的 schema，耦合往上移了一層。**write-around** 則是寫只進 DB、完全不碰快取，靠之後的讀 miss 自然回填，適合「寫一次之後很久才（或根本不）被讀」的資料，避免把一次性寫資料灌進快取浪費記憶體；它的代價是「寫完立刻讀」必然 miss、首讀延遲高。這些都不是新的第三條路，而是上面三種主軸在「回填責任放哪」「寫要不要碰快取」這兩個維度上的變奏。

真正要記住的判準是：**讀遠多於寫、容忍短暫舊值 → cache-aside；寫後立刻要讀到自己剛寫的 → write-through；寫吞吐是瓶頸且資料丟得起 → write-behind。** 而且這三者經常不是互斥的選擇，而是按 key、按資料類型混用——熱計數器走 write-behind，使用者檔案走 write-through，其餘一律 cache-aside。

順帶一個反向的判準，因為快取本身也是有成本的：如果讀寫比接近 1:1，或者資料總量小到 DB 自帶的 buffer pool 已經能全部駐留記憶體，那麼外加一層快取買到的命中率有限，卻憑空多出一個會失效、會分叉、會崩潰的狀態源。先確認「DB 真的扛不住、且讀真的遠多於寫」，再來談用哪種策略——不然你解的是一個還不存在的問題。

## 當快取本身不是一層：多層快取的甜頭與新債

到目前為止我們談的是「一層快取＋一個 DB」。實務上請求路徑常常串了好幾級快取，每一級的容量、延遲、命中率、一致性都不同：

```
client/CDN  →  in-process local cache  →  remote cache (Redis)  →  DB
  最遠最大            最近最小、最快           居中、跨節點共享        真相
                      最易與真相分叉                               最一致最慢
```

驅動這個結構的第一原理是：**離請求越近的層越快但越小、也越容易跟真相分叉；越遠的層越大越一致但越慢。** 在應用程式行程內放一個 in-process 的 LRU map（local cache），命中時是純記憶體存取、連網路往返都省了，比打 Redis 還快一個數量級；但它每台機器各有一份、互不相通。把熱資料用 local cache 吃掉大部分讀，再用遠端 Redis 接住 local miss，最後才落到 DB。

這個甜頭可以算出來。設請求路徑 local → Redis → DB，local 命中率 80%、Redis 對 local-miss 的命中率 90%、其餘才打 DB。每 1,000 個請求：local 命中 800，剩 200 進 Redis，Redis 命中 180，最後只有 **20 個打到 DB——2%**。若各層延遲是 local 0.01 ms、Redis 0.5 ms、DB 10 ms，平均讀延遲 ≈ 0.8×0.01 + 0.18×0.5 + 0.02×10 ≈ **0.3 ms**，相較全打 DB 的 10 ms 降了約 97%，而 DB 的讀負載只剩 1/50。

但多層快取欠的債，比單層更難纏，而且最經典的那個 bug 幾乎人人踩過：**失效傳播不完整。** 一次寫入，現在要讓**每一層**都作廢。你很自然地更新了 DB、刪了 Redis——然後忘了那些散在各台機器行程內的 local cache。結果是：刪過 Redis 的那台機器沒事，但其他每一台還守著自己 local cache 裡的舊值，會持續回舊資料，直到各自的 local TTL 到期。更糟的是這個錯誤**只發生在部分機器上**——同一個請求打到不同的後端實例會得到不同的答案，難重現、難 debug，因為它不是「壞了」而是「有些機器壞了」。

要傳播 local 層的失效，常見做法是用一個 pub/sub 通道廣播失效事件：寫入時除了刪 Redis，還往一個 channel `PUBLISH` 一條「key X 失效了」，每台機器訂閱它、收到就清掉自己的 local cache。但這裡藏著一個語意陷阱——Redis Pub/Sub 是 fire-and-forget、at-most-once 的：訊息只送給「發佈當下正連線」的訂閱者，某台機器若恰好在重連、GC 暫停、或網路抖動，就會**永久錯過**這條失效訊息，於是它的 local cache 殘留舊值直到 TTL（Pub/Sub 的這個交付特性見〈Redis Pub/Sub 與 Streams：交付語意的取捨〉）。所以 local 層的 TTL 不能設太長——它是漏掉的失效訊息的最後保底。多一層快取，就多一處要失效、多一種不一致來源、多一個故障點；命中率的甜頭不是免費的，層數本身就是成本。

## 快取掛了，你退到哪裡

多層快取還逼出最後一個、也最致命的問題：**當某一層（尤其是大家共用的遠端 Redis）整個掛掉，請求路徑要能退到下一層，而不是整條鏈一起倒。**

這裡有個違反直覺的危險。直覺會說「快取掛了？沒關係，退回去直接打 DB 就好，本來就是這樣設計的」。但想清楚那一瞬間發生了什麼：平常 Redis 替 DB 擋下了 98% 的讀，DB 只在處理那 2%。Redis 一掛，原本被擋住的**全部讀流量在同一個瞬間直撲 DB**——那是平常的 50 倍負載砸下來。DB 撐不住、開始變慢甚至倒下，於是你把「快取故障」升級成了「資料庫故障」，一個本來只該掉一點命中率的事件，演變成全站不可用。這正是「快取雪崩」的一種觸發路徑（雪崩本身與它的全套防禦見〈快取的失效災難：雪崩、穿透、擊穿〉）。

所以快取層的「退到 DB」這條降級路徑，不能是裸退——必須在退路上同時對 DB 套保護：限流（只放 DB 扛得住的那部分讀過去）、single-flight（同一個 key 同時只讓一個請求去 DB 重建、其餘等它的結果）、或保留一層 local cache 在遠端掛掉時兜住最熱的那批資料（回的值可能更舊，但好過直接 503）。換句話說，**「快取掛了退 DB」這句話只有配上「並且保護好 DB」才成立**，否則它只是把單點故障從快取搬到了 DB。

值得把這裡的降級和更廣義的服務降級分清楚：本章談的是「快取這一層自己掛了，請求在快取鏈裡怎麼往下退」這件局部的事。服務之間呼叫的整體降級——壞掉時退到一個預設值、退到一個簡化的回應、或乾脆熔斷不再嘗試——是一個更大的韌性主題，由它的 owning 章去講透（見〈fallback 與降級：壞掉時退到哪〉）；觸發降級的熔斷機制本身也另有專章（見〈circuit breaker：三態與半開試探〉）。本章只負責把「快取層的退路」這一段講清楚：退路必須存在，而且退路盡頭的 DB 必須有人護著。

## 為什麼是這個形狀

退一步看，三種快取策略、多層結構、降級路徑——它們全部是從同一個無法迴避的事實裡長出來的：**你為了快，給同一份資料造了第二個來源，而這兩個來源不可能被一個原子動作同時更新。**

正因如此，才有「先寫哪個、後寫哪個、刪還是更新」這些看似吹毛求疵、實則決定正確性的順序問題——它們都是在管理那個「一個已更新、另一個還沒」的窗口。正因如此，cache-aside 選擇用 TTL 讓裂縫自癒、把責任收斂到讀路徑的回填；write-through 選擇用同步雙寫換讀的新鮮、把裂縫推到寫的兩段之間；write-behind 選擇用「先認帳」換極致吞吐、把裂縫變成一個「崩潰就丟」的明碼標價的賭注。正因如此，多層快取在每加一層、買到一截延遲的同時，都欠下一處新的失效傳播；正因如此，快取一掛不能裸退，因為那第二個來源平常正替真相來源擋著絕大部分的火力。

快取從來不是「把資料放快一點」這麼無辜的動作。它是一個鄭重的取捨：**你拿一致性的一小塊裂縫，去換吞吐與延遲的一大塊收益。** 選對策略，就是選一個你的業務真正付得起的裂縫——讀多寫少容忍舊值，就讓它自癒（cache-aside）；要讀己之寫，就同步寫對（write-through）；資料丟得起而吞吐要命，就先認帳慢慢還（write-behind）。沒有一種選擇能讓裂縫消失。能消失的那種叫「不用快取」，而你之所以走到這裡，正是因為那條路你已經走不通了。
