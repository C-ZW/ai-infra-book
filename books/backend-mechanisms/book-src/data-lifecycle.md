# 資料的生命週期：retention、TTL、軟刪除與合規刪除

法務轉來一封信，內容兩行：某位歐盟使用者依 GDPR 第 17 條行使「被遺忘權」，要求你把他的個人資料**從你的系統裡刪掉**，限期一個月內回覆已完成。你打開資料庫，敲下那行你寫過一千遍的指令：

```sql
DELETE FROM users WHERE id = 'u_8842';
```

回車。`DELETE 1`。一筆刪掉了。問題是——你真的刪掉了嗎？

那一筆 `users` 列確實從主庫的可見結果裡消失了。但這位使用者的資料不只住在這一張表。它在兩台 read replica 上各有一份正等著複製延遲追平；它在 Redis 快取裡以 `user:u_8842` 為鍵躺著，TTL 還有四十分鐘；它被攤平進了 Elasticsearch 的搜尋索引；昨晚的全量備份已經把它封進了一個保留三十天的快照；資料倉裡的 analytics 表抄了一份去算月活；而過去三個月的應用日誌，因為某行 `logger.info(user)` 順手把整個 user 物件序列化了進去，到處夾帶著他的 email 和手機號。你的 `DELETE 1` 動到的是這座冰山露出水面的那一角。

這一章談的就是這座冰山：一筆資料從寫入到真正消失之間，會經過哪些「結束」的形態，每一種結束在底層到底發生了什麼，以及為什麼「刪除」是後端裡最容易被低估、又最容易出事的一個動作。

## 「刪掉」這個詞底下藏了四件不同的事

工程師嘴上的「刪掉」，其實混用了四種語意上完全不同的操作。把它們分清楚，是這整章的地基。

第一種是**過期回收**。你不在乎這筆資料的「消失」對誰負責，你只在乎它別把磁碟塞爆。session、快取、原始日誌、監控指標——這些資料本質短命，活過它的有用窗口就是純粹的負擔。對這類資料你要的是一個自動化的清道夫：設一個存活時長（TTL），時間到了，引擎自己把它收走。

第二種是**封存**。資料還有價值，只是不再「熱」——一筆三年前的訂單，法規要求你留七年，但沒人會在毫秒級的交易路徑上查它。你不刪它，你把它從昂貴的熱儲存搬到便宜的冷儲存（物件儲存的低頻層，見〈物件儲存：S3 與大 payload 的交接〉）。它還在，只是搬了家、降了級、慢了。

第三種是**軟刪除**。使用者按下「刪除這篇草稿」，你希望這個動作**可以反悔**：三十天內他點「復原」，東西要回得來；同時你的審計需要知道「這筆曾經存在、誰在何時刪的」。於是你不真的刪——你寫一個標記 `deleted_at = now()`，讓查詢層假裝它不在。資料一個位元組都沒少，只是被一層過濾條件藏了起來。

第四種是**合規刪除**，也叫硬刪除。這才是法務那封信要的東西：把這筆資料**真正地、不可復原地、從每一個副本裡**抹掉，包括備份、快取、衍生索引、日誌。它的對象不是「一張表」，而是「這筆資料在你整個系統裡的所有影子」。

這四件事被同一個動詞「刪除」綁架，是這個領域所有災難的根源。最致命的一條是：**軟刪除不是合規刪除**。一筆 `deleted_at IS NOT NULL` 的列，對使用者「看起來刪了」，但它的個資原封不動躺在磁碟上、躺在每一份備份裡。如果你拿軟刪除去回應 GDPR 第 17 條，你回覆了「已完成」，而你其實一筆資料都沒抹掉——這不是技術瑕疵，是把「假裝刪了」當成「刪了」上報給監理機關。

## TTL：把回收交給引擎，但別以為「過期」就是「不在了」

先看最溫和、也最容易被誤解的一種：TTL（time-to-live）。

它的承諾很簡單：你給每一筆資料標一個到期時間，引擎在到期後**自動**把它清掉，你一行清理程式都不用寫，也不用養一個定時 job 去掃。Redis 的 `EXPIRE`、DynamoDB 的 TTL 屬性、時序資料庫的 retention policy，全是這個模式。聽起來像是把問題外包給引擎，乾淨俐落。

但這裡藏著一個幾乎所有人第一次都會踩的陷阱：**「過期」和「被刪掉」是兩個不同的時刻，中間有一段你無法忽略的窗口。**

為什麼不能在到期那一秒立刻刪？因為「立刻」太貴了。設想 Redis 裡有上千萬個 key，每個都帶不同的到期時間。要做到「到期那一刻精準刪除」，引擎得維護一個按到期時間排序的優先佇列、不斷檢查隊首——這會把寶貴的單執行緒 CPU（Redis 指令執行為單執行緒，見〈Redis 的內部〉）耗在掃描上。所以 Redis 選了一條便宜得多的路，由兩個機制合起來逼近「大致準時」：

- **惰性過期（lazy）**：只有當你**去碰**一個 key 時，Redis 才順手檢查它過期了沒。過期了就當場刪掉、回你一個「不存在」。沒人碰的過期 key，可以在記憶體裡賴著不走。
- **主動過期（active）**：一個背景循環，預設**每秒跑十次**。每次從帶 TTL 的 key 裡**隨機抽 20 個**取樣，刪掉其中過期的；如果這 20 個裡超過 **25%** 已過期，它判斷「過期密度很高」，立刻再抽一輪，直到取樣的過期比例降到門檻以下才歇手。

把這兩個機制疊起來看，一個非顯然的後果浮現了：**一個已經過期、但剛好沒人去碰、又一直沒被隨機取樣抽中的 key，會在記憶體裡多活一陣子。** 它在邏輯上「不存在」（你 `GET` 它會得到 nil），但它在物理上還佔著記憶體。對快取場景這完全無所謂——你要的就是「邏輯上消失」。但只要你動了「拿 TTL 當刪除手段」的念頭，這個窗口就會反咬你。

DynamoDB 把這個窗口擺得更明白。它的官方文件白紙黑字寫著：到期項目會在到期時間之後「**within a few days**（幾天之內）」被刪除，是 best-effort、不消耗寫入吞吐的背景流程。更要命的是中間那段：**到期但尚未被刪除的項目，仍然會出現在你的 `Scan`、`Query`、`GetItem` 結果裡**。也就是說，如果你的應用直接信任「設了 TTL 的東西到期就讀不到」，你會讀到本該消失的資料——除非你在每個查詢上都掛一個 `filter expression` 把過期項目濾掉。DynamoDB 的文件甚至直接點名這個用途：「如果你的應用要求項目在到期後**立刻**不可用（例如 GDPR 的被遺忘權），TTL 本身不足以滿足。」

這就把 TTL 的定位釘死了：**它是給「可以晚一點、可以不精準」的回收用的清道夫，不是給「必須準時、必須真的不在」的合規刪除用的手術刀。** 把短命快取、過期 session 交給它，完美；把「必須在某一刻徹底消失的個資」交給它，是把法律義務押在一個 best-effort 的背景循環上。

還有一個容易被忽略的 TTL 副作用：在 DynamoDB 裡，TTL 觸發的刪除會**流進 DynamoDB Streams**，被標記為 `service deletion` 而非使用者刪除，並且會從本地與全域次級索引一併移除。這意味著你可以掛一個下游 consumer，在資料到期被刪的那一刻收到事件、去清理它在別處的副本——TTL 不只是回收，它還能當一條「資料死亡」的通知源。

## 按時間分區輪替：把「刪一億列」變成「砍掉一塊」

TTL 解決的是「逐筆資料各有各的壽命」。但有一大類資料的壽命是**齊頭**的：日誌、事件、監控指標、稽核軌跡——它們按時間源源不斷地進來，政策也按時間走，「只留最近 90 天」。對這類資料，逐筆過期是個糟糕的工具，因為刪除本身就會變成一場災難。

設想一張存了三億列日誌的表，你要刪掉 90 天前的一億列：

```sql
DELETE FROM logs WHERE created_at < now() - interval '90 days';
```

這一行看著無辜，跑起來是場噩夢。在 PostgreSQL 這種 MVCC 引擎裡（見〈MVCC：讓讀不擋寫〉），`DELETE` 並不會立刻把那一億列的空間還給作業系統——它只是把每一列標記成「對之後的交易不可見」的死元組（dead tuple）。真正回收空間要等 `VACUUM` 背景跑過。於是你這一刀下去，製造了一億個死元組，撐爆了 WAL，讓 autovacuum 追著你喘，表還膨脹（bloat）了一大塊——刪完之後磁碟用量可能不降反升，要等 vacuum 慢慢追平。在 LSM 引擎上（見〈LSM-tree 與 B-tree〉）更糟，後面會講到。

成熟的做法是釜底抽薪：**從一開始就按時間把表切成分區（partition）**，每天（或每月）一個分區。資料寫進來，依 `created_at` 自動落到對應的那塊。當某一天的資料整段過期，你不去逐列刪它，你直接把那一整塊卸下來：

```sql
ALTER TABLE logs DETACH PARTITION logs_2026_03_14;   -- 先卸下，可選擇先封存
DROP TABLE logs_2026_03_14;                            -- 或直接砍掉整塊
```

`DROP` 掉一個分區是一個近乎 O(1) 的中繼資料操作——它不掃描、不產生死元組、不需要 vacuum。一億列的清除從「跑半小時、把資料庫拖垮」變成「一瞬間，砍掉一個檔案」。PostgreSQL 14 起還能 `DETACH PARTITION CONCURRENTLY`，只需要父表的 `SHARE UPDATE EXCLUSIVE` 鎖、不會把寫入擋在門外。`DETACH` 與 `DROP` 的差別正好對上前面的語意分類：`DETACH` 是把這塊資料移出主表但留著（拿去封存到冷層），`DROP` 是直接抹掉。

這就是為什麼凡是「資料按時間進、按時間退」的系統，retention 幾乎總是建在分區輪替上、而不是建在 `DELETE` 上。代價是你得在建表那一刻就決定按時間分區、選好分區粒度——粒度太粗（一年一個分區）會讓「砍掉」的最小單位太大、retention 不精細；太細（一小時一個分區）會讓分區數量爆炸、planner 變慢。這個決定很難事後追加，所以它屬於「設計時就要想清楚」的那一類。

## 軟刪除：用一個標記換來可復原與審計，欠下兩筆隱形的債

現在換一個完全不同的需求。使用者刪的不是日誌，是他自己的東西——一篇文章、一個專案、一筆訂單——而你希望這個刪除**可逆**。三十天垃圾桶、誤刪復原、「這筆是誰在何時刪的」的審計，這些需求把你推向軟刪除。

軟刪除的機制簡單到幾乎不像個機制：加一欄 `deleted_at TIMESTAMP NULL`，刪除時不 `DELETE` 而是 `UPDATE ... SET deleted_at = now()`，然後在**每一個查詢**裡加上 `WHERE deleted_at IS NULL`。資料還在，只是被這個過濾條件藏起來了。

正因為它簡單，它欠下的兩筆債特別隱蔽。

**第一筆債：那個 `WHERE deleted_at IS NULL` 必須出現在每一個查詢裡，漏一處就是一次資料外洩。** 你的列表 API 記得加了，但那個三個月後新寫的報表查詢忘了；那個直接跑在資料庫上的 BI dashboard 沒人想到要過濾；那個 join 了五張表的複雜查詢，只在外層加了過濾、漏了某張中間表。每一個遺漏，都是把一筆「使用者以為刪掉了」的資料端到了某個人眼前。這種債的可怕之處在於它不會報錯——查詢跑得好好的，只是多吐了它不該吐的列。對付它的辦法是把過濾下沉到一個沒人能繞過的地方：資料庫視圖（只暴露 `deleted_at IS NULL` 的視圖、應用只准查視圖）、ORM 的全域 scope、或 row-level security。靠「每個人寫每個查詢時都記得加」是註定要漏的。

**第二筆債更刁鑽，是軟刪除和唯一約束的衝突。** 假設 `users.email` 上有唯一索引。某使用者用 `alice@example.com` 註冊，後來軟刪了帳號（`deleted_at` 被設上）——那一列還躺在表裡，email 欄還是 `alice@example.com`。現在 Alice 想用同一個 email 重新註冊，你的 `INSERT` 撞上唯一索引：`duplicate key value violates unique constraint`。從使用者視角，她的帳號明明「刪掉了」，卻被一個看不見的幽靈擋住了重註冊。

天真的修法是把唯一約束改成 `UNIQUE(email, deleted_at)`——讓「同 email、不同 deleted_at」可以共存。但這在 PostgreSQL 上有個致命陷阱：**`NULL` 在唯一索引裡彼此不相等。** 兩個 active 使用者的 `deleted_at` 都是 `NULL`，而 `(alice@example.com, NULL)` 和 `(alice@example.com, NULL)` 在唯一索引眼中**不算重複**——於是你反而允許了兩個 active 的同 email 帳號，把唯一性整個破壞掉。正解是用**部分唯一索引**（partial unique index），只對 active 列強制唯一：

```sql
CREATE UNIQUE INDEX users_email_active_uniq
  ON users (email)
  WHERE deleted_at IS NULL;
```

這條索引只覆蓋 `deleted_at IS NULL` 的列：兩個 active 同 email 會撞、被擋；而軟刪掉的那筆因為 `deleted_at` 非 NULL，根本不在索引裡，不擋任何人重註冊。一行索引定義，把「可復原」和「唯一性」這對看似矛盾的需求調和了。（PostgreSQL 15 起另有一條路：唯一約束可加 `NULLS NOT DISTINCT`，讓多個 NULL 被視為相等，於是 `UNIQUE NULLS NOT DISTINCT (email, deleted_at)` 也能擋住兩個 active 同 email。但部分唯一索引語意更清楚、不替軟刪除的列建索引省空間、也適用於更舊版本，這裡仍以它為主解。）這個 edge case 之所以值得講透，是因為它幾乎不會在開發階段暴露——只有當第一個真實使用者刪了帳號又想回來時才會炸，而那時你的軟刪除策略已經鋪滿整個 schema。

## LSM 引擎的刪除：你寫的「刪除」其實是一次寫入

前面說 PostgreSQL 的 `DELETE` 會留下死元組、要等 vacuum。在 LSM-tree 引擎上（Cassandra、RocksDB、ScyllaDB，見〈LSM-tree 與 B-tree〉），刪除這件事的反直覺程度還要再上一層——而且它直接決定了你能不能在這類系統上做時序資料的 retention。

LSM 的根本特性是**寫入永不就地修改**：所有寫都是往一個不可變的有序檔（SSTable）追加。那麼「刪除一筆資料」怎麼辦？你沒辦法回去把舊 SSTable 裡那筆改掉——它是不可變的。於是 LSM 用了一個漂亮又危險的招數：**刪除也是一次寫入。** 它寫一個帶時間戳的「墓碑」（tombstone）標記，蓋在那個鍵上方。讀取時，引擎從新到舊掃過各層 SSTable，一旦先碰到墓碑，就知道「這個鍵已被刪」，回你一個「不存在」——即使底下舊 SSTable 裡那筆原始資料還好端端躺著。

這帶來三個層層遞進的後果。

**第一，刪除不釋放空間，反而先佔用空間。** 直覺上刪資料該騰出磁碟，但在這裡刪除等於又寫了一筆（墓碑）進去。原始資料和墓碑要一起等到背景 compaction 把它們合併、確認墓碑蓋掉了所有更舊的版本後，才一起被丟棄、空間才回收。

**第二，墓碑會拖慢讀取。** 設想一個時序場景：你對某個 partition 大量寫入又大量按 TTL 過期，累積了成千上萬個墓碑。現在一個讀取要掃過這個 partition——它得跳過所有這些墓碑才能拼出當前的真實狀態。墓碑堆積會讓本該很快的讀取慢成爬行，這是時序資料庫最經典的效能陷阱之一。Cassandra 甚至設了一個「單次查詢掃過的墓碑數」上限，超過就直接讓查詢失敗，逼你正視這個問題。

**第三，也是最微妙的：墓碑不能太早刪，否則「死人會復活」。** Cassandra 給每個墓碑一個寬限期 `gc_grace_seconds`，**預設 864000 秒，整整十天**。compaction 即使遇到一個已經可以丟棄的墓碑，也要等過了這十天才真的把它和被它蓋掉的資料一起清走。為什麼要等？因為 Cassandra 是多副本、最終一致的——刪除是靠墓碑**傳播**到各副本來生效的。如果某個節點在你下刪除指令時剛好離線，它身上那筆舊資料還在、也還沒收到墓碑。要是墓碑在這個節點復原前就被別的節點清掉了，那麼當這個落後的節點重新上線、跑 anti-entropy repair（見〈gossip 與 anti-entropy〉）時，它會把自己那筆「還活著」的舊資料當成新資訊，**反過來複製給其他節點**——一筆你明明刪過的資料就這樣詐屍復活了。這種還魂的記錄有個專門的名字：zombie（殭屍）。`gc_grace_seconds` 那十天，就是留給「掉隊的節點回來、把墓碑也收下」的安全餘量。

把這三點合起來，一個對運維極其重要的結論浮現了：**在 LSM 引擎上，「大量刪除」和「按 TTL 大量過期」在機制上是同一件事——它們都在製造海量墓碑。** 這就是為什麼在 Cassandra 上做時序 retention，正確姿勢不是讓幾百萬列各自 TTL 過期（那會生出幾百萬個墓碑、把讀取拖垮），而是和關聯式庫一樣走分區/整表輪替——直接 `DROP` 掉整張舊表，繞開墓碑機制。刪除的形狀，徹底被底層引擎的結構決定了。

## 合規刪除：一個 DELETE 動作，一條覆蓋全副本的管線

現在回到開場那封信。法務要的不是過期回收、不是封存、不是軟刪除——是把一個人的資料從你整個系統裡**真正抹掉**。這是本章最難的一種「刪除」，難的不是技術，是**範圍**。

先把法律的形狀講清楚，因為它決定了工程的邊界。GDPR 第 17 條給了使用者「被遺忘權」，但它不是無條件的。能觸發刪除義務的情形包括：資料對當初蒐集的目的已不再必要、使用者撤回了當初的同意且無其他合法依據、使用者依第 21 條反對處理且你無壓倒性正當理由、資料被非法處理等。反過來，第 17 條第 3 項也給了你**拒絕刪除的正當理由**：當保留資料是為了履行法律義務（例如稅務、財務法規要求你保存交易紀錄若干年）、為了行使或防禦法律請求、為了言論自由、或為了公共利益的研究與統計時，你可以、有時甚至**必須**留著。這就帶出合規刪除的第一個非顯然點：**「全部刪掉」往往是錯的。** 一個收到刪除請求的訂單系統，可能必須保留訂單的金額與時間（稅法要求），但抹掉其中的個人識別資訊（姓名、地址、email），讓剩下的資料**無法再合理地連回任何個人**——這叫**匿名化**（anonymization），在法律允許時是比物理刪除更務實的解：真正匿名化後的資料不再構成個資，已脫離 GDPR 的管轄。要注意它和**假名化**（pseudonymization）不是一回事：假名化只是把識別欄位換成代號、把對照表另存他處，憑那份額外資訊還能回推到本人，所以假名化資料**仍然是個資、仍受 GDPR 規範、仍須在被要求時刪除**。把假名化當成「替代刪除」用，正是本章開頭那個陷阱的變體——看起來抹掉了，其實只要拿出對照表就還在。

講清楚邊界之後，看工程上真正的難題：**一筆資料的副本，遠不只主庫那一份。** 把一筆使用者資料在一個典型系統裡的分身數一遍：

| 副本所在 | 份數 | 抹除難度 |
|---|---|---|
| 主庫 | 1 | `DELETE` 一行，簡單 |
| read replica | 2–3 | 隨主庫複製自動消失，但有複製延遲窗口 |
| Redis 快取 | 1 | 主動 `DEL`，否則等 TTL（不可靠） |
| 搜尋索引（Elasticsearch） | 1 | 要單獨發刪除請求、近即時生效 |
| 資料倉／analytics | 1+ | 抄過去的，要單獨清，常被遺忘 |
| 每日備份（保留 30 天） | 30 | **最難**：備份是不可變快照，沒法逐筆改 |
| 應用日誌裡夾帶的 PII | 不定 | 散落、無結構，最容易漏 |

收到**一筆**刪除請求，「真正刪掉」要協調的是這 **36 份以上**的副本。漏掉任何一處——尤其是那份你壓根沒想到搜尋索引也抄了一份、或那行 `logger.info` 順手記下的 email——就是「名義上合規、實際上仍在外洩」。而 GDPR 對被遺忘權這類「資料主體權利」的違規，罰則落在最高一檔：**€2000 萬，或全球年營業額的 4%，取較高者**（2026-06 查證）。一行 `DELETE FROM users WHERE id=?` 顯然扛不起這個爆炸半徑。

所以合規刪除的正確形狀，不是一個動作，而是**一條管線**：

```
收到刪除請求
  → 主庫硬刪 / 匿名化（依法律是抹除還是去識別化）
  → 連鎖清理所有衍生副本：搜尋索引、快取、analytics、訊息佇列裡在途的訊息
  → 處理備份：標記該主體「beyond use」（封存、不再使用、輪替到期時隨備份一起消失）
  → 留下「已於某時完成刪除」的稽核憑證 —— 但憑證本身不得含個資
```

這裡每一步都有自己的陷阱。備份那一步尤其反直覺：你**沒辦法**回去把一個三十天前的不可變快照裡的某一筆改掉——那等於偽造歷史備份、且技術上往往不可行。監理實務上接受的做法是把該主體標記為「beyond use」（鎖起來、保證絕不再用於任何處理），並依賴備份的自然輪替——三十天後那份備份過期銷毀時，裡頭的影子一併消失。這就是為什麼**備份保留期越長，你的合規刪除完成時間就被拉得越久**：你最多只能承諾「在最後一份含此資料的備份輪替到期後」徹底完成。

還有一個收尾的悖論值得點出：**你需要記錄「我刪了某人的資料」這件事本身，但這條記錄不能含被刪者的個資。** 如果你的刪除稽核軌跡裡寫著「於 2026-06-22 刪除 alice@example.com 的全部資料」，那你又把 email 留了下來——你在記錄「我刪了它」的同時沒刪掉它。正解是用一個與本人無法反推的代號（pseudonymous id）來記，證明「某個請求被處理了」，而不洩漏「是誰」。

## 為什麼資料的死法值得當成設計問題

退一步看，這四種「刪除」之所以各長一個樣，是因為它們在回答四個完全不同的問題：

- **TTL／過期**回答「這東西誰都不負責、別讓它塞爆磁碟就好」——所以它可以惰性、可以不精準、可以 best-effort。
- **分區輪替**回答「這一整段時間的資料齊頭過期」——所以它把刪除降維成砍掉一塊檔案，繞開逐筆刪除的所有代價。
- **軟刪除**回答「這個動作要可逆、要可審計」——所以它根本不刪，只藏，代價是欠下「處處要過濾」和「唯一約束打架」兩筆債。
- **合規刪除**回答「法律要求這個人從我系統裡徹底消失」——所以它的對象不是一張表，而是所有副本，是一條管線而非一個動作。

把這四者混為一談，就是這個領域所有事故的共同根源：拿 TTL 當合規刪除（押在 best-effort 上）、拿軟刪除冒充硬刪除（資料根本沒走）、在 LSM 上逐筆刪除製造墓碑風暴、或是 `DELETE` 完主庫就以為結案、忘了那座副本的冰山。

更底層的那條設計原則是：**資料既是資產，也是負債。** 你留的每一筆資料，都同時是一份可能有用的紀錄，和一份你必須保護、必須能在被要求時抹除、出事時會擴大外洩面的責任。「先全留著以防萬一」不是中性的——它在悄悄擴大你的攻擊面和合規負擔。一個成熟系統不是「能存多少存多少」，而是在寫下每一類資料時就已經想好了它該怎麼活、活多久、老了去哪、被要求消失時怎麼真的消失。資料的生命週期，和它的寫入路徑一樣，是設計的一部分，不是事後才補的清潔工作。
