# 阻塞與非阻塞 I/O

一個剛上線的聊天服務，壓力測試跑到大約一萬個同時連線時，記憶體曲線開始陡峭地爬升，接著行程被作業系統的 OOM killer 砍掉。看 CPU，使用率不到三成；看網路，吞吐遠未飽和。資源明明都還有，服務卻撐不住——它不是被「算不完」壓垮，而是被「等不完」壓垮的。

這一章要拆的，就是這個「等」字。一個後端行程花在 I/O 上的時間——等資料庫回一行、等外部 API 回一包、等 socket 把下一個位元組送來——往往遠多於它花在計算上的時間。問題不在等本身（等是 I/O 的天性），而在**等的時候，發起這次等待的執行緒在做什麼**。答案有兩種，整章的分水嶺就在這裡：要嘛這條執行緒被掛起、原地枯等，什麼別的事都做不了；要嘛它登記一聲「好了叫我」，立刻回頭去服務別人。前者叫阻塞，後者叫非阻塞。聽起來只是一個 API 旗標的差別，但它決定了你那台機器到底能不能用一條執行緒照看上萬個連線——也決定了開場那條記憶體曲線會不會把你的服務拖死。

## 一次 `read` 到底發生了什麼

先把最底層那一步看清楚，因為所有討論都從這裡長出來。你呼叫 `read(fd, buf, n)`，想從一個 socket 讀進來一些位元組。此刻 kernel 那邊的接收緩衝區可能有資料，也可能還空著（對端還沒送、或封包還在路上）。資料還沒到，這個呼叫該怎麼辦？

預設情況下，這個 fd 是**阻塞模式**的，kernel 的處理是：把你這條執行緒**從可執行佇列裡拿掉、標記為睡眠**，直到接收緩衝區裡有了至少一個位元組，才把它喚醒、讓 `read` 帶著資料返回。在這整段等待期間，這條執行緒不消耗 CPU（這點常被誤會——阻塞不是 busy-wait、不是空轉燒 CPU），但它也**完全不能做任何別的事**。它佔著一條執行緒的全部身價——那塊 stack、那個排程器名額——只為了等一件還沒發生的事。

如果你把這個 fd 設成**非阻塞模式**（在 Linux 上對 fd 設 `O_NONBLOCK` 旗標），同一個 `read` 的劇本完全變了。資料還沒到時，kernel **不掛起你**，而是讓 `read` 立刻返回 `-1`，並把 `errno` 設成 `EAGAIN`（在 Linux 上 `EWOULDBLOCK` 與 `EAGAIN` 同值，但 POSIX 並不保證兩者數值相同，可移植的程式要同時判這兩個——這是個會在跨平台時咬人的小邊角）。這個錯誤不是「出事了」，它是 kernel 在跟你說：「現在沒東西、要是真讀就得讓你等了，但你說過不要等，所以我先還你，你自己看著辦。」

於是非阻塞模式把一個尖銳的新問題丟回給你：**既然 kernel 不替你等了，你怎麼知道資料「什麼時候」到？** 你總不能寫一個 `while` 迴圈拼命重試 `read`、撞 `EAGAIN` 就再試一次——那叫忙輪詢（busy polling），它確實不阻塞，但會把一整顆 CPU 核心燒在「問了沒、問了沒」上，比阻塞還糟。非阻塞 `read` 只是把執行緒解放出來的**前半步**；要讓它真的有用，你需要一個機制，能在「真的有資料了」的那一刻才回頭來通知你。這個機制，就是整個高並發伺服器的地基。

## 為什麼天真的解法撐不住一萬個連線

退一步看大局。你的服務要同時照看 N 個連線，每個連線大半時間都在閒等——等使用者打字、等對端回應。問題是：怎麼用有限的執行緒，撐住這麼多同時在等的連線？

最直覺、也最古老的答案是 **thread-per-connection（每連線一條執行緒）**：來一個連線，開一條執行緒，這條執行緒就用最樸素的阻塞 `read`／`write` 跟它線性對話——讀一行、處理、寫一行回去，讀不到就阻塞等。這個模型的好處無可取代：**程式碼是線性的、好懂的**，沒有 callback、沒有狀態機，控制流就跟你腦中想的一樣由上往下走。連線數不多時，這是最舒服的寫法。

但把連線數推到一萬，這個模型會從兩個方向同時崩掉，而且第一個方向比多數人記得的更微妙。

**第一個方向是記憶體，但細節是關鍵。** 在 Linux 上，每開一條執行緒，預設要為它的 stack **保留**一塊位址空間，這塊預設是 **8 MB**（由 `ulimit -s` 決定）。一萬條執行緒，乍看就是 80 GB——但這裡有個非顯然的真相：那 8 MB 是**虛擬位址空間的保留**，不是當場吃掉的實體記憶體。stack 的實體頁是 kernel 在你真的踩到那一頁時，靠 page fault **惰性配置**的；一條剛開、stack 還很淺的執行緒，實際駐留記憶體（RSS）可能只有幾十 KB。所以「一萬條執行緒立刻吃 80 GB」是個會誤導人的演算法。

真正的天花板是這三件事裡**最先撞到的那一件**：一是 32 位元行程的虛擬位址空間會被 8 MB × N 的保留直接撐爆（64 位元下這條較鬆，但每個 mapping 仍佔 `vm.max_map_count` 之類的核心上限）；二是**實體記憶體**——只要這些連線真的在工作、stack 真的長到幾十上百 KB，一萬條乘起來就是好幾 GB 的硬實體開銷，而開場那條被 OOM 砍掉的曲線，正是這條在動；三是**排程成本**——這往往才是壓垮駱駝的那根。一萬條執行緒裡，任一瞬間絕大多數都睡在自己的阻塞 `read` 上，但只要有風吹草動，kernel 排程器就得在這上萬個實體中做選擇、做 context switch。每次切換要儲存與還原暫存器、刷新部分 CPU 快取與 TLB，代價是微秒級——單看很小，但當你的執行緒數遠超過 CPU 核心數，CPU 會有可觀的比例不是花在處理請求上，而是花在「決定下一個輪到誰」上。

**第二個方向是浪費得荒謬。** 這上萬條執行緒，絕大多數時刻都睡在阻塞 `read` 上、什麼也沒做——你為「等待」這件事付出了「一條完整 OS 執行緒」的全副身價，而等待本身根本不需要一條執行緒。這就是天真做法的核心荒謬：**用一個重量級的並發單元（OS 執行緒），去承載一件輕量級的事（閒等一個 fd）。**

這個歷史上著名的瓶頸有個名字——**C10K 問題**（怎麼讓單機同時服務一萬個連線）。它的解法路線，全都指向同一個方向：**別再讓「等待」綁住一條執行緒。** 既然一萬個連線裡同一瞬間真正「有資料要處理」的可能只有幾十個，那理想的模型就該是——用極少數執行緒，只在「某個 fd 真的就緒了」時才去碰它，其餘時間這幾條執行緒要嘛在處理就緒的 fd、要嘛整個睡著（連它們也不該空轉）。要做到這件事，就需要前一節懸而未決的那個機制：一個能「同時盯著上萬個 fd、只把就緒的那幾個告訴你」的東西。

## readiness 通知：從 `select` 到 `epoll`

這個機制叫 **I/O 多工（I/O multiplexing）**：一條執行緒，一次系統呼叫，同時盯著一大票 fd，問 kernel「這些裡面，哪些現在可以無阻塞地讀或寫了？」。它的演化史，就是一部「怎麼把這個問題問得更便宜」的歷史。

最早的兩個答案是 `select` 和 `poll`。它們的形狀一樣：你每次呼叫時，把**你關心的全部 fd**（那一萬個）打包傳進 kernel，kernel 掃過一遍，把就緒狀態標記回來給你。問題就在「每次都傳全部、kernel 每次都掃全部」這件事上——它的成本是 **O(n)**，n 是你監看的 fd 總數，不是真正就緒的數量。一萬個連線裡只有 5 個有新資料，你每一輪也得把一萬個 fd 複製進 kernel、讓 kernel 線性掃過一萬個。連線越多，每一輪的固定稅越重；`select` 還額外有一個 fd 數量上限（經典的 `FD_SETSIZE`，通常 1024）的硬牆。這兩個 API 能跑，但跑不到 C10K 的彼岸。

突破來自 Linux 的 `epoll`（BSD/macOS 上對應的是 `kqueue`，思路相同）。它的關鍵設計，是把「一次性傳全部」拆成「**註冊一次、長期維護**」。你用 `epoll_ctl` 把感興趣的 fd **一個一個註冊**進一個 kernel 維護的 epoll 實例（kernel 內部用紅黑樹管這個 interest set）；之後每次呼叫 `epoll_wait`，kernel **只回給你那些此刻真正就緒的 fd**——它在內部維護一張「就緒清單」（一個 linked list），fd 一就緒就被 callback 掛上這張清單，`epoll_wait` 直接把這張清單交給你。

這一下把成本從 O(總 fd 數) 降到 **O(就緒 fd 數)**。盯著一萬個連線、其中 5 個就緒，`epoll_wait` 的工作量就是那 5 個的量級，跟另外 9995 個閒著的連線**無關**。你不再為「監看」付線性稅，只為「真正發生的事」付錢。這個從「每輪掃全部」到「只回就緒的」的轉變，正是現代高並發伺服器（Nginx、Node.js、Redis）全都選 epoll 風格事件驅動的根本原因。

值得多看一眼的是 `epoll` 的兩種觸發語意，因為選錯會直接生 bug。**level-triggered（水平觸發，epoll 的預設）**：只要某個 fd 的接收緩衝區裡**還有**沒讀完的資料，每一次 `epoll_wait` 都會繼續報它就緒——像水位，只要水還在線上就一直響。**edge-triggered（邊緣觸發）**：只在 fd「**從沒資料變成有資料**」的那個瞬間報一次，之後即使緩衝區裡還有殘留、只要沒有新資料進來，就**不再報**——像門鈴，按下的那一刻響一聲，按著不放也不會一直響。

邊緣觸發更省（少很多重複喚醒），但它把一個義務壓到你頭上：被通知就緒後，你**必須把這個 fd 一路讀到 `EAGAIN` 為止**，把當下緩衝區裡的資料**全部**抽乾。為什麼？因為若你只讀了一半就停手，剩下的那半因為「沒有新的邊緣事件」永遠不會再被報——那些資料就**靜靜地卡在緩衝區裡，連線像死了一樣再也不動**，但 socket 其實是好的。這是 epoll 程式設計裡最經典、最難 debug 的陷阱之一：一切看起來都對，連線就是莫名其妙地僵住。也正因為邊緣觸發要求「一路讀到 `EAGAIN`」，它和**非阻塞 fd 是綁死的**——你必須用非阻塞 `read`，才能在抽乾的最後一步靠 `EAGAIN` 知道「好了，這次到底了」，而不是讓最後那一下 `read` 把你阻塞住。前一節埋的伏筆，在這裡接上了：非阻塞 `read` 的 `EAGAIN`，正是 readiness 迴圈賴以運作的訊號。

## 算一筆帳：一萬個連線，兩種模型差在哪

把 C10K 的取捨用具體數字攤開，差距會大到嚇人。

**thread-per-connection 那邊**：一萬個並發連線，一萬條執行緒。先看硬成本——這些連線在工作，stack 不會停在最淺，假設每條執行緒實際駐留的 stack 平均長到約 64 KB（保守估計，真實負載常更多），光 stack 的實體記憶體就是 `10,000 × 64 KB ≈ 640 MB`，而且這還沒算每條執行緒在 kernel 裡的 task 結構等固定開銷。更致命的是排程：一萬條執行緒遠超任何機器的核心數（就算 64 核也超了上百倍），CPU 有相當比例耗在 context switch 而非處理請求上——你會看到一個怪現象：CPU 用滿了，但吞吐很低，因為算力被排程本身吃掉了。

**單執行緒 + epoll 那邊**：一萬個 fd 不過是 kernel 那棵紅黑樹裡的一萬個節點，加上應用層每個連線一小塊狀態（讀寫緩衝區、解析到一半的狀態機），單連線開銷以 KB 計，一萬個合起來是幾十 MB 的量級。沒有一萬條執行緒，就沒有那筆排程稅——這條執行緒大部分時間睡在 `epoll_wait` 上（這一個睡眠是有意義的、kernel 會在有 fd 就緒時精準喚醒它），醒來就成批處理已就緒的那幾十個 fd，處理完再睡回去。記憶體從 640 MB 量級降到幾十 MB，CPU 從「泰半花在切換」變成「幾乎全花在處理」——**同一個負載，兩個量級的差異**，而且差異全部來自一個選擇：要不要為「等待」綁住一條執行緒。

這筆帳也劃出了 epoll 的適用邊界，別把它當銀彈。連線數穩定在數百以內、每個連線吞吐高、邏輯複雜時，thread-per-connection 那份「線性程式碼」的可讀性紅利是真金白銀，過早上 epoll 的事件迴圈反而是拿可維護性換一個你還用不到的擴展性。epoll 真正不可取代，是在連線數上萬、且每個連線大半在閒等的場景——長連線推播、聊天、IoT。判準是**連線數與閒置比**，不是「哪個比較潮」。

## 第二根軸：阻塞 vs 非阻塞，不等於同步 vs 非同步

到這裡有個極容易混淆、也極值得講透的點：**「阻塞 vs 非阻塞」和「同步 vs 非同步」是兩根正交的軸，不是同義詞。** 把它們混為一談，是讀 I/O 模型時最常見的迷霧。

前面講的那套——非阻塞 fd ＋ epoll readiness 通知——雖然「不阻塞」，但它骨子裡仍然是**同步**的。為什麼？因為 epoll 只告訴你「這個 fd 現在可以讀了」（readiness，就緒），**真正那一下 `read` 還是得你自己去呼叫、資料還是 kernel 在你的執行緒上拷貝給你**。你被通知的是「可以開始了」，不是「已經完成了」。這就是 readiness 模型（就緒模型）的本質：kernel 替你盯著、就緒了喊你，但搬運的活兒是你同步去做的。

真正的**非同步 I/O** 是另一回事——它是 **completion 模型（完成模型）**。你把整個操作（「從這個 fd 讀 n 個位元組到這塊 buffer」）連同 buffer 一起**提交**給 kernel，然後就走人去做別的；kernel 在背景把這件事**從頭做到完**——包括資料拷貝——做完了才通知你「這件事完成了，結果在你給的 buffer 裡」。Windows 的 IOCP 一直是這個模型；Linux 上長期缺一個好用的完成式介面，直到 **io_uring** 出現才補齊。

io_uring 的設計很值得看一眼，因為它把「減少系統呼叫」這件事做到了極致。它在 user space 和 kernel 之間架兩個共享記憶體的環形佇列（ring buffer，名字就是這麼來的）：一個**提交佇列（SQ）**、一個**完成佇列（CQ）**。你要發 I/O，就把請求（SQE）寫進 SQ 的尾巴；kernel 從 SQ 頭部取走、做完後把結果（CQE）放進 CQ 的尾巴；你從 CQ 頭部收結果。妙處在**批次**：傳統模型下，發 1000 個 I/O 就要 1000 次（甚至更多）系統呼叫，每次系統呼叫進出 kernel 都有固定開銷；io_uring 可以一口氣把 1000 個 SQE 排進環裡、用**一次** `io_uring_enter` 通知 kernel（搭配 kernel 輪詢模式甚至能做到趨近零次系統呼叫）。在每秒幾十萬 I/O 的場景，省下的系統呼叫開銷相當可觀。

把這兩根軸畫成四格，整張地圖就清楚了：

| | 同步（自己動手） | 非同步（kernel 代勞到完成） |
|---|---|---|
| 阻塞 | 樸素的 `read`：呼叫就睡，醒來資料在手 | （罕見/退化組合，實務不用） |
| 非阻塞 | epoll/kqueue readiness：喊你就緒、你自己 `read` | io_uring / IOCP completion：提交即走、完成才通知 |

一般後端語境裡講的「非阻塞 I/O」，幾乎都是指左下那格——非阻塞 fd ＋ readiness 通知，也就是 event loop 賴以成立的地基（event loop 怎麼把這套底層機制包成單執行緒並發，見〈event loop：單執行緒怎麼同時做很多事〉）。而 io_uring 那種完成模型雖然更先進，普及上卻有現實阻力：它需要較新的 Linux kernel（5.1 起才有），且因為 2022–2023 年間它成了 kernel 安全漏洞的重災區，Google 已在 ChromeOS、Android 與部分生產環境主動停用或限制它，至 2026-06 仍未全面恢復。新、強、但生態與信任都還在補課——這正是「最先進的機制不一定是預設選擇」的活教材。

## 非阻塞解的是「等」，不是「算」

最後要釘死一個邊界，因為這是把非阻塞 I/O 誤當萬靈丹的人最常摔的跤：**非阻塞 I/O 解決的是「等 I/O」浪費執行緒的問題，它對「算 CPU 卡住執行緒」完全無能為力。**

設想你那條跑著 epoll 迴圈的執行緒，盯著一萬個連線，運轉得漂亮。某個請求進來，需要做一件 CPU 密集的事——同步解析一個 50 MB 的 JSON、跑一段重的加解密、縮放一張大圖——耗時 500 毫秒。在這 500 毫秒裡，這條執行緒**被這個計算牢牢佔住**，它回不到 `epoll_wait`，於是另外 9999 個連線即使資料早就就緒、躺在 kernel 緩衝區裡，也**沒有人去處理**——它們全部一起凍結 500 毫秒。非阻塞 I/O 在這裡幫不上任何忙：fd 是非阻塞的、epoll 是高效的，但卡住執行緒的根本不是 I/O，是那段不肯讓出的計算。

這個區分之所以要命，是因為它劃清了非阻塞 I/O 的能力邊界。它讓「等待」不再綁架執行緒，但它沒有、也不可能讓「計算」不綁架執行緒——一條執行緒在算一段同步程式碼時，它就是在那條執行緒上、不在別處。要解 CPU 卡住事件迴圈的問題，得換一套完全不同的工具：把計算搬離事件迴圈所在的那條執行緒，丟給別的執行緒或行程去平行地算（這套機制——thread pool 與 worker threads——見〈thread pool 與 worker threads：CPU 密集怎麼辦〉）。

同一個道理還藏著一個悄悄咬人的角落：**並非所有「看起來像 I/O」的操作都走非阻塞那條路。** 最典型的是 DNS 解析。直覺會把 `getaddrinfo`（解主機名）當成網路操作、預期它走 epoll，但它在許多平台上其實是一個**阻塞的 libc 呼叫**——底層沒有乾淨的非阻塞介面，runtime（如 Node.js 的 libuv）只好偷偷把它丟到一個**小小的執行緒池**裡用阻塞方式做掉（這個池預設只有 4 條，網路 socket 的 epoll 反而不佔它）。平時無感，但當你的服務瞬間要解大量主機名、把那 4 條執行緒佔滿，新的 DNS 查詢就得排隊——而你會百思不解：明明是「非阻塞」的服務，怎麼 DNS 慢得像塞車？因為它走的根本不是直覺以為的那條路。**「非阻塞」是 fd 與機制的性質，不是「凡是 I/O 就自動非阻塞」的保證**——哪些操作真有非阻塞的底層支援、哪些只能靠執行緒池假裝，是你選 runtime 時得親自查清的事。

## 為什麼是這個形狀

退一步看，這一整套機制——非阻塞 fd、`EAGAIN`、epoll 的就緒清單、邊緣觸發要讀到底、完成模型的環形佇列——全都是從一個樸素的事實裡長出來的：**I/O 的本質是「等」，而「等」這件事，不該綁住一個能做事的執行緒。**

阻塞模型把「等」和「一條執行緒」死死綁在一起，於是 N 個連線要 N 條執行緒，撞上 C10K 的牆。非阻塞模型做的，是把「等」從執行緒身上**剝離**出來——讓 kernel 用一張就緒清單去替你盯著上萬個 fd，你的少數幾條執行緒只在「真的有事」時才動。剝離得越乾淨，一條執行緒能照看的連線就越多：readiness 模型（epoll）讓你不必為閒等付執行緒，completion 模型（io_uring）更進一步讓你連系統呼叫的次數都省下來。每一步演化，都是在同一個方向上把「等」這件事和「算」這件事拆得更開。

所以當你看到一個服務用一條主執行緒、配著 epoll，從容地照看著幾萬個長連線，你看到的不是什麼黑魔法，而是這個第一原理被貫徹到底的樣子：它只是**拒絕為等待付出一條執行緒的代價**。也正因為它把這件事做得這麼徹底，它才會在那唯一沒被拆開的地方——一段不肯讓出的 CPU 計算——暴露出全部的脆弱：那一刻，整條執行緒連同它照看的所有連線，一起停在那裡等一件，本可以搬到別處去算的事。
