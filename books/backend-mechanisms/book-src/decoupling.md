# 解耦：時間、空間、同步性、資料、實作五種耦合

一個服務上線三年了，沒人敢動它的某張資料表。不是因為那張表特別神聖，而是因為沒人說得清還有誰在讀它。報表系統直連這張表跑夜間批次、另一個團隊的後台直接 `JOIN` 它做查詢、一個早就忘了誰寫的同步腳本每小時拉一次。你想把一個 `VARCHAR(50)` 的欄位改成 `TEXT`、或把一個布林旗標拆成兩個狀態——一個十分鐘的改動，卡在一個沒人能回答的問題上：**改了會弄裂誰？**

這就是耦合（coupling）的真面目。它不是架構圖上畫得出來的那種「A 有一條箭頭指向 B」，而是那些**沒畫在圖上、卻綁得最死的假設**：某人假設這個欄位永遠是 50 字以內、某人假設這張表的主鍵格式不變、某人假設這個值改完後一秒內就會出現在下游。每一條這樣的假設，都是一條繩子。解耦這件事，講穿了就是**有意識地數清楚有哪些繩子、剪掉值得剪的、留下該留的**——而不是把「拆開」當成一種美德去盲目追求。

這一章要做兩件事。第一，把「耦合」這個被講爛、卻幾乎總是被當成單一純量在用的詞，拆成幾個**正交的維度**——因為唯有拆開，你才回答得了那個讓所有人卡住的問題：「我加了一個訊息佇列，到底解了哪一條繩子、又沒解哪一條？」第二，把它的攣生兄弟「內聚」擺到旁邊一起看——因為只追求低耦合會把你帶向一個災難性的反面。

## 耦合是「假設量」，不是「有沒有連線」

先把耦合的定義收緊。工程師日常講「這兩個服務耦合很重」，腦中浮現的往往是「它們之間有很多呼叫」。但呼叫多不等於耦合重。真正的度量是：**A 對 B 抱持了多少假設，使得 B 一變、A 就被迫跟著變、跟著重新部署、甚至跟著失敗。**

把這句話拆開，每一種「假設」就是一個獨立的耦合維度。Gregor Hohpe 在 *Enterprise Integration Patterns* 之後持續打磨這套分類，到後來列出八個面向（technology、location、topology、data format、semantic、conversation、order、temporal）。下面這五條，是把其中對後端工程師最關鍵、最常絆倒人的幾個面向挑出來、並刻意把「同步性」從 Hohpe 歸在 temporal 底下的部分**單獨拆出來**——因為它和時間耦合會分家，這個分岔待會兒會看到。先記住一件事：**它們彼此正交**——你可以在某一維上鬆綁，卻在另一維上綁得死死的，而這正是大多數「我以為我解耦了」的故事出錯的地方。

**時間耦合（temporal）**：A 要做事的當下，B 必須也活著、也準備好回應。一個同步 HTTP 呼叫是最強的時間耦合——B 此刻掛了，A 這次呼叫就跟著失敗。兩端在這次互動期間是命運共同體。

**空間耦合（spatial / location）**：A 必須知道 B 在哪——具體的 IP、host、port。把 endpoint 寫死在設定檔裡是強空間耦合；B 一搬家、一擴副本、一換 IP，A 就得改設定、重部署。

**同步性耦合（synchronization）**：A 發出請求後，必須**卡在原地**等 B 回來才能往下走。這一維常和時間耦合被混為一談，但它們是兩回事，待會兒會看到一個它們分家的場景。

**資料格式耦合（data / schema）**：A 必須知道 B 的資料**長什麼樣**——有哪些欄位、型別是什麼、語意是什麼。章首那張「沒人敢動的表」就是最強的資料耦合：報表系統直接讀 B 的內部結構，B 一改欄位，讀它的那一端就地裂開。

**實作耦合（implementation / technology）**：A 假設了 B 用什麼語言、什麼框架、內部用什麼演算法、回應藏在 XML 還是 JSON 的哪一層。透過一份穩定的契約把這些藏起來，B 換掉整個實作 A 都不該有感覺。

這五條的價值，在於它讓「解耦」從一句口號變成一個**可逐條盤點的清單**。當有人說「我們上了 Kafka 來解耦」，你現在可以追問：解了時間耦合嗎（生產者不必等消費者在線）？解了空間耦合嗎（生產者不必知道誰在訂）？——但**解了資料格式耦合嗎？** 通常一條都沒解。兩端還是共用同一份事件 schema，你把事件結構一改，所有消費者照樣一起裂。這就帶到全章最重要的一個認知。

## 「加佇列＝解耦」是最貴的一個誤會

把「上訊息佇列」當成萬用解耦劑，是這個領域最普遍的誤判，而它之所以危險，正是因為它**部分為真**——真到足以讓人停止思考。

佇列確實剪了兩條繩子。生產者把訊息丟進佇列就走，不必等消費者處理完——剪掉同步性耦合。消費者此刻掛了，訊息留在佇列裡等它回來，生產者不受影響——剪掉時間耦合。這兩刀剪得很漂亮，於是人們順手得出結論：「現在它們解耦了。」

但 Hohpe 有一個一針見血的提醒：**事件驅動，不等於鬆耦合**。把同步呼叫換成丟訊息，你動到的只是「互動風格」這一維；不管走哪種風格，「發送方和接收方仍然必須對一份資料格式與其語意達成共識」。換句話說，**資料格式耦合一刀未剪**。更微妙的是，佇列還可能讓事情在某一維上**更糟**：本來只有一個下游讀你的資料，上了 pub/sub 之後變成五個未知的下游各自存了一份對你 schema 的依賴——你要改那個事件結構時，要協調的下游從一個變成五個，而且其中有些你根本不知道存在。Hohpe 把這個現象講得很白：能在不驚動發送方的情況下加訂閱者，這份便利會在系統長大後反過來咬你，因為「更多接收方，意味著訊息格式一變時，潛在的衝擊面更大」。

所以判斷一次「解耦」是否真的發生，不能看「中間有沒有插一個 middleware」，要看**剪掉了哪幾維的繩子**。把這個視角套回各種常見手段，每一種都能精準定位它的作用面：

| 手段 | 主要剪掉哪一維 | 沒剪掉的（容易被忽略） |
|---|---|---|
| 非同步訊息佇列 | 時間 + 同步性 | 資料格式（共用 schema 仍綁死） |
| 服務發現 / DNS / 負載均衡 | 空間 | 時間、同步性、資料格式 |
| 穩定的 API 契約 + 相容演進 | 資料格式 + 實作 | 時間（同步呼叫仍命運共同體） |
| 發布／訂閱 | 空間（不知有誰在聽）+ 時間 + 同步性 | 資料格式（事件 schema 仍是耦合點） |
| 每服務私有資料庫 | 資料格式（最強形式） | — |
| 依賴反轉（依賴介面而非實作） | 實作 | 資料格式（介面簽章的型別／語意一變，雙方仍一起裂） |

這張表不是要你背，是要你養成一個反射：每次要「解耦」前，先問自己**痛的是哪一維**，再挑剪那一維的工具。空間耦合在痛（B 老是換 IP 害 A 要改設定），你需要的是服務發現（見〈DNS 與服務發現〉），上佇列一點忙都幫不上。資料格式耦合在痛（B 一改欄位下游就裂），你需要的是一份有版本治理、能向後相容演進的契約（見〈序列化與 schema 演進〉），佇列同樣救不了你——訊息照樣帶著那份脆弱的 schema 流過去。

## 時間與同步性：一個容易被當成同一件事的分岔

前面列維度時埋了一個伏筆：時間耦合和同步性耦合常被當成同義詞，但它們會分家。把這個分岔看清楚，是真正讀懂「同步 vs 非同步」的前提（那一維的完整取捨見〈同步與非同步通訊〉，這裡只看它和「解耦」的接點）。

設想一個非同步的請求—回應模式：A 把一個請求丟進「請求佇列」，附上一個 correlation id 和一個「回覆佇列」的位址，然後**繼續卡在那裡，等回覆佇列冒出帶著同一個 correlation id 的結果**。在這個設計裡：

- 時間耦合**被剪了**：B 不必在 A 發請求的當下就在線，訊息會在佇列裡等它。
- 同步性耦合**沒被剪**：A 還是卡著等回應才能往下走。

反過來也成立。一個 fire-and-forget 的事件（A 發出「使用者登入了」就走，永遠不在乎有沒有人處理）是時間和同步性**雙雙解開**。而一個帶逾時的同步 HTTP 呼叫，兩維都綁著——但綁的程度因為逾時而有了上限，不至於無限等下去。

為什麼要把這兩維拆這麼細？因為它直接決定**失敗怎麼傳遞**。同步性耦合存在時，B 的延遲就直接累加成 A 的延遲；一旦 B 落到長尾，A 跟著落到長尾（這個尾延遲在同步鏈上的放大見〈長尾延遲：p99 從哪裡來〉）。時間耦合存在時，B 的**死活**直接決定 A 這次呼叫的死活。你可以只想解決其中一個——比如你不在乎多等一會兒（同步性可以留著），但很在乎 B 半夜掛掉時別把 A 也一起拖垮（時間耦合一定要剪）——這時候「丟佇列、但仍等回覆」就是對的形狀，而「全改成 fire-and-forget」反而會讓你丟掉本來需要的那個回應。

## 一個帶數字的取捨：解耦買到了什麼，又欠下了什麼

抽象講了這麼多，把它壓進一個能手算的場景。一個再普通不過的流程：使用者下單成功後，寄一封確認信。

最直覺的寫法是同步串接——下單 API 的處理流程裡，直接呼叫郵件服務，等它回 200 才回覆使用者「下單成功」。問題出在可用性會**相乘**。假設下單服務本身的月可用性是 **99.9%**，郵件服務是 **99.5%**。在串接架構下，使用者要看到「下單成功」，這兩個服務在那一刻都得活著——這是一個**序列依賴**，整條鏈的可用性是兩者相乘：

```
combined availability = 0.999 × 0.995 = 0.994005 ≈ 99.4%
```

數字本身不嚇人，把它翻成停機時間才有感。一個三十天的月份共 43,200 分鐘。郵件服務的 99.5% 自己就對應每月 **216 分鐘（正好 3.6 小時）** 的不可用；而在同步串接下，這 3.6 小時會**整段傳染給下單**——郵件服務一掛，使用者連單都下不了。下單的可用性從 99.9%（每月約 43 分鐘的停機）被硬生生拖到 99.4%（每月約 259 分鐘）。**你的下單能力，被一個遠不如它關鍵的寄信動作給綁架了。**

現在剪掉那條繩子。把「寄信」從同步呼叫改成「丟一則訊息到佇列就走」，下單 API 不再等郵件服務。這一刀剪的正是時間 + 同步性耦合：郵件服務掛掉，影響的只是「信晚一點到」，下單本身回到它自己的 **99.9%**。一個欄位都沒動、一行業務邏輯都沒改，只是把一個同步呼叫換成一則訊息，下單的月停機從 259 分鐘掉回 43 分鐘。

但解耦從來不是免費的午餐，它只是**把一種複雜度換成另一種**。剪掉那條繩子的同一瞬間，你欠下了三筆新的債：

- **最終一致取代了即時確認**。本來「下單成功」就保證「信寄了」，現在「下單成功」只保證「寄信的請求進了佇列」。信到底寄出去沒有，不再是同步可知的事實。
- **「寄信失敗」從同步異常變成沉默的卡住**。同步架構下郵件服務報錯，你當場就接到；非同步架構下它失敗了，沒有人會當場知道——你得靠佇列的監控、靠死信佇列（見〈死信與毒訊息：DLQ〉）把卡住的訊息撈出來，否則錯誤就被掃進了地毯下。
- **重複投遞變成常態**。佇列的投遞語意幾乎都是 at-least-once，同一則「寄確認信」可能被投兩次，消費端必須冪等、否則使用者收到兩封信（冪等的完整機制見〈重複是常態：冪等與去重〉）。

這就是解耦的本質交換，沒有例外：**你用「信的即時確認性」換到了「下單的可用性」，再用「一套監控 + DLQ + 冪等的營運負擔」付清這筆交易。** 划不划算，取決於這個流程能不能容忍最終一致。寄確認信可以——晚三秒到沒人會死。但如果換成「扣庫存」呢？「扣庫存」必須當下成功才能確認訂單，硬把它改成非同步事件，換來的「最終一致」會製造一個致命的窗口：**訂單已經成立、但庫存還沒扣，超賣就發生在這個縫裡。** 在這種強一致需求下，那個時間耦合**不是缺陷，是特性**——它正是你要的「不確認成功就不放行」。

這帶出一條容易被遺忘的原則：**不是所有耦合都該剪。** 解耦是有成本的工具，不是放諸四海的美德。在「該強一致」的地方剪掉時間耦合，等於主動製造了一個對帳問題。

## 隱性耦合：最該剪、卻最看不見的那幾條

到目前為止講的都是「看得見」的耦合——API 上的、設定檔裡的、訊息 schema 上的。但綁得最死、也最常在某次改動時把你炸傷的，往往是那些**不在任何契約上**的假設。

最經典的就是章首那張共享資料表。它危險不在於「有人讀它」，而在於這個依賴**沒有出現在任何 API 文件、任何介面定義裡**——它藏在某個你看不到的報表 SQL、某個別團隊的 `JOIN`、某個忘了誰寫的同步腳本中。你改 schema 時，編譯器不會警告你、型別檢查不會攔你、CI 不會紅——它在生產環境的某個夜間批次裡靜靜地裂開，而你要花半天才追到是誰在讀。這就是為什麼「每服務私有資料庫、禁止跨服務直接讀彼此的表」是微服務裡最關鍵、卻最常被偷懶跳過的一條紀律：它把資料格式耦合從「藏在 SQL 裡的隱性繩子」逼回到「攤在 API 上的顯性契約」，讓任何一次破壞性變更都至少**看得見**。

同類的隱性繩子還有幾種，值得各記一筆：

- **魔術字串狀態值**：A 寫進資料庫一個 `status = 'PENDING_REVIEW'`，B 直接字串比對這個值來決定行為。這個耦合不在任何介面上，但 A 一改字串、B 就地失靈。
- **跨服務的順序假設**：B 假設 A 一定先發了 `OrderCreated` 才會發 `OrderPaid`，於是收到 `OrderPaid` 時直接去查那筆訂單。等哪天訊息亂序到達（分散式系統裡這是常態，不是意外），B 就查到了不存在的訂單。這條繩子綁在「時序」上，圖上完全畫不出來。
- **共用的可變狀態**：兩個模組讀寫同一個全域變數、同一塊快取、同一張表。Constantine 的耦合光譜裡，這叫 common coupling，是僅次於「一個模組直接竄改另一個內部」的次強耦合。

真正徹底的解耦，要連這些檯面下的假設一起盤點。一個實用的自測：**如果我半夜改了某個東西，會在哪些我此刻想不到的地方爆掉？** 想不全，正說明隱性耦合的存在。

## 翻到另一面：低耦合若沒有高內聚，是另一場災難

如果解耦的故事到此為止，會留下一個危險的暗示：耦合越低越好，所以盡量拆。這個推論會把你帶進一個經典的反面災難，而要看清它，必須請出耦合的攣生概念——內聚（cohesion）。

這對概念是 Larry Constantine 在 1960 年代末提出、後與 Edward Yourdon 在 1979 年的 *Structured Design* 裡系統化的，是「東西該怎麼切」這個問題上最古老也最耐用的一把尺。兩者的定義互補：

- **耦合**＝模組**之間**的依賴強度。越低越好（改一個不必動另一個）。
- **內聚**＝模組**內部**元素的相關程度。越高越好（一個模組裡的東西都在服務同一個目的）。

關鍵在於：**這兩個量必須一起看，單盯一個一定切錯。** 證明很簡單，看兩個極端就懂。

只追求**低耦合**：你可以把所有程式碼塞進一個巨大的模組。模組之間的依賴？零，因為根本只有一個模組。耦合度完美。但內部是一鍋什麼都攪在一起的粥——使用者驗證、訂單計算、寄信、報表，全混在一起。這是內聚爛到底，沒人能安全地改任何一處。

只追求**高內聚**：你可以把每一個函式都拆成一個獨立的微服務，每個都極度專注、只做一件事。內聚度完美。但它們彼此呼叫成一張密網，一個簡單請求要跨七個服務的網路呼叫，耦合爆炸到天上去。這就是惡名昭彰的「分散式單體」——你買下了分散式系統的全部痛（部分失敗、最終一致、難以追蹤的跨服務流程），卻沒換到任何獨立演化的好處，因為這些服務根本綁在一起、必須同步部署。**服務邊界畫錯，比不拆還糟。**

好的切法是**同時**滿足兩者：每個模組內部高內聚（職責單一），模組之間低耦合（依賴少而穩定）。而要找到這條正確的接縫，最誠實的證據不在任何漂亮的架構圖上，而在你的版本控制歷史裡。

## 版本歷史是內聚最誠實的證人

怎麼判斷一條邊界切對沒？有一個近乎作弊的具體判準：**看哪些檔案總是出現在同一個 commit 裡。**

設想一個 200 個檔案的單體，你用工具掃過去半年的 commit，量出「平均一次功能變更，要動 12 個檔案、分散在 5 個模組」。這個數字本身就是一張診斷書：它同時是**低內聚**（一個功能的相關邏輯散落在五個地方）和**高耦合**（改一處被迫連帶改另外四處）的徵狀。相關的東西沒住在一起，而住在一起的東西又彼此糾纏。

修法叫「按變更軸線（axis of change）重切」——把「總是一起改」的東西聚進同一個模組，目標是把那個數字壓到「一次功能變更只動 1–2 個模組」。而判斷「哪些東西總是一起改」，最準的資料不是工程師的記憶、不是架構文件，而是 **co-change（共變）紀錄**：如果某幾個檔案在過去半年的 PR 裡反覆地、總是一起出現，它們多半本來就該住在一起。版本控制不會說謊——它記錄的是這個系統**實際上**怎麼被改的，而不是某人**希望**它被怎麼切。

這裡藏著一個反直覺、卻能擋下很多誤判的陷阱：**有時候，刻意重複比錯誤的抽象更解耦。** DRY（don't repeat yourself）是好原則，但用過頭會反咬一口。你看到兩段程式碼長得像，於是把它們抽成一個共用模組。問題是，如果這兩段碼**碰巧長得像、但會各自往不同方向演化**——一個是「計算稅率」、一個是「計算折扣」，今天公式恰好一樣——你就在它們之間製造了一條本來不存在的繩子。明天稅率邏輯要變，你一改那個共用模組，折扣計算跟著裂。用耦合／內聚這把尺去看，這件事其實很清楚：你為了消除重複，引入了一個**虛假的內聚**（把兩個目的不同的東西硬說成同一個），代價是真實的耦合。重複是看得見的成本，錯誤抽象是看不見的負債——而看不見的那種，總是更貴。

## 為什麼是這個形狀

退一步看，這一章講的所有東西，都是從同一個第一原理長出來的：**耦合是「假設量」，而假設是多維的、彼此正交的。**

正因為它是「假設量」而非「連線數」，你才不能靠「插一個 middleware」來解耦——middleware 只在某一兩維上剪繩子，卻可能在另一維上綁得更緊。正因為它是多維的，你才必須先問「痛的是哪一維」，再挑剪那一維的工具，否則就會發生「上了 Kafka 卻沒解掉資料耦合」這種昂貴的空忙。正因為剪每一條繩子都要付代價（最終一致、新的故障點、新的監控負擔），解耦才不是一種美德，而是一筆**要逐項核算划不划算的交易**——在該強一致的地方，那條時間耦合是你刻意保留的特性。

而它的攣生兄弟內聚提醒你：解耦不是把東西拆得越散越好。低耦合配上低內聚，得到的是一鍋粥或一個分散式單體；真正的目標永遠是那條讓「常一起改的住在一起、不相干的彼此獨立」的接縫。最誠實的指南針不在架構圖上，在你的 commit 歷史裡——它記得這個系統真正怎麼被改，也因此最清楚那些繩子到底綁在哪。

下次你站在那張「沒人敢動的表」前面，你會知道該怎麼想了：先別急著加佇列、加 gateway、加抽象層。先把繩子一條條數出來——誰假設了這個欄位的型別、誰假設了它的語意、誰假設了它一改就同步生效。數清楚了，你才知道哪幾條值得剪、哪幾條剪了反而會把你本來想要的保證一起剪掉。
