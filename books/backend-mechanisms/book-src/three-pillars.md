# 三本柱：metric、log 與 trace

凌晨兩點，有人在群組裡丟一句：「結帳變慢了，使用者在抱怨。」你打開儀表板，一條延遲曲線在過去十分鐘從 80 毫秒爬到 600 毫秒。然後呢？這條曲線告訴你「有事」，但它一個字都沒說「是什麼事」。它不知道是哪些使用者、哪個 endpoint、哪一台機器、卡在第幾跳。你盯著那條往上爬的線，手裡握著的全部資訊就是一個數字隨時間變大——而你需要的是一個故事。

可觀測性（observability）這個詞，剝掉行銷的外殼，問的是一個很樸素的問題：**系統在生產環境出事時，你能不能只靠它往外吐的訊號，推斷出它內部發生了什麼，而不必登進去重現？** 重現往往是辦不到的——那個導致雪崩的請求組合、那一瞬間的快取冷熱分布、那個只在特定使用者身上觸發的邊界，你重跑一百次也湊不回來。所以你只剩下系統當時留下的痕跡。問題是，留什麼痕跡、留多細、付多少代價，這裡面藏著一組逃不掉的取捨。三本柱——metric、log、trace——就是工程界對這組取捨給出的三個不同答案。它們不是三個可以挑一個的選項，理解它們的關鍵，是看清**為什麼非得是三個、而不是一個**。

## 為什麼一種訊號不夠

先做一個思想實驗：假設你只能留一種痕跡，你會留哪一種？

如果只留**最詳細**的——把每個請求發生的每一件事，連同所有欄位，原原本本寫成一條紀錄——那你什麼都答得出來，代價是你會被自己的資料淹死。一個每秒一千請求的服務，每個請求寫一條一 KB 的紀錄，一天就是：

```
1,000 req/s × 86,400 s/day × 1 KB = 86,400,000 KB ≈ 86 GB / 天
```

而且這只是一條紀錄。真實的服務一個請求往往落十幾條 log。86 GB 變成幾百 GB，乘上三十天，再乘上你想保留的月數——光是儲存帳單就能讓人重新思考人生。高流量下保留全量明細，財務上幾乎一定不可行，於是你被迫取樣（sampling）：每一百條留一條、每一千條留一條。一取樣，這種訊號就不再「什麼都答得出來」了——你想回頭看的那一筆，有很大機率正好是被丟掉的那九十九分之一。

如果只留**最便宜**的——把事件壓成數字，不記「這一筆是什麼」，只記「這一秒發生了幾筆」——那你能用幾乎可以忽略的成本看趨勢，代價是細節在你壓縮的那一刻就永遠消失了。一條時間序列（time series）不管承載一筆請求還是一億筆，存下來都只是一串 `(timestamp, value)`。便宜到誇張。但等你想問「凌晨兩點那一秒，到底是哪些 user 觸發了慢請求」，答案是：資訊已經不在了。你當初沒留，現在補不回來。聚合是一個單向的門，事件一旦走進去變成數字，就再也走不回來變成事件。

這就是三本柱背後那條唯一的第一原理：**一種訊號的資訊量，和它的成本成反比。** 你不可能同時要「無損的細節」和「可忽略的成本」——那違反熱力學等級的常識。資訊要嘛在源頭被你丟掉（取樣），要嘛被你壓扁（聚合），要嘛你就得付錢全部留著。三本柱是在這條「資訊—成本」的光譜上，釘下的三個工程上最有用的點。

| 訊號 | 本質 | 資訊量 | 成本 | 它擅長回答 |
|---|---|---|---|---|
| Metric | 把事件聚合成隨時間的數字 | 低（聚合即失真） | 極低、固定 | 「現在健康嗎、趨勢如何」 |
| Log | 離散、帶時間戳的事件紀錄 | 最高（單筆無損） | 高、隨流量長 | 「這一筆到底發生了什麼」 |
| Trace | 一個請求跨服務的因果樹 | 中（補上跨服務維度） | 高（幾乎必取樣） | 「這次慢/錯在哪一跳」 |

接下來把這三個點各自看清楚——它們各自買到什麼、各自在哪裡騙你。

## Metric：用失真換來幾乎免費的「持續在看」

metric 的本質是聚合。你不記「第 8,142,330 號請求花了 230 毫秒、回了 200」，你記「過去這一秒，`http_requests_total` 這個計數器又 +1 了」。個別請求的身分、時間、欄位，全部在 +1 的那一刻被丟棄，只留下計數本身。換來的是一個驚人的性質：**這條訊號的成本由維度決定，不由流量決定。** 一條計數器扛一筆請求或一億筆，存下來都是同一串時間序列。這就是為什麼 metric 是唯一你能放心開全量、永不取樣的訊號——它根本不在乎流量多大。

要把不同的問題壓成數字，需要不同形狀的聚合，這就是 metric 的幾種基本型別。**counter（計數器）**是只增不減、重啟歸零的累計值，「總共發生幾次」用它（請求數、錯誤數）。**gauge（量表）**是可上可下的瞬時值，「此刻是多少」用它（佇列長度、記憶體用量、連線數）。這兩個直觀。真正需要想清楚的是延遲這種**分布**——你不能用 counter，因為你要的不是「總共幾次」；也不能用一個 gauge 存「平均延遲」，因為平均值會騙你（後面會看到怎麼騙）。延遲要用 **histogram（直方圖）**：預先切好一組桶（bucket），比如 `≤10ms`、`≤50ms`、`≤100ms`、`≤500ms`、`≤1s`、`+Inf`，每來一筆請求，就在它落到的桶（以及所有比它大的累計桶）上 +1。histogram 本質上是一組以延遲為邊界的累計計數器，外加一個觀測值總和。

histogram 的精妙之處，要在「跨多台機器算 p99」這個具體場景下才看得出來，而這裡正藏著一個會出錯的邊界。假設你的服務有三個副本，你想知道整個服務的 p99 延遲。一個非常自然、而且**錯誤**的做法是：拿三台各自算好的 p99，平均一下。

**分位數不能平均。** 這不是近似不近似的問題，是數學上錯的。設想副本 A 處理了一百萬筆都很快的請求、它的 p99 是 50ms；副本 B 處理了一千筆但全都很慢、它的 p99 是 2000ms。把 50 和 2000 平均得到 1025ms——可是合併後的真實資料集裡，B 那一千筆慢請求在一百萬零一千筆裡連 0.1% 都不到，根本碰不到第 99 百分位，真實的合併 p99 其實很接近 50ms。平均分位數既不知道每台處理了多少筆、也不知道分布長什麼樣，它把資訊量級完全不同的兩個數字當成等重的兩票，算出來的東西沒有任何意義。

histogram 正是為了堵住這個洞而設計的。因為它存的不是算好的分位數，而是**桶的累計計數**，而計數是可加的：三台機器同一個桶 `≤100ms` 的計數，可以直接相加，得到「整個服務裡 ≤100ms 的請求總數」。把所有桶都這樣對齊相加（關鍵是別在加總時弄丟標示桶邊界的那個 `le` 標籤），你就得到一個合併後的直方圖，再在這個合併直方圖上插值算 p99——順序是**先合併桶、後算分位數**，絕不能反過來。histogram 的代價是準確度受限於桶邊界：兩個桶之間靠線性插值估計，桶切得太寬，分位數就只是個大概。（新一代的 native histogram 用動態、指數間距的桶解掉了「邊界要事先猜對」這個老問題，2026 年已普遍可用，但「桶計數可加、算好的分位數不可加」這條根本道理不變。）

順帶說清楚一個常被混用的近親：**summary**。它和 histogram 看起來很像，差別在分位數**在 client 端就算好了**——它直接吐給你「這台的 p99 是 230ms」，而不是一堆桶。好處是單台的分位數精確、不需後端插值；致命的壞處正是上面那條——算好的分位數不能跨實例聚合。所以判準很乾脆：只要你的服務有不只一個副本、需要算整體分位數，就用 histogram；summary 只在「單一實例、永不需要合併」時才合適。

那 metric 在哪裡騙你？最深的陷阱不是它不準，而是它**誘惑你把不該放進去的東西放進去**。metric 那麼便宜、那麼好用，你會很想在請求計數上多掛一個 `user_id` 標籤，「這樣就能看 per-user 的量了」。這一掛，成本模型就從「由流量決定」翻轉成「由維度組合決定」——每一個不同的標籤組合都是一條獨立的時間序列，而組合數是各標籤取值數的**乘積**。一個有五十萬活躍 user 的服務，這一個標籤就能讓一條 metric 炸成幾千萬條序列，把監控系統的記憶體吃乾。高基數維度（user_id、request_id、完整 URL）天生不屬於 metric，它們屬於 log 和 trace——這條分工線本身值得單獨講透（見〈cardinality 爆炸：一個 label 如何拖垮監控〉）。這裡只需記住：metric 用「丟掉身分」換來便宜，你一旦試圖用標籤把身分塞回去，就是在退掉它最核心的那筆交易。

## Log：唯一無損、但最會在你最需要時消失的訊號

如果 metric 是「丟掉身分、只留計數」，log 就是它的反面：**留下這一筆事件的全部身分，一個欄位都不丟。** 一條 log 就是「在這個時刻，發生了這一件具體的事，這是它的全部欄位」。它是三本柱裡資訊量最高、最不失真的——當你需要知道「那個失敗的請求，它的 error message 到底是什麼、帶了哪些參數、走了哪個分支」，只有 log 答得出來，因為只有它把那一筆的細節原封不動留了下來。

但這份無損是要付兩種代價的，而且這兩種代價都在你最不希望它們出現的時候找上門。

第一種代價是**量**，前面已經算過——高流量下 log 大到幾乎一定要取樣。一旦取樣，它就退化成「有代表性的樣本」而非「完整真相」，而尾端那些最有價值的稀有事件（錯誤、超慢請求），在隨機取樣下被命中的機率最低，恰好最容易溜走。

第二種代價更陰險，因為它違反直覺：**log pipeline 最常丟資料的時刻，正是系統出事的時刻。** log 不是寫了就一定到得了後端。它先進一個記憶體緩衝區，由 agent 批次送往日誌系統。當系統出事，log 量會暴增（每個請求都在記錄錯誤、堆疊、重試），緩衝區瞬間被灌滿——滿了之後，agent 通常選擇**靜默丟棄**新進的 log，因為阻塞住會把背壓傳回應用程式、讓本來就在掙扎的服務雪上加霜。於是你得到一個殘酷的反諷：事故期間，正是你最需要每一條 log 的時候，恰恰是 log 最不完整的時候。事後你從 log 統計「事故期間的請求數」，數字常常莫名其妙地偏低——不是請求真的變少了，是記錄它們的那條路在背壓下把資料丟了。任何「系統最忙時還能完整記下每一件事」的假設，都會在第一場真正的事故裡破掉。

正因如此，有一條判準值得記死：**任何要拿去告警、要當作正確性依據的數字，都不要從 log 去算，要用埋點的 metric。** counter 是每個事件在程式裡原子地 +1，不經過那條會在背壓下丟資料的 pipeline，因此它的計數不受取樣與丟棄影響。從 log 反推 metric 看似省事（反正都記了），但它把數字的正確性綁死在 log 的完整性上——而 log 偏偏是最會在關鍵時刻失真的訊號，這裡面的取樣、遺漏、視窗三重陷阱深到值得單獨拆解（見〈從 log 算 metric 的陷阱：取樣、遺漏與視窗〉）。

log 真正不可取代的地方，是**事後可查的任意維度**。你沒辦法為每一個還沒發生、還沒想到的問題事先埋好 metric——但只要當下把事件完整寫進了結構化 log，事後你想按任何欄位切、任何條件濾，資料都還在那裡等你。這是 metric 給不了的彈性：metric 只能回答你**事先決定要問**的問題，log 能回答你**事後才想到**要問的問題。代價就是那份「把細節都留著」的帳單與脆弱。

## Trace：當「慢在哪一跳」變成一個沒人認帳的羅生門

metric 和 log 在單一服務裡幾乎夠用了。trace 要解的，是一個只在請求穿過**多個服務**時才浮現、而前兩本柱都答不了的問題。

看一個具體的場景。一個結帳請求的路徑是 `gateway → orders → payments → 資料庫`，端到端 p99 是 800 毫秒，慢得使用者在抱怨。你打開四個服務各自的儀表板——每一個都顯示「我很健康」，各自的 p99 都在 250 毫秒以下。四個團隊在群組裡互相看著對方：不是我。這是個無解的羅生門，因為每一本「per-service」的帳，單獨看都是清白的。metric 告訴你整體 p99 是 800ms，卻切不開這 800ms 是怎麼分攤到四跳的；log 告訴你某台機器某條紀錄，卻不知道這條紀錄屬於哪個端到端請求的哪一段。缺的是一個能把「同一個請求在不同服務留下的痕跡」**串成一條因果鏈**的維度。

trace 補的正是這個維度。它把一個請求穿越多個服務的完整路徑，重建成一棵樹。樹上每個節點叫一個 **span**，代表一段有起止時間的工作——一次 RPC、一次 DB 查詢、一段內部計算。每個 span 記著自己的 id、指向上游的父 span id、起止時間戳，外加一些屬性（HTTP status、DB statement）。父子 span 的時間區間一層層嵌套——root span（gateway）的區間橫跨整條請求、把所有下游 span 都包在裡面——就還原出「誰呼叫了誰、各自花了多久」。把上面那條慢請求攤開，下面這張圖標的是每個服務**自己花的時間**（self-time，扣掉等下游回來的等待），一目了然：

```
trace（端到端 ≈ 800 ms）
  gateway        [== 20 ms ==]
    orders          [== 30 ms ==]
      payments         [============ 620 ms ============]   <-- 瓶頸
        db                 [== 80 ms ==]
```

四段的 self-time 加起來約 750ms，和 root span 涵蓋的端到端 800ms 差的那 50ms，落在跨服務的網路傳輸與序列化等沒被任何一段 span 計入的縫隙裡。但重點已經喊出來了：瓶頸在 payments——它對某個外部 API 的呼叫沒設好逾時，一個請求在那裡乾等。沒有 trace，這個結論是四個團隊吵一下午都吵不出來的；有了 trace，它是一張圖。

讓這一切成立的核心機制，是一個容易被忽略、卻一斷全毀的東西：**context 傳播（context propagation）。** 每個服務在往下游發請求時，必須把這個請求的 `trace_id` 和當前的 span id **隨請求一起傳下去**——通常塞在 HTTP header 裡（W3C 把這個格式標準化成 `traceparent` header）。下游服務收到後，用這個 trace_id 開自己的 span、把 parent 指回上游。所有服務共用同一個 trace_id，後端才能把散落各處的 span 認回同一棵樹。

這裡藏著 trace 最脆弱的故障模式：**傳播鏈只要斷一環，那一跳之後的因果就全丟了。** 某個老服務沒轉發 `traceparent`，或者請求跨過一個非同步邊界——丟進訊息佇列、稍後被另一個服務消費——而沒有人把 trace context 塞進訊息的 metadata，那麼下游產生的 span 就成了找不到父親的孤兒，拼不回原來那棵樹。你會得到一堆孤立的 span，看起來有資料，實際上拼不成任何一條完整的鏈。跨非同步邊界的傳播是最常被漏掉的一環，因為它不像 HTTP 呼叫那樣有現成的 header 自動帶過去，得手動把 context 塞進訊息裡再從另一端取出來。

trace 也幾乎一定要取樣——全量留存每一條請求路徑太貴。而**怎麼取樣**藏著一個比 log 取樣更要命的決策。最便宜的做法是 **head-based sampling**：在請求一開始、root span 剛產生時就擲骰子決定「這整條 trace 留不留」（比如留 10%）。無狀態、低延遲、成本固定——但它有個致命盲區：**錯誤和超慢的請求只占極少數，隨機取樣大機率正好把它們丟掉**，而那恰恰是你唯一想看的。你發現某筆出了錯想回頭追，它的 span 在請求一開始就被決定不記錄了，根本不存在。要精準留住「壞的」trace，得用 **tail-based sampling**：等整條 trace 跑完、看到全貌（有沒有錯、夠不夠慢）再決定留不留，於是可以下「凡是慢於 500ms 或含錯誤的全留、正常的只留 10%」這種規則。代價是後端必須緩衝一條 trace 的所有 span 直到它完成，而且同一條 trace 的 span 得被路由到同一個收集器實例才湊得齊——記憶體與架構成本都高得多。trace 這套機制——span 樹、context 傳播的細節、取樣策略的完整取捨——深講還有很多層，是另一個主題的份內事（見〈分散式追蹤：trace context 傳播、span 與取樣〉）；這裡要帶走的是它在三本柱裡的定位：**它是唯一補上「跨服務因果」這個維度的訊號**，而它的命脈是那條一斷全毀的傳播鏈。

## 1+1+1 為什麼大於 3：同一個事件的三種投影

到這裡，三本柱看起來像三個各管一段、互不相干的工具。但它們最有威力的用法，恰恰在於它們**不是**三個獨立的東西——它們是**同一個事件的三種投影**。

回到那個結帳請求。它穿過系統時，可以**同時**讓 `http_requests_total` 計數器 +1（metric）、寫下一行帶完整欄位的 access log（log）、產生一個 root span 並沿途長出子 span（trace）。這是同一次請求，被三個不同的鏡頭從三個角度拍下來：metric 拍下它對「整體趨勢」的貢獻、log 拍下它「自己是什麼」、trace 拍下它「怎麼穿過系統」。

而把這三張照片**對應回同一次請求**的那把鑰匙，是一個共同的 id：`trace_id`。如果你的採集層在寫 log 時把當前的 trace_id 一起寫進去、在 metric 上用 **exemplar** 把某個 histogram 桶連到一條代表性的 trace_id，那麼排障的路徑就被打通成一條直線：你在儀表板上看到 p99 那個桶異常高（metric）→ 點開這個桶掛著的 exemplar，跳到一條真實的慢 trace（trace）→ 在 trace 裡看到 payments 那段最慢、抓出它的 trace_id → 拿這個 trace_id 去 log 系統，撈出 payments 在那次呼叫裡寫下的所有 log，看到確切的 error message。一個 id，串起三本柱，把「有事」一路追到「是這件事」。

反過來，如果三本柱是三個各自為政的專案、沒有植入這個共同的 id，那你排障時就只能在三個 UI 之間靠時間戳人肉猜對應——「metric 上 02:13 有個尖峰，我去 log 裡撈 02:13 前後一分鐘的東西，看看哪條像」。能不能對上全憑運氣，而高流量下同一秒有成千上萬條 log，這種「靠時間戳猜」基本上等於白建。三本柱真正的價值不在於各自存在，而在於它們被一條 trace_id 縫成一個整體——這也是為什麼 2026 年的事實標準 OpenTelemetry 把三者收進**同一個採集層、同一套語意慣例**，讓共同的 context 從一開始就被植入，而不是事後才想辦法對齊（OpenTelemetry 的 traces 規格早在 2021 年就 stable，metrics 資料模型也已 stable，logs 在 2024–2026 間陸續穩定；持續效能剖析 profiling 作為「第四訊號」在 2026 年 3 月進入 public alpha、後續才往 Beta／GA 走，但三本柱的格局是穩固的地基）。

值得一提的是，三本柱回答的「現在系統健康嗎」這個問題，業界也淬煉出了固定的少數幾個訊號，讓你不必每次從零想要量什麼。Google SRE 的**四個黃金訊號（four golden signals）**——延遲（latency）、流量（traffic）、錯誤（errors）、飽和度（saturation）——主張如果一個面向使用者的服務你只能盯四個 metric，就盯這四個，因為它們足以涵蓋幾乎所有有意義的故障模式。Tom Wilkie 在 2015 年從中提煉出更適合請求驅動服務的 **RED 方法**——Rate（每秒請求數）、Errors（每秒失敗數）、Duration（請求耗時分布）——把「該為每個服務埋哪幾條 metric」收斂成一張極簡的清單。這些不是 metric 的替代品，而是「先量這幾個準沒錯」的起手式，省掉你在無窮多可埋的指標裡迷路。（至於怎麼把這些訊號變成可談判的可靠性目標、燒多快該叫醒人，那是 SLI 與 error budget 的份內事，見〈SLI、SLO 與 error budget：p99 與 p999〉。）

## 為什麼是這個形狀

退一步看，三本柱不是某個委員會拍腦袋分的三類，它是「資訊—成本反比」這條約束被逼到牆角後，唯一站得住的形狀。

你想要無損的細節，就得付 log 那份隨流量暴漲、還會在事故時靜默丟失的帳單。你想要可忽略的成本，就得接受 metric 在聚合那一刻把個別事件的身分永久丟掉。這兩個是光譜的兩端，互相到不了對方那裡——所以你需要兩種訊號，一種便宜而粗、一種昂貴而細，分別守「持續在看趨勢」和「事後查單筆真相」這兩件事。而當系統從單體長成跨服務的網狀，又冒出一個前兩者都拍不到的維度——「同一個請求在不同服務之間的因果順序」——於是需要第三種訊號 trace，用一條 trace_id 把散落的痕跡串成一棵樹，代價是它得靠一條一斷全毀的傳播鏈、還幾乎一定得取樣。

所以是三個，不是一個，也不是五個。三個，因為「資訊—成本」這條軸上有三個工程上真正划算的駐點，加上「跨服務因果」這個正交的維度。它們各自承認了一個無法迴避的取捨：metric 用失真換便宜，log 用昂貴換無損，trace 用脆弱的傳播鏈換跨服務的因果。下次你盯著那條凌晨兩點往上爬的延遲曲線，你會知道它只是三張投影裡的一張——而真正的故事，要靠那條把三張投影縫在一起的 trace_id，才講得完整。
