# API gateway 的角色：authz、限流、轉換與聚合

你拆了一個單體。原本一支應用程式裡的訂單、使用者、商品、付款，現在是四個各自部署、各自擴展的服務。架構圖畫得很漂亮——直到你坐下來寫第一個對外端點，才發現一個沒人在白板上提的問題：**驗 token 的程式碼，現在該放哪？**

放進訂單服務，它就得會解 JWT、會抓 JWKS、會處理金鑰輪替、會判斷 token 過期。然後使用者服務也要一份一樣的。商品服務也要。付款服務更要。四份幾乎一模一樣的驗簽邏輯，散在四個 repo、四個部署管線裡。某天簽發金鑰的端點換了位址，你要改四個地方、發四次版、祈禱沒漏。再加上限流——每個服務各自數「這個 client 這秒打了幾次」，但 client 的請求其實是打散到四個服務的，沒有任何一個服務看得到全貌，於是「每秒 100 次」這個限制根本算不準。再加上 TLS 終結、CORS、請求日誌、追蹤 header 的注入……

這些東西有個共同的名字：**橫切關注（cross-cutting concerns）**。它們不屬於任何一個服務的業務邏輯，卻是每一個服務都躲不掉的。把它們複製到每個服務裡，你買到的是重複、漂移、和一個「改一處要動 N 處」的維護地獄。API gateway 這層存在的全部理由，就是把這些橫切關注從 N 份收斂回一份——擺在所有外部流量都必經的那個入口上，做掉一次。

這一章要把這層薄薄的反向代理講透：它在邊界上到底做了什麼、每一件事的保證邊界在哪、以及最關鍵的——它在哪些地方會悄悄騙你，讓你以為某件事被守住了，其實沒有。

## 它不是負載均衡器，雖然長得很像

先把一個容易混淆的東西分開。閘道坐的位置和負載均衡器（見〈負載均衡：L4 與 L7、演算法與健康檢查〉）幾乎重疊——都在 client 和後端之間、都做 L7 路由、都終結 TLS。差別在它們**關心請求的哪一層**。

負載均衡器關心的是「把這個請求送到哪台健康的機器」。它看 host、看 path，挑一個 target，轉過去。它對請求的**內容**基本無感——header 裡有沒有合法 token、這個 client 這分鐘打了幾次、回應的欄位該不該重塑，這些都不在它的職責裡。

閘道關心的是請求的**語意與身分**。同一個 `POST /orders`，閘道要問：這個 token 是真的嗎？這個 API key 還有額度嗎？這個欄位要不要過濾掉？要不要先去問三個後端再把答案拼起來？它做的是「**內容感知（content-aware）的決策**」，不只是「轉發」。

這條界線在實作上常常糊掉——AWS 的 ALB 是個 L7 負載均衡器，但它也能掛 OIDC 認證；很多閘道底層就是一個負載均衡器加上一堆 plugin。但理解這層職責分工很重要，因為它解釋了後面一個反覆出現的取捨：**閘道每多扛一件事，它就多一分變成瓶頸與單點的風險。** 負載均衡器之所以能做得又快又薄，正是因為它什麼都不問；閘道每問一個問題，都要在延遲和可靠性上付帳。

## authz：邊界驗一次，但只驗得了一半

閘道的第一份工作，也是最值得拿來解剖的，是認證與授權。把它講透，你會發現它買到的保證比直覺以為的小一圈。

典型的做法是這樣。client 帶著一個 JWT 打進來，閘道在邊界把它攔下，做三件事：驗簽（這個 token 真的是我們的授權伺服器簽的嗎，用 JWKS 公鑰驗）、驗時效（`exp` 過了沒）、驗 claim（`iss`、`aud` 對不對）。三關都過，閘道才放行。後端服務收到的，是一個**已經驗過身分**的請求——它不必再碰一次 JWT、不必認得密碼學、不必知道 JWKS 端點在哪。

關鍵的工程手法在這裡：閘道驗完之後，會把它解出來的身分**注入成一個內部 header**，比如 `X-User-Id: 4821`、`X-Tenant: acme`、`X-Roles: buyer`，然後才把請求轉給後端。後端不再去解那個又長又難的 JWT，它只讀這個簡單的 header，並且**信任**這個 header——因為它知道，能到它面前的請求，一定先穿過了閘道。

```
                  +-----------------------+
   Authorization: |  API gateway          |   X-User-Id: 4821
   Bearer eyJ...  |  - verify signature   |   X-Tenant: acme
   ------------>  |  - check exp / claims |   ----------------->  backend
   (external)     |  - inject identity    |   (internal, trusted)  (reads header,
                  +-----------------------+                         no crypto)
```

這個設計很漂亮，但它藏著一個必須講死的安全前提，也是這套機制最容易被戳破的地方：**後端對 `X-User-Id` 的信任，完全建立在「外部請求進不來、只能經閘道」這個網路假設上。** 如果有人能繞過閘道、直接打到後端的內網位址，他只要自己捏一個 `X-User-Id: 1`（管理員）的 header，後端就會乖乖把他當管理員。閘道做的不是「讓後端變安全」，而是「把驗證的責任集中到一個點」——而這個集中要成立，後端就**絕不能**接受來自閘道以外的流量。實務上這靠網路隔離（後端只在私有網段、不對公網開放）、或者更嚴格的 mTLS（後端只認閘道的客戶端憑證，見〈TLS 與 mTLS：握手與憑證管理〉）來兜住。少了這道，整個「邊界驗一次」的模型就是紙糊的。

更要命的是第二件事：**閘道驗的是「你是誰」（authentication），不是「你能不能做這件事」（fine-grained authorization）。** 這兩個常被混為一談，但它們的歸屬天差地別。

閘道能做的授權是**粗粒度**的：這個 path 需不需要登入、這個 role 能不能碰 `/admin/*` 這一段。這些規則只看請求本身就能判。但「使用者 4821 能不能修改**訂單 9 號**」——這個問題閘道答不了，因為答案藏在資料裡：訂單 9 號是不是他下的？他的方案允不允許改已出貨的單？這是**業務規則**，只有握著資料的後端服務才知道。把這種細粒度授權外包給閘道，等於要閘道去查訂單資料庫、懂訂單的業務邏輯——它一旦開始做這個，就不再是薄薄的橫切層，而是把業務邏輯吸進了基礎設施。

所以正確的分工是：**閘道擋掉「沒登入」和「明顯越權」的請求，後端負責「這一筆資料這個人到底能不能動」。** 閘道是第一道粗篩，不是唯一一道。任何「閘道驗過了，後端就不用管授權」的系統，都在等一個越權漏洞——某個登入了的普通使用者，帶著合法 token，去改了不屬於他的那筆訂單，而後端因為「反正閘道驗過了」，連問都沒問。

## 限流：閘道是最好的位置，也是最容易失準的位置

第二份工作是限流。為什麼是閘道做？答案在開場那個算不準的「每秒 100 次」——因為閘道是**唯一一個看得到單一 client 全部流量的點**。請求進來時還沒扇散到各個後端，所有的計數都能在這一個位置匯總。把限流放在這，是讓它站在能看到全貌的地方。

但「閘道適合做限流」和「閘道做的限流是準的」是兩回事，這中間的縫隙，是這一章最該講透的非顯然處。限流的演算法本身（token bucket、leaky bucket、分散式計數怎麼同步）是另一個主題的核心（見〈rate limiting：token bucket、leaky bucket 與分散式限流〉），這裡只看「放在閘道層」這個位置帶來的、演算法層面以下的兩個坑。

**第一個坑：閘道自己是多副本的。** 閘道既然在關鍵路徑上，就絕不能是單台——它一定有好幾個副本在跑（理由下一節講）。問題來了：如果「每個 API key 每秒 100 次」這個計數是每個副本**各記各的**，而你有 4 個副本、流量被負載均衡平均打散，那實際允許的就是 4 × 100 = 400 次/秒——你的限制被悄悄放大了 4 倍，而且副本數一變，這個倍率還會跟著變。要讓計數準，多個副本必須共享一個計數狀態（通常丟進一個 Redis 之類的集中計數器，見〈Redis 的內部：單執行緒、I/O threads、持久化與多重角色〉），但這又給每個請求加了一跳網路、一個新的故障點——共享計數器掛了，限流要嘛全放（fail-open，後端裸奔）、要嘛全擋（fail-closed，自己造成停機）。「限流到底放在哪一層、用多準的計數」本身就是一個取捨，沒有免費的準確。

**第二個坑：限流擋的是「進來的速率」，不是「進來之後造成的負載」。** 閘道能擋住「這個 client 每秒打太多」，但它擋不住**已經放進去的那些請求在後端慢慢拖垮系統**。設想一個請求合法、不超速，但它觸發了一個沒加索引的查詢，在後端跑了 30 秒。閘道對此一無所知——它在請求進來那一刻就放行了，後面那 30 秒它看不到。限流管的是入口的水龍頭，管不了水進去之後在管路裡淤積。要兜住「慢請求拖垮後端」，得靠後端自己的逾時、艙壁隔離（見〈timeout、deadline 與 budget：把等待變成有界〉〈bulkhead：艙壁隔離〉）。把這兩件事混為一談——以為「閘道做了限流，後端就安全了」——是一個會讓你在某次慢查詢風暴裡措手不及的錯覺。

值得手算一遍的是雲端託管閘道的限流上限到底意味著什麼。以 AWS API Gateway 的 REST API 為例，它的帳號層預設限流（2026-06）是：**每個 Region、每秒 10,000 次（steady-state）、burst 5,000**，用 token bucket 實作，超過回 `429 Too Many Requests`（部分較新 Region 如 Cape Town、Milan、Jakarta 等預設較低，約 2,500 RPS / 1,250 burst）。看起來很寬。陷阱在「帳號層、該 Region **所有** REST API 合計」這幾個字。

假設你的 API 上線初期只有 200 RPS，離 10,000 遠得很，你睡得很安穩。但同一個帳號同一個 Region 裡，另一個團隊跑了一個批次作業，平時不顯眼，某個瞬間卻尖到一萬多 RPS。token bucket 是一個共用的桶：穩態每秒補 10,000 個 token，但桶的容量（burst）只有 5,000——那一下尖峰把桶裡的 token 一口氣抽乾，補的速度跟不上抽的速度，桶見底了。等你那個只有 200 RPS、什麼都沒做錯的 API 在這一瞬要拿 token，桶裡已經沒有了，於是你開始莫名其妙地吃 `429`。這不是 bug，是「共用一個帳號層 token bucket」的必然後果：吃虧的不是「平均流量最大」的人，而是「在桶被別人抽乾的那一瞬剛好來要 token」的人。正解是別只靠帳號層預設：在 stage 或 method 設 per-stage 限流、或對第三方用 usage plan 設 per-key 限制，把不同 API 的流量在計數上**彼此隔離**——這正是艙壁隔離的思想用在限流配額上。AWS 官方也明說這些數字是 **best-effort 的目標值、不是保證上限**：真實的觸發點會在你設的值附近浮動，不會卡在精確的那一刀。

順帶看一個更隱蔽的限流時間語意。如果你用 AWS WAF 的 rate-based rule 在閘道前擋 per-IP 的洪水（ALB 沒有原生限流，要靠 WAF 補這一塊），它的計數窗口（2026-06）可設 60／120／300／600 秒，預設 **300 秒（5 分鐘）的滑動窗口**。這裡有個容易踩的非顯然行為：WAF 的演算法**給較近的請求更高權重**、而且**不保證精確卡在你設的閾值**，從超速發生到 WAF 真的開始擋，**可能慢上好幾分鐘**。也就是說，這種限流是給「持續性濫用」設計的緩坡防線，不是給「瞬間尖峰」設計的硬牆——你拿它擋一個跑了三分鐘的攻擊綽綽有餘，拿它擋一個 2 秒的突刺則幾乎來不及反應。把限流的「時間粒度」和你要擋的「威脅的時間尺度」對齊，是個常被忽略卻會決定成敗的細節。

## 轉換：把後端的方言翻成對外的標準語

第三份工作是轉換（transformation）。閘道可以在請求和回應流經它時改寫內容：注入或剝除 header、把對外的 REST 翻成對內的 gRPC、過濾掉後端回應裡不該外洩的內部欄位、把多個後端各自的錯誤格式統一成一種對外的錯誤結構。

這件事的價值在於**解耦對外契約與後端實作**。後端服務可以用它最順手的協定和資料形狀（內部 gRPC、欄位命名隨團隊習慣），而對外永遠呈現一套乾淨、一致、穩定的 API。後端重構、換協定、改欄位名，只要閘道的轉換層補一刀，對外契約可以紋風不動。前面 authz 那段的 header 注入，其實就是轉換的一個特例——把驗出來的身分塞成下游能讀的形狀。

但轉換是閘道功能裡**最該節制**的一項，因為它最容易讓閘道從「薄薄的橫切層」膨脹成「胖閘道」。一旦你開始在閘道裡寫複雜的欄位映射、條件分支、資料格式的深度改寫，業務邏輯就一點一點滲進了基礎設施。閘道本該是所有團隊共用、很少改動的東西；當它塞滿了各個服務專屬的轉換規則，它就變成了**每個團隊都要改、每次改都要協調、改錯就全掛**的瓶頸——一個披著微服務外皮的單體，這個反模式常被叫做「分散式單體（distributed monolith）」。判準很簡單：閘道做的轉換，應該是**與業務無關的、機械式的**（協定翻譯、header 整形、欄位白名單）；只要某個轉換需要懂業務語意、需要分支判斷「這種訂單要這樣轉、那種要那樣轉」，它就該退回後端去做。

## 聚合：把一次外部請求扇成多次內部呼叫

第四份工作是聚合（aggregation），也是閘道能力的上限所在。

場景很具體。一個行動 App 的訂單詳情頁，要顯示「訂單本身 + 買家名字 + 三個品項的縮圖」。在純粹的微服務切分下，這四塊資料分屬訂單、使用者、商品三個服務。如果讓 client 自己去湊，它得依序打 `GET /orders/1`、`GET /users/9`、再三個 `GET /products/{id}`——五次往返。在行動網路 200 ms RTT 下、若這些請求有先後依賴（先拿到訂單才知道要查哪些商品），使用者要乾等近一秒看著轉圈圈。

聚合的做法：client 只打**一次** `GET /order-detail/1` 到閘道，閘道在它和後端之間那段又快又近的內網裡，把這次請求**扇出**成多個後端呼叫、等它們回來、把結果拼成一個回應，一次還給 client。

```
   client                gateway                    backends
     |   GET /order-detail/1  |
     | ---------------------> |
     |                        | --> GET /orders/1        (1 ms)
     |                        | <-- 拿到 user_id、品項 id
     |                        | --> GET /users/9         (1 ms, 並行)
     |                        | --> GET /products/...    (1 ms, 並行)
     |                        | <-- 合併三份回應
     |   一份聚合回應         |
     | <--------------------- |
```

注意那條依賴：閘道得先拿到訂單、才知道買家是誰、要查哪三個品項，所以它不是三個呼叫一次全發，而是**先打訂單、再用拿到的 id 並行打使用者與商品**——兩段內網往返。省下的延遲可以算出來。原本同一條依賴攤在公網上：client 先打一次拿訂單（200 ms），再打使用者與三個商品；就算它夠聰明、把這後四個並行（多數行動端其實是一個 `await` 接一個序列發），也得再吃一次 200 ms，總共約 400 ms；若寫成全序列，五次往返累積到約 1,000 ms。改成閘道聚合後，client 只付一次 200 ms 的往返；閘道到後端那兩段走的是內網（單程約 1 ms），各取最慢的一條，合計約 3~4 ms。client 感受到的總延遲掉到約 200 ms——**省下來的不是計算，而是公網往返的次數**。聚合的全部價值，就是把「N 次昂貴的公網往返」換成「1 次公網往返 + 幾段便宜的內網往返」。

但聚合一旦做進閘道，就帶出兩個必須正視的後果。

第一，**它把扇出失敗的處理也搬到了閘道**。三個後端呼叫，萬一商品服務超時、使用者服務回 500，閘道該怎麼辦？整個回 500，還是回一個「訂單和買家有了、縮圖暫缺」的部分結果？這是個降級決策（見〈fallback 與降級：壞掉時退到哪〉），而它現在是閘道的責任。閘道做聚合，就繼承了「協調多個可能各自失敗的呼叫」這整套複雜度——它不再只是轉發，而是成了一個小小的編排器。

第二，**通用閘道做聚合，做久了會四分五裂**。Web 端要的欄位、iOS 要的、Android 要的，往往不一樣；每加一種前端、每改一次頁面，聚合邏輯就要動。把所有前端的聚合需求全塞進一個通用閘道，它會被各前端的特殊需求撕扯。這正是 **BFF（Backend for Frontend）** 模式登場的地方——這個名稱由 SoundCloud 的 Phil Calçado 命名（約 2013 年的實踐），後經 Sam Newman 撰文推廣而廣為人知：與其用一個通用閘道伺候所有前端，不如**為每一種前端各設一個專屬的聚合層**，讓它只貼合那一種前端的需求、由那個前端團隊自己擁有。聚合這個職責，於是從「通用閘道的一個功能」被拆成「每種前端一個專屬後端」。一個系統裡常常是兩者並存：一個通用閘道管 authz、限流、TLS 這些真正橫切、人人都需要的事；幾個 BFF 各自管自己前端的聚合與裁剪。

## 為什麼這層東西本身必須比它保護的後端更可靠

把四份工作講完，一個尾大不掉的後果浮上來，必須收尾講清楚：**閘道在所有外部流量的關鍵路徑上，它掛了，全系統就掛了。**

這是閘道整個架構選擇裡最沉的那塊取捨。你把橫切關注集中到一個點，買到了一致性與不重複——代價是這個點成了**單一失敗點（single point of failure）**。後端服務還能各自有各自的故障域，一個掛了另一個還活著；但閘道一旦倒下，再健康的後端也收不到任何請求。它的可用性，是整個系統可用性的天花板。

所以閘道有一條鐵律：**它必須比它所保護的任何後端都更可靠。** 這推導出幾個非選不可的設計：它一定是多副本的（前面限流計數的麻煩正源於此——可靠性逼出多副本，多副本又逼出共享計數）；它前面要有健康檢查把掛掉的副本摘掉（見〈health check：liveness 與 readiness〉）；它自己要刻意保持**薄**——邏輯越少、出錯面越小、越不容易因為某次業務改動而崩。前面反覆出現的那條告誡——別把業務邏輯、複雜轉換、重編排堆進閘道——到這裡才顯出它真正的份量：那不只是整潔的潔癖，而是因為**閘道每胖一分，整個系統的可用性下限就低一分**。一個塞滿邏輯、人人都在改的閘道，會用它自己的高故障率，拖垮它本來要保護的所有後端。

退一步看，閘道的整個形狀，都是從一個簡單的張力裡長出來的：**橫切關注既不該散在每個服務裡（會重複、會漂移），也不該全堆在一個點上（會變胖、會變脆）。** authz、限流、轉換、聚合之所以被收上來，是因為它們確實橫切、確實該收;而它們之所以被要求「只收機械式的、與業務無關的那一層」，是因為那個收上來的點付不起變胖的代價。閘道是這條張力上找到的平衡：**收得夠多，讓後端能專注業務、讓治理能一致；又收得夠克制，讓自己薄到撐得起「比所有後端都可靠」這個承諾。** 它在邊界上替你驗了身分、擋了洪水、翻譯了方言、拼好了答案——但你得始終記得，它驗的身分只到「你是誰」、擋的洪水只到「進來的速率」、而它自己，是你整個系統裡最不能倒的那一塊。
