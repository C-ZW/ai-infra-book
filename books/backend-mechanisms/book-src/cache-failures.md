# 快取的失效災難：雪崩、穿透、擊穿

設想一個平常得不能再平常的系統：一個讀多寫少的商品服務，前面擺了一層 Redis 當快取，後面是一台 PostgreSQL。命中率長期穩在 98%，DB 的 QPS 看起來總是低低的、貼著地面爬，連報表都懶得看。團隊對這套配置很滿意——直到某個整點，監控圖上 DB 的連線數在三秒內從 200 衝到 4,000，CPU 打滿，慢查詢堆積，接著是一連串 `connection pool exhausted`，然後整個服務 503。

奇怪的是，這段時間進來的請求量並沒有暴增。流量曲線平得跟平常一樣。打爆 DB 的不是「更多請求」，而是**同一批請求突然全部變成了 cache miss**。快取這層東西，平時把 98% 的讀擋在身前，讓你幾乎忘了它的存在；可一旦它在某個瞬間集體失手，那 98% 會在同一刻全部漏到後端——而後端的容量，是按「只承接 2%」設計的。

這就是快取失效災難的本質。快取的省力，全部建立在「命中」之上；而命中率不是一條平滑的線，它會在某些特定時刻**塌陷**。這一章要拆的，就是命中率塌陷的三種典型形狀——雪崩、穿透、擊穿——它們長得像、後果像（都是 DB 被打爆），但**根因完全不同，因此解藥也完全不同**。把它們當成同一種病來醫，是這個領域最常見、也最致命的誤判：對著穿透加 TTL 抖動沒有任何用，對著擊穿擺 Bloom filter 也擋不住。要對症，得先看清三者各自是怎麼壞的。

## 三個共通的脊椎，三條不同的根因

先把三種災難擺在一起，用最精煉的一句話釘住各自的根因，後面再逐一展開：

- **雪崩（avalanche）**：大量不同的鍵在**同一個時刻**集體失效（或快取整個重啟，記憶體清空），於是大批本來互不相干的請求同時 miss，洪水般打到 DB。根因是**失效時間的集中**。
- **穿透（penetration）**：被查的鍵**在資料來源裡根本不存在**，於是快取永遠是 miss——查一次 miss 一次、查一次打一次 DB，快取形同虛設。根因是**「不存在」這個答案沒有被快取下來**。
- **擊穿（breakdown / hotspot invalid）**：**單一一個熱門鍵**在過期的那一瞬間，大量並發請求同時撞上它的 miss，於是它們一窩蜂全擠去 DB 重建同一份資料。根因是**熱點 ＋ 同時重建**。

把這三句話再壓縮，會看到一條共通的脊椎：**在快取失效的那一刻，後端會不會、要不要被流量擊垮。** 失效之所以是快取裡最難的部分，正因為它同時牽涉三個維度——時間（什麼時候過期）、空間（哪些鍵過期）、並發（有多少請求同時撞上同一次失效）。三種災難，不過是這三個維度上的三種極端組合。

接下來逐一進深水區。

## 雪崩：當所有鍵約好同一秒死

雪崩最容易在一種看似無害的操作裡埋下：**批次預熱**。

想像一個很合理的做法。每天凌晨兩點，一個排程任務把當天會用到的熱門商品全部預先載入快取，每一筆都設一個整齊的 `TTL = 3600`（一小時）。任務跑得很快，幾秒鐘把十萬個鍵刷進 Redis。系統一整天都很順——直到第二天，當你發現災難總是發生在某些**特定的整點**。

原因藏在那個「整齊」裡。所有鍵都在凌晨兩點零幾秒被寫入，又都帶著一模一樣的一小時 TTL，於是它們會在凌晨三點零幾秒**同一個瞬間集體到期**。那一刻，十萬個鍵在 Redis 裡同時消失，十萬筆請求同時 miss，十萬個查詢在同一秒撲向 DB——而且這不是一次性的，它會在每個整點週期性地復發，因為回填時你又給了它們同樣整齊的 TTL，於是下一輪的集體死亡時刻又被你親手對齊了。

看清根因，解法就幾乎是自明的：**不要讓鍵的失效時間對齊。** 給每個 TTL 加一道隨機抖動（jitter），把「同一秒到期」攤平成「在一段視窗內陸續到期」：

```
ttl = base + random(0, spread)
# e.g. base = 3600s, spread = 600s
# → 每個鍵在 3600~4200s 之間隨機某點過期
```

這一道抖動把原本擠在一個時間點的失效尖峰，攤成了一段十分鐘的緩坡。原本一秒鐘十萬筆 miss，現在變成每秒約 167 筆，DB 完全感覺不到。代價幾乎為零——只是有一小部分鍵會比理論上「該活的時間」早幾分鐘失效，犧牲掉一點點命中率，換來尖峰被削平。

但 `spread` 這個參數本身藏著一個非顯然的權衡，值得停下來看。`spread` 太小，攤平的窗口太窄，尖峰削得不夠，雪崩只是被推遲、沒被解決；`spread` 太大，又會有相當比例的鍵過早失效，整體命中率明顯下滑。一個常被忽略的細節是：**抖動只能是正向的偏移、或圍繞基準的雙向偏移，但不能讓 TTL 有機會抖成接近零或負值**——若你寫成 `base + random(-spread, spread)` 而 `spread` 接近 `base`，某些鍵會一寫進去幾乎立刻過期，反而製造出大量無謂的 miss。抖動的方向與幅度，得讓最短 TTL 仍遠大於一次重建的耗時。

雪崩還有一種更兇、抖動救不了的版本：**快取節點整個重啟或被清空**。這不是「鍵陸續到期」，而是「所有鍵在同一個物理時刻一起消失」——記憶體歸零，命中率瞬間從 98% 掉到 0%，全部流量直撲 DB。TTL 抖動對這種情況毫無幫助，因為問題不在過期時間的分布，而在快取本身的可用性。這時要靠的是另外兩件事：一是**重啟後的預熱**（warmup，重建前先把熱資料灌回快取，而不是讓線上流量去填一個空快取），二是**快取掛掉時後端要有保護**——後面講擊穿時會看到的 single-flight，以及對 DB 本身的限流，是這裡最後的防線。換句話說，「快取一掛、流量直撲 DB」這條路徑，本質上就是一場被觸發的雪崩，而能兜住它的，是讓後端不至於在那一瞬間被打穿的機制（這類後端保護的整體設計，見〈快取策略〉裡的降級路徑，以及 owning 它的〈rate limiting：token bucket、leaky bucket 與分散式限流〉）。

## 穿透：快取永遠攔不住一個不存在的鍵

雪崩是「太多鍵同時失效」，穿透則是相反的病理：**一個鍵根本進不了快取，因為它對應的資料壓根不存在。**

機制上極其簡單，也正因此極其陰險。標準的 cache-aside 讀路徑是這樣的：先查快取，miss 就去 DB 撈，撈到了就回填快取、設 TTL，下次就命中了。但如果這個鍵在 DB 裡**查無此物**呢？DB 回一個「找不到」，於是你**沒有東西可以回填**——快取裡仍然是空的。下一個查同一個鍵的請求，又是 miss，又去問 DB，又得到「找不到」，又沒得回填。這個鍵就像快取上的一個洞，每一次查詢都直接穿過快取、命中 DB，無論你的命中率機制設計得多漂亮，它都繞過了你全部的防線。

正常使用下，這個洞可能無傷大雅——沒人會反覆查一個不存在的商品。但只要把場景換成**對外開放、且 ID 可以被枚舉的端點**，它立刻變成攻擊面。一個惡意腳本拿著 `GET /product/{id}`，從 `id=1` 一路掃到 `id=99999999`，其中絕大多數是不存在的 ID。每一個不存在的 ID 都是一次穿透，每一次穿透都是一發直擊 DB 的子彈。攻擊者不需要多大的流量，就能讓你的快取層形同不存在、讓 DB 獨自承受全部掃描壓力。

要堵這個洞，得回到根因：**問題是「不存在」這個答案從沒被快取下來。** 那就把它快取下來。兩條主要的路：

**第一條，快取空值（null caching）。** 當 DB 回「找不到」時，仍然往快取裡寫一個特殊的「空」標記，設一個**短** TTL。下次再查同一個不存在的鍵，命中快取、直接回「不存在」，不再打 DB。這條路簡單、直接，但那個「短 TTL」是有講究的——它必須短，否則會撞上一個討厭的邊界：**如果這個鍵之後真的被新增了呢？** 假設你給空值設了一小時 TTL，而某人在 10:00 查了一個還不存在的鍵（快取寫入空值，有效到 11:00），結果 10:05 這筆資料真的被建立了——接下來的 55 分鐘裡，所有查這個鍵的人都會被那個過期的空值騙，明明資料已經存在，卻一直被告知「查無此物」。空值 TTL 越長，這個「資料已存在卻查不到」的不一致窗口就越長。所以空值 TTL 通常設得很短（幾秒到幾十秒），只夠擋住短時間內的重複掃描，又不至於長到遮蔽真實的新增。一個更乾淨的補強是：在寫入路徑（真的新增該鍵時）主動刪掉對應的空值快取，把不一致窗口收到最小。

**第二條，前置一個 Bloom filter。** 這是穿透問題裡最漂亮的一招，值得把機制講透，因為它的保證有一個剛好對症的非對稱性。

Bloom filter 是一個機率性的集合資料結構，由 Burton Bloom 在 1970 年提出。它的構造簡單到可以一句話說完：一個長度為 `m` 的位元陣列（初始全 0），外加 `k` 個獨立的雜湊函數。要把一個元素「加入」集合，就用 `k` 個雜湊函數算出 `k` 個位置，把這些位置的 bit 全設成 1。要查詢一個元素「在不在」集合裡，同樣算出那 `k` 個位置，看它們**是不是全都是 1**：只要有任何一個是 0，就斷定「一定不在」；如果全是 1，就回答「可能在」。

關鍵就在這個「可能在」與「一定不在」的不對稱。因為加入元素只會把 bit 從 0 翻成 1、從不翻回去，所以一個真的在集合裡的元素，它對應的那 `k` 個位置必然都被設過 1——**Bloom filter 絕不會把一個存在的元素誤判為不存在**（沒有偽陰性）。但反過來，一個不在集合裡的元素，它的 `k` 個位置也可能恰好都被別的元素設成了 1，於是被誤判為「可能在」——這就是**偽陽性**，是 Bloom filter 唯一的錯誤類型，發生率由 `m`、`k` 和已存元素數量共同決定。

這個「無偽陰性、有偽陽性」的非對稱，正好是穿透問題要的形狀。把所有**真實存在的鍵**都灌進 Bloom filter，擺在快取前面當守門員：每個查詢先問 Bloom filter——

```
if not bloom.contains(key):
    return NOT_FOUND          # 100% 確定不存在，連 DB 都不必碰
else:
    # 可能存在（也可能是偽陽性），照常走 cache → DB
```

當 Bloom filter 說「不在」，那是**百分之百可靠**的——因為它沒有偽陰性，凡是它否定的，就絕對不存在，可以當場回 `NOT_FOUND`，連 DB 都不碰。攻擊者枚舉的那海量不存在 ID，絕大多數會被 Bloom filter 在第一關直接擋下。偶爾的偽陽性會放一個不存在的鍵穿過去、白查一次 DB，但這只是**效率的小損失**，不影響正確性——它從不會擋掉一個真實存在的鍵。空間效率也驚人：擋住一億個鍵的枚舉，一個偽陽性率約 1% 的 Bloom filter 在最佳參數下只需約 120 MB（要把偽陽性壓到 0.1% 也才約 170 MB），而且它從不儲存鍵的原文。

當然，天下沒有白吃的午餐，Bloom filter 自己也有脾氣。最棘手的是**刪除**：因為一個 bit 可能被多個元素共用，你不能在元素刪除時把那 `k` 個 bit 清 0——那會誤傷其他剛好共用該 bit 的元素，製造出真正的偽陰性（把存在的東西誤判為不存在），而偽陰性是 Bloom filter 絕不能有的破口。所以標準 Bloom filter 是「只進不出」的；要支援刪除，得換成 counting Bloom filter（每格存計數而非單一 bit）或 cuckoo filter 之類的變體。這也意味著，當你的鍵集合會增刪時，得有一套機制讓 filter 跟上——通常是定期重建，或用支援刪除的變體。

## 擊穿：一個熱鍵過期，一萬個請求同時去重建

雪崩是「很多鍵同時失效」，穿透是「鍵不存在」，擊穿則是第三種——**一個極熱的鍵，在它過期的那一個瞬間，被大量並發請求同時撞上。**

注意它和雪崩的分別：雪崩是**多個**鍵在同一刻失效，擊穿是**單一一個**鍵失效卻被千軍萬馬同時撞上。一個是橫向的廣度（很多鍵），一個是縱向的並發（一個鍵、極高 QPS）。混淆這兩者，就會開錯藥——對擊穿加 TTL 抖動沒有意義，因為問題從來不是「這個鍵在不對的時間過期」，而是「在它過期的那一瞬間，太多請求同時想重建它」。

把這個瞬間用真實數字算一遍，才能體會它有多兇。假設某個熱門商品頁的快取項過期了，而此刻針對它的讀取速率是 **5,000 QPS**——平均每毫秒約有 5 個請求進來。重建這份快取（去 DB 跑那串查詢、組裝、寫回 Redis）大約需要 **20 毫秒**。問題來了：在這 20 毫秒的「空窗」裡，會有多少請求撞進來？

```
空窗期 = 20 ms
到達速率 = 5,000 QPS = 5 req/ms
空窗內到達 = 20 ms × 5 req/ms = 100 個請求
```

100 個請求，在沒有任何保護的情況下，會**全部** miss、**全部**判定「快取沒有、我得去重建」、於是**全部**衝向 DB 去跑同一串查詢、重建同一份一模一樣的資料。本來穩態下 DB 每隔一陣子才被這個鍵打擾一次，現在在 20 毫秒內被同一個查詢砸了 100 次——相對穩態瞬間放大了約 100 倍。而這還只是一個鍵。若同時有幾個熱鍵一起過期，或這個鍵的 QPS 更高，放大倍數會更可怕。這就是擊穿：**穩態下毫不起眼的一個鍵，在失效的剎那把後端的負載放大兩三個數量級。**

看清「100 個請求重建的是同一份資料」這件事，解法就浮現了：**既然大家要的是同一份結果，那就只讓一個請求去做重建，其餘的等它、共享它的結果。** 這就是 single-flight（也叫 request coalescing、請求合併）：

```
on cache miss for key K:
    if acquire_lock(K):          # 只有一個請求搶到鎖
        value = rebuild_from_db(K)
        cache.set(K, value, ttl)
        release_lock(K)
        return value
    else:                        # 其餘請求沒搶到鎖
        wait_or_retry()          # 等一下、再讀一次快取
        return cache.get(K)      # 此時通常已被那一個請求填好
```

回到剛才的數字：加上 single-flight 後，那 100 個撞進空窗的請求裡，只有**第一個**搶到鎖、去 DB 重建；其餘 99 個發現鎖被佔著，就乖乖等待（或短暫退避後重讀快取），等那一個請求把結果填回快取，它們直接命中新值。**20 毫秒空窗內打到 DB 的查詢，從約 100 個降為 1 個。** 這是兩個數量級的削減，而且代價只是讓那 99 個請求多等了最多 20 毫秒——對它們而言，等 20 毫秒命中，遠勝於一起去把 DB 壓垮、然後大家一起超時。

但 single-flight 那把鎖，本身又是一個會反咬的東西，它的 TTL（鎖的超時）藏著一個雙向的陷阱，這正是把這個機制講透與只會喊「加個鎖」的分界。

**鎖的 TTL 設太長**：假設你給鎖設了 30 秒 TTL，而那個搶到鎖的請求在重建時卡住了（DB 慢查詢、網路抖動、甚至那個行程崩潰了）——在它釋放鎖之前，**所有其他請求都被擋在門外乾等**，等的不是 20 毫秒，而是整整 30 秒的鎖超時。你本來想擋住「驚群打爆 DB」，結果換來了「所有人一起卡 30 秒」，把一個 DB 過載問題變成了一個全面延遲爆炸問題。

**鎖的 TTL 設太短**：假設鎖只設 10 毫秒，而重建實際要 20 毫秒——那麼鎖會在重建還沒完成時就先過期。鎖一過期，等在外面的請求看到鎖沒了，又會有一個（或多個）衝進去搶鎖、重新觸發重建。**驚群再現**，你費勁加的鎖等於沒加。

所以鎖的 TTL 必須**穩穩地大於重建時間的合理上限、又不能大到在重建真的卡死時把大家鎖太久**——這是一個沒有完美解的張力。實務上常見的緩解是給鎖一個「夠重建用、但有上限」的 TTL，搭配「搶不到鎖的請求短暫退避後重讀快取，而不是死等鎖」，再加上：如果這個鍵真的熱到不容許任何一次直穿 DB，就根本不要讓它走「物理過期 → 大家撞 miss」這條路。

還有一個 pseudo-code 沒畫出來、卻一定要想清楚的分支：**那個搶到鎖的請求，重建本身失敗了怎麼辦？** 上面的碼假設 `rebuild_from_db(K)` 總會成功並寫回快取，但它完全可能拋例外、DB 回錯、或行程在 `cache.set` 之前就崩潰。這時鎖必須在 `finally` 裡釋放（崩潰路徑則靠鎖 TTL 過期兜底），否則鎖會懸著、把所有等待者一路卡到 TTL 才解套。更要命的是，等待者退避完重讀快取時，讀到的**仍是空的**——於是它們要嘛一起再撞一次 miss、重新觸發重建（驚群復發），要嘛 `cache.get(K)` 回 `None` 把錯誤直接吐給使用者。所以等待者的退避必須有**重試上限**，並明確定義「重試耗盡、快取仍空」時的回退：回一個稍舊的值、回明確錯誤、還是有限次數內讓下一個請求接手重建——三選一都行，但不能不選，否則一次 rebuild 失敗就讓 single-flight 整個退化回它本來要消滅的驚群。

這就帶出擊穿的另一條解法，思路和 single-flight 正交：**邏輯過期（logical expiration），讓熱鍵永遠不要在物理上過期。** 做法是快取項本身不設 TTL（值永遠在），但在值裡額外存一個「邏輯過期時間戳」。讀取時檢查這個戳：

- 還沒到邏輯過期時間 → 直接回快取值（命中，零延遲）。
- 已過邏輯過期時間 → **仍然回那個（稍舊的）快取值**，同時觸發一個背景刷新（或用 single-flight 只讓一個請求去刷），把新值寫回。

關鍵在於：因為鍵從不物理過期，**任何時刻讀取都有東西可回**，沒有「空窗」，也就沒有「大家撞進空窗一起去重建」的擊穿。代價是過期後的短暫期間，請求拿到的是稍微陳舊的值（最終一致），用「可控的舊」換「絕不直穿 DB」。對首頁、爆款商品、明星貼文這類熱到不能有任何一次穿透的鍵，這個取捨幾乎總是划算的。

值得一提的是，「讓請求各自隨機決定要不要提前刷新」這個思路有過嚴謹的最佳化研究：Vattani、Chierichetti 與 Lowenstein 在 2015 年 VLDB 的論文裡證明，讓每個請求在快取**物理過期之前**，以一個隨著逼近過期時間而上升的機率提前觸發重建，能在「儘量不過早重算」與「儘量不撞集體 miss」之間取得最佳平衡，而且指數分布是最優的隨機化方式。其判定式可以寫成這樣的形狀：

```
# 每次讀取時計算，成立則由本請求負責提前重建
now - delta * beta * ln(random()) >= expiry
#   delta : 重建一次的耗時（秒）
#   beta  : 調節積極程度的參數，1.0 是常見預設
#   random() ∈ (0, 1]
```

`delta`（重建越貴）越大、或時間越逼近 `expiry`，這個判定式成立的機率就越高，於是越可能有某個請求「自願」提前重建——而因為是隨機觸發、且大多數在物理過期前就完成，集體撞上同一次過期的機率被大幅壓低。這比硬鎖更平滑：沒有「搶鎖、等鎖、鎖超時」那一整套脆弱的協調，只靠每個請求各自擲一次骰子。

## 同一場病的近親：stale-while-revalidate

把擊穿的「邏輯過期」和雪崩的「快取掛掉退到舊值」放在一起看，會發現它們共享同一個更廣的模式——**用「可控的舊」換「可用性」**。這個模式在 HTTP 快取層有一個正式的名字：stale-while-revalidate，由 RFC 5861 定義（它是一份 Informational RFC，標題就叫〈HTTP Cache-Control Extensions for Stale Content〉，同時也定義了 stale-if-error）。

語意和應用層的邏輯過期幾乎一模一樣：快取項過期後，在一段指定的窗口內，快取**仍然立刻回那個過期的值**給請求者，同時在背景非同步去源站重新驗證、取回新值。請求者完全不必等待重建，所以既沒有 miss 風暴、也沒有重建延遲——尖峰被「先回舊值、背景慢慢刷」這個動作徹底吸收掉了。`stale-if-error` 則是它的容錯近親：當背景重新驗證失敗（源站回 5xx 或根本連不上）時，與其把錯誤吐給使用者，不如繼續回那個過期的舊值，用一點點陳舊換取在故障期間仍然可用。

這條模式之所以值得單獨點出來，是因為它把前面三種災難的解法收束成了一個共同的世界觀：**快取的災難，幾乎都發生在「失效」這個從有到無的瞬間；而最深刻的解法，往往不是去加快重建、或去擋住流量，而是消滅那個「無」的瞬間本身——讓快取在過期後仍然有東西可回，把陡峭的失效懸崖，改造成一道有舊值可以踩著走過去的緩坡。** single-flight 是在懸崖底下接住摔下來的人，邏輯過期與 stale-while-revalidate 則是乾脆把懸崖填平。

## 先分類，再開藥

繞了一圈，回到開場那個被打爆的 DB。現在你有能力對它做一次精準的鑑別診斷，而不是慌亂地把所有「快取保護」一股腦全堆上去。

問三個問題就夠了。**是很多不同的鍵在同一時刻一起 miss 嗎？** 如果是，那是雪崩——去查 TTL 是不是設得太整齊（批次預熱的整齊 TTL 是頭號嫌犯），加 jitter；如果是快取剛重啟，那要的是預熱與後端保護。**是同一個（或少數幾個）不存在的鍵被反覆查嗎？** 如果是，那是穿透——快取空值（短 TTL）或前置 Bloom filter，把「不存在」這個答案也擋在 DB 之前。**是某一個極熱的鍵在過期瞬間被千軍萬馬同時撞上嗎？** 如果是，那是擊穿——single-flight 讓只有一個請求去重建，或乾脆用邏輯過期 / stale-while-revalidate 讓它根本不物理過期。

這三道題的答案互斥，解法也互斥：對穿透加 jitter 毫無用處（不存在的鍵根本進不了快取，談何過期時間的分布），對擊穿擺 Bloom filter 也擋不住（那個熱鍵是真實存在的，Bloom filter 會放行）。**先分類，再開藥**——這六個字，是整章最該帶走的東西。

而在所有這些技巧之上，還有一條最該先問、卻最常被跳過的問題：**這層保護，你現在真的需要嗎？** 如果流量本來就不大，DB 即使承接「整批 miss」也綽綽有餘，那麼 Bloom filter、single-flight、邏輯過期這些東西，每一個都是你日後要維護、要 debug、要在某個凌晨三點搞懂為什麼它行為怪異的複雜度。它們是真實災難的解藥，不是預設的標配。先量一量「失效的那一瞬間，DB 到底扛不扛得住」，再決定要不要、要哪一帖。把為一百 QPS 的系統套上為十萬 QPS 設計的全套防護，和把 DB 裸奔在十萬 QPS 的洪水前，是同一種沒有先看清問題就行動的錯誤——只是方向相反。

退一步看，這三種災難之所以長成這個形狀，根源都在快取這個東西的先天結構：**它是一份「較快但可能過時、且隨時可能消失」的副本，而你的後端容量，是按照它「大部分時候在工作」來規劃的。** 正因為後端只準備承接漏網的那一小部分，所以快取在任何時刻的集體失手——無論是時間上集中（雪崩）、是某個鍵根本攔不住（穿透）、還是某個熱點在一瞬間被擊穿——都會被那道「2% 的容量接住 100% 的流量」的落差放大成災難。看懂了這道落差，你就看懂了：這三帖藥治的從來不是快取，而是**快取失手時，後端那道來不及加寬的門。**
