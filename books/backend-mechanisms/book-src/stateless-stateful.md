# 無狀態與有狀態：為什麼 serverless 要無狀態

凌晨三點，自動擴縮把你的服務從 4 個實例放大到 40 個，業務尖峰過了又縮回 4 個——這一切你睡得正熟，毫無感覺。隔天早上有人回報：一小撮使用者在尖峰那段時間「被登出了」，或者購物車裡的東西莫名其妙清空了一半。日誌乾乾淨淨，沒有錯誤、沒有崩潰、沒有例外堆疊。你重現不出來，因為現在只剩 4 個實例、流量也平了。

這不是 bug，至少不是你寫錯了哪一行的那種 bug。它是一個更深的東西在發作：**你把某些「下一次請求需要用到的記憶」放進了某個實例的記憶體裡，而那個實例在縮容時被平台殺掉了，記憶就跟著蒸發。** 縮容把承載狀態的那台機器當垃圾回收，平台沒覺得自己做錯任何事——它本來就保留「隨時換掉你任何一個實例」的權利。問題出在那個藏在程式裡的隱含假設——記憶體會一直活著——而它不會。

這一章要把「無狀態」這個常被當成口號、邊界卻很少被講清的概念，拆到機制級。它表面上是一條 serverless 的紀律（「函式不能有狀態」），底下其實是一個關於「process 記憶體的壽命」的硬事實，以及一連串「狀態到底躲在哪裡」的非顯然陷阱。

## 第一原理：process 的記憶體是 process 的私產

先把「狀態」這個詞釘死，否則整章都會在含糊裡打轉。

一個運算單元是**無狀態（stateless）**的，意思非常具體：**它在兩次請求之間，不持有任何「下一次請求要被正確處理所必需」的記憶體狀態。** 任意一個實例都能處理任意一個請求；把這個實例殺掉、換一個全新的上來，從外部看行為不變。**有狀態（stateful）**則相反：實例的記憶體裡握著外面取不到的東西——一段累加中的計數、一個尚未 flush 的排序緩衝、一份使用者會話、一個進行到一半的多步驟流程——殺掉這個實例，這些東西就跟著消失，而且通常沒有人知道它消失了。

注意這個定義裡完全沒有提到「你有沒有寫 `global` 變數」。無狀態與否的檢驗標準只有一句話：**「殺掉這個實例、換一個全新的，行為會不會變？」** 不變就是無狀態，會變就是有狀態。這個檢驗標準稍後會反覆用到，因為「我到底有沒有狀態」這件事，直覺常常會給出錯的答案。

而這一切之所以是個問題，根源是一句殘酷到近乎廢話的事實：**process 的記憶體是 process 的私產，process 一死即蒸發。** 沒有任何作業系統、任何 runtime、任何雲端平台，會在你的 process 被殺掉之後替你保管它堆積在 heap 上的那些物件。RAM 不是持久層。它從來都不是。

在傳統的「一台長命機器跑一個長命 process」的世界裡，這個事實的殺傷力被掩蓋了——機器很少重啟，process 一跑就是好幾個月，於是「把東西放記憶體裡、它就一直在」這個錯覺可以維持很久，久到讓人忘記它只是錯覺。雲端與 serverless 做的事，本質上是把這個錯覺**戳破並制度化**：它讓 process 的死亡從「罕見意外」變成「家常便飯、而且是平台主動發起的」。

## 牲口，不是寵物

業界有個粗糙但精準的比喻：雲端時代的運算實例是**牲口（cattle），不是寵物（pets）**。寵物有名字、生病了你帶去看醫生、你記得牠的脾氣；牲口有編號、生病了就換一頭、你對任何一頭都不該有感情。

serverless 的 execution environment、容器編排裡的一個 Pod——它們全是牲口。它們會因為這些理由在你沒有預期的時刻消失：

- **自動擴縮（autoscaling）**：尖峰過了就縮容，被縮掉的那台連同它的記憶體一起回收。
- **滾動更新（rolling update）**：發新版時，舊實例被逐一換成新實例（發布策略的機制見〈blue-green、canary 與 rolling〉）。
- **故障自癒**：健康檢查判定某實例不健康，就把它殺掉重建（探針機制見〈health check〉）。
- **平台維護**：底層主機要打補丁、要遷移，你的實例被驅逐到別處。

這不是平台的缺陷，正好相反——**這正是你買 serverless / 編排換來的東西。** 你交出「狀態可以安穩留在記憶體裡」這個便利，換來「實例可以被任意替換」這個彈性：能無痛擴縮、能滾動發布、能自動把死掉的換成活的。這是一筆明明白白的交易，而無狀態就是這筆交易要你付的價。平台之所以**敢**隨時換掉你的實例，正是因為它假設你的實例是可替換的；而要可替換，實例就不能在記憶體裡藏著別人取不到的狀態。

Kubernetes 把這條哲學直接刻進了它的物件設計。一個 `Deployment` 管理的 Pod 是**可互換的（interchangeable）**：它們從同一份 spec 長出來，沒有穩定身分，本地磁碟是 ephemeral 的——Pod 一旦重排到別的節點，本地寫的東西就丟了。如果你的工作負載**真的**需要穩定身分與持久的本地磁碟，Kubernetes 要你改用另一種物件 `StatefulSet`，它給每個 Pod 一個固定的名字和一個會跟著它重新掛載的 PersistentVolume。換句話說，平台逼你**在物件層級就先宣告**「我是無狀態還是有狀態」——因為這兩者的調度規則根本不同，混為一談就會在重排的瞬間丟資料。

## 那條鐵律：持久化是主動動作

從「記憶體是私產、process 一死即蒸發」這個第一原理，直接推出一條貫穿整個雲端後端設計的鐵律：

> **持久化（durability）是一個主動動作，不是預設行為。**

你寫進記憶體的任何東西，預設的命運就是隨 process 一起死。要它活過一次重啟，你得**主動**把它寫到一個「重啟之後仍然存在」的地方——資料庫、Redis、物件儲存、訊息佇列。沒有被主動寫出去的，重啟就是失憶，而且平台不會替你哀悼，也不會給你任何警告。失憶是無聲的。

這條鐵律聽起來像常識，但它真正咬人的地方，是那些「看起來已經安全、其實還沒落地」的瞬間。最經典的反面教材是這個模式：

```
// ANTI-PATTERN: memory as a write buffer
function handleEvent(e) {
  buffer.push(e);              // in-memory
  if (buffer.length >= 100) {
    flushToDatabase(buffer);   // only NOW does it become durable
    buffer = [];
  }
}
```

這段碼的算盤是「先攢在記憶體裡、湊夠一批再寫資料庫」，為的是減少寫入次數。它在 happy path 上跑得很好。但問一句：在 `buffer` 累積到第 37 筆、還沒湊滿 100、`flushToDatabase` 還沒被呼叫的那一刻，實例被縮容殺掉了，會發生什麼事？

那 37 筆事件**憑空消失**。client 端可能早就收到了「我收到了」的回應（因為 `handleEvent` 成功返回了），於是它不會重送。沒有人會知道這 37 筆丟了——沒有錯誤日誌，因為從程式的角度看一切正常返回；資料庫裡就是少了 37 筆，靜默地少。這正是開場那個「日誌乾乾淨淨卻丟資料」的劇本。

鐵律在這裡給出明確的判準：**持久化必須在「對外宣告成功」之前完成。** 你不能先回 client「OK」、再打算背景把它落地——這兩個動作的順序一旦顛倒，中間那個窗口就是一個丟資料的黑洞。如果你確實需要「先快速回應、再非同步落地」的效能，那這個「之後再落地」的承諾本身就必須是持久的——也就是說，那筆「待落地」的記錄要先進一個持久的地方（一張 outbox 表、一個持久佇列），由它來保證最終一定會被處理。把「我保證之後會做這件事」這個承諾本身變持久，是唯一安全的做法（這套 outbox 機制見〈跨服務的交付一致〉）。

## 狀態藏在你沒看的地方

如果無狀態的全部難度就只是「別寫全域變數」，這一章就不必存在了。真正讓人翻車的，是**狀態藏在那些你根本不覺得是狀態的東西裡**。回到那個檢驗標準——「殺掉換一個，行為會變嗎」——拿它去掃一遍一個自認無狀態的服務，往往會挖出一堆東西：

- **模組級的快取**。`const cache = new Map()` 放在檔案頂層，每個請求往裡塞一點東西。它就是狀態。實例 A 的 cache 和實例 B 的 cache 內容不一樣，同一個 client 兩次請求落到不同實例，會看到不一致的結果。
- **連線池**。它確實是狀態，只是它是**無害**的狀態——因為它不承載「正確性所需的記憶」。連線池放在全域是對的、是慣例（連線池機制見〈連線池〉），因為任一個池都能服務任一個請求，殺掉重建只是慢一點、不會錯。這劃出了關鍵的界線：**全域裡可以放「無狀態的東西」（連線、設定、編譯好的 regex），但不能放「跨請求的、與正確性相關的記憶」。**
- **累加器**。`let requestCount = 0; requestCount++`。在多實例下這個計數毫無意義——每個實例各數各的，永遠湊不齊。要全域計數，得用外部的原子操作（Redis `INCR` 或資料庫的原子更新），而不是「讀進記憶體 +1 再寫回」，後者連在單實例裡都是 race（見〈race condition 與原子性〉）。
- **本地檔案系統**。寫到本地磁碟的暫存檔，在容器或 serverless 裡同樣是易失的。

Lambda 把這個「狀態藏在哪」的陷阱演到了最極致、也最危險的版本，值得單獨看清楚。

## Lambda 的 warm reuse：最危險的那種「不無狀態」

很多人對 Lambda 的第一印象是「它天生無狀態，我不用操心」。這個印象一半對、一半會害死你。

機制是這樣的。Lambda 在事件到來時，啟動一個受隔離的 execution environment（一個 microVM）來跑你的函式。它的生命週期被官方明確切成三個相位：**Init → Invoke → Shutdown**。Init 階段跑你 handler **外面**的頂層程式碼（import 套件、建連線、讀設定）；Invoke 階段才跑 handler 本身;Shutdown 在環境要被回收前發生。

關鍵在於：一次 invoke 結束後，Lambda **不會立刻**銷毀這個環境，而是把它**凍結（freeze）**起來保留一段時間。如果短時間內又有同一個函式的請求進來，Lambda 把它**解凍重用**——這就是 warm start，省掉了整個 Init 階段，所以快得多。官方文件白紙黑字寫明了重用帶來的後果：**handler 外面宣告的物件，在下一次 invoke 時仍然是初始化好的狀態**（這正是把 DB 連線放在 handler 外的理由——連線被重用、不必每次重開）；而且 **`/tmp` 目錄的內容在環境凍結期間也會保留**，跨多次 invoke 都還在。甚至，官方還補了一個很多人不知道的細節：當一次 invoke 出錯、Lambda 重設環境時，**它並不會清掉 `/tmp` 的內容**——殘留會一路帶進下一次 init。

把這兩件事擺在一起，陷阱就現形了。execution environment 會被重用，意味著**你的全域變數和 `/tmp` 檔案會在不同的 invoke 之間殘留**。而不同的 invoke，極可能是**不同使用者**的請求。於是這個寫法：

```
// DANGEROUS: caching per-request data in global scope
let currentUser = null;

exports.handler = async (event) => {
  currentUser = await loadUser(event.userId);   // overwrites global
  return { greeting: `Hello, ${currentUser.name}` };
};
```

在冷啟動的第一個請求下看起來完全正常。但當環境被重用、而下一個請求**沒有**走到某個會覆寫 `currentUser` 的分支（或在賦值前就因某條件提早返回），它就會讀到**上一個使用者**留在全域裡的資料。這不是效能問題，這是一條真實的**資料外洩路徑**——A 使用者的姓名、權杖、購物車，洩漏給 B 使用者。它在低流量下幾乎不會發作（每次都冷啟動，全域是乾淨的），偏偏在高流量、環境密集重用時最容易爆，而那正是你最不希望出事的時候。

所以官方對 Lambda 的無狀態建議，講得非常精確：**handler 外的全域只放「跨 invoke 共用安全」的東西（連線、設定）；任何「只屬於這一次請求」的資料，一律放進 handler 內的區域變數**，用完即隨 invoke 結束而消失。判準還是那一句老話——這個東西如果殘留到下一個請求，會不會出事？

還有兩個 Lambda 特有的、容易被當成「平台幫我保管狀態」的誤會，得一併澄清：

- **`/tmp` 不是你的持久快取。** 它在環境凍結期間確實還在，於是有人拿它當跨請求快取用。但環境隨時會被回收——官方明說：環境會被保留「一段時間」，但即使是持續被呼叫的函式，平台也會**每隔幾小時就終止並重建環境**以便更新與維護，並警告你「**不要假設 execution environment 會無限期存在**」。`/tmp` 的內容會在某個你預測不到的時刻一起消失。它是個易失的、機會主義的加速層，不是儲存。
- **背景任務不會在 invoke 之間替你跑完。** 如果你在 handler 裡發起了一個非同步任務、但在它完成前就返回了，這個任務會被**凍結**；它可能在下一次 invoke 解凍時才接著跑（在一個完全不相干的請求的時間裡），也可能環境直接被回收、它永遠跑不完。所以任何背景工作都必須在 handler 返回前確實完成，別指望 invoke 之間的縫隙。

## 那會話、長連線、進行中的流程怎麼辦

講到這裡，一個誠實的反問會浮上來：可是現實系統就是有狀態啊。使用者得登入、購物車得記得、WebSocket 連著一條長命的通道、一個三步驟的下單流程跑到第二步——這些狀態不會憑空不存在，它們得**待在某個地方**。無狀態不是「假裝沒有狀態」，而是「**把狀態從會被殺掉的地方，搬到不會被殺掉的地方**」。這個搬遷有一套相當固定的對照：

| 狀態的種類 | 搬去哪 | 換來什麼 |
|---|---|---|
| 會話 / 使用者狀態 | 外部 store（Redis / DB），請求自帶 token | 任一實例可服務任一請求 |
| 進行中的多步驟流程 | 持久佇列 / durable workflow | 實例死了，流程能從上次的持久點續跑 |
| 累計型狀態（計數、聚合） | 外部原子操作（`INCR` / 原子 UPDATE） | 多實例下數字才湊得齊、不 race |
| 本地快取 | 接受它易失，定位成可重建的加速層 | 重啟就重建，不當權威來源 |
| 連線 / 暫存檔 | 外部連線池；`/tmp` 不假設跨請求存在 | 連線可重用，但不承載正確性 |

最常見的那條——把會話外置——值得多看一眼，因為它解釋了 JWT 這類 token 為什麼長那個樣子。把使用者狀態放進外部 store 之後，每個請求得自帶一張「我是誰」的憑證，讓**任意一個**實例都能驗證它、不必依賴「這個實例剛好記得你上次登入過」。這就是為什麼無狀態服務幾乎總是搭配「請求自帶 token」的設計：HTTP 協定本身（RFC 9110）就被定義成 stateless——每個請求的語意都能被**獨立理解**，伺服器不該假設同一條連線上的兩個請求來自同一個使用者；正是這個無狀態性，才讓「把請求動態分散到多台伺服器、隨意做負載均衡」成為可能。狀態外置與 HTTP 的無狀態語意是同一個哲學的兩面（token 的生命週期見〈token 的生命週期與撤銷〉）。

## sticky session：把無狀態的好處讓回去一點

但有一類狀態外置的成本高到讓人卻步：**長連線本身就是狀態**。一條 WebSocket 連著一個特定實例的記憶體（緩衝、訂閱關係、連線進度），這不是一個值、是一條活的通道，你沒辦法把它「寫進 Redis」然後讓別的實例接手——通道是綁在那個 process 的 socket 上的（長連線水平擴展的 backplane／pub-sub fan-out 細節見〈WebSocket：frame、ping-pong 與水平擴展〉）。

於是有了一個務實的退讓：**sticky session（會話親和性 / session affinity）**——讓負載均衡器把同一個 client 的請求，固定送到同一個實例。實作上靠一個 cookie 或來源 IP 雜湊來認人。它讓「狀態留在某個實例的記憶體裡」這件事重新變得可行，代價是你**把無狀態換來的好處部分讓了回去**：

- **負載不均**。請求不再能自由地灑到最閒的實例上，而是被綁死在「它第一次落到的那台」。幾個重量級 client 黏到同一台，那台就過載，別台閒著。
- **擴縮時仍會斷**。你新增一台實例來分擔負載，但既有的 client 全黏在舊實例上、不會自己遷過去；要等新連線才用得到新容量。縮容更糟——被縮掉那台上的所有連線**當場全斷**。
- **sticky 不是高可用**。這是最常被誤解的一點：sticky 只決定「請求路由去哪」，它**不複製狀態**。被綁的那台實例一死，黏在它上面的所有 client 的記憶體狀態**照樣全丟**——sticky session 不是備援，它沒有為你保存任何東西。

正因為這些代價，12-Factor 那條著名的 stateless processes 原則把話講得很硬：它要求 process 無狀態、share-nothing，記憶體與磁碟頂多當「單次交易內的短暫快取」，而 sticky session 被它直接點名為**對無狀態原則的違反、不該依賴**。

這話在原則層是對的，但工程現實沒那麼非黑即白。判準應該是這樣：**能外置就外置**——它換來自由擴縮，這幾乎總是值得的。只有當外置的成本真的過高，才退而用 sticky，並且**清醒地接受它的脆弱**。這個「成本過高」不是修辭，把數字攤開就看得很清楚：從本地記憶體讀一個值大約是 100 奈秒（0.0001 ms）等級，而向同機房的 Redis 跑一趟往返，就算網路理想也要 0.5 ms 上下——差了三、四個數量級。對一般的會話狀態，這點延遲完全吃得下：一個請求本來就要花幾毫秒到幾十毫秒，多這 0.5 ms 微不足道，換來「任一實例可服務任一請求」絕對划算。但對即時媒體那種每秒要搬幾十 KB、且每個封包都對延遲敏感的連線緩衝，狀態若不留在本地、而是每次都去外部繞一趟，那 0.5 ms 疊在每個封包上就足以把即時性吃掉——這時 sticky 把連線釘在本地記憶體才合理：接受擴縮時的不均、接受實例一死就丟那批 client 的連線態。把 sticky 當成「明知有代價、但這個場景值得付」的選擇，而不是「反正它能讓我繼續把狀態留在記憶體裡」的偷懶——後者會在某次縮容的凌晨三點回來找你。

## 反過來說：無狀態不是無條件的美德

把無狀態講得這麼重，得補上另一面，否則就成了教條。**無狀態是擴縮的手段，不是道德。**

如果一個工具本來單機就跑得好、永遠不會橫向擴展、也不上 serverless——一個內部的批次腳本、一個開發期原型、一個明確不需要擴縮的後台工具——硬把它的每一點狀態都外置到 Redis 或資料庫，只是徒增每次操作的網路往返延遲、徒增一個要運維的外部相依、徒增可能失敗的環節。你為了「可擴縮」付了成本，卻買了一個永遠用不到的能力。這是不折不扣的過度工程（這種「為不會發生的需求預先付費」的權衡見〈簡單性與過度工程：YAGNI 的兩面〉）。

判準因此很清楚：**你的運算層會不會被自動擴縮、滾動更新、或跑在 serverless 上？** 會——那無狀態不是選項、是平台強加的前提，因為它本來就會替你殺實例，你沒有商量餘地。不會、而且確定不會——那有狀態（把東西留在記憶體裡）反而更簡單、更快，沒必要為一個用不到的彈性付稅。一旦哪天它**要**橫向擴展了，那時記憶體裡的狀態就會立刻變成擴縮的天花板——你會在那一刻、而不是更早，去償還這筆債。

## 為什麼是這個形狀

退一步看，「serverless 要無狀態」這條規則，不是雲廠商拍腦袋定的紀律，而是從一個無法迴避的物理事實裡長出來的唯一形狀：**process 的記憶體是 process 的私產，process 一死即蒸發；而雲端與 serverless 的整個價值，恰恰建立在「平台可以隨時殺掉並替換你的 process」之上。**

這兩件事擺在一起，結論就被逼出來了。正因為平台要能隨時換掉實例才能給你彈性，實例就**不能**在記憶體裡藏著別人取不到的狀態——所以要無狀態。正因為記憶體一死即蒸發，持久化就**不能**是預設、必須是你主動把東西寫到外面去的動作——所以「先回成功、再背景落地」是個丟資料的黑洞。正因為長連線這種狀態外置的成本太高，才有 sticky session 這個「明知會讓回好處」的退讓存在。連 Lambda 那個 warm reuse 的資料外洩陷阱，都是同一個事實的另一張臉——平台重用了那段被預期會清空、實則殘留的記憶體，而不該跨請求活著的東西被放了進去。

所以下次你看到一份架構文件劈頭就寫「所有服務必須無狀態」，你會知道那句話底下不是一條風格偏好，而是一個物理約束的投影：在一個實例隨時會被當牲口換掉的世界裡，凡是你想要它活過下一秒的東西，都得自己主動送到一個不會被殺掉的地方去——否則它就只是 RAM 上一段等著被回收的位元組，而沒有人會替你記得它曾經存在。
