# 跨區與異質庫同步

到目前為止，談資料同步時我們其實一直偷偷假設了兩件事：兩端離得夠近、兩端長得夠像。CDC、outbox、衝突解決、對帳——這些機制在「同一個機房裡、PostgreSQL 對 PostgreSQL」的世界裡都優雅得很。但真實世界很少這麼客氣。

把兩個假設一起拿掉，整套同步的難度就換了一個量級。第一個假設崩掉，是因為你的副本不在隔壁機架，而在地球另一端——東京到維吉尼亞，封包來回一趟要走過半個地球，這段時間是光速釘死的，沒有任何調參能讓它消失。第二個假設崩掉，是因為來源庫和目的地庫根本不是同一種東西：一邊是 PostgreSQL，另一邊是 MySQL，或是一邊 OLTP、另一邊資料倉。它們的型別系統不同、交易語意不同、連「一列資料長什麼樣」「一個時刻怎麼存」都不同。

這一章談的，就是當「距離」和「異質」這兩個現實同時壓上來時，同步會變成什麼樣子。重點不是再發明新機制——CDC 還是 CDC，衝突解決還是衝突解決——而是看清楚：在這兩個約束下，你**實際上拿到的是什麼一致性**，以及哪些你以為理所當然的東西會悄悄壞掉。

## 兩條繞不過去的物理

先把難點釘在第一原理上。跨區同步的所有痛苦，幾乎都能追溯到兩條無法繞過的事實。

第一條是**光速**。資訊在光纖裡的傳播速度大約是真空光速的三分之二，再加上沿途路由器、交換機的處理延遲——所以實測的跨洲往返，永遠比「直線距離除以光速」那個理論下限更糟。具體一點：美東維吉尼亞（us-east-1）到東京（ap-northeast-1）的大圓直線距離換算成光纖往返，理論下限大約 **107 毫秒**；但實測的中位往返延遲（P50）其實落在 **150 毫秒上下**，因為真實的網路路徑不走直線、還要過一堆設備（2026-06，AWS region 間實測，cloudping）。就近一點的美東到愛爾蘭（eu-west-1）大約 **70 毫秒**。記住這個量級：跨洲一趟往返是「上百毫秒」，不是「幾毫秒」。這個數字是物理常數附近的工程現實，加機器、調 TCP 參數都動不了它。

這條物理直接判了一件事的死刑：**任何「每個請求都要跨區同步往返一次」的設計，它的延遲天花板就被釘死在那個 RTT 上。** 你的程式碼再快、機器再多，一筆需要等東京確認的寫入在維吉尼亞發起，使用者就是得等那兩百多毫秒。

第二條是 **CAP 在跨區時不再是理論玩具**。同機房裡網路分區罕見到可以當邊角情況；但跨洲的長鏈路上，分區是**會固定發生的營運事件**——海底電纜被船錨刮斷、某個 region 的網路退化、BGP 路由出包。當維吉尼亞和東京之間的連線斷掉時，CAP 逼你當場二選一：要嘛兩邊都拒絕寫入、保住一致（犧牲可用性），要嘛兩邊都繼續各寫各的、接受資料分岔（犧牲一致性，事後再收斂）。你**沒有**「分區時又一致又可用」這個選項——這不是設計得不夠好，是定理（理論面屬於另一個主題，這裡只取它的營運後果）。

異質庫則加上第三條，性質不太一樣但同樣繞不過：**兩個不同的資料庫之間，沒有共同的交易邊界，也沒有共同的型別語意。** 你不能用一個 `BEGIN ... COMMIT` 把「寫 PostgreSQL」和「寫 MySQL」包成一個原子單位——它們是兩個獨立的交易管理器。而且就算值搬過去了，「意思」也可能在轉換中走樣，因為兩邊對「一個時刻是什麼」「NULL 是什麼」「一個數字有多少精度」的定義並不一致。

這三條——光速、CAP、無共同交易與型別——就是這一章所有取捨的根。下面分兩條線走：先看「距離」這一側（跨區），再看「異質」這一側（跨庫）。

## 跨區：你只能買到三選二

把跨區同步的選擇攤開，本質上是在三件你都想要的東西之間做取捨：**低延遲讀、低延遲寫、強一致**。跨上百毫秒的距離，這三者很難同時拿到。三種主流架構，正好對應三種不同的放棄。

**單主跨區（single-leader）**是最常見、也最該作為預設的形狀。指定一個 region 當主（比如維吉尼亞），其他 region 擺非同步唯讀副本。讀可以就近——東京的使用者讀東京副本，幾毫秒搞定；但所有寫都得回到維吉尼亞主庫。它放棄的是「低延遲寫」：東京使用者的每一筆寫入，網路成本就是一趟跨區 RTT。換來的好處是**語意極簡**——只有一個地方能寫，所有寫入天然被主庫定序，根本不存在並發衝突這回事。

**多主跨區（multi-leader, active-active）**走另一條路：每個 region 都能在本地寫，寫完才非同步地把變更互相複製過去。它放棄的是「強一致」——既然兩個 region 可以同時各寫各的、誰都還沒看到對方，**並發衝突就成了必然**而非意外。於是多主架構的第一個設計問題，從來不是「怎麼把資料複製過去」（那是簡單的部分），而是「同一筆資料在兩區被同時改了，最後算誰的、怎麼讓所有副本收斂到同一個結果」。換來的好處是本地寫低延遲、且單一 region 掛掉另一區還能繼續寫。

**跨區同步寫 / quorum**則放棄「低延遲寫」放得更徹底：每一筆寫都要等跨區的多數確認才算數（比如資料分三區、`W=2`，每寫一筆都要等至少一個遠端區回覆）。換來的是真正的跨區強一致——但每一筆寫都釘在「最近的那個遠端 region 的 RTT」量級上，而且分區時為了不分岔，它會直接拒絕服務。

下面這張表把三者並排，但別把它當骨架看——真正的理解在於上面那句「三選二」：你選的不是一行表格，是「願意在哪件事上付代價」。

| 架構 | 放棄什麼 | 換到什麼 | 衝突 | 分區時 |
|---|---|---|---|---|
| 單主 + 非同步副本 | 低延遲寫（寫要回主） | 就近低延遲讀、單一真相、語意最簡 | 無（單寫者定序） | 副本繼續服務舊讀；主區掛要 failover |
| 多主 active-active | 強一致 | 各區本地低延遲讀寫、區故障仍可寫 | 必然，要選收斂策略 | 各區繼續寫、事後收斂 |
| 跨區 quorum / 同步 | 低延遲寫（每寫付跨區 RTT） | 跨區強一致 | 無 | 拒寫保一致（犧牲可用性） |

### 把單主跨區算到底

抽象的「三選二」要變成肌肉記憶，得用真實數字走一遍。設一個典型場景：主庫在維吉尼亞，唯讀副本在東京，跨區往返就用上面查到的數字——抓 **150 毫秒 RTT**（實測在約 130–170 ms 間浮動，取一個接近實測中位的整數方便推算）。

東京使用者**讀**東京副本，只付本地往返，幾毫秒；這是單主架構送你的免費午餐，讀就近、又快又便宜。但東京使用者**寫**，請求得先飛回維吉尼亞主庫——一趟去、寫入、一趟回，光網路就吃掉約 **150 毫秒**，這還沒算主庫本身的處理時間。對一個習慣了同區個位數毫秒延遲的系統來說，這是兩個數量級的差別。

更微妙的坑在「寫完馬上讀」。東京使用者寫了一筆（飛去維吉尼亞主庫成功），然後立刻刷新頁面想看結果——這次讀走的是**東京副本**。但這筆寫才剛在維吉尼亞落地，還在以非同步複製的方式往東京飄，副本此刻**落後主庫一個複製延遲**。健康時這個延遲是數十到數百毫秒，網路一抖可達秒級。於是使用者很可能讀到一個「還沒有自己剛剛那筆寫」的舊狀態——他明明剛存了，卻看不到。這就是讀己之寫（read-your-writes）在跨區下的具體長相，成因是複製延遲（一致性後果的細節見〈複製延遲與讀己之寫〉）。

現在把架構換成**跨區 quorum 同步寫**（資料分三區、`W=2`、`R+W>N`）。一致性問題消失了——每筆寫都等到至少一個遠端區確認才回（`W=2`），而讀也要湊到 quorum（`R≥2`，使 `R+W>N`）才保證看到最新。但代價立刻現身：東京使用者那筆寫，現在得等「最近的那個遠端 region」回覆，假設最近的遠端區也要 **80 毫秒**單程，這筆寫的延遲下限就被釘在約 **160 毫秒**（一趟去等確認、一趟回）。你用延遲買到了強一致。

這就是跨區的鐵律，攤開來看就是物理：**低延遲讀、低延遲寫、強一致，跨上百毫秒的距離，你能挑兩個。** 單主放棄低延遲寫，多主放棄強一致，quorum 放棄低延遲寫且分區時放棄可用性。沒有哪一行是「最佳實踐」——它取決於你的資料是讀多還是寫多、能不能容忍讀到舊值、分區時你寧可拒服務還是寧可分岔。

### 一個常被當成高可用的災備

跨區最危險的誤解，是把「有個跨區副本」當成「就有多區高可用」。這兩件事差得遠。

非同步唯讀副本是**災備（disaster recovery）**，不是 active-active。平時它只能讀、不能寫。主區整個掛掉時，你得做 failover——把某個副本提升為新主。這個過程有兩個讓人睡不著的細節。第一，**failover 不是瞬間且無痛的**：要偵測到主真的死了（而不是只是慢，這個「分不清慢和死」的根本困難屬於另一個主題）、要選出並提升新主、要把流量切過去，這中間有一段真空。第二，也是更致命的——**非同步複製意味著主區可能帶著「還沒複製出去的尾端寫入」一起死掉**。那些寫入已經回覆過 client「成功」了，但還沒飛到任何副本，主區一崩，它們就永久消失。使用者收到了成功確認，資料卻沒了。

所以「跨區副本＝高可用」是個會在最壞時刻背叛你的假設。副本給你的是「主區沒了之後，能用一份**可能少了最後幾筆**的資料重新站起來」，不是「主區沒了你毫無感覺地繼續跑」。要真正的多區無縫接管，你得付 active-active 或跨區同步的代價——而那兩條路，前者要你解衝突，後者要你付延遲。天下沒有白吃的高可用。

### 託管多主的現實：LWW 是預設，強一致要明選

多主聽起來複雜，但雲端託管服務把它包裝得很順手——順手到你可能沒意識到自己選了什麼衝突策略。

以 DynamoDB Global Tables 為例（2026-06）。它是託管的多區、多活複製：每個 region 的副本都能讀也能寫，一個 region 的寫入會非同步複製到其他所有 region，通常一秒內收斂。問題來了——同一筆 item 在兩個 region「同時」被改，怎麼辦？預設模式（multi-Region eventual consistency）的答案是 **last writer wins**：DynamoDB 比較每筆改動的內部時間戳，留時間戳最大的那個，所有副本最終收斂到「最後寫入者」的版本。

這裡藏著一個你必須主動意識到的取捨：**LWW 的「最後」是時間戳的最後，不是真實因果的最後。** 兩個 region 並發改同一筆、靠時間戳決勝——而跨區的時鐘各走各的，時鐘偏移會讓「真實上較晚、但時間戳較小」的那筆寫被靜默丟棄。沒有錯誤、沒有日誌，一筆寫就這麼不見了（這正是 LWW 配牆鐘的經典陷阱，機制細節與版本向量／CRDT 的對策見〈衝突解決〉）。如果你開了 active-active 卻沒主動想過衝突，等於把「最後算誰的」交給了預設的 LWW + 時間戳——對某些資料（覆蓋一個 last-seen 旗標）這沒問題，對另一些資料（餘額、計數）這就是會吃資料的 bug。

值得補一句現況：DynamoDB Global Tables 在 2026-06 也提供 **multi-Region strong consistency（MRSC）** 模式——它不再用 LWW 默默吞掉衝突，而是對並發衝突回一個**可重試的衝突錯誤**，把「該怎麼辦」的決定權交還給你。代價當然是回到了「跨區強一致要付延遲與可用性」那條鐵律上。預設給你便宜的最終一致 + LWW、強一致要你明確選並付代價——這個設計本身就是前面三選二的具體化。

## 異質：值搬過去了，意思未必

換到另一條線。現在兩端離得不遠，但它們是**不同的庫**：PostgreSQL 對 MySQL、關聯式對文件庫、OLTP 對資料倉。

最先要打消的幻想是「用內建複製就好」。PostgreSQL 的 logical replication 是個好東西，但它是**同構**方案——PG 對 PG。你沒辦法用它把資料複製到 MySQL，因為它輸出的是 PG 自己的邏輯變更格式，MySQL 根本不認。異質同步沒有「內建」這條路，你得在中間放一層**把變更標準化、再做型別映射**的東西。

主流做法兩種。一是 **log-based CDC + 型別映射管線**：用 Debezium 這類工具讀來源的交易日誌（PG 的 WAL、MySQL 的 binlog），把每筆變更解析成一個與資料庫無關的中介事件格式，經過一個轉換層做型別與編碼的映射，再寫進異質目的地（CDC 本身的機制見〈CDC：從資料庫變更長出事件流〉）。二是 **ETL/ELT 批次**，或像 AWS DMS 這類專用遷移工具——它的典型模式是「先做一次全量初始載入，再切換到 CDC 持續增量同步」，而對異質遷移，schema 與型別的轉換是獨立的一步（AWS 用 SCT 這類 schema 轉換工具先把來源 schema 譯成目的地的形狀，再讓 DMS 搬資料，2026-06）。

不論走哪條，異質同步的真正難點都不在「搬」，而在**轉換層**。值從一個型別系統倒進另一個型別系統時，意思會在邊界上悄悄走樣，而且——這是最陰險的部分——**這種走樣不會報錯**。它只會在某天的報表、某次排序、某個 join 上，給你一個無聲的錯誤答案。

### 一個會錯好幾個小時、卻不報錯的時間戳

把這個「意思走樣」講到能在腦中重放，最好的例子是時間。

設一個具體場景：把 MySQL 的一張表同步到 PostgreSQL。來源欄位是 MySQL 的 `DATETIME`，目的地欄位是 PostgreSQL 的 `timestamptz`。這兩個型別看起來都在存「一個時刻」，但它們對「時刻是什麼」的定義根本不同，這是已查證的語意差（2026-06，兩家官方文件）：

- MySQL 的 `DATETIME` 存的是一個**字面上的本地時間，完全不帶時區意識**。你存進去 `2026-06-21 09:00:00`，它就原封不動是這串字，不管伺服器在哪個時區、session 設了什麼時區，這個值都不變。它不知道、也不關心這個 `09:00` 是哪裡的九點。
- PostgreSQL 的 `timestamptz` 則相反：它在**寫入時把值依「當前 session 的時區」換算成 UTC 存起來**，讀取時再依讀取者的 session 時區換回去。它的內部永遠是 UTC，時區只是進出時的換算規則。

現在看轉換層怎麼出事。一個存在 MySQL 裡的 `2026-06-21 09:00:00`——這個字串本身**不攜帶任何時區資訊**，它到底是東京的九點、UTC 的九點、還是維吉尼亞的九點，MySQL 不知道。當這個裸值流進寫 PostgreSQL 的那一步，PG 必須替它補上一個時區才能算出要存成哪個 UTC 時刻。如果轉換層沒有**顯式指定**「這個來源值該被當成哪個時區」，那麼到底補上什麼時區，就取決於那條寫入路徑當下的 session 時區設定——可能是 UTC、可能是資料庫伺服器的本地時區、可能是某個函式庫的預設。

假設 MySQL 那台機器和它的應用其實是用東京時間（UTC+9）在記這個 `09:00`，意思是「東京早上九點」（也就是 UTC 同一天的 `00:00`）。但寫 PG 的 session 時區若是 UTC，PG 會把這個 `09:00` 當成「UTC 的九點」存進去——同一筆資料，在兩個庫裡代表的**絕對時刻差了整整 9 個小時**。

致命之處在於：**這一步不會丟出任何錯誤。** 兩邊都存了一個合法的時間戳，型別檢查全過，同步管線一片綠燈。錯誤只會在下游浮現——一份「按時間排序的訂單」突然亂序、一個「過去 24 小時的成交額」報表把跨午夜的交易算錯邊、一個「兩個時間點之間的事件」查詢漏掉或多抓了一批。等有人發現報表不對、回頭追，會發現資料早就錯了好幾週，而且**每一筆**時間都偏了固定的幾小時。這就是異質同步最典型的靜默資料損壞：值搬過去了，意思沒搬過去。

防它的辦法不是「更小心」，而是把它變成轉換層的**顯式契約**：來源的每一個時間欄位，都要明確宣告「它記的是哪個時區」，轉換層據此換算，並對這個換算寫測試。同樣的紀律適用於每一種會在兩庫間語意分叉的東西——NULL 的處理（一邊的空字串到另一邊是不是 NULL）、數值精度（`DECIMAL` 的位數、浮點的捨入）、字元編碼（latin1 對 utf8mb4）、自增主鍵的語意。**異質同步的轉換層，本質是一份「兩個型別系統之間的逐欄翻譯規約」**，而不是一條無腦的資料管道。沒有顯式定義並測試這份規約，你搬的就不是資料，是一堆看起來對、其實意思已經跑掉的位元組。

### schema 變更：第一次加欄位就會卡住

異質（甚至同構的跨區）同步還有一個會在「第一次改 schema」時引爆的坑：**複製機制搬的是資料變更，不是結構變更。**

以 PostgreSQL logical replication 為例（即使是同構 PG→PG 也一樣，這是已查證的官方限制，2026-06）：它**不複製 DDL**。你在發佈端（publisher）`ALTER TABLE ADD COLUMN` 加了一欄，這個 DDL **不會**自動跑到訂閱端（subscriber）。更糟的是接下來會發生什麼：發佈端那張表現在多了一欄、開始產生「帶新欄位的變更」往訂閱端送，而訂閱端的表還是舊結構、塞不進這個新欄位——**複製會直接報錯卡住**，而且會一直卡到你手動把訂閱端的 schema 補齊為止。

這帶出一條反直覺但救命的操作順序：**additive（新增型）的 schema 變更，要先在訂閱端做，再在發佈端做。** 先給訂閱端加好那一欄（它一開始是空的、沒人寫，無害），這樣等發佈端之後也加欄、開始送帶新欄位的資料時，訂閱端早已準備好接住，複製不會中斷。反過來先改發佈端，就會撞上上面那個「資料來了但表接不住」的卡死。這個「訂閱端先行」的紀律，是讓跨庫／跨區同步能在不停機下演進 schema 的關鍵（結構變更本身的完整機制是另一個主題，本章只取它在同步場景下的後果；序列化層的向前後相容是又一層、見〈序列化與 schema 演進〉）。

異質同步只是把這個問題放大：DDL 不只是「不會自動同步」，連「對應的結構」在兩庫間都未必存在——MySQL 的 `ENUM` 在 PG 裡沒有直接對應、PG 的陣列型別 MySQL 沒有、兩邊的索引語意也不同。所以異質同步的 schema 演進，永遠是「兩端各自手動改、且要協調好先後」的人工活，沒有自動駕駛。

## 為什麼是這個形狀

退一步看整章。跨區與異質同步之所以長成這副麻煩樣子，不是因為工具不夠好，而是因為它被兩條無法迴避的現實夾在中間。

**距離**這一側，光速把「跨區同步往返」釘成上百毫秒、CAP 把「分區時又一致又可用」判了死刑——於是你永遠在「低延遲讀、低延遲寫、強一致」裡只能挑兩個。單主、多主、quorum 不是三個任選的選項，而是三種「你決定放棄哪一個」的具體後果。你想要哪種保證，就得認哪種代價：放棄低延遲寫、放棄強一致、或放棄分區時的可用性。

**異質**這一側，沒有共同交易、沒有共同型別語意——於是「值搬過去」和「意思搬過去」是兩件事，而後者只能靠一份顯式、被測試過的逐欄翻譯規約來保證，否則時區、精度、編碼會在邊界上悄悄走樣，給你不報錯的錯資料。schema 變更則因為「DDL 不被複製」，逼出「訂閱端先行」這條反直覺的操作紀律。

把兩側合起來，這一章真正想留給你的判斷是：**跨區先問「能不能單主就好」**——讀多寫少的系統，一個主區加幾個就近唯讀副本（純災備）通常就夠了，別為了「聽起來很強」就跳進 active-active 那口同時要解衝突、又要扛跨區運維的深井。**異質先問「能不能就用 CDC 加一份明確的型別映射」**——別輕易裸著雙寫到兩種庫，那是把 dual-write 的分叉風險、和型別走樣的靜默損壞，一次全收下。多區、active-active、異質雙寫這些字眼都很有重量，但每一個都在向你收一筆具體的、跑不掉的稅。看清楚那筆稅是什麼，再決定要不要付——這就是跨上百毫秒的距離、跨兩個不同型別系統做同步時，唯一站得住的姿勢。
