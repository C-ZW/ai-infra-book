# load shedding：主動丟掉一部分流量

凌晨兩點，一個本來健康的服務開始出怪事。流量沒有暴增——只是某個下游慢了幾百毫秒，或一台機器掉出叢集、剩下的人接住了多出來的負載。儀表板上 CPU 衝到 100%、佇列越積越深，延遲從 20 毫秒爬到 800 毫秒、再爬到 3 秒。然後是最詭異的一段：吞吐量明明還在每秒幾千筆，**成功率卻一路下滑**，到後來幾乎沒有一筆請求真的回到使用者手上。服務沒有崩潰、行程沒被殺、CPU 滿載地忙著——但它做的全是白工。

這一幕的核心是一個違反直覺的事實：**一個過載的系統，越努力處理每一筆請求，能完成的有用工作反而越少。** load shedding 這套機制存在的唯一理由，就是讓系統在這種時刻做一件反本能的事——**主動、有選擇地丟掉一部分進來的請求**，好把資源留給剩下那些還救得回來的。它不是「資源不夠所以擺爛」，而是一個冷靜的取捨：寧可服務 80% 的請求且都快又正常、明確拒絕另外 20%，也不要假裝能服務 100% 然後讓所有人一起超時。

要看懂為什麼「丟掉一部分」反而是救活整體的唯一辦法，得先把那條下墜的曲線講清楚。

## goodput：被超時偷走的吞吐量

吞吐量（throughput）是一個騙人的指標。它數的是「系統每秒處理了幾筆請求」，但沒問一個更要緊的問題：這些被處理的請求，**有沒有人還在等它的回應**？

把這兩件事拆開，你會得到一個更誠實的量：**goodput**——單位時間內，**真正在客戶端還沒放棄之前、成功完成且有用**的請求數。throughput 數的是 CPU 忙了多少，goodput 數的是其中有多少沒有白費。在輕載時兩者幾乎相等：請求進來、很快做完、客戶端收到，每一筆 throughput 都是 goodput。

問題出在過載之後。假設客戶端的超時設在 100 毫秒。當佇列開始積壓、一筆請求要排隊 200 毫秒才輪到處理，那麼**在服務動手做它之前，客戶端早已超時離開了**。服務不知道這件事——它埋頭把這筆請求處理完、回了一個漂亮的 200 OK，送到一個已經沒人接聽的連線上。這筆請求消耗了完整的 CPU，goodput 卻是零。負載越高，佇列越深，這種「為死人做工」的比例就越大，於是出現那條招牌曲線：**throughput 還撐在高點，goodput 卻一路俯衝向零。**

這就是開場那一幕的真相。系統不是慢，是它把所有算力都花在了注定沒人要的回應上。AWS 在它的工程文獻裡把 load shedding 的目標講得很白：讓 goodput 在系統接近滿載時**平掉成一條水平線**，之後不管你再往它身上灌多少 throughput，goodput 都穩住不掉。load shedding 做的事，就是在請求注定要超時之前**早早把它拒掉**，把那筆本來要燒在死人身上的 CPU，還給一個還活著、還等得到回應的人。

## 為什麼過載會自己惡化：goodput 崩塌的算術

光說「goodput 會掉」還不夠，得看到它**崩塌**的數字，才知道這件事有多急。

設想一個單核服務，每筆請求耗 1 毫秒 CPU，所以飽和吞吐是每秒 1,000 筆。客戶端的超時是 100 毫秒。現在進來的流量升到 1,500 RPS——比它能消化的多出 50%。

沒有 load shedding 時會發生什麼？請求以 1,500/秒進來、只能以 1,000/秒做完，每秒淨積壓 500 筆。佇列像漲潮一樣往上堆。把佇列深度換算成等待時間很直接——前面排著 N 筆、每筆耗 1 毫秒、單核一次做一筆，新進來的這筆就得等約 N 毫秒（用消化速率 1,000/秒換算，不是流入的 1,500/秒）。一筆新請求進來時前面排著的人越來越多，等它排到時，牆上的鐘早就走過了 100 毫秒。於是**每一筆被處理完的請求，客戶端都已經超時走人**。服務 100% 滿載地忙著，goodput 卻趨近於零——它在以全速生產廢品。

現在加一條最樸素的 load shedding 規則：**請求進佇列時，若預估它「等待＋處理」湊不進 100 毫秒（這裡約等於排隊已超過 99 毫秒，因為處理本身還要 1 毫秒）就當場丟掉、回 503。** 反正排到它時客戶端也走了，做了也是白做，不如早點拒了、省下那 1 毫秒 CPU。結果是：系統只放約 1,000 RPS 進來處理（剛好它的容量），其餘約 500 RPS 在門口立刻被拒；這 1,000 筆全都在超時前完成。goodput 從約 0 一口氣回到 1,000。

同樣的 CPU、同樣的硬體，差別只在「願不願意丟」——goodput 從 0 變成 1,000。這不是優化，是急救。而且這個算術還藏著一層更急的東西：**為什麼流量降回正常之後，系統往往不會自己好。** 積壓的佇列、超時後立刻重試的客戶端，會持續把系統按在過載狀態裡（這個自我維持的惡性循環，是更廣的故障家族，見〈retry storm 與 metastable failure〉）。load shedding 之所以是過載防禦的第一道閘，正因為它能在迴圈成形之前就把多餘的流量擋在門外。

## 跟 rate limiting 長得像，但不是一回事

load shedding 和限流（rate limiting）都在「拒絕請求」，初看像同一件事的兩個名字。但它們觸發的依據根本不同，這個差別決定了你在什麼場景該用哪一個——混用會在最需要保護時失靈。

限流依**預先設定的固定速率**擋（見〈rate limiting：token bucket、leaky bucket 與分散式限流〉）。你跟它說「這個 API key 每秒最多 100 筆」，它就在第 101 筆下手——**不管系統當下忙不忙**。它是一份事前的合約：保護下游不被某個租戶打爆、把配額當成可賣的東西。它的判斷依據是「你」，不是「我」。

load shedding 依**系統當下的真實飽和度**擋。它不在乎你是誰、有沒有超過你的配額；它盯著自己的 CPU、佇列深度、event-loop 延遲、in-flight 請求數，**只有當這些指標說「我快撐不住了」才開始丟**。它是一份事中的自保：在系統真的接近崩潰點時，動態地把多餘的負載卸掉。它的判斷依據是「我」，不是「你」。

這個差別在一個具體情境裡會變得很鋒利：流量完全在所有人的配額之內、沒有任何一個租戶超速，但因為一台機器掛了、剩下的機器各自吃下多出來的一份，**整個叢集還是過載了**。限流此刻一筆都不會擋——每個人都守規矩。能救命的只有 load shedding，因為它看的是系統的真實狀態，不是規則手冊。反過來，一個惡意租戶在系統明明還很閒的時候狂打，load shedding 不會動手（系統還沒飽和），那是限流的活。

所以成熟的系統兩者都有，各擋一種威脅：限流防**可預期的配額濫用**，load shedding 防**不可預期的真實過載**。把 load shedding 設成固定速率，等於閹掉它的全部價值——它的靈魂就在於「依當下狀態動態決定」。

## 用什麼訊號決定該丟了：飽和度指標的陷阱

load shedding 的整個機制可以濃縮成兩個問題：**用什麼指標判斷「我過載了」**，以及**過載時先丟誰**。前一個問題看似簡單，卻是最多人踩坑的地方。

最直覺的指標是 CPU 使用率：CPU 超過某個水位（比如 90%）就開始拒。對 CPU 密集的工作這沒問題。但在 **event-loop 模型**裡（單執行緒、靠非阻塞 I/O 同時服務大量連線，見〈event loop：單執行緒怎麼同時做很多事〉），CPU 使用率會騙你——一個 event-loop 可能 CPU 還沒滿，但因為某個回呼卡了一下、或同時湧進太多待處理的 callback，**event-loop 的單圈延遲早已嚴重劣化**，請求在 loop 的佇列裡乾等。此刻用 CPU 當觸發會「太晚」：等 CPU 真的滿了，延遲早就爆了。對這種模型，**event-loop lag**（一圈走多久）才是最早反映飽和的訊號。

更普遍的教訓是：**沒有一個放諸四海皆準的飽和指標，你得選「對你這個系統最早翹起來」的那一個。** CPU、記憶體、佇列深度、in-flight 請求數、p99 延遲——每一個都在某些系統裡是先行指標、在另一些裡是落後指標。選錯了，load shedding 不是太早誤殺可服務的請求（水位設太低、浪費容量），就是太晚（水位設太高、等你開始丟，系統已經栽進惡性循環）。

這裡藏著一個最常見也最致命的 bug：**健康檢查被一起丟掉。** 一個粗暴的「飽和就回 503」規則，如果連 liveness／readiness 探針（見〈health check：liveness 與 readiness〉）的請求都一視同仁地擋下，後果分兩種、都不好：**liveness 探針**被拒時，編排器（Kubernetes 之類）會判定這個容器死了、**直接把它重啟**——本來只是暫時過載、卸掉一點負載就能撐過去的服務，反而因為探針被殺掉，過載瞬間升級成停機；**readiness 探針**被拒雖不觸發重啟，卻會把這個 Pod 從 Service 的 endpoint 摘掉、停止導流，若全叢集同時這樣摘，等於集體下線。load shedding 必須對控制面流量（健康檢查、心跳、最高優先級的內部請求）網開一面：這些請求量很小、卻是系統「還活著」的證明，丟它們等於自斷生路。

## 過載時先丟誰：把請求分級

決定了「何時丟」，接著是「丟誰」。最笨的做法是隨機丟、或一律丟最新進來的。但過載恰恰是最該**精挑細選**的時刻——因為不是所有請求都一樣值錢。

把請求按重要性分級，過載時從最不重要的開始丟，是 load shedding 從「鈍器」變成「手術刀」的關鍵。Google 在它的基礎設施裡把請求標上四個明確的優先級：`CRITICAL_PLUS`、`CRITICAL`、`SHEDDABLE_PLUS`、`SHEDDABLE`，名字本身就說明了設計意圖——有些請求生來就是「可被卸載的」（sheddable）。過載時，系統先把 `SHEDDABLE` 的丟光，還不夠再丟 `SHEDDABLE_PLUS`，一層層往上，**最關鍵的 `CRITICAL_PLUS` 只在萬不得已時才碰**。

這個分級在實務上常見的切法：付費用戶 vs 免費用戶、互動式請求 vs 批次背景任務、寫入 vs 可重算的讀取、面向使用者的請求 vs 內部預取。一個典型的決定是「過載時先停掉背景的資料預熱，保住使用者點下去那一下的回應」——同樣是卸載，但卸在使用者感覺不到的地方。

另一條正交的軸是**依成本丟**：不丟「不重要的」，而丟「最貴的」。一個無快取的複雜聚合查詢，可能抵得上一百個簡單的鍵值讀取；過載時丟掉那一筆貴的，能換回一百筆便宜的存活。用最小的犧牲數，保最多的成功請求數。

但分級不是免費的午餐，它欠下兩筆債。第一，請求得**攜帶**它的優先級資訊——這意味著從入口到每一跳都要把 criticality 傳遞下去，否則下游無從判斷。第二，被持續降級的低優先級流量會**餓死**：如果系統長期處在「剛好得丟 sheddable」的狀態，那些請求就永遠輪不到，得有機制兜底（比如保留一個極小的配額、或在持續飢餓時告警）。分級買到了「過載時保住要緊的」，付出的是「多了一套要維護的優先級語意，和一群可能被永久犧牲的低端請求」。

## 把延遲本身當成卸載訊號：CoDel 與自適應併發

到目前為止的 load shedding 都需要你**手動設一個水位**——CPU 90%、佇列 1000 筆。但水位是個脆弱的魔術數字：設高了來不及，設低了浪費容量，而且機器升級、流量結構一變，昨天對的數字今天就錯了。有沒有辦法讓系統**自己**算出該丟的時機，不靠人去猜門檻？

有，而且這個想法是從一個看似無關的領域借來的：**網路壅塞控制**。

第一個版本叫 **CoDel**（Controlled Delay；Nichols 與 Jacobson 2012 年發表於 ACM Queue，2018 年 1 月才標準化為 RFC 8289）。它本是為了解決路由器的「緩衝膨脹」（bufferbloat）——封包在過大的緩衝區裡排太久。CoDel 的洞見漂亮得近乎狡猾：**它不看佇列有多長，而看封包在佇列裡待了多久（sojourn time，停留時間）。** 一個塞滿但快速流動的佇列是健康的；一個不長卻一直排不動的佇列才是病了。CoDel 盯著一段時間窗（預設 interval 100 毫秒）內停留時間的**最小值**，只要這個最小值持續超過目標（預設 target 5 毫秒）整整一個 interval，它就判定「這不是一陣突發、是真的堵了」，開始丟封包；而且丟的頻率隨著持續堵塞**越來越快**（丟棄間隔反比於已丟數量的平方根），直到停留時間掉回目標之下。

這個「看停留時間、而非佇列長度」的思路，被搬進了應用層的請求佇列。一個服務可以給每筆進佇列的請求蓋一個時間戳，出佇列時算它等了多久——**等太久的請求直接丟，不送進處理**。因為一筆等了 200 毫秒才出佇列的請求，客戶端極可能早超時了；用停留時間判斷，等於直接逼近了「這筆做了會不會白做」的本質，比拍腦袋設一個佇列長度閾值精準得多。

更進一步的版本，是把整個系統的**併發上限**做成自適應的，這條路同樣抄自 TCP 的壅塞控制（TCP 靠觀察 RTT 變化來推測網路堵不堵、動態調整發送窗口）。它的數學底座是 Little's Law——**佇列裡的請求數 ≈ 到達率 × 平均延遲**——反過來說，給定一個延遲目標，能容許的併發數就被框死了。Netflix 的 concurrency-limits、Envoy 的 adaptive concurrency 都走這條路：先在「故意只放極少請求進去」的理想狀態下，量到一個基準延遲 `minRTT`（系統不堵時一筆請求該花多久）；之後每個取樣窗用實際觀測到的延遲 `sampleRTT`（通常取個高百分位）和它比，算出一個梯度：

```
gradient   = (minRTT + buffer) / sampleRTT
limit_new  = gradient * limit_old + headroom
```

這裡 `buffer` 與 `headroom` 是兩個不同的量、別混為一談：`buffer` 是加在分子上、容許延遲正常抖動的緩衝（Envoy 把它設成 `minRTT × buffer_pct`，典型 25%），讓 `sampleRTT` 只比 `minRTT` 略大時梯度仍 ≈1、不誤判成過載；`headroom` 是加在 `limit` 更新式上的驅動項（Envoy 把它釘死成 `sqrt(limit)`、不可設定），負責在系統不堵時主動把上限往上推、探一探還能放多少。

讀懂這兩行就讀懂了整個機制。當系統不堵，`sampleRTT` 貼近 `minRTT`，梯度約等於 1（甚至因為 `buffer` 讓分子略大於 `sampleRTT` 而稍稍超過 1），併發上限**緩緩往上爬**，試著多放點流量進來。一旦系統開始堵、`sampleRTT` 明顯大於 `minRTT`，梯度**掉到 1 以下**，併發上限隨之**自動收緊**——超過這個動態上限的請求就被卸載掉。沒有人設過任何固定的 CPU 或佇列門檻；系統純粹從「延遲有沒有比它健康時變糟」這一個訊號，自己推導出當下該收多緊。機器換了、流量變了，它重新量一次 `minRTT` 就跟著校準。

代價是這套東西**精巧而脆弱**：它需要對請求的併發有完整掌控（中間若有別的東西也在排隊、或 `minRTT` 在系統從沒真正空閒過的情況下量不準，梯度就會失真），調校 `buffer_pct` 與取樣窗也是門手藝。它買到的是「不必維護魔術數字、自動適應」，付出的是「更難理解、更難除錯、出問題時行為更難預測」。對多數系統，一個調好的固定水位 + 優先級分級就夠了；自適應併發是給那種「流量結構多變、人工調水位調到崩潰」的大規模場景準備的重武器。

## 卸載必須是一個完整的對話

最後一個故障模式最隱蔽，因為它在單看一個服務時完全看不出來：**你卸載掉的負載，會不會原封不動地又被打回來？**

load shedding 拒掉一筆請求、回了 503，這在服務這端是「我成功保護了自己」。但故事還有客戶端那一半。如果客戶端收到 503 後**立刻無退避地重試**，那筆被你卸掉的負載一秒後就回來了——而且因為重試，可能還變多了。你以為丟掉了 500 RPS，實際上它們在門口轉了一圈又排回隊伍，你的卸載成了空轉，CPU 還是滿的。最壞的情況，這正是把過載放大成持續性崩潰的那個正回饋迴圈的起點。

所以 load shedding 從來不是一個服務單方面能做完的事，它是一場**雙方的對話**。服務端拒絕時，回應裡得帶上退避指引（`Retry-After`、或明確的「現在過載、稍後再來」訊號）；客戶端收到後，得真的退避——指數退避加隨機 jitter，把重試攤開到未來、而不是所有人同時殺回來（重試的具體做法見〈重試、退避與 jitter〉）。少了客戶端這一半，服務端再聰明的卸載都只是把同一批流量在門口推來推去。

這也是為什麼最徹底的 load shedding 把卸載的決定**前推到客戶端自己身上**。與其讓請求大老遠跑到服務端、被拒、再退回來，不如讓客戶端在發出前就判斷「這趟值不值得」。Google 的客戶端自適應節流就是這個思路：每個客戶端記住自己最近兩分鐘發了多少請求（`requests`）、其中被後端接受了多少（`accepts`）。正常時兩者貼近；當後端開始大量拒絕、`requests` 漲到 `accepts` 的 K 倍（典型 K=2）以上，客戶端**自己**就開始按一個機率在本地丟棄請求——

```
rejection_probability = max(0, (requests - K * accepts) / (requests + 1))
```

這些被本地丟掉的請求**連網路都不會碰**。妙處在於它的自我收斂：後端拒得越凶，`accepts` 越低，`requests / accepts` 的比值越大，客戶端本地丟棄的機率就越高，打到後端的流量自動往下收——而當後端緩過來、`accepts` 回升，比值縮小，客戶端又慢慢把流量放回去。整個過載防禦變成一個分散在所有客戶端身上、無需中央協調的自穩定迴圈。沿著 deadline 在多跳呼叫鏈上的傳播也是同一哲學：每一跳檢查「剩下的時間還夠不夠做完」，不夠就提早放棄，絕不送出一筆注定超時的請求去白佔下游的資源（見〈timeout、deadline 與 budget：把等待變成有界〉）。

## 為何是這個形狀

退一步看，load shedding 的整個樣貌，都是從一個無法迴避的物理事實裡長出來的：**一個系統的處理能力是有限的，而過載時把這份有限的能力花在注定失敗的請求上，是雙重的浪費——既沒服務到那筆請求，又排擠了本來能成功的請求。**

正因為過載會讓 goodput 崩塌、而不只是線性地變慢，所以它必須**主動**丟，不能被動等系統自己撐——撐的結果是全員陣亡。正因為「該不該丟」取決於系統當下的真實狀態而非預設規則，所以它跟限流分屬兩件事、得各看各的訊號。正因為不是每筆請求一樣值錢，所以它要分級、要做手術刀而非鈍器。正因為固定水位是脆弱的魔術數字，所以才有人把網路壅塞控制的智慧借過來、讓系統從延遲本身自己推導該收多緊。正因為被丟的負載會反彈，所以它必須是客戶端與服務端之間一場有退避、有節流的完整對話，而非服務端一廂情願的拒絕。

load shedding 看似是在「放棄」一部分使用者，本質上卻是一個系統在過載這個殘酷時刻裡，唯一能做的負責任的事：**誠實地承認自己的極限，然後在這個極限之內，把能做好的事情做到最好。** 一個拒絕承認極限、堅持要服務每一筆請求的系統，最後一筆也服務不了。下次你看到一個服務在尖峰時回出一串 503、卻始終沒有真的倒下，你會知道那不是它壞了——那是它在丟掉一部分，好讓剩下的活著。
