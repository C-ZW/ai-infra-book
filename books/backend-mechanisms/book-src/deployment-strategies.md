# blue-green、canary 與 rolling

凌晨兩點，你要把一個服務從版本 N 換成 N+1。程式碼通過了所有測試、code review 也過了，按理說該是個無事件的部署。但這裡藏著一個任何測試都覆蓋不到的時刻：**換的那一瞬間，舊版還沒退場、新版還沒被真實流量驗證過，兩者必須在同一個生產環境裡交接。** 交接窗口裡任何一個不相容、任何一個只在真實負載下才現形的 bug、任何一次資源不足，都會直接打到正在付錢、正在等回應的使用者身上。

部署策略要管的，從頭到尾就是這一段窗口。它不負責讓你的程式碼變正確——那是測試與 review 的事；它負責的是：**當新版本注定有某種你還沒發現的問題時，怎麼讓這個問題打到最少的人、被最快地發現、被最快地收回。** 三種主流策略——rolling、blue-green、canary——其實是對同一個問題的三種不同回答，差別在於它們各自把「window 風險」往哪個方向擠。要真正讀懂它們，得先看清楚天真的做法在哪裡破掉。

## 天真的做法：先停舊的，再起新的

最直覺的換版方式叫 **recreate（重建式）**：把舊版全部停掉，再把新版全部起來。它的好處是乾淨——任何一個瞬間，線上只有一個版本，你永遠不必擔心「新舊版同時在跑會不會打架」。

代價也一樣直白：從舊版停掉、到新版第一個 process 能接流量之間，**服務是全停的**。這段空窗有多長，取決於新版的啟動時間——載入 runtime、建立連線池、暖機快取、跑完啟動健康檢查（見〈health check：liveness 與 readiness〉），動輒十幾秒到數十秒。對一個夜間批次系統，這沒什麼;對一個面向使用者的服務，每次部署都讓所有人看幾十秒的 502，是不能接受的。

於是真正的需求浮現了：**我們要的是零停機（zero-downtime）的換版**。而一旦你要求零停機，你就被迫接受一件事——**在換版的某段時間裡，舊版和新版必須同時對外服務**。recreate 用「全停」換來了「永遠只有一個版本在線」的單純；要拿掉那段全停，就得放棄那份單純。三種零停機策略的全部複雜度，幾乎都是從「新舊共存」這個被迫接受的前提長出來的。記住這條因果鏈，後面每一個陷阱你都能自己推出來。

## rolling：一批一批換，總容量不塌

rolling（滾動更新）是最節制的零停機做法，也是 Kubernetes Deployment 的預設策略。它的想法簡單到近乎樸素：**不要一次換完，一批一批換。** 假設你有 10 個 replica 跑著舊版，rolling 會先起幾個新版的、確認它們健康、再停掉等量的舊版，如此一批批推進，直到全部換成新版。任何一個時刻，在線的 replica 總數大致維持不變，所以總容量不塌、服務不中斷。

控制「換得多激進」的是兩個旋鈕，理解它們的交互作用是讀懂 rolling 的關鍵：

- **`maxUnavailable`**：更新過程中，相對於期望副本數，**最多允許幾個不可用**。設成 0，就是「絕不讓總可用容量低於目標」。
- **`maxSurge`**：更新過程中，**最多允許超額起幾個** replica（暫時超過期望副本數）。設成 0，就是「絕不為了換版多佔資源」。

Kubernetes 對這兩者的預設值都是 **25%**（2026-06；可設成整數或百分比，但兩者不能同時為 0——同時為 0 等於「不准少、也不准多」，那就動彈不得了）。把這兩個旋鈕的組合想清楚，rolling 的行為就完全可預測了：

- `maxSurge=1, maxUnavailable=0`：先多起 1 個新版（總數暫時 11），它 ready 後再停 1 個舊版（回到 10），如此推進。全程可用容量**從不低於 10**，代價是部署期間要多出 1 個 replica 的資源。這是對使用者最安全的設定。
- `maxSurge=0, maxUnavailable=1`：先停 1 個舊版（總數暫時 9），再起 1 個新版補回。不佔額外資源，代價是部署期間容量**會短暫掉到 9**——如果你的服務本來就貼著容量上限在跑，這一掉可能就過載了。

這裡有個容易被忽略的細節：rolling 判斷「一個新 replica 可以接流量了沒」，靠的是 **readiness probe**（見〈health check：liveness 與 readiness〉）。readiness 一旦回報就緒，這個新 replica 就被加進負載均衡的後端、開始分到真實流量。**問題在於 readiness 通常只查「process 活著、能回 200」，查不出業務邏輯對不對。** 一個會把每筆訂單金額算錯、但 `/healthz` 照樣回 200 的新版，在 rolling 看來是完全健康的——它會一路把 10 個 replica 全換成這個會算錯帳的版本，全程綠燈。rolling 對「壞版本」毫無免疫力，它只保證「換的過程不掉容量」，不保證「換上去的是對的」。

而 rolling 最痛的地方在回滾。rolling 沒有「上一套環境原封不動留著」這回事——舊版的 replica 在滾動過程中被逐批銷毀了。所以當你發現新版有問題、要退回去，**回滾本身又是一輪 rolling**：再起舊版、再停新版，一批批來。這段時間裡，壞版本還在線上服務一部分流量。換句話說，rolling 的回滾速度和它的部署速度一樣慢——你不是按一個鈕就瞬間回到從前，而是要再花一整輪滾動的時間，才能把毒清乾淨。

## blue-green：把回滾變成一次路由切換

如果回滾速度是你的硬需求——比方一個金流服務，發布出錯每多一秒都在虧錢——rolling 那種「再滾一輪」的回滾就太慢了。blue-green（藍綠部署）就是衝著這個痛點來的。

它的做法是空間換時間：**開一整套全新的環境（green）跑新版，舊環境（blue）原封不動地繼續服務。** green 起來後，你可以對它做完整的驗證——跑 smoke test、打內部流量、確認它真的健康——這段期間真實使用者完全感覺不到，因為流量還全在 blue。確認 green 沒問題，你把流量入口**一次性**從 blue 切到 green。出事了？把流量切回 blue 就好。

關鍵在於「切換」這個動作落在哪一層，因為**切換的原子性與回滾速度，完全由這一層決定**：

```
client ──▶ [ traffic switch ] ──┬──▶ blue  (version N,   still warm)
                                └──▶ green (version N+1, new)

switch 落在哪一層，決定回滾多快、多乾淨：
  - LB target group / Service selector : near-atomic, 立即生效
  - reverse proxy / Ingress weight      : near-atomic, 立即生效
  - DNS record (A / CNAME)              : 受 TTL 與 client 快取拖累, 慢且不乾淨
```

切換點選在負載均衡器的 target group、或 Kubernetes Service 的 label selector、或反向代理的後端權重，切換幾乎是原子的、即刻生效，回滾也就近乎瞬間。但如果你貪圖簡單、用 **DNS** 來做切換（把網域從指向 blue 改成指向 green），就踩進一個經典陷阱：**DNS 有 TTL，而且很多 client（包括作業系統、JVM、各種函式庫）會無視 TTL 自己快取解析結果。** 你改了 DNS 紀錄，但舊的解析結果還活在無數 client 的快取裡，它們會繼續往 blue 打，可能持續好幾分鐘甚至更久。這讓 DNS 層的「切換」既不原子、回滾也不乾淨——記錄已指向 green，卻仍有一部分流量黏在 blue 上，直到那些快取自然過期。blue-green 的「秒切換、秒回滾」承諾，建立在切換點本身是即時可控的；DNS 不是。

blue-green 買到的保證很明確：**新版可以在零使用者風險下被完整驗證，回滾近乎瞬間**。代價也很明確：**部署期間你要同時養兩套等量環境，資源翻倍**。對一個跑著 100 台機器的服務，blue-green 的那一刻意味著 200 台的帳單。這通常是短暫的（切換完成、確認穩定後就把 blue 收掉），但它要求你的基礎設施有能力臨時撐起雙倍容量。

### blue-green 真正的難點不在應用層，在資料庫

blue-green 看起來最乾淨，但它藏著整個部署主題裡最隱蔽的一個陷阱，而這個陷阱恰恰來自它「兩套環境」的賣點。應用層你可以開兩套、瞬間切換——**但 blue 和 green 通常共用同一個資料庫。** 資料庫沒有藍綠之分。

於是問題來了：green 跑的新版如果改了 schema（比如 `ALTER TABLE` 加了個 NOT NULL 的欄位、或 rename 了一個欄位），這個變更會直接作用在那個**共用的**資料庫上。而你準備用來回滾的後路——blue，跑的是舊版——可能就**讀不了被改過的 schema 了**。blue 本該是隨時能切回去的安全網，但 green 的 migration 已經把這張網割破了。回滾路徑被自己的資料庫變更給切斷，這是 blue-green 最致命、也最容易被忽略的失效模式。

破解之道不是 blue-green 獨有，而是任何「新舊版共存」策略都必須遵守的鐵律：**schema 變更必須對前後兩個版本都相容**，絕不能一步到位。這正是 expand-contract（先擴張、後收縮）這類零停機遷移模式存在的理由——加欄位要先部署「能讀新欄位的版本」、再部署「會寫新欄位的版本」，把一個破壞性變更拆成數次各自相容的小步（這套機制本身的推導見〈零停機 schema 遷移：expand-contract 與 backfill〉）。對 blue-green 來說，重點是認清：**你的「秒回滾」只在 schema 對 blue 仍然相容時才成立**。一旦 green 跑了個破壞相容性的 migration，blue-green 的核心賣點就失效了，而且失效得無聲無息——直到你真的需要回滾、卻發現切回 blue 後它整片報錯。

## canary：讓一小撮流量替全體試錯

rolling 管的是「換的過程不掉容量」，blue-green 管的是「回滾要快」，但它們有個共同的盲點：**新版要嘛沒接真實流量（blue-green 切換前的 green），要嘛接的流量比例不可控地一路放大（rolling 換到一半時，約一半流量已在新版手上）。** 如果新版有個只在真實流量、真實負載下才現形的問題，這兩種策略都讓它有機會打到相當大一片使用者。

canary（金絲雀）就是來補這個盲點的。名字來自礦工下礦坑時帶一隻金絲雀——金絲雀對毒氣比人敏感，牠先倒下，礦工就知道該撤了。canary 部署也是這個邏輯：**讓新版先只接一小撮流量（比如 5%），用這一小撮使用者當哨兵，盯著新版在真實流量下的錯誤率、延遲、各項業務指標；確認健康，才逐步把流量放大——5%、25%、50%、100%。** 這組級數與每階停留時間只是示意，實務上由 SLO 閾值與 progressive delivery 的設定決定，不是任何工具的固定預設。任何一階出問題，就把流量打回 0，受影響的永遠只是當下那一小撮。

canary 的整套價值，可以用一個具體的數字看清楚。

### 一個 worked example：同樣的 bug，三種策略打到多少人

假設一個服務每秒處理 2,000 次請求，新版藏了一個 bug：它會讓 **1%** 的請求回 500。同樣這個 bug，三種策略下的傷害天差地別。

**rolling**（10 個 replica，`maxSurge=2, maxUnavailable=0`，逐批替換）：當你滾到第 5 個 replica、新版承擔了約 50% 的流量時，故障率是 `50% × 1% = 0.5%` 的總請求出錯——即每秒約 `2000 × 0.5% = 10` 筆 500。而且你按下回滾後，rolling 還得再滾一整輪把新版換回舊版，這段時間錯誤還在累積——只是隨著新版 replica 被逐批換掉，每秒錯誤量會從 10 筆一路遞減到 0。假設你 30 秒後才察覺（這段維持約 10 筆/秒，累積 `10 × 30 = 300` 筆），回滾又花 60 秒（錯誤率從 10 筆/秒遞減到 0、平均約 5 筆/秒，再累積 `5 × 60 = 300` 筆），整場約累積數百筆失敗請求。

**blue-green**：切換前 green 沒接真實流量，所以這個「只在真實流量下現形」的 bug 在驗證階段可能根本看不出來；一旦你把流量**一次性**全切到 green，瞬間 **100%** 流量撞上 bug——每秒 `2000 × 1% = 20` 筆 500。好消息是你能在數秒內把流量切回 blue，所以爆發強度高但持續時間極短。blue-green 賭的是「我切回去夠快」，不是「我打到的人夠少」。

**canary**（先放 5%）：故障率是 `5% × 1% = 0.05%`——即每秒 `2000 × 0.05% = 1` 筆 500。自動分析在數十秒內偵測到這一小撮流量的錯誤率異常、自動把權重打回 0。整場事故受影響的請求數，比 rolling 少了一個數量級，比 blue-green 的瞬間全量更是天壤之別。

差距不在「會不會出事」——三種策略都攔不住一個已經混進新版的 bug；差距在**出事時打到多少人、多快能止血**。這正是發布策略真正在買的東西：不是正確性，是 blast radius（爆炸半徑）的可控性。

### 為什麼「先發一台看看」不算 canary

canary 最常見的誤用，是把它降格成「先發一台、人去肉眼看 dashboard、覺得沒事就全面放量」。這有 canary 的形、沒 canary 的魂，差在兩個地方，而這兩個地方恰恰是 canary 的全部價值所在。

第一是**統計意義**。5% 的流量要能當哨兵，前提是這 5% 大到足以讓指標有統計意義。回到上面的例子：每秒 2,000 請求的服務，5% 是每秒 100 請求，幾十秒就能累積數千個樣本，1% 的故障率會清清楚楚地浮出來。但如果你的服務每秒只有 10 個請求，5% 是每秒 0.5 個請求——你盯著它看十分鐘，可能連一個 500 都還沒等到，自然「沒看到問題」。**canary 的解析度受限於 canary 流量的絕對量**：流量太小，canary 期看到的「綠燈」只是樣本不足的假象，不是真的安全。這就是為什麼 canary 是大流量服務的奢侈品——你得有足夠的流量，讓一小撮也具備統計意義。

第二是**自動化的判準與回滾**。真正的 canary 要事先定義好「達標」與「不達標」的客觀判準（錯誤率、p99 延遲、特定業務指標的閾值），由系統在放量途中持續比對 canary 與 baseline 的指標，達標自動往下一階放、不達標自動把權重打回 0。靠人盯 dashboard 憑感覺放量，問題在於人會累、會分心、會在凌晨三點把「看起來還行」當成「沒問題」。**沒有自動分析與自動回滾的 canary，等於把炸彈的引信拉長——它讓壞版本以慢動作鋪開，而不是真的在拆彈。**

在 Kubernetes 生態裡，把 canary 做成自動化的工具（如 Argo Rollouts、Flagger 這類 progressive delivery controller，2026-06）正是在補這一塊：它們接上 metric provider（Prometheus、Datadog、CloudWatch 等），把「放量還是回滾」的決定從「時間驅動」（等個幾分鐘就放）變成「指標驅動」（指標達標才放），這才是 canary 與「分批慢慢發」的本質區別。

## 三者不是三選一

讀到這裡很容易以為要在三者間挑一個，但實務上它們常常疊在一起用，因為它們管的是不同維度：**canary 管「放量的節奏與判準」，rolling 管「底層 replica 怎麼一批批替換」，blue-green 式的「兩套環境」管「出大事時整批切回的速度」。** 一個成熟的 progressive delivery 流程，可能是用 canary 的權重控制決定每一階放多少流量、底層用 rolling 把 pod 逐批換成新版、同時保留舊版的一套環境隨時可切回。它們不是競爭者，是同一套「管理換版窗口」工具箱裡的不同工具。

選哪一套、疊到多複雜，回到一條判準：**這套機制擋下的風險，值不值得它的複雜度與成本。** 一個每天部署一次、流量很小的內部服務，硬上一整套帶自動 metric 分析的 canary pipeline，維運這套東西的成本遠超過它擋下的風險——那是把複雜度本身變成了新的故障源。反過來，一個每秒幾萬請求、發布出錯會上新聞的服務，rolling 的「慢動作回滾」和 blue-green 的「全量瞬切」可能都不夠，canary 的細粒度放量才配得上它的風險。沒有最好的策略，只有與你的流量規模、回滾速度需求、可觀測性成熟度相稱的策略。

## 那條所有策略都繞不過的暗礁

不管你用 rolling、canary 還是 blue-green，只要你要零停機，你就接受了「新舊版同時在線」這個前提；而這個前提拖出一連串約束與失效模式，是整個部署主題真正的硬核。把它們收在一起看：

**新舊版必須雙向相容。** rolling 和 canary 期間，舊 client 可能打到新版、新版寫的資料舊版可能要讀。任何單向的破壞性變更——直接 rename 一個 API 欄位、改掉一個訊息格式、移除一個舊版還在用的欄位——都會在共存期間炸掉。發布策略管的是「流量怎麼挪」，它**完全不管你的變更本身相不相容**。一個破壞相容性的變更，配上再完美的 canary，照樣在共存的第一秒就出事。相容性是部署策略的前提，不是它能替你解決的問題。

**回滾不是時光機。** 流量可以切回舊版，但新版已經寫進資料庫的資料、已經發出去的訊息、已經觸發的外部副作用（扣了款、寄了信、呼叫了第三方 API），**不會隨著流量回滾而被撤銷**。回滾是停損，把傷害止在當下；它不能讓已經發生的事沒發生過。一個部署如果在出錯期間污染了資料或發錯了訊息，光把流量切回舊版是不夠的——那些副作用要靠補償邏輯或冪等設計（見〈重複是常態：冪等與去重〉）另外清理。把回滾當成「一鍵當沒事過」，是很多事故在「已經回滾了為什麼還有問題」階段才被發現的根源。

**健康檢查綠燈不等於版本正確。** 前面在 rolling 那裡提過：readiness probe 通常只查 process 活著、能回 200，查不到業務邏輯。這對所有自動化策略都成立——rolling 靠它判斷新 pod 能否接流量、canary 的自動分析如果只盯 HTTP 狀態碼而不盯業務指標，都會被一個「能回 200 但算錯帳」的版本騙過去。所以 canary 的判準要盯到真正的業務 SLI，而不只是「服務還活著嗎」。綠燈只證明「沒死」，不證明「沒錯」。

**舊版要能優雅退場。** 每次替換、每次切換，被換下來的舊 replica 身上可能還有處理到一半的請求、還沒寫完的連線。如果你粗暴地一個 SIGKILL 把它打死，這些在途請求就被截斷、變成使用者眼裡的錯誤。零停機部署的「零停機」，不只要求新版順利接手，也要求舊版**好好地把手上的活做完再退**——這需要負載均衡器先停止往要退場的實例送新請求、等它把在途請求消化完（這個等待窗口在 AWS ALB 上叫 deregistration delay，預設 300 秒，2026-06），同時應用本身收到終止訊號後要 graceful shutdown（這套退場機制見〈graceful shutdown：怎麼好好地關掉一個行程〉）。少了這一環，你的「零停機部署」會在每次部署時悄悄丟掉一小批在途請求——一個只在部署那幾秒、用一般監控很難抓到的失效。

## 為什麼是這個形狀

退一步看，三種策略的全部樣貌，都是從一個無法迴避的取捨裡長出來的：**recreate 用「全停機」買來了「永遠只有一個版本在線」的單純；一旦你要拿掉那段停機，就必須接受新舊共存，而新舊共存的所有複雜度——相容性、回滾的不徹底、健康檢查的盲點、優雅退場——就全都跟著來了。**

rolling、blue-green、canary 的差異，本質上是它們各自選擇把「換版窗口的風險」往哪裡擠：rolling 擠向「省資源、但回滾慢」，blue-green 擠向「回滾快、但資源翻倍且被共用資料庫卡住」，canary 擠向「blast radius 最小、但要有足夠流量與夠好的可觀測性才玩得起來」。沒有一種策略消滅了風險——它們只是把同一份風險，重新分配到一個你更能承受的維度上。

理解了這一點，下次你看到一個部署在 canary 階段被自動打回、或一個 blue-green 切換後又緊急切回 blue，你會知道那不是流程出了錯——那正是這套機制在做它該做的事：在新版注定有某個你還沒發現的問題的前提下，把那個問題擋在它能傷到的最小範圍之內。
