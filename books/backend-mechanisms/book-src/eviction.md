# 驅逐：LRU、LFU、TTL 與 LRM

你給快取設了 `maxmemory 4gb`。它運行了三個月，相安無事——直到某天凌晨，監控開始噴出零星的寫入錯誤，接著是一連串 `OOM command not allowed when used memory > 'maxmemory'`。資料量爬到了上限。這一刻，快取面對一個它每天都要回答無數次、但你可能從沒認真想過的問題：**記憶體滿了，又有一筆新資料要進來，我該丟掉現有的哪一個鍵？**

這個問題沒有正確答案，只有「對你的存取樣式最不糟」的答案。丟錯了——把馬上要被讀的熱鍵踢出去——下一秒它又 miss 回填，你的快取命中率塌一截，後端跟著扛起本來該被擋住的流量。丟對了——騰掉一個再也不會被碰的冷鍵——使用者甚至不會察覺。而快取沒有預知能力，它只能用「過去的存取痕跡」去**猜**未來。整個驅逐（eviction）這門學問，講的就是怎麼用最少的記憶體、最少的 CPU，把這個猜測做得夠準。

## 先分清楚：驅逐不是過期

開始之前要釘死一組常被混為一談的詞，否則後面全會錯位：**驅逐（eviction）和過期（expiration）是兩個獨立的機制，由兩件完全不同的事觸發。**

過期是**時間**驅動的。你給一個鍵設了 TTL（`SET session:abc ... EX 3600`），一小時後它就該消失——不管記憶體還剩多少、不管你有沒有在寫新資料。Redis 清理過期鍵有兩條路：被動的（lazy，下次有人來存取這個鍵時，發現它過期了，當場刪掉再回 nil）和主動的（active，背景任務週期性地隨機採樣一批設了 TTL 的鍵、把已過期的清掉）。兩條路合起來，保證過期鍵不會無限期佔著記憶體，但也意味著「一個鍵的 TTL 到了」和「它真的從記憶體消失」之間有個時間差——它可能已經邏輯上死了，卻還佔著位子，直到下次被採樣到或被存取到。

驅逐是**空間**驅動的。它跟 TTL 無關，只在一個條件下被觸發：**已用記憶體碰到 `maxmemory`，而你又要寫入新東西。** 這時 Redis 必須先騰出空間，才能容下新資料——它會依照你設的 `maxmemory-policy`，挑一個（或幾個）既有鍵刪掉，直到記憶體降回上限以下。

兩者的關係是：過期讓鍵「該走的時候走」，驅逐讓鍵「沒地方站的時候被推下車」。一個鍵可能 TTL 還沒到、卻因為記憶體滿了被驅逐；也可能根本沒設 TTL（永不過期），卻在某次寫入壓力下被驅逐。把這兩件事分開，是讀懂後面所有策略的前提——尤其是 `volatile-*` 那一組，它正卡在兩者的交界上。

## 天真的做法：精確 LRU，與它的隱形成本

直覺上，最該被丟的鍵是「最久沒被用到的那個」——這就是 **LRU（Least Recently Used，最近最少使用）**。它背後的賭注很合理：剛被存取過的資料，很可能馬上又被存取（時間局部性）；很久沒人碰的，大概也不會再被碰。對絕大多數真實 workload，這個賭注贏面很大。

問題是，要做到**精確**的 LRU，代價比你想的高。教科書的標準解法是一個 hash map 配一個雙向鏈結串列（doubly-linked list）：hash map 給你 O(1) 的查找，鏈結串列維護一條「從最近用到最久沒用」的順序。每次某個鍵被存取，你把它對應的節點從串列裡摘下來、接到串列頭；要驅逐時，砍掉串列尾那個節點就是最久沒用的。查找、移動、刪除全是 O(1)，漂亮。

漂亮在演算法課上，落到一個要管上千萬個鍵的記憶體儲存上就變成奢侈。問題出在那個雙向鏈結串列：**每一個鍵，都得額外掛兩根指標**（prev 與 next），在 64-bit 機器上就是每個鍵 16 bytes 的純管理開銷。一個快取若存著一千萬個小鍵，光是維護這條 LRU 順序就要燒掉約 160 MB——這些記憶體沒存任何使用者資料，純粹是為了「記住誰最久沒被用」。更糟的是並發：每一次讀取——連單純的 `GET`——都要去改那條全域串列的頭，這意味著每個讀操作都得碰一個全域共享的資料結構。在 Redis 那種「指令在單執行緒上序列執行」的模型裡（見〈Redis 的內部〉），這還只是多幾個指標操作；但在多執行緒的快取裡，這條全域串列會立刻變成一把人人爭搶的鎖，把多核的吞吐全卡死。

於是真實系統面對一個取捨：要嘛付出可觀的記憶體與並發代價，換來「分毫不差的 LRU 順序」；要嘛承認——**我其實不需要分毫不差。** 我只是要丟掉一個「夠舊」的鍵，不是非得丟「全域最舊」的那一個。這個讓步，正是現代快取驅逐的設計起點。

## 近似 LRU：用採樣換掉那條串列

Redis 的選擇是放棄精確，改用**近似 LRU（approximated LRU）**——而且這個近似做得相當聰明，值得在腦中重放一遍。

它先把「最近用過沒」這件事壓縮到極致。每個物件的標頭裡塞一個 **24-bit 的 LRU 時鐘值**（秒級解析度），存的是「這個鍵上次被存取的時間」。24 bits 在秒級下能表示約 1678 萬秒、繞約 194 天才回卷一次——對快取夠用了。關鍵是：這個時鐘戳就藏在物件本來就有的標頭裡，**不額外掛任何指標、不維護任何串列**。每次存取一個鍵，就把當前時間寫進它的時鐘戳，僅此而已。記住 recency 的成本，從「每鍵兩根指標 + 一條全域串列」降到「每鍵幾個 bit、各自為政」。

代價是你失去了那條現成的排序——要驅逐時，沒有一條串列的尾巴可以直接砍。Redis 補上的是**隨機採樣**：要騰位時，它從鍵空間隨機抓 `maxmemory-samples` 個鍵（預設 **5** 個），比較它們的閒置時間（idle time，當前時間減去時鐘戳），把最閒的那個驅逐掉。只看 5 個、不看全部——所以它幾乎一定不是「全域最舊」，但統計上會是「相當舊」的一個。把 `maxmemory-samples` 調大（比如 10），抽樣越多、結果越接近真 LRU，但每次驅逐燒的 CPU 也越多。這是個準確度對 CPU 的旋鈕。

光是隨機抽 5 個還不夠準——每次驅逐都是獨立一抽，上次抽到的好候選這次就忘了。所以 Redis 3.0 起加了一個 **eviction pool**：一個固定 16 格的小池子。每次採樣，不是抽完就丟，而是把抽到的鍵按閒置時間插進這個池子（最閒的排前面），跨多次驅逐**累積**好候選；真要刪時，從池子裡挑最閒的那個下手。這個池子讓近似 LRU 的品質明顯往真 LRU 靠攏，而成本只是 16 格的記憶體。

把這套放慢看一遍：**24-bit 時鐘戳記下 recency（省掉串列）→ 驅逐時隨機採樣幾個鍵（省掉全域排序）→ 用一個 16 格的池子跨輪累積候選（把品質補回來）。** 整條設計的每一步，都是在「精確」和「便宜」之間刻意選了後者，再用工程手段把精確度盡量贖回來。理解了這個，你就懂為什麼 Redis 文件總是強調「我們的 LRU 是近似的」——那不是它做不到精確，是精確不划算，而近似在實務上已經夠好。

## LRU 的盲點：一次掃描就能毀掉它

近似不近似是實作細節，LRU 還有一個**原理層**的弱點，不管你做得多精確都躲不掉——它只看「最近有沒有被碰」，不看「被碰過幾次」。這個盲點有個經典的觸發情境：掃描。

設想一個快取容量上限是 **1,000 個鍵**，裡頭住著 10 個長期熱鍵——每秒各被讀 100 次的那種真正的明星資料——加上一些常用鍵，剛好把 1,000 格填滿。現在有人跑了一個分析查詢，做了一次全表掃描，陸續把 **5,000 個一次性的冷鍵**讀進快取。容量只有 1,000，每讀進一個新冷鍵，就得驅逐一個既有鍵騰位。

問題來了：這些冷鍵**剛剛**才被讀進來，它們的 recency 是全場最新的。在 LRU 眼裡，「剛被讀」就是「最近用過」，所以這 5,000 個用完即丟的冷鍵全排在驅逐隊伍的最後面，而那 10 個真正的熱鍵——它們上次被讀也許是 50 毫秒前——反而相對「舊」，被一個個擠出去。掃描結束後，快取裝滿了再也不會被碰的冷鍵，10 個熱鍵全沒了。接下來每一個對熱鍵的請求都是 miss，全打到後端。這叫**快取污染（cache pollution）**：一次性的大量存取，把 recency 訊號徹底沖刷，把真正有價值的資料洗出去。

LRU 在這裡輸，輸在它的賭注前提——「剛被碰的很可能再被碰」——對一次性掃描根本不成立。那些冷鍵被碰了，但永遠不會再被碰。要修這個盲點，你得換一個賭注：別賭 recency，賭 **frequency**。

## LFU：賭頻率，但要會遺忘

**LFU（Least Frequently Used，最近最不常使用）**把問題重新定義：該丟的不是「最久沒碰的」，而是「被碰得最少的」。回到剛才那個場景——10 個熱鍵每秒被讀 100 次，5,000 個冷鍵各只被讀一次。在 LFU 的記分板上，熱鍵的計數遠遠輾壓冷鍵，所以掃描湧進來時，先被驅逐的是那些只被讀一次的冷鍵，10 個熱鍵穩穩留存。同樣的掃描，LFU 毫髮無傷。這就是讀重、又有掃描型查詢的 workload 上，LFU 勝過 LRU 的典型情形。

但「數每個鍵被存取幾次」這件事，天真地做又會撞上兩道牆。

第一道是**記憶體**——又是它。若每個鍵都掛一個精確的 64-bit 計數器，你又回到了「為了管理而燒記憶體」的老問題。Redis 的解法是一個叫 **Morris counter** 的機率計數器：每個鍵只用 **8 bits** 來近似它的存取頻率。它不是每次存取就 +1，而是**機率性地**增加——計數越高，下一次存取能讓它再漲的機率越低。這條對數型的曲線由 `lfu-log-factor`（預設 10）控制，讓這個 8-bit 的小計數器能覆蓋從「被讀幾次」到「被讀上百萬次」的巨大動態範圍：預設配置下，計數器要到約一百萬次存取才會飽和。代價是它記的是個**估計值**、不是精確次數——但驅逐要的本來就只是「誰比誰常用」的相對排序，估計值足矣。

第二道牆更微妙，是**時間**。純粹的「累積次數」有個致命缺陷：一個曾經爆紅、現在已經沒人看的鍵——比如上週的熱門商品——它的歷史計數高得嚇人，會永遠賴在快取裡不走，把記憶體佔給一段早就過去的人氣。LFU 若不會遺忘，就會被歷史綁架。所以 Redis 的 LFU 計數器會**隨時間衰減**：由 `lfu-decay-time`（預設 1 分鐘）控制，一個鍵越久沒被存取，採樣到它時計數就被往下調得越多。這樣「曾經很熱、現在已冷」的鍵會慢慢掉下記分板，最終被驅逐。**「衰減」是 LFU 能用的關鍵——沒有它，LFU 不是在追蹤頻率，而是在膜拜歷史。** LFU 容易被想成一個單純的累加計數器；但少了衰減這個修飾詞，它追蹤的就不是頻率、而是歷史。真正的 LFU 是「帶遺忘的近似頻率」，這兩個修飾詞缺一不可。

到這裡，LRU 與 LFU 的分工就清楚了：**LRU 賭時間局部性，對「忽冷忽熱、近期活躍」的資料反應快，但怕掃描污染；LFU 賭存取頻率，對「穩定熱點」守得住、抗掃描，但新竄起的熱鍵需要時間累積計數才會被認可（冷啟動較慢）。** 沒有哪個全勝，只有哪個更貼合你的存取分布。

## volatile 那一條岔路，與一個會靜默退化的陷阱

到目前為止講的都是「淘汰準則」——用 recency 還是 frequency 來排序。Redis 的策略名其實是兩個維度的笛卡兒積：**準則**（lru / lfu / random / ttl / lrm）乘上**候選範圍**（`allkeys-*` 對上 `volatile-*`）。準則決定「在候選裡挑誰」，候選範圍決定「誰有資格被挑」。

- `allkeys-*`：**所有**鍵都是驅逐的候選，不管有沒有設 TTL。
- `volatile-*`：**只有設了 TTL 的鍵**才是候選；那些沒設過期時間的鍵，被視為「不可丟」，驅逐永遠不會動到它們。

`volatile-*` 存在的理由很實際：你常常會在**同一個 Redis 實例裡混放兩種資料**——一批是純快取（可以隨時重算、丟了無所謂），另一批是只存在於 Redis 裡的真相（某些 session、計數器，丟了就回不來）。你想要的行為是「記憶體滿了，去淘汰快取那批，但別碰真相那批」。做法就是：給快取資料設 TTL，給真相資料不設 TTL，然後用 `volatile-lru`（或 lfu）。驅逐只在「設了 TTL 的鍵」裡挑，真相資料因為沒 TTL 而免疫。這是個乾淨的設計——用 TTL 的有無，當「這個鍵可不可以被犧牲」的標記。

但這條岔路上埋著一個**會靜默吃掉你系統**的陷阱，務必看清。`volatile-*` 的前提是「候選集合裡有東西可挑」。如果某個時刻，**所有設了 TTL 的鍵都剛好被過期清掉了，或者你壓根忘了給任何快取鍵設 TTL**——那麼當記憶體撞上 `maxmemory`、需要驅逐時，Redis 在 `volatile-*` 的候選集合裡**找不到任何鍵可以淘汰**。這時它的行為會**退化成 `noeviction`**：寫入直接報錯。

這個失敗模式陰險在哪？它不是設定錯誤那種一啟動就炸的明顯故障，而是**潛伏的**。你的系統可能跑了好幾個月都沒事——因為記憶體一直沒滿，或者每次要驅逐時剛好有 TTL 鍵可挑。直到某天，業務量上來、記憶體爬到上限，而那一刻可淘汰的 TTL 鍵又恰好不夠，寫入就開始噴 OOM 錯誤。直覺上設了 `volatile-lru` 就等於有了驅逐保護；但在「沒有可淘汰的 TTL 鍵」這個邊界上，它會悄悄變回「滿了就掛」。用 `volatile-*` 時，你其實是在對自己承諾「我會持續保證有足夠的、設了 TTL 的鍵可供犧牲」——這個承諾若沒人守，保護就是空的。

順帶說，`volatile-*` 家族裡還有一個 `allkeys-*` 沒有的成員：`volatile-ttl`。它的準則既不是 recency 也不是 frequency，而是「剩餘 TTL 最短的先走」——反正快過期的鍵留著也活不久，不如提早送走、把位子讓給能活更久的。它只在 `volatile-*` 語境下有意義（沒 TTL 的鍵根本沒有「剩餘 TTL」可比），所以沒有對應的 `allkeys-ttl`。

## LRM：當「被讀」不該算數

到這裡，Redis 的驅逐策略已經能排出一張九宮格：`{lru, lfu, random}` × `{allkeys, volatile}` 共六種，加上只屬於 volatile 的 `volatile-ttl`，再加 `noeviction`——這就是長年穩定的**八種**經典策略。但 2026 年，Redis 8.6 又補進了一對新成員，填上一個此前沒有任何策略能覆蓋的縫隙：**LRM（Least Recently Modified，最近最少修改）**，包括 `volatile-lrm` 和 `allkeys-lrm`。8.6 起，核心策略總數來到**十種**（2026-06）。

LRM 機制上幾乎是 LRU 的孿生兄弟——同樣用近似採樣、同樣比較一個時間戳挑最舊的。唯一、但決定性的差別在於：**那個時間戳只在「寫入」時更新，「讀取」完全不算數。** LRU 的時鐘戳是讀也更新、寫也更新（任何存取都刷新 recency）；LRM 的時鐘戳只認修改，一個鍵被讀一萬次，它在 LRM 眼裡的「最後修改時間」一動不動。

這個差別解的是什麼問題？設想一類資料：**被狂讀，但很久不再被改。** 一份已經定稿、之後只供查詢的設定；一個寫入後就凍結、只剩讀流量的紀錄。在 LRU 下，這些鍵因為被讀得很勤，recency 一直被刷新，永遠排在驅逐隊尾、賴著不走——可是它們其實是「死資料」，內容早就定型，把它們從快取丟掉、需要時再從後端讀回來，成本並不高。真正該被保護的，是那些**正在被更新的活資料**。LRM 正是為這個場景設計：它把「最近有沒有被改」當作「這個鍵還活不活躍」的訊號，於是「只被狂讀、不再被改」的死資料會被優先淘汰，把寶貴的記憶體留給真正在變動的活資料。

但這也劃出了它的危險邊界，得講清楚：LRM 會**優先驅逐被高頻讀取的鍵**——這正是它的目的，卻也正是它最容易被選錯的地方。如果你的意圖其實是「保住熱讀資料」，那 LRM 會做出和你期望完全相反的事：它會專挑你讀得最勤、卻不去改的那些鍵下手。LRM 適合的是「讀重寫少、且想先趕走只讀不改的靜態資料」的 workload；用在「想留住熱讀資料」的場景，就是拿錯了工具。選驅逐策略時，先問自己一句：對我的資料，「該留下」的訊號到底是「最近被讀過」「常被讀」還是「最近被改過」？——這三個答案，分別指向 LRU、LFU、LRM。

## 不只是 Redis：同一個問題的不同長相

驅逐不是 Redis 獨有的難題，任何「容量有限、要決定丟誰」的快取都得面對。看一眼別的系統怎麼解，能讓你更看清這類問題的形狀。

Memcached 走的是另一條路。它的記憶體按 **slab class** 切成不同大小的塊，而 **LRU 是 slab-specific 的**——每個 slab class 各自維護一條 LRU 鏈。這帶來一個 Redis 沒有的怪現象：如果裝小物件的 slab class 滿了、但裝大物件的 slab class 還很空，Memcached **不會**為了塞一個小物件去驅逐一個大物件騰的空間——它只會在那個塞不下的 slab class 內部，砍掉最舊的小物件——驅逐預設被鎖在 class 內。（現代 Memcached 自 1.5.0 起預設開啟 slab automover，背景執行緒會把某個 class 多出來的整頁搬給驅逐壓力大的 class，緩解這種卡死；但它保守、慢，且仍是「先有 class 內驅逐壓力、才觸發跨 class 搬頁」，不是即時的記憶體自由流動。）

更有意思的是 Memcached 後來的 **segmented LRU**：把每條 LRU 再切成 HOT / WARM / COLD 三段子串列（另有一段 TEMP 專收極短 TTL 的物件），由背景執行緒搬運。新鍵先進 HOT；HOT 裡的鍵走到尾端時，若還活躍就升去 WARM、不活躍就降去 COLD；驅逐永遠從 COLD 的尾巴開始。為什麼要這麼麻煩？因為前面提過的那個並發痛點——精確 LRU 每次讀都要去動全域串列的頭（item bumping），在多執行緒下是把全域鎖。分段 LRU 讓「直接命中的讀取不必每次都搬鍵到串列頭」，於是你可以一口氣讀 500 個鍵而不必為每一個都去搶 LRU 鎖。**這是 Memcached 用結構複雜度，去換 Redis 用單執行緒模型自然避開的那個並發問題**——同一個「LRU 維護成本」的難題，在不同的執行模型下長出了不同的解。

兩相對照，你會看到驅逐這件事沒有銀彈：Redis 用近似採樣 + 單執行緒迴避並發；Memcached 用 slab 隔離 + 分段 LRU 對抗多執行緒鎖爭用。它們的存取局部性假設、記憶體佈局、執行模型都不同，於是同一個「丟誰」的問題，被各自的約束壓成了不同的形狀。

## 別忘了：很多時候你根本不在驅逐

收束前，講一個容易被忽略、卻最該先確認的事實：**在很多系統裡，驅逐策略其實從來沒被觸發過。**

驅逐只在「已用記憶體撞上 `maxmemory`」時才啟動。如果你的資料總量遠小於 `maxmemory`——比如你配了 16 GB，實際只用 2 GB——那麼無論你把 `maxmemory-policy` 設成 LRU、LFU 還是 LRM，**它們一行程式碼都不會被執行到**，因為永遠沒有「需要騰位」的時刻。這種情況下糾結於選哪個驅逐策略，是在優化一條走不到的路徑。你真正在用的清理機制，是 **TTL 過期**——那才是把舊資料請出去的力量。

所以面對一個快取，順序應該是：先量你的資料量會不會逼近 `maxmemory`。會，驅逐策略才是個真問題，這時前面講的 LRU/LFU/LRM 取捨才值得認真挑。不會，那把心思花在 TTL 設計上（多長、要不要加抖動防雪崩、見〈快取的失效災難〉）遠比挑驅逐策略實在。**驅逐是「記憶體不夠」的應急機制，不是「資料該不該留」的日常管理工具**——把這兩件事搞反，你會在一個從不觸發的旋鈕上耗掉精力。

## 為何是這個形狀

退一步看，整章的每一個設計選擇，都從同一個無法迴避的處境裡長出來：**快取的記憶體有限，而它無法預知未來——它只能用過去的存取痕跡，去猜哪個鍵最不會被需要，再賭一把把它丟掉。**

正因為要「記住過去」本身就有成本，所以連最直覺的精確 LRU 都嫌貴——那條雙向鏈結串列的指標與鎖，逼出了近似採樣、24-bit 時鐘戳、eviction pool 這一整套「夠好就行」的工程。正因為「過去」可以用不同方式解讀，才分岔出 recency（LRU）、frequency（LFU）、modification（LRM）三種賭注，各自押中一類存取分布、也各自有一種能打敗它的樣式——掃描污染之於 LRU、歷史綁架之於不會遺忘的 LFU、熱讀資料之於 LRM。正因為一個實例常混放「可丟的快取」與「不可丟的真相」，才需要 `volatile-*` 用 TTL 當犧牲標記——也才埋下「沒有可淘汰的 TTL 鍵時靜默退化成拒寫」這個陷阱。

驅逐策略不是一張隨意拼湊的選單，而是「在有限記憶體裡，用不完整的資訊做不可撤回的丟棄決定」這個問題，被各種約束反覆擠壓後留下的幾種形狀。下次你在 `maxmemory-policy` 那一行猶豫該填什麼，真正要回答的不是「哪個最好」，而是一連串更具體的問題：我的資料量會不會真的撞到上限？對我的鍵，「值得留」的訊號是最近被讀、常被讀、還是最近被改？我這個實例裡，有沒有混著不能被當柴燒的真相資料？把這幾個問題答清楚，那一行設定自己就浮出來了。
