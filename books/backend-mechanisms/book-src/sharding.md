# 分片：水平切分與路由

有一張訂單表，二十億列，跑在一台你能租到的最大機器上。讀已經靠副本擋掉了，熱資料也都進了快取，索引調到不能再調——但寫入還是頂到天花板。每一筆下單都要進這台機器的磁碟，而磁碟的 IOPS、WAL 的吞吐、單機的連線數，全都是硬上限。你沒辦法再把這台機器變大兩倍，因為市面上根本沒有大兩倍的機器；就算有，價格曲線也早就翹到不值得。

到這一步，唯一的出路是把這張表**切開**，分到很多台機器上，每台只存其中一部分列。這就是分片（sharding，水平切分）。聽起來只是「把資料攤平到多台」這麼一句話，但這一刀下去，你失去的東西遠比想像中多——而拿回那些東西的代價，就是這一章要講的全部。

## 分片把規模問題換成另一個問題

先把第一原理講清楚，不然後面所有的痛都會顯得像意外。

單台資料庫之所以好用，是因為**所有資料都在同一個地方**。一筆交易可以原子地改三張表，因為它們共用同一個交易引擎；一個 `UNIQUE` 約束能保證全表沒有重複的 email，因為整張表的索引就在那一台機器上；一句 `JOIN` 能把訂單和使用者拼起來，因為兩邊的資料庫頁面都在同一塊記憶體裡可達。這些保證——原子交易、全域唯一、任意 JOIN、線性一致的讀——全都建立在「資料共處一地」這個前提上。

分片做的事，是**把這個前提拆掉**。資料一旦散到多台機器，上面那些保證就不再免費。同一個分片**內部**，你還保有資料庫原本的全套保證（單分片內的交易、單分片內的唯一性、單分片內的 JOIN 都還在）；但只要一個操作要碰到**兩個以上**的分片，你幾乎什麼都得自己重新補。

所以分片的本質不是「把規模問題解決了」，而是**把「規模」這個問題，換成「跨分片協調」這個問題**。換得划不划算，完全取決於你切的那一刀，有沒有讓「絕大多數操作只落在單一分片內」。切對了，你用十六台機器扛十六倍的寫入，而幾乎每個操作還是像在單機上跑；切錯了，你付了十六台的營運複雜度，每個操作卻都要跨遍所有分片——規模沒擴成，只是把一台機器的麻煩乘以十六。

這就是為什麼這一章的重心不在「怎麼把表切開」（那是最簡單的部分），而在**那把切分的鍵怎麼選、請求怎麼被路由到對的分片、以及那些被迫跨分片的操作怎麼辦**。

## 切分鍵：一切的源頭

每一筆資料要落到哪個分片，由它的**切分鍵（shard key，又稱 partition key）**決定。你拿這筆資料的某個欄位——`user_id`、`order_id`、`tenant_id`——餵給一個函式，函式吐出一個分片編號。切分鍵選什麼、那個函式長什麼樣，決定了整個系統的命運：資料分佈得均不均、熱點會不會集中、哪些查詢能落單片、哪些被迫掃全部。

最直覺的函式是雜湊取模：`shard = hash(key) % N`，N 是分片數。把鍵雜湊一下打散，再對分片數取模分桶。資料分佈會很均勻——好的雜湊函式讓任何鍵都等機率落到任何一桶，不會像「按字母 A–M、N–Z」那樣讓姓 S 開頭的人擠爆某一片。

但 `hash(key) % N` 藏著一個會在你擴容那天爆炸的地雷，值得停下來算清楚。

假設現在 N=4，某個鍵 `hash=100`，`100 % 4 = 0`，它住在 0 號分片。有一天你加一台機器，N 變成 5，同一個鍵 `100 % 5 = 0`——剛好沒搬。但換個鍵 `hash=101`：原本 `101 % 4 = 1`，擴容後 `101 % 5 = 1`，也沒搬。再換 `hash=102`：`102 % 4 = 2` 變成 `102 % 5 = 2`……看起來都沒搬？別被小數字騙了。算一批：`hash` 從 100 到 119 這二十個鍵，N 從 4 變 5 之後，**有多少個鍵的歸屬分片改變了**？

| hash | % 4 | % 5 | 搬了嗎 |
|---|---|---|---|
| 100 | 0 | 0 | 否 |
| 101 | 1 | 1 | 否 |
| 102 | 2 | 2 | 否 |
| 103 | 3 | 3 | 否 |
| 104 | 0 | 4 | **搬** |
| 105 | 1 | 0 | **搬** |
| 106 | 2 | 1 | **搬** |
| 107 | 3 | 2 | **搬** |
| 108 | 0 | 3 | **搬** |
| 109 | 1 | 4 | **搬** |
| 110 | 2 | 0 | **搬** |
| 111 | 3 | 1 | **搬** |
| 112 | 0 | 2 | **搬** |
| 113 | 1 | 3 | **搬** |
| 114 | 2 | 4 | **搬** |
| 115 | 3 | 0 | **搬** |
| 116 | 0 | 1 | **搬** |
| 117 | 1 | 2 | **搬** |
| 118 | 2 | 3 | **搬** |
| 119 | 3 | 4 | **搬** |

前四個碰巧沒動，是因為 100 剛好是 lcm(4,5)=20 的倍數、落在週期邊界上，於是 100–103（餘數 0–3）的 `%4` 與 `%5` 暫時同步；過了餘數 3 就開始分岔，從 104 起幾乎每個都搬。數一下這二十列：搬了 16 個、留了 4 個，正好 **80% 換了家**。一般地，`hash % N` 在 N 變到 N+1 時，平均只有約 `1/N` 那一小撮鍵留在原地（更精確是 `min(M,N)/lcm(M,N)`，這裡 4/20=1/5），**絕大多數鍵都得搬到新機器**。

「搬」在分片的世界裡不是免費複製一下。每一個換家的鍵，都意味著一次跨機器的資料搬移；搬移期間，這筆資料到底該去舊位置找還是新位置找，是個競態；幾百 GB 的資料在網路上流動，會吃滿頻寬、拖慢正常流量；而且這是**幾乎全表一起搬**，不是搬一小撮。一次「加一台機器」的擴容，變成一場波及全叢集、持續數小時、期間一致性與可用性都岌岌可危的大遷徙。

讓擴縮容時只搬「應該搬的那一小撮」（約 1/N），正是 **consistent hashing** 要解的問題——它的機制（雜湊環、virtual node 怎麼讓負載均勻）是另一個主題（見〈consistent hashing〉）。這裡只要記住結論：**`hash % N` 能用來把資料打散，但不能拿來當「節點數會變動」的長久方案。**

## 真正的系統都隔了一層

不過，連 consistent hashing 都不是大型生產系統最常見的做法。看一眼真實系統怎麼路由，會發現它們幾乎都在「鍵」和「實體分片」之間，**插了一層間接（indirection）**。

Redis Cluster 是最乾淨的例子。它不把鍵直接對應到節點，而是先把鍵映射到 **16384 個固定的雜湊槽（hash slot）**之一：`slot = CRC16(key) % 16384`。注意這個取模的分母是 **16384，永遠不變**，跟你有幾台機器無關。然後，這 16384 個槽再被分配給實體節點——三台機器時，可能 0–5460 給 A、5461–10922 給 B、剩下給 C。鍵到槽的映射是純函式、永遠固定；會變的只是「槽到節點」這層分配表。

這一層間接帶來的好處是決定性的。當你加一台機器，你**不重算任何鍵的雜湊**——CRC16 的結果不變、槽號不變。你只是把一部分槽（連同它們底下的資料）從舊節點搬到新節點，並更新「槽到節點」這張小小的分配表。要搬多少資料、什麼時候搬、搬哪些槽，全在你掌控之中，而不是「N 一變，全表雞飛狗跳」。Redis Cluster 搬槽時，被搬的槽會進入 `MIGRATING`／`IMPORTING` 的過渡狀態，逐 key 搬移，期間還能對外服務，搬完才更新分配表——整個過程是可控的滴漏，不是潰堤。

為什麼是 16384 這個數字？是個務實的折衷：槽的歸屬要在節點間用心跳訊息廣播，16384 個 bit 的點陣圖剛好約 2 KB、塞得進心跳封包；同時它又遠大於可能的節點數（幾百台），保證每台機器能分到夠細的槽、負載能切得夠均。槽數固定、且遠多於節點數——這兩點是它能優雅擴縮的關鍵。

Vitess（MySQL 的水平分片方案）走的是同一個哲學，只是換了名字。它把切分鍵透過一個叫 **vindex** 的函式算成一個 **keyspace ID**，再把 keyspace ID 的值域切成連續區間分給各分片。要 resharding（改變分片數）時，它在背景把來源分片的資料複製到目標分片、追平複製、逐筆比對校驗，全程舊分片照常服務讀寫；等資料追齊、校驗通過，才切換流量，只有**幾秒鐘的唯讀視窗**。DynamoDB 在底層也是類似的概念：你給一個 partition key，它雜湊後決定資料落在哪個內部 partition，而 partition 怎麼分裂、怎麼搬，是它替你管的。

這裡浮現一個值得記住的模式：**幾乎所有成熟的分片系統，都不讓「鍵」直接綁死「實體機器」，而是中間隔一層邏輯單位（槽、keyspace ID、虛擬桶）。** 因為一旦綁死，擴縮容就會像 `hash % N` 那樣牽一髮動全身；隔了一層，物理拓樸的變動就被這層間接吸收掉了。這是分片設計裡最反覆出現、最值得內化的一招。

## 範圍分片，與它換來的東西

雜湊不是唯一的切法。另一條主線是**範圍分片（range sharding）**：不打散，而是按鍵的**區間**連續地切。`user_id` 1–1000 萬在 shard1、1000 萬–2000 萬在 shard2，依此類推；或者按 `order_date`，一月的訂單在這片、二月在那片。

範圍分片買到一個雜湊分片給不了的東西：**範圍查詢落在單一分片**。「查這個使用者 ID 區間的所有訂單」「拉出三月一號到三月七號的所有交易」——這類查詢在範圍分片下只碰一兩個相鄰分片，因為相近的鍵物理上就住在一起。雜湊分片把鍵打散後，相鄰的 `order_date` 會被雜湊到天南地北的分片，同樣一句範圍查詢就得問遍所有分片。

但範圍分片用一個東西換來這份便利，而那個東西的名字叫**熱點（hotspot）**。

考慮以 `order_date` 做範圍分片。每一筆新訂單的日期都是「現在」，於是**所有的寫入，全部打在持有最新日期的那一個分片上**。其他十五個分片存著歷史資料，幾乎沒有寫流量、在那邊曬太陽；最後一個分片獨自扛下全站的寫壓力，跟你沒分片之前一台機器扛的一模一樣。你買了十六台機器，寫入吞吐卻一點沒漲——因為寫入的分佈不取決於你有幾片，而取決於**切分鍵的值有沒有把寫流量攤開**。任何單調遞增的切分鍵（時間戳、自增 ID）配上範圍分片，都會掉進這個陷阱：新資料永遠擠在尾端那一片。

這就是切分鍵選擇的核心張力，也是它沒有標準答案的原因：

- **雜湊分片**把寫入攤得很均勻、天然避開熱點，代價是**範圍查詢要 scatter-gather**（問遍所有分片再合併）。
- **範圍分片**讓範圍查詢落單片、有序好掃，代價是**單調鍵會造出寫熱點**。

選哪一個，取決於你最高頻、最在意延遲的存取模式長什麼樣。如果系統幾乎全是「給一個 ID、點查一筆」，雜湊分片完勝——點查在哪種切法下都只碰一片，那就選分佈最均的。如果系統的命脈是「掃某個時間窗、某個區間」，範圍分片才對得起那些查詢——但你得另想辦法處理寫熱點（例如在鍵前面加個雜湊前綴把熱點打散，代價是犧牲掉一部分範圍掃描的能力，本質上是在兩種切法之間調一個旋鈕）。

## 熱點不只發生在範圍分片

有一個常被忽略的真相：**就算你用了雜湊分片、自以為躲過了熱點，熱點還是可能找上門。**

雜湊分片保證的是**鍵的數量**均勻分佈——每個分片分到的鍵數量差不多。但它對**每個鍵的存取量**一無所知。如果你的切分鍵是 `tenant_id`，而某一個租戶是其他人的一千倍大（一個爆紅的大客戶），那麼這個租戶的所有資料和所有流量，會全部集中在它雜湊到的那**單一個分片**上。雜湊把它和別人放在同一片，卻沒法把它一個人的流量再切細。這個分片會先燒起來，而其他分片還在閒置。

DynamoDB 把這個問題暴露得最清楚，因為它的限制是寫死的數字：**每個底層 partition 的吞吐上限約是每秒 3,000 個讀取單位、1,000 個寫入單位。** 你的某個熱門 partition key 一旦把流量推過這條線，請求就開始被節流（throttle）——即使你整張表的總配額還遠遠用不完。流量的不均，撞上單 partition 的硬上限。

DynamoDB 的對策叫 **split for heat**：它觀察到某個 partition 持續高熱，就把它一分為二，讓熱資料分到兩個新 partition、吞吐翻倍。但這個機制有幾個非顯然的邊界，恰好示範了「分片救不了什麼」：

- 它**需要幾分鐘才生效**，意味著突發的熱點會先讓你嚐到五到十分鐘的節流，自動分裂才追上來——對一場限時搶購來說，這幾分鐘可能就是事故本身。
- 如果你的熱 partition key 配的是**單調遞增的 sort key**（典型如時間戳），split for heat **判定分裂沒用而不分裂**——因為所有新寫入都落在 sort key 的最尾端，無論你把 partition 從哪裡切開，熱永遠在最新那一半，分裂只是換個地方繼續燒。這跟範圍分片的單調鍵熱點是同一個病根，只是換了層皮。
- 表上若有 local secondary index，會**阻止同一個 item collection 內的分裂**——一個你為了查詢方便而加的索引，悄悄關掉了系統替你救熱點的能力。

把這些串起來看，結論很樸素也很重要：**分片擴展的是「不同鍵之間」的並行度，它從來無法把「同一個鍵」的負載切細。** 一個天生會被瘋狂存取的單一鍵（一篇病毒貼文、一個超大租戶、一個全域計數器），不管你怎麼分片，它都會壓在某一個分片上。對付這種熱點得換武器——把那個鍵本身再拆（給大租戶獨立的子分片、把計數器拆成 N 個分片計數器再彙總），而不是指望分片機制自己變出來。切分鍵選得再好，也只在「鍵夠分散」時有效。

## 跨分片操作：你失去的那些保證，怎麼一個個還

現在來到分片最痛、也最定義它的部分：那些被迫跨越分片邊界的操作。每一個都對應一個你在單機時免費擁有、分片後得自己掏錢買回的保證。

**全域唯一性。** 單機上一句 `UNIQUE (email)` 就保證全表沒有重複 email。分片後，這個約束**只在單一分片內有效**——每個分片各自檢查自己那份資料裡有沒有重複，但兩個相同的 email 如果被路由到不同分片，兩邊各自檢查都通過，重複就這樣溜進系統。資料庫的唯一約束在分片邊界處失效，而且是**靜默地**失效——沒有報錯，只是約束不再成立。要拿回全域唯一，你得在分片機制之外另建：一個中央發號器負責發放保證唯一的 ID，或者把唯一性欄位本身設成切分鍵（這樣相同的值必然路由到同一分片，分片內的 `UNIQUE` 就重新生效了），或者維護一張獨立的全域唯一性索引表。沒有一個是免費的。

**跨分片交易。** 單機上改三張表可以包在一個原子交易裡，要嘛全成、要嘛全敗。如果這三張表被分到三個分片，單一資料庫交易就管不到了——它的原子性止於一台機器的邊界。要讓跨分片的多筆寫入原子地成功或失敗，你得引入分散式的原子提交協定，那是另一套機制、另一套延遲與阻塞代價（見〈2PC 與分散式交易〉）；更常見的做法是退一步，用補償式的流程容忍中間態、事後對帳。無論哪條路，都比「BEGIN; … COMMIT;」貴得多。**好的切分鍵的全部價值，就在於讓需要原子性的那組資料天然落在同一分片**——以 `tenant_id` 分片，單一租戶內部的操作就從來不跨片，交易在分片內就解決了。

**跨分片查詢（scatter-gather）。** 「全站訂單總額」這種要彙總所有使用者的查詢，從一句 SQL 變成**對每個分片各查一次、再把結果合併**——這叫 scatter-gather（散播—收集）。它不只是「多查幾次」這麼簡單，它有一個惡毒的延遲特性：**一個 scatter-gather 查詢的延遲，由它問到的所有分片裡最慢的那一個決定。** 你問十六個分片，只要其中一個正在 GC、正在被別的查詢拖住、正好 p99 抖了一下，整個查詢就跟著慢。問的分片越多，「至少有一個正在抖」的機率越高——這就是**長尾延遲放大（tail latency amplification）**：個別分片偶爾的慢，被 scatter-gather 聚合成「幾乎每次查詢都會撞上某個慢分片」。十六分片時，就算每片 p99 抖動的機率只有 1%，一次 scatter-gather 撞上至少一個慢分片的機率是 `1 − 0.99¹⁶ ≈ 15%`——個別的小機率被放大成系統性的常態。

**二級索引：問題最深的一層。** 假設你以 `user_id` 分片，但常常要「查所有狀態為 pending 的訂單」。`status` 不是切分鍵，那這個查詢怎麼辦？有兩種根本不同的索引擺法，各自的痛點正好相反：

| | 本地索引（document-partitioned） | 全域索引（term-partitioned） |
|---|---|---|
| 索引放哪 | 每個分片各自為自己的資料建 `status` 索引 | 把 `status` 當索引的切分鍵、跨所有資料另切一份全域索引 |
| 寫入 | 只動到資料所在的那一個分片，便宜 | 一筆寫可能動到索引的多個分片（不同 term 落不同片），且常是非同步更新 |
| 按 `status` 查 | 得 scatter-gather 問遍所有分片 | 路由到索引的特定分片，一次命中、不必散播 |
| 本質取捨 | 寫便宜、讀貴（讀要散播） | 讀便宜、寫貴（寫要扇出且最終一致） |

本地索引把痛留給讀——每次按非切分鍵查都得 scatter-gather；全域索引把痛留給寫——每次寫都要更新可能散落在別片的索引，而且為了不讓寫變慢，索引的更新往往做成**非同步**，於是你查全域索引時可能讀到還沒更新完的舊結果。沒有一種擺法能同時讓讀和寫都便宜——這個取捨是分片下二級索引的固有形狀，DynamoDB 的 LSI（本地）與 GSI（全域）正是把這兩種擺法做成了產品選項，連 GSI 的更新是最終一致這件事，也誠實地寫在它的文件裡。

## 怎麼讓相關的資料留在同一片

看完上面那一長串「跨分片有多貴」，你會得到一個強烈的直覺：**設計分片的全部藝術，就是想盡辦法讓「會被一起操作的資料」落在同一個分片上。** 一旦它們同片，交易、JOIN、唯一性全都回到單分片的便宜世界；一旦它們散落，每個操作都要付跨片的稅。

切分鍵的選擇本身就是這件事的第一道防線——以 `tenant_id` 分片，是賭「絕大多數操作都在單一租戶內」；以 `user_id` 分片，是賭「絕大多數查詢都帶著 `user_id`」。選對了，賭贏了，系統大多時候像在單機上跑。

但有些系統需要更明確的「把這幾筆釘在一起」的手段。Redis Cluster 的 **hash tag** 是個很乾淨的示範。預設它對整個鍵算 CRC16 決定槽，但如果鍵裡含有 `{...}`，它**只拿大括號裡的那段**去算槽。於是 `{user1000}:profile`、`{user1000}:cart`、`{user1000}:orders` 三個鍵，因為大括號裡都是 `user1000`，會被算到**同一個槽、落在同一個節點**——這樣你就能對它們跑 `MGET`、跑需要多鍵的 Lua 腳本，而不會撞上 `CROSSSLOT` 錯誤（Redis Cluster 對「鍵分散在不同節點」的多鍵操作會直接回絕，因為它沒法原子地跨節點執行）。

但 hash tag 也示範了同一招的反面風險。如果你貪圖方便，給一大堆鍵都加同一個低基數的 tag（比如所有鍵都 `{global}`），那它們會全部擠進同一個槽、同一個節點——你親手把一個本來能均勻分散的系統，重新捏回了單機熱點。**「把相關資料放一起」和「把資料攤開避免熱點」是一對直接對立的目標**：放得越在一起，操作越便宜、熱點風險越高；攤得越開,熱點越少、跨片操作越多。切分鍵與 co-location 策略，本質上就是在這條軸上選一個點，而那個點該選哪，取決於你的存取模式更怕跨片的稅、還是更怕熱點的火。

## 一個從頭到尾的數字

把這些原理收束在一個具體的例子上。回到開頭那張二十億列的訂單表，單機寫入到頂。

你決定以 `user_id` 雜湊分到 16 個分片。立刻的好處是算得出來的：二十億列平均攤成每片 1.25 億列，寫壓力除以 16——因為任兩筆訂單的 `user_id` 幾乎不同，雜湊把它們均勻打散到十六片，沒有哪一片獨自扛全部。最高頻的查詢 `SELECT * FROM orders WHERE user_id = ?` 依然只碰**一片**，因為切分鍵就是 `user_id`，路由層算一次雜湊就知道該問誰——這個查詢分片前後一樣快，你白賺了十六倍的寫入容量。

代價也算得出來。「全站今日訂單總額」這種跨所有使用者的彙總，從一句 SQL 變成對 16 片各查一次再加總的 scatter-gather，而它的延遲被最慢的那片綁架——前面算過，就算每片 p99 抖動只有 1%，這個彙總撞上慢分片的機率約 15%。「保證 email 全站唯一」也不再能靠 `UNIQUE`，因為兩個相同 email 可能落在不同片各自通過檢查，你得另設中央發號或把 email 設成額外的路由維度。

現在做一個錯誤的對照，看切分鍵選錯有多致命。假設你不選 `user_id`，改用 `order_date` 範圍分片。每一筆新訂單的日期都是今天，於是**全站的寫入，每一筆都打在持有「今天」的那一片上**——其餘 15 片存著歷史訂單在那邊閒置，最後一片獨自扛下你分片前一台機器扛的全部寫壓力。你買了 16 台機器，付了 16 台的營運複雜度、跨片查詢的稅、唯一性的重建成本，而寫入吞吐**一點沒漲**。同一張表、同樣切成 16 片，選 `user_id` 是「寫壓力÷16」，選 `order_date` 是「寫壓力全壓一片」。這兩個結果之間的全部差別，就是那一個切分鍵的選擇——這也是為什麼分片這件事，幾乎所有的重量都壓在「鍵選什麼」這一個決定上。

## 為什麼是這個形狀

退一步看，分片的整個樣貌，都從一個最樸素的物理事實長出來：**一台機器的容量有上限，而資料和流量沒有。**

正因為一台不夠，你得攤到多台——而資料一旦不在同一個地方，「資料共處一地」所支撐的全套保證（原子交易、全域唯一、任意 JOIN、線性讀）就一個個失效，於是分片的核心工作，從來不是「怎麼切開」，而是**怎麼讓絕大多數操作不必跨越那些被切開的邊界**。切分鍵就是那把賭注：賭對了存取模式，操作落單片、像沒分片一樣便宜；賭錯了，每個操作都付跨片的稅，規模沒擴成、複雜度先翻倍。

正因為節點數會變動，而 `hash % N` 會在變動時把全表掀翻，成熟系統才一律在鍵與機器之間隔一層邏輯單位（槽、keyspace ID），用它吸收拓樸的變動。正因為分片只能切開「不同鍵之間」的負載、切不開「同一個鍵」的負載，熱點才會在你以為躲過時找上門，逼你對那一個燒起來的鍵另想辦法。

分片不是一套讓資料庫變大的魔法，而是一筆精算過的交易：你拿「跨分片協調的複雜度」去換「水平擴展的容量」。划不划算，全寫在那一個切分鍵裡。下次你看到一個系統糾結於 shard key 怎麼選、糾結於某個查詢要不要 scatter-gather、糾結於熱租戶把一片燒紅——你會知道那不是它沒設計好，而是它在償還水平擴展這件事，從一開始就欠下的、無法迴避的債。
