# 資料同步的全景：full/incremental、push/pull、batch/streaming

有一張使用者表躺在你的主庫裡，五千萬列。某天，產品要上線一個搜尋功能，於是這份資料得在另一個地方再活一次——一份在 Elasticsearch 裡的索引。問題立刻來了：使用者改了暱稱、刪了帳號、補了個 email，主庫裡的這些變動，要怎麼讓搜尋索引也跟著變？

你寫下第一版，是任何人都會寫的那版：每晚跑一個 job，把整張表 `SELECT *` 拉出來，刪掉舊索引，重建一份新的。它能跑、好懂、無狀態，掛了重跑就好。然後資料長到一億列，這個 job 開始跑四個小時，期間搜尋結果是昨天的；再然後產品要求「改完暱稱五秒內就能搜到」，於是你被逼著去碰那條看起來更聰明的路——只搬「變動的那幾筆」。而就在你動手的那一刻，一整個你原本沒看見的世界打開了：你得知道「上次同步到哪」，你得處理「被刪掉的列」（它連個影子都不留），你得想清楚「同一秒內改了三次的列」怎麼辦。原本一個四行的全量 job，變成一套會在某個深夜悄悄漏資料的管線。

這一章不教你某個特定工具怎麼設定。它要拆的是更底層的東西：**所有資料同步方案，不管叫 ETL、CDC、replication 還是 outbox，骨子裡都是同三個決定的不同取值。** 把這三個決定看穿，後面那些名詞就只是它們的排列組合。

## 同步的第一原理：你永遠在搬「差異」

先把問題剝到只剩一句話。資料同步要做的是：**一份資料的來源版本變了，讓它的目的地版本也跟著變——在可接受的時間內、用可接受的代價。** 來源和目的地可以是同一份資料的兩個副本（主庫到唯讀副本）、兩種儲存（DB 到搜尋索引、DB 到快取）、或兩個服務各自的私有庫。形態千變萬化，但底下的動作只有一個：**把「變更」從一端搬到另一端。**

這句話聽起來像廢話，但它藏著整章的樞紐。「搬變更」這件事，逼你連續回答三個問題，而且這三個問題彼此正交——回答其中一個，不限制另外兩個：

1. **你怎麼界定「變更」？** 是「整份資料就是變更」（每次都當全新的搬），還是「只有跟上次比起來不同的那部分」？這是 **full vs incremental**。
2. **誰來發起這次搬運？** 是來源端主動把變更推出去，還是目的地端週期性回來問「有什麼新的」？這是 **push vs pull**。
3. **變更累積多少才搬一次？** 是攢一大批一起搬，還是一筆變更發生就近即時搬一筆？這是 **batch vs streaming**。

三個軸、各兩個值，理論上八種組合。實務上常見的不過幾種，但重點不是背組合，而是看清**每一軸各自在跟什麼東西做交易**。我們一軸一軸拆，拆到能在腦中重放每個取值會在哪裡幫你、又會在哪裡反咬你一口。

## 第一軸：full 還是 incremental——「自我修復」與「會漂移」的對立

開場那個每晚重建的全量 job，它的好不在簡單，而在一個很容易被低估的性質：**它天生自我修復。** 因為每一次它都把目的地整個覆蓋成來源的當前樣子，所以無論前一晚漏了什麼、錯了什麼、被誰手動改壞了什麼，今晚一覆蓋就乾淨了。它**沒有記憶，所以也沒有會累積的錯**。一個全量同步永遠不會「悄悄漂移」——它要嘛是對的，要嘛差最多一個批次窗口就會被修正。這個「永不漂移」的保證，是它真正的賣點，而代價寫在臉上：成本隨資料量線性成長。一億列就是每次都搬一億列，不管實際上只動了一千列。

增量正好是它的鏡像。增量只搬「上次同步之後變動的那部分」，便宜、可以做到很即時。但它要付一個全量不必付的稅：**它必須記住「上次同步到哪」。** 這個位置有很多名字——watermark、cursor、log offset——但本質都是同一個東西：一個你必須正確維護的狀態。而只要是狀態，它就會錯。算錯一格，你要嘛漏掉介於兩次同步之間的某些變更，要嘛重複搬同一批。**全量把錯誤關在「成本」這個籠子裡；增量把錯誤放進了「正確性」這個更危險的房間。**

最經典的增量做法是時間戳輪詢：來源表有個 `updated_at`，目的地記住上次同步看到的最大時間戳，每輪只拉 `updated_at > watermark` 的列。看起來無懈可擊，直到你撞上三個它沒處理的洞，而這三個洞都是**靜默的**——不會報錯，只會讓兩邊慢慢對不上。

**第一個洞：硬刪除。** `DELETE FROM users WHERE id = 42` 之後，第 42 列從表裡消失了。它沒有 `updated_at` 可以變大，因為它根本不存在了。下一輪輪詢 `SELECT ... WHERE updated_at > X` 永遠掃不到它——它不在結果集裡，因為它不在表裡。於是目的地那邊的第 42 列**永遠留著**，變成一筆「來源已經刪了、索引還在」的鬼資料。使用者刪了帳號，卻還能在搜尋裡被搜到。要堵這個洞，你得改用軟刪除（把 DELETE 變成 `UPDATE ... SET deleted_at = now()`，這樣它就是一筆 `updated_at` 會變大的更新，輪詢看得見），或者根本換一條路——直接讀資料庫的交易日誌，那裡面 DELETE 是一筆實實在在的事件（這正是 CDC 的核心，見〈CDC：從資料庫變更長出事件流〉）。

**第二個洞：邊界那一秒的競態。** 假設你用 `updated_at > watermark`，嚴格大於。某一輪你讀到的最大時間戳是 `10:00:00`，於是你把 watermark 設成 `10:00:00`，下一輪從 `> 10:00:00` 開始。聽起來對。但想想這個排程：有一筆交易在 `10:00:00` 這一秒**開始**寫入、`updated_at` 被設成 `10:00:00`，但它**還沒 commit**——你這一輪的 `SELECT` 看不到未提交的列，所以沒讀到它。等它 commit 完，你的 watermark 已經前進到 `10:00:00` 之後了，下一輪 `> 10:00:00` 又跳過了它。這一筆就此**永久蒸發**，不報任何錯。修法是用「重疊窗口」：watermark 用 `>=` 而非 `>`，每輪重新掃描一小段邊界、容忍重複，再靠目的地用主鍵去重（去重的完整機制見〈重複是常態：冪等與去重〉）。把「可能漏」換成「可能重複」——而重複是能被去重消化的，漏不行。這個取捨，你會在這一章反覆看到，它幾乎是同步世界的母題。但要說清楚：重疊窗口只**縮小**漏的機率窗、不根除它——只要某筆交易的 `updated_at` 早於 watermark、卻在更晚才 commit（一筆跑很久的長交易就會這樣），而它落在你回掃的那一小段之外，它仍會被跳過。牆鐘時間戳當 watermark 的根本缺陷，唯一的根除是下到交易日誌、用日誌位置（而非牆鐘）當游標（這正是 CDC 的核心，見〈CDC：從資料庫變更長出事件流〉）。

**第三個洞：同一格內被改了好幾次。** 一列在你兩次輪詢之間從 `pending` 改成 `paid`、再改成 `shipped`，輪詢只會看到最後的 `shipped`——中間那兩個狀態對下游而言**從未存在**。如果下游只是要「目前狀態」，這無所謂；但如果下游是個審計系統、或在做需要完整變更歷史的事，這就是無聲的資料缺失。輪詢拿的是「現在的快照」，而不是「發生過的順序」，這個區別在很多時候不重要，在某些時候致命。

到這裡，一個帶數字的權衡能讓整件事落地。回到那張表：五千萬列、每列約 1 KB，總共約 50 GB，要同步到搜尋索引。

走**全量**：每次搬 50 GB。假設網路有效吞吐 100 MB/s，光是傳輸就要 `50,000 MB ÷ 100 MB/s = 500 秒`，八分多鐘，而且這 50 GB 對來源庫是一次沉重的全表重讀。資料新鮮度＝最多一個批次窗口的陳舊，若每晚跑一次，就是最多 24 小時舊。

走**增量輪詢**：假設每天實際變動 0.5%，也就是 25 萬列、約 250 MB。傳輸只要約 2.5 秒，新鮮度可以壓到輪詢間隔（比如一分鐘）。便宜了兩個數量級，新鮮多了三個數量級。但這 250 MB 的便宜不是白拿的——你用它換來了上面那三個洞的維護責任。**全量用成本換正確性（永遠不漏），增量用正確性的風險換成本與新鮮度。** 哪個對，取決於你的資料量撐不撐得住全量、你的業務容不容忍陳舊、以及——這是最常被忽略的——**你的來源端能不能可靠地告訴你「什麼變了」。**

## 那個誰都繞不開的硬問題：怎麼對一個「正在動的東西」拍快照

到目前為止我們把 full 和 incremental 講成兩條岔路，好像選一條就走到底。但真實系統幾乎從不純走一條。原因藏在一個你一旦開始做增量就躲不掉的問題裡，而它深到值得單獨花一節，因為它是「把同步講到能在腦中重放」這件事的試金石。

問題是這樣的：你決定上 log-based 增量同步——直接讀交易日誌，每一筆變更都抓得到。很好。但日誌只記錄「從你開始讀的那一刻起」發生的變更。表裡**已經有的**一億列歷史資料呢？日誌裡沒有它們的身影（或者早就被回收了）。所以任何增量管線啟動時都得先做一次**初始全量快照**：把現有的一億列整個搬一遍，然後**無縫接上**從某個位置開始的日誌串流。

魔鬼就在「無縫」這兩個字。你要對一張**正在被線上交易持續修改**的表做全量快照，這是在對一個移動的目標拍照。具體看哪裡會破：

假設你 `10:00:00` 開始做全量快照，慢慢掃，`10:30:00` 才掃完。掃到第 500 萬列（時間 `10:05`）時，這一列的值是 A。但 `10:10` 的時候，有筆線上交易把它改成了 B，而你的全量掃描早在 `10:05` 就掃過它、記下的是 A，不會回頭。同時，你的日誌串流如果是從 `10:30`（快照結束）才開始接，那 `10:10` 那筆「A→B」的變更就掉進了一個縫裡——快照記成 A，日誌沒接到那筆改動。**目的地永遠停在 A，而來源是 B。** 一筆無聲的不一致，誕生於兩個本該嚴絲合縫的階段之間的接縫。

天真的修法是「鎖表」：快照期間禁止任何寫入。正確，但對一個線上系統等於停機，沒人能接受。

業界真正的解法，漂亮到值得記住，它有兩個層次。

**第一個層次：用資料庫的一致性快照。** 關聯式資料庫早就有「對一個一致時間點拍照」的能力，就藏在隔離級別裡。在 PostgreSQL，你開一個 `REPEATABLE READ` 交易，這個交易看到的就是它**第一個查詢執行那一瞬間**的一致快照（精確說是交易的第一個非交易控制語句，而非 `BEGIN` 本身——若你 `BEGIN` 後等了一會才下第一個 `SELECT`，快照凍結在那個 `SELECT`）——之後別人怎麼改都跟它無關，它眼裡的世界凍結在出發那一刻。`pg_dump` 的平行匯出正是靠這個：它先用 `pg_export_snapshot()` 匯出一個快照 ID，再讓多條連線都 `SET TRANSACTION SNAPSHOT` 到同一個快照，於是這些連線即使各跑各的、各掃一塊，看到的也是**完全同一個瞬間**的資料——拼起來是 100% 一致的一份 dump，就像單一個 `REPEATABLE READ` 交易掃出來的。這把「移動目標」釘成了「靜止目標」：你不是對正在動的表拍照，而是對它在某個確切 LSN（log 位置）上的凍結版本拍照。只要你記住那個 LSN，日誌串流從那個 LSN 接著讀，接縫就消失了——快照是 LSN 之前的全部狀態，日誌是 LSN 之後的全部變更，無重疊、無遺漏。

**第二個層次：當你連「整表凍結一個快照」都付不起時——watermark 交織法。** 一個 `REPEATABLE READ` 長交易掃一億列可能跑很久，期間它會釘住資料庫的舊版本不讓回收（在 PostgreSQL 會撐大 dead tuple、拖累 vacuum）。Netflix 的 DBLog 框架（2020）提出了一個不鎖、不開長交易、且能隨時暫停續傳的辦法，後來 Debezium 的「增量快照」也採用了它。它的核心是**把快照切成小塊，並用寫進日誌本身的「水位標記」來消解快照塊和即時變更流之間的競態**。一塊的處理是這樣：

```
對每一個 chunk（一批主鍵）：
  1. 往日誌裡寫一筆「低水位 low watermark」標記（藉由 update 一張 signal 表）
  2. SELECT 出這個 chunk 的當前列（一次普通的讀，不鎖）
  3. 往日誌裡寫一筆「高水位 high watermark」標記
  4. 繼續正常消費日誌；當看到自己剛寫的低、高水位之間
     出現「某個主鍵的即時變更」時，就把這個主鍵
     從步驟 2 讀到的 chunk 結果裡「剔除」
  5. 低水位之前 + 剔除後的 chunk + 高水位之後的即時流，拼起來
```

為什麼這樣就對了？關鍵在第 4 步的剔除。如果某一列在你 `SELECT` 這個 chunk 的同時、又被線上改了一筆，那這筆即時變更會以日誌事件的形式出現在低、高水位之間。一旦你在日誌裡看到它，就知道「這一列我從 chunk 裡讀到的值已經過期了」——所以把它從 chunk 結果剔掉，讓那筆**更新的**日誌事件去代表它的最新狀態。沒看到的列，表示快照期間沒人動它，chunk 讀到的值就是對的。於是每一列都恰好被表達一次、且是最新的一次：要嘛來自 chunk（沒被並發改動），要嘛來自日誌（被改了，以日誌的新值為準）。**不鎖任何東西，卻得到了和「凍結整表」一樣的一致性**，而且 chunk 化讓它可以暫停、續傳、限速，對來源的衝擊被攤平。

你不一定每天都要親手實作這個——成熟工具替你做了。但**理解它**是分水嶺：它告訴你「初始快照 + 增量串流」這句話裡，那個「+」號藏著整套同步系統最容易出錯、也最值得敬畏的工程。任何宣稱「先全量、再增量」的方案，你都該追問一句：**你的接縫是怎麼縫的？** 縫不好，第一次啟動就會在某處留下一筆永遠對不上的資料。

## 第二軸：push 還是 pull——「來源要不要認識下游」的根本選擇

第二個決定看起來只是技術細節，實則是一道架構分水嶺：**變更是來源主動推出去，還是目的地週期性回來拉？**

**Pull（拉）**是開場那個輪詢 job 的形態：目的地說了算，它決定多久問一次來源「有什麼新的」。它最大的好處是**解耦**——來源完全不需要知道有誰在消費它、有幾個下游、它們在哪。來源只管把資料放在那兒，誰要誰自己來拿。下游也自己控制節奏：忙不過來就慢點拉，這天然就是一種背壓（背壓的機制見〈背壓：當下游跟不上〉）。代價是兩頭都不討好的輪詢延遲與空轉：拉得太稀疏，新鮮度差；拉得太密，大多數輪次是「沒有新東西」的空查詢，白白消耗來源的 IOPS。你想要一秒的新鮮度，就得每秒打一次來源，哪怕一整天只有零星幾筆變更。

**Push（推）**反過來：來源一有變更就主動送給下游。延遲低，沒有空轉。但它把一堆責任反推回了來源端，而這些責任在 pull 模型裡是不存在的：

- 來源得**知道有哪些下游**（或至少把變更交給一個能扇出的中介，比如一個 broker）。
- 下游掛了怎麼辦？推過去沒人收，這筆變更**重試嗎、緩存嗎、丟棄嗎**？來源端得有一套投遞失敗的處理。
- 最關鍵的：**背壓往哪走？** push 是來源主導節奏，如果來源狂推、下游消化不了，訊息要嘛在中間無限堆積把記憶體吃光，要嘛反過來把來源拖慢。一個只會「有變更就推」、卻不管下游死活的 push 管線，是經典的過載放大器。push 模型必須配一個有界的緩衝（通常就是 broker 那個佇列）和一套背壓機制，否則它在流量尖峰時會放大故障而不是傳遞資料。

這裡有個常被忽略的洞察：**現實中最好用的同步管線，往往兩者都不純。** 一個 log-based CDC 管線，來源端（資料庫）其實是被動的——它只是寫自己的日誌，根本不知道有 CDC 在讀。讀日誌的 connector 是去**拉**日誌的（pull 自資料庫的角度）。但 connector 把解析出來的變更**推**進 Kafka，下游 consumer 又是**拉** Kafka。同一條管線，三段，push 和 pull 交替出現。所以「push 還是 pull」不是要你給整條管線貼一個標籤，而是要你**在每一段交接處都問一次**：這一跳是誰主導節奏、誰負責背壓、來源認不認識下游。看清每一跳，你才知道某次故障時資料會堆在哪、會丟在哪。

## 第三軸：batch 還是 streaming——「窗口陳舊度」換「長運維的重量」

第三個決定是關於**粒度與時機**：攢一大批一起搬，還是一筆一筆即時搬？

**Batch（批次）**是把變更攢起來、每小時或每天跑一個 job 一次搬完。它的優點是工程意義上的「踏實」：吞吐高（一次處理一大批，攤掉固定開銷）、單位成本低、出錯了**整批重跑**就好——批次 job 天然有「冪等地重跑一整段」的能力，因為一個批次有明確的邊界。它的代價是新鮮度被批次窗口釘死：每小時一批，資料就最多舊一小時。對「每日報表」這種需求，這完全夠；對「改完五秒內要生效」這種需求，這是死刑。

**Streaming（串流）**是一筆變更發生就近即時搬一筆。新鮮度可以做到秒級甚至亞秒級。但它要付的是一種**性質完全不同**的成本——不是「每次貴一點」，而是「你現在養了一個必須 24×7 活著的東西」：

- 它是個**長運行管線**，不是一個跑完就退的 job。它得有人管它的存活、它的延遲、它落後多少。
- 它得處理**亂序**：串流裡的事件不保證嚴格按發生順序到達（順序保證的機制見〈訊息順序：FIFO 與 partition key〉），下游得能容忍或重排。
- 它得處理**背壓**：上游持續產出、下游持續消費，一旦下游變慢，整條管線的緩衝就開始漲，你得有東西在漲到爆之前介入。
- 它最現實的陷阱，是某些 streaming 同步會在來源端留下一個**必須一直被消費的位置**。以 PostgreSQL 的 logical replication 為例，它靠一個 replication slot 記住「下游消費到哪」，並為此**保留那個位置之後的所有 WAL 不回收**。如果你的下游 connector 掛了一整天沒人發現，PostgreSQL 為了不丟資料會死死守著那一整天的 WAL——磁碟就在無人察覺中被一點點填滿，最後可能在某個半夜把整個主庫寫爆。**這是 batch 完全不會有的故障模式**：一個批次 job 掛了，來源毫髮無傷；一個 streaming 消費端掛了，可能反過來殺死來源。所以「消費端的存活與落後量」在 streaming 同步裡不是可選的監控項，是一級指標。

把這一軸講白：**batch 用「陳舊度」換「運維的輕」，streaming 用「運維的重」換「新鮮度」。** 而那個運維的重，最容易被低估——上 streaming 的決定往往在三個月後變成「沒人會修的那條管線」。

## 三軸組合：名詞只是座標上的點

現在三軸都拆透了，回頭看那些唬人的名詞，它們會塌縮成座標上的幾個點：

| 你聽過的名字 | full/incr | push/pull | batch/stream | 一句話 |
|---|---|---|---|---|
| 每晚 dump 整表的 ETL | full | pull | batch | 最樸素，自我修復，最舊 |
| `updated_at` 輪詢 | incremental | pull | 可批可串 | 便宜，但漏刪除、漏邊界 |
| log-based CDC | incremental | 混合 | streaming | 抓得到刪除與順序，但要養管線 |
| outbox + 事件投遞 | incremental | push | streaming | 變更與事件原子一致，至少一次 |
| 主庫→唯讀副本複製 | incremental | push | streaming | 資料庫內建，靠複製延遲 |

這張表不是要你背。它要證明的是：**沒有「資料同步」這個單一技術，只有三個正交決定的具體取值。** 當有人說「我們用 CDC」，你腦中該自動展開成「所以是 incremental、streaming、且來源被動」，於是你立刻知道該擔心什麼：接縫縫好了嗎？slot 有人盯著嗎？下游冪等嗎？名詞是壓縮包，三軸是解壓縮的鑰匙。

## 為什麼這三軸正交，又為什麼選擇收斂

值得停下來想一個問題：為什麼偏偏是這三軸，而且它們真的彼此獨立嗎？

獨立，是因為它們各自回答一個不同維度的問題，而且回答一個不約束另一個。「界定變更的方式」（full/incremental）是關於**搬什麼**；「誰發起」（push/pull）是關於**搬的控制權在哪**；「攢多少」（batch/streaming）是關於**搬的粒度**。你完全可以 incremental + pull + batch（輪詢攢一批），也可以 incremental + push + streaming（CDC 即時推），更可以 full + push + streaming（雖然少見，比如把整個狀態快照持續推給一個鏡像）。八種組合在邏輯上都成立，這就是正交的意思。

但實務中為什麼大家總往那幾種收斂？因為**約束不在這三軸之間，而在每一軸的取值各自會召喚出的責任，常常彼此匹配或衝突**。incremental 要求你正確界定「什麼變了」——而最可靠的「什麼變了」的來源是交易日誌，日誌天生是**有序的事件流**，於是它自然親近 streaming（一筆一筆來）而非 batch（攢一批本身就違背了日誌的逐筆性質）。streaming 要求 24×7 的活性，而 push 的「來源主動」恰好供得上這種即時性，pull 的輪詢則會在追求低延遲時退化成高頻空轉。於是 incremental + streaming + push 這一組會彼此加強，凝結成我們叫做「CDC 管線」的東西。反過來，full + batch + pull 也彼此加強——全量本就無狀態，批次本就有邊界，拉取本就解耦，三者湊在一起就是那個踏實的「每晚重建」job。

這就是為什麼這一章不給你一張「最佳實踐」清單。**不存在普世最優的那一組。** 存在的是一連串可以推導的判準：

- 先問「**能不能容忍 N 小時的陳舊**」。能，且資料量小→ full + batch，要的就是它自我修復、好除錯、半夜出事也不會殺死來源。
- 不能容忍陳舊、或資料量大到全量搬不動→ 往 incremental + streaming 走，並且**睜大眼睛接受**你即將欠下的債：刪除要處理、接縫要縫、slot 要盯、下游要冪等。
- 再問「**來源端能不能可靠地告訴我什麼變了**」。有可信的 `updated_at` 且沒有硬刪除→ 輪詢就夠；有硬刪除、要完整順序、要餵多個下游→ 只能下到交易日誌那一層（見〈CDC：從資料庫變更長出事件流〉）。

每一步都是在拿一個你不想要的東西，換一個你更不想要的東西。同步沒有免費的選項，只有想清楚了的取捨。

## 最後一塊拼圖：沒有對帳的同步，等於沒有保證的同步

把三軸選定、管線搭好，還剩一件事，而它重要到能讓前面所有的工程功虧一簣，卻最常被當成「以後再說」。

是這樣一個殘酷的事實：**任何長期運行的增量／串流同步，最終都會漂移。** 不是「可能會」，是「一定會」。漏處理一則訊息、消費端靜默重啟、一個邊界競態、一次 schema 變更的接縫沒接好、有人手動改了一筆庫——每一個都會在兩端之間留下一點點差，而且**這些差幾乎都不會報錯**。你的管線一路綠燈，dashboard 上沒有任何紅色，但你的搜尋索引和主庫，正在以你看不見的速度悄悄分家。

這就是為什麼成熟的同步系統永遠不只是「搬資料的管線」，它一定還配一個**持續驗證兩邊是否一致、並在發現漂移時修回去**的機制——對帳。它可能是每晚比一次兩邊的總列數和某個金額欄位的總和當哨兵，可能是用 Merkle tree 之類的結構讓比對成本只正比於差異量而非總資料量，發現差了再以來源為準覆蓋或重放缺漏的那幾筆（這套比對與修復的完整機制見〈對帳：怎麼確認兩邊一致、漂移怎麼修〉）。機制的細節不在這一章，但這一章必須把這個觀念釘進你對「同步」的定義裡：

**「管線沒報錯」和「兩邊真的一致」是兩件完全不同的事。** 一致是要被**主動驗證**的，不是「沒出錯就自動擁有」的。一個跑了半年、從沒報過錯、其實早就漏了上萬筆的同步管線，比一個你知道它不可靠的管線更危險——因為它給了你虛假的安心。所以當你設計任何 incremental 同步時，「對帳」不是一個可選的加值功能，它和管線本身是同一件事的兩半。把同步搭起來只完成了一半；另一半是建立一個能在三週後告訴你「你已經漂移了」的東西。

## 為何是這個形狀

退一步看整章，所有的複雜度都從一個無法迴避的源頭長出來：**你想讓散在多處的同一份資料保持一致，但「把變更搬過去」這個動作，本身就會在時間、順序、狀態這三個維度上出岔子。**

正因為「搬什麼」可以是全部或差異，你才有了 full 與 incremental 的對立——一邊用成本買「永不漂移」，一邊用「會漂移的風險」買便宜與新鮮。正因為「誰發起」決定了控制權與背壓往哪走，你才有了 push 與 pull，以及那個更深的提醒：在管線的每一跳分別問，而不是給整條貼一個標籤。正因為「攢多少」決定了你養的是一個跑完就走的 job、還是一個必須 24×7 活著、還可能反殺來源的長物，你才有了 batch 與 streaming 的重量差。而正因為任何有狀態的增量管線都會漂移，對帳才不是附件，是同步的另一半身體。

下次你面對「這份資料怎麼讓那邊也有一份」這個問題，你不會再去找「該用哪個工具」。你會先在心裡轉三個旋鈕——搬全部還是搬差異、誰來發起、攢多少搬一次——讓答案從你能容忍多少陳舊、來源能不能告訴你什麼變了、團隊養不養得起一條長管線裡浮現出來。工具只是這三個旋鈕被轉到某個位置後，剛好有人替你打包好的那個盒子。看穿盒子裡的三個旋鈕，你就再也不會被名詞唬住了。
