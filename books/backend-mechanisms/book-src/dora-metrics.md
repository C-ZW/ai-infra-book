# DORA 四指標

一個工程主管在季度檢討會上被問了一句很合理的問題：「我們團隊的交付能力，到底是好還是不好？」

這是個比聽起來難回答得多的問題。你可以說「我們發布很頻繁」——但頻繁地發爛 code 算好嗎？你可以說「我們幾乎不出包」——但代價是每次發布前都要凍結兩週、層層簽核，這也算好嗎？速度與穩定看起來天生對立：想跑得快就得冒險，想穩就得放慢。憑直覺，每個人對「我們團隊好不好」都有一個答案，而且每個人的答案都不一樣，誰也說服不了誰。會議室裡真正缺的，不是更多意見，而是**一把所有人都認的尺**。

DORA 四指標想當的，就是這把尺。它的野心不大也不小：用四個能客觀量測、跨團隊可比、有大規模研究背書的數字，把「交付能力好不好」這個含糊的判斷，變成一個可以指著數據談的事實。但這把尺有一個非常容易被誤用的形狀——用對了，它幫團隊定位真正的瓶頸；用錯了，它會在一季之內把整個團隊的行為扭曲成一場數字表演。要把它用對，你得先看懂這四個數字為什麼是這四個、它們各自在量什麼、以及它們合在一起時想對抗的到底是什麼。

## 這四個數字從哪裡來

先講清楚 DORA 不是某個人拍腦袋設計的指標框架。DORA（DevOps Research and Assessment）是一個始於 2014 年的研究計畫，靠的是每年一份大規模問卷——後來以《Accelerate State of DevOps Report》為名發布。它的方法論是這樣的：蒐集數以萬計的團隊回報的各種工程做法與結果（早年的《Accelerate》一書記載分析了約 23,000 份資料點），用統計方法去找「哪些做法和『交付得又快又好』有相關」。

這個出身決定了四指標的性格。它們不是「理論上應該重要的東西」，而是「在資料裡，能把高效能團隊和低效能團隊分開的那幾個量」。2018 年 Forsgren、Humble、Kim 三人把研究結論寫成《Accelerate》，丟出一個當時相當反直覺的主張：**速度與穩定不是取捨關係**。資料顯示，跑得最快的那群團隊，同時也是最穩的——它們部署最頻繁，卻同時故障率最低、出事後恢復最快。換句話說，會議室裡那個「想快就得犧牲穩」的直覺，被資料打臉了：真正的高手是兩者都好，而把速度和穩定當成蹺蹺板兩端的團隊，往往兩端都不怎麼樣。

這個發現是理解四指標的鑰匙。它們之所以是「四個」而不是「一個」，正是為了把這個「又快又穩」的雙重要求釘死成可量測的形狀——兩個量速度，兩個量穩定，逼你不能只顧一頭。

## 兩個量速度，兩個量穩定

四個指標，傳統上分成兩組看。

量速度（吞吐）的兩個：

- **Deployment Frequency（部署頻率）**——一段時間內你把變更推上生產的次數，或者反過來看，兩次部署之間隔多久。它在問：你多敢、多能頻繁地交付？
- **Lead Time for Changes（變更前置時間）**——一筆變更從 commit 進版本控制，到它真的部署上線，中間經過多久。它在問：一個寫完的改動，要在你的流程裡卡多久才能見到使用者？

量穩定的兩個：

- **Change Failure Rate（變更失敗率）**——你的部署裡，有多少比例上線後需要立刻補救（hotfix、rollback、緊急修補）。它在問：你發出去的東西，有多常出包？
- **Failed Deployment Recovery Time（失敗部署復原時間）**——一次失敗的部署，要多久才能恢復服務。它在問：出事之後，你多快能站起來？

把它們擺在一起看，這個分組的用意就浮出來了。前兩個衡量「你交付得多快」，後兩個衡量「你交付的東西多可靠、壞了多快好」。任何一組單獨衝高都很容易，難的是四個一起好——而四個一起好，正是那把尺真正想量的東西。

不過這裡藏了一個很容易說錯、又 load-bearing 的細節，值得拆清楚：**「兩兩對抗」只是一個防呆的記法，不代表速度和穩定之間有內建的取捨。** 把它想成「左邊兩個是速度、右邊兩個是穩定，互相牽制」只是為了提醒你別只盯一頭；DORA 研究的招牌結論恰恰相反——這四個在高效能團隊裡是**同向移動**的，不是此消彼長的。如果你心裡把它當成蹺蹺板，你就已經誤解了這個框架最核心的發現。

## 為什麼非得四個一起看

理解四指標，最快的路是看「只盯其中一個會發生什麼」。每一個指標單獨拿出來當目標，都會立刻長出一個荒謬的最佳解。

只盯 **Deployment Frequency**：團隊學會的不是「更頻繁地交付有價值的變更」，而是「把一次部署拆成五次假部署」，或者乾脆在 pipeline 裡加一個每小時空跑的 deploy。數字漂亮了，交付能力一點沒變。

只盯 **Lead Time**：團隊會把「commit」這個起點往後挪——等所有東西都準備好了才 commit，於是從 commit 到上線的那段時間自然很短。前面那段沒被計入的等待，照樣存在，只是藏到了量測窗口外。

只盯 **Change Failure Rate**：最簡單的壓低方法是**不發布**。不上線就不會失敗。退而求其次，是把「失敗」的定義悄悄收窄——這個只是小問題不算失敗、那個是 hotfix 但我們當作常規發布——於是失敗率好看了，系統照樣在使用者面前出包。

只盯 **Failed Deployment Recovery Time**：你會誘導團隊草草「緩解」而不去「理解」——回滾了、服務恢復了、計時器停了，數字漂亮，但根因沒查、下次照壞。

看出共通的形狀了嗎？每一個指標，單獨成為目標，都會被一個「優化數字而非優化真實能力」的捷徑攻陷。這正是這個框架的設計要點：**四個指標互相當對方的監視器**。狂衝部署頻率，變更失敗率會跳出來抓你；拼命壓低失敗率（靠不發布），部署頻率和前置時間會立刻露餡；草草緩解換取復原時間，會在變更失敗率與返工率上留下痕跡。任何一個被遊戲化的捷徑，都會在另一個指標上現形。**四個一起看，才逼得出「又快又穩」**——這是它們作為一組存在的全部理由，也是為什麼把任何一個單獨抽出來當 KPI，都等於拆掉這個框架的安全機制。

## 一個團隊怎麼用這把尺定位瓶頸

抽象的原理講完，看一個具體的情境，這把尺才會活起來。

一個團隊有個模糊的感覺：「我們發布好像很慢，但說不上來慢在哪。」這種感覺最危險——它會誘導出錯誤的補救，比如「大家手腳快一點」「加班趕一下」，而瓶頸根本不在那裡。

導入 DORA 量測，先別急著下結論，先把四個數字攤開。假設量出來是這樣：

| 指標 | 量到的值 |
|---|---|
| Deployment Frequency | 每天 3–5 次 |
| Lead Time for Changes | 中位數 6 天 |
| Change Failure Rate | 8% |
| Failed Deployment Recovery Time | 中位數 40 分鐘 |

第一個反直覺的發現：**部署頻率一點都不低**——每天好幾次，這已經是相當好的節奏。所以「我們很慢」的直覺，問題不在部署這個動作本身。但 Lead Time 中位數 6 天，相對於「每天部署好幾次」顯得很不協調：既然能一天部署這麼多次，為什麼一筆變更從 commit 到上線要拖 6 天？

這個不協調就是線索。把這 6 天的 Lead Time 拆開——一筆變更的時間軸通常長這樣：

```
commit ──► CI 跑完 ──► 等人 review ──► merge ──► 等部署窗口 ──► 上線
  |          |              |            |            |
  0      ~15 min        ~4 days      ~10 min      ~2 days
```

手算一下：CI 本身 15 分鐘，merge 動作 10 分鐘，這些都不是瓶頸。6 天裡，約 4 天耗在「等人 review」，約 2 天耗在「等手動部署排程」。真正吃掉時間的，是兩段**等待**，不是任何人寫 code 寫得慢。

到這裡，DORA 的價值就兌現了：它沒告訴團隊「去逼大家寫快一點」（那會傷士氣又沒用），而是把手指精準地按在兩個瓶頸上——review 的回應太慢、部署沒有自動化。對策因此變得具體：縮小 PR 尺寸讓 review 更快（小 PR 本來就更容易被認真看完），以及把手動部署排程換成自動化部署。

三個月後再量一次：Lead Time 中位數從 6 天掉到 1 天出頭，而且——這是關鍵——**Change Failure Rate 沒有惡化**，還是 8% 上下。如果壓 Lead Time 的代價是失敗率飆升，那只是把瓶頸從一處推到另一處，不算改善。四個指標一起看，才能確認這是真的變好，而不是拆東牆補西牆。這正是這把尺被設計來做的事：**用數字定位瓶頸，而不是拿數字鞭打人。**

順帶說一個量測上的選擇。要得到上面這些數字，有兩條路：一條是接上 CI/CD 和版控的工程資料自動算——客觀、可持續，但要花力氣接管線；另一條是用 DORA 官方的快速問卷自評——主觀、但零接入成本。對剛起步、只想知道「我們大概落在哪個檔位」的團隊，問卷自評就夠了；要長期追蹤趨勢、要數字經得起爭論，才值得投資自動化的工程資料管線。別在還沒確認瓶頸之前，就先花一季去建一套精密的儀表板——那本身可能就是一種把工具當目標的過度投入。

## 那條容易絆倒人的歷史：第四個指標改過名

四指標聽起來是個一勞永逸的框架，但它其實一直在演化，而其中一次演化埋了一個很容易讓人說錯話的坑。

第四個指標——失敗部署復原時間——**原本不叫這個名字**。它早年叫 MTTR（Mean Time to Recovery）或 Time to Restore Service，定義很寬：服務從任何一種劣化中恢復要多久。問題出在這個「任何一種」。資料中心斷電、上游服務商掛掉、海底電纜被挖斷——這些劣化跟你的**交付能力**毫無關係，卻全被算進這個指標。一個交付做得很好的團隊，可能因為一次跟自己無關的雲端區域故障，把 MTTR 拉得很難看。這讓它跟其他三個「衡量交付管線」的指標格格不入：那三個都在量「你把變更送上線的這條路」，唯獨它在量「世界對你做了什麼」。

於是 **2023 年，DORA 正式把它更名為 Failed Deployment Recovery Time，並收窄定義**：只算「因為一次變更（部署）而導致的劣化」的復原時間，把那些跟你的部署無關的外部故障排除在外。這個改動讓第四個指標終於跟前三個對齊——四個都在量同一條交付管線上的事。

這個坑在哪裡？如果你或你的工具仍在用舊的廣義 MTTR——把資料中心斷電、第三方故障都算進去——你算出來的數字就**和 DORA 的最新定義不一致**，跟別人的數字一比就會錯位，而且你會以為自己在量「交付復原能力」，實際上量的是混了一堆外部噪音的東西。要對齊現在的框架（2026-06），記住：**這個指標只算變更造成的劣化的復原時間。**

## 而且，它早就不只四個了

更新的一次演化更徹底。**2024 年的報告，DORA 加入了第五個指標，並且重新切了分組。**

第五個指標叫 **Deployment Rework Rate（部署返工率）**——你的部署裡，有多少是「非計畫的、為了修生產上的問題而做的」。它跟變更失敗率很像，但角度不同：變更失敗率問「這次部署本身有沒有立刻出包」，返工率問「我們有多少部署其實是在替先前的問題擦屁股」。一個團隊可以變更失敗率不高（每次部署當下都沒爆），卻返工率很高（不斷地在補先前漏掉的洞）——返工率抓的是這種「持續性的不穩定」。

同時，2024 年的分組也變了，而且變得比「兩兩對抗」更精確。新的切法是：

- **Software Delivery Throughput（吞吐，3 個）**：Deployment Frequency、Lead Time for Changes，以及——**移過來的**——Failed Deployment Recovery Time。
- **Software Delivery Instability（不穩定，2 個）**：Change Failure Rate、Deployment Rework Rate。

兩件事值得注意。其一，復原時間**從穩定組移到了吞吐組**——DORA 的推理是：出事後快速恢復，本質上是在維持交付的流動，更像吞吐能力而非穩定性。其二，那一組的名字從「stability（穩定）」改成了「**instability（不穩定）**」——因為高的變更失敗率或高的返工率，量到的其實是「不穩定的程度」，數字越大代表越不穩，這樣命名在語意上更誠實。

那「DORA 四指標」這個說法還算數嗎？算。**作為通稱，「DORA 四指標」仍然指最早那四個**（部署頻率、變更前置時間、變更失敗率、失敗部署復原時間），這是最通用的入門記法，也是大多數團隊實際在追的。但如果你要跟人精確討論最新框架，你得知道：它在 2024 年已經是**五個指標、兩個新分組（吞吐／不穩定）**，而且第四個指標所在的組別已經換了邊（2026-06）。把這段歷史搞清楚，你才不會在跟人對數字時，發現大家口中的「第四個指標」根本不是同一個東西。

## 為什麼 elite 不是終點

DORA 還會把團隊分成 elite / high / medium / low 四檔，給出每檔的大致基準——最頂的 elite 大致是「隨需部署、前置時間低於一天、變更失敗率約 5%、復原時間不到一小時」這個量級。這些數字很誘人，誘人到容易讓人把「衝上 elite」當成目標本身。這是個需要踩剎車的衝動，理由有兩層。

第一層，這些檔位是**描述性的分布，不是道德排名**。它描述的是「在受訪的這群團隊裡，數字大致呈現幾個聚落」，而且這些聚落每年都在動——2024 年的報告裡就出現了反常現象：medium 檔的變更失敗率竟然比 high 檔還低，而 high 檔的占比從前一年的 31% 縮到 22%。四個指標一向被認為會同向移動，這一年卻出現了不齊步的情況。這恰恰說明：檔位是對真實世界的一個快照，不是一條刻好的階梯，更不是「越高越有道德」的排序。

第二層，也更實際：**並非每個系統都該追 elite 級的部署頻率。** 一個低變更需求的內部工具，本來就沒幾個變更要發，硬要把部署頻率衝到「隨需、一天好幾次」毫無意義——你會為了餵飽指標而製造出根本不需要的部署。指標服務於「這個系統需要什麼」，不是反過來讓系統去服務指標。對某些系統，medium 檔的數字就是它健康的樣子。

## 它最危險的失敗模式：尺變成了靶

把上面所有的陷阱收束起來，會匯到同一個根源，也是這個框架唯一真正致命的失敗模式：**把四指標拿來考核個人或團隊的績效。**

社會科學裡有一條 Goodhart 定律——人類學家 Marilyn Strathern 在 1997 年的論文裡讓最流傳的一句廣為人知：「**當一個量測變成目標，它就不再是個好的量測。**」這條定律對 DORA 指標的殺傷力是即時且徹底的。前面講的每一種遊戲化——拆假部署衝頻率、挪後 commit 起點壓前置時間、收窄「失敗」定義壓失敗率、草草緩解換取復原時間——平時不會發生，因為沒人有動機去做這些蠢事。但一旦這些數字跟某人的考績、獎金、升遷掛鉤，動機瞬間出現，而且勢不可擋。指標一旦成為目標，它衡量的就不再是「交付能力」，而是「員工規避懲罰、爭取獎勵的創造力」——而後者的創造力是無限的。

所以這把尺的保證有一個沉默但 load-bearing 的成立條件：**它只在「誠實量測、整體觀察、用於團隊自我診斷」的前提下有效。** 滿足這個前提，四指標是一面誠實的鏡子，照出流程的瓶頸與系統的脆弱。一旦違反——把它變成個人 KPI、變成團隊間的排名競賽——鏡子立刻變成哈哈鏡，每個人都在對著它整理表情，沒人在乎鏡子背後的真實。這不是危言聳聽的口號，而是 Goodhart 定律在這個框架上的具體兌現方式。

還有一個相關的誤解值得點破：**四指標不是 DORA 的全部。** DORA 的研究包含一大堆「能力」——文化、自動化、精實管理、架構解耦等等——四指標只是這些能力作用之後、顯現在**結果面**的量測。光盯指標不看背後的做法，等於只量體溫不看病因：體溫高告訴你「有問題」，但退燒不等於治病。指標幫你定位「哪裡不對」，但「為什麼不對、怎麼修」要回到底層的工程與協作做法去找。

## 為什麼是這個形狀

退一步看，四指標的整個樣貌，都是從一個張力裡長出來的：**人們本能地相信速度和穩定是取捨，而資料說它們不是。**

正因為要對抗「只顧一頭」的本能，它才是四個而不是一個——兩個盯速度、兩個盯穩定，互相當監視器，逼出「又快又穩」。正因為任何單一指標都會被捷徑攻陷，它才必須**一起看**——讓每個遊戲化的小聰明，都在另一個指標上現形。正因為它量的是「結果」而非「能力」，它才只能當診斷的尺、不能當改善的藥——告訴你哪裡有瓶頸，不告訴你怎麼修。也正因為它一旦變成考核的靶就會失真，它才必須牢牢守在「團隊自我診斷」的位置，絕不下放成個人 KPI。

它不是一套憑空設計的儀表板，而是「如何客觀回答『這個團隊交付得好不好』」這個問題，被大規模資料逼到牆角後，唯一站得住的形狀。下次有人在會議室問「我們到底好不好」，你會知道答案不在誰的嗓門大，而在四個一起看、誠實量出來的數字裡——以及在你有沒有忍住，沒把這把尺變成一根打人的棍子。
