# 12-factor：哪些在容器/serverless 時代仍成立

有一份文件，你大概沒讀過全文，卻在無數面試、設計討論、PR 評論裡被當成共識引用：「這不夠 12-factor」「config 要外置，這是 12-factor 的第三條」。它叫 *The Twelve-Factor App*，2011 年由 Heroku 共同創辦人 Adam Wiggins 整理發表，把「一個應用要怎麼寫，才能在 PaaS 上跑得好、好擴展、好維運」歸納成十二條。

弔詭的是，它寫成的那一年——2011——Docker 還沒誕生（2013 才開源），Kubernetes 還要再三年（2014）才出現，serverless 更是遙遠的概念。它描述的世界是 Heroku 那種「你 `git push`、平台幫你跑」的 PaaS。可是當容器與編排器席捲了整個產業，人們發現一件怪事：這份前容器時代的文件，它定義的好幾個模式——config 從程式碼裡抽出來、process 不存本地狀態、log 當成一條 stream 吐出去——竟然不偏不倚地變成了雲原生世界的**預設假設**。Kubernetes 的 ConfigMap、Pod 的 ephemeral 本質、容器 runtime 收 stdout 的方式，幾乎像是照著 12-factor 的原則長出來的。

但這不是說十二條全都通過了時間的考驗。有幾條被現代平台直接內建成預設、再也不必特別提；有幾條原則對、但 2011 年給的具體做法已經顯老、得換個實作；還有幾條，整個議題在 2011 年根本還不存在，是 12-factor 留下的盲區。這一章不打算逐條複述十二條的內容——那種清單網路上到處都是，背它沒有意義。我們要做的是更有用的事：搞清楚**每一條背後到底是一個恆久的工程約束，還是一個綁死在 2011 年 PaaS 上的時代產物**，以及為什麼這個區分在 2026 年比文件本身更值得理解。

## 為什麼一份十五年前的文件還沒過期

先回答一個該被先問的問題：為什麼這份東西沒有像同期的大多數技術文件一樣，被掃進歷史的角落？

關鍵在於它說的不是「用什麼工具」，而是「應用對外界做什麼假設」。一條「用 Heroku 的 `Procfile` 宣告 process」這種建議，會隨工具死去；但「**應用不該假設它跑在哪台特定機器上、不該假設那台機器重啟後本地檔案還在**」這種約束，是從一個更深的事實裡長出來的——只要你想讓一個應用能被任意複製、任意搬移、任意殺掉重建，它就**不能**把任何要緊的狀態藏在自己的行程記憶體或本地磁碟裡。這個約束不關 Heroku 的事，它關的是「水平擴展」這件事的物理本質。

於是 12-factor 真正在做的，是把「一個應用要能被一個調度系統自由地擺佈」這個需求，翻譯成十二條對應用程式作者的具體要求。Heroku 當年的調度系統是它的 dyno manager；今天的調度系統是 Kubernetes 的 controller、是 Lambda 的 invocation model。調度系統換了，但「應用要可被自由擺佈」這個前提沒換——反而更強了。容器把「process 隨時可被殺、可被複製到任何一台 node」變成了日常操作，serverless 更把它推到極致：你的程式碼可能每一次請求都跑在一個全新的、上一秒還不存在的執行環境裡。在這種世界，一個違反 12-factor 核心假設的應用不是「不夠優雅」，而是**根本跑不起來**——它無法水平擴展、每次部署都掉資料、log 收不到。

這就是為什麼它沒過期。它描述的不是某個平台的操作手冊，而是「可被調度的應用」這個物種的解剖結構。平台會換，物種的解剖結構不太會。

## 三條已經贏到變成空氣的原則

有幾條原則贏得如此徹底，以至於今天的工程師根本意識不到自己在遵守它們——它們被平台內建、被工具預設，成了像空氣一樣的背景。

**第一條是「process 無狀態、share-nothing」（factor VI）。** 12-factor 要求應用的每個 process 都是無狀態的：它不在自己的記憶體或本地磁碟裡保存任何**跨請求、跨重啟需要存活**的資料，所有這類狀態都外放到 backing service（資料庫、快取、物件儲存）。2011 年這是個需要說服人的主張——當時很多應用理所當然地把 session 存在本地記憶體、把上傳的檔案寫進本地目錄。

到了容器與 serverless 時代，這條不再是「建議」，它變成了**入場券**。想一下水平擴展實際發生的事：負載上升，調度器多開三個一模一樣的 replica，把流量分散過去。如果使用者的 session 存在第一個 replica 的記憶體裡，那他的下一個請求被分到第三個 replica 時就「登出」了——除非你硬上 sticky session 把他釘回原機，但那等於放棄了無狀態的全部好處（這類「連線狀態怎麼跨節點共享」的拉扯，在即時連線那類長連線服務裡會被逼到檯面，見〈WebSocket〉）。serverless 把這條推到無可迴避：Lambda 的每個 execution environment 隨時可能被回收，下一次調用可能落在一個全新的環境，你**沒有**本地狀態可以依賴。所以無狀態不是一種美德，它是「能不能 scale-to-zero、能不能讓平台隨意回收與複製你的執行環境」的前提（無狀態為什麼是 serverless 的硬性前提，見〈無狀態與有狀態〉）。

這裡有個微妙的誤讀值得拆掉：無狀態**不等於**系統不能有狀態。它說的是**應用 process 不在本地保存跨請求的狀態**，狀態被誠實地推到外部的 backing service 去。一個處理金流的系統當然有大量狀態——但那些狀態活在資料庫裡，不活在某個 web process 的記憶體變數裡。把「無狀態」理解成「系統不准有任何狀態」是搞錯了層次；真正的要求是「**狀態要放在一個能被所有 replica 共享、且不隨任一 process 之死而消失的地方**」。

**第二條是「config 存在環境裡」（factor III）。** 12-factor 的洞見是把「會隨部署環境而變的東西」（資料庫連線字串、外部 API 的 URL、feature flag）從程式碼裡徹底抽出來，因為同一份 codebase 要能部署成 dev、staging、prod 多份，差異全在 config。這條原則——**config 與程式碼分離**——恆久正確，而且現代平台把它變成了預設：K8s 用 ConfigMap，serverless 用環境變數注入，沒有人會把生產資料庫密碼硬編進原始碼了（至少不會承認）。

但這條藏著一個 2026 年必須拆開的細節，我們待會單獨講——因為「config 與程式碼分離」這個**原則**通過了考驗，而 2011 年給的具體做法「**一律塞進環境變數**」沒有完全通過。

**第三條是「logs 當成 event stream」（factor XI）。** 12-factor 要求應用**不要自己管理 log 檔**——不要決定 log 寫到哪個檔案、不要自己做 rotation、不要自己壓縮歸檔。應用該做的只有一件事：把 log 當成一條按時間排序的事件流，無緩衝地寫到 `stdout`。至於這條流被導去哪、怎麼聚合、存多久，是**執行環境**的事，不是應用的事。

2011 年這是個有點反直覺的主張，因為當時「應用自己寫 log 檔」是天經地義。但在容器世界，這條變成了**鐵律**，原因很物理：容器是 ephemeral 的，它的檔案系統隨容器銷毀而消失。如果你的應用把 log 寫進容器內的某個檔案，那麼這個容器一旦被殺——而在 K8s 裡容器被殺是家常便飯——log 就跟著蒸發了，你連事後查都查不了。正確的形狀是應用只管往 stdout 吐，容器 runtime 接住這條流，交給 log agent（如 Fluent Bit 這類）聚合、路由到後端。

這條因此產生了一個現代的反模式：**在容器裡還自己 rotate log 檔**。一旦應用自作主張地寫檔、切割、壓縮，它就破壞了「平台負責收集」的前提，反而在容器裡製造出一個會隨容器一起死掉的、誰也收不到的 log 黑洞。在容器裡寫 stdout 就好，剩下的交給平台——這不是偷懶，這是把職責放對了地方。

## 一個最常被一句帶過的因子：disposability 到底要求什麼

十二條裡有一條，名字聽起來很抽象、最常被一筆帶過，但它其實是整套東西能否在容器時代成立的關鍵——**disposability（factor IX）：process 要能被即時地、優雅地拋棄。** 它包含兩個要求：**快速啟動**和**優雅關閉**，而這兩件事在 K8s 的日常運作裡每天被觸發無數次，值得把機制看清楚。

先看為什麼這條在 2011 年只是「最佳實踐」、到了容器時代卻變成生死攸關。K8s 的滾動更新（見〈blue-green、canary 與 rolling〉）、HPA 的擴縮、node 的維護排空、雲端 Spot 機器被回收——這些操作的共同點是：它們會**頻繁地、預期之內地**殺掉你的 process 再起新的。把啟動時間的代價算到底就知道為什麼這不是優雅問題而是可用性問題：一個 50 replica 的 Deployment 做滾動更新，若一次只換一個 Pod（`maxUnavailable=1`、`maxSurge=1`），新 Pod 啟動要 90 秒，那麼把整批換完約需 `50 × 90s = 4500s`，**75 分鐘**——這段時間裡新舊版本混跑、回滾窗口也被拉得一樣長；若把啟動壓到 5 秒，同樣 50 個 Pod 只要 `50 × 5s = 250s`，約 **4 分鐘**。同一個操作，啟動時間差一個量級，部署時間就差到一個量級，HPA 在尖峰來臨時能不能及時擴出容量也由它決定（冷啟動為什麼會把第一個請求拖那麼慢，是另一個完整的題目，見〈冷啟動與暖機〉）。而一個**不會優雅關閉**的 process，會在每一次這類操作中默默地傷害使用者。

優雅關閉的機制具體是這樣運作的，把它在腦中重放一遍：當 K8s 決定要終止一個 Pod，它**不是**直接砍掉，而是先送一個 `SIGTERM` 訊號給容器裡的 PID 1，然後開始等一段寬限期（terminationGracePeriodSeconds，預設 30 秒）。這裡有個常見到會讓「明明寫了 handler 卻沒生效」的陷阱：訊號只送給 PID 1，所以前提是你的應用**真的是** PID 1——若它被一層 shell 包住（`sh -c "node app.js"`），PID 1 是那個 shell，預設**不會**把 SIGTERM 轉發給子行程，於是你的 handler 永遠收不到訊號、寬限期一過直接 SIGKILL（解法是用 `exec` 形式啟動讓應用接管 PID 1，或塞一個 `tini` 之類的 init 負責轉發）。這個 SIGTERM 是平台對應用說的一句話：「我要關你了，把手上的事收一收。」一個實作了優雅關閉的 web process，收到 SIGTERM 後該做的事有明確的順序：

```
SIGTERM received
  1. stop accepting new connections   (close the listener / leave the LB pool)
  2. let in-flight requests finish     (drain, up to a deadline)
  3. close DB/cache connections, flush
  4. exit(0)
```

順序是重點。先**停止接受新請求**——對 web process 而言就是不再 accept 新連線、或先讓自己從負載均衡的後端池裡被移除（在 K8s 裡，readiness 變 false 會讓這個 Pod 的 IP 從 Service endpoints 移除，見〈health check〉）。然後**把手上還沒處理完的請求做完**（drain，排空），而不是一刀切斷讓使用者收到半截回應。最後才關掉資料庫連線、把緩衝刷出去、乾淨退出。對一個 worker process（從佇列取訊息來處理的那種），優雅關閉的對應動作是：把當前正在處理的那筆工作**退回佇列**，讓別的 worker 或重啟後的自己再接手，而不是讓它消失。

現在看一個沒實作優雅關閉的 process 會發生什麼。它收到 SIGTERM 後**完全不理會**——因為 SIGTERM 的作業系統預設動作就是終止行程，除非應用顯式註冊 handler 覆寫它。於是 K8s 等不到它自己退出，寬限期一過（30 秒後）直接補一個 `SIGKILL`——這是一個**不可被捕捉、不可被忽略**的訊號，作業系統當場處決這個 process，不給任何收尾機會。此刻那些還在處理中的請求全部當場斷掉：使用者看到 502、那筆轉帳的後半段沒跑完、那條訊息送了一半。最要命的是，這不是「偶爾運氣不好」——**每一次部署、每一次擴縮、每一次 node 排空都會發生**，因為這些操作天天在跑。一個不優雅關閉的服務，等於把「丟掉在途請求」設成了部署的內建副作用。

這裡有個非顯然的邊界值得點出來：**寬限期是有限的，所以 drain 不能無限等。** 如果某個請求要跑 5 分鐘、而寬限期是 30 秒，那這條請求注定會在 drain 途中被 SIGKILL 砍斷——優雅關閉只在「在途請求能在寬限期內跑完」的前提下才優雅。這把一個設計約束逼到檯面上：要嘛你的請求都夠短（能在寬限期內排空），要嘛你把長任務做成可中斷、可恢復的（拆成 worker 從佇列取、收到 SIGTERM 就退回佇列），不能假裝一個 5 分鐘的 HTTP 請求能被優雅地關掉。disposability 不只是「記得寫個 SIGTERM handler」，它逼著你重新想清楚「我的工作單元到底有多大、能不能在 30 秒內收尾」。

還有一個常被忽略的孿生要求藏在 disposability 裡：**process 要能扛得住「非優雅」的猝死。** SIGTERM 是平台**禮貌地**通知你，但機器可能直接斷電、kernel 可能直接 OOM kill（這正是容器記憶體超限的下場，見〈Docker：容器隔離的原理〉）、SIGKILL 可能因為各種原因提前到來。所以一個真正 disposable 的應用，不能把正確性**寄託在**優雅關閉一定會發生上——它的工作得設計成「就算被攔腰斬斷，重啟後也能正確地接續或重來」。對 worker 來說這意味著訊息處理要冪等、要靠 at-least-once + 去重兜住「處理到一半就死」的窗口（見〈重複是常態〉）。優雅關閉是讓**好情況更乾淨**；扛得住猝死是讓**壞情況不致命**。兩者都要，缺一個都不算真的 disposable。

## 原則對、做法老了：config 不該全塞進環境變數

回到剛才掛起來的那條。factor III 說「config 存在環境裡」，這句話其實混進了兩個必須拆開的東西：一個是**恆久正確的原則**（config 與程式碼分離），一個是**2011 年的具體實作建議**（而且實作就用環境變數）。原則通過了時間考驗；實作沒有完全通過——而且破口正好出現在 12-factor 自己沒有特別區分的地方：**機密**。

12-factor 把所有 config 一視同仁——資料庫的 URL、log level、feature flag、以及**資料庫密碼、API key、私鑰**——全部都建議塞進環境變數。這在 2011 年的單機 PaaS 心智模型下尚可接受，但它埋了一個第一原理層級的錯誤：**機密不是「比較敏感的 config」，它是一個性質不同的問題類別。** 一般 config 的目標是「在對的環境注入對的值」；機密的目標完全不同——它要在「這個值遲早會被某種方式洩漏」的悲觀假設下，把**爆炸半徑**和**洩漏後的修復成本**壓到最小。用同一套機制（環境變數）處理兩者，等於用 config 的安全等級去保護機密。

環境變數對機密的具體洩漏面，值得逐一看清楚，因為每一個都是真實的事故管道：

- **它會被子 process 繼承。** 你的應用 fork 出一個子 process（跑個外部命令、起個 helper），整包環境變數預設原封不動傳下去——包括那個資料庫密碼。子 process 是不是該看到它？通常不是，但它看到了。
- **它會被 dump 進 log。** 大量框架在啟動時或崩潰時為了方便除錯，會把整個環境變數印出來。機密就這樣大搖大擺地進了 log，再被 log 系統長期保存、被一整個團隊看到——而且因為 log 通常被視為低敏感資料，這份洩漏可能很久都沒人發現。
- **它躺在 `/proc/<pid>/environ` 裡。** 在 Linux 上，同一台機器上有足夠權限的 process 可以直接讀到別的 process 的環境變數。容器的隔離縮小了這個面，但沒有消滅它。
- **它沒有版本、沒有輪替、沒有稽核。** 環境變數天生不記錄「這個值是什麼時候設的、上次換是何時、誰讀過」。當你某天懷疑某個 key 外洩、要回答「它上次輪替是什麼時候、現在誰還在用它」時，環境變數一個字都答不出來。

所以 2026 年正確的態度是：守住**「config 與程式碼分離」這個原則**，但放掉**「機密也一定用環境變數」這個實作**。一般的、低敏感的 config，環境變數依然是個好選擇——簡單、語言無關、不會被誤 commit。但機密該走不同的路：用掛載檔案（K8s 的 Secret 掛成 tmpfs volume，不落盤、不進環境）而非環境變數，或更進一步用專門的機密管理器拿短命的動態憑證（這條線該怎麼鋪、機密該被當成什麼性質的問題，見〈config 與機密管理〉）。值得補一句：K8s 的 Secret 物件**預設只是 base64 編碼、不是加密**——把機密放進 Secret 不等於保護了它，那是另一個容易被「我用了 Secret 物件」這句話騙過去的陷阱。

這條的教訓比它本身更通用：**讀一份方法論時，要會分辨哪一層是它真正的洞見、哪一層只是它寫成那年的環境留下的指紋。** factor III 的洞見是「分離」，指紋是「環境變數」。十五年後，洞見還站著，指紋該被擦掉重畫。

## 12-factor 沒看見的那一面

最後一類，是 12-factor 不是說錯、而是**根本沒談**的東西——2011 年那些議題要嘛還不成氣候，要嘛還沒被認為是應用作者該操心的事。把這些盲區看清楚，才不會把一份 checklist 當成完整的工程指南。

**最大的盲區是可觀測性。** 12-factor 整套東西裡，關於「怎麼知道你的系統現在好不好」的全部內容，就是 factor XI 那一條——**log**。它完全沒提 metrics，也完全沒提分散式追蹤（trace）。這在 2011 年的單體或少數服務的世界裡勉強夠用：log 一條 stream 大致能還原發生了什麼。但到了今天這種一個請求橫跨十幾個服務的分散式世界，光有 log 是遠遠不夠的——你需要 metrics 來看聚合的趨勢與分布（p99 延遲從哪來、錯誤率有沒有爬升），需要 trace 來把一個請求在十幾個服務間的完整路徑串起來、定位是哪一跳慢了。把 12-factor 當成完整的觀測指引，會留下一個巨大的洞（log、metric、trace 為什麼是三件不同的事、各補什麼，見〈三本柱：metric、log 與 trace〉）。這也是它最常被點名的時代局限。

**第二個盲區是有狀態服務。** 整套 12-factor 建立在「應用無狀態、狀態全外放」的假設上，這對絕大多數無狀態的 web 與 worker 服務都對。但它對「**那些被外放進去的狀態本身**」幾乎隻字未提——資料庫叢集怎麼做、有狀態的串流處理怎麼辦、需要穩定身分與本地持久卷的服務（K8s 裡用 StatefulSet 而非 Deployment 的那一類）該遵守什麼。12-factor 把狀態推給了「backing service」這個黑盒，但現實世界裡，**有人得去寫那個 backing service**，而 12-factor 對那個人沒有任何指引。一個刻意的有狀態服務違反 factor VI 的「無狀態」，不是它做錯了，是 12-factor 的適用範圍本來就不包括它——把「違反 12-factor」當成一種罪名扣在資料庫頭上，是搞錯了這份文件的定位。

理解這些盲區，才能擺正 12-factor 在 2026 年的位置：它是一張**「這個服務雲原生就緒了嗎」的快速體檢表**，是評估一個無狀態服務能不能在容器/K8s/serverless 上跑得順的好起點；但它不是一份完整的現代分散式系統工程指南。當 checklist 的**起點**很好，當**終點**會漏掉可觀測性、機密、有狀態系統這幾個大洞。

## 一份正在被改寫的「定本」

還有一個事實會改變你引用它的方式：**12-factor 不再是一份十五年沒動的凍結文件了。** 2024 年 11 月（2024-11-12），這套方法論被開源——從 Heroku 名下搬到 GitHub 上的 `twelve-factor/twelve-factor` 社群組織，轉為社群治理，用 pull request、issue、review 這套大家熟悉的流程協作演進（2026-06）。一個「更新版」正在 repo 的 `next` 分支上開發，目標是把這套東西刷新到 Kubernetes、GitOps、workload identity 的時代，最終會取代 12factor.net 上的舊版。

維護者定了一個有意思的約束：**因子數量維持十二個不變**——他們把「十二」這個限制本身看成一種價值，逼著這份文件保持聚焦、不膨脹成一份什麼都收的清單。社群提案的方向，恰好印證了前面拆出來的那幾個破口：有人提議把一條因子改寫成關於 **workload identity**（用平台簽發的身分取代到處散落的長命機密），config 那條被討論納入**掛載檔案**而不只是環境變數，log 那條被討論擴展到涵蓋**遙測（telemetry）**——也就是把 metrics 與 trace 補進那個只有 log 的盲區。這些更新在 2026-06 仍在 `next` 分支演進中、尚未定案，所以引用時別把任何單一條目的新措辭當成已凍結的權威。

這件事本身就是一條值得帶走的提醒：當你說「這要符合 12-factor」時，你引用的其實是一份**正在被它的社群重寫**的活文件。它的核心抽象——無狀態、config 外置、log as stream、disposability——依然站得住、依然是判斷一個服務能不能在現代平台上跑好的有效透鏡；但它的細節在 2026 年確實顯老，而且**官方自己正在改**。把它當成不可動搖的石板，恰恰違背了它如今的存在方式。

## 為什麼是這個形狀

退一步看，12-factor 之所以能跨過十五年、跨過一整代基礎設施的更替還沒死，是因為它在無意間把握住了一個比任何具體平台都更持久的東西：**「一個應用要能被一個調度系統自由地擺佈」這件事，對應用程式作者提出的硬性要求。**

正因為應用要能被任意複製到任何一台機器——所以它不能在本地藏狀態。正因為它要能被任意殺掉重建——所以它要能秒起、收到 SIGTERM 要能優雅收尾、還要扛得住來不及收尾的猝死。正因為同一份程式碼要部署成多份不同環境——所以差異要外置成 config。正因為它跑在哪、log 被導去哪都不該由它決定——所以它只管往 stdout 吐。這些不是 Heroku 的偏好，是「可被調度的應用」這個物種的解剖結構，平台換了、結構沒換。

而它顯出年代局限的地方，也來自同一個根源：它是一份回答「**單一應用該怎麼寫**」的文件，所以它對「**一群應用組成的分散式系統該怎麼被觀測、狀態本身該怎麼被管**」這些跨越單一應用邊界的問題，天生沒有著墨——那些問題在 2011 年的單體世界裡還沒被逼到非解不可。理解 12-factor 最好的方式，不是把它當成一張要逐條打勾的清單，而是看穿每一條背後那個「為了能被自由調度，應用必須放棄什麼、必須具備什麼」的取捨——看懂了取捨，你既知道為什麼那幾條至今顛撲不破，也知道為什麼有幾條該被這個時代溫柔地改寫。
