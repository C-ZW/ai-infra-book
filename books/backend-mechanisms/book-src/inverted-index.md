# 倒排索引：全文搜尋的資料結構

有人在你的系統裡丟進一句 `WHERE description LIKE '%refund%'`，當資料只有幾千列時它跑得飛快，沒人多想。三年後資料長到兩千萬列，同一句查詢開始要跑七、八秒，把連線池佔滿，把 p99 拉到天上。你打開查詢計畫，看到那行讓人胃一沉的字：`Seq Scan`。索引完全幫不上忙——你給 `description` 建的 B-tree 索引，對 `LIKE '%refund%'` 連碰都不碰。

這不是調校能救的事。問題出在你**用錯了資料結構**：你想做的是「全文搜尋」，手上拿的卻是一個「為前綴比對而生的有序樹」。要理解為什麼這兩者天差地遠、以及全文搜尋底下那個翻轉一切的結構長什麼樣，得先看清楚 `LIKE '%word%'` 到底笨在哪。

## 為什麼 `LIKE '%word%'` 一定走全表掃描

B-tree 索引（見〈LSM-tree 與 B-tree：寫放大換讀放大〉）的全部本事，建立在一個前提上：**鍵是有序的，而你查的東西是這個順序的前綴。** 索引把所有 `description` 的值排好，`"refund issued"` 排在 `"refunded"` 前面、`"refunded"` 排在 `"region"` 前面。查 `LIKE 'refund%'`（前綴比對）時，B-tree 能一路二分到 `refund` 開頭的那一段、把其餘整棵樹剪掉——這正是它快的原因。

但 `LIKE '%refund%'` 要的是「`refund` 出現在字串**任何位置**」。`refund` 可能在 `"a full refund was given"` 的中間、`"please refund me"` 的尾巴。這些字串在 B-tree 裡四散各處，沒有任何前綴能把它們圈在一起。索引的有序性對「子字串」這個問題**完全使不上力**，查詢規劃器只剩一條路：把每一列的 `description` 整個讀出來、逐字元掃一遍看裡面有沒有 `refund`。這是 `O(N × L)`——`N` 是列數、`L` 是平均字串長度的全表掃描。資料一大就是線性地慢下去，沒有天花板。

更深一層的問題是，就算 `LIKE` 不慢，它做的也根本不是使用者要的搜尋。使用者搜「running」，他多半也想找到「ran」「runs」這些同一個詞的變形；他搜「machine learning」，他要的是這兩個詞**相鄰成片語**、不是分別散落在文件兩端。`LIKE` 對這些一無所知——它是純粹的位元組子字串比對，沒有語意、不認得詞、不認得詞的變形。全文搜尋要解的，是一個比「字串裡有沒有這串位元組」更大的問題。

## 翻轉：從「文件→詞」到「詞→文件」

先把「全文搜尋」這件事拆成它的第一原理。你有一堆文件，使用者給你一個詞，你要回答「**哪些文件包含這個詞**」。`LIKE` 的做法是：對每一份文件問「你裡面有這個詞嗎」——以**文件**為迴圈的外層、詞為要找的東西。這個方向天生是 `O(文件數)`。

倒排索引（inverted index）做的，是把這個迴圈整個翻過來。它預先建好一張表，鍵是**詞**、值是**包含這個詞的所有文件清單**：

```
forward (天真方向):   doc1 -> [the, quick, brown, fox]
                      doc2 -> [the, lazy, brown, dog]

inverted (倒排方向):  "brown" -> [doc1, doc2]
                      "fox"   -> [doc1]
                      "quick" -> [doc1]
                      "lazy"  -> [doc2]
                      "dog"   -> [doc2]
```

「倒排」這個名字就是這個意思：正常的映射方向是 `文件 → 它包含哪些詞`，倒排索引把箭頭反過來，存的是 `詞 → 它出現在哪些文件`。一旦這張表建好，查「哪些文件含 `brown`」就退化成一次**字典查找**：直接拿 `brown` 這個鍵去取它的清單，`O(1)` 拿到 `[doc1, doc2]`，完全不碰其他任何文件。線性掃描變成了直接查找——這就是倒排索引存在的全部理由，其餘所有機制都是圍繞「怎麼把這張表建好、存小、查快」長出來的。

那張詞對清單的表，術語叫法各有不同但結構一致：每個詞叫一個 **term**；某個 term 對應的那份「我出現在哪些文件」的清單，叫它的 **posting list**（倒排串列），清單裡的每一筆（「我在 doc7、出現了 3 次、在第 12 和 40 個位置」）叫一個 **posting**。所有 term 收集起來、排好序，構成 **term dictionary**（詞典）。整個倒排索引，就是「一本詞典 + 每個詞掛一條 posting list」。

## 索引建出來之前：analysis 決定了你搜不搜得到

在能建 posting list 之前，得先回答一個比想像中關鍵得多的問題：**「詞」到底是什麼？** 一份文件是一串連續的字元，把它切成一個個 term 的這道工序叫 **analysis（分析）**，它是整個全文搜尋裡最被低估、卻最常決定「搜得到搜不到」的一步。

analysis 把原始文字推過一條管線：

1. **Tokenization（斷詞）**：把字元流切成一個個 token。英文這類有空格的語言可以粗暴地按空白與標點切，`"The quick-brown fox."` 切成 `the / quick / brown / fox`。
2. **正規化（normalization）**：把 token 折成一個標準形式——轉小寫（`The` → `the`，這樣搜尋大小寫不敏感）、去掉標點、把全形半形統一。
3. **Stemming / lemmatization（取詞幹）**：把同一個詞的各種屈折變化收斂到同一個根。經典的 Porter stemmer 純粹做後綴剝除（suffix stripping、不查詞庫），會把 `running`、`runs` 都折成 `run`，於是搜 `running` 也能命中只寫了 `runs` 的文件；但 `ran` 這種不規則變化沒有可剝的後綴，純 stemmer 對它無能為力（`ran` 會原封不動留下），要把 `ran` 也收斂到 `run` 得靠 lemmatization（帶詞庫的形態分析）。
4. **去停用詞（stop words）**：把 `the`、`a`、`of`、`is` 這類幾乎每份文件都有、區分力等於零的高頻詞丟掉，省下巨大的 posting list（`the` 的 posting list 幾乎涵蓋全部文件，存它幾乎沒有資訊量）。

這條管線跑完，`"The dogs were running"` 變成 `[dog, run]`——只剩兩個正規化後的 term。建索引時存的是這兩個 term。

這裡藏著倒排索引**最容易出錯、也最隱蔽**的一條鐵律：**建索引和查詢，必須用同一套 analyzer。** 想清楚為什麼。建索引時你把 `"running"` 經 stemming 存成了 `run`。現在使用者搜 `"running"`——如果查詢這一側**沒有**跑同一套 stemming，系統會拿著原封不動的 `running` 去 term dictionary 裡找，而字典裡只有 `run`、沒有 `running`，於是查無此詞、零結果。文件明明在那裡，卻搜不到。反過來，查詢端 stem 了、索引端沒 stem，一樣對不上。這個 bug 特別陰險的地方在於：它不會報錯、不會慢、不會崩，它只是**安靜地少給你結果**——而「少給結果」這種失敗，在搜尋裡是最難被發現的，因為你根本不知道本該有什麼。

中文、日文、泰文這類沒有詞間空格的語言，把這道工序的重要性又往上推了一級。`"全文搜尋很有用"` 沒有空格可以切，你需要一個斷詞器（如 jieba、ICU 的 word break）把它切成 `全文 / 搜尋 / 很 / 有用`。沒配斷詞器、或退化成「一字一 token」，整個搜尋的語意就垮了——使用者搜「搜尋」搜不到一篇講「全文搜尋」的文件，因為索引裡根本沒有「搜尋」這個 term，只有一個個孤立的單字。**analysis 配錯，後面的資料結構再精巧都是白搭**，因為你索引的根本不是使用者心裡的那些詞。

## posting list 怎麼查：交集、聯集，與「往前跳」

term dictionary 把單一詞的查找變成 `O(1)`，但真實查詢往往不是單一詞。使用者搜 `payment AND refund`，要的是「同時含這兩個詞」的文件——這需要把 `payment` 的 posting list 和 `refund` 的 posting list 做**交集**。`payment OR refund` 是聯集，`payment NOT refund` 是差集。這些集合運算的效率，決定了多詞查詢快不快，而它快的祕密全在一個設計決定上：**posting list 裡的文件 ID 永遠保持遞增排序。**

排序為什麼是關鍵？想算兩個清單的交集。如果它們是亂序的，你只能拿一邊的每個元素去另一邊做雜湊查找或全掃，那是 `O(m × n)`。但如果兩邊都**已經排好序**，你可以用一個漂亮的「拉鏈式合併」（merge）：兩個指標各指一邊的開頭，比較它們指向的文件 ID——相等就是一個交集命中、兩邊同時前進；不等就讓**指向較小 ID 的那一側**單獨前進。每個元素至多被看一次，交集成本降到 `O(m + n)`。

把這套算術落到真實量級上看效益有多大。假設一個語料有 100 萬篇文件，要查同時含 `payment` 與 `refund` 的文件。`payment` 在語料裡的 posting list 命中 5,000 篇、`refund` 命中 3,000 篇。`LIKE` 路徑要對 100 萬篇全部做子字串比對；倒排路徑只取出這兩條 posting list，做一次排序合併，成本約 `5000 + 3000 = 8,000` 次比較。**從百萬級的掃描，降到萬級的合併——這就是倒排把搜尋從「掃描」變成「查找」的具體標價。**

但搜尋引擎還更貪心。注意上面那兩條清單**長度懸殊**：一條 5,000、一條 3,000。當其中一條極短、另一條極長時（比如 `the`（若沒被當停用詞濾掉）命中 90 萬篇、`fox` 只命中 50 篇），純線性合併還是得把長清單從頭走到尾，浪費。真正的引擎用的是 **galloping / leapfrog（跳躍式前進）**：每條 posting list 都支援一個 `advance(target)` 操作——「跳到第一個 ≥ target 的文件 ID」。做交集時，先看短清單的下一個文件 ID，然後叫長清單**直接跳**到那個位置附近，跳過中間一大段不可能命中的文件。要讓 `advance` 本身也快，posting list 上會疊一層 **skip list（跳躍表）**：一個稀疏的索引，記著「第 128 筆在這裡、第 256 筆在這裡」，讓 `advance` 能大步邁過整個區塊，而不是逐筆走。Lucene 的 posting 格式正是這樣——postings 按區塊（block）存、區塊上掛 skip data，讓 `advance` 跳過整個區塊。

這也解釋了一個常被忽略的優化：做多詞 AND 時，引擎會**從最短的 posting list 開始驅動**整個交集——因為命中數的上界就是最短那條清單的長度，沒理由讓最長的清單當主迴圈。一個含 50 篇的 `fox`，無論 `the` 命中多少篇，最終交集都不會超過 50 筆，那就以 `fox` 為基準、拿著它的每個 ID 去其他清單 `advance`。

## 片語查詢：為什麼還得存「位置」

到目前為止，posting 裡只要存「哪些文件」就夠回答布林查詢。但使用者搜 `"machine learning"` 加了引號——他要的是這兩個詞**恰好相鄰、構成一個片語**，而不是一份文件裡 `machine` 出現在第一段、`learning` 出現在第十段這種「分別都有、但根本不相關」的情況。光靠「文件含這兩個詞」的交集，無法區分這兩種文件。

要支援片語查詢，posting 裡就不能只存文件 ID，還得存這個 term 在該文件裡的**每一個出現位置**（position，第幾個 token）。於是 `machine` 在 doc7 的 posting 變成「doc7,位置 [4, 88]」，`learning` 的變成「doc7,位置 [5, 200]」。查片語時，先用文件 ID 交集找出同時含兩詞的候選文件，再對每個候選比對位置：`machine` 在位置 4、`learning` 在位置 5——位置差恰好是 1，相鄰，命中。doc7 雖然也含 `learning`（在位置 200），但那個和最近的 `machine`（位置 88）差了 112，不構成片語。位置資訊讓「相鄰」這個概念第一次變得可計算。

但這個能力不是免費的。**存位置會讓索引顯著膨脹**——原本一個 posting 只記一個文件 ID，現在要記該詞在文件裡出現的每一個位置；一個高頻詞在一份長文件裡可能出現幾十次，就是幾十個位置整數。索引大小可能因此翻倍甚至更多。這帶出一個實打實的取捨：**如果你的搜尋不需要片語查詢，就別索引位置。** Lucene 允許你按欄位設定索引選項（只索引文件、文件+詞頻、文件+詞頻+位置），關掉位置能省下可觀的空間與寫入成本。把「以防萬一」的位置索引預設打開，是很多搜尋索引悄悄變肥的原因。

## 為什麼倒排索引塞得進記憶體：壓縮

一個百萬文件、上億 token 的語料，posting list 加起來的原始大小非常可觀。搜尋引擎能把它壓到記憶體裝得下、且查得飛快，靠的是幾層精巧的壓縮——而這些壓縮能成立，全因為前面那個「文件 ID 保持遞增排序」的決定。

第一層是 **delta encoding（差值編碼）**。一條 posting list 的文件 ID 是嚴格遞增的，比如 `[1042, 1051, 1063, 1078, ...]`。與其存這些動輒四、五位數的絕對值，不如存**相鄰差值**：`[1042, 9, 12, 15, ...]`。關鍵洞察是：一個高頻詞——比如出現在 40% 文件裡的詞——它的 posting list 又長又密，相鄰文件 ID 的間隔平均很小（一個出現在 40% 文件的詞，平均間隔約 2.5）。差值因此擠成一堆很小的數字，而小數字可以用兩類策略省下大量位元：一是 **variable-byte（VByte）**——逐個數字用一到數個位元組、小數字只花一兩個位元組；二是**整塊定寬打包（PackedInts / FOR）**——把整個區塊的數字用同一個剛好夠用的固定位元寬度打包，避開為少數大數而每個都用滿四位元組。Lucene 的 postings 走的正是後者：按區塊（每塊 128 筆）把文件 ID 差值與詞頻做定寬打包的區塊壓縮（block-compressed delta）。

注意這層壓縮和「往前跳」的 `advance` 之間有個微妙的張力：差值編碼是**相對**的，要知道第 1000 筆的絕對文件 ID，理論上得從頭把前面所有差值累加。skip list 正是來化解這個張力——它在稀疏的幾個點上記下**絕對**文件 ID 與該點在壓縮串流裡的位置，於是 `advance` 能跳到最近的 skip point、從那裡的絕對值開始往下累加，而不必從清單開頭重算。壓縮與隨機存取這兩個看似矛盾的需求，靠 skip list 同時滿足。

term dictionary 那一側也有自己的壓縮故事。詞典是「所有不重複 term 的有序清單」，可能有數百萬個詞。Lucene 不是把它們平鋪存著，而是用 **FST（finite state transducer，有限狀態轉換器）**——一種把共享前綴折疊起來的緊湊結構，類似一棵被高度壓縮的 trie。`run`、`running`（若未 stem）、`runner` 共享前綴 `run`，FST 只存一次這個前綴的路徑。FST 不只省空間，還快：它能在微秒級做前綴查找，甚至在不碰磁碟的情況下就判定「這個詞根本不存在」——`seekExact` 常常無需任何 I/O 就能回答「查無此詞」。對於「使用者打了個拼錯的詞」這種高頻情況，能在記憶體裡瞬間否決、不去碰磁碟上的 posting，是個不小的省。

## 它建在哪裡：從一句 SQL 到一座叢集

倒排索引不是只有大型搜尋引擎才有，它是一個**資料結構**，可以裝在很多地方，而「裝在哪」決定了你付出的維運與一致性代價。

最近的地方，是你的關聯式資料庫**本身**。PostgreSQL 的 `tsvector` / `tsquery` 配 **GIN 索引**，做的就是一個道地的倒排索引——GIN 內部存的正是「(鍵, posting list)」對，每個 lexeme（正規化後的詞）一個鍵，掛一條壓縮過的 row ID（命中文件的列指標）清單。注意標準 GIN 的 posting list 只存列指標、**不**存 lexeme 在文件裡的位置，所以片語查詢（`<->`）在索引層只能粗篩、命中後得回表（recheck）讀原始 `tsvector` 比對位置；要在索引層就支援片語，得改用 RUM 這類 extension。MySQL 的 `FULLTEXT` 索引同理。選它的最大好處，是搜尋索引和你的權威資料**在同一個資料庫、同一筆交易裡**：寫進去的那一刻就可搜尋，沒有「來源庫和搜尋索引之間的同步延遲」這回事，也不必多養一套系統。代價是 analysis 能力（多語言、同義詞、自訂分詞）和查詢豐富度，通常不如專用引擎。

中等而專用的地方，是嵌入式搜尋庫——**Lucene**（Java）、**Tantivy**（Rust）、**SQLite FTS5**。FTS5 在一個 SQLite 檔案裡建出和 Lucene 同款的倒排索引，它的 doclist 一樣用 delta-encoded 的 rowid 和 position list 來壓縮、支援片語查詢。它們把倒排索引嵌進你自己的行程，免去維運一座外部叢集。

最重、也最強的地方，是專用的分散式搜尋引擎——**Elasticsearch / OpenSearch**（底層都是 Lucene）、**Apache Solr**。它們提供完整的 analysis 管線、分散式分片、近即時索引、豐富的查詢 DSL。但引入它，等於引入一個**獨立的系統**和一條「來源資料庫 → 搜尋索引」的資料同步管線——而這條管線把全文搜尋從一個資料結構問題，變成了一個分散式一致性問題。

## 它會怎麼騙你：搜尋索引不是你的權威資料

最後這個故障模式，是把倒排索引從專用引擎取用時最容易踩、後果也最重的一個，值得單獨講透。

Elasticsearch 這類引擎，和你的來源資料庫之間是**最終一致**的（見〈一致性光譜：線性、因果、讀己之寫、最終一致〉）。你的 source of truth 在 PostgreSQL，每次寫入要再經過一條管線——可能是應用層 dual-write、可能是 CDC 從資料庫變更長出事件流（見〈CDC：從資料庫變更長出事件流〉）——把變更同步進搜尋索引。這條管線有延遲、會落後、偶爾會斷。於是會出現各種「資料庫裡明明刪了，搜尋還搜得到」「剛寫的東西搜尋還看不到」的鬼影，本質都是這條同步管線的最終一致性在現身。

從這個事實會推出一條硬規則：**永遠不要把搜尋索引當權威資料庫。** 它是個為「搜得快」而高度優化的衍生視圖，不是交易性的、不保證不丟資料。它可以在某次重建索引時被整個丟掉重灌，也理當能被——因為它的真相在別處，隨時能從來源重新索引出來。如果你把唯一一份權威資料只存在 Elasticsearch 裡，你是把系統的命脈押在一個設計上就不為持久權威而生的東西上。它丟了資料不是它壞了，是你用錯了它的定位。

這條規則的另一面，是一個同樣常見的反向錯誤：**為了「未來可能要搜尋」就先架一座 Elasticsearch 叢集**，結果只有幾萬筆資料。在那個規模，PostgreSQL 的 GIN 索引綽綽有餘，而你卻付出了多一個分散式系統的維運、多一條同步管線的一致性負擔——用實打實的複雜度，換一個還沒到來的需求。倒排索引該裝在哪，取決於你的規模和搜尋對業務的權重，不是取決於哪個名字聽起來最厲害。

## 為何是這個形狀

退一步看，倒排索引的每個部件，都是從「把『哪些文件含這個詞』這個問題答快」這一個目標裡，被逼出來的必然形狀。

要把線性掃描變成直接查找，就得預先翻轉映射方向、用詞當鍵——於是有了 term dictionary 和 posting list。要讓多詞查詢的集合運算快，posting list 就**必須**保持文件 ID 遞增排序——於是交集退化成排序合併、`advance` 變得可能。正因為排序了，相鄰文件 ID 的差值才小、才壓得動——於是有了 delta encoding，而它和隨機跳躍的矛盾，又催生了 skip list。要讓搜尋認得「詞」而非位元組、認得詞的變形與相鄰，就得在建索引前跑一道 analysis、在 posting 裡存位置——而 analysis 必須兩端一致，否則整套精巧結構索引的根本不是使用者要的詞。

`LIKE '%word%'` 的悲劇，是它用一個為「有序前綴」而生的結構，去答一個「任意位置的詞」的問題，方向從一開始就錯了。倒排索引不是 `LIKE` 的加速版——它是把同一個問題**換一個方向問**之後，自然長出來的答案。
