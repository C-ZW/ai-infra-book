# 失敗偵測：逾時是唯一手段、慢與死分不開

凌晨三點，監控面板上一個節點的指標斷了。它不再回報 CPU、不再回應健康檢查、不再對任何請求出聲。值班的人盯著那塊變灰的方格，要做一個看似簡單的判斷：這台機器**死了**，還是只是**很慢**？

這個判斷不能拖。如果它死了，它正在負責的工作得趕快交給別人——它持有的那個分散式鎖要釋放、它當的那個 leader 要重選、它沒處理完的訊息要重派。每多等一秒，故障就多影響一秒。但如果它沒死，只是某個 GC 暫停卡了它五秒、或它和監控之間的網路抖了一下，那麼「把它的工作交給別人」就是一場災難：兩個節點會同時自認是 leader、同一筆訂單會被處理兩次、同一把鎖會被兩個人以為自己持有。

整個失敗偵測這門學問，就卡在這個三點鐘的判斷上。而它最殘酷的地方在於：**你手上根本沒有能做出這個判斷的工具。** 你以為你在問「它死了嗎」，但你能觀測到的只有一件事——「我發出去的東西，在我等的時間內沒有回來」。從這件事，到「它死了」，中間隔著一道**原則上跨不過去**的鴻溝。

## 你能依靠的只有一件事，而那件事證明不了任何事

把那塊變灰的方格放慢看。你（或你的偵測程式）對那個節點發了一個探測——一個心跳請求、一個 health endpoint 的 GET、一個 RPC——然後沒收到回應。這個「沒收到回應」背後，可能是四種完全不同的真相：

1. 探測請求在路上就丟了，對方根本沒收到；
2. 對方收到了、正在處理，回應還沒生出來（它可能卡在一次長 GC、一次磁碟 I/O、或單純 CPU 排隊排不到）；
3. 對方處理完了、回應也發出來了，但回應在回程丟了；
4. 對方整個當機了，連 OS 都沒了。

這四種情況，你**該做的反應南轅北轍**：第 (4) 種你該接手它的工作；第 (2)(3) 種你該耐心再等或重試，絕對不能接手；第 (1) 種你該重送探測。但在一個訊息延遲沒有上界的網路裡，這四種情況從你這端看出去**完全一樣**——都是「我等的窗口內沒有東西回來」。你唯一能用的偵測工具是逾時（timeout），而逾時能告訴你的，只有「它沒在我選的這個時限內回應」。它永遠無法告訴你「它死了」。

這不是工具不夠好，是更深的東西。把問題逼到底，會發現它和分散式系統最根本的那個難題是同一個——**部分失敗（partial failure）**：在非同步網路裡，一個沉默的節點是慢是死，外部原則上不可區分（這個第一原理見〈為什麼分散式這麼難〉）。失敗偵測就站在這個原理最赤裸的位置上：別的機制還能繞，失敗偵測無處可繞，因為它的全部工作就是去做那個「原則上做不到」的區分。

於是有一個你必須先接受、否則後面都會做錯的事實：**完美的失敗偵測器不存在。** 在一個訊息可能任意延遲的非同步網路裡，你造不出一個「絕不誤判」的偵測器。這不是還沒有人發明出來，是它被證明了不可能——而從這個不可能裡，長出了失敗偵測的整個形狀。

## 一個逾時值，就是一個賭注

既然逾時是唯一的手段，而它又證明不了任何事，那設計失敗偵測就退化成一件事：**選一個逾時值。** 而每一個逾時值，都是一個賭注——你在「把健康節點誤判成死」和「真死了卻遲遲沒發現」這兩種錯誤之間下注，而且你**不可能同時把兩個都壓到零**。

這不是空話，可以算給你看。設想一個服務，正常回應時間的分布是 p99 = 200 毫秒、p999 = 900 毫秒——也就是 99% 的請求在 200 ms 內回來，但有千分之一的請求會慢到 900 ms（這很常見：偶發的 GC、快取 miss、鄰居吵架式的資源競爭，都會製造這條長尾）。現在你要替你的探測選一個逾時值。

先試 **250 ms**（略高於 p99）。看起來合理——絕大多數請求都在這之內回來。但代價是：所有跑超過 250 ms 的請求，都會被你當成「失敗」。那是 p99 以上那一撮，略低於**百分之一**的請求（逾時卡在 250 ms 而非 p99 的 200 ms，所以只罩到那 1% 裡更慢的一部分）。這些節點其實**活得好好的**，只是這一次慢了，卻被你誤判為死，觸發了重試、甚至 failover。更糟的是，如果這些重試又疊到本來就慢的請求上，會把更多請求推過 250 ms 這條線，製造更多誤判——一個正回饋的雪球（這條雪球怎麼滾成系統性崩潰，見〈retry storm 與 metastable failure〉）。

那把逾時放寬呢。試 **900 ms**（約等於 p999）。現在誤判率掉到大約**千分之一**——只剩最尾巴那一撮真正異常慢的會被冤枉。誤判問題幾乎消失了。但你買到的這個「準」，是用**偵測延遲**換的：一個**真的死掉**的節點，現在要等滿將近一秒，你才肯承認它死了。在這將近一秒裡，每一個打進這個黑洞的請求都白白等滿、然後失敗。如果這是一個每秒幾千請求的服務，那將近一秒就是幾千個請求被葬送。

把這兩個數字並排，你就看見了那枚硬幣的兩面：

| 逾時值 | 誤判率（把活的當死） | 偵測延遲（真死了多久才發現） |
|---|---|---|
| 250 ms（≈ p99） | ~1% | 快，但代價是頻繁誤判與重試風暴 |
| 900 ms（≈ p999） | ~0.1% | 慢，真死的節點要等近 1s 才被判定 |

沒有第三欄能讓兩格都變綠。你往左移逾時值，誤判上升、偵測變快；往右移，誤判下降、偵測變慢。這條取捨曲線是失敗偵測的物理，不是你調參技巧不夠好。任何宣稱「我們的偵測又準又快」的說法，要嘛是網路在那段時間恰好很乖、要嘛是它偷偷把賭注挪到了別的地方——下一節就是在講怎麼挪。

值得記死的是這條取捨在學術上的名字。1996 年 Chandra 與 Toueg 把失敗偵測器的性質拆成兩個正交的維度：**completeness（完整性）**——真的死掉的節點最終都會被懷疑；**accuracy（準確性）**——不去懷疑那些其實沒死的節點。前者對應「別漏掉真的死」，後者對應「別冤枉沒死的」。他們證明了一件呼應上面那枚硬幣的事：在純非同步系統裡，你拿不到一個「既永遠完整又永遠準確」的偵測器（那叫 perfect failure detector，P）；你最多只能期待**最終準確**——容許前期犯錯、但網路穩定後不再冤枉好人（◇P / ◇S 這一類）。換句話說，理論早就告訴你：別追求那個三點鐘判斷的「絕對正確」，去追求「最終會對、且錯的時候不要釀災」。

## 把單次賭注，變成一連串更聰明的賭注

既然單次逾時這個賭注必輸一頭，工程上的整套發展，都是在做同一件事：**不去消除誤判（消不掉），而是加入更多訊號，把賭注下得更聰明、並讓賭輸時的損失更小。** 幾種手段，由簡到精：

**心跳，連續漏 N 拍才判死。** 與其讓「一次逾時」就定生死，不如要求對方週期性地主動發「我還在」，而你連續漏掉 N 拍才宣告它死。這等於把一次性的賭注，攤成 N 次獨立的觀測——一次偶發的封包遺失或一次短暫的 GC，只會讓你漏一拍，不會誤判；要連續 N 拍都沒訊號，才比較可能是真的死了。代價很直接：**偵測延遲變成 N × 心跳間隔**。心跳每秒一次、要漏 3 拍才判死，那一個真死的節點最長要 3 秒才被發現。你又在那枚硬幣上移動了——用更慢的偵測，換更低的誤判。

**動態逾時，讓賭注跟著網路走。** 固定逾時的問題是它假設網路的脾氣是固定的。但跨可用區的 RTT 和同機架的 RTT 差一個數量級；同一條鏈路在尖峰和離峰也不一樣。聰明的做法是持續量測「對方回應的到達間隔」的歷史分布，讓逾時值跟著這個分布浮動——網路本來就慢的時候，自動把逾時放寬，別去冤枉它。最漂亮的一個版本是 **phi accrual failure detector**（Hayashibara 等人，2004）。它的洞見是：別輸出「死/活」這個二元判斷，改輸出一個連續的**懷疑值 φ**。φ 的定義是「假設到達間隔服從歷史量到的分布，現在這麼久還沒等到下一拍，這件事有多不可能」——用對數尺度表達。φ 越高，代表「它還活著」這個假設越站不住腳。

這個設計的好處是它把「下注的門檻」從偵測器手裡，交還給了每一個消費者。同一個 φ 值，一個對誤判極敏感的消費者（比如會觸發 failover 的那種）可以設「φ 超過 12 才信」，一個無所謂的消費者可以設「φ 超過 8 就行」——同一套量測，服務不同的敏感度需求。這正是 Cassandra 與 Akka 採用的路線：Cassandra 的 `phi_convict_threshold` 預設是 **8**，而運維文件特別建議：跑在像 EC2 這種網路偶爾會抽風的雲環境上，把它調到 **10–12**——這就是「在更不穩的網路上，把賭注下得更保守一點」的具體旋鈕。

**問別人，別只信自己的一面之詞。** 你判不準對方，很可能是**你和它之間**那段網路出了問題，而不是它本身死了。那就別只憑你一個人的視角下判斷——去問第三方。這正是 SWIM 這類 gossip 式成員協定（gossip 機制本身見〈gossip 與 anti-entropy〉）的核心招數，值得把它的節奏看清楚，因為它把前面所有手段縫成了一個完整的東西。

SWIM 每隔一個固定的「協定週期」T（量級在數秒；最廣泛部署的 memberlist/Serf 預設約 1 秒、跨地端 WAN 設定約 5 秒）做一次：隨機挑一個節點，直接 ping 它。如果在逾時內收到 ack，沒事。如果沒收到——關鍵就在這裡，它**不立刻判死**——它改為隨機找 **k 個別的節點**，請它們「幫我代探一下那個傢伙」（ping-req）。這 k 個節點各自去 ping 目標，只要其中**任何一個**收到 ack，就把這個好消息轉達回來。只有當「我直接 ping 不到，且 k 個代探者也全都 ping 不到」時，目標才會被標記。這一招直接拆掉了「我和它之間網路壞了」這個最常見的誤判來源：如果是你這條鏈路的問題，那 k 個從別的角度出發的代探者多半連得上，目標就洗清了嫌疑。

SWIM 還加了一層叫 **suspicion（懷疑態）** 的緩衝：代探也失敗時，目標不是直接被宣告「死」，而是先進入「suspect（疑似死亡）」狀態，並把這個懷疑 gossip 給全叢集。被懷疑的節點如果還活著、收到了這個傳聞，可以發一個反駁把自己洗白；只有在懷疑態維持夠久、沒人替它說話，才升級成「confirmed dead」。這又是那枚硬幣的同一個動作——**用一點偵測延遲，換顯著更低的誤判率**。後來的 Lifeguard 擴充（HashiCorp 2017）更進一步：它讓每個節點意識到「搞不好是**我自己**最近反應慢、是我的偵測器有問題」——當一個節點發現自己最近錯過很多訊息（自身健康度差），它會主動**放寬自己的逾時、延長自己發出的懷疑的緩衝**，避免一個被 CPU 打爆、自身都慢的節點，到處亂指別人死。這是對「誤判源頭可能是觀測者自己」這件事最徹底的回應。

值得停下來說一句的是：你也許會想「TCP 不是有 keepalive 嗎，讓它幫我偵測不就好了」。可以，但它的預設值會讓你失望——Linux 的 `tcp_keepalive_time` 預設是 **7200 秒（兩小時）**：一條閒置的連線要閒置滿兩小時，才開始發第一個 keepalive 探測，之後再經過幾次探測失敗（預設 9 次探測×75 秒，再加約十一分鐘）才認定連線死掉。這個預設是為「清掉殭屍連線、別浪費網路」設計的，**不是為「快速偵測對端故障」設計的**。誰拿它當失敗偵測器，誰就會在對端死掉後等上兩個多小時才知道。要快，你得自己在應用層發心跳、自己設那個賭注值——這也再次說明，失敗偵測這個賭注，沒有人能替你下。

## 賭一定會輸，所以真正的解法是讓「輸」不致命

把上面所有手段加起來，你能把誤判率壓得很低、把偵測延遲調得還能接受。但你**消不掉**誤判——非同步網路裡它原則上就在那。所以一個成熟的設計，最後一定會繞到一個和「把偵測做得更準」正交的想法上：

> 既然誤判一定會發生，那就別把全部力氣花在「不要誤判」上，把力氣花在「**誤判發生時不要釀成資料災難**」上。

這個轉向是失敗偵測這章真正的高潮，因為它承認了前面所有努力的天花板，然後從天花板上方找到了出路。具體長什麼樣？回到開場那個三點鐘的判斷。假設你誤判了——你斷定那個節點死了，把它持有的分散式鎖搶過來、把它的 leader 角色讓給了別人。然後那個節點**醒過來了**（它根本沒死，只是剛剛卡在一次七秒的 GC 暫停裡）。它一醒來，渾然不覺自己已經被廢，拿著它以為還握著的鎖，提交一筆寫入。

如果你的系統對這筆寫入毫無防備，這就是一場資料損壞：兩個「持鎖者」先後寫入同一份資料，覆蓋、錯亂、雙重扣款。但如果你在每次「換手」時都附帶一個**單調遞增的版本號**——一個 fencing token——故事就不同了。新的持鎖者拿到的 token 比舊的大；下游的儲存只接受「token 不小於它見過的最大值」的寫入。那個醒過來的舊節點，手裡攥著一個**過期的、較小的** token，它的寫入一到下游就被乾脆地回絕。它的「自以為還是持鎖者」是空的——誤判依然發生了，但它**沒有造成任何損壞**。（fencing token 的完整機制，包括它為什麼必須單調、為什麼不能只靠「我記得我有鎖」，見〈分散式鎖〉。）

這就是面對「完美偵測不存在」的正解：**不是追求零誤判（做不到），是讓殘餘的誤判無害。** 同樣的思路貫穿失敗偵測會觸發的每一種動作——只要失敗偵測會導致「重做」（重試一個請求）或「換人做」（failover 一個角色），下游就必須能容忍「被判死的那個其實沒死、稍後會帶著陳舊的認知回來」。重做的那條路，靠**冪等**兜住（同一筆操作做兩次的效果等於做一次，定義見〈重複是常態〉）；換人做的那條路，靠 fencing token 兜住。把這兩道防線備齊，你才敢去做那個三點鐘的判斷——因為你知道，就算判錯了，最壞也只是白忙一場，不會把帳目寫成兩份。

## 在雲端與排程這一層，逾時的賭注被攤在你面前

前面講的偵測器、心跳、SWIM，多半藏在你依賴的基礎設施底下。但有一層，這個「逾時是唯一手段」的約束會**直接攤到你的設計桌上**，逼你親自下注——就是 serverless 與排程這一層。它的執行單元（function 實例、排程觸發出來的 Job）是「牲口不是寵物」：隨時可能被回收、搬遷、因擴縮而生滅。平台和你的工作之間隔著一道故障邊界，而平台判斷你的工作死活、你判斷平台給你的實例死活，用的都是同一個工具：逾時。

最乾淨的例子是雲端 function 的逾時上限。它不是一個「建議值」，而是平台替你下的那個賭注的**硬邊界**：時間到了，平台就認定「這個 invocation 要嘛死了、要嘛慢到不值得再等」，然後直接砍掉它——不管它其實是不是正卡在某個馬上就要回來的下游呼叫上。AWS Lambda 的這個上限是 **900 秒（15 分鐘）**（2026-06）。這正是前一節那枚硬幣在 serverless 世界的化身：平台不可能等你的 function 無限久（那它就無法回收資源、無法區分「卡死」和「正常的慢」），所以它必須設一個逾時；而這個逾時一旦觸發在一個其實只是慢的工作上，就是一次誤判——你那個跑到第 14 分鐘的批次，會在毫無預警下被腰斬。設計長流程時，你不是在「避免逾時」，你是在**和平台對這個賭注達成共識**：把超過上限的工作拆步、或交給能持久化中間狀態的工作流，讓任何一段被砍掉時都能從斷點續跑——這又回到了「讓誤判無害」。

排程觸發這一層更微妙，因為「逾時的賭注」在這裡會偽裝成「漏跑與重複」。一個排程器（K8s CronJob、雲端 cron）和它觸發的工作之間隔著故障邊界，於是它**無法保證 exactly-once**——Kubernetes 的文件就白紙黑字寫著：在某些情況下，一個 Job 可能被建立**兩次**，或**一次都不建立**。為什麼？因為排程器要判斷「上一次觸發成功了沒、現在該不該補」，靠的還是那套「在時限內有沒有看到預期的訊號」——controller 重啟、API server 短暫不可達、時鐘抖動，都會讓它把「其實成功了」誤判成「沒成功，補一個」（於是重複），或把「該觸發」錯過（於是漏跑）。這就是失敗偵測的不確定性，換了一身排程的衣服。

它甚至有一個會把人坑慘的具體陷阱。CronJob 的 controller 會算「自上次排程以來錯過了幾次」，而如果這個數字**超過 100**，它會**放棄補跑這一整批錯過的排程**、記一行 `TooManyMissedTimes` 警示，然後在**下一個排程時間點照常恢復觸發**（除非你設了 `startingDeadlineSeconds` 來縮短它計數的窗口）。把這個機制和一個高頻排程放在一起，算一下就知道有多脆：一個每分鐘觸發的 CronJob，只要叢集 control plane 升級或故障，讓 controller 停了 **1 小時 41 分（101 分鐘）**，錯過次數就破百——這批漏掉的工作**全部被靜默放棄**，恢復後雖然下一分鐘就照常觸發，但中間那一百多次該跑沒跑的窗，沒有任何工作會回頭補上。而它留下的全部痕跡，只有 controller log 裡那一行警示。監控如果只盯 Pod 健康、沒專門盯這條，運維會以為任務一直在好好地跑，直到某天有人發現中間那段時間的報表整片空了。這是「失敗偵測的不確定性沒被正視」會付出的真實代價：你以為排程器在替你可靠地數時間，其實它也在那道故障邊界前，做著和你一樣的、會出錯的逾時判斷。

對策和前面是同一套，因為問題的根是同一個。既然觸發不是 exactly-once、漏跑與重複都是常態，那麼**每一個被觸發的工作都必須冪等**——同一個邏輯時間點被觸發兩次，效果不能等於做兩次（見〈重複是常態〉）。而那個 100 次靜默放棄，要嘛設 `startingDeadlineSeconds` 收窄它的計數窗口、要嘛把這條警示納入監控。你會發現你又繞回了同一個結論：在這道邊界上，你下的賭注一定會偶爾輸，所以真正的功夫不在「賭贏」，在「輸了也不損壞、輸了你會知道」。

順帶澄清一個容易混的點：本章講的是「在 serverless / 排程這層，活與死只能用逾時去猜」這個分散式視角下的約束；至於一個服務該怎麼設計它的存活探針——liveness 探針和 readiness 探針怎麼分、探什麼、設多敏感——那是探針自己的一套機制（見〈health check〉）。值得借過來印證的，只是它的預設值同樣是那枚硬幣的產物：Kubernetes liveness 探針預設每 **10 秒**探一次、連續失敗 **3 次**才判定容器不健康（2026-06）——也就是大約 **30 秒**的連續失敗才會觸發重啟。為什麼是「連續 3 次」而不是「1 次」？正是為了用一點偵測延遲，去濾掉一次偶發的慢，避免把一個只是卡了一下的健康容器當死的給殺了。連這個最日常的設定背後，都站著本章開頭那個三點鐘的判斷。

## 為什麼是這個形狀

退一步看，失敗偵測整套東西——逾時、心跳、phi 值、代探、suspicion、fencing token、function 的 15 分鐘上限、CronJob 的非 exactly-once——全都是從同一個無法迴避的事實裡長出來的：**在會當機、會延遲、會丟包的網路上，你分不清沉默是慢還是死。**

正因為分不清，你只剩逾時這一個手段，而逾時只能告訴你「沒在時限內回應」、證明不了「死」。正因為任何一個逾時值都是賭注，你被迫在「誤判率」和「偵測延遲」之間選一個點，永遠拿不到兩全。正因為單次賭注必輸一頭，工程上才演化出心跳、動態逾時、phi 值、代探、suspicion——一層層把賭注下得更聰明，但沒有一層能跨過那個「完美偵測不存在」的天花板。也正因為跨不過去，最成熟的系統最後都不再執著於「不要誤判」，而是轉去**讓誤判無害**——用冪等兜住重做、用 fencing token 兜住換手，承認偵測會錯，但讓錯不致命。

所以下次你在凌晨三點盯著那塊變灰的方格，要決定一台機器是死是活時，記住你做的不是一個「查明真相」的判斷——真相在那道網路邊界後面，你**取不到**。你做的是一個**賭注**：選一個逾時、賭它死了、然後動手接管。而你之所以敢賭、敢按下那個 failover 按鈕，不是因為你確定它死了——你永遠不會確定——而是因為你早就備好了那兩道防線：就算它沒死、稍後醒來，它陳舊的寫入也進不了下游。失敗偵測不是一門「如何知道它死了」的學問，它是一門「**在你永遠無法確定它死了的前提下，依然能安全地往前走**」的學問。
