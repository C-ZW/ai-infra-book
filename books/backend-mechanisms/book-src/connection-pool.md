# 連線池：為什麼不每次都重開連線

一個剛上線的服務，程式碼讀起來無懈可擊：每次要查資料庫，就開一條連線、跑查詢、關掉連線。乾淨、對稱、沒有狀態殘留——課本會給滿分。然後流量上來了，你開始看到兩種症狀同時發作：一邊是請求延遲莫名其妙地高，一條 2 毫秒就能跑完的查詢，端到端要花掉 30 毫秒；另一邊是資料庫那頭開始拒絕新連線，log 裡冒出 `FATAL: sorry, too many clients already`。兩個症狀看似無關，其實是同一個決定的兩張臉：**你把「建立連線」這個昂貴又有總量上限的動作，放進了每一個請求的熱路徑。**

連線池要解的，就是這件事。它的核心主張一句話講得完——**連線很貴，但連線可以重用**——可是真正把它講透，要回答的問題遠不只「快取幾條連線」這麼簡單：池子滿了該等還是該拒？借出去的連線怎麼確定還活著？池該開多大，憑什麼？這些問題的答案，每一個都是一個取捨，而且每一個取捨選錯，都會在某個壓力點上以一種難以一眼看穿的方式爆掉。

## 建一條連線，到底貴在哪

先把「貴」這個字拆開。一條到資料庫的新連線，從零到能跑第一句查詢，要依序走過三段獨立的成本：

第一段是 **TCP 三向交握**。client 送 SYN、server 回 SYN-ACK、client 再回 ACK——這是一個完整的來回（1-RTT）才能讓雙方都確認連線可用。在同一個資料中心內，RTT 可能只有零點幾毫秒；但只要跨可用區、跨區域，這個 RTT 動輒十幾、幾十毫秒，而它是雷打不動的物理下限，光速擺在那裡。

第二段是 **TLS 交握**。如果連線加密（對外服務幾乎都該加密），TLS 1.3 把交握壓到了 1-RTT（相對 TLS 1.2 的 2-RTT 是一大進步），但那仍是**在 TCP 交握之上再疊一個來回**。所以一條冷的加密連線，光是「把管子接通」就要兩個 RTT，外加雙方算共享金鑰的 CPU 成本。

第三段是**應用層的認證與初始化**。對資料庫而言，這意味著傳帳號密碼（或走 SCRAM 之類的挑戰-回應，本身又是幾個來回）、server 在後端 fork 出一個專屬的行程或執行緒、配置該連線的記憶體、設定 session 參數。PostgreSQL 為每條連線開一個獨立的後端行程，這一步在 server 端的開銷尤其實在——不只是時間，還有記憶體與排程成本。

把這三段加起來，「開一條連線」的固定成本，從幾毫秒（同機房、不加密）到上百毫秒（跨區、加密、認證來回多）都有可能。關鍵在於：**這筆成本和你接下來那句查詢要跑多久，幾乎沒有關係。** 一句 `SELECT` 跑 2 毫秒，前面卻墊了 30 毫秒去接管子——這 30 毫秒不是在做你要的工作，是純粹的協定稅。而連線一旦建好，它**可以被下一個請求原封不動地接著用**：管子接通了就接通了，沒有理由用完一次就拆掉重接。連線池存在的全部理由，就是把這筆一次性的固定成本,從「每個請求付一次」攤平成「整個服務生命週期付幾十次」。

順帶看清一個常被忽略的對稱事實：HTTP/1.1 的 keep-alive、HTTP/2 與 gRPC 的單連線多工，本質上是**同一個想法在不同協定層的化身**——都是「別重開連線、在既有連線上接著做事」。連線池是這個想法在「呼叫端主動管理一組可借用連線」這個形態上的版本。

## 池子的四個動作

把連線池想成一個小小的櫃台，櫃台後面放著一排已經接通、隨時可借的連線。它對外只做四件事，但每一件事的邊界條件都藏著故事。

**borrow（借）**：請求要連線時，向櫃台借一條。理想情況下櫃台有閒置連線，直接遞給你，零成本。沒有閒置的呢？這裡分岔出第一個關鍵決定——如果池子還沒到上限，就現開一條（付一次建立成本）給你；如果已經到上限，櫃台**沒有連線可給**，這時它能做的只有兩件事：讓你**等**（阻塞，直到有人歸還或等到逾時），或當場**拒絕**你。這個「等還是拒」的選擇，後面會看到它其實是一個背壓決策。

**return（還）**：用完把連線還回櫃台——注意是**還**，不是**關**。連線回到閒置清單，等下一個人借。這一步聽起來最不起眼，卻是整個機制最容易出人命的地方：只要有一條「借了不還」的路徑，池子就會慢慢失血。

**validate（驗活）**：在借出或歸還的瞬間，櫃台可以選擇先確認這條連線「還活著」——通常是一句極輕量的探測（如 `SELECT 1`，或更現代的驅動層 ping）。為什麼需要？因為連線可能在閒置躺在櫃台裡的期間，被對端或中間的網路設備**悄悄掐死了**，而櫃台對此一無所知。

**evict（汰除）**：閒置太久的連線會被主動關掉回收。一條連線白白佔著資源卻沒人用，不只浪費，還可能正在慢慢腐爛（見下文死連線）。汰除讓池子在低載時縮回合理大小。

這四個動作裡，borrow 與 validate 的細節決定了池子在壓力下的行為，而 return 與 evict 的疏漏決定了池子怎麼在某個深夜悄悄壞掉。我們一個一個看。

## 池滿了：等、拒、還是往上游推

池子有上限，這是它的本質,不是缺陷。上限的意義是：**它是你對下游施加的並發閘門**——同一時刻最多只有這麼多條連線在跟資料庫講話。所以當請求多到把池借空、又有人來借時，發生什麼，是一個必須**刻意設計**的決定,不該交給預設值替你決定。

三條路：

**阻塞等待，直到逾時。** 借不到就排隊等,賭很快有人歸還。好處是能吸收短暫的尖峰——一個 100 毫秒的突刺,讓幾個請求多等十幾毫秒就過去了,使用者甚至沒感覺。壞處是,**等待時間會原封不動地算進請求延遲**:如果尖峰不是短暫的而是持續的,排隊會越積越長,借用逾時設得越寬,這條隊就拖得越久,最後變成所有請求都在「等連線」上耗掉大半時間。等待是把過載**藏起來**,藏到某個點藏不住為止。

**立刻失敗（fail-fast）。** 借不到就當場拋錯。好處是過載**立刻可見**,不會堆積成隱形的延遲債——系統寧可現在就掉一部分請求,也不要讓所有請求的延遲一起爆炸。壞處是尖峰期錯誤率直接上升,需要呼叫端配合重試(而重試本身又是一門學問,做不好會變成壓垮系統的重試風暴,見〈重試、退避與 jitter〉)。

**當成背壓往上游拒絕。** 這是把「池滿」這個訊號,當成一個明確的「我吃不下了」往上傳遞,讓更上游(負載均衡、閘道、甚至客戶端)去吸收或限流。連線池其實是系統裡**最天然的一個背壓觀測點**——池的等待佇列長度,直接反映了「需求超過下游處理能力」的程度(背壓的完整機制見〈背壓〉)。

這三條沒有絕對的對錯,差別在你想在哪裡承接過載。一個務實的組合常是:設一個**不太長**的借用逾時(比如幾百毫秒到一兩秒),讓短尖峰能靠等待平滑掉,但超過就 fail-fast,不讓延遲無限膨脹。逾時設多長,本質上是在回答「我願意為了不掉請求,讓使用者多等多久」——這是個業務決定,不是技術預設。

## 池該開多大:反直覺的那條曲線

「池開多大」是連線池最重要、也最違反直覺的一個參數。直覺會說:連線越多,能同時處理的請求越多,吞吐越高,所以池越大越好。**這個直覺是錯的,而且錯得很危險。**

先把直覺錯在哪講清楚。連線數不是憑空的並發度,每一條到資料庫的活躍連線,背後對應的是資料庫**真的要去服務的一個工作單位**——它要佔一個行程或執行緒、要爭搶 CPU、要競爭鎖、要分一塊記憶體。一台機器的 CPU 核心數是固定的,同一時刻真正能平行跑的查詢數,被核心數卡住。當活躍連線遠多於核心數,資料庫並沒有「同時做更多事」,它只是在大量連線之間**瘋狂切換上下文**——而每一次上下文切換都是純損耗,不產出任何有用工作。連線堆得越高,切換的損耗越大,有效吞吐**不升反降**。

這就是為什麼會有那條著名的、反直覺的建議:**你要的是一個小池,後面擠著一堆等著借連線的執行緒**,而不是一個大池讓所有請求都能立刻拿到連線去把資料庫淹掉。PostgreSQL 社群與 HikariCP 給出的經驗起點公式是:

```
connections = (core_count * 2) + effective_spindle_count
```

`core_count` 不含超執行緒的邏輯核(用實體核);`effective_spindle_count` 大致是「磁碟能同時服務多少個並行 I/O」——SSD 約算 1,傳統機械硬碟陣列則接近實際碟片數,而若資料集完全在記憶體裡命中、不碰碟,這項趨近於 0。一台 4 實體核、單 SSD 的資料庫,照公式約是 `(4*2)+1 = 9`,湊個整數叫它 10。**一台正經的資料庫伺服器,最佳連線數常常是個位數到幾十,不是幾百幾千。** 這個數字小到讓多數人第一次看到時不敢相信。

那道理的證據有多強?HikariCP 的文件引了 Oracle 的一段壓測示範:把連線池從 **2048** 條砍到 **96** 條,回應時間從約 **100 毫秒**降到約 **2 毫秒**——足足 **50 倍**的改善,而且是**砍**連線、不是加。背後正是上面那條曲線:2048 條連線把資料庫推進了上下文切換的泥沼,96 條讓每條連線都能順暢拿到 CPU 做事。

所以真正的判準翻轉了過來:**池的上限,應該由「下游能舒服承受多少並發」回推,而不是由「我有多少請求想同時打進去」決定。** 如果你的請求並發遠超過這個健康的連線數,正確的做法是讓多出來的請求在池外**排隊等待**(被那個小池節流),而不是把池開大、把過載直接灌進資料庫。連線池在這裡的角色,與其說是「加速器」,不如說是**保護資料庫不被自己人打死的限流閥**。

## 一筆帳:無池、好池、壞池

把上面的道理落到一筆能手算的帳上。假設:建立一條資料庫連線(含交握與認證)約 **30 毫秒**;連線建好後,一句業務查詢約 **2 毫秒**。

**無池**——每個請求現開現關:每請求成本 = 30 + 2 = **32 毫秒**,其中 **94%** 純粹花在接管子上,真正做事只佔 6%。在 2000 QPS 下,你每秒還要付 2000 次建立成本,server 端光是 fork 後端行程就先被拖垮。這是開頭那個「乾淨對稱」寫法的真實代價。

**好池**——固定 20 條的池,穩態 2000 QPS:一條連線跑一句 2 毫秒的查詢,每秒理論上能服務 `1/0.002 = 500` 次;20 條的理論吞吐上限是 `20 * 500 = 10,000 QPS`。實際只跑 2000 QPS,池利用率約 `2000/10000 = 20%`——大量時間池裡都有閒置連線,借用幾乎零等待,而那 30 毫秒的建立成本只在服務啟動、暖機那幾十條連線時付過一次,之後攤平到趨近於零。每請求的有效延遲回到接近查詢本身的 2 毫秒。

**壞池**——同樣 2000 QPS,但有人「為了省資源」把 max 砍到 4 條:理論上限變成 `4 * 500 = 2000 QPS`,**恰好等於**到來的負載。這正是排隊理論裡最凶險的位置:當利用率 ρ 逼近 1,排隊延遲不是線性上升,而是**指數爆炸**(這個直覺見〈排隊的直覺〉)。一點點流量抖動,或一條查詢偶爾慢了一下,等待佇列就瞬間堆起來,借用等待時間從零暴衝到幾百毫秒甚至逾時。表面上「資源用好用滿、利用率 100%」聽起來很美,實際上你把系統頂在了懸崖邊上。

這筆帳的三種結局,把連線池的整個取捨空間畫了出來:無池是把固定成本付進每個請求;好池讓你在「夠用的並發」與「不打爆下游」之間留了餘量;壞池則是把池當成省錢工具、砍到剛好夠用——而「剛好夠用」在排隊的世界裡等於「隨時會爆」。

## 連線洩漏:借了不還的慢性失血

連線池最常見的生產事故,不是池太小或太大,而是**連線洩漏(connection leak)**:某條程式路徑借了連線卻沒還回去。

機制上它為什麼致命,值得在腦中重放一遍。池有 N 條連線。每洩漏一條,可借的就少一條,而且**永遠**少一條——洩漏的連線既沒被歸還、也沒被關閉回收,它就那樣懸在「已借出」的狀態裡,直到行程重啟。所以洩漏是**單調累積**的:跑一陣子,N 條一條條漏光,最後池空了,所有新的 borrow 全部卡在等待、然後集體逾時。症狀很有特徵——**服務剛起來好好的,跑了幾小時或幾天後,突然所有請求一起卡死或逾時**,重啟就好、過陣子再犯。這個「健康一段時間後集體猝死」的模式,幾乎是連線洩漏的簽名。

洩漏從哪來?幾乎總是**例外路徑**。正常路徑大家都記得 `close`,但一旦中間拋了例外、或提早 `return`、或某個分支忘了走到釋放那行,連線就漏了。所以防禦的核心是**讓歸還不依賴程式設計師記得**——用語言提供的資源綁定機制把它變成自動的:Java 的 try-with-resources、Go 的 `defer conn.Close()`、Python 的 `with` context manager、Rust 的 RAII drop。它們的共同點是:**無論正常返回還是拋例外,離開作用域時一定執行歸還。** 把這件事交給編譯器或執行期保證,而不是交給人的紀律。

```
# 反例:例外路徑漏掉歸還
conn = pool.borrow()
result = conn.query(sql)   # 若這裡拋例外,下面那行永遠不執行 -> 洩漏
pool.return(conn)

# 正確:用作用域綁定,離開即還(示意)
with pool.borrow() as conn:     # __exit__ 保證歸還,即使 query 拋例外
    result = conn.query(sql)
```

成熟的池實作通常還提供一張安全網:**洩漏偵測**——如果一條連線被借走超過某個門檻時間(比如幾十秒)還沒還,就在 log 裡印出當初借它的那段呼叫堆疊。這不修復洩漏,但它把一個「幾天後才爆、爆了也不知道哪漏的」謎題,變成一條指著兇手的線索。對一個會在 3am 把你叫醒的故障來說,「能不能在不重現的情況下查出根因」往往比「能不能自動修復」更值錢。

## 死連線:櫃台以為活著的那條

第二個藏得很深的故障:池裡的一條連線,**在櫃台看來是閒置可借的,但它其實早已斷了**。

怎麼會這樣?連線閒置躺在池裡時,沒有任何流量經過它。而 TCP 連線是個雙方各自維護的狀態——如果中間有東西**單方面**把它拆了,而沒有(或沒能)通知到你這端,你的櫃台就會一直以為它還在。誰會幹這種事?太多了:雲環境裡的 NAT 閘道、負載均衡器、防火牆,普遍會**主動掐掉閒置超過 N 分鐘的連線**以回收自己的狀態表;資料庫自己也有 `wait_timeout`(MySQL)之類的設定,會關掉閒太久的後端;網路抖動、對端重啟,都可能讓一條連線變成單向的殭屍。

於是經典的炸法是這樣:一條連線閒置了十分鐘,雲端的 NAT 在第五分鐘就悄悄把它掐了,但你的池毫不知情。下一個請求來借,櫃台高高興興遞出這條「閒置」連線,查詢一發出去——`connection reset by peer`,或者更糟,卡住直到 TCP 逾時(可能是幾十秒)才報錯。使用者收到一個莫名其妙的失敗,而那條查詢本身一點問題都沒有。

防禦有兩個層次,要一起用:

第一,**借出前驗活**:在把連線遞給請求前,先跑一個極輕量的探測,確認它還能說話。代價是每次借用多一個來回(現代驅動有更省的協定層 ping,成本很低),換來「遞出去的一定是活的」。

第二,也是更根本的,**讓池的時間設定比下游的掐殺時間更短**。這是一條鐵律,但方向很容易記反:

- 池的**閒置汰除(idle timeout)**要比「下游掐閒置連線的時間」**更短**——這樣連線在被外部掐死之前,池自己就先把它汰掉重建了,根本不給它變殭屍的機會。
- 池的**最大壽命(max lifetime)**要比「資料庫/雲端強制關閉連線的硬上限」**更短**——主動在到期前淘汰並換新,而不是等對端某天突然把它關掉、讓你在下次借用時踩雷。
- 另外可以開**保活探測(keepalive)**:對閒置連線週期性地送一個極小的 ping,讓中間設備認為這條連線「還在用」、不要掐它;這個間隔要設得比「下游中間設備掐閒置連線的時間」短才有意義——讓 ping 在設備的閒置計時器歸零之前先把它刷新(注意參照物是中間設備的閒置窗口,不是池的最大壽命:後者是另一條管強制汰換的獨立計時器)。

換句話說,死連線的防禦哲學是:**永遠不要讓「外部設備」成為第一個關掉你連線的人。** 你要搶在它前面,主動地、有計畫地汰換,把「對端隨機掐你」這個不可控的故障,轉成「我自己按時換新」這個可控的維護。

## 上限的乘法陷阱:當你有很多個你

到目前為止,我們把連線池當成一個池在談。但生產系統幾乎從不只有一個。你水平擴展,跑了 K 個應用實例,**每個實例都有自己的一個連線池**,而它們**共享同一個資料庫**。這時一個極容易被忽略、又極容易致命的算術出現了:**資料庫看到的總連線數,是所有實例的池上限相加。**

具體一點。資料庫 `max_connections = 100`(這正是 PostgreSQL 的預設值;其中預設還保留 3 條給超級使用者,實際留給應用的約 97 條)。你的應用實例,每個把池 max 設成 20——單看一個實例,完全合理。但當你擴到第 6 個實例:`6 * 20 = 120 > 100`。第 6 個實例的池在嘗試把連線開滿時,會撞上資料庫的硬上限,拿到 `too many clients already`——而這個錯誤**不會發生在你壓測單一實例時**,只在「實例數 × 每池上限」越過資料庫總額度的那一刻,在生產的某次自動擴容後,毫無預警地出現。更陰險的是,擴容本來是為了抗住更大流量,結果擴著擴著反而把資料庫的連線額度撐爆——你越想活下去,死得越快。

所以連線池的容量規劃,必須算**總帳**,而且要留餘量:

```
實例數 × 每實例池上限  +  維運/遷移/監控等帶外連線  ≤  資料庫 max_connections
```

那留給每個實例的額度,常常小到反直覺——如果你預期擴到 10 個實例、資料庫 `max_connections=100`、再留 10 條給維運,那每個實例的池上限只剩 `(100-10)/10 = 9` 條。又是個位數。而前面那條「小池反而更快」的曲線告訴你:這其實往往**正好夠用**,甚至是健康的。問題只在於,當實例數會自動伸縮、上限不固定時,這個乘法就特別難算對。

這也正是 serverless 把連線池逼到牆角的原因。serverless 函式的實例數隨流量自動暴漲,每個實例又各自開池——`實例數` 這一項變成了一個你不太能控制、可能瞬間衝到幾百上千的數字,乘上每池上限,輕鬆把任何資料庫的連線額度碾碎。傳統的「每實例一個池」模型在這裡直接失效。業界的解法是再加一層**外部連線池中介(external pooler)**:讓所有函式實例不直接連資料庫,而是連到一個中間的 pooler(如 PgBouncer),由它維護一個**收斂後**的、對資料庫的小連線池,把成百上千的客戶端連線多工到資料庫的幾十條真實連線上(2026-06)。

但這層中介不是免費的,它買到了「收斂連線數」,卻欠下一筆**狀態**的債——而這筆債,正好把我們帶到連線最容易被誤解的本質。

## 連線不是無狀態的水管

關於連線池,最深的一個誤解是把連線想成一根無狀態的水管:借哪一條都一樣,反正都是通到同一個資料庫。**大多數時候對,但在幾個關鍵地方,致命地錯。**

一條資料庫連線,在它的整個生命裡,是有 **session 狀態**的——它記得這條連線上設過的 session 變數、開著的交易、宣告的暫存表、`PREPARE` 過的預備語句、持有的 session 級 advisory lock。這些狀態**綁在這條特定連線上**,不會跟著你「歸還又借出」而轉移到另一條連線。

平常這不出問題,因為「借一條、在它上面跑完一個完整的工作單位、再還回去」這個模式,讓每個工作單位都待在同一條連線上、自始至終。但一旦中間插進一個會**打散連線歸屬**的東西,事情就裂開了。前面那個 serverless 的外部 pooler 正是最典型的例子——當 PgBouncer 跑在**交易級(transaction)pooling 模式**下,它在每個交易**結束**時就把後端連線收回、給下一個客戶端用。這意味著你這個客戶端的兩個相鄰交易,很可能跑在**兩條不同的後端連線**上。

於是依賴「跨交易的連線狀態」的東西全部失靈:在連線 A 上 `PREPARE` 的語句,下一個交易換到連線 B 上 `EXECUTE`,在未啟用 pooler 端追蹤時會找不到那個語句、報錯;session 級的設定、暫存表、advisory lock,統統不跨交易保留。這不是 pooler 的 bug,而是它的收斂能力的**直接代價**——它能把上千客戶端多工到幾十條連線,正是因為它不讓任何客戶端「霸佔」一條連線;而不霸佔,就意味著連線上的 session 狀態無法被你信賴。出路有三:把 pooling 模式退回 session 級(犧牲收斂效果);在驅動層關掉 server 端預備語句、改用別的方式;或(自 PgBouncer 1.21、2023-10 起)把 pooler 的 `max_prepared_statements` 設為非零,讓它在後端連線上代為追蹤並即時重建協定層的預備語句——但這只救得了協定層的 prepared statement(JDBC `PreparedStatement`、libpq `PQprepare` 那種),救不了以純文字送出的 SQL `PREPARE name AS …`,而 advisory lock、暫存表、跨交易的 `SET` 這些 session 狀態仍然不跨交易保留。這是個必須當面承認、不能假裝不存在的取捨。

這件事的教訓比 PgBouncer 本身大:**連線池能成立的前提,是「連線可互換」這個假設成立。** 在絕大多數無狀態的查詢上,它成立得很好,於是池子可以自由地把任何一條閒置連線遞給任何請求。但凡你在連線上建立了**跨借用週期的狀態**,你就在偷偷違反這個假設,而池子並不會阻止你——它會照常把連線換來換去,直到某天那個狀態落在了錯的連線上,給你一個極難 debug 的錯誤。理解連線池,某種程度上就是理解「連線在什麼條件下可以被當成無狀態的、什麼條件下不行」這條界線劃在哪。

## 為什麼是這個形狀

退一步看,連線池的整個樣貌,是被兩個無法迴避的事實夾出來的:**連線建立很貴,而連線總量有硬上限。**

正因為建立貴,所以要預先建好、重複借用,把一次性成本攤平——這給了它「池」的基本形態。正因為總量有上限,所以池必須有 max、必須在借空時做出「等或拒」的決定、必須在多實例相乘時算總帳、必須在 serverless 下靠外部中介收斂——這些全都是「有限資源該怎麼分配」這個老問題的具體答案。正因為連線是會被外部悄悄掐死、又攜帶 session 狀態的活物,所以池不能只是個被動的快取,它得**主動驗活、按時汰換**,還得清楚知道自己手上的連線在什麼條件下可以當成無狀態的水管、什麼條件下不行。

而那條最反直覺的曲線——小池往往比大池快——把這一切收束成一句設計哲學:連線池不是一個讓你「儘量多並發」的加速器,它是一個**節流閥**。它最重要的工作,不是讓更多請求衝進資料庫,而是攔住多餘的請求、把對下游的並發壓在資料庫真正能舒服承受的那個小數字上,讓多出來的負載在池外排隊等待,而不是進去把資料庫淹死。下次你看到一個服務在高峰期請求都卡在「等連線」、而資料庫 CPU 卻沒滿,你會知道那未必是池太小——很可能正是那個小池在盡職地擋著,寧可讓請求排隊,也不讓資料庫被自己人打垮。
