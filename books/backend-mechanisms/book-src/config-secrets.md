# config 與機密管理

凌晨兩點，值班的人盯著告警：付款服務全面回 500，但程式碼一個字沒動，部署 pipeline 也乾乾淨淨。翻 log 才看出來——有人為了讓 staging 連上一份新的測試資料庫，順手在生產環境的 console 上改了一個環境變數，把資料庫連線字串裡的 host 改錯了。一行純文字，沒進版控、沒走 review、沒人知道誰改的、也沒辦法乾淨地回滾，因為「上一個值是什麼」根本沒被記錄在任何地方。

這就是 config 這個看似最無聊的主題底下藏的張力：它是**程式之外、卻能左右程式行為的那一層輸入**。一個服務的程式碼可以是 immutable 的、可以被測試到滴水不漏、可以一個位元組都不差地在 dev 與 prod 跑同一份 image——然後被一個外部注入的字串整個改變命運。config 管的就是這層輸入：哪些值該從程式碼裡抽出來、用什麼通道遞給跑著的行程、能不能在不重啟的前提下換掉、換錯了怎麼追怎麼退。

而在這層輸入裡，有一小撮值的性質截然不同：資料庫密碼、API key、私鑰、簽章用的對稱秘密。它們和 log level、超時毫秒數一樣是「程式之外的輸入」，但洩漏的後果天差地別——一個錯誤的超時值頂多讓請求慢一點，一把外洩的資料庫密碼是整個系統被打穿。把這兩類東西當成同一個問題、用同一套機制處理，是這個領域最常見、也最致命的設計錯誤。所以這一章其實在講兩個問題：**怎麼把會變的輸入正確地餵進系統**，以及**怎麼在「機密遲早會洩漏」的前提下，把洩漏的爆炸半徑與善後成本壓到最小**。

## 為什麼 config 必須離開程式碼

先把第一個問題講難。最天真的做法是把這些值直接寫進程式碼：連線字串、超時、第三方服務的 URL，全都硬編在原始檔裡。這在單機、單環境、一個人寫的時候完全沒問題——直到你需要第二份部署。

同一份程式碼要跑在 dev、staging、production 三個環境，連的是三套不同的資料庫、三組不同的下游 URL、三種不同的 log 詳細程度。如果環境差異寫死在程式碼裡，你就得為每個環境維護一個分支，或在程式碼裡塞滿 `if (env === "prod")` 的判斷——而那一刻，「程式碼」與「這份程式碼跑在哪」這兩件本該正交的事，被焊死在了一起。每次改環境設定都要改程式碼、重新建置、重新走完整個發布流程；每個環境跑的其實是**不同的 build**，dev/prod parity（環境一致性）就此破功。

把 config 抽離程式碼，買到的正是「**同一份建置產物，跑在任何環境**」。CI 建一次 image，dev 與 prod 跑的是同一份位元組，差別只在啟動時被餵進的那組外部值。12-factor 方法論的第 III 條給了一個漂亮的判準：一份應用的 config 有沒有正確地與程式碼分離，看「**這份程式碼能不能在任何時刻被開源，而不洩漏任何 credential**」（這個 litmus test 出自 12factor.net）。如果把 repo 設成公開就會洩密，代表有不該在程式碼裡的東西混在了程式碼裡。

但要小心 12-factor 那條原則被過度引申。它的核心主張是「config 與程式碼分離」——這條恆久正確；它順手給的實作建議是「存進環境變數」——這條在 2011 年的 PaaS 脈絡下合理，但在後來的容器與 Kubernetes 時代不該被當成鐵律。原則與實作要分開記：**分離是目標，環境變數只是其中一種注入手段，而且對機密來說是相當漏的一種**。後面會看到為什麼。

## 把 config 遞進行程的三種通道

值已經在程式碼之外了，下一個問題是：用什麼通道把它遞給跑著的行程？三種主流做法，各自買到不同的東西。

**環境變數**是最簡單的：啟動行程時把 `DATABASE_URL`、`LOG_LEVEL` 設進環境，程式從 `process.env` 之類的介面讀。語言無關、零相依、人人看得懂。它的弱點有兩個面向：一是不適合大型或有結構的 config（一坨巢狀的 JSON 硬塞進一個環境變數很難看也難維護）；二是對機密有實打實的洩漏面，這留到下一節細講。

**掛載的設定檔**把 config 以檔案的形式掛進容器——在 Kubernetes 裡就是把一個 ConfigMap 以 volume 的形式掛到 pod 的某個路徑。它適合量大、有結構的 config，而且開了一扇環境變數沒有的門：**不重啟就換值**。環境變數在行程啟動那一刻就被凍進行程的記憶體，之後改不了；掛載的檔案則可以在行程運行中被更新，只要應用願意 watch 那個檔案、偵測到變動就重讀。但這裡有個容易踩的時間差陷阱：Kubernetes 更新掛載的 ConfigMap **不是即時的**——kubelet 週期性地同步，從你改完 ConfigMap 到 pod 裡的檔案真的變掉，有一段傳播延遲：最壞情況約等於 kubelet 同步週期（預設約 1 分鐘）加上 ConfigMap 快取 TTL（預設約 1 分鐘），合計可達約兩分鐘（且用 `subPath` 掛載的檔案根本不會自動更新）。「我改了 ConfigMap，為什麼服務還是舊行為」十之八九是卡在這個延遲，而不是設定沒生效。

**集中式 config 服務**走得更遠：應用啟動時、甚至運行中，向一個中央系統拉取設定。這買到動態更新、版本控制、按比例放量（同一個 feature flag 先對 5% 的流量開）這些環境變數與檔案都給不了的能力，代價是引入一個新的相依——而且是一個在啟動路徑上的相依。這就帶出一個非顯然的故障模式：**如果應用啟動時必須先連上 config 服務才能起來，那 config 服務就成了所有應用的單點故障**。config 服務一抖，整個機隊起不來；更糟的是，這常發生在你最需要把機隊重啟的時候（比如一場正在擴大的事故裡），於是 config 服務的故障與應用的故障耦合成一個彼此放大的死結。成熟的做法是讓應用**快取上一份成功拉到的 config**、在 config 服務不可用時用快取值照常啟動，把這個相依從「硬」變「軟」。

這三種通道不互斥，真實系統常疊著用：環境變數放少量啟動關鍵值、掛載檔案放大塊靜態 config、集中服務放需要動態調整的 feature flag。選哪個的判準不是哪個先進，而是「這個值需要多動態、結構多複雜、洩漏了多嚴重」。

## 機密是另一個問題，不是更敏感的 config

現在轉到第二個、也是更難的問題。一個誘人但錯誤的心智模型是把機密看成「比較敏感的 config」——既然都是程式之外的輸入，那用同一套機制、調高一點權限不就好了？

不行，因為兩者的**目標函數根本不同**。一般 config 的目標是「正確注入」——把對的值在對的環境餵給對的行程，事情就成了。機密的目標完全是另一回事：你必須假設**這個機密遲早會洩漏**——被某個粗心的 log dump 帶出去、被某個有權限的人複製到一個不該複製的地方、被某次資料庫外洩撈走——然後在這個悲觀假設下，把「洩漏後爆炸半徑有多大、善後要動多少地方、能不能查出是誰在何時拿走的」這幾件事的成本壓到最小。

換句話說，一般 config 問的是「怎麼把值送對」，機密問的是「值送漏了之後，傷害能控制在多小」。用 config 的安全等級去保護機密，等於只回答了前一個問題，把後一個問題整個跳過。理解了這個分野，後面所有的機制——加密儲存、最小權限、輪替、稽核、短命動態憑證——就都不是「進階選配」，而是這個更難的目標逼出來的必要條件。

具體看環境變數為什麼對機密特別漏。把一個資料庫密碼塞進環境變數，它就以明文活在行程的環境塊裡，而環境塊有好幾條安全洩漏的縫：

- **子行程繼承**：環境變數預設會被 `fork`/`exec` 出來的子行程整碗繼承。你的應用呼叫一個外部工具、跑一段 build script、起一個 sidecar，這些子行程全都自動拿到了父行程的所有機密——遠遠超出它們該有的權限。
- **`/proc/<pid>/environ`**：在 Linux 上，同一個使用者（或有 ptrace 能力者）可以直接讀 `/proc/<pid>/environ` 看到任一行程的完整環境變數（這道權限不是無條件的——目標若是 setuid／non-dumpable 行程，或啟用了 Yama `ptrace_scope`，同 UID 也可能讀不到；這正是「為什麼有時讀得到、有時讀不到」的原因）。在 Kubernetes 裡，一個 `kubectl exec` 進到 pod 的人預設就有這個能力。
- **崩潰與除錯的 dump**：很多框架在啟動時或拋例外時會把整包環境變數印進 log 方便除錯——機密就這樣進了 log，再被 log 系統長期保存、被一整批有 log 存取權的人看到，而且通常沒人意識到那裡躺著一把密碼。

把機密改成**以檔案掛載**（而且掛在 tmpfs 這種不落盤的記憶體檔案系統上）能堵掉其中幾條縫：檔案不會被子行程自動繼承、不出現在 `/proc/<pid>/environ`、不會被「印出整包 env」的除錯習慣順手帶走。代價是應用得改成從檔案讀機密、並在輪替時能重讀。這是一個小但實在的攻擊面縮減，也是為什麼「機密一律用環境變數」這條從 12-factor 慣性延續下來的習慣，在認真對待安全的系統裡正被「掛載檔案」取代。

## 為什麼 Kubernetes Secret 不等於「機密被保護了」

Kubernetes 提供了一個叫 Secret 的物件，把機密和一般 config（ConfigMap）在 API 層分了開來。很多人看到「我用了 Secret 物件」就鬆一口氣，以為機密被加密保護了。這是一個危險的誤解，值得把底下發生的事看清楚。

Kubernetes Secret 的值預設**只是 base64 編碼，不是加密**。base64 是個可逆的編碼——任何人 `echo ... | base64 -d` 一下就還原了，它存在的理由是讓二進位資料能安全地塞進 YAML，而不是保護任何東西。Secret 最終存進叢集的後端儲存 etcd；如果 etcd 沒有額外開啟 encryption-at-rest，那麼 etcd 裡躺的就是 base64 過的明文。任何能直接讀到 etcd 資料的人——拿到 etcd 備份、攻進 etcd 節點、或在某些設定下有足夠 RBAC 權限的人——就能讀出叢集裡的每一把機密（Kubernetes 官方文件對這點寫得毫不含糊）。

要讓 Secret 在 etcd 裡真的被加密，得明確設定 API server 的 `EncryptionConfiguration`，讓 API server 在寫進 etcd 前加密、讀出時解密。但這裡有個容易自我安慰的細節：最簡單的 provider 是用一把存在本機檔案裡的本地 AES 金鑰——這擋得住「etcd 被偷」，卻擋不住「host 被攻破」，因為那把金鑰就明文躺在 host 上的 `EncryptionConfiguration` 檔案裡，攻進 host 的人連 etcd 都不用碰，直接撈金鑰即可。要把這個缺口補上，得用 KMS provider 把加密金鑰本身放到叢集之外的金鑰管理服務裡，讓 host 上永遠不存在能解密一切的那把總鑰。

所以「我用了 Secret」這句話本身不提供任何加密保證。它買到的只有一件事：**把機密和一般 config 在物件層級分了開來**，於是你可以對 Secret 設更嚴的 RBAC、不把它和 log level 混在同一個 ConfigMap 裡給同一群人讀。加密、最小權限、輪替這些真正的防護，得在 Secret 之外另外搭。把 Secret 物件的存在誤當成防護到位，是這個領域最普遍的盲點。

順帶提一個架構選擇：因為原生 Secret 終究要落進 etcd，很多團隊乾脆讓機密**不進 etcd**，改用兩類外部整合。一類是 External Secrets Operator（ESO）這種「同步器」：它去外部的機密管理器拉值、在叢集裡產生標準的 Secret 物件，好處是相容一切吃 Secret 的東西、易於除錯，代價是機密仍然落進了 etcd（只是來源變成外部單一真相）。另一類是 Secrets Store CSI Driver：它把機密直接以 volume 掛進 pod、繞過 etcd，攻擊面更小，代價是和 pod 生命週期綁得更緊、相容性與除錯較麻煩。兩者解的是同一個焦慮——別讓長命機密大量沉澱在叢集裡——只是在「相容性」與「攻擊面」之間選了不同的點（2026-06）。

## 一把密碼用三年的代價

把上面的原則收束成一條可操作的尺：衡量一個系統機密管理成熟度的，不是它用了哪個工具，而是「**輪替一次機密，要動幾個地方、要不要停機**」。把這把尺架在兩種極端上，差距會自己跳出來。

設想最差的形態：同一個資料庫密碼，硬編在 5 個服務的環境變數裡，三年沒換過。某天安全團隊懷疑它外洩了，要求立刻輪替。你得做的事是這樣一串：改掉資料庫上的密碼、更新這 5 個服務各自的環境變數設定、依序重新部署這 5 個服務——而且這中間有一段沒法消除的空窗：從你在資料庫端改掉密碼的那一刻，到 5 個服務全部更新並重啟完成之間，任何還拿著舊密碼去連的服務都會被資料庫拒絕。這是一次需要小心編排的協調式停機，在凌晨進行，每一步都可能出錯。更糟的是，因為這把密碼三年來被複製到了十個你已經記不全的地方，你甚至不確定改完這 5 個就真的沒有漏網的——而且你**永遠無法知道它到底有沒有真的外洩、是誰在何時拿走的**，因為一個靜態長命機密的每一次使用都長得一模一樣，沒有任何痕跡可查。

現在看另一個極端：動態機密。每個服務啟動時，不是去讀一個寫死的密碼，而是向一個機密管理器（HashiCorp Vault 這類）**即時申請一份臨時資料庫憑證**，附帶一個租約（lease），TTL 比如一小時。機密管理器在資料庫上即時建一個短命使用者、把帳密發給服務，並記下「這份憑證是誰、在何時、為了什麼租走的」。租約到期後，機密管理器自動把那個資料庫使用者撤銷、憑證失效；服務在到期前自動續租、或重新申請一份新的（Vault 的租約 TTL 由你設定——常見的生產配置是 default TTL 1 小時、max TTL 24 小時；Vault 系統層的出廠預設其實是 768 小時／32 天，要短命憑證得自己在 mount 或 role 上調低。到期後 Vault 用自己的撤銷系統「在合理時間內」自動 revoke，並非保證在到期那一瞬間立刻失效——這是官方文件的措辭）。

在這個世界裡，「輪替」這個動作幾乎消失了：根本沒有一把長命密碼需要被人工輪替，每份憑證活一小時就自然死去。要緊急作廢一份憑證？撤銷它的租約，立刻失效，不必動任何服務的設定、不必重新部署、沒有協調空窗。要查「誰在用這個資料庫」？稽核日誌裡每一份租出去的憑證都有名有姓有時間。從「一把密碼用三年、散落十處、洩了你不知道、換它要協調式停機」，到「每份憑證一小時、集中發放、全程有記錄、撤銷即失效」——這兩端之間的距離，就是機密管理這件事的全部進步空間。

當然動態機密不是免費的：你引入了一個 Vault 之類的關鍵相依（它本身又是個必須高可用、否則所有服務拿不到憑證的單點），應用得改成「啟動時去拉、到期前續租」的模式，運維這套基礎設施有實打實的成本。對一個單人 side project、兩三個機密、無合規要求的場景，硬上一整套動態機密反而是過度工程——一個加密儲存、不進 repo、和一般 config 分離的設定來源就夠了。判準還是那條尺：你的「輪替一次要動幾個地方」如果答案是「一個地方、自動完成」，你就到位了，用什麼工具是其次。

## 輪替為什麼比它看起來難：一個四步交接

「輪替機密」聽起來像「把舊的換成新的」這麼簡單，但真正自動化它的時候，會撞上一個分散式系統的老問題：**在新舊憑證交接的那一瞬間，系統裡有些地方還拿著舊的、有些已經拿到新的，你不能讓任何一刻出現「沒有任何一份憑證是有效的」的空窗**，否則輪替本身就成了一次故障。把這個交接做對，需要的步驟比直覺多。

AWS Secrets Manager 的 Lambda 輪替函式把這件事拆成了**四個步驟**，是理解「安全交接」的好教材。它在內部用三個 staging label 標記同一個機密的不同版本：`AWSCURRENT`（當前生效的版本）、`AWSPENDING`（正在輪替、還沒生效的新版本）、`AWSPREVIOUS`（上一個已知良好的舊版本）。輪替時 Secrets Manager 用不同參數**多次**呼叫同一個 Lambda，依序走完這四步（步驟名與行為以 AWS 官方文件為準）：

```
createSecret  生成一份新的機密值，存成 AWSPENDING 版本。
              此刻 AWSCURRENT 仍是舊值、仍然有效，服務照常用它。
              新值放在 AWSPENDING 是為了「冪等」——若後面任一步失敗
              重試，能直接沿用這份已生成的新值，不會每次重試都生一把新的。
setSecret     去資料庫/服務端，把憑證真的改成 AWSPENDING 裡的新值。
              這是唯一真正動到外部資源的一步。
testSecret    用 AWSPENDING 的新憑證實際連一次資料庫，確認它真的能用。
              在把流量切過去之前先驗證，避免切到一把壞憑證。
finishSecret  把 AWSCURRENT 標籤從舊版本移到新版本（同一次 API 呼叫裡
              一併移除 AWSPENDING），舊版本自動被打上 AWSPREVIOUS。
              到這一步，新憑證才正式「生效」。
```

看清楚這個順序為什麼非如此不可。新值先靜靜地待在 `AWSPENDING` 裡、不碰 `AWSCURRENT`，所以在 `createSecret` 與 `setSecret` 之間，所有讀 `AWSCURRENT` 的服務拿到的還是穩穩有效的舊憑證——**沒有空窗**。`setSecret` 改完資料庫後、`finishSecret` 切標籤前，先用 `testSecret` 實連一次驗明正身，確保不會把生效標籤切到一把根本連不上的壞憑證。而整個過程用 `AWSPENDING` 來保證冪等：因為 Secrets Manager 在任一步失敗時會把整個輪替流程**重試多次**，新值必須只在第一次 `createSecret` 時生成一次、之後重試都沿用同一份，否則每次重試生一把新密碼，會在資料庫端留下一堆孤兒。

但這個設計也藏著一個讀者該預期的非顯然邊界：在 `setSecret`（資料庫密碼已改）與 `finishSecret`（生效標籤還沒切）之間，如果有任何服務**沒有在每次連線時去 Secrets Manager 重讀、而是把舊密碼快取在記憶體裡**，它手上的舊密碼此刻已經在資料庫端失效了，於是這段窗口它的連線會被拒。這正是「單一使用者輪替」策略的固有空窗。AWS 為此提供了**交替使用者（alternating users）**策略：維護兩個資料庫使用者，每次輪替只改其中閒著的那一個、改完才切過去，於是任何時刻都有一份當前有效的憑證可用、消除了上述空窗——代價是你的輪替週期得砍半、也就是輪替動作得跑得更勤（因為兩個使用者輪流，要達到「每 90 天每個使用者換一次」就得設成每 45 天輪一次；這個 45 天是從「每次只輪一個 user」推出的結果，不是 AWS 明列的數字），而且這個策略和 RDS Proxy 不相容（這條相容性限制是 AWS 官方明載的）。「為什麼輪替週期要砍半」這種非直覺的細節，正是把輪替當成「換個值」會完全想不到、卻會在生產裡真實咬你的地方。

## 為何是這個形狀

退一步看，這一章的兩半其實是同一個第一原理的兩面：**程式之外那層輸入，你既要它正確、又要在它落入壞人之手時傷害可控。**

對一般 config，「正確」是主旋律——把它從程式碼裡抽出來，換來同一份 build 跑遍所有環境；用環境變數、掛載檔案、集中服務裡合適的通道遞進去，按值的動態程度與結構複雜度選。它會壞的地方在交接的時間差：掛載的 ConfigMap 改了不是即時生效，集中式 config 服務會在你最需要重啟機隊時變成放大故障的單點。

對機密，主旋律換成了「洩漏可控」——因為你被迫假設它遲早會洩。於是每一個機制都是這個悲觀假設逼出來的：和一般 config 分離，是為了能對它上更嚴的權限；掛載而非環境變數，是為了堵掉子行程繼承與 `/proc` 那幾條縫；加密儲存，是為了讓拿到 etcd 備份的人撈不出明文；短命的動態憑證，是為了讓每把鑰匙活不過一小時、讓「輪替」這個痛苦的人工動作自然消失；稽核日誌，是為了在洩漏發生後至少答得出「誰、何時、拿了什麼」。一把用三年、散落十處、無聲無息的靜態密碼，之所以是最差的形態，正因為它在這每一個維度上都站在反面。

下次你在 console 上看到一個機密管理器發出一份一小時就過期的臨時憑證、或一個 Secret 在 etcd 裡是被 KMS 加密過的密文，你會知道那些麻煩不是儀式——它們是「機密遲早會洩」這個冷靜假設，被推到底之後，唯一還站得住的形狀。
