# REST 設計、版本化與 OpenAPI 契約

某天早上，一個第三方合作夥伴的工程師寄來一封怒氣沖沖的郵件：他們的整合昨晚開始大量報錯，而你們什麼都沒做——沒發版、沒改 schema，至少你以為沒有。翻 commit 紀錄才發現，前天有人在訂單狀態的 enum 裡加了一個新值 `partially_refunded`。對你們而言這是純粹的新增，舊功能一個字都沒動。但對方的 client 用了一個窮舉式的 `switch`，碰到沒見過的 enum 值就直接拋例外。一個你眼中「絕對安全」的變更，在防火牆的另一邊變成了破壞性變更。

這封郵件濃縮了 API 設計的全部難處。你寫的不只是程式碼，而是一份**契約**——一旦有人在你看不見的地方依賴它，它就脫離了「你想怎麼改就怎麼改」的私有領地，進入「動一下就可能弄壞別人」的公共領地。REST、版本化、OpenAPI 這三樣東西，表面上是三個獨立的話題，骨子裡都在回答同一個問題：**當介面的兩端由不同的人、在不同的時間、以不同的步調演進時，要靠什麼把它們綁在一起又不至於互相絆倒。** 這一章講的就是這套機制：它買到什麼保證、在哪裡買不到、以及那些「看起來安全其實會炸」的邊界。

## REST 不是 URL 漂不漂亮，是一組可機器驗證的承諾

先把一個常見的混淆撥開。很多關於 REST 的爭論——複數還是單數名詞、要不要尾斜線、巢狀資源該嵌幾層——其實無關痛癢。Fielding 在 2000 年的博士論文裡把 REST 定義成一組**架構約束**，核心是「以資源為中心、透過統一介面操作」：每個資源有自己的 URI，用 HTTP 方法表達意圖，用狀態碼表達結果。但對一個要把系統接起來的後端工程師來說，REST 真正值錢的不是這套哲學的美感，而是它附帶的幾條**可以被機器驗證、且 client 可以據此做決定**的承諾。

最重要的一條是**方法語意**。RFC 9110（HTTP Semantics，2022 年取代了舊的 RFC 7230–7235 系列）把每個方法的兩個性質釘死了：

- **safe（安全）**：方法的語意本質上是唯讀的，client 不請求、也不預期任何狀態改變。GET、HEAD、OPTIONS、TRACE 是 safe 的。
- **idempotent（冪等）**：同一個請求送一次和送 N 次，對伺服器的**預期效果**相同。GET、HEAD、OPTIONS、TRACE、PUT、DELETE 是冪等的；POST 不是。

注意 idempotent 的定義用的是「預期效果」，不是「位元層級完全相同的回應」。一個 `DELETE /orders/42` 送第一次回 `204 No Content`、送第二次因為東西已經沒了而回 `404 Not Found`——回應碼不同，但它仍然是冪等的，因為**伺服器的狀態**在第二次之後沒有再改變。同理，RFC 9110 明說伺服器可以對每個冪等請求做「非冪等的副作用」，比如每次都寫一筆 access log、留一份修訂歷史——這些不影響資源狀態，所以不破壞冪等性。把冪等理解成「資源的最終狀態與重送次數無關」，而不是「回應永遠一字不差」，才抓到了它的真意。

這條承諾為什麼值錢？因為它直接決定了**逾時之後能不能安全重試**。設想 client 送出一個請求後連線斷了，它根本不知道伺服器到底收到沒、處理了沒。如果這是個冪等的 PUT，client 可以閉著眼睛重送——最壞情況是伺服器收到兩次、但結果跟收到一次一樣。如果這是個非冪等的 POST（比如「建立一筆付款」），盲目重送就可能扣兩次款。整個分散式系統的重試、退避邏輯，都建立在「這個方法到底冪不冪等」這個判斷上。把刪除做成 `GET /orders/42/delete` 之所以是災難，正是因為它撒了謊：它用一個被定義為 safe 的方法去做一件有破壞性的事，於是任何瀏覽器的預抓取、任何爬蟲、任何重試框架，都會理直氣壯地把你的資料刪掉——它們有充分的理由相信 GET 不會改變任何東西。

順帶澄清一個糾纏已久的細節：**PATCH 不保證冪等**。它冪不冪等，完全取決於 patch 文件的語意。`PATCH {"status": "shipped"}` 這種「設定成某個絕對值」的寫法是冪等的（送幾次結果都是 shipped）；但 `PATCH {"op": "increment", "field": "retry_count"}` 這種「在現值上做相對運算」的寫法，送兩次就加兩次，徹底不冪等。需要冪等性的時候，別指望方法本身給你，得自己加冪等鍵（idempotency 的完整機制有它自己的歸屬章節，這裡只需要知道：當業務要求「同一個操作不管送幾次只生效一次」時，靠的是 client 帶一個唯一的 key、server 端記住「這個 key 我處理過了」——這套做法見〈重複是常態：冪等與去重〉）。值得一提的是，IETF 的 httpapi 工作組正在把這個慣例標準化成一個 `Idempotency-Key` 請求標頭（`draft-ietf-httpapi-idempotency-key-header`，2025-10 出到 -07 版、Standards Track，2026-06 仍是 draft），讓「為 POST/PATCH 加冪等鍵」這件原本各家各做的事有一份共同規範。

第二條值錢的承諾是**用狀態碼說真話**。把失敗包在 `200 OK` 的 body 裡（`{"success": false, "error": "..."}`）是一種特別陰險的反模式，因為它在語意層騙了所有只看狀態碼的中間人：load balancer 以為這台後端健康、監控系統把它算成成功、client 的重試框架不會重試、CDN 可能還把這個「成功」的回應快取起來。錯誤就該回 4xx（client 的問題）或 5xx（server 的問題），讓整條鏈路上基於狀態碼的告警、重試、熔斷都能正確運作。至於錯誤的**細節**該長什麼樣，IETF 已經給了標準答案：RFC 9457（Problem Details for HTTP APIs，2023-07 發布，取代了 2016 年的 RFC 7807）定義了一個 `application/problem+json` 的 body 格式，帶 `type`、`title`、`status`、`detail`、`instance` 幾個標準欄位，讓「機器可讀的錯誤」有一份跨服務通用的形狀，而不是每個 API 自己發明一套 `{code, message}`。

## 為什麼「就直接改」會出事：契約一旦有人依賴就凍結了

理解了 REST 的承諾，下一個問題就浮上來了：這份契約會**演進**。需求會變，欄位會增刪，型別會調整。如果介面只有你自己一個 client、而且你能把 client 和 server 一起部署，那根本沒有版本化問題——改完一起上線即可，這種情況下提前搞一套版本機制反而是過度工程。

麻煩出在介面有了**你無法強迫它同步升級**的 client：一個獨立部署的前端團隊、一個第三方整合、一支已經裝在使用者手機上、可能半年都不更新的舊版 App。這些 client 對你的介面形成了一種隱形的依賴——你看不到他們的程式碼，但他們在依賴你回傳的每一個欄位、每一個 enum 值、每一個狀態碼的確切形狀。從這一刻起，你的介面就**凍結**了：不是不能改，而是任何改動都得先回答「這會不會弄壞某個我看不見的 client」。

把可能的變更按「破壞性」排個序，是理清這團亂麻的第一步。**不破壞式變更（additive change）**是最安全的一檔：新增一個可選欄位、新增一個端點、新增一個可選的 query 參數。舊 client 看不到這些新東西，自然也不會被影響——它讀它原本就在讀的那些欄位，多出來的當作不存在。絕大多數日常演進都能塞進這條路，這應該是你的第一選擇。

但「additive 一定安全」這個信念，正是開場那封郵件的陷阱所在。**新增一個 enum 值，在規格上是新增，在某些 client 眼裡卻是破壞性的。** 一個用嚴格 schema 驗證的 client，看到回應裡出現它的 schema 沒列舉的 `partially_refunded`，驗證直接失敗；一個用窮舉 `switch` 的 client，沒有 `default` 分支接住未知值，就拋例外。問題的根源在於：**「安全」從來不是由你這端單方面決定的，而是由 client 怎麼解讀你的輸出決定的。** 在發送端，這只是多給了一點資訊；但對一個假設「我見過所有可能值」的 client 來說，多出來的那一點就是違約。這也是為什麼成熟的 API 在文件裡會明寫一句「client 必須容忍未知的 enum 值」——把「對未知值要寬容」這條義務，明確地寫進契約、推給 client，而不是默默假設。

## 版本化的幾條路，以及它們各自欠的債

當一個變更**真的**無法用 additive 的方式塞進去——比如你必須改掉一個欄位的型別、或重組整個回應結構——你就被迫面對版本化。版本化的本質，是讓**舊契約和新契約並存一段時間**，給 client 一個按自己步調遷移的窗口。怎麼讓兩個版本並存，有幾條路，每條都買到不同的東西、也各自欠下不同的債。

**把版本放進 URI**（`/v1/orders` → `/v2/orders`）是最直白的一條。它的好處全來自「版本是顯式的、寫在最顯眼的地方」：路由器一眼就能把 `/v1` 和 `/v2` 導到不同的後端或不同的程式碼路徑；CDN 和快取天然地把它們當成兩個不同的資源、不會搞混；出問題時翻 access log，每一行都寫著它打的是哪個版本，debug 不必猜。代價是哲學上的：嚴格說 `/v2/orders` 並不是「orders 的新版本」，而是「另一個叫 orders 的東西」——資源的身分被版本號污染了，REST 純粹主義者看了難受。但工程上這條路最省心，這也是為什麼絕大多數對外公開的 API 走的是 URI 版本化。它欠的債是另一種：v1 和 v2 並存期間，你的程式碼會分叉，兩套邏輯都要維護、都要測；版本一多，淘汰機制要是沒跟上，你會養出一窩沒人敢關掉的殭屍版本。

**把版本放進 header**（`Accept: application/vnd.myapi+json; version=2`，或一個自訂的 `X-API-Version`）走的是相反的取捨。URI 保持乾淨——`/orders` 永遠是 `/orders`，資源身分純粹了。但版本藏進了 header 這個不顯眼的角落，於是前面 URI 版本化白送的好處全要自己掙回來：CDN 和任何快取層**必須**把這個 header 納入 cache key，否則一個快取住的 v1 回應會被原封不動地塞給一個要 v2 的請求——這是個很安靜、很難查的 bug，因為它只在快取命中時發作；access log 預設不記 request header，你得自己把版本顯式地記進 log，否則出事時連對方打的是哪個版本都看不出來。header 版本化適合內部 API、單一團隊掌控所有 client 的場景，那裡乾淨的 URI 的價值，蓋過了「版本藏起來」帶來的觀測性損失。

還有一條更細粒度、不需要翻整個端點版本的路：**欄位層級的滾動演進**。不動 URI、不換大版本，靠「新增欄位、舊欄位標記 deprecated 但繼續回傳、設一個下線日期」來逐步搬遷。這條路的關鍵在於它必須**真的被執行**——deprecated 不能只是文件裡一句話，得有機器可讀的下線預告，而 HTTP 剛好有對應的標頭。RFC 8594 定義的 `Sunset` 標頭帶一個 HTTP-date 時間戳，明確告訴 client「這個資源預計在這個時間點之後變得不可用」；搭配 RFC 9745（The Deprecation HTTP Response Header Field，2025-03 發布，Proposed Standard）定義的 `Deprecation` 標頭，可以把「已棄用」和「將下線」兩個階段分開傳達——`Deprecation` 說「這東西不推薦用了、但還能用」，`Sunset` 說「它會在這個時刻徹底關掉」。OpenAPI 那邊也有對應的 `deprecated: true` 標記，讓棄用狀態同時出現在契約文件裡。這套機制買到的是「不必為了改一個欄位就翻整個端點版本」的細粒度，欠的債是**紀律**：如果你標了 deprecated 卻從不真的下線、從不發告警去推 client 遷移，那些 deprecated 欄位會永遠死不了，你的回應會越來越胖、契約會越來越髒，最後變成一座「誰也不敢動」的沼澤。

## 一個攤開來算的遷移：90 天，36 MB，換零中斷

把版本化從原則落到數字上，才看得出它到底貴不貴。設想一個 API 日均 200 萬次呼叫，其中 `GET /v1/orders` 佔了六成——大約每天 120 萬次。產品要把回應裡的金額欄位改個形狀：原本 `amount` 是一個整數（以「分」為單位，比如 `12300` 代表 123 元），現在要改成帶幣別的物件 `{"currency": "TWD", "minor_units": 12300}`，因為系統要開始支援多幣別了。

最直接的做法——把 `amount` 從整數原地改成物件——是純粹的破壞性變更。每天那 120 萬次呼叫的 client，凡是把 `amount` 當數字來讀的，瞬間全部炸掉：算數的地方拿到一個物件、做比較的地方型別錯誤、序列化的地方崩潰。這條路不能走。

不破壞式的做法是這樣鋪的：**新增**一個 `amount_v2` 物件欄位，與此同時把舊的 `amount` 標記為 deprecated，並設一個 90 天後的 `Sunset` 日期。在這 90 天的窗口裡，兩個欄位**同時回傳**——舊 client 繼續讀它的 `amount`、毫無感覺；新 client 和已經遷移的 client 讀 `amount_v2`。你在這段時間做的關鍵動作是**監控**：盯著「仍然只讀舊 `amount`、完全沒碰 `amount_v2` 的 client」佔多少比例，透過某種 client 識別（API key、User-Agent、或要求 client 在遷移後帶一個標記）來追蹤。等這個比例降到趨近於零——意味著幾乎所有人都搬完了——你才把 `amount` 真正移除。

這套做法不是免費的，代價可以算出來。多出來的那個 `amount_v2` 物件，序列化成 JSON 大約是 `"amount_v2":{"currency":"TWD","minor_units":12300}` 這麼長，連欄位名帶結構約 50 個位元組，保守抓個 40 位元組的淨增量。乘上每天 120 萬次呼叫，就是約 4,800 萬位元組、大約 48 MB 的額外出向流量，每天。90 天累積下來約 4.3 GB 的多餘傳輸。

這個數字本身不痛——雲端出向流量這個量級的成本是幾分錢的事。但把它擺出來，是為了看清這筆交易的兩端：你付的是**90 天裡每天約 48 MB 的冗餘流量、加上兩套金額邏輯並存的程式碼複雜度**，買到的是**零中斷遷移**——那 120 萬次/日的 client 沒有一個在某個瞬間突然炸掉，每個人都在自己的時間表裡平順地搬過去。把這筆交易跟「原地改型別、然後在 incident 頻道裡滅火、跟第三方道歉、緊急回滾」的代價一比，48 MB/日 便宜得近乎免費。這就是為什麼「貴一點、慢一點，但零中斷」幾乎總是對的選擇——版本化付的那點冗餘，是拿可量化的小成本，去贖掉一個不可控的大風險。

## OpenAPI：把契約變成一份機器讀得懂的單一真相

到目前為止，「契約」都還是個抽象詞——它散落在文件、在程式碼、在工程師的腦子裡。一旦 client 多起來、一旦你想自動生成 SDK 或 mock server，這種散落就成了問題：哪份文件是對的？這個欄位到底可不可選？這個端點會回哪些錯誤碼？OpenAPI（前身叫 Swagger）解的就是這件事——它把端點、參數、請求與回應的 schema、錯誤碼，全部寫成一份**機器可讀**的 YAML 或 JSON，讓 client SDK、mock server、契約測試、API 文件**全部從同一份真相生成**，而不是各自維護、各自漂移。

OpenAPI 的版本史上有一個值得記住的里程碑：**3.1.0（2021-02 發布）讓 OpenAPI 的 schema 與 JSON Schema Draft 2020-12 完全對齊**。聽起來像個無聊的技術細節，但它解掉的是一個真實的痛：3.1.0 之前，OpenAPI 用的是自己魔改的 JSON Schema 子集，跟標準的 JSON Schema 似是而非——於是整個工具鏈各自為政，你用 JSON Schema 寫的驗證規則沒法直接餵給 OpenAPI 工具，反之亦然。3.1.0 把這道牆拆了，schema 從此可以在兩個世界之間直接流通。最新的版本是 **3.2.0（2025-09 發布）**，它在 3.1 的基礎上加了結構化的 tag（可以分類、巢狀、做出階層）、串流媒體型別的一級支援、以及——很應景地——對 HTTP `QUERY` 方法的支援。

說到 `QUERY`，這是個正在發生、值得知道的變化。它要解的是一個 REST 由來已久的尷尬：當查詢條件複雜到塞不進 URL（想想一個帶十幾個過濾條件、巢狀結構的搜尋），你只剩兩個爛選項——用 GET 但被迫把一大坨參數硬編進 URI（撞上 URL 長度上限、還難讀），或用 POST 帶 body（但 POST 既不 safe 也不冪等，於是快取和自動重試全都不能用）。`QUERY` 方法是第三條路：它允許帶 request body，同時保有 GET 那樣的 **safe 且冪等**語意，於是「複雜查詢」終於可以既帶結構化的 body、又能被快取、又能安全重試。這個方法在 2025-11-20 由 IESG 核准，並於 2026-06-15 正式發布為 RFC 10008（The HTTP QUERY Method，Proposed Standard），OpenAPI 3.2.0 在它定案前就已經先支援了它。

但 OpenAPI 最關鍵的一個**保證邊界**，必須講清楚，否則它會給你虛假的安心：**OpenAPI 只是「描述」，不是「強制」。** 一份 OpenAPI 文件保證的是「存在一份契約」，它**不保證實作真的符合這份契約**。你完全可以寫一份漂亮的 spec 說某端點回傳 `{id, name, email}`，然後在程式碼裡因為一個 bug 回了 `{id, name}`——少了 email——而那份 spec 對此一無所知，它只是一份宣告，不會去攔你。要讓「描述」變成「保證」，你得加一道把兩者對起來的機制：**契約測試（contract testing）**——把 OpenAPI 生成的契約拿去比對真實的回應，確認實作沒有偷偷偏離 spec。沒有這道機制，OpenAPI 給你的只是「我們**聲稱**契約長這樣」，而不是「實作**真的**是這樣」。這個區別在 incident 的時候會變得非常具體：當 client 抱怨「你回的東西跟文件不一樣」，問題往往不是文件錯了，而是**沒有任何機制保證文件和實作同步**——spec 和 code 各自演進，慢慢就對不上了。

## 什麼時候不該做這些

技術章容易給人一種「全都要上」的錯覺，但版本化和 OpenAPI 都是有成本的，盲目套用本身就是一種反模式。判準其實很乾淨，回到「介面有沒有你無法同步升級的 client」這條線就好。

**版本化**在「介面一有外部 client」時是必需的——因為你無法強迫所有 client 同步升級，舊契約必須留著。但反過來，當介面**只有內部、單一團隊、可以把 client 和 server 一起部署**時，預先扛起一套版本機制就是過度工程：你在為一個「未來也許會有外部 client」的假想需求，預付真實的維護成本（每個版本都要測、要維護、要想淘汰策略）。這種時候直接改、一起部署，遠比養一套用不到的版本基礎設施划算。

**OpenAPI 契約**的報酬率，在「多 client、要自動生成 SDK、要 mock、要契約測試」時最高——這些場景下，一份單一真相能同時餵養好幾條下游管線，維護 spec 的成本被攤薄到近乎免費。但如果你只有一個 client、而且介面改得很勤，那麼維護「spec 與實作同步」這件事本身就是純負擔：每改一次 code 就要追著改 spec，而你又沒有從「自動生成 SDK／mock」裡賺回任何東西。這種時候，把 spec 延後、甚至先不寫，是合理的判斷。

兩個決定背後是同一個原則：**這些機制買的是「跨越組織與時間邊界的協調能力」。** 當你的介面確實跨越了這種邊界——多個你管不到的 client、多個會在不同時間遷移的消費者——它們的價值就浮現了。當邊界不存在時，你付的是成本、卻沒有東西可買。

## 為什麼是這個形狀

退一步看，這一整套東西——方法的 safe/idempotent 語意、用狀態碼說真話、additive-first 的演進策略、版本並存的窗口、OpenAPI 這份單一真相——都是從同一個無法迴避的事實裡長出來的：**介面的兩端，由不同的人在不同的時間以不同的步調演進，而你只控制得了其中一端。**

正因為你控制不了 client 何時升級，你才不能「就直接改」，才需要讓舊契約和新契約並存一個窗口。正因為 client 在你看不見的地方解讀你的每一個輸出，「安全」才不是你單方面說了算——一個新增的 enum 值，在規格上無辜、在某個窮舉 switch 面前就是違約。正因為逾時之後 client 必須自己決定要不要重試，方法的冪等性才不是學術細節、而是 client 賴以做決定的硬承諾。正因為「我聲稱契約長這樣」和「實作真的是這樣」是兩件事，OpenAPI 才必須配上契約測試，否則它只是一份會慢慢撒謊的文件。

REST、版本化、OpenAPI 不是三條孤立的最佳實踐，而是同一個問題——「如何在無法強制同步的兩端之間維持一份不互相絆倒的契約」——被逼到牆角後，分別在語意、演進、描述三個層面長出的答案。下次當你準備改一個對外的欄位，停下來問一句「在防火牆的另一邊，有沒有一個我看不見的 client，正假設這個欄位永遠是它現在的樣子」——這個問句本身，就是這整套機制存在的理由。
