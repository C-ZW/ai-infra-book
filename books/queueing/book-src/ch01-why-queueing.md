## 從你已知的出發

三段你大概不需要翻監控紀錄就能想起來的現場記憶。

**RDS CPU 的 80% 警戒線。** 你做過 profiling、把 RDS 的 CPU 尖峰壓下 40%。當初為什麼要動手？因為儀表板上那條線逼近 80%，而「80% 是警戒線」是團隊裡不需要解釋的常識。問題是：警戒的到底是什麼？CPU 在 82% 的時候並沒有燒起來，查詢還是一條一條在完成。那條線背後如果有一個數學理由，它應該長什麼樣？

**k6 壓測曲線的形狀。** 你用 k6 和 Vegeta 做過負載測試。每次曲線都長一個樣：負載低的時候延遲平平的，加壓上去也只是緩慢爬升，然後在某個點——每次都讓人覺得「剛剛明明還好好的」——P99 像被點燃一樣豎直起飛。那個拐點不是 bug，不是 GC，不是某次部署引入的 regression。它每次都在，換一套系統還在。

**SQS backlog 的尖峰記憶。** 結算 pipeline 的 SQS 佇列，平常深度貼著零。尖峰時段 backlog 堆起來，過了尖峰自己消化掉。worker 的總處理能力明明高於訊息的平均到達速率——「平均上」永遠追得上——backlog 為什麼還是會堆？

這三件事是同一件事。本書用一個貫穿全書的例子來釘住它：**推播管線**——API 收到事件、推播 worker 負責發送，原型是你維護過的 WebSocket 通知系統（約 10M 則訊息/月；以 30 天計，平均到達率約 3.86 則/s）。基準數字如下，全書到處會用到，值得現在就記住：

| 量 | 值 | 這個數字怎麼來的 |
|---|---|---|
| 單一 worker 平均處理時間 | E[S] = 100 ms（服務率 μ = 10 則/s） | 量測得到 |
| 尖峰到達率 | λ = 8 則/s | 量測得到 |
| 使用率 | ρ = λ/μ = 0.8 | 算術 |
| 尖峰平均回應時間 | E[T] = 0.5 s | ch08 推導 |
| 尖峰 P99 回應時間 | ≈ 2.30 s | ch08 推導 |

注意這組數字裡的反差：worker 有兩成的時間是閒著的，但平均回應時間是處理時間的 5 倍、P99 是 23 倍。沒有任何元件故障，沒有任何資源耗盡。**這不是 bug，是數學**——而且是可以從第一原理推出來的數學。本章先讓你看到「等待是怎麼無中生有的」，推導留給後面的章節。

順帶一句歷史定位：這門數學的起點不是網路服務，是電話。A.K. Erlang 在哥本哈根電話公司工作時，1909 年證明了隨機撥出的電話呼叫服從 Poisson 分布，1917 年給出交換機的損失與等待公式（見 ch09）；「排隊理論」長成一門一般化的學科則是 1950 年代的事，Kendall（1953，記號系統的出處）與 Lindley（1952）等人把它建制化。電話交換機房和你的推播管線，在數學上是同一個房間。

## 確定性的世界不排隊：D/D/1 對照組

要看清楚等待從哪裡來，先看一個**不會排隊**的世界。

設定：1 秒內到達 8 個 job（λ = 8），每個 job 的處理時間**恰好** 100 ms（μ = 10），單一 worker，先到先服務。如果到達是完美等距的——每 125 ms 來一個——時間線是這樣：

| Job | 到達 (ms) | 開始服務 (ms) | 完成 (ms) | 等待 (ms) |
|---|---|---|---|---|
| J1 | 0 | 0 | 100 | 0 |
| J2 | 125 | 125 | 225 | 0 |
| J3 | 250 | 250 | 350 | 0 |
| J4 | 375 | 375 | 475 | 0 |
| J5 | 500 | 500 | 600 | 0 |
| J6 | 625 | 625 | 725 | 0 |
| J7 | 750 | 750 | 850 | 0 |
| J8 | 875 | 875 | 975 | 0 |

每個 job 到達時 worker 都剛好閒著（前一個 job 在 100 ms 內做完，下一個 125 ms 才來），**等待恆等於零**。使用率一樣是 0.8——worker 每 125 ms 裡忙 100 ms——但佇列從不存在。

這種「到達間隔固定、服務時間固定、單一伺服器」的系統，排隊理論的速記寫作 **D/D/1**（D = deterministic；這套記號叫 Kendall 記號，本章稍後給你一張對照表，正式定義在 ch08）。D/D/1 的結論可以一句話講完：只要 λ < μ，永遠沒有人等。如果世界是確定性的，這本書到這裡就可以結束了。

## 等待從哪裡生出來：手排一條叢發時間線

現在只改一件事：**同樣這 8 個 job**，把前 4 個擠成一團、全部在 t = 0 到達，後 4 個維持原時刻。λ 還是 8 則/s，μ 還是 10 則/s，總工作量還是 800 ms，使用率還是 0.8。手排時間線（這是本章的 worked example，值得你拿紙跟著走一遍）：

| Job | 到達 (ms) | 開始服務 (ms) | 完成 (ms) | 等待 (ms) |
|---|---|---|---|---|
| J1 | 0 | 0 | 100 | 0 |
| J2 | 0 | 100 | 200 | 100 |
| J3 | 0 | 200 | 300 | 200 |
| J4 | 0 | 300 | 400 | 300 |
| J5 | 500 | 500 | 600 | 0 |
| J6 | 625 | 625 | 725 | 0 |
| J7 | 750 | 750 | 850 | 0 |
| J8 | 875 | 875 | 975 | 0 |

總等待 0 + 100 + 200 + 300 = 600 ms，平均等待 75 ms，最倒楣的 J4 等了 300 ms——三個完整的服務時間。兩條時間線的 worker 忙碌狀態並排看：

```text
1 格 = 25 ms；「#」= 服務中、「.」= 閒置
         0 ms                                    1000 ms
         |----------------------------------------|
等距到達  ####.####.####.####.####.####.####.####.
叢發到達  ################....####.####.####.####.
```

兩條線的「#」都是 32 格（800 ms 的工作）、「.」都是 8 格（200 ms 閒置）。**忙碌總量一模一樣，等待卻從 0 變成 600 ms。**這個例子裡藏著三個觀察，是全書反覆回收的母題：

**觀察一：等待誕生於局部超載。**那一團 4 個 job 在一瞬間到達，瞬間的「到達速率」遠超過 μ。λ < μ 說的是長期平均追得上，對任何一個 125 ms 的窗口它什麼都沒承諾。

**觀察二：閒置不可儲存。**等距版裡 worker 每輪閒 25 ms，叢發版裡這些閒置集中在 t = 400–500 之間和尾段。但 t = 400 的閒置救不了 t = 0 被擠在佇列裡的 J3、J4——服務能力是「易腐品」，當下沒用掉就永久蒸發。等待不能用之後的空檔退款。

**觀察三：叢發的代價是平方的。**一團 b 個 job 同時抵達一台閒著的 worker，第 i 個（從 0 數起）要等前面 i 個做完，等待 i·E[S]；整團的總等待是 (0+1+⋯+(b−1))·E[S] = b(b−1)/2 · E[S]。這是完整推導——只是算術，但結論有牙齒：**團的大小翻倍，總等待大約翻四倍**。適用條件：服務時間固定為 E[S]、worker 在整團抵達時閒置、排空前沒有其他 job 插進來、先到先服務。本例 b = 4：4·3/2 × 100 ms = 600 ms，和手排的表一致。

注意，到目前為止**沒有任何機率出場**——兩條時間線都是純算術，隨機性還沒登場，等待已經出現了。機率（Part II）要解決的是下一個問題：真實世界的到達不是你手排的，叢發以什麼頻率、什麼大小出現？「隨機到達」本質上就是永不停歇的小叢發。把這件事定量化之後你會得到全書最重要的一條敘事線：**隨機性（變異數）本身就是延遲的來源**——到達的不規則要付費（ch14），服務時間的不規則也要付費（ch12），帳單上會有它們各自的明細。

## 使用率階梯：80% 警戒線下面是懸崖

上面的例子固定了 ρ = 0.8。現在問：等待對 ρ 多敏感？

先把答案的形狀給你。對最簡單的隨機佇列模型 M/M/1（Poisson 到達、指數分布服務時間、單一伺服器、先到先服務、穩態、ρ < 1——每個詞在 ch08 之前都會有精確定義，現在先當「教科書第一個模型」），平均回應時間是：

```text
E[T] = E[S] / (1 − ρ)
```

本章不推導它（ch08 用生滅過程一步一步兌現），但你現在就可以用它讀數字。取 E[S] = 100 ms，這就是全書的**使用率階梯**：

| ρ | 剩餘容量 1−ρ | E[T] |
|---|---|---|
| 0.50 | 0.50 | 0.2 s |
| 0.80 | 0.20 | 0.5 s |
| 0.90 | 0.10 | 1.0 s |
| 0.95 | 0.05 | 2.0 s |
| 0.99 | 0.01 | 10 s |

```text
E[T]   （示意，兩軸皆非等比例）
10 s  ┤                                        *  ← ρ = 0.99
      │                                        *
      │                                       *
 2 s  ┤                                      *  ← ρ = 0.95
 1 s  ┤                                    *  ← ρ = 0.90
0.5 s ┤                                *  ← ρ = 0.80（基準尖峰）
0.2 s ┤                    *  ← ρ = 0.50
0.1 s ┤    *   *   *   *  ← 低使用率時 E[T] ≈ E[S]
      └───────────────────────────────────────────► ρ
       0                   0.5         0.8     1.0
```

這就是你在每一次壓測裡看到的那條曲線——hockey stick。讀它的正確姿勢有三條：

第一，**分母是 1−ρ，不是 ρ**。直覺把使用率當線性刻度：「80% 用掉了，還有 20% 餘裕」。但延遲關心的是剩餘容量的倒數：從 0.8 到 0.9，流量只增加 12.5%，剩餘容量卻砍半，平均回應時間從 0.5 s 翻倍到 1.0 s。從 0.95 到 0.99，流量只再多 4.2%，延遲從 2 s 變 10 s。**剩餘容量每砍半，延遲翻倍**——這才是值得放進心算的形式。

第二，低使用率端幾乎免費。ρ → 0 時 E[T] → E[S] = 0.1 s（把 ρ = 0 代入公式即得）；ρ = 0.5 也才 0.2 s。曲線的左半邊平到讓人誤以為它永遠平下去——這正是「上週還好好的，這週多了 15% 流量就炸了」的數學機制。

第三，**爆炸是漸近的，不是踩到開關**。不存在一個「超過就壞掉」的魔法數字；80% 只是人類選了一個落在「曲線開始彎」的整數。它該不該是 80，取決於這條曲線的形狀，而曲線的形狀取決於模型假設——M/M/1 的階梯數字搬到別的系統會錯多少，是「陷阱與防禦」一節的主題，也是 ch09／ch12／ch14 各自要修正的帳。

## 平均值只說了一半：P99 的預告

階梯表給的是平均。你的 SLA 不是用平均訂的。

預告一個 ch08 的結果：在 M/M/1（同樣的適用條件：Poisson 到達、指數服務、單一伺服器、FCFS、穩態）裡，回應時間 T 的完整分布是已知的——指數分布，參數 μ−λ——所以任何百分位都能直接寫出來。P99 是：

```text
t₉₉ = ln 100 / (μ − λ) ≈ 4.6052 / (μ − λ)
```

而平均是 E[T] = 1/(μ−λ)，所以 **M/M/1 的 P99 永遠約等於 4.6 倍平均**（ln 100 ≈ 4.6052）。推播管線基準：μ−λ = 2，E[T] = 0.5 s，t₉₉ ≈ 2.30 s。

兩件事值得現在記下。其一，「平均 0.5 s」與「百分之一的使用者等超過 2.3 s」是同一個系統的同一個穩態——平均值正常從來不代表尾巴正常，監控只看平均等於閉一隻眼。其二，4.6× 這個倍數**是 M/M/1 專屬的**：它依賴指數分布那條瘦得很有教養的尾巴。當服務時間是重尾分布（少數 job 大得離譜——你那支偶爾跑 30 秒的報表 query），P99 與平均會徹底脫鉤，4.6× 規則死得很難看，多難看 ch15 會算給你看。平均與 tail 是兩回事，這是全書六條主線之一。

## 地圖的語言：Kendall 記號一瞥

前面已經偷跑了 D/D/1 和 M/M/1。這套速記是 Kendall 1953 年引入的，格式是 **A/S/c**，有時加第四欄 **/K**：

| 欄位 | 意思 | 常見取值 |
|---|---|---|
| A | 到達過程 | M = Poisson（無記憶，memoryless）、D = 固定間隔、G = 任意分布 |
| S | 服務時間分布 | M = 指數、D = 固定、G = 任意 |
| c | 伺服器數 | 1、c、∞ |
| K | 系統容量上限 | 省略 = 無限；寫了就是「滿了拒收」 |

你只需要把它當地圖上的地名，正式定義與預設慣例（FCFS 等）在 ch08。本書的主要停靠站：

| 記號 | 白話 | 章 |
|---|---|---|
| M/M/1 | 隨機到達、指數服務、一台機器——第一個能完整解出來的模型 | ch08 |
| M/M/c | c 台機器共用一條佇列——池化與 Erlang 公式 | ch09 |
| M/M/1/K | 容量有上限、滿了丟棄——backpressure 的數學 | ch10 |
| M/G/1 | 服務時間任意分布——變異數開始收費 | ch12 |
| G/G/1 | 到達與服務都任意——沒有精確解，只有 Kingman 的近似 | ch14 |

「G」是個誠實的字母：它承認你對分布一無所知，而理論能告訴你「只知道平均和變異數時，最多還能說什麼」。

## 這門理論能回答什麼、不能回答什麼

買地圖之前先看清楚它涵蓋的疆域。**排隊理論能回答的**（在各模型的假設成立時）：

1. **穩態的平均量**：平均佇列長、平均等待、系統內平均人數——有些模型還給完整分布，連 P99 都能寫成公式（ch08、ch12）。
2. **穩定性的邊界**：什麼條件下佇列必然無限長大、什麼條件下存在穩態（ch07、ch08），以及 retry 迴圈怎麼把一個本來穩定的系統推過邊界（ch16）。
3. **配置決策**：給定 SLA 要幾台伺服器（ch09）、buffer 該多大與丟棄率多少（ch10）、一台快機與多台慢機誰贏（ch09）、排程順序怎麼選與代價是什麼（ch13）。
4. **變異數的價格**：到達或服務的不規則會讓等待多多少——精確到公式（ch12、ch14）。
5. **網路層級的行為**：多站系統的瓶頸在哪、各站佔用多少、整體回應時間多長（ch16–18）。
6. **公式失效時的退路**：什麼時候只能模擬、模擬要怎麼做才不騙人（ch19）。

**它不能回答、或天生不直接回答的**：

1. **個案的因果**——「昨天 14:03 那次尖峰的根因是什麼」。理論給的是分布層級的敘述（這種尖峰多常發生、多大），不是單一事件的因果鏈；事後歸因是 trace 和 log 的工作。
2. **暫態動態的細節**——過載怎麼升起、退潮要多久。本書主攻穩態；過載動態的數學（fluid／diffusion 近似）只在 ch21 的地圖上標位置。
3. **確定性的 worst-case 保證**——「最壞情況下絕不超過 X ms」屬於 network calculus 的世界，本書一句話帶過（見 ch21）。
4. **服務時間為什麼長這樣**——cache 命中、GC、資料傾斜是系統知識；理論把 E[S] 與它的變異數當輸入，不解釋它們的成因。
5. **人的策略行為**——「隊太長使用者就走了」如果是使用者的理性決策，那是 strategic queueing 的題目，本書不展開（見 ch21）。

## 經驗法則的審判：全書的案件清單

你的工程生涯累積了一批「大家都這麼說」的黑話。它們多數有真實的經驗基礎，但經驗法則和定理的差別在於：定理自帶適用條件，黑話沒有——所以黑話會在你最需要它的時候背叛你。本書把它們一條一條送上法庭：**立案在這裡，各章審理，ch20 逐條宣判**。判決只有三種：成立（升格為定理）、有條件成立（附上精確的適用範圍）、推翻。

這份清單是全書的契約，之後各章引用時寫「審判 #N」：

| # | 工程黑話 | 開庭要回答的問題 | 承審章 |
|---|---|---|---|
| 1 | 「使用率不要超過 80%」 | 80 是定理還是迷信？什麼時候警戒線該是 60、什麼時候可以是 95？ | ch08 立案、ch09/ch14 修正 |

| 4 | 「retry 是免費的保險」 | retry 把 λ 放大多少？什麼時候把系統推下懸崖、而且鬆手也回不來？ | ch16、ch20 |
| 5 | 「buffer 開大一點總沒錯」 | 大 buffer 治什麼、不治什麼、什麼時候只是把死刑改成凌遲？ | ch10 |
| 6 | 「上下游各自做容量規劃就好」 | 「上游的輸出＝下游的輸入」在什麼數學條件下合法？ | ch11、ch16、ch18 |
| 7 | 「壓測跑三次取平均就穩了」 | 模擬與壓測的輸出統計要怎麼做才誠實？固定 VU 的壓測看得到真實過載嗎？ | ch17、ch19 |
| 8 | 「已經等這麼久了，應該快輪到了」 | 等待的歷史對剩餘等待透露了什麼——還是什麼都沒透露？ | ch04、ch12 |

全部案件在 ch20 結案重審——到時你手上會有本章還沒給你的全部工具。

## 陷阱與防禦

本章沒教你任何可以推導的東西，但已經給了三個可以**誤用**的數字（階梯表、4.6× 規則、b(b−1)/2）。先把護欄裝上：

| 陷阱 | 它怎麼騙你 | 怎麼自我察覺與防禦 |
|---|---|---|
| 把使用率階梯當萬用常數表 | 階梯數字是 M/M/1 專屬。服務時間固定時等待砍半（M/D/1，見 ch12）；同樣 ρ 之下多台伺服器共用佇列等待更低（見 ch09）；重尾服務時間下可以差一個數量級以上（見 ch15） | 引用任何數字前先問「這是哪個模型算的、我的系統哪裡像哪裡不像」；說不出模型名，就只信形狀、不信數字 |
| 用平均率判定「不會排隊」 | λ < μ 只保證長期追得上，對任何短窗口零承諾——本章手排例 λ = 8 < μ = 10，平均等待 75 ms、最倒楣的 job 等 300 ms | 聽到「容量夠所以不會等」就去找叢發：到達間隔真的接近等距嗎？上游有沒有批次、retry、cron？ |
| 拿「P99 ≈ 4.6× 平均」走天下 | 它是 M/M/1 的精確結果，靠的是指數分布的瘦尾巴；重尾之下 P99 與平均脫鉤，誤差可以大到一個數量級（ch15 會算給你看） | 規則跟著模型走：講得出假設才准用；講不出來，P99 一律實測 |
| 把 ρ 讀成線性刻度 | 「還有 20% 餘裕」聽起來像「還能加 20% 的量、代價不變」；實際上 0.8 → 0.9 流量只多 12.5%，平均延遲翻倍 | 心算一律用 1/(1−ρ) 的形式：剩餘容量減半 ⇒ 延遲翻倍 |

## 紙上推演

### 題 1：手排兩團叢發 **[15 分鐘] ★**

同樣 8 個 job、每個服務恰好 100 ms、單一 worker、先到先服務（λ = 8、μ = 10 不變）。這次到達時刻是 0, 0, 0, 0, 500, 500, 500, 500 ms——兩團各 4 個。

(a) 手排完整時間線（每個 job 的開始、完成、等待），算總等待與平均等待，和本章兩條時間線（等距：總等待 0；一團 4 個：總等待 600 ms）比較。
(b) 如果第二團提早到 t = 300 ms（第一團還沒排空），總等待變多少？為什麼變多？

### 推演解答

(a) 第一團在 t = 400 排空，worker 閒置到 500，第二團抵達時又是一台閒機器：

| Job | 到達 (ms) | 開始 (ms) | 完成 (ms) | 等待 (ms) |
|---|---|---|---|---|
| J1–J4 | 0 | 0/100/200/300 | 100/200/300/400 | 0/100/200/300 |
| J5–J8 | 500 | 500/600/700/800 | 600/700/800/900 | 0/100/200/300 |

總等待 600 + 600 = 1,200 ms，平均 150 ms。三條時間線同樣的 λ 與 μ：等距 0、一團 75 ms、兩團 150 ms 的平均等待——**「到達的形狀」是一個獨立的變數**，而我們還沒有任何語言去量化它（ch03 的 C²ₐ 就是那個語言，ch14 把它寫進帳單）。每團恰好貢獻 b(b−1)/2 · E[S] = 600 ms，因為兩團彼此不重疊：適用條件「整團抵達時 worker 閒置」對兩團都成立。

(b) 第二團在 t = 300 抵達時，J4 還有整整 100 ms 的工作沒做。J5–J8 的每一個都得多扛這 100 ms 的殘留 backlog：等待變成 100/200/300/400，第二團小計 1,000 ms，總等待 1,600 ms（= 1,200 + 4×100，與逐項手排一致；worker 從 0 連續忙到 800）。教訓：**叢發的代價不只平方於團的大小，重疊的叢發還會把殘留等待往後傳**。隨機到達會三不五時送你重疊的叢發——這是 hockey stick 右端那麼陡的原因之一。常見錯路：把 (b) 算成「跟 (a) 一樣，只是提早 200 ms 發生」——錯在忘了檢查「抵達時 worker 是否閒置」這個適用條件。

### 題 2：把階梯反過來用 **[5 分鐘] ★**

沿用 M/M/1 假設與 E[T] = E[S]/(1−ρ)，單一 worker。

(a) worker 不變（E[S] = 100 ms），要求尖峰平均回應 E[T] ≤ 0.25 s，使用率最多能到多少？對應的 λ 上限是多少？
(b) 尖峰 λ = 8 則/s 不變，要求同樣的 E[T] ≤ 0.25 s，worker 要多快（μ 至少多少、E[S] 至多多少）？

### 推演解答

(a) 0.1/(1−ρ) ≤ 0.25 ⇒ 1−ρ ≥ 0.4 ⇒ **ρ ≤ 0.6**，即 λ ≤ 6 則/s。複核：0.1/0.4 = 0.25 ✓。注意要求「平均回應壓在 2.5 倍服務時間內」就已經把可用容量砍到六成——SLA 越緊，你能用的使用率越低，這是用白紙黑字買 headroom。

(b) 改 μ 的時候 E[S] = 1/μ 跟著變，不能只動分母。把 E[S] = 1/μ、ρ = λ/μ 代入：

```text
E[T] = (1/μ) / (1 − λ/μ) = 1/(μ − λ)   ← 純代數整理
```

要 1/(μ−8) ≤ 0.25 ⇒ **μ ≥ 12 則/s**（E[S] ≤ 83.3 ms）。複核：μ = 12 時 ρ = 8/12 ≈ 0.667，E[T] = (1/12)/(1/3) = 0.25 s ✓。常見錯路：拿 (a) 的「ρ ≤ 0.6」直接解 8/μ ≤ 0.6 得 μ ≥ 13.3——錯在沿用了舊的 E[S]；換了 μ，整條階梯的縱軸刻度都變了。順帶看見一個漂亮的事實：M/M/1 的平均回應時間只由「剩餘服務能力 μ−λ」決定（這個形式 ch08 會正式推導）。

### 題 3：三個系統，各歸地圖哪一塊 **[10 分鐘] ★★**

判斷下列系統各該用排隊理論的哪一塊分析，說出推理（提示：對每個系統問三個問題——到達是開放還是封閉？容量有限嗎？有沒有回饋迴圈？）：

(a) SQS 佇列＋8 個 worker consumer；backlog 實務上無上限；訊息來自幾十萬個互相獨立的使用者。
(b) k6 壓測：200 個 VU，每個 VU 收到回應才送下一個 request。
(c) API gateway 限制同時在途請求 1,000 個，超過直接回 429；client 收到 429 之後會 retry。

### 推演解答

(a) **開放、容量近似無限、無回饋**：到達者不管系統多滿都照來（幾十萬獨立來源的匯流也是 ch05 解釋「為什麼到達常像 Poisson」的典型場景）。8 個 worker 共用一條佇列 → M/M/c 一族（ch09）；若 job 處理時間的變異大或有重尾，再加上 ch12／ch15 的修正。
(b) **封閉**：系統裡永遠最多 200 個「在途請求」，回應變慢時 VU 自動變慢——負載會自我節流。這是 ch17 的封閉網路（ch02 的 bounds 先給你粗答案）；它也是「固定 VU 壓測天生看不到真實過載」的原因，ch19 會正式開庭。
(c) **開放、容量有限（K = 1,000）、有回饋**：滿了拒收是 M/M/c/K 一族（ch10）；但 429 觸發 retry，把被拒的流量送回到達端——這個回饋迴圈是 ch16 的 feedback 與 ch20 retry storm 的主場。三個問題裡「有沒有回饋」最容易漏看，而它恰好是把系統從「可解」推向「會雪崩」的那一個。

### 題 4：口頭重講 **[10 分鐘] ★**

不看書，對空氣講 90 秒：「為什麼 ρ < 1 的系統還會排隊？」講完再對照解答。

### 推演解答

合格的講稿至少要有這兩個成分，缺一個都不算懂：

1. **等待來自局部超載**：λ < μ 是長期平均的承諾，對短窗口無效；只要到達有叢發（而隨機到達就是持續的小叢發），瞬間需求就會超過瞬間供給，超出的部分變成佇列。
2. **閒置不可儲存**：佇列消化靠的是「之後的空檔」，但空檔不能預存、也不能回溯使用——叢發前 worker 閒掉的每一毫秒都永久蒸發了，所以叢發造成的等待是淨損失，不會被平均率「攤平」。

加分項：指出 D/D/1 是對照組（確定性世界裡 λ < μ ⇒ 零等待），所以等待的來源精確地說是**變異**，不是負載本身。常見錯路：「因為運氣不好」（沒有機制，不及格）；「因為瞬間 λ 超過 μ」（只講了成分 1——如果閒置可以儲存，瞬間超載也不會累積等待）。

## 自我檢核

口頭自答，講得出來才算過關：

1. 為什麼 λ < μ 的系統還會排隊？你的答案有沒有同時包含「局部叢發」與「閒置不可儲存」兩個成分？
2. D/D/1 為什麼完全不排隊？這對「延遲的真正來源是什麼」說了什麼？
3. 使用率從 0.8 升到 0.9，平均回應時間怎麼變？為什麼用「剩餘容量 1−ρ」當座標比用 ρ 誠實？
4. 使用率階梯的五個數字是在哪些假設下算出來的？你現在的系統最可能違反其中哪一個？
5. 「P99 ≈ 4.6× 平均」的適用條件是什麼？什麼樣的服務時間分布會讓它死掉？
6. Kendall 記號 M/G/1 的三個欄位各說了什麼？「G」承認了什麼樣的無知？
7. 舉兩個排隊理論不能回答的問題，並分辨：是「本書後面會教」還是「理論天生不管」？
