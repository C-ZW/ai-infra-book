# ch09 — M/M/c 與 Erlang 公式：池化的威力

> **本章解決什麼問題**：ch08 的 M/M/1 只有一台機器，但你的世界裡 worker 永遠是一群。本章把 ch07 的生滅引擎裝上 c 台伺服器：推出 M/M/c 的平穩分布與 Erlang-C（等待機率）、用手算友善的遞迴拿下 Erlang-B（損失機率），並順著 Erlang 在哥本哈根機房遇到的原始問題，引入「沒位子就走」的損失系統。然後結兩樁懸案：共用佇列為什麼贏過獨立佇列、一台快機什麼時候贏過多台慢機；最後用 √a 緩衝把「pool 該開幾台」從經驗值升級成定理。位子有限但還能排幾個的中間地帶，ch10 接手。

## 從你已知的出發

你天天在做本章的決策，只是用的是儀表板而不是公式。SQS 的 consumer 要開幾個？autoscaling 的目標值要釘在哪？3,200 條 WebSocket 並發連線背後該配幾台推播 worker？你的做法大概是：抓歷史尖峰、乘個安全係數、上線盯 backlog，不夠再加。能動，但每一步都是經驗值——這章把它換成能推導、能質疑的東西。

更重要的是一個你可能沒意識到自己做過的架構決策。「一條 SQS、四個 consumer 搶著領」與「按 user id hash 到四條 queue、各配一個 consumer」，在架構圖上只差一個分流器。本章會給這個差別標上精確的價格：同樣四台機器、同樣的總流量，hash 分流的平均延遲是共用佇列的 2.9 倍（0.5 s 對 0.175 s）。順序保證與 affinity 從來不是免費的，只是帳單以前沒寄到你手上。

第三條線接 ch04 與 ch08。ch04 結尾算過「兩台 worker 全忙、你排第一」：你的等待是兩路殘餘指數的競賽、Exp(2μ)，當時就預告這個結構是本章的骨架。ch08 最後埋的陷阱題——流量翻倍、機器也翻倍，延遲會怎樣——也在本章結案，而答案會讓「使用率不要超過 80%」這條經驗法則的適用範圍開始動搖。

## 把生滅引擎裝上 c 台機器：M/M/c

模型設定：Poisson(λ) 到達、每台機器的服務時間獨立同分布 Exp(μ)、c 台同質機器共用**一條**佇列，誰空誰拿。狀態取「系統內人數 n」（含服務中），它構成生滅過程（ch07），只要把死亡率寫對：

- n ≤ c 時，n 台機器各自忙著一個工作，下一次完工是 n 路獨立 Exp(μ) 的競賽——競賽引理（ch04）：min ~ Exp(nμ)。
- n ≥ c 時，只有 c 台在做事，排隊的人不貢獻離開率，死亡率卡在 cμ。

合起來：μₙ = min(n, c)·μ，λₙ ≡ λ。rate diagram：

```text
      λ          λ          λ          λ          λ          λ
   ───────►  ───────►  ───────►  ───────►  ───────►  ───────►
 0          1          2          ⋯        c         c+1        ⋯
   ◄───────  ◄───────  ◄───────  ◄───────  ◄───────  ◄───────
      μ         2μ         3μ         cμ         cμ         cμ
```

代入 ch07 的生滅通解 πₙ = π₀·Π(λᵢ/μᵢ₊₁)（i 從 0 到 n−1），分兩段（完整推導）：

```text
n ≤ c：分母 = μ·2μ·…·nμ = n!·μⁿ
       πₙ = π₀ · (λ/μ)ⁿ/n! = π₀ · aⁿ/n!          ← a = λ/μ，offered load
n ≥ c：超過 c 之後每一步都除以 cμ
       πₙ = π_c · ρⁿ⁻ᶜ = π₀ · (aᶜ/c!) · ρⁿ⁻ᶜ      ← ρ = λ/(cμ) = a/c
```

正規化（幾何級數收斂要求 ρ < 1，這就是 ergodicity 條件，ch07）：

```text
1/π₀ = Σ aⁿ/n!（n = 0 到 c−1） + (aᶜ/c!)·1/(1−ρ)
```

這個分布是個混血兒：c 以下是 Poisson 形的權重（待會 M/M/∞ 會解釋為什麼）、c 以上是公比 ρ 的幾何尾（M/M/1 的形狀，ch08）：

```text
 πₙ
  │     *  *
  │   *      *
  │ *          *               ← n ≤ c：Poisson 形（像 M/M/∞）
  │              *
  │               *  *         ← n > c：幾何尾，每步乘 ρ（像 M/M/1）
  │                     *  *
  │                           *   *
  └──────────────┴────────────────────►  n
  0              c
```

兩個符號的分工值得停一秒。**a = λ/μ 是工作量**：由 Little's Law 套在「服務中」這一段（ch02 的 E[N_服務中] = ρ 的多機版），平均忙碌台數 = λ·E[S] = a——所以 a 的單位叫 erlang，讀作「平均需要幾台機器的活」。**ρ = a/c 是使用率**：每台機器平均忙碌的時間比例。穩態條件 ρ < 1 就是 c > a：開的機器要多過工作量，剩下的問題只是「多多少」——本章最後一節的主題。

適用條件記在這裡，後面每個公式都繼承：Poisson 到達、指數服務、c 台同質、共用單一佇列、穩態 ρ < 1。其中 πₙ 對「排隊的人誰先上」不敏感（只要不偷看工作長度、不讓機器空等），但等待時間的**分布**推導用 FCFS。

順帶回答一個 ch06 訓練過的問題：憑什麼狀態只記人數 n 就夠？兩個 M 各自無記憶——到達端不必記「離上次到達多久」、服務端不必記「每台已經做了多久」，所以 n 一個數字就裝下了未來需要的一切；連「忙的是哪幾台」都不用記，因為機器同質。這個狀態選擇在服務不是指數時整個崩潰（人數不再吃得下歷史），那是 ch12 的開場白。

## Erlang-C：你會不會等、等多久

到達的你需要排隊，當且僅當抵達瞬間 N ≥ c。到達者眼中的分布等於時間平均——這是 PASTA（ch08），Poisson 到達才有的特權：

```text
P(等待) = P(N ≥ c) = Σ πₙ（n ≥ c） = π_c·(1 + ρ + ρ² + …) = π_c/(1−ρ)
```

代入 π_c，得到本章第一個招牌公式 **Erlang-C**（延遲機率）：

```text
                  aᶜ/c! · 1/(1−ρ)
C(c, a) = ─────────────────────────────────────── ，Σ 取 n = 0 到 c−1
           Σ aⁿ/n!  +  aᶜ/c! · 1/(1−ρ)
```

先做兩個極限的體檢（自我複核的習慣）：c = 1 時分母只剩 1 + ρ/(1−ρ)，整式收成 C = ρ——正是 M/M/1 的 P(N ≥ 1) = ρ，舊結果是新公式的特例 ✓；反過來固定 a、讓 c → a⁺（機器剛好夠用），1/(1−ρ) 那一項炸大、C → 1，人人都得等 ✓。公式在兩端都接上了直覺，中間的值才值得信。

光知道會不會等不夠，要知道等多久。以下是全章最值得抄一遍的推導（完整推導），它把 ch03–ch08 的工具各用一次：

**第一步**：已知要等（N ≥ c），眼前佇列長度的條件分布。πₙ 在 n ≥ c 段是幾何的，條件化之後還是幾何：P(排在你前面有 j 人 | 等待) = (1−ρ)ρʲ。

**第二步**：等待期間 c 台全忙（你還在等 ⇒ 你前面有人 ⇒ 沒有機器閒著），完工流是 c 路 Exp(μ) 的競賽，無記憶性讓它成為速率 cμ 的 Poisson 流（ch04 + ch05）。前面有 j 人時，你要等 **j+1 次完工**才輪到你開始服務。

**第三步**：把幾何混合的 Erlang 和算出來。關鍵一招是 ch05 的計數對偶——「第 j+1 次完工晚於 t」等價於「t 內完工次數 ≤ j」：

```text
P(T_Q > t | 等待)
  = Σⱼ (1−ρ)ρʲ · P(t 內完工 ≤ j 次)                ← j 從 0 到 ∞
  = Σₖ P(t 內完工 k 次) · Σⱼ≥ₖ (1−ρ)ρʲ             ← 交換求和順序
  = Σₖ e^(−cμt)·(cμt)ᵏ/k! · ρᵏ                      ← 內層幾何和 = ρᵏ
  = e^(−cμt) · e^(cμρt)
  = e^(−cμ(1−ρ)t) = e^(−(cμ−λ)t)
```

條件等待是乾淨的指數：T_Q |等待 ~ Exp(cμ−λ)，跟 M/M/1 的 T ~ Exp(μ−λ) 同一個家族，只是速率換成「總完工率減總到達率」。於是全套指標出列：

```text
E[T_Q] = C(c,a) · 1/(cμ−λ)                ← 等的機率 × 等的人平均等多久
P(T_Q > t) = C(c,a) · e^(−(cμ−λ)t)        ← 等待的 tail，FCFS
E[T]   = E[T_Q] + 1/μ
E[N_Q] = λ·E[T_Q] = C(c,a)·ρ/(1−ρ)        ← Little 套佇列
E[N]   = E[N_Q] + a                        ← Little 套服務段
```

每一步的工具來歷：PASTA 換掉「到達者視角」、幾何級數（ch03）收尾、競賽引理（ch04）給 cμ、計數對偶（ch05）拆 Erlang 和、Little（ch02）把時間與人數互換。這條推導鏈是紙上推演第 3 題的考題。

## 哥本哈根的機房：損失系統與 Erlang-B

1908 年，A.K. Erlang 以科學協作者的身分加入哥本哈根電話公司，主持新成立的物理技術實驗室，領到的題目是把機率論用在「電話呼叫的等待問題」上。具體的工程問題長這樣：兩個交換局之間拉 c 條中繼線，銅線昂貴，開太多浪費、開太少接不通——而當 c 條全忙時，新呼叫聽到忙音、直接消失。**沒位子就走、不排隊**：這類系統叫**損失系統（loss system）**，跟前面「無限排隊」正好是光譜的兩端（位子有限但能排幾個的中間地帶，連同丟棄與逾時的權衡，見 ch10）。

Erlang 1909 年先證明隨機呼叫流服從 Poisson 分布並部分解決延遲問題；1917 年給出損失公式與等待時間公式——就是本章的 Erlang-B 與 Erlang-C——很快被各國電話公司採用，包括英國郵政總局。1946 年，國際標準機構把話務量單位命名為 erlang 以紀念他（a 的單位就是這麼來的）。誠實註記：說「Erlang 發明排隊理論」並不精確——他解的是電話工程的具體問題；排隊理論作為一般化學科，是 1950 年代 Kendall（1953 記號）、Lindley（1952）等人建制起來的。

損失系統的模型是 **M/M/c/c**（Kendall 記號第四欄：系統容量也是 c）：生滅鏈砍在 c——λₙ = λ（n < c）、滿了到達率歸零；μₙ = nμ。通解同前段，只是正規化變有限和：

```text
πₙ = (aⁿ/n!) / Σ aᵏ/k!（k = 0 到 c），  n = 0, 1, …, c
```

注意**沒有穩態條件**：a 再大都穩定，損失系統永遠活著，代價付在丟掉的呼叫上。被擋掉的呼叫比例 = 到達者看到全滿的機率 = π_c（又是 PASTA；它在 blocking 場景的精確用法 ch10 細談）。這就是 **Erlang-B**（損失機率）：

```text
B(c, a) = (aᶜ/c!) / Σ aᵏ/k!（k = 0 到 c）
```

直接算要疊階乘，c 一大，aᶜ 與 c! 在計算上雙雙爆掉（兩者相除才有限）。手算與程式都該用遞迴。令 I(c) = 1/B(c, a)，從定義出發（完整推導）：

```text
1/B(c,a) = Σ aᵏ/k! · c!/aᶜ（k = 0 到 c）
         = 1 + (c/a) · Σ aᵏ/k! · (c−1)!/aᶜ⁻¹（k = 0 到 c−1）   ← 拆出 k = c 項，餘項提出 c/a
         = 1 + (c/a) · 1/B(c−1, a)
```

也就是 **I(0) = 1、I(c) = 1 + (c/a)·I(c−1)**：每多一台機器，一乘一加。

### Worked example：a = 3.2 的手算全程 → Erlang-C → E[T_Q]

基準場景：尖峰 λ = 32 則/s、單 worker μ = 10 則/s，所以 a = 3.2 erlang——平均要 3.2 台機器的活，至少開 4 台才穩。先把 c/a 備好：1/3.2 = 0.3125、2/3.2 = 0.625、3/3.2 = 0.9375、4/3.2 = 1.25。遞迴跑四步：

```text
I(0) = 1
I(1) = 1 + 0.3125·1      = 1.3125    → B(1) = 0.762
I(2) = 1 + 0.625·1.3125  = 1.8203    → B(2) = 0.549
I(3) = 1 + 0.9375·1.8203 = 2.7065    → B(3) = 0.369
I(4) = 1 + 1.25·2.7065   = 4.3832    → B(4) = 0.228
```

讀法：如果推播 worker 是損失系統（滿了就丟），1 台會丟掉 76.2% 的訊息、4 台還是丟 22.8%——損失系統對容量非常飢渴，這個數字 ch10 會再回來鞭。

但我們的 worker 會排隊，要的是 Erlang-C。兩個公式分母只差一項，代數整理可得轉換式（完整推導）：

```text
記 S = Σ aᵏ/k!（k = 0 到 c）、x = aᶜ/c!，則 B = x/S，而
C = (x/(1−ρ)) / (S − x + x/(1−ρ))      ← C 的分母 = S 把第 c 項換成放大版
  = x / ((S−x)(1−ρ) + x)               ← 上下同乘 (1−ρ)
  = B / ((1−B)(1−ρ) + B)               ← 上下同除 S
  = B / (1 − ρ(1−B))
```

代入 ρ = 0.8、B(4, 3.2) = 0.228145：

```text
C(4, 3.2) = 0.228145 / (1 − 0.8·0.771855)
          = 0.228145 / 0.382516
          = 0.596
```

到達的訊息有 59.6% 要排隊。等多久？cμ − λ = 40 − 32 = 8/s：

```text
E[T_Q] = 0.596/8 = 74.6 ms
E[T]   = 74.6 ms + 100 ms = 175 ms
```

自我複核走另一條路：E[N_Q] = C·ρ/(1−ρ) = 0.596·4 = 2.39，E[N] = 2.39 + 3.2 = 5.59，Little 給 E[T] = 5.59/32 = 175 ms ✓。再給兩個 tail 視角：等的人平均等 1/8 s = 125 ms；P(T_Q > 0.5 s) = 0.596·e⁻⁴ ≈ 1.09%——平均看起來無害，但門檻一畫，百分之一的訊息等超過半秒，SLA 要不要收這個由你決定，數學只負責把它算出來。

最後一句話的預告：Erlang-B 還有個本章不展開的超能力——它對服務時間分布**不敏感（insensitive）**，M/G/c/c 的損失機率只透過 E[S] 依賴服務分布，直方圖長什麼樣都行（為什麼，見 ch13 的 insensitivity 與 ch18 的 BCMP）。Erlang-C **沒有**這個性質。這個不對稱是陷阱表的常客。

## M/M/∞：人人自帶一台機器

把 c 推到無限大：μₙ = nμ 對所有 n 成立、永遠不用排隊。通解的正規化變成 Σ aⁿ/n! = eᵃ（指數級數，ch03 的老朋友），於是：

```text
πₙ = e⁻ᵃ · aⁿ/n!     ——  N ~ Poisson(a)
```

系統內人數是不折不扣的 Poisson 分布：E[N] = a、Var(N) = a、T_Q ≡ 0、E[T] = E[S]，任何 λ 都穩定。這解釋了 M/M/c 分布前半段的 Poisson 形——c 還沒用完之前，系統的行為跟機器無限多沒有兩樣；也解釋了 M/M/c/c 的平穩分布為什麼是「截斷的 Poisson」。

它模什麼？所有「沒有資源競爭、即到即服務」的階段：3,200 條 WebSocket 並發連線（人數 = 連線數、「服務」= 連線存續時間，E[N] = λ·E[存續] 直接給你容量帳）、ch17 封閉網路裡的 think time 站。而且 M/G/∞ 同樣給 Poisson(a)——insensitivity 家族的另一員。

但 M/M/∞ 在本章真正的任務是當一把尺：**如果機器無限多，忙碌台數的平均是 a、標準差是 √a**。記住這個 √a，最後一節靠它吃飯。

## 池化三連比：獨立佇列、共用佇列、一台快機

先結 ch08 的懸案。流量翻倍、機器也翻倍（λ = 16、c = 2、μ = 10）：ρ 還是 0.8，但 E[T] 從 0.5 s 降到 0.278 s（C(2, 1.6) = 0.711，E[T_Q] = 0.711/4 = 178 ms）。**同樣的使用率，池子越大延遲越低。**這不是巧合，是本節與下一節的主旋律。

現在上主菜。同樣的總流量 λ = 32 則/s、同樣的總容量 40 則/s，三種架構：

- **(a) hash 分流四條獨立佇列**：4 × M/M/1，各 λ = 8、μ = 10——ch08 基準原樣。這裡有一步常被略過的嚴謹：憑什麼每條子流還是 Poisson？因為 hash 對到達流做的是機率分流，thinning 定理（ch05）保證每條子流是獨立的 Poisson(8)——「各是 M/M/1」不是近似，是定理。
- **(b) 共用佇列**：M/M/4，本章 worked example。
- **(c) 一台四倍快機**：M/M/1，λ = 32、μ = 40（E[S] = 25 ms）。

| 指標 | (a) 4 × M/M/1 | (b) M/M/4 共用佇列 | (c) 快機 M/M/1 |
|---|---|---|---|
| ρ | 0.8 | 0.8 | 0.8 |
| E[S] | 100 ms | 100 ms | 25 ms |
| P(等待) | 0.8 | 0.596 | 0.8 |
| E[T_Q] | 400 ms | 74.6 ms | 100 ms |
| E[T] | **500 ms** | **175 ms** | **125 ms** |
| E[N]（全系統） | 16 | 5.59 | 4 |

（(a)、(c) 的 P(等待) = P(N ≥ 1) = ρ，PASTA；(c) 的 E[T_Q] = ρ/(μ−λ) = 0.8/8，ch08。）

**(a) → (b)：池化的威力。**硬體一樣、使用率一樣，只改了拓撲，E[T] 砍 2.9 倍、E[T_Q] 砍 5.4 倍。浪費在哪被消滅了？獨立佇列容許「2 號機前排著五個人、3 號機在發呆」這種荒謬同時發生：(a) 中任一台機器閒著的機率是 0.2，平均有 0.8 台機器閒置，同時平均 12.8 人在排隊。共用佇列把「有人等且有機器閒」這個事件**邏輯上**消滅——不是調參調掉的，是結構上不可能。這正是「一條 SQS 多個 consumer」（天生共用佇列）與「按 user hash 拆四條 queue」（獨立佇列）的差價。真實的 load balancer（輪詢、隨機）落在兩者之間；join-the-shortest-queue 已非常接近 (b)，而「隨機問兩台、選短的」這手便宜魔法留給 ch21。

**(b) vs (c)：快機的勝負帳。**快機總回應贏（125 對 175 ms），但拆開看：E[T] = E[T_Q] + E[S]，快機是 100 + 25，M/M/4 是 74.6 + 100。**等待是 M/M/4 短**——四台一起清佇列，59.6% 的人要等對上 80%；快機贏在服務本身快四倍，把省下的 75 ms 服務時間拿去抵銷多付的 25 ms 等待還有剩。在 M/M 世界裡這是通則：同 ρ 之下單台快機的 E[T] = 1/(cμ−λ)，永遠 ≤ M/M/c 的 C/(cμ−λ) + 1/μ。低負載時贏最多（沒人排隊，比的就是 E[S]，差距 c 倍）；ρ → 1 時兩者都被 1/(cμ−λ) 主宰，差距相對消失——ρ = 0.95 時是 0.5 s 對 0.546 s，只剩 9%。

**裁決：一台快機 vs 多台慢機。**指數服務、看平均、機器不壞：快機贏。但現實有三張扣分卡：

1. **故障域**。快機一掛是 100% 容量蒸發；pool 掛一台剩 75%——雖然在 a = 3.2 之下剩三台意謂 ρ > 1、得靠丟棄硬撐（ch10），但「降級」與「全黑」是兩種人生。還有部署：pool 能 rolling restart，單機只能停機或加倍買藍綠。
2. **搶占與變異數**。FCFS 快機上一個 30 秒的巨型工作會 head-of-line 堵死後面所有人；pool 裡它只佔走 1/c 的容量，短工作從旁邊流走。指數服務把這個風險定價得很溫和，重尾的現實（ch15）會狠狠放大它。反過來說，若快機能搶占（PS／time-slicing），優勢又被它搶回去——這場恩怨 ch13 開庭。
3. **採購現實**。四倍快的機器常常不存在、或貴得超線性；scale-out 是唯一能一直走下去的路。

一句話版本：**平均延遲投快機，工程現實投 pool；而只要你用了 pool，請務必共用佇列。**

## √a 緩衝：square-root staffing 與 QED

工作量 a 已知，c 該開幾台？兩個直覺答案都錯：c = a 是 ρ = 1，爆炸（ch08）；c = a/0.8——「使用率不要超過 80%」的線性思維——對小 pool 浪費感受不深，對大 pool 是燒錢。正確的縮放是：

```text
c ≈ a + β·√a        （β 是你選的服務品質旋鈕，常在 1 上下）
```

為什麼是 √a？直覺版推導（本書不證，嚴格版指路）：用 M/M/∞ 那把尺——機器無限多時，忙碌台數 ~ Poisson(a)，平均 a、標準差 √a，a 大時近似常態。需求的自然波動是 **√a 級**，不是 a 的固定比例；開 a + β√a 台，就是把波動的 β 個標準差擋在「全忙」之外。緩衝該按波動定價，而波動長得比工作量慢——這就是規模經濟的數學來源。

用精確的 Erlang-C 驗證這個直覺（β = 1，c = a + √a，全部重算）：

| a | c = a + √a | ρ = a/c | P(等待)（Erlang-C） |
|---|---|---|---|
| 4 | 6 | 0.667 | 0.285 |
| 25 | 30 | 0.833 | 0.250 |
| 100 | 110 | 0.909 | 0.237 |
| 400 | 420 | 0.952 | 0.230 |
| 10,000 | 10,100 | 0.990 | 0.225 |

等待機率釘在四分之一附近不動，使用率卻一路爬到 99%。這個現象的正式版本是 **QED regime**（Quality- and Efficiency-Driven）：Halfin & Whitt（1981）證明，當 c 與 λ 一起放大、保持 (1−ρ)√c → β（等價於 c ≈ a + β√a），P(等待) 收斂到一個只依賴 β、嚴格介於 0 與 1 之間的極限——上表的 0.285 → 0.225 正在逼近 β = 1 的極限（約 0.22）。品質與效率，√a 讓你兩個都拿。

歸屬的誠實註記（容易被誤傳）：√ 法則本身是電話工程的老經驗，可上溯至 Erlang 的年代；Halfin–Whitt 1981 給的是 many-server 極限**定理**；把「該開 a + β√a 台」當最佳化問題嚴格證成的是 Borst–Mandelbaum–Reiman 2004。β 怎麼選沒有通用值——它是「機器的錢」對「等待的痛」的匯率，等待越貴 β 越大。

對「80% 警戒線」的判決於是有了細則：它隱含的對手是小 pool。c = 4 時 ρ = 0.8 已經給你 59.6% 的等待機率，警戒有理；c = 10,100 時 ρ = 0.99 仍然體面。**使用率上限不是常數，是 pool 大小的函數**——pool 越小，每一分使用率越貴。完整結案在 ch20。

最後把適用條件釘上：Poisson 到達（尖峰時段內近似平穩——非平穩的 λ(t) 用 busy hour 流量餵 a，這是電話工程百年的老紀律）、同質機器、指數服務給出表中的精確值（一般服務下 QED 的結構仍在，數值修正的方向見 ch14）。

## 陷阱與防禦

| 陷阱 | 它怎麼騙你 | 防禦與自我察覺 |
|---|---|---|
| 有 c 台機器 ≠ M/M/c | 公式假設單一共用佇列、誰空誰拿；hash 分流其實是 c 個獨立 M/M/1，同樣硬體 E[T] 差 2.9 倍。拿 175 ms 的模型去規劃 500 ms 的架構，容量帳全錯 | 先看路由：consumer 主動領（拉式）≈ 共用佇列；hash／sticky 推式 ≈ 獨立佇列；輪詢落在中間 |
| 把 ρ 餵進 Erlang 公式 | 遞迴吃的是 a = λ/μ（erlang 數），不是 ρ = a/c；餵 0.8 算出來的是「0.8 erlang 小系統」的答案，c 越大錯得越離譜 | a 可以遠大於 1、ρ 必須小於 1——發現餵進公式的數字永遠小於 1，先停下來查單位 |
| Erlang-C 假設沒人放棄 | 真實呼叫者會掛斷、請求會 timeout：高負載下放棄是天然洩壓閥，Erlang-C 系統性高估壅塞，照它 staffing 會超編 | 監控裡有 abandonment／timeout 而模型裡沒有，模型就錯了一邊；M/M/c+M（Erlang-A）見 ch10 |
| 服務分布不是指數 | Erlang-C 對服務分布**敏感**：等待大致按 (1+C²ₛ)/2 縮放（ch12/ch14），C²ₛ = 10 時低估 5.5 倍；Erlang-B 反而 insensitive——兩條最容易記反 | 口訣：**loss 看均值就夠，delay 要看變異數**。動手前先量 C²ₛ（ch03） |
| B 與 C 拿錯場景 | 系統會排隊卻用 Erlang-B（低估壅塞——被「擋掉」的人其實在排隊堆積）；滿了就丟卻用 Erlang-C（在 a ≥ c 時根本算不出來，公式要求 ρ < 1，而損失系統偏偏常開在 a > c） | 先問一句：「滿了之後，工作去哪？」消失 → B；等待 → C；等一陣子放棄 → 都不對，ch10 |
| 「c 台同質」是假設 | 混 instance 世代、noisy neighbor、冷啟動中的 worker，讓有效 μ 參差：名目 c 台 ≠ 有效 c 台，公式給樂觀值 | 看 per-worker 吞吐直方圖；差異大時用保守的有效容量重算，或交給模擬（ch19） |
| 拿全日平均 λ 算 staffing | λ(t) 非平穩（ch05）：用平均流量算出的 c，在尖峰時段必塌 | 用最忙時段（busy hour）的 λ 當 a 的輸入；尖峰差異大就分時段 staffing |

## 紙上推演

**第 1 題 [15 分鐘 ★★]**：λ = 2、μ = 1（a = 2）。用遞迴手算 B(1, 2)、B(2, 2)、B(3, 2)，轉成 Erlang-C C(3, 2)，再算 E[T_Q]。對照：同容量單台快機（μ = 3）的 E[T_Q] 是多少？哪邊等得短？

**第 2 題 [10 分鐘 ★★]**：承 worked example（a = 3.2），SLA 要求 P(等待) < 0.2，至少要開幾台？把遞迴從 I(4) 繼續往下算。

**第 3 題 [15 分鐘 ★★★]**：闔上書，重講「M/M/c 條件等待 ~ Exp(cμ−λ)」的完整推導。要求指出：幾何分布從哪來、cμ 憑什麼在整段等待期間都成立、j+1 的 +1 是什麼、交換求和那一步在做什麼。

**第 4 題 [10 分鐘 ★★]**：「拆 pool 一定變差嗎？」用本章結果先給 M/M 世界的答案，再找出這個「一定」綁定的假設——什麼樣的工作組成會讓「拆出一個專用 pool」反而合理？（提示：ch13 的排程基準裡有 90% 短工作、10% 長工作。）

### 推演解答

**第 1 題**：c/a 依次 = 0.5、1、1.5。遞迴：

```text
I(0) = 1
I(1) = 1 + 0.5·1   = 1.5     → B(1) = 2/3 ≈ 0.667
I(2) = 1 + 1·1.5   = 2.5     → B(2) = 0.400
I(3) = 1 + 1.5·2.5 = 4.75    → B(3) = 4/19 ≈ 0.211
```

轉 Erlang-C（ρ = 2/3）：C = B/(1 − ρ(1−B)) = (4/19)/(1 − (2/3)(15/19)) = (4/19)/(9/19) = **4/9 ≈ 0.444**。E[T_Q] = C/(cμ−λ) = (4/9)/(3−2) = 0.444（單位是平均服務時間 1/μ）。快機 M/M/1（μ = 3、λ = 2）：E[T_Q] = ρ/(μ−λ) = (2/3)/1 = 0.667——**等待還是 pool 短**（0.444 對 0.667），與三連比的結論一致；快機贏的是 E[S] 那一段。常見錯路：把 ρ = 2/3 餵進遞迴（陷阱表第 2 條），或忘了 E[T_Q] 的分母是 cμ−λ 而不是 μ−λ。

**第 2 題**：繼續遞迴（c/a：5/3.2 = 1.5625、6/3.2 = 1.875）：

```text
I(5) = 1 + 1.5625·4.3832 = 7.849    → B(5) = 0.127
I(6) = 1 + 1.875·7.849   = 15.72    → B(6) = 0.0636
```

C(5, 3.2)：ρ = 0.64，C = 0.127/(1 − 0.64·0.873) = 0.127/0.442 = **0.289** > 0.2，不夠。
C(6, 3.2)：ρ = 0.533，C = 0.0636/(1 − 0.533·0.936) = 0.0636/0.501 = **0.127** < 0.2，達標。答案：**6 台**。順帶看邊際收益：c = 5 時 E[T_Q] = 0.289/18 = 16.0 ms、c = 6 時 = 0.127/28 = 4.54 ms——從 4 台的 74.6 ms 起，每加一台等待砍好幾倍，但 P(等待) < 0.2 這個門檻本身是 SLA 的選擇，不是數學的命令。

**第 3 題**：推導鏈四個零件。①幾何分布：πₙ 在 n ≥ c 段是公比 ρ 的幾何（生滅通解的直接後果），條件在「N ≥ c」上仍是幾何——P(前面 j 人 | 等待) = (1−ρ)ρʲ；到達者看得到這個分布，靠 PASTA。②cμ 全程成立：你還在等 ⇒ 你前面有人 ⇒ c 台全忙；無記憶性讓「全忙時的完工流」是速率 cμ 的 Poisson 流，與歷史無關（競賽引理）。③+1：第 j+1 次完工才輪到你**開始服務**——前面 j 人各離開一次、再加一台空出來給你；少算這個 +1 是最常見的錯。④交換求和：把「幾何個 Erlang 相加」翻成「對 Poisson 計數條件化」（第 j+1 次完工晚於 t ⟺ t 內完工 ≤ j 次），內層收成幾何和 ρᵏ，外層收成 e 的指數級數——兩個 ch03 的級數各用一次，指數分布就掉出來了。

**第 4 題**：M/M 世界（工作同質、同 μ）：拆 pool 一定變差——三連比 (a) 對 (b)，同硬體 E[T] 差 2.9 倍，因為拆開後「有人等且有機器閒」變得可能。但「一定」綁在**工作同質**上。若服務高變異（90% 短、10% 長），共用 pool 裡幾個長工作可能同時佔滿全部 c 台，短工作集體陪葬（ch12 的 inspection paradox 會量化這件事）；拆出「短工作專用 pool」等於用拓撲實作一個窮人版的優先權排程——犧牲池化效率，買回變異數隔離。誰划算取決於 C²ₛ 與負載：同質工作永遠共用；重尾混合工作，最好的解是「共用＋會看長度的排程」（ch13 的 SRPT 戰場），排程動不了手才用拆 pool 硬隔（ch15 結案）。
