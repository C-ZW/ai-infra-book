## 從你已知的出發

你值過那種班：一支月底報表 query 跑了 30 秒，整條 API 的延遲跟著飆。事後檢討時大家盯著那支 query 罵，但儀表板上有件事沒人解釋得了——**平均**服務時間幾乎沒動。一萬個請求裡混進幾支慢的，平均被稀釋得無影無蹤，可是 P99 已經翻了好幾倍。你的 SQS 結算 pipeline 也有同款劇情：批次有大有小，平均處理時間穩穩的 100 ms，等待卻在尖峰時段莫名其妙地堆。

ch03 已經給了這個現象一個名字：C²ₛ，服務時間的變異係數平方。那裡你算過 90% 短工作（50 ms）混 10% 長工作（550 ms）的 C²ₛ = 2.25，也從一張截斷的監控直方圖估出某服務的 C²ₛ ≈ 3.73。ch04 的陷阱表則開出了預告帳單：把 C²ₛ = 10 的現實硬套指數模型，等待會低估 5.5 倍。本章就是結帳的地方——你會親手推出那條公式，知道它每一項從哪裡來、什麼條件下成立、什麼時候會反過來騙你。

## M/G/1：佇列長度為什麼不再是完整狀態

記號先對齊（Kendall 記號的正式定義見 ch08）：M/G/1 ＝ Poisson(λ) 到達、服務時間 S 是任意分布的 i.i.d. 隨機變數（General）、單一伺服器、預設 FCFS、無限緩衝、ρ = λE[S] < 1。

M/M/1 能用一條生滅鏈解掉，靠的是 ch04 那句話：指數分布下「已經做了多久」對未來毫無資訊量，所以**一個整數**（系統內人數 N）就是完整狀態。現在服務換成一般分布，這句話反過來咬人：一支已經跑了 25 秒的報表 query，剩餘時間的分布跟剛開跑時完全不同。要描述系統，你得記 (N, a)——a 是服務中工作已進行的時間，一個連續量。狀態空間從可數變成連續，Part III 的生滅引擎熄火。ch04 證過指數是唯一無記憶的連續分布，所以這不是技術困難，是本質的：**只要服務不是指數，佇列長度就不夠當狀態**。

出路有兩條。第一條是 Kendall 1953 的 embedded chain：只在「工作離開的瞬間」觀察系統——那一刻 a 恰好歸零，N 在這些瞬間構成一條離散時間 Markov chain（他論文標題裡的 imbedded Markov chain 就是這個），可以解、但要對付一整套無窮維的方程。第二條是本書要走的 mean-value 路線：放棄解分布，只追平均——材料只需要 Little's Law（ch02）、PASTA（ch08），加上一個新角色：**殘餘服務時間**。代價是誠實的：這條路只給你平均值，等待的完整分布要靠 transform 方法（本章末節有它的存在聲明）。

## 殘餘服務時間：你總是撞上長工作

問題：你在一個隨機時刻抵達，伺服器正忙著某支工作，它還要多久才完？ch04 給過指數的答案——殘餘分布跟全新的一樣，期望仍是完整的 E[S]。當時也立了牌：一般分布是另一個世界。現在進這個世界。

直覺先行。隨機時刻抵達的你，不是在「工單清單」上均勻抽一支工作，而是在**時間軸**上均勻抽一個點——一支 2 秒的工作在時間軸上占的長度是 20 ms 工作的 100 倍，你撞上它的機率也大 100 倍。這叫 inspection paradox（檢視悖論）：**你撞上的工作，是被長度加權抽樣過的**，平均比 E[S] 長。ch04 推演裡那個公車站的第二層陷阱（「隨機抵達的人傾向落在偏長的間隔裡」）就是它。

把直覺變成數字，用一張圖。定義 r(t)：時刻 t 服務中工作的殘餘服務時間，伺服器閒置時 r(t) = 0。每支工作開始服務時，r 跳到它的長度 Sᵢ，然後以斜率 −1 滑回 0：

```
 r(t)  服務中工作的殘餘服務時間（閒置時 r = 0）
 ▲
 │\
 │ \
 │  \
 │   \        |\
 │    \       | \        |\
 │     \      |  \       | \
 └──────┴─────┴───┴──────┴──┴─────▶ t
     S₁ 完成   S₂ 完成    S₃ 完成
   面積 S₁²/2  面積 S₂²/2  面積 S₃²/2
```

r(t) 的長期時間平均，就是把鋸齒下的面積除以總時間。每支工作貢獻一個底為 Sᵢ、高為 Sᵢ 的直角三角形，面積 Sᵢ²/2。設 [0, T] 內完成了 M(T) 支工作：

```text
(1/T)∫₀ᵀ r(t) dt = (1/T) · Σᵢ Sᵢ²/2              ← 面積＝三角形相加（邊界那塊未完成的，T→∞ 後可忽略）
                 = (M(T)/T) · (Σᵢ Sᵢ²/2)/M(T)     ← 乘除一個 M(T)
                 → λ · E[S²]/2                     ← 穩態下完成率＝到達率（ρ<1）；樣本平均→期望值（大數法則）
```

嚴謹度標示：**工程師的嚴謹**——每一步都能口頭說出理由，但「邊界項可忽略」與「極限交換」沒有嚴格證（它們需要 E[S²] < ∞；這個條件 ch15 會正面炸掉）。教科書用 renewal-reward 語言把這個論證寫嚴格，見延伸閱讀。

所以**時間平均的殘餘服務時間是 E[r̄] = λE[S²]/2**（含閒置時段的 0）。只看忙的時段，就是把面積除以忙的總時間 ΣSᵢ：

```text
E[r | 忙] = (Σᵢ Sᵢ²/2)/(Σᵢ Sᵢ) → E[S²]/(2E[S]) = E[S]·(1 + C²ₛ)/2
```

最後一步用了 E[S²] = Var(S) + E[S]² = E[S]²(1 + C²ₛ)——ch03 的定義直接代入。這條公式值得拆開讀：E[S²]/(2E[S]) = (E[S²]/E[S]) × (1/2)。第一個因子 E[S²]/E[S] 是**你撞上的那支工作的平均長度**（長度加權後的平均），第二個因子 1/2 是「平均撞在中點」。兩個偏差，一個把工作變長、一個砍半，乘起來才是殘餘。

對帳三連發，均值全部釘在 E[S] = 100 ms：

- **確定性**：E[S²] = E[S]²，殘餘 = E[S]/2 = 50 ms。「做到一半」的直覺**只在這裡**成立。
- **指數**：E[S²] = 2E[S]²，殘餘 = E[S] = 100 ms——ch04 的「殘餘＝全新」被一般公式收編為特例。
- **二點混合（90% × 50 ms ＋ 10% × 550 ms，ch13 的排程基準）**：E[S²] = 0.9·2,500 + 0.1·302,500 = 32,500 ms²，殘餘 = 32,500/200 = **162.5 ms**——比一支全新工作的平均還長。

第三個例子值得多看一眼，它就是 inspection paradox 的數值現場。長工作只占工單的 10%，但占時間軸的比例是 0.1×550/100 = **55%**——你超過一半的機率撞上它。你撞上的工作平均長 E[S²]/E[S] = 325 ms，是真平均的 3.25 倍。交叉驗算（全機率定理，ch03）：0.55 × (550/2) + 0.45 × (50/2) = 151.25 + 11.25 = 162.5 ms ✓，兩條路同帳。監控上「平均服務時間 100 ms」沒有說謊，但對隨機抵達的你而言，世界由長工作主宰。

## Pollaczek–Khinchine：把帳單算出來

歷史一句：Pollaczek 1930 用複變分析先得到這個結果，Khinchine 1932 用機率語言重新推導，兩人沒有合作，公式因此掛雙名（拼法 Khinchin、Khintchine 亦通行）。我們走的 mean-value 路線比兩位原版都輕——因為 Little 和 PASTA 都是後來才有的工具。

標記一位乘客（tagged customer）：穩態下隨機抵達的你。FCFS 之下，你的等待由兩塊組成——服務中那支工作的殘餘，加上佇列裡排在你前面每一位的**完整**服務：

```text
E[T_Q] = E[r 你所見] + E[N_Q 你所見] · E[S]   ← FCFS：先等殘餘，再等前面每位做完
       = λE[S²]/2   + E[N_Q] · E[S]           ← PASTA（ch08）：Poisson 到達看到的＝時間平均
       = λE[S²]/2   + λE[T_Q] · E[S]          ← Little 用在佇列上（ch02）：E[N_Q] = λE[T_Q]
       = λE[S²]/2   + ρ · E[T_Q]              ← ρ = λE[S]

⇒ E[T_Q](1 − ρ) = λE[S²]/2

⇒ E[T_Q] = λE[S²]/(2(1−ρ))                    ← Pollaczek–Khinchine 平均等待公式
```

第一行有個容易滑過去的細節，值得停下來：為什麼佇列中每位貢獻的是**不偏的** E[S]，而服務中那位要用被長度加權的殘餘？因為佇列裡的工作**還沒開始跑**——它們的長度還沒在時間軸上占據任何位置，對你而言就是一批新鮮的 i.i.d. 抽樣；而且 FCFS 不看長度出工，佇列長度與這些工作的長度互相獨立，所以期望可以直接相乘（這步的獨立性我們口頭論證、不嚴格證）。只有服務中那一位被時間軸偏置污染。把這兩種身分搞混，正是本章推演第 4 題那個錯誤論證的死法。

代入 E[S²] = E[S]²(1 + C²ₛ)，得到本書接下來三章都會反覆引用的形狀：

```text
E[T_Q] = ρ/(1−ρ) · E[S] · (1 + C²ₛ)/2
          ───┬──    ─┬──    ───┬─────
          使用率     時間      變異性
          槓桿       尺度      槓桿
```

三個因子各管一件事。ρ/(1−ρ) 是 ch08 的 hockey stick，原封不動地活了下來——放掉指數假設，爆炸的形狀不變。E[S] 定尺度。新角色是 (1 + C²ₛ)/2：**變異數本身就是延遲的來源，C² 是它的計價單位**。ch14 的 Kingman 近似會把到達端的 C²ₐ 也接進來、變成 (C²ₐ + C²ₛ)/2，而 P-K 正是它在 Poisson 到達（C²ₐ = 1）時的精確特例。

兩個立即的健全性檢查。C²ₛ = 1（指數服務）：E[T_Q] = ρE[S]/(1−ρ)，基準數字下 = 0.4 s，與 ch08 的 M/M/1 完全吻合 ✓。C²ₛ = 0（確定性服務）：因子變成 1/2——**M/D/1 的等待恰好是同參數 M/M/1 的一半**。這句金句的精確內容是：把服務時間標準化到完全一致，你能拿回的等待改善就是這一半，不多不少；剩下那一半來自 Poisson 到達的隨機性，不關服務的事。

### 適用條件：哪一步用掉了什麼

這張表是本章的靈魂——P-K 不是「M/G/1 的公式」，是「這五個條件的公式」：

| 條件 | 在推導的哪一步被用掉 | 破掉時找誰 |
|---|---|---|
| Poisson 到達 | PASTA 那步：「你所見＝時間平均」 | ch14（Kingman，C²ₐ 旋鈕） |
| FCFS（見下方放寬） | 分解式「殘餘＋前面每位的完整服務」 | ch13（排程） |
| 服務 i.i.d.、與到達獨立、不看長度出工 | 佇列中每位貢獻不偏的 E[S] | ch13 |
| ρ < 1、穩態長期平均 | 完成率＝到達率；Little | ch10（過載靠丟棄）、ch20（動態） |
| E[S²] < ∞ | 面積論證的三角形加總 | ch15（重尾） |

FCFS 那條要說精確，因為它常被講錯。推導本身用了 FCFS，但結論的**平均值**適用範圍更寬：在「非搶占、work-conserving、不看工作長度」的出工紀律家族裡——FCFS、非搶占 LCFS、隨機抽人——伺服器忙閒的時間軸完全相同（誰先誰後不影響「現在有沒有人在做事」），系統內人數過程 N(t) 的統計規律因此一樣，於是 E[N_Q] 一樣，Little 一除，E[T_Q] 一樣。**平均等待是這個家族的紀律不變量；等待的分布不是**——LCFS 的等待變異數遠大於 FCFS，倒霉的人可以等非常久。一旦紀律看長度（SJF、SRPT）、會搶占（PS）、或分優先級，連平均都變——那是 ch13 的主場，那裡的守恆律會把「總量不變、分配可選」講到底。

### Worked example：基準三連比——同一個平均，三張帳單

推播管線基準：λ = 8/s、E[S] = 100 ms、ρ = 0.8，全程不動。只動服務時間的形狀，C²ₛ ∈ {0, 1, 10}。

C²ₛ = 10 的那檔需要一個具體分布。用二點分布：保留短工作 50 ms，設長工作長度 L、占比 q，解兩條帳：

```text
E[S]  = (1−q)·50 + q·L  = 100        ← 均值釘住
E[S²] = (1−q)·2,500 + q·L² = 110,000  ← 110,000 = E[S]²(1+C²ₛ) = 10,000 × 11

⇒ q = 1/41 ≈ 0.0244、L = 2,100 ms
驗算：E[S]  = (40×50 + 2,100)/41 = 4,100/41 = 100 ✓
      E[S²] = (40×2,500 + 2,100²)/41 = 4,510,000/41 = 110,000 ✓
      C²ₛ  = (110,000 − 10,000)/10,000 = 10 ✓
```

ch03 末尾為了示範「C²ₛ = 10 做得到」推過同一組參數；本章把它升格為**基準分布**：50 ms（機率 40/41）／2,100 ms（機率 1/41）。翻成人話：**2.4% 的請求各拖 2.1 秒**，C²ₛ 就是 10——你的系統裡幾乎一定有這種請求。

三檔全算（E[T_Q] = λE[S²]/(2(1−ρ)) = 8·E[S²]/0.4 = 20·E[S²]，E[S²] 用秒²）：

| C²ₛ | 分布 | E[S²] | E[r│忙] | E[T_Q] | E[T] | E[N_Q] | E[N] |
|---|---|---|---|---|---|---|---|
| 0 | 確定性 100 ms | 0.01 s² | 50 ms | **0.2 s** | 0.3 s | 1.6 | 2.4 |
| 1 | 指數，均值 100 ms | 0.02 s² | 100 ms | **0.4 s** | 0.5 s | 3.2 | 4.0 |
| 10 | 二點 50 ms／2,100 ms | 0.11 s² | 550 ms | **2.2 s** | 2.3 s | 17.6 | 18.4 |

（E[T] = E[T_Q] + E[S]；E[N_Q] = λE[T_Q]、E[N] = E[N_Q] + ρ，都是 Little，ch02。）

把這張表貼在心裡：三條管線的平均服務時間都是 100 ms、使用率都是 0.8、QPS 都一樣——**監控大盤上它們長得一模一樣**——等待差 11 倍，佇列裡平均囤的工作從 1.6 個變成 17.6 個。「一支報表 query 拖垮整條 API」的肇事機制現在有名字了：它不是吃掉了多少容量（ρ 沒變），是它把 E[S²] 撐爆了。

## Transform 方法與 busy period：地圖上的兩個鄰居

**等待的分布在哪裡？** mean-value 路線只給平均。完整分布有路可走：把分布打包成 Laplace–Stieltjes transform（LST）S̃(s) = E[e^(−sS)]，則 FCFS 的 M/G/1 等待時間的 LST 有封閉形式：

```text
W̃(s) = (1−ρ)s / (s − λ(1 − S̃(s)))     ← P-K transform 公式（本書不推導）
```

從它可以擠出所有矩（等待的變異數、偏度），數值反演可得整條分布與 tail。注意：**這條公式對紀律敏感，FCFS 是必要的**——上一節說的「平均是紀律不變量」到分布層級就失效了。本書不走這條路；想看完整處理，延伸閱讀的 Adan & Resing 與 Kleinrock 都有。直覺版（本書不證）：它存在這件事本身就值錢——意思是 M/G/1 的等待分布原則上完全可知，不必模擬。

**Busy period（忙碌期）**：從一支工作抵達空系統開始、到系統再次清空為止的時段。它是伺服器視角的自然單位——「這波要多久才消化完」。系統在閒（idle）與忙（busy）之間交替：閒期是「等下一個 Poisson 到達」，無記憶（ch04），E[閒] = 1/λ；長期忙的時間比例是 ρ（ch02 的 Utilization Law）。自我一致地解：

```text
ρ = E[B]/(E[B] + 1/λ)   ← 忙的比例＝忙期長度占循環的比例（工程師的嚴謹：長期平均的比值）
⇒ E[B] = E[S]/(1−ρ) = 1/(μ−λ)
```

基準數字：E[B] = 0.1/0.2 = **0.5 s**，每個忙碌期平均消化 1/(1−ρ) = 5 支工作——而且**三種分布都一樣**。平均忙碌期對 C²ₛ 完全盲目，這乍看跟整章的主旋律打架。沒有打架：帳單藏在分布裡。直覺版（本書不證）：一支長度 x 的工作開頭的忙碌期，它服務期間平均招來 λx 支新工作，每支又各自展開一段平均 E[B] 的「子忙碌期」，自我一致地解出 E[B | 開頭是 x] = x/(1−ρ)。基準的那支 2,100 ms 長工作一進門，期望就是一段 2.1/0.2 = **10.5 秒**不間斷的忙碌——你的 30 秒報表 query 按同款公式是 150 秒。平均忙碌期不動，但忙碌期的變異數隨 C²ₛ 爆漲：絕大多數忙碌期短促（一支 50 ms 工作、沒人跟上，0.25 s 就收工），偶爾一段超長——P99 的等待全部囤積在那些超長忙碌期裡。忙碌期的完整分布同樣有 transform 刻畫（Kendall functional equation：φ(s) = S̃(s + λ − λφ(s))，存在聲明，本書不解）。

## 陷阱與防禦

| 陷阱 | 它怎麼騙你 | 防禦 |
|---|---|---|
| 只量平均服務時間就建模 | C²ₛ 在平均裡不可見：把 C²ₛ = 10 的現實當指數（C²ₛ=1）算，預測 0.4 s、實際 2.2 s，低估 5.5 倍；當確定性算低估 11 倍 | 量 E[S²] 或完整直方圖；用 ch03 的直方圖估法時記得結果是**下界** |
| 到達不是 Poisson 卻套 P-K | P-K 沒有 C²ₐ 旋鈕——它硬寫死 C²ₐ = 1。上游是批次/retry 叢發（C²ₐ > 1）時低估；上游被 rate limiter 整形過（C²ₐ < 1，ch10）時高估 | 先量到達間隔的 C²ₐ；離 1 遠就換 ch14 的 Kingman |
| 出工紀律不在「非搶占＋不看長度」家族 | 系統裡藏著優先級、搶占、或 worker 偷看工作大小，P-K 連平均都不再成立 | 確認消費端真正的出工順序；有 class 之分就用 ch13 的 priority 公式 |
| 重尾服務 | Pareto α ≤ 2 時 E[S²] = ∞，P-K 回答「∞」——這不是公式壞掉，是真警報；更陰的是有限樣本估出的 E[S²] 永遠偏低且不收斂，你以為 C²ₛ = 12，其實沒有上限 | 看見直方圖右尾掛著就提高警覺；ch15 整章處理 |
| 拿穩態公式套尖峰動態 | P-K 是長期平均：尖峰剛起或 ρ ≥ 1 的時段裡它沒有定義的意義 | 確認量測窗口處在近似穩態；過載與動態見 ch10/ch20 |

自我察覺的徵兆：**ρ 沒動、平均服務時間沒動，等待卻在爬**——這個組合在 M/M/1 的世界裡無法解釋，在 P-K 的世界裡只有一個嫌犯：C²ₛ 漂移了。新上線的慢 endpoint、變大的批次、開始超時重試的下游，都會在平均裡隱身、在 E[S²] 裡現形。把 E[S²]（或 P99 服務時間）放上儀表板，比再多加三條平均值曲線都有用。

## 紙上推演

**推演 1 [15 分鐘] ★★ — P-K 推導重講**
闔上書，從「標記一位乘客」開始把 E[T_Q] = λE[S²]/(2(1−ρ)) 推到底。要求：說出推導用掉的四個材料（分解式、PASTA、面積論證、Little），並指出——如果出工紀律換成 SJF，哪一步先斷？如果到達換成 cron 等距觸發，哪一步先斷？

**推演 2 [10 分鐘] ★ — E[S²] 計算兩題**
基準管線 λ = 8/s、ρ = 0.8 不變，算 E[T_Q]：
(a) 服務時間均勻分布在 [50, 150] ms；
(b) 服務時間是 ch13 的排程基準混合：90% × 50 ms ＋ 10% × 550 ms。

**推演 3 [10 分鐘] ★★ — 均值不變、變異數砍半**
維持 E[S] = 100 ms、ρ = 0.8，把 Var(S) 砍半。等待改善幾 %？分別對 C²ₛ = 10、1、0.1 的起點回答，並用一句話總結「壓變異數」這根槓桿什麼時候值得拉。

**推演 4 [15 分鐘] ★★★ — 找出錯誤論證的破綻**
審這段話：「佇列裡平均有 E[N_Q] 個人，每人平均要 E[S]；服務中那位平均也要 E[S]，但平均已經做到一半，所以再等 E[S]/2。故 E[T_Q] = E[N_Q]·E[S] + ρ·E[S]/2。」把這個式子解到底，看看它「推導」出了什麼公式；指出錯在哪一步、錯的本質是什麼假設。

### 推演解答

**推演 1。** 主鏈：

```text
E[T_Q] = E[r 所見] + E[N_Q 所見]·E[S]   ← 材料一：FCFS 分解（殘餘＋前面每位的完整服務）
所見 → 時間平均                          ← 材料二：PASTA（Poisson 到達）
E[r̄] = λE[S²]/2                         ← 材料三：鋸齒面積論證（需 E[S²] < ∞、穩態完成率 = λ）
E[N_Q] = λE[T_Q]                         ← 材料四：Little 用在佇列
⇒ E[T_Q] = λE[S²]/2 + ρE[T_Q] ⇒ E[T_Q] = λE[S²]/(2(1−ρ))
```

換 SJF：材料一先斷——排在你前面的不再是「先到的人」，後到的短工作會插隊，分解式整個重寫；同時「佇列長度與工作長度獨立」也死了（SJF 出工順序就是看長度）。換 cron 等距到達：材料二先斷——等距到達者看到的不是時間平均（ch08 用等距到達當過 PASTA 的反例），而且面積論證裡「完成率 = λ」雖然還活著，E[r 所見] ≠ E[r̄] 已足以讓答案錯掉。常見錯路：以為 cron 到達會讓面積論證本身失效——不會，r(t) 的時間平均照樣是 λE[S²]/2，死的是「你看到的＝時間平均」這座橋。

**推演 2。** (a) Uniform[50, 150]：Var = (150−50)²/12 = 833.3 ms²，E[S²] = 10,000 + 833.3 = 10,833 ms²，C²ₛ = 0.0833。E[T_Q] = 0.2 × (1 + 0.0833) = **0.217 s**——離 M/D/1 的 0.2 s 不遠，±50% 的均勻抖動只貴 8%。(b) 混合：E[S²] = 32,500 ms²（本章算過），C²ₛ = 2.25，E[T_Q] = 0.2 × 3.25 = **0.65 s**。這個數字記住——ch13 會拿同一條管線問：不動硬體、只動出工順序，能把 0.65 s 壓到多少。

**推演 3。** E[T_Q] ∝ 1 + C²ₛ，Var 砍半 ⇒ C²ₛ 砍半，改善比例 = (C²ₛ/2)/(1 + C²ₛ)。C²ₛ = 10：5/11 ≈ **45.5%**（2.2 s → 1.2 s）；C²ₛ = 1：**25%**（0.4 s → 0.3 s）；C²ₛ = 0.1：**4.5%**（幾乎白忙）。總結：變異數槓桿的收益跟你的叢發程度成正比——系統越亂，整治越值錢；已經接近確定性的系統，這根槓桿沒肉，去拉 ρ 槓桿（兩根槓桿的正式比價是 ch14 的 worked example）。

**推演 4。** 把錯式解到底：E[T_Q] = λE[T_Q]·E[S] + ρE[S]/2 ⇒ E[T_Q](1−ρ) = ρE[S]/2 ⇒ E[T_Q] = ρE[S]/(2(1−ρ))——它「推導」出了 **M/D/1 的公式**（基準下 0.2 s）。錯在「平均已經做到一半，所以再等 E[S]/2」：這句對服務中那位用了**不偏的**平均長度、再砍半，完全忽略 inspection paradox——你撞上的工作平均長 E[S²]/E[S]，不是 E[S]。「做到一半」的直覺恰好等價於假設服務是確定性的（本章對帳三連發的第一條），所以這個論證在 D 的世界裡碰巧全對，在指數世界低估 2 倍，在 C²ₛ = 10 的世界低估 11 倍。錯誤論證最危險的形態就是這種：有一個世界讓它碰巧正確。

### Python 小實驗

事件驅動模擬，骨架沿用 ch08 的 DES（event list ＋ 模擬時鐘），只把服務時間換成三種分布；只用標準函式庫。把前 20,000 筆等待丟掉，是為了沖掉「從空系統起跑」的偏差——這手法的正式名字與紀律見 ch19。

```python
import heapq
import random

random.seed(42)
LAM, N_JOBS, WARMUP = 8.0, 200_000, 20_000

def svc_d():  # deterministic: C2s = 0
    return 0.1
def svc_m():  # exponential: C2s = 1
    return random.expovariate(10.0)
def svc_b():  # two-point: 50 ms w.p. 40/41, 2100 ms w.p. 1/41 -> C2s = 10
    return 0.05 if random.random() < 40 / 41 else 2.1

def mg1_mean_wait(svc):
    """Event-driven M/G/1 FCFS; reuses the ch08 DES skeleton."""
    events = [(random.expovariate(LAM), "arr")]  # (time, kind) min-heap
    queue, waits, busy, done = [], [], False, 0
    while done < N_JOBS:
        t, kind = heapq.heappop(events)
        if kind == "arr":
            heapq.heappush(events, (t + random.expovariate(LAM), "arr"))
            if busy:
                queue.append(t)
            else:
                busy = True
                waits.append(0.0)
                heapq.heappush(events, (t + svc(), "dep"))
        else:
            done += 1
            if queue:
                waits.append(t - queue.pop(0))
                heapq.heappush(events, (t + svc(), "dep"))
            else:
                busy = False
    tail = waits[WARMUP:]  # drop start-up bias from the empty start (ch19)
    return sum(tail) / len(tail)

for name, svc in [("D (C2s=0)", svc_d), ("M (C2s=1)", svc_m), ("2pt (C2s=10)", svc_b)]:
    print(f"{name:13s} mean wait = {mg1_mean_wait(svc):.3f} s")
```

理論值 0.2／0.4／2.2 s。每檔 200,000 支工作下的合理落點：D 應落在 0.200 ± 0.005、M 應落在 0.40 ± 0.03、二點分布應落在 2.2 ± 0.35（seed 42 的實跑：0.200、0.409、2.079）。注意第三檔的誤差帶寬了一個量級——**C²ₛ 越大，模擬收斂越慢**：等待全囤在偶發的超長忙碌期裡，樣本之間又高度相關，2.2 秒的帳單要跑很久才量得準。換幾顆種子（把 42 改成 0–9）你會看到第三檔在 1.9 到 2.4 之間晃、前兩檔紋風不動。這不是 bug，是 ch15（重尾讓它更糟）與 ch19（怎麼系統化地處理）的預告片。
