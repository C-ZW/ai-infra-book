# ch02 — 操作分析：不用機率的鐵律

> **本章解決什麼問題**：ch01 解釋了為什麼會排隊，但還沒給你任何能算的東西。本章給第一批武器：operational laws——只靠「數數字」就成立的恆等式，不需要任何分布假設。主角是 Little's Law（E[N] = λE[T]），配角是 Utilization Law、Forced Flow Law 與 Response Time Law；終場用 bottleneck 分析，不解任何模型就算出封閉系統的吞吐天花板與拐點。之後每一章的機率模型（ch03 起）都必須服從這些鐵律——它們既是地基，也是你的驗算器。

## 從你已知的出發

你在容量規劃的試算表裡寫過這條公式：

```text
concurrent ≈ RPS × avg latency
```

要撐 8,000 RPS、平均延遲 250 ms，就需要 8000 × 0.25 = 2,000 個並發槽位——連線、worker、goroutine，隨便什麼。你用它估過 worker pool 大小，也用它反推過「3,200 條 WebSocket 並發連線到底對應多少訊息流量」。

這條公式就是 Little's Law。你一直在用一條 1961 年才被一般性證明的定理，而且——這是本章的重點——你大概沒想過它什麼時候會騙你：觀察窗沒選對會騙你、系統邊界沒劃清會騙你、拿 P95 代替平均會騙你。本章把它從黑話升級成定理：精確的敘述、你能口頭重講的證明、以及適用邊界。

另一件你每天在做的事：k6 開 50 個 VU 壓 API。那是一個**封閉系統**——在途請求數固定、上一個回來才發下一個——它與線上真實流量（開放系統）的行為差異大到值得一整章（見 ch17）。本章先發給你封閉系統的第一批武器：Response Time Law 與 bottleneck bounds，讓你不寫模擬、不解模型，就能畫出壓測曲線的天花板。

## 操作分析：只信量得到的數

排隊理論的多數結果長這樣：「假設到達是 Poisson、服務是指數分布，則……」。操作分析（operational analysis）反其道而行：**只從可量測量出發，不做任何分布假設**。這套世界觀在 1970 年代由 Buzen 與 Denning 建制化（Denning & Buzen 1978 的 tutorial 至今仍是標準入口），動機很工程：機房裡的人量得到吞吐與忙碌時間，但沒有任何辦法「量到」到達過程是不是 Poisson。

選一個觀察窗，長度 τ。對窗內的一個系統（或系統裡的一台裝置），你能直接量到四個數：

| 量測量 | 意義 | 監控對應物 |
|---|---|---|
| A(τ) | 窗內到達數 | request counter 差分 |
| C(τ) | 窗內完成數 | response counter 差分 |
| B(τ) | 窗內忙碌總時間 | CPU busy time、`1 − idle` 的積分 |
| J(τ) | 系統內個數對時間的累積面積（單位：則·秒） | in-flight gauge 每秒取樣加總 |

從這四個數定義出所有導出量：吞吐 X = C/τ、使用率 ρ = B/τ、平均服務時間 E[S] = B/C（每個完成的 job 平均消耗多少服務時間）、系統內平均個數 E[N] = J/τ、平均逗留時間 E[T] = J/C。本章的 E[N]、E[T] 都先讀成這種「窗口平均」；ch03 之後它們會升級成機率期望值，兩種讀法在表現良好的系統裡相等（ch06 的 ergodic 定理會發這張執照）。

整套方法只有一個信仰之躍：**把窗口平均當成長期平均**。窗內的恆等式是算術、無可懷疑；「這 60 秒量到的 ρ 代表這個系統的 ρ」才是假設——窗口要夠長、要落在行為平穩的時段。本章所有「定律」都繼承這一個（而且只有這一個）假設。

第一條定律現在是免費的。**Utilization Law**：

```text
ρ = B/τ = (C/τ)·(B/C) = X·E[S]
```

**嚴謹度：完整推導**——其實只是把一個分數拆成兩個分數，窗內精確成立，沒有任何假設。但讀法很有內容：使用率 = 吞吐 × 平均服務時間。你的服務 X = 8 則/s、每則平均吃 100 ms，那 ρ = 0.8 不是巧合、不是量測、是算術。operational laws 全部長這樣：恆等式本身是算術，「定律」的力量在於它把幾個你以為獨立的監控指標鎖死在一起——其中一個量錯了，另外兩個會供出它。

## Little's Law：一塊面積的兩種切法

### 直覺版：面積雙算

J(τ) 是一塊面積。關鍵觀察：這塊面積有兩種切法。

三個 job 的迷你例子：job 1 在 t=0 到達、t=3 離開；job 2 在 t=1 到達、t=4 離開；job 3 在 t=2 到達、t=6 離開。先直著切——畫出 N(t)，每個瞬間數人頭：

```text
N(t)
 3 │        ┌───┐                  ← 直切：細直條，高度 = 那一瞬間的 N(t)
 2 │    ┌───┘   └───┐
 1 │┌───┘           └───────┐
 0 │┘                       └──
   └───────────────────────────→ t
    0   1   2   3   4       6
```

再橫著切——同一塊面積，每條橫帶是一個 job 從到達到離開的逗留：

```text
   t →  0   1   2   3   4   5   6
 job 1  ████████████             ← T₁ = 3 s
 job 2      ████████████         ← T₂ = 3 s
 job 3          ████████████████ ← T₃ = 4 s
```

直切加總：1+2+3+2+2 = 10 則·秒。橫切加總：3+3+4 = 10 則·秒。同一塊面積，當然相等。現在除以不同的東西：

```text
J/τ = 10/6 ≈ 1.67   ← 除以窗長：N 的時間平均
J/C = 10/3 ≈ 3.33 s ← 除以完成數：每個 job 的平均逗留
X   = C/τ = 3/6 = 0.5/s
驗證：1.67 = 0.5 × 3.33 ✓
```

這就是 Little's Law 的全部直覺：**E[N] = X·E[T] 只是同一塊面積除以兩個不同分母，再用吞吐把它們黏起來**。

### 工程師的嚴謹：有限觀察窗的證明

**嚴謹度：工程師的嚴謹——每一步都能口頭說出理由**。設窗口 [0, τ] 頭尾系統皆空（稍後放寬）。令 1ⱼ(t) 為 indicator：job j 在時刻 t 在系統內時取 1，否則取 0。

```text
J(τ) = ∫₀^τ N(t) dt          ← 面積的定義
     = ∫₀^τ Σⱼ 1ⱼ(t) dt      ← N(t) 本來就是「在場 job」的 indicator 之和
     = Σⱼ ∫₀^τ 1ⱼ(t) dt      ← 有限和與積分交換：窗內 job 數有限，合法
     = Σⱼ Tⱼ                 ← 每個 indicator 的積分就是該 job 的逗留時間
```

兩邊除以 τ，再把右邊乘除 C：

```text
J/τ = (C/τ) · (Σⱼ Tⱼ / C)
E[N] = X · E[T]              ← 窗內精確成立，零機率假設
```

窗口頭尾不空怎麼辦？跨越邊界的 job 讓 J 與 ΣTⱼ 差了一截「邊角面積」。這截誤差最多是（頭尾在場數 × 它們的窗內逗留），除以 τ 之後，只要**系統不無限堆積**（N(τ)/τ → 0）且窗夠長，邊角項就被攤薄到零。於是取 τ → ∞：若長期平均到達率 λ = lim A(τ)/τ 存在、E[T] 的極限存在，且系統穩定（完成追得上到達，X → λ），就得到正式版：

**E[N] = λE[T]。**

這個推導值得你抄一遍——它是全書最便宜也最耐用的一條定理。

### 嚴格版：Stidham 1974（指路，不證）

L = λW 在 1961 年之前是 folk theorem：所有人都在用，沒有人給過一般性證明。Little 1961 在平穩隨機過程的假設下給出第一個證明；Stidham 1974 把假設削到極薄——論文標題就叫 "A Last Word on L = λW"（「最後一句話」，數學家的冷幽默）：

```text
Stidham (1974)：對單一條樣本路徑，只要
  λ = lim A(t)/t            存在且有限
  E[T] = lim (T₁+…+Tₙ)/n    存在且有限
則 E[N] = lim (1/t)∫₀ᵗ N(s)ds 存在，且 E[N] = λE[T]。
```

注意它要的東西少到什麼程度：不需要平穩性、不需要獨立性、不需要任何分布——甚至不需要隨機。對一條「已經發生的歷史」它也成立。本書不證（證明只有幾頁，延伸閱讀有原文）；Little 本人在 50 週年回顧（2011）裡詳述了這段從黑話到定理的歷史。

### 適用條件：「無關」才是它可怕的地方

Little's Law 與以下**全部無關**：

- 到達過程的分布與相關性（Poisson 與否、叢發與否）；
- 服務時間分布（C²ₛ 是 0 還是 100 都一樣——C² 是什麼見 ch03）；
- 排程（FCFS、LCFS、processor sharing、priority……只要不弄丟 job）；
- 內部結構（單伺服器、worker pool、整條 microservice 鏈——邊界框起來就算數）。

它**需要**的只有兩件事：

1. **穩定、長期平均存在**：λ 的極限存在有限、逗留時間的平均收斂、系統不無限堆積。過載系統（到達超過容量）不滿足——E[T] 發散，定律以「兩邊都是無限大」的形式空洞地成立，對你毫無用處。
2. **邊界一致**：E[N]、λ、E[T] 三個量必須用**同一條**系統邊界、數**同一群** job。

「無關性」是雙面刃。正面：它在任何邊界上都能用，是萬用驗算器——三個平均量你量到兩個，第三個免費；三個都量到，互相對帳，對不上就是監控有 bug。反面：**它對分布不敏感，所以它對分布也一無所知**。兩個系統 E[T] 相同、P99 差 20 倍，Little 完全看不出差別。它是守恆律，不是效能模型：它說三個平均值永遠綁在一起，但從不告訴你它們各自是多少、更不告訴你 tail 長什麼樣。「為什麼是這個值」要等機率模型（見 ch08），「tail 多肥」要等 ch12 與 ch15。

### 與教科書記號的對照（全書只說這一次）

教科書傳統用 L 與 W 敘述這條定律。對照表如下，附錄 A 會再收錄一次；本書內文一律用 E[N]/E[T]——因為 N 和 T 是隨機變數，之後要談它們的分布與 tail，L/W 這種「平均值專用名」會讓你誤以為平均就是全部。

| 本書記號 | 教科書記號 | 讀法 |
|---|---|---|
| E[N] | L | 系統內平均 job 數 |
| E[N_Q] | L_Q（亦寫 Lq） | 佇列中平均 job 數（不含服務中） |
| E[T] | W | 平均逗留時間（含服務） |
| E[T_Q] | W_Q（亦寫 Wq） | 平均等待時間（不含服務） |

Little's Law 的教科書寫法 L = λW，即 E[N] = λE[T]。

## E[N] = λE[T] 三連發：邊界框到哪，定律管到哪

定律裡的「系統」是**你自己畫的虛線框**。同一台機器，框三次，得到三條不同的結論：

```text
   ┌-------------------------------------┐
   :      [o][o][o]        (( S ))       :
λ ─→      佇列 N_Q          服務中        ─→ X
   └-------------------------------------┘
     邊界 1：全框   邊界 2：只框佇列   邊界 3：只框服務位
```

1. **框整個系統**：E[N] = λE[T]。
2. **只框佇列**（服務位不算）：E[N_Q] = λE[T_Q]。等待中的 job 數對等待時間，同一條定律。
3. **只框服務位**：單伺服器的服務位裡不是 0 個就是 1 個 job，所以「服務位內平均個數」就是「服務位非空的時間比例」，也就是使用率。而 Little 說它等於 λE[S]。所以 **E[N_服務中] = λE[S] = ρ**——Utilization Law 被 Little 重新發明了一次，而且多送你一個讀法：*使用率就是「平均有多少個 job 正在被服務」*。c 台伺服器時同理：平均忙碌台數 = λE[S]，除以 c 就是每台的平均使用率。

三條合起來還白送一個恆等式：E[N] = E[N_Q] + ρ、E[T] = E[T_Q] + E[S]。

順手用它重看一眼你的舊系統：WebSocket 通知系統 ~10M 訊息/月，平均到達率 λ = 10⁷/(30 × 86400) ≈ 3.86 則/s；就算每則端到端處理要 100 ms，E[N] = λE[T] ≈ 0.386——**平均同時在處理的訊息不到一則**。3,200 條並發連線聽起來嚇人，但連線數不是負載，in-flight 才是；那個系統真正的工程難題在尖峰倍率與連線狀態管理，不在平均吞吐。一條定律就把「該擔心什麼」排了序。

### Worked example：從 60 秒的監控數據推出全套指標

推播管線的發送 worker（單一服務位），你抓了一段 60 秒的觀察窗，量到四個原始數字：完成 C = 480 則、忙碌 B = 48 s、in-flight gauge 每秒取樣積分 J = 240 則·秒；窗口頭尾在場數相同、A = C = 480（進出平衡的乾淨窗口）。一步一步推：

| 量 | 怎麼算 | 數值 |
|---|---|---|
| 吞吐 X | C/τ = 480/60 | 8.00 則/s |
| 使用率 ρ | B/τ = 48/60 | 0.800 |
| 平均服務時間 E[S] | B/C = 48/480 | 0.100 s（μ = 1/E[S] = 10 則/s） |
| 系統內平均 E[N] | J/τ = 240/60 | 4.00 則 |
| 平均逗留 E[T] | J/C = 240/480 | 0.500 s |
| 服務中平均 | = ρ | 0.800 則 |
| 佇列平均 E[N_Q] | E[N] − ρ = 4 − 0.8 | 3.20 則 |
| 平均等待 E[T_Q] | E[N_Q]/X = 3.2/8 | 0.400 s |

自我複核（操作分析的紀律就是對帳）：Utilization Law：X·E[S] = 8 × 0.1 = 0.8 = ρ ✓；Little：X·E[T] = 8 × 0.5 = 4 = E[N] ✓；分解：E[T_Q] + E[S] = 0.4 + 0.1 = 0.5 = E[T] ✓。全部咬合。

注意我們得到了什麼、沒得到什麼。得到：平均值的完整一套帳，零機率假設。沒得到：*為什麼* ρ = 0.8 時 E[N] 是 4 而不是 2？P99 是多少？這兩題操作分析天生答不了——ch08 會用 M/M/1（λ = 8、μ = 10）從機率推出一模一樣的這組平均值，外加分布與 P99。模型與量測在平均值上對得起來，是模型及格的最低門檻，不是它正確的證明。

## 封閉系統：Response Time Law

換一種系統：互動式。N 個使用者繞圈圈——想 Z 秒（think time），送出一個請求，等 E[T] 秒拿到回應，再想 Z 秒……。k6 的 VU、固定大小的 connection pool、配額固定的 batch worker，全是這種封閉迴圈。

把 Little 的虛線框畫在**整個迴圈**上：迴圈裡永遠剛好 N 個使用者（這就是「封閉」的意思），每人的一圈平均耗時 Z + E[T]，迴圈的流量是系統吞吐 X：

```text
N = X · (E[T] + Z)        ← Little 框住整個迴圈
⇒ E[T] = N/X − Z          ← Response Time Law
```

**嚴謹度：完整推導**（它就是 Little 換個邊界）。實用價值：壓測時 X 和 N 是你最容易量準的兩個數，E[T] 用這條反推，比相信 client 端統計可靠。數字感受一下：50 個 VU、think time 1 s 的腳本，量到吞吐 41.7/s，那麼 E[T] = 50/41.7 − 1 = 0.199 s——不用看任何 client 端的延遲直方圖就拿到了平均。它同時是一句警告：固定 VU 的壓測裡 N 被釘死，X 與 E[T] 被這條式子鎖在蹺蹺板兩端——這個系統**天生不會過載**，跟線上開放流量是兩種動物（見 ch17）。

## Forced Flow Law 與 service demand：瓶頸的量尺

真實系統不是一台裝置。一個 job 在系統裡會訪問多台裝置、有的還訪問多次：一則推播打一次 API node、查三次 DB。操作分析用兩個概念接住這件事。

**Visit ratio** Vᵢ = Cᵢ/C：窗內裝置 i 完成 Cᵢ 次服務、系統整體完成 C 個 job，所以每個 job 平均訪問裝置 i 共 Vᵢ 次。立刻得到 **Forced Flow Law**：

```text
Xᵢ = Cᵢ/τ = Vᵢ·(C/τ) = Vᵢ·X    ← 各裝置的流量被整體吞吐「強制」鎖定
```

**Service demand** Dᵢ = Vᵢ·E[Sᵢ]：一個 job 一生在裝置 i 消耗的總服務時間。它有個漂亮的量測性質：Dᵢ = Bᵢ/C——拿裝置忙碌時間除以系統完成數就量到了，**不需要**分別知道 Vᵢ 和 E[Sᵢ]。把 Forced Flow 塞進 Utilization Law：

```text
ρᵢ = Xᵢ·E[Sᵢ] = X·Vᵢ·E[Sᵢ] = X·Dᵢ
```

每台裝置的使用率 = 系統吞吐 × 該裝置的 service demand。Dᵢ 最大的那台裝置使用率最高、最先撞到 100%——它就是**瓶頸（bottleneck）**。小例子：一則推播查 DB 三次、每次 5 ms，D_DB = 15 ms；ρ_DB = X × 0.015 ≤ 1 直接給出 X ≤ 66.7 則/s——不管 DB 前面排了什麼、其他裝置多快，這個天花板跑不掉。

這給了你一個五分鐘的瓶頸盤點法：同一個觀察窗，抓每台裝置的忙碌時間 Bᵢ 和系統完成數 C，逐台算 Dᵢ = Bᵢ/C，排序——榜首就是瓶頸，而且你順便拿到了它卡死的吞吐上限 1/D_max。注意這跟「看誰的 CPU 最高」不同：使用率會隨當下流量浮動，Dᵢ 是每個 job 的固有成本，流量高低都量得到、也才值得進容量規劃的試算表。

## Bottleneck bounds：不解模型就畫出天花板

現在把工具串起來，處理封閉系統的核心問題：N 個使用者的互動式系統，吞吐 X(N) 和延遲 E[T(N)] 會怎麼走？精確答案要 ch17 的 MVA；但**上下界現在就能畫**，而且實務上常常夠用。

**嚴謹度：工程師的嚴謹**。兩條天花板：

```text
瓶頸天花板（任何 N 都成立）：
  ρᵢ = X·Dᵢ ≤ 1  對每台裝置 i    ← 忙碌比例不可能超過 100%
  ⇒ X ≤ 1/D_max                  ← Dᵢ 最大的那台卡死全場

人數天花板（小 N 時更緊）：
  E[T] ≥ ΣDᵢ                     ← 零排隊也得把服務需求做完
  X = N/(E[T] + Z)               ← Response Time Law 改寫
  ⇒ X ≤ N/(ΣDᵢ + Z)

拐點：兩條天花板的交點
  N* = (ΣDᵢ + Z)/D_max
```

第一條的條件：Dᵢ 是常數（service demand 不隨負載變化）。第二條另需：job 同一時刻只在一個地方接受服務（沒有 fork-join 並行——有的話 E[T] ≥ ΣDᵢ 不再成立，見 ch18）。

### Worked example：封閉基準配置的 bounds 全程

基準配置：兩站互動式系統，D₁ = 0.1 s、D₂ = 0.2 s、think time Z = 1 s。代入：

```text
ΣDᵢ = 0.3 s、D_max = 0.2 s
X(N) ≤ min( N/1.3 , 1/0.2 ) = min( N/1.3 , 5 ) 個/s
N*   = 1.3/0.2 = 6.5
E[T(N)] ≥ max( 0.3 , N·0.2 − 1 ) s    ← 第二式由 X ≤ 5 代回 Response Time Law
```

```text
X(N)
  5 │        _________________________   ← 天花板 X = 1/D_max = 5/s
    │       /:
    │      / :
    │     /  :   實際曲線只能活在屋頂下面（精確形狀見 ch17）
    │    /   :
    │   /    :  ← 斜坡 X = N/(ΣDᵢ+Z) = N/1.3
    │  /     :
    └─────────┴──────────────────────→ N
    0        N* = 6.5
```

逐點算出來（3 位有效數字）：

| N | N/1.3 | 1/D_max | X(N) 上界（取小） | E[T(N)] 下界 |
|---|---|---|---|---|
| 1 | 0.769 | 5 | 0.769 | 0.300 s |
| 2 | 1.54 | 5 | 1.54 | 0.300 s |
| 4 | 3.08 | 5 | 3.08 | 0.300 s |
| 6 | 4.62 | 5 | 4.62 | 0.300 s |
| 6.5 = N* | 5.00 | 5 | 5.00 | 0.300 s |
| 8 | 6.15 | 5 | 5.00 | 0.600 s |
| 10 | 7.69 | 5 | 5.00 | 1.00 s |
| 13 | 10.0 | 5 | 5.00 | 1.60 s |

讀法。N ≤ N*：系統空曠，每加一個使用者吞吐近乎線性成長、延遲貼著 0.3 s 的地板。N > N*：瓶頸（node 2）飽和，X 釘死在 5/s，之後每加一個使用者，多出來的時間**全數變成排隊**。N = 13 那格可以口頭重算一遍：13 個使用者每圈都要在 node 2 耗 0.2 s，node 2 每秒最多供 5 圈，一圈至少 13/5 = 2.6 s，扣掉想的 1 s，在系統裡至少 1.6 s——跟表格一致。

N* = 6.5 的讀法：這個配置「養得起」大約 6 個使用者不互相踩；第 7 個起，排隊成為常態。它不是人數所以不必是整數——它是兩條漸近線的交點。三個警告：bounds 是漸近線不是預測，真實曲線在拐點附近是圓角、離 bound 最遠；對應的悲觀下界（所有人擠在瓶頸）本書不展開，見延伸閱讀 Lazowska et al.；想要曲線本身，用 MVA（見 ch17，arrival theorem 是它的引擎，這兩條 bounds 是它的伏筆）。

但注意這件事有多便宜：我們沒有假設任何分布、沒有解任何 chain，只用了「使用率 ≤ 1」和 Little's Law，就拿到了吞吐天花板、延遲地板、和系統開始排隊的人數。這就是操作分析的全部美德——**它給的少，但它要的更少**。

## 陷阱與防禦

| 陷阱 | 它怎麼騙你 | 自我察覺與防禦 |
|---|---|---|
| 非穩態窗口 | 事故中 backlog 單調上漲的 10 分鐘窗：A(τ) ≫ C(τ)，邊角面積不可忽略，J/C 算出的「平均延遲」既不是已完成 job 的平均、也不是被堆住 job 的命運——是個沒有指涉物的數字 | 算 Little 前先看 A/C：偏離 1 超過幾個百分點、或窗口頭尾 in-flight 差距大，就換窗（拉長、或避開斜坡段） |
| 邊界不一致 | 拿「佇列深度」當 E[N]、拿「端到端延遲」當 E[T]，相除得到一個鬼吞吐；或 λ 數的是 client 發出量（含 retry）、T 卻只量成功請求 | 寫下一句話：「系統 = 從哪裡進到哪裡出」；N、λ、T 三個量逐一檢查是否服從同一句話 |
| 平均 × tail 的類別錯誤 | 「RPS × P99 = 並發 P99」——Little 只連平均；M/M/1 下 P99(T) ≈ 4.6 × E[T]（見 ch08），重尾下倍數更離譜（見 ch15），用平均公式塞 tail 數字，錯的不是數值是類別 | 看到任何「Little + 百分位」的組合直接打回票；tail 要分布，分布要模型或量測 |
| λ 用錯流 | 有丟棄、retry、dedup 時，「進入系統的流」有好幾種數法，各自給出不同的 E[T] | 在邊界上明確選定一群 job（如「被接受且最終完成」），三個量都只數它們；丟棄流的處理見 ch10、retry 迴圈見 ch16 |
| 把 Dᵢ 當常數 | bottleneck bounds 假設 service demand 與負載無關；GC、lock contention、cache 失效率會讓 Dᵢ(N) 隨 N 上升——真實 X 連 1/D_max 都摸不到、拐點比 N* 更早 | 徵兆：使用率還沒滿、吞吐已經平掉。對策：分段量 Dᵢ（不同負載各量一次），看它是不是在漂 |
| 把 bound 當預測 | 「N* = 6.5，所以 6 個使用者以內延遲就是 0.3 s」——bound 附近真實曲線是圓角，N = 6 時排隊已經明顯 | 永遠把 bounds 報成「不可能更好」；要曲線，MVA（見 ch17）或模擬（見 ch19） |

## 紙上推演

**題 1（從殘缺監控補全帳本）[10 分鐘 ★]**
一個服務節點，5 分鐘窗（τ = 300 s）量到：完成 21,600 個請求、in-flight 取樣平均 18.0 個、使用率 0.720（單服務位）。窗口進出平衡。求 X、E[T]、E[S]、E[N_Q]、E[T_Q]，並做至少兩個交叉驗算。

**題 2（升級哪台收益最大）[15 分鐘 ★★]**
封閉基準配置（D₁ = 0.1 s、D₂ = 0.2 s、Z = 1 s）、N = 10 個使用者。預算只夠把一台裝置換成兩倍速：方案 A 讓 D₁ → 0.05 s，方案 B 讓 D₂ → 0.1 s。分別算出新的 X(10) 上界與新拐點 N*，下結論：錢該花在哪、買到了什麼、沒買到什麼。

**題 3（抓出 Little 的誤用）[20 分鐘 ★★]**
事故檢討會上有三段論證，各藏一個本章等級的錯誤，指出來並給正確算法：
(a) 「事故那 600 秒，backlog 從 0 漲到 90,000 則，完成率平均 150 則/s，in-flight 取樣平均 45,000 則，所以平均延遲 = 45000/150 = 300 s。」
(b) 「尖峰 300 則/s 進來、SLA 要 P99 < 2 s，所以系統內同時最多 300 × 2 = 600 則，buffer 開 600 就絕對夠。」
(c) 「dashboard 的佇列深度平均 3.2，client 量到平均延遲 0.5 s，所以吞吐 = 3.2/0.5 = 6.4 則/s；可是 counter 明明顯示 8 則/s——監控壞了。」

### 推演解答

**題 1。**
X = C/τ = 21600/300 = 72.0/s。E[T] = E[N]/X = 18/72 = 0.250 s。E[S] = ρ/X = 0.72/72 = 0.0100 s（10.0 ms）——這是 Utilization Law 反著用。服務位平均 = ρ = 0.720，所以 E[N_Q] = 18 − 0.72 = 17.3（17.28）。E[T_Q] = E[N_Q]/X = 17.28/72 = 0.240 s。
驗算一：E[T_Q] + E[S] = 0.240 + 0.010 = 0.250 = E[T] ✓。驗算二：X·E[T] = 72 × 0.25 = 18 = E[N] ✓。常見錯路：把 ρ 當成「服務位的 E[T]」之類的時間量——它是個比例，同時也是「服務位內平均 job 數」，後者才接得上 Little。

**題 2。**
基準：X(10) ≤ min(10/1.3, 5) = 5.00/s，N* = 6.5。
方案 A（D₁ → 0.05）：ΣDᵢ = 1.25 s 含 Z…不對，先停下來把帳算清楚——ΣDᵢ = 0.05 + 0.2 = 0.25 s，分母 ΣDᵢ + Z = 1.25 s。D_max 仍是 0.2。X(10) ≤ min(10/1.25, 5) = min(8, 5) = **5.00/s，天花板紋風不動**。N* = 1.25/0.2 = 6.25——拐點還**提早**了（每圈變短，使用者回來得更快，瓶頸更早飽和）。買到的只有輕載時延遲地板 0.3 → 0.25 s。
方案 B（D₂ → 0.1）：ΣDᵢ = 0.2 s、D_max = 0.1（兩台並列）。X(10) ≤ min(10/1.2, 10) = **8.33/s**，N* = 1.2/0.1 = 12（N = 10 回到拐點以下，系統重新「空曠」）。
結論：升級非瓶頸近乎把錢丟進水裡；升級瓶頸把天花板從 5 抬到 10、把養得起的人數從 6.5 抬到 12。沒買到的：D₁ = D₂ = 0.1 並列成為新瓶頸，下一輪優化的單台收益砍半——瓶頸分析每做完一輪就要重做。

**題 3。**
(a) **非穩態窗口**。A(τ) = 90000 + 90000 = 180,000、C(τ) = 90,000，A/C = 2.0，Little 的前提（邊角項可攤薄）粉碎。300 s 這個數字內部一致（45000/150）但沒有指涉物：已完成的 9 萬則多半是事故早期的快件，平均遠小於 300 s；still-in-flight 的 9 萬則命運未卜、多半遠慘於 300 s。正確做法：已完成者直接拿時間戳記算平均；被堆住的 9 萬則另立帳本（它們至少還要 90000/150 = 600 s 才消化得完）。
(b) **平均 × tail 類別錯誤**。Little 能說的是 E[N] = 300 × E[T]——平均對平均；λ × P99 不是 N 的 P99，也不是任何分布量。佇列長度的 tail 需要 N 的分布（M/M/1 版見 ch08；真要保證不溢出，用有限容量模型算丟棄率，見 ch10）。Buffer 開 600 之後溢不溢出，這段論證一個字都沒回答。
(c) **邊界不一致**。3.2 框的是「只有佇列」，0.5 s 框的是「整個系統含服務」，相除得到的 6.4 不屬於任何邊界。同邊界配對立刻和解：佇列邊界 3.2/0.4 = 8.00 則/s；系統邊界 4.0/0.5 = 8.00 則/s（這正是本章 worked example 的那組數字）。監控沒壞，是論證壞了。

## 自我檢核

口頭回答，講不清楚就回去重讀：

1. Little's Law 的「無關性」具體指與哪些東西無關？它需要的兩個前提是什麼？少了任何一個會發生什麼事？
2. 面積雙算證明裡，「直切」和「橫切」各對應哪個物理量？哪一步用到了「窗口頭尾皆空」？放寬之後靠什麼條件救回來？
3. 為什麼 E[N_服務中] = ρ？把 Little 的虛線框畫在哪裡、單伺服器的哪個特性讓「平均個數」變成「時間比例」？
4. Stidham 1974 的版本比 Little 1961 強在哪？「對一條已經發生的歷史也成立」這句話為什麼重要？
5. Response Time Law 裡的 Z 是什麼？為什麼固定 VU 的壓測天生不會過載、而線上開放流量會？
6. X ≤ 1/D_max 的推導用了哪兩條定律？什麼工程現實會讓 Dᵢ 不是常數，此時這條 bound 還可信嗎（提示：它會變成高估還是低估）？
7. N* 的公式與直覺意義？N < N* 與 N > N* 各是哪條天花板在管事？
8. 為什麼「RPS × P99」不是任何東西的 P99？Little 在這裡的精確管轄範圍是什麼？
