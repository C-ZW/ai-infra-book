# ch08 — M/M/1：第一個完整的佇列

> **本章解決什麼問題**：零件到齊了——ch04 的指數原子、ch05 的 Poisson 到達、ch07 的生滅引擎。本章把它們組裝成第一個能完整解出來的佇列模型 M/M/1，並一次兌現前面各章開出的支票：ch01 的使用率階梯與「P99 ≈ 4.6× 平均」從體感變成定理、ch03 的級數微分正式上工、「80% 警戒線」開庭初審。途中你會拿到一個全書反覆偷用的隱形定理（PASTA）、Kendall 記號的正式戶籍資料，以及「穩態」這個詞的精確意義。ch09 之後的每一個模型，都是本章的變奏。

## 從你已知的出發

ch01 的開場是你熟悉的場景：推播管線在尖峰時段延遲飆升，worker 明明追得上總量（λ = 8 則/s < μ = 10 則/s），佇列卻照樣積壓。當時我給了你一張使用率階梯表和一條「P99 ≈ 4.6× 平均」的規則，但全部是賒帳——「ch08 推導兌現」。今天是兌現日。

先盤點你手上已經有什麼。ch05 證明了大量獨立使用者匯流出來的到達近似 Poisson(8)，而且你知道怎麼檢驗這個假設；ch04 給了無記憶性與競賽引理；ch07 把兩者組裝成生滅過程，推出了通解 πₙ = π₀·Π(λᵢ/μᵢ₊₁) 和 ergodicity 判準。本章不需要任何新零件——M/M/1 是生滅過程最簡單的特例，把 λₙ ≡ λ、μₙ ≡ μ 代進去，整個模型就解完了。剩下的工作是把解讀懂：每個數字對應你監控面板上的哪個指標、每個假設對應你架構圖上的哪個元件、每個結論在什麼條件下會反咬你。

具體一點。本章結束時，下面這些問題全部有可推導的答案，而不是經驗值：

- RDS CPU 的 80% 警戒線——80 這個數字是定理還是迷信（審判 #1 立案開庭）？

- 「平均 0.5 s」的系統，P99 是多少？這兩個數字的關係什麼時候是常數、什麼時候脫鉤（審判 #3 的第一手證據）？
- 流量從 ρ = 0.8 漲到 0.9（只多 12.5%），哪些指標翻倍、哪些翻得更兇？

另外有一件方法論的事值得先說破：πₙ 是「時間平均」——系統處於狀態 n 的時間比例。但你的使用者不住在時間軸上，他們在某個瞬間**到達**，看到的是那一瞬間的佇列。「到達者看到的」與「時間平均」憑什麼是同一件事？這個被整個行業日用而不知的等號，是本章後半的主角 PASTA。

## 先報戶口：Kendall 記號與 M/M/1 的完整假設

ch01 你已經把 Kendall 記號當地名用過，現在補辦正式戶籍。Kendall 1953 年引入的原始格式是三欄 **A/S/c**：到達過程／服務時間分布／伺服器數。後人擴充出更多欄位，通行的完整版是 **A/S/c/K/N/D**——K 系統容量上限、N 呼叫母體大小、D 服務紀律（discipline）。這個擴充常稱 Kendall–Lee 記號，通行說法歸給 A.M. Lee 1966 年的教科書（2026-06 查證於二手來源，原書未直查）。後三欄省略時的**預設值**是整套記號的隱形條款：

> K = ∞（不丟棄）、N = ∞（到達率不因在場人數改變）、D = FCFS（先到先服務）。

所以「M/M/1」這三個字元展開來是一份完整的合約。逐條攤開，並標注每條在你系統裡對應什麼、哪一章處理它破掉的情形：

| # | 假設 | 精確內容 | 工程對應與破口 |
|---|---|---|---|
| 1 | 到達 M | Poisson(λ)：增量獨立、平穩、不同時到（ch05 三定義） | 大量獨立使用者匯流≈成立；retry／批次／cron 流破它（ch05 防禦流程） |
| 2 | 服務 M | i.i.d. Exp(μ)，且與到達過程獨立 | **最常破的一條**：服務時間沒有極限定理護航（ch05 尾段），C²ₛ ≠ 1 見 ch12、重尾見 ch15 |
| 3 | c = 1 | 單一伺服器 | 單 worker、單連線池槽位；多台見 ch09 |
| 4 | K = ∞ | 佇列無上限、永不丟棄 | 現實的 buffer 都有限；有限容量見 ch10 |
| 5 | N = ∞ | 開放系統：到達率與在場人數無關 | 固定 VU 的壓測是封閉系統，先天違反（見 ch17） |
| 6 | FCFS | 先到先服務 | 平均量不挑這條（ch13 守恆律），**但回應時間的分布挑**——本章後面就會用到 |
| 7 | ρ = λ/μ < 1 | 穩定性 | 這不是建模假設，是穩態存在的條件——本章倒數第二節專門講它 |

這張表是本章的合約書。接下來每個結果出場，我都會說它用了表中哪幾條；「陷阱與防禦」再從反方向逐條驗收。

## 平穩分布：把 ch07 的通解代進去

M/M/1 的 rate diagram 是生滅過程最樸素的長相——出生率與死亡率都是常數：

```
        lam         lam         lam         lam
     ────────►   ────────►   ────────►   ────────►
  (0)         (1)         (2)         (3)   . . .
     ◄────────   ◄────────   ◄────────   ◄────────
        mu          mu          mu          mu
```

ch07 已經對一般的 λₙ、μₙ 推過通解，這裡不重推，直接代入 λₙ ≡ λ、μₙ ≡ μ（**嚴謹度：完整推導**——代入是合法的推導步驟，前提是你能口頭重講 ch07 的通解怎麼來的；紙上推演第 1 題會逼你做這件事）：

```text
πₙ = π₀ · Π(i=0..n−1) λᵢ/μᵢ₊₁ = π₀ · (λ/μ)ⁿ = π₀·ρⁿ     ← ch07 通解，連乘全是同一個因子
正規化：Σₙ πₙ = π₀ · Σₙ ρⁿ = π₀/(1−ρ) = 1                ← 幾何級數（ch03），需要 ρ < 1
⇒ π₀ = 1−ρ，  πₙ = (1−ρ)ρⁿ
```

兩行就解完了，但每一行都值得停一下。

**第一行**是 ch07 local balance 的化身：在狀態 n 與 n+1 之間下一刀（ch06 的 cut 讀法），往右的機率流 πₙ·λ 必須等於往左的 πₙ₊₁·μ，所以 πₙ₊₁ = ρ·πₙ——每往上爬一階，機率打一次 ρ 折。**第二行**是 ch03 那條幾何級數，而它的收斂條件 ρ < 1 在這裡不是技術細節：ρ ≥ 1 時級數發散、π₀ 只能是 0、不存在可正規化的平穩分布——「佇列穩不穩定」與「一條級數收不收斂」是同一個問題（ch07 的 ergodicity 判準在最簡單情形的現身）。

**適用條件**：假設表 #1–#5、#7。注意**不含 FCFS**：只要伺服器不偷懶（系統非空就以總速率 μ 出活）、且不依工作大小挑單，N(t) 的生滅結構就不變——πₙ 對 FCFS／LCFS／隨機挑單都一樣（ch13 細講）。

把基準數字代進來（λ = 8、μ = 10、ρ = 0.8）：πₙ = 0.2·0.8ⁿ。讀這個分布的三個姿勢：

- **它是幾何分布**，嚴格遞減：最有可能的單一狀態居然是「系統全空」（π₀ = 0.2）。一個忙到 80% 使用率的系統，最常見的瞬間長相是空的——但這不是好消息，因為……
- **尾巴是 ρⁿ**。對任何 n：P(N ≥ n) = Σ(k≥n) (1−ρ)ρᵏ = ρⁿ（**完整推導**：提出 ρⁿ 後又是同一條幾何級數）。代數字：P(N ≥ 10) = 0.8¹⁰ ≈ 0.107——每十個瞬間就有一個積壓上雙位數；P(N ≥ 20) = 0.8²⁰ ≈ 0.0115。分布的頭睡在 0，尾巴卻拖得很長，平均值是被尾巴抬起來的。
- **π₀ = 1−ρ 是一次免費對帳**：系統非空的時間比例 = 1−π₀ = ρ，正是 ch02 的 Utilization Law（E[N_服務中] = ρ）。機率模型與操作分析在這裡咬合，咬不合就是哪裡算錯了。

## 四個平均量與一座階梯

### E[N]：ch03 的微分技巧上工

```text
E[N] = Σₙ n·(1−ρ)ρⁿ
     = (1−ρ) · Σₙ n·ρⁿ
     = (1−ρ) · ρ/(1−ρ)²        ← ch03 微分技巧：Σ n·ρⁿ = ρ/(1−ρ)²
     = ρ/(1−ρ)
```

代 ρ = 0.8：**E[N] = 0.8/0.2 = 4**。其實你在 ch03 的紙上推演就算過這題——當時是練手感，現在它有了身分：尖峰時段的推播管線，系統內平均掛著 4 則訊息。

### Little 三連推：E[T]、E[T_Q]、E[N_Q]

有了 E[N]，剩下三個量全靠 Little's Law（ch02）與「回應時間 = 等待 + 服務」這條會計恆等式，不需要再碰任何機率：

```text
E[T]   = E[N]/λ = ρ/(λ(1−ρ)) = 1/(μ−λ) = E[S]/(1−ρ)    ← Little；三種寫法同一件事
E[T_Q] = E[T] − E[S] = ρ/(μ−λ)                            ← 扣掉自己的服務
E[N_Q] = λ·E[T_Q] = ρ²/(1−ρ)                              ← Little 框在「只有佇列」的邊界上
```

基準數字與自我複核一次做完：

| 量 | 公式 | ρ = 0.8 的值 | 對帳 |
|---|---|---|---|
| E[N] | ρ/(1−ρ) | 4 | ch03 推演重現 ✓ |
| E[T] | 1/(μ−λ) | 1/2 = **0.5 s** | λ·E[T] = 8×0.5 = 4 = E[N] ✓ |
| E[T_Q] | ρ/(μ−λ) | 0.8/2 = **0.4 s** | 0.4 + 0.1 = 0.5 = E[T] ✓ |
| E[N_Q] | ρ²/(1−ρ) | 0.64/0.2 = **3.2** | E[N] − E[N_Q] = 0.8 = ρ = E[N_服務中]（ch02）✓ |

**適用條件**：與 πₙ 相同（#1–#5、#7，不挑 discipline——但見下節，分布就挑了）。

### 帳本二：不解 πₙ 的捷徑（這裡偷用了一個定理）

同一組平均量有一條完全不碰平穩分布的路（**工程師的嚴謹**）：一個到達者眼前平均有 E[N] 個工作，每個——包括做到一半的那個——平均還要耗 1/μ（無記憶性，ch04：殘餘服務時間仍是 Exp(μ)），再加上自己的服務：

```text
E[T] = E[N]·E[S] + E[S]      ← 「到達者看到的平均人數 = E[N]」：這一步是 PASTA，下下節開庭
E[N] = λ·E[T]                ← Little
⇒ E[T] = ρ·E[T] + E[S]  ⇒  E[T] = E[S]/(1−ρ)
```

兩條方程、兩個未知數，解完。注意這條路用了三樣東西：PASTA（先賒著）、無記憶性、**FCFS**（你恰好要等完眼前那些人、不多不少——換成插隊制這行就錯了）。兩本帳算出同一個 E[T]，是本章第二次對帳成功。

### 使用率階梯：ch01 的賒帳清償

E[T] = E[S]/(1−ρ) 就是 ch01 那張階梯表的出處。代 E[S] = 100 ms：

| ρ | 1−ρ | E[T] = E[S]/(1−ρ) |
|---|---|---|
| 0.50 | 0.50 | 0.2 s |
| 0.80 | 0.20 | 0.5 s |
| 0.90 | 0.10 | 1.0 s |
| 0.95 | 0.05 | 2.0 s |
| 0.99 | 0.01 | 10 s |

ch01 教你的讀法「剩餘容量每砍半，延遲翻倍」現在升格為定理：E[T] 反比於 1−ρ，hockey stick 的形狀不是壓測工具的渲染效果，是 1/(1−ρ) 的圖形。階梯的適用條件就是 M/M/1 的合約書全文——這張表搬到別的模型會錯多少，ch01 的陷阱表已經預告，ch09／ch12／ch14 逐一結帳。

## 回應時間的完整分布：T ~ Exp(μ−λ)

平均值只是分布的第一個矩，而你的 SLA 訂在分位數上。M/M/1 慷慨到給出整條分布：

> **定理**：穩態 M/M/1 在 **FCFS** 下，回應時間 T ~ Exp(μ−λ)：P(T > t) = e^(−(μ−λ)t)。
> **適用條件**：合約書全文＋FCFS。平均量不挑 discipline，這條分布挑——換成 PS 或 LCFS，E[T] 不變、分布整個變形（ch13）。

**證明 sketch（工程師的嚴謹；每步標出引用的零件）**。到達者看到系統裡有 k 個工作的機率是 πₖ（**PASTA**，下節正式開庭）；看到 k 個，在 FCFS 下它的 T 是 k+1 段服務之和——前面 k 個各一段、自己一段，其中做到一半那個的殘餘仍是完整的 Exp(μ)（無記憶性，ch04），且各段獨立。k+1 段 i.i.d. Exp(μ) 之和是 Erlang(k+1, μ) 分布，密度 μ(μt)ᵏe^(−μt)/k!（ch05 你見過它的 k = 2 版本——round-robin 那段；一般版由卷積歸納，本書引用不重推）。把 k 用 πₖ 混起來：

```text
f_T(t) = Σ(k≥0) (1−ρ)ρᵏ · μ(μt)ᵏ e^(−μt)/k!
       = (1−ρ)μ·e^(−μt) · Σ(k≥0) (ρμt)ᵏ/k!     ← ρᵏ 與 (μt)ᵏ 併成 (ρμt)ᵏ
       = (1−ρ)μ·e^(−μt) · e^(ρμt)               ← 指數級數原地收回
       = (μ−λ)·e^(−(μ−λ)t)                       ← (1−ρ)μ = μ−λ
```

幾何權重把無窮多條 Erlang 密度疊成一條乾淨的指數——這是排隊理論最漂亮的計算之一，值得親手抄一遍。直覺版的旁註：「幾何個指數之和還是指數」也可以用 ch05 的分流定理看——服務完成流是速率 μ 的事件流，每個完成以機率 1−ρ「輪到你出場」，分流後的第一個事件自然是 Exp((1−ρ)μ)。

順帶送兩件事。其一，**等待時間** T_Q 的分布同一招可得：P(T_Q = 0) = π₀ = 1−ρ（到達看到全空，又是 PASTA），P(T_Q > t) = ρ·e^(−(μ−λ)t)——一個在 0 有質點的混合分布；ρ = 0.8 時**八成的請求都要排隊**。其二，一個反直覺的推論：T 的整條分布只透過 **μ−λ**（剩餘容量）出現。λ = 8、μ = 10 與 λ = 98、μ = 100 的回應時間分布**一模一樣**（都是 Exp(2)），儘管後者 ρ = 0.98、系統內平均掛著 49 個工作。回應時間看剩餘容量、佇列長度看使用率，兩者由 Little 黏在一起，並不矛盾——但「ρ 高 = 慢」這句直覺在比較不同規模的系統時會騙你（M/M/1 專屬的巧合，別帶去別的模型）。

### 分位數：4.6× 規則的出生證明

指數分布的 tail 是一條直線（半對數紙上），分位數隨手解：

```text
P(T > t_p) = e^(−(μ−λ)t_p) = 1−p   ⇒   t_p = ln(1/(1−p)) / (μ−λ)
p = 0.99：t₉₉ = ln 100/(μ−λ)
```

基準：t₉₉ = 4.6052/2 ≈ **2.30 s**。而 E[T] = 1/(μ−λ)，所以 t₉₉/E[T] = ln 100 ≈ 4.6052——**M/M/1 + FCFS 的 P99 永遠約是平均的 4.6 倍**，與負載無關（ρ 變了，兩者等比例變）。ch01 賒的第二筆帳清了。同場加映：中位數 t₅₀ = ln 2/(μ−λ) ≈ 0.347 s，遠低於平均 0.5 s——指數分布的平均被尾巴拉著走，「一半的使用者體感比平均快得多」與「1% 的使用者等 4.6 倍平均」是同一條分布的兩端。每多要一個 9（P99 → P99.9 → …），代價是再加 ln 10 ≈ 2.3026 倍的 E[T]：t₉₉.₉ = ln 1000/2 ≈ 3.45 s。輕尾世界裡，nines 是線性計價的；重尾世界這個價目表會被撕掉（ch15）。

這條規則的死法預告（審判 #3 的證物清單）：它依賴指數服務那條瘦尾巴與 FCFS。服務時間重尾時 P99 與平均徹底脫鉤（ch15）；discipline 換掉時分布變形（ch13）；到達叢發時整條分布右移（ch14）。「P99 ≈ 4.6× 平均」是合約內的定理、合約外的謠言。

## PASTA：到達看到的就是時間平均

本章已經兩次寫下「到達者看到 k 個的機率是 πₖ」。現在開庭審它。

πₙ 的身分是**時間平均**：系統處於狀態 n 的時間比例（ch06 的遍歷定理＋ch07 的 ergodic CTMC 給的執照）。但「到達者看到什麼」是在**特定瞬間**取樣——到達的那些瞬間。兩者憑什麼相等？

> **PASTA**（Poisson Arrivals See Time Averages；Wolff 1982，縮寫直接取自論文標題）：若到達流是 Poisson，且滿足 **lack of anticipation**（系統的現在不洩漏到達流的未來：未來的到達增量與系統當前狀態獨立），則到達者抵達瞬間（**不含自己**）看到系統處於狀態 n 的長期比例，等於時間平均 πₙ。

### 為什麼這不是廢話：一個反例

把到達換成等距（D 流），等號立刻碎掉。間隔 1 s 的確定性到達、固定 0.5 s 的確定性服務（D/D/1，ch01 的老朋友）：

```
time:     0     0.5    1     1.5    2     2.5    3
          |      |     |      |     |      |     |
arrive:   A            A            A            A
server:   [=====]      [=====]      [=====]      [==
N(t):     |--1--|--0--||--1--|--0--||--1--|--0--|
```

時間平均：系統一半時間有 1 個工作、一半時間全空——時間視角的「分布」是 (1/2, 1/2)。但**每一個到達者看到的都是空系統**：到達視角的「分布」是 (1, 0)。兩個視角給出完全不同的答案，誰都沒算錯——它們本來就是不同的問題。等距到達「挑時間」：它們永遠在伺服器剛好排空之後抵達，取樣到的全是系統最空的瞬間。任何有節律的到達流都在用自己的節律與系統的節律共舞，看到的是被共舞扭曲過的世界。

### 證明思路（直覺版＋指路）

Poisson 為什麼免疫？因為它**沒有節律可以共舞**。用 Bayes 把問題攤開（h → 0）：

```text
P(狀態 n | (t, t+h] 有到達)
= P((t,t+h] 有到達 | 狀態 n) · P(狀態 n) / P((t,t+h] 有到達)
= (λh + o(h)) · πₙ / (λh + o(h))     ← 關鍵步：條件在「狀態 n」上，到達機率仍是 λh + o(h)
→ πₙ
```

關鍵步成立的理由：時刻 t 的系統狀態是「過去的到達與服務」的函數，而 Poisson 的獨立增量＋lack of anticipation 保證未來那一小段 (t, t+h] 的到達與整個過去獨立——所以也與狀態獨立。到達「不挑時間」，取樣就無偏。回頭看 D/D/1 反例斷在哪裡：P(下一瞬有到達 | 伺服器忙) = 0——確定性到達的未來被現在完全洩漏，關鍵步的條件機率根本不等於無條件機率。把這個直覺做成嚴格證明（對軌跡層級的時間平均、用 martingale 處理收斂）是 Wolff 1982 的貢獻，**本書不證**，見延伸閱讀。

### 本章在哪裡偷用了它（自首清單）

- **帳本二**的第一行：「到達者眼前平均有 E[N] 個」——到達者看到的期望 = 時間平均的期望，PASTA。
- **T ~ Exp(μ−λ)** 證明的第一步：「到達者看到 k 個的機率是 πₖ」。
- 「**八成的請求要排隊**」：P(T_Q > 0) = 1−π₀ = ρ 把時間平均 π₀ 讀成了到達者的命運——PASTA。

同樣重要的是**哪裡沒用它**：πₙ 本身、E[N]、以及經由 Little 的 E[T]／E[T_Q]／E[N_Q] 全程不需要 PASTA——Little 是樣本路徑恆等式（ch02），對 D 流照樣成立。所以就算到達不是 Poisson，階梯表的「形狀」邏輯還在，死掉的是「到達者體感 = 時間平均」這層翻譯。

### 工程旁注：取樣的金科玉律

PASTA 給監控一條實用紀律：想無偏估計「使用者看到的系統」，用 Poisson 時間點取樣（間隔抽 Exp 的主動探針），而不是固定週期的 cron 探針——後者會跟系統的週期負載共舞出 alias。反過來，固定 VU 的壓測器是封閉迴路：上一個回應沒回來就不發下一個，到達時刻與系統狀態**強相關**，lack of anticipation 死透——它量到的「到達者視角」既不是 πₙ 也不是線上使用者的體感（ch17、ch19 接手）。

## 穩態的精確意義，以及 ρ ≥ 1 的世界

本章所有結果前面都掛著「穩態」。把這個詞攤開，它包含三件可分辨的事（ch06／ch07 的定理在 M/M/1 的落地）：

1. **極限分布**：不管從幾個工作起步，P(N(t) = n) → πₙ。
2. **時間平均**：單一軌跡待在狀態 n 的時間比例 → πₙ（遍歷定理——你的監控與 ch19 的模擬靠這條吃飯）。
3. **平穩性**：若 N(0) 就按 π 抽樣，則之後每一刻都是 π。

三件事都在說分布，**沒有一件在說軌跡會安靜下來**。穩態的 M/M/1 永遠在抖：π 自己就說了 P(N ≥ 10) ≈ 0.107——穩態「內含」風暴，風暴是分布的一部分而不是對穩態的偏離。看到佇列瞬間衝高就宣布「系統異常」，是把穩態誤解成靜止；該問的是衝高的**頻率**是否超出 π 的預算。

另一個誠實聲明：收斂（第 1 件事）需要時間，而且 ρ 越貼近 1 越慢——量級大約隨 1/(μ(1−ρ)²) 成長（**量級論證，本書不證**，精確常數略去：ρ = 0.8 時是秒級，ρ = 0.99 時是千秒級）。所以「尖峰窗內局部平穩」（ch05）這個立足點在 ρ = 0.8 很穩——幾秒就收斂、尖峰窗有兩小時；但對一個 ρ = 0.99 的系統，穩態公式描述的狀態可能比你的尖峰窗還長壽——公式沒錯，是「穩態」根本來不及發生。

### ρ ≥ 1：兩種不同的死法

把 ch07 的 ergodicity 判準代進來：正規化級數 Σρⁿ 收斂 ⇔ ρ < 1。等號與大於號是兩種病（生滅判準的分類，**引用不證**；ch06 預告過的「零常返」在此兌現）：

| 區間 | 診斷 | 症狀 |
|---|---|---|
| ρ < 1 | 正常返（positive recurrent） | 唯一平穩分布，本章全部結果成立 |
| ρ = 1 | **零常返**（null recurrent） | 必定一再排空（機率 1），但平均排空時間無限；無平穩分布，佇列像對稱隨機漫步一樣以 √t 量級遊蕩——「容量剛好打平」不是完美調校，是病 |
| ρ > 1 | 暫態（transient） | N(t)/t → λ−μ：佇列以每秒 λ−μ 個的速度線性長大，永不回頭 |

ρ > 1 的算術冷酷而簡單：λ = 12、μ = 10 的系統每秒淨積壓 2 則，一小時 7,200 則。**過載不是「變慢」，是發散**——不存在「過載下的穩態延遲」這種東西，任何有限的延遲數字都只是你量測時刻的快照，下一刻更糟。出口只有三個：降 λ（限流、丟棄——ch10 把這件事定理化）、升 μ／加 c（ch09）、或眼睜睜看 backlog 吃掉記憶體。

最後一個防呆：公式自己不會報警。把 ρ = 1.2 代進 ρ/(1−ρ)，它面不改色地吐出 −6——幾何級數在收斂半徑外的胡言亂語（ch03 的陷阱表早有預告）。**負數的 E[N] 不是 bug，是你違約的回執。**

## 開庭：0.8 → 0.9 對照表（worked example 總整理）

把本章全部結果裝進一個情境：推播尖峰從 λ = 8 漲到 λ = 9（流量 +12.5%），μ = 10 不動。每一格都用前面的公式獨立重算：

| 指標 | 公式 | ρ = 0.8 | ρ = 0.9 | 變化 |
|---|---|---|---|---|
| π₀（全空比例） | 1−ρ | 0.2 | 0.1 | 砍半 |
| E[N] | ρ/(1−ρ) | 4 | 9 | ×2.25 |
| E[N_Q] | ρ²/(1−ρ) | 3.2 | 8.1 | ×2.53 |
| E[T] | 1/(μ−λ) | 0.5 s | 1.0 s | ×2 |
| E[T_Q] | ρ/(μ−λ) | 0.4 s | 0.9 s | ×2.25 |
| t₉₉ | ln 100/(μ−λ) | 2.30 s | 4.61 s | ×2 |
| P(N ≥ 20) | ρ²⁰ | 0.0115 | 0.122 | ×10.5 |

讀法比數字重要：

- **翻倍不是巧合，是定理**。μ−λ 從 2 砍到 1，而 T 的整條分布是 Exp(μ−λ)——所以不只平均翻倍，**每一個分位數都翻倍**（t₉₉ 從 2.30 到 4.61）。「流量多一成、延遲翻一倍」這句話精確的版本是：剩餘容量砍半，回應時間分布等比例放大兩倍。
- 越往尾巴越慘烈：平均 ×2，佇列 ×2.5，「積壓破 20」的機率 ×10.5。**監控告警如果訂在佇列深度上，它對 ρ 的變化比平均延遲敏感一個數量級**——這是把告警訂在 backlog 而不是平均 latency 上的數學理由。
- **審判 #1 初審意見**：「80%」本身無罪也無功——E[N] = 4、t₉₉ = 2.30 s 可不可接受是 SLA 的事，數學管不著。數學管的是：在 ρ = 0.8 這個位置，你距離「一切翻倍」只有 12.5% 的流量成長，而流量預測的誤差常常不只 12.5%。警戒線的正確讀法不是「80 以下安全」，而是「你的反應速度追不追得上 1/(1−ρ) 的爬升速度」。全案發回 ch09（池化會移動懸崖）與 ch14（變異數會移動懸崖，方向相反）繼續審理。

### 推演解答

**解 1**：(a) 在 n 與 n+1 之間下刀：穩態下跨刀兩向機率流相等，πₙλ = πₙ₊₁μ ⇒ πₙ₊₁ = ρπₙ ⇒ πₙ = π₀ρⁿ。到這裡**還不需要** ρ < 1。正規化 Σπₙ = π₀·Σρⁿ = 1 才需要：幾何級數收斂（ρ < 1）時 π₀ = 1−ρ；ρ ≥ 1 時級數發散、π₀ 被迫為 0，平穩分布不存在——ρ < 1 的用途是讓「機率總和為 1」有解，它就是穩定條件本身。(b) 帳本一：E[N] = (1−ρ)Σnρⁿ = (1−ρ)·ρ/(1−ρ)² = ρ/(1−ρ)。帳本二：P(N ≥ n) = ρⁿ，故 E[N] = Σ(n≥1) ρⁿ = ρ/(1−ρ)。對齊 ✓。常見錯路：tail 求和從 n = 0 起跑（多算一個 1）——ch03 的 tail 公式是 Σ(n≥1) P(N ≥ n)。

**解 2**：0.8² = 0.64 → 0.8⁴ = 0.4096 → 0.8⁵ = 0.328 → 0.8¹⁰ ≈ 0.107 → 0.8²⁰ ≈ **0.0115**；0.9² = 0.81 → 0.9⁴ ≈ 0.656 → 0.9⁵ ≈ 0.590 → 0.9¹⁰ ≈ 0.349 → 0.9²⁰ ≈ **0.122**。負載 +12.5%，深積壓機率 ×10.5。第二問：**不能直接當溢出率**。P(N ≥ 20) 是「無限 buffer 系統」站在 20 以上的時間比例；真把 buffer 截在 20，到達會被拒收、系統的動態整個改變（拒收降低了有效 λ，反而把分布往下壓），正確的模型是 M/M/1/K，blocking 機率另有公式（ch10）。ρⁿ 是個堪用的**量級**估計，不是精確答案——而且 ρ 越高兩者差越多。

**解 3**：(a) 單台快機：E[T] = 1/(μ−λ) = 1/(20−16) = **0.25 s**，t₉₉ = 4.6052/4 ≈ **1.15 s**——雖然 ρ 沒變，延遲砍半。(b) 對半分流：每台是獨立的 M/M/1（λ = 8、μ = 10，分流定理 ch05 保證子流仍是 Poisson），E[T] = **0.5 s**、t₉₉ ≈ **2.30 s**——跟現在一模一樣，一毫秒都沒改善。差異來源：回應時間只看剩餘容量 μ−λ，方案 (a) 的剩餘容量是 4 則/s、方案 (b) 每台還是 2 則/s。同樣的 ρ，**規模大的系統天生佔便宜**——這就是規模經濟的佇列論雛形，也是「加機器就好」（審判 #2）案情複雜的原因：怎麼加，差很多。共用佇列方案落在兩者之間的哪裡，ch09 揭曉。

**解 4**：(i) 到達近似 Poisson ⇒ PASTA 適用：請求自記的佇列深度直方圖**同時**估計 πₙ 與到達者視角（兩者相等正是 PASTA 的內容），「需等待比例」= 看到非空的比例 → ρ。(ii) 封閉迴路：到達時刻與系統狀態強相關，lack of anticipation 不成立——它量到的分布既不是 πₙ 也不是開放流量下的到達者視角，只能描述「這個壓測腳本自己的世界」（ch17 會給它正確的數學身分）。(iii) cron 取樣估的是**時間平均** πₙ（遍歷定理，與 PASTA 無關；前提是取樣週期與系統負載沒有共同節律、且窗口夠長）——但它**不是**到達者視角；恰好線上到達是 Poisson 時，兩者數值相等（PASTA 搭的橋），到達一旦不是 Poisson，(iii) 與真實體感就分道揚鑣。

### Python 小實驗

本書的**事件驅動模擬（DES）骨架**在此首次登場——之後 ch12／ch15／ch19 的模擬都是這個骨架換零件。只用標準函式庫；統計方法學（warmup、信賴區間、batch means）**刻意全部不做**，那是 ch19 的整章主題——今天先讓骨架跑起來、跟理論對上號。

DES 的世界觀只有兩樣東西：**事件清單**（一個以時間排序的 heap，存「未來會發生什麼」）與**模擬時鐘**（不勻速流動，直接跳到下一個事件的時刻）。M/M/1 只有兩種事件：到達、離開。迴圈每一輪做三件事：從 heap 取出最早的事件 → 把時鐘撥到那一刻（途中 N 不變，順手把 N×經過時間累加進面積，這就是 E[N] 的時間平均）→ 處理事件：到達就排程下一個到達（heap 裡永遠恰好掛著一個未來到達）、伺服器閒著就立刻開工，否則進佇列；離開就記下這個工作的回應時間、從佇列拉下一個開工。

```python
import heapq
import random
import statistics

random.seed(42)

LAM, MU = 8.0, 10.0   # arrival rate, service rate (jobs/s)
N_JOBS = 200_000      # finished jobs before we stop

events = [(random.expovariate(LAM), 0, 0)]  # (time, kind, seq); kind: 0=arrival, 1=departure
queue = []            # arrival times of waiting jobs, FCFS order
serving = None        # arrival time of the job now in service
clock = area = 0.0    # simulation clock; integral of N(t) dt
n_sys = seq = 0       # jobs in system; arrival sequence number
sojourns = []         # response time T of each finished job

while len(sojourns) < N_JOBS:
    t, kind, _ = heapq.heappop(events)   # next event in time order
    area += n_sys * (t - clock)          # N(t) was constant on [clock, t)
    clock = t
    if kind == 0:                        # --- arrival ---
        seq += 1
        heapq.heappush(events, (clock + random.expovariate(LAM), 0, seq))
        if serving is None:              # server idle: service starts now
            serving = clock
            heapq.heappush(events, (clock + random.expovariate(MU), 1, seq))
        else:
            queue.append(clock)          # busy: join the FCFS queue
        n_sys += 1
    else:                                # --- departure ---
        sojourns.append(clock - serving)
        n_sys -= 1
        serving = queue.pop(0) if queue else None
        if serving is not None:          # pull next job, schedule its departure
            heapq.heappush(events, (clock + random.expovariate(MU), 1, seq))

sojourns.sort()
print(f"E[N] (time average) = {area / clock:.3f}")
print(f"E[T] (mean)         = {statistics.mean(sojourns):.4f}")
print(f"P99 of T            = {sojourns[int(0.99 * N_JOBS)]:.3f}")
```

理論值：E[N] = 4、E[T] = 0.5 s、P99 = t₉₉ ≈ 2.303 s。20 萬個工作（約 25,000 模擬秒）下，模擬值應落在 E[N] = 4 ± 0.3、E[T] = 0.5 ± 0.02、P99 = 2.30 ± 0.10——區間給得比你直覺的寬，因為相鄰工作的回應時間**高度相關**（同一段忙期裡的工作共享同一場塞車），有效樣本數遠小於 20 萬。兩個誠實聲明：①從空系統起跑會讓估計略偏低（暖機污染）；②上面的落點區間是經驗保守值，不是信賴區間——這兩件事怎麼正確處理，正是 ch19 的主題。

想拷打它：把 LAM 改成 10.5（ρ = 1.05），看 E[N] 的「時間平均」隨模擬越長越大、永不收斂——暫態沒有穩態可言，模擬器比公式誠實，它直接演給你看。再試 LAM = 10.0（ρ = 1）：遊蕩得更曖昧，這就是零常返的體感。
