# ch10 — 有限容量與不耐煩：backpressure 的定理

> **本章解決什麼問題**：ch08 的 M/M/1 有一個沒人會買單的假設——佇列無限長。真實系統處處是牆：bounded channel、connection pool 上限、會逾時放棄的客戶端、rate limiter。本章把牆蓋進模型：M/M/1/K 讓 ρ > 1 第一次合法、blocking 機率與 goodput 把 load shedding 變成可以算的定理、Erlang-A 給 timeout 一個數學身分、token bucket 與 leaky bucket 拿到佇列論戶口。ch09 給你更多伺服器、本章給你更少容量，合起來才是完整的設計空間；有限母體一段是 ch17 封閉網路的種子，整形壓低 C²ₐ 是 ch14 的伏筆。

## 從你已知的出發

你架過的每一道防線，其實都是這一章：Node.js stream 的 highWaterMark 是一面牆，寫入超過水位就回 false 叫上游收手；nginx 的 `max_conns` 是一面牆，超出的連線直接失敗（或進一條有限的等待佇列，逾時回 502）；你給 SQS consumer 設的 prefetch 上限是牆；回 429 的 rate limiter 是牆；客戶端的 timeout 加 DLQ，是「等不下去的人」的制度化。

但我猜你心裡有一條沒驗證過的道德律：**丟請求＝失敗＝可恥**。每次設計 load shedding 都像在認輸。本章要用推導翻案：在持續過載下，「不丟」這個選項根本不存在——無限 buffer 只是把丟棄改名叫遲到。本章的 worked example 會給你看一個 λ = 12、μ = 10 的過載系統（offered load 超出容量 20%），靠著丟掉 19.3% 的到達，給倖存者 0.693 秒的回應時間與接近滿載的吞吐——**過載系統靠丟棄活著，而且活得有模有樣**。「buffer 開大一點就好」這句機房黑話，本章開庭、給出判決（ch20 的審判總表會結案）。

還有一條 ch08 留下的尾巴。ch08 說 ρ ≥ 1 沒有穩態、佇列發散——可是你的系統過載時並沒有消失在數學的虛空裡，它在丟東西、在逾時、在自我節流。不是定理錯了，是模型少了牆。把牆蓋回去，ρ > 1 就從「禁區」變成「日常」，而且算得乾乾淨淨。

## M/M/1/K：把牆蓋進模型裡

模型設定：到達 Poisson(λ)、服務 Exp(μ)、單一伺服器、FCFS，系統**最多容納 K 個**（含服務中那一個）。滿了之後的到達**直接消失**——不排別處、不重試（重試是 feedback，另一隻怪物，見 ch16 與 ch20）。Kendall 記號 A/S/c/K 的第四格（ch08）終於派上用場。狀態取 n＝系統內人數，生滅率：

```text
λₙ = λ（n = 0, …, K−1）、λ_K = 0（牆）
μₙ = μ（n = 1, …, K）
```

```text
            λ                λ                    λ
      ┌─────┐ ─────► ┌─────┐ ─ ... ─► ┌─────┐
      │  0  │        │  1  │          │  K  │ ──X 滿了：到達被丟（rate λ·π_K）
      └─────┘ ◄───── └─────┘ ◄─ ... ─ └─────┘
            μ                μ                    μ
```

這是 ch07 生滅機器的標準客戶。在 n 與 n+1 之間下一刀（local balance）：

```text
λ·πₙ = μ·πₙ₊₁   ⇒   πₙ₊₁ = ρ·πₙ，其中 ρ = λ/μ
⇒ πₙ = ρⁿ·π₀（n = 0, …, K）
```

到這一步跟 ch08 一字不差——牆只改了一件事：**正規化從無窮級數變成有限和**（完整推導，ρ ≠ 1）：

```text
1 = π₀·(1 + ρ + ρ² + … + ρ^K) = π₀·(1−ρ^{K+1})/(1−ρ)   ← ch03 的有限幾何和
⇒ π₀ = (1−ρ)/(1−ρ^{K+1})
⇒ πₙ = (1−ρ)·ρⁿ/(1−ρ^{K+1})，n = 0, …, K
```

ρ = 1 時分子分母同歸於零，回頭看原式：πₙ = π₀ 對所有 n，正規化給 **πₙ = 1/(K+1)**——均勻分布。

三件值得停下來看的事：

**第一，ρ > 1 合法了。**ch08 需要 Σρⁿ 收斂，所以 ρ < 1 是生死線；這裡的和永遠是 K+1 項，無條件有限。更深一層：有限、不可約的 CTMC 必有唯一平穩分布（ch06／ch07 的存在唯一定理在有限狀態空間無條件成立）——**穩定性的全部工作都由牆承擔了**。連 ρ = 1 都從 ch08 的零常返病態（見 ch08）變成最無聊的均勻分布。

**第二，幾何形狀會翻轉。**ρ < 1 時 πₙ 遞減、機率堆在 0 那端（跟 M/M/1 同款）；ρ = 1 時均勻；ρ > 1 時 πₙ **遞增**、機率堆在牆邊——過載系統大部分時間幾乎是滿的，偶爾才透一口氣。同一條公式，三種人生。

**第三，ρ 的語義裂開了。**在 M/M/1，ρ = λ/μ 同時是「公式參數」與「伺服器忙的時間比例」（使用率），兩個身分重合（ch08）。在 M/M/1/K 它們分家：ρ 只剩公式參數，**真正的使用率是 1−π₀**，永遠小於 1——連 λ = 12 的過載系統，伺服器也只忙 96.9% 的時間（數字見 worked example）。說「這個系統 ρ = 1.2」不再表示「忙 120% 的時間」，只表示「offered load 超出容量 20%」。

## blocking、λ_eff 與 goodput：PASTA 的正確用法

**π_K 有兩個身分**，而它們相等需要一條定理。身分一（時間平均）：系統處於滿狀態的長期時間比例。身分二（到達者視角）：一個到達被擋在門外的長期比例。這兩件事在一般系統**不相等**——ch08 的反例還熱著：等距到達的系統，每個到達都看到空系統，但系統忙的時間比例可以很高。

把它們黏起來的是 **PASTA**（Wolff 1982，敘述與證明思路見 ch08）：到達是 Poisson、且到達不偷看系統狀態（lack of anticipation），則到達者看到的狀態分布＝時間平均分布。M/M/1/K 兩個條件都滿足，所以：

```text
P(到達被擋) = π_K（精確，不是近似）
```

先掛出本章的 PASTA 陣亡名單，後文各自驗屍：**有限母體**（到達率隨狀態變，lack of anticipation 直接死）、**整形後的流**（不再 Poisson）。PASTA 是定理不是反射動作。

**有效到達率**：被擋的客人永遠離開，真正進入系統的速率是

```text
λ_eff = λ·(1 − π_K)
```

進得來的都會被服務完（系統有限、FCFS、沒有人能卡到永遠），所以穩態吞吐 throughput = λ_eff。複核這件事有一條漂亮的恆等式——穩態下流入必等於流出：

```text
λ·(1 − π_K) = μ·(1 − π₀)
左邊：門口放行的速率；右邊：伺服器有人可服務的時間比例 × 服務率
代數驗證：兩邊同除 μ ⇒ ρ − ρ^{K+1}·π₀ = 1 − π₀ ⇔ π₀·(1 − ρ^{K+1}) = 1 − ρ ✓（π₀ 的定義）
```

worked example 會再用數字驗它兩次。

**Little's Law 要用 λ_eff。**E[T] = E[N]/λ_eff，不是 E[N]/λ。理由是 ch02 的邊界紀律：被擋的客人從未進入系統，不貢獻 E[N]、也沒有「逗留時間」可言；分子算的是牆內的人，分母就必須是穿牆而入的流量。用錯 λ 會低估 E[T]，過載時錯得不小（陷阱表有數字）。

**goodput**：被服務完成**且結果仍有人要**的速率。throughput 是供給側的帳，goodput 是需求側的帳。當 E[T] 超過客戶端的 timeout，每一件完工都沒人收貨——throughput 不動、goodput 歸零，伺服器全力以赴地做白工。這個區分在下一節變成數字。

## Worked example：K = 10 的兩個世界（λ = 8 與 λ = 12）

基準：K = 10、μ = 10。兩個到達情境：λ = 8（正常，ρ = 0.8）與 λ = 12（過載，ρ = 1.2）。全程手算、逐項複核；先把要用的冪次用平方法疊出來：

```text
ρ = 0.8：0.8² = 0.64、0.8⁴ = 0.4096、0.8⁸ = 0.16777
         0.8¹⁰ = 0.8⁸·0.8² = 0.16777·0.64 = 0.10737
         0.8¹¹ = 0.10737·0.8 = 0.085899
ρ = 1.2：1.2² = 1.44、1.2⁴ = 2.0736、1.2⁸ = 4.29982
         1.2¹⁰ = 4.29982·1.44 = 6.19174
         1.2¹¹ = 6.19174·1.2 = 7.43008
```

**正常情境 ρ = 0.8**：

```text
π₀  = (1−0.8)/(1−0.8¹¹) = 0.2/0.91410 = 0.21879
π_K = π₀·0.8¹⁰ = 0.21879·0.10737 = 0.023493        ← 損失 2.35%
λ_eff = 8·(1−0.023493) = 7.8121/s
複核（流量守恆）：μ·(1−π₀) = 10·0.78121 = 7.8121 ✓
```

E[N] 的封閉式：對有限幾何和做 ch03 的微分技巧（Σnρⁿ = ρ·d/dρ Σρⁿ），整理後得

```text
E[N] = ρ/(1−ρ) − (K+1)·ρ^{K+1}/(1−ρ^{K+1})
```

第一項是老朋友 M/M/1 的 E[N]，第二項是牆的折扣。代入：

```text
E[N] = 4 − 11·0.085899/0.91410 = 4 − 0.94489/0.91410 = 4 − 1.0337 = 2.9663
複核（直接加總）：Σ n·0.8ⁿ（n=1..10）= 13.5575，×π₀ = 13.5575·0.21879 = 2.9663 ✓
E[T] = E[N]/λ_eff = 2.9663/7.8121 = 0.380 s，E[T_Q] = 0.380 − 0.100 = 0.280 s
```

**過載情境 ρ = 1.2**：

```text
π₀  = (1−1.2)/(1−1.2¹¹) = (−0.2)/(−6.43008) = 0.031104
π_K = 0.031104·6.19174 = 0.19259                    ← 損失 19.3%
λ_eff = 12·(1−0.19259) = 9.6890/s
複核（流量守恆）：μ·(1−π₀) = 10·0.96890 = 9.6890 ✓
E[N] = 1.2/(−0.2) − 11·7.43008/(1−7.43008) = −6 + 81.731/6.43008 = −6 + 12.711 = 6.711
複核（直接加總）：Σ n·1.2ⁿ（n=1..10）= 215.75，×π₀ = 215.75·0.031104 = 6.711 ✓
E[T] = 6.711/9.6890 = 0.693 s，E[T_Q] = 0.593 s
```

順手加贈中間點 λ = 10（ρ = 1，均勻分布，全部口算）：πₙ = 1/11、π_K = 0.0909、λ_eff = 9.09/s、E[N] = 5、E[T] = 5/9.09 = 0.550 s。整理成一張表：

| 量 | λ = 8（ρ = 0.8） | λ = 10（ρ = 1） | λ = 12（ρ = 1.2） |
|---|---|---|---|
| π₀ | 0.219 | 0.0909 | 0.0311 |
| π_K（blocking） | 0.0235 | 0.0909 | 0.193 |
| λ_eff＝throughput | 7.81/s | 9.09/s | 9.69/s |
| 真實使用率 1−π₀ | 0.781 | 0.909 | 0.969 |
| E[N] | 2.97 | 5.00 | 6.71 |
| E[T] | 0.380 s | 0.550 s | 0.693 s |
| 丟棄速率 λ·π_K | 0.188/s | 0.909/s | 2.31/s |
| M/M/1 對照 E[T] | 0.500 s | 不存在（零常返） | 不存在（發散） |

讀表的三個重點：

1. **正常情境下牆幾乎免費還倒貼**：丟 2.35%（每分鐘約 11 則），換到 E[T] 從 M/M/1 的 0.500 s 降到 0.380 s——牆剁掉了佇列長度的尾巴，倖存者反而更快。損失可不可接受是 SLA 的事，但「有限 buffer 讓平均延遲變好」是定理。
2. **過載情境下系統活著、而且體面**：λ = 12 的 M/M/1 佇列以 λ−μ = 2/s 線性成長、延遲隨時間無限上升；M/M/1/K 丟掉 19.3%，給倖存者 0.693 s 與 9.69/s 的吞吐（容量 μ = 10 的 97%）。**過載下沒有「不丟」的選項，只有「在哪裡丟、誰付等待成本」的選項。**
3. ρ 從 0.8 → 1.2，E[T] 只從 0.380 s → 0.693 s——不到兩倍。ch08 那條 hockey stick 在這裡被牆壓平了：**延遲的爆炸被轉換成損失的成長**（2.35% → 19.3%）。爆炸守恆，只是換了出口。

## throughput–latency–loss：你只能選邊

```text
          throughput
              ^
             / \
            /   \
           /     \
      latency --- loss
```

K 是這個三角形的旋鈕。ρ < 1 時三角形是鬆的：K 開大，loss → 0、latency → M/M/1 的值，幾乎可以全拿——這就是輕載下沒人感覺到三角存在的原因。ρ > 1 時三角形繃緊，而且有一條地板定理（完整推導，兩行）：

```text
π_K = (ρ−1)·ρ^K/(ρ^{K+1}−1) = (1−1/ρ)·1/(1−ρ^{−(K+1)})   ← 上下同除 ρ^{K+1}
K→∞：ρ^{−(K+1)} → 0  ⇒  π_K → 1 − 1/ρ
```

λ = 12 時地板是 1 − 1/1.2 = 1/6 ≈ 16.7%。**再大的 buffer 都丟不少於 16.7%**——因為伺服器每秒最多消化 10 則，多出來的 2 則/s 必須從某個出口離開，這是流量守恆，不是工程不夠努力。對照同一條極限：λ_eff → λ/ρ = μ、E[T] → ∞。所以 K 在過載下只決定一件事：**在註定丟掉至少 1/6 的前提下，你願意為了那一點點損失改善付出多少延遲**。K = 10 → K = 100 的具體帳在紙上推演第 2 題：loss 19.3% → 16.7%，E[T] 0.693 s → 9.50 s——損失改善 2.6 個百分點，延遲變 13.7 倍。網路工程界給「buffer 大到延遲失控」起過一個名字：bufferbloat。

再疊上 goodput 就更難看：若客戶端 timeout 是 2 秒，K = 100 的過載系統 throughput 高達 10/s，goodput 卻趨近 0——每件工作完工時客戶早走了。**這就是 load shedding 的定理版**：過載時要在門口丟、趁早丟，把延遲留給活著的請求，把伺服器的每一秒花在會被收貨的工作上。丟棄不是放棄服務品質，是保衛 goodput。

## M/M/c/K：家族全圖

把 ch09 的多伺服器和本章的牆組起來：μₙ = min(n, c)·μ、到達在 K 截斷。πₙ 照 ch07 通解是分段幾何（n ≤ c 是階乘段、c < n ≤ K 是幾何段），推導機器完全一樣，本書不展開數值。值得帶走的是光譜的兩端：**K = c 就是 M/M/c/c（Erlang-B，見 ch09）**——完全不排隊的損失系統；**K = ∞ 就是 M/M/c（Erlang-C）**——完全不損失的等待系統。中間的 M/M/c/K 是「等待室有限」的呼叫中心、連線池、附帶佇列上限的 worker pool：現實幾乎都住在中間。一句話預告：佇列**網路**裡的 blocking（下游滿了堵住上游）會打破 ch16 的乘積形式，那是 ch18 的戰場。

## 不耐煩的數學：balking、reneging 與 timeout 的身分

牆是系統設的；不耐煩是客人自帶的。兩種經典行為（命名出處：Haight 1957、1959）：

**Balking（望之卻步）**：到達者看到 n 個人在系統，以機率 bₙ 決定加入。建模上就是狀態相依的出生率 λₙ = λ·bₙ——ch07 的生滅機器直接吃，連新定理都不用。M/M/1/K 其實是極端 balking：n < K 必進、n = K 必走。

**Reneging（等到一半放棄）**：進來排了，但每人有自己的耐心時鐘，走到先放棄。耐心取 Exp(θ) 時得到 **M/M/1+M**：佇列裡 n−1 個等待者的放棄時鐘與服務完成賽跑（ch04 競賽引理），死亡率變成

```text
μₙ = μ + (n−1)·θ（n ≥ 1）
```

多伺服器版 M/M/c+M 在 call center 文獻通稱 **Erlang-A**（A＝Abandonment；定錨論文 Garnett–Mandelbaum–Reiman 2002）。本書只取敘述層級的三個結論，但每條給理由：

1. **對任何 λ 都穩定。**死亡率隨 n 線性成長，ch07 的正規化級數通項像 λⁿ/(μ·(μ+θ)·(μ+2θ)…)，分母是階乘級成長——階乘壓死指數，級數必收斂。**不耐煩是天然的 backpressure**：系統不需要任何人設計牆，客人自己會走。
2. **過載下的損失地板與 blocking 同款**（直覺版，流體論證）：穩態時放棄率必須吸收超載量 λ−μ，所以 P(放棄) ≈ (λ−μ)/λ = 1 − 1/ρ——跟上一節 buffer 開到無限大的 blocking 地板一模一樣。流體帳上，「門口丟」和「等到放棄」丟得一樣多；差別在**成本落點**：blocking 讓被丟的人立刻知道、零等待；reneging 讓被丟的人先白等一段，甚至已佔用部分服務——無效服務推高實際負載，goodput 更難看。早拒絕優於晚放棄，這是可以推導的偏好，不是品味。
3. **timeout 的數學身分**：客戶端 timeout 就是 reneging，只是耐心是固定值（C² = 0）而不是指數。Erlang-A 選指數耐心純粹是為了讓生滅機器可解；換成固定耐心，數值會變，但「永遠穩定」「過載地板 1 − 1/ρ」的定性結論不變（敘述層級，本書不證）。倒有一道數學救不了的工程裂縫：**客戶端放棄了、伺服器不知道**，繼續替鬼魂服務——timeout 之後通常還跟著重試，這對組合是 retry storm 的前奏（見 ch20）。

## 有限母體：M/M/1//N——會自我節流的世界

到目前為止 λ 都是天上掉下來的常數。換個世界：**修理工模型（repairman model）**——N 台機器，運轉中的每台以率 λ_u 故障，故障送到唯一的修理工（Exp(μ)）前排隊。狀態 n＝待修台數，出生率

```text
λₙ = (N − n)·λ_u   ← 在場的人越多，外面能來的人越少
```

仍是生滅過程，πₙ 由 ch07 通解直接給出（πₙ = π₀·N!/(N−n)!·(λ_u/μ)ⁿ），數值本書不展開。要帶走的是兩個結構性後果：

**第一，永不爆炸。**狀態空間有限、到達率自我遞減，哪怕 N·λ_u 遠大於 μ，系統頂多全員待修，不存在發散。封閉世界天生溫柔。

**第二，PASTA 死亡。**最小反例 N = 1：唯一一台機器送修時，**永遠**看到空的修理站（它是世上唯一的客人），但修理站忙的時間比例明明大於零——到達者視角與時間平均徹底脫鉤。死因：到達率依賴狀態（機器在站內時，到達率是 0），「到達不偷看系統」的假設粉碎。一般版的正確答案是：封閉系統的到達者看到的是**少了自己的世界**（N−1 個同伴構成的系統的穩態）——這句話的精確版叫 arrival theorem，是 ch17 封閉網路與 MVA 的引擎，這裡只埋種子、不展開。

工程化身你天天在用：DB connection pool、固定數量的 worker、k6 固定 VU 的壓測。它們都是有限母體——**自我節流，所以永遠看不到開放世界的爆炸**。你的壓測為什麼總比生產樂觀？一半的答案在這裡（另一半見 ch19 的 coordinated omission；全論見 ch17）。

## rate limiting 的佇列論身分：token bucket 與 leaky bucket

M/M/1/K 的牆管**庫存**（系統內人數上限）；rate limiter 的牆管**流量**（進入速率上限）。兩種經典機制，佇列論身分完全不同：

**Token bucket**：桶容量 B 枚 token、以速率 r 補充；到達拿到 token 才放行，桶空則丟棄（policing）或標記。判定性質：任何長度 τ 的時間窗，放行量 ≤ B + r·τ。它是**容忍叢發的 admission control**：≤ B 的叢發原樣穿過——**間隔一動不動**——只有長期平均被壓到 ≤ r。對下游的意義：保證長期 ρ ≤ r/μ（U 槓桿），但到達流的形狀幾乎沒整：C²ₐ 基本原封不動。小數字：r = 8/s、B = 20，一團 30 則的瞬間叢發 → 20 則整團放行、10 則丟棄。下游還是會被那 20 則砸一下。

**Leaky bucket（佇列形態，整形器／shaper）**：進來先排進整形佇列，以固定間隔 1/r 放出。忙期內下游看到的到達間隔是常數——**C²ₐ → 0**（V 槓桿）。歷史出處：leaky bucket 由 Turner 1986 提出；token bucket 與「當計量器用的 leaky bucket」是彼此的鏡像，與「當佇列用的 leaky bucket」（整形器）是兩回事——這兩種身分常被混為一談，引用時小心。

整形值多少錢？借用一次 ch14 的 Kingman／VUT 近似先算帳（推導與適用條件全在 ch14；它對 M/M/1 恰好還原精確值，所以這個對照是公平的）。下游 μ = 10、λ = 8、C²ₛ = 1：

```text
Poisson 輸入（C²ₐ = 1）：E[T_Q] = 0.4 s              ← M/M/1 精確值（ch08 基準）
完美整形後（C²ₐ ≈ 0）：E[T_Q] ≈ (ρ/(1−ρ))·((C²ₐ+C²ₛ)/2)·E[S]
                              = 4·(1/2)·0.1 = 0.2 s   ← Kingman 近似（ch14）
```

同樣的 λ、同樣的 μ，下游等待**砍半**——你一行容量都沒加。但誠實記帳：整形是變異數的搬運工，不是焚化爐。那團 30 則的叢發在整形器裡排隊，最後一則要等 29/8 ≈ 3.6 s 才放出去——等待從下游搬到了整形器。划不划算取決於兩處等待的單價：在整形器裡等是一條 in-memory 佇列，在下游等可能佔著 DB 連線或一個 worker。**「在哪裡等」是設計變數**——這是本章三角形的第四個隱藏角。

一句話收束：token bucket 動「量」不動「形」（押 U 槓桿、保叢發）；leaky bucket 動「形」順便動量（押 V 槓桿）。ch14 的 VUT 世界觀會把這兩根槓桿放上同一張天平。

## 陷阱與防禦

| 陷阱 | 它怎麼騙你 | 防禦與察覺 |
|---|---|---|
| 拿無限 buffer 公式算有牆的系統 | E[S]/(1−ρ) 在 ρ→1 爆炸，但你的系統有牆：實際延遲飽和、損失登場，公式與現實分道揚鑣。反向也騙：只報 M/M/1/K 漂亮的 E[T] 不報 π_K，容量規劃樂觀到危險 | 監控裡出現 429/503/drop 計數的那一刻，無限 buffer 公式出局；報 E[T] 必須同時報 loss——它們是同一個三角形的兩個角 |
| Little's Law 用 λ 不用 λ_eff | λ = 12 時 E[N]/λ = 6.71/12 = 0.559 s，比真值 0.693 s 低估 19%；越過載騙得越凶 | 進 Little 的 λ 是「真的進了系統」的流量；先畫邊界、再對齊分子分母（ch02 的紀律） |
| 把 throughput 當 goodput | 過載大 buffer 系統 throughput ≈ μ 滿載漂亮，但 E[T] 一旦超過客戶端 timeout，每件完工都沒人收貨 | E[T] 與客戶端 timeout 放同一張面板；goodput 才是要守的 SLA 指標 |
| PASTA 當反射動作用 | 整形後的流（近等距）、有限母體（N=1 的機器永遠看到空站）、balking 放行後的流，到達者看到的都≠時間平均 | 用之前問兩題：到達是 Poisson 嗎？到達率會不會看狀態？兩題都過才准用（ch08） |
| 「加大 buffer」當免費修復 | ρ > 1 時損失地板 1−1/ρ 不動如山；K = 10→100 只把 loss 從 19.3% 降到 16.7%，E[T] 卻變 13.7 倍——把丟棄改名叫遲到 | 先分辨暫態尖峰還是持續過載：尖峰用 buffer 吸收合法；持續過載只有降 λ（shed／limit）或加 μ 兩條路，buffer 不在選單上 |
| 對 reneging 視而不見的伺服器 | 客戶端早已放棄，伺服器照單全做：無效服務膨脹實際負載，讓過載更過載 | 請求帶 deadline、出隊先查「還來得及嗎」；這是 ch20 retry storm 的第一道防線 |

## 紙上推演

**1. [20 分鐘 ★★] 把 M/M/1/K 重推一遍。**從生滅過程的 local balance 出發推出 πₙ 與 π₀（含 ρ = 1 的特例），然後驗收一個邊界情形：K = 1 時的 π₁ 應該等於什麼？把它跟 ch09 的 Erlang-B（c = 1、a = ρ）對一次帳，並代 λ = 8、μ = 10 算出數值。

**2. [25 分鐘 ★★★] 加大 buffer 能救過載嗎？**λ = 12、μ = 10。(a) 先推 K→∞ 時 π_K 的極限。(b) 對 K = 100 估算 π_K、λ_eff、E[N]、E[T]（提示：ρ > 1 時把狀態從牆往回數，令 m = K−n，看 π_{K−m} 像什麼分布）。(c) 與 K = 10 的結果並列，寫出「加大 buffer」的判決書。

**3. [15 分鐘 ★★] reneging 對 tail 的定性分析。**在 M/M/1+M 裡，考慮「實際被服務完成的客人」的等待時間分布。不用算，給三條定性論證：它的 tail 為什麼比同負載 FCFS M/M/1 短？哪一類客人被系統性地從「完成者」名單中剔除？這對「以完成請求量測到的 P99」這個監控指標意味著什麼偏差？

### 推演解答

**1.** 相鄰 cut 給 λπₙ = μπₙ₊₁，所以 πₙ = ρⁿπ₀。正規化：ρ ≠ 1 時 π₀ = (1−ρ)/(1−ρ^{K+1})；ρ = 1 時每項相等，πₙ = 1/(K+1)。K = 1 特例：

```text
π₀ = (1−ρ)/(1−ρ²) = 1/(1+ρ)，π₁ = ρ/(1+ρ)
```

這正是 Erlang-B 在 c = 1 的值：B(1, a) = a/(1+a)（ch09 遞迴的第一步），合理——K = c = 1 的 M/M/1/1 就是單線損失系統。代 λ = 8、μ = 10：π₁ = 0.8/1.8 = 0.444。沒有等待室時，44.4% 的到達被丟；對照 K = 10 的 2.35%——等待室的功用是把損失換成等待，這正是三角形的 latency–loss 邊。常見錯路：忘了 ρ = 1 要回到原式重新正規化，直接代公式得 0/0。

**2.** (a) 上下同除 ρ^{K+1}：π_K = (1−1/ρ)/(1−ρ^{−(K+1)}) → **1 − 1/ρ**（ρ > 1）。λ = 12 時地板 1/6 ≈ 0.167。(b) ρ > 1 時 πₙ ∝ ρⁿ 集中在牆邊。令 m = K−n：π_{K−m} ∝ ρ^{−m}——離牆的距離近似幾何分布（比率 1/ρ），E[K−N] ≈ Σm·ρ^{−m}/Σρ^{−m} = 1/(ρ−1) = 5。所以：

```text
π_K ≈ 1/6 = 0.167（修正項 ρ^{−101} 小到可忽略）
λ_eff ≈ 12·(5/6) = 10.0 = μ（伺服器近乎永不空轉：π₀ ≈ 0.2/ρ¹⁰¹ ≈ 0）
E[N] ≈ K − 1/(ρ−1) = 100 − 5 = 95.0
E[T] = E[N]/λ_eff ≈ 95.0/10.0 = 9.50 s
```

封閉式複核：E[N] = ρ/(1−ρ) − 101·ρ¹⁰¹/(1−ρ¹⁰¹) ≈ −6 + 101 = 95 ✓。(c) 判決書：K 10→100，loss 19.3% → 16.7%（改善 2.6 個百分點，地板 16.7% 永遠到不了底下），E[T] 0.693 → 9.50 s（13.7 倍）。若客戶端 timeout 在個位數秒以內，goodput 反而從「還行」變成「近零」。**加大 buffer 不能救過載；它把丟棄變成遲到，而遲到在 timeout 面前是更貴的丟棄。**

**3.** 三條定性論證（敘述層級）：（i）等待最久的客人正是耐心時鐘最可能先響的客人——「等很久才完成」這種樣本被放棄事件**選擇性移除**，完成者的等待分布右尾被系統性修剪；（ii）放棄讓死亡率隨 n 線性成長，佇列長度分布的尾巴比幾何衰減更快（分母出現階乘級），長佇列狀態本身就罕見，長等待的供給源頭被掐住；（iii）固定耐心（真實 timeout）更極端：完成者的等待被硬截在 timeout 附近之下。對監控的含義：**「以完成請求量測的 P99」在過載時會系統性偏樂觀**——最痛苦的那批使用者根本沒留下完成樣本，他們留下的是放棄與重試。量測 goodput 之外，還得量放棄率，否則面板會在系統最痛時最好看。這與 ch19 的 coordinated omission 是表親。
